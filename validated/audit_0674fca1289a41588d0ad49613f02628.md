### Title
Multi-Collateral Liquidation DoS via Arithmetic Underflow in `repaidUnits` Input Path — (`src/Midnight.sol`)

### Summary

In `Midnight.sol`'s `liquidate` function, when using the `repaidUnits` input path in a multi-collateral market where one collateral has crashed in price, the Recovery Close Factor (RCF) check can pass for a `repaidUnits` value that causes `seizedAssets` to exceed the available balance of the specific collateral being liquidated. The subsequent unchecked subtraction at line 670 then underflows, reverting the liquidation.

### Finding Description

The `liquidate` function computes `seizedAssets` from `repaidUnits` at line 652:

```solidity
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
``` [1](#0-0) 

Then at line 670, the collateral balance is decremented:

```solidity
uint128 newCollateral = _position.collateral[collateralIndex] - UtilsLib.toUint128(seizedAssets);
``` [2](#0-1) 

There is no cap ensuring `seizedAssets <= _position.collateral[collateralIndex]`. The only guard is the RCF check at lines 655–668:

```solidity
uint256 maxRepaid = lltv < WAD
    ? (_position.debt - maxDebt).mulDivUp(WAD * WAD, WAD * WAD - lif * lltv)
    : type(uint256).max;
require(
    repaidUnits <= maxRepaid
        || _position.collateral[collateralIndex].mulDivDown(...)
            .zeroFloorSub(maxRepaid) < market.rcfThreshold,
    RecoveryCloseFactorConditionsViolated()
);
``` [3](#0-2) 

The critical flaw: `maxDebt` is computed as the sum across **all** collaterals (line 613), but `seizedAssets` is computed only from the **specific** collateral being liquidated. In a multi-collateral market where one collateral has crashed, healthy collaterals inflate `maxDebt`, which shrinks `debt - maxDebt`, which in turn inflates `maxRepaid` (via the `1 / (1 - lif*lltv)` multiplier). This allows `repaidUnits` values that would require seizing far more than the crashed collateral's balance. [4](#0-3) 

**Concrete scenario:**

- Collateral 0 (liquidated): `c0 = 1000e18`, `price0 = 0.5e36` (50% crash), `lltv = 0.77e18`, `maxLif ≈ 1.0609e18`
- Collateral 1 (healthy): `c1 = 1000e18`, `price1 = 1e36`, same lltv/maxLif
- `debt = 1500e18`

After bad debt realization: `_position.debt ≈ 1413.9e18`

`maxDebt = 1000e18 × 0.5 × 0.77 + 1000e18 × 1 × 0.77 = 1155e18`

`maxRepaid = (1413.9e18 − 1155e18) / (1 − 1.0609 × 0.77) ≈ 1414e18`

RCF check passes for `repaidUnits = 1413.9e18` (≤ `_position.debt`).

But: `seizedAssets = 1413.9e18 × 1.0609 / 0.5 ≈ 2999e18 >> c0 (1000e18)` → **underflow at line 670**.

The second RCF condition (`zeroFloorSub(maxRepaid) < rcfThreshold`) evaluates to `0 < 0 = false` when `rcfThreshold = 0` (the default), so it provides no protection. [5](#0-4) 

### Impact Explanation

Liquidators using the `repaidUnits` input path in multi-collateral markets where one collateral has crashed will receive unexpected arithmetic reverts, even though the RCF check passed. This delays bad-debt realization and collateral seizure. Lenders suffer prolonged exposure to bad debt that should have been socialized. Automated liquidation bots that rely on the `repaidUnits` path will silently fail.

### Likelihood Explanation

Multi-collateral markets are a core feature of the protocol (up to 128 collaterals per market). Price crashes on individual collateral assets are realistic market events. The `repaidUnits` input path is a natural choice for liquidators who know the debt amount they want to repay. The default `rcfThreshold = 0` means the protective second condition in the RCF check is always false, maximizing exposure. No privileged access is required — any liquidator can trigger this.

### Recommendation

Before computing `seizedAssets` from `repaidUnits`, cap `seizedAssets` at the available collateral balance:

```solidity
seizedAssets = UtilsLib.min(
    repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice),
    _position.collateral[collateralIndex]
);
// Recompute repaidUnits from the capped seizedAssets to keep accounting consistent
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```

Alternatively, add an explicit check in the RCF validation that `repaidUnits` does not exceed `maxRepayableFromCollat = collateral[collateralIndex] * price / lif`.
