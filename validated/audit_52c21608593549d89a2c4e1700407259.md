Audit Report

## Title
Reentrancy via `onLiquidate` callback enables unauthorized `setConsumed` cancellation of borrower sell offers - (File: src/Midnight.sol, src/periphery/EcrecoverAuthorizer.sol)

## Summary
`Midnight.liquidate` issues an external `onLiquidate` callback to an attacker-controlled address before pulling the repayment token, with no reentrancy guard present in the contract. An attacker holding a valid pre-signed `Authorization` from the borrower can, during that callback window, submit it to `EcrecoverAuthorizer.setIsAuthorized` to gain `isAuthorized[borrower][attacker] = true`, then immediately call `Midnight.setConsumed(group, type(uint256).max, borrower)` to permanently cancel all of the borrower's sell offers in that group. The Certora formal verification suite explicitly acknowledges this reentrancy gap is out of scope.

## Finding Description

**Callback ordering in `liquidate`**

`Midnight.liquidate` executes in this order:
1. Position state is mutated (debt reduced, collateral seized, loss factor updated).
2. Seized collateral is transferred to `receiver` — line 696.
3. `ILiquidateCallback(callback).onLiquidate(...)` is called — lines 698–714 — with `callback` being fully attacker-controlled.
4. Repayment token is pulled from `payer` — line 717. [1](#0-0) 

There is no reentrancy guard anywhere in `Midnight.sol` (confirmed: zero matches for `nonReentrant`, `ReentrancyGuard`, `_locked`, `_entered`).

**`EcrecoverAuthorizer.setIsAuthorized` is permissionless**

Any address can call `EcrecoverAuthorizer.setIsAuthorized` with a pre-signed `Authorization` struct. It verifies the EIP-712 signature, increments `nonce[authorization.authorizer]`, and calls `IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer)`. [2](#0-1) 

**`setIsAuthorized` and `setConsumed` only check `isAuthorized`**

Both functions gate access solely on `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` with no additional lock or reentrancy check. [3](#0-2) [4](#0-3) 

**`liquidationLocked` does not protect here**

The contract comment explicitly states: *"Liquidations are locked for the seller during the callbacks of take."* This lock applies only during `take` callbacks, not during `liquidate` callbacks, so it provides no protection against this attack. [5](#0-4) 

**Certora specs explicitly exclude reentrancy**

The `onlyAuthorizedCanChangeConsumedExceptTake` rule in `OnlyAuthorizedCanChange.spec` carries the comment: *"Assumes no reentrancy: callbacks and token transfers are not modeled as re-entering Midnight, so re-entrant consumed changes are not covered."* [6](#0-5) 

**Full exploit flow**

Preconditions:
- Borrower has `isAuthorized[borrower][EcrecoverAuthorizer] = true` (standard periphery setup).
- Attacker holds a valid pre-signed `Authorization{authorizer: borrower, authorized: attacker, isAuthorized: true, nonce: N, deadline: T}` with nonce `N` still unused.
- Borrower is liquidatable.

Steps:
1. Attacker calls `Midnight.liquidate(..., borrower, ..., callback=attacker, ...)`.
2. Midnight mutates position state, transfers collateral to `receiver`, then calls `attacker.onLiquidate(...)`.
3. Inside `onLiquidate`, attacker calls `EcrecoverAuthorizer.setIsAuthorized(authorization, signature)`.
4. EcrecoverAuthorizer verifies the borrower's signature, increments `nonce[borrower]` to `N+1`, calls `Midnight.setIsAuthorized(attacker, true, borrower)`.
5. `isAuthorized[borrower][attacker]` is now `true`.
6. Attacker calls `Midnight.setConsumed(G, type(uint256).max, borrower)`.
7. `consumed[borrower][G] = type(uint256).max` — all sell offers in group `G` are permanently cancelled (the `consumeNonDecreasing` invariant ensures this cannot be reversed).
8. `onLiquidate` returns `CALLBACK_SUCCESS`; liquidation completes normally.

## Impact Explanation

`consumed[borrower][G]` is permanently set to `type(uint256).max`. Every sell offer the borrower has in group `G` is effectively cancelled: any subsequent `take` call will revert on `require(newConsumed <= offer.maxUnits/maxAssets, ConsumedUnits/Assets())`. The `consumeNonDecreasing` Certora rule confirms this state is irreversible — the borrower cannot recover those offers and must migrate to a new group. This constitutes unauthorized permanent state corruption of the borrower's trading positions, falling under "Unauthorized state changes" per RESEARCHER.md. [7](#0-6) 

## Likelihood Explanation

The attack requires the attacker to hold a valid pre-signed `Authorization` from the borrower. This is realistic in two concrete scenarios: (a) the borrower previously authorized an operator/trading bot that turned adversarial and holds an unused nonce; (b) the borrower authorized a contract that was later compromised. The borrower must also be liquidatable, which is a normal market condition. The attack is repeatable across multiple groups if the attacker holds authorizations with sequential nonces. Note: if the borrower has already revoked `EcrecoverAuthorizer` in Midnight (`isAuthorized[borrower][EcrecoverAuthorizer] = false`), the `setIsAuthorized` call from EcrecoverAuthorizer will revert, blocking the attack — this limits the realistic population of affected users to those who have not revoked the periphery contract.

## Recommendation

Add a reentrancy guard (e.g., a `_locked` flag checked and set at the entry of all state-mutating external functions, or OpenZeppelin's `ReentrancyGuard`) to `Midnight.sol`. Alternatively, apply a narrower fix by setting a `_liquidateLocked` flag for the duration of the `liquidate` call that blocks `setConsumed` and `setIsAuthorized` from being called re-entrantly, mirroring the existing `liquidationLocked` pattern used for `take` callbacks. [8](#0-7)

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

/// Assumes no reentrancy: callbacks and token transfers are not modeled as re-entering Midnight, so re-entrant consumed changes are not covered.
rule onlyAuthorizedCanChangeConsumedExceptTake(env e, method f, calldataarg args, address user, bytes32 group) filtered { f -> !f.isView && f.selector != sig:take(Midnight.Offer, bytes, uint256, address, address, address, bytes).selector } {
    bool userIsAuthorized = user == e.msg.sender || isAuthorized(user, e.msg.sender);

    uint256 consumedBefore = consumed(user, group);
    f(e, args);
    uint256 consumedAfter = consumed(user, group);

    assert consumedAfter == consumedBefore || userIsAuthorized;
