### Title
DoS on `repay()` via Frontrunning with `liquidate()` - (File: src/Midnight.sol)

---

### Summary

The `repay()` function performs an uncapped subtraction `position[id][onBehalf].debt -= UtilsLib.toUint128(units)`. Because `liquidate()` is permissionless (for markets without a liquidator gate) and also reduces a borrower's `debt`, an attacker can frontrun a borrower's full-debt repayment with a dust liquidation, causing the repay to revert on underflow. The attacker can repeat this indefinitely at near-zero cost, permanently blocking the borrower from repaying.

---

### Finding Description

**Root cause:** `repay()` at line 508 performs a raw checked subtraction with no cap to the current debt:

```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
``` [1](#0-0) 

Solidity 0.8.34 (the compiler version used) reverts on underflow by default. If `units` exceeds the current `debt` by even 1, the transaction reverts.

**Attacker-controlled entry path:** `liquidate()` is permissionless for markets where `liquidatorGate == address(0)` (the default). It directly decrements the borrower's debt:

```solidity
_position.debt -= UtilsLib.toUint128(repaidUnits);
``` [2](#0-1) 

**Exploit flow:**

1. Borrower has `debt = D` and is either unhealthy (pre-maturity) or in any state (post-maturity, since `postMaturityMode` only requires `block.timestamp > market.maturity`).
2. Borrower reads their current debt on-chain and submits `repay(units = D)`.
3. Attacker sees the pending transaction and frontruns with `liquidate(repaidUnits = 1, seizedAssets = 0)`.
   - The RCF check passes trivially because `1 <= maxRepaid` for any unhealthy position.
   - In post-maturity mode the RCF check is skipped entirely.
   - Cost to attacker: 1 wei of loan token (18-decimal token → ~$0).
4. Borrower's debt is now `D - 1`.
5. Borrower's `repay(units = D)` executes: `(D - 1) - D` → arithmetic underflow → revert.
6. Attacker repeats on every retry.

The liquidation condition check:

```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
``` [3](#0-2) 

Post-maturity, this condition is satisfied for **any** borrower with `debt > 0`, making the attack universally applicable after market expiry.

The liquidator gate is the only mitigation, but it is optional:

```solidity
require(
    market.liquidatorGate == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
``` [4](#0-3) 

---

### Impact Explanation

A borrower who attempts to repay their exact full debt (a common UX pattern: "repay all") can be permanently DoS'd. Each retry is frontrun at ~$0 cost to the attacker. The borrower's position remains open, accruing risk (liquidation, bad-debt socialization). In post-maturity markets, **every borrower** is vulnerable regardless of health status. This constitutes a permanent lock/freeze of user state reachable by a no-privilege attacker.

---

### Likelihood Explanation

- **No privileged access required.** Any EOA or contract can call `liquidate()` on a liquidatable position.
- **Near-zero cost.** 1 wei of loan token per frontrun; the attacker may even receive dust collateral in return via `seizedAssets`.
- **Post-maturity universality.** After `market.maturity`, every borrower with `debt > 0` is a valid target, making this a systemic risk at market expiry — precisely when borrowers most need to repay.
- **Repeatable.** The borrower cannot escape by retrying with the same `units` value; the attacker simply frontruns again.

---

### Recommendation

Cap `units` to the borrower's current debt inside `repay()`, mirroring the recommended fix from M-09:

```solidity
function repay(Market memory market, uint256 units, address onBehalf, address callback, bytes calldata data)
    external
{
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    bytes32 id = touchMarket(market);

    uint128 currentDebt = position[id][onBehalf].debt;
    uint128 repaidUnits = units > currentDebt ? currentDebt : UtilsLib.toUint128(units);
    position[id][onBehalf].debt -= repaidUnits;
    marketState[id].withdrawable += repaidUnits;
    ...
}
```

This ensures that passing `type(uint256).max` (or any value exceeding current debt) safely repays the full outstanding debt without reverting, eliminating the frontrunning surface entirely.

### Proof of Concept

**Setup:**
- Market with `liquidatorGate = address(0)`, `maturity = block.timestamp + 1` (expires in 1 second).
- Borrower has `debt = 1000e18`.

**Steps:**

1. Wait for `block.timestamp > market.maturity`.
2. Borrower calls `repay(market, 1000e18, borrower, address(0), "")`.
3. Attacker frontruns with `liquidate(market, 0, 0, 1, borrower, true, attacker, address(0), "")`.
   - `postMaturityMode = true` → no health check, no RCF check.
   - `repaidUnits = 1` → `position[id][borrower].debt` becomes `1000e18 - 1`.
   - Attacker pays 1 wei of loan token.
4. Borrower's `repay(units = 1000e18)` executes against `debt = 1000e18 - 1`:
   - `(1000e18 - 1) - 1000e18` → **underflow revert**.
5. Borrower retries; attacker frontruns again. Cycle repeats indefinitely.
