Audit Report

## Title
Arithmetic Underflow in `liquidate` When Seizing All Collateral at Price Just Above Bad-Debt Threshold - (File: `src/Midnight.sol`)

## Summary

When a liquidator passes `seizedAssets = collateral` at a price where the two-step `mulDivUp` ceiling expression evaluates to `originalDebt + 1`, the bad-debt block is skipped (`badDebt = 0`, so `_position.debt` remains `originalDebt`), but `repaidUnits` — computed with the identical ceiling expression — equals `originalDebt + 1`. The checked subtraction `_position.debt -= UtilsLib.toUint128(repaidUnits)` at line 676 then reverts with an arithmetic underflow, permanently blocking the `seizedAssets`-input liquidation path at this price boundary.

## Finding Description

**Root cause — rounding asymmetry between bad-debt check and `repaidUnits` computation.**

The bad-debt residual is zeroed via ceiling rounding at lines 614–616:

```solidity
badDebt = badDebt.zeroFloorSub(
    _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
);
```

`badDebt = 0` whenever `_collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif) >= originalDebt`. When `badDebt = 0`, the `if (badDebt > 0)` block at line 626 is skipped, so `_position.debt` remains `originalDebt`.

When the liquidator passes `seizedAssets = collateral`, `repaidUnits` is computed at line 650 with the **identical two-step ceiling expression**:

```solidity
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```

In pre-maturity mode, `lif = _maxLif = maxLif`, so `repaidUnits = collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif)` — the same expression as in the bad-debt computation.

The two-step ceiling can produce a result strictly greater than the real value `v = collateral * price / ORACLE_PRICE_SCALE * WAD / maxLif` by up to 2 units due to intermediate rounding. Specifically, when the intermediate result `ceil(collateral * price / ORACLE_PRICE_SCALE)` is not an integer multiple of `maxLif / WAD`, the second ceiling adds another unit. This means `repaidUnits` can equal `originalDebt + 1` even when `v` is exactly `originalDebt` (or in `(originalDebt, originalDebt + 1)`).

**Concrete numerical example** (toy values to isolate the two-step ceiling compounding):
- `collateral = 2`, `price = 3`, `ORACLE_PRICE_SCALE = 4`, `WAD = 4`, `maxLif = 2`, `units = 3`
- Real value: `2 * 3 / 4 * 4 / 2 = 3`
- Two-step ceiling: `ceil(2 * 3 / 4) = ceil(1.5) = 2`, then `ceil(2 * 4 / 2) = 4`
- `repaidUnits = 4 > _position.debt = 3` → underflow

In production with `ORACLE_PRICE_SCALE = 1e36` and `WAD = 1e18`, the same intermediate rounding occurs whenever `collateral * price` is not divisible by `ORACLE_PRICE_SCALE`, and the intermediate ceiling result is not divisible by `maxLif / WAD`.

**Why the RCF check does not prevent this.**

The RCF check at lines 655–668 computes `maxRepaid` as:

```solidity
uint256 maxRepaid = lltv < WAD
    ? (_position.debt - maxDebt).mulDivUp(WAD * WAD, WAD * WAD - lif * lltv)
    : type(uint256).max;
require(
    repaidUnits <= maxRepaid
        || _position.collateral[collateralIndex].mulDivDown(...)
            .zeroFloorSub(maxRepaid) < market.rcfThreshold,
    RecoveryCloseFactorConditionsViolated()
);
```

Neither branch checks `repaidUnits <= _position.debt`. For a position modestly underwater, `maxRepaid` is a multiple of `(_position.debt - maxDebt)` and can easily exceed `originalDebt + 1`. Markets with `rcfThreshold = type(uint256).max` trivially satisfy the second condition for any `repaidUnits`.

**Exploit flow:**

1. Market: single collateral, any `lltv < WAD`, normal pre-maturity mode, `rcfThreshold = type(uint256).max` (or any value large enough).
2. Borrower has `debt = units`, `collateral = units.mulDivUp(WAD, lltv)`.
3. Oracle set to a price where `collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif) = units + 1` (i.e., the two-step ceiling overshoots by 1).
4. At this price: `badDebt = 0` (position unhealthy but no bad debt), `_position.debt = units`.
5. Liquidator calls `liquidate(seizedAssets = collateral, repaidUnits = 0, postMaturityMode = false)`.
6. `repaidUnits = units + 1`.
7. RCF check passes.
8. `_position.debt -= UtilsLib.toUint128(units + 1)` → arithmetic underflow → revert.

## Impact Explanation

The `seizedAssets = collateral` liquidation path reverts unconditionally at this price boundary. The position is genuinely unhealthy and the liquidator is entitled to seize collateral, but the transaction always reverts. This is a service availability failure: a valid, documented liquidation input mode is permanently broken at a deterministic price value. The `repaidUnits = debt` input path (which uses `mulDivDown` at line 652) is unaffected, so the position is not permanently unliquidatable, but the full-collateral-seizure path is blocked for any liquidator observing this oracle price.

## Likelihood Explanation

The price window is a single integer value, which is narrow in relative terms but deterministic and repeatable. Any oracle returning integer prices will hit it with nonzero probability. No privileged access is required — any unprivileged liquidator can trigger the revert by passing `seizedAssets = collateral`. Markets with `rcfThreshold = type(uint256).max` are unconditionally vulnerable whenever the oracle returns this exact price. The condition is reachable by any liquidator observing the oracle price.

## Recommendation

Cap `repaidUnits` at `_position.debt` before the subtraction at line 676 to prevent the underflow:

```solidity
repaidUnits = UtilsLib.min(repaidUnits, _position.debt);
_position.debt -= UtilsLib.toUint128(repaidUnits);
```

Alternatively, apply the same cap immediately after computing `repaidUnits` at line 650, before the RCF check, so that `maxRepaid` and the RCF condition are evaluated against the capped value. This preserves the invariant that `repaidUnits <= _position.debt` at all times.

## Proof of Concept

**Minimal manual steps:**

1. Deploy a market with a single collateral, `lltv = 0.5e18`, `maxLif = 2e18`, `rcfThreshold = type(uint256).max`.
2. Borrower supplies `collateral = 2` and borrows `units = 3` (set via storage write as in `testLiquidateFullyRepayOrFullySeizeWhenRcfDeactivated`).
3. Set oracle price to `3` with `ORACLE_PRICE_SCALE = 4`.
4. Verify: `collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif) = ceil(ceil(2*3/4)*4/2) = ceil(ceil(1.5)*2) = ceil(2*2) = 4 > 3 = units`. Bad debt = 0.
5. Call `liquidate(seizedAssets = 2, repaidUnits = 0, postMaturityMode = false)`.
6. Observe revert with arithmetic underflow at line 676.

**Fuzz test plan:** Add a fuzz test in `LiquidationTest.sol` that, for a single-collateral market with `rcfThreshold = type(uint256).max`, iterates over `(units, collateral, price)` triples and asserts that `liquidate(seizedAssets = collateral, ...)` never reverts with an arithmetic underflow when `badDebt == 0` and the position is liquidatable. The existing `testLiquidateFullyRepayOrFullySeizeWhenRcfDeactivated` test does not cover this case because it bounds `units <= repayableDebt = mulDivDown(collateral, WAD, maxLif)` (floor), which does not exclude the case where the two-step ceiling overshoots.
