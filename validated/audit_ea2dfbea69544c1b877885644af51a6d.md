### Title
Zero Oracle Price Not Guarded in `seizedAssets` Path of `liquidate()` — (`src/Midnight.sol`)

---

### Summary

In `Midnight.sol`, the `liquidate()` function initializes `liquidatedCollatPrice` to `0` and only assigns it when the bitmap loop reaches `collateralIndex`. When the oracle for the liquidated collateral returns `0`, the `seizedAssets` branch silently computes `repaidUnits = 0` via `mulDivUp(seizedAssets, 0, ORACLE_PRICE_SCALE)`, allowing a liquidator to seize collateral while repaying zero loan tokens and reducing zero debt. The `repaidUnits` path is correctly protected (it reverts on division by zero), but the `seizedAssets` path is not, creating an asymmetric and exploitable gap.

---

### Finding Description

**Vulnerability class:** Pricing/accounting — incomplete zero-price guard (direct analog of the external report's `priceSource == address(0)` issue).

In `liquidate()`:

```solidity
uint256 liquidatedCollatPrice;          // initialized to 0
...
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    uint256 price = IOracle(_collateralParam.oracle).price();
    if (i == collateralIndex) liquidatedCollatPrice = price;   // only set here
    ...
}
``` [1](#0-0) 

If the oracle for `collateralIndex` returns `0`, `liquidatedCollatPrice` remains `0`. The two branches then diverge:

```solidity
if (seizedAssets > 0) {
    // mulDivUp(seizedAssets, 0, ORACLE_PRICE_SCALE) == 0  → repaidUnits = 0  ← NO REVERT
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
} else {
    // mulDivDown(ORACLE_PRICE_SCALE, 0) → division by zero  ← REVERTS
    seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
}
``` [2](#0-1) 

`mulDivUp` is implemented as `(x * y + (d - 1)) / d`: [3](#0-2) 

With `y = 0`: `(seizedAssets * 0 + (ORACLE_PRICE_SCALE - 1)) / ORACLE_PRICE_SCALE = 0`. So `repaidUnits = 0` — no revert, no debt repaid.

Execution then continues:
- Line 670: `_position.collateral[collateralIndex] -= seizedAssets` — collateral is deducted from borrower
- Line 675: `_marketState.withdrawable += 0` — no loan tokens added to pool
- Line 676: `_position.debt -= 0` — borrower's debt unchanged
- Line 696: `safeTransfer(token, receiver, seizedAssets)` — collateral sent to liquidator
- Line 717: `safeTransferFrom(loanToken, payer, address(this), 0)` — liquidator pays nothing [4](#0-3) 

The protocol's own LIVENESS documentation acknowledges only the `repaidUnits` path:

> "If the liquidated collateral oracle returns 0 on price, liquidate with repaid input reverts." [5](#0-4) 

The `seizedAssets` path is not mentioned and is not protected. The Certora `NoDivisionByZero.spec` also explicitly assumes `ghostPrice(...) > 0` to prove safety, confirming the zero-price case is unhandled: [6](#0-5) 

---

### Impact Explanation

A liquidator can call `liquidate(market, collateralIndex, seizedAssets=N, repaidUnits=0, borrower, ...)` when the oracle for `collateralIndex` returns `0`. The result:

- Liquidator receives `N` collateral tokens at zero cost (zero loan tokens transferred in).
- Borrower's debt is not reduced — they still owe the full amount but have lost collateral.
- `withdrawable` is not increased — lenders receive nothing.
- Protocol accounting is corrupted: collateral balance decreases, debt balance unchanged.

This is direct theft of borrower collateral with no compensation to the protocol or lenders.

---

### Likelihood Explanation

An oracle returning `0` is a realistic condition:

1. **Price feed staleness / circuit breaker**: Many oracle implementations (Chainlink, custom wrappers) return `0` or a sentinel value when the feed is stale, paused, or the aggregator has no valid round. A market creator deploying a collateral with such an oracle creates the precondition.
2. **Attacker-controlled oracle**: `touchMarket` validates `lltv`, `maxLif`, and collateral token ordering, but does **not** validate that `oracle != address(0)` or that `oracle.price() > 0`. A market can be created with an oracle that returns `0` under attacker-controlled conditions.
3. **No privilege required**: Any external caller can call `liquidate()`. The only precondition is that the position is liquidatable (which is trivially satisfied when the oracle returns `0`, since `maxDebt` collapses to `0` for that collateral, making `originalDebt > maxDebt`). [7](#0-6) 

---

### Recommendation

Add an explicit guard in the `seizedAssets` branch mirroring the implicit revert in the `repaidUnits` branch:

```solidity
if (seizedAssets > 0) {
    require(liquidatedCollatPrice > 0, ZeroOraclePrice());
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
} else {
    seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
}
```

This mirrors the fix applied in Morpho PR 590: only use the price when it is non-zero.

---

### Proof of Concept

1. Deploy a market with a collateral whose oracle can be set to return `0` (e.g., the test `Oracle` contract).
2. Borrower supplies collateral and borrows.
3. Set oracle price to `0`: `Oracle(market.collateralParams[0].oracle).setPrice(0)`.
4. Attacker calls:
   ```solidity
   midnight.liquidate(
       market,
       0,                          // collateralIndex
       collateralAmount,           // seizedAssets = full collateral
       0,                          // repaidUnits = 0
       borrower,
       false,
       attacker,
       address(0),
       ""
   );
   ```
5. **Expected (correct)**: revert.
6. **Actual**: attacker receives `collateralAmount` tokens, pays `0` loan tokens, borrower's debt unchanged.

The position is liquidatable because `maxDebt = collateral * 0 / ORACLE_PRICE_SCALE * lltv / WAD = 0 < originalDebt`. [8](#0-7)
