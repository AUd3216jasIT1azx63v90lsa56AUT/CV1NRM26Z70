Audit Report

## Title
Fully-consumed assets-based buy offer with `buyerPrice < WAD` can be taken indefinitely with zero asset transfer but non-zero position mutations - (`src/Midnight.sol`)

## Summary
When a buy offer with `maxAssets > 0` is fully consumed and `buyerPrice < WAD`, the consumed-assets guard can be bypassed indefinitely by supplying `units = 1`. Because `mulDivDown(1 * buyerPrice, WAD)` truncates to `0`, the consumed counter does not advance past `maxAssets`, yet all position mutations (credit, debt, `totalUnits`) execute on the full `units = 1` value. The protocol comment at line 94 and the test `testBugBuyMaxAssetsBypass` both explicitly acknowledge this as a bug.

## Finding Description
**Exact code path — `src/Midnight.sol` lines 363–373:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // truncates to 0 when buyerPrice < WAD, units = 1
    : units.mulDivUp(buyerPrice, WAD);

uint256 newConsumed;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // maxAssets + 0 ≤ maxAssets → passes
}
``` [1](#0-0) 

**Root cause:** The guard at line 369 checks `newConsumed <= offer.maxAssets`. When `buyerPrice < WAD` (e.g., tick set to `MAX_TICK - 16`), `mulDivDown(1 * buyerPrice, WAD) = 0`. Adding `0` to an already-maxed `consumed` value satisfies `<= maxAssets`, so the guard passes. All subsequent position mutations at lines 408–417 execute unconditionally on `units = 1`, not on `buyerAssets = 0`. [2](#0-1) 

**Protocol acknowledgement:** The comment at line 94 explicitly states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* [3](#0-2) 

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — passes because `maxAssets + 0 ≤ maxAssets`
- No check enforces `units == 0` when `consumed >= maxAssets`
- No check enforces `buyerAssets > 0` when `units > 0`

## Impact Explanation
After an offer reaches `consumed == maxAssets`, any unprivileged taker can call `take(offer, ..., units=1)` indefinitely. Each call:
- Passes the `ConsumedAssets` guard (consumed does not increase)
- Increases the buyer/maker's `credit` by `buyerCreditIncrease` derived from `units = 1`
- Increases the seller/taker's `debt` by `sellerDebtIncrease` derived from `units = 1`
- Increases `totalUnits` by `buyerCreditIncrease`
- Transfers zero loan tokens (`buyerAssets = sellerAssets = 0`)

This corrupts the credit/debt accounting invariant and inflates `totalUnits` without any corresponding asset backing, at zero token cost to the attacker. [4](#0-3) 

## Likelihood Explanation
All preconditions are attacker-reachable without any privileged access:
1. `offer.buy = true` and `offer.maxAssets > 0` — standard offer configuration
2. `buyerPrice < WAD` — achievable at any tick where `tickToPrice(tick) + settlementFee < WAD` (e.g., `MAX_TICK - 16`)
3. Offer fully consumed — attacker can self-consume as first taker, or wait for organic fills

The attack is repeatable in every block at negligible gas cost since no token transfers occur.

## Recommendation
Add an explicit guard requiring `buyerAssets > 0` (or equivalently `units == 0`) when the offer is assets-capped and the computed `buyerAssets` rounds to zero:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, after computing `newConsumed`, add `require(units == 0 || newConsumed > consumed_before)` to ensure every non-zero-unit take actually advances the consumed counter.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` at line 858 is a complete, self-contained proof of concept: [5](#0-4) 

It:
1. Sets `maxAssets = 1`, `tick = MAX_TICK - 16` (ensuring `offerPrice < WAD`)
2. Pre-consumes the offer to `maxAssets` via `setConsumed`
3. Calls `take(1, borrower, lenderOffer)` — the bypass
4. Asserts `buyerAssets == 0`, `sellerAssets == 0`, consumed unchanged, token balances unchanged
5. Asserts `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased

        lenderOffer.maxAssets = 1;

        // Fully consume the offer before the take.

        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);

        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
