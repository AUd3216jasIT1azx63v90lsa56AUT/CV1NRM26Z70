Audit Report

## Title
Fee-on-Transfer Loan Token Inflates `claimableSettlementFee` Beyond Actual Contract Receipts - (File: src/Midnight.sol / src/libraries/SafeTransferLib.sol)

## Summary
In `Midnight.sol`'s `take` function, `claimableSettlementFee[offer.market.loanToken]` is incremented by the nominal `buyerAssets - sellerAssets` at line 418 before the inbound transfer executes. `SafeTransferLib.safeTransferFrom` at line 455 only verifies the boolean return value and non-revert; it performs no balance-before/balance-after check. When the market's `loanToken` is a fee-on-transfer token, the contract receives strictly less than the nominal amount, permanently inflating `claimableSettlementFee` relative to actual holdings. Subsequent calls to `claimSettlementFee` will attempt to transfer the inflated amount, either reverting (DoS) or silently draining lender-withdrawable funds co-mingled in the same ERC-20 balance.

## Finding Description

**Confirmed code path:**

1. **`src/Midnight.sol` line 418** — state written unconditionally before any external call:
   ```solidity
   claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
   ```

2. **`src/Midnight.sol` lines 420–422** — `payer` resolves to `buyerCallback` (taker-supplied) when non-zero:
   ```solidity
   address buyerCallback = offer.buy ? offer.callback : takerCallback;
   address payer = buyerCallback != address(0) ? buyerCallback : ...;
   ```

3. **`src/Midnight.sol` line 455** — inbound transfer pulls `buyerAssets - sellerAssets` from `payer`:
   ```solidity
   SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
   ```

4. **`src/libraries/SafeTransferLib.sol` lines 27–33** — only checks revert propagation and boolean return; no balance delta verification:
   ```solidity
   (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
   ...
   require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
   ```

**Root cause:** `safeTransferFrom` does not verify the actual amount received by `address(this)`. A fee-on-transfer token deducts a fee silently while returning `true`, so the call succeeds but the contract receives `(buyerAssets - sellerAssets) * (1 - fee_rate)`. Because `claimableSettlementFee` was already incremented by the full nominal amount at line 418, the accounting is permanently wrong by `(buyerAssets - sellerAssets) * fee_rate` per `take`.

**Formal verification explicitly excludes this case.** `certora/specs/Solvency.spec` line 31 states:
```
// Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing...
```
The `tokenBalanceCorrect` strong invariant is therefore not proven for fee-on-transfer tokens, confirming the protocol's own proofs do not cover this scenario.

**Why existing checks fail:** The only checks on the transfer are revert-propagation and the boolean return value. There is no `balanceOf(address(this))` snapshot before/after the transfer. The `claimableSettlementFee` increment at line 418 is unconditional and precedes the transfer. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

## Impact Explanation

After each such `take`, `claimableSettlementFee[loanToken]` exceeds the contract's actual token balance attributable to settlement fees by `(buyerAssets - sellerAssets) * fee_rate`. When the protocol owner calls `claimSettlementFee`, it will attempt to transfer more tokens than the contract holds for that purpose. Since all loan token balances (withdrawable lender assets, settlement fees) are co-mingled in the same ERC-20 balance, the shortfall is covered by lender-withdrawable funds — constituting direct theft of lender assets. Alternatively, if the balance is insufficient, `claimSettlementFee` reverts, causing a DoS on fee collection. The core solvency invariant — "contract token balances must cover withdrawable assets, collateral claims, credit redemptions, and accrued fees" — is violated. This matches the `live_context.json` best bug classes of "protocol insolvency" and "credit/debt accounting corruption." [6](#0-5) [7](#0-6) 

## Likelihood Explanation

**Preconditions:**
1. A market exists whose `loanToken` is a fee-on-transfer token. Market creation is permissionless; no protocol check restricts the loan token type. The `live_context.json` explicitly lists "Can non-standard token behavior break accounting assumptions?" as a highest-priority question, and `live_context.json` line 232–233 states fee-on-transfer "should be tested if not explicitly excluded" — and SECURITY.md contains no such exclusion.
2. A taker submits a sell offer take with `takerCallback` pointing to a contract implementing `IBuyCallback` that holds the fee-on-transfer token and approves Midnight in `onBuy`. `takerCallback` is a free, unchecked parameter.

Both preconditions are fully attacker-controllable. The attack is repeatable on every `take` in such a market, compounding the discrepancy linearly. No privileged access is required. [8](#0-7) [9](#0-8) 

## Recommendation

Perform a balance-before/balance-after check around the inbound `safeTransferFrom` call and use the actual received amount to update `claimableSettlementFee`:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived;
```

This requires moving the `claimableSettlementFee` increment to after the transfer. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens (e.g., via a market creation check or clear documentation), but this does not fix existing markets. ## Proof of Concept

**Foundry fork test outline:**

1. Deploy a mock ERC-20 with a 1% transfer fee (deducted from the received amount on every `transferFrom`).
2. Create a Midnight market with this token as `loanToken`.
3. Create a buy offer (maker) and a matching sell offer take (taker with `takerCallback`).
4. Deploy a `IBuyCallback` contract that holds the fee-on-transfer token and approves Midnight in `onBuy`.
5. Call `take` with `takerCallback` set to the callback contract.
6. Assert: `claimableSettlementFee[feeToken]` equals `buyerAssets - sellerAssets` (nominal).
7. Assert: `IERC20(feeToken).balanceOf(address(midnight))` equals `(buyerAssets - sellerAssets) * 0.99` (actual received).
8. Assert: the gap equals `(buyerAssets - sellerAssets) * 0.01` — the invariant `tokenBalances >= claimableSettlementFee + withdrawableSum + collateralSum` is broken.
9. Call `claimSettlementFee` and observe it either reverts (DoS) or drains lender withdrawable balance to cover the shortfall. [10](#0-9) [11](#0-10)

```text
// Settlement fee receipts pending settlement: claimableSettlementFee is incremented in take before
// the inbound fee transfer happens, so we track the gap and clear it in CVL_transferFrom.
persistent ghost mapping(address => mathint) pendingFeeReceipt {
    init_state axiom (forall address token. pendingFeeReceipt[token] == 0);
}

hook Sstore claimableSettlementFee[KEY address token] uint256 newVal (uint256 oldVal) {
    // Except for claimSettlementFee, the claimableSettlementFee is non-decreasing, see WithdrawableMonotonicity.spec.
    if (newVal > oldVal) {
        pendingFeeReceipt[token] = pendingFeeReceipt[token] + newVal - oldVal;
    }
}
```
