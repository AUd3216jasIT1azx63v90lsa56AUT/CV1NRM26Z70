Audit Report

## Title
Debt Increase Permitted at Exact Maturity Timestamp via Off-by-One in `take()` Guard - (File: `src/Midnight.sol`)

## Summary
The `take()` function guards debt increases with `block.timestamp <= offer.market.maturity` (line 391), which evaluates to `true` at the exact maturity boundary, allowing `sellerDebtIncrease > 0` to be committed on-chain. The post-maturity liquidation path in `liquidate()` uses the strict inequality `block.timestamp > market.maturity` (line 622), so the newly created debt cannot be post-maturity liquidated in the same block but becomes immediately liquidatable in the very next block regardless of collateral health. This violates the protocol invariant that debt must not increase at or after maturity.

## Finding Description

**Root cause — asymmetric boundary operators:**

`src/Midnight.sol` line 391:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
At `block.timestamp == maturity` the left operand is `true`, short-circuiting the `require` entirely. Any non-zero `sellerDebtIncrease` passes.

`src/Midnight.sol` line 359:
```solidity
uint256 timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp);
```
At equality `timeToMaturity = 0`, so `buyerPendingFeeIncrease` becomes 0 and the settlement fee uses the 0-duration breakpoint. Neither sub-computation reverts.

`src/Midnight.sol` line 476:
```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```
`isHealthy` (lines 944–960) computes only `maxDebt >= debt` against oracle prices; it has no maturity awareness. A well-collateralised borrower passes this check, and `sellerPos.debt += sellerDebtIncrease` is committed at line 414.

**Why the liquidation guard does not save it:**

`src/Midnight.sol` line 622:
```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
```
Post-maturity liquidation requires `block.timestamp > market.maturity` (strictly greater). At `block.timestamp == maturity` the condition is `false`, so the position cannot be post-maturity liquidated in the same block. In the very next block the strict inequality flips and the position is immediately post-maturity liquidatable regardless of collateral health.

**Exploit flow (two valid paths):**

*Path A — malicious taker harms maker/borrower:*
A sell offer (`offer.buy == false`) is live with `offer.expiry >= maturity`. The maker is the seller/borrower (`seller = offer.maker`, line 375). A malicious taker submits a take in the block where `block.timestamp == maturity` with `units > sellerPos.credit`, causing `sellerDebtIncrease > 0`. The maker's debt is increased on-chain. In the next block the maker's position is post-maturity liquidatable and can be seized by any liquidator.

*Path B — borrower self-harms or front-runs settlement:*
A buy offer (`offer.buy == true`) is live with `offer.expiry >= maturity`. The taker is the seller/borrower (`seller = taker`, line 375). The taker submits a take at `block.timestamp == maturity`, increasing their own debt. The position is immediately post-maturity liquidatable in the next block.

**Existing protections reviewed and found insufficient:**
- `block.timestamp <= offer.expiry` — passes since `expiry >= maturity`.
- `CannotIncreaseDebtPostMaturity` guard — uses `<=`, not `<`; passes at equality.
- `SellerIsLiquidatable` guard — health-only, not maturity-aware.
- `IRatifier.isRatified` — validates the Merkle proof against the offer struct; has no maturity-time restriction.

## Impact Explanation
A borrower's debt is increased at the exact maturity timestamp, a state the protocol explicitly forbids. In the immediately following block the position is post-maturity liquidatable (via `postMaturityMode = true`) regardless of collateral health, exposing the borrower to forced liquidation with a growing LIF ramp on debt that should never have been created. In Path A the harm is inflicted on the maker by an unprivileged external taker, constituting unauthorized state change and potential asset loss through forced liquidation of a previously healthy position.

## Likelihood Explanation
Block timestamps on proof-of-stake Ethereum are deterministic slot values, making the exact maturity block reliably identifiable. Any unprivileged actor monitoring the chain can submit the take transaction targeting that block. The preconditions — a live offer with `expiry >= maturity` and a sufficiently collateralised borrower — are normal operating state for any active market. The attack is repeatable on every market at its maturity block and requires no privileged access, leaked keys, or victim mistakes.

## Recommendation
Change the debt-increase guard from non-strict to strict inequality so that the maturity boundary is excluded:

```solidity
// Before (line 391):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After:
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
This aligns the `take()` guard with the strict `>` used in `liquidate()`, closing the one-block window where debt can be created but not yet post-maturity liquidated.

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];
                uint256 price = IOracle(collateralParam.oracle).price();
                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                    .mulDivDown(collateralParam.lltv, WAD);
                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
    }
