Audit Report

## Title
Maker-controlled sell callback can transiently manipulate oracle price, causing `isHealthy` to return false and reverting valid taker transactions - (File: src/Midnight.sol)

## Summary
When a taker fills a sell offer (`offer.buy == false`), the maker's `offer.callback` is assigned as `sellerCallback` and invoked before the post-callback health check. Because the `LIQUIDATION_LOCK_SLOT` is reset to `false` before `isHealthy` is evaluated, a malicious maker callback that transiently manipulates an on-chain oracle causes `isHealthy(offer.market, id, seller)` to return `false` for a genuinely healthy seller, reverting the taker's transaction with `SellerIsLiquidatable`. Since the entire transaction reverts, the oracle manipulation also reverts, making the attack repeatable at gas cost only.

## Finding Description

**Callback assignment** (`src/Midnight.sol`, line 421):
```solidity
address sellerCallback = offer.buy ? takerCallback : offer.callback;
```
When `offer.buy == false`, `sellerCallback = offer.callback` — fully controlled by the maker at offer-creation time.

**Seller identity** (`src/Midnight.sol`, line 375):
```solidity
(address buyer, address seller) = offer.buy ? (offer.maker, taker) : (taker, offer.maker);
```
When `offer.buy == false`, `seller = offer.maker`. The post-callback health check at line 476 is therefore on the maker's own position.

**Callback invocation** (`src/Midnight.sol`, lines 458–474):
`ISellCallback(sellerCallback).onSell(...)` is called with no restrictions on external state mutations. The callback can interact with any external contract, including a DEX pool whose reserves an oracle reads.

**Lock reset and health check** (`src/Midnight.sol`, lines 475–476):
```solidity
if (!wasLocked) UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, false);
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```
In the normal case (`wasLocked == false`), the lock is reset to `false` before the `require`. Therefore `liquidationLocked(id, seller)` evaluates to `false`, and `isHealthy` must return `true` for the transaction to succeed. If the callback has shifted the oracle price, `isHealthy` reads the manipulated value and returns `false`.

**`isHealthy` oracle read** (`src/Midnight.sol`, lines 953–955):
```solidity
uint256 price = IOracle(collateralParam.oracle).price();
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```
The oracle is called live with no snapshot or TWAP enforcement at the protocol level.

**Exploit flow:**
1. Maker deploys `MaliciousCallback` implementing `ISellCallback.onSell` that swaps tokens in the oracle's underlying pool (using a flash loan), dropping the reported collateral price below the health threshold.
2. Maker creates a sell offer with `offer.callback = address(MaliciousCallback)`.
3. Taker calls `take()`. State is updated (seller debt increases), lock is set to `true` (line 444).
4. `MaliciousCallback.onSell` executes the pool swap; oracle price drops below the health threshold.
5. Lock is reset to `false` (line 475).
6. `isHealthy(offer.market, id, seller)` reads the manipulated price → returns `false`.
7. `require(false || false, SellerIsLiquidatable())` → transaction reverts.
8. The revert unwinds the swap and the flash loan repayment — the attack costs only gas.

**Why existing protections are insufficient:**
The `LIQUIDATION_LOCK_SLOT` mechanism (lines 444, 475) is designed solely to prevent reentrant liquidation calls during callbacks, as confirmed by the comment at line 335: *"The seller cannot be liquidated during the callbacks of a take."* It does not protect the post-callback `isHealthy` evaluation from reading oracle state that was mutated during the callback. The lock is explicitly cleared before `isHealthy` is called, leaving the health check fully exposed to any oracle state changes made during `onSell`.

## Impact Explanation
Any taker attempting to fill a malicious maker's sell offer will have their transaction revert with `SellerIsLiquidatable` even though the seller's position was healthy at the pre-callback oracle price. The maker's offer is never consumed (all state changes revert), so the attack is repeatable at zero net cost to the maker (gas only, since the oracle manipulation also reverts). This permanently blocks market activity on any offer the maker chooses to poison and can be used to grief specific takers or deny liquidity to the entire market. The impact is a permanent, targeted denial-of-service on taker transactions — a concrete service unavailability impact. `SECURITY.md` explicitly states oracle manipulation/flash-loan attacks are not excluded.

## Likelihood Explanation
**Preconditions:**
- Market uses an oracle whose `price()` reads mutable on-chain state (e.g., Uniswap v2/v3 spot price, any reserve-based oracle). TWAP oracles with short windows are also susceptible.
- Maker's position must be healthy before the callback (so the offer passes normal validation) but close enough to the liquidation threshold that a feasible pool swap can push it below.
- Maker must be able to execute the swap within the callback — trivially achievable via flash loan, which also reverts with the transaction.

**Feasibility:** High. The maker controls `offer.callback` unconditionally; no privileged access is required. The attack is atomic (single transaction), repeatable, and costs only gas. Any market with a spot oracle is vulnerable. The maker can deliberately position themselves near the liquidation threshold to minimize the swap size required.

## Recommendation
1. **Snapshot the oracle price before callbacks:** Read and cache all oracle prices needed for the health check before invoking any callback. Use the cached values in the post-callback `isHealthy` evaluation rather than re-querying the oracle.
2. **Alternatively, move the health check before the seller callback:** Evaluate `isHealthy` before `onSell` is invoked (while the lock is still `true`), so the callback cannot influence the result.
3. **Restrict seller callback to taker-controlled address:** When `offer.buy == false`, require that `sellerCallback` is provided by the taker (i.e., `takerCallback`) rather than `offer.callback`, analogous to how `buyerCallback` is taker-controlled when `offer.buy == true`.

## Proof of Concept
**Minimal fork test plan:**
1. Deploy a mock spot oracle that reads reserves from a mock Uniswap V2 pool.
2. Create a market using this oracle with LLTV = 0.8.
3. Maker supplies collateral and borrows to reach 79% LTV (healthy but near threshold).
4. Deploy `MaliciousCallback` implementing `ISellCallback`: in `onSell`, flash-borrow the collateral token and dump it into the pool, dropping the oracle price by ~2%.
5. Maker creates a sell offer with `offer.callback = address(MaliciousCallback)`.
6. Taker calls `take()` on the offer.
7. **Expected result:** Transaction reverts with `SellerIsLiquidatable` despite the maker's position being healthy at the pre-callback price.
8. **Confirm:** After the revert, call `isHealthy` directly — it returns `true` (oracle price is back to normal since the swap reverted).
9. Repeat step 6 — same revert, confirming the attack is indefinitely repeatable at gas cost only.
