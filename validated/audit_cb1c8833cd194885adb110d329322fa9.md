Audit Report

## Title
`maxAssets` cap bypassed via zero-rounding `buyerAssets` on buy offers with `tick < MAX_TICK` — (`src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and `tick < MAX_TICK` (the entire practical tick range), calling `take()` with `units = 1` computes `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0` due to integer truncation. Because `consumed` is incremented by `buyerAssets`, it never advances, the `ConsumedAssets` guard always passes, and an unprivileged taker can call `take()` indefinitely — even after the offer is fully consumed — accumulating unbounded credit for the maker and unbounded debt for themselves with no asset transfer.

## Finding Description
**Root cause — `src/Midnight.sol` lines 358–373:**

For a buy offer, `buyerPrice = offerPrice = TickLib.tickToPrice(offer.tick)`. `tickToPrice` returns values in `(0, WAD]`; only `tick = MAX_TICK = 5820` yields exactly `WAD`. Every other valid tick yields `offerPrice < WAD` (prices are multiples of `PRICE_ROUNDING_STEP = 1e12`, so the maximum sub-WAD price is `WAD - 1e12`).

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // line 363 — (1 * buyerPrice) / 1e18 = 0
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group]
        += offer.buy ? buyerAssets : sellerAssets;  // line 368 — += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets()); // line 369 — always passes
}
```

With `units = 1` and `buyerPrice < 1e18`: `mulDivDown(1, buyerPrice, WAD) = (1 * buyerPrice) / 1e18 = 0`. `consumed` is incremented by zero, `newConsumed` is unchanged, and the cap check passes unconditionally — even when `consumed` was already pre-set to `maxAssets`.

Position state is still fully updated at lines 410 and 414:
```solidity
buyerPos.credit += toUint128(buyerCreditIncrease);  // lender gains 1 unit of credit
sellerPos.debt   += toUint128(sellerDebtIncrease);  // borrower gains 1 unit of debt
```

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — never triggers because `newConsumed` never increases.
- There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in `take()`.
- The protocol's own NatSpec at line 94 acknowledges the symptom: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*
- The existing test `testBugBuyMaxAssetsBypass` (line 858) explicitly reproduces the scenario, pre-fills `consumed` to `maxAssets`, calls `take(1, borrower, lenderOffer)`, and asserts that `creditOf(lender)` and `debtOf(borrower)` both increase while `consumed` stays frozen and no tokens move.

## Impact Explanation
The `maxAssets` cap is the maker's primary mechanism to bound their total lending exposure. Bypassing it allows an unprivileged taker to:
1. Inflate the maker's credit position to an arbitrary size without the maker spending any assets, increasing their susceptibility to bad-debt socialization losses beyond what they consented to.
2. Accumulate their own debt position of arbitrary size without paying any assets — usable to extract collateral or manipulate market state.
3. Violate the core accounting invariant that `consumed[maker][group]` must increase by at least the fill's contribution on every successful non-zero-unit take.

This constitutes unauthorized state mutation and a broken accounting invariant, both in-scope impact classes per `RESEARCHER.md`.

## Likelihood Explanation
The precondition is trivially satisfied: any buy offer with `maxAssets > 0` and any `tick` in `[0, MAX_TICK)` — the entire practical tick range — is vulnerable. No privileged access, oracle manipulation, or special token behavior is required. The attack is repeatable in a single transaction via `multicall`. Gas cost on cheap chains makes position inflation economically viable.

## Recommendation
Add a guard in `take()` that rejects a non-zero-unit take when the resulting asset contribution rounds to zero:

```solidity
if (offer.maxAssets > 0) {
    uint256 contribution = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || contribution > 0, ZeroAssetContribution());
    newConsumed = consumed[offer.maker][offer.group] += contribution;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum tick or minimum `units` such that the asset contribution is always non-zero when `maxAssets > 0`.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` at `test/TakeTest.sol` line 858 is a complete, self-contained PoC:

1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (price < WAD).
2. Pre-fill `consumed[lender][group] = maxAssets` via `setConsumed`.
3. Call `take(1, borrower, lenderOffer)`.
4. Assert: `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged at `maxAssets`, but `creditOf(lender)` and `debtOf(borrower)` both strictly increased.
