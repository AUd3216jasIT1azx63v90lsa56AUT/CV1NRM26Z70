Audit Report

## Title
`buyerAssetsToUnits` reverts outside try/catch for sell offers where `offerPrice + settlementFee > WAD`, breaking skip-on-revert guarantee - (`File: src/periphery/MidnightBundles.sol`)

## Summary
In `buyWithAssetsTargetAndWithdrawCollateral`, the call to `TakeAmountsLib.buyerAssetsToUnits` at line 209 sits outside the try/catch block that wraps `take()`. For sell offers where `tickToPrice(offer.tick) + settlementFee > WAD`, `buyerAssetsToUnits` reverts with `PriceGreaterThanOne` while `Midnight.take()` itself would succeed. This causes the entire bundler transaction to revert rather than skipping the offer, breaking the function's documented skip-on-revert guarantee and blocking collateral withdrawals and all other offers in the bundle.

## Finding Description

**Root cause — `TakeAmountsLib.buyerAssetsToUnits`:**

In `src/periphery/TakeAmountsLib.sol` lines 26–28, for a sell offer (`offer.buy == false`):
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + settlementFee;   // = offerPrice + settlementFee
require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());  // ← REVERTS
```
When `offerPrice + settlementFee > WAD`, this unconditionally reverts.

**`Midnight.take()` has no equivalent guard:**

In `src/Midnight.sol` lines 361–363, the identical arithmetic is performed for a sell offer:
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
```
There is no `require(buyerPrice <= WAD)`. The NatSpec at lines 333–334 explicitly acknowledges this asymmetry: *"All sellerAssets are reachable with the units input, and all buyerAssets are reachable only if buyerPrice <= WAD."* `take()` is intentionally permissive for sell offers with `buyerPrice > WAD`.

**`buyerAssetsToUnits` is called outside the try/catch:**

In `src/periphery/MidnightBundles.sol` lines 208–221:
```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.buyerAssetsToUnits(   // ← NOT in try/catch; revert propagates
        MIDNIGHT, id, takes[i].offer, targetFilledBuyerAssets - filledBuyerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) {      // ← only this is caught
    ...
} catch {}
```
The NatSpec at line 174 states *"Skips every reason why take can revert"* while line 175 states *"Reverts if TakeAmountsLib or ConsumableUnitsLib reverts."* These are contradictory for sell offers where `buyerPrice > WAD`: `take()` would not revert, but `buyerAssetsToUnits` does, and the revert propagates uncaught.

**Test suite confirms the precondition is known:** Every test exercising `buyWithAssetsTargetAndWithdrawCollateral` at `MAX_TICK` explicitly zeroes out all settlement fees (e.g., `test/MidnightBundlesTest.sol` lines 201–203, 260–262, 296–298), confirming that non-zero settlement fees at high ticks are a known problematic combination.

**Exploit flow:**
1. Maker creates a sell offer with `offer.tick` such that `tickToPrice(tick)` is near WAD (e.g., `tick = MAX_TICK`).
2. Market has any non-zero settlement fee, so `offerPrice + settlementFee > WAD`.
3. Taker calls `buyWithAssetsTargetAndWithdrawCollateral` with this offer in the `takes` array.
4. At line 209, `buyerAssetsToUnits` reverts with `PriceGreaterThanOne`.
5. The revert propagates — the entire transaction reverts, including collateral withdrawals and all other offers in the bundle.
6. Calling `Midnight.take()` directly on the same offer succeeds.

## Impact Explanation
`buyWithAssetsTargetAndWithdrawCollateral` reverts entirely instead of skipping the problematic offer. This prevents the taker from completing any collateral withdrawals and all other takes in the bundle. A malicious maker can deliberately place a sell offer at a high-price tick in any market with a non-zero settlement fee to reliably grief any taker who includes that offer in a bundle. The impact is a denial-of-service on the bundler's core skip-on-revert functionality.

## Likelihood Explanation
**Preconditions:** A sell offer with `tickToPrice(offer.tick) + settlementFee > WAD` exists and is included in a `buyWithAssetsTargetAndWithdrawCollateral` call. Any maker can set `offer.tick = MAX_TICK`. Any non-zero settlement fee (the normal operating state of the protocol) satisfies the second condition. The condition can also arise retroactively if the settlement fee is increased after offer creation. No special privilege is required. The trigger is deterministic and repeatable.

## Recommendation
Wrap the entire `unitsToTake` computation (including `TakeAmountsLib.buyerAssetsToUnits` and `ConsumableUnitsLib.consumableUnits`) inside the try/catch block, or add a pre-flight guard that skips the offer when `buyerPrice > WAD` before invoking `buyerAssetsToUnits`. The simplest fix is to check `tickToPrice(takes[i].offer.tick) + settlementFee(id, ...) <= WAD` before calling `buyerAssetsToUnits`, and `continue` if the condition fails, consistent with the skip-on-revert contract.

## Proof of Concept
1. Deploy Midnight with a market where `settlementFee > 0` for the relevant maturity bucket.
2. Maker creates a sell offer with `tick = MAX_TICK` such that `tickToPrice(MAX_TICK) + settlementFee > WAD`.
3. Taker calls `buyWithAssetsTargetAndWithdrawCollateral` with this offer in `takes`.
4. Observe the transaction reverts with `PriceGreaterThanOne` (from `TakeAmountsLib`).
5. Call `Midnight.take()` directly with the same offer and parameters — observe it succeeds.
6. The existing test pattern at `test/MidnightBundlesTest.sol` lines 201–203 already demonstrates the workaround (zeroing fees), confirming the precondition is reachable when fees are non-zero.
