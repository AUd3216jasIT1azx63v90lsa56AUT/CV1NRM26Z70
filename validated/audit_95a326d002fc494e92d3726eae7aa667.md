### Title
Continuous Fee Precision Loss in `updatePositionView` Allows Lenders to Avoid Paying Fees via Permissionless `updatePosition` Calls — (File: `src/Midnight.sol`)

---

### Summary

The `updatePositionView` function computes the accrued continuous fee using integer division (`mulDivDown`), which truncates to zero when `postSlashPendingFee * dt < (maturity - lastAccrual)`. Critically, `_updatePosition` always advances `lastAccrual` to `block.timestamp` regardless of whether any fee was accrued. Because `updatePosition` is a permissionless external function, a lender can call it repeatedly with small time intervals to consume the entire market duration without accruing any continuous fee, causing the protocol to lose all continuous fee revenue from that position.

---

### Finding Description

**Root cause — fee formula truncates to zero, but timestamp always advances:**

In `updatePositionView`, the accrued fee is:

```solidity
uint128 fee = _lastAccrual < market.maturity
    ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
    : 0;
``` [1](#0-0) 

This computes `fee = floor(pendingFee * dt / (maturity - lastAccrual))`. When `pendingFee * dt < (maturity - lastAccrual)`, integer division truncates to **zero**.

In `_updatePosition`, `lastAccrual` is unconditionally advanced to `block.timestamp` regardless of whether `fee > 0`:

```solidity
_position.lastAccrual = uint128(block.timestamp);
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
``` [2](#0-1) 

This is the exact same bug class as the MultiRewards report: the accumulator (`lastAccrual`) advances while the accrued value (`fee`) stays zero, permanently discarding the time interval.

**Permissionless trigger:**

`updatePosition` is a public, unrestricted external function callable for any user:

```solidity
function updatePosition(Market memory market, address user) external returns (uint128, uint128, uint128) {
    bytes32 id = toId(market);
    require(marketState[id].tickSpacing > 0, MarketNotCreated());
    return _updatePosition(market, id, user);
}
``` [3](#0-2) 

**Condition for zero fee per call:**

`fee = 0` when:
```
pendingFee * dt < (maturity - lastAccrual)
```

Substituting `pendingFee = credit * continuousFee * timeToMaturity / WAD` and `timeToMaturity ≈ (maturity - lastAccrual)` at position creation:

```
credit * continuousFee * dt / WAD < 1
=> credit * continuousFee < WAD   (for dt = 1 second)
``` [4](#0-3) 

**`MAX_CONTINUOUS_FEE` value:**

```solidity
uint32 constant MAX_CONTINUOUS_FEE = uint32(uint256(0.01e18) / uint256(365 days));
// ≈ 317,097,919
``` [5](#0-4) 

At max fee rate, the zero-fee threshold is:
```
credit < WAD / MAX_CONTINUOUS_FEE = 1e18 / 3.17e8 ≈ 3.15e9
```

- **18-decimal tokens (ETH):** threshold ≈ 3.15e9 wei ≈ 0.000000003 ETH (dust-level, low impact)
- **6-decimal tokens (USDC):** threshold ≈ 3.15e9 / 1e6 = **3,150 USDC** (realistic position size)

At lower fee rates (e.g., `continuousFee = 1e6`), the threshold rises to `1e18 / 1e6 = 1e12`, meaning USDC positions up to **1,000,000 USDC** can achieve zero fee accrual per second.

**Attack flow:**

1. Lender opens a position with `credit` below the threshold for the given `continuousFee`.
2. Lender calls `updatePosition(market, self)` every block (or every second on L2s).
3. Each call: `fee = 0`, but `lastAccrual` advances by `dt`.
4. After the full market duration, `lastAccrual == maturity`, all time consumed, total fee accrued = 0.
5. Protocol's `continuousFeeCredit` never increases; lender retains full credit.

Even for positions above the zero-fee threshold, frequent calls amplify the general precision loss: each call discards `pendingFee * dt % (maturity - lastAccrual)` units of fee. For a 1M USDC position called every second over 1 year, the maximum cumulative loss is up to 31,536,000 USDC units ≈ **31.5 USDC** of fee revenue.

---

### Impact Explanation

**Medium.** The protocol loses continuous fee revenue. For USDC markets with positions below ~3,150 USDC (at max fee rate) or up to ~1M USDC (at lower fee rates), a lender can reduce their continuous fee to zero by calling `updatePosition` every block. On L2s (Arbitrum, Base), gas costs are negligible, making this economically rational for any lender whose avoided fee exceeds gas cost. Additionally, every invocation of `updatePosition` (including those triggered internally by `take`) causes a small irrecoverable precision loss that accumulates over the market lifetime.

---

### Likelihood Explanation

**Medium.** The attack requires no privileged access — only a valid lender position and repeated calls to a public function. On L2 networks where Morpho typically deploys, per-call gas cost is low enough that avoiding even a few USDC of fees is profitable. The condition is automatically satisfied for small positions or markets with lower-than-maximum continuous fee rates, which are common in practice.

---

### Recommendation

Store the residual `pendingFee * dt % (maturity - lastAccrual)` and carry it forward to the next accrual period, analogous to the MultiRewards recommendation. Concretely, add a `pendingFeeResidue` field to `Position` and re-add it to the numerator on the next `updatePositionView` call:

```solidity
uint256 numerator = postSlashPendingFee * (accrualEnd - _lastAccrual) + _position.pendingFeeResidue;
uint256 denominator = market.maturity - _lastAccrual;
uint128 fee = uint128(numerator / denominator);
// store remainder
_position.pendingFeeResidue = uint128(numerator % denominator);
```

This ensures no time interval is silently discarded and eliminates the griefing vector entirely.

### Proof of Concept

**Setup:** USDC market, `continuousFee = MAX_CONTINUOUS_FEE ≈ 3.17e8`, 1-year maturity, lender credit = 3,000 USDC (3e9 units, below threshold).

**Verify zero-fee condition:**
```
pendingFee = 3e9 * 317097919 * 31536000 / 1e18
           = 3e9 * 317097919 * 31536000 / 1e18
           ≈ 3e9 * 1e10 / 1e18
           = 3e9 * 1e-8
           = 30
```

`fee per second = floor(30 * 1 / 31536000) = 0` ✓

    function updatePositionView(Market memory market, bytes32 id, address user)
        public
        view
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        uint128 credit = _position.credit;
        uint128 _lastLossFactor = _position.lastLossFactor;
        uint256 postSlashCredit = _lastLossFactor < type(uint128).max
            ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
            : 0;
        uint128 _pendingFee = _position.pendingFee;
        uint256 postSlashPendingFee =
            credit > 0 ? _pendingFee - _pendingFee.mulDivUp(credit - postSlashCredit, credit) : 0;
        uint256 accrualEnd = UtilsLib.min(block.timestamp, market.maturity);
        uint128 _lastAccrual = _position.lastAccrual;
        // forge-lint: disable-next-item(unsafe-typecast) as fee <= pending <= credit which are uint128 position fields
        uint128 fee = _lastAccrual < market.maturity
            ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
            : 0;
        // forge-lint: disable-next-item(unsafe-typecast) as credit and pending are <= uint128 position fields
        return (uint128(postSlashCredit) - fee, uint128(postSlashPendingFee) - fee, fee);
    }

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
