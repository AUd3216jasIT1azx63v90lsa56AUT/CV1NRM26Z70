Audit Report

## Title
Buy offer `maxAssets` cap bypass via `mulDivDown` rounding to zero enables unbounded unbacked credit/debt creation - (`src/Midnight.sol`)

## Summary
When a buy offer uses asset-mode capping (`maxAssets > 0`) and `offerPrice < WAD` (any tick below `MAX_TICK`), the consumed-accounting increment computes `buyerAssets = units.mulDivDown(buyerPrice, WAD)`, which truncates to 0 for `units = 1`. An attacker can call `take()` with `units = 1` after `consumed == maxAssets`, bypassing the cap check while still mutating buyer credit, seller debt, `totalUnits`, and triggering the maker's `onBuy` callback — all with zero asset transfer. The attack is repeatable indefinitely at zero cost.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

`buyerAssets` is computed with truncating division: [1](#0-0) 

`mulDivDown` is `(x * y) / d` with integer truncation: [2](#0-1) 

When `buyerPrice < WAD` (which holds for any `offerPrice < WAD`, i.e., any tick below `MAX_TICK`), `mulDivDown(1, buyerPrice, WAD) = 0`.

The consumed increment then adds 0: [3](#0-2) 

If `consumed[maker][group]` is already equal to `maxAssets`, `newConsumed = maxAssets + 0 = maxAssets`, which satisfies `require(newConsumed <= offer.maxAssets)`. The function does not revert.

Execution continues to mutate position state using the full `units = 1`: [4](#0-3) 

The maker's `onBuy` callback is invoked with `buyerAssets = 0` and `units = 1`: [5](#0-4) 

**Why existing checks fail:**

The `require(newConsumed <= offer.maxAssets)` at line 369 is the only guard against over-consumption. It passes because `newConsumed = N + 0 = N`. There is no check that `units == 0` when `buyerAssets == 0`, and no check that `consumed == maxAssets` must cause a revert.

The Certora spec rule `takeConsumedAtMaxUnchangedAssets` only asserts that `consumed` does not increase past `maxAssets` — it does not assert that `units` must be zero or that position state is unchanged when the offer is exhausted: [6](#0-5) 

The analogous strong rule (`fullyConsumedOfferRevertsOnNonTrivialTake`) exists only for the `maxAssets == 0` (units-mode) path: [7](#0-6) 

Critically, the Certora spec summaries use `NONDET` for both `mulDivDown` and `mulDivUp`, meaning the formal verification never models the rounding-to-zero behavior that enables this bypass: [8](#0-7) 

## Impact Explanation

After `consumed == maxAssets`, each `take(units=1)` call:
- Returns `buyerAssets = 0`, `sellerAssets = 0` — no loan tokens move
- Leaves `consumed` unchanged at `maxAssets`
- Increases `creditOf(buyer)`, `debtOf(seller)`, and `totalUnits` by 1 unit each
- Triggers the maker's `onBuy` callback with `buyerAssets = 0, units = 1`

This violates the core accounting invariant that credit and debt are backed by asset transfers. An unprivileged attacker can create unbounded unbacked credit/debt positions and spam the maker's callback at zero asset cost, repeatable indefinitely. The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` directly confirms this behavior.

## Likelihood Explanation

All preconditions are reachable by an unprivileged taker:
1. `offer.buy = true` — standard buy offer, no privilege required.
2. `offerPrice < WAD` — any tick below `MAX_TICK`; a normal, common configuration.
3. `consumed[maker][group] == maxAssets` — reachable by a prior legitimate fill or by the maker calling `setConsumed`.
4. `units = 1` — attacker-controlled, no constraint.

Since `consumed` never increases past `maxAssets` during the attack, the exploit is repeatable on every subsequent call with no additional setup.

## Recommendation

Add an explicit guard that reverts when `buyerAssets == 0` and `units > 0` in assets mode, or equivalently, require that `units == 0` whenever the computed asset amount rounds to zero. A minimal fix at line 363–369:

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
+   require(buyerAssets > 0 || units == 0, ZeroAssetTake()); // prevent rounding bypass
}
```

Alternatively, mirror the `fullyConsumedOfferRevertsOnNonTrivialTake` Certora rule for the assets-mode path, and remove the `NONDET` summaries for `mulDivDown`/`mulDivUp` in `Consume.spec` so the formal verification captures rounding behavior.

## Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` is the direct PoC. Manual reproduction steps:

1. Create a buy offer with `maxAssets = N > 0` and `tick = MAX_TICK - 16` (so `tickToPrice(tick) < WAD`).
2. Fill the offer legitimately until `consumed[maker][group] == N`.
3. Call `take(units=1)` as an unprivileged attacker.
4. Observe: call succeeds, `consumed` remains `N`, but `creditOf(buyer) += 1`, `debtOf(seller) += 1`, `totalUnits += 1`, and the maker's `onBuy` callback fires with `buyerAssets=0, units=1`.
5. Repeat step 3 indefinitely — each call succeeds and further inflates unbacked credit/debt.

rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
