Audit Report

## Title
ECDSA Signature Malleability Enables Nonce-Griefing Front-Run on `setIsAuthorized` - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` calls raw `ecrecover` at line 31 without enforcing a low-s constraint. Because secp256k1 ECDSA produces two valid `(v, r, s)` pairs per `(message, key)`, any mempool observer can derive a second valid signature for the same authorization and front-run the victim's transaction, consuming the nonce. The victim's original transaction then reverts with `InvalidNonce`. If the victim submitted the call inside an atomic multicall bundle (e.g., authorize-then-invoke a protective action), the entire bundle fails and must be resubmitted, introducing a timing gap exploitable in liquidation-sensitive contexts.

## Finding Description
**Exact code path — `EcrecoverAuthorizer.setIsAuthorized` (`src/periphery/EcrecoverAuthorizer.sol`, lines 24–48):**

- Line 26: `require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());` — nonce is checked and incremented atomically; the first caller to reach this line wins.
- Line 31: `address signer = ecrecover(digest, signature.v, signature.r, signature.s);` — raw `ecrecover` with no low-s guard.
- No `msg.sender` restriction anywhere in the function; any address may call it with any valid-looking signature. This is confirmed by the test `testEcrecoverAuthorizerPermissionless` in `test/SetIsAuthorizedWithSigTest.sol` (lines 74–86), which explicitly documents "Anyone can submit — no caller auth needed."

**Root cause:** secp256k1 ECDSA produces two valid `(v, r, s)` pairs per `(message, key)`: the canonical `(v, r, s)` and the malleable `(55−v, r, n−s)` where `n` is the curve order (`0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141`). Both pass `ecrecover` and return the identical signer address. Without `require(uint256(s) <= 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0)`, both are accepted.

**Exploit flow:**
1. Victim signs `Authorization{authorizer=V, authorized=X, isAuthorized=true, nonce=N, deadline=D}` and broadcasts `setIsAuthorized(auth, sig)` with `sig = (v, r, s)`.
2. Attacker observes the pending transaction, extracts `(v, r, s)`, computes `s' = n − s`, `v' = 55 − v` (flipping between 27 and 28).
3. Attacker submits `setIsAuthorized(auth, Signature{v', r, s'})` with higher gas.
4. Attacker's transaction executes first: `nonce[V] == N` passes, nonce becomes `N+1`, `ecrecover` returns `V`, authorization is set to the victim's intended value.
5. Victim's transaction executes: `nonce[V] == N+1` but `auth.nonce == N` → `InvalidNonce()` revert.

**Why existing checks fail:**
- `require(signer != address(0))` — both signatures recover the same non-zero address; passes.
- `require(signer == authorization.authorizer || isAuthorized(...))` — same signer recovered; passes.
- The nonce is already consumed before the victim's transaction executes.

## Impact Explanation
The concrete impact is a forced revert of any atomic bundle that includes `setIsAuthorized` as its first step. In a liquidation-protection scenario where a user bundles `EcrecoverAuthorizer.setIsAuthorized` + a `MidnightBundles` protective call (repay debt, withdraw collateral) into a single multicall transaction, the attacker's front-run causes the entire bundle to fail. The victim must detect the failure, resubmit (now without the `setIsAuthorized` step, since authorization is already set), and race against the liquidation deadline. Even a single-block delay can be sufficient for a liquidator to execute. The attacker pays only gas; the victim bears the liquidation loss. The attack is non-repeatable for the same authorization intent (the state is already set correctly after the front-run), but the one-time bundle disruption is sufficient to cause financial harm.

## Likelihood Explanation
**Preconditions:** Victim broadcasts a `setIsAuthorized` transaction (routine operation). Attacker monitors the public mempool (standard, zero-cost capability on Ethereum mainnet and most L2s with public mempools). No special role, funds, or credentials required beyond gas.

**Feasibility:** Computing `s' = n − s` and `v' = 55 − v` is trivial off-chain arithmetic, executable in milliseconds. The attack window is the standard mempool propagation window. The attacker pays gas but requires no upfront capital. The permissionless nature of `setIsAuthorized` (confirmed by the codebase's own test) is the enabling condition.

## Recommendation
Add a low-s constraint immediately after computing the digest, before calling `ecrecover`:

```solidity
require(
    uint256(signature.s) <= 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0,
    InvalidSignature()
);
address signer = ecrecover(digest, signature.v, signature.r, signature.s);
```

This is the same guard used by OpenZeppelin's `ECDSA.recover` and enforced by EIP-2 for transaction signatures. It eliminates the malleable counterpart by restricting `s` to the lower half of the curve order, making each `(message, key)` pair correspond to exactly one accepted signature.

## Proof of Concept
**Minimal Foundry test plan:**

```solidity
function testMalleabilityFrontRun() public {
    // Setup: borrower authorizes ecrecoverAuthorizer on Midnight
    vm.prank(borrower);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

    // Victim signs authorization
    Authorization memory auth = makeAuthorization(borrower, lender, true);
    Signature memory sig = signAuthorization(auth, borrower);

    // Attacker computes malleable counterpart
    uint256 n = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141;
    Signature memory malSig = Signature({
        v: sig.v == 27 ? 28 : 27,
        r: sig.r,
        s: bytes32(n - uint256(sig.s))
    });

    // Attacker front-runs with malleable signature
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(auth, malSig);

    // Authorization is set correctly (attacker's tx succeeded)
    assertEq(midnight.isAuthorized(borrower, lender), true);
    assertEq(ecrecoverAuthorizer.nonce(borrower), 1);

    // Victim's original transaction now reverts
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);
}
```

**Multicall bundle disruption:** Wrap the victim's `setIsAuthorized` + `MidnightBundles.repayAndWithdrawCollateral` in a Multicall3 `aggregate` call.     function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
```text
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );
```
