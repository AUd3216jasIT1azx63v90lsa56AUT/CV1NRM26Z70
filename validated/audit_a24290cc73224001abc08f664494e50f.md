Audit Report

## Title
`onLiquidate` callback enables post-bad-debt `supplyCollateral`, causing permanent lender over-slashing - (File: src/Midnight.sol)

## Summary
In `liquidate()`, bad debt is fully computed from a local bitmap snapshot and immediately committed to `lossFactor`/`totalUnits` before the `onLiquidate` callback fires. A callback authorized for the borrower can call `supplyCollateral` during this window, injecting collateral that was never counted in the bad-debt formula. Lenders are permanently slashed for a shortfall larger than the true post-liquidation shortfall, with no mechanism to reverse the over-accounting.

## Finding Description
**Exact code path in `src/Midnight.sol`:**

**Step 1 — Bitmap snapshot (line 606):**
```solidity
uint128 _collateralBitmap = _position.collateralBitmap;
```
The bad-debt loop (lines 607–618) iterates over this local copy. Any collateral index not set in this snapshot at the moment of the snapshot is invisible to the loop.

**Step 2 — Bad debt committed before any external call (lines 626–641):**
```solidity
_position.debt -= uint128(badDebt);
_marketState.lossFactor = ...;
_marketState.totalUnits -= UtilsLib.toUint128(badDebt);
_marketState.continuousFeeCredit = ...;
```
All lender-slashing state is finalized here. No external call has occurred yet.

**Step 3 — `onLiquidate` callback fires (lines 698–715):**
```solidity
if (callback != address(0)) {
    require(
        ILiquidateCallback(callback).onLiquidate(...) == CALLBACK_SUCCESS, ...
    );
}
```
At this point, `lossFactor` and `totalUnits` are already written. The callback executes with full re-entry capability into Midnight.

**Step 4 — `supplyCollateral` called inside callback (lines 524–546):**
```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
...
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
_position.collateralBitmap = newCollateralBitmap;
```
Authorization is satisfied if `callback == borrower` (self-liquidation, borrower is a contract) or if the borrower pre-authorized the callback via `setIsAuthorized`. The new collateral and its bitmap bit are written to storage, but the bad-debt loop already completed using the old snapshot — these writes have no retroactive effect on the committed `badDebt`.

**Step 5 — No re-check after callback (line 717):**
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```
With `repaidUnits == 0`, zero tokens are pulled. The function returns without re-evaluating collateral or bad debt.

**Why existing guards fail:**

- `liquidationLocked` is set only in `take()` at line 444 (`UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`). It is never set inside `liquidate()`. The check at line 621 (`!liquidationLocked(id, borrower)`) only guards against liquidating a borrower who is mid-`take()` callback — it does not block `supplyCollateral` from being called during `onLiquidate`.
- There is no reentrancy guard on `liquidate()` or `supplyCollateral()`.
- The `atMostOneNonZero(repaidUnits, seizedAssets)` check (line 595) permits both to be zero, enabling a pure bad-debt realization with no token transfer required.

## Impact Explanation
The over-slashing amount per attack is `A · price_new / maxLif_new` units, where `A` is the collateral amount supplied in the callback. This quantity is subtracted from `totalUnits` and added to `lossFactor` in excess of the true shortfall. Because `lossFactor` is used in `updatePositionView` to proportionally reduce every lender's credit:
```solidity
uint256 postSlashCredit = credit.mulDivDown(
    type(uint128).max - marketState[id].lossFactor,
    type(uint128).max - _lastLossFactor
);
```
every lender who calls `updatePosition` after the attack permanently loses additional credit proportional to the inflated `lossFactor`. The loss is irreversible — there is no mechanism to reduce `lossFactor` or restore `totalUnits`. This constitutes direct, permanent theft of lender value, an in-scope critical impact.

## Likelihood Explanation
**Required preconditions:**
1. The borrower position has genuine bad debt (oracle price drop below `1/maxLif` coverage) — the normal trigger for bad-debt realization.
2. The borrower is a smart contract implementing `ILiquidateCallback`, or has pre-authorized a callback contract via `setIsAuthorized`. Protocol-integrated borrowers (vaults, aggregators) routinely satisfy this.
3. The callback contract holds collateral tokens to supply — obtainable via flash loan within the same callback since `supplyCollateral` pulls from `msg.sender` and no repayment is required within `liquidate()`.

Self-liquidation (borrower == liquidator == callback) requires zero third-party cooperation. The attack is repeatable on any market with bad debt and a callback-capable borrower, and can be executed atomically in a single transaction.

## Recommendation
Set a transient-storage liquidation lock for the borrower at the start of `liquidate()` (analogous to the lock set for the seller in `take()` at line 444), and clear it after the callback. This prevents `supplyCollateral` (and any other position-mutating function) from being called for the borrower during the `onLiquidate` callback window. Alternatively, move the bad-debt computation and commitment to after the callback, so any collateral supplied during the callback is included in the formula before `lossFactor` and `totalUnits` are updated.

## Proof of Concept
**Minimal attack sequence (single transaction):**

1. Deploy `AttackContract` implementing `ILiquidateCallback`. Pre-fund it with collateral tokens for index `N` (not in borrower's current bitmap). Pre-authorize it: `midnight.setIsAuthorized(attackContract, true, borrower)` (or make `borrower == attackContract`).
2. Wait for borrower's position to have bad debt: `debt > Σ(collateral_i · price_i / maxLif_i)`.
3. Call `midnight.liquidate(market, anyIndex, 0, 0, borrower, false, receiver, attackContract, data)`.
4. Inside `AttackContract.onLiquidate`: call `midnight.supplyCollateral(market, N, largeAmount, borrower)`. The authorization check passes (`msg.sender == borrower` or via `isAuthorized`). `_position.collateral[N]` and `_position.collateralBitmap` are updated, but `lossFactor` and `totalUnits` were already committed with the inflated `badDebt`.
5. `liquidate()` returns. Lenders have been slashed for `badDebt` that included the full pre-callback shortfall, while the borrower's position now holds `largeAmount` of collateral index `N` that was never counted.
6. Any lender calling `updatePosition` now receives reduced credit proportional to the inflated `lossFactor`.

**Verification:** Assert `marketState[id].lossFactor` after the attack equals the value it would have if `largeAmount · price_N / maxLif_N` had been subtracted from `badDebt` before commitment — the difference is the over-slash.
