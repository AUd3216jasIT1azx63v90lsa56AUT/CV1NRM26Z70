Audit Report

## Title
Taker-Controlled `takerCallback` Sets `payer` to Arbitrary Third Party on Sell Offers - (File: `src/Midnight.sol`)

## Summary
In `Midnight.take`, when processing a sell offer (`offer.buy = false`), `takerCallback` is accepted without any constraint and is used directly as both `buyerCallback` and `payer`. An attacker can supply any address as `takerCallback`, causing Midnight to invoke `onBuy` on that address and then pull loan tokens from it. Any contract that implements `IBuyCallback`, returns `CALLBACK_SUCCESS` from `onBuy` without verifying `buyer == address(this)`, and holds a token balance (or approves inside `onBuy`) can have its tokens drained by an unprivileged caller.

## Finding Description

**Root cause — no authorization constraint on `takerCallback`:**

The only taker-side check in `take` is:

```solidity
// src/Midnight.sol:346
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());
```

This guards who may act as `taker`; it places no constraint on `takerCallback`.

**Payer derivation for sell offers:**

```solidity
// src/Midnight.sol:420-422
address buyerCallback = offer.buy ? offer.callback : takerCallback;   // sell: = takerCallback
address sellerCallback = offer.buy ? takerCallback : offer.callback;
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
// sell + non-zero takerCallback → payer = takerCallback (attacker-controlled)
```

**`onBuy` is invoked on the attacker-chosen address:**

```solidity
// src/Midnight.sol:445-452
if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback).onBuy(id, offer.market, buyerAssets, units, buyerPendingFeeIncrease, buyer, buyerCallbackData)
            == CALLBACK_SUCCESS,
        WrongBuyCallbackReturnValue()
    );
}
```

**Transfers then pull from `payer = victimContract`:**

```solidity
// src/Midnight.sol:455-456
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

**`buyer` argument for sell offers:** Line 375 sets `buyer = taker` (the attacker's address), not the callback contract's address. A callback that checks `buyer == address(this)` would catch this; one that checks only `msg.sender == midnight` or the market ID would not.

**Protocol's own `LendCallback` demonstrates the vulnerable pattern** (`test/TakeTest.sol:1452-1471`): it checks only `id == IdLib.toId(market, block.chainid, msg.sender)` (verifying the market ID and that `msg.sender` is Midnight), approves `buyerAssets` to Midnight inside `onBuy`, and returns `CALLBACK_SUCCESS` — no check that `buyer == address(this)`. This makes it directly exploitable as a victim if it holds tokens.

**Exploit flow:**
1. Maker signs a sell offer (`offer.buy = false`).
2. Attacker calls `take(offer, ratifierData, units, attacker, receiver, victimContract, data)`.
3. `buyerCallback = victimContract`, `payer = victimContract`.
4. Midnight calls `victimContract.onBuy(..., buyer=attacker, ...)`. If `victimContract` checks only `msg.sender == midnight`, it approves tokens and returns `CALLBACK_SUCCESS`.
5. `safeTransferFrom(loanToken, victimContract, ...)` pulls `buyerAssets` from `victimContract`.
6. The attacker's credit position is credited; the victim paid.

**Why existing checks fail:**
- `TakerUnauthorized` (line 346) guards only who may act as `taker`; it does not constrain `takerCallback`.
- The ratifier validates the maker's signature over `offer` fields; `takerCallback` is not part of the signed offer.
- The Certora spec `OnlyExplicitPayerCanLoseTokens.spec` (lines 91-114) explicitly models and proves that tokens may be pulled from any `buyerCallback` that returned `CALLBACK_SUCCESS` — confirming this path is reachable by design with no additional guard on who may supply `takerCallback`.

## Impact Explanation
Direct theft of ERC-20 tokens from any contract that (a) implements `IBuyCallback`, (b) returns `CALLBACK_SUCCESS` from `onBuy` without verifying `buyer == address(this)`, and (c) holds a token balance with approval to Midnight or approves inside `onBuy`. The victim never authorized the specific trade. This constitutes unauthorized movement of assets triggered by an unprivileged external caller.

## Likelihood Explanation
The attacker requires no privileged access — only the ability to call `take` as a taker. Any contract purpose-built to act as a buy callback in Midnight (flash-loan wrappers, routing contracts, aggregator adapters) will implement `IBuyCallback` and will commonly guard only `msg.sender == midnight` or the market ID, not `buyer == address(this)`. The protocol's own `LendCallback` test contract follows exactly this vulnerable pattern. The attack is repeatable as long as the victim retains its balance and callback interface.

## Recommendation
Constrain `takerCallback` to be either `msg.sender` or an address explicitly authorized by `taker` (e.g., via `isAuthorized[taker][takerCallback]`). Alternatively, pass `msg.sender` or `taker` as the `buyer` argument to `onBuy` and require that callback contracts verify `buyer == address(this)` — and document this requirement explicitly. The fix must be at the protocol level in `Midnight.take`, not delegated to callback implementors.

## Proof of Concept
1. Deploy a `LendCallback` instance pre-funded with loan tokens (or any `IBuyCallback` that checks only `msg.sender == midnight` and approves inside `onBuy`).
2. Have a maker sign a sell offer for the relevant market.
3. Attacker calls `midnight.take(offer, ratifierData, units, attacker, receiver, address(lendCallback), "")`.
4. Midnight calls `lendCallback.onBuy(id, market, buyerAssets, units, fee, attacker, "")`.
5. `lendCallback` passes the market ID check, approves `buyerAssets` to Midnight, returns `CALLBACK_SUCCESS`.
6. Midnight executes `safeTransferFrom(loanToken, lendCallback, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, lendCallback, receiver, sellerAssets)`.
7. Attacker's position is credited; `lendCallback`'s token balance is drained. Assert `lendCallback` token balance decreased by `buyerAssets` and attacker's credit position increased accordingly.
