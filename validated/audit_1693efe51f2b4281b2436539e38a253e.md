Audit Report

## Title
Phantom Collateral via Fee-on-Transfer Token in `supplyCollateral` Enables Undercollateralized Borrowing - (File: src/Midnight.sol)

## Summary
`supplyCollateral` records `assets` worth of collateral and sets the collateral bitmap bit before invoking `SafeTransferLib.safeTransferFrom`, which only verifies the call did not revert and returned `true` — it never measures the actual amount received. A 100%-fee-on-transfer collateral token causes 0 tokens to arrive while the protocol records `assets` phantom collateral, allowing the borrower to borrow against collateral the contract does not hold.

## Finding Description
**Root cause — checks-effects-interactions violation with no received-amount guard:**

In `supplyCollateral` (`src/Midnight.sol` lines 531–545), state is mutated before the transfer executes:

```solidity
uint256 oldCollateral = _position.collateral[collateralIndex];           // line 532
_position.collateral[collateralIndex] = toUint128(oldCollateral + assets); // line 533 ← written first

if (oldCollateral == 0 && assets > 0) {
    _position.collateralBitmap = _position.collateralBitmap.setBit(collateralIndex); // line 536-537
    require(countBits(...) <= MAX_COLLATERALS_PER_BORROWER, ...);         // line 538-540
}

SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545
```

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34) only checks that the low-level call succeeded and that the return value decodes to `true`. It performs no balance-before/after comparison and does not verify that `assets` tokens were actually received.

**Exploit flow:**
1. Attacker deploys an ERC20 token with a 100% transfer fee (all transferred tokens go to a fee recipient; the recipient receives 0).
2. Attacker calls `touchMarket` (permissionless, `src/Midnight.sol` lines 755–791) with this token as `collateralParams[i].token`. `touchMarket` validates only LLTV tiers, token ordering, and `maxLif` — no token-type restriction exists.
3. Attacker calls `supplyCollateral(market, i, assets, self)` with any `assets > 0`.
4. Protocol writes `_position.collateral[i] = assets` and sets the bitmap bit. The `safeTransferFrom` call succeeds (the fee token returns `true`) but delivers 0 tokens to the contract.
5. `isHealthy` (`src/Midnight.sol` lines 944–960) iterates the bitmap and computes `maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)` — a non-zero value derived entirely from phantom collateral.
6. Attacker borrows up to `maxDebt` in real loan tokens against collateral the contract does not hold.

**Why existing checks fail:**
- `safeTransferFrom` has no received-amount check (`src/libraries/SafeTransferLib.sol` lines 24–34).
- `touchMarket` imposes no restriction on token type (`src/Midnight.sol` lines 762–773).
- Certora specs summarize `safeTransferFrom` as `NONDET` (`certora/specs/CreatedMarkets.spec` line 33), so formal verification does not model fee-on-transfer behavior.
- The `nonZeroCollateralsAreActivated` invariant holds on the stored value, but the stored value diverges from actual holdings.

## Impact Explanation
The protocol records `assets` phantom collateral for a borrower who deposited 0 tokens. `isHealthy` computes a non-zero `maxDebt` from this phantom collateral, allowing the borrower to take a loan backed by nothing. When the loan is not repaid, the shortfall becomes bad debt socialized across all lenders in the market. The contract's core solvency invariant — `sum(_position.collateral[i]) ≤ token.balanceOf(midnight)` — is broken. This constitutes direct theft of lender funds, a critical in-scope impact.

## Likelihood Explanation
**Preconditions:**
1. Market creation is permissionless; the attacker deploys the fee-on-transfer token and creates the market themselves — no admin access required.
2. No whitelist or token-type restriction exists in `touchMarket`.
3. The attacker needs only to call `supplyCollateral` once with `assets > 0` and `oldCollateral == 0`.

**Feasibility:** The attack is trivially repeatable. The attacker controls the token contract and can set fee = 100%. No special privileges, no oracle manipulation, and no reentrancy are required. The only practical dependency is attracting lenders to the attacker-created market, which is achievable by mimicking a legitimate market configuration or by exploiting any legitimate token that introduces a transfer fee post-deployment.

## Recommendation
Add a balance-before/after check in `supplyCollateral` to verify the actual amount received:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
require(received == assets, InvalidReceivedAmount());
```

Alternatively, move all state mutations after the transfer and use `received` (not `assets`) as the credited amount, consistent with the checks-effects-interactions pattern. This also resolves the CEI ordering violation.

## Proof of Concept
```solidity
// 1. Deploy a 100%-fee ERC20: transferFrom always returns true but sends 0 to recipient.
// 2. Create market with this token as collateral (touchMarket — permissionless).
// 3. feeToken.approve(address(midnight), 1e18);
// 4. midnight.supplyCollateral(market, 0, 1e18, attacker);
//    → _position.collateral[0] == 1e18 (phantom)
//    → feeToken.balanceOf(address(midnight)) == 0
// 5. Assert midnight.isHealthy(market, id, attacker) == true  (maxDebt > 0 from phantom)
// 6. Take a lender offer to borrow up to maxDebt in real loan tokens.
// 7. Assert loanToken.balanceOf(attacker) > 0 and midnight holds 0 collateral.
```

A Foundry fuzz/invariant test asserting `sum(position.collateral[i]) <= IERC20(token).balanceOf(address(midnight))` for all markets and borrowers would catch this invariant violation deterministically with a fee-on-transfer token in the collateral set.
