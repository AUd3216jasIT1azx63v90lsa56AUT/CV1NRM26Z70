Audit Report

## Title
Inclusive `<=` boundary in `CannotIncreaseDebtPostMaturity` allows seller debt increase at exactly `block.timestamp == maturity` - (File: src/Midnight.sol)

## Summary
The guard at line 391 of `src/Midnight.sol` uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` when `block.timestamp == maturity`, permitting `sellerDebtIncrease > 0` to proceed in that exact block. The expiry check at line 353 uses the same inclusive operator, so a sell offer with `offer.expiry == offer.market.maturity` remains valid at that timestamp, completing the exploit path. The error name `CannotIncreaseDebtPostMaturity` and the surrounding logic make clear the protocol intends to block debt increases at or after maturity, but the `<=` boundary leaves a one-block window unprotected.

## Finding Description

**Exact code path (`src/Midnight.sol`):**

```solidity
// line 353
require(block.timestamp <= offer.expiry, OfferExpired());
// at block.timestamp == maturity: maturity <= maturity → passes

// line 391
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0,
        CannotIncreaseDebtPostMaturity());
// at block.timestamp == maturity: maturity <= maturity → TRUE → does NOT revert
// even when sellerDebtIncrease > 0

// line 414 (executes unconditionally after line 391 passes)
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
```

**Root cause:** The `<=` operator on line 391 classifies `block.timestamp == maturity` as pre-maturity for the debt-increase guard. The correct semantic is strict: debt must not increase at or after maturity (`<`).

**Exploit flow:**
1. Maker signs a sell offer (`offer.buy = false`) with `offer.expiry >= offer.market.maturity`. `EcrecoverRatifier.isRatified` (lines 33–46, `src/ratifiers/EcrecoverRatifier.sol`) validates only the signature and Merkle proof — no maturity check.
2. Taker waits until the block where `block.timestamp == maturity`.
3. Taker calls `Midnight.take` with `units > sellerPos.credit` so `sellerDebtIncrease = units - sellerCreditDecrease > 0` (line 384).
4. Line 353: `maturity <= maturity` → passes.
5. Line 391: `maturity <= maturity` → `true` → `CannotIncreaseDebtPostMaturity` does not revert.
6. Line 414: `sellerPos.debt += sellerDebtIncrease` executes, increasing the maker's debt at exactly maturity.
7. From the next block onward (`block.timestamp > maturity`), the position enters post-maturity liquidation mode (per the protocol's own comment at lines 62–64: "post-maturity mode, available after the market's maturity"), making the newly created debt immediately subject to liquidation with no grace period.

**Why existing checks fail:** The sole guard against post-maturity debt increase is line 391. It uses `<=` instead of `<`, leaving the single-block window at `block.timestamp == maturity` unprotected. No other check in `take` or in `EcrecoverRatifier.isRatified` closes this gap.

## Impact Explanation
An unprivileged taker can force a maker's debt to increase at exactly `block.timestamp == maturity` by taking a qualifying sell offer in that block. The resulting debt is immediately subject to post-maturity liquidation from the next block onward, with no grace period, even if the position was healthy one block earlier. This violates the protocol's core accounting invariant that debt must not increase at or after maturity, and directly exposes the maker's collateral to liquidation under conditions they could not have anticipated or defended against.

## Likelihood Explanation
**Required preconditions:**
- A valid sell offer exists with `offer.expiry >= offer.market.maturity` (routine; makers commonly set expiry at or beyond maturity).
- The maker has no existing credit in the market (or less credit than `units`), so `sellerDebtIncrease > 0`.
- The taker submits the transaction in the block where `block.timestamp == maturity`.

**Feasibility:** Maturity timestamps are public and fixed at market creation. A taker can monitor the chain and target the maturity block. Block timestamps are validator-controlled within a small range, making precise targeting straightforward. The attack is repeatable across every market and every qualifying offer, and requires no privileged access.

## Recommendation
Change the comparison on line 391 from `<=` to `<`:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0,
        CannotIncreaseDebtPostMaturity());

// After (correct):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0,
        CannotIncreaseDebtPostMaturity());
```

This ensures that `block.timestamp == maturity` is treated as post-maturity for the debt-increase guard, consistent with the protocol's intent and the post-maturity liquidation mode boundary.

## Proof of Concept
**Minimal test steps:**
1. Deploy a market with `maturity = T`.
2. Maker signs a sell offer with `offer.expiry = T`, `offer.buy = false`, and no existing credit in the market.
3. Warp to `block.timestamp = T`.
4. Taker calls `Midnight.take` with `units = 1` (so `sellerDebtIncrease = 1 > 0`).
5. Assert: the call succeeds (does not revert with `CannotIncreaseDebtPostMaturity`).
6. Assert: `sellerPos.debt` increased by 1.
7. Warp to `block.timestamp = T + 1`.
8. Assert: the maker's position is liquidatable via post-maturity mode.

**Expected (buggy) result:** Steps 5–8 all pass, confirming debt was increased at maturity and is immediately liquidatable afterward.
