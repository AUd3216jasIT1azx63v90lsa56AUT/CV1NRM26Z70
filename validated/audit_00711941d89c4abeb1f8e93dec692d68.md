Audit Report

## Title
Zero-rounding in `maxAssets` branch allows unbounded offer reuse with no token transfer — (`src/Midnight.sol`, `src/libraries/UtilsLib.sol`)

## Summary
When a buy offer has `maxAssets > 0`, the `consumed` mapping is incremented by `buyerAssets`, which is computed via `mulDivDown(units, buyerPrice, WAD)`. For sufficiently small `units`, integer division truncates this to zero, so `consumed` never advances and the offer cap is never reached. Each such call still mutates buyer credit and seller debt by `units`, creating unbounded credit and debt with zero token transfer.

## Finding Description
**Root cause:** `mulDivDown` in `UtilsLib.sol` line 29–31 is `(x * y) / d` with no rounding-up guard. For a buy offer, `buyerPrice = offerPrice` (lines 361–362: `sellerPrice = offerPrice - fee`, `buyerPrice = sellerPrice + fee`). When `units * offerPrice < WAD`, `buyerAssets = 0`.

**Affected branch** (`src/Midnight.sol` lines 367–369):
```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
```
`consumed += 0` leaves `newConsumed` unchanged; the `require` passes every time.

**Position mutation still occurs** (lines 382–414): `buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)` and `sellerDebtIncrease = units - sellerCreditDecrease` are both non-zero when the buyer has no debt and the seller has no credit. `buyerPos.credit += units` and `sellerPos.debt += units` execute unconditionally.

**Token transfers** (lines 455–456): `safeTransferFrom(..., buyerAssets - sellerAssets)` and `safeTransferFrom(..., sellerAssets)` both transfer 0 tokens.

**No guard exists:** there is no `require(units > 0)` and no `require(buyerAssets > 0)` anywhere in `take`. The `grep` confirms zero `require.*units` matches in `src/Midnight.sol`.

**Why the `maxUnits` branch is immune:** line 371 uses `+= units`, not `+= buyerAssets`, so it always advances.

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = M > 0` at any tick where `offerPrice < WAD` (every valid tick).
2. Attacker computes `U = floor((WAD - 1) / offerPrice)` off-chain; at `PRICE_ROUNDING_STEP = 1e12` (minimum price), `U = 999_999`.
3. Attacker calls `take(offer, ..., units=U)` repeatedly.
4. Each call: `buyerAssets = 0`, `consumed` unchanged, `require(consumed <= M)` passes, buyer gains `U` credit, seller gains `U` debt, 0 tokens transferred.
5. Repeat indefinitely — `consumed` never reaches `M`.

## Impact Explanation
Every iteration grants the buyer (maker) `U` units of credit and burdens the seller (taker) with `U` units of debt, both unbacked by any token transfer. Repeated calls create arbitrarily large credit balances unsupported by the contract's loan-token holdings, directly violating the solvency invariant: the contract's balance of `loanToken` does not cover the credit it has issued. This is a critical accounting/solvency break exploitable by any unprivileged taker.

## Likelihood Explanation
Preconditions are minimal: any buy offer with `maxAssets > 0` at any tick (the entire valid tick range satisfies `offerPrice < WAD`). No special permissions, oracle manipulation, or victim cooperation is required. The required `units` value is trivially computed off-chain. The attack is repeatable without limit since `consumed` never advances.

## Recommendation
Add a guard in the `maxAssets` branch requiring that the asset increment is non-zero when `units > 0`:
```solidity
if (offer.maxAssets > 0) {
    uint256 assetIncrement = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetIncrement > 0, ZeroAssetIncrement());
    newConsumed = consumed[offer.maker][offer.group] += assetIncrement;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```
Alternatively, enforce a minimum `units` value such that `units * min(buyerPrice, sellerPrice) >= WAD`, or switch the `maxAssets` branch to use `mulDivUp` for the consumed increment.
