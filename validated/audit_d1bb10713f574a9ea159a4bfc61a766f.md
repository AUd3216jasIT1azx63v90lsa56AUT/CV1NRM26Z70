Audit Report

## Title
Settlement Fee Interpolation Floor Division Causes Systematic Protocol Fee Undercollection - (File: `src/Midnight.sol`)

## Summary
`settlementFee()` at line 979 uses Solidity integer (floor) division to interpolate between two CBP-aligned breakpoint values over a time interval whose width in seconds is not a multiple of `CBP = 1e12`. This produces a result that is always truncated downward by up to `CBP − 1 = 999 999 999 999` wei relative to the true rational interpolation. Because the truncation is always in the same direction, every `take()` call at a non-breakpoint TTM with `feeLower ≠ feeUpper` causes the protocol to collect strictly less settlement fee than intended, and the shortfall accumulates permanently in `claimableSettlementFee`.

## Finding Description

**Root cause — `src/Midnight.sol` line 979:**
```solidity
return (feeLower * (end - timeToMaturity) + feeUpper * (timeToMaturity - start)) / (end - start);
```
`feeLower` and `feeUpper` are `settlementFeeCbpN * CBP` where `CBP = 1e12` (`src/libraries/ConstantsLib.sol` line 10) and `settlementFeeCbpN` is a `uint16`. The numerator is therefore always a multiple of `CBP = 1e12`. The denominator `(end − start)` is a pure second-count (e.g. `518 400` for the [1 d, 7 d] interval). Because `gcd(1e12, 518400) = 86400`, the division truncates whenever the weighted sum of the uint16 coefficients is not divisible by `518400 / 86400 = 6`.

**Concrete example** — interval [1 d, 7 d], `feeLower = 1 × CBP`, `feeUpper = 2 × CBP`, TTM = 3 days:
- numerator = `1e12 × 345600 + 2e12 × 172800 = 691200e12`
- denominator = `518400`
- exact result = `1 333 333 333 333.33…`
- Solidity result = `1 333 333 333 333`
- truncation = `333 333 333 333` wei ≈ 0.33 × CBP

**Exploit flow:**
1. Taker observes a market where `feeLower ≠ feeUpper` for the relevant breakpoint interval (the common case).
2. Taker computes the `block.timestamp` (i.e. TTM) that maximises `numerator mod denominator`.
3. Taker submits `take()` at that block. `timeToMaturity = zeroFloorSub(offer.market.maturity, block.timestamp)` is determined by the taker's choice of when to transact.
4. `_settlementFee` is the truncated value; `buyerPrice = sellerPrice + _settlementFee` is lower than the true interpolated price.
5. `claimableSettlementFee[token] += buyerAssets − sellerAssets` is incremented by the truncated spread, permanently under-crediting the protocol.

**Why existing checks do not stop it:**
`setMarketSettlementFee` and `setDefaultSettlementFee` enforce `newSettlementFee % CBP == 0` on the stored breakpoint values, but impose no constraint on the interpolated result. There is no rounding-up or ceiling division anywhere in `settlementFee()`.

## Impact Explanation
On every `take()` where TTM falls strictly between two breakpoints and `feeLower ≠ feeUpper`, the protocol collects up to `CBP − 1 ≈ 1e12` wei less settlement fee per call than the true linear interpolation. The shortfall is a permanent deficit in `claimableSettlementFee` — it cannot be recovered. For a high-volume market the aggregate loss is unbounded (loss per fill × number of fills). The maximum per-trade loss in collected assets is `units × (CBP − 1) / WAD`; for a 1 WAD trade this is ≈ 1e12 wei, which is small per trade but accumulates continuously.

## Likelihood Explanation
Preconditions are minimal: any market where the fee setter has configured at least two adjacent breakpoints with different values (the standard configuration). The taker needs only to choose the right block to submit in — `block.timestamp` is not freely settable by the taker, but the taker can select which block their transaction lands in, giving them practical control over TTM at second-level granularity. The condition is met on every fill in the interior of any breakpoint interval, making it continuous and repeatable without any special setup.

## Recommendation
Replace the floor division in `settlementFee()` with ceiling division (round up toward the protocol) to ensure the protocol always collects at least the true interpolated value:

```solidity
uint256 numerator = feeLower * (end - timeToMaturity) + feeUpper * (timeToMaturity - start);
uint256 denominator = end - start;
return (numerator + denominator - 1) / denominator;
```

Alternatively, use a `mulDivUp`-style helper consistent with the rest of the codebase. This guarantees the interpolated fee is always ≥ the true rational value, bounded by at most `CBP − 1` wei above it — within the existing fee granularity.

## Proof of Concept
Minimal Foundry test (extend `SettlementFeeTest`):

```solidity
function testInterpolationTruncation() public {
    // Set feeLower = 1 CBP, feeUpper = 2 CBP for [1d, 7d] interval
    midnight.setDefaultSettlementFee(address(loanToken), 1, 1e12);
    midnight.setDefaultSettlementFee(address(loanToken), 2, 2e12);

    market.maturity = block.timestamp + 3 days; // TTM = 3 days
    bytes32 id = midnight.touchMarket(market);

    uint256 fee = midnight.settlementFee(id, 3 days);
    // True value: (1e12*345600 + 2e12*172800) / 518400 = 1333333333333.33...
    // Solidity returns: 1333333333333
    assertEq(fee, 1333333333333);          // truncated
    // Show truncation: true value * 518400 > fee * 518400
    assertGt(1e12 * 345600 + 2e12 * 172800, fee * 518400); // remainder != 0
}
```

Running this test will confirm the truncation. A fuzz test over all TTM values in [1d, 7d] with `feeLower ≠ feeUpper` will show that the remainder `(numerator % denominator) > 0` for the majority of second-level TTM values, confirming systematic undercollection.
