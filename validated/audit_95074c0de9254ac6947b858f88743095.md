### Title
Staker Cannot Specify Minimum Expiry Bound in `stake()`, Enabling Sponsor to Front-Run and Reduce Pool Duration - (File: src/ConfidencePool.sol)

### Summary
`stake()` accepts only `amount` as a parameter, providing no mechanism for a staker to specify a minimum acceptable `expiry`. Because `expiry` is sponsor-mutable until the first stake, a malicious sponsor can front-run a staker's transaction with `setExpiry()` to reduce the pool's duration to the protocol minimum (30 days), atomically locking the staker into a materially different pool than they evaluated.

### Finding Description
`stake()` is a single-parameter function with no slippage guard on pool terms:

```solidity
function stake(uint256 amount) external nonReentrant whenPoolNotPaused {
``` [1](#0-0) 

`expiry` is the upper bound `T` in the k=2 bonus formula for the EXPIRED path and defines the staking window. It is freely mutable by the sponsor via `setExpiry()` until `expiryLocked` is set:

```solidity
function setExpiry(uint256 newExpiry) external onlyOwner {
    if (expiryLocked) revert ExpiryLocked();
    if (newExpiry < block.timestamp + _MIN_EXPIRY_LEAD) revert ExpiryTooSoon();
``` [2](#0-1) 

`setExpiry()` has no registry-state check, so it is callable even while the registry is in `UNDER_ATTACK`. `expiryLocked` is set to `true` only inside `stake()` itself:

```solidity
if (!expiryLocked) {
    expiryLocked = true;
}
``` [3](#0-2) 

**Concrete attack sequence:**

1. Sponsor creates pool with `expiry = block.timestamp + 365 days`.
2. Registry transitions to `UNDER_ATTACK`; `riskWindowStart == 0` (no one has poked yet).
3. Staker evaluates the 1-year pool and submits `stake(amount)`.
4. Sponsor front-runs with `setExpiry(block.timestamp + 30 days)` — valid because `expiryLocked == false`.
5. Staker's `stake()` mines: `_observePoolState()` seals `riskWindowStart = block.timestamp` (registry is `UNDER_ATTACK`), then `expiryLocked = true` is set.
6. Staker is now locked in a 30-day pool with `riskWindowStart != 0`; `withdraw()` is permanently disabled.

The staker had no way to express "revert if `expiry < X`" in their transaction.

### Impact Explanation
- The staker is irrevocably locked into a pool whose duration is up to ~12× shorter than evaluated, with no withdrawal path because `riskWindowStart != 0` is a one-way latch.
- The k=2 bonus formula's `T = expiry` is compressed: if the sponsor reduces `expiry` to just above `riskWindowStart`, every staker's `(T − entryTime)²` collapses toward zero, driving `globalScore` to zero and triggering the amount-weighted fallback — eliminating the early-staker advantage the formula is designed to provide.
- The staker's risk profile is materially different from what they consented to: a 30-day window has a different probability of reaching `CORRUPTED` than a 1-year window.

### Likelihood Explanation
- The sponsor is not listed as a trusted actor (only the moderator/DAO and factory owner are trusted); malicious sponsor behavior is in scope.
- The attack requires the registry to be in `UNDER_ATTACK` at the time of the first stake (so `riskWindowStart` seals and `withdraw()` is disabled), which is the normal operating state for an active agreement.
- The sponsor controls the pool deployment and has full knowledge of pending mempool transactions, making front-running straightforward.
- `DESIGN.md §10` acknowledges `expiry` is mutable until first stake to "protect staker reliance," but does not address or accept the front-running race against the first staker.

### Recommendation
Add a `minExpiry` slippage guard to `stake()`:

```solidity
function stake(uint256 amount, uint256 minExpiry) external nonReentrant whenPoolNotPaused {
    if (expiry < minExpiry) revert ExpiryBelowMinimum();
    ...
}
```

This allows stakers to atomically assert the pool's deadline meets their minimum requirement, reverting if the sponsor has reduced it between the staker's off-chain evaluation and on-chain execution.

### Proof of Concept

```solidity
// Setup: registry is UNDER_ATTACK, pool has expiry = now + 365 days, no stakes yet
// (expiryLocked == false, riskWindowStart == 0)

// Step 1: Staker submits stake(1000e18) — sees 1-year pool
// Step 2: Sponsor front-runs:
pool.setExpiry(block.timestamp + 30 days); // valid: expiryLocked == false

// Step 3: Staker's tx mines:
pool.stake(1000e18);
// Inside stake():
//   _observePoolState() → registry is UNDER_ATTACK → _markRiskWindowStart() fires
//   riskWindowStart = block.timestamp  (one-way latch sealed)
//   expiryLocked = true

// Step 4: Staker tries to exit:
pool.withdraw(); // reverts WithdrawsDisabled — riskWindowStart != 0
// Staker is locked in a 30-day pool they never agreed to
```
