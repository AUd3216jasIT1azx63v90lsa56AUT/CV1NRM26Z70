Audit Report

## Title
Asymmetric Maturity Boundary Operators Enable Debt Creation at Exact Maturity Followed by Immediate Health-Agnostic Liquidation — (File: `src/Midnight.sol`)

## Summary
`take` uses `block.timestamp <= offer.market.maturity` (line 391) to guard debt creation, permitting `sellerDebtIncrease > 0` when `block.timestamp == maturity`. `liquidate` uses the strict `block.timestamp > market.maturity` (line 622) for post-maturity mode, meaning a position whose debt was created at exactly `T` becomes unconditionally liquidatable at `T+1` regardless of collateral health. This directly violates `live_context.json` line 221, which explicitly names timestamp equality as a forbidden debt-increase path.

## Finding Description
**Root cause — asymmetric boundary operators:**

`src/Midnight.sol` line 391:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
At `block.timestamp == T`, `T <= T` evaluates to `true`, so the require passes unconditionally. `sellerDebtIncrease` is computed at line 384 and written to state at line 414:
```solidity
uint256 sellerDebtIncrease = units - sellerCreditDecrease;
...
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
```

`src/Midnight.sol` lines 620–622:
```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```
`postMaturityMode` is a caller-supplied `bool` (line 587). At `T`, `T > T` is `false` — post-maturity liquidation is blocked. At `T+1`, `T+1 > T` is `true` — any borrower with debt is liquidatable regardless of health, per `live_context.json` line 88: `"overdue_debt_after_maturity_is_liquidatable_even_if_healthy": true`.

**Protocol invariants explicitly violated:**
- `live_context.json` line 221: `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"` — names this exact case.
- `live_context.json` line 86: `"debt_increase_after_maturity_forbidden": true`.
- `live_context.json` line 197 (core_invariants): `"debt must not increase after maturity"`.

**Exploit flow:**
1. A market with `maturity = T` is deployed. The maturity timestamp is a public on-chain parameter.
2. A taker submits `take` in the block where `block.timestamp == T`, targeting a sell offer where `sellerDebtIncrease > 0`. The check `T <= T` passes; debt is written to the seller's position.
3. At `T+1`, a liquidator calls `liquidate(..., postMaturityMode=true)`. The check `T+1 > T` passes. The position is liquidatable even if fully collateralized.
4. LIF ramps from ~1 at `T+1` to `maxLif` over 15 minutes (`live_context.json` line 142), at which point the liquidator seizes collateral at full incentive from a position that was healthy at creation.

If the seller is a maker whose offer is taken at exactly `T` by a front-running taker, the seller had no ability to prevent the take, and their collateral becomes seizable within 15 minutes.

**Existing checks shown insufficient:**
The only guard against post-maturity debt creation is the `<=` check at line 391. There is no secondary check, no cooldown, and no health-based protection in the post-maturity liquidation path. The `postMaturityMode` flag is entirely caller-controlled with no on-chain validation beyond the `block.timestamp > market.maturity` comparison.

## Impact Explanation
Unauthorized collateral seizure from a healthy position. A borrower whose debt is created at exactly `T` is immediately subject to post-maturity liquidation at `T+1` with no health-based protection. This matches two of the protocol's listed best bug classes: `"unauthorized collateral seizure"` and `"healthy-account liquidation"` (`live_context.json` lines 58–61). The borrower suffers a concrete, irreversible loss of collateral from a position that the protocol's own invariant (`live_context.json` line 221) explicitly forbids creating.

## Likelihood Explanation
Requires `block.timestamp == market.maturity` exactly. Market maturities are immutable public on-chain parameters (`live_context.json` line 70), making the target timestamp known in advance. On proof-of-stake Ethereum, validators control block timestamps within the allowed drift window, making exact timestamp targeting feasible for a validator-attacker. For non-validators, the condition requires a transaction landing in the exact maturity block, which is achievable by monitoring the mempool and submitting with appropriate gas. The condition is repeatable across every market whose maturity falls on a future block. No privileged access is required — the attacker is an unprivileged taker.

## Recommendation
Change the maturity guard in `take` from `<=` to `<` to match the strict post-maturity semantics used in `liquidate`:

```solidity
// Before (line 391):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After:
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This ensures that at `block.timestamp == maturity`, debt creation is forbidden, eliminating the one-second asymmetry. Alternatively, change the liquidation guard from `>` to `>=`, but the `<` fix in `take` is preferable as it aligns with the invariant that debt must not increase at or after maturity.
