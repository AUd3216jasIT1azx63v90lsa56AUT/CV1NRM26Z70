Audit Report

## Title
Zero-rounding in `maxAssets` branch allows unbounded offer replay with free credit minting and no token transfer — (`src/Midnight.sol`, `src/libraries/UtilsLib.sol`)

## Summary
When a buy offer has `maxAssets > 0` and the taker supplies `units` small enough that `units * buyerPrice < WAD`, `mulDivDown` returns zero and `consumed` is incremented by zero. The `require(newConsumed <= offer.maxAssets)` check passes unchanged on every call, so the offer is never exhausted. Meanwhile, position state mutates normally — the maker accrues credit and the taker accrues debt — with zero token transfers, creating unbacked credit and violating the protocol solvency invariant.

## Finding Description

**Root cause:** `mulDivDown(units, buyerPrice, WAD)` performs integer division `(units * buyerPrice) / WAD`. When `units * buyerPrice < WAD`, the quotient is 0. The `maxAssets` branch increments `consumed` by this zero value, so the cap check `require(newConsumed <= offer.maxAssets)` is trivially satisfied on every call.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` enforces an upper bound but does not require `consumed` to have increased. [6](#0-5) 
- There is no `require(units > 0)` or `require(buyerAssets > 0)` guard anywhere in `take`. Grep confirms no `require.*units` check in `src/`.
- The `maxUnits` branch (line 371) uses `+= units` and is immune; only the `maxAssets` branch is affected. [7](#0-6) 

**Exploit flow:**
1. Attacker controls address A (maker) and address B (taker), satisfying `offer.maker != taker` at line 354.
2. A posts a buy offer with `maxAssets = M > 0` at a low tick (e.g., tick 0, `offerPrice ≈ 1e12` given `PRICE_ROUNDING_STEP = 1e12`).
3. B calls `take(offer, ..., units = U)` where `U < WAD / offerPrice` (e.g., `U = 999_999` at `offerPrice = 1e12`).
4. `buyerAssets = mulDivDown(999_999, 1e12, 1e18) = 0`.
5. `consumed += 0` → `consumed` unchanged; `require(consumed <= M)` passes.
6. A's credit increases by `U`; B's debt increases by `U`.
7. Both token transfers are for 0 tokens.
8. Steps 3–7 repeat indefinitely; `consumed` never reaches `M`.

## Impact Explanation
Each iteration grants the maker (A) `U` units of credit with zero token payment and burdens the taker (B) with `U` units of debt with zero token receipt. Repeated indefinitely, A accumulates unbounded credit not backed by any token deposit. When A redeems this credit, the protocol pays out tokens deposited by legitimate users, directly draining the contract's loan-token balance. This is a direct protocol insolvency / theft-of-funds impact.

## Likelihood Explanation
**Preconditions:** A buy offer with `maxAssets > 0` must exist at any tick below the maximum — true for virtually all real offers. The attacker needs two addresses (trivially created). No admin access, oracle manipulation, or victim cooperation is required.

**Feasibility:** The required `units` value is computed off-chain as `floor((WAD - 1) / offerPrice)`. At `offerPrice = 1e12` (tick 0), `U = 999_999`. At higher ticks the window narrows but remains non-zero for any tick where `offerPrice < WAD` (i.e., all valid ticks below MAX_TICK). [8](#0-7) 

**Repeatability:** Unlimited — `consumed` never advances, so the offer never closes.

## Recommendation
Add a guard in the `maxAssets` branch requiring that the asset delta is non-zero before updating `consumed`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0, ZeroAssetsDelta());   // NEW
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` value such that `units * offerPrice >= WAD` before entering the accounting logic, or use `mulDivUp` for the buy-offer `buyerAssets` calculation in the `maxAssets` branch to prevent rounding to zero.

## Proof of Concept
**Minimal manual steps:**
1. Deploy with a standard ERC-20 loan token.
2. Address A creates a buy offer: `tick = 0`, `maxAssets = 1e18`, `maxUnits = 0`.
3. Address B calls `take(offer, ..., units = 999_999)` repeatedly (e.g., 1000 times).
4. After each call, verify `consumed[A][group]` remains 0 and `position[id][A].credit` increases by `999_999` per call.
5. After 1000 calls, A has `999_999_000` units of credit; token balance of contract is unchanged.
6. A redeems credit — contract pays out tokens it never received, confirming insolvency.

**Fuzz/invariant test:** Assert `position[id][buyer].credit_after - position[id][buyer].credit_before <= buyerAssets_transferred` for every `take` call. This invariant will be violated immediately on the first call with the described parameters.
