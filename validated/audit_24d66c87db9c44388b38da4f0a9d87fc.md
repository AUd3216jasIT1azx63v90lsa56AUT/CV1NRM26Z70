### Title
Zero-payout `claimAttackerBounty` emits misleading event without state change - (File: src/ConfidencePool.sol)

### Summary
`claimAttackerBounty()` does not guard against a zero `payout` before emitting `AttackerBountyClaimed`. When the contract's token balance is zero at call time, the function silently no-ops — no tokens move, no meaningful state changes — yet still emits the event with `payout = 0`, wasting gas and producing a misleading on-chain record.

### Finding Description
In `claimAttackerBounty`, after all access-control checks pass, the payout is computed as:

```solidity
uint256 remaining = bountyEntitlement - bountyClaimed;
uint256 freeBalance = stakeToken.balanceOf(address(this));
uint256 payout = remaining <= freeBalance ? remaining : freeBalance;
``` [1](#0-0) 

When `freeBalance == 0`, `payout` resolves to `0`. The subsequent `if (payout > 0)` block is skipped entirely — no `corruptedReserve` decrement, no `claimsStarted` flip, no token transfer:

```solidity
if (payout > 0) {
    corruptedReserve -= payout;
    if (!claimsStarted) claimsStarted = true;
    stakeToken.safeTransfer(attacker, payout);
}
``` [2](#0-1) 

However, the event emission is unconditional and fires regardless:

```solidity
emit AttackerBountyClaimed(attacker, payout, newBountyClaimed, bountyEntitlement);
``` [3](#0-2) 

Because `bountyClaimed` is incremented by zero (`bountyClaimed = bountyClaimed + 0`), the `bountyClaimed == bountyEntitlement` guard at line 434 is never satisfied, so the named attacker can call this function repeatedly in the zero-balance state, each time emitting a spurious `AttackerBountyClaimed(attacker, 0, ...)` event. [4](#0-3) 

### Impact Explanation
The direct consequences are: (1) misleading `AttackerBountyClaimed` events with `payout = 0` are emitted on-chain, which off-chain indexers and UIs may misinterpret as successful bounty claims; (2) gas is consumed for a no-op call; (3) `claimsStarted` is never flipped, so the moderator's re-flag window remains open longer than expected; (4) `claimCorrupted` remains blocked by `MustClaimBountyFirst` even though no progress was made. No principal is lost and no third-party funds are at risk.

### Likelihood Explanation
Requires `stakeToken.balanceOf(address(this)) == 0` at the moment `claimAttackerBounty` is called in a good-faith CORRUPTED pool. Under normal operation `bountyEntitlement = snapshotTotalStaked + snapshotTotalBonus` equals the entire pool balance at resolution time, so the balance should always cover the entitlement. The zero-balance edge case is only reachable through an unusual token behavior (e.g., a token that self-drains) or a race with another sweep path — both unlikely given the factory allowlist and the `MustClaimBountyFirst` / `ClaimWindowNotExpired` guards on the other sweep functions. Likelihood is low.

### Recommendation
Add an explicit zero-payout guard immediately after computing `payout`, before the state update and event emission, mirroring the pattern already used in `claimCorrupted` and `sweepUnclaimedCorrupted`:

```solidity
uint256 payout = remaining <= freeBalance ? remaining : freeBalance;
if (payout == 0) revert NothingToSweep();
```

This makes the function revert rather than emit a misleading event, consistent with how every other sweep/claim function in the contract handles a zero-transfer scenario. [5](#0-4) [6](#0-5) 

### Proof of Concept
1. Moderator calls `flagOutcome(CORRUPTED, true, attackerAddr)` — pool enters good-faith CORRUPTED; `bountyEntitlement = snapshotTotalStaked + snapshotTotalBonus > 0`.
2. Through any mechanism that reduces `stakeToken.balanceOf(pool)` to `0` before the attacker claims (e.g., a defective token that self-drains, or a test environment where the balance was never funded), `freeBalance == 0`.
3. Named attacker calls `claimAttackerBounty()`. All guards pass (`bountyClaimed(0) != bountyEntitlement(>0)`, `goodFaith == true`, `msg.sender == attacker`, deadline not passed).
4. `payout = min(remaining, 0) = 0`. The `if (payout > 0)` block is skipped. `bountyClaimed` stays at `0`.
5. `emit AttackerBountyClaimed(attacker, 0, 0, bountyEntitlement)` fires — a zero-payout event with no state change.
6. Step 3–5 can be repeated indefinitely, each time emitting another spurious event and consuming gas, while `claimCorrupted` remains permanently blocked by `MustClaimBountyFirst`. [7](#0-6)
