Audit Report

## Title
Reentrancy via `IBuyCallback.onBuy` allows buyer to drain `withdrawable` before paying `buyerAssets` - (File: `src/Midnight.sol`)

## Summary

In `take`, `buyerPos.credit` is written to storage at line 410 before any external call. The `onBuy` callback fires at lines 447–452, and the corresponding `safeTransferFrom` payment only executes at lines 455–456 after the callback returns. A malicious `IBuyCallback` can reenter `withdraw` during `onBuy`, consuming the freshly-credited units against the market's pre-existing `withdrawable` balance before the buyer has paid a single token. The net result is the attacker receives `units` loan tokens while paying only `buyerAssets = units * buyerPrice / WAD`, pocketing `units * (1 - buyerPrice/WAD)` for free.

## Finding Description

**Broken CEI ordering in `take`:**

`buyerPos.credit` is incremented at line 410 before any external interaction: [1](#0-0) 

The `onBuy` callback fires at lines 447–452, while payment has not yet been collected: [2](#0-1) 

Payment only arrives after the callback returns: [3](#0-2) 

**`withdraw` has no reentrancy guard — only an authorization check:** [4](#0-3) 

The state mutations and token transfer in `withdraw`: [5](#0-4) 

**Exploit flow (taking a sell offer, `offer.buy = false`):**

1. Attacker calls `setIsAuthorized(maliciousCallback, true, taker)` — permissionless, one tx.
2. A market exists with `withdrawable >= units` (any active market after repayments/liquidations).
3. A sell offer exists with `offerPrice + settlementFee < WAD` (standard discounted lending offer).
4. Attacker calls `take(sellOffer, ..., units, taker, ..., maliciousCallback, ...)`.
5. Inside `take`:
   - `buyer = taker`, `buyerCallback = takerCallback = maliciousCallback`, `payer = maliciousCallback`.
   - Line 410: `buyerPos.credit += units` (taker has zero prior debt, so `buyerCreditIncrease = units`).
   - Lines 447–452: `maliciousCallback.onBuy(...)` is called.
6. Inside `onBuy`, `maliciousCallback` calls `withdraw(market, units, taker, maliciousCallback)`:
   - Auth check passes: `isAuthorized[taker][maliciousCallback] == true`.
   - `_position.credit -= units` → 0.
   - `_marketState.withdrawable -= units` (drains other lenders' repaid funds).
   - `safeTransfer(loanToken, maliciousCallback, units)` → callback receives `units` tokens.
7. `onBuy` returns `CALLBACK_SUCCESS`.
8. Back in `take`, lines 455–456: `maliciousCallback` pays `buyerAssets` total.
9. **Net gain = `units − buyerAssets = units * (1 − buyerPrice/WAD) > 0`.**

**Why existing checks fail:**

- `withdraw`'s authorization check is satisfied by the attacker's pre-authorization of the callback — a standard, permissionless operation via `setIsAuthorized`. [6](#0-5) 
- There is no reentrancy guard anywhere in `Midnight.sol`.
- The TOKEN SAFETY REQUIREMENTS comment (`"It should not re-enter Midnight on transfer nor transferFrom"`) applies only to the loan token itself, not to the `onBuy` callback mechanism — the reentrancy here occurs through the callback, not a token transfer. [7](#0-6) 
- The `withdrawable` underflow guard only requires `withdrawable >= units`, which is the stated precondition.

## Impact Explanation

Direct theft of loan tokens from the contract. The attacker drains `withdrawable` — the pool of tokens repaid by borrowers and owed to other lenders — without depositing the corresponding assets. Other lenders who hold credit in the same market and attempt to `withdraw` after the attack will find `withdrawable` insufficient and their calls will revert. The stolen amount per attack is `units * (1 − buyerPrice/WAD)`, which approaches `units` as the offer price approaches zero. This is a critical-severity direct theft of user funds.

## Likelihood Explanation

All preconditions are trivially satisfiable by any unprivileged user:
- `withdrawable >= units`: satisfied in any active market after any repayment or liquidation.
- Sell offer with `offerPrice + settlementFee < WAD`: satisfied by virtually every real lending offer.
- Attacker pre-authorizes callback: one transaction, no privilege, no cost beyond gas.

The attack is fully repeatable. The callback contract is self-funding: it receives `units` tokens from `withdraw` and only needs to return `buyerAssets < units` to `take`, requiring zero net capital beyond gas.

## Recommendation

Apply the CEI (Checks-Effects-Interactions) pattern strictly in `take`. Move both `safeTransferFrom` calls (lines 455–456) to **before** the `onBuy` callback invocation (lines 447–452), so that payment is collected before any external call can reenter the contract. Alternatively, add a transient-storage reentrancy guard (the contract already uses `tload`/`tstore` for the liquidation lock) that prevents any reentrant call to `withdraw`, `repay`, `take`, or other state-mutating functions during callback execution. [8](#0-7) 

## Proof of Concept

**Minimal Foundry test outline:**

1. Deploy `Midnight` and a mock ERC-20 loan token.
2. Create a market; have a legitimate borrower repay `units` tokens so `withdrawable = units`.
3. Deploy `MaliciousCallback` implementing `IBuyCallback`:
   - In `onBuy`, call `midnight.withdraw(market, units, taker, address(this))`.
   - Return `CALLBACK_SUCCESS`.
4. Attacker EOA calls `midnight.setIsAuthorized(address(maliciousCallback), true, taker)`.
5. Place a sell offer with `offerPrice = 0.5e18` (50% of WAD).
6. Attacker calls `midnight.take(sellOffer, ..., units, taker, ..., address(maliciousCallback), ...)`.
7. **Assert:** `maliciousCallback` token balance increased by `units − buyerAssets > 0`; `marketState.withdrawable == 0`; other lenders' subsequent `withdraw` calls revert.
