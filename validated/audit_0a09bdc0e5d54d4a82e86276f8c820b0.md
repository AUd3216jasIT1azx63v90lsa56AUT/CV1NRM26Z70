Looking at the `stake()` function in `ConfidencePool.sol` for an analog to the missing-deadline vulnerability class.

### Title
Missing registry-state guard on `stake()` allows delayed transaction to permanently lock user funds — (File: src/ConfidencePool.sol)

### Summary
`stake()` accepts no registry-state bound parameter. A transaction submitted while the registry is in a pre-risk state (`NOT_DEPLOYED`, `NEW_DEPLOYMENT`, `ATTACK_REQUESTED`) — where `withdraw()` is still open — can sit in the mempool and execute after the registry has transitioned to `UNDER_ATTACK`. The user's own `stake()` call then triggers `_markRiskWindowStart()` via `_observePoolState()`, permanently sealing the one-way `riskWindowStart` latch and irrevocably disabling `withdraw()`. The user's principal is locked until pool resolution, which may be months away, and is fully at risk of loss under a CORRUPTED outcome.

### Finding Description
`stake()` calls `_observePoolState()` unconditionally before recording the deposit:

```solidity
function stake(uint256 amount) external nonReentrant whenPoolNotPaused {
    ...
    _assertDepositsAllowed(_observePoolState());   // line 227
    ...
}
```

`_observePoolState()` contains the one-way risk-window latch:

```solidity
if (riskWindowStart == 0 && _isActiveRiskState(state)) {
    _markRiskWindowStart();   // sets riskWindowStart = block.timestamp (capped at expiry)
}
```

`_assertDepositsAllowed` permits `UNDER_ATTACK` deposits (by design, per DESIGN.md §3), so the call does not revert. The deposit is recorded, but `riskWindowStart` is now non-zero.

`withdraw()` gates on the one-way latch:

```solidity
if (
    riskWindowStart != 0                          // one-way; never reset
        || (state != NOT_DEPLOYED && state != NEW_DEPLOYMENT && state != ATTACK_REQUESTED)
) {
    revert WithdrawsDisabled();
}
```

Both conditions are now true. The user cannot withdraw regardless of any future registry rewind (the latch is explicitly designed to survive upstream state reversals).

DESIGN.md §3 states "A staker who does not want this can read the live registry state before staking," but this only helps when the state has not changed between the read and execution. It does not address the mempool-delay scenario and does not document the TX-delay race as an accepted trade-off.

### Impact Explanation
A user who submitted `stake()` in good faith during a pre-risk window — intending to retain the `withdraw()` escape hatch — has their principal permanently locked until pool resolution. Resolution can be deferred up to `expiry + MODERATOR_CORRUPTED_GRACE` (pool expiry plus 180 days). Under a CORRUPTED outcome the user loses their entire principal to `recoveryAddress`. The user earned near-zero bonus (k=2 entry floored at `riskWindowStart ≈ now` collapses `(T − entry)²` to ~0) while bearing full downside risk they did not intend to accept.

### Likelihood Explanation
The registry transition from `NEW_DEPLOYMENT` / `ATTACK_REQUESTED` to `UNDER_ATTACK` is a discrete, publicly observable on-chain event. Any period of elevated gas prices, block congestion, or validator non-inclusion near that transition is sufficient. The pool is designed for broad staker participation (unprivileged stakers are a named actor class), and many stakers cannot be assumed to understand mempool mechanics or how to invalidate a pending transaction. The risk is continuous and probabilistic: every pending `stake()` transaction near a registry-state boundary is exposed.

### Recommendation
Add a `maxState` (or equivalent deadline) parameter to `stake()` that reverts if the observed registry state has advanced beyond what the caller intended:

```solidity
function stake(uint256 amount, IAttackRegistry.ContractState maxAllowedState) external nonReentrant whenPoolNotPaused {
    ...
    IAttackRegistry.ContractState state = _observePoolState();
    if (uint8(state) > uint8(maxAllowedState)) revert RegistryStateTooAdvanced();
    _assertDepositsAllowed(state);
    ...
}
```

This mirrors the `deadline` parameter on AMM swap functions and gives users a trustless guarantee that their stake only executes under the registry conditions they evaluated.

### Proof of Concept

1. Registry is in `NEW_DEPLOYMENT`. Alice checks the state off-chain and submits `stake(1000e18)`, intending to retain the ability to `withdraw()` if the agreement goes live.
2. Gas prices spike. Alice's transaction sits in the mempool.
3. The registry transitions to `UNDER_ATTACK`. A separate pool interaction (or `pokeRiskWindow()`) may or may not have already sealed `riskWindowStart`; it does not matter for the outcome.
4. Alice's `stake()` executes. `_observePoolState()` observes `UNDER_ATTACK` and calls `_markRiskWindowStart()`, setting `riskWindowStart = block.timestamp`. [1](#0-0) 
5. `_assertDepositsAllowed(UNDER_ATTACK)` does not revert (UNDER_ATTACK is explicitly permitted). [2](#0-1) 
6. Alice's 1000e18 tokens are recorded as eligible stake.
7. Alice calls `withdraw()`. The check `riskWindowStart != 0` is true (set in step 4). `WithdrawsDisabled` is reverted. [3](#0-2) 
8. Alice's principal is locked until the pool resolves. If the outcome is CORRUPTED, her entire 1000e18 is swept to `recoveryAddress` — a total loss she never consented to. [4](#0-3)
