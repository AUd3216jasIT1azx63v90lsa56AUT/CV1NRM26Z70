Audit Report

## Title
Division-by-Zero in `buyerAssetsToUnits` on Zero-Price Sell Offer Causes Bundler DoS - (File: src/periphery/TakeAmountsLib.sol)

## Summary
When a sell offer has `tick=0` and the market's `settlementFee=0`, `TickLib.tickToPrice(0)` returns `0`, making `buyerPrice=0`. `buyerAssetsToUnits` then executes `targetBuyerAssets.mulDivDown(WAD, 0)`, which is `(targetBuyerAssets * WAD) / 0` and panics. Because the `buyerAssetsToUnits` call in `buyWithAssetsTargetAndWithdrawCollateral` sits outside the `try/catch` block, the entire bundler transaction reverts unconditionally.

## Finding Description
`TickLib.tickToPrice(0)` returns `0`, confirmed by the Certora spec rule `tickToPriceIsZeroAtZero` in `certora/specs/TickToPrice.spec` lines 34–36.

In `buyerAssetsToUnits` (`src/periphery/TakeAmountsLib.sol` lines 22–29), with `offer.buy=false`, `tick=0`, and `settlementFee=0`:
- `offerPrice = TickLib.tickToPrice(0) = 0`
- `sellerPrice = offerPrice = 0` (sell-offer branch: `offer.buy ? offerPrice - settlementFee : offerPrice`)
- `buyerPrice = sellerPrice + settlementFee = 0 + 0 = 0`
- `require(0 <= WAD)` passes
- `return targetBuyerAssets.mulDivDown(WAD, 0)` → `(targetBuyerAssets * WAD) / 0` → Solidity 0.8 division-by-zero panic

`mulDivDown` is implemented as plain `(x * y) / d` with no `unchecked` wrapper (`src/libraries/UtilsLib.sol` lines 29–31), so `d=0` panics unconditionally. This is confirmed by `testMulDivDownDivisionByZero` in `test/UtilsLibTest.sol` lines 58–63, which expects `stdError.divisionError` when `d=0`.

In `buyWithAssetsTargetAndWithdrawCollateral` (`src/periphery/MidnightBundles.sol` lines 208–214), the `TakeAmountsLib.buyerAssetsToUnits(...)` call is evaluated as an argument to `min(...)` **before** the `try` at line 215. A panic inside `buyerAssetsToUnits` is not caught and propagates to the caller.

The `NoDivisionByZero.spec` only verifies `Midnight.sol` methods and explicitly does not cover `TakeAmountsLib` or `MidnightBundles`. The existing `TakeAmountsTest` fuzz tests bound `tick` away from zero (`tick = bound(tick, 4, ...)` at lines 99, 121, 145, 168), so this path is never exercised.

## Impact Explanation
Any call to `buyWithAssetsTargetAndWithdrawCollateral` that includes a sell offer with `tick=0` and a market where `settlementFee=0` will revert unconditionally when `targetBuyerAssets > 0`. The bundler's asset-denominated targeting functionality is completely unavailable for zero-price offers. Funds are not permanently frozen (the `pullToken` is unwound on revert), so the impact is a targeted DoS of the periphery contract.

## Likelihood Explanation
The preconditions are minimal and fully attacker-controllable. Any unprivileged user can post a sell offer with `tick=0` — `tickToPrice` only requires `tick <= MAX_TICK`, and `tick=0` satisfies this. Settlement fee of `0` is a common and expected market configuration, explicitly used across the bundler test suite (e.g., `MidnightBundlesTest.sol` lines 261, 297, 401, 596, 721, 857). The condition is stable and repeatable: every call to `buyWithAssetsTargetAndWithdrawCollateral` targeting such an offer will revert.

## Recommendation
Add a guard in `buyerAssetsToUnits` (and symmetrically in `sellerAssetsToUnits`) to revert with a descriptive error when `buyerPrice == 0` (or `sellerPrice == 0`), before reaching the `mulDivDown` call. For example:

```solidity
require(buyerPrice > 0, ZeroPrice());
```

Alternatively, add a `tick > 0` precondition check at the top of both functions, since `tickToPrice(0) == 0` is a known degenerate case confirmed by the Certora spec.

Additionally, update the `TakeAmountsTest` fuzz bounds to include `tick=0` (or at minimum `tick=1`) to catch this class of regression.

## Proof of Concept
Minimal Foundry test:
```solidity
function testBuyerAssetsToUnitsZeroTick() public {
    // Set all settlement fees to 0.
    for (uint256 i; i <= 6; i++) {
        midnight.setMarketSettlementFee(id, i, 0);
    }
    offer.tick = 0; // tickToPrice(0) == 0
    offer.buy = false;

    // buyerAssetsToUnits panics: mulDivDown(WAD, 0)
    vm.expectRevert(stdError.divisionError);
    TakeAmountsLib.buyerAssetsToUnits(address(midnight), id, offer, 1e18);
}
```

For the bundler-level DoS, wrap the above in a `buyWithAssetsTargetAndWithdrawCollateral` call with a zero-tick sell offer and `targetBuyerAssets > 0`; the entire transaction reverts with a division-by-zero panic rather than the expected `OutOfOffers` or a successful fill.
