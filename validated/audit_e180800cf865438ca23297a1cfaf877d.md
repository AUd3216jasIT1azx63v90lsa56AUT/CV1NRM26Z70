### Title
EOA Address Accepted as `oracle` in `CollateralParams` Permanently Locks Borrower Collateral ŌĆö (File: `src/Midnight.sol`)

---

### Summary

`touchMarket` creates a market without validating that each `collateralParams[i].oracle` is a deployed contract. If an EOA address is used as an oracle ŌĆö whether by mistake or by a malicious market creator ŌĆö every subsequent call that invokes `IOracle(oracle).price()` reverts. Because markets are immutable once created, any borrower who accumulates debt in such a market has their collateral permanently locked: `withdrawCollateral` reverts on the health check and `liquidate` reverts on the oracle call.

---

### Finding Description

**Vulnerability class**: Input validation ŌĆö address that must be a contract is not checked for code existence.

**Root cause ŌĆö `touchMarket` (src/Midnight.sol lines 755ŌĆō791)**

The loop that validates each `CollateralParams` entry checks only the collateral token sort order, the LLTV allowlist, and the `maxLif` formula. It never checks whether `market.collateralParams[i].oracle` has any deployed code:

```solidity
for (uint256 i = 0; i < market.collateralParams.length; i++) {
    address collateralToken = market.collateralParams[i].token;
    require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
    uint256 lltv = market.collateralParams[i].lltv;
    require(isLltvAllowed(lltv), LltvNotAllowed());
    require(
        market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_LOW)
            || market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_HIGH),
        InvalidMaxLif()
    );
    previousCollateralToken = collateralToken;
}
``` [1](#0-0) 

The `oracle` field of `CollateralParams` is never touched during market creation.

**Downstream failure ŌĆö `isHealthy` (src/Midnight.sol line 953)**

```solidity
uint256 price = IOracle(collateralParam.oracle).price();
``` [2](#0-1) 

A low-level call to an EOA returns empty data and `success = false`; Solidity's high-level call reverts with no return data. This causes `isHealthy` to revert, which in turn causes `withdrawCollateral` to revert.

**Downstream failure ŌĆö `liquidate` (src/Midnight.sol line 610)**

```solidity
uint256 price = IOracle(_collateralParam.oracle).price();
``` [3](#0-2) 

Same revert path: liquidators cannot seize collateral or realize bad debt.

**Market immutability seals the fate**

Once `touchMarket` stores the market (via `IdLib.storeInCode`), the oracle address is permanently encoded. There is no setter for per-market oracle addresses. [4](#0-3) 

---

### Impact Explanation

Any borrower who supplies collateral and takes on debt in a market whose oracle is an EOA:

- **Cannot withdraw collateral** ŌĆö `withdrawCollateral` calls `isHealthy`, which calls `oracle.price()`, which reverts.
- **Cannot be liquidated** ŌĆö `liquidate` calls `oracle.price()` directly, which reverts.
- **Collateral is permanently locked** in the `Midnight` contract with no recovery path.

This constitutes a permanent, irrecoverable loss of the borrower's collateral assets. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

Two realistic paths exist:

1. **Accidental misconfiguration**: A user creating a market copy-pastes their own wallet address or another EOA as the oracle. Markets are created permissionlessly with no UI guardrails enforced at the contract level.
2. **Malicious market creation**: An attacker deliberately creates a market with an EOA oracle, makes it appear legitimate (correct loan token, valid LLTV/maxLif), and lures lenders/borrowers into it. Once borrowers have debt, their collateral is trapped.

No privileged access is required. `touchMarket` is callable by anyone, including indirectly through `take`, `repay`, `supplyCollateral`, `withdrawCollateral`, `withdraw`, and `liquidate`. [7](#0-6) 

---

### Recommendation

Add an `extcodesize` check for each oracle address inside the `touchMarket` validation loop:

```solidity
address oracle = market.collateralParams[i].oracle;
uint256 size;
assembly { size := extcodesize(oracle) }
require(size > 0, OracleNotAContract());
```

Apply the same check to `enterGate` and `liquidatorGate` when they are non-zero, as an EOA in either role also causes irreversible market dysfunction (all credit increases or all liquidations revert permanently). [1](#0-0)

### Proof of Concept

**Setup**

- `alice` is an EOA (no deployed code).
- `bob` is a normal user.

**Steps**

1. Alice calls `touchMarket` with a `Market` struct where `collateralParams[0].oracle = address(alice)`. The call succeeds ŌĆö no oracle code-size check exists.
2. Bob calls `supplyCollateral` on this market, depositing 1000 WETH as collateral. Succeeds (oracle is not called here).
3. Bob calls `take` as seller (increasing debt). Succeeds if Bob has no prior debt (oracle not called when `debt == 0` before the take, and the health check at line 476 calls `isHealthy` ŌĆö which **does** call the oracle if Bob now has debt). The take reverts at the final health check.

   *Alternatively*: Bob takes a buy offer (becoming a buyer/lender), then a separate borrower takes a sell offer against Bob's credit, giving the borrower debt. The borrower's collateral is now locked.

4. Any attempt to call `withdrawCollateral` for a borrower with debt reverts at `isHealthy` ŌåÆ `oracle.price()`.
5. Any attempt to call `liquidate` reverts at the oracle loop.
6. The borrower's collateral is permanently locked with no on-chain recovery mechanism.

**Expected outcome**: `withdrawCollateral` and `liquidate` both revert with a low-level call failure on the EOA oracle address. The collateral balance in `position[id][borrower].collateral[0]` remains non-zero indefinitely. [8](#0-7) [9](#0-8)
