### Title
Continuous Fee on Withdrawn Credit Is Silently Discarded Instead of Credited to the Protocol — (`src/Midnight.sol`)

### Summary

In `withdraw()`, the proportional future continuous fee (`pendingFeeDecrease`) on the withdrawn credit is removed from the lender's `pendingFee` but is never added to `continuousFeeCredit` and is never deducted from the token transfer. The lender receives the full `units` of loan tokens, but the protocol's stated invariant is that a lender's face value is `credit - pendingFee`. The future fee is silently destroyed, causing a direct, permanent revenue loss to the protocol on every pre-maturity withdrawal.

### Finding Description

The contract explicitly documents the accounting invariant at line 55:

> "In the absence of bad debt realizations, the face value of a lender's position is `credit - pendingFee`."

`pendingFee` is set at position-open time to the **total** continuous fee owed over the position's lifetime: [1](#0-0) 

`_updatePosition` accrues only the **elapsed** portion of `pendingFee` into `continuousFeeCredit`: [2](#0-1) 

After `_updatePosition` returns, the remaining `pendingFee` is the **future** fee owed from `now` to `maturity`. In `withdraw()`, this future fee is computed proportionally and removed from the position: [3](#0-2) 

But `pendingFeeDecrease` is **never routed anywhere**. It is not added to `continuousFeeCredit`, and the transfer sends the full `units` to the receiver: [4](#0-3) 

Compare this to `_updatePosition`, where the accrued fee is correctly credited: [5](#0-4) 

The `claimContinuousFee` function draws from `withdrawable` to pay the fee claimer: [6](#0-5) 

Because `pendingFeeDecrease` is never added to `continuousFeeCredit`, the fee claimer permanently loses that revenue. The lender receives `units` tokens when their face value entitles them to only `units - pendingFeeDecrease`.

The Certora spec `certora/specs/BalanceEffects.spec` confirms this is not caught by formal verification — it only asserts `continuousFeeCredit` increases by the **accrued** fee (`userFee`), not by `pendingFeeDecrease`: [7](#0-6) 

### Impact Explanation

Every pre-maturity withdrawal causes a permanent loss of `pendingFeeDecrease` tokens to the protocol. The lender receives more than their face value (`credit - pendingFee`). The fee claimer's `continuousFeeCredit` is understated by the sum of all `pendingFeeDecrease` values across all early withdrawals. In markets with high continuous fees and long maturities, this loss is proportionally large. There is no recovery mechanism.

### Likelihood Explanation

Any lender can trigger this by calling `withdraw()` at any time before maturity. No special privileges, flash loans, or external conditions are required. The incentive is clear: a lender who withdraws early avoids paying the remaining continuous fee while receiving the full token amount. This is a normal, permissionless user action.

### Recommendation

Add `pendingFeeDecrease` to `continuousFeeCredit` and deduct it from the token transfer so the future fee stays in the contract for the fee claimer:

```solidity
// In withdraw(), after computing pendingFeeDecrease:
_position.pendingFee -= pendingFeeDecrease;
_marketState.continuousFeeCredit += pendingFeeDecrease; // ADD: credit future fee to protocol

_position.credit -= UtilsLib.toUint128(units);
_marketState.withdrawable -= UtilsLib.toUint128(units);
_marketState.totalUnits -= UtilsLib.toUint128(units);

// Transfer only the lender's face value, not the full units
SafeTransferLib.safeTransfer(market.loanToken, receiver, units - pendingFeeDecrease);
```

This preserves `pendingFeeDecrease` inside the contract's `withdrawable` pool, correctly attributing it to `continuousFeeCredit` so the fee claimer can claim it.

### Proof of Concept

**Setup:** Market with `continuousFee = MAX_CONTINUOUS_FEE`, maturity = 360 days. Lender opens a position of 1,000,000 units at T=0. `pendingFee ≈ 114,155` units (≈11.4% total fee).

**Step 1 (T = 180 days):** Borrower repays 1,000,000 units → `withdrawable = 1,000,000`.

**Step 2:** Lender calls `withdraw(1,000,000 - accruedFee, lender, lender)` (withdraws all post-accrual credit).
- `_updatePosition` accrues ~57,077 units → `continuousFeeCredit = 57,077`, lender `credit ≈ 942,923`, `pendingFee ≈ 57,077`.
- `pendingFeeDecrease = 57,077` (proportional future fee on full withdrawal).
- `pendingFee → 0`, `credit → 0`.
- `withdrawable = 1,000,000 - 942,923 = 57,077`.
- Lender receives **942,923 tokens** (their face value should be `942,923 - 57,077 = 885,846`).

**Step 3:** Fee claimer calls `claimContinuousFee(57,077)` → receives 57,077 tokens. `withdrawable → 0`.

**Result:** Protocol received 57,077 tokens. It should have received **114,154 tokens** (the full `pendingFee`). The lender extracted **57,077 extra tokens** that belonged to the protocol. The `pendingFeeDecrease` of 57,077 was silently destroyed.
