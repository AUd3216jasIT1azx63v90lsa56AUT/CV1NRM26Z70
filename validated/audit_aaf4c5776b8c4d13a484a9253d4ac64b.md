### Title
Continuous Fee Accrual Rounds to Zero, Enabling Lenders to Avoid Paying Continuous Fees via Frequent `updatePosition` Calls — (`src/Midnight.sol`)

---

### Summary

The `updatePositionView` function computes the continuous fee accrued since the last update using integer division. Because `updatePosition` is permissionless and callable by anyone for any user, a lender can call it on their own position every second (every block on L2). For positions below a calculable credit threshold, the per-second fee rounds to zero on every call, and the protocol never collects the continuous fee.

---

### Finding Description

**Root cause — line 814–815 of `src/Midnight.sol`:**

```solidity
uint128 fee = _lastAccrual < market.maturity
    ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
    : 0;
``` [1](#0-0) 

The fee for a given update interval is:

```
fee = pendingFee × (accrualEnd − lastAccrual) / (maturity − lastAccrual)
```

`mulDivDown` performs plain integer division (no rounding up). [2](#0-1) 

If `updatePosition` is called every second, `accrualEnd − lastAccrual = 1`. For the fee to be non-zero, the numerator must satisfy:

```
pendingFee × 1 ≥ maturity − lastAccrual
```

For a 1-year market: `maturity − lastAccrual ≈ 31,536,000`.

**`pendingFee` is set at `take` time:**

```solidity
uint128 buyerPendingFeeIncrease =
    UtilsLib.toUint128(buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD));
``` [3](#0-2) 

With `MAX_CONTINUOUS_FEE = 0.01e18 / 365 days ≈ 317,097,919` (WAD/second): [4](#0-3) 

For a USDC (6-decimal) position of **1,000 USDC** (`credit = 1e9`) in a 1-year market at max fee:

```
pendingFee = 1e9 × 317,097,919 × 31,536,000 / 1e18 ≈ 1e7
fee/second = 1e7 × 1 / 31,536,000 ≈ 0.317  →  rounds to 0
```

The rounding-to-zero threshold for a 1-year market at max fee is:

```
credit × 0.01 < 31,536,000  →  credit < 3,153,600,000 units  →  < 3,153 USDC
```

Every call to `updatePosition` sets `_position.lastAccrual = uint128(block.timestamp)`: [5](#0-4) 

So the next call one second later again computes `fee = 0`. The `continuousFeeCredit` accumulator is never incremented, and the lender's credit is never reduced.

**`updatePosition` is fully permissionless:**

```solidity
function updatePosition(Market memory market, address user) external returns (uint128, uint128, uint128) {
    bytes32 id = toId(market);
    require(marketState[id].tickSpacing > 0, MarketNotCreated());
    return _updatePosition(market, id, user);
}
``` [6](#0-5) 

No authorization check. Any lender can call this on their own position every second.

The protocol's own inline comment acknowledges the risk: *"pendingFee updates are rounded in favor of the user. It could lead to fees manipulations too."* [7](#0-6) 

---

### Impact Explanation

A lender with a USDC position below ~3,153 USDC (in a 1-year market at max continuous fee) can call `updatePosition` on their own position every second and pay **zero** continuous fees for the entire market lifetime. The protocol's `continuousFeeCredit` is never incremented for that position, so the fee claimer loses all continuous fee revenue from that lender. The lender retains full credit without the fee deduction.

The impact scales with the number of small-position lenders and the total continuous fee rate. For 6-decimal tokens (USDC, USDT), the threshold is in the thousands of dollars — a realistic position size.

---

### Likelihood Explanation

- **Attacker preconditions**: Hold any credit position in a market with a 6-decimal loan token and a continuous fee set. No privileged access required.
- **Trigger**: Call `updatePosition(market, self)` once per second (or once per block on L2 where block time ≥ 1 second).
- **Cost vs. benefit**: On Arbitrum (L2), a simple external call costs ~$0.0001–$0.001. Avoiding 10 USDC/year in fees on a 1,000 USDC position requires 31,536,000 calls/year — economically marginal on mainnet but viable on L2 for larger positions near the threshold. For positions just below the threshold (e.g., 3,000 USDC, annual fee ~30 USDC), the economics favor the attack on L2.
- **Self-beneficial**: Unlike pure griefing, the lender directly benefits (retains credit, avoids fee deduction), providing clear economic motivation.

---

### Recommendation

Apply a minimum accrual interval (e.g., require `block.timestamp - lastAccrual >= MIN_ACCRUAL_INTERVAL`) before updating `lastAccrual`, or accumulate fractional fees using a fixed-point remainder carried across calls (similar to how Compound/Aave use per-second index accumulators). Alternatively, use `mulDivUp` for the fee computation so that even a 1-second interval charges at least 1 unit, preventing the zero-rounding exploit.

---

### Proof of Concept

**Setup:**
- Market: USDC loan token (6 decimals), 1-year maturity, `continuousFee = MAX_CONTINUOUS_FEE`
- Lender Alice: 1,000 USDC credit → `pendingFee ≈ 1e7`

**Steps:**
1. Alice calls `take` to acquire 1,000 USDC credit. `pendingFee = 1e7`, `lastAccrual = T0`.
2. At `T0 + 1`: Alice calls `updatePosition(market, alice)`.
   - `fee = 1e7 × 1 / 31,535,999 = 0` (rounds down).
   - `continuousFeeCredit += 0`. `lastAccrual = T0 + 1`. Credit unchanged.
3. Repeat every second for the entire year.
4. At maturity: Alice's `pendingFee` is still `1e7`. She has paid **0** in continuous fees. The protocol has collected **0** from Alice.
5. Alice calls `withdraw` to redeem full credit. `_updatePosition` is called; `accrualEnd = maturity`, `lastAccrual = T_last` (1 second ago), `fee = 1e7 × 1 / 1 = 1e7` — but this is only the last second's fee, not the full year's fee.

**Expected (correct) behavior:** Alice should have paid `1e7` total in fees (10 USDC) distributed over the year.
**Actual behavior:** Alice pays at most `1e7` in the final second, but all prior seconds contribute 0. The protocol loses up to 10 USDC per 1,000 USDC lender per year for positions below the threshold.
