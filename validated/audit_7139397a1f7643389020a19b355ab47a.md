Audit Report

## Title
Debt Increase Permitted at Exact Maturity Boundary via `<=` Operator in `CannotIncreaseDebtPostMaturity` Check - (File: src/Midnight.sol)

## Summary
The guard at `src/Midnight.sol` line 391 uses `block.timestamp <= offer.market.maturity`, permitting `sellerDebtIncrease > 0` when `block.timestamp == maturity`. At that same timestamp, `UtilsLib.zeroFloorSub` yields `timeToMaturity = 0`, so the new lender credit accrues zero continuous fee. The post-maturity liquidation check at line 622 uses strict `block.timestamp > market.maturity`, meaning post-maturity liquidation is simultaneously unavailable, and normal-mode liquidation is blocked for any healthy position, creating a one-block window of liquidation immunity for the borrower.

## Finding Description

**Root cause — `UtilsLib.zeroFloorSub` (`src/libraries/UtilsLib.sol` line 24):**
```solidity
z := mul(gt(x, y), sub(x, y))
```
`gt(maturity, block.timestamp)` evaluates to `0` when `block.timestamp == maturity`, so `timeToMaturity = 0`. [1](#0-0) 

**Guard at `src/Midnight.sol` line 391:**
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
At `block.timestamp == maturity`, the left operand is `true`, so the `require` passes unconditionally regardless of `sellerDebtIncrease`. The debt increase at line 414 (`sellerPos.debt += sellerDebtIncrease`) executes. [2](#0-1) [3](#0-2) 

**`timeToMaturity` computation at line 359:**
```solidity
uint256 timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp);
```
Yields `0` at exactly maturity. [4](#0-3) 

**Zero continuous-fee consequence (lines 385–386):**
```solidity
uint128 buyerPendingFeeIncrease =
    UtilsLib.toUint128(buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD));
```
With `timeToMaturity = 0`, `buyerPendingFeeIncrease = 0`. The lender receives no continuous-fee obligation for credit extended at maturity. [5](#0-4) 

**Post-maturity liquidation guard at line 622:**
```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
```
Uses strict `>`. At `block.timestamp == maturity`, post-maturity liquidation is unavailable. Normal-mode liquidation requires `originalDebt > maxDebt` (unhealthy position). A borrower who passes the health check at line 476 after increasing debt at exactly maturity is immune to both liquidation paths for that block. [6](#0-5) [7](#0-6) 

**Test coverage gap:** All existing post-maturity tests warp to `market.maturity + 1`, not `market.maturity` exactly, leaving the boundary untested. [8](#0-7) 

**Exploit flow:**
1. Attacker identifies a market whose `maturity` falls on a reachable block timestamp.
2. At `block.timestamp == maturity`, attacker submits a `take` with `sellerDebtIncrease > 0`.
3. Line 391 passes (`block.timestamp <= maturity` is `true`).
4. Debt is increased; new lender credit carries `pendingFee = 0`.
5. Health check at line 476 passes (position is healthy with sufficient collateral).
6. Post-maturity liquidation (line 622) is blocked for this block (`block.timestamp > maturity` is `false`).
7. Normal liquidation is also blocked (position is healthy).
8. Next block, post-maturity liquidation becomes available, but the fee-free credit record has already been committed.

## Impact Explanation
The protocol invariant that debt increase is forbidden at or after maturity is violated at the exact maturity boundary. The lender providing credit at maturity receives `buyerPendingFeeIncrease = 0`, constituting a direct accounting inconsistency: the credit is recorded with no fee obligation despite the protocol's continuous-fee model. The borrower gains a one-block window of liquidation immunity at a timestamp where the position is immediately overdue. The state inconsistency is permanent: the fee-free credit record persists after the block. While the continuous fee for zero remaining time is mathematically zero, the protocol's intent is to block all new debt at or after maturity — the `<=` boundary permits a class of transactions the protocol explicitly names `CannotIncreaseDebtPostMaturity` as forbidden.

## Likelihood Explanation
The precondition is a single deterministic timestamp (`block.timestamp == maturity`). Any borrower or MEV searcher with standard transaction submission capability can target this block. No privileged access is required. The offer need only have a non-expired `expiry` covering that timestamp and sufficient collateral pre-supplied. The condition is repeatable for every market whose maturity falls on a reachable block timestamp.

## Recommendation
Change the guard at line 391 from `<=` to `<`:
```solidity
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
This makes the debt-increase boundary consistent with the post-maturity liquidation boundary (`block.timestamp > market.maturity` becomes available at the same timestamp that debt increase is blocked). Add a test case warping to exactly `market.maturity` (not `market.maturity + 1`) to cover this boundary.

## Proof of Concept
```solidity
function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;
    uint256 timestamp = market.maturity; // exactly at maturity, not +1
    vm.warp(timestamp);
    borrowerOffer.expiry = timestamp;
    borrowerOffer.maxUnits = units;
    deal(address(loanToken), lender, units);
    collateralize(market, borrower, units);

    // This should revert with CannotIncreaseDebtPostMaturity but does NOT
    take(units, lender, borrowerOffer);

    // Debt was increased at maturity with zero pendingFee
    assertGt(midnight.debtOf(id, borrower), 0);
    assertEq(midnight.pendingFee(id, lender), 0); // fee-free credit

    // Post-maturity liquidation is blocked at this block
    vm.expectRevert(IMidnight.NotLiquidatable.selector);
    midnight.liquidate(market, 0, 0, units, borrower, true, lender, address(0), "");
}
```
