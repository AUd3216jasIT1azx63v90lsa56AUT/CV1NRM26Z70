Audit Report

## Title
Reentrancy in `repay` callback allows authorized operator to drain borrower collateral via transient debt zeroing - (File: src/Midnight.sol)

## Summary
The `repay` function reduces `position[id][onBehalf].debt` at line 508 before invoking the `IRepayCallback.onRepay` external call at line 516 and before pulling loan tokens at line 520. During the callback window, the position's debt is zero while no loan tokens have been received. An authorized operator acting as the callback contract can re-enter `withdrawCollateral`, which passes the `isHealthy` check because `debt == 0` causes the oracle loop to be skipped and returns `0 >= 0 = true`. The attacker funds the repayment using proceeds from the drained collateral, completing the attack atomically with zero upfront capital.

## Finding Description

**Root cause:** `repay` writes `position[id][onBehalf].debt -= units` before the external callback and before `safeTransferFrom`. The health invariant — "collateral cannot be withdrawn unless debt is backed" — is violated because `isHealthy` evaluates against a transiently zeroed debt that has not yet been backed by received loan tokens.

**Code path — `repay`** (`src/Midnight.sol:502-521`): [1](#0-0) 

- Line 505: auth check passes if `isAuthorized[onBehalf][msg.sender]`
- Line 508: debt zeroed **before** callback or payment
- Line 516: external `onRepay` callback fires with debt already zero
- Line 520: loan tokens pulled **last**

**Code path — `withdrawCollateral`** (`src/Midnight.sol:549-573`): [2](#0-1) 

- Line 556: auth check passes (X is authorized)
- Line 568: `isHealthy` passes because `debt == 0` at this point
- Line 572: collateral transferred to attacker

**`isHealthy` short-circuits when debt is zero:** [3](#0-2) 

When `debt == 0`, the oracle loop is never entered, `maxDebt` stays 0, and `return maxDebt >= debt` evaluates to `0 >= 0 = true` regardless of collateral state.

**No reentrancy guard exists on `repay`.** The `LIQUIDATION_LOCK_SLOT` transient lock: [4](#0-3) 

is only set during `take` callbacks at line 444: [5](#0-4) 

It is never set during `repay`. The TOKEN SAFETY REQUIREMENTS only exclude reentrancy through token `transfer`/`transferFrom` calls: [6](#0-5) 

The `onRepay` callback is an explicit protocol-level callback, not a token transfer, so this exclusion does not apply.

**Exploit flow:**

1. Victim A authorizes malicious contract X via `setIsAuthorized` or via `EcrecoverAuthorizer.setIsAuthorized` (permissionless relay — anyone can submit a signed authorization): [7](#0-6) 

2. X calls `repay(market, D, A, X, data)` — X is both `msg.sender` and `callback`.
3. Line 508 zeros A's debt. Line 516 calls `X.onRepay(...)`.
4. Inside `onRepay`, X re-enters `withdrawCollateral(market, idx, C_total, A, X)`:
   - Auth check passes (X is authorized for A).
   - Health check passes (debt is 0).
   - All of A's collateral is transferred to X.
5. X swaps collateral for D loan tokens via a DEX, returns `CALLBACK_SUCCESS`.
6. Line 520: `safeTransferFrom(loanToken, X, midnight, D)` succeeds — funded by the swap.

**Net result:** A's debt is 0 and A's collateral is 0. X extracted `value(C) − D` profit with zero upfront capital.

**Why existing checks fail:**
- Authorization check in `withdrawCollateral` passes because X is legitimately authorized.
- `isHealthy` passes because debt was zeroed at line 508 before any loan tokens were received.
- `LIQUIDATION_LOCK_SLOT` is not set during `repay`, only during `take`.
- Token reentrancy guard does not cover the explicit `onRepay` callback.

## Impact Explanation
An authorized operator can drain 100% of a borrower's overcollateralized position in a single atomic transaction with zero upfront capital. The invariant "collateral cannot be withdrawn outside health/liquidation rules" is broken because the health check is evaluated against a transiently zeroed debt that has not yet been backed by received loan tokens. This constitutes direct theft of user assets — a critical in-scope impact. The overcollateralization premium (`value(C) − D`) is extracted entirely by the attacker.

## Likelihood Explanation
**Preconditions:**
1. A must have authorized X (e.g., to allow automated repayment, or via a malicious UI/integrator).
2. A must have a healthy, overcollateralized position.
3. The collateral token must be swappable for loan tokens within one transaction (true for any token with DEX liquidity).

All three are routine conditions. The `EcrecoverAuthorizer.setIsAuthorized` relay is permissionless — any party can submit a signed authorization. The attack is repeatable across all markets and all authorized operators. A malicious contract deployed as a "repayment helper" or "position manager" can exploit this against any user who grants it authorization.

## Recommendation
Apply one of the following fixes:

1. **Checks-Effects-Interactions ordering:** Move the debt reduction and `withdrawable` increase to **after** the callback and `safeTransferFrom`, so the position's debt is only zeroed once tokens have been received.

2. **Reentrancy lock during repay callback:** Set a transient lock (analogous to `LIQUIDATION_LOCK_SLOT`) on the borrower's position before invoking `onRepay` and clear it after, then add a guard in `withdrawCollateral` that reverts if the lock is set.

3. **Post-callback health assertion:** After the callback and token pull, assert that the position is either debt-free or healthy — mirroring the pattern used in `take` at line 476.

Option 1 is the cleanest and most consistent with standard CEI discipline.

## Proof of Concept

**Minimal Foundry test outline:**

```solidity
// 1. Deploy Midnight, create market, victim A supplies collateral C and borrows D.
// 2. Deploy MaliciousOperator X implementing IRepayCallback.
// 3. A calls setIsAuthorized(address(X), true).
// 4. X calls midnight.repay(market, D, A, address(X), "").
//    - Inside onRepay: X calls midnight.withdrawCollateral(market, 0, C_total, A, address(X)).
//      Assert: succeeds (isHealthy returns true because debt == 0).
//    - X swaps C for D loan tokens via mock DEX.
//    - Returns CALLBACK_SUCCESS.
// 5. Assert: A.debt == 0, A.collateral == 0, X holds value(C) - D profit.
// 6. Assert: attack completed with zero X token balance at start.
```

The test is fully deterministic on a local fork with a mock oracle and mock DEX. No external dependencies beyond standard ERC-20 tokens are required.

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];
                uint256 price = IOracle(collateralParam.oracle).price();
                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                    .mulDivDown(collateralParam.lltv, WAD);
                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
    }
