### Title
`contributeBonus()` Incorrectly Applies Stake-Deposit Guard, Blocking Bonus Contributions During `PROMOTION_REQUESTED` - (File: src/ConfidencePool.sol)

### Summary
`contributeBonus()` calls `_assertDepositsAllowed(_observePoolState())`, a guard function designed exclusively to prevent late-join gaming of stake deposits. This guard reverts with `StakingClosed()` during `PROMOTION_REQUESTED`, `PRODUCTION`, and `CORRUPTED` registry states. Bonus contributions carry no timing-based gaming risk, so the guard's rationale does not apply to them — yet the same restriction is imposed, causing `contributeBonus()` to revert in states where it should succeed.

### Finding Description
`_assertDepositsAllowed` is documented and designed to block stake deposits during `PROMOTION_REQUESTED` because a late staker could enter in the closing window and earn a disproportionate bonus share relative to their risk exposure. Its natspec explicitly frames the asymmetry as being "about deposit *timing*": [1](#0-0) 

`contributeBonus()` calls this same guard: [2](#0-1) 

Bonus contributors receive no share of the bonus pool — they only add to it. There is no timing-based gaming vector for bonus contributions: a contributor cannot shift their own payout by choosing when to contribute. The `_assertDepositsAllowed` guard's entire rationale (preventing a late joiner from gaming the k=2 time-weighted formula) is inapplicable to `contributeBonus()`. Calling it here is the direct analog to the external report's defect: using a more-restricted function (one with a guard designed for a different operation) when a less-restricted path should be used.

The protocol documentation explicitly states bonus contributions are permissionless with no state-based restriction: [3](#0-2) [4](#0-3) 

### Impact Explanation
During `PROMOTION_REQUESTED` — the closing stretch of the active-risk window, when stakers are already locked in and cannot withdraw — any call to `contributeBonus()` reverts with `StakingClosed()`. A sponsor or third party wishing to top up the bonus pool to reward stakers who bore risk during the attack window is completely blocked from doing so. The bonus pool is frozen at whatever level it reached before `PROMOTION_REQUESTED` was observed, reducing the payout to stakers who took on real, locked-in risk. The same block applies when the registry is in `PRODUCTION` or `CORRUPTED` but the moderator has not yet flagged the outcome (pool is still `UNRESOLVED`).

### Likelihood Explanation
`PROMOTION_REQUESTED` is a normal, expected lifecycle state for any agreement that survives its term. Any bonus contribution attempted during this state fails unconditionally. The likelihood is high that a sponsor or contributor would attempt to add bonus during the final risk window stretch, particularly as an incentive for stakers who are locked in.

### Recommendation
Remove `_assertDepositsAllowed` from `contributeBonus()`. The existing guards — `outcome != UNRESOLVED` and `block.timestamp >= expiry` — are sufficient to bound bonus contributions to the pre-resolution, pre-expiry window. No timing-based gaming guard is needed because bonus contributors have no claim on the bonus pool and cannot influence their own payout by choosing when to contribute.

```solidity
function contributeBonus(uint256 amount) external nonReentrant whenPoolNotPaused {
    if (amount == 0) revert InvalidAmount();
    if (outcome != PoolStates.Outcome.UNRESOLVED) revert OutcomeAlreadySet();
    if (block.timestamp >= expiry) revert StakingClosed();

    // Remove: _assertDepositsAllowed(_observePoolState());
    // Bonus contributions carry no timing-based gaming risk.
    // The two guards above are sufficient.

    uint256 balanceBefore = stakeToken.balanceOf(address(this));
    stakeToken.safeTransferFrom(msg.sender, address(this), amount);
    uint256 balanceAfter = stakeToken.balanceOf(address(this));
    uint256 received = balanceAfter > balanceBefore ? balanceAfter - balanceBefore : 0;
    if (received == 0) revert NoTokensReceived();

    totalBonus += received;
    emit BonusContributed(msg.sender, received);
}
```

### Proof of Concept

1. Pool is deployed; stakers deposit during `UNDER_ATTACK`. `riskWindowStart` is sealed.
2. Registry transitions to `PROMOTION_REQUESTED`. Stakers are locked in (withdrawals disabled).
3. Sponsor calls `contributeBonus(1000e18)` to top up the bonus pool for locked-in stakers.
4. `_observePoolState()` returns `PROMOTION_REQUESTED`.
5. `_assertDepositsAllowed(PROMOTION_REQUESTED)` reverts with `StakingClosed()`.
6. The bonus contribution fails. The bonus pool remains at its pre-`PROMOTION_REQUESTED` level.
7. Stakers who bore risk during the closing window receive a smaller bonus than the sponsor intended to provide. [5](#0-4) [6](#0-5)

    /// @dev `UNDER_ATTACK` is intentionally NOT blocked while `PROMOTION_REQUESTED` is. Both are
    /// active-risk (attackable, still corruptible); the asymmetry is about deposit *timing*, not
    /// safety — UNDER_ATTACK deposits earn ~zero k=2 bonus and self-lock (no trap), whereas
    /// PROMOTION_REQUESTED is the closing-window stretch where a late join would be gameable. See
    /// docs/DESIGN.md (deposit gating).
    function _assertDepositsAllowed(IAttackRegistry.ContractState state) private pure {
        if (
            state == IAttackRegistry.ContractState.PROMOTION_REQUESTED
                || state == IAttackRegistry.ContractState.PRODUCTION || state == IAttackRegistry.ContractState.CORRUPTED
        ) {
            revert StakingClosed();
        }
    }
