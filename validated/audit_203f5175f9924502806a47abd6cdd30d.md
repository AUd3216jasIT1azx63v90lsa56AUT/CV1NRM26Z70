### Title
Flawed `_clampUserSums` detection silently skips pre-risk clamping for mixed stakers, overstating `userScore` above `globalScore` and permanently locking principal — (File: src/ConfidencePool.sol)

### Summary
`_clampUserSums` uses a single aggregate inequality (`userSumStakeTime[u] < eligibleStake[u] × riskWindowStart`) to decide whether a user's pre-risk deposits need their entry times promoted to `riskWindowStart`. This condition is only correct for users whose entire stake was deposited before the risk window opened. For a user who has both a pre-risk deposit and a subsequent post-risk deposit (permitted during `UNDER_ATTACK`), the post-risk deposit inflates `userSumStakeTime[u]` enough to make the condition false, so clamping is silently skipped. The resulting stale per-user sums cause `_bonusShare` to compute a `userScore` that exceeds the correctly-computed `globalScore`, making the bonus share exceed `snapshotTotalBonus`. The `safeTransfer` in `claimSurvived` / `claimExpired` then reverts, permanently locking the user's principal inside the contract.

### Finding Description

`_markRiskWindowStart` eagerly resets the **global** accumulators for all currently-staked capital: [1](#0-0) 

Per-user sums are left stale and are lazily corrected by `_clampUserSums` on the next operation touching that user: [2](#0-1) 

The detection condition on line 681 assumes that `userSumStakeTime[u] < stake_ * start` is the invariant that distinguishes "not yet clamped" from "already clamped." This holds only when every deposit was made before `riskWindowStart`. When a user also has a post-risk deposit (recorded at wall-clock time ≥ `riskWindowStart` per the floor in `stake()`): [3](#0-2) 

the post-risk contribution `amount_post × entryTime_post` (where `entryTime_post ≥ riskWindowStart`) can push `userSumStakeTime[u]` to or above `stake_ × riskWindowStart`, making the condition false even though the pre-risk deposit's entry time has never been promoted. Clamping is skipped entirely.

After the skip, `userSumStakeTime[u]` equals `amount_pre × entryTime_pre + amount_post × entryTime_post` instead of the correct `amount_pre × riskWindowStart + amount_post × entryTime_post`. Similarly, `userSumStakeTimeSq[u]` equals `amount_pre × entryTime_pre² + amount_post × entryTime_post²` instead of `amount_pre × riskWindowStart² + amount_post × entryTime_post²`. Both are understated relative to their correct clamped values.

In `_bonusShare`, the k=2 score is: [4](#0-3) 

Because `userSumStakeTime[u]` is understated, `userMinus` is understated, so `userScore` is overstated. Simultaneously, `userSumStakeTimeSq[u]` is understated, so `userPlus` is understated. The net overstatement of `userScore` relative to the correct value equals:

```
amount_pre × (riskWindowStart − entryTime_pre) × (2T − riskWindowStart − entryTime_pre)
```

Since `T ≥ riskWindowStart > entryTime_pre`, this quantity is strictly positive — `userScore` is always overstated when clamping is missed. The global score is computed from the correctly-maintained snapshot sums: [5](#0-4) 

so `userScore > globalScore` is reachable, causing `Math.mulDiv(userScore, snapshotTotalBonus, globalScore)` to return a value greater than `snapshotTotalBonus`. The subsequent `safeTransfer(msg.sender, userEligible + bonusShare)` reverts because the contract balance is insufficient.

### Impact Explanation

A user who (a) staked before the risk window and (b) added a second deposit during `UNDER_ATTACK` (explicitly permitted by `_assertDepositsAllowed`) can be permanently unable to execute `claimSurvived` or `claimExpired`. Their principal (`eligibleStake[msg.sender]`) is locked inside the contract with no recovery path: `withdraw` is disabled once `riskWindowStart != 0`, and the claim entrypoints revert. The bonus pool is also distorted for all other claimants because the global snapshot is correct while the affected user's numerator is inflated.

### Likelihood Explanation

The trigger requires only two ordinary user actions — one pre-risk stake and one post-risk stake during `UNDER_ATTACK` — both of which are explicitly supported and documented as intentional. No privileged role is needed. The exact numeric threshold (`amount_post × (entryTime_post − riskWindowStart) ≥ amount_pre × (riskWindowStart − entryTime_pre)`) is easily satisfied when the second deposit is equal to or larger than the first, or when the second deposit occurs well after `riskWindowStart`. This is a common staking pattern.

### Recommendation

Replace the aggregate inequality with a per-deposit structural check. The simplest correct fix is to track whether the user has any pre-risk deposits that have not yet been promoted. One approach: always apply the clamp unconditionally and let the assignment be a no-op when the values are already correct:

```solidity
function _clampUserSums(address u) internal {
    uint256 start = riskWindowStart;
    uint256 stake_ = eligibleStake[u];
    if (start == 0 || stake_ == 0) return;

    // Correct clamped values: treat every pre-risk deposit as entering at riskWindowStart.
    // For a user with only post-risk deposits, userSumStakeTime[u] >= stake_ * start already,
    // so the max() leaves the values unchanged.
    uint256 clampedTime   = stake_ * start;
    uint256 clampedTimeSq = stake_ * start * start;

    // Decompose: post-risk portion is already correct; only the pre-risk portion needs lifting.
    // Equivalent: replace the pre-risk contribution with amount_pre * start.
    // Since post-risk entries are recorded at >= start, the correct aggregate is:
    //   userSumStakeTime   = max(userSumStakeTime[u], clampedTime)   -- wrong for mixed case
    // Instead, track a separate pre-risk sum, OR unconditionally rewrite both sums to
    // reflect the clamped pre-risk portion:
    if (userSumStakeTime[u] < clampedTime) {
        // Pure pre-risk case: all deposits were before riskWindowStart.
        userSumStakeTime[u]   = clampedTime;
        userSumStakeTimeSq[u] = clampedTimeSq;
    } else {
        // Mixed case: subtract the stale pre-risk contribution and add the clamped one.
        // Requires storing userSumStakeTimePreRisk separately, or restructuring accounting.
    }
}
```

The cleanest fix is to store `userSumStakeTimePreRisk` and `userSumStakeTimeSqPreRisk` separately so the clamp can surgically replace only the pre-risk portion, leaving the post-risk portion intact.

### Proof of Concept

**Setup:**
- `riskWindowStart = 2000`, `riskWindowEnd = outcomeFlaggedAt = T = 4000`, `snapshotTotalBonus = 1000`
- Alice stakes `100` tokens at `t = 1000` (pre-risk)
- Risk window opens at `t = 2000` (`_markRiskWindowStart` eagerly sets `sumStakeTime = 100 × 2000 = 200000`, `sumStakeTimeSq = 100 × 4000000 = 400000000`)
- Alice stakes `100` tokens at `t = 3000` (post-risk, during `UNDER_ATTACK`)
  - `sumStakeTime += 100 × 3000 = 500000`; `sumStakeTimeSq += 100 × 9000000 = 1300000000`
  - `userSumStakeTime[Alice] = 100×1000 + 100×3000 = 400000`
  - `userSumStakeTimeSq[Alice] = 100×1000000 + 100×9000000 = 1000000000`
- Snapshot at `flagOutcome`: `snapshotTotalStaked = 200`, `snapshotSumStakeTime = 500000`, `snapshotSumStakeTimeSq = 1300000000`

**At `claimSurvived`:**

`_clampUserSums(Alice)`: `stake_ × start = 200 × 2000 = 400000`. Condition: `400000 < 400000` → **false → no clamp**.

`_bonusShare`:
- `userPlus = 4000² × 200 + 1000000000 = 3200000000 + 1000000000 = 4200000000`
- `userMinus = 2 × 4000 × 400000 = 3200000000`
- `userScore = 1000000000`
- `plus = 4000² × 200 + 1300000000 = 4500000000`
- `minus = 2 × 4000 × 500000 = 4000000000`
- `globalScore = 500000000`
- `bonusShare = mulDiv(1000000000, 1000, 500000000) = 2000` (200% of bonus pool)

`payout = 200 + 2000`. Contract holds `200 + 1000 = 1200`. **`safeTransfer` reverts. Alice's principal is permanently locked.**

Correct clamped result: `userSumStakeTime = 100×2000 + 100×3000 = 500000`, `userSumStakeTimeSq = 100×4000000 + 100×9000000 = 1300000000`, `userScore = globalScore = 500000000`, `bonusShare = 1000` (100% of bonus, correct for sole staker).
