### Title
Dust Liquidation Front-Run Causes `repay()` to Revert via Debt Underflow — (File: src/Midnight.sol)

### Summary

The `repay` function subtracts `units` directly from `position[id][onBehalf].debt` with no upper-bound check. A permissionless `liquidate` call can reduce a borrower's debt by a dust amount between when the borrower reads their debt and when their `repay` transaction executes, causing the subtraction to underflow and revert. This is the direct analog of the Cooler M-4 vulnerability class.

### Finding Description

In `repay`, the debt decrement is unchecked against the current balance:

```solidity
// src/Midnight.sol:508
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
``` [1](#0-0) 

Solidity 0.8.x checked arithmetic means this reverts if `units > debt`. There is no `min(units, debt)` guard and no cap on the caller-supplied `units`.

The `liquidate` function is fully permissionless for any liquidatable borrower (unhealthy pre-maturity, or any borrower post-maturity). It reduces the borrower's debt:

```solidity
// src/Midnight.sol:676
_position.debt -= UtilsLib.toUint128(repaidUnits);
``` [2](#0-1) 

**Exploit path:**

1. Borrower has `debt = D` and is liquidatable (post-maturity **or** unhealthy pre-maturity).
2. Borrower submits `repay(units = D)` to fully close their position.
3. Attacker front-runs with `liquidate(repaidUnits = 1)` (dust), which is permissionless and costs only 1 loan token unit.
4. Borrower's on-chain `debt` is now `D - 1`.
5. Borrower's `repay(units = D)` hits line 508: `(D-1) - D` underflows → **reverts**.
6. Borrower must re-query debt and retry; in the meantime they remain exposed to further liquidation.

### Impact Explanation

The borrower's repay transaction is griefed: it reverts and wastes gas. The borrower must re-query their current debt and resubmit. In the post-maturity scenario the LIF (liquidation incentive factor) grows linearly over time toward `maxLif`, so repeated griefing forces the borrower to lose progressively more collateral per unit repaid to liquidators. Automated repay contracts (e.g., keeper bots or periphery bundles) that read debt off-chain and pass it as `units` are particularly vulnerable because they may not retry or may retry with a stale value.

There is no permanent, irrecoverable loss of funds in the base case — the borrower can always retry — but the attack degrades service availability and, in the post-maturity growing-LIF window, causes incremental collateral loss proportional to how long the griefing persists.

### Likelihood Explanation

- **Precondition**: The borrower must be liquidatable (post-maturity or unhealthy). This is a realistic and common state.
- **Attacker cost**: 1 loan token unit (dust) plus gas per front-run. Economically trivial.
- **No privileged access required**: `liquidate` is fully permissionless.
- **MEV infrastructure**: Front-running a pending `repay` with a `liquidate` is a standard MEV pattern executable by any searcher.

### Recommendation

Cap the repaid units to the borrower's actual debt inside `repay`, mirroring the Cooler report's recommendation:

```solidity
uint128 currentDebt = position[id][onBehalf].debt;
uint128 actualUnits = units > currentDebt ? currentDebt : UtilsLib.toUint128(units);
position[id][onBehalf].debt -= actualUnits;
marketState[id].withdrawable += actualUnits;
// pull only actualUnits from the payer
```

This ensures a full-repay call always succeeds regardless of any concurrent dust liquidation.

### Proof of Concept

```
State:
  market.maturity = T (already passed, so block.timestamp > T)
  position[id][borrower].debt = 1000e18

Step 1 — Borrower broadcasts:
  repay(market, units=1000e18, onBehalf=borrower, ...)

Step 2 — Attacker sees the pending tx and front-runs:
  liquidate(market, collateralIndex=0, seizedAssets=0, repaidUnits=1,
            borrower=borrower, postMaturityMode=true, ...)
  → position[id][borrower].debt becomes 999999999999999999 (1000e18 - 1)

Step 3 — Borrower's repay executes:
  position[id][borrower].debt -= 1000e18
  → 999999999999999999 - 1000000000000000000 underflows → REVERT

Result: Borrower's repay fails. They must re-query debt and resubmit.
        Attacker spent 1 loan token unit and gained dust collateral via LIF.
``` [3](#0-2) [4](#0-3)
