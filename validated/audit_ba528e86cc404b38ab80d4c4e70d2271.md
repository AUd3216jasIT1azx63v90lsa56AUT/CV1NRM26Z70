Audit Report

## Title
`SetterRatifier` stores chain-agnostic ratification booleans, enabling cross-chain replay of maker offer roots after a hard fork - (File: src/ratifiers/SetterRatifier.sol)

## Summary
`SetterRatifier` records offer-root ratifications as a plain `bool` in `isRootRatified[maker][root]` with no reference to `block.chainid`. After a hard fork both chains inherit identical storage, so every root ratified on the original chain is automatically valid on the forked chain. An unprivileged taker can fill maker offers on the forked chain that the maker never intended to be active there, draining the maker's assets without any privileged access.

## Finding Description

**Root cause — chain-agnostic storage in `SetterRatifier`:**

`setIsRootRatified` writes a boolean keyed only on `(maker, root)`:

```solidity
// src/ratifiers/SetterRatifier.sol:18
mapping(address maker => mapping(bytes32 root => bool)) public isRootRatified;

// lines 24-27
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
```

`isRatified` checks only the Merkle proof and this boolean — no `block.chainid` anywhere:

```solidity
// lines 30-36
function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
    ...
    require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
    require(isRootRatified[offer.maker][root], NotRatified());
    return CALLBACK_SUCCESS;
}

**Contrast with `EcrecoverRatifier`**, which explicitly binds to `block.chainid` via EIP-712 and documents the protection:

```solidity
// src/ratifiers/EcrecoverRatifier.sol:10-11
/// @dev If block.chainid changes (hard fork), the EIP-712 domain separator changes
/// and previously signed offers are no longer valid.
...
// line 40
bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
``` [3](#0-2) [4](#0-3) 

**`Midnight` itself explicitly acknowledges hard fork scenarios** and captures `INITIAL_CHAIN_ID` at construction to keep market IDs stable — confirming the protocol is designed to remain operational post-fork:

```solidity
// src/Midnight.sol:173-175
/// @dev INITIAL_CHAIN_ID is captured at construction and used in place of block.chainid when computing market ids,
/// so a hard fork that changes block.chainid does not strand existing accounting. But as a result, after a hard-fork
/// there can be some market id clashes.
``` [5](#0-4) 

This confirms the protocol explicitly anticipates continued operation on both chains post-fork, making the `SetterRatifier` gap a concrete design inconsistency rather than a theoretical edge case.

**Exploit flow:**

1. Maker calls `setIsRootRatified(maker, root, true)` on chain A (pre-fork). State: `isRootRatified[maker][root] = true`.
2. Hard fork occurs. Chain B inherits identical storage: `isRootRatified[maker][root] = true` on chain B.
3. Attacker (taker) on chain B calls `midnight.take(offer, abi.encode(root, leafIndex, proof), units, taker, ...)`.
4. `Midnight.take` calls `IRatifier(offer.ratifier).isRatified(offer, ratifierData)` → `SetterRatifier.isRatified`.
5. `isRatified` passes: Merkle proof is valid (same offer hash, same root), `isRootRatified[maker][root]` is `true`.
6. Take executes, draining maker's assets on chain B.

All attacker inputs (`root`, `leafIndex`, `proof`) are public on-chain data. No privileged access is required.

**Existing checks are insufficient:** The only guards in `isRatified` are (a) `msg.sender == MIDNIGHT` and (b) Merkle proof validity — neither involves chain identity. The maker's only recourse post-fork is to call `setIsRootRatified(maker, root, false)` on chain B, but this requires the maker to be aware of the fork and act before an attacker front-runs the cancellation.

## Impact Explanation

Direct theft of maker assets on the forked chain. Any maker using `SetterRatifier` with active ratified roots at fork time has those roots automatically valid on the forked chain. A taker can fill offers the maker never intended to be active on that chain, taking the maker's loan tokens or collateral. The impact is equivalent to unauthorized order execution — a concrete, in-scope asset loss.

## Likelihood Explanation

Hard forks are rare but have occurred on major EVM chains (ETH/ETC, ETH/ETH PoW). The precondition — a `SetterRatifier` deployment with active ratified roots — is the normal operating state for any maker using this ratifier. No special attacker capability is required beyond being a taker. The attack is repeatable for every ratified root that exists at fork time, and the attacker can front-run any cancellation attempt by the maker on the forked chain.

## Recommendation

Include `block.chainid` in the storage key for `isRootRatified`, mirroring the chain-binding already present in `EcrecoverRatifier`:

```solidity
mapping(uint256 chainId => mapping(address maker => mapping(bytes32 root => bool))) public isRootRatified;
```

Update `setIsRootRatified` and `isRatified` to key on `block.chainid`:

```solidity
isRootRatified[block.chainid][maker][root] = newIsRootRatified;
// and
require(isRootRatified[block.chainid][offer.maker][root], NotRatified());
```

This ensures that a ratification on chain A is not automatically valid on chain B after a fork, consistent with the protection already documented and implemented in `EcrecoverRatifier`.

## Proof of Concept

**Fork test plan:**

1. Deploy `Midnight` and `SetterRatifier` on a local fork of mainnet (chain ID 1).
2. Maker calls `setIsRootRatified(maker, root, true)`.
3. Fork the chain state (simulate hard fork by changing `block.chainid` to 2 in the test environment, e.g., via `vm.chainId(2)` in Foundry).
4. As an unprivileged taker, call `midnight.take(offer, abi.encode(root, leafIndex, proof), units, taker, ...)` on the forked chain.
5. Assert the take succeeds and maker assets are transferred — demonstrating the replay without any maker action on chain B.

Expected result: take succeeds on chain B using only data observable from chain A, with `isRootRatified[maker][root]` still `true` on chain B.
