### Title
Missing Transaction Deadline in `MidnightBundles` Bundle Functions - (File: src/periphery/MidnightBundles.sol)

### Summary

The four trade-execution functions in `MidnightBundles.sol` — `buyWithUnitsTargetAndWithdrawCollateral`, `supplyCollateralAndSellWithUnitsTarget`, `buyWithAssetsTargetAndWithdrawCollateral`, and `supplyCollateralAndSellWithAssetsTarget` — accept no `deadline` parameter. Because the settlement fee is computed at execution time from `block.timestamp`, a validator who delays the transaction can cause it to execute at a settlement fee that is materially different from what the user anticipated, within the bounds of the user's slippage parameters.

### Finding Description

In `Midnight.sol`, the `take()` function computes the settlement fee dynamically at execution time:

```solidity
uint256 timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp);
uint256 _settlementFee = settlementFee(id, timeToMaturity);
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;
``` [1](#0-0) 

The settlement fee is a piecewise-linear function of TTM with seven breakpoints (0 d, 1 d, 7 d, 30 d, 90 d, 180 d, 360 d): [2](#0-1) 

None of the four `MidnightBundles` entry points accept a `deadline` argument: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

A validator can therefore hold a pending bundle transaction in the mempool and include it in any block before the embedded offers expire. Because `block.timestamp` advances, the settlement fee at inclusion time may differ from the fee at submission time, shifting the effective price against the user.

By contrast, `EcrecoverAuthorizer.setIsAuthorized` correctly enforces `require(block.timestamp <= authorization.deadline, Expired())`: [7](#0-6) 

### Impact Explanation

**For a buyer** (`buyWithUnitsTargetAndWithdrawCollateral` / `buyWithAssetsTargetAndWithdrawCollateral`):
`buyerPrice = offerPrice + settlementFee`. If the settlement fee rises between submission and inclusion (e.g., TTM crosses a breakpoint where the fee schedule increases), the buyer pays more per unit. The `maxBuyerAssets` / `minUnits` guards only prevent execution beyond the user's stated worst-case tolerance — they do not prevent execution at any price within that tolerance window.

**For a seller** (`supplyCollateralAndSellWithUnitsTarget` / `supplyCollateralAndSellWithAssetsTarget`):
`sellerPrice = offerPrice - settlementFee`. A rising settlement fee reduces proceeds. The `minSellerAssets` / `maxUnits` guards are the only protection.

A user who sets generous slippage parameters (common practice to avoid failed transactions) can be forced into the worst-case execution simply by a validator delaying inclusion. The settlement fee can vary by up to 50 bps across the 360-day schedule, which on large notional amounts represents meaningful value. [8](#0-7) 

### Likelihood Explanation

Any block proposer (validator on PoS Ethereum) who observes a pending `MidnightBundles` transaction in the public mempool can delay inclusion until the settlement fee moves in a direction that exhausts the user's slippage budget. No privileged access is required — only the ability to order transactions within a block, which every validator possesses. The attack is especially attractive near TTM breakpoints (1 d, 7 d, 30 d, etc.) where the fee schedule changes discontinuously.

### Recommendation

Add a `deadline` parameter to each bundle function and revert if `block.timestamp > deadline`:

```solidity
function buyWithUnitsTargetAndWithdrawCollateral(
    uint256 targetUnits,
    uint256 maxBuyerAssets,
    uint256 deadline,          // <-- add
    address taker,
    ...
) external {
    require(block.timestamp <= deadline, DeadlineExceeded());
    ...
}
```

Apply the same pattern to all four trade-execution functions. Users should pass a deadline that is only a few blocks in the future (e.g., `block.timestamp + 2 minutes` at submission time), consistent with standard DEX router practice.

### Proof of Concept

1. User submits `buyWithUnitsTargetAndWithdrawCollateral` targeting a market with TTM = 7 days + 1 hour, with `maxBuyerAssets` set 30 bps above the current expected cost (generous slippage).
2. The settlement fee schedule has `settlementFeeCbp2` (7 d breakpoint) > `settlementFeeCbp3` (30 d breakpoint), so the fee rises as TTM crosses below 7 days.
3. A validator withholds the transaction for ~1 hour until TTM < 7 days, causing the settlement fee to jump to the higher `settlementFeeCbp2` value.
4. The transaction executes: `buyerPrice = offerPrice + newHigherFee`. The buyer pays up to `maxBuyerAssets` — the full slippage budget — rather than the expected amount.
5. The validator has caused the user to overpay by up to 30 bps of the trade notional with zero privileged access. [9](#0-8)

    function settlementFee(bytes32 id, uint256 timeToMaturity) public view returns (uint256) {
        MarketState storage _marketState = marketState[id];
        require(_marketState.tickSpacing > 0, MarketNotCreated());

        if (timeToMaturity >= 360 days) return _marketState.settlementFeeCbp6 * CBP;

        // forgefmt: disable-start
        (uint256 start, uint256 end, uint256 feeLower, uint256 feeUpper) =
            timeToMaturity < 1 days   ? (  0 days,   1 days, _marketState.settlementFeeCbp0 * CBP, _marketState.settlementFeeCbp1 * CBP) :
            timeToMaturity < 7 days   ? (  1 days,   7 days, _marketState.settlementFeeCbp1 * CBP, _marketState.settlementFeeCbp2 * CBP) :
            timeToMaturity < 30 days  ? (  7 days,  30 days, _marketState.settlementFeeCbp2 * CBP, _marketState.settlementFeeCbp3 * CBP) :
            timeToMaturity < 90 days  ? ( 30 days,  90 days, _marketState.settlementFeeCbp3 * CBP, _marketState.settlementFeeCbp4 * CBP) :
            timeToMaturity < 180 days ? ( 90 days, 180 days, _marketState.settlementFeeCbp4 * CBP, _marketState.settlementFeeCbp5 * CBP) :
                                        (180 days, 360 days, _marketState.settlementFeeCbp5 * CBP, _marketState.settlementFeeCbp6 * CBP);
        // forgefmt: disable-end

        return (feeLower * (end - timeToMaturity) + feeUpper * (timeToMaturity - start)) / (end - start);
    }
