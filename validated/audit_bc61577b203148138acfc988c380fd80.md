Audit Report

## Title
Post-maturity `seizedAssets` liquidation reverts on bad-debt positions when `lif < maxLif` - (File: `src/Midnight.sol`)

## Summary
When a position carries bad debt and a liquidator calls `liquidate` with `seizedAssets > 0` and `postMaturityMode = true` before `TIME_TO_MAX_LIF` (15 minutes) has elapsed, the bad-debt reduction at line 628 uses `maxLif` to shrink `_position.debt`, while `repaidUnits` at line 650 is computed with the smaller ramping `lif`. Because `lif < maxLif`, the resulting `repaidUnits` exceeds the reduced `_position.debt`, and the unchecked subtraction at line 676 reverts with an arithmetic underflow. The `maxRepaid`/RCF guard that would otherwise cap `repaidUnits` is unconditionally skipped in `postMaturityMode`.

## Finding Description

**Root cause:** The bad-debt loop at lines 607–618 computes each collateral's maximum supportable debt using `maxLif` (line 615):

```solidity
badDebt = badDebt.zeroFloorSub(
    _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
);
``` [1](#0-0) 

After subtracting `badDebt` at line 628, `_position.debt` is bounded by `Σ collateral[i] * price_i / ORACLE_PRICE_SCALE * WAD / maxLif_i`. [2](#0-1) 

However, when `block.timestamp < market.maturity + TIME_TO_MAX_LIF`, the `lif` computed at lines 645–647 is strictly less than `maxLif`:

```solidity
uint256 lif = postMaturityMode
    ? UtilsLib.min(_maxLif, WAD + (_maxLif - WAD) * (block.timestamp - market.maturity) / TIME_TO_MAX_LIF)
    : _maxLif;
``` [3](#0-2) 

The `seizedAssets` branch at line 650 then computes `repaidUnits = seizedAssets.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif)`, which is larger per unit of collateral than the debt ceiling established by the bad-debt step. [4](#0-3) 

The `maxRepaid`/RCF guard at lines 655–668 is entirely skipped because it is gated on `!postMaturityMode`: [5](#0-4) 

The subtraction at line 676 therefore underflows and reverts:

```solidity
_position.debt -= UtilsLib.toUint128(repaidUnits);
``` [6](#0-5) 

**Concrete verification** (`TIME_TO_MAX_LIF = 15 minutes` per `ConstantsLib.sol` line 19): [7](#0-6) 

| Parameter | Value |
|---|---|
| `collateral[0]` | 901 |
| `price_0` | `ORACLE_PRICE_SCALE` (1e36) |
| `maxLif_0` | `2 * WAD` |
| `originalDebt` | 452 |
| `Δt` | 1 s |

- `badDebt = 452 − ceil(901/2) = 452 − 451 = 1`; `_position.debt → 451`
- `lif = 1e18 + floor(1e18/900) = 1001111111111111111`
- `repaidUnits = ceil(901e18 / 1001111111111111111)`: since `1001111111111111111 × 900 + 100 = 901e18`, `repaidUnits = 901`
- `901 > 451` → arithmetic underflow revert at line 676 (Solidity 0.8+ checked arithmetic)

## Impact Explanation
Any liquidator calling `liquidate(seizedAssets > 0, postMaturityMode = true)` on a bad-debt position during the 15-minute post-maturity ramp window receives an unconditional revert. The collateral-seizure entry point — the natural choice when targeting a specific collateral amount — is completely broken for this combination of conditions. This delays timely liquidation during a critical window, degrading protocol liveness and leaving bad-debt positions unresolvable via the primary liquidation path.

## Likelihood Explanation
All three preconditions are reachable by any unprivileged user without special access:
1. Bad debt is a normal protocol outcome when collateral value falls below `debt * maxLif / ORACLE_PRICE_SCALE`.
2. The 15-minute ramp window is a fixed protocol constant (`TIME_TO_MAX_LIF = 15 minutes`).
3. Any liquidator using the `seizedAssets`-input form hits this deterministically within the window.

The revert is deterministic and requires no coordination or privileged action.

## Recommendation
Cap `repaidUnits` at `_position.debt` before the subtraction at line 676, or extend the `maxRepaid` guard to also apply in `postMaturityMode`. The simplest fix is to add a cap immediately after line 650:

```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
    repaidUnits = UtilsLib.min(repaidUnits, _position.debt); // cap to avoid underflow
}
```

Alternatively, move the `repaidUnits <= _position.debt` guard outside the `!postMaturityMode` branch so it applies universally.

## Proof of Concept
Deploy a market with one collateral (`maxLif = 2e18`, `lltv` any allowed value). Supply a borrower with `collateral[0] = 901` and `debt = 452`. Advance time to `market.maturity + 1`. Call `liquidate(market, 0, 901, 0, borrower, true, receiver, address(0), "")`. The call reverts with an arithmetic underflow at line 676. Confirm by also calling with `repaidUnits = 451` (non-`seizedAssets` path), which succeeds, demonstrating the bug is isolated to the `seizedAssets` branch in `postMaturityMode`.
