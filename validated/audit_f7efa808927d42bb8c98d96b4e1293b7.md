Audit Report

## Title
Taker-Controlled `takerCallback` Sets `payer` to Arbitrary Third Party on Sell Offers - (File: `src/Midnight.sol`)

## Summary
In `Midnight.take`, when processing a sell offer (`offer.buy = false`), `buyerCallback` is set directly to the taker-supplied `takerCallback` and `payer` is derived from it, with no check that `takerCallback` is the taker or authorized by the taker. An attacker can set `takerCallback = victimContract`, causing the protocol to call `onBuy` on the victim and then pull loan tokens from it. Any contract that implements `IBuyCallback`, returns `CALLBACK_SUCCESS` from `onBuy` without verifying `buyer == address(this)`, and holds a token balance with approval to Midnight can have its tokens drained by an unprivileged caller.

## Finding Description

**Root cause — no authorization constraint on `takerCallback`:**

The only taker-side authorization check in `take` is:

```solidity
// src/Midnight.sol:346
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());
```

This verifies that `msg.sender` may act as `taker`, but places no constraint on `takerCallback`. The parameter is accepted as-is.

**Payer derivation for sell offers:**

```solidity
// src/Midnight.sol:420-422
address buyerCallback = offer.buy ? offer.callback : takerCallback;   // sell: = takerCallback (attacker-controlled)
address sellerCallback = offer.buy ? takerCallback : offer.callback;
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
// sell + non-zero takerCallback → payer = takerCallback
```

**`onBuy` is invoked on the attacker-chosen address before any transfer:**

```solidity
// src/Midnight.sol:445-452
if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback).onBuy(id, offer.market, buyerAssets, units, buyerPendingFeeIncrease, buyer, buyerCallbackData)
            == CALLBACK_SUCCESS,
        WrongBuyCallbackReturnValue()
    );
}
```

**Transfers then pull from `payer = victimContract`:**

```solidity
// src/Midnight.sol:455-456
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

**Critical detail on the `buyer` argument:** For a sell offer, `buyer = taker` (line 375), i.e., the attacker's address — not the callback contract's address. A callback that naively checks `buyer == address(this)` would catch this, but a callback that only checks `msg.sender == midnight` would not. The protocol's own `LendCallback` test contract (`test/TakeTest.sol:1452-1470`) demonstrates the vulnerable pattern: it checks only the market ID, approves `buyerAssets` to Midnight inside `onBuy`, and returns `CALLBACK_SUCCESS` — making it directly exploitable as a victim.

**Exploit flow:**
1. Maker signs a sell offer (`offer.buy = false`).
2. Attacker calls `take(offer, ratifierData, units, attacker, receiver, victimContract, data)`.
3. `buyerCallback = victimContract`, `payer = victimContract`.
4. `victimContract.onBuy(...)` is called by Midnight; if `victimContract` checks only `msg.sender == midnight` (or only the market ID), it approves tokens and returns `CALLBACK_SUCCESS`.
5. `safeTransferFrom(loanToken, victimContract, ...)` pulls `buyerAssets` from `victimContract`.
6. The attacker's credit position is credited; the victim paid.

**Why existing checks fail:**
- The `TakerUnauthorized` check (line 346) only guards who may act as `taker`; it does not constrain `takerCallback`.
- The ratifier validates the maker's signature over `offer` fields; `takerCallback` is not part of the signed offer and is not validated.
- The Certora spec `OnlyExplicitPayerCanLoseTokens.spec` (lines 91-114) explicitly models and proves that tokens may be pulled from any `buyerCallback` that returned `CALLBACK_SUCCESS` — confirming this path is reachable by design with no additional guard.

## Impact Explanation
Direct theft of ERC-20 tokens from any contract that (a) implements `IBuyCallback`, (b) returns `CALLBACK_SUCCESS` from `onBuy` without verifying `buyer == address(this)`, and (c) holds a token balance with approval to Midnight (or approves inside `onBuy`). The victim never authorized the specific trade. This constitutes unauthorized movement of assets triggered by an unprivileged external caller.

## Likelihood Explanation
The attacker requires no privileged access — only the ability to call `take` as a taker. Any contract purpose-built to act as a buy callback in Midnight (flash-loan wrappers, routing contracts, aggregator adapters) will implement `IBuyCallback` and will commonly guard only `msg.sender == midnight` or the market ID, not `buyer == address(this)`. The protocol's own `LendCallback` test contract is vulnerable. The attack is repeatable as long as the victim retains its balance and callback interface.

## Recommendation
Constrain `takerCallback` to be either `address(0)`, `msg.sender`, or an address explicitly authorized by `taker` via `isAuthorized`. Specifically, add a check such as:

```solidity
require(
    takerCallback == address(0) || takerCallback == msg.sender || isAuthorized[taker][takerCallback],
    TakerCallbackUnauthorized()
);
```

This mirrors the existing `TakerUnauthorized` pattern and ensures the payer is always an address the taker has explicitly authorized. Additionally, document in `IBuyCallback` that implementations must verify `buyer == address(this)` to prevent being used as an unauthorized payer.

## Proof of Concept
Deploy a `LendCallback`-style contract holding loan tokens with Midnight approval. Have it implement `onBuy` checking only `msg.sender == midnight` (or the market ID) and returning `CALLBACK_SUCCESS`. Call `midnight.take(sellOffer, ..., units, attacker, receiver, address(lendCallback), "")` from an unprivileged attacker address. Observe that `lendCallback`'s loan token balance decreases by `buyerAssets` and the attacker's credit position increases by `units`. The existing `LendCallback` contract in `test/TakeTest.sol:1442-1476` can be used directly as the victim in a fork test.
