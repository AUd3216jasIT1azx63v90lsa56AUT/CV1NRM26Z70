Audit Report

## Title
Authorized Delegate Can Consume Authorizer's Nonce via Delegated Signature, Permanently Blocking EcrecoverAuthorizer Usage â€” (`src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` post-increments `nonce[authorization.authorizer]` at line 26 before the signer is identified, and the signer check at lines 33â€“36 accepts any address for which `IMidnight.isAuthorized(authorizer, signer)` returns `true`. An attacker who holds any Midnight authorization from the victim can therefore sign an `Authorization` struct naming the victim as `authorizer`, pass both checks without reverting, and permanently consume the victim's nonce â€” invalidating any pending victim-signed authorizations at that nonce. The attack is repeatable every block at zero capital cost.

## Finding Description
In `src/periphery/EcrecoverAuthorizer.sol`:

```solidity
// Line 26 â€” nonce consumed before signer is known
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

// Lines 33â€“36 â€” delegate accepted as valid signer
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);

// Lines 46â€“47 â€” Midnight call succeeds if EcrecoverAuthorizer is authorized by victim
IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
```

The nonce increment at line 26 is unconditional and is not rolled back when the signer is a delegate rather than the authorizer â€” the transaction does not revert in this path, so the storage write persists. `Midnight.isAuthorized` is a flat, permissionless mapping (`src/Midnight.sol` line 192); any address the victim has ever authorized satisfies the check. `Midnight.setIsAuthorized` at line 732 checks `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`, so the call at lines 46â€“47 succeeds whenever `isAuthorized[victim][EcrecoverAuthorizer] == true`.

**Exploit flow:**

1. **Preconditions:** `isAuthorized[victim][attacker] == true` (victim authorized attacker as an operator/keeper) and `isAuthorized[victim][EcrecoverAuthorizer] == true` (victim uses the sig-based flow). Victim has a pending off-chain signed `Authorization{authorizer=victim, nonce=N}`.
2. Attacker constructs `Authorization{authorizer=victim, authorized=<any third address>, isAuthorized=<any>, nonce=N, deadline=<future>}` and signs it with the **attacker's own key**.
3. Attacker calls `EcrecoverAuthorizer.setIsAuthorized(auth, attackerSig)`.
4. Line 26: `N == nonce[victim]` â†’ passes; `nonce[victim]` becomes `N+1`.
5. Lines 33â€“36: `signer == victim` â†’ false; `isAuthorized[victim][attacker]` â†’ true â†’ passes.
6. Lines 46â€“47: `IMidnight.setIsAuthorized(thirdAddress, <any>, victim)` executes successfully.
7. Victim's nonce is now `N+1`; victim's pending authorization at nonce `N` reverts with `InvalidNonce`.
8. Attacker sets `authorized` to a third address (not themselves), preserving their own authorization, and repeats for `N+1`, `N+2`, etc.

The existing `testEcrecoverAuthorizerInvalidSignature` test (`test/SetIsAuthorizedWithSigTest.sol` lines 88â€“97) only verifies that a fully unauthorized signer (`lender` signing for `borrower` with no prior authorization) is rejected. It does not cover the authorized-delegate path, where the signer check at lines 33â€“36 passes and the transaction succeeds.

## Impact Explanation
An attacker holding any Midnight authorization from the victim can grief the victim's `EcrecoverAuthorizer` nonce sequence indefinitely. Any off-chain signed authorizations the victim has distributed â€” including emergency revocations of malicious operators â€” are rendered permanently invalid. Additionally, the attacker can set `authorized` to any arbitrary address with any `isAuthorized` value, directly manipulating the victim's Midnight authorization state. If the victim is a smart contract that relies solely on the EIP-712 signature flow through `EcrecoverAuthorizer`, they lose all ability to manage their Midnight authorizations, constituting a permanent freeze of a critical authorization management path.

## Likelihood Explanation
Preconditions are common: any user who has authorized an operator (keeper, ratifier, liquidation bot) on Midnight while also having authorized `EcrecoverAuthorizer` is vulnerable. The attacker requires no capital, no special timing, and no privileged access beyond a pre-existing authorization from the victim. The attack is executable every block, front-runnable against any victim-submitted `setIsAuthorized` transaction, and costs only gas.

## Recommendation
Move the nonce increment to after the signer check, or restructure so the nonce is only consumed when `signer == authorization.authorizer`. One concrete fix:

```solidity
address signer = ecrecover(digest, signature.v, signature.r, signature.s);
require(signer != address(0), InvalidSignature());
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

Alternatively, restrict the delegate path so that only `signer == authorization.authorizer` is accepted, removing the delegate signing capability entirely if it is not an intended feature.

## Proof of Concept
Add the following test to `test/SetIsAuthorizedWithSigTest.sol`:

```solidity
function testDelegateCanConsumeAuthorizerNonce() public {
    // Setup: borrower authorizes EcrecoverAuthorizer and lender (as delegate)
    vm.startPrank(borrower);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);
    midnight.setIsAuthorized(lender, true, borrower);
    vm.stopPrank();

    // Victim has a pending authorization at nonce 0
    Authorization memory victimAuth = makeAuthorization(borrower, otherLender, true);
    Signature memory victimSig = signAuthorization(victimAuth, borrower);

    // Attacker (lender) constructs a different auth at the same nonce and signs with their own key
    Authorization memory attackerAuth = makeAuthorization(borrower, otherLender, false);
    attackerAuth.nonce = 0;
    Signature memory attackerSig = signAuthorization(attackerAuth, lender);

    // Attacker submits first â€” consumes nonce 0
    ecrecoverAuthorizer.setIsAuthorized(attackerAuth, attackerSig);
    assertEq(ecrecoverAuthorizer.nonce(borrower), 1);

    // Victim's pending authorization at nonce 0 now reverts
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(victimAuth, victimSig);
}
