Audit Report

## Title
Unprivileged Attacker Can Drain Any Approved `IBuyCallback` Contract via Unvalidated `takerCallback` Payer Assignment - (File: src/Midnight.sol)

## Summary
In `Midnight.take`, when processing a sell offer (`offer.buy = false`), the `takerCallback` parameter is used directly as both `buyerCallback` and `payer` with no check that it equals `msg.sender`, `taker`, or is authorized by either. An attacker can supply any victim contract as `takerCallback`; if that contract has approved Midnight for the loan token and returns `CALLBACK_SUCCESS` from `onBuy`, Midnight will pull `buyerAssets` from the victim and credit the attacker's position at zero personal cost to the attacker.

## Finding Description

**Root cause — `src/Midnight.sol` lines 420–456:** [1](#0-0) 

For `offer.buy = false`:
- `buyerCallback = takerCallback` — fully attacker-supplied, no validation
- `payer = takerCallback` whenever `takerCallback != address(0)`

The only authorization check in `take` is: [2](#0-1) 

This validates the caller's right to act as `taker`, but says nothing about `takerCallback`.

After the callback returns `CALLBACK_SUCCESS`, tokens are pulled from `payer` (the victim): [3](#0-2) 

**Exploit flow:**
1. Victim contract `V` has `loanToken.approve(midnight, type(uint256).max)` and implements `IBuyCallback`, returning `CALLBACK_SUCCESS` from `onBuy` for any call from Midnight (the natural pattern for a legitimate integrator that trusts Midnight as caller).
2. Attacker calls `midnight.take(sellOffer, ratifierData, units, attacker, receiver, address(V), callbackData)`.
3. `buyerCallback = address(V)`, `payer = address(V)`.
4. Midnight calls `V.onBuy(...)` → `V` returns `CALLBACK_SUCCESS`.
5. Midnight executes `safeTransferFrom(loanToken, V, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, V, receiver, sellerAssets)`.
6. `V` loses `buyerAssets` in loan tokens; attacker's position gains credit for `units` at zero personal cost.

**Why the Certora spec fails to catch this:**

The `takeOnlyExplicitPayer` rule sets `buyCallbackAllowed = true` unconditionally: [4](#0-3) 

The `onCallBackSummary` function models the callback non-deterministically returning `CALLBACK_SUCCESS`: [5](#0-4) 

The spec proves tokens are only pulled from the designated payer, but never verifies that the designated payer consented to being the payer for this specific `take`. The spec's comment at L91–94 explicitly lists "the buyerCallback that returned `CALLBACK_SUCCESS`" as an allowed payer, which is precisely the victim scenario. [6](#0-5) 

## Impact Explanation
Direct theft of loan tokens from any contract that (a) has approved Midnight and (b) implements `IBuyCallback` returning `CALLBACK_SUCCESS`. The attacker acquires protocol credit for free. The victim's token balance decreases by `buyerAssets = units.mulDivUp(buyerPrice, WAD)`, unbounded up to the victim's balance and approval. This is a direct, unrecoverable loss of user funds.

## Likelihood Explanation
All three preconditions are realistic in a live ecosystem:
1. Any router, aggregator, or vault that interacts with Midnight will have approved the loan token.
2. Any legitimate Midnight buy-callback integrator will return `CALLBACK_SUCCESS` from `onBuy` when called by Midnight, without being able to verify it was legitimately designated as the callback for this specific `take`.
3. Valid sell offers exist in any active market.

The attack requires no special privilege, is repeatable (limited only by victim's balance/approval and offer capacity), and costs the attacker only gas.

## Recommendation
Validate that `takerCallback` is authorized by the taker. The simplest fix is to require that when `takerCallback != address(0)` and `offer.buy = false`, `takerCallback == taker` or `isAuthorized[taker][takerCallback]`. This mirrors the existing `TakerUnauthorized` pattern and ensures the payer has consented to being designated for this specific `take`.

```solidity
if (!offer.buy && takerCallback != address(0)) {
    require(
        takerCallback == taker || isAuthorized[taker][takerCallback],
        TakerCallbackUnauthorized()
    );
}
```

## Proof of Concept
1. Deploy a victim contract `V` that: (a) calls `loanToken.approve(midnight, type(uint256).max)`, and (b) implements `IBuyCallback.onBuy` returning `CALLBACK_SUCCESS` unconditionally.
2. Create a valid sell offer from a maker account.
3. Call `midnight.take(sellOffer, ratifierData, units, attacker, receiver, address(V), "")` from the attacker EOA.
4. Observe: `V`'s loan token balance decreases by `buyerAssets`; attacker's `position[id][attacker].credit` increases by `buyerCreditIncrease`; attacker paid nothing.

/// Proves that in `take`, the only addresses whose tokens can be pulled are:
/// 1. msg.sender (when !offer.buy and buyerCallback == 0),
/// 2. the buyerCallback that returned CALLBACK_SUCCESS,
/// 3. the offer maker (when offer.buy and buyerCallback == 0, i.e. maker is the buyer with no callback).
rule takeOnlyExplicitPayer(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiverIfTakerIsSeller, address takerCallback, bytes takerCallbackData) {
    require e.msg.sender != currentContract, "only external calls";

    address buyerCallback = offer.buy ? offer.callback : takerCallback;

    msgSender = e.msg.sender;
    msgSenderAllowed = !offer.buy && buyerCallback == 0;
    callbackAllowed = false;
    maker = offer.maker;
    makerAllowed = offer.buy && buyerCallback == 0;

    buyCallbackAllowed = true;
    liquidateCallbackAllowed = false;
    repayCallbackAllowed = false;
    flashLoanCallbackAllowed = false;
    badPullSeen = false;

    take(e, offer, ratifierData, units, taker, receiverIfTakerIsSeller, takerCallback, takerCallbackData);

    assert !badPullSeen;
}
