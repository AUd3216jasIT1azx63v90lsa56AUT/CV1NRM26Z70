Audit Report

## Title
Free Collateral Seizure via Zero-Price Oracle in `seizedAssets` Input Path - (File: src/Midnight.sol)

## Summary
When `liquidate` is called with `seizedAssets > 0` and the oracle for `collateralIndex` returns `0`, the expression at line 650 evaluates `repaidUnits` to `0` without reverting. The liquidator receives the full `seizedAssets` of collateral tokens while repaying zero debt and transferring zero loan tokens, destroying collateral backing without any corresponding debt reduction.

## Finding Description

**Root cause — `mulDivUp` with a zero numerator factor silently returns 0:**

`UtilsLib.mulDivUp` is implemented as `(x * y + (d - 1)) / d`. [1](#0-0) 

With `liquidatedCollatPrice = 0` and `ORACLE_PRICE_SCALE = 1e36`:
- `seizedAssets.mulDivUp(0, 1e36)` = `(0 + (1e36 − 1)) / 1e36` = `0`
- `0.mulDivUp(WAD, lif)` = `(0 + (lif − 1)) / lif` = `0` (because `lif ≥ WAD = 1e18`)

So `repaidUnits = 0` at line 650. [2](#0-1) 

**Why the `NotLiquidatable` guard does not block this:**

The while loop at lines 607–618 iterates over the borrower's active collateral bitmap. The zero-price collateral contributes `0` to `maxDebt`, but a second collateral with a non-zero price can make `originalDebt > maxDebt` true, satisfying the health check. [3](#0-2) [4](#0-3) 

**Why the `maxRepaid` guard does not block this:**

With `repaidUnits = 0`, the condition `repaidUnits <= maxRepaid` is trivially `0 <= maxRepaid`, which is always true for a `uint256`. [5](#0-4) 

**Net accounting effect:**

```
_position.collateral[collateralIndex] -= seizedAssets;  // collateral destroyed
_position.debt -= 0;                                    // debt unchanged
_marketState.withdrawable += 0;                         // pool unchanged
safeTransfer(collateralToken, receiver, seizedAssets);  // attacker receives collateral
safeTransferFrom(loanToken, payer, address(this), 0);   // attacker pays nothing
``` [6](#0-5) [7](#0-6) 

**Contrast with the `repaidUnits > 0` path:**

Line 652 divides by `liquidatedCollatPrice` directly, so a zero price causes a division-by-zero revert. The `seizedAssets` path has no such division. [8](#0-7) 

**Protocol's own Certora specs confirm the gap:**

`NoDivisionByZero.spec` line 124 explicitly assumes `ghostPrice(...) > 0` to prove no division by zero in `liquidate`, acknowledging the zero-price case is unproven. [9](#0-8) 

`Reverts.spec` lines 245–253 (`oracleZeroCausesLiquidateWithRepaidRevert`) only proves the `repaidUnits > 0` path reverts on zero price; there is no corresponding rule for the `seizedAssets > 0` path. [10](#0-9) 

`certora/README.md` line 98 explicitly states: *"NoDivisionByZero.spec checks that division by zero never occur, **except for liquidate if the liquidated collateral is not activated or has a price of 0**."* [11](#0-10) 

## Impact Explanation
An unprivileged liquidator can seize any amount of a borrower's collateral at `collateralIndex` (up to the borrower's full balance) while repaying zero debt. The borrower loses collateral but their debt is unchanged. Lenders lose because collateral backing is destroyed without any corresponding debt reduction, directly worsening pool solvency. The core accounting invariant — that collateral can only be seized in exchange for debt repayment — is violated.

## Likelihood Explanation
**Required preconditions:**
1. A market with at least two collateral params (standard multi-collateral setup).
2. The borrower has non-zero collateral at both indices (both bits set in `collateralBitmap`).
3. The oracle for the target `collateralIndex` returns `0` — explicitly modeled as a reachable, non-failure state by the protocol's own Certora specs.
4. The borrower is unhealthy based on the other collateral's price, so `NotLiquidatable` passes.

A price feed returning `0` for a delisted, deprecated, paused, or uninitialized asset is a realistic on-chain condition. The attack is repeatable as long as the oracle continues returning `0` and the borrower retains collateral at that index. No special privileges are required.

## Recommendation
Add an explicit revert when `liquidatedCollatPrice == 0` and `seizedAssets > 0`, mirroring the protection that the `repaidUnits > 0` path already receives implicitly via division. For example, before line 650:

```solidity
require(liquidatedCollatPrice > 0, ZeroCollateralPrice());
```

Alternatively, restructure the `seizedAssets` branch to also divide by `liquidatedCollatPrice` (as the `repaidUnits` branch does), so a zero price causes a natural revert. The corresponding Certora rule in `Reverts.spec` should be extended to cover the `seizedAssets > 0` path, and the assumption in `NoDivisionByZero.spec` line 124 should be removed once the fix is in place.

## Proof of Concept
**Minimal manual steps:**
1. Deploy a market with two collateral params: `collateral[0]` with a normal oracle, `collateral[1]` with an oracle that returns `0`.
2. Create a borrower position with collateral at both indices and debt such that `debt > maxDebt` (unhealthy due to `collateral[0]`'s contribution alone).
3. Call `liquidate(market, collateralIndex=1, seizedAssets=X, repaidUnits=0, borrower, ...)` from an attacker address.
4. Observe: attacker receives `X` tokens of `collateral[1]`; borrower's debt is unchanged; attacker's loan token balance is unchanged.

**Invariant/fuzz test plan:**
- Assert that after any `liquidate` call, `repaidUnits > 0` whenever `seizedAssets > 0`.
- Assert that `borrower.debt_after < borrower.debt_before` whenever `seizedAssets > 0`.
- Fuzz `liquidatedCollatPrice` across the full `uint256` range including `0`.
