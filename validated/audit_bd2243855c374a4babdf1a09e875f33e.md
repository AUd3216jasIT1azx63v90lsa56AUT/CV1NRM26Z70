### Title
Fee-on-Transfer Tokens Cause Improper Accounting in `supplyCollateral` and `repay` — (File: `src/Midnight.sol`)

---

### Summary

The Midnight protocol records accounting values using the caller-supplied `assets`/`units` parameter before the actual token transfer executes. `SafeTransferLib` only checks for call success and a boolean return value — it does not verify the amount actually received. If a fee-on-transfer (deflationary) token is used as a collateral or loan token, less than the recorded amount arrives at the contract, creating a persistent accounting surplus that can cause undercollateralized borrowing, failed liquidations, and lender fund loss. The protocol is permissionless: any ERC-20 can be used as a loan or collateral token with no enforcement of the documented TOKEN SAFETY REQUIREMENTS.

---

### Finding Description

**Vulnerability type:** Accounting (improper balance tracking)

**Root cause — `supplyCollateral`:** [1](#0-0) 

`_position.collateral[collateralIndex]` is incremented by the caller-supplied `assets` before the transfer executes. [2](#0-1) 

`safeTransferFrom` is called after the state write. If the collateral token charges a transfer fee, only `assets * (1 - fee)` arrives, but `assets` is permanently recorded as the borrower's collateral.

**Root cause — `repay`:** [3](#0-2) 

Both `position[id][onBehalf].debt` and `marketState[id].withdrawable` are updated using the caller-supplied `units` before the transfer. [4](#0-3) 

If the loan token is deflationary, only `units * (1 - fee)` arrives, but `withdrawable` is inflated by the full `units`. Subsequent `withdraw` calls drain the contract faster than tokens arrive.

**`SafeTransferLib` does not check received amount:** [5](#0-4) 

The library only checks call success and the boolean return value. There is no balance-before/balance-after check. A fee-on-transfer token passes this check silently.

**Documented but unenforced requirement:** [6](#0-5) 

The protocol documents "Midnight must send/receive exactly the requested amount on transfers" as a TOKEN SAFETY REQUIREMENT, but this is a comment-level assumption with zero on-chain enforcement. `touchMarket` validates collateral sorting, LLTV, and maxLif, but performs no token-type validation.

**Downstream impact on `withdraw` and `liquidate`:**

`withdraw` calls `safeTransfer(market.loanToken, receiver, units)` after reducing `withdrawable` by `units`. [7](#0-6) 

`liquidate` calls `safeTransfer(collateralToken, receiver, seizedAssets)` after reducing the position's collateral by `seizedAssets`. [8](#0-7) 

Both will revert if the contract's actual token balance is lower than the accounting balance, which is guaranteed when a deflationary token has been used.

---

### Impact Explanation

**Deflationary collateral token path:**
- Borrower calls `supplyCollateral` with `assets = 100`. Only 95 tokens arrive; 100 is recorded.
- Borrower borrows against 100 units of collateral (e.g., 80 loan tokens at 80% LLTV), but the contract only holds 95 collateral tokens.
- Position is undercollateralized from inception.
- Liquidator calls `liquidate` specifying `seizedAssets = 100`. `safeTransfer(collateralToken, receiver, 100)` reverts because the contract holds only 95 tokens.
- Liquidation is blocked; lenders absorb bad debt they cannot recover.

**Deflationary loan token path:**
- Borrower calls `repay` with `units = 100`. Only 95 tokens arrive; `withdrawable` increases by 100.
- Lenders call `withdraw`. The first lenders succeed; the last lenders find the contract balance exhausted and their `withdraw` calls revert.
- Direct loss of principal for late-withdrawing lenders.

---

### Likelihood Explanation

Market creation is fully permissionless via `touchMarket`. Any address can deploy a market with any ERC-20 as the loan token or collateral token — no whitelist, no token validation, no governance approval. A malicious actor can deliberately construct a market around a fee-on-transfer token. Additionally, tokens that add fee mechanisms post-deployment (upgradeable proxies) would silently corrupt all existing markets using that token. No privileged access is required for either scenario.

---

### Recommendation

Enforce the TOKEN SAFETY REQUIREMENTS in code rather than only in documentation. For every `safeTransferFrom` call that is expected to increase the contract's balance, record the balance before and after the transfer and use the actual delta as the credited amount:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Apply the same pattern in `repay` (credit `received` to `withdrawable`, not `units`) and in `take` (credit `received` to `claimableSettlementFee`). Alternatively, implement a token whitelist enforced in `touchMarket`.

### Proof of Concept

```
Setup:
  - Deploy FeeToken: ERC-20 with 5% fee on every transfer.
  - Deploy Midnight.

Step 1 — Create market:
  touchMarket(Market { loanToken: USDC, collateralParams: [{ token: FeeToken, lltv: 0.8e18, ... }], ... })

Step 2 — Supply collateral:
  supplyCollateral(market, 0, 100e18, attacker)
  → FeeToken.transferFrom(attacker, Midnight, 100e18) executes
  → Midnight receives 95e18 (5% fee deducted)
  → position[id][attacker].collateral[0] = 100e18  ← INFLATED

Step 3 — Borrow:
  take(sellOffer, ..., units=80e18)  // borrow 80 USDC against 100e18 recorded collateral
  → position[id][attacker].debt = 80e18
  → Midnight actually holds only 95e18 FeeToken as collateral

Step 4 — Default (do not repay).

Step 5 — Liquidation attempt:
  liquidate(market, 0, 100e18, 0, attacker, false, liquidator, address(0), "")
  → _position.collateral[0] -= 100e18  (accounting update succeeds)
  → safeTransfer(FeeToken, liquidator, 100e18)
  → REVERTS: Midnight only holds 95e18 FeeToken

Result:
  Liquidation is permanently blocked at seizedAssets=100e18.
  Liquidator must guess the true balance (95e18) to succeed,
  but the accounting records 100e18 — the 5e18 gap is unrecoverable bad debt
  socialized onto lenders who never consented to a deflationary collateral token.
```

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
