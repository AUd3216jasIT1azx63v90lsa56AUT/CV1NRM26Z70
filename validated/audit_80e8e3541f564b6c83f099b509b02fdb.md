Audit Report

## Title
Fully-Consumed Buy Offer (maxAssets Mode) Bypassable via Zero-Rounding Units — (File: src/Midnight.sol)

## Summary
When `offer.buy == true` and `offer.maxAssets > 0`, the consumed-cap check uses `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For any tick below `MAX_TICK`, `buyerPrice < WAD`, so choosing `units = 1` makes `buyerAssets = 0`. The cap check `newConsumed <= offer.maxAssets` then passes even when `consumed` is already equal to `maxAssets`, allowing a taker to mutate `credit`, `debt`, and `totalUnits` with zero token transfer. The protocol's own NatSpec acknowledges this at line 94 and the test `testBugBuyMaxAssetsBypass` reproduces it exactly.

## Finding Description
**Exact code path — `src/Midnight.sol` `take()`:**

```solidity
// line 363
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // integer division, rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

// lines 367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [1](#0-0) 

When `units * buyerPrice < WAD`, `mulDivDown` returns 0. `consumed` is incremented by 0, so `newConsumed` equals its pre-call value. If `consumed[maker][group]` is already `maxAssets`, the check `newConsumed <= maxAssets` is trivially satisfied.

Execution continues to the position-mutation block:

```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);   // line 410
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);   // line 414
_marketState.totalUnits = UtilsLib.toUint128(
    _marketState.totalUnits + buyerCreditIncrease - sellerCreditDecrease); // line 416-417
``` [2](#0-1) 

And the token transfer at line 455 sends `buyerAssets - sellerAssets = 0 - 0 = 0` tokens. [3](#0-2) 

**Root cause:** The only guard against over-filling is `require(newConsumed <= offer.maxAssets)`. When the increment is 0 due to rounding, this check is a no-op. There is no `require(units == 0 || buyerAssets > 0)` guard and no `require(newConsumed > consumedBefore)` guard.

The protocol's own NatSpec documents this at line 94:

> `@dev It is possible to give units to a fully consumed assets-based buy offer with price < 1.` [4](#0-3) 

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = M`, `maxUnits = 0`, tick below `MAX_TICK` (so `buyerPrice < WAD`).
2. Offer is legitimately filled until `consumed[maker][group] == M`, or maker calls `setConsumed` directly.
3. Attacker calls `take(offer, ..., units=1)`. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. `consumed` stays at `M`. Check passes.
4. `buyerPos.credit` increases, `sellerPos.debt` increases, `totalUnits` increases — all with zero token transfer.
5. Steps 3–4 repeat indefinitely (via `multicall` or across blocks).

## Impact Explanation
- **Maker's `maxAssets` cap is bypassed**: the maker's intended exposure limit is rendered meaningless; their credit position grows without bound beyond `maxAssets`.
- **Unbacked credit/debt inflation**: `totalUnits` grows without any corresponding asset deposit, breaking the invariant that every credit unit has a matching asset payment. Lenders' withdrawable share is diluted.
- **Zero-cost debt creation**: the attacker accrues debt at zero cost. Since no assets were paid, the attacker has no economic incentive to repay, converting the maker's forced credit into bad debt.
- **Griefing**: the maker is forced into credit positions they explicitly capped, with no recourse.

## Likelihood Explanation
All preconditions are attacker-controllable with no privileged access:
- `offer.buy == true`, `offer.maxAssets > 0`, `offer.maxUnits == 0` — standard buy offer configuration.
- `buyerPrice < WAD` — true for any tick below `MAX_TICK`; the vast majority of real offers.
- `consumed[maker][group] >= maxAssets` — reachable by normal fills or by the maker calling `setConsumed`.
- `units = 1` — always satisfies `1 * buyerPrice < WAD` when `buyerPrice < WAD`.

The attack requires no privileged keys, no flash loan, and no external oracle manipulation. It is repeatable in the same transaction via `multicall`. [5](#0-4) 

## Recommendation
Add a guard that rejects a non-zero `units` take when the resulting asset increment rounds to zero under the cap path:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetIncrement());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce `require(newConsumed > consumed[offer.maker][offer.group])` before the increment (snapshot the old value first) whenever `units > 0`.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) is a complete, self-contained reproduction: [6](#0-5) 

It:
1. Sets `lenderOffer.maxAssets = 1`, tick = `MAX_TICK - 16` (so `buyerPrice < WAD`).
2. Calls `setConsumed` to fully consume the offer (`consumed = maxAssets = 1`).
3. Calls `take(1, borrower, lenderOffer)`.
4. Asserts `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged at `maxAssets`.
5. Asserts `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all **strictly increased** — confirming state mutation with zero payment.

        lenderOffer.maxAssets = 1;

        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
