Audit Report

## Title
Division by Zero in `sellerAssetsToUnits` via `tickToPrice(0) = 0` Causes DoS of Bundle Flows - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary

`tickToPrice(0)` returns `0`, confirmed by a passing unit test. When `consumableUnits` is called on a sell offer with `tick=0`, `maxUnits=0`, and `maxAssets>0`, it delegates to `sellerAssetsToUnits`, which sets `sellerPrice = 0` and then calls `mulDivDown(targetSellerAssets, WAD, 0)`, triggering an unconditional division-by-zero revert. Because `MidnightBundles` does not wrap `consumableUnits` in a `try/catch`, the entire bundle transaction reverts.

## Finding Description

**Confirmed zero-price fact:**

`test/TickLibTest.sol` line 16 asserts and passes: [1](#0-0) 

The same test confirms tick 2 is the first non-zero tick (`1e12`), meaning ticks 0 and 1 both produce `sellerPrice = 0`.

**Root cause in `tickToPrice`:**

The double rounding in `tickToPrice` collapses the result to 0 for tick=0: `divHalfDownUnchecked(1e36, 1e18 + wExp(...))` yields a value smaller than `PRICE_ROUNDING_STEP = 1e12`, so the final `* PRICE_ROUNDING_STEP` multiplication produces 0. [2](#0-1) 

**Reachable division-by-zero path:**

`consumableUnits` reaches `sellerAssetsToUnits` when `offer.maxUnits == 0`, `offer.buy == false`, and `offer.maxAssets > 0`: [3](#0-2) 

Inside `sellerAssetsToUnits`, for a sell offer (`offer.buy = false`), `sellerPrice` is set directly to `offerPrice = tickToPrice(0) = 0`, then `mulDivDown(targetSellerAssets, WAD, sellerPrice)` is called with denominator 0: [4](#0-3) 

**Bundle functions propagate the revert:**

`buyWithUnitsTargetAndWithdrawCollateral` calls `consumableUnits` at line 77 outside any `try/catch`. The `try/catch` only wraps the subsequent `IMidnight.take(...)` call. Its NatSpec explicitly documents "Reverts if ConsumableUnitsLib reverts": [5](#0-4) [6](#0-5) 

`buyWithAssetsTargetAndWithdrawCollateral` has the same pattern — `consumableUnits` is called at line 213 outside any `try/catch`: [7](#0-6) [8](#0-7) 

**Why existing checks fail:**

- `tickToPrice` only guards `tick > MAX_TICK`; tick=0 is in range.
- `consumableUnits` has no guard for `sellerPrice == 0`.
- `sellerAssetsToUnits` has no guard for `sellerPrice == 0`.
- The `NoDivisionByZero.spec` Certora spec covers only `src/Midnight.sol` (not periphery) and uses `NONDET` for `tickToPrice`, so it does not catch this path: [9](#0-8) 

## Impact Explanation

Any bundle transaction that calls `consumableUnits` on a sell offer with `tick=0`, `maxUnits=0`, `maxAssets>0` reverts unconditionally. Because `MidnightBundles` does not wrap `consumableUnits` in a `try/catch`, the entire bundle call fails. This is a concrete DoS of sell-side bundle flows: any automated system (aggregator, router, front-end) that selects offers from an on-chain order book and includes a malicious offer in the `takes` array will have its entire bundle transaction griefed. Legitimate takers lose gas and fail to execute their intended trades.

## Likelihood Explanation

The preconditions are trivially satisfiable by any unprivileged maker: `tick=0` is always a valid, accessible tick (it satisfies `tick <= MAX_TICK` and `0 % spacing == 0` for any spacing), and setting `maxUnits=0` with `maxAssets>0` is a normal asset-capped offer configuration. The attack is zero-cost (no funds required, no state setup), repeatable on every block, and requires no special role or privilege.

## Recommendation

Add a zero-price guard in `sellerAssetsToUnits` (and symmetrically in `buyerAssetsToUnits`) before performing the division. If `sellerPrice == 0`, return 0 or revert with a descriptive error. Alternatively, add a guard in `consumableUnits` to return 0 when `tickToPrice(offer.tick) == 0` for a sell offer with `maxUnits == 0`. A third option is to add a validity check in the offer creation path to reject `tick` values that produce a zero price (i.e., tick < 2).

## Proof of Concept

1. Deploy `MidnightBundles` pointing to a live `Midnight` instance.
2. As an unprivileged maker, create a sell offer with `tick=0`, `maxUnits=0`, `maxAssets=1e18`.
3. As a taker, call `buyWithUnitsTargetAndWithdrawCollateral` with a `takes` array containing the above offer.
4. Observe that `ConsumableUnitsLib.consumableUnits` is called, which calls `TakeAmountsLib.sellerAssetsToUnits`, which calls `mulDivDown(targetSellerAssets, WAD, 0)`, causing the entire transaction to revert.

A minimal Foundry test:
```solidity
function testDivByZeroDoS() public {
    // Construct a sell offer with tick=0, maxUnits=0, maxAssets=1e18
    Offer memory offer = Offer({tick: 0, maxUnits: 0, maxAssets: 1e18, buy: false, ...});
    // Confirm tickToPrice(0) == 0
    assertEq(TickLib.tickToPrice(0), 0);
    // Confirm sellerAssetsToUnits reverts
    vm.expectRevert();
    TakeAmountsLib.sellerAssetsToUnits(midnight, id, offer, 1e18);
}
```
