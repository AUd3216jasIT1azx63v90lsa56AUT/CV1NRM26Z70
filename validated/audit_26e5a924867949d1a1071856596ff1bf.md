Audit Report

## Title
Front-running `buyWithAssetsTargetAndWithdrawCollateral` via partial offer fill causes DoS through exact equality check - (File: src/periphery/MidnightBundles.sol)

## Summary
`buyWithAssetsTargetAndWithdrawCollateral` and `supplyCollateralAndSellWithAssetsTarget` compute `unitsToTake` using a live read of `consumed[maker][group]` via `ConsumableUnitsLib.consumableUnits`. Any unprivileged taker can front-run the victim's bundle by partially filling the same offer, reducing available capacity so the bundle fills fewer assets than `targetFilledBuyerAssets`/`targetFilledSellerAssets`. The post-loop exact equality check then reverts the entire bundle. The attacker receives a legitimate fill on each front-run, making the attack economically neutral or profitable.

## Finding Description

**Root cause — live state read combined with exact equality check and no slippage tolerance.**

`ConsumableUnitsLib.consumableUnits` reads the live `consumed[offer.maker][offer.group]` value at call time: [1](#0-0) 

For a sell offer (`buy == false`) with `maxAssets > 0`, it returns `sellerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed))`. Any increase in `consumed` before this call reduces the returned value.

In `Midnight.take`, every successful take of an offer with `maxAssets > 0` increments the shared `consumed[offer.maker][offer.group]` mapping: [2](#0-1) 

This mapping is global — any taker of the same offer group increments it.

Inside `buyWithAssetsTargetAndWithdrawCollateral`, `unitsToTake` is bounded by `consumableUnits`: [3](#0-2) 

After the loop, an exact equality is enforced: [4](#0-3) 

The same pattern exists in `supplyCollateralAndSellWithAssetsTarget`: [5](#0-4) 

**Exploit flow:**

1. Maker posts a sell offer with `maxAssets = M`, `maxUnits = 0`, and a permissive ratifier.
2. Victim constructs a bundle with `targetBuyerAssets = X`, `takes = [offer]`, requiring `N = buyerAssetsToUnits(X)` units of capacity.
3. Victim broadcasts the bundle transaction.
4. Attacker front-runs with a direct `Midnight.take(offer, ..., Y_units, ...)` where `Y_units >= 1`. This increments `consumed[maker][group]` by `sellerAssets(Y_units)`.
5. Victim's bundle executes. `consumableUnits` now returns `sellerAssetsToUnits(M - consumed_after_attack)`, which is less than `N`.
6. `unitsToTake = min(N, takes[0].units, N - delta) = N - delta` for some `delta > 0`.
7. The take succeeds but fills fewer buyer assets: `filledBuyerAssets = X - epsilon < targetFilledBuyerAssets`.
8. `require(filledBuyerAssets == targetFilledBuyerAssets, OutOfOffers())` reverts.

**Why existing checks do not stop it:**

The `try/catch` around the inner `take` call only catches reverts from `take`. Here, `take` succeeds — it fills `N - delta` units without reverting. The revert originates from the post-loop equality check in the bundler, not from `take` itself: [6](#0-5) 

There is no slippage tolerance or minimum-fill parameter on the assets-target variants. `consumableUnits` has no snapshot or reservation mechanism. `Midnight.take` is permissionless for any taker acting as their own `taker` address: [7](#0-6) 

The only gate is the ratifier check; offers with permissive ratifiers (the common case) are fully exposed.

## Impact Explanation

Any victim using `buyWithAssetsTargetAndWithdrawCollateral` or `supplyCollateralAndSellWithAssetsTarget` targeting an offer with `maxAssets > 0` and a permissive ratifier can be griefed indefinitely. The attacker front-runs each retry with a minimal partial fill (even 1 unit), causing every bundle attempt to revert with `OutOfOffers`. The victim cannot complete their intended trade through the bundler as long as the attacker continues to front-run. This constitutes a targeted, sustained denial of service against a specific user's use of the periphery contract.

## Likelihood Explanation

**Preconditions:**
- Victim uses `buyWithAssetsTargetAndWithdrawCollateral` or `supplyCollateralAndSellWithAssetsTarget` targeting an offer with `maxAssets > 0`.
- The offer's ratifier allows the attacker to take (permissive ratifier — the common case).
- Attacker is any address that can call `Midnight.take` on the same offer (no special role required).
- Attacker can observe the mempool (standard on all non-private-mempool chains).

**Feasibility:** High. The attacker submits a `take` with 1 unit before the victim's bundle lands. The attack is repeatable on every retry. The attacker's only cost is gas; they receive a legitimate fill in return, making the attack economically neutral or profitable.

## Recommendation

1. **Replace the exact equality check with a `>=` check and return excess funds.** Change `require(filledBuyerAssets == targetFilledBuyerAssets, OutOfOffers())` to `require(filledBuyerAssets >= targetFilledBuyerAssets, OutOfOffers())` and refund `targetBuyerAssets - filledBuyerAssets - referralFeeAssets` to the caller. This tolerates partial front-running while still guaranteeing the caller pays no more than `targetBuyerAssets`.
2. **Alternatively, add a `minFilledBuyerAssets` slippage parameter** so the caller can express acceptable partial fills, and remove the exact equality requirement.
3. **Alternatively, allow the loop to continue to subsequent offers** when a partially-consumed offer yields fewer assets than needed, rather than capping `unitsToTake` at `consumableUnits` and accepting a short fill.

## Proof of Concept

**Minimal manual steps (single-offer scenario):**

1. Deploy `Midnight` and `MidnightBundles` on a local fork.
2. Maker creates a sell offer: `buy = false`, `maxAssets = 1000e18`, `maxUnits = 0`, permissive ratifier.
3. Victim calls `buyWithAssetsTargetAndWithdrawCollateral` with `targetBuyerAssets` requiring all 1000e18 of capacity.
4. Before victim's tx lands, attacker calls `Midnight.take(offer, ..., 1_unit, ...)` directly, incrementing `consumed[maker][group]` by `sellerAssets(1)`.
5. Victim's bundle executes: `consumableUnits` returns capacity for `1000e18 - sellerAssets(1)` assets, so `unitsToTake` is reduced by 1 unit's worth.
6. `take` inside the bundle succeeds with fewer units; `filledBuyerAssets < targetFilledBuyerAssets`.
7. `require(filledBuyerAssets == targetFilledBuyerAssets, OutOfOffers())` reverts.
8. Repeat step 4 on every victim retry to sustain the DoS.
