Audit Report

## Title
Stale Root Ratification in SetterRatifier Survives Operator Authorization Revocation - (File: src/ratifiers/SetterRatifier.sol)

## Summary
`SetterRatifier.isRatified` checks only the stored `isRootRatified[offer.maker][root]` boolean and never re-validates whether the entity that originally set it is still authorized. When a maker revokes an operator via `Midnight.setIsAuthorized(operator, false)`, any roots that operator previously ratified remain permanently active, allowing takers to continue filling offers the maker intended to stop.

## Finding Description
**Root cause — `SetterRatifier.isRatified` (lines 30–36):**

```solidity
function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
    require(msg.sender == MIDNIGHT, NotMidnight());
    (bytes32 root, uint256 leafIndex, bytes32[] memory proof) =
        abi.decode(ratifierData, (bytes32, uint256, bytes32[]));
    require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
    require(isRootRatified[offer.maker][root], NotRatified());   // ← only checks stored flag
    return CALLBACK_SUCCESS;
}
```

There is no live call to `IMidnight(MIDNIGHT).isAuthorized(offer.maker, setter)`. By contrast, `EcrecoverRatifier.isRatified` (line 44) performs a live re-check on every call:

```solidity
require(_signer == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer), Unauthorized());
```

This asymmetry is confirmed by `testIsRatifiedRevokeAuthorizationInvalidates` in `test/EcrecoverRatifierTest.sol` (lines 185–205), which proves `EcrecoverRatifier` correctly rejects after revocation. No equivalent test or guard exists in `SetterRatifier`.

**Exploit path:**
1. `maker` calls `midnight.setIsAuthorized(operator, true, maker)` — operator is authorized.
2. `operator` calls `setterRatifier.setIsRootRatified(maker, root, true)` — passes the authorization check at line 25; `isRootRatified[maker][root]` is set to `true`.
3. `maker` calls `midnight.setIsAuthorized(operator, false, maker)` — operator is deauthorized.
4. `taker` calls `midnight.take(offer, abi.encode(root, leafIndex, proof), ...)` — `SetterRatifier.isRatified` is called; the only check is `isRootRatified[maker][root]`, which is still `true`. The take succeeds.

**Why existing checks fail:**
- `Midnight.take` checks `isAuthorized[taker][msg.sender]` — this is the *taker's* authorization, not the maker's.
- The maker's authorization revocation only writes to `isAuthorized[onBehalf][authorized]`; it does not cascade to any ratifier state.
- The maker must manually call `setIsRootRatified(maker, root, false)` to stop fills, but the protocol provides no indication this is required and no automatic mechanism to do it.

## Impact Explanation
Any taker can fill offers from a root ratified by a now-deauthorized operator for as long as the offer's `expiry` has not passed and the market exists. The maker's explicit revocation action does not stop further offer fills, violating the core authorization model. This constitutes unauthorized state changes (new credit/debt positions created against the maker's intent) and unauthorized movement of value.

## Likelihood Explanation
All preconditions are normal, unprivileged user actions: authorizing an operator, having that operator ratify roots (the operator's primary purpose), and later revoking the operator (key rotation, end of service). No oracle manipulation, admin access, or impossible values are required. The scenario is repeatable for every root the operator ratified before revocation, and the test suite at `test/SetterRatifierTest.sol` (lines 48–77) confirms the authorized-setter flow works exactly as described.

## Recommendation
Add a live authorization re-check in `SetterRatifier.isRatified`, tracking which address set each root and verifying it is still authorized at call time. Alternatively, store the setter address alongside the boolean and require `setter == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, setter)` in `isRatified`, mirroring the pattern in `EcrecoverRatifier` line 44.

## Proof of Concept
Extend `test/SetterRatifierTest.sol` with:

```solidity
function testIsRatifiedRevokeAuthorizationDoesNotInvalidate() public {
    Offer memory offer = makeOffer(lender);
    bytes32 _root = HashLib.hashOffer(offer);

    // Step 1: authorize borrower as operator
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, true, lender);

    // Step 2: operator sets root while authorized
    vm.prank(borrower);
    setterRatifier.setIsRootRatified(lender, _root, true);

    // Step 3: maker revokes operator
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, false, lender);

    // Step 4: isRatified still succeeds — demonstrates the bug
    vm.prank(address(midnight));
    bytes32 result = setterRatifier.isRatified(offer, abi.encode(_root, 0, new bytes32[](0)));
    assertEq(result, CALLBACK_SUCCESS); // passes, but should revert
}
```

This test will pass against the current implementation, proving the stale ratification survives revocation.
