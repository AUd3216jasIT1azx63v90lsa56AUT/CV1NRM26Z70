Audit Report

## Title
Buy-offer `consumed` counter frozen at sub-cap value when `buyerAssets` rounds to zero, enabling indefinite fills - (File: src/Midnight.sol)

## Summary
In `take`, when `offer.buy = true` and `offer.maxAssets > 0`, the `consumed` mapping is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. When `buyerPrice < WAD` and `units * buyerPrice < WAD`, this rounds to zero, leaving `consumed` unchanged and the cap check trivially satisfied. Each such call mutates buyer credit, seller debt, and `totalUnits` without transferring any tokens, permanently bypassing the `maxAssets` cap.

## Finding Description
**Exact code path** — `src/Midnight.sol:363-373`:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → can be 0
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // passes when increment == 0
}
```

**Root cause** — `mulDivDown(units, buyerPrice, WAD)` returns 0 whenever `units * buyerPrice < WAD`. The `consumed` mapping is incremented by zero, so `newConsumed` equals the pre-call value. The guard `newConsumed <= offer.maxAssets` is trivially satisfied for any pre-call value ≤ `maxAssets`.

**Exploit flow**:
1. Attacker observes a buy offer with `maxAssets > 0` at a tick where `tickToPrice(tick) < WAD` (e.g., `MAX_TICK - 16`).
2. Calls `take(offer, ..., units=1, ...)` where `1 * buyerPrice < WAD`.
3. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`.
4. `consumed` is unchanged; `require(consumed <= maxAssets)` passes.
5. `buyerCreditIncrease` and `sellerDebtIncrease` are computed from `units = 1`, so the maker gains 1 unit of credit and the taker gains 1 unit of debt — with zero token transfer (`safeTransferFrom(..., 0 - 0)` at line 455–456).
6. Steps 2–5 repeat indefinitely via `multicall`.

**Why existing checks fail** — The only guard is `require(newConsumed <= offer.maxAssets)`. There is no check that `units == 0 || assetsIncrement > 0`. The protocol's own comment at line 94 acknowledges this: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The Certora spec `Consume.spec` rule `takeConsumedAtMaxUnchangedAssets` (lines 88–96) uses `NONDET` summaries for `mulDivDown`/`mulDivUp` (lines 11–12), so the prover never models the zero-rounding case.

## Impact Explanation
An unprivileged taker can call `take` an unbounded number of times against a buy offer at any `consumed` value from 0 to `maxAssets`. Each call with `units` in the rounding-zero range transfers zero tokens yet increments the buyer's credit and the seller's debt by `units`, and increments `totalUnits`. The result is unbacked credit/debt positions: accounting state diverges from actual token balances, violating the core invariant that credit and debt must be backed by real token flows. This constitutes a concrete accounting integrity failure and unauthorized state mutation reachable by any unprivileged user.

## Likelihood Explanation
The precondition `buyerPrice < WAD` is reachable on any market whose tick spacing allows ticks below the WAD-price tick — no privileged action is required. The attacker only needs to be a valid taker (not the maker). The attack is repeatable in a single transaction via `multicall`. The existing test `testBugBuyMaxAssetsBypass` (line 858) already demonstrates the identical behavior with `consumed = maxAssets` (fully consumed), confirming the path is live and the behavior is deterministic.

## Recommendation
Add a guard that rejects zero-asset fills when `maxAssets > 0` and `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetIncrement = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetIncrement > 0, ZeroAssetFill());
    newConsumed = consumed[offer.maker][offer.group] += assetIncrement;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` value such that `units * price >= WAD` when `maxAssets > 0`, or switch to `mulDivUp` for the buy-side assets increment in the consumed accounting path.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` at `test/TakeTest.sol:858` is a complete, passing reproduction:

1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (price < WAD).
2. Pre-set `consumed[lender][group] = maxAssets` via `setConsumed`.
3. Call `take(1, borrower, lenderOffer)`.
4. Assert `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged at `maxAssets`.

        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:

        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);

/// If consumed is already at or above maxAssets before a take in assets mode, it remains unchanged.
rule takeConsumedAtMaxUnchangedAssets(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxUnits == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumedBefore >= offer.maxAssets => consumed(offer.maker, offer.group) == consumedBefore;
}
