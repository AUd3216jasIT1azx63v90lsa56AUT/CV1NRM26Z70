### Title
Post-Maturity Liquidation DoS via `repaidUnits` Underflow When `lif < maxLif` and `badDebt > 0` — (`File: src/Midnight.sol`)

---

### Summary

In `liquidate()`, bad debt is socialized using `maxLif` as the denominator, reducing `_position.debt` to approximately `collateral * price / maxLif`. Immediately after, `repaidUnits` is derived from the seized collateral using the current `lif`, which is strictly less than `maxLif` during the 15-minute post-maturity ramp-up window. Because `lif < maxLif`, the computed `repaidUnits` exceeds the already-reduced `_position.debt`, causing an arithmetic underflow revert. Any liquidator who attempts to both realize bad debt and seize collateral in a single call is blocked for up to `TIME_TO_MAX_LIF = 15 minutes` after maturity.

---

### Finding Description

**Root cause — two logically separate steps share a single debt variable with incompatible denominators.**

**Step 1 — bad debt reduction (lines 626–641):**

```solidity
if (badDebt > 0) {
    _position.debt -= uint128(badDebt);   // debt now ≈ collateral*price/maxLif
    ...
}
```

`badDebt` is computed by subtracting each collateral's value quoted at `maxLif`:

```solidity
badDebt = badDebt.zeroFloorSub(
    _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
);
``` [1](#0-0) 

After this block, for a single-collateral position:

```
_position.debt  ≈  collateral * price / ORACLE_PRICE_SCALE * WAD / maxLif
```

**Step 2 — repaidUnits from seized collateral (lines 649–650):**

```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
}
``` [2](#0-1) 

In post-maturity mode `lif` ramps from `WAD` to `maxLif` over `TIME_TO_MAX_LIF = 15 minutes`:

```solidity
uint256 lif = postMaturityMode
    ? UtilsLib.min(_maxLif, WAD + (_maxLif - WAD) * (block.timestamp - market.maturity) / TIME_TO_MAX_LIF)
    : _maxLif;
``` [3](#0-2) [4](#0-3) 

When `lif < maxLif`:

```
repaidUnits  =  seizedAssets * price / ORACLE_PRICE_SCALE * WAD / lif
             >  seizedAssets * price / ORACLE_PRICE_SCALE * WAD / maxLif
             ≈  _position.debt   (after bad-debt reduction)
```

**Step 3 — underflow revert (line 676):**

```solidity
_position.debt -= UtilsLib.toUint128(repaidUnits);   // reverts: repaidUnits > debt
``` [5](#0-4) 

Solidity 0.8 checked arithmetic causes this to revert unconditionally whenever `repaidUnits > _position.debt`.

---

### Impact Explanation

- Any liquidator calling `liquidate()` with `seizedAssets > 0` on a position where `badDebt > 0` will receive a revert for the entire 15-minute window after maturity.
- Bad debt is already socialized (lenders' `lossFactor` is updated) in the same call, but the collateral cannot be seized until `lif` reaches `maxLif`.
- The collateral remains locked in the position for up to 15 minutes, during which no liquidator can recover it via the normal combined path.
- A workaround exists (two separate calls: first `seizedAssets=0, repaidUnits=0` to realize bad debt, then a second call to seize collateral), but this is non-obvious, requires two transactions, and is not how the function is documented or expected to be used.

---

### Likelihood Explanation

- Trigger conditions are realistic and require no privileged access: a market must have passed maturity, and a borrower's position must be underwater enough that `badDebt > 0` (i.e., collateral value at `maxLif` < debt). This is the normal bad-debt scenario the protocol is designed to handle.
- The 15-minute window (`TIME_TO_MAX_LIF`) is short but guaranteed to occur for every such position at every maturity.
- Liquidator bots that compute `seizedAssets` off-chain and submit a single combined transaction will hit this revert systematically during the window.

---

### Recommendation

Cap `repaidUnits` to the remaining debt **after** bad debt reduction before performing the subtraction, or split the two operations so they use a consistent denominator. Concretely, after the bad-debt block, clamp:

```solidity
repaidUnits = UtilsLib.min(repaidUnits, _position.debt);
```

immediately before line 676. This mirrors the fix described in the referenced report: update the two accounting variables in independent steps that each account for their own component, rather than assuming the combined result is non-negative.

---

### Proof of Concept

**Setup:**
- Single-collateral market, `maxLif = 1.1e18`, `ORACLE_PRICE_SCALE = 1e36`
- Borrower: `debt = 100`, `collateral = 50`, oracle `price = 1e36`

**At maturity + 7.5 minutes** (`lif ≈ 1.05e18`, halfway through ramp):

1. `badDebt = 100 - ceil(50 * 1e36/1e36 * 1e18/1.1e18) = 100 - 46 = 54`
2. After bad-debt block: `_position.debt = 100 - 54 = 46`
3. Liquidator calls `liquidate(..., seizedAssets=50, repaidUnits=0, postMaturityMode=true)`
4. `repaidUnits = ceil(50 * 1e36/1e36 * 1e18/1.05e18) = ceil(47.619...) = 48`
5. `_position.debt (46) -= repaidUnits (48)` → **arithmetic underflow → revert**

The call reverts. The same call at `maturity + 15 minutes` (when `lif = maxLif = 1.1e18`) succeeds:
- `repaidUnits = ceil(50/1.1) = 46 = _position.debt` → subtraction yields 0, no revert.
