Audit Report

## Title
Permit Front-Running Causes Repeatable DoS on Bundler Collateral Supply - (File: src/periphery/MidnightBundles.sol)

## Summary
In `MidnightBundles.pullToken`, when `PermitKind.ERC2612` is used, the permit call is wrapped in a silent `try/catch` but the subsequent `SafeTransferLib.safeTransferFrom` is unconditional. An attacker who observes the victim's permit parameters in the mempool can consume the permit nonce directly on the token contract before the victim's transaction lands, causing `safeTransferFrom` to revert due to insufficient allowance. This makes `supplyCollateralAndSellWithUnitsTarget` (and `supplyCollateralAndSellWithAssetsTarget`) permanently griefable for any victim relying solely on the ERC-2612 permit with no pre-existing bundler allowance.

## Finding Description
**Exact code path:**

`supplyCollateralAndSellWithUnitsTarget` iterates over `collateralSupplies` and for each entry calls `pullToken` before `supplyCollateral`: [1](#0-0) 

Inside `pullToken` for `PermitKind.ERC2612`: [2](#0-1) 

The comment at line 382 explicitly acknowledges the front-run scenario ("a third party may have already consumed the permit"), but the design assumption embedded in that comment is that the victim has a pre-existing allowance. The code does not enforce this assumption — `safeTransferFrom` at line 384 is unconditional regardless of whether the permit succeeded or was already consumed.

**Exploit flow:**
1. Victim constructs a transaction calling `supplyCollateralAndSellWithUnitsTarget` with `PermitKind.ERC2612` and zero pre-existing allowance on the bundler.
2. The permit parameters `(owner, spender, value, deadline, v, r, s)` are visible as calldata in the mempool.
3. Attacker extracts them and calls `collateralToken.permit(owner, bundler, value, deadline, v, r, s)` directly, consuming the nonce.
4. Victim's transaction lands: `try permit(...)` fails silently (nonce already consumed), then `safeTransferFrom` reverts because allowance is 0.
5. The entire transaction reverts — `supplyCollateral` in `Midnight.sol` is never entered, so neither the collateral balance nor the bitmap is modified. [3](#0-2) 

**Why existing checks fail:** The `try/catch` at line 383 is the sole protection and is intentionally designed to swallow the revert. There is no fallback check such as "if `allowance(from, address(this)) >= amount`, skip permit" before the unconditional `safeTransferFrom`.

## Impact Explanation
Any victim who submits `supplyCollateralAndSellWithUnitsTarget` or `supplyCollateralAndSellWithAssetsTarget` using `PermitKind.ERC2612` with no pre-existing bundler allowance can be griefed indefinitely. Each new permit signature the victim generates produces a new nonce that can again be front-run. The victim is effectively locked out of the bundler's collateral-supply path without switching to a pre-approval or Permit2 flow. Protocol state (collateral balances, bitmap) remains consistent — this is a pure liveness break / targeted DoS on a specific user operation, matching the "service unavailability under realistic attacker input" impact class in RESEARCHER.md.

## Likelihood Explanation
**Preconditions:** (1) Victim uses `PermitKind.ERC2612` in `CollateralSupply.permit`; (2) victim has no pre-existing ERC-20 allowance to the bundler for the collateral token; (3) attacker monitors the public mempool. All three are realistic on any public EVM network. The NatSpec at line 113 says the sender "should" pre-approve, but this is advisory and not enforced on-chain, so users can and will rely on the permit-only path. Cost to attacker is one `permit` call per grief attempt (minimal gas). [4](#0-3) 

## Recommendation
Before the unconditional `safeTransferFrom`, check whether the current allowance already covers the transfer and skip the permit call (or skip the transfer revert) accordingly. A minimal fix:

```solidity
if (permit.kind == PermitKind.ERC2612) {
    (uint256 deadline, uint8 v, bytes32 r, bytes32 s) =
        abi.decode(permit.data, (uint256, uint8, bytes32, bytes32));
    if (IERC20(token).allowance(from, address(this)) < amount) {
        // Only attempt permit if allowance is insufficient.
        try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
    }
    SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
}
```

This ensures that if the permit was front-run but the victim had a pre-existing allowance, the transfer still succeeds; and if the victim has no pre-existing allowance and the permit was consumed, the `safeTransferFrom` will still revert — but the victim can detect this and use Permit2 or pre-approve instead of being silently griefed. Alternatively, enforce at the call-site that a sufficient allowance exists before entering the permit path, or document and gate the ERC2612 path to require a pre-existing allowance.

## Proof of Concept
**Minimal fork test outline (Foundry):**

1. Deploy `MidnightBundles` against a local fork.
2. Create a victim EOA with a valid ERC-2612 collateral token and zero bundler allowance.
3. Victim signs a permit `(owner=victim, spender=bundler, value=X, deadline=T, v, r, s)`.
4. Attacker calls `collateralToken.permit(victim, bundler, X, T, v, r, s)` directly — succeeds, nonce incremented.
5. Victim calls `supplyCollateralAndSellWithUnitsTarget(...)` with the same permit parameters.
6. Assert the transaction reverts (insufficient allowance on `safeTransferFrom`).
7. Repeat steps 3–6 with a fresh permit signature to confirm repeatability.

Expected: step 6 reverts every time; victim cannot supply collateral through the bundler without switching to pre-approval or Permit2.
