### Title
`rcfThreshold` Lacks Input Validation in `touchMarket()`, Allowing Permanent Disabling of Recovery Close Factor Protection — (File: src/Midnight.sol)

---

### Summary

The `Market` struct contains a `rcfThreshold` parameter that controls when the Recovery Close Factor (RCF) is deactivated during liquidations. Unlike `lltv` and `maxLif`, which are validated against allowed values in `touchMarket()`, `rcfThreshold` has no validation whatsoever. Since market creation is permissionless, any user can create a market with `rcfThreshold = type(uint256).max`, permanently disabling RCF protection and allowing liquidators to fully liquidate borrowers even when a partial liquidation would restore health.

---

### Finding Description

In `touchMarket()`, the protocol enforces strict validation on two collateral parameters:

- `lltv` is checked against a whitelist of nine allowed values via `isLltvAllowed()`.
- `maxLif` is required to equal `maxLif(lltv, LIQUIDATION_CURSOR_LOW)` or `maxLif(lltv, LIQUIDATION_CURSOR_HIGH)`. [1](#0-0) [2](#0-1) 

However, `rcfThreshold` — the third critical risk parameter in `Market` — receives no validation at all in `touchMarket()`. [3](#0-2) [4](#0-3) 

The `rcfThreshold` is consumed in `liquidate()` as the gate for the RCF bypass condition:

```solidity
uint256 maxRepaid = lltv < WAD
    ? (_position.debt - maxDebt).mulDivUp(WAD * WAD, WAD * WAD - lif * lltv)
    : type(uint256).max;
require(
    repaidUnits <= maxRepaid
        || _position.collateral[collateralIndex].mulDivDown(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
            .mulDivDown(WAD, lif).zeroFloorSub(maxRepaid) < market.rcfThreshold,
    RecoveryCloseFactorConditionsViolated()
);
``` [5](#0-4) 

The second disjunct — `... < market.rcfThreshold` — is the RCF deactivation condition. When `rcfThreshold = type(uint256).max`, this condition evaluates to `true` for every position (since `uint256` values are always `< type(uint256).max` unless they equal it exactly), making `RecoveryCloseFactorConditionsViolated` unreachable. The liquidator can pass any `repaidUnits` up to the full debt, bypassing the `maxRepaid` cap entirely.

The market ID is derived from the full `abi.encode(market)`, so `rcfThreshold` is immutable once the market is created — there is no way to correct a maliciously configured market after the fact. [6](#0-5) 

---

### Impact Explanation

A borrower who enters a market with `rcfThreshold = type(uint256).max` can be fully liquidated the moment their position becomes even slightly unhealthy. Under normal RCF enforcement, the liquidator is capped at `maxRepaid` — the minimum repayment needed to restore health. With RCF disabled, the liquidator seizes the maximum collateral and repays the full debt in one transaction, extracting the maximum liquidation incentive (`maxLif`) on the entire position rather than on the marginal unhealthy portion. The borrower suffers a loss proportional to `(maxLif - 1) * fullDebt` instead of `(maxLif - 1) * marginalUnhealthyDebt`. For a position that is 1% unhealthy with `maxLif ≈ 1.15`, the over-liquidation multiplies the borrower's loss by roughly 15×.

---

### Likelihood Explanation

Market creation is fully permissionless — any EOA can call `touchMarket()` with `rcfThreshold = type(uint256).max` at the cost of gas only. The attacker then provides liquidity at competitive rates (or simply waits for organic lenders to fill the book), attracting borrowers. Liquidation is also permissionless (unless `liquidatorGate` is set, which the attacker controls as market creator). The attacker can act as both market creator and liquidator. Frontends and aggregators that list markets without surfacing `rcfThreshold` make it easy to lure unsuspecting borrowers.

---

### Recommendation

Add a validation bound on `rcfThreshold` inside `touchMarket()`, analogous to the existing checks on `lltv` and `maxLif`. For example, enforce a protocol-defined maximum:

```solidity
require(market.rcfThreshold <= MAX_RCF_THRESHOLD, RcfThresholdTooHigh());
```

A reasonable upper bound would be a value denominated in loan-token units (e.g., `1e24` for a 1 million token threshold), preventing the parameter from being set to a value that trivially satisfies the bypass condition for all realistic positions. Alternatively, document and enforce that `rcfThreshold` must be strictly less than `type(uint256).max`.

---

### Proof of Concept

1. Deploy `Midnight`.
2. Call `touchMarket()` with a `Market` struct where `rcfThreshold = type(uint256).max` and valid `lltv`/`maxLif` values (e.g., `lltv = LLTV_0 = 0.385e18`).
3. As a lender, call `take()` on a buy offer in this market to provide liquidity.
4. As a borrower, call `take()` on a sell offer to borrow, supplying collateral via `supplyCollateral()`.
5. Manipulate the oracle price downward so that `debt > maxDebt` by a small margin (e.g., 1%).
6. As the attacker/liquidator, call `liquidate()` with `repaidUnits = position.debt` (full debt).
7. Observe: the `RecoveryCloseFactorConditionsViolated` check passes because `... < type(uint256).max` is always true.
8. The borrower is fully liquidated. Under normal RCF enforcement with the same parameters, only ~1% of the debt would have been repayable. [7](#0-6) [8](#0-7)

        bytes32 id = toId(market);
        if (marketState[id].tickSpacing == 0) {

            require(market.collateralParams.length > 0, NoCollateralParams());
            require(market.collateralParams.length <= MAX_COLLATERALS, TooManyCollateralParams());
            address previousCollateralToken;
            for (uint256 i = 0; i < market.collateralParams.length; i++) {
                address collateralToken = market.collateralParams[i].token;
                require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
                uint256 lltv = market.collateralParams[i].lltv;
                require(isLltvAllowed(lltv), LltvNotAllowed());
                require(
                    market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_LOW)
