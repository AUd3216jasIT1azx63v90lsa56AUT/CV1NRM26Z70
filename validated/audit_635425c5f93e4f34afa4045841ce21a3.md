Audit Report

## Title
Referral fee not budgeted within `maxBuyerAssets`, causing revert that blocks a valid fill - (File: src/periphery/MidnightBundles.sol)

## Summary
In `buyWithUnitsTargetAndWithdrawCollateral`, the bundler pulls exactly `maxBuyerAssets` from `msg.sender` and spends `filledBuyerAssets` of that in the take loop. The referral fee is then computed as `filledBuyerAssets * referralFeePct / (WAD - referralFeePct)` and transferred from the bundler's remaining balance. When `maxBuyerAssets == filledBuyerAssets` and `referralFeePct > 0`, the bundler holds zero tokens post-loop, causing the `safeTransfer` to the referrer at line 103 to revert. The user loses gas and cannot complete an otherwise valid fill.

## Finding Description
**Code path:**

Line 66 pulls exactly `maxBuyerAssets` from `msg.sender`:
```solidity
pullToken(loanToken, msg.sender, maxBuyerAssets, loanTokenPermit);
```

The take loop (lines 71–86) causes MIDNIGHT to pull `resBuyerAssets` from the bundler per iteration, accumulating into `filledBuyerAssets`. After the loop, the bundler holds `maxBuyerAssets - filledBuyerAssets` tokens.

Line 102 computes the referral fee:
```solidity
uint256 referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
```

When `referralFeePct > 0` and `filledBuyerAssets > 0`, `referralFeeAssets` is strictly positive. Line 103 then attempts:
```solidity
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
```

If `maxBuyerAssets == filledBuyerAssets`, the bundler holds zero tokens at this point, so the ERC20 transfer fails. The subsequent line 104 subtraction `maxBuyerAssets - filledBuyerAssets - referralFeeAssets` would also underflow under Solidity 0.8.34 checked arithmetic, but is never reached.

**Root cause:** No check enforces `maxBuyerAssets >= filledBuyerAssets + referralFeeAssets`. The NatSpec at lines 47–48 correctly documents the total cost formula:
```solidity
/// @dev Total loan assets transferred from msg.sender is
/// filledBuyerAssets + filledBuyerAssets * referralFeePct / (WAD - referralFeePct).
```
but the contract never validates that `maxBuyerAssets` is large enough to cover both the fill and the referral fee before attempting the transfer.

**Existing guards** (lines 60–61) only check taker authorization and `referralFeePct < WAD`:
```solidity
require(taker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(taker, msg.sender), Unauthorized());
require(referralFeePct < WAD, PctExceeded());
```
Neither prevents the revert scenario.

## Impact Explanation
All offers are successfully taken (`filledUnits == targetUnits` passes at line 88), collateral withdrawals complete, but the transaction reverts at the referral-fee transfer (line 103). The user loses gas and cannot complete the fill in a single call even though the market state is valid and the fill is economically sound. Because the revert is deterministic for the given `(maxBuyerAssets, referralFeePct)` pair, the user is blocked unless they increase `maxBuyerAssets` or set `referralFeePct = 0`. This constitutes service unavailability / severe degradation of a core protocol function for any user employing tight slippage with a nonzero referral fee.

## Likelihood Explanation
Any unprivileged taker who sets `maxBuyerAssets` equal to the expected fill cost (tight slippage, a common and reasonable pattern) and passes a nonzero `referralFeePct` hits this revert. No privileged role, special state, or external dependency is required. The precondition is trivially reachable and repeatable across every call with these parameters.

## Recommendation
Before the referral fee transfer, add a check that the bundler's remaining balance covers the fee:
```solidity
require(maxBuyerAssets >= filledBuyerAssets + referralFeeAssets, InsufficientBudgetForReferralFee());
```
Alternatively, compute `referralFeeAssets` before the take loop and enforce `maxBuyerAssets >= filledBuyerAssets + referralFeeAssets` as a post-loop invariant, or document clearly (and enforce on-chain) that `maxBuyerAssets` must be set to at least `expectedFillCost * WAD / (WAD - referralFeePct)` to account for the fee.

## Proof of Concept
1. Deploy `MidnightBundles` against a local fork with a valid market and a sell offer for `N` units at price `P` (so `filledBuyerAssets = N * P`).
2. Call `buyWithUnitsTargetAndWithdrawCollateral` with:
   - `targetUnits = N`
   - `maxBuyerAssets = N * P` (tight slippage, exactly the fill cost)
   - `referralFeePct = 1e16` (1%, any nonzero value)
   - `referralFeeRecipient = <any address>`
3. Observe: the take loop succeeds, `filledUnits == targetUnits` passes, but the transaction reverts at line 103 with an ERC20 insufficient-balance error because the bundler holds 0 tokens after the loop.
4. Confirm fix: set `maxBuyerAssets = N * P * WAD / (WAD - referralFeePct)` (rounded up) and observe the transaction succeeds.
