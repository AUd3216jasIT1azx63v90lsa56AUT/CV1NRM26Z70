Audit Report

## Title
Maker callback can call `setConsumed` mid-`take` to inflate consumed and grief taker multicall fills - (File: src/Midnight.sol)

## Summary
During `take`, `consumed[offer.maker][offer.group]` is incremented and bounds-checked at lines 367ŌĆō373 before any external call is made. For a buy offer, `offer.callback` is assigned as `buyerCallback` (line 420), and `IBuyCallback(buyerCallback).onBuy(...)` is invoked at lines 448ŌĆō452 with no reentrancy guard protecting `setConsumed`. A maker-controlled callback can therefore call `setConsumed` to inflate `consumed` to `maxAssets` mid-`take`, causing every subsequent fill of the same offer within the same `multicall` to revert with `ConsumedAssets`, atomically reverting the taker's entire bundle.

## Finding Description

**Step 1 ŌĆö consumed is incremented and capped before any callback:**

`take` writes and bounds-checks `consumed` at lines 367ŌĆō373, before any external call is made:

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Step 2 ŌĆö For a buy offer, `buyerCallback` is the maker's own address:**

Line 420: `address buyerCallback = offer.buy ? offer.callback : takerCallback;` ŌĆö for buy offers, `buyerCallback = offer.callback`, which is fully maker-controlled.

**Step 3 ŌĆö The only transient lock set during `take` is the liquidation lock for the seller:**

Line 444: `bool wasLocked = UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true);` ŌĆö this lock only prevents liquidating the seller during callbacks. There is no transient lock or mutex preventing `setConsumed` from being called during the `onBuy` callback at lines 448ŌĆō452.

**Step 4 ŌĆö `setConsumed` only requires authorization and a non-decreasing amount:**

Lines 723ŌĆō728:
```solidity
function setConsumed(bytes32 group, uint256 amount, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    require(amount >= consumed[onBehalf][group], AlreadyConsumed());
    consumed[onBehalf][group] = amount;
}
```
No check prevents this from being called during an active `take`.

**Exploit flow:**
1. Maker deploys malicious contract `M` and calls `setIsAuthorized(M, true, maker)` ŌĆö a normal, permissionless operation.
2. Maker publishes a buy offer with `offer.callback = M` and `offer.maxAssets = 1000`.
3. Taker submits `multicall([take(offer, 400 units), take(offer, 400 units)])`.
4. First `take`: `consumed[maker][group] += 400` ŌåÆ 400 Ōēż 1000, passes.
5. `M.onBuy(...)` is called. Inside, `M` calls `midnight.setConsumed(group, 1000, maker)`. Authorization check passes (`isAuthorized[maker][M] == true`). `AlreadyConsumed` check passes (`1000 >= 400`). `consumed[maker][group]` is now 1000.
6. First `take` completes normally.
7. Second `take`: `consumed[maker][group] += 400` ŌåÆ 1400 > 1000 ŌåÆ reverts `ConsumedAssets`.
8. Entire `multicall` reverts; taker's atomic bundle fails and all gas is wasted.

**Why existing checks fail:**
- `AlreadyConsumed` (line 725) only prevents *decreasing* consumed; it explicitly allows any upward inflation, so the re-entrant call passes unconditionally.
- The `LIQUIDATION_LOCK_SLOT` transient guard (line 444) only protects against liquidating the seller during callbacks ŌĆö it has no effect on `setConsumed`.
- The Certora `Consume.spec` rule `takeConsumedBoundedByMax` (lines 59ŌĆō64) does not catch this because the callback sets consumed to exactly `maxAssets`, so `consumed <= maxAssets` still holds after the single `take` call. The rule `takeConsumedDelta` (lines 67ŌĆō75) does not apply because it is scoped to units mode (`offer.maxAssets == 0`), while the exploit uses assets mode. The rule `takeConsumedAtMaxUnchangedAssets` (lines 89ŌĆō97) checks the pre-`take` state, not the mid-`take` state mutated by the callback.

## Impact Explanation
A maker can deterministically grief any taker who attempts multiple partial fills of the same offer atomically via `multicall`. The entire atomic bundle reverts, wasting the taker's gas and preventing the intended fill. The maker's offer is also permanently blocked from further fills (self-harm griefing). No direct fund loss occurs, but the attack constitutes a concrete, repeatable denial-of-service against atomic multi-fill workflows, which are a core use case of the protocol's `multicall` and group/offer-cap design.

## Likelihood Explanation
All preconditions are normal, permissionless protocol operations: deploying a callback contract and calling `setIsAuthorized` are both standard flows already used in the protocol. No special privileges, oracle manipulation, or token-owner action is required. The attack is deterministic ŌĆö it always succeeds given the preconditions ŌĆö and the maker can deploy a fresh callback and offer for each target taker. Any taker using `multicall` for multi-fill atomic transactions is a potential victim.

## Recommendation
Introduce a transient per-maker-group lock during `take` that prevents `setConsumed` from being called for the active `(maker, group)` pair while a `take` is in progress. Concretely, before the consumed increment at line 367, set a transient slot keyed on `(offer.maker, offer.group)` to `true`, and clear it after the callbacks complete. In `setConsumed`, revert if the corresponding transient slot is set. Alternatively, snapshot `consumed[offer.maker][offer.group]` before the callbacks and re-validate the cap after all callbacks complete, using the snapshotted pre-callback value as the baseline.

## Proof of Concept
1. Deploy `MaliciousCallback` implementing `IBuyCallback.onBuy` that calls `midnight.setConsumed(offer.group, offer.maxAssets, maker)`.
2. Maker calls `midnight.setIsAuthorized(address(MaliciousCallback), true, maker)`.
3. Maker constructs a buy offer with `offer.callback = address(MaliciousCallback)`, `offer.maxAssets = 1000`, `offer.group = someGroup`.
4. Taker calls `midnight.multicall([abi.encodeCall(take, (offer, ..., 400 units, ...)), abi.encodeCall(take, (offer, ..., 400 units, ...))])`.
5. Observe: first `take` succeeds, `MaliciousCallback.onBuy` sets `consumed[maker][group] = 1000`, second `take` reverts with `ConsumedAssets`, entire `multicall` reverts.
6. Expected: both fills succeed with `consumed[maker][group] = 800`. Actual: full revert.
