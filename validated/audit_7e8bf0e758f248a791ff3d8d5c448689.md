Audit Report

## Title
Fee-on-transfer loanToken causes `repayAndWithdrawCollateral` to always revert due to bundler balance shortfall - (File: src/periphery/MidnightBundles.sol)

## Summary

In `repayAndWithdrawCollateral`, `units` is computed from the nominal `assets` parameter before `pullToken` is called. When the loan token charges a transfer fee, the bundler receives fewer tokens than `assets`, but `Midnight.repay` subsequently attempts to pull the full `units` amount from the bundler. This causes an unconditional revert for any fee-on-transfer loan token when `referralFeePct = 0`, and for any token whose per-transfer fee exceeds `referralFeeAssets` otherwise.

## Finding Description

**Code path** (`src/periphery/MidnightBundles.sol` lines 328–347):

```solidity
address loanToken = market.loanToken;
uint256 referralFeeAssets = assets.mulDivDown(referralFeePct, WAD);  // L329
uint256 units = assets - referralFeeAssets;                           // L330
pullToken(loanToken, msg.sender, assets, loanTokenPermit);            // L331
forceApproveMax(loanToken, MIDNIGHT);                                 // L332

IMidnight(MIDNIGHT).repay(market, units, onBehalf, address(0), "");  // L334
```

`pullToken` resolves to `SafeTransferLib.safeTransferFrom(token, from, address(this), amount)` with `amount = assets` (lines 384 and 396). For a fee-on-transfer token with fee `f`, the bundler receives `assets - f`, not `assets`.

`units = assets - referralFeeAssets` is derived from the nominal input, not the actual received balance. Inside `Midnight.repay` (line 511 and 520):

```solidity
address payer = callback != address(0) ? callback : msg.sender;  // callback=address(0) → payer=bundler
...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
```

Midnight pulls exactly `units` from the bundler. The bundler holds `assets - f`. The pull fails whenever `units > assets - f`, i.e., `f > referralFeeAssets`. With `referralFeePct = 0`, this reduces to `f > 0` — true for any fee-on-transfer token.

**Existing protections:** The only token disclaimer in `MidnightBundles.sol` (lines 23–24) excludes tokens that revert on `approve(0); approve(max)`. Fee-on-transfer tokens are not excluded. No balance snapshot or received-amount reconciliation exists between `pullToken` and the `repay` call.

## Impact Explanation

Every call to `repayAndWithdrawCollateral` with a fee-on-transfer loan token reverts at the `safeTransferFrom` inside `Midnight.repay`. Borrowers with open positions in such markets cannot repay debt or withdraw collateral through the bundler, permanently blocking the intended repay-and-withdraw flow for that token type. This constitutes a permanent freeze of user state reachable through a standard user action.

## Likelihood Explanation

The precondition is a market whose `loanToken` charges a transfer fee. Tokens with configurable fees (owner-enabled) or inherently fee-on-transfer tokens satisfy this. No privileged action is required; any borrower with an open position in such a market triggers the revert by calling `repayAndWithdrawCollateral` with `referralFeePct = 0`. The condition is deterministic and repeatable on every call.

## Recommendation

Replace the nominal `assets` basis for `units` with the actual received amount. Snapshot the bundler's balance before and after `pullToken`, then derive `units` from the delta:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
pullToken(loanToken, msg.sender, assets, loanTokenPermit);
uint256 received = IERC20(loanToken).balanceOf(address(this)) - balanceBefore;
uint256 referralFeeAssets = received.mulDivDown(referralFeePct, WAD);
uint256 units = received - referralFeeAssets;
```

This ensures `units` never exceeds the bundler's actual balance regardless of token transfer behavior.

## Proof of Concept

1. Deploy a mock ERC-20 loan token that deducts a 1% fee on every `transferFrom`.
2. Create a Midnight market with this token as `loanToken`.
3. Open a borrow position on behalf of `borrower`.
4. Call `MidnightBundles.repayAndWithdrawCollateral` with `assets = 1000`, `referralFeePct = 0`.
5. Observe: `pullToken` delivers 990 tokens to the bundler; `Midnight.repay` attempts to pull 1000 tokens from the bundler; `safeTransferFrom` reverts due to insufficient balance.
6. Confirm the revert is deterministic across all positive `assets` values and all fee magnitudes `f > 0`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

        if (permit.kind == PermitKind.ERC2612) {

                abi.decode(permit.data, (uint256, uint8, bytes32, bytes32));
            // Tolerate revert: a third party may have already consumed the permit.

                    IPermit2.SignatureTransferDetails(address(this), amount),
                    from,
                    signature
                );
        } else {
            SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
