Audit Report

## Title
Maker-controlled `offer.callback` designates arbitrary third-party as `payer` for buy offers, enabling drain of victim's loan tokens without maker payment - (`File: src/Midnight.sol`)

## Summary
In `Midnight.take`, when `offer.buy = true`, the `payer` for both `safeTransferFrom` calls is derived directly from `offer.callback`, a field freely set by the maker with no authorization check tying the callback address to the maker. A malicious maker can pair a self-deployed permissive ratifier with `offer.callback = victimContract` to cause the protocol to pull `buyerAssets` from the victim instead of the maker, while the maker receives `buyerCreditIncrease` credit at zero cost. The existing ratifier gate is insufficient because the maker controls which ratifier they authorize.

## Finding Description

**Exact code path — `src/Midnight.sol` L420–456:**

```solidity
address buyerCallback = offer.buy ? offer.callback : takerCallback;   // L420
address payer = buyerCallback != address(0)
    ? buyerCallback
    : (offer.buy ? buyer : msg.sender);                                // L422

// state mutations (credit to offer.maker) ...
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);           // L410

if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback).onBuy(...) == CALLBACK_SUCCESS,    // L448-451
        WrongBuyCallbackReturnValue()
    );
}
SafeTransferLib.safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets); // L455
SafeTransferLib.safeTransferFrom(loanToken, payer, receiver, sellerAssets);                    // L456
```

**Root cause:** There is no check of the form `require(isAuthorized[offer.maker][offer.callback], ...)` analogous to the ratifier gate at L355. The `offer.callback` field is included in the offer hash (`HashLib.hashOffer`, L129) and is therefore covered by the ratifier's signature — but the maker controls which ratifier they authorize. Any maker can call `setIsAuthorized(permissiveRatifier, true)` (L731–734) and deploy a ratifier that unconditionally returns `CALLBACK_SUCCESS`, bypassing the only structural guard.

**Exploit flow:**
1. Attacker deploys `PermissiveRatifier` returning `CALLBACK_SUCCESS` for any offer.
2. Attacker calls `midnight.setIsAuthorized(permissiveRatifier, true)` — no privilege required.
3. Attacker identifies `victimContract`: a contract that (a) holds an ERC-20 approval to Midnight for `loanToken` and (b) implements `IBuyCallback.onBuy()` returning `CALLBACK_SUCCESS` without validating the `buyer` argument (e.g., a general-purpose lending vault or aggregator router).
4. Attacker constructs `offer` with `offer.buy = true`, `offer.maker = attacker`, `offer.callback = victimContract`, `offer.ratifier = permissiveRatifier`.
5. Attacker (or any third party) calls `take(offer, ...)`.
6. `buyerCallback = offer.callback = victimContract`; `payer = victimContract`.
7. `victimContract.onBuy(...)` is called → returns `CALLBACK_SUCCESS`.
8. `safeTransferFrom(loanToken, victimContract, address(this), buyerAssets - sellerAssets)` executes.
9. `safeTransferFrom(loanToken, victimContract, receiver, sellerAssets)` executes.
10. `buyerPos.credit += buyerCreditIncrease` (L410) — attacker gains credit; victim paid.

**Why existing checks fail:**
- L346 (`TakerUnauthorized`): gates the taker, irrelevant to callback.
- L355 (`RatifierUnauthorized`): requires `isAuthorized[offer.maker][offer.ratifier]` — the attacker satisfies this trivially by authorizing their own permissive ratifier.
- L356 (`RatifierFail`): the permissive ratifier returns `CALLBACK_SUCCESS` unconditionally.
- The Certora rule `takeOnlyExplicitPayer` (L95–115 of `OnlyExplicitPayerCanLoseTokens.spec`) proves only that tokens are pulled from "the buyerCallback that returned `CALLBACK_SUCCESS`" — it sets `callbackAllowed = true` for any address returning `CALLBACK_SUCCESS` (L62–64) regardless of maker authorization. The spec does not model the constraint that the callback must be authorized by the maker, so this attack class is outside its proof scope.

## Impact Explanation
The attacker gains `buyerCreditIncrease` credit in the market (L410) — a real, withdrawable asset — while paying nothing. The victim contract loses up to `offer.maxAssets` / `offer.maxUnits` worth of loan tokens per invocation. The attack is repeatable across multiple `take` calls until the victim's approval or balance is exhausted. This constitutes direct theft of assets from a third party, violating the core protocol invariant that every credit increase must be matched by a corresponding payment from the buyer.

## Likelihood Explanation
**Attacker requirements:** no privileged access; any address can deploy a permissive ratifier and call `setIsAuthorized`. **Victim requirements:** (1) ERC-20 approval to Midnight for the loan token — standard for any contract participating as a buy-callback payer; (2) `onBuy` returns `CALLBACK_SUCCESS` without validating the `buyer` argument — realistic for general-purpose vaults, aggregators, or routers that do not enforce a specific buyer identity. The attack requires no oracle manipulation, no flash loan, no admin key, and no user mistake. It is fully repeatable up to the victim's approval limit.

## Recommendation
Add an explicit authorization check for `offer.callback` mirroring the ratifier gate:

```solidity
require(
    offer.callback == address(0) || isAuthorized[offer.maker][offer.callback],
    CallbackUnauthorized()
);
```

This ensures the maker must have explicitly authorized the callback address on-chain before it can be designated as payer, eliminating the ability to point the callback at an arbitrary third-party contract.

## Proof of Concept

**Minimal Foundry test outline:**

```solidity
// 1. Deploy a permissive ratifier
PermissiveRatifier rat = new PermissiveRatifier(); // returns CALLBACK_SUCCESS always

// 2. Deploy victim: has loanToken approval to Midnight, onBuy returns CALLBACK_SUCCESS
VictimCallback victim = new VictimCallback();
deal(address(loanToken), address(victim), 1000e18);
vm.prank(address(victim));
loanToken.approve(address(midnight), type(uint256).max);

// 3. Attacker authorizes permissive ratifier
vm.prank(attacker);
midnight.setIsAuthorized(address(rat), true, attacker);

// 4. Construct malicious offer
Offer memory offer;
offer.buy      = true;
offer.maker    = attacker;
offer.ratifier = address(rat);
offer.callback = address(victim);   // <-- victim designated as payer
offer.market   = market;
offer.expiry   = block.timestamp + 1 days;
offer.tick     = MAX_TICK;
offer.maxUnits = 1000e18;

// 5. Taker (attacker themselves) triggers take
collateralize(market, attacker, 1000e18); // attacker posts collateral as seller-side
vm.prank(attacker);
midnight.take(offer, hex"", 1000e18, attacker, attacker, address(0), hex"");

// 6. Assert: victim lost tokens, attacker gained credit
assertEq(loanToken.balanceOf(address(victim)), 0);
assertGt(midnight.creditOf(id, attacker), 0);
```

Expected result: victim's loan tokens are transferred to Midnight/receiver; attacker holds credit with zero personal expenditure.
