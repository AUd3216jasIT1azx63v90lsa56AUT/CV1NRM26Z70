### Title
Former pool owner retains CORRUPTED fund sweep via stale `recoveryAddress` after ownership transfer - (File: src/ConfidencePool.sol)

### Summary
`ConfidencePool` inherits `Ownable2Step` for ownership management, but `transferOwnership`/`acceptOwnership` does not clear or update `recoveryAddress`. A former pool owner who set `recoveryAddress` to their own address retains the ability to receive the entire pool (stakers' principal + bonus) in any CORRUPTED resolution, even after they have transferred ownership to a new party.

### Finding Description
`ConfidencePool` stores two independent authority addresses: `owner` (managed by `Ownable2Step`) and `recoveryAddress` (set at `initialize` time and updatable only by the current owner via `setRecoveryAddress`). [1](#0-0) [2](#0-1) 

At pool creation, the factory passes `msg.sender` as both `owner_` and the caller-supplied `recoveryAddress` (the typical sponsor pattern — the sponsor sets recovery to their own address): [3](#0-2) 

When the pool owner later calls `transferOwnership(newOwner)` and the new owner calls `acceptOwnership()`, the `Ownable2Step` machinery transfers `owner` to the new address, but `recoveryAddress` is never touched. No override of `_transferOwnership` exists in `ConfidencePool` to clear or redirect it. [4](#0-3) 

After the transfer, the former owner's address remains as `recoveryAddress`. `claimCorrupted()` is permissionless — any caller triggers a full sweep of the pool balance to `recoveryAddress`: [5](#0-4) 

Similarly, `sweepUnclaimedCorrupted()` and `sweepUnclaimedBonus()` both send to `recoveryAddress` without any ownership check: [6](#0-5) [7](#0-6) 

The new owner can call `setRecoveryAddress` to fix this, but there is a race: `claimCorrupted()` is callable by anyone the instant `flagOutcome(CORRUPTED, ...)` is mined, so the former owner can front-run the new owner's `setRecoveryAddress` transaction with a `claimCorrupted()` call, draining the entire pool to their own address before the update lands.

### Impact Explanation
Under a CORRUPTED (bad-faith) resolution, `claimCorrupted()` sweeps `stakeToken.balanceOf(address(this))` — the full pool including all stakers' principal and bonus — to `recoveryAddress`. If `recoveryAddress` still points to the former owner, every staker loses their principal to a party who no longer has any legitimate claim on the pool. The new owner, who accepted the pool in good faith, has no recourse once the sweep executes. [8](#0-7) 

### Likelihood Explanation
Pool ownership transfer is a documented, supported operation (the factory enforces `IAgreement(agreement).owner() == msg.sender` at creation, but agreement ownership can change independently). A sponsor selling or delegating a pool is a realistic lifecycle event. The new owner has no on-chain prompt to update `recoveryAddress` after `acceptOwnership()`, and the window between `flagOutcome` and `claimCorrupted` is a single block — easily front-run by the former owner monitoring the mempool. [9](#0-8) 

### Recommendation
Override `_transferOwnership` in `ConfidencePool` to atomically redirect `recoveryAddress` to the incoming owner, or add a dedicated `transferOwnershipAndChangeRecovery(address newOwner, address newRecovery)` function (mirroring the IdRegistry fix of `transferAndChangeRecovery`) that updates both in one transaction. At minimum, emit a prominent warning event when `acceptOwnership` fires while `recoveryAddress != pendingOwner`, so the new owner is alerted on-chain.

### Proof of Concept
1. Sponsor (Alice) calls `factory.createPool(..., recoveryAddress=Alice, ...)` → pool deployed with `owner=Alice`, `recoveryAddress=Alice`.
2. Alice sells the pool: calls `pool.transferOwnership(Bob)`.
3. Bob calls `pool.acceptOwnership()` → `owner=Bob`, but `recoveryAddress` is still `Alice`.
4. Moderator calls `pool.flagOutcome(CORRUPTED, false, address(0))`.
5. Alice (or any front-runner) calls `pool.claimCorrupted()` before Bob's `setRecoveryAddress(Bob)` lands.
6. `stakeToken.balanceOf(pool)` — all stakers' principal + bonus — is transferred to Alice.
7. Bob's subsequent `setRecoveryAddress(Bob)` succeeds but there is nothing left to recover. [5](#0-4)

    function claimCorrupted() external nonReentrant {
        if (outcome != PoolStates.Outcome.CORRUPTED) revert OutcomeNotSet();
        if (goodFaith && bountyClaimed < bountyEntitlement) revert MustClaimBountyFirst();

        // aderyn-fp-next-line(reentrancy-state-change)
        uint256 toSweep = stakeToken.balanceOf(address(this));
        if (toSweep == 0) revert NothingToSweep();

        // Clamp the decrement — `toSweep` can exceed the original reserve when post-resolution
        // donations have inflated the balance.
        corruptedReserve = toSweep <= corruptedReserve ? corruptedReserve - toSweep : 0;
        if (!goodFaith) {
            bountyClaimed = bountyEntitlement;
        }
        if (!claimsStarted) claimsStarted = true;
        stakeToken.safeTransfer(recoveryAddress, toSweep);

        emit ClaimCorrupted(msg.sender, recoveryAddress, toSweep);
    }
