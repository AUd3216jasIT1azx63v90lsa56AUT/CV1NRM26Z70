Audit Report

## Title
Zero-price sell offer in `maxAssets` mode allows unlimited reuse — `consumed` never increments - (File: src/Midnight.sol)

## Summary
When a sell offer (`offer.buy == false`) is created with `tick = 0` (yielding `tickToPrice == 0`) and `offer.maxAssets > 0`, the assets-mode consumed accounting computes `sellerAssets = mulDivUp(units * 0, WAD) = 0` on every call to `take`. Because `consumed` is incremented by zero on each call, the `require(newConsumed <= offer.maxAssets)` guard is permanently satisfied and the offer can be taken an unlimited number of times, completely bypassing the `maxAssets` cap.

## Finding Description

**Root cause — `src/Midnight.sol` lines 358–369:**

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);          // line 358: tickToPrice(0) == 0
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice; // line 361: sellerPrice = 0
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD)
                                 : units.mulDivUp(sellerPrice, WAD);        // line 364: mulDivUp(units*0, WAD) = 0

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets; // line 368: += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());               // line 369: 0 <= N, always passes
}
```

`UtilsLib.mulDivUp` is defined as `(x * y + (d - 1)) / d`. With `x = units`, `y = 0`, `d = WAD`: `(0 + WAD - 1) / WAD = 0` by integer division. [1](#0-0) 

`tickToPrice(0) == 0` is a confirmed protocol invariant, proven in the Certora spec and the test suite. [2](#0-1) [3](#0-2) 

Tick 0 is always accessible because `0 % tickSpacing == 0` for any `tickSpacing`. [4](#0-3) 

**Exploit flow:**
1. Maker (borrower) creates a sell offer with `tick = 0`, `maxAssets = N` (any positive value), `maxUnits = 0`
2. Taker calls `take(offer, ..., units = K, ...)` repeatedly
3. Each call: `sellerAssets = 0`, `consumed += 0`, check `0 <= N` passes
4. Each call: `sellerPos.debt += K`, `buyerPos.credit += K`
5. After M calls: `consumed[maker][group] = 0`, `debtOf(maker) = M * K`

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — always passes since `newConsumed = 0` [5](#0-4) 
- Post-take health check `require(... || isHealthy(...), SellerIsLiquidatable())` — limits per-take debt increase to the seller's current health headroom, but does not fix the consumed accounting; as collateral value fluctuates the window reopens [6](#0-5) 
- The Certora `Consume.spec` rule `takeConsumedBoundedByMax` uses `NONDET` summaries for `tickToPrice` and `mulDivUp`, so the concrete zero-price path is not covered [7](#0-6) 
- All existing zero-price tests use `maxUnits` mode, not `maxAssets` mode; all `maxAssets` tests use `MAX_TICK` (non-zero price) [8](#0-7) [9](#0-8) 

## Impact Explanation
The `maxAssets` cap — the sole mechanism preventing an assets-mode offer from being overfilled — is permanently bypassed. The offer is reusable indefinitely: `consumed` stays at zero while the maker's debt grows without bound (bounded only by the per-call health check). The maker's `maxAssets` intent is completely unenforceable. As the maker's debt accumulates and collateral value fluctuates, the position becomes liquidatable and any shortfall becomes bad debt socialized across all lenders in the market.

## Likelihood Explanation
**Preconditions:**
1. A sell offer exists with `tick = 0` and `offer.maxAssets > 0` — tick 0 is always accessible (`0 % tickSpacing == 0`), and a maker may set it as a near-zero-rate offer
2. Taker is any unprivileged address (not the maker itself due to `SelfTake` check)

**Feasibility:** Fully reachable with no special privileges. The taker pays at most `mulDivUp(units, _settlementFee, WAD)` per call in loan tokens (0 if settlement fee is 0). Repeatable in a loop or across blocks. The `DEFAULT_TICK_SPACING = 4` means tick 0 is always a valid tick. [10](#0-9) 

## Recommendation
Add a guard before the `maxAssets` consumed update to require that `sellerAssets > 0` (for sell offers) or `buyerAssets > 0` (for buy offers) when `maxAssets > 0`, or alternatively require `offerPrice > 0` when `maxAssets > 0`. A minimal fix:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0, ZeroPriceMaxAssetsOffer()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, document that `maxAssets` mode is undefined/unsupported for zero-price ticks and add a validation in `take` that rejects `maxAssets > 0` when `offerPrice == 0`.

## Proof of Concept
Minimal Foundry test (extend `TakeTest`):

```solidity
function testZeroPriceMaxAssetsBypass() public {
    uint256 maxAssetsCap = 1e18;
    uint256 units = 1e18;

    borrowerOffer.tick = 0;          // tickToPrice(0) == 0
    borrowerOffer.maxUnits = 0;
    borrowerOffer.maxAssets = maxAssetsCap;

    collateralize(market, borrower, units * 100); // enough collateral

    // First take: consumed should reach maxAssetsCap but doesn't
    take(units, lender, borrowerOffer);
    assertEq(midnight.consumed(borrower, borrowerOffer.group), 0); // BUG: still 0

    // Second take: should revert with ConsumedAssets but doesn't
    take(units, lender, borrowerOffer);
    assertEq(midnight.consumed(borrower, borrowerOffer.group), 0); // BUG: still 0

    // Maker's debt grew unboundedly despite maxAssets cap
    assertEq(midnight.debtOf(id, borrower), units * 2);
}
```
