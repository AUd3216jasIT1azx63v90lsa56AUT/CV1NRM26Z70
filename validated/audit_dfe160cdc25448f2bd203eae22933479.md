Audit Report

## Title
Authorized Operator Can Grief Victim's Pre-Signed EcrecoverAuthorizer Nonce via No-Op Authorization - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` unconditionally increments `nonce[authorization.authorizer]` on every successful call, regardless of whether the downstream `Midnight.setIsAuthorized` call produces any state change. Any address already authorized by the victim in Midnight can craft a no-op `Authorization` signed with its own key, consume the victim's current nonce, and permanently invalidate any pre-signed `Authorization{nonce=N}` the victim intended to submit. The attack is repeatable for every subsequent nonce as long as the attacker remains authorized.

## Finding Description

**Root cause:** The nonce increment at `EcrecoverAuthorizer.sol` line 26 is unconditional — it fires before any downstream state-change check:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

The signer check at lines 33–36 permits any address already authorized by the authorizer in Midnight to submit on their behalf:

```solidity
require(
    signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

`Midnight.setIsAuthorized` at line 733 performs an unconditional assignment with no prior-state check:

```solidity
isAuthorized[onBehalf][authorized] = newIsAuthorized;
```

The contract explicitly documents "No-ops are allowed" at line 168, confirming this is an intentional design choice that creates the attack surface.

**Exploit flow:**
1. Victim `V` authorizes attacker `A` in Midnight: `isAuthorized[V][A] = true` (normal relayer/adapter setup).
2. `V` pre-signs off-chain: `Authorization{authorizer=V, authorized=criticalContract, isAuthorized=true, nonce=N}`.
3. `A` crafts: `Authorization{authorizer=V, authorized=neverAuthorizedAddr, isAuthorized=false, nonce=N, deadline=future}` and signs it with `A`'s own key.
4. `A` calls `ecrecoverAuthorizer.setIsAuthorized(noOpAuth, aSig)`.
5. All checks pass: deadline valid; `nonce[V]==N` (incremented to N+1); `ecrecover` returns `A`; `isAuthorized[V][A]==true`.
6. `Midnight.setIsAuthorized(neverAuthorizedAddr, false, V)` executes as a no-op (sets `false` for an address never authorized).
7. `nonce[V]` is now `N+1`. `V`'s pre-signed `Authorization{nonce=N}` now reverts with `InvalidNonce`.

**Why existing checks fail:** The Certora spec (`certora/specs/EcrecoverAuthorizer.spec`) only verifies that a successful call increments the authorizer's nonce and that expired/wrong/reused nonces revert. It does not assert that nonce consumption requires a meaningful state change in Midnight.

## Impact Explanation
The victim's pre-signed `Authorization{nonce=N}` is permanently invalidated. If the victim cannot immediately re-sign (hardware wallet unavailable, pre-signed batch, or a contract authorizer that pre-committed to nonce `N`), their ability to delegate via `EcrecoverAuthorizer` is frozen until a new signature is produced. The attack is repeatable: the attacker can consume every subsequent nonce the victim pre-signs for as long as the attacker remains authorized in Midnight, creating a sustained griefing/DoS on the victim's off-chain signing workflow.

## Likelihood Explanation
Required preconditions are both common in normal protocol usage:
- Victim has authorized the attacker in Midnight (`isAuthorized[victim][attacker] = true`) — a routine interaction when setting up relayers, smart contract wallets, or protocol adapters.
- Victim has a pre-signed `Authorization` in flight (in a mempool, stored in a UI, or shared with a relayer).

The attack costs one transaction, requires no special protocol-level privileges beyond being an authorized operator (a user-level delegation, not a governance/admin role), and is repeatable indefinitely until the victim revokes the attacker's Midnight authorization.

## Recommendation
Before incrementing the nonce, verify that the `Authorization` would produce a meaningful state change, or restrict who can submit on behalf of the authorizer to only the authorizer themselves (removing the `isAuthorized` delegation path in `EcrecoverAuthorizer`). The simplest targeted fix is to check whether `Midnight.isAuthorized(authorization.authorizer, authorization.authorized) != authorization.isAuthorized` before consuming the nonce, and revert or skip the nonce increment if the call would be a no-op. Alternatively, move the nonce increment to after the downstream call and only increment if the call succeeds with a state change.

## Proof of Concept
```solidity
// 1. V authorizes A in Midnight
midnight.setIsAuthorized(A, true, V); // called by V

// 2. V pre-signs off-chain (simulated here):
Authorization memory victimAuth = Authorization({
    authorizer: V,
    authorized: criticalContract,
    isAuthorized: true,
    nonce: 0,       // nonce[V] == 0
    deadline: type(uint256).max
});
// victimSig = sign(victimAuth) with V's key

// 3. A crafts a no-op authorization and signs with A's key
Authorization memory noOpAuth = Authorization({
    authorizer: V,
    authorized: address(0xdead), // never authorized
    isAuthorized: false,
    nonce: 0,                    // same nonce
    deadline: type(uint256).max
});
Signature memory aSig = sign(noOpAuth, A_privateKey);

// 4. A submits the no-op
ecrecoverAuthorizer.setIsAuthorized(noOpAuth, aSig);
// nonce[V] is now 1

// 5. V's pre-signed authorization now reverts
ecrecoverAuthorizer.setIsAuthorized(victimAuth, victimSig);
// reverts: InvalidNonce (nonce[V]==1, authorization.nonce==0)
```
