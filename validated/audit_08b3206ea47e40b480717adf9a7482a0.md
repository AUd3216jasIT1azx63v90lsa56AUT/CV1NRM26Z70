### Title
Fee-on-Transfer Loan Token in `repay()` Inflates `withdrawable`, Causing Lender Fund Loss — (File: src/Midnight.sol)

### Summary

`repay()` credits `marketState[id].withdrawable` with the full `units` argument before calling `safeTransferFrom`, which does not verify the actual received amount. If the market's loan token charges a transfer fee, the contract receives fewer tokens than it records as withdrawable. Over time the accounting surplus grows until the last lenders to call `withdraw()` cannot be paid, suffering a direct loss of funds.

### Finding Description

**Root cause — `repay()`, lines 508–520:** [1](#0-0) 

```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);   // ← full units credited

address payer = callback != address(0) ? callback : msg.sender;
// ...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
// ← only (units − fee) actually arrives
```

`SafeTransferLib.safeTransferFrom` checks only the boolean return value; it does not measure the balance delta: [2](#0-1) 

So when the loan token deducts a transfer fee, `withdrawable` is over-stated by exactly that fee on every repayment. Lenders who call `withdraw()` later are paid from a pool that is smaller than the recorded `withdrawable`, and the last lenders to withdraw receive nothing (or the call reverts).

**Secondary surface — `supplyCollateral()`, lines 533–545:** [3](#0-2) 

The same pattern applies to collateral tokens: the position is credited with `assets` but only `assets − fee` arrives. This inflates the borrower's collateral value, allowing them to borrow more than their real collateral supports. When the position is liquidated, `liquidate()` attempts to `safeTransfer` the full recorded collateral amount out, which either reverts (if the contract is short) or drains other users' collateral of the same token.

**Why the TOKEN SAFETY REQUIREMENTS comment does not close this:**

The NatSpec at lines 133–140 lists "Midnight must send/receive exactly the requested amount on transfers" as an *assumption* for correct behaviour, not an on-chain enforcement. There is no guard preventing a market from being created with a fee-on-transfer loan or collateral token. The `core_invariants.external_calls` entry in `live_context.json` explicitly states "fee-on-transfer … should be tested if not explicitly excluded," and `SECURITY.md` contains no such exclusion. [4](#0-3) 

### Impact Explanation

- **`repay()` path:** Every repayment with a fee-on-transfer loan token leaves `withdrawable` larger than the actual contract balance. Lenders who withdraw early are made whole at the expense of later lenders, who lose funds proportional to the cumulative fee shortfall. This is a direct, permanent loss of principal for affected lenders.
- **`supplyCollateral()` path:** Inflated collateral records allow a borrower to take on more debt than their real collateral covers, creating bad debt that is socialized across all lenders in the market.

### Likelihood Explanation

Midnight markets are permissionlessly created with any ERC-20 as the loan or collateral token. A malicious actor can deliberately deploy a market using a fee-on-transfer token (or a token they control whose fee they can activate after liquidity is deposited) to trap lenders. Even without malicious intent, legitimate fee-bearing tokens (e.g., tokens with protocol-level transfer taxes) can be used innocently, triggering the same accounting drift. No privileged access is required.

### Recommendation

Measure the contract's token balance before and after each inbound `safeTransferFrom` and use the delta — not the argument — for all accounting updates. For example, in `repay()`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
// use `received` instead of `units` for withdrawable and debt accounting
```

Apply the same pattern in `supplyCollateral()` and anywhere else an inbound transfer feeds protocol accounting.

### Proof of Concept

1. Deploy a loan token `FeeToken` that deducts 2% on every `transferFrom`.
2. Create a Midnight market with `FeeToken` as the loan token.
3. Maker A lends 100 units (buys credit). Seller B borrows 100 units (sells credit).
4. Seller B calls `repay(market, 100, B, ...)`.
   - `withdrawable` increases by 100.
   - Contract receives only 98 `FeeToken`.
5. Maker A calls `withdraw(market, 50, A, A)` — succeeds (48 tokens available after fee on the outbound transfer, but the accounting allows it).
6. A second lender C (who also holds 50 credit units from a separate take) calls `withdraw(market, 50, C, C)` — **reverts** because the contract balance is insufficient.
7. Lender C's 50 credit units are permanently unclaimable; funds are lost.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
