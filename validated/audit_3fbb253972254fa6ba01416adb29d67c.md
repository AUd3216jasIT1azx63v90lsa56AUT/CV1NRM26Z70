Audit Report

## Title
Reentrant `withdraw` via `onBuy` callback drains pre-existing `withdrawable` before payer tokens are pulled - (File: `src/Midnight.sol`)

## Summary
In `take`, `buyerPos.credit` is incremented before the `onBuy` callback fires, but `_marketState.withdrawable` is never increased during `take`. Because `withdraw` has no reentrancy guard and only checks `isAuthorized`, a malicious `buyerCallback` (self-authorized by the attacker) can call `withdraw` on behalf of the buyer inside `onBuy`, consuming the freshly-granted credit to drain pre-existing `withdrawable` tokens. The payer transfer executes only after `onBuy` returns, and when `buyerPrice < WAD` the protocol receives fewer tokens than it sent out, permanently breaking the solvency invariant.

## Finding Description

**Root cause — storage ordering in `take`:** [1](#0-0) 

`buyerPos.credit += buyerCreditIncrease` is committed to storage at line 410 before any external call. `_marketState.withdrawable` is never touched inside `take`.

**Callback fires before token pull:** [2](#0-1) 

`IBuyCallback(buyerCallback).onBuy(...)` is called at line 448–452. The `safeTransferFrom` calls that pull tokens from `payer` happen only at lines 455–456, after `onBuy` returns.

**`withdraw` has no reentrancy guard:** [3](#0-2) 

`withdraw` only checks `isAuthorized[onBehalf][msg.sender]` (line 482). It then decrements `_position.credit` (line 493), `_marketState.withdrawable` (line 494), and transfers tokens (line 499). There is no guard preventing it from being called while a `take` is in progress.

**The LIQUIDATION_LOCK_SLOT does not help:** [4](#0-3) 

The transient lock set at line 444 protects only the *seller* from liquidation during callbacks. It does not block reentrant calls to `withdraw` for the buyer.

**`buyerPrice < WAD` is reachable:** [5](#0-4) 

`buyerPrice = sellerPrice + _settlementFee`. For a sell offer at a sufficiently low tick (e.g., tick = 0), `offerPrice` is near zero, making `buyerAssets = units * buyerPrice / WAD < units`.

**Exploit flow:**

1. Attacker deploys `MaliciousCallback` implementing `IBuyCallback`.
2. Attacker calls `setIsAuthorized(callback, true)` → `isAuthorized[attacker][callback] = true`.
3. Attacker (or a second controlled address) creates a sell offer at tick 0 (or any tick where `buyerPrice < WAD`).
4. Attacker calls `take(sellOffer, ..., units, attacker, ..., callback, ...)`.
5. `take` sets `buyerPos.credit += units` (line 410). `withdrawable` is unchanged.
6. `take` calls `callback.onBuy(..., buyer=attacker, ...)`.
7. Inside `onBuy`, callback calls `midnight.withdraw(market, units, attacker, attacker)`:
   - Auth check passes (`isAuthorized[attacker][callback]` = true).
   - `attacker.credit -= units` (zeroes the credit just granted).
   - `withdrawable -= units` (drains pre-existing pool).
   - Transfers `units` tokens to attacker.
8. `onBuy` returns `CALLBACK_SUCCESS`.
9. `take` calls `safeTransferFrom(callback, ..., buyerAssets)` — total `buyerAssets = units * buyerPrice / WAD < units` tokens pulled.
10. Net protocol loss: `units − buyerAssets > 0` tokens per execution.

**Why existing checks fail:**

- No `nonReentrant` modifier on `take` or `withdraw`.
- The token safety requirement ("should not re-enter Midnight on transfer") covers token callbacks only, not `onBuy` callbacks.
- The `isHealthy` check at line 476 applies to the seller, not the buyer.
- Certora specs explicitly assume no reentrancy, so formal verification does not cover this path.

## Impact Explanation
The protocol's token balance decreases by `units − buyerAssets` per attack. Pre-existing `withdrawable` (belonging to other lenders) is drained at a discount. At tick = 0 with zero settlement fee, `buyerAssets = 0` and the attacker extracts the full `units` tokens for free. The core solvency invariant — contract token balance covers all withdrawable assets — is violated: `withdrawable` is decremented but the corresponding tokens are not replaced by the payer. This constitutes direct theft of lender funds.

## Likelihood Explanation
All three preconditions are trivially satisfiable by an unprivileged attacker:
1. `withdrawable > 0` — satisfied in any market that has seen at least one repayment or liquidation.
2. `isAuthorized[attacker][callback]` — self-granted by the attacker in a single transaction before calling `take`.
3. A sell offer with `buyerPrice < WAD` — the attacker can use a second controlled address to create one at tick 0 (bypassing the `offer.maker != taker` self-take check).

The attack is fully permissionless, requires no privileged role, and is repeatable as long as `withdrawable > 0`.

## Recommendation
Apply a reentrancy guard to `take` and `withdraw` (or all state-mutating entry points) using a transient storage lock, similar to the existing `LIQUIDATION_LOCK_SLOT` pattern. Alternatively, move all `safeTransferFrom` calls to *before* any external callbacks (checks-effects-interactions), or add a per-market "in-take" flag checked by `withdraw`. The simplest targeted fix is to add a global transient reentrancy lock that `take` sets on entry and `withdraw` checks before proceeding.

## Proof of Concept
**Minimal Foundry test outline:**

1. Deploy `Midnight` and create a market with `loanToken = MockERC20`.
2. Fund the market with `withdrawable = 100e18` (simulate via a prior `repay` call from a borrower).
3. Deploy `MaliciousCallback`:
   - Implements `IBuyCallback`.
   - In `onBuy`, calls `midnight.withdraw(market, units, attacker, attacker)`.
   - Returns `CALLBACK_SUCCESS`.
4. Attacker calls `midnight.setIsAuthorized(maliciousCallback, true)`.
5. Second attacker address creates a sell offer at tick 0 (price ≈ 0).
6. Attacker calls `midnight.take(sellOffer, ..., 100e18, attacker, ..., maliciousCallback, ...)`.
7. Assert: attacker received `100e18` tokens from `withdraw`; protocol received `≈ 0` tokens from `safeTransferFrom`; `marketState.withdrawable` underflows or equals 0 while protocol balance is negative relative to remaining lender claims.
