### Title
DoS of `buyWithAssetsTargetAndWithdrawCollateral` via Sell Offers at Tick ≥ 0 Causing `TakeAmountsLib.buyerAssetsToUnits()` to Revert Outside `try/catch` — (File: `src/periphery/MidnightBundles.sol`)

---

### Summary

`MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral()` iterates over a `takes[]` array and calls `TakeAmountsLib.buyerAssetsToUnits()` **outside** the `try/catch` block that is meant to gracefully skip failing takes. For any sell offer where `offerPrice + settlementFee > WAD` (i.e., tick ≥ 0 with a non-zero settlement fee), `buyerAssetsToUnits()` reverts with `PriceGreaterThanOne()`. A malicious maker can sign a sell offer at tick = 0 (0% interest — a valid, economically rational offer) to force the entire bundle transaction to revert, permanently DoS-ing this bundle function for any taker whose off-chain order book includes that offer.

---

### Finding Description

**Root cause — `TakeAmountsLib.buyerAssetsToUnits()` is called outside the `try/catch`:**

In `buyWithAssetsTargetAndWithdrawCollateral`, the loop body is:

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.buyerAssetsToUnits(          // ← OUTSIDE try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledBuyerAssets - filledBuyerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // ← OUTSIDE try/catch
);
try IMidnight(MIDNIGHT).take(...) returns (...) {
    ...
} catch {}
``` [1](#0-0) 

The `try/catch` only wraps the `take()` call itself. Both pre-computation helpers are called bare.

**Why `buyerAssetsToUnits()` reverts for sell offers at tick ≥ 0:**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + settlementFee;
require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());
``` [2](#0-1) 

For a sell offer (`offer.buy == false`): `sellerPrice = offerPrice`, so `buyerPrice = offerPrice + settlementFee`. At tick = 0, `offerPrice = WAD` exactly. Any non-zero settlement fee makes `buyerPrice = WAD + settlementFee > WAD`, triggering the revert.

**Critically, `take()` itself does NOT perform this check** — it computes `buyerAssets = units.mulDivUp(buyerPrice, WAD)` without reverting when `buyerPrice > WAD`. So the offer is perfectly executable via a direct `take()` call; only the bundle path is broken. [3](#0-2) 

**The NatSpec acknowledges the gap but does not treat it as a security boundary:**

> "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." [4](#0-3) 

This acknowledgment describes the symptom but does not prevent exploitation.

**Attack flow:**

1. Malicious maker signs a sell offer at tick = 0 (`offerPrice = WAD`), with a large `maxAssets` to appear as deep liquidity.
2. The off-chain order book includes this offer (it is a valid, signed, non-expired offer at a legitimate tick).
3. A taker calls `buyWithAssetsTargetAndWithdrawCollateral` with this offer anywhere in the `takes[]` array.
4. When the loop reaches that offer, `TakeAmountsLib.buyerAssetsToUnits()` reverts with `PriceGreaterThanOne()`.
5. The revert propagates out of the loop (no `try/catch` wraps it), reverting the entire bundle transaction.
6. The malicious maker can re-sign the offer indefinitely at zero cost to maintain the DoS.

---

### Impact Explanation

Any taker relying on `buyWithAssetsTargetAndWithdrawCollateral` to execute an assets-denominated buy is blocked as long as the malicious offer is present in the order book. The taker's transaction reverts, wasting gas and preventing the trade. The core `Midnight.take()` function is unaffected, but the bundle convenience layer — which is the primary UX path for retail users — is rendered unusable for the affected market. The malicious maker bears no cost beyond signing a message.

---

### Likelihood Explanation

The attack requires no privileged access, no capital, and no on-chain transaction from the attacker. Signing an EIP-712 message is free. A sell offer at tick = 0 (0% interest) is a valid, well-formed offer that passes all off-chain validation checks. Any off-chain aggregator or front-end that does not explicitly filter out offers where `offerPrice + settlementFee > WAD` will include such an offer. The attacker can sustain the DoS indefinitely by re-signing.

---

### Recommendation

Move both `TakeAmountsLib.buyerAssetsToUnits()` and `ConsumableUnitsLib.consumableUnits()` inside the `try/catch` block, or wrap the entire per-offer computation in a `try/catch`. The simplest fix is to restructure the loop so that any revert from the pre-computation helpers is caught and treated as a skipped offer, consistent with the existing intent of the `try/catch` on `take()`:

```solidity
// Pseudocode fix
try this._computeAndTake(takes[i], ...) returns (uint256 units, uint256 assets) {
    filledUnits += units;
    filledBuyerAssets += assets;
} catch {}
```

Alternatively, add an explicit pre-check: skip any offer where `offerPrice + settlementFee > WAD` before calling `buyerAssetsToUnits()`.

---

### Proof of Concept

**Preconditions:**
- A market exists with a non-zero settlement fee (e.g., 1 bps).
- Attacker controls any EOA (no funds required).

**Steps:**

1. Attacker signs a sell offer: `tick = 0`, `maxAssets = type(uint128).max`, valid `expiry`, using `EcrecoverRatifier`.
2. Off-chain order book ingests the offer (it is valid by all standard checks).
3. Legitimate taker calls `buyWithAssetsTargetAndWithdrawCollateral` with `takes[0]` = attacker's offer.
4. Loop iteration 0: `TakeAmountsLib.buyerAssetsToUnits(MIDNIGHT, id, takes[0].offer, ...)` is called.
   - `offerPrice = TickLib.tickToPrice(0) = WAD`
   - `buyerPrice = WAD + settlementFee > WAD`
   - `require(buyerPrice <= WAD, ...)` → **reverts**
5. Entire transaction reverts. Taker's gas is wasted. Trade is not executed.
6. Attacker re-signs the same offer (free) to maintain the DoS. [5](#0-4) [6](#0-5)
