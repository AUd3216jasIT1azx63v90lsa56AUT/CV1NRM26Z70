### Title
Maker's `setConsumed` Can Be DoSed via Front-Running with `take` — (File: src/Midnight.sol)

### Summary
An attacker can prevent a maker from setting a specific partial consumed value for their offer group by front-running the `setConsumed` call with a `take` that pushes `consumed[maker][group]` above the target value, causing the `AlreadyConsumed` revert. This is structurally identical to the InstaDApp pattern: a permissionless function (`take`) can be used to pre-empt a state-transition function (`setConsumed`) by violating its monotonicity guard.

### Finding Description

`setConsumed` enforces a one-way monotonicity invariant:

```solidity
// src/Midnight.sol:724-727
function setConsumed(bytes32 group, uint256 amount, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    require(amount >= consumed[onBehalf][group], AlreadyConsumed());
    consumed[onBehalf][group] = amount;
``` [1](#0-0) 

The `consumed[maker][group]` counter is incremented by any taker calling `take`:

```solidity
// src/Midnight.sol:367-373
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
} else {
    newConsumed = consumed[offer.maker][offer.group] += units;
    require(newConsumed <= offer.maxUnits, ConsumedUnits());
}
``` [2](#0-1) 

`take` is fully permissionless for any valid taker — there is no access restriction preventing an attacker from calling it:

```solidity
// src/Midnight.sol:346
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());
``` [3](#0-2) 

**Attack path:**
1. Maker has an offer with `maxUnits = 1000`, current `consumed[maker][group] = 0`.
2. Maker broadcasts `setConsumed(group, 500, maker)` to cap remaining capacity at 500 units.
3. Attacker observes the pending transaction in the mempool and front-runs it with `take(offer, ..., 501 units, attacker, ...)`.
4. `consumed[maker][group]` becomes `501`.
5. Maker's `setConsumed(group, 500, maker)` reverts: `require(500 >= 501)` fails with `AlreadyConsumed`.

The maker is left unable to set the intended partial cap. The only remaining option is `setConsumed(group, type(uint256).max, maker)` for full cancellation, which the contract itself documents as the only path that "never reverts." [4](#0-3) 

### Impact Explanation

A maker who wishes to reduce — but not eliminate — their remaining offer capacity (e.g., to manage risk exposure mid-market) is permanently blocked from setting any specific consumed value below the attacker-inflated counter. The maker is forced into a binary choice: accept the attacker-determined remaining capacity, or fully cancel the offer group. In volatile markets where precise capacity management matters, this degrades the maker's ability to manage risk without fully exiting.

### Likelihood Explanation

The attacker must execute a valid `take`, which requires holding sufficient loan token assets to pay `buyerAssets`. This is a real cost, but the attacker receives units in return, making the attack economically neutral or even profitable if the offer price is favorable. Any market participant with sufficient capital and visibility into the mempool can execute this. On chains with a public mempool (Ethereum mainnet, most L2s), the attack is straightforward.

### Recommendation

The `setConsumed` function should be redesigned to allow a maker to set an absolute cap on future consumption rather than requiring the new value to exceed the current counter. One approach: add a separate `setMaxConsumed(bytes32 group, uint256 cap, address onBehalf)` that stores a per-group cap checked during `take`, independent of the monotonically increasing `consumed` counter. Alternatively, document clearly that `setConsumed` with a specific value is not front-running safe and that `type(uint256).max` is the only reliable cancellation path.

### Proof of Concept

**Preconditions:**
- Maker has a valid offer: `maker = Alice`, `group = G`, `maxUnits = 1000`, `consumed[Alice][G] = 0`.
- Attacker = Bob, holds sufficient loan tokens to buy 501 units.

**Steps:**
1. Alice broadcasts: `setConsumed(G, 500, Alice)` — intending to allow only 500 more units.
2. Bob observes the pending tx and front-runs with: `take(aliceOffer, ratifierData, 501, Bob, ...)`.
3. `consumed[Alice][G]` is now `501`.
4. Alice's `setConsumed(G, 500, Alice)` executes and hits: `require(500 >= 501)` → reverts with `AlreadyConsumed`.
5. Alice can no longer set consumed to any value ≤ 501. She must either accept 499 remaining units of capacity or call `setConsumed(G, type(uint256).max, Alice)` to fully cancel. [5](#0-4) [6](#0-5)
