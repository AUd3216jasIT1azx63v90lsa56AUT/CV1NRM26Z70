Audit Report

## Title
Oracle Price Manipulated via `onSell` Callback Allows Undercollateralized Borrowing - (File: src/Midnight.sol)

## Summary
In `take()`, the seller's debt is committed to storage at line 414 before the `onSell` callback fires at lines 458–474. Because `IOracle.price()` is declared `external view` and called via `staticcall`, the oracle cannot write state during the health check — but it can freely read state that the callback already wrote. An attacker who deploys a malicious oracle reading from an attacker-controlled `StateContract`, and a malicious `onSell` callback writing to that same contract, can inflate the oracle price during the callback window, causing `isHealthy` to pass with a fabricated price and leaving the seller with undercollateralized debt that becomes bad debt socialized to lenders.

## Finding Description
**Verified code path in `src/Midnight.sol`:**

1. `take()` is called (line 337).
2. `sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease)` (line 414) — seller's debt committed to storage.
3. Liquidation lock set for seller (line 444): `bool wasLocked = UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`.
4. `onSell` callback fires (lines 458–474) via a regular `CALL`. The callback can write to any external contract, including an attacker-controlled `StateContract`.
5. Liquidation lock released (line 475): `if (!wasLocked) UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, false)`.
6. Health check (line 476): `require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable())`.
7. Inside `isHealthy` (line 953): `uint256 price = IOracle(collateralParam.oracle).price()`. Because `IOracle.price()` is declared `external view` (`src/interfaces/IOracle.sol:6`), Solidity issues a `staticcall`. The `staticcall` prevents the oracle from writing state during its own execution, but does not prevent the oracle from reading state already written by the callback in step 4. A malicious oracle that reads `StateContract.value` returns the inflated value written by the callback.
8. `maxDebt` is computed with the inflated price and exceeds the seller's actual debt — health check passes.

**Why existing checks fail:**
- The liquidation lock (lines 444/475) only prevents liquidation during callbacks; it does not prevent oracle price manipulation.
- `staticcall` on `price()` prevents the oracle from writing state during the health check, but the callback already wrote the manipulated value before `isHealthy` is called.
- No snapshot of the oracle price is taken before the callback to compare against.
- The Certora formal verification in `certora/specs/Healthiness.spec` lines 14–16 explicitly acknowledges this gap: `// Assumption: price does not change during rules.` and models `price()` as a `persistent ghost summaryPrice(address)` returning a fixed value per oracle address (line 42). This assumption is not enforced on-chain.
- `touchMarket` (lines 755–791) is permissionless with no restriction on the oracle address supplied in `CollateralParams`.

**Attacker-controlled inputs:**
- Market creation: `touchMarket` is permissionless. The attacker deploys `StateContract`, `MaliciousOracle` (reads `StateContract.value`), and `MaliciousCallback` (writes to `StateContract.value`), then calls `touchMarket` with `CollateralParams.oracle = address(MaliciousOracle)`.
- Seller callback: when `offer.buy == true`, the seller is the taker and controls `takerCallback`. When `offer.buy == false`, the seller is `offer.maker` and controls `offer.callback`.

## Impact Explanation
The seller can borrow an arbitrary number of units backed by negligible collateral (e.g., 1 wei). After `take()` completes, the seller's position is undercollateralized at the true oracle price. The attacker then calls `withdrawCollateral` (line 568: `require(isHealthy(market, id, onBehalf), UnhealthyBorrower())`), which calls `isHealthy` with the same manipulable oracle while `StateContract.value` remains inflated, allowing full collateral withdrawal. Once collateral is withdrawn, the attacker resets `StateContract.value` to zero, leaving the protocol with bad debt socialized to lenders via the loss factor mechanism (lines 631–634). This directly violates the invariant that every debt unit must be backed by sufficient collateral at the oracle price, and constitutes bad debt creation — a top-priority bug class per `live_context.json`.

## Likelihood Explanation
All preconditions are fully attacker-controlled: the attacker deploys three contracts (`StateContract`, `MaliciousOracle`, `MaliciousCallback`), creates a market (permissionless, no privileged access required), and supplies minimal collateral. The `live_context.json` explicitly lists "market creator" and "callback receiver" as valid attacker profiles. `SECURITY.md` explicitly states oracle manipulation attacks are not excluded. The attack is repeatable across any number of takes and is not blocked by any on-chain guard. The only cost is gas and a small collateral deposit. A victim lender who provides a buy offer in the malicious market is the counterparty whose credit is destroyed by the resulting bad debt.

## Recommendation
Snapshot the oracle price(s) before the `onSell` callback fires and re-verify that the price has not changed after the callback returns, or alternatively call `isHealthy` before the `onSell` callback (after debt is committed but before any external call). A simpler mitigation is to record the `maxDebt` value computed from oracle prices immediately after debt is committed (before any callback), then assert that the same `maxDebt` value still satisfies the health check after the callback — preventing the callback from influencing the oracle read used in the health check. Alternatively, restrict oracle addresses to a protocol-approved allowlist, though this conflicts with the permissionless market design.

## Proof of Concept
```solidity
// StateContract: holds the manipulated price value
contract StateContract {
    uint256 public value;
    function set(uint256 v) external { value = v; }
}

// MaliciousOracle: reads StateContract.value as price (view function, staticcall-safe)
contract MaliciousOracle {
    StateContract public sc;
    constructor(StateContract _sc) { sc = _sc; }
    function price() external view returns (uint256) { return sc.value; }
}

// MaliciousCallback: writes inflated value to StateContract during onSell
contract MaliciousCallback is ISellCallback {
    StateContract public sc;
    uint256 public inflatedPrice;
    constructor(StateContract _sc, uint256 _inflatedPrice) { sc = _sc; inflatedPrice = _inflatedPrice; }
    function onSell(...) external returns (bytes32) {
        sc.set(inflatedPrice); // write inflated price before isHealthy is called
        return CALLBACK_SUCCESS;
    }
}

// Test:
// 1. Deploy StateContract, MaliciousOracle(sc), MaliciousCallback(sc, 1e50)
// 2. touchMarket with CollateralParams.oracle = address(MaliciousOracle)
// 3. Attacker supplies 1 wei collateral
// 4. Lender creates buy offer in malicious market
// 5. Attacker takes lender's offer as seller with takerCallback = address(MaliciousCallback)
//    - debt committed (line 414)
//    - onSell fires: sc.value = 1e50 (inflated)
//    - isHealthy: oracle returns 1e50, maxDebt >> debt, health check passes
// 6. Attacker calls withdrawCollateral: oracle still returns 1e50, passes
// 7. Attacker calls sc.set(0): oracle now returns 0
// 8. Protocol has bad debt; lender credit slashed via lossFactor
```
