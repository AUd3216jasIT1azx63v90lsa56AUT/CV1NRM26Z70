Audit Report

## Title
Stale Signed Authorization Replay After Direct Revocation Re-grants Operator Access — (File: `src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer` maintains its own per-address `nonce` mapping that is only incremented when `EcrecoverAuthorizer.setIsAuthorized` is called. When a victim revokes an operator directly via `Midnight.setIsAuthorized`, the `EcrecoverAuthorizer` nonce is never consumed, leaving any previously signed `Authorization` at the current nonce valid and replayable by anyone until its deadline. An attacker can replay the stale signature to re-grant the operator, who can then drain the victim's credit via `withdraw`.

## Finding Description

**Root cause:**

`EcrecoverAuthorizer.setIsAuthorized` (line 26) enforces:
```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```
The `nonce` mapping lives exclusively in `EcrecoverAuthorizer` and is only mutated here. `Midnight.setIsAuthorized` (line 731–735) only writes to `isAuthorized[onBehalf][authorized]` and emits an event — it has no knowledge of and makes no call into `EcrecoverAuthorizer.nonce`.

**Exploit flow:**

1. Victim calls `midnight.setIsAuthorized(ecrecoverAuthorizer, true, victim)` — standard setup. `EcrecoverAuthorizer.nonce[victim] == 0`.
2. Victim signs `Authorization{authorizer=victim, authorized=operator, isAuthorized=true, nonce=0, deadline=now+1day}`.
3. Victim revokes operator directly: `midnight.setIsAuthorized(operator, false, victim)`. Sets `isAuthorized[victim][operator] = false` in Midnight but **leaves `EcrecoverAuthorizer.nonce[victim] == 0` unchanged**.
4. Attacker (before deadline) calls `EcrecoverAuthorizer.setIsAuthorized(auth, sig)` with the old signed message.
5. Nonce check: `0 == nonce[victim]++` → passes; nonce becomes 1.
6. Signature verifies against victim's key → passes.
7. Authorization check at line 33–36: `signer (victim) == authorization.authorizer (victim)` → passes.
8. `midnight.setIsAuthorized(operator, true, victim)` is called; `isAuthorized[victim][ecrecoverAuthorizer] == true` (never revoked), so the `onBehalf` check at line 732 passes.
9. `isAuthorized[victim][operator] = true` is restored.
10. Operator calls `midnight.withdraw(market, units, victim, attacker)` — authorization check at line 482 passes, victim's credit is drained.

**Why existing checks fail:** The nonce check only prevents replay of a nonce already consumed *through `EcrecoverAuthorizer`*. A direct `Midnight.setIsAuthorized` revocation does not consume any nonce in `EcrecoverAuthorizer`, so the signed message at nonce 0 remains unconsumed and valid until its deadline.

## Impact Explanation
A victim who revokes an operator via the natural `Midnight.setIsAuthorized` path believes the revocation is final. An attacker holding any unexpired signed authorization from the victim can immediately re-grant the operator, who can then call `withdraw` (or any other authorized action) to steal the victim's credit. This is a direct, concrete theft of user funds with no recovery path.

## Likelihood Explanation
All three preconditions are realistic for any user of the `EcrecoverAuthorizer` peripheral:
- Victim authorized `EcrecoverAuthorizer` in Midnight (standard setup for signed-authorization flow).
- Victim signed at least one `Authorization` with `isAuthorized=true` whose deadline has not yet expired.
- Victim revoked the operator via `Midnight.setIsAuthorized` directly (the natural, documented revocation path).

The attack is permissionless (any address can submit the old signature), repeatable until the deadline expires, and requires no special privileges. The attacker only needs to observe the victim's prior signed authorization (e.g., from mempool or event logs).

## Recommendation
Add a `invalidateNonce` (or `incrementNonce`) function to `EcrecoverAuthorizer` that allows a user to atomically increment their own nonce, invalidating all outstanding signed authorizations at the current nonce:

```solidity
function invalidateNonce() external {
    emit NonceInvalidated(msg.sender, nonce[msg.sender]);
    nonce[msg.sender]++;
}
```

Additionally, document clearly that users who wish to revoke a signed authorization must either (a) call this function before or alongside `Midnight.setIsAuthorized`, or (b) revoke via `EcrecoverAuthorizer.setIsAuthorized` with `isAuthorized=false` (which consumes the nonce). This is the standard pattern used by Uniswap Permit2 and similar systems.

## Proof of Concept

**Minimal manual steps:**

1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. As victim: call `midnight.setIsAuthorized(ecrecoverAuthorizer, true, victim)`.
3. As victim: sign `Authorization{authorizer=victim, authorized=operator, isAuthorized=true, nonce=0, deadline=block.timestamp+1 days}`.
4. As victim: call `midnight.setIsAuthorized(operator, false, victim)` — victim believes operator is revoked.
5. Assert: `midnight.isAuthorized(victim, operator) == false`. ✓
6. Assert: `ecrecoverAuthorizer.nonce(victim) == 0`. ✓ (nonce not consumed)
7. As attacker: call `ecrecoverAuthorizer.setIsAuthorized(auth, sig)` with the signature from step 3.
8. Assert: `midnight.isAuthorized(victim, operator) == true`. ✓ (operator re-granted)
9. As operator: call `midnight.withdraw(market, victimCredit, victim, attacker)`.
10. Assert: attacker received victim's funds. ✓
