### Title
Miner-Controlled `block.timestamp` Enables Immediate Maximum LIF at Maturity, Bypassing the 15-Minute Ramp-Up — (File: src/Midnight.sol)

---

### Summary

`TIME_TO_MAX_LIF` is set to exactly `15 minutes = 900 seconds`, which is the same upper bound by which a miner can shift `block.timestamp` forward. A miner acting as a liquidator can set `block.timestamp = market.maturity + TIME_TO_MAX_LIF` at the exact moment of maturity, causing the LIF calculation to return `_maxLif` immediately — bypassing the entire ramp-up period designed to give borrowers time to repay.

---

### Finding Description

In `liquidate()`, when `postMaturityMode = true`, the Liquidation Incentive Factor (LIF) is computed as:

```solidity
uint256 lif = postMaturityMode
    ? UtilsLib.min(_maxLif, WAD + (_maxLif - WAD) * (block.timestamp - market.maturity) / TIME_TO_MAX_LIF)
    : _maxLif;
``` [1](#0-0) 

The constant `TIME_TO_MAX_LIF` is defined as:

```solidity
uint256 constant TIME_TO_MAX_LIF = 15 minutes;
``` [2](#0-1) 

`15 minutes = 900 seconds` is precisely the maximum amount a miner can shift `block.timestamp` forward per the Ethereum PoW specification. A miner who is also a liquidator can, at the exact moment of maturity, produce a block with `block.timestamp = market.maturity + 900`. This causes:

```
lif = min(_maxLif, WAD + (_maxLif - WAD) * 900 / 900) = _maxLif
```

The entire 15-minute ramp-up is collapsed to zero. The liquidatability gate at line 622 is also satisfied by the shifted timestamp:

```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
``` [3](#0-2) 

Additionally, if the market is within 900 seconds of maturity, the miner can shift `block.timestamp` past `market.maturity` to trigger post-maturity mode on a market that has not yet actually matured, enabling liquidation of healthy borrowers with no RCF restriction.

---

### Impact Explanation

For a market with `LLTV_0 = 0.385` and `LIQUIDATION_CURSOR_HIGH = 0.5`:

```
maxLif = WAD / (WAD - 0.5 * (WAD - 0.385 * WAD))
       = 1e18 / 0.6925e18
       ≈ 1.444e18  (44.4% liquidation incentive)
``` [4](#0-3) 

A miner-liquidator seizes 44.4% more collateral than the debt repaid, **immediately at maturity**, while a non-miner liquidator must wait the full 15 minutes to reach the same incentive. Borrowers who intended to repay within the 15-minute window are liquidated at maximum penalty with no warning. The RCF is also deactivated in post-maturity mode, so the entire position can be liquidated in one transaction.

---

### Likelihood Explanation

- Requires the attacker to control block production (miner on PoW, validator on PoS).
- On PoW chains (and EVM-compatible chains with loose timestamp rules), the 900-second shift is directly exploitable.
- On Ethereum PoS, the per-slot timestamp flexibility is narrower (~12 seconds), but the pre-maturity trigger (shifting into post-maturity mode up to 12 seconds early) still applies.
- The attack is economically rational: the miner profits directly from the increased LIF on any sufficiently large position at maturity.
- No privileged protocol access is required — `liquidate()` is permissionless (subject to the optional `liquidatorGate`). [5](#0-4) 

---

### Recommendation

Increase `TIME_TO_MAX_LIF` to a value significantly greater than 900 seconds so that a 900-second timestamp manipulation is negligible relative to the ramp-up window. For example, setting `TIME_TO_MAX_LIF = 7 days` would reduce the maximum LIF gain from a 900-second manipulation to `(_maxLif - WAD) * 900 / 604800 ≈ 0.15%` of the total incentive range — effectively harmless. Alternatively, document the accepted risk if the protocol is intended only for Ethereum PoS where timestamp manipulation is bounded to ~12 seconds.

### Proof of Concept

**Setup:**
- Market with `loanToken = USDC`, `LLTV = 0.385`, `maxLif ≈ 1.444e18`, `maturity = T`.
- Borrower has 1000 USDC of debt and 1444 units of collateral (healthy position).

**Attack steps:**

1. Attacker is a block proposer/miner.
2. At real-world time `T` (market maturity), the attacker produces a block with `block.timestamp = T + 900` (shifting forward by `TIME_TO_MAX_LIF`).
3. In the same block, attacker calls:
   ```solidity
   midnight.liquidate(
       market,
       collateralIndex,
       0,           // seizedAssets = 0 (use repaidUnits path)
       1000,        // repaidUnits = full debt
       borrower,
       true,        // postMaturityMode
       attacker,
       address(0),
       ""
   );
   ```
4. LIF check: `block.timestamp - market.maturity = 900 = TIME_TO_MAX_LIF`, so `lif = _maxLif ≈ 1.444e18`.
5. `seizedAssets = 1000 * 1.444e18 / WAD * ORACLE_PRICE_SCALE / liquidatedCollatPrice` — attacker seizes the maximum possible collateral.
6. A non-miner liquidator calling at the same real-world time `T` (with honest `block.timestamp = T`) would get `lif = WAD` (no incentive), making the liquidation unprofitable and leaving the borrower's position intact.

**Expected outcome:** Miner-liquidator extracts maximum collateral at maturity with zero waiting time, while honest liquidators cannot compete. [1](#0-0) [2](#0-1)
