Audit Report

## Title
Fully-Consumed Assets-Based Buy Offer With `buyerPrice < WAD` Bypasses Consumption Guard, Allowing Unbounded Position Mutation - (File: src/Midnight.sol)

## Summary
When a buy offer has `maxAssets > 0` and `buyerPrice < WAD`, calling `take` with `units = 1` after the offer is fully consumed produces `buyerAssets = 0` via truncating division, causing the consumed guard to pass as a no-op. All downstream position mutations — buyer credit increase, seller debt increase, and `totalUnits` inflation — still execute against `units = 1` with zero token transfer, corrupting the credit/debt accounting invariant indefinitely.

## Finding Description
**Root cause:**

At `src/Midnight.sol:363`, `buyerAssets` is computed with truncating division:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
``` [1](#0-0) 

When `offer.buy = true`, `units = 1`, and `buyerPrice < WAD`, `mulDivDown(1 * buyerPrice, WAD)` truncates to `0`.

**Consumed guard bypass:**

At lines 367–369, the guard increments `consumed` by `buyerAssets` (i.e., `0`) and checks `<= maxAssets`:

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
``` [2](#0-1) 

When the offer is already fully consumed (`consumed == maxAssets`), `maxAssets + 0 <= maxAssets` trivially passes.

**Position mutations proceed on `units = 1`:**

```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt);
uint256 sellerDebtIncrease = units - sellerCreditDecrease;
``` [3](#0-2) 

```solidity
_marketState.totalUnits =
    UtilsLib.toUint128(_marketState.totalUnits + buyerCreditIncrease - sellerCreditDecrease);
``` [4](#0-3) 

These mutations operate on `units` (= 1), not `buyerAssets` (= 0). The token transfer at line 455 sends zero:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
``` [5](#0-4) 

**Protocol acknowledgment of the behavior:**

The codebase explicitly documents this at line 94:

```
/// @dev It is possible to give units to a fully consumed assets-based buy offer with price < 1.
``` [6](#0-5) 

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — passes because `maxAssets + 0 ≤ maxAssets`.
- No guard enforces `units == 0` when `consumed >= maxAssets`.
- No guard enforces `buyerAssets > 0` when `units > 0` on a buy offer.

## Impact Explanation
Each post-consumption call with `units = 1` and `buyerAssets = 0`:
- Increases the maker's (buyer's) `credit` by `buyerCreditIncrease` derived from `units = 1`, with zero loan tokens paid.
- Increases the taker's (seller's) `debt` by `sellerDebtIncrease` derived from `units = 1`, with zero loan tokens received.
- Increases `totalUnits` by `buyerCreditIncrease` with no asset backing, diluting all lenders' pro-rata claims at maturity.

An attacker controlling both the maker address and a throwaway taker address can inflate the maker's credit arbitrarily at negligible gas cost. The taker's resulting bad debt is socialized among all lenders via the slashing mechanism, directly reducing their recoverable assets. This constitutes unauthorized state change, accounting corruption, and indirect theft from other lenders.

## Likelihood Explanation
All preconditions are attacker-reachable without any privileged access:
1. `offer.buy = true`, `offer.maxAssets > 0` — standard offer configuration available to any user.
2. `buyerPrice < WAD` — achieved by setting `offer.tick` to any value where `tickToPrice(tick) + settlementFee < WAD`.
3. Offer fully consumed — the attacker can self-consume as the first taker using a separate address, or wait for organic fills.

The attack is repeatable every block with no token transfers, making it essentially free beyond gas. Any external account can act as the taker.

## Recommendation
Add a guard immediately after computing `buyerAssets` on the buy-offer path to reject zero-asset takes when `units > 0`:

```solidity
if (offer.buy && offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
}
```

Alternatively, add a guard that rejects any `take` call when `consumed >= maxAssets` and `maxAssets > 0`, regardless of the computed asset delta:

```solidity
if (offer.maxAssets > 0) {
    require(consumed[offer.maker][offer.group] < offer.maxAssets, OfferFullyConsumed());
}
```

Either fix closes the bypass by ensuring a fully consumed offer cannot be re-entered with a zero-asset increment.

## Proof of Concept
The repository already contains a reproducing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol`: [7](#0-6) 

Minimal manual steps:
1. Deploy a market with a loan token.
2. Create a buy offer with `maxAssets = N` and `tick` such that `tickToPrice(tick) < WAD` (e.g., `MAX_TICK - 16` as used in the existing test).
3. Fully consume the offer from a second address so `consumed == maxAssets`.
4. Call `take` again with `units = 1` from any address. Observe: `buyerAssets = 0`, consumed guard passes, buyer credit increases by 1, seller debt increases by 1, `totalUnits` inflates by 1, zero tokens transferred.
5. Repeat step 4 arbitrarily many times to inflate credit/debt without bound.
