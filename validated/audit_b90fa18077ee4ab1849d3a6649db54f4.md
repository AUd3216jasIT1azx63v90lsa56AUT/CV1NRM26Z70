Audit Report

## Title
`sellerAssetsToUnits` panics on division-by-zero when `sellerPrice == 0`, unconditionally reverting `supplyCollateralAndSellWithAssetsTarget` - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary
`TakeAmountsLib.sellerAssetsToUnits` calls `mulDivUp(WAD, sellerPrice)` where `sellerPrice = offerPrice - settlementFee`. When `offerPrice == settlementFee`, `sellerPrice` is exactly `0`, and `mulDivUp`'s implementation `(x * y + (d - 1)) / d` triggers an arithmetic underflow panic on `(d - 1)` before any division occurs. Because `sellerAssetsToUnits` is evaluated as a function argument outside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, the panic propagates and reverts the entire bundler transaction. The core `Midnight.take` does not revert for this same input, creating a confirmed divergence.

## Finding Description

**Root cause — `mulDivUp` with `d = 0`:**

`UtilsLib.mulDivUp` is implemented as:
```solidity
return (x * y + (d - 1)) / d;
``` [1](#0-0) 

This is not in an `unchecked` block. With `d = 0`, the sub-expression `(d - 1)` underflows under Solidity 0.8+ checked arithmetic, producing an arithmetic panic before any division.

**`sellerAssetsToUnits` passes `sellerPrice` as the divisor:**

When `offer.buy == true` and `offerPrice == settlementFee`, `sellerPrice = offerPrice - settlementFee = 0`. The subtraction itself does not underflow (equal values), but the resulting `0` is then passed as `d` to `mulDivUp`: [2](#0-1) 

The NatSpec on line 34 states "Reverts if `offerPrice < settlementFee`" — the `==` case is not mentioned and not guarded. [3](#0-2) 

**Divergence from `Midnight.take`:**

In the core, `sellerAssets = units.mulDivDown(sellerPrice, WAD) = (units * 0) / WAD = 0` — no revert. The core proceeds normally; the periphery panics. [4](#0-3) 

**Unprotected call site in `supplyCollateralAndSellWithAssetsTarget`:**

`sellerAssetsToUnits` is evaluated as the first argument to `min(...)` at lines 286–288, before the `try/catch` block that begins at line 292. Solidity evaluates all function arguments before the call, so the panic propagates unconditionally: [5](#0-4) 

The bundler's own NatSpec at line 246 explicitly acknowledges this: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." [6](#0-5) 

**Exploit flow:**
1. Attacker observes `settlementFee(id, ttm)` for a target market (public view function).
2. Attacker identifies tick `T` such that `tickToPrice(T) == settlementFee`. Both values are multiples of `PRICE_ROUNDING_STEP = 1e12` (CBP), making exact equality achievable. [7](#0-6) [8](#0-7) 
3. Attacker publishes a buy offer (`offer.buy = true`) at tick `T` — permissionless, gas cost only.
4. An off-chain aggregator includes this offer in a victim's `takes` array.
5. Victim calls `supplyCollateralAndSellWithAssetsTarget`; the loop reaches the poisoned offer, `sellerAssetsToUnits` panics, entire transaction reverts.

## Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee(id, ttm)` at call time reverts unconditionally. This blocks the sell-side bundle flow for borrowers who rely on the bundler to atomically supply collateral and borrow in one transaction. The DoS is repeatable at negligible cost to the attacker and cannot be mitigated by the victim without removing the poisoned offer from their `takes` array.

## Likelihood Explanation
Settlement fees and tick prices are both multiples of `1e12`; exact equality is achievable at specific TTM values. The settlement fee schedule is deterministic and public, so the attacker can predict the exact block at which a given tick's price equals the fee. The attacker only needs to publish one offer (permissionless). The victim must include the offer in their `takes` array — realistic when using an off-chain aggregator that sorts by price without pre-simulating `sellerAssetsToUnits`. The attacker can maintain a portfolio of offers at multiple ticks to cover a range of future fee values.

## Recommendation
Add an explicit guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case when `offer.buy == true`. If `targetSellerAssets == 0`, return `0` units. If `targetSellerAssets > 0`, revert with a descriptive error (e.g., `SellerPriceIsZero`) rather than an opaque arithmetic panic. This mirrors the existing guard pattern in `buyerAssetsToUnits` and aligns the periphery's revert behavior with the core's documented semantics.

Alternatively, restructure `supplyCollateralAndSellWithAssetsTarget` so that `sellerAssetsToUnits` is called inside the `try/catch` block, consistent with how `take` itself is wrapped, allowing the bundler to skip the poisoned offer rather than reverting the entire transaction.

## Proof of Concept
Minimal Foundry test:
1. Deploy `Midnight` and `MidnightBundles` with a market having a non-zero settlement fee at the current TTM (e.g., `settlementFee = 1e12`).
2. Find tick `T` such that `TickLib.tickToPrice(T) == 1e12` (iterate ticks or use `priceToTick`).
3. Create a buy offer with `offer.buy = true`, `offer.tick = T`, `offer.maker = attacker`, valid ratifier.
4. Call `supplyCollateralAndSellWithAssetsTarget` with `takes = [poisonedOffer]` and `targetSellerAssets = 1`.
5. Observe the transaction reverts with an arithmetic panic (not `OutOfOffers` or any graceful error).
6. Confirm that calling `Midnight.take` directly with the same offer and `units = 1` succeeds and returns `sellerAssets = 0`.
