### Title
Zero Oracle Price Enables Free Collateral Seizure in `liquidate()` — (`File: src/Midnight.sol`)

---

### Summary

The `liquidate()` function in `Midnight.sol` computes `repaidUnits` from `seizedAssets` using the oracle price. When the oracle for the liquidated collateral returns `0`, the `mulDivUp` arithmetic silently resolves `repaidUnits = 0`, allowing an attacker to seize a borrower's entire collateral balance while repaying zero loan tokens. The protocol's own LIVENESS comments acknowledge the zero-price case only for the inverse path (`repaidUnits > 0` reverts via division-by-zero), leaving the `seizedAssets > 0` path completely unguarded.

---

### Finding Description

**Root cause — unvalidated oracle price in the `seizedAssets` branch of `liquidate()`**

In `liquidate()`, the oracle price for the collateral being seized is fetched without any non-zero check: [1](#0-0) 

`liquidatedCollatPrice` is then used to compute `repaidUnits` when the caller supplies `seizedAssets > 0`: [2](#0-1) 

`mulDivUp` is defined as: [3](#0-2) 

When `liquidatedCollatPrice == 0`:

```
mulDivUp(seizedAssets, 0, 1e36)
  = (seizedAssets * 0 + (1e36 - 1)) / 1e36
  = (1e36 - 1) / 1e36
  = 0   ← integer division
```

Then:
```
mulDivUp(0, WAD, lif)
  = (0 * WAD + (lif - 1)) / lif
  = (lif - 1) / lif
  = 0   ← integer division (lif ≥ WAD = 1e18)
```

So `repaidUnits = 0`. The subsequent state mutations then:
- Deduct `seizedAssets` from the borrower's collateral balance [4](#0-3) 
- Add `repaidUnits (= 0)` to `withdrawable` and subtract from `_position.debt` [5](#0-4) 
- Transfer `seizedAssets` collateral tokens to the attacker's `receiver` [6](#0-5) 
- Pull `repaidUnits (= 0)` loan tokens from the attacker [7](#0-6) 

The borrower loses all collateral; their debt is unchanged; the attacker pays nothing.

**Contrast with the `repaidUnits > 0` path** (the only case the protocol documents): [8](#0-7) 

Here `liquidatedCollatPrice` is the divisor, so a zero price causes a division-by-zero revert. The LIVENESS section explicitly notes this: [9](#0-8) 

The `seizedAssets` path is the unacknowledged, unguarded counterpart.

**Liquidatability precondition with zero price**

`isHealthy()` computes `maxDebt` by summing `collateral * price / ORACLE_PRICE_SCALE * lltv / WAD` across all activated collaterals: [10](#0-9) 

When the oracle for a collateral returns `0`, that collateral contributes `0` to `maxDebt`. If the borrower's debt exceeds the remaining `maxDebt`, the `NotLiquidatable()` guard passes: [11](#0-10) 

---

### Impact Explanation

- **Attacker gains**: all collateral tokens of the targeted borrower at zero cost.
- **Borrower loses**: entire collateral balance; debt remains unchanged (full bad debt).
- **Protocol**: bad debt is socialized via the loss-factor mechanism onto all lenders in the market, permanently diluting their credit.
- **Severity**: Critical — direct theft of collateral assets with no capital requirement.

---

### Likelihood Explanation

The attack requires the oracle for a collateral to return `0`. This is realistic in the following scenarios (all without privileged access):

1. **Uninitialized oracle** (direct analog to the external report): a freshly deployed oracle that has not yet received its first price update returns the Solidity default value of `0`.
2. **Buggy or edge-case oracle**: an oracle that returns `0` on underflow, stale data, or circuit-breaker conditions.
3. **Malicious market creator**: `touchMarket` accepts any `oracle` address supplied in `CollateralParams` with no validation. An attacker can create a market with a collateral whose oracle is a contract that always returns `0`, then wait for a borrower to use it. [12](#0-11) 

No privileged keys are required. The attacker only needs to call `liquidate()` with `seizedAssets > 0` and `repaidUnits == 0`.

---

### Recommendation

Add a non-zero price guard in the `seizedAssets` branch of `liquidate()`, mirroring the implicit protection that division-by-zero provides in the `repaidUnits` branch:

```solidity
if (seizedAssets > 0) {
    require(liquidatedCollatPrice > 0, "PRICE_NOT_INITIALIZED");
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
} else {
    // division by zero already reverts when liquidatedCollatPrice == 0
    seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
}
```

Additionally, consider adding the same guard in `isHealthy()` and the `liquidate()` loop so that a zero-price oracle causes a revert rather than silently zeroing out a collateral's contribution to `maxDebt`.

### Proof of Concept

**Setup**:
1. Deploy a `ZeroOracle` contract whose `price()` always returns `0`.
2. Create a market with one collateral whose `oracle` is `ZeroOracle`.
3. Victim calls `supplyCollateral` (1000e18 tokens) and borrows via `take` (e.g., 500e18 debt units).

**Attack**:
```solidity
// ZeroOracle.price() returns 0
// isHealthy: maxDebt = 1000e18 * 0 / 1e36 * lltv / 1e18 = 0 < 500e18 debt → liquidatable

midnight.liquidate(
    market,
    0,                  // collateralIndex
    1000e18,            // seizedAssets = victim's full collateral
    0,                  // repaidUnits (computed internally → 0)
    victim,
    false,              // normal mode
    attacker,           // receiver
    address(0),
    ""
);
// Result:
// - attacker receives 1000e18 collateral tokens
// - attacker pays 0 loan tokens
// - victim's debt remains 500e18 (bad debt)
```

**Expected outcome**: attacker receives full collateral for free; victim's debt is uncleared; lenders absorb the bad debt via loss-factor socialization.
