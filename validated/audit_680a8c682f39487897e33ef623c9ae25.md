Audit Report

## Title
Authorized Delegate Can Front-Run Victim's `EcrecoverAuthorizer` Transaction by Signing a Different `Authorization` Struct with the Same Nonce - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts signatures from any address that `Midnight.isAuthorized(authorizer, signer)` returns `true` for, not just the authorizer themselves. Because the nonce is incremented unconditionally before the signature check, an attacker who is already authorized by the victim in Midnight can construct a different `Authorization` struct (different `authorized` or `isAuthorized` fields) using the victim's current nonce, sign it with their own key, and submit it ahead of the victim's pending transaction. The nonce is consumed, causing the victim's in-flight transaction to revert with `InvalidNonce`. The attack is repeatable for every new nonce the victim produces.

## Finding Description

**Code path â€” `src/periphery/EcrecoverAuthorizer.sol` lines 24â€“48:**

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // nonce consumed here (1)

    bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
    bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
    bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
    address signer = ecrecover(digest, signature.v, signature.r, signature.s);
    require(signer != address(0), InvalidSignature());
    require(
        signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer), // (2)
        Unauthorized()
    );
    ...
    IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
}
```

**Root cause:** At (1), the nonce is incremented against `authorization.authorizer` before the signer is verified. At (2), the signer check accepts any address for which `Midnight.isAuthorized(authorizer, signer)` is `true`. This means an attacker who holds a Midnight authorization from the victim can construct an entirely different `Authorization` struct â€” with a different `authorized` address or different `isAuthorized` value â€” using the victim's current nonce `N`, sign it with their own key, and submit it. The EIP-712 digest is computed over the attacker's struct, so `ecrecover` returns the attacker's address, which passes check (2). The nonce is permanently consumed at (1), and the victim's signed message (which commits to a different struct) becomes invalid.

**Exploit flow:**

| Step | Actor | Action | State after |
|------|-------|--------|-------------|
| 0 | Victim | `midnight.setIsAuthorized(attacker, true, victim)` | `isAuthorized[victim][attacker] = true` |
| 1 | Victim | Signs `Authorization{authorizer=victim, authorized=operatorX, isAuthorized=true, nonce=N}`, broadcasts tx | Pending in mempool |
| 2 | Attacker | Observes mempool; constructs `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N}`, signs with attacker key | â€” |
| 3 | Attacker | Calls `ecrecoverAuthorizer.setIsAuthorized(attackerAuth, attackerSig)` with higher gas | `nonce[victim] = N+1`; attacker's chosen auth set |
| 4 | Victim's tx | Lands: `nonce[victim]=N+1 != N` â†’ reverts `InvalidNonce` | Victim's intended authorization never set |

Check (2) passes at step 3 because `IMidnight(MIDNIGHT).isAuthorized(victim, attacker)` is `true`. The attacker can repeat this for every new nonce the victim signs.

**Why existing checks fail:** The deadline check (line 25) does not help â€” the attacker uses the same or a fresh deadline. The `InvalidNonce` check (line 26) is the mechanism being abused, not a guard against it. The `InvalidSignature` check (line 32) passes because the attacker's signature over their own struct is valid. The authorization check (line 33â€“36) passes because the attacker is legitimately authorized in Midnight.

## Impact Explanation
The victim's `EcrecoverAuthorizer`-based authorization flow is blocked for as long as the attacker remains authorized by the victim in Midnight. Every signed message the victim produces can be front-run and its nonce consumed. The victim cannot use `EcrecoverAuthorizer` to authorize their intended operator via off-chain signatures. Additionally, the attacker can set the victim's Midnight authorization state to an arbitrary value (e.g., authorize a new address) via the same mechanism â€” though this specific sub-impact is also reachable via a direct `Midnight.setIsAuthorized` call given the precondition. The primary unique impact is a sustained, targeted DoS of the `EcrecoverAuthorizer` flow.

## Likelihood Explanation
**Preconditions:**
- The victim must have previously called `midnight.setIsAuthorized(attacker, true, victim)`. This is a normal protocol operation (authorizing an operator, bot, or contract).
- The attacker must observe the victim's pending `EcrecoverAuthorizer.setIsAuthorized` transaction in the mempool.

Both conditions are realistic on any public EVM chain with a visible mempool. The attack is repeatable: every time the victim signs a new message with nonce `N+1`, the attacker can front-run again. The victim's only escape is to revoke the attacker's Midnight authorization via a direct `midnight.setIsAuthorized(attacker, false, victim)` call (which does not go through `EcrecoverAuthorizer` and cannot be front-run in the same way).

## Recommendation
Restrict the signer check so that only the authorizer themselves can sign an `Authorization` struct for submission via `EcrecoverAuthorizer`. Remove the `IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)` branch from the check at line 33â€“36:

```solidity
require(signer == authorization.authorizer, Unauthorized());
```

If the intent is to allow authorized delegates to submit authorizations on behalf of the authorizer, the nonce should be keyed per `(authorizer, signer)` pair rather than per `authorizer` alone, so that a delegate consuming a nonce does not invalidate the authorizer's own signed messages. Alternatively, require that the signature always come from the authorizer, and let delegates relay the authorizer's pre-signed messages without being able to substitute their own.

## Proof of Concept
**Minimal manual steps:**
1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. As `victim`, call `midnight.setIsAuthorized(attacker, true, victim)`.
3. As `victim`, sign `Authorization{authorizer=victim, authorized=operatorX, isAuthorized=true, nonce=0, deadline=T}` off-chain. Do not submit yet.
4. As `attacker`, construct `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=T}`, sign with attacker's key, and call `ecrecoverAuthorizer.setIsAuthorized(attackerAuth, attackerSig)`.
5. Verify: `nonce[victim] == 1`, `midnight.isAuthorized(victim, attacker) == true` (set by attacker's call).
6. Now submit victim's original signed transaction. Observe it reverts with `InvalidNonce`.
7. Repeat from step 3 with `nonce=1` to confirm the attack is indefinitely repeatable.
