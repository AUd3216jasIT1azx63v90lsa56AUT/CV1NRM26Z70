Audit Report

## Title
Fee-on-Transfer Collateral Token Inflates `_position.collateral[i]`, Enabling Undercollateralized Borrowing and Bad Debt - (`src/Midnight.sol`)

## Summary

`supplyCollateral` records the caller-supplied `assets` value directly into `_position.collateral[collateralIndex]` before executing the token transfer, with no balance-delta check. When the collateral token charges a fee on transfer, fewer tokens arrive at the contract than are recorded. `isHealthy` then computes `maxDebt` from the inflated stored value, making the position appear healthier than it is, allowing undercollateralized borrowing and creating bad debt that is socialized among lenders.

## Finding Description

**Root cause — `supplyCollateral` (`src/Midnight.sol` lines 532–545):**

The accounting update uses the raw input `assets` and occurs before the transfer:

```solidity
// line 533 — records input, not actual received amount
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
// ...
// line 545 — transfer happens after; fee-on-transfer tokens deliver assets - fee
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34) only verifies the call succeeded and that the return value is `true`; it performs no balance-before/balance-after check. A fee-on-transfer token returns `true` while delivering fewer tokens than requested.

**`isHealthy` uses the inflated stored value (`src/Midnight.sol` lines 954–955):**

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

The stored `_position.collateral[i]` is the inflated value, so `maxDebt` is overstated relative to actual contract holdings.

**`touchMarket` performs no token-type validation (`src/Midnight.sol` lines 755–791):**

Market creation validates only LLTV tiers, `maxLif`, collateral sort order, and count. Any ERC20 address is accepted as a collateral token.

**Exploit flow:**
1. Attacker deploys a `FeeOnTransferERC20` (e.g., 1% fee) and calls `touchMarket` with it as collateral — permissionless.
2. Attacker calls `supplyCollateral(market, 0, 100e18, attacker)`:
   - `_position.collateral[0]` is written as `100e18`.
   - Only `99e18` tokens arrive at the contract (1% fee deducted).
3. Attacker takes a lender's buy offer, borrowing up to `100e18 * price * LLTV / WAD` units.
4. `isHealthy` computes `maxDebt` using `100e18` — overstated by `1e18 * price * LLTV / WAD`.
5. Actual backing is only `99e18` tokens; the position is undercollateralized from inception.
6. At liquidation, the seized collateral is insufficient to cover the debt, creating bad debt socialized among lenders.

**Existing checks are insufficient:**
- `SafeTransferLib` only checks call success and boolean return, not actual balance delta.
- `touchMarket` has no token allowlist or fee-on-transfer detection.
- `isHealthy` trusts stored accounting without any cross-check against actual token balances.

## Impact Explanation

The borrower's recorded collateral exceeds the actual contract balance from the moment of the first `supplyCollateral` call. The position is undercollateralized at inception. When the position is liquidated, the seized collateral is insufficient to cover the repaid debt, creating bad debt that is immediately socialized among lenders via the loss factor mechanism. This directly violates the protocol's core invariant stated in `live_context.json`: *"ERC20 transfer deltas must match accounting deltas"* and *"contract token balances must cover withdrawable assets, collateral claims, credit redemptions, and accrued fees."* The impact class is **bad debt creation / protocol insolvency**, which is explicitly listed as a best bug class.

## Likelihood Explanation

Market creation is fully permissionless. Any ERC20 can be used as collateral. Fee-on-transfer tokens are a well-known token class (USDT with fee enabled, STA, PAXG, or any custom token). The attacker needs only to deploy or use an existing fee-on-transfer token, create a market, and call `supplyCollateral`. The attack is repeatable on every `supplyCollateral` call with such a token and scales linearly with the fee rate and supplied amount. No privileged access is required.

## Recommendation

Record the actual received amount by measuring the balance delta around the transfer:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

This ensures the accounting always reflects the actual token balance increase, regardless of token transfer mechanics. Apply the same pattern symmetrically in `withdrawCollateral` if needed.

## Proof of Concept

**Minimal Foundry test plan:**

1. Deploy a `FeeOnTransferERC20` with a 1% transfer fee.
2. Deploy `Midnight` and create a market using the fee token as collateral via `touchMarket`.
3. Call `supplyCollateral(market, 0, 100e18, attacker)`.
4. Assert `midnight.collateral(id, attacker, 0) == 100e18` (recorded).
5. Assert `IERC20(feeToken).balanceOf(address(midnight)) == 99e18` (actual).
6. Have a lender post a buy offer; attacker takes it, borrowing `100e18 * price * LLTV / WAD` units.
7. Assert `isHealthy(market, id, attacker) == true` despite actual collateral being only `99e18`.
8. Drop oracle price slightly; assert liquidation results in bad debt (seized collateral value < repaid debt).

        Position storage _position = position[id][borrower];

        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
