Audit Report

## Title
Post-maturity bad-debt liquidation reverts with arithmetic underflow when `lif < maxLif` and `seizedAssets > 0` - (File: src/Midnight.sol)

## Summary
During the 15-minute `TIME_TO_MAX_LIF` ramp window after market maturity, the bad-debt write-down reduces `_position.debt` using `maxLif` (line 628), but `repaidUnits` is then computed using the smaller ramped `lif` (line 650). Because dividing by a smaller denominator yields a strictly larger `repaidUnits`, the subtraction `_position.debt -= repaidUnits` at line 676 reverts with an arithmetic underflow, making any position with bad debt permanently un-liquidatable via the `seizedAssets` path for the entire ramp window.

## Impact Explanation
Any liquidator calling `liquidate` with `seizedAssets > 0` on a position carrying bad debt during the 15-minute post-maturity ramp window receives a guaranteed revert. The `seizedAssets` path is the natural path for liquidators who want to seize all available collateral. The position cannot be cleared via this path for the entire ramp window, violating the core protocol invariant that unhealthy positions must always remain liquidatable. This constitutes a deterministic, temporary freeze of the liquidation mechanism for affected positions, with direct risk of protocol insolvency if bad-debt positions cannot be cleared promptly.

## Likelihood Explanation
All three preconditions are permissionless and time-based: market maturity is reached naturally, bad debt arises from normal price movements, and the 15-minute window is precisely when liquidation incentives are lowest (lif near WAD) and liquidators are most likely to seize all available collateral via `seizedAssets`. The bug is deterministic and repeatable for any parameter set satisfying `badDebt > 0` and `lif < maxLif`.

## Recommendation
In the post-maturity branch, cap `repaidUnits` at `_position.debt` before the subtraction at line 676. For example, after computing `repaidUnits` at line 650, add:

```solidity
if (postMaturityMode) {
    repaidUnits = UtilsLib.min(repaidUnits, _position.debt);
    seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
}
```

Alternatively, use `lif` instead of `maxLif` in the bad-debt write-down loop so that both computations use a consistent LIF value, though this changes the economic semantics of the bad-debt accounting.

## Proof of Concept
Minimal Foundry unit test:

1. Create a market with a single collateral, `maxLif = 1.1e18`, `lltv` any allowed value.
2. Supply loan tokens; borrow such that `originalDebt = 1000` units against `collateral = 1000` at `price = ORACLE_PRICE_SCALE`.
3. Advance `block.timestamp` to `market.maturity + 1` (inside the 15-minute ramp, so `lif ã WAD`).
4. Call `liquidate(..., seizedAssets = 1000, repaidUnits = 0, ..., postMaturityMode = true)`.
5. Observe revert with arithmetic underflow at line 676.
6. Confirm the `repaidUnits` path (`seizedAssets = 0, repaidUnits = 910`) succeeds, proving only the `seizedAssets` path is broken.
