Audit Report

## Title
Unbounded zero-unit take loop in bundle functions allows gas griefing via fully-consumed group front-run - (File: src/periphery/MidnightBundles.sol)

## Summary
All four bundle functions in `MidnightBundles.sol` dispatch `IMidnight.take()` with `units=0` when `ConsumableUnitsLib.consumableUnits()` returns 0 for a fully-consumed group, because no guard skips the `try` call when `unitsToTake == 0`. Since `Midnight.take()` explicitly permits no-op takes on fully-consumed offers (confirmed by the inline comment at line 93), each such call succeeds and returns `(0, 0)`, leaving `filledUnits` unchanged. An attacker who controls the maker address can front-run a taker's bundle transaction by calling `setConsumed(group, type(uint256).max, maker)`, converting every offer in the `takes[]` array into a zero-unit no-op and forcing the taker to burn O(N × per-take gas) before the transaction reverts with `OutOfOffers()`.

## Finding Description

**Root cause:** No `if (unitsToTake == 0) continue;` guard exists before the `try` call in any of the four bundle loops.

**Exact code path in `supplyCollateralAndSellWithUnitsTarget`:**

Loop at lines 144–160 computes: [1](#0-0) 

`ConsumableUnitsLib.consumableUnits()` returns `offer.maxUnits.zeroFloorSub(consumed)`, which is 0 when `consumed >= maxUnits`: [2](#0-1) 

So `unitsToTake = min(targetUnits - filledUnits, takes[i].units, 0) = 0`. The `try` is dispatched unconditionally: [3](#0-2) 

Inside `Midnight.take()`, with `units=0` and `consumed == maxUnits`, the consumed check at lines 371–372 evaluates `newConsumed = maxUnits + 0 = maxUnits`, so `require(maxUnits <= maxUnits)` passes: [4](#0-3) 

The protocol comment at line 93 explicitly confirms this is intentional behavior: [5](#0-4) 

Before the consumed check, the ratifier external call executes unconditionally: [6](#0-5) 

`_updatePosition` is called for buyer and/or seller if they hold credit: [7](#0-6) 

The health check at the end of `take()` may invoke an oracle if the seller has debt: [8](#0-7) 

`take()` returns `(0, 0)`. Back in the bundle, `filledUnits += 0`, the loop condition `filledUnits < targetUnits` remains true, and the next iteration begins. After all N iterations the loop exits by exhausting `takes.length`, and `require(filledUnits == targetUnits, OutOfOffers())` reverts the entire transaction.

**`setConsumed` is callable by the maker (any unprivileged user acting as their own `onBehalf`):** [9](#0-8) 

**Why existing checks fail:**
- The `try/catch` at lines 152–160 is designed to tolerate asynchrony (e.g., an offer taken by someone else between block construction and execution). It silently swallows both reverts and zero-return successes, providing no protection against this case.
- The loop condition `filledUnits < targetUnits` does not terminate early because `filledUnits` never advances when all takes return 0.
- `ConsumableUnitsLib.consumableUnits()` correctly returns 0, but the bundle does not act on this signal before dispatching the take.

The same missing guard exists identically in all four bundle functions: `buyWithUnitsTargetAndWithdrawCollateral` (lines 74–85), `supplyCollateralAndSellWithUnitsTarget` (lines 147–160), `buyWithAssetsTargetAndWithdrawCollateral` (lines 208–221), and `supplyCollateralAndSellWithAssetsTarget` (lines 285–300). ## Impact Explanation
Every zero-unit take in `Midnight.take()` executes: one external ratifier call (line 356), up to two `_updatePosition` calls with multiple SLOADs/SSTOREs (lines 379–380), position arithmetic touching storage slots (lines 408–414), a `claimableSettlementFee` SLOAD+SSTORE (line 418), a `Take` event emission (lines 425–442), a TSTORE liquidation lock (line 444), two `safeTransferFrom` calls with amount 0 (lines 455–456), and an `isHealthy` call that may invoke an oracle (line 476). Conservatively, each zero-unit take costs on the order of 30,000–80,000 gas depending on position state and oracle complexity. For N = 100 offers, the taker's transaction burns 3,000,000–8,000,000 gas before reverting with `OutOfOffers()`. The taker loses all gas; no funds are transferred because the transaction reverts. The attack is a repeatable, targeted DoS against any user of the bundle functions.

## Likelihood Explanation
**Preconditions:**
1. Attacker controls a maker address — any unprivileged user qualifies.
2. Attacker creates N offers sharing the same `(maker, group)` pair — no cost beyond gas.
3. Attacker calls `setConsumed(group, type(uint256).max, maker)` — one cheap transaction, callable by the maker on their own behalf.
4. A victim taker calls a bundle function with those N offers in `takes[]`.

The front-run variant is straightforward on any chain with a public mempool. The social-engineering variant (listing N exhausted offers in an off-chain order book) requires no front-running at all. The attack is repeatable: the attacker can re-exhaust a group after any partial fill, or create new groups. Likelihood is medium-high given that bundle functions are designed for use with aggregators and order-book integrations that fetch offer lists off-chain.

## Recommendation
Add a zero-unit guard immediately before each `try` call in all four bundle loops:

```solidity
if (unitsToTake == 0) continue;
```

This single-line fix in each of the four loops eliminates the ability to dispatch a no-op take, ensuring that fully-consumed offers are skipped without incurring the gas cost of a cross-contract call. The fix is consistent with the existing intent of `ConsumableUnitsLib.consumableUnits()` returning 0 as a signal that the offer is exhausted.

## Proof of Concept
**Minimal manual steps:**
1. Deploy `Midnight` and `MidnightBundles` on a fork.
2. Attacker creates N buy offers (e.g., N = 50) all sharing `group = keccak256("g")` with `maxUnits = 1000`.
3. Attacker calls `setConsumed("g", type(uint256).max, attacker)`.
4. Victim calls `supplyCollateralAndSellWithUnitsTarget` with `targetUnits = 1`, `takes[]` containing all N offers.
5. Observe: the loop iterates all N times, each dispatching `take(..., 0, ...)` which succeeds and returns `(0, 0)`. After N iterations, the transaction reverts with `OutOfOffers()`. Victim loses all gas; attacker spent only the gas for one `setConsumed` call.

**Fuzz/invariant test plan:** Fuzz `takes.length` from 1 to 200 with all offers pre-exhausted via `setConsumed`. Assert that gas consumed by the victim's transaction grows linearly with `takes.length` and always reverts — confirming unbounded gas consumption with no state change.
