Audit Report

## Title
Maker-controlled `offer.callback` designates arbitrary third-party as `payer` for buy offers, enabling drain of victim's loan tokens without maker payment - (File: src/Midnight.sol)

## Summary
In `Midnight.take`, when `offer.buy = true`, the `payer` for both `safeTransferFrom` calls is derived directly from `offer.callback`, a field freely set by the maker with no authorization check tying the callback address to the maker. A malicious maker can pair a self-deployed permissive ratifier with `offer.callback = victimContract` to cause the protocol to pull `buyerAssets` from the victim instead of the maker, while the maker receives `buyerCreditIncrease` credit at zero cost. The existing ratifier gate is insufficient because the maker controls which ratifier they authorize.

## Finding Description

**Exact code path — `src/Midnight.sol` L420–456:**

```solidity
address buyerCallback = offer.buy ? offer.callback : takerCallback;   // L420
address payer = buyerCallback != address(0)
    ? buyerCallback
    : (offer.buy ? buyer : msg.sender);                                // L422

buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);           // L410

if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback).onBuy(...) == CALLBACK_SUCCESS,    // L448-451
        WrongBuyCallbackReturnValue()
    );
}
SafeTransferLib.safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets); // L455
SafeTransferLib.safeTransferFrom(loanToken, payer, receiver, sellerAssets);                    // L456
```

**Root cause:** There is no check of the form `require(isAuthorized[offer.maker][offer.callback], ...)` analogous to the ratifier gate at L355. The `offer.callback` field is included in the offer hash and is covered by the ratifier's signature — but the maker controls which ratifier they authorize. Any maker can call `setIsAuthorized(permissiveRatifier, true)` (L731–734) and deploy a ratifier that unconditionally returns `CALLBACK_SUCCESS`, bypassing the only structural guard.

**Exploit flow:**
1. Attacker deploys `PermissiveRatifier` returning `CALLBACK_SUCCESS` for any offer.
2. Attacker calls `midnight.setIsAuthorized(permissiveRatifier, true, attacker)` — no privilege required (`onBehalf == msg.sender` satisfies L732).
3. Attacker identifies `victimContract`: a contract that (a) holds loan tokens with an ERC-20 approval to Midnight and (b) implements `IBuyCallback.onBuy()` returning `CALLBACK_SUCCESS` without validating the `buyer` argument. The protocol's own reference `LendCallback` (test/TakeTest.sol L1452–1470) follows exactly this pattern — it validates only the market ID, not `buyer`.
4. Attacker constructs `offer` with `offer.buy = true`, `offer.maker = attacker`, `offer.callback = victimContract`, `offer.ratifier = permissiveRatifier`.
5. Attacker (or any third party) calls `take(offer, ...)`.
6. `buyerCallback = offer.callback = victimContract`; `payer = victimContract`.
7. `victimContract.onBuy(...)` is called → returns `CALLBACK_SUCCESS`.
8. `safeTransferFrom(loanToken, victimContract, address(this), buyerAssets - sellerAssets)` executes.
9. `safeTransferFrom(loanToken, victimContract, receiver, sellerAssets)` executes.
10. `buyerPos.credit += buyerCreditIncrease` (L410) — attacker gains credit; victim paid.

**Why existing checks fail:**
- L346 (`TakerUnauthorized`): gates the taker, irrelevant to callback.
- L355 (`RatifierUnauthorized`): requires `isAuthorized[offer.maker][offer.ratifier]` — the attacker satisfies this trivially by authorizing their own permissive ratifier.
- L356 (`RatifierFail`): the permissive ratifier returns `CALLBACK_SUCCESS` unconditionally.
- The Certora rule `takeOnlyExplicitPayer` (L95–114 of `OnlyExplicitPayerCanLoseTokens.spec`) proves only that tokens are pulled from "the buyerCallback that returned `CALLBACK_SUCCESS`" — it sets `callbackAllowed = true` for any address returning `CALLBACK_SUCCESS` (L62–64) regardless of maker authorization. The spec does not model the constraint that the callback must be authorized by the maker, so this attack class is outside its proof scope.

## Impact Explanation
The attacker gains `buyerCreditIncrease` credit in the market (L410) — a real, withdrawable asset — while paying nothing. The victim contract loses up to `offer.maxAssets` / `offer.maxUnits` worth of loan tokens per invocation. The attack is repeatable across multiple `take` calls until the victim's approval or balance is exhausted. This constitutes direct theft of assets from a third party, violating the core protocol invariant that every credit increase must be matched by a corresponding payment from the buyer.

## Likelihood Explanation
**Attacker requirements:** no privileged access; any address can deploy a permissive ratifier and call `setIsAuthorized`. **Victim requirements:** (1) ERC-20 approval to Midnight for the loan token — standard for any contract participating as a buy-callback payer; (2) `onBuy` returns `CALLBACK_SUCCESS` without validating the `buyer` argument — the protocol's own reference `LendCallback` implementation follows this exact pattern, making it a realistic and likely victim profile for any integrator following the reference design. The attack requires no oracle manipulation, no flash loan, no admin key, and no user mistake. It is fully repeatable up to the victim's approval limit.

## Recommendation
Add an authorization check requiring that `offer.callback` is explicitly authorized by `offer.maker`, mirroring the existing ratifier gate:

```solidity
require(
    offer.callback == address(0) || isAuthorized[offer.maker][offer.callback],
    CallbackUnauthorized()
);
```

This ensures the maker cannot designate an arbitrary third-party contract as the payer. Alternatively, document and enforce (via interface or protocol convention) that all `IBuyCallback` implementations must validate that `buyer == address(this)` or an explicitly authorized address before returning `CALLBACK_SUCCESS` and granting token approvals — and update the reference `LendCallback` accordingly.

## Proof of Concept
**Minimal Forge test outline:**

```solidity
// 1. Deploy permissive ratifier
PermissiveRatifier ratifier = new PermissiveRatifier(); // returns CALLBACK_SUCCESS always

// 2. Attacker authorizes it
vm.prank(attacker);
midnight.setIsAuthorized(address(ratifier), true, attacker);

// 3. Deploy victim: implements onBuy, does NOT validate buyer, has tokens + approval
VictimCallback victim = new VictimCallback();
deal(loanToken, address(victim), LARGE_AMOUNT);
vm.prank(address(victim));
ERC20(loanToken).approve(address(midnight), type(uint256).max);

// 4. Construct malicious offer

offer.buy = true;
offer.maker = attacker;
offer.callback = address(victim);
offer.ratifier = address(ratifier);
offer.market = market;
offer.maxUnits = units;
offer.tick = MAX_TICK;
offer.expiry = block.timestamp + 1 days;

// 5. Take the offer (attacker or anyone)
vm.prank(attacker);
midnight.take(offer, "", units, taker, taker, address(0), "");

// 6. Assert: attacker gained credit, victim lost tokens
assertGt(midnight.creditOf(id, attacker), 0);
assertLt(ERC20(loanToken).balanceOf(address(victim)), LARGE_AMOUNT);
```

`VictimCallback.onBuy` validates only `msg.sender == address(midnight)` and returns `CALLBACK_SUCCESS` — matching the pattern of the protocol's own reference `LendCallback` which does not validate `buyer`.
