Audit Report

## Title
Maker-controlled `offer.callback` designates arbitrary third-party as `payer` for buy offers, enabling drain of victim's loan tokens without maker payment - (File: src/Midnight.sol)

## Summary
In `Midnight.take`, when `offer.buy = true`, the `payer` for both `safeTransferFrom` calls is derived directly from `offer.callback`, a field freely set by the maker with no authorization check tying the callback address to the maker. A malicious maker can pair a self-deployed permissive ratifier with `offer.callback = victimContract` to cause the protocol to pull `buyerAssets` from the victim instead of the maker, while the maker receives `buyerCreditIncrease` credit at zero cost.

## Finding Description

**Exact code path — `src/Midnight.sol` L420–456:**

`buyerCallback` is set to `offer.callback` when `offer.buy = true` [1](#0-0) , and `payer` is set to `buyerCallback` when non-zero [2](#0-1) . Both `safeTransferFrom` calls then pull from `payer` [3](#0-2) .

**Root cause:** No check of the form `require(isAuthorized[offer.maker][offer.callback], ...)` exists. The only structural gates are the ratifier checks at L355–356 [4](#0-3) , which the attacker trivially satisfies by calling `setIsAuthorized(permissiveRatifier, true, attacker)` — permitted for any address with no privilege required [5](#0-4) .

**Exploit flow:**
1. Attacker deploys `PermissiveRatifier` returning `CALLBACK_SUCCESS` unconditionally.
2. Attacker calls `setIsAuthorized(permissiveRatifier, true, attacker)`.
3. Attacker identifies `victimContract`: holds loan tokens with ERC-20 approval to Midnight, and implements `onBuy` returning `CALLBACK_SUCCESS` without validating `buyer`. The protocol's own reference `LendCallback` (test/TakeTest.sol L1452–1470) follows this exact pattern: it validates only the market ID, approves Midnight for `buyerAssets`, and returns `CALLBACK_SUCCESS` — without checking `buyer` [6](#0-5) .
4. Attacker constructs `offer` with `offer.buy = true`, `offer.maker = attacker`, `offer.callback = victimContract`, `offer.ratifier = permissiveRatifier`.
5. Attacker calls `take(offer, ...)`.
6. `buyerCallback = victimContract`; `payer = victimContract`.
7. `victimContract.onBuy(...)` is called → approves Midnight → returns `CALLBACK_SUCCESS`.
8. `safeTransferFrom(loanToken, victimContract, address(this), buyerAssets - sellerAssets)` executes.
9. `safeTransferFrom(loanToken, victimContract, receiver, sellerAssets)` executes.
10. `buyerPos.credit += buyerCreditIncrease` (L410) — attacker gains credit; victim paid.

**Why existing checks fail:**
- L346 (`TakerUnauthorized`): gates the taker, irrelevant to callback.
- L355–356 (`RatifierUnauthorized` / `RatifierFail`): attacker satisfies both by authorizing their own permissive ratifier.
- Certora rule `takeOnlyExplicitPayer` (L95–114): sets `buyCallbackAllowed = true` for any address [7](#0-6) , and `callbackAllowed = true` for any address returning `CALLBACK_SUCCESS` [8](#0-7)  — with no constraint that the callback is authorized by the maker. The attack class is outside the proof scope.

## Impact Explanation
The attacker gains `buyerCreditIncrease` credit in the market (L410) — a real, withdrawable asset — while paying nothing. The victim contract loses up to `offer.maxAssets` / `offer.maxUnits` worth of loan tokens per invocation. The attack is repeatable across multiple `take` calls until the victim's approval or balance is exhausted. This constitutes direct theft of assets from a third party, violating the core protocol invariant that every credit increase must be matched by a corresponding payment from the buyer.

## Likelihood Explanation
Attacker requirements: no privileged access; any address can deploy a permissive ratifier and call `setIsAuthorized`. Victim requirements: (1) ERC-20 approval to Midnight for the loan token — standard for any contract participating as a buy-callback payer; (2) `onBuy` returns `CALLBACK_SUCCESS` without validating the `buyer` argument. The protocol's own reference `LendCallback` implementation follows this exact pattern [9](#0-8) , making it a realistic and likely victim profile for any integrator following the reference design. The attack requires no oracle manipulation, no flash loan, no admin key, and no user mistake. It is fully repeatable up to the victim's approval limit.

## Recommendation
Add an authorization check tying `offer.callback` to `offer.maker` when `offer.buy = true`. One approach: require `offer.callback == address(0) || offer.callback == offer.maker || isAuthorized[offer.maker][offer.callback]` before accepting the callback as `payer`. Alternatively, document that `onBuy` implementations MUST validate the `buyer` argument equals their own address (or an address they control), and update the reference `LendCallback` to enforce this.

## Proof of Concept
1. Deploy `PermissiveRatifier` returning `CALLBACK_SUCCESS` for any offer.
2. Deploy `VictimCallback` mirroring `LendCallback` (validates market ID, approves Midnight, returns `CALLBACK_SUCCESS`, does not check `buyer`); fund it with loan tokens.
3. Call `midnight.setIsAuthorized(permissiveRatifier, true, attacker)`.
4. Construct `offer` with `offer.buy = true`, `offer.maker = attacker`, `offer.callback = VictimCallback`, `offer.ratifier = permissiveRatifier`, valid market and tick.
5. Call `midnight.take(offer, ...)`.
6. Assert: `VictimCallback` loan token balance decreased by `buyerAssets`; `attacker` position credit increased by `buyerCreditIncrease`; attacker paid zero tokens.
