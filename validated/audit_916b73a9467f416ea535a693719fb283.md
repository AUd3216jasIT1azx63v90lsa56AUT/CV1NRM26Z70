### Title
Liquidation Chaining via High-LLTV Collateral Selection Bypasses Recovery Close Factor — (`src/Midnight.sol`)

### Summary

In Midnight's multi-collateral markets, the Recovery Close Factor (RCF) is computed using only the LLTV of the **specific collateral being seized** in each `liquidate` call. A liquidator can exploit this by targeting the highest-LLTV collateral first, which inflates `maxRepaid` to a value exceeding the collateral's own worth, triggering the RCF deactivation clause. After seizing all high-LLTV collateral, the position remains unhealthy, and the liquidator chains a second call on the remaining collateral. The total amount liquidated can be an order of magnitude larger than the minimum required to restore health.

### Finding Description

The `liquidate` function computes `maxDebt` as the sum across **all** activated collaterals:

```
maxDebt += collateral[i] * price[i] / ORACLE_PRICE_SCALE * lltv[i] / WAD
``` [1](#0-0) 

Then, in normal mode, the RCF cap is computed using only the LLTV of the **liquidated collateral**:

```solidity
uint256 lltv = market.collateralParams[collateralIndex].lltv;
uint256 maxRepaid = lltv < WAD
    ? (_position.debt - maxDebt).mulDivUp(WAD * WAD, WAD * WAD - lif * lltv)
    : type(uint256).max;
``` [2](#0-1) 

The RCF is **deactivated** when `maxRepaid` exceeds the collateral's own value divided by `lif`:

```solidity
require(
    repaidUnits <= maxRepaid
        || _position.collateral[collateralIndex].mulDivDown(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
            .mulDivDown(WAD, lif).zeroFloorSub(maxRepaid) < market.rcfThreshold,
    RecoveryCloseFactorConditionsViolated()
);
``` [3](#0-2) 

When `lltv_liquidated` is high (e.g., `LLTV_7 = 0.98`), the denominator `WAD² - lif * lltv` becomes tiny (≈ 0.01 × WAD²), making `maxRepaid` ≈ 99× the deficit. This almost always exceeds the collateral's value, so `zeroFloorSub` returns 0, which is `< rcfThreshold` (any positive threshold), and the RCF is deactivated. The liquidator seizes the entire high-LLTV collateral bucket.

After this first call, the position is still unhealthy because the high-LLTV collateral was contributing the most to `maxDebt`. The liquidator then calls `liquidate` again on the remaining low-LLTV collateral, this time with the RCF properly enforced. The two-call chain liquidates far more than a single optimal call on the low-LLTV collateral would have.

### Impact Explanation

A borrower with a multi-collateral position (e.g., one high-LLTV and one low-LLTV collateral) can be liquidated for **an order of magnitude more** than the minimum needed to restore health. In the concrete scenario below, the borrower loses ~104.6 units of debt repaid (and corresponding collateral seized) instead of ~7.9 units — a **13× over-liquidation**. This is a direct, quantifiable loss of collateral for the borrower beyond what the protocol's RCF mechanism is designed to permit.

### Likelihood Explanation

Multi-collateral markets are a first-class feature of Midnight (up to 128 collaterals per market, up to 16 per borrower). Any market that includes at least two collaterals with significantly different LLTVs (e.g., one at `0.98` and one at `0.385`) is vulnerable. No privileged access is required — any external liquidator can execute this by calling `liquidate` twice (or via `multicall`). The only precondition is that the borrower is unhealthy, which is a normal protocol state.

### Recommendation

After each `liquidate` call in normal mode, verify that the position is either healthy or that the seized collateral was fully exhausted (i.e., the RCF deactivation was triggered by a genuine dust constraint, not by a high-LLTV amplification). Concretely, add a post-liquidation health check:

```solidity
// After updating collateral and debt:
require(
    isHealthy(market, id, borrower) || newCollateral == 0,
    ChainLiquidationNotAllowed()
);
```

Alternatively, compute `maxRepaid` using the **weighted average LLTV** across all collaterals rather than only the liquidated collateral's LLTV, so the RCF correctly reflects the global position health improvement from the seizure.

### Proof of Concept

**Setup** (using allowed LLTV tiers from `ConstantsLib.sol`):

| Collateral | LLTV | maxLif (cursor=0.5) | Amount | Price | Value | Contribution to maxDebt |
|---|---|---|---|---|---|---|
| A | 0.98 | ≈1.0101 | 100 | 1 | 100 | 98.0 |
| B | 0.385 | ≈1.4440 | 100 | 1 | 100 | 38.5 |

- `maxDebt = 136.5`
- `debt = 140` (unhealthy; deficit = 3.5)

**Attack — Call 1: liquidate collateral A (high LLTV = 0.98)**

- `maxRepaid_A = 3.5 / (1 - 1.0101 × 0.98) ≈ 3.5 / 0.0101 ≈ 346`
- `collateral_A_value / lif_A = 100 / 1.0101 ≈ 99`
- `zeroFloorSub(99, 346) = 0 < rcfThreshold` → **RCF deactivated**
- Liquidator seizes all 100 units of collateral A, repays ≈99 units of debt
- State after: `debt = 41`, `maxDebt = 38.5` (only B remains) → **still unhealthy**

**Attack — Call 2: liquidate collateral B (low LLTV = 0.385)**

- `maxRepaid_B = (41 - 38.5) / (1 - 1.4440 × 0.385) ≈ 2.5 / 0.4441 ≈ 5.63`
- RCF enforced normally; liquidator repays 5.63 units, seizes ≈8.13 units of B
- State after: `debt ≈ 35.4`, `maxDebt ≈ 35.4` → **healthy**

**Total repaid by attacker: ≈104.6 units**

**Baseline — single optimal liquidation of collateral B only:**

- `maxRepaid_B = 3.5 / 0.4441 ≈ 7.88`
- Repays 7.88 units, seizes ≈11.4 units of B → position healthy

**Total repaid in baseline: ≈7.88 units**

**Result: 13× over-liquidation.** The borrower loses ≈96.7 additional units of collateral A and ≈3.3 additional units of collateral B beyond what was necessary to restore health.
