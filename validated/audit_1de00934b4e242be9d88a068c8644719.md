Audit Report

## Title
Bad Debt Over-Realization via `supplyCollateral` Reentrancy in `onLiquidate` Callback - (File: src/Midnight.sol)

## Summary

`liquidate()` snapshots the borrower's collateral bitmap, computes and commits `badDebt` to storage (slashing lenders via `_marketState.lossFactor`), and only then invokes the `onLiquidate` external callback. Because no reentrancy guard is set on `liquidate()` and `supplyCollateral` is callable by any authorized address at any time, a malicious callback can supply additional collateral for the borrower mid-execution. The `badDebt` already written to storage is larger than the true undercollateralization at callback-completion time, causing lenders to suffer a permanent, unwarranted credit loss while the attacker recovers the supplied collateral.

## Finding Description

**Exact code path in `src/Midnight.sol`:**

- **Line 606**: Bitmap snapshot taken: `uint128 _collateralBitmap = _position.collateralBitmap;`
- **Lines 607–618**: `badDebt` computed by iterating only over bits present in the snapshot. Any collateral added after this point is invisible to the computation.
- **Lines 626–641**: `badDebt` irreversibly written to storage — `_position.debt -= uint128(badDebt)` and `_marketState.lossFactor` updated (lenders slashed).
- **Lines 698–715**: `ILiquidateCallback(callback).onLiquidate(...)` is called **after** the bad-debt write. No reentrancy lock is set before this call.
- **Line 444**: The `liquidationLocked` / `tExchange` mechanism is set exclusively inside `take()`, never inside `liquidate()`.
- **Lines 524–546**: `supplyCollateral` only checks `isAuthorized[onBehalf][msg.sender]`. It sets a new bit in `_position.collateralBitmap` and increases `_position.collateral[newIndex]` with no reentrancy guard.

**Root cause:** The protocol assumes that the collateral state is frozen between the `badDebt` computation and the external callback. This assumption is broken because `supplyCollateral` is callable by any authorized address at any time, including during the `onLiquidate` callback, and `liquidate()` has no reentrancy protection.

**Exploit flow:**

1. Borrower (= attacker) holds debt `D`, collateral `C₀` at index 0 only. Position has genuine bad debt: `D > C₀ · price₀ / ORACLE_PRICE_SCALE · lltv₀ / WAD`.
2. Attacker calls `setIsAuthorized(callbackContract, true, borrower)`.
3. Attacker calls `liquidate(market, 0, 0, 0, borrower, false, receiver, callbackContract, data)`.
4. `badDebt = D − C₀ · price₀ / ORACLE_PRICE_SCALE · WAD / maxLif₀` computed from snapshot (index 0 only).
5. `_position.debt -= badDebt`; `_marketState.lossFactor` updated — lenders slashed by `badDebt`.
6. `onLiquidate` fires. Callback calls `supplyCollateral(market, 1, largeAmount, borrower)`. Bit 1 set in `_position.collateralBitmap`; `_position.collateral[1] = largeAmount`. This collateral was absent from the snapshot and therefore not subtracted from `badDebt`.
7. Callback returns `CALLBACK_SUCCESS`. Zero loan tokens pulled (`repaidUnits = 0`).
8. Post-liquidation: position debt = `C₀ · price₀ / ORACLE_PRICE_SCALE · WAD / maxLif₀`; collateral[1] = `largeAmount`.
9. Attacker withdraws collateral[1] down to the minimum needed to satisfy `isHealthy`, recovering the bulk of `largeAmount`. The excess debt reduction (`largeAmount · price₁ / ORACLE_PRICE_SCALE · WAD / maxLif₁`) is the attacker's net gain at lenders' expense.

**Why existing checks fail:** The `liquidationLocked` guard checks whether the borrower is locked during a `take()` call — it is not a reentrancy guard on `liquidate()` itself and is never set during `liquidate()`. `supplyCollateral` has no awareness of an in-progress liquidation.

## Impact Explanation

Lenders suffer a permanent, unwarranted credit loss. `_marketState.lossFactor` is updated with a `badDebt` value that is larger than the true undercollateralization at the time the callback completes. The difference — `largeAmount · price₁ / ORACLE_PRICE_SCALE · WAD / maxLif₁` loan-token units — is the excess slashing. Because `WAD / maxLif > lltv` for all valid LLTV tiers, the excess debt reduction per unit of collateral exceeds the collateral value needed to maintain position health, making the attack profitable. The loss is irreversible: `lossFactor` is monotonically non-decreasing.

## Likelihood Explanation

All preconditions are satisfiable by a single actor who is simultaneously borrower and liquidator (self-liquidation is explicitly supported per the line 577 comment). No oracle manipulation, governance action, or victim mistake is required. The attack is applicable to any multi-collateral market with at least one unused collateral index for the borrower, which is the common case. It is repeatable and permissionless given the self-authorization path.

## Recommendation

Set a transient reentrancy lock at the start of `liquidate()` (analogous to the `tExchange` pattern already used in `take()`) and clear it at the end. Alternatively, re-compute `badDebt` after the callback using the post-callback collateral state, or prohibit `supplyCollateral` calls that would affect a position currently undergoing bad-debt realization by checking a transient "liquidation in progress" flag inside `supplyCollateral`.

## Proof of Concept

**Minimal Forge test outline:**

```solidity
// 1. Deploy market with two collateral params (index 0 and index 1).
// 2. Borrower supplies C0 of collateral[0], borrows D (genuine bad debt: D > C0*price0/lltv0).
// 3. Borrower authorizes MaliciousCallback contract.
// 4. MaliciousCallback.onLiquidate() calls midnight.supplyCollateral(market, 1, largeAmount, borrower).
// 5. Call midnight.liquidate(market, 0, 0, 0, borrower, false, receiver, maliciousCallback, "").
// 6. Assert _marketState.lossFactor reflects badDebt computed WITHOUT largeAmount.
// 7. Assert borrower can withdraw largeAmount - min_collateral from collateral[1].
// 8. Assert lender credit loss > true undercollateralization at tx end.
```

The test proves that `lossFactor` encodes a `badDebt` larger than `D − (C₀·price₀/ORACLE_PRICE_SCALE·WAD/maxLif₀ + largeAmount·price₁/ORACLE_PRICE_SCALE·WAD/maxLif₁)`, and that the attacker recovers the bulk of `largeAmount` in a subsequent `withdrawCollateral` call.
