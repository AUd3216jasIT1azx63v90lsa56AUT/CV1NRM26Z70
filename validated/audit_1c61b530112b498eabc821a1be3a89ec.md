Audit Report

## Title
Unbounded oracle gas consumption in `liquidate()` collateral bitmap loop enables permanent liquidation DoS - (File: src/Midnight.sol)

## Summary
The `liquidate()` function iterates over all activated collateral slots and calls `IOracle.price()` on each with no gas stipend or cap. Because `touchMarket()` is permissionless and never validates oracle addresses, a malicious borrower can register 16 attacker-controlled oracles that each consume near-block-gas-limit gas per call, making any liquidation of their position revert with out-of-gas. This permanently blocks liquidation of an undercollateralized position, causing unrecoverable bad debt for lenders.

## Finding Description

**Exact code path — `liquidate()` at `src/Midnight.sol:607-618`:**

```solidity
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price(); // unbounded external call
    ...
    _collateralBitmap = _collateralBitmap.clearBit(i);
}
``` [1](#0-0) 

Each `price()` call is an unconstrained external call. With `MAX_COLLATERALS_PER_BORROWER = 16`, an attacker can force 16 × ~1.9M gas ≈ 30M gas per liquidation attempt, exceeding the block gas limit. [2](#0-1) 

**`touchMarket()` validates tokens, LLTV, and `maxLif` — but never oracle addresses:** [3](#0-2) 

**`IOracle` imposes no gas restriction:** [4](#0-3) 

**Oracle is never called in `supplyCollateral()`**, so gas-consuming oracles are invisible at collateral activation time. The protocol's own comments acknowledge this: [5](#0-4) 

**Exploit flow:**
1. Attacker deploys 16 oracle contracts whose `price()` each consume ~1.9M gas (tight SLOAD loop) but return a valid non-zero price.
2. Attacker calls `touchMarket()` with a `Market` struct referencing these 16 oracles. All other parameters are valid (sorted tokens, allowed LLTV, correct `maxLif`).
3. Attacker calls `supplyCollateral()` 16 times, activating all 16 bits in `collateralBitmap`. Oracle is never called here.
4. Attacker borrows (self-lends from a second address to fund the market).
5. Attacker's oracles return a low price, making the position unhealthy.
6. Any liquidator calling `liquidate()` triggers the while loop: 16 × ~1.9M gas → OOG revert.

**Why existing checks fail:**
- `MAX_COLLATERALS_PER_BORROWER = 16` bounds iteration count, not gas per call.
- The `liquidatorGate` check at lines 597–600 runs before the loop and cannot prevent OOG.
- The LIVENESS section at lines 142–158 acknowledges reverting oracles block liquidation as a known caveat, but does not address deliberate gas-exhaustion attacks as an exploit vector — the documented caveat covers accidental oracle failure, not attacker-controlled permanent DoS. [6](#0-5) 

## Impact Explanation
An undercollateralized borrower position with 16 gas-consuming oracles activated cannot be liquidated by any liquidator regardless of gas price or block gas limit. The core protocol invariant — "unhealthy positions remain liquidatable" — is violated. Bad debt accrues permanently and is never realized, directly harming all lenders in the market through the `lossFactor` mechanism. This constitutes a permanent freeze of the liquidation path and unrecoverable loss for lenders.

## Likelihood Explanation
**Preconditions:**
- Deploy 16 gas-consuming oracle contracts (cheap, one-time).
- Create a market with these oracles (permissionless, no admin approval).
- Deploy 16 minimal ERC-20 tokens at chosen addresses to satisfy the sort-order requirement.
- Activate all 16 collateral slots via `supplyCollateral()`.
- Have debt in the market (self-lend from a second address).

**Feasibility:** Moderate. The main practical cost is deploying 16 ERC-20 tokens and 16 oracle contracts. All steps are permissionless and fully attacker-controlled. The DoS is indefinite — the borrower has no incentive to voluntarily withdraw collateral. The attack is repeatable across any number of markets.

## Recommendation
1. **Gas-cap oracle calls:** Use a gas-limited low-level call when invoking `IOracle.price()`, e.g., `(bool ok, bytes memory data) = oracle.call{gas: GAS_STIPEND}(abi.encodeCall(IOracle.price, ()));` with a fixed `GAS_STIPEND` (e.g., 100k gas). Treat a failed or out-of-gas call as a revert with a descriptive error.
2. **Oracle validation at market creation:** In `touchMarket()`, optionally call each oracle's `price()` with a gas cap to verify it responds within budget before the market is registered.
3. **Document the gas cap as a market parameter:** Allow market creators to specify a per-oracle gas budget, enforced at liquidation time.

## Proof of Concept
**Minimal Foundry test plan:**
1. Deploy a `GasGrinder` contract: `function price() external view returns (uint256) { uint256 x; for (uint256 i; i < 60_000; i++) { assembly { x := sload(i) } } return 1e36; }` — consumes ~1.9M gas per call.
2. Deploy 16 instances of `GasGrinder` and 16 minimal ERC-20 tokens with addresses sorted ascending.
3. Construct a `Market` with these 16 `CollateralParams` (valid LLTV, valid `maxLif`, sorted tokens, each oracle = a `GasGrinder`).
4. Call `touchMarket(market)`.
5. Call `supplyCollateral()` 16 times (one per collateral index) with non-zero amounts.
6. Self-lend from a second address; call `take()` to create debt.
7. Manipulate oracle return values to make position unhealthy (or simply use `price() = 1` from the start).
8. Call `liquidate()` with `gasleft()` logged before and after — assert it reverts with OOG or that gas consumed exceeds 30M.
