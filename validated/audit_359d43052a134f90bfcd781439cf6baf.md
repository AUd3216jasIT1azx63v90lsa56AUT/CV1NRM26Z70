Audit Report

## Title
Authorized-Signer Front-Run Advances Victim's Nonce and Invalidates Pending Signed Authorization - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` permits any address that `IMidnight.isAuthorized(authorizer, signer)` returns `true` for to sign and submit an `Authorization` struct with attacker-chosen `authorized` and `isAuthorized` values at the authorizer's current nonce. A delegated signer (attacker) can front-run the victim's pending transaction, consume nonce N with a different struct, permanently invalidate the victim's signed authorization, and simultaneously set arbitrary `isAuthorized` entries on behalf of the victim.

## Finding Description
**Root cause:**

The nonce in `EcrecoverAuthorizer` is keyed solely on `authorization.authorizer` and increments on any valid call from any authorized signer. [1](#0-0) 

The signer check allows either the authorizer or any address for which `IMidnight.isAuthorized(authorization.authorizer, signer)` is `true`. [2](#0-1) 

The `Authorization` struct contains five independently attacker-controllable fields: `authorizer`, `authorized`, `isAuthorized`, `nonce`, `deadline`. [3](#0-2) 

There is no binding between the signer's identity and the `authorized` field. A delegated signer can freely choose any values for `authorized` and `isAuthorized` â€” there is no check that the struct content was pre-approved by the authorizer.

**Exploit flow:**

1. **Precondition:** `IMidnight.isAuthorized(victim, attacker) == true` (victim previously authorized attacker as an operator/relayer â€” a routine, expected state).
2. Victim signs `Authorization{authorizer=victim, authorized=X, isAuthorized=true, nonce=N}` and broadcasts.
3. Attacker observes victim's tx in mempool; crafts `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N}` (same nonce, different `authorized`).
4. Attacker signs this struct with their own private key â†’ `ecrecover(digest, ...) == attacker`.
5. Attacker submits with higher gas priority.
6. Contract execution:
   - Line 25: deadline check passes (attacker sets any valid future deadline).
   - Line 26: `N == nonce[victim]` â†’ passes; nonce increments to N+1.
   - Line 31: `signer = attacker`.
   - Lines 33â€“36: `IMidnight.isAuthorized(victim, attacker) == true` â†’ passes.
   - Line 47: `IMidnight.setIsAuthorized(attacker, true, victim)` executes â€” attacker self-authorizes or sets arbitrary entries.
7. Victim's tx arrives: `N != N+1` â†’ reverts with `InvalidNonce()`. [4](#0-3) 

**Why existing checks fail:**

- The nonce check prevents replay of the *same* signed message but does not prevent a *different* message at the same nonce from a delegated signer.
- The `isAuthorized` delegation path does not scope what struct content the agent may sign â€” the agent can sign any `authorized`/`isAuthorized` combination.
- There is no binding between the signer's identity and the `authorized` field in the struct.

## Impact Explanation
Two concrete impacts:

1. **DoS of delegation (permanent, repeatable):** The victim's pending signed `Authorization` is permanently invalidated. The victim must sign a new message at nonce N+1, which the attacker can front-run again indefinitely. This constitutes permanent service degradation for the victim's signature-gated delegation path.

2. **Unauthorized state mutation / privilege escalation:** The attacker can set arbitrary `isAuthorized` entries on behalf of the victim â€” authorizing malicious addresses or revoking existing authorizations. This maps directly to unauthorized state changes and privilege escalation from authorized-agent to full authorization-manager, both of which are in-scope impact classes per `RESEARCHER.md`. [5](#0-4) 

## Likelihood Explanation
The precondition (`isAuthorized[victim][attacker] == true`) is a common, expected state â€” users routinely authorize operators, bots, and relayers via `IMidnight.setIsAuthorized`. [6](#0-5) 

Once met, the attack requires only mempool monitoring and a higher-gas competing transaction, which is standard MEV infrastructure. The attack is trivially repeatable on every future signed authorization the victim broadcasts, making it a persistent griefing vector with no natural expiry.

## Recommendation
**Option 1 (preferred):** Require that the signer is the `authorization.authorizer` themselves â€” remove the delegated signer path from `EcrecoverAuthorizer` entirely. Delegation is already handled by `IMidnight.setIsAuthorized` directly; the periphery contract need not re-expose it.

**Option 2:** If delegated signing must be supported, bind the signer's identity to the `authorized` field: require `signer == authorization.authorized` when `signer != authorization.authorizer`. This prevents a delegated signer from choosing an arbitrary `authorized` address.

**Option 3:** Use a per-(authorizer, signer) nonce instead of a per-authorizer nonce, so a delegated signer can only consume their own nonce slot and cannot invalidate the authorizer's own pending transactions.
