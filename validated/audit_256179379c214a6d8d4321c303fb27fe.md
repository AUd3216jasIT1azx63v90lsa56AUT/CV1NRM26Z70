Audit Report

## Title
Off-by-One in Maturity Guard Permits Debt Increase at Exact Maturity Timestamp - (File: `src/Midnight.sol`)

## Summary
The guard at `src/Midnight.sol:391` uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` when `block.timestamp == market.maturity`, allowing `sellerDebtIncrease > 0` at that exact timestamp. This directly violates two explicit protocol invariants that name timestamp equality as a forbidden path for debt increase. The fee calculation simultaneously treats this timestamp as post-maturity (`timeToMaturity = 0`), creating an internal inconsistency where debt is increased with zero fee accrual and the resulting position is immediately liquidatable in the next block.

## Finding Description

**Root cause — `<=` instead of `<` at line 391:**

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

When `block.timestamp == market.maturity`, the left side is `true`, so the `require` passes unconditionally regardless of `sellerDebtIncrease`. The correct boundary is `block.timestamp < offer.market.maturity`.

**Fee calculation already treats exact maturity as post-maturity:**

```solidity
uint256 timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp);
```

At exact maturity, `zeroFloorSub(maturity, maturity) = 0`, so `timeToMaturity = 0` and `buyerPendingFeeIncrease = 0`. The fee path and the debt guard are inconsistent at this boundary.

**Exploit flow:**
1. A sell offer exists (`offer.buy == false`, maker = borrower) with `offer.expiry >= market.maturity`.
2. The seller has zero credit (`sellerPos.credit == 0`), so `sellerCreditDecrease = 0` and `sellerDebtIncrease = units`.
3. Taker waits for `block.timestamp == market.maturity`.
4. Taker calls `take(sellOffer, ..., units, taker, ...)`.
5. Line 353: `block.timestamp <= offer.expiry` passes (expiry ≥ maturity).
6. Line 391: `maturity <= maturity` → `true` → passes.
7. Line 414: `sellerPos.debt += units` — debt is increased at maturity with zero fee accrual.

**Why existing checks do not stop it:**
- The expiry check (line 353) passes if `expiry >= maturity`.
- The maturity guard (line 391) uses `<=` not `<`.
- Post-maturity liquidation (line 622) requires `block.timestamp > market.maturity` (strict `>`), so the newly created debt cannot be liquidated in post-maturity mode in the same block, leaving the borrower with no recourse until the next block.

## Impact Explanation
A borrower's debt is increased at `block.timestamp == market.maturity` with zero fee accrual, violating the core credit/debt accounting invariant (`"debt must not increase after maturity"`) and the maturity boundary invariant (`"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"`). The position is immediately subject to post-maturity liquidation in the next block (`block.timestamp == maturity + 1`) with no recourse for the borrower. This constitutes credit/debt accounting corruption and enables unauthorized collateral seizure — both explicitly listed in-scope impact classes.

## Likelihood Explanation
Any block whose `block.timestamp` equals a market's maturity is sufficient. On PoS chains, validators can influence `block.timestamp` within a small window; a taker can also simply submit the transaction and have it land in the correct block. The precondition — a sell offer with `expiry >= maturity` — is normal and expected protocol behavior. The attack is repeatable for every market whose maturity falls on a block timestamp and requires no special privileges beyond being an unprivileged taker.

## Recommendation
Change the comparison at line 391 from `<=` to `<`:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (correct):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the debt guard with the fee calculation, which already treats `block.timestamp == maturity` as post-maturity (`timeToMaturity = 0`), and with the liquidation check, which uses strict `>` for post-maturity mode.
