Audit Report

## Title
Authorized Operator Can Transitively Grant Persistent Backdoor Authorization, Surviving Victim's Revocation - (File: src/Midnight.sol)

## Summary
`setIsAuthorized` permits any currently-authorized operator to write new entries into the victim's `isAuthorized` mapping, including entries the victim never granted. When the victim revokes the original operator, any backdoor entries planted by that operator remain active, giving an attacker permanent, unrevoked access to all victim-gated operations including `withdrawCollateral` and `withdraw`.

## Finding Description
**Root cause:** `setIsAuthorized` at line 731–735 checks only `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` before writing `isAuthorized[onBehalf][authorized] = newIsAuthorized`. There is no restriction preventing an authorized operator from writing additional entries into the victim's authorization mapping on behalf of the victim.

**Exploit flow:**

1. Victim calls `setIsAuthorized(attacker, true, victim)` for a limited purpose. State: `isAuthorized[victim][attacker] = true`.
2. Attacker calls `setIsAuthorized(attacker2, true, victim)`. The check at line 732 passes because `isAuthorized[victim][attacker] == true`. State: `isAuthorized[victim][attacker2] = true`.
3. Victim calls `setIsAuthorized(attacker, false, victim)` to revoke. State: `isAuthorized[victim][attacker] = false`. `isAuthorized[victim][attacker2]` is **unchanged**.
4. `attacker2` calls `withdrawCollateral(market, collateralIndex, assets, victim, attacker2)`. Line 556 check: `isAuthorized[victim][attacker2] == true` → passes. Victim's collateral is transferred to `attacker2`.
5. `attacker2` calls `withdraw(market, units, victim, attacker2)`. Same check passes. Victim's credit is drained.

**Why existing checks fail:** The Certora rule `onlyAuthorizedCanChangeIsAuthorized` (spec line 102–110) only asserts the caller was authorized at call time — it does not constrain the depth of delegation. The isolation rule `setIsAuthorizedIsolation` (spec line 125–130) only verifies that unrelated `(user, authorized)` pairs are unaffected, which is true; the attack writes to `(victim, attacker2)` which is a new pair. No on-chain check prevents this. The test `testSetIsAuthorizedAuthorization` (lines 290–304) explicitly asserts this behavior succeeds, confirming it is unguarded.

## Impact Explanation
`attacker2` gains permanent `isAuthorized[victim][attacker2] = true` that survives victim's revocation of `attacker`. `attacker2` can call `withdrawCollateral` to seize all of victim's collateral tokens and `withdraw` to drain all of victim's redeemable credit units. This is direct, unauthorized theft of user assets with no further preconditions once the backdoor is planted.

## Likelihood Explanation
Preconditions: victim must authorize `attacker` at least once — a routine action in DeFi for wallet integrations, periphery contracts, or delegated position management. Attacker needs only one transaction during the authorization window to plant `attacker2`. The victim has no on-chain mechanism to detect the backdoor without enumerating all `SetIsAuthorized` events. The attack is repeatable: attacker can plant multiple backdoor addresses before being revoked.

## Recommendation
Restrict `setIsAuthorized` so that only `onBehalf == msg.sender` (the account itself) can modify its own authorization mapping. Authorized operators should not be permitted to sub-delegate. Change line 732 from:

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
```

to:

```solidity
require(onBehalf == msg.sender, Unauthorized());
```

If delegated authorization management is a required feature, introduce a separate, explicitly scoped permission bit (e.g., `canDelegate`) that must be separately granted and is not implied by standard authorization.

## Proof of Concept
The existing test `testSetIsAuthorizedAuthorization` in `test/AuthorizationTest.sol` at lines 290–304 already demonstrates the full backdoor-planting step (lines 300–303) and asserts it succeeds. A complete PoC extends this test:

1. After line 303, have `attacker2` call `withdrawCollateral` with `onBehalf = user`.
2. Assert the call succeeds and victim's collateral balance decreases.
3. Have victim call `setIsAuthorized(authorized, false, user)` (revoke `attacker`).
4. Assert `attacker2` can still call `withdrawCollateral` successfully.
