### Title
Recovery Address Not Reset on Ownership Transfer Allows Previous Sponsor to Retain CORRUPTED Sweep Destination - (File: src/ConfidencePool.sol)

### Summary
`ConfidencePool` inherits `Ownable2Step`, enabling two-step ownership transfer. However, `recoveryAddress` — the permissionless sweep destination for all CORRUPTED-path funds — is never cleared or updated during the transfer. A previous sponsor who set `recoveryAddress` to their own address at pool creation retains that sweep destination after relinquishing ownership, allowing the entire pool balance (staker principal + bonus) to be swept to them on any CORRUPTED resolution.

### Finding Description
`ConfidencePool` inherits `Ownable2Step`, exposing `transferOwnership` and `acceptOwnership` for pool ownership handoff. [1](#0-0) 

`recoveryAddress` is set at initialization to the sponsor-supplied value and is subsequently mutable only by the current owner via `setRecoveryAddress`. [2](#0-1) [3](#0-2) 

Neither `transferOwnership` nor `acceptOwnership` (inherited from OpenZeppelin's `Ownable2Step`) resets or updates `recoveryAddress`. After the two-step transfer completes, the new owner holds all `onlyOwner` privileges, but `recoveryAddress` still points to whatever address the previous sponsor set — typically their own address.

Three permissionless sweep functions unconditionally send funds to `recoveryAddress`:

- `claimCorrupted()` — sweeps the entire pool balance (staker principal + bonus) to `recoveryAddress` on bad-faith CORRUPTED. [4](#0-3) 

- `sweepUnclaimedCorrupted()` — sweeps remaining balance to `recoveryAddress` after the 180-day attacker claim window. [5](#0-4) 

- `sweepUnclaimedBonus()` — sweeps excess/unclaimed bonus to `recoveryAddress` on SURVIVED/EXPIRED. [6](#0-5) 

Because all three callers are permissionless (no `onlyOwner` or `onlyModerator` guard), any party can trigger the sweep at any time after the outcome is set, sending funds to the stale previous-owner address.

### Impact Explanation
Under bad-faith CORRUPTED resolution, `claimCorrupted` sweeps `stakeToken.balanceOf(address(this))` — the full pool, including all staker principal and all bonus contributions — to `recoveryAddress`. [7](#0-6) 

If `recoveryAddress` still points to the previous sponsor after an ownership transfer, every staker loses their entire principal and bonus to an address the new owner does not control. The new owner has no mechanism to intercept or redirect the sweep once `flagOutcome(CORRUPTED, ...)` has been called and `claimsStarted` is set.

### Likelihood Explanation
The scenario does not require the previous sponsor to be malicious. Setting `recoveryAddress` to one's own address at pool creation is the natural, expected behavior — the sponsor is the intended recovery beneficiary. Ownership transfer is an explicit feature of `Ownable2Step` and is a realistic operational event (e.g., protocol handoff, multisig migration, agreement ownership change). The new owner may not immediately call `setRecoveryAddress` after accepting, especially if they are unaware that the field is not automatically updated. A CORRUPTED resolution is the only additional prerequisite, and the moderator flags it based on registry state, not on the ownership history of the pool.

### Recommendation
Override `_transferOwnership` in `ConfidencePool` to reset `recoveryAddress` to the incoming owner address, or override `acceptOwnership` to require the pending owner to supply a new `recoveryAddress` atomically. At minimum, emit a prominent warning event on transfer noting that `recoveryAddress` was not updated, and document that the new owner must call `setRecoveryAddress` before any CORRUPTED resolution can occur.

### Proof of Concept
```
1. Alice (original sponsor) calls factory.createPool(..., recoveryAddress=Alice, ...)
   → pool.recoveryAddress == Alice

2. Alice calls pool.transferOwnership(Bob)
3. Bob calls pool.acceptOwnership()
   → pool.owner() == Bob, but pool.recoveryAddress == Alice (unchanged)

4. Bob does not immediately call setRecoveryAddress (or is unaware of the stale value).

5. The agreement is breached; moderator calls pool.flagOutcome(CORRUPTED, false, address(0))
   → outcome = CORRUPTED, goodFaith = false

6. Anyone (including Alice) calls pool.claimCorrupted()
   → stakeToken.transfer(Alice, entirePoolBalance)
   → All staker principal + bonus swept to Alice; Bob and stakers receive nothing.
```

The call at step 6 succeeds because `claimCorrupted` has no caller restriction and unconditionally transfers to the stale `recoveryAddress`. [8](#0-7)

```text
    function claimCorrupted() external nonReentrant {
        if (outcome != PoolStates.Outcome.CORRUPTED) revert OutcomeNotSet();
        if (goodFaith && bountyClaimed < bountyEntitlement) revert MustClaimBountyFirst();

        // aderyn-fp-next-line(reentrancy-state-change)
        uint256 toSweep = stakeToken.balanceOf(address(this));
        if (toSweep == 0) revert NothingToSweep();

        // Clamp the decrement — `toSweep` can exceed the original reserve when post-resolution
        // donations have inflated the balance.
        corruptedReserve = toSweep <= corruptedReserve ? corruptedReserve - toSweep : 0;
        if (!goodFaith) {
            bountyClaimed = bountyEntitlement;
        }
        if (!claimsStarted) claimsStarted = true;
        stakeToken.safeTransfer(recoveryAddress, toSweep);

        emit ClaimCorrupted(msg.sender, recoveryAddress, toSweep);
```

```text
    function sweepUnclaimedCorrupted() external nonReentrant {
        if (outcome != PoolStates.Outcome.CORRUPTED) revert OutcomeNotSet();
        if (!goodFaith) revert NotGoodFaithCorrupted();
        if (block.timestamp <= corruptedClaimDeadline) revert ClaimWindowNotExpired();

        // aderyn-fp-next-line(reentrancy-state-change)
        uint256 amount = stakeToken.balanceOf(address(this));
        if (amount == 0) revert NothingToSweep();

        corruptedReserve = 0;
        bountyClaimed = bountyEntitlement;
        if (!claimsStarted) claimsStarted = true;
        stakeToken.safeTransfer(recoveryAddress, amount);
```
