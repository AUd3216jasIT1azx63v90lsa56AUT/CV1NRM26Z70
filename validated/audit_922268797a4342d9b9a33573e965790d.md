Audit Report

## Title
Post-maturity no-op `liquidate(0,0)` repeatable via multicall triggers O(N×16) oracle calls with zero state change - (File: `src/Midnight.sol`)

## Summary
In post-maturity mode, after the first `liquidate(seizedAssets=0, repaidUnits=0)` call partially realizes bad debt, subsequent identical calls pass all guards and each unconditionally execute the full `collateralBitmap` while-loop (up to 16 `IOracle.price()` calls) before returning `(0, 0)` with zero state change. Because `multicall` propagates reverts immediately and calls 2..N never revert, an attacker can compose a multicall of N such calls, causing exactly N×16 oracle calls in a single transaction bounded only by the block gas limit.

## Finding Description

**Root cause — oracle loop runs before the liquidatability guard:**

The while-loop at [1](#0-0)  iterates every bit in `_position.collateralBitmap`, calling `IOracle(_collateralParam.oracle).price()` for each activated collateral (up to `MAX_COLLATERALS_PER_BORROWER = 16`). This loop executes **unconditionally** before the `NotLiquidatable()` guard at [2](#0-1) .

In post-maturity mode the guard reduces to `block.timestamp > market.maturity` — a condition that is permanently and trivially true for every matured market, regardless of the relationship between debt and maxDebt.

**Call-by-call trace (post-maturity, 16 collaterals, partial bad debt):**

- **Call 1**: `require(_position.debt > 0)` at [3](#0-2)  passes → 16 oracle calls → `badDebt > 0` → `_position.debt -= badDebt` (partial: debt remains > 0) at [4](#0-3)  → `seizedAssets == 0 && repaidUnits == 0` skips the seize block at [5](#0-4)  → `safeTransfer(..., 0)` and `safeTransferFrom(..., 0)` at [6](#0-5)  both succeed per the TOKEN SAFETY REQUIREMENTS ("It should not revert because a transfer is no-op") → returns `(0, 0)`.

- **Calls 2..N**: `_position.debt > 0` still passes (partial bad debt left) → 16 oracle calls → `badDebt = 0` (remaining debt ≈ floor already realized via mulDivUp rounding) → `NotLiquidatable()` passes (`block.timestamp > market.maturity`) → `if (badDebt > 0)` skipped → `if (repaidUnits > 0 || seizedAssets > 0)` skipped → 0-amount transfers succeed → returns `(0, 0)`. **Zero state change, 16 oracle calls consumed per call.**

**`multicall` propagates reverts immediately with no try/catch:** [7](#0-6) 

Since calls 2..N do not revert, the entire multicall succeeds, and the attacker can pack N entries bounded only by the block gas limit (~30 M gas on Ethereum mainnet).

**Why existing checks fail:**

| Check | Why it does not stop calls 2..N |
|---|---|
| `require(_position.debt > 0)` | Partial bad debt leaves `debt > 0` after call 1 |
| `NotLiquidatable()` post-maturity | Reduces to `block.timestamp > market.maturity`, permanently true |
| `if (badDebt > 0)` | `badDebt == 0` on calls 2..N; block skipped but loop already ran |
| `if (repaidUnits > 0 \|\| seizedAssets > 0)` | Both are 0; block skipped |
| `multicall` revert propagation | Calls 2..N do not revert |

## Impact Explanation
A single multicall transaction with N entries of `liquidate(0, 0, postMaturityMode=true)` against a borrower with 16 activated collaterals and partial bad debt triggers exactly N×16 external `IOracle.price()` calls. This is bounded only by the block gas limit, not by any protocol-level cap. The attacker can saturate a block with oracle computation at the cost of their own gas, constituting concrete gas griefing of the liquidation path and block-space denial-of-service against competing liquidators. In a time-sensitive post-maturity liquidation race, delaying competing liquidators by even one block can cause measurable harm (bad debt accumulation, missed liquidation incentives).

## Likelihood Explanation
**Preconditions:**
1. Market is post-maturity (`block.timestamp > market.maturity`) — inevitable for every market.
2. Borrower has partial bad debt: `debt > sum(collateral_i × price_i / maxLif_i)` but debt not fully zeroed — the common case when collateral prices drop significantly but not to zero.
3. Borrower has up to 16 activated collaterals — the maximum allowed, achievable by any borrower.
4. No `liquidatorGate` (or attacker passes it).

All preconditions are reachable by an unprivileged liquidator with no special access. The attack is repeatable across any post-maturity market with partial bad debt and is not blocked by any nonce, consumed counter, or state transition after call 1.

## Recommendation
Add a guard after the oracle loop and bad-debt computation that reverts when the call would be a complete no-op — i.e., when `badDebt == 0` and both `seizedAssets == 0` and `repaidUnits == 0`. This preserves the intentional "realize bad debt with 0 tokens transferred" use case (call 1 where `badDebt > 0`) while preventing the repeated zero-effect calls:

```solidity
require(badDebt > 0 || repaidUnits > 0 || seizedAssets > 0, NoOpLiquidation());
```

This check should be placed immediately after the while-loop and before the `NotLiquidatable()` guard, or alternatively after the `NotLiquidatable()` guard. Either placement eliminates the repeated oracle-call amplification.

## Proof of Concept
**Minimal fork test plan:**
1. Deploy `Midnight` on a local fork; create a market with 16 collateral params each backed by a mock oracle.
2. Have a borrower supply all 16 collaterals and take on debt such that `debt > sum(collateral_i × price_i / maxLif_i)` (partial bad debt scenario).
3. Advance `block.timestamp` past `market.maturity`.
4. Record oracle call counts via mock oracle event logs or call counters.
5. Call `multicall` with N=100 entries of `liquidate(market, 0, 0, 0, borrower, true, receiver, address(0), "")`.
6. Assert: transaction succeeds, oracle call count equals exactly N×16 (1600), and `_position.debt` is unchanged after call 1's bad-debt realization.
7. Confirm zero state change on calls 2..100 by asserting `_position.debt` is identical before call 2 and after call 100.
