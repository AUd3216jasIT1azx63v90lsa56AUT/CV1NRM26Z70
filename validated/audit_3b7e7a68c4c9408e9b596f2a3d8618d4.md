Audit Report

## Title
Unprivileged `touchMarket` snapshots zero settlement fees when called before `setDefaultSettlementFee` - (`src/Midnight.sol`)

## Summary
`touchMarket` is an unrestricted `public` function that snapshots `defaultSettlementFeeCbp[market.loanToken]` at market creation time with no guard requiring that defaults have been initialized. Any caller can invoke it before the `feeSetter` sets non-zero defaults, permanently writing all seven `settlementFeeCbp` breakpoints as zero for that market. The only remediation path is a privileged `setMarketSettlementFee` call by the `feeSetter`, requiring active monitoring and manual intervention per affected market.

## Finding Description
**Root cause:** `touchMarket` (`src/Midnight.sol:755`) is `public` with no access restriction. The creation branch (`marketState[id].tickSpacing == 0`) reads `defaultSettlementFeeCbp[market.loanToken]` at lines 777–784 and copies all seven values directly into `_marketState.settlementFeeCbp0`–`settlementFeeCbp6`. No guard checks that any of these values are non-zero, nor that `setDefaultSettlementFee` has ever been called for the loan token. The constructor (`src/Midnight.sol:203`) sets no default fees; `defaultSettlementFeeCbp` is a plain mapping that starts at Solidity's zero default for every loan token.

**Exploit flow:**
1. Contract is deployed; `defaultSettlementFeeCbp[loanToken]` is `[0,0,0,0,0,0,0]`.
2. Any address calls `touchMarket(market)` where `market.loanToken` is the target token.
3. `touchMarket` enters the creation branch, reads the all-zero array, and writes `settlementFeeCbp0`–`settlementFeeCbp6 = 0` into `marketState[id]`.
4. `feeSetter` later calls `setDefaultSettlementFee(loanToken, i, nonZeroFee)` for `i = 0..6` — this only updates `defaultSettlementFeeCbp[loanToken]`, not any already-created market's `MarketState`.
5. `settlementFee(id, ttm)` (`src/Midnight.sol:963`) returns 0 for all TTM values on the affected market for its entire lifetime.

**Why existing checks fail:** The guards in `touchMarket` (lines 758–771) validate maturity range, collateral list structure, LLTV, and `maxLif`. None check the state of `defaultSettlementFeeCbp`. The Certora rule `newMarketSettlementFeesMatchDefault` (`certora/specs/SettlementFeeBoundaries.spec:67`) verifies that market fees equal defaults at creation time — this is satisfied even when both are zero, so the formal spec does not catch this ordering issue. `setDefaultSettlementFee` (`src/Midnight.sol:277`) only writes to the default mapping and has no retroactive effect on existing markets. `setMarketSettlementFee` (`src/Midnight.sol:258`) can correct a specific market but is gated to `feeSetter`.

## Impact Explanation
The affected market collects zero settlement fees on every `take` call for its entire lifetime until the `feeSetter` manually invokes `setMarketSettlementFee` for each of the seven breakpoints. This is a concrete, measurable protocol revenue loss. The protocol's own comment at `src/Midnight.sol:41` states "A default settlement fee (per loan token) is set on new markets" — the design intent is that non-zero defaults precede market creation, but this ordering is not enforced. The impact is bounded to fee revenue loss (not user fund loss), and is recoverable only through privileged intervention, making it a medium-severity finding.

## Likelihood Explanation
The precondition — `defaultSettlementFeeCbp[loanToken]` being all-zero — holds for every loan token until `feeSetter` explicitly initializes it. This is the natural state immediately after deployment and for any newly introduced loan token. An attacker can front-run the `feeSetter`'s `setDefaultSettlementFee` transaction, or the condition arises organically if any user creates a market before the `feeSetter` acts. The attack is a single permissionless call requiring no capital, and is repeatable for any loan token whose defaults have not yet been set.

## Recommendation
Add a check in `touchMarket`'s creation branch that at least one of the seven `defaultSettlementFeeCbp[market.loanToken]` values is non-zero before snapshotting, or require that the `feeSetter` has explicitly initialized defaults for the loan token (e.g., via a separate `bool` flag set by `setDefaultSettlementFee`). Alternatively, restrict `touchMarket` to only be callable after defaults are initialized, or allow `touchMarket` to be called by a permissioned role for initial market creation. The `multicall` function already allows atomic batching of `setDefaultSettlementFee` + `touchMarket` in a single transaction, which is the recommended deployment pattern and should be enforced at the contract level.

## Proof of Concept
```solidity
// 1. Deploy Midnight (defaultSettlementFeeCbp[loanToken] == [0,0,0,0,0,0,0])
// 2. Attacker calls touchMarket before feeSetter initializes defaults:
midnight.touchMarket(market); // market.loanToken = targetToken
// 3. feeSetter sets non-zero defaults (too late for market above):
midnight.setDefaultSettlementFee(targetToken, 6, 5000e12); // 50 bps at 360d
// 4. Verify affected market has zero fees:
assert(midnight.settlementFee(id, 360 days) == 0); // passes — protocol earns nothing
// 5. A second market (different maturity) created now gets non-zero fees:
bytes32 id2 = midnight.touchMarket(market2);
assert(midnight.settlementFee(id2, 360 days) == 5000e12); // passes
```
The existing test at `test/SettlementFeeTest.sol:170` contains the comment "Must be set before touchMarket, which snapshots defaultFees at creation time," confirming the team is aware of the ordering dependency but has not enforced it in the contract.
