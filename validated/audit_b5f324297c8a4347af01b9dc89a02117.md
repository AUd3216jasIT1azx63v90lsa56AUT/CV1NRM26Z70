Audit Report

## Title
Repay Eliminates Debt Without Token Receipt When loanToken Returns True on No-Op transferFrom - (File: src/Midnight.sol)

## Summary
The `repay` function in `src/Midnight.sol` permanently commits `position[id][onBehalf].debt -= units` and `marketState[id].withdrawable += units` before invoking `SafeTransferLib.safeTransferFrom`. `SafeTransferLib.safeTransferFrom` validates only the boolean return value of `transferFrom`, not the actual token balance delta. A `loanToken` whose `transferFrom` returns `true` without moving tokens satisfies all three library checks, causing debt to be zeroed and `withdrawable` to be inflated with zero token receipt, directly violating the protocol's core solvency invariant.

## Finding Description
**Exact code path:**

`repay` writes state before the transfer: [1](#0-0) 

Lines 508–509 decrement debt and increment `withdrawable`; line 520 calls `safeTransferFrom` last. If the transfer is a no-op, the state mutations are already committed and cannot be rolled back.

`SafeTransferLib.safeTransferFrom` performs three checks: [2](#0-1) 

1. `token.code.length > 0` — passes for any deployed contract.
2. The low-level call does not revert — passes if `transferFrom` returns without reverting.
3. `returndata.length == 0 || abi.decode(returndata, (bool))` — passes if the decoded bool is `true`.

There is no balance snapshot before the call and no balance verification after it. A token whose `transferFrom` emits no transfer event, moves no tokens, but returns `true` satisfies all three checks.

`touchMarket` performs zero validation on `market.loanToken`: [3](#0-2) 

It validates collateral params, LLTV tiers, and maturity, but applies no code-size check, interface check, or allowlist to `loanToken`. The protocol explicitly supports `"arbitrary loan token"` and `"permissionless market creation"` as design features. [4](#0-3) 

**Exploit flow:**
1. Attacker deploys `FakeLoanToken` — a contract with `code.length > 0` whose `transferFrom` always returns `true` and moves nothing.
2. Attacker calls `touchMarket` with `loanToken = address(FakeLoanToken)` and valid collateral/LLTV params. Market is created.
3. Attacker (as lender) takes a sell offer; `safeTransferFrom` on `FakeLoanToken` passes, credit is recorded, no real tokens enter the contract.
4. Attacker (as borrower) now holds `position[id][borrower].debt = D` units.
5. Attacker calls `repay(market, D, borrower, address(0), "")`:
   - L505: authorization check passes (`onBehalf == msg.sender`).
   - L508: `position[id][borrower].debt` decremented to 0.
   - L509: `marketState[id].withdrawable` incremented by D.
   - L520: `safeTransferFrom(FakeLoanToken, attacker, address(this), D)` — returns `true`, no tokens move.
6. Debt is zero; `withdrawable` is D; the contract's actual `FakeLoanToken` balance is unchanged (still 0).

A more impactful realistic variant: a legitimate token (e.g., one with a pausable `transferFrom` that returns `true` instead of reverting when paused) used in a real market. Any borrower in that market could repay during the pause window, zeroing their debt while inflating `withdrawable` with no backing, directly harming lenders who deposited real tokens.

**Why existing checks fail:**

`safeTransferFrom` is the sole guard on token receipt. It is a return-value check, not a balance-delta check. No function in `repay` inspects `IERC20(loanToken).balanceOf(address(this))` before or after the call. [5](#0-4) 

## Impact Explanation
The borrower's debt is permanently zeroed without any token payment. `marketState[id].withdrawable` is inflated by D units with no backing. Any lender who subsequently calls `withdraw` against that market will either revert (fund freeze) or drain tokens deposited by other legitimate repayments in the same market (direct loss to other lenders). The core invariant — *"contract token balances must cover withdrawable assets"* and *"ERC20 transfer deltas must match accounting deltas"* — is violated. [6](#0-5) [7](#0-6) 

## Likelihood Explanation
Preconditions: (a) a `loanToken` that returns `true` on `transferFrom` without moving tokens, and (b) permissionless market creation (confirmed by design). Condition (a) is reachable by the attacker themselves as market creator deploying a custom token, or by targeting any real token that silently no-ops transfers under certain conditions (paused state, blacklist, etc.) while still returning `true`. The call requires no privilege beyond being the borrower (`onBehalf == msg.sender`). The attack is repeatable across any number of markets and any debt size up to `type(uint128).max`. The `live_context.json` explicitly flags `"false-return"` tokens as a test target that is in scope unless explicitly excluded, and `SECURITY.md` contains no such exclusion. [8](#0-7) 

## Recommendation
Replace the return-value-only check with a balance-delta check in `repay` (and symmetrically in `liquidate` at line 717). Before calling `safeTransferFrom`, snapshot `IERC20(market.loanToken).balanceOf(address(this))`; after the call, assert the balance increased by at least `units`. This is the standard pattern used by protocols that support arbitrary ERC20 tokens. Alternatively, move the `safeTransferFrom` call to before the state mutations (checks-effects-interactions), which would cause a revert before any state is written if the transfer fails silently — though a balance-delta check is more robust against tokens that return `true` on no-ops.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
