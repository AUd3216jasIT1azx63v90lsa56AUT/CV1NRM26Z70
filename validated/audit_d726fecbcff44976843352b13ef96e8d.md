Audit Report

## Title
Missing authorization check on `callback` parameter allows arbitrary third-party payer designation - (File: src/Midnight.sol)

## Summary
`Midnight.liquidate` unconditionally sets `payer = callback` at line 679 with no requirement that `callback == msg.sender` or that `callback` has authorized `msg.sender`. An attacker can designate any deployed flash-liquidation contract that implements `ILiquidateCallback`, returns `CALLBACK_SUCCESS`, and holds a loanToken approval to Midnight as the payer, causing the protocol to pull `repaidUnits` of loanToken from that contract while the attacker's `receiver` collects the seized collateral at zero personal cost.

## Finding Description

**Exact code path:**

`src/Midnight.sol` line 679 sets payer unconditionally to `callback`: [1](#0-0) 

`src/Midnight.sol` lines 696–717 transfer collateral to `receiver`, invoke `callback.onLiquidate(...)`, then pull `repaidUnits` from `payer` (i.e., `callback`): [2](#0-1) 

**Root cause:** No check of the form `require(callback == msg.sender || isAuthorized[callback][msg.sender])` exists anywhere in `liquidate`. The `isAuthorized` mapping is consulted in `setConsumed` (line 724) and `setIsAuthorized` (line 732) but is never consulted in `liquidate` for the `callback` parameter: [3](#0-2) 

The only gate on `callback` is that it must return `CALLBACK_SUCCESS` from `onLiquidate`. The `ILiquidateCallback` interface passes `receiver` as a parameter to `onLiquidate`, but the protocol does not require the callback to verify it received the collateral: [4](#0-3) 

**Exploit flow:**
1. Precondition: `victimContract` has called `loanToken.approve(midnight, N)` and its `onLiquidate` checks only `msg.sender == midnight` (returning `CALLBACK_SUCCESS` without verifying `receiver == address(this)`). This is a realistic pattern for flash-liquidation bots.
2. Attacker calls `midnight.liquidate(market, idx, 0, repaidUnits, borrower, false, attackerEOA, victimContract, data)`.
3. `payer = victimContract` (line 679).
4. `seizedAssets` of collateral transferred to `attackerEOA` (line 696).
5. `victimContract.onLiquidate(attacker, ...)` called — returns `CALLBACK_SUCCESS` because `msg.sender == midnight` (lines 700–714).
6. `safeTransferFrom(loanToken, victimContract, midnight, repaidUnits)` executes (line 717) — pulls from victim.
7. Net: attacker receives collateral worth `repaidUnits × LIF`, pays nothing; victim loses `repaidUnits` of loanToken.

**Why existing checks fail:**
- The `liquidatorGate` check (line 598) gates who can call `liquidate`, not what `callback` is: [5](#0-4) 
- The Certora spec `otherEntryPointsOnlyPullFromCaller` (line 128) sets `liquidateCallbackAllowed = true` unconditionally for any address passed as callback to `liquidate`. It proves only the weaker property "tokens only leave from a callback that returned `CALLBACK_SUCCESS`," not "callback must be `msg.sender` or authorized by `msg.sender`": [6](#0-5) 

## Impact Explanation
Any contract that (a) implements `ILiquidateCallback` returning `CALLBACK_SUCCESS` and (b) holds a loanToken approval to Midnight can have its entire approved balance drained by an unprivileged attacker in a single `liquidate` call. The attacker receives seized collateral at zero personal cost. This directly violates the protocol invariant that liquidation must not transfer assets from unauthorized parties, and constitutes unauthorized token loss for the victim.

## Likelihood Explanation
Preconditions are routinely satisfied in production: (1) unhealthy or post-maturity borrowers exist as a normal market condition; (2) deployed flash-liquidation bots and callback contracts pre-approve Midnight for loanToken to enable the pull after `onLiquidate`, and many check only `msg.sender == midnight` without validating that the collateral receiver is themselves or that the liquidation is profitable for them. The attacker requires no special privilege, no oracle manipulation, and no governance access. The attack is repeatable as long as the victim's approval persists.

## Recommendation
Add an authorization check in `liquidate` when `callback != address(0)`:

```solidity
if (callback != address(0)) {
    require(callback == msg.sender || isAuthorized[callback][msg.sender], Unauthorized());
}
```

This mirrors the pattern already used in `setConsumed` and `setIsAuthorized`, ensuring that only the callback contract itself (or a party it has explicitly authorized) can designate it as the payer.

## Proof of Concept

**Minimal Foundry test plan:**

1. Deploy a `VictimLiquidator` contract that:
   - Implements `ILiquidateCallback.onLiquidate`
   - Returns `CALLBACK_SUCCESS` if `msg.sender == midnight` (no receiver check)
   - Pre-approves Midnight for `N` loanToken units
2. Set up a market with an undercollateralized borrower.
3. From an attacker EOA, call:
   ```solidity
   midnight.liquidate(market, 0, 0, repaidUnits, borrower, false, attackerEOA, address(victimLiquidator), "");
   ```
4. Assert: `attackerEOA` received `seizedAssets` of collateral; `victimLiquidator` lost `repaidUnits` of loanToken; attacker spent zero loanToken.
