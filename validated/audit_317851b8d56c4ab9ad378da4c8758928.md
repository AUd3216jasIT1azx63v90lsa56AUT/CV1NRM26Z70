Audit Report

## Title
Continuous Fee Permanently Unclaimable After Full Lender Withdrawal With Partial Borrower Repayment - (File: src/Midnight.sol)

## Summary
`_updatePosition` accrues fees into `continuousFeeCredit` without increasing `withdrawable`, while `withdraw` decreases `withdrawable` with no floor relative to `continuousFeeCredit`. When a lender fully exits after a borrower has only partially repaid (covering the lender's net credit but not the accrued fee), `withdrawable` drops to zero while `continuousFeeCredit` remains positive, causing every subsequent `claimContinuousFee` call to revert via uint128 underflow. The accrued fee is permanently locked in the contract.

## Finding Description

**Root cause — three independent code facts combine to produce the bug:**

1. `_updatePosition` (`src/Midnight.sol:846`) increases `continuousFeeCredit` by `accruedFee` but leaves `withdrawable` unchanged: [1](#0-0) 

2. `withdraw` (`src/Midnight.sol:494`) decreases `withdrawable` by the full `units` withdrawn, with no lower-bound guard relative to `continuousFeeCredit`: [2](#0-1) 

3. `claimContinuousFee` (`src/Midnight.sol:318-320`) performs three unchecked-underflow subtractions; it reverts if `withdrawable < amount`: [3](#0-2) 

The Certora spec confirms the revert condition: `claimContinuousFee` succeeds iff `amount <= withdrawable && amount <= continuousFeeCredit && amount <= totalUnits`: [4](#0-3) 

**Concrete exploit flow (single lender, single borrower, credit = C, accrued fee = F):**

| Step | Action | `withdrawable` | `continuousFeeCredit` | lender `credit` |
|---|---|---|---|---|
| 0 | Initial | 0 | 0 | 0 |
| 1 | `take(C)` | 0 | 0 | C |
| 2 | `repay(C − F)` | C − F | 0 | C |
| 3 | `updatePosition` | C − F | F | C − F |
| 4 | `withdraw(C − F)` | **0** | F | 0 |
| 5 | `claimContinuousFee(F)` | **REVERT** (0 − F underflows) | F | 0 |

At step 4, `withdraw(C − F)` succeeds because `withdrawable = C − F ≥ C − F`. At step 5, `claimContinuousFee(F)` reverts because `withdrawable = 0 < F`.

**Why existing checks fail:**

No invariant enforces `withdrawable >= continuousFeeCredit`. The `withdraw` function has no lower-bound guard on `withdrawable` relative to `continuousFeeCredit`. The Certora `WithdrawableMonotonicity.spec` only verifies that `withdraw` decreases `withdrawable` by exactly `units`, not that it preserves any relationship with `continuousFeeCredit`: [5](#0-4) 

The existing `testClaimContinuousFee` test always performs a full repay of `credit` (not `credit − fee`) before claiming, which masks the bug: [6](#0-5) 

## Impact Explanation
The accrued continuous fee `F` stored in `continuousFeeCredit` becomes permanently unclaimable: `claimContinuousFee(F)` always reverts because `withdrawable = 0 < F`. No other function increases `withdrawable` without a corresponding new repayment from the borrower. If the borrower has already fully repaid their net obligation (`C − F`), there is no mechanism to recover the locked fee tokens. This constitutes a permanent, irreversible loss of protocol fee revenue for every market with a non-zero continuous fee where this sequence occurs.

## Likelihood Explanation
The required conditions are: (1) a market with a non-zero continuous fee (the default path for any market using `defaultContinuousFee`), (2) time elapsed so that `F > 0`, and (3) the borrower repaying exactly `C − F` or any amount in `[C − F, C)`. Condition 3 is not adversarial — it is the natural repayment amount when a borrower repays their net obligation after fee accrual has reduced the lender's credit. The lender calling `withdraw(credit)` after `updatePosition` is the standard full-exit path. The scenario is repeatable across any market with continuous fees and is reachable by any unprivileged lender.

## Recommendation
In `withdraw` (`src/Midnight.sol`), add a guard after computing the post-withdrawal `withdrawable` to ensure it does not fall below `continuousFeeCredit`:

```solidity
uint128 newWithdrawable = _marketState.withdrawable - UtilsLib.toUint128(units);
require(newWithdrawable >= _marketState.continuousFeeCredit, WithdrawWouldLockFee());
_marketState.withdrawable = newWithdrawable;
```

This enforces the implicit invariant that `withdrawable >= continuousFeeCredit` at all times, guaranteeing the fee claimer can always recover accrued fees up to `continuousFeeCredit`.

## Proof of Concept
Minimal Foundry test (add to `test/ContinuousFeeTest.sol`):

```solidity
function testClaimContinuousFeeLockedAfterPartialRepay() public {
    uint256 credit = 1e18;
    uint256 feeRate = 1e4; // non-zero continuous fee
    uint256 ttm = 365 days;
    setupLender(credit, feeRate, ttm);

    // Advance time so fee accrues
    vm.warp(vm.getBlockTimestamp() + 30 days);
    midnight.updatePosition(market, lender);

    uint256 feeAmount = midnight.continuousFeeCredit(id);
    assert(feeAmount > 0);

    uint256 lenderCredit = midnight.creditOf(id, lender);
    // Borrower repays exactly lender's net credit (not the fee)
    deal(address(loanToken), borrower, lenderCredit);
    vm.prank(borrower);
    midnight.repay(market, lenderCredit, borrower, address(0), hex"");

    // Lender fully withdraws
    vm.prank(lender);
    midnight.withdraw(market, lenderCredit, lender, lender);

    // Fee claimer cannot claim — reverts with underflow
    vm.prank(feeClaimer);
    vm.expectRevert();
    midnight.claimContinuousFee(market, feeAmount, feeClaimer);
}
```
