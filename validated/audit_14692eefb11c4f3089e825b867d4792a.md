Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via `mulDivDown` rounding to zero in `take()` — (File: `src/Midnight.sol`)

## Summary
In `take()`, when processing a buy offer with `maxAssets > 0`, the consumed counter is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For any tick below `MAX_TICK` (5820), `buyerPrice < WAD`, so passing `units = 1` yields `buyerAssets = 0`. The `consumed` counter never advances, the `require(newConsumed <= offer.maxAssets)` guard passes trivially on every call, and the taker accumulates real credit/debt positions at zero token cost while the maker's cap is never enforced.

## Finding Description

**Exact code path** — `src/Midnight.sol` lines 361–369:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;
// For a buy offer: buyerPrice == offerPrice == tickToPrice(offer.tick)
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause**: `tickToPrice` is proven to return values in `[0, WAD]` (`tickToPriceAtMostWad` in `certora/specs/TickToPrice.spec` line 48). The minimum non-zero price is `1e12` (tick 2, confirmed by `testTickToPriceMinMax`). For any tick below `MAX_TICK`, `buyerPrice < WAD`, so `mulDivDown(1, buyerPrice, WAD) = 0`. The consumed counter is incremented by zero, and `0 <= maxAssets` is trivially satisfied.

**Token transfer**: Lines 455–456 transfer `buyerAssets - sellerAssets` and `sellerAssets` respectively — both zero. The taker pays nothing.

**Position updates**: Lines 382–384 and 408–414 use `units` (= 1), not `buyerAssets`. So the maker (buyer) gains 1 unit of credit and the taker (seller) gains 1 unit of debt per call, with no token payment and no consumed counter advance.

**Exploit flow**:
1. Maker posts a buy offer: `offer.buy = true`, `maxAssets = M`, any tick < `MAX_TICK` (i.e., `buyerPrice < WAD`).
2. Taker calls `take(1, ...)` in a loop (or via `multicall`).
3. Each call: `buyerAssets = 0`, `consumed += 0`, check `0 <= M` passes.
4. Maker accumulates 1 unit of credit per call; taker accumulates 1 unit of debt per call; no tokens transferred.
5. `consumed` remains at its pre-attack value indefinitely.

**Why existing checks fail**: The sole guard is `require(newConsumed <= offer.maxAssets)`. Since `newConsumed` is incremented by `0`, it never approaches `maxAssets`. The Certora rule `takeConsumedBoundedByMax` (line 62 of `certora/specs/Consume.spec`) only asserts `consumed <= maxAssets` post-call — it does not verify that the total units delivered are bounded by what `maxAssets` implies, so formal verification does not catch this.

## Impact Explanation
The maker's `maxAssets` cap — the primary mechanism limiting their credit exposure — is rendered completely ineffective for any tick below `MAX_TICK`. A taker can inflate the maker's credit position and their own debt position to an arbitrary size without paying any tokens and without the maker's consent beyond the original offer. This constitutes an unauthorized state change and a critical accounting integrity failure: the protocol's core invariant that an offer cannot be overfilled is violated.

## Likelihood Explanation
Preconditions: `offer.buy == true`, `offer.maxAssets > 0`, and `buyerPrice < WAD`. The third condition holds for every tick except `MAX_TICK` (5820), covering virtually all real offers. The attacker needs no special privileges, no oracle manipulation, and no front-running. Passing `units = 1` is sufficient. The attack is repeatable in a single transaction via `multicall` and requires only a valid taker authorization.

## Recommendation
Enforce a minimum `buyerAssets` (or `sellerAssets`) of at least 1 when `maxAssets > 0`, or require that `units` is large enough that the asset amount is non-zero. Concretely, add a check such as:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetDelta > 0, ZeroAssetDelta());
    newConsumed = consumed[offer.maker][offer.group] += assetDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that every successful `take` in asset-cap mode advances the consumed counter by at least 1, making the cap meaningful.

## Proof of Concept

**Minimal manual steps**:
1. Deploy with any market where `tickToPrice(tick) < WAD` (any tick < 5820).
2. Maker creates a buy offer: `buy = true`, `maxAssets = 100`, tick = 2910 (mid-range, `buyerPrice ≈ 0.5e18`).
3. Taker calls `take(1, ...)` 200 times (or once via `multicall` with 200 identical calls).
4. Assert: `consumed[maker][group] == 0` (never advanced), `position[id][maker].credit >= 200` (far exceeds what `maxAssets = 100` should allow), taker paid 0 tokens total.

**Fuzz/invariant test**: Add an invariant asserting `position[id][maker].credit <= maxAssets / buyerPrice * WAD` after any sequence of `take` calls on a buy offer with `maxAssets > 0`. A fuzzer will immediately find the violation with `units = 1`.
