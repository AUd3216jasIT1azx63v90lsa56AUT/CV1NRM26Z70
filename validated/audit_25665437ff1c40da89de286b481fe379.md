Audit Report

## Title
Co-authorized agent can deauthorize any other authorized agent via `Midnight.setIsAuthorized` and `EcrecoverAuthorizer`, enabling fund freeze - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`Midnight.setIsAuthorized` permits any authorized agent to write any authorization entry on behalf of the user, including setting `isAuthorized=false` for a different authorized agent. `EcrecoverAuthorizer.setIsAuthorized` amplifies this by accepting a signature from any authorized agent — not only the authorizer — allowing an attacker who is a co-authorized agent to sign and submit a deauthorization struct targeting the victim's withdraw agent. The result is that the victim's designated withdrawal operator is silently removed, and any subsequent `withdraw` call by that operator reverts with `Unauthorized`.

## Finding Description

**Root cause — `Midnight.setIsAuthorized` (lines 731–735):** [1](#0-0) 

The authorization check at line 732 is:
```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
```
Any address in `isAuthorized[onBehalf]` satisfies this check. There is no restriction on the direction of the change (`newIsAuthorized=false`) or on which `authorized` address is targeted. An authorized agent can therefore revoke any other agent's authorization on behalf of the user.

**Amplification — `EcrecoverAuthorizer.setIsAuthorized` (lines 33–36):** [2](#0-1) 

The signer check is:
```solidity
signer == authorization.authorizer
    || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)
```
Any co-authorized agent's key satisfies this check. The function then calls: [3](#0-2) 

**Exploit flow (EcrecoverAuthorizer path):**

Preconditions (all set by victim, all normal usage):
- `isAuthorized[victim][EcrecoverAuthorizer] = true`
- `isAuthorized[victim][withdrawAgent] = true`
- `isAuthorized[victim][attacker] = true`

Steps:
1. Attacker reads `ecrecoverAuthorizer.nonce(victim)` → `N`.
2. Attacker constructs `Authorization{authorizer=victim, authorized=withdrawAgent, isAuthorized=false, nonce=N, deadline=...}`.
3. Attacker signs with their own private key.
4. Attacker calls `ecrecoverAuthorizer.setIsAuthorized(auth, sig)`.
5. `ecrecover` returns attacker's address; `isAuthorized[victim][attacker] == true` passes.
6. `EcrecoverAuthorizer` calls `Midnight.setIsAuthorized(withdrawAgent, false, victim)`; `isAuthorized[victim][EcrecoverAuthorizer] == true` passes.
7. `isAuthorized[victim][withdrawAgent]` is set to `false`.
8. Any call by `withdrawAgent` to `Midnight.withdraw(..., victim, ...)` reverts with `Unauthorized`.

**Simpler direct path:** The attacker can skip `EcrecoverAuthorizer` entirely and call `Midnight.setIsAuthorized(withdrawAgent, false, victim)` directly, since `isAuthorized[victim][attacker] == true` already satisfies line 732.

**Existing test confirms the behavior:** [4](#0-3) 

`testSetIsAuthorizedAuthorization` explicitly confirms that an authorized agent can call `setIsAuthorized` for any `newAuthorized` address on behalf of the user. The test only covers the `true` (grant) case, leaving the `false` (revoke) case untested and unguarded.

## Impact Explanation
A lender relying on an authorized agent (e.g., a smart contract wallet, keeper, or automation bot) to call `Midnight.withdraw` on their behalf can have that agent silently deauthorized by any other co-authorized agent. If the lender is a smart contract that cannot call Midnight directly, their credit is permanently frozen in the market. The attacker incurs only gas cost and can repeat the attack each time the victim re-authorizes the withdraw agent. This constitutes a permanent, repeatable fund freeze triggered by an unprivileged user action.

## Likelihood Explanation
The precondition — attacker is one of the victim's authorized agents — is realistic in multi-agent setups (e.g., a keeper bot plus a backup agent, or a protocol integration plus a user-controlled agent). The attacker position can be obtained legitimately or through social engineering. No special timing, oracle conditions, or privileged keys are required. The direct path (calling `Midnight.setIsAuthorized` directly) requires no signature infrastructure at all.

## Recommendation
Restrict `Midnight.setIsAuthorized` so that only `onBehalf == msg.sender` can modify authorization entries — remove the `isAuthorized[onBehalf][msg.sender]` branch from the check on line 732. Authorized agents should be able to act *as* the user for operational functions (withdraw, repay, etc.) but should not be able to modify the user's authorization registry. Correspondingly, `EcrecoverAuthorizer.setIsAuthorized` should only accept a signature from `authorization.authorizer` directly (remove the `isAuthorized` branch at line 34), since the purpose of the contract is to allow off-chain key holders to authorize agents — not to allow agents to re-delegate or revoke on behalf of the user.

## Proof of Concept
Minimal Foundry test (extend `AuthorizationTest`):

```solidity
function testCoAuthorizedAgentDeauthorizesWithdrawAgent() public {
    address victim = makeAddr("victim");
    address withdrawAgent = makeAddr("withdrawAgent");
    address attacker = makeAddr("attacker");

    // Victim sets up normal multi-agent configuration
    vm.startPrank(victim);
    midnight.setIsAuthorized(withdrawAgent, true, victim);
    midnight.setIsAuthorized(attacker, true, victim);
    vm.stopPrank();

    assertEq(midnight.isAuthorized(victim, withdrawAgent), true);

    // Attacker (co-authorized) directly deauthorizes withdrawAgent
    vm.prank(attacker);
    midnight.setIsAuthorized(withdrawAgent, false, victim);

    assertEq(midnight.isAuthorized(victim, withdrawAgent), false);

    // withdrawAgent can no longer withdraw on behalf of victim
    vm.prank(withdrawAgent);
    vm.expectRevert(IMidnight.Unauthorized.selector);
    midnight.withdraw(market, 1, victim, withdrawAgent);
}
```

Expected result: test passes, confirming the deauthorization succeeds and the subsequent withdraw reverts.
