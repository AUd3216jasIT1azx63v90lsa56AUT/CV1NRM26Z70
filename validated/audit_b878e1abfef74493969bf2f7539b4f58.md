Audit Report

## Title
Zero-asset rounding bypass allows unlimited takes on fully-consumed buy offers - (File: src/Midnight.sol)

## Summary
When a buy offer has `maxAssets > 0` and its tick yields `buyerPrice < WAD`, calling `take` with `units = 1` causes `mulDivDown(1, buyerPrice, WAD)` to truncate to zero. Because `consumed` is incremented by this zero value, the guard `newConsumed <= offer.maxAssets` never triggers even when the offer is already fully consumed. An unprivileged taker can repeat this indefinitely, permanently inflating `credit`, `debt`, and `totalUnits` without transferring any tokens.

## Finding Description

**Exact code path:**

`buyerAssets` is computed with truncating division: [1](#0-0) 

`consumed` is then incremented by `buyerAssets` (not `units`), and the cap check uses `<=`: [2](#0-1) 

**Root cause:** When `buyerPrice < WAD` (which holds for any tick above `MAX_TICK / 2 = 2910`, including the normal range up to `MAX_TICK = 5820`) and `units = 1`, `mulDivDown(1, buyerPrice, WAD) = 0`. The increment to `consumed` is zero, so `newConsumed = consumed + 0 = consumed`. The guard `require(newConsumed <= offer.maxAssets)` passes even when `consumed == maxAssets`. [3](#0-2) 

**Position state still mutates** because `buyerCreditIncrease`, `sellerDebtIncrease`, and `totalUnits` are all computed from `units` (not `buyerAssets`): [4](#0-3) [5](#0-4) 

**Token transfer is zero** because `safeTransferFrom` is called with `buyerAssets - sellerAssets = 0 - 0 = 0`: [6](#0-5) 

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = M`, `tick = MAX_TICK - 16` (so `buyerPrice < WAD`), `maxUnits = 0`.
2. Offer is fully consumed through normal fills until `consumed[maker][group] == M`.
3. Taker calls `take(offer, ..., units=1)`. `buyerAssets = 0`. `newConsumed = M + 0 = M ≤ M` → passes. `credit`, `debt`, `totalUnits` increase. Zero tokens transferred.
4. Step 3 repeats without bound.

**Why existing checks fail:** The only guard against over-consumption is `require(newConsumed <= offer.maxAssets)`. There is no check requiring `units > 0` to produce a non-zero `buyerAssets`, no strict `consumed < maxAssets` pre-check, and no revert path when `buyerAssets == 0` but `units > 0`.

## Impact Explanation
Each zero-asset take permanently inflates the buyer's (maker's) `credit`, the seller's (taker's) `debt`, and `totalUnits` without any token movement. This corrupts the core accounting invariants of the protocol: credit/debt ledgers become permanently inflated, `totalUnits` diverges from actual economic exposure, and the offer replay protection (`maxAssets` cap) is bypassed entirely. This constitutes unauthorized state mutation and accounting integrity failure — a concrete in-scope impact per `RESEARCHER.md`. [7](#0-6) 

## Likelihood Explanation
All three preconditions are reachable by any unprivileged user in normal protocol operation: (1) buy offers with `maxAssets > 0` are standard; (2) ticks above `MAX_TICK / 2` (yielding `buyerPrice < WAD`) cover the entire upper half of the valid tick range; (3) full consumption of an offer happens in ordinary use. No admin access, oracle manipulation, or special permissions are required. The attack is repeatable without limit once the offer is fully consumed. [8](#0-7) 

## Recommendation
Add a guard that reverts when `buyerAssets == 0` but `units > 0` in the `maxAssets` branch, or alternatively track consumption in `units` rather than assets. A minimal fix:

```solidity
if (offer.maxAssets > 0) {
    require(buyerAssets > 0, ZeroAssets()); // <-- add this
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any `take` call that would produce zero assets is rejected, preventing the rounding bypass entirely. [9](#0-8) 

## Proof of Concept
The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` directly reproduces the scenario: [10](#0-9) 

It sets `maxAssets = 1`, `tick = MAX_TICK - 16`, pre-fills `consumed` to `maxAssets` via `setConsumed`, then calls `take(1, borrower, lenderOffer)`. The assertions confirm: `buyerAssets == 0`, token balances unchanged, `consumed` stays at `maxAssets`, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase — proving the bypass and the accounting corruption.
