Audit Report

## Title
`SetterRatifier` Lacks Chain Binding, Enabling Cross-Fork Offer Replay - (File: src/ratifiers/SetterRatifier.sol)

## Summary
`SetterRatifier` stores ratification state as `mapping(address maker => mapping(bytes32 root => bool)) public isRootRatified` with no chain discriminator. After a hard fork that changes `block.chainid`, storage is copied verbatim to the forked chain. Because neither `SetterRatifier.isRatified` nor `HashLib.hashOffer` includes any chain-specific value, and because `Midnight.toId` uses the immutable `INITIAL_CHAIN_ID` (intentionally stable across forks), a taker can fill a maker's offers on the forked chain using ratification state the maker set only on the original chain, modifying the maker's position without post-fork consent.

## Finding Description

**Root cause â€” `SetterRatifier` has no chain binding:**

`setIsRootRatified` writes to a storage key of `(maker, root)` only: [1](#0-0) 

`isRatified` checks only that same key, with no `block.chainid` or `INITIAL_CHAIN_ID` guard: [2](#0-1) 

**`HashLib.hashOffer` is chain-agnostic:**
The offer hash covers market fields, maker, expiry, tick, ratifier, etc. but never `block.chainid` or any chain-specific value: [3](#0-2) 

**`Midnight.toId` uses `INITIAL_CHAIN_ID` (immutable), not `block.chainid`:**
The protocol explicitly and intentionally uses `INITIAL_CHAIN_ID` to keep market IDs stable across hard forks: [4](#0-3) [5](#0-4) [6](#0-5) 

After a fork, `block.chainid` changes but `INITIAL_CHAIN_ID` does not, so `toId` returns the **same** market ID on both chains, meaning `touchMarket` finds the already-existing market and proceeds directly to the ratifier check.

**Contrast with `EcrecoverRatifier`**, which explicitly binds to the current chain via `block.chainid` in the EIP-712 domain separator: [7](#0-6) [8](#0-7) 

Its NatSpec explicitly states that a `block.chainid` change invalidates previously signed offers. `SetterRatifier` has no equivalent protection, creating an unintended asymmetry.

**Full exploit path:**
1. Pre-fork: maker calls `setIsRootRatified(maker, root, true)` â†’ `isRootRatified[maker][root] = true`.
2. Hard fork: `block.chainid` changes; all storage is copied; `isRootRatified[maker][root]` remains `true` on the forked chain.
3. Post-fork: taker calls `midnight.take(offer, abi.encode(root, leafIndex, proof), units, taker, ...)` on the forked chain.
4. `touchMarket` resolves the same market ID (via unchanged `INITIAL_CHAIN_ID`).
5. `isAuthorized[offer.maker][offer.ratifier]` â€” also copied from original chain, still `true`.
6. `isRatified` verifies the Merkle proof (chain-agnostic) and checks `isRootRatified[maker][root]` â€” `true`.
7. Take executes: maker's credit/debt position is modified on the forked chain without any post-fork authorization act by the maker.

The `take` function confirms the full call chain: [9](#0-8) 

## Impact Explanation
A taker can fill any `SetterRatifier`-backed offer on the forked chain using ratification state the maker set only on the original chain. The maker's position (credit increased or debt incurred) is modified on the forked chain without post-fork consent. This constitutes unauthorized state change and unauthorized movement of value â€” both in-scope impact classes per RESEARCHER.md. The asymmetry with `EcrecoverRatifier` (which is explicitly protected) demonstrates this is an unintended gap, not a design choice.

## Likelihood Explanation
Preconditions: (1) a contentious hard fork that changes `block.chainid` while keeping contract addresses identical (ETH/ETC-style), and (2) at least one maker using `SetterRatifier` with an active, non-expired offer. Both are realistic â€” hard forks are historical precedent, and the codebase explicitly anticipates the hard fork scenario (NatSpec comment at `Midnight.sol:173-175`, dedicated test `testToIdStableAcrossHardfork` in `test/OtherFunctionsTest.sol`). The attack requires no special privilege, is executable by any taker, and is repeatable for every ratified root not explicitly revoked post-fork.

## Recommendation
Add `block.chainid` as a dimension in the `isRootRatified` mapping:

```solidity
mapping(address maker => mapping(uint256 chainId => mapping(bytes32 root => bool))) public isRootRatified;
```

Update `setIsRootRatified` and `isRatified` to key on `block.chainid`. This mirrors the protection `EcrecoverRatifier` achieves via its EIP-712 domain separator. Alternatively, include `block.chainid` in the `hashOffer` computation in `HashLib`, though this would be a broader change affecting both ratifiers.

## Proof of Concept
1. Deploy `Midnight` and `SetterRatifier` on a local fork at `chainid = 1`.
2. Maker calls `setIsRootRatified(maker, root, true)` and `midnight.setIsAuthorized(address(setterRatifier), true, maker)`.
3. Snapshot the EVM state (simulating fork storage copy).
4. In the snapshot, use `vm.chainId(2)` to simulate the forked chain's new `block.chainid`.
5. Confirm `INITIAL_CHAIN_ID` remains `1` (immutable), so `toId` returns the same market ID.
6. Taker calls `midnight.take(offer, abi.encode(root, 0, new bytes32[](0)), units, taker, ...)`.
7. Observe the call succeeds and the maker's position is modified â€” without any post-fork action by the maker.
