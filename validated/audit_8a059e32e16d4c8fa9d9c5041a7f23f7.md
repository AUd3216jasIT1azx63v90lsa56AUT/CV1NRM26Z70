Audit Report

## Title
Fee-on-Transfer Loan Token Causes Permanent `claimableSettlementFee` Overstatement in `take` - (File: src/Midnight.sol)

## Summary
In `Midnight.take`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` at line 418, before the corresponding `safeTransferFrom` at line 455. When the loan token is fee-on-transfer, the contract receives less than the recorded amount, creating a permanent per-take shortfall. This violates the solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee` and can cause fee-claimer or lender withdrawals to fail.

## Finding Description
**Root cause:** At `src/Midnight.sol:418`, the settlement fee is recorded at full nominal value:

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

The actual token pull occurs later at lines 455–456:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` does not verify the amount actually received by `address(this)`. For a fee-on-transfer token with rate `f`, the contract receives only `(buyerAssets - sellerAssets) * (1 - f)` but records `buyerAssets - sellerAssets`. The shortfall `(buyerAssets - sellerAssets) * f` is permanent and non-reversible per take.

**Exploit flow:** An unprivileged taker creates a market with a fee-on-transfer loan token (markets are permissionlessly created per the README). They call `take` (or batch N takes via `multicall`) against valid offers. Each take independently increments `claimableSettlementFee` by the full nominal spread while the contract receives a fee-discounted amount. After N takes, the total overstatement is `N * (buyerAssets - sellerAssets) * f`.

**Why existing checks fail:** `SafeTransferLib` only checks that the call does not revert; it does not assert `balanceAfter - balanceBefore == amount`. The Certora `tokenBalanceCorrect` invariant is verified under the assumption that `safeTransferFrom` delivers exactly the requested amount — an assumption that fee-on-transfer tokens violate by design. [2](#0-1) 

## Impact Explanation
`claimableSettlementFee[loanToken]` exceeds actual tokens held for fees by `N * (buyerAssets - sellerAssets) * f`. When the fee claimer calls `claimSettlementFee` for the full recorded amount, the transfer either reverts (insufficient balance) or drains tokens backing lender `withdrawable` credits, causing lender withdrawals to fail. The insolvency magnitude scales linearly with the number of takes and the fee rate. This constitutes direct theft/freeze of lender assets — a concrete in-scope impact. [1](#0-0) 

## Likelihood Explanation
**Preconditions:** The loan token must be fee-on-transfer. Markets are permissionlessly created, so any actor can deploy a market with such a token. The taker needs sufficient balance and approval for `buyerAssets` per take, and distinct offer groups to avoid `ConsumedUnits`/`ConsumedAssets` reverts. No privileged access is required — `taker == msg.sender` satisfies the authorization check. The shortfall is permanent and repeatable across transactions, accumulating unbounded overstatement over time. [1](#0-0) 

## Recommendation
Record `claimableSettlementFee` based on the actual received amount rather than the nominal amount. Measure the contract's token balance before and after the `safeTransferFrom` call and use the delta:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived;
```

Alternatively, explicitly document and enforce (via market creation checks) that fee-on-transfer tokens are not supported as loan tokens.

## Proof of Concept
1. Deploy a fee-on-transfer ERC20 token with a 1% transfer fee.
2. Permissionlessly create a Midnight market using this token as `loanToken`.
3. Post a buy offer as maker.
4. As an unprivileged taker, call `take` (or batch N calls via `multicall`) against the offer.
5. After each take, assert: `claimableSettlementFee[loanToken] > IERC20(loanToken).balanceOf(address(midnight))`.
6. Call `claimSettlementFee` for the full recorded amount — observe revert or observe that lender `withdrawable` balances are drained, causing subsequent lender `withdraw` calls to revert.
