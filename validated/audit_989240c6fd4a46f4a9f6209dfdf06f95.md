Audit Report

## Title
Nested `take()` in `onBuy` callback increases seller debt, causing outer `take()` to revert with `SellerIsLiquidatable` - (File: src/Midnight.sol)

## Summary
The `take()` function sets a transient liquidation lock for the seller before invoking buyer callbacks, but the lock-release and health-check sequence at lines 475â€“476 can be exploited via reentrancy. A malicious buyer's `onBuy` callback can call a nested `take()` on a second sell offer from the same seller, increasing the seller's debt. The nested call passes its own health check because the lock is still held, but the outer call's health check fails after the lock is released, reverting with `SellerIsLiquidatable` and undoing all state changes.

## Finding Description
**Root cause â€” lock/health-check ordering:**

At line 444, the outer `take()` sets the liquidation lock and captures the prior value:
```solidity
bool wasLocked = UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true);
```
The `onBuy` callback is then invoked at lines 445â€“453. After both callbacks complete, line 475 conditionally releases the lock:
```solidity
if (!wasLocked) UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, false);
```
and line 476 performs the health check:
```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```

**Exploit path (attacker is maker of a buy offer):**

1. Attacker deploys a contract and posts a buy offer (`offer.buy = true`) with `offer.callback` pointing to the malicious contract.
2. Victim (seller) calls `take()` on the attacker's buy offer â€” victim is `taker`, so `seller = taker = victim`.
3. **Outer take, line 444**: lock set to `true`; `wasLocked = false`.
4. **Outer take, lines 445â€“453**: attacker's `onBuy` is called.
5. **Inside `onBuy`**: attacker calls `take()` on a *second* sell offer where the same victim is `offer.maker` (seller).
   - **Nested take, line 444**: `tExchange` returns `true` (already locked); `wasLocked = true` in nested frame.
   - Nested take executes normally: seller's debt increases (lines 412â€“414).
   - **Nested take, line 475**: `if (!wasLocked)` is `false` â†’ lock is **not** released.
   - **Nested take, line 476**: `liquidationLocked(id, seller)` returns `true` â†’ health check passes even if seller is now unhealthy.
6. **Back in outer take, line 475**: `wasLocked = false` â†’ lock **is** released (`tstore` sets it to `false`).
7. **Outer take, line 476**: `liquidationLocked(id, seller)` returns `false`; `isHealthy` returns `false` (debt was increased by nested take) â†’ **reverts with `SellerIsLiquidatable()`**.

The `tExchange` utility confirms the transient-slot semantics: it atomically reads the old value and writes the new one, so the nested call correctly observes `wasLocked = true` and the outer call correctly observes `wasLocked = false`.

The `setConsumed` mechanism is irrelevant: line 725 enforces `amount >= consumed[onBehalf][group]`, so consumed can never be decreased; the nested-take path alone is sufficient.

## Impact Explanation
Every state mutation â€” including the nested take's debt increase â€” is reverted when the outer take reverts, so no funds are permanently lost. The concrete impact is a **targeted transaction-level DoS**: any seller whose position can be made unhealthy via a second sell offer cannot successfully take the attacker's buy offer. The victim wastes gas and cannot complete the intended trade through that offer. If the attacker posts offers at competitive prices across multiple ticks, they can selectively block a specific seller from trading with them entirely, forcing the seller to use less favorable offers elsewhere.

## Likelihood Explanation
The attacker only needs to: (1) post a buy offer with a malicious callback contract, and (2) ensure the victim has at least one other active sell offer in the same market whose execution would push the victim's health below the threshold. Both conditions are observable on-chain. No privileged access, leaked keys, or victim mistakes are required. The attack is repeatable at negligible cost (gas only, since the outer take reverts and the attacker pays nothing net). Any unprivileged user can execute this.

## Recommendation
Move the health check **before** the callbacks, or snapshot the seller's debt before callbacks and assert it has not increased afterward. Alternatively, record the seller's health state before the `onBuy` callback and use that snapshot at line 476 rather than re-evaluating post-callback. A simpler mitigation is to check `isHealthy` immediately after line 444 (pre-callback) and again at line 476, reverting if the post-callback state is worse than the pre-callback state in a way attributable to debt increase.
