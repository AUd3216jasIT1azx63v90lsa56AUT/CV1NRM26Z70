Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via `mulDivDown` zero-rounding when `buyerPrice < WAD` - (File: src/Midnight.sol)

## Summary
When a buy offer has `maxAssets > 0` and `offer.tick < MAX_TICK` (i.e., `buyerPrice < WAD`), calling `take` with `units = 1` produces `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. The `consumed` accumulator never advances, so the `require(newConsumed <= offer.maxAssets)` cap check is permanently satisfied regardless of how many times `take` is called. The protocol's own comment and a dedicated test both confirm this behavior.

## Finding Description
In `src/Midnight.sol`, `take()`:

```
line 363: buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...
line 368: newConsumed = consumed[offer.maker][offer.group] += buyerAssets;  // += 0
line 369: require(newConsumed <= offer.maxAssets, ConsumedAssets());         // trivially passes
```

`buyerPrice = tickToPrice(offer.tick)`, which is `<= WAD` for all ticks and strictly `< WAD` for any tick below `MAX_TICK`. For `units = 1`, `mulDivDown(1, buyerPrice, WAD) = floor(buyerPrice / WAD) = 0`. The consumed accumulator does not advance, and the cap check is permanently satisfied.

The protocol explicitly acknowledges this at `src/Midnight.sol` line 94: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*

The test `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol` lines 858–889) confirms: a buy offer with `maxAssets = 1` and `tick = MAX_TICK - 16` (price < WAD), already fully consumed (`consumed = 1`), is successfully taken with `units = 1`; `buyerAssets = 0`, `consumed` stays at `1`, yet lender credit and borrower debt both increase.

The Certora rule `takeConsumedBoundedByMax` (`certora/specs/Consume.spec` line 62) asserts `consumed <= maxAssets` post-take but uses `NONDET` summaries for `mulDivDown`/`mulDivUp`, so it does not model this rounding collapse and provides no protection.

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N`, any `tick < MAX_TICK`.
2. Taker calls `take(offer, ..., units=1, ...)` repeatedly (N or more times).
3. Each call: `buyerAssets = 0`, `consumed` unchanged, cap check passes, lender credit +1, borrower debt +1, zero loan tokens transferred.
4. After `k > N` calls, the maker holds `k` units of credit while `consumed` never exceeded `N`; the cap is bypassed.

## Impact Explanation
The `maxAssets` cap on buy offers is rendered ineffective for any `buyerPrice < WAD`, which covers virtually every realistic buy offer. A taker can deliver an unbounded number of units to the maker without the maker paying any loan tokens, violating the core accounting invariant that `consumed[maker][group]` accurately tracks total buyer asset exposure. The maker accrues credit beyond `maxAssets`; the taker accrues matching debt with zero compensation — a griefing vector against the taker and a cap-bypass against the maker's stated intent.

## Likelihood Explanation
Preconditions: a buy offer with `maxAssets > 0` and any tick below `MAX_TICK` — i.e., virtually every realistic buy offer, since `MAX_TICK` corresponds to price = WAD = 1.0, the maximum possible price. No special permissions are required; any address can be the taker. The attack is trivially scriptable and repeatable at the cost of gas only.

## Recommendation
Track consumed units rather than (or in addition to) rounded asset amounts when `buyerPrice < WAD` can cause `mulDivDown` to return zero. One approach: when `offer.maxAssets > 0` and `offer.buy == true`, additionally enforce `newConsumedUnits <= derivedUnitCap` using a units-based accumulator, or require `buyerAssets > 0` before allowing the take to proceed (i.e., revert if `units * buyerPrice < WAD`). Alternatively, enforce a minimum `units` threshold such that `units.mulDivDown(buyerPrice, WAD) >= 1` before accepting the take.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) is a complete, self-contained proof of concept. It:
1. Creates a buy offer with `maxAssets = 1` and `tick = MAX_TICK - 16`.
2. Pre-fills `consumed` to `maxAssets` via `setConsumed`.
3. Calls `take` with `units = 1`.
4. Asserts `buyerAssets == 0`, `consumed` unchanged at `1` (== `maxAssets`), yet lender credit and borrower debt both strictly increased — confirming the cap bypass. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

**File:** test/TakeTest.sol (L858-889)
```text
    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);
        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
```
