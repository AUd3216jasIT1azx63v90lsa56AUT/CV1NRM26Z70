Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via `mulDivDown` rounding to zero, minting unbacked maker credit - (`src/Midnight.sol`)

## Summary
In `take`, when `offer.buy == true` and `offer.maxAssets > 0`, the consumed-offer cap is tracked by adding `buyerAssets` to `consumed[maker][group]`. Because `buyerAssets` is computed with `mulDivDown` and rounds to zero when `units * buyerPrice < WAD`, the consumed counter never advances, the cap check always passes, and the take executes full position state changes — including a `buyerCreditIncrease` of `units` — while transferring zero tokens. The maker accumulates unbacked credit that can be withdrawn, draining tokens belonging to other depositors.

## Finding Description
**Root cause — `src/Midnight.sol`, `take`:**

Line 363 computes `buyerAssets` with rounding-down division:
```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
```
When `units * buyerPrice < WAD` (achievable at any tick where `offerPrice < WAD`, e.g. `MAX_TICK - 16` as used in the existing test), `buyerAssets == 0`.

Lines 367–369 then add this zero to the consumed counter:
```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
// += 0 → newConsumed unchanged → passes even if already at cap
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```
The cap check compares the unchanged `newConsumed` against `offer.maxAssets` and passes unconditionally, regardless of how many times the offer has already been taken or whether it was fully consumed.

Execution continues to line 382:
```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt);
```
When `buyerPos.debt == 0` (normal lender state), `buyerCreditIncrease = units`. Line 410 mints this credit:
```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
```
Lines 455–456 then transfer `buyerAssets - sellerAssets = 0` and `sellerAssets = 0` respectively — zero tokens move.

**Why existing checks fail:**
- The `ConsumedAssets` check at line 369 is the only cap guard; it is defeated by the zero-increment.
- There is no `require(buyerAssets > 0 || units == 0)` guard.
- There is no check that `buyerCreditIncrease == 0` when `buyerAssets == 0`.
- `reduceOnly` does not apply here (it guards against credit increase on the maker side, but the maker is the buyer, and `buyerCreditIncrease > 0` is the normal case for a buy offer).

**Existing test proof — `test/TakeTest.sol`, lines 857–889 (`testBugBuyMaxAssetsBypass`):**
The test pre-consumes the offer to its full `maxAssets = 1` cap, then calls `take(1, borrower, lenderOffer)` and asserts:
- `buyerAssets == 0`, `sellerAssets == 0`
- `consumed` counter unchanged (still equals `maxAssets`)
- `creditOf(id, lender)` strictly increased
- `debtOf(id, borrower)` strictly increased
- Token balances of both parties unchanged

## Impact Explanation
The maker (lender) accumulates credit — a claim on the protocol's loan token pool — without depositing any tokens. Every unit of unbacked credit is a direct shortfall: when the maker later calls `withdraw`, they drain tokens belonging to other depositors. Because the consumed counter never advances, the offer can be replayed an unlimited number of times in a single transaction (via multicall or a loop), inflating unbacked credit to an arbitrary amount. This is a direct, concrete loss to protocol solvency and constitutes theft from other depositors.

## Likelihood Explanation
All preconditions are trivially satisfiable by any unprivileged user:
1. A buy offer with `offer.maxAssets > 0` and a tick where `offerPrice < WAD` (e.g. `MAX_TICK - 16` as shown in the test).
2. `buyerPos.debt == 0` — the normal state for any lender who has not previously borrowed.
3. `units = 1` (or any value where `units * buyerPrice < WAD`).

No special privileges, victim mistakes, or external dependencies are required. The attack is repeatable in a single transaction and is not blocked by any slippage, health, authorization, or callback check.

## Recommendation
Add a guard at the point where `buyerAssets` is computed to reject takes that would result in zero asset movement when `units > 0`:
```solidity
require(units == 0 || (offer.buy ? buyerAssets > 0 : sellerAssets > 0), ZeroAssets());
```
Alternatively, enforce that the consumed counter must strictly increase when `maxAssets > 0`:
```solidity
if (offer.maxAssets > 0) {
    uint256 delta = offer.buy ? buyerAssets : sellerAssets;
    require(delta > 0, ZeroAssets());
    newConsumed = consumed[offer.maker][offer.group] += delta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```
Either fix closes the bypass by ensuring that any take with `units > 0` must move at least one token unit.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` at lines 857–889 is a complete, self-contained proof of concept. It:
1. Sets `maxAssets = 1` and `tick = MAX_TICK - 16` (where `offerPrice < WAD`).
2. Pre-consumes the offer to its full cap via `setConsumed`.
3. Calls `take(1, borrower, lenderOffer)` and asserts zero token transfers, unchanged consumed counter, and strictly increased lender credit and borrower debt.

Running `forge test --match-test testBugBuyMaxAssetsBypass` reproduces the issue.
