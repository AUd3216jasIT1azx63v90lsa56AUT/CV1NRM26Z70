Audit Report

## Title
Oracle Reentrancy into `supplyCollateral` Corrupts Stale Local Bitmap in Liquidation Loop, Causing Excess Bad Debt Socialization - (File: src/Midnight.sol)

## Summary

In `Midnight.liquidate`, the collateral bitmap is snapshotted into a local stack variable at line 606 before the oracle-calling loop begins. Because `IOracle.price()` is an unconstrained external call and `supplyCollateral` has no reentrancy guard, an attacker-deployed oracle can reenter `supplyCollateral` during that call, writing a new collateral bit into storage. The local bitmap never reflects this update, so the loop exits without accounting for the newly added collateral's value. The resulting `badDebt` figure is inflated, and the excess is socialized among lenders via `lossFactor`, while the borrower retains the injected collateral with their debt zeroed.

## Finding Description

**Root cause — stale local bitmap vs. live storage bitmap:**

At `src/Midnight.sol:606`, the bitmap is snapshotted into a stack variable:

```solidity
uint128 _collateralBitmap = _position.collateralBitmap;   // local copy
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price();  // ← unconstrained external call
    ...
    _collateralBitmap = _collateralBitmap.clearBit(i);         // only local cleared
}
``` [1](#0-0) 

The loop only ever clears bits from the local copy. Any bit written to `_position.collateralBitmap` in storage during the `price()` call is invisible to the loop.

**Reentrancy path — `supplyCollateral` writes to storage bitmap:**

`supplyCollateral` has no reentrancy guard and is callable by any account authorized for `onBehalf`:

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
``` [2](#0-1) 

When a new collateral slot is first activated, it writes directly to the storage bitmap:

```solidity
if (oldCollateral == 0 && assets > 0) {
    uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
    _position.collateralBitmap = newCollateralBitmap;   // ← storage write
    ...
}
``` [3](#0-2) 

**Exploit flow (all steps attacker-controlled):**

1. Attacker (market creator) deploys a market with collateral index `i` using a `ReentrantOracle`, and collateral index `j > i` with a normal token. Both pass `touchMarket` validation (sorted, valid LLTV/maxLif).
2. Attacker (as borrower) supplies collateral `i`, borrows `D` units, becomes unhealthy (oracle price drops).
3. Attacker calls `setIsAuthorized(reentrantOracle, true, borrower)` — authorizes the oracle to supply on their behalf.
4. Attacker pre-funds `ReentrantOracle` with collateral `j` tokens and approves Midnight.
5. Anyone calls `liquidate(market, i, 0, 0, borrower, false, ...)` (seizedAssets=0, repaidUnits=0 — pure bad-debt realization path).
6. Loop starts: local `_collateralBitmap = {i}`.
7. Loop calls `ReentrantOracle.price()` for index `i`.
8. Oracle reenters: `supplyCollateral(market, j, largeAmount, borrower)` → storage bitmap becomes `{i, j}`, `_position.collateral[j] = largeAmount`.
9. Oracle returns `0` (or any price making `maxDebt < originalDebt`).
10. Loop reads `_position.collateral[i]` from storage (unchanged), computes `maxDebt += 0`, `badDebt` unchanged.
11. Loop clears bit `i` from local bitmap → local bitmap = `0`. Loop exits.
12. Bit `j` was never processed. `badDebt = D` (full original debt).
13. `_position.debt -= D` (borrower's debt zeroed), `lossFactor` updated to slash lenders by `D`.
14. Borrower retains collateral `j` (never seized, never counted) and can freely withdraw it since debt is now 0.

**Why existing checks do not stop it:**

`liquidationLocked` is only set during `take` callbacks (transient storage, scoped to the seller during `take`). It is never set during `liquidate`'s oracle calls, so it does not block `supplyCollateral` reentrancy: [4](#0-3) 

The `NotLiquidatable` check at lines 620–624 uses the stale `maxDebt` computed without collateral `j`, so it passes even though the position would be healthy if `j` were counted: [5](#0-4) 

There is no `nonReentrant` modifier anywhere on `liquidate` or `supplyCollateral`. The TOKEN SAFETY REQUIREMENTS in the contract comments explicitly restrict token reentrancy but make no equivalent restriction on oracle contracts.

## Impact Explanation

Lenders are slashed via `lossFactor` by the full `badDebt = D` instead of the correct `badDebt = D − value(collateral_j)/maxLif_j`. The attacker (borrower) has their debt zeroed and retains collateral `j` — effectively extracting `value(collateral_j)/maxLif_j` worth of loan tokens from lenders at no cost. The invariant "bad debt realization must account for all active collateral" is violated: collateral `j` is active in storage at the time of realization but is excluded from the computation. This constitutes direct theft of lender funds via inflated loss socialization. [6](#0-5) 

## Likelihood Explanation

All preconditions are reachable by a single unprivileged actor: market creation is permissionless via `touchMarket`, the attacker is both market creator and borrower, and `setIsAuthorized` is callable by any user for their own account. The attack is repeatable across any market the attacker creates. The attacker can size collateral `j` to maximize lender loss up to the full debt amount `D`. No gate or guard blocks the reentrancy path.

## Recommendation

Apply one or more of the following mitigations:

1. **Read the bitmap from storage after the oracle loop**, not before, so any storage writes during reentrancy are reflected. Alternatively, re-read `_position.collateralBitmap` from storage at the start of each loop iteration.
2. **Add a transient-storage reentrancy guard to `liquidate`** (analogous to `liquidationLocked`) that blocks `supplyCollateral` (and other state-mutating functions) for the borrower's position while a liquidation is in progress.
3. **Document and enforce an oracle trust assumption**: add an explicit check or note that oracles must be non-reentrant, and consider a registry or whitelist for oracle contracts.

The most robust fix is option 2: set a per-borrower transient lock at the start of `liquidate` and check it in `supplyCollateral`.

## Proof of Concept

**Minimal Foundry test outline:**

1. Deploy `Midnight`. Deploy `MockToken` (loan token), `CollateralTokenI`, `CollateralTokenJ`.
2. Deploy `ReentrantOracle` that, on `price()`, calls `Midnight.supplyCollateral(market, j, largeAmount, borrower)` using pre-approved `CollateralTokenJ`, then returns `0`.
3. Create market with `collateralParams = [{token: CollateralTokenI, oracle: ReentrantOracle, lltv: X, maxLif: Y}, {token: CollateralTokenJ, oracle: normalOracle, lltv: X, maxLif: Y}]` (sorted by token address).
4. Borrower: `supplyCollateral(market, 0, amount_i, borrower)`, then `take(...)` to borrow `D` units.
5. Borrower: `setIsAuthorized(reentrantOracle, true, borrower)`.
6. Pre-fund `ReentrantOracle` with `CollateralTokenJ`; approve Midnight.
7. Call `liquidate(market, 0, 0, 0, borrower, false, ...)`.
8. Assert: `position[id][borrower].debt == 0`, `position[id][borrower].collateral[j] == largeAmount`, `marketState[id].lossFactor` reflects full `D` slashed from lenders.
9. Borrower calls `withdrawCollateral(market, j, largeAmount, borrower, borrower)` — succeeds since debt is 0. [7](#0-6)
