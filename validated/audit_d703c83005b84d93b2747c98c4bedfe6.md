### Title
`TakeAmountsLib` Calls Outside `try/catch` in Bundler Loops Cause Full Transaction Revert — (File: `src/periphery/MidnightBundles.sol`)

---

### Summary

`MidnightBundles` wraps each `IMidnight.take()` call in a `try/catch` block to gracefully skip failing offers. However, `TakeAmountsLib.buyerAssetsToUnits` and `TakeAmountsLib.sellerAssetsToUnits` are called **outside** the `try/catch` to compute `unitsToTake`. Both functions contain revert conditions that can be triggered by a malicious maker's offer parameters. When triggered, the revert propagates out of the loop entirely, defeating the graceful-skip design and DoS-ing the entire bundler call.

---

### Finding Description

**Root cause — `buyerAssetsToUnits` outside `try/catch`:**

In `buyWithAssetsTargetAndWithdrawCollateral`, the loop computes `unitsToTake` before the `try` block:

```solidity
// MidnightBundles.sol lines 205-221
for (uint256 i; i < takes.length && filledBuyerAssets < targetFilledBuyerAssets; i++) {
    ...
    uint256 unitsToTake = min(
        TakeAmountsLib.buyerAssetsToUnits(          // <-- NOT inside try/catch
            MIDNIGHT, id, takes[i].offer, targetFilledBuyerAssets - filledBuyerAssets
        ),
        takes[i].units,
        ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
    );
    try IMidnight(MIDNIGHT).take(...) returns (...) {
        ...
    } catch {}
}
``` [1](#0-0) 

`TakeAmountsLib.buyerAssetsToUnits` contains:

```solidity
// TakeAmountsLib.sol lines 26-28
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
uint256 buyerPrice = sellerPrice + settlementFee;
require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());
``` [2](#0-1) 

For a **sell offer** (`offer.buy == false`), `buyerPrice = offerPrice + settlementFee`. At `tick = MAX_TICK = 5820`, `tickToPrice(5820) ≈ WAD`. Any non-zero settlement fee then makes `buyerPrice > WAD`, reverting `buyerAssetsToUnits` — outside the `try/catch`. [3](#0-2) 

**Root cause — `sellerAssetsToUnits` outside `try/catch`:**

The same pattern exists in `supplyCollateralAndSellWithAssetsTarget`:

```solidity
// MidnightBundles.sol lines 282-300
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(             // <-- NOT inside try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    ...
);
try IMidnight(MIDNIGHT).take(...) returns (...) { ... } catch {}
``` [4](#0-3) 

For a **buy offer** (`offer.buy == true`), `sellerPrice = offerPrice - settlementFee`. At `tick = 0`, `tickToPrice(0) ≈ 5e11` (≈ 5×10⁻⁷ WAD). The maximum settlement fee is 50 bps = 5×10¹⁵. Since `5e15 >> 5e11`, any non-zero settlement fee causes an arithmetic underflow revert in `sellerAssetsToUnits` for tick-0 buy offers. [5](#0-4) 

---

### Impact Explanation

The bundler's documented invariant — *"If taking an offer reverts, the bundler will completely skip this offer"* — is violated. A single malicious offer in the `takes` array causes the **entire bundler transaction to revert**, not just that offer to be skipped. The taker:

- Cannot execute any of the other valid offers in the batch
- Loses gas on the reverted transaction
- Is effectively DoS'd if their routing system automatically includes such offers [6](#0-5) 

---

### Likelihood Explanation

The attack is cheap and permissionless. Any maker can:
1. Sign a sell offer at `tick = MAX_TICK` (the highest-price tick, which looks attractive to buyers) in a market with non-zero settlement fee — this triggers the `buyerPrice > WAD` revert.
2. Sign a buy offer at `tick = 0` (the lowest-price tick) in a market with non-zero settlement fee — this triggers the underflow revert.

The offer does not need to be on-chain. It only needs to be included in the taker's `takes` array. Automated routing systems that sort offers by price would naturally rank a tick-`MAX_TICK` sell offer first (best price for the buyer), making it likely to be included.

---

### Recommendation

Move the `TakeAmountsLib` calls inside the `try/catch` scope, or wrap the entire `unitsToTake` computation in its own `try/catch` that skips the offer on revert. For example:

```solidity
try this._computeUnitsToTake(takes[i], id, targetFilledBuyerAssets - filledBuyerAssets)
    returns (uint256 unitsToTake) {
    try IMidnight(MIDNIGHT).take(...) returns (...) { ... } catch {}
} catch {}
```

Alternatively, add a pre-check `if (buyerPrice > WAD) continue;` before calling `buyerAssetsToUnits`.

---

### Proof of Concept

**Attack on `buyWithAssetsTargetAndWithdrawCollateral`:**

1. Deploy a market with a non-zero settlement fee (e.g., `settlementFeeCbp6 > 0`).
2. Malicious maker signs a sell offer with `tick = MAX_TICK = 5820` (price ≈ WAD).
3. Taker calls `buyWithAssetsTargetAndWithdrawCollateral` with `takes = [maliciousOffer, legitimateOffer1, legitimateOffer2]`.
4. On iteration `i=0`, `TakeAmountsLib.buyerAssetsToUnits` computes `buyerPrice = tickToPrice(5820) + settlementFee > WAD` and reverts with `PriceGreaterThanOne`.
5. The revert propagates out of the loop — the `try/catch` around `take()` never executes.
6. The entire transaction reverts. `legitimateOffer1` and `legitimateOffer2` are never attempted. [7](#0-6) [8](#0-7)

    function buyerAssetsToUnits(address midnight, bytes32 id, Offer memory offer, uint256 targetBuyerAssets)
        internal
        view
        returns (uint256)
    {
        uint256 offerPrice = TickLib.tickToPrice(offer.tick);
        uint256 settlementFee =
            IMidnight(midnight).settlementFee(id, UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp));
        // Mirrors Midnight's computation to revert if offerPrice < settlementFee in case of a buy offer.
        uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
        uint256 buyerPrice = sellerPrice + settlementFee;
        require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());
        return offer.buy ? targetBuyerAssets.mulDivUp(WAD, buyerPrice) : targetBuyerAssets.mulDivDown(WAD, buyerPrice);
    }
