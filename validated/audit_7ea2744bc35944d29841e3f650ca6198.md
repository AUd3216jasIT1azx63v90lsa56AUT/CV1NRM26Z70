Audit Report

## Title
`claimContinuousFee` reverts via arithmetic underflow when bad-debt liquidation front-runs the feeClaimer - (File: `src/Midnight.sol`)

## Summary
`claimContinuousFee` subtracts `amount` from `continuousFeeCredit` with no guard that `amount <= continuousFeeCredit`. A zero-cost bad-debt liquidation (`seizedAssets=0, repaidUnits=0`) atomically reduces `continuousFeeCredit` via the proportional slash formula, allowing any unprivileged caller to front-run the feeClaimer and cause its transaction to revert with a Solidity 0.8 arithmetic underflow, persistently griefing protocol fee collection.

## Finding Description

**Root cause — `claimContinuousFee`** (`src/Midnight.sol:318`):
```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
```
No `require(amount <= _marketState.continuousFeeCredit)` guard exists. Solidity 0.8 checked arithmetic reverts on underflow. [1](#0-0) 

**Bad-debt slash in `liquidate`** (`src/Midnight.sol:635–640`):
```solidity
_marketState.continuousFeeCredit = _lossFactor < type(uint128).max
    ? UtilsLib.toUint128(
        _marketState.continuousFeeCredit
            .mulDivDown(type(uint128).max - _marketState.lossFactor,
                        type(uint128).max - _lossFactor)
      )
    : 0;
```
Because `newLossFactor > oldLossFactor`, the ratio `(MAX − newLF) / (MAX − oldLF) < 1`, so `continuousFeeCredit` is strictly reduced on every bad-debt event. [2](#0-1) 

**Zero-cost liquidation path** (`src/Midnight.sol:577–578`, `src/Midnight.sol:643`):
The protocol explicitly documents that passing `seizedAssets=0, repaidUnits=0` realizes bad debt with zero tokens transferred. The `if (repaidUnits > 0 || seizedAssets > 0)` block at line 643 is skipped entirely, but the bad-debt slash at lines 626–640 executes unconditionally whenever `badDebt > 0`. [3](#0-2) [4](#0-3) 

**Exploit flow**:
1. `continuousFeeCredit = X`. feeClaimer broadcasts `claimContinuousFee(market, X, receiver)`.
2. Attacker front-runs with `liquidate(market, 0, 0, 0, borrower, false, receiver, address(0), "")` — zero tokens spent, only gas. `continuousFeeCredit` is reduced to `X' < X`.
3. feeClaimer's transaction executes: `X' -= X` → arithmetic underflow → revert.

**Why existing checks fail**:
- `liquidatorGate` check (`src/Midnight.sol:598`) only restricts the attacker when `market.liquidatorGate != address(0)`; markets with no gate (the default) are fully open. [5](#0-4) 
- The Certora spec (`certora/specs/Role.spec:264`) itself encodes `amount <= continuousFeeCreditBefore` as a necessary non-revert condition, confirming no on-chain guard enforces this invariant. [6](#0-5) 

## Impact Explanation
Protocol fee collection via `claimContinuousFee` is persistently griefed. No funds are permanently lost — `continuousFeeCredit` is reduced, not stolen — but the feeClaimer must re-read state and resubmit with a reduced amount on every attempt, and the attack can be replayed immediately on each resubmission as long as any bad-debt-eligible position exists. This constitutes sustained, low-cost denial of a core protocol revenue function.

## Likelihood Explanation
Attacker cost is gas only — no tokens required. Preconditions: (a) a borrower position with `debt > 0` and `originalDebt > maxDebt` (bad debt) exists, and (b) the feeClaimer submits `amount = continuousFeeCredit` (the natural "claim all" call). Both are routine market conditions. The attack is repeatable on every subsequent claim attempt and is accessible to any unprivileged address on markets without a `liquidatorGate`.

## Recommendation
Add an explicit cap in `claimContinuousFee` before the subtraction:
```solidity
amount = UtilsLib.min(amount, _marketState.continuousFeeCredit);
```
Or add a revert guard:
```solidity
require(amount <= _marketState.continuousFeeCredit, InsufficientContinuousFeeCredit());
```
The former is preferable as it makes the "claim all" pattern atomic and griefing-resistant. Additionally, consider whether the zero-cost bad-debt liquidation path should require a minimum `repaidUnits` or `seizedAssets` to prevent gas-only state manipulation.

## Proof of Concept
1. Deploy a market with no `liquidatorGate`.
2. Create a borrower position that accrues bad debt (`debt > maxDebt` and `debt > sum of collateral / maxLif`).
3. Accrue `continuousFeeCredit = X` via normal market activity.
4. Submit `claimContinuousFee(market, X, receiver)` from the feeClaimer EOA.
5. In the same block, front-run with `liquidate(market, 0, 0, 0, borrower, false, attacker, address(0), "")`.
6. Observe: `continuousFeeCredit` is reduced to `X' < X`; feeClaimer's tx reverts with arithmetic underflow.
7. Repeat from step 4 — the attack is immediately replayable.
