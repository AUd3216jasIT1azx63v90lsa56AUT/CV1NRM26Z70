### Title
Sponsor can redirect staker principal by changing `recoveryAddress` after staking - (File: src/ConfidencePool.sol)

### Summary
`setRecoveryAddress` imposes no post-stake lock, allowing the pool sponsor to silently redirect the entire pool — including stakers' principal — to an arbitrary address at any point before a CORRUPTED sweep executes.

### Finding Description
`setRecoveryAddress` (lines 611–618) contains only a zero-address guard:

```solidity
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    address oldRecoveryAddress = recoveryAddress;
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
```

There is no check on `expiryLocked`, `claimsStarted`, or any other lifecycle latch. By contrast, `setExpiry` (lines 622–632) explicitly gates on `expiryLocked` with the documented rationale "protects staker reliance: once anyone has deposited against a given deadline." The same reliance argument applies to `recoveryAddress` — stakers commit capital against a specific sweep destination — yet no equivalent one-way latch exists.

`recoveryAddress` is the live destination read at execution time by three sweep paths:
- `claimCorrupted` (line 423): sends the **entire pool** (`stakeToken.balanceOf(address(this))`) to `recoveryAddress` under bad-faith CORRUPTED.
- `sweepUnclaimedCorrupted` (line 468): sends the full remaining balance to `recoveryAddress` after the bounty window.
- `sweepUnclaimedBonus` (line 506): sends excess/dust to `recoveryAddress` under SURVIVED/EXPIRED.

Because all three read `recoveryAddress` at call time rather than snapshotting it at `flagOutcome`, any update between staking and sweep execution takes immediate effect.

### Impact Explanation
Under a bad-faith CORRUPTED resolution, `claimCorrupted` transfers `stakeToken.balanceOf(address(this))` — the full pool including every staker's principal — to whatever `recoveryAddress` is at that moment. A sponsor who updates `recoveryAddress` to their own address between `flagOutcome` and the first `claimCorrupted` call steals the entire pool. Stakers have no recourse: `withdraw` is permanently disabled once `riskWindowStart != 0`, and the outcome is already flagged. The same vector applies to `sweepUnclaimedCorrupted` (good-faith path remainder) and `sweepUnclaimedBonus` (bonus contributions).

### Likelihood Explanation
The sponsor is the agreement owner and pool creator — a non-trusted actor with direct `onlyOwner` access to `setRecoveryAddress`. No special precondition beyond pool creation is required. The window between `flagOutcome` and the first sweep call is open to any block, and the update is a single low-cost transaction. The asymmetry with `expiry` (locked after first stake) makes this an exploitable gap rather than a documented trade-off: the DESIGN.md §10 rationale for locking `expiry` ("protects staker reliance") applies identically to `recoveryAddress` but was never implemented for it.

### Recommendation
Apply a one-way latch to `recoveryAddress` mirroring the `expiryLocked` pattern for `expiry`. Snapshot `recoveryAddress` into an immutable storage slot at the first stake (when `expiryLocked` is set), or add a `recoveryAddressLocked` flag that is set at `flagOutcome` time and checked in `setRecoveryAddress`. At minimum, `setRecoveryAddress` should revert once `outcome != PoolStates.Outcome.UNRESOLVED` or once `claimsStarted` is true, so the sweep destination cannot be altered after the outcome is determined.

### Proof of Concept
1. Sponsor creates pool with `recoveryAddress = legitimateDAO`.
2. Stakers deposit; `expiryLocked` flips `true`.
3. Agreement is breached; moderator calls `flagOutcome(CORRUPTED, false, address(0))` (bad-faith). `snapshotTotalStaked + snapshotTotalBonus` is now owed to `recoveryAddress`.
4. Before any call to `claimCorrupted`, sponsor calls `setRecoveryAddress(sponsorWallet)`. No revert — no guard exists.
5. Any caller invokes `claimCorrupted()`:
   ```solidity
   uint256 toSweep = stakeToken.balanceOf(address(this)); // full pool
   stakeToken.safeTransfer(recoveryAddress, toSweep);     // → sponsorWallet
   ```
6. Entire pool (stakers' principal + bonus) is transferred to the sponsor. Stakers receive nothing. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
