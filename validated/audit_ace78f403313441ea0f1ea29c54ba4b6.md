Audit Report

## Title
Division by Zero in `sellerAssetsToUnits` via `tickToPrice(0) = 0` Causes DoS of Bundle Flows - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary

`tickToPrice(0)` provably returns `0` due to double rounding by `PRICE_ROUNDING_STEP`. When `consumableUnits` is called on a sell offer with `offer.maxUnits == 0` and `offer.tick == 0`, it delegates to `sellerAssetsToUnits`, which sets `sellerPrice = 0` and then calls `mulDivDown(targetSellerAssets, WAD, 0)`, triggering an unconditional division-by-zero revert. Because `MidnightBundles` calls `consumableUnits` outside any `try/catch`, the entire bundle transaction reverts.

## Finding Description

**Confirmed zero-price fact:**

`test/TickLibTest.sol` line 16 asserts and passes: [1](#0-0) 

The root cause is the double rounding in `tickToPrice`: for tick=0, the intermediate result of `1e36 / (1e18 + wExp(...))` is smaller than `PRICE_ROUNDING_STEP = 1e12`, so `divHalfDownUnchecked(..., PRICE_ROUNDING_STEP) * PRICE_ROUNDING_STEP` collapses to `0`: [2](#0-1) 

**Reachable division-by-zero path:**

`consumableUnits` reaches `sellerAssetsToUnits` when `offer.maxUnits == 0`, `offer.buy == false`: [3](#0-2) 

Inside `sellerAssetsToUnits`, for a sell offer (`offer.buy = false`), `sellerPrice` is set directly to `offerPrice = tickToPrice(0) = 0`, then `mulDivDown(targetSellerAssets, WAD, sellerPrice)` is called with denominator `0`: [4](#0-3) 

**Bundle functions propagate the revert:**

`buyWithUnitsTargetAndWithdrawCollateral` calls `consumableUnits` at line 77 outside any `try/catch`; only the subsequent `take` call is wrapped: [5](#0-4) [6](#0-5) 

`buyWithAssetsTargetAndWithdrawCollateral` has the same pattern at line 213: [7](#0-6) [8](#0-7) 

`supplyCollateralAndSellWithAssetsTarget` also calls `consumableUnits` at line 290 outside any `try/catch`: [9](#0-8) 

**Why existing checks fail:**

- `tickToPrice` only guards `tick > MAX_TICK`; tick=0 is in range.
- `consumableUnits` has no guard for `sellerPrice == 0`.
- `sellerAssetsToUnits` has no guard for `sellerPrice == 0`.
- The `NoDivisionByZero.spec` Certora spec covers only `Midnight.sol` and uses `NONDET` for `tickToPrice`, so it does not catch this periphery path: [10](#0-9) 

## Impact Explanation

Any bundle transaction that calls `consumableUnits` on a sell offer with `tick=0` and `maxUnits=0` reverts unconditionally. Because `MidnightBundles` does not wrap `consumableUnits` in a `try/catch`, the entire bundle call fails. This is a concrete DoS of bundle flows: any automated system (aggregator, router, front-end) that selects offers from an on-chain order book and includes a malicious offer in the `takes` array will have its entire bundle transaction griefed. Legitimate takers lose gas and fail to execute their intended trades.

## Likelihood Explanation

The preconditions are trivially satisfiable by any unprivileged maker: `tick=0` is always a valid, accessible tick (it satisfies `tick <= MAX_TICK` and `0 % spacing == 0` for any spacing), and setting `maxUnits=0` is a normal asset-capped offer configuration. The attack is zero-cost (no funds required beyond gas for offer creation), repeatable on every block, and requires no special role or privilege. The attack is most effective against automated bundle-construction systems that pull offers from an on-chain order book without filtering for zero-price ticks.

## Recommendation

Add a zero-price guard in `sellerAssetsToUnits` (and symmetrically in `buyerAssetsToUnits` for the sell-offer branch) before performing the division:

```solidity
require(sellerPrice > 0, ZeroPrice());
```

Alternatively, add the guard in `consumableUnits` before delegating to `sellerAssetsToUnits`, or in `tickToPrice` to revert on tick=0 (though this would be a breaking change to the tick model). The most targeted fix is a `require(sellerPrice > 0)` check in `TakeAmountsLib.sellerAssetsToUnits` before the `mulDivDown`/`mulDivUp` call. [11](#0-10) 

## Proof of Concept

1. Deploy `MidnightBundles` pointing to a live `Midnight` instance.
2. As an attacker (maker), create a sell offer with `tick=0`, `maxUnits=0`, `maxAssets=1` (any positive value).
3. As a victim (taker or automated bundler), call `buyWithUnitsTargetAndWithdrawCollateral` with a `takes` array that includes the attacker's offer.
4. Observe the transaction reverts unconditionally with a division-by-zero panic.

Minimal Forge unit test:
```solidity
function testDivByZeroDoS() public {
    // Confirm tickToPrice(0) == 0
    assertEq(TickLib.tickToPrice(0), 0);

    // Construct a sell offer at tick=0 with maxUnits=0, maxAssets=1
    Offer memory offer = Offer({
        tick: 0,
        buy: false,
        maxUnits: 0,
        maxAssets: 1,
        // ... other fields
    });

    // consumableUnits -> sellerAssetsToUnits -> mulDivDown(..., 0) -> revert
    vm.expectRevert(); // Panic: division by zero
    ConsumableUnitsLib.consumableUnits(address(midnight), id, offer);
}
```
