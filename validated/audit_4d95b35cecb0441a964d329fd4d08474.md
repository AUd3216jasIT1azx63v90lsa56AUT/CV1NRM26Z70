Audit Report

## Title
`sellerAssetsToUnits` reverts with arithmetic underflow when `sellerPrice == 0` on a valid buy offer at the settlement fee boundary - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary

`TakeAmountsLib.sellerAssetsToUnits` calls `targetSellerAssets.mulDivUp(WAD, sellerPrice)` where `sellerPrice = offerPrice - settlementFee` for buy offers. When `offerPrice == settlementFee`, `sellerPrice` is exactly `0`, and `UtilsLib.mulDivUp` evaluates `(x * y + (d - 1)) / d` with `d = 0`, causing an arithmetic underflow on `(d - 1)` in Solidity 0.8+ checked arithmetic before any division occurs. The core `Midnight.take()` accepts this offer without reverting (computing `units.mulDivDown(0, WAD) = 0`), so the boundary is valid in the core protocol but unhandled in the periphery inverse function. Because `supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside any `try/catch`, the revert propagates and DoSes the entire bundler call.

## Finding Description

**Root cause — `UtilsLib.mulDivUp` with `d = 0`:**

`UtilsLib.mulDivUp` is implemented as: [1](#0-0) 

With `d = 0`, the sub-expression `(d - 1)` underflows under Solidity 0.8 checked arithmetic, producing an `arithmeticError` revert before any division occurs. This is explicitly tested and confirmed in the test suite: [2](#0-1) 

**`TakeAmountsLib.sellerAssetsToUnits` — zero denominator path:**

For buy offers, `sellerPrice` is computed as `offerPrice - settlementFee`: [3](#0-2) 

When `offer.buy == true` and `tickToPrice(offer.tick) == settlementFee(id, ttm)`, line 44 yields `sellerPrice = 0`. Line 46 then calls `targetSellerAssets.mulDivUp(WAD, 0)` → arithmetic revert. The NatSpec comment on line 34 only documents the `offerPrice < settlementFee` revert (underflow in subtraction), not the `offerPrice == settlementFee` case (zero denominator in the inverse): [4](#0-3) 

**Why `Midnight.take()` accepts this offer:**

In the core protocol, `sellerPrice = 0` is computed identically, but the forward direction uses `mulDivDown`: [5](#0-4) 

`units.mulDivDown(0, WAD) = (units * 0) / WAD = 0` — no revert. The seller receives zero assets and all buyer payment goes to the settlement fee. The core protocol explicitly documents this: "Taking buy offers with price < settlement fee will revert" (line 331), but `price == settlementFee` is not rejected.

**Why the bundler call reverts entirely:**

`supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` at line 286 **outside** the `try/catch` block that wraps `IMidnight.take()`: [6](#0-5) 

The `try/catch` at line 292 only catches reverts from `take()` itself. The NatSpec at line 246 even explicitly acknowledges this: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." The loop condition `filledSellerAssets < targetFilledSellerAssets` (line 282) guarantees `targetFilledSellerAssets - filledSellerAssets > 0` when `sellerAssetsToUnits` is called, so `targetSellerAssets > 0` always — the zero-numerator escape does not apply.

**Fuzz test coverage gap:**

All fuzz tests in `TakeAmountsTest.sol` set `offer.buy = false` in `setUp()` and never exercise the buy-offer path through `sellerAssetsToUnits`: [7](#0-6) 

The `_maxTick` helper bounds ticks to ensure `buyerPrice <= WAD` for sell offers, but does not exclude `sellerPrice == 0` for buy offers: [8](#0-7) 

## Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee(id, ttm)` at call time reverts entirely. The taker's authorized bundler flow is completely blocked for that transaction. Collateral already supplied within the same call (lines 269–275) is locked in the bundler until the transaction reverts and unwinds. The attack is a permissionless, repeatable griefing DoS against any user of the `supplyCollateralAndSellWithAssetsTarget` bundler function.

## Likelihood Explanation

**Preconditions:**
1. A buy offer exists with `tickToPrice(offer.tick) == settlementFee(id, ttm)` at the time of the bundler call.
2. A victim taker has authorized the bundler and includes this offer in `takes[]`.

**Feasibility:** Settlement fees are set in multiples of `1e12` and tick prices are discrete values from `tickToPrice`. An attacker can enumerate tick prices and find one matching the current or near-future settlement fee. The condition is repeatable: the attacker can create a new offer each time the settlement fee drifts to a matching tick price. No privileged access is required — any address can create a buy offer.

## Recommendation

Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case on buy offers, mirroring the existing guard in `buyerAssetsToUnits`. The simplest fix is to require `sellerPrice > 0` before calling `mulDivUp`, or to return `type(uint256).max` (signaling "infinite units needed") when `sellerPrice == 0`, consistent with the semantics that a zero-price seller receives nothing regardless of units. Alternatively, add a `require(sellerPrice > 0)` with a descriptive error, since a buy offer at `offerPrice == settlementFee` yields zero seller assets and is economically meaningless for a seller-assets-target fill.

```solidity
// In sellerAssetsToUnits, after computing sellerPrice:
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
if (offer.buy && sellerPrice == 0) return type(uint256).max; // seller receives 0 assets per unit
```

The NatSpec comment on line 34 should also be updated to document the `offerPrice == settlementFee` boundary.
