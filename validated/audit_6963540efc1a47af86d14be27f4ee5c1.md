Audit Report

## Title
Fee-on-transfer collateral token inflates stored collateral, enabling undercollateralized borrowing - (File: src/Midnight.sol)

## Summary
`supplyCollateral` writes the caller-supplied `assets` value into `_position.collateral[collateralIndex]` before invoking `SafeTransferLib.safeTransferFrom`, which only verifies the boolean return of `transferFrom` and performs no balance-delta check. For a fee-on-transfer collateral token the contract receives `assets - fee` but records `assets`, inflating stored collateral. `isHealthy` computes `maxDebt` directly from the inflated stored value, allowing the borrower to take debt that exceeds the actual on-chain collateral value and creating unrecoverable bad debt.

## Finding Description

**Root cause — `supplyCollateral` (`src/Midnight.sol:533, 545`):**

```solidity
// line 533 — credit recorded BEFORE transfer
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
// ...
// line 545 — only checks transferFrom return value, no balance snapshot
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol:27-33`) calls `transferFrom` and checks only the boolean return value. It takes no before/after balance snapshot. For a fee-on-transfer token with fee rate `f`, the contract receives `assets*(1-f)` but `_position.collateral[collateralIndex]` stores `assets`.

**`isHealthy` uses the inflated stored value (`src/Midnight.sol:954-955`):**

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

`_position.collateral[i]` is the inflated `assets`, not the actual `assets*(1-f)` held by the contract.

**`touchMarket` imposes no token-type restriction (`src/Midnight.sol:762-773`):**

The only per-collateral checks are: address strictly ascending, `isLltvAllowed(lltv)`, and `maxLif` in the two allowed tiers. No check prevents a fee-on-transfer token.

**Exploit flow:**
1. Attacker calls `touchMarket` with a market whose `collateralParams[i].token` is a fee-on-transfer ERC20. Passes all validation.
2. Attacker calls `supplyCollateral(market, i, assets, borrower)`. Contract records `assets` in `_position.collateral[i]` but receives only `assets - fee`.
3. Attacker (as seller) calls `take` to increase debt. At line 476, `isHealthy` is called with the inflated stored collateral, returning `true` for debt up to `assets * lltv` instead of the correct `(assets-fee) * lltv`.
4. Position is now undercollateralized: actual collateral value supports only `(assets-fee)*lltv` debt but `assets*lltv` debt was taken.

**Why existing checks fail:**
- `SafeTransferLib` checks only `transferFrom` success, not balance delta.
- `isHealthy` reads `_position.collateral[i]` directly — it has no access to the actual token balance.
- `touchMarket` imposes no restriction on fee-on-transfer tokens.
- The TOKEN SAFETY REQUIREMENTS comment (`src/Midnight.sol:139`) is documentation only; it is not enforced on-chain.
- Certora formal verification summarizes `SafeTransferLib.safeTransferFrom` as `NONDET` (`certora/specs/CreatedMarkets.spec:33`), so the proofs do not cover fee-on-transfer behavior.
- `live_context.json:233` explicitly states fee-on-transfer tokens "should be tested if not explicitly excluded" — no explicit scope exclusion exists in `SECURITY.md`.

## Impact Explanation
An undercollateralized position is created. The gap between recorded collateral and actual collateral is `fee * assets` per `supplyCollateral` call, compounding with each deposit. For a 1% fee token with LLTV 0.77, supplying 1000 units records 1000 but holds 990; the borrower can take 770 units of debt while actual collateral supports only 762.3 units. When liquidated, the seized collateral is worth less than the repaid debt, socializing the shortfall as bad debt to lenders — a direct, concrete protocol insolvency impact matching the "bad debt creation" and "protocol insolvency" best-bug classes listed in `live_context.json:55-56`.

## Likelihood Explanation
`touchMarket` is permissionless — any address can create a market with a fee-on-transfer collateral token with no governance approval, whitelist, or privileged role required. The attacker can simultaneously be the market creator, the borrower, and the lender. The attack is repeatable on every `supplyCollateral` call, compounding the inflation with each deposit. Fee-on-transfer tokens exist on mainnet (deflationary mechanics, explicit transfer taxes).

## Recommendation
In `supplyCollateral`, snapshot the contract's collateral token balance before and after the transfer and credit only the actual received delta:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

This ensures `_position.collateral[i]` always reflects the actual on-chain balance, eliminating the inflation vector regardless of token fee mechanics. Alternatively, enforce the token assumption on-chain by reverting if `received != assets`.

        require(token.code.length > 0, NoCode());
