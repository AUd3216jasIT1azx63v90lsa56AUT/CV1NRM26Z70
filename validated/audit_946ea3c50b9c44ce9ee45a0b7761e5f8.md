Audit Report

## Title
`sellerAssetsToUnits` panics on division-by-zero when `sellerPrice == 0`, unconditionally reverting `supplyCollateralAndSellWithAssetsTarget` - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `targetSellerAssets.mulDivUp(WAD, sellerPrice)` for buy offers, where `sellerPrice = offerPrice - settlementFee`. When `offerPrice == settlementFee`, `sellerPrice` is exactly `0`, and `mulDivUp`'s implementation `(x * y + (d - 1)) / d` triggers an arithmetic underflow panic on `(d - 1)` before any division occurs. Because `sellerAssetsToUnits` is called outside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, the panic propagates and reverts the entire bundler transaction. The core `Midnight.take` does not revert for this same input, creating a divergence between core and periphery.

## Finding Description

**Root cause — `mulDivUp` with `d = 0`:** [1](#0-0) 

The function is not in an `unchecked` block. With `d = 0`, the sub-expression `(d - 1)` underflows under Solidity 0.8+ checked arithmetic, producing an arithmetic panic before any division.

**`sellerAssetsToUnits` passes `sellerPrice` as the divisor:** [2](#0-1) 

When `offer.buy == true` and `offerPrice == settlementFee`, `sellerPrice = offerPrice - settlementFee = 0`. The subtraction itself does not underflow (equal values), but the resulting `0` is then passed as `d` to `mulDivUp`. The NatSpec comment on line 34 states "Reverts if `offerPrice < settlementFee`" — the `==` case is not mentioned and not guarded. [3](#0-2) 

**Divergence from `Midnight.take`:** [4](#0-3) 

In the core, `sellerAssets = units.mulDivDown(0, WAD) = (units * 0) / WAD = 0` — no revert. The core proceeds normally; the periphery panics.

**Unprotected call site in `supplyCollateralAndSellWithAssetsTarget`:** [5](#0-4) 

`sellerAssetsToUnits` is called outside the `try/catch`. The bundler's own NatSpec explicitly acknowledges this: [6](#0-5) 

**Exploit flow:**
1. Attacker observes `settlementFee(id, ttm)` for a target market (public view function).
2. Attacker finds tick `T` such that `tickToPrice(T) == settlementFee` (both are multiples of `1e12`, as confirmed by test setup).
3. Attacker publishes a buy offer (`offer.buy = true`) at tick `T` — permissionless, gas cost only.
4. An off-chain aggregator includes this offer in a victim's `takes` array.
5. Victim calls `supplyCollateralAndSellWithAssetsTarget`; the loop reaches the poisoned offer, `sellerAssetsToUnits` panics, entire transaction reverts.

## Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee(id, ttm)` at call time reverts unconditionally. This blocks the sell-side bundle flow for borrowers who rely on the bundler to atomically supply collateral and borrow in one transaction. The DoS is repeatable at negligible cost to the attacker.

## Likelihood Explanation
Settlement fees and tick prices are both multiples of `1e12`; exact equality is achievable at specific TTM values. The settlement fee schedule is deterministic and public, so the attacker can predict the exact block at which a given tick's price equals the fee. The attacker only needs to publish one offer (permissionless). The victim must include the offer in their `takes` array — realistic when using an off-chain aggregator that sorts by price without pre-simulating `sellerAssetsToUnits`. The attacker can maintain a portfolio of offers at multiple ticks to cover a range of future fee values.

## Recommendation
In `sellerAssetsToUnits`, add an explicit guard before the `mulDivUp` call for the buy-offer branch:

```solidity
// In sellerAssetsToUnits, after computing sellerPrice:
if (offer.buy) {
    require(sellerPrice > 0, SellerPriceIsZero());
    return targetSellerAssets.mulDivUp(WAD, sellerPrice);
}
return targetSellerAssets.mulDivDown(WAD, sellerPrice);
```

This mirrors the existing guard in `buyerAssetsToUnits` (`require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne())`). When `sellerPrice == 0`, the seller receives zero assets for any number of units, so a nonzero `targetSellerAssets` is unreachable and should revert with a meaningful error rather than an opaque arithmetic panic. [7](#0-6) 

## Proof of Concept
Minimal Foundry test:
1. Deploy `Midnight` and `MidnightBundles`.
2. Create a market and set `settlementFee` for the relevant TTM bucket to a value `F` that is a multiple of `1e12`.
3. Find tick `T` such that `TickLib.tickToPrice(T) == F`.
4. Create a buy offer at tick `T` with `offer.buy = true`, `offer.maxUnits > 0`.
5. Call `midnightBundles.supplyCollateralAndSellWithAssetsTarget(...)` with `takes[0]` pointing to this offer.
6. Observe the transaction reverts with `Panic(0x11)` (arithmetic underflow) rather than completing or skipping the offer.

Alternatively, a direct unit test on `TakeAmountsLib.sellerAssetsToUnits` with `sellerPrice = 0` (achieved by mocking `settlementFee` to equal `tickToPrice(offer.tick)`) will reproduce the panic in isolation.
