Audit Report

## Title
Double `mulDivDown` in `isHealthy` Produces False-Negative at Exact Health Boundary, Causing `take` to Revert with `SellerIsLiquidatable()` - (File: src/Midnight.sol)

## Summary
`isHealthy` computes `maxDebt` via two sequential `mulDivDown` calls, which can produce a result 1 unit below the true single-step floor. When a `take` brings the seller's debt to exactly the true mathematical `maxDebt`, the protocol's computed `maxDebt` equals `true_maxDebt - 1`, causing `isHealthy` to return `false` and the `take` to revert with `SellerIsLiquidatable()` even though the position is genuinely healthy. No funds are lost, but the seller's effective borrowing capacity is artificially capped by 1 loan-token unit.

## Finding Description

**Code path:**

`take()` at `src/Midnight.sol:414` writes `sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease)`, then at line 476 calls:

```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```

`isHealthy` at `src/Midnight.sol:954–955` computes:

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

`mulDivDown` in `src/libraries/UtilsLib.sol:29–31` is plain integer division:

```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
```

**Root cause — two-step rounding loss:**

Let `A = floor(C * P / ORACLE_PRICE_SCALE)`. The protocol computes `maxDebt = floor(A * lltv / WAD)`. The exact single-step floor is `floor(C * P * lltv / (ORACLE_PRICE_SCALE * WAD))`. Because `A` already lost up to `(ORACLE_PRICE_SCALE - 1) / ORACLE_PRICE_SCALE` in the first division, the second division can lose an additional unit, making `maxDebt = exact_floor - 1`.

**Concrete numerical example** (with `ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`):

- `C = 7`, `P = 7e35` (= 0.7 × ORACLE_PRICE_SCALE), `lltv = 7e17` (= 0.7 × WAD)
- Step 1: `floor(7 × 7e35 / 1e36) = floor(4.9) = 4`
- Step 2: `floor(4 × 7e17 / 1e18) = floor(2.8) = 2`
- Exact single-step: `floor(7 × 7e35 × 7e17 / (1e36 × 1e18)) = floor(3.43) = 3`

If `debt = 3` after the take, `isHealthy` returns `2 >= 3 = false` → revert, but the position is truly healthy (3.43 ≥ 3).

**Exploit path:**

1. Seller has existing debt `D` and collateral `C` with oracle price `P` and `lltv` satisfying the rounding condition above.
2. Taker calls `take()` with `units` chosen so that `sellerDebtIncrease = exact_floor - D`, landing `sellerPos.debt` at exactly `exact_floor`.
3. `isHealthy` computes `maxDebt = exact_floor - 1 < debt = exact_floor` → returns `false`.
4. `take` reverts with `SellerIsLiquidatable()`.

The taker controls `units` directly; `sellerDebtIncrease = units - min(units, sellerPos.credit)`, which equals `units` when the seller has no credit.

**Why existing checks fail:**

The Certora `stayHealthy` rule in `certora/specs/Healthiness.spec:260` explicitly excludes `take`:

```solidity
rule stayHealthy(env e, method f, calldataarg args) filtered { 
    f -> f.selector != sig:liquidate(...).selector && 
         f.selector != sig:take(...).selector 
}
```

No formal proof covers the invariant that a healthy seller remains healthy after a `take`. The `liquidate` function at `src/Midnight.sol:613` uses the identical double-`mulDivDown` pattern, so the protocol is internally self-consistent — but both diverge from the true mathematical boundary by 1 unit at the rounding edge.

## Impact Explanation
A legitimate `take` that would leave the seller's position at the true mathematical health boundary reverts with `SellerIsLiquidatable()`. The seller's effective borrowing capacity is capped at `computed_maxDebt` rather than `exact_floor`. This is a repeatable, 1-unit denial of service at the boundary — no funds are lost or frozen, and the seller's position is unchanged on revert. Severity is low.

## Likelihood Explanation
Requires `C * P` to be non-divisible by `ORACLE_PRICE_SCALE` (so step-1 rounding occurs) and the fractional loss to propagate through step 2. With arbitrary oracle prices and collateral amounts this condition is reachable without any contrived overflow or impossible-oracle scenario. The taker must choose `units` precisely to land debt at `exact_floor`, which is straightforward given knowledge of the seller's current state. It is repeatable for any `(collateral, price, lltv)` triple satisfying the condition.

## Recommendation
Replace the two sequential `mulDivDown` calls with a single three-argument computation that avoids the intermediate rounding loss:

```solidity
// Instead of:
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);

// Use a single-step equivalent (requires intermediate product to not overflow):
maxDebt += (_position.collateral[i] * price / ORACLE_PRICE_SCALE * collateralParam.lltv) / WAD;
// Or, if overflow is a concern, use a 512-bit mulDiv (e.g., FullMath.mulDiv):
maxDebt += FullMath.mulDiv(_position.collateral[i], price * collateralParam.lltv, ORACLE_PRICE_SCALE * WAD);
```

Apply the same fix to the identical pattern in `liquidate` at `src/Midnight.sol:613` to keep both checks consistent with the true mathematical boundary.

## Proof of Concept
Minimal Foundry unit test plan:

1. Deploy `Midnight` with a market using `lltv = 7e17` and an oracle returning `price = 7e35`.
2. Supply `collateral = 7` units to a borrower (seller).
3. Verify `isHealthy` returns `true` with `debt = 0`.
4. Directly write `debt = 2` into the seller's position storage (below `computed_maxDebt = 2`).
5. Construct a sell offer from the seller. Have a taker call `take` with `units = 1` (bringing `debt` to `3 = exact_floor`).
6. Observe the transaction reverts with `SellerIsLiquidatable()`.
7. Confirm `isHealthy` returns `false` (computed `maxDebt = 2 < debt = 3`).
8. Confirm the true mathematical value `floor(7 × 7e35 × 7e17 / (1e36 × 1e18)) = 3 ≥ 3`, proving the position is genuinely healthy.
