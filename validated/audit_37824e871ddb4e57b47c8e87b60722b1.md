Audit Report

## Title
Debt increase permitted at exact maturity timestamp due to `<=` boundary check - (File: src/Midnight.sol)

## Summary
The `take()` function guards against post-maturity debt increases with `require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity())` at line 391. The `<=` operator allows the condition to pass when `block.timestamp == offer.market.maturity`, permitting `sellerDebtIncrease > 0` at the exact maturity boundary. This directly violates the protocol invariant that debt must not increase at or after maturity, which is explicitly stated in `live_context.json` as "maturity boundary must not allow debt increase by rounding, **timestamp equality**, or callback sequencing."

## Finding Description
**Root cause:** The guard at line 391 uses `<=` instead of `<`:

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

When `block.timestamp == offer.market.maturity`, the left operand evaluates to `true`, making the entire `||` expression unconditionally `true`. The require never reverts regardless of `sellerDebtIncrease`.

**Code path:**
1. `timeToMaturity` is computed at line 359 as `zeroFloorSub(offer.market.maturity, block.timestamp)`. At equality, this returns `0`.
2. `settlementFee(id, 0)` returns the 0-day breakpoint fee; all price/unit computations proceed normally without reverting.
3. At line 384, `sellerDebtIncrease = units - sellerCreditDecrease`. If the seller has no existing credit, `sellerCreditDecrease = 0` and `sellerDebtIncrease = units > 0`.
4. The guard at line 391 passes unconditionally because `block.timestamp <= offer.market.maturity` is `true`.
5. Line 414 executes: `sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease)` — new debt is minted at maturity.

**Why existing checks fail:** No other check in `take()` prevents this. The expiry check (`block.timestamp <= offer.expiry`) is orthogonal. The `reduceOnly` check only applies if the maker set that flag. The gate checks at lines 402–406 only apply if `sellerDebtIncrease > 0` AND a gate is configured — they do not substitute for the maturity guard. The only maturity guard is the flawed `<=` at line 391.

## Impact Explanation
A taker can cause `sellerPos.debt` to increase at the exact maturity block. This corrupts the debt/credit accounting invariant: a borrower who should be in a pure-repayment/settlement phase has new debt minted against them. This can push a previously healthy position into an unhealthy state, enabling liquidation of a position that should have been protected from new debt. The impact class is "credit/debt accounting corruption" and "bad debt creation," both listed as highest-priority bug classes in `live_context.json`.

## Likelihood Explanation
The precondition `block.timestamp == offer.market.maturity` is a single-block window that occurs exactly once per market. A taker monitoring the chain can submit a transaction in that specific block. The offer requires only a valid ratifier (e.g., a signature-based ratifier the maker already authorized) and a seller with no existing credit. No privileged access, oracle manipulation, or admin action is required. This is repeatable across every market whose maturity falls on a future block.

## Recommendation
Change `<=` to `<` at line 391:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns with the explicit invariant in `live_context.json` line 221: *"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."* The `high_value_attack_flows` entry `maturity_boundary_trade` also explicitly tests "warp to maturity" → "try debt-increasing trade" → "assert debt cannot increase after maturity," confirming the intended behavior.

## Proof of Concept
Minimal Foundry test plan:
1. Create a market with `maturity = block.timestamp + 1000`.
2. Open a sell offer (maker is seller) where the seller has zero existing credit.
3. `vm.warp(maturity)` — set `block.timestamp == offer.market.maturity`.
4. Call `take()` with `units > 0` as an unprivileged taker.
5. Assert: `sellerPos.debt` increased (demonstrating the bug).
6. With the fix applied (`<` instead of `<=`), assert the call reverts with `CannotIncreaseDebtPostMaturity`.
