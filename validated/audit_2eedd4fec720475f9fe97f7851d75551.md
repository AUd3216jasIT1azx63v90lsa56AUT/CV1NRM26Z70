Audit Report

## Title
Fee-on-transfer loan token inflates `_marketState.withdrawable` beyond actual contract balance in `liquidate` and `repay` - (File: src/Midnight.sol)

## Summary

In `liquidate` and `repay`, `_marketState.withdrawable` is incremented by the full nominal `repaidUnits`/`units` before `SafeTransferLib.safeTransferFrom` pulls those tokens from the payer. When the loan token silently deducts a transfer fee, the contract receives fewer tokens than recorded, permanently inflating `withdrawable` relative to the actual balance. Lenders who subsequently call `withdraw` drain the contract faster than tokens arrived, causing the final lender(s) to receive a revert or shortfall — protocol insolvency within that market.

## Finding Description

**Root cause:** `_marketState.withdrawable` is updated with the full nominal amount before the token transfer executes, and `SafeTransferLib.safeTransferFrom` performs no pre/post balance snapshot — it only checks `success` and the boolean return value.

**Exact code paths:**

`liquidate()` in `src/Midnight.sol`:
- Line 675: `_marketState.withdrawable += UtilsLib.toUint128(repaidUnits);` — accounting credited in full
- Line 717: `SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);` — only `repaidUnits - fee` arrives

`repay()` in `src/Midnight.sol`:
- Line 509: `marketState[id].withdrawable += UtilsLib.toUint128(units);` — same pattern
- Line 520: `SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);` — only `units - fee` arrives

`withdraw()` in `src/Midnight.sol`:
- Line 494: `_marketState.withdrawable -= UtilsLib.toUint128(units);`
- Line 499: `SafeTransferLib.safeTransfer(market.loanToken, receiver, units);` — transfers the full `units` out, drawing down the actual balance by the same amount the inflated accounting claims

`SafeTransferLib.safeTransferFrom` in `src/libraries/SafeTransferLib.sol` lines 24–34: calls `transferFrom`, checks `success` and the decoded boolean, and returns. No balance delta is measured. A fee silently retained by the token contract is invisible.

`touchMarket()` in `src/Midnight.sol` lines 755–791: validates collateral params (lltv tiers, maxLif, sort order) but imposes no restriction on the loan token type. Any ERC20, including a fee-on-transfer token, is accepted.

**Exploit flow:**
1. Attacker deploys (or uses) a fee-on-transfer ERC20 `FeeToken` with per-transfer fee `f`.
2. Attacker calls `touchMarket(market)` with `market.loanToken = address(FeeToken)` — succeeds unconditionally.
3. Lenders supply credit; borrowers take debt denominated in `FeeToken`.
4. Borrower becomes liquidatable.
5. Liquidator calls `liquidate(..., repaidUnits=R, ...)`.
6. Line 675 executes: `_marketState.withdrawable += R`.
7. Line 717 executes: `FeeToken.transferFrom(payer, address(this), R)` — contract receives `R - f`.
8. Post-call: `IERC20(FeeToken).balanceOf(address(this)) < _marketState.withdrawable` by exactly `f`.
9. Each subsequent `liquidate` or `repay` compounds the deficit by another `f`.
10. When lenders call `withdraw()`, the last lender(s) trigger a revert because the contract holds fewer tokens than `withdrawable` promises.

**Why existing checks fail:**
- `SafeTransferLib.safeTransferFrom` has no balance-delta check — only verifies call success and boolean return.
- `touchMarket` has no loan token allowlist or fee-on-transfer guard.
- `live_context.json` core invariant "ERC20 transfer deltas must match accounting deltas" (line 232) is violated; the same file notes fee-on-transfer tokens "should be tested if not explicitly excluded" (line 233) — they are not excluded anywhere in the code or documentation.

## Impact Explanation

Every `liquidate` or `repay` call against a fee-on-transfer loan token inflates `_marketState.withdrawable` by the fee amount without a corresponding token balance increase. Once cumulative fees exceed the contract's actual balance for that market, `safeTransfer` in `withdraw()` reverts for the last withdrawing lender(s), permanently locking their credit. This is protocol insolvency: lenders collectively hold more withdrawable credit than the contract can honour. The impact class is direct loss of user funds / permanent fund freeze, both listed as highest-priority bug classes in `live_context.json`.

## Likelihood Explanation

All three preconditions are reachable by unprivileged actors:
1. Market creation is permissionless — any address can call `touchMarket` with any ERC20 as loan token.
2. A borrower becoming liquidatable is normal market operation.
3. A liquidator calling `liquidate` with `repaidUnits > 0` is the standard liquidation flow.

The bug is triggered on every single liquidation or repayment in such a market, making it repeatable and cumulative. Real-world fee-on-transfer tokens (tokens with built-in tax mechanisms) are deployed on mainnet. The protocol's own design states "arbitrary loan token" as a feature, and no exclusion of fee-on-transfer tokens exists in `SECURITY.md`, `RESEARCHER.md`, or the contract code.

## Recommendation

**Option A (preferred — balance-delta check):** Measure the contract's token balance before and after `safeTransferFrom` and use the actual received delta for accounting updates:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
_marketState.withdrawable += UtilsLib.toUint128(received);
_position.debt -= UtilsLib.toUint128(received);
```

Apply the same pattern in `repay()`.

**Option B (explicit exclusion):** If fee-on-transfer tokens are intentionally out of scope, add an explicit check in `touchMarket` (e.g., a balance-delta probe during market creation) or document the exclusion clearly and enforce it via a token allowlist.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
