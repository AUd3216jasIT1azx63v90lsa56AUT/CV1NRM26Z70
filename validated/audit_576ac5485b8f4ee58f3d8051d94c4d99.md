Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via zero-rounding of `buyerAssets` when `offerPrice < WAD` - (File: src/Midnight.sol)

## Summary
When `offer.buy = true` and `offerPrice < WAD`, `buyerAssets` is computed with `mulDivDown` at line 363, which floors to zero for any `units` where `units * buyerPrice < WAD`. Because the `consumed` accumulator at line 368 is incremented by `buyerAssets` (not `units`), a taker can call `take` with such small `units` values indefinitely without ever advancing `consumed`, bypassing the maker's `maxAssets` cap entirely. The protocol's own test suite contains `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol`, confirming this is a live defect.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–373:** [1](#0-0) 

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

`WAD = 1e18` per `src/libraries/ConstantsLib.sol` line 8. [2](#0-1) 

`tickToPrice` at `src/libraries/TickLib.sol` lines 44–52 computes `1e36 / (1e18 + wExp(LN_ONE_PLUS_DELTA * (MAX_TICK/2 - tick)))`. For any tick > `MAX_TICK/2 = 2910`, the exponent is negative, `wExp` returns a value < 1e18, and the resulting price is < 1e18 = WAD. This covers ticks 2911–5820, roughly half the valid tick range. [3](#0-2) 

With `offerPrice < WAD` and `units = 1`: `buyerAssets = 1 * offerPrice / WAD = 0` (floor division). The `consumed` accumulator is incremented by zero, so `newConsumed` never grows and the `ConsumedAssets` guard never fires.

**Position state still mutates on every call** (lines 408–414): `buyerPos.credit += buyerCreditIncrease` and `sellerPos.debt += sellerDebtIncrease` execute regardless of `buyerAssets`. [4](#0-3) 

**Token transfers are also zero** (lines 455–456): `safeTransferFrom(..., buyerAssets - sellerAssets)` and `safeTransferFrom(..., sellerAssets)` both transfer 0, so the maker accumulates credit without paying loan tokens. [5](#0-4) 

**Why existing checks fail:** The `atMostOneNonZero` guard at line 350 only enforces that at most one of `maxAssets`/`maxUnits` is set; it does not prevent zero-valued asset increments. The `else` branch at line 371 (the `maxUnits` path) correctly increments by `units` and is not affected. The `isRatified` check at line 356 validates the Merkle proof/signature over the offer struct and does not inspect `units` or `buyerAssets`. [6](#0-5) 

**Protocol-confirmed:** `testBugBuyMaxAssetsBypass` appears twice in `test/TakeTest.sol`, confirming the protocol team has already identified this path. [7](#0-6) 

## Impact Explanation
The `maxAssets` invariant — intended to bound total buyer-asset exposure per maker/group — is rendered completely ineffective for buy offers at ticks > 2910. `totalUnits` and position credit/debt grow without bound while zero loan tokens back the new credit. The maker accumulates unbacked credit claims; the taker accumulates debt with zero loan-token receipt. This breaks the protocol's core accounting integrity: the invariant "offers cannot be overfilled beyond `maxAssets`" is violated, and the protocol holds unbacked credit obligations. Severity: Critical.

## Likelihood Explanation
All preconditions are reachable by an unprivileged taker with no special access: (1) `offer.buy = true` — standard offer type; (2) `offer.maxAssets > 0` — standard cap usage; (3) tick > 2910 — normal, covering half the valid tick range; (4) `units = 1` — minimum valid input. The attack is repeatable in a single transaction via `multicall`. No victim mistake or external dependency is required.

## Recommendation
Add a guard requiring `buyerAssets > 0` before the consumed check, or change the consumed accumulator for buy offers to use `mulDivUp` (matching the sell-offer rounding) so that any nonzero `units` always produces a nonzero asset increment:

```solidity
// Option A: reject dust takes that round to zero
require(buyerAssets > 0 || units == 0, ZeroBuyerAssets());

// Option B: use ceiling division for consumed tracking on buy offers
uint256 consumedIncrease = offer.buy
    ? units.mulDivUp(buyerPrice, WAD)   // was mulDivDown
    : sellerAssets;
newConsumed = consumed[offer.maker][offer.group] += consumedIncrease;
```

Option B is preferable as it is conservative (slightly over-counts consumed assets) and does not introduce a new revert path for legitimate small takes.

## Proof of Concept
The protocol's own `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` is the minimal reproducer. Manually:

1. Deploy Midnight; create a market with tick spacing 1.
2. Maker publishes a buy offer with `maxAssets = 1e6` and `tick = 5820` (price ≈ 1e12 << WAD).
3. Taker calls `take(units=1, ...)` in a loop (or via `multicall`).
4. Each iteration: `buyerAssets = 1 * 1e12 / 1e18 = 0`; `consumed += 0`; cap check passes; `buyerPos.credit` increments by 1.
5. After N iterations: `consumed = 0` (cap never triggered), `buyerPos.credit = N`, zero loan tokens transferred. The `maxAssets` cap is fully bypassed.
