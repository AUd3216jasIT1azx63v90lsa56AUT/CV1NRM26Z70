Audit Report

## Title
Silent No-Op `transferFrom` Bypasses Loan Token Payment in `liquidate()`, Breaking `withdrawable` Accounting Invariant - ([File: src/Midnight.sol])

## Summary
`SafeTransferLib.safeTransferFrom` passes silently when a token's `transferFrom` returns empty calldata, because the guard short-circuits to `true` on zero-length returndata. In `liquidate()`, all critical state mutations — collateral seizure, debt reduction, and `withdrawable` increment — are committed and the real collateral is transferred out before the loan token pull at line 717. A permissionless market creator who sets `loanToken` to a no-op ERC20 can therefore liquidate an unhealthy position, receive real collateral, and pay zero loan tokens, permanently inflating `withdrawable` beyond the protocol's actual loan token balance.

## Finding Description

**Root cause — `SafeTransferLib.safeTransferFrom`:**

`src/libraries/SafeTransferLib.sol` line 33:
```solidity
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
```
A contract whose `transferFrom` body is empty (or simply `return`s with no value) satisfies this: the low-level call succeeds (`success = true`), `returndata` is zero-length, and the require is trivially satisfied. No tokens move.

**State mutations before the transfer in `liquidate()`:**

`src/Midnight.sol` lines 670–676:
```solidity
_position.collateral[collateralIndex] = newCollateral;
...
_marketState.withdrawable += UtilsLib.toUint128(repaidUnits);
_position.debt -= UtilsLib.toUint128(repaidUnits);
```
All accounting is finalized before any external call. Real collateral is then sent to the receiver at line 696:
```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
```
Only after collateral is gone does the protocol attempt to pull loan tokens at line 717:
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```
There is no balance snapshot before/after this call, so a silent no-op is indistinguishable from a real transfer.

**Permissionless market creation — no `loanToken` validation:**

`touchMarket` (lines 755–791) validates `lltv`, `maxLif`, collateral ordering, and maturity, but imposes zero restrictions on `market.loanToken`. Any contract address with `code.length > 0` is accepted.

**Exploit flow:**
1. Attacker deploys `ERC20NoRevert` (contract with `transferFrom() {}` — no-op, no return value).
2. Attacker calls `touchMarket` with `loanToken = ERC20NoRevert` and a real collateral token.
3. Attacker posts a buy offer as lender — pays nothing (no-op loanToken). Victim takes the sell side (borrows), supplies real collateral via `supplyCollateral`, and has debt recorded on-chain despite receiving no real loan tokens.
4. Position becomes unhealthy (attacker controls oracle address, also unchecked in `touchMarket`, or waits for maturity).
5. Attacker calls `liquidate(market, collateralIndex, seizedAssets > 0, 0, borrower, ...)`.
6. Lines 649–650 compute `repaidUnits` from `seizedAssets` and oracle price (non-zero).
7. Lines 670–676: collateral deducted, `withdrawable += repaidUnits`, `debt -= repaidUnits`.
8. Line 696: real collateral token transferred to attacker's `receiver`.
9. Line 717: `safeTransferFrom(ERC20NoRevert, payer, address(this), repaidUnits)` — no-op, succeeds, zero tokens received.

**Why existing checks fail:**
- `require(UtilsLib.atMostOneNonZero(repaidUnits, seizedAssets))` — satisfied; attacker passes `seizedAssets > 0` as input; `repaidUnits` is computed internally.
- `require(_position.debt > 0)` — satisfied; victim has debt.
- `require(!liquidationLocked && originalDebt > maxDebt)` — satisfied; position is genuinely unhealthy.
- `liquidatorGate` — `address(0)` in attacker-created market.
- `SafeTransferLib` — passes on empty returndata.
- No post-transfer balance assertion exists anywhere in `liquidate()`.

The contract's TOKEN SAFETY REQUIREMENTS (lines 133–140) document that "Midnight must send/receive exactly the requested amount on transfers," but this is never enforced on-chain.

## Impact Explanation

After the call: `_marketState.withdrawable` has increased by `repaidUnits` but `loanToken.balanceOf(address(this))` is unchanged. The invariant `withdrawable <= loanToken.balanceOf(midnight)` is broken. The attacker receives real collateral assets for free. Any lender who later calls `withdraw` against this market will find the protocol insolvent for the inflated `withdrawable` amount. The borrower's debt is erased without repayment. This constitutes direct theft of collateral assets and permanent corruption of protocol accounting state.

## Likelihood Explanation

Market creation is fully permissionless — no governance approval is required. Deploying an `ERC20NoRevert` contract costs one transaction. The oracle address is also unchecked in `touchMarket`, so the attacker can control when a position becomes unhealthy. The attack is repeatable across any number of markets and positions. The social engineering surface (convincing a victim to supply real collateral to a market with a no-op loanToken) is realistic in DeFi given permissionless market creation and frontend abstraction.

## Recommendation

1. **Add a post-transfer balance check in `liquidate()`**: snapshot `loanToken.balanceOf(address(this))` before line 717 and assert it increased by at least `repaidUnits` after the call.
2. **Alternatively, enforce token safety at market creation**: in `touchMarket`, require that `market.loanToken` passes a basic sanity check (e.g., a test `transferFrom` with value 0, or a registry of approved tokens).
3. **Checks-effects-interactions**: move the `safeTransferFrom` loan token pull (line 717) before the collateral transfer (line 696) and state mutations, so a failed or no-op pull reverts the entire transaction before any assets leave the protocol.
