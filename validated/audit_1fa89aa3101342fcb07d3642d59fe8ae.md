All five factual claims verified against the actual source code. The vulnerability is real.

Audit Report

## Title
`buyWithAssetsTargetAndWithdrawCollateral` reverts entirely when any sell offer has `tick = MAX_TICK` and non-zero settlement fee - (File: `src/periphery/MidnightBundles.sol`)

## Summary

`buyWithAssetsTargetAndWithdrawCollateral` computes `unitsToTake` via `TakeAmountsLib.buyerAssetsToUnits` at lines 208–214, which is placed entirely **outside** the `try/catch` block at line 215. For any sell offer at `tick = MAX_TICK` (5820) in a market with a non-zero settlement fee, `buyerPrice = WAD + settlementFee > WAD`, triggering the `require(buyerPrice <= WAD)` guard and reverting the entire bundle transaction. A malicious maker can exploit this to permanently grief any taker bundle that includes their offer.

## Finding Description

**Root cause — pre-computation outside `try`:**

The loop at lines 205–222 of `MidnightBundles.sol` evaluates `unitsToTake` as:

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.buyerAssetsToUnits(...),   // line 209 — OUTSIDE try
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(...)
);
try IMidnight(MIDNIGHT).take(...) { ... } catch {}  // line 215
``` [1](#0-0) 

**Revert trigger in `buyerAssetsToUnits`:**

For a sell offer (`offer.buy = false`), `TakeAmountsLib.buyerAssetsToUnits` computes:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;  // = offerPrice
uint256 buyerPrice = sellerPrice + settlementFee;                            // = offerPrice + settlementFee
require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());
``` [2](#0-1) 

**`tickToPrice(MAX_TICK)` returns exactly WAD:**

At `tick = MAX_TICK = 5820`, the exponent argument `LN_ONE_PLUS_DELTA * (2910 - 5820)` is large and negative. `wExp` of a large negative number returns `1e36 / wExp(large_positive) ≈ 498`. Thus `1e18 + 498` is the denominator, yielding `1e36 / (1e18 + 498) ≈ 1e18 - 498`. After `divHalfDownUnchecked` rounding to `PRICE_ROUNDING_STEP = 1e12`, the result is exactly `1e18 = WAD`. [3](#0-2) [4](#0-3) 

With any `settlementFee > 0`, `buyerPrice = WAD + settlementFee > WAD`, so the `require` reverts unconditionally.

**`ConsumableUnitsLib` does not rescue this:**

For a sell offer with `maxUnits == 0`, `consumableUnits` calls `sellerAssetsToUnits` (which has no `buyerPrice <= WAD` guard). For `maxUnits > 0`, it returns directly. Neither path preempts the revert from `buyerAssetsToUnits`. [5](#0-4) [6](#0-5) 

**NatSpec guarantee violated:**

Line 176 promises "If taking an offer reverts, the bundler will completely skip this offer." The revert from `buyerAssetsToUnits` is not a revert from `take()` — it occurs before the `try` keyword and unwinds the entire transaction, violating the documented skip guarantee. [7](#0-6) 

## Impact Explanation

The entire `buyWithAssetsTargetAndWithdrawCollateral` transaction reverts. All other valid offers in the `takes[]` array are blocked. The taker's `targetBuyerAssets` transfer at line 197 is also reverted so funds are not permanently frozen, but the taker's intended purchase is completely blocked for that bundle invocation. This constitutes a permanent, costless, structural DoS of the bundler function for any market with non-zero settlement fee. [8](#0-7) 

## Likelihood Explanation

The precondition requires only: (1) a sell offer at `tick = MAX_TICK` exists in the market, and (2) the market has a non-zero settlement fee. `MAX_TICK = 5820` is a valid tick accessible at any tick spacing that divides 5820 (e.g., spacing 1, 2, 3, 5, 6, 10, …). Non-zero settlement fees are standard. The condition is permanent and structural — it does not depend on market state, timing, or victim mistakes. A malicious maker posts one offer at `tick = MAX_TICK` and any taker who aggregates offers from the order book and includes it triggers the revert. The attack is free to set up and infinitely repeatable.

## Recommendation

Move the computation of `unitsToTake` inside the `try` block, or wrap each call to `buyerAssetsToUnits` and `consumableUnits` in its own `try/catch` so that a revert from either skips the offer rather than aborting the bundle. Alternatively, add a pre-check: if `tickToPrice(offer.tick) + settlementFee > WAD`, skip the offer (set `unitsToTake = 0`) rather than calling `buyerAssetsToUnits`. A third option is to add a guard in `buyerAssetsToUnits` that returns 0 instead of reverting when `buyerPrice > WAD`, consistent with the bundler's skip-on-failure contract.

## Proof of Concept

**Minimal manual steps:**

1. Deploy Midnight with a market where `settlementFee > 0`.
2. Maker posts a sell offer at `tick = MAX_TICK` (5820) with any valid `maxAssets`/`maxUnits`.
3. Taker calls `buyWithAssetsTargetAndWithdrawCollateral` with a `takes[]` array that includes the maker's offer alongside other valid offers.
4. Observe: the entire transaction reverts with `PriceGreaterThanOne()` before any `take()` is attempted.

**Fuzz/unit test plan:**

```solidity
