Audit Report

## Title
Front-Running Partial Fill Causes Exact-Equality DoS in `buyWithAssetsTargetAndWithdrawCollateral` and `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/MidnightBundles.sol)

## Summary

`buyWithAssetsTargetAndWithdrawCollateral` and `supplyCollateralAndSellWithAssetsTarget` compute `unitsToTake` using `ConsumableUnitsLib.consumableUnits`, which reads live on-chain `consumed[maker][group]` state at execution time. Any unprivileged actor who partially fills a target offer before the bundle executes reduces `consumableUnits`, causing the bundle to underfill and revert at the strict equality checks on lines 224 and 303. The victim's tokens are returned on revert, but the bundle cannot complete its intended action, and the attack is repeatable.

## Finding Description

**Root cause — live `consumed` read in `ConsumableUnitsLib.sol` lines 14–22:**
`consumableUnits` calls `IMidnight(midnight).consumed(offer.maker, offer.group)` at execution time with no snapshot from bundle submission. Any intervening `take` on the same offer increments `consumed`, reducing the returned unit capacity.

**Bundle loop caps `unitsToTake` by live capacity — `MidnightBundles.sol` lines 208–213:**
```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.buyerAssetsToUnits(..., targetFilledBuyerAssets - filledBuyerAssets),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // live state
);
```
If `consumableUnits` is reduced by a front-run, `unitsToTake` is smaller, `take` succeeds but fills fewer assets than needed, and `filledBuyerAssets` ends below `targetFilledBuyerAssets`.

**Strict equality check — line 224:**
```solidity
require(filledBuyerAssets == targetFilledBuyerAssets, OutOfOffers());
```
The `try/catch` on lines 215–221 only handles reverts from `take`; it does not handle the case where `take` succeeds but fills fewer assets. There is no slippage tolerance or minimum-fill threshold.

**Identical pattern — `supplyCollateralAndSellWithAssetsTarget` line 303:**
```solidity
require(filledSellerAssets == targetFilledSellerAssets, OutOfOffers());
```

**Exploit flow:**
1. Victim submits `buyWithAssetsTargetAndWithdrawCollateral` with offer O (`maxAssets > 0`), `targetFilledBuyerAssets = X`.
2. Attacker front-runs with a direct `Midnight.take(O, ..., smallUnits)` — permissionless.
3. `consumed[O.maker][O.group]` increases; `consumableUnits(O)` returns reduced capacity.
4. Victim's bundle executes: `unitsToTake` is capped at reduced value; `take` succeeds but `resBuyerAssets < X`.
5. Loop ends; `filledBuyerAssets < targetFilledBuyerAssets`; `require` at line 224 reverts.

**Why existing checks fail:**
- `try/catch` only catches reverts, not successful-but-underfilled `take` calls.
- No snapshot of `consumed` at bundle submission time.
- No fallback fill path or slippage tolerance.
- Additionally, `buyWithAssetsTargetAndWithdrawCollateral` pulls `targetBuyerAssets` at line 197 but has no refund path for unused tokens (line 239 only sends referral fee), meaning the strict equality check is the sole guard against token loss — relaxing it requires simultaneously adding a refund.

## Impact Explanation

Any unprivileged actor can repeatedly grief a victim's `buyWithAssetsTargetAndWithdrawCollateral` or `supplyCollateralAndSellWithAssetsTarget` bundle by front-running with a minimal partial fill of the target offer. The victim's transaction reverts with `OutOfOffers` every time, permanently blocking the bundle from executing as long as the attacker repeats the front-run. This constitutes sustained, repeatable denial-of-service against a specific user's bundle execution, matching the "service unavailability or severe degradation under realistic attacker input" impact class in RESEARCHER.md.

## Likelihood Explanation

Taking an offer is permissionless — any address can call `Midnight.take`. The attacker needs only enough loan tokens to take 1 unit from the target offer, receiving a market credit position in return, so the net cost is gas plus any bid-ask spread. The attack is repeatable every block with no cooldown. The only natural limit is the offer's `maxAssets` being fully consumed, but at that point the victim's bundle also cannot use the offer. Preconditions are minimal: victim uses an assets-target bundle with an offer that has `maxAssets > 0` and insufficient fallback offers.

## Recommendation

1. Replace the strict equality checks at lines 224 and 303 with `>=`:
   ```solidity
   require(filledBuyerAssets >= targetFilledBuyerAssets, OutOfOffers());
   ```
2. Add a refund path for unused tokens in `buyWithAssetsTargetAndWithdrawCollateral` (since `targetBuyerAssets` is pulled upfront at line 197 but only `filledBuyerAssets + referralFeeAssets` is consumed):
   ```solidity
   SafeTransferLib.safeTransfer(loanToken, msg.sender, targetBuyerAssets - filledBuyerAssets - referralFeeAssets);
   ```
3. Apply the symmetric fix in `supplyCollateralAndSellWithAssetsTarget` at line 303.
4. Consider snapshotting `consumed` at bundle submission time or using a minimum-fill threshold parameter to give users explicit slippage control.

## Proof of Concept

**Minimal fork test plan:**
1. Deploy Midnight and MidnightBundles on a local fork.
2. Create a sell offer O with `maxAssets = 1000`, `group = G`.
3. Victim calls `buyWithAssetsTargetAndWithdrawCollateral` with `takes = [O]`, `targetBuyerAssets` corresponding to full fill.
4. Before victim's tx lands, attacker calls `Midnight.take(O, ..., 1_unit)` — `consumed[O.maker][G]` increments.
5. Victim's tx executes: `consumableUnits(O)` returns reduced value; `unitsToTake` is smaller; `filledBuyerAssets < targetFilledBuyerAssets`; tx reverts with `OutOfOffers`.
6. Repeat step 4–5 to demonstrate sustained DoS.
