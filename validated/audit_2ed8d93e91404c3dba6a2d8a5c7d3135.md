### Title
Fee-on-Transfer Collateral Token Inflates Position Accounting in `supplyCollateral` — (File: src/Midnight.sol)

### Summary

`supplyCollateral` credits the caller's position with the full `assets` parameter before performing the token transfer. If the collateral token silently deducts a fee during `transferFrom`, the contract receives `assets - fee` but records `assets` in the position. Because markets are immutable after creation and there is no mechanism to disable a collateral, this accounting gap persists for the lifetime of the market, enabling over-borrowing and eventual bad debt.

### Finding Description

In `supplyCollateral`, the position is updated with the caller-supplied `assets` value on line 533, and the token pull happens on line 545:

```solidity
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets); // line 533
...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545
``` [1](#0-0) 

`SafeTransferLib.safeTransferFrom` only verifies that the call did not revert and that the return value (if any) is `true`. It does not verify that the contract's balance increased by exactly `assets`. [2](#0-1) 

The protocol documents a token safety requirement that "Midnight must send/receive exactly the requested amount on transfers," but this is a comment-level assumption with no on-chain enforcement. [3](#0-2) 

Markets are created permissionlessly via `touchMarket` and their `collateralParams` are immutable once set. There is no function to disable a market or remove a collateral entry. [4](#0-3) 

The `live_context.json` core invariants section explicitly lists "ERC20 transfer deltas must match accounting deltas" and notes that "fee-on-transfer, rebasing, false-return, ERC777-like hooks, and non-standard decimals should be tested if not explicitly excluded," confirming this is an in-scope concern. [5](#0-4) 

**Exploit path:**

1. A market is created with collateral token C (e.g., USDT with fee currently 0).
2. Token C later activates a fee-on-transfer (e.g., 1%).
3. Borrower calls `supplyCollateral(market, idx, 1000e6, borrower)`.
4. `_position.collateral[idx]` is set to `oldCollateral + 1000e6`.
5. The contract only receives `990e6` (after 1% fee).
6. The health check in `isHealthy` / `withdrawCollateral` / `take` uses the inflated `1000e6` value to compute `maxDebt`, allowing the borrower to take on more debt than the actual collateral backing supports.
7. On liquidation, `safeTransfer(collateralToken, receiver, seizedAssets)` attempts to send `seizedAssets` tokens, but the contract may not hold enough, causing the liquidation to revert or leaving lenders with unrecoverable bad debt. [6](#0-5) 

### Impact Explanation

The inflated collateral balance is used directly in `isHealthy` to compute `maxDebt` (via oracle price × LLTV). A borrower can borrow more loan tokens than their real collateral supports. When the position is eventually liquidated, the contract may be unable to transfer the recorded `seizedAssets` because it holds fewer tokens than the position ledger claims. This creates bad debt that is socialized among lenders via the `lossFactor` mechanism, constituting direct loss of lender funds and protocol insolvency.

### Likelihood Explanation

Market creation is permissionless; any token can be used as collateral. Tokens like USDT have a fee-on-transfer switch that is currently disabled but can be activated by the token issuer at any time. Once activated, every subsequent `supplyCollateral` call for that token in any market that includes it will silently inflate the position. Because markets are immutable and there is no collateral-disable mechanism, the protocol cannot stop accepting the token or correct existing positions.

### Recommendation

Replace the static `assets` credit with a balance-delta measurement:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

This ensures the position always reflects the actual amount held by the contract, regardless of token fee behavior.

### Proof of Concept

1. Deploy a mock ERC-20 collateral token with a toggleable 1% fee-on-transfer.
2. Create a Midnight market using this token as collateral (fee initially off).
3. Enable the fee on the token.
4. Call `midnight.supplyCollateral(market, 0, 1000e18, borrower)`.
5. Assert: `midnight.collateral(id, borrower, 0) == 1000e18` (inflated).
6. Assert: `collateralToken.balanceOf(address(midnight))` increased by only `990e18`.
7. The gap of `10e18` allows the borrower to borrow against phantom collateral, and any liquidation attempting to seize the full `1000e18` will revert or drain other users' collateral.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
