### Title
Lender Can Front-Run Bad Debt Realization to Escape Proportional Loss Socialization — (File: `src/Midnight.sol`)

---

### Summary

Bad debt socialization in `Midnight.sol` is applied **lazily**: `lossFactor` is only updated inside `liquidate()` when a borrower's bad debt is realized. The `withdraw()` function applies slashing only against the **current** (potentially stale) `lossFactor` via `_updatePosition`. A lender who monitors the chain for insolvent borrowers can front-run the liquidation transaction, withdrawing at the pre-slashing rate and escaping their proportional share of the loss entirely at the expense of other lenders.

---

### Finding Description

**Vulnerability type**: State transition / timing / front-running

**Root cause — lazy `lossFactor` update**

`lossFactor` is only mutated inside `liquidate()` when `badDebt > 0`: [1](#0-0) 

Until `liquidate()` is called for an insolvent borrower, `marketState[id].lossFactor` remains at its old value. Every lender's stored `position.lastLossFactor` therefore equals the global `lossFactor`, so `_updatePosition` computes zero slashing: [2](#0-1) 

**Exploit path through `withdraw()`**

`withdraw()` calls `_updatePosition` before processing the withdrawal: [3](#0-2) 

Because `_updatePosition` reads the current (stale) `lossFactor`, the lender's credit is **not reduced** before the subtraction on line 493. The lender withdraws at a 1:1 ratio (1 credit unit = 1 loan token) even though, once the pending bad debt is realized, each credit unit will be worth less than 1 token.

**No guard prevents this**: `withdraw()` has no check for unhealthy borrowers in the market, no maturity restriction, and no cooldown. The only constraint is `units ≤ min(credit, withdrawable, totalUnits)`, all of which are satisfied when `withdrawable > 0`.

The protocol itself acknowledges the race to withdraw: [4](#0-3) 

However, that comment addresses the benign race after a repayment or liquidation has already occurred — not the malicious race of escaping bad debt *before* it is realized.

---

### Impact Explanation

**Concrete scenario**:

| State | `totalUnits` | `withdrawable` | credit[A] | credit[B] |
|---|---|---|---|---|
| Initial | 200 | 100 (from Borrower D's repayment) | 100 | 100 |
| After Lender A front-runs `withdraw(100)` | 100 | 0 | 0 | 100 |
| After `liquidate(C, badDebt=100)` | 0 | 0 | 0 | 0 (slashed) |

**Fair outcome** (no front-running): each lender receives 50 tokens.
**Actual outcome**: Lender A receives 100 tokens; Lender B receives 0 tokens.

Lender A extracts 50 tokens that should have been Lender B's. In a market with many lenders, a single attacker can drain the entire `withdrawable` pool before bad debt is realized, forcing all remaining lenders to absorb 100% of the loss instead of their proportional share. This is a direct, quantifiable theft of value from other protocol participants.

---

### Likelihood Explanation

Three conditions must hold simultaneously:

1. **`withdrawable > 0`** — any repayment or partial liquidation in the market satisfies this; realistic in any active market.
2. **An insolvent borrower exists but has not yet been liquidated** — the window between a price drop making a position unhealthy and a liquidation bot's transaction being included is typically 1–several blocks, which is sufficient.
3. **The attacker acts before the liquidation** — a lender monitoring the mempool or on-chain state can submit a `withdraw()` with a higher gas price to front-run the liquidation bot. No privileged access is required; this is a standard MEV/front-running technique available to any EOA or contract.

This is a realistic, no-privilege attack executable by any sophisticated lender.

---

### Recommendation

The core fix is to ensure that bad debt is accounted for **before** a lender can withdraw at the pre-loss rate. Options in increasing order of complexity:

1. **Require explicit bad-debt realization before withdrawal**: Before allowing `withdraw()`, check that no borrower in the market is currently liquidatable (i.e., `originalDebt > maxDebt` or `block.timestamp > maturity`). This is gas-intensive but fully closes the window.

2. **Atomic bad-debt sweep on `withdraw()`**: Allow anyone to pass a list of insolvent borrowers to `withdraw()` so their bad debt is realized atomically in the same transaction, updating `lossFactor` before the caller's credit is evaluated.

3. **Two-step withdrawal with a signal delay** (closest to the external report's recommendation): Require lenders to signal intent to withdraw at least one epoch in advance, analogous to the RPL fix. This ensures that any bad debt that materializes during the delay period is applied before the withdrawal is processed.

### Proof of Concept

**Setup**:
- Market with `loanToken = T`, two lenders A and B each with `credit = 100`, `totalUnits = 200`.
- Borrower D has already repaid 100 units → `withdrawable = 100`.
- Borrower C has `debt = 100`, zero collateral (fully insolvent). `liquidate()` has not yet been called.

**Attack**:
```
// Step 1 — Lender A sees Borrower C is insolvent (e.g., via eth_call to isHealthy)
// Step 2 — Lender A submits withdraw() with higher gas than the pending liquidation tx
midnight.withdraw(market, 100, lenderA, lenderA);
  // _updatePosition: lossFactor unchanged → credit[A] stays 100, no slashing
  // credit[A] = 0, withdrawable = 0, totalUnits = 100
  // Lender A receives 100 T tokens

// Step 3 — Liquidation bot's tx lands (or anyone calls liquidate)
midnight.liquidate(market, 0, 0, 0, borrowerC, false, address(0), address(0), "");
  // badDebt = 100, totalUnits = 0, lossFactor maxed out
  // credit[B] will be slashed to 0 at next interaction

// Step 4 — Lender B calls withdraw() → reverts or receives 0
// withdrawable = 0, credit[B] (post-slash) = 0
```

**Expected fair outcome**: Lender A = 50 T, Lender B = 50 T.
**Actual outcome**: Lender A = 100 T, Lender B = 0 T.

Lender A stole 50 T from Lender B with no privileged access, using only a standard front-run.
