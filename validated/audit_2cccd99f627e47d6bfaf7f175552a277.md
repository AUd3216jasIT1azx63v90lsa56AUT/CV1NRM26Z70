Audit Report

## Title
Debt increase permitted at exact maturity timestamp via `<=` boundary — (File: `src/Midnight.sol`)

## Summary
The maturity guard in `Midnight.take` at line 391 uses `block.timestamp <= offer.market.maturity`, which permits `sellerDebtIncrease > 0` when `block.timestamp == maturity`. The liquidation guard at line 622 uses the strict `block.timestamp > market.maturity` for post-maturity mode, creating a one-second asymmetry: debt can be created at exactly `T`, but post-maturity liquidation becomes available at `T+1` for any borrower with debt, regardless of health. This directly violates the protocol's own stated invariant in `live_context.json` line 221.

## Finding Description
**Root cause — asymmetric boundary operators:**

`src/Midnight.sol` line 391:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
At `block.timestamp == T` (maturity), `T <= T` is `true`, so the require passes unconditionally. `sellerDebtIncrease` is computed at line 384 and written to state at line 414:
```solidity
uint256 sellerDebtIncrease = units - sellerCreditDecrease;
...
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
```

`src/Midnight.sol` lines 620–622:
```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```
At `T`, `T > T` is `false` → post-maturity liquidation blocked. At `T+1`, `T+1 > T` is `true` → any borrower with debt is liquidatable regardless of health.

**Protocol invariant explicitly violated:**

`live_context.json` line 221 reads: `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"` — naming this exact case. `live_context.json` line 86 also states `"debt_increase_after_maturity_forbidden": true`.

**Exploit flow:**
1. A market with `maturity = T` is public on-chain. Attacker (or cooperating validator) targets the block with `block.timestamp == T`.
2. Attacker submits `take` with a sell offer where `sellerDebtIncrease > 0`. The check `T <= T` passes; debt is written to the seller's position.
3. At `T+1`, any liquidator calls `liquidate(..., postMaturityMode=true)`. The check `T+1 > T` passes. The position is liquidatable even if fully collateralized.
4. LIF grows from ~1 at `T+1` to `maxLif` over 15 minutes, at which point the liquidator seizes collateral at full incentive from a position that was healthy at creation.

If the victim is a maker (seller) whose offer is taken at exactly `T` by a front-running taker, the victim had no ability to prevent the take and their collateral becomes seizable within 15 minutes.

## Impact Explanation
Unauthorized collateral seizure from a healthy position. A borrower whose debt is created at exactly `T` is immediately subject to post-maturity liquidation at `T+1` with no health-based protection. This falls directly under the protocol's listed best bug class: `"unauthorized collateral seizure"` and `"healthy-account liquidation"`. The borrower suffers a loss of collateral from a position that should never have had debt created at that timestamp per the protocol's own invariant.

## Likelihood Explanation
Requires `block.timestamp == market.maturity` exactly. Market maturities are public on-chain parameters, making the target timestamp known in advance. On proof-of-stake Ethereum, validators control block timestamps within the allowed drift window (~12 seconds), making exact timestamp targeting feasible for a validator-attacker. For non-validators, the condition requires a transaction landing in the exact maturity block, which is probabilistically achievable by monitoring the mempool and submitting with appropriate gas. The condition is repeatable across any market whose maturity falls on a future block.

## Recommendation
Change the `<=` to `<` in the maturity guard in `take`:

```solidity
// Before
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This makes both guards use strict inequality (`<` for debt creation, `>` for post-maturity liquidation), eliminating the one-second overlap. At `T`, debt creation is blocked; at `T+1`, post-maturity liquidation is available — consistent with the invariant.

## Proof of Concept
Minimal Foundry test:
```solidity
function testDebtCreationAtExactMaturity() public {
    // Setup: create market with maturity = T
    uint256 T = block.timestamp + 1 days;
    // ... market setup ...
    
    // Warp to exactly T
    vm.warp(T);
    
    // Take a sell offer — should revert but does not
    // sellerDebtIncrease > 0 is written to state
    midnight.take(...); // expect revert CannotIncreaseDebtPostMaturity, but passes
    
    // Warp to T+1
    vm.warp(T + 1);
    
    // Post-maturity liquidation succeeds on a healthy position
    midnight.liquidate(..., postMaturityMode=true); // succeeds — unauthorized seizure
}
```
The existing test `testLiquidatePostMaturityModeBeforeMaturity` confirms that at `block.timestamp == maturity`, post-maturity liquidation reverts — but no existing test verifies that debt creation also reverts at that same timestamp, leaving the boundary untested for the `take` path.
