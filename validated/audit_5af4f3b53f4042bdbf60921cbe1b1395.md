## Title
Oracle Price Manipulated via `onSell` Callback Allows Undercollateralized Borrowing - (File: src/Midnight.sol)

## Summary
In `take()`, the seller's debt is committed to storage at line 414 before the `onSell` callback fires at lines 458–474 via an unrestricted `CALL`. The health check at line 476 invokes `isHealthy`, which queries `IOracle.price()` after the callback completes. Because the callback can write to any external contract, a malicious oracle that reads from attacker-controlled storage will return an inflated price during the health check, causing `isHealthy` to pass with fabricated collateral value and leaving the seller with undercollateralized debt. Lenders bear the resulting bad debt.

## Finding Description

**Exact code path:**

1. `take()` is called at `src/Midnight.sol:337`.
2. `sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease)` at line 414 — seller's debt is committed to storage before any callback.
3. Liquidation lock is set for the seller at line 444 via `UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`.
4. Loan tokens are transferred to the receiver at line 456 — the seller has already received `sellerAssets` before the health check.
5. `onSell` callback fires at lines 458–474 via a regular `CALL`. The callback can write to any external contract, including an attacker-controlled `StateContract`.
6. Liquidation lock is released at line 475.
7. `require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable())` at line 476.
8. Inside `isHealthy` at line 953: `uint256 price = IOracle(collateralParam.oracle).price()` — called via `staticcall` because `price()` is declared `external view` in `src/interfaces/IOracle.sol:6`. The `staticcall` cannot write state but freely reads state already written by the callback. The oracle reads from the `StateContract` whose storage was written in step 5, returning an inflated price.
9. `maxDebt` is computed with the inflated price at lines 954–955 and exceeds the seller's actual debt — health check passes.

**Root cause:** The protocol fires an unrestricted external `CALL` (`onSell`) between committing the seller's debt increase and evaluating the health check. No snapshot of the oracle price is taken before the callback. Any state written by the callback is visible to the oracle when `isHealthy` is called.

**Why existing checks fail:**
- The liquidation lock (lines 444, 475) only prevents liquidation during callbacks; it does not prevent oracle price manipulation.
- `staticcall` on `price()` prevents the oracle from writing state during the health check, but the callback already wrote the manipulated value before `isHealthy` is called. `staticcall` can freely read state written earlier in the same transaction.
- `touchMarket` at lines 755–791 validates only `lltv`, `maxLif`, collateral token ordering, and maturity — it does not validate oracle behavior, so any address can be supplied as the oracle.
- The Certora formal verification at `certora/specs/Healthiness.spec:14–16` explicitly assumes `price()` is constant during execution and does not cover this case. The `stayHealthy` rule at line 260 explicitly excludes `take` from its coverage.

## Impact Explanation
The seller can borrow an arbitrary number of loan tokens backed by negligible collateral. After `take()` completes, the oracle returns to its real price, leaving the seller's position undercollateralized. Lenders bear the resulting bad debt. This is direct theft of lender funds and violates the core protocol invariant that every debt position must be backed by sufficient collateral at the real oracle price.

## Likelihood Explanation
All preconditions are fully attacker-controlled and require no privileged access. Anyone can call `touchMarket` with a `CollateralParams.oracle` pointing to a malicious oracle that reads from an attacker-controlled `StateContract`. The seller controls `offer.callback` (when `offer.buy == false`) or `takerCallback` (when `offer.buy == true`). The attacker deploys three contracts (`StateContract`, malicious oracle, malicious callback), creates a market, and creates or takes an offer. No governance, admin, or leaked-key access is needed. `SECURITY.md:26` explicitly states oracle manipulation attacks are not excluded. The attack is repeatable across any number of takes.

## Recommendation
Snapshot the oracle price(s) before any external callback is invoked and use the snapshot inside `isHealthy` rather than re-querying the oracle after the callback. Alternatively, call `isHealthy` using a `staticcall` context that encompasses the entire post-debt-commit section, or restructure `take()` so that the health check precedes all external calls (with debt changes applied atomically after). A simpler mitigation is to read and cache all oracle prices needed for the health check before the liquidation lock is set at line 444, then pass those cached values into the health check at line 476.

## Proof of Concept

Deploy three contracts:
1. `StateContract` — stores a `uint256 inflatedPrice`; exposes `setPrice(uint256)` and `getPrice() view`.
2. `MaliciousOracle` — implements `IOracle`; `price()` reads and returns `StateContract.getPrice()`.
3. `MaliciousCallback` — implements `ISellCallback`; `onSell(...)` calls `StateContract.setPrice(INFLATED)` and returns `CALLBACK_SUCCESS`.

Steps:
1. Deploy all three contracts. Set `StateContract.inflatedPrice` to a real price initially.
2. Call `touchMarket` with `CollateralParams.oracle = address(MaliciousOracle)` and minimal real collateral deposited.
3. Create a sell offer (`offer.buy = false`) with `offer.callback = address(MaliciousCallback)` and `offer.maker = attacker`.
4. Have a counterparty (or a second attacker account) call `take()` on the offer.
5. During `onSell`, `MaliciousCallback` calls `StateContract.setPrice(type(uint256).max / 2)`.
6. `isHealthy` at line 476 calls `MaliciousOracle.price()` which returns the inflated value — health check passes.
7. After the transaction, call `StateContract.setPrice(realPrice)`. The attacker's position is now undercollateralized at the real price.
