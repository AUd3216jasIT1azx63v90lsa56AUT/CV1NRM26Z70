Audit Report

## Title
Authorized Delegate Can Perpetually Front-Run Revocations via EcrecoverAuthorizer Delegate-Signing Loop - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts signatures from any address already authorized in Midnight on behalf of the `authorizer`, not just the `authorizer` itself. An attacker controlling an authorized operator can perpetually consume every revocation nonce the victim broadcasts by front-running with a fresh re-grant signed by the operator key. A smart contract victim with no direct `Midnight.setIsAuthorized` call path is permanently unable to revoke the operator's authorization.

## Finding Description
**Root cause — `src/periphery/EcrecoverAuthorizer.sol` lines 33–36:**

```solidity
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
``` [1](#0-0) 

Any address `signer` for which `isAuthorized[victim][signer] == true` in Midnight can produce a valid `Authorization` struct with `authorizer = victim` and any future nonce. This creates a circular dependency: granting authorization to an operator gives that operator the power to sign further authorizations on the victim's behalf, including re-grants that consume the exact nonce the victim is attempting to use for revocation.

**Nonce mechanism — line 26:**

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
``` [2](#0-1) 

Nonces are strictly sequential. Consuming nonce `N` with a re-grant forces the victim's revocation (also targeting nonce `N`) to fail with `InvalidNonce`.

**Exploit path:**

1. Victim (smart contract, no direct `Midnight.setIsAuthorized` call path) previously authorized `operator` via `EcrecoverAuthorizer`. State: `isAuthorized[victim][operator] = true`, `nonce[victim] = 1`.
2. Victim signs `Authorization{authorizer=victim, authorized=operator, isAuthorized=false, nonce=1, deadline=T}` and broadcasts.
3. Attacker (controlling `operator`) observes the pending tx in the mempool. Attacker signs `Authorization{authorizer=victim, authorized=operator, isAuthorized=true, nonce=1, deadline=T+1}` with `operator`'s key. The line 34 check passes because `isAuthorized[victim][operator] == true`. Attacker front-runs: `nonce[victim] = 2`, authorization remains `true`.
4. Victim's revocation lands with `nonce=1`, fails `InvalidNonce`.
5. Steps 2–4 repeat for every nonce increment. The attacker needs only one EOA key and a mempool watcher.

**Why existing checks are insufficient:**
- The nonce check (line 26) prevents replay of the *same* signature but does not prevent the attacker from consuming the *next* nonce with a freshly signed re-grant. [2](#0-1) 
- The deadline check (line 25) is irrelevant; the attacker can use any future deadline. [3](#0-2) 
- `Midnight.setIsAuthorized` (lines 731–735) would allow direct revocation when `onBehalf == msg.sender`, but this requires the victim contract to have a code path that calls it directly. [4](#0-3) 

## Impact Explanation
The operator retains permanent `isAuthorized[victim][operator] = true` in Midnight. With this authorization the operator can call `withdraw`, `withdrawCollateral`, `repay`, `take`, `setConsumed`, and any other `onBehalf`-gated entry point on behalf of the victim. [4](#0-3)  The victim's funds (collateral, withdrawable credit, pending fees) are fully accessible to the attacker with no on-chain revocation path available, constituting a permanent fund drain / fund freeze. This is a **Critical** severity impact: direct theft of assets and permanent loss of access control.

## Likelihood Explanation
**Required preconditions:**
1. Victim is a smart contract whose code has no direct call to `Midnight.setIsAuthorized` — realistic for protocol integrations, smart-contract wallets, or multisigs that route all Midnight interaction through `EcrecoverAuthorizer`.
2. Victim signed at least one `isAuthorized=true` authorization and the attacker obtained the signature (public broadcast, failed tx, or mempool observation).
3. Attacker controls the `authorized` address (operator) and can sign new `Authorization` structs with it.

All three preconditions require no privileged access, oracle manipulation, or user mistake beyond the initial signature issuance. The front-running loop is repeatable every block at negligible gas cost for the attacker. MEV infrastructure makes same-block front-running trivially reliable.

## Recommendation
Remove the delegate-signing capability from `EcrecoverAuthorizer` entirely, requiring only the authorizer's direct signature:

```solidity
require(signer == authorization.authorizer, Unauthorized());
```

This eliminates the circular dependency. If delegate-signing is a desired feature, restrict it to `isAuthorized=true` grants only, and require the authorizer's direct signature for any revocation (`isAuthorized=false`):

```solidity
require(
    signer == authorization.authorizer
        || (authorization.isAuthorized && IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)),
    Unauthorized()
);
``` [1](#0-0) 

## Proof of Concept
**Minimal Foundry test outline:**

1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. Create `victim` as a contract with no `setIsAuthorized` call path.
3. Sign `Authorization{authorizer=victim, authorized=operator, isAuthorized=true, nonce=0}` with `victim`'s key; call `EcrecoverAuthorizer.setIsAuthorized`. Assert `isAuthorized[victim][operator] == true`.
4. Sign `Authorization{authorizer=victim, authorized=operator, isAuthorized=false, nonce=1}` with `victim`'s key (revocation).
5. Before submitting the revocation, sign `Authorization{authorizer=victim, authorized=operator, isAuthorized=true, nonce=1}` with `operator`'s key; call `EcrecoverAuthorizer.setIsAuthorized`. Assert success and `nonce[victim] == 2`.
6. Submit the victim's revocation from step 4. Assert it reverts with `InvalidNonce`.
7. Assert `isAuthorized[victim][operator] == true` still holds.
8. Repeat steps 4–7 for nonces 2, 3, … to demonstrate the loop is unbounded. [5](#0-4)

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
```

        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());

        emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
