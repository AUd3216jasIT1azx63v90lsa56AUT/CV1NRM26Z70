Audit Report

## Title
Front-running `repay` with partial liquidation causes arithmetic underflow DoS - (`src/Midnight.sol`)

## Summary
The `repay` function subtracts `units` from `position[id][onBehalf].debt` with no guard that `units ≤ debt`. An attacker can front-run any `repay(market, D, …)` call with a partial `liquidate` that reduces the borrower's debt below `D`, causing the repay to revert with a Solidity 0.8 checked-arithmetic underflow. The attacker earns the liquidation incentive on every front-run, making the attack self-funding and repeatable.

## Finding Description

**Root cause — missing bounds check in `repay`:** [1](#0-0) 

`position[id][onBehalf].debt -= UtilsLib.toUint128(units)` performs a checked subtraction on a `uint128` storage field. There is no `require(units <= position[id][onBehalf].debt)` guard anywhere in the function. [2](#0-1) 

`UtilsLib.toUint128` only guards against cast overflow (`x <= type(uint128).max`); it does not enforce `units ≤ debt`.

**State-modifying paths in `liquidate` that write to the same slot:** [3](#0-2) [4](#0-3) 

Both execute before any external call, so the debt reduction is committed to storage before the borrower's pending `repay` is included.

**Exploit flow:**
1. Borrower has unhealthy position: `debt = D`, position is liquidatable.
2. Borrower broadcasts `repay(market, D, onBehalf, …)`.
3. Attacker sees the transaction in the mempool and front-runs with `liquidate(…, repaidUnits=1, borrower, …)`.
4. Liquidation succeeds: `_position.debt` is now `D − 1`.
5. Borrower's `repay` lands: `(D−1) − D` underflows → **revert**.
6. Attacker received collateral worth `1 * lif / WAD` in step 3, paying only 1 loan token — net profit from the liquidation incentive.
7. Attacker repeats on every retry, continuously front-running with `repaidUnits = 1`.

**Why existing checks do not stop it:**

The only check in `repay` is the authorization guard: [5](#0-4) 

This is irrelevant to the underflow. The `liquidatorGate` check in `liquidate`: [6](#0-5) 

…is only present when the market creator sets a non-zero gate; permissionless markets have `liquidatorGate == address(0)`, bypassing this guard entirely. The `liquidationLocked` check uses transient storage scoped to a single transaction's callback reentrancy — it does not prevent a separate front-running transaction.

**Test gap confirmed:** [7](#0-6) 

`testRepay` always bounds `repaid ≤ units`, so the underflow path is never exercised by the existing test suite.

## Impact Explanation
Any borrower whose position is unhealthy (liquidatable) can be permanently denied the ability to repay via `repay`. Every `repay(market, D, onBehalf, …)` call reverts as long as the attacker front-runs with even a 1-unit liquidation. The borrower is forced into full liquidation, losing the liquidation incentive premium on their collateral rather than repaying at par. This constitutes a concrete, repeatable DoS against the `repay` function for any unhealthy borrower in any market without a `liquidatorGate`, with direct financial loss to the victim.

## Likelihood Explanation
**Preconditions:** (a) borrower's position is unhealthy (price dropped below LLTV threshold), (b) market has no `liquidatorGate` (the default for permissionless markets). Both are routine on-chain states. **Feasibility:** front-running is standard on EVM chains with a public mempool; the attacker needs only to submit a higher-gas-price `liquidate` with `repaidUnits = 1`. **Profitability:** the attacker earns the liquidation incentive (`lif > WAD`) on every front-run, so the attack is net-positive and self-funding. **Repeatability:** the attack can be repeated on every retry until the borrower's collateral is fully seized. Front-running is not excluded by `SECURITY.md`.

## Recommendation
Add an explicit bounds check in `repay` before the subtraction:

```solidity
require(units <= position[id][onBehalf].debt, RepayExceedsDebt());
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
```

Alternatively, support a sentinel value (e.g., `type(uint256).max`) to mean "repay full current debt", computed atomically from storage, so the borrower's intent cannot be invalidated by a concurrent liquidation.

## Proof of Concept
```solidity
// Foundry fork test outline
function testFrontRunRepayDoS() public {
    // 1. Setup: borrower has debt D in an unhealthy position (price dropped)
    uint256 D = position[id][borrower].debt;

    // 2. Attacker front-runs with a 1-unit liquidation
    vm.prank(attacker);
    midnight.liquidate(market, collateralIndex, 0, 1, borrower, false, attacker, address(0), "");
    // _position.debt is now D - 1

    // 3. Borrower's repay(D) now underflows and reverts
    vm.prank(borrower);
    vm.expectRevert(); // arithmetic underflow
    midnight.repay(market, D, borrower, address(0), "");
}
```
The test confirms that after a 1-unit liquidation, `repay(D)` reverts. Repeating step 2 before every retry demonstrates the sustained DoS.
