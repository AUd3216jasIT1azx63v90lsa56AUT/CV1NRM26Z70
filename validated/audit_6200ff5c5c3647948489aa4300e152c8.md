### Title
`ConfidencePool.initialize()` Callable By Anyone, Bypassing Factory Token Allowlist and Agreement-Owner Authorization — (File: src/ConfidencePool.sol)

### Summary
`ConfidencePool.initialize()` has no access control restricting it to the factory. Anyone can deploy a minimal proxy pointing to `poolImplementation` and call `initialize()` directly, bypassing the two critical validations that `ConfidencePoolFactory.createPool()` enforces: the stake-token allowlist check and the agreement-owner authorization check.

### Finding Description
`ConfidencePoolFactory.createPool()` enforces two validations before initializing a pool clone:

1. **Token allowlist** — [1](#0-0)  `if (!allowedStakeToken[stakeToken]) revert StakeTokenNotAllowed();`
2. **Agreement-owner authorization** — [2](#0-1)  `if (IAgreement(agreement).owner() != msg.sender) revert UnauthorizedCreator();`

`ConfidencePool.initialize()` carries neither of these guards: [3](#0-2) 

The function is declared `external initializer` with no `onlyFactory` or equivalent modifier. Since `ConfidencePool` is a clone implementation, any caller can:

1. Deploy a minimal proxy pointing to the public `poolImplementation` address (via `Clones.clone(poolImplementation)` or any EIP-1167 factory).
2. Call `initialize()` on that proxy with fully attacker-controlled parameters, including a non-allowlisted `stakeToken_`, themselves as `outcomeModerator_`, themselves as `owner_`, and themselves as `recoveryAddress_`.

The only agreement-related checks that survive in `initialize()` are `safeHarborRegistry.isAgreementValid(agreement_)` and `IAgreement(agreement).isContractInScope(account)` inside `_replaceScope` — neither checks agreement ownership. [4](#0-3) 

### Impact Explanation
An attacker who creates a rogue pool gains:

- **Arbitrary stake token**: the factory allowlist exists specifically because fee-on-transfer and rebasing tokens break the pool's balance accounting, causing under-payment on every claim/withdraw. A malicious token (e.g., one with a backdoor `transferFrom`) can drain depositors immediately. [5](#0-4) 
- **Self-appointed `outcomeModerator`**: the attacker can call `flagOutcome()` at will once the registry reaches a terminal state, directing the resolution outcome. Under bad-faith CORRUPTED, the entire pool (stake + bonus) sweeps to `recoveryAddress`. [6](#0-5) 
- **Self-appointed `owner_` / `recoveryAddress`**: the attacker controls `sweepUnclaimedBonus()` destination and can update `recoveryAddress` at any time before outcome. [7](#0-6) 

The rogue pool is indistinguishable from a factory-deployed pool on-chain: it holds real funds, emits the same events, and resolves through the same logic. The factory's `_poolsByAgreement` mapping will not list it, but off-chain tooling or a misleading UI can direct stakers to it. Stakers who deposit cannot withdraw once `riskWindowStart` is set, trapping their principal. [8](#0-7) 

### Likelihood Explanation
The attack requires deploying a clone (one transaction, no special privilege) and calling `initialize()` (one transaction). The only friction is social engineering stakers into using the rogue pool. Given that the pool address is not inherently tied to the factory registry, a phishing site or a misleading integration can plausibly direct users to it. The attacker does not need to own the agreement or hold any privileged role in the protocol.

### Recommendation
Add an `onlyFactory` access control check to `initialize()`. The simplest approach is to store the factory address in the implementation (set once in the constructor or via an immutable) and revert in `initialize()` if `msg.sender != factory`. Alternatively, the factory can call a separate `initializeFromFactory()` entry point that is gated by `msg.sender == factory`, while the public `initialize()` is removed or made internal.

### Proof of Concept

//    - allowedStakeToken check (maliciousToken is NOT on the allowlist)

    /// @inheritdoc IConfidencePool
    function initialize(
        address agreement_,
        address stakeToken_,
        address safeHarborRegistry_,
        address outcomeModerator_,
        uint256 expiry_,
        uint256 minStake_,
        address recoveryAddress_,
        address owner_,
        address[] calldata accounts
    ) external initializer {
        if (agreement_ == address(0)) revert ZeroAddress();
        if (stakeToken_ == address(0)) revert ZeroAddress();
        if (safeHarborRegistry_ == address(0)) revert ZeroAddress();
        if (outcomeModerator_ == address(0)) revert ZeroAddress();
        if (owner_ == address(0)) revert ZeroAddress();
        if (recoveryAddress_ == address(0)) revert InvalidRecoveryAddress();
        if (expiry_ < block.timestamp + _MIN_EXPIRY_LEAD) revert ExpiryTooSoon();
        if (expiry_ > type(uint32).max) revert ExpiryTooFar();
        if (minStake_ == 0) revert InvalidAmount();
        // aderyn-fp-next-line(reentrancy-state-change)
        if (!IBattleChainSafeHarborRegistry(safeHarborRegistry_).isAgreementValid(agreement_)) {
            revert InvalidAgreement();
        }

        agreement = agreement_;
        stakeToken = IERC20(stakeToken_);
        safeHarborRegistry = IBattleChainSafeHarborRegistry(safeHarborRegistry_);
        outcomeModerator = outcomeModerator_;
        // forge-lint: disable-next-line(unsafe-typecast)
        expiry = uint32(expiry_);
        minStake = minStake_;
        recoveryAddress = recoveryAddress_;
        outcome = PoolStates.Outcome.UNRESOLVED;

        _replaceScope(accounts);

        // Direct assignment (skipping Ownable2Step's two-step) so no `owner() == initializer-caller`
        // window exists between init and the new owner accepting. Two-step still applies to later transfers.
        _transferOwnership(owner_);
    }
