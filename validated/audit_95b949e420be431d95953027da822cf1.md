Audit Report

## Title
`TakeAmountsLib.sellerAssetsToUnits` panics on `sellerPrice == 0` outside try/catch, reverting entire bundle for a case `Midnight.take` accepts - (`src/periphery/TakeAmountsLib.sol` / `src/periphery/MidnightBundles.sol`)

## Summary
When a buy offer's tick price equals the settlement fee exactly, `sellerPrice` computes to zero in `TakeAmountsLib.sellerAssetsToUnits`. The subsequent call to `mulDivUp(targetSellerAssets, WAD, 0)` triggers an arithmetic underflow panic at `(d - 1)` = `(0 - 1)` before any division occurs. Because this call sits outside the `try/catch` guard in `supplyCollateralAndSellWithAssetsTarget`, the panic propagates and reverts the entire bundle — including all collateral supplies — even though `Midnight.take` would accept the same offer without reverting.

## Finding Description

**Root cause — `mulDivUp` with `d = 0`:**

`UtilsLib.mulDivUp` is implemented in checked arithmetic: [1](#0-0) 

When `d = 0`, the sub-expression `(d - 1)` = `(0 - 1)` underflows under Solidity 0.8 checked arithmetic, producing panic code `0x11` before any division.

**Trigger path — `sellerAssetsToUnits` with `sellerPrice = 0`:**

`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and passes it as the divisor to `mulDivUp`: [2](#0-1) 

When `offer.buy == true` and `tickToPrice(offer.tick) == settlementFee(id, timeToMaturity)`, `sellerPrice = 0`, and `mulDivUp(..., WAD, 0)` panics.

**Why `Midnight.take` does not revert for the same input:**

`Midnight.take` computes `sellerAssets = units.mulDivDown(sellerPrice, WAD)` — dividing by `WAD`, not by `sellerPrice`. With `sellerPrice = 0` this yields `0` cleanly: [3](#0-2) 

There is no guard in `take` that rejects `sellerPrice == 0`; the NatSpec at line 331 only says "Taking buy offers with price **<** settlement fee will revert." [4](#0-3) 

**The panic is outside the `try/catch`:**

In `supplyCollateralAndSellWithAssetsTarget`, `sellerAssetsToUnits` is evaluated unconditionally before the guarded `take` call: [5](#0-4) 

The `try/catch` at lines 292–300 only guards `IMidnight.take`; the pre-computation panic propagates to the caller.

**`consumableUnits` does not panic for the same offer:**

For `offer.buy == true` and `offer.maxUnits == 0`, `consumableUnits` routes to `buyerAssetsToUnits`, which divides by `buyerPrice = sellerPrice + settlementFee = settlementFee` (non-zero): [6](#0-5) 

The panic is exclusively from the direct `sellerAssetsToUnits` call at line 286.

## Impact Explanation
The entire `supplyCollateralAndSellWithAssetsTarget` transaction reverts. All collateral supplies executed before the loop (lines 269–275) are also rolled back. The taker cannot sell units or supply collateral even though all other offers in the bundle are valid and `Midnight.take` would accept them individually. The stated design intent of the bundler is to skip offers that `take` would reject; here it panics for an offer `take` would accept, violating that invariant. [7](#0-6) 

## Likelihood Explanation
Settlement fees are interpolated linearly between breakpoints as time-to-maturity decreases. For any market with a non-zero settlement fee, there exists a moment when the interpolated fee equals `tickToPrice(t)` for some tick `t`. The protocol's own NatSpec acknowledges this: "if the settlement fee gets increased, it might implicitly cancel offers with very low price." [8](#0-7) 

A maker can create a buy offer at such a tick. Any taker who includes that offer in a `supplyCollateralAndSellWithAssetsTarget` bundle at the exact moment the condition holds will have their entire bundle reverted. The condition is transient but repeatable and can be targeted by monitoring on-chain fee state against tick prices.

## Recommendation
Guard `sellerAssetsToUnits` against `sellerPrice == 0` in one of two ways:

1. **In `TakeAmountsLib.sellerAssetsToUnits`**: add an early return of `type(uint256).max` (or `0`) when `sellerPrice == 0` for buy offers, mirroring how `Midnight.take` handles it.
2. **In `supplyCollateralAndSellWithAssetsTarget`**: wrap the entire `unitsToTake` pre-computation (lines 285–291) in a `try/catch`-equivalent (e.g., a helper that returns `0` on revert), so a panicking pre-computation is treated the same as a reverting `take` — the offer is skipped.

Option 2 is more robust because it also handles any future revert paths in `ConsumableUnitsLib` for the same function.

## Proof of Concept
1. Deploy a market with a non-zero settlement fee schedule.
2. Create a buy offer at tick `t` where `tickToPrice(t) == settlementFee(id, timeToMaturity)` at a known future timestamp.
3. At that timestamp, call `supplyCollateralAndSellWithAssetsTarget` with `takes[0]` set to that offer and `targetSellerAssets > 0`.
4. Observe the transaction reverts with arithmetic panic `0x11` at `UtilsLib.mulDivUp` line 35, despite `IMidnight.take` accepting the offer when called directly.

A Foundry fuzz test can find the triggering tick/fee pair by fuzzing `offer.tick` and `timeToMaturity` until `tickToPrice(tick) == settlementFee(id, timeToMaturity)`, then asserting that `supplyCollateralAndSellWithAssetsTarget` does not revert when `take` alone does not revert.
