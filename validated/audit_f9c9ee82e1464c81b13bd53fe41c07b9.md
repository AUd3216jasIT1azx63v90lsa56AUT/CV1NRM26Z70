Audit Report

## Title
Fee-on-Transfer Collateral Inflates Stored Collateral Without Actual Token Receipt, Enabling Uncollateralized Debt - (File: src/Midnight.sol / src/libraries/SafeTransferLib.sol)

## Summary
`supplyCollateral` writes `_position.collateral[collateralIndex]` and sets the `collateralBitmap` bit before calling `SafeTransferLib.safeTransferFrom`. `safeTransferFrom` only validates call success and the boolean return value, never the actual balance delta. A fee-on-transfer collateral token with a 100% fee rate returns `true` and does not revert, yet delivers zero tokens to Midnight. `isHealthy` then computes `maxDebt` from the phantom stored collateral, allowing the borrower to pass the health check in `take` and incur real debt with zero actual collateral.

## Finding Description
**Root cause:** `supplyCollateral` (src/Midnight.sol:533) writes `_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets)` and (src/Midnight.sol:535-537) sets the bitmap bit before the transfer at line 545. `SafeTransferLib.safeTransferFrom` (src/libraries/SafeTransferLib.sol:27-33) only checks `success` and the decoded boolean return value — no balance delta is verified. A 100%-fee token satisfies both checks while delivering zero tokens.

**Exploit flow:**
1. Attacker deploys a token whose `transferFrom` deducts 100% as a fee, returns `true`, and does not revert.
2. Attacker calls `touchMarket` with this token as `collateralParams[0].token`. `touchMarket` validates LLTV tier, maxLif, and sort order — it does not inspect token behavior.
3. Attacker calls `supplyCollateral(market, 0, 1e18, borrower)`:
   - Line 533: `_position.collateral[0] = 1e18` — stored.
   - Lines 535-537: bitmap bit 0 set.
   - Line 545: `safeTransferFrom` called — token returns `true`, 0 tokens arrive.
4. State after: `position[id][borrower].collateral[0] == 1e18`, `collateralBitmap == 1`, `token.balanceOf(midnight) == 0`.
5. Attacker calls `take(buyOffer, ..., units, borrower, ...)`:
   - Borrower incurs debt.
   - Line 476: `isHealthy` is called. It reads `_position.collateral[0] == 1e18` (src/Midnight.sol:954), multiplies by oracle price and LLTV, computes `maxDebt >= debt` → returns `true`.
   - Take succeeds; borrower receives real loan tokens.

**Why existing checks fail:**
- `safeTransferFrom` (src/libraries/SafeTransferLib.sol:27-33) checks only `success` and `returndata` boolean — no balance delta check.
- `isHealthy` (src/Midnight.sol:954-955) reads `_position.collateral[i]`, not `IERC20(token).balanceOf(address(this))`.
- The Certora `nonZeroCollateralsAreActivated` invariant proves bitmap ↔ stored-collateral consistency, not stored-collateral ↔ actual-balance consistency.
- No whitelist or token-behavior validation exists in `touchMarket` or `supplyCollateral`.

## Impact Explanation
The borrower holds real loan tokens received from the sell side of `take` and has recorded debt, but Midnight holds zero collateral tokens. The debt is permanently uncollateralized. If the borrower does not repay, lenders absorb the loss through the bad-debt socialization mechanism. This is direct theft of lender funds via pure bad debt creation with no collateral backing, violating the core protocol invariant that every borrow must have correct collateral obligation.

## Likelihood Explanation
Market creation via `touchMarket` is permissionless — any address can register a market with an arbitrary collateral token. The attacker only needs to deploy a 100%-fee token, create a market, and attract or self-fund a lender position. The attack is repeatable across any number of markets and requires no privileged access. With a self-funded lender account the attacker moves their own loan tokens but creates permanent bad debt, socializing losses to all lenders in the market.

## Recommendation
Add a balance-delta check in `supplyCollateral`: record `balanceBefore = IERC20(collateralToken).balanceOf(address(this))` before the transfer, call `safeTransferFrom`, then assert `IERC20(collateralToken).balanceOf(address(this)) - balanceBefore == assets`. Use the actual received amount (the delta) as the value stored in `_position.collateral[collateralIndex]` rather than the caller-supplied `assets` parameter. This pattern is standard for fee-on-transfer token compatibility and eliminates the phantom collateral vector entirely.

## Proof of Concept
1. Deploy `FeeToken` — a standard ERC-20 whose `transferFrom` burns 100% of `value`, emits a `Transfer` event, and returns `true`.
2. Deploy a Midnight market via `touchMarket` with `collateralParams[0].token = address(feeToken)` and a valid LLTV tier.
3. Mint `1e18` `FeeToken` to attacker; approve Midnight.
4. Call `supplyCollateral(market, 0, 1e18, borrower)`. Assert: `position[id][borrower].collateral[0] == 1e18`, `feeToken.balanceOf(midnight) == 0`.
5. Place a buy offer in the market funded with real loan tokens (attacker second account or third party).
6. Call `take(buyOffer, ..., units, borrower, ...)`. Assert: call succeeds, borrower receives loan tokens, `position[id][borrower].debt > 0`, `feeToken.balanceOf(midnight) == 0`.
7. Confirm `isHealthy(market, id, borrower) == true` despite zero actual collateral.
