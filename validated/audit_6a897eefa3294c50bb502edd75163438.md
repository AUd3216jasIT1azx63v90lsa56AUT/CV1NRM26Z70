### Title
Missing Deadline Parameter in `stake()` Allows Stale Transactions to Involuntarily Lock Stakers During Active-Risk State - (File: src/ConfidencePool.sol)

### Summary
`stake()` accepts no deadline/expiry parameter. A transaction submitted when the registry is in a safe pre-attack state can sit in the mempool and execute after the registry has transitioned to `UNDER_ATTACK`, permanently locking the staker's principal with near-zero bonus and no withdrawal path — an outcome the staker never consented to.

### Finding Description
`stake()` performs a live registry read via `_observePoolState()` and then calls `_assertDepositsAllowed()`, which blocks `PROMOTION_REQUESTED`, `PRODUCTION`, and `CORRUPTED` but explicitly permits `UNDER_ATTACK`. There is no mechanism for a caller to bound the latest acceptable execution time.

A staker who submits a `stake()` transaction while the registry is in `NOT_DEPLOYED`, `NEW_DEPLOYMENT`, or `ATTACK_REQUESTED` (all states where `withdraw()` is still open) may have their transaction delayed in the mempool due to network congestion, gas-price fluctuations, or MEV activity. If the registry transitions to `UNDER_ATTACK` during that delay:

1. `_observePoolState()` observes `UNDER_ATTACK` and calls `_markRiskWindowStart()`, setting `riskWindowStart = block.timestamp` (if not already set).
2. `_assertDepositsAllowed()` passes — `UNDER_ATTACK` is not blocked.
3. The stake is recorded with `newEntry` floored at `riskWindowStart ≈ block.timestamp`, so `(T − entry)² ≈ 0` and the staker earns near-zero bonus.
4. Because `riskWindowStart != 0`, `withdraw()` is now permanently disabled for this staker.

The staker's principal is now locked into the resolution path with no exit and negligible bonus, despite having submitted the transaction under conditions where they could have freely withdrawn.

```solidity
// stake() — no deadline parameter
function stake(uint256 amount) external nonReentrant whenPoolNotPaused {
    ...
    _assertDepositsAllowed(_observePoolState()); // UNDER_ATTACK passes here
    ...
    uint256 newEntry = block.timestamp;
    uint256 start = riskWindowStart;
    if (start != 0 && newEntry < start) newEntry = start; // entry floored at riskWindowStart ≈ now
    ...
}
```

```solidity
// withdraw() — permanently gated once riskWindowStart is set
if (
    riskWindowStart != 0
        || (state != IAttackRegistry.ContractState.NOT_DEPLOYED
            && state != IAttackRegistry.ContractState.NEW_DEPLOYMENT
            && state != IAttackRegistry.ContractState.ATTACK_REQUESTED)
) {
    revert WithdrawsDisabled();
}
```

### Impact Explanation
An unprivileged staker loses:
- **Withdrawal rights** — permanently, due to the one-way `riskWindowStart != 0` latch.
- **Bonus entitlement** — near-zero, because entry is floored at `riskWindowStart ≈ now`, collapsing `(T − entry)²` to approximately zero.
- **Principal safety** — principal is now at risk until resolution (SURVIVED/EXPIRED returns it; CORRUPTED sweeps it to `recoveryAddress`).

This is materially different from the documented "voluntary UNDER_ATTACK deposit" scenario. DESIGN.md §3 frames the UNDER_ATTACK deposit as a deliberate, informed choice: *"A staker who does not want this can read the live registry state before staking."* A stale transaction is not a deliberate choice — the staker read a safe state, submitted the transaction, and the state changed beneath them before execution.

### Likelihood Explanation
Moderate. EVM mempools routinely hold transactions for multiple blocks during congestion or gas-price spikes. MEV searchers can also delay inclusion of specific transactions. The `ATTACK_REQUESTED → UNDER_ATTACK` registry transition is a real, on-chain event that can occur while a stake transaction is pending. No privileged actor is required; ordinary network conditions are sufficient.

### Recommendation
Add a `deadline` parameter to `stake()` (and symmetrically to `contributeBonus()`) and revert if `block.timestamp > deadline`. This is the standard mitigation used by Uniswap and other DeFi protocols:

```solidity
function stake(uint256 amount, uint256 deadline) external nonReentrant whenPoolNotPaused {
    if (block.timestamp > deadline) revert DeadlineExpired();
    ...
}
```

This gives stakers a way to express "execute only while conditions are still as I observed them," directly analogous to adding a timeout to an RPC call to prevent execution under stale or unfavorable conditions.

### Proof of Concept

1. Registry is in `ATTACK_REQUESTED`. Staker reads state off-chain, confirms `withdraw()` is open, and submits `stake(amount)` with a moderate gas price.
2. Network congestion delays inclusion. Registry transitions `ATTACK_REQUESTED → UNDER_ATTACK` (an on-chain event by the trusted registry).
3. Any party calls `pokeRiskWindow()` (or any pool interaction), setting `riskWindowStart = block.timestamp_of_transition`.
4. Staker's pending `stake()` transaction is eventually included. `_observePoolState()` returns `UNDER_ATTACK`; `_assertDepositsAllowed` passes.
5. `newEntry = block.timestamp` is floored at `riskWindowStart` (already set), so the staker's k=2 contribution is `amount × (T − riskWindowStart)²` — near zero if flagged soon after.
6. `withdraw()` now reverts `WithdrawsDisabled` for this staker (riskWindowStart != 0 latch).
7. Staker's principal is locked until resolution. If the agreement is later flagged `CORRUPTED`, the staker loses principal to `recoveryAddress` — an outcome they never voluntarily accepted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
