Audit Report

## Title
`TakeAmountsLib.sellerAssetsToUnits` panics on `sellerPrice == 0` outside try/catch, reverting entire bundle for a case `Midnight.take` accepts - (`src/periphery/TakeAmountsLib.sol` / `src/periphery/MidnightBundles.sol`)

## Summary
When a buy offer's tick price equals the settlement fee exactly, `sellerPrice` computes to zero in `TakeAmountsLib.sellerAssetsToUnits`. The call `targetSellerAssets.mulDivUp(WAD, 0)` triggers an arithmetic underflow panic at `(d - 1)` before any division. Because this call is evaluated unconditionally before the `try/catch` guard in `supplyCollateralAndSellWithAssetsTarget`, the panic propagates and reverts the entire bundle — including all collateral supplies — even though `Midnight.take` accepts the same offer without reverting.

## Finding Description

**Root cause — `mulDivUp` with `d = 0`:**

`UtilsLib.mulDivUp` is implemented in checked arithmetic:

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
``` [1](#0-0) 

When `d = 0`, the sub-expression `(d - 1)` = `(0 - 1)` underflows under Solidity 0.8 checked arithmetic, producing panic code `0x11`. This is explicitly confirmed by the existing test `testMulDivUpDivisionByZero`, which expects `stdError.arithmeticError` (not `divisionError`): [2](#0-1) 

**Trigger path — `sellerAssetsToUnits` with `sellerPrice = 0`:**

`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and passes it as the divisor to `mulDivUp`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
``` [3](#0-2) 

When `offer.buy == true` and `tickToPrice(offer.tick) == settlementFee(id, timeToMaturity)`, `sellerPrice = 0`, and `mulDivUp(..., WAD, 0)` panics. The NatSpec comment on `sellerAssetsToUnits` states it "Reverts if offerPrice < settlementFee" but does not account for the `==` case, which also panics. [4](#0-3) 

**Why `Midnight.take` does not revert for the same input:**

`Midnight.take` computes `sellerAssets = units.mulDivDown(sellerPrice, WAD)` — dividing by `WAD`, not by `sellerPrice`. With `sellerPrice = 0` this yields `0` cleanly:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD);
``` [5](#0-4) 

The NatSpec at line 331 only says "Taking buy offers with price **<** settlement fee will revert." The `==` case is not rejected by `take`. [6](#0-5) 

**The panic is outside the `try/catch`:**

In `supplyCollateralAndSellWithAssetsTarget`, `sellerAssetsToUnits` is evaluated unconditionally before the guarded `take` call:

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) returns (...) { ... } catch {}
``` [7](#0-6) 

The `try/catch` only guards `IMidnight.take`; the pre-computation panic propagates to the caller. The function's own NatSpec acknowledges this: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." [8](#0-7) 

**`consumableUnits` does not panic for the same offer:**

For `offer.buy == true` and `offer.maxUnits == 0`, `consumableUnits` routes to `buyerAssetsToUnits`, which divides by `buyerPrice = sellerPrice + settlementFee = settlementFee` (non-zero when fee > 0): [9](#0-8) 

The panic is exclusively from the direct `sellerAssetsToUnits` call at line 286.

## Impact Explanation
The entire `supplyCollateralAndSellWithAssetsTarget` transaction reverts. All collateral supplies executed before the loop (lines 269–275) are also rolled back. The taker cannot sell units or supply collateral even though all other offers in the bundle are valid and `Midnight.take` would accept them individually. The stated design intent of the bundler is to skip offers that `take` would reject; here it panics for an offer `take` would accept, violating that invariant. [10](#0-9) 

## Likelihood Explanation
Settlement fees are interpolated linearly between breakpoints as time-to-maturity decreases. For any market with a non-zero settlement fee, there exists a moment when the interpolated fee equals `tickToPrice(t)` for some tick `t`. The protocol's own NatSpec acknowledges this: "if the settlement fee gets increased, it might implicitly cancel offers with very low price." [6](#0-5) 

A maker can create a buy offer at such a tick. Any taker who includes that offer in a `supplyCollateralAndSellWithAssetsTarget` bundle at the exact moment the condition holds will have their entire bundle reverted. The condition is transient but repeatable and can be targeted by monitoring on-chain fee state against tick prices.

## Recommendation
In `sellerAssetsToUnits`, add an explicit guard for the `sellerPrice == 0` case for buy offers. When `sellerPrice == 0`, the seller receives zero assets regardless of units taken, so the function should return `type(uint256).max` (signaling "any number of units satisfies the target") or mirror the `buyerAssetsToUnits` approach of explicitly checking `offerPrice - settlementFee` before calling `mulDivUp`. A minimal fix:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
if (offer.buy && sellerPrice == 0) return type(uint256).max;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

This ensures the bundler skips (via `min` with `takes[i].units`) rather than panics, consistent with the design intent. [11](#0-10) 

## Proof of Concept
Minimal Foundry test steps:

1. Deploy a market with a non-zero settlement fee at breakpoint index 1.
2. Create a lender buy offer at tick `t` such that `tickToPrice(t)` equals the interpolated settlement fee at the current TTM.
3. Call `supplyCollateralAndSellWithAssetsTarget` with that offer in the `takes` array and any non-empty `collateralSupplies`.
4. Observe the transaction reverts with `arithmeticError` (panic `0x11`) rather than skipping the offer.
5. Confirm that calling `Midnight.take` directly with the same offer and `units = 0` succeeds (returns `(0, 0)`).

The existing test `testMulDivUpDivisionByZero` already confirms the panic mechanism at the `UtilsLib` level. [2](#0-1)
