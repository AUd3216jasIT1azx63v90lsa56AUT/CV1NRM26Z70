Audit Report

## Title
Front-Running Partial Fill Causes Exact-Equality DoS in `buyWithAssetsTargetAndWithdrawCollateral` and `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/MidnightBundles.sol)

## Summary

`buyWithAssetsTargetAndWithdrawCollateral` enforces a strict exact-equality check `filledBuyerAssets == targetFilledBuyerAssets` at line 224. The `unitsToTake` per offer is bounded by `ConsumableUnitsLib.consumableUnits`, which reads live on-chain `consumed[maker][group]` state at execution time with no snapshot. Any unprivileged actor who takes even a minimal amount from a target offer before the bundle executes will reduce `consumableUnits`, causing the bundle to underfill and revert with `OutOfOffers`. The identical flaw exists in `supplyCollateralAndSellWithAssetsTarget` at line 303.

## Finding Description

**`consumed` tracking in `Midnight.sol` lines 367–369:**
Every call to `take()` with `offer.maxAssets > 0` increments `consumed[offer.maker][offer.group]` by the assets transacted:

```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
``` [1](#0-0) 

**`ConsumableUnitsLib.consumableUnits` lines 14–23:**
For a sell offer with `maxAssets > 0`, remaining capacity is computed from live state:

```solidity
uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
// ...
return TakeAmountsLib.sellerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
``` [2](#0-1) 

There is no snapshot of `consumed` at bundle submission time; the value is read fresh at execution.

**Bundle loop and exact equality check in `MidnightBundles.sol` lines 208–224:**

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.buyerAssetsToUnits(..., targetFilledBuyerAssets - filledBuyerAssets),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // live state
);
try IMidnight(MIDNIGHT).take(..., unitsToTake, ...) returns (uint256 resBuyerAssets, uint256) {
    filledBuyerAssets += resBuyerAssets;
} catch {}
// ...
require(filledBuyerAssets == targetFilledBuyerAssets, OutOfOffers());
``` [3](#0-2) 

**Exploit flow:**
1. Victim submits `buyWithAssetsTargetAndWithdrawCollateral` with offer O (`maxAssets` set), `targetFilledBuyerAssets = X`.
2. Attacker front-runs with a direct `Midnight.take(O, ..., smallUnits, ...)` — permissionless, any address can call `take`.
3. `consumed[O.maker][O.group]` increases by Y assets.
4. Victim's bundle executes: `consumableUnits(O)` returns `sellerAssetsToUnits(maxAssets - Y)` — reduced capacity.
5. `unitsToTake` is capped at the reduced value; `take` succeeds but `resBuyerAssets < targetFilledBuyerAssets`.
6. Loop ends (no more offers); `filledBuyerAssets < targetFilledBuyerAssets`.
7. `require(filledBuyerAssets == targetFilledBuyerAssets, OutOfOffers())` reverts.

**Why existing checks do not stop it:**
- The `try/catch` only handles reverts from `take`; it does not handle the case where `take` succeeds but fills fewer assets than needed.
- There is no slippage tolerance or minimum-fill threshold — the check is strict equality.
- There is no snapshot of `consumed` at bundle submission time.
- The attacker's `take` is fully valid and passes all core checks.

The identical pattern exists in `supplyCollateralAndSellWithAssetsTarget` at line 303: [4](#0-3) 

Note also that `buyWithAssetsTargetAndWithdrawCollateral` pulls `targetBuyerAssets` from the caller but has no refund path for unused tokens — the strict equality check is the only guard preventing token loss in the contract, which means relaxing it requires adding a refund mechanism simultaneously. [5](#0-4) 

## Impact Explanation

Any unprivileged actor holding loan tokens can repeatedly grief a victim's `buyWithAssetsTargetAndWithdrawCollateral` or `supplyCollateralAndSellWithAssetsTarget` bundle by front-running with a minimal partial fill of the target offer. The victim's transaction reverts with `OutOfOffers` every time, permanently blocking the bundle from executing as long as the attacker repeats the front-run. The victim's tokens are returned on revert, but the victim cannot complete the intended market action via the bundle. This constitutes a sustained, repeatable denial-of-service against a specific user's bundle execution — matching the "service unavailability or severe degradation under realistic attacker input" impact class in RESEARCHER.md.

## Likelihood Explanation

**Preconditions:**
- Victim uses `buyWithAssetsTargetAndWithdrawCollateral` or `supplyCollateralAndSellWithAssetsTarget` with an offer that has `maxAssets > 0`.
- The victim's `takes` array has insufficient fallback offers to absorb the capacity reduction (e.g., a single-offer bundle, or the attacker front-runs all listed offers).
- Attacker holds any nonzero amount of loan tokens.

**Feasibility:** High. Taking an offer is permissionless — any address can call `Midnight.take`. The attacker needs only enough loan tokens to take 1 unit from the target offer, reducing `consumableUnits` by at least 1. The attacker receives a market credit position in return, so the net cost is gas plus any bid-ask spread, not the full loan amount. The attack is repeatable every block with no cooldown.

## Recommendation

1. **Replace exact equality with a `>=` check and add a refund path.** Change `require(filledBuyerAssets == targetFilledBuyerAssets, OutOfOffers())` to `require(filledBuyerAssets >= targetFilledBuyerAssets, OutOfOffers())` and transfer any surplus tokens back to `msg.sender` at the end of the function. This requires auditing the token accounting carefully since `targetBuyerAssets` is pulled upfront.
2. **Alternatively, add a `minFilledBuyerAssets` parameter** (slippage tolerance) and replace the exact equality with `require(filledBuyerAssets >= minFilledBuyerAssets, ...)`, returning unused tokens to the caller. This gives users explicit control over acceptable partial fills.
3. **For `supplyCollateralAndSellWithAssetsTarget`**, apply the symmetric fix: replace `require(filledSellerAssets == targetFilledSellerAssets, OutOfOffers())` with a `>=` check and adjust the downstream `SafeTransferLib.safeTransfer(loanToken, receiver, targetSellerAssets)` to transfer the actual filled amount minus fees.

## Proof of Concept

**Minimal Foundry fork test plan:**

```solidity
// 1. Deploy Midnight + MidnightBundles on a local fork.
// 2. Maker posts a sell offer with maxAssets = 1000e18, group = 1.
// 3. Victim calls buyWithAssetsTargetAndWithdrawCollateral with
//    targetBuyerAssets = 1000e18, takes = [offer O], referralFeePct = 0.
// 4. Before victim's tx lands, attacker calls Midnight.take(O, ..., 1 unit, ...).
//    consumed[maker][1] is now > 0.
// 5. Victim's tx executes:
//    - consumableUnits(O) = sellerAssetsToUnits(maxAssets - consumed) < full capacity
//    - unitsToTake is reduced; take succeeds with resBuyerAssets < 1000e18
//    - filledBuyerAssets < targetFilledBuyerAssets
//    - require(filledBuyerAssets == targetFilledBuyerAssets) reverts with OutOfOffers
// 6. Assert: victim's tx reverted; attacker's take succeeded.
// 7. Repeat step 4 every block to demonstrate sustained DoS.
```

The test is deterministic: controlling transaction ordering in a local fork (e.g., via `vm.prank` + manual sequencing) is sufficient to reproduce without any probabilistic assumptions.
