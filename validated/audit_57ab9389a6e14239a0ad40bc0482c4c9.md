Audit Report

## Title
`ConsumableUnitsLib.consumableUnits` arithmetic overflow outside try/catch reverts bundler transactions for buy offers with large `maxAssets` - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary
When a buy offer has `maxUnits == 0` and `maxAssets` large enough that `targetBuyerAssets * WAD` overflows `uint256`, `TakeAmountsLib.buyerAssetsToUnits` panics via Solidity 0.8.x checked arithmetic. Because `ConsumableUnitsLib.consumableUnits` is invoked outside the try/catch block in every bundler loop, the panic propagates and reverts the entire transaction. No tokens are lost (revert returns them), but gas is permanently consumed and the bundler is rendered unusable for any offer matching this condition.

## Finding Description

**Root cause — overflow in `mulDivUp`:**

`UtilsLib.mulDivUp` at `src/libraries/UtilsLib.sol:34-36` computes `(x * y + (d - 1)) / d` in default checked arithmetic. When `x = type(uint256).max` and `y = WAD (1e18)`, the product overflows `uint256`, triggering a Panic(0x11).

**Trigger path through `ConsumableUnitsLib`:**

`src/periphery/ConsumableUnitsLib.sol:18-19` — when `offer.maxUnits == 0` and `offer.buy == true`:
```solidity
} else if (offer.buy) {
    return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
```
With `consumed == 0` and `offer.maxAssets = type(uint256).max`, `remaining = type(uint256).max` is passed as `targetBuyerAssets`.

`src/periphery/TakeAmountsLib.sol:28-29`:
```solidity
require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());
return offer.buy ? targetBuyerAssets.mulDivUp(WAD, buyerPrice) : ...
```
The `require` passes (valid prices satisfy `buyerPrice <= WAD`), then `type(uint256).max * WAD` overflows on the next line.

**Why the overflow is outside the try/catch:**

In `supplyCollateralAndSellWithUnitsTarget` (`src/periphery/MidnightBundles.sol:147-160`) and `supplyCollateralAndSellWithAssetsTarget` (`src/periphery/MidnightBundles.sol:285-300`), `ConsumableUnitsLib.consumableUnits` is called in the `min(...)` expression that computes `unitsToTake`, which is outside the try/catch. Only `IMidnight(MIDNIGHT).take(...)` is wrapped:
```solidity
uint256 unitsToTake = min(
    ...,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // outside try/catch
);
try IMidnight(MIDNIGHT).take(...) { ... } catch {}  // only take() is guarded
```
The panic from `consumableUnits` propagates through `min(...)` and reverts the entire transaction.

**Why existing checks fail:**

The `require(buyerPrice <= WAD)` guard in `buyerAssetsToUnits` checks the price, not the magnitude of `targetBuyerAssets`. It passes for any valid price, then the unchecked-free `mulDivUp` overflows. The NatSpec comment "Reverts if ConsumableUnitsLib reverts" acknowledges the propagation but does not prevent it.

**Overflow threshold:** any `offer.maxAssets > type(uint256).max / WAD ≈ 1.157e59`. `type(uint256).max` is the natural "no cap" sentinel.

## Impact Explanation
The bundler is rendered unusable for any buy offer where `maxUnits == 0` and `maxAssets > ~1.157e59`. A taker routing such an offer through any of the four bundler entry-points will have their transaction revert. Gas is permanently lost; tokens are returned by the revert. A malicious maker can deliberately craft such an offer to grief specific takers who rely on the bundler, repeatably and at no cost beyond gas for offer creation.

## Likelihood Explanation
Setting `maxAssets = type(uint256).max` as an "unlimited" sentinel is a natural and common pattern for makers who do not wish to cap asset exposure. Any taker who routes such an offer through the bundler triggers the revert. No special privileges are required — the maker is an unprivileged user who freely sets `maxAssets` to any `uint256` value with no on-chain cap. The condition is deterministic, requires no timing or oracle manipulation, and is repeatable.

## Recommendation
Wrap the `consumableUnits` call in a try/catch (or equivalent) so that a revert from it causes the bundler to skip the offer rather than abort the transaction, consistent with the existing handling of `take()` failures. Alternatively, add an overflow-safe cap inside `buyerAssetsToUnits`: before calling `mulDivUp`, clamp `targetBuyerAssets` to `type(uint256).max / WAD` so the multiplication cannot overflow. A third option is to add an explicit cap on `maxAssets` in `consumableUnits` before passing it to `buyerAssetsToUnits`.

## Proof of Concept
1. Deploy `MidnightBundles` against a local fork.
2. Create a buy offer with `maxUnits = 0`, `maxAssets = type(uint256).max`, and a valid tick such that `buyerPrice <= WAD`.
3. Call `supplyCollateralAndSellWithUnitsTarget` with `takes[0]` referencing that offer.
4. Observe the transaction reverts with `Panic(0x11)` (arithmetic overflow) rather than skipping the offer.
5. Confirm gas is consumed and no tokens are transferred.

Fuzz test: set `offer.maxAssets` to any value in `[type(uint256).max / WAD + 1, type(uint256).max]` with `offer.maxUnits = 0` and `offer.buy = true`; all such values trigger the overflow.
