Audit Report

## Title
Front-running partial liquidation reduces `maxRepaid` below victim's `repaidUnits`, causing `RecoveryCloseFactorConditionsViolated()` revert - (File: src/Midnight.sol)

## Summary

In normal (pre-maturity) liquidation mode, `maxRepaid` is computed live from the current position state at call time. An unprivileged attacker can front-run any competing liquidator with a profitable partial liquidation of `A` units, which reduces `maxRepaid` by exactly `A` while leaving the `rcfThreshold` small-collateral exception value unchanged. This causes the victim's transaction to revert with `RecoveryCloseFactorConditionsViolated()` whenever `victimRepaidUnits > originalMaxRepaid - A`. The attack is self-funding and indefinitely repeatable.

## Finding Description

**Code path:** `src/Midnight.sol` lines 655–667.

`maxRepaid` is recomputed from live on-chain state on every call (lines 607–618, 659–661). After an attacker's partial liquidation of `A` units, the state transitions are:

- `_position.debt` → `debt - A` (line 676)
- `seizedAssets = A * lif * ORACLE_PRICE_SCALE / (WAD * price)` (line 652)
- `_position.collateral[collateralIndex]` → `collateral - seizedAssets` (line 670)
- `maxDebt` (recomputed next call) → `maxDebt - A * lif * lltv / WAD²` (line 613)

Therefore:
```
newDebt - newMaxDebt = (debt - maxDebt) - A*(WAD² - lif*lltv)/WAD²
newMaxRepaid         = originalMaxRepaid - A
```

The `rcfThreshold` exception (lines 664–665) is invariant under partial liquidations:
```
newCollateral*price/lif - newMaxRepaid
= (collateral - A*lif/price)*price/lif - (originalMaxRepaid - A)
= collateral*price/lif - originalMaxRepaid   ← identical to pre-attack value
```

**Exploit flow:**

1. Borrower is unhealthy (`debt > maxDebt`), no bad debt, RCF active (rcfThreshold exception does not apply).
2. `originalMaxRepaid = (debt - maxDebt) * WAD² / (WAD² - lif*lltv)`.
3. Victim liquidator broadcasts tx with `victimRepaidUnits` where `1 < victimRepaidUnits ≤ originalMaxRepaid`.
4. Attacker front-runs with `attackerRepaidUnits` satisfying:
   - `attackerRepaidUnits < originalMaxRepaid` (attacker passes RCF check)
   - `attackerRepaidUnits > originalMaxRepaid - victimRepaidUnits` (so `newMaxRepaid < victimRepaidUnits`)
5. Victim's tx executes: `newMaxRepaid = originalMaxRepaid - attackerRepaidUnits < victimRepaidUnits`; rcfThreshold exception still does not apply → revert with `RecoveryCloseFactorConditionsViolated()`.

The `liquidationLocked` check at line 621 only prevents liquidation during take callbacks (transient storage), not between separate transactions, so it provides no protection here.

## Impact Explanation

An unprivileged attacker can permanently prevent any competing liquidator from executing a full or large partial liquidation of any unhealthy position by repeatedly front-running with small profitable partial liquidations. The victim's transaction always reverts with `RecoveryCloseFactorConditionsViolated()`. The position remains unhealthy and continues to accumulate protocol risk, while the attacker monopolizes all liquidation profit. This constitutes both a denial-of-service on liquidators and theft of liquidation opportunity.

## Likelihood Explanation

Preconditions are the standard conditions for any contested liquidation on a chain with a public mempool: borrower is unhealthy, RCF is active, victim submits `repaidUnits > 1`, and no `liquidatorGate` restricts the attacker (or the attacker also passes the gate). The attack is economically self-funding (attacker profits from the partial liquidation at `lif > 1` discount) and can be repeated indefinitely every time the victim resubmits.

## Recommendation

Replace the hard revert with a cap: when `repaidUnits > maxRepaid` and the rcfThreshold exception does not apply, clamp `repaidUnits` (and the corresponding `seizedAssets`) to `maxRepaid` rather than reverting. Alternatively, add a caller-supplied `minRepaidUnits` slippage parameter: if `maxRepaid < minRepaidUnits`, revert with a dedicated slippage error so the victim's intent is preserved without being silently griefed. Either approach eliminates the front-running vector by making the outcome of the victim's transaction independent of whether a prior partial liquidation occurred in the same block.
