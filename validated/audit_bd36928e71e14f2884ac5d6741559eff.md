Audit Report

## Title
Fully-consumed assets-mode buy offer can be re-taken with `units > 0` when `buyerPrice < WAD`, minting unbacked credit and debt - (`File: src/Midnight.sol`, `src/libraries/TickLib.sol`)

## Summary
In `Midnight.take`, when `offer.buy == true` and `offer.maxAssets > 0`, the consumed guard uses `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `tickToPrice` always returns a value strictly less than `WAD` for every valid tick, `mulDivDown(1, buyerPrice, WAD) == 0` for `units = 1`. A fully-consumed buy offer (`consumed == maxAssets`) can therefore be re-taken indefinitely with `units = 1`, each call minting 1 unit of unbacked credit to the maker and 1 unit of unbacked debt to the taker with zero token transfer.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

`buyerAssets` is computed with `mulDivDown`, which floors the result: [1](#0-0) 

For `units = 1` and any valid tick, `buyerPrice < WAD = 1e18`, so `floor(1 * buyerPrice / 1e18) = 0`. The guard `require(newConsumed <= offer.maxAssets)` then evaluates as `require(M + 0 <= M)`, which always passes.

**Why `buyerPrice < WAD` is structural — `src/libraries/TickLib.sol` lines 44–52:**

`tickToPrice` computes `1e36 / (1e18 + wExp(...))` rounded to `PRICE_ROUNDING_STEP = 1e12`. The denominator is always strictly greater than `1e18`, so the result is always strictly less than `1e18 = WAD` for every tick in `[0, MAX_TICK]`. [2](#0-1) 

**Position accounting still executes with `units = 1`:**

After the guard passes, the full accounting path runs: [3](#0-2) 

With `units = 1`, `buyerCreditIncrease = 1` (if maker has no debt), `sellerDebtIncrease = 1` (if taker has no credit). `buyerPos.credit += 1` and `sellerPos.debt += 1` are written with no corresponding token transfer, since `safeTransferFrom(..., buyerAssets - sellerAssets)` transfers 0 tokens. [4](#0-3) 

**Existing formal verification gaps — `certora/specs/Consume.spec`:**

- `takeConsumedAtMaxUnchangedAssets` (lines 88–97) only asserts `consumed` is unchanged — it does not assert `units == 0`, so it does not catch the case where `units > 0` but `buyerAssets = 0`. [5](#0-4) 
- `fullyConsumedOfferRevertsOnNonTrivialTake` (lines 99–111) only covers `maxAssets == 0` (units mode); no equivalent rule exists for assets mode. [6](#0-5) 
- Critically, `mulDivDown` is summarized as `NONDET` in the spec (line 11), so the formal verifier never reasons about the rounding-to-zero case. [7](#0-6) 

**Protocol acknowledgment:**

The NatDoc at line 94 explicitly states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* A dedicated test `testBugBuyMaxAssetsBypass` exists in `test/TakeTest.sol`, confirming this is a reachable, known state. [8](#0-7) 

## Impact Explanation

Each successful re-take with `units = 1` on a fully-consumed buy offer:
- Mints 1 unit of credit to the maker without any loan-token deposit.
- Mints 1 unit of debt to the taker without any loan-token transfer.
- Increments `totalUnits` via `buyerCreditIncrease`, inflating the continuous-fee accrual base for all lenders in the market.

The contract's loan-token balance no longer covers all outstanding credit, violating the core solvency invariant. The `maxAssets` cap — intended to bound total maker exposure — is rendered permanently ineffective once an offer is fully consumed.

## Likelihood Explanation

**Preconditions:**
- A buy offer with `maxAssets > 0` at any valid tick (all 5820 ticks satisfy `price < WAD`).
- `consumed[maker][group] == maxAssets` (normal end state after full consumption).
- Attacker is any unprivileged address (not the maker, due to `SelfTake` guard).

**Feasibility:** High. The `price < WAD` condition is structural and permanent — no oracle manipulation, admin action, or special token behavior is required. The attacker only needs to call `take` with `units = 1` after an offer is exhausted.

**Repeatability:** Unlimited. Each call with `units = 1` succeeds; `consumed` stays pinned at `maxAssets` forever.

## Recommendation

Add a guard that rejects non-trivial takes on fully-consumed assets-mode offers, analogous to the existing units-mode protection. Specifically, after computing `buyerAssets`, require that if `offer.maxAssets > 0` and `consumed[offer.maker][offer.group] >= offer.maxAssets`, then `units == 0` (or equivalently, `buyerAssets > 0` whenever `units > 0`):

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroAssetsTake());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, change the consumed check to a strict less-than before incrementing: `require(consumed[offer.maker][offer.group] < offer.maxAssets || units == 0)`. The Certora rule `fullyConsumedOfferRevertsOnNonTrivialTake` should also be extended to cover the `maxAssets > 0` case with a concrete (non-NONDET) summary for `mulDivDown`.

## Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` provides a reproducible path. Manual steps:

1. Create a buy offer with `offer.buy = true`, `offer.maxAssets = M`, any valid tick.
2. Fill the offer normally until `consumed[maker][group] == M`.
3. Call `take(units=1, taker=attacker, offer=...)` — observe it succeeds.
4. Verify `attacker`'s debt increased by 1 and `maker`'s credit increased by 1 with zero token transfer.
5. Repeat step 3 arbitrarily many times; `consumed` remains pinned at `M`.

/// If consumed is already at or above maxAssets before a take in assets mode, it remains unchanged.
rule takeConsumedAtMaxUnchangedAssets(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxUnits == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumedBefore >= offer.maxAssets => consumed(offer.maker, offer.group) == consumedBefore;
}

/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
