Audit Report

## Title
Missing budget invariant check causes revert when markup-formula referral fee exceeds remaining balance - (`src/periphery/MidnightBundles.sol`)

## Summary
`buyWithUnitsTargetAndWithdrawCollateral` pulls `maxBuyerAssets` upfront and runs a fill loop bounded only by `targetUnits`, accumulating `filledBuyerAssets` without any cap relative to the referral-fee-adjusted budget. After the loop, the markup formula at line 102 computes `referralFeeAssets = filledBuyerAssets * referralFeePct / (WAD - referralFeePct)`, which can exceed the contract's remaining balance (`maxBuyerAssets - filledBuyerAssets`), causing the `safeTransfer` at line 103 to revert with insufficient balance or the subtraction at line 104 to panic with arithmetic underflow under Solidity 0.8.34 checked arithmetic.

## Finding Description

**Code path** — `src/periphery/MidnightBundles.sol`, `buyWithUnitsTargetAndWithdrawCollateral`:

Line 66 pulls exactly `maxBuyerAssets` tokens from `msg.sender` into the bundler: [1](#0-0) 

The fill loop (lines 71–86) accumulates `filledBuyerAssets` as MIDNIGHT pulls tokens from the bundler via the `type(uint256).max` approval. The loop terminates only when `filledUnits == targetUnits` or all takes are exhausted — there is no cap on `filledBuyerAssets` relative to `maxBuyerAssets`: [2](#0-1) 

The only post-fill check is `require(filledUnits == targetUnits)` — no budget invariant check exists: [3](#0-2) 

After the loop, the bundler holds `maxBuyerAssets - filledBuyerAssets` tokens. The referral fee is then computed using the markup formula and transferred: [4](#0-3) 

**Root cause — mathematical breakdown**: For the transfers to succeed, the bundler needs `referralFeeAssets <= maxBuyerAssets - filledBuyerAssets`, which simplifies to:

```
filledBuyerAssets <= maxBuyerAssets * (WAD - referralFeePct) / WAD
```

The fill loop enforces no such bound. If `filledBuyerAssets` exceeds this threshold, `referralFeeAssets > maxBuyerAssets - filledBuyerAssets`, and:
- Line 103's `safeTransfer` reverts (bundler has insufficient balance), or
- Line 104's subtraction underflows (Solidity 0.8.34 checked arithmetic panic).

**Concrete example** (`referralFeePct = 10%`, `maxBuyerAssets = 100`):
- Threshold: `100 * 0.9 = 90`
- Fill loop produces `filledBuyerAssets = 95` (valid, since `95 <= 100`)
- `referralFeeAssets = 95 * 0.1 / 0.9 ≈ 10.56`
- Bundler holds `100 - 95 = 5` tokens
- Line 103 attempts to transfer `10.56` → revert

**Existing guard is insufficient**: `require(referralFeePct < WAD)` only ensures the fee percentage is below 100%; it does not bound the markup fee relative to the remaining budget: [5](#0-4) 

**Contrast with `buyWithAssetsTargetAndWithdrawCollateral`**: That function correctly pre-computes `referralFeeAssets = targetBuyerAssets * referralFeePct / WAD` and sets `targetFilledBuyerAssets = targetBuyerAssets - referralFeeAssets` before the fill loop, bounding fills to the fee-adjusted budget. `buyWithUnitsTargetAndWithdrawCollateral` has no equivalent pre-computation: [6](#0-5) 

The NatSpec invariant "The msg.sender will pay at most `maxBuyerAssets`" and the documented total cost formula `filledBuyerAssets + filledBuyerAssets * referralFeePct / (WAD - referralFeePct)` are both violated: [7](#0-6) 

## Impact Explanation

The entire transaction reverts atomically. All state changes roll back: the user acquires no credit, no collateral is withdrawn, and `maxBuyerAssets` tokens are returned. The user bears only the gas cost. The documented invariant is violated in the sense that the function fails to complete rather than succeeding with a refund, rendering `buyWithUnitsTargetAndWithdrawCollateral` unusable for any caller who sets `maxBuyerAssets` as their total budget with a non-zero referral fee and a tight fill. No funds are permanently lost. Severity is low-to-medium: the slippage-protection guarantee is broken and the function is effectively DoS'd under common usage patterns.

## Likelihood Explanation

**Preconditions**:
1. `referralFeePct > 0` — any non-zero referral fee, set by the caller or a frontend.
2. `filledBuyerAssets > maxBuyerAssets * (WAD - referralFeePct) / WAD` — triggered whenever fills consume more than `(1 - referralFeePct/WAD)` of the budget.

For `referralFeePct = 10%`, the revert fires whenever fills consume more than 90% of `maxBuyerAssets`. A user who sets `maxBuyerAssets` as their total budget (the natural interpretation given the NatSpec) and uses any non-trivial referral fee will hit this on nearly every tight fill. No privileged role, oracle manipulation, or external state is required. The condition is deterministic and repeatable by any unprivileged caller.

## Recommendation

Mirror the pattern used in `buyWithAssetsTargetAndWithdrawCollateral`. Before the fill loop, pre-compute the fee-adjusted fill budget and cap `filledBuyerAssets` accordingly:

```solidity
// Pre-compute the maximum assets available for fills after reserving the referral fee.
// Using the markup formula: totalCost = fills + fills * pct / (WAD - pct)
// => fills <= maxBuyerAssets * (WAD - referralFeePct) / WAD
uint256 maxFilledBuyerAssets = maxBuyerAssets.mulDivDown(WAD - referralFeePct, WAD);
```

Then either:
- **Option A**: Add a post-loop invariant check: `require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets, BudgetExceeded())` after line 102.
- **Option B** (preferred): Cap the fill loop by tracking remaining budget, analogous to how `buyWithAssetsTargetAndWithdrawCollateral` bounds by `targetFilledBuyerAssets`.

## Proof of Concept

**Minimal manual steps**:
1. Deploy `MidnightBundles` against a local fork.
2. Create a market with a single sell offer priced such that filling `targetUnits` costs `filledBuyerAssets = 95` assets.
3. Call `buyWithUnitsTargetAndWithdrawCollateral` with `maxBuyerAssets = 100`, `referralFeePct = 0.1e18` (10%), `targetUnits` matching the offer.
4. Observe revert at line 103 (`safeTransfer` fails: bundler holds 5 tokens, attempts to send ≈10.56).

**Fuzz test plan**:
```solidity

    uint256 maxBuyerAssets,
    uint256 referralFeePct,
    uint256 filledBuyerAssets

    // Assert that referralFeeAssets exceeds remaining balance
    assertGt(referralFeeAssets, maxBuyerAssets - filledBuyerAssets);
```
