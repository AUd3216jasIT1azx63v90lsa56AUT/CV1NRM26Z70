Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via zero-rounding `buyerAssets` — (File: src/Midnight.sol)

## Summary
When a buy offer is taken with `maxAssets > 0` and any `tick < MAX_TICK`, integer division in `mulDivDown(units, buyerPrice, WAD)` floors `buyerAssets` to zero for small `units` values (e.g. `units = 1`). Because `consumed[maker][group]` is incremented by `buyerAssets = 0`, the guard `require(newConsumed <= offer.maxAssets)` passes unconditionally on every call. An attacker can take the offer an unbounded number of times, causing the maker to accumulate credit and the taker to accumulate debt with zero tokens transferred per fill, completely bypassing the maker's stated cap.

## Finding Description

**Root cause — `tickToPrice` always returns a value strictly less than `WAD` for `tick < MAX_TICK`:**

`tickToPrice` computes `1e36 / (1e18 + wExp(LN_ONE_PLUS_DELTA * (MAX_TICK/2 - tick)))` rounded to `PRICE_ROUNDING_STEP = 1e12`. [1](#0-0) 

At `tick = MAX_TICK = 5820`, the exponent argument is `LN_ONE_PLUS_DELTA * (2910 - 5820)`, a large negative, so `wExp` ≈ 0 and the result ≈ `1e36 / 1e18 = WAD`. For every `tick < MAX_TICK` the denominator is strictly larger, so the price is strictly less than `WAD = 1e18`. The minimum non-zero price is `PRICE_ROUNDING_STEP = 1e12`. [2](#0-1) 

**`mulDivDown` has no minimum-result guard:**

```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
``` [3](#0-2) 

With `units = 1` and any `buyerPrice < 1e18`: `(1 * buyerPrice) / 1e18 = 0`.

**The `maxAssets` guard is defeated:**

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [4](#0-3) 

When `buyerAssets = 0`, `consumed` is incremented by 0, so `newConsumed` never grows and the `require` never fires.

**Position mutation still executes with `units = 1`:**

With `units = 1` and no prior debt/credit, `buyerCreditIncrease = zeroFloorSub(1, 0) = 1` and `sellerDebtIncrease = 1`. The maker gains 1 unit of credit and the taker gains 1 unit of debt per call. [5](#0-4) [6](#0-5) 

**Token transfers are zero-value:**

Both `safeTransferFrom` calls transfer `buyerAssets - sellerAssets = 0` and `sellerAssets = 0` respectively. Standard ERC-20 tokens accept zero-value transfers, so the call completes successfully. [7](#0-6) 

**No minimum-units check exists:** There is no `require(units > 0)` or equivalent anywhere in `src/Midnight.sol`. `units = 1` is accepted without restriction.

**Why existing checks fail:**

- `takeConsumedDelta` in `Consume.spec` is scoped to `maxAssets == 0` only (line 68); no analogous delta rule exists for assets mode. [8](#0-7) 

- `mulDivDown` and `mulDivUp` are summarized as `NONDET` in the Certora spec, so the formal proofs never model the rounding-to-zero case. [9](#0-8) 

- `takeConsumedBoundedByMax` (line 62) only asserts `consumed <= maxAssets` after a single take; it does not assert that `consumed` strictly increases when `units > 0`. [10](#0-9) 

## Impact Explanation
The `maxAssets` cap — the sole mechanism preventing a buy offer from being filled beyond the maker's stated limit — is completely bypassed. The maker accumulates unbounded credit exposure without any corresponding token inflow per fill. At maturity, the maker holds credit positions backed by taker debt that was created without token deposit, creating an accounting imbalance. If the taker defaults, the maker's excess credit is unbacked. This violates the core protocol invariant that offers cannot be overfilled beyond the maker's stated cap.

## Likelihood Explanation
**Preconditions**: (1) A buy offer with `maxAssets > 0` — a normal maker configuration. (2) Any `tick < MAX_TICK` — satisfied by virtually every real offer since `MAX_TICK` is the par tick. (3) Attacker willing to accumulate debt at zero token cost per unit. No privileged access, no oracle manipulation, and no special token behavior is required. The attack is repeatable in a single transaction via `multicall` or across multiple blocks.

## Recommendation
Add a minimum-assets guard in the `maxAssets` branch to ensure that each fill transfers at least 1 token unit:

```solidity
if (offer.maxAssets > 0) {
    require(buyerAssets > 0, ZeroBuyerAssets());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` value such that `mulDivDown(units, buyerPrice, WAD) >= 1`, i.e. `units >= WAD / buyerPrice + 1`. A simpler approach is to require `units >= WAD / PRICE_ROUNDING_STEP = 1e6` as a protocol-wide floor when `maxAssets > 0`.

Additionally, add a `takeConsumedDelta` analog for assets mode in `Consume.spec` that asserts `consumed` strictly increases when `units > 0` and `maxAssets > 0`, and remove the `NONDET` summaries for `mulDivDown`/`mulDivUp` in that rule.

## Proof of Concept
**Minimal Foundry test:**

```solidity
// 1. Deploy Midnight, create a market.
// 2. Maker posts a buy offer: maxAssets = 1000e18, tick = 2910 (mid-range), valid ratifier/expiry.
// 3. Record consumed[maker][group] = 0.
// 4. Attacker calls take(offer, ..., units=1, ...) 1000 times (or via multicall).
// 5. Assert consumed[maker][group] == 0  (cap never advanced).
// 6. Assert maker.credit == 1000         (unbounded credit accumulated).
// 7. Assert attacker.debt == 1000        (unbounded debt accumulated).
// 8. Assert loanToken.balanceOf(address(midnight)) unchanged (zero tokens transferred).
```

All four assertions will pass, demonstrating the cap bypass.
