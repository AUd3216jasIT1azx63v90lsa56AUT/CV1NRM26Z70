Audit Report

## Title
Division by Zero in `sellerAssetsToUnits` via `tickToPrice(0) = 0` Causes DoS of Bundle Flows - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary

`tickToPrice(0)` provably returns `0` due to double rounding by `PRICE_ROUNDING_STEP`. When `consumableUnits` is called on a sell offer with `offer.maxUnits == 0` and `offer.tick == 0`, it delegates to `sellerAssetsToUnits`, which sets `sellerPrice = 0` and then calls `mulDivDown(targetSellerAssets, WAD, 0)`, triggering an unconditional division-by-zero revert. Because `MidnightBundles` calls `consumableUnits` outside any `try/catch`, the entire bundle transaction reverts.

## Finding Description

**Confirmed zero-price fact:**

`test/TickLibTest.sol` line 16 asserts and passes: [1](#0-0) 

The root cause is the double rounding in `tickToPrice`: for tick=0, the intermediate result of `1e36 / (1e18 + wExp(...))` is smaller than `PRICE_ROUNDING_STEP = 1e12`, so `divHalfDownUnchecked(..., PRICE_ROUNDING_STEP) * PRICE_ROUNDING_STEP` collapses to `0`: [2](#0-1) 

**Reachable division-by-zero path:**

`consumableUnits` reaches `sellerAssetsToUnits` when `offer.maxUnits == 0` and `offer.buy == false`: [3](#0-2) 

Inside `sellerAssetsToUnits`, for a sell offer (`offer.buy = false`), `sellerPrice` is set directly to `offerPrice = tickToPrice(0) = 0`, then `mulDivDown(targetSellerAssets, WAD, sellerPrice)` is called with denominator `0`: [4](#0-3) 

**Bundle functions propagate the revert:**

`buyWithUnitsTargetAndWithdrawCollateral` calls `consumableUnits` at line 77 outside any `try/catch`; only the subsequent `take` call is wrapped: [5](#0-4) 

`buyWithAssetsTargetAndWithdrawCollateral` has the same pattern at line 213: [6](#0-5) 

`supplyCollateralAndSellWithAssetsTarget` also calls `consumableUnits` at line 290 outside any `try/catch`: [7](#0-6) 

**Why existing checks fail:**

- `tickToPrice` only guards `tick > MAX_TICK`; tick=0 is in range. [8](#0-7) 
- `consumableUnits` has no guard for `sellerPrice == 0`.
- `sellerAssetsToUnits` has no guard for `sellerPrice == 0`.

## Impact Explanation

Any bundle transaction that calls `consumableUnits` on a sell offer with `tick=0` and `maxUnits=0` reverts unconditionally. Because `MidnightBundles` does not wrap `consumableUnits` in a `try/catch` (this is even documented in the NatSpec: "Reverts if ConsumableUnitsLib reverts"), the entire bundle call fails. This is a concrete DoS of bundle flows: any automated system (aggregator, router, front-end) that selects offers from an on-chain order book and includes a malicious offer in the `takes` array will have its entire bundle transaction griefed. Legitimate takers lose gas and fail to execute their intended trades.

## Likelihood Explanation

The preconditions are trivially satisfiable by any unprivileged maker: `tick=0` is always a valid, accessible tick (it satisfies `tick <= MAX_TICK` and `0 % spacing == 0` for any spacing), and setting `maxUnits=0` is a normal asset-capped offer configuration. The attack is zero-cost (no funds required beyond gas for offer creation), repeatable on every block, and requires no special role or privilege. The attack is most effective against automated bundle-construction systems that pull offers from an on-chain order book without filtering for zero-price ticks.

## Recommendation

Add a zero-price guard in `TakeAmountsLib.sellerAssetsToUnits` (and symmetrically in `buyerAssetsToUnits`): if `sellerPrice == 0`, return `0` (treating a zero-price offer as fully consumed / non-consumable). This causes `min(..., 0) = 0` in the bundle loop, making the offer a no-op rather than a revert. Alternatively, add the guard in `ConsumableUnitsLib.consumableUnits` by checking `tickToPrice(offer.tick) == 0` and returning `0` early. A third option is to validate `tickToPrice(offer.tick) > 0` at offer creation time in the core protocol, but this is a more invasive change.

## Proof of Concept

1. Deploy `MidnightBundles` against a live or forked Midnight deployment.
2. As an attacker, create a sell offer with `tick = 0`, `maxUnits = 0`, `maxAssets = 1e18`, `buy = false`.
3. As a legitimate taker, call `buyWithUnitsTargetAndWithdrawCollateral` with `takes` array containing the attacker's offer.
4. Observe the transaction reverts with a division-by-zero Panic (0x12).
5. Confirm with a Forge unit test: call `ConsumableUnitsLib.consumableUnits(midnight, id, offer)` directly with the above offer parameters and assert it reverts.
