Audit Report

## Title
ECDSA Signature Malleability Enables Nonce-Griefing Front-Run on `setIsAuthorized` - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` uses raw `ecrecover` at line 31 without enforcing a low-s constraint, allowing any observer to derive a second valid signature `(v^1, r, secp256k1_order − s)` for the same message and signer. An attacker who front-runs a victim's pending `setIsAuthorized` transaction with the malleable counterpart atomically consumes the victim's nonce, causing the victim's original transaction to revert with `InvalidNonce`. When the victim's call is part of an atomic bundle (e.g., authorize-then-invoke a protective action via `MidnightBundles`), the entire bundle fails, introducing a timing gap that can be exploited in time-sensitive liquidation-protection scenarios.

## Finding Description
**Exact code path — `EcrecoverAuthorizer.setIsAuthorized` (`src/periphery/EcrecoverAuthorizer.sol`, lines 24–48):**

- **Line 26:** `require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());` — nonce is checked and incremented atomically; the first caller to supply any valid signature for nonce N wins.
- **Line 31:** `address signer = ecrecover(digest, signature.v, signature.r, signature.s);` — raw `ecrecover` with no low-s guard.
- **No `msg.sender` restriction** anywhere in the function; any address may call it with any signature.

**Root cause:** secp256k1 ECDSA produces two valid `(v, r, s)` pairs per `(message, key)`: the canonical `(v, r, s)` and the malleable `(v^1, r, n − s)` where `n` is the curve order (`0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141`). Both pass `ecrecover` and return the identical signer address. Without `require(uint256(s) <= 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0)`, both are accepted.

**Exploit flow:**
1. Victim signs `Authorization{authorizer=V, authorized=X, isAuthorized=true, nonce=N, deadline=D}` and broadcasts `setIsAuthorized(auth, sig=(v,r,s))`.
2. Attacker observes the pending transaction, extracts `(v, r, s)`, computes `s' = n − s`, `v' = v ^ 1` — trivial off-chain arithmetic.
3. Attacker submits `setIsAuthorized(auth, Signature{v', r, s'})` with higher gas.
4. Attacker's transaction executes first: `nonce[V] == N` passes, nonce becomes `N+1`, `ecrecover` returns `V`, authorization is set to the intended value.
5. Victim's transaction executes: `nonce[V] == N+1` but `auth.nonce == N` → `InvalidNonce()` revert.

**Why existing checks fail:**
- `require(signer != address(0))` — both signatures recover the same non-zero address; passes for both.
- `require(signer == authorization.authorizer || isAuthorized(...))` — same signer recovered; passes for both.
- The nonce is already consumed before the victim's transaction executes; no check can recover it.

## Impact Explanation
The victim's `setIsAuthorized` transaction reverts. Although the authorization state is correctly set by the attacker's front-run, any atomic bundle that included the `setIsAuthorized` call (e.g., authorize a protective contract and immediately invoke `MidnightBundles.repayAndWithdrawCollateral` or a buy/sell function to avoid liquidation) fails entirely and must be re-submitted. The re-submission requires the victim to reconstruct the bundle without the `setIsAuthorized` step (since authorization is now set), introducing a non-trivial timing gap. In a liquidation-protection scenario on a chain with a public mempool, this gap can be sufficient for the victim's position to become liquidatable before the protective action executes. The attacker can repeat this for any future `setIsAuthorized` call the victim makes (e.g., revoking or changing authorization), causing persistent griefing with potential for indirect financial loss.

## Likelihood Explanation
**Preconditions:** Victim broadcasts a `setIsAuthorized` transaction (routine operation). Attacker monitors the public mempool (standard, zero-cost capability on Ethereum mainnet and most L2s with public mempools). No special role, funds beyond gas, or credentials required.

**Feasibility:** Computing `s' = n − s` and `v' = v ^ 1` is a single subtraction and XOR — executable in milliseconds with any standard secp256k1 library. The attack is repeatable for every new `setIsAuthorized` nonce the victim attempts to use. The attacker pays gas but gains the ability to persistently delay time-sensitive operations.

## Recommendation
Add a low-s check immediately before or after the `ecrecover` call in `setIsAuthorized`:

```solidity
uint256 SECP256K1_ORDER_HALF = 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0;
require(uint256(signature.s) <= SECP256K1_ORDER_HALF, InvalidSignature());
```

This ensures only the canonical form of each signature is accepted, making the malleable counterpart invalid and eliminating the front-run vector. This is the same approach used by OpenZeppelin's `ECDSA.recover` and enforced by EIP-2 for transaction signatures.

## Proof of Concept
**Minimal fork test plan:**

1. Deploy `EcrecoverAuthorizer` pointing to a mock `IMidnight`.
2. Generate a secp256k1 key pair `(privKey, V)`.
3. Construct `Authorization{authorizer=V, authorized=X, isAuthorized=true, nonce=0, deadline=block.timestamp+1000}`.
4. Sign the EIP-712 digest to obtain canonical `(v, r, s)`.
5. Compute malleable signature: `s' = secp256k1_order - s`, `v' = v ^ 1`.
6. As attacker, call `setIsAuthorized(auth, Signature{v', r, s'})` — assert it succeeds and `nonce[V] == 1`.
7. As victim, call `setIsAuthorized(auth, Signature{v, r, s})` — assert it reverts with `InvalidNonce()`.
8. Confirm `IMidnight.isAuthorized(V, X) == true` (authorization was set by attacker's tx, not victim's). [1](#0-0)

```text
    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );
```
