Audit Report

## Title
Fully-Consumed Buy Offer (maxAssets Mode) Bypassable via Zero-Rounding Units ā€” (File: src/Midnight.sol)

## Summary
In `take()`, when `offer.buy == true` and `offer.maxAssets > 0`, the consumed-cap increment uses `mulDivDown(units, buyerPrice, WAD)`. For any tick below `MAX_TICK`, `buyerPrice < WAD`, so `units = 1` yields `buyerAssets = 0`. The cap check `newConsumed <= maxAssets` becomes a no-op when `consumed` is already at `maxAssets`, yet position state (`credit`, `debt`, `totalUnits`) is still mutated with zero token transfer, bypassing the maker's intended exposure limit and inflating unbacked accounting state.

## Finding Description
**Root cause:** The consumed-cap guard at lines 367ā€“369 increments `consumed` by `buyerAssets`, which rounds to zero for `units * buyerPrice < WAD`. There is no `require(buyerAssets > 0)` or `require(newConsumed > consumedBefore)` guard. The position-mutation block at lines 408ā€“417 operates on `units` directly, not on `buyerAssets`, so state changes occur regardless of the zero asset amount.

**Exploit flow:**
1. Maker creates a buy offer: `maxAssets = M`, `maxUnits = 0`, tick below `MAX_TICK` (so `buyerPrice < WAD`).
2. Offer is legitimately filled until `consumed[maker][group] == M`.
3. Attacker calls `take(offer, ..., units=1)`. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. `consumed` stays at `M`. Cap check passes.
4. `buyerPos.credit += 1`, `sellerPos.debt += 1`, `totalUnits` increases ā€” all with zero token transfer (line 455: `safeTransferFrom(..., 0)`).
5. Repeatable indefinitely via `multicall` or across blocks.

**Existing checks reviewed and found insufficient:**
- `require(newConsumed <= offer.maxAssets)` ā€” passes because increment is 0.
- No `require(units > 0)` guard exists in `take()`.
- No `require(buyerAssets > 0 || units == 0)` guard exists.

## Impact Explanation
- **maxAssets cap bypass:** The maker's explicit exposure limit is rendered meaningless; their credit position grows without bound beyond `maxAssets` with no corresponding asset inflow.
- **Unbacked totalUnits inflation:** `totalUnits` grows without any asset deposit, breaking the invariant that every credit unit has a matching asset payment. All lenders' withdrawable share is diluted.
- **Griefing at zero cost:** The attacker pays only gas. The maker is forced into credit positions they explicitly capped, with no recourse. The taker address accrues debt without receiving assets (throwaway address scenario).

## Likelihood Explanation
All preconditions are attacker-controllable with no privileged access: `offer.buy == true` and `offer.maxAssets > 0` are standard buy offer parameters; `buyerPrice < WAD` holds for any tick below `MAX_TICK` (the vast majority of real offers); `consumed == maxAssets` is reachable by normal fills. The attack requires no flash loan, no oracle manipulation, and no privileged keys. It is repeatable in the same transaction via `multicall`.

## Recommendation
Add a guard in the `maxAssets` branch to reject takes that produce zero asset movement when the cap is already reached:

```solidity
if (offer.maxAssets > 0) {
    // Reject zero-asset takes that would bypass the cap
    require(offer.buy ? buyerAssets > 0 : sellerAssets > 0, ZeroAssets());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, add `require(units == 0 || buyerAssets > 0)` specifically for the buy+maxAssets path, or enforce a minimum `units` value such that `units * buyerPrice >= WAD`.

## Proof of Concept
The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` reproduces this exactly. Manual steps:
1. Deploy market with tick below `MAX_TICK`.
2. Create buy offer with `maxAssets = M`.
3. Fill offer to `consumed == M`.
4. Call `take(offer, ..., units=1)` ā€” observe `consumed` unchanged at `M`, but `buyerPos.credit` incremented by 1 and zero tokens transferred.
5. Repeat step 4 N times; `buyerPos.credit` grows to `M + N` with zero asset cost.
