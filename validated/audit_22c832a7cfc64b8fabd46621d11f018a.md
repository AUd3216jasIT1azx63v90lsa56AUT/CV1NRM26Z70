Audit Report

## Title
Unbounded `tokens[]` array in `flashLoan` enables zero-cost block-stuffing DoS - (File: src/Midnight.sol)

## Summary
The `flashLoan` function at lines 737–752 of `src/Midnight.sol` accepts caller-supplied `tokens[]` and `assets[]` arrays of arbitrary length, with no upper-bound check. Each iteration dispatches two external ERC20 calls via `SafeTransferLib`. An unprivileged attacker can pass a very large array with zero-value assets and stub contracts, consuming block gas without holding any tokens or requiring any permissions, enabling a repeatable block-stuffing DoS.

## Finding Description
The sole guard in `flashLoan` is `tokens.length == assets.length` (line 740). There is no cap on array length. The function then executes two loops of `tokens.length` iterations each (lines 742–744 and 749–751), each calling `SafeTransferLib.safeTransfer` and `SafeTransferLib.safeTransferFrom` respectively. Each of these performs an `EXTCODESIZE` check (`token.code.length > 0`) plus a `CALL` into the token contract (lines 13–15 and 25–27 of `src/libraries/SafeTransferLib.sol`).

`ConstantsLib.sol` defines `MAX_COLLATERALS = 128` and `MAX_COLLATERALS_PER_BORROWER = 16` for analogous array bounds elsewhere, but defines no `MAX_FLASH_LOAN_TOKENS` constant.

**Exploit flow:**
1. Attacker deploys a minimal ERC20 stub with code (satisfying `token.code.length > 0`) that accepts any `transfer`/`transferFrom` including zero-value and returns `true`.
2. Attacker deploys a callback stub that returns `CALLBACK_SUCCESS`.
3. Attacker calls `flashLoan(tokens=[stub]*N, assets=[0]*N, callback, "")` with N arbitrarily large.
4. The function executes 2N external calls, consuming gas linearly with N.
5. At N ≈ 3,000–5,000, the transaction approaches the Ethereum block gas limit (~30M gas).

**Why existing checks fail:** The `tokens.length == assets.length` check (line 740) is trivially satisfied. With `assets[i] = 0`, the protocol needs zero token balance for the outbound transfers, and the callback needs zero token balance for the inbound transfers. No access control exists on `flashLoan`.

## Impact Explanation
An attacker can submit repeated `flashLoan` transactions with large `tokens[]` arrays to consume the majority of consecutive blocks' gas budgets. This constitutes a block-stuffing DoS: legitimate users' transactions are delayed or excluded. The attacker requires zero token balance and zero protocol permissions. This matches the valid impact category of "service unavailability or severe degradation under realistic attacker input" per `RESEARCHER.md`.

## Likelihood Explanation
**Preconditions:** One-time deployment of two trivial stub contracts. **Feasibility:** On Ethereum mainnet, sustained block stuffing is costly but feasible for targeted short-duration attacks. On L2s (Arbitrum, Base, Optimism) where gas is cheap, this is highly practical and repeatable. **Repeatability:** No rate limiting, nonce, or cooldown exists on `flashLoan`.

## Recommendation
Add an array length cap using a new constant in `ConstantsLib.sol`, consistent with the existing pattern:

```solidity
uint256 constant MAX_FLASH_LOAN_TOKENS = 32; // or similar reasonable bound
```

Then enforce it at the top of `flashLoan`:

```solidity
require(tokens.length <= MAX_FLASH_LOAN_TOKENS, TooManyTokens());
```

This mirrors the existing `MAX_COLLATERALS` and `MAX_COLLATERALS_PER_BORROWER` bounds already applied to analogous array inputs.

## Proof of Concept
1. Deploy `StubERC20` with `transfer` and `transferFrom` returning `true` for any input including zero-value.
2. Deploy `StubCallback` with `onFlashLoan` returning `CALLBACK_SUCCESS`.
3. Construct `tokens = new address[](5000)` filled with `address(StubERC20)` and `assets = new uint256[](5000)` filled with `0`.
4. Call `midnight.flashLoan(tokens, assets, address(StubCallback), "")`.
5. Observe gas consumption approaching the block gas limit (~30M gas), with no token balance required by the attacker.

A Foundry fuzz test parameterizing `N` from 1 to 5000 and measuring `gasleft()` before and after the call will confirm linear gas growth with no revert.
