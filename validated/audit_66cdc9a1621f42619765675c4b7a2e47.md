Audit Report

## Title
Maker callback can call `setConsumed` mid-`take` to inflate consumed and grief taker multicall fills - (File: src/Midnight.sol)

## Summary
`setConsumed` has no reentrancy guard and no check that it is being called from within an active `take`. For a buy offer, `offer.callback` is the maker's address, and the maker's callback is invoked via `onBuy` after `consumed` has already been incremented and bounds-checked. A malicious maker callback can call `setConsumed` to inflate `consumed` to `maxAssets` during this window, causing every subsequent `take` of the same offer in the same `multicall` to revert with `ConsumedAssets`, rolling back the taker's entire atomic bundle.

## Finding Description

**Root cause confirmed at lines 723–728:** `setConsumed` has only two guards — authorization and non-decrease — with no transient lock or reentrancy protection: [1](#0-0) 

**Step 1 — `consumed` incremented and bounds-checked before any callback (lines 367–373):** [2](#0-1) 

**Step 2 — For `offer.buy == true`, `buyerCallback = offer.callback` (line 420), making the maker's contract the callback target:** [3](#0-2) 

**Step 3 — `onBuy` is invoked at lines 445–453 with no reentrancy protection on `setConsumed`:** [4](#0-3) 

**Exploit flow:**
1. Maker deploys malicious callback contract `M` and calls `setIsAuthorized(M, true, maker)`.
2. Maker creates a buy offer with `offer.callback = M` and `maxAssets = X`.
3. Taker calls `multicall` with two `take` calls on the same offer.
4. First `take`: `consumed` increments from `0` to `delta1`; bounds check passes. `onBuy` is called on `M`.
5. Inside `onBuy`, `M` calls `midnight.setConsumed(group, maxAssets, maker)`. Authorization passes (`isAuthorized[maker][M]`). Non-decrease check passes (`maxAssets >= delta1`). `consumed` is now `maxAssets`.
6. First `take` completes normally.
7. Second `take`: `consumed` increments from `maxAssets` to `maxAssets + delta2`. Bounds check fails: `maxAssets + delta2 > maxAssets` → reverts `ConsumedAssets`. The entire `multicall` reverts.

**Why existing checks fail:**

The `LIQUIDATION_LOCK_SLOT` / `tExchange` at line 444 is a transient lock specifically for preventing liquidation of the seller during callbacks — it provides no protection against `setConsumed` being called: [5](#0-4) 

The Certora rule `takeConsumedBoundedByMax` (lines 59–64) only asserts `consumed <= maxAssets` post-`take`. Inflating `consumed` to exactly `maxAssets` satisfies this assertion, so the rule does not catch the inflation: [6](#0-5) 

The rule `takeConsumedDelta` (lines 67–75) only applies to units mode (`offer.maxAssets == 0`) and does not cover the assets-mode exploit path: [7](#0-6) 

## Impact Explanation
A maker can grief any taker who attempts multiple partial fills of the same offer atomically via `multicall` or a bundler. The maker's callback inflates `consumed` to `maxAssets` after the first fill, causing every subsequent fill in the same transaction to revert. The taker's entire atomic bundle fails, wasting gas and preventing the intended fills. Atomic execution guarantees are broken for any taker relying on `multicall` for multi-fill bundles.

## Likelihood Explanation
Preconditions are trivial and permissionless: the maker deploys a callback contract and calls `setIsAuthorized(M, true, maker)` — both are normal protocol operations requiring no special privileges. The attack is deterministic and repeatable; the maker can deploy a fresh callback and offer with a new group for each target taker. Any taker using `multicall` for atomic multi-fill transactions is a potential victim.

## Recommendation
Add a transient reentrancy guard to `setConsumed` that prevents it from being called while a `take` is in progress. Specifically, set a transient flag at the start of `take` (similar to the existing `LIQUIDATION_LOCK_SLOT` pattern) and check for it in `setConsumed`, reverting if the flag is set. Alternatively, snapshot `consumed` at the start of `take` and re-validate the cap against the snapshot value after all callbacks complete, ignoring any mid-callback inflation.

## Proof of Concept
1. Deploy malicious callback contract `M` implementing `IBuyCallback`. In `onBuy`, call `midnight.setConsumed(group, offer.maxAssets, maker)`.
2. Maker calls `setIsAuthorized(address(M), true, maker)`.
3. Maker creates a buy offer: `offer.callback = address(M)`, `offer.maxAssets = 1000e18`, `offer.group = keccak256("g")`.
4. Taker calls `multicall` with two `take` calls: first for `500e18` assets, second for `100e18` assets.
5. Observe: first `take` succeeds; `M.onBuy` inflates `consumed` to `1000e18`; second `take` reverts `ConsumedAssets`; entire `multicall` reverts.
6. Confirm: without the malicious callback, both fills succeed atomically.

/// After a successful take in units mode, the change in consumed equals the units taken.
rule takeConsumedDelta(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    assert consumed(offer.maker, offer.group) == consumedBefore + units;
