Audit Report

## Title
Fee-on-Transfer Loan Token Breaks `withdrawable` Accounting in `liquidate()` and `repay()`, Causing Market Insolvency - (File: src/Midnight.sol)

## Summary
In `liquidate()` and `repay()`, `marketState[id].withdrawable` is incremented by the full nominal `repaidUnits`/`units` value before `SafeTransferLib.safeTransferFrom` is called. Because `SafeTransferLib` only checks for revert and a `bool` return without measuring actual tokens received, a fee-on-transfer loan token causes Midnight to record more withdrawable credit than it actually holds. Lenders who withdraw later will find the contract's real balance insufficient, making the market permanently insolvent.

## Finding Description

**Exact code path — `liquidate()`:** [1](#0-0) 

Accounting is updated first (full `repaidUnits`), then the collateral transfer and optional callback execute, and finally: [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only verifies the call does not revert and returns `true`; it performs no before/after balance measurement. If the loan token charges a 1% fee, Midnight receives `repaidUnits * 0.99` but `withdrawable` was increased by `repaidUnits`.

**Same pattern in `repay()`:** [3](#0-2) [4](#0-3) 

**`touchMarket()` imposes no restriction on `loanToken`:** [5](#0-4) 

Only collateral params, LLTV, maxLif, and maturity are validated. Any ERC20 address is accepted as `market.loanToken`.

**Certora formal verification explicitly excludes fee-on-transfer tokens:** [6](#0-5) 

The `tokenBalanceCorrect` solvency invariant is therefore not proven for fee-on-transfer loan tokens.

**`live_context.json` explicitly flags this as a required test surface with no code-level exclusion:** [7](#0-6) 

**Exploit flow:**
1. Attacker calls `touchMarket(market)` where `market.loanToken` is a fee-on-transfer ERC20 (e.g., 1% fee). Market creation is permissionless.
2. Lenders supply credit; borrowers take debt in this market via normal `take()` calls.
3. Borrower's position becomes liquidatable (price drop or post-maturity).
4. Liquidator calls `liquidate(...)` with `repaidUnits = X`.
5. Line 675: `_marketState.withdrawable += X` — accounting records full `X`.
6. Line 676: `_position.debt -= X` — borrower's debt fully reduced.
7. Line 717: `safeTransferFrom(loanToken, payer, address(this), X)` — fee-on-transfer token delivers only `X * 0.99` to Midnight. Call succeeds.
8. Post-call: `withdrawable` exceeds actual `loanToken` balance by `X * 0.01`. Gap widens with every subsequent `liquidate()` or `repay()` call.

## Impact Explanation

After one or more liquidations or repayments with a fee-on-transfer loan token, `marketState[id].withdrawable` permanently exceeds the contract's actual `loanToken` balance. Lenders calling `withdraw()` trigger `safeTransfer(loanToken, receiver, units)` — early withdrawers succeed; later lenders find the real balance insufficient and their calls revert. The market is insolvent: lenders collectively cannot redeem all their credit. The shortfall equals the cumulative fee taken across all affected `liquidate()` and `repay()` calls. This matches the "protocol insolvency" and "credit/debt accounting corruption" impact classes listed as highest priority in `live_context.json`. [8](#0-7) 

## Likelihood Explanation

**Preconditions:** A market with a fee-on-transfer loan token must exist and have at least one liquidation or repayment. Market creation is fully permissionless — any user can call `touchMarket()` with any ERC20 as `loanToken`. The protocol explicitly supports "arbitrary loan token" as a design feature. [9](#0-8) 

No privileged action is required. The attacker can be the market creator, a lender, a borrower, or a liquidator. The exploit is repeatable: every `liquidate()` or `repay()` call with `repaidUnits > 0` widens the accounting gap. Fee-on-transfer tokens (e.g., USDT with fee enabled, STA, PAXG) are real, deployed ERC20s.

## Recommendation

1. **Measure actual received tokens** using a balance-before/after pattern in both `liquidate()` and `repay()`:
   ```solidity
   uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
   SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
   uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
   require(received == repaidUnits, FeeOnTransferNotSupported());
   ```
   Or alternatively, use `received` instead of `repaidUnits` when updating `withdrawable` and `debt`.

2. **Alternatively**, add an explicit exclusion in `touchMarket()` by requiring the loan token to be on an allowlist, or document and enforce that fee-on-transfer tokens are unsupported at the market creation level.

## Proof of Concept

**Foundry test outline:**

```solidity
// 1. Deploy a MockFeeToken with 1% transferFrom fee
// 2. Create a market with loanToken = address(feeToken)
// 3. Lender supplies 1000 units; borrower takes debt of 1000 units
// 4. Drop oracle price to make position liquidatable
// 5. Liquidator calls liquidate(..., repaidUnits=1000, ...)
//    - feeToken delivers 990 to Midnight
//    - marketState[id].withdrawable == 1000 (overcounted by 10)
// 6. Assert: feeToken.balanceOf(address(midnight)) < marketState[id].withdrawable
// 7. Lender calls withdraw(1000) — succeeds (drains real balance)
// 8. Second lender calls withdraw(any) — reverts (balance exhausted)
// Invariant broken: sum of lender credits > actual token balance
```

The same test applies to `repay()` by having the borrower repay instead of being liquidated.

        bytes32 id = toId(market);
        if (marketState[id].tickSpacing == 0) {
            require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
            require(market.collateralParams.length > 0, NoCollateralParams());

            address previousCollateralToken;

                    InvalidMaxLif()
