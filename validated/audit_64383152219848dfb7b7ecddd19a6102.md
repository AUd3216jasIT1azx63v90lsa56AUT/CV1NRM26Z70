### Title
Missing deadline parameter in `withdraw()` permanently locks user funds when transaction is delayed past `ATTACK_REQUESTED` → `UNDER_ATTACK` transition — (File: src/ConfidencePool.sol)

### Summary
`withdraw()` reads the live registry state at execution time and permanently disables withdrawals once the registry reaches `UNDER_ATTACK`. Because there is no deadline parameter, a user who submits a withdrawal while the registry is in `ATTACK_REQUESTED` (a permitted state) may have their transaction delayed until the registry transitions to `UNDER_ATTACK`, at which point the call reverts and their funds are permanently locked until pool resolution — potentially months away, and with full principal loss possible under a bad-faith CORRUPTED outcome.

### Finding Description
`withdraw()` gates on the live registry state read inside `_observePoolState()`:

```solidity
IAttackRegistry.ContractState state = _observePoolState();
if (
    riskWindowStart != 0
        || (state != IAttackRegistry.ContractState.NOT_DEPLOYED
            && state != IAttackRegistry.ContractState.NEW_DEPLOYMENT
            && state != IAttackRegistry.ContractState.ATTACK_REQUESTED)
) {
    revert WithdrawsDisabled();
}
``` [1](#0-0) 

The function accepts no deadline or `latestWithdrawTime` parameter. A user who observes the registry in `ATTACK_REQUESTED` and submits a withdrawal transaction has no on-chain mechanism to express "execute only if the registry is still in a permitted state." If the transaction sits in the mempool — due to gas price fluctuation, network congestion, or validator ordering — and the registry transitions to `UNDER_ATTACK` before execution, the call reverts with `WithdrawsDisabled`. The one-way `riskWindowStart` latch then seals permanently on the next successful `_observePoolState()` call by any actor, making re-entry impossible even if the registry were to rewind. [2](#0-1) 

DESIGN.md §9 documents the one-way closure as intentional but does not acknowledge the TX-delay scenario or the absence of a deadline as an accepted trade-off. [3](#0-2) 

### Impact Explanation
The user's entire eligible stake is permanently locked until pool resolution. Under a bad-faith CORRUPTED outcome, `claimCorrupted()` sweeps the full pool — including the user's principal — to `recoveryAddress`, resulting in total loss of principal. Even under SURVIVED or EXPIRED, the user is locked for the remaining pool duration (up to `expiry`, which must be at least 30 days from deploy, and up to `expiry + 180 days` for the CORRUPTED backstop path). The user had no intent to bear this risk and had no on-chain tool to prevent it.

### Likelihood Explanation
The `ATTACK_REQUESTED` state is a defined pre-attack staging window during which withdrawals are explicitly permitted. Users who monitor the registry and wish to exit before risk materializes will naturally submit withdrawals during this window. Gas price spikes and mempool delays are routine on EVM chains; no adversarial action is required. The closer the submission is to the `ATTACK_REQUESTED` → `UNDER_ATTACK` transition, the higher the probability the TX executes in the wrong state. This is a probabilistic, non-targeted risk that grows with pool usage.

### Recommendation
Add a `latestWithdrawTime` deadline parameter to `withdraw()` and revert at the top of the function if `block.timestamp > latestWithdrawTime`. This mirrors the standard deadline pattern used in AMM swap functions and gives users a trustless guarantee that their transaction will not execute under unintended registry conditions:

```solidity
function withdraw(uint256 latestWithdrawTime) external nonReentrant {
    if (block.timestamp > latestWithdrawTime) revert DeadlineExceeded();
    // ... existing logic
}
```

### Proof of Concept
1. Registry is in `ATTACK_REQUESTED`. User has `eligibleStake[user] > 0` and wants to exit before risk materializes.
2. User submits `withdraw()`. Gas price spikes; the transaction sits in the mempool.
3. The BattleChain registry transitions from `ATTACK_REQUESTED` to `UNDER_ATTACK`.
4. User's transaction executes. `_observePoolState()` returns `UNDER_ATTACK` (an active-risk state), which is not in `{NOT_DEPLOYED, NEW_DEPLOYMENT, ATTACK_REQUESTED}`.
5. The condition at line 293–300 is true; the call reverts with `WithdrawsDisabled`.
6. `_markRiskWindowStart()` is called inside `_observePoolState()` before the revert — but since the whole transaction reverts, `riskWindowStart` is not yet set. However, the next call to any pool entrypoint (e.g., `pokeRiskWindow()`, `stake()` by another user) will set the one-way latch.
7. User's funds are permanently locked. Under a subsequent bad-faith CORRUPTED flag, `claimCorrupted()` sweeps the full pool to `recoveryAddress`, and the user loses their entire principal. [4](#0-3)

        IAttackRegistry.ContractState state = _observePoolState();
        // `riskWindowStart` is the pool's one-way record that risk has materialised;

        if (
            riskWindowStart != 0
                || (state != IAttackRegistry.ContractState.NOT_DEPLOYED
                    && state != IAttackRegistry.ContractState.NEW_DEPLOYMENT
                    && state != IAttackRegistry.ContractState.ATTACK_REQUESTED)

```markdown
## 9. Withdraw lifecycle

`withdraw()` exits the caller's full eligible stake in one call. It is gated by registry state:
allowed in `NOT_DEPLOYED`, `NEW_DEPLOYMENT`, and `ATTACK_REQUESTED`; permanently disabled from
`UNDER_ATTACK` onward (gated additionally on the one-way `riskWindowStart != 0` latch, so an
upstream registry rewind cannot re-open it).

This closes the race in which a staker observes an attack on-chain and front-runs `flagOutcome`
with `withdraw()` to escape with full value. Withdrawing before a flag forfeits any bonus the
caller would have earned (the claim paths require nonzero `eligibleStake`).

The withdraw escape hatch (open for the entire pre-attack window) is also why "no observed risk →
no bonus" is fair: a staker only forfeits the exit option once risk has actually materialized,
which is exactly when they begin earning the risk premium. A sponsor cannot grief stakers by
keeping the agreement out of attackable mode — stakers can freely exit until risk materializes.

```
