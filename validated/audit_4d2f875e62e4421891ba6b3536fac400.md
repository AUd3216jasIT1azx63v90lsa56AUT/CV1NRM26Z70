Audit Report

## Title
Continuous Fee Permanently Unclaimable When `withdrawable < continuousFeeCredit` Due to Missing Guard in `claimContinuousFee` - (File: src/Midnight.sol)

## Summary
`claimContinuousFee` unconditionally decrements `_marketState.withdrawable` by `amount` at line 320, but `_updatePosition` increases `continuousFeeCredit` without ever increasing `withdrawable`. In any market where borrowers have not fully repaid, `withdrawable` will be less than `continuousFeeCredit`, causing every `claimContinuousFee` call to revert with an arithmetic underflow. Legitimately accrued continuous fees are permanently unclaimable for the entire duration of outstanding borrower debt.

## Finding Description

**Root cause — `claimContinuousFee` (`src/Midnight.sol:312-325`):**

The function decrements `withdrawable` at line 320 with no prior guard:

```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount); // line 318
_marketState.totalUnits -= UtilsLib.toUint128(amount);          // line 319
_marketState.withdrawable -= UtilsLib.toUint128(amount);        // line 320 — no guard
```

The only pre-conditions checked are `msg.sender == feeClaimer` and `tickSpacing > 0`. Solidity 0.8 checked arithmetic causes line 320 to revert whenever `withdrawable < amount`. [1](#0-0) 

**How `continuousFeeCredit` grows without `withdrawable` growing:**

`_updatePosition` at line 846 increments `continuousFeeCredit` by `accruedFee` but makes no change to `withdrawable`:

```solidity
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
``` [2](#0-1) 

`withdrawable` only increases in two places: `repay` (line 509, `+= units`) and `liquidate` (line 675, `+= repaidUnits`). [3](#0-2) [4](#0-3) 

The Certora spec `withdrawableUnchanged` in `certora/specs/WithdrawableMonotonicity.spec` formally confirms that `updatePosition` (and `take`) leave `withdrawable` unchanged — it is excluded from the filter precisely because it does not modify `withdrawable`: [5](#0-4) 

**Exploit path (no privileged action needed to reach the bad state):**

1. Lender provides `C` units via `take`. State: `withdrawable = 0`, `continuousFeeCredit = 0`.
2. Time passes. Anyone calls `updatePosition(market, lender)`. Lender's credit decreases by `fee`; `continuousFeeCredit += fee`. `withdrawable` remains `0`.
3. Borrower does **not** repay. State: `continuousFeeCredit = fee > 0`, `withdrawable = 0`.
4. feeClaimer calls `claimContinuousFee(market, fee, receiver)` with `fee <= continuousFeeCredit`.
5. Line 318: `continuousFeeCredit -= fee` → OK.
6. Line 319: `totalUnits -= fee` → OK.
7. Line 320: `withdrawable -= fee` → **arithmetic underflow, revert**.

**Formal verification independently confirms the revert condition:**

The Certora liveness rule `feeClaimerCanClaimContinuousFee` in `certora/specs/Role.spec:264` asserts that success requires `amount <= withdrawableBefore` — a constraint the code never enforces as a pre-condition, relying instead on the implicit underflow revert: [6](#0-5) 

**The existing test silently works around the bug:**

`testClaimContinuousFee` in `test/ContinuousFeeTest.sol` calls `repay` before claiming, with the explicit comment *"Repay so withdrawable covers the claim."* This masks the bug in the test suite — no test exercises the normal case where fees are claimed without prior full repayment: [7](#0-6) 

## Impact Explanation
Accrued continuous fees are permanently unclaimable whenever `withdrawable < continuousFeeCredit`. The feeClaimer receives zero revenue from any market with outstanding borrower debt. In markets with defaulted borrowers or post-maturity bad debt, fees may never become claimable. This constitutes a complete loss of protocol fee revenue for the duration of any active loan, which is the normal operating state of the protocol.

## Likelihood Explanation
The condition `continuousFeeCredit > withdrawable` is the **default state** of every active lending market from the moment the first fee accrues. No special preconditions, attacker inputs, or oracle manipulation are required. The state is reached by ordinary protocol usage (lend → time passes → `updatePosition`) and persists for the entire life of the loan. The feeClaimer is a trusted role, but the inability to claim is triggered by normal user behavior (borrowers not repaying), not by any privileged action.

## Recommendation
Two viable fixes:

1. **Do not decrement `withdrawable` in `claimContinuousFee`**: The fee was already deducted from lender credit in `_updatePosition`; the corresponding tokens remain locked as borrower debt and are not in the withdrawable pool. Remove line 320 (`_marketState.withdrawable -= ...`) and track fee claims solely against `continuousFeeCredit` and `totalUnits`. The `SafeTransferLib.safeTransfer` at line 324 will naturally revert if the contract lacks sufficient token balance, providing the real solvency guard.

2. **Increment `withdrawable` in `_updatePosition`**: When `continuousFeeCredit += accruedFee` is applied, also apply `withdrawable += accruedFee`. This treats accrued fees as immediately withdrawable, consistent with the accounting model used by `repay` and `liquidate`. This requires verifying that the contract's token balance invariant (`balance >= withdrawable + claimableSettlementFee`) is maintained.

Option 1 is the minimal, lower-risk fix.

## Proof of Concept
Minimal Foundry test (no repay before claim):

```solidity
function testClaimContinuousFeeRevertsWithoutRepay() public {
    uint256 credit = 1e18;
    uint256 feeRate = 0.01e18;
    uint256 ttm = 365 days;

    setupLender(credit, feeRate, ttm);
    vm.warp(vm.getBlockTimestamp() + 30 days);
    midnight.updatePosition(market, lender);

    uint256 feeAmount = midnight.continuousFeeCredit(id);
    assertGt(feeAmount, 0, "fee accrued");
    assertEq(midnight.withdrawable(id), 0, "withdrawable is zero");

    // feeClaimer attempts to claim — reverts with arithmetic underflow
    vm.prank(feeClaimer);
    vm.expectRevert(); // underflow on withdrawable -= amount
    midnight.claimContinuousFee(market, feeAmount, feeClaimer);
}
```

This test requires no privileged setup beyond the normal `feeClaimer` role and reproduces the revert on any market where the borrower has not repaid.
