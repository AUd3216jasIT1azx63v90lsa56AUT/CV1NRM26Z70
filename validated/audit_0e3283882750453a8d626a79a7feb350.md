Audit Report

## Title
Referral fee not budgeted within `maxBuyerAssets`, causing revert that blocks a valid fill - (File: src/periphery/MidnightBundles.sol)

## Summary
In `buyWithUnitsTargetAndWithdrawCollateral`, the bundler pulls exactly `maxBuyerAssets` from `msg.sender` and spends `filledBuyerAssets` of that in the take loop. The referral fee (`referralFeeAssets`) is then computed as `filledBuyerAssets * referralFeePct / (WAD - referralFeePct)` and transferred out of the bundler's remaining balance. When `maxBuyerAssets == filledBuyerAssets` and `referralFeePct > 0`, the bundler's post-loop balance is zero, causing the `safeTransfer` to the referrer at line 103 to revert with an ERC20 insufficient-balance error. The user loses gas and cannot complete an otherwise valid fill without adjusting parameters.

## Finding Description
**Code path:**

At line 66, the bundler pulls exactly `maxBuyerAssets` from `msg.sender`: [1](#0-0) 

The take loop (lines 71–86) causes MIDNIGHT to pull `resBuyerAssets` from the bundler per iteration. After the loop completes, the bundler holds `maxBuyerAssets - filledBuyerAssets` tokens: [2](#0-1) 

At line 102, `referralFeeAssets` is computed as `filledBuyerAssets * referralFeePct / (WAD - referralFeePct)`. When `referralFeePct > 0` and `filledBuyerAssets > 0`, this is strictly positive. The contract then attempts to transfer `referralFeeAssets` from the bundler's balance to the referrer: [3](#0-2) 

If `maxBuyerAssets == filledBuyerAssets`, the bundler holds zero tokens at line 103, so the ERC20 transfer fails and `safeTransfer` propagates the revert. The subsequent subtraction `maxBuyerAssets - filledBuyerAssets - referralFeeAssets` at line 104 would also underflow under Solidity 0.8.34 checked arithmetic, but it is never reached.

**Root cause:** No check enforces `maxBuyerAssets >= filledBuyerAssets + referralFeeAssets`. The NatSpec at lines 47–48 documents the total cost formula correctly: [4](#0-3) 

but the contract never validates that `maxBuyerAssets` is large enough to cover both the fill and the referral fee before attempting the transfer.

**Existing guards** (lines 60–61) only check taker authorization and `referralFeePct < WAD`: [5](#0-4) 

Neither prevents the revert scenario.

## Impact Explanation
All offers are successfully taken (`filledUnits == targetUnits` passes at line 88), collateral withdrawals complete, but the transaction reverts at the referral-fee transfer (line 103). The user loses gas and cannot complete the fill in a single call even though the market state is valid and the fill is economically sound. Because the revert is deterministic for the given `(maxBuyerAssets, referralFeePct)` pair, the user is blocked unless they increase `maxBuyerAssets` or set `referralFeePct = 0`. This constitutes a service unavailability / severe degradation of a core protocol function for any user employing tight slippage with a nonzero referral fee.

## Likelihood Explanation
Any unprivileged taker who sets `maxBuyerAssets` equal to the expected fill cost (tight slippage, a common and reasonable pattern) and passes a nonzero `referralFeePct` hits this revert. No privileged role, special state, or external dependency is required. The precondition is trivially reachable and repeatable across every call with these parameters.

## Recommendation
Validate before the referral-fee transfer that the bundler's remaining balance is sufficient, or require `maxBuyerAssets >= filledBuyerAssets + referralFeeAssets` explicitly:

```solidity
uint256 referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
require(maxBuyerAssets >= filledBuyerAssets + referralFeeAssets, InsufficientBuyerAssets());
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - filledBuyerAssets - referralFeeAssets);
```

Alternatively, document clearly that `maxBuyerAssets` must cover `filledBuyerAssets + referralFeeAssets` and add an off-chain helper to compute the correct value.

## Proof of Concept
1. Deploy `MidnightBundles` against a local fork with a standard ERC20 loan token.
2. Create a sell offer for `N` units at price `P` (so `filledBuyerAssets = N * P`).
3. Call `buyWithUnitsTargetAndWithdrawCollateral` with:
   - `targetUnits = N`
   - `maxBuyerAssets = N * P` (tight, equal to expected fill cost)
   - `referralFeePct = 1e16` (1%, any nonzero value `< WAD`)
   - `referralFeeRecipient = any address`
4. Observe: the take loop succeeds, `filledUnits == targetUnits` passes at line 88, collateral withdrawals complete, but the transaction reverts at line 103 with an ERC20 insufficient-balance error because the bundler holds 0 tokens after the fill.
5. Confirm fix: repeat with `maxBuyerAssets = N * P + (N * P * referralFeePct / (WAD - referralFeePct))`; the transaction succeeds.
