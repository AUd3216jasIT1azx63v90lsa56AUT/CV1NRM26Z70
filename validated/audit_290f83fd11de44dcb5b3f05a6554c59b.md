### Title
Continuous Fee Avoidance via Zero-Rounding in `updatePositionView` — (File: src/Midnight.sol)

---

### Summary

The `updatePositionView` function computes the continuous fee accrual as `postSlashPendingFee * elapsed / remaining`, which truncates to zero via integer division when `pendingFee * elapsed < (maturity - lastAccrual)`. Critically, `_updatePosition` unconditionally advances `_position.lastAccrual` to `block.timestamp` even when the computed `fee` is zero. A lender can exploit this by triggering position updates at intervals short enough to keep `fee = 0`, then withdrawing credit in the same block as the final update — cancelling the entire `pendingFee` without any of it ever reaching `continuousFeeCredit`. The fee claimer loses the continuous fee revenue that should have accrued.

---

### Finding Description

**Root cause — division truncation in `updatePositionView`:**

```solidity
// src/Midnight.sol:814-816
uint128 fee = _lastAccrual < market.maturity
    ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
    : 0;
```

`mulDivDown` is plain integer division: `(x * y) / d`. When `postSlashPendingFee * (accrualEnd - _lastAccrual) < (market.maturity - _lastAccrual)`, the result is `0`.

**State advancement regardless of zero fee — `_updatePosition`:**

```solidity
// src/Midnight.sol:845
_position.lastAccrual = uint128(block.timestamp);
```

`lastAccrual` is written unconditionally. When `fee = 0`, `accruedFee = 0` is added to `continuousFeeCredit` (line 846), and `pendingFee` is unchanged — but the accrual window has been consumed.

**Fee cancellation on withdrawal — `withdraw`:**

```solidity
// src/Midnight.sol:489-491
if (_position.credit > 0) {
    pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
    _position.pendingFee -= pendingFeeDecrease;
}
```

`pendingFeeDecrease` is subtracted from `pendingFee` but is **never added to `continuousFeeCredit`**. It is simply discarded. This is the intended mechanism for cancelling the "future" portion of the fee when a lender exits early — but it becomes exploitable when the "past" portion was never accrued due to rounding.

**End-to-end exploit path:**

The condition for `fee = 0` at each interaction is:

```
pendingFee * elapsed < (maturity - lastAccrual)
⟺  elapsed < (maturity - lastAccrual) / pendingFee
```

A lender with `pendingFee = P` can keep `fee = 0` by interacting at intervals shorter than `(maturity - lastAccrual) / P` seconds. After N such interactions, `lastAccrual ≈ block.timestamp` but `pendingFee` is still `P`. The lender then calls `withdraw` in the same block as the last update: `_updatePosition` computes `fee = P * 0 / (maturity - lastAccrual) = 0`, leaving `pendingFee = P`. The subsequent `pendingFeeDecrease = P` zeroes out `pendingFee` without crediting the protocol.

**`pendingFee` is set at trade time:**

```solidity
// src/Midnight.sol:385-386
uint128 buyerPendingFeeIncrease =
    UtilsLib.toUint128(buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD));
```

`MAX_CONTINUOUS_FEE = uint32(0.01e18 / 365 days) ≈ 317,097,919` (≈1 % per year). For a lender with `credit = C` units, `pendingFee ≈ C * 0.01` for a one-year market.

---

### Impact Explanation

The fee claimer loses the continuous fee revenue that should have been collected from lenders who exploit this path. The maximum loss per position equals `pendingFee = credit * continuousFee * timeToMaturity / WAD`. At the maximum fee rate (1 %/year) on a one-year market:

| Credit (USDC, 6 dec) | pendingFee (units) | Max loss (USDC) | Required interaction interval (Ethereum, 12 s blocks) |
|---|---|---|---|
| 1,000 USDC | ~10,000 | ~0.01 USDC | < 3,153 s (~52 min) — trivially feasible |
| 100,000 USDC | ~1,000,000 | ~1 USDC | < 31.5 s — feasible (3 blocks) |
| 1,000,000 USDC | ~10,000,000 | ~10 USDC | < 3.15 s — infeasible on Ethereum mainnet |

On L2s with sub-second block times and negligible gas costs, the threshold shifts dramatically upward, making the exploit economically viable for large positions. The absolute loss per position is bounded by `pendingFee`, but across many positions in a high-volume market the aggregate loss to the fee claimer is material.

---

### Likelihood Explanation

Any lender can execute this without any privileged access. The only cost is gas for the repeated `updatePosition` or `take` calls that advance `lastAccrual`. On Ethereum mainnet the gas cost exceeds the fee savings for large positions, but on EVM-compatible L2s (Arbitrum, Base, Optimism) with cheap gas and fast blocks the attack is economically rational for positions above ~10,000 USDC. The protocol's own comment at line 115 acknowledges the vector: *"pendingFee updates are rounded in favor of the user. It could lead to fees manipulations too."* — confirming the team is aware but has not applied the M-9-style fix of skipping the `lastAccrual` update when `fee = 0`.

---

### Recommendation

Mirror the fix applied to M-9 in Surge: **do not advance `lastAccrual` when the computed `fee` is zero**. In `_updatePosition`, guard the timestamp write:

```solidity
// Only advance lastAccrual when a non-zero fee was actually accrued
if (accruedFee > 0 || accrualEnd == _position.lastAccrual) {
    _position.lastAccrual = uint128(block.timestamp);
}
```

This ensures that a zero-fee interaction does not consume the accrual window, so the next interaction will include the skipped elapsed time in its numerator and eventually produce a non-zero fee. Alternatively, store `pendingFee` with extra precision (e.g., scaled by `1e18`) to reduce the rounding error per interaction to negligible levels.

### Proof of Concept

**Setup:**
- Loan token: USDC (6 decimals)
- `continuousFee` = `MAX_CONTINUOUS_FEE` ≈ 317,097,919
- Market maturity: 1 year from now (`T = block.timestamp + 31,536,000`)
- Lender acquires `credit = 1,000,000` USDC units (1 USDC)
- `pendingFee = 1,000,000 * 317,097,919 * 31,536,000 / 1e18 ≈ 9,997` units

**Condition for `fee = 0`:**
`9,997 * elapsed < 31,536,000 ⟹ elapsed < 3,154 seconds (~52 minutes)`

**Steps:**
1. Every ~50 minutes, call `updatePosition(market, lender)`. Each call: `fee = 9,997 * 3,000 / 31,536,000 = 0` (integer division). `lastAccrual` advances. `pendingFee` stays at 9,997. `continuousFeeCredit` receives 0.
2. After ~8,760 such calls (one year), `lastAccrual ≈ T - 3,000`.
3. In the same block as the final `updatePosition`, call `withdraw(market, credit, lender, receiver)`.
4. Inside `withdraw`: `_updatePosition` computes `fee = 9,997 * 0 / 3,000 = 0`. `pendingFee` remains 9,997.
5. `pendingFeeDecrease = 9,997 * credit / credit = 9,997`. `pendingFee → 0`. `continuousFeeCredit` unchanged.
6. **Result:** Lender paid 0 continuous fees. Fee claimer lost 9,997 units (~0.01 USDC) for this position. Scaled to 100,000 USDC credit: ~1 USDC loss per year per position, with gas costs on L2 of ~$0.001 per call × 8,760 calls = ~$8.76 — still profitable.
