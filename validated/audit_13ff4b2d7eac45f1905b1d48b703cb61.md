Looking at the exact code paths in `src/Midnight.sol` and `certora/specs/Consume.spec` to validate each claim.

Audit Report

## Title
Maker callback can call `setConsumed` mid-`take` to inflate consumed and grief taker multicall fills - (File: src/Midnight.sol)

## Summary
In `take`, `consumed[offer.maker][offer.group]` is incremented and bounds-checked before any external callback is invoked. For a buy offer, `offer.callback` is the maker's own address, and `setConsumed` has no guard against being called during an active `take`. A maker-controlled callback can therefore re-enter `setConsumed` to inflate `consumed` to `maxAssets` after the first fill, causing every subsequent fill of the same offer within the same `multicall` to revert with `ConsumedAssets`, atomically reverting the taker's entire bundle.

## Finding Description

**Step 1 — consumed is incremented and capped before any callback:**

`take` writes and bounds-checks `consumed` at lines 367–373, before any external call is made. [1](#0-0) 

**Step 2 — For a buy offer, `buyerCallback` is the maker's own address:**

`offer.callback` is assigned to `buyerCallback` when `offer.buy == true`. [2](#0-1) 

**Step 3 — The maker's `onBuy` callback is invoked with no reentrancy guard on `setConsumed`:**

`IBuyCallback(buyerCallback).onBuy(...)` is called at lines 448–452. There is no transient lock or mutex preventing `setConsumed` from being called during this external call. [3](#0-2) 

**Step 4 — `setConsumed` only requires authorization and a non-decreasing amount:**

`setConsumed` passes as long as `msg.sender` is authorized for `onBehalf` and `amount >= consumed[onBehalf][group]`. There is no check that it is not being called during an active `take`. [4](#0-3) 

**Exploit flow:**
1. Maker deploys malicious contract `M` and calls `setIsAuthorized(M, true, maker)` — a normal, permissionless operation.
2. Maker publishes a buy offer with `offer.callback = M` and `offer.maxAssets = 1000`.
3. Taker submits `multicall([take(offer, 400 units), take(offer, 400 units)])`.
4. First `take`: `consumed[maker][group] += 400` → 400 ≤ 1000, passes.
5. `M.onBuy(...)` is called. Inside, `M` calls `midnight.setConsumed(group, 1000, maker)`. Authorization check passes (`isAuthorized[maker][M] == true`). `AlreadyConsumed` check passes (`1000 >= 400`). `consumed[maker][group]` is now 1000.
6. First `take` completes normally.
7. Second `take`: `consumed[maker][group] += 400` → 1400 > 1000 → reverts `ConsumedAssets`.
8. Entire `multicall` reverts; taker's atomic bundle fails and all gas is wasted.

**Why existing checks fail:**
- `AlreadyConsumed` (line 725) only prevents *decreasing* consumed; it explicitly allows any upward inflation, so the re-entrant call passes unconditionally.
- There is no transient lock or reentrancy guard on `setConsumed` during `take`.
- The Certora `Consume.spec` rules `takeConsumedBoundedByMax` and `takeConsumedDelta` (lines 59–75) do not catch this because `multicall` is summarized as `HAVOC_ALL DELETE` (line 4), meaning the tool does not model the callback re-entering `setConsumed` mid-`take`. [5](#0-4) [6](#0-5) 

## Impact Explanation
A maker can deterministically grief any taker who attempts multiple partial fills of the same offer atomically via `multicall` or a bundler. The entire atomic bundle reverts, wasting the taker's gas and preventing the intended fill. The maker's offer is also permanently blocked from further fills (self-harm griefing). No direct fund loss occurs, but the attack constitutes a concrete, repeatable denial-of-service against atomic multi-fill workflows, which are a core use case of the protocol's `multicall` and group/offer-cap design.

## Likelihood Explanation
All preconditions are normal, permissionless protocol operations: deploying a callback contract and calling `setIsAuthorized` are both standard flows already used in the protocol (e.g., `BorrowCallback`, `LendCallback` patterns). No special privileges, oracle manipulation, or token-owner action is required. The attack is deterministic — it always succeeds given the preconditions — and the maker can deploy a fresh callback and offer for each target taker. Any taker using `multicall` or a bundler for multi-fill atomic transactions is a potential victim.

## Recommendation
Add a transient reentrancy guard to `setConsumed` that reverts if called during an active `take` execution. The simplest approach is to use a transient storage slot (already used in the codebase for `LIQUIDATION_LOCK_SLOT`) to mark that a `take` is in progress, and require that slot to be unset at the entry of `setConsumed`. Alternatively, snapshot `consumed` into a local variable at the start of `take` and re-validate the cap against that snapshot after the callback returns, ignoring any external mutations made during the callback window. [7](#0-6) 

## Proof of Concept
**Minimal Foundry test plan:**
1. Deploy `Midnight` and a mock loan token.
2. Deploy malicious callback contract `M` that, inside `onBuy`, calls `midnight.setConsumed(offer.group, offer.maxAssets, maker)`.
3. Maker calls `setIsAuthorized(address(M), true, maker)`.
4. Maker creates a buy offer: `offer.callback = address(M)`, `offer.maxAssets = 1000`, `offer.group = bytes32("g1")`.
5. Taker calls `midnight.multicall([abi.encodeCall(take, (offer, ..., 400, ...)), abi.encodeCall(take, (offer, ..., 400, ...))])`.
6. Assert the call reverts with `ConsumedAssets`.
7. Confirm that calling each `take` individually (not via `multicall`) succeeds for the first and fails for the second (since `consumed` was inflated to 1000 by `M` during the first fill).

**Expected result:** Step 6 reverts, confirming the griefing. Step 7 confirms the inflation persists after the first individual fill. [8](#0-7) [4](#0-3)

rule takeConsumedBoundedByMax(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert offer.maxAssets > 0 => consumed(offer.maker, offer.group) <= offer.maxAssets;
    assert offer.maxAssets == 0 => consumed(offer.maker, offer.group) <= offer.maxUnits;
}

/// After a successful take in units mode, the change in consumed equals the units taken.
rule takeConsumedDelta(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumed(offer.maker, offer.group) == consumedBefore + units;
}
