### Title
Lenders Can Front-Run Bad Debt Realization to Avoid Pro-Rated Loss — (`src/Midnight.sol`)

---

### Summary

When bad debt is realized via `liquidate()`, the `lossFactor` is updated globally and all lenders are slashed proportionally at their next interaction. However, a lender who withdraws credit units before the `liquidate()` transaction executes avoids their share of the loss entirely. Because `withdraw()` charges no fee and requires only that `withdrawable >= units`, any lender who monitors the mempool can front-run a bad-debt liquidation and leave remaining lenders to absorb a disproportionately larger loss.

---

### Finding Description

**Bad debt socialization mechanism:**

When `liquidate()` realizes bad debt, it updates the global `lossFactor`:

```solidity
// Midnight.sol lines 631-634
_marketState.lossFactor = UtilsLib.toUint128(
    type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
);
_marketState.totalUnits -= UtilsLib.toUint128(badDebt);
``` [1](#0-0) 

Each lender's credit is only slashed at their next interaction via `_updatePosition()` / `updatePositionView()`:

```solidity
// Midnight.sol lines 805-807
uint256 postSlashCredit = _lastLossFactor < type(uint128).max
    ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
    : 0;
``` [2](#0-1) 

**Withdrawal path:**

`withdraw()` calls `_updatePosition()` first, then reduces credit and transfers tokens:

```solidity
// Midnight.sol lines 481-500
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    ...
    _updatePosition(market, id, onBehalf);
    ...
    _position.credit -= UtilsLib.toUint128(units);
    _marketState.withdrawable -= UtilsLib.toUint128(units);
    _marketState.totalUnits -= UtilsLib.toUint128(units);
    ...
    SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
}
``` [3](#0-2) 

**No withdrawal fee:** There is no fee charged on `withdraw()`. The only fee in the protocol is the settlement fee charged at `take()` time.

**Attack path:**

1. Market has `withdrawable > 0` (from prior repayments or partial liquidations — a normal market condition).
2. A borrower's position has bad debt. A liquidator submits `liquidate()` to the mempool.
3. A malicious lender with `C` credit units sees the pending transaction.
4. The lender front-runs with `withdraw(C)`: `_updatePosition()` reads the current (pre-bad-debt) `lossFactor`, so the lender receives full face value `C` loan tokens.
5. `liquidate()` executes: `lossFactor` increases, `totalUnits` decreases. The bad debt `B` is now socialized only among the remaining lenders.
6. The malicious lender optionally re-enters by taking a sell offer (paying only the small settlement fee).

The protocol itself acknowledges a related race condition at lines 27–29:

> "When some assets become withdrawable before maturity (after a repayment or a liquidation), there is an incentive to take resting sell offers with price < 1 and withdraw instantly. Lenders (and the fee claimer) might also race to withdraw first." [4](#0-3) 

However, this comment addresses a different arbitrage scenario (taking cheap sell offers). The bad-debt front-running attack is a distinct, unmitigated vector.

---

### Impact Explanation

Remaining lenders bear a larger fraction of bad debt than they would if all lenders shared the loss proportionally. In a market with total credit `T`, bad debt `B`, and a front-running lender holding `C` credit:

- **Expected loss per remaining unit** (if all share): `B / T`
- **Actual loss per remaining unit** (after front-run): `B / (T - C)`

For large `C` relative to `T`, the remaining lenders can suffer a significantly amplified loss. This is a direct, quantifiable financial harm to honest lenders — a loss-of-funds impact.

---

### Likelihood Explanation

- **Precondition**: `withdrawable > 0`. This is a normal market state after any repayment or partial liquidation. It does not require any special setup.
- **Attacker capability**: Any lender. No privileged access required.
- **Mempool monitoring**: Standard practice for MEV bots and sophisticated DeFi participants on Ethereum mainnet.
- **Profitability**: The gain is `C * B / T` (avoided loss). The cost is gas + settlement fee on re-entry (max ~50 bps annualized). For any meaningful bad debt event, the gain far exceeds the cost.

Likelihood is **medium-high**: the precondition is common, the attacker is unprivileged, and the tooling (mempool monitoring, flashbots) is widely available.

---

### Recommendation

1. **Withdrawal fee / exit fee**: Impose a small fee on `withdraw()` proportional to the credit withdrawn. This makes the front-run unprofitable for small bad debt events and reduces the incentive for large ones.

2. **Snapshot-based loss application**: Apply the `lossFactor` at the time of `withdraw()` using the *pending* (post-liquidation) loss factor. This is architecturally difficult in the current design since the loss is applied lazily.

3. **Withdrawal delay / lock**: Introduce a withdrawal queue or cooldown period so that lenders cannot exit atomically in the same block as a bad debt event. This is the most robust mitigation but adds UX friction.

4. **Private RPC for liquidators**: As a partial off-chain mitigation (analogous to the Napier sponsor's response), liquidators can submit bad-debt liquidations via private RPC (e.g., Flashbots) to prevent front-running. However, this is not enforced at the protocol level and should not be the sole mitigation.

### Proof of Concept

```
Setup:
- Market with loanToken = USDC, maturity = T+90 days
- Lender A: 1,000,000 credit units
- Lender B (attacker): 500,000 credit units
- Borrower: 200,000 debt units, undercollateralized (bad debt = 200,000)
- withdrawable = 600,000 (from prior repayments)
- totalUnits = 1,500,000

Step 1: Liquidator submits liquidate(borrower, ...) to mempool.
        Expected bad debt = 200,000 units.

Step 2: Attacker (Lender B) front-runs with withdraw(500,000, B, B).
        _updatePosition reads lossFactor = 0 (no bad debt yet).
        B receives 500,000 USDC. withdrawable = 100,000. totalUnits = 1,000,000.

Step 3: liquidate() executes.
        badDebt = 200,000.
        lossFactor increases: remaining lenders (only Lender A, 1,000,000 units) absorb all 200,000 bad debt.
        Lender A's effective credit = 1,000,000 * (1 - 200,000/1,000,000) = 800,000.

Step 4: Attacker re-enters by taking a sell offer (lending 500,000 USDC again).
        Pays settlement fee ~= 500,000 * 0.0025 = 1,250 USDC (at 90-day rate, max).

Result:
- Attacker avoided loss: 500,000 * 200,000 / 1,500,000 = ~66,667 USDC
- Attacker cost: ~1,250 USDC settlement fee + gas
- Net gain: ~65,417 USDC
- Lender A's loss: 200,000 USDC instead of 133,333 USDC (50% more than fair share)
``` [3](#0-2) [1](#0-0) [5](#0-4)

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
