Audit Report

## Title
Zero-unit `take` unconditionally dispatches `onBuy` to attacker-supplied `takerCallback` - (File: `src/Midnight.sol`)

## Summary
The `take` function dispatches the `onBuy` callback solely based on `buyerCallback != address(0)`, with no guard requiring `units > 0` or `buyerAssets > 0`. For a sell offer (`offer.buy = false`), `buyerCallback` is set directly from the caller-supplied `takerCallback`. An unprivileged attacker can pass any `IBuyCallback`-implementing address as `takerCallback` and call `take` with `units=0`, causing `onBuy` to fire on that address with zero-value parameters and no real token transfer. Because `consumed` is incremented by zero, the attack is repeatable indefinitely against the same offer.

## Finding Description
**Root cause:** No `units > 0` (or `buyerAssets > 0`) guard exists anywhere in `take` before the callback dispatch.

**Exact code path:**

- `src/Midnight.sol` lines 363–364: with `units=0`, `buyerAssets = 0.mulDivDown(buyerPrice, WAD) = 0` and `sellerAssets = 0`. [1](#0-0) 

- Lines 366–373: `consumed` is incremented by `buyerAssets` (0) or `units` (0); the `require(newConsumed <= offer.maxAssets/maxUnits)` passes trivially. The offer fill limit is never advanced. [2](#0-1) 

- Line 420: for a sell offer (`offer.buy = false`), `buyerCallback = takerCallback` — fully attacker-controlled. [3](#0-2) 

- Lines 445–453: the only guard before dispatching `onBuy` is `buyerCallback != address(0)`. No check on `units` or `buyerAssets`. [4](#0-3) 

- Lines 455–456: `safeTransferFrom(..., buyerAssets - sellerAssets)` and `safeTransferFrom(..., sellerAssets)` are both called with amount 0 — no-ops. [5](#0-4) 

**Exploit flow:**
1. Attacker identifies any valid, live sell offer (`offer.buy = false`) with a passing ratifier.
2. Attacker calls `take(sellOffer, ratifierData, 0, taker, receiver, victimContract, data)`.
3. `buyerAssets = 0`, `sellerAssets = 0`; consumed unchanged.
4. `buyerCallback = victimContract` (attacker-supplied).
5. `IBuyCallback(victimContract).onBuy(id, market, 0, 0, 0, buyer, data)` is called.
6. Victim contract executes its full `onBuy` body (state mutations, approvals, nested calls, event emissions) with zero-value parameters.
7. Both `safeTransferFrom` calls are no-ops; no tokens move.
8. If victim returns `CALLBACK_SUCCESS`, `take` succeeds and all callback-induced state changes persist.
9. Step 2 can be repeated indefinitely — `consumed` never advances.

**Why existing checks fail:**
- `require(taker == msg.sender || isAuthorized[taker][msg.sender])` validates taker identity only, not the callback target. [6](#0-5) 
- The `atMostOneNonZero(offer.maxAssets, offer.maxUnits)` check only prevents both being non-zero simultaneously; it does not require either to be non-zero, and does not block `units=0`. [7](#0-6) 
- `WrongBuyCallbackReturnValue` only reverts if the victim returns the wrong magic value; a well-formed callback returning `CALLBACK_SUCCESS` passes through. [8](#0-7) 

## Impact Explanation
Any contract implementing `IBuyCallback` and returning `CALLBACK_SUCCESS` — including legitimate protocol integrations such as lending callbacks — can have its `onBuy` invoked by an arbitrary unprivileged caller with zero-value parameters and no real trade. The callback executes its full body: state mutations, token approvals, reentrancy vectors, and event emissions all fire without any economic transfer. Because `consumed` is never incremented, the attack is repeatable against the same offer indefinitely. This constitutes an unauthorized state change on third-party contracts and a broken protocol invariant: a zero-unit take must have no side effects on external parties.

## Likelihood Explanation
Preconditions are minimal: a valid live sell offer (any public offer works), knowledge of a victim contract address implementing `IBuyCallback` that returns `CALLBACK_SUCCESS`, and the ability to call `take` as an unprivileged taker. No special role, no privileged access, no oracle manipulation is required. Any deployed `IBuyCallback` integration that does not internally assert `units > 0` or `buyerAssets > 0` is a viable victim. The attack is fully repeatable.

## Recommendation
Add an explicit guard at the top of `take` requiring `units > 0`:

```solidity
require(units > 0, ZeroUnits());
```

Alternatively, gate the `onBuy` dispatch on both `buyerCallback != address(0)` **and** `buyerAssets > 0` (or `units > 0`):

```solidity
if (buyerCallback != address(0) && units > 0) { ... }
```

The former is preferable as it eliminates the zero-unit path entirely and prevents analogous issues in the seller callback path.
