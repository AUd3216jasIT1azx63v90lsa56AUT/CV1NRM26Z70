### Title
Authorized Operator Charged as Payer Instead of Taker-Buyer in `take`, Breaking Composability — (File: `src/Midnight.sol`)

### Summary

In `Midnight.take`, when a sell offer is taken (`offer.buy = false`, meaning the taker is the buyer) and no `takerCallback` is provided, the `payer` for loan token transfers is set to `msg.sender` rather than `taker` (the actual buyer). Because the authorization model explicitly allows `msg.sender != taker`, any authorized operator that does not pre-fund itself with loan tokens will have its `safeTransferFrom` call revert, making it impossible to relay buy-side takes on behalf of users.

### Finding Description

**Root cause — line 422, `src/Midnight.sol`:**

```solidity
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
```

Role assignments at line 375:
```solidity
(address buyer, address seller) = offer.buy ? (offer.maker, taker) : (taker, offer.maker);
```

When `offer.buy = false` (taker is buyer, maker is seller):
- `buyer = taker`
- `payer = msg.sender` ← **wrong when `msg.sender != taker`**

When `offer.buy = true` (maker is buyer, taker is seller):
- `buyer = offer.maker`
- `payer = buyer = offer.maker` ← correct, maker pays

The asymmetry is the bug. For buy offers the maker (buyer) pays directly. For sell offers the taker is the buyer, but `msg.sender` (the operator) is charged instead.

The authorization check at line 346 explicitly permits `msg.sender != taker`:
```solidity
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());
```

The actual token pull happens at lines 455–456:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

When `payer = msg.sender` and `msg.sender` is an operator that does not hold loan tokens, both calls revert.

**Why `MidnightBundles` is not a counter-example:**
`MidnightBundles.buyWithUnitsTargetAndWithdrawCollateral` explicitly pulls tokens from `msg.sender` into itself before calling `take` (lines 66–67), and then calls `take` where `MidnightBundles` itself is `msg.sender`. It is designed to be the payer. Any other operator that does not pre-fund itself will fail.

### Impact Explanation

Any authorized smart-contract operator (aggregator, router, meta-transaction relayer, etc.) that calls `take` with a sell offer (`offer.buy = false`) and no `takerCallback` will have the transaction revert unless it holds and has pre-approved loan tokens to Midnight. The taker's position is updated (state changes are applied before the transfer), but the transfer fails, reverting the entire transaction. This breaks composability: operators cannot relay buy-side takes on behalf of users without becoming custodians of loan tokens, which is an unexpected and undocumented requirement.

### Likelihood Explanation

The authorization model (`isAuthorized`) is a first-class feature of the protocol, explicitly designed for operators and smart-contract integrations. Any integrator that builds an operator contract to call `take` on behalf of users — without studying the exact `payer` derivation — will hit this revert. The condition is reachable with no privileged access: any user can authorize any contract, and any contract can call `take`.

### Recommendation

Change the `payer` derivation so that when `offer.buy = false` and no `buyerCallback` is set, the payer is `taker` (the actual buyer) rather than `msg.sender`:

```solidity
// Before
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);

// After
address payer = buyerCallback != address(0) ? buyerCallback : buyer;
```

`buyer` is already correctly set to `offer.maker` when `offer.buy = true` and to `taker` when `offer.buy = false`, so this single change makes the payer consistent in both directions. `MidnightBundles` would need to be updated accordingly (it would need to ensure the taker has approved it, or use a `takerCallback`).

### Proof of Concept

1. Alice holds 1000 USDC and has approved `OperatorContract` on Midnight via `setIsAuthorized(OperatorContract, true, alice)`.
2. Alice has also approved Midnight to spend her USDC (`USDC.approve(midnight, 1000e6)`).
3. Bob has a resting sell offer (`offer.buy = false`) for 100 units at price 1.0.
4. `OperatorContract` calls:
   ```solidity
   midnight.take(bobSellOffer, ratifierData, 100, alice, address(0), address(0), "");
   ```
5. Inside `take`:
   - `buyer = taker = alice`
   - `buyerCallback = takerCallback = address(0)`
   - `payer = msg.sender = OperatorContract` ← charged instead of Alice
6. `safeTransferFrom(USDC, OperatorContract, midnight, buyerAssets - sellerAssets)` reverts because `OperatorContract` holds no USDC and has no allowance.
7. The entire transaction reverts. Alice's buy cannot be executed through `OperatorContract` even though she authorized it and holds sufficient funds. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
