### Title
`withdrawable` Accounting Corrupted When `receiver == address(this)` in `withdraw` — (File: src/Midnight.sol)

---

### Summary

`Midnight.withdraw` accepts an arbitrary `receiver` address and unconditionally decrements `_marketState.withdrawable`, `_marketState.totalUnits`, and `_position.credit` before transferring loan tokens to `receiver`. When `receiver == address(this)`, the tokens are immediately returned to the contract, leaving the actual token balance unchanged while the accounting permanently understates `withdrawable`. Any subsequent `withdraw` or `claimContinuousFee` call that relies on `withdrawable` being accurate will underflow and revert, DoS-ing all lenders in that market.

---

### Finding Description

**Root cause — `withdraw` (L481–500):**

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    ...
    _position.credit      -= UtilsLib.toUint128(units);   // L493
    _marketState.withdrawable -= UtilsLib.toUint128(units); // L494
    _marketState.totalUnits   -= UtilsLib.toUint128(units); // L495

    SafeTransferLib.safeTransfer(market.loanToken, receiver, units); // L499
}
``` [1](#0-0) 

There is no guard preventing `receiver == address(this)`. When that address is supplied:

1. `safeTransfer` sends `units` of the loan token to `address(this)` — the Midnight contract itself — so the contract's actual ERC-20 balance is **unchanged**.
2. `_marketState.withdrawable` is decremented by `units` — permanently understating how many tokens are available for withdrawal.
3. `_position.credit` and `_marketState.totalUnits` are also decremented, so the attacker's position is burned with no recovery path.

**Why `withdrawable` matters:**

`withdrawable` is the sole accounting field that gates whether a `withdraw` or `claimContinuousFee` call can succeed without underflowing. It is incremented only by `repay` (L509) and `liquidate` (L675). After the attack, every subsequent call that decrements `withdrawable` by any amount ≥ the new (understated) value will revert with an arithmetic underflow, even though the tokens are physically present in the contract. [2](#0-1) [3](#0-2) 

**`claimContinuousFee` is also affected (L318–325):**

```solidity
_marketState.withdrawable -= UtilsLib.toUint128(amount); // L320
``` [4](#0-3) 

This path reverts too once `withdrawable` is understated below the claimed amount.

---

### Impact Explanation

- **All lenders in the targeted market are DoS-ed**: their `withdraw` calls revert with underflow on `_marketState.withdrawable` even though the loan tokens are physically in the contract.
- **Fee claimer is DoS-ed**: `claimContinuousFee` for the market also reverts.
- **Tokens are permanently stranded**: the tokens returned to `address(this)` have no accounting entry; there is no admin function to recover them or to correct `withdrawable` directly.
- **Partial recovery only**: the DoS can be partially mitigated if borrowers repay debt (which increments `withdrawable`), but the attacker can repeat the attack to re-drain the counter.

Severity: **Medium** — permanent accounting corruption causing market-wide withdrawal DoS, at the cost of the attacker's own credit.

---

### Likelihood Explanation

- **No privileged access required**: any address holding credit in a market can call `withdraw` with `receiver == address(this)`.
- **Attacker cost**: the attacker sacrifices their own credit (a real economic cost), making this a griefing attack rather than a profit-seeking one.
- **Realistic scenario**: a malicious actor who wants to freeze a competitor's market, or a borrower who wants to prevent lenders from withdrawing before maturity, has a clear incentive.
- **Repeatability**: the attack can be executed multiple times (by different accounts or the same account after re-acquiring credit) to keep `withdrawable` at zero.

---

### Recommendation

Add a recipient check at the top of `withdraw` (and analogously in `withdrawCollateral` and `liquidate` for their respective receivers):

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    require(receiver != address(this), InvalidReceiver()); // <-- add this
    ...
}
```

This mirrors the fix recommended in the external report (`require(to != address(this))`). The same guard should be applied to `withdrawCollateral` and the `receiver` parameter of `liquidate` to prevent collateral tokens from being similarly stranded. [5](#0-4) 

---

### Proof of Concept

**Setup**: Market M exists with loan token T. Alice has 1000 units of credit. `withdrawable` = 1000.

**Attack steps**:

1. Alice calls `withdraw(market, 1000, alice, address(this))`.
2. Inside `withdraw`:
   - `alice.credit` → 0
   - `_marketState.withdrawable` → 0
   - `_marketState.totalUnits` → decremented by 1000
   - `safeTransfer(T, address(this), 1000)` — tokens return to Midnight; net balance change = 0.
3. Bob (another lender with 500 units of credit and `withdrawable` now = 0) calls `withdraw(market, 500, bob, bob)`.
4. `_marketState.withdrawable -= 500` **underflows → revert**. Bob cannot withdraw despite the tokens being physically present.
5. The fee claimer calls `claimContinuousFee(market, amount, receiver)` → same underflow → revert.

**Expected outcome**: Bob's withdrawal and all fee claims for market M are permanently blocked until borrowers repay enough debt to restore `withdrawable`, at which point Alice (or another attacker) can repeat step 1. [6](#0-5)
