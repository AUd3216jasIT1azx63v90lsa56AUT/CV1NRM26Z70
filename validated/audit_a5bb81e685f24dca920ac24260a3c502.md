### Title
Authorized Account Can Front-Run Revocation by Granting Authorization to a Controlled Address — (`src/Midnight.sol`)

### Summary

`setIsAuthorized` permits any currently-authorized account to grant authorization to additional addresses on behalf of the user. A malicious authorized account can observe a pending revocation transaction in the mempool and front-run it by authorizing a second controlled address, permanently retaining full access to the victim's account even after the revocation completes.

### Finding Description

**Vulnerability type:** Authorization bypass / privilege persistence (front-running)

The root cause is in `setIsAuthorized`: [1](#0-0) 

The check `isAuthorized[onBehalf][msg.sender]` allows any authorized account to call `setIsAuthorized` on behalf of the authorizer — including to add new authorized addresses. This is explicitly confirmed by the protocol's own documentation: [2](#0-1) 

And in `live_context.json`: [3](#0-2) 

**Attack path:**

1. Alice authorizes Bob: `setIsAuthorized(Bob, true, Alice)` → `isAuthorized[Alice][Bob] = true`
2. Alice decides to revoke Bob and broadcasts: `setIsAuthorized(Bob, false, Alice)`
3. Bob observes Alice's pending transaction in the mempool and front-runs it by calling: `setIsAuthorized(Charlie, true, Alice)` where Charlie is a Bob-controlled address
4. Alice's revocation executes: `isAuthorized[Alice][Bob] = false`
5. Result: `isAuthorized[Alice][Charlie] = true` — Bob retains full control through Charlie

Bob (via Charlie) can now:
- Call `withdraw` to drain Alice's credit
- Call `withdrawCollateral` to seize Alice's collateral
- Call `setIsAuthorized` to add further controlled addresses on Alice's behalf
- Call `setConsumed` to cancel Alice's offers [4](#0-3) [5](#0-4) 

### Impact Explanation

**Impact: High.** A successfully front-run revocation leaves the victim's entire Midnight account — credit, collateral, and offer management — permanently under attacker control. The attacker can drain all withdrawable credit and collateral tokens. The victim has no recourse because each new revocation attempt can itself be front-run by the same mechanism.

### Likelihood Explanation

**Likelihood: Low.** The attacker must be monitoring the mempool for revocation transactions targeting their address, and must be able to submit a higher-gas transaction to front-run it. On Ethereum mainnet this is realistic (MEV infrastructure, private relay networks). The scenario requires the attacker to have previously been legitimately authorized by the victim, which is a realistic precondition (e.g., a DeFi operator, a compromised smart-contract operator, or a malicious integrator).

### Recommendation

Restrict `setIsAuthorized` so that only the account owner (`onBehalf == msg.sender`) can grant new authorizations. Authorized accounts should only be permitted to remove authorizations (i.e., set `newIsAuthorized = false`), not add new ones. Alternatively, introduce a time-lock or nonce-based revocation mechanism that cannot be front-run.

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    if (newIsAuthorized) {
        // Only the account owner can grant new authorizations
        require(onBehalf == msg.sender, Unauthorized());
    } else {
        // Either the owner or an authorized account can revoke
        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    }
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
}
```

### Proof of Concept

```solidity
// 1. Alice authorizes Bob (legitimate use)
vm.prank(alice);
midnight.setIsAuthorized(bob, true, alice);

// 2. Alice decides to revoke Bob — this tx is in the mempool

// 3. Bob front-runs Alice's revocation by authorizing Charlie (Bob-controlled)
vm.prank(bob);
midnight.setIsAuthorized(charlie, true, alice); // succeeds: isAuthorized[alice][bob] == true

// 4. Alice's revocation executes
vm.prank(alice);
midnight.setIsAuthorized(bob, false, alice);

// 5. Bob is revoked, but Charlie still has full access
assertEq(midnight.isAuthorized(alice, bob), false);
assertEq(midnight.isAuthorized(alice, charlie), true); // Bob retains control via Charlie

// 6. Charlie drains Alice's credit
vm.prank(charlie);
midnight.withdraw(market, aliceCredit, alice, charlie); // succeeds
