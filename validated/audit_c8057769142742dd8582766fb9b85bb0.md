Audit Report

## Title
Pending `EcrecoverAuthorizer` Signature Survives Direct Midnight Revocation — (`src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer` maintains a per-authorizer nonce in its own storage that is only incremented by successful calls through `EcrecoverAuthorizer.setIsAuthorized`. A victim who revokes an attacker's authorization by calling `Midnight.setIsAuthorized` directly does not touch that nonce, leaving any pending (signed-but-not-yet-submitted) authorization signature valid. The attacker can submit the withheld signature to restore their authorization, fully defeating the revocation.

## Finding Description

**Root cause — nonce isolation:**

`EcrecoverAuthorizer.nonce` is a standalone mapping in `EcrecoverAuthorizer` storage: [1](#0-0) 

It is only incremented inside `EcrecoverAuthorizer.setIsAuthorized`: [2](#0-1) 

`Midnight.setIsAuthorized` is a completely separate entry point that writes `isAuthorized[onBehalf][authorized]` but has zero knowledge of, and makes no change to, `EcrecoverAuthorizer.nonce`: [3](#0-2) 

There is no `invalidateNonce()` or equivalent function anywhere in `EcrecoverAuthorizer` or its interface: [4](#0-3) 

**Exploit flow:**

1. Victim calls `Midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim)` — standard setup. `isAuthorized[victim][ecrecoverAuthorizer] = true`. `EcrecoverAuthorizer.nonce[victim] = 0`.
2. Victim signs `Authorization{authorizer: victim, authorized: attacker, isAuthorized: true, nonce: 0, deadline: block.timestamp + 365 days}` and shares it with the attacker (off-chain delegation, intercepted mempool tx, or a previously issued permit).
3. Attacker deliberately withholds submission.
4. Victim decides to revoke and calls `Midnight.setIsAuthorized(attacker, false, victim)`. `isAuthorized[victim][attacker]` is now `false`. `EcrecoverAuthorizer.nonce[victim]` is still `0`.
5. Attacker submits the withheld signature to `EcrecoverAuthorizer.setIsAuthorized`:
   - Deadline check passes (far future).
   - Nonce check passes: `0 == nonce[victim]++` → `0 == 0` ✓.
   - Signature valid (victim signed it) → `signer == authorization.authorizer` ✓.
   - `EcrecoverAuthorizer` calls `Midnight.setIsAuthorized(attacker, true, victim)` with `msg.sender = ecrecoverAuthorizer`.
   - `Midnight` authorization check: `isAuthorized[victim][ecrecoverAuthorizer] == true` ✓ (victim never revoked the authorizer itself). [5](#0-4) 
6. `isAuthorized[victim][attacker]` is `true` again. Revocation is undone.

**Why existing checks fail:**

The nonce prevents *true replay* of an already-consumed signature, but it does not help the victim invalidate a *pending* signature. The victim's only escape is to also revoke `EcrecoverAuthorizer`'s own authorization via `Midnight.setIsAuthorized(ecrecoverAuthorizer, false, victim)`, but nothing in the protocol enforces or documents this as a required step. The AUTHORIZATIONS comment in `Midnight.sol` warns generically that "authorized accounts can authorize other accounts on behalf of the user," but does not address nonce invalidation: [6](#0-5) 

## Impact Explanation
An attacker holding any pending (signed, deadline-not-expired) `EcrecoverAuthorizer` authorization for `isAuthorized: true` can restore their own authorization at any time after the victim revokes it via the direct Midnight path. This gives the attacker persistent, victim-unrevocable access to all operations gated by `isAuthorized`: `take` on behalf, `repay` on behalf, `withdraw` on behalf, `setConsumed`, and `setIsAuthorized` chaining (the attacker can authorize further addresses on the victim's behalf). The impact is unauthorized state mutation and potential asset loss across all markets where the victim participates.

## Likelihood Explanation
Preconditions are realistic: off-chain delegation flows routinely produce signed-but-not-yet-submitted authorizations. The attacker only needs to withhold submission until after the victim's revocation transaction confirms, then submit. The attack is repeatable for every pending signature the attacker holds, is not blocked by any on-chain mechanism, and requires no privileged access — only possession of a valid signed `Authorization` struct.

## Recommendation
Add an `invalidateNonce()` function to `EcrecoverAuthorizer` that allows the authorizer to increment their own nonce, rendering all previously signed but unsubmitted authorizations invalid:

```solidity
function invalidateNonce() external {
    nonce[msg.sender]++;
    emit NonceInvalidated(msg.sender, nonce[msg.sender]);
}
```

This mirrors the pattern used by EIP-2612 permit implementations and Uniswap's Permit2. Additionally, document clearly that revoking via `Midnight.setIsAuthorized` directly does **not** invalidate pending `EcrecoverAuthorizer` signatures, and that users must call `invalidateNonce()` (or revoke `EcrecoverAuthorizer`'s own Midnight authorization) to achieve a complete revocation.

## Proof of Concept
Minimal Foundry test extending `SetIsAuthorizedWithSigTest.sol`:

```solidity
function testPendingSignatureSurvivesDirectRevocation() public {
    // Step 1: victim authorizes EcrecoverAuthorizer in Midnight
    vm.prank(borrower); // borrower = victim
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

    // Step 2: victim signs authorization for attacker (nonce=0, far deadline)
    Authorization memory auth = Authorization({
        authorizer: borrower,
        authorized: attacker,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 365 days
    });
    Signature memory sig = signAuthorization(auth, borrower);
    // attacker withholds sig

    // Step 4: victim revokes attacker directly via Midnight
    vm.prank(borrower);
    midnight.setIsAuthorized(attacker, false, borrower);
    assertEq(midnight.isAuthorized(borrower, attacker), false);
    // nonce is still 0 — pending sig is still valid
    assertEq(ecrecoverAuthorizer.nonce(borrower), 0);

    // Step 5: attacker submits withheld signature
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // Step 6: revocation is undone
    assertEq(midnight.isAuthorized(borrower, attacker), true);
}
```

Expected result: the final assertion passes, confirming the revocation was defeated.
