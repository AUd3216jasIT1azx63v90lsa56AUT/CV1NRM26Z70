Audit Report

## Title
Fee-on-Transfer Loan Token in `repay` Inflates `withdrawable` Beyond Actual Balance — (File: src/Midnight.sol)

## Summary
The `repay` function credits `marketState[id].withdrawable` by the caller-supplied `units` before executing the token transfer. When the loan token charges a transfer fee, the contract receives fewer tokens than recorded, permanently inflating `withdrawable` relative to the actual balance. Subsequent lender withdrawals drain tokens belonging to other lenders or other markets sharing the same token.

## Finding Description
In `repay` (lines 508–520 of `src/Midnight.sol`), the accounting update unconditionally uses the argument `units`:

```solidity
position[id][onBehalf].debt      -= UtilsLib.toUint128(units);  // L508
marketState[id].withdrawable     += UtilsLib.toUint128(units);  // L509
...
SafeTransferLib.safeTransferFrom(
    market.loanToken, payer, address(this), units               // L520
);
```

`SafeTransferLib.safeTransferFrom` (lines 24–34 of `src/libraries/SafeTransferLib.sol`) validates only `token.code.length > 0`, call success, and the boolean return value — no balance-before/balance-after measurement is performed. For a fee-on-transfer token, `transferFrom` returns `true` and emits a transfer event for `units`, but only `units - fee` arrives at the contract.

Market creation is permissionless and performs no validation on `loanToken` properties, so any actor can create a market with a fee-on-transfer ERC-20 as the loan token.

**Exploit flow:**
1. Attacker creates a market with a fee-on-transfer ERC-20 as `loanToken`.
2. Borrower takes a loan of `D` units via normal protocol flow.
3. Borrower calls `repay(market, D, onBehalf, address(0), "")`.
4. Protocol records: `debt[onBehalf] -= D` and `withdrawable += D`.
5. `safeTransferFrom` transfers `D` tokens; only `D * (1 - feeRate)` arrives at `address(this)`.
6. `withdrawable` now exceeds actual balance by `D * feeRate`.
7. Lenders call `withdraw`; the protocol transfers `D` tokens, draining funds belonging to other lenders or other markets sharing the same token.

Each `repay` call widens the deficit by `units * feeRate`. No reentrancy guard addresses this because the state mutation precedes the transfer.

## Impact Explanation
Every `repay` with a fee-on-transfer loan token permanently inflates `marketState[id].withdrawable` relative to the contract's actual token balance. Lenders who withdraw after such repayments receive tokens deposited by other lenders or belonging to other markets, causing direct, irreversible loss of lender principal. This is a direct theft-of-assets impact.

## Likelihood Explanation
Market creation is permissionless with no token validation, so any unprivileged actor can create a market with a fee-on-transfer loan token. The exploit requires only a standard borrow-then-repay flow. It is repeatable: each repayment widens the deficit. Tokens with owner-configurable fees (e.g., USDT's historical fee switch) make this reachable without any malicious action at market-creation time — the fee only needs to be active when `repay` is called. Fee-on-transfer tokens are not excluded by `SECURITY.md`.

## Recommendation
Measure the actual balance delta in `repay` and use that for accounting instead of the caller-supplied `units`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;

position[id][onBehalf].debt      -= UtilsLib.toUint128(received);
marketState[id].withdrawable     += UtilsLib.toUint128(received);
```

Alternatively, explicitly document and enforce (via `touchMarket` validation or a registry) that only standard ERC-20 tokens without transfer fees are permitted as loan tokens.

## Proof of Concept
1. Deploy a mock ERC-20 that deducts a 1% fee on every `transferFrom`, returning `true`.
2. Create a Midnight market with this token as `loanToken`.
3. Supply liquidity and borrow `1000` units.
4. Call `repay(market, 1000, borrower, address(0), "")`.
5. Assert: `marketState[id].withdrawable == 1000` but `token.balanceOf(address(midnight)) == 990`.
6. Call `withdraw` for `1000` units — succeeds by draining 10 tokens from another depositor's balance, confirming the shortfall.

        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));

                revert(add(returndata, 0x20), mload(returndata))
            }
