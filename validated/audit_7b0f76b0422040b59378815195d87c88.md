Audit Report

## Title
Unbounded `tokens[]` loop in `flashLoan` enables permissionless block-gas DoS - (File: src/Midnight.sol)

## Summary
`flashLoan` accepts arbitrarily large `tokens[]` and `assets[]` calldata arrays with no length cap. Two sequential loops each issue a full external call per element via `SafeTransferLib`, which executes even for zero-amount transfers. An unprivileged attacker can supply a large enough array to consume the entire block gas limit, blocking all other protocol transactions for as long as they are willing to pay their own gas.

## Finding Description
**Code path:**

`flashLoan` (lines 737–752, `src/Midnight.sol`) accepts `tokens[]` and `assets[]` as calldata. The only input validation is the length-equality check at line 740:
```solidity
require(tokens.length == assets.length, InconsistentInput());
```

Loop 1 (lines 742–744) calls `SafeTransferLib.safeTransfer(tokens[i], callback, assets[i])` for every element.

Loop 2 (lines 749–751) calls `SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i])` for every element.

`SafeTransferLib.safeTransfer` (lines 12–22, `src/libraries/SafeTransferLib.sol`) unconditionally checks `token.code.length > 0` and then issues a full external `.call()` to the token contract — even when `value == 0`. The same applies to `safeTransferFrom` (lines 24–34).

**Root cause:** No `tokens.length` cap exists in `flashLoan`. By contrast, `touchMarket` enforces `require(market.collateralParams.length <= MAX_COLLATERALS)` (line 760, `src/Midnight.sol`), where `MAX_COLLATERALS = 128` is defined in `src/libraries/ConstantsLib.sol` (line 20) but is never applied to `flashLoan`.

**Exploit flow:**
1. Attacker pre-deploys N minimal ERC20 contracts that accept zero-amount transfers and return `true` on both `transfer` and `transferFrom`.
2. Attacker deploys a callback contract that returns `CALLBACK_SUCCESS` immediately.
3. Attacker calls `flashLoan(tokens, zeros, callback, "")` with `tokens.length = N` and all `assets[i] = 0`.
4. Loop 1 makes N external calls (each: `code.length` check + external `.call()`).
5. Callback returns `CALLBACK_SUCCESS` cheaply.
6. Loop 2 makes another N external calls.
7. At ~5,000–10,000 gas per call × 2N calls, N ≈ 1,500–3,000 fills a 30M gas block.

**Why existing checks fail:** The only guard is `tokens.length == assets.length`, which the attacker trivially satisfies. `SafeTransferLib` requires `token.code.length > 0`, but the attacker supplies real deployed contracts. There is no reentrancy guard, no length cap, no fee, and no access control on `flashLoan`.

## Impact Explanation
An unprivileged caller can repeatedly submit transactions that each consume the full block gas limit, effectively blocking all other protocol interactions — borrows, repayments, liquidations, and withdrawals — for as long as the attacker is willing to pay their own gas. This constitutes a sustained, permissionless DoS against the entire protocol. The impact is concrete and not hypothetical: the attacker's own gas cost is the only limiting factor.

## Likelihood Explanation
- **Preconditions:** Caller needs only valid ERC20 contract addresses (attacker-deployed) and a callback that returns `CALLBACK_SUCCESS`. No protocol position, no tokens held, no approval required.
- **Feasibility:** Deploying a batch of minimal ERC20s is a one-time setup cost. The attack is then repeatable every block.
- **Repeatability:** High — the attacker pays their own gas but can fill every block indefinitely.

## Recommendation
Apply a maximum length cap to the `tokens[]` array in `flashLoan`, consistent with the existing `MAX_COLLATERALS` pattern used in `touchMarket`:

```solidity
function flashLoan(address[] calldata tokens, uint256[] calldata assets, address callback, bytes calldata data)
    external
{
    require(tokens.length == assets.length, InconsistentInput());
    require(tokens.length <= MAX_COLLATERALS, TooManyTokens()); // add this
    ...
}
```

Alternatively, introduce a dedicated `MAX_FLASH_LOAN_TOKENS` constant with an appropriate bound. The value should be chosen to keep worst-case gas consumption well within a single block's gas limit.

## Proof of Concept
1. Deploy N minimal ERC20 contracts (e.g., always returning `true` on `transfer`/`transferFrom`, no-op on zero amounts).
2. Deploy a callback contract returning `CALLBACK_SUCCESS` from `onFlashLoan`.
3. Build `tokens[]` of length N (all attacker ERC20 addresses) and `assets[]` of length N (all zeros).
4. Call `Midnight.flashLoan(tokens, assets, callback, "")`.
5. Measure gas consumed: it scales linearly with N. At N ≈ 2,000, gas consumption approaches 30M.
6. Confirm no other transaction can be included in the same block when the attacker fills it with such calls.

A Foundry fuzz test parameterizing `tokens.length` and asserting gas usage stays below a safe threshold would reproduce this automatically.
