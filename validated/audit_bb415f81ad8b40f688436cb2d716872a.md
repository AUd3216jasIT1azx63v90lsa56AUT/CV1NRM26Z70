### Title
DoS of Full Liquidations via Frontrunning Partial Liquidation — (`src/Midnight.sol`)

---

### Summary

Midnight's `liquidate()` function requires callers to specify exact `repaidUnits` or `seizedAssets` amounts. Because any unprivileged address can call `liquidate()` on any unhealthy or post-maturity borrower, an attacker can frontrun a full-liquidation transaction with a dust partial liquidation, causing the original liquidator's transaction to revert via arithmetic underflow. This is the same vulnerability class as the DSCEngine report, and it is directly reachable in Midnight.

---

### Finding Description

**Root cause:** The `liquidate()` function subtracts caller-supplied `repaidUnits` directly from `_position.debt` at line 676, and caller-supplied `seizedAssets` directly from `_position.collateral[collateralIndex]` at line 670, with no tolerance for the position having changed between when the liquidator observed the state and when their transaction executes.

```solidity
// src/Midnight.sol:670
uint128 newCollateral = _position.collateral[collateralIndex] - UtilsLib.toUint128(seizedAssets);

// src/Midnight.sol:676
_position.debt -= UtilsLib.toUint128(repaidUnits);
```

Both are plain Solidity 0.8.34 arithmetic. If `repaidUnits > _position.debt` or `seizedAssets > _position.collateral[collateralIndex]`, the transaction reverts with an arithmetic underflow panic.

**Attack path (post-maturity mode, most impactful):**

1. Alice has `debt = D` units, collateral `C`, market is past maturity. Her position is liquidatable via `postMaturityMode = true`.
2. Bob (liquidator) broadcasts `liquidate(..., repaidUnits = D, postMaturityMode = true, ...)` to fully liquidate Alice.
3. Attacker (any address holding ≥ 1 loan token unit) sees Bob's transaction in the mempool and frontruns with `liquidate(..., repaidUnits = 1, postMaturityMode = true, ...)`.
4. After the attacker's transaction: `_position.debt = D - 1`.
5. Bob's transaction executes: `_position.debt -= D` → `(D - 1) - D` → arithmetic underflow → **revert**.

The same attack works with the `seizedAssets` input path: attacker frontruns with `seizedAssets = 1`, reducing `_position.collateral[collateralIndex]` by 1, causing Bob's `seizedAssets = C` to underflow at line 670.

**Why existing checks do not prevent this:**

- `require(_position.debt > 0, NotBorrower())` at line 596 only checks that debt is non-zero, not that it is ≥ the caller's `repaidUnits`.
- `require(UtilsLib.atMostOneNonZero(repaidUnits, seizedAssets), InconsistentInput())` at line 595 only validates mutual exclusivity of inputs.
- The RCF check at lines 659–667 uses `_position.debt` (already reduced by the attacker's frontrun), so `maxRepaid` is smaller, and the check may also revert with `RecoveryCloseFactorConditionsViolated()` before even reaching line 676 — either way, Bob's transaction is blocked.
- There is no slippage/tolerance parameter for the liquidator to specify a minimum acceptable debt to repay. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

- **Full liquidations can be blocked indefinitely** at a cost of 1 loan token unit per block to the attacker.
- In **post-maturity mode**, the LIF ramps linearly from 1 to `maxLif` over `TIME_TO_MAX_LIF` (15 minutes). Blocking liquidation during this window means the LIF grows, increasing the collateral seized per unit repaid. If the LIF reaches a level where `collateral * price / LIF < debt`, the protocol realizes bad debt — losses socialized to all lenders in the market.
- Even before bad debt, delayed post-maturity liquidations harm lenders who cannot recover their loan tokens until the borrower is liquidated.
- In **normal (pre-maturity) mode**, the same attack applies when the RCF dust condition is met (full liquidation is allowed). Blocking it forces the liquidator to use partial liquidations, leaving dust debt and potentially keeping the position unhealthy longer. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

- **High.** `liquidate()` is fully permissionless (absent a `liquidatorGate`). Any address with 1 unit of the loan token can execute the attack.
- The borrower being liquidated is the most motivated attacker (protecting their collateral), and they can use a secondary address since the function has no `msg.sender != borrower` guard.
- The cost is negligible: 1 wei of loan token plus gas. The attacker does not lose the repaid unit — they receive collateral in return (at LIF discount), so the frontrun is economically neutral or even profitable.
- Frontrunning is straightforward on any chain with a public mempool. [5](#0-4) 

---

### Recommendation

Allow liquidators to pass `type(uint256).max` for `repaidUnits` (or `seizedAssets`) to mean "use the current on-chain value", resolved at execution time:

```solidity
if (repaidUnits == type(uint256).max) {
    repaidUnits = _position.debt; // use current debt after bad-debt subtraction
}
```

This should be applied **after** the bad-debt subtraction block (line 641) and **before** the RCF check and the debt/collateral arithmetic. This mirrors the fix recommended in the DSCEngine report and eliminates the TOCTOU window between observation and execution. [6](#0-5)

### Proof of Concept

```
Setup:
- Market: loanToken = USDC, collateral = WETH, past maturity
- Alice: debt = 1000e18 units, collateral = 2000e18 WETH
- Bob: holds 1000e18 loan tokens, calls:
    liquidate(market, 0, 0, 1000e18, alice, true, bob, address(0), "")
    // repaidUnits = 1000e18 (full debt)

Attack:
- Attacker (Alice's secondary address) frontruns Bob with:
    liquidate(market, 0, 0, 1, alice, true, attacker, address(0), "")
    // repaidUnits = 1 (dust)
- After attacker tx: position[id][alice].debt = 1000e18 - 1

Bob's tx executes:
- Line 676: _position.debt -= 1000e18
  = (1000e18 - 1) - 1000e18
  → arithmetic underflow → REVERT

Result: Bob's full liquidation is blocked. Attacker spent 1 wei of loan token
and received 1 wei worth of collateral (at LIF). Attack can be repeated every
block for negligible cost.
``` [7](#0-6)
