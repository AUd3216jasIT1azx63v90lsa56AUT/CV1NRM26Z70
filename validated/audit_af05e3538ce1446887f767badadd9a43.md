Audit Report

## Title
Zero-price oracle enables free collateral seizure with full bad-debt socialization to lenders - (File: `src/Midnight.sol`)

## Summary
When an oracle returns `0` for a collateral's price, `mulDivUp` arithmetic causes `repaidUnits` to evaluate to `0` even when `seizedAssets > 0`. The `liquidate` function simultaneously writes off the borrower's entire debt as bad debt (socializing the loss to lenders via `lossFactor`) and transfers the borrower's collateral to the caller, while pulling zero loan tokens from the caller. Every existing guard is bypassed by the zero-price condition.

## Finding Description

**Root cause — `src/Midnight.sol` line 650:**
```solidity
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```
`mulDivUp` (`src/libraries/UtilsLib.sol` lines 34–35) computes `(x * y + (d - 1)) / d`. With `liquidatedCollatPrice = 0`:
- `mulDivUp(seizedAssets, 0, ORACLE_PRICE_SCALE)` = `(ORACLE_PRICE_SCALE − 1) / ORACLE_PRICE_SCALE` = `0`
- `mulDivUp(0, WAD, lif)` = `(lif − 1) / lif` = `0`

Result: `repaidUnits = 0`.

**`NotLiquidatable` check trivially passes (lines 620–624):** With price=0, `maxDebt` accumulates `mulDivDown(_collateral, 0, ORACLE_PRICE_SCALE) * lltv / WAD = 0` for every collateral. `originalDebt > 0` is already guaranteed by the `NotBorrower` check at line 596.

**Full bad debt realized (lines 626–641):** `badDebt` starts at `originalDebt`. For each collateral, `zeroFloorSub(_collateral.mulDivUp(0, ORACLE_PRICE_SCALE)...) = zeroFloorSub(0) = 0`, so `badDebt = originalDebt`. The entire debt is written off: `_position.debt -= badDebt`, `lossFactor` is updated to socialize the full loss to lenders, and `totalUnits` is decremented.

**Collateral seizure path executes (lines 643–677):** The branch `seizedAssets > 0` is entered. After bad debt write-off, `_position.debt = 0` and `maxDebt = 0`, so the RCF check at lines 659–667 evaluates `maxRepaid = 0` and `repaidUnits = 0 ≤ 0` — passes. Then:
- Line 670: `_position.collateral[collateralIndex] -= seizedAssets` — collateral removed from borrower
- Line 675: `_marketState.withdrawable += 0` — pool receives no compensation
- Line 676: `_position.debt -= 0` — already zeroed
- Line 696: `safeTransfer(collateralToken, receiver, seizedAssets)` — collateral sent to attacker
- Line 717: `safeTransferFrom(loanToken, payer, address(this), 0)` — zero loan tokens pulled

The protocol's own TOKEN SAFETY REQUIREMENTS (line 136) state tokens "should not revert because a transfer is no-op," so the zero-amount `safeTransferFrom` succeeds unconditionally.

**Formal proof gap:** `certora/specs/NoDivisionByZero.spec` line 124 explicitly requires `ghostPrice(...) > 0` as a precondition, meaning formal proofs do not cover price=0. `certora/specs/Reverts.spec` lines 246–253 only proves that `liquidate` with `repaidUnits > 0` reverts when oracle returns 0 (division by zero at line 652); it does not prove that `seizedAssets > 0` reverts. The `seizedAssets > 0` path with price=0 is entirely unguarded.

**Intended behavior confirmed:** `test/LiquidationTest.sol` line 876 calls `liquidate(market, 0, 0, 0, borrower, false, ...)` (both inputs zero) to realize bad debt only, after which the borrower withdraws their own collateral. The `seizedAssets > 0` variant with price=0 is not tested and not guarded.

## Impact Explanation
An attacker receives the borrower's real ERC-20 collateral tokens for free. Lenders suffer a complete, irreversible loss on the borrower's full debt socialized via `lossFactor` write-down. The protocol's `withdrawable` pool receives zero compensation. The affected market is rendered insolvent. This constitutes direct theft of collateral value and permanent accounting corruption — both concrete, in-scope critical impacts.

## Likelihood Explanation
Required preconditions: (1) an oracle transiently returns `0` — explicitly modeled as reachable in the Certora specs (`forceOracleReturnZero`, `singleZeroOracle`); (2) the borrower has `debt > 0` and non-zero collateral at the targeted index. No special privilege is required unless `liquidatorGate` is configured (and `liquidatorGate = address(0)` is the default unrestricted case). Oracles can return `0` due to uninitialized price feeds, stale/gap conditions, sequencer downtime, or error-handling paths that return 0 instead of reverting. The exploit is repeatable for any such market and requires no victim cooperation.

## Recommendation
Add an explicit guard at the entry of the `seizedAssets > 0` branch (or before line 650) requiring `liquidatedCollatPrice > 0`:

```solidity
require(liquidatedCollatPrice > 0, ZeroCollateralPrice());
```

Alternatively, add a global guard at the top of `liquidate` that reverts if any activated collateral oracle returns 0, consistent with the precondition already assumed in `certora/specs/NoDivisionByZero.spec` line 124. The `repaidUnits > 0` input path already reverts on price=0 (division by zero at line 652); the `seizedAssets > 0` path must be given equivalent protection.

## Proof of Concept
1. Deploy a market with a collateral oracle that can be set to return `0`.
2. Have a borrower supply collateral and take on debt (`_position.debt > 0`).
3. Set the oracle price to `0`.
4. As an unprivileged attacker, call:
   ```solidity
   midnight.liquidate(market, collateralIndex, seizedAssets, 0, borrower, false, attacker, address(0), "");
   ```
   where `seizedAssets` equals the borrower's full collateral balance at `collateralIndex`.
5. Observe: attacker receives `seizedAssets` collateral tokens; attacker pays 0 loan tokens; borrower's debt is fully written off as bad debt; lenders' `lossFactor` is updated to `type(uint128).max`; `withdrawable` is unchanged.

This is directly reproducible by extending `test/LiquidationTest.sol::testFullBadDebtWithdrawCollateral` to pass `seizedAssets > 0` instead of `0` as the third argument on line 876, and asserting that the attacker receives the collateral while paying nothing.
