### Title
Missing `invalidateNonce()` in `EcrecoverAuthorizer` Allows Replay of Revoked Signed Authorizations — (File: src/periphery/EcrecoverAuthorizer.sol)

---

### Summary

`EcrecoverAuthorizer` uses a sequential nonce to prevent replay of signed authorizations, but exposes no function to increment/invalidate the nonce without consuming a valid signature. This means a user who has shared a signed authorization cannot cancel it through this contract. An attacker holding the signed message can submit it at any time — including after the user has attempted to revoke the authorization directly on `Midnight` — re-granting themselves access and enabling theft of user funds.

---

### Finding Description

`EcrecoverAuthorizer.setIsAuthorized` enforces replay protection via a per-user sequential nonce:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
``` [1](#0-0) 

The nonce is only ever incremented by successfully processing a signed authorization. There is no `invalidateNonce()`, `cancelAuthorization()`, or equivalent function that would let a user advance their nonce without submitting a valid signature. [2](#0-1) 

The signed authorization grants the `authorized` address full access to all `onBehalf`-gated functions on `Midnight`:

```solidity
IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
``` [3](#0-2) 

Those authorized functions include `take`, `withdraw`, `withdrawCollateral`, `repay`, `setIsAuthorized`, and `setConsumed` — all of which can move user funds or escalate privileges further. [4](#0-3) 

**Exploit flow:**

1. User signs `Authorization{authorized: attacker, isAuthorized: true, nonce: 0, deadline: far_future}` and shares it with a trading bot or relayer.
2. User decides to stop using the bot and calls `Midnight.setIsAuthorized(attacker, false, user)` directly.
3. Attacker (bot operator) submits the previously signed authorization to `EcrecoverAuthorizer.setIsAuthorized()`.
4. `EcrecoverAuthorizer` validates the signature (still valid — nonce 0 was never consumed), increments the nonce, and calls `Midnight.setIsAuthorized(attacker, true, user)`.
5. Attacker is re-authorized. In the same transaction, attacker calls `Midnight.withdraw(market, credit, user, attacker)` or `Midnight.withdrawCollateral(...)` to drain the user's position.

The user's direct revocation on `Midnight` is completely undone by step 4, with no on-chain mechanism to prevent it.

---

### Impact Explanation

**Impact: High.** A successful exploit grants the attacker full authorization over the victim's `Midnight` positions. The attacker can withdraw all credit (loan tokens), withdraw all collateral, or call `setIsAuthorized` to create a persistent backdoor by authorizing additional addresses — leading to total loss of user funds.

---

### Likelihood Explanation

**Likelihood: Medium.** The precondition is that the user has previously shared a signed authorization (e.g., with a trading bot, relayer, or counterparty) and later wants to revoke it. This is a standard and expected use-case for gasless authorization systems. The attacker requires no privileged access — only possession of the signed message, which the user intentionally provided. The attack is atomic and requires no special timing beyond front-running the user's revocation attempt.

---

### Recommendation

Add a permissionless `invalidateNonce()` function that allows a user to advance their own nonce, rendering any outstanding signed authorizations with the old nonce invalid:

```solidity
function invalidateNonce() external {
    emit NonceInvalidated(msg.sender, nonce[msg.sender]);
    nonce[msg.sender]++;
}
```

This mirrors the pattern used by Permit2 (`invalidateNonces`) and is the standard mitigation for this class of issue in EIP-712 authorization systems.

---

### Proof of Concept

**Setup:**
- `user` has credit in a `Midnight` market.
- `user` previously signed: `Authorization{authorizer: user, authorized: attacker, isAuthorized: true, nonce: 0, deadline: type(uint256).max}`.

**Steps:**
1. `user` calls `Midnight.setIsAuthorized(attacker, false, user)` — attacker is revoked on Midnight.
2. `attacker` calls `EcrecoverAuthorizer.setIsAuthorized(authorization, signature)` with the original signed message.
3. `EcrecoverAuthorizer` checks: `nonce[user] == 0` ✓, signature valid ✓ → calls `Midnight.setIsAuthorized(attacker, true, user)`.
4. `attacker` (now re-authorized) calls `Midnight.withdraw(market, userCredit, user, attacker)`.
5. All of `user`'s loan tokens are transferred to `attacker`. [5](#0-4) [6](#0-5)

contract EcrecoverAuthorizer is IEcrecoverAuthorizer {
    address public immutable MIDNIGHT;
    mapping(address => uint256) public nonce;

    constructor(address _midnight) {
        MIDNIGHT = _midnight;
    }

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
