Audit Report

## Title
Bad Debt Over-Realization via `onLiquidate` Callback Supplying Collateral After Bitmap Snapshot - (File: `src/Midnight.sol`)

## Summary
The `liquidate` function snapshots `_position.collateralBitmap` into a local variable at line 606, computes `badDebt` from that snapshot, and irreversibly applies the bad debt to `_position.debt`, `_marketState.lossFactor`, and `_marketState.totalUnits` at lines 626–641 before invoking the `onLiquidate` callback at lines 698–715. A borrower-controlled smart contract can exploit this ordering by calling `supplyCollateral` inside the callback, depositing real collateral that was never subtracted from `badDebt`, causing the protocol to over-realize bad debt and over-slash lenders.

## Finding Description

**Root cause:** The local variable `_collateralBitmap` (line 606) is snapshotted from storage and never refreshed after the external callback. Any collateral index activated during the callback is invisible to the `badDebt` computation.

**Exact code path (verified against source):**

- `src/Midnight.sol` line 606: `uint128 _collateralBitmap = _position.collateralBitmap;` — snapshot taken. [1](#0-0) 

- Lines 607–618: `while (_collateralBitmap != 0)` loop computes `badDebt` exclusively from indices present in the snapshot. [2](#0-1) 

- Lines 626–641: `badDebt` is irreversibly applied — `_position.debt -= uint128(badDebt)`, `_marketState.lossFactor` is ratcheted up, `_marketState.totalUnits` is reduced — all before any external call. [3](#0-2) 

- Lines 698–715: `ILiquidateCallback(callback).onLiquidate(...)` is invoked after full bad debt realization. [4](#0-3) 

**No reentrancy guard exists anywhere in `Midnight.sol`** — confirmed by absence of `nonReentrant`, `ReentrancyGuard`, `_locked`, or equivalent. [5](#0-4) 

**`supplyCollateral` authorization check:**

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
``` [6](#0-5) 

When the borrower contract calls `supplyCollateral(market, newIndex, largeAmount, address(this))` from inside `onLiquidate`, `onBehalf == msg.sender` is trivially satisfied — no external authorization is needed.

**`supplyCollateral` writes to storage:**

```solidity
if (oldCollateral == 0 && assets > 0) {
    uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
    _position.collateralBitmap = newCollateralBitmap;
``` [7](#0-6) 

This storage write is invisible to the already-computed `badDebt`.

**Exploit flow:**

1. Borrower deploys a smart contract holding a position where all existing collateral is worth less than debt at `maxLif` prices (genuine bad debt condition).
2. Borrower contract calls `liquidate(market, 0, 0, 0, address(this), false, receiver, address(this), "")` — `seizedAssets = repaidUnits = 0`, `callback = address(this)`.
3. `liquidate` snapshots `_collateralBitmap` (line 606), loops over existing collateral, computes `badDebt = originalDebt − Σ(collateral_i × price_i / maxLif_i)` (positive).
4. `liquidate` applies `badDebt` at lines 626–641 — irreversibly modifies `_position.debt`, `_marketState.lossFactor`, `_marketState.totalUnits`.
5. `liquidate` calls `onLiquidate` on the borrower contract (line 700).
6. Inside `onLiquidate`, the borrower contract calls `supplyCollateral(market, newIndex, largeAmount, address(this))`. Auth passes because `onBehalf == msg.sender`. This sets `_position.collateral[newIndex]` and sets the corresponding bit in `_position.collateralBitmap` in storage.
7. `liquidate` returns. The new collateral is now in the position, but `badDebt` was already applied without accounting for it. With `repaidUnits = 0`, the `safeTransferFrom` at line 717 transfers 0 tokens — no revert.

**Why existing checks fail:**

- No reentrancy guard on `liquidate` or `supplyCollateral`.
- No post-callback re-computation or validation of `badDebt`.
- The local `_collateralBitmap` variable is never refreshed from storage after the callback.
- `supplyCollateral`'s authorization check is satisfied trivially when the borrower contract acts on behalf of itself.

## Impact Explanation

`_marketState.lossFactor` is set higher than warranted and `_marketState.totalUnits` is reduced by more than the true uncollateralized shortfall. Every subsequent lender `updatePosition` call slashes lender credit proportionally to the inflated loss factor — lenders bear losses they should not bear. The borrower's debt is reduced by a `badDebt` amount that exceeds the true shortfall, effectively forgiving debt that was actually backed by the newly supplied collateral. This constitutes direct, unauthorized theft of lender value with no recovery path, as `lossFactor` is monotonically increasing. [8](#0-7) 

## Likelihood Explanation

Preconditions are minimal and realistic: (1) a position with genuine bad debt — a normal liquidation scenario; (2) the borrower is a smart contract — standard in DeFi; (3) the borrower passes itself as the callback — requires no external authorization. The attack requires no privileged access, no oracle manipulation, no flash loan (the borrower can pre-fund the collateral), and is repeatable on any market with bad debt. The borrower must hold sufficient collateral tokens to supply during the callback, but this is a trivial economic precondition.

## Recommendation

Apply one or more of the following mitigations:

1. **Reentrancy guard:** Add a per-market or global reentrancy lock that prevents `supplyCollateral` (and other state-mutating functions) from being called during `liquidate`.
2. **Post-callback validation:** After the callback returns, re-read `_position.collateralBitmap` from storage and verify it equals the pre-callback snapshot. Revert if it differs.
3. **Move bad debt application after the callback:** Compute `badDebt` before the callback but apply it only after the callback returns, re-reading collateral state from storage at that point.
4. **Snapshot and compare:** Record `_position.collateralBitmap` before the callback and assert it is unchanged after.

Option 1 (reentrancy guard) is the most robust and prevents analogous issues in other callback paths (`repay`, `take`, `flashLoan`).

## Proof of Concept

**Minimal Foundry test outline:**

```solidity
contract MaliciousBorrower is ILiquidateCallback {
    IMidnight midnight;
    Market market;
    uint256 newCollateralIndex;
    uint256 newCollateralAmount;

    function attack() external {
        // Pre-fund with new collateral token
        // Position already has genuine bad debt (existing collateral < debt at maxLif)
        midnight.liquidate(market, 0, 0, 0, address(this), false, address(this), address(this), "");
    }

    function onLiquidate(...) external returns (bytes4) {
        // Supply new collateral AFTER badDebt has been applied
        IERC20(market.collateralParams[newCollateralIndex].token)
            .approve(address(midnight), newCollateralAmount);
        midnight.supplyCollateral(market, newCollateralIndex, newCollateralAmount, address(this));
        return CALLBACK_SUCCESS;
    }
}

function testBadDebtOverRealization() public {
    // 1. Setup market with borrower having genuine bad debt
    // 2. Record lossFactor and totalUnits before attack
    // 3. Execute attack()
    // 4. Assert lossFactor increased beyond what true bad debt warrants
    // 5. Assert borrower position now has newCollateralAmount collateral
    //    but debt was reduced as if that collateral didn't exist
    // 6. Assert lender credit is slashed more than the true shortfall
}
```

**Expected result:** `_marketState.lossFactor` after the attack exceeds the value it would have if `newCollateralAmount` had been included in the `badDebt` computation, and the borrower retains the newly supplied collateral while having received unwarranted debt forgiveness.
