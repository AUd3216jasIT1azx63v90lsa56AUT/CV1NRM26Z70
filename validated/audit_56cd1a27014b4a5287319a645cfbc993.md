Audit Report

## Title
Systematic per-lender credit over-slash via compounded `mulDivDown` rounding in `updatePositionView` — (File: src/Midnight.sol)

## Summary
In `liquidate`, the `lossFactor` update uses `mulDivDown` to floor the surviving fraction, rounding `lossFactor` upward. In `updatePositionView`, `postSlashCredit` applies a second `mulDivDown` against the already-rounded `lossFactor`. Both floors compound in the same direction: lenders collectively lose up to N extra units per bad-debt event (N = number of lenders with non-zero credit), with those units permanently orphaned inside `totalUnits` and unclaimable by any party.

## Finding Description
**Root cause — double floor in the same direction:**

`lossFactor` update (`src/Midnight.sol:631-633`):
```solidity
_marketState.lossFactor = UtilsLib.toUint128(
    type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
);
```
Let `MAX = type(uint128).max`, `A = MAX − old_lossFactor`. The surviving multiplier stored is `⌊A·(T−B)/T⌋`, so `new_lossFactor = MAX − ⌊A·(T−B)/T⌋ ≥ exact`. The first floor makes `lossFactor` too large.

`postSlashCredit` (`src/Midnight.sol:805-806`):
```solidity
uint256 postSlashCredit = _lastLossFactor < type(uint128).max
    ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
    : 0;
```
For a lender synced before the event (`_lastLossFactor = old_lossFactor`):
`postSlashCredit_i = ⌊c_i · ⌊A·(T−B)/T⌋ / A⌋`

This is a double floor. The `mulDivAddDownDown` Certora rule (`certora/specs/MulDiv.spec:30-33`) proves inductively that `Σ postSlashCredit_i ≤ ⌊(Σc_i)·⌊A·(T−B)/T⌋/A⌋ ≤ T−B`. Each floor loses at most 1 unit, so the lower bound is `Σ postSlashCredit_i ≥ (T−B) − N`.

**Concrete minimal example:**
- 2 lenders, `credit_1 = credit_2 = 1`; `totalUnits = 2`, `badDebt = 1`, `old_lossFactor = 0`
- `new_lossFactor = MAX − ⌊MAX/2⌋ = MAX − (2^127−1) = 2^127`
- Each lender: `postSlashCredit = ⌊1·(2^127−1)/(2^128−1)⌋ = 0`
- `totalUnits` after bad debt = 1, `Σ credit_i = 0` → 1 unit permanently orphaned

**Why existing checks fail:**
- No Certora rule or on-chain check verifies `Σ credit_i ≥ totalUnits − withdrawable` after slashing.
- `totalUnitsEqualsSumNegativeDebtPlusWithdrawable` (`certora/specs/Midnight.spec:123-124`) tracks debt+withdrawable, not lender credits.
- `BalanceEffects.spec` `liquidateEffects` (`certora/specs/BalanceEffects.spec:170-184`) only checks per-user credit is unchanged by `liquidate` itself (before `updatePosition`).
- `LossFactor.spec` rules (`certora/specs/LossFactor.spec:63-70`) verify per-lender sync, not aggregate.
- Test tolerance `assertApproxEqAbs(..., 1)` (`test/LiquidationTest.sol:361`) covers single-lender rounding only.

## Impact Explanation
Lenders collectively lose up to N extra units per bad-debt event beyond their proportional share of the actual bad debt. These units remain counted in `totalUnits` (and eventually in `withdrawable` after borrower repayment) but no lender holds credit to claim them, making them permanently locked in the contract. Over K bad-debt events with N lenders each, up to N·K units are irreversibly orphaned. This violates the accounting invariant `Σ credit_losses = total_bad_debt_realized` and constitutes permanent, irreversible value leakage from lenders. For lenders with minimum-unit positions (credit = 1), the rounding can eliminate 100% of their remaining post-slash credit.

## Likelihood Explanation
Triggered by any bad-debt liquidation in a market with more than one lender — the standard operating condition for any active market. No privileged access, no oracle manipulation, and no special sequencing is required beyond a normal bad-debt liquidation call. The effect accumulates monotonically over the market lifetime and is amplified by markets with many lenders or frequent bad-debt events.

## Recommendation
Eliminate the double floor by storing the surviving multiplier directly rather than recomputing it. One approach: store `survivingFraction = MAX − lossFactor` as a monotonically decreasing value, so `postSlashCredit = credit.mulDivDown(new_survivingFraction, old_survivingFraction)` uses only a single `mulDivDown`. Alternatively, use `mulDivUp` for one of the two operations so the two rounding errors partially cancel rather than compound. At minimum, add an invariant test verifying `Σ credit_i + continuousFeeCredit ≥ totalUnits − withdrawable − ε·N` after each bad-debt event.

## Proof of Concept
Minimal Foundry test:
```solidity
// Setup: market with 2 lenders each supplying 1 unit, 1 borrower with 1 unit debt
// Set oracle price to trigger full bad debt (badDebt = 1)
// Call: midnight.liquidate(market, 0, 0, 0, borrower, false, address(this), address(0), "")
// Assert: midnight.totalUnits(id) == 1
// Call: midnight.updatePosition(market, lender1); midnight.updatePosition(market, lender2)
// Assert: midnight.creditOf(id, lender1) == 0  // both lenders wiped
// Assert: midnight.creditOf(id, lender2) == 0
// Assert: midnight.totalUnits(id) == 1          // 1 unit orphaned — invariant violated
```
The existing `testRealizeOnlyBadDebt` fuzz test (`test/LiquidationTest.sol:347-362`) can be extended to a two-lender variant to reproduce this deterministically.
