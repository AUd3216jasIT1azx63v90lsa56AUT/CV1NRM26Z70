Audit Report

## Title
`consumableUnits` ignores `offer.reduceOnly`, causing bundler to silently skip permanently-reverting offers and revert with `OutOfOffers` - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary
`ConsumableUnitsLib.consumableUnits` computes fillable units solely from `consumed[maker][group]` vs `offer.maxUnits`/`offer.maxAssets`, never reading `offer.reduceOnly` or the maker's on-chain credit/debt. When a `reduceOnly` offer's maker holds zero credit (sell) or zero debt (buy), `consumableUnits` returns a positive value, but every subsequent `Midnight.take()` call reverts with `MakerCreditOrDebtIncreased`. The bundler's `catch {}` silently absorbs the revert, `filledUnits` is never incremented, and the post-loop `require(filledUnits == targetUnits, OutOfOffers())` fires, reverting the entire bundler transaction.

## Finding Description

**Root cause — `consumableUnits` never reads `reduceOnly`:**

`src/periphery/ConsumableUnitsLib.sol` lines 14–23 confirm the function reads only `consumed`, `maxUnits`, `maxAssets`, and `offer.buy`:

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    if (offer.maxUnits > 0) {
        return offer.maxUnits.zeroFloorSub(consumed);   // no reduceOnly check
    } else if (offer.buy) {
        return TakeAmountsLib.buyerAssetsToUnits(...);
    } else {
        return TakeAmountsLib.sellerAssetsToUnits(...);
    }
}
``` [1](#0-0) 

The function's NatSpec states it "Returns a number of units such that it fully consumes the offer." For a `reduceOnly` offer whose maker has no position to reduce, the correct return value is 0; the function instead returns `maxUnits - consumed > 0`.

**`Midnight.take()` enforces the constraint that `consumableUnits` ignores:**

`src/Midnight.sol` contains the `reduceOnly` guard (confirmed: 26 matches for `MakerCreditOrDebtIncreased|sellerDebtIncrease|buyerCreditIncrease`). For a `reduceOnly` sell offer where `sellerPos.credit == 0`, `sellerDebtIncrease = units`, so the require always fails for any `units > 0`. The same applies to a `reduceOnly` buy offer where `debt == 0`.

**Bundler silently absorbs the revert across all four bundle functions:**

All four bundle functions in `src/periphery/MidnightBundles.sol` follow the same pattern:

```solidity
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // returns N > 0
);
try IMidnight(MIDNIGHT).take(..., unitsToTake, ...) returns (...) {
    filledUnits += unitsToTake;
} catch {}   // MakerCreditOrDebtIncreased swallowed here
``` [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The bundler's own comment at line 42 acknowledges: "Skips every reason why take can revert (including ones that are not asynchrony related)." The `try/catch` is intentional, but `consumableUnits` is the designated pre-filter; it fails to return 0 for permanently-unfillable `reduceOnly` offers. [6](#0-5) 

**Exploit flow:**

1. Maker (attacker) signs a `reduceOnly` sell offer with `maxUnits = N` while holding `credit = 0` in the target market (or withdraws all credit after signing — no privileged access required).
2. Taker (victim) constructs a bundler call with `targetUnits = N` and `takes = [attackerOffer]`.
3. Bundler calls `consumableUnits` → returns `N` (positive; no `reduceOnly` check).
4. Bundler calls `take(offer, ..., N, ...)` → `sellerDebtIncrease = N`, `reduceOnly` check fails → reverts with `MakerCreditOrDebtIncreased`.
5. `catch {}` swallows the revert; `filledUnits` stays 0.
6. Loop ends; `require(0 == N, OutOfOffers())` → entire bundler call reverts.

The same path applies to a `reduceOnly` buy offer where the maker's `debt == 0`.

## Impact Explanation

A maker can publish a `reduceOnly` offer with no existing credit/debt. Any taker who routes through the bundler with this offer as the sole (or gap-filling) entry will have their transaction revert with `OutOfOffers`. The taker loses gas and may miss time-sensitive operations (e.g., liquidation-adjacent collateral withdrawal, rate-sensitive borrowing). The bundler's `pullToken`/`forceApproveMax` sequence executes before the loop, so the taker's tokens are temporarily pulled and then returned on revert — no permanent fund loss, but the transaction is reliably and repeatably griefable. This constitutes service unavailability/severe degradation under realistic attacker input, an explicitly in-scope impact class per RESEARCHER.md. [7](#0-6) 

## Likelihood Explanation

Preconditions require only that a maker creates a `reduceOnly` offer with `maxUnits > 0` and `credit == 0` (sell) or `debt == 0` (buy) — a valid, signable offer state requiring no privileged access. The maker can withdraw all credit after signing to satisfy the precondition retroactively. The attack is trivially reachable by any unprivileged user, repeatable across any market and all four bundler functions, and the maker's offer remains valid (not consumed, not expired) after the bundler reverts, enabling repeated griefing of the same taker until they manually exclude the offer.

## Recommendation

In `ConsumableUnitsLib.consumableUnits`, add a `reduceOnly` check before computing units. If `offer.reduceOnly` is true, query the maker's current credit (for sell offers) or debt (for buy offers) from `IMidnight(midnight)` and return `min(computedUnits, makerCredit)` or `min(computedUnits, makerDebt)` respectively. This ensures the pre-filter correctly returns 0 for permanently-unfillable `reduceOnly` offers, preventing the bundler from attempting a `take()` that will always revert. [1](#0-0) 

## Proof of Concept

1. Deploy Midnight and MidnightBundles on a local fork.
2. Attacker creates a `reduceOnly` sell offer signed with `maxUnits = 100`, `group = G`, in market `M`, while holding `credit = 0` (or calls `withdrawCredit` after signing).
3. Victim calls `buyWithUnitsTargetAndWithdrawCollateral(targetUnits=100, ..., takes=[attackerOffer])`.
4. Observe: `consumableUnits` returns 100; `take()` reverts with `MakerCreditOrDebtIncreased`; `catch {}` swallows it; `filledUnits == 0`; `require(0 == 100, OutOfOffers())` fires.
5. Confirm the entire bundler transaction reverts with `OutOfOffers`.
6. Repeat steps 3–5 to demonstrate repeatability — the offer is never consumed.
