Audit Report

## Title
LiquidatorGate checks `msg.sender` but loan tokens are pulled from `callback` payer, enabling gate bypass - (File: src/Midnight.sol)

## Summary
The `liquidate` function validates the `liquidatorGate` exclusively against `msg.sender`, but when `callback != address(0)` the actual loan token repayment is pulled from the `callback` address and collateral is sent to the caller-controlled `receiver`. A whitelisted address A can call `liquidate` with `callback=B` and `receiver=B`, making gate-blocked address B the true economic actor â€” paying loan tokens and receiving collateral â€” while the gate never checks B. The protocol's stated invariant that "the liquidator gate restricts who may liquidate" is broken.

## Finding Description

**Gate check** (`src/Midnight.sol` lines 597â€“600): Only `msg.sender` is validated. [1](#0-0) 

**Payer assignment** (`src/Midnight.sol` line 679): When a callback is provided, the payer becomes the callback address. [2](#0-1) 

**Token pull** (`src/Midnight.sol` line 717): Loan tokens are pulled from `payer` (the callback address). [3](#0-2) 

**Collateral transfer** (`src/Midnight.sol` line 696): Collateral is sent to `receiver`, which is caller-controlled. [4](#0-3) 

**Exploit flow:**
1. Market has a `liquidatorGate` that whitelists A and blocks B.
2. B deploys a contract implementing `ILiquidateCallback`, pre-approves Midnight to pull loan tokens from it.
3. A (whitelisted) calls `midnight.liquidate(market, ..., receiver=B, callback=B, ...)`.
4. Gate calls `canLiquidate(A)` â†’ `true` â†’ passes.
5. Collateral is transferred to B (`receiver=B`).
6. `onLiquidate` is called on B; B returns `CALLBACK_SUCCESS`.
7. Loan tokens are pulled from B (`payer=B`).
8. B has paid the loan tokens and received the collateral â€” the full economic outcome of a liquidation â€” despite being blocked by the gate.

**Why existing checks fail:**

The Certora rule `liquidatorGateBlocksLiquidation` (`certora/specs/Reverts.spec` lines 324â€“332) only verifies that `canLiquidate(msg.sender)` returning false causes a revert. It does not constrain the `callback` or `receiver` addresses against the gate. [5](#0-4) 

The `OnlyExplicitPayerCanLoseTokens.spec` line 128 explicitly sets `liquidateCallbackAllowed = true` for `liquidate`, confirming the callback-as-payer path is intentional but the gate gap is unaddressed. [6](#0-5) 

The existing `GateTest.sol` liquidator gate tests (lines 269â€“300) only test with `callback=address(0)` and never exercise the callback bypass path. [7](#0-6) 

The `live_context.json` test scenario `liquidator_gate_bypass` (lines 302â€“308) explicitly lists "alternate actor/callback/authorized address" as a bypass vector to test and asserts "only approved liquidator can liquidate," confirming the protocol designers identified this surface â€” but the gap remains unaddressed in both the implementation and the formal verification rules. [8](#0-7) 

## Impact Explanation
The `liquidatorGate` access control is fully bypassed for the actual economic actor. The protocol documentation states `"liquidator_gate_restricts_who_may_liquidate": true` and the NatSpec states "The liquidator gate can prevent the user from liquidating borrowers in the market." [9](#0-8) [10](#0-9) 

Any market relying on `liquidatorGate` to enforce KYC, regulatory, or trust-based restrictions on liquidators has those restrictions rendered ineffective. A gate-blocked address B can pay for and profit from liquidations (receiving seized collateral) in a market configured to exclude it. This is a direct authorization bypass with concrete asset movement consequences.

## Likelihood Explanation
The attack requires a whitelisted liquidator A to submit the transaction with `callback=B`. If A is a public/permissionless liquidation router contract (a common DeFi pattern), any external user can supply the `callback` argument directly, requiring zero cooperation from A. Even with a private EOA as A, the collusion is low-friction: B deploys a callback contract, pre-approves Midnight, and asks any whitelisted address to submit one transaction. The condition is repeatable on every liquidatable position in every gated market.

## Recommendation
When a `liquidatorGate` is set and `callback != address(0)`, also validate the callback address against the gate:

```solidity
require(
    market.liquidatorGate == address(0)
        || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
// Add: also gate the callback (actual payer)
require(
    callback == address(0)
        || market.liquidatorGate == address(0)
        || ILiquidatorGate(market.liquidatorGate).canLiquidate(callback),
    LiquidatorGatedFromLiquidating()
);
```

Additionally, consider whether `receiver` (collateral recipient) should also be gated, depending on the intended semantics of the gate. Update the Certora rule `liquidatorGateBlocksLiquidation` and add a `GateTest.sol` test exercising the callback path with a blocked callback address.

## Proof of Concept
Minimal Foundry test extending `GateTest`:

```solidity
contract BlockedCallbackLiquidator is ILiquidateCallback {
    IMidnight midnight;
    IERC20 loanToken;
    constructor(address _midnight, address _loanToken) {
        midnight = IMidnight(_midnight);
        loanToken = IERC20(_loanToken);
        loanToken.approve(_midnight, type(uint256).max);
    }
    function onLiquidate(...) external returns (bytes4) {
        return CALLBACK_SUCCESS;
    }
}

function testLiquidatorGateBypassViaCallback(uint256 units) public {
    units = bound(units, 1, MAX_TEST_AMOUNT * 3 / 4);
    gate.setWhitelisted(lender, true);
    gate.setWhitelisted(borrower, true);
    gate.setWhitelisted(liquidator, true);   // A is whitelisted
    // blockedActor (B) is NOT whitelisted

    BlockedCallbackLiquidator blocked = new BlockedCallbackLiquidator(address(midnight), address(loanToken));
    deal(address(loanToken), address(blocked), units); // B pre-funds

    collateralize(gatedMarket, borrower, units);
    take(units, lender, borrowerOffer);
    Oracle(gatedMarket.collateralParams[0].oracle).setPrice(ORACLE_PRICE_SCALE / 2);

    uint256 collateralBefore = collateralToken1.balanceOf(address(blocked));

    // A calls liquidate with callback=blocked, receiver=blocked
    vm.prank(liquidator);
    midnight.liquidate(gatedMarket, 0, 1, 0, borrower, false, address(blocked), address(blocked), "");

    // Assert: blocked (gate-excluded) received collateral
    assertGt(collateralToken1.balanceOf(address(blocked)), collateralBefore);
}
```

Expected result without fix: test passes (bypass succeeds). Expected result with fix: test reverts with `LiquidatorGatedFromLiquidating`.
