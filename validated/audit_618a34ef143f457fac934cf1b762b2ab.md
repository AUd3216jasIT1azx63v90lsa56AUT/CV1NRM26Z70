Audit Report

## Title
Zero-`buyerAssets` rounding bypasses `maxAssets` cap, allowing unlimited free position mutations - (File: `src/Midnight.sol`)

## Summary
In `take()`, when `offer.maxAssets > 0` and `units * buyerPrice < WAD`, `buyerAssets` truncates to zero via `mulDivDown`. The `consumed` mapping is incremented by zero, the `maxAssets` cap check passes trivially, yet position state (`credit`, `debt`, `totalUnits`) is mutated using `units` directly with zero token transfer. This allows a taker to bypass a fully-consumed offer's cap an unlimited number of times at zero cost.

## Finding Description
In `src/Midnight.sol`, `take()` computes `buyerAssets` via fixed-point division before using it to gate the `maxAssets` cap:

**Truncation to zero**: At line 363, `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. With `units = 1` and any tick except `MAX_TICK`, `tickToPrice` returns a value that is a multiple of `PRICE_ROUNDING_STEP = 1e12` but strictly less than `WAD = 1e18`. Thus `mulDivDown(1, price, 1e18) = 0`. [1](#0-0) [2](#0-1) 

**Cap check bypassed**: At lines 367–369, `consumed[offer.maker][offer.group] += buyerAssets` adds `0`, so `newConsumed` is unchanged. `require(newConsumed <= offer.maxAssets)` passes trivially even when the offer is fully consumed. [3](#0-2) 

**Position mutations proceed from `units`**: Lines 382–414 compute `buyerCreditIncrease`, `sellerDebtIncrease`, and `totalUnits` changes directly from `units`, not from `buyerAssets`. With `units = 1`, all three increase by 1. [4](#0-3) 

**Zero token transfer**: Line 455 transfers `buyerAssets - sellerAssets = 0` tokens. [5](#0-4) 

**Root cause**: A denomination mismatch — `consumed` is asset-denominated but position mutations are unit-denominated — with no guard enforcing `buyerAssets > 0` before proceeding.

**Existing checks that fail**:
- `require(newConsumed <= offer.maxAssets)` — passes because `+= 0`.
- `require(atMostOneNonZero(maxAssets, maxUnits))` — validates offer structure only, not fill size.
- No `require(units > 0)` or `require(buyerAssets > 0)` guard exists. [6](#0-5) 

## Impact Explanation
A taker can fill a buy offer with `maxAssets > 0` an unlimited number of times at zero token cost. Each fill increases the maker's credit and the taker's debt by `units` without any token transfer and without advancing `consumed`. This violates the core accounting invariant that every credit increase must correspond to valid consumed accounting, and that offers cannot be overfilled or reused. The taker accrues unbounded debt obligations and the maker gains unbounded credit — both without any economic exchange — which can be used to manipulate market state, drain settlement fee accounting, or exploit downstream liquidation and settlement logic.

## Likelihood Explanation
Preconditions are minimal: (1) a buy offer with `maxAssets > 0` exists (standard offer type); (2) `tickToPrice(offer.tick) < WAD` — true for every valid tick except `MAX_TICK`; (3) taker uses `units = 1`. No privileged access, oracle manipulation, or front-running is required. The attack is executable in a single transaction and is trivially repeatable.

## Recommendation
Add a guard immediately after computing `buyerAssets` and `sellerAssets` to reject fills that produce zero asset amounts:

```solidity
require(buyerAssets > 0 || units == 0, ZeroBuyerAssets());
```

Alternatively, enforce a minimum `units` value such that `units * buyerPrice >= WAD` before proceeding, or switch the `consumed` accounting to be unit-denominated when `maxAssets` is used (eliminating the denomination mismatch entirely).

## Proof of Concept
The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` directly reproduces the issue:

1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (price < WAD).
2. Pre-fill `consumed` to `maxAssets` via `setConsumed`.
3. Call `take(1, borrower, lenderOffer)`.
4. Assert `buyerAssets == 0`, `consumed` unchanged at `maxAssets`, token balances unchanged.
5. Assert `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased. [7](#0-6)

        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
