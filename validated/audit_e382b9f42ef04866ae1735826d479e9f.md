### Title
Sponsor Can Redirect Full Pool Sweep Post-Outcome by Changing `recoveryAddress` Without a Finality Lock — (File: src/ConfidencePool.sol)

### Summary
`setRecoveryAddress` carries no finality gate. Unlike `expiry` (locked by `expiryLocked` after the first stake) and pool scope (locked by `scopeLocked` once the registry leaves pre-attack staging), `recoveryAddress` remains freely mutable by the pool owner at any point — including after the moderator has flagged a CORRUPTED outcome and before the first sweep call executes. A malicious sponsor can therefore attract stakers with a legitimate recovery address, wait for a bad-faith CORRUPTED flag, and atomically redirect the entire pool to themselves.

### Finding Description
`setRecoveryAddress` in `ConfidencePool.sol` imposes only a zero-address guard:

```solidity
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
``` [1](#0-0) 

There is no check on `expiryLocked`, `claimsStarted`, or `outcome`. By contrast, `setExpiry` is gated by `expiryLocked` (set on the first stake) and `setPoolScope` is gated by `scopeLocked` (set on the first non-staging registry observation). Both locks exist precisely to protect staker reliance on parameters they verified before depositing. [2](#0-1) [3](#0-2) 

`recoveryAddress` is the destination for every fund-moving sweep path:

- `claimCorrupted` — sweeps the entire pool balance to `recoveryAddress` in bad-faith CORRUPTED [4](#0-3) 
- `sweepUnclaimedCorrupted` — sweeps remainder after attacker window to `recoveryAddress` [5](#0-4) 
- `sweepUnclaimedBonus` — sweeps excess bonus/dust to `recoveryAddress` [6](#0-5) 

After `flagOutcome(CORRUPTED, false, address(0))` is called by the moderator, `claimsStarted` is still `false` and `outcome` is `CORRUPTED`, but `setRecoveryAddress` remains callable. The sponsor can front-run any `claimCorrupted` call in the same block or in a subsequent block before anyone sweeps. [7](#0-6) 

### Impact Explanation
In a bad-faith CORRUPTED resolution the entire pool — `snapshotTotalStaked + snapshotTotalBonus` — is swept to `recoveryAddress`. A sponsor who changes `recoveryAddress` to their own EOA immediately after the moderator's flag captures 100 % of all staker principal and bonus contributions. Stakers have no recourse: `withdraw` is permanently disabled once `riskWindowStart != 0`, and the claim paths (SURVIVED/EXPIRED) are blocked by the CORRUPTED outcome. The loss is total and irreversible. [8](#0-7) 

### Likelihood Explanation
The pool sponsor is not a trusted actor under the contest's adversarial model (only the moderator/DAO and factory owner carry the "trusted" qualifier). A malicious sponsor can execute this attack whenever the moderator flags bad-faith CORRUPTED, which is the normal resolution path for an in-scope breach with no named whitehat. The sponsor controls pool creation and can deliberately set up a pool with a reputable-looking `recoveryAddress` to attract stakers, then swap it out the moment the outcome is sealed. No special on-chain conditions beyond a CORRUPTED registry state are required. [9](#0-8) 

### Recommendation
Add a one-way finality latch to `setRecoveryAddress` mirroring the pattern already used for `expiry` and scope. The simplest consistent fix is to block changes once the first stake has been received (i.e., once `expiryLocked` is true), since that is the moment stakers begin relying on pool parameters. Alternatively, block changes once `outcome != UNRESOLVED` or once `claimsStarted` is true. Either gate eliminates the front-run window while preserving the sponsor's ability to correct a misconfigured address before any capital is at risk.

```solidity
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    if (expiryLocked) revert RecoveryAddressLocked(); // add analogous lock
    ...
}
```

### Proof of Concept
1. Sponsor deploys a pool via `createPool` with `recoveryAddress = 0xDAO` (a credible DAO treasury address).
2. Stakers verify `recoveryAddress == 0xDAO` on-chain and deposit; `expiryLocked` flips `true`.
3. The agreement is breached; the moderator calls `flagOutcome(CORRUPTED, false, address(0))` (bad-faith). `outcome = CORRUPTED`, `claimsStarted = false`.
4. In the same block (or any subsequent block before `claimCorrupted` is called), the sponsor calls `setRecoveryAddress(sponsorEOA)`. The call succeeds — no lock exists.
5. Any caller (including the sponsor) calls `claimCorrupted`. The function reads `recoveryAddress` live and transfers the entire pool balance to `sponsorEOA`. [8](#0-7) 

All staker principal and bonus are permanently lost to the sponsor. The attack requires no flash loan, no reentrancy, and no registry manipulation — only a single `setRecoveryAddress` transaction between the moderator's flag and the first sweep.

## 10. Sponsor trust surface

- **`recoveryAddress`** — CORRUPTED sweep destination. Receives the full pool (including stakers'
  principal) under bad-faith CORRUPTED; only excess/dust under SURVIVED/EXPIRED/good-faith.
- **`expiry`** — sponsor-mutable only **until the first stake** (one-way `expiryLocked` latch).
  This protects staker reliance: once anyone has deposited against a given deadline (which feeds
  the k=2 weighting as `T` for the EXPIRED path), the sponsor cannot move it. The latch
  intentionally does **not** reset when stake is withdrawn — resetting it would let the sponsor
  move `expiry` during an all-stakers-exited moment and harm the next cohort.
- **Pool scope** — see §8.

All staker-relevant state is on-chain; stakers should verify pool parameters before depositing.
The `recoveryAddress` value is informational only on the BattleChain agreement side (the
agreement's `Chain.assetRecoveryAddress` is a CAIP-2 string), so the pool stores its own typed
`address recoveryAddress` for token transfers.
