Audit Report

## Title
Re-entrant `take()` via `onBuy()` Bypasses State-Dependent `enterGate.canIncreaseCredit` Check - (File: src/Midnight.sol)

## Summary
`Midnight.take()` commits `buyerPos.credit` to storage before invoking the `onBuy()` buyer callback, and no reentrancy guard prevents a re-entrant `take()` call during that callback. A malicious buyer callback can re-enter `take()` for the same buyer in the same gated market; the inner call's `canIncreaseCredit` gate check executes against the already-mutated intermediate credit state, allowing a gate with credit-threshold logic (e.g., `creditOf(id, buyer) < X`) to be bypassed and the buyer to accumulate credit beyond the gate's intended cap.

## Finding Description

**Exact code path in `src/Midnight.sol`:**

1. Gate check (lines 397–401): `canIncreaseCredit(buyer)` is called with pre-update state.
2. State update (line 410): `buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease)` is written to persistent storage.
3. Buyer callback (lines 445–453): `IBuyCallback(buyerCallback).onBuy(...)` is called with credit already committed to storage. [1](#0-0) [2](#0-1) [3](#0-2) 

The only transient lock in `take()` is `LIQUIDATION_LOCK_SLOT`, which exclusively protects the seller from liquidation during callbacks. It does not prevent re-entry into `take()` itself. [4](#0-3) [5](#0-4) 

**Exploit flow:**

Precondition: gate implements `canIncreaseCredit(buyer)` as `creditOf(id, buyer) < X` (reads Midnight state; a natural credit-cap design).

1. Outer `take()`: buyer credit = 0. Gate sees 0 < X → returns `true`. Gate check passes.
2. Outer `take()` writes `buyerPos.credit = Y` (where Y < X) to storage.
3. Outer `take()` calls `onBuy()` on malicious callback.
4. Malicious callback re-enters `take()` for same buyer, different offer, same market.
5. Inner `take()`: gate calls `canIncreaseCredit(buyer)` → reads `creditOf = Y`. Since Y < X, returns `true`. Gate check passes.
6. Inner `take()` writes `buyerPos.credit = Y + Z` to storage.
7. Both calls complete. Final credit = Y + Z > X. Gate's cap is violated.

**Why existing checks fail:**

The Certora spec (`certora/specs/Reverts.spec` lines 30–34) explicitly models callbacks as pure (no state changes), with the comment: *"For gate rules, gate checks precede callbacks so re-entrant state changes cannot affect them."* This is only true for the outer call's gate check. The inner call's gate check executes after the outer call's state mutation. [6](#0-5) 

The `enterGateBlocksCreditIncrease` rule (lines 295–307) uses `ghostCanIncreaseCredit`, a ghost function returning a fixed value per gate address. It does not model state-dependent gate logic and cannot catch this reentrancy scenario. [7](#0-6) 

The attacker controls `takerCallback` (for a sell offer taken by the attacker) — this is a fully attacker-controlled input with no privilege requirement. [8](#0-7) 

## Impact Explanation
A buyer whose callback is a malicious contract can accumulate credit in a gated market beyond the gate's intended per-user cap. Any gate whose `canIncreaseCredit` logic reads Midnight's own `creditOf` (or any state that `take()` mutates before the callback) is vulnerable. This directly violates the protocol invariant that gate checks must apply to all entry paths and that callbacks must not corrupt partial state through reentrancy. The `IEnterGate` interface declares `canIncreaseCredit` as `external view`, making state-reading gate implementations a natural and expected design. [9](#0-8) 

## Likelihood Explanation
Preconditions are realistic: (1) a gated market exists with a credit-threshold gate (a natural design for permissioned lending pools), (2) the attacker is the taker and sets `takerCallback` to a malicious contract (fully attacker-controlled, no privilege required), (3) a second valid offer exists in the same market (easily arranged by the attacker as maker). The attack is repeatable within a single transaction and requires no special timing or oracle manipulation.

## Recommendation
Add a transient-storage reentrancy guard to `take()` (analogous to `LIQUIDATION_LOCK_SLOT` but scoped to the entire `take()` execution), or move the gate checks to after all state mutations are complete and before the callbacks are invoked. Alternatively, move the buyer callback to before the state update (checks-effects-interactions pattern). The Certora spec's assumption that callbacks are pure should also be revisited to model re-entrant `take()` calls.

## Proof of Concept
Deploy a `CreditCapGate` implementing `canIncreaseCredit(address buyer) → creditOf(id, buyer) < CAP`. Deploy a `MaliciousCallback` implementing `onBuy(...)` that re-enters `midnight.take()` with a second buy offer for the same buyer in the same gated market. In a single transaction:
1. Call `midnight.take()` with `takerCallback = address(MaliciousCallback)` for a sell offer where the attacker is taker (buyer).
2. Inside `onBuy()`, `MaliciousCallback` calls `midnight.take()` again for a second offer.
3. Assert `midnight.creditOf(id, buyer) > CAP` after the transaction completes.
