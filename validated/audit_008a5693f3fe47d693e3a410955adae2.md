### Title
Front-Running DoS via Nonce Consumption in `EcrecoverAuthorizer.setIsAuthorized()` - (File: src/periphery/EcrecoverAuthorizer.sol)

---

### Summary

`EcrecoverAuthorizer.setIsAuthorized()` accepts a signed `Authorization` struct and can be called by any `msg.sender`. Because the function increments `nonce[authorization.authorizer]` on every successful call and performs no check binding `msg.sender` to `authorization.authorizer`, an attacker can observe a pending transaction in the mempool, copy the `authorization` and `signature` parameters verbatim, and submit the same call with higher gas. This consumes the authorizer's nonce, causing the original submitter's transaction to revert with `InvalidNonce()`. The attacker can repeat this indefinitely, permanently denying use of the `EcrecoverAuthorizer` path for the targeted authorizer.

---

### Finding Description

`EcrecoverAuthorizer.setIsAuthorized()` is a meta-transaction entry point: it verifies an EIP-712 signature from `authorization.authorizer` (or a delegate already authorized on Midnight), then calls `IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer)`.

The nonce check and increment are:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

There is no requirement that `msg.sender == authorization.authorizer`. The function is `external` with no caller restriction. [1](#0-0) 

Because the full `authorization` struct and `signature` are calldata visible on-chain before inclusion, any observer can:

1. Watch the mempool for a call to `setIsAuthorized`.
2. Re-submit the identical `(authorization, signature)` pair with higher gas.
3. Their transaction mines first, consuming `nonce[authorization.authorizer]`.
4. The original submitter's transaction reverts: `authorization.nonce` no longer matches the incremented storage value.

The attacker gains nothing from the authorization itself (the parameters are fixed by the signature), but they can repeat this for every new signed message the authorizer produces, making the `EcrecoverAuthorizer` path permanently unusable for that account. [2](#0-1) 

---

### Impact Explanation

- **No direct fund loss**: the authorization content (`authorized`, `isAuthorized`) is fixed by the signature; the attacker cannot alter what gets authorized.
- **Denial of Service**: the authorizer's signed message is consumed before their own transaction lands. They must sign a fresh message with the new nonce, which the attacker can front-run again indefinitely.
- **Broken meta-transaction flow**: the entire purpose of `EcrecoverAuthorizer` is to let a third party (relayer, bundler) submit authorizations on behalf of a user. Persistent front-running makes this path unusable for the targeted authorizer.
- The authorizer retains the fallback of calling `Midnight.setIsAuthorized` directly (which requires `msg.sender == onBehalf` or an existing on-chain authorization), but the off-chain-signature convenience path is fully griefable. [3](#0-2) 

---

### Likelihood Explanation

- **No privileged access required**: any unprivileged attacker with mempool visibility can execute this.
- **Cheap to execute**: the attacker only needs to copy calldata and bump gas; no capital is locked.
- **Realistic scenario**: MEV bots already monitor mempools for profitable or griefing opportunities. A competitor or griever targeting a specific user's authorization flow is a realistic threat on any public EVM chain.
- **Persistent**: the attack can be repeated for every new nonce the victim signs, requiring no state setup between rounds. [4](#0-3) 

---

### Recommendation

Bind the nonce consumption to `msg.sender` when the caller is the authorizer themselves, or add a `msg.sender` check when the intent is to restrict who may submit a given signed message. The simplest mitigation (matching the fix applied in the referenced PoolTogether report) is to require that the caller is either the authorizer or an already-authorized account before processing the signature:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(
        msg.sender == authorization.authorizer
            || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, msg.sender),
        Unauthorized()
    );
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
    // ... rest unchanged
}
```

This ensures only the authorizer (or their existing delegate) can consume the nonce, eliminating the front-running surface entirely. [2](#0-1)

### Proof of Concept

**Setup**: Alice wants to authorize Bob on Midnight via `EcrecoverAuthorizer`. She signs:

```
Authorization {
    authorizer: Alice,
    authorized: Bob,
    isAuthorized: true,
    nonce: 0,          // current nonce[Alice] == 0
    deadline: T+3600
}
```

and broadcasts the transaction `setIsAuthorized(authorization, signature)`.

**Attack**:

1. Attacker Eve observes Alice's pending transaction in the mempool.
2. Eve copies `(authorization, signature)` verbatim and submits with `gasPrice = Alice.gasPrice + 1`.
3. Eve's transaction mines first:
   - `nonce[Alice]` advances from `0` → `1`.
   - `Midnight.setIsAuthorized(Bob, true, Alice)` executes — Bob is authorized (same outcome Alice intended).
4. Alice's transaction mines next:
   - `require(0 == nonce[Alice]++)` → `require(0 == 1)` → **reverts with `InvalidNonce()`**.

**Persistent DoS**: Alice signs a new message with `nonce: 1`. Eve front-runs again. Alice can never successfully submit through `EcrecoverAuthorizer`. She must call `Midnight.setIsAuthorized` directly, defeating the purpose of the meta-transaction contract. [5](#0-4)

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
