### Title
Ineffective balance-diff defense against rebasing tokens breaks claim accounting — (File: src/ConfidencePool.sol)

### Summary
The balance-diff check in `stake()` and `contributeBonus()` is explicitly described as "defense-in-depth against the factory allowlist admitting a fee-on-transfer **or rebasing** token." This claim is false for rebasing tokens. The check only detects fee-on-transfer discrepancies at the moment of transfer; a rebasing token passes it cleanly at deposit time. If a proxy stake token is upgraded to rebasing behavior post-allowlist, the pool's strict internal accounting diverges from the actual balance, and later claimants cannot withdraw their full entitlement.

### Finding Description
`stake()` and `contributeBonus()` both execute a before/after balance snapshot:

```solidity
// src/ConfidencePool.sol lines 236-241
uint256 balanceBefore = stakeToken.balanceOf(address(this));
stakeToken.safeTransferFrom(msg.sender, address(this), amount);
uint256 balanceAfter = stakeToken.balanceOf(address(this));
uint256 received = balanceAfter > balanceBefore ? balanceAfter - balanceBefore : 0;
if (received == 0) revert NoTokensReceived();
if (received < minStake) revert BelowMinStake();
```

For a fee-on-transfer token this correctly catches `received < amount`. For a rebasing token it does nothing useful: rebasing adjusts every holder's balance via a global multiplier change, not via a transfer, so `received == amount` at deposit time regardless of whether the token will later rebase. The check passes, and the pool records the full nominal amount in `eligibleStake[staker]`, `totalEligibleStake`, and `totalBonus`.

The factory's `allowedStakeToken` mapping is the only real gate, and it is checked only at `createPool` time:

```solidity
// src/ConfidencePoolFactory.sol line 31
// Checked only at `createPool` time, so de-listing a token does not affect existing pools.
mapping(address token => bool allowed) public override allowedStakeToken;
```

A proxy stake token that is a standard ERC20 when allowlisted and when the pool is created can later have its implementation upgraded by its own owner to introduce negative rebasing. From that point the pool's actual `stakeToken.balanceOf(address(this))` shrinks below the sum of all recorded liabilities (`totalEligibleStake + totalBonus`), while the accounting variables remain at their original nominal values.

At resolution, `claimSurvived()` and `claimExpired()` pay each claimant `userEligible + bonusShare` — both derived from the strict recorded amounts:

```solidity
// src/ConfidencePool.sol lines 393-403
uint256 bonusShare = _bonusShare(msg.sender, userEligible);
uint256 payout = userEligible + bonusShare;
...
stakeToken.safeTransfer(msg.sender, payout);
```

Early claimants drain the actual balance. Later claimants' `safeTransfer` calls revert because the contract no longer holds enough tokens, permanently locking their principal and bonus share.

### Impact Explanation
Stakers who claim after the pool balance has been eroded by a negative rebase cannot recover their stake or bonus. Their funds are permanently locked in the contract with no recovery path: `claimSurvived`/`claimExpired` revert on transfer, `sweepUnclaimedBonus` reserves `totalEligibleStake + (snapshotTotalBonus - claimedBonus)` and sweeps only the excess above that reserve (which is zero or negative after a rebase), and there is no admin escape hatch for stakers. The loss is proportional to the magnitude of the rebase and the number of stakers who have not yet claimed.

### Likelihood Explanation
The factory allowlist is the primary defense and requires the factory owner to make a governance error to allowlist a natively rebasing token. However, the code comment in `stake()` explicitly names "a proxy-token upgrade post-allowlist" as the scenario the balance-diff is supposed to guard against. A proxy token whose implementation is upgraded by its own owner after allowlisting is a realistic path that bypasses the allowlist entirely and is not controlled by any trusted actor in the ConfidencePool system. The factory owner cannot retroactively protect existing pools by de-listing the token.

### Recommendation
1. Remove or correct the comment claiming the balance-diff is a defense against rebasing tokens — it is not. The comment creates a false sense of security.
2. Document clearly that the balance-diff only catches fee-on-transfer discrepancies at deposit time and provides no protection against post-deposit rebasing.
3. Consider adding a runtime invariant check at resolution (`flagOutcome`, `claimExpired`) that asserts `stakeToken.balanceOf(address(this)) >= snapshotTotalStaked + snapshotTotalBonus` and reverts or emits a warning if the balance has drifted, so the moderator can act before claims begin.
4. Prefer allowlisting only non-upgradeable ERC20 tokens, or tokens whose upgrade path is controlled by the same DAO that controls the factory.

### Proof of Concept
1. Factory owner allowlists token `T`, a standard ERC20 implemented behind an upgradeable proxy.
2. Sponsor creates pool `P` with stake token `T`.
3. Staker A deposits 100 T → `eligibleStake[A] = 100`, `totalEligibleStake = 100`. Balance-diff check passes (`received == 100`).
4. Staker B deposits 100 T → `eligibleStake[B] = 100`, `totalEligibleStake = 200`. Balance-diff check passes.
5. Bonus contributor deposits 50 T → `totalBonus = 50`. Balance-diff check passes.
6. Pool's actual balance: 250 T. Recorded liabilities: 250 T. Consistent.
7. Token `T`'s owner upgrades the proxy implementation to introduce a 60% negative rebase. All balances globally shrink by 60%.
8. Pool's actual balance: 100 T. Recorded liabilities still 250 T.
9. Registry reaches PRODUCTION; moderator flags SURVIVED. `snapshotTotalStaked = 200`, `snapshotTotalBonus = 50`.
10. Staker A calls `claimSurvived()`: payout ≈ 100 + 25 = 125 T. `safeTransfer(A, 125)` succeeds (balance was 100, reverts). Actually with 100 T in the contract, A's transfer of 125 reverts immediately — or if A's share is smaller, A drains the pool and B gets nothing.
11. Staker B calls `claimSurvived()`: `safeTransfer(B, ...)` reverts — insufficient balance. B's 100 T principal and bonus share are permanently locked. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
