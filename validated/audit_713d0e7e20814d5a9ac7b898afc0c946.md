Audit Report

## Title
Zero-price tick enables unbacked credit and debt creation with no loan token transfer - (`src/libraries/TickLib.sol`, `src/Midnight.sol`)

## Summary
`TickLib.tickToPrice(0)` returns `0`, confirmed by the protocol's own test suite and Certora formal verification. Because tick 0 satisfies `0 % tickSpacing == 0` for any spacing, and `take` contains no guard requiring `offerPrice > 0`, a sell offer at tick 0 in a zero-settlement-fee market produces `buyerAssets = 0` and `sellerAssets = 0` while still granting the buyer real credit units and accumulating real debt on the seller with zero loan token transfer. When `maxAssets > 0` is used, `consumed` is never incremented (since `sellerAssets = 0`), allowing the same offer to be taken an unlimited number of times. The attacker's accumulated credit can drain `withdrawable` funded by legitimate borrowers' repayments.

## Finding Description

**Root cause — `tickToPrice(0) == 0`:**

The protocol's own test explicitly asserts this: [1](#0-0) 

And the Certora formal verification rule confirms it: [2](#0-1) 

The `tickToPrice` implementation rounds to multiples of `PRICE_ROUNDING_STEP = 1e12`, and for tick 0 the result rounds down to 0: [3](#0-2) 

**Tick 0 always passes the spacing check:** [4](#0-3) 

`DEFAULT_TICK_SPACING = 4` and `0 % 4 == 0`, so tick 0 is always accessible: [5](#0-4) 

**Settlement fee defaults to zero:**

`defaultSettlementFeeCbp` is a zero-initialized storage mapping. `touchMarket` copies it into the market state: [6](#0-5) 

All markets start with `_settlementFee = 0` unless the privileged `feeSetter` explicitly configures fees.

**No guard on zero price in `take`:** [7](#0-6) 

With `offerPrice = 0` and `_settlementFee = 0`, both `buyerAssets` and `sellerAssets` compute to 0. There is no `require(offerPrice > 0)` anywhere in `take`.

**Credit and debt are updated regardless:** [8](#0-7) 

`buyerPos.credit += units` and `sellerPos.debt += units` execute unconditionally, independent of the asset amounts.

**Zero tokens are transferred:** [9](#0-8) 

The protocol's own token safety requirements explicitly state tokens "should not revert because a transfer is no-op": [10](#0-9) 

**`maxAssets` consumption is never incremented:** [11](#0-10) 

For a sell offer, `sellerAssets = 0`, so `consumed[offer.maker][offer.group] += 0`. `newConsumed` stays at 0, which always satisfies `<= offer.maxAssets`. The same offer can be taken an unlimited number of times.

**Health check does not prevent the invariant violation:** [12](#0-11) 

`isHealthy` only verifies that collateral covers debt at oracle price. It does not verify that the protocol's loan token balance covers outstanding credit. A seller with sufficient collateral passes the check even though zero loan tokens entered the contract.

**Exploit flow (two-account attack):**

1. Attacker account A (maker/seller) supplies sufficient collateral and creates a sell offer at tick 0 with `maxAssets > 0`.
2. Attacker account B (taker/buyer) calls `take` with `units > 0`. Self-take is blocked by `require(offer.maker != taker, SelfTake())` at line 354, so two accounts are required — trivially satisfied.
3. B pays 0 loan tokens, receives `units` of credit. A accumulates `units` of debt, receives 0 loan tokens.
4. `_marketState.totalUnits` increases by `units` with no corresponding loan token deposit.
5. Step 2 is repeated arbitrarily (since `sellerAssets = 0` never increments `consumed`), bounded only by A's collateral-backed `maxDebt`.
6. When legitimate borrowers repay, `withdrawable` increases via `repay`: [13](#0-12) 

7. B calls `withdraw` to drain those tokens, stealing from legitimate lenders: [14](#0-13) 

## Impact Explanation

B holds credit claims unbacked by any deposited loan tokens. When legitimate borrowers repay debt, `withdrawable` increases and B can drain it before legitimate lenders. The `totalUnits == sumDebt + withdrawable` accounting invariant is maintained in ledger terms (A's debt equals B's credit), but the solvency invariant is violated: the contract's loan token balance does not cover all outstanding credit redemptions. This constitutes direct, concrete theft of legitimate lenders' deposited loan tokens. Severity: Critical.

## Likelihood Explanation

All preconditions are trivially satisfied by any unprivileged user: tick 0 is always accessible at any tick spacing, zero settlement fees are the default for all new markets, and the two-account requirement is trivial. No privileged access is required. The exploit is repeatable without bound when `maxAssets > 0` is used. Any market with default (zero) settlement fees is vulnerable from the moment of creation.

## Recommendation

Add a guard in `take` requiring a non-zero offer price:

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
require(offerPrice > 0, ZeroPrice());
```

This prevents any offer at tick 0 (or tick 1, which also maps to 0) from being taken, eliminating the root cause. Alternatively, enforce a minimum tick in `take` (e.g., `require(offer.tick >= 2)`), or add a minimum `buyerAssets` / `sellerAssets` floor. Additionally, consider adding a guard that `sellerAssets > 0` when `offer.maxAssets > 0` to prevent the unlimited-take bypass independently.

## Proof of Concept

Minimal Foundry test plan:

```solidity
// 1. Deploy Midnight, create a market with default (zero) settlement fees.
// 2. Account A: supplyCollateral sufficient to back N units of debt.
// 3. Account A: authorize a ratifier that always returns CALLBACK_SUCCESS.
// 4. Account A: construct a sell Offer at tick=0, maxAssets=type(uint256).max.
// 5. Account B: call take(offer, ..., units=1000e18, taker=B, ...).
//    Assert: B.credit increased by 1000e18, A.debt increased by 1000e18,
//            loanToken.balanceOf(Midnight) unchanged.
// 6. Repeat step 5 N times. Assert consumed[A][group] == 0 after each call.
// 7. Simulate a legitimate lender depositing via a buy offer and a borrower repaying.
//    Assert: withdrawable > 0.
// 8. Account B: call withdraw(market, 1000e18, B, B).
//    Assert: B receives real loan tokens; legitimate lender cannot withdraw their full amount.
```

The test directly demonstrates that `consumed` never increments (step 6), that `totalUnits` grows without token deposit (step 5), and that B can drain legitimate lenders' repayments (step 8).
