Audit Report

## Title
`maxAssets` cap on buy offers bypassed via zero-rounding when `buyerPrice < WAD` - (File: `src/Midnight.sol`)

## Summary
When a buy offer uses `maxAssets > 0` as its fill cap, the consumed delta is `buyerAssets = mulDivDown(units, buyerPrice, WAD)`. For any tick where `buyerPrice < WAD` (all ticks below `MAX_TICK`), a taker submitting `units = 1` causes `buyerAssets` to floor to zero, so `consumed` never advances and the `ConsumedAssets` guard never fires. The offer can be filled an unlimited number of times, inflating the maker's credit and `totalUnits` without any corresponding asset transfer. The protocol's own NatDoc at line 94 and the test `testBugBuyMaxAssetsBypass` confirm the path is reachable and the state change is real.

## Finding Description

**Exact code path — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // floors to 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

uint256 newConsumed;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause:** For a buy offer (`offer.buy == true`) with `maxAssets > 0`, the consumed delta is `buyerAssets`. When `buyerPrice < WAD` and `units = 1`, `mulDivDown(1, buyerPrice, WAD) = floor(buyerPrice / WAD) = 0`. The `consumed` mapping is incremented by zero, so `newConsumed` never changes and `require(newConsumed <= offer.maxAssets)` is trivially satisfied on every call regardless of how many times the offer has been filled.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — `newConsumed` is always 0 (or whatever it was before), so this never fires.
- `require(atMostOneNonZero(offer.maxAssets, offer.maxUnits))` — satisfied; only `maxAssets` is set.
- No minimum-units or minimum-assets floor guard exists.

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = M > 0`, `maxUnits = 0`, and any tick where `buyerPrice < WAD` (e.g., `MAX_TICK - 16`).
2. Taker calls `take(offer, ..., units=1, ...)` N times.
3. Each call: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; `consumed` stays at 0; guard passes; `buyerPos.credit += 1` (line 410); `sellerPos.debt += 1` (line 414); `totalUnits` incremented (line 416); zero tokens transferred (line 455).
4. After N calls: `consumed[maker][group] == 0`, maker's credit has grown by N, taker's debt has grown by N, with zero asset transfer.

**Protocol acknowledgment:** The NatDoc at line 94 states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test `testBugBuyMaxAssetsBypass` (lines 858–889) explicitly proves the state change occurs even when `consumed` is already at `maxAssets`, asserting `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase.

## Impact Explanation

The `maxAssets` cap — the protocol's primary mechanism for limiting a buy offer's total fill exposure — is completely ineffective whenever `buyerPrice < WAD`. This condition holds for all ticks below `MAX_TICK`, which is the normal operating range. Consequences:

- **Accounting invariant broken:** `consumed` is supposed to track and bound total fill activity, but it never advances, so the invariant `consumed <= maxAssets` is vacuously satisfied while actual fill activity is unbounded.
- **Unbacked credit inflation:** The maker accumulates credit (units) without any corresponding loan token deposit. At maturity, credit entitles the holder to receive loan tokens; unbacked credit inflates claims against the protocol's asset pool.
- **`totalUnits` inflation:** `_marketState.totalUnits` grows without bound (line 416–417), corrupting market-wide accounting used in fee calculations and withdrawals.
- **Taker self-harm / griefing vector:** The taker takes on debt with zero assets received. A malicious actor controlling both sides (maker and taker via separate addresses) can manufacture unbounded credit for the maker at the cost of unbounded debt on the taker address, then exploit the maker's inflated credit in downstream interactions.

## Likelihood Explanation

**Preconditions:**
1. A buy offer with `maxAssets > 0` and any tick below `MAX_TICK` (the normal range for all practical prices below 1 WAD per unit). Zero settlement fee is not required; the condition `buyerPrice < WAD` is satisfied by the vast majority of valid ticks.
2. The taker is any unprivileged address — no special role, no admin key, no oracle manipulation required.
3. The taker must supply collateral for the debt they take on, but the token cost per fill is zero.

**Feasibility:** Trivially repeatable. The only cost is gas. The condition `buyerPrice < WAD` is the default operating range. No external dependencies or timing constraints are required.

## Recommendation

Add a minimum-assets floor check before updating `consumed`. Specifically, for a buy offer with `maxAssets > 0`, require that `buyerAssets > 0` before proceeding, or revert if `units * buyerPrice < WAD`:

```solidity
if (offer.maxAssets > 0) {
    require(buyerAssets > 0, ZeroBuyerAssets()); // prevent zero-rounding bypass
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` such that `units * buyerPrice >= WAD` when `maxAssets > 0` and `offer.buy == true`. This ensures every fill advances `consumed` by at least 1, preserving the cap invariant.

## Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` lines 858–889 is a complete, self-contained PoC:

1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.maxUnits = 0`, `lenderOffer.tick = MAX_TICK - 16` (price < WAD).
2. Pre-set `consumed` to `maxAssets` via `setConsumed` (simulating a fully consumed offer).
3. Call `take(1, borrower, lenderOffer)`.
4. Assert: `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged at `maxAssets`, yet `creditOf(lender)` increased, `debtOf(borrower)` increased, `totalUnits` increased.
