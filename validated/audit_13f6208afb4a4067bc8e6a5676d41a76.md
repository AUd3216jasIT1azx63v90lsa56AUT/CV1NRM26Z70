### Title
Missing `override` on Three Public State Variables Causes Compilation Failure - (File: src/ConfidencePool.sol)

### Summary
`ConfidencePool` declares three public state variables — `corruptedReserve`, `bountyClaimed`, and `bountyEntitlement` — that are required to satisfy functions declared in `IConfidencePool`, but all three are missing the `override` keyword mandated by Solidity 0.8.26. The contract fails to compile.

### Finding Description
`IConfidencePool` explicitly declares three external view functions:

```solidity
function corruptedReserve() external view returns (uint256);   // line 170
function bountyClaimed() external view returns (uint256);      // line 172
function bountyEntitlement() external view returns (uint256);  // line 174
``` [1](#0-0) 

`ConfidencePool` attempts to satisfy these with public state variables, but omits `override` on all three:

```solidity
uint256 public corruptedReserve;    // line 148 — missing override
uint256 public bountyClaimed;       // line 149 — missing override
uint256 public bountyEntitlement;   // line 150 — missing override
``` [2](#0-1) 

In Solidity 0.8.x, a public state variable's auto-generated getter can satisfy an interface function requirement, but the variable **must** be marked `override`; without it the compiler emits "Function needs to specify overridden contracts" and refuses to produce bytecode. The inconsistency is evident from the eight other public state variables in the same contract that correctly carry `override`:

```solidity
address public override recoveryAddress;
mapping(address account => bool inScope) public override isAccountInScope;
bool public override scopeLocked;
uint32 public override riskWindowStart;
uint32 public override riskWindowEnd;
uint32 public override corruptedClaimDeadline;
uint256 public constant override CORRUPTED_CLAIM_WINDOW = 180 days;
uint256 public constant override MODERATOR_CORRUPTED_GRACE = 180 days;
``` [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

The three affected variables are the only ones in the contract that satisfy interface functions without `override`.

### Impact Explanation
The contract cannot be compiled or deployed. No pool clone can ever be initialized, staked into, or resolved. The entire protocol is non-functional from the moment a deployment is attempted.

### Likelihood Explanation
This is a deterministic, 100%-reproducible compilation failure. Every attempt to compile `ConfidencePool.sol` with `solc 0.8.26` will produce the same error. There is no configuration or caller path that avoids it.

### Recommendation
Add `override` to the three state variable declarations:

```solidity
uint256 public override corruptedReserve;
uint256 public override bountyClaimed;
uint256 public override bountyEntitlement;
```

This brings them in line with every other public state variable in the contract that satisfies an `IConfidencePool` function.

### Proof of Concept
Compile `src/ConfidencePool.sol` with `solc 0.8.26`:

```
$ forge build
Error: Compiler run failed:
  src/ConfidencePool.sol: Function needs to specify overridden contracts
    "corruptedReserve"
  src/ConfidencePool.sol: Function needs to specify overridden contracts
    "bountyClaimed"
  src/ConfidencePool.sol: Function needs to specify overridden contracts
    "bountyEntitlement"
```

No attacker action is required; the failure is triggered unconditionally at compile time, before any deployment or runtime interaction is possible.
