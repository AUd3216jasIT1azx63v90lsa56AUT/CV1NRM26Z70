Audit Report

## Title
Zero-asset take bypasses full-consumption guard on buy offers with `buyerPrice < WAD` - (File: `src/Midnight.sol`)

## Summary
When `offer.buy == true` and `offer.maxAssets > 0`, `buyerAssets` is computed with `mulDivDown`, which truncates to zero when `units * buyerPrice < WAD`. Because the consumed cap check uses `<=`, a taker can call `take` with `units > 0` after the offer is fully consumed (`consumed == maxAssets`), pass the guard with `newConsumed = maxAssets + 0 = maxAssets`, and mutate position state (`credit`, `debt`, `totalUnits`) without transferring any assets. This is repeatable indefinitely on the same fully-consumed offer.

## Finding Description

**Root cause — `src/Midnight.sol` line 363:**

`buyerAssets` is computed via `mulDivDown`, which performs plain integer division:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
``` [1](#0-0) 

`mulDivDown` is `(x * y) / d` with no rounding protection: [2](#0-1) 

When `units * buyerPrice < WAD` (= 1e18), this returns exactly 0. [3](#0-2) 

**Root cause — `src/Midnight.sol` lines 367–369:**

The consumed cap check adds this (potentially zero) value and enforces `<=`:

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [4](#0-3) 

When `consumed == maxAssets` and `buyerAssets == 0`: `newConsumed = maxAssets + 0 = maxAssets ≤ maxAssets` — the check passes unconditionally.

**Position state still mutates with `units > 0`:**

After the guard passes, `buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` all increase by `units`: [5](#0-4) 

**Token transfers are zero:**

Both transfers use `buyerAssets` and `sellerAssets`, which are both 0 (since `sellerPrice < buyerPrice < WAD` means `sellerAssets` also truncates to zero): [6](#0-5) 

**Exploit flow:**
1. Attacker constructs a buy offer with `tick` set to a negative tick (any tick where `offerPrice < WAD`), `maxAssets > 0`.
2. Attacker fills the offer legitimately until `consumed[maker][group] == maxAssets`.
3. Attacker calls `take(offer, ..., units=1)`. `mulDivDown(1 * buyerPrice, WAD) = 0`. Cap check passes.
4. `buyerPos.credit += 1`, `sellerPos.debt += 1`, `_marketState.totalUnits += 1`. Zero tokens transferred.
5. Step 3–4 is repeatable indefinitely.

**Why existing checks fail:** There is no floor check requiring `buyerAssets > 0` when `units > 0`. The `<=` guard only prevents `newConsumed` from exceeding `maxAssets`; it does not prevent a zero-increment take from proceeding once the cap is already reached.

The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` explicitly names and demonstrates this as a bug. [7](#0-6) 

## Impact Explanation

A fully-consumed buy offer can be taken an unbounded number of additional times with `units > 0` and zero asset transfer. Each such take manufactures unbacked credit for the maker (buyer) and unbacked debt for the taker (seller), and inflates `_marketState.totalUnits`. This directly violates the core accounting invariant that `consumed == maxAssets` means no further fills are permitted, enabling creation of unbacked credit/debt — a critical integrity failure in the protocol's financial state.

## Likelihood Explanation

All preconditions are reachable by an unprivileged taker:
- `offer.buy == true`, `offer.maxAssets > 0` — standard offer configuration.
- `buyerPrice < WAD` — any tick where `offerPrice < WAD` (negative tick); fully attacker-controlled via `offer.tick`.
- `consumed == maxAssets` — reachable by the attacker themselves via prior legitimate fills.

No privileged access, leaked keys, or victim mistakes are required. The attack is repeatable indefinitely on the same offer after full consumption.

## Recommendation

Add a floor check requiring `buyerAssets > 0` when `units > 0` and `offer.maxAssets > 0` with `offer.buy == true`. Concretely, after computing `buyerAssets` on line 363, add:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(buyerAssets > 0, ZeroAssetTake());
}
```

This ensures that any take with `units > 0` must result in a non-zero asset transfer, preventing zero-increment takes from bypassing the consumption cap. An equivalent fix is to enforce `require(units == 0 || buyerAssets > 0)` before the consumed accounting block.

## Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` demonstrates the bug. A minimal manual reproduction:

1. Deploy with `offer.buy = true`, `offer.maxAssets = N`, `offer.tick` set to a negative tick (e.g., tick = -1, so `offerPrice < WAD`).
2. Call `take` with `units` sized so that `buyerAssets` accumulates to exactly `N` (legitimate fills).
3. Assert `consumed[maker][group] == N`.
4. Call `take` again with `units = 1`. Observe: no revert, `buyerPos.credit` increases by 1, `sellerPos.debt` increases by 1, `_marketState.totalUnits` increases by 1, zero tokens transferred.
5. Repeat step 4 indefinitely — each call succeeds and inflates unbacked state. [8](#0-7)
