### Title
Borrower Can DoS Their Own Liquidation by Activating a Reverting-Oracle Collateral — (File: src/Midnight.sol)

### Summary
In `liquidate()`, the function iterates over every collateral in the borrower's `collateralBitmap` and calls `IOracle(...).price()` on each one. If any single oracle reverts, the entire liquidation reverts. Because `supplyCollateral` never calls the oracle, a borrower whose position is underwater can supply 1 wei of a collateral whose oracle is reverting, activate that collateral in their bitmap, and permanently block all liquidation attempts against their position — causing unrecoverable bad debt for lenders.

### Finding Description

**Root cause — oracle loop with no skip mechanism**

In `liquidate()`, every activated collateral is queried unconditionally:

```solidity

    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price();   // ← reverts here
    ...
    _collateralBitmap = _collateralBitmap.clearBit(i);
}

The protocol itself acknowledges this gap in the contract-level NatSpec:

> *"Liquidation reverts if any of the activated collaterals' oracle reverts (see LIVENESS). Note that a borrower can activate a collateral once its oracle is reverting because the oracle is not called in supplyCollateral."* [4](#0-3) 

The authorization check on `supplyCollateral` only prevents a third party from poisoning a victim's bitmap; it does **not** prevent the borrower from poisoning their own bitmap.

**Exploit path**

1. A market is created with ≥ 2 collateral params. At least one oracle (`oracleB`) is reverting (paused, deprecated, or buggy — a realistic condition for any long-lived market).
2. Borrower supplies collateral A, borrows, and the position becomes unhealthy.
3. Before a liquidator acts, the borrower calls `supplyCollateral(market, indexB, 1, self)` — supplying 1 wei of collateral B. This sets bit `indexB` in `_position.collateralBitmap` at zero cost.
4. Every subsequent call to `liquidate(market, ...)` enters the bitmap loop, hits `oracleB.price()`, and reverts.
5. `isHealthy` also reverts, so `withdrawCollateral` and `take` (when the seller has debt) also revert for this borrower.
6. The borrower's debt is never cleared; lenders absorb the bad debt permanently.

### Impact Explanation

- **Liquidation is permanently blocked** for the poisoned position as long as `oracleB` reverts.
- **Bad debt accrues to lenders** via the loss-factor mechanism, diluting all credit holders in the market.
- **Post-maturity bad-debt realization** (`liquidate` with `seizedAssets = repaidUnits = 0`) is equally blocked, so the loss cannot even be socialized.
- The borrower retains their borrowed assets while their collateral is locked — a direct financial gain at lenders' expense.

### Likelihood Explanation

- **Oracle failure is realistic**: Chainlink feeds can be paused; custom oracles can be deprecated or contain bugs. Any market with a long maturity is exposed.
- **Attacker cost is 1 wei** of the reverting collateral token plus gas.
- **No privileged access required**: the borrower is a normal user exercising a standard protocol function.
- The protocol's own Certora spec formally proves this revert propagation (`oracleRevertCausesLiquidateRevert`), confirming the path is deterministic. [5](#0-4) 

### Recommendation

Add an `indicesToSkip` parameter to `liquidate` (mirroring EigenLayer's `slashQueuedWithdrawal` fix), allowing liquidators to exclude a specific collateral index from the oracle loop. The skipped collateral's value is treated as zero for `maxDebt`/`badDebt` computation, and its assets are not seized. This preserves liveness for all other collaterals while accepting a conservative (zero-value) treatment of the broken one.

Alternatively, wrap each `IOracle(...).price()` call in a try/catch and treat a reverting oracle as returning price = 0 (the most conservative assumption), consistent with the existing `oracleZeroCausesIsHealthyReturnFalse` Certora rule.

### Proof of Concept

```
Setup:
  market.collateralParams = [
    { token: tokenA, oracle: workingOracle,   lltv: 0.8e18, ... },
    { token: tokenB, oracle: revertingOracle, lltv: 0.8e18, ... }
  ]

Step 1: Borrower supplies tokenA, borrows units.
Step 2: workingOracle.setPrice(price_below_lltv)  → position is unhealthy.
Step 3: Borrower calls supplyCollateral(market, 1 /*indexB*/, 1 /*1 wei*/, borrower)
        → collateralBitmap now has bits 0 and 1 set.
Step 4: Liquidator calls liquidate(market, 0, 0, repaidUnits, borrower, false, ...)
        → loop reaches i=1, calls revertingOracle.price() → REVERT.
Step 5: Liquidator tries liquidate(market, 1, ...) → same revert.
Step 6: Position remains unliquidated; lenders absorb bad debt.
```

The `oracleRevertCausesLiquidateRevert` Certora rule already proves step 4 deterministically for any activated reverting oracle. [6](#0-5) [1](#0-0)

/// If any activated collateral oracle reverts on price, liquidate reverts.
rule oracleRevertCausesLiquidateRevert(env e, Midnight.Market market, uint256 collateralIndex, uint256 seizedAssets, uint256 repaidUnits, address borrower, address receiver, address callback, bytes data, uint256 revertingCollateralIndex, bool postMaturityMode) {
    require singleRevertingOracle == market.collateralParams[revertingCollateralIndex].oracle, "oracle is reverting";

    bytes32 id = summaryToId(market);
    uint128 bitmap = collateralBitmap(id, borrower);
    require summaryGetBit(bitmap, revertingCollateralIndex), "revertingCollateralIndex is activated";

    liquidate@withrevert(e, market, collateralIndex, seizedAssets, repaidUnits, borrower, postMaturityMode, receiver, callback, data);

    assert lastReverted;
}
