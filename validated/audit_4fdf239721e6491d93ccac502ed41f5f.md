Audit Report

## Title
`sellerAssetsToUnits` Division-by-Zero When `sellerPrice == 0` DoS's Periphery Bundler - (File: src/periphery/TakeAmountsLib.sol)

## Summary
In `TakeAmountsLib.sellerAssetsToUnits`, when `offer.buy = true` and `offerPrice == settlementFee`, `sellerPrice` computes to `0`, causing a Solidity 0.8 division-by-zero panic on `targetSellerAssets.mulDivUp(WAD, 0)`. The core `Midnight.take()` does not revert in this state — it computes `sellerAssets = 0` via `mulDivDown(0, WAD)`. Because `sellerAssetsToUnits` is called outside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, the panic propagates and permanently DoS's that bundler function for any offer in this state.

## Finding Description

**Root cause — `TakeAmountsLib.sol` lines 44–46:**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

When `offer.buy = true` and `offerPrice == settlementFee`, `sellerPrice = 0`. The return statement immediately calls `targetSellerAssets.mulDivUp(WAD, 0)` — a Solidity 0.8 panic (division by zero). There is no zero-guard on `sellerPrice`.

The NatSpec at line 34 states *"Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)"* — this only documents the strict underflow case (`<`). For the equal case (`offerPrice == settlementFee`), the NatSpec is silent, yet the periphery still panics while the core does not revert.

**Core does not revert — `Midnight.sol` lines 361–364:**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...;
```

With `sellerPrice = 0`, `units.mulDivDown(0, WAD) = 0`. No revert. `take()` proceeds normally, transferring 0 assets to the seller.

**Bundler call site — `MidnightBundles.sol` lines 285–300:**

`TakeAmountsLib.sellerAssetsToUnits(...)` is called at line 286, **outside** the `try/catch` block that wraps `IMidnight(MIDNIGHT).take(...)` at lines 292–300. The panic propagates uncaught and reverts the entire `supplyCollateralAndSellWithAssetsTarget` call.

**Simplest exploit path (no governance required):**
1. `tickToPrice(0)` evaluates to `0` (confirmed by the formula in `TickLib.sol` lines 44–51: for tick=0, `wExp(LN_ONE_PLUS_DELTA * 2910) ≈ 2e24`, so `1e36 / (1e18 + 2e24) ≈ 5e11`, which rounds down to `0` after `divHalfDownUnchecked(PRICE_ROUNDING_STEP=1e12)`).
2. A market with `settlementFee = 0` satisfies `offerPrice == settlementFee == 0`.
3. An unprivileged lender creates a buy offer at tick 0.
4. Any caller invokes `supplyCollateralAndSellWithAssetsTarget` with `takes` containing that offer and `targetSellerAssets > 0`.
5. `sellerAssetsToUnits` panics; the entire bundler transaction reverts.

**Alternative path:** Both tick prices and settlement fees are multiples of `PRICE_ROUNDING_STEP = 1e12`. The settlement fee is a piecewise-linear function of TTM that drifts continuously with time and can naturally pass through any tick price value, triggering the same condition without governance action.

**Why existing checks fail:** The NatSpec guard only documents the `offerPrice < settlementFee` case. The equal case is unguarded. The bundler NatSpec at line 246 acknowledges propagation ("Reverts if TakeAmountsLib or ConsumableUnitsLib reverts") but does not identify this specific undocumented trigger.

## Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee` and `targetSellerAssets > 0` reverts unconditionally. The bundler is DoS'd for that offer. Because the `try/catch` only wraps `take()` and not `sellerAssetsToUnits`, there is no graceful fallback. Funds are not frozen in the core (`take()` still works directly), but the periphery path is completely blocked for affected offers.

## Likelihood Explanation
- **No governance required for the simplest case:** tick 0 gives `offerPrice = 0`, and any market with `settlementFee = 0` satisfies the condition.
- **Natural fee drift:** Settlement fees are piecewise-linear in TTM and change continuously. Exact equality with a tick price (both multiples of `1e12`) is structurally reachable without any malicious action.
- **Unprivileged trigger:** Any lender can create a buy offer at the relevant tick; any caller of the bundler triggers the DoS.
- **Repeatability:** Deterministic — every bundler call with that offer reverts until the offer expires or the fee changes.

## Recommendation
Add a zero-guard on `sellerPrice` in `sellerAssetsToUnits` before the division. If `sellerPrice == 0`, either revert with a descriptive error (mirroring the `offerPrice < settlementFee` case) or return a sentinel value. The NatSpec should be updated to explicitly document the equal case. Optionally, move `sellerAssetsToUnits` inside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, or wrap it in its own `try/catch`, to prevent panics from propagating.

## Proof of Concept
1. Deploy a market with `settlementFee = 0` (or wait until TTM causes the interpolated fee to reach 0).
2. Create a buy offer at tick 0 (so `tickToPrice(0) = 0 = settlementFee`).
3. Call `supplyCollateralAndSellWithAssetsTarget` with `takes = [{offer: buyOfferAtTick0, ...}]` and `targetSellerAssets = 1`.
4. Observe the transaction reverts with a Solidity panic (division by zero) at `TakeAmountsLib.sellerAssetsToUnits` line 46.
5. Confirm that calling `IMidnight.take()` directly with the same offer and `units > 0` succeeds and returns `sellerAssets = 0`.
