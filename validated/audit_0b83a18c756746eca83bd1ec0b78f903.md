Audit Report

## Title
`buyWithAssetsTargetAndWithdrawCollateral` reverts entirely when any sell offer has `tick = MAX_TICK` and non-zero settlement fee - (File: `src/periphery/MidnightBundles.sol`)

## Summary

`buyWithAssetsTargetAndWithdrawCollateral` computes `unitsToTake` via `TakeAmountsLib.buyerAssetsToUnits` at line 209, which is placed **outside** the `try/catch` block at line 215. For any sell offer at `tick = MAX_TICK` (5820) in a market with a non-zero settlement fee, `buyerPrice = tickToPrice(MAX_TICK) + settlementFee = WAD + settlementFee > WAD`, triggering the `require(buyerPrice <= WAD)` guard in `buyerAssetsToUnits` and reverting the entire bundle transaction. A malicious maker can exploit this to permanently grief any taker bundle that includes their offer.

## Finding Description

**Exact code path:**

1. The loop in `buyWithAssetsTargetAndWithdrawCollateral` (lines 205–222) evaluates `unitsToTake` via `min(TakeAmountsLib.buyerAssetsToUnits(...), takes[i].units, ConsumableUnitsLib.consumableUnits(...))` at lines 208–214 — entirely before the `try` keyword at line 215.

2. Inside `buyerAssetsToUnits` (TakeAmountsLib.sol lines 26–28), for a sell offer (`offer.buy = false`): `sellerPrice = offerPrice` and `buyerPrice = sellerPrice + settlementFee`. The guard `require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne())` is then evaluated.

3. `tickToPrice(MAX_TICK)` returns exactly `WAD` (1e18): with `tick = 5820`, the exponent argument `LN_ONE_PLUS_DELTA * (2910 - 5820)` is a large negative number, `wExp(...)` collapses to ~0, and the formula yields `1e36 / 1e18 = 1e18 = WAD`.

4. With any `settlementFee > 0`, `buyerPrice = WAD + settlementFee > WAD`, so the `require` reverts. This revert propagates out of the `min(...)` expression, past the `try` keyword, and unwinds the entire transaction.

5. `ConsumableUnitsLib.consumableUnits` does not rescue this: for a sell offer with `maxUnits == 0` it calls `sellerAssetsToUnits` (which has no `buyerPrice <= WAD` guard), and for `maxUnits > 0` it returns directly — neither path triggers the revert first.

**Why existing checks do not stop it:**

- The `try/catch` at lines 215–221 only wraps `IMidnight.take(...)`. The pre-computation of `unitsToTake` is unconditionally outside it.
- `sellerAssetsToUnits` has no `require(buyerPrice <= WAD)` guard, so `ConsumableUnitsLib` does not preempt the failure.
- The NatSpec at line 175 acknowledges "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts," but line 176 promises "If taking an offer reverts, the bundler will completely skip this offer" — the revert from `buyerAssetsToUnits` is not a revert from `take()`, yet it kills the entire bundle, violating the skip guarantee.

## Impact Explanation

The entire `buyWithAssetsTargetAndWithdrawCollateral` transaction reverts. All other valid offers in the `takes[]` array are blocked. The taker's `targetBuyerAssets` transfer (line 197) is also reverted, so funds are not permanently frozen, but the taker's intended purchase is completely blocked for that bundle invocation. A malicious maker can deliberately post a sell offer at `tick = MAX_TICK` in any market with non-zero settlement fee to permanently grief any taker bundle that includes it.

## Likelihood Explanation

The precondition requires only: (1) a sell offer at `tick = MAX_TICK` exists in the market, and (2) the market has a non-zero settlement fee. Both are trivially satisfiable — `MAX_TICK = 5820` is a valid tick accessible at any tick spacing that divides 5820, and non-zero settlement fees are standard. The condition is permanent and structural: it does not depend on market state, timing, or victim mistakes. Any taker who aggregates offers from an order book and includes such an offer triggers the revert.

## Recommendation

Move `TakeAmountsLib.buyerAssetsToUnits` inside the `try/catch` block, or wrap the entire `unitsToTake` computation in a `try/catch`. One approach: extract the `min(...)` computation into an internal function and call it with `try`, catching any revert to skip the offer. Alternatively, add a pre-check that skips offers where `tickToPrice(offer.tick) + settlementFee > WAD` before calling `buyerAssetsToUnits`.

## Proof of Concept

1. Deploy a market with non-zero settlement fee.
2. Maker posts a sell offer at `tick = MAX_TICK` (5820).
3. Taker calls `buyWithAssetsTargetAndWithdrawCollateral` with a `takes[]` array containing this offer (among others).
4. The transaction reverts with `PriceGreaterThanOne()` at `TakeAmountsLib.buyerAssetsToUnits` line 28, before the `try` block is ever reached.
5. All other valid offers in the bundle are blocked.

A minimal unit test: construct an `Offer` with `buy = false`, `tick = MAX_TICK`, mock `settlementFee > 0`, and call `buyerAssetsToUnits` directly — it reverts. Then confirm the same revert propagates through `buyWithAssetsTargetAndWithdrawCollateral` with a forked or local deployment.
