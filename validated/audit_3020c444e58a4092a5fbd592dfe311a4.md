Audit Report

## Title
Reentrancy via `IBuyCallback.onBuy` allows buyer to drain `withdrawable` before paying `buyerAssets` - (File: `src/Midnight.sol`)

## Summary
In `take`, `buyerPos.credit` is committed to storage at line 410 before any external interaction. The `onBuy` callback fires at lines 447–452 while payment has not yet been collected, and a malicious `IBuyCallback` can reenter `withdraw` during this window. Because `withdraw` only checks authorization (satisfied by a pre-authorized callback) and requires `withdrawable >= units` (satisfied in any active market), the attacker drains `units` loan tokens from `withdrawable` and then repays only `buyerAssets = units * buyerPrice / WAD < units`, pocketing the difference for free.

## Finding Description

**Broken CEI ordering in `take`:**

`buyerPos.credit` is incremented at line 410 before any external call:
```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease); // line 410
```

The `onBuy` callback fires at lines 447–452 while payment has not yet been collected:
```solidity
IBuyCallback(buyerCallback).onBuy(...) // lines 447-452
```

Payment only arrives after the callback returns:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // line 455
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets); // line 456
```

**`withdraw` has no reentrancy guard — only an authorization check:**
```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized()); // line 482
```

**The state mutations and token transfer in `withdraw`:**
```solidity
_position.credit -= UtilsLib.toUint128(units);       // line 493
_marketState.withdrawable -= UtilsLib.toUint128(units); // line 494
_marketState.totalUnits -= UtilsLib.toUint128(units);   // line 495
SafeTransferLib.safeTransfer(market.loanToken, receiver, units); // line 499
```

**Exploit flow (taking a sell offer, `offer.buy = false`):**

1. Attacker calls `setIsAuthorized(maliciousCallback, true, taker)` — permissionless, one tx.
2. A market exists with `withdrawable >= units` (any active market after repayments/liquidations).
3. A sell offer exists with `offerPrice + settlementFee < WAD` (standard discounted lending offer).
4. Attacker calls `take(sellOffer, ..., units, taker, ..., maliciousCallback, ...)`.
5. Inside `take`:
   - `buyer = taker`, `buyerCallback = takerCallback = maliciousCallback`, `payer = maliciousCallback` (line 422: `buyerCallback != address(0) ? buyerCallback : ...`).
   - Line 379: `_updatePosition` called for buyer — sets `lastLossFactor = lossFactor`, `lastAccrual = block.timestamp`.
   - Line 410: `buyerPos.credit += units` (taker has zero prior debt, so `buyerCreditIncrease = units`).
   - Lines 447–452: `maliciousCallback.onBuy(...)` is called.
6. Inside `onBuy`, `maliciousCallback` calls `withdraw(market, units, taker, maliciousCallback)`:
   - Auth check passes: `isAuthorized[taker][maliciousCallback] == true`.
   - Line 485: `_updatePosition` re-runs for buyer — since `lastLossFactor == lossFactor` and `lastAccrual == block.timestamp`, no slashing or fee accrual occurs; credit remains `units`.
   - `_position.credit -= units` → 0.
   - `_marketState.withdrawable -= units` (drains other lenders' repaid funds).
   - `safeTransfer(loanToken, maliciousCallback, units)` → callback receives `units` tokens.
7. `onBuy` returns `CALLBACK_SUCCESS`.
8. Back in `take`, lines 455–456: `maliciousCallback` pays `buyerAssets` total.
9. **Net gain = `units − buyerAssets = units * (1 − buyerPrice/WAD) > 0`.**

**Why existing checks fail:**

- `withdraw`'s authorization check is satisfied by the attacker's pre-authorization of the callback — a standard, permissionless operation via `setIsAuthorized`.
- There is no reentrancy guard anywhere in `Midnight.sol`.
- The TOKEN SAFETY REQUIREMENTS comment (`"It should not re-enter Midnight on transfer nor transferFrom"`) applies only to the loan token itself, not to the `onBuy` callback mechanism — the reentrancy here occurs through the callback, not a token transfer.
- The `withdrawable` underflow guard only requires `withdrawable >= units`, which is the stated precondition.

## Impact Explanation
Direct theft of loan tokens from the contract. The attacker drains `withdrawable` — the pool of tokens repaid by borrowers and owed to other lenders — without depositing the corresponding assets. Other lenders who hold credit in the same market and attempt to `withdraw` after the attack will find `withdrawable` insufficient and their calls will revert. The stolen amount per attack is `units * (1 − buyerPrice/WAD)`, which approaches `units` as the offer price approaches zero. This is a critical-severity direct theft of user funds.

## Likelihood Explanation
All preconditions are trivially satisfiable by any unprivileged user:
- `withdrawable >= units`: satisfied in any active market after any repayment or liquidation.
- Sell offer with `offerPrice + settlementFee < WAD`: satisfied by virtually every real lending offer.
- Attacker pre-authorizes callback: one transaction, no privilege, no cost beyond gas.

The attack is fully repeatable. The callback contract is self-funding: it receives `units` tokens from `withdraw` and only needs to return `buyerAssets < units` to `take`, requiring zero net capital beyond gas.

## Recommendation
Apply the checks-effects-interactions (CEI) pattern strictly. Move the `onBuy` callback invocation (lines 447–452) to **after** the `safeTransferFrom` payment calls (lines 455–456), or alternatively add a nonreentrant transient-storage guard (using `tload`/`tstore`, already available per the contract's Cancun dependency) that locks `withdraw`, `repay`, and other state-mutating functions for the duration of any callback execution in `take`.

## Proof of Concept
**Minimal manual steps:**
1. Deploy `MaliciousCallback` implementing `IBuyCallback`. In `onBuy`, call `Midnight.withdraw(market, units, taker, address(this))`.
2. Call `Midnight.setIsAuthorized(address(maliciousCallback), true, taker)` from `taker`.
3. Ensure the target market has `withdrawable >= units` (e.g., after any borrower repayment).
4. Place or identify a sell offer with `offerPrice + settlementFee < WAD`.
5. Call `Midnight.take(sellOffer, ..., units, taker, ..., address(maliciousCallback), ...)`.
6. Observe: `maliciousCallback` receives `units` tokens from `withdraw` inside `onBuy`, then pays only `buyerAssets < units` back in lines 455–456. Net profit = `units - buyerAssets`. Other lenders' subsequent `withdraw` calls revert due to insufficient `withdrawable`.
