### Title
Authorized User Can Front-Run Authorizer's Nonce to Invalidate Pending `setIsAuthorized` Transactions — (`src/periphery/EcrecoverAuthorizer.sol`)

---

### Summary

`EcrecoverAuthorizer.setIsAuthorized` accepts a valid signature from **any address already authorized by the `authorizer` on Midnight**, not only from the `authorizer` themselves. Because the nonce is sequential and keyed per `authorizer`, an authorized user (Bob) can craft and submit a competing `Authorization` message signed by himself using Alice's current nonce, consuming it before Alice's own pending transaction reaches the chain. This permanently invalidates Alice's signed message and can be repeated to block Alice from revoking Bob's access through the meta-transaction flow indefinitely.

---

### Finding Description

**Root cause — dual-signer acceptance on a shared sequential nonce:**

`EcrecoverAuthorizer` stores one sequential nonce per `authorizer` address:

```solidity
mapping(address => uint256) public nonce;
``` [1](#0-0) 

In `setIsAuthorized`, the nonce is checked and incremented **before** the signer is verified:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
``` [2](#0-1) 

The signer check then accepts either the `authorizer` themselves **or** any address already authorized by the `authorizer` on Midnight:

```solidity
require(
    signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
``` [3](#0-2) 

`Midnight.isAuthorized` is a flat `mapping(address authorizer => mapping(address authorized => bool))`:

```solidity
mapping(address authorizer => mapping(address authorized => bool)) public isAuthorized;
``` [4](#0-3) 

**Exploit path:**

1. Alice wants to revoke Bob's authorization. She signs off-chain:
   `Authorization{authorizer=alice, authorized=bob, isAuthorized=false, nonce=N}` and hands it to a relayer.
2. Bob (still authorized by Alice at this moment) observes the relayer's pending transaction in the mempool.
3. Bob signs his own `Authorization{authorizer=alice, authorized=<any address>, isAuthorized=true, nonce=N}` — valid because `isAuthorized[alice][bob] == true`.
4. Bob submits with higher gas, front-running the relayer. Alice's nonce advances to `N+1`.
5. The relayer's transaction reverts with `InvalidNonce`. Alice's revocation never executes.
6. Bob repeats for every subsequent revocation attempt Alice makes through `EcrecoverAuthorizer`.

---

### Impact Explanation

Bob can permanently block Alice from revoking his authorization through the `EcrecoverAuthorizer` meta-transaction flow. Additionally, in step 3 Bob can set `authorized=<arbitrary address>, isAuthorized=true`, silently granting a new address access to Alice's Midnight account as a side-effect of the front-run. Alice retains the ability to call `Midnight.setIsAuthorized` directly on-chain, so the attack is scoped to the off-chain-signed / relayer-submitted authorization path. The impact is **medium**: persistent DoS of the meta-transaction revocation flow plus potential unauthorized authorization grants. [5](#0-4) 

---

### Likelihood Explanation

**Low-to-medium.** Preconditions: Bob must already hold an active authorization from Alice on Midnight, and must monitor the public mempool for Alice's pending `EcrecoverAuthorizer` transactions. Both conditions are realistic on any public EVM chain with a visible mempool. The attack requires no capital and no privileged protocol role beyond the user-level authorization Alice herself granted.

---

### Recommendation

Bind the nonce to the **signer**, not only to the `authorizer`. One approach: require that the `Authorization` struct includes the `signer` address and that the nonce is keyed `nonce[signer]` (or `nonce[authorizer][signer]`). This ensures that Bob consuming a nonce can never invalidate Alice's own signed message, because they draw from separate counters.

Alternatively, restrict `setIsAuthorized` so that only `authorization.authorizer` themselves (not delegated signers) may sign the EIP-712 message, and handle delegation at the Midnight layer directly.

---

### Proof of Concept

```
Setup:
  - Alice calls midnight.setIsAuthorized(ecrecoverAuthorizer, true, alice)
    → EcrecoverAuthorizer is authorized to act on Alice's behalf on Midnight.
  - Alice calls midnight.setIsAuthorized(bob, true, alice)
    → Bob is authorized by Alice on Midnight.
  - ecrecoverAuthorizer.nonce(alice) == 0

Attack:
  1. Alice signs off-chain:
       Authorization{authorizer=alice, authorized=bob, isAuthorized=false, nonce=0, deadline=T}
     and submits to a relayer.

  2. Bob observes the relayer's pending tx. Bob signs:
       Authorization{authorizer=alice, authorized=charlie, isAuthorized=true, nonce=0, deadline=T}
     Bob's signature is valid because isAuthorized[alice][bob] == true.

  3. Bob calls ecrecoverAuthorizer.setIsAuthorized(bobsAuth, bobsSig) with higher gas.
     → nonce[alice] becomes 1. Charlie is now authorized by Alice.

  4. Relayer's tx executes: nonce check fails (0 != 1). Reverts with InvalidNonce.
     → Bob's authorization is never revoked.

  5. Alice re-signs with nonce=1. Bob repeats step 2-4 indefinitely.

Expected result: Alice cannot revoke Bob through EcrecoverAuthorizer.
``` [5](#0-4) [6](#0-5)

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
