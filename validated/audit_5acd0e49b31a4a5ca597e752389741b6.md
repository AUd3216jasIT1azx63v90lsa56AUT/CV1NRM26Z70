Audit Report

## Title
`maxAssets` Cap Bypass via `mulDivDown` Zero-Rounding on Buy Offers - (File: `src/Midnight.sol`)

## Summary
When `offer.buy == true` and `offer.maxAssets > 0`, the consumed tracker is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `mulDivDown` performs plain integer division (`(x * y) / d`), any call where `units * buyerPrice < WAD` produces `buyerAssets = 0`, leaving `consumed[offer.maker][offer.group]` unchanged. An attacker controlling both a maker and a taker address can call `take` repeatedly with sub-WAD unit fills, each passing the `ConsumedAssets` guard, while the maker's credit grows beyond the intended cap.

## Finding Description

**Root cause** — `src/Midnight.sol` lines 363–369 and `src/libraries/UtilsLib.sol` line 29–31:

`mulDivDown` is implemented as plain integer division:
```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;  // truncates toward zero
}
```

In `take`, when `offer.buy == true`:
- `buyerPrice = sellerPrice + _settlementFee = (offerPrice - _settlementFee) + _settlementFee = offerPrice`
- `buyerAssets = units.mulDivDown(buyerPrice, WAD) = (units * offerPrice) / WAD`

With `offerPrice = tickToPrice(2) = 1e12` (confirmed by `test/TickLibTest.sol` line 17) and `units = 1`: `buyerAssets = (1 * 1e12) / 1e18 = 0`.

The consumed guard:
```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy
        ? buyerAssets   // ← 0 when units * offerPrice < WAD
        : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```
never increments `consumed` and never trips. Meanwhile, position state at lines 408–414 is updated based on `units` (not `buyerAssets`), so the maker's credit and taker's debt grow by `units` per call regardless.

Token transfers at lines 455–456 both become 0-value transfers (`buyerAssets - sellerAssets = 0`, `sellerAssets = 0`), which succeed on standard ERC-20 tokens.

**Why existing checks fail**: The health check at line 476 (`require(... || isHealthy(...), SellerIsLiquidatable())`) applies to the taker/seller and requires collateral backing the taker's accumulated debt — this is a real precondition but does not prevent the attack; the attacker pre-funds the taker address with collateral. There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in the path, and no minimum-units check exists in `take`.

**Exploit flow**:
1. Attacker controls `makerAddr` and `takerAddr` (distinct, due to `SelfTake` check at line 354).
2. `makerAddr` posts a buy offer: `offer.buy = true`, `offer.maxAssets = M`, `tick < MAX_TICK` (i.e., `offerPrice < WAD`).
3. `takerAddr` pre-supplies collateral sufficient to remain healthy under accumulated debt.
4. `takerAddr` calls `take` with `units = 1` (or any value where `units * offerPrice < WAD`) N times.
5. Each call: `buyerAssets = 0` → `consumed` unchanged → guard passes → `makerAddr.credit += 1`, `takerAddr.debt += 1`.
6. After N calls: `makerAddr` holds N units of credit; `consumed` remains 0 (or its pre-attack value); the `maxAssets = M` cap is completely inert.
7. As `withdrawable` grows from other borrowers' repayments, `makerAddr` can withdraw N tokens — far exceeding what `maxAssets` was meant to allow.

## Impact Explanation
The `maxAssets` cap on buy offers is rendered completely ineffective for any `tick < MAX_TICK` (i.e., `offerPrice < WAD`), which covers virtually all real buy offers. The maker accumulates unbounded credit beyond the intended cap. Since `withdraw` (line 481–499) transfers `units` of loan tokens directly from the protocol's balance, the maker can drain tokens deposited by other participants up to the `withdrawable` pool, violating the core protocol invariant that offer fills cannot exceed `maxAssets`.

## Likelihood Explanation
- **Preconditions**: Any buy offer with `maxAssets > 0` and `tick < MAX_TICK`. This is the normal operating range — `MAX_TICK` (price = 1.0 WAD) is the par-value boundary.
- **Attacker capability**: Must control two addresses and provide collateral for the taker address to pass the health check. No privileged role required.
- **Feasibility**: Straightforward — choose `units = 1` (or any value below `WAD / offerPrice`) per call.
- **Repeatability**: Unlimited; each fill is independent and the guard never trips.

## Recommendation
Add a guard requiring that a non-zero `units` fill produces a non-zero consumed increment:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)
    : units.mulDivUp(buyerPrice, WAD);

// Add: prevent zero-rounding bypass of maxAssets cap
require(units == 0 || buyerAssets > 0 || sellerAssets > 0, ZeroAssets());
```

Alternatively, use `mulDivUp` for the consumed tracking on buy offers so that even a 1-unit fill counts as at least 1 asset consumed:

```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy
    ? units.mulDivUp(buyerPrice, WAD)   // round UP for cap enforcement
    : sellerAssets;
```

This preserves the buyer-favorable rounding for the actual token transfer while ensuring the cap is never silently bypassed.

## Proof of Concept

Minimal Foundry test (add to `test/TakeTest.sol`):

```solidity
function testMaxAssetsBypassViaZeroRounding() public {
    // Use tick=2: offerPrice = 1e12, so units=1 → buyerAssets = (1*1e12)/1e18 = 0
    uint256 tick = 2;
    uint256 maxAssetsCap = 1e12; // cap = 1 asset unit
    lenderOffer.buy = true;
    lenderOffer.tick = tick;
    lenderOffer.maxAssets = maxAssetsCap;
    lenderOffer.maxUnits = 0;

    // Taker needs collateral to pass health check
    collateralize(market, borrower, 1000);

    // Each fill: buyerAssets=0, consumed unchanged, guard never trips
    for (uint256 i = 0; i < 1000; i++) {
        take(1, borrower, lenderOffer);
    }

    // Maker has 1000 units of credit; consumed is still 0; cap was M=1e12
    assertEq(midnight.creditOf(id, lender), 1000);
    assertEq(midnight.consumed(lender, lenderOffer.group), 0); // cap completely bypassed
}
