Audit Report

## Title
Division by Zero in `sellerAssetsToUnits` via `tickToPrice(0) = 0` Causes DoS of Bundle Flows - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary

`tickToPrice(0)` provably returns `0` due to rounding by `PRICE_ROUNDING_STEP`, as confirmed by a passing unit test. When `consumableUnits` is called with a sell offer (`offer.buy = false`) where `offer.maxUnits = 0` and `offer.maxAssets > 0` at `offer.tick = 0`, it delegates to `sellerAssetsToUnits`, which computes `sellerPrice = 0` and then calls `mulDivDown(targetSellerAssets, WAD, 0)`, triggering an unconditional division-by-zero revert. No guard in `consumableUnits` or `sellerAssetsToUnits` prevents this path, and `MidnightBundles` does not wrap `consumableUnits` in a `try/catch`.

## Finding Description

**Confirmed zero-price fact:**

`test/TickLibTest.sol` line 16 asserts and passes: [1](#0-0) 

The root cause is the double rounding in `tickToPrice`: for tick=0, the intermediate result of `1e36 / (1e18 + wExp(...))` is smaller than `PRICE_ROUNDING_STEP = 1e12`, so `divHalfDownUnchecked(..., PRICE_ROUNDING_STEP) * PRICE_ROUNDING_STEP` collapses to `0`: [2](#0-1) 

**Reachable division-by-zero path:**

`consumableUnits` reaches `sellerAssetsToUnits` when `offer.maxUnits == 0`, `offer.buy == false`, and `offer.maxAssets > 0`: [3](#0-2) 

Inside `sellerAssetsToUnits`, for a sell offer (`offer.buy = false`), `sellerPrice` is set directly to `offerPrice = tickToPrice(0) = 0`, then `mulDivDown(targetSellerAssets, WAD, sellerPrice)` is called with denominator `0`: [4](#0-3) 

**Bundle functions propagate the revert:**

`buyWithUnitsTargetAndWithdrawCollateral` calls `consumableUnits` outside any `try/catch`, and its NatSpec explicitly documents "Reverts if ConsumableUnitsLib reverts": [5](#0-4) [6](#0-5) 

`buyWithAssetsTargetAndWithdrawCollateral` has the same pattern: [7](#0-6) [8](#0-7) 

**Why existing checks fail:**

- `tickToPrice` only guards `tick > MAX_TICK`; tick=0 is in range.
- `consumableUnits` has no guard for `sellerPrice == 0`.
- `sellerAssetsToUnits` has no guard for `sellerPrice == 0`.
- The `NoDivisionByZero.spec` Certora spec covers only `Midnight.sol` and uses `NONDET` for `tickToPrice`, so it does not catch this periphery path: [9](#0-8) 

## Impact Explanation

Any bundle transaction that calls `consumableUnits` on a sell offer with `tick=0`, `maxUnits=0`, `maxAssets>0` reverts unconditionally. Because `MidnightBundles` does not wrap `consumableUnits` in a `try/catch`, the entire bundle call fails. This is a concrete DoS of sell-side bundle flows: any automated system (aggregator, router, front-end) that selects offers from an on-chain order book and includes a malicious offer in the `takes` array will have its entire bundle transaction griefed. Legitimate takers lose gas and fail to execute their intended trades.

## Likelihood Explanation

The preconditions are trivially satisfiable by any unprivileged maker: `tick=0` is always a valid, accessible tick (it satisfies `tick <= MAX_TICK` and `0 % spacing == 0` for any spacing), and setting `maxUnits=0` with `maxAssets>0` is a normal asset-capped offer configuration. The attack is zero-cost (no funds required, no state setup), repeatable on every block, and requires no special role or privilege. The attack is most effective against automated bundle-construction systems (aggregators, routers) that pull offers from an on-chain order book without filtering for zero-price ticks.

## Recommendation

Add a zero-price guard in `sellerAssetsToUnits` (or in `consumableUnits` before delegating to it) to revert or return 0 when `sellerPrice == 0`:

```solidity
// In TakeAmountsLib.sellerAssetsToUnits, after computing sellerPrice:
if (sellerPrice == 0) return type(uint256).max; // or revert with a descriptive error
```

Alternatively, add a guard in `tickToPrice` or `consumableUnits` to handle the zero-price case explicitly. The fix should be consistent with how `buyerAssetsToUnits` handles edge cases.

## Proof of Concept

1. Deploy `MidnightBundles` pointing to a live `Midnight` instance.
2. As an attacker (maker), create a sell `Offer` with `tick=0`, `buy=false`, `maxUnits=0`, `maxAssets=1`.
3. As a taker, call `buyWithUnitsTargetAndWithdrawCollateral` or `buyWithAssetsTargetAndWithdrawCollateral` with a `takes` array that includes the malicious offer.
4. Observe that the call reverts unconditionally due to division by zero in `sellerAssetsToUnits`.

The existing test `testTickToPriceMinMax` already proves `tickToPrice(0) == 0`; a minimal forge test can extend this to call `sellerAssetsToUnits` directly with `tick=0`, `buy=false`, and `targetSellerAssets=1` to confirm the revert. [1](#0-0)
