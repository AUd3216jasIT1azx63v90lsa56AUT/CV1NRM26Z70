Audit Report

## Title
Double `mulDivDown` Rounding Understates `maxDebt` by 1, Enabling Liquidation of Economically Healthy Positions - (File: src/Midnight.sol)

## Summary
Both `isHealthy` and `liquidate` compute `maxDebt` via two sequential `mulDivDown` calls, each discarding a remainder. The combined rounding error can understate `maxDebt` by exactly 1 unit relative to the true single-step floor. When a position sits exactly at the health boundary (`exact maxDebt == debt`), the protocol computes `maxDebt = debt ŌłÆ 1`, causing `isHealthy` to return `false` and the `liquidate` guard to pass, allowing seizure of collateral from an economically healthy borrower.

## Finding Description

**Confirmed code path:**

`mulDivDown` is a plain integer floor division with no rounding protection: [1](#0-0) 

`isHealthy` applies it twice in sequence: [2](#0-1) 

`liquidate` uses the identical double-floor pattern: [3](#0-2) 

The liquidatability guard then compares the understated `maxDebt` against `originalDebt`: [4](#0-3) 

**Root cause ŌĆö double floor rounding:**

Let `V1 = floor(C ├Ś P / ORACLE_PRICE_SCALE)` (step 1 result) and `V2 = floor(V1 ├Ś L / WAD)` (step 2 result). The exact single-step bound is `M = floor(C ├Ś P ├Ś L / (ORACLE_PRICE_SCALE ├Ś WAD))`. Because step 1 discards remainder `r = (C ├Ś P) mod ORACLE_PRICE_SCALE` before step 2 multiplies by `L`, the combined error satisfies `M ŌłÆ 1 Ōēż V2 Ōēż M`. The off-by-one (`V2 = M ŌłÆ 1`) is reachable whenever the two discarded remainders together exceed one full unit.

**Concrete counterexample** using confirmed protocol constants (`ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`, valid LLTV tier `0.77e18`): [5](#0-4) [6](#0-5) 

| Parameter | Value |
|---|---|
| `collateral` (C) | 10 |
| `price` (P) | `1.3e35` |
| `lltv` (L) | `0.77e18` |
| `debt` (D) | 1 |

- Step 1: `floor(10 ├Ś 1.3e35 / 1e36) = floor(1.3) = 1`
- Step 2: `floor(1 ├Ś 0.77e18 / 1e18) = floor(0.77) = 0`
- Exact: `10 ├Ś 1.3e35 ├Ś 0.77e18 / 1e54 = 1.001` ŌåÆ `floor = 1 = debt` ŌåÆ **healthy**
- Protocol sees: `maxDebt = 0 < 1 = debt` ŌåÆ **unhealthy** ŌåÆ liquidation guard passes

**Exploit flow:**

1. Borrower opens a position with C = 10 collateral units and borrows D = 1 debt unit at a price above `1.3e35` (so `isHealthy` returns `true` at the `take` call site at line 476).
2. Oracle price drifts to exactly `1.3e35` ŌĆö the position remains economically healthy (`exact maxDebt = 1 = debt`).
3. Liquidator calls `liquidate(market, collateralIndex, seizedAssets, 0, borrower, false, ...)`.
4. `maxDebt` is computed as 0; guard `originalDebt (1) > maxDebt (0)` passes; collateral is seized.

**Why existing checks fail:**

There are no minimum debt or collateral dust guards anywhere in the codebase that would prevent positions with small integer values from being created. [7](#0-6) 

The health check at the `take` path (line 476) uses the same `isHealthy` function with the same double-floor computation, so it does not prevent the boundary condition from being reached.

## Impact Explanation
A liquidator can seize collateral from a borrower whose position is economically healthy (exact `maxDebt Ōēź debt`) purely due to the double floor-rounding understating `maxDebt` by 1. The borrower suffers an unwarranted collateral loss. The core protocol invariant ŌĆö healthy positions cannot be liquidated ŌĆö is violated. This is a direct, concrete loss of user funds triggered by an unprivileged external caller.

## Likelihood Explanation
No oracle manipulation is required. For any fixed collateral amount `C` and LLTV `L`, there exists a positive-measure interval of oracle prices `P` that triggers the off-by-one. A liquidator needs only to monitor positions and call `liquidate` when the price enters such an interval. With many positions across many markets, the probability of at least one being in the vulnerable range at any given time is non-negligible. The condition is passively reachable through normal price movement.

## Recommendation
Replace the two sequential `mulDivDown` calls with a single combined computation to eliminate the intermediate rounding step:

```solidity
// Instead of:
maxDebt += collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);

// Use a single-step division (requires 512-bit intermediate or restructured denominator):
maxDebt += mulDiv(collateral * lltv, price, ORACLE_PRICE_SCALE * WAD);
// or using a library that handles 512-bit intermediates (e.g., OZ Math.mulDiv):
maxDebt += Math.mulDiv(collateral, price * lltv, ORACLE_PRICE_SCALE * WAD);
```

This must be applied consistently in both `isHealthy` (lines 954ŌĆō955) and `liquidate` (line 613). [2](#0-1) [3](#0-2) 

## Proof of Concept
Minimal Foundry unit test:

```solidity
function test_doubleRoundingOffByOne() public {
    uint256 C = 10;
    uint256 P = 1.3e35;
    uint256 L = 0.77e18;
    uint256 ORACLE_PRICE_SCALE = 1e36;
    uint256 WAD = 1e18;
    uint256 debt = 1;

    // Double-floor (protocol behavior)
    uint256 v1 = (C * P) / ORACLE_PRICE_SCALE;          // = 1
    uint256 maxDebt = (v1 * L) / WAD;                    // = 0

    // Single-step exact floor
    uint256 exact = (C * P / ORACLE_PRICE_SCALE * L) ;   // demonstrates loss
    // True exact (using 512-bit): floor(10 * 1.3e35 * 0.77e18 / 1e54) = 1

    assertEq(maxDebt, 0);   // protocol sees unhealthy
    assert(debt > maxDebt); // liquidation guard passes ŌĆö WRONG
    // Exact bound = 1 = debt, position is healthy
}
```

Deploy a market with the above parameters, supply 10 collateral units, borrow 1 debt unit, set oracle to `1.3e35`, and call `liquidate` ŌĆö the call will succeed despite the position being economically healthy.
