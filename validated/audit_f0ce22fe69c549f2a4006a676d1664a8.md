Audit Report

## Title
Missing `caller` Parameter in `onRepay` Callback Enables Arbitrary Third-Party Token Drain - (File: `src/Midnight.sol`)

## Summary
`Midnight.repay` unconditionally sets `payer = callback` when a non-zero callback is supplied, then invokes `IRepayCallback(callback).onRepay(...)` without passing the transaction initiator (`msg.sender`). Unlike `onLiquidate` and `onFlashLoan`, the `onRepay` interface structurally prevents callback contracts from verifying who triggered the repayment. An attacker with any debt position can designate an arbitrary third-party contract as `callback`, causing that contract's pre-approved loan tokens to be consumed to repay the attacker's debt without the third party's consent.

## Finding Description

**Exact code path:**

`src/Midnight.sol` lines 511–520:
```solidity
address payer = callback != address(0) ? callback : msg.sender;   // line 511

if (callback != address(0)) {
    require(
        IRepayCallback(callback).onRepay(id, market, units, onBehalf, data) == CALLBACK_SUCCESS,
        WrongRepayCallbackReturnValue()
    );
}
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
```

`src/interfaces/ICallbacks.sol` line 21:
```solidity
function onRepay(bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
```

**Structural inconsistency confirmed:**

| Callback | Passes `caller`? | Protocol call site |
|---|---|---|
| `onFlashLoan(address caller, ...)` | ✓ | `flashLoan` line 746: `.onFlashLoan(msg.sender, ...)` |
| `onLiquidate(address caller, ...)` | ✓ | `liquidate` line 701: `.onLiquidate(msg.sender, ...)` |
| `onRepay(bytes32 id, ...)` | ✗ | `repay` line 516: `.onRepay(id, market, units, onBehalf, data)` |

Inside `onRepay`, `msg.sender` is always `Midnight`. Without `caller` being passed, a callback contract has no protocol-supplied mechanism to verify who triggered the repayment.

**Exploit path:**

1. Attacker holds any debt position (trivially opened).
2. `thirdParty` is any contract that: (a) implements `IRepayCallback` and returns `CALLBACK_SUCCESS` from `onRepay`, and (b) has approved Midnight to spend its loan tokens (e.g., a repayment helper, bundler, or position manager).
3. Attacker calls `repay(market, units, onBehalf=attacker, callback=thirdParty, data)`.
4. Authorization check passes: `onBehalf == msg.sender`.
5. `payer = thirdParty`.
6. `thirdParty.onRepay(id, market, units, attacker, data)` is called. It cannot check the initiator because `caller` is not passed; checking `msg.sender == Midnight` only confirms the protocol called it, not who triggered it.
7. `safeTransferFrom(loanToken, thirdParty, address(this), units)` executes, draining `thirdParty`'s tokens.
8. Attacker's debt is reduced; `thirdParty` bears the full cost.

**Why existing checks fail:**

- The authorization check (`onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`) only validates the caller's right to act for `onBehalf`. It does not restrict which address may be designated as `callback`/`payer`.
- There is no check that `callback == msg.sender` or that `isAuthorized[callback][msg.sender]`.
- The `onRepay` interface structurally prevents the callback from self-protecting via the initiator address, unlike `onLiquidate` and `onFlashLoan`.

## Impact Explanation
Any contract implementing `IRepayCallback` with a Midnight loan token allowance is subject to direct, unconditional theft of its tokens. The attacker's debt is reduced at the victim contract's expense; the victim receives nothing in return and never consented to the payment. This constitutes direct theft of assets from protocol integrators (repayment helpers, bundlers, position managers). The ERC-20 allowance granted to Midnight is consumed without the approving contract's consent, violating the invariant that approvals bind the right user, action, and amount.

## Likelihood Explanation
Required preconditions: (1) Attacker has any debt position — trivially achievable. (2) `thirdParty` implements `IRepayCallback` and has approved Midnight — realistic for any repayment helper or bundler participating in the protocol. (3) `thirdParty.onRepay` returns `CALLBACK_SUCCESS` without out-of-band initiator tracking — structurally forced by the missing `caller` parameter; transient-storage-based self-call tracking is a non-standard mitigation that most integrators will not implement. No oracle manipulation, admin access, victim authorization, or token misbehavior is required. The attack is repeatable as long as the allowance and debt remain.

## Recommendation
Add `msg.sender` as the first parameter to `onRepay` in `IRepayCallback`, consistent with `onLiquidate` and `onFlashLoan`:

```solidity
// ICallbacks.sol
interface IRepayCallback {
    function onRepay(address caller, bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
}
```

Update the call site in `repay`:
```solidity
IRepayCallback(callback).onRepay(msg.sender, id, market, units, onBehalf, data)
```

Additionally, consider requiring `isAuthorized[callback][msg.sender]` or `callback == msg.sender` before accepting a third-party payer, analogous to how authorization is enforced for `onBehalf`.
