Audit Report

## Title
Zero oracle price in `seizedAssets` path allows free collateral seizure with zero debt repaid - (File: `src/Midnight.sol`)

## Summary
When `liquidate` is called with `seizedAssets > 0` and the oracle for `collateralIndex` returns `price = 0`, the computation at line 650 evaluates `repaidUnits` to exactly `0` due to integer arithmetic in `mulDivUp`. A liquidator receives real collateral tokens while repaying zero debt, violating the core invariant that every collateral seizure must correspond to proportional debt repayment.

## Finding Description

**Root cause — `mulDivUp` with zero numerator factor:**

`mulDivUp` at `src/libraries/UtilsLib.sol` line 34–36:
```solidity
return (x * y + (d - 1)) / d;
```
With `y = liquidatedCollatPrice = 0`, `d = ORACLE_PRICE_SCALE = 1e36`:
- `mulDivUp(seizedAssets, 0, 1e36)` = `(0 + 1e36 - 1) / 1e36` = `0`
- `mulDivUp(0, WAD, lif)` = `(0 + lif - 1) / lif` = `0`

So `repaidUnits = 0`.

**Exact code path (`src/Midnight.sol`):**

1. **Line 595**: `atMostOneNonZero(repaidUnits=0, seizedAssets=N)` → passes (one is zero).
2. **Lines 607–618**: Bitmap loop calls `IOracle(...).price()`. If the oracle for `collateralIndex` returns `0`, then `liquidatedCollatPrice = 0` (line 611). That collateral contributes `0` to `maxDebt`, making the position *more* likely to satisfy `originalDebt > maxDebt`.
3. **Lines 620–624**: Liquidatability check passes.
4. **Line 649–650**: `seizedAssets > 0` branch taken:
   ```solidity
   repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
   ```
   Evaluates to `0` as shown above.
5. **Lines 655–668**: RCF check: `0 <= maxRepaid` is trivially true; no revert.
6. **Line 670–676**: `_position.collateral[collateralIndex] -= seizedAssets` (real collateral removed); `_position.debt -= 0` (no debt removed).
7. **Line 696**: `safeTransfer(collateralToken, receiver, seizedAssets)` — real collateral sent to liquidator.
8. **Line 717**: `safeTransferFrom(loanToken, payer, address(this), 0)` — zero loan tokens pulled.

**Why the `repaid input` path reverts but `seizedAssets` path does not:**

Line 652 (the `repaid input` branch):
```solidity
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
```
`mulDivDown(..., 0)` divides by zero → revert. The `seizedAssets` path has no such division by `liquidatedCollatPrice`, so it silently produces `repaidUnits = 0`.

**Protocol's own acknowledgment of the gap:**

The LIVENESS comment at line 146 explicitly states:
> *"If the liquidated collateral oracle returns 0 on price, liquidate with repaid input reverts."*

This documents only the `repaid input` path. The `seizedAssets` path is unmentioned and unprotected.

`NoDivisionByZero.spec` line 124 explicitly excludes price=0 from its proof scope:
```
require ghostPrice(market.collateralParams[collateralIndex].oracle) > 0, "Assumption: the collateral price is not zero";
```

`Reverts.spec` rule `oracleZeroCausesLiquidateWithRepaidRevert` (line 246–253) only proves the `repaidUnits > 0` path reverts; no analogous rule exists for the `seizedAssets > 0` path.

## Impact Explanation
An unprivileged liquidator can seize any amount of a borrower's collateral at `collateralIndex` (up to the full balance) while repaying exactly 0 debt. The borrower loses collateral with no corresponding debt reduction. The lender pool loses collateral backing without any loan repayment. This is direct, unauthorized theft of user assets — a critical-severity impact.

## Likelihood Explanation
**Required preconditions:**
1. A borrower has ≥1 active collateral and is unhealthy (`originalDebt > maxDebt`) or the market is post-maturity.
2. The oracle for the targeted `collateralIndex` returns `0`.

The protocol explicitly acknowledges condition (2) as reachable in its LIVENESS section and models it with dedicated ghost axioms in `Reverts.spec`. Zero oracle prices occur via circuit-breaker oracles, stale Chainlink feeds returning 0, or custom oracle implementations. Condition (1) is made *easier* to satisfy when the oracle returns 0, since that collateral contributes 0 to `maxDebt`. Once conditions are met, any unprivileged liquidator can exploit this repeatedly until the collateral at that index is fully drained.

## Recommendation
Add an explicit guard before the `seizedAssets` branch to revert when `liquidatedCollatPrice == 0`:

```solidity
require(liquidatedCollatPrice > 0, ZeroCollateralPrice());
```

This should be placed before line 649 (the `if (seizedAssets > 0)` branch), or alternatively as a unified guard before the entire `if (repaidUnits > 0 || seizedAssets > 0)` block at line 643. This mirrors the implicit protection already present in the `repaid input` path via division-by-zero revert, making both paths consistent and safe.

## Proof of Concept
**Minimal Foundry test outline:**

1. Deploy a market with two collaterals: collateral A (normal oracle, price > 0) and collateral B (zero-price oracle, `price()` returns 0).
2. Have a borrower supply both collaterals and take debt such that the position is unhealthy (possible because collateral B contributes 0 to `maxDebt`).
3. Call `liquidate(market, collateralIndexB, seizedAssets=N, repaidUnits=0, borrower, false, attacker, address(0), "")`.
4. Assert: attacker received `N` units of collateral B token; borrower's debt is unchanged; attacker paid 0 loan tokens.

**Expected result without fix:** transaction succeeds, attacker receives free collateral.
**Expected result with fix:** transaction reverts with `ZeroCollateralPrice`.
