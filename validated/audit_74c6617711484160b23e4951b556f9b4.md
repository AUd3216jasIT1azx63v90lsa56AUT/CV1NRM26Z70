Audit Report

## Title
Lender can drain `withdrawable` below `continuousFeeCredit`, permanently DoS-ing `claimContinuousFee()` - (File: src/Midnight.sol)

## Summary
`claimContinuousFee()` unconditionally subtracts `amount` from `_marketState.withdrawable` with no guard that `withdrawable >= amount`. Because fee accrual in `_updatePosition()` increases `continuousFeeCredit` without a matching increase to `withdrawable`, and because any lender can freely reduce `withdrawable` via `withdraw()`, an unprivileged lender can drain `withdrawable` below `continuousFeeCredit` and cause every subsequent `claimContinuousFee()` call to revert with an arithmetic underflow, permanently blocking protocol fee collection.

## Finding Description

**Root cause — accounting divergence**

`withdrawable` increases only in `repay()` and `liquidate()` (when borrowers return tokens): [1](#0-0) [2](#0-1) 

`_updatePosition()` increases `continuousFeeCredit` by the accrued fee with **no corresponding increase to `withdrawable`**: [3](#0-2) 

`withdraw()` decrements `withdrawable` by the full withdrawn amount: [4](#0-3) 

`claimContinuousFee()` then blindly subtracts `amount` from `withdrawable` with no guard: [5](#0-4) 

**Exploit flow**

Preconditions: market exists, fees have accrued (`continuousFeeCredit = F > 0`), some repayments have been made so `withdrawable = W >= F`.

1. Attacker (lender with credit ≥ `W − F + 1`) calls `withdraw(market, W − F + 1, attacker, attacker)`.
   - `withdrawable` drops to `F − 1`.
   - Fully authorized — the attacker is withdrawing their own credit.
2. feeClaimer calls `claimContinuousFee(market, F, receiver)`.
   - Line 318: `continuousFeeCredit -= F` — succeeds (`F ≤ continuousFeeCredit`).
   - Line 319: `totalUnits -= F` — succeeds.
   - Line 320: `withdrawable -= F` → `(F−1) − F` → **arithmetic underflow → revert**.

The attack is repeatable: every time a borrower repays and `withdrawable` rises back above `continuousFeeCredit`, the attacker can drain it again with a single `withdraw()` call.

**Why existing checks are insufficient**

`claimContinuousFee()` only checks `msg.sender == feeClaimer` and `tickSpacing > 0`: [6](#0-5) 

There is no `require(withdrawable >= amount)`. The Certora liveness rule in `Role.spec` at line 264 explicitly lists `amount <= withdrawableBefore` as a necessary condition for non-revert — confirming the underflow is a real revert path, not an intentional guard: [7](#0-6) 

The protocol's own test confirms the gap: it explicitly repays before claiming, with the comment "Repay so withdrawable covers the claim", acknowledging that without repayment the claim would fail: [8](#0-7) 

## Impact Explanation

`claimContinuousFee()` is permanently DoS-able at negligible cost. Any lender can block the feeClaimer from collecting all accrued continuous-fee revenue for a market. The feeClaimer can only succeed after a borrower repays enough to restore `withdrawable ≥ continuousFeeCredit`, at which point the attacker can immediately drain it again. Protocol fee revenue is effectively held hostage indefinitely.

## Likelihood Explanation

- **Preconditions**: any market with a non-zero continuous fee and at least one lender who has credit to withdraw — the normal operating state of any active market.
- **Attacker cost**: zero beyond gas — the attacker withdraws their own credit and receives their own tokens back.
- **Repeatability**: unlimited; the attacker can re-execute after every repayment.
- **No special role required**: any lender (or anyone authorized by a lender) qualifies.

## Recommendation

Enforce the invariant `withdrawable >= continuousFeeCredit` at the point where it can be broken. The cleanest fix is to add a guard in `withdraw()` preventing `withdrawable` from being drained below `continuousFeeCredit`:

```solidity
require(
    _marketState.withdrawable - UtilsLib.toUint128(units) >= _marketState.continuousFeeCredit,
    WithdrawableWouldFallBelowContinuousFeeCredit()
);
```

Alternatively, maintain separate accounting for the tokens backing `continuousFeeCredit` so that fee-backing tokens are not withdrawable by lenders.

## Proof of Concept

Minimal Foundry test steps:

1. Create a market with a non-zero continuous fee.
2. Lender provides credit `C`; borrower takes the loan.
3. Borrower repays `R` units → `withdrawable = R`.
4. Time passes; call `updatePosition(market, lender)` → `continuousFeeCredit = F > 0`, lender credit = `C − F`.
5. Ensure `R >= F` (so feeClaimer could normally claim).
6. Attacker (the lender) calls `withdraw(market, R − F + 1, lender, lender)` → `withdrawable = F − 1`.
7. feeClaimer calls `claimContinuousFee(market, F, receiver)` → **reverts** with arithmetic underflow on line 320.
8. Borrower repays again; repeat from step 6 to demonstrate repeatability.
