Audit Report

## Title
`sellerAssetsToUnits` division-by-zero when `offerPrice == settlementFee` causes unconditional revert in `supplyCollateralAndSellWithAssetsTarget` - (`src/periphery/TakeAmountsLib.sol`)

## Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and then calls `mulDivUp(targetSellerAssets, WAD, sellerPrice)`. When `offerPrice == settlementFee(id, 0)` post-maturity, `sellerPrice = 0` and `mulDivUp` reverts via uint256 underflow in `d - 1`. Midnight's core `take()` handles this case correctly (computing `sellerAssets = 0`), so the periphery diverges from core. Because `sellerAssetsToUnits` is called outside the `try/catch` in `supplyCollateralAndSellWithAssetsTarget`, any `takes` array containing such an offer causes the entire transaction to revert unconditionally.

## Finding Description

**Root cause — `TakeAmountsLib.sellerAssetsToUnits`, lines 41–46:**

`sellerPrice` is computed as `offerPrice - settlementFee` for buy offers. When `offerPrice == settlementFee`, `sellerPrice = 0`. The subsequent call `targetSellerAssets.mulDivUp(WAD, 0)` executes `(x * y + (0 - 1)) / 0`, where `0 - 1` underflows in unchecked uint256 arithmetic, causing an arithmetic revert. [1](#0-0) 

This is confirmed by `testMulDivUpDivisionByZero`, which documents that `mulDivUp(x, y, 0)` reverts with `stdError.arithmeticError` due to the `d - 1` underflow: [2](#0-1) 

The `mulDivUp` implementation: [3](#0-2) 

**Why the condition is reachable:** `PRICE_ROUNDING_STEP = 1e12` means `tickToPrice(tick)` returns multiples of `1e12`. `MAX_SETTLEMENT_FEE_0_DAYS = 14e12` and `CBP = 1e12`, so `settlementFee(id, 0)` takes values in `{0, 1e12, 2e12, ..., 14e12}`. For any non-zero `settlementFeeCbp0`, a tick exists where `tickToPrice(tick) == settlementFee(id, 0)`. [4](#0-3) [5](#0-4) 

**Why Midnight's `take()` succeeds:** In `Midnight.take()`, `sellerAssets = units.mulDivDown(sellerPrice, WAD) = units.mulDivDown(0, WAD) = 0`, which is valid. The `CannotIncreaseDebtPostMaturity` check passes when `sellerDebtIncrease == 0` (seller has sufficient credit). [6](#0-5) [7](#0-6) 

**Why the periphery reverts:** `sellerAssetsToUnits` is called at line 286 **outside** the `try/catch` block that wraps `take()`. The NatSpec at line 175 explicitly acknowledges this: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." The `try/catch` only covers the `IMidnight(MIDNIGHT).take(...)` call. [8](#0-7) [9](#0-8) 

**NatSpec divergence:** `sellerAssetsToUnits` documents "Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)." The `offerPrice == settlementFee` case is not covered by this comment, and Midnight does NOT revert in this case — only the periphery does. [10](#0-9) 

**Additional surface:** `ConsumableUnitsLib.consumableUnits` also calls `sellerAssetsToUnits` (line 21) when `offer.buy = true` and `offer.maxAssets > 0`, and this call is also outside the `try/catch` (line 290 of `MidnightBundles.sol`). [11](#0-10) 

**Exploit flow:**
1. Market passes maturity (`block.timestamp > offer.market.maturity`).
2. Attacker creates a buy offer at a tick where `tickToPrice(tick) == settlementFee(id, 0)` (e.g., `settlementFeeCbp0 = 1` → `settlementFee = 1e12`, tick with `tickToPrice = 1e12`).
3. Attacker broadcasts the offer off-chain. Off-chain routers include it in the `takes` array.
4. Victim calls `supplyCollateralAndSellWithAssetsTarget` with this offer in `takes`.
5. `sellerAssetsToUnits` is called with `targetSellerAssets > 0` and `sellerPrice = 0` → arithmetic revert.
6. Entire transaction reverts. The offer remains valid and the condition persists as long as the market is post-maturity with the same fee.

**Existing protections are insufficient:** `buyerAssetsToUnits` has a `require(buyerPrice <= WAD)` guard that incidentally catches `offerPrice < settlementFee` via underflow, but `sellerAssetsToUnits` has no analogous guard for `sellerPrice == 0`. [12](#0-11) 

## Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee(id, 0)` post-maturity will revert unconditionally, regardless of `targetSellerAssets`. This is a permanent DoS of the sell-via-periphery path for post-maturity markets whenever such an offer appears in the routing set. The attacker's cost is only the gas to create the offer; no funds are at risk for the attacker.

## Likelihood Explanation
**Preconditions:**
- Market is post-maturity (inevitable for any market).
- `settlementFeeCbp0 >= 1` (any non-zero post-maturity fee, the common case).
- A tick exists where `tickToPrice(tick) == settlementFee(id, 0)` — achievable given `PRICE_ROUNDING_STEP = 1e12` and max fee of `14e12`.
- The attacker's offer is included in the victim's `takes` array via off-chain routing.

The condition is repeatable: the offer can be recreated after expiry, and the revert is deterministic. Any routing system that automatically aggregates available buy offers without filtering for `sellerPrice > 0` is vulnerable.

## Recommendation
Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case. When `sellerPrice == 0`, the forward direction yields `sellerAssets = 0` for any number of units, so the inverse should return `type(uint256).max` (or 0 if `targetSellerAssets == 0`) to signal that no finite unit count achieves a positive `targetSellerAssets`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
if (offer.buy && sellerPrice == 0) {
    return targetSellerAssets == 0 ? 0 : type(uint256).max;
}
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

Alternatively, move the `sellerAssetsToUnits` call inside the `try/catch` block, or add a pre-check in `supplyCollateralAndSellWithAssetsTarget` to skip offers where `sellerPrice == 0`.

## Proof of Concept
Minimal Foundry fork test:
1. Deploy a market with `settlementFeeCbp0 = 1` (fee = `1e12`).
2. Warp past maturity.
3. Find tick `t` such that `TickLib.tickToPrice(t) == 1e12` (iterate from tick 0 upward).
4. Create a buy offer at tick `t` with `offer.buy = true`, valid expiry, and any `maxUnits > 0`.
5. Call `supplyCollateralAndSellWithAssetsTarget` with `targetSellerAssets = 1` and `takes = [Take({offer: poisonOffer, units: 1, ratifierData: ""})]`.
6. Assert the call reverts with `stdError.arithmeticError`.
