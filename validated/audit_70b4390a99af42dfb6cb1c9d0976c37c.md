### Title
`isHealthy` Misses Maturity-Based Liquidatability, Allowing Post-Maturity Collateral Drain — (File: `src/Midnight.sol`)

---

### Summary

`isHealthy()` evaluates only the collateral-ratio condition (`maxDebt >= debt`) and never checks whether the market's maturity has passed. The protocol's own NatSpec states that accounts are liquidatable if they are **either** unhealthy **or** maturity has passed. Because `withdrawCollateral()` relies solely on `isHealthy()` as its guard, a borrower whose market has expired can withdraw collateral freely — even though they are already liquidatable in post-maturity mode — leaving liquidators with nothing to seize and forcing bad debt onto lenders.

---

### Finding Description

The protocol defines two independent liquidatability conditions (lines 58–59):

> "Accounts are liquidatable only if they are either **unhealthy** or the **maturity has passed**."

`liquidate()` correctly encodes both conditions: [1](#0-0) 

```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```

But `isHealthy()` only checks the collateral-ratio arm: [2](#0-1) 

```solidity
function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
    ...
    return maxDebt >= debt;   // ← no maturity check
}
```

`withdrawCollateral()` uses `isHealthy()` as its sole guard: [3](#0-2) 

```solidity
require(isHealthy(market, id, onBehalf), UnhealthyBorrower());
```

After `block.timestamp > market.maturity`, a borrower with outstanding debt is liquidatable in post-maturity mode. Yet `isHealthy()` returns `true` for any position where `maxDebt >= debt`, regardless of maturity. The borrower can therefore call `withdrawCollateral()` and remove collateral that liquidators are entitled to seize.

---

### Impact Explanation

A borrower can drain their collateral immediately after maturity, before any liquidator can act. With collateral removed:

- Liquidators calling `liquidate()` with `postMaturityMode = true` receive little or no collateral.
- The remaining debt becomes bad debt, socialized among all lenders in the market via the `lossFactor` mechanism.
- Lenders suffer a direct, unrecoverable loss of principal proportional to the bad debt realized.

This is a **direct theft of lender funds** enabled by a missing condition in a guard function.

---

### Likelihood Explanation

- **No privileged access required**: any borrower can call `withdrawCollateral()` permissionlessly.
- **Timing is predictable**: market maturity is a fixed, public timestamp. A borrower can prepare and submit the withdrawal in the same block maturity is reached, front-running liquidators.
- **Incentive is strong**: the borrower eliminates their repayment obligation while keeping the borrowed assets.
- **Affects every fixed-maturity market** in the protocol where a borrower's collateral ratio remains healthy at maturity.

---

### Recommendation

Add a maturity check to `isHealthy()`, or add a separate guard in `withdrawCollateral()` that rejects withdrawals when `block.timestamp > market.maturity` and the caller has outstanding debt:

```solidity
// In withdrawCollateral, after the collateral update:
require(
    block.timestamp <= market.maturity || _position.debt == 0,
    PostMaturityWithdrawalForbidden()
);
require(isHealthy(market, id, onBehalf), UnhealthyBorrower());
```

Alternatively, update `isHealthy()` itself to return `false` when `block.timestamp > market.maturity && debt > 0`, making it consistent with the protocol's stated liquidatability definition and ensuring all callers inherit the correct semantics.

---

### Proof of Concept

**Setup:**
- Market with `maturity = T`, single collateral, `lltv = 0.8`, `maxLif = 1.1`.
- Borrower supplies 100 units of collateral (oracle price = 1), borrows 80 units of loan token.
- At `T - ε`, position is healthy: `maxDebt = 80 = debt`.

**Exploit (at `block.timestamp = T + 1`):**

1. Borrower calls `withdrawCollateral(market, 0, 50, borrower, receiver)`.
2. `isHealthy()` computes `maxDebt = 50 * 1 * 0.8 = 40`, `debt = 80` → returns `false`. Wait — in this example the position becomes unhealthy. Let me adjust: use `lltv = 0.95`, borrow 80, collateral = 100. `maxDebt = 95`. Withdraw 15 collateral → `maxDebt = 85 * 0.95 = 80.75 >= 80` → `isHealthy()` returns `true`. Withdrawal succeeds.
3. Liquidator calls `liquidate(market, 0, 0, 80, borrower, true, ...)` (post-maturity mode, repay 80 units).
4. `seizedAssets = 80 * 1.0 * (ORACLE_PRICE_SCALE / liquidatedCollatPrice) = 80` but only 85 collateral remains — liquidator gets 80 collateral for 80 debt repaid (LIF = 1 at maturity). The borrower has already extracted 15 collateral units for free.
5. Scale this to a borrower who withdraws the maximum collateral that keeps `isHealthy()` returning `true` — they extract the maximum possible collateral before liquidation, forcing residual bad debt onto lenders. [4](#0-3) [5](#0-4) [2](#0-1) [1](#0-0)

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];
                uint256 price = IOracle(collateralParam.oracle).price();
                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                    .mulDivDown(collateralParam.lltv, WAD);
                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
    }
