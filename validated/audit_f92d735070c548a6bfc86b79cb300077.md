Audit Report

## Title
Missing oracle address validation in `touchMarket` enables permanent position freeze - (File: `src/Midnight.sol`)

## Summary
`touchMarket` is permissionless and validates collateral token addresses, LLTV tiers, and `maxLif` values, but performs no check that `collateralParams[i].oracle != address(0)`. Any unprivileged caller can create a market with a zero oracle address. Any borrower who subsequently supplies collateral to that index and acquires debt will have `isHealthy`, `withdrawCollateral`, and `liquidate` permanently revert, freezing their collateral and debt with no recovery path.

## Finding Description
**Root cause:** In `src/Midnight.sol`, `touchMarket` (lines 762–773) iterates over `collateralParams` and enforces `collateralToken > previousCollateralToken`, `isLltvAllowed(lltv)`, and `maxLif` validity. There is no `require(market.collateralParams[i].oracle != address(0))` check.

**Exploit flow:**
1. Attacker calls `touchMarket` with `collateralParams[0].token = <valid token>`, `collateralParams[0].oracle = address(0)`, valid LLTV and `maxLif`. The call succeeds; the market is stored.
2. Victim calls `supplyCollateral(market, 0, amount, victim)`. No oracle is consulted here; the collateral bit is set in `_position.collateralBitmap` (line 536–537). The call succeeds.
3. Victim borrows via `take`. Because `_position.debt == 0` before the first borrow, `isHealthy` short-circuits at line 948 (`if (debt > 0)`) and never calls the oracle. The borrow succeeds.
4. Victim now has `debt > 0`. Any subsequent call that reaches `isHealthy` enters the bitmap loop (line 950), reads `collateralParam.oracle == address(0)`, and executes `IOracle(address(0)).price()` (line 953). Since `address(0)` has no code, the low-level call returns empty bytes; Solidity 0.8.x ABI-decodes the expected `uint256` from zero bytes and reverts.
5. `withdrawCollateral` (line 568: `require(isHealthy(...))`) reverts permanently. `liquidate` (line 610: same oracle call in its own bitmap loop) reverts permanently. The direct `isHealthy` view reverts permanently.

**Why existing checks fail:** The sorted-token check (`collateralToken > previousCollateralToken`) only prevents `token == address(0)` and duplicates; it says nothing about the `oracle` field. No other check in `touchMarket` or anywhere else in the creation path validates the oracle address. The `supplyCollateral` comment ("checks authorization to prevent activated collateral poisoning") addresses only unauthorized activation by third parties, not oracle-field poisoning at market creation time.

## Impact Explanation
Any borrower who activates the poisoned collateral index and holds debt has their position permanently frozen: they cannot withdraw collateral, cannot be liquidated (their debt cannot be recovered by the protocol), and `isHealthy` always reverts for them. Their collateral is locked in the contract with no recovery path, and their debt accrues indefinitely. This constitutes a permanent, irrecoverable freeze of user funds and violates the core protocol invariant that unhealthy positions remain liquidatable.

## Likelihood Explanation
The precondition is trivially achievable: `touchMarket` is permissionless, requires no tokens or privileges, and the only constraint on the oracle field is that it be ABI-encodable as an `address`. A single transaction creates the trap. Victims are any borrowers who interact with the market without inspecting the raw `collateralParams[i].oracle` field off-chain — a realistic scenario given that market parameters are stored via `IdLib.storeInCode` and not surfaced as a prominent on-chain warning. The freeze is permanent and irreversible once a borrower has debt with the activated index.

## Recommendation
Add an explicit non-zero oracle check inside the `touchMarket` collateral-params loop, immediately after the token check:

```solidity
// in touchMarket, inside the for loop (after line 764):
require(market.collateralParams[i].oracle != address(0), ZeroOracle());
```

Optionally, also validate that the oracle address contains code (e.g., using `address.code.length > 0`) to reject EOA addresses that would produce the same ABI-decode revert.

## Proof of Concept
**Minimal Foundry test outline:**
```solidity
function test_zeroOracleFreezesBorrower() public {
    // 1. Build a Market with collateralParams[0].oracle = address(0)
    Market memory m = _buildMarket(address(0) /* oracle */);

    // 2. touchMarket succeeds — no revert
    bytes32 id = midnight.touchMarket(m);

    // 3. Victim supplies collateral — succeeds
    vm.prank(victim);
    midnight.supplyCollateral(m, 0, 1e18, victim);

    // 4. Victim borrows (debt == 0 before, oracle not called) — succeeds
    _victimBorrow(m, id, victim, 1e17);

    // 5. withdrawCollateral reverts permanently
    vm.prank(victim);
    vm.expectRevert();
    midnight.withdrawCollateral(m, 0, 1, victim, victim);

    // 6. liquidate reverts permanently
    vm.expectRevert();
    midnight.liquidate(m, 0, 0, 1, victim, false, address(this), address(0), "");
}
```
Expected: steps 2–4 succeed; steps 5–6 revert unconditionally, demonstrating permanent freeze.
