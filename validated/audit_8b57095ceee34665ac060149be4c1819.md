Audit Report

## Title
Authorized-Party Nonce Burn via Delegated Signature Consumes Authorizer's Nonce — (`src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts a valid signature from any party already authorized by `authorization.authorizer` in Midnight, not only from the authorizer themselves. Because `nonce[authorization.authorizer]` is incremented on every successful call regardless of who signed, an authorized party can craft and submit their own EIP-712 signature over an `Authorization` struct naming the victim as `authorizer`, consuming the victim's current nonce and permanently invalidating any pending off-chain authorization the victim has already distributed with that nonce.

## Finding Description
**Code path** — `src/periphery/EcrecoverAuthorizer.sol` lines 24–48:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // nonce consumed here

    ...
    address signer = ecrecover(digest, signature.v, signature.r, signature.s);
    require(signer != address(0), InvalidSignature());
    require(
        signer == authorization.authorizer
            || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer), // delegation branch
        Unauthorized()
    );
    ...
    IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
}
```

**Root cause** — The nonce at line 26 belongs to `authorization.authorizer` (victim A), but the `Unauthorized` guard at lines 33–36 has a second branch that accepts any signer for whom `isAuthorized[A][signer] == true`. There is no requirement that the signer is the same party who generated the `Authorization` struct, nor that the signer is acting on a struct that A produced. An authorized party (ATTACKER) can independently construct and sign an `Authorization{authorizer=A, authorized=ATTACKER, isAuthorized=true, nonce=N}` with their own private key, and the call will succeed because `isAuthorized[A][ATTACKER] == true`.

**Exploit flow:**

| Step | Action |
|------|--------|
| 0 | A calls `midnight.setIsAuthorized(ecrecoverAuthorizer, true, A)` (standard EcrecoverAuthorizer setup) and `midnight.setIsAuthorized(ATTACKER, true, A)` (e.g., authorizing a keeper or market maker). |
| 1 | A signs off-chain: `Authorization{authorizer=A, authorized=LEGITIMATE_OPERATOR, isAuthorized=true, nonce=N}` and hands the signature to `LEGITIMATE_OPERATOR`. |
| 2 | ATTACKER independently signs: `Authorization{authorizer=A, authorized=ATTACKER, isAuthorized=true, nonce=N}` with ATTACKER's own private key. |
| 3 | ATTACKER calls `ecrecoverAuthorizer.setIsAuthorized(authAttacker, sigAttacker)`. |
| 4 | Line 26: `nonce[A]` advances N → N+1. Lines 33–36: `isAuthorized[A][ATTACKER] == true` → passes. Call succeeds. |
| 5 | `LEGITIMATE_OPERATOR` submits A's signed authorization → reverts `InvalidNonce` because `nonce[A]` is now N+1. |

**Why existing checks fail** — The `Unauthorized` check is satisfied by the delegation branch; it does not distinguish between "the authorizer is signing their own nonce" and "a delegate is consuming the authorizer's nonce on a struct the delegate constructed themselves." There is no guard preventing an authorized party from targeting `authorization.authorizer = A` with a nonce they did not generate.

## Impact Explanation
A's off-chain signed authorization with nonce=N is permanently invalidated. `LEGITIMATE_OPERATOR` cannot submit it; A must re-sign with nonce=N+1 and redistribute. ATTACKER can repeat this indefinitely (each time submitting a no-op self-authorization `isAuthorized=true` for themselves), making it impossible for A to successfully delegate to any third party through `EcrecoverAuthorizer` as long as ATTACKER remains authorized. A's only escape is to revoke ATTACKER's authorization via a direct on-chain call to `midnight.setIsAuthorized`, but ATTACKER can front-run that revocation with another nonce burn. This constitutes a persistent, repeatable griefing/DoS on A's authorization mechanism with zero cost beyond gas.

## Likelihood Explanation
**Preconditions:**
1. A has authorized `ecrecoverAuthorizer` in Midnight — required for any EcrecoverAuthorizer user.
2. A has authorized ATTACKER in Midnight at any prior point (e.g., a market maker, keeper, or liquidation bot).
3. A has a pending off-chain signed authorization that ATTACKER knows about (e.g., via mempool observation).

All three are realistic for any active protocol user. The Midnight authorization model is explicitly broad — `Midnight.sol` lines 101–110 document that "authorized accounts can authorize other accounts on behalf of the user," meaning users who authorize counterparties for position management do not expect those counterparties to also gain nonce-burning capability in `EcrecoverAuthorizer`. The attack is repeatable with zero cost beyond gas.

## Recommendation
Restrict the `Unauthorized` check in `EcrecoverAuthorizer.setIsAuthorized` so that only `authorization.authorizer` themselves can sign an authorization that consumes their own nonce. Remove the delegation branch entirely, or — if delegation is intentional — require that the signer is the authorizer and separately handle delegation at the call-submission layer. Concretely:

```solidity
require(signer == authorization.authorizer, Unauthorized());
```

If delegation is desired, it should be implemented in a way that does not allow a delegate to construct and sign their own `Authorization` struct targeting the authorizer's nonce.

## Proof of Concept
**Minimal Foundry test outline:**
1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. As A: call `midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, A)` and `midnight.setIsAuthorized(ATTACKER, true, A)`.
3. As A: sign off-chain `Authorization{authorizer=A, authorized=LEGITIMATE_OPERATOR, isAuthorized=true, nonce=0, deadline=...}` → `sigA`.
4. As ATTACKER: sign `Authorization{authorizer=A, authorized=ATTACKER, isAuthorized=true, nonce=0, deadline=...}` → `sigATTACKER`.
5. As ATTACKER: call `ecrecoverAuthorizer.setIsAuthorized(authAttacker, sigATTACKER)` → succeeds; assert `ecrecoverAuthorizer.nonce(A) == 1`.
6. As LEGITIMATE_OPERATOR: call `ecrecoverAuthorizer.setIsAuthorized(authA, sigA)` → reverts with `InvalidNonce`.
