### Title
Sponsor can redirect CORRUPTED sweep to steal all staker principal by updating `recoveryAddress` after outcome is flagged — (File: src/ConfidencePool.sol)

### Summary
`setRecoveryAddress()` carries no temporal guard. After the moderator flags bad-faith CORRUPTED, the sponsor can atomically change `recoveryAddress` to their own wallet and call `claimCorrupted()`, draining the entire pool — all staker principal and bonus — to themselves.

### Finding Description
`setRecoveryAddress()` is gated only by `onlyOwner` with no check on `outcome`, `claimsStarted`, or any other lifecycle state:

```solidity
// src/ConfidencePool.sol L611-L618
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    address oldRecoveryAddress = recoveryAddress;
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
```

`claimCorrupted()` is permissionless and reads `recoveryAddress` at call time:

```solidity
// src/ConfidencePool.sol L408-L425
function claimCorrupted() external nonReentrant {
    if (outcome != PoolStates.Outcome.CORRUPTED) revert OutcomeNotSet();
    if (goodFaith && bountyClaimed < bountyEntitlement) revert MustClaimBountyFirst();
    uint256 toSweep = stakeToken.balanceOf(address(this));
    ...
    stakeToken.safeTransfer(recoveryAddress, toSweep);   // ← live read
}
```

Once the moderator calls `flagOutcome(CORRUPTED, false, address(0))`, the entire pool balance is destined for `recoveryAddress`. Because `setRecoveryAddress()` has no post-outcome lock, the sponsor can change the destination between the flag and the sweep. Since `claimCorrupted()` is itself permissionless, the sponsor can execute both steps atomically in a single transaction (or in two back-to-back transactions in the same block), leaving no window for stakers to react.

The same attack applies to the auto-CORRUPTED path in `claimExpired()`: after the backstop fires and sets `outcome = CORRUPTED`, the sponsor can immediately call `setRecoveryAddress(sponsorWallet)` and then `claimCorrupted()`.

DESIGN.md §10 explicitly documents a temporal latch for `expiry` ("sponsor-mutable only **until the first stake**") but provides no equivalent latch for `recoveryAddress`. The document describes `recoveryAddress` only as the "CORRUPTED sweep destination" without stating it is mutable post-outcome. This omission is not listed in the Accepted-Behavior Filter and is not acknowledged anywhere in the design document as an intentional trade-off.

### Impact Explanation
Complete loss of all staker principal and bonus. In a bad-faith CORRUPTED resolution the entire `stakeToken.balanceOf(address(this))` — which equals `snapshotTotalStaked + snapshotTotalBonus` — is transferred to the sponsor-controlled address. Every staker loses 100 % of their deposit with no recourse.

### Likelihood Explanation
The precondition is that the moderator flags bad-faith CORRUPTED, which is the honest moderator's correct action when an in-scope breach occurs. Under honest configuration this is a routine, expected event. Once it fires, the sponsor needs only two owner-gated calls (`setRecoveryAddress` + `claimCorrupted`) that can be bundled in one transaction. No special timing, no oracle manipulation, and no cooperation from any other party is required. The sponsor is explicitly a non-trusted actor in the stated threat model.

### Recommendation
Lock `recoveryAddress` updates once the outcome is no longer `UNRESOLVED`, mirroring the `expiryLocked` pattern used for `expiry`:

```solidity
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (outcome != PoolStates.Outcome.UNRESOLVED) revert OutcomeAlreadySet();
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    address oldRecoveryAddress = recoveryAddress;
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
```

Alternatively, snapshot `recoveryAddress` into an immutable local variable inside `flagOutcome()` and use that snapshot in all sweep functions, so post-flag changes have no effect on in-flight resolutions.

### Proof of Concept
1. Sponsor deploys pool with `recoveryAddress = legitimateAddress`.
2. Stakers deposit; `totalEligibleStake = S`, `totalBonus = B`.
3. Agreement is breached; moderator calls `flagOutcome(CORRUPTED, false, address(0))` (bad-faith). `snapshotTotalStaked = S`, `snapshotTotalBonus = B`, `corruptedReserve = S + B`.
4. In the same block, sponsor calls `setRecoveryAddress(sponsorWallet)` — no revert; `recoveryAddress` is now `sponsorWallet`.
5. Sponsor calls `claimCorrupted()` — `toSweep = stakeToken.balanceOf(address(this)) = S + B`; `stakeToken.safeTransfer(sponsorWallet, S + B)` executes.
6. All stakers receive nothing; sponsor has drained `S + B` tokens. [1](#0-0) [2](#0-1) [3](#0-2)

## 10. Sponsor trust surface

- **`recoveryAddress`** — CORRUPTED sweep destination. Receives the full pool (including stakers'

  move `expiry` during an all-stakers-exited moment and harm the next cohort.

The `recoveryAddress` value is informational only on the BattleChain agreement side (the
agreement's `Chain.assetRecoveryAddress` is a CAIP-2 string), so the pool stores its own typed
`address recoveryAddress` for token transfers.
