Audit Report

## Title
Fee-on-Transfer Loan Token Causes Permanent `claimableSettlementFee` Overstatement in `take` - (File: src/Midnight.sol)

## Summary
In `Midnight.take`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` at line 418 before the `safeTransferFrom` at line 455. When the loan token charges a transfer fee, the contract receives only `(buyerAssets - sellerAssets) * (1 - f)` but records the full nominal amount. This creates a permanent per-take shortfall that accumulates linearly, violating the solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee` and eventually causing lender withdrawals to fail.

## Finding Description
**Root cause:** `src/Midnight.sol:418` increments `claimableSettlementFee` by the full nominal fee before the token transfer. `src/Midnight.sol:455` executes `SafeTransferLib.safeTransferFrom` without verifying the amount actually received. With a fee-on-transfer token at rate `f`, the contract receives `(buyerAssets - sellerAssets) * (1 - f)` but records `buyerAssets - sellerAssets`, leaving a permanent shortfall of `(buyerAssets - sellerAssets) * f` per take.

**Why existing checks fail:** `certora/specs/Solvency.spec:31` explicitly states: *"no fee taking from sender or receiver, no rebasing."* The `CVL_transferFrom` summary transfers the exact requested amount, so `pendingFeeReceipt` always reaches zero after the modeled transfer (`Solvency.spec:55–56`). With a real fee-on-transfer token, the transfer only partially closes the `pendingFeeReceipt` gap — the residual is never captured by the formal model. The `tokenBalanceCorrect` strong invariant and `pendingFeeReceiptZero` weak invariant are therefore only proven under the standard-ERC20 assumption. Additionally, `multicall` is summarized as `HAVOC_ALL DELETE` (`Solvency.spec:4`), so no multicall path is formally verified.

**Exploit path:** A taker calls `take` on a market whose `loanToken` is fee-on-transfer. Each call increments `claimableSettlementFee` by the full nominal spread while the contract receives a fee-discounted amount. After N takes, `claimableSettlementFee` exceeds actual held tokens by `N * (buyerAssets - sellerAssets) * f`. When the fee claimer calls `claimSettlementFee` for the full recorded amount, the transfer either reverts (insufficient balance) or drains tokens backing lender `withdrawable` credits, causing lender withdrawals to fail.

## Impact Explanation
The solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee` is violated. The shortfall is permanent and non-reversible. Lender withdrawals fail when the drained balance can no longer cover `withdrawable` credits. The insolvency magnitude scales linearly with the number of takes and the fee rate, making it unbounded over time. This constitutes a direct loss of funds for lenders.

## Likelihood Explanation
Preconditions: the loan token must be fee-on-transfer; the taker needs sufficient balance and approval. No governance access, oracle manipulation, or privileged keys are required. Any unprivileged taker can trigger this in a single transaction on any market using such a token. The attack is repeatable across multiple transactions, accumulating unbounded shortfall. Fee-on-transfer tokens are a deployed token class. `SECURITY.md` contains no exclusion for this token type.

## Recommendation
Add a balance check around the inbound transfer at `src/Midnight.sol:455`. Record the contract's token balance before and after the `safeTransferFrom`, and use the actual received amount (post-balance minus pre-balance) to increment `claimableSettlementFee` rather than the nominal `buyerAssets - sellerAssets`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are unsupported as loan tokens, either via an allowlist or a market creation guard.

## Proof of Concept
1. Deploy a mock ERC20 with a 1% transfer fee.
2. Create a Midnight market with this token as `loanToken`.
3. Call `take` repeatedly; after each call, assert `claimableSettlementFee[token] > token.balanceOf(midnight)`.
4. Call `claimSettlementFee` for the full recorded amount; observe revert or observe that subsequent lender `withdraw` calls revert due to insufficient balance.
