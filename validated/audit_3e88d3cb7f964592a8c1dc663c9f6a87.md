Audit Report

## Title
Unconstrained `takerCallback` with fallback returning `CALLBACK_SUCCESS` enables unauthorized token drain from third-party contracts in `take()` - (File: src/Midnight.sol)

## Summary
`take()` allows the taker to freely nominate any address as `takerCallback`, which becomes both `buyerCallback` and `payer` for sell offers. The protocol's sole consent check — that `IBuyCallback(takerCallback).onBuy(...)` returns `CALLBACK_SUCCESS` — is satisfiable by any contract whose `fallback()` returns `keccak256("morpho.midnight.callbackSuccess")`. An attacker can therefore designate such a contract as `takerCallback` and drain its entire approved `loanToken` balance without its explicit consent.

## Finding Description
**Code path** (`src/Midnight.sol`):

```solidity
// Line 346 — taker is checked, takerCallback is not
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());

// Line 420 — sell offer: buyerCallback = attacker-supplied takerCallback
address buyerCallback = offer.buy ? offer.callback : takerCallback;

// Line 422 — payer = buyerCallback = takerCallback = victimContract
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);

// Lines 447–452 — sole guard: return-value check only
require(
    IBuyCallback(buyerCallback)
        .onBuy(id, offer.market, buyerAssets, units, buyerPendingFeeIncrease, buyer, buyerCallbackData)
    == CALLBACK_SUCCESS,
    WrongBuyCallbackReturnValue()
);

// Lines 455–456 — tokens pulled from payer = victimContract
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

**Root cause**: `take()` enforces taker identity (`taker == msg.sender || isAuthorized[taker][msg.sender]`) but imposes no constraint that `takerCallback` must be the taker, authorized by the taker, or a contract that deliberately opted in as payer. The only enforcement is the `CALLBACK_SUCCESS` return-value check. Because Solidity dispatches unrecognized function selectors to `fallback()`, any contract whose `fallback()` returns `keccak256("morpho.midnight.callbackSuccess")` satisfies the check and becomes the payer.

**Exploit flow**:
1. Attacker calls `take(sellOffer, ratifierData, units, attacker, receiver, victimContract, "")`.
2. `buyerCallback = victimContract`, `payer = victimContract`.
3. `IBuyCallback(victimContract).onBuy(...)` dispatches to `victimContract`'s `fallback()`.
4. `fallback()` returns `CALLBACK_SUCCESS`; `require` passes.
5. `safeTransferFrom(loanToken, victimContract, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, victimContract, receiver, sellerAssets)` execute, draining `victimContract`'s approved balance.

**Why existing checks fail**:
- `taker == msg.sender || isAuthorized[taker][msg.sender]` — checks taker identity, not `takerCallback`.
- `isAuthorized[offer.maker][offer.ratifier]` + `IRatifier.isRatified` — validates the maker's offer signature; irrelevant to `takerCallback`.
- `require(... == CALLBACK_SUCCESS)` — the only guard, satisfied by a fallback returning the constant.
- Certora spec `takeOnlyExplicitPayer` (`OnlyExplicitPayerCanLoseTokens.spec` lines 95–115) — proves tokens are only pulled from the callback that returned `CALLBACK_SUCCESS`, but does not constrain who the taker may nominate as `takerCallback`. The spec's invariant is technically upheld even in the attack scenario.

## Impact Explanation
Direct, complete theft of `loanToken` from `victimContract` up to its full approved balance and the offer's `maxUnits`/`maxAssets` cap. Tokens flow to `address(this)` (Midnight, as settlement fee) and to the attacker-controlled `receiver`. The attack is permissionless, repeatable, and requires no privileged access. This is a direct theft of user/contract funds — the highest-severity smart contract impact class.

## Likelihood Explanation
**Required preconditions**:
1. `victimContract` has a `fallback()` returning `keccak256("morpho.midnight.callbackSuccess")` — not accidental; requires deliberate use of this specific 32-byte constant. Realistic for universal adapter contracts, proxy pass-throughs, or "accept-all" callback wrappers that return a fixed magic value.
2. `victimContract` has approved `loanToken` to Midnight — routine for any contract that interacts with the protocol (lending aggregators, routers, vault wrappers).
3. A valid sell offer exists (EcrecoverRatifier-signed) — freely obtainable from any maker who has published a signed offer.

Likelihood is **low**: condition 1 is non-trivial but not impossible. Once both conditions hold, the attack is fully permissionless and repeatable.

## Recommendation
Add an authorization check on `takerCallback` mirroring the existing `taker` check. The simplest fix is to require that `takerCallback` is either the taker or explicitly authorized by the taker:

```solidity
require(
    takerCallback == address(0)
        || takerCallback == taker
        || isAuthorized[taker][takerCallback],
    TakerCallbackUnauthorized()
);
```

This ensures that only the taker (or a contract the taker has explicitly authorized) can be nominated as payer, closing the payer-confusion path regardless of what the callback returns.
