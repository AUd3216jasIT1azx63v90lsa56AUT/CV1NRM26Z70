Audit Report

## Title
Front-running partial liquidation reduces `maxRepaid` below victim's `repaidUnits`, causing `RecoveryCloseFactorConditionsViolated()` revert - (File: src/Midnight.sol)

## Summary

In normal (pre-maturity) liquidation mode, `maxRepaid` is recomputed from live on-chain state (`_position.debt` and `maxDebt`) on every call. An unprivileged attacker can front-run any competing liquidator with a profitable partial liquidation of `A` units, which reduces `maxRepaid` by exactly `A` while leaving the `rcfThreshold` small-collateral exception value unchanged. This causes the victim's transaction to revert with `RecoveryCloseFactorConditionsViolated()` whenever `victimRepaidUnits > originalMaxRepaid - A`, and the attack is self-funding and indefinitely repeatable.

## Finding Description

**Root cause:** `maxRepaid` at lines 659–661 is computed from `_position.debt` (mutable storage) and `maxDebt` (recomputed from `_position.collateral[i]` in the loop at lines 607–618). Both values change after every partial liquidation (lines 670, 676), so `maxRepaid` is not stable across transactions.

**State transitions after attacker liquidates `A` units (lines 652, 670, 676):**
- `_position.debt` → `debt - A`
- `seizedAssets = A * lif * ORACLE_PRICE_SCALE / (WAD * price)` (line 652)
- `_position.collateral[collateralIndex]` → `collateral - seizedAssets` (line 670)
- `maxDebt` (recomputed next call, line 613) → `maxDebt - A * lif * lltv / WAD²`

Therefore:
```
newDebt - newMaxDebt = (debt - maxDebt) - A*(WAD² - lif*lltv)/WAD²
newMaxRepaid         = originalMaxRepaid - A
```

**`rcfThreshold` exception (lines 664–665) is invariant under partial liquidations:**
```
newCollateral*price/lif - newMaxRepaid
= (collateral - A*lif/price)*price/lif - (originalMaxRepaid - A)
= collateral*price/lif - originalMaxRepaid   ← identical to pre-attack value
```
If the exception did not apply before the attack, it still does not apply after.

**Exploit flow:**
1. Borrower is unhealthy (`debt > maxDebt`), no bad debt (checked at line 626), RCF active, `market.liquidatorGate == address(0)` (line 598) or attacker passes the gate.
2. `originalMaxRepaid = (_position.debt - maxDebt).mulDivUp(WAD*WAD, WAD*WAD - lif*lltv)` (line 659–660).
3. Victim broadcasts tx with `victimRepaidUnits` where `1 < victimRepaidUnits ≤ originalMaxRepaid`.
4. Attacker front-runs with `attackerRepaidUnits` satisfying:
   - `attackerRepaidUnits < originalMaxRepaid` → attacker passes RCF check (line 662)
   - `attackerRepaidUnits > originalMaxRepaid - victimRepaidUnits` → `newMaxRepaid < victimRepaidUnits`
5. Victim's tx executes: `newMaxRepaid = originalMaxRepaid - attackerRepaidUnits < victimRepaidUnits`; rcfThreshold exception still does not apply → revert at line 666 with `RecoveryCloseFactorConditionsViolated()`.

**Existing guards are insufficient:**
- `liquidationLocked` (line 621) uses transient storage (`LIQUIDATION_LOCK_SLOT`, ConstantsLib.sol line 24) and only blocks liquidation during `take` callbacks within the same transaction (comment at line 60). It provides no cross-transaction protection.
- The bad debt branch (lines 626–641) is a precondition exclusion, not a guard against this attack.
- The `liquidatorGate` (lines 597–600) is optional (`address(0)` = unrestricted) and does not prevent a gated attacker from also passing the gate. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

## Impact Explanation

An unprivileged attacker can permanently prevent any competing liquidator from executing a full or large partial liquidation of any unhealthy position by repeatedly front-running with small profitable partial liquidations. Each front-run is economically self-funding (attacker receives collateral at the `lif > 1` discount). The victim's transaction always reverts with `RecoveryCloseFactorConditionsViolated()`. The position remains unhealthy and continues to accumulate protocol risk, while the attacker monopolizes all liquidation profit. This constitutes both a denial-of-service on liquidators and theft of liquidation opportunity. [6](#0-5) [7](#0-6) 

## Likelihood Explanation

Preconditions are the standard conditions for any contested liquidation on a chain with a public mempool: borrower is unhealthy, RCF is active, victim submits `repaidUnits > 1`, and `liquidatorGate == address(0)` (or the attacker also passes the gate). The attack is economically self-funding (attacker profits from the partial liquidation at `lif > 1` discount) and can be repeated indefinitely every time the victim resubmits. No special privileges are required. [8](#0-7) 

## Recommendation

Decouple the RCF cap from live state by one of the following approaches:

1. **Slippage parameter:** Allow the liquidator to pass a `minRepaidUnits` parameter and revert only if `repaidUnits < minRepaidUnits`, rather than reverting if `repaidUnits > maxRepaid`. This shifts the cap enforcement to the liquidator's own slippage tolerance.
2. **Clamp instead of revert:** Replace the `require(repaidUnits <= maxRepaid ...)` with `repaidUnits = min(repaidUnits, maxRepaid)` so that a front-run partial liquidation silently reduces the victim's liquidation amount rather than reverting it entirely.
3. **Commit-reveal / private mempool guidance:** Document that liquidators should use private transaction relays (e.g., Flashbots) to avoid front-running, and that `repaidUnits` should be set conservatively below `maxRepaid`.

Option 2 is the most robust on-chain fix and eliminates the revert path entirely. [9](#0-8)
