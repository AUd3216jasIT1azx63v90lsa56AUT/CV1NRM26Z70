### Title
No Way to Invalidate Issued Signed Authorizations in `EcrecoverAuthorizer` â€” (`src/periphery/EcrecoverAuthorizer.sol`)

---

### Summary

`EcrecoverAuthorizer` allows gasless, signature-based authorization of accounts on Midnight via EIP-712 signed messages. Once a user signs an `Authorization` with a long-dated `deadline`, there is no mechanism to invalidate that signature without first consuming the nonce. A user who calls `Midnight.setIsAuthorized` directly to revoke an authorized account can have that revocation immediately undone by an attacker replaying the outstanding signed authorization, because the nonce cannot be advanced without a valid signature.

---

### Finding Description

`EcrecoverAuthorizer.setIsAuthorized` enforces sequential nonces:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

The nonce only advances when a valid signed authorization is consumed. There is no standalone `increaseNonce()` or equivalent function that lets a user advance their nonce without providing a signature. [1](#0-0) 

**Attack path:**

1. Alice signs `Authorization { authorized: Mallory, isAuthorized: true, nonce: N, deadline: far_future }` and hands it to Mallory (e.g., as part of a gasless UX flow).
2. Alice later decides to revoke Mallory and calls `Midnight.setIsAuthorized(Mallory, false, Alice)` directly on the core contract.
3. Mallory (or anyone holding the signed message) calls `EcrecoverAuthorizer.setIsAuthorized` with the original signature. The nonce is still `N`, the deadline has not passed, so the call succeeds and re-authorizes Mallory on Midnight.
4. Alice's revocation is void. Mallory is authorized again.

The core contract's `setIsAuthorized` is accessible to anyone authorized on behalf of the user, and `isAuthorized` grants sweeping permissions across all protocol entry-points: [2](#0-1) 

---

### Impact Explanation

`isAuthorized` in Midnight is a blanket delegation. An authorized account can call `withdraw`, `withdrawCollateral`, `repay`, `supplyCollateral`, `setConsumed`, `setIsAuthorized` (further delegation), and act as `taker` in `take` â€” all on behalf of the victim. [3](#0-2) [4](#0-3) 

A malicious authorized account can drain all collateral and credit from a victim's positions. The inability to invalidate a signed authorization means a user who has signed one with a long deadline has no reliable way to revoke it â€” the revocation can always be front-run or back-run by replaying the permit.

**Severity: High** â€” direct theft of user assets is reachable with no privileged access required beyond possession of the signed message.

---

### Likelihood Explanation

- The signed authorization is an off-chain artifact that can be stored, leaked, or handed to a counterparty.
- The attacker only needs to watch the mempool for a `setIsAuthorized(Mallory, false, Alice)` transaction and front-run it with the signed permit, or simply replay it after the fact.
- No privileged keys are required. The attacker is a normal user who holds a valid signed message.
- Long-deadline authorizations are the expected UX pattern for gasless flows (the `deadline` field exists precisely to allow long-lived signatures). [5](#0-4) 

---

### Recommendation

Add a permissionless `increaseNonce()` function to `EcrecoverAuthorizer` that advances the caller's nonce without requiring a signature, invalidating all outstanding signed authorizations for that nonce:

```solidity
/// @notice Invalidates all outstanding signed authorizations for the caller at the current nonce.
function increaseNonce() external returns (uint256) {
    return ++nonce[msg.sender];
}
```

This mirrors the fix applied in the referenced eBTC report (PR 672 via `PermitNonce` parent) and is the standard pattern for EIP-712 permit invalidation. [6](#0-5) 

---

### Proof of Concept

```
1. Alice signs: Authorization { authorizer: Alice, authorized: Mallory, isAuthorized: true, nonce: 0, deadline: block.timestamp + 365 days }
2. Alice submits tx: Midnight.setIsAuthorized(Mallory, false, Alice)
3. Mallory observes Alice's tx in mempool.
4. Mallory front-runs with: EcrecoverAuthorizer.setIsAuthorized(authorization, aliceSig)
   â†’ nonce[Alice] becomes 1, Mallory is authorized on Midnight.
5. Alice's tx executes: Midnight.setIsAuthorized(Mallory, false, Alice) â†’ Mallory is deauthorized.
6. Mallory back-runs: impossible now (nonce consumed). BUT if Mallory front-ran step 4 before step 5, Mallory is authorized.
   Alternatively, if Alice never submitted step 2 and just wants to revoke, she has no mechanism at all â€” she cannot advance her nonce without a valid signature.
7. Mallory calls: Midnight.withdrawCollateral(market, idx, allCollateral, Alice, Mallory)
   â†’ Alice's collateral is drained.
```

The root cause is the absence of a standalone nonce-advancement function in `EcrecoverAuthorizer`, making it impossible to invalidate outstanding signed authorizations. [7](#0-6)

contract EcrecoverAuthorizer is IEcrecoverAuthorizer {
    address public immutable MIDNIGHT;
    mapping(address => uint256) public nonce;

    constructor(address _midnight) {

    }

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
