### Title
Unvalidated `rcfThreshold` Allows Permanent Disabling of Recovery Close Factor Protection for Borrowers — (File: `src/Midnight.sol`)

---

### Summary

The `touchMarket` function in `Midnight.sol` performs no validation on the `Market.rcfThreshold` field. Setting `rcfThreshold = type(uint256).max` at market creation permanently disables the Recovery Close Factor (RCF) for all normal-mode liquidations in that market. Any liquidator can then fully liquidate a borrower who is only marginally unhealthy, causing the borrower to lose their entire collateral position instead of only the minimum needed to restore health. Market creation is permissionless, so no privileged access is required.

---

### Finding Description

**Root cause — no validation on `rcfThreshold` in `touchMarket`:**

`touchMarket` validates `maturity`, `lltv`, `maxLif`, and collateral ordering, but applies zero constraints to `market.rcfThreshold`: [1](#0-0) 

The RCF enforcement in `liquidate` reads:

```solidity
require(
    repaidUnits <= maxRepaid
        || _position.collateral[collateralIndex].mulDivDown(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
            .mulDivDown(WAD, lif).zeroFloorSub(maxRepaid) < market.rcfThreshold,
    RecoveryCloseFactorConditionsViolated()
);
``` [2](#0-1) 

The second branch of the `||` is the RCF deactivation condition. Per the inline documentation, it is intended to fire only when the remaining collateral after a max-RCF liquidation would be too small to cover gas costs: [3](#0-2) 

**Extreme value — `rcfThreshold = type(uint256).max`:**

The left-hand side of `< market.rcfThreshold` is:

```
collateral * liquidatedCollatPrice / ORACLE_PRICE_SCALE / lif  -  maxRepaid
```

`collateral` is stored as `uint128` (max ≈ 3.4 × 10³⁸). `ORACLE_PRICE_SCALE = 1e36`, `lif ≥ WAD = 1e18`. The maximum realistic result of this expression is therefore bounded well below `type(uint256).max` (≈ 1.15 × 10⁷⁷). Consequently, when `rcfThreshold = type(uint256).max`, the condition is **always true**, the RCF check is permanently bypassed, and `repaidUnits` is unconstrained. [4](#0-3) 

**Extreme value — `rcfThreshold = 0`:**

`zeroFloorSub` returns a `uint256` (always ≥ 0). The condition `uint256 < 0` is always false, so the RCF deactivation safety valve is permanently disabled. Small, dust-level positions can never be fully liquidated in normal mode, blocking bad-debt realization and socializing losses onto lenders. [5](#0-4) 

---

### Impact Explanation

**`rcfThreshold = type(uint256).max`:** A borrower who is 1% underwater can be fully liquidated — losing 100% of their collateral — instead of only the minimum amount needed to restore health. The liquidator profits from the excess collateral seized at the LIF discount. This is a direct, quantifiable financial loss to borrowers proportional to the difference between their actual collateral and the RCF-capped amount.

**`rcfThreshold = 0`:** The RCF deactivation safety valve never fires. Dust positions that are unhealthy but too small to be worth liquidating under the RCF constraint accumulate bad debt, which is then socialized among all lenders in the market via the `lossFactor` slashing mechanism. [6](#0-5) 

---

### Likelihood Explanation

`touchMarket` is fully permissionless — any externally owned account can call it with arbitrary `Market` parameters. An attacker creates a market with `rcfThreshold = type(uint256).max` alongside otherwise legitimate parameters (valid `lltv`, reputable oracle, attractive maturity). Users who borrow in this market are exposed to over-liquidation the moment their position becomes marginally unhealthy. The `rcfThreshold` field is not surfaced prominently in UIs and its implication (RCF deactivation) is non-obvious to most users. [7](#0-6) 

---

### Recommendation

Add explicit bounds on `rcfThreshold` inside `touchMarket`:

```solidity
require(market.rcfThreshold > 0, InvalidRcfThreshold());
require(market.rcfThreshold <= SOME_MAX_RCF_THRESHOLD, InvalidRcfThreshold());
```

`SOME_MAX_RCF_THRESHOLD` should be calibrated so that the RCF deactivation only fires for positions whose remaining collateral value is genuinely below a meaningful dust threshold (e.g., a few hundred loan-token units, consistent with the comment at lines 70–76). This mirrors the fix pattern from the referenced external report: impose explicit `minTermDuration` / `minPrice` style guards on parameters that currently have only one-sided or no bounds.

### Proof of Concept

**Setup:**
- Loan token: `USDC`
- Collateral: `WETH`, oracle price = 2000 USDC/ETH, `lltv = 0.86e18`
- `rcfThreshold = type(uint256).max`
- Borrower supplies 1 ETH collateral, borrows 1700 USDC (healthy at 86% LTV)

**Attack steps:**

1. Attacker calls `touchMarket` with the above `Market` struct where `rcfThreshold = type(uint256).max`. Market creation succeeds — no revert. [8](#0-7) 

2. Borrower interacts with the market normally (supply collateral, take sell offer to borrow).

3. ETH price drops to 1980 USDC. Borrower's `maxDebt = 1 ETH × 1980 / 1e36 × 0.86e18 / 1e18 = 1702.8 USDC`. Borrower's debt = 1700 USDC. Position is **healthy** — not yet liquidatable.

4. ETH price drops to 1960 USDC. `maxDebt = 1685.6 USDC < 1700 USDC`. Position is now unhealthy. Under normal RCF, `maxRepaid ≈ (1700 - 1685.6) / (1 - lif × lltv)` — only ~50 USDC needs to be repaid to restore health.

5. Attacker calls `liquidate` with `repaidUnits = 1700` (full debt). The RCF check evaluates:
   - `repaidUnits (1700) <= maxRepaid (~50)` → **false**
   - `collateral_value / lif - maxRepaid < type(uint256).max` → **always true**
   - Require passes. Borrower's entire 1 ETH collateral is seized. [2](#0-1) 

6. Borrower loses ~1 ETH (≈ 1960 USDC) to repay 1700 USDC of debt — a ~260 USDC excess loss that should have been protected by the RCF.

        if (marketState[id].tickSpacing == 0) {
            require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
            require(market.collateralParams.length > 0, NoCollateralParams());
            require(market.collateralParams.length <= MAX_COLLATERALS, TooManyCollateralParams());

                    InvalidMaxLif()
                );
