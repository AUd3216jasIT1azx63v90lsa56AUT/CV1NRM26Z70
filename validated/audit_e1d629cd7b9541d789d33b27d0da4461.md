### Title
Borrower Can Permanently Block Their Own Liquidation by Activating a Collateral With a Reverting Oracle — (File: src/Midnight.sol)

---

### Summary

The `liquidate` function iterates over every activated collateral in a borrower's bitmap and calls each collateral's oracle. If any single oracle reverts, the entire liquidation reverts. Because `supplyCollateral` never calls the oracle, a borrower can activate a collateral whose oracle is already reverting (or will revert) at zero cost, permanently shielding their position from liquidation and forcing bad debt onto lenders.

---

### Finding Description

**Root cause — `supplyCollateral` activates without oracle check:**

`supplyCollateral` sets the collateral balance and flips the bit in `collateralBitmap` without ever calling the oracle. [1](#0-0) 

**Root cause — `liquidate` calls every activated oracle in a loop:**

`liquidate` iterates the full `collateralBitmap` and calls `IOracle(_collateralParam.oracle).price()` for every activated collateral. A single revert in this loop causes the entire transaction to revert. [2](#0-1) 

**Exploit path:**

1. A market is created with collateral A (healthy oracle) and collateral B (oracle that reverts — e.g., a deprecated Chainlink feed, a paused oracle, or a custom oracle the attacker controls in a permissionlessly-created market).
2. Borrower supplies collateral A and borrows loan tokens normally.
3. Borrower's position becomes unhealthy (debt > maxDebt).
4. Before liquidation, borrower calls `supplyCollateral` with 1 wei of collateral B. No oracle is called; the bit for collateral B is set in `collateralBitmap`.
5. Any call to `liquidate` now iterates the bitmap, hits collateral B's oracle, and reverts unconditionally.
6. `isHealthy` also reverts for this borrower (same oracle loop path), so `take` and `withdrawCollateral` also revert for the seller/borrower when they have debt — but `repay` does not call oracles, so the borrower retains the option to repay voluntarily while lenders have no enforcement path.

The protocol's own LIVENESS comment acknowledges this: *"Note that a borrower can activate a collateral once its oracle is reverting because the oracle is not called in `supplyCollateral`."* [3](#0-2) 

---

### Impact Explanation

- **Bad debt socialized to lenders**: An underwater borrower who activates a reverting oracle cannot be liquidated. Their bad debt cannot be realized via `liquidate`. Lenders in the market absorb the loss with no recourse.
- **Permanent lock**: There is no admin escape hatch, no way to forcibly deactivate a collateral, and no migration path. The position is permanently un-liquidatable as long as the oracle reverts.
- **Scope**: All lenders in the affected market are impacted proportionally to their credit share.

---

### Likelihood Explanation

- **Realistic oracle failure**: Chainlink feeds can be deprecated, paused, or have circuit breakers that cause `latestRoundData` to revert. Any market with a collateral backed by such a feed is vulnerable.
- **Permissionless market creation amplifies risk**: Markets are created permissionlessly. An attacker can deploy a market with a malicious oracle that returns normally until the attacker's position is about to be liquidated, then switches to reverting. Lenders who enter the market before the oracle is weaponized are exposed.
- **Low attacker cost**: Activating the reverting collateral costs only 1 wei of collateral B plus gas. The borrower retains their borrowed assets.
- **No privileged access required**: The attacker is a normal borrower exploiting a valid protocol flow (`supplyCollateral`).

---

### Recommendation

1. **Wrap oracle calls in try/catch inside `liquidate`**: If an oracle reverts for a collateral, treat its contribution to `maxDebt` as 0 (worst case for the borrower) and continue. This prevents a single broken oracle from blocking liquidation of the entire position.
2. **Alternatively, allow liquidation to proceed on a per-collateral basis**: Let liquidators specify which collateral to seize and only call that collateral's oracle, skipping the full-bitmap health loop for the purpose of determining liquidatability.
3. **Document and enforce oracle liveness requirements** at market creation, or add an on-chain oracle liveness check that gates collateral activation.

### Proof of Concept

```
Setup:
  - Market M with collateralParams[0] = {token: WETH, oracle: HealthyOracle}
                   collateralParams[1] = {token: USDC, oracle: RevertingOracle}
  - RevertingOracle.price() always reverts (or attacker flips it to revert on demand)

Step 1: Borrower supplies 100 WETH (collateral index 0), borrows 80 units.
        collateralBitmap = 0b01 (only index 0 set)

Step 2: WETH price drops; borrower's position becomes unhealthy (debt > maxDebt).

Step 3: Borrower calls supplyCollateral(M, 1 /*index*/, 1 /*1 wei USDC*/, borrower).
        supplyCollateral does NOT call RevertingOracle.
        collateralBitmap = 0b11 (both index 0 and 1 set)

Step 4: Liquidator calls liquidate(M, 0, 0, repaidUnits, borrower, false, ...).
        Loop hits index 1: IOracle(RevertingOracle).price() → REVERT
        Entire liquidate() call reverts.

Step 5: Borrower keeps borrowed assets. Lenders cannot recover via liquidation.
        Bad debt accrues indefinitely.

    function supplyCollateral(Market memory market, uint256 collateralIndex, uint256 assets, address onBehalf)
        external
    {
        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
        bytes32 id = touchMarket(market);
        address collateralToken = market.collateralParams[collateralIndex].token;

        Position storage _position = position[id][onBehalf];
        uint256 oldCollateral = _position.collateral[collateralIndex];
        _position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);

        if (oldCollateral == 0 && assets > 0) {
            uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
            _position.collateralBitmap = newCollateralBitmap;
            require(
                UtilsLib.countBits(newCollateralBitmap) <= MAX_COLLATERALS_PER_BORROWER, TooManyActivatedCollaterals()
            );
        }

        emit EventsLib.SupplyCollateral(msg.sender, id, collateralToken, assets, onBehalf);

        SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
    }
