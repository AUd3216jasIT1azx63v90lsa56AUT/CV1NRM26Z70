Audit Report

## Title
Missing `caller` Parameter in `onRepay` Callback Enables Arbitrary Third-Party Token Drain - (File: `src/Midnight.sol`)

## Summary
`Midnight.repay` unconditionally sets `payer = callback` when a non-zero callback address is supplied, then calls `IRepayCallback(callback).onRepay(...)` before pulling tokens from `payer` via `safeTransferFrom`. Unlike `onLiquidate` and `onFlashLoan`, the `onRepay` interface omits the initiating `caller` address, structurally preventing any callback contract from distinguishing a legitimate self-initiated invocation from a malicious one. An attacker with any debt position can designate an arbitrary third-party contract as `callback`, causing that contract's pre-approved tokens to be consumed to repay the attacker's own debt without the third party's consent.

## Finding Description

**Exact code path:**

`src/Midnight.sol` lines 502–521:
```solidity
function repay(Market memory market, uint256 units, address onBehalf, address callback, bytes calldata data)
    external
{
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    // ...
    address payer = callback != address(0) ? callback : msg.sender;   // line 511
    // ...
    if (callback != address(0)) {
        require(
            IRepayCallback(callback).onRepay(id, market, units, onBehalf, data) == CALLBACK_SUCCESS,
            WrongRepayCallbackReturnValue()
        );
    }
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
}
```

`src/interfaces/ICallbacks.sol` line 21:
```solidity
function onRepay(bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
```

**Root cause — missing `caller` parameter in `onRepay`:**

The three payer-designating callbacks are structurally inconsistent:

| Callback | Passes `caller`? |
|---|---|
| `onFlashLoan(address caller, ...)` | ✓ (`ICallbacks.sol` line 25) |
| `onLiquidate(address caller, ...)` | ✓ (`ICallbacks.sol` line 17) |
| `onRepay(bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data)` | ✗ |

Inside `onRepay`, `msg.sender` is always `Midnight` — it identifies the protocol, not the transaction initiator. Without `caller` being passed, a callback contract has no protocol-supplied way to verify who triggered the repayment.

**Exploit path (no victim authorization required):**

1. Attacker holds any debt position in any market (or opens one trivially).
2. `thirdParty` is any contract that: (a) implements `IRepayCallback` and returns `CALLBACK_SUCCESS` from `onRepay`, and (b) has approved Midnight to spend its loan tokens — e.g., a repayment helper, bundler, or position manager.
3. Attacker calls `repay(market, units, onBehalf=attacker, callback=thirdParty, data)`.
4. Authorization check passes: `onBehalf == msg.sender`.
5. `payer = thirdParty`.
6. `thirdParty.onRepay(id, market, units, attacker, data)` is called. It cannot check the initiator because `caller` is not passed; checking `msg.sender == Midnight` only confirms the protocol called it, not who triggered it.
7. `safeTransferFrom(loanToken, thirdParty, address(this), units)` executes, draining `thirdParty`'s tokens.
8. Attacker's debt is reduced; `thirdParty` bears the full cost.

**Why existing checks fail:**

- The authorization check (`onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`) only validates the caller's right to act for `onBehalf`. It does not restrict which address may be designated as `callback`/`payer`.
- There is no check that `callback == msg.sender` or that `isAuthorized[callback][msg.sender]`.
- The `onRepay` interface structurally prevents the callback from self-protecting via the initiator address, unlike `onLiquidate` and `onFlashLoan`.

## Impact Explanation
Any contract implementing `IRepayCallback` with a Midnight token allowance is subject to direct, unconditional theft of its loan tokens. The attacker's debt is reduced at the victim contract's expense; the victim receives nothing in return and never consented to the payment. This is direct theft of assets from protocol integrators (repayment helpers, bundlers, position managers). The ERC-20 allowance granted to Midnight is consumed without the approving contract's consent, violating the invariant that approvals bind the right user/action/amount.

## Likelihood Explanation
**Required preconditions:**
1. Attacker has any debt position — trivially achievable by any user.
2. `thirdParty` implements `IRepayCallback` and has approved Midnight — realistic for any repayment helper or bundler participating in the protocol.
3. `thirdParty.onRepay` returns `CALLBACK_SUCCESS` without out-of-band initiator tracking — structurally forced by the missing `caller` parameter; transient-storage-based self-call tracking is a non-standard mitigation that most integrators will not implement.

No oracle manipulation, admin access, victim authorization, or token misbehavior is required. The attack is repeatable as long as the allowance and debt remain.

## Recommendation
Add `msg.sender` as the first `caller` parameter to `onRepay`, consistent with `onLiquidate` and `onFlashLoan`:

```solidity
// ICallbacks.sol
interface IRepayCallback {
    function onRepay(address caller, bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
}

// Midnight.sol — inside repay()
IRepayCallback(callback).onRepay(msg.sender, id, market, units, onBehalf, data)
```

This allows callback contracts to verify that the initiating caller is the expected party (e.g., `require(caller == expectedRepayer)`), closing the attack surface.

## Proof of Concept

**Minimal Foundry test:**

```solidity
contract MaliciousRepayCallback {
    // Simulates a repayment helper that has approved Midnight and returns CALLBACK_SUCCESS
    function onRepay(bytes32, Market memory, uint256, address, bytes memory)
        external returns (bytes32)
    {
        return CALLBACK_SUCCESS; // cannot check initiator — caller not passed
    }
}

function testThirdPartyDrain() public {
    // Setup: attacker has debt, thirdParty has tokens and Midnight approval
    address attacker = makeAddr("attacker");
    MaliciousRepayCallback thirdParty = new MaliciousRepayCallback();

    // Give thirdParty loan tokens and approve Midnight
    deal(address(loanToken), address(thirdParty), units);
    vm.prank(address(thirdParty));
    loanToken.approve(address(midnight), units);

    // Attacker opens debt position (onBehalf = attacker, so auth check passes)
    collateralize(market, attacker, units);
    setupMarket(market, units);

    uint256 thirdPartyBefore = loanToken.balanceOf(address(thirdParty));

    // Attacker designates thirdParty as callback/payer
    vm.prank(attacker);
    midnight.repay(market, units, attacker, address(thirdParty), hex"");

    // thirdParty's tokens were consumed to repay attacker's debt
    assertEq(loanToken.balanceOf(address(thirdParty)), thirdPartyBefore - units);
    assertEq(midnight.debtOf(id, attacker), 0);
}
```

**Expected result:** `thirdParty` loses `units` of loan tokens; attacker's debt is cleared. The test passes, confirming the drain.
