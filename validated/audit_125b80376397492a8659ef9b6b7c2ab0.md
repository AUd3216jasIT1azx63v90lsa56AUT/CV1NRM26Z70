### Title
Authorized Account Can Implant Persistent Sub-Authorization That Survives Revocation — (File: src/Midnight.sol)

### Summary
`Midnight.sol`'s `setIsAuthorized` allows any currently-authorized account to grant further authorizations on behalf of a user. When the user later revokes the original authorized account, sub-authorizations that account created are **not cascaded** and remain active. An attacker who briefly held authorization (or compromised an authorized smart contract) can plant a persistent backdoor before being revoked, then drain the victim's funds after revocation.

### Finding Description

The root cause is in `setIsAuthorized`: [1](#0-0) 

The check `isAuthorized[onBehalf][msg.sender]` permits any authorized account to call this function and write any value into `isAuthorized[onBehalf][attacker]`. There is no parent-tracking or revocation cascade. When the user later calls `setIsAuthorized(Bob, false, Alice)`, only `isAuthorized[Alice][Bob]` is cleared; `isAuthorized[Alice][Attacker]` — written by Bob — remains `true`.

The same delegation path is reachable through `EcrecoverAuthorizer.setIsAuthorized`, which also accepts a signature from any currently-authorized signer: [2](#0-1) 

Once `isAuthorized[Alice][Attacker]` is `true`, the attacker can call:

- `withdraw(market, units, Alice, attacker)` — drains Alice's credit [3](#0-2) 
- `withdrawCollateral(market, index, assets, Alice, attacker)` — drains Alice's collateral [4](#0-3) 
- `setIsAuthorized(AnotherAttacker, true, Alice)` — propagates the backdoor further [1](#0-0) 

The contract comment acknowledges the delegation capability but does not warn that revocation is non-cascading: [5](#0-4) 

### Impact Explanation
An attacker with a planted sub-authorization can call `withdraw` and `withdrawCollateral` with an arbitrary `receiver`, achieving **direct, complete theft** of the victim's credit units and all collateral assets across every market where the victim has a position. No further preconditions are needed after the backdoor is planted.

### Likelihood Explanation
Users routinely authorize smart contracts for trading bots, bundlers (`MidnightBundles`), and ratifiers. Any of these contracts that contain a re-entrancy path, an access-control bug, or a malicious upgrade can be used to call `setIsAuthorized(attacker, true, victim)` before the victim notices and revokes. The attacker only needs a single transaction window while the contract is authorized. The `EcrecoverAuthorizer` path additionally allows a signed authorization message to be submitted by anyone, so a leaked or front-run signature is sufficient.

### Recommendation
Track the authorization parent at grant time and cascade revocations, **or** restrict `setIsAuthorized` so that only `onBehalf == msg.sender` (the account itself) can grant new authorizations — authorized accounts may only act on existing permissions, not create new ones. At minimum, add explicit NatSpec warning that revoking an account does **not** revoke sub-authorizations it created, so integrators can audit their exposure.

### Proof of Concept

```
1. Alice calls: midnight.setIsAuthorized(Bob, true, Alice)
   → isAuthorized[Alice][Bob] = true

2. Bob (compromised smart contract or malicious actor) calls:
   midnight.setIsAuthorized(Attacker, true, Alice)
   → isAuthorized[Alice][Attacker] = true

3. Alice detects the compromise and calls:
   midnight.setIsAuthorized(Bob, false, Alice)
   → isAuthorized[Alice][Bob] = false
   → isAuthorized[Alice][Attacker] is STILL true  ← not cleared

4. Attacker calls:
   midnight.withdraw(market, aliceCredit, Alice, Attacker)
   → passes: isAuthorized[Alice][Attacker] == true
   → Alice's credit is transferred to Attacker

5. Attacker calls:
   midnight.withdrawCollateral(market, 0, aliceCollateral, Alice, Attacker)
   → passes: isAuthorized[Alice][Attacker] == true
   → Alice's collateral is transferred to Attacker
```

Expected result: Alice's full position is drained despite having revoked Bob, because the sub-authorization Bob planted was never cleared.
