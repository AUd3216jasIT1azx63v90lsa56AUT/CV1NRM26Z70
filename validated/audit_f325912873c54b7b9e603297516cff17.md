Audit Report

## Title
Off-by-one in `CannotIncreaseDebtPostMaturity` guard permits debt creation at `block.timestamp == maturity` - (File: `src/Midnight.sol`)

## Summary
The maturity guard at line 391 of `src/Midnight.sol` uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` at exact equality, allowing `sellerDebtIncrease > 0` at the maturity timestamp itself. The post-take health check (`isHealthy`) is a pre-maturity check that passes at this moment, so debt is written to state. One second later, the liquidation path at line 622 evaluates `postMaturityMode ? block.timestamp > market.maturity` as `true`, making the position immediately liquidatable regardless of collateral health, in direct violation of the protocol's no-debt-increase-at-or-after-maturity invariant.

## Finding Description
**Root cause — line 391, `src/Midnight.sol`:**
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
`<=` passes when `block.timestamp == offer.market.maturity`, so `sellerDebtIncrease > 0` is not blocked at the maturity boundary.

**Exploit flow:**
1. Attacker holds any valid sell offer (standard protocol operation, no privilege required).
2. Transaction lands in the block where `block.timestamp == market.maturity`.
3. `take` is called with `units > sellerPos.credit`, so `sellerDebtIncrease = units - sellerCreditDecrease > 0`.
4. Line 391: `maturity <= maturity` → `true` → guard passes.
5. Line 476: `isHealthy` evaluates the pre-maturity condition (`maxDebt >= debt`); at this timestamp the position is healthy, so the check passes.
6. `sellerPos.debt += sellerDebtIncrease` executes — debt is created at maturity.
7. At `block.timestamp == maturity + 1`, line 622: `postMaturityMode ? block.timestamp > market.maturity` → `maturity+1 > maturity` → `true` → position is immediately liquidatable regardless of collateral health.

**Why existing checks fail:**
- The `CannotIncreaseDebtPostMaturity` guard uses `<=` instead of `<`, creating a one-block window.
- The `SellerIsLiquidatable` check at line 476 calls `isHealthy`, which is the pre-maturity health formula (`maxDebt >= debt`). At `block.timestamp == maturity`, the position can be healthy under this formula, so the check does not block the operation.
- No existing test exercises `block.timestamp == market.maturity` with `sellerDebtIncrease > 0`; all post-maturity tests warp to `maturity + 1`.

## Impact Explanation
Debt is created in violation of the core protocol invariant that debt must not increase at or after maturity. The affected position enters post-maturity mode in the very next block and becomes immediately liquidatable regardless of collateral health. The seller has no opportunity to repay before liquidation. While the LIF at `maturity + 1` equals `WAD` (no liquidation bonus), the position is still liquidatable and the debt was created in a state the protocol explicitly prohibits, constituting an accounting integrity failure and unauthorized state change.

## Likelihood Explanation
Requires a transaction to land in the exact block where `block.timestamp == market.maturity`. On EVM chains with predictable block timestamps (e.g., post-Merge Ethereum with 12-second slots), a validator or MEV searcher using Flashbots can guarantee this. A regular attacker can monitor the mempool and submit with appropriate timing. The only precondition is a valid sell offer, which is a normal, unprivileged protocol operation. The attack is repeatable for every market whose maturity timestamp coincides with a block boundary.

## Recommendation
Change `<=` to `<` at line 391:
```solidity
// Before
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
This aligns the guard with the liquidation check's strict `>` at line 622 and closes the one-block window. Add a test case that warps `block.timestamp` to exactly `market.maturity` and asserts the call reverts with `CannotIncreaseDebtPostMaturity` when `sellerDebtIncrease > 0`.
