Audit Report

## Title
Fee-on-Transfer Collateral Token Inflates `position.collateral` Accounting, Enabling Undercollateralized Debt Creation - (File: `src/Midnight.sol`)

## Summary

`supplyCollateral` writes the caller-supplied `assets` value directly to `_position.collateral[collateralIndex]` at line 533 before executing the ERC20 transfer at line 545, with no balance-before/after snapshot. When the collateral token charges a transfer fee, Midnight receives fewer tokens than recorded. `isHealthy` then computes `maxDebt` from the inflated stored value, allowing a seller to carry debt that exceeds the actual on-hand collateral, creating undercollateralized positions at inception and causing bad debt for lenders upon liquidation.

## Finding Description

**Root cause — `supplyCollateral` (`src/Midnight.sol` lines 531–545):**

```solidity
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets); // line 533
...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545
```

The accounting write at line 533 uses the caller-controlled `assets` parameter. The transfer at line 545 executes after the state update with no balance snapshot. For a 1% fee-on-transfer token: `supplyCollateral(..., 100, ...)` records `position.collateral[i] = 100` but Midnight receives 99.

**No restriction on collateral token type — `touchMarket` (`src/Midnight.sol` lines 757–773):**

Market creation validates only sorted order, non-zero address, allowed LLTV tiers, and valid `maxLif`. There is no check that the collateral token is not a fee-on-transfer token. Any unprivileged actor can create such a market.

**Health check uses inflated value — `isHealthy` (`src/Midnight.sol` lines 954–955):**

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

`_position.collateral[i]` is the inflated recorded value (100), not the actual balance (99). `maxDebt` is therefore ~1% higher than the true collateral-backed capacity.

**Health check at end of `take` (`src/Midnight.sol` line 476):**

```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```

`isHealthy` returns `true` using the inflated collateral, so the take succeeds and the seller's debt is increased beyond what their actual collateral covers.

**Exploit flow:**
1. Attacker deploys or uses an existing fee-on-transfer ERC20 (1% fee) as collateral token.
2. Anyone calls `touchMarket` with this token as `collateralParams[0].token` — succeeds, no restriction.
3. Seller calls `supplyCollateral(market, 0, 100, seller)` → `position.collateral[0] = 100`, Midnight balance increases by 99.
4. Taker calls `take(offer, ratifierData, units, taker, ...)`.
5. `isHealthy` computes `maxDebt` using `position.collateral[0] = 100`, returns `true`.
6. Take succeeds. Seller holds debt backed by 99 units of actual collateral, but protocol believes it is backed by 100.

**Existing checks are insufficient:** `touchMarket` validates only structural market parameters (sort order, LLTV, maxLif) — no token behavior check. `isHealthy` reads only the stored accounting value, not the actual contract balance. No balance-delta check exists anywhere in `supplyCollateral`.

## Impact Explanation

The invariant "contract balances cover recorded collateral" is violated from the moment `supplyCollateral` is called with a fee-on-transfer token. The seller's position is undercollateralized at inception: actual collateral held < collateral used to compute `maxDebt`. The seller can borrow up to `100 * price * lltv / WAD` units of debt while only 99 units of collateral are held. The shortfall scales with position size and fee rate. When the position is eventually liquidated, the protocol realizes bad debt proportional to the fee, causing direct loss to lenders via the `lossFactor` mechanism. This constitutes direct theft of lender assets.

## Likelihood Explanation

All preconditions are reachable by any unprivileged user: fee-on-transfer ERC20 tokens exist and are deployed on mainnet (USDT has historically had this mode; many DeFi tokens implement it), market creation is permissionless with no token-type restriction, and normal user actions (signing and taking offers) complete the exploit. No admin keys, oracle manipulation, or impossible values are required. The attack is repeatable on every `supplyCollateral` call with a fee-on-transfer token.

## Recommendation

Use a balance-before/after snapshot pattern in `supplyCollateral` to credit only the actual received amount:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Alternatively, whitelist collateral tokens at the protocol level to exclude fee-on-transfer tokens, or add an explicit invariant check that the contract's token balance is at least equal to the sum of all recorded collateral for that token.

## Proof of Concept

**Minimal Foundry test plan:**

1. Deploy a mock ERC20 that deducts a 1% fee on every `transferFrom`, crediting the fee to a fee collector address.
2. Deploy Midnight and create a market using the fee-on-transfer token as `collateralParams[0].token` via `touchMarket`.
3. Mint 100 tokens to the seller. Seller calls `supplyCollateral(market, 0, 100, seller)`.
4. Assert `position[id][seller].collateral[0] == 100`.
5. Assert `IERC20(feeToken).balanceOf(address(midnight)) == 99`.
6. Construct a signed offer from the seller with debt = `100 * price * lltv / WAD`.
7. Taker calls `take(offer, ...)`.
8. Assert the call succeeds (does not revert with `SellerIsLiquidatable`).
9. Assert `position[id][seller].debt > 99 * price * lltv / WAD` — debt exceeds actual collateral-backed capacity.
10. Confirm that if the seller's actual collateral (99 tokens) were liquidated at oracle price, the proceeds are insufficient to cover the debt, demonstrating bad debt.
