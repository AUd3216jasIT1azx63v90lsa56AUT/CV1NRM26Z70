### Title
`setConsumed()` Silent No-Op Enables Double-Fill via Front-Running in `multicall` Cancel-and-Take Pattern — (File: `src/Midnight.sol`)

---

### Summary

When a maker wants to atomically cancel their existing offer and take a counter-offer, they use `multicall([setConsumed(group, type(uint256).max), take(counterOffer)])`. Because `setConsumed` with `type(uint256).max` **never reverts** — even if the offer was already fully consumed — an attacker can front-run the multicall by filling the original offer first. The `setConsumed` call then silently succeeds as a no-op, and `take(counterOffer)` also succeeds, leaving the user with double the intended exposure.

---

### Finding Description

**Root cause — `setConsumed` never reverts when cancelling:** [1](#0-0) 

```solidity
/// @dev Passing type(uint256).max cancels all offers in the group (and never reverts).
function setConsumed(bytes32 group, uint256 amount, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    require(amount >= consumed[onBehalf][group], AlreadyConsumed());
    consumed[onBehalf][group] = amount;
    emit EventsLib.SetConsumed(msg.sender, group, amount, onBehalf);
}
```

The check `require(amount >= consumed[onBehalf][group])` always passes when `amount = type(uint256).max`, regardless of whether the offer was already fully consumed by a prior `take()`. This is the exact analog of PuttyV2's `cancel()` not reverting when an order was already filled.

**Attack path via `multicall`:** [2](#0-1) 

The standard pattern for a maker to atomically cancel their offer and take a counter-offer is:

```solidity
midnight.multicall([
    abi.encodeCall(midnight.setConsumed, (group, type(uint256).max, alice)),
    abi.encodeCall(midnight.take, (counterOffer, ratifierData, units, alice, ...))
]);
```

**Front-run scenario:**

1. Alice posts offer A (sell offer, `maxUnits = 100`, `group = G`) — broadcast publicly on the order book.
2. Bob creates counter-offer B. Alice wants to cancel A and take B.
3. Alice submits `multicall([setConsumed(G, max), take(B)])` to the mempool.
4. Attacker sees the pending multicall, extracts offer A's details (previously broadcast), and front-runs with `take(A, ...)` — filling Alice's original offer.
5. Alice's multicall executes:
   - `setConsumed(G, type(uint256).max)` → `require(max >= 100)` passes → **silent no-op**, `consumed[Alice][G]` already at 100.
   - `take(B)` → succeeds normally.
6. **Result**: Alice's original sell offer A was filled (she took on debt she didn't intend) AND she also took counter-offer B (more debt). Alice is twice as leveraged as intended.

---

### Impact Explanation

A maker using the standard cancel-and-replace pattern via `multicall` can have both their original offer and the counter-offer filled simultaneously. If both are sell offers (borrowing), the user ends up with **twice the debt** they intended. If both are buy offers (lending), the user commits **twice the capital**. In either case, the user's position is materially different from their intent, with direct financial loss (unexpected debt, locked capital, or liquidation risk from over-leverage).

---

### Likelihood Explanation

- **Attacker preconditions**: No privileged access required. The attacker only needs the original offer's parameters and ratifier data, which are publicly broadcast when the maker posts the offer to the order book.
- **Trigger**: Standard mempool front-running (MEV). Any searcher monitoring the mempool for `multicall` transactions containing `setConsumed` + `take` can execute this.
- **Frequency**: Any user who uses the cancel-and-replace workflow (a common UX pattern for updating offers) is vulnerable. This is a realistic, everyday operation.

---

### Recommendation

Add a `maxConsumed` guard to `setConsumed` so the caller can assert the offer has not been filled beyond an expected threshold. If the offer was already consumed, the multicall reverts atomically:

```solidity
function setConsumed(bytes32 group, uint256 amount, address onBehalf, uint256 maxCurrentConsumed) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    require(consumed[onBehalf][group] <= maxCurrentConsumed, AlreadyConsumed()); // NEW: revert if already filled
    require(amount >= consumed[onBehalf][group], AlreadyConsumed());
    consumed[onBehalf][group] = amount;
    emit EventsLib.SetConsumed(msg.sender, group, amount, onBehalf);
}
```

Alternatively, document that users must check `consumed[maker][group]` before constructing the multicall and use a slippage-style check contract wrapper.

### Proof of Concept

**Setup:**
- Alice posts sell offer A: `maker=Alice`, `group=G`, `maxUnits=100`, `tick=T` (broadcast publicly).
- Bob creates buy counter-offer B: `maker=Bob`, `group=G2`, `maxUnits=100`.

**Execution:**

1. Alice submits:
   ```solidity
   midnight.multicall([
       abi.encodeCall(midnight.setConsumed, (G, type(uint256).max, Alice)),
       abi.encodeCall(midnight.take, (offerB, ratifierDataB, 100, Alice, ...))
   ]);
   ```

2. Attacker front-runs (higher gas):
   ```solidity
   midnight.take(offerA, ratifierDataA, 100, attacker, ...);
   // consumed[Alice][G] = 100 (offer A fully filled)
   ```

3. Alice's multicall executes:
   - `setConsumed(G, type(uint256).max, Alice)`:
     - `require(type(uint256).max >= 100)` → **passes** (no revert)
     - `consumed[Alice][G] = type(uint256).max` (no-op, offer already consumed)
   - `take(offerB, ...)`:
     - Succeeds; Alice takes on debt from offer B as well.

4. **Final state**: Alice has debt from offer A (filled by attacker) **and** debt from offer B (filled by Alice). Alice is 2× leveraged vs. intent. [1](#0-0) [2](#0-1) [3](#0-2)
