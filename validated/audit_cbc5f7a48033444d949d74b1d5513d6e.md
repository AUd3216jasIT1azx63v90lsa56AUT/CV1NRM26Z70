Audit Report

## Title
Missing `caller` Parameter in `onRepay` Callback Enables Arbitrary Third-Party Token Drain - (`File: src/Midnight.sol`)

## Summary
`Midnight.repay` sets `payer = callback` when a non-zero callback is supplied, then calls `IRepayCallback(callback).onRepay(...)` before pulling tokens from `payer` via `safeTransferFrom`. Unlike `onLiquidate` and `onFlashLoan`, the `onRepay` interface omits the `caller` address, structurally preventing any callback contract from verifying who initiated the repayment. An attacker with their own debt position can designate any third-party contract as `callback`, causing that contract's approved tokens to be pulled to repay the attacker's own debt without the third party's consent.

## Finding Description

**Exact code path:**

`src/Midnight.sol` lines 502–521:
```solidity
function repay(Market memory market, uint256 units, address onBehalf, address callback, bytes calldata data)
    external
{
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    // ...
    address payer = callback != address(0) ? callback : msg.sender;   // line 511
    // ...
    if (callback != address(0)) {
        require(
            IRepayCallback(callback).onRepay(id, market, units, onBehalf, data) == CALLBACK_SUCCESS,
            WrongRepayCallbackReturnValue()
        );
    }
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
}
```

`src/interfaces/ICallbacks.sol` line 21:
```solidity
function onRepay(bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
```

**Root cause — missing `caller` parameter in `onRepay`:**

The three payer-designating callbacks are inconsistent:
- `onFlashLoan(address caller, ...)` — passes `msg.sender` ✓
- `onLiquidate(address caller, ...)` — passes `msg.sender` ✓ (line 702, `src/Midnight.sol`)
- `onRepay(bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data)` — **no caller** ✗

Because `onRepay` does not receive the initiating caller's address, a callback contract cannot distinguish a legitimate invocation from a malicious one. Checking `msg.sender == Midnight` is insufficient — it only confirms the call came from the protocol, not who triggered it.

**Simplest exploit path (no victim authorization required):**

The claim focuses on the `isAuthorized` path, but the authorization check `require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender])` is trivially satisfied when `onBehalf = msg.sender`. An attacker with their own debt position needs no victim at all:

1. Attacker has debt in any market (or opens a position).
2. `thirdParty` is any contract that has approved Midnight to spend its loan tokens and whose `onRepay` returns `CALLBACK_SUCCESS` (e.g., a repayment helper, bundler, or position manager).
3. Attacker calls `repay(market, units, onBehalf=attacker, callback=thirdParty, data)`.
4. Authorization check passes: `onBehalf == msg.sender`.
5. `payer = thirdParty`.
6. `thirdParty.onRepay(...)` is called — returns `CALLBACK_SUCCESS`; it cannot check the initiator because `caller` is not passed.
7. `safeTransferFrom(loanToken, thirdParty, address(this), units)` executes, draining `thirdParty`.
8. Attacker's own debt is reduced; `thirdParty` bears the cost.

**Why existing checks fail:**
- The authorization check only validates the caller's right to act for `onBehalf`, not the right to designate an arbitrary third party as payer.
- There is no check `callback == msg.sender` or `isAuthorized[callback][msg.sender]`.
- The `onRepay` interface structurally prevents the callback from self-protecting via the caller address, unlike `onLiquidate` and `onFlashLoan`.

## Impact Explanation
`thirdParty`'s loan tokens are transferred to Midnight to repay the attacker's debt. `thirdParty` receives nothing in return and did not consent to the payment. This constitutes direct theft of assets from any contract implementing `IRepayCallback` with a Midnight token allowance. The invariant that "callbacks, ERC20 transfers, multicall, or reentrancy cannot corrupt partial state" and that "approvals bind the right user/market/action/amount/deadline" are both violated: the callback is invoked against a party that never agreed to be the payer, and the ERC20 allowance granted to Midnight is consumed without the approving contract's consent.

## Likelihood Explanation
**Preconditions:**
1. Attacker has any debt position in the protocol — trivially achievable by any user.
2. `thirdParty` has approved Midnight to spend its loan tokens — realistic for any repayment helper, bundler, or position manager contract that participates in the protocol.
3. `thirdParty.onRepay` returns `CALLBACK_SUCCESS` without verifying the initiator — structurally forced by the missing `caller` parameter; any contract that cannot use out-of-band state (e.g., transient storage tracking self-initiated calls) is vulnerable.

The attack requires no oracle manipulation, no admin access, no victim authorization, and no token misbehavior. It is repeatable as long as the allowance and debt remain. Any protocol integrator that builds a repayment helper contract implementing `IRepayCallback` and approving Midnight is immediately at risk.

## Recommendation
Add `address caller` as the first parameter to `onRepay`, matching the established pattern of `onLiquidate` and `onFlashLoan`:

```solidity
// src/interfaces/ICallbacks.sol
interface IRepayCallback {
    function onRepay(address caller, bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
}
```

And pass `msg.sender` at the call site in `src/Midnight.sol`:
```solidity
IRepayCallback(callback).onRepay(msg.sender, id, market, units, onBehalf, data)
```

This allows callback contracts to verify that the entity that called `repay` is the one they authorized to use their tokens, consistent with how `onLiquidate` and `onFlashLoan` are protected.

## Proof of Concept
```solidity
// Minimal Foundry test
contract MaliciousRepayCallback is IRepayCallback {
    // Returns CALLBACK_SUCCESS unconditionally — cannot check caller
    function onRepay(bytes32, Market memory, uint256, address, bytes memory) external returns (bytes32) {
        return CALLBACK_SUCCESS;
    }
}

function testDrainThirdParty() public {
    // Setup: thirdParty has loan tokens and has approved Midnight
    address thirdParty = address(new MaliciousRepayCallback());
    deal(address(loanToken), thirdParty, 1000);
    vm.prank(thirdParty);
    loanToken.approve(address(midnight), 1000);

    // Attacker opens a debt position
    collateralize(market, attacker, 1000);
    setupMarket(market, 1000); // attacker now has 1000 units of debt

    // Attacker repays their own debt using thirdParty's tokens
    vm.prank(attacker);
    midnight.repay(market, 1000, attacker, thirdParty, "");

    // thirdParty's tokens are gone; attacker's debt is cleared
    assertEq(loanToken.balanceOf(thirdParty), 0);
    assertEq(midnight.debtOf(id, attacker), 0);
}
```
