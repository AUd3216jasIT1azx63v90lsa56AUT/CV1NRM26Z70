### Title
Permissionless `liquidate` with Attacker-Controlled `receiver` Enables Front-Running to Steal Liquidation Profit — (File: `src/Midnight.sol`)

---

### Summary

The `liquidate` function in `Midnight.sol` performs no authorization check on the caller when `liquidatorGate == address(0)`, and accepts a fully attacker-controlled `receiver` address that determines where seized collateral is sent. An attacker can monitor the mempool for pending liquidation transactions, front-run them with an identical call but a substituted `receiver`, and redirect the entire liquidation profit to themselves — using the native `callback` mechanism to avoid any upfront capital requirement.

---

### Finding Description

**Root cause — two cooperating design choices with no binding between caller and receiver:**

**1. No liquidator authorization when gate is absent.** [1](#0-0) 

When `market.liquidatorGate == address(0)` (the default for any market created without a gate), any `msg.sender` may call `liquidate`. There is no check that `msg.sender` is the intended liquidator.

**2. Seized collateral is sent to an unconstrained `receiver` parameter.** [2](#0-1) 

The collateral transfer destination is whatever address the caller passes. There is no requirement that `receiver == msg.sender` or any other binding.

**3. Repayment is collected *after* collateral is already sent, via a caller-supplied `callback`.** [3](#0-2) [4](#0-3) 

When `callback != address(0)`, the `payer` becomes the callback contract. The seized collateral is transferred to `receiver` first (line 696), then `onLiquidate` is invoked on the callback (lines 698–714), and only then are loan tokens pulled from `payer` (line 717). This ordering lets an attacker's callback contract sell the already-received collateral to fund the repayment — zero upfront capital required.

**Exploit flow:**

1. Legitimate liquidator `L` identifies an unhealthy borrower and broadcasts:
   `liquidate(..., receiver=L, callback=address(0), ...)`
2. Attacker `A` sees the pending transaction in the mempool.
3. `A` front-runs with a higher gas price, identical market/borrower/amount parameters, but:
   - `receiver = A` (or `A`'s profit address)
   - `callback = A_contract` (flash-loan-enabled liquidation contract)
4. Midnight sends `seizedAssets` of collateral to `A` (line 696).
5. Midnight calls `A_contract.onLiquidate(...)` (line 700).
6. Inside the callback, `A_contract` sells the received collateral (or uses a flash loan) to obtain `repaidUnits` of loan token, which is then pulled by Midnight at line 717.
7. `A` nets the spread: `seizedAssets × collatPrice / LIF − repaidUnits × loanPrice > 0`.
8. `L`'s original transaction reverts (`NotLiquidatable`) because the position is already closed.

The `ILiquidateCallback` interface confirms the callback receives all necessary context (`seizedAssets`, `repaidUnits`, `receiver`) to execute this atomically: [5](#0-4) 

---

### Impact Explanation

- **Direct theft of liquidation profit** from any liquidator who broadcasts to the public mempool. The attacker captures the full LIF-based spread without having identified the position themselves.
- **Systemic degradation of liquidation incentives**: if front-running is reliable, rational liquidators stop monitoring positions (their work is always stolen), leading to delayed liquidations, bad debt accumulation, and lender losses via the `lossFactor` socialization mechanism. [6](#0-5) 

---

### Likelihood Explanation

- **High.** The preconditions are minimal: any market without a `liquidatorGate` (the default) is vulnerable. The attacker needs only a mempool monitor and a simple flash-loan callback contract — both are standard MEV infrastructure. No privileged access is required. The native callback mechanism eliminates the need for external flash loan providers.

---

### Recommendation

1. **Bind `receiver` to `msg.sender`** — remove the `receiver` parameter from `liquidate` and always send seized collateral to `msg.sender`. Liquidators who need a different destination can wrap the call in their own contract.
2. **Alternatively, require `liquidatorGate` for markets where liquidation profit is significant** — market creators should deploy a gate that restricts liquidators to registered addresses, preventing anonymous front-running.
3. **Emit a `liquidator` field tied to `msg.sender`** in the event regardless, so off-chain monitoring can detect front-running patterns.

```
// Attacker contract (simplified)
contract FrontRunLiquidator is ILiquidateCallback {
    address immutable midnight;
    address immutable collateralToken;
    address immutable loanToken;

    function attack(
        Market calldata market,
        uint256 collateralIndex,
        uint256 seizedAssets,
        address borrower
    ) external {
        // Front-run with receiver = address(this), callback = address(this)
        IMidnight(midnight).liquidate(
            market, collateralIndex, seizedAssets, 0,
            borrower, false,
            address(this),   // <-- receiver: steal collateral here
            address(this),   // <-- callback: fund repayment from seized collateral
            ""
        );
        // Profit: collateral value - repaid loan value
        IERC20(collateralToken).transfer(msg.sender, IERC20(collateralToken).balanceOf(address(this)));
    }

    function onLiquidate(
        address, bytes32, Market memory market, uint256,
        uint256 seizedAssets, uint256 repaidUnits,
        address, address, bytes memory, uint256
    ) external returns (bytes32) {
        // collateral already received; sell it to repay
        // (swap collateralToken -> loanToken via DEX, then approve midnight)
        _swapAndApprove(collateralToken, loanToken, seizedAssets, repaidUnits);
        return CALLBACK_SUCCESS;
    }
}
```

**Steps:**
1. Deploy `FrontRunLiquidator`.
2. Monitor mempool for `liquidate(...)` calls targeting markets with `liquidatorGate == address(0)`.
3. Call `attack(...)` with higher gas, same borrower/amounts, `receiver = address(this)`.
4. Legitimate liquidator's transaction reverts with `NotLiquidatable`.
5. Attacker receives seized collateral minus repaid debt — the full liquidation spread. [7](#0-6) [3](#0-2) [8](#0-7)
