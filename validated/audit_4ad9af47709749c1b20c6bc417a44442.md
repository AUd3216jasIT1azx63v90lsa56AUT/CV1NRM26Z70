Audit Report

## Title
Rebasing Collateral Token Negative Rebase Inflates `_position.collateral` Accounting, Blocking Liquidation and Creating Bad Debt - (File: src/Midnight.sol)

## Summary
`supplyCollateral` writes the caller-supplied `assets` value directly into `_position.collateral[collateralIndex]` with no balance-before/after reconciliation. When a rebasing collateral token undergoes a negative rebase, the stored accounting exceeds the protocol's actual token holdings. `isHealthy` and `liquidate` both read the stale stored value, causing the position to appear healthy and blocking liquidators with `NotLiquidatable()`, while lenders absorb irrecoverable bad debt.

## Finding Description

**Root cause — `supplyCollateral` (src/Midnight.sol:533, 545)**

The accounting write at line 533 unconditionally sets `_position.collateral[collateralIndex] = oldCollateral + assets` using the caller-supplied `assets` parameter. The actual token transfer occurs at line 545 via `safeTransferFrom`. For a standard ERC-20 these are equal, but for a rebasing token the stored value permanently diverges from the real balance after any rebase event — there is no snapshot, no balance-diff check, and no hook to reconcile the stored value. [1](#0-0) 

**No token-type restriction in `touchMarket` (src/Midnight.sol:755-791)**

`touchMarket` validates LLTV tiers, `maxLif`, sorted order, and maturity, but imposes no restriction on the ERC-20 type of the collateral token. Any address satisfying the sort order is accepted, including rebasing tokens (stETH, aTokens, etc.). Market creation is permissionless — any unprivileged actor can create a market with a rebasing collateral token. [2](#0-1) 

**`isHealthy` uses stored accounting, not actual balance (src/Midnight.sol:944-960)**

`maxDebt` is computed from `_position.collateral[i]` — the stale stored value — multiplied by the oracle price and LLTV. After a negative rebase the stored value exceeds the real balance, so `maxDebt` is overestimated and the position appears healthy when it is actually insolvent. [3](#0-2) 

**`liquidate` uses the same stale value (src/Midnight.sol:612-624)**

The liquidatability check `originalDebt > maxDebt` at line 622 uses `_position.collateral[i]` read at line 612 to compute `maxDebt`. If the stored collateral is large enough to make `maxDebt >= debt`, the `NotLiquidatable()` revert fires and no liquidator can touch the position. [4](#0-3) 

**Exploit flow:**
1. Unprivileged actor calls `touchMarket` with a rebasing token (e.g., stETH) as `collateralParams[0].token` — no check prevents this.
2. Borrower calls `supplyCollateral(market, 0, C, borrower)` — `_position.collateral[0]` is set to `C`; Midnight holds `C` stETH.
3. Borrower borrows up to the LLTV limit: `debt = C * price * lltv / ORACLE_PRICE_SCALE`.
4. A negative rebase occurs (e.g., validator slashing): Midnight's actual stETH balance drops to `C' < C`; `_position.collateral[0]` remains `C`.
5. `isHealthy()` returns `true` because `maxDebt = C * price * lltv / ORACLE_PRICE_SCALE >= debt`.
6. Actual collateral value is only `C' * price * lltv / ORACLE_PRICE_SCALE < debt` — position is insolvent.
7. `liquidate` reverts with `NotLiquidatable()` because `debt <= maxDebt` (inflated).
8. Borrower walks away; lenders absorb the bad debt with no recourse.

**Existing protections are insufficient:** `SECURITY.md` excludes "depegging of an external stablecoin," but a negative rebase on stETH from validator slashing is a distinct mechanism — it is a protocol-level balance reduction, not a stablecoin peg failure. No documentation in the repository restricts rebasing tokens from being used as collateral, and no on-chain guard enforces such a restriction.

## Impact Explanation
The core accounting invariant `sum(_position.collateral[i]) <= actualTokenBalance` is violated. The borrower's position is reported healthy by `isHealthy()` while the actual collateral is insufficient to cover the debt. Liquidators are permanently blocked by `NotLiquidatable()`. The shortfall becomes irrecoverable bad debt absorbed by lenders, constituting direct, permanent loss of funds for an in-scope user class.

## Likelihood Explanation
All three preconditions are realistic and require no privileged access: (1) market creation with a rebasing token is permissionless via `touchMarket`; (2) borrowing near the LLTV limit is normal user behavior; (3) negative rebases on stETH (validator slashing) and aTokens (protocol losses) are documented, recurring external events. The scenario is repeatable for every market that lists a rebasing token and is independent of any malicious actor — a passive borrower benefits from the accounting divergence without taking any exploitative action.

## Recommendation
1. **Preferred — balance-diff accounting:** In `supplyCollateral`, record `balanceAfter - balanceBefore` rather than the caller-supplied `assets` value. This makes the stored collateral equal to tokens actually received, regardless of token type.
2. **Alternative — token allowlist:** Restrict collateral tokens in `touchMarket` to a protocol-maintained allowlist of audited, non-rebasing ERC-20s.
3. **Complementary — documentation:** If rebasing tokens are intentionally out of scope, enforce this on-chain or document it explicitly and add a revert in `touchMarket` for known rebasing token interfaces.

## Proof of Concept

**Minimal Foundry fork test outline:**
1. Deploy or fork with a mock rebasing ERC-20 (implements `rebase(int256 delta)` that adjusts all balances proportionally).
2. Call `touchMarket` with the rebasing token as `collateralParams[0].token` — confirm it succeeds.
3. Call `supplyCollateral(market, 0, 1000e18, borrower)` — confirm `_position.collateral[0] == 1000e18`.
4. Call `borrow` up to the LLTV limit.
5. Call `rebase(-100e18)` on the mock token to simulate a 10% negative rebase — Midnight's balance drops to `900e18`, stored value remains `1000e18`.
6. Assert `isHealthy(market, id, borrower) == true` (inflated `maxDebt`).
7. Attempt `liquidate` — assert it reverts with `NotLiquidatable()`.
8. Confirm `debt > actualCollateralValue * price * lltv / ORACLE_PRICE_SCALE` — bad debt is realized with no liquidation path.
