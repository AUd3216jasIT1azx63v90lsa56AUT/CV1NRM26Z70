### Title
Maker-Controlled `offer.callback` Executes Under Taker's `tx.origin`, Enabling Drain of Taker Funds via Third-Party `tx.origin`-Gated Contracts â€” (File: src/Midnight.sol)

---

### Summary

In `take()`, the maker's `offer.callback` is invoked mid-execution while `tx.origin` is still the taker's EOA. A malicious maker can point `offer.callback` at a contract that uses `tx.origin` to authorize withdrawals from any third-party protocol the taker holds funds in, draining those funds without the taker's consent.

---

### Finding Description

`take()` resolves the buyer and seller callbacks from the offer and the taker-supplied argument:

```solidity
// src/Midnight.sol L420-421
address buyerCallback = offer.buy ? offer.callback : takerCallback;
address sellerCallback = offer.buy ? takerCallback : offer.callback;
```

Depending on `offer.buy`, the maker's `offer.callback` is assigned to either `buyerCallback` or `sellerCallback`. Both are then invoked inside the taker's transaction:

```solidity
// src/Midnight.sol L445-453  (buyerCallback path)
if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback)
            .onBuy(id, offer.market, buyerAssets, units, buyerPendingFeeIncrease, buyer, buyerCallbackData)
        == CALLBACK_SUCCESS, ...
    );
}

// src/Midnight.sol L458-474  (sellerCallback path)
if (sellerCallback != address(0)) {
    require(
        ISellCallback(sellerCallback)
            .onSell(id, offer.market, sellerAssets, units, sellerPendingFeeDecrease, seller, receiver, sellerCallbackData)
        == CALLBACK_SUCCESS, ...
    );
}
```

Throughout both calls, `tx.origin` equals the taker's EOA â€” the address that submitted the original transaction. There is no restriction on what `offer.callback` may do during its execution. A malicious maker deploys a callback that, upon `onBuy`/`onSell`, calls into any third-party contract that gates withdrawals or approvals on `tx.origin` (e.g., legacy wallet contracts, certain DeFi vaults, permit-style contracts). Because `tx.origin` is the taker, those contracts treat the call as originating from the taker and execute the privileged action â€” draining the taker's balance â€” without the taker's knowledge.

The same surface exists for the ratifier call at line 356:

```solidity
// src/Midnight.sol L355-356
require(isAuthorized[offer.maker][offer.ratifier], RatifierUnauthorized());
require(IRatifier(offer.ratifier).isRatified(offer, ratifierData) == CALLBACK_SUCCESS, RatifierFail());
```

The ratifier is maker-authorized and maker-controlled; it is also called under the taker's `tx.origin`.

---

### Impact Explanation

Any taker whose EOA holds a balance or approval in a third-party contract that uses `tx.origin` for authorization can have those funds drained in a single `take()` call. The taker receives no warning; the protocol enforces no restriction on callback behavior. This is a direct, irreversible loss of user funds external to the Midnight protocol itself.

---

### Likelihood Explanation

- `tx.origin`-gated contracts exist in production (legacy multisigs, certain vault designs, permit-style wrappers).
- Market creation and offer creation are both permissionless; a malicious maker has no barrier to entry.
- The taker can inspect `offer.callback` off-chain, but the callback can be a proxy whose implementation is swapped between inspection and execution, or the taker may simply not audit the address.
- The attack requires no privileged access, no oracle manipulation, and no governance capture â€” only a crafted offer and a taker who executes it.

---

### Recommendation

Document explicitly that takers' EOAs should be single-purpose addresses holding no balances or authorizations in third-party contracts. Warn integrators and end-users that `offer.callback` (and `offer.ratifier`) execute arbitrary code under the taker's `tx.origin`. Optionally, consider emitting the callback address in the `Take` event so off-chain tooling can flag unknown or unverified callbacks before execution.

---

### Proof of Concept

```solidity
// Attacker-deployed callback
contract MaliciousCallback is IBuyCallback {
    address immutable victim;        // taker's EOA
    IVault  immutable legacyVault;   // third-party contract gated on tx.origin

    function onBuy(bytes32, Market memory, uint256, uint256, uint256, address, bytes memory)
        external returns (bytes32)
    {
        // tx.origin == taker's EOA here
        legacyVault.withdrawAll(tx.origin); // drains taker's balance in legacyVault
        return CALLBACK_SUCCESS;
    }
}
```

**Steps:**
1. Attacker deploys `MaliciousCallback` referencing a `legacyVault` that gates `withdrawAll` on `tx.origin`.
2. Attacker creates an attractive buy offer (`offer.buy = true`) with `offer.callback = address(MaliciousCallback)`.
3. Taker calls `take(offer, ...)`.
4. `take()` invokes `MaliciousCallback.onBuy(...)` at `src/Midnight.sol L447-452`.
5. Inside `onBuy`, `tx.origin == taker's EOA`; `legacyVault.withdrawAll(tx.origin)` succeeds and transfers the taker's vault balance to the attacker.
6. `take()` completes normally; the taker's Midnight position is unaffected but their external funds are gone. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
