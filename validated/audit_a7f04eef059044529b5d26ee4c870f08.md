### Title
Lenders Can Frontrun Bad Debt Realization to Avoid Proportional Loss — (`src/Midnight.sol`)

### Summary

The `lossFactor` in Morpho Midnight is updated lazily: it only increases when `liquidate()` is called and bad debt is realized. The `withdraw()` function applies only the **current** (pre-liquidation) `lossFactor` when computing a lender's credit. A lender who calls `withdraw()` before `liquidate()` is called on an undercollateralized borrower escapes their proportional share of the bad debt. The remaining lenders absorb a disproportionately larger loss — in the extreme case, a single remaining lender absorbs the entire bad debt.

### Finding Description

**Vulnerability type**: Accounting / state-transition race (frontrunning)

**Root cause**: `lossFactor` is a lazy, market-wide accumulator. It is only updated inside `liquidate()` when `badDebt > 0`. The `withdraw()` function calls `_updatePosition()`, which reads the **current** `lossFactor` from storage: [1](#0-0) 

If `liquidate()` has not yet been called, `marketState[id].lossFactor` still holds the pre-bad-debt value, so `postSlashCredit == credit` — no loss is applied. The lender then withdraws their full credit: [2](#0-1) 

The `withdrawable` pool (funded by prior repayments) is the only constraint on how much can be withdrawn, mirroring the "deposit pool" constraint in the Rio report.

Bad debt is realized in `liquidate()` only after the call: [3](#0-2) 

Because `liquidate()` is permissionless, any party can call it. This creates a race: a lender who observes an undercollateralized borrower (oracle prices are public) can call `withdraw()` in the same block or earlier, before any liquidator calls `liquidate()`.

**End-to-end exploit path**:

1. Oracle price drops; borrower's collateral value (at `maxLif`) falls below their debt — bad debt is now latent but not yet realized on-chain.
2. Attacker (lender) calls `withdraw(market, units, self, self)` — `_updatePosition()` reads the stale `lossFactor`, credit is unchanged, and `units` loan tokens are transferred out.
3. Liquidator (or anyone) calls `liquidate()` — `lossFactor` increases, `totalUnits` decreases.
4. Remaining lenders call `_updatePosition()` at their next interaction and have their credit slashed proportionally to the **full** bad debt, even though one lender already exited at full value.

### Impact Explanation

- The withdrawing lender receives full credit value, avoiding their proportional share of bad debt.
- Remaining lenders bear a larger-than-proportional loss. If all but one lender frontrun, the last lender absorbs 100% of the bad debt.
- This directly violates the core protocol invariant stated in `live_context.json`: *"bad debt must reduce lender credit exactly once and proportionally."* [4](#0-3) 

The loss is bounded by the available `withdrawable` liquidity (repaid loan tokens), which is the same constraint as the Rio deposit pool. In active markets with regular repayments, this pool can be substantial.

### Likelihood Explanation

- Bad debt events are realistic in any collateralized lending protocol (sharp oracle price drops, cascading liquidations).
- `liquidate()` is permissionless — the race condition is real and observable on-chain.
- Oracle prices are public; any lender or MEV bot can compute when a borrower's collateral value (at `maxLif`) falls below their debt before `liquidate()` is called.
- No privileged access is required. The attacker only needs to be a lender with credit in the market.

### Recommendation

Apply the bad debt loss **before** allowing withdrawal, or introduce a two-step withdrawal mechanism with a delay. The simplest fix is to require that any position with pending bad debt (i.e., `marketState[id].lossFactor > position[id][user].lastLossFactor`) has `_updatePosition()` applied — which `withdraw()` already does — but additionally ensure that any known-undercollateralized borrower's bad debt is realized (via a forced `liquidate()` call or a snapshot mechanism) before lenders can withdraw. Alternatively, track a "pending bad debt" accumulator that is applied to withdrawals even before `liquidate()` is called.

### Proof of Concept

```
Setup:
  - Market with loanToken T, one collateral C, LLTV=0.8
  - Alice lends 100 T (credit=100, totalUnits=100)
  - Bob borrows 80 T against 100 C (debt=80)
  - Bob repays 20 T → withdrawable=20, debt=60
  - Oracle price drops: C now worth 50 T → Bob's maxDebt = 50*0.8 = 40 < 60 debt
    → badDebt = 60 - 50/maxLif > 0 (latent, not yet realized)

Attack:
  1. Alice calls withdraw(market, 20, alice, alice)
     - _updatePosition: lossFactor unchanged → credit stays 100
     - credit -= 20 → credit=80
     - withdrawable -= 20 → withdrawable=0
     - Alice receives 20 T (full value, no loss)

  2. Liquidator calls liquidate(market, ..., bob, ...)
     - badDebt realized → lossFactor increases
     - totalUnits reduced by badDebt
     - Alice's remaining credit=80 is now slashed by the full bad debt
       even though she already withdrew 20 T at full value

Result:
  Alice avoided her proportional share of the bad debt on the 20 T she withdrew.
  The loss is concentrated on her remaining credit and any other lenders.
```
