Audit Report

## Title
Zero-price oracle allows collateral seizure with zero debt repayment via `seizedAssets` input path - (`src/Midnight.sol`)

## Summary
When the oracle for the liquidated collateral returns exactly 0, calling `liquidate` with `seizedAssets > 0` and `repaidUnits = 0` causes `repaidUnits` to be silently computed as 0 at line 650 without reverting. The function then transfers `seizedAssets` collateral tokens to the receiver while subtracting 0 from `_position.debt`, violating the invariant that collateral seizure must correspond to proportional debt repayment. The symmetric `repaidUnits > 0` input path correctly reverts via division-by-zero at line 652, but the `seizedAssets > 0` path has no equivalent guard and is explicitly excluded from formal verification coverage.

## Finding Description

**Root cause — missing post-computation zero check at line 650:**

`mulDivUp` is defined as `(x * y + (d - 1)) / d`.

At line 650 with `liquidatedCollatPrice = 0`:
```
seizedAssets.mulDivUp(0, ORACLE_PRICE_SCALE)
  = (seizedAssets * 0 + (1e36 - 1)) / 1e36
  = (1e36 - 1) / 1e36
  = 0   ← integer division, no revert
```
Then:
```
0.mulDivUp(WAD, lif)
  = (0 * WAD + (lif - 1)) / lif
  = (lif - 1) / lif
  = 0   ← since lif >= WAD = 1e18 >> 1, no revert
```

So `repaidUnits = 0` silently. The `mulDivUp` implementation at `src/libraries/UtilsLib.sol:34-36` confirms this: `(x * y + (d - 1)) / d` with `y = 0` and large `d` always yields 0.

**Contrast with the `repaidUnits > 0` path (line 652):**
```solidity
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
```
`mulDivDown(ORACLE_PRICE_SCALE, 0)` → `ORACLE_PRICE_SCALE / 0` → division-by-zero → reverts. This path is correctly blocked, and `Reverts.spec` lines 246–253 formally proves it. The `seizedAssets > 0` path has no analogous protection.

**Existing checks reviewed and shown insufficient:**
- `atMostOneNonZero` (line 595): checks input values only, not computed values.
- `_position.debt > 0` (line 596): checked before bad debt realization; passes trivially.
- `NotLiquidatable` (line 620–624): passes because price=0 makes `maxDebt` contribution from this collateral = 0, so `originalDebt > maxDebt`.
- RCF check (lines 662–667): `repaidUnits = 0 <= maxRepaid` always passes trivially.
- No check anywhere that `repaidUnits > 0` after computation when `seizedAssets > 0`.

**Exploit flow — multi-collateral variant (most impactful, no bad debt realization):**

1. Position has collateral 0 (oracle returns 0) and collateral 1 (oracle returns non-zero price P1). Position is unhealthy: `originalDebt > maxDebt` (driven by collateral 1's price drop).
2. `badDebt = 0` because collateral 1's value covers the bad-debt threshold. `_position.debt` is NOT zeroed.
3. Attacker calls `liquidate(market, 0, seizedAssets>0, 0, borrower, false, receiver, address(0), "")`.
4. `repaidUnits` computed as 0 (shown above).
5. Line 670: `_position.collateral[0] -= seizedAssets` — collateral 0 reduced.
6. Line 675: `_marketState.withdrawable += 0`.
7. Line 676: `_position.debt -= 0` — debt **unchanged**.
8. Line 696: `safeTransfer(collateralToken0, receiver, seizedAssets)` — collateral transferred to attacker.
9. Line 717: `safeTransferFrom(loanToken, payer, address(this), 0)` — zero loan tokens paid.

Borrower loses real collateral tokens with zero debt reduction.

**Single-collateral variant:** With price=0 and a single collateral, `badDebt = originalDebt`, so `_position.debt` is zeroed at line 628 before the seizure block. The seizure block still executes (line 643: `seizedAssets > 0`), seizes collateral, and subtracts 0 from an already-zero debt. The attacker front-runs the borrower's `withdrawCollateral` (confirmed valid by `testFullBadDebtWithdrawCollateral` at line 870–892), stealing collateral tokens the borrower would otherwise recover.

**Formal verification gap confirmed:**
- `NoDivisionByZero.spec` line 124 explicitly requires `ghostPrice > 0` as a precondition, acknowledging the zero-price case is unverified.
- `Reverts.spec` lines 246–253 only proves revert for the `repaidUnits > 0` input; no analogous rule exists for `seizedAssets > 0`.
- `LiquidationBoundedByLIF.spec` lines 92–109 (`liquidationProfitBoundedSeizedAssets`) asserts `seizedResult * price * WAD <= repaidResult * ORACLE_PRICE_SCALE * maxLif`; with `price = 0` and `repaidResult = 0`, this trivially holds (0 ≤ 0), masking the invariant violation.

## Impact Explanation

An unprivileged liquidator seizes an arbitrary amount of collateral tokens (up to `_position.collateral[collateralIndex]`) while transferring zero loan tokens. In the multi-collateral case, `_position.debt` is completely unchanged after the call — the borrower loses real collateral with no debt reduction, a direct theft of collateral tokens. In the single-collateral case, the attacker steals collateral the borrower would otherwise recover via `withdrawCollateral`. The protocol's core invariant — that collateral seizure corresponds to proportional debt repayment — is broken. The attack is repeatable as long as the oracle returns 0 and collateral remains in the position.

## Likelihood Explanation

**Preconditions:**
1. A market collateral's oracle returns 0. The protocol explicitly handles this as a valid scenario: `testFullBadDebtWithdrawCollateral` (line 875) sets price to 0 and calls `liquidate`; `oracleZeroCausesIsHealthyReturnFalse` formally proves `isHealthy` returns false at price=0. This is not oracle failure — it is a valid price for a worthless or depegged asset.
2. The borrower must have `_position.debt > 0` (trivially satisfied for any active borrower).
3. The position must be liquidatable (`originalDebt > maxDebt` or post-maturity). Price=0 guarantees this for any single-collateral position; for multi-collateral, the position must already be unhealthy on other collaterals.

**Feasibility:** Any unprivileged liquidator can execute this in a single external transaction. No special role, signature, or governance action is required. The call is repeatable as long as the oracle returns 0 and collateral remains.

## Recommendation

Add a guard after computing `repaidUnits` in the `seizedAssets > 0` branch to revert when the computed `repaidUnits` is 0 but `seizedAssets > 0`. The simplest fix is to require `liquidatedCollatPrice > 0` before entering the seizure computation block when `seizedAssets > 0`, mirroring the implicit protection that the `repaidUnits > 0` path already has via division-by-zero:

```solidity
if (seizedAssets > 0) {
    require(liquidatedCollatPrice > 0, ZeroPriceLiquidation());
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
} else {
    seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
}
```

Alternatively, add `require(repaidUnits > 0)` after the computation. The formal verification rule `noDivisionByZeroLiquidate` in `NoDivisionByZero.spec` should also be updated to remove the `ghostPrice > 0` precondition and instead prove that the function reverts when price=0 and `seizedAssets > 0`.
