Audit Report

## Title
Off-by-one `<=` in maturity debt-increase guard allows forced debt creation at exact maturity timestamp - (File: `src/Midnight.sol`)

## Summary
The `take()` function guards against post-maturity debt increases using `block.timestamp <= offer.market.maturity` (line 391), which permits `sellerDebtIncrease > 0` when `block.timestamp == offer.market.maturity`. The post-maturity liquidation gate uses the strict `block.timestamp > market.maturity` (line 622), creating a one-second window where debt can be forced onto a seller who cannot yet be liquidated in post-maturity mode. One block later, the position is liquidatable via post-maturity mode even though it is collateral-healthy, causing the borrower to lose collateral to a liquidator.

## Finding Description

**Root cause — asymmetric boundary operators:**

`take()`, line 391:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
When `block.timestamp == offer.market.maturity`, the left operand is `true`, so the `require` passes unconditionally regardless of `sellerDebtIncrease`. Debt is then written to storage at line 414:
```solidity
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
```

The end-of-take health check at line 476:
```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```
`isHealthy` (line 944–960) returns `maxDebt >= debt` — a purely collateral-coverage check with no maturity component. A collateral-healthy seller passes this check even after debt is forced onto them at maturity.

`liquidate()`, line 620–624, uses strict `>`:
```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```
At `block.timestamp == maturity` this evaluates to `false` for `postMaturityMode=true`, so `NotLiquidatable()` reverts. At `block.timestamp == maturity + 1` it evaluates to `true`.

This asymmetry is confirmed by the existing test at `test/LiquidationTest.sol` lines 170–177, which explicitly documents that post-maturity liquidation is unavailable at exact maturity but available one second later. However, no test covers the case of debt increase at exact maturity.

**Why existing checks fail:**
- `CannotIncreaseDebtPostMaturity` uses `<=` instead of `<`, creating a one-second window at the exact maturity timestamp.
- The `SellerIsLiquidatable` check at line 476 only enforces collateral health, not maturity-based liquidatability.
- The protocol's own invariant document (`live_context.json`, line 221) explicitly lists this as forbidden: `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"`.
- The `credit_debt` invariant (`live_context.json`, line 197) states `"debt must not increase after maturity"`. The maturity timestamp itself is "after maturity" in the sense that the debt obligation has expired.

**Exploit flow:**
1. A sell offer exists with `expiry >= maturity` and the seller has no existing credit to absorb units (so `sellerDebtIncrease > 0`).
2. Taker waits until `block.timestamp == offer.market.maturity`.
3. Taker calls `take()` with `units` such that `sellerDebtIncrease > 0`.
4. Line 391: `block.timestamp <= maturity` → `true` → passes. Seller's debt increases. Health check at line 476 passes (seller is collateral-healthy).
5. In the next block (`block.timestamp == maturity + 1`), taker calls `liquidate(..., postMaturityMode=true)`.
6. Line 622: `block.timestamp > market.maturity` → `true` → liquidation proceeds on a collateral-healthy position.
7. LIF at `maturity + 1` = `WAD + (maxLif - WAD) * 1 / TIME_TO_MAX_LIF` (just above 1, ramping over 15 minutes per line 646).

## Impact Explanation

A taker can force a collateral-healthy borrower to take on new debt at the exact maturity second. In the immediately following block, the borrower's position is liquidatable via post-maturity mode even though it is collateral-healthy (`live_context.json` line 88: `"overdue_debt_after_maturity_is_liquidatable_even_if_healthy": true`). The borrower has no opportunity to repay between debt creation and liquidation (consecutive blocks). The liquidator seizes collateral at LIF > 1 (even at `maturity + 1`, LIF = `WAD + (maxLif - WAD)/900`), causing direct loss of collateral value beyond the debt repaid. This matches the `best_bug_classes` entries `"healthy-account liquidation"` and `"unauthorized collateral seizure"`.

## Likelihood Explanation

**Preconditions:**
- A sell offer with `expiry >= maturity` exists (common for offers meant to be filled up to maturity).
- The seller has no existing credit to absorb the units (`sellerCreditDecrease < units`).
- The seller has sufficient collateral to pass the health check after the debt increase.

**Feasibility:** On chains with 1-second block times (post-merge Ethereum, Arbitrum, Base), landing a transaction at exactly `block.timestamp == maturity` is achievable via MEV or simple block monitoring. The attacker (taker) controls the timing. The attack is repeatable for any market where a qualifying sell offer exists. No privileged access is required — the attacker is a normal unprivileged taker.

## Recommendation

Change the guard in `take()` from `<=` to `<` so that debt cannot increase at the exact maturity timestamp:

```solidity
// Before (line 391):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After:
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the debt-increase boundary with the post-maturity liquidation boundary (`block.timestamp > market.maturity`), closing the one-second window. The fix is consistent with the protocol invariant that explicitly forbids debt increase by "timestamp equality."

## Proof of Concept

Minimal Foundry test plan:

```solidity
function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;
    // Warp to exact maturity
    vm.warp(market.maturity);
    borrowerOffer.expiry = market.maturity; // offer valid at maturity
    borrowerOffer.maxUnits = units;
    deal(address(loanToken), lender, units);
    collateralize(market, borrower, units); // enough collateral to be healthy

    // This should revert with CannotIncreaseDebtPostMaturity but currently does NOT
    take(units, lender, borrowerOffer);
    assertGt(midnight.debtOf(id, borrower), 0, "debt was created at maturity");

    // One block later: post-maturity liquidation succeeds on a collateral-healthy position
    vm.warp(market.maturity + 1);
    midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");
}
```

Expected (buggy) behavior: `take()` succeeds at `block.timestamp == maturity` with `sellerDebtIncrease > 0`, and `liquidate(..., postMaturityMode=true)` succeeds at `maturity + 1` on a collateral-healthy position.

Expected (fixed) behavior: `take()` reverts with `CannotIncreaseDebtPostMaturity` at `block.timestamp == maturity` when `sellerDebtIncrease > 0`.
