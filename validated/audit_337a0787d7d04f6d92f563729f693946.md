### Title
Griefer can front-run `EcrecoverAuthorizer::setIsAuthorized` to grief atomic bundles and relayer transactions — (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer::setIsAuthorized` is permissionless and uses a sequential nonce. A griefer can observe a pending transaction in the mempool, front-run it with the identical parameters, and cause the original transaction to revert with `InvalidNonce()`. When the authorization call is embedded in an atomic multicall or `MidnightBundles` bundle alongside downstream protocol actions, the entire bundle reverts — blocking the user from completing their intended operation in that transaction.

### Finding Description
The root cause is in `EcrecoverAuthorizer::setIsAuthorized`:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // @audit nonce consumed on first call
    ...
    IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
}
``` [1](#0-0) 

There is no `msg.sender` restriction — the function is explicitly permissionless by design (confirmed by the test comment `// Anyone can submit — no caller auth needed`). [2](#0-1) 

This creates the identical front-running vector as the EIP-2612 permit vulnerability described in the external report:

1. User signs an `Authorization` struct and submits (or has a relayer submit) a transaction
2. Griefer observes the mempool, extracts the signature and authorization data
3. Griefer front-runs by calling `setIsAuthorized` with the same parameters
4. Nonce is consumed; authorization IS set correctly to the intended value
5. The original transaction reverts with `InvalidNonce()`

For a standalone `setIsAuthorized` call, the authorization is already set and the user only loses gas. However, the critical impact path is when the call is embedded in an atomic `Midnight::multicall` or a `MidnightBundles` bundle that combines the authorization step with subsequent protocol actions (`take`, `repay`, `supplyCollateral`, etc.). [3](#0-2) 

In that case, the entire bundle reverts at step (a), and the downstream protocol operations never execute.

### Impact Explanation
- Any atomic bundle combining `EcrecoverAuthorizer::setIsAuthorized` with downstream protocol calls (`take`, `repay`, `withdraw`, etc.) reverts entirely when front-run
- The griefer can persistently repeat this on every retry that includes the authorization step in the same bundle, creating a sustained DoS
- Users relying on gasless/meta-transaction flows via relayers suffer repeated failed transactions and wasted gas
- No funds are directly stolen, but the gasless authorization pattern is rendered unreliable

### Likelihood Explanation
- Requires only mempool observation — no privileged access, no capital, no special role
- The function is explicitly permissionless, making the attack trivially executable by any external actor
- All chains with public mempools (Ethereum mainnet, most L2s) are fully exposed
- Likelihood is high for any user relying on the gasless authorization pattern in a bundle

### Recommendation
Mirror the pattern already used in `MidnightBundles::pullToken` for ERC-2612 permits, which wraps the `permit` call in a try-catch and tolerates a revert if a third party already consumed the permit: [4](#0-3) 

Apply the same pattern when `setIsAuthorized` is used inside a bundle:

```solidity
try ecrecoverAuthorizer.setIsAuthorized(authorization, signature) {} catch {
    // Tolerate revert: a third party may have already consumed the authorization.
    // Verify the desired authorization state is already set.
    require(
        IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, authorization.authorized)
            == authorization.isAuthorized,
        "Authorization not set as expected"
    );
}
```

### Proof of Concept

1. Alice signs `Authorization{authorizer: alice, authorized: bundler, isAuthorized: true, nonce: 0, deadline: T+1h}`
2. Alice (or a relayer) submits a `Midnight::multicall` containing:
   - Call A: `ecrecoverAuthorizer.setIsAuthorized(auth, sig)`
   - Call B: `midnight.take(offer, ...)` — requires `bundler` to be authorized
3. Bob (griefer) observes the mempool, extracts `auth` and `sig`
4. Bob calls `EcrecoverAuthorizer.setIsAuthorized(auth, sig)` directly — succeeds; nonce becomes 1; authorization is set
5. Alice's multicall executes: Call A reverts with `InvalidNonce()` (nonce is now 1, not 0)
6. `Midnight::multicall` propagates the revert — Call B never executes
7. Bob repeats on every retry that bundles the authorization step, persistently blocking Alice's atomic operation [5](#0-4) [6](#0-5)

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()

            authorization.isAuthorized,
            authorization.nonce
        );

    }
