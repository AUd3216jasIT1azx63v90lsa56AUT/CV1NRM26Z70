Audit Report

## Title
Zero-`buyerAssets` rounding in `maxAssets`-mode buy-offer consumed accounting allows indefinite offer reuse — (File: src/Midnight.sol)

## Summary
When a buy offer is configured with `offer.maxAssets > 0`, the `take` function tracks capacity consumption by incrementing `consumed[maker][group]` by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `tickToPrice` always returns a value strictly less than `WAD` for every valid tick, filling with `units = 1` produces `buyerAssets = 0`, so `consumed` never advances. The `ConsumedAssets` guard is therefore never triggered, and the offer can be filled an unlimited number of times, allowing the maker to accumulate unbounded credit units while transferring zero loan tokens.

## Finding Description

**Root cause — `src/Midnight.sol` lines 361–369:**

`sellerPrice` and `buyerPrice` are both derived from `tickToPrice(offer.tick)`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice = sellerPrice + _settlementFee;
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);
``` [1](#0-0) 

For `offer.buy = true`, `buyerPrice = (offerPrice - _settlementFee) + _settlementFee = offerPrice = tickToPrice(offer.tick)`.

**`tickToPrice` always returns strictly less than `WAD`:**

```solidity
function tickToPrice(uint256 tick) internal pure returns (uint256) {
    require(tick <= MAX_TICK, TickOutOfRange());
    unchecked {
        return uint256(1e36)
            .divHalfDownUnchecked(1e18 + wExp(LN_ONE_PLUS_DELTA * (int256(MAX_TICK / 2) - int256(tick))))
            .divHalfDownUnchecked(PRICE_ROUNDING_STEP) * PRICE_ROUNDING_STEP;
    }
}
``` [2](#0-1) 

The denominator `1e18 + wExp(...)` is always strictly greater than `1e18` for every valid tick (0 through `MAX_TICK = 5820`), so `1e36 / (1e18 + wExp(...)) < 1e18 = WAD`. After rounding to `PRICE_ROUNDING_STEP = 1e12`, the result is at most `WAD - 1e12`.

Therefore, for any valid tick with `units = 1`:
```
buyerAssets = mulDivDown(1, buyerPrice, 1e18) = (1 * buyerPrice) / 1e18 = 0
``` [3](#0-2) 

**The `maxAssets` guard is never triggered:**

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [4](#0-3) 

With `buyerAssets = 0`, `consumed[maker][group] += 0` stays permanently at 0, and `require(0 <= maxAssets)` always passes regardless of `maxAssets`.

**Credit accumulates with each fill:**

```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt);
// ...
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
``` [5](#0-4) 

Each call with `units = 1` gives the maker +1 credit and the taker +1 debt, while both `safeTransferFrom` calls transfer 0 tokens (which succeed per the protocol's own token safety requirements at line 136). [6](#0-5) 

**No guard prevents this:** There is no `require(units > 0)`, no `require(buyerAssets > 0)`, and no minimum-units check anywhere in the `take` path.

**The protocol's own comments acknowledge this behavior:**

```
/// @dev It is possible to give units to a fully consumed assets-based buy offer with price < 1.
``` [7](#0-6) 

This comment acknowledges the rounding behavior but does not assert it is safe or bounded.

**Existing checks are insufficient:**
- `require(newConsumed <= offer.maxAssets)` — `newConsumed` is always 0, so this never fires. [8](#0-7) 
- `require(offer.maker != taker, SelfTake())` — bypassed by using a second address. [9](#0-8) 
- `isHealthy` check at line 476 — only constrains the taker's collateral capacity, not the number of fills. [10](#0-9) 

## Impact Explanation

The maker accumulates N units of credit while paying zero loan tokens. At maturity, the maker withdraws N loan tokens from the pool. The taker's N units of debt, if unpaid, become bad debt socialized among all lenders in the market. This constitutes direct theft from the protocol's loan token pool: the maker extracts loan tokens that legitimate lenders deposited, with the loss distributed across all lenders via the `lossFactor` slashing mechanism. [11](#0-10) 

## Likelihood Explanation

**Preconditions:**
- Any buy offer with `maxAssets > 0` at any valid tick (all 5821 ticks are affected).
- A second address controlled by the maker to act as taker, supplying enough collateral to remain healthy under accumulated debt.
- No special permissions, governance access, or oracle manipulation required.

**Feasibility:** Trivially repeatable in a loop. The only cost is gas and the collateral the taker must lock. The number of iterations is bounded only by the taker's collateral capacity (which can be made arbitrarily large). The maker's credit gain is unbounded relative to loan tokens paid.

## Recommendation

Add a minimum-assets guard in the `maxAssets` branch of `take`. Specifically, when `offer.maxAssets > 0` and `offer.buy = true`, require that `buyerAssets > 0` before proceeding:

```solidity
if (offer.maxAssets > 0) {
    require(offer.buy ? buyerAssets > 0 : sellerAssets > 0, ZeroAssets());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` value such that `mulDivDown(units, buyerPrice, WAD) >= 1`, i.e., `units >= WAD / buyerPrice + 1`.

## Proof of Concept

1. Deploy a market with any valid collateral and loan token.
2. Maker creates a buy offer: `offer.buy = true`, `offer.maxAssets = 1e18` (any `M > 0`), `offer.tick = MAX_TICK` (or any valid tick), valid `expiry`, authorized `ratifier`.
3. Taker (maker's second address) supplies sufficient collateral to remain healthy.
4. Taker calls `take(offer, ..., units=1, ...)` in a loop N times.
5. Assert after each iteration: `consumed[maker][group] == 0`, `position[id][maker].credit == N`, `position[id][taker].debt == N`, loan token balance of contract unchanged.
6. After N iterations, maker calls `withdraw` once sufficient `withdrawable` exists (e.g., after taker repays or other borrowers repay), extracting N loan tokens that were never paid in.
