### Title
Borrower Can DoS Liquidation by Frontrunning with a Minimal Repayment - (File: src/Midnight.sol)

### Summary

The `liquidate()` function in `src/Midnight.sol` accepts a caller-supplied `repaidUnits` parameter and applies it directly against `_position.debt` without capping it to the current on-chain debt. Because Midnight exposes a standalone `repay()` function that any authorized party can call to reduce a borrower's debt, a borrower can frontrun a liquidation transaction by repaying as little as 1 unit, causing the liquidator's transaction to revert while the position remains liquidatable.

### Finding Description

The root cause is a TOCTOU (Time-of-Check to Time-of-Use) gap between when the liquidator reads `_position.debt` off-chain and when the liquidation transaction executes on-chain.

**Entry path for the borrower's frontrun:** `midnight.repay(market, 1, borrower, address(0), hex"")` — confirmed available and callable by any authorized address, reducing `_position.debt` by the supplied amount.

**Post-maturity attack path (arithmetic underflow):**

In post-maturity mode (`postMaturityMode = true`), there is no Recovery Close Factor (RCF) cap. The liquidator legitimately reads `_position.debt = D` and submits `liquidate(..., repaidUnits = D, postMaturityMode = true)`. The borrower frontruns with `repay(..., 1, ...)`, making `_position.debt = D - 1`. When the liquidator's transaction executes, the subtraction at line 676:

```solidity
_position.debt -= UtilsLib.toUint128(repaidUnits);  // D - 1 - D → underflow
```

reverts with an arithmetic error. The position is still liquidatable (still post-maturity, `_position.debt > 0`), but the liquidator's transaction fails.

This is confirmed by `testCannotRepayMoreThanDebt` in `test/LiquidationTest.sol` which explicitly expects `stdError.arithmeticError` when `repaidUnits > debt`.

**Normal mode (pre-maturity) attack path (RCF revert):**

In normal mode, the liquidator computes `maxRepaid` off-chain and submits `repaidUnits = maxRepaid`. The borrower frontruns with `repay(..., 1, ...)`. On execution, `_position.debt` is now `D - 1`, so the on-chain `maxRepaid` recalculates to a smaller value:

```solidity
uint256 maxRepaid = lltv < WAD
    ? (_position.debt - maxDebt).mulDivUp(WAD * WAD, WAD * WAD - lif * lltv)
    : type(uint256).max;
require(
    repaidUnits <= maxRepaid || ...,
    RecoveryCloseFactorConditionsViolated()
);
```

Since `repaidUnits` (the original `maxRepaid`) now exceeds the freshly computed `maxRepaid`, the transaction reverts with `RecoveryCloseFactorConditionsViolated()`. The position is still unhealthy and liquidatable.

### Impact Explanation

Liquidators attempting to liquidate the full debt (post-maturity) or the maximum RCF-allowed amount (normal mode) are denied. The borrower can sustain this DoS by spending gas on minimal repayments (1 unit = 1 loan token at face value, a negligible cost for large positions). This delays or prevents liquidation, allowing an underwater position to accumulate further bad debt that is socialized across all lenders via the `lossFactor` mechanism. Repeated DoS across multiple liquidators degrades protocol solvency guarantees.

### Likelihood Explanation

- The borrower has a direct economic incentive to delay liquidation (preserving their collateral).
- The `repay()` function is permissionless for the borrower and requires only 1 unit of loan token.
- On Ethereum's public mempool, frontrunning is straightforward with standard gas-price bumping.
- The attack must be repeated for each liquidator attempt, but the cost per frontrun (1 unit + gas) is trivially small relative to the collateral at stake.
- Likelihood is **Medium**: frontrunning is feasible on public mempools; liquidators using private relays (Flashbots) are immune, but not all liquidators do so.

### Recommendation

Cap `repaidUnits` to the current `_position.debt` before applying the RCF check and the debt subtraction, rather than reverting when the caller-supplied value exceeds the live debt:

```solidity
// After bad-debt subtraction, before the RCF block:
if (repaidUnits > _position.debt) {
    repaidUnits = _position.debt;
    // Recompute seizedAssets from the capped repaidUnits
    seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
}
```

This mirrors the mitigation recommended in the Venus M-08 report: instead of reverting when the supplied amount exceeds the current maximum, silently clamp to the achievable maximum and proceed.

### Proof of Concept

**Setup:** Market is post-maturity. Borrower has `debt = 100e18` units. Liquidator intends to liquidate the full position.

1. Liquidator reads `midnight.debtOf(id, borrower) = 100e18` off-chain.
2. Liquidator submits: `midnight.liquidate(market, 0, 0, 100e18, borrower, true, receiver, address(0), "")`.
3. Borrower observes the pending tx in the mempool and submits with higher gas: `midnight.repay(market, 1, borrower, address(0), hex"")`.
4. Borrower's repay executes first → `_position.debt = 100e18 - 1`.
5. Liquidator's tx executes: reaches line 676 `_position.debt -= 100e18` → `(100e18 - 1) - 100e18` → arithmetic underflow → **revert**.
6. Position remains liquidatable (`block.timestamp > market.maturity`, `_position.debt = 100e18 - 1 > 0`).
7. Borrower repeats step 3 for every subsequent liquidation attempt.

        uint256 collateral = midnight.collateral(id, borrower, 0);

        // Price must be high enough that seized assets for (units + 1) don't exceed available collateral.
        uint256 minPrice = (units + 1).mulDivUp(_maxLif, WAD).mulDivUp(ORACLE_PRICE_SCALE, collateral);
        liquidationOraclePrice = bound(liquidationOraclePrice, minPrice, ORACLE_PRICE_SCALE);
        Oracle(market.collateralParams[0].oracle).setPrice(liquidationOraclePrice);

        midnight.liquidate(market, 0, 0, repaid, borrower, true, address(this), address(0), "");
