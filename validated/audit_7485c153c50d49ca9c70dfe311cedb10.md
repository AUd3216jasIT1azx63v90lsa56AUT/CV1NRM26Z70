Audit Report

## Title
Front-running victim's EcrecoverAuthorizer revocation with pre-signed authorization grants unauthorized Midnight control - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` is permissionless: any caller can submit a valid signed authorization on behalf of the signer. When a victim attempts to cancel a pending signed authorization by revoking `EcrecoverAuthorizer`'s Midnight permission, an attacker possessing the victim's pre-signed authorization can front-run that revocation, causing the authorization to land before the revocation and leaving `isAuthorized[victim][X] = true` in Midnight even though the victim believed they had cancelled it.

## Finding Description

**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` ( [1](#0-0) ):

- Line 25: deadline check — passes if deadline is in the future.
- Line 26: nonce check — passes if `authorization.nonce == nonce[authorization.authorizer]`.
- Lines 33–36: signer check — `require(signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer))`. When the victim themselves signed the authorization, `signer == authorization.authorizer` is true and the `||` short-circuits — **no check is made that `EcrecoverAuthorizer` is currently authorized in Midnight**.
- Line 47: calls `IMidnight(MIDNIGHT).setIsAuthorized(authorized, isAuthorized, authorizer)`.

`Midnight.setIsAuthorized` ( [2](#0-1) ):

- Line 732: `require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender])` — evaluates `isAuthorized[victim][EcrecoverAuthorizer]` **at call time**.

**Root cause:** There is no check in `EcrecoverAuthorizer.setIsAuthorized` that `EcrecoverAuthorizer` is currently authorized in Midnight before proceeding. The Midnight-level gate at line 732 is evaluated at call time, so if the attacker's transaction executes before the victim's revocation, `isAuthorized[victim][EcrecoverAuthorizer]` is still `true` and the gate passes.

**Exploit flow:**

1. Victim authorizes `EcrecoverAuthorizer` in Midnight: `isAuthorized[victim][EcrecoverAuthorizer] = true`.
2. Victim signs `Authorization{authorizer=victim, authorized=X, isAuthorized=true, nonce=0, deadline=block.timestamp+365 days}`. The signature is known to the attacker (mempool, off-chain relay, dApp event).
3. Victim decides to cancel and broadcasts `midnight.setIsAuthorized(ecrecoverAuthorizer, false, victim)`.
4. Attacker observes the revocation in the mempool and front-runs it with `ecrecoverAuthorizer.setIsAuthorized(auth, sig)`.
5. At the time of the attacker's call: deadline passes; nonce=0 matches; `signer == authorization.authorizer` is true (short-circuits the Midnight authorization check on the signer path); `isAuthorized[victim][EcrecoverAuthorizer]` is still `true` in `Midnight.setIsAuthorized`. All checks pass.
6. `isAuthorized[victim][X] = true` is written in Midnight.
7. Victim's revocation executes: `isAuthorized[victim][EcrecoverAuthorizer] = false`. But `isAuthorized[victim][X]` remains `true`.

**Why existing checks fail:**

- The nonce check ( [3](#0-2) ) passes because nonce=0 matches the current state before the attacker's transaction.
- The deadline check ( [4](#0-3) ) passes because the deadline is 365 days away.
- The signer check ( [5](#0-4) ) passes via the `signer == authorization.authorizer` branch — the victim signed it, so this is true regardless of whether `EcrecoverAuthorizer` is authorized in Midnight.
- The Midnight gate ( [6](#0-5) ) passes because `isAuthorized[victim][EcrecoverAuthorizer]` is still `true` at the moment the attacker's transaction executes.
- There is no `invalidateNonce` or equivalent function in `EcrecoverAuthorizer` ( [7](#0-6) ) — the only way to burn a nonce is to call `setIsAuthorized`, which itself requires `EcrecoverAuthorizer` to be authorized in Midnight to complete, creating a circular dependency.

## Impact Explanation
X receives `isAuthorized[victim][X] = true` in Midnight, granting X full Midnight control over the victim — ability to take offers, set consumed, withdraw collateral, etc. — even though the victim believed they had cancelled the authorization. The victim's revocation transaction succeeds but is rendered meaningless because the authorization was already applied. This is an unauthorized state change / privilege escalation directly caused by a protocol-level race condition, matching an in-scope impact class per RESEARCHER.md.

## Likelihood Explanation
Preconditions: (1) Victim has a signed authorization with a future deadline in circulation — common in dApp flows where signatures are shared off-chain or submitted to a relayer. (2) Victim attempts to cancel by revoking `EcrecoverAuthorizer`'s Midnight permission — a natural and reasonable cancellation action given the absence of a direct nonce-invalidation function. (3) Attacker has the signed authorization — obtainable from the mempool, off-chain relay, or a prior on-chain submission attempt. All three are realistic. The attack is executable by any MEV searcher monitoring the mempool, requires no special privilege, and is repeatable.

## Recommendation
Add a permissionless `invalidateNonce` function to `EcrecoverAuthorizer` that allows the authorizer to increment their own nonce directly, without requiring Midnight authorization:

```solidity
function invalidateNonce(address authorizer) external {
    require(msg.sender == authorizer, Unauthorized());
    nonce[authorizer]++;
}
```

This allows the victim to atomically burn the pending nonce in the same transaction as (or before) the Midnight revocation, making the attacker's pre-signed authorization invalid regardless of transaction ordering. Alternatively, document clearly that the only safe cancellation path is nonce invalidation, not Midnight permission revocation.

## Proof of Concept
Minimal Foundry test:

1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. `vm.prank(victim)`: call `midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim)`.
3. Victim signs `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=block.timestamp+365 days}` using `vm.sign`.
4. Simulate victim broadcasting `midnight.setIsAuthorized(address(ecrecoverAuthorizer), false, victim)` (not yet executed).
5. `vm.prank(attacker)`: call `ecrecoverAuthorizer.setIsAuthorized(auth, sig)` — this executes first (front-run).
6. Execute victim's revocation: `vm.prank(victim)`: `midnight.setIsAuthorized(address(ecrecoverAuthorizer), false, victim)`.
7. Assert: `midnight.isAuthorized(victim, attacker) == true` and `midnight.isAuthorized(victim, address(ecrecoverAuthorizer)) == false`.

Expected result: both assertions pass, confirming the attacker gained unauthorized control over the victim's Midnight account despite the victim's revocation succeeding.

```text
contract EcrecoverAuthorizer is IEcrecoverAuthorizer {
    address public immutable MIDNIGHT;
    mapping(address => uint256) public nonce;

    constructor(address _midnight) {
        MIDNIGHT = _midnight;
    }

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
```
