Audit Report

## Title
Fee-on-Transfer Token Flash Loan Drains Protocol Reserves Without Balance Invariant Check - (File: src/Midnight.sol)

## Summary
The `flashLoan` function transfers `assets[i]` out and pulls `assets[i]` back with no pre/post balance snapshot on the protocol's own holdings. When the token is fee-on-transfer, the protocol disburses `assets[i]` but receives only `assets[i]*(1-f)` on repayment, producing a net loss of `assets[i]*f` per call. The only formal exclusion of this token class is a Certora prover-level assumption in `Solvency.spec`, not an on-chain guard.

## Finding Description
**Exact code path — `flashLoan` lines 737–752 of `src/Midnight.sol`:**

Outbound leg: `SafeTransferLib.safeTransfer(tokens[i], callback, assets[i])` at line 743. With fee rate `f`, the protocol's balance decreases by `assets[i]` while the callback receives only `assets[i]*(1-f)`. [1](#0-0) 

Inbound leg: `SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i])` at line 750. The call requests `assets[i]` from the callback; the fee is deducted in transit, so the protocol receives only `assets[i]*(1-f)`. [2](#0-1) 

**Root cause:** A grep across `src/Midnight.sol` confirms zero occurrences of `balanceOf`, `balanceBefore`, or `balanceAfter` anywhere in the file. There is no `require(balanceAfter >= balanceBefore)` guard in or around `flashLoan`. The function also has no token allowlist, no role gate, and no market-membership check on the supplied token addresses. [3](#0-2) 

**Why the Certora proof does not protect against this:** The only formal exclusion of fee-on-transfer tokens is the prover-level assumption at line 31 of `certora/specs/Solvency.spec`:

> "Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits."

The `CVL_transferFrom` summary models lossless transfers, so the `tokenBalanceCorrect` invariant proven in the spec is unsound for fee-on-transfer tokens in production. [4](#0-3) 

**Why existing checks fail:** The only post-inbound check is that `safeTransferFrom` does not revert. It succeeds as long as the callback holds `assets[i]` tokens (satisfied by pre-funding with `assets[i]*f`). No balance invariant is enforced on-chain.

## Impact Explanation
The protocol's token balance for any fee-on-transfer loan token decreases by `assets*f` per flash loan call. Repeated calls drain the protocol's reserves for that token, directly violating the core solvency invariant that contract balances must cover collateral, withdrawable, and claimable settlement fee amounts. Lenders and borrowers whose positions are denominated in that token face insolvency as backing reserves are eroded. This constitutes direct, concrete loss of protocol-held assets. Neither `SECURITY.md` nor `RESEARCHER.md` excludes fee-on-transfer tokens or griefing attacks from scope. [5](#0-4) 

## Likelihood Explanation
**Preconditions:**
- A fee-on-transfer token must be used as a market's loan token (USDT has a dormant fee-on-transfer switch; many DeFi-native tokens carry transfer fees by design).
- The attacker must pre-fund their callback with `assets*f` tokens per iteration.

**Feasibility:** `flashLoan` is callable by any unprivileged address with no token allowlist, no role gate, and no market-membership check on the supplied token addresses. [6](#0-5) 

**Cost model:** This is a 1:1 griefing attack — the attacker spends `assets*f` tokens per iteration and the protocol loses the same amount. It is not directly profitable, but it is a credible griefing/DoS vector against any market whose loan token carries a transfer fee.

## Recommendation
Add a pre/post balance snapshot around the full flash loan cycle:

```solidity
uint256[] memory balancesBefore = new uint256[](tokens.length);
for (uint256 i = 0; i < tokens.length; i++) {
    balancesBefore[i] = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
}
// ... callback ...
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(
        IERC20(tokens[i]).balanceOf(address(this)) >= balancesBefore[i],
        InsufficientRepayment()
    );
}
```

This ensures the protocol's balance is at least restored regardless of token transfer mechanics, and is the standard pattern used by Aave v3 and Morpho Blue.

## Proof of Concept
1. Deploy a fee-on-transfer ERC-20 token with fee rate `f = 1%` and seed a Midnight market with it as the loan token (e.g., 1,000,000 tokens deposited by lenders).
2. Deploy a malicious callback contract that holds `assets*f` tokens and returns `CALLBACK_SUCCESS`.
3. Call `flashLoan([feeToken], [assets], maliciousCallback, "")` repeatedly.
4. Each call: protocol sends `assets`, receives `assets*(1-f)`, net loss = `assets*f`.
5. After `1/f` iterations (100 calls at 1% fee), the protocol has lost the equivalent of one full `assets` tranche from its reserves.
6. Verify with a Foundry fork test: assert `IERC20(feeToken).balanceOf(address(midnight))` decreases by `assets*f` per call. [3](#0-2)
