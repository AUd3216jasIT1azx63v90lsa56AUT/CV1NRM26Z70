Audit Report

## Title
ECDSA Signature Malleability Enables Nonce-Griefing Front-Run on `setIsAuthorized` - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` calls raw `ecrecover` at line 31 without enforcing a low-s constraint, and imposes no `msg.sender` restriction. Any observer can derive a second valid signature `(v', r, n − s)` for the same message and signer, where `v' = (v == 27 ? 28 : 27)` and `n` is the secp256k1 curve order. An attacker who front-runs a victim's pending `setIsAuthorized` with this malleable counterpart atomically consumes the victim's nonce, causing the victim's original transaction to revert with `InvalidNonce`. When the victim's call is part of an atomic multicall bundle (e.g., authorize-then-invoke a protective action), the entire bundle fails, introducing a timing gap exploitable in time-sensitive scenarios.

## Finding Description
**Code path — `EcrecoverAuthorizer.setIsAuthorized` (`src/periphery/EcrecoverAuthorizer.sol`, lines 24–48):**

- **Line 26:** `require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());` — nonce is checked and incremented atomically; the first caller to supply any valid signature for nonce N wins.
- **Line 31:** `address signer = ecrecover(digest, signature.v, signature.r, signature.s);` — raw `ecrecover` with no low-s guard.
- **No `msg.sender` restriction** anywhere in the function; any address may call it with any signature.

**Root cause:** secp256k1 ECDSA produces two valid `(v, r, s)` pairs per `(message, key)`: the canonical `(v, r, s)` and the malleable `(v', r, n − s)` where `n = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141` and `v' = (v == 27 ? 28 : 27)`. Both pass `ecrecover` and return the identical signer address. Without `require(uint256(s) <= 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0)`, both are accepted.

**Exploit flow:**
1. Victim signs `Authorization{authorizer=V, authorized=X, isAuthorized=true, nonce=N, deadline=D}` and broadcasts `setIsAuthorized(auth, sig=(v,r,s))`.
2. Attacker observes the pending transaction, extracts `(v, r, s)`, computes `s' = n − s`, `v' = (v == 27 ? 28 : 27)` — trivial off-chain arithmetic with any secp256k1 library.
3. Attacker submits `setIsAuthorized(auth, Signature{v', r, s'})` with higher gas.
4. Attacker's transaction executes first: `nonce[V] == N` passes, nonce becomes `N+1`, `ecrecover` returns `V`, authorization is set to the intended value.
5. Victim's transaction executes: `nonce[V] == N+1` but `auth.nonce == N` → `InvalidNonce()` revert.

**Why existing checks fail:**
- `require(signer != address(0))` — both signatures recover the same non-zero address; passes for both.
- `require(signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(...))` — same signer recovered; passes for both.
- The nonce is already consumed before the victim's transaction executes; no subsequent check can recover it.

## Impact Explanation
The victim's `setIsAuthorized` transaction reverts. Although the authorization state is correctly set by the attacker's front-run, any atomic bundle that included the `setIsAuthorized` call fails entirely and must be re-submitted. The re-submission requires the victim to reconstruct the bundle without the `setIsAuthorized` step, introducing a non-trivial timing gap. In a liquidation-protection scenario on a chain with a public mempool, this gap can be sufficient for the victim's position to become liquidatable before the protective action executes. The attacker can repeat this for any future `setIsAuthorized` call the victim makes (e.g., revoking or changing authorization), causing persistent griefing with potential for indirect financial loss.

## Likelihood Explanation
Preconditions: victim broadcasts a `setIsAuthorized` transaction (routine operation); attacker monitors the public mempool (standard, zero-cost capability on Ethereum mainnet and most L2s with public mempools). No special role, funds beyond gas, or credentials required. Computing `s' = n − s` and the v-flip is a single subtraction and conditional — executable in milliseconds. The attack is repeatable for every new `setIsAuthorized` nonce the victim attempts to use.

## Recommendation
Add a low-s constraint immediately after recovering the signer:

```solidity
uint256 SECP256K1_ORDER_HALF = 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0;
require(uint256(signature.s) <= SECP256K1_ORDER_HALF, InvalidSignature());
address signer = ecrecover(digest, signature.v, signature.r, signature.s);
```

Alternatively, replace raw `ecrecover` with OpenZeppelin's `ECDSA.recover`, which enforces the low-s constraint and rejects malleable signatures by default. This ensures only one of the two mathematically valid `(v, r, s)` pairs is accepted per message, eliminating the front-run vector entirely.

## Proof of Concept
**Minimal fork test plan (Foundry):**

1. Deploy `EcrecoverAuthorizer` pointing to a mock `Midnight`.
2. Generate a valid EIP-712 `Authorization` struct and sign it with a test private key, obtaining `(v, r, s)`.
3. Compute the malleable counterpart: `s' = secp256k1_order - s`, `v' = (v == 27 ? 28 : 27)`.
4. As the attacker, call `setIsAuthorized(auth, Signature(v', r, s'))` — assert it succeeds and `nonce[authorizer] == 1`.
5. As the victim, call `setIsAuthorized(auth, Signature(v, r, s))` — assert it reverts with `InvalidNonce()`.
6. Confirm `ecrecover(digest, v', r, s') == ecrecover(digest, v, r, s)` to prove both signatures recover the same address.
