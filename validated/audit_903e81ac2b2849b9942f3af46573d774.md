Audit Report

## Title
Front-running partial liquidation reduces `maxRepaid` below victim's `repaidUnits`, causing `RecoveryCloseFactorConditionsViolated()` revert - (File: src/Midnight.sol)

## Summary

In normal (pre-maturity) liquidation mode, `maxRepaid` is computed live from the current `_position.debt - maxDebt` at call time. An attacker can front-run any victim liquidator with a profitable partial liquidation of `A` units, which reduces `maxRepaid` by exactly `A`, while leaving the `rcfThreshold` small-collateral exception value unchanged. This causes the victim's transaction to revert with `RecoveryCloseFactorConditionsViolated()` whenever `victimRepaidUnits > originalMaxRepaid - A`. The attack is self-funding (attacker profits from the partial liquidation) and indefinitely repeatable.

## Finding Description

**Code path:** `src/Midnight.sol` lines 655–667.

```solidity
uint256 maxRepaid = lltv < WAD
    ? (_position.debt - maxDebt).mulDivUp(WAD * WAD, WAD * WAD - lif * lltv)
    : type(uint256).max;
require(
    repaidUnits <= maxRepaid
        || _position.collateral[collateralIndex].mulDivDown(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
            .mulDivDown(WAD, lif).zeroFloorSub(maxRepaid) < market.rcfThreshold,
    RecoveryCloseFactorConditionsViolated()
);
``` [1](#0-0) 

`maxDebt` is recomputed from current on-chain collateral at lines 607–618 on every call: [2](#0-1) 

**Root cause — `maxRepaid` is not monotone across sequential liquidations.**

After the attacker's partial liquidation of `A` units, the state changes are (from lines 652, 670–676):
- `_position.debt` → `debt - A`
- `_position.collateral[collateralIndex]` → `collateral - A·lif·ORACLE_PRICE_SCALE/(WAD·price)`
- `maxDebt` (recomputed) → `maxDebt - A·lif·lltv/WAD²` [3](#0-2) 

Therefore:

```
newDebt - newMaxDebt
  = (debt - A) - (maxDebt - A·lif·lltv/WAD²)
  = (debt - maxDebt) - A·(WAD² - lif·lltv)/WAD²

newMaxRepaid = (newDebt - newMaxDebt)·WAD²/(WAD² - lif·lltv)
             = originalMaxRepaid - A
```

The new `maxRepaid` is exactly `originalMaxRepaid - A`.

**The `rcfThreshold` exception is invariant under partial liquidations.** The exception fires when `collateral·price/lif - maxRepaid < rcfThreshold`. After the attacker's partial liquidation:

```
newCollateral·price/lif - newMaxRepaid
= (collateral - A·lif/price)·price/lif - (originalMaxRepaid - A)
= collateral·price/lif - A - originalMaxRepaid + A
= collateral·price/lif - originalMaxRepaid   ← identical to pre-attack value
```

So if the exception did not apply before the attack, it does not apply after.

**Exploit flow:**

1. Borrower is unhealthy (`debt > maxDebt`), no bad debt, RCF active (rcfThreshold exception does not apply).
2. `originalMaxRepaid = (debt - maxDebt)·WAD²/(WAD² - lif·lltv)`.
3. Victim liquidator broadcasts tx with `victimRepaidUnits` where `1 < victimRepaidUnits ≤ originalMaxRepaid`.
4. Attacker front-runs with `attackerRepaidUnits` chosen so that:
   - `attackerRepaidUnits < originalMaxRepaid` (position stays unhealthy, attacker passes RCF)
   - `attackerRepaidUnits > originalMaxRepaid - victimRepaidUnits` (so `newMaxRepaid < victimRepaidUnits`)
5. Victim's tx executes: `newMaxRepaid = originalMaxRepaid - attackerRepaidUnits < victimRepaidUnits`; rcfThreshold exception still does not apply → revert with `RecoveryCloseFactorConditionsViolated()`.

The `liquidationLocked` check at line 621 only prevents liquidation during take callbacks (transient storage), not between separate transactions, so it provides no protection here. [4](#0-3) 

## Impact Explanation

An unprivileged attacker can permanently prevent any competing liquidator from executing a full or large partial liquidation of any unhealthy position by repeatedly front-running with small profitable partial liquidations. The victim's transaction always reverts with `RecoveryCloseFactorConditionsViolated()`. The position remains unhealthy and continues to accumulate protocol risk, while the attacker monopolizes all liquidation profit. This constitutes both a denial-of-service on liquidators and theft of liquidation opportunity (the attacker extracts value that would otherwise go to the victim).

## Likelihood Explanation

**Preconditions:**
- Borrower is unhealthy (`debt > maxDebt`) in normal (pre-maturity) mode.
- RCF is active: `collateral·price/lif - maxRepaid ≥ rcfThreshold`.
- Victim liquidator submits `repaidUnits > 1`.
- No `liquidatorGate` restricting the attacker (or the attacker also passes the gate).

These are the standard conditions for any contested liquidation on a chain with a public mempool. The attack is economically self-funding (attacker profits from the partial liquidation at `lif > 1` discount) and can be repeated indefinitely every time the victim resubmits.

## Recommendation

Add a caller-supplied `minRepaidUnits` (or equivalently `maxRepaidFloor`) parameter to `liquidate`. If the computed `maxRepaid` falls below this floor, revert with a slippage error rather than silently allowing the attacker to shrink the window. This gives victim liquidators a standard slippage-protection mechanism: they specify the minimum `maxRepaid` they expect, and their transaction reverts cleanly (rather than with `RecoveryCloseFactorConditionsViolated()`) if the position has been partially liquidated ahead of them. Alternatively, document this as an accepted liveness limitation and recommend that liquidators use private mempools or commit-reveal schemes.

## Proof of Concept

**Minimal Foundry test outline:**

```solidity
// Setup: unhealthy position, RCF active, no rcfThreshold exception
// 1. Compute originalMaxRepaid
// 2. Victim prepares tx: liquidate(..., repaidUnits = originalMaxRepaid, ...)
// 3. Attacker front-runs: liquidate(..., repaidUnits = 1, ...)
//    → attacker succeeds, newMaxRepaid = originalMaxRepaid - 1
// 4. Victim tx executes:
//    → repaidUnits (= originalMaxRepaid) > newMaxRepaid (= originalMaxRepaid - 1)
//    → rcfThreshold exception still does not apply (invariant proven above)
//    → vm.expectRevert(RecoveryCloseFactorConditionsViolated.selector) ✓
// 5. Repeat step 3-4 to show indefinite repeatability
```

The existing test infrastructure in `test/LiquidationTest.sol` (e.g., `_setupUnhealthy`, `_maxRepaid`, `testRcfThresholdEnforcesRecoveryCloseFactor`) provides all the scaffolding needed to implement this as a concrete fork test. [5](#0-4)
