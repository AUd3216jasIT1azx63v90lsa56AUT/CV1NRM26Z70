Audit Report

## Title
Fee-on-Transfer Collateral Token Inflates `position.collateral` Accounting, Enabling Undercollateralized Debt Creation - (File: `src/Midnight.sol`)

## Summary

`supplyCollateral` credits the caller-supplied `assets` parameter directly to `position.collateral[collateralIndex]` at line 533 before executing the ERC20 transfer at line 545, with no balance-before/after snapshot. When the collateral token charges a transfer fee, Midnight receives fewer tokens than recorded. `isHealthy` then computes `maxDebt` from the inflated stored value, allowing a seller to carry more debt than their actual on-hand collateral supports, creating undercollateralized positions at inception.

## Finding Description

**Root cause — `supplyCollateral` (`src/Midnight.sol` lines 531–545):**

```solidity
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets); // line 533
...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545
```

The accounting write at line 533 uses the caller-controlled `assets` parameter. The transfer at line 545 executes after the state update with no balance snapshot. For a 1% fee-on-transfer token: `supplyCollateral(..., 100, ...)` records `position.collateral[i] = 100` but Midnight receives 99.

**No restriction on collateral token type — `touchMarket` (`src/Midnight.sol` lines 757–773):**

Market creation validates only sorted order, non-zero address, allowed LLTV tiers, and valid `maxLif`. There is no check that the collateral token is not a fee-on-transfer token. Any unprivileged actor can create such a market.

**Health check uses inflated value — `isHealthy` (`src/Midnight.sol` lines 954–955):**

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

`_position.collateral[i]` is the inflated recorded value (100), not the actual balance (99). `maxDebt` is therefore ~1% higher than the true collateral-backed capacity.

**Health check at end of `take` (`src/Midnight.sol` line 476):**

```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```

`isHealthy` returns `true` using the inflated collateral, so the take succeeds and the seller's debt is increased beyond what their actual collateral covers.

**Exploit flow:**
1. Attacker deploys or uses an existing fee-on-transfer ERC20 (1% fee) as collateral token.
2. Anyone calls `touchMarket` with this token as `collateralParams[0].token` — succeeds, no restriction.
3. Seller calls `supplyCollateral(market, 0, 100, seller)` → `position.collateral[0] = 100`, Midnight balance increases by 99.
4. Taker calls `take(offer, ratifierData, units, taker, ...)`.
5. `isHealthy` computes `maxDebt` using `position.collateral[0] = 100`, returns `true`.
6. Take succeeds. Seller holds debt backed by 99 units of actual collateral, but protocol believes it is backed by 100.

**Existing checks are insufficient:** `isRatified` in `IEcrecoverRatifier` is a pure signature verification — it has no visibility into collateral token behavior, actual balances, or the accounting delta. The `touchMarket` checks do not gate on token transfer behavior.

## Impact Explanation

The invariant "contract balances cover collateral" is violated from the moment `supplyCollateral` is called with a fee-on-transfer token. The seller's position is undercollateralized at inception: actual collateral held < collateral used to compute `maxDebt`. The seller can borrow up to `100 * price * lltv / WAD` units of debt while only 99 units of collateral are held. The shortfall scales with position size and fee rate. When the position is eventually liquidated, the protocol realizes bad debt proportional to the fee, causing direct loss to lenders via the `lossFactor` mechanism.

## Likelihood Explanation

All preconditions are reachable by any unprivileged user: fee-on-transfer ERC20 tokens exist (USDT has historically had this mode; many DeFi tokens implement it), market creation is permissionless with no token-type restriction, and normal user actions (signing and taking offers) complete the exploit. No admin keys, oracle manipulation, or impossible values are required. The attack is repeatable on every `supplyCollateral` call with a fee-on-transfer token.

## Recommendation

Apply a balance-before/after pattern in `supplyCollateral`: record the contract's balance of `collateralToken` before the transfer, execute the transfer, then credit only the actual received amount (`balanceAfter - balanceBefore`) to `_position.collateral[collateralIndex]`. This eliminates the discrepancy regardless of token fee behavior:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

The state update must move after the transfer. Alternatively, explicitly whitelist collateral tokens at market creation to exclude fee-on-transfer tokens.

## Proof of Concept

**Minimal Foundry test plan:**
1. Deploy a mock ERC20 that deducts 1% on every `transferFrom`.
2. Call `touchMarket` with this token as `collateralParams[0].token` — assert it succeeds.
3. Call `supplyCollateral(market, 0, 1000e18, seller)` — assert `position[id][seller].collateral[0] == 1000e18` but `mockToken.balanceOf(midnight) == 990e18`.
4. Have seller sign a sell offer; have taker call `take`.
5. Assert `isHealthy` returns `true` despite actual collateral being 990e18.
6. Assert seller's `debt` exceeds what 990e18 collateral at the given oracle price and LLTV would support.
7. Confirm the position is undercollateralized: `990e18 * price * lltv / WAD < debt`.
