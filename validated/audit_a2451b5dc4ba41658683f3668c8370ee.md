### Title
Authorized Accounts Can Grant Unrestricted Further Authorizations on Behalf of Users, Enabling Full Account Takeover — (`src/Midnight.sol`)

### Summary

The `setIsAuthorized` function in `Midnight.sol` allows any account that has been authorized by a user to grant full, unrestricted authorization to arbitrary third parties on that user's behalf. This is the direct analog to the external report's finding that "authorized accounts can authorize other accounts on behalf of the user" and "delegations are unrestricted by default." A user who authorizes a smart contract or EOA to manage their positions may not realize that the authorized party can silently escalate access to any attacker, enabling complete account takeover and direct theft of all assets.

### Finding Description

The authorization system in Midnight is coarse-grained and flat: `isAuthorized[onBehalf][authorized]` is a simple boolean with no scope restrictions. The `setIsAuthorized` function checks only that the caller is currently authorized by `onBehalf`, but places no restriction on what the authorized account can do — including granting further authorizations to arbitrary addresses:

```solidity
// src/Midnight.sol:731-735
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
}
```

The protocol itself documents this behavior in the `AUTHORIZATIONS` section of `Midnight.sol`:

> "In particular, authorized accounts can authorize other accounts on behalf of the user."

The test suite at `test/AuthorizationTest.sol:290-303` confirms this works exactly as designed — `authorized` can call `setIsAuthorized(newAuthorized, true, user)` and it succeeds.

**Exploit path:**

1. Alice authorizes Bob (e.g., a DeFi aggregator, bundler, or any smart contract) via `setIsAuthorized(bob, true, alice)`.
2. Bob (malicious or compromised) calls `setIsAuthorized(attacker, true, alice)`.
3. Attacker now has full, unrestricted access to Alice's account on Midnight.
4. Attacker calls `withdrawCollateral(market, 0, alice_collateral_amount, alice, attacker)` — steals all collateral.
5. Attacker calls `withdraw(market, alice_credit, alice, attacker)` — steals all credit/loan tokens.
6. Attacker calls `setConsumed(group, type(uint256).max, alice)` — cancels all of Alice's active offers.
7. Attacker calls `setIsAuthorized(bob, false, alice)` — revokes Bob's access to cover tracks.

No additional preconditions are needed beyond step 1. The `EcrecoverAuthorizer` periphery contract (`src/periphery/EcrecoverAuthorizer.sol:33-36`) also allows an already-authorized signer to submit authorization signatures on behalf of the user, widening the attack surface further.

### Impact Explanation

**Direct theft of all user assets.** Any user who has ever authorized any account (smart contract or EOA) is at risk of having that authorized party silently escalate access to an attacker. The attacker can drain all collateral, all credit, and manipulate all positions and offers. There is no scope restriction, no caveat system, and no way for the user to limit what an authorized account can do. The impact is a complete account takeover with full asset loss.

### Likelihood Explanation

The likelihood is **high** in practice. Users are expected to authorize smart contracts (bundlers like `MidnightBundles`, ratifiers, aggregators) to manage their positions — this is a core protocol workflow. Any of these contracts, if malicious or compromised (e.g., via an upgrade, a bug, or a supply-chain attack), can immediately escalate access to an attacker. The attack requires no privileged keys, no oracle manipulation, and no special timing — just being authorized by the victim, which is the normal operating state for users of the protocol.

### Recommendation

1. **Restrict `setIsAuthorized` to the account itself**: Only `onBehalf == msg.sender` should be allowed to grant new authorizations. Authorized accounts should not be able to sub-delegate.
2. **If sub-delegation is intentional**, add explicit scope restrictions (e.g., a bitmask of allowed function selectors) so that an authorized account cannot grant broader access than it itself holds.
3. **Prominently document** in user-facing materials that authorizing any account grants it the ability to further authorize arbitrary third parties, so users understand the full scope of what they are consenting to.

### Proof of Concept

The existing test at `test/AuthorizationTest.sol:290-303` already demonstrates the full exploit path:

```solidity
function testSetIsAuthorizedAuthorization(address user, address authorized, address newAuthorized) public {
    vm.assume(user != authorized);

    // Step 1: user authorizes `authorized`
    vm.prank(user);
    midnight.setIsAuthorized(authorized, true, user);

    // Step 2: `authorized` grants access to `newAuthorized` (attacker) — no user consent needed
    vm.prank(authorized);
    midnight.setIsAuthorized(newAuthorized, true, user);

    // Step 3: attacker now has full access to user's account
    assertEq(midnight.isAuthorized(user, newAuthorized), true);
    // attacker can now call withdraw, withdrawCollateral, take, setConsumed, etc. on behalf of user
}
```

To extend to asset theft, add after step 3:
```solidity
vm.prank(newAuthorized); // attacker
midnight.withdrawCollateral(market, 0, collateralAmount, user, newAuthorized); // steals collateral
```

**Root cause**: `src/Midnight.sol:731-735` — the authorization check `isAuthorized[onBehalf][msg.sender]` permits any authorized account to call `setIsAuthorized` with `onBehalf` set to the victim, granting arbitrary further authorizations with no restriction.
