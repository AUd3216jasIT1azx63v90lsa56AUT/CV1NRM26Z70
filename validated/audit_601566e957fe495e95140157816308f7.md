Audit Report

## Title
Fee-on-transfer loan token silently overcounts `claimableSettlementFee` in `take`, enabling protocol insolvency - (`File: src/Midnight.sol`)

## Summary
In `take`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal `buyerAssets - sellerAssets` at line 418 before the inbound `safeTransferFrom` at line 455. Because `SafeTransferLib.safeTransferFrom` only checks the boolean return value of `transferFrom` and performs no balance-before/after accounting, a fee-on-transfer loan token causes only `(buyerAssets - sellerAssets - transferFee)` to actually arrive at `address(this)` while the full nominal amount is credited. The deficit accumulates across every `take`, and when `feeClaimer` calls `claimSettlementFee` against the overcounted balance, it drains tokens that back lender `withdrawable` credits or borrower collateral, causing direct protocol insolvency.

## Finding Description

**Exact code path:**

1. `src/Midnight.sol` line 418: `claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;` — accounting state written before any transfer occurs.
2. `src/Midnight.sol` line 455: `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);` — for a fee-on-transfer token, only `(buyerAssets - sellerAssets - transferFee)` arrives.
3. `src/libraries/SafeTransferLib.sol` lines 24–34: `safeTransferFrom` calls `transferFrom` and checks only the boolean return value. No balance delta is measured. A fee-on-transfer token returns `true` while silently withholding the fee, so no revert occurs.
4. `src/Midnight.sol` lines 305–309: `claimSettlementFee` decrements `claimableSettlementFee[token]` by `amount` and calls `safeTransfer(token, receiver, amount)`. If `claimableSettlementFee` exceeds actual tokens held for that purpose, this transfer pulls from tokens backing lender `withdrawable` or borrower collateral.

**Root cause:** The protocol credits `claimableSettlementFee` with the nominal transfer amount rather than the actual received amount. `SafeTransferLib` provides no mechanism to detect the shortfall.

**Why existing checks fail:** `certora/specs/Solvency.spec` line 31 explicitly states: *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver."* The `CVL_transferFrom` ghost (lines 43–59) moves the full `value` between ghost balances, so the `tokenBalanceCorrect` invariant (lines 162–177) and `pendingFeeReceiptZero` invariant (lines 157–158) are proven only under this assumption. Fee-on-transfer tokens violate the assumption silently — the formal proofs provide no coverage here.

**No loan token validation exists:** `touchMarket` (lines 755–791) validates collateral params, maturity, and LLTV but performs zero validation on `market.loanToken`. The protocol design explicitly states "arbitrary loan token" (`live_context.json` line 17).

## Impact Explanation

Each `take` with a fee-on-transfer loan token creates a deficit: `claimableSettlementFee[token]` exceeds actual tokens held for that purpose by `transferFee` per call. The deficit accumulates across every take in the affected market. When `feeClaimer` calls `claimSettlementFee` (acting normally, not maliciously), `safeTransfer` is executed against the overcounted balance. Once the cumulative shortfall exceeds the slack in the contract's token balance, the transfer pulls from tokens backing lender `withdrawable` credits or borrower collateral. This breaks the core solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee`, causing lenders to be unable to withdraw and/or collateral to be unrecoverable — direct protocol insolvency and loss of user funds.

## Likelihood Explanation

**Preconditions:**
- A market exists (or is created by the attacker) with a fee-on-transfer loan token.
- The attacker (as taker) has approved Midnight for the loan token and holds sufficient balance.

Market creation is permissionless with no loan token validation. The attacker can be simultaneously the token deployer, market creator, offer maker, and taker — requiring no cooperation from other users. `live_context.json` explicitly lists "token charges fee" as an external behavior to test (line 389) and states fee-on-transfer tokens "should be tested if not explicitly excluded" (line 233); they are not excluded anywhere in the codebase. The attack is repeatable on every `take` call, accumulating the deficit until `feeClaimer` claims fees. Fee-on-transfer tokens (tax tokens, some rebasing tokens) are a well-known ERC20 variant.

## Recommendation

Measure the actual received amount using a balance-before/after pattern in `take`, and credit `claimableSettlementFee` only with the delta actually received:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived; // move to after transfer, use actual delta
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens by adding a validation check in `touchMarket` or via a registry of allowed tokens. If fee-on-transfer tokens are intended to be unsupported, this must be enforced on-chain, not just documented, since market creation is permissionless.

## Proof of Concept

**Minimal Foundry fork test plan:**

1. Deploy a mock ERC20 that deducts a 1% fee on every `transferFrom` (fee sent to a burn address, returns `true`).
2. Call `touchMarket` to create a market with this token as `loanToken`.
3. Create a buy offer (maker = attacker address A) and take it (taker = attacker address B, `takerCallback = address(0)`).
4. After `take`, assert: `claimableSettlementFee[feeToken] == buyerAssets - sellerAssets` (nominal) while `feeToken.balanceOf(midnight) < claimableSettlementFee[feeToken]` (actual shortfall).
5. Deposit legitimate lender funds via a separate `withdrawable`-increasing path to establish a baseline balance.
6. Have `feeClaimer` call `claimSettlementFee(feeToken, claimableSettlementFee[feeToken], receiver)`.
7. Assert that the lender's `withdrawable` balance can no longer be fully redeemed — demonstrating direct fund loss.

**Expected result:** Step 4 shows the accounting discrepancy. Step 7 shows the insolvency: `safeTransfer` in `claimSettlementFee` pulls from tokens backing lender withdrawables, leaving lenders unable to withdraw their full entitlement.
