Audit Report

## Title
Bad Debt Over-Realization via `onLiquidate` Callback Supplying Collateral After Bitmap Snapshot - (File: `src/Midnight.sol`)

## Summary
The `liquidate` function snapshots `_position.collateralBitmap` into a local variable at line 606, computes and fully applies `badDebt` against `_position.debt`, `_marketState.lossFactor`, and `_marketState.totalUnits` at lines 626–641, and only then invokes the `onLiquidate` callback at lines 698–715. A borrower-controlled smart contract can use this callback window to call `supplyCollateral` on its own behalf, activating a new collateral index in storage that was invisible to the bad debt computation, causing the protocol to over-realize bad debt and over-slash lenders.

## Finding Description

**Bitmap snapshot before bad debt realization:**

At line 606, `_collateralBitmap` is copied into a local stack variable: [1](#0-0) 

The while loop iterates only over this local copy. Any collateral index not set in the bitmap at snapshot time is invisible to the `badDebt` computation.

**Bad debt fully applied before callback:**

Lines 626–641 irreversibly reduce `_position.debt`, ratchet up `_marketState.lossFactor`, and reduce `_marketState.totalUnits` — all before any external call: [2](#0-1) 

**Callback invoked after realization:**

The `onLiquidate` callback is called at lines 698–715, after all state mutations: [3](#0-2) 

**`supplyCollateral` authorization is trivially satisfied:**

`supplyCollateral` checks `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. When the borrower is a smart contract `BorrowerContract` that passes `callback = address(this)` to `liquidate`, the callback executes as `BorrowerContract`. When `BorrowerContract` calls `supplyCollateral(..., address(this))` inside `onLiquidate`, `msg.sender == BorrowerContract` and `onBehalf == address(this) == BorrowerContract`, so the check passes with no preconditions: [4](#0-3) 

`supplyCollateral` then sets the new bit in the storage bitmap: [5](#0-4) 

**No reentrancy guard exists:**

The `liquidationLocked` mechanism (transient storage via `tExchange`) is only set during `take` callbacks for the seller: [6](#0-5) 

`liquidate` itself never sets a liquidation lock for the borrower during its own callback window. There is no `nonReentrant` modifier or equivalent lock on `liquidate` or `supplyCollateral`.

**Exploit flow:**
1. `BorrowerContract` holds a position where all existing collateral is worth less than debt at `maxLif` prices (genuine bad debt).
2. `BorrowerContract` calls `liquidate(market, 0, 0, 0, address(this), false, receiver, address(this), "")` — `seizedAssets = repaidUnits = 0`, `callback = address(this)`.
3. `liquidate` snapshots `_collateralBitmap` (line 606), computes `badDebt` from existing collateral only.
4. `badDebt` is applied: `_position.debt` reduced, `_marketState.lossFactor` increased, `_marketState.totalUnits` reduced (lines 626–641).
5. `onLiquidate` is called on `BorrowerContract` (line 700).
6. Inside `onLiquidate`, `BorrowerContract` calls `supplyCollateral(market, newIndex, largeAmount, address(this))`, depositing real collateral and setting a new bit in `_position.collateralBitmap` in storage.
7. `liquidate` returns. The new collateral is in the position, but `badDebt` was already applied without accounting for it.

**Why existing checks fail:**
- No reentrancy guard on `liquidate` or `supplyCollateral`.
- No post-callback re-computation or validation of `badDebt`.
- The local `_collateralBitmap` variable is never refreshed from storage after the callback.
- `supplyCollateral`'s authorization check is trivially satisfied when the borrower is its own callback.

## Impact Explanation
`_marketState.lossFactor` is set higher than warranted, and `_marketState.totalUnits` is reduced by more than the true uncollateralized shortfall. Every lender who subsequently calls `updatePosition` has their credit slashed proportionally to the inflated loss factor — a direct, concrete loss of lender funds. The borrower's debt is forgiven by an amount that was actually backed by the newly supplied collateral, constituting an illegitimate debt reduction at lenders' expense. This is a high-severity accounting integrity violation with direct fund loss to lenders.

## Likelihood Explanation
Preconditions are minimal and realistic: (1) a position with genuine bad debt — a normal liquidation scenario; (2) the borrower is a smart contract — standard in DeFi; (3) the borrower passes itself as the callback — requires no external authorization, no oracle manipulation, no privileged access, and no victim mistakes. The attack is repeatable on any market with bad debt and is fully self-contained.

## Recommendation
Set a liquidation lock for the borrower in transient storage at the start of `liquidate` (analogous to how `take` locks the seller via `tExchange`) and clear it at the end. This prevents `supplyCollateral` (or any other state-mutating call on the borrower's position) from executing during the `onLiquidate` callback window. Alternatively, re-read `_position.collateralBitmap` from storage after the callback and revert if it differs from the pre-callback snapshot, or move the callback invocation before bad debt realization.

## Proof of Concept
Deploy `BorrowerContract` implementing `ILiquidateCallback`. Pre-condition: `BorrowerContract` has debt `D` and collateral worth `C < D` at `maxLif` prices (genuine bad debt of `D - C`). Also hold a separate collateral token for index `newIndex` (not yet activated). Call `liquidate(market, 0, 0, 0, address(this), false, receiver, address(this), "")`. In `onLiquidate`, call `supplyCollateral(market, newIndex, largeAmount, address(this))`. After the call: assert `midnight.lossFactor(id)` reflects a loss of `D - C` units, but `midnight.collateral(id, address(this), newIndex) == largeAmount` is now in the position. The lender's credit after `updatePosition` is slashed by `D - C` despite the position now holding `largeAmount` of real collateral that was never counted against the bad debt.
