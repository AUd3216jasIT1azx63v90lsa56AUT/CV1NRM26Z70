### Title
Fee-on-Transfer / Rebasing Collateral and Loan Tokens Cause Accounting Desync, Enabling Undercollateralized Borrowing and Protocol Insolvency - (File: src/Midnight.sol)

---

### Summary

Morpho Midnight is a permissionless protocol where any ERC-20 token can be used as a loan token or collateral. The protocol's internal accounting in `supplyCollateral()`, `repay()`, and `flashLoan()` credits or debits the full requested `assets`/`units` amount without verifying the actual token balance delta before and after the transfer. If a fee-on-transfer or rebasing-down token is used, the protocol receives fewer tokens than it records, creating a persistent accounting desync that enables undercollateralized borrowing, bad debt creation, and protocol insolvency.

---

### Finding Description

**Vulnerability class:** Accounting (ERC20 transfer delta mismatch)

The protocol documents a token safety requirement at lines 133–139:

> "Midnight must send/receive exactly the requested amount on transfers."
> "Midnight's balance of the token should only decrease on transfer and transferFrom."

However, this requirement is **not enforced on-chain**. The protocol is fully permissionless — any address can call `touchMarket()` to create a market with any ERC-20 token as loan token or collateral. There is no whitelist, no balance-delta check, and no revert path if the received amount is less than requested.

**Attack surface 1 — `supplyCollateral()` (most critical):** [1](#0-0) 

```
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);  // line 533
...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545
```

The position is credited with the full `assets` value **before** the transfer, and no post-transfer balance check is performed. If the collateral token charges a transfer fee (e.g., 5%), the protocol receives `assets * 0.95` but records `assets` as collateral. The borrower's `_position.collateral[collateralIndex]` is inflated relative to actual tokens held. [2](#0-1) 

**Attack surface 2 — `repay()`:** [3](#0-2) 

```
position[id][onBehalf].debt -= UtilsLib.toUint128(units);      // line 508
marketState[id].withdrawable += UtilsLib.toUint128(units);     // line 509
...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
```

`withdrawable` is incremented by `units` but only `units - fee` is actually received. Lenders' redeemable credit is backed by fewer tokens than the accounting reflects. [4](#0-3) 

**Attack surface 3 — `flashLoan()`:** [5](#0-4) 

```
SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);              // line 743
...
SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]); // line 750
```

The protocol sends `assets[i]` and demands `assets[i]` back. With a fee-on-transfer token, the repayment transfer delivers `assets[i] - fee` to the contract. Each flash loan call permanently drains the protocol's balance of that token by the fee amount, with no accounting correction. [6](#0-5) 

The `live_context.json` explicitly lists these as test axes:

> "fee-on-transfer, rebasing, false-return, ERC777-like hooks, and non-standard decimals should be tested if not explicitly excluded" [7](#0-6)

### Impact Explanation

**`supplyCollateral()` path (highest severity):**

1. Borrower calls `supplyCollateral()` with `assets = 1000` of a 5%-fee-on-transfer collateral token.
2. Protocol records `_position.collateral[index] = 1000` but holds only `950` tokens.
3. `isHealthy()` computes `maxDebt` using the inflated `1000` collateral value via the oracle.
4. Borrower borrows against the phantom `50` units of collateral value.
5. When liquidated, the protocol can only seize `950` tokens but must repay `units` computed from `1000`.
6. The shortfall becomes bad debt, socialized among all lenders in the market via `lossFactor`.

This is a direct, repeatable path to bad debt creation and lender fund loss with no privileged access required.

**`repay()` path:**
Protocol's `withdrawable` exceeds actual token balance. Lenders who withdraw last receive nothing; the protocol is insolvent by the cumulative fee amount across all repayments.

**`flashLoan()` path:**
Any caller can drain the protocol's entire balance of a fee-on-transfer token to zero by calling `flashLoan()` repeatedly, each call consuming the fee amount from the protocol's reserves.

---

### Likelihood Explanation

- Market creation is **fully permissionless** — no admin approval, no token whitelist.
- Fee-on-transfer tokens are a well-established, non-malicious token class (transfer taxes are common in DeFi governance and meme tokens).
- The attacker does not need any privileged role, leaked key, or governance capture.
- The attacker only needs to: (a) identify or deploy a market using a fee-on-transfer collateral token, and (b) call `supplyCollateral()` followed by a borrow via `take()`.
- The `live_context.json` explicitly flags this as a test axis, confirming the protocol team considers it a realistic external behavior. [8](#0-7) 

---

### Recommendation

1. **Add a balance-delta check in `supplyCollateral()`**: Record `balanceBefore`, execute the transfer, then assert `balanceAfter - balanceBefore == assets`. Revert if the received amount is less than requested.
2. **Apply the same pattern to `repay()`**: Verify the actual received amount equals `units` before crediting `withdrawable`.
3. **Apply the same pattern to `flashLoan()` repayment**: Verify the returned amount equals the loaned amount.
4. Alternatively, document and enforce a token registry or market creation gate that rejects fee-on-transfer tokens at market creation time.

### Proof of Concept

**Setup:**
- Deploy a mock ERC-20 collateral token `FeeToken` that deducts 5% on every `transferFrom`.
- Create a Midnight market with `FeeToken` as the collateral, LLTV = 0.8.

**Steps:**

1. Attacker calls `supplyCollateral(market, 0, 1000e18, attacker)`.
   - `_position.collateral[0]` is set to `1000e18`.
   - Contract receives `950e18` (5% fee deducted).

2. Attacker calls `take()` on a sell offer to borrow loan tokens against the inflated `1000e18` collateral.
   - `isHealthy()` computes `maxDebt = 1000e18 * price * 0.8`, allowing borrowing against phantom `50e18`.

3. Attacker defaults (does not repay).

4. Liquidator calls `liquidate()` with `seizedAssets = 1000e18`.
   - `_position.collateral[0]` is `1000e18` in accounting.
   - `SafeTransferLib.safeTransfer(collateralToken, receiver, 1000e18)` — but the contract only holds `950e18`.
   - Transfer reverts OR (if the contract holds other users' tokens of the same type) drains other users' collateral.

5. Even if the liquidation succeeds by consuming other depositors' tokens, the `badDebt` path in `liquidate()` socializes the shortfall across all lenders via `lossFactor`, causing lenders to lose credit proportional to the phantom collateral. [9](#0-8) [10](#0-9) [11](#0-10) [12](#0-11)

        external
    {
        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
        bytes32 id = touchMarket(market);
        address collateralToken = market.collateralParams[collateralIndex].token;

        Position storage _position = position[id][onBehalf];
        uint256 oldCollateral = _position.collateral[collateralIndex];
        _position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);

        if (oldCollateral == 0 && assets > 0) {
            uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
            _position.collateralBitmap = newCollateralBitmap;
            require(
                UtilsLib.countBits(newCollateralBitmap) <= MAX_COLLATERALS_PER_BORROWER, TooManyActivatedCollaterals()
            );
        }

        emit EventsLib.SupplyCollateral(msg.sender, id, collateralToken, assets, onBehalf);

        SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
    }
