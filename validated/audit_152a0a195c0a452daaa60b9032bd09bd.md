Audit Report

## Title
`consumableUnits` Missing Tick Accessibility Check Enables Malicious Maker to Grief Bundler Takers via Silent `OutOfOffers` - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary
`ConsumableUnitsLib.consumableUnits` returns a positive unit count for any offer with remaining capacity without verifying that `offer.tick` is divisible by the market's `tickSpacing`. When the bundler passes this non-zero unit count to `IMidnight.take()`, the core contract unconditionally reverts with `TickNotAccessible` at line 351 of `src/Midnight.sol`. The bundler's `catch {}` block silently swallows this revert, `filledUnits` is not incremented, and the transaction ultimately reverts with `OutOfOffers`. A malicious maker can sign and publish offers at inaccessible ticks to reliably cause any taker's bundler transaction to fail.

## Finding Description

**Root cause — `consumableUnits` performs no tick check:**

`ConsumableUnitsLib.consumableUnits` computes remaining capacity purely from `consumed`, `maxUnits`, and `maxAssets`: [1](#0-0) 

There is no `offer.tick % tickSpacing == 0` guard anywhere in this function or in `TakeAmountsLib`.

**Core contract enforces the tick check unconditionally inside `take()`:**

`src/Midnight.sol` line 351 fires before any units-related logic, meaning even a call with a valid `units` value reverts for an inaccessible tick: [2](#0-1) 

**Bundler swallows the revert across all four entry points:**

All four bundler functions follow the same pattern — compute `unitsToTake` using `consumableUnits`, attempt `take()` inside a `try`, and silently discard any revert in `catch {}`: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

The NatDoc on each function explicitly states: *"Skips every reason why take can revert (including ones that are not asynchrony related)"*, confirming the catch is intentionally broad and not limited to asynchrony-related failures.

After the loop, the unfilled target triggers: [7](#0-6) 

**Exploit flow:**
1. Market has `tickSpacing = 4` (the `DEFAULT_TICK_SPACING` used in all markets).
2. Malicious maker signs `Offer{tick: 2921, maxUnits: type(uint256).max, ...}` — valid EIP-712 data, passes ratifier checks.
3. Offer is published to the off-chain order book.
4. Taker's aggregator/frontend fetches it and includes it in `takes[]` passed to any bundler entry point.
5. `consumableUnits` returns `type(uint256).max` (no tick check).
6. `unitsToTake` is non-zero.
7. `take()` reverts with `TickNotAccessible` at `src/Midnight.sol:351`.
8. `catch {}` swallows the revert; `filledUnits` is not incremented.
9. If no other valid offers cover `targetUnits`, the bundler reverts with `OutOfOffers`.

## Impact Explanation
A malicious maker can reliably cause any taker's bundler transaction to revert with `OutOfOffers` by polluting the order book with inaccessible-tick offers. The taker loses gas and fails to complete their intended borrow or lend operation. Because the bundler is the documented integration path for takers, this constitutes concrete, repeatable service unavailability through normal protocol usage. The attack requires no on-chain state and is free beyond signing a message.

## Likelihood Explanation
All preconditions are trivially satisfiable by any unprivileged actor: (1) all markets start with `DEFAULT_TICK_SPACING = 4`; (2) any maker can sign an offer with an arbitrary `tick` value — there is no on-chain validation at offer creation time; (3) automated taker systems (aggregators, frontends) that fetch from open order books will naturally include such poisoned offers. The attack is repeatable at zero marginal cost and is undetectable by `consumableUnits` as currently written. [8](#0-7) 

## Recommendation
Add a tick accessibility check in `ConsumableUnitsLib.consumableUnits` before computing capacity:

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    MarketState memory state = IMidnight(midnight).marketState(id);
    if (offer.tick % state.tickSpacing != 0) return 0;  // inaccessible tick → skip
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    ...
}
```

Alternatively, add the same guard in the bundler loop before computing `unitsToTake`, returning 0 for inaccessible ticks so the offer is naturally skipped. Either location prevents `TickNotAccessible` from ever reaching the `catch {}` block with a non-zero `unitsToTake`. [1](#0-0) 

## Proof of Concept
**Minimal fork test plan:**
1. Deploy Midnight with a market where `tickSpacing = 4`.
2. Malicious maker signs `Offer{tick: 2921, maxUnits: type(uint256).max, buy: false, ...}` (2921 % 4 = 1, inaccessible).
3. Legitimate maker signs a valid offer at `tick = 2920` covering only half of `targetUnits`.
4. Call `buyWithUnitsTargetAndWithdrawCollateral` with `takes = [poisonedOffer, legitimateOffer]` and `targetUnits` equal to the full amount.
5. Observe: `consumableUnits` returns `type(uint256).max` for the poisoned offer; `take()` reverts with `TickNotAccessible`; `catch {}` swallows it; `filledUnits` only reaches half of `targetUnits`; transaction reverts with `OutOfOffers`.
6. Confirm: removing the poisoned offer from `takes[]` allows the transaction to succeed with the legitimate offer alone (if it covers `targetUnits`). [2](#0-1)
