Audit Report

## Title
`sellerAssetsToUnits` reverts with arithmetic underflow when `sellerPrice == 0` on a valid buy offer at the settlement fee boundary - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and then calls `targetSellerAssets.mulDivUp(WAD, sellerPrice)`. When `offerPrice == settlementFee`, `sellerPrice` is zero and `UtilsLib.mulDivUp` evaluates `(d - 1)` with `d = 0`, triggering an arithmetic underflow revert under Solidity 0.8 checked arithmetic. Because `supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside its `try/catch` block, the revert propagates and DoSes the entire bundler transaction for any user whose `takes[]` array includes such an offer.

## Finding Description

**Root cause — `UtilsLib.mulDivUp` with `d = 0`:**

`UtilsLib.mulDivUp` is implemented as:

```solidity
return (x * y + (d - 1)) / d;
``` [1](#0-0) 

With `d = 0`, the sub-expression `(d - 1)` underflows under Solidity 0.8 checked arithmetic before any division occurs.

**`TakeAmountsLib.sellerAssetsToUnits` — zero denominator path:**

For buy offers, `sellerPrice` is computed as `offerPrice - settlementFee`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...
``` [2](#0-1) 

When `offer.buy == true` and `tickToPrice(offer.tick) == settlementFee(id, ttm)`, line 44 yields `sellerPrice = 0`. Line 46 then calls `targetSellerAssets.mulDivUp(WAD, 0)` → arithmetic revert. The NatSpec at line 34 only documents the `offerPrice < settlementFee` revert (underflow in subtraction), not the `offerPrice == settlementFee` case (zero denominator in the inverse): [3](#0-2) 

**Why `Midnight.take()` accepts this offer:**

In the core protocol, `sellerPrice = 0` is used in the forward direction via `mulDivDown`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...
``` [4](#0-3) 

`units.mulDivDown(0, WAD) = (units * 0) / WAD = 0` — no revert. The seller receives zero assets and all buyer payment goes to the settlement fee. The core protocol does not reject `offerPrice == settlementFee` for buy offers; only `offerPrice < settlementFee` causes an underflow revert at line 361.

**Why the bundler call reverts entirely:**

`supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` at line 286 **outside** the `try/catch` block that wraps `IMidnight.take()`: [5](#0-4) 

The `try/catch` at line 292 only catches reverts from `take()` itself. The NatSpec at line 246 explicitly acknowledges this: [6](#0-5) 

The loop condition `filledSellerAssets < targetFilledSellerAssets` (line 282) guarantees `targetFilledSellerAssets - filledSellerAssets > 0` when `sellerAssetsToUnits` is called, so `targetSellerAssets > 0` always — the zero-numerator escape does not apply. [7](#0-6) 

**Fuzz test coverage gap:**

All fuzz tests in `TakeAmountsTest.sol` set `offer.buy = false` in `setUp()` and never exercise the buy-offer path through `sellerAssetsToUnits`: [8](#0-7) 

The `_maxTick` helper bounds ticks to ensure `buyerPrice <= WAD` for sell offers, but does not exclude `sellerPrice == 0` for buy offers: [9](#0-8) 

## Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee(id, ttm)` at call time reverts entirely. The taker's authorized bundler flow is completely blocked for that transaction. Collateral already supplied within the same call (lines 269–275) is locked in the bundler until the transaction reverts and unwinds. The attack is a permissionless, repeatable griefing DoS against any user of the `supplyCollateralAndSellWithAssetsTarget` bundler function. [10](#0-9) 

## Likelihood Explanation
**Preconditions:**
1. A buy offer exists with `tickToPrice(offer.tick) == settlementFee(id, ttm)` at the time of the bundler call.
2. A victim taker has authorized the bundler and includes this offer in `takes[]`.

**Feasibility:** Settlement fees are set in multiples of `1e12` and tick prices are discrete values from `tickToPrice`. An attacker can enumerate tick prices and find one matching the current or near-future settlement fee. The condition is repeatable: the attacker can create a new offer each time the settlement fee drifts to a matching tick price. No privileged access is required — any address can create a buy offer.

## Recommendation
The most targeted fix is to guard against `sellerPrice == 0` inside `sellerAssetsToUnits`. When `sellerPrice == 0`, the seller receives zero assets regardless of units, so no finite number of units can yield a positive `targetSellerAssets`; the function should revert with a descriptive error (e.g., `SellerPriceIsZero`) rather than underflowing. Alternatively, move the `TakeAmountsLib.sellerAssetsToUnits` call inside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget` so that a revert from the library causes the offer to be skipped rather than the entire transaction to revert — consistent with the bundler's stated design intent of skipping reverts. [11](#0-10) 

## Proof of Concept
Minimal Foundry test:
1. Deploy the protocol with a market where `settlementFee(id, ttm) = S` for some `S > 0`.
2. Find tick `t` such that `tickToPrice(t) == S` (enumerate ticks; settlement fees are multiples of `1e12`).
3. Attacker creates a buy offer with `offer.tick = t`.
4. Victim calls `supplyCollateralAndSellWithAssetsTarget` with `takes[0]` pointing to the attacker's offer.
5. Observe: the call reverts with arithmetic underflow originating from `UtilsLib.mulDivUp` inside `sellerAssetsToUnits`, before `take()` is ever reached.

Alternatively, add a fuzz test to `TakeAmountsTest.sol` that sets `offer.buy = true`, sets `settlementFee` equal to `tickToPrice(offer.tick)`, and calls `TakeAmountsLib.sellerAssetsToUnits` — it will revert, confirming the bug.
