Audit Report

## Title
Fully-consumed assets-based buy offer with `buyerPrice < WAD` can be taken indefinitely with zero asset transfer but non-zero position mutations - (File: src/Midnight.sol)

## Summary
When a buy offer with `maxAssets > 0` is fully consumed and `buyerPrice < WAD`, calling `take()` with `units = 1` causes `mulDivDown(1, buyerPrice, WAD)` to truncate to `0`. The consumed counter does not advance, so the `ConsumedAssets` guard passes indefinitely, while all position mutations (`credit`, `debt`, `totalUnits`) execute on the full `units = 1` value with zero token transfer. The protocol explicitly acknowledges this behavior at line 94 and the test `testBugBuyMaxAssetsBypass` reproduces it exactly.


When `buyerPrice < WAD` (e.g., `tick = MAX_TICK - 16`) and `units = 1`, `buyerAssets = 0`. Adding `0` to an already-maxed `consumed` value satisfies `<= maxAssets`, so the guard passes. Position mutations at lines 408â€“417 then execute unconditionally on `units = 1`, and the token transfer at line 455 sends `buyerAssets - sellerAssets = 0` tokens.

**Protocol acknowledgement at line 94:** `"It is possible to give units to a fully consumed assets-based buy offer with price < 1."` â€” confirming the behavior is real, not a misread.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` â€” passes because `maxAssets + 0 â‰¤ maxAssets`
- No check enforces `units == 0` when `consumed >= maxAssets`
- No check enforces `buyerAssets > 0` when `units > 0`

## Impact Explanation
After an offer reaches `consumed == maxAssets`, any unprivileged taker can call `take(offer, ..., units=1)` indefinitely. Each call increases the buyer/maker's `credit`, the seller/taker's `debt`, and `totalUnits` by amounts derived from `units = 1`, while transferring zero loan tokens. This corrupts the credit/debt accounting invariant and inflates `totalUnits` without any corresponding asset backing, at zero token cost to the attacker. The test `testBugBuyMaxAssetsBypass` explicitly asserts `buyerAssets == 0`, `sellerAssets == 0`, token balances unchanged, yet `creditOf`, `debtOf`, and `totalUnits` all strictly increase.

## Likelihood Explanation
All preconditions are attacker-reachable without any privileged access:
1. `offer.buy = true` and `offer.maxAssets > 0` â€” standard offer configuration
2. `buyerPrice < WAD` â€” achievable at any tick where `tickToPrice(tick) + settlementFee < WAD` (e.g., `MAX_TICK - 16`)
3. Offer fully consumed â€” attacker can self-consume as first taker, or wait for organic fills

The attack is repeatable in every block at negligible gas cost since no token transfers occur.

## Recommendation
After computing `buyerAssets`, add a guard that rejects zero-asset takes when units are nonzero and the offer is assets-capped:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(units == 0 || buyerAssets > 0, ZeroAssetTake());
}
```

Alternatively, enforce that the consumed counter strictly advances when `units > 0`: require `newConsumed > consumed[offer.maker][offer.group]` before the update. A simpler approach is to pre-check `consumed[offer.maker][offer.group] < offer.maxAssets` before entering the take logic when `maxAssets > 0`.
