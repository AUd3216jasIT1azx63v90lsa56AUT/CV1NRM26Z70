Audit Report

## Title
Authorized agent can grind victim's nonce in `EcrecoverAuthorizer` to permanently invalidate all pending pre-signed authorizations - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` increments `nonce[authorization.authorizer]` on every successful call, but the authorization check accepts any signer for whom `IMidnight.isAuthorized(authorizer, signer)` returns `true`, not only the authorizer's own key. An attacker who holds a valid on-chain authorization from a victim can submit an unbounded sequence of no-op re-authorizations of themselves, each consuming one nonce slot at gas cost only, permanently invalidating every pre-signed authorization the victim distributed offline.

## Finding Description
**Exact code path — `src/periphery/EcrecoverAuthorizer.sol` lines 24–48:**

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // line 26
// ...
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer), // lines 33-36
    Unauthorized()
);
IMidnight(MIDNIGHT)
    .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer); // lines 46-47
```

**Root cause.** The nonce slot is keyed on `authorization.authorizer` (the victim), but the authorization check accepts any `signer` for whom `IMidnight.isAuthorized(victim, signer)` is `true`. There is no requirement that `signer == authorization.authorizer`. The nonce increment on line 26 is committed whenever the full call succeeds, including no-op writes.

**Exploit flow (N iterations).**

Preconditions:
1. `isAuthorized[victim][EcrecoverAuthorizer] == true` — required for `Midnight.setIsAuthorized` (called at line 47) to not revert; this is the normal operational state for any user of the contract.
2. `isAuthorized[victim][attacker] == true` — victim has granted the attacker on-chain authorization for any purpose.

For `i = 0 … N-1`:
1. Attacker constructs `Authorization(authorizer=victim, authorized=attacker, isAuthorized=true, nonce=i, deadline=future)`.
2. Attacker signs the EIP-712 digest with their own private key → `signer = attacker`.
3. Attacker calls `EcrecoverAuthorizer.setIsAuthorized(auth, sig)`.
4. Line 25: deadline check passes.
5. Line 26: `i == nonce[victim]` → passes; `nonce[victim]` becomes `i+1`.
6. Lines 33–36: `attacker != victim`; `isAuthorized[victim][attacker] == true` → passes.
7. Line 47: `Midnight.setIsAuthorized(attacker, true, victim)` → no-op write (attacker already authorized).

After N calls: `nonce[victim] == N`. Every pre-signed authorization the victim distributed with `nonce < N` reverts with `InvalidNonce()`.

**Why existing checks fail.** The `Unauthorized` guard is satisfied by the attacker's own valid ECDSA signature because the delegation branch is true. The nonce check enforces sequential ordering only; it does not restrict which party may consume nonce slots. No check requires `signer == authorization.authorizer`.

## Impact Explanation
All pending pre-signed authorizations the victim has distributed offline (for `take`, `repay`, `withdraw`, `withdrawCollateral`, `liquidate`, `setConsumed`, or any other action gated on `isAuthorized`) become permanently invalid. Counterparties holding those signatures cannot execute them. The attacker can immediately repeat the attack after any re-signing, creating a sustained, low-cost denial-of-service against every signature-based action in the protocol. This constitutes permanent corruption of user state and severe service degradation under realistic attacker input.

## Likelihood Explanation
Both preconditions are normal operational states for any user relying on the signature-based authorization flow: authorizing `EcrecoverAuthorizer` is required to use it, and authorizing other agents (e.g., smart contracts, keepers, counterparties) is the protocol's intended delegation model. The attacker requires no capital, no special role, and no privileged access beyond the on-chain authorization the victim already granted. The attack costs only gas and is repeatable indefinitely. The attacker can front-run any revocation attempt by submitting additional nonce-consuming transactions.

## Recommendation
Restrict nonce consumption to the authorizer's own key. The simplest fix is to require `signer == authorization.authorizer` unconditionally, removing the delegation branch from `EcrecoverAuthorizer.setIsAuthorized`. If agent-signed authorizations are intentionally supported, the nonce must be keyed on the signer (not the authorizer), or a separate per-(authorizer, signer) nonce mapping must be used, so that one agent cannot advance another party's nonce counter.

## Proof of Concept
1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. As victim: call `Midnight.setIsAuthorized(EcrecoverAuthorizer, true, victim)` and `Midnight.setIsAuthorized(attacker, true, victim)`.
3. Victim pre-signs `Authorization(authorizer=victim, authorized=thirdParty, isAuthorized=true, nonce=0, deadline=future)` and distributes it offline.
4. As attacker: construct `Authorization(authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=future)`, sign with attacker key, call `EcrecoverAuthorizer.setIsAuthorized`. Observe `nonce[victim] == 1`.
5. Attempt to submit victim's pre-signed authorization (nonce=0): reverts with `InvalidNonce()`.
6. Repeat step 4 with `nonce=1, 2, …` to permanently outpace any re-signing by the victim.
