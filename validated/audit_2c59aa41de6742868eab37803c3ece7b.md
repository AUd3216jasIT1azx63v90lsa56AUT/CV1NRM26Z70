### Title
Settlement Fee Rounds to Zero on Small `take()` Calls, Enabling Fee-Free Trading - (File: src/Midnight.sol)

---

### Summary

In `Midnight.sol`'s `take()` function, the settlement fee accrued to the protocol is computed as the arithmetic difference `buyerAssets - sellerAssets`, where each term is independently rounded via `mulDivDown`/`mulDivUp`. When `units * _settlementFee < WAD (1e18)`, both terms round to the same integer and the protocol collects zero fee, even though a real trade with real asset transfers occurs. Any unprivileged user can exploit this by chunking trades into small `units` values.

---

### Finding Description

**Root cause — independent rounding of buyer and seller assets:** [1](#0-0) 

```solidity
uint256 _settlementFee = settlementFee(id, timeToMaturity);
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;
uint256 buyerAssets  = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD);
```

The fee credited to the protocol is: [2](#0-1) 

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
```

Because `buyerAssets` and `sellerAssets` are each floored (or ceiled) independently, the difference can be zero even when `_settlementFee > 0`. The condition for zero fee is:

```
units * _settlementFee < WAD  (= 1e18)
```

The minimum possible settlement fee is `1 CBP = 1e12` (one centi-basis-point): [3](#0-2) 

```solidity
uint256 constant CBP = 1e12;
```

So with a 1-cbp fee, any call with `units < 1e18 / 1e12 = 1e6` pays **zero** settlement fee. There is no minimum-units guard anywhere in `take()`.

The protocol's own ROUNDINGS comment acknowledges this: [4](#0-3) 

> "the settlement fee has no defined rounding direction, which could lead to fees manipulations on chains with very cheap gas."

The acknowledgment confirms the root cause but does not constitute a fix or an explicit out-of-scope declaration.

---

### Impact Explanation

A malicious taker can accumulate an arbitrarily large credit or debt position while paying **zero** settlement fees by splitting a single large `take()` into many small calls each with `units < WAD / _settlementFee`.

- The protocol's `feeClaimer` receives no settlement fee revenue for those trades.
- The taker still receives full credit/debt units and the maker still receives/pays full `sellerAssets` — only the protocol's cut is zeroed out.
- The continuous fee (`buyerPendingFeeIncrease`) is also rounded down via `mulDivDown` for small `buyerCreditIncrease`, compounding the fee loss: [5](#0-4) 

```solidity
uint128 buyerPendingFeeIncrease =
    UtilsLib.toUint128(buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD));
```

**Impact: Low** — only protocol fee revenue is lost; no user funds are at risk.

---

### Likelihood Explanation

- **No privileged access required.** Any taker can call `take()` with an arbitrarily small `units` value.
- **Economically viable on L2s.** On Arbitrum, Base, or Optimism, gas per call is a fraction of a cent, making thousands of micro-calls profitable even for small fee savings.
- **Trivially scriptable.** A bot can split any desired trade size into chunks of `units = WAD / _settlementFee - 1` automatically.

**Likelihood: High** on cheap-gas EVM chains (L2s), where the protocol is explicitly designed to operate.

---

### Recommendation

Add a minimum-fee guard inside `take()` after computing `buyerAssets` and `sellerAssets`:

```solidity
uint256 feeCollected = buyerAssets - sellerAssets;
require(
    _settlementFee == 0 || feeCollected > 0,
    SettlementFeeRoundsToZero()
);
```

Alternatively, compute the fee directly from `units` before rounding the asset amounts:

```solidity
uint256 feeUnits = units.mulDivUp(_settlementFee, WAD); // round up in protocol's favor
require(feeUnits > 0 || _settlementFee == 0, SettlementFeeRoundsToZero());
```

This mirrors the recommendation in the referenced external report: reject inputs where the fee calculation would round to zero.

### Proof of Concept

**Setup:**
- Market with `loanToken = USDC` (6 decimals), settlement fee = 1 cbp (`_settlementFee = 1e12`).
- Offer: buy offer, `offerPrice = WAD` (tick = 0, price = 1.0).

**Exploit call (repeated N times):**
```
take(offer, ..., units = 999_999, ...)
```

**Arithmetic (buy offer path):**
```
buyerPrice  = 1e18 + 1e12
sellerPrice = 1e18

buyerAssets  = floor(999_999 * (1e18 + 1e12) / 1e18)
             = floor(999_999 + 0.000999999)
             = 999_999

sellerAssets = floor(999_999 * 1e18 / 1e18)
             = 999_999

claimableSettlementFee += 999_999 - 999_999 = 0
```

**Result:** The taker acquires `999_999` credit units per call, the maker receives `999_999` USDC, and the protocol receives **0** settlement fee. Repeating 1,000 times yields ~1 USDC of credit with zero fees paid. On an L2 at 0.001 USD/tx, total gas cost is ~$1 — break-even at any fee rate above ~0.1 bps. [6](#0-5)
