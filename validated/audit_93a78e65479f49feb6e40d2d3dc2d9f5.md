Audit Report

## Title
Referral fee not budgeted within `maxBuyerAssets`, causing revert that blocks a valid fill - (File: src/periphery/MidnightBundles.sol)

## Summary
In `buyWithUnitsTargetAndWithdrawCollateral`, the bundler pulls `maxBuyerAssets` from `msg.sender` and the take loop drains `filledBuyerAssets` of that balance via MIDNIGHT. The referral fee (`referralFeeAssets`) is then computed on top of `filledBuyerAssets` and transferred from the bundler's remaining balance. When `maxBuyerAssets == filledBuyerAssets` and `referralFeePct > 0`, the bundler holds zero tokens after the loop, causing the `safeTransfer` to the referrer at line 103 to revert with an ERC20 insufficient-balance error. The fill itself succeeded, but the entire transaction is rolled back.

## Finding Description
**Code path:**

`src/periphery/MidnightBundles.sol`, function `buyWithUnitsTargetAndWithdrawCollateral`:

1. Line 66 pulls exactly `maxBuyerAssets` from `msg.sender` into the bundler. Line 67 grants `type(uint256).max` approval to MIDNIGHT.
2. Lines 79–85: the take loop calls `MIDNIGHT.take(...)`, which pulls `resBuyerAssets` from the bundler per iteration, accumulating into `filledBuyerAssets`. After the loop the bundler holds `maxBuyerAssets - filledBuyerAssets` tokens.
3. Line 102: `referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct)`. When `referralFeePct > 0` and `filledBuyerAssets > 0`, this is strictly positive.
4. Line 103: `if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets)`. If `maxBuyerAssets == filledBuyerAssets`, the bundler's balance is zero here; the ERC20 transfer fails and `safeTransfer` propagates the revert.
5. Line 104: `SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - filledBuyerAssets - referralFeeAssets)` would also underflow under Solidity 0.8.34 checked arithmetic, but is never reached because line 103 reverts first.

**Root cause:** No check enforces `maxBuyerAssets >= filledBuyerAssets + referralFeeAssets`. The NatSpec at lines 47–48 correctly documents the total cost formula (`filledBuyerAssets + filledBuyerAssets * referralFeePct / (WAD - referralFeePct)`), but the contract never validates that `maxBuyerAssets` is large enough to cover both components. By contrast, the sibling functions `buyWithAssetsTargetAndWithdrawCollateral` (line 200–201) and `supplyCollateralAndSellWithAssetsTarget` (line 277–278) compute the referral fee upfront and incorporate it into the fill target, avoiding this class of failure entirely.

**Existing guards:** Only `referralFeePct < WAD` (line 61) and taker authorization (line 60) are checked. Neither prevents the scenario where `maxBuyerAssets` is set tightly to `filledBuyerAssets`.

## Impact Explanation
All offers are successfully taken (`filledUnits == targetUnits` passes at line 88) and collateral withdrawals complete, but the transaction reverts at the referral-fee transfer (line 103), rolling back all state changes. The user loses gas and cannot complete an economically valid fill in a single call. Because the revert is deterministic for the given `(maxBuyerAssets, referralFeePct)` pair, the user is blocked until they either increase `maxBuyerAssets` to cover the referral fee or set `referralFeePct = 0`. This constitutes a service-availability failure and a denial-of-fill for any user who follows the common tight-slippage pattern.

## Likelihood Explanation
Any unprivileged taker can trigger this by setting `maxBuyerAssets` equal to the expected fill cost (computed off-chain from offer prices) and passing any `referralFeePct > 0`. No special state, privileged role, or external dependency is required. The precondition is trivially reachable and repeatable across every call with these parameters. Tight slippage is a standard integration pattern, making accidental triggering highly probable for integrators who do not account for the referral fee in `maxBuyerAssets`.

## Recommendation
Compute `referralFeeAssets` before the fill loop and validate that `maxBuyerAssets` is sufficient to cover both the fill and the fee, consistent with the approach used in `buyWithAssetsTargetAndWithdrawCollateral`. One concrete fix: after line 102, add `require(maxBuyerAssets >= filledBuyerAssets + referralFeeAssets, InsufficientBuyerAssets())` before the transfer at line 103. Alternatively, mirror the `buyWithAssetsTargetAndWithdrawCollateral` pattern by computing the referral fee upfront and using `maxBuyerAssets - referralFeeAssets` as the effective cap for the fill loop, so the fee is always pre-reserved.

## Proof of Concept
1. Deploy `MidnightBundles` against a local fork with a valid market and a sell offer priced at exactly 1000 loan tokens for `targetUnits`.
2. Call `buyWithUnitsTargetAndWithdrawCollateral` with:
   - `targetUnits` = offer's unit count
   - `maxBuyerAssets` = 1000 (equal to expected `filledBuyerAssets`)
   - `referralFeePct` = 1e16 (1%, i.e., `WAD / 100`)
   - `referralFeeRecipient` = any non-zero address
3. Observe: the take loop succeeds, `filledUnits == targetUnits` passes, but the transaction reverts at line 103 with an ERC20 insufficient-balance error because `referralFeeAssets = 1000 * 1e16 / (1e18 - 1e16) ≈ 10` and the bundler holds 0 tokens.
4. Confirm fix: retry with `maxBuyerAssets` = 1011 (≥ 1000 + 10); transaction succeeds and 10 tokens are sent to the referrer with 1 token returned to `msg.sender`.
