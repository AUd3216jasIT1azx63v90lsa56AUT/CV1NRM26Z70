Audit Report

## Title
Nonce-Race Front-Run on `EcrecoverAuthorizer.setIsAuthorized` Enables Unauthorized Withdrawal - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` is fully permissionless and encodes `isAuthorized` into the EIP-712 signed digest, making a grant (`isAuthorized=true`) and a revocation (`isAuthorized=false`) for the same nonce two cryptographically distinct but independently valid messages. An attacker holding a victim-signed grant for nonce=N can front-run the victim's mempool revocation (also nonce=N) by submitting the grant first, consuming the nonce and causing the revocation to revert with `InvalidNonce()`. The attacker remains authorized and can immediately call `Midnight.withdraw` to drain the victim's withdrawable loan-token balance.

## Finding Description

**Permissionless entry point:** `setIsAuthorized` performs no `msg.sender` check; any caller may submit any signed authorization. [1](#0-0) 

**Nonce consumed atomically on first use:** The nonce slot is incremented on the first message that uses it, regardless of whether that message is a grant or a revocation. [2](#0-1) 

**`isAuthorized` is a field in the signed struct:** Because `isAuthorized` is part of `Authorization`, `{victim, attacker, true, N, far}` and `{victim, attacker, false, N, ...}` produce two cryptographically distinct digests that both carry a valid victim signature and both satisfy `nonce == N`. The nonce mechanism prevents replay of the *same* message; it does not prevent two *different* messages from racing on the same nonce slot. [3](#0-2) 

**Digest computed over the full struct including `isAuthorized`:** [4](#0-3) 

**Signature check only requires `ecrecover` to return `authorization.authorizer`:** The attacker's held grant carries a genuine victim signature, so `ecrecover` returns the victim's address and the `Unauthorized()` check passes. [5](#0-4) 

**Calls `Midnight.setIsAuthorized` with attacker-controlled `authorized` and `isAuthorized`:** [6](#0-5) 

**Exploit flow:**
1. Victim authorizes `EcrecoverAuthorizer` in Midnight and signs `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N, deadline=far}` for a relayer (the attacker), but the grant is never submitted.
2. Victim changes mind, signs `Authorization{..., isAuthorized=false, nonce=N}`, and broadcasts the revocation to the public mempool.
3. Attacker observes the revocation in the mempool and front-runs it by submitting the old grant at higher gas.
4. Attacker's grant executes: `nonce[victim]` becomes N+1; `isAuthorized[victim][attacker] = true` in Midnight.
5. Victim's revocation reverts: `authorization.nonce == N` but `nonce[victim] == N+1` → `InvalidNonce()`.
6. Attacker calls `Midnight.withdraw(market, units, victim, attacker)`.

**Why the direct-revoke escape hatch also fails:** The victim can call `Midnight.setIsAuthorized` directly to set `isAuthorized[victim][attacker] = false`, but this does not consume any nonce in `EcrecoverAuthorizer`. [7](#0-6) 

After the direct revoke, the attacker can still submit the signed grant via `EcrecoverAuthorizer` (nonce=N is still unconsumed), re-setting `isAuthorized[victim][attacker] = true`, provided `EcrecoverAuthorizer` remains authorized in Midnight. There is no mechanism to invalidate a signed grant without consuming its nonce, and nonce consumption can be front-run.

**Why existing checks fail:**
- `InvalidNonce()`: prevents replay of the *same* message, not two *different* signed messages (grant vs. revoke) racing on the same nonce.
- `Unauthorized()`: the attacker's grant carries a genuine victim signature (`ecrecover` returns `victim`), so this passes.
- `Expired()`: the attacker's grant carries a far deadline, so this passes.

## Impact Explanation

Once `isAuthorized[victim][attacker]` is `true`, the attacker calls `Midnight.withdraw`, which checks only `isAuthorized[onBehalf][msg.sender]`: [8](#0-7) 

The authorization check passes, `_position.credit` and `_marketState.withdrawable` are decremented, and loan tokens are transferred directly to the attacker-controlled `receiver`: [9](#0-8) 

This constitutes direct theft of the victim's withdrawable loan-token balance. The reduction in `_marketState.withdrawable` and `totalUnits` also reduces the pool's ability to cover legitimate lender redemptions.

## Likelihood Explanation

**Required preconditions:**
- Victim signed a grant off-chain and shared it with the attacker (standard in relayer/meta-transaction flows — the victim authorizes a relayer, then decides to revoke before the relayer submits).
- Victim attempts revocation via `EcrecoverAuthorizer` (the natural path for a gasless/meta-tx user who cannot pay gas directly).
- Attacker monitors the public mempool (standard capability on any EVM chain with a public mempool).

**Feasibility:** High. The attacker needs only to submit a higher-gas transaction with the already-held valid signature. No special privileges, no cryptographic breaks, no oracle manipulation. The attack is repeatable for each new nonce if the victim continues using `EcrecoverAuthorizer` for revocations.

## Recommendation

Remove `isAuthorized` from the signed struct and instead derive it from a separate, caller-supplied parameter that is not part of the digest. The signed message should commit only to *who* is being authorized and *which nonce* is being consumed, not to the direction of the authorization. Concretely:

1. Change the `Authorization` struct to remove `isAuthorized`.
2. Accept `isAuthorized` as a separate, unsigned calldata parameter in `setIsAuthorized`.
3. The digest then covers only `{authorizer, authorized, nonce, deadline}`, making grant and revocation the *same* signed message — the nonce is consumed by whichever direction the caller supplies, and a held grant cannot be replayed as a revocation or vice versa.

Alternatively, implement a separate `invalidateNonce` function that the authorizer can call directly (without a signature) to advance their nonce, invalidating any outstanding signed messages for that nonce slot.

## Proof of Concept

**Minimal Foundry test (extend `SetIsAuthorizedWithSigTest.sol`):**

```solidity
function testFrontRunRevocation() public {
    // 1. Victim authorizes EcrecoverAuthorizer in Midnight
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);

    // 2. Victim signs a grant for attacker (nonce=0, far deadline)
    Authorization memory grant = Authorization({
        authorizer: victim,
        authorized: attacker,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 365 days
    });
    Signature memory grantSig = signAuthorization(grant, victim);

    // 3. Victim signs a revocation for the same nonce and broadcasts it
    Authorization memory revoke = Authorization({
        authorizer: victim,
        authorized: attacker,
        isAuthorized: false,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory revokeSig = signAuthorization(revoke, victim);

    // 4. Attacker front-runs: submits grant first
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(grant, grantSig);
    assertEq(midnight.isAuthorized(victim, attacker), true);
    assertEq(ecrecoverAuthorizer.nonce(victim), 1);

    // 5. Victim's revocation reverts
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(revoke, revokeSig);

    // 6. Attacker drains victim's withdrawable balance
    vm.prank(attacker);
    midnight.withdraw(market, victimUnits, victim, attacker);
    assertEq(loanToken.balanceOf(attacker), victimUnits);
}
```
