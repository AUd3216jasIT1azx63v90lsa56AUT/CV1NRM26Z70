Looking at the ConfidencePool codebase for an analog to the wrong-formula vulnerability class (incorrect denominator/numerator in a rate/score calculation causing inflated values and incorrect distribution).

### Title
`_clampUserSums` Wrong Formula Inflates Bonus Score for Mixed Pre/Post-Risk Stakers, Breaking Last-Claimer Payouts — (`File: src/ConfidencePool.sol`)

### Summary
`_clampUserSums` resets a user's entire per-user stake-time accumulators to `riskWindowStart` whenever the weighted-average entry time falls below that threshold. When a user holds both pre-risk and post-risk deposits and the pre-risk weight dominates, the function incorrectly clamps the post-risk deposit's entry time down to `riskWindowStart` as well. This inflates the user's k=2 score above its correct value, causing the sum of all user scores to exceed `globalScore`, so later claimers cannot receive their full payout.

### Finding Description
`_clampUserSums` is called before every claim and on every new stake. Its purpose is to retroactively treat pre-risk deposits as if they entered at `riskWindowStart`, so that pre-risk capital earns no time weight. The implementation is:

```solidity
function _clampUserSums(address u) internal {
    uint256 start = riskWindowStart;
    uint256 stake_ = eligibleStake[u];
    if (start == 0 || stake_ == 0) return;
    if (userSumStakeTime[u] < stake_ * start) {   // ← fires when weighted avg < riskWindowStart
        userSumStakeTime[u] = stake_ * start;      // ← resets ALL deposits to riskWindowStart
        userSumStakeTimeSq[u] = stake_ * start * start;
    }
}
```

The condition `userSumStakeTime[u] < stake_ * start` is equivalent to:

```
Σ(aᵢ · tᵢ) < (Σaᵢ) · riskWindowStart
⟺  Σ(aᵢ · (tᵢ − riskWindowStart)) < 0
```

This is negative whenever pre-risk deposits dominate by weight. When it fires, the function sets both accumulators as if **every** deposit entered at `riskWindowStart`, including post-risk deposits whose actual entry time `t₂ > riskWindowStart`.

The correct behavior is to clamp only the pre-risk portion:

```
userSumStakeTime[u]   = Σ aᵢ · max(tᵢ, riskWindowStart)
userSumStakeTimeSq[u] = Σ aᵢ · max(tᵢ, riskWindowStart)²
```

The global accumulators (`snapshotSumStakeTime`, `snapshotSumStakeTimeSq`) are **correct**: `_markRiskWindowStart` eagerly resets them for all pre-risk stakers, and post-risk deposits add their actual `block.timestamp` to the globals. So `globalScore` (the denominator in `_bonusShare`) is correct, but the per-user numerator is inflated for any user with mixed deposits.

### Impact Explanation
Because `_bonusShare` computes `Math.mulDiv(userScore, snapshotTotalBonus, globalScore)`, and the inflated `userScore` can push the sum of all user scores above `globalScore`, the collective bonus entitlement calculated across all claimers exceeds `snapshotTotalBonus`. The first claimers drain the bonus balance; later claimers find insufficient tokens and their `safeTransfer` (which includes both principal and bonus in a single call) reverts, permanently locking their principal inside the pool. The severity scales with the size of the post-risk deposit and the time gap `t₂ − riskWindowStart`.

### Likelihood Explanation
The scenario requires a staker to hold a large pre-risk deposit and then add a smaller post-risk deposit during `UNDER_ATTACK` (which is explicitly allowed by `_assertDepositsAllowed`). This is a natural and documented usage pattern — a staker who deposited early and tops up once the risk window opens. No privileged role is needed; any unprivileged staker can trigger the condition.

### Recommendation
Replace the wholesale reset with a per-deposit-aware clamp that only adjusts the pre-risk contribution. Since individual deposit records are not stored, the correct approach is to compute the pre-risk excess and subtract it:

```solidity
function _clampUserSums(address u) internal {
    uint256 start = riskWindowStart;
    uint256 stake_ = eligibleStake[u];
    if (start == 0 || stake_ == 0) return;
    uint256 floor = stake_ * start;
    if (userSumStakeTime[u] < floor) {
        // Only the pre-risk portion is wrong; compute the deficit and correct it.
        uint256 deficit = floor - userSumStakeTime[u];
        // deficit = Σ aᵢ·(start − tᵢ) for pre-risk deposits only.
        // Correct sumTime: add deficit to bring pre-risk entries up to start.
        userSumStakeTime[u] = floor;
        // For sumTimeSq: each pre-risk deposit i contributes aᵢ·(start²−tᵢ²).
        // Without per-deposit records, the safe conservative fix is to store
        // the minimum correct value: treat all pre-risk stake as entering at start.
        // This requires tracking pre-risk vs post-risk stake separately at deposit time,
        // or storing the post-risk sums independently so they are never overwritten.
        userSumStakeTimeSq[u] = /* post-risk portion */ + stake_preRisk * start * start;
    }
}
```

The cleanest fix is to track `userPreRiskStake`, `userPostRiskSumStakeTime`, and `userPostRiskSumStakeTimeSq` separately at deposit time, so `_clampUserSums` can reconstruct the correct totals without iterating deposits.

### Proof of Concept
Concrete numbers (timestamps in seconds, amounts in tokens):

| Variable | Value |
|---|---|
| `riskWindowStart` | 1 000 |
| `T = riskWindowEnd` | 2 000 |
| Alice pre-risk deposit | 1 000 tokens at t₁ = 0 |
| Alice post-risk deposit | 100 tokens at t₂ = 1 500 |
| Bob pre-risk deposit | 100 tokens at t₁ = 500 |
| `snapshotTotalBonus` | 1 000 tokens |

**Global accumulators after `_markRiskWindowStart` + Alice's post-risk stake:**
- `snapshotTotalStaked` = 1 200
- `snapshotSumStakeTime` = 1 100·1 000 + 100·1 500 = 1 250 000
- `snapshotSumStakeTimeSq` = 1 100·1 000² + 100·1 500² = 1 325 000 000
- `globalScore` = 2 000²·1 200 − 2·2 000·1 250 000 + 1 325 000 000 = **1 125 000 000**

**Alice's per-user sums before clamp:**
- `userSumStakeTime[Alice]` = 1 000·0 + 100·1 500 = 150 000
- Clamp condition: 150 000 < 1 100·1 000 = 1 100 000 → **TRUE**
- After clamp: `userSumStakeTime[Alice]` = 1 100 000, `userSumStakeTimeSq[Alice]` = 1 100 000 000

**Alice's wrong score:**
= 2 000²·1 100 − 2·2 000·1 100 000 + 1 100 000 000 = **1 100 000 000**

**Alice's correct score** (clamping only t₁=0 → 1 000, keeping t₂=1 500):
= 2 000²·1 100 − 2·2 000·(1 000·1 000 + 100·1 500) + (1 000·1 000² + 100·1 500²)
= 4 400 000 000 − 4 600 000 000 + 1 225 000 000 = **1 025 000 000**

**Bob's score** (clamp fires, correct): **100 000 000**

Sum of wrong scores: 1 100 000 000 + 100 000 000 = **1 200 000 000 > globalScore 1 125 000 000**

Alice's bonus share: `mulDiv(1 100 000 000, 1 000, 1 125 000 000)` = **977 tokens**
Bob's bonus share: `mulDiv(100 000 000, 1 000, 1 125 000 000)` = **88 tokens**
Total claimed bonus: **1 065 tokens > 1 000 available**

If Alice claims first, she receives 977 bonus tokens. The pool then holds only 23 bonus tokens for Bob. Bob's `safeTransfer(Bob, 100_principal + 88_bonus)` reverts because the contract balance is `100 + 23 = 123 < 188`. Bob's principal is permanently locked. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function _bonusShare(address u, uint256 userEligible) internal view returns (uint256) {
        if (snapshotTotalBonus == 0) return 0;
        // No observable risk → no bonus (see contract natspec).
        if (riskWindowStart == 0) return 0;
        uint256 T = outcomeFlaggedAt;

        // Underflow guards on both subtractions: globally the sum of squares is nonneg, but
        // truncation/rounding pathologies could push individual terms over.
        uint256 userPlus = T * T * userEligible + userSumStakeTimeSq[u];
        uint256 userMinus = 2 * T * userSumStakeTime[u];
        uint256 userScore = userPlus > userMinus ? userPlus - userMinus : 0;

        uint256 plus = T * T * snapshotTotalStaked + snapshotSumStakeTimeSq;
        uint256 minus = 2 * T * snapshotSumStakeTime;
        uint256 globalScore = plus > minus ? plus - minus : 0;

        if (globalScore == 0) {
            // No time elapsed in the risk window for anyone → fallback to amount-weighted.
            if (snapshotTotalStaked == 0) return 0;
            return Math.mulDiv(userEligible, snapshotTotalBonus, snapshotTotalStaked);
        }
        // mulDiv handles the final multiply-then-divide via 512-bit intermediates, so a very
        // large `snapshotTotalBonus` cannot push the numerator over uint256 before division.
        return Math.mulDiv(userScore, snapshotTotalBonus, globalScore);
```

```text
    function _markRiskWindowStart() internal {
        // Cap at expiry: accrual is bounded by the pool's lifecycle. Without the cap, a late
        // observation could pin riskWindowStart > expiry, and `_clampUserSums` would record every
        // pre-risk deposit as entering past T = expiry (EXPIRED path). The k=2 sums then encode
        // post-deadline "at-risk" time that never actually existed.
        uint256 t = block.timestamp;
        if (t > expiry) t = expiry;
        // Cast is truncation-safe: `t` is capped at `expiry`, which is itself a uint32.
        // forge-lint: disable-next-line(unsafe-typecast)
        riskWindowStart = uint32(t);
        // Eagerly reset the global accumulators so every currently-eligible staker is treated as
        // entering at `t`. Per-user sums stay stale until `_clampUserSums` runs on the next op
        // touching that user.
        sumStakeTime = totalEligibleStake * t;
        sumStakeTimeSq = totalEligibleStake * t * t;
        emit RiskWindowStarted(t);
```
