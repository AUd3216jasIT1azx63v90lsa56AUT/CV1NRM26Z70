### Title
`_clampUserSums` Wholesale Replacement Overestimates Per-User k=2 Score for Mixed Pre/Post-Risk Depositors, Causing Claim Revert and Permanent Principal Lock - (File: src/ConfidencePool.sol)

### Summary

`_clampUserSums` uses a single wholesale replacement to update a user's per-user accumulators when a pre-risk deposit is detected. When a user has both pre-risk and post-risk deposits, the replacement incorrectly clamps the post-risk portion to `riskWindowStart` as well, producing a `userSumStakeTime` that is lower than the correct value. Because the global snapshot correctly reflects the post-risk deposit at its actual entry time, the user's computed score exceeds the global score, making `bonusShare > snapshotTotalBonus`. The resulting `payout` overflows the contract's available balance and `safeTransfer` reverts, permanently locking the user's principal.

### Finding Description

`_clampUserSums` is designed to lazily promote stale pre-risk per-user accumulators to the `riskWindowStart` floor:

```solidity
function _clampUserSums(address u) internal {
    uint256 start = riskWindowStart;
    uint256 stake_ = eligibleStake[u];
    if (start == 0 || stake_ == 0) return;
    if (userSumStakeTime[u] < stake_ * start) {
        userSumStakeTime[u] = stake_ * start;
        userSumStakeTimeSq[u] = stake_ * start * start;
    }
}
``` [1](#0-0) 

The condition `userSumStakeTime[u] < stake_ * start` fires whenever the user's accumulated `Σ(amount × entryTime)` is below `totalStake × riskWindowStart`. This is correct for a pure pre-risk depositor (all entries below `riskWindowStart`). However, when a user has **both** a pre-risk deposit and a later post-risk deposit (permitted during `UNDER_ATTACK` per DESIGN.md §3), the post-risk deposit's contribution to `userSumStakeTime` is at `postTime > riskWindowStart`, which partially offsets the pre-risk deficit but may not fully close it. The condition still triggers, and the replacement sets:

```
userSumStakeTime[u]   = eligibleStake × riskWindowStart
userSumStakeTimeSq[u] = eligibleStake × riskWindowStart²
```

This treats **all** of the user's stake as entering at `riskWindowStart`, including the post-risk tranche that should retain its actual `postTime > riskWindowStart`.

Meanwhile, `_markRiskWindowStart` eagerly resets the **global** accumulators to `totalEligibleStake × riskWindowStart`, and subsequent post-risk `stake()` calls correctly add `received × postTime` to both global sums:

```solidity
sumStakeTime = totalEligibleStake * t;          // eager reset
sumStakeTimeSq = totalEligibleStake * t * t;
``` [2](#0-1) 

```solidity
sumStakeTime += contribTime;    // received * block.timestamp
sumStakeTimeSq += contribTimeSq;
``` [3](#0-2) 

So the global snapshot at `flagOutcome` correctly encodes `preAmount × riskWindowStart + postAmount × postTime`, while `_clampUserSums` sets the per-user value to the strictly lower `(preAmount + postAmount) × riskWindowStart`.

**Concrete example** (T = 4000, riskWindowStart = 2000):

| | Pre-risk 100 @ t=1000 | Post-risk 50 @ t=3000 | Total |
|---|---|---|---|
| `userSumStakeTime` (raw) | 100 000 | 150 000 | 250 000 |
| `eligibleStake × riskWindowStart` | — | — | 300 000 |
| Condition `250 000 < 300 000` | **TRUE → clamp fires** | | |
| After clamp: `userSumStakeTime` | — | — | 300 000 ← **wrong** |
| Correct value | 200 000 | 150 000 | 350 000 |

User score with clamped values:
```
userPlus  = 4000² × 150 + 600 000 000 = 3 000 000 000
userMinus = 2 × 4000 × 300 000       = 2 400 000 000
userScore = 600 000 000
```

Global score (correct snapshot):
```
plus        = 4000² × 150 + 850 000 000 = 3 250 000 000
minus       = 2 × 4000 × 350 000        = 2 800 000 000
globalScore = 450 000 000
```

`bonusShare = (600 000 000 / 450 000 000) × snapshotTotalBonus = 1.333 × snapshotTotalBonus`

The payout `userEligible + bonusShare` exceeds the contract's available balance; `safeTransfer` reverts. [4](#0-3) 

### Impact Explanation

When the claim reverts, the user's principal (`eligibleStake[msg.sender]`) remains in the contract. No alternative claim path exists for SURVIVED or EXPIRED outcomes — `sweepUnclaimedBonus` reserves `totalEligibleStake` for remaining stakers but sends sweeps to `recoveryAddress`, not to the staker. The principal is permanently locked. In the single-staker case this is a total loss of the deposited principal. In the multi-staker case the overestimated score redistributes bonus away from other stakers, and the affected user's own claim still reverts.

### Likelihood Explanation

The trigger requires a user to stake in two separate calls: one before `riskWindowStart` is sealed and one after (during `UNDER_ATTACK`, which is explicitly permitted). This is a natural usage pattern — an early staker topping up their position once an attack is publicly visible. The condition fires whenever the pre-risk time-deficit (`preAmount × (riskWindowStart − preTime)`) exceeds the post-risk time-surplus (`postAmount × (postTime − riskWindowStart)`), which is common when the pre-risk deposit is large or was made well before the attack window opened.

### Recommendation

Track the user's pre-risk stake separately (e.g. a `preRiskStake` mapping set at `_markRiskWindowStart` time or on the first post-risk interaction) so that `_clampUserSums` can compute the correct mixed value:

```
userSumStakeTime[u]   = preRiskStake[u] × riskWindowStart
                      + (eligibleStake[u] − preRiskStake[u]) × (actual post-risk weighted avg)
```

Alternatively, store per-deposit entries so the clamp can iterate only over pre-risk tranches. At minimum, the clamp replacement should be guarded to only fire when the user has **no** post-risk deposits (i.e. all stake was deposited before `riskWindowStart`), which can be detected by comparing `eligibleStake[u]` against a stored pre-risk snapshot.

    function _bonusShare(address u, uint256 userEligible) internal view returns (uint256) {
        if (snapshotTotalBonus == 0) return 0;
        // No observable risk → no bonus (see contract natspec).
        if (riskWindowStart == 0) return 0;
        uint256 T = outcomeFlaggedAt;

        // Underflow guards on both subtractions: globally the sum of squares is nonneg, but
        // truncation/rounding pathologies could push individual terms over.
        uint256 userPlus = T * T * userEligible + userSumStakeTimeSq[u];
        uint256 userMinus = 2 * T * userSumStakeTime[u];
        uint256 userScore = userPlus > userMinus ? userPlus - userMinus : 0;

        uint256 plus = T * T * snapshotTotalStaked + snapshotSumStakeTimeSq;
        uint256 minus = 2 * T * snapshotSumStakeTime;
        uint256 globalScore = plus > minus ? plus - minus : 0;

        if (globalScore == 0) {
            // No time elapsed in the risk window for anyone → fallback to amount-weighted.
            if (snapshotTotalStaked == 0) return 0;
            return Math.mulDiv(userEligible, snapshotTotalBonus, snapshotTotalStaked);
        }
        // mulDiv handles the final multiply-then-divide via 512-bit intermediates, so a very
        // large `snapshotTotalBonus` cannot push the numerator over uint256 before division.
        return Math.mulDiv(userScore, snapshotTotalBonus, globalScore);
    }
