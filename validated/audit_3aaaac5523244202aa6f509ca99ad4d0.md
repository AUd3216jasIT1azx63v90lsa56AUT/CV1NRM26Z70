Audit Report

## Title
Fee-on-Transfer Loan Token Inflates `claimableSettlementFee` Without Actual Balance Coverage - (File: `src/Midnight.sol`)

## Summary
In `take`, `claimableSettlementFee[offer.market.loanToken]` is incremented at line 418 before the inbound transfer at line 455. Because `SafeTransferLib.safeTransferFrom` only checks call success and return value — not actual received amount — a fee-on-transfer loan token causes the protocol to credit more than it receives. Repeated takes compound the shortfall, enabling either a DoS on fee collection or, when lender `withdrawable` balances of the same token exist, direct drainage of lender-redeemable assets.

## Finding Description

**Root cause:** `claimableSettlementFee` accounting is updated unconditionally before the transfer that is supposed to back it, and no guard verifies the actual amount received.

**Exact code path:**

- `src/Midnight.sol` line 418: `claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;` — accounting updated before any transfer. [1](#0-0) 

- `src/Midnight.sol` line 455: `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);` — protocol receives the fee portion after accounting is already committed. [2](#0-1) 

- `src/libraries/SafeTransferLib.sol` lines 24–34: `safeTransferFrom` only checks that the call did not revert and that the return value is `true`. No before/after balance comparison is performed. [3](#0-2) 

- `src/Midnight.sol` lines 305–310: `claimSettlementFee` decrements `claimableSettlementFee[token] -= amount` and then calls `safeTransfer(token, receiver, amount)`. If `claimableSettlementFee` exceeds the actual token balance, this either reverts or drains other lenders' `withdrawable` funds. [4](#0-3) 

**Why existing checks fail:**

The Certora `Solvency.spec` formal proof explicitly assumes at lines 31–33 that tokens perform no fee-taking: *"no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits."* The `pendingFeeReceipt` ghost (lines 142–151) tracks the gap between the `claimableSettlementFee` increment and the inbound transfer, but the CVL model assumes the transfer delivers exactly the requested amount — it does not model fee-on-transfer deductions. [5](#0-4) [6](#0-5) 

Market creation via `touchMarket` is permissionless and accepts any `loanToken` with no token-type validation.

## Impact Explanation

`claimableSettlementFee[loanToken]` exceeds the protocol's actual token balance by `(buyerAssets - sellerAssets) * feeRate` per `take` call. Two concrete impacts:

1. **DoS on fee collection:** `claimSettlementFee` reverts when the credited amount exceeds the actual balance, permanently blocking fee recovery for that token.
2. **Lender fund drainage:** When the same fee-on-transfer token is used as a loan token across multiple markets with lender `withdrawable` balances (e.g., via `repay`), the fee claimer's `safeTransfer` draws from those lender-redeemable assets — direct protocol insolvency.

Both impacts are in-scope per `SECURITY.md` (direct theft/freeze of user assets, protocol insolvency).

## Likelihood Explanation

All three preconditions — (1) fee-on-transfer loan token, (2) `offer.buy = true`, (3) `offer.callback != address(0)` — are fully attacker-controlled. Market creation is permissionless. The attack is repeatable on every `take` call with no privileged access required. Legitimate fee-on-transfer tokens (e.g., PAXG) used by honest market creators also trigger this path without any attacker involvement.

## Recommendation

Measure the actual received amount by comparing the contract's token balance before and after the transfer at line 455, and use the measured delta to update `claimableSettlementFee` rather than the nominal `buyerAssets - sellerAssets`. Alternatively, move the `claimableSettlementFee` increment to after the transfer and compute it from the balance delta. A market-level flag or a protocol-wide guard rejecting fee-on-transfer tokens in `touchMarket` would also mitigate the issue.

## Proof of Concept

1. Deploy a fee-on-transfer ERC20 with a 1% transfer fee.
2. Call `touchMarket` with this token as `loanToken` (permissionless).
3. Create a buy offer (`offer.buy = true`, `offer.callback = attackerBuyCallback`).
4. Call `take` with `buyerAssets - sellerAssets = 100`.
5. Observe: `claimableSettlementFee[feeToken]` increments by 100 (line 418), but `safeTransferFrom` at line 455 delivers only 99 tokens to the contract.
6. Repeat N times; shortfall accumulates to N units.
7. Call `claimSettlementFee(feeToken, 100*N, receiver)`: if the contract holds lender `withdrawable` balances of `feeToken`, those are drained; otherwise the call reverts.

A Foundry fork test can confirm this by deploying a mock fee-on-transfer ERC20, asserting `claimableSettlementFee` after each `take`, and asserting the contract's actual token balance is less than `claimableSettlementFee` after multiple takes.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
