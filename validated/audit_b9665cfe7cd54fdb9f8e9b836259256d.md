### Title
Nonce Frontrunning in `EcrecoverAuthorizer.setIsAuthorized()` Enables Perpetual DoS on Authorization Transactions — (File: `src/periphery/EcrecoverAuthorizer.sol`)

---

### Summary

`EcrecoverAuthorizer.setIsAuthorized()` is an unrestricted external function that increments the authorizer's nonce before completing signature verification. An attacker who observes a pending `setIsAuthorized()` transaction in the mempool can copy its calldata verbatim and frontrun it with higher gas, consuming the nonce. The victim's original transaction then reverts with `InvalidNonce`. This can be repeated indefinitely, permanently preventing the victim from successfully submitting their own authorization transactions.

---

### Finding Description

`EcrecoverAuthorizer.setIsAuthorized()` has no access control — any address may call it: [1](#0-0) 

The nonce is incremented on line 26 as a side-effect of the `require` expression:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

This increment is committed to state before the signature is verified (lines 28–36). The signature check only happens after: [2](#0-1) 

Because the full `authorization` struct and `signature` are passed as calldata and are visible in the public mempool, an attacker can:

1. Observe a pending `setIsAuthorized(authorization, signature)` transaction.
2. Copy the exact same `authorization` and `signature` values.
3. Submit an identical call with a higher gas price, frontrunning the victim.
4. The attacker's transaction succeeds: the nonce is incremented and the authorization is set.
5. The victim's original transaction reverts with `InvalidNonce` because `authorization.nonce` no longer matches the incremented `nonce[authorization.authorizer]`.

The victim must now sign a new message with `nonce + 1`. The attacker can frontrun that too, indefinitely.

The `Authorization` struct that is signed and exposed in calldata: [3](#0-2) 

---

### Impact Explanation

- The victim can never successfully submit their own `setIsAuthorized()` transaction as long as the attacker keeps frontrunning.
- Every victim attempt wastes gas and reverts.
- In a scenario where the victim urgently needs to revoke a compromised authorized address, the attacker can block the revocation indefinitely (the attacker's frontrun does execute the revocation, but the victim cannot confirm this on-chain and may be confused into re-authorizing).
- More critically: if the victim alternates between grant and revoke attempts due to confusion, the attacker can selectively frontrun only the revoke transactions, leaving the authorized address active.

---

### Likelihood Explanation

- Requires only mempool access and the ability to submit transactions with higher gas — no privileged keys, no special role.
- Frontrunning is a well-established, low-cost attack on Ethereum mainnet.
- The full signature is exposed in calldata, making replay trivial.
- Any user of `EcrecoverAuthorizer` is a potential victim.

---

### Recommendation

Add a check that `msg.sender` is either the `authorization.authorizer` or an address already authorized by them in `Midnight.isAuthorized`. This mirrors the pattern already used in `EcrecoverRatifier.cancelRoot()`: [4](#0-3) 

Alternatively, move the nonce increment to after all validation checks so that a failed signature check does not leave the nonce in an inconsistent state (though this alone does not prevent frontrunning if the function remains open).

---

### Proof of Concept

**Setup:**
- `authorizer` = Alice (victim)
- `attacker` = Bob

**Steps:**

1. Alice signs an `Authorization` struct: `{authorizer: Alice, authorized: X, isAuthorized: true, nonce: 0, deadline: T}` and submits `setIsAuthorized(authorization, sig)` to the mempool.

2. Bob observes Alice's pending transaction and copies `authorization` and `sig` exactly.

3. Bob submits `setIsAuthorized(authorization, sig)` with higher gas. Bob's transaction is mined first:
   - `nonce[Alice]` is `0`, matches `authorization.nonce = 0` → incremented to `1`.
   - Signature verifies correctly (it's Alice's valid signature).
   - `IMidnight.setIsAuthorized(X, true, Alice)` is called.

4. Alice's original transaction is mined next:
   - `nonce[Alice]` is now `1`, but `authorization.nonce = 0` → `require` fails with `InvalidNonce`.
   - Alice's transaction reverts.

5. Alice signs a new message with `nonce: 1` and resubmits. Bob frontruns again. Repeat indefinitely. [5](#0-4)

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
