Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via zero-rounding `buyerAssets` — (File: src/Midnight.sol)

## Summary
When a buy offer is taken with `maxAssets > 0` and any `tick < MAX_TICK`, integer division in `mulDivDown(units, buyerPrice, WAD)` floors `buyerAssets` to zero for small `units` values (e.g. `units = 1`). Because `consumed[maker][group]` is incremented by `buyerAssets = 0`, the guard `require(newConsumed <= offer.maxAssets)` passes unconditionally on every call. An attacker can take the offer an unbounded number of times, causing the maker to accumulate credit and the taker to accumulate debt with zero tokens transferred per fill, completely bypassing the maker's stated cap.

## Finding Description

**Exact code path — `src/Midnight.sol` lines 358–369:**

```solidity
uint256 offerPrice  = TickLib.tickToPrice(offer.tick);                    // line 358
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice; // 361
uint256 buyerPrice  = sellerPrice + _settlementFee;                        // 362
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)                                    // 363 ← rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group]
        += offer.buy ? buyerAssets : sellerAssets;                         // 368 ← += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());             // 369 ← never fires
}
```

**Root cause — `TickLib.tickToPrice` returns values strictly less than `WAD` for every `tick < MAX_TICK`:**

`tickToPrice` computes `1e36 / (1e18 + wExp(...))` rounded to `PRICE_ROUNDING_STEP = 1e12`. At `MAX_TICK = 5820` the result equals `WAD`; for every lower tick it is strictly smaller. The minimum non-zero price is `1e12` (one rounding step). With `units = 1`:

```
mulDivDown(1, buyerPrice, WAD) = (1 * buyerPrice) / WAD
```

Since `buyerPrice < WAD = 1e18`, integer division yields 0 for any `buyerPrice < 1e18`. This holds for all `tick < MAX_TICK`, which covers virtually every real offer.

**`mulDivDown` implementation — `src/libraries/UtilsLib.sol` line 30:**

```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
```

Plain integer division with no minimum-result guard.

**Position mutation still executes — `src/Midnight.sol` lines 408–414:**

With `units = 1`, `buyerCreditIncrease = zeroFloorSub(1, buyerPos.debt)` and `sellerDebtIncrease = 1 - sellerCreditDecrease`. If the maker has no prior debt and the taker has no prior credit, the maker gains 1 unit of credit and the taker gains 1 unit of debt per call, with zero tokens transferred.

**Token transfers — `src/Midnight.sol` lines 455–456:**

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

Both are zero-value transfers. Standard ERC-20 tokens accept zero-value `transferFrom` calls, so the take completes successfully.

**No minimum-units check exists:** A `grep` over `src/Midnight.sol` for any `require(units > 0)` or equivalent returns no matches. The function accepts `units = 1` without restriction.

**Why existing checks fail:**

- The `require(newConsumed <= offer.maxAssets)` guard is the sole enforcement of the cap and relies entirely on `buyerAssets` being non-zero — which is not guaranteed for `units = 1` and `tick < MAX_TICK`.
- `Consume.spec` line 68 scopes `takeConsumedDelta` to `maxAssets == 0` only; no analogous delta rule exists for assets mode.
- `mulDivDown` and `mulDivUp` are summarized as `NONDET` in `certora/specs/Consume.spec` lines 11–12, so the formal proofs never model the rounding-to-zero case.
- `takeConsumedBoundedByMax` (line 62) only asserts `consumed <= maxAssets` after a single take; it does not assert that consumed strictly increases when `units > 0`.

**Exploit flow:**
1. Maker posts a buy offer: `maxAssets = X`, any `tick < MAX_TICK`, valid ratifier, valid expiry.
2. Attacker calls `take(offer, ..., units=1, ...)` in a loop (or via `multicall`).
3. Each iteration: `buyerAssets = 0`; `consumed` unchanged; cap check passes; maker credit +1; taker debt +1; 0 tokens transferred.
4. After N iterations, maker holds N units of credit while `consumed[maker][group]` remains 0, far exceeding `maxAssets`.

## Impact Explanation
The `maxAssets` cap — the sole mechanism preventing a buy offer from being filled beyond the maker's stated limit — is completely bypassed. The maker accumulates unbounded credit exposure (claims on future repayment at maturity) without any corresponding token inflow per fill. At maturity, the maker holds credit positions backed by taker debt that was created without token deposit, creating an accounting imbalance. If the taker defaults, the maker's excess credit is unbacked. This violates the core protocol invariant that offers cannot be overfilled beyond the maker's stated cap.

## Likelihood Explanation
**Preconditions**: (1) A buy offer with `maxAssets > 0` — a normal maker configuration. (2) Any `tick < MAX_TICK` — satisfied by virtually every real offer since `MAX_TICK` is the par tick. (3) Attacker willing to accumulate debt at zero token cost per unit. No privileged access, no oracle manipulation, and no special token behavior is required. The attack is repeatable in a single transaction via `multicall` or across multiple blocks.

## Recommendation
Add a minimum-result guard after computing `buyerAssets` and `sellerAssets`. Specifically, when `offer.maxAssets > 0` and `units > 0`, require that the computed asset amount is non-zero before incrementing `consumed`:

```solidity
if (offer.maxAssets > 0) {
    uint256 delta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || delta > 0, ZeroAssetDelta());
    newConsumed = consumed[offer.maker][offer.group] += delta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` value such that the resulting asset amount is always non-zero given the offer's tick and price. Additionally, the formal spec in `Consume.spec` should replace the `NONDET` summary for `mulDivDown`/`mulDivUp` with a concrete lower-bound axiom (e.g., `result >= 1` when both inputs are non-zero and `y >= d`) to catch this class of rounding bug.

## Proof of Concept
**Minimal Foundry test:**

```solidity
function test_maxAssets_bypass() public {
    // Setup: create market, maker posts buy offer with maxAssets = 1e18, tick = MAX_TICK - 4
    // tickToPrice(MAX_TICK - 4) < WAD, so mulDivDown(1, price, WAD) = 0

    Offer memory offer = Offer({
        buy: true,
        tick: MAX_TICK - 4,       // price < WAD
        maxAssets: 1e18,          // cap: 1e18 tokens
        maxUnits: 0,
        // ... other fields
    });

    // Attacker calls take 1000 times with units=1
    for (uint i = 0; i < 1000; i++) {
        midnight.take(offer, ratifierData, 1, attacker, attacker, address(0), "");
    }

    // Assert: consumed[maker][group] == 0 (cap never incremented)
    assertEq(midnight.consumed(maker, offer.group), 0);
    // Assert: maker credit == 1000 (far beyond maxAssets cap)
    assertEq(midnight.position(marketId, maker).credit, 1000);
}
```

**Expected result:** All 1000 `take` calls succeed; `consumed` remains 0; maker credit reaches 1000 units; zero tokens transferred throughout.
