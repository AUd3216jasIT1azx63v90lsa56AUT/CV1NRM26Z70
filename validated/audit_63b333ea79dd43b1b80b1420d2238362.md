Audit Report

## Title
Reentrancy in `repay` callback allows authorized operator to drain borrower collateral via transient debt zeroing - (`src/Midnight.sol`)

## Summary
`repay` reduces `position[id][onBehalf].debt` at line 508 before invoking the `IRepayCallback.onRepay` external call at line 516 and before pulling loan tokens at line 520. During the callback window, the position's debt is zero while no loan tokens have been received, allowing a re-entrant call to `withdrawCollateral` to pass the `isHealthy` check and drain all collateral. The attacker funds the repayment using the proceeds from the drained collateral, completing the attack atomically with zero upfront capital.

## Finding Description

**Root cause:** `repay` writes `position[id][onBehalf].debt -= units` before the external callback and before `safeTransferFrom`. The health invariant — "collateral cannot be withdrawn unless debt is backed" — is violated because `isHealthy` evaluates against a transiently zeroed debt that has not yet been backed by received loan tokens.

**Code path — `repay`** (`src/Midnight.sol:502-521`): [1](#0-0) 

- Line 505: auth check passes if `isAuthorized[onBehalf][msg.sender]`
- Line 508: debt zeroed **before** callback or payment
- Line 516: external `onRepay` callback fires with debt already zero
- Line 520: loan tokens pulled **last**

**Code path — `withdrawCollateral`** (`src/Midnight.sol:549-573`): [2](#0-1) 

- Line 556: auth check passes (X is authorized)
- Line 568: `isHealthy` passes because `debt == 0` at this point
- Line 572: collateral transferred to attacker

**No reentrancy guard exists** on either function. The `LIQUIDATION_LOCK_SLOT` transient lock is defined: [3](#0-2) 

but it is only set during liquidation `take` callbacks, not during `repay`.

**Exploit flow:**

1. Victim A authorizes malicious contract X via `EcrecoverAuthorizer.setIsAuthorized` (permissionless relay — anyone can submit a signed authorization): [4](#0-3) 

2. X calls `repay(market, D, A, X, data)` — X is both `msg.sender` and `callback`.
3. Line 508 zeros A's debt. Line 516 calls `X.onRepay(...)`.
4. Inside `onRepay`, X re-enters `withdrawCollateral(market, idx, C_total, A, X)`:
   - Auth check passes (X is authorized for A).
   - Health check passes (debt is 0).
   - All of A's collateral is transferred to X.
5. X swaps collateral for D loan tokens via a DEX, returns `CALLBACK_SUCCESS`.
6. Line 520: `safeTransferFrom(loanToken, X, midnight, D)` succeeds — funded by the swap.

**Net result:** A's debt is 0 and A's collateral is 0. X extracted `value(C) − D` profit with zero upfront capital.

**Why existing checks fail:**
- Authorization check in `withdrawCollateral` passes because X is legitimately authorized.
- `isHealthy` passes because debt was zeroed at line 508 before any loan tokens were received.
- `LIQUIDATION_LOCK_SLOT` is not set during `repay`, only during liquidation.

## Impact Explanation
An authorized operator can drain 100% of a borrower's overcollateralized position in a single atomic transaction. The invariant "collateral cannot be withdrawn outside health/liquidation rules" is broken because the health check is evaluated against a transiently zeroed debt that has not yet been backed by received loan tokens. This constitutes direct theft of user assets — a critical in-scope impact.

## Likelihood Explanation
**Preconditions:**
1. A must have signed an `Authorization` for X (e.g., to allow automated repayment, or via a malicious UI/integrator).
2. A must have a healthy, overcollateralized position.
3. The collateral token must be swappable for loan tokens within one transaction (true for any token with DEX liquidity).

All three are routine conditions. The `EcrecoverAuthorizer.setIsAuthorized` relay is permissionless — any party can submit a signed authorization. The attack is repeatable across all markets and all authorized operators. A malicious contract deployed as a "repayment helper" or "position manager" can exploit this against any user who grants it authorization.

## Recommendation
Apply one or more of the following fixes:

1. **Pull loan tokens before zeroing debt and before the callback** (checks-effects-interactions): move `safeTransferFrom` to before the state mutation and callback.
2. **Add a reentrancy guard** (e.g., a transient storage lock analogous to `LIQUIDATION_LOCK_SLOT`) that is set at the entry of `repay` and checked in `withdrawCollateral`.
3. **Zero debt only after payment is confirmed**: restructure `repay` so that `position[id][onBehalf].debt` is decremented after `safeTransferFrom` completes.

Option 1 (CEI ordering) is the most robust and eliminates the root cause entirely.

## Proof of Concept
Minimal Foundry fork test:

1. Deploy a `MaliciousOperator` contract implementing `IRepayCallback.onRepay`.
2. Inside `onRepay`, call `midnight.withdrawCollateral(market, 0, C_total, victim, address(this))`.
3. Swap received collateral for loan tokens via a DEX fork.
4. Return `CALLBACK_SUCCESS`.
5. In the test: set up victim with collateral C and debt D; authorize `MaliciousOperator`; call `maliciousOperator.attack(market, D, victim)`.
6. Assert: victim collateral == 0, victim debt == 0, attacker loan token balance == `value(C) - D`.

The test should pass against the current code and fail after applying the CEI fix.
