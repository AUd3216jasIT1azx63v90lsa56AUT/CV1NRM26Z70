Audit Report

## Title
Maker callback can call `setConsumed` mid-`take` to inflate consumed and grief taker multicall fills - (File: src/Midnight.sol)

## Summary
During `take`, `consumed[offer.maker][offer.group]` is incremented and bounds-checked at lines 367–373 before any external call. For a buy offer, `offer.callback` is the maker's own address, and `IBuyCallback(buyerCallback).onBuy(...)` is invoked at lines 448–452 with no reentrancy guard on `setConsumed`. A maker-controlled callback can therefore call `setConsumed` to inflate `consumed` to `maxAssets` after the first fill, causing every subsequent fill of the same offer within the same `multicall` to revert with `ConsumedAssets`, atomically reverting the taker's entire bundle.

## Finding Description

**Step 1 — consumed is incremented and capped before any callback:**

`take` writes and bounds-checks `consumed` at lines 367–373, before any external call is made. [1](#0-0) 

**Step 2 — For a buy offer, `buyerCallback` is the maker's own address:**

`offer.callback` is assigned to `buyerCallback` when `offer.buy == true`. [2](#0-1) 

**Step 3 — The only transient lock set during `take` is the liquidation lock for the seller:**

`LIQUIDATION_LOCK_SLOT` is set for the seller at line 444 to prevent liquidation during callbacks. There is no transient lock or mutex preventing `setConsumed` from being called during this external call. [3](#0-2) [4](#0-3) 

**Step 4 — `setConsumed` only requires authorization and a non-decreasing amount:**

`setConsumed` passes as long as `msg.sender` is authorized for `onBehalf` and `amount >= consumed[onBehalf][group]`. There is no check that it is not being called during an active `take`. [5](#0-4) 

**Exploit flow:**
1. Maker deploys malicious contract `M` and calls `setIsAuthorized(M, true, maker)` — a normal, permissionless operation.
2. Maker publishes a buy offer with `offer.callback = M` and `offer.maxAssets = 1000`.
3. Taker submits `multicall([take(offer, 400 units), take(offer, 400 units)])`.
4. First `take`: `consumed[maker][group] += 400` → 400 ≤ 1000, passes.
5. `M.onBuy(...)` is called. Inside, `M` calls `midnight.setConsumed(group, 1000, maker)`. Authorization check passes (`isAuthorized[maker][M] == true`). `AlreadyConsumed` check passes (`1000 >= 400`). `consumed[maker][group]` is now 1000.
6. First `take` completes normally.
7. Second `take`: `consumed[maker][group] += 400` → 1400 > 1000 → reverts `ConsumedAssets`.
8. Entire `multicall` reverts; taker's atomic bundle fails and all gas is wasted.

**Why existing checks fail:**
- `AlreadyConsumed` (line 725) only prevents *decreasing* consumed; it explicitly allows any upward inflation, so the re-entrant call passes unconditionally.
- The `LIQUIDATION_LOCK_SLOT` transient guard (line 444) only protects against liquidating the seller during callbacks — it has no effect on `setConsumed`.
- The Certora `Consume.spec` rule `takeConsumedBoundedByMax` (lines 59–64) does not catch this because the callback sets consumed to exactly `maxAssets`, so `consumed <= maxAssets` still holds after the single `take` call. The rule `takeConsumedDelta` (lines 67–75) does not apply because it is scoped to units mode (`offer.maxAssets == 0`), while the exploit uses assets mode. [6](#0-5) [7](#0-6) [8](#0-7) 

## Impact Explanation
A maker can deterministically grief any taker who attempts multiple partial fills of the same offer atomically via `multicall`. The entire atomic bundle reverts, wasting the taker's gas and preventing the intended fill. The maker's offer is also permanently blocked from further fills (self-harm griefing). No direct fund loss occurs, but the attack constitutes a concrete, repeatable denial-of-service against atomic multi-fill workflows, which are a core use case of the protocol's `multicall` and group/offer-cap design.

## Likelihood Explanation
All preconditions are normal, permissionless protocol operations: deploying a callback contract and calling `setIsAuthorized` are both standard flows already used in the protocol. No special privileges, oracle manipulation, or token-owner action is required. The attack is deterministic — it always succeeds given the preconditions — and the maker can deploy a fresh callback and offer for each target taker. Any taker using `multicall` for multi-fill atomic transactions is a potential victim.

## Recommendation
Add a transient reentrancy guard scoped to `setConsumed` during an active `take`. Concretely: set a transient storage slot (e.g. `CONSUME_LOCK_SLOT` keyed by `(maker, group)`) immediately after the consumed increment and bounds-check at lines 367–373, and clear it just before `take` returns. In `setConsumed`, revert if the corresponding slot is set. This prevents the callback from inflating consumed mid-take while preserving all legitimate uses of `setConsumed` outside of an active take.

## Proof of Concept
1. Deploy `MaliciousCallback M` that implements `onBuy`: calls `midnight.setConsumed(offer.group, offer.maxAssets, maker)` then returns `CALLBACK_SUCCESS`.
2. Maker calls `setIsAuthorized(address(M), true, maker)`.
3. Maker creates a buy offer with `offer.callback = address(M)`, `offer.maxAssets = 1000`, `offer.group = someGroup`.
4. Taker calls `midnight.multicall([abi.encodeCall(take, (offer, ..., 400, ...)), abi.encodeCall(take, (offer, ..., 400, ...))])`.
5. Observe: the entire `multicall` reverts with `ConsumedAssets` from the second `take`, despite `400 + 400 = 800 <= 1000`.
6. Confirm: a single `take` of 400 units succeeds, proving the offer is valid and the revert is caused solely by the mid-take inflation of `consumed`.
