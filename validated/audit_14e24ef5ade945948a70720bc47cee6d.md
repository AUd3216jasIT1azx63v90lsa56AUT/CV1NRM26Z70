### Title
Missing Zero-Address Validation for `oracle` in `CollateralParams` Enables Permanent DoS of Borrower Positions — (File: src/Midnight.sol)

---

### Summary

The `touchMarket` function validates several `CollateralParams` fields (`token`, `lltv`, `maxLif`) but never checks that `collateralParams[i].oracle != address(0)`. Because `touchMarket` is permissionless, any attacker can create a market with a zero-address oracle. Once a borrower activates that collateral, every oracle-dependent function (`isHealthy`, `liquidate`, `take` as seller with debt) reverts, locking the borrower's collateral until full debt repayment — which may be impossible.

---

### Finding Description

**Root cause — `touchMarket` validation loop, `src/Midnight.sol` lines 762–773:**

```solidity
for (uint256 i = 0; i < market.collateralParams.length; i++) {
    address collateralToken = market.collateralParams[i].token;
    require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
    uint256 lltv = market.collateralParams[i].lltv;
    require(isLltvAllowed(lltv), LltvNotAllowed());
    require(
        market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_LOW)
            || market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_HIGH),
        InvalidMaxLif()
    );
    previousCollateralToken = collateralToken;
}
```

`token` is implicitly non-zero (the `> previousCollateralToken` check where `previousCollateralToken` starts at `address(0)` enforces `token > address(0)`). `lltv` and `maxLif` are validated. **`oracle` is never validated.** A market with `collateralParams[i].oracle = address(0)` is accepted and stored immutably.

**Propagation — `supplyCollateral`, `src/Midnight.sol` lines 524–546:**

`supplyCollateral` calls `touchMarket` (which creates the market if new) and then activates the collateral by setting its bit in `collateralBitmap` when `oldCollateral == 0 && assets > 0`. No oracle check is performed here either. The activation is irreversible unless the borrower withdraws the full collateral balance — but withdrawal requires `isHealthy`, which calls the oracle.

**Revert cascade — `isHealthy`, `src/Midnight.sol` line 953:**

```solidity
uint256 price = IOracle(collateralParam.oracle).price();
```

In Solidity 0.8.34, a high-level interface call to `address(0)` (which has no code) succeeds at the EVM level but returns empty data. The ABI decoder then attempts to decode a `uint256` from 0 bytes and **reverts**. Any function that reaches this line with a zero-oracle collateral activated and `debt > 0` will revert:

- `withdrawCollateral` → calls `isHealthy` → reverts
- `take` (seller with debt) → calls `isHealthy` at line 476 → reverts
- `liquidate` → calls `IOracle(_collateralParam.oracle).price()` at line 610 directly → reverts

`repay` is the only debt-reducing path that does not call the oracle, so the borrower can escape only by repaying the full debt.

---

### Impact Explanation

A borrower who has debt and has activated a zero-oracle collateral (even a dust amount supplied after borrowing) is DoS'd from:

- Withdrawing any collateral (including collaterals with valid oracles, because `isHealthy` iterates all activated collaterals)
- Being liquidated (liquidators cannot recover the position)
- Taking new sell-side positions

All collateral is locked until the borrower repays 100% of their debt. If the borrower lacks the funds to repay — a realistic scenario for leveraged positions — the lock is permanent and the collateral is unrecoverable. The market is immutable, so the broken oracle cannot be patched after creation.

---

### Likelihood Explanation

`touchMarket` is fully permissionless — no privileged key is required. An attacker needs only to:

1. Deploy or identify a market with `collateralParams[i].oracle = address(0)` for one collateral slot.
2. Advertise the market (e.g., via a front-end or off-chain tooling) as a legitimate multi-collateral market.
3. Wait for a victim to supply the zero-oracle collateral (e.g., believing it is a secondary collateral type).

Because markets are identified by a `bytes32` hash of their full parameter set and are immutable once created, the broken market persists forever. The attack requires no on-chain privilege and no cryptographic break.

---

### Recommendation

Add an explicit non-zero check for `oracle` inside the `touchMarket` validation loop:

```solidity
require(market.collateralParams[i].oracle != address(0), InvalidOracle());
```

This mirrors the implicit non-zero enforcement already applied to `token` via the sorting check, and is consistent with the pattern used in analogous lending protocols (e.g., Gearbox's recommended fix for `_interestRateModel`).

---

### Proof of Concept

**Setup:**
- Deploy `Midnight`.
- Construct a `Market` with one valid collateral (index 0, valid oracle) and one zero-oracle collateral (index 1, `oracle = address(0)`).

**Steps:**
1. Attacker calls `touchMarket(market)` — succeeds, market is created and stored immutably.
2. Victim calls `supplyCollateral(market, 0, 1e18, victim)` — supplies valid collateral, activates index 0.
3. Victim calls `take(offer, ...)` as seller — borrows against collateral 0. `isHealthy` is called; only index 0 is activated, oracle is valid → succeeds.
4. Victim calls `supplyCollateral(market, 1, 1e6, victim)` — supplies dust of zero-oracle collateral, activates index 1 in `collateralBitmap`.
5. Victim calls `withdrawCollateral(market, 0, 1e18, victim, victim)` → internally calls `isHealthy` → iterates `collateralBitmap` → hits index 1 → calls `IOracle(address(0)).price()` → **reverts**.
6. Liquidator calls `liquidate(market, 0, ...)` → hits `IOracle(address(0)).price()` at line 610 → **reverts**.
7. Victim's collateral at index 0 is locked. Only path out is `repay` to zero debt, then `withdrawCollateral`.
