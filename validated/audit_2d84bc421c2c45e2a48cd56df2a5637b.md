### Title
`EcrecoverAuthorizer` Cannot Call `Midnight.setIsAuthorized()` Without Prior On-Chain Authorization — (File: src/periphery/EcrecoverAuthorizer.sol)

---

### Summary

`EcrecoverAuthorizer` is a periphery contract designed to let users set Midnight authorizations via off-chain EIP-712 signatures, avoiding the need to be online. However, `Midnight.setIsAuthorized()` enforces a strict `msg.sender` check that requires the user to have already authorized `EcrecoverAuthorizer` on-chain before the periphery contract can act on their behalf. This creates a circular dependency that renders `EcrecoverAuthorizer` non-functional for any user who has not already performed an on-chain authorization — which is precisely the population the contract is designed to serve.

---

### Finding Description

`EcrecoverAuthorizer.setIsAuthorized()` verifies an off-chain signature and then calls into the core contract:

```solidity
IMidnight(MIDNIGHT)
    .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
``` [1](#0-0) 

Inside `Midnight.setIsAuthorized()`, the access check is:

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
``` [2](#0-1) 

At the call site:
- `msg.sender` = `EcrecoverAuthorizer`
- `onBehalf` = `authorization.authorizer` (the user)

The condition `onBehalf == msg.sender` is false (the user is not `EcrecoverAuthorizer`). The condition `isAuthorized[onBehalf][msg.sender]` requires `isAuthorized[user][EcrecoverAuthorizer] == true`, which is only possible if the user has already called `Midnight.setIsAuthorized(EcrecoverAuthorizer, true, user)` directly on-chain.

This is the same vulnerability class as the external report: a periphery contract calls a core function whose `msg.sender` restriction is too strict, blocking the periphery contract unless a pre-authorization step is completed — a step that defeats the entire purpose of the periphery contract.

---

### Impact Explanation

`EcrecoverAuthorizer` is completely non-functional for any user who has not already authorized it on Midnight. Every call to `EcrecoverAuthorizer.setIsAuthorized()` for such a user reverts with `Unauthorized()`. The contract's stated purpose — enabling authorization via off-chain signatures without requiring the user to be online — is entirely negated. Users who rely on this contract (e.g., smart contract wallets, delegated signers) cannot use it to bootstrap their Midnight authorization state.

---

### Likelihood Explanation

The failure is deterministic and affects every new user of `EcrecoverAuthorizer`. No special attacker is required; any user who attempts to use the contract without having first called `Midnight.setIsAuthorized(EcrecoverAuthorizer, true, self)` on-chain will have their transaction revert. Since the contract's value proposition is precisely to avoid that on-chain step, the affected population is effectively all intended users.

---

### Recommendation

`Midnight.setIsAuthorized()` should accept calls from `EcrecoverAuthorizer` without requiring prior on-chain authorization from the user, provided the periphery contract has already verified the user's off-chain signature. Two options:

1. **Preferred:** Add `EcrecoverAuthorizer` as a trusted periphery in Midnight (similar to how Sense Finance's fix allowed Periphery to bypass the restriction), so the check becomes:
   ```solidity
   require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender] || msg.sender == trustedAuthorizer, Unauthorized());
   ```
2. **Alternative:** Have `EcrecoverAuthorizer` use a separate, dedicated entry-point in Midnight that accepts a verified signature directly, bypassing the `isAuthorized` mapping check.

---

### Proof of Concept

1. Alice (a new user, `isAuthorized[alice][EcrecoverAuthorizer] == false`) signs an `Authorization` struct off-chain granting Bob access to her Midnight position.
2. Bob calls `EcrecoverAuthorizer.setIsAuthorized(authorization, signature)`.
3. `EcrecoverAuthorizer` verifies Alice's signature successfully.
4. `EcrecoverAuthorizer` calls `Midnight.setIsAuthorized(bob, true, alice)` with `msg.sender = EcrecoverAuthorizer`.
5. Midnight evaluates: `alice == EcrecoverAuthorizer` → false; `isAuthorized[alice][EcrecoverAuthorizer]` → false.
6. Transaction reverts with `Unauthorized()`.
7. Alice must call `Midnight.setIsAuthorized(EcrecoverAuthorizer, true, alice)` directly on-chain first — negating the off-chain signature mechanism entirely. [3](#0-2) [4](#0-3)

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
