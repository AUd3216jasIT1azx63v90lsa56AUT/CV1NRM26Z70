### Title
Lender Can Front-Run Bad Debt Realization to Escape Proportional Slash — (`File: src/Midnight.sol`)

---

### Summary

The protocol socializes bad debt among lenders by lazily updating each lender's credit via `_updatePosition`, which applies the accumulated `lossFactor` delta at the lender's next interaction. Because `withdraw()` calls `_updatePosition` with the **current** (pre-bad-debt) `lossFactor` before the slash is committed, a lender who monitors the mempool and front-runs a pending `liquidate()` call that realizes bad debt can exit at full face value, shifting their share of the loss entirely onto remaining lenders.

---

### Finding Description

**Root cause — lazy slashing via `lossFactor`:**

`_updatePosition` computes the slashed credit as:

```
postSlashCredit = credit * (MAX - currentLossFactor) / (MAX - lastLossFactor)
``` [1](#0-0) 

It then writes `lastLossFactor = currentLossFactor`, locking in the snapshot: [2](#0-1) 

**Bad debt realization in `liquidate()`** increases `lossFactor` atomically: [3](#0-2) 

**`withdraw()` calls `_updatePosition` first**, then decreases `credit` and `withdrawable`: [4](#0-3) 

**Attack path:**

1. Market has `totalUnits = T`, `lossFactor = L`, `withdrawable = W > 0` (from prior repayments).
2. A borrower's position has `badDebt = B` units; a liquidation transaction is visible in the mempool.
3. Attacker (lender with `credit = C`) front-runs the liquidation with `withdraw(market, min(C, W), ...)`.
4. Inside `withdraw`, `_updatePosition` runs with the current (pre-bad-debt) `lossFactor = L` → no slash, `newCredit = C`. The lender's `lastLossFactor` is set to `L`.
5. The lender's `credit` is reduced to `C - W` (or 0), `totalUnits` drops to `T - W`, and they receive `W` tokens at full face value.
6. The liquidation executes: `lossFactor` is now computed over the reduced `totalUnits = T - W`, so the proportional loss for remaining lenders is `B / (T - W)` instead of `B / T`.

The protocol's stated invariant is violated: [5](#0-4) 

The protocol itself acknowledges the race condition for withdrawals: [6](#0-5) 

---

### Impact Explanation

A lender escapes their proportional share of bad debt (`C/T * B` units of loss). The remaining lenders collectively absorb the full `B` units of bad debt over a smaller `totalUnits` base, meaning each remaining lender's credit is slashed more than it should be. In markets with large bad debt events and meaningful `withdrawable` balances, this can represent a material, direct theft of value from honest lenders.

---

### Likelihood Explanation

**Preconditions (all realistic):**
- `withdrawable > 0`: satisfied whenever any borrower has repaid or any prior liquidation has occurred in the market — a normal operating condition.
- A pending `liquidate()` call with `badDebt > 0` is visible in the mempool: standard for any public EVM chain.
- The attacker is a lender with `credit > 0`: no privileged access required.

MEV bots routinely monitor mempools for liquidation transactions. The front-run is a single `withdraw()` call with no special setup. Likelihood is **high** on any chain with a public mempool.

---

### Recommendation

Apply the bad debt slash **before** allowing withdrawals in the same block, or snapshot the `lossFactor` at the start of each block and apply it atomically. Concretely, one approach is to require that `_updatePosition` is called with the **post-liquidation** `lossFactor` before any withdrawal can reduce `totalUnits`. A simpler mitigation is to move the `lossFactor` update into a separate, pre-committed step (analogous to UMA's `snapshotCurrentRound`) so that the slash is locked before any lender can exit. Alternatively, track a "pending bad debt" accumulator that is applied to all positions before any withdrawal is processed in the same block.

---

### Proof of Concept

**Setup:**
- Market with `loanToken = USDC`, `totalUnits = 1000`, `lossFactor = 0`, `withdrawable = 200` (from prior repayments).
- Alice is a lender: `credit = 100` (10% of total).
- Bob is a borrower with `debt = 300`, collateral worth only 100 units → `badDebt = 200`.

**Without front-run (honest path):**
1. Liquidator calls `liquidate(Bob, ...)`.
2. `lossFactor` increases: `MAX - (MAX - 0) * (1000 - 200) / 1000 = MAX * 0.2`.
3. Alice calls `withdraw(market, 100, ...)`.
4. `_updatePosition`: `postSlashCredit = 100 * (MAX - MAX*0.2) / MAX = 80`.
5. Alice can withdraw at most 80 units (her slashed credit).

**With front-run (attack path):**
1. Alice sees the liquidation in the mempool and front-runs with `withdraw(market, 100, alice, alice)`.
2. `_updatePosition`: `lossFactor = 0 = lastLossFactor` → no slash, `newCredit = 100`.
3. `credit -= 100 → 0`, `withdrawable -= 100 → 100`, `totalUnits -= 100 → 900`.
4. Alice receives **100 USDC** (full face value, no slash).
5. Liquidation executes: `lossFactor = MAX - (MAX - 0) * (900 - 200) / 900 = MAX * 200/900 ≈ 22.2%`.
6. Remaining lenders (900 units) bear the full 200-unit loss instead of 180 units.

**Alice's gain:** 20 USDC (her fair share of bad debt, shifted to others).
**Other lenders' extra loss:** 20 USDC distributed across 900 units instead of 900 units of a 1000-unit pool. [7](#0-6) [8](#0-7) [9](#0-8)

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
