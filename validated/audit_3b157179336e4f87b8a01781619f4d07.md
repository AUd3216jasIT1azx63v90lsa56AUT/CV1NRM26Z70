### Title
Unbounded `rcfThreshold` Permanently Disables Recovery Close Factor Protection for Borrowers — (File: `src/Midnight.sol`)

---

### Summary

The `Market` struct contains a `rcfThreshold` field that controls the dust exception for the Recovery Close Factor (RCF). The RCF is the protocol's core mechanism that caps how much of a borrower's position a liquidator may seize in a single call to only what is needed to restore health. Because `touchMarket()` applies no upper-bound validation on `rcfThreshold`, any market creator can set it to `type(uint256).max`, permanently and silently disabling the RCF for every borrower in that market. Liquidators can then fully liquidate a position that is only marginally unhealthy, extracting the LIF premium on the entire collateral balance rather than just the unhealthy portion.

---

### Finding Description

**Root cause — no validation of `rcfThreshold` at market creation**

`touchMarket()` validates `lltv` against an explicit allowlist and `maxLif` against a formula-derived pair of values, but applies zero validation to `rcfThreshold`: [1](#0-0) 

`rcfThreshold` is accepted as-is from the caller-supplied `Market` struct and stored immutably as part of the market identity (via `IdLib.storeInCode`).

**How the RCF check uses `rcfThreshold`**

In `liquidate()`, the normal-mode RCF guard is: [2](#0-1) 

The `require` passes if **either**:
1. `repaidUnits <= maxRepaid` — the liquidation is within the RCF cap, **or**
2. `collateral_value / lif − maxRepaid < market.rcfThreshold` — the residual collateral is below the dust threshold.

When `rcfThreshold = type(uint256).max`, condition 2 evaluates to `x < type(uint256).max`, which is true for every practically reachable value of `x`. The RCF cap (`maxRepaid`) is therefore never enforced; a liquidator may pass any `repaidUnits` up to the borrower's full debt.

---

### Impact Explanation

A borrower who is 1 % below the health threshold can have 100 % of their debt repaid and 100 % of their collateral seized in a single transaction. The liquidator collects the LIF premium (`maxLif`) on the entire collateral balance instead of only the unhealthy slice. For a position with 100 ETH collateral and 90 ETH debt at LLTV = 0.86 (maxDebt = 86 ETH):

- **With RCF**: liquidator repays ≈ 5 ETH, seizes ≈ 5 × LIF ETH of collateral.
- **Without RCF** (`rcfThreshold = max`): liquidator repays 90 ETH, seizes ≈ 90 × LIF ETH of collateral — the borrower loses the LIF premium on ≈ 85 ETH of collateral unnecessarily.

This is a direct, quantifiable financial loss to borrowers proportional to position size and LIF.

---

### Likelihood Explanation

Market creation is permissionless — no privileged key is required. [4](#0-3) 

The `live_context.json` explicitly lists "market creator" as an attacker model. [5](#0-4) 

A realistic attack path:
1. Attacker deploys a market with attractive parameters (competitive LLTV, low fees, reputable loan/collateral tokens) but sets `rcfThreshold = type(uint256).max`.
2. Borrowers enter the market — `rcfThreshold` is buried inside the `Market` struct and not surfaced by typical frontends.
3. When any borrower becomes marginally unhealthy (e.g., due to a small oracle price move), the attacker (or any liquidator) fully liquidates the position.

The attack requires no flash loan, no oracle manipulation, and no privileged role. The only prerequisite is that borrowers enter the market, which is realistic given the permissionless, offer-book design.

---

### Recommendation

Add an upper-bound check on `rcfThreshold` inside `touchMarket()`, analogous to the existing `lltv` and `maxLif` validations:

```solidity
// In touchMarket(), alongside the existing collateral param checks:
require(market.rcfThreshold <= MAX_RCF_THRESHOLD, RcfThresholdTooHigh());
```

`MAX_RCF_THRESHOLD` should be set to a value that covers only genuine dust positions (e.g., a few hundred dollars of collateral value at oracle scale), not an unbounded sentinel. Alternatively, require `rcfThreshold > 0` and `rcfThreshold < type(uint256).max` as a minimum sanity guard, mirroring how the external report's resolution enforced a sane `minBlockHeight` default and warned on unsafe overrides.

    false,       // normal mode
    attacker,
    address(0),
    ""

// Attacker seizes 90 * lif / price worth of collateral
// Victim loses LIF premium on ~85 units of collateral unnecessarily
