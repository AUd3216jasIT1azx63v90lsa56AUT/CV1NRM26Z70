Audit Report

## Title
Buy-offer `mulDivDown` rounding to zero bypasses `maxAssets` cap and mints unbacked credit — (`src/Midnight.sol`)

## Summary
When `offer.buy = true` and `buyerPrice < WAD`, calling `take` with `units = 1` causes `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. The `consumed` counter is never incremented, so the `maxAssets` cap is never exhausted regardless of how many times the offer is taken. Each call still increments the maker's credit and the taker's debt by 1 unit while transferring zero loan tokens, minting unbacked credit and debt without limit.

## Finding Description

**Root cause — `src/Midnight.sol` line 363:** [1](#0-0) 

With `units = 1` and any `buyerPrice < WAD` (i.e., any tick below `MAX_TICK`), `1 * buyerPrice < WAD`, so `mulDivDown` returns 0. `WAD = 1e18` is confirmed in `src/libraries/ConstantsLib.sol`. [2](#0-1) 

**Cap bypass — lines 367–369:** [3](#0-2) 

When `offer.buy = true` and `buyerAssets = 0`, `newConsumed` is incremented by zero on every call. The `require(newConsumed <= offer.maxAssets)` check passes unconditionally, even after `maxAssets` would otherwise be exhausted.

**Unbacked credit/debt — lines 382, 410, 414:** [4](#0-3) [5](#0-4) 

`buyerCreditIncrease` and `sellerDebtIncrease` are computed from `units`, not from `buyerAssets`, so they increase by 1 per call regardless of the zero-asset transfer.

**Zero transfer — lines 455–456:** [6](#0-5) 

No loan tokens move. `totalUnits` grows without a matching increase in the protocol's loan-token balance. [7](#0-6) 

**EcrecoverRatifier bypass:** The ratifier only verifies the Merkle-proof signature over the offer struct. It does not inspect `units`, `buyerAssets`, or the consumed counter, so it passes for any validly-signed offer. [8](#0-7) 

**Existing checks that fail:**
- `require(newConsumed <= offer.maxAssets)` — passes because `newConsumed` is unchanged.
- `require(offer.maker != taker)` — trivially satisfied with two attacker-controlled addresses.
- `reduceOnly` — not set in the attack offer.
- Health check on seller — taker can pre-supply collateral.

**Confirmed by test:** `testBugBuyMaxAssetsBypass` exists in `test/TakeTest.sol` and explicitly documents and reproduces this state. [9](#0-8) 

## Impact Explanation
The maker (buyer/lender) accumulates unbounded credit without ever depositing loan tokens. The taker accumulates debt without receiving loan tokens. When the taker eventually repays, the maker withdraws tokens that were never deposited, directly draining other lenders. This breaks the core invariant that every credit increase must correspond to a valid asset transfer, constituting a direct loss of funds for honest lenders — critical severity.

## Likelihood Explanation
Preconditions are trivially met: any buy offer with `buyerPrice < WAD` (every tick except the maximum) and `maxAssets > 0` is vulnerable. The maker and taker can be two addresses controlled by the same party. The attack is repeatable in a single transaction via `multicall`. No oracle manipulation, admin access, or special token behavior is required.

## Recommendation
Add a guard requiring that `buyerAssets > 0` whenever `units > 0` in a buy offer, or switch the buy-offer asset calculation to `mulDivUp` (consistent with how sell offers already use `mulDivUp` at line 363). Alternatively, enforce a minimum `units` value such that `units * buyerPrice >= WAD` before proceeding. The consumed counter should also be incremented by at least 1 per `take` call when `maxAssets > 0` to prevent cap bypass via dust rounding. [10](#0-9) 

## Proof of Concept
1. Deploy with a buy offer: `offer.buy = true`, `tick` set to any value below `MAX_TICK` (so `buyerPrice < WAD`), `maxAssets = N` for any `N > 0`.
2. Call `take(offer, units=1, taker=attackerB, ...)` from `attackerA` (two distinct addresses).
3. Observe: `buyerAssets = 0`, `consumed` unchanged, `attackerA.credit += 1`, `attackerB.debt += 1`, zero tokens transferred.
4. Repeat step 2 arbitrarily many times — the `maxAssets` cap is never hit.
5. `attackerB` repays debt; `attackerA` withdraws tokens never deposited, draining the pool.
6. Run `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` for a self-contained reproduction.
