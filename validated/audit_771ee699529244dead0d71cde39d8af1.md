Audit Report

## Title
Oracle Reentrancy in `liquidate()` Bitmap Loop Enables Unauthorized Collateral Withdrawal with Corrupted Health Accounting - (File: src/Midnight.sol)

## Summary

The `liquidate()` function iterates over collaterals and calls `IOracle.price()` as an unconstrained external call at line 610, before the `NotLiquidatable` check and without any liquidation lock being set. An attacker who controls the oracle for a low-index collateral `i` can reenter `withdrawCollateral()` for a higher-index collateral `j` (already valued in `maxDebt`) during that oracle call, extracting `j`'s tokens while the outer liquidation proceeds on stale `maxDebt` accounting. The residual debt is socialized as bad debt among lenders.

## Finding Description

**Root cause:** `liquidate()` takes a local snapshot of `_collateralBitmap` at line 606 and iterates MSB-first. For each index `i`, it calls `IOracle(_collateralParam.oracle).price()` at line 610 — an unconstrained external call — and only *after* that call reads `_position.collateral[i]` from storage at line 612. The liquidation lock (`LIQUIDATION_LOCK_SLOT`) is never written inside `liquidate()`; it is only written in `take()` at line 444 to protect the seller during take callbacks. The `NotLiquidatable` check at line 620–624 only *reads* the lock after the loop completes. `withdrawCollateral()` at lines 549–573 has no reentrancy guard and no liquidation-lock check — its only safety net is `require(isHealthy(...))` at line 568, which itself calls `IOracle.price()` on every activated collateral.

**Exploit flow:**

Preconditions:
- Attacker is market creator (permissionless) and borrower.
- Market has collateral `j` (high index, honest oracle) and collateral `i` (low index, `i < j`, attacker-controlled malicious oracle).
- Borrower holds `C_j` valuable tokens at index `j` and `C_i` tokens at index `i`.
- Borrower has debt `D` where `D > C_j · price_j · lltv_j / (ORACLE_PRICE_SCALE · WAD)` (position is unhealthy when oracle-i returns 0).
- Borrower has called `setIsAuthorized(maliciousOracle, true, borrower)`.

Steps:

1. Attacker calls `liquidate(market, collateralIndex=i, seizedAssets=C_i, repaidUnits=0, borrower, false, ...)`.
2. `_collateralBitmap` is snapshotted locally at line 606 (contains bits for both `j` and `i`).
3. Loop iteration 1 (MSB = `j`): honest oracle returns `price_j`; `_position.collateral[j]` read as `C_j`; `maxDebt += C_j · price_j · lltv_j`; `badDebt` reduced accordingly.
4. Loop iteration 2 (MSB = `i`): `maliciousOracle.price()` is called at line 610.
5. Inside `maliciousOracle.price()`, the oracle reenters `withdrawCollateral(market, j, C_j, borrower, attackerWallet)`:
   - Authorization passes: `isAuthorized[borrower][maliciousOracle] = true`.
   - `_position.collateral[j]` set to 0 in storage (line 562); bit `j` cleared from `_position.collateralBitmap` in storage (line 565).
   - `isHealthy()` called (line 568): reads updated storage bitmap (only bit `i`), calls malicious oracle recursively, which returns `price_i_high` large enough so `C_i · price_i_high · lltv_i ≥ D` → returns `true`.
   - `C_j` tokens transferred to `attackerWallet` (line 572). ✓
6. Malicious oracle returns `price_i_low = 0` to `liquidate()`.
7. `liquidatedCollatPrice = 0`; `_position.collateral[i]` read as `C_i`; `maxDebt += 0`.
8. Loop ends: `maxDebt = C_j · price_j · lltv_j` (computed using the already-withdrawn `C_j`).
9. `NotLiquidatable` check (line 620–624): `liquidationLocked = false` (never set); `originalDebt > maxDebt` → passes by precondition. ✓
10. `badDebt = D − C_j · price_j / (ORACLE_PRICE_SCALE · maxLif_j) > 0` is realized; lenders are slashed.
11. `repaidUnits = C_i · 0 / ORACLE_PRICE_SCALE · WAD / lif = 0`; attacker seizes worthless `C_i` tokens and repays 0 debt.

**Why existing checks fail:**

- The liquidation lock is never set in `liquidate()`, so `withdrawCollateral()` cannot detect it is being called mid-liquidation.
- The `isHealthy()` check inside `withdrawCollateral()` is defeated because the malicious oracle returns an arbitrarily high price on the recursive call, making the position appear healthy without `j`.
- The `NotLiquidatable` check uses the stale `maxDebt` that still includes `j`'s contribution, so it passes even though `j` has been withdrawn.
- The TOKEN SAFETY REQUIREMENTS (lines 133–140) explicitly prohibit token reentrancy but impose no equivalent constraint on oracles, confirming this path was not guarded.

## Impact Explanation

An attacker who is both market creator and borrower can withdraw valuable collateral `j` during a liquidation — bypassing the health check — while the liquidation proceeds on corrupted `maxDebt` accounting. The attacker recovers `C_j` tokens without repaying any debt, and the residual debt is socialized as bad debt among lenders via the loss-factor mechanism (lines 631–640). This constitutes direct theft of collateral assets and unauthorized bad-debt imposition on lenders.

## Likelihood Explanation

All preconditions are self-inflicted by the attacker with no external cooperation: market creation is permissionless, borrowing is permissionless, deploying a malicious oracle contract requires no privilege, and self-authorization via `setIsAuthorized` is a standard user action. The only constraint is that the position must be genuinely unhealthy (debt exceeds `j`'s collateral value alone), which the attacker engineers by borrowing the maximum against `j` and supplying worthless `i`. The attack is repeatable across any market where the attacker controls an oracle for any collateral index lower than another activated collateral.

## Recommendation

Set the liquidation lock for the borrower at the start of `liquidate()` (before the oracle loop) and clear it at the end, mirroring the pattern used in `take()` at line 444. This prevents any reentry into `withdrawCollateral()` (or any other entry point that checks `liquidationLocked`) during the liquidation's oracle calls. Alternatively, add a CEI-compliant snapshot of all `_position.collateral[i]` values before any external oracle call, so the loop operates entirely on memory rather than re-reading storage after each external call.

## Proof of Concept

Minimal Foundry test outline:

1. Deploy `MaliciousOracle` with a flag: first call (from `liquidate`) triggers reentry into `withdrawCollateral(market, j, C_j, borrower, attacker)` and then returns `0`; second call (from `isHealthy` inside `withdrawCollateral`) returns a price high enough to satisfy the health check.
2. Create a market with two collaterals: index 0 = `TokenI` with `MaliciousOracle`, index 1 = `TokenJ` with `HonestOracle`.
3. Borrower supplies `C_j` of `TokenJ` and `C_i` of `TokenI`; borrows `D` units such that `D > C_j · price_j · lltv_j`.
4. Borrower calls `setIsAuthorized(address(maliciousOracle), true, borrower)`.
5. Attacker calls `liquidate(market, 0, C_i, 0, borrower, false, attacker, address(0), "")`.
