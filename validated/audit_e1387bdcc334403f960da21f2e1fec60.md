Audit Report

## Title
Buy offer at `tick=0` with zero settlement fee allows unlimited offer reuse via zero-advancing `consumed` counter - (File: src/Midnight.sol)

## Summary

`TickLib.tickToPrice(0)` returns exactly `0`, confirmed by the Certora rule `tickToPriceIsZeroAtZero` and the Foundry test `testTickToPriceMinMax`. For a buy offer placed at `tick = 0` in a market with zero settlement fee (the default for any freshly created market), `buyerAssets` computes to `mulDivDown(units, 0, WAD) = 0` for any `units`. Because the `consumed` accounting branch for `maxAssets`-capped buy offers increments by `buyerAssets`, the counter never advances, and the `require(newConsumed <= offer.maxAssets)` guard passes unconditionally on every call, allowing the offer to be taken an unlimited number of times.

## Finding Description

**Exact code path — `Midnight.take` (`src/Midnight.sol` lines 358–373):**

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);          // = 0 when tick == 0
uint256 _settlementFee = settlementFee(id, timeToMaturity);    // = 0 (default zero-initialized)
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee  // 0 - 0 = 0
                                : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;            // 0 + 0 = 0
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)                        // units * 0 / 1e18 = 0
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group]
                    += offer.buy ? buyerAssets : sellerAssets; // += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets()); // 0 <= maxAssets → always passes
}
```

`TickLib.tickToPrice` is confirmed to return `0` at tick `0` by both the Certora rule `tickToPriceIsZeroAtZero` (`certora/specs/TickToPrice.spec` line 34–36) and the Foundry assertion `assertEq(TickLib.tickToPrice(0), 0, "tick 0")` (`test/TickLibTest.sol` line 16). The `settlementFeeCbp*` fields in `MarketState` are zero-initialized (`src/interfaces/IMidnight.sol` lines 45–51), so `settlementFee()` returns `0` for any freshly created market. The `consumed` increment for a buy offer is `buyerAssets` (`src/Midnight.sol` line 368), which is identically zero when `offerPrice = 0` and `_settlementFee = 0`, regardless of `units`. The guard at line 369 therefore never trips.

**Token transfer consequence:** Both `SafeTransferLib.safeTransferFrom` calls at lines 455–456 transfer `0` tokens (since `buyerAssets = sellerAssets = 0`). The maker (buyer) receives `units` of credit at zero token cost; the taker (seller) takes on `units` of debt and receives `0` tokens.

**Preconditions:**
1. Market with default (zero) settlement fee — zero-initialized by default, no admin action required.
2. Buy offer (`offer.buy = true`) placed at `tick = 0` — always valid since `0 % tickSpacing == 0` for any spacing.
3. `offer.maxAssets > 0` — any positive value signals an assets-capped offer.
4. A ratifier authorized by the maker that returns `CALLBACK_SUCCESS` (standard setup).

**Why existing checks fail:**
- `ConsumedAssets` guard compares `newConsumed` (always `0`) against `offer.maxAssets` (positive) — always passes.
- `SelfTake` check is satisfied by using two distinct addresses.
- The underflow guard on `offerPrice - _settlementFee` only fires when `_settlementFee > offerPrice`; with both zero, `0 - 0 = 0`, no revert.
- No minimum-tick or minimum-price check exists anywhere in `take` or offer validation.

## Impact Explanation

The `maxAssets` cap on a buy offer is completely bypassed: the offer can be taken an unlimited number of times, each time crediting `units` to the maker at zero token cost. The core accounting invariant — that `consumed[maker][group]` faithfully tracks how much of an offer has been filled and enforces the `maxAssets` ceiling — is broken. A maker controlling a second address with sufficient collateral can accumulate unbounded credit in the market without paying any loan tokens, and can subsequently withdraw that credit as tokens whenever `withdrawable` liquidity exists in the market (e.g., from borrower repayments).

## Likelihood Explanation

All preconditions are trivially reachable by any unprivileged user. Every newly deployed market has zero settlement fees by default. Tick `0` is always accessible. The attacker needs only two addresses and enough collateral on the taker address to satisfy the health check at line 476. The attack is repeatable an unlimited number of times within a single transaction via `multicall`.

## Recommendation

Track consumed units rather than consumed assets when `offerPrice` can be zero, or enforce a minimum tick (e.g., `tick >= 1`) in offer validation. Alternatively, add a check that `offerPrice > 0` before allowing an assets-capped offer to proceed, or fall back to units-based accounting when `buyerAssets == 0` to ensure the consumed counter always advances for non-zero `units`.

## Proof of Concept

1. Deploy a market; leave all `settlementFeeCbp*` at their default zero values.
2. Address A (maker) creates a buy offer: `tick = 0`, `buy = true`, `maxAssets = 1e18`, `maxUnits = 0`.
3. Address B (taker, with collateral) calls `take(offer, ..., units = 1e18, taker = B, ...)`.
4. Observe: `offerPrice = 0`, `buyerAssets = 0`, `consumed[A][group] = 0`, check `0 <= 1e18` passes. A gains `1e18` credit; B gains `1e18` debt; zero tokens transferred.
5. Repeat step 3 arbitrarily many times — `consumed` remains `0` after every fill.
6. Assert `consumed[A][group] == 0` and `creditOf(id, A) == N * 1e18` after `N` iterations.
