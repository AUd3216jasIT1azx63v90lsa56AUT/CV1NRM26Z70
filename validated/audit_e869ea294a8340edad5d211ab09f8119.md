Audit Report

## Title
Fee-on-transfer loan token permanently inflates `claimableSettlementFee`, draining lenders' withdrawable balance - (`src/Midnight.sol` / `src/libraries/SafeTransferLib.sol`)

## Summary
`take()` increments `claimableSettlementFee[loanToken]` by the full spread (`buyerAssets - sellerAssets`) at line 418 before executing the inbound transfer at line 455. `SafeTransferLib.safeTransferFrom` only validates the boolean return value of `transferFrom` and never measures the actual amount received. When `loanToken` is a fee-on-transfer token, the contract receives `spread - tokenFee` but `claimableSettlementFee` is credited the full `spread`, creating a permanent per-call deficit. Subsequent `claimSettlementFee` calls drain this shortfall from lenders' `withdrawable` balance, violating the `tokenBalanceCorrect` solvency invariant.

## Finding Description

**Root cause — no token restriction in `touchMarket`:**

`touchMarket` validates maturity, collateral params, LLTV, and `maxLif` but imposes no restriction on `market.loanToken`. Any ERC20, including a fee-on-transfer token, is accepted. [1](#0-0) 

**Accounting write precedes transfer in `take()`:**

At line 418, `claimableSettlementFee` is incremented by the full spread before any transfer occurs: [2](#0-1) 

The actual inbound transfer happens later, after optional buyer callbacks: [3](#0-2) 

**`SafeTransferLib.safeTransferFrom` performs no received-amount check:**

It calls `transferFrom(from, to, value)` and only verifies the call did not revert and returned `true`. There is no balance-before/after measurement, so a fee-on-transfer token silently delivering `value - tokenFee` is indistinguishable from a compliant transfer. [4](#0-3) 

**Exploit flow (`offer.buy == true`, no buyer callback):**
1. Attacker deploys a fee-on-transfer ERC20 (e.g., 1% fee) and calls `touchMarket` with it as `loanToken`.
2. Lenders supply credit; `marketState[id].withdrawable` accumulates.
3. A buy offer exists. Taker calls `take()` with `units` such that `spread = 1000`.
4. Line 418: `claimableSettlementFee[loanToken] += 1000`.
5. Line 455: `safeTransferFrom(..., 1000)` — token charges 10; contract receives 990.
6. Post-`take()`: `claimableSettlementFee = 1000`, actual balance held for fees = 990. Deficit = 10.
7. Each subsequent `take()` accumulates the deficit linearly.
8. `claimSettlementFee(claimableSettlementFee)` transfers the full inflated amount, consuming 10 units that backed lenders' `withdrawable`.
9. Lenders calling `withdraw()` find the contract insolvent proportional to cumulative fee deductions.

**Why existing checks fail:**

The Certora `Solvency.spec` explicitly models this away: line 31–33 states *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits"* and `CVL_transferFrom` delivers exactly `value` tokens. [5](#0-4) 

The `pendingFeeReceipt` ghost (lines 140–151) tracks the gap between the `claimableSettlementFee` increment and the inbound transfer, and clears it in `CVL_transferFrom` only when the exact expected amount arrives — an assumption that breaks for fee-on-transfer tokens. [6](#0-5) 

The `tokenBalanceCorrect` strong invariant is therefore only proven under the no-fee-on-transfer assumption and does not hold in practice for such tokens. [7](#0-6) 

## Impact Explanation
Every `take()` on a market whose `loanToken` is fee-on-transfer inflates `claimableSettlementFee` by `tokenFee`. When `claimSettlementFee` is called (a routine privileged operation), it transfers the inflated amount from the contract, consuming funds that back lenders' `withdrawable`. The `tokenBalanceCorrect` invariant — `balance >= collateralSum + withdrawableSum + claimableSettlementFee` — is violated after each such `take()`. Lenders suffer credit destruction: their on-chain `withdrawable` credit is valid but the underlying token balance is insufficient to honour it, causing insolvency proportional to cumulative fee deductions. This constitutes direct, quantifiable theft of lender funds.

## Likelihood Explanation
All three preconditions are reachable without any privileged action:
1. `loanToken` is a fee-on-transfer ERC20 — deployable by any unprivileged market creator at zero cost, or a legitimate real-world token with transfer fees.
2. `offer.buy == true` and settlement fee > 0 — the default settlement fee schedule is non-zero for non-zero TTM markets.
3. A buy offer exists with an approved payer — normal market operation.

The attack is repeatable on every `take()` call, scales linearly with trade volume, and requires no oracle manipulation, admin key, or user mistake. The only social precondition is lenders supplying credit to a market with a fee-on-transfer loan token, which is realistic for legitimate tokens that carry transfer fees.

## Recommendation
Two complementary fixes:

1. **Balance-delta check in `SafeTransferLib.safeTransferFrom`**: Record `balanceBefore = IERC20(token).balanceOf(to)` before the call and assert `IERC20(token).balanceOf(to) - balanceBefore == value` after. This makes the transfer atomic with its accounting guarantee.

2. **Token allowlist or fee-on-transfer guard in `touchMarket`**: Require that `loanToken` is on a protocol-maintained allowlist of audited, standard-compliant ERC20 tokens, or perform a probe transfer at market creation to verify no fee is deducted.

Fix (1) is the minimal, self-contained defence; fix (2) provides defence-in-depth and prevents other token-behaviour edge cases (rebasing, blacklisting).

## Proof of Concept
**Minimal Foundry fork test plan:**

```solidity
// 1. Deploy FeeToken: transferFrom deducts 1% and delivers 99% to recipient.
// 2. Call touchMarket(market) where market.loanToken = address(feeToken).
// 3. Lender calls supply/deposit so marketState[id].withdrawable = 10_000.
// 4. Create a buy offer (maker = buyer, approved for feeToken).
// 5. Taker calls take() with units producing spread = 1_000.
//    Assert: claimableSettlementFee[feeToken] == 1_000
//    Assert: feeToken.balanceOf(midnight) increased by only 990
// 6. Repeat take() 10 times.
//    Assert: claimableSettlementFee[feeToken] == 10_000
//    Assert: feeToken.balanceOf(midnight) increased by only 9_900 (deficit = 100)
// 7. Call claimSettlementFee(feeToken, 10_000).
//    Assert: feeToken.balanceOf(midnight) decreased by 10_000
//    Assert: lender's withdrawable (10_000) is now unbacked by 100 tokens
// 8. Lender calls withdraw(10_000) — reverts or delivers 9_900, proving insolvency.
```

The test directly demonstrates that `tokenBalanceCorrect` is violated: `balance < collateralSum + withdrawableSum + claimableSettlementFee` after step 5, and that lender funds are irrecoverably consumed after step 7.

```text
// Settlement fee receipts pending settlement: claimableSettlementFee is incremented in take before
// the inbound fee transfer happens, so we track the gap and clear it in CVL_transferFrom.
persistent ghost mapping(address => mathint) pendingFeeReceipt {
    init_state axiom (forall address token. pendingFeeReceipt[token] == 0);
}

hook Sstore claimableSettlementFee[KEY address token] uint256 newVal (uint256 oldVal) {
    // Except for claimSettlementFee, the claimableSettlementFee is non-decreasing, see WithdrawableMonotonicity.spec.
    if (newVal > oldVal) {
        pendingFeeReceipt[token] = pendingFeeReceipt[token] + newVal - oldVal;
    }
}
```
