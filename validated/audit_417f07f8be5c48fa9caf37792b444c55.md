Audit Report

## Title
`claimContinuousFee` reverts via arithmetic underflow when bad-debt liquidation front-runs the feeClaimer - (File: `src/Midnight.sol`)

## Summary
`claimContinuousFee` performs an unchecked subtraction on `continuousFeeCredit` with no guard that `amount <= continuousFeeCredit`. An unprivileged attacker can call `liquidate` with `seizedAssets=0, repaidUnits=0` at zero token cost to realize existing bad debt, atomically reducing `continuousFeeCredit` before the feeClaimer's pending transaction executes, causing it to revert with a Solidity 0.8 arithmetic underflow. The attack is repeatable and constitutes a sustained denial-of-service against protocol fee collection.

## Finding Description
**Root cause — `claimContinuousFee`** (`src/Midnight.sol:318`):

```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
```

No `require(amount <= continuousFeeCredit)` guard exists. Solidity 0.8 checked arithmetic reverts on underflow. The Certora spec at `certora/specs/Role.spec:264` encodes this revert condition explicitly:

```
assert !reverted <=> ... && amount <= continuousFeeCreditBefore;
```

confirming `amount <= continuousFeeCredit` is a necessary non-revert condition with no on-chain enforcement.

**Bad-debt slash — `liquidate`** (`src/Midnight.sol:635-640`):

```solidity
_marketState.continuousFeeCredit = _lossFactor < type(uint128).max
    ? UtilsLib.toUint128(
        _marketState.continuousFeeCredit
            .mulDivDown(type(uint128).max - _marketState.lossFactor,
                        type(uint128).max - _lossFactor)
      )
    : 0;
```

Because `newLossFactor > oldLossFactor`, the ratio `(MAX − newLF) / (MAX − oldLF) < 1`, so `continuousFeeCredit` is strictly reduced on every bad-debt event.

**Zero-cost liquidation path**: The protocol explicitly supports `seizedAssets=0, repaidUnits=0` calls (`src/Midnight.sol:577`):

> "Passing both 0 for seizedAssets and repaidUnits allows to realize bad debt with 0 token transferred."

The `atMostOneNonZero` check at line 595 passes when both are zero (zero non-zero values satisfies "at most one"). The token-transfer block at line 643 (`if (repaidUnits > 0 || seizedAssets > 0)`) is skipped entirely. The only cost to the attacker is gas.

**Exploit flow**:
1. `continuousFeeCredit = X`. feeClaimer broadcasts `claimContinuousFee(market, X, receiver)`.
2. Attacker front-runs with `liquidate(market, 0, 0, 0, borrower, false, receiver, address(0), "")` — zero tokens transferred. Bad debt is realized; `continuousFeeCredit` drops to `X' < X`.
3. feeClaimer's transaction executes: `X' -= X` → arithmetic underflow → revert.

**Why existing checks fail**: The `liquidatorGate` check (`src/Midnight.sol:597-600`) only applies when `market.liquidatorGate != address(0)`; markets with `liquidatorGate == address(0)` allow any caller. No minimum-amount or slippage guard exists in `claimContinuousFee`.

## Impact Explanation
The feeClaimer's `claimContinuousFee` transactions are persistently griefed and revert. Protocol fee collection is disrupted: the feeClaimer must re-read on-chain state and resubmit on every attempt. The attack is zero-cost (gas only) and can be replayed across multiple bad-debt positions or repeated price-drop events, constituting a sustained denial-of-service against a core protocol revenue function.

## Likelihood Explanation
**Attacker cost**: gas only — no tokens required. **Preconditions**: a liquidatable position with `badDebt > 0` must exist (routine in any lending market experiencing price volatility) and the feeClaimer must submit `amount` equal to the current `continuousFeeCredit` (the natural "claim all" pattern). **Repeatability**: each distinct bad-debt position or subsequent price drop creates a new front-run opportunity. **Access**: any unprivileged external address can call `liquidate` on markets without a `liquidatorGate`.

## Recommendation
Add a guard in `claimContinuousFee` to cap `amount` at the available credit, or use a safe subtraction:

```solidity
uint128 credit = _marketState.continuousFeeCredit;
require(UtilsLib.toUint128(amount) <= credit, InsufficientContinuousFeeCredit());
_marketState.continuousFeeCredit = credit - UtilsLib.toUint128(amount);
```

Alternatively, allow the feeClaimer to pass `type(uint256).max` as a sentinel to claim all available credit atomically, reading the current value on-chain rather than relying on a pre-observed value.

## Proof of Concept
1. Deploy a market with `liquidatorGate == address(0)`.
2. Create a borrower position that becomes undercollateralized (bad debt: collateral value < debt even at max liquidation incentive).
3. Observe `continuousFeeCredit = X > 0` (accrued over time).
4. Submit `claimContinuousFee(market, X, receiver)` from the feeClaimer account.
5. Before it lands, call `liquidate(market, 0, 0, 0, borrower, false, attacker, address(0), "")` — zero tokens transferred, bad debt realized, `continuousFeeCredit` drops to `X' < X`.
6. feeClaimer's transaction reverts with arithmetic underflow at `src/Midnight.sol:318`.
7. Repeat step 4–6 on every feeClaimer retry to sustain the DoS.
