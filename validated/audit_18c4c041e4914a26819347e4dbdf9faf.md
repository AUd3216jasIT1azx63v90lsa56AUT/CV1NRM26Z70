Audit Report

## Title
Fully-Consumed Assets-Mode Buy Offer with `buyerPrice < WAD` Allows Unbounded Credit/Debt Creation with Zero Token Transfer — (`src/Midnight.sol`)

## Summary
When `offer.buy = true`, `offer.maxAssets > 0`, and `buyerPrice < WAD`, calling `take()` with a small `units` value causes `buyerAssets = units.mulDivDown(buyerPrice, WAD)` to round down to `0`. The consumed-cap check adds `0` to `consumed[maker][group]`, so it trivially passes even when the offer is fully consumed. Position accounting then increases `buyerPos.credit` and `sellerPos.debt` by `units` while transferring zero tokens, breaking the solvency invariant that contract balances cover all credit. The attack is repeatable without limit.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

`buyerAssets` is computed with `mulDivDown`, which rounds toward zero: [1](#0-0) 

When `units` is small enough that `units * buyerPrice < WAD`, `buyerAssets == 0`. The consumed-cap branch then adds `0` to `consumed`: [2](#0-1) 

Because `newConsumed` is unchanged, `newConsumed <= offer.maxAssets` always passes even when `consumed[maker][group]` is already at `maxAssets`.

**Position accounting uses `units`, not `buyerAssets` — lines 382–414:** [3](#0-2) [4](#0-3) 

`buyerPos.credit` increases by `units`, `sellerPos.debt` increases by `units`, and `totalUnits` increases — all with no token backing.

**Token transfer — line 455:** [5](#0-4) 

`buyerAssets - sellerAssets == 0 - 0 == 0`, so no tokens are pulled.

**Why existing checks fail:**
- `ConsumedAssets` guard (line 369): checks `newConsumed <= maxAssets`. Since `buyerAssets == 0`, `newConsumed` is unchanged and the check trivially passes.
- `reduceOnly` guard (line 393): blocks takes where `buyerCreditIncrease == 0` for a buy offer, but here `buyerCreditIncrease == units > 0`, so it does not trigger. [6](#0-5) 
- No guard requires `buyerAssets > 0` when `units > 0`.

**Protocol acknowledgment — line 94:** [7](#0-6) 

The NatSpec documents this path as reachable and does not block it.

## Impact Explanation

After a buy offer is fully consumed (`consumed == maxAssets`), any unprivileged taker can call `take(units=1, ...)` repeatedly. Each call increases `creditOf(maker)` by `units` and `debtOf(taker)` by `units` while transferring zero tokens. The maker's credit grows without bound beyond `maxAssets` with no token deposit backing it. The contract's loan-token balance does not increase to cover the new credit, directly violating the solvency invariant (confirmed by `certora/specs/Solvency.spec`). At maturity, the maker can attempt to withdraw credit that has no corresponding token deposit, causing insolvency or failed withdrawals for other lenders. [8](#0-7) 

## Likelihood Explanation

All preconditions are reachable by an unprivileged taker:
1. `offer.buy = true` — standard lender offer.
2. `offer.maxAssets > 0` — maker uses assets-mode cap.
3. `offer.tick < MAX_TICK` — any tick where `tickToPrice(tick) < WAD`.
4. `units = 1` — trivially satisfies `units.mulDivDown(buyerPrice, WAD) == 0` at any sub-WAD price.
5. `consumed[maker][group] >= maxAssets` — offer is fully consumed (legitimately or via `setConsumed`).

The attack is repeatable indefinitely since `consumed` never advances past `maxAssets`. Any taker observing a fully-consumed buy offer at a sub-WAD tick can execute this with no special access.

## Recommendation

Add a guard in the `maxAssets` branch requiring that `buyerAssets > 0` whenever `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` such that `units.mulDivDown(buyerPrice, WAD) >= 1` before proceeding with position accounting. This ensures the consumed counter always advances when `units > 0`, making the cap check meaningful.

## Proof of Concept

A passing test already exists in the repository at `test/TakeTest.sol` lines 858–889, named `testBugBuyMaxAssetsBypass`: [9](#0-8) 

The test:
1. Sets `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (price < WAD).
2. Pre-consumes the offer fully via `midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender)`.
3. Calls `take(1, borrower, lenderOffer)`.
4. Asserts `buyerAssets == 0`, `sellerAssets == 0` (zero transfer).
5. Asserts `consumed` is unchanged (still at `maxAssets`).
6. Asserts `creditOf(lender)` **increased** — credit grew without token backing.
7. Asserts `debtOf(borrower)` **increased** — debt grew without token transfer.
8. Asserts `totalUnits` **increased**.

The test passes, confirming the exploit path is live and reproducible.
