Audit Report

## Title
Missing `loanToken != address(0)` validation in `touchMarket` allows creation of permanently broken markets - (`src/Midnight.sol`)

## Summary
`touchMarket` validates collateral tokens, LLTV, maxLif, and maturity but contains no guard against `market.loanToken == address(0)`. Any unprivileged caller can create a market with `loanToken = address(0)`, which succeeds silently. Every subsequent `take()` on that market permanently reverts with `SafeTransferLib.NoCode()` because `address(0).code.length == 0`, and the market cannot be deleted once created.

## Finding Description
**Root cause — `touchMarket` (`src/Midnight.sol`, lines 755–791):**

The creation guard checks maturity, collateral count, sort order, LLTV tier, and maxLif, but contains no `require(market.loanToken != address(0))`: [1](#0-0) 

The collateral sort check (`collateralToken > previousCollateralToken`, where `previousCollateralToken` starts at `address(0)`) only prevents collateral tokens from being `address(0)` — it does not protect the loan token field. [2](#0-1) 

When `market.loanToken = address(0)`, the market is created with all-zero settlement fees (since `defaultSettlementFeeCbp[address(0)]` returns a zero-initialized array): [3](#0-2) 

**Trigger — `take()` (`src/Midnight.sol`, lines 455–456):**

Every `take()` call reaches the token transfer step and calls `SafeTransferLib.safeTransferFrom` with `offer.market.loanToken`: [4](#0-3) 

`SafeTransferLib.safeTransferFrom` immediately reverts because `address(0).code.length == 0`: [5](#0-4) 

**Misleading helper — `TakeAmountsLib.buyerAssetsToUnits` (`src/periphery/TakeAmountsLib.sol`, lines 17–30):**

With `settlementFee = 0` (from the zero-initialized market), `buyerPrice = offerPrice` (nonzero for any valid tick), and the function returns a nonzero unit count — signaling to callers that a take is feasible when it will always revert: [6](#0-5) 

**Confirmed reachable — `test/SettersTest.sol`, lines 267–276:**

The test suite explicitly creates a market with `loanToken: address(0)` and calls `midnight.touchMarket(market)` successfully, confirming the path is exercised without error: [7](#0-6) 

## Impact Explanation
Any market created with `loanToken = address(0)` is permanently broken: all `take()` calls revert with `NoCode()`, the market state is immutable (no deletion mechanism exists), and `TakeAmountsLib.buyerAssetsToUnits` returns nonzero values that falsely indicate takes are executable. Any offers posted to such a market are permanently frozen. This matches the "permanent lock/freeze of user/project state" impact class in RESEARCHER.md.

## Likelihood Explanation
`touchMarket` is a public, permissionless function — no role, signature, or special permission is required. The precondition (supply `loanToken = address(0)` with any valid collateral/LLTV/maturity) is trivially satisfiable in a single transaction. The attack is repeatable for any `(address(0), maturity)` pair not yet created, and the broken market is immutable once deployed.

## Recommendation
Add an explicit zero-address guard for the loan token at the start of the market creation block in `touchMarket`:

```solidity
require(market.loanToken != address(0), InvalidLoanToken());
```

This should be placed alongside the existing `maturity` and `collateralParams` checks (around line 758–759 of `src/Midnight.sol`). Optionally, extend the Certora spec `createdMarketsHaveNonZeroCollaterals` to also assert `market.loanToken != address(0)` for all created markets.

## Proof of Concept
1. Deploy `Midnight` on a local fork.
2. Construct a `Market` with `loanToken = address(0)`, a valid collateral token, allowed LLTV, valid maxLif, and `maturity = block.timestamp + 1 days`.
3. Call `midnight.touchMarket(market)` — succeeds, market is created.
4. Have any address post an offer on this market.
5. Call `midnight.take(offer, ...)` — reverts with `SafeTransferLib.NoCode()`.
6. Confirm via `TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, amount)` that it returns a nonzero value despite the take being permanently broken.

The existing test at `test/SettersTest.sol:267–276` already demonstrates step 3 succeeds; steps 4–6 can be appended to that test to complete the PoC.
