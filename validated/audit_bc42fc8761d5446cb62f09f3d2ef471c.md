Audit Report

## Title
`sellerAssetsToUnits` division-by-zero when `sellerPrice == 0` blocks bundler sell flows for affected buy offers - (`File: src/periphery/TakeAmountsLib.sol`)

## Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and uses it as the denominator in `mulDivUp(targetSellerAssets, WAD, sellerPrice)` with no guard against `sellerPrice == 0`. When the feeSetter sets the index-6 settlement fee to `MAX_SETTLEMENT_FEE_360_DAYS = 0.005e18` and a buy offer exists at the tick whose price equals exactly `0.005e18`, every call to `sellerAssetsToUnits` for that offer panics with a Solidity 0.8 arithmetic panic. Because this call in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` occurs before the `try/catch` wrapping `IMidnight.take`, the panic propagates and reverts the entire bundler transaction.

## Finding Description
**Vulnerable code:**

`src/periphery/TakeAmountsLib.sol`, `sellerAssetsToUnits`, lines 41–46:
```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
uint256 settlementFee =
    IMidnight(midnight).settlementFee(id, UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp));
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : targetSellerAssets.mulDivDown(WAD, sellerPrice);
``` [1](#0-0) 

There is no `require(sellerPrice > 0)` guard before the division. The sibling function `buyerAssetsToUnits` computes `sellerPrice` at line 26 and then checks `require(buyerPrice <= WAD)` at line 28, but this guard does not protect against `sellerPrice == 0` — it only catches `buyerPrice > WAD`. `sellerAssetsToUnits` has no analogous guard at all. [2](#0-1) 

**Why `sellerPrice == 0` is reachable:**

- `PRICE_ROUNDING_STEP = 1e12` and `MAX_SETTLEMENT_FEE_360_DAYS = 0.005e18 = 5e15`. Since `5e15` is a multiple of `1e12`, the rounding in `tickToPrice` (which rounds to multiples of `PRICE_ROUNDING_STEP`) can produce exactly `0.005e18` for some tick `t*`. [3](#0-2) [4](#0-3) 

- `maxSettlementFee(6) = MAX_SETTLEMENT_FEE_360_DAYS = 0.005e18`. Setting index 6 to `0.005e18` passes the `<= maxSettlementFee(index)` check and is a valid feeSetter action. [5](#0-4) 

- For a market with TTM ≥ 360 days, `settlementFee(id, ttm) = 0.005e18`, so `sellerPrice = 0.005e18 − 0.005e18 = 0`.

**Why the bundler `try/catch` does NOT protect against this:**

In `MidnightBundles.supplyCollateralAndSellWithAssetsTarget`, `TakeAmountsLib.sellerAssetsToUnits` is called at line 286, *before* the `try { IMidnight(MIDNIGHT).take(...) } catch {}` block at line 292. The arithmetic panic from `mulDivUp(..., 0)` is thrown at line 286 and propagates directly to the caller, reverting the entire bundler call. [6](#0-5) 

**Why `Midnight.take()` does NOT protect against this:** `Midnight.take()` computes `sellerAssets = units.mulDivDown(0, WAD) = 0` — no revert. The equal-to case silently produces `sellerAssets = 0` and succeeds, meaning the core protocol does not gate this condition.

## Impact Explanation
Every call to `supplyCollateralAndSellWithAssetsTarget` that includes the affected buy offer reverts unconditionally. Users cannot use the bundler to sell against that offer. The bundler sell path — the primary interface for asset-targeted selling — is frozen for all affected offers until the fee is lowered or the offer expires. Direct `Midnight.take()` still works (producing `sellerAssets = 0`), but the bundler path is completely blocked.

## Likelihood Explanation
The feeSetter setting the 360-day fee to its maximum (`0.005e18`) is a routine governance action with no indication it is unsafe. The tick whose price equals exactly `0.005e18` is a valid, accessible tick (price is a multiple of `PRICE_ROUNDING_STEP`). Any maker who placed a buy offer at that tick is affected. The condition is deterministic and repeatable: every bundler call with that offer will panic until the fee is lowered or the offer expires. No attacker action is required beyond the normal feeSetter governance action and the existence of a buy offer at the affected tick.

## Recommendation
Add a guard in `sellerAssetsToUnits` before the division, mirroring the intent of the NatDoc comment that already documents the revert expectation:

```solidity
require(sellerPrice > 0, SomeError());
```

This should be placed after line 44 in `src/periphery/TakeAmountsLib.sol`, before the `return` statement. Alternatively, move the `sellerAssetsToUnits` call inside the `try/catch` block in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` so that a panic on any individual offer does not revert the entire bundler transaction — though the explicit guard is the more correct fix since `sellerPrice == 0` represents a degenerate offer state. [7](#0-6) 

## Proof of Concept
1. Deploy a market with TTM ≥ 360 days.
2. feeSetter calls `setMarketSettlementFee(id, 6, 0.005e18)` — valid, within bounds.
3. A maker places a buy offer at tick `t*` where `tickToPrice(t*) = 0.005e18`.
4. Any user calls `supplyCollateralAndSellWithAssetsTarget(...)` including that offer in the `takes` array.
5. At line 286, `TakeAmountsLib.sellerAssetsToUnits` is called → `sellerPrice = 0.005e18 − 0.005e18 = 0` → `mulDivUp(target, WAD, 0)` → Solidity 0.8 arithmetic panic (division by zero) → entire bundler call reverts.

A fuzz test can confirm by iterating over all ticks to find `t*` where `tickToPrice(t*) == 0.005e18`, then asserting that `sellerAssetsToUnits` reverts for any nonzero `targetSellerAssets` when the settlement fee equals the offer price.
