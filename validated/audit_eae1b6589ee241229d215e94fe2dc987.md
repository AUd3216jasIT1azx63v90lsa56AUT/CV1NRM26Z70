### Title
Post-Maturity Liquidation Reverts Due to Arithmetic Underflow When Bad Debt and Collateral Seizure Occur in Same Call — (File: src/Midnight.sol)

### Summary

In `Midnight.sol`'s `liquidate` function, when a position has bad debt (`badDebt > 0`) and the liquidation is in post-maturity mode with `lif < maxLif` (i.e., within the first `TIME_TO_MAX_LIF = 15 minutes` after maturity), the function first reduces `_position.debt` by `badDebt` (computed using `maxLif`), then attempts to subtract `repaidUnits` (computed using the smaller `lif`). Because `lif < maxLif`, `repaidUnits` can exceed the remaining debt, causing an arithmetic underflow revert that blocks the liquidation entirely.

---

### Finding Description

**Root Cause**

In `liquidate`, two sequential subtractions from `_position.debt` use inconsistent LIF values:

**Step 1 — Bad debt removal (uses `maxLif`):** [1](#0-0) 

`badDebt` is computed by subtracting each collateral's coverage at `maxLif` from `originalDebt`:

```solidity
badDebt = badDebt.zeroFloorSub(
    _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
);
```

After this block, `_position.debt -= uint128(badDebt)`, leaving:

```
remaining_debt ≈ Σ( collateral[i] * price[i] / ORACLE_PRICE_SCALE * WAD / maxLif[i] )
```

**Step 2 — Collateral seizure (uses `lif`, which is < `maxLif` in early post-maturity):** [2](#0-1) 

```solidity
uint256 lif = postMaturityMode
    ? UtilsLib.min(_maxLif, WAD + (_maxLif - WAD) * (block.timestamp - market.maturity) / TIME_TO_MAX_LIF)
    : _maxLif;

if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
}
...
_position.debt -= UtilsLib.toUint128(repaidUnits);  // ← UNDERFLOW
```

Since `lif < maxLif`, dividing by `lif` yields a larger `repaidUnits` than dividing by `maxLif`. There is **no cap** ensuring `repaidUnits <= _position.debt`.

**The Mismatch:**

| Quantity | Divisor Used |
|---|---|
| `badDebt` (reduces `_position.debt`) | `maxLif` |
| `repaidUnits` (further reduces `_position.debt`) | `lif` (≤ `maxLif`) |

When `lif < maxLif`, `repaidUnits` can exceed the remaining debt, causing the subtraction at line 676 to underflow and revert the entire transaction. [3](#0-2) 

**Vulnerable window:** `TIME_TO_MAX_LIF = 15 minutes` after maturity. [4](#0-3) 

---

### Impact Explanation

Any call to `liquidate` that simultaneously:
1. Triggers bad-debt realization (`badDebt > 0`), AND
2. Passes `seizedAssets > 0` (or `repaidUnits > 0`)

...will revert with an arithmetic underflow for the first 15 minutes after market maturity. Since the entire transaction reverts, **bad debt is not socialized** and **collateral is not seized** during this window. Lenders cannot withdraw funds, and the protocol's accounting remains in an unhealthy state until either:
- The 15-minute window expires (lif reaches maxLif), or
- A liquidator knows to split into two separate calls (realize bad debt first with both inputs = 0, then seize collateral in a second call with a carefully bounded `seizedAssets`)

Even with the two-call workaround, the liquidator must cap `seizedAssets` to `_position.debt * lif / WAD * ORACLE_PRICE_SCALE / price` to avoid underflow in the second call, meaning they cannot seize all collateral during the window.

---

### Likelihood Explanation

- Post-maturity mode is a normal, expected protocol state for every market.
- Bad debt is a realistic outcome when collateral prices drop below the debt value.
- Any liquidator bot that calls `liquidate` with `seizedAssets > 0` on a bad-debt position within 15 minutes of maturity will have their transaction revert.
- The 15-minute window is short but is precisely the period when liquidation incentives are lowest (lif ≈ WAD), meaning liquidators are most likely to attempt liquidation to capture the growing incentive — making this a realistic trigger.

---

### Recommendation

Before executing the collateral seizure, cap `repaidUnits` to the remaining `_position.debt`:

```solidity
// After bad debt removal and repaidUnits computation:
repaidUnits = UtilsLib.min(repaidUnits, _position.debt);
// Recompute seizedAssets accordingly if repaidUnits was capped
```

Alternatively, add an explicit check:

```solidity
require(repaidUnits <= _position.debt, RepaidExceedsDebt());
```

so the revert is informative rather than a silent underflow.

---

### Proof of Concept

**Setup:**
- Single-collateral market, `lltv = 0.86e18`, `maxLif ≈ 1.0753e18` (cursor = 0.5)
- `originalDebt = 100e18`, `collateral = 100e18`, `price = ORACLE_PRICE_SCALE` (1:1)
- `maxDebt = 100 * 0.86 = 86` → position is unhealthy ✓

**At maturity (`block.timestamp == market.maturity`):**

1. `badDebt = 100 - ceil(100 * 1e18 / 1.0753e18) = 100 - 93 = 7`
2. `_position.debt -= 7` → `_position.debt = 93`
3. `lif = WAD = 1e18` (at maturity, lif starts at 1.0)
4. Liquidator passes `seizedAssets = 100` (all collateral)
5. `repaidUnits = ceil(100 * 1e36 / 1e36 * 1e18 / 1e18) = 100`
6. `_position.debt -= 100` → `93 - 100` → **arithmetic underflow → revert**

The liquidation fails. The bad debt is not socialized. The collateral remains locked in the position. The transaction must be retried after 15 minutes when `lif = maxLif`, at which point `repaidUnits = ceil(100 * 1e18 / 1.0753e18) = 93 ≤ 93 = _position.debt`, and the liquidation succeeds.
