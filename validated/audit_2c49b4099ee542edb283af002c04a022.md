Audit Report

## Title
Unconditional zero-amount `safeTransfer` in `liquidate()` permanently blocks bad-debt-only liquidations for zero-revert collateral tokens - (File: src/Midnight.sol)

## Summary
`liquidate()` unconditionally calls `SafeTransferLib.safeTransfer(collateralToken, receiver, seizedAssets)` at line 696 even when `seizedAssets = 0`, directly contradicting the NatSpec at line 577 which explicitly guarantees "Passing both 0 for seizedAssets and repaidUnits allows to realize bad debt with 0 token transferred." For any market whose collateral token reverts on zero-amount `transfer()` calls, the bad-debt-only liquidation path is permanently DoS'd, preventing `lossFactor` updates and lender loss socialization.

## Finding Description

**Verified code path:**

The NatSpec at line 577 explicitly documents the zero-zero path: [1](#0-0) 

The guard at line 595 permits both inputs being zero (`atMostOneNonZero` returns true when both are zero): [2](#0-1) 

The block at line 643 is skipped entirely when both inputs are zero, leaving `seizedAssets = 0`: [3](#0-2) 

Bad-debt accounting at lines 626–641 executes correctly and modifies `_position.debt` and `_marketState`: [4](#0-3) 

Execution then falls through unconditionally to: [5](#0-4) 

`SafeTransferLib.safeTransfer` has no zero-amount guard — it always issues the low-level `token.call`: [6](#0-5) 

For a collateral token that reverts on zero-amount `transfer()`, this causes the entire transaction to revert, rolling back all state changes including the bad-debt accounting.

Additionally, `SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits)` at line 717 is also called unconditionally with `repaidUnits = 0` in the same path, compounding the issue if the loan token also reverts on zero transfers: [7](#0-6) 

**Root cause:** Missing `if (seizedAssets > 0)` guard before the `safeTransfer` call at line 696 (and symmetrically, missing `if (repaidUnits > 0)` before line 717).

## Impact Explanation

Bad-debt-only liquidations (`seizedAssets = 0, repaidUnits = 0`) are permanently blocked for any market whose collateral token reverts on zero-amount transfers. The consequences are:

- `_marketState.lossFactor` is never updated — lenders cannot recover proportional losses
- `_position.debt` is never reduced — the borrower's bad debt persists indefinitely
- The protocol invariant "bad debt must reduce lender credit exactly once and proportionally" is violated
- The protocol's own documented guarantee at line 577 is broken

This constitutes a permanent freeze of bad-debt realization and credit/debt accounting corruption, matching the "permanent or long-term fund freeze" and "bad debt creation" impact classes listed in `live_context.json`. [8](#0-7) 

## Likelihood Explanation

**Preconditions:**
1. A market exists with a collateral token that reverts on zero-amount `transfer()`. Several deployed ERC20 tokens exhibit this behavior. The market creator role is explicitly listed as an unprivileged attacker in scope.
2. A borrower's position reaches `badDebt > 0` (collateral value insufficient to cover debt at `maxLif`), which is a normal market outcome.

Non-standard token behavior is explicitly listed as in-scope in `live_context.json` under `external_calls` and `recommended_fuzz_axes`: [9](#0-8) [10](#0-9) 

Once such a market exists and a position reaches bad-debt state, the DoS is triggered by any liquidator on every attempt. It is repeatable and permanent — market configuration is immutable post-deployment, so the collateral token cannot be replaced.

## Recommendation

Add a zero-amount guard before the `safeTransfer` call at line 696:

```solidity
if (seizedAssets > 0) {
    SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
}
```

Symmetrically, guard the `safeTransferFrom` at line 717:

```solidity
if (repaidUnits > 0) {
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
}
```

This aligns the implementation with the documented guarantee at line 577 and ensures bad-debt-only liquidations succeed regardless of token behavior on zero-amount transfers.

## Proof of Concept

1. Deploy a mock ERC20 collateral token that reverts on `transfer(to, 0)`.
2. Create a market with this token as `collateralParams[0].token`.
3. Have a borrower deposit collateral and take debt.
4. Manipulate the oracle price so that `badDebt > 0` (collateral value < debt / maxLif).
5. Call `liquidate(market, 0, 0, 0, borrower, false, receiver, address(0), "")`.
6. Observe: transaction reverts at line 696 due to the zero-amount transfer revert.
7. Confirm: `_marketState.lossFactor` is unchanged, `_position.debt` is unchanged.
8. Apply fix (guard `safeTransfer` with `if (seizedAssets > 0)`), re-run step 5.
9. Observe: transaction succeeds, `lossFactor` is updated, `_position.debt` is reduced.

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
