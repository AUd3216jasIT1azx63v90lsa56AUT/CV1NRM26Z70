Audit Report

## Title
Zero Oracle Price Enables Free Collateral Seizure via `seizedAssets` Path — (`File: src/Midnight.sol`)

## Summary
When a liquidated collateral's oracle returns `0`, calling `liquidate` with `seizedAssets > 0` and `repaidUnits = 0` computes `repaidUnits = 0` without reverting. The liquidator receives real collateral tokens while transferring zero loan tokens. The `repaidUnits > 0` path correctly reverts via division-by-zero, but the `seizedAssets > 0` path has no equivalent guard, directly contradicting the protocol's own Certora README claim that zero-returning oracles block `liquidate`.

## Finding Description

**Oracle price loop — lines 607–618:**

When `price = 0` for the liquidated collateral:
- `liquidatedCollatPrice = 0` (line 611)
- `maxDebt += mulDivDown(_collateral, 0, ORACLE_PRICE_SCALE) = 0` (line 613)
- `badDebt = zeroFloorSub(originalDebt, mulDivUp(_collateral, 0, ORACLE_PRICE_SCALE)) = originalDebt` (lines 614–616) [1](#0-0) 

**Liquidatability check — line 622:**

`originalDebt > maxDebt` → `originalDebt > 0` → `true`. The position is liquidatable because `_position.debt > 0` is enforced at line 596. [2](#0-1) 

**Bad debt zeroes the debt — line 628:**

`_position.debt -= uint128(badDebt)` = `_position.debt -= originalDebt` → `_position.debt = 0`. [3](#0-2) 

**`seizedAssets > 0` branch — line 650:**

```solidity
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```

- `mulDivUp(seizedAssets, 0, ORACLE_PRICE_SCALE)`: denominator is `ORACLE_PRICE_SCALE = 1e36 > 0`; numerator product is `seizedAssets * 0 = 0`; result = `(0 + 1e36 - 1) / 1e36 = 0`. No revert.
- `mulDivUp(0, WAD, lif)`: `lif >= WAD > 0`; result = `0`. No revert.
- `repaidUnits = 0`. [4](#0-3) 

**Contrast with `repaidUnits > 0` path — line 652:**

```solidity
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
```

`mulDivDown(x, ORACLE_PRICE_SCALE, 0)` → division by zero → revert. This asymmetry is the root cause. [5](#0-4) 

**RCF check — lines 659–667:**

After bad debt, `_position.debt = 0` and `maxDebt = 0`, so `_position.debt - maxDebt = 0` and `maxRepaid = mulDivUp(0, ...) = 0`. The check `repaidUnits <= maxRepaid` → `0 <= 0` → passes silently. [6](#0-5) 

**Token transfers — lines 696 and 717:**

- `safeTransfer(collateralToken, receiver, seizedAssets)` — liquidator receives `seizedAssets` collateral tokens.
- `safeTransferFrom(loanToken, payer, address(this), 0)` — liquidator pays zero loan tokens. [7](#0-6) [8](#0-7) 

**Certora spec gap confirmed:**

`Reverts.spec` rule `oracleZeroCausesLiquidateWithRepaidRevert` (line 246) only covers `repaidUnits > 0` with `seizedAssets = 0`. There is no corresponding rule for `seizedAssets > 0` with `repaidUnits = 0`. [9](#0-8) 

`NoDivisionByZero.spec` explicitly requires `ghostPrice(...) > 0` as a precondition for `noDivisionByZeroLiquidate`, confirming the zero-price case is a known gap, not an impossible scenario. [10](#0-9) 

`LiquidationProfitability.spec` rule `liquidationLifSeizedAssets` asserts `seizedResult * price > (repaidResult - 1) * ORACLE_PRICE_SCALE()`. With `price = 0` and `repaidResult = 0`, this becomes `0 > -ORACLE_PRICE_SCALE()` in `mathint` — vacuously true. The profitability spec does not catch this case. [11](#0-10) 

The `certora/README.md` line 70 states: *"A reverting or zero-returning collateral oracle blocks `liquidate`, `withdrawCollateral`, `isHealthy` and `take` whenever the borrower has debt."* This is false for `liquidate` with `seizedAssets > 0`. [12](#0-11) 

## Impact Explanation

An unprivileged liquidator calls `liquidate(market, collateralIndex, seizedAssets=N, repaidUnits=0, borrower, ...)` when the collateral oracle returns `0`. The call succeeds: `N` collateral tokens are transferred to the liquidator's `receiver` at zero cost. The borrower's entire debt is simultaneously written off as bad debt, socializing losses to lenders. The liquidator extracts real collateral value — whatever the collateral is worth on the open market — while paying nothing. This constitutes direct theft of assets from the borrower and indirect loss to lenders via bad debt socialization.

## Likelihood Explanation

Preconditions: (1) a market collateral oracle returns `0` — explicitly modeled as a realistic scenario in the protocol's own Certora specs (`NoDivisionByZero.spec` carves it out as a known exception); (2) the borrower has debt and is liquidatable — trivially satisfied when the oracle returns `0` since `maxDebt = 0`. The attack is permissionless, requires no special role or privileged access, is repeatable for any position in any market whose collateral oracle returns `0`, and can be executed atomically in a single transaction.

## Recommendation

Add a guard in the `seizedAssets > 0` branch that reverts when `liquidatedCollatPrice == 0`, mirroring the natural revert that already protects the `repaidUnits > 0` branch:

```solidity
if (seizedAssets > 0) {
    require(liquidatedCollatPrice > 0, ZeroOraclePrice());
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
}
```

Additionally, add a corresponding Certora rule in `Reverts.spec` analogous to `oracleZeroCausesLiquidateWithRepaidRevert` for the `seizedAssets > 0` path, and correct the README claim to accurately reflect the actual protection scope.

## Proof of Concept

Minimal Foundry test:

1. Deploy a market with a mock collateral oracle that returns `0`.
2. Create a borrower position with nonzero debt and nonzero collateral at `collateralIndex`.
3. Call `liquidate(market, collateralIndex, seizedAssets=N, repaidUnits=0, borrower, false, attacker, address(0), "")` from an unprivileged attacker address.
4. Assert: attacker's collateral token balance increased by `N`; attacker's loan token balance is unchanged (paid 0); borrower's debt is 0 (written off as bad debt); borrower's collateral at `collateralIndex` decreased by `N`.
