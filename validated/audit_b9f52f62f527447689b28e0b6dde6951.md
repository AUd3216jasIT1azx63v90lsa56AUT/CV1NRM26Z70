### Title
No User-Controlled Slippage Protection in `liquidate` Allows Oracle Price Changes to Drain Liquidator Profit — (`File: src/Midnight.sol`)

### Summary

The `liquidate` function in `src/Midnight.sol` computes either `seizedAssets` or `repaidUnits` at execution time using a live oracle price. There is no caller-supplied minimum collateral output (`minSeizedAssets`) or maximum debt input (`maxRepaidUnits`) guard. An oracle price update mined before the liquidator's transaction executes can silently shift the computed amount against the liquidator, making the liquidation unprofitable or causing a flash-loan-funded liquidation to revert entirely.

### Finding Description

`liquidate` enforces that at most one of `seizedAssets` / `repaidUnits` is non-zero on entry, then derives the other from the live oracle price:

**Path A — caller supplies `repaidUnits`, protocol computes `seizedAssets`:** [1](#0-0) 

```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
} else {
    seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
}
```

The oracle price used in both branches is fetched unconditionally during the collateral loop: [2](#0-1) 

```solidity
uint256 price = IOracle(_collateralParam.oracle).price();
if (i == collateralIndex) liquidatedCollatPrice = price;
```

There is no check after the computation that `seizedAssets >= someCallerMinimum` (Path A) or `repaidUnits <= someCallerMaximum` (Path B). The function signature confirms no such parameter exists: [3](#0-2) 

**Exploit flow (Path A — most impactful):**

1. Alice observes an unhealthy borrower and simulates the call: repay 1 000 units → receive 500 collateral tokens at oracle price P.
2. Alice constructs a flash-loan-funded liquidation: borrow 1 000 loan tokens, call `liquidate(repaidUnits=1000)`, sell seized collateral to repay the flash loan.
3. Before Alice's tx is mined, a legitimate oracle update (or MEV sandwich) pushes the oracle price to 2P.
4. At execution: `seizedAssets = 1000 * lif / WAD * ORACLE_PRICE_SCALE / (2P) = 250` — half the expected collateral.
5. Alice's callback tries to repay the flash loan with 250 tokens worth of collateral but needs 500; the callback reverts, or Alice absorbs a direct loss.

### Impact Explanation

- **Direct financial loss**: A liquidator who pre-committed to a swap or flash loan to fund the repayment receives fewer collateral assets than required to break even, resulting in a net loss of loan tokens.
- **Liquidation DoS**: Flash-loan-funded liquidators (the dominant pattern in DeFi) will have their callbacks revert when the seized collateral falls below the flash-loan repayment threshold, preventing the liquidation from completing and leaving unhealthy positions open.
- **Severity**: High — direct loss of liquidator funds and potential protocol-level liveness failure (unhealthy positions cannot be liquidated profitably).

### Likelihood Explanation

- Oracle prices update on every heartbeat (typically every block or every few minutes). Any latency between tx submission and inclusion is sufficient to trigger the discrepancy.
- MEV bots routinely sandwich liquidation transactions; front-running with an oracle-price-moving transaction (e.g., a large trade on the underlying DEX that the oracle tracks) is a well-known attack pattern.
- No privileged access is required: the attacker only needs to submit a transaction that causes the oracle to report a different price before the victim's liquidation lands.
- Likelihood: **Medium-High** — natural price drift makes this a constant background risk; active MEV sandwiching makes it an on-demand exploit.

### Recommendation

Add two optional caller-controlled guard parameters to `liquidate`:

```solidity
function liquidate(
    Market calldata market,
    uint256 collateralIndex,
    uint256 seizedAssets,
    uint256 repaidUnits,
    uint256 minSeizedAssets,   // NEW: revert if computed seizedAssets < this
    uint256 maxRepaidUnits,    // NEW: revert if computed repaidUnits > this
    address borrower,
    ...
)
```

After the oracle-price-dependent computation, add:

```solidity
if (repaidUnits > 0 && minSeizedAssets > 0)
    require(seizedAssets >= minSeizedAssets, SlippageExceeded());
if (seizedAssets > 0 && maxRepaidUnits > 0)
    require(repaidUnits <= maxRepaidUnits, SlippageExceeded());
```

Passing `0` for both guards preserves backward compatibility. This mirrors the `amountOutMin` / `amountInMax` pattern used by Uniswap and Morpho Blue.

### Proof of Concept

**Setup**: Market with one collateral, oracle initially reporting `price = 2e18` (scaled), `lif = 1.05e18`, `ORACLE_PRICE_SCALE = 1e18`.

**Step 1 — Simulate off-chain** (Alice's expectation):
```
seizedAssets = repaidUnits * lif / WAD * ORACLE_PRICE_SCALE / price
             = 1000 * 1.05e18 / 1e18 * 1e18 / 2e18
             = 525 collateral tokens
```

**Step 2 — Oracle update** (mined before Alice's tx):
Oracle price moves to `4e18`.

**Step 3 — Alice's tx executes**:
```solidity
seizedAssets = 1000 * 1.05e18 / 1e18 * 1e18 / 4e18 = 262 collateral tokens
```

Alice receives 262 instead of 525 — a 50% shortfall. If Alice's `onLiquidate` callback expected ≥ 525 tokens to repay a flash loan, the callback reverts and the liquidation fails entirely.

**No privileged access required.** The oracle price change can be caused by any large trade on the underlying price feed source, or simply by natural market movement between block inclusion. [4](#0-3) [5](#0-4)
