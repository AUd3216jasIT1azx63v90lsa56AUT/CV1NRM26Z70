Audit Report

## Title
`maxAssets` cap bypass via integer truncation in assets-mode buy offers - (File: `src/Midnight.sol`)

## Summary

When a buy offer is created with `maxAssets > 0` and any `tick < MAX_TICK`, calling `take` with `units` small enough that `units * buyerPrice < WAD` causes `buyerAssets` to truncate to zero via `mulDivDown`. Because `consumed[maker][group]` is incremented by `buyerAssets` (zero), the `ConsumedAssets` cap check `require(newConsumed <= offer.maxAssets)` is trivially satisfied on every call, and the offer can be filled an unlimited number of times. The maker's unit position grows without bound while the protocol's core invariant — that offers cannot be overfilled — is violated.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369 and `src/libraries/UtilsLib.sol` line 30:**

```solidity
// UtilsLib.sol:30
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;   // plain integer division, truncates toward zero
}

// Midnight.sol:363
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // (units * buyerPrice) / 1e18
    : units.mulDivUp(buyerPrice, WAD);

// Midnight.sol:367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

`tickToPrice` rounds all prices to multiples of `PRICE_ROUNDING_STEP = 1e12` (`src/libraries/TickLib.sol` lines 6–8, 44–52). For any tick below `MAX_TICK = 5820`, `tickToPrice(tick) < WAD`. The minimum representable price is `1e12`. For `units = 1` and `buyerPrice = 1e12`: `(1 * 1e12) / 1e18 = 0`. More generally, truncation to zero occurs whenever `units * buyerPrice < 1e18`, i.e., `units < WAD / buyerPrice`. At the minimum price this threshold is `units < 1_000_000`.

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N` (any positive value) and any `tick < MAX_TICK`.
2. Taker calls `take(offer, ..., units=1, ...)` repeatedly (or via `multicall` in one transaction).
3. Each iteration: `buyerAssets = 0`; `consumed[maker][group] += 0`; `require(0 <= N)` passes unconditionally.
4. Position accounting at lines 408–414 still executes with `units = 1`: maker's credit increases by 1 (or debt decreases by 1), taker's debt increases by 1.
5. Token transfers at lines 455–456 are `safeTransferFrom(..., 0)` — succeed silently.
6. Only gate remaining is the seller health check at line 476; taker must maintain sufficient collateral, which is self-imposed.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — `newConsumed` is always 0, so this never fires.
- No `require(units > 0)` guard exists.
- No `require(buyerAssets > 0)` guard exists.
- The Certora spec `takeConsumedBoundedByMax` (`certora/specs/Consume.spec` line 62) only asserts `consumed <= maxAssets`, which is vacuously true when consumed stays at 0. It does not assert a minimum increment.
- The `takeConsumedDelta` rule (`Consume.spec` lines 67–74) explicitly scopes itself to units mode (`require offer.maxAssets == 0`) and never verifies the assets-mode delta.
- `UtilsLib.mulDivDown` is summarized as `NONDET` in the Certora spec (`Consume.spec` line 11), so the prover never reasons about truncation-to-zero.

## Impact Explanation

The `maxAssets` cap — the protocol's primary mechanism for limiting how many times an offer can be filled in assets mode — is completely bypassed. The maker's unit position (credit or debt reduction) grows without bound while `consumed` stays at zero. This violates the core protocol invariant that "offers cannot be replayed, overfilled, or reused." The maker accumulates a position far exceeding their intended limit, and the taker accumulates unbounded debt without receiving any loan tokens per fill, breaking the expected asset-to-unit correspondence. Any downstream logic (integrations, risk management, fee accounting) that relies on `maxAssets` being enforced is also broken.

## Likelihood Explanation

Preconditions are minimal and fully attacker-controlled: any buy offer with `maxAssets > 0` at any tick except `MAX_TICK` (which corresponds to price = WAD = par, an atypical edge case). The vast majority of realistic lending offers are below par. The taker needs only enough collateral to remain healthy after accumulating debt — a self-imposed cost. No privileged access is required. The attack is executable by any external address in a single transaction via `multicall`.

## Recommendation

Add a minimum-increment guard in the assets-mode branch of `take`. The simplest fix is to require that the computed asset amount is nonzero before updating `consumed`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0, ZeroAssetsDelta());   // prevent truncation bypass
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, increment `consumed` by `units` in assets mode as well (tracking unit consumption regardless of mode), or add a `require(units > 0)` guard and enforce a minimum tick/price such that `1 * minPrice >= WAD`. The Certora spec should also be updated to assert `consumed(offer.maker, offer.group) > consumedBefore` when `units > 0` and `offer.maxAssets > 0`, and `mulDivDown` should not be summarized as `NONDET` in specs that reason about consumed-mapping increments.

## Proof of Concept

**Minimal manual steps:**
1. Deploy with `settlementFee = 0` for simplicity.
2. Maker calls `postOffer` (or equivalent) with `buy = true`, `tick = 0` (price ≈ `1e12`), `maxAssets = 1e18`, `group = bytes32(0)`.
3. Taker (with sufficient collateral) calls `take(offer, ..., units=1, ...)` 1,000,001 times (or via `multicall`).
4. Assert: `consumed[maker][group] == 0` (never incremented), `position[id][maker].credit >= 1_000_001` (far exceeds `maxAssets / tickToPrice(0) ≈ 1e6`).
5. The `require(newConsumed <= offer.maxAssets)` check never reverts despite the offer being filled 1,000,001× its intended limit.

**Fuzz test plan:** Fuzz `(tick, units, maxAssets)` with `tick < MAX_TICK`, `units < WAD / tickToPrice(tick)`, `maxAssets > 0`; assert that after any successful `take`, `consumed[maker][group]` strictly increases by at least 1.
