### Title
ZKsync-Incompatible `create2` with Runtime-Constructed Bytecode in `IdLib.storeInCode` Breaks All Market Creation — (File: src/libraries/IdLib.sol)

---

### Summary

`IdLib.storeInCode()` deploys market configuration data as contract bytecode using `create2` with a bytecode buffer assembled entirely at runtime. On ZKsync Era, `create2` requires the compiler to know the deployed bytecode at compile time; runtime-assembled bytecode is not supported. Since `storeInCode` is called inside `touchMarket`, which is the entry point for every core protocol action (`take`, `withdraw`, `repay`, `supplyCollateral`, `withdrawCollateral`, `liquidate`), this makes the entire Morpho Midnight protocol non-functional on ZKsync.

---

### Finding Description

**Root cause — `IdLib.storeInCode`:** [1](#0-0) 

```solidity
function storeInCode(Market memory market, uint256 chainId) internal returns (address create2Address) {
    bytes memory creationCode = abi.encodePacked(SSTORE2_PREFIX, abi.encode(market));
    assembly ("memory-safe") {
        create2Address := create2(0, add(creationCode, 0x20), mload(creationCode), chainId)
    }
    require(create2Address != address(0), SStore2DeploymentFailed());
}
```

`creationCode` is built at runtime from `abi.encodePacked(SSTORE2_PREFIX, abi.encode(market))`. The ZKsync compiler has no visibility into this bytecode at compile time. Per ZKsync docs, `create`/`create2` on ZKsync Era work by passing the **hash** of the bytecode to the `ContractDeployer` system contract; the compiler must resolve `datasize`/`dataoffset` at compile time. A runtime-assembled buffer passed directly to `create2` in assembly bypasses this mechanism entirely, causing the deployment to fail or produce an incorrect result.

**Call path — `touchMarket` → `storeInCode`:** [2](#0-1) 

`IdLib.storeInCode(market, INITIAL_CHAIN_ID)` is called unconditionally the first time any market is touched: [3](#0-2) 

`touchMarket` is invoked by every state-changing function: `take`, `withdraw`, `repay`, `supplyCollateral`, `withdrawCollateral`, and `liquidate`.

**Downstream breakage — `toMarket`:** [4](#0-3) 

`toMarket` reads market configuration back from the deployed contract's code. If `storeInCode` fails, `toMarket` returns garbage or reverts, corrupting market lookups.

**Market ID derivation depends on the same bytecode hash:** [5](#0-4) 

`toId` computes the market ID as the CREATE2 address using the same `creationCode` hash. If the actual deployment address differs from what ZKsync's `ContractDeployer` produces, market IDs computed off-chain will not match on-chain state.

---

### Impact Explanation

On ZKsync, `storeInCode` will either revert (triggering `SStore2DeploymentFailed()`) or deploy to an unexpected address. Since `touchMarket` calls `storeInCode` on every new market creation, **no market can ever be created on ZKsync**. Without market creation, `take`, `withdraw`, `repay`, `supplyCollateral`, `withdrawCollateral`, and `liquidate` all revert. The protocol is completely non-functional. Any ETH sent with a `take` call (e.g., via `msg.value` forwarding) would be lost.

---

### Likelihood Explanation

ZKsync is a widely-used EVM-compatible L2. The protocol is described as targeting the EVM broadly. Any deployment attempt on ZKsync will immediately fail at the first `touchMarket` call. No attacker action is required — the failure is deterministic and triggered by normal user interaction.

---

### Recommendation

Replace the runtime-assembled `create2` call with a pattern the ZKsync compiler can resolve at compile time. The ZKsync-recommended approach is to use `type(Contract).creationCode` so the compiler can embed the bytecode hash in the factory deps field of the EIP-712 transaction. For the SSTORE2 use case specifically, consider using a ZKsync-compatible SSTORE2 library that uses `CODECOPY`-based storage reads rather than deploying arbitrary runtime-assembled bytecode, or store market data in a mapping instead of deploying it as a contract.

---

### Proof of Concept

1. Deploy `Midnight.sol` on a ZKsync Era fork.
2. Call `touchMarket(market)` with any valid `Market` struct.
3. Internally, `IdLib.storeInCode(market, INITIAL_CHAIN_ID)` executes `create2` with a runtime-assembled `creationCode` buffer.
4. ZKsync's `ContractDeployer` cannot resolve the bytecode hash (it was never registered in `factoryDeps`).
5. `create2` returns `address(0)`.
6. `require(create2Address != address(0), SStore2DeploymentFailed())` reverts.
7. `touchMarket` reverts, and with it every call to `take`, `withdraw`, `repay`, `supplyCollateral`, `withdrawCollateral`, and `liquidate`.
