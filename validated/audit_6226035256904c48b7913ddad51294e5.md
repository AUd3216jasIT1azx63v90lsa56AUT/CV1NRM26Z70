Audit Report

## Title
Authorized agent can grind victim's nonce in `EcrecoverAuthorizer` to invalidate all pending pre-signed authorizations - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts signatures from any address that `Midnight.isAuthorized` recognises as an agent of the authorizer, not only the authorizer's own key. An attacker who holds on-chain authorization from a victim can submit an unbounded sequence of no-op re-authorizations of themselves, each consuming one nonce slot at gas cost only. Every pre-signed authorization the victim distributed offline with a nonce below the attacker's final count becomes permanently unreplayable.

## Finding Description
**Exact code path** — `src/periphery/EcrecoverAuthorizer.sol` lines 24–48:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // line 26
// ... digest construction and ecrecover ...
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer), // lines 33-36
    Unauthorized()
);
IMidnight(MIDNIGHT)
    .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer); // lines 46-47
```

**Root cause.** The nonce slot is keyed on `authorization.authorizer` (the victim), but the authorization check accepts any `signer` for whom `IMidnight.isAuthorized(victim, signer)` returns `true`. There is no requirement that the signer equal the authorizer. Because the nonce increment on line 26 is committed whenever the full call succeeds — including no-op writes — any authorized agent can advance the victim's nonce at will.

**Exploit flow (N iterations).**

Preconditions:
1. `isAuthorized[victim][EcrecoverAuthorizer] == true` (required for the contract to function at all).
2. `isAuthorized[victim][attacker] == true` (victim has granted the attacker on-chain authorization for some purpose).

For `i = 0 … N-1`:
1. Attacker constructs `Authorization(authorizer=victim, authorized=attacker, isAuthorized=true, nonce=i, deadline=future)`.
2. Attacker signs the EIP-712 digest with their own private key → `signer = attacker`.
3. Attacker calls `EcrecoverAuthorizer.setIsAuthorized(auth, sig)`.
4. Line 25: deadline check passes.
5. Line 26: `i == nonce[victim]` → passes; `nonce[victim]` becomes `i+1`.
6. Lines 33–36: `attacker != victim`; `isAuthorized[victim][attacker] == true` → passes.
7. Line 47: `Midnight.setIsAuthorized(attacker, true, victim)` → no-op write.

After N calls: `nonce[victim] == N`. Every pre-signed authorization the victim distributed with `nonce < N` reverts with `InvalidNonce()`.

**Why existing checks fail.** The `Unauthorized` guard is satisfied by the attacker's own valid ECDSA signature because the delegation branch is true. The nonce check enforces sequential ordering only; it does not restrict which party may consume nonce slots. No check requires `signer == authorization.authorizer`.

## Impact Explanation
All pending pre-signed authorizations the victim has distributed offline (for `take`, `repay`, `withdraw`, `withdrawCollateral`, `liquidate`, `setConsumed`, or any other action gated on `isAuthorized`) become permanently invalid. Counterparties holding those signatures cannot execute them; the victim must re-sign and redistribute every authorization. The attacker can immediately repeat the attack after any re-signing, creating a sustained, low-cost denial-of-service against every signature-based action in the protocol. This constitutes permanent corruption of user state and severe service degradation under realistic attacker input.

## Likelihood Explanation
Both preconditions are normal operational states for any user relying on the signature-based authorization flow: authorizing `EcrecoverAuthorizer` is required to use it, and authorizing other agents (e.g., smart contracts, keepers, counterparties) is the protocol's intended delegation model. The attacker requires no capital, no special role, and no privileged access beyond the on-chain authorization the victim already granted. The attack costs only gas and is repeatable indefinitely as long as the attacker's authorization is not revoked. The attacker can front-run any revocation attempt by submitting additional nonce-consuming transactions.

## Recommendation
Restrict the signer to the authorizer's own key only — remove the `isAuthorized` delegation branch from `EcrecoverAuthorizer.setIsAuthorized`:

```solidity
require(signer == authorization.authorizer, Unauthorized());
```

If delegation is intentionally desired, key the nonce on the `(authorizer, signer)` pair rather than on `authorizer` alone, so that an agent can only consume nonce slots in their own sub-namespace and cannot affect the authorizer's nonce:

```solidity
mapping(address authorizer => mapping(address signer => uint256)) public nonce;
// ...
require(authorization.nonce == nonce[authorization.authorizer][signer]++, InvalidNonce());
```

Either change eliminates the ability of an authorized agent to grind the authorizer's nonce.

## Proof of Concept
```
1. Deploy Midnight and EcrecoverAuthorizer.
2. victim calls Midnight.setIsAuthorized(EcrecoverAuthorizer, true, victim).
3. victim calls Midnight.setIsAuthorized(attacker, true, victim).
4. victim pre-signs Authorization(authorizer=victim, authorized=thirdParty, isAuthorized=true, nonce=0, deadline=future) and distributes it offline.
5. attacker constructs Authorization(authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=future), signs with attacker key, calls EcrecoverAuthorizer.setIsAuthorized.
   → nonce[victim] becomes 1.
6. thirdParty attempts to submit victim's pre-signed authorization (nonce=0).
   → reverts with InvalidNonce().
7. Repeat step 5 with nonce=1,2,… to invalidate any future pre-signed authorizations.
```
A Foundry fork test can confirm this by asserting that `nonce[victim]` advances after each attacker call and that the victim's pre-signed transaction reverts.
