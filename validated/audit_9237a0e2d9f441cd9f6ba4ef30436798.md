### Title
Block Timestamp Manipulation Inflates Post-Maturity Liquidation Incentive Factor — (File: src/Midnight.sol)

### Summary
`Midnight.sol` computes the post-maturity Liquidation Incentive Factor (LIF) using `block.timestamp` over a ramp window of `TIME_TO_MAX_LIF = 15 minutes`. Because miners/validators can shift `block.timestamp` by up to ~15 seconds, they can advance their position on the LIF ramp by ~1.67% of its full range, seizing more collateral than they are entitled to for a given debt repayment. The attack window is narrow but the financial impact per liquidation can be significant on large positions.

### Finding Description

In `liquidate()`, the post-maturity LIF is computed as:

```solidity
// src/Midnight.sol line 645-647
uint256 lif = postMaturityMode
    ? UtilsLib.min(_maxLif, WAD + (_maxLif - WAD) * (block.timestamp - market.maturity) / TIME_TO_MAX_LIF)
    : _maxLif;
``` [1](#0-0) 

`TIME_TO_MAX_LIF` is defined as exactly **15 minutes (900 seconds)**:

```solidity
// src/libraries/ConstantsLib.sol line 19
uint256 constant TIME_TO_MAX_LIF = 15 minutes;
``` [2](#0-1) 

The LIF ramp is linear: at `t = maturity` it equals `WAD` (1.0), and at `t = maturity + 900s` it equals `_maxLif`. A miner who includes a liquidation transaction can set `block.timestamp` up to ~15 seconds ahead of real time. This shifts the numerator `(block.timestamp - market.maturity)` by up to 15 out of 900 seconds — a **~1.67% jump** along the full ramp.

The `seizedAssets` formula (when the liquidator specifies `repaidUnits`) is:

```solidity
// src/Midnight.sol line 652
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
``` [3](#0-2) 

A higher `lif` directly increases `seizedAssets` for the same `repaidUnits`. The borrower loses more collateral than the protocol intends at that moment in time.

Additionally, the maturity boundary check at line 622 also uses `block.timestamp`:

```solidity
// src/Midnight.sol line 620-624
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
``` [4](#0-3) 

A miner can set `block.timestamp` to just past `market.maturity` to unlock post-maturity mode up to ~15 seconds early, enabling liquidation of borrowers who have not yet had a chance to repay, while simultaneously the check at line 391 blocks new debt:

```solidity
// src/Midnight.sol line 391
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [5](#0-4) 

### Impact Explanation

A miner-liquidator manipulating `block.timestamp` by +15 seconds on a position with `_maxLif ≈ 1.0178e18` (e.g., LLTV = 0.965, cursor = 0.5) gains:

```
extra_lif = (_maxLif - WAD) * 15 / 900
           = 0.0178e18 * 15 / 900
           ≈ 2.97e14  (~0.03% of WAD)
```

On a $10M collateral position, this translates to approximately **$3,000 in extra collateral seized per liquidation**, directly at the borrower's expense. The borrower receives less collateral back than the protocol intends. For the highest-LLTV markets (LLTV = 1, `maxLif = WAD`, no incentive), this is zero. For lower-LLTV markets the delta is larger. [6](#0-5) 

### Likelihood Explanation

- **Attacker profile**: Must be a block proposer (validator) or collude with one via MEV infrastructure (e.g., private mempool / builder). This is realistic on Ethereum post-merge given the prevalence of MEV-boost and builder APIs.
- **Preconditions**: A market must be in post-maturity mode (or within ~15 seconds of maturity), and a liquidatable position must exist. Both are routine protocol states.
- **No privileged keys required**: The attacker only needs to propose a block and include their own `liquidate()` call.
- **Profitability**: The gain scales linearly with position size. On large positions (common in DeFi), the gain easily exceeds block proposal costs.

### Recommendation

Replace the `block.timestamp`-based LIF ramp with a `block.number`-based equivalent, approximating elapsed time using expected block intervals (e.g., 12 seconds/block on Ethereum mainnet):

```solidity
uint256 constant BLOCKS_TO_MAX_LIF = TIME_TO_MAX_LIF / 12; // 75 blocks

uint256 lif = postMaturityMode
    ? UtilsLib.min(_maxLif, WAD + (_maxLif - WAD) * (block.number - maturityBlock) / BLOCKS_TO_MAX_LIF)
    : _maxLif;
```

This requires storing `maturityBlock` at market creation. Alternatively, increase `TIME_TO_MAX_LIF` significantly (e.g., to several hours or days) so that a 15-second manipulation becomes negligible relative to the ramp window. A 15-second shift over a 1-hour ramp is only ~0.004%, making the attack economically unviable.

### Proof of Concept

1. A market with `maturity = T` and a large borrower position (e.g., $10M collateral, LLTV = 0.86) exists.
2. At real time `T + 1 second`, the attacker (a validator) proposes a block with `block.timestamp = T + 16 seconds` (15 seconds ahead).
3. The attacker calls `liquidate(..., postMaturityMode=true, repaidUnits=X, ...)`.
4. The contract computes:
   - Honest LIF: `WAD + (_maxLif - WAD) * 1 / 900`
   - Manipulated LIF: `WAD + (_maxLif - WAD) * 16 / 900`
5. `seizedAssets` is proportionally larger under the manipulated LIF.
6. The borrower loses ~1.67% more collateral value than the protocol intends at `T + 1 second`. [7](#0-6) [2](#0-1)
