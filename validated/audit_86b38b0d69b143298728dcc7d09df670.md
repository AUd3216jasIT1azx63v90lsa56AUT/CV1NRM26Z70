Audit Report

## Title
`maxAssets` cap bypass via integer truncation in assets-mode buy offers - (File: `src/Midnight.sol`)

## Summary
When a buy offer is created with `maxAssets > 0` and any `tick < MAX_TICK`, calling `take` with `units` small enough that `units * buyerPrice < WAD` causes `buyerAssets` to truncate to zero via `mulDivDown`. Because `consumed[maker][group]` is incremented by zero, the `require(newConsumed <= offer.maxAssets)` cap check passes unconditionally on every call, allowing the offer to be filled an unlimited number of times while the maker's unit position grows without bound.

## Finding Description

**Root cause:**

`UtilsLib.mulDivDown` performs plain integer division that truncates toward zero: [1](#0-0) 

For buy offers, `buyerAssets` is computed as: [2](#0-1) 

The `maxAssets` cap check then increments `consumed` by `buyerAssets` and enforces the cap: [3](#0-2) 

`tickToPrice` rounds all prices to multiples of `PRICE_ROUNDING_STEP = 1e12`, and for any `tick < MAX_TICK = 5820`, the resulting price is strictly less than `WAD = 1e18`: [4](#0-3) [5](#0-4) 

At the minimum price of `1e12`, truncation to zero occurs for any `units < 1_000_000` (specifically whenever `units * buyerPrice < 1e18`). With `units = 1` and `buyerPrice = 1e12`: `(1 * 1e12) / 1e18 = 0`.

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N` (any positive value) and any `tick < MAX_TICK`.
2. Taker calls `take(offer, ..., units=1, ...)` repeatedly (or via `multicall` in one transaction).
3. Each iteration: `buyerAssets = 0`; `consumed[maker][group] += 0`; `require(0 <= N)` passes unconditionally.
4. Position accounting at lines 408–414 still executes with `units = 1`: maker's credit increases by 1 unit per call.
5. Token transfers at lines 455–456 are `safeTransferFrom(..., 0)` — succeed silently. [6](#0-5) [7](#0-6) 

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — `newConsumed` is always 0, so this never fires.
- No `require(units > 0)` guard exists anywhere in `take`.
- No `require(buyerAssets > 0)` guard exists.
- The Certora spec `takeConsumedBoundedByMax` (line 62) only asserts `consumed <= maxAssets`, which is vacuously true when `consumed` stays at 0 — it does not assert a minimum increment per fill. [8](#0-7) 
- The `takeConsumedDelta` rule (lines 67–74) explicitly scopes itself to units mode (`require offer.maxAssets == 0`) and never verifies the assets-mode delta. [9](#0-8) 
- `UtilsLib.mulDivDown` is summarized as `NONDET` in the Certora spec (line 11), so the prover never reasons about truncation-to-zero. [10](#0-9) 

## Impact Explanation
The `maxAssets` cap — the protocol's primary mechanism for limiting how many times an offer can be filled in assets mode — is completely bypassed. The maker's unit position (credit or debt reduction) grows without bound while `consumed` stays at zero. This violates the core protocol invariant that offers cannot be replayed, overfilled, or reused. The maker accumulates a position far exceeding their intended limit, and the taker accumulates unbounded debt without receiving any loan tokens per fill, breaking the expected asset-to-unit correspondence. Any downstream logic relying on `maxAssets` enforcement is also broken.

## Likelihood Explanation
Preconditions are minimal and fully attacker-controlled: any buy offer with `maxAssets > 0` at any `tick < MAX_TICK` (the vast majority of realistic lending offers are below par). The taker needs only enough collateral to remain healthy — a self-imposed cost. No privileged access is required. The attack is executable by any external address in a single transaction via `multicall`. [11](#0-10) 

## Recommendation
Add a guard requiring `buyerAssets > 0` (or equivalently `units > 0`) before the `maxAssets` cap check in assets mode. For example, in `src/Midnight.sol` after line 363, add:

```solidity
if (offer.maxAssets > 0) {
    require(buyerAssets > 0, ZeroBuyerAssets());
}
```

Alternatively, add `require(units > 0)` at the top of `take`. Additionally, update the Certora spec to replace the `NONDET` summary for `mulDivDown`/`mulDivUp` with a sound over-approximation that preserves the zero-truncation property, and add a rule asserting that `consumed` increases by at least a minimum positive amount per fill in assets mode.

## Proof of Concept
**Minimal manual steps:**
1. Deploy with a market at any `tick` where `tickToPrice(tick) < 1e18` (any `tick < 5820`).
2. Maker creates a buy offer: `maxAssets = 1e18`, `tick = 0` (minimum price `~1e12`).
3. Taker calls `take(offer, ..., units=1, ...)` in a loop 1,000,000 times (or via `multicall`).
4. Observe: `consumed[maker][group]` remains `0` throughout; maker's `credit` increases by 1,000,000 units; `maxAssets` cap is never triggered.

**Fuzz/invariant test plan:**
- Invariant: for any successful `take` call in assets mode (`offer.maxAssets > 0`), `consumed[offer.maker][offer.group]` after the call must be strictly greater than before.
- Fuzz over `units` in range `[1, WAD / tickToPrice(tick) - 1]` to reliably hit the truncation-to-zero case. [12](#0-11)

/// After a successful take in units mode, the change in consumed equals the units taken.
rule takeConsumedDelta(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);
