Audit Report

## Title
`claimableSettlementFee` Over-Credited on Fee-on-Transfer Loan Token in `take` - (File: `src/Midnight.sol`)

## Summary

In `Midnight.take`, `claimableSettlementFee[offer.market.loanToken]` is incremented at line 418 by the full nominal spread `buyerAssets - sellerAssets` before the token transfer executes. When the loan token charges a transfer fee, the contract receives only `(buyerAssets - sellerAssets) * (1 - feeRate)` at line 455, but the accounting records the full nominal amount. Because `claimSettlementFee` performs no balance guard, the fee claimer can transfer more tokens than were ever received as fees, with the shortfall drawn from lenders' `withdrawable` balances, causing direct protocol insolvency.

## Finding Description

**Root cause — accounting before transfer, no received-amount check:**

At `src/Midnight.sol` line 418, the settlement fee accounting is updated unconditionally with the nominal spread:

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;  // line 418
``` [1](#0-0) 

The actual token transfer to the contract occurs later at line 455:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` (lines 24–34) only checks for revert and a false return value. It performs no balance-before/balance-after check. [3](#0-2) 

With a fee-on-transfer token charging rate `f`, the contract receives only `(buyerAssets - sellerAssets) * (1 - f)`, while `claimableSettlementFee` was credited with the full `buyerAssets - sellerAssets`. The discrepancy per `take` call is `(buyerAssets - sellerAssets) * f`.

**`claimSettlementFee` has no balance guard:**

```solidity
function claimSettlementFee(address token, uint256 amount, address receiver) external {
    require(msg.sender == feeClaimer, OnlyFeeClaimer());
    claimableSettlementFee[token] -= amount;
    SafeTransferLib.safeTransfer(token, receiver, amount);  // line 309
}
``` [4](#0-3) 

The function transfers `amount` from the contract's unified token balance, which also covers lenders' `withdrawable` amounts. When the fee claimer calls this for the full (inflated) `claimableSettlementFee`, the shortfall is taken from lender funds.

**Why existing checks fail:**

- `SafeTransferLib` does not measure received amounts. [3](#0-2) 
- `claimSettlementFee` has no invariant check comparing `claimableSettlementFee[token]` against actual token balance minus `withdrawable`. [4](#0-3) 
- The Certora rule `takeIncreasesClaimableSettlementFee` (line 75 of `certora/specs/WithdrawableMonotonicity.spec`) asserts only that `claimableSettlementFee` increases by `buyerAssets - sellerAssets`, not that the actual token balance increases by the same amount — it cannot detect the discrepancy introduced by a fee-on-transfer token. [5](#0-4) 

**Exploit path:**

1. Attacker calls `touchMarket` (public, no access control, lines 755–791) with a fee-on-transfer token as `loanToken`. [6](#0-5) 
2. Attacker (or any taker) calls `take` on a valid offer in that market.
3. Line 418 credits `claimableSettlementFee` with the full nominal spread. [1](#0-0) 
4. Line 455 transfers only `(1 - f)` of that spread to the contract. [2](#0-1) 
5. Repeated `take` calls accumulate the discrepancy.
6. When `feeClaimer` calls `claimSettlementFee` for the full `claimableSettlementFee` amount, `safeTransfer` draws the shortfall from lenders' `withdrawable` balances. [4](#0-3) 
7. Lenders' subsequent `withdraw` calls revert or receive less than their credit entitles them to.

## Impact Explanation

Direct protocol insolvency: lenders lose funds proportional to `(buyerAssets - sellerAssets) * feeRate` per `take` call, compounding across all takes in fee-on-transfer markets. The unified token balance invariant — that the contract's ERC-20 balance must cover both `claimableSettlementFee` and all lender `withdrawable` amounts — is broken. This matches the highest-priority bug class "direct loss of user funds / protocol insolvency."

## Likelihood Explanation

Market creation is permissionless (`touchMarket` is `public`). [7](#0-6) 
`SECURITY.md` contains no exclusion for fee-on-transfer tokens. [8](#0-7) 
Fee-on-transfer tokens are a well-known token class. No privileged access, oracle manipulation, or special permissions are required — any unprivileged market creator and taker can trigger this. The effect is repeatable and accumulates with every `take` in the affected market.

## Recommendation

Measure the actual received amount using a balance-before/balance-after pattern in `take`:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 received = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += received;
```

Move the `claimableSettlementFee` increment to after the transfer and credit only `received` rather than the nominal `buyerAssets - sellerAssets`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens, and add a guard in `touchMarket` or `take` to reject such tokens.

## Proof of Concept

1. Deploy a standard ERC-20 with a 1% transfer fee as `loanToken`.
2. Call `touchMarket` with this token — succeeds with no access control.
3. Create a buy offer and call `take` with `units` such that `buyerAssets - sellerAssets = 1000`.
4. Observe: `claimableSettlementFee[loanToken]` increases by 1000, but contract balance increases by only 990.
5. Repeat 100 times: `claimableSettlementFee` = 100,000; actual fees held = 99,000; discrepancy = 1,000 tokens.
6. Call `claimSettlementFee(loanToken, 100000, attacker)` — the 1,000-token shortfall is drawn from lender `withdrawable` balances.
7. Confirm: a lender calling `withdraw` for their full `withdrawable` amount reverts or receives less than entitled.
