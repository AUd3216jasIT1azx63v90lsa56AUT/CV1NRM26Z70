Audit Report

## Title
Fee-on-Transfer Collateral Token Inflates Recorded Collateral Beyond Actual Balance - (File: src/Midnight.sol)

## Summary

`supplyCollateral` credits `_position.collateral[collateralIndex]` with the full caller-supplied `assets` value at line 533 before calling `SafeTransferLib.safeTransferFrom` at line 545, with no pre/post balance measurement. When a fee-on-transfer token is used as collateral, the protocol records `assets` but receives only `assets * (1 - fee)`, creating phantom collateral. `isHealthy` uses the inflated recorded value directly, allowing the attacker to open debt positions that exceed the actual token balance held by the contract.

## Finding Description

**Exact code path â€” `src/Midnight.sol` lines 524â€“546:**

```solidity
// line 533: records full `assets` unconditionally
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);

// line 545: transfer may deliver less than `assets` if token charges a fee
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

`SafeTransferLib.safeTransferFrom` only checks that the call did not revert and that the return value (if any) is `true`; it does not compare pre/post balances.

**`touchMarket` imposes no token-type restriction.** Lines 755â€“791 show that `touchMarket` validates only LLTV tiers, sorted collateral addresses, and `maxLif`. Any ERC20 address, including a fee-on-transfer token, is accepted as a valid collateral token.

**`isHealthy` uses the recorded (inflated) collateral value directly** (`src/Midnight.sol` lines 954â€“955):

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

**Formal verification explicitly excludes this case.** `certora/specs/Solvency.spec` line 31â€“33 states: *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits."* The `CVL_transferFrom` summary transfers the exact amount specified, so the `tokenBalanceCorrect` solvency invariant is not verified for fee-on-transfer tokens.

**Exploit flow:**
1. Attacker deploys a fee-on-transfer ERC20 (e.g., 10% fee) and calls `touchMarket` to create a permissionless Midnight market with it as collateral.
2. Attacker calls `supplyCollateral(market, 0, 1000e18, attacker)`.
3. `_position.collateral[0]` is set to `1000e18`; Midnight receives only `900e18`.
4. `isHealthy` computes `maxDebt` using `1000e18`, allowing the attacker to borrow up to `1000e18 * price * lltv / WAD`.
5. Actual collateral backing is only `900e18 * price * lltv / WAD` â€” the position is immediately undercollateralized.
6. On liquidation, the protocol attempts to seize up to `1000e18` tokens but only holds `900e18`, causing a shortfall and bad debt socialized across lenders via the loss-factor mechanism.

## Impact Explanation

The solvency invariant `tokenBalances[token][currentContract] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token)` is violated. Phantom collateral allows the attacker to open debt positions exceeding the actual token balance held by the contract. On liquidation, the seized amount cannot be fully transferred out, leaving bad debt that is socialized across lenders via the loss-factor mechanism. This constitutes direct theft of lender funds and protocol insolvency for the affected market.

## Likelihood Explanation

Market creation is permissionless â€” any address can call `touchMarket` with a fee-on-transfer token as collateral. The attacker only needs to be the `onBehalf` address (or authorized for it), which is trivially satisfied by self-authorization. The attack is repeatable: each `supplyCollateral` call inflates collateral by `fee * assets`. Fee-on-transfer tokens exist in production (e.g., USDT with fee enabled, STA, PAXG). No admin action is required.

## Recommendation

Measure the actual tokens received by comparing pre- and post-transfer balances in `supplyCollateral`, and credit only the delta:

```solidity
uint256 balanceBefore = ERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = ERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Alternatively, document and enforce that only standard (non-fee-on-transfer, non-rebasing) ERC20 tokens are supported as collateral, and add a validation hook in `touchMarket` or a registry of approved collateral tokens.

## Proof of Concept

**Minimal Foundry test plan:**

1. Deploy a mock ERC20 that deducts a 10% fee on every `transferFrom` call, crediting only 90% to the recipient.
2. Deploy Midnight and call `touchMarket` with the fee token as collateral, a valid LLTV tier, and a mock oracle returning `ORACLE_PRICE_SCALE` (price = 1).
3. Approve and call `supplyCollateral(market, 0, 1000e18, attacker)`.
4. Assert `midnight.collateral(id, attacker, 0) == 1000e18` (recorded).
5. Assert `feeToken.balanceOf(address(midnight)) == 900e18` (actual).
6. Demonstrate that `isHealthy` returns `true` for a borrow of `1000e18 * lltv / WAD` units.
7. Attempt liquidation and assert that the transfer of `1000e18` tokens reverts or leaves bad debt, confirming the shortfall.
