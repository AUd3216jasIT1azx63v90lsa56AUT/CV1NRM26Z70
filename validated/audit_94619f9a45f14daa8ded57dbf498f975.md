Audit Report

## Title
`buyWithAssetsTargetAndWithdrawCollateral` permanently reverts for sell offers at high ticks with non-zero settlement fee due to uncaught `PriceGreaterThanOne` revert in `buyerAssetsToUnits` - (File: src/periphery/TakeAmountsLib.sol)

## Summary
`TakeAmountsLib.buyerAssetsToUnits` enforces `require(buyerPrice <= WAD)` at line 28. For sell offers, `buyerPrice = offerPrice + settlementFee`, which exceeds `WAD` at `tick = MAX_TICK` with any non-zero settlement fee, since `tickToPrice(MAX_TICK) ≈ WAD - 1e12`. This call is placed outside the `try/catch` block in `MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` (lines 208–214), so the revert propagates and aborts the entire transaction. `IMidnight.take()` has no equivalent guard and succeeds on the same offer.

## Finding Description

**Root cause — `TakeAmountsLib.sol` lines 26–28:**
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + settlementFee;
require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());
```
For sell offers (`offer.buy = false`), `sellerPrice = offerPrice` and `buyerPrice = offerPrice + settlementFee`. At `tick = MAX_TICK = 5820`, `tickToPrice(5820)` evaluates to approximately `WAD - 1e12 = 999_999_000_000_000_000` after `PRICE_ROUNDING_STEP` rounding. Any `settlementFee >= 1e12` (one CBP, the minimum non-zero fee) makes `buyerPrice > WAD`, triggering the revert. `MAX_SETTLEMENT_FEE_1_DAY = 14e12` far exceeds this threshold.

**Structural gap — `MidnightBundles.sol` lines 208–221:**
```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.buyerAssetsToUnits(...),   // outside try — revert propagates
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(...)
);
try IMidnight(MIDNIGHT).take(...) { ... } catch {}
```
The NatSpec at line 175 explicitly documents: *"Reverts if TakeAmountsLib or ConsumableUnitsLib reverts."* Only `take()` is guarded; the `min(...)` argument evaluation is not.

**No equivalent guard in `Midnight.sol` lines 361–363:**
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
```
No `require(buyerPrice <= WAD)` exists. Core `take()` proceeds unconditionally with `buyerPrice > WAD`.

**Exploit flow:**
1. Maker creates a sell offer (`offer.buy = false`) at `tick = MAX_TICK = 5820` in any market with `settlementFee >= 1e12` (any real market with non-zero settlement fee).
2. Taker calls `buyWithAssetsTargetAndWithdrawCollateral` with this offer in `takes[]`.
3. Loop reaches this offer; `buyerAssetsToUnits` is called with `buyerPrice = tickToPrice(MAX_TICK) + settlementFee > WAD`.
4. `require(buyerPrice <= WAD)` reverts with `PriceGreaterThanOne()`.
5. Revert escapes the `min(...)` argument evaluation, bypasses the `try/catch`, and aborts the entire transaction including all other valid offers.
6. Calling `IMidnight.take()` directly on the same offer succeeds.

**Confirming evidence in the test suite:** `testBuyBuyerAssetsTargetWithReferralFee` (line 596–598) and `testBuyBuyerAssetsTargetPartiallyConsumed` (line 1293–1295) both explicitly zero all settlement fees before calling `buyWithAssetsTargetAndWithdrawCollateral` at `MAX_TICK`, confirming the authors were aware this function fails with non-zero settlement fees at that tick.

## Impact Explanation
`buyWithAssetsTargetAndWithdrawCollateral` is permanently unusable for any sell offer where `tickToPrice(offer.tick) + settlementFee(id, ttm) > WAD`. At `tick = MAX_TICK`, the condition is triggered by any `settlementFee >= 1e12`. Any taker transaction that includes even one such offer in its `takes[]` array reverts entirely, including all other valid offers in the same call. Users are forced to call `IMidnight.take()` directly or use `buyWithUnitsTargetAndWithdrawCollateral`, defeating the purpose of the assets-target bundle. No funds are lost, but the bundler function is rendered permanently non-functional for a common, valid market configuration.

## Likelihood Explanation
The condition is deterministically reachable without any special privileges, oracle manipulation, or admin action. A maker places a sell offer at a high tick — a normal protocol action — in any market with a non-zero settlement fee (the default for all real markets). `MAX_SETTLEMENT_FEE_1_DAY = 14e12` is 14× the minimum threshold. No victim mistake is required; the taker's transaction simply reverts. The condition is stable across blocks and cannot be avoided by the taker without excluding the offer from `takes[]`.

## Recommendation
Wrap the `TakeAmountsLib.buyerAssetsToUnits` call in a `try/catch` equivalent (using a helper contract or an internal try pattern) so that a revert from `buyerAssetsToUnits` causes the offer to be skipped rather than aborting the entire transaction. Alternatively, add a pre-check before calling `buyerAssetsToUnits`: if `tickToPrice(offer.tick) + settlementFee(id, ttm) > WAD`, skip the offer (or cap `unitsToTake` using the units-based path). This aligns with the existing behavior for `take()` reverts, which are already caught and skipped.

## Proof of Concept
1. Deploy a market with any non-zero `settlementFeeCbp6` (e.g., `settlementFeeCbp6 = 1`, giving `settlementFee = 1e12`).
2. Maker creates a sell offer at `tick = MAX_TICK = 5820`.
3. Taker calls `buyWithAssetsTargetAndWithdrawCollateral` with `takes = [Take({offer: sellerOffer, units: type(uint256).max, ratifierData: ""})]`.
4. Observe revert with `PriceGreaterThanOne()`.
5. Call `IMidnight.take(sellerOffer, "", 1, taker, address(0), address(0), "")` directly — observe success.

Forge test skeleton:
```solidity
function testBuyAssetsTargetRevertsAtMaxTickWithFee() public {

}
```
