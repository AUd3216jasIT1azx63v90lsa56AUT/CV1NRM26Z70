Audit Report

## Title
Fully-consumed assets-mode buy offer with `offerPrice < WAD` allows unbounded credit/debt inflation via zero-`buyerAssets` rounding bypass - (File: src/Midnight.sol)

## Summary
When a buy offer uses `maxAssets` mode and `offerPrice < WAD`, `buyerAssets` is computed via `mulDivDown` and truncates to zero for sufficiently small `units`. The consumed-cap check increments `consumed` by zero, so a fully-consumed offer passes the cap check indefinitely. Position state (`credit`, `debt`, `totalUnits`) is updated using the nonzero `units` value while no tokens are transferred, enabling unbounded phantom credit/debt inflation at zero cost.

## Finding Description

**Root cause — dual accounting split between `buyerAssets` and `units`:**

At `src/Midnight.sol` line 363, `buyerAssets` is computed with `mulDivDown`: [1](#0-0) 

When `buyerPrice < WAD` and `units * buyerPrice < WAD`, `buyerAssets` truncates to `0`.

The consumed-cap branch at lines 367–369 increments `consumed` by `buyerAssets` (zero) and checks the result: [2](#0-1) 

If `consumed[maker][group]` is already at `maxAssets`, adding zero leaves `newConsumed == maxAssets`, and `maxAssets <= maxAssets` passes.

Position state is then updated using `units` (nonzero), not `buyerAssets`: [3](#0-2) 

Token transfers use `buyerAssets` and `sellerAssets` (both zero), so no tokens move: [4](#0-3) 

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = N`, `maxUnits = 0`, tick set such that `offerPrice < WAD`.
2. Attacker (or maker) takes normally or calls `setConsumed` until `consumed[maker][group] == maxAssets`.
3. Attacker calls `take(units=1, ...)` — `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`.
4. Consumed check: `maxAssets + 0 <= maxAssets` → passes.
5. `buyerCreditIncrease = 1`, `sellerDebtIncrease = 1` → maker's credit +1, taker's debt +1, `totalUnits` +1.
6. No tokens transferred. `consumed` unchanged.
7. Steps 3–6 repeat indefinitely.

**Why existing checks fail:**

The `ConsumedAssets` check (line 369) guards against `consumed` exceeding `maxAssets`, but the increment is `buyerAssets` (zero), not `units`. There is no check that `units == 0` when `buyerAssets == 0`.

The Certora rule `takeConsumedAtMaxUnchangedAssets` (lines 88–97 of `certora/specs/Consume.spec`) only asserts that `consumed` is unchanged when already at max — it does not assert that `credit`, `debt`, or `totalUnits` are unchanged: [5](#0-4) 

Furthermore, the Certora spec uses `NONDET` summaries for `mulDivDown` and `mulDivUp` (lines 11–12), meaning the formal verification does not model the rounding behavior that causes this issue: [6](#0-5) 

The units-mode analogue `fullyConsumedOfferRevertsOnNonTrivialTake` enforces that a fully-consumed units-mode offer only allows `units == 0` takes, but there is no equivalent for assets mode with `buyerAssets == 0`: [7](#0-6) 

The protocol comment at line 94 of `src/Midnight.sol` acknowledges this as a known behavior but does not exclude it from scope or mark it as intentionally accepted: [8](#0-7) 

## Impact Explanation
The maker's credit grows beyond their intended `maxAssets` cap with zero asset inflow. The taker accumulates debt without receiving any assets. `totalUnits` is inflated without bound, corrupting protocol-wide accounting (e.g., interest rate calculations, solvency checks). Since `consumed` never increments, the attack is repeatable without limit at zero token cost. This breaks the invariant that a fully-consumed offer admits only no-op takes, and corrupts the credit/debt accounting for both the maker and taker.

## Likelihood Explanation
All preconditions are attacker-controllable: the offer tick must satisfy `offerPrice < WAD` (any tick below the WAD-equivalent tick), `maxAssets > 0`, `maxUnits = 0`, and `consumed >= maxAssets`. The attacker can set `consumed` to `maxAssets` themselves via `setConsumed` (permissionless for their own group) or by taking normally. The call requires only that the taker is authorized and the ratifier approves — standard conditions. The attack is repeatable in every block with no cost beyond gas (zero token transfer).

## Recommendation
Add a check that `units == 0` whenever `buyerAssets == 0` (and `offer.buy == true` with `maxAssets > 0`), or alternatively enforce that the consumed increment uses `units` (converted to assets) rather than `buyerAssets` when `buyerAssets` rounds to zero. The simplest fix is to require `buyerAssets > 0 || units == 0` before updating position state in the assets-mode branch, mirroring the behavior enforced for units mode by `fullyConsumedOfferRevertsOnNonTrivialTake`.

## Proof of Concept
1. Deploy with a market where `tickToPrice(tick) < WAD`.
2. Maker creates a buy offer: `maxAssets = 100`, `maxUnits = 0`, `tick` such that `buyerPrice = WAD/2`.
3. Call `setConsumed(group, 100, maker)` to set `consumed[maker][group] = 100`.
4. Call `take(units=1, taker, makerOffer, ...)` — `buyerAssets = mulDivDown(1, WAD/2, WAD) = 0`.
5. Assert: `consumed[maker][group]` is still `100`, but `position[id][maker].credit` increased by 1, `position[id][taker].debt` increased by 1, and `totalUnits` increased by 1.
6. Repeat step 4 N times; verify credit/debt/totalUnits each increase by N with zero token transfer.

```text
/// If consumed is already at or above maxAssets before a take in assets mode, it remains unchanged.
rule takeConsumedAtMaxUnchangedAssets(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxUnits == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumedBefore >= offer.maxAssets => consumed(offer.maker, offer.group) == consumedBefore;
}
```

```text
/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
```
