Audit Report

## Title
`onLiquidate` callback enables `supplyCollateral` after bad-debt accounting is finalized, causing lenders to be over-slashed - (File: src/Midnight.sol)

## Summary
In `liquidate()`, bad-debt is fully computed from a collateral bitmap snapshot and irrevocably committed to `lossFactor` and `totalUnits` before the `onLiquidate` callback fires. A callback contract authorized for the borrower can call `supplyCollateral` during this window, injecting collateral that was invisible to the bad-debt formula. Lenders are permanently slashed for a shortfall larger than the true post-liquidation shortfall, while the borrower retains and can freely withdraw the newly supplied collateral.

## Finding Description
**Exact code path:**

1. **Line 606** — collateral bitmap is snapshotted into a local variable: `uint128 _collateralBitmap = _position.collateralBitmap;`. Any collateral added after this point is invisible to the bad-debt loop.

2. **Lines 607–618** — `badDebt` is computed by iterating only over indices present in the snapshot. The loop clears bits from the local copy, not from storage.

3. **Lines 626–641** — `badDebt` is irrevocably committed before any external call: `_position.debt -= uint128(badDebt)`, `_marketState.lossFactor` is updated, and `_marketState.totalUnits` is reduced. Lenders are slashed here.

4. **Lines 698–715** — `onLiquidate` callback fires. All bad-debt accounting is already final at this point.

5. Inside the callback, `supplyCollateral` is called for `borrower`. The authorization check at **line 527** (`onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`) is satisfied if the callback contract is the borrower (self-liquidation) or if the borrower pre-authorized the callback contract via `setIsAuthorized`.

6. `supplyCollateral` writes `_position.collateral[newIndex]` and sets the corresponding bit in `_position.collateralBitmap` (**lines 533–537**). Neither write was visible to the already-completed bad-debt loop.

7. **Line 717** — `safeTransferFrom` pulls loan tokens (0 if `repaidUnits == 0`). No re-check of collateral or bad debt occurs.

**Why existing checks do not stop it:**

- `liquidationLocked` is a transient-storage flag set only inside `take()` at **line 444** via `UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`. It is never set inside `liquidate()`. It does not block `supplyCollateral` from being called during the `onLiquidate` callback.
- There is no reentrancy guard on `liquidate()` or `supplyCollateral()`.
- The Certora `OnlyAuthorizedCanChange.spec` invariant at **line 73** explicitly acknowledges: *"Assumes no reentrancy: callbacks and token transfers are not modeled as re-entering Midnight, so re-entrant collateral changes are not covered."* This is a documented gap in the formal verification, not a design guarantee of safety.

## Impact Explanation
`badDebt` realized = `D − Σ(collateral_i · price_i / maxLif_i)` (pre-callback collateral only).
True shortfall after callback = `D − Σ(collateral_i · price_i / maxLif_i) − A · price_new / maxLif_new`.
Over-slashing per attack = `A · price_new / maxLif_new` units, permanently subtracted from `totalUnits` and added to `lossFactor`. Every lender's effective credit is permanently reduced by this excess amount when they call `updatePosition`. The attacker (borrower) retains the newly supplied collateral in their position. Because bad debt realization sets `_position.debt` to zero (in the full-bad-debt case), the borrower can immediately withdraw all collateral including the newly supplied amount, recovering the flash-loaned tokens at near-zero net cost. This constitutes direct, permanent theft of lender credit with no privileged access required.

## Likelihood Explanation
**Required preconditions:**
1. Borrower is a smart contract (or has pre-authorized the callback) — realistic for any protocol-integrated borrower or DeFi vault.
2. Position has genuine bad debt (oracle price drop) — the normal trigger for bad-debt realization.
3. Callback contract holds collateral tokens to supply — sourceable from a flash loan within the same callback, since `supplyCollateral` pulls from `msg.sender` and no repayment is required within `liquidate()`.

Self-liquidation (`borrower == liquidator == callback`) requires no third-party cooperation. The attack is repeatable on any market with bad debt and a callback-capable borrower. The attacker controls `seizedAssets = 0`, `repaidUnits = 0`, `callback`, and the collateral amount, requiring no privileged access.

## Recommendation
Move the `onLiquidate` callback invocation to **before** the bad-debt accounting block, or re-snapshot `_position.collateralBitmap` from storage after the callback returns and recompute `badDebt` using the post-callback state. Alternatively, set a transient reentrancy lock at the entry of `liquidate()` (analogous to `liquidationLocked` in `take()`) that blocks `supplyCollateral` and `withdrawCollateral` for the affected borrower for the duration of the call. The lock must cover the entire execution of `liquidate()`, not just the callback window.

## Proof of Concept
**Minimal self-liquidation attack (no third-party required):**

1. Deploy `AttackerBorrower` implementing `ILiquidateCallback`. The contract holds or can flash-loan collateral token `T`.
2. `AttackerBorrower` borrows from a market until a genuine oracle price drop creates bad debt (`debt > Σ collateral_value`).
3. `AttackerBorrower` calls `liquidate(market, collateralIndex, 0, 0, address(this), false, address(this), address(this), "")`.
4. Inside `liquidate()`, lines 606–618 compute `badDebt` from the current (pre-callback) collateral bitmap. Lines 626–641 commit the full bad debt to `lossFactor`/`totalUnits`.
5. Lines 698–715 invoke `AttackerBorrower.onLiquidate(...)`.
6. Inside `onLiquidate`, `AttackerBorrower` flash-loans `A` units of token `T` and calls `supplyCollateral(market, newIndex, A, address(this))`. Authorization passes because `msg.sender == onBehalf == address(this)`.
7. `_position.collateral[newIndex] = A` and the bitmap bit is set — invisible to the already-committed bad-debt calculation.
8. `onLiquidate` returns `CALLBACK_SUCCESS`. Line 717 transfers 0 loan tokens (repaidUnits == 0).
9. After `liquidate()` returns, `_position.debt == 0` (full bad debt was realized). `AttackerBorrower` calls `withdrawCollateral` to recover all collateral including the `A` tokens, repays the flash loan.
10. **Net result:** lenders permanently over-slashed by `A · price_T / maxLif_T` units; attacker pays only gas.
