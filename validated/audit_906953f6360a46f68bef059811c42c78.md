### Title
No Way to Invalidate Pending Signed Authorizations in `EcrecoverAuthorizer` — (`src/periphery/EcrecoverAuthorizer.sol`)

---

### Summary

`EcrecoverAuthorizer` allows users to grant or revoke Midnight authorizations via off-chain EIP-712 signatures. Once a user signs an `Authorization` struct with a long-dated `deadline`, there is no mechanism to invalidate it short of consuming the nonce in a race. A user who revokes an authorization directly on `Midnight` can have it silently re-granted by anyone holding the original signed message.

---

### Finding Description

`EcrecoverAuthorizer` maintains a sequential nonce per address:

```solidity
mapping(address => uint256) public nonce;
``` [1](#0-0) 

`setIsAuthorized` validates and consumes the nonce atomically:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
``` [2](#0-1) 

There is **no** `increaseNonce()` or equivalent function anywhere in the contract or its interface. [3](#0-2) 

**Exploit path:**

1. Alice signs `Authorization{authorizer=Alice, authorized=Attacker, isAuthorized=true, nonce=0, deadline=block.timestamp + 365 days}` and shares it (e.g., via a UI or off-chain channel).
2. Alice later decides to revoke and calls `Midnight.setIsAuthorized(Attacker, false, Alice)` directly — this updates `isAuthorized[Alice][Attacker] = false` on Midnight but **does not touch** `nonce[Alice]` in `EcrecoverAuthorizer` (still `0`).
3. The attacker calls `EcrecoverAuthorizer.setIsAuthorized(authorization, sig)` with the original signed message. Nonce `0` is still valid, deadline has not passed, so the call succeeds and sets `isAuthorized[Alice][Attacker] = true` again on Midnight. [4](#0-3) 

The `setIsAuthorized` call on Midnight itself is permissionless for the authorized caller: [5](#0-4) 

---

### Impact Explanation

An authorized account on Midnight can call every position-mutating function on behalf of the victim: `take()`, `withdrawCollateral()`, `repay()`, `supplyCollateral()`, and even `setIsAuthorized()` (granting further delegations). [6](#0-5) 

The bundler also enforces `isAuthorized` as its only access check, meaning an attacker re-granted authorization can drain collateral through the bundler's `withdrawCollateral` path: [7](#0-6) 

Concrete worst case: attacker calls `withdrawCollateral` on all of Alice's collateral positions to an attacker-controlled address — **direct theft of all collateral**.

---

### Likelihood Explanation

- **Precondition:** A signed `Authorization` with `isAuthorized=true` and a future deadline must exist. This is the normal operating mode for any UI that uses `EcrecoverAuthorizer` for gasless delegation.
- **Attacker capability:** No privileged access required. Any address that received or observed the signed message can execute the replay.
- **Trigger:** The victim revokes on-chain (a normal, expected user action). The attacker simply submits the old signed message afterward — no front-running required, the window is open until the deadline.
- **Likelihood: High** for any user who has ever used the gasless authorization flow and later attempted to revoke.

---

### Recommendation

Add a public `increaseNonce()` function to `EcrecoverAuthorizer`, mirroring the fix applied in the referenced eBTC audit:

```solidity
/// @notice Invalidates all outstanding signed authorizations for the caller.
function increaseNonce() external returns (uint256) {
    return ++nonce[msg.sender];
}
```

This allows a user to atomically burn their current nonce, rendering any previously signed `Authorization` with that nonce permanently invalid, without requiring them to race against an attacker. [8](#0-7) 

---

### Proof of Concept

```
1. Alice signs: Authorization{authorizer=Alice, authorized=Bob, isAuthorized=true, nonce=0, deadline=now+365d}
2. Bob (or anyone with the sig) calls EcrecoverAuthorizer.setIsAuthorized(auth, sig)
   → Midnight.isAuthorized[Alice][Bob] = true, nonce[Alice] = 1
3. Alice calls Midnight.setIsAuthorized(Bob, false, Alice) directly
   → Midnight.isAuthorized[Alice][Bob] = false, nonce[Alice] still = 1
   (Alice cannot reset nonce[Alice] without a new signed message)
4. Alice signs a new Authorization{nonce=1, isAuthorized=false} and submits it
   → nonce[Alice] = 2, isAuthorized[Alice][Bob] = false ✓
   BUT: if Alice had signed nonce=0 with deadline far in the future and Bob had NOT yet consumed it,
        Bob can call setIsAuthorized with nonce=0 BEFORE Alice's revocation tx lands (or after step 3
        if nonce=0 was never consumed), re-granting himself access.

Concrete no-front-run scenario (nonce never consumed):
1. Alice signs auth{nonce=0, isAuthorized=true, deadline=+365d} — never submitted on-chain
2. Alice decides she no longer wants Bob authorized; calls Midnight.setIsAuthorized(Bob,false,Alice)
   → isAuthorized[Alice][Bob]=false, but nonce[Alice]=0 still
3. Bob submits the signed auth → nonce[Alice]=1, isAuthorized[Alice][Bob]=true again
4. Bob calls Midnight.withdrawCollateral(..., Alice, Bob) → steals Alice's collateral
``` [9](#0-8)

contract EcrecoverAuthorizer is IEcrecoverAuthorizer {
    address public immutable MIDNIGHT;
    mapping(address => uint256) public nonce;

    constructor(address _midnight) {
        MIDNIGHT = _midnight;
    }

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
```
