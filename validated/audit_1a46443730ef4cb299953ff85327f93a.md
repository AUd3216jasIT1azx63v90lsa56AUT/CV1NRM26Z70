Audit Report

## Title
Double `mulDivDown` truncation in `isHealthy` produces `maxDebt` one unit below exact value, blocking valid `withdrawCollateral` - (File: `src/Midnight.sol`)

## Summary
`isHealthy` chains two independent floor-rounding calls — `collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)` — where each call independently truncates. The compounded truncation can yield a `maxDebt` that is 1 less than the mathematically exact value, causing `isHealthy` to return `false` for a position that is genuinely healthy. Any borrower whose debt equals the exact health boundary is permanently blocked from withdrawing collateral via `withdrawCollateral`, which enforces `isHealthy` after reducing collateral.

## Finding Description

**Root cause — `UtilsLib.mulDivDown` is plain integer division with no 512-bit intermediate:** [1](#0-0) 

Each call independently truncates. Chaining two such calls means the first truncation's discarded remainder propagates into the second call's input, potentially causing the second result to be 1 less than `floor(C * P * L / (S * W))`.

**Double truncation in `isHealthy`:** [2](#0-1) 

Let `S = ORACLE_PRICE_SCALE = 1e36`, `W = WAD = 1e18`. The first call computes `q1 = floor(C * P / S)`, discarding remainder `r1 ∈ [0, S-1]`. The second call computes `floor(q1 * L / W)`. Because `q1 ≤ C*P/S`, the result satisfies `computed ≤ floor(C*P*L/(S*W))`. Since `L ≤ W`, the propagated error `r1 * L / (S * W) < 1`, so the gap is at most 1.

**`withdrawCollateral` enforces `isHealthy` after reducing collateral:** [3](#0-2) 

The collateral is written before the check. If `isHealthy` returns `false`, the transaction reverts (state rolled back), but the withdrawal is permanently blocked as long as the position remains at this boundary.

**Concrete exploit path (verified arithmetic):**

Constants: `ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`, `lltv = LLTV_2 = 0.77e18` (allowed by `isLltvAllowed`). [4](#0-3) 

Parameters: `price = 1e36/77 + 1`, `collateral = 100`, `debt = 1`.

- **First `mulDivDown`:** `q1 = floor(100 * (1e36/77 + 1) / 1e36) = floor(1.2987...) = 1`
- **Second `mulDivDown`:** `q2 = floor(1 * 0.77e18 / 1e18) = floor(0.77) = 0`
- **Computed `maxDebt` = 0; `isHealthy` returns `false` (0 ≥ 1 is false)**

Exact single-operation value:
```
floor(100 * (1e36/77 + 1) * 0.77e18 / (1e36 * 1e18))
= floor((1e54 + 1e18) / 1e54)
= floor(1 + 1e-36) = 1
```

Exact `maxDebt = 1 ≥ debt = 1` → position is truly healthy, but `isHealthy` returns `false`.

**Existing checks are insufficient:** The authorization check at line 556 (`onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`) passes normally. The only revert comes from the `isHealthy` call at line 568, which is structurally broken for this boundary case. [5](#0-4) 

## Impact Explanation
A borrower whose debt equals the exact health boundary cannot withdraw collateral that would leave `computed_maxDebt < debt`, even though `exact_maxDebt ≥ debt`. The collateral is not seized — it remains locked in the contract — but the withdrawal is permanently blocked unless the oracle price rises or the borrower repays debt. This is a concrete, in-scope fund freeze / valid withdrawal blocked, matching "Permanent lock, freeze, or unrecoverable corruption of user/project state" per RESEARCHER.md.

## Likelihood Explanation
The condition is deterministic and repeatable: any borrower who borrows the maximum allowed amount and then attempts to withdraw collateral to the exact health boundary can hit this. The specific `(collateral, price)` pairs that trigger it are reachable with small token amounts (1 unit of debt). No privileged access is required — the borrower calls `withdrawCollateral` directly. The preconditions are achievable by any unprivileged user on any market using an allowed LLTV tier.

## Recommendation
Replace the chained `mulDivDown` with a single combined operation that computes `floor(collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD))` in one step, avoiding intermediate truncation:

```solidity
maxDebt += _position.collateral[i].mulDivDown(
    price * collateralParam.lltv,
    ORACLE_PRICE_SCALE * WAD
);
```

This requires verifying that `price * collateralParam.lltv` does not overflow `uint256`. Given `price` is oracle-supplied and `lltv ≤ WAD = 1e18`, the product can reach up to `~1e54` for large oracle prices, which fits in `uint256` (max ~1.16e77). Alternatively, use a 512-bit `mulDiv` (e.g., from `PRBMath` or Solady) for the full three-operand product.

## Proof of Concept
Minimal Foundry test:

```solidity

// Deploy a mock oracle returning price = 1e36/77 + 1
// Supply collateral = 100 to a borrower position
// Borrow debt = 1 (passes initial isHealthy: q1=2, q2=1 with collateral=200; or set collateral=200 initially)
// Call withdrawCollateral(market, 0, 100, borrower, receiver)
// Expected: reverts with UnhealthyBorrower()
// Verify: isHealthy with collateral=100 returns false (computed maxDebt=0 < debt=1)
// Verify exact math: floor(100*(1e36/77+1)*0.77e18/(1e36*1e18)) = 1 >= 1 (truly healthy)
```

The revert is deterministic and reproducible with the exact parameters above.
