Audit Report

## Title
Fully-consumed buy offer (`maxAssets` mode) can be re-taken with zero-asset units, mutating position state without payment - (File: src/Midnight.sol)

## Summary
When a buy offer has `maxAssets > 0` and `consumed[maker][group]` has already reached `maxAssets`, a taker can call `take()` with a small `units` value such that `units * buyerPrice < WAD`. Because `mulDivDown` performs plain integer division, `buyerAssets` rounds to zero, the consumed counter is not incremented, and the cap check passes trivially. The take then proceeds to increase `buyerPos.credit`, `sellerPos.debt`, and `totalUnits` with zero token transfer, breaking the invariant that every credit unit corresponds to an asset payment.

## Finding Description
**Exact code path in `src/Midnight.sol`:**

At line 363, `buyerAssets` is computed with rounding-down division:
```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // (units * buyerPrice) / WAD
    : units.mulDivUp(buyerPrice, WAD);
``` [1](#0-0) 

`mulDivDown` is implemented as plain integer division `(x * y) / d` in `src/libraries/UtilsLib.sol`:
```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
``` [2](#0-1) 

When `units * buyerPrice < WAD`, this returns 0. At lines 367–369, the consumed counter is incremented by `buyerAssets` (which is 0), and the cap check `newConsumed <= offer.maxAssets` is evaluated against the unchanged value:
```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [3](#0-2) 

If `consumed[maker][group]` is already equal to `maxAssets`, then `newConsumed == maxAssets` and the require passes. Execution continues to mutate `buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` at lines 408–417, and the token transfer at line 455 transfers `buyerAssets - sellerAssets = 0 - 0 = 0` tokens. [4](#0-3) 

**Root cause:** The only guard against over-filling is `require(newConsumed <= offer.maxAssets)`. When the increment is 0 due to rounding, this check is a no-op. There is no guard requiring `buyerAssets > 0` when `units > 0`, and no guard requiring `newConsumed > consumedBefore`.

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = M`, `maxUnits = 0`, and any tick below `MAX_TICK` (so `buyerPrice < WAD`).
2. Offer is legitimately filled until `consumed[maker][group] == M`.
3. Attacker calls `take(offer, ..., units=1)`. Since `1 * buyerPrice < WAD`, `buyerAssets = 0`. `consumed` stays at `M`. The cap check passes.
4. `buyerPos.credit` increases, `sellerPos.debt` increases, `totalUnits` increases — all with zero token transfer.
5. Steps 3–4 are repeatable indefinitely, including within a single transaction via multicall.

**Why existing checks fail:** The `require(newConsumed <= offer.maxAssets)` check is the sole guard. When the increment is 0, it reduces to `require(M <= M)`, which is always true. No other check in the `take()` function prevents a zero-asset take from proceeding.

A reproducing test `testBugBuyMaxAssetsBypass` already exists in `test/TakeTest.sol` and confirms the behavior. [5](#0-4) 

## Impact Explanation
An attacker can repeatedly take a fully-consumed buy offer at zero cost, accruing unbounded `credit` for the maker and unbounded `debt` for themselves (or any target seller). `totalUnits` and `pendingFee` state inflate without any corresponding asset transfer, permanently breaking the protocol invariant that every credit unit has a matching asset payment. This constitutes unauthorized state mutation and accounting corruption reachable by any unprivileged external user.

## Likelihood Explanation
All preconditions are attacker-controllable with no privileged access:
- `offer.buy == true`, `offer.maxAssets > 0`, `offer.maxUnits == 0` — standard buy offer configuration.
- `buyerPrice < WAD` — true for any tick below `MAX_TICK`; the vast majority of real offers.
- `consumed[maker][group] >= maxAssets` — reachable by normal fills or by the maker calling `setConsumed`.
- `units = 1` — sufficient when `buyerPrice < WAD`.

The attack requires no special role, no flash loan, and no oracle manipulation. It is repeatable in the same block via multicall.

## Recommendation
Add a guard that rejects a take when `units > 0` but the computed asset amount rounds to zero. For the `maxAssets` branch, require that the consumed counter strictly increases:

```solidity
if (offer.maxAssets > 0) {
    uint256 consumedBefore = consumed[offer.maker][offer.group];
    newConsumed = consumedBefore + (offer.buy ? buyerAssets : sellerAssets);
    require(newConsumed > consumedBefore || units == 0, ZeroAssetTake());
    consumed[offer.maker][offer.group] = newConsumed;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, add a top-level guard `require(buyerAssets > 0 || units == 0)` before the consumed accounting block to reject any take where the buyer pays nothing for a non-zero unit amount.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` reproduces this exactly. To verify manually:

1. Deploy with a buy offer: `maxAssets = M`, `maxUnits = 0`, tick such that `tickToPrice(tick) < WAD`.
2. Fill the offer until `consumed[maker][group] == M`.
3. Call `take(offer, units=1, ...)`.
4. Assert: `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged at `M`, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.
5. Repeat step 3 to confirm unbounded repetition.
