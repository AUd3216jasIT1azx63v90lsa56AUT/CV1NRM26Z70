Audit Report

## Title
Nonce-Race Front-Run on `EcrecoverAuthorizer.setIsAuthorized` Enables Unauthorized Withdrawal - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` is fully permissionless and binds `isAuthorized` (grant vs. revoke) into the EIP-712 signed digest, meaning a grant and a revocation for the same nonce are two distinct valid messages. An attacker holding a victim-signed grant for nonce=N can front-run the victim's mempool revocation (also nonce=N) by submitting the grant first, consuming the nonce and causing the revocation to revert with `InvalidNonce()`. The attacker remains authorized in Midnight and can immediately call `Midnight.withdraw` to drain the victim's withdrawable loan-token balance.

## Finding Description

**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` is permissionless — `msg.sender` is never checked: [1](#0-0) 

The nonce is consumed atomically on first use: [2](#0-1) 

The `Authorization` struct includes `isAuthorized` as a field in the signed digest: [3](#0-2) 

The digest is computed over the full struct including `isAuthorized`: [4](#0-3) 

The signature check only requires `ecrecover` to return `authorization.authorizer`: [5](#0-4) 

The function then calls `Midnight.setIsAuthorized` with attacker-controlled `authorized` and `isAuthorized` fields: [6](#0-5) 

**Root cause:** Because `isAuthorized` is part of the signed digest, `{victim, attacker, true, N, far}` and `{victim, attacker, false, N, ...}` are two cryptographically distinct messages that both carry a valid victim signature and both satisfy `nonce == N`. The nonce mechanism only prevents replay of the *same* message; it does not prevent two different messages from racing on the same nonce slot. Since the function is permissionless, the attacker can submit the grant at any time, including as a front-run.

**Exploit flow:**
1. Victim signs `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N, deadline=far}` off-chain for a relayer and shares the signature with the attacker (relayer), but the grant is never submitted.
2. Victim changes mind, signs `Authorization{authorizer=victim, authorized=attacker, isAuthorized=false, nonce=N}`, and broadcasts the revocation to the public mempool.
3. Attacker observes the revocation in the mempool and front-runs it by submitting the old grant with the victim's valid signature at higher gas.
4. Attacker's grant executes: `nonce[victim]` becomes N+1; `isAuthorized[victim][attacker] = true` in Midnight.
5. Victim's revocation reverts: `authorization.nonce == N` but `nonce[victim] == N+1` → `InvalidNonce()`.
6. Attacker calls `Midnight.withdraw(market, units, victim, attacker)`.

**Why existing checks fail:**
- `InvalidNonce()`: prevents replay of the *same* message, but does not prevent two *different* signed messages (grant vs. revoke) from racing on the same nonce.
- `Unauthorized()`: the attacker's grant carries a genuine victim signature (`ecrecover` returns `victim`), so this passes.
- `Expired()`: the attacker's grant carries a far deadline, so this passes.
- Direct `Midnight.setIsAuthorized` call: the victim could call this directly to revoke, but it does not consume the nonce. The attacker can still submit the signed grant afterward via `EcrecoverAuthorizer`, re-setting `isAuthorized[victim][attacker] = true`. There is no mechanism to invalidate a signed grant without consuming its nonce. [7](#0-6) 

## Impact Explanation

Once `isAuthorized[victim][attacker]` is `true`, the attacker calls `Midnight.withdraw(market, units, victim, attacker)`: [8](#0-7) 

The authorization check passes, `_position.credit` and `_marketState.withdrawable` are decremented, and loan tokens are transferred directly to the attacker: [9](#0-8) 

This constitutes direct theft of the victim's withdrawable loan-token balance. The reduction in `_marketState.withdrawable` and `totalUnits` also reduces the pool's ability to cover legitimate lender redemptions, matching the scoped insolvency impact class.

## Likelihood Explanation

**Required preconditions:**
- Victim signed a grant off-chain and shared it with the attacker (standard in relayer/meta-transaction flows — the victim authorizes a relayer, then decides to revoke before the relayer submits).
- Victim attempts revocation via `EcrecoverAuthorizer` (the only path that consumes the nonce and would invalidate the signed grant).
- Attacker monitors the public mempool (standard capability on any EVM chain with a public mempool).

**Feasibility:** High. The attacker needs only to submit a higher-gas transaction with the already-held valid signature. No special privileges, no cryptographic breaks, no oracle manipulation. The attack is repeatable for each new nonce if the victim continues using `EcrecoverAuthorizer` for revocations.

## Recommendation

1. **Add a standalone nonce-invalidation function** (e.g., `invalidateNonce(address authorizer)`) that increments `nonce[authorizer]` without requiring a signed message. This allows a victim to atomically burn nonce=N, rendering any outstanding signed grant for that nonce unsubmittable, without racing against the attacker.
2. **Alternatively**, remove `isAuthorized` from the signed digest and instead use a separate `revoke` path that does not share the nonce namespace with grants, so a revocation cannot be front-run by a grant on the same nonce.
3. **Document clearly** that once a signed grant is shared with a counterparty, the only safe revocation path is the nonce-invalidation function, not a signed revocation message.

## Proof of Concept

**Minimal Foundry test outline:**

```solidity
// 1. Deploy Midnight + EcrecoverAuthorizer
// 2. victim signs Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=far}
// 3. victim signs Authorization{authorizer=victim, authorized=attacker, isAuthorized=false, nonce=0, deadline=far}
// 4. Simulate front-run: attacker calls ecrecoverAuthorizer.setIsAuthorized(grantAuth, grantSig) first
// 5. victim's revocation call reverts with InvalidNonce()
// 6. Assert isAuthorized[victim][attacker] == true in Midnight
// 7. Attacker calls midnight.withdraw(market, victimCredit, victim, attacker)
// 8. Assert attacker received loan tokens; victim credit == 0
```

The test requires no privileged keys, no oracle manipulation, and no chain-specific assumptions beyond a public mempool. The `ecrecover` path is deterministic given the victim's private key and the struct fields. [10](#0-9)

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
