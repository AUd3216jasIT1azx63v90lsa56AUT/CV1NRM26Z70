Audit Report

## Title
Debt Created at `maturity == block.timestamp` Is Immediately Liquidatable in the Next Block via Post-Maturity Mode - (File: src/Midnight.sol)

## Summary
`touchMarket` enforces no lower bound on `market.maturity`, allowing a market to be created with `maturity = block.timestamp`. The `take` guard uses `<=` permitting debt creation at the exact maturity timestamp, while `liquidate`'s post-maturity guard uses strict `>`, creating a one-block asymmetry where a healthy borrower's position is immediately liquidatable in the next block with no opportunity to repay.

## Finding Description

**`touchMarket` (`src/Midnight.sol:758`)** enforces only an upper bound:
```solidity
require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
```
No `require(market.maturity > block.timestamp)` exists. A market with `maturity = block.timestamp` is created without error.

**`SetterRatifier.isRatified` (`src/ratifiers/SetterRatifier.sol:30-36`)** checks only Merkle proof membership and `isRootRatified[offer.maker][root]` — zero validation of `offer.market.maturity`.

**`take` (`src/Midnight.sol:391`):**
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
When `maturity = block.timestamp`, `block.timestamp <= block.timestamp` is `true`, so `sellerDebtIncrease > 0` is permitted — debt is written at the exact maturity timestamp.

**`liquidate` (`src/Midnight.sol:620-624`):**
```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```
One block later (`block.timestamp = maturity + 1`), `block.timestamp > market.maturity` is `true`, so post-maturity liquidation of a healthy borrower succeeds.

**Exploit flow:**
1. Attacker (maker) constructs an offer with `offer.market.maturity = block.timestamp`.
2. Attacker calls `setterRatifier.setIsRootRatified(maker, root, true)` — no privilege required beyond being the maker.
3. In the same block, a taker (victim borrower) calls `midnight.take(offer, ...)` with `units > 0`. The `take` check at line 391 passes (`T <= T`). The `SellerIsLiquidatable` check at line 476 passes if the borrower has sufficient collateral. Debt is written to `sellerPos.debt`.
4. In the next block, any caller invokes `midnight.liquidate(market, ..., postMaturityMode=true)`. The guard `block.timestamp > market.maturity` is satisfied. The healthy borrower's position is liquidated.

**Why existing checks fail:**
- `touchMarket`: no lower-bound maturity check.
- `SetterRatifier.isRatified`: no maturity check at all.
- `take` line 391: uses `<=`; the boundary case `maturity == block.timestamp` is not excluded.
- `liquidate` line 622: uses strict `>`; the asymmetry between `<=` in `take` and `>` in `liquidate` creates a one-block window where debt exists but is already post-maturity-liquidatable.

## Impact Explanation
A healthy borrower who takes a loan in a market where `maturity = block.timestamp` has their collateral immediately seizable in the next block via `postMaturityMode=true` liquidation, with no opportunity to repay. The borrower suffers an immediate, unavoidable collateral loss on a position that was healthy at the time of borrowing. The LIF at `block.timestamp = maturity + 1` is slightly above 1, so the liquidator receives slightly more collateral than the debt repaid.

## Likelihood Explanation
All preconditions are fully attacker-controlled: any unprivileged address can be a maker, call `setIsRootRatified` on themselves, and construct an offer with `maturity = block.timestamp`. The taker can be an unsuspecting victim who does not inspect the maturity field. The attack requires no admin action, oracle manipulation, or special privilege, and is repeatable in any block.

## Recommendation
Add a minimum maturity lower bound in `touchMarket`:
```solidity
require(market.maturity > block.timestamp, MaturityInPast());
```
Alternatively, change the `take` guard from `<=` to `<` so that debt creation is disallowed at the exact maturity timestamp:
```solidity
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
Both fixes eliminate the asymmetry; applying both is the most robust approach.

## Proof of Concept
1. Deploy `Midnight` and `SetterRatifier` on a local fork.
2. In block N: call `setterRatifier.setIsRootRatified(maker, root, true)`, then call `midnight.take(offer, ...)` where `offer.market.maturity = block.timestamp`. Assert debt is written and position is healthy.
3. Advance one block (block N+1).
