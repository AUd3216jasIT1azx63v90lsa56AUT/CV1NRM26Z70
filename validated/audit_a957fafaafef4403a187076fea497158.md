Audit Report

## Title
Bad Debt Over-Realization via `onLiquidate` Callback Supplying Collateral After Bitmap Snapshot - (File: `src/Midnight.sol`)

## Summary
The `liquidate` function snapshots `_position.collateralBitmap` into a local variable at line 606, computes and fully applies `badDebt` (modifying `_position.debt`, `_marketState.lossFactor`, and `_marketState.totalUnits`) at lines 626–641, and only then invokes the `onLiquidate` callback at lines 698–715. A borrower-controlled smart contract can use this callback window to call `supplyCollateral` on its own behalf, activating a new collateral index in storage — but since `badDebt` was already computed and applied from the pre-callback snapshot, that collateral value is never deducted from `badDebt`, causing the protocol to over-realize bad debt and over-slash lenders.

## Finding Description

**Root cause — bitmap snapshot before bad debt realization:**

At line 606, `_collateralBitmap` is copied into a local variable: [1](#0-0) 

The loop iterates only over this local copy. Any collateral index not set in the bitmap at snapshot time is invisible to the `badDebt` computation.

**Bad debt fully applied before callback:**

Lines 626–641 irreversibly reduce `_position.debt`, ratchet up `_marketState.lossFactor`, and reduce `_marketState.totalUnits` — all before any external call: [2](#0-1) 

**Callback invoked after realization:**

The `onLiquidate` callback is called at lines 698–715, after all state mutations: [3](#0-2) 

**`supplyCollateral` authorization is trivially satisfied:**

`supplyCollateral` checks `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. When the borrower is a smart contract that passes `callback = address(this)` to `liquidate`, calling `supplyCollateral(..., address(this))` inside `onLiquidate` satisfies `onBehalf == msg.sender` with no preconditions: [4](#0-3) 

`supplyCollateral` then sets the new bit in the storage bitmap: [5](#0-4) 

**No reentrancy guard exists** — confirmed by absence of any `nonReentrant` modifier or lock in `liquidate` or `supplyCollateral`.

**Exploit flow:**
1. Borrower is a smart contract with a position where all existing collateral is worth less than debt at `maxLif` prices (genuine bad debt).
2. Borrower calls `liquidate(market, 0, 0, 0, address(this), false, receiver, address(this), "")` — `seizedAssets = repaidUnits = 0`, `callback = address(this)`.
3. `liquidate` snapshots `_collateralBitmap` (line 606), computes `badDebt` from existing collateral only.
4. `badDebt` is applied: `_position.debt` reduced, `_marketState.lossFactor` increased, `_marketState.totalUnits` reduced (lines 626–641).
5. `onLiquidate` is called on the borrower contract (line 700).
6. Inside `onLiquidate`, the borrower calls `supplyCollateral(market, newIndex, largeAmount, address(this))`, depositing real collateral and setting a new bit in `_position.collateralBitmap` in storage.
7. `liquidate` returns. The new collateral is in the position, but `badDebt` was already applied without accounting for it.

**Why existing checks fail:**
- No reentrancy guard on `liquidate` or `supplyCollateral`.
- No post-callback re-computation or validation of `badDebt`.
- The local `_collateralBitmap` variable is never refreshed from storage after the callback.
- `supplyCollateral`'s authorization check is trivially satisfied when the borrower is its own callback.

## Impact Explanation
`_marketState.lossFactor` is set higher than warranted, and `_marketState.totalUnits` is reduced by more than the true uncollateralized shortfall. Every lender who subsequently calls `updatePosition` has their credit slashed proportionally to the inflated loss factor — a direct, concrete loss of lender funds. The borrower's debt is forgiven by an amount that was actually backed by the newly supplied collateral, constituting an illegitimate debt reduction at lenders' expense. This is a high-severity accounting integrity violation with direct fund loss to lenders.

## Likelihood Explanation
Preconditions are minimal and realistic: (1) a position with genuine bad debt — a normal liquidation scenario; (2) the borrower is a smart contract — standard in DeFi; (3) the borrower passes itself as the callback — requires no external authorization, no oracle manipulation, no privileged access, and no victim mistakes. The attack is repeatable on any market with bad debt and is fully self-contained.

## Recommendation
Apply one or more of the following:
1. **Move the callback before bad debt realization**: invoke `onLiquidate` before computing or applying `badDebt`, so any collateral supplied during the callback is visible to the bitmap loop.
2. **Re-read `_position.collateralBitmap` from storage after the callback** and recompute `badDebt` (or validate that the realized amount is still correct).
3. **Add a reentrancy guard** (e.g., a transient storage lock) covering both `liquidate` and `supplyCollateral` to prevent state changes to the borrower's position during the liquidation callback.
4. **Restrict `supplyCollateral` during active liquidation callbacks** by tracking a per-borrower liquidation-in-progress flag (similar to the existing `LIQUIDATION_LOCK_SLOT` used for `take`).

## Proof of Concept
**Minimal unit test outline:**
1. Deploy a market with two collateral types (index 0 and index 1).
2. Create a borrower smart contract that implements `onLiquidate`.
3. Borrower supplies only index-0 collateral (small amount) and borrows such that `debt > collateral_0_value / maxLif_0` (genuine bad debt).
4. Record `_marketState.lossFactor` and `_marketState.totalUnits` before the attack.
5. Borrower calls `liquidate(..., callback = address(this))` with `seizedAssets = repaidUnits = 0`.
6. Inside `onLiquidate`, borrower calls `supplyCollateral(market, 1, largeAmount, address(this))`.
7. After `liquidate` returns, assert that `_marketState.lossFactor` is higher than it would be if `largeAmount` of index-1 collateral had been counted in `badDebt`.
8. Assert that a lender's credit after `updatePosition` is lower than it should be given the total collateral now in the position.
