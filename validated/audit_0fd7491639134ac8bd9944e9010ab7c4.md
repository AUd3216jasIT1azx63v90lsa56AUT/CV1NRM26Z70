Audit Report

## Title
`buyWithAssetsTargetAndWithdrawCollateral` unconditionally reverts when any sell offer has `tick = MAX_TICK` and non-zero settlement fee - (`File: src/periphery/MidnightBundles.sol`)

## Summary

`buyWithAssetsTargetAndWithdrawCollateral` calls `TakeAmountsLib.buyerAssetsToUnits` at line 209 of `MidnightBundles.sol` to compute `unitsToTake`, but this call is placed **outside** the `try/catch` block that begins at line 215. For any sell offer with `tick = MAX_TICK` (5820) and a non-zero settlement fee, `buyerPrice = tickToPrice(MAX_TICK) + settlementFee = WAD + settlementFee > WAD`, triggering the `require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne())` guard in `TakeAmountsLib.buyerAssetsToUnits` and reverting the entire bundle transaction. This is in direct tension with the NatSpec guarantee on line 176 that "If taking an offer reverts, the bundler will completely skip this offer."

## Finding Description

**Exact code path:**

1. `buyWithAssetsTargetAndWithdrawCollateral` is called with a `takes[]` array containing a sell offer (`offer.buy = false`) at `tick = MAX_TICK` (5820). [1](#0-0) 

2. The loop reaches that offer and evaluates the `min(...)` expression. The first argument calls `TakeAmountsLib.buyerAssetsToUnits` at line 209 — **before** the `try` keyword on line 215. [2](#0-1) 

3. Inside `buyerAssetsToUnits`, for a sell offer (`offer.buy = false`): `sellerPrice = offerPrice` and `buyerPrice = sellerPrice + settlementFee = offerPrice + settlementFee`. [3](#0-2) 

4. `tickToPrice(MAX_TICK)` returns exactly `WAD` (1e18). With any `settlementFee > 0`, `buyerPrice = WAD + settlementFee > WAD`, so `require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne())` reverts. [4](#0-3) [5](#0-4) 

5. This revert propagates out of the `min(...)` call, past the `try` keyword, and unwinds the entire transaction. The `try/catch` on lines 215–221 never executes. [6](#0-5) 

**Why existing checks do not stop it:**

- `ConsumableUnitsLib.consumableUnits` (line 213) does **not** call `buyerAssetsToUnits` for sell offers — it calls `sellerAssetsToUnits` (when `maxUnits == 0`) or returns `maxUnits - consumed` directly (when `maxUnits > 0`). Neither path has a `PriceGreaterThanOne` guard for sell offers. [7](#0-6) 

- `sellerAssetsToUnits` has no `require(buyerPrice <= WAD)` guard at all. [8](#0-7) 

- The NatSpec on line 175 acknowledges "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts," but line 176 promises "If taking an offer reverts, the bundler will completely skip this offer." These are contradictory: the revert from `buyerAssetsToUnits` is not a revert from `take()`, yet it kills the entire bundle. [9](#0-8) 

- `MAX_TICK = 5820` is a valid, accessible tick (5820 is divisible by any spacing that divides it). Settlement fees are non-zero by default in any deployed market with non-zero fee settings. [10](#0-9) 

## Impact Explanation

The entire `buyWithAssetsTargetAndWithdrawCollateral` transaction reverts. All other valid offers in the `takes[]` array are blocked. The taker's `targetBuyerAssets` transfer (line 197) is also reverted, so funds are not permanently frozen, but the taker's intended purchase is completely blocked for that bundle invocation. A malicious maker can deliberately post a sell offer at `tick = MAX_TICK` in any market with non-zero settlement fee to permanently grief any taker bundle that includes it, since the condition is structural and does not depend on market state or timing.

## Likelihood Explanation

The precondition requires only: (1) a sell offer at `tick = MAX_TICK` exists in the market, and (2) the market has a non-zero settlement fee. Both conditions are trivially satisfiable — `MAX_TICK` is a valid tick accessible at any spacing dividing 5820, and settlement fees are non-zero by default. A maker can deliberately post such an offer to disrupt any taker bundle that includes it. The condition is permanent and does not require the market to be near maturity. Any taker who aggregates offers from an order book and includes such an offer triggers the revert.

## Recommendation

Move the `TakeAmountsLib.buyerAssetsToUnits` call inside the `try/catch` block, or wrap the entire `min(...)` computation in a `try/catch` so that a revert from `buyerAssetsToUnits` causes the offer to be skipped rather than the entire transaction to revert. Alternatively, add a pre-check before calling `buyerAssetsToUnits` that skips the offer if `tickToPrice(offer.tick) + settlementFee > WAD`, consistent with the existing skip-on-revert intent.

## Proof of Concept

1. Deploy a market with non-zero settlement fee settings.
2. Have a maker post a sell offer at `tick = MAX_TICK` (5820).
3. Have a taker call `buyWithAssetsTargetAndWithdrawCollateral` with a `takes[]` array containing that offer (plus any other valid offers).
4. Observe that the transaction reverts with `PriceGreaterThanOne` before the `try` block is ever reached, blocking all other offers in the bundle.

A minimal Foundry test: set `offer.tick = MAX_TICK`, `offer.buy = false`, confirm `settlementFee(id, ttm) > 0`, then call `buyWithAssetsTargetAndWithdrawCollateral` and assert it reverts with `TickLib.PriceGreaterThanOne`.

        internal
        view
        returns (uint256)

        uint256 offerPrice = TickLib.tickToPrice(offer.tick);
        uint256 settlementFee =
            IMidnight(midnight).settlementFee(id, UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp));
        uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
        return
