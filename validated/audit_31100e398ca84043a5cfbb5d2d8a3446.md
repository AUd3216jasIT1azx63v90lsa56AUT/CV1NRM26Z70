Audit Report

## Title
Buy offer `maxAssets` cap bypass via rounding-to-zero allows unbounded unbacked credit/debt creation and free maker callback spam - (`src/Midnight.sol`)

## Summary
When a buy offer uses `maxAssets`-mode and `offerPrice < WAD` (any tick below `MAX_TICK`), the consumed-accounting increment computes `buyerAssets = units.mulDivDown(buyerPrice, WAD)`, which rounds to 0 for `units = 1`. An attacker can call `take()` with `units = 1` after `consumed == maxAssets`, bypassing the exhaustion guard while still mutating `creditOf(buyer)`, `debtOf(seller)`, `totalUnits`, and triggering the maker's `onBuy` callback — all at zero asset cost, repeatable indefinitely.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

`buyerAssets` for a buy offer is computed with `mulDivDown`: [1](#0-0) 

When `buyerPrice < WAD` (which holds for any `offerPrice < WAD`, i.e., any tick below `MAX_TICK`) and `units = 1`, `mulDivDown(1, buyerPrice, WAD) = (1 * buyerPrice) / WAD = 0` by integer truncation.

The `maxAssets` branch then increments `consumed` by this zero value: [2](#0-1) 

If `consumed[maker][group]` is already equal to `maxAssets`, `newConsumed = maxAssets + 0 = maxAssets`, which satisfies `require(newConsumed <= offer.maxAssets)`. The function does not revert.

Execution continues to mutate position state with the full `units = 1`: [3](#0-2) [4](#0-3) 

The maker's `onBuy` callback is then invoked with `buyerAssets = 0` and `units = 1`: [5](#0-4) 

And the token transfer executes `safeTransferFrom(..., buyerAssets - sellerAssets)` = `safeTransferFrom(..., 0)`: [6](#0-5) 

**Why existing checks fail:**

The only guard is `require(newConsumed <= offer.maxAssets)` at line 369, which passes because `newConsumed = maxAssets`. There is no check that `units == 0` when `buyerAssets == 0`, and no check that `consumed == maxAssets` must cause a revert.

The Certora spec rule `takeConsumedAtMaxUnchangedAssets` only asserts that `consumed` does not increase past `maxAssets` — it does not assert that `units` must be zero or that position state is unchanged when the offer is exhausted: [7](#0-6) 

The analogous strong rule (`fullyConsumedOfferRevertsOnNonTrivialTake`) that asserts `units == 0` on a fully-consumed offer exists only for the `maxAssets == 0` (units-mode) path: [8](#0-7) 

Furthermore, the Certora spec summarizes `mulDivDown` as `NONDET`: [9](#0-8) 

This means the formal verification never reasons about the actual rounding behavior, so the spec cannot detect this class of truncation-to-zero bypass.

## Impact Explanation

After `consumed == maxAssets`, each `take(units=1)` call:
- Returns `buyerAssets = 0`, `sellerAssets = 0` — no loan tokens move
- Leaves `consumed` unchanged at `maxAssets`
- Increases `creditOf(buyer)` and `debtOf(seller)` by 1 unit each with zero asset backing
- Increases `totalUnits` by 1 with no corresponding collateral
- Triggers the maker's `onBuy` callback with `buyerAssets = 0, units = 1`

This violates the core accounting invariant that credit and debt are backed by assets. An unprivileged attacker can create unbounded unbacked credit/debt positions and inflate `totalUnits` at zero cost, and can spam the maker's callback indefinitely. The attack is repeatable on every call since `consumed` never advances past `maxAssets`.

## Likelihood Explanation

All preconditions are reachable by an unprivileged taker with no special access:
1. `offer.buy = true` — standard buy offer
2. `offerPrice < WAD` — any tick below `MAX_TICK`; a normal, common configuration
3. `consumed[maker][group] == maxAssets` — reachable by a prior legitimate fill or by the maker calling `setConsumed`
4. `units = 1` — attacker-controlled, no constraint

The attack costs only gas and is repeatable indefinitely.

## Recommendation

Add an explicit guard that reverts when `buyerAssets == 0` and `units > 0` in the `maxAssets` branch, or equivalently require that `units == 0` whenever the computed asset amount rounds to zero:

```solidity
// In the maxAssets branch, after computing buyerAssets/sellerAssets:
if (offer.maxAssets > 0) {
    uint256 assetDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetDelta > 0 || units == 0, ZeroAssetTake());
    newConsumed = consumed[offer.maker][offer.group] += assetDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Additionally, add a Certora spec rule analogous to `fullyConsumedOfferRevertsOnNonTrivialTake` for the assets-mode path, and remove the `NONDET` summary for `mulDivDown`/`mulDivUp` in `Consume.spec` so that rounding behavior is formally verified.

## Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` directly confirms the bug: [10](#0-9) 

Minimal reproduction steps:
1. Create a buy offer with `maxAssets = N > 0` and `tick = MAX_TICK - 16` (so `tickToPrice(tick) < WAD`)
2. Fill the offer legitimately until `consumed[maker][group] == N`
3. Call `take(offer, ..., units=1, ...)` as an unprivileged attacker
4. Observe: call succeeds, `consumed` remains `N`, but `creditOf(buyer) += 1`, `debtOf(seller) += 1`, `totalUnits += 1`, and `onBuy` is called with `buyerAssets=0, units=1`
5. Repeat step 3 indefinitely — each call succeeds and further inflates unbacked positions

rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
