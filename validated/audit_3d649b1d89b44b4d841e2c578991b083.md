Audit Report

## Title
Zero-asset rounding bypass allows offer reuse after full consumption in assets mode - (File: src/Midnight.sol)

## Summary
When a buy offer has `maxAssets > 0` and `offerPrice < WAD`, calling `take` with a sufficiently small `units` value causes `mulDivDown(units * buyerPrice, WAD)` to round to zero. Because the consumption guard uses `<=` rather than `<`, adding zero to an already-saturated `consumed == maxAssets` still passes, allowing the take to proceed and mutate position state (credit, debt, `totalUnits`) with zero token transfer. The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` explicitly confirms this behavior.

## Finding Description

**Code path** — `src/Midnight.sol`, function `take`:

`buyerAssets` is computed with floor rounding: [1](#0-0) 

The consumption guard then adds `buyerAssets` (which may be 0) to the running total and checks with `<=`: [2](#0-1) 

When `consumed` is already exactly `maxAssets`, adding 0 yields `newConsumed == maxAssets`, which satisfies `<= maxAssets`. Execution continues into the full position-mutation block: [3](#0-2) 

Token transfer at line 455–456 transfers `buyerAssets - sellerAssets = 0 - 0 = 0`, so no tokens move.

**Root cause — two compounding issues:**
1. `mulDivDown` returns 0 for `units * buyerPrice < WAD`, which occurs whenever `buyerPrice < WAD` (any tick strictly below the WAD tick) and `units` is small enough.
2. The guard is `<=` not `<`, so a fully-saturated `consumed == maxAssets` is not rejected when 0 is added.

**Exploit flow:**

| Step | Action | `buyerAssets` added | `consumed` after |
|------|--------|---------------------|-----------------|
| 1 | Normal take filling offer to `maxAssets - 1` | `maxAssets - 1` | `maxAssets - 1` |
| 2 | Take with `units` s.t. `mulDivDown = 1` | `1` | `maxAssets` |
| 3+ | Repeated takes with `units = 1`, `buyerPrice < WAD` | `0` | `maxAssets` ≤ `maxAssets` ✓ |

Steps 1–2 are achievable by the attacker themselves. Step 3 is repeatable indefinitely.

**Why existing checks fail:** There is no `require(buyerAssets > 0)` guard when `units > 0`, and no pre-check that `consumed < maxAssets` (strict) before the addition. The `<=` guard is the sole gate and is defeated by zero-rounding.

## Impact Explanation
Each bypass take grants the maker (lender) free credit units and burdens the taker (borrower) with debt units, while transferring zero loan tokens. `_marketState.totalUnits` is inflated without any corresponding asset backing. This directly violates the core protocol invariant that offers cannot be replayed after full consumption, and corrupts the credit/debt accounting that underpins solvency and liquidation logic. The impact is unauthorized state mutation and accounting inflation, achievable by any unprivileged user.

## Likelihood Explanation
All preconditions are attacker-controlled: `offer.maxAssets > 0` and `offer.buy == true` are standard lender offer parameters; `offerPrice < WAD` holds for any tick below the WAD tick (the test uses `MAX_TICK - 16`). No admin access, oracle manipulation, or victim cooperation is required. The attacker can fill the offer to `maxAssets` themselves in prior takes, then replay indefinitely. The condition `mulDivDown(1 * buyerPrice, WAD) == 0` holds for any `buyerPrice < WAD`, which is the common case.

## Recommendation
Apply one or more of the following fixes in `src/Midnight.sol`:

1. **Strict pre-check**: Before the addition, require `consumed[offer.maker][offer.group] < offer.maxAssets` (strict `<`) so a fully-consumed offer is rejected before any addition.
2. **Non-zero asset guard**: Add `require(units == 0 || buyerAssets > 0)` (and equivalently for `sellerAssets`) immediately after line 364 to reject takes that round to zero assets.
3. **Change `<=` to `<`**: Replace `require(newConsumed <= offer.maxAssets)` with `require(newConsumed < offer.maxAssets)` — but this alone would break the exact-fill case; combining with fix (2) is cleaner.

The most robust fix is (1) + (2) together.

## Proof of Concept
The test `testBugBuyMaxAssetsBypass` at `test/TakeTest.sol` lines 858–889 is a complete, self-contained reproduction: [4](#0-3) 

It pre-saturates `consumed` to `maxAssets` via `setConsumed`, then calls `take(1, borrower, lenderOffer)` with `tick = MAX_TICK - 16` (ensuring `buyerPrice < WAD`). The assertions confirm `buyerAssets == 0` and `sellerAssets == 0` (zero transfer), yet `creditOf(id, lender)`, `debtOf(id, borrower)`, and `totalUnits(id)` all strictly increase — proving the bypass.

        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
