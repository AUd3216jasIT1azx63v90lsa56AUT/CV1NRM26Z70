Audit Report

## Title
Zero-asset rounding bypass allows unbounded position mutation on fully-consumed assets-based buy offers — (`src/Midnight.sol`)

## Summary
When a buy offer uses `maxAssets`-based consumption and `offerPrice < WAD`, a taker can call `take()` with `units = 1` such that `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. Because the consumed accounting increments by zero, the cap check `newConsumed <= maxAssets` passes even when the offer is already fully consumed. Execution continues unconditionally: position credit/debt mutates, `totalUnits` increases, and the maker's `onBuy` callback fires — all with zero tokens transferred and zero cap advancement, repeatable without bound.

## Finding Description

**Root cause** — `src/Midnight.sol`, `take()`: [1](#0-0) 

```
Line 363: buyerAssets = units.mulDivDown(buyerPrice, WAD)
          → units=1, buyerPrice<WAD: mulDivDown(1, buyerPrice, WAD) = 0

Lines 367–369:
  if (offer.maxAssets > 0) {
      newConsumed = consumed[maker][group] += buyerAssets;  // += 0, unchanged
      require(newConsumed <= offer.maxAssets, ConsumedAssets()); // maxAssets <= maxAssets ✓
  }
```

The sole guard preventing execution on an exhausted offer is the `ConsumedAssets` check. Because `buyerAssets` rounds to zero, `newConsumed` does not advance, and the require is trivially satisfied regardless of how many times the call is repeated.

**Execution continues unconditionally after the guard:** [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

- Lines 382–384: `buyerCreditIncrease = zeroFloorSub(1, buyerPos.debt)` — equals 1 if buyer has no debt; `sellerDebtIncrease = 1 - sellerCreditDecrease`.
- Lines 408–414: `buyerPos.credit += 1`, `sellerPos.debt += 1`.
- Lines 416–417: `_marketState.totalUnits` increases by `buyerCreditIncrease`.
- Lines 445–453: `IBuyCallback(buyerCallback).onBuy(..., buyerAssets=0, units=1, ...)` — maker callback fires.
- Lines 455–456: `safeTransferFrom(..., 0)` — zero tokens move, no revert.

**Protocol acknowledgment of the behavior:** [6](#0-5) 

Line 94 states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* This confirms the developers are aware of the rounding path but have not closed it.

**Confirmed by existing test** `testBugBuyMaxAssetsBypass()`: [7](#0-6) 

The test is named "Bug", does not use `vm.expectRevert`, and asserts post-take that `creditOf(lender) > before`, `debtOf(borrower) > before`, `totalUnits > before`, while `consumed` and token balances are unchanged — exactly the divergence described.

## Impact Explanation

1. **Unbounded free credit for the maker**: Each call with `units=1` increments `buyerPos.credit` by 1 at zero token cost. Credit represents a claim on loan tokens at settlement; inflating it allows the maker to redeem more assets than were ever deposited, constituting direct theft from the protocol's asset pool.
2. **Unbounded free debt imposed on the taker**: `sellerPos.debt` increases by 1 per call. A malicious maker can construct a scenario where a victim taker (e.g., one interacting via an automated router) accumulates debt with no corresponding asset receipt.
3. **Uncontrolled callback invocation**: The maker's `onBuy` callback fires with `buyerAssets=0, units=1` on every call. Callback logic that assumes it is only triggered by real asset flows (e.g., minting, state updates, reentrancy-sensitive logic) can be exploited or corrupted.
4. **`totalUnits` diverges from actual asset flows**: Protocol-wide accounting invariants break, potentially affecting liquidation thresholds, fee calculations, and settlement math for all participants in the market.

## Likelihood Explanation

**Required preconditions:**
- `offer.buy = true`, `offer.maxAssets > 0` — standard buy offer configuration.
- `offerPrice < WAD` — any tick below the WAD-price tick; the confirmed test uses `MAX_TICK - 16`.
- `consumed[maker][group] == maxAssets` — reachable after a normal full fill (no special access needed), or self-set by the maker via `setConsumed`.

All preconditions are reachable by an unprivileged external user. The attack requires zero capital (zero tokens transferred), is repeatable without bound on every qualifying exhausted offer, and leaves no on-chain trace distinguishing it from a legitimate take.

## Recommendation

Add a guard immediately after computing `buyerAssets` (line 363) to reject takes that produce zero assets when the offer uses `maxAssets`-based consumption:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(buyerAssets > 0, ZeroAssets());
}
```

Alternatively, enforce that `newConsumed` strictly increases when `maxAssets > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 prev = consumed[offer.maker][offer.group];
    newConsumed = prev + (offer.buy ? buyerAssets : sellerAssets);
    require(newConsumed > prev, ZeroAssets());          // strict increase
    consumed[offer.maker][offer.group] = newConsumed;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Either fix closes the rounding bypass without affecting `maxUnits`-based offers or sell-side `maxAssets` offers where rounding is upward (`mulDivUp`).

## Proof of Concept

The existing test `testBugBuyMaxAssetsBypass()` at lines 858–889 of `test/TakeTest.sol` is a complete, passing PoC: [7](#0-6) 

1. Lender creates a buy offer with `maxAssets = 1`, `tick = MAX_TICK - 16` (so `offerPrice < WAD`).
2. Lender self-sets `consumed` to `maxAssets` via `setConsumed` (simulating a fully consumed offer; equivalently, a prior legitimate fill achieves the same state).
3. Borrower calls `take(1, borrower, lenderOffer)`.
4. Assertions confirm: `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged, token balances unchanged — yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.
5. Step 3 can be repeated indefinitely, amplifying the divergence without bound.
