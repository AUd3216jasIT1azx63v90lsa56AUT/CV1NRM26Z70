Audit Report

## Title
Missing collateral capacity guard in `repaidUnits` path causes arithmetic underflow/`CastOverflow` in `liquidate` - (File: `src/Midnight.sol`)

## Summary
When `liquidate` is called with `repaidUnits > 0` and the liquidated collateral oracle price is very low, `seizedAssets` is computed without any cap against `_position.collateral[collateralIndex]`, producing a value that can vastly exceed available collateral. The RCF check that follows does not enforce `seizedAssets <= collateral`; at near-zero prices it reduces to a trivially-true `0 < rcfThreshold`, allowing the execution to reach line 670 where either `UtilsLib.toUint128` reverts with `CastOverflow` or the subtraction reverts with arithmetic underflow.

## Finding Description
At `src/Midnight.sol:652`, the `repaidUnits` branch computes:

```solidity
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
``` [1](#0-0) 

No subsequent check enforces `seizedAssets <= _position.collateral[collateralIndex]`. The RCF check at lines 662–667:

```solidity
require(
    repaidUnits <= maxRepaid
        || _position.collateral[collateralIndex].mulDivDown(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
            .mulDivDown(WAD, lif).zeroFloorSub(maxRepaid) < market.rcfThreshold,
    RecoveryCloseFactorConditionsViolated()
);
``` [2](#0-1) 

is designed to limit over-liquidation relative to debt recovery, not to bound `seizedAssets` against available collateral. When `liquidatedCollatPrice` is near zero, `collateral * price / ORACLE_PRICE_SCALE * WAD / lif ≈ 0`, so `zeroFloorSub(maxRepaid) = 0`, and `0 < rcfThreshold` is trivially true for any positive `rcfThreshold`. The check passes unconditionally.

Execution then reaches line 670:

```solidity
uint128 newCollateral = _position.collateral[collateralIndex] - UtilsLib.toUint128(seizedAssets);
``` [3](#0-2) 

If `seizedAssets > type(uint128).max`, `UtilsLib.toUint128` reverts with `CastOverflow`. If `seizedAssets <= type(uint128).max` but `seizedAssets > collateral[collateralIndex]`, the subtraction reverts with arithmetic underflow.

**Bad-debt scenario (single collateral):** After bad debt is socialized (lines 626–641), `_position.debt` is reduced. With `collateral = 1`, `price = 1`, `ORACLE_PRICE_SCALE = 1e36`, `lif = 1.1e18`, `repaidUnits = 1`: `seizedAssets = 1 * 1.1e18 / 1e18 * 1e36 / 1 = 1.1e36`. Then `1 - 1.1e36` → underflow revert. [4](#0-3) 

**Multi-collateral RCF bypass:** With collateral A at near-zero price and collateral B covering all debt, the position is unhealthy but has no bad debt. `repaidUnits > maxRepaid` fails the first RCF condition, but `collateral_capacity ≈ 0` makes the second condition trivially true. `seizedAssets = repaidUnits * lif / WAD * ORACLE_PRICE_SCALE / price_A` overflows `uint128` → `CastOverflow`.

## Impact Explanation
Any liquidator calling `liquidate` with `repaidUnits > 0` on a position whose liquidated collateral oracle price has dropped to near zero receives an unexpected revert (`CastOverflow` or arithmetic underflow) instead of a successful liquidation or a descriptive protocol error. This breaks the protocol invariant that a valid `repaidUnits` input (within the borrower's debt) must produce a successful liquidation. In the multi-collateral case, the RCF bypass makes this reachable for any market with `rcfThreshold > 0` without requiring bad debt. The `seizedAssets` path remains functional as a workaround, but the `repaidUnits` path is a documented, supported interface.

## Likelihood Explanation
Requires: (1) a collateral whose oracle price has dropped to near zero — a normal liquidation trigger; (2) a liquidator using the `repaidUnits` input path with a value exceeding `collateral * price / ORACLE_PRICE_SCALE * WAD / lif`. In the multi-collateral case, any positive `rcfThreshold` (the default in all test markets) enables the RCF bypass without bad debt. Oracle price crashes are a standard liquidation scenario, making this realistic and repeatable.

## Recommendation
Add an explicit cap before line 670 to ensure `seizedAssets` does not exceed available collateral:

```solidity
seizedAssets = UtilsLib.min(seizedAssets, _position.collateral[collateralIndex]);
```

Alternatively, add a `require(seizedAssets <= _position.collateral[collateralIndex], ...)` guard immediately after the `seizedAssets` computation in the `repaidUnits` branch (line 652), so the revert is descriptive rather than an arithmetic panic. The RCF check should not be relied upon to enforce collateral capacity bounds.

## Proof of Concept
**Minimal unit test (single-collateral bad debt):**
1. Deploy a market with one collateral, `maxLif = 1.1e18`, `ORACLE_PRICE_SCALE = 1e36`.
2. Create a borrower position: `collateral = 1`, `debt = D` (unhealthy at `price = 1`).
3. Set oracle to return `price = 1`.
4. Call `liquidate(..., seizedAssets=0, repaidUnits=1, ...)`.
5. Observe: bad debt loop reduces `_position.debt`; then `seizedAssets = 1 * 1.1e18 / 1e18 * 1e36 / 1 = 1.1e36`; line 670 reverts with arithmetic underflow.

**Multi-collateral fuzz test:** Fuzz `liquidatedCollatPrice` in `[1, fullRepaymentPrice(repaidUnits) - 1]` with `repaidUnits > maxRepaid` and `rcfThreshold > 0`; assert no `CastOverflow` or arithmetic underflow revert occurs. The test suite already bounds `liquidationOraclePrice >= fullRepaymentPrice(units)` in `testLiquidateUnitsInput` and caps `repaid` to `maxRepaidFromCollat` in `testLiquidateWithBadDebtRepaidInput`, confirming awareness of this gap.
