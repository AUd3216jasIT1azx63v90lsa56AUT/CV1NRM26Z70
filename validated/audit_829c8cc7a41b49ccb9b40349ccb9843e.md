### Title
Lenders Can Front-Run Bad Debt Realization to Avoid Socialized Losses — (File: src/Midnight.sol)

---

### Summary

Midnight socializes bad debt by updating a global `lossFactor` at the moment `liquidate()` is called. The `withdraw()` function applies the `lossFactor` that is current at withdrawal time. A lender who observes a pending liquidation transaction in the mempool can front-run it with a `withdraw()` call, exiting at full pre-slash credit value before the `lossFactor` is updated. The proportional loss that should have been borne by that lender is instead shifted entirely onto the remaining lenders.

---

### Finding Description

**Vulnerability class**: Accounting / timing front-run (MEV sandwich on bad-debt realization).

**Bad debt socialization path** — `liquidate()` in `src/Midnight.sol`:

When `badDebt > 0`, the global `lossFactor` is updated:

```solidity
// src/Midnight.sol lines 631–634
_marketState.lossFactor = UtilsLib.toUint128(
    type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
);
_marketState.totalUnits -= UtilsLib.toUint128(badDebt);
``` [1](#0-0) 

This is the only moment the loss is recorded on-chain. Every lender's credit is reduced lazily — only when `_updatePosition()` is next called for that lender.

**Lender exit path** — `withdraw()` in `src/Midnight.sol`:

```solidity
// src/Midnight.sol lines 484–499
_updatePosition(market, id, onBehalf);   // applies CURRENT lossFactor
...
_position.credit -= UtilsLib.toUint128(units);
_marketState.withdrawable -= UtilsLib.toUint128(units);
...
SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
``` [2](#0-1) 

`_updatePosition` computes the slashed credit as:

```solidity
// src/Midnight.sol lines 805–807
uint256 postSlashCredit = _lastLossFactor < type(uint128).max
    ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor,
                        type(uint128).max - _lastLossFactor)
    : 0;
``` [3](#0-2) 

If `withdraw()` executes **before** `liquidate()`, `marketState[id].lossFactor` still holds the old value, so `postSlashCredit == credit` — no slashing occurs. The lender withdraws the full pre-loss amount.

**Exploit flow**:

1. Market has `withdrawable > 0` (some borrowers have already repaid).
2. A borrower's position has bad debt; a liquidation transaction is broadcast to the mempool.
3. Attacker (lender) observes the pending liquidation and submits `withdraw(units = fullCredit)` with a higher gas price.
4. Attacker's `withdraw()` lands first: `_updatePosition` sees the old `lossFactor`, credit is not slashed, attacker receives `fullCredit` loan tokens.
5. Liquidation lands: `lossFactor` is updated, `totalUnits` is reduced.
6. Remaining lenders' credit is now slashed by a larger factor than if the attacker had stayed, because the bad debt is spread over a smaller `totalUnits` base.

The `withdrawable` constraint is the only gate:

```solidity
_marketState.withdrawable -= UtilsLib.toUint128(units);
``` [4](#0-3) 

This reverts if `withdrawable < units`, so the attack requires that repaid assets are available. In any active market where some borrowers have repaid, this condition is routinely satisfied.

---

### Impact Explanation

A front-running lender extracts their full credit value (e.g., 100 units) instead of the post-slash value (e.g., 90 units after a 10% bad-debt event). The 10-unit shortfall is absorbed by remaining lenders, who each bear a larger proportional loss than the protocol's socialization invariant intends. Repeated across multiple bad-debt events or by multiple coordinated lenders, this can materially drain the withdrawable pool and concentrate losses on passive lenders who do not monitor the mempool. This is a direct, quantifiable financial loss to other protocol participants.

---

### Likelihood Explanation

- **No privileged access required**: any lender with credit in the market can execute this.
- **Observable trigger**: liquidation transactions are public in the Ethereum mempool; bad-debt candidates are identifiable on-chain by comparing `position.debt` to `maxDebt`.
- **Precondition** (`withdrawable > 0`) is routinely met in any market with partial repayments.
- **Cost**: only higher gas fees (standard MEV tooling).
- **Frequency**: every bad-debt liquidation event is an opportunity.

Likelihood is **medium** — it requires mempool monitoring and available withdrawable liquidity, but both are realistic in a live market.

---

### Recommendation

1. **Snapshot-based loss application**: Record each lender's `lastLossFactor` at deposit time and apply the full accumulated loss at withdrawal time regardless of ordering — this is already the design intent of `lastLossFactor`, but the lazy application window is the attack surface. Consider requiring that `_updatePosition` is called (and the resulting slashed credit is used) before computing the withdrawable amount, ensuring the lender cannot withdraw more than their post-slash entitlement even if they race ahead of the liquidation.

2. **Withdrawal queue / delay**: Introduce a minimum delay between a credit position being opened and it becoming withdrawable, or between a withdrawal request and its execution, so that pending bad-debt liquidations can settle first.

3. **Off-chain monitoring**: At minimum, monitor for lenders withdrawing in the same block as or immediately before a bad-debt liquidation, as an indicator of MEV extraction.

### Proof of Concept

```
Setup:
  - Market M with loanToken T, one collateral C, LLTV = 0.86e18.
  - Borrower B supplies collateral, borrows 1000 units of debt.
  - Lender L buys 1000 credit units (via take).
  - Borrower B2 repays 500 units → marketState[M].withdrawable = 500.
  - Borrower B's collateral price drops; B has 200 units of bad debt.
  - Liquidator X submits liquidate(M, B, ...) to mempool.

Attack:
  1. Lender L observes X's pending tx.
  2. L submits withdraw(M, 500, L, L) with gas price > X's gas price.
  3. L's tx lands first:
       _updatePosition: lossFactor unchanged → postSlashCredit = 1000 (no slash)
       credit[L] -= 500 → credit[L] = 500
       withdrawable -= 500 → withdrawable = 0
       L receives 500 T tokens at full value.
  4. X's liquidate lands:
       badDebt = 200
       lossFactor updated: remaining lenders (credit = 500) absorb 200 units of loss
       effective slash ≈ 40% on remaining credit.
  5. L's remaining 500 credit is now worth ~300 T (after 40% slash).
     Without front-running, L's 1000 credit would have been slashed by 20% → 800 T total.
     L extracted 500 T at full value instead of 400 T → gained ~100 T at others' expense.
```

        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
