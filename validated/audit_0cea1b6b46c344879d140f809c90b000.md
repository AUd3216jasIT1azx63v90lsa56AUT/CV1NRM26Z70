Audit Report

## Title
Zero-price oracle enables collateral seizure with zero repayment via `seizedAssets` input path - (`src/Midnight.sol`)

## Summary
When `IOracle.price()` returns `0` for the liquidated collateral, calling `liquidate` with `seizedAssets > 0` and `repaidUnits = 0` computes `repaidUnits = 0` due to integer arithmetic, transferring the borrower's collateral to the liquidator while pulling zero loan tokens. The `repaidUnits > 0` input path correctly reverts via division-by-zero, but the `seizedAssets > 0` path has no such guard. Simultaneously, the bad-debt path zeros the borrower's debt and degrades `lossFactor`, causing lenders to absorb both the loan loss and the collateral loss.

## Finding Description

**Root cause — asymmetric division in `src/Midnight.sol` lines 649–652:**

```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
                              .mulDivUp(WAD, lif);   // = 0 when price = 0
} else {
    seizedAssets = repaidUnits.mulDivDown(lif, WAD)
                              .mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice); // div-by-zero revert
}
```

With `liquidatedCollatPrice = 0`:
- `mulDivUp(seizedAssets, 0, 1e36)` = `(0 + 1e36 − 1) / 1e36` = `0`
- `mulDivUp(0, WAD, lif)` = `(0 + lif − 1) / lif` = `0`
- → `repaidUnits = 0`

The `repaidUnits > 0` branch divides by `liquidatedCollatPrice` directly (line 652), causing a Solidity division-by-zero revert. The `seizedAssets > 0` branch never divides by `liquidatedCollatPrice`; its denominators are always `ORACLE_PRICE_SCALE` (1e36) and `lif` (≥ WAD), both non-zero.

**Full exploit flow:**

1. **Oracle price capture** (lines 610–611): `price = 0` → `liquidatedCollatPrice = 0`
2. **maxDebt/badDebt** (lines 613–616): `maxDebt = 0`, `badDebt = originalDebt`
3. **Liquidatability check** (lines 620–624): `originalDebt > 0 = maxDebt` → passes
4. **Bad-debt write-down** (lines 626–641): `_position.debt` zeroed, `lossFactor` degraded, lenders absorb full principal loss
5. **repaidUnits computation** (lines 649–650): `repaidUnits = 0`
6. **State mutation and transfers** (lines 670–717):
   - `_position.collateral[collateralIndex] -= seizedAssets` (collateral removed)
   - `_marketState.withdrawable += 0` (no credit to lenders)
   - `safeTransfer(collateralToken, receiver, seizedAssets)` (collateral out)
   - `safeTransferFrom(loanToken, payer, address(this), 0)` (nothing in)

**Existing protections reviewed and found insufficient:**

- `atMostOneNonZero(repaidUnits, seizedAssets)` (line 595): only enforces mutual exclusivity; does not block `seizedAssets > 0, repaidUnits = 0`
- `_position.debt > 0` (line 596): satisfied before bad-debt write-down
- `touchMarket` validation (lines 762–773): validates `lltv`, `maxLif`, token sorting — no oracle price bounds check
- Certora `Reverts.spec` rule `oracleZeroCausesLiquidateWithRepaidRevert` (line 246): explicitly scoped to `repaidUnits > 0` (line 248: `require repaidUnits > 0`); no corresponding rule exists for `seizedAssets > 0`
- Certora `NoDivisionByZero.spec` line 124: explicitly requires `ghostPrice(...) > 0` as a precondition, acknowledging the zero-price case is unverified
- Certora `README.md` line 70: states "A reverting or zero-returning collateral oracle blocks `liquidate`" — this invariant is **not proven** for the `seizedAssets` input path and is in fact violated by this code path [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

## Impact Explanation
An unprivileged liquidator extracts the borrower's entire collateral balance at zero cost. Simultaneously, the bad-debt path zeros the borrower's debt and degrades `lossFactor`, so lenders absorb both the full principal loss and the collateral loss — a double loss. This violates the core protocol invariant that collateral cannot be seized outside health/liquidation rules at fair value, and constitutes a direct theft of collateral from the protocol at lenders' expense.

## Likelihood Explanation
No privileged access is required. Market creation is permissionless (`touchMarket` is public) and the oracle address is set at market creation with no price-bounds validation. Any oracle adapter that can transiently return `0` (e.g., a Chainlink wrapper that does not validate `answeredInRound`, a TWAP on a low-liquidity pool, or any adapter with an unhandled edge case) creates the precondition. The attack requires only: (1) a market with such an oracle, (2) a borrower with active debt and collateral, and (3) a call to `liquidate` with `seizedAssets > 0` while the oracle returns `0`. The attack is repeatable across all borrowers in the affected market for as long as the oracle returns `0`. [9](#0-8) [10](#0-9) 

## Recommendation
Add an explicit guard in the `seizedAssets > 0` branch requiring `liquidatedCollatPrice > 0` before computing `repaidUnits`, mirroring the implicit protection the `repaidUnits > 0` branch gets from division-by-zero:

```solidity
if (seizedAssets > 0) {
    require(liquidatedCollatPrice > 0, ZeroOraclePrice());
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
                              .mulDivUp(WAD, lif);
}
```

Additionally, update `Reverts.spec` to add a corresponding rule `oracleZeroCausesLiquidateWithSeizedRevert` covering the `seizedAssets > 0` input path, and remove the `ghostPrice > 0` assumption from `NoDivisionByZero.spec` line 124 once the guard is in place.

## Proof of Concept
Minimal Foundry test plan:
1. Deploy `Midnight` and create a market with a mock oracle that returns `0`.
2. Have a borrower supply collateral and take on debt.
3. Set the mock oracle to return `0`.
4. Call `liquidate(market, collateralIndex, borrower.collateralBalance, 0, borrower, false, attacker, address(0), "")`.
5. Assert: attacker received `borrower.collateralBalance` of collateral token, `0` loan tokens were transferred from attacker, borrower's debt is `0`, and `lossFactor` has degraded (lenders absorbed the loss).
