Audit Report

## Title
Zero-price oracle enables collateral seizure with zero loan token repayment via `seizedAssets` input path - (File: src/Midnight.sol)

## Summary
When the oracle for the liquidated collateral returns `0`, calling `liquidate` with `seizedAssets > 0` and `repaidUnits = 0` causes `repaidUnits` to be computed as `0` due to the `mulDivUp` rounding behavior. The borrower's collateral is fully transferred to the liquidator while their debt is decremented by zero, violating the core invariant that collateral cannot be seized without repaying correct debt value.

## Finding Description

**Root cause — `src/Midnight.sol` lines 607–618:**

The while loop iterates the borrower's `collateralBitmap`, calls `IOracle(_collateralParam.oracle).price()` for each active collateral, and assigns the result to `liquidatedCollatPrice` when `i == collateralIndex`. When the oracle returns `0`, `liquidatedCollatPrice = 0`. That collateral also contributes `0` to `maxDebt` (line 613), making `originalDebt > maxDebt` trivially true.

**`mulDivUp` implementation — `src/libraries/UtilsLib.sol` line 35:**
```solidity
return (x * y + (d - 1)) / d;
```

**Computation at line 650 when `seizedAssets > 0` and `liquidatedCollatPrice = 0`:**
```solidity
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```
- Step 1: `mulDivUp(seizedAssets, 0, ORACLE_PRICE_SCALE)` = `(seizedAssets * 0 + (ORACLE_PRICE_SCALE − 1)) / ORACLE_PRICE_SCALE` = `0` (integer division, since `ORACLE_PRICE_SCALE > 1`)
- Step 2: `mulDivUp(0, WAD, lif)` = `(0 * WAD + (lif − 1)) / lif` = `0` (since `lif ≥ WAD > 1`)

This is confirmed by the `Reverts.spec` axiom at lines 65–68:
```
axiom forall uint256 a. forall uint256 d. d > 0 => CVL_mulDivUpGhost(a, 0, d) == 0;
```

**Resulting state mutations (lines 670–717):**
- `_position.collateral[collateralIndex] -= seizedAssets` — collateral fully removed
- `_marketState.withdrawable += 0` — pool unchanged
- `_position.debt -= 0` — debt unchanged
- `safeTransfer(collateralToken, receiver, seizedAssets)` — collateral sent to attacker
- `safeTransferFrom(loanToken, payer, address(this), 0)` — zero loan tokens paid

**Why existing checks fail:**
- `atMostOneNonZero(0, seizedAssets)` → `true` (attacker passes `repaidUnits = 0`)
- `_position.debt > 0` → `true` (borrower has debt)
- `NotLiquidatable`: oracle returning `0` zeroes the collateral's `maxDebt` contribution, making `originalDebt > maxDebt` trivially true
- `maxRepaid` RCF check (lines 655–668): `repaidUnits = 0 ≤ maxRepaid` always passes

**Formal verification gap — `certora/specs/NoDivisionByZero.spec` line 124:**
The `noDivisionByZeroLiquidate` rule requires `ghostPrice(...) > 0` as a precondition, explicitly excluding the `price = 0` case from coverage. The `certora/README.md` line 98 states: *"except for `liquidate` if the liquidated collateral is not activated or has a price of 0."*

**Coverage gap in `certora/specs/Reverts.spec` lines 246–253:**
The `oracleZeroCausesLiquidateWithRepaidRevert` rule only proves that `liquidate` reverts when `repaidUnits > 0` is passed as input with a zero-price oracle. There is no corresponding rule proving that `liquidate` reverts when `seizedAssets > 0` is passed with a zero-price oracle. This is the uncovered path.

## Impact Explanation
A liquidator seizes the borrower's entire collateral at `collateralIndex` while repaying zero loan tokens. The borrower's debt is not reduced, yet their collateral balance drops to zero and the collateral bitmap bit is cleared. The protocol's `withdrawable` pool is not increased. This constitutes direct, irreversible theft of borrower collateral with no corresponding debt relief — a critical accounting invariant violation causing direct loss of user funds.

## Likelihood Explanation
**Required preconditions:**
1. The oracle for the targeted collateral returns `0` — explicitly modeled as reachable in the Certora specs (`Reverts.spec` ghost `singleZeroOracle`, `forceOracleReturnZero`).
2. The borrower has `debt > 0` and a non-zero balance at `collateralIndex` with the bitmap bit set.
3. No special privilege required — any address can call `liquidate`.

Oracle returning `0` is reachable via: uninitialized price feed, price feed gap, oracle contract returning `0` on error, or a transiently stale feed. The exploit is repeatable for any market whose oracle can transiently return `0`. The protocol's own formal specs explicitly model this scenario, confirming it is considered a reachable state.

## Recommendation
Add an explicit revert when `liquidatedCollatPrice == 0` and `seizedAssets > 0` in the `seizedAssets > 0` branch (line 649), mirroring the existing behavior for the `repaidUnits > 0` branch (which reverts via division by zero at line 652 when `liquidatedCollatPrice = 0`). For example:

```solidity
if (seizedAssets > 0) {
    require(liquidatedCollatPrice > 0, ZeroPriceLiquidation());
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
}
```

Alternatively, add a `require(liquidatedCollatPrice > 0)` guard immediately after the while loop when `seizedAssets > 0` is the input mode. The `Reverts.spec` should also be extended with a rule analogous to `oracleZeroCausesLiquidateWithRepaidRevert` for the `seizedAssets > 0` input path.

## Proof of Concept
**Minimal reproduction steps:**
1. Deploy a market with a collateral whose oracle can be set to return `0`.
2. Have a borrower supply collateral and take on debt (`_position.debt > 0`).
3. Set the oracle to return `0`.
4. Call `liquidate(market, collateralIndex, borrower.collateral[collateralIndex], 0, borrower, false, attacker, address(0), "")`.
5. Observe: attacker receives `seizedAssets` collateral tokens, pays `0` loan tokens, borrower's debt is unchanged, borrower's collateral is `0`.

**Invariant fuzz test:** Assert `debtBefore - debtAfter >= seizedAssets * price / ORACLE_PRICE_SCALE / lif` after every `liquidate` call. With oracle returning `0` and `seizedAssets > 0`, this assertion fails with `debtBefore == debtAfter` and `seizedAssets > 0`.
