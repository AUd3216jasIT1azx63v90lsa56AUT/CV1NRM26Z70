### Title
Lenders Can Withdraw Before Bad Debt Socialization, Causing Unfair Loss Distribution — (File: src/Midnight.sol)

---

### Summary

In Midnight, bad debt is only socialized at the moment `liquidate()` is called, which updates `lossFactor` and slashes all lenders' credit proportionally. Because `withdraw()` applies only the *current* (already-committed) `lossFactor`, any lender who exits before `liquidate()` is called escapes the forthcoming loss entirely. Remaining lenders absorb a disproportionately larger share of the bad debt, mirroring the unfair loss distribution described in the external report.

---

### Finding Description

**Root cause — `withdraw()` has no awareness of unsocialized bad debt:**

`withdraw()` calls `_updatePosition()` at line 485, which applies `marketState[id].lossFactor` to the caller's credit. This factor is only updated inside `liquidate()` (lines 631–634). Until `liquidate()` is called for an insolvent borrower, `lossFactor` does not reflect that borrower's bad debt. Any lender who calls `withdraw()` in this window receives their full, unslashed credit. [1](#0-0) 

**Bad debt is only committed to `lossFactor` inside `liquidate()`:** [2](#0-1) 

**`withdrawable` is populated by ordinary repayments, making the window exploitable:**

When any borrower calls `repay()`, `withdrawable` increases. In a market with multiple borrowers, some will repay while others default. The repaid liquidity creates a withdrawal window before the defaulter is liquidated. [3](#0-2) 

**The protocol itself acknowledges the race:** [4](#0-3) 

The comment acknowledges a race to withdraw but does not restrict it. No mechanism prevents a lender from exploiting this race specifically to avoid bad-debt losses.

**Exploit path:**

1. Market has N lenders with total credit `T`.
2. Borrower A is insolvent: collateral value at `maxLif` < debt → `badDebt = D`.
3. Borrower B has already repaid `R` units → `withdrawable = R`.
4. Lender L (credit `C`) monitors on-chain state, detects Borrower A's insolvency before any liquidator acts.
5. Lender L calls `withdraw(C)` — `_updatePosition` applies the *current* (pre-liquidation) `lossFactor`, so L receives full `C` tokens.
6. A liquidator calls `liquidate()` for Borrower A. `lossFactor` is updated, `totalUnits` drops by `D`.
7. Remaining lenders (total credit `T − C`) absorb the full `D` bad debt: each loses `D / (T − C)` instead of `D / T`.

---

### Impact Explanation

Lenders who exit before `liquidate()` is called bear **zero** of the bad debt. Every unit of bad debt is instead distributed over a smaller remaining credit base, amplifying the per-lender loss for those who stay. In the extreme case where `withdrawable` is large enough for most lenders to exit, a single remaining lender could absorb the entire bad debt alone. The loss is permanent — `lossFactor` is monotonically increasing and credit is never restored. [5](#0-4) 

---

### Likelihood Explanation

The preconditions are realistic and require no privileged access:

- **Multiple borrowers** in a market is the normal operating state.
- **`withdrawable > 0`** is guaranteed whenever any borrower has repaid, which is routine.
- **Detecting insolvency** requires only reading public on-chain state (`position`, oracle prices).
- **No front-running of the liquidator is needed** — the lender simply needs to call `withdraw()` before any liquidator submits `liquidate()`. Given that liquidations are not atomic with insolvency onset, this window can last blocks or longer. [6](#0-5) 

---

### Recommendation

Apply a prospective bad-debt haircut at withdrawal time. Before crediting the lender with `units` tokens, compute the *expected* bad debt from all currently insolvent positions (those where `collateral * price / maxLif < debt`) and reduce the withdrawable amount accordingly, or alternatively track a "pending loss factor" that is applied at withdrawal even before formal liquidation. A simpler mitigation is to require that `withdrawable` only be accessible pro-rata (i.e., `units ≤ credit / totalUnits * withdrawable`), preventing any single lender from draining the pool ahead of a loss event.

---

### Proof of Concept

```
Setup:
  - 10 lenders, each with credit = 100 units  →  totalUnits = 1000
  - Borrower A: debt = 500, collateral value at maxLif = 400  →  badDebt = 100
  - Borrower B: repaid 300 units  →  withdrawable = 300

Step 1: Lender 1 calls withdraw(100)
  _updatePosition applies current lossFactor (0) → no slash
  Lender 1 receives 100 tokens.  withdrawable = 200, totalUnits = 900

Step 2: Lender 2 calls withdraw(100)
  Same result. withdrawable = 100, totalUnits = 800

Step 3: Lender 3 calls withdraw(100)
  Same result. withdrawable = 0, totalUnits = 700

Step 4: liquidate() called for Borrower A
  badDebt = 100 socialized over totalUnits = 700
  Each remaining lender loses 100/700 ≈ 14.3%

Expected (fair) outcome:
  badDebt = 100 socialized over totalUnits = 1000
  Each lender loses 100/1000 = 10%

Lenders 1–3 avoided ~4.3% loss each at the expense of lenders 4–10.
``` [7](#0-6) [8](#0-7)
