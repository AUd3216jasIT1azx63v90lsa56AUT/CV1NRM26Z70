## Title
`claimContinuousFee` reverts via arithmetic underflow when bad-debt liquidation front-runs the feeClaimer - (File: `src/Midnight.sol`)

## Summary
`claimContinuousFee` performs an unchecked subtraction on `continuousFeeCredit` with no guard that `amount <= continuousFeeCredit`. An unprivileged caller can invoke `liquidate` with `seizedAssets=0, repaidUnits=0` at zero token cost to realize existing bad debt, atomically reducing `continuousFeeCredit` before the feeClaimer's pending transaction executes, causing it to revert with a Solidity 0.8 arithmetic underflow. The attack is repeatable across distinct bad-debt positions and price-drop events, persistently griefing protocol fee collection.

## Finding Description
**Root cause — `claimContinuousFee`** (`src/Midnight.sol:318`):

```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
```

No `require(amount <= _marketState.continuousFeeCredit)` guard exists. The only protection is Solidity 0.8 checked arithmetic, which reverts on underflow. The Certora spec at `certora/specs/Role.spec:264` encodes this revert condition explicitly:

```
assert !reverted <=> ... && amount <= continuousFeeCreditBefore;
```

confirming the absence of any on-chain guard.

**Bad-debt slash — `liquidate`** (`src/Midnight.sol:635-640`):

```solidity
_marketState.continuousFeeCredit = _lossFactor < type(uint128).max
    ? UtilsLib.toUint128(
        _marketState.continuousFeeCredit
            .mulDivDown(type(uint128).max - _marketState.lossFactor,
                        type(uint128).max - _lossFactor)
      )
    : 0;
```

Because `newLossFactor > oldLossFactor`, the ratio `(MAX − newLF) / (MAX − oldLF) < 1`, so `continuousFeeCredit` is strictly reduced on every bad-debt event.

**Zero-cost liquidation path**: The protocol explicitly supports `seizedAssets=0, repaidUnits=0` calls (`src/Midnight.sol:577`):

> "Passing both 0 for seizedAssets and repaidUnits allows to realize bad debt with 0 token transferred."

The `atMostOneNonZero` check at line 595 passes when both are zero (`or(iszero(0), iszero(0)) = 1`). The token-transfer block at line 643 (`if (repaidUnits > 0 || seizedAssets > 0)`) is skipped entirely. The only cost to the attacker is gas.

**Exploit flow**:
1. `continuousFeeCredit = X`. feeClaimer broadcasts `claimContinuousFee(market, X, receiver)`.
2. Attacker front-runs with `liquidate(market, 0, 0, 0, borrower, false, receiver, address(0), "")` — zero tokens transferred. Bad debt is realized; `continuousFeeCredit` drops to `X' < X`.
3. feeClaimer's transaction executes: `X' -= X` → arithmetic underflow → revert.

**Why existing checks fail**: The `liquidatorGate` check (`src/Midnight.sol:597-600`) only applies when `market.liquidatorGate != address(0)`; markets with `liquidatorGate == address(0)` allow any caller. No minimum-amount or slippage guard exists in `claimContinuousFee`.

## Impact Explanation
The feeClaimer's `claimContinuousFee` transactions are persistently griefed and revert. While no funds are directly stolen, protocol fee collection is disrupted: the feeClaimer must re-read on-chain state and resubmit on every attempt. The attack is zero-cost (gas only) and can be replayed across multiple bad-debt positions or repeated price-drop events, constituting a sustained denial-of-service against a core protocol revenue function.

## Likelihood Explanation
**Attacker cost**: gas only — no tokens required. **Preconditions**: a liquidatable position with `badDebt > 0` must exist (routine in any lending market experiencing price volatility) and the feeClaimer must submit `amount` equal to the current `continuousFeeCredit` (the natural "claim all" pattern). **Repeatability**: each distinct bad-debt position or subsequent price drop creates a new front-run opportunity. **Access**: any unprivileged external address can call `liquidate` on markets without a `liquidatorGate`.

## Recommendation
Add an explicit guard in `claimContinuousFee` before the subtraction:

```solidity
uint128 credit = _marketState.continuousFeeCredit;
uint128 amt128 = UtilsLib.toUint128(amount);
require(amt128 <= credit, InsufficientContinuousFeeCredit());
_marketState.continuousFeeCredit = credit - amt128;
```

Alternatively, clamp `amount` to `min(amount, continuousFeeCredit)` so the feeClaimer always succeeds and collects whatever is available, eliminating the griefing surface entirely.

7. Now broadcast the feeClaimer's transaction — observe it reverts with arithmetic underflow.
8. Confirm `continuousFeeCredit == X'` on-chain (fees still present but unclaimed).
