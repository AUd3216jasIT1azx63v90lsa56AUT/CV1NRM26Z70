### Title
Missing deadline parameter in `stake()` allows pending transactions to execute in unfavorable registry states, permanently locking staker principal - (File: src/ConfidencePool.sol)

### Summary
The `stake()` function accepts no deadline or maximum-state guard. A transaction submitted in a safe pre-attack registry state (`NOT_DEPLOYED`, `NEW_DEPLOYMENT`, `ATTACK_REQUESTED`) can sit in the mempool and execute after the registry has transitioned to `UNDER_ATTACK`, permanently locking the staker's principal with near-zero bonus and no withdrawal path.

### Finding Description
`stake()` uses only `block.timestamp` as the deposit entry time and has no user-supplied deadline. When a staker submits the transaction in a pre-attack state (where `withdraw()` is still available), they expect to be able to exit if conditions change. However, if the transaction is delayed in the mempool and the registry transitions to `UNDER_ATTACK` before mining:

1. `_observePoolState()` seals `riskWindowStart = block.timestamp` [1](#0-0) 
2. `_assertDepositsAllowed` does **not** revert for `UNDER_ATTACK` — by design, but the staker did not choose this state [2](#0-1) 
3. The deposit entry time is recorded as `≈ riskWindowStart`, so `(T − entry)² ≈ 0` → bonus share ≈ 0 [3](#0-2) 
4. `withdraw()` is permanently disabled because `riskWindowStart != 0` [4](#0-3) 

The staker is locked into a position they did not intend to take, with near-zero bonus and principal at risk until resolution. DESIGN.md §3 states the UNDER_ATTACK deposit path is "voluntary" and that "a staker who does not want this can read the live registry state before staking" — but this reasoning applies to a staker who *chooses* to stake in that state, not to one whose transaction was submitted in a safe state and executed in a dangerous one due to mempool delay. The document does not address the pending-transaction scenario, so this is a distinct implementation defect, not a documented accepted trade-off. [5](#0-4) 

### Impact Explanation
If the pool resolves CORRUPTED (bad-faith), the staker's entire principal is swept to `recoveryAddress` with zero recovery. [6](#0-5)  Even under good-faith CORRUPTED, the staker's principal goes to the named attacker's bounty. [7](#0-6)  The staker intended to stake in a safe state where they retained the right to exit — that right is permanently forfeited by the delayed execution.

### Likelihood Explanation
Requires the registry to transition from a pre-attack state to `UNDER_ATTACK` while the staker's transaction is pending. On an EVM-compatible L2 like BattleChain, this is less common than on mainnet, but network congestion events can cause transactions to sit in the mempool. The registry transition from pre-attack to `UNDER_ATTACK` is a normal, expected protocol event for every active agreement. [8](#0-7) 

### Recommendation
Add a `deadline` parameter to `stake()` that reverts if `block.timestamp > deadline`:

```solidity
function stake(uint256 amount, uint256 deadline) external nonReentrant whenPoolNotPaused {
    if (block.timestamp > deadline) revert DeadlineExpired();
    // ... rest of function
}
```

Optionally, add a `maxState` parameter that reverts if the current registry state exceeds the staker's intended maximum, allowing stakers to specify they only want to stake in pre-attack states.

### Proof of Concept
1. Alice submits `stake(1000e18)` when the registry is in `NEW_DEPLOYMENT` (withdrawals allowed, no risk). [9](#0-8) 
2. The transaction sits in the mempool due to network congestion on BattleChain.
3. The registry transitions to `UNDER_ATTACK`; `riskWindowStart` is sealed by another pool interaction or `pokeRiskWindow()`. [10](#0-9) 
4. Alice's transaction is mined: `_observePoolState()` observes `UNDER_ATTACK`, `_assertDepositsAllowed` does not revert. [11](#0-10) 
5. Alice's stake is recorded with `newEntry ≈ riskWindowStart` → `(T − entry)² ≈ 0` → bonus share ≈ 0. [12](#0-11) 
6. Alice calls `withdraw()` → reverts `WithdrawsDisabled` (`riskWindowStart != 0`). [13](#0-12) 
7. The moderator flags CORRUPTED (bad-faith) → Alice's 1000e18 tokens are swept to `recoveryAddress`. [14](#0-13) 
8. Alice loses her entire principal despite having submitted the transaction in a safe state with the intent to retain withdrawal rights.

```text
    function _markRiskWindowStart() internal {
        // Cap at expiry: accrual is bounded by the pool's lifecycle. Without the cap, a late
        // observation could pin riskWindowStart > expiry, and `_clampUserSums` would record every
        // pre-risk deposit as entering past T = expiry (EXPIRED path). The k=2 sums then encode
        // post-deadline "at-risk" time that never actually existed.
        uint256 t = block.timestamp;
        if (t > expiry) t = expiry;
        // Cast is truncation-safe: `t` is capped at `expiry`, which is itself a uint32.
        // forge-lint: disable-next-line(unsafe-typecast)
        riskWindowStart = uint32(t);
        // Eagerly reset the global accumulators so every currently-eligible staker is treated as
        // entering at `t`. Per-user sums stay stale until `_clampUserSums` runs on the next op
        // touching that user.
        sumStakeTime = totalEligibleStake * t;
        sumStakeTimeSq = totalEligibleStake * t * t;
        emit RiskWindowStarted(t);
```

**File:** docs/DESIGN.md (L57-71)
```markdown
- **`UNDER_ATTACK`:** a depositor takes on real, visible, voluntary risk. Their bonus share is
  crushed to ~zero by the k=2 weighting — entry is floored at `riskWindowStart ≈ now`, so
  `(T − entry)² ≈ 0` — and the deposit self-locks into the resolution path (`withdraw` is
  disabled once `riskWindowStart != 0`). There is no late-join *advantage* to gate against and
  no trap: it is voluntary, near-zero-reward risk capital. A staker who does not want this can
  read the live registry state before staking.
- **`PROMOTION_REQUESTED`:** the agreement has requested to exit toward `PRODUCTION`, so the risk
  window is about to close (`riskWindowEnd` imminent). Blocking deposits here stops a late join
  in the final stretch — entering with almost no remaining risk-bearing time before resolution.
  The state is still attackable; it is the *closing window*, not assured survival, that justifies
  the block.

**Why this is not a "trap" finding:** the withdraw-lock from `UNDER_ATTACK` onward is the *same*
intended commitment every staker accepts once risk materializes, and the late entrant earns
negligible bonus. Principal remains recoverable through resolution.
```

```markdown
3. **Risk window.** `riskWindowStart` is sealed on first observation of `UNDER_ATTACK` or `PROMOTION_REQUESTED` (active-risk states only). `riskWindowEnd` is sealed on first observation of `PRODUCTION` or `CORRUPTED`. Both happen lazily on the next pool interaction, or eagerly via permissionless `pokeRiskWindow()`. If the registry transitions straight from `NEW_DEPLOYMENT` to a terminal state without anyone observing an active-risk state, `riskWindowStart` never seals — see "no observed risk" below.
4. **Withdraw window.** Stakers can fully exit via `withdraw()` while the registry is in any pre-attack state (`NOT_DEPLOYED`, `NEW_DEPLOYMENT`, or `ATTACK_REQUESTED`). Withdrawals are permanently disabled from `UNDER_ATTACK` onward. Withdrawing forfeits any bonus claim.
```
