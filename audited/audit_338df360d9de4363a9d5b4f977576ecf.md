### Title
Authorized address can de-ratify maker's offer tree root, freezing all takes - (`src/ratifiers/SetterRatifier.sol`)

### Summary
`SetterRatifier.setIsRootRatified` gates write access with the same coarse-grained `isAuthorized(maker, msg.sender)` check used across the entire Midnight protocol. Because this check is a flat boolean with no action-level scoping, any address that makerB has ever authorized for any purpose can call `setIsRootRatified(makerB, root, false)`, setting `isRootRatified[makerB][root]` to `false` and causing every subsequent `take` against makerB's offer tree to revert with `NotRatified`.

### Finding Description

**Exact code path:**

`SetterRatifier.setIsRootRatified` (line 24–26):

```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
``` [1](#0-0) 

The guard delegates to `IMidnight.isAuthorized(maker, msg.sender)`:

```solidity
mapping(address maker => mapping(bytes32 root => bool)) public isRootRatified;
``` [2](#0-1) 

`isRatified` then hard-reverts if the flag is `false`:

```solidity
require(isRootRatified[offer.maker][root], NotRatified());
``` [3](#0-2) 

**Root cause:** `newIsRootRatified` is an unconstrained `bool`. The authorization check does not distinguish between setting the flag to `true` (ratifying) and setting it to `false` (de-ratifying/freezing). Any address that passes `isAuthorized(maker, msg.sender)` can flip the flag in either direction.

**Exploit flow:**

1. makerB calls `midnight.setIsAuthorized(makerA, true, makerB)` for any legitimate reason (e.g., to let makerA repay debt or manage collateral on makerB's behalf). This sets `isAuthorized[makerB][makerA] = true`. [4](#0-3) 

2. makerA calls `setterRatifier.setIsRootRatified(makerB, root, false)`.
   - `maker == msg.sender` → `makerB == makerA` → **false**
   - `IMidnight(MIDNIGHT).isAuthorized(makerB, makerA)` → **true** (step 1)
   - Require passes; `isRootRatified[makerB][root] = false` is written.

3. Any taker calling `midnight.take(offer, ...)` where `offer.maker == makerB` and `offer.ratifier == address(setterRatifier)` triggers `isRatified`, which now reverts `NotRatified`.

**Why existing checks fail:** There is no check on the value of `newIsRootRatified`. The authorization model is a flat boolean with no action scope; the same `isAuthorized` mapping governs `withdraw`, `repay`, `setConsumed`, `setIsAuthorized`, and now `setIsRootRatified` in the external ratifier.

### Impact Explanation

makerB's entire offer tree for the targeted `root` is frozen: every `take` call against any offer in that Merkle tree reverts `NotRatified`. makerB cannot fill any orders until they call `setIsRootRatified(makerB, root, true)` themselves — but makerA can immediately re-freeze it in the same block. If makerA is a smart contract, this can be made atomic and permanent (grief loop). This matches the scoped impact: long-term freezing of the take path for makerB's offer tree.

### Likelihood Explanation

**Preconditions:**
- makerB must have called `setIsAuthorized(makerA, true, makerB)` at any point in the past for any reason.
- makerB's offers must use `SetterRatifier` as their ratifier.

Both conditions are realistic: the protocol explicitly supports and tests authorized-party ratification (`testIsRatifiedAuthorizedSetterCanRatifyOnBehalf`, `testTakeAuthorizedSetterCanRatifyOnBehalf`). [5](#0-4) 

The attack is repeatable: makerA can re-call `setIsRootRatified(makerB, root, false)` after every recovery attempt by makerB, as long as the authorization remains active. makerB revoking the authorization is the only mitigation, but the damage is already done for any in-flight takes.

### Recommendation

Restrict de-ratification (setting `newIsRootRatified = false`) to `maker == msg.sender` only, while allowing authorized parties to ratify (`true`) on behalf of the maker:

```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    if (!newIsRootRatified) {
        require(maker == msg.sender, Unauthorized());
    } else {
        require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    }
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
```

This preserves the ability for authorized operators to ratify offer trees on behalf of a maker (the intended use case) while ensuring only the maker themselves can freeze their own tree.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {SetterRatifier} from "../src/ratifiers/SetterRatifier.sol";
import {ISetterRatifier} from "../src/ratifiers/interfaces/ISetterRatifier.sol";
import {BaseTest} from "./BaseTest.sol";
import {HashLib} from "../src/ratifiers/libraries/HashLib.sol";
import {Offer, CollateralParams, Market} from "../src/interfaces/IMidnight.sol";
import {MAX_TICK} from "../src/libraries/TickLib.sol";

contract SetterRatifierDeRatifyTest is BaseTest {
    SetterRatifier internal setterRatifier;

    function setUp() public override {
        super.setUp();
        setterRatifier = new SetterRatifier(address(midnight));
    }

    function testAuthorizedCanDeRatifyMakersRoot() public {
        address makerB = lender;
        address makerA = borrower;

        // Build an offer for makerB
        Market memory market;
        market.loanToken = address(loanToken);
        market.maturity = vm.getBlockTimestamp() + 100;
        market.collateralParams = new CollateralParams[](1);
        market.collateralParams[0] = CollateralParams({
            token: address(collateralToken1), lltv: 0.77e18,
            maxLif: maxLif(0.77e18, 0.25e18), oracle: address(oracle1)
        });
        Offer memory offer;
        offer.market = market;
        offer.buy = true;
        offer.maker = makerB;
        offer.ratifier = address(setterRatifier);
        offer.maxUnits = type(uint256).max;
        offer.expiry = vm.getBlockTimestamp() + 200;
        offer.tick = MAX_TICK;

        bytes32 root = HashLib.hashOffer(offer);

        // makerB ratifies their own root
        vm.prank(makerB);
        setterRatifier.setIsRootRatified(makerB, root, true);
        assertTrue(setterRatifier.isRootRatified(makerB, root));

        // makerB authorizes makerA for unrelated purposes
        vm.prank(makerB);
        midnight.setIsAuthorized(makerA, true, makerB);

        // makerA de-ratifies makerB's root — should revert Unauthorized but does NOT
        vm.prank(makerA);
        setterRatifier.setIsRootRatified(makerB, root, false);

        // Assert: makerB's root is now frozen
        assertFalse(setterRatifier.isRootRatified(makerB, root));

        // Assert: any take against makerB's offer now reverts NotRatified
        vm.prank(address(midnight));
        vm.expectRevert(ISetterRatifier.NotRatified.selector);
        setterRatifier.isRatified(offer, abi.encode(root, 0, new bytes32[](0)));
    }
}
```

**Expected assertions:**
- `setterRatifier.isRootRatified(makerB, root)` is `false` after makerA's call (line `assertFalse`).
- `isRatified` reverts `NotRatified` (line `vm.expectRevert`).
- The test passes without any revert on `setIsRootRatified`, confirming the missing guard.
