### Title
Liquidator Gate Bypassed via Repay Callback Reentrancy - (File: src/interfaces/IGate.sol / src/Midnight.sol)

### Summary
The `liquidate` function checks `msg.sender` against `ILiquidatorGate.canLiquidate`, but when `liquidate` is called from within a `repay` callback (`IRepayCallback.onRepay`), `msg.sender` is the callback contract rather than the original `repay` caller. Because `repay` sets no liquidation lock and propagates no caller identity through transient storage, an address blocked by the liquidator gate can route a liquidation through a gate-allowed callback contract, fully bypassing the gate invariant.

### Finding Description

**Code path:**

`repay` (lines 502–521) performs no gate check and sets no `liquidationLocked` flag for any borrower. It calls the callback at lines 514–519:

```solidity
if (callback != address(0)) {
    require(
        IRepayCallback(callback).onRepay(id, market, units, onBehalf, data) == CALLBACK_SUCCESS,
        WrongRepayCallbackReturnValue()
    );
}
``` [1](#0-0) 

Inside that callback, any call to `liquidate` has `msg.sender == callback`. The gate check in `liquidate` is:

```solidity
require(
    market.liquidatorGate == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
``` [2](#0-1) 

`liquidationLocked` is only set during `take` callbacks (line 444) for the seller: [3](#0-2) 

`repay` never sets it. There is no transient-storage mechanism in `repay` that records the original caller for downstream gate checks.

**Attacker-controlled inputs:**
- `callback` parameter of `repay` — set to a contract B that is whitelisted by the gate
- `units` — can be 0 (no underflow: `uint128(0) - uint128(0) = 0`), so A need not hold any debt
- `onBehalf` — set to A itself (always satisfies `onBehalf == msg.sender`)

**Exploit flow:**
1. Market is created with `liquidatorGate` that blocks address A but allows contract B.
2. A calls `midnight.repay(market, 0, A, B, encodedLiquidateArgs)`.
3. `repay` calls `B.onRepay(...)` — at this point `msg.sender` inside B is `midnight`.
4. B's `onRepay` calls `midnight.liquidate(market, collateralIndex, seizedAssets, 0, victimBorrower, false, receiver, address(0), "")`.
5. Inside `liquidate`, `msg.sender == B`; gate returns `true` for B → gate check passes.
6. `victimBorrower` is liquidated; A is the economic beneficiary (B forwards seized collateral to A).

**Why existing checks fail:**
- The `liquidationLocked` guard only protects sellers during `take` callbacks; it is never set during `repay` callbacks. [4](#0-3) 
- The Certora rule `liquidatorGateBlocksLiquidation` models `liquidate` called directly (with `e.msg.sender` as the top-level caller), not via a nested call from a repay callback, so this path is not covered. [5](#0-4) 

### Impact Explanation
An address explicitly blocked by a market's `liquidatorGate` can liquidate any unhealthy borrower in that market by routing the call through a gate-allowed repay callback contract. The gate invariant — that only permitted addresses may liquidate — is fully violated. The blocked address suffers no economic barrier and can repeat the attack at will.

### Likelihood Explanation
Preconditions: (1) a market with a non-zero `liquidatorGate` that distinguishes between EOAs and contracts, or between specific addresses; (2) the attacker can deploy or control a contract that the gate allows. Both are realistic: permissioned markets commonly whitelist smart-contract liquidators (e.g., MEV bots, liquidation routers) while blocking EOAs or vice versa. The attack requires no special privilege, no oracle manipulation, and no user mistake. It is repeatable on every unhealthy position.

### Recommendation
Mirror the pattern used by `take` for the seller lock: before invoking the repay callback, write the original `msg.sender` into a transient storage slot (e.g., `REPAY_CALLER_SLOT`). In `liquidate`, if that slot is non-zero, use its value instead of `msg.sender` for the gate check, then clear it after the check. This ensures the gate always evaluates the entity that initiated the outermost protocol interaction, regardless of call depth.

Alternatively, add a transient reentrancy guard in `repay` that prevents `liquidate` from executing during the callback phase, though this would break legitimate flash-liquidation patterns that combine repay and liquidate atomically.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import {IMidnight, Market} from "src/interfaces/IMidnight.sol";
import {IRepayCallback} from "src/interfaces/ICallbacks.sol";
import {ILiquidatorGate} from "src/interfaces/IGate.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";

/// Gate: blocks attacker EOA, allows this callback contract.
contract BypassGate is ILiquidatorGate {
    address public immutable blocked;
    address public immutable allowed;
    constructor(address _blocked, address _allowed) {
        blocked = _blocked; allowed = _allowed;
    }
    function canLiquidate(address account) external view returns (bool) {
        return account == allowed; // blocks EOA, allows callback contract
    }
}

/// Callback contract: whitelisted by gate, calls liquidate on behalf of attacker.
contract MaliciousRepayCallback is IRepayCallback {
    IMidnight public midnight;
    Market public market;
    address public victimBorrower;
    address public receiver;

    constructor(IMidnight _midnight) { midnight = _midnight; }

    function setup(Market calldata _market, address _victim, address _receiver) external {
        market = _market; victimBorrower = _victim; receiver = _receiver;
    }

    function onRepay(bytes32, Market memory, uint256, address, bytes memory)
        external returns (bytes32)
    {
        // msg.sender here is midnight; nested liquidate sees msg.sender == address(this)
        midnight.liquidate(market, 0, 1, 0, victimBorrower, false, receiver, address(0), "");
        return CALLBACK_SUCCESS;
    }
}

contract GateBypassTest is BaseTest {
    function testLiquidatorGateBypassViaRepayCallback() public {
        address attacker = makeAddr("attacker");
        MaliciousRepayCallback cb = new MaliciousRepayCallback(midnight);
        BypassGate gate = new BypassGate(attacker, address(cb));

        // Create gated market, supply collateral, open position for victimBorrower
        // ... (standard setup) ...
        // Drop oracle price to make victimBorrower unhealthy

        cb.setup(gatedMarket, victimBorrower, attacker);

        // Attacker calls repay with 0 units — no debt required
        vm.prank(attacker);
        midnight.repay(gatedMarket, 0, attacker, address(cb), "");

        // Assert: liquidation succeeded despite attacker being gated
        assertLt(midnight.debtOf(gatedId, victimBorrower), originalDebt, "debt reduced: gate bypassed");
        // Assert: direct liquidate by attacker still reverts
        vm.prank(attacker);
        vm.expectRevert(IMidnight.LiquidatorGatedFromLiquidating.selector);
        midnight.liquidate(gatedMarket, 0, 1, 0, victimBorrower, false, attacker, address(0), "");
    }
}
```

Expected assertions:
- `midnight.debtOf(gatedId, victimBorrower) < originalDebt` — liquidation succeeded via callback
- Direct `liquidate` by `attacker` reverts with `LiquidatorGatedFromLiquidating` — confirming the gate blocks the attacker directly but not via the callback path
