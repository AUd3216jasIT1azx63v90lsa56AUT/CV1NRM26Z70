### Title
`onLiquidate` Callback Can Supply Collateral After Bad Debt Is Realized, Over-Slashing Lenders - (File: src/Midnight.sol)

### Summary

In `liquidate()`, bad debt is computed from a local snapshot of `collateralBitmap` and immediately applied to `lossFactor` and `totalUnits` before the `onLiquidate` callback fires. Because `supplyCollateral` has no reentrancy guard and the liquidation lock is never set during `liquidate()`, an authorized callback can add new collateral to the borrower's position after the bad debt accounting is already finalized. The new collateral is never counted against `badDebt`, so lenders are slashed by more than the true shortfall.

### Finding Description

**Exact code path:**

`liquidate()` takes a local copy of the bitmap at line 606:

```solidity
uint128 _collateralBitmap = _position.collateralBitmap;   // local snapshot
while (_collateralBitmap != 0) { ... }                    // badDebt computed here
``` [1](#0-0) 

Bad debt is then immediately committed to storage — `_position.debt`, `_marketState.lossFactor`, and `_marketState.totalUnits` are all updated — before any external call: [2](#0-1) 

Only after all of that does the callback fire: [3](#0-2) 

`supplyCollateral` has no reentrancy guard and does not check `liquidationLocked` (which is only set during `take()`, not `liquidate()`): [4](#0-3) 

The liquidation lock is only ever written in `take()`: [5](#0-4) 

**Attacker-controlled inputs and exploit flow:**

1. Borrower has debt `D`, collateral `C0` at index 0; oracle price drops so `D > C0·price0/maxLif0` (bad debt territory).
2. Borrower calls `setIsAuthorized(maliciousCallback, true, borrower)`.
3. Anyone calls `liquidate(market, 0, 0, 0, borrower, false, receiver, maliciousCallback, data)`.
4. Inside `liquidate()`:
   - `badDebt = D − C0·price0/maxLif0` computed from local bitmap snapshot (only index 0 present).
   - `lossFactor` updated, `totalUnits` reduced by `badDebt` — lenders are slashed.
   - `_position.debt -= badDebt`.
   - `onLiquidate` callback fires.
5. Inside `onLiquidate`, `maliciousCallback` calls `supplyCollateral(market, 1, largeAmount, borrower)`. Because `isAuthorized[borrower][maliciousCallback]` is true, this succeeds. `_position.collateral[1] = largeAmount` and bit 1 is set in `_position.collateralBitmap`.
6. Callback returns `CALLBACK_SUCCESS`; loan token pulled from payer.

**Why checks fail:**

- The `liquidationLocked` transient flag is never set inside `liquidate()`, so `supplyCollateral` is not blocked.
- `supplyCollateral`'s only guard is the authorization check, which the attacker satisfies.
- The Certora formal verification explicitly assumes callbacks do not re-enter Midnight (README line 113), so this path is not covered by the proofs. [6](#0-5) 

### Impact Explanation

Lenders are over-slashed by `W1 = C1·price1/ORACLE_PRICE_SCALE·WAD/maxLif1` units of credit. The borrower's debt is reduced by this same extra amount while the borrower retains the `C1` collateral tokens in their position (withdrawable once healthy). The net cost to the attacker is only gas; the net loss to lenders is `W1` units of credit, which is a permanent reduction in their redeemable value. This directly violates the invariant that bad debt realization must account for all collateral present after the liquidation.

### Likelihood Explanation

**Preconditions:**
1. Position must be in bad debt territory (oracle price sufficiently low).
2. Borrower must have authorized the callback contract (`setIsAuthorized`).

Condition 2 is fully attacker-controlled: the borrower can authorize their own callback contract at any time before the liquidation. Self-liquidation (borrower calls `liquidate` on themselves) is not blocked. The attack is repeatable across any market with bad debt and is not gated by any privileged role.

### Recommendation

Set the liquidation lock for the borrower at the start of `liquidate()` and clear it at the end, mirroring the pattern used in `take()`. This prevents any re-entrant call to `supplyCollateral` (or any other state-mutating function) during the `onLiquidate` callback:

```solidity
// At the top of liquidate(), after the gate check:
bool wasLocked = UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, borrower, true);

// ... existing logic ...

// At the very end, before returning:
if (!wasLocked) UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, borrower, false);
```

Then add a check in `supplyCollateral` (or rely on the lock being checked there) to revert if the borrower's liquidation is locked. Alternatively, snapshot the full position state (debt and all collateral values) before the callback and revert if any collateral was added during the callback.
