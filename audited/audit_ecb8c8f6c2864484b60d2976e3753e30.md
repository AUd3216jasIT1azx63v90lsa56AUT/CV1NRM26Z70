### Title
Fee-on-Transfer Loan Token in `take` Causes `claimableSettlementFee` to Exceed Actual Contract Balance - (`File: src/libraries/SafeTransferLib.sol` / `src/Midnight.sol`)

### Summary
In `Midnight.take`, `claimableSettlementFee[loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` before the inbound transfer occurs. `SafeTransferLib.safeTransferFrom` only verifies the boolean return value of `transferFrom`; it does not measure the actual tokens received. If the loan token deducts a fee on transfer, the contract receives less than the recorded amount, permanently breaking the invariant that the contract's token balance must cover all claimable settlement fees.

### Finding Description

**Code path:**

`src/Midnight.sol` line 418 increments the accounting counter before any transfer: [1](#0-0) 

Then line 455 pulls the settlement-fee portion from the payer into the contract: [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only checks that `transferFrom` returns `true` (or no return data); it performs no before/after balance measurement: [3](#0-2) 

**Root cause:** The accounting write (`+= buyerAssets - sellerAssets`) is unconditional and uses the nominal requested amount, not the amount actually credited to the contract. There is no balance-delta check anywhere in `take`.

**Exploit flow:**
1. Deploy or use any existing ERC-20 that deducts a percentage fee on every `transferFrom` (e.g., 1 % fee).
2. Create a market with this token as `loanToken` via the permissionless `touchMarket`.
3. As an unprivileged taker, call `take` against any valid offer. Suppose `buyerAssets - sellerAssets = 1000`.
4. `claimableSettlementFee[loanToken] += 1000` (line 418).
5. `safeTransferFrom(loanToken, payer, address(this), 1000)` executes; the contract receives only 990 (1 % fee deducted by the token).
6. After the call: `claimableSettlementFee[loanToken] == 1000` but `IERC20(loanToken).balanceOf(address(midnight))` increased by only 990.
7. The gap grows with every subsequent `take`. Eventually `claimableSettlementFee > actual_balance_for_fees`.

**Why existing checks fail:**
- `SafeTransferLib` has no received-amount verification.
- `touchMarket` imposes no restriction on the loan token type.
- The Certora formal verification explicitly assumes "no fee taking from sender or receiver" as a precondition (`Solvency.spec` line 31), meaning this invariant is not enforced on-chain: [4](#0-3) 
- The `tokenBalanceCorrect` strong invariant in Certora would be violated under this scenario: [5](#0-4) 

### Impact Explanation
`claimableSettlementFee[loanToken]` records more tokens than the contract holds for that purpose. When the `feeClaimer` calls `claimSettlementFee`, the `safeTransfer` will revert once the cumulative shortfall exceeds the remaining balance, permanently locking or destroying accrued protocol revenue. The solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee` is violated for the affected token.

### Likelihood Explanation
Precondition: the loan token must charge a fee on `transferFrom`. Market creation is permissionless — any address can call `touchMarket` with any ERC-20. The bug is triggered on every `take` call in such a market, is fully repeatable, and requires no special privilege. The attacker need not even be adversarial; ordinary takers in a fee-token market will silently corrupt the accounting.

### Recommendation
After the `safeTransferFrom` on line 455, measure the actual received amount via a balance delta and use that value to update `claimableSettlementFee` instead of the nominal `buyerAssets - sellerAssets`:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 received = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
// Replace the earlier unconditional increment with:
claimableSettlementFee[offer.market.loanToken] += received; // was: += buyerAssets - sellerAssets
```

Alternatively, document and enforce (via `touchMarket`) that fee-on-transfer tokens are prohibited as loan tokens.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
// FeeToken: 1% fee on every transferFrom, deducted from recipient
contract FeeToken is ERC20 {
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount / 100;
        super.transferFrom(from, to, amount - fee); // recipient gets amount - fee
        _burn(from, fee);                           // fee destroyed (or sent to owner)
        return true;
    }
}

contract FeeOnTransferSettlementFeeTest is Test {
    Midnight midnight;
    FeeToken loanToken;

    function testFeeOnTransferBreaksSettlementFeeAccounting() public {
        // Setup: create market with fee-on-transfer loan token
        // ... (standard market/offer setup with loanToken = FeeToken) ...

        uint256 spread = buyerAssets - sellerAssets; // e.g., 1000

        uint256 balanceBefore = loanToken.balanceOf(address(midnight));
        midnight.take(offer, ratifierData, units, taker, receiver, address(0), "");
        uint256 balanceAfter = loanToken.balanceOf(address(midnight));

        uint256 actualReceived = balanceAfter - balanceBefore; // 990 (1% fee)
        uint256 recorded = midnight.claimableSettlementFee(address(loanToken)); // 1000

        // KEY ASSERTIONS:
        assertLt(actualReceived, spread, "fee-on-transfer reduced received amount");
        assertGt(recorded, actualReceived, "claimableSettlementFee exceeds actual balance");
        // Invariant violation:
        assertLt(
            loanToken.balanceOf(address(midnight)),
            midnight.claimableSettlementFee(address(loanToken)),
            "balance < claimableSettlementFee: solvency invariant broken"
        );
    }
}
```

Expected: the final `assertLt` passes, confirming `balance < claimableSettlementFee` and the solvency invariant is broken.

            }
