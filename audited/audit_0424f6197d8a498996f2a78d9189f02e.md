### Title
Zero-`buyerAssets` rounding allows unlimited reuse of `maxAssets`-capped buy offers — (`src/Midnight.sol`)

### Summary
When `offer.buy == true` and `offer.maxAssets > 0`, the consumed counter is incremented by `buyerAssets`, which is computed as `units.mulDivDown(buyerPrice, WAD)`. For any tick below `MAX_TICK`, `buyerPrice = tickToPrice(tick) < 1e18`, so a call with `units = 1` yields `buyerAssets = 0`. The consumed counter therefore never increases, and the `ConsumedAssets` cap is never triggered regardless of how many times the offer is taken. The codebase itself acknowledges this in `test/TakeTest.sol` via the test named `testBugBuyMaxAssetsBypass`.

### Finding Description

**Exact code path — `src/Midnight.sol` lines 358–369:**

```
offerPrice  = tickToPrice(offer.tick)          // rounds to multiples of 1e12
sellerPrice = offerPrice - _settlementFee      // for offer.buy == true
buyerPrice  = sellerPrice + _settlementFee     // == offerPrice
buyerAssets = units.mulDivDown(buyerPrice, WAD)// == 0 when units*buyerPrice < 1e18

if (offer.maxAssets > 0) {
    newConsumed = consumed[maker][group] += buyerAssets;  // += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets()); // always passes
}
``` [1](#0-0) 

For `offer.buy == true`, `buyerPrice == offerPrice == tickToPrice(offer.tick)`. Because `tickToPrice` rounds to multiples of `PRICE_ROUNDING_STEP = 1e12` and is strictly less than `WAD = 1e18` for every tick below `MAX_TICK`, the product `1 * buyerPrice` is always less than `WAD`, so `mulDivDown` truncates to zero. [2](#0-1) 

**Attacker inputs:** `offer.buy = true`, `offer.maxAssets > 0`, any `offer.tick < MAX_TICK` (i.e., the entire valid tick range except the single maximum tick), `units = 1`. No privileged access required; any address that is not the maker can be the taker.

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N` at any tick < `MAX_TICK`.
2. Taker calls `take(offer, ..., units=1, ...)` repeatedly.
3. Each call: `buyerAssets = 0`, `consumed += 0`, `newConsumed == consumed_before <= maxAssets` → no revert.
4. Each call still executes the full position update: maker's credit increases by 1 unit, taker's debt increases by 1 unit, zero tokens are transferred.
5. The offer is never consumed; the loop can run indefinitely.

**Why existing checks fail:** The only guard against over-filling is `require(newConsumed <= offer.maxAssets)`. Because `newConsumed` is unchanged when `buyerAssets = 0`, this check is trivially satisfied forever. There is no `require(units == 0 || buyerAssets > 0)` guard.

The in-repo test `testBugBuyMaxAssetsBypass` explicitly pre-sets `consumed = maxAssets` (fully consumed state) and then demonstrates that a `units=1` take at `tick = MAX_TICK - 16` still succeeds with `buyerAssets = 0`, `sellerAssets = 0`, while lender credit and borrower debt both increase. [3](#0-2) 

### Impact Explanation

The `maxAssets` budget is completely bypassed for any buy offer at any tick below `MAX_TICK`. The maker's credit position grows without bound (one unit per call) while `consumed` stays frozen at its pre-attack value. The taker accumulates unbounded debt without receiving any tokens. The core invariant "offers cannot be replayed, overfilled, or reused" is violated: a fully-consumed offer (consumed == maxAssets) remains indefinitely takeable.

### Likelihood Explanation

Preconditions are minimal and universally present: any buy offer with `maxAssets > 0` at any tick in `[0, MAX_TICK)` is affected. The only tick that is immune is `MAX_TICK` itself (where `tickToPrice = 1e18` exactly, giving `buyerAssets = 1` for `units = 1`). The attack requires no special permissions, no oracle manipulation, and no token with unusual behavior. It is repeatable in every block at negligible gas cost.

### Recommendation

Add a guard that prevents a non-zero `units` take from contributing zero to the consumed counter in assets mode:

```solidity
if (offer.maxAssets > 0) {
    uint256 delta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || delta > 0, ZeroAssetTake());
    newConsumed = consumed[offer.maker][offer.group] += delta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum tick or minimum `units` such that `buyerAssets >= 1` is guaranteed when `units > 0` and `maxAssets > 0`. A complementary fix is to fall back to incrementing by `units` (as in the `maxUnits` branch) when the computed asset delta rounds to zero.

### Proof of Concept

```solidity
function testConsumedNeverIncreasesOnZeroBuyerAssets() public {
    // Setup: buy offer at any tick < MAX_TICK with maxAssets cap
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 100;
    lenderOffer.tick = MAX_TICK - 16; // tickToPrice < WAD → buyerAssets = 0 for units=1

    collateralize(market, borrower, 1000);
    deal(address(loanToken), lender, 0); // lender pays nothing

    uint256 consumedBefore = midnight.consumed(lender, lenderOffer.group);

    // Take 1 unit — should increment consumed by at least 1
    (uint256 buyerAssets,) = take(1, borrower, lenderOffer);

    assertEq(buyerAssets, 0, "buyerAssets rounds to zero");
    // BUG: consumed did not increase
    assertEq(midnight.consumed(lender, lenderOffer.group), consumedBefore,
        "consumed unchanged despite non-zero units take");

    // Repeat 200 times — offer never exhausted despite maxAssets=100
    for (uint256 i; i < 200; i++) {
        take(1, borrower, lenderOffer);
    }
    assertLt(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets + 1,
        "consumed still at or below maxAssets after 200 takes");
    // Assert maker credit grew far beyond what maxAssets should allow
    assertGt(midnight.creditOf(id, lender), lenderOffer.maxAssets,
        "maker credit exceeds maxAssets budget");
}
```

Expected assertions: `buyerAssets == 0`, `consumed` unchanged after each take, maker credit exceeds `maxAssets` after repeated takes, no revert on any iteration.
