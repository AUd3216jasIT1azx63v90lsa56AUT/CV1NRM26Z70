### Title
Unbounded zero-unit take loop in bundle functions allows gas griefing via fully-consumed group front-run - (File: src/periphery/MidnightBundles.sol)

### Summary
The loop in `supplyCollateralAndSellWithUnitsTarget` (and the three sibling bundle functions) does not skip iterations where `unitsToTake == 0`. When `ConsumableUnitsLib.consumableUnits()` returns 0 for a fully-consumed group, `unitsToTake = min(targetUnits - filledUnits, takes[i].units, 0) = 0`, yet `try IMidnight.take(..., units=0, ...)` is still dispatched. Because `Midnight.take()` explicitly permits no-op takes on fully-consumed offers (confirmed by the inline comment at line 93), the call succeeds and returns `(0, 0)`, leaving `filledUnits` unchanged and the loop condition `filledUnits < targetUnits` perpetually true until all N offers are exhausted. An attacker who controls the maker address can front-run a taker's bundle transaction by calling `setConsumed(group, type(uint256).max, maker)`, converting every offer in the array into a zero-unit no-op and forcing the taker to pay O(N × per-take gas) before the transaction reverts with `OutOfOffers()`.

7. `SelfTake` check (line 354) — passes (maker ≠ taker).

9. **External call `IRatifier.isRatified()`** (line 356) — executes fully.
10. `buyerAssets = 0.mulDivDown(...) = 0`, `sellerAssets = 0`.
11. Consumed check: `consumed += 0 == maxUnits`; `require(maxUnits <= maxUnits)` — **passes**.
12. `_updatePosition` for buyer and/or seller if they hold credit — SLOADs + SSTOREs.
13. Position arithmetic (all `+= 0` / `-= 0`) — still touches storage slots.
14. `claimableSettlementFee += 0` — SLOAD + SSTORE.
15. `Take` event emitted.
16. Liquidation lock TSTORE.

19. Returns `(0, 0)`.

Back in the bundle: `filledUnits += 0`, loop condition still true, next iteration begins. After all N iterations, `require(filledUnits == targetUnits, OutOfOffers())` reverts.

**Root cause:** No `if (unitsToTake == 0) continue;` guard before the `try` call at line 152.

**Attacker inputs:**
- Maker creates N buy offers (`offer.buy = true`) all sharing the same `(maker, group)` pair with `maxUnits = X`.
- Maker calls `setConsumed(group, type(uint256).max, maker)` to exhaust the group (permissionless for the maker, line 723–728 of `Midnight.sol`).
- Alternatively, the maker front-runs the taker's pending transaction with this `setConsumed` call.
- Taker's `takes[]` array contains all N offers.

**Why existing checks fail:**
- The `try/catch` at line 152–160 is designed to tolerate asynchrony (e.g., an offer taken by someone else between block construction and execution). It silently swallows both reverts and zero-return successes, so it provides no protection.
- The loop condition `filledUnits < targetUnits` does not terminate early because `filledUnits` never advances when all takes return 0.
- `ConsumableUnitsLib.consumableUnits()` correctly returns 0, but the bundle does not act on this signal before dispatching the take.

### Impact Explanation
Every zero-unit take in `Midnight.take()` executes: one external ratifier call, up to two `_updatePosition` calls (each with multiple SLOADs/SSTOREs), two ERC-20 `transferFrom` calls with amount 0, one `isHealthy` call (potentially invoking an oracle), one event emission, and two TSTORE operations. Conservatively, each zero-unit take costs on the order of 30,000–80,000 gas depending on position state and oracle complexity. For N = 100 offers, the taker's transaction burns 3,000,000–8,000,000 gas before reverting with `OutOfOffers()`. The taker loses all gas; no funds are transferred because the transaction reverts. The attack is repeatable at the cost of one `setConsumed` call per front-run.

### Likelihood Explanation
**Preconditions:**
1. Attacker controls a maker address (any unprivileged user can be a maker).
2. Attacker creates N offers in a single group (no cost beyond gas).
3. Attacker calls `setConsumed(group, type(uint256).max, maker)` — one cheap transaction.
4. A taker (victim) calls a bundle function with those N offers in `takes[]`.

The front-run variant is straightforward on any chain with a public mempool. The social-engineering variant (listing N exhausted offers in an off-chain order book) requires no front-running at all. The attack is repeatable: the attacker can re-exhaust a group after any partial fill, or simply create new groups. Likelihood is medium-high given that bundle functions are designed for use with aggregators and order-book integrations that fetch offer lists off-chain.

### Recommendation
Add an explicit zero-unit skip guard immediately before each `try IMidnight.take()` call in all four bundle loop bodies:

```solidity
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
if (unitsToTake == 0) continue;   // ← add this guard
try IMidnight(MIDNIGHT).take(...) ...
```

This applies identically to `buyWithUnitsTargetAndWithdrawCollateral` (line 74–85), `supplyCollateralAndSellWithUnitsTarget` (line 147–160), `buyWithAssetsTargetAndWithdrawCollateral` (line 208–221), and `supplyCollateralAndSellWithAssetsTarget` (line 285–300).

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {MidnightBundles, Take} from "src/periphery/MidnightBundles.sol";
// ... standard test setup imports

contract ZeroUnitGasGriefTest is Test {
    Midnight midnight;
    MidnightBundles bundles;
    address maker = address(0xA);
    address taker = address(0xB);
    uint256 N = 50; // number of griefing offers

    function setUp() public {
        midnight = new Midnight();
        bundles = new MidnightBundles(address(midnight));
        // deploy mock loanToken, collateralToken, oracle, ratifier
        // fund taker with collateral; authorize bundles for taker
        // maker creates N buy offers all with group = keccak256("g"), maxUnits = 1000
    }

    function test_gasGriefZeroUnitLoop() public {
        // Step 1: exhaust the group as maker
        vm.prank(maker);
        midnight.setConsumed(keccak256("g"), type(uint256).max, maker);

        // Step 2: build takes[] with N offers from the exhausted group
        Take[] memory takes = new Take[](N);
        for (uint256 i; i < N; i++) {
            takes[i] = Take({units: 100, offer: buildOffer(maker, keccak256("g")), ratifierData: ""});
        }

        // Step 3: taker calls bundle — measure gas
        uint256 gasBefore = gasleft();
        vm.prank(taker);
        vm.expectRevert(IMidnightBundles.OutOfOffers.selector);
        bundles.supplyCollateralAndSellWithUnitsTarget(
            1,      // targetUnits > 0
            0,      // minSellerAssets
            taker,
            taker,
            new CollateralSupply[](0),
            takes,
            0,
            address(0)
        );
        uint256 gasUsed = gasBefore - gasleft();

        // Assertions:
        // 1. Transaction reverted with OutOfOffers
        // 2. gasUsed is proportional to N (assert gasUsed > N * MIN_GAS_PER_TAKE)
        assertGt(gasUsed, N * 20_000, "gas not proportional to N");

        // Contrast: with the fix (if unitsToTake == 0 continue), gasUsed should be
        // near-constant regardless of N. A fuzz test can sweep N from 1 to 200
        // and assert gasUsed grows linearly without the fix, and stays flat with it.
    }
}
```

**Expected assertions:**
- Transaction reverts with `OutOfOffers()` in all cases.
- `gasUsed` scales linearly with `N` (proving O(N) loop execution).
- With the `if (unitsToTake == 0) continue;` fix applied, `gasUsed` is O(1) in N (loop exits immediately after the first zero-unit check).
