### Title
Fee-on-Transfer Collateral Token Creates Phantom Collateral Enabling Undercollateralized Borrowing - (File: src/Midnight.sol)

### Summary
`supplyCollateral` credits `position[id][onBehalf].collateral[collateralIndex]` with the full caller-supplied `assets` value at line 533 before the external transfer at line 545, and `SafeTransferLib.safeTransferFrom` only validates the boolean return value — it never measures the actual received amount. Because `touchMarket` is permissionless and imposes no restriction on collateral token type, an attacker can create a market whose collateral token silently deducts a transfer fee, causing the protocol to record more collateral than it holds and enabling borrowing against phantom collateral.

### Finding Description
**Code path:**

`supplyCollateral` (src/Midnight.sol lines 524–546): [1](#0-0) 

Line 533 writes `oldCollateral + assets` into storage unconditionally. The transfer at line 545 then calls `SafeTransferLib.safeTransferFrom`: [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only checks `success` and the boolean return value — it performs no before/after balance measurement: [3](#0-2) 

**Market creation is fully permissionless.** `touchMarket` validates maturity, sorted collateral addresses, allowed LLTV tiers, and valid `maxLif` — but imposes **no restriction on the collateral token's transfer behavior**: [4](#0-3) 

The protocol documents the requirement that tokens must transfer exactly the requested amount, but this is a comment-level assumption with no on-chain enforcement: [5](#0-4) 

**Exploit flow (fee_rate = f, LLTV = 0.77, oracle_price = 1):**

1. Attacker deploys a fee-on-transfer ERC20 with fee rate `f` (e.g., 30%).
2. Attacker calls `touchMarket` with this token as the sole collateral — succeeds because no token-type check exists.
3. Attacker calls `supplyCollateral(market, 0, 1e18, attacker)`:
   - Line 533 sets `position.collateral[0] = 1e18` (phantom collateral).
   - Line 545 transfers `1e18` from attacker; protocol actually receives `0.70e18`.
4. Attacker takes a buy offer (borrows loan tokens). `isHealthy` evaluates against the recorded `1e18`, allowing borrowing up to `0.77e18` loan tokens — `0.07e18` more than the actual `0.70e18` collateral supports.
5. Attacker keeps the loan tokens and does not repay.
6. Liquidator attempts to seize `1e18` collateral; `safeTransfer` at `withdrawCollateral` line 572 reverts because the protocol only holds `0.70e18`. The liquidator can only seize `0.70e18`, leaving `0.30e18` phantom collateral in the position. The remaining debt becomes irrecoverable bad debt socialized to lenders.

**Why existing checks fail:** The only checks in `supplyCollateral` are authorization (line 527) and the collateral bitmap cap (lines 538–540). Neither prevents fee-on-transfer tokens. `SafeTransferLib` does not measure received amounts. [6](#0-5) 

### Impact Explanation
The invariant `position.collateral[i] <= collateralToken.balanceOf(midnight)` is broken for every slot backed by a fee-on-transfer token. The attacker borrows real loan tokens against phantom collateral. Liquidators cannot fully seize the phantom portion, creating bad debt that is socialized to lenders. With LLTV = 0.77 and fee = 30%, the attacker extracts `0.07e18` loan tokens per `1e18` nominal collateral supplied, with lenders bearing the loss.

### Likelihood Explanation
**Preconditions:** Attacker must deploy a fee-on-transfer ERC20 and create a market — both are permissionless, zero-cost on-chain operations. No privileged role is required. The attack is repeatable at any scale and requires no oracle manipulation or external cooperation. The only constraint is that lenders must supply loan tokens to the attacker's market; this can be achieved by offering attractive rates or by the attacker acting as both market creator and lender in a multi-step setup targeting other borrowers.

### Recommendation
Measure the actual received amount in `supplyCollateral` using a before/after balance check, and credit only the delta:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

This must be applied consistently to all token inflows (`repay`, `take` loan-token pulls) to maintain the full balance invariant. Alternatively, enforce the token safety requirement at market creation time by adding a whitelist or a transfer-amount verification probe in `touchMarket`.

        if (marketState[id].tickSpacing == 0) {

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
