### Title
Zero-asset `withdrawCollateral` unconditionally triggers oracle-calling health check, blocking no-op for borrowers with reverting oracle - (File: src/Midnight.sol)

### Summary
`withdrawCollateral` calls `isHealthy` unconditionally regardless of whether `assets == 0`. When `assets == 0`, no collateral state changes and no bitmap bit is cleared, yet the full oracle-querying health check still executes. A borrower with outstanding debt and any activated collateral whose oracle is reverting will have their zero-asset (no-op) withdrawal revert unnecessarily.

When `assets == 0`:
- Line 561: `newCollateral = oldCollateral - 0 = oldCollateral` — no change.
- Line 564: `assets > 0` is `false`, so the bitmap is **not** cleared.
- Line 568: `isHealthy` is called **unconditionally**.

When `debt > 0`, every bit in the bitmap triggers an external `oracle.price()` call. If any activated oracle reverts, `isHealthy` reverts, and `withdrawCollateral` reverts with the oracle's revert (not `UnhealthyBorrower()`).

**Exploit flow:**
1. Borrower takes on debt (debt > 0).
2. Borrower has an activated collateral at some index whose oracle begins reverting (e.g., Chainlink feed paused, oracle contract bricked).
3. Borrower calls `withdrawCollateral(market, anyIndex, 0, onBehalf, receiver)`.
4. `newCollateral = oldCollateral`, bitmap unchanged, `isHealthy` called.
5. `isHealthy` iterates bitmap, hits reverting oracle, propagates revert.
6. Call reverts — a complete no-op is blocked.

**Why existing checks don't stop it:** The bitmap-clear guard at line 564 only fires when `assets > 0`. There is no early-exit guard for `assets == 0` before the `isHealthy` call. The Certora spec `oracleRevertCausesWithdrawCollateralRevert` (Reverts.spec lines 196–208) formally proves this revert behavior for all values of `assets` including 0, confirming the path is reachable and the spec treats it as expected — but the spec does not distinguish the zero-asset case as a special no-op.

### Impact Explanation
A borrower with debt whose market has any activated collateral with a reverting oracle cannot execute a zero-asset withdrawal. The call reverts even though no state would change, no tokens would move, and the borrower's health ratio is unaffected. Any system or integration that calls `withdrawCollateral(..., 0, ...)` as a no-op (e.g., in a multicall to trigger accounting side-effects or as a defensive call) is blocked for the duration of the oracle outage.

### Likelihood Explanation
Oracle reverts are realistic: Chainlink feeds can be paused, oracle contracts can be upgraded or bricked, and sequencer downtime on L2s can cause oracle calls to revert. The preconditions — borrower has debt, one activated oracle reverts, caller passes `assets == 0` — are all independently reachable without any privileged action. The condition is repeatable for the entire duration of the oracle outage.

### Recommendation
Add an early return when `assets == 0`, before any state mutation or health check:

```solidity
function withdrawCollateral(..., uint256 assets, ...) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    if (assets == 0) return;   // no-op: skip state changes, bitmap update, and health check
    bytes32 id = touchMarket(market);
    ...
}
```

Alternatively, guard the `isHealthy` call: `if (assets > 0) require(isHealthy(market, id, onBehalf), UnhealthyBorrower());`. The early-return approach is cleaner and consistent with how `supplyCollateral` implicitly handles zero amounts.

    function withdrawCollateral(
        Market memory market,

        address onBehalf,

        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;

                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)

            }
        }

/// If an activated collateral oracle reverts on price different than withdrawn collateral, withdrawCollateral reverts when the borrower has debt.
rule oracleRevertCausesWithdrawCollateralRevert(env e, Midnight.Market market, uint256 collateralIndex, uint256 assets, address onBehalf, address receiver, uint256 revertingCollateralIndex) {

    require revertingCollateralIndex != collateralIndex, "withdrawCollateral may clear the bit at collateralIndex before calling isHealthy";

    bytes32 id = summaryToId(market);
    uint128 bitmap = collateralBitmap(id, onBehalf);
    require summaryGetBit(bitmap, revertingCollateralIndex), "revertingCollateralIndex is activated";

    withdrawCollateral@withrevert(e, market, collateralIndex, assets, onBehalf, receiver);
    bool reverted = lastReverted;

    assert debtOf(id, onBehalf) > 0 => reverted;
}
