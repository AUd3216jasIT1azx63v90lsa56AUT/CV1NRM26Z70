**Why the claimed exploit path is self-defeating:**

The liquidation check fires when `originalDebt > maxDebt`. For a borrower to have `debt > maxDebt_computed` at the current oracle prices, they must have been allowed to borrow that amount in the first place. Borrowing is gated by `isHealthy`, which is called after every `take` (line 476) and after `withdrawCollateral` (line 568): [4](#0-3) [5](#0-4) 

`isHealthy` uses the **same** `mulDivDown`-based formula to compute `maxDebt`. Therefore, after any valid borrow, the invariant `debt <= maxDebt_computed` holds by construction — the health check would have rejected a borrow that pushed `debt` above `maxDebt_computed`. At the same oracle prices, `liquidate` computes the identical `maxDebt_computed`, so `originalDebt > maxDebt` is false and `NotLiquidatable()` reverts.
