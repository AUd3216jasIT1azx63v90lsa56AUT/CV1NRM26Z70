Audit Report

## Title
Zero-asset free fill on low-price buy offers bypasses `maxAssets` cap and corrupts accounting - (`src/Midnight.sol`)

## Summary
In `Midnight.take()`, for a buy offer at any tick below `MAX_TICK`, `mulDivDown(units * buyerPrice, WAD)` truncates to zero when `units * buyerPrice < WAD`. When `maxAssets` mode is active, the consumed counter increments by zero on every such call, so the cap check never advances and the offer can be taken an unlimited number of times. Each iteration grants the maker (lender) one unit of credit and the taker (borrower) one unit of debt while transferring zero tokens and accruing zero settlement fee, permanently skewing the protocol's accounting invariant that every credit unit is backed by a deposited asset.

## Finding Description
**Root cause ‚Äî truncation to zero:**

In `src/Midnight.sol` lines 363‚Äì364, for a buy offer (`offer.buy = true`):
```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD);
```
`mulDivDown` truncates toward zero. For `units = 1` and any `buyerPrice < WAD` (i.e., any tick < `MAX_TICK = 5820`), `1 * buyerPrice / WAD = 0`. `TickLib.tickToPrice(2) = 1e12` and `tickToPrice(MAX_TICK - 2) = 1e18 - 1e12`, so the condition holds for virtually the entire tick range (5819 of 5820 ticks).

**Cap bypass ‚Äî consumed counter stalls:**

Lines 367‚Äì369:
```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```
`consumed += 0` on every call. `newConsumed` never advances past its pre-call value, so the cap check `newConsumed <= offer.maxAssets` passes unconditionally, even when the offer is already fully consumed.

**Position state mutated unconditionally:**

Lines 408‚Äì414 and 416‚Äì418 update `buyerPos.credit`, `sellerPos.debt`, `_marketState.totalUnits`, and `claimableSettlementFee` by `units = 1` and `buyerAssets - sellerAssets = 0` respectively, regardless of whether any assets moved.

**Zero token transfers:**

Lines 455‚Äì456:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // 0
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets); // 0
```

**Exploit flow:**
1. Attacker (lender/maker) creates a buy offer at any tick < `MAX_TICK` with `maxAssets = N`.
2. Taker (or an authorized operator) calls `Midnight.take(offer, ..., units=1, taker, ...)` in a loop.
3. Each iteration: `buyerAssets = 0`, `sellerAssets = 0`, `consumed` unchanged, maker gains 1 unit of credit, taker gains 1 unit of debt, zero tokens transferred, zero fee accrued.
4. Loop runs indefinitely; maker accumulates unbounded credit for free.

**Existing checks are insufficient:**
- The `require(newConsumed <= offer.maxAssets)` check at line 369 is bypassed because `newConsumed` never increases.
- There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in `take()`.
- The codebase explicitly acknowledges this at line 94: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*
- The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` lines 858‚Äì889 reproduces the exact scenario and confirms: `buyerAssets = 0`, `consumed` unchanged, lender credit increases, borrower debt increases, no token transfers.

## Impact Explanation
- **Direct theft of assets**: The lender accumulates credit (a claim on future repayments and withdrawals) without depositing any loan assets. When the lender later calls `withdraw`, they drain assets deposited by other lenders, constituting direct theft from the pool.
- **Offer cap bypass**: In `maxAssets` mode, the consumed counter never advances, so the offer is never exhausted regardless of how many times it is taken.
- **Solvency skew**: `totalUnits` grows without a matching increase in `withdrawable` or contract token balance, violating the core accounting invariant that every credit unit is backed by a deposited or repaid asset.
- **Zero settlement fee**: `claimableSettlementFee` is never incremented; the protocol collects nothing for these fills.
- **Severity: Critical** ‚Äî direct, unbounded theft of other lenders' deposited assets with no cost to the attacker.

## Likelihood Explanation
- Precondition: any buy offer at any tick < `MAX_TICK` (5819 of 5820 ticks qualify). Default tick spacing is 4, so the lowest accessible tick is 0 (price = 0) and the second-lowest is 4 (price = `1e12`).
- `units = 1` is always a valid input; no minimum-units check exists.
- The attacker only needs to be a normal lender (maker) and either be the taker themselves (via a separate address) or have any willing taker call `take()`.
- Repeatable indefinitely at negligible gas cost on L2 deployments.
- No privileged access required.

## Recommendation
Add a guard at the start of the asset-computation block to reject zero-asset fills when `units > 0`:
```solidity
require(units == 0 || buyerAssets > 0, ZeroAssetFill());
```
Alternatively, switch buy offers to use `mulDivUp` for `buyerAssets` (consistent with sell offers), ensuring that any non-zero `units` input always produces a non-zero asset amount. Additionally, consider adding a minimum-units check or a minimum-price check to prevent tick-0 offers from being used in `maxAssets` mode.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` at lines 858‚Äì889 is a complete, passing reproduction:
1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (price < WAD).
2. Pre-consume the offer to its cap via `midnight.setConsumed(lenderOffer.group, 1, lender)`.
3. Call `take(1, borrower, lenderOffer)`.
4. Assert: `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged at cap, lender credit increased, borrower debt increased, no token transfers.

        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:

        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
