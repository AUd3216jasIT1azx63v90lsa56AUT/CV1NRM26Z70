### Title
Sequential nonce allows attacker to invalidate multiple pre-signed `nonce==0` authorizations — (`src/periphery/EcrecoverAuthorizer.sol`)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` is fully permissionless: any caller can submit a valid `Authorization` + `Signature` pair. The nonce is a single global counter per authorizer, not per `authorized` address. If an authorizer pre-signs multiple `Authorization` structs all with `nonce==0` (e.g., to authorize several different addresses), an attacker who observes any one of them can submit it, advancing `nonce[authorizer]` to 1 and permanently invalidating all remaining `nonce==0` authorizations.

### Finding Description
The critical code path is in `src/periphery/EcrecoverAuthorizer.sol`:

```solidity
// line 24-26
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
``` [1](#0-0) 

There is no `msg.sender` restriction — the function is permissionless, as explicitly confirmed by the existing test `testEcrecoverAuthorizerPermissionless`. [2](#0-1) 

The `Authorization` struct binds `authorizer`, `authorized`, `isAuthorized`, `nonce`, and `deadline`. [3](#0-2) 

Because `nonce` is a single counter per `authorizer` (not per `authorized` address), an authorizer who pre-signs `N` authorizations all with `nonce==0` (for `N` different `authorized` addresses) produces `N` independently valid signatures. An attacker who observes any one of them can submit it, incrementing `nonce[authorizer]` to 1. The remaining `N-1` authorizations now fail the `InvalidNonce()` check and are permanently unusable without re-signing.

Existing checks do not stop this:
- The deadline check only prevents expired signatures, not front-running.
- The signature validity check only ensures the signer is the authorizer or an authorized delegate — it does not restrict who submits.
- The nonce check enforces sequential ordering but does not prevent a third party from consuming the nonce.

### Impact Explanation
All pre-signed `nonce==0` authorizations except the one the attacker chose to submit are permanently invalidated. The authorizer must re-sign each one with the new nonce. If the authorizer is offline, a smart contract, or a hardware wallet in a time-sensitive context, this constitutes a denial-of-service on delegated authorization. Any protocol flow that depends on those pre-signed authorizations (e.g., a relayer submitting them on behalf of a user) is broken.

### Likelihood Explanation
Preconditions are realistic: a user pre-signs multiple authorizations off-chain (e.g., to authorize several DeFi contracts) before any are submitted on-chain. All naturally share `nonce==0`. An attacker monitoring the mempool or receiving one signature out-of-band submits it first. The attack is repeatable: after the authorizer re-signs with `nonce==1`, the attacker can repeat the same griefing. No special privilege is required.

### Recommendation
Replace the sequential global nonce with an unordered nonce (bitmap or per-`authorized`-address nonce), so that consuming one authorization does not invalidate others. Alternatively, restrict submission to `msg.sender == authorization.authorizer` only, removing the permissionless relay property. A minimal fix:

```solidity
// per-(authorizer, authorized) nonce
mapping(address => mapping(address => uint256)) public nonce;
require(authorization.nonce == nonce[authorization.authorizer][authorization.authorized]++, InvalidNonce());
```

Or use a `usedNonces` bitmap (EIP-2612-style unordered nonces) so each nonce is independent.

### Proof of Concept
```solidity
function testNonceDesyncGriefing() public {
    // Authorizer pre-signs two nonce==0 authorizations for different addresses
    vm.prank(borrower);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

    Authorization memory auth1 = Authorization({
        authorizer: borrower, authorized: lender,
        isAuthorized: true, nonce: 0, deadline: block.timestamp + 1 days
    });
    Authorization memory auth2 = Authorization({
        authorizer: borrower, authorized: otherLender,
        isAuthorized: true, nonce: 0, deadline: block.timestamp + 1 days
    });
    Signature memory sig1 = signAuthorization(auth1, borrower);
    Signature memory sig2 = signAuthorization(auth2, borrower);

    // Attacker submits auth1 first
    vm.prank(address(0xdead)); // unprivileged attacker
    ecrecoverAuthorizer.setIsAuthorized(auth1, sig1);
    assertEq(ecrecoverAuthorizer.nonce(borrower), 1);

    // auth2 (nonce==0) is now invalid
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(auth2, sig2);

    // Assert: authorizer's second pre-signed auth is permanently invalidated
    assertEq(midnight.isAuthorized(borrower, otherLender), false);
}
```

Expected assertions: `auth2` reverts with `InvalidNonce`, `isAuthorized(borrower, otherLender)` remains `false`, confirming the invariant violation.
