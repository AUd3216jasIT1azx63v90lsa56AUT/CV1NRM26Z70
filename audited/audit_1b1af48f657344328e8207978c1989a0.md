### Title
Double `mulDivDown` in `isHealthy` rounds `maxDebt` below exact value, blocking valid `withdrawCollateral` - (`src/Midnight.sol`)

### Summary
`isHealthy` computes `maxDebt` with two chained floor-rounding operations: `collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)`. Because `UtilsLib.mulDivDown` is a plain `(x * y) / d` with no 512-bit intermediate, the double truncation can produce a result that is 1 less than the exact value `floor(collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD))`. A position whose debt equals the exact `maxDebt` is therefore reported as unhealthy, and `withdrawCollateral` reverts with `UnhealthyBorrower()`, permanently freezing the collateral.

### Finding Description

**Root cause — `UtilsLib.mulDivDown` is plain integer division:**

```solidity
// src/libraries/UtilsLib.sol:29-31
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
```

No 512-bit intermediate; every call truncates independently. [1](#0-0) 

**Double truncation in `isHealthy`:**

```solidity
// src/Midnight.sol:954-955
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

Let `q1 = floor(C * P / S)` (first call, `S = ORACLE_PRICE_SCALE`). The exact intermediate is `C * P / S`; the truncation discards a remainder `r1 ∈ [0, S-1]`. The second call then computes `floor(q1 * L / W)` (`L = lltv`, `W = WAD`). Because `q1 ≤ C*P/S`, the second result satisfies:

```
computed_maxDebt ≤ floor(C * P * L / (S * W))  =  exact_maxDebt
```

The gap is at most 1 (since `L ≤ W`, the propagated error `r1 * L / (S * W) < 1`). So `computed_maxDebt` can equal `exact_maxDebt - 1`. [2](#0-1) 

**`withdrawCollateral` enforces `isHealthy` after reducing collateral:**

```solidity
// src/Midnight.sol:561-568
uint256 newCollateral = _position.collateral[collateralIndex] - assets;
_position.collateral[collateralIndex] = UtilsLib.toUint128(newCollateral);
...
require(isHealthy(market, id, onBehalf), UnhealthyBorrower());
```

The collateral is already written before the check; if `isHealthy` returns false the call reverts, but the state is rolled back — the collateral is not lost, but the withdrawal is permanently blocked as long as the position stays at this boundary. [3](#0-2) 

**Concrete exploit path (allowed LLTV = 0.77e18):**

| Variable | Value |
|---|---|
| `lltv` | `0.77e18` (allowed by `isLltvAllowed`) |
| `price` | `ORACLE_PRICE_SCALE / 77 + 1` |
| `collateral` (initial) | `200` |
| `debt` | `1` |

1. With `collateral = 200`: `q1 = floor(200*(S/77+1)/S) = 2`; `q2 = floor(2*0.77e18/1e18) = 1`. Health check passes; borrower takes `debt = 1`.
2. Borrower (or authorized operator via `EcrecoverAuthorizer`) calls `withdrawCollateral` with `assets = 100`, leaving `collateral = 100`.
3. `isHealthy` recomputes: `q1 = floor(100*(S/77+1)/S) = 1`; `q2 = floor(1*0.77e18/1e18) = 0`. Returns `false` (0 < 1).
4. Exact value: `floor(100*(S/77+1)*0.77e18/(S*1e18)) = floor(1 + tiny) = 1 ≥ 1`. Position is truly healthy.
5. `withdrawCollateral` reverts with `UnhealthyBorrower()`.

**Authorization path via `EcrecoverAuthorizer`:**

`EcrecoverAuthorizer.setIsAuthorized` verifies an EIP-712 signature from the borrower and calls `IMidnight(MIDNIGHT).setIsAuthorized(authorized, true, authorizer)`, setting `isAuthorized[borrower][operator] = true`. The operator then calls `withdrawCollateral(market, 0, 100, borrower, receiver)`. The authorization check at line 556 passes; the revert comes solely from the `isHealthy` double-rounding. [4](#0-3) [5](#0-4) 

### Impact Explanation
A borrower (or their authorized operator) whose position sits exactly at the true health boundary cannot withdraw any collateral that would leave `computed_maxDebt < debt`, even though `exact_maxDebt ≥ debt`. The collateral is not seized — it remains in the contract — but the withdrawal is blocked indefinitely unless the oracle price rises or the borrower repays debt. This constitutes a fund freeze / valid withdrawal blocked.

### Likelihood Explanation
The condition requires `debt = exact_maxDebt` and `computed_maxDebt = exact_maxDebt - 1`. This is achievable with small token amounts (1 unit of debt) and specific `(collateral, price)` pairs that maximize the double-rounding loss. It is deterministic and repeatable: any borrower who borrows the maximum allowed amount and then attempts to withdraw collateral down to the exact health boundary will hit this. The preconditions (authorization granted, position at exact boundary) are reachable by any unprivileged user.

### Recommendation
Replace the two-step `mulDivDown` chain with a single operation over the combined denominator. Since `price * lltv` fits in `uint256` under the existing overflow bounds (`price ≤ max_uint256 / max_uint128` and `lltv ≤ WAD = 1e18`), but `collateral * price * lltv` may overflow, introduce a 512-bit `mulDiv` (e.g., Solmate's full-precision variant) and compute:

```solidity
maxDebt += mulDiv512Down(collateral, price * lltv, ORACLE_PRICE_SCALE * WAD);
```

This eliminates the intermediate truncation entirely. Alternatively, if 512-bit arithmetic is undesirable, document the 1-unit rounding slack and adjust the health check to `maxDebt + 1 >= debt` (i.e., allow a 1-unit tolerance), accepting that positions can be 1 unit over the strict boundary.

### Proof of Concept

```solidity
// Foundry unit test
function testDoubleRoundingBlocksWithdrawCollateral() public {
    // lltv = 0.77e18 (allowed), ORACLE_PRICE_SCALE = 1e36, WAD = 1e18
    uint256 S = 1e36;
    uint256 price = S / 77 + 1;
    oracle1.setPrice(price);

    // Supply 200 units of collateral; computed_maxDebt(200) = 1
    deal(address(collateralToken1), borrower, 200);
    vm.prank(borrower);
    collateralToken1.approve(address(midnight), 200);
    vm.prank(borrower);
    midnight.supplyCollateral(market, 0, 200, borrower);

    // Borrow 1 unit (health check passes: computed_maxDebt = 1 >= 1)
    setupMarket(market, 1); // lender supplies 1 unit

    // Authorize operator via EcrecoverAuthorizer
    vm.prank(borrower);
    midnight.setIsAuthorized(address(this), true, borrower);

    // Attempt to withdraw 100 collateral (leaving 100)
    // exact_maxDebt(100) = 1 >= 1 = debt  → truly healthy
    // computed_maxDebt(100) = 0 < 1 = debt → isHealthy returns false
    vm.expectRevert(IMidnight.UnhealthyBorrower.selector);
    midnight.withdrawCollateral(market, 0, 100, borrower, borrower);

    // Assert: exact arithmetic says healthy
    // floor(100 * (S/77+1) / S) = 1; floor(1 * 0.77e18 / 1e18) = 0  → BUG
    assertEq(
        uint256(100) * price / S * uint256(0.77e18) / 1e18,
        0,  // computed_maxDebt
        "computed_maxDebt is 0"
    );
    // exact: floor(100 * price * 0.77e18 / (S * 1e18)) = 1
    // (100 * price = 100*(S/77+1) = 100*S/77+100; *0.77e18/(S*1e18) ≈ 1)
}
```

Expected assertions: `vm.expectRevert(UnhealthyBorrower)` passes (withdrawal is blocked), while the exact arithmetic check confirms `exact_maxDebt = 1 ≥ debt = 1`.
