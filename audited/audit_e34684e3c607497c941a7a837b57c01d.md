Audit Report

## Title
`maxAssets` Cap Bypass via Zero `buyerAssets` Rounding on Buy Offers with `buyerPrice < WAD` - (File: `src/Midnight.sol`)

## Summary
When `offer.buy = true` and `offer.maxAssets > 0`, the `take` function computes `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `tickToPrice(tick) < WAD` for every tick below `MAX_TICK`, a fill with `units = 1` yields `buyerAssets = 0`, leaving `consumed[maker][group]` unchanged while still increasing `buyerPos.credit`, `sellerPos.debt`, and `totalUnits`. The `maxAssets` cap is therefore entirely ineffective across the practical price range, and lender credit can be minted with zero token deposit.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

`buyerAssets` is computed with `mulDivDown`, which floors the result: [1](#0-0) 

When `units = 1` and `buyerPrice < WAD` (i.e., any tick below `MAX_TICK`), `floor(1 * buyerPrice / WAD) = 0`. The cap check then becomes: [2](#0-1) 

`consumed` is incremented by 0, so `newConsumed` never exceeds `offer.maxAssets`, and the `require` trivially passes — even when the offer was already fully consumed before the call.

**Price bound confirmed by Certora:**

`tickToPriceAtMostWad` proves `tickToPrice(tick) <= WAD` for all ticks, and `tickToPriceIsOneAtMaxTick` proves equality holds only at `MAX_TICK = 5820`: [3](#0-2) 

**Position state still mutates despite zero asset transfer:**

`buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` are updated based on `units`, not `buyerAssets`: [4](#0-3) 

The token transfer is `buyerAssets - sellerAssets = 0 - 0 = 0`: [5](#0-4) 

**Protocol acknowledgment:**

Line 94 explicitly documents this behavior as a known limitation: [6](#0-5) 

**Existing guard failure:**

`require(offer.maker != taker, SelfTake())` blocks self-takes but not two coordinated addresses: [7](#0-6) 

## Impact Explanation

An attacker controlling a maker address and a separate taker address can mint unbounded `buyerPos.credit` without depositing any loan tokens. At maturity, this credit is redeemable against the protocol pool, draining funds from legitimate lenders. The `maxAssets` cap — the maker's primary mechanism to bound total buyer-side commitment — is rendered entirely inoperative for every offer at a non-par price, which is the entire practical price range. Additionally, `claimableSettlementFee` accounting is corrupted: `buyerAssets - sellerAssets = 0` is added per fill despite real unit-level exposure being created. [8](#0-7) 

## Likelihood Explanation

Required preconditions are trivially satisfied: `offer.buy = true`, `offer.maxAssets > 0`, and `offer.tick < MAX_TICK` describe the entire practical price range. The attacker needs only two EOAs (or one EOA and one contract). The taker must have sufficient collateral for the debt incurred, which is a standard protocol precondition. The attack is repeatable indefinitely within a single block via `multicall`.

## Recommendation

Replace `mulDivDown` with `mulDivUp` for `buyerAssets` when `offer.buy = true`, or add an explicit guard rejecting fills where `buyerAssets == 0` and `offer.maxAssets > 0`:

```solidity
// Option A: round up buyerAssets for buy offers
uint256 buyerAssets = offer.buy
    ? units.mulDivUp(buyerPrice, WAD)
    : units.mulDivUp(buyerPrice, WAD);

// Option B: explicit guard
if (offer.maxAssets > 0 && offer.buy) {
    require(buyerAssets > 0, ZeroBuyerAssets());
}
```

Option A changes rounding direction for the buyer (maker), which may conflict with the `makerFavorableRounding` Certora property. Option B is a targeted guard that preserves existing rounding semantics while closing the bypass.

## Proof of Concept

The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` is a complete, self-contained reproduction: [9](#0-8) 

It sets `maxAssets = 1`, pre-sets `consumed = maxAssets` (fully consumed), takes with `units = 1` at `tick = MAX_TICK - 16`, and asserts:
- `buyerAssets == 0` and `sellerAssets == 0`
- `consumed` is unchanged (still equals `maxAssets`)
- `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased
- Token balances of both parties are unchanged
