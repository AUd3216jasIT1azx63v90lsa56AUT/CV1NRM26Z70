Audit Report

## Title
Dust-collateral rounding in `isHealthy` and `liquidate` enables liquidation of economically healthy positions - (File: src/Midnight.sol)

## Summary
When a position holds exactly 1 wei of collateral, the two chained `mulDivDown` calls used to compute `maxDebt` can round the result to 0 even though the position's true collateral value exceeds its debt. Because `liquidate` uses the identical rounding path and checks `originalDebt > maxDebt`, a position that is genuinely overcollateralized in real arithmetic becomes liquidatable on-chain, violating the core invariant that healthy positions cannot be liquidated.

## Finding Description
**Root cause â€” chained integer division in `isHealthy` and `liquidate`:**

`isHealthy` at `src/Midnight.sol:954â€“955`: [1](#0-0) 

`liquidate` at `src/Midnight.sol:613`: [2](#0-1) 

Both use `collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)`. With `ORACLE_PRICE_SCALE = 1e36` and `WAD = 1e18`: [3](#0-2) 

When `collateral[i] = 1` wei, the first division yields a small integer (e.g., 1 or 2), and the second division by `WAD` floors it to 0 for any `lltv < WAD`.

**Exploit path (LLTV_2 = 0.77e18):**

1. Borrower deposits collateral, borrows 1 unit of debt, then calls `withdrawCollateral` with `assets = initial_collateral âˆ’ 1`, leaving 1 wei.
2. `withdrawCollateral` only clears the bitmap bit when `newCollateral == 0`, so the 1-wei position remains tracked: [4](#0-3) 
3. At withdrawal time with price = 2e36: `mulDivDown(1, 2e36, 1e36) = 2`; `mulDivDown(2, 0.77e18, 1e18) = 1`. `maxDebt = 1 >= debt = 1` â†’ `isHealthy` returns true â†’ withdrawal succeeds.
4. Oracle price drops to 1.5e36 (a ~25% drop, a routine market event): `mulDivDown(1, 1.5e36, 1e36) = 1`; `mulDivDown(1, 0.77e18, 1e18) = 0`. `maxDebt = 0`.
5. Liquidator calls `liquidate`. The check `originalDebt (1) > maxDebt (0)` passes: [5](#0-4) 
6. True economic health: collateral value = 1 Ã— 1.5e36 / 1e36 = 1.5 units; true max debt = 1.5 Ã— 0.77 = 1.155 > 1. The position IS overcollateralized in real arithmetic.

**Why existing checks fail:** `withdrawCollateral` calls `isHealthy` only at withdrawal time (price = 2e36), which passes. There is no minimum-collateral floor, no dust guard, and no check preventing a subsequent price drop from triggering rounding collapse. The `liquidate` path recomputes `maxDebt` with the same two-step `mulDivDown`, so the same rounding error recurs unconditionally.

## Impact Explanation
A liquidator seizes 1 wei of collateral worth 1.5 loan-token units by repaying 1 unit of debt, netting ~0.5 loan-token units of profit at the borrower's expense. The borrower loses collateral that the protocol's own health formula should have protected. The invariant "healthy positions are not liquidatable" is concretely violated: `collateral_value Ã— lltv > debt` holds in real arithmetic but fails on-chain due to integer rounding on dust collateral. This constitutes direct theft of user assets caused by a protocol accounting bug.

## Likelihood Explanation
**Required preconditions:**
1. Borrower reduces collateral to 1 wei via `withdrawCollateral` â€” reachable by any borrower calling `withdrawCollateral(market, collateralIndex, initial_collateral âˆ’ 1, ...)`.
2. Oracle price at withdrawal â‰¥ `ceil(WAD / lltv) Ã— ORACLE_PRICE_SCALE` (â‰¥ 2e36 for LLTV_2).
3. Oracle price later drops into `(ORACLE_PRICE_SCALE / lltv, ceil(WAD/lltv) Ã— ORACLE_PRICE_SCALE)` â€” a ~25â€“33% drop depending on LLTV tier, a normal market event.

No privileged access is required. The vulnerability applies to all LLTV tiers except LLTV_8 = 1e18 (where `mulDivDown(x, 1e18, 1e18) = x`, avoiding the collapse). The dust state can be created deliberately by a borrower or arise from partial withdrawals. The liquidator is any unprivileged external account. [6](#0-5) 

## Recommendation
**Option 1 (preferred):** Enforce a minimum collateral amount per position slot â€” reject `withdrawCollateral` calls that would leave a nonzero collateral below a protocol-defined dust threshold (e.g., require `newCollateral == 0 || newCollateral >= MIN_COLLATERAL`).

**Option 2:** Replace the two-step `mulDivDown` with a single combined multiplication before dividing: `collateral[i].mulDiv(price * lltv, ORACLE_PRICE_SCALE * WAD)`. This eliminates the intermediate rounding step and computes the correct floor of the full product in one operation.

**Option 3:** Add a dust guard in `withdrawCollateral` that rejects withdrawals leaving a collateral amount whose `mulDivDown(price, ORACLE_PRICE_SCALE)` result is 0 at current oracle price.
