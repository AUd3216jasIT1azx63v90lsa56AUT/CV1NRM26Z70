Audit Report

## Title
`sellerAssetsToUnits` reverts on division-by-zero when `offerPrice == settlementFee`, causing uncaught revert in `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/TakeAmountsLib.sol)

## Summary
When `offer.buy = true` and `offerPrice == settlementFee`, `sellerAssetsToUnits` computes `sellerPrice = 0` and calls `mulDivUp(targetSellerAssets, WAD, 0)`. Because `mulDivUp` evaluates `d - 1` before dividing, Solidity 0.8 checked arithmetic underflows on `0 - 1` and reverts. This revert occurs outside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, causing the entire bundler call to revert rather than skipping the problematic offer.

## Finding Description

**Root cause — `sellerAssetsToUnits`:**

In `src/periphery/TakeAmountsLib.sol` lines 44–46:
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...
```
When `offer.buy = true` and `offerPrice == settlementFee`, `sellerPrice = 0`, and the call becomes `mulDivUp(targetSellerAssets, WAD, 0)`. [1](#0-0) 

**`mulDivUp` implementation:**

In `src/libraries/UtilsLib.sol` line 35:
```solidity
return (x * y + (d - 1)) / d;
```
With `d = 0`, the sub-expression `d - 1` = `0 - 1` underflows under Solidity 0.8 checked arithmetic, reverting with `stdError.arithmeticError` before any division occurs. [2](#0-1) 

**Test suite confirms this exact behavior:**

`test/UtilsLibTest.sol` lines 80–84 explicitly documents and tests this revert path with the comment `// because there is d-1`:
```solidity
function testMulDivUpDivisionByZero(uint256 x, uint256 y) public {
    // because there is d-1.
    vm.expectRevert(stdError.arithmeticError);
    this.mulDivUp(x, y, 0);
}
``` [3](#0-2) 

**Uncaught revert path in `supplyCollateralAndSellWithAssetsTarget`:**

In `src/periphery/MidnightBundles.sol` lines 285–300, `TakeAmountsLib.sellerAssetsToUnits(...)` is evaluated as an argument to `min(...)` at line 286, **before** the `try` block at line 292. The `try/catch` only wraps `IMidnight(MIDNIGHT).take(...)`. The revert from `sellerAssetsToUnits` propagates uncaught, reverting the entire function. [4](#0-3) 

**`sellerAssetsToUnits` NatDoc gap:**

The function's NatDoc at line 34 states: *"Reverts if offerPrice < settlementFee in case of a buy offer"* — the equality case (`offerPrice == settlementFee`) is not documented as a revert condition, yet it triggers one. [5](#0-4) 

**Contrast with `buyerAssetsToUnits`:**

`buyerAssetsToUnits` is not affected: when `sellerPrice = 0`, it computes `buyerPrice = 0 + settlementFee = settlementFee > 0` and calls `mulDivUp(targetBuyerAssets, WAD, settlementFee)`, which is safe. [6](#0-5) 

## Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `offerPrice == settlementFee` reverts entirely. Collateral already supplied in the same call (lines 269–275) is locked in the taker's position, but the seller assets are never received. Borrowers relying on this bundler function to sell units and service their debt are blocked for the duration the condition holds. If the settlement fee is raised to match multiple outstanding offers simultaneously, the bundler is bricked for all of them at once. The impact is a targeted, repeatable DoS on a core borrower-facing bundler function. [7](#0-6) 

## Likelihood Explanation

The condition `offerPrice == settlementFee` is reachable in two ways: (1) a maker deliberately places a buy offer at the tick whose price equals the current settlement fee — tick prices and settlement fee values are both discrete, making exact equality achievable; (2) the `feeSetter` admin raises the settlement fee post-offer-creation to a value that coincidentally matches an existing offer's price, a normal operational action. The condition is transient (it resolves as time-to-maturity shifts the interpolated fee), but it can be re-triggered repeatedly. No special privilege is required for path (1); any unprivileged maker can create such an offer.

## Recommendation

Add a zero-price guard in `sellerAssetsToUnits`. When `sellerPrice == 0`, the seller receives zero assets for any number of units — the function should either revert with a descriptive error (consistent with the `offerPrice < settlementFee` revert) or return a sentinel value. The simplest fix mirrors the existing `buyerAssetsToUnits` pattern and adds an explicit check:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
require(sellerPrice > 0 || !offer.buy, ZeroSellerPrice());
```

Alternatively, `supplyCollateralAndSellWithAssetsTarget` could wrap the `sellerAssetsToUnits` call in a `try/catch`-equivalent (e.g., a static-call wrapper) so that a zero-price offer is skipped rather than reverting the entire bundler call, consistent with the function's stated intent to skip problematic offers.

## Proof of Concept

Minimal Foundry test plan:

1. Deploy `MidnightBundles` against a mock `IMidnight` that returns `settlementFee = P` for a given market/time.
2. Create a buy offer at tick `T` where `TickLib.tickToPrice(T) == P`.
3. Call `supplyCollateralAndSellWithAssetsTarget` with `takes = [Take(offer, ...)]` and any nonzero `targetSellerAssets`.
4. Observe revert with `stdError.arithmeticError` (not `OutOfOffers`), confirming the revert originates from `mulDivUp` inside `sellerAssetsToUnits` before the `try` block is reached.
5. Confirm that calling `supplyCollateralAndSellWithUnitsTarget` (which uses `ConsumableUnitsLib.consumableUnits` → `buyerAssetsToUnits`, not `sellerAssetsToUnits`) with the same offer does **not** revert, isolating the root cause to `sellerAssetsToUnits`.
