### Title
Continuous Fee Accrual Rounds to Zero via Permissionless `updatePosition` Griefing — (`src/Midnight.sol`)

### Summary

`updatePosition` is a permissionless external function that resets `_position.lastAccrual` to `block.timestamp` for any user. The continuous fee accrual formula uses integer division that rounds down. A lender can call `updatePosition` on their own position every block, keeping the elapsed time interval tiny, causing the accrued fee to truncate to zero on every call. The protocol fee claimer permanently loses continuous fee revenue.

### Finding Description

**Vulnerability type**: Accounting / Rounding (DoS on fee accrual)

The fee accrual formula in `updatePositionView` is:

```solidity
uint128 fee = _lastAccrual < market.maturity
    ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
    : 0;
``` [1](#0-0) 

`mulDivDown` is plain integer floor division:

```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
``` [2](#0-1) 

`_updatePosition` (called by the public `updatePosition`) unconditionally writes `block.timestamp` into `lastAccrual`:

```solidity
_position.lastAccrual = uint128(block.timestamp);
``` [3](#0-2) 

`updatePosition` has **no authorization check** — any caller can trigger it for any user:

```solidity
function updatePosition(Market memory market, address user) external returns (uint128, uint128, uint128) {
    bytes32 id = toId(market);
    require(marketState[id].tickSpacing > 0, MarketNotCreated());
    return _updatePosition(market, id, user);
}
``` [4](#0-3) 

**Rounding condition**: For `fee` to truncate to zero:

```
postSlashPendingFee * (accrualEnd - lastAccrual) < (maturity - lastAccrual)
```

With a 12-second block interval and 365 days remaining, this requires:

```
postSlashPendingFee < 31,536,000 / 12 = 2,628,000 units
```

`pendingFee` is set at `take()` time:

```solidity
uint128 buyerPendingFeeIncrease =
    UtilsLib.toUint128(buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD));
``` [5](#0-4) 

`MAX_CONTINUOUS_FEE = 0.01e18 / 365 days ≈ 317` (WAD/second): [6](#0-5) 

For a USDC market (6 decimals), a lender with 1,000 USDC of credit (`credit = 1e9` raw units):

```
pendingFee = 1e9 * 317 * 31,536,000 / 1e18 ≈ 9,997 units
fee per 12s = 9,997 * 12 / 31,536,000 ≈ 0.0038 → truncates to 0
```

The threshold below which fees always round to zero (12-second blocks, 365-day market) is `credit < 2.628e14` raw units. For USDC this equals **$262 million** — meaning every realistic USDC position is affected. The lender calls `updatePosition` on their own position once per block, `lastAccrual` is refreshed, and `continuousFeeCredit` never increases. [7](#0-6) 

### Impact Explanation

The fee claimer permanently receives zero continuous fee revenue from affected positions. The lender retains the full `pendingFee` amount that should have been transferred to `continuousFeeCredit`. For a USDC market with $10 million TVL at 1% annual fee, the annual loss to the protocol is ~$100,000. The lender's `credit` is never reduced by the fee, so they also withdraw more at maturity than they should.

### Likelihood Explanation

The attack is self-beneficial: the lender saves fees equal to the continuous fee rate on their position. On L2 networks (Optimism, Base, Arbitrum) with 2-second blocks and gas costs of fractions of a cent, the cost of calling `updatePosition` once per block is negligible compared to the fees avoided on any position above a few hundred dollars. No privileged access is required. The attacker is the lender themselves, acting on a normal protocol flow.

### Recommendation

Introduce a minimum accrual interval (e.g., 1 hour or 1 day) before `_updatePosition` resets `lastAccrual`, or restructure the fee formula to accumulate a running `feePerUnit` index (similar to Synthetix staking rewards) so that short-interval calls cannot cause rounding loss. Alternatively, round the fee calculation **up** (`mulDivUp`) so that even a 1-second interval accrues at least 1 unit of fee when any pending fee exists.

### Proof of Concept

1. Deploy a USDC market with `continuousFee = MAX_CONTINUOUS_FEE`, maturity = 365 days from now.
2. Lender Alice buys 1,000 USDC of credit via `take()`. Her `pendingFee ≈ 9,997` units, `lastAccrual = block.timestamp`.
3. Advance 12 seconds (1 Ethereum block).
4. Alice calls `updatePosition(market, alice)`.
   - `fee = 9,997 * 12 / 31,536,000 = 0` (truncated).
   - `continuousFeeCredit` does not increase. `lastAccrual` is reset to current timestamp.
5. Repeat step 3–4 indefinitely.
6. After 365 days, `continuousFeeCredit` remains 0. Alice withdraws her full `credit` with zero fee paid. The fee claimer receives nothing.
