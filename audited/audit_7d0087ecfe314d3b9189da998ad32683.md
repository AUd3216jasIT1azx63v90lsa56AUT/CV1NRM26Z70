### Title
Fee-on-transfer loan token causes seller underpayment and `claimableSettlementFee` overcount in `take()` - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` before the two `safeTransferFrom` calls execute. Because `SafeTransferLib.safeTransferFrom` performs no balance-before/after check, a fee-on-transfer loan token silently delivers less than the requested amount to both `address(this)` and `receiver`. The result is that the seller receives `sellerAssets * (1-F)` instead of `sellerAssets`, and `claimableSettlementFee` is overcounted by `F * (buyerAssets - sellerAssets)`, breaking the solvency invariant the protocol relies on.

### Finding Description

**Permissionless market creation with arbitrary loan token.**
`touchMarket` is callable by anyone and imposes no restriction on `market.loanToken` beyond collateral-param validation. [1](#0-0) 
The `live_context.json` explicitly lists "permissionless market creation" and "arbitrary loan token" as design properties, so a fee-on-transfer token is a reachable precondition without any privileged action.

**State write before transfer.**
Inside `take()`, the settlement-fee accounting is updated at line 418 before any token movement: [2](#0-1) 

The two inbound transfers happen later: [3](#0-2) 

**No balance check in `SafeTransferLib`.**
`safeTransferFrom` calls `transferFrom` and only checks the boolean return value. It does not compare balances before and after the call, so a fee silently reduces the received amount without reverting. [4](#0-3) 

**Exploit flow (fee rate F, 0 < F < 1):**

1. Unprivileged market creator calls `touchMarket` with a fee-on-transfer ERC20 as `loanToken`.
2. Lender and borrower interact normally; taker calls `take(offer, ..., units, ...)`.
3. At line 418: `claimableSettlementFee[loanToken] += (buyerAssets - sellerAssets)` — full nominal amount recorded.
4. At line 455: `safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets)` — contract actually receives `(buyerAssets - sellerAssets) * (1-F)`.
5. At line 456: `safeTransferFrom(loanToken, payer, receiver, sellerAssets)` — receiver actually receives `sellerAssets * (1-F)`.

**Why existing checks do not stop it.**
- No token whitelist or fee-on-transfer guard in `touchMarket` or `take`.
- `SafeTransferLib` only checks `transferFrom` success, not received amount.
- The Certora `pendingFeeReceiptZero` and `tokenBalanceCorrect` invariants are proved against a standard ERC20 model; they are not verified against fee-on-transfer tokens. [5](#0-4) 

### Impact Explanation

**Seller underpaid:** The seller's `receiver` address receives `sellerAssets * (1-F)` loan tokens instead of the contractually expected `sellerAssets`. This is a direct, repeatable loss of user funds proportional to the token's fee rate.

**Settlement fee overcount:** `claimableSettlementFee[loanToken]` records `buyerAssets - sellerAssets` but the contract only holds `(buyerAssets - sellerAssets) * (1-F)` for that purpose. The gap `F * (buyerAssets - sellerAssets)` is a phantom liability. When `claimSettlementFee` is later called for the full recorded amount, the transfer either reverts (fee claimer cannot collect) or, if the contract holds surplus tokens from other users (lenders' `withdrawable`), drains those tokens — a solvency violation. [6](#0-5) 

### Likelihood Explanation

**Preconditions:**
- A fee-on-transfer ERC20 is used as `loanToken` in a market. This requires only a permissionless `touchMarket` call — no admin action.
- Any subsequent `take` call on that market triggers the bug.

**Feasibility:** High. Many real tokens (e.g., tokens with reflection mechanics or protocol-fee-on-transfer) implement this pattern. The market creator role is unprivileged. The bug fires on every `take` in such a market, making it repeatable and cumulative.

### Recommendation

After each `safeTransferFrom`, verify the actual received amount using a balance-before/after check, and use the measured received amount to update `claimableSettlementFee` and to credit the seller. Concretely:

1. Record `balanceBefore = IERC20(loanToken).balanceOf(address(this))` before line 455.
2. Execute the first `safeTransferFrom`.
3. Compute `actualFeeReceived = IERC20(loanToken).balanceOf(address(this)) - balanceBefore`.
4. Replace the line-418 increment with `claimableSettlementFee[loanToken] += actualFeeReceived`.
5. For the second transfer (to `receiver`), record the receiver's balance before and after, and emit/assert the actual delivered amount equals `sellerAssets` (or revert if it does not).

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens by adding a validation step in `touchMarket` (e.g., a self-transfer probe), though this is harder to make robust.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

// FeeOnTransferToken: charges fee F (e.g. 1%) on every transferFrom
// Deploy a market with this as loanToken via touchMarket (permissionless)
// Then call take() and assert:

function testFeeOnTransferSellerUnderpaidAndFeeOvercount() public {
    uint256 F_BPS = 100; // 1% fee
    FeeOnTransferERC20 feeToken = new FeeOnTransferERC20(F_BPS);

    // Market creator (unprivileged) creates market with fee-on-transfer loanToken
    market.loanToken = address(feeToken);
    bytes32 id = midnight.touchMarket(market);

    // Setup: fund payer, collateralize seller
    feeToken.mint(payer, 1_000e18);
    vm.prank(payer); feeToken.approve(address(midnight), type(uint256).max);
    collateralize(market, seller, MAX_DEBT);

    uint256 receiverBalBefore = feeToken.balanceOf(receiver);
    uint256 contractBalBefore = feeToken.balanceOf(address(midnight));
    uint256 claimableBefore = midnight.claimableSettlementFee(address(feeToken));

    (uint256 buyerAssets, uint256 sellerAssets) = take(units, buyer, sellerOffer);

    uint256 spread = buyerAssets - sellerAssets;
    uint256 expectedFeeReceived = spread * (10000 - F_BPS) / 10000;
    uint256 expectedSellerReceived = sellerAssets * (10000 - F_BPS) / 10000;

    // Assert 1: receiver got less than sellerAssets (seller underpaid)
    assertEq(
        feeToken.balanceOf(receiver) - receiverBalBefore,
        expectedSellerReceived,
        "receiver underpaid"
    );
    assertLt(
        feeToken.balanceOf(receiver) - receiverBalBefore,
        sellerAssets,
        "receiver received full sellerAssets — should not happen"
    );

    // Assert 2: claimableSettlementFee overcounts by F * spread
    uint256 claimableIncrease = midnight.claimableSettlementFee(address(feeToken)) - claimableBefore;
    assertEq(claimableIncrease, spread, "claimable incremented by full spread");

    // Assert 3: actual contract balance increase is less than claimable increase
    uint256 actualContractIncrease = feeToken.balanceOf(address(midnight)) - contractBalBefore;
    assertEq(actualContractIncrease, expectedFeeReceived, "contract received fee*(1-F)");
    assertLt(actualContractIncrease, claimableIncrease, "double accounting: claimable > actual balance");

    // Assert 4: feeClaimer cannot claim the full recorded amount (reverts)
    vm.prank(feeClaimer);
    vm.expectRevert();
    midnight.claimSettlementFee(address(feeToken), claimableIncrease, feeClaimer);
}
```

**Expected assertions:**
- `receiver` balance delta = `sellerAssets * (1-F)` < `sellerAssets` ✓
- `claimableSettlementFee` delta = `buyerAssets - sellerAssets` (full, overcounted) ✓
- Contract balance delta = `(buyerAssets - sellerAssets) * (1-F)` < `claimableSettlementFee` delta ✓
- `claimSettlementFee` for the full recorded amount reverts ✓

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }

// For any token, the pending settlement fee receipt after a transaction is 0: every claimableSettlementFee
// increment in take is paid back in by the same-function inbound transfer.
weak invariant pendingFeeReceiptZero(address token)
    pendingFeeReceipt[token] == 0;

// For any token, the balance of the contract is always greater than or equal to the sum of all collateral, withdrawable, and claimable settlement fee amounts for that token minus the flash loaned amount.
// Note: this invariant is strong, so it also holds before each external call.
strong invariant tokenBalanceCorrect(address token)
    tokenBalances[token][currentContract] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token) - flashloans[token] - pendingFeeReceipt[token]
