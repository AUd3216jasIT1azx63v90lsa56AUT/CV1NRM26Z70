Audit Report

## Title
Debt Created at `maturity == block.timestamp` Is Immediately Liquidatable in the Next Block via Post-Maturity Mode - (File: src/Midnight.sol)

## Summary
`touchMarket` enforces no lower bound on `market.maturity`, allowing a market to be created with `maturity = block.timestamp`. The `take` guard uses `<=`, permitting debt creation at the exact maturity timestamp, while `liquidate`'s post-maturity guard uses strict `>`. This one-block asymmetry means a borrower's position is immediately liquidatable in the next block with no opportunity to repay, despite being healthy at the time of borrowing.

## Finding Description
**Root cause — missing lower-bound in `touchMarket` (`src/Midnight.sol:758`):**
```solidity
require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
```
No `require(market.maturity > block.timestamp)` exists. A market with `maturity = block.timestamp` is created without error.

**`SetterRatifier.isRatified` (`src/ratifiers/SetterRatifier.sol:24-36`):** Any address can call `setIsRootRatified(maker, root, true)` for themselves — no privilege required. The `isRatified` check validates only Merkle proof membership and `isRootRatified[offer.maker][root]`, with zero validation of `offer.market.maturity`.

**`take` (`src/Midnight.sol:391`):**
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
When `maturity = block.timestamp`, `block.timestamp <= block.timestamp` is `true`, so `sellerDebtIncrease > 0` is permitted. Additionally, `timeToMaturity = zeroFloorSub(maturity, block.timestamp) = 0`, so `buyerPendingFeeIncrease = 0` — debt is written at the exact maturity timestamp with no fee accrual.

**`liquidate` (`src/Midnight.sol:620-624`):**
```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```
One block later (`block.timestamp = maturity + 1`), `block.timestamp > market.maturity` is `true`, so post-maturity liquidation of a healthy borrower succeeds.

**Exploit flow:**
1. Attacker (maker) constructs an offer with `offer.market.maturity = block.timestamp`.
2. Attacker calls `setterRatifier.setIsRootRatified(maker, root, true)` — no privilege required.
3. In the same block, a taker (victim borrower) calls `midnight.take(offer, ...)` with `units > 0`. The check at line 391 passes (`T <= T`). The `SellerIsLiquidatable` check at line 476 passes because the borrower has sufficient collateral. Debt is written to `sellerPos.debt`.
4. In the next block, any caller invokes `midnight.liquidate(market, ..., postMaturityMode=true)`. The guard `block.timestamp > market.maturity` is satisfied. The healthy borrower's position is liquidated.

**Why existing checks fail:**
- `touchMarket`: no lower-bound maturity check.
- `SetterRatifier.isRatified`: no maturity validation.
- `take` line 391: `<=` boundary case `maturity == block.timestamp` is not excluded.
- `liquidate` line 622: strict `>` creates a one-block window where debt exists but is already post-maturity-liquidatable.
- `repay` (line 502): exists but cannot protect the borrower if the liquidator front-runs in the next block.

## Impact Explanation
A healthy borrower who takes a loan in a market where `maturity = block.timestamp` has their collateral immediately seizable in the next block via `postMaturityMode=true` liquidation, with no opportunity to repay. The borrower suffers an immediate, unavoidable collateral loss on a position that was healthy at the time of borrowing. The LIF at `block.timestamp = maturity + 1` is slightly above 1, so the liquidator receives slightly more collateral than the debt repaid.

## Likelihood Explanation
All preconditions are fully attacker-controlled: any unprivileged address can be a maker, call `setIsRootRatified` on themselves, and construct an offer with `maturity = block.timestamp`. The attack requires no admin action, oracle manipulation, or special privilege, and is repeatable in any block. The taker need only fail to inspect the maturity field of the offer — a realistic condition in any UI-mediated interaction.

## Recommendation
Add a lower-bound maturity check in `touchMarket`:
```solidity
require(market.maturity > block.timestamp, MaturityInPast());
```
This prevents market creation with `maturity <= block.timestamp`, closing the boundary-case asymmetry between `take` (line 391, `<=`) and `liquidate` (line 622, `>`). Alternatively, change `take` line 391 to use strict `<` (`block.timestamp < offer.market.maturity`) to exclude the exact maturity timestamp from debt creation.

//    → block.timestamp > market.maturity is true → healthy position liquidated

    uint256 maturity = block.timestamp;
    Market memory market = buildMarket(maturity);
    Offer memory offer = buildOffer(market, /* buy=false, units>0 */);
    // maker ratifies root
    setterRatifier.setIsRootRatified(maker, root, true);
    // victim takes offer in same block
    vm.prank(victim);
    midnight.take(offer, ratifierData, units, victim, address(0), "");
    // next block
    vm.roll(block.number + 1);
    vm.warp(block.timestamp + 1);
    // liquidator liquidates healthy position
    midnight.liquidate(market, 0, seizedAssets, 0, victim, true, liquidator, address(0), "");
    // assert victim's collateral seized despite healthy LTV at time of borrow
}
