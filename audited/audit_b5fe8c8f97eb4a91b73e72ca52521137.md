Audit Report

## Title
`TakeAmountsLib.sellerAssetsToUnits` panics on `sellerPrice == 0` outside try/catch, reverting entire bundle - (`src/periphery/TakeAmountsLib.sol` / `src/periphery/MidnightBundles.sol`)

## Summary
When a buy offer's tick price equals the settlement fee exactly, `sellerPrice` computes to zero in `TakeAmountsLib.sellerAssetsToUnits`. The subsequent call `targetSellerAssets.mulDivUp(WAD, 0)` triggers an arithmetic underflow panic at `(d - 1)` before any division. Because this call is evaluated unconditionally before the `try/catch` guard in `supplyCollateralAndSellWithAssetsTarget`, the panic propagates and reverts the entire bundle — including all collateral supplies — even though `Midnight.take` accepts the same offer without reverting.

## Finding Description

**Root cause — `mulDivUp` with `d = 0`:**

`UtilsLib.mulDivUp` is implemented in checked arithmetic:

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
``` [1](#0-0) 

When `d = 0`, the sub-expression `(d - 1)` = `(0 - 1)` underflows under Solidity 0.8 checked arithmetic, producing panic code `0x11` (arithmetic overflow/underflow), not a division-by-zero error. This is a pre-division panic that cannot be caught by a `try/catch` on an external call.

**Trigger path — `sellerAssetsToUnits` with `sellerPrice = 0`:**

`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and passes it as the divisor to `mulDivUp`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
``` [2](#0-1) 

When `offer.buy == true` and `tickToPrice(offer.tick) == settlementFee(id, timeToMaturity)`, `sellerPrice = 0`, and `mulDivUp(..., WAD, 0)` panics. The NatSpec comment on `sellerAssetsToUnits` states it "Reverts if offerPrice < settlementFee" but does not account for the `==` case, which also panics. [3](#0-2) 

**The panic is outside the `try/catch`:**

In `supplyCollateralAndSellWithAssetsTarget`, `sellerAssetsToUnits` is evaluated unconditionally before the guarded `take` call:

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) returns (...) { ... } catch {}
``` [4](#0-3) 

The `try/catch` only guards `IMidnight.take`; the pre-computation panic propagates to the caller. The function's own NatSpec acknowledges this: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." [5](#0-4) 

**`consumableUnits` does not panic for the same offer:**

For `offer.buy == true` and `offer.maxUnits > 0`, `consumableUnits` returns `offer.maxUnits.zeroFloorSub(consumed)` — no panic. For `offer.maxUnits == 0`, it routes to `buyerAssetsToUnits`, which divides by `buyerPrice = sellerPrice + settlementFee = 0 + settlementFee = settlementFee` (non-zero when fee > 0). [6](#0-5) 

The panic is exclusively from the direct `sellerAssetsToUnits` call.

**Collateral supplies are rolled back:**

Collateral supplies are executed unconditionally before the take loop: [7](#0-6) 

When the panic occurs inside the loop, the entire transaction reverts, rolling back all collateral supplies already executed.

## Impact Explanation
The entire `supplyCollateralAndSellWithAssetsTarget` transaction reverts. All collateral supplies executed before the loop are also rolled back. The taker cannot sell units or supply collateral even though all other offers in the bundle are valid and `Midnight.take` would accept them individually. The stated design intent of the bundler is to skip offers that `take` would reject; here it panics for an offer `take` would accept, violating that invariant. The impact is a complete DoS of the bundle function for any taker who includes such an offer.

## Likelihood Explanation
Settlement fees are interpolated linearly between breakpoints as time-to-maturity decreases. For any market with a non-zero settlement fee, there exists a moment when the interpolated fee equals `tickToPrice(t)` for some tick `t`. A maker can create a buy offer at such a tick. Any taker who includes that offer in a `supplyCollateralAndSellWithAssetsTarget` bundle at the exact moment the condition holds will have their entire bundle reverted. The condition is transient but repeatable and can be targeted by monitoring on-chain fee state against tick prices.

## Recommendation
Guard `sellerAssetsToUnits` against `sellerPrice == 0` before calling `mulDivUp`. The simplest fix is to add an early return of `type(uint256).max` (or `0`) when `sellerPrice == 0` in `sellerAssetsToUnits`, consistent with how `Midnight.take` handles this case (yielding `sellerAssets = 0`). Alternatively, wrap the `sellerAssetsToUnits` and `consumableUnits` calls in a `try/catch` at the call site in `supplyCollateralAndSellWithAssetsTarget`, or add an explicit `if (sellerPrice == 0) return 0;` guard in `sellerAssetsToUnits` before the `mulDivUp` call. [2](#0-1) 

## Proof of Concept
1. Deploy a market with a non-zero settlement fee schedule.
2. Create a buy offer at tick `t` such that `tickToPrice(t) == settlementFee(id, timeToMaturity)` at some future timestamp.
3. At that timestamp, call `supplyCollateralAndSellWithAssetsTarget` with `collateralSupplies` non-empty and `takes` containing the above offer.
4. Observe: the transaction reverts with arithmetic panic `0x11` (not `OutOfOffers`), and all collateral supplies are rolled back.
5. Confirm: calling `Midnight.take` directly with the same offer and `units > 0` succeeds and returns `sellerAssets = 0`.

A fuzz test can target this by fuzzing `offerPrice` and `settlementFee` with the constraint `offerPrice == settlementFee` and asserting that `sellerAssetsToUnits` does not revert when `Midnight.take` would not revert.
