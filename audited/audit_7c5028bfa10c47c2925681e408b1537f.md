Audit Report

## Title
Buy offer `maxAssets` cap bypass via `mulDivDown` zero-rounding allows unbounded unit overfill - (File: src/Midnight.sol)

## Summary
When a buy offer has `maxAssets > 0` and `tick < MAX_TICK` (i.e., `buyerPrice < WAD`), a taker can call `take` with `units` small enough that `mulDivDown(units, buyerPrice, WAD)` truncates to zero. Because `consumed` is incremented by `buyerAssets = 0`, the `ConsumedAssets` guard never fires and the offer can be filled for arbitrarily many units beyond the maker's intended cap. The protocol's own NatSpec and test suite explicitly acknowledge and demonstrate this behavior.

## Finding Description
**Code path** — `src/Midnight.sol` lines 363–369:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN to zero when units * buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause** — `mulDivDown(units, buyerPrice, WAD)` truncates to zero whenever `units * buyerPrice < WAD`. `tickToPrice` is proven (Certora rule `tickToPriceIsOneAtMaxTick`, `tickToPriceAtMostWad`) to return `WAD` only at `MAX_TICK = 5820` and strictly less than `WAD` for all lower ticks. Choosing `units = 1` (or any value below `WAD / buyerPrice`) makes `buyerAssets = 0`.

**Exploit flow**:
1. Maker publishes a buy offer with `maxAssets = N`, `tick < MAX_TICK`.
2. Offer is fully consumed: `consumed[maker][group] = N`.
3. Taker calls `take(units=1, taker, offer, ...)` where `1 * buyerPrice < WAD`.
4. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`.
5. `consumed += 0` → `newConsumed = N` → `require(N <= N)` passes.
6. Position state mutates: maker's credit increases by 1 unit, taker's debt increases by 1 unit, zero tokens transferred.
7. Steps 3–6 repeat indefinitely; each iteration adds units to the maker's credit with no asset cost and no consumed increment.

**Why existing checks fail** — The `ConsumedAssets` guard at line 369 is purely asset-denominated. It cannot detect unit-level overfill when `buyerAssets` rounds to zero. The Certora `takeConsumedBoundedByMax` rule (`certora/specs/Consume.spec` line 62) only asserts `consumed <= maxAssets`, which is trivially satisfied when the increment is zero. No unit-level invariant is formally verified for the `maxAssets` branch.

The protocol's own NatSpec at `src/Midnight.sol` line 94 explicitly acknowledges: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol` lines 858–889) confirms the state change is real: `creditOf(lender)` and `debtOf(borrower)` both increase while `consumed` and token balances remain unchanged.

## Impact Explanation
The maker's `maxAssets` cap is completely ineffective in unit terms when `buyerPrice < WAD`. A taker can fill an unbounded number of units on a fully-consumed buy offer, giving the maker more credit than they intended and imposing matching debt on themselves. This directly violates the core protocol invariant that offers cannot be overfilled. The maker receives credit they did not intend to provide; the taker accumulates debt with no corresponding asset transfer. This constitutes unauthorized state mutation and accounting corruption reachable by any unprivileged user.

## Likelihood Explanation
The precondition (`buyerPrice < WAD`) holds for every tick except `MAX_TICK = 5820`, covering the vast majority of real-world offers. No special permissions are required; any unprivileged taker can trigger this. The attack is repeatable in a single transaction via `multicall` or across multiple transactions. The only cost to the attacker is gas and the debt they accumulate (which they may not care about if they have sufficient collateral or intend to default).

## Recommendation
Add a unit-level consumed check in the `maxAssets` branch, or reject takes where `buyerAssets == 0` and `units > 0`. Concretely, either:

1. **Reject zero-asset takes when `units > 0`**: Add `require(units == 0 || buyerAssets > 0, ZeroAssets())` before the consumed update.
2. **Track consumed in units when `buyerAssets` rounds to zero**: Fall back to a unit-denominated cap check when `buyerAssets == 0`.
3. **Enforce a minimum `units` value**: Require `units.mulDivDown(buyerPrice, WAD) > 0` for any `maxAssets`-capped buy offer.

Option 1 is the simplest and most conservative fix.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` at lines 858–889 is a complete, self-contained PoC:

- Sets `maxAssets = 1`, `tick = MAX_TICK - 16` (so `buyerPrice < WAD`).
- Pre-consumes the offer to its cap via `setConsumed`.
- Calls `take(1, borrower, lenderOffer)`.
- Asserts `buyerAssets == 0`, `consumed` unchanged, token balances unchanged.
- Asserts `creditOf(lender)` and `debtOf(borrower)` both strictly increased.

This test passes, confirming the vulnerability is real and exploitable.
