### Title
Inclusive `<=` boundary in `CannotIncreaseDebtPostMaturity` allows seller debt increase at exactly `block.timestamp == maturity` - (`src/Midnight.sol`)

### Summary
`Midnight.take` guards against post-maturity debt increases with `require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity())` at line 391. Because the comparison is `<=` rather than `<`, the guard passes without reverting when `block.timestamp == maturity` even if `sellerDebtIncrease > 0`. The expiry check at line 353 uses the same inclusive operator, so an offer with `offer.expiry == offer.market.maturity` is also valid at that exact timestamp, completing the exploit path. The protocol's own documented invariant explicitly states "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."

### Finding Description

**Exact code path:**

`Midnight.take` (lines 337–395, `src/Midnight.sol`):

```
line 353: require(block.timestamp <= offer.expiry, OfferExpired());
          // at block.timestamp == maturity, maturity <= maturity → passes

line 391: require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0,
                  CannotIncreaseDebtPostMaturity());
          // at block.timestamp == maturity, maturity <= maturity → TRUE → passes
          // even when sellerDebtIncrease > 0
```

**Root cause:** The `<=` operator on line 391 treats `block.timestamp == maturity` as pre-maturity, allowing `sellerDebtIncrease > 0` to proceed. The correct boundary is strict: debt must not increase at or after maturity (`<`).

**Attacker-controlled inputs:**
- `offer.buy = false` (sell offer, so `seller = offer.maker`, `buyer = taker`)
- `offer.expiry = offer.market.maturity` (or any value ≥ maturity)
- `units` chosen so that `sellerDebtIncrease = units - sellerCreditDecrease > 0` (maker has no or insufficient credit)
- Call submitted in the block where `block.timestamp == maturity`

**Exploit flow:**
1. Maker signs a sell offer (`offer.buy = false`) with `offer.expiry >= offer.market.maturity`. The `EcrecoverRatifier.isRatified` validates the signature without any maturity check (lines 33–46, `src/ratifiers/EcrecoverRatifier.sol`).
2. Taker waits until `block.timestamp == maturity`.
3. Taker calls `Midnight.take` with `units > sellerPos.credit` so `sellerDebtIncrease > 0`.
4. Line 353: `maturity <= maturity` → passes.
5. Line 391: `maturity <= maturity` → TRUE → `CannotIncreaseDebtPostMaturity` does **not** revert.
6. Line 414: `sellerPos.debt += sellerDebtIncrease` executes, increasing the maker's debt at exactly maturity.
7. The maker's newly created debt is immediately overdue (maturity has been reached), making the position immediately liquidatable even if it was healthy one block earlier.

**Why existing checks fail:** The only guard against post-maturity debt increase is line 391. It uses `<=` instead of `<`, so the single-block window at `block.timestamp == maturity` is unprotected. No other check in `take` or in `EcrecoverRatifier.isRatified` closes this gap.

### Impact Explanation
An unprivileged taker can force a maker's debt to increase at exactly `block.timestamp == maturity` by taking a sell offer in that block. The resulting debt is immediately overdue and liquidatable under the protocol rule that "overdue debt after maturity is liquidatable even if healthy." The maker receives `sellerAssets` loan tokens but their collateral is immediately exposed to liquidation with no grace period, violating the invariant that debt must not increase at or after maturity.

### Likelihood Explanation
**Preconditions:**
- A valid sell offer exists with `offer.expiry >= offer.market.maturity` (common; makers routinely set expiry at or beyond maturity).
- The maker has no existing credit in the market (or less credit than `units`), so `sellerDebtIncrease > 0`.
- The taker submits the transaction in the exact block where `block.timestamp == maturity`.

**Feasibility:** The maturity timestamp is public and fixed at market creation. A taker can monitor the chain and submit the transaction in the maturity block. Block timestamps are miner/validator-controlled within a small range, making precise targeting straightforward. The attack is repeatable for every market and every qualifying offer.

### Recommendation
Change the comparison on line 391 from `<=` to `<`:

```solidity
// Before (buggy):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This makes `block.timestamp == maturity` fall into the post-maturity branch, requiring `sellerDebtIncrease == 0` and matching the documented invariant.
