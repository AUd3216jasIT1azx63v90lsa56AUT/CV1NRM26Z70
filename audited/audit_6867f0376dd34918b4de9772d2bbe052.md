Audit Report

## Title
Gas-griefing via unbounded `takes[]` loop and uncapped `isRatified()` call in `MidnightBundles` - (File: `src/periphery/MidnightBundles.sol`)

## Summary
Every bundler function in `MidnightBundles` iterates over a caller-supplied, unbounded `takes[]` array and wraps each `Midnight.take()` call in a bare `try {} catch {}`. Because `Midnight.take()` makes a plain external call to `IRatifier.isRatified()` with no gas cap, a malicious maker can deploy a ratifier that burns nearly all forwarded gas then reverts. A victim taker whose routing software includes even two such offers in `takes[]` will exhaust their entire gas budget with zero fills, paying full gas cost for nothing.

## Finding Description

**Confirmed code path — `MidnightBundles.sol` lines 71–86:**

```solidity
for (uint256 i; i < takes.length && filledUnits < targetUnits; i++) {
    ...
    try IMidnight(MIDNIGHT)
        .take(takes[i].offer, takes[i].ratifierData, unitsToTake, taker, ...) returns (...) {
        filledUnits += unitsToTake;
        filledBuyerAssets += resBuyerAssets;
    } catch {}   // silent — catches OOG-induced reverts too
}
```

The identical pattern appears in all four bundler functions at lines 144–161, 205–222, and 282–301. [1](#0-0) 

**Confirmed code path — `Midnight.sol` line 356:**

```solidity
require(IRatifier(offer.ratifier).isRatified(offer, ratifierData) == CALLBACK_SUCCESS, RatifierFail());
```

This is a plain external call. Under EIP-150, the EVM forwards up to 63/64 of the caller's remaining gas to `isRatified()` with no cap. [2](#0-1) 

**Confirmed: `setIsAuthorized` is fully permissionless for self-authorization:**

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    ...
}
```

An attacker calls `setIsAuthorized(maliciousRatifier, true, attacker)` where `onBehalf == msg.sender`. This satisfies the `isAuthorized[offer.maker][offer.ratifier]` check at line 355 with zero privilege. [3](#0-2) 

**Gas cascade per malicious iteration (EIP-150 63/64 rule):**

Let the bundler enter an iteration with G gas:
- Bundler calls `take()` via `try`: `take()` receives ≤63G/64; bundler retains ≥G/64.
- `take()` calls `isRatified()`: `isRatified()` receives ≤63²G/64²; `take()` retains ≥63G/64².
- `isRatified()` burns all forwarded gas and reverts.
- `take()` has ≈63G/64² remaining when it reverts; bundler recovers that plus its retained G/64.
- Bundler exits iteration 1 with ≈(64+63)/64² = 127/4096 ≈ **3.1% of G**.

After iteration 2 with 0.031G: bundler exits with ≈3.1% × 3.1% ≈ **0.096% of G** — effectively zero for any realistic gas budget. The next external call (e.g., `ConsumableUnitsLib.consumableUnits()` or the next `try`) OOGs, consuming the victim's full gas budget. The final `require(filledUnits == targetUnits, OutOfOffers())` is never reached; the transaction reverts OOG. [4](#0-3) 

**Existing protections reviewed and found insufficient:**

- `isAuthorized[offer.maker][offer.ratifier]` (line 355): Attacker self-authorizes their own malicious ratifier — no barrier.
- `ConsumableUnitsLib.consumableUnits()` (called before `try`): Pure view call, does not invoke the ratifier — no barrier.
- `catch {}` (lines 85, 160, 221, 300): Designed to handle asynchrony, but also catches gas-burning reverts — no barrier.
- No `takes.length` cap anywhere in `MidnightBundles`. [5](#0-4) 

## Impact Explanation
Any user of the bundler who includes even two offers from a malicious maker in their `takes[]` array will have their entire transaction gas consumed with zero fills. The victim pays full block-gas-limit cost and receives nothing. This constitutes concrete, repeatable griefing/DoS of takers using the bundler — a service unavailability impact under realistic attacker input.

## Likelihood Explanation
**Preconditions:** Attacker deploys one contract and calls `setIsAuthorized` once — both permissionless. Offers are distributed off-chain by protocol design. Routing software that aggregates offers from public orderbooks is the expected usage pattern. The attacker can post offers with competitive prices to maximize inclusion probability. The same two malicious offers can be reused against any number of victims at zero marginal cost per victim.

## Recommendation
Apply at least one of the following mitigations:

1. **Gas-cap the `isRatified()` call** in `Midnight.take()`: forward a fixed gas stipend (e.g., 100k gas) rather than all available gas.
2. **Gas-check in `catch`**: after each `catch {}`, verify `gasleft() > MINIMUM_GAS_THRESHOLD` and revert or break if not.
3. **Cap `takes.length`** in each bundler function to a small constant (e.g., 32).
4. **Combine (2) and (3)** for defense-in-depth.

## Proof of Concept

**Minimal reproduction steps:**

1. Deploy `MaliciousRatifier` whose `isRatified()` spins in a loop consuming ≈all forwarded gas, then reverts.
2. Call `midnight.setIsAuthorized(address(maliciousRatifier), true, attacker)`.
3. Construct two `Offer` structs with `offer.maker = attacker`, `offer.ratifier = address(maliciousRatifier)`, competitive price, valid expiry.
4. Build a `takes[]` array of length 2 containing these offers.
5. Call `bundler.buyWithUnitsTargetAndWithdrawCollateral(...)` with a realistic gas limit (e.g., 3M gas).
6. Observe: transaction reverts OOG; `filledUnits == 0`; victim paid full gas cost.

**Invariant fuzz test plan:** Fuzz `takes[]` with a mix of legitimate and malicious offers; assert that `gasleft()` after the loop is never below a safe threshold, and that the transaction never OOGs when `takes.length <= MAX_TAKES`.
