Audit Report

## Title
`onLiquidate` Callback Enables Post-Bad-Debt Collateral Injection, Over-Slashing Lenders - (File: src/Midnight.sol)

## Summary
In `liquidate()`, bad debt is computed from a local snapshot of `_position.collateralBitmap` and immediately committed to `lossFactor`, `totalUnits`, and `_position.debt` before the `onLiquidate` callback fires. Because `supplyCollateral` has no reentrancy guard and `liquidationLocked` is never set during `liquidate()`, a borrower-authorized callback can inject new collateral into the position after bad debt accounting is finalized. The injected collateral is never counted against `badDebt`, causing lenders to be permanently over-slashed by the collateral's oracle-weighted value while the borrower retains the injected collateral in their position.

## Finding Description

**Root cause — local bitmap snapshot used for bad debt, storage mutated before callback:**

At line 606, `liquidate()` captures a local copy of the bitmap:
```solidity
uint128 _collateralBitmap = _position.collateralBitmap;   // line 606
while (_collateralBitmap != 0) { ... }                    // badDebt computed here
```
Bad debt is then immediately written to storage at lines 626–641 — `_position.debt`, `_marketState.lossFactor`, and `_marketState.totalUnits` are all updated — before any external call. The `onLiquidate` callback fires only after all accounting is finalized, at lines 698–715.

**`supplyCollateral` is unguarded during `liquidate()`:**

`liquidationLocked` is exclusively set inside `take()` at line 444 via `UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`, never inside `liquidate()`. The protocol comment at line 60 confirms: "Liquidations are locked for the seller during the callbacks of take." `supplyCollateral`'s only gate is the authorization check (`isAuthorized[onBehalf][msg.sender]`), which the attacker satisfies by pre-authorizing the callback contract.

**Exploit flow:**

1. Borrower has debt `D`, collateral `C0` at index 0; oracle price drops so `D > C0·price0/maxLif0` (bad debt territory, position also unhealthy so `D > maxDebt`).
2. Borrower calls `setIsAuthorized(maliciousCallback, true, borrower)`. Callback contract is pre-funded with `C1` tokens (collateral at index 1).
3. Anyone (including the borrower) calls `liquidate(market, 0, 0, 0, borrower, false, receiver, maliciousCallback, data)`. Both `repaidUnits` and `seizedAssets` are 0, which passes the `atMostOneNonZero` check at line 595.
4. Inside `liquidate()`:
   - `badDebt = D − C0·price0/maxLif0` computed from local bitmap snapshot (only index 0 present).
   - `lossFactor` updated, `totalUnits` reduced by `badDebt` — lenders are slashed.
   - `_position.debt -= badDebt`.
   - Since `repaidUnits == 0 && seizedAssets == 0`, the collateral seizure block at line 643 is skipped entirely.
5. `onLiquidate` fires at line 698. `maliciousCallback` calls `supplyCollateral(market, 1, C1amount, borrower)`. Because `isAuthorized[borrower][maliciousCallback]` is true, this succeeds. `_position.collateral[1] = C1amount` and bit 1 is set in `_position.collateralBitmap`. Tokens are pulled from the callback contract.
6. Callback returns `CALLBACK_SUCCESS`; 0 loan tokens pulled (line 717: `repaidUnits = 0`).
7. No post-callback health or state check exists (lines 717–719 only transfer tokens and return).
8. Borrower's position now has reduced debt (`C0·price0/maxLif0`) and collateral C0 + C1. Lenders have been slashed as if C1 never existed.

**Why existing checks fail:**

- `liquidationLocked` is never set in `liquidate()`, so `supplyCollateral` is not blocked during the callback window.
- `supplyCollateral`'s only guard is the authorization check, which the attacker satisfies by pre-authorizing the callback.
- The bad debt loop iterates over the local `_collateralBitmap` snapshot, so any collateral added to `_position.collateralBitmap` during the callback is invisible to the already-completed bad debt computation.
- There is no post-callback invariant check on the position state.

## Impact Explanation

Lenders are permanently over-slashed by `C1amount · price1 / ORACLE_PRICE_SCALE · WAD / maxLif1` units of credit relative to what they should have been slashed. The borrower's debt is reduced by this same extra amount while the borrower retains the C1 collateral in their position (usable to support future borrowing or withdrawable once the position is healthy). This is a direct, permanent reduction in lender redeemable value and violates the protocol invariant that bad debt realization must account for all collateral present at the time of liquidation. The attack is repeatable across any market with bad debt.

## Likelihood Explanation

**Preconditions:**
1. Position must be in bad debt territory — a realistic market condition requiring only an oracle price drop.
2. Borrower must have authorized the callback contract — fully attacker-controlled, as the borrower can call `setIsAuthorized` at any time before the liquidation.

Self-liquidation is not blocked anywhere in `liquidate()`. The borrower can be both the liquidator and the callback operator. No privileged role is required. The attack is executable by any normal user who controls a borrower position.

## Recommendation

Set `liquidationLocked` for the borrower during the `liquidate()` execution window (analogous to how `take()` sets it for the seller at line 444), and clear it after the callback returns. This prevents `supplyCollateral` (and any other state-mutating function gated on `liquidationLocked`) from being called during the callback. Alternatively, re-compute bad debt after the callback using the live `_position.collateralBitmap` and reconcile any discrepancy before finalizing the loss factor update.

## Proof of Concept

**Minimal manual steps:**

1. Deploy a market with two collateral tokens (C0, C1) and a loan token.
2. Borrower supplies C0, borrows `D` such that `D > C0·price0/maxLif0` after an oracle price drop.
3. Borrower calls `setIsAuthorized(MaliciousCallback, true, borrower)`.
4. `MaliciousCallback` is pre-funded with `C1amount` of token C1.
5. Call `liquidate(market, 0, 0, 0, borrower, false, receiver, MaliciousCallback, "")`.
6. Inside `MaliciousCallback.onLiquidate(...)`, call `midnight.supplyCollateral(market, 1, C1amount, borrower)`.
7. Assert: `marketState[id].lossFactor` reflects slashing of `D - C0·price0/maxLif0` units.
8. Assert: `position[id][borrower].collateral[1] == C1amount` (C1 is in the position).
9. Assert: lender redeemable value is reduced by `C1amount · price1 / ORACLE_PRICE_SCALE · WAD / maxLif1` more than it should be.
10. Optionally: repay enough debt to make the position healthy, then call `withdrawCollateral` to recover C1.
