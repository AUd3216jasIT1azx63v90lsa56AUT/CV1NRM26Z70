### Title
Fee-on-Transfer Loan Token Causes `consumed` to Overstate Economic Value Exchanged in `take()` - (File: src/Midnight.sol)

### Summary

In `take()`, `consumed[offer.maker][offer.group]` is incremented by the arithmetically computed `buyerAssets` (or `sellerAssets`) before any token transfer occurs, and the subsequent `safeTransferFrom` calls do not verify the actual amount received. When the loan token is fee-on-transfer, the actual tokens delivered to Midnight and the receiver are reduced by the fee, but `consumed` is still incremented by the full nominal amount. This causes the offer's consumption cap to be reached before the maker's intended economic exposure (`maxAssets`) has actually been filled.

### Finding Description

**Exact code path:**

In `src/Midnight.sol`, `take()` computes `buyerAssets` and `sellerAssets` purely arithmetically:

```solidity
// lines 363-364
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD);
```

Then, when `offer.maxAssets > 0`, consumption is tracked using these nominal values:

```solidity
// lines 367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

The actual token transfers happen later:

```solidity
// lines 455-456
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

`SafeTransferLib.safeTransferFrom` (lines 24-34 of `src/libraries/SafeTransferLib.sol`) only checks the boolean return value of `transferFrom`; it does not perform a balance-before/balance-after check and does not verify the actual amount received.

**Root cause:** `consumed` is incremented by the nominal computed amount, not the actual tokens received. With a fee-on-transfer loan token, the actual received amount is `nominal - fee`, but `consumed` grows by `nominal`.

**Attacker-controlled inputs:**
- Any unprivileged taker can call `take()` with a valid offer where `offer.maxAssets > 0` and the market's `loanToken` is a fee-on-transfer token.
- The taker does not need any special privilege; the market creator chose the loan token, and the taker simply fills the offer.

**Exploit flow:**
1. A market is created with a fee-on-transfer `loanToken` (e.g., 1% fee per transfer).
2. Maker creates a buy offer with `offer.maxAssets = 1000e18` and `offer.buy = true`.
3. Taker calls `take()` with `units` such that `buyerAssets = 1000e18`.
4. `consumed[maker][group]` is incremented by `1000e18`, reaching `maxAssets`.
5. `safeTransferFrom` transfers `1000e18` nominal, but Midnight receives `(buyerAssets - sellerAssets) * (1 - fee%)` and the receiver gets `sellerAssets * (1 - fee%)`.
6. The offer is now fully consumed (`consumed == maxAssets`), but the actual economic value delivered is only `~990e18`.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets, ConsumedAssets())` at line 369 only enforces the nominal cap; it has no knowledge of the fee deduction.
- There is no balance-before/balance-after pattern anywhere in `take()`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The offer's `consumed` counter reaches `maxAssets` before the actual economic value delivered equals `maxAssets`. The maker's effective liquidity is reduced: they intended to lend/borrow up to `maxAssets` worth of tokens, but the offer is exhausted after only `maxAssets * (1 - fee_rate)` of real value is exchanged. The deficit is proportional to the fee rate and `maxAssets`. This is repeatable across every take on any fee-on-transfer loan token market with an asset-based cap.

### Likelihood Explanation

**Preconditions:**
1. `offer.maxAssets > 0` — explicitly set by the maker; a common configuration per the protocol's own docs (lines 90-92 of `src/Midnight.sol`).
2. `loanToken` is fee-on-transfer — the market creator chooses the token; no protocol-level exclusion of such tokens exists.
3. Taker calls `take()` — permissionless, no special role required.

All three preconditions are reachable without any privileged action. The bug is triggered on every single `take()` call in such a market, making it fully repeatable.

### Recommendation

Track actual received amounts using a balance-before/balance-after pattern for the loan token transfers, and use the actual received delta to increment `consumed` rather than the computed nominal `buyerAssets`/`sellerAssets`. Alternatively, explicitly document and enforce (via a check or a registry) that fee-on-transfer tokens are not supported as loan tokens, reverting market creation for such tokens.

### Proof of Concept

```solidity
// Foundry unit test plan
contract FeeOnTransferLoanToken is ERC20 {
    uint256 constant FEE_BPS = 100; // 1% fee
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount * FEE_BPS / 10000;
        super.transferFrom(from, to, amount - fee); // receiver gets amount - fee
        _burn(from, fee);                           // fee is burned
        return true;
    }
}

function testFeeOnTransferConsumedOverstated() public {
    // Setup: market with FeeOnTransferLoanToken as loanToken
    // Maker creates buy offer: maxAssets = 1000e18, maxUnits = 0
    // Taker fills offer with units s.t. buyerAssets = 1000e18

    uint256 consumedBefore = midnight.consumed(maker, offer.group);
    uint256 receiverBalBefore = feeToken.balanceOf(receiver);

    take(units, taker, offer);

    uint256 consumedAfter = midnight.consumed(maker, offer.group);
    uint256 actualReceived = feeToken.balanceOf(receiver) - receiverBalBefore;

    // consumed reaches maxAssets (1000e18)
    assertEq(consumedAfter - consumedBefore, 1000e18);
    // but actual value delivered is only ~990e18
    assertLt(actualReceived, 1000e18);
    // invariant violated: consumed overstates actual economic value
    assertEq(consumedAfter, offer.maxAssets); // offer exhausted
    assertLt(actualReceived, offer.maxAssets); // but real value < maxAssets
}
```

**Expected assertions:** `consumed` equals `maxAssets` while `actualReceived < maxAssets`, proving the offer is exhausted before the maker's intended economic exposure is reached.
