### Title
Cross-denomination `consumed` subtraction in `consumableUnits` silently under-reports remaining capacity for `maxAssets` offers when a same-group `maxUnits` offer has been partially filled - (`File: src/periphery/ConsumableUnitsLib.sol`)

### Summary
The `consumed[maker][group]` mapping stores a single scalar that is units-denominated when the last fill was against a `maxUnits` offer, and assets-denominated when the last fill was against a `maxAssets` offer. `consumableUnits` reads this scalar and unconditionally subtracts it from `offer.maxAssets` before passing the result to `buyerAssetsToUnits`/`sellerAssetsToUnits`, treating a units-denominated number as if it were assets. When a maker holds two offers in the same group — one `maxUnits` and one `maxAssets` — and the `maxUnits` offer is partially filled first, the periphery computes a dimensionally incorrect remaining-assets figure for the `maxAssets` offer, causing `MidnightBundles` to under-fill or completely skip a still-valid offer and revert with `OutOfOffers()`.

### Finding Description

**Root cause — `ConsumableUnitsLib.consumableUnits` (lines 14-23):**

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);  // single scalar, denomination depends on which offer was filled last
    if (offer.maxUnits > 0) {
        return offer.maxUnits.zeroFloorSub(consumed);
    } else if (offer.buy) {
        return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed)); // BUG: consumed may be units, not assets
    } else {
        return TakeAmountsLib.sellerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed)); // BUG: same
    }
}
``` [1](#0-0) 

**Core accounting — `Midnight.take` (lines 367-373):**

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets; // assets-denominated
    require(newConsumed <= offer.maxAssets, ConsumedAssets());

    require(newConsumed <= offer.maxUnits, ConsumedUnits());

``` [2](#0-1) 

The mapping `consumed[maker][group]` is a single `uint256` with no denomination tag. Its meaning depends entirely on which offer type was filled last.

**Protocol note (not enforced):**

> To work as expected, all offers in the same group should have the same direction (offer.buy), max values and loan token. [3](#0-2) 

This is a comment, not a `require`. The `take()` function only checks `atMostOneNonZero(offer.maxAssets, offer.maxUnits)` per individual offer — there is no cross-offer group homogeneity check. [4](#0-3) 

**Exploit path:**

1. Maker (unprivileged) creates **Offer A** in group G: `maxUnits = 1000`, `maxAssets = 0`.
2. Maker creates **Offer B** in group G: `maxUnits = 0`, `maxAssets = 500`, `offer.buy = true`.
3. Taker fills Offer A for 600 units via `take()` → `consumed[maker][G] = 600` (units-denominated).
4. Taker calls `MidnightBundles.supplyCollateralAndSellWithUnitsTarget(...)` with Offer B in the `takes` array.
5. Bundler calls `ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, offerB)`:
   - `consumed = 600` (units, but treated as assets)
   - `offer.maxAssets.zeroFloorSub(consumed)` = `500 - 600` → saturates to `0` via `zeroFloorSub`
   - Returns `buyerAssetsToUnits(midnight, id, offerB, 0)` = **0 units**
6. `unitsToTake = min(remaining, takes[i].units, 0) = 0` — Offer B is completely skipped.
7. Loop exhausts all offers; `filledUnits < targetUnits` → reverts `OutOfOffers()`. [5](#0-4) 

The same path applies to `buyWithUnitsTargetAndWithdrawCollateral` (line 77), `buyWithAssetsTargetAndWithdrawCollateral` (line 213), and `supplyCollateralAndSellWithAssetsTarget` (line 290). [6](#0-5) 

**Why existing checks do not stop it:**

- `atMostOneNonZero` is per-offer, not per-group — mixed groups are permitted.
- `zeroFloorSub` prevents underflow/revert inside `consumableUnits` itself, but silently returns 0, masking the error.
- The core `take()` would accept a fill of Offer B (its own `consumed` accounting for Offer B starts at 0 assets), but the periphery never reaches `take()` because `unitsToTake = 0`.

### Impact Explanation

A taker using any `MidnightBundles` bundle function that relies on `consumableUnits` to size fills will have valid `maxAssets` offers silently zeroed out whenever the shared group `consumed` value (set by a prior `maxUnits` fill) numerically meets or exceeds `maxAssets`. The transaction reverts with `OutOfOffers()` even though sufficient offer capacity exists on-chain. The taker loses gas and fails to execute a time-sensitive trade (e.g., borrowing before maturity, or filling at a favorable price). In the partial-mismatch case (consumed units < maxAssets), the taker is under-filled by an amount proportional to the denomination difference, receiving fewer units than the offer actually permits.

### Likelihood Explanation

**Preconditions:**
- A maker must have two offers in the same group with different max types (`maxUnits` vs `maxAssets`). This is not prevented by the protocol.
- The `maxUnits` offer must be partially or fully filled before the taker attempts to fill the `maxAssets` offer via the periphery.

**Feasibility:** Moderate. The documentation recommends homogeneous groups but does not enforce it. A maker building an OCO strategy across different markets or offer types could inadvertently mix denominations. The condition is also triggerable by any prior taker who fills the `maxUnits` offer.

**Repeatability:** Deterministic once the group state is set. Every subsequent periphery call for the `maxAssets` offer in that group will compute the wrong remaining capacity until `setConsumed` is called to reset the group.

### Recommendation

In `consumableUnits`, track the denomination of the `consumed` value by branching on the offer type before subtracting:

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    if (offer.maxUnits > 0) {
        // consumed is units-denominated for this offer type
        return offer.maxUnits.zeroFloorSub(consumed);
    } else if (offer.buy) {
        // consumed is assets-denominated only if this offer's own fills set it;
        // if the group was previously consumed by a maxUnits offer, consumed is
        // units-denominated and must not be subtracted from maxAssets directly.
        // Safe approach: treat consumed as assets only when offer.maxAssets > 0
        // and add a group-homogeneity invariant, or expose per-offer consumed tracking.
        return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
    } else {
        return TakeAmountsLib.sellerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
    }
}
```

The correct fix is either: (a) enforce at the core level that all offers in a group share the same max type (add a group registry), or (b) document and enforce in `MidnightBundles` that all `takes` in a bundle must have homogeneous max types within each group, reverting if mixed types are detected. The periphery could check `offer.maxUnits > 0` vs `offer.maxAssets > 0` across all takes sharing the same `(maker, group)` pair and revert with a descriptive error before computing `consumableUnits`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Offer} from "src/Midnight.sol";
import {ConsumableUnitsLib} from "src/periphery/ConsumableUnitsLib.sol";
import {TakeAmountsLib} from "src/periphery/TakeAmountsLib.sol";

contract ConsumableUnitsDenominationTest is Test {
    // Setup: deploy Midnight, create market, create maker with two offers in same group G:
    //   offerA: maxUnits=1000, maxAssets=0
    //   offerB: maxUnits=0,    maxAssets=500, offer.buy=true
    // Both offers share the same (maker, group) pair.

    function testDimensionalMismatch() public {
        // ... standard Midnight test setup (market, loanToken, collateral) ...

        bytes32 group = keccak256("G");
        Offer memory offerA = /* maxUnits=1000, group=G */;
        Offer memory offerB = /* maxAssets=500, group=G, buy=true */;

        // Step 1: Partially fill offerA for 600 units.
        // consumed[maker][G] = 600 (units-denominated)
        take(offerA, 600, taker);
        assertEq(midnight.consumed(maker, group), 600);

        // Step 2: Call consumableUnits for offerB.
        bytes32 id = midnight.toId(offerB.market);
        uint256 result = ConsumableUnitsLib.consumableUnits(address(midnight), id, offerB);

        // BUG: result == 0 because 500.zeroFloorSub(600) == 0
        // Expected: buyerAssetsToUnits(500) > 0 (offerB has never been filled)
        assertEq(result, 0, "consumableUnits incorrectly returns 0 for a fully-available maxAssets offer");

        // Confirm offerB is actually fillable at the core level (take does not revert for units > 0)
        // consumed for offerB in assets mode starts at 0 from offerB's perspective,
        // but the shared consumed[maker][G] = 600 would cause take() to revert with ConsumedAssets
        // if 600 > 500 — showing the group is now poisoned for offerB at the core level too.

        // Invariant assertion: consumableUnits must return > 0 when no assets have been consumed
        // against offerB's maxAssets cap.
        // This assertion FAILS, proving the bug.
        assertTrue(result > 0, "INVARIANT VIOLATED: valid maxAssets offer reports zero consumable units");
    }
}
```

**Expected assertions:**
- `consumableUnits(offerB)` should return `buyerAssetsToUnits(500)` (the full capacity of Offer B).
- Instead it returns `0` (because `500 - 600` saturates to 0).
- A subsequent `MidnightBundles` call with only Offer B in the `takes` array reverts with `OutOfOffers()`.
- A direct `midnight.take(offerB, ..., someUnits, ...)` also reverts with `ConsumedAssets()` because `consumed[maker][G] = 600 > maxAssets = 500`, confirming the group is poisoned at the core level as well — the maker's `maxAssets` offer is permanently blocked until `setConsumed` is called.
