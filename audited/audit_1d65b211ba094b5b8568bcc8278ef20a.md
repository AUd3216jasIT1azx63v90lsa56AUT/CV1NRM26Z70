### Title
Buy offer `maxAssets` cap bypass via `mulDivDown` zero-rounding allows unbounded unit overfill - (File: src/Midnight.sol)

### Summary
When a buy offer has `maxAssets > 0` and `buyerPrice < WAD` (any tick strictly below `MAX_TICK`), a taker can call `take` with `units` small enough that `mulDivDown(units, buyerPrice, WAD) = 0`. Because `consumed` is incremented by `buyerAssets`, which is zero, the `ConsumedAssets` guard never triggers and the offer can be filled for arbitrarily many units beyond the maker's intended cap. The protocol's own test suite names and demonstrates this exact path in `testBugBuyMaxAssetsBypass`.

### Finding Description
**Code path** — `src/Midnight.sol` lines 363–369:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // only asset-level guard
}
```

**Root cause** — `mulDivDown(units, buyerPrice, WAD)` truncates to zero whenever `units * buyerPrice < WAD`. Because `tickToPrice` is proven to return values `<= WAD` (and equals `WAD` only at `MAX_TICK`), every tick below `MAX_TICK` yields `buyerPrice < WAD`. Choosing `units = 1` (or any value below `WAD / buyerPrice`) makes `buyerAssets = 0`.

**Attacker-controlled inputs** — `units` (any value satisfying `units * buyerPrice < WAD`), `offer.tick` (any tick below `MAX_TICK`).

**Exploit flow**:
1. Maker (lender) publishes a buy offer with `maxAssets = N`, `tick < MAX_TICK`.
2. Offer reaches fully-consumed state: `consumed[maker][group] = N`.
3. Taker calls `take(units=1, taker, offer, ...)` where `1 * buyerPrice < WAD`.
4. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`.
5. `consumed += 0` → `newConsumed = N` → `require(N <= N)` passes.
6. Position state mutates: maker's credit increases by 1 unit, taker's debt increases by 1 unit, zero tokens transferred.
7. Step 3–6 can be repeated indefinitely; each iteration adds 1 unit to the maker's credit with no asset cost and no consumed increment.

**Why existing checks fail** — The `ConsumedAssets` guard at line 369 is purely asset-denominated. It cannot detect unit-level overfill when `buyerAssets` rounds to zero. The Certora `takeConsumedBoundedByMax` rule (Consume.spec line 62) only asserts `consumed <= maxAssets`, which is trivially satisfied when the increment is zero. No unit-level invariant is formally verified for the `maxAssets` branch.

The protocol's own NatSpec at line 94 acknowledges: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test `testBugBuyMaxAssetsBypass` (TakeTest.sol line 858) confirms the state change is real: `creditOf(lender)` and `debtOf(borrower)` both increase while `consumed` and token balances remain unchanged.

### Impact Explanation
The maker's `maxAssets` cap is completely ineffective in unit terms when `buyerPrice < WAD`. A taker can fill an unbounded number of units on a fully-consumed buy offer, giving the maker more credit than they intended and imposing matching debt on themselves. This directly violates the core invariant that "offers cannot be overfilled."

### Likelihood Explanation
The precondition (`buyerPrice < WAD`) holds for every tick except `MAX_TICK` (5820), covering the vast majority of real offers. No special permissions are required; any unprivileged taker can trigger this. The attack is repeatable in a single transaction via `multicall` or across multiple transactions. The only cost to the taker is the gas for each call and the debt they accumulate (which they may not care about if they have sufficient collateral or intend to repay).

### Recommendation
Add a guard that rejects non-zero `units` when the resulting `buyerAssets` (or `sellerAssets`) rounds to zero in the `maxAssets` branch:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetDelta());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` threshold such that `units >= WAD / buyerPrice` (ceiling division) before accepting a non-zero take on a `maxAssets`-capped buy offer.

### Proof of Concept
The existing Foundry test `testBugBuyMaxAssetsBypass` (TakeTest.sol line 858) already constitutes a passing unit PoC. A fuzz/invariant extension:

```solidity
function invariant_maxAssetsBuyOfferUnitCap(
    uint256 units, uint256 tick, uint256 maxAssets
) public {
    // Setup: buy offer, maxAssets = maxAssets, tick < MAX_TICK
    // Fuzz: call take(units, ...) repeatedly
    // Assert after each take:
    uint256 totalUnitsFilledByMaker = midnight.creditOf(id, maker);
    uint256 buyerPrice = computeBuyerPrice(tick);
    // Invariant: totalUnitsFilledByMaker * buyerPrice / WAD <= maxAssets
    assertLe(
        totalUnitsFilledByMaker.mulDivDown(buyerPrice, WAD),
        maxAssets,
        "unit overfill beyond maxAssets intent"
    );
}
```

    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);
