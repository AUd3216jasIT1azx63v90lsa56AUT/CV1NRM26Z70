### Title
Authorized address can de-ratify maker's offer root, permanently freezing all takes - (`src/ratifiers/SetterRatifier.sol`)

### Summary
`SetterRatifier.setIsRootRatified` applies the same coarse-grained `isAuthorized` check for both ratifying (`true`) and de-ratifying (`false`) a root. Any address that a maker has authorized in Midnight for any purpose can call `setIsRootRatified(maker, root, false)`, setting `isRootRatified[maker][root] = false` and causing every subsequent `take` on that maker's offer tree to revert with `NotRatified`.

### Finding Description
**Code path:**
- [1](#0-0)  — the single guard `maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender)` is applied uniformly for both `true` and `false` values of `newIsRootRatified`.
- [2](#0-1)  — `require(isRootRatified[offer.maker][root], NotRatified())` hard-reverts every `take` if the root is de-ratified.

**Root cause:** The authorization check does not distinguish between a write of `true` (ratify, benign delegation) and a write of `false` (de-ratify, destructive). Midnight's `isAuthorized` mapping is a flat boolean with no action-scoping; granting authorization for any purpose (e.g., to allow an automated taker to act on the maker's behalf) simultaneously grants the power to destroy the maker's ratification state.

**Exploit flow:**
1. makerB calls `midnight.setIsAuthorized(makerA, true, makerB)` for any legitimate reason (e.g., delegating take authority).
2. makerA calls `setterRatifier.setIsRootRatified(makerB, root, false)`.
3. The guard passes: `IMidnight(MIDNIGHT).isAuthorized(makerB, makerA) == true`.
4. `isRootRatified[makerB][root]` is set to `false`.
5. Every call to `midnight.take(offer, ...)` where `offer.maker == makerB` and the ratifier data encodes `root` now reverts `NotRatified`.

**Why existing checks fail:** There is no check on the value of `newIsRootRatified`. The `Unauthorized()` revert only fires if `msg.sender` is neither the maker nor an authorized address — it does not restrict what an authorized address may write.

### Impact Explanation
makerB's entire offer tree (all offers whose Merkle root is `root`) is permanently frozen for takes. Lenders/borrowers whose positions depend on those offers being fillable cannot execute; funds committed to those offers are effectively locked until makerB re-ratifies the root — but makerA can immediately de-ratify again in the same transaction or block, making recovery impossible while the authorization persists.

### Likelihood Explanation
Precondition: makerB must have called `setIsAuthorized(makerA, true, makerB)` at any prior point. This is a normal, expected operation (e.g., delegating to a keeper or co-signer). The authorization is coarse-grained, so any such grant is sufficient. The attack is a single external call, requires no special funds, and is immediately repeatable. makerB cannot atomically revoke authorization and re-ratify without a race condition.

### Recommendation
Restrict de-ratification (`false`) to the maker only; allow authorized addresses only to ratify (`true`):

```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    if (newIsRootRatified) {
        require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    } else {
        require(maker == msg.sender, Unauthorized());
    }
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
```

This preserves the delegation use-case (automated ratification) while ensuring only the maker can revoke their own roots.
