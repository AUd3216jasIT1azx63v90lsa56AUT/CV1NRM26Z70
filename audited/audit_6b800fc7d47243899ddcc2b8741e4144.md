### Title
Post-badDebt-realization `supplyCollateral` via `onLiquidate` callback inflates bad debt, over-slashing lenders - (File: src/Midnight.sol)

### Summary

In `liquidate()`, the `badDebt` computation and its application to `lossFactor`/`_position.debt` are fully committed before the `onLiquidate` callback fires. An authorized callback can call `supplyCollateral` during that window, adding collateral to the borrower's position that was never counted in `badDebt`. Lenders are slashed for a larger shortfall than the true post-liquidation one.

### Finding Description

**Exact code path:**

`liquidate()` in `src/Midnight.sol`:

1. **Line 606** — snapshot of `_collateralBitmap` is taken into a local variable.
2. **Lines 607–618** — the `while` loop iterates over that snapshot, computing `maxDebt` and `badDebt` from collateral present at that moment.
3. **Lines 626–641** — if `badDebt > 0`, it is immediately committed: `_position.debt -= uint128(badDebt)`, `_marketState.lossFactor` is updated (lenders slashed), `_marketState.totalUnits` is reduced.
4. **Lines 670–676** — optional seizure/repayment applied.
5. **Line 696** — seized collateral transferred out.
6. **Lines 698–715** — `onLiquidate` callback fires. **No lock prevents re-entry into `supplyCollateral`.**
7. **Line 717** — loan token pulled from payer. [1](#0-0) [2](#0-1) [3](#0-2) 

`supplyCollateral()` at lines 524–546 only requires `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. If the borrower has previously called `setIsAuthorized(callback, true, borrower)`, the callback contract satisfies this check and can supply any amount of any collateral index on behalf of the borrower. [4](#0-3) 

**Exploit flow:**

**Preconditions:**
- Borrower has a bad-debt position: `debt D`, collateral `C` at index 0 with price `P`, such that `C * P / ORACLE_PRICE_SCALE * WAD / maxLif < D`.
- Borrower has executed `setIsAuthorized(callbackContract, true, borrower)` at any prior time.
- `callbackContract` holds (or can obtain) collateral tokens for index `newIndex`.

**Steps:**
1. Liquidator calls `liquidate(market, 0, 0, 0, borrower, false, receiver, callbackContract, data)`.
2. `badDebt = D - C*P/ORACLE_PRICE_SCALE*WAD/maxLif` is computed and applied — lenders' `lossFactor` is updated, borrower's debt reduced.
3. `onLiquidate` fires on `callbackContract`.
4. Inside the callback: `callbackContract` calls `supplyCollateral(market, newIndex, W, borrower)`.
   - `_position.collateral[newIndex] = W` is written.
   - Bit `newIndex` is set in `_position.collateralBitmap`.
   - `W` collateral tokens are transferred from `callbackContract` to Midnight.
5. Callback returns `CALLBACK_SUCCESS`.
6. `liquidate()` completes normally.

**Why existing checks fail:**

- `liquidationLocked` is only SET inside `take` (for the seller), never inside `liquidate`. The callback is free to call any Midnight function.
- There is no reentrancy guard on `liquidate`.
- `supplyCollateral`'s only guard is the authorization check, which passes because the borrower pre-authorized the callback.
- The `badDebt` value was already finalized and emitted before the callback; there is no re-computation after it returns. [5](#0-4) [6](#0-5) 

### Impact Explanation

After the callback, the borrower's position holds collateral worth `W * P2 / ORACLE_PRICE_SCALE` (in loan-token terms) at index `newIndex` that was never subtracted from `badDebt`. Lenders are over-slashed by:

```
over_slash = W * P2 / ORACLE_PRICE_SCALE * WAD / maxLif2   (loan-token units)
```

`_marketState.lossFactor` is permanently inflated by this amount, so every lender who calls `updatePosition` after the attack has their credit reduced by more than the true shortfall. The borrower's debt is reduced by `badDebt` (the inflated figure) while they retain the `W` collateral, giving them a net benefit of `over_slash` units of debt reduction at lenders' expense. The invariant *"every credit has matching debt or valid settled/loss state"* is broken: the sum of lender credits after slashing is less than the sum of borrower debts plus withdrawable assets.

### Likelihood Explanation

**Preconditions are low-friction:**
- Any borrower whose position reaches bad-debt territory (a normal market event) can execute this.
- `setIsAuthorized` is a standard, permissionless call the borrower makes on their own account.
- The callback contract needs collateral tokens; these can be pre-funded or sourced via a flash loan from Midnight itself (the `flashLoan` entry point is available inside the callback window).
- No admin action, no oracle manipulation, and no special role is required.

The attack is repeatable: a borrower can set up the authorization once and exploit every future bad-debt liquidation of their position.

### Recommendation

Compute and apply `badDebt` **after** the callback returns, or take a post-callback snapshot and recompute. The simplest correct fix is to move the bad-debt block (lines 626–641) and the `repaidUnits`/`seizedAssets` block (lines 643–677) to **after** line 715 (after the callback returns), so that any collateral added during the callback is included in the `badDebt` calculation.

Alternatively, set the `LIQUIDATION_LOCK_SLOT` for the borrower at the start of `liquidate` (and clear it at the end) to block `supplyCollateral` — but this requires `supplyCollateral` to check the lock, which it currently does not.

A targeted fix: re-read `_position.collateralBitmap` and re-run the `badDebt` loop after the callback, then apply only the delta to `lossFactor`.
