### Title
Undifferentiated `isAuthorized` Mapping Grants Ratifier-Authorized Addresses Full Protocol Access Including Privilege Escalation via `setIsAuthorized` ŌĆö (File: src/Midnight.sol)

---

### Summary

The `isAuthorized[onBehalf][msg.sender]` mapping in `Midnight.sol` is a flat, undifferentiated authorization used simultaneously for two distinct purposes: (1) ratifier authorization (a specific, scoped intent ŌĆö allowing a contract to validate offers), and (2) general operation authorization (granting full access to every state-mutating function, including `setIsAuthorized` itself). This mirrors the external report's root cause exactly: a shared executor holding permissions in two systems with different access-control levels. When a maker authorizes a ratifier, they inadvertently grant it the ability to authorize arbitrary third parties on their behalf, enabling full position drain without any further user interaction.

---

### Finding Description

**Root cause ŌĆö dual use of a single flat mapping:**

In `take()`, the ratifier authorization check is:

```solidity
require(isAuthorized[offer.maker][offer.ratifier], RatifierUnauthorized());
``` [1](#0-0) 

Every other state-mutating function (`withdraw`, `repay`, `supplyCollateral`, `withdrawCollateral`, `setConsumed`, and critically `setIsAuthorized`) uses the identical mapping for general operation authorization:

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
``` [2](#0-1) 

The protocol itself acknowledges this in the `AUTHORIZATIONS` natspec:

> "authorized accounts can authorize other accounts on behalf of the user" [3](#0-2) 

**The shared mapping is also re-used by peripheral contracts**, compounding the scope:

- `EcrecoverRatifier.cancelRoot` checks `IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender)` ŌĆö any address authorized in Midnight can cancel the maker's entire offer-root tree.
- `SetterRatifier.setIsRootRatified` checks `IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender)` ŌĆö any address authorized in Midnight can de-ratify the maker's offer roots. [4](#0-3) [5](#0-4) 

**Exploit path (privilege escalation ŌåÆ full drain):**

1. Maker calls `setIsAuthorized(maliciousRatifier, true, maker)` ŌĆö intending only to allow offer validation.
2. `maliciousRatifier` calls `Midnight.setIsAuthorized(attacker, true, maker)` ŌĆö this succeeds because `isAuthorized[maker][maliciousRatifier] == true`.
3. `attacker` calls `Midnight.withdraw(market, maker.credit, maker, attacker)` ŌĆö drains all credit.
4. `attacker` calls `Midnight.withdrawCollateral(market, i, maker.collateral[i], maker, attacker)` ŌĆö drains all collateral (if maker has no debt, health check passes trivially). [6](#0-5) [7](#0-6) 

The test suite explicitly confirms that an authorized address can re-authorize further addresses:

```solidity
vm.prank(authorized);
midnight.setIsAuthorized(newAuthorized, true, user);
assertEq(midnight.isAuthorized(user, newAuthorized), true);
``` [8](#0-7) 

---

### Impact Explanation

A maker who authorizes any ratifier (the required step to use `EcrecoverRatifier` or `SetterRatifier`) inadvertently grants that ratifier ŌĆö and any address it subsequently authorizes ŌĆö the ability to:

- Drain all credit via `withdraw`
- Drain all collateral via `withdrawCollateral`
- Cancel all pending offer roots via `EcrecoverRatifier.cancelRoot`
- De-ratify all offer roots via `SetterRatifier.setIsRootRatified`
- Propagate the authorization to arbitrary additional addresses via `setIsAuthorized`

The impact is **complete loss of all assets held in Midnight positions** for affected makers.

---

### Likelihood Explanation

Every maker who uses `EcrecoverRatifier` or `SetterRatifier` must call `setIsAuthorized(ratifier, true, maker)` ŌĆö this is the mandatory setup step shown in every integration test and the test harness `BaseTest.sol`. [9](#0-8) 

The authorization scope is not visible in the ratifier's interface; a maker reading `EcrecoverRatifier` sees only `isRatified` and `cancelRoot`, with no indication that the authorization also covers all core Midnight operations. A malicious or compromised ratifier deployed to look legitimate can exploit this silently after authorization. The likelihood is **medium**: it requires a malicious ratifier to be deployed and authorized, but the structural precondition (every maker must authorize their ratifier) is universally present.

---

### Recommendation

Separate ratifier authorization from general operation authorization. Options include:

1. **Dedicated ratifier mapping**: Add a separate `isRatifierAuthorized[maker][ratifier]` mapping checked only in `take()`, distinct from the general `isAuthorized` mapping used by all other functions.
2. **Scoped authorization with function selectors**: Extend `setIsAuthorized` to accept a bitmask or list of permitted function selectors, so a maker can authorize a ratifier only for `take`-related validation without granting access to `withdraw`, `withdrawCollateral`, or `setIsAuthorized`.
3. **Peripheral contracts use dedicated authorization**: `EcrecoverRatifier.cancelRoot` and `SetterRatifier.setIsRootRatified` should check a ratifier-specific authorization rather than the general Midnight `isAuthorized` mapping.

---

### Proof of Concept

```solidity
// 1. Deploy malicious ratifier that always returns CALLBACK_SUCCESS
//    and has a backdoor to call setIsAuthorized.
MaliciousRatifier rat = new MaliciousRatifier(address(midnight));

// 2. Maker authorizes the ratifier (standard required setup).
vm.prank(maker);
midnight.setIsAuthorized(address(rat), true, maker);

// 3. Malicious ratifier escalates privilege to attacker.
//    This succeeds because isAuthorized[maker][rat] == true.
vm.prank(address(rat));
midnight.setIsAuthorized(attacker, true, maker);

// 4. Attacker drains maker's credit.
vm.prank(attacker);
midnight.withdraw(market, midnight.creditOf(id, maker), maker, attacker);

// 5. Attacker drains maker's collateral (if no debt).
vm.prank(attacker);
midnight.withdrawCollateral(market, 0, midnight.collateral(id, maker, 0), maker, attacker);

// Result: maker's entire position is drained with no further user interaction.
``` [10](#0-9) [1](#0-0)
