### Title
Authorized Address Can Sub-Delegate Signing Rights to Attacker Without Maker Consent - (File: src/ratifiers/EcrecoverRatifier.sol)

### Summary
`Midnight.setIsAuthorized` permits any address already authorized by a maker to grant that same authorization to arbitrary third parties on the maker's behalf. Because `EcrecoverRatifier.isRatified` accepts any signer for whom `isAuthorized(offer.maker, _signer)` returns true, an attacker who receives a sub-delegated authorization can produce valid offer signatures that Midnight treats as coming from the maker, without the maker ever directly consenting to the attacker.

### Finding Description

**Root cause — `Midnight.setIsAuthorized` (line 731–733):**

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
```

The guard only checks that `msg.sender` is already authorized by `onBehalf`; it places no restriction on *who* `authorized` may be. An authorized address therefore has the full power to write any entry into `isAuthorized[onBehalf][*]`. [1](#0-0) 

**Ratifier check — `EcrecoverRatifier.isRatified` (line 44):**

```solidity
require(_signer == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer), Unauthorized());
```

Any address for which `isAuthorized[maker][_signer]` is true passes this check, regardless of how that entry was written. [2](#0-1) 

**Exploit flow (all calls are attacker-controlled after step 1):**

1. Maker legitimately calls `midnight.setIsAuthorized(A, true, maker)` — e.g., to let A manage positions.
2. A (now authorized) calls `midnight.setIsAutgorized(attacker, true, maker)`. The guard `isAuthorized[maker][A] == true` passes; the mapping is updated: `isAuthorized[maker][attacker] = true`.
3. Attacker constructs any offer with `offer.maker = maker`, builds a valid Merkle root over it, and signs the root with their own key.
4. Anyone calls `midnight.take(offer, ratifierData, ...)`. Midnight calls `ecrecoverRatifier.isRatified`. The recovered signer is the attacker; `isAuthorized(maker, attacker)` returns `true`; the ratifier returns `CALLBACK_SUCCESS`.
5. The offer executes as if the maker signed it.

**Why existing checks do not stop it:**

- The `setIsAuthorized` guard is satisfied by A's existing authorization; there is no "only the principal can delegate" restriction.
- `isRatified` performs a flat, single-level lookup — it does not distinguish between authorizations granted directly by the maker and those granted by a delegate.
- The protocol's own test `testSetIsAuthorizedAuthorization` explicitly asserts this three-party delegation succeeds, confirming the path is reachable. [3](#0-2) 

### Impact Explanation
An attacker who receives a sub-delegated authorization (or who is themselves a legitimate authorized address) can sign arbitrary offers on behalf of any maker. This bypasses the `EcrecoverRatifier` gate entirely: the attacker can create buy or sell offers that drain the maker's supplied liquidity, open debt positions against the maker's collateral, or trigger liquidations — all without the maker's direct consent. The scoped impact (ratifier bypass allowing unauthorized access to restricted markets, credit/debt changes) is fully realized.

### Likelihood Explanation
The precondition is that the maker has authorized at least one other address for any reason (e.g., an operator, a frontend relayer, or the `EcrecoverAuthorizer` contract). This is the normal operating mode of the protocol. Once any authorized address exists, a single additional transaction from that address completes the attack. The step is repeatable and requires no special privileges beyond being an authorized address.

### Recommendation
Restrict `setIsAuthorized` so that only the principal (`onBehalf == msg.sender`) can grant new authorizations; authorized delegates should not be permitted to sub-delegate. Alternatively, add a separate, non-delegatable "root authorization" level and restrict `isRatified` to only accept signers that were authorized directly by the maker (i.e., track the granting principal and reject entries written by delegates).

### Proof of Concept

```solidity
// Foundry unit test
function testAttackerSubDelegateSigningRights() public {
    // Setup: maker is lender, A is borrower, attacker is a fresh address
    address A = borrower;
    (address attacker, uint256 attackerKey) = makeAddrAndKey("attacker");

    // Step 1: maker authorizes A for some legitimate purpose
    vm.prank(lender);
    midnight.setIsAuthorized(A, true, lender);

    // Step 2: A sub-delegates to attacker on maker's behalf
    vm.prank(A);
    midnight.setIsAuthorized(attacker, true, lender);

    // Assert: isAuthorized[maker][attacker] == true without maker's direct action
    assertTrue(midnight.isAuthorized(lender, attacker));

    // Step 3: attacker signs an offer root for maker
    Offer memory offer = makeOffer(lender); // offer.maker = lender
    bytes32 _root = HashLib.hashOffer(offer);
    Signature memory sig = signature(_root, attackerKey, address(ecrecoverRatifier), 0);
    bytes memory ratifierData = abi.encode(sig, _root, 0, new bytes32[](0));

    // Step 4: isRatified accepts attacker's signature
    vm.prank(address(midnight));
    bytes32 result = ecrecoverRatifier.isRatified(offer, ratifierData);
    assertEq(result, CALLBACK_SUCCESS); // passes — attacker signs for maker
}
```

Expected assertion: `result == CALLBACK_SUCCESS`, confirming the attacker's signature is accepted as valid for the maker's offer without the maker ever authorizing the attacker directly.
