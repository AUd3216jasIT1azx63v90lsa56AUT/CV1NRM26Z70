Audit Report

## Title
Buy Offer `maxAssets` Cap Fully Bypassed via Rounding-to-Zero `buyerAssets` on Small-Unit Fills - (File: src/Midnight.sol)

## Summary
When a buy offer uses `maxAssets`-based consumption tracking and `tickToPrice(offer.tick) < WAD` (any tick below `MAX_TICK = 5820`), calling `take()` with `units = 1` causes `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. Because `consumed` is incremented by `buyerAssets`, it never advances, and the guard `require(newConsumed <= offer.maxAssets)` passes unconditionally on every call. Position state (credit, debt, `totalUnits`) mutates while no loan tokens are transferred, allowing unbounded credit accumulation at zero cost.

## Finding Description
**Root cause** — `src/Midnight.sol` lines 363–369:

```solidity
// line 363
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;
// line 368
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
// line 369
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

For a buy offer, `buyerPrice = offerPrice` (the settlement fee cancels: `sellerPrice = offerPrice - fee`, `buyerPrice = sellerPrice + fee = offerPrice`). `tickToPrice` returns exactly `WAD` only at `MAX_TICK = 5820`; every tick below that yields a strictly smaller price. This is proven by the Certora rule `tickToPriceIsOneAtMaxTick` and `tickToPriceAtMostWad` in `certora/specs/TickToPrice.spec`, and confirmed by `testTickToPriceMinMax` in `test/TickLibTest.sol`.

When `units = 1` and `buyerPrice < WAD`:
```
buyerAssets = mulDivDown(1, buyerPrice, WAD) = floor(buyerPrice / WAD) = 0
```

`consumed` increments by 0. The check `0 <= maxAssets` is trivially satisfied regardless of how many times it has already been called. The token transfer at lines 455–456 (`safeTransferFrom(payer, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(payer, receiver, sellerAssets)`) transfers 0 tokens. Yet the position mutations at lines 408–417 still execute: `buyerPos.credit += buyerCreditIncrease`, `sellerPos.debt += sellerDebtIncrease`, and `_marketState.totalUnits` increases.

**Existing checks and why they fail:**
- `require(newConsumed <= offer.maxAssets)` — never triggered because `newConsumed` never increases.
- `require(offer.maker != taker, SelfTake())` — trivially bypassed with two addresses.
- `isHealthy` on seller — satisfied as long as the taker supplies collateral.

**Confirmed by test** `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` lines 858–889: the test pre-sets `consumed = maxAssets = 1`, then calls `take(1, borrower, lenderOffer)` with `tick = MAX_TICK - 16`, and asserts `buyerAssets == 0`, `consumed` unchanged, but `creditOf(lender)` increased, `debtOf(borrower)` increased, and `totalUnits` increased.

## Impact Explanation
The `maxAssets` cap — the sole mechanism limiting how many units a buy offer can fill — is completely inoperative for any tick below `MAX_TICK`. A maker colluding with a separate taker address can loop `take(1, ...)` to accumulate an unbounded number of credit units for zero loan-token cost. At maturity, those credit units are redeemable for full face value in loan tokens. The taker address defaults on the corresponding debt; the protocol absorbs bad debt. This breaks the core accounting invariant that contract loan-token balances cover all outstanding credit, enabling direct theft of loan tokens from the protocol at the expense of other lenders.

## Likelihood Explanation
No privileged access is required. The precondition (any tick below `MAX_TICK`) covers the entire usable price range below par. The attack requires only two addresses (trivially satisfied), a buy offer with `maxAssets > 0`, and collateral on the taker side. It is repeatable in a single transaction via a loop or multicall; gas cost is the only practical limit.

## Recommendation
Add a guard that rejects fills where `units > 0` but `buyerAssets == 0` (for buy offers) or `sellerAssets == 0` (for sell offers). For example, after line 364:

```solidity
require(units == 0 || (offer.buy ? buyerAssets > 0 : sellerAssets > 0), ZeroAssets());
```

This ensures that every non-trivial fill advances the asset-denominated consumed counter by at least 1, making the `maxAssets` cap enforceable. Alternatively, enforce a minimum `units` value such that `units * price >= WAD` before allowing a fill against a `maxAssets`-capped offer.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) is a complete, self-contained reproduction:

1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (price < WAD).
2. Pre-consume the offer fully: `midnight.setConsumed(group, 1, lender)`.
3. Call `take(1, borrower, lenderOffer)`.
4. Observe: `buyerAssets == 0`, `consumed` still equals `maxAssets`, but `creditOf(lender)` increased, `debtOf(borrower)` increased, `totalUnits` increased.

Run with: `forge test --match-test testBugBuyMaxAssetsBypass`.
