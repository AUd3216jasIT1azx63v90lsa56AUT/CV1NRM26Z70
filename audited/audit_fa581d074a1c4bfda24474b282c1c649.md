### Title
Fee-on-Transfer Loan Token Causes `claimableSettlementFee` Overstatement, Breaking Solvency Invariant in `take()` - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee[loanToken]` is incremented by the nominal `buyerAssets - sellerAssets` before the actual `safeTransferFrom` call, and `SafeTransferLib.safeTransferFrom` performs no balance-before/after check. When the loan token is a fee-on-transfer token, Midnight receives less than the nominal fee amount, permanently overstating `claimableSettlementFee` relative to the contract's actual balance. This breaks the core solvency invariant and creates a race condition where either `claimSettlementFee()` or `withdraw()` reverts for the last caller.

### Finding Description

**Exact code path:**

`src/Midnight.sol` line 418 writes the accounting entry unconditionally before any transfer:

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

Then lines 455–456 execute the two transfers:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only checks the boolean return value; it does not measure the actual balance delta received by `address(this)`: [3](#0-2) 

**Root cause:** With a fee-on-transfer token charging fee rate `f`:
- Transfer 1 (to `address(this)`): Midnight receives `(buyerAssets - sellerAssets) × (1 - f)`, not `buyerAssets - sellerAssets`.
- Transfer 2 (to `receiver`): goes directly to the seller's receiver, never touches Midnight's balance.

`claimableSettlementFee` is overstated by exactly `(buyerAssets - sellerAssets) × f` per `take()` call.

**Attacker-controlled inputs:**
1. Deploy a fee-on-transfer ERC20 token (e.g., 1% fee on every `transferFrom`).
2. Call `touchMarket()` with this token as `loanToken` — permissionless, no admin required. [4](#0-3) 
3. Post a buy offer (as lender) and a sell offer (as borrower with collateral).
4. Call `take()` — the accounting discrepancy is written permanently.

**Why existing checks fail:**
- `SafeTransferLib` only validates the boolean return, not the received amount.
- There is no `require(balanceAfter - balanceBefore == expected)` guard anywhere in `take()`.
- The Certora formal verification explicitly excludes fee-on-transfer tokens with the assumption `"no fee taking from sender or receiver"`, confirming this is an unverified gap: [5](#0-4) 
- The `pendingFeeReceipt` ghost in the Certora spec tracks the gap between the accounting write and the transfer, but it is only cleared when `dest == currentContract && pendingFeeReceipt[token] == value` — a condition that is never satisfied with a fee-on-transfer token because the actual received value is less than `pendingFeeReceipt[token]`: [6](#0-5) 

### Impact Explanation

After one `take()` with a fee-on-transfer loan token:

```
claimableSettlementFee[token] = buyerAssets - sellerAssets
actual_balance_for_fees       = (buyerAssets - sellerAssets) × (1 - f)
```

The invariant `tokenBalance(Midnight) >= collateralSum + withdrawableSum + claimableSettlementFee` is broken. Concretely:

- If the fee claimer calls `claimSettlementFee(token, claimableSettlementFee[token], receiver)`, the `safeTransfer` at line 309 reverts because Midnight lacks sufficient balance. [7](#0-6) 
- If the fee claimer claims first (draining the actual balance), subsequent lender `withdraw()` calls revert at line 499 for the same reason. [8](#0-7) 

This is a permanent, non-recoverable state: every subsequent `take()` compounds the deficit. The race condition between the fee claimer and lenders is a direct, concrete DoS on core market actions.

### Likelihood Explanation

**Preconditions:**
- Market creation is fully permissionless; any address can call `touchMarket()` with any `loanToken`.
- The protocol explicitly claims support for "arbitrary loan token" and lists "token charges fee" as an external behavior test case in `live_context.json`. [9](#0-8) [10](#0-9) 

**Feasibility:** Trivial. The attacker deploys a standard fee-on-transfer ERC20, creates a market, and executes a single `take()`. No privileged access, no oracle manipulation, no timing dependency.

**Repeatability:** Every `take()` in the affected market compounds the deficit. The bug is deterministic and reproducible.

### Recommendation

Replace the two separate `safeTransferFrom` calls with a balance-before/after check, or restructure to pull the full `buyerAssets` in a single transfer and push `sellerAssets` out of Midnight's balance:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 received = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
require(received == buyerAssets - sellerAssets, InsufficientFeeReceived());
```

Alternatively, document that fee-on-transfer tokens are explicitly unsupported as loan tokens and add a validation in `touchMarket()` (though this is difficult to enforce on-chain without a test transfer). The cleaner fix is the balance-delta check.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));

                revert(add(returndata, 0x20), mload(returndata))
