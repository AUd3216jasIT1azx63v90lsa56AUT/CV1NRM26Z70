### Title
Fee-on-Transfer Token Accounting Discrepancy in `supplyCollateral` and `repay` Enables Over-Borrowing and Protocol Insolvency — (File: src/Midnight.sol)

---

### Summary

`Midnight.sol` updates internal accounting (collateral balances, withdrawable units) using the caller-supplied `assets`/`units` parameter before executing the token transfer, with no post-transfer balance check. When a fee-on-transfer token is used as a collateral or loan token, the protocol records more than it actually receives, inflating collateral positions and the `withdrawable` pool. This enables over-borrowing against phantom collateral and leaves lenders unable to redeem their full credit.

---

### Finding Description

**Root cause — `supplyCollateral`** [1](#0-0) 

```solidity
// line 533 — accounting written with caller-supplied value
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
// ...
// line 545 — actual transfer; only (assets - fee) arrives
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

The position is credited with `assets` collateral units, but the contract receives only `assets − fee`. No balance snapshot is taken before or after the transfer to reconcile the two.

**Root cause — `repay`** [2](#0-1) 

```solidity
// lines 508-509 — accounting written with caller-supplied value
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);
// ...
// line 520 — actual transfer; only (units - fee) arrives
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
```

`withdrawable` is incremented by `units`, but the contract receives only `units − fee`. Every repayment widens the gap between the recorded withdrawable pool and the actual token balance.

**Why the documented assumption does not protect the protocol**

The NatSpec at lines 133–140 lists "Midnight must send/receive exactly the requested amount on transfers" as a *token safety assumption*, not an enforced invariant. [3](#0-2) 

The protocol is permissionless: any address can call `touchMarket` with an arbitrary `loanToken` or `collateralParams[i].token`. There is no token allowlist, no registry check, and no balance-delta validation at any entry point. The assumption is therefore unenforced.

The `live_context.json` audit context explicitly lists this class as in-scope: [4](#0-3) 

> "fee-on-transfer, rebasing, false-return, ERC777-like hooks, and non-standard decimals should be tested if not explicitly excluded"

`SECURITY.md` contains no explicit exclusion for fee-on-transfer tokens. [5](#0-4) 

---

### Impact Explanation

**Vector A — `supplyCollateral` with fee-on-transfer collateral token**

1. Attacker supplies `100e18` of a 10%-fee collateral token.
2. Protocol receives `90e18` tokens; position records `100e18` collateral.
3. Attacker borrows against the inflated `100e18` collateral value (e.g., `80e18` loan tokens at LLTV = 0.8).
4. Attacker walks away with the loan. Liquidator attempts to seize `100e18` collateral but the contract only holds `90e18`, causing the liquidation to revert or leave unrecoverable bad debt.
5. Lenders absorb the loss via the `lossFactor` socialization mechanism.

**Vector B — `repay` with fee-on-transfer loan token**

1. Borrower repays `100e18` units; `withdrawable` increases by `100e18`; contract receives `90e18`.
2. Repeated repayments accumulate a deficit: `withdrawable > actual balance`.
3. Lenders calling `withdraw` will eventually hit a transfer revert (insufficient balance), permanently freezing a portion of their credit. [6](#0-5) 

The `withdraw` function transfers `units` tokens unconditionally; if the balance is short, the call reverts and the lender's credit is permanently locked.

---

### Likelihood Explanation

- Market creation is fully permissionless (`touchMarket` has no token validation).
- Fee-on-transfer tokens exist on mainnet (reflection tokens, tokens with configurable transfer taxes).
- No privileged role is required; any unprivileged user can create a market or supply collateral.
- The attacker does not need to own or control the token contract — they only need to use an existing fee-on-transfer token.

---

### Recommendation

In both `supplyCollateral` and `repay`, record a balance snapshot before the transfer and use the actual received delta as the accounting value:

**`supplyCollateral`:**
```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

**`repay`:**
```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
require(received == units, TokensWithFeesNotSupported());
```

Alternatively, add an explicit revert if `received != units` to enforce the documented assumption at the contract level rather than relying on off-chain documentation.

    function supplyCollateral(Market memory market, uint256 collateralIndex, uint256 assets, address onBehalf)
        external
    {
        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
        bytes32 id = touchMarket(market);
        address collateralToken = market.collateralParams[collateralIndex].token;

        Position storage _position = position[id][onBehalf];
        uint256 oldCollateral = _position.collateral[collateralIndex];
        _position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);

        if (oldCollateral == 0 && assets > 0) {
            uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);

                UtilsLib.countBits(newCollateralBitmap) <= MAX_COLLATERALS_PER_BORROWER, TooManyActivatedCollaterals()
            );
