Audit Report

## Title
Post-maturity `seizedAssets` liquidation reverts on bad-debt positions when `lif < maxLif` - (`src/Midnight.sol`)

## Summary
When a position carries bad debt and a liquidator calls `liquidate` with `seizedAssets > 0` and `postMaturityMode = true` before `TIME_TO_MAX_LIF` (900 s) has elapsed, the bad-debt reduction at line 628 uses `maxLif` to shrink `_position.debt`, while `repaidUnits` at line 650 is computed with the smaller ramping `lif`. Because `lif < maxLif`, the resulting `repaidUnits` exceeds the reduced `_position.debt`, and the unchecked subtraction at line 676 reverts with an arithmetic underflow. No guard in the `postMaturityMode` branch caps `repaidUnits` at `_position.debt`.

## Finding Description

**Root cause:** The bad-debt loop (lines 607–618) computes each collateral's maximum supportable debt using `maxLif` (line 615), which is the largest possible liquidation incentive factor. After subtracting `badDebt` at line 628, `_position.debt` is bounded by `Σ collateral[i] * price_i / ORACLE_PRICE_SCALE * WAD / maxLif_i`. However, when `block.timestamp < market.maturity + TIME_TO_MAX_LIF`, the `lif` computed at lines 645–647 is strictly less than `maxLif`. The `seizedAssets` branch at line 650 then computes `repaidUnits = seizedAssets * price / ORACLE_PRICE_SCALE * WAD / lif`, which is larger per unit of collateral than the debt ceiling established by the bad-debt step. The `maxRepaid` / RCF guard at lines 655–668 is entirely skipped because it is gated on `!postMaturityMode`. The subtraction `_position.debt -= UtilsLib.toUint128(repaidUnits)` at line 676 therefore underflows and reverts.

**Concrete verification** (`ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`, `TIME_TO_MAX_LIF = 900 s`):

| Parameter | Value |
|---|---|
| `collateral[0]` | 901 |
| `price_0` | `ORACLE_PRICE_SCALE` |
| `maxLif_0` | `2 * WAD` |
| `originalDebt` | 452 |
| `Δt` | 1 s |

- Bad debt: `ceil(901/2) = 451`; `badDebt = 452 − 451 = 1`; `_position.debt → 451`
- `lif = WAD + WAD/900 ≈ 1.00111 * WAD` (integer truncation makes it slightly less than `901*WAD/900`)
- `repaidUnits = ceil(901 * WAD / lif) ≥ 900`
- `900 > 451` → **arithmetic underflow revert at line 676**

**Why existing checks fail:** The `maxRepaid` guard (lines 659–667) is the only mechanism that would cap `repaidUnits`, but it is unconditionally skipped when `postMaturityMode == true`. `UtilsLib.toUint128` only checks for uint128 overflow, not for the subtraction underflow.

## Impact Explanation
Any liquidator calling `liquidate(seizedAssets > 0, postMaturityMode = true)` on a position with bad debt during the 15-minute post-maturity ramp window receives an unconditional revert. The `seizedAssets`-input liquidation path — the natural choice when targeting a specific collateral amount — is completely broken for this combination of conditions. Workarounds exist (use `repaidUnits` input, or first call with `(0, 0)` to realize bad debt), but the primary collateral-seizure entry point silently fails, delaying timely liquidation and degrading protocol liveness during a critical window.

## Likelihood Explanation
All three preconditions are reachable by any unprivileged user without any special access:
1. Bad debt is a normal protocol outcome when collateral price falls below `debt / (collateral * maxLif / ORACLE_PRICE_SCALE)`.
2. The 15-minute ramp window is a fixed protocol constant (`TIME_TO_MAX_LIF = 15 minutes` in `src/libraries/ConstantsLib.sol`).
3. Any liquidator using the collateral-input form of the call hits this deterministically.

The revert is deterministic within the window and requires no coordination or privileged action.

## Recommendation
After computing `repaidUnits` in the `seizedAssets` branch (line 650), cap it at `_position.debt` in the `postMaturityMode` path. For example, after line 650:

```solidity
if (postMaturityMode) {
    repaidUnits = UtilsLib.min(repaidUnits, _position.debt);
    // recompute seizedAssets to match the capped repaidUnits if needed
}
```

Alternatively, restructure so that the bad-debt computation and the `lif` used for `repaidUnits` are consistent — either both use `maxLif` or both use the ramping `lif`. The simplest invariant to enforce is that `repaidUnits` never exceeds `_position.debt` before the subtraction at line 676, regardless of mode.
