Looking at the exact code path in `src/Midnight.sol` and `src/libraries/TickLib.sol`, plus the Certora spec gaps in `certora/specs/Consume.spec`.

### Title
Buy-offer `maxAssets` cap bypassed via zero-rounding `buyerAssets` — (`src/Midnight.sol`)

### Summary
When a buy offer is taken with `maxAssets > 0` and a tick where `tickToPrice(tick) < WAD`, the consumed increment is `buyerAssets = mulDivDown(units, buyerPrice, WAD)`. For any `units` satisfying `units * buyerPrice < WAD` (e.g. `units = 1` at any tick < `MAX_TICK`), integer division floors this to zero, so `consumed[maker][group]` never increments. The guard `require(newConsumed <= offer.maxAssets)` therefore never triggers, and the offer can be taken an unbounded number of times beyond the maker's intended cap.

### Finding Description

**Code path** — `src/Midnight.sol` lines 358–369:

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);          // line 358
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice; // 361
uint256 buyerPrice  = sellerPrice + _settlementFee;            // 362  → equals offerPrice for buy offers
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)                        // 363  ← rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group]
        += offer.buy ? buyerAssets : sellerAssets;             // 368  ← += 0 when buyerAssets == 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets()); // 369  ← never fires
}
```

**Root cause** — For a buy offer, `buyerPrice = offerPrice = tickToPrice(tick)`. The Certora spec `TickToPrice.spec` proves `tickToPrice(tick) == WAD` only at `MAX_TICK = 5820`, and `tickToPrice(tick) < WAD` for every tick below that (including `tickToPrice(0) == 0`, confirmed by `testTickToPriceMinMax`). `mulDivDown` is plain integer division: `(units * buyerPrice) / WAD`. For `units = 1` and `buyerPrice < WAD`, the product is less than `WAD`, so the result is 0. The consumed mapping is incremented by 0, the cap check passes trivially, and the take succeeds.

**Attacker-controlled inputs**:
- `offer.buy = true`, `offer.maxAssets > 0`, any `offer.tick` in `[0, MAX_TICK)` (i.e. `tickToPrice(tick) < WAD`)
- `units = 1` (or any value where `units * tickToPrice(tick) < WAD`)
- Attacker is the taker (seller); `offer.maker != taker` is satisfied

**Exploit flow**:
1. Maker posts a buy offer: `maxAssets = X`, tick < `MAX_TICK`, valid ratifier, valid expiry.
2. Attacker calls `take(offer, ..., units=1, ...)` in a loop.
3. Each iteration: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; `consumed` stays at its prior value; `require(newConsumed <= maxAssets)` passes; maker's credit increases by 1 unit; taker's debt increases by 1 unit; 0 tokens transferred.
4. After N iterations the maker holds N units of credit, far exceeding `maxAssets`, while `consumed[maker][group]` remains 0.

**Why existing checks fail**:
- The `require(newConsumed <= offer.maxAssets)` guard is the only enforcement of the cap; it relies entirely on `buyerAssets` being non-zero, which is not guaranteed.
- The Certora `Consume.spec` rule `takeConsumedDelta` (lines 67–75) is scoped to `maxAssets == 0` only — there is no analogous delta rule for assets mode.
- `Consume.spec` summarises both `mulDivDown` and `mulDivUp` as `NONDET` (lines 11–12), so the formal proofs never model the rounding-to-zero case.
- `takeConsumedBoundedByMax` (line 62) only asserts `consumed <= maxAssets` after a single take; it does not assert that consumed strictly increases when `units > 0`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation
The `maxAssets` cap — the sole mechanism preventing a buy offer from being filled beyond the maker's stated limit — is completely bypassed. An attacker can call `take` with `units = 1` an arbitrary number of times, each time incrementing the maker's credit by 1 unit at zero token cost, while `consumed` stays at 0. The offer is overfilled unboundedly, violating the core invariant that "offers cannot be replayed, overfilled, reused, or filled after cancel/deadline." The maker accumulates credit exposure far beyond their intended `maxAssets` limit with no corresponding token inflow per fill. [5](#0-4) 

### Likelihood Explanation
**Preconditions**: (1) A buy offer with `maxAssets > 0` exists — a normal maker configuration. (2) The tick is any value below `MAX_TICK` — satisfied by virtually every real offer since `MAX_TICK` is the par (price = 1) tick. (3) The attacker must be willing to take on 1 unit of debt per call with 0 tokens received — a griefing cost, but trivially automatable. No privileged access, no oracle manipulation, and no special token behaviour is required. The attack is repeatable in a single transaction via `multicall` or across multiple blocks. [6](#0-5) [7](#0-6) 

### Recommendation
Add a minimum-increment guard in the `maxAssets` branch: when `units > 0`, require that the computed asset amount is also non-zero before updating consumed.

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetIncrement()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` such that `units * price >= WAD` before accepting a fill in assets mode, or switch the consumed tracking for buy offers to use `mulDivUp` so that any non-zero `units` always produces at least 1 asset increment.

Also update `Consume.spec` to replace the `NONDET` summaries for `mulDivDown`/`mulDivUp` with axiom-constrained ghosts (as done in `LiquidationBoundedByLIF.spec`) and add a rule asserting `units > 0 => consumed(offer.maker, offer.group) > consumedBefore` for the assets-mode branch. [8](#0-7) [9](#0-8) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity 0.8.34;

import {BaseTest} from "./BaseTest.sol";
import {TickLib, MAX_TICK} from "../src/libraries/TickLib.sol";
import {IMidnight} from "../src/interfaces/IMidnight.sol";

contract ConsumedZeroIncrementTest is BaseTest {
    // Fuzz: any tick in [tickSpacing, MAX_TICK) gives buyerPrice < WAD.
    function testBuyOfferMaxAssetsConsumedNeverIncrements(uint256 tick) public {
        // Use any tick strictly below MAX_TICK so tickToPrice(tick) < WAD.
        tick = bound(tick, 0, (MAX_TICK / DEFAULT_TICK_SPACING - 1)) * DEFAULT_TICK_SPACING;
        uint256 price = TickLib.tickToPrice(tick);
        // Precondition: price < WAD so mulDivDown(1, price, WAD) == 0.
        assertLt(price, WAD, "price must be < WAD");

        // Maker posts a buy offer capped at maxAssets = 1000.
        lenderOffer.buy      = true;
        lenderOffer.tick     = tick;
        lenderOffer.maxAssets = 1000;
        lenderOffer.maxUnits  = 0;

        // Fund maker and give taker collateral for debt.
        deal(address(loanToken), lender, 1e30);
        collateralize(market, borrower, 1e6); // enough collateral for many units of debt

        uint256 consumedBefore = midnight.consumed(lender, lenderOffer.group);

        // Attacker (borrower/taker) calls take with units=1 three times.
        for (uint256 i = 0; i < 3; i++) {
            take(1, borrower, lenderOffer);
        }

        uint256 consumedAfter = midnight.consumed(lender, lenderOffer.group);

        // BUG: consumed never moved despite 3 fills.
        assertEq(consumedAfter, consumedBefore, "consumed did not increment — bug confirmed");

        // Maker has 3 units of credit beyond the maxAssets=1000 cap (cap is in assets, not units,
        // but consumed=0 means the cap check never fired at all).
        assertEq(midnight.creditOf(id, lender), 3, "maker credit grew unboundedly");

        // Invariant assertion that SHOULD hold but does not:
        // assertGt(consumedAfter, consumedBefore, "consumed must increase per fill");
    }
}
```

**Expected assertions**:
- `consumedAfter == consumedBefore` (demonstrates the bug — consumed stays at 0)
- `creditOf(lender) == 3` (maker received 3 units of credit beyond the cap)
- A correct implementation would require `consumedAfter >= consumedBefore + 1` per fill when `units > 0` [10](#0-9) [11](#0-10) [12](#0-11)

rule takeConsumedBoundedByMax(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {

/// After a successful take in units mode, the change in consumed equals the units taken.
rule takeConsumedDelta(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {

    assert consumed(offer.maker, offer.group) == consumedBefore + units;
