### Title
`mulDivUp` arithmetic overflow in `buyerAssetsToUnits` DoS-es bundler for buy offers with large `maxAssets` - (File: src/periphery/ConsumableUnitsLib.sol)

### Summary
`ConsumableUnitsLib.consumableUnits` delegates to `TakeAmountsLib.buyerAssetsToUnits` for buy offers, which calls `targetBuyerAssets.mulDivUp(WAD, buyerPrice)`. Because `UtilsLib.mulDivUp` uses Solidity 0.8.x checked arithmetic with no `unchecked` block, the intermediate product `targetBuyerAssets * WAD` reverts with an arithmetic panic whenever `targetBuyerAssets > type(uint256).max / WAD`. No existing check bounds `offer.maxAssets` below this threshold, so any buy offer with `maxAssets ≳ 1.157e59` permanently DoS-es every bundler entry-point that calls `consumableUnits`.

### Finding Description

**Exact code path:**

`ConsumableUnitsLib.consumableUnits` (line 18–19):
```
offer.buy == true, offer.maxUnits == 0
→ TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed))
``` [1](#0-0) 

With `consumed = 0`, `targetBuyerAssets = offer.maxAssets`.

`TakeAmountsLib.buyerAssetsToUnits` (line 29):
```
return targetBuyerAssets.mulDivUp(WAD, buyerPrice);   // offer.buy == true branch
``` [2](#0-1) 

`UtilsLib.mulDivUp` (line 34–36) — **no `unchecked` block**:
```solidity
return (x * y + (d - 1)) / d;   // x * y overflows if x > type(uint256).max / WAD
``` [3](#0-2) 

The existing guard `require(buyerPrice <= WAD)` only constrains the denominator; it does nothing to bound `targetBuyerAssets`. [4](#0-3) 

**Overflow threshold:** `type(uint256).max / WAD ≈ 1.157 × 10⁵⁹`. Any `offer.maxAssets` above this value (e.g. `type(uint256).max`, a common "unlimited" sentinel) triggers the panic.

**Attacker-controlled input:** The `takes[i].offer` struct is caller-supplied memory in every bundler entry-point. No on-chain validation bounds `offer.maxAssets`.

**Affected bundler entry-points** (both enforce `offer.buy == true` before calling `consumableUnits`):

- `supplyCollateralAndSellWithUnitsTarget` line 145 + 150
- `supplyCollateralAndSellWithAssetsTarget` line 283 + 290 [5](#0-4) [6](#0-5) 

The core `Midnight.take` is unaffected; it never calls `consumableUnits`. The formal overflow proof (`NoMultiplicationOverflow.spec`) covers only `Midnight.sol`, not the periphery. [7](#0-6) 

### Impact Explanation
Any buy offer whose maker sets `maxAssets > type(uint256).max / WAD` (including the idiomatic `type(uint256).max` "no-limit" value) causes every call to `supplyCollateralAndSellWithUnitsTarget` or `supplyCollateralAndSellWithAssetsTarget` that includes that offer to revert unconditionally. Takers cannot use the bundler to fill such offers; they must fall back to calling `Midnight.take` directly with a manually computed `units` value, losing all bundler convenience and composability guarantees.

### Likelihood Explanation
`type(uint256).max` is the idiomatic Solidity sentinel for "no cap". A maker who sets `offer.maxAssets = type(uint256).max` on a buy offer — a natural choice — silently breaks all bundler paths for that offer. The condition is trivially reproducible: set `offer.maxAssets = type(uint256).max`, `offer.maxUnits = 0`, `offer.buy = true`, `consumed = 0`. No privileged access, no oracle manipulation, and no special market state is required.

### Recommendation
Cap `targetBuyerAssets` before the multiplication in `buyerAssetsToUnits`, or saturate rather than overflow. The minimal fix is to add a guard at the top of the `offer.buy` branch:

```solidity
// In TakeAmountsLib.buyerAssetsToUnits, before the mulDivUp call:
if (targetBuyerAssets > type(uint256).max / WAD) return type(uint256).max;
return targetBuyerAssets.mulDivUp(WAD, buyerPrice);
```

Returning `type(uint256).max` is safe because `consumableUnits` feeds into a `min(...)` call in the bundler; an overestimate is harmless and the subsequent `take` will enforce the real cap.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {ConsumableUnitsLib} from "src/periphery/ConsumableUnitsLib.sol";
import {TakeAmountsLib} from "src/periphery/TakeAmountsLib.sol";
import {TickLib, MAX_TICK} from "src/libraries/TickLib.sol";
import {WAD} from "src/libraries/ConstantsLib.sol";
import {Offer} from "src/interfaces/IMidnight.sol";

contract OverflowPoC is Test {
    /// @dev Fuzz offer.maxAssets in [type(uint256).max / WAD, type(uint256).max].
    /// Invariant: consumableUnits must never revert for any valid offer.maxAssets.
    function testFuzz_consumableUnitsNoOverflow(uint256 maxAssets) public {
        maxAssets = bound(maxAssets, type(uint256).max / WAD, type(uint256).max);

        Offer memory offer;
        offer.buy      = true;
        offer.maxUnits = 0;
        offer.maxAssets = maxAssets;
        offer.tick     = MAX_TICK; // buyerPrice = WAD (maximum, so buyerPrice <= WAD check passes)

        // Expect: should NOT revert.
        // Actual: reverts with arithmetic overflow in mulDivUp(maxAssets, WAD, buyerPrice).
        uint256 buyerPrice = TickLib.tickToPrice(MAX_TICK); // = WAD = 1e18
        // Direct call to the inner function to isolate the overflow:
        vm.expectRevert(); // remove this line to assert no-revert invariant
        TakeAmountsLib.buyerAssetsToUnits(address(0), bytes32(0), offer, maxAssets);
    }
}
```

**Expected assertion:** `consumableUnits` (and `buyerAssetsToUnits`) must not revert for any `offer.maxAssets` value. The test demonstrates that for `maxAssets ≥ type(uint256).max / WAD` the call panics with `stdError.arithmeticError`, violating the invariant.
