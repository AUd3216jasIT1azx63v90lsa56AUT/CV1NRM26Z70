Audit Report

## Title
Zero-`buyerAssets` rounding bypasses `maxAssets` cap, allowing unlimited free position mutations - (File: `src/Midnight.sol`)

## Summary
In `take()`, when `offer.maxAssets > 0` and `units * buyerPrice < WAD`, `buyerAssets` truncates to zero via `mulDivDown`. The `consumed` mapping is incremented by zero, the `maxAssets` cap check trivially passes, yet position state (`credit`, `debt`, `totalUnits`) is mutated using `units` directly — with zero token transfer. This allows a taker to bypass a fully-consumed offer's cap an unlimited number of times at zero cost.

## Finding Description
In `src/Midnight.sol`, the `take()` function computes `buyerAssets` and `sellerAssets` via fixed-point division before using them to gate the `maxAssets` cap:

- **Line 363**: `buyerAssets = units.mulDivDown(buyerPrice, WAD)` truncates to `0` when `units * buyerPrice < WAD`.
- **Lines 367–369**: When `offer.maxAssets > 0`, `consumed[offer.maker][offer.group] += buyerAssets` adds `0`, and `require(newConsumed <= offer.maxAssets)` passes trivially because `newConsumed` is unchanged.
- **Lines 382–414**: Position mutations (`buyerPos.credit`, `sellerPos.debt`, `_marketState.totalUnits`) are computed from `units` directly, not from `buyerAssets`. With `units = 1`, these all increase by 1.
- **Line 455**: `safeTransferFrom(..., buyerAssets - sellerAssets)` transfers `0` tokens.

The root cause is a denomination mismatch: `consumed` is asset-denominated but position mutations are unit-denominated. No guard enforces `buyerAssets > 0` before proceeding.

**Price conditions**: `tickToPrice` returns values that are multiples of `PRICE_ROUNDING_STEP = 1e12`. For every tick except `MAX_TICK`, `price < WAD = 1e18`. With `units = 1`, `mulDivDown(1, price, 1e18) = 0` for any such tick. The test uses `tick = MAX_TICK - 16` where `price < WAD`.

**Existing checks that fail**:
- `require(newConsumed <= offer.maxAssets)` — passes because `+= 0`.
- `require(atMostOneNonZero(maxAssets, maxUnits))` — validates offer structure only, not fill size.
- No `require(units > 0)` or `require(buyerAssets > 0)` guard exists.

**Confirmed by `testBugBuyMaxAssetsBypass`** (`test/TakeTest.sol`, lines 857–889): the test pre-fills `consumed` to `maxAssets` via `setConsumed`, then calls `take(1, borrower, lenderOffer)` with `tick = MAX_TICK - 16`. It asserts `buyerAssets == 0`, `consumed` unchanged at `maxAssets`, token balances unchanged, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.

## Impact Explanation
A taker can fill a buy offer with `maxAssets > 0` an unlimited number of times at zero token cost. Each fill increases the maker's credit and the taker's debt by `units` without any token transfer and without advancing `consumed`. This violates the core accounting invariant that every credit increase must correspond to valid consumed accounting, and that offers cannot be overfilled or reused. The taker accrues unbounded debt obligations and the maker gains unbounded credit — both without any economic exchange — which can be used to manipulate market state, drain settlement fee accounting, or exploit downstream liquidation and settlement logic.

## Likelihood Explanation
Preconditions are minimal: (1) a buy offer with `maxAssets > 0` exists (standard offer type); (2) `tickToPrice(offer.tick) < WAD` — true for every valid tick except `MAX_TICK`; (3) taker uses `units = 1`. No privileged access, oracle manipulation, or front-running is required. The attack is executable in a single transaction via `multicall` and is trivially repeatable.

## Recommendation
Add a guard immediately after computing `buyerAssets` (and `sellerAssets`) that enforces a non-zero asset amount whenever `units > 0` and the offer uses asset-denominated cap:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || (offer.buy ? buyerAssets : sellerAssets) > 0, ZeroBuyerAssets());
}
```

This ensures that any non-trivial fill (i.e., `units > 0`) produces at least one asset unit, so `consumed` always advances and the `maxAssets` cap is meaningful. Alternatively, enforce a minimum `units` value such that `units >= WAD / buyerPrice` (i.e., at least one asset is produced), or switch to tracking `consumed` in units when `maxAssets` is used as a soft cap only.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) is a complete, minimal reproducer:

1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (price < WAD).
2. Pre-fill `consumed` to `maxAssets` via `midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender)`.
3. Call `take(1, borrower, lenderOffer)`.
4. Assert: `buyerAssets == 0`, `consumed` unchanged at `maxAssets`, token balances unchanged.
5. Assert: `creditOf(lender) > lenderCreditBefore`, `debtOf(borrower) > borrowerDebtBefore`, `totalUnits > totalUnitsBefore`.

All assertions pass, confirming the bypass. Repeating step 3 indefinitely inflates credit and debt without bound. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
