Audit Report

## Title
Fee-on-Transfer `loanToken` Inflates `claimableSettlementFee` Beyond Actual Contract Balance — (`src/libraries/SafeTransferLib.sol`, `src/Midnight.sol`)

## Summary
`SafeTransferLib.safeTransferFrom` verifies only that the call does not revert and that the return value is `true`; it never checks that the amount actually received equals the requested `value`. In `take()`, `claimableSettlementFee[loanToken]` is incremented unconditionally before the transfer executes, so a partial-transfer token (one that silently delivers `min(requested, cap)` and returns `true`) causes the recorded fee obligation to permanently exceed the contract's actual token balance. Because `touchMarket` imposes no restriction on `loanToken`, any unprivileged actor can register such a market and trigger the shortfall.

## Finding Description

**Root cause — `SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol`, lines 24–34):**

```solidity
function safeTransferFrom(address token, address from, address to, uint256 value) internal {
    require(token.code.length > 0, NoCode());
    (bool success, bytes memory returndata) =
        token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
    if (!success) { assembly { revert(...) } }
    require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
}
```

No balance-delta check. A token that executes `transfer(to, min(value, cap))` and returns `true` passes both guards (`NoCode` and `TransferFromReturnedFalse`).

**Accounting in `take()` (`src/Midnight.sol`, lines 418, 455–456):**

Line 418 credits the fee before the transfer:
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;   // line 418
```
Lines 455–456 then call `safeTransferFrom` without verifying the received amount:
```solidity
SafeTransferLib.safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets); // line 455
SafeTransferLib.safeTransferFrom(loanToken, payer, receiver,      sellerAssets);               // line 456
```
If the token delivers only `X < (buyerAssets - sellerAssets)`, the contract's balance increases by `X` while `claimableSettlementFee` increases by the full `buyerAssets - sellerAssets`. The shortfall `(buyerAssets - sellerAssets) - X` is permanently unrecoverable.

**Permissionless market creation — `touchMarket()` (`src/Midnight.sol`, lines 755–791):**

`touchMarket` validates collateral params, maturity, and LLTV but imposes no restriction on `loanToken`. Any address with deployed code is accepted, allowing an unprivileged actor to register a market whose `loanToken` is a partial-transfer contract they control.

**Exploit flow:**
1. Attacker deploys `PartialToken`: `transferFrom(from, to, amount)` transfers `min(amount, CAP)` and returns `true`.
2. Attacker calls `touchMarket(market)` with `market.loanToken = address(PartialToken)`.
3. Attacker (as maker or taker) calls `take()` with `units` chosen so that `buyerAssets - sellerAssets > CAP`.
4. Line 418 credits `claimableSettlementFee[PartialToken] += buyerAssets - sellerAssets`.
5. Line 455 calls `safeTransferFrom(..., buyerAssets - sellerAssets)` — token delivers only `CAP`; call succeeds.
6. Contract balance of `PartialToken` is `CAP`; `claimableSettlementFee[PartialToken]` is `buyerAssets - sellerAssets`. Invariant violated.
7. Repeating the attack compounds the shortfall indefinitely.

**Existing protections reviewed and found insufficient:**
- `NoCode` check: passes (attacker deploys a real contract).
- `TransferFromReturnedFalse` check: passes (token returns `true`).
- No balance-delta check anywhere in `SafeTransferLib` or `take()`.
- The Certora `Solvency.spec` (line 31–33) explicitly assumes "no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits" — the formal proof does not cover this case.
- The `pendingFeeReceiptZero` invariant in `Solvency.spec` (lines 157–158) and the `tokenBalanceCorrect` invariant (lines 162–163) are both proven only under the exact-transfer assumption; they do not hold for partial-transfer tokens.

## Impact Explanation
`claimableSettlementFee[token]` exceeds the contract's actual balance of that token. `claimSettlementFee` will revert when the fee claimer attempts to withdraw the full recorded amount, permanently locking fee revenue for that token. Repeated calls compound the shortfall without bound. This directly violates the solvency invariant: `tokenBalances[token][contract] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token)`. While collateral and withdrawable balances for other users in the same market are funded by separate transfers and are not directly drained, the protocol's accounting integrity for the affected token is permanently broken.

## Likelihood Explanation
- **Preconditions:** Attacker deploys a partial-transfer ERC20 and creates a market — both are permissionless, requiring no privileged access.
- **Feasibility:** Trivial — no oracle manipulation, no flash loan, no user mistake required.
- **Repeatability:** The attack can be repeated across multiple `take()` calls and multiple markets using the same token, compounding the shortfall indefinitely.
- **Constraint:** Impact is scoped to the specific `loanToken` used; markets with standard tokens are unaffected. However, the protocol makes no distinction and the invariant is still broken.

## Recommendation
Add a balance-before/after check in `take()` (or in `safeTransferFrom`) to verify the actual amount received equals the expected amount:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets);
require(IERC20(loanToken).balanceOf(address(this)) - balanceBefore == buyerAssets - sellerAssets, UnexpectedTransferAmount());
```

Alternatively, move the `claimableSettlementFee` increment to after the transfer and derive the credited amount from the actual balance delta rather than the requested amount. A third option is to whitelist allowed `loanToken` addresses at the protocol level, though this would reduce permissionlessness.

**Minimal Foundry test plan:**

```solidity
// 1. Deploy PartialToken: transferFrom transfers min(amount, CAP) and returns true.
// 2. Call touchMarket with loanToken = address(PartialToken).
// 3. Post a buy offer and call take() with units such that buyerAssets - sellerAssets > CAP.
// 4. Assert: PartialToken.balanceOf(address(midnight)) == CAP
// 5. Assert: midnight.claimableSettlementFee(address(PartialToken)) == buyerAssets - sellerAssets
// 6. Assert: claimableSettlementFee > actual balance  →  invariant violated
// 7. Call claimSettlementFee and observe revert (or partial claim leaving shortfall).
```

The test requires no privileged keys, no mainnet fork, and no external dependencies beyond a standard Foundry setup.
