### Title
Fee-on-Transfer Loan Token Causes Receiver Shortfall and Accounting Discrepancy in `withdraw` - (File: src/Midnight.sol)

### Summary
`Midnight.withdraw` deducts exactly `units` from `position[id][onBehalf].credit` and `marketState[id].withdrawable`, then calls `SafeTransferLib.safeTransfer(market.loanToken, receiver, units)`. `SafeTransferLib.safeTransfer` only verifies the call succeeded and returned `true`; it performs no balance-before/after check. When `market.loanToken` is a fee-on-transfer token with fee rate `f`, the receiver receives `units*(1-f)` while the protocol's accounting records a full `units` deduction, creating a permanent discrepancy between on-chain token balance and internal state.

### Finding Description
**Exact code path:**

`Midnight.withdraw` (`src/Midnight.sol:481-500`):
```
_position.credit      -= units;          // line 493
_marketState.withdrawable -= units;      // line 494
_marketState.totalUnits   -= units;      // line 495
SafeTransferLib.safeTransfer(market.loanToken, receiver, units); // line 499
```

`SafeTransferLib.safeTransfer` (`src/libraries/SafeTransferLib.sol:12-22`):
```
(bool success, bytes memory returndata) =
    token.call(abi.encodeCall(IERC20.transfer, (to, value)));
// checks: success == true, returndata == true or empty
// NO check: receiver balance increased by exactly value
```

**Root cause:** `safeTransfer` validates call success and ERC20 boolean return only. It does not assert `receiver.balanceOf(after) - receiver.balanceOf(before) == value`. A fee-on-transfer token deducts `units*f` from the transferred amount before crediting the receiver, returning `true` and succeeding — passing all existing checks.

**Attacker-controlled inputs:**
- `market.loanToken`: any address; `touchMarket` imposes no restriction on the loan token type (lines 755-791 validate only collateral params, LLTV, maxLif, and maturity).
- `units`: any value up to the lender's credit.
- `receiver`: any address.

**Exploit flow:**
1. Deploy or use an existing fee-on-transfer ERC20 with fee rate `f > 0`.
2. Create a market with this token as `loanToken` (permissionless, no token whitelist).
3. Lender acquires `units` credit via normal `take` flow.
4. Borrower repays `units` → Midnight receives `units*(1-f)` actual tokens but `withdrawable += units` (discrepancy already introduced at repay).
5. Lender calls `withdraw(market, units, lender, receiver)`.
6. `_position.credit -= units`, `_marketState.withdrawable -= units` execute.
7. `safeTransfer` calls `token.transfer(receiver, units)` → token charges fee, receiver gets `units*(1-f)`, fee goes to fee collector, call returns `true`.
8. No revert. Accounting shows `units` withdrawn; receiver holds `units*(1-f)`.

**Why existing checks fail:**
- `SafeTransferLib` only checks `success` and boolean return value, not actual received amount.
- No balance-before/after guard anywhere in `withdraw`.
- The Certora `Solvency.spec` (line 31-33) explicitly assumes "no fee taking from sender or receiver" — the formal proofs do not cover this case.
- `live_context.json` (line 233) lists fee-on-transfer as a class that "should be tested if not explicitly excluded" — no explicit exclusion exists in the core contract.

### Impact Explanation
The receiver receives `units*(1-f) < units` loan tokens while Midnight's accounting permanently records a deduction of `units` from both `position[id][onBehalf].credit` and `marketState[id].withdrawable`. The `units*f` tokens are irrecoverably lost to the fee mechanism. Because the discrepancy also originates at `repay` (Midnight receives `units*(1-f)` but credits `withdrawable += units`), the contract's actual token balance is `units*f` short per repay-withdraw cycle. This violates the core invariant that contract balances must cover withdrawable assets and credit redemptions, and can cascade to make other lenders unable to withdraw their full credit.

### Likelihood Explanation
- Market creation is permissionless and accepts any `loanToken` address with no type restrictions.
- Fee-on-transfer tokens exist on mainnet (e.g., tokens with deflationary mechanics or protocol fees).
- The exploit requires no special privilege: any lender in a market using such a token triggers it by calling `withdraw` normally.
- The condition is repeatable on every `withdraw` call in any affected market.

### Recommendation
Add a balance-before/after check in `withdraw` to assert the receiver received exactly `units`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(receiver);
SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
require(
    IERC20(market.loanToken).balanceOf(receiver) - balanceBefore == units,
    FeeOnTransferNotSupported()
);
```

Alternatively, apply the same guard to `repay`'s `safeTransferFrom` (where the discrepancy first enters) and document that fee-on-transfer tokens are explicitly unsupported.

    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {

        _burn(from, fee);
        return super.transferFrom(from, to, amount - fee);

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
