### Title
Lenders Cannot Reliably Withdraw Full Credit Balance Due to Continuous Fee Accrual - (File: src/Midnight.sol)

### Summary
The `withdraw` function in `Midnight.sol` accepts a caller-specified `units` amount and subtracts it directly from the lender's credit after calling `_updatePosition`. Because `_updatePosition` recomputes credit using `block.timestamp`, and credit decreases every second via continuous fee accrual, any lender who queries their full credit balance off-chain and submits a `withdraw` for that exact amount will have their transaction revert with an arithmetic underflow if the transaction is included even one second later than the block in which the amount was computed.

### Finding Description

**Root cause — time-sensitive credit value with no "withdraw-all" sentinel:**

`withdraw` (line 481) calls `_updatePosition` (line 485) before performing the subtraction:

```solidity
_updatePosition(market, id, onBehalf);          // line 485 — credit snapped to block.timestamp
...
_position.credit -= UtilsLib.toUint128(units);  // line 493 — reverts on underflow
```

`_updatePosition` delegates to `updatePositionView`, which computes the accrued continuous fee as:

```solidity
uint256 accrualEnd = UtilsLib.min(block.timestamp, market.maturity);   // line 811
uint128 fee = _lastAccrual < market.maturity
    ? uint128(postSlashPendingFee.mulDivDown(
          accrualEnd - _lastAccrual,
          market.maturity - _lastAccrual))                              // line 814-815
    : 0;
return (uint128(postSlashCredit) - fee, ...);                          // line 818
```

The returned `newCredit` is strictly smaller for every additional second elapsed since `lastAccrual`. After `_updatePosition` writes this smaller value back to storage (line 842), the subtraction on line 493 uses the freshly-written, time-reduced credit. If the caller supplied `units` equal to the credit they read at an earlier timestamp, `units > _position.credit` and the transaction reverts with an arithmetic underflow.

**Exploit path:**

1. Lender calls `updatePositionView` (or reads `position[id][user].credit`) at timestamp `T` and observes `credit = C`.
2. Lender submits `withdraw(market, C, onBehalf, receiver)`.
3. Transaction is mined at timestamp `T + Δ` (even one block later, Δ ≥ 1 s on mainnet).
4. `_updatePosition` computes `newCredit = C - fee(Δ)` where `fee(Δ) > 0`.
5. Line 493 executes `C - (C - fee(Δ))` → underflow → revert.

There is no `type(uint256).max` sentinel, no slippage tolerance parameter, and no "withdraw all" path in `withdraw`. The only way to avoid the revert is to withdraw strictly less than the full credit, permanently leaving dust in the protocol, or to use an atomic on-chain helper that reads and withdraws in the same transaction.

### Impact Explanation

Any lender attempting a full exit of their credit position from an EOA will have their transaction revert whenever it is not mined in the exact same second as the off-chain credit query. On Ethereum mainnet (12-second blocks) this is the common case. The lender is forced to either:
- Accept leaving residual credit (dust) in the protocol forever, or
- Build or use a smart-contract wrapper that atomically reads and withdraws in one call.

This degrades protocol usability for the standard EOA lender workflow and can cause repeated failed transactions with wasted gas.

### Likelihood Explanation

The continuous fee accrues every second (`block.timestamp` granularity). Any lender who wants to fully exit a position — a routine and expected action — will encounter this revert unless they happen to use an atomic on-chain helper. No privileged access or adversarial setup is required; the condition is triggered by normal user behavior.

### Recommendation

Add a `type(uint256).max` sentinel in `withdraw` that resolves to the current post-update credit:

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    bytes32 id = touchMarket(market);
    MarketState storage _marketState = marketState[id];
    _updatePosition(market, id, onBehalf);

    Position storage _position = position[id][onBehalf];
    if (units == type(uint256).max) units = _position.credit; // <-- add this
    ...
}
```

This mirrors the pattern used by ERC-4626 vaults and Morpho Blue's `withdraw` to allow atomic full-exit without requiring the caller to predict the exact post-accrual balance.
