Audit Report

## Title
Double `mulDivDown` Rounding Understates `maxDebt` by 1, Enabling Liquidation of Economically Healthy Positions - (File: src/Midnight.sol)

## Summary
`isHealthy` and `liquidate` both compute `maxDebt` via two sequential `mulDivDown` (floor division) calls. Each call independently discards a remainder, and the combined rounding error can understate `maxDebt` by exactly 1 relative to the true single-step floor. When a position sits at the exact health boundary, the protocol computes `maxDebt = debt − 1`, causing `isHealthy` to return `false` and the `liquidate` guard to pass, allowing seizure of collateral from an economically healthy borrower.

## Finding Description

**Confirmed code path:**

`mulDivDown` is plain integer floor division with no overflow protection or rounding correction: [1](#0-0) 

`isHealthy` applies it twice in sequence — first dividing by `ORACLE_PRICE_SCALE`, then by `WAD`: [2](#0-1) 

`liquidate` uses the identical double-floor pattern: [3](#0-2) 

The liquidatability guard compares the understated `maxDebt` against `originalDebt`: [4](#0-3) 

**Root cause — double floor rounding:**

Let `V1 = floor(C × P / ORACLE_PRICE_SCALE)` and `V2 = floor(V1 × L / WAD)`. The exact single-step bound is `M = floor(C × P × L / (ORACLE_PRICE_SCALE × WAD))`. Because step 1 discards remainder `r = (C × P) mod ORACLE_PRICE_SCALE` before step 2 multiplies by `L`, the combined error satisfies `M − 1 ≤ V2 ≤ M`. The off-by-one (`V2 = M − 1`) is reachable whenever the two discarded remainders together exceed one full unit.

**Concrete counterexample** using confirmed protocol constants (`ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`, valid LLTV tier `LLTV_2 = 0.77e18`): [5](#0-4) [6](#0-5) 

| Parameter | Value |
|---|---|
| `collateral` (C) | 10 |
| `price` (P) | `1.3e35` |
| `lltv` (L) | `0.77e18` |
| `debt` (D) | 1 |

- Step 1: `floor(10 × 1.3e35 / 1e36) = floor(1.3) = 1`
- Step 2: `floor(1 × 0.77e18 / 1e18) = floor(0.77) = 0`
- Exact: `10 × 1.3e35 × 0.77e18 / 1e54 = 1.001` → `floor = 1 = debt` → **healthy**
- Protocol sees: `maxDebt = 0 < 1 = debt` → **unhealthy** → liquidation guard passes

**Exploit flow:**

1. Borrower opens a position with C = 10 collateral units and borrows D = 1 debt unit at price P = 2e35. At that price: `V1 = floor(10 × 2e35 / 1e36) = 2`, `V2 = floor(2 × 0.77e18 / 1e18) = 1 ≥ 1 = debt` → `isHealthy` returns `true` at the `take` call site (line 476), so the borrow succeeds.
2. Oracle price drifts to exactly `1.3e35` through normal market movement. The position remains economically healthy: exact `maxDebt = floor(1.001) = 1 = debt`.
3. Liquidator calls `liquidate(market, collateralIndex, seizedAssets, 0, borrower, false, ...)`.
4. Protocol computes `maxDebt = 0`; guard `originalDebt (1) > maxDebt (0)` passes; collateral is seized from a healthy borrower.

**Why existing checks fail:**

There are no minimum debt or collateral dust guards anywhere in the codebase that would prevent positions with small integer values from being created. [7](#0-6) 

The health check at the `take` path uses the same `isHealthy` function with the same double-floor computation, so it does not prevent the boundary condition from being reached after price movement. [8](#0-7) 

The contract's own natspec acknowledges `maxDebt` is rounded down but does not account for the compounding of two independent floor operations: [9](#0-8) 

## Impact Explanation
A liquidator can seize collateral from a borrower whose position is economically healthy (exact `maxDebt ≥ debt`) purely due to the double floor-rounding understating `maxDebt` by 1. The borrower suffers an unwarranted collateral loss. The core protocol invariant — healthy positions cannot be liquidated — is violated. This is a direct, concrete loss of user funds triggered by an unprivileged external caller.

## Likelihood Explanation
No oracle manipulation is required. For any fixed collateral amount `C` and LLTV `L`, there exists a positive-measure interval of oracle prices `P` that triggers the off-by-one. A liquidator needs only to monitor positions and call `liquidate` when the price enters such an interval. With many positions across many markets, the probability of at least one being in the vulnerable range at any given time is non-negligible. The condition is passively reachable through normal price movement.

## Recommendation
Replace the two sequential `mulDivDown` calls with a single-step computation that avoids the intermediate floor. Use a 512-bit full-precision multiply-then-divide (e.g., Uniswap's `FullMath.mulDiv`) so that `maxDebt` is computed as `floor(C × P × L / (ORACLE_PRICE_SCALE × WAD))` in one operation:

```solidity
// Instead of:
maxDebt += collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);

// Use (with overflow-safe 512-bit arithmetic):
maxDebt += FullMath.mulDiv(collateral * price, lltv, ORACLE_PRICE_SCALE * WAD);
```

Note that `ORACLE_PRICE_SCALE * WAD = 1e54` fits in a uint256 (max ~1.16e77), so the denominator itself does not overflow. The numerator `collateral * price * lltv` may require 512-bit intermediate arithmetic. Alternatively, restructure as `mulDivDown(collateral, price * lltv / WAD, ORACLE_PRICE_SCALE)` only if `price * lltv / WAD` can be computed without precision loss, which requires careful analysis.

## Proof of Concept
Minimal Foundry unit test:

```solidity
function test_doubleFloorOffByOne() public pure {
    uint256 C = 10;
    uint256 P = 1.3e35;
    uint256 L = 0.77e18;
    uint256 S = 1e36;
    uint256 W = 1e18;
    uint256 D = 1;

    // Two-step (protocol behavior)
    uint256 v1 = (C * P) / S;          // = 1
    uint256 v2 = (v1 * L) / W;         // = 0
    assert(v2 < D);                     // protocol: unhealthy

    // Single-step (true math)
    // C*P*L = 10 * 1.3e35 * 0.77e18 = 1.001e54
    // floor(1.001e54 / 1e54) = 1
    // Use 512-bit or check: (C*P*L) / (S*W) = 1
    // Simplified check: C*P*L >= D * S * W
    assert(C * P / S * L / W < D);     // double-floor: 0 < 1 → liquidatable
    // True: 10 * 1.3e35 * 0.77e18 = 1.001e54 >= 1 * 1e54 → healthy
}
```
