### Title
`mulDivUp` Rounding in `sellerPendingFeeDecrease` Drains Seller's `pendingFee` Faster Than Proportional via Small-Unit Takes - (`src/Midnight.sol`)

### Summary

In `take()`, the seller's `pendingFee` decrease is computed with `mulDivUp`, which rounds up each individual take. When a seller (or a sybil they control) takes in increments of 1 unit, each call rounds up by up to 1 wei, causing cumulative over-drain of `pendingFee` relative to the credit decrease. A seller with `credit=C` and `pendingFee=1` can drain their entire `pendingFee` by selling just 1 unit, leaving `C-1` units of credit with zero `pendingFee` and no further fee obligation.

### Finding Description

**Exact code path** — `src/Midnight.sol` lines 387–389 and 412–413:

```solidity
uint128 sellerPendingFeeDecrease = sellerPos.credit > 0
    ? UtilsLib.toUint128(sellerPos.pendingFee.mulDivUp(sellerCreditDecrease, sellerPos.credit))
    : 0;
// ...
sellerPos.pendingFee -= sellerPendingFeeDecrease;
sellerPos.credit    -= UtilsLib.toUint128(sellerCreditDecrease);
``` [1](#0-0) [2](#0-1) 

**Root cause** — `mulDivUp(pendingFee, 1, credit)` = `ceil(pendingFee / credit)`. When `pendingFee < credit`, this rounds up to 1 even though the true proportional share is `< 1`. Each small take over-drains `pendingFee` by up to 1 unit. Over N takes of 1 unit each, the cumulative over-drain can equal the entire initial `pendingFee`.

**Attacker-controlled inputs** — `units=1` per call; the seller signs a sell offer via `EcrecoverRatifier` (offer.buy=false, offer.maker=seller, offer.maxUnits=N or multiple groups), then a sybil address (different from `offer.maker` to pass the `SelfTake` check) calls `take()` with `units=1` repeatedly. [3](#0-2) [4](#0-3) 

**Exploit flow (concrete example)**:
- State: `credit=100`, `pendingFee=1` (naturally reached near maturity as fees accrue)
- Take 1 (units=1): `sellerCreditDecrease=1`, `sellerPendingFeeDecrease = ceil(1·1/100) = 1`
- After: `credit=99`, `pendingFee=0`
- Seller retains 99 units of credit with zero `pendingFee`; no further continuous fee accrues on those 99 units

**Why existing checks fail**:
- The `SelfTake` check (`offer.maker != taker`) is bypassed by using a sybil address.
- The `isHealthy` check at line 476 only evaluates collateral vs. debt; a pure lender (no debt) always passes.
- The Certora rule `pendingFeeDecreasesProportionallyForSeller` verifies only the single-take formula `ceil(P·Δcredit/credit)` — it does not verify the multi-take invariant that `pendingFee/credit` is monotonically non-increasing across sequential takes. [5](#0-4) [6](#0-5) 

### Impact Explanation

The seller avoids paying the continuous fee on their remaining credit. The protocol's `continuousFeeCredit` never receives the fee that should have accrued on the retained credit, because `pendingFee` was zeroed out via rounding rather than via actual accrual. The maximum loss per attack is bounded by the seller's `pendingFee` at the time of the attack (which equals the remaining fee obligation). This directly violates the core invariant that `pendingFee` must decrease proportionally to credit decrease across multiple operations.

### Likelihood Explanation

The precondition `pendingFee ≪ credit` arises naturally near maturity as continuous fees accrue and `pendingFee` approaches zero while `credit` remains large. No special market conditions are required. The attack requires only: (1) the seller to sign a sell offer (standard operation), and (2) a sybil address to call `take()` with `units=1`. Both are permissionless. The attack is repeatable across any market with a non-zero continuous fee rate.

### Recommendation

Replace `mulDivUp` with `mulDivDown` for `sellerPendingFeeDecrease`:

```solidity
uint128 sellerPendingFeeDecrease = sellerPos.credit > 0
    ? UtilsLib.toUint128(sellerPos.pendingFee.mulDivDown(sellerCreditDecrease, sellerPos.credit))
    : 0;
```

This ensures the seller never loses more `pendingFee` than proportional to their credit decrease. The rounding now favors the seller by at most 1 unit per take (the seller retains a negligible excess `pendingFee`), which is the correct conservative direction — the protocol should not lose fee revenue due to rounding. Alternatively, track `pendingFee` as a fixed-point ratio relative to credit to eliminate rounding accumulation entirely.

### Proof of Concept

```solidity
// Foundry invariant/fuzz test
function testPendingFeeRatioDrainedBySmallTakes(
    uint256 credit,   // e.g., 1000
    uint256 feeRate,
    uint256 ttm,
    uint256 elapsed,
    uint256 numTakes  // e.g., 10
) public {
    credit   = bound(credit,   100, MAX_CREDIT);
    feeRate  = bound(feeRate,  1,   MAX_CONTINUOUS_FEE);
    ttm      = bound(ttm,      10,  360 days);
    elapsed  = bound(elapsed,  ttm * 9 / 10, ttm - 1); // near maturity
    numTakes = bound(numTakes, 2, 50);

    setupLender(credit, feeRate, ttm);
    vm.warp(vm.getBlockTimestamp() + elapsed);
    midnight.updatePosition(market, lender);

    uint256 creditBefore  = midnight.creditOf(id, lender);
    uint256 pendingBefore = midnight.pendingFee(id, lender);
    vm.assume(pendingBefore > 0 && pendingBefore < creditBefore);

    // Seller signs a sell offer; sybil takes 1 unit at a time
    for (uint256 i = 0; i < numTakes && midnight.creditOf(id, lender) > 0; i++) {
        deal(address(loanToken), sybil, 1);
        takeSingleUnit(lender, sybil); // take(units=1, maker=lender, taker=sybil)
    }

    uint256 creditAfter  = midnight.creditOf(id, lender);
    uint256 pendingAfter = midnight.pendingFee(id, lender);

    // INVARIANT: pendingFee/credit ratio must be non-increasing
    // i.e., pendingAfter * creditBefore <= pendingBefore * creditAfter
    // (cross-multiply to avoid division)
    assertLe(
        pendingAfter * creditBefore,
        pendingBefore * creditAfter,
        "pendingFee/credit ratio increased: over-drain via mulDivUp"
    );
}
```

**Expected assertion failure**: With `credit=100`, `pendingFee=1`, `numTakes=1`, after one take of 1 unit: `pendingAfter=0`, `creditAfter=99`. The invariant check becomes `0 * 100 <= 1 * 99` → `0 <= 99` (passes), but the ratio went from `1/100` to `0/99 = 0` — a strict decrease of the entire fee obligation from a 1% credit reduction. A stateful fuzz test asserting `pendingAfter * creditBefore <= pendingBefore * creditAfter` will catch cases where the ratio strictly drops to zero prematurely.

// When a seller's credit decreases via a take, their pendingFee decreases by ceil(PendingFee * creditDelta / postUpdateCredit).
rule pendingFeeDecreasesProportionallyForSeller(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    address seller = offer.buy ? taker : offer.maker;

    bytes32 id;
    uint128 postUpdateCredit;
    uint128 postUpdatePendingFee;

    postUpdateCredit, postUpdatePendingFee, _ = updatePositionView(e, offer.market, id, seller);

    require postUpdateCredit > 0 || postUpdatePendingFee == 0, "See noRemainingContinuousFeeWithoutCredit in Midnight.spec";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    require id == lastId, "id should be derived from market";

    uint256 creditAfter = creditOf(id, seller);
    uint256 pendingFeeAfter = pendingFee(id, seller);

    require creditAfter > 0 || pendingFeeAfter == 0, "See noRemainingContinuousFeeWithoutCredit in Midnight.spec";

    mathint creditDelta = postUpdateCredit - creditAfter;

    // When postUpdateCredit == 0: noRemainingContinuousFeeWithoutCredit gives postUpdatePendingFee == 0; credit is non-increasing for a seller, therefore creditAfter == 0;
    // noRemainingContinuousFeeWithoutCredit gives pendingFeeAfter == 0; hence pendingFeeDelta == 0.
    assert postUpdateCredit == 0 ? postUpdatePendingFee == pendingFeeAfter : postUpdatePendingFee == pendingFeeAfter + (postUpdatePendingFee * creditDelta + postUpdateCredit - 1) / postUpdateCredit;
}
