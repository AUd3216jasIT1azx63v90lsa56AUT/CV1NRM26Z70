Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via rounding-to-zero `buyerAssets` on small-unit takes - (File: `src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and `buyerPrice < WAD` (tick above `MAX_TICK/2`), `buyerAssets` is computed with `mulDivDown`, which rounds to zero for sufficiently small `units`. Because `consumed` is incremented by `buyerAssets`, a zero-asset take leaves `consumed` unchanged, passes the `ConsumedAssets` cap check, yet still creates real credit and debt and increases `totalUnits`. This can be repeated without bound, even after the offer is fully consumed in asset terms, violating the maker's intended exposure cap.

## Finding Description
**Exact code path — `src/Midnight.sol`:** [1](#0-0) 

For a buy offer, `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. When `buyerPrice < WAD` (i.e., `tickToPrice(tick) < 1e18`) and `units * buyerPrice < WAD`, the result is `0`. The cap check on line 369 then trivially passes because `consumed` was not incremented.

**Root cause:** The `consumed` accumulator for asset-capped buy offers uses the rounded-down asset value rather than a unit-based counter. When rounding produces zero, the accumulator is frozen while real position state (credit, debt, `totalUnits`) continues to change.

**Position state still mutates despite zero assets:** [2](#0-1) 

`buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` all increase by `units` regardless of whether `buyerAssets` is zero.

**Protocol's own acknowledgment — `src/Midnight.sol` line 94:** [3](#0-2) 

**Exploit flow:**
1. Maker creates a buy offer: `maxAssets = N`, `tick > MAX_TICK/2` (so `buyerPrice < WAD`), `maxUnits = 0`.
2. Taker calls `take(1, ...)` repeatedly (or in a loop via callback).
3. Each call: `buyerAssets = 1 * buyerPrice / WAD = 0`. `consumed` stays at its current value. Cap check passes. Lender credit and borrower debt each increase by 1 unit.
4. After `N` such calls, `consumed == maxAssets` but the offer remains permanently takeable; every subsequent call still produces `buyerAssets = 0` and passes the cap check.

**Why existing checks fail:** The sole guard is `require(newConsumed <= offer.maxAssets)`. Since `newConsumed` never increases when `buyerAssets = 0`, this check is permanently satisfied regardless of how many unit-level fills have occurred.

## Impact Explanation
The maker's `maxAssets` cap is intended to bound total economic exposure in asset terms. Due to rounding, an attacker can fill an unbounded number of units beyond what `maxAssets / buyerPrice` would imply. The lender (maker) accumulates credit redeemable at maturity for real assets, and the taker (borrower) accumulates debt that must be repaid — both without any asset transfer occurring at fill time. This violates the core protocol invariant that offers cannot be overfilled or replayed beyond their stated cap.

## Likelihood Explanation
Preconditions are minimal and common: any buy offer at a tick above `MAX_TICK/2` with `maxAssets > 0` is vulnerable. No privileged access is required — any address can call `take()` as the taker. The attack is repeatable within a single transaction via callback or across multiple transactions. Ticks above `MAX_TICK/2` correspond to below-par lending rates, which are a normal and expected use case.

## Recommendation
Add a guard that prevents a take from proceeding when `offer.maxAssets > 0`, `offer.buy == true`, and `buyerAssets == 0`. For example, require `buyerAssets > 0` before the cap check, or enforce a minimum `units` such that `units * buyerPrice >= WAD`. Alternatively, use `mulDivUp` for the `consumed` increment on buy offers so that even the smallest take increments the accumulator by at least 1, ensuring the cap is eventually reached.

## Proof of Concept
The repository contains a direct reproduction at `test/TakeTest.sol`: [4](#0-3) 

`testBugBuyMaxAssetsBypass` sets `maxAssets = 1`, `tick = MAX_TICK - 16` (so `offerPrice < WAD`), pre-consumes the offer to `maxAssets` via `setConsumed`, then calls `take(1, ...)`. The test asserts `buyerAssets == 0`, `consumed` remains at `maxAssets` (cap not incremented), yet `creditOf`, `debtOf`, and `totalUnits` all strictly increase — confirming the bypass.
