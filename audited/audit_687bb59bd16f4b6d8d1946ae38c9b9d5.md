Audit Report

## Title
Re-entrant `take()` via `onBuy()` Callback Bypasses State-Dependent `enterGate.canIncreaseCredit` Check - (File: src/Midnight.sol)

## Summary
`Midnight.take()` writes the buyer's credit increase to persistent storage at line 410 before invoking the `onBuy()` buyer callback at lines 445–453, with no reentrancy guard on `take()` itself. A malicious buyer callback can re-enter `take()` for the same buyer in the same gated market; the inner call's `canIncreaseCredit` gate check executes against the already-mutated intermediate credit state, allowing a gate whose logic reads Midnight's own `creditOf` to be bypassed and the buyer to accumulate credit beyond the gate's intended cap.

## Finding Description

**Exact code path in `src/Midnight.sol`:**

1. Gate check (lines 397–401): `canIncreaseCredit(buyer)` is called against pre-update state.
2. State update (line 410): `buyerPos.credit += buyerCreditIncrease` is committed to storage.
3. Buyer callback (lines 445–453): `IBuyCallback(buyerCallback).onBuy(...)` is called with credit already written.

```solidity
// lines 397-401 — gate check BEFORE state update
require(
    offer.market.enterGate == address(0) || buyerCreditIncrease == 0
        || IEnterGate(offer.market.enterGate).canIncreaseCredit(buyer),
    BuyerGatedFromIncreasingCredit()
);

buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease); // line 410

// lines 445-453 — callback AFTER state update, NO reentrancy guard
if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback)
            .onBuy(id, offer.market, buyerAssets, units, buyerPendingFeeIncrease, buyer, buyerCallbackData)
        == CALLBACK_SUCCESS, WrongBuyCallbackReturnValue()
    );
}
```

The only transient lock in `take()` is `LIQUIDATION_LOCK_SLOT` (line 444), which is keyed by `(id, seller)` and only prevents the seller from being liquidated during callbacks. It does not prevent re-entry into `take()` itself.

**Root cause:** No reentrancy guard (transient-storage lock, mutex, or `nonReentrant` modifier) protects `take()` against re-entry. The `tExchange`/`tGet` helpers in `UtilsLib.sol` exist and are used for the liquidation lock, but are not applied to `take()` entry.

**Exploit flow (concrete):**

Precondition: gate implements `canIncreaseCredit(buyer)` as `IMidnight(midnight).creditOf(id, buyer) < X` (reads Midnight state; a natural credit-cap design).

1. Outer `take()`: buyer credit = 0. Gate sees 0 < X → returns `true`. Gate check passes.
2. Outer `take()` writes `buyerPos.credit = Y` (where Y < X) to storage.
3. Outer `take()` calls `onBuy()` on malicious callback.
4. Malicious callback re-enters `take()` for same buyer, different offer, same market.
5. Inner `take()`: gate calls `canIncreaseCredit(buyer)` → reads `creditOf = Y`. Since Y < X, returns `true`. Gate check passes.
6. Inner `take()` writes `buyerPos.credit = Y + Z` to storage.
7. Both calls complete. Final credit = Y + Z, potentially > X. Gate's cap is violated.

**Why existing checks fail:**

- `Reverts.spec` lines 30–34 explicitly model all callbacks as pure CVL functions with no state changes, and justify this by stating "gate checks precede callbacks so re-entrant state changes cannot affect them." This is only true for the *outer* call's gate check. The *inner* call's gate check executes after the outer call's state mutation.
- The `enterGateBlocksCreditIncrease` rule (lines 295–307) uses `ghostCanIncreaseCredit(offer.market.enterGate)` — a fixed ghost value per gate address — and does not model a gate whose return value depends on Midnight's own storage. The rule therefore cannot detect this scenario.
- No `nonReentrant` modifier or transient-storage lock guards `take()` against re-entry.

## Impact Explanation
A buyer whose callback is a malicious contract can accumulate credit in a gated market beyond the gate's intended per-user cap within a single transaction. Any gate whose `canIncreaseCredit` logic reads Midnight's own `creditOf` (or any state that `take()` mutates before the callback) is vulnerable. This directly violates the protocol invariant that gate checks must apply to all entry paths and that callbacks must not corrupt partial state through reentrancy. The impact is unauthorized credit accumulation, which translates to unbacked lending exposure in the affected market.

## Likelihood Explanation
All preconditions are attacker-controlled: (1) the attacker is the taker (or an authorized account) and sets `takerCallback` to a malicious contract — no privilege required; (2) for a buy offer, the attacker can also be the maker and set `offer.callback`; (3) a second valid offer in the same market can be created by the attacker as maker. The attack requires no special timing, oracle manipulation, or victim mistakes. It is repeatable within a single transaction and applies to any market whose gate reads Midnight state.

## Recommendation
Add a transient-storage reentrancy guard to `take()` using the existing `UtilsLib.tExchange`/`tGet` infrastructure. A dedicated `TAKE_LOCK_SLOT` (analogous to `LIQUIDATION_LOCK_SLOT`) should be set to `true` at the start of `take()` and cleared at the end, with a `require(!tGet(TAKE_LOCK_SLOT, id, buyer))` check at entry. Alternatively, move the `canIncreaseCredit` gate check to after all state mutations and before the callback, so the check always sees the final state — though this changes gate semantics. The cleanest fix is a per-call reentrancy lock on `take()`.

## Proof of Concept
Deploy a malicious `IBuyCallback` contract that, in `onBuy()`, calls `Midnight.take()` again with a second pre-signed buy offer for the same buyer and market. Configure a gate contract that implements `canIncreaseCredit(buyer)` as `IMidnight(midnight).creditOf(id, buyer) < CAP`. Run the outer `take()` with `units` such that the resulting credit Y < CAP. Observe that after both calls complete, `creditOf(id, buyer) = Y + Z > CAP`, demonstrating the gate cap is bypassed. A Foundry fork test can confirm this by asserting `creditOf(id, buyer) <= CAP` after the transaction and observing the assertion fail. rule enterGateBlocksCreditIncrease(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData, address user) {
    require !ghostCanIncreaseCredit(offer.market.enterGate), "canIncreaseCredit blocked";
    require offer.market.enterGate != 0, "enter gate is set";

    bytes32 id = summaryToId(offer.market);
    uint256 creditBefore = creditOf(id, user);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    uint256 creditAfter = creditOf(id, user);

    assert creditAfter <= creditBefore;
}
