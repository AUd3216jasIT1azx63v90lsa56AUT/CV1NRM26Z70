Audit Report

## Title
Two-step `mulDivDown` rounding in health check allows liquidation of truly-healthy positions - (File: src/Midnight.sol)

## Summary
Both `isHealthy` and `liquidate` compute maximum debt capacity via two sequential floor divisions. This two-step floor can produce a result exactly 1 unit less than the mathematically correct single-step `floor(collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD))`. When the oracle price lands in the band where this discrepancy occurs and the borrower's debt equals the single-step result, the position is genuinely solvent but the protocol treats it as liquidatable, allowing an unprivileged liquidator to seize collateral amplified by the LIF.

## Finding Description

**Exact code path:**

`mulDivDown` at `src/libraries/UtilsLib.sol` lines 29–31 is plain integer floor division: [1](#0-0) 

`isHealthy` at `src/Midnight.sol` lines 954–955 applies two sequential floors: [2](#0-1) 

`liquidate` at `src/Midnight.sol` line 613 applies the identical two-step pattern: [3](#0-2) 

The liquidation gate at lines 620–624 fires when `originalDebt > maxDebt`: [4](#0-3) 

**Root cause — two-step floor loses up to 1 unit:**

Let `q = floor(C*P / S)`, `r = (C*P) mod S`, with `S = ORACLE_PRICE_SCALE = 1e36`, `W = WAD = 1e18`, `L = lltv`. [5](#0-4) 

- Two-step result: `floor(q * L / W)`
- Single-step result: `floor(C*P*L / (S*W)) = floor(q*L/W + r*L/(S*W))`

The difference equals exactly 1 when `{q*L/W} + r*L/(S*W) >= 1`. Since `r < S` and `L <= W`, the term `r*L/(S*W) < 1`, so the maximum discrepancy is **exactly 1**.

**Concrete witness (`LLTV_7 = 0.98e18`):** [6](#0-5) 

Choose `C = 1`, `L = 0.98e18`, `P = 51e36 + 2.1e34`:
- `q = floor((51e36 + 2.1e34) / 1e36) = 51`
- Two-step: `floor(51 * 0.98) = floor(49.98) = 49`
- Single-step: `floor((51 + 0.021) * 0.98) = floor(50.00058) = 50`

If `debt = 50`: two-step gives `maxDebt = 49`, so `50 > 49` → liquidation proceeds. Single-step gives `maxDebt = 50 >= 50` → position is truly healthy.

**Exploit flow:**

1. Borrower supplies `C = 1` unit of collateral.
2. At prior oracle price `P' = 52e36`, both formulas give `maxDebt = floor(52 * 0.98) = 50`. Borrower borrows 50 units. The `take` health check at line 476 passes.
3. Oracle moves passively to `P = 51e36 + 2.1e34` (no manipulation required).
4. Liquidator calls `liquidate()`. Two-step gives `maxDebt = 49`. The `NotLiquidatable` guard evaluates `50 > 49 = true` and does not revert.
5. Liquidator repays debt and seizes collateral amplified by `lif > WAD`.

**Why existing checks fail:**

- `liquidationLocked` is transient storage set only during `take` callbacks (line 444: `UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`). A standalone `liquidate` call has it as `false`. [7](#0-6) 
- The `NotLiquidatable` guard uses the same two-step `maxDebt`, so it confirms the (incorrect) liquidatability.
- The protocol comment at line 116 ("maxDebt is rounded down in isHealthy and liquidate") documents the rounding direction but does not acknowledge or justify the additional 1-unit error introduced by two-step composition versus a single-step floor. [8](#0-7) 

`LLTV_8 = WAD = 1e18` is immune because `mulDivDown(x, WAD, WAD) = x` introduces no rounding in the second step. All other tiers (`LLTV_0` through `LLTV_7`) are affected. [9](#0-8) 

## Impact Explanation

A liquidator can call `liquidate()` on a position that is genuinely solvent at the current oracle price. The borrower loses collateral at a rate amplified by `lif` (which exceeds `WAD` for all non-`LLTV_8` tiers), receiving less than fair value in return. This directly violates the core protocol invariant that healthy positions are not liquidatable, constituting unauthorized movement of user assets.

## Likelihood Explanation

The vulnerable price band has width approximately `S * W / (L * C)` price units. For `C = 1` and `LLTV_7`, this is roughly `1e36 * 1e18 / (0.98e18) ≈ 1.02e36` — wide relative to oracle granularity. For larger collateral amounts the band narrows proportionally, but the condition remains passively observable without any oracle manipulation. The vulnerability applies to all eight non-`LLTV_8` tiers. Any liquidator monitoring oracle prices can trigger this the moment the price enters the band, and the condition is repeatable across any market using a non-WAD LLTV.

## Recommendation

Replace the two sequential `mulDivDown` calls with a single combined floor division:

```solidity
// Instead of:
maxDebt += collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);

// Use:
maxDebt += collateral.mulDivDown(price * lltv, ORACLE_PRICE_SCALE * WAD);
```

This computes `floor(collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD))` in one step, eliminating the intermediate floor that causes the 1-unit discrepancy. Apply the same fix in both `isHealthy` and `liquidate`. Verify that `price * lltv` cannot overflow `uint256` given the oracle price scale and LLTV bounds.

## Proof of Concept

**Minimal manual steps:**

1. Deploy a market with `LLTV_7 = 0.98e18` and a mock oracle.
2. Set oracle price to `P' = 52e36`. Borrower supplies `C = 1` collateral unit and borrows `50` debt units. Confirm `isHealthy` returns `true`.
3. Set oracle price to `P = 51e36 + 2.1e34`.
4. Confirm `isHealthy` returns `false` (two-step gives `maxDebt = 49 < 50`).
5. Confirm that the single-step formula `floor(1 * (51e36 + 2.1e34) * 0.98e18 / (1e36 * 1e18)) = 50 >= 50` would return healthy.
6. Call `liquidate()` as an unprivileged liquidator. Confirm it succeeds and the borrower's collateral is seized.

**Fuzz test plan:** Fuzz over `(collateral, price, lltv)` and assert that whenever `isHealthy` returns `false`, the single-step formula `floor(collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD))` also gives a result strictly less than `debt`. Any counterexample is a witness to the bug.
