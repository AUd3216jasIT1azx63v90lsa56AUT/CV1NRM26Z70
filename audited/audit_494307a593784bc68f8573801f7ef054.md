### Title
`TakeAmountsLib.sellerAssetsToUnits` panics on `sellerPrice == 0` outside try/catch, reverting entire bundle for a case `Midnight.take` accepts - (`src/periphery/ConsumableUnitsLib.sol` / `src/periphery/TakeAmountsLib.sol`)

### Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and then calls `mulDivUp(targetSellerAssets, WAD, sellerPrice)`. When `offerPrice == settlementFee` exactly, `sellerPrice = 0`, and `mulDivUp`'s implementation `(x * y + (d - 1)) / d` underflows at `(0 - 1)` before any division, producing an arithmetic panic. `Midnight.take` does not revert for this same condition — it computes `sellerAssets = units.mulDivDown(0, WAD) = 0` (dividing by `WAD`, not by `sellerPrice`) and succeeds. Because the `sellerAssetsToUnits` call in `supplyCollateralAndSellWithAssetsTarget` sits outside the `try/catch`, the panic propagates and reverts the entire bundle.

### Finding Description

**Exact code path:**

`supplyCollateralAndSellWithAssetsTarget` (MidnightBundles.sol lines 285–291) calls `TakeAmountsLib.sellerAssetsToUnits` unconditionally for every offer in the loop, before the `try IMidnight(MIDNIGHT).take(...)` block:

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(          // ← outside try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) { ... } catch {}  // ← only take() is guarded
```

Inside `sellerAssetsToUnits` (TakeAmountsLib.sol lines 41–46):

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
uint256 settlementFee = IMidnight(midnight).settlementFee(...);
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

When `offer.buy == true` and `offerPrice == settlementFee`: `sellerPrice = 0`. `mulDivUp` (UtilsLib.sol line 35) is `(x * y + (d - 1)) / d`; with `d = 0`, the sub-expression `(d - 1)` = `(0 - 1)` triggers a Solidity 0.8 arithmetic underflow panic (panic code `0x11`) before any division occurs.

**Why `Midnight.take` does not revert for the same input:**

In `Midnight.take` (Midnight.sol lines 361–364):
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;  // = 0
uint256 buyerPrice  = sellerPrice + _settlementFee;                           // = settlementFee
uint256 buyerAssets = units.mulDivDown(buyerPrice, WAD);  // divides by WAD, not sellerPrice
uint256 sellerAssets = units.mulDivDown(sellerPrice, WAD); // = 0, divides by WAD
```
`mulDivDown` is `(x * y) / d` with `d = WAD` — no panic. The take proceeds and returns `sellerAssets = 0`.

**Why `consumableUnits` does not panic for this offer:**

For `offer.buy == true` and `offer.maxAssets > 0`, `consumableUnits` (ConsumableUnitsLib.sol line 19) calls `buyerAssetsToUnits`, not `sellerAssetsToUnits`. `buyerAssetsToUnits` divides by `buyerPrice = offerPrice` (non-zero), so no panic.

**Attacker-controlled inputs:**

The taker controls the `takes[]` array passed to `supplyCollateralAndSellWithAssetsTarget`. The taker can include any offer with `offer.buy == true`, `offer.maxAssets > 0`, and a tick satisfying `tickToPrice(tick) == settlementFee(id, timeToMaturity)`. The settlement fee is a function of time-to-maturity and can equal a tick price at specific moments (e.g., as maturity approaches and the interpolated fee crosses a tick-price boundary, or when the admin sets a fee that coincidentally matches a tick price). The taker need not be the offer maker.

**Why existing checks fail:**

The `try/catch` at lines 292–300 only guards `Midnight.take`. The pre-computation calls (`sellerAssetsToUnits`, `consumableUnits`) are outside it. The NatSpec comment at line 246 ("Reverts if TakeAmountsLib or ConsumableUnitsLib reverts") acknowledges this but does not justify it as correct behavior — the bundler's stated design intent is to skip offers that `Midnight.take` would reject, yet here `Midnight.take` would accept the offer.

### Impact Explanation

The entire `supplyCollateralAndSellWithAssetsTarget` transaction reverts. All collateral supplies (lines 269–275, executed before the loop) are also rolled back. The taker cannot sell units or supply collateral even though all other offers in the bundle are valid and `Midnight.take` would accept them individually. The taker is blocked from executing the bundle until they identify and remove the offending offer.

### Likelihood Explanation

**Preconditions:**
1. A buy offer with `offer.buy == true`, `offer.maxAssets > 0` must exist.
2. `tickToPrice(offer.tick) == settlementFee(id, timeToMaturity)` must hold at call time.
3. `targetSellerAssets > 0` (loop must execute).

**Feasibility:** Settlement fees are interpolated linearly between breakpoints as time-to-maturity decreases. For any market with a non-zero settlement fee, there exists a moment in time when the interpolated fee equals `tickToPrice(t)` for some tick `t`. A lender can create a buy offer at that tick. The taker including that offer in a bundle triggers the panic. The condition is not permanent — it holds only at specific time windows — but it is repeatable and can be targeted by fuzzing tick and fee values. The comment in Midnight.sol ("if the settlement fee gets increased, it might implicitly cancel offers with very low price") confirms the protocol designers are aware that fee/price equality is reachable.

### Recommendation

Add a guard for `sellerPrice == 0` in `TakeAmountsLib.sellerAssetsToUnits`. The cleanest fix mirrors the existing `buyerAssetsToUnits` guard pattern:

```solidity
// In sellerAssetsToUnits, after computing sellerPrice:
require(sellerPrice > 0 || !offer.buy, SellerPriceZero());
```

Alternatively, return `type(uint256).max` when `sellerPrice == 0` (signalling "infinite units needed"), which causes `min(...)` to select 0 from the other bounds and the offer is effectively skipped. Additionally, the call to `sellerAssetsToUnits` at MidnightBundles.sol line 286 should be wrapped in a try/catch or preceded by an inline `sellerPrice > 0` check so that a revert from this library does not propagate out of the bundle.
