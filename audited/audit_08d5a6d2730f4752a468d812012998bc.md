### Title
Authorized party can revoke EcrecoverAuthorizer's Midnight delegation, permanently blocking victim's off-chain revocation path - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` calls `midnight.setIsAuthorized` on behalf of the victim, which requires `isAuthorized[victim][ecrecoverAuthorizer]` to be true at call time. Once X is authorized via EcrecoverAuthorizer, X can call `midnight.setIsAuthorized(address(ecrecoverAuthorizer), false, victim)` directly, revoking EcrecoverAuthorizer's delegation. After this, any subsequent EcrecoverAuthorizer call on behalf of the victim reverts at Midnight's authorization check, making off-chain revocation of X impossible.

### Finding Description
**Root cause:** `Midnight.setIsAuthorized` (line 732) allows any currently-authorized delegate to revoke the authorization of any other address, including the EcrecoverAuthorizer contract itself. There is no restriction preventing a delegate from revoking the authorization of the intermediary that granted them access.

**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` (lines 33–36) verifies the signer is the authorizer or an authorized delegate, then unconditionally calls: [1](#0-0) 

`Midnight.setIsAuthorized` (line 732) checks: [2](#0-1) 

When EcrecoverAuthorizer is the `msg.sender` and victim is `onBehalf`, the check `isAuthorized[victim][ecrecoverAuthorizer]` must be true. X, once authorized, can call `midnight.setIsAuthorized(ecrecoverAuthorizer, false, victim)` directly — satisfying line 732 via `isAuthorized[victim][X]` — and set `isAuthorized[victim][ecrecoverAuthorizer] = false`. All subsequent EcrecoverAuthorizer calls on behalf of victim then fail at line 732.

**Attacker-controlled inputs:** The attacker controls the `Authorization` struct fields and the timing of the `midnight.setIsAuthorized(ecrecoverAuthorizer, false, victim)` call. No privileged access is required.

**Why existing checks fail:** The `signer == authorization.authorizer` check in EcrecoverAuthorizer (line 33) only validates the signature; it does not guarantee EcrecoverAuthorizer remains authorized in Midnight at call time. Midnight's `setIsAuthorized` has no restriction on delegates revoking other delegates' authorizations. [3](#0-2) 

### Impact Explanation
After X executes Step 3, the victim's only recourse is a direct on-chain transaction to `midnight.setIsAuthorized`. For smart contract wallets that exclusively support EIP-712 off-chain signing (e.g., certain account abstraction wallets with no direct execution path), this is permanently impossible. The victim cannot re-authorize EcrecoverAuthorizer via EcrecoverAuthorizer itself (circular dependency: EcrecoverAuthorizer needs to be authorized to re-authorize itself), and cannot revoke X via EcrecoverAuthorizer. X retains full Midnight control over the victim's account indefinitely.

### Likelihood Explanation
**Preconditions:** (1) Victim has authorized EcrecoverAuthorizer in Midnight — the normal setup for any EcrecoverAuthorizer user. (2) Victim has signed an authorization for X — the normal use case. Both are standard, non-exceptional states. The attack requires X to be malicious and execute one additional direct call after being authorized. It is repeatable: any time a victim uses EcrecoverAuthorizer to authorize a new party, that party can execute this attack.

### Recommendation
In `Midnight.setIsAuthorized`, add a guard preventing a delegate from revoking the authorization of any address other than themselves when acting on behalf of another user:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
+   // A delegate cannot revoke other delegates; only the account owner or the delegate itself can.
+   if (msg.sender != onBehalf && !newIsAuthorized) {
+       require(authorized == msg.sender, Unauthorized());
+   }
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
}
```

This ensures that only the account owner (direct on-chain call) can revoke third-party authorizations, while delegates retain the ability to self-revoke.
