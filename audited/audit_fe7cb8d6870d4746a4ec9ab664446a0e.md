Audit Report

## Title
Off-by-one `<=` in `CannotIncreaseDebtPostMaturity` guard permits debt creation at exact maturity timestamp - (File: `src/Midnight.sol`)

## Summary
The guard at `src/Midnight.sol:391` uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` at the exact maturity second, allowing `sellerDebtIncrease > 0` to pass. New debt is written with zero `pendingFee` (because `timeToMaturity == 0` zeroes the continuous-fee term), and post-maturity liquidation is simultaneously unavailable for that block because it requires `block.timestamp > market.maturity` (strict `>`).

## Finding Description
**Root cause confirmed at line 391:** [1](#0-0) 

`block.timestamp <= offer.market.maturity` is `true` at equality, so the `require` passes unconditionally regardless of `sellerDebtIncrease`.

**Exploit flow (all lines confirmed):**

1. `timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp)` yields `0` at equality. [2](#0-1) 

2. `buyerPendingFeeIncrease = buyerCreditIncrease.mulDivDown(_marketState.continuousFee * 0, WAD)` → `0`. No continuous fee is charged on new credit. [3](#0-2) 

3. `sellerDebtIncrease = units - sellerCreditDecrease` is `> 0` when the seller has no existing credit. [4](#0-3) 

4. The guard at line 391 passes; `sellerPos.debt += sellerDebtIncrease` is written to storage. [5](#0-4) 

5. Post-maturity liquidation requires strict `block.timestamp > market.maturity`; at exactly maturity this path is closed for the entire block. [6](#0-5) 

**Why existing checks fail:**
- The `CannotIncreaseDebtPostMaturity` guard is the sole maturity-boundary enforcement for debt creation; its `<=` makes it ineffective at the exact maturity second.
- `SellerIsLiquidatable` at line 476 enforces collateral health, not the maturity boundary — a well-collateralized seller passes. [7](#0-6) 
- Pre-maturity liquidation (`postMaturityMode = false`) remains available during that block only if the seller becomes undercollateralized, but the `isHealthy` check at line 476 ensures the seller is healthy at take-time, so the window of unprotected debt is real.

**Dual-boundary inconsistency:** At `block.timestamp == market.maturity`, the protocol simultaneously treats the timestamp as pre-maturity for the debt-increase guard (`<=` passes) and as post-maturity for fee accounting (`timeToMaturity == 0`, zero continuous fee on new credit). [8](#0-7) 

## Impact Explanation
Debt is created at the exact maturity second in violation of the core protocol invariant. The buyer's credit position carries zero `pendingFee` despite being created at the terminal boundary, representing an accounting loss to lenders. For one full block, the newly created debt cannot be liquidated via the post-maturity path. If collateral value drops in that window (e.g., oracle update in the same or next block), the position can become bad debt before any liquidator can act under post-maturity mode, resulting in a concrete loss of lender funds.

## Likelihood Explanation
Any unprivileged taker can trigger this by calling `take()` with a sell offer at a block where `block.timestamp == market.maturity`. On Ethereum, validator-influenced timestamps make exact-second targeting feasible; on L2s (Arbitrum, Base) sequencers set timestamps with fine granularity, making this straightforward. The precondition — seller has collateral and no existing credit — is easily arranged. The attack is repeatable across any market whose maturity falls on a reachable block timestamp.

## Recommendation
Change `<=` to `<` at line 391:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the debt-creation guard with the post-maturity liquidation check (`block.timestamp > market.maturity`) and eliminates the dual-boundary inconsistency. [1](#0-0) 

## Proof of Concept
**Minimal Foundry test plan:**

1. Deploy a market with `maturity = T`.
2. Warp to `block.timestamp = T` (`vm.warp(T)`).
3. Create a sell offer from a seller with collateral and zero existing credit.
4. Call `take()` with `units > 0` as an unprivileged taker.
5. Assert: `sellerPos.debt > 0` (debt was written).
6. Assert: `buyerPos.pendingFee == 0` (zero continuous fee charged).
7. Attempt `liquidate(..., postMaturityMode = true, ...)` — assert it reverts with `NotLiquidatable()` because `block.timestamp > market.maturity` is `false`.
8. Warp to `T + 1`; assert post-maturity liquidation now succeeds.

Steps 5–7 together prove the invariant violation and the one-block liquidation gap.
