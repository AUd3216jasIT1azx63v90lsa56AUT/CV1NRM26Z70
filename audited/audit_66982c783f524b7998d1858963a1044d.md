Audit Report

## Title
Buy offer `maxAssets` cap bypassed via zero-rounding when `buyerPrice < WAD` — (File: `src/Midnight.sol`)

## Summary
When a buy offer sets `maxAssets > 0` and the tick yields a `buyerPrice < WAD`, calling `take` with `units = 1` causes `buyerAssets` to floor to zero via `mulDivDown`. Because `consumed` is incremented by zero, the guard `newConsumed <= offer.maxAssets` never advances and the offer can be replayed an unlimited number of times. Each replay mutates lender credit, borrower debt, and `totalUnits` while transferring zero tokens, completely defeating the assets-mode consumption cap.

## Finding Description
**Root cause — `src/Midnight.sol` line 363:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // floors to 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);
```

`mulDivDown(1, buyerPrice, WAD)` returns `0` for any `buyerPrice < WAD` because `1 * buyerPrice < WAD`.

**Guard — lines 367–369:**

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());  // += 0, always passes
}
```

`consumed` is incremented by `buyerAssets = 0`, so `newConsumed` never advances regardless of replay count.

**`tickToPrice` range — `src/libraries/TickLib.sol` lines 44–52:**

`tickToPrice` is bounded to `[0, WAD]`. For every tick below `MAX_TICK` (5820), the returned price is strictly less than WAD, making the zero-rounding condition reachable at virtually every valid tick.

**State mutations still execute — lines 408–417:**

Even with `buyerAssets = 0`, the position accounting runs unconditionally:
- `buyerPos.credit += buyerCreditIncrease`
- `sellerPos.debt += sellerDebtIncrease`
- `_marketState.totalUnits` incremented

The token transfer on line 455 (`safeTransferFrom(..., buyerAssets - sellerAssets)`) sends 0 tokens, so the attacker pays nothing.

**Sell path is safe** because it uses `mulDivUp`, which returns ≥ 1 for any `units > 0` and `sellerPrice > 0`, ensuring `consumed` always advances.

**Exploit flow:**
1. Attacker (or any user) creates a buy offer: `offer.buy = true`, `offer.maxAssets = X > 0`, any tick where `tickToPrice < WAD` (e.g. `MAX_TICK - 16`).
2. Calls `take(offer, ..., units = 1, ...)` repeatedly.
3. Each call: `buyerAssets = 0`, `consumed` unchanged, cap check passes, positions mutated, zero tokens transferred.
4. Offer is structurally "fully consumed" yet accepts unlimited further fills.

## Impact Explanation
The `maxAssets` cap — the sole mechanism preventing a buy offer from being overfilled in assets mode — is completely ineffective for any tick below `MAX_TICK`. An offer intended to consume at most `X` assets can be replayed indefinitely: the maker accumulates unbounded credit units while paying zero loan tokens, and the taker accumulates unbounded debt while receiving zero loan tokens. This directly violates the protocol invariant that offers cannot be replayed, overfilled, or reused, and constitutes unauthorized state mutation (unbounded credit/debt/totalUnits inflation) with zero economic cost.

## Likelihood Explanation
No privileged access, oracle manipulation, or special token behavior is required. The condition `tickToPrice < WAD` holds for every tick except `MAX_TICK` itself. The attacker controls `units = 1` directly. The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` reproduces the exact scenario, confirming the path is reachable with standard setup.

## Recommendation
Add a floor-of-one guard before the cap check when `units > 0`:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// Prevent zero-rounding bypass of the maxAssets cap
if (offer.buy && offer.maxAssets > 0) require(buyerAssets > 0, ZeroBuyerAssets());
```

Alternatively, track consumed in units (not assets) when `buyerAssets` would round to zero, or enforce a minimum `units` such that `units * buyerPrice >= WAD` before proceeding. The sell path's use of `mulDivUp` is the correct pattern and should be mirrored for the buy-offer assets accounting.

## Proof of Concept
The repository already contains a reproducing test at `test/TakeTest.sol` (`testBugBuyMaxAssetsBypass`). Minimal manual steps:

1. Deploy with `tickSpacing = 1`.
2. Create buy offer: `buy = true`, `maxAssets = 1`, `tick = MAX_TICK - 16`.
3. Call `take(..., units = 1)` twice.
4. Assert: `consumed[maker][group]` remains `0` after both calls; `totalUnits` has increased by 2; zero loan tokens transferred.
5. Expected (correct) behavior: second call reverts with `ConsumedAssets()`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
