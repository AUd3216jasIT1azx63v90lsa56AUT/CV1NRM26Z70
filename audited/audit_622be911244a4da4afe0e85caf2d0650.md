### Title
Unbounded Oracle Gas Consumption in `liquidate()` While Loop Enables Permanent Liquidation DoS - (File: src/Midnight.sol)

### Summary

The `liquidate()` function iterates over all activated collaterals in an unbounded while loop, calling `IOracle(_collateralParam.oracle).price()` for each with no gas cap. Since market creation is permissionless and any address implementing `IOracle` is accepted as an oracle, an attacker who is both market creator and borrower can deploy gas-consuming oracles, activate all 16 collaterals, and render their position permanently unliquidatable by exhausting transaction gas before the repay/seize logic executes.

``` [1](#0-0) 

The loop runs up to `MAX_COLLATERALS_PER_BORROWER = 16` times. [2](#0-1) 

Each iteration makes an uncapped external `view` call to an oracle address that is fixed at market creation time and embedded in `CollateralParams`. [3](#0-2) 

**Root cause:** No gas stipend or limit is applied to `IOracle.price()` calls. Market creation is permissionless — any contract implementing `IOracle` is accepted. There is no oracle validation at market creation.

**Exploit flow:**

1. Attacker (market creator) deploys 16 oracle contracts, each implementing `IOracle.price()` to return a valid price while consuming approximately `block_gas_limit / 17` gas (e.g., via a tight computation loop).
2. Attacker creates a market with these 16 collateral params.
3. Attacker (borrower) calls `supplyCollateral` for all 16 collaterals, activating each in the bitmap. The `TooManyActivatedCollaterals` check at line 539 allows exactly 16. [4](#0-3) 

4. Attacker borrows, making the position healthy initially.
5. Attacker allows the position to become unhealthy (e.g., oracle prices drop legitimately, or post-maturity).
6. Any liquidator calls `liquidate()`. The while loop executes 16 oracle calls, each consuming `~block_gas_limit / 17` gas. Total oracle gas ≈ `16 × (block_gas_limit / 17)` ≈ `0.94 × block_gas_limit`. The transaction runs out of gas before reaching the bad-debt realization and repay/seize logic at lines 626–677. [5](#0-4) 

**Why existing checks fail:** The LIVENESS section (lines 142–158) acknowledges oracle *reverts* blocking liquidation but does not address gas exhaustion. No gas limit is imposed anywhere on oracle calls. The `liquidatorGate` check at line 598 and the `NotLiquidatable` check at line 620 are both *after* the while loop, so they cannot rescue a gas-exhausted transaction. [6](#0-5) 

### Impact Explanation

The unhealthy position becomes permanently unliquidatable. Bad debt is never realized, violating the core invariant: *"unhealthy positions remain liquidatable."* The borrower retains their borrowed assets while lenders bear unrecognized losses. No liquidator can succeed regardless of gas price or block gas limit, because the attacker can tune oracle gas consumption to always exceed available gas.

### Likelihood Explanation

**Preconditions:**
- Permissionless market creation (confirmed — no admin approval required).
- Attacker controls oracle contracts (trivially achievable as market creator).
- Attacker activates exactly 16 collaterals (enforced by `MAX_COLLATERALS_PER_BORROWER`).
- Position must be unhealthy or post-maturity (attacker can engineer this).

**Feasibility:** High. The attacker is the same address acting as market creator and borrower — both are permissionless roles. The gas threshold is tunable: the attacker can binary-search the exact gas consumption needed to exhaust any liquidation attempt. The attack is repeatable and irreversible once the position is established.

### Recommendation

Apply a gas cap to each oracle call in the while loop, e.g.:

```solidity
uint256 price = IOracle(_collateralParam.oracle).price{gas: MAX_ORACLE_GAS}();
```

Define `MAX_ORACLE_GAS` as a protocol constant (e.g., 100,000 gas) sufficient for legitimate oracles. Alternatively, validate oracle gas consumption at market creation time by calling `price()` with a gas measurement and rejecting oracles that exceed a threshold. A revert-on-out-of-gas wrapper (try/catch with a gas check) could also be used, though it would need to handle the revert-equals-liveness-failure case already documented.
