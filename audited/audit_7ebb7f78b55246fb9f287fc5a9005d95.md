### Title
Reentrancy via `onBuy` Callback in `take()` Allows Maker to Drain `withdrawable` Pool — (File: `src/Midnight.sol`)

---

### Summary

In `take()`, the buyer's credit position is fully updated before the `onBuy` callback fires, but the loan token payment is only pulled **after** the callback returns. A malicious maker contract can exploit this ordering by calling `withdraw()` inside `onBuy`, redeeming the newly credited units at 1:1 against the `withdrawable` pool, then paying only the discounted `buyerAssets` price. The difference — `units − buyerAssets` — is permanently extracted from the protocol's `withdrawable` reserve.

---

### Finding Description

**Root cause — violated CEI ordering in `take()`:**

In `src/Midnight.sol`, `take()` executes in this order:

1. **State written** — buyer credit increased, `totalUnits` updated (lines 408–418)
2. **`onBuy` callback fired** — external call to `buyerCallback` (lines 445–453)
3. **Payment pulled** — `safeTransferFrom(payer, …, buyerAssets)` (lines 455–456) [1](#0-0) [2](#0-1) 

The `payer` for the `safeTransferFrom` is set to `buyerCallback` when it is non-zero:

```solidity
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
``` [3](#0-2) 

When `offer.buy = true`, `buyerCallback = offer.callback` and `buyer = offer.maker`. If the maker is a smart contract that sets `offer.callback = address(this)`, then during `onBuy` the callback is `msg.sender` and `buyer == msg.sender`. This satisfies `withdraw()`'s authorization check:

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
``` [4](#0-3) 

**Exploit path inside `onBuy`:**

The callback calls `withdraw(market, N, maker, maker)`. `withdraw()` requires only that `_position.credit >= N` and `_marketState.withdrawable >= N`:

```solidity
_position.credit     -= UtilsLib.toUint128(units);   // credit just granted by take()
_marketState.withdrawable -= UtilsLib.toUint128(units); // drains existing repaid pool
_marketState.totalUnits   -= UtilsLib.toUint128(units);
SafeTransferLib.safeTransfer(market.loanToken, receiver, units); // N tokens out
``` [5](#0-4) 

The maker's credit was just set to `C + N` by `take()`. The callback withdraws `N` units, receiving **N loan tokens**. Control returns to `take()`, which then pulls only `buyerAssets = N × price / WAD` tokens from the maker (the payer). Since `price < WAD` for any discounted zero-coupon offer, `buyerAssets < N`.

**Net accounting:**

| Flow | Tokens |
|---|---|
| Maker receives via `withdraw()` | `+N` |
| Maker pays via `safeTransferFrom` | `−N × price / WAD` |
| **Maker profit** | `N × (1 − price/WAD)` |
| Contract balance change | `−N × (1 − price/WAD)` |

The `withdrawable` pool is permanently reduced by `N` tokens while only `buyerAssets` are returned. There is no reentrancy guard on `take()` and no check preventing `withdraw()` from being called during the `onBuy` callback. [6](#0-5) 

The TOKEN SAFETY REQUIREMENTS (line 138) only prohibit token-level reentrancy; callback-level reentrancy into `withdraw()` is not restricted.

---

### Impact Explanation

**Direct theft of loan tokens from the `withdrawable` pool.** The attacker extracts `N × (1 − price/WAD)` loan tokens per exploit invocation. With a 10% discount (price = 0.9 WAD), 10% of `N` tokens are stolen per call. The attacker can repeat this for any `N ≤ withdrawable`, draining the entire pool in a single transaction. Lenders who held withdrawable credit lose the ability to redeem early and, critically, the contract's token balance is permanently short — meaning some lenders cannot be made whole at maturity even if all borrowers repay in full.

Severity: **Critical** — direct, repeatable theft of protocol funds with no privileged access required.

---

### Likelihood Explanation

The attacker only needs to:
1. Deploy a smart contract that implements `IBuyCallback` and calls `withdraw()` inside `onBuy`.
2. Create a signed buy offer with `offer.callback = address(this)`.
3. Have any counterparty (or themselves via an authorized address) call `take()`.

No admin keys, no oracle manipulation, no flash loan required. The `withdrawable` pool is funded by normal borrower repayments, so it will be non-zero in any active market. This is a **realistic, low-barrier attack** for any sophisticated DeFi actor.

---

### Recommendation

Apply the Checks-Effects-Interactions pattern strictly: pull the payment **before** invoking the `onBuy` callback. Swap the order of lines 445–453 and 455–456 so that `safeTransferFrom` executes first:

```solidity
// 1. Pull payment first
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);

// 2. Then fire the callback
if (buyerCallback != address(0)) {
    ...onBuy(...)...
}
``` [2](#0-1) 

Alternatively, add a `nonReentrant` guard to `take()` and `withdraw()`. The flash-loan-style callback design can still be preserved by restructuring the callback to operate post-payment (as `onSell` already does).

### Proof of Concept

```solidity
contract MaliciousMaker is IBuyCallback {
    Midnight midnight;
    Market  market;

    // Maker creates a buy offer: offer.callback = address(this), offer.buy = true
    // offer.tick set to a discounted price (e.g. price = 0.9 WAD)

    function onBuy(
        bytes32 id,
        Market memory _market,
        uint256 buyerAssets,
        uint256 units,
        uint128,
        address buyer,
        bytes memory
    ) external returns (bytes4) {
        // buyer == address(this) (maker == callback)
        // credit was just increased by `units` in take()
        // withdrawable >= units (precondition: market has repaid liquidity)
        midnight.withdraw(_market, units, address(this), address(this));
        // Received `units` loan tokens; will only pay `buyerAssets < units` after return
        return CALLBACK_SUCCESS;
    }
}
```

**Steps:**
1. Deploy `MaliciousMaker`, fund it with `buyerAssets` loan tokens, approve Midnight.
2. Create a buy offer signed by `MaliciousMaker` at a discounted tick (price < WAD), `offer.callback = address(MaliciousMaker)`.
3. Call `take()` against the offer with `units = withdrawable`.
4. `onBuy` fires → `withdraw()` transfers `units` tokens to `MaliciousMaker`.
5. `take()` pulls only `buyerAssets = units × price / WAD` from `MaliciousMaker`.
6. **Result:** `MaliciousMaker` holds `units − buyerAssets` extra tokens; `withdrawable` pool is drained.
