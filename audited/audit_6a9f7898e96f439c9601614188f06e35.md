### Title
Nonce post-increment overflow permanently locks authorizer at `type(uint256).max` - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
In `EcrecoverAuthorizer.setIsAuthorized`, the nonce check uses a post-increment inside the `require`: `require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce())`. Under Solidity 0.8.34's checked arithmetic, if `nonce[authorizer]` has reached `type(uint256).max`, the `++` write overflows and reverts with `Panic(0x11)` rather than succeeding. This makes nonce `type(uint256).max` permanently unreachable and permanently locks the authorizer from ever calling `setIsAuthorized` again.

### Finding Description
The root cause is on line 26 of `src/periphery/EcrecoverAuthorizer.sol`:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

Solidity 0.8.34 compiles post-increment with checked arithmetic. The sequence is:
1. Read `nonce[authorizer]` (current value used in comparison).
2. Write `nonce[authorizer] + 1` back to storage — this write is checked.

If `nonce[authorizer] == type(uint256).max`, step 1 makes the equality true (when `authorization.nonce == type(uint256).max`), but step 2 attempts `type(uint256).max + 1`, which overflows and reverts with `Panic(0x11)`. The `require` never resolves to `true`; the entire call reverts. After a user has successfully consumed nonce `type(uint256).max - 1` (leaving `nonce[authorizer] == type(uint256).max`), no further valid call to `setIsAuthorized` can ever succeed for that authorizer. ### Impact Explanation
Non-critical behavior divergence: the nonce space is effectively `[0, type(uint256).max - 1]` rather than `[0, type(uint256).max]`. An authorizer who exhausts all `2^256 - 1` nonces is permanently unable to call `setIsAuthorized` again. No funds are lost or frozen; only the authorization management capability is permanently disabled for that address.

### Likelihood Explanation
Precondition: `nonce[authorizer]` must equal `type(uint256).max`, requiring exactly `2^256 - 1` prior successful calls by the same authorizer. This is computationally and economically impossible in practice. The behavior divergence is real and reproducible in a unit test by directly manipulating storage, but it cannot be triggered by any realistic on-chain actor. Likelihood is effectively zero.

### Recommendation
Replace the inline post-increment inside `require` with an explicit pre-check and separate increment:

```solidity
uint256 currentNonce = nonce[authorization.authorizer];
require(authorization.nonce == currentNonce, InvalidNonce());
nonce[authorization.authorizer] = currentNonce + 1; // or use unchecked if wrap is acceptable
```

Alternatively, use `unchecked { nonce[authorization.authorizer]++; }` after the check if wrapping is acceptable, or add an explicit guard `require(currentNonce < type(uint256).max)` to surface a cleaner error.

contract NonceOverflowTest is Test {
    EcrecoverAuthorizer auth;

    function setUp() public {
        auth = new EcrecoverAuthorizer(address(0)); // mock midnight
    }

    function test_nonceOverflowAtMaxMinus1() public {
        address authorizer = vm.addr(1);

        // Force nonce to type(uint256).max - 1 via stdstore
        stdstore
            .target(address(auth))
            .sig("nonce(address)")
            .with_key(authorizer)
            .checked_write(type(uint256).max - 1);

        assertEq(auth.nonce(authorizer), type(uint256).max - 1);

        // Submit authorization.nonce == type(uint256).max - 1 with valid sig
        // (build EIP-712 digest, sign with key 1, call setIsAuthorized)
        // Assert: call succeeds
        // Assert: auth.nonce(authorizer) == type(uint256).max

        // Now submit authorization.nonce == type(uint256).max with valid sig
        // Assert: call reverts with Panic(0x11) arithmetic overflow
        // NOT with InvalidNonce()
        vm.expectRevert(stdError.arithmeticError);
        // auth.setIsAuthorized(authorizationWithMaxNonce, sig);
    }
}

Expected assertions:
- First call (nonce `type(uint256).max - 1`): succeeds, `nonce[authorizer]` becomes `type(uint256).max`.
- Second call (nonce `type(uint256).max`): reverts with `Panic(0x11)` (arithmetic overflow), not `InvalidNonce()`.
