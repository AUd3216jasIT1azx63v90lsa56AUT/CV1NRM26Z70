### Title
Off-by-one in maturity boundary check allows debt increase at exact maturity timestamp - (`src/Midnight.sol`)

### Summary

`Midnight.take` guards against post-maturity debt creation with `block.timestamp <= offer.market.maturity`, which is inclusive of the maturity timestamp itself. Because post-maturity liquidation requires `block.timestamp > market.maturity` (strictly greater), a position that receives new debt at the exact maturity second is immediately liquidatable one second later. An authorized operator can exploit this to force debt onto a victim at maturity and then seize their collateral.

### Finding Description

**Root cause — line 391 of `src/Midnight.sol`:**

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

The condition `block.timestamp <= offer.market.maturity` evaluates to `true` when `block.timestamp == offer.market.maturity`, so `sellerDebtIncrease > 0` is permitted at the exact maturity second.

**Liquidation boundary — line 622 of `src/Midnight.sol`:**

```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
```

Post-maturity liquidation requires `block.timestamp > market.maturity` (strictly greater). This is confirmed by `testLiquidatePostMaturityModeBeforeMaturity` (lines 170–173), which asserts `NotLiquidatable` at `block.timestamp == market.maturity` and succeeds at `market.maturity + 1`.

**Authorization — line 346 of `src/Midnight.sol`:**

```solidity
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());
```

An authorized operator can supply any `taker` address. Authorization is coarse-grained, not scoped by action or market (`live_context.json` lines 158–162).

**Exploit flow:**

1. Attacker obtains `isAuthorized[victim][attacker] = true` (e.g., victim authorized a router contract the attacker controls, or via `EcrecoverAuthorizer.setIsAuthorized` with a victim-signed `Authorization`).
2. Victim has collateral deposited in the market.
3. A buy offer exists with `offer.buy = true`.
4. Attacker warps/waits to `block.timestamp == offer.market.maturity`.
5. Attacker calls `take(buyOffer, ..., taker=victim, receiverIfTakerIsSeller=attacker)`.
6. `sellerDebtIncrease = units - sellerCreditDecrease > 0` passes line 391 because `block.timestamp <= maturity` is `true`.
7. `sellerPos.debt += sellerDebtIncrease` executes (line 414).
8. The `SellerIsLiquidatable` check at line 476 passes because the victim has sufficient collateral to remain healthy at maturity.
9. Attacker receives `sellerAssets` (line 456, paid by the buy-offer maker).
10. At `block.timestamp = maturity + 1`, `postMaturityMode` liquidation becomes available. Attacker calls `liquidate(..., borrower=victim, postMaturityMode=true, ...)` and seizes victim's collateral.

**Why existing checks fail:**

- Line 391 uses `<=` instead of `<`, so the maturity second is not excluded.
- The `SellerIsLiquidatable` check (line 476) only verifies health at the time of the take; it does not prevent debt creation at maturity.
- The `enterGate` check (lines 402–406) only applies if a gate is configured; most markets have `address(0)`.

### Impact Explanation

A victim with collateral in a market has debt imposed on them at the exact maturity second by an authorized operator. One second later, the victim's position is liquidatable via post-maturity mode regardless of collateral health. The attacker can seize the victim's collateral and also receives `sellerAssets` from the buy-offer maker during the take itself. The victim suffers collateral loss without having voluntarily borrowed.

### Likelihood Explanation

**Preconditions:**
- Attacker is authorized for victim. This is realistic: users routinely authorize routers, composors, or periphery contracts. The `EcrecoverAuthorizer` also allows authorization via off-chain signatures, which could be phished.
- `block.timestamp == market.maturity`. On chains with 1-second block times (e.g., Ethereum post-merge), this is a single target block. On chains with sub-second blocks, the window is wider. A searcher/MEV bot can target this block precisely.
- A buy offer exists. Resting buy offers are the normal operating state of the protocol.
- Victim has collateral. Required for the `SellerIsLiquidatable` check to pass; victims with collateral are the most valuable targets.

The attack is repeatable across every market at its maturity second and requires no special privileges beyond a prior authorization.

### Recommendation

Change the comparison operator from `<=` to `<` on line 391:

```solidity
// Before (buggy):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the debt-increase guard with the post-maturity liquidation guard (`block.timestamp > market.maturity`) and with the protocol invariant stated in `live_context.json` line 221: *"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."*

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IMidnight, Market, Offer} from "src/interfaces/IMidnight.sol";

contract MaturityBoundaryDebtTest is Test {
    Midnight midnight;
    address victim   = address(0xBEEF);
    address attacker = address(0xBAD);
    address maker    = address(0xABC);   // buy-offer maker / lender

    // ... (standard market/token/oracle setup as in TakeTest.sol) ...

    function testDebtIncreaseAtExactMaturity() public {
        // 1. Setup: victim has collateral, maker has a resting buy offer.
        //    attacker is authorized for victim.
        midnight.setIsAuthorized(attacker, true, victim);  // victim authorizes attacker
        // collateralize victim so SellerIsLiquidatable passes
        collateralize(market, victim, UNITS);

        // 2. Warp to exact maturity.
        vm.warp(market.maturity);

        // 3. Attacker takes the buy offer with taker=victim, receiver=attacker.
        vm.prank(attacker);
        midnight.take(buyOffer, "", UNITS, victim, attacker, address(0), "");

        // 4. Assert: victim now has debt at maturity — INVARIANT VIOLATED.
        assertGt(midnight.debtOf(id, victim), 0, "victim has debt at maturity");

        // 5. One second later: victim is liquidatable via post-maturity mode.
        vm.warp(market.maturity + 1);
        // Should NOT revert — victim is immediately liquidatable.
        midnight.liquidate(market, 0, 0, 0, victim, true, attacker, address(0), "");

        // 6. Assert: victim's collateral was seized.
        assertEq(midnight.collateral(id, victim, 0), 0, "victim collateral seized");

        // Fuzz assertion: for any timestamp >= maturity, sellerDebtIncrease must be 0.
        // assertEq(sellerDebtIncrease, 0) when block.timestamp >= market.maturity
    }
}
```

**Expected assertions:**
- `debtOf(id, victim) > 0` after the take at `block.timestamp == maturity` — demonstrates the invariant breach.
- `liquidate(..., postMaturityMode=true)` succeeds at `maturity + 1` — demonstrates immediate liquidatability.
