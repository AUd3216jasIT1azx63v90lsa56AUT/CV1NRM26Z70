### Title
Same-ChainId Fork Signature Replay via Call-Time Domain Separator - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` computes the EIP-712 domain separator at call time using `block.chainid`, not at construction. When a hard fork retains the original chain's chainid (at least temporarily), the domain separator is byte-for-byte identical on both chains. A victim's signed `Authorization` — including its nonce — is therefore valid on the fork, and any party in possession of the signature can replay it there to set an unauthorized `isAuthorized` mapping in `Midnight`.

### Finding Description
**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` (line 29):
```solidity
bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
``` [1](#0-0) 

The domain separator is `keccak256(EIP712_DOMAIN_TYPEHASH || block.chainid || address(this))`. No construction-time chainid is captured; the value is read fresh on every call.

**Root cause:** If a hard fork launches with the same chainid as the original chain (as Ethereum Classic did with chainid 1 before EIP-155 replay protection was deployed), `block.chainid` returns the same value on both chains. The domain separator is therefore identical, and any signature produced on the original chain is cryptographically valid on the fork.

**Nonce does not save you:** The nonce stored in `EcrecoverAuthorizer` is part of the forked state. If the fork diverges at block N (before the victim's transaction is mined), the fork's nonce for the victim is still 0. The victim later submits their signature on the original chain (nonce advances to 1 there), but on the fork the nonce remains 0. The attacker replays the original-chain signature on the fork; the nonce check `authorization.nonce == nonce[authorization.authorizer]++` passes. [2](#0-1) 

**Exploit flow:**
1. Pre-fork: victim calls `midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim)` — this state is shared by both chains after the fork.
2. Fork occurs at block N; both chains share identical state including `EcrecoverAuthorizer.nonce[victim] == 0`.
3. Victim signs `Authorization{authorizer: victim, authorized: attacker, isAuthorized: true, nonce: 0, deadline: T}` on the original chain and submits it there (nonce → 1 on original chain).
4. Attacker submits the same `(auth, sig)` tuple to `EcrecoverAuthorizer.setIsAuthorized` on the fork. `block.chainid` is identical → domain separator matches → `ecrecover` recovers `victim` → nonce check passes (fork nonce is still 0) → `Midnight.setIsAuthorized(attacker, true, victim)` is called and succeeds. [3](#0-2) 

**Why existing checks fail:**
- The nonce prevents same-chain replay but not cross-fork replay when the fork diverged before the signature was consumed.
- The deadline check is irrelevant; the attacker replays before the deadline.
- The developer comment ("If block.chainid changes…") explicitly scopes protection only to forks that change the chainid, leaving same-chainid forks unprotected. [4](#0-3) 

`Midnight.setIsAuthorized` only checks `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`; since `ecrecoverAuthorizer` was already authorized by the victim (shared fork state), this passes. [5](#0-4) 

### Impact Explanation
On the forked chain, the attacker gains `isAuthorized[victim][attacker] == true`. This lets the attacker call any `onBehalf`-gated function in `Midnight` as the victim: `withdraw`, `withdrawCollateral`, `repay`, `setConsumed`, and further `setIsAuthorized` calls — enabling complete drain of the victim's funds on the fork.

### Likelihood Explanation
Preconditions: (a) a hard fork that does not immediately change chainid; (b) the fork diverges before the victim's authorization transaction is mined. Condition (a) has historical precedent (Ethereum/Ethereum Classic). Condition (b) is trivially satisfied if the attacker front-runs the victim's submission on the fork, or if the fork diverges before the victim signs. The attack is repeatable for every victim who signed an authorization after the fork point.

### Recommendation
Capture the chainid at construction time (as `Midnight` does with `INITIAL_CHAIN_ID`) and use it in the domain separator, or cache the domain separator at construction and recompute only when `block.chainid` differs from the cached value (the pattern used by OpenZeppelin's `EIP712` base). Either approach ensures that a signature produced before a chainid change cannot be replayed after the fork adopts a new chainid, and that a same-chainid fork cannot replay signatures because the construction-time chainid was recorded in a different deployment context. [6](#0-5) 

### Proof of Concept
```solidity
// Foundry differential test
function testSameChainIdForkReplay() public {
    // 1. Setup: victim authorizes ecrecoverAuthorizer in Midnight
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);

    // 2. Victim signs Authorization on "original chain" (chainid = 1)
    Authorization memory auth = Authorization({
        authorizer: victim,
        authorized: attacker,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    bytes32 domainSep = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(ecrecoverAuthorizer)));
    bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSep, keccak256(abi.encode(AUTHORIZATION_TYPEHASH, auth))));
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(victimKey, digest);
    Signature memory sig = Signature(v, r, s);

    // 3. Simulate fork: chainid remains the same (vm.chainId stays at 1)
    //    Fork state: nonce[victim] == 0 (victim's tx not yet mined on fork)

    // 4. Attacker replays on fork
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // 5. Assertions
    assertEq(midnight.isAuthorized(victim, attacker), true);  // unauthorized authorization set
    assertEq(ecrecoverAuthorizer.nonce(victim), 1);           // nonce consumed on fork

    // 6. Attacker drains victim's funds
    vm.prank(attacker);
    midnight.withdraw(market, victimUnits, victim, attacker); // succeeds
    assertEq(loanToken.balanceOf(attacker), victimUnits);     // funds drained
}
```

Expected: all assertions pass, demonstrating that a signature signed on the original chain is replayable on a same-chainid fork, enabling fund drain.
