Audit Report

## Title
Authorized-Signer Front-Run Advances Victim's Nonce and Invalidates Pending Signed Authorization - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts any address that `authorization.authorizer` has previously authorized in Midnight as a valid signer for an arbitrary `Authorization` struct at the current nonce. Because the nonce is keyed only on `authorization.authorizer` and there is no binding between the signer's identity and the `authorized` field, an already-authorized attacker can front-run the victim's pending transaction with a different struct at the same nonce, consuming nonce N and causing the victim's transaction to revert with `InvalidNonce()`. This is indefinitely repeatable and constitutes a persistent DoS of the victim's signature-gated delegation path.

## Finding Description

**Root cause:**

In `EcrecoverAuthorizer.setIsAuthorized`, the nonce is keyed solely on `authorization.authorizer`:

```solidity
// src/periphery/EcrecoverAuthorizer.sol line 26
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

The signer check accepts any address for which `IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)` returns `true`:

```solidity
// lines 33-36
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

There is no binding between the signer's identity and the `authorized` field in the struct. The `Authorization` struct (`src/periphery/interfaces/IEcrecoverAuthorizer.sol` lines 11-17) contains five independently attacker-controllable fields: `authorizer`, `authorized`, `isAuthorized`, `nonce`, `deadline`.

**Exploit flow:**

1. **Precondition**: `isAuthorized[victim][attacker] == true` in Midnight (victim previously authorized attacker as an operator/relayer — a normal, expected state).
2. Victim signs `Authorization{authorizer=victim, authorized=X, isAuthorized=true, nonce=N}` and broadcasts.
3. Attacker observes victim's tx in mempool; crafts `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N}` (same nonce, different `authorized`).
4. Attacker signs this struct with their own private key → `ecrecover(digest, ...) == attacker`.
5. Attacker submits with higher gas priority.
6. Contract execution:
   - Line 25: deadline check passes (attacker sets any valid future deadline).
   - Line 26: `N == nonce[victim]` → passes; nonce increments to N+1.
   - Line 31: `signer = attacker`.
   - Lines 33–36: `isAuthorized[victim][attacker] == true` → passes.
   - Line 47: `Midnight.setIsAuthorized(attacker, true, victim)` executes.
7. Victim's tx arrives: `N != N+1` → reverts with `InvalidNonce()`.

**Why existing checks fail:**

- The nonce check prevents replay of the *same* signed message but does not prevent a *different* message at the same nonce from a delegated signer.
- The `isAuthorized` delegation path was designed to allow authorized agents to act on behalf of the authorizer, but it does not scope what struct content the agent may sign — the agent can sign any `authorized`/`isAuthorized` combination.
- There is no binding between the signer's identity and the `authorized` field in the struct.

## Impact Explanation

The victim's pending signed `Authorization` is permanently invalidated (DoS of delegation). The victim must sign a new message at nonce N+1, which the attacker can front-run again indefinitely. This constitutes a persistent, unrecoverable service degradation for the victim's signature-gated delegation path through `EcrecoverAuthorizer`. Additionally, the attacker can set arbitrary `isAuthorized` entries on behalf of the victim via `EcrecoverAuthorizer` — though note that an already-authorized operator can also call `Midnight.setIsAuthorized` directly (as confirmed by `testSetIsAuthorizedAuthorization` in `test/AuthorizationTest.sol` lines 290–304). The primary unique impact is the indefinite DoS of the gasless authorization flow.

## Likelihood Explanation

The precondition (`isAuthorized[victim][attacker] == true`) is a common, expected state — users routinely authorize operators, bots, and relayers to act on their behalf. Once met, the attack requires only mempool monitoring and a higher-gas competing transaction, which is standard MEV infrastructure. The attack is trivially repeatable on every future signed authorization the victim broadcasts, making it a persistent griefing vector with no natural expiry.

## Recommendation

Bind the signer's identity to the struct content. The simplest fix is to require that a delegated signer (one who is not the authorizer themselves) may only sign a struct where `authorization.authorized == signer`. This prevents a delegated signer from crafting a struct that authorizes a third party:

```solidity
require(
    signer == authorization.authorizer
        || (IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)
            && signer == authorization.authorized),
    Unauthorized()
);
```

Alternatively, remove the delegated-signer path from `EcrecoverAuthorizer` entirely and require that only the authorizer themselves may sign (the delegation use-case can be served by direct `Midnight.setIsAuthorized` calls). A third option is to include the expected signer address in the signed struct itself, so the digest commits to who is permitted to submit it.

## Proof of Concept

Minimal Foundry test (extending `EcrecoverAuthorizerTest` / `BaseTest`):

```solidity
function testFrontRunNonceInvalidation() public {
    // Setup: victim (borrower) authorizes attacker in Midnight
    address attacker = makeAddr("attacker");
    uint256 attackerKey = 0xDEAD; // attacker's private key
    attacker = vm.addr(attackerKey);

    vm.prank(borrower);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);
    vm.prank(borrower);
    midnight.setIsAuthorized(attacker, true, borrower);

    // Victim prepares a signed authorization for address X
    address X = makeAddr("X");
    Authorization memory victimAuth = makeAuthorization(borrower, X, true);
    Signature memory victimSig = signAuthorization(victimAuth, borrower);

    // Attacker front-runs with a different struct at the same nonce
    Authorization memory attackerAuth = Authorization({
        authorizer: borrower,
        authorized: attacker,
        isAuthorized: true,
        nonce: victimAuth.nonce, // same nonce N
        deadline: block.timestamp + 1 days
    });
    bytes32 structHash = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, attackerAuth));
    bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(ecrecoverAuthorizer)));
    bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, structHash));
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(attackerKey, digest);
    Signature memory attackerSig = Signature({v: v, r: r, s: s});

    // Attacker submits first — consumes nonce N
    ecrecoverAuthorizer.setIsAuthorized(attackerAuth, attackerSig);
    assertEq(ecrecoverAuthorizer.nonce(borrower), 1);

    // Victim's tx now reverts
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(victimAuth, victimSig);

    // X was never authorized; attacker was
    assertEq(midnight.isAuthorized(borrower, X), false);
    assertEq(midnight.isAuthorized(borrower, attacker), true);
}
