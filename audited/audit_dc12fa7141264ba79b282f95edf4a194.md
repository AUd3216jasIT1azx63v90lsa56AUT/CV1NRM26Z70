Audit Report

## Title
Fee-on-Transfer Loan Token Causes `withdrawable` Over-Accounting in `repay` - (File: `src/Midnight.sol`)

## Summary
The `repay` function increments `marketState[id].withdrawable` by `units` before calling `safeTransferFrom`, which only validates that `transferFrom` did not revert and returned `true`. When the loan token charges a transfer fee `f`, the contract receives `units*(1-f)` but records `withdrawable += units`, creating a per-call deficit of `f*units`. This deficit accumulates until lenders' `withdraw` calls revert due to insufficient contract balance, rendering the market insolvent.

## Finding Description
**Code path:**

`src/Midnight.sol` lines 508–520:
```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);   // line 508
marketState[id].withdrawable += UtilsLib.toUint128(units);  // line 509
...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
```

`src/libraries/SafeTransferLib.sol` lines 24–34: `safeTransferFrom` calls `transferFrom(from, to, value)` and only checks that the call did not revert and that the return value is `true` (or absent). It performs no balance-before/balance-after check and cannot detect that the contract received fewer tokens than `value`.

**Root cause:** Optimistic accounting in `repay` combined with the absence of received-amount verification in `safeTransferFrom`.

**Exploit flow:**
1. A market is created (permissionlessly) with a fee-on-transfer ERC-20 as `loanToken` (fee rate `f`).
2. Lenders supply to the market; borrower takes a loan of `units` via `take`.
3. Borrower calls `repay(market, units, onBehalf, address(0), "")`.
4. `position[id][onBehalf].debt -= units` — debt fully cleared.
5. `marketState[id].withdrawable += units` — full `units` credited.
6. `safeTransferFrom(loanToken, msg.sender, address(this), units)` — token deducts fee; Midnight receives `units*(1-f)`.
7. `safeTransferFrom` returns without error because `transferFrom` returned `true`.
8. Net result: `withdrawable` overstated by `f*units`; contract balance short by `f*units`.

**Why existing checks fail:** `SafeTransferLib.safeTransferFrom` validates only `success` and the boolean return value — no `balanceOf(address(this))` snapshot comparison is performed. The Certora invariant `tokenBalanceCorrect` in `Solvency.spec` and the rule `repayIncreasesWithdrawable` in `WithdrawableMonotonicity.spec` are both proven under the explicit assumption that tokens transfer with no fee-taking, so they provide no coverage for this token class.

## Impact Explanation
Each `repay` call with a fee-on-transfer loan token causes `withdrawable` to exceed the contract's actual token balance by `f*units`. Lenders calling `withdraw` will drain the real balance before all `withdrawable` credit is consumed, causing the final lenders' `withdraw` calls to revert with insufficient balance. The protocol becomes insolvent by the cumulative sum of all fee shortfalls. Debt is fully erased while the backing assets are not fully present. This constitutes direct, permanent loss of lender funds — a critical in-scope impact.

## Likelihood Explanation
Market creation is permissionless; any actor can deploy a market with a fee-on-transfer loan token. The borrower need only have an outstanding debt position and call `repay` normally — no special privilege, no reentrancy, no oracle manipulation. The attack is repeatable on every `repay` call and compounds linearly with repayment volume. Fee-on-transfer ERC-20 tokens exist in production (e.g., tokens with configurable fee mechanisms). No victim mistake is required; lenders who supplied in good faith are harmed by the accounting drift.

## Recommendation
Replace the optimistic accounting pattern with a balance-delta check. Before and after `safeTransferFrom`, snapshot `IERC20(market.loanToken).balanceOf(address(this))` and use the actual received amount (the delta) to increment `withdrawable` rather than the nominal `units` parameter. Alternatively, document and enforce at market creation that fee-on-transfer tokens are unsupported (e.g., via a whitelist or an explicit received-amount assertion), and add a corresponding check in `safeTransferFrom` or at the call site in `repay`.

## Proof of Concept
1. Deploy a mock ERC-20 with a 1% transfer fee.
2. Create a Midnight market with this token as `loanToken`.
3. Lender calls `supply` for 1000 units; borrower calls `take` for 1000 units.
4. Borrower calls `repay(market, 1000, borrower, address(0), "")`.
5. Assert: `marketState[id].withdrawable == 1000` but `token.balanceOf(midnight) == 990` (10 units short).
6. Repeat step 4 ten times; assert lender's `withdraw` for the full credited amount reverts due to insufficient balance.
7. Foundry fuzz test: fuzz over fee rate `f ∈ [1, 10000]` and repay count `n ∈ [1, 100]`; assert `token.balanceOf(midnight) >= marketState[id].withdrawable` after each repay — this invariant will fail for any `f > 0`.
