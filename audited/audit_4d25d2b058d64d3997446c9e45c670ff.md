### Title
Missing allowlist enforcement in `ConfidencePool.initialize()` allows non-allowlisted tokens to bypass factory guard, risking permanent fund freeze - (File: src/ConfidencePool.sol)

### Summary
`ConfidencePoolFactory.createPool()` enforces the stake-token allowlist before deploying a pool, but `ConfidencePool.initialize()` is `external` with no access control and no corresponding allowlist check. Anyone can clone the pool implementation directly and call `initialize()` with a non-allowlisted token, creating a structurally valid pool that the factory's guard never vetted. A pool seeded with a negative-rebasing or fee-on-transfer token will have its on-chain balance eroded below tracked liabilities, permanently freezing staker funds.

### Finding Description
`ConfidencePoolFactory.createPool()` enforces the allowlist before deploying any clone: [1](#0-0) 

However, `ConfidencePool.initialize()` carries only the `initializer` modifier, which prevents re-initialization but places no restriction on the caller and performs no check that `stakeToken_` is allowlisted: [2](#0-1) 

Because `poolImplementation` is a public state variable on the factory, any actor can read it, call `Clones.clone(poolImplementation)` to obtain a fresh clone, and immediately call `initialize()` on that clone with an arbitrary `stakeToken_`. The factory's `_poolsByAgreement` mapping will not record this pool, but the clone is otherwise a fully functional `ConfidencePool` — same bytecode, same interface, same events — indistinguishable to a staker who does not independently verify pool provenance via `getPoolsByAgreement()`.

The factory's own NatDoc acknowledges exactly what happens when a defective token enters a pool: [3](#0-2) 

And DESIGN.md §12 states the allowlist is the mechanism that excludes this failure mode: [4](#0-3) 

The balance-diff defense inside `stake()` and `contributeBonus()` catches fee-on-transfer tokens only at deposit time (received amount vs. `minStake`): [5](#0-4) 

It provides no protection against post-deposit balance erosion from a negative-rebasing token, nor does it guard `withdraw()`, `claimSurvived()`, `claimExpired()`, or any sweep path.

### Impact Explanation
A pool initialized with a negative-rebasing token will have its actual `stakeToken` balance continuously eroded while `eligibleStake`, `totalEligibleStake`, and all snapshot values remain at their original figures. When stakers attempt to call `withdraw()`, `claimSurvived()`, or `claimExpired()`, the `safeTransfer` calls will revert with an insufficient-balance error, permanently freezing principal. Because `hasClaimed` is set before the transfer in `claimSurvived()` and `claimExpired()`, a failed transfer also marks the staker as having claimed, blocking any retry. The entire staked principal and bonus are irrecoverable. [6](#0-5) 

### Likelihood Explanation
The technical barrier is minimal: the `poolImplementation` address is public, `Clones.clone` is a standard library call, and `initialize()` is unrestricted. An attacker who wants to harm stakers of a specific agreement can create a rogue pool for that agreement (the `isAgreementValid` check in `initialize()` only requires a valid registry agreement, not factory authorization), advertise it as the official pool, and wait for deposits. Unsophisticated stakers who do not cross-reference `getPoolsByAgreement()` are the realistic victim population.

### Recommendation
Restrict `initialize()` so it can only be called by the factory. The cleanest approach is to pass the factory address as a parameter, store it, and add an `onlyFactory` modifier:

```solidity
address public factory;

modifier onlyFactory() {
    if (msg.sender != factory) revert NotFactory();
    _;
}

function initialize(
    address factory_,
    address agreement_,
    address stakeToken_,
    ...
) external initializer {
    factory = factory_;
    // existing zero-address checks ...
    if (!IConfidencePoolFactory(factory_).allowedStakeToken(stakeToken_)) {
        revert StakeTokenNotAllowed();
    }
    ...
}
```

The factory must pass `address(this)` as `factory_` when calling `initialize()`. This closes the bypass and makes the allowlist check load-bearing inside the pool itself, mirroring the intent of the factory guard.

### Proof of Concept
```solidity
// 1. Read the public implementation address from the factory.
address impl = factory.poolImplementation();

// 2. Deploy a clone without going through the factory.
address roguePool = Clones.clone(impl);

// 3. Initialize with a non-allowlisted negative-rebasing token.
//    `legitimateAgreement` passes isAgreementValid; no allowlist check fires.
IConfidencePool(roguePool).initialize(
    legitimateAgreement,
    rebasingToken,          // NOT in allowedStakeToken
    address(safeHarborRegistry),
    outcomeModerator,
    block.timestamp + 31 days,
    1e18,
    attackerAddress,        // attacker controls recoveryAddress
    attackerAddress,        // attacker is pool owner
    accounts
);

// 4. Stakers directed to roguePool deposit; eligibleStake is recorded.
// 5. Rebasing token deflates; stakeToken.balanceOf(roguePool) < totalEligibleStake.
// 6. claimSurvived() / withdraw() revert on safeTransfer → funds permanently frozen.
```
