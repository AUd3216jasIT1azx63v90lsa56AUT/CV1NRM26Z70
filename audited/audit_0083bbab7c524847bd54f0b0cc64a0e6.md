### Title
Callback Payer Confusion Allows Arbitrary Third-Party Drain in Liquidation - (File: src/Midnight.sol)

### Summary
The `liquidate` function accepts a caller-controlled `callback` address and unconditionally sets `payer = callback` at line 679 without verifying that `callback == msg.sender`. Because the loan token pull at line 717 draws from `payer`, any unprivileged liquidator can designate a third-party contract as `callback`, causing Midnight to drain that contract's loan tokens to fund the liquidation while the liquidator's `receiver` collects the seized collateral.

### Finding Description
**Code path:**

`liquidate` (src/Midnight.sol, lines 581–719):

- Line 589: `address callback` is a free, unvalidated caller input. [1](#0-0) 
- Line 679: `address payer = callback != address(0) ? callback : msg.sender;` — payer is set to the attacker-supplied callback with no check that `callback == msg.sender`. [2](#0-1) 
- Line 696: Seized collateral is transferred to `receiver` **before** any payment is collected. [3](#0-2) 
- Lines 698–715: `ILiquidateCallback(callback).onLiquidate(...)` is called by Midnight, so `msg.sender` inside the callback is Midnight — a check the victim contract may pass. [4](#0-3) 
- Line 717: `safeTransferFrom(market.loanToken, payer, address(this), repaidUnits)` pulls loan tokens from the victim. [5](#0-4) 

**Root cause:** There is no `require(callback == msg.sender || callback == address(0))` guard. The `callback` parameter is purely attacker-controlled and is used directly as the token source.

**Attacker inputs:**
- `callback` = address of victim contract (any contract that has approved Midnight and whose `onLiquidate` returns `CALLBACK_SUCCESS`)
- `receiver` = attacker-controlled address to receive seized collateral
- `borrower` = any legitimately liquidatable position

**Why existing checks fail:**
- The `liquidatorGate` check (line 598) only gates whether `msg.sender` may liquidate at all; it does not constrain `callback`. [6](#0-5) 
- The `WrongLiquidateCallbackReturnValue` check (line 713) only verifies the return value equals `CALLBACK_SUCCESS`; it does not verify the callback is the initiator. [7](#0-6) 
- `ILiquidateCallback.onLiquidate` receives `caller` (the liquidator's address) but the interface imposes no obligation on the victim to reject calls it did not initiate. [8](#0-7) 

### Impact Explanation
A victim contract that (a) holds or has approved Midnight for loan tokens and (b) implements `onLiquidate` returning `CALLBACK_SUCCESS` without verifying it initiated the liquidation loses `repaidUnits` of loan token. The attacker's receiver gains `seizedAssets` of collateral while contributing zero loan tokens. This is a direct, concrete fund loss from the victim to the attacker.

### Likelihood Explanation
The precondition — victim implements `ILiquidateCallback` returning `CALLBACK_SUCCESS` and has approved Midnight — is restrictive for EOAs but realistic for: liquidation helper/aggregator contracts, flash-loan wrapper contracts, or any protocol-level contract that integrates with Midnight's liquidation flow and holds a standing approval. Such contracts are common in DeFi composability. The attack is permissionless, repeatable, and requires no privileged access.

### Recommendation
Enforce that the callback must be the caller:

```solidity
require(callback == address(0) || callback == msg.sender, InvalidCallback());
```

Add this check immediately after the `callback` parameter is received (before line 679). This ensures only the liquidator itself can be designated as payer via callback, eliminating the ability to redirect payment to a third party.

// Victim: a naive liquidation helper that approves Midnight and blindly returns CALLBACK_SUCCESS
contract VictimCallback is ILiquidateCallback {
    address public midnight;
    address public loanToken;
    constructor(address _midnight, address _loanToken) {
        midnight = _midnight;
        loanToken = _loanToken;
        IERC20(_loanToken).approve(_midnight, type(uint256).max);
    }
    function onLiquidate(address, bytes32, Market memory, uint256, uint256, uint256,
        address, address, bytes memory, uint256) external returns (bytes32) {
        return CALLBACK_SUCCESS; // no caller check
    }
}

contract CallbackPayerConfusionTest is Test {
    function test_liquidatorDrainsVictimCallback() public {
        // Setup: fund VictimCallback with loanToken, create liquidatable position
        // ...
        uint256 victimBalanceBefore = loanToken.balanceOf(address(victim));
        uint256 attackerCollateralBefore = collateralToken.balanceOf(attacker);

        vm.prank(attacker); // attacker has zero loanToken
        midnight.liquidate(market, collateralIndex, seizedAssets, 0,
            borrower, false, attacker, address(victim), "");

        // Assertions:
        assertLt(loanToken.balanceOf(address(victim)), victimBalanceBefore,
            "victim lost loan tokens");
        assertGt(collateralToken.balanceOf(attacker), attackerCollateralBefore,
            "attacker gained collateral");
        assertEq(loanToken.balanceOf(attacker), 0,
            "attacker spent zero loan tokens");
    }
}
