Audit Report

## Title
Zero-asset rounding bypass allows unbounded position mutation on fully-consumed assets-based buy offers — (File: src/Midnight.sol)

## Summary
When a buy offer uses `maxAssets`-based consumption and `offerPrice < WAD`, calling `take()` with `units=1` produces `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. The `ConsumedAssets` guard advances by zero and passes unconditionally, while `buyerPos.credit`, `sellerPos.debt`, and `totalUnits` mutate as if a real trade occurred — with zero tokens transferred. This is repeatable without bound on any exhausted qualifying offer, enabling free credit inflation for the maker and costless debt imposition on any taker.

## Finding Description

**Root cause** — `src/Midnight.sol`, `take()`:

```solidity
// Line 363
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;
// units=1, buyerPrice < WAD → mulDivDown(1, buyerPrice, WAD) = 0

// Lines 367–369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += buyerAssets; // += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());       // trivially passes
}
``` [1](#0-0) 

The `ConsumedAssets` check is the sole guard preventing execution on an exhausted offer. Because `buyerAssets` rounds to zero, `newConsumed` does not advance, and the require is satisfied regardless of how many times the call is repeated.

**Execution continues unconditionally after the guard:**

```solidity
// Lines 382–384
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt); // = 1 if no debt
uint256 sellerCreditDecrease = UtilsLib.min(units, sellerPos.credit);       // = 0 if no credit
uint256 sellerDebtIncrease = units - sellerCreditDecrease;                  // = 1

// Lines 408–414
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);  // +1
sellerPos.debt  += UtilsLib.toUint128(sellerDebtIncrease);   // +1

// Lines 416–417
_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease; // +1

// Line 455: safeTransferFrom(..., 0) — no revert
``` [2](#0-1) [3](#0-2) [4](#0-3) 

**Developer acknowledgment of the behavior:**

Line 94 states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* This confirms awareness of the rounding path. The presence of a test named `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` — with "Bug" explicitly in the name — further confirms this is treated as a defect, not an accepted design choice. [5](#0-4) 

**Existing checks reviewed and shown insufficient:**

- `ConsumedAssets` (line 369): bypassed because `buyerAssets = 0`.
- `CannotIncreaseDebtPostMaturity` (line 391): only blocks post-maturity; pre-maturity debt increase is unrestricted.
- `SellerIsLiquidatable` (line 476): bounds total iterations per seller by collateral value, but does not prevent the attack — only limits its per-seller scale.
- `SelfTake` (line 354): prevents the maker from self-exploiting, but does not prevent a third-party taker (e.g., an automated router) from being victimized. [6](#0-5) [7](#0-6) [8](#0-7) 

## Impact Explanation

1. **Free credit for the maker (buyer)**: Each call increments `buyerPos.credit` by 1 at zero token cost. Credit is a claim on loan tokens at settlement; inflating it allows the maker to redeem more assets than were ever deposited, constituting direct theft from the protocol's asset pool at the expense of other lenders.
2. **Debt imposed on the taker (seller) without asset receipt**: `sellerPos.debt` increases by 1 per call. An automated router that does not verify consumed state before calling `take()` accumulates debt with no corresponding loan token receipt, and must repay at settlement.
3. **`totalUnits` diverges from real asset flows**: Protocol-wide accounting invariants break, undermining the formal verification guarantees claimed in the Certora README.
4. **Uncontrolled callback invocation**: The maker's `onBuy` callback fires with `buyerAssets=0, units=1` on every call, which can corrupt callback-side state that assumes it is only triggered by real asset flows.

## Likelihood Explanation

Required preconditions are all reachable by an unprivileged external user:
- `offer.buy = true`, `offer.maxAssets > 0` — standard buy offer configuration.
- `offerPrice < WAD` — any tick below the WAD-price tick.
- `consumed[maker][group] == maxAssets` — reachable after a normal full fill, or self-set by the maker via `setConsumed` on their own group.

The attack requires zero capital (zero tokens transferred), is repeatable without bound on every qualifying exhausted offer (up to the seller's health limit per seller), and leaves no on-chain trace distinguishing it from a legitimate take. The maker can construct the offer and wait for any automated taker to interact.

## Recommendation

Add a check that rejects takes where the computed asset amount rounds to zero when `maxAssets > 0`:

```solidity
if (offer.maxAssets > 0) {
    require(buyerAssets > 0, ZeroAssets()); // add this guard
    newConsumed = consumed[offer.maker][offer.group] += buyerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` value such that `mulDivDown(units, buyerPrice, WAD) >= 1` before proceeding with state mutations. This ensures that any take which mutates credit/debt/totalUnits also transfers at least one token unit.

## Proof of Concept

The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` directly demonstrates this:

1. Create a buy offer with `maxAssets > 0` and `offerPrice < WAD` (e.g., tick = `MAX_TICK - 16`).
2. Fully consume the offer via a normal take until `consumed[maker][group] == maxAssets`.
3. Call `take()` with `units=1` from a third-party taker.
4. Assert post-take: `creditOf(maker) > before`, `debtOf(taker) > before`, `totalUnits > before`, while `consumed` and token balances are unchanged. [9](#0-8)
