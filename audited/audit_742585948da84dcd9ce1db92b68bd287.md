### Title
Incorrect `_clampUserSums` over-clamps post-risk deposits, overestimating bonus shares and causing DoS on `claimSurvived` / `claimExpired` — (File: src/ConfidencePool.sol)

---

### Summary

`_clampUserSums` is designed to floor pre-risk deposit entry times to `riskWindowStart`. However, when a user has staked both **before** and **after** the risk window opens, the function incorrectly clamps the user's **entire** stake — including post-risk deposits — to `riskWindowStart`. This inflates the user's `userScore` in `_bonusShare` beyond what the global snapshot accounts for, causing the sum of all claimants' bonus shares to exceed `snapshotTotalBonus`. The last claimer's `safeTransfer` then reverts because the contract balance is insufficient, permanently locking their principal.

---

### Finding Description

When the risk window opens, `_markRiskWindowStart` eagerly resets the global accumulators to `totalEligibleStake × riskWindowStart`. Per-user sums are updated lazily via `_clampUserSums`, which is called at the start of every claim. [1](#0-0) 

The clamp condition is:

```solidity
if (userSumStakeTime[u] < stake_ * start) {
    userSumStakeTime[u] = stake_ * start;
    userSumStakeTimeSq[u] = stake_ * start * start;
}
```

This replaces the user's sums with `eligibleStake[u] × riskWindowStart` and `eligibleStake[u] × riskWindowStart²`. The problem arises when the user has staked **after** the risk window opened. Their post-risk deposit was recorded at its actual wall-clock entry time `t2 > riskWindowStart` in both the per-user sums and the global snapshot sums. After clamping, the per-user sums treat that post-risk deposit as if it entered at `riskWindowStart`, but the global snapshot (`snapshotSumStakeTime`, `snapshotSumStakeTimeSq`) correctly retains `t2`. [2](#0-1) 

This divergence means:

- **`userScore_A`** (computed from clamped per-user sums) = `(A1 + A2) × (T − t_r)²`
- **Correct `userScore_A`** = `A1 × (T − t_r)² + A2 × (T − t2)²`

Since `t2 > t_r`, `(T − t_r)² > (T − t2)²`, so `userScore_A` is **overestimated**. The sum of all users' scores then exceeds `globalScore`, and consequently:

```
Σ bonusShare_i  =  Σ mulDiv(userScore_i, snapshotTotalBonus, globalScore)
                >  snapshotTotalBonus
``` [3](#0-2) 

The first claimer (the user with the inflated score) receives more bonus than the pool holds for them. After their payout, the remaining balance is insufficient to cover the next claimer's `payout = userEligible + bonusShare`, causing `safeTransfer` to revert. [4](#0-3) 

The stuck principal cannot be recovered: `sweepUnclaimedBonus` reserves exactly `totalEligibleStake + (snapshotTotalBonus − claimedBonus)`, which equals the remaining balance, so `amount == 0` and it reverts with `NothingToSweep`. [5](#0-4) 

---

### Impact Explanation

**Impact: Medium–High.** Any staker who has not yet claimed when the overestimated bonus is paid out cannot claim their principal or bonus. Their funds are permanently locked in the contract with no recovery path. The DESIGN.md documents no accepted trade-off for this scenario. [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: Medium.** The trigger condition — a user staking both before and after the risk window — is a natural usage pattern. Staking during `UNDER_ATTACK` is explicitly allowed by design (§3 of DESIGN.md), so the two-deposit scenario is expected. An adversary can also deliberately exploit it: stake a large amount before the risk window, stake a dust amount after it opens, claim first to extract the inflated bonus, and leave all other claimants unable to withdraw. [7](#0-6) 

---

### Recommendation

Track pre-risk and post-risk stake separately per user, or modify `_clampUserSums` to clamp only the pre-risk portion. One approach: store a `userPreRiskStake[u]` counter incremented only for deposits made before `riskWindowStart`. At clamp time, recompute:

```solidity
uint256 preRisk = userPreRiskStake[u];
uint256 postRisk = eligibleStake[u] - preRisk;
// Clamp only the pre-risk portion; leave post-risk sums untouched.
userSumStakeTime[u]   = preRisk * start + (userSumStakeTime[u] - userPreRiskSumStakeTime[u]);
userSumStakeTimeSq[u] = preRisk * start * start + (userSumStakeTimeSq[u] - userPreRiskSumStakeTimeSq[u]);
```

This keeps the per-user sums consistent with the global snapshot, ensuring `Σ userScore_i == globalScore` and `Σ bonusShare_i ≤ snapshotTotalBonus`.

### Proof of Concept

**Parameters:**
- User A stakes `A1 = 1000` tokens at `t1 = 100` (before risk window)
- Risk window opens at `t_r = 200`
- User A stakes `A2 = 1` token at `t2 = 300` (after risk window)
- User B stakes `B = 100` tokens at `t3 = 400` (after risk window)
- `T = outcomeFlaggedAt = 500`, `snapshotTotalBonus = 1,000,000`

**Global snapshot sums** (built correctly):
- `snapshotSumStakeTime = 1000×200 + 1×300 + 100×400 = 240,300`
- `snapshotSumStakeTimeSq = 1000×200² + 1×300² + 100×400² = 56,090,000`
- `globalScore = 500²×1101 + 56,090,000 − 2×500×240,300 = 91,040,000`

**`_clampUserSums(A)` at claim time:**
- `userSumStakeTime[A] = 100,300 < 1001×200 = 200,200` → clamp fires
- After clamp: `userSumStakeTime[A] = 200,200`, `userSumStakeTimeSq[A] = 40,040,000`
- `userScore_A = 500²×1001 + 40,040,000 − 2×500×200,200 = 90,090,000`

**User B's score** (no clamp, t3 > t_r):
- `userScore_B = 500²×100 + 16,000,000 − 2×500×40,000 = 1,000,000`

**Invariant violation:** `userScore_A + userScore_B = 91,090,000 > globalScore = 91,040,000`

**Bonus shares:**
- `bonusShare_A = ⌊90,090,000 × 1,000,000 / 91,040,000⌋ = 989,558`
- `bonusShare_B = ⌊1,000,000 × 1,000,000 / 91,040,000⌋ = 10,984`
- `bonusShare_A + bonusShare_B = 1,000,542 > 1,000,000`

**DoS:**
- Initial contract balance: `1101 + 1,000,000 = 1,001,101`
- User A claims `1001 + 989,558 = 990,559` → remaining balance: `10,542`
- User B attempts to claim `100 + 10,984 = 11,084` → **`safeTransfer` reverts** (balance `10,542 < 11,084`)
- User B's 100-token principal is permanently locked. [1](#0-0) [8](#0-7)

```text
    function _bonusShare(address u, uint256 userEligible) internal view returns (uint256) {
        if (snapshotTotalBonus == 0) return 0;
        // No observable risk → no bonus (see contract natspec).
        if (riskWindowStart == 0) return 0;
        uint256 T = outcomeFlaggedAt;

        // Underflow guards on both subtractions: globally the sum of squares is nonneg, but
        // truncation/rounding pathologies could push individual terms over.
        uint256 userPlus = T * T * userEligible + userSumStakeTimeSq[u];
        uint256 userMinus = 2 * T * userSumStakeTime[u];
        uint256 userScore = userPlus > userMinus ? userPlus - userMinus : 0;

        uint256 plus = T * T * snapshotTotalStaked + snapshotSumStakeTimeSq;
        uint256 minus = 2 * T * snapshotSumStakeTime;
        uint256 globalScore = plus > minus ? plus - minus : 0;

        if (globalScore == 0) {
            // No time elapsed in the risk window for anyone → fallback to amount-weighted.
            if (snapshotTotalStaked == 0) return 0;
            return Math.mulDiv(userEligible, snapshotTotalBonus, snapshotTotalStaked);
        }
        // mulDiv handles the final multiply-then-divide via 512-bit intermediates, so a very
        // large `snapshotTotalBonus` cannot push the numerator over uint256 before division.
        return Math.mulDiv(userScore, snapshotTotalBonus, globalScore);
    }
```

**File:** docs/DESIGN.md (L52-71)
```markdown

`_assertDepositsAllowed` blocks `PROMOTION_REQUESTED`, `PRODUCTION`, and `CORRUPTED`, but **not**
`UNDER_ATTACK`. This asymmetry is about deposit *timing*, not relative safety (both active-risk
states are attackable and can still go `CORRUPTED`):

- **`UNDER_ATTACK`:** a depositor takes on real, visible, voluntary risk. Their bonus share is
  crushed to ~zero by the k=2 weighting — entry is floored at `riskWindowStart ≈ now`, so
  `(T − entry)² ≈ 0` — and the deposit self-locks into the resolution path (`withdraw` is
  disabled once `riskWindowStart != 0`). There is no late-join *advantage* to gate against and
  no trap: it is voluntary, near-zero-reward risk capital. A staker who does not want this can
  read the live registry state before staking.
- **`PROMOTION_REQUESTED`:** the agreement has requested to exit toward `PRODUCTION`, so the risk
  window is about to close (`riskWindowEnd` imminent). Blocking deposits here stops a late join
  in the final stretch — entering with almost no remaining risk-bearing time before resolution.
  The state is still attackable; it is the *closing window*, not assured survival, that justifies
  the block.

**Why this is not a "trap" finding:** the withdraw-lock from `UNDER_ATTACK` onward is the *same*
intended commitment every staker accepts once risk materializes, and the late entrant earns
negligible bonus. Principal remains recoverable through resolution.
```

**File:** docs/DESIGN.md (L167-178)
```markdown
The bonus pool is split with a k=2 time-weighted formula: each deposit contributes
`amount × (T − entryTime)²` to its score, summed per-deposit (top-ups are not blended into a
single weighted-average entry time — that would lose the Jensen gap and let users farm share by
splitting across addresses). Squaring crushes late entrants. Per-user and global sums maintain
the score in O(1) per claim via `T²·Σa − 2T·Σ(a·t) + Σ(a·t²)`.

`T` is `riskWindowEnd` for SURVIVED resolutions or `expiry` for EXPIRED. Effective entry time is
floored at `riskWindowStart`: pre-risk capital earns no time weight. So "squaring crushes late
entrants" applies only *within* the observed risk window — every deposit made before the seal
(including the one that triggers it) floors to `riskWindowStart` and is amount-weighted among that
cohort. Intentional: pre-risk calendar time is not at-risk time, so equal per-token weight there is
correct, not a leak.
```
