### Title
Authorized Operator Can Create Persistent Sub-Authorizations That Survive Revocation - (File: src/Midnight.sol)

### Summary
`Midnight.setIsAuthorized` permits any currently-authorized address to grant new authorizations on behalf of the user with no restriction on sub-delegation. When a victim later revokes the original operator, any secondary addresses that operator added remain authorized indefinitely. The victim has no on-chain enumeration mechanism to discover or bulk-revoke these hidden sub-authorizations.

### Finding Description
The access check in `setIsAuthorized` is:

```solidity
// src/Midnight.sol L731-734
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
``` [1](#0-0) 

The guard only verifies that `msg.sender` is currently authorized for `onBehalf`; it places no restriction on what the authorized caller may do next. Concretely:

1. **Precondition**: victim calls `setIsAuthorized(attacker, true, victim)` → `isAuthorized[victim][attacker] = true`.
2. **Attacker call**: attacker calls `setIsAuthorized(attacker2, true, victim)`. The check `isAuthorized[victim][attacker]` is `true`, so it passes. Result: `isAuthorized[victim][attacker2] = true`.
3. **Revocation**: victim calls `setIsAuthorized(attacker, false, victim)` → `isAuthorized[victim][attacker] = false`.
4. **Bad state**: `isAuthorized[victim][attacker2]` remains `true`. attacker2 can now call `withdraw`, `withdrawCollateral`, `repay`, `setConsumed`, or `setIsAuthorized` on behalf of victim indefinitely.

The existing test `testSetIsAuthorizedAuthorization` in `test/AuthorizationTest.sol` (lines 290–304) confirms step 2 is reachable and succeeds, but it never tests the revocation scenario. [2](#0-1) 

The Certora rule `onlyAuthorizedCanChangeIsAuthorized` only asserts that the caller was authorized at call time; it does not assert anything about the state after a subsequent revocation, so it does not catch this scenario. [3](#0-2) 

### Impact Explanation
Any address that victim ever authorizes can silently plant one or more additional authorized addresses. After the victim revokes the original operator, the planted addresses retain full operator-level access: they can withdraw victim's credit units, withdraw victim's collateral, cancel victim's offers via `setConsumed`, and recursively add yet more authorized addresses. The victim has no enumeration API to discover these hidden entries and no atomic "revoke all" primitive.

### Likelihood Explanation
The precondition is that the victim must have authorized the attacker at some point (e.g., a DeFi aggregator, a bundle executor, or a compromised key). This is a normal, expected usage pattern. The attacker needs only one transaction while authorized to plant attacker2. The attack is repeatable and the planted authorization is permanent until the victim explicitly revokes attacker2 by address — which they cannot do if they are unaware of it.

### Recommendation
Restrict `setIsAuthorized` so that only the account itself (`onBehalf == msg.sender`) may grant new authorizations. Authorized operators should be permitted to act on behalf of the user for protocol operations (withdraw, repay, etc.) but not to modify the authorization mapping itself:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender, Unauthorized()); // only self can grant/revoke
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
}
```

If sub-delegation is intentionally desired, it must be explicitly scoped and documented, and a revocation of a parent operator must cascade to all sub-authorizations it created.

### Proof of Concept

```solidity
// Foundry unit test — add to test/AuthorizationTest.sol
function testPersistentSubAuthorizationAfterRevocation() public {
    address victim   = makeAddr("victim");
    address attacker = makeAddr("attacker");
    address attacker2 = makeAddr("attacker2");

    // Step 1: victim authorizes attacker (normal usage)
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, true, victim);
    assertEq(midnight.isAuthorized(victim, attacker), true);

    // Step 2: attacker plants a secondary authorization for attacker2
    vm.prank(attacker);
    midnight.setIsAuthorized(attacker2, true, victim);
    assertEq(midnight.isAuthorized(victim, attacker2), true);

    // Step 3: victim revokes attacker, believing access is fully removed
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, false, victim);
    assertEq(midnight.isAuthorized(victim, attacker), false);

    // Step 4: attacker2 still authorized — invariant violated
    // Expected (secure): false  |  Actual (buggy): true
    assertEq(midnight.isAuthorized(victim, attacker2), false); // FAILS
}
```

Expected assertion failure: `midnight.isAuthorized(victim, attacker2)` returns `true` instead of `false`, proving the persistent sub-authorization survives revocation of the primary operator.
