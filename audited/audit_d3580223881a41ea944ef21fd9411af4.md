### Title
Bad Debt Over-Realization via `supplyCollateral` Reentrancy in `onLiquidate` Callback - (File: src/Midnight.sol)

### Summary

`liquidate()` computes `badDebt` from a local snapshot of `_position.collateralBitmap` and immediately applies it to `_position.debt` and `_marketState.lossFactor` before invoking the `onLiquidate` callback. Because there is no reentrancy guard on `liquidate()`, an authorized callback can call `supplyCollateral` for the borrower during the callback, setting a new bitmap bit and adding collateral that was never counted in the already-finalized `badDebt`. Lenders are slashed by a `badDebt` figure that is larger than the actual undercollateralization at callback-completion time.

### Finding Description

**Code path:**

`liquidate()` in `src/Midnight.sol`:

1. **Line 606** — local bitmap snapshot: `uint128 _collateralBitmap = _position.collateralBitmap;`
2. **Lines 607–618** — loop over local snapshot, computing `maxDebt` and `badDebt` from live storage collateral amounts but only for bits present in the snapshot.
3. **Lines 626–641** — `badDebt` is committed to storage: `_position.debt -= uint128(badDebt)` and `_marketState.lossFactor` is updated (lenders slashed). This is irreversible within the call.
4. **Lines 643–677** — collateral seized, debt reduced by `repaidUnits`.
5. **Lines 698–715** — `ILiquidateCallback(callback).onLiquidate(...)` is called. **No reentrancy lock is set before this call.** The `liquidationLocked` / `tExchange` mechanism exists only in `take()` (line 444), not in `liquidate()`.
6. **Line 717** — loan tokens pulled from payer.

**Root cause:** `badDebt` is finalized and written to storage (step 3) before the external callback (step 5). The callback can call `supplyCollateral(market, newIndex, largeAmount, borrower)` (lines 524–546), which:
- Passes the authorization check `isAuthorized[borrower][callbackContract]`
- Sets a new bit in `_position.collateralBitmap` (storage)
- Increases `_position.collateral[newIndex]`

This new collateral was not present in the local `_collateralBitmap` snapshot and therefore was not subtracted from `badDebt`. The already-applied `badDebt` is larger than the true undercollateralization.

**Attacker inputs:**
- `callback` = attacker-controlled contract implementing `ILiquidateCallback`
- `borrower` = account for which `isAuthorized[borrower][callbackContract] = true`
- `seizedAssets = 0`, `repaidUnits = 0` (pure bad-debt realization, explicitly supported per line 577 comment)
- `newIndex` = any collateral index with zero prior balance (so `oldCollateral == 0` triggers `setBit`)

**Exploit flow:**
1. Borrower (= attacker or attacker-controlled) has debt D, collateral C₀ at index 0 only. Position has bad debt: `C₀ * price / ORACLE_PRICE_SCALE * WAD / maxLif < D`.
2. Borrower calls `setIsAuthorized(callbackContract, true, borrower)`.
3. Attacker calls `liquidate(market, 0, 0, 0, borrower, false, receiver, callbackContract, data)`.
4. `badDebt = D - C₀ * price / ORACLE_PRICE_SCALE * WAD / maxLif > 0` computed from snapshot.
5. `_position.debt -= badDebt`; `_marketState.lossFactor` updated — lenders slashed.
6. `onLiquidate` callback fires. Callback calls `supplyCollateral(market, 1, largeAmount, borrower)`. Bit 1 set in `_position.collateralBitmap`; `_position.collateral[1] = largeAmount`.
7. Callback returns `CALLBACK_SUCCESS`.
8. Zero loan tokens pulled (repaidUnits = 0).
9. Attacker later calls `withdrawCollateral(market, 1, largeAmount, borrower, attacker)` — position is now healthy because debt was reduced by the inflated `badDebt`.

**Why checks fail:** There is no reentrancy guard on `liquidate()`. `supplyCollateral` only checks authorization. The `liquidationLocked` transient-storage mechanism is set exclusively inside `take()` (line 444) and is never set during `liquidate()`.

### Impact Explanation

Lenders are slashed via `_marketState.lossFactor` by a `badDebt` value that exceeds the true undercollateralization. The new collateral supplied during the callback would have reduced (or eliminated) `badDebt` had it been present at computation time. The attacker recovers the supplied collateral tokens by withdrawing them after the call, so the net cost to the attacker is zero (excluding gas). Lenders suffer a permanent, unwarranted credit loss proportional to the difference between the inflated `badDebt` and the true bad debt.

### Likelihood Explanation

**Preconditions:**
1. `isAuthorized[borrower][callbackContract] = true` — requires borrower cooperation or the borrower IS the liquidator (self-liquidation). Self-liquidation is explicitly permissionless in the protocol.
2. The market must have at least one collateral index with zero prior balance for the borrower (to trigger `setBit`). Any multi-collateral market satisfies this.
3. The position must have genuine bad debt (collateral value < debt at `maxLif` prices). This is the normal trigger for bad-debt realization.
4. The liquidator must pass `liquidatorGate` (or no gate exists).

All preconditions are achievable by a single actor who is both borrower and liquidator. The attack is repeatable across any bad-debt liquidation in any multi-collateral market. No oracle manipulation or impossible values are required.

### Recommendation

Set a transient reentrancy lock at the start of `liquidate()` (analogous to the `tExchange` pattern already used in `take()`) and clear it after the callback and token transfer. Alternatively, move the `onLiquidate` callback invocation to before the bad-debt accounting block, or re-read `_position.collateralBitmap` from storage after the callback and recompute/verify `badDebt` has not been understated. The simplest fix is to add a per-borrower liquidation lock at the entry of `liquidate()` that blocks `supplyCollateral` (or any state-mutating reentrant call) for the borrower's position during the liquidation execution.
