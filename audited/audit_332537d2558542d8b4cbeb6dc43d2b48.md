### Title
Authorized repay callback activates reverting-oracle collateral slot, permanently blocking liquidation of unhealthy positions - (File: src/Midnight.sol / src/periphery/EcrecoverAuthorizer.sol)

### Summary
`Midnight.repay` fires an `IRepayCallback` with no reentrancy guard and no snapshot of the borrower's collateral bitmap. An attacker who holds a valid signed `Authorization` from the borrower can use `EcrecoverAuthorizer.setIsAuthorized` to become authorized, then call `repay` with a callback that calls `supplyCollateral` to activate a previously-unused collateral slot whose oracle always reverts. Because `liquidate` unconditionally calls `IOracle.price()` for every bit set in the borrower's collateral bitmap, all subsequent liquidation attempts on that position revert permanently, violating the invariant that unhealthy positions must remain liquidatable.

### Finding Description

**Code path**

1. `EcrecoverAuthorizer.setIsAuthorized` (`src/periphery/EcrecoverAuthorizer.sol:24-48`):
   - Verifies deadline, nonce, and that `ecrecover` returns the borrower (or someone the borrower has authorized).
   - Calls `IMidnight(MIDNIGHT).setIsAuthorized(attackerContract, true, borrower)`.
   - In `Midnight.setIsAuthorized` (`src/Midnight.sol:731-735`): requires `isAuthorized[borrower][EcrecoverAuthorizer] == true` (the borrower must have pre-authorized the periphery contract, which is the standard gasless-authorization setup). Sets `isAuthorized[borrower][attackerContract] = true`.

2. `Midnight.repay` (`src/Midnight.sol:502-521`):
   - Line 505: `require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender])` — passes because `isAuthorized[borrower][attackerContract] = true`.
   - Line 508: decrements `position[id][borrower].debt` by 1.
   - Line 516: calls `IRepayCallback(attackerContract).onRepay(...)` — **no reentrancy guard, no bitmap snapshot**.

3. Inside `onRepay`, the attacker contract calls `Midnight.supplyCollateral(market, revertingOracleIndex, 1, borrower)` (`src/Midnight.sol:524-546`):
   - Line 527: `require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender])` — `msg.sender` is `attackerContract`, `isAuthorized[borrower][attackerContract] = true` → passes.
   - Lines 535-540: `oldCollateral == 0 && assets > 0` → sets bit `revertingOracleIndex` in `position[id][borrower].collateralBitmap`. Only check is `countBits <= MAX_COLLATERALS_PER_BORROWER`. **No oracle liveness check.**
   - Line 545: pulls 1 wei of collateral token from the attacker contract.

4. `Midnight.liquidate` (`src/Midnight.sol:607-618`):
   ```solidity
   while (_collateralBitmap != 0) {
       uint256 i = UtilsLib.msb(_collateralBitmap);
       CollateralParams memory _collateralParam = market.collateralParams[i];
       uint256 price = IOracle(_collateralParam.oracle).price(); // ← REVERTS for poisoned slot
       ...
   }
   ```
   Every future call to `liquidate` for this borrower iterates the full bitmap, hits the reverting oracle, and reverts. The protocol's own NatDev at lines 34-36 confirms: *"Liquidation reverts if any of the activated collaterals' oracle reverts"* and *"a borrower can activate a collateral once its oracle is reverting because the oracle is not called in supplyCollateral."*

**Root cause**: `supplyCollateral` relies solely on the authorization check to prevent collateral poisoning, but an authorized third party (obtained via a signed `Authorization`) can call it from inside a `repay` callback with no additional guard. The `repay` function neither snapshots the collateral bitmap before the callback nor validates it afterward.

**Why existing checks fail**: The authorization check at `supplyCollateral:527` is the only protection against collateral poisoning. Once the attacker is authorized (via `EcrecoverAuthorizer`), this check is satisfied. There is no reentrancy lock, no oracle liveness probe in `supplyCollateral`, and no post-callback bitmap invariant check in `repay`.

### Impact Explanation
An unhealthy borrower position becomes permanently unliquidatable. Lenders cannot recover their funds through liquidation. The core invariant — *unhealthy positions remain liquidatable* — is broken for the duration of the oracle outage (which may be permanent if the oracle contract is deprecated or bricked). Bad debt accrues without recourse.

### Likelihood Explanation
**Preconditions:**
1. Borrower has authorized `EcrecoverAuthorizer` on Midnight — standard for any user relying on gasless/meta-transaction authorization flows.
2. Borrower has signed an `Authorization` authorizing the attacker contract — could have been signed before the position became unhealthy (e.g., to allow a keeper to repay debt).
3. Market has at least one collateral param whose oracle is currently reverting — realistic for long-lived markets where an oracle is deprecated, paused, or suffers a liveness failure.
4. Attacker holds 1 wei of the collateral token and 1 unit of the loan token — trivially satisfiable.

The attack is repeatable: if the poisoned collateral is somehow removed (e.g., the oracle recovers and the attacker withdraws), the attacker can re-execute. The attacker loses only 1 wei of collateral (locked in the contract) and 1 unit of loan token (repaid to the market).

### Recommendation
**Option A (preferred):** In `repay`, snapshot `position[id][onBehalf].collateralBitmap` before the callback and assert it is unchanged after:
```solidity
uint128 bitmapBefore = position[id][onBehalf].collateralBitmap;
// ... callback ...
require(position[id][onBehalf].collateralBitmap == bitmapBefore, CollateralPoisoningDetected());
```

**Option B:** In `supplyCollateral`, when activating a new slot (`oldCollateral == 0 && assets > 0`), call `IOracle(market.collateralParams[collateralIndex].oracle).price()` and revert if it reverts, preventing activation of a broken oracle slot.

**Option C:** Add a reentrancy guard scoped to state-mutating functions so that `supplyCollateral` cannot be called from within a `repay` callback.
