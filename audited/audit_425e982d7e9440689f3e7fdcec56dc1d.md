### Title
Unconditional zero-value `safeTransferFrom` calls in `take()` DoS dust trades on tokens that revert on zero transfers - (`src/Midnight.sol`)

### Summary
`take()` makes two unconditional `safeTransferFrom` calls — one for `sellerAssets` and one for the fee spread `buyerAssets - sellerAssets` — with no zero-value guard. Both values can legitimately round to zero for small `units` or low-tick offers. Any ERC20 `loanToken` that reverts on zero-value transfers will cause `take()` to revert for these dust-sized inputs, DoS-ing valid offers.

### Finding Description
In `src/Midnight.sol` lines 363–364, `buyerAssets` and `sellerAssets` are computed via `mulDivDown`/`mulDivUp`:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD);
```

Both can be zero. For a buy offer with `tick=0` (`sellerPrice=0`), `sellerAssets = units.mulDivDown(0, WAD) = 0` for any `units`. For the fee spread, when `units * _settlementFee < WAD`, `buyerAssets - sellerAssets = 0`. The existing test suite explicitly confirms both paths: `testPriceZeroNoSettlementFeeSell` asserts `buyerAssets=0, sellerAssets=0` succeeds; `testPriceZeroWithSettlementFeeSell` asserts `sellerAssets=0` succeeds — both with a standard test token. [1](#0-0) 

At lines 455–456, both values are passed unconditionally to `safeTransferFrom`:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` has no zero-value guard — it unconditionally calls `token.transferFrom(from, to, value)` and propagates any revert: [3](#0-2) 

If the `loanToken` reverts on zero-value `transferFrom` calls (a known behavior of some real ERC20 tokens), both call sites become revert paths for dust inputs. No existing check in `take()` prevents `units` from being chosen such that either computed value is zero. [4](#0-3) 

### Impact Explanation
Any taker attempting to fill a valid offer with `units` small enough that `sellerAssets=0` or `buyerAssets-sellerAssets=0` will have their `take()` call revert. The offer remains valid and unconsumed (state changes are reverted), but the specific `(tick, units)` combination is permanently un-takeable on that token. This violates the invariant that valid offers can always be taken, and constitutes a low-cost, repeatable DoS for dust-sized trades on affected markets.

### Likelihood Explanation
Preconditions: (1) the market's `loanToken` reverts on zero-value transfers — a documented behavior of some deployed ERC20 tokens; (2) a taker submits `units` small enough that `units * sellerPrice < WAD` (buy offer) or `units * _settlementFee < WAD` (fee spread). Both conditions are easily satisfied: the market creator selects the token, and the taker controls `units`. The attack is repeatable at negligible cost (only gas for the reverting call).

### Recommendation
Add zero-value guards before each `safeTransferFrom` call in `take()`:

```solidity
uint256 spread = buyerAssets - sellerAssets;
if (spread > 0) SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), spread);
if (sellerAssets > 0) SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

This matches the pattern already used implicitly by the protocol's own test expectations for zero-asset takes.

### Proof of Concept
```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

// Token that reverts on zero-value transferFrom
contract ZeroRevertToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    function mint(address to, uint256 amt) external { balanceOf[to] += amt; }
    function approve(address sp, uint256 amt) external returns (bool) {
        allowance[msg.sender][sp] = amt; return true;
    }
    function transfer(address to, uint256 amt) external returns (bool) {
        require(amt > 0, "zero transfer"); balanceOf[msg.sender] -= amt; balanceOf[to] += amt; return true;
    }
    function transferFrom(address from, address to, uint256 amt) external returns (bool) {
        require(amt > 0, "zero transfer"); // <-- reverts on zero
        allowance[from][msg.sender] -= amt; balanceOf[from] -= amt; balanceOf[to] += amt; return true;
    }
}

// Foundry fuzz test
function testZeroValueTransferDoS(uint256 units) public {
    // Bound units so that units * sellerPrice < WAD (sellerAssets rounds to 0)
    // tick=0 => sellerPrice=0 for buy offer => sellerAssets=0 always
    units = bound(units, 1, 1e18);
    
    // Deploy market with ZeroRevertToken as loanToken
    // Set up buy offer at tick=0 (sellerPrice=0)
    // Taker calls take(units, ...)
    
    // Expected: take() reverts because safeTransferFrom(token, payer, receiver, 0) reverts
    vm.expectRevert("zero transfer");
    midnight.take(buyOffer, hex"", units, taker, receiver, address(0), hex"");
    
    // Assert: offer consumed counter unchanged (no state mutation)
    assertEq(midnight.consumed(offer.maker, offer.group), 0);
}
```

**Assertions:** `take()` reverts; `consumed` is unchanged; the offer remains valid but un-takeable for this `units` value. Fuzz over `units` in `[1, WAD/sellerPrice - 1]` to find all affected inputs.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
