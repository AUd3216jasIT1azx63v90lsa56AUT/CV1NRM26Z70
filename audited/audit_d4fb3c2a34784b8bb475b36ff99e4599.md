### Title
Fully-consumed buy offer with `buyerPrice < WAD` allows indefinite zero-asset re-takes that mutate position state — (`src/Midnight.sol`)

### Summary

When `offer.maxAssets > 0` and `offer.buy == true`, the consumed accounting increments by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. If `buyerPrice < WAD` (reachable with valid tick values), a taker can supply `units = 1` and receive `buyerAssets = 0`, so `newConsumed = maxAssets + 0 = maxAssets ≤ maxAssets` passes the guard. Because `units` is still 1, all downstream position mutations (credit, debt, `totalUnits`) execute normally. This can be repeated without bound after the offer is fully consumed.

### Finding Description

**Code path — `src/Midnight.sol` lines 363–373:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → can be 0
    : units.mulDivUp(buyerPrice, WAD);

uint256 newConsumed;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // ← only guard
}
```

**Root cause:** The consumed cap is enforced by comparing `newConsumed` (incremented by `buyerAssets`) against `maxAssets`. When `buyerPrice < WAD`, `mulDivDown(1 * buyerPrice, WAD) = 0`, so the increment is zero and the check is trivially satisfied even when `consumed` is already at `maxAssets`. There is no guard requiring `buyerAssets > 0` when `units > 0`.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick` set to any tick where `tickToPrice(tick) < WAD` (e.g., `MAX_TICK - 16` per the existing test)
- `units = 1` (or any value small enough that `mulDivDown(units * buyerPrice, WAD) = 0`)

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = M` and `tick` such that `offerPrice < WAD`.
2. Offer is legitimately filled until `consumed[maker][group] == M`.
3. Taker calls `take(offer, ..., units=1)`. `buyerAssets = mulDivDown(1 * offerPrice, WAD) = 0`. `newConsumed = M + 0 = M ≤ M` — `ConsumedAssets` check passes.
4. Execution continues: `buyerCreditIncrease`, `sellerDebtIncrease`, `totalUnits` all change by 1 unit. No tokens transfer (`buyerAssets = sellerAssets = 0`).
5. Step 3–4 can be repeated indefinitely; `consumed` never changes.

**Why existing checks fail:** The sole protection is `require(newConsumed <= offer.maxAssets, ConsumedAssets())` at line 369. When `buyerAssets = 0`, `newConsumed` is unchanged, so the check is always satisfied regardless of how many times the offer has already been fully consumed.

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) already proves this path executes successfully and confirms the position-state divergence. [1](#0-0) [2](#0-1) 

### Impact Explanation

A fully-consumed buy offer (consumed == maxAssets) can be taken indefinitely with `units = 1` when `buyerPrice < WAD`. Each such call:
- Increases the maker's credit by 1 unit (or decreases their debt)
- Increases the taker's debt by 1 unit (or decreases their credit)
- Increases `totalUnits` by 1

No tokens are transferred and `consumed` does not change, so the offer appears unconsumed to the accounting layer while position state diverges from the asset-transfer record. This breaks the invariant that a fully-consumed offer must not allow any further fills, and corrupts the credit/debt/totalUnits accounting without any corresponding asset movement. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

**Preconditions:**
1. `offer.buy == true` and `offer.maxAssets > 0` — a standard buy offer configuration.
2. `offerPrice < WAD` — requires `tick` to map to a price below `1e18`. The test uses `MAX_TICK - 16`, confirming this is a reachable tick value.
3. The offer must have been filled to exactly `maxAssets` consumed — achievable by any taker through normal fills.

All three preconditions are reachable by an unprivileged taker with no special access. The attack is repeatable an unlimited number of times after the offer is fully consumed, and requires only a single `take` call per iteration with `units = 1`. [5](#0-4) 

### Recommendation

Add an explicit check that the asset delta is non-zero whenever `units > 0` and `offer.maxAssets > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetDelta > 0, ZeroAssetDelta());
    newConsumed = consumed[offer.maker][offer.group] += assetDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any non-trivial fill (units > 0) must produce a non-zero asset increment, preventing zero-asset re-takes from bypassing the consumed cap. [6](#0-5) 

### Proof of Concept

The existing Foundry test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` is a complete PoC. It already asserts the exact bad state:

```solidity
// After fully consuming the offer (consumed == maxAssets == 1):
(uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

assertEq(buyerAssets, 0);                                          // zero assets transferred
assertEq(sellerAssets, 0);
assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets); // consumed unchanged
// But position state mutated:
assertGt(midnight.creditOf(id, lender), lenderCreditBefore);      // credit increased
assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);      // debt increased
assertGt(midnight.totalUnits(id), totalUnitsBefore);              // totalUnits increased
```

A complementary fuzz/invariant test should assert: for all `(units, tick)` where `offer.maxAssets > 0` and `consumed == maxAssets`, `take` with `units > 0` must revert with `ConsumedAssets`. [2](#0-1)
