### Title
Fee-on-Transfer Loan Token in `repay` Credits Full `units` to `withdrawable` While Receiving Only `units - fee` — (`src/Midnight.sol`)

### Summary
The `repay` function unconditionally increments `marketState[id].withdrawable` by the caller-supplied `units` before executing the token transfer, and `SafeTransferLib.safeTransferFrom` only validates the boolean return value of `transferFrom`, never the actual balance delta. When the market's `loanToken` is a fee-on-transfer token, the contract receives `units - fee` but credits lenders with `units`, permanently breaking the invariant `IERC20(loanToken).balanceOf(address(midnight)) >= sum(withdrawable[id])` and enabling direct loss of lender funds.

### Finding Description

**Exact code path:**

`repay` in `src/Midnight.sol` lines 502–521:

```
position[id][onBehalf].debt      -= toUint128(units);   // L508 – full units
marketState[id].withdrawable     += toUint128(units);   // L509 – full units
...
SafeTransferLib.safeTransferFrom(
    market.loanToken, payer, address(this), units       // L520 – transfer
);
``` [1](#0-0) 

`SafeTransferLib.safeTransferFrom` in `src/libraries/SafeTransferLib.sol` lines 24–34 only checks `token.code.length > 0`, call success, and the boolean return value. It performs no balance-before/balance-after measurement. [2](#0-1) 

**Root cause:** The accounting update (lines 508–509) is unconditionally based on the argument `units`, not on the actual tokens received. For a fee-on-transfer token, `transferFrom(payer, address(this), units)` returns `true` and emits a transfer event for `units`, but only `units - fee` arrives at the contract. The protocol has no balance-delta check and no restriction on fee-on-transfer tokens anywhere in market creation or the `repay` path (confirmed: no `feeOnTransfer`, `deflationary`, or equivalent guard exists in the codebase).

**Exploit flow:**
1. Attacker (borrower) creates or uses a market whose `loanToken` is a fee-on-transfer ERC-20 (e.g., a token with a 1% transfer fee).
2. Borrower takes a loan of `D` units via normal protocol flow.
3. Borrower calls `repay(market, D, onBehalf, address(0), "")`.
4. Protocol executes: `debt[onBehalf] -= D` and `withdrawable += D` (lines 508–509).
5. `safeTransferFrom` transfers `D` tokens from borrower; only `D * 0.99` arrives at `address(midnight)`.
6. `withdrawable` now exceeds actual balance by `D * 0.01`.
7. Lenders call `withdraw(market, D, ...)` — protocol executes `withdrawable -= D` and `safeTransfer(loanToken, receiver, D)` (lines 494, 499).
8. The `safeTransfer` of `D` drains tokens that belong to other lenders or other markets sharing the same token, causing a shortfall.

**Why existing checks fail:** `SafeTransferLib` is a return-value-only wrapper; it has no balance accounting. There is no market-creation validation rejecting fee-on-transfer tokens. The state mutation precedes the transfer, so even a reentrancy guard (none is present on `repay`) would not help. [3](#0-2) 

### Impact Explanation
Every `repay` call with a fee-on-transfer loan token permanently inflates `marketState[id].withdrawable` relative to the contract's actual token balance. Lenders who withdraw after such repayments receive tokens that were deposited by other lenders or belong to other markets, causing direct, irreversible loss of lender principal. The core invariant `IERC20(loanToken).balanceOf(address(midnight)) >= withdrawable[id]` is violated after the first such repayment.

### Likelihood Explanation
Any unprivileged actor can create a market with a fee-on-transfer loan token (market creation is permissionless and has no token validation). The borrower needs only a standard borrow-then-repay flow. The exploit is repeatable: each `repay` call widens the deficit by `units * feeRate`. Tokens with configurable fees (e.g., USDT's historical fee switch, ERC-20 tokens with owner-settable fees) make this reachable without any malicious token-owner action at exploit time — the fee just needs to be active when `repay` is called.

### Recommendation
Measure the actual balance delta inside `repay` and use it — not the argument `units` — for all accounting updates:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;

position[id][onBehalf].debt      -= UtilsLib.toUint128(received);
marketState[id].withdrawable     += UtilsLib.toUint128(received);
```

Alternatively, explicitly document and enforce (via a registry or market-creation check) that fee-on-transfer and rebasing tokens are unsupported as loan tokens, and add an invariant test that asserts `balanceOf(midnight) >= withdrawable` after every repay.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
// FeeToken: ERC-20 that deducts 1% on every transferFrom
// (implement as a simple mock in test/erc20s/)

contract FeeOnTransferRepayTest is Test {
    Midnight midnight;
    FeeToken loanToken;   // 1% fee-on-transfer mock
    address lender  = address(0xA);
    address borrower = address(0xB);

    function setUp() public {
        midnight = new Midnight(...);
        loanToken = new FeeToken();
        // create market with loanToken, supply liquidity from lender,
        // borrower takes a loan of 1000 units
    }

    function test_feeOnTransferRepayInflatesWithdrawable() public {
        uint256 units = 1000e18;
        uint256 fee   = units / 100; // 1%

        // snapshot balance before repay
        uint256 balBefore = loanToken.balanceOf(address(midnight));

        vm.prank(borrower);
        midnight.repay(market, units, borrower, address(0), "");

        uint256 balAfter = loanToken.balanceOf(address(midnight));
        uint256 received = balAfter - balBefore;

        // Assert: only (units - fee) arrived
        assertEq(received, units - fee);

        // Assert: withdrawable was credited the full units — INVARIANT BROKEN
        uint256 withdrawable = midnight.marketState(id).withdrawable;
        assertGt(withdrawable, loanToken.balanceOf(address(midnight)),
            "invariant violated: withdrawable > balance");

        // Assert: lender can drain more than was deposited
        vm.prank(lender);
        vm.expectRevert(); // or succeeds and steals from other lenders
        midnight.withdraw(market, units, lender, lender);
    }
}
```

Expected assertions:
- `received == units - fee` (only partial amount arrived)
- `marketState[id].withdrawable > IERC20(loanToken).balanceOf(address(midnight))` (invariant broken)
- Subsequent `withdraw` of `units` either reverts (DoS for lenders) or succeeds by consuming tokens belonging to other depositors (theft)
