### Title
Missing deadline parameter in `stake()` allows stale transactions to permanently lock stakers under unintended registry state - (File: src/ConfidencePool.sol)

### Summary
`stake()` accepts no deadline parameter. A transaction submitted when the registry is in a pre-risk state (where `withdraw()` is still available) can execute after the registry has transitioned to `UNDER_ATTACK`, permanently disabling the staker's only exit path and exposing their principal to CORRUPTED resolution loss.

### Finding Description
`stake()` takes only an `amount` argument with no deadline or maximum-state guard: [1](#0-0) 

Internally, `_observePoolState()` is called on every `stake()` invocation: [2](#0-1) 

`_observePoolState()` permanently seals `riskWindowStart` the first time it observes `UNDER_ATTACK`: [3](#0-2) 

Once `riskWindowStart != 0`, `withdraw()` is permanently disabled for all stakers: [4](#0-3) 

A staker who submits `stake()` while the registry is in `NEW_DEPLOYMENT` or `ATTACK_REQUESTED` (pre-risk, withdrawals open) may have their transaction sit in the mempool while the registry transitions to `UNDER_ATTACK`. When the transaction finally executes, `_observePoolState()` seals `riskWindowStart`, `_assertDepositsAllowed` passes (UNDER_ATTACK is explicitly allowed), and the stake is accepted â€” but the staker is now permanently locked in with no exit.

`_assertDepositsAllowed` intentionally permits `UNDER_ATTACK` deposits by design (DESIGN.md Â§3), but that design decision addresses *voluntary* UNDER_ATTACK staking. It does not address â€” and DESIGN.md does not document as an accepted trade-off â€” the scenario where a staker's transaction executes in a materially different state than the one they observed when signing. [5](#0-4) 

### Impact Explanation
A staker who intended to stake in a pre-risk state (where `withdraw()` is available as an escape hatch) is permanently locked into the pool without their informed consent. Their bonus share is crushed to ~zero by the k=2 weighting (entry floored at `riskWindowStart â‰ˆ now`, so `(T âˆ’ entry)Â² â‰ˆ 0`). If the agreement subsequently resolves CORRUPTED, the staker loses their entire principal to `recoveryAddress` â€” a direct, unrecoverable financial loss from a transaction they submitted under entirely different conditions.

### Likelihood Explanation
Requires a mempool delay while the registry transitions from a pre-risk state to `UNDER_ATTACK`. This is realistic on any congested network or when a staker submits with a low gas price. The transition itself is a normal, expected lifecycle event. No adversarial action is required â€” the loss arises purely from timing.

### Recommendation
Add a `deadline` parameter to `stake()` (and symmetrically to `withdraw()` and `contributeBonus()`) that reverts if `block.timestamp > deadline`:

```solidity
function stake(uint256 amount, uint256 deadline) external nonReentrant whenPoolNotPaused {
    if (block.timestamp > deadline) revert DeadlineExpired();
    // ... rest of function
}
```

This gives stakers a standard slippage-protection mechanism: they specify the latest block timestamp at which their transaction is still acceptable, and any stale execution reverts harmlessly rather than locking them into an unintended state.

### Proof of Concept

1. Registry is in `NEW_DEPLOYMENT`. Alice approves the pool and submits `stake(1000e18)` with a low gas price, intending to stake in a state where `withdraw()` is still available.
2. The transaction sits in the mempool. The registry transitions to `UNDER_ATTACK`.
3. Bob (or any keeper) calls `pokeRiskWindow()`, sealing `riskWindowStart = block.timestamp`. *(Alternatively, Alice's own `stake()` call seals it in step 4.)*
4. Alice's `stake()` transaction executes. `_observePoolState()` returns `UNDER_ATTACK` and sets `riskWindowStart` (if not already set). `_assertDepositsAllowed` passes. Alice's 1000e18 tokens are transferred in.
5. Alice calls `withdraw()`. It reverts `WithdrawsDisabled` because `riskWindowStart != 0`.
6. The agreement resolves CORRUPTED. The moderator flags bad-faith CORRUPTED. `claimCorrupted()` sweeps the full pool â€” including Alice's 1000e18 â€” to `recoveryAddress`.
7. Alice has lost her entire principal from a transaction she submitted when the registry was safely in `NEW_DEPLOYMENT`.
