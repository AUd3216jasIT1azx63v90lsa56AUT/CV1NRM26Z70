### Title
Stale Operator Signatures Resurrected on Re-Authorization - ([File: src/ratifiers/EcrecoverRatifier.sol])

### Summary
`EcrecoverRatifier.isRatified` checks operator authorization at call time against the live `Midnight.isAuthorized` mapping with no binding to the authorization epoch in which the signature was created. When a maker revokes an operator and later re-authorizes the same address (even for an entirely different purpose), every root that operator signed during the prior authorization period becomes fillable again, because `isRootCanceled` was never set and the live authorization check now passes.

### Finding Description
**Code path:**

`isRatified` (line 44):
```solidity
require(_signer == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer), Unauthorized());
``` [1](#0-0) 

The only durable invalidation mechanism is `isRootCanceled`, set by an explicit `cancelRoot` call: [2](#0-1) 

`cancelRoot` is never called automatically by `setIsAuthorized` in `Midnight.sol`. There is no nonce, epoch counter, or timestamp embedded in the signed digest that would bind a signature to a specific authorization window.

**Exploit flow:**

1. Maker calls `Midnight.setIsAuthorized(operatorA, true, maker)`.
2. Operator A signs root R (containing offer O) and shares the `(sig, root, leafIndex, proof)` tuple with a taker.
3. Maker calls `Midnight.setIsAuthorized(operatorA, false, maker)` — does **not** call `cancelRoot(maker, R)`.
4. Maker later calls `Midnight.setIsAuthorized(operatorA, true, maker)` for a new, unrelated purpose.
5. Taker submits the old tuple from step 2 to fill offer O.
6. `isRatified` evaluates: `isRootCanceled[maker][R]` → false ✓; `ecrecover(digest)` → operatorA ✓; `isAuthorized(maker, operatorA)` → true ✓ → returns `CALLBACK_SUCCESS`.

**Why existing checks fail:**

- `isRootCanceled` is only set by an explicit `cancelRoot` call; revocation does not trigger it.
- The signed digest covers only `(chainId, verifyingContract, root)` — no authorization nonce or timestamp.
- The maker cannot enumerate all roots operator A ever signed, so silent resurrection is possible even with diligent makers.

The existing test `testIsRatifiedRevokeAuthorizationInvalidates` confirms revocation alone invalidates signatures, but no test covers the re-authorization resurrection path. [3](#0-2) 

### Impact Explanation
Stale operator signatures from a revoked authorization period become fillable again upon re-authorization of the same operator address. Offers the maker believed were permanently invalidated (e.g., old price levels, old amounts, old counterparties) can be executed by any taker who retained the original `ratifierData`, violating the invariant that cancelled or expired offers cannot be filled.

### Likelihood Explanation
Preconditions: maker must revoke and later re-authorize the same operator address without calling `cancelRoot` in between. This is a natural operational pattern (e.g., rotating keys, temporarily suspending an operator, or re-using an address across sessions). The taker only needs to have saved the original `ratifierData` tuple, which is available on-chain or off-chain from the original signing event. The attack is repeatable for every root the operator signed during the prior period.

### Recommendation
Bind signatures to a specific authorization epoch by introducing a per-`(maker, operator)` revocation nonce in `Midnight` (incremented on each `setIsAuthorized(op, false, maker)` call) and including that nonce in the EIP-712 digest signed by the operator. Alternatively, automatically call `cancelRoot` for all known roots on revocation, though root enumeration is impractical on-chain. The minimal fix is to add an `authorizationNonce` field to the signed struct:

```solidity
bytes32 structHash = keccak256(abi.encode(
    offerTreeTypeHash,
    root,
    IMidnight(MIDNIGHT).authorizationNonce(offer.maker, _signer)
));
```

so that incrementing the nonce on revocation permanently invalidates all prior signatures from that operator.

### Proof of Concept
```solidity
function testStaleSignatureResurrectedOnReauthorization() public {
    // 1. Maker authorizes operatorA
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, true, lender);

    // 2. OperatorA signs root R for offer O
    Offer memory offer = makeOffer(lender);
    bytes32 root = HashLib.hashOffer(offer);
    bytes memory ratifierData = buildRatifierData(root, borrower);

    // 3. Maker revokes operatorA (does NOT cancel root)
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, false, lender);

    // Confirm revocation invalidates the signature
    vm.prank(address(midnight));
    vm.expectRevert(IEcrecoverRatifier.Unauthorized.selector);
    ecrecoverRatifier.isRatified(offer, ratifierData);

    // 4. Maker re-authorizes operatorA for a new purpose
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, true, lender);

    // 5. Old signature is now valid again — assert this SUCCEEDS (unexpected)
    vm.prank(address(midnight));
    bytes32 result = ecrecoverRatifier.isRatified(offer, ratifierData);
    assertEq(result, CALLBACK_SUCCESS); // BUG: stale signature accepted
}
```

Expected: the final `isRatified` call reverts with `Unauthorized`. Actual: it returns `CALLBACK_SUCCESS`, confirming the resurrection.
