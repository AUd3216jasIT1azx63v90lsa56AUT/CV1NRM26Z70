### Title
Division by zero in `liquidate()` when `collateralIndex` absent from borrower's `collateralBitmap` with `repaidUnits > 0` - (File: `src/Midnight.sol`)

### Summary
`liquidatedCollatPrice` is initialized to `0` and only assigned inside the while loop when `i == collateralIndex`. If the caller supplies a `collateralIndex` not set in the borrower's `collateralBitmap`, the loop never writes it, leaving it `0`. The subsequent `mulDivDown(..., liquidatedCollatPrice)` at line 652 performs integer division by zero, causing an unhandled panic revert instead of a protocol-defined error.

### Finding Description
**Code path:**
- `liquidatedCollatPrice = 0` declared at [1](#0-0) 
- While loop iterates only over `_position.collateralBitmap` bits; assignment is conditional on `i == collateralIndex` at [2](#0-1) 
- Division by zero at [3](#0-2) 
- `mulDivDown` has no zero-denominator guard: [4](#0-3) 

**Attacker inputs:** `collateralIndex` = any index not set in borrower's `collateralBitmap`; `seizedAssets = 0`; `repaidUnits > 0`.

**Exploit flow:**
1. Borrower supplies collateral at index 0 only → `collateralBitmap = 0b1`.
2. Borrower borrows; position becomes unhealthy (oracle price drops).
3. Liquidator calls `liquidate(market, collateralIndex=1, seizedAssets=0, repaidUnits=1, borrower, false, ...)`.
4. `atMostOneNonZero(1, 0)` passes (line 595). `debt > 0` passes (line 596). `NotLiquidatable` passes because `originalDebt > maxDebt` (line 622).
5. While loop runs for `i=0` only; `i != 1` so `liquidatedCollatPrice` stays `0`.
6. `repaidUnits > 0` branch entered at line 643; line 652 executes `mulDivDown(ORACLE_PRICE_SCALE, 0)` → panic.

**Why checks fail:** No check validates that `collateralIndex` is present in `_position.collateralBitmap` before the non-zero branch. The NatSpec at lines 578–579 explicitly permits unactivated `collateralIndex` for the zero-zero case, but the same permissiveness is silently inherited by the non-zero branches without a guard. [5](#0-4) 

### Impact Explanation
Any liquidator can cause their own liquidation call to revert with a division-by-zero panic (Solidity panic code `0x12`) by supplying a `collateralIndex` not in the borrower's bitmap alongside `repaidUnits > 0`. The revert is ungraceful — no protocol-defined error is emitted. While this does not corrupt state (the revert is total), it means liquidation of a genuinely unhealthy position fails silently for any caller who passes an invalid index, and the protocol provides no diagnostic. Repeated calls with invalid indices constitute a self-inflicted DoS on the liquidation path for that caller.

### Likelihood Explanation
Preconditions are trivially reachable: any borrower with debt and a subset of activated collaterals, combined with a liquidator (or automated bot) passing a `collateralIndex` outside that subset. Multi-collateral markets make this more likely as the valid index set is a strict subset of all market collateral indices. The position only needs to be unhealthy or post-maturity, which is the normal liquidation precondition. The attack is repeatable at zero cost beyond gas.

### Recommendation
Before entering the `repaidUnits > 0 || seizedAssets > 0` block, add an explicit check that `collateralIndex` is set in `_position.collateralBitmap`:

```solidity
if (repaidUnits > 0 || seizedAssets > 0) {
    require(
        _position.collateralBitmap & (1 << collateralIndex) != 0,
        InvalidCollateralIndex()
    );
    // ... existing logic
}
```

This mirrors the pattern already used in `supplyCollateral` and `withdrawCollateral` where `collateralIndex` is implicitly validated through bitmap operations, and aligns with the existing comment that zero-zero liquidations are the only intended use of unactivated collateral indices. [6](#0-5)
