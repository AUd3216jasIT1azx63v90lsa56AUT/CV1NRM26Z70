### Title
Fee-on-Transfer Collateral Tokens Enable Undercollateralized Borrowing - (File: src/Midnight.sol)

### Summary
`supplyCollateral` credits a borrower's position with the full requested `assets` amount before verifying what was actually received. When a fee-on-transfer (deflationary) collateral token is used, the protocol records more collateral than it holds. Because `isHealthy` computes `maxDebt` from the inflated recorded collateral, the borrower can carry debt that exceeds the real collateral value, creating guaranteed bad debt for lenders. A parallel accounting gap in `repay` inflates `withdrawable` beyond actual token receipts, enabling lenders to drain the protocol.

### Finding Description

**Root cause — `supplyCollateral` (lines 524–546)**

```solidity
// line 533 — accounting written BEFORE transfer, using caller-supplied `assets`
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
...
// line 545 — actual transfer; fee-on-transfer token delivers (assets − fee)
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

The position records `assets` but the contract receives `assets − fee`. No before/after balance check exists.

**Health check uses the inflated figure (lines 944–960)**

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
               .mulDivDown(collateralParam.lltv, WAD);
```

`_position.collateral[i]` is the over-stated value, so `maxDebt` is computed against phantom collateral. The borrower passes the health check and can borrow against tokens the protocol never received.

**Secondary gap — `repay` (lines 502–521)**

```solidity
// line 508-509 — accounting updated with full `units`
position[id][onBehalf].debt     -= UtilsLib.toUint128(units);
marketState[id].withdrawable    += UtilsLib.toUint128(units);
...
// line 520 — only (units − fee) actually arrives
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
```

`withdrawable` grows by `units` while the vault receives `units − fee`, creating a shortfall that lenders can exploit via `withdraw`.

**Documented assumption does not prevent exploitation**

The TOKEN SAFETY REQUIREMENTS comment (lines 133–140) states *"Midnight must send/receive exactly the requested amount on transfers"* as an assumption, but the protocol imposes no on-chain enforcement. Any unprivileged user can call `touchMarket` / `supplyCollateral` with an arbitrary token, including fee-on-transfer tokens, without any gate or check.

### Impact Explanation

- **Undercollateralized borrowing**: A borrower supplying a 1%-fee collateral token and recording `100` units while depositing `99` can borrow at the LLTV of the inflated `100`. The resulting position is immediately undercollateralized by the fee amount, socializing bad debt onto all lenders in the market.
- **Protocol insolvency via `repay`**: Each repayment with a fee-on-transfer loan token inflates `withdrawable` beyond actual holdings. Lenders calling `withdraw` drain the vault faster than debt is repaid, leaving later lenders unable to withdraw.
- **Severity**: High — direct, permanent loss of lender funds with no recovery path once the position is opened.

### Likelihood Explanation

- No privileged access is required. Any user can create a market (`touchMarket` is permissionless) and supply any ERC-20 token as collateral or loan token.
- Fee-on-transfer tokens are deployed on mainnet (e.g., STA, PAXG, various reflection tokens). A malicious actor can deploy their own minimal fee-on-transfer ERC-20 to exploit this deterministically.
- The attack is profitable for the borrower (free leverage) and requires only standard user-level interactions.

### Recommendation

Record the actual received amount by measuring the balance delta around each inbound transfer:

```solidity
// supplyCollateral — measure actual receipt
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Apply the same pattern to `repay` (credit `withdrawable` with the measured receipt, not `units`) and to the settlement-fee leg of `take`. Alternatively, enforce the token requirement at market creation by reverting if a test transfer returns a balance delta that differs from the requested amount.

### Proof of Concept

1. Deploy `FeeToken` — an ERC-20 that deducts 1% on every `transferFrom`.
2. Create a Midnight market with `FeeToken` as the sole collateral token and any standard loan token.
3. Call `supplyCollateral(market, 0, 10_000e18, attacker)`.
   - Position records `collateral[0] = 10_000e18`.
   - Contract receives `9_900e18` (1% fee deducted).
4. Call `take` to borrow the maximum units permitted by `isHealthy`, which computes `maxDebt` against the inflated `10_000e18`.
5. The position is immediately undercollateralized by `100e18 * price * lltv / WAD` in debt units.
6. Liquidating the position realizes bad debt that is socialized across all lenders, reducing their credit proportionally via the `lossFactor` mechanism. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
