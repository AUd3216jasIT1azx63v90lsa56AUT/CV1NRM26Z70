Audit Report

## Title
Maker-controlled sell callback can transiently manipulate oracle price, causing `isHealthy` to return false and reverting valid taker transactions - (File: src/Midnight.sol)

## Summary
When a taker fills a sell offer (`offer.buy == false`), the maker's `offer.callback` is assigned as `sellerCallback` and invoked before the post-callback health check at line 476. A malicious maker can implement `onSell` to transiently depress a spot/reserve-based oracle within the callback, causing `isHealthy(offer.market, id, seller)` to return `false` for a position that was healthy at pre-callback prices. The transaction reverts with `SellerIsLiquidatable`, all state changes are undone atomically, and the attack is repeatable at near-zero cost (gas only).

## Finding Description

**Callback assignment (line 421):**
```solidity
address sellerCallback = offer.buy ? takerCallback : offer.callback;
```
When `offer.buy == false`, `sellerCallback = offer.callback` — fully controlled by the maker at offer-creation time. No validation of `offer.callback` exists anywhere in `take()`.

**Seller identity (line 375):**
```solidity
(address buyer, address seller) = offer.buy ? (offer.maker, taker) : (taker, offer.maker);
```
When `offer.buy == false`, `seller = offer.maker`. The health check at line 476 is therefore on the maker's own position.

**State mutation before callback (line 414):**
`sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease)` executes before the callback fires, so `isHealthy` will evaluate against a position that now carries more debt.

**Callback invocation (lines 458–474):**
`ISellCallback(sellerCallback).onSell(...)` is called with no restrictions on external state mutations. The callback can freely interact with pool reserves, oracle state, etc.

**Lock reset and health check (lines 475–476):**
```solidity
if (!wasLocked) UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, false);
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```
In the normal case (`wasLocked == false`), the lock is reset to `false` before the `require`. `liquidationLocked(id, seller)` therefore evaluates to `false`, and `isHealthy` must return `true`. If the callback has shifted the oracle price, `isHealthy` reads the manipulated value and returns `false`.

**`isHealthy` oracle read (line 953):**
```solidity
uint256 price = IOracle(collateralParam.oracle).price();
```
This is a live call to the oracle with no snapshot or TWAP enforcement at the protocol level.

**Exploit flow:**
1. Maker deploys `MaliciousCallback` implementing `ISellCallback.onSell` that swaps tokens in the oracle's underlying pool (using tokens held by the callback contract), dropping the reported collateral price.
2. Maker creates a sell offer with `offer.callback = address(MaliciousCallback)` and sufficient collateral that the position is healthy at the pre-callback price.
3. Taker calls `take()`. Maker's debt increases (line 414), lock is set to `true` (line 444).
4. `MaliciousCallback.onSell` executes the pool swap; oracle price drops below the health threshold. Callback returns `CALLBACK_SUCCESS`.
5. Lock is reset to `false` (line 475).
6. `isHealthy(offer.market, id, seller)` reads the manipulated price → returns `false`.
7. `require(false || false, SellerIsLiquidatable())` → transaction reverts; all state changes (including the pool swap) are undone atomically. Maker's tokens are returned.

**Why existing protections are insufficient:**
The `LIQUIDATION_LOCK_SLOT` mechanism (lines 444, 475) is designed solely to prevent reentrant liquidation calls during callbacks — confirmed by the comment at line 335: *"The seller cannot be liquidated during the callbacks of a take."* It does not protect the post-callback `isHealthy` evaluation from reading oracle state that was mutated during the callback. The Certora `Healthiness.spec` (lines 14–16) explicitly assumes `price()` does not change during rules (`function _.price() external => summaryPrice(calledContract)`), and the `genericCallback` summary at line 172 uses `require callIsHealthy(...)` as an assumption — not a proven property — meaning this attack vector is entirely outside the formal verification's scope.

The existing test `testSellSellerCallbackLiquidateRevertsWhileLiquidationLocked` (TakeTest.sol lines 1362–1375) even demonstrates oracle price manipulation within a sell callback, but the test restores the price before returning (`oracle.setPrice(healthyPrice)` at line 1375). The vulnerability arises when the price is NOT restored before the callback returns.

## Impact Explanation
Any taker attempting to fill a malicious maker's sell offer will have their transaction revert with `SellerIsLiquidatable`. No funds are lost (revert undoes all state changes), but the offer is never consumed, making the attack repeatable indefinitely. The maker can permanently block market activity on any offer they choose to poison, grief specific takers, or deny liquidity to an entire market. This constitutes service unavailability and severe degradation of market function under realistic attacker input — an in-scope impact per RESEARCHER.md. Oracle manipulation attacks are explicitly not excluded per SECURITY.md line 26.

## Likelihood Explanation
**Preconditions:**
- Market uses an oracle whose `price()` reads mutable on-chain state (spot price, reserve-based oracle, short-window TWAP). This is the common case for DeFi markets.
- Maker's position must be healthy before the callback (so the offer is valid) but the oracle must be manipulable enough to push it below the health threshold.

**Feasibility:** High. The maker controls `offer.callback` unconditionally; no privileged access is required. The attack is atomic (single transaction), repeatable, and costs only gas — since the revert undoes the pool swap and returns the maker's tokens. Any market with a spot oracle is vulnerable.

## Recommendation
1. **Snapshot the oracle price before the callback** and reuse the snapshot in the post-callback `isHealthy` call, preventing the callback from influencing the health evaluation.
2. Alternatively, **move the health check before the seller callback** is invoked, so oracle state at callback time is irrelevant to the health evaluation.
3. As a defense-in-depth measure, require oracles integrated with the protocol to implement TWAP or manipulation-resistant price feeds, and document this requirement explicitly.

## Proof of Concept
Deploy a `MaliciousCallback` contract that, in `onSell`, performs a large swap in the oracle's underlying AMM pool to depress the collateral price below the health threshold, then returns `CALLBACK_SUCCESS` without restoring the price. Create a sell offer with `offer.callback = address(MaliciousCallback)`. Have a taker call `take()` on this offer. The transaction will revert with `SellerIsLiquidatable` because:
- `wasLocked` is `false` (normal case), so the lock is reset before the health check
- `liquidationLocked(id, seller)` returns `false`
- `isHealthy` reads the manipulated oracle price and returns `false`
- The revert undoes the pool swap, returning the maker's tokens

The existing test infrastructure at TakeTest.sol lines 1362–1375 already demonstrates the oracle manipulation pattern within a sell callback; removing the `oracle.setPrice(healthyPrice)` restoration at line 1375 would reproduce the vulnerability exactly.
