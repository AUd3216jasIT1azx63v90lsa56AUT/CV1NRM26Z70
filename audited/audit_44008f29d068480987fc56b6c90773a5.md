### Title
`liquidate` reverts with division-by-zero when `collateralIndex` is not in `collateralBitmap` and `repaidUnits > 0` - (File: src/Midnight.sol)

### Summary
In `Midnight.liquidate`, `liquidatedCollatPrice` is initialized to `0` and only assigned inside the bitmap loop when `i == collateralIndex`. If the caller passes a `collateralIndex` whose bit is not set in the borrower's `collateralBitmap`, `liquidatedCollatPrice` remains `0` after the loop. When `repaidUnits > 0` and `seizedAssets == 0`, the function then executes `repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, 0)`, which reverts with division by zero. No runtime check enforces that `collateralIndex` must be active in the bitmap when a non-zero amount is specified.

### Finding Description

**Root cause — missing bitmap membership check before price-dependent arithmetic:**

`liquidatedCollatPrice` is declared at line 603 and defaults to `0`: [1](#0-0) 

It is only set inside the bitmap loop when the iterated index matches the caller-supplied `collateralIndex`: [2](#0-1) 

If bit `collateralIndex` is absent from `_position.collateralBitmap`, the condition `i == collateralIndex` is never true, and `liquidatedCollatPrice` stays `0`.

The `NotLiquidatable` guard at line 620–624 only checks `originalDebt > maxDebt` (normal mode) or `block.timestamp > market.maturity` (post-maturity mode) — neither condition requires `collateralIndex` to be active: [3](#0-2) 

After that guard, when `repaidUnits > 0` and `seizedAssets == 0`, execution reaches: [4](#0-3) 

`mulDivDown(ORACLE_PRICE_SCALE, 0)` reverts with division by zero.

**Exploit flow:**
1. Borrower has `debt > 0` and is unhealthy (`originalDebt > maxDebt`), with collateral only at index `j` (bit `j` set in bitmap).
2. Unprivileged liquidator calls `liquidate(market, collateralIndex=i, seizedAssets=0, repaidUnits=N>0, borrower, false, ...)` where bit `i` is NOT set.
3. `atMostOneNonZero` check passes (only `repaidUnits` is nonzero).
4. `debt > 0` check passes.
5. Bitmap loop runs over bit `j` only; `liquidatedCollatPrice` is never assigned → remains `0`.
6. `NotLiquidatable` check passes (position is genuinely unhealthy).
7. `repaidUnits > 0` branch entered; `seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, 0)` → **division by zero revert**.

**Why existing checks are insufficient:**
The only input validation is `atMostOneNonZero` (line 595) and `debt > 0` (line 596). There is no check that `collateralIndex` is a member of `_position.collateralBitmap` when `repaidUnits > 0` or `seizedAssets > 0`.

The Certora formal verification spec explicitly acknowledges this gap — it requires bitmap membership as a *precondition assumption*, not as a runtime guarantee: [5](#0-4) 

The `LiquidationBoundedByLIF.spec` comment further confirms the revert is a known consequence, not a protected invariant: [6](#0-5) 

The NatSpec comment at line 578–579 states that zero-amount liquidations may use a non-activated collateral, implying non-zero liquidations should not — but this is never enforced at runtime: [7](#0-6) 

### Impact Explanation
Any caller can make `liquidate` revert with an opaque division-by-zero panic instead of a descriptive error, for any unhealthy position, by supplying a `collateralIndex` not present in the borrower's bitmap alongside `repaidUnits > 0`. The position remains liquidatable via a correct call, so this is a DoS scoped to the specific call with the invalid index — not a protocol-wide liquidation freeze. However, it produces an unexpected, uninformative revert path that violates the invariant that liquidation of a genuinely unhealthy position must not revert unexpectedly for structurally valid (non-zero amount) inputs.

### Likelihood Explanation
Preconditions are easy to satisfy: any unhealthy borrower with at least one active collateral suffices. The liquidator only needs to pass a `collateralIndex` value that is within the `market.collateralParams` array bounds but whose bit is not set in the borrower's bitmap. This is reachable by mistake (off-by-one index) or deliberately. It is repeatable as long as the position remains unhealthy and the chosen index remains inactive.

### Recommendation
Add an explicit runtime check immediately before the price-dependent arithmetic block that enforces `collateralIndex` is active in the bitmap whenever a non-zero amount is being liquidated:

```solidity
if (repaidUnits > 0 || seizedAssets > 0) {
    require(
        _position.collateralBitmap.getBit(collateralIndex),
        InactiveCollateralIndex()
    );
    // ... existing arithmetic
}
```

Alternatively, assert `liquidatedCollatPrice != 0` before the division, which also catches the case where the oracle itself returns `0`.

### Proof of Concept

```solidity
function testLiquidateReverts_CollateralIndexNotInBitmap() public {
    // Setup: borrower supplies collateral at index 0 only
    midnight.supplyCollateral(market, 0, 1e18, borrower);
    // Borrow enough to become unhealthy
    // ... (drop oracle price so originalDebt > maxDebt)
    oracle0.setPrice(1); // make position unhealthy

    // Sanity: bit 0 is set, bit 1 is NOT set
    uint128 bitmap = midnight.collateralBitmap(toId(market), borrower);
    assertEq(bitmap & (1 << 1), 0, "bit 1 must not be set");

    // Liquidator calls with collateralIndex=1 (not in bitmap) and repaidUnits>0
    deal(address(loanToken), address(this), 1e18);
    loanToken.approve(address(midnight), 1e18);

    // Expected: reverts with division by zero (Panic(0x12))
    vm.expectRevert(stdError.divisionError);
    midnight.liquidate(market, 1, 0, 1e18, borrower, false, address(this), address(0), "");
}
```

Expected assertion: the call reverts with a division-by-zero panic (`Panic(0x12)`), confirming `liquidatedCollatPrice == 0` was used as the denominator at line 652. [8](#0-7)
