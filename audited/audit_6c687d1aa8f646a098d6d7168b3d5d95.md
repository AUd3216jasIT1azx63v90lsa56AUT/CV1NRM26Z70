### Title
Unchecked `uint256` multiplication overflow in `isHealthy()` permanently freezes `withdrawCollateral()` and `take()` for borrowers with large collateral and high oracle price — (`src/Midnight.sol`)

### Summary

`UtilsLib.mulDivDown` computes `(x * y) / d` using plain Solidity 0.8.x checked arithmetic with no intermediate overflow guard. Inside `isHealthy()`, the sub-expression `_position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)` multiplies a `uint128` collateral value by a `uint256` oracle price; if their product exceeds `type(uint256).max`, Solidity's checked arithmetic reverts. Because `withdrawCollateral()` calls `isHealthy()` unconditionally when the borrower has debt, and `take()` calls it to evaluate the seller's liquidatability, both entry points permanently revert for any borrower whose stored collateral and live oracle price satisfy the overflow condition. The protocol's own NatSpec explicitly acknowledges this as a known liveness limitation but provides no on-chain mitigation.

### Finding Description

**Exact code path:**

`mulDivDown` in `src/libraries/UtilsLib.sol` lines 29–31:
```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;   // checked mul — reverts on overflow
}
```

`isHealthy` in `src/Midnight.sol` lines 944–960:
```solidity
uint256 price = IOracle(collateralParam.oracle).price();
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

`_position.collateral[i]` is stored as `uint128` (max ≈ 3.4 × 10³⁸). `price` is an unconstrained `uint256`. The product `collateral[i] * price` overflows `uint256` when:

```
price > type(uint256).max / collateral[i]
     ≈ 2^256 / 2^128  =  2^128  ≈  3.4 × 10³⁸
```

Since `ORACLE_PRICE_SCALE = 1e36`, this threshold corresponds to a collateral-to-loan-token exchange rate of roughly **340 loan tokens per collateral unit** when collateral is at its maximum storable value — a realistic condition for many token pairs (e.g., a low-decimal collateral token against a high-decimal loan token, or any token that appreciates significantly after the position is opened).

**Why existing checks do not stop it:**

The Certora `NoMultiplicationOverflow.spec` proof is explicitly conditional on a bounded oracle price assumption (`boundedPrice` ghost, line 46–49 of that spec), not an on-chain invariant. No `require` in `supplyCollateral`, `touchMarket`, or anywhere else caps the oracle price or validates `collateral * price` before it is used. The protocol's own NatSpec (lines 147–149 of `src/Midnight.sol`) documents the revert as a known liveness limitation but provides no fix.

**Exploit flow:**

1. Borrower calls `supplyCollateral` with `assets` near `type(uint128).max` (permitted — `toUint128` only reverts above `type(uint128).max`).
2. Borrower calls `take` (sell offer) to acquire debt.
3. Oracle price rises (or was always high) such that `collateral * price > type(uint256).max`.
4. Any call to `withdrawCollateral` or `take` (when seller has debt) internally calls `isHealthy`, which calls `mulDivDown(collateral, price, ORACLE_PRICE_SCALE)`, triggering the checked-arithmetic revert.
5. The borrower's collateral is permanently frozen — they cannot withdraw it and cannot repay via `take`.

### Impact Explanation

The borrower's collateral is permanently inaccessible: `withdrawCollateral` reverts at line 568 (`require(isHealthy(...), UnhealthyBorrower())`) because `isHealthy` itself reverts before returning. `take` similarly reverts when evaluating the seller's health. The borrower cannot recover their collateral tokens even though their position may be economically healthy. This is a direct fund freeze scoped exactly to the stated impact.

### Likelihood Explanation

The overflow requires `collateral[i] * price > 2^256`. With `collateral[i] = type(uint128).max`, the price threshold is ≈ 340 × `ORACLE_PRICE_SCALE` — achievable for real token pairs with mismatched decimals or significant price appreciation. The protocol's own test suite (`MaxAmountsTest.sol` line 58) already sets `oracle1.setPrice(ORACLE_PRICE_SCALE * 1e36)` (= 10⁷²), demonstrating that extreme prices are considered in testing; with a collateral amount of only ≈ 10⁵ at that price, the product would overflow. The condition is not attacker-controlled in isolation (the borrower does not control the oracle), but it is reachable through normal market appreciation after a large collateral deposit, making it a realistic liveness failure rather than a theoretical one.

### Recommendation

Replace the bare multiplication in `mulDivDown` with a 512-bit intermediate or use a well-audited full-precision `mulDiv` (e.g., Solady's `FullMath.mulDiv`) that avoids intermediate overflow. Alternatively, add an on-chain oracle price cap enforced at market creation or at the point of use:

```solidity
require(price <= type(uint256).max / collateral, OraclePriceTooHigh());
```

before each `mulDivDown(collateral, price, ORACLE_PRICE_SCALE)` call inside `isHealthy` and `liquidate`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {Test, stdError} from "forge-std/Test.sol";
import {Midnight} from "../src/Midnight.sol";
import {Market, CollateralParams} from "../src/interfaces/IMidnight.sol";
import {ORACLE_PRICE_SCALE} from "../src/libraries/ConstantsLib.sol";
import {BaseTest} from "./BaseTest.sol";

contract OverflowFreezePoC is BaseTest {
    Market internal market;

    function setUp() public override {
        super.setUp();
        market.loanToken = address(loanToken);
        market.maturity = vm.getBlockTimestamp() + 365 days;
        market.collateralParams.push(CollateralParams({
            token: address(collateralToken1),
            lltv: 0.77e18,
            maxLif: maxLif(0.77e18, 0.25e18),
            oracle: address(oracle1)
        }));
        market.rcfThreshold = 0;
    }

    /// @dev Fuzz: assert withdrawCollateral never reverts with arithmetic overflow
    ///      when price < type(uint256).max / collateral (safe zone).
    function testFuzz_noOverflowInSafeZone(uint128 collateralAmt, uint256 price) public {
        vm.assume(collateralAmt > 0);
        // Safe zone: product fits in uint256
        vm.assume(price <= type(uint256).max / collateralAmt);

        oracle1.setPrice(price);
        deal(address(collateralToken1), address(this), collateralAmt);
        midnight.supplyCollateral(market, 0, collateralAmt, borrower);

        // Create debt so isHealthy is called
        setupMarket(market, 1);

        vm.prank(borrower);
        // Must NOT revert with arithmetic overflow
        try midnight.withdrawCollateral(market, 0, 0, borrower, borrower) {} catch (bytes memory err) {
            // Only UnhealthyBorrower is acceptable; arithmetic overflow is not
            assertNotEq(bytes4(err), bytes4(stdError.arithmeticError));
        }
    }

    /// @dev Unit: demonstrate the overflow revert when price exceeds safe threshold.
    function testUnit_overflowFreezesWithdraw() public {
        uint128 collateralAmt = type(uint128).max;
        // price chosen so collateralAmt * price > type(uint256).max
        uint256 price = type(uint256).max / collateralAmt + 2; // just above threshold

        oracle1.setPrice(price);
        deal(address(collateralToken1), address(this), collateralAmt);
        midnight.supplyCollateral(market, 0, collateralAmt, borrower);
        setupMarket(market, 1); // give borrower debt

        vm.prank(borrower);
        vm.expectRevert(stdError.arithmeticError); // overflow in mulDivDown
        midnight.withdrawCollateral(market, 0, 0, borrower, borrower);
    }
}
```

**Expected assertions:**
- `testFuzz_noOverflowInSafeZone`: passes for all `(collateralAmt, price)` pairs in the safe zone — no arithmetic overflow revert.
- `testUnit_overflowFreezesWithdraw`: the `vm.expectRevert(stdError.arithmeticError)` assertion passes, confirming the freeze.
