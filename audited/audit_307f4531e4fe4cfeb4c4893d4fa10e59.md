### Title
Fee-on-Transfer Loan Token Inflates `withdrawable` in `liquidate`, Causing Protocol Insolvency - (File: src/Midnight.sol)

### Summary
In `liquidate`, `_marketState.withdrawable` is increased by `repaidUnits` before `SafeTransferLib.safeTransferFrom` is called to pull the loan token from the payer. When the loan token is fee-on-transfer, the contract receives only `repaidUnits * (1 - fee)` but records the full `repaidUnits` as withdrawable. This inflates `withdrawable` beyond the actual token balance held by the contract, making the protocol insolvent for lenders who withdraw last.

### Finding Description

**Code path:**

In `src/Midnight.sol`, the `liquidate` function at lines 675–676 unconditionally updates accounting state before the external token pull at line 717:

```solidity
// src/Midnight.sol:675-676
_marketState.withdrawable += UtilsLib.toUint128(repaidUnits);
_position.debt -= UtilsLib.toUint128(repaidUnits);
```

Then, after emitting an event and executing the collateral transfer and optional callback:

```solidity
// src/Midnight.sol:717
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```

`SafeTransferLib.safeTransferFrom` only checks that `transferFrom` returns `true` (or no return data); it does not measure the contract's balance delta:

```solidity
// src/libraries/SafeTransferLib.sol:24-34
function safeTransferFrom(address token, address from, address to, uint256 value) internal {
    require(token.code.length > 0, NoCode());
    (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
    ...
    require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
}
```

For a fee-on-transfer token, `transferFrom(payer, address(this), R)` deducts `R` from the payer and delivers only `R * (1 - fee)` to the contract. The return value is still `true`, so `safeTransferFrom` does not revert. The result is:

- `_marketState.withdrawable` increases by `R`
- Contract balance increases by only `R * (1 - fee)`
- Gap per liquidation: `R * fee`

**Attacker-controlled inputs:**

Any unprivileged user can call `touchMarket` with a fee-on-transfer token as `market.loanToken` — `touchMarket` has no restriction on token type. Once the market exists, any liquidator can call `liquidate` with any valid `repaidUnits`. The liquidator does not profit from this; the damage is to lender solvency.

**Why existing checks fail:**

The Certora solvency spec explicitly assumes away this class of token at line 31–33:

```
// Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing...
```

There is no on-chain guard in `touchMarket`, `liquidate`, or `SafeTransferLib` that detects or rejects fee-on-transfer behavior. The same accounting pattern in `repay` (lines 509, 520) is identically vulnerable.

### Impact Explanation

After each liquidation with a fee-on-transfer loan token, `marketState[id].withdrawable` exceeds `loanToken.balanceOf(address(this))` by `repaidUnits * fee`. The `withdraw` function transfers `units` of loan tokens and decreases `withdrawable` by `units` without any balance check. As lenders withdraw, the contract's actual balance is exhausted before `withdrawable` reaches zero. The last lenders to withdraw receive nothing, constituting a direct loss of funds and protocol insolvency.

### Likelihood Explanation

**Preconditions:**
1. A market must exist with a fee-on-transfer loan token. Any unprivileged user can create such a market via `touchMarket`.
2. The market must have at least one liquidatable position (unhealthy borrower or post-maturity).

**Feasibility:** Fee-on-transfer tokens are a well-known ERC20 variant (e.g., tokens with deflationary mechanics or tax tokens). The `live_context.json` explicitly flags this as a risk category that "should be tested if not explicitly excluded," and no exclusion exists in the protocol. The exploit is repeatable: every liquidation in such a market widens the insolvency gap by `repaidUnits * fee`.

### Recommendation

Use a balance-before/after pattern in `safeTransferFrom` calls that credit internal accounting, or add an explicit check after the transfer:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
require(received == repaidUnits, FeeOnTransferToken());
```

Alternatively, document and enforce that fee-on-transfer tokens are not supported as loan tokens, and add a check in `touchMarket` that reverts if the token takes a fee (e.g., by performing a test transfer during market creation).

contract FeeOnTransferToken is ERC20 {
    uint256 public feeBps; // e.g. 100 = 1%
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount * feeBps / 10000;
        super.transferFrom(from, to, amount - fee); // deliver less
        _burn(from, fee); // or send to fee collector
        return true;
    }
}

function testFuzz_liquidateInflatesWithdrawable(uint256 feeBps, uint256 repaidUnits) public {
    feeBps = bound(feeBps, 1, 1000); // 0.01% to 10%
    repaidUnits = bound(repaidUnits, 1, MAX_DEBT);

    FeeOnTransferToken fot = new FeeOnTransferToken(feeBps);
    // Create market with fot as loanToken
    // Set up borrower position with debt >= repaidUnits
    // Make position liquidatable (drop oracle price)

    uint256 balBefore = fot.balanceOf(address(midnight));
    uint256 withdrawableBefore = midnight.withdrawable(id);

    midnight.liquidate(market, 0, 0, repaidUnits, borrower, false, receiver, address(0), "");

    uint256 withdrawableAfter = midnight.withdrawable(id);
