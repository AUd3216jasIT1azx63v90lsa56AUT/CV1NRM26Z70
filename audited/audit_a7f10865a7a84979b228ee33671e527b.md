Audit Report

## Title
`liquidatorGate` access control bypassed via `callback`/`receiver` payer substitution - (File: src/Midnight.sol)

## Summary
The `liquidate` function checks `liquidatorGate.canLiquidate` only against `msg.sender`, but when `callback != address(0)` the actual loan token repayment is pulled from the `callback` address and collateral is sent to the caller-controlled `receiver`. A whitelisted address A can call `liquidate(callback=B, receiver=B)`, causing gate-blocked address B to be the true economic actor â€” paying loan tokens and receiving seized collateral â€” while the gate never inspects B. This fully circumvents the access control the gate is designed to enforce.

## Finding Description
**Gate check** (`src/Midnight.sol` lines 597â€“600): only `msg.sender` is evaluated by `canLiquidate`:
```solidity
require(
    market.liquidatorGate == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
```

**Payer substitution** (line 679): when `callback` is non-zero, the payer becomes the callback address:
```solidity
address payer = callback != address(0) ? callback : msg.sender;
```

**Collateral transfer** (line 696): collateral is sent to `receiver`, which is a caller-supplied parameter with no gate check.

**Loan token pull** (line 717): tokens are pulled from `payer` (the callback address B), not `msg.sender` (A):
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```

**Exploit flow:**
1. Market has `liquidatorGate` that whitelists A and blocks B.
2. B deploys a contract implementing `ILiquidateCallback`, pre-approves Midnight to pull loan tokens from it.
3. A calls `midnight.liquidate(market, ..., receiver=B, callback=B, ...)`.
4. Gate calls `canLiquidate(A)` â†’ `true` â†’ passes.
5. Collateral transferred to B (`receiver`).
6. `onLiquidate` called on B; B returns `CALLBACK_SUCCESS`.
7. Loan tokens pulled from B (`payer=B`).
8. B has fully executed the liquidation despite being gate-blocked.

**Why existing checks fail:** The Certora rule `liquidatorGateBlocksLiquidation` (`certora/specs/Reverts.spec` lines 325â€“332) only proves that a call reverts when `canLiquidate(msg.sender)` is false â€” it does not constrain the callback address. The `OnlyExplicitPayerCanLoseTokens.spec` rule (line 128) explicitly sets `liquidateCallbackAllowed = true` for `liquidate`, confirming the callback-as-payer path is intentional but the gate gap is unaddressed.

## Impact Explanation
The `liquidatorGate` access control is fully circumvented for the actual economic actor. Any market relying on the gate for KYC, regulatory compliance, or trust-based liquidator restrictions has those restrictions rendered ineffective. B receives seized collateral and bears the loan repayment cost â€” the full economic outcome of a liquidation â€” without ever being checked by the gate. This constitutes unauthorized state change and unauthorized movement of assets (collateral) to a gate-blocked address.

## Likelihood Explanation
Requires whitelisted address A to cooperate with blocked address B, or for A to be a public/permissionless liquidation router contract (common in DeFi). In the router case, any external user can supply `callback=B` and `receiver=B` as arguments with no collusion needed from A's operator. Even with a cooperative EOA, the setup is low-friction: B deploys a callback contract, pre-approves Midnight, and asks any whitelisted address to submit the transaction. The condition is repeatable on every liquidatable position in every gated market.

## Recommendation
Extend the gate check to also validate the `callback` address (and optionally `receiver`) when they are non-zero:
```solidity
require(
    market.liquidatorGate == address(0)
        || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
require(
    callback == address(0)
        || market.liquidatorGate == address(0)
        || ILiquidatorGate(market.liquidatorGate).canLiquidate(callback),
    LiquidatorGatedFromLiquidating()
);
```
Alternatively, disallow `callback` addresses that differ from `msg.sender` in gated markets, or pass both `msg.sender` and `callback` to a single `canLiquidate(address caller, address payer)` interface.

## Proof of Concept
1. Deploy a market with a `liquidatorGate` that returns `true` for address A and `false` for address B.
2. Create a liquidatable borrower position.
3. Deploy `BCallback` implementing `ILiquidateCallback.onLiquidate` to return `CALLBACK_SUCCESS`; pre-approve Midnight for the required loan token amount from `BCallback`.
4. From address A, call `liquidate(market, ..., receiver=BCallback, callback=BCallback, ...)`.
5. Observe: gate check passes (A is whitelisted), collateral is transferred to `BCallback`, `onLiquidate` is called on `BCallback`, loan tokens are pulled from `BCallback`.
6. Assert: `BCallback` (gate-blocked) received collateral and paid loan tokens â€” full liquidation executed without gate approval.
