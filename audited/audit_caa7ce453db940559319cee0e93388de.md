## Title
Zero-`buyerAssets` rounding in `maxAssets`-mode buy-offer consumed accounting allows indefinite offer reuse ‚ÄĒ (File: src/Midnight.sol)

## Summary
When a buy offer is configured with `offer.maxAssets > 0`, the `take` function tracks capacity via `consumed[maker][group] += buyerAssets`, where `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `tickToPrice` always returns strictly less than `WAD` for every valid tick (0‚Äď5820), filling with `units = 1` yields `buyerAssets = 0`, so `consumed` never advances. The `ConsumedAssets` guard is permanently bypassed, and the offer can be filled an unlimited number of times, allowing the maker to accumulate unbounded credit units while transferring zero loan tokens.

## Finding Description

**Root cause ‚ÄĒ `src/Midnight.sol` lines 358‚Äď369:**

`offerPrice = tickToPrice(offer.tick)`. For `offer.buy = true`:

```solidity
uint256 sellerPrice = offerPrice - _settlementFee;
uint256 buyerPrice  = sellerPrice + _settlementFee; // == offerPrice
uint256 buyerAssets = units.mulDivDown(buyerPrice, WAD); // rounds DOWN
```

`mulDivDown` is `(x * y) / d` (plain integer division). With `units = 1` and `buyerPrice < WAD`:

```
buyerAssets = (1 * buyerPrice) / 1e18 = 0
```

**`tickToPrice` always returns < `WAD` ‚ÄĒ `src/libraries/TickLib.sol` lines 44‚Äď52:**

```solidity
return uint256(1e36)
    .divHalfDownUnchecked(1e18 + wExp(LN_ONE_PLUS_DELTA * (...)))
    .divHalfDownUnchecked(PRICE_ROUNDING_STEP) * PRICE_ROUNDING_STEP;
```

`wExp` always returns a strictly positive value, so the denominator `1e18 + wExp(...)` is always `> 1e18`, making the result always `< 1e18 = WAD`. After rounding to `PRICE_ROUNDING_STEP = 1e12`, the maximum representable price is `WAD - 1e12`. This holds for all 5821 valid ticks.

**The `maxAssets` guard is never triggered ‚ÄĒ `src/Midnight.sol` lines 367‚Äď369:**

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

With `buyerAssets = 0`, `consumed` stays permanently at 0. `require(0 <= maxAssets)` always passes regardless of `maxAssets`.

**Credit accumulates with each fill ‚ÄĒ `src/Midnight.sol` lines 382‚Äď414:**

```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt);
// ...
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt  += UtilsLib.toUint128(sellerDebtIncrease);
```

With `units = 1` and `buyerPos.debt = 0`: `buyerCreditIncrease = 1`. Each call gives the maker +1 credit and the taker +1 debt. `sellerAssets` is also 0 (same rounding), so both `safeTransferFrom` calls at lines 455‚Äď456 transfer 0 tokens, which succeed per the protocol's own token safety requirement (line 136: "it should not revert because a transfer is no-op").

**Protocol comment explicitly acknowledges this behavior ‚ÄĒ `src/Midnight.sol` line 94:**

```
/// @dev It is possible to give units to a fully consumed assets-based buy offer with price < 1.
```

This confirms the rounding behavior is known but does not assert it is safe or bounded.

**No guard prevents this:** There is no `require(units > 0)`, no `require(buyerAssets > 0)`, and no minimum-units check anywhere in the `take` path.

## Impact Explanation

The maker accumulates N credit units while paying zero loan tokens. `_marketState.totalUnits` increases by N (line 416‚Äď417) but `_marketState.withdrawable` does not, creating an undercollateralization of the pool. At maturity, the maker can withdraw N loan tokens from the pool that legitimate lenders deposited. The taker's N units of debt, if unpaid, become bad debt socialized among all lenders via the `lossFactor` slashing mechanism (lines 631‚Äď634). This constitutes direct theft from the protocol's loan token pool: the maker extracts loan tokens that legitimate lenders deposited, with the loss distributed across all lenders.

## Likelihood Explanation

**Preconditions:**
- Any buy offer with `maxAssets > 0` at any valid tick (all 5821 ticks are affected).
- A second address controlled by the maker to act as taker, supplying enough collateral to satisfy the `isHealthy` check at line 476.
- No special permissions, governance access, or oracle manipulation required.

**Feasibility:** Trivially repeatable in a loop via `multicall`. The only cost is gas and the collateral the taker must lock. The number of iterations is bounded only by the taker's collateral capacity (which can be made arbitrarily large by supplying more collateral). The maker's credit gain is unbounded relative to loan tokens paid.

## Recommendation

1. **Primary fix:** Add a check `require(offer.maxAssets == 0 || buyerAssets > 0 || units == 0, ZeroAssets())` before the consumed accounting block, or equivalently require `units.mulDivDown(buyerPrice, WAD) > 0` when `maxAssets > 0`.
2. **Alternative:** Track consumed in units (not assets) for buy offers too, or enforce a minimum `units` such that `buyerAssets >= 1` (i.e., `units >= WAD / buyerPrice + 1`).
3. **Supplementary:** Add `require(units > 0)` as a general guard at the top of `take` to prevent no-op fills that bypass all asset-based caps.

4. Taker calls `take(offer, ..., units=1, ...)` in a loop N times.
