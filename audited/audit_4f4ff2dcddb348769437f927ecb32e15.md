Audit Report

## Title
`liquidatorGate` access control bypassed via `callback`/`receiver` payer substitution - (File: src/Midnight.sol)

## Summary
The `liquidate` function checks `liquidatorGate.canLiquidate` only against `msg.sender`, but when `callback != address(0)` the actual loan token repayment is pulled from the `callback` address and collateral is sent to `receiver`. A whitelisted address A can call `liquidate(callback=B, receiver=B)` causing gate-blocked address B to be the true economic actor — paying loan tokens and receiving seized collateral — while the gate never inspects B.

## Finding Description
**Gate check** — [1](#0-0)  only `msg.sender` is evaluated:
```solidity
require(
    market.liquidatorGate == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
```

**Payer substitution** — [2](#0-1)  when `callback` is non-zero, the payer becomes the callback address:
```solidity
address payer = callback != address(0) ? callback : msg.sender;
```

**Collateral transfer** — [3](#0-2)  collateral is sent to `receiver`, which is caller-controlled and can be set to B.

**Loan token pull** — [4](#0-3)  tokens are pulled from `payer` (B), not `msg.sender` (A):
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```

**Exploit flow:**
1. Market has `liquidatorGate` that whitelists A and blocks B.
2. B deploys a contract implementing `ILiquidateCallback`, pre-approves Midnight to pull loan tokens from it.
3. A calls `midnight.liquidate(market, ..., receiver=B, callback=B, ...)`.
4. Gate calls `canLiquidate(A)` → `true` → passes.
5. Collateral transferred to B (`receiver`).
6. `onLiquidate` called on B; B returns `CALLBACK_SUCCESS`.
7. Loan tokens pulled from B (`payer=B`).
8. B has fully executed the liquidation despite being gate-blocked.

**Why existing checks fail:** The Certora rule `liquidatorGateBlocksLiquidation` [5](#0-4)  only proves that a call reverts when `canLiquidate(msg.sender)` is false — it does not constrain the callback address. The `OnlyExplicitPayerCanLoseTokens.spec` rule [6](#0-5)  explicitly sets `liquidateCallbackAllowed = true` for `liquidate`, confirming the callback-as-payer path is intentional but the gate gap is unaddressed.

## Impact Explanation
The `liquidatorGate` access control is fully circumvented for the actual economic actor. Any market relying on the gate for KYC, regulatory compliance, or trust-based liquidator restrictions has those restrictions rendered ineffective. B receives seized collateral and bears the loan repayment cost — the full economic outcome of a liquidation — without ever being checked by the gate. This constitutes unauthorized state change and unauthorized movement of assets (collateral) to a gate-blocked address.

## Likelihood Explanation
Requires whitelisted address A to cooperate with blocked address B. If A is a public/permissionless liquidation router contract (common in DeFi), any external user can supply `callback=B` and `receiver=B` as arguments — no collusion with A's operator is needed. Even with a cooperative EOA, the setup is low-friction: B deploys a callback contract, pre-approves Midnight, and asks any whitelisted address to submit the transaction. The condition is repeatable on every liquidatable position in every gated market.

## Recommendation
Extend the gate check to also validate the `callback` address (and optionally `receiver`) when they differ from `msg.sender`:

```solidity
require(
    market.liquidatorGate == address(0)
        || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
// Also gate the callback payer if it differs from msg.sender
if (callback != address(0)) {
    require(
        market.liquidatorGate == address(0)
            || ILiquidatorGate(market.liquidatorGate).canLiquidate(callback),
        LiquidatorGatedFromLiquidating()
    );
}
```

Alternatively, restrict `callback` and `receiver` to `msg.sender` when a `liquidatorGate` is set, or pass both `msg.sender` and `callback` to a multi-address `canLiquidate` interface.

## Proof of Concept
1. Deploy a market with a `liquidatorGate` that whitelists address A and blocks address B.
2. Create a borrower position that is liquidatable.
3. Deploy a contract at address B implementing `ILiquidateCallback.onLiquidate` that returns `CALLBACK_SUCCESS`, and pre-approve Midnight to pull loan tokens from B.
4. From address A, call `midnight.liquidate(market, collateralIndex, 0, repaidUnits, borrower, false, receiver=B, callback=B, "")`.
5. Observe: the call succeeds, B's loan tokens are consumed, B receives the seized collateral, and `canLiquidate` was never called on B.

A Foundry test can assert: (a) the call does not revert, (b) B's loan token balance decreases by `repaidUnits`, (c) B's collateral token balance increases by `seizedAssets`, and (d) `canLiquidate(B)` returns `false` on the gate — confirming the bypass.
