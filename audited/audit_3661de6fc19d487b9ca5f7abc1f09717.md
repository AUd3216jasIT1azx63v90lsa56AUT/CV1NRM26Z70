Audit Report

## Title
Fee-on-Transfer Collateral Token Enables Phantom Collateral Accounting and Undercollateralized Borrowing - ([File: src/Midnight.sol])

## Summary
`supplyCollateral` in `src/Midnight.sol` commits `_position.collateral[collateralIndex]` and the bitmap bit using the caller-supplied `assets` value before executing the ERC-20 transfer. `SafeTransferLib.safeTransferFrom` validates only that `transferFrom` does not revert and returns `true`, with no balance-before/after guard. A fee-on-transfer collateral token where the fee equals `assets` satisfies both checks while delivering zero tokens, leaving the protocol with phantom accounting that `isHealthy` treats as real collateral, enabling undercollateralized borrowing.

## Finding Description
**Root cause — checks-effects-interactions ordering with no received-amount guard:**

`supplyCollateral` (`src/Midnight.sol` lines 524–546):
- Line 533 writes `_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets)` using the caller-supplied `assets` before any transfer occurs.
- Lines 535–541 set the bitmap bit and enforce `MAX_COLLATERALS_PER_BORROWER`, also before the transfer.
- Line 545 is the actual transfer: `SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets)`.

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34):
- Calls `token.transferFrom(from, to, value)`, checks `success == true`, and decodes the return value as `bool`.
- No balance snapshot is taken before or after. A fee-on-transfer token with a 100% fee satisfies both checks while delivering 0 tokens.

`isHealthy` (`src/Midnight.sol` lines 944–960):
- Iterates the bitmap, reads `_position.collateral[i]` (the phantom value), and computes `maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(collateralParam.lltv, WAD)`.
- This produces a positive `maxDebt` from collateral that was never received, allowing the borrower to pass health checks and take debt.

**Exploit flow:**
1. Attacker deploys a fee-on-transfer ERC-20 token (fee = 100%) and creates a permissionless market with it as the collateral token.
2. Attacker calls `supplyCollateral(market, collateralIndex, assets, attacker)` with `assets > 0`.
3. Accounting and bitmap are committed; `transferFrom` succeeds (returns `true`) but delivers 0 tokens.
4. `_position.collateral[collateralIndex] == assets`, `token.balanceOf(address(this))` unchanged.
5. Attacker calls a borrow path; `isHealthy` computes positive `maxDebt` from the phantom value and approves the borrow.
6. Attacker receives real loan tokens backed by zero real collateral.

**Why existing checks fail:**
- `SafeTransferLib` has no received-amount guard (lines 24–34).
- Accounting is unconditionally committed before the external call (line 533 before line 545).
- Certora formal verification explicitly assumes well-behaved ERC-20 tokens (`certora/README.md` line 112), so `Solvency.spec` invariants are not proved against fee-on-transfer tokens.
- `touchMarket` validates LLTV tiers, sorted collateral addresses, and maxLif, but imposes no constraint on token transfer behavior.

## Impact Explanation
**Protocol insolvency / direct theft of lender funds.** `_position.collateral[collateralIndex]` records `assets` while the contract holds 0 tokens. `isHealthy` computes a positive `maxDebt` from this phantom value, allowing the borrower to take real loan-token debt with no real backing. When the phantom debt cannot be repaid, lenders bear the loss. This directly violates the core invariants stated in `live_context.json`: "ERC20 transfer deltas must match accounting deltas" and "collateral bitmap/list state must match actual deposited collateral." The impact class matches "protocol insolvency," "bad debt creation," and "direct loss of user funds" — all listed as highest-priority bug classes.

## Likelihood Explanation
Market creation is permissionless (`live_context.json`: "permissionless market creation"; `touchMarket` is `public`). An attacker can deploy a fee-on-transfer token and create a market with no admin involvement. `supplyCollateral` requires only `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`, which the attacker satisfies trivially. Fee-on-transfer tokens with configurable tax rates exist on mainnet; a 100% fee is an extreme but deployable configuration. Any fee > 0 causes partial accounting inflation. The attack is repeatable across collateral indices and markets. `live_context.json` explicitly lists "token charges fee" as a recommended fuzz axis and "fee-on-transfer" tokens as something that "should be tested if not explicitly excluded," confirming this is an in-scope attack surface.

## Recommendation
In `supplyCollateral`, record the contract's token balance before and after the transfer and use the actual received amount for accounting:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
require(received == assets, InsufficientTransfer());
```

Alternatively, update `_position.collateral[collateralIndex]` and the bitmap only after the transfer, using `received` instead of `assets`. The same pattern should be applied to any other function that calls `safeTransferFrom` and then uses the caller-supplied amount for accounting (e.g., the loan-token supply path at line 520).

## Proof of Concept
**Minimal Foundry test plan:**

1. Deploy a `MockFeeToken` implementing ERC-20 where `transferFrom` deducts 100% as a fee (transfers 0 tokens to recipient) but returns `true`.
2. Deploy `Midnight` and call `touchMarket` with `MockFeeToken` as `collateralParams[0].token`, a real ERC-20 as `loanToken`, valid LLTV, and a mock oracle returning a nonzero price.
3. Mint `MockFeeToken` to the attacker and approve `Midnight`.
4. Call `supplyCollateral(market, 0, 1e18, attacker)`.
5. Assert: `position[id][attacker].collateral[0] == 1e18` and `MockFeeToken.balanceOf(address(midnight)) == 0`.
6. Have a lender supply real loan tokens and publish an offer.
7. Have the attacker take the offer (borrow); assert `isHealthy` returns `true` and the attacker receives real loan tokens.
8. Assert `MockFeeToken.balanceOf(address(midnight)) == 0` throughout — confirming zero real collateral backs the debt.
