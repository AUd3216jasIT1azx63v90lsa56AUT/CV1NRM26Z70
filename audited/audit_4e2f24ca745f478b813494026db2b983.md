### Title
Unbounded returndata propagation in `SafeTransferLib` enables returndata-bomb OOG DoS on `take()` - (File: src/libraries/SafeTransferLib.sol)

### Summary
`SafeTransferLib.safeTransferFrom` propagates the token's full revert payload with no size cap via `assembly { revert(add(returndata, 0x20), mload(returndata)) }`. Because `touchMarket()` imposes no restriction on `loanToken`, an unprivileged market creator can register a market whose loan token returns a megabyte-scale revert payload, forcing any caller of `take()` to pay quadratic memory-expansion gas for the copy of that payload, causing an OOG revert.

### Finding Description
**Root cause — `SafeTransferLib.safeTransferFrom` (lines 27-31):**

```solidity
(bool success, bytes memory returndata) =
    token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
if (!success) {
    assembly ("memory-safe") {
        revert(add(returndata, 0x20), mload(returndata))   // no size cap
    }
}
```

When `token.call(...)` returns, the EVM copies the entire returndata into the caller's memory as part of allocating `bytes memory returndata`. Memory-expansion gas is paid by the **caller** (Midnight), not the token. The assembly block then re-reverts with the same uncapped blob, propagating it up the call stack.

**Permissionless market creation — `touchMarket()` (lines 755-791):**

`touchMarket` validates collateral params, LLTV tiers, maxLif, and maturity, but performs **zero validation on `market.loanToken`**. Any address with deployed bytecode is accepted.

**Reachable path in `take()` (lines 455-456):**

```solidity
SafeTransferLib.safeTransferFrom(
    offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(
    offer.market.loanToken, payer, receiver, sellerAssets);
```

`payer` resolves to `buyer` (the offer maker) when no `buyerCallback` is set (line 422). The attacker is the maker; the victim is the taker who calls `take()`.

**Exploit flow:**
1. Attacker deploys `MaliciousToken` whose `transferFrom()` executes `mstore` in a loop to fill memory with N bytes, then `revert(0, N)`.
2. Attacker calls `touchMarket(market)` with `market.loanToken = address(MaliciousToken)` — succeeds because no token validation exists.
3. Attacker signs a buy offer (`offer.buy = true`, `offer.maker = attacker`) in this market.
4. Victim taker calls `take(offer, ...)` with a gas budget sufficient for normal execution.
5. `safeTransferFrom(MaliciousToken, attacker, address(this), ...)` is called; `MaliciousToken.transferFrom` reverts with N bytes.
6. The EVM copies N bytes into `returndata` in Midnight's memory frame — memory expansion cost ≈ `N²/512/32²` gas, paid by the victim's transaction.
7. The assembly `revert` propagates the blob; victim's transaction OOGs.

**Why existing checks do not stop it:**
- `require(token.code.length > 0, NoCode())` — satisfied; the malicious token has code.
- All offer validity checks (signature, ratifier, tick, expiry, consumed) pass before the transfer.
- No returndata size limit anywhere in `SafeTransferLib`.

### Impact Explanation
Any caller of `take()` (or `repay`, `liquidate`, `supplyCollateral`, `flashLoan`) against a market whose `loanToken` is the malicious contract will OOG if their gas budget does not account for the inflated memory cost. Because the gas overhead is invisible from the ABI and unpredictable at estimation time, legitimate takers, liquidators, and repayers are reliably denied service. The attacker pays only deployment + `touchMarket` + offer-signing costs (a few hundred thousand gas, one-time).

### Likelihood Explanation
**Preconditions:**
- Attacker deploys one malicious token contract (trivial).
- Attacker calls `touchMarket` with that token (permissionless, no admin approval).
- Attacker signs a buy offer in that market (off-chain, zero on-chain cost).
- A victim must call `take()` on that offer — requires the victim to be deceived about the token's legitimacy (e.g., token address spoofed to look like USDC/WETH, or the offer is surfaced through an aggregator that does not verify `loanToken`).

The attack is repeatable at negligible marginal cost once the market is created. The social-engineering barrier is the primary limiting factor, but it is not a protocol-level protection.

### Recommendation
Cap the returndata size before propagating it in both `safeTransfer` and `safeTransferFrom`. A standard pattern:

```solidity
if (!success) {
    uint256 len = returndata.length;
    if (len > 256) len = 256;          // cap at 256 bytes
    assembly ("memory-safe") {
        revert(add(returndata, 0x20), len)
    }
}
```

Alternatively, revert with a fixed custom error instead of forwarding the token's payload, which also eliminates the gas-amplification vector entirely.

        assembly {
            // write 800 KB of zeros into memory starting at offset 0

            mstore(0, size)             // length slot (not used by revert)
            revert(0, size)
        }
    }
}
