### Title
Fee-on-Transfer Loan Token Causes Systematic Debt/Balance Deficit in `repay` - (File: src/Midnight.sol)

### Summary

`repay` reduces `position[id][onBehalf].debt` and increases `marketState[id].withdrawable` by exactly `units`, then calls `SafeTransferLib.safeTransferFrom` which only checks that `transferFrom` returns `true` — it never verifies the actual token balance received by the contract. When the market's `loanToken` is a fee-on-transfer token, the contract receives `units - fee` tokens while accounting records `units` as repaid, creating a permanent deficit between `withdrawable` and the contract's real token balance. The `onRepay` callback does not change this outcome; the deficit is structural and repeatable.

**Root cause:** `safeTransferFrom` validates only the boolean return of `transferFrom`. A fee-on-transfer token's `transferFrom(payer, midnight, units)` returns `true` and deducts `units` from the payer, but delivers only `units - fee` to the contract. Steps (1) and (2) have already committed `units` to accounting before the transfer, and there is no reconciliation afterward.

**Attacker inputs and exploit flow:**

1. Any unprivileged actor (market creator role is permissionless per `live_context.json`: `"arbitrary loan token"`) creates a market with a fee-on-transfer token as `loanToken`. `touchMarket` performs no token-type validation.
2. A borrower takes a sell offer, acquiring `units` of debt.
3. The borrower (or any authorized caller) calls `repay(market, units, onBehalf, address(0), "")`.
4. `position[id][onBehalf].debt` decreases by `units`; `marketState[id].withdrawable` increases by `units`.
5. `safeTransferFrom(feeToken, msg.sender, midnight, units)` executes: the token transfers `units - fee` to Midnight and returns `true`. No revert occurs.
6. Midnight's actual token balance increased by `units - fee`, but `withdrawable` increased by `units`. Deficit = `fee`.

**With callback (alternative path, same outcome):** When `callback != address(0)`, `payer = callback`. The callback's `onRepay` can source `units` tokens (e.g., via a flash loan of a non-fee token, swapped to the fee token). After `onRepay` returns, step (3) executes identically — the fee-on-transfer token still delivers only `units - fee`. The callback changes who holds the tokens, not how many arrive.

**Why existing checks fail:**

- `safeTransferFrom` checks only the `bool` return value, not `balanceOf(address(this))` before and after.
- `touchMarket` has no token allowlist or fee-on-transfer exclusion.
- The Certora `WithdrawableMonotonicity` spec models `safeTransferFrom` as a no-op (no balance tracking), so formal verification does not cover this class of token.
- The `live_context.json` core invariants explicitly flag: `"ERC20 transfer deltas must match accounting deltas"` and `"fee-on-transfer … should be tested if not explicitly excluded"` — confirming no explicit exclusion exists.

### Impact Explanation

Each `repay` call with a fee-on-transfer loan token leaves `marketState[id].withdrawable` overstated by `fee` relative to the contract's actual token balance. Lenders holding credit units attempt to `withdraw`, which calls `safeTransfer(loanToken, receiver, units)`. The last lenders to withdraw will find the contract balance insufficient, causing their withdrawals to revert. The shortfall is permanent and grows with each repayment. This is direct protocol insolvency / bad debt borne by lenders.

### Likelihood Explanation

**Preconditions:**
- A market must exist with a fee-on-transfer token as `loanToken`. Market creation is permissionless and accepts any token address.
- The borrower must repay using that token (which is the only token accepted for repayment in that market).

**Feasibility:** Fee-on-transfer tokens are a legitimate and deployed token design (e.g., tokens with built-in redistribution or burn mechanics). No special privilege is required. The attack is repeatable on every `repay` call, compounding the deficit. The `liquidate` function has the identical pattern at line 717 (`SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits)`) and is equally affected.

### Recommendation

Add a balance-before/after check in `repay` (and `liquidate`) to enforce that the contract received exactly `units` tokens:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
require(IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore == units, InsufficientTokensReceived());
```

Alternatively, explicitly document and enforce (via `touchMarket` validation) that fee-on-transfer tokens are not supported as loan tokens.

        uint256 net = amount - FEE;

        return super.transferFrom(from, to, net); // delivers amount-FEE, returns true

function testFeeOnTransferRepayDeficit() public {
    FeeOnTransferLoanToken feeToken = new FeeOnTransferLoanToken();
    // Create market with feeToken as loanToken (permissionless)
    Market memory mkt = buildMarket(address(feeToken), ...);
    midnight.touchMarket(mkt);
    // Setup: lender supplies credit, borrower takes debt of `units`
    uint256 units = 1000;
    setupBorrow(mkt, borrower, units);

    uint256 withdrawableBefore = midnight.withdrawable(id);
    uint256 contractBalBefore = feeToken.balanceOf(address(midnight));

    vm.prank(borrower);
    feeToken.approve(address(midnight), units);
    vm.prank(borrower);
    midnight.repay(mkt, units, borrower, address(0), "");

    // Assertions that expose the bug:
    assertEq(midnight.debtOf(id, borrower), 0);                                    // debt fully cleared
    assertEq(midnight.withdrawable(id), withdrawableBefore + units);               // withdrawable = units

    assertEq(actualReceived, units - FEE);                                         // only units-FEE received
    // INVARIANT VIOLATION: withdrawable > actual balance
    assertGt(midnight.withdrawable(id), feeToken.balanceOf(address(midnight)));
