### Title
Lenders Can Front-Run Bad Debt Realization to Avoid Proportional Credit Loss — (File: src/Midnight.sol)

---

### Summary

Morpho Midnight socializes bad debt among lenders by updating `marketState.lossFactor` inside `liquidate()`. This loss is applied lazily to each lender's `credit` only when `_updatePosition()` is called. Because `withdraw()` calls `_updatePosition()` with the **current** (pre-bad-debt) `lossFactor`, a lender who observes a pending `liquidate()` transaction in the mempool can front-run it with a `withdraw()` call, extracting their credit at full face value before the loss is recorded. The remaining lenders absorb a disproportionately larger share of the bad debt.

---

### Finding Description

**Root cause — lazy loss application decoupled from withdrawal gate:**

`_updatePosition()` computes slashed credit using `marketState[id].lossFactor` at call time: [1](#0-0) 

If `liquidate()` has not yet executed, `lossFactor` still reflects the pre-bad-debt state, so `postSlashCredit == credit` (no reduction). `withdraw()` calls `_updatePosition()` first, then immediately transfers tokens: [2](#0-1) 

The only gate preventing withdrawal is the `withdrawable` counter, which is incremented by `repay()` and `liquidate()` (for the repaid portion): [3](#0-2) [4](#0-3) 

In any active market where borrowers have been repaying, `withdrawable > 0`. This is the normal operating state.

**Bad debt realization path (the event being front-run):** [5](#0-4) 

`lossFactor` is updated here, after which every remaining lender's credit is proportionally reduced at their next interaction. A lender who has already withdrawn is no longer in `totalUnits` and takes zero loss.

**Exploit flow:**

1. Market has `withdrawable = W > 0` (from prior repayments).
2. A borrower becomes undercollateralized; a liquidator submits `liquidate()` to the mempool.
3. Lender L (holding `credit = C`, representing fraction `C / totalUnits` of the pool) observes the pending transaction.
4. L front-runs with `withdraw(units = min(C, W))`. `_updatePosition()` applies the old `lossFactor` — no credit reduction. L receives `min(C, W)` loan tokens.
5. Liquidation executes: `lossFactor` is updated using the now-smaller `totalUnits` (L's credit already removed), so remaining lenders absorb a larger proportional loss.

---

### Impact Explanation

- **Direct financial loss to remaining lenders.** The bad debt that should have been shared proportionally across all lenders is instead concentrated on those who did not (or could not) front-run.
- **Violated invariant:** `live_context.json` line 191 states: *"bad debt must reduce lender credit exactly once and proportionally."* Front-running breaks the "proportionally" guarantee.
- **Severity:** High — direct, quantifiable loss of funds for honest lenders; no privileged access required. [6](#0-5) 

---

### Likelihood Explanation

- **Precondition:** `withdrawable > 0`. This is the normal state of any market that has seen any repayment activity.
- **Attacker capability:** Standard mempool observation (no privileged keys, no oracle manipulation). Any lender can do this.
- **Incentive:** The larger the bad debt event relative to the lender's credit share, the larger the gain from front-running.
- **Likelihood:** High in practice — bad debt events in lending protocols are visible on-chain before mining, and MEV bots routinely monitor mempools for exactly this class of opportunity.

---

### Recommendation

Apply the bad debt loss **before** allowing withdrawal, or prevent withdrawal during the same block as bad debt realization. Concretely:

- **Option A:** Before decrementing `withdrawable` in `withdraw()`, check that `_updatePosition()` has been called with the latest `lossFactor`. This is already done, but the issue is that `lossFactor` is stale until `liquidate()` runs. A snapshot-based approach (e.g., recording a "pending bad debt" flag or amount before the liquidation settles) would close the window.
- **Option B:** In `liquidate()`, when bad debt is realized, immediately slash all lenders' credits (eager update). This is gas-intensive but eliminates the race condition entirely.
- **Option C:** Restrict `withdraw()` so that `withdrawable` is only usable after a cooldown block following any bad debt event, preventing same-block or front-run extraction.

### Proof of Concept

```
Setup:
  - Market with loanToken = USDC, one collateral
  - totalUnits = 1000, withdrawable = 400 (from prior repayments)
  - Lender A: credit = 200 (20% of pool)
  - Lender B: credit = 800 (80% of pool)
  - Borrower X: debt = 500, collateral now worth 300 → badDebt = 200

Step 1 (expected, no front-run):
  liquidate(borrower=X, seizedAssets=0, repaidUnits=0)
    → badDebt = 200
    → new lossFactor updated: remaining totalUnits = 800
    → Lender A's credit at next interaction: 200 * (800/1000) = 160  (loses 40)
    → Lender B's credit at next interaction: 800 * (800/1000) = 640  (loses 160)

Step 2 (attack, with front-run):
  Lender A sees liquidate() in mempool, sends withdraw(units=200) with higher gas:
    → _updatePosition() uses OLD lossFactor → credit unchanged = 200
    → withdrawable: 400 - 200 = 200 ✓ (no revert)
    → totalUnits: 1000 - 200 = 800
    → Lender A receives 200 USDC, exits with zero loss

  liquidate(borrower=X) now executes:
    → badDebt = 200, _totalUnits = 800
    → lossFactor updated against 800 units
    → Lender B's credit: 800 * (600/800) = 600  (loses 200, the FULL bad debt)

Result: Lender A avoided 40 USDC loss; Lender B absorbs 200 instead of 160.
