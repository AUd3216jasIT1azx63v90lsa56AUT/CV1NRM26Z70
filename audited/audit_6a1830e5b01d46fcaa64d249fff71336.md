Audit Report

## Title
`buyerAssetsToUnits` division-by-zero DoS in `buyWithAssetsTargetAndWithdrawCollateral` when `tick = 0` and `settlementFee = 0` - (File: src/periphery/TakeAmountsLib.sol)

## Summary
`TickLib.tickToPrice(0)` evaluates to exactly `0` due to integer rounding in the two-step `divHalfDownUnchecked` chain. When a sell offer has `tick = 0` and effective `settlementFee = 0`, `buyerAssetsToUnits` computes `buyerPrice = 0`, passes the `require(buyerPrice <= WAD)` guard (which has no zero-check), and then panics with division-by-zero at `targetBuyerAssets.mulDivDown(WAD, 0)`. Because `MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` calls `buyerAssetsToUnits` outside any try/catch, the panic propagates and reverts the entire bundler transaction, while `Midnight.take` on the same offer succeeds.

## Finding Description

**Root cause — `tickToPrice(0) = 0`:** [1](#0-0) 

With `tick = 0`:
- `LN_ONE_PLUS_DELTA * (2910 - 0) ≈ 14.514e18`
- `wExp(14.514e18) ≈ 2.009e24`
- `1e36 / (1e18 + 2.009e24) ≈ 4.978e11`
- `divHalfDownUnchecked(4.978e11, 1e12)` = `(497800000000 + 499999999999) / 1e12` = `997799999999 / 1e12` = `0` (integer division, result < 1)
- `0 * PRICE_ROUNDING_STEP = 0`

**Root cause — missing zero-guard in `buyerAssetsToUnits`:** [2](#0-1) 

With `offer.buy = false`, `tick = 0`, `settlementFee = 0`:
1. `offerPrice = 0`
2. `sellerPrice = offerPrice = 0` (sell branch)
3. `buyerPrice = 0 + 0 = 0`
4. `require(0 <= WAD)` — passes (no zero-guard)
5. `targetBuyerAssets.mulDivDown(WAD, 0)` — **panic: division by zero**

**Root cause — `buyerAssetsToUnits` is outside the try/catch in the bundler:** [3](#0-2) [4](#0-3) 

The `TakeAmountsLib.buyerAssetsToUnits(...)` call at lines 209–211 is computed before the `try IMidnight(MIDNIGHT).take(...)` block at lines 215–221. A panic inside `buyerAssetsToUnits` is not caught and propagates to revert the entire transaction.

**Divergence with `Midnight.take`:** For a sell offer at `tick = 0`, `Midnight.take` computes `buyerAssets = units.mulDivUp(0, WAD) = 0` — no division, no revert. The bundler and core contract disagree on whether this offer is takeable.

## Impact Explanation
Any lender calling `buyWithAssetsTargetAndWithdrawCollateral` with a `takes` array containing a sell offer at `tick = 0` (and effective `settlementFee = 0`) will have their entire bundler transaction reverted. The pre-pulled `targetBuyerAssets` tokens are returned (no permanent fund loss), but the bundler flow is completely DoS'd for that call. The lender cannot use this bundler function to fill zero-price offers, even though `Midnight.take` accepts them. The `sellerAssetsToUnits` path in `supplyCollateralAndSellWithAssetsTarget` has an analogous issue when `sellerPrice = 0`. [5](#0-4) 

## Likelihood Explanation
Preconditions: (1) a sell offer at `tick = 0` — valid since `0 <= MAX_TICK` and `0 % tickSpacing == 0` for any spacing; (2) effective `settlementFee = 0`. Both are reachable permissionlessly. A borrower can create the offer without any privileged access. Any lender who then includes it in a `buyWithAssetsTargetAndWithdrawCollateral` call triggers the revert unconditionally. The condition is stable and repeatable. [6](#0-5) 

## Recommendation
Add a zero-guard for `buyerPrice` in `buyerAssetsToUnits` before the division:

```solidity
require(buyerPrice != 0, ZeroPrice());
require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());
```

Similarly add a zero-guard for `sellerPrice` in `sellerAssetsToUnits`. Alternatively, handle `buyerPrice == 0` as a special case returning `type(uint256).max` (infinite units needed), which would cause `unitsToTake` to be clamped to `takes[i].units` or `consumableUnits`, matching the behavior of `Midnight.take`. [7](#0-6) 

## Proof of Concept
1. Deploy Midnight with `settlementFee = 0`.
2. Borrower creates a sell offer with `tick = 0`.
3. Lender calls `buyWithAssetsTargetAndWithdrawCollateral` with `takes = [{offer: sellOffer_tick0, ...}]` and any `targetBuyerAssets > 0`.
4. Observe: transaction reverts with panic code `0x12` (division by zero) inside `buyerAssetsToUnits`.
5. Confirm: calling `IMidnight.take(sellOffer_tick0, ..., 1, ...)` directly succeeds and returns `(0, 0)`.

Unit test sketch:
```solidity
// tickToPrice(0) == 0
assertEq(TickLib.tickToPrice(0), 0);
// buyerAssetsToUnits reverts
vm.expectRevert(stdError.divisionError);
TakeAmountsLib.buyerAssetsToUnits(midnight, id, sellOffer_tick0, 1e18);
// Midnight.take succeeds
(uint256 ba, uint256 sa) = IMidnight(midnight).take(sellOffer_tick0, "", 1, taker, address(0), address(0), "");
assertEq(ba, 0); assertEq(sa, 0);
```

    function buyerAssetsToUnits(address midnight, bytes32 id, Offer memory offer, uint256 targetBuyerAssets)
        internal
        view
        returns (uint256)
    {
        uint256 offerPrice = TickLib.tickToPrice(offer.tick);
        uint256 settlementFee =
            IMidnight(midnight).settlementFee(id, UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp));
        // Mirrors Midnight's computation to revert if offerPrice < settlementFee in case of a buy offer.
        uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
        uint256 buyerPrice = sellerPrice + settlementFee;
        require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());
        return offer.buy ? targetBuyerAssets.mulDivUp(WAD, buyerPrice) : targetBuyerAssets.mulDivDown(WAD, buyerPrice);
    }
