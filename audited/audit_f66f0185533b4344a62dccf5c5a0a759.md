Audit Report

## Title
`onLiquidate` callback enables post-bad-debt `supplyCollateral`, causing permanent lender over-slashing - (File: src/Midnight.sol)

## Summary
In `liquidate()`, bad debt is fully computed from a local bitmap snapshot and immediately committed to `lossFactor`/`totalUnits` before the `onLiquidate` callback fires. A callback authorized for the borrower can call `supplyCollateral` during this window, injecting collateral that was never counted in the bad-debt formula. Lenders are permanently slashed for a shortfall larger than the true post-liquidation shortfall, with no mechanism to reverse the over-accounting.

## Finding Description

**Step 1 — Bitmap snapshot (line 606):**

The bad-debt loop iterates over a local copy of `_position.collateralBitmap`. Any collateral index not set in this snapshot at the moment of the snapshot is invisible to the loop. [1](#0-0) 

**Step 2 — Bad debt committed before any external call (lines 626–641):**

All lender-slashing state (`lossFactor`, `totalUnits`, `continuousFeeCredit`) is finalized here. No external call has occurred yet. [2](#0-1) 

**Step 3 — `onLiquidate` callback fires (lines 698–715):**

At this point, `lossFactor` and `totalUnits` are already written. The callback executes with full re-entry capability into Midnight. There is no reentrancy guard on `liquidate()` or `supplyCollateral()`. [3](#0-2) 

**Step 4 — `supplyCollateral` called inside callback (lines 524–546):**

Authorization is satisfied if `callback == borrower` (self-liquidation: `onBehalf == msg.sender`) or if the borrower pre-authorized the callback via `setIsAuthorized`. The new collateral and its bitmap bit are written to storage, but the bad-debt loop already completed using the old snapshot — these writes have no retroactive effect on the committed `badDebt`. [4](#0-3) 

**Step 5 — No re-check after callback (line 717):**

With `repaidUnits == 0`, zero tokens are pulled. The function returns without re-evaluating collateral or bad debt. [5](#0-4) 

**Why existing guards fail:**

- `liquidationLocked` is set only in `take()` via `UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`. It is never set inside `liquidate()`. The check at line 621 only guards against liquidating a borrower who is mid-`take()` callback — it does not block `supplyCollateral` from being called during `onLiquidate`. [6](#0-5) [7](#0-6) 

- There is no reentrancy guard on `liquidate()` or `supplyCollateral()`. [8](#0-7) 

- `atMostOneNonZero(repaidUnits, seizedAssets)` permits both to be zero (since `or(iszero(0), iszero(0)) == 1`), enabling a pure bad-debt realization with no token transfer required. This is explicitly documented in the protocol comments. [9](#0-8) [10](#0-9) 

## Impact Explanation

The over-slashing amount per attack is proportional to the collateral amount `A` supplied in the callback. This quantity is subtracted from `totalUnits` and added to `lossFactor` in excess of the true shortfall. Because `lossFactor` is used in `_updatePosition` to proportionally reduce every lender's credit, every lender who calls `updatePosition` after the attack permanently loses additional credit proportional to the inflated `lossFactor`. The loss is irreversible — there is no mechanism to reduce `lossFactor` or restore `totalUnits`. [11](#0-10) 

Additionally, the attacker's own debt is reduced by the full `badDebt` (computed without the injected collateral), while the injected collateral remains in their position and can be withdrawn once the position is healthy. This constitutes direct, permanent theft of lender value.

## Likelihood Explanation

**Required preconditions:**
1. The borrower position has genuine bad debt (oracle price drop below `1/maxLif` coverage) — the normal trigger for bad-debt realization.
2. The borrower is a smart contract implementing `ILiquidateCallback`, or has pre-authorized a callback contract via `setIsAuthorized`. Protocol-integrated borrowers (vaults, aggregators) routinely satisfy this.
3. The callback contract holds collateral tokens to supply — obtainable via flash loan within the same callback since `supplyCollateral` pulls from `msg.sender` and no repayment is required within `liquidate()`.

Self-liquidation (borrower == liquidator == callback) requires zero third-party cooperation. The attack is repeatable on any market with bad debt and a callback-capable borrower, and can be executed atomically in a single transaction.

## Recommendation

Move the `onLiquidate` callback invocation to **before** the bad-debt accounting block (lines 626–641), or re-compute bad debt **after** the callback returns and use the post-callback collateral state for the final `lossFactor`/`totalUnits` update. Alternatively, add a reentrancy lock (e.g., transient storage flag) set at the entry of `liquidate()` and checked in `supplyCollateral()` to prevent collateral injection during an active liquidation.

## Proof of Concept

**Minimal self-liquidation PoC (single transaction):**

1. Deploy `AttackerBorrower` implementing `ILiquidateCallback`. The contract holds a position with genuine bad debt in market `M`.
2. `AttackerBorrower` calls `liquidate(market, collateralIndex, 0, 0, address(this), false, address(this), address(this), data)`.
3. Inside `liquidate()`, bad debt `D` is computed from the current collateral bitmap and committed: `lossFactor` increases, `totalUnits` decreases by `D`.
4. `onLiquidate` fires on `AttackerBorrower`. Inside the callback:
   - Flash-borrow `A` collateral tokens.
   - Call `supplyCollateral(market, collateralIndex, A, address(this))` — authorized because `onBehalf == msg.sender`.
   - Repay flash loan (or use pre-held tokens).
5. `liquidate()` returns. `repaidUnits == 0`, so no loan tokens are transferred.
6. **Observe:** `marketState[id].lossFactor` is inflated by the amount corresponding to `A * price / maxLif` units of bad debt that did not actually exist. All lenders' credits are permanently reduced at their next `updatePosition` call.
7. `AttackerBorrower` calls `withdrawCollateral` to recover the injected `A` collateral (position is now healthy due to debt reduction).

**Invariant fuzz test:** Assert that after any sequence of `liquidate` + `supplyCollateral` calls, `sum(lender credits) + sum(borrower debts) <= totalUnits` holds. The attack violates this invariant by inflating the `lossFactor` beyond the true bad debt.
