### Title
`isRatified` reverts with `LeafIndexOutOfRange` instead of `InvalidProof` when `leafIndex` has bits set above `proof.length` - (File: src/ratifiers/interfaces/IEcrecoverRatifier.sol)

### Summary
`EcrecoverRatifier.isRatified` wraps the `HashLib.isLeaf` call with `require(..., InvalidProof())`, intending that all invalid-proof inputs revert with `InvalidProof`. However, when `leafIndex >> proof.length != 0`, `HashLib.isLeaf` itself reverts with `LeafIndexOutOfRange` before returning, and Solidity's `require(expr, CustomError())` does not catch reverts from within `expr` — it only emits `CustomError()` on a `false` return. The inner revert propagates unchanged, so callers observe `LeafIndexOutOfRange` instead of `InvalidProof`.

### Finding Description
**Code path:**

`EcrecoverRatifier.isRatified` (line 37):
```solidity
require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
``` [1](#0-0) 

`HashLib.isLeaf` (line 58):
```solidity
require(leafIndex >> proof.length == 0, LeafIndexOutOfRange());
``` [2](#0-1) 

**Root cause:** Solidity's `require(f(), CustomError())` syntax only substitutes `CustomError()` when `f()` returns `false`. If `f()` itself reverts, the inner revert propagates up the call stack unmodified. `HashLib.isLeaf` reverts (not returns `false`) when `leafIndex >> proof.length != 0`.

**Attacker-controlled inputs:** `ratifierData` is fully attacker-supplied. It is `abi.decode`d into `(sig, root, leafIndex, proof)` with no pre-validation. [3](#0-2) 

**Exploit flow:**
1. Taker calls `midnight.take(offer, ratifierData, ...)` where `ratifierData` encodes `leafIndex = 2`, `proof.length = 1`.
2. Midnight calls `EcrecoverRatifier.isRatified`.
3. `isRatified` calls `HashLib.isLeaf(..., 2, proof)`.
4. Inside `isLeaf`: `2 >> 1 == 1 != 0` → `revert LeafIndexOutOfRange()`.
5. The revert bubbles through `require(HashLib.isLeaf(...), InvalidProof())` unchanged.
6. Caller receives `LeafIndexOutOfRange` selector, not `InvalidProof`.

**Existing protections:** None. The `require(msg.sender == MIDNIGHT)` guard is irrelevant here — Midnight itself calls `isRatified` during `take`, so the path is fully reachable. There is no pre-check on `leafIndex` before the `isLeaf` call.

The same divergence exists in `SetterRatifier.isRatified` at line 34 for the same reason. [4](#0-3) 

### Impact Explanation
No funds are lost or frozen. The transaction correctly reverts for an invalid proof. The only concrete impact is that the error selector observed by the caller is `HashLib.LeafIndexOutOfRange` instead of `IEcrecoverRatifier.InvalidProof`, diverging from the documented/expected interface contract. Off-chain tooling, SDKs, or integrators that match on `InvalidProof` to detect bad-proof submissions will fail to classify this revert correctly.

### Likelihood Explanation
Trivially reachable: any taker can craft `ratifierData` with `leafIndex = 2` and `proof` of length 1 (or any `leafIndex` with bits set above `proof.length`). No special state, no privileged access, no prior setup required. Fully repeatable on every call.

### Recommendation
Add an explicit bounds check in `isRatified` before calling `isLeaf`, and revert with `InvalidProof` if the index is out of range:

```solidity
require(leafIndex >> proof.length == 0, InvalidProof());
require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
```

Alternatively, remove the `LeafIndexOutOfRange` revert from `HashLib.isLeaf` and instead return `false` when `leafIndex >> proof.length != 0`, so the outer `require(..., InvalidProof())` handles it uniformly. Apply the same fix to `SetterRatifier.isRatified`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity 0.8.34;

import {Test} from "forge-std/Test.sol";
import {EcrecoverRatifier} from "src/ratifiers/EcrecoverRatifier.sol";
import {IEcrecoverRatifier, Signature} from "src/ratifiers/interfaces/IEcrecoverRatifier.sol";
import {HashLib} from "src/ratifiers/libraries/HashLib.sol";
import {Offer} from "src/interfaces/IMidnight.sol";

contract LeafIndexDivergenceTest is Test {
    EcrecoverRatifier ratifier;
    address midnight = address(0xBEEF);

    function setUp() public {
        ratifier = new EcrecoverRatifier(midnight);
    }

    function testLeafIndexOutOfRangeNotInvalidProof() public {
        Offer memory offer; // zero-value offer

        // leafIndex=2, proof.length=1: 2 >> 1 == 1 != 0 → triggers LeafIndexOutOfRange
        bytes32[] memory proof = new bytes32[](1);
        proof[0] = bytes32(0);

        Signature memory sig = Signature({v: 27, r: bytes32(0), s: bytes32(0)});
        bytes memory ratifierData = abi.encode(sig, bytes32(0), uint256(2), proof);

        vm.prank(midnight);
        // Assert: reverts with LeafIndexOutOfRange, NOT InvalidProof
        vm.expectRevert(HashLib.LeafIndexOutOfRange.selector);
        ratifier.isRatified(offer, ratifierData);

        // Confirm InvalidProof selector is different
        assertTrue(
            HashLib.LeafIndexOutOfRange.selector != IEcrecoverRatifier.InvalidProof.selector
        );
    }
}
```

Expected: `vm.expectRevert(HashLib.LeafIndexOutOfRange.selector)` passes, confirming the error selector diverges from `IEcrecoverRatifier.InvalidProof.selector`. Changing the expectation to `IEcrecoverRatifier.InvalidProof.selector` would cause the test to fail, proving the divergence.
