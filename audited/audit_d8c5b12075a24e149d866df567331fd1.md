### Title
Oracle Reentrancy in `liquidate()` Bitmap Loop Enables Unauthorized Collateral Withdrawal with Corrupted Health Accounting - (File: src/Midnight.sol)

### Summary

The `liquidate()` bitmap loop calls `IOracle.price()` as an external call (line 610) before the `NotLiquidatable` check and without any reentrancy guard or liquidation lock being set. An attacker who controls the oracle for collateral index `i` can reenter `withdrawCollateral()` for a higher-index collateral `j` (already valued in `maxDebt`) during that oracle call, causing `maxDebt` to be computed with `position.collateral[j]` while the actual storage value is zeroed out, and allowing the attacker to extract `j`'s tokens while the outer liquidation proceeds on corrupted accounting.

### Finding Description

**Exact code path:**

`liquidate()` at `src/Midnight.sol:607-618` takes a local snapshot of `_collateralBitmap` (line 606) and iterates MSB-first. For each index `i`, it calls `IOracle(_collateralParam.oracle).price()` (line 610) — an unconstrained external call — and only *after* that call reads `_position.collateral[i]` from storage (line 612). No liquidation lock is set before this loop; `liquidationLocked` is only *read* at line 621, after the loop completes.

`withdrawCollateral()` at `src/Midnight.sol:549-573` has no reentrancy guard and no liquidation-lock check. Its only safety net is `require(isHealthy(...))` at line 568, which itself calls `IOracle.price()` on every activated collateral.

**Root cause:** The liquidation lock (`LIQUIDATION_LOCK_SLOT`) is never written inside `liquidate()`. It is only written in `take()` (line 444) to protect the seller during take callbacks. The oracle call at line 610 is therefore a fully open external call that can reenter any Midnight entry point.

**Attacker inputs and exploit flow:**

Preconditions:
- Attacker is market creator and borrower.
- Market has collateral `j` (high index, valuable token, honest oracle) and collateral `i` (low index, `i < j`, any token, attacker-controlled malicious oracle).
- Borrower has `C_j` valuable tokens at index `j` and `C_i` tokens at index `i`.
- Borrower has debt `D` where `D > C_j · price_j · lltv_j / (ORACLE_PRICE_SCALE · WAD)` (position is unhealthy with oracle-i returning 0).
- Borrower has called `setIsAuthorized(maliciousOracle, true, borrower)`.

Attack steps:

1. Attacker calls `liquidate(market, collateralIndex=i, seizedAssets=C_i, repaidUnits=0, borrower, false, ...)`.
2. Bitmap loop processes `j` first (MSB): oracle for `j` returns `price_j`; `_position.collateral[j]` is read as `C_j`; `maxDebt += C_j · price_j · lltv_j`.
3. Bitmap loop processes `i`: calls `maliciousOracle.price()`.
4. Inside `maliciousOracle.price()`, the oracle reenters `withdrawCollateral(market, j, C_j, borrower, attackerWallet)`:
   - `_position.collateral[j]` is set to 0 in storage (line 562).
   - `_position.collateralBitmap` clears bit `j` (line 565).
   - `isHealthy()` is called (line 568): reads the updated bitmap (without `j`), calls malicious oracle again (recursive call), which returns `price_i_high` large enough so `C_i · price_i_high · lltv_i ≥ D` → returns `true`.
   - `C_j` valuable tokens are transferred to `attackerWallet` (line 572). ✓
5. Malicious oracle returns `price_i_low = 0` to the `liquidate()` loop.
6. `_position.collateral[i]` is read as `C_i`; `maxDebt += 0`.
7. Loop ends: `maxDebt = C_j · price_j · lltv_j` (computed using the now-withdrawn `C_j`).
8. `NotLiquidatable` check (line 622): `D > maxDebt` → passes (by precondition).
9. `badDebt = D - C_j · price_j / maxLif_j > 0` is realized; `_position.debt` is reduced by `badDebt`; lenders are slashed.
10. Collateral `i` (worthless) is seized; `repaidUnits = 0` (since `liquidatedCollatPrice = 0`).

**Why existing checks fail:**

- The liquidation lock is never set in `liquidate()`, so `withdrawCollateral()` has no way to detect it is being called mid-liquidation.
- The `isHealthy()` check inside `withdrawCollateral()` is defeated because the malicious oracle returns an arbitrarily high price on the recursive call, making the position appear healthy without `j`.
- The `NotLiquidatable` check uses the stale `maxDebt` that still includes `j`'s contribution, so it passes even though `j` is gone.

### Impact Explanation

An attacker who is both market creator and borrower can withdraw valuable collateral `j` during a liquidation — bypassing the health check — while the liquidation proceeds on corrupted `maxDebt` accounting. The attacker recovers `C_j` tokens without repaying debt, and the residual debt is socialized as bad debt among lenders. This is unauthorized collateral withdrawal during liquidation with incorrect health accounting, directly matching the scoped impact.

### Likelihood Explanation

Requires the attacker to be both market creator (permissionless) and borrower, to deploy a malicious oracle, and to pre-authorize it. All preconditions are self-inflicted by the attacker with no external cooperation needed. The attack is repeatable across any market where the attacker controls an oracle for any collateral index lower than another activated collateral. The only constraint is that the position must be genuinely unhealthy (debt exceeds `j`'s collateral value alone), which the attacker can engineer by borrowing the maximum against `j` and supplying worthless `i`.

### Recommendation

Set the liquidation lock for the borrower at the start of `liquidate()`, before the bitmap loop, and add a check in `withdrawCollateral()` that reverts if the liquidation lock is set for `onBehalf`. Concretely:

In `liquidate()`, before line 607:
```solidity
bool wasLocked = UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, borrower, true);
```
And restore it after the function (or use a try/finally pattern via a modifier).

In `withdrawCollateral()`, after the authorization check:
```solidity
require(!liquidationLocked(id, onBehalf), LiquidationLocked());
```

Alternatively, apply a general reentrancy guard (transient-storage-based) to all state-mutating entry points.

    address borrower;
    address attackerWallet;

        if (!inLiquidateLoop) {
            inLiquidateLoop = true;

function testOracleReentrancyWithdrawCollateral() public {
    // Setup: market with j=1 (honest oracle), i=0 (malicious oracle)

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);

                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
