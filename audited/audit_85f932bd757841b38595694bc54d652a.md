### Title
Fee-on-transfer loan token causes `claimableSettlementFee` over-credit, making fee claimer unable to withdraw - (`src/Midnight.sol`)

### Summary
In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full spread (`buyerAssets - sellerAssets`) at line 418 before the actual token transfer at line 455. `SafeTransferLib.safeTransferFrom` only verifies the boolean return value of `transferFrom`; it does not measure the amount actually received by `address(this)`. When the loan token is a fee-on-transfer token, Midnight receives `spread - tax` while `claimableSettlementFee` is credited the full `spread`, permanently breaking the invariant that `claimableSettlementFee[token] ≤ actual balance attributable to fees`.

### Finding Description
**Code path:**

`src/Midnight.sol` line 418 (state write, before transfer):
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

`src/Midnight.sol` line 455 (actual inbound transfer):
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
``` [2](#0-1) 

`src/libraries/SafeTransferLib.sol` lines 24–34: `safeTransferFrom` calls `transferFrom` and checks only the boolean return value. There is no before/after balance check and no received-amount verification. [3](#0-2) 

**Root cause:** The accounting write (`claimableSettlementFee +=`) is unconditionally based on the nominal transfer amount, not the amount actually received. For a fee-on-transfer token with tax rate `t`, Midnight receives `spread × (1 - t)` but records `spread`.

**Attacker-controlled inputs:**
- `offer.market.loanToken` — any address with deployed code; market creation is permissionless via `touchMarket` called inside `take()`.
- The attacker deploys or uses an existing fee-on-transfer token, creates a market with it as `loanToken`, and calls `take()` as taker (or maker of a sell offer).

**Exploit flow:**
1. Deploy `TaxToken` with a `transferFrom` hook that sends `tax` to a treasury and transfers `amount - tax` to `to`, returning `true`.
2. Create a market with `loanToken = TaxToken` (permissionless via `touchMarket`).
3. Call `take()` with any non-zero `units` and a non-zero settlement fee.
4. At line 418, `claimableSettlementFee[TaxToken] += spread`.
5. At line 455, Midnight receives `spread - tax`; `claimableSettlementFee` is now `tax` units ahead of the actual balance.
6. Repeat to accumulate deficit.
7. `feeClaimer` calls `claimSettlementFee(TaxToken, claimableSettlementFee[TaxToken], receiver)` — the `safeTransfer` at line 309 attempts to send the full recorded amount, but Midnight's balance is short by the accumulated tax, causing a revert.

**Why existing checks fail:** The Certora solvency spec explicitly assumes away this case with the comment "Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver" and models `transferFrom` as an exact balance move. [4](#0-3) 

The `pendingFeeReceiptZero` weak invariant and `tokenBalanceCorrect` strong invariant are both proved only under this assumption. [5](#0-4) 

There is no code-level guard in `take()` or `touchMarket` that rejects fee-on-transfer tokens.

### Impact Explanation
`claimableSettlementFee[token]` exceeds the actual token balance attributable to fees by the cumulative tax amount. When `feeClaimer` calls `claimSettlementFee` for the full recorded amount, `safeTransfer` reverts because Midnight's balance is insufficient, making fee withdrawal impossible. If Midnight holds surplus balance from other sources (collateral, withdrawable), the transfer succeeds but silently drains those buckets, causing insolvency for lenders and borrowers who can no longer withdraw or redeem collateral. [6](#0-5) 

### Likelihood Explanation
**Preconditions:** The loan token must be a fee-on-transfer token. Such tokens exist in production (e.g., tokens with buy/sell taxes). Market creation is permissionless — any user can create a market with any ERC20 as `loanToken`. No privileged action is required. The exploit is repeatable: every `take()` call on such a market widens the deficit. A single attacker acting as taker (or as maker of a sell offer) is sufficient.

### Recommendation
Measure the actual amount received using a before/after balance check in `take()` for the fee portion, and use the measured amount to increment `claimableSettlementFee`:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 received = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
// claimableSettlementFee was already incremented by buyerAssets - sellerAssets above;
// adjust downward by the shortfall:
claimableSettlementFee[offer.market.loanToken] -= (buyerAssets - sellerAssets) - received;
```

Alternatively, move the `claimableSettlementFee` increment to after the transfer and base it on `received`. A simpler long-term fix is to document and enforce (via a registry or allowlist) that only standard ERC20 tokens with no transfer tax are permitted as loan tokens.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
// ... standard test imports

contract TaxToken {
    address public treasury;
    uint256 public taxBps = 100; // 1% tax
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;

    constructor(address _treasury) { treasury = _treasury; }

    function mint(address to, uint256 amount) external { balanceOf[to] += amount; }
    function approve(address spender, uint256 amount) external returns (bool) {
        allowance[msg.sender][spender] = amount; return true;
    }
    function transfer(address to, uint256 amount) external returns (bool) {
        return _transfer(msg.sender, to, amount);
    }
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        allowance[from][msg.sender] -= amount;
        return _transfer(from, to, amount);
    }
    function _transfer(address from, address to, uint256 amount) internal returns (bool) {
        uint256 tax = amount * taxBps / 10000;
        balanceOf[from] -= amount;
        balanceOf[treasury] += tax;
        balanceOf[to] += amount - tax;
        return true;
    }
}

contract FeeOnTransferTakeTest is Test {
    // Setup: deploy Midnight, TaxToken, create market with TaxToken as loanToken
    // Fund lender with TaxToken, set up collateral, create buy offer

    function testFeeOnTransferOverCredit() public {
        // ... setup omitted for brevity

        uint256 feeBefore = midnight.claimableSettlementFee(address(taxToken));
        uint256 balanceBefore = taxToken.balanceOf(address(midnight));

        // Taker executes take() — spread = buyerAssets - sellerAssets
        (uint256 buyerAssets, uint256 sellerAssets) = midnight.take(offer, ...);
        uint256 spread = buyerAssets - sellerAssets;
        uint256 tax = spread * 100 / 10000; // 1% tax

        uint256 feeAfter = midnight.claimableSettlementFee(address(taxToken));
        uint256 balanceAfter = taxToken.balanceOf(address(midnight));

        // claimableSettlementFee credited full spread
        assertEq(feeAfter - feeBefore, spread, "fee credited full spread");
        // but Midnight only received spread - tax
        assertEq(balanceAfter - balanceBefore, spread - tax, "midnight received less");
        // invariant broken: claimableSettlementFee > actual balance for fees
        assertGt(feeAfter - feeBefore, balanceAfter - balanceBefore, "INVARIANT BROKEN");

        // Fee claimer cannot withdraw full amount
        vm.prank(feeClaimer);
        vm.expectRevert(); // insufficient balance
        midnight.claimSettlementFee(address(taxToken), feeAfter, receiver);
    }
}
```

**Expected assertions:**
- `claimableSettlementFee[taxToken]` increases by `spread` (full nominal amount)
- `taxToken.balanceOf(midnight)` increases by only `spread - tax`
- `claimSettlementFee` reverts when attempting to claim the full `claimableSettlementFee` amount
