### Title
Fee-on-Transfer loanToken Inflates `claimableSettlementFee` Beyond Actual Balance in `take()` - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee[loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` before the inbound transfer executes. With a fee-on-transfer loanToken, the contract receives less than the nominal amount, creating a permanent gap between the accounting counter and the actual token balance. Because `claimableSettlementFee` is keyed by token address and shared across all markets using the same loanToken, this gap accumulates across every take in every market using that token, eventually making it impossible for the feeClaimer to claim the full recorded settlement fee.

### Finding Description
**Exact code path:**

`take()` at [1](#0-0)  increments the shared counter:
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
```

Then at [2](#0-1)  the actual transfers occur:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

`SafeTransferLib.safeTransferFrom` [3](#0-2)  calls `transferFrom` and checks only that it does not return `false` — it does **not** check the amount actually received by `address(this)`. For a fee-on-transfer token with fee rate `r`, the contract receives `(buyerAssets - sellerAssets) * (1 - r)` but `claimableSettlementFee` was already incremented by the full `buyerAssets - sellerAssets`. The shortfall per take is `(buyerAssets - sellerAssets) * r`.

**Market creation is permissionless.** `touchMarket` [4](#0-3)  accepts any `loanToken` with no restriction against fee-on-transfer tokens, and is callable by anyone. The protocol's own design document confirms "permissionless market creation" and "arbitrary loan token."

**Attacker inputs:** Deploy or use an existing fee-on-transfer ERC20 as `loanToken`. Create N markets with different `collateralParams` or `maturity` values (all sharing the same `loanToken`). In each market, act as lender and borrower (or cooperate with another address) to execute takes with non-zero settlement fee. Each take inflates `claimableSettlementFee[loanToken]` by more than the tokens actually deposited.

**Why existing checks fail:** The Certora `tokenBalanceCorrect` invariant in `Solvency.spec` [5](#0-4)  is proven only under the assumption that `safeTransferFrom` performs exact accounting — the CVL summary `cvlSafeTransferFrom` credits the full `amount` to the recipient with no deduction. [6](#0-5)  The formal proof is therefore unsound for fee-on-transfer tokens. There is no balance-check after the inbound transfer in `take()`, no token allowlist, and no fee-on-transfer guard anywhere in the protocol.

### Impact Explanation
After N takes across markets sharing the same fee-on-transfer loanToken:

```
claimableSettlementFee[token]  = N * spread
actual balance held for fees   = N * spread * (1 - r)
```

`claimSettlementFee` [7](#0-6)  calls `safeTransfer(token, receiver, amount)`. If `amount` equals the full recorded `claimableSettlementFee`, the transfer reverts because the contract does not hold enough tokens. The feeClaimer can only claim a partial amount (at most `actual_balance - collateral - withdrawable`), and the recorded `claimableSettlementFee` permanently overstates what is claimable. The invariant `balance >= collateral + withdrawable + claimableSettlementFee` is violated.

### Likelihood Explanation
**Preconditions:**
1. A fee-on-transfer token is used as `loanToken` — permissionless, no admin action required.
2. At least one take with non-zero settlement fee occurs — normal protocol usage.

No special privileges are needed. The attacker can be the market creator, lender, and borrower simultaneously. The effect occurs on the very first take and compounds with every subsequent take. It is not reversible: there is no mechanism to reduce `claimableSettlementFee` except by claiming it, which requires the actual balance to be present.

### Recommendation
After the inbound `safeTransferFrom` in `take()`, measure the actual received amount using a before/after balance check, and use that measured amount (rather than the nominal `buyerAssets - sellerAssets`) to increment `claimableSettlementFee`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loanTokens (e.g., via an allowlist or a received-amount assertion that reverts if the actual receipt differs from the nominal amount).

contract FeeToken is ERC20 {
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount / 100;
        super.transferFrom(from, to, amount - fee); // recipient gets amount - fee
        _burn(from, fee);                            // fee is burned
        return true;
    }
}

contract FeeOnTransferSettlementFeeTest is Test {
    Midnight midnight;
    FeeToken token;

    function testClaimableExceedsBalance() public {
        // Setup: create N markets with same fee-on-transfer loanToken
        // Execute one take per market (lender pays buyerAssets - sellerAssets to midnight)
        // Each take: claimableSettlementFee += spread, but midnight receives spread * 0.99

        uint256 N = 10;
        for (uint256 i = 0; i < N; i++) {
            // create market[i] with token as loanToken, different maturity
            // execute take with non-zero settlement fee
        }

        uint256 claimable = midnight.claimableSettlementFee(address(token));
        uint256 balance   = token.balanceOf(address(midnight));

        // KEY ASSERTION: claimable > balance (invariant violated)
        assertGt(claimable, balance, "claimableSettlementFee exceeds actual balance");

        // feeClaimer cannot claim the full amount
        vm.prank(feeClaimer);
        vm.expectRevert(); // safeTransfer reverts: insufficient balance
        midnight.claimSettlementFee(address(token), claimable, feeClaimer);
    }

**Expected assertions:**
- `claimableSettlementFee[token] > token.balanceOf(address(midnight))` — invariant broken
- `claimSettlementFee(token, claimableSettlementFee[token], receiver)` reverts
- The gap equals `N * spread * fee_rate` and grows linearly with each additional take

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
