Audit Report

## Title
`consumableUnits` ignores `offer.reduceOnly`, causing bundler to silently skip permanently-reverting offers and revert with `OutOfOffers` - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary
`ConsumableUnitsLib.consumableUnits` computes fillable units solely from `consumed[maker][group]` vs `offer.maxUnits`/`offer.maxAssets`, with no awareness of `offer.reduceOnly` or the maker's current `credit`/`debt`. When a `reduceOnly` sell offer's maker holds zero credit (or a `reduceOnly` buy offer's maker holds zero debt), `consumableUnits` returns a positive value, but every subsequent `Midnight.take()` call will deterministically revert with `MakerCreditOrDebtIncreased`. The bundler's `catch {}` silently absorbs the revert, `filledUnits` is never incremented, and the bundler ultimately reverts with `OutOfOffers`, griefing any taker who routes through it.

## Finding Description

**Root cause — missing `reduceOnly` guard in `consumableUnits`:**

`src/periphery/ConsumableUnitsLib.sol` lines 14–23 read only the `consumed` counter and `offer.maxUnits`/`offer.maxAssets`: [1](#0-0) 

`offer.reduceOnly` and the maker's on-chain `credit`/`debt` are never consulted.

**`take()` enforces the constraint that `consumableUnits` ignores:**

`src/Midnight.sol` lines 382–395 compute `sellerDebtIncrease = units - min(units, sellerPos.credit)` and then enforce: [2](#0-1) 

For a `reduceOnly` sell offer where `sellerPos.credit == 0`: `sellerCreditDecrease = 0`, `sellerDebtIncrease = units`. The require evaluates to `false || (units == 0)`, which always fails for any `units > 0`.

**Bundler silently absorbs the revert:**

All four bundle functions follow the same pattern — `consumableUnits` returns N > 0, `take()` reverts with `MakerCreditOrDebtIncreased`, `catch {}` swallows it, `filledUnits` stays 0, and `require(filledUnits == targetUnits, OutOfOffers())` fires: [3](#0-2) 

The same pattern appears in `supplyCollateralAndSellWithUnitsTarget` (lines 147–163), `buyWithAssetsTargetAndWithdrawCollateral` (lines 208–224), and `supplyCollateralAndSellWithAssetsTarget` (lines 285–303). [4](#0-3) 

**`reduceOnly` is a standard, signable field:** [5](#0-4) 

## Impact Explanation

A maker can publish a `reduceOnly` offer with no existing credit/debt. Any taker who routes through the bundler with this offer as the sole (or gap-filling) entry will have their transaction revert with `OutOfOffers`. The taker loses gas and may miss time-sensitive operations (e.g., liquidation-adjacent collateral withdrawal, rate-sensitive borrowing). The bundler's `pullToken`/`forceApproveMax` sequence executes before the loop, so the taker's tokens are temporarily pulled and then returned on revert — no permanent fund loss, but the transaction is reliably and repeatably griefable. This constitutes service unavailability/severe degradation under realistic attacker input.

## Likelihood Explanation

**Preconditions:** Maker creates a `reduceOnly` offer with `maxUnits > 0` and `credit == 0` (sell) or `debt == 0` (buy). This is a valid, signable offer state — no privileged access required. **Feasibility:** Trivially reachable. A maker can sign such an offer at any time. Off-chain order books and aggregators rely on `consumableUnits` to filter offers; a positive return value will cause them to include the offer. **Repeatability:** The maker's offer remains valid (not consumed, not expired) after the bundler reverts, so the same offer can grief repeated bundler calls until the taker manually excludes it.

## Recommendation

In `ConsumableUnitsLib.consumableUnits`, add a `reduceOnly` guard that queries the maker's on-chain position before returning a non-zero value. Specifically:

```solidity
if (offer.reduceOnly) {
    (uint128 credit,,,, uint128 debt,) = IMidnight(midnight).position(id, offer.maker);
    uint256 reducible = offer.buy ? uint256(debt) : uint256(credit);
    if (reducible == 0) return 0;
    // optionally cap the result by reducible
}
```

This mirrors the constraint enforced by `take()` at lines 392–395 of `Midnight.sol` and ensures `consumableUnits` returns 0 for any `reduceOnly` offer whose maker has no position to reduce, preventing the bundler from attempting a deterministically-reverting `take()`.

## Proof of Concept

1. Deploy Midnight and MidnightBundles in a fork/unit test environment.
2. Maker signs a `reduceOnly` sell offer with `maxUnits = 100`, `credit = 0` in the target market.
3. Taker calls `buyWithUnitsTargetAndWithdrawCollateral` with `targetUnits = 100` and `takes = [makerOffer]`.
4. Observe: `consumableUnits` returns 100; `take()` reverts with `MakerCreditOrDebtIncreased`; `catch {}` swallows it; `filledUnits == 0`; transaction reverts with `OutOfOffers`.
5. Confirm the offer is still unconsumed and the attack is repeatable.
