### Title
Lenders Can Front-Run Bad Debt Liquidations to Escape Proportional Loss Socialization - (File: src/Midnight.sol)

### Summary

In `Midnight.sol`, bad debt is socialized among lenders by updating `marketState.lossFactor` inside `liquidate()`. Because `lossFactor` is applied lazily — only when `_updatePosition()` is called — a lender who calls `withdraw()` before a bad debt `liquidate()` transaction is mined escapes their proportional share of the loss. The remaining lenders bear a disproportionately larger loss.

### Finding Description

**Root cause — lazy `lossFactor` application:**

When `liquidate()` realizes bad debt, it updates `lossFactor` and reduces `totalUnits`:

```solidity
// src/Midnight.sol:631-634
_marketState.lossFactor = UtilsLib.toUint128(
    type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
);
_marketState.totalUnits -= UtilsLib.toUint128(badDebt);
``` [1](#0-0) 

This updated `lossFactor` is only applied to a lender's credit at their next `_updatePosition()` call, which computes:

```solidity
// src/Midnight.sol:805-806
uint256 postSlashCredit = _lastLossFactor < type(uint128).max
    ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
    : 0;
``` [2](#0-1) 

**The front-run path:**

`withdraw()` calls `_updatePosition()` at the very start:

```solidity
// src/Midnight.sol:485
_updatePosition(market, id, onBehalf);
``` [3](#0-2) 

If a lender's `withdraw()` is mined **before** the `liquidate()` that raises `lossFactor`, then `_updatePosition()` sees the old (pre-bad-debt) `lossFactor`, applies no slash, and the lender withdraws their full credit. After the liquidation executes, the lender is already out of the market and bears zero loss.

**Attack flow:**

1. Market state: `totalUnits = 1000`, `withdrawable = 600` (some borrowers have repaid). A borrower has `debt = 200` with collateral worth only 80 (bad debt = 120).
2. A liquidator submits `liquidate(borrower, seizedAssets=0, repaidUnits=0)` to realize the bad debt. This transaction enters the public mempool.
3. Lender A (credit = 100) sees the pending liquidation. Their fair share of the loss is `100 * 120 / 1000 = 12` units.
4. Lender A submits `withdraw(100)` with higher gas. It mines first.
5. `_updatePosition()` applies the current (pre-bad-debt) `lossFactor` — no slash. Lender A receives 100 tokens. `withdrawable` drops to 500.
6. The liquidation mines: `lossFactor` increases, `totalUnits` drops to 880. The 120-unit loss is now spread over the remaining 900 units of credit, not 1000.
7. Lender A has escaped their 12-unit loss, which is now borne by the remaining lenders.

The protocol's own comment acknowledges a related race: [4](#0-3) 

However, that comment refers to racing to withdraw repaid assets. The bad debt front-run is a distinct, targeted attack where a lender deliberately avoids a loss that the protocol intends to socialize.

### Impact Explanation

- **Direct financial loss** to remaining lenders: they absorb a larger-than-intended share of every bad debt event that is front-run.
- The attacker recovers their full credit with zero loss; the deficit is permanently shifted to honest lenders.
- Severity scales with position size and bad debt magnitude. In markets with high LLTV (especially LLTV = 1, where bad debt is almost guaranteed on liquidation per the protocol's own comment at line 22–25), this is highly profitable. [5](#0-4) 

### Likelihood Explanation

- **Realistic and low-cost**: Any lender with credit in a market can monitor the public mempool for `liquidate()` calls that will realize bad debt (detectable by simulating the call off-chain).
- No privileged access is required — only a standard lender position and enough `withdrawable` balance in the market (which exists whenever any borrower has repaid).
- The attack is profitable whenever `badDebt * credit / totalUnits > gas_cost`, a condition easily met on L2s with cheap gas.
- The protocol's design (lazy `lossFactor` application, no withdrawal delay) makes this structurally exploitable on every bad debt event.

### Recommendation

Introduce a **withdrawal delay / two-step withdrawal** mechanism:

1. **Step 1 — Request**: Record the lender's requested withdrawal amount and the current `lossFactor` at request time. Apply the slash immediately at request time (not at execution time).
2. **Step 2 — Execute**: After a mandatory delay (e.g., 1 block or a configurable timelock), allow the lender to collect the already-slashed amount.

This ensures that any bad debt realized between the request and execution still applies to the lender's position, eliminating the front-run incentive. Alternatively, applying the `lossFactor` slash atomically at the time of `withdraw()` using the **post-liquidation** state (i.e., requiring the lender to accept any `lossFactor` update that occurred in the same block) would also mitigate this, though a timelock is more robust.

### Proof of Concept

**Setup:**
- `totalUnits = 1000`, `withdrawable = 600`
- Borrower B: `debt = 200`, collateral value = 80 → `badDebt = 120`
- Lender A: `credit = 100`, `lastLossFactor = 0` (current market `lossFactor`)

**Step 1** — Liquidator broadcasts:
```
liquidate(market, collateralIndex, seizedAssets=0, repaidUnits=0, borrower=B, ...)
```
This is visible in the mempool. Simulating it shows `badDebt = 120` will be realized.

**Step 2** — Lender A front-runs with higher gas:
```
withdraw(market, units=100, onBehalf=A, receiver=A)
```
- `_updatePosition()` runs with `lossFactor = 0` (unchanged) → `postSlashCredit = 100`, no slash.
- `_position.credit -= 100` → credit = 0
- `_marketState.withdrawable -= 100` → withdrawable = 500
- `_marketState.totalUnits -= 100` → totalUnits = 900
- Lender A receives 100 tokens.

**Step 3** — Liquidation executes:
```solidity
// badDebt = 120, _totalUnits = 900
lossFactor = type(uint128).max - (type(uint128).max - 0) * (900 - 120) / 900
           = type(uint128).max - type(uint128).max * 780/900
           ≈ type(uint128).max * 120/900
totalUnits = 780
```
The 120-unit loss is spread over 780 remaining credit units (≈15.4% loss), instead of 900 (≈13.3% loss).

**Result:** Lender A escaped a ~12-unit loss. Remaining lenders bear ~2.1% extra loss per unit of credit — a direct, measurable transfer of value from honest lenders to the front-runner.
