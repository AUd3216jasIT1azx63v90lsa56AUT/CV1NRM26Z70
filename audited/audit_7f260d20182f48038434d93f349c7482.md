Audit Report

## Title
Oracle Price Manipulated via `onSell` Callback Allows Undercollateralized Borrowing - (File: src/Midnight.sol)

## Summary
In `take()`, the seller's debt is committed to storage at line 414 before the `onSell` callback fires at lines 458–474. The health check at line 476 calls `isHealthy`, which queries `IOracle.price()` after the callback completes. Because the callback executes via an unrestricted `CALL`, it can write to any external contract, including an attacker-controlled `StateContract`. A malicious oracle that reads from that `StateContract` will return an inflated price during the health check, causing `isHealthy` to pass with fabricated collateral value and leaving the seller with undercollateralized debt.

## Finding Description
**Exact code path:**

1. `take()` is called (`src/Midnight.sol:337`).
2. `sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease)` at line 414 — seller's debt is committed to storage before any callback.
3. Liquidation lock is set for the seller at line 444 via `UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`.
4. `onSell` callback fires at lines 458–474 via a regular `CALL` (not `staticcall`). The callback can write to any external contract, including an attacker-controlled `StateContract`.
5. Liquidation lock is released at line 475.
6. `require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable())` at line 476.
7. Inside `isHealthy` at line 953: `uint256 price = IOracle(collateralParam.oracle).price()` — called via `staticcall` because `price()` is declared `external view` in `src/interfaces/IOracle.sol:6`. The `staticcall` cannot write state but freely reads state already written by the callback. The oracle reads from the `StateContract` whose storage was written in step 4, returning an inflated price.
8. `maxDebt` is computed with the inflated price at line 954–955 and exceeds the seller's actual debt — health check passes.

**Root cause:** The protocol fires an unrestricted external `CALL` (`onSell`) between committing the seller's debt increase and evaluating the health check. No snapshot of the oracle price is taken before the callback. Any state written by the callback is visible to the oracle when `isHealthy` is called.

**Why existing checks fail:**
- The liquidation lock (lines 444, 475) only prevents liquidation during callbacks; it does not prevent oracle price manipulation.
- `staticcall` on `price()` prevents the oracle from writing state during the health check, but the callback already wrote the manipulated value before `isHealthy` is called. `staticcall` can freely read state written earlier in the same transaction.
- The Certora formal verification at `certora/specs/Healthiness.spec:14–16` explicitly assumes `price()` is constant during execution (`// Assumption: price does not change during rules. ... function _.price() external => summaryPrice(calledContract) expect(uint256)`) and does not cover this case. The `stayHealthy` rule at line 260 explicitly excludes `take` from its coverage.

## Impact Explanation
The seller can borrow an arbitrary number of units backed by negligible collateral (e.g., 1 wei). The seller receives `sellerAssets` loan tokens during the take. After `take()` completes, the oracle returns to its real price, leaving the seller's position undercollateralized. Lenders bear the resulting bad debt. This is a direct theft of lender funds and violates the core protocol invariant that every debt position must be backed by sufficient collateral at the real oracle price. Bad debt creation is an explicitly listed top-priority impact class in `live_context.json`.

## Likelihood Explanation
All preconditions are fully attacker-controlled and require no privileged access:
- Anyone can call `touchMarket` with a `CollateralParams.oracle` pointing to a malicious oracle that reads from an attacker-controlled `StateContract`. `touchMarket` at line 755 validates only `lltv`, `maxLif`, collateral token ordering, and maturity — it does not validate oracle behavior.
- The seller controls `offer.callback` (when `offer.buy == false`, seller = `offer.maker`) or `takerCallback` (when `offer.buy == true`, seller = taker).
- The attacker deploys three contracts (`StateContract`, malicious oracle, malicious callback), creates a market, and creates or takes an offer. No governance, admin, or leaked-key access is needed.
- `SECURITY.md:26` explicitly states: "Note: This does not exclude oracle manipulation/flash-loan attacks."
- The attack is repeatable across any number of takes and is not blocked by any on-chain guard.

## Recommendation
Snapshot the oracle price for each activated collateral before the `onSell` callback fires, and use those snapshots inside `isHealthy` rather than re-querying the oracle after the callback. Alternatively, call `isHealthy` (or at minimum query all oracle prices) before the `onSell` callback and compare against a post-callback re-query, reverting if any price changed. A simpler mitigation is to move the health check to before the `onSell` callback, accepting that the seller must already be healthy before receiving `sellerAssets`.

## Proof of Concept
Deploy three contracts:

```solidity
// StateContract: holds a manipulable price slot
contract StateContract {
    uint256 public manipulatedPrice;
    function setPrice(uint256 p) external { manipulatedPrice = p; }
}

// MaliciousOracle: reads from StateContract; returns real price when slot is 0
contract MaliciousOracle {
    StateContract sc;
    uint256 realPrice = 1e36; // ORACLE_PRICE_SCALE
    function price() external view returns (uint256) {
        uint256 p = sc.manipulatedPrice();
        return p == 0 ? realPrice : p;
    }
}

// MaliciousSellCallback: inflates price during onSell, resets after
contract MaliciousSellCallback is ISellCallback {
    StateContract sc;
    function onSell(...) external returns (bytes32) {
        sc.setPrice(type(uint256).max / 2); // inflate to pass health check
        return CALLBACK_SUCCESS;
    }
}
```

Steps:
1. Create a market via `touchMarket` with `CollateralParams.oracle = address(maliciousOracle)`.
2. Supply 1 wei of collateral for the seller.
3. Have a victim lender create a buy offer for N units.
4. Seller calls `take()` with `takerCallback = address(maliciousSellCallback)`.
5. During `onSell`, `StateContract.manipulatedPrice` is set to a huge value.
6. `isHealthy` queries `maliciousOracle.price()` which returns the huge value — health check passes.
7. After `take()`, `StateContract.manipulatedPrice` can be reset to 0 (or left; the oracle returns real price on next call).
