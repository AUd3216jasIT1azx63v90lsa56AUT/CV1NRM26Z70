Audit Report

## Title
`claimContinuousFee` reverts via arithmetic underflow when bad-debt liquidation front-runs the feeClaimer - (File: `src/Midnight.sol`)

## Summary
`claimContinuousFee` performs `_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount)` with no guard that `amount <= continuousFeeCredit`. A bad-debt liquidation with `seizedAssets=0, repaidUnits=0` costs zero tokens yet atomically reduces `continuousFeeCredit` via the proportional slash formula. Any unprivileged caller can therefore front-run the feeClaimer's pending claim and cause it to revert with a Solidity 0.8 arithmetic underflow, griefing protocol fee collection.

## Finding Description
**`claimContinuousFee`** (`src/Midnight.sol:318`):
```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
```
No `require(amount <= _marketState.continuousFeeCredit)` guard exists. Solidity 0.8 checked arithmetic reverts on underflow.

**Bad-debt slash in `liquidate`** (`src/Midnight.sol:635–640`):
```solidity
_marketState.continuousFeeCredit = _lossFactor < type(uint128).max
    ? UtilsLib.toUint128(
        _marketState.continuousFeeCredit
            .mulDivDown(type(uint128).max - _marketState.lossFactor,
                        type(uint128).max - _lossFactor)
      )
    : 0;
```
Because `newLossFactor > oldLossFactor`, the ratio `(MAX − newLF) / (MAX − oldLF) < 1`, so `continuousFeeCredit` is strictly reduced on every bad-debt event.

**Zero-cost liquidation path** (`src/Midnight.sol:577–578`, comment; `src/Midnight.sol:643`):
The protocol explicitly documents that passing `seizedAssets=0, repaidUnits=0` realizes bad debt with zero tokens transferred. The `if (repaidUnits > 0 || seizedAssets > 0)` block at line 643 is skipped, but the bad-debt slash at lines 626–640 executes unconditionally.

**Exploit flow**:
1. `continuousFeeCredit = X`. feeClaimer broadcasts `claimContinuousFee(market, X, receiver)`.
2. Attacker front-runs with `liquidate(market, 0, 0, 0, borrower, false, receiver, address(0), "")` — zero tokens spent, only gas. `continuousFeeCredit` is reduced to `X' < X`.
3. feeClaimer's transaction executes: `X' -= X` → arithmetic underflow → revert.

**Why existing checks fail**: The `liquidatorGate` check (`src/Midnight.sol:598`) only restricts the attacker when `market.liquidatorGate != address(0)`; markets with no gate (the default) are fully open. The Certora spec (`certora/specs/Role.spec:264`) itself encodes `amount <= continuousFeeCreditBefore` as a necessary non-revert condition, confirming no on-chain guard enforces this invariant.

## Impact Explanation
The feeClaimer's `claimContinuousFee` transaction is griefed and reverts. No funds are permanently lost — `continuousFeeCredit` is reduced (not stolen) — but protocol fee collection is persistently blocked as long as any bad-debt-eligible position exists. The feeClaimer must re-read state and resubmit with a reduced amount on every attempt, and the attack can be replayed immediately on each resubmission.

## Likelihood Explanation
**Attacker cost**: gas only — no tokens required. **Preconditions**: (a) a borrower position with `debt > 0` and `originalDebt > maxDebt` (bad debt) exists, and (b) the feeClaimer submits `amount = continuousFeeCredit` (the natural "claim all" call). Both are routine market conditions. **Repeatability**: the attack can be replayed on every subsequent claim attempt until all bad-debt positions are exhausted. **Access**: any unprivileged address can call `liquidate` on markets without a `liquidatorGate`.

## Recommendation
Add an explicit cap in `claimContinuousFee` before the subtraction:

```solidity
amount = UtilsLib.min(amount, _marketState.continuousFeeCredit);
```

Or add a strict guard:

```solidity
require(amount <= _marketState.continuousFeeCredit, InsufficientContinuousFeeCredit());
```

The capping approach is preferable as it makes "claim all" calls robust against concurrent state changes without reverting.

## Proof of Concept
1. Deploy a market with no `liquidatorGate`.
2. Create a borrower position that accrues bad debt (oracle price drops below `badDebtPriceDown`).
3. Allow `continuousFeeCredit` to accumulate to value `X`.
4. Submit `claimContinuousFee(market, X, receiver)` from the feeClaimer account.
5. In the same block (before step 4 mines), call `liquidate(market, 0, 0, 0, borrower, false, attacker, address(0), "")` from any EOA.
6. Observe: step 5 mines first, `continuousFeeCredit` drops to `X' < X`, step 4 reverts with arithmetic underflow.
7. Repeat from step 4 to demonstrate repeatability.
