Audit Report

## Title
`sellerAssetsToUnits` reverts on `sellerPrice = 0` while `Midnight.take()` succeeds, causing `supplyCollateralAndSellWithAssetsTarget` to revert for valid buy offers - (File: src/periphery/TakeAmountsLib.sol)

## Summary

When a buy offer exists at tick `T` where `tickToPrice(T) == settlementFee(id, ttm)`, `sellerPrice` computes to exactly `0`. `TakeAmountsLib.sellerAssetsToUnits` then calls `mulDivUp(targetSellerAssets, WAD, 0)`, which triggers an arithmetic underflow at `d - 1` under Solidity 0.8.x checked arithmetic, reverting unconditionally. `Midnight.take()` does not revert in this case — it computes `sellerAssets = units.mulDivDown(0, WAD) = 0` and proceeds normally. Because `sellerAssetsToUnits` is called outside the `try/catch` block in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget`, the revert propagates out of the bundler entirely.

## Finding Description

**Root cause — `mulDivUp` with `d = 0`:**

`UtilsLib.mulDivUp` is implemented as:

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
``` [1](#0-0) 

When `d = 0`, `d - 1` underflows under Solidity 0.8.x checked arithmetic, reverting unconditionally. This is explicitly confirmed by the test: [2](#0-1) 

**Code path in `TakeAmountsLib.sellerAssetsToUnits`:**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy
    ? targetSellerAssets.mulDivUp(WAD, sellerPrice)   // d = 0 → REVERT
    : targetSellerAssets.mulDivDown(WAD, sellerPrice);
``` [3](#0-2) 

When `offer.buy = true` and `offerPrice == settlementFee`, `sellerPrice = 0`. The call becomes `mulDivUp(targetSellerAssets, WAD, 0)`, which reverts via the `d - 1` underflow.

**Contrast with `Midnight.take()`:**

`mulDivDown(units, 0, WAD)` computes `(units * 0) / WAD = 0` — no revert. `Midnight.take()` succeeds and yields `sellerAssets = 0`. The NatSpec on `sellerAssetsToUnits` at line 34 states: *"Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)."* This is correct for the strict-less-than case, but the equal case (`offerPrice == settlementFee`) is undocumented and diverges: `Midnight.take()` accepts `sellerPrice = 0` while `sellerAssetsToUnits` reverts. [4](#0-3) 

**Call site in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget`:**

`TakeAmountsLib.sellerAssetsToUnits` is called outside the `try/catch` block. The `try/catch` only wraps `IMidnight(MIDNIGHT).take(...)`. The bundler's own NatSpec at line 246 confirms: *"Reverts if TakeAmountsLib or ConsumableUnitsLib reverts."* [5](#0-4) [6](#0-5) 

## Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer at tick `T` where `tickToPrice(T) == settlementFee(id, ttm)` reverts entirely. The taker cannot complete the bundled sell-with-assets-target operation through this offer. The attacker's offer is never consumed (the bundler reverts before `take()` is called), so the griefing is permanent and repeatable at zero cost beyond the initial offer placement. This constitutes a targeted, repeatable DoS of a core bundler function for any taker routing through such an offer.

## Likelihood Explanation

**Preconditions:**
1. A buy offer exists at tick `T` where `tickToPrice(T) == settlementFee(id, ttm)`. `tickToPrice` returns multiples of `PRICE_ROUNDING_STEP = 1e12` across ticks 0–5820. Settlement fees are also multiples of `1e12`. An attacker can enumerate ticks offline and post an offer at the matching tick — no privileged access required. [7](#0-6) 

2. A taker routes through this offer via `supplyCollateralAndSellWithAssetsTarget` — possible if the offer appears in an off-chain order book and is included in the `takes` array.
3. `targetFilledSellerAssets - filledSellerAssets > 0` — always true inside the loop due to the `filledSellerAssets < targetFilledSellerAssets` loop guard. [8](#0-7) 

The attack is repeatable: the attacker maintains the offer indefinitely since the bundler reverts before `take()` is called, so the offer is never consumed.

## Recommendation

In `TakeAmountsLib.sellerAssetsToUnits`, handle the `sellerPrice = 0` edge case explicitly. When `sellerPrice == 0`, any nonzero `targetSellerAssets` is unreachable (since `sellerAssets` will always be 0 regardless of units), so the function should either return `type(uint256).max` (signaling that no finite unit count achieves the target) or revert with a descriptive error. Alternatively, mirror the behavior of `Midnight.take()` by returning `type(uint256).max` when `sellerPrice == 0` and `targetSellerAssets > 0`, so the `min(...)` in the bundler naturally clamps to `takes[i].units` or `consumableUnits`, and the `try/catch` around `take()` handles any downstream revert gracefully.

## Proof of Concept

1. Deploy Midnight and MidnightBundles on a fork.
2. Find tick `T` such that `tickToPrice(T) == settlementFee(id, ttm)` for a given market (enumerate offline using `TickLib.tickToPrice` and `IMidnight.settlementFee`).
3. Place a buy offer at tick `T` with nonzero units.
4. Call `supplyCollateralAndSellWithAssetsTarget` with `takes = [{offer at tick T, ...}]` and `targetSellerAssets > 0`.
5. Observe the call reverts with `arithmeticError` at `UtilsLib.mulDivUp` before `take()` is ever called.
6. Confirm `IMidnight.take(offer at tick T, ..., units, ...)` succeeds directly, returning `sellerAssets = 0`. [1](#0-0) [9](#0-8)
