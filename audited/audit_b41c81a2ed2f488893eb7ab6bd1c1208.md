Audit Report

## Title
Zero-increment `consumed` bypass allows unlimited takes on fully-consumed buy offers — (File: `src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and `buyerPrice < WAD` (true for every valid tick), calling `take` with `units = 1` produces `buyerAssets = 0` via integer truncation in `mulDivDown`. The guard `require(newConsumed <= offer.maxAssets)` then passes unconditionally even when the offer is fully consumed, while all downstream position mutations still execute on the non-zero `units` value with zero token transfer.

## Finding Description
**Root cause — `src/Midnight.sol` lines 363–373:**

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For buy offers, `consumed` advances by `buyerAssets`, computed as `(units * buyerPrice) / WAD` with truncation. `buyerPrice` equals `tickToPrice(offer.tick)` (the settlement fee cancels: `sellerPrice = offerPrice - fee`, `buyerPrice = sellerPrice + fee = offerPrice`).

`tickToPrice` in `src/libraries/TickLib.sol` returns:
```
1e36 / (1e18 + wExp(LN_ONE_PLUS_DELTA * (MAX_TICK/2 - tick)))
```
For every tick in `[0, MAX_TICK]`, `wExp(...)` is strictly positive, so the denominator exceeds `1e18`, and the result is strictly less than `1e18 = WAD`. Therefore `buyerPrice < WAD` for all reachable ticks.

With `units = 1`: `1 * buyerPrice / WAD = 0` (integer division). `newConsumed = consumed + 0 = consumed`. The guard `require(consumed <= maxAssets)` passes even when `consumed == maxAssets`.

Downstream state mutations at lines 382–417 all operate on `units = 1`:
- `buyerCreditIncrease = zeroFloorSub(1, buyerPos.debt)` — non-zero if buyer has no debt
- `sellerDebtIncrease = 1 - sellerCreditDecrease` — non-zero if seller has no credit
- `totalUnits` incremented by `buyerCreditIncrease - sellerCreditDecrease`
- `claimableSettlementFee` incremented by `buyerAssets - sellerAssets = 0`

Token transfers at lines 455–456 send `0` tokens (`safeTransferFrom(payer, address(this), 0)` and `safeTransferFrom(payer, receiver, 0)`). No `require(units > 0)` or `require(buyerAssets > 0)` guard exists anywhere in `take`.

## Impact Explanation
A fully-consumed buy offer (`consumed == maxAssets`) can be taken an unlimited number of times. Each call increases the maker's credit and taker's debt by `units` with zero token payment, increments `totalUnits` without a corresponding increase in `claimableSettlementFee` or token reserves, and permanently breaks the protocol invariant that units outstanding are backed by deposited assets. Repeated exploitation accumulates unbacked credit and debt, leading to market insolvency.

## Likelihood Explanation
Preconditions are minimal: a buy offer with `maxAssets > 0` (standard configuration) and any tick below `MAX_TICK` (the entire normal operating range). `units = 1` trivially satisfies `1 * buyerPrice < WAD` for any such tick. No special permissions, oracle manipulation, or victim cooperation is required. Any unprivileged taker can trigger this after any partial or full fill, and the attack is repeatable without limit.

## Recommendation
Add a guard that rejects takes where `units > 0` but the asset-denominated increment is zero. For buy offers:

```solidity
require(buyerAssets > 0 || units == 0, ZeroAssetTake());
```

Alternatively, enforce a minimum `units` value such that `units.mulDivDown(buyerPrice, WAD) >= 1`, or switch the `maxAssets` accounting for buy offers to use `mulDivUp` so that even `units = 1` produces a non-zero increment (though this changes rounding semantics). The most conservative fix is the explicit `require(buyerAssets > 0)` guard placed immediately after line 363.

## Proof of Concept
1. Deploy a market with any valid tick `t < MAX_TICK` and `tickSpacing`.
2. Maker signs a buy offer with `maxAssets = M` for some small `M`.
3. Fill the offer normally until `consumed[maker][group] == M`.
4. Attacker calls `take` with `units = 1`, `offer.buy = true`, same offer parameters.
5. Observe: `buyerAssets = 0`, `newConsumed = M`, guard passes, `buyerPos.credit` or `sellerPos.debt` increases by 1, zero tokens transferred.
6. Repeat step 4 indefinitely — `consumed` never advances past `M`, position state mutates each time.

A fuzz test asserting `consumed[maker][group] <= maxAssets` after any sequence of `take` calls, or an invariant test asserting `totalUnits` equals the sum of all `buyerAssets` transferred, will reproduce the violation.
