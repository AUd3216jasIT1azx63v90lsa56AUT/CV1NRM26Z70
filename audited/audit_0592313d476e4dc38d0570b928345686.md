### Title
Debt Increase Permitted at Exact Maturity Boundary via `<=` Operator in `take` — (`src/Midnight.sol`)

### Summary
The maturity guard in `Midnight.take` uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` when `block.timestamp == market.maturity`, allowing `sellerDebtIncrease > 0` to pass unchecked at the exact maturity timestamp. This directly violates the explicitly stated invariant: *"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."* The `multicall` function compounds this by allowing an attacker to execute multiple debt-increasing `take` calls in the same block at maturity, all sharing the same `block.timestamp`.

### Finding Description

**Code path:**

`Midnight.multicall` (line 211–220) → `delegatecall` → `Midnight.take` (line 337–479)

The critical guard is at line 391:

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [1](#0-0) 

When `block.timestamp == market.maturity`, the left operand `block.timestamp <= offer.market.maturity` is `true`, so the entire `require` passes regardless of the value of `sellerDebtIncrease`. The `sellerDebtIncrease` is computed at line 384:

```solidity
uint256 sellerDebtIncrease = units - sellerCreditDecrease;
``` [2](#0-1) 

If the seller (borrower) has no existing credit to offset (`sellerPos.credit == 0`), then `sellerCreditDecrease == 0` and `sellerDebtIncrease == units > 0`. The guard passes, and line 414 writes the debt increase to storage:

```solidity
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
``` [3](#0-2) 

**Attacker-controlled inputs:**
- `offer.market.maturity` is fixed at market creation (immutable).
- The attacker warps (or waits) to `block.timestamp == market.maturity`.
- The attacker constructs one or more valid sell-side offers (or uses existing maker offers) and calls `multicall([take(offer1, units1, seller), take(offer2, units2, seller)])`.
- Both calls share the same `block.timestamp`, both pass the `<=` guard, and both write `sellerDebtIncrease` to the seller's debt.

**Why existing checks fail:**
- The `CannotIncreaseDebtPostMaturity` guard uses `<=` instead of `<`, so it does not fire at `block.timestamp == maturity`.
- The `SellerIsLiquidatable` check at line 476 only prevents the take if the seller is *already* liquidatable before the trade; it does not prevent debt creation at maturity.
- The `enterGate.canIncreaseDebt` check (lines 402–406) is a market-level gate, not a maturity guard, and is optional.
- All existing tests for this guard use `market.maturity + 1` (strictly after maturity), leaving the exact boundary untested. [4](#0-3) [5](#0-4) 

**Explicitly violated invariant** (live_context.json line 221):

> "maturity boundary must not allow debt increase by rounding, **timestamp equality**, or callback sequencing" [6](#0-5) 

Also violated: `core_invariants.credit_debt`: "debt must not increase after maturity" (line 197). [7](#0-6) 

### Impact Explanation
A borrower can take on new debt at exactly `block.timestamp == market.maturity`. This debt is immediately overdue — the market's repayment deadline has arrived — and is immediately subject to post-maturity liquidation (per `overdue_debt_after_maturity_is_liquidatable_even_if_healthy`). The borrower has received loan assets in exchange for debt that should never have been created, violating the protocol's solvency invariant. Via `multicall`, this can be repeated across multiple offers in a single transaction, amplifying the bad debt created in one atomic operation.

### Likelihood Explanation
The precondition is `block.timestamp == market.maturity`, which is a single specific second. However:
- Any market with a known maturity timestamp is predictably reachable by any participant.
- The attacker simply submits the `multicall` transaction in the block whose timestamp equals `market.maturity` (achievable by monitoring the mempool or by being a block proposer/builder).
- No privileged access, no oracle manipulation, and no user mistake is required.
- The attack is repeatable across every market at its maturity second.

### Recommendation
Change the operator in the maturity guard from `<=` to `<`:

```solidity
// Before (line 391):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After:
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [1](#0-0) 

This ensures that at `block.timestamp == market.maturity`, any `sellerDebtIncrease > 0` causes a revert, consistent with the invariant that debt must not increase at or after maturity.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {IMidnight} from "../src/interfaces/IMidnight.sol";

contract MaturityBoundaryDebtTest is BaseTest {
    function testDebtIncreasesAtExactMaturity() public {
        uint256 units = 100;
        // Setup: lender supplies credit, borrower has collateral
        collateralize(market, borrower, units * 2);
        setupMarket(market, units);

        // Warp to EXACTLY maturity (not maturity + 1)
        vm.warp(market.maturity);

        uint256 debtBefore = midnight.debtOf(id, borrower);

        // Construct a buy offer (maker=lender, taker=borrower as seller)
        // borrowerOffer is a sell offer: taker=borrower becomes seller
        borrowerOffer.expiry = market.maturity;
        borrowerOffer.maxUnits = units;
        deal(address(loanToken), lender, units);

        // Single take at exact maturity — should revert but does NOT
        // because block.timestamp <= market.maturity is true
        take(units, lender, borrowerOffer);

        uint256 debtAfter = midnight.debtOf(id, borrower);

        // ASSERTION: debt must not have increased at maturity
        // This assertion FAILS, proving the bug
        assertEq(debtAfter, debtBefore, "debt must not increase at maturity");
    }

    function testMulticallDoubleDebtAtMaturity() public {
        uint256 units = 50;
        collateralize(market, borrower, units * 4);
        setupMarket(market, units);

        vm.warp(market.maturity);

        uint256 debtBefore = midnight.debtOf(id, borrower);

        // Two separate buy offers, both filled at exact maturity via multicall
        borrowerOffer.expiry = market.maturity;
        borrowerOffer.maxUnits = units;
        deal(address(loanToken), lender, units * 2);

        bytes[] memory calls = new bytes[](2);
        calls[0] = abi.encodeCall(
            midnight.take,
            (borrowerOffer, "", units, lender, address(0), address(0), "")
        );
        // Use a second offer (different group/nonce) for the second call
        calls[1] = abi.encodeCall(
            midnight.take,
            (borrowerOffer2, "", units, lender, address(0), address(0), "")
        );

        midnight.multicall(calls);

        uint256 debtAfter = midnight.debtOf(id, borrower);

        // ASSERTION: debt increased twice at maturity — both should have reverted
        assertEq(debtAfter, debtBefore, "debt must not increase at maturity via multicall");
        // Expected: debtAfter == debtBefore (test fails, proving the bug)
        // Actual:   debtAfter == debtBefore + 2 * units
    }
}
```

Expected test outcome: both assertions fail, with `debtAfter > debtBefore`, confirming that `sellerDebtIncrease` is written to storage at `block.timestamp == market.maturity` because the `<=` guard does not revert.
