Audit Report

## Title
Fully-consumed assets-capped buy offer with `buyerPrice < WAD` can be taken indefinitely for free units â€” (`src/Midnight.sol`)

## Summary
When a buy offer uses `maxAssets` as its cap and `offerPrice < WAD`, calling `take()` with `units = 1` produces `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. Because the consumed accounting increments by `buyerAssets` (zero), the `ConsumedAssets` guard passes even when `consumed == maxAssets`, allowing the offer to be taken an unlimited number of times after full consumption. Each such take mutates real position state (credit, debt, `totalUnits`) and fires the maker callback, all without transferring any assets.

## Finding Description
**Root cause** â€” `src/Midnight.sol` `take()`:

```solidity
// line 363
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;

// lines 367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For a buy offer, `consumed` is incremented by `buyerAssets`, not by `units`. When `buyerPrice < WAD` (any tick where `offerPrice < 1e18`) and `units = 1`, `mulDivDown(1, buyerPrice, WAD)` truncates to `0`. Adding `0` to a `consumed` already at `maxAssets` yields `newConsumed == maxAssets`, satisfying `<= maxAssets`. The guard is bypassed.

**Exploit flow**:
1. Attacker observes a buy offer with `maxAssets > 0` and `offerPrice < WAD` (e.g., `tick = MAX_TICK - 16`).
2. Offer is fully consumed: `consumed[maker][group] == maxAssets`.
3. Attacker calls `take(units=1, ...)`. `buyerAssets = 0`.
4. `newConsumed = maxAssets + 0 = maxAssets`. Check passes.
5. Position accounting executes with `units = 1`: buyer gains 1 unit of credit, seller gains 1 unit of debt, `totalUnits` increases by 1 (lines 408â€“417).
6. Maker callback fires with `buyerAssets=0, units=1`.
7. No assets are transferred. Steps 3â€“6 repeat indefinitely.

**Why existing checks fail**: The `ConsumedAssets` check at line 369 is asset-denominated, but position mutation and callback use `units` (unit-denominated). There is no guard rejecting a take when `buyerAssets == 0` and `units > 0`.

The protocol's own test suite documents this exact behavior in `testBugBuyMaxAssetsBypass` (lines 857â€“889 of `test/TakeTest.sol`), and the NatDoc at line 94 of `src/Midnight.sol` explicitly states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*

## Impact Explanation
A fully-consumed buy offer (assets mode, `buyerPrice < WAD`) can be taken an unlimited number of times by any unprivileged taker. Each extra take increases the seller's debt and the buyer's credit by 1 unit at zero cost, inflates `totalUnits` and `marketState` accounting, and fires the maker's `onBuy` callback with `buyerAssets=0, units=1`. This violates the core invariant that `consumed == maxAssets` means the offer is exhausted, constitutes unauthorized state mutation (credit/debt/totalUnits inflation without asset backing), and may trigger unintended maker-side logic via the callback.

## Likelihood Explanation
Both preconditions â€” a buy offer with `maxAssets > 0` and `offerPrice < WAD`, and the offer being fully consumed â€” are normal, reachable protocol states. The tick range `offerPrice < WAD` is explicitly supported and documented. Any unprivileged taker can call `take()` directly. The attack is repeatable in a single transaction via multicall or a loop, with zero cost to the attacker.

## Recommendation
Add a guard in the `maxAssets` branch that rejects a take when the computed asset amount rounds to zero but `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0 || units == 0, ZeroAssetTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, track consumed in units (not assets) when `maxAssets > 0` and `offerPrice < WAD`, or enforce a minimum `units` value such that `buyerAssets >= 1` always holds.
