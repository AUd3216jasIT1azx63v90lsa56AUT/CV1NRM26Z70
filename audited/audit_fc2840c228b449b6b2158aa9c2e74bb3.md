Audit Report

## Title
`onBuy` Callback Reentrancy Drains `withdrawable` Pool Before Asset Transfer - (File: `src/Midnight.sol`)

## Summary
In `take`, buyer credit is written to storage at line 410 and `totalUnits` is updated at line 416–417 before the external `onBuy` callback fires at lines 448–452, while the token pull from the payer does not occur until lines 455–456 after the callback returns. An attacker controlling the `buyerCallback` contract can reenter `withdraw` during `onBuy`, redeeming the freshly committed credit against the pre-existing `withdrawable` pool to receive `units` tokens, then using those tokens to satisfy the payer transfer of only `buyerAssets < units`. The net result is extraction of `units − buyerAssets` tokens from other lenders' withdrawable balance in a single atomic transaction.

## Finding Description

**Root cause:** `take` commits all buyer-side state mutations (`buyerPos.credit`, `totalUnits`, `claimableSettlementFee`) before making the external `onBuy` call, violating the checks-effects-interactions pattern. `withdraw` has no reentrancy guard and no transient-storage lock preventing it from being called mid-`take`.

**Exact code path:**

`take` (sell offer, `offer.buy = false`):
- Line 375: `buyer = taker` (attacker), `seller = offer.maker`
- Line 382: `buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)` = `units` (attacker has no prior debt)
- Line 410: `buyerPos.credit += units` — credit committed to storage before any external call
- Lines 416–417: `_marketState.totalUnits += units` — committed to storage
- Line 418: `claimableSettlementFee[...] += buyerAssets - sellerAssets` — committed to storage
- Line 420: `buyerCallback = takerCallback` (attacker-controlled contract C); line 422: `payer = C`
- Line 444: `LIQUIDATION_LOCK_SLOT` set only for the *seller*; no lock on `withdraw`
- Lines 448–452: `IBuyCallback(buyerCallback).onBuy(...)` — external call with no reentrancy protection
- Lines 455–456: `safeTransferFrom(payer, ...)` — token pull happens **after** callback returns

`withdraw` (called inside `onBuy` by C on behalf of attacker):
- Line 482: `isAuthorized[attacker][C]` — satisfied via prior `setIsAuthorized(C, true, attacker)`
- Line 485: `_updatePosition` — no-op in same block (same `lastAccrual`, no intervening bad debt)
- Line 493: `_position.credit -= units` — succeeds; credit was just set to `units` at line 410
- Line 494: `_marketState.withdrawable -= units` — succeeds if `withdrawable >= units`; no guard prevents this during a `take` callback
- Line 495: `_marketState.totalUnits -= units` — cancels the `totalUnits` increase from `take`
- Line 499: `safeTransfer(loanToken, C, units)` — `units` tokens leave the contract

**Exploit flow:**
1. Attacker calls `setIsAuthorized(C, true, attacker)`.
2. Market has `withdrawable = W >= units` (from any prior repayment or liquidation).
3. Attacker calls `take(sellOffer, ..., attacker, ..., C, ...)` with `units` chosen so `buyerAssets = units × buyerPrice / WAD < units` (any tick with `buyerPrice < WAD`).
4. `take` writes `buyerPos.credit += units` (line 410), then calls `C.onBuy(...)` (line 448).
5. Inside `C.onBuy`: C calls `midnight.withdraw(market, units, attacker, C)`.
   - `attacker.credit -= units` → 0
   - `withdrawable -= units` → `W − units`
   - `totalUnits` reverts to pre-take value
   - Contract sends `units` tokens to C
6. `C.onBuy` returns `CALLBACK_SUCCESS`.
7. `take` resumes: `safeTransferFrom(C, address(this), buyerAssets − sellerAssets)` and `safeTransferFrom(C, receiver, sellerAssets)` — C pays only `buyerAssets` total using the tokens it just received.
8. C retains `units − buyerAssets` tokens as profit.
9. Line 476 health check on the seller passes if the seller has sufficient collateral (normal for any active sell offer).

**Why existing checks fail:**
- `LIQUIDATION_LOCK_SLOT` (line 444) only blocks liquidation of the seller; it does not block `withdraw`.
- There is no reentrancy guard on `withdraw` or on the `onBuy` callback section of `take`.
- The TOKEN SAFETY REQUIREMENTS (lines 133–140) prohibit re-entry from token `transfer`/`transferFrom` calls, not from `onBuy` callbacks — this attack re-enters through the callback, not the token.
- `withdraw` performs no check that `withdrawable >= credit` beyond the implicit Solidity 0.8 underflow revert, which the attacker satisfies by precondition.

## Impact Explanation

The attacker extracts `units − buyerAssets = units × (WAD − buyerPrice) / WAD` tokens per call from the `withdrawable` pool. These tokens represent repaid loan assets owed to other lenders. After the attack, `withdrawable` is permanently reduced by `units` while the attacker's credit is zero and no new net debt is outstanding in `totalUnits`. Other lenders holding credit cannot redeem their proportional share until new repayments refill the pool. The entire `withdrawable` pool can be drained in a single transaction by setting `units = withdrawable`. This constitutes direct, irreversible theft of lender funds — a critical impact matching the "direct loss of user funds" and "protocol insolvency" categories.

## Likelihood Explanation

**Preconditions:**
1. `withdrawable > 0` — satisfied any time a borrower has repaid or been liquidated (normal market operation).
2. A sell offer exists at any tick with `buyerPrice < WAD` — standard discount lending, always present in active markets.
3. Attacker pre-authorizes their callback contract via one `setIsAuthorized` call — no privilege required.

The attack is fully permissionless, requires no oracle manipulation, no admin access, and no special token behavior. It is executable in a single transaction and is repeatable across any market with repayment history.

## Recommendation

Apply the checks-effects-interactions pattern strictly: move the `safeTransferFrom` token pulls (lines 455–456) to **before** the `onBuy` callback (lines 448–452), so the payer must deliver tokens before any external call is made. Alternatively, introduce a transient-storage reentrancy lock (analogous to `LIQUIDATION_LOCK_SLOT`) that is set at the start of `take` and checked at the start of `withdraw`, `repay`, and any other state-mutating function, preventing reentrant calls during the callback window. The lock must cover `withdraw` specifically, not just `liquidate`.

## Proof of Concept

**Minimal Foundry test plan:**

```solidity
// 1. Deploy MockBuyCallback contract C that, inside onBuy, calls:
//    midnight.withdraw(market, units, attacker, address(C))
//    and then returns CALLBACK_SUCCESS.

// 2. Setup:
//    - Create market with loanToken = USDC
//    - Lender supplies credit; borrower borrows and repays, leaving withdrawable = W

// 3. Attack:
//    attacker.setIsAuthorized(address(C), true, attacker);
//    // Place or find a sell offer at tick T where buyerPrice < WAD
//    uint256 units = W; // drain entire pool
//    midnight.take(sellOffer, ratifierData, units, attacker, receiver, address(C), "");

// 4. Assertions:
//    assertEq(midnight.withdrawable(id), 0);           // pool drained
//    assertEq(USDC.balanceOf(address(C)), units - buyerAssets); // profit retained
//    assertEq(midnight.creditOf(id, attacker), 0);    // credit zeroed
//    // Contract token balance decreased by (units - buyerAssets)
```

The test confirms that C's token balance increases by `units − buyerAssets` and `withdrawable` drops to zero, with no revert, in a single transaction.
