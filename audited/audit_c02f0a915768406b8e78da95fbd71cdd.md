### Title
Unbounded `_poolsByAgreement` array causes `getPoolsByAgreement` OOG DoS for on-chain integrations â€” (`File: src/ConfidencePoolFactory.sol`)

### Summary
`getPoolsByAgreement` returns the entire `_poolsByAgreement[agreement]` storage array in one call with no pagination or size cap. A pool sponsor (agreement owner) can call `createPool` N times for the same agreement, growing the array to N entries. Any on-chain consumer that calls `getPoolsByAgreement` inside a gas-constrained context (e.g., a callback forwarding â‰¤100 000 gas) will OOG once N exceeds the cold-SLOAD budget, permanently DoS-ing that integration for the affected agreement.

### Finding Description
**Root cause â€” `ConfidencePoolFactory.sol` lines 117â€“119:**

```solidity
function getPoolsByAgreement(address agreement) external view returns (address[] memory) {
    return _poolsByAgreement[agreement];
}
```

The function performs one cold SLOAD for the array length and one cold SLOAD per element (each 20-byte address occupies one 32-byte storage slot). There is no pagination, no size guard, and no gas check.

**Growth path â€” `createPool` lines 86â€“103:**

```solidity
bytes32 salt = keccak256(abi.encode(agreement, _poolsByAgreement[agreement].length));
pool = Clones.cloneDeterministic(poolImplementation, salt);
// ...
_poolsByAgreement[agreement].push(pool);
```

`createPool` requires only: (a) `allowedStakeToken[stakeToken] == true`, (b) `expiry >= block.timestamp + 30 days`, (c) `safeHarborRegistry.isAgreementValid(agreement) == true`, and (d) `IAgreement(agreement).owner() == msg.sender`. All four are reachable by the agreement owner under normal operating conditions. There is no per-agreement pool cap and no fee beyond gas.

**Reachable sequence:**
1. Sponsor holds a valid agreement and the factory has an allowlisted token.
2. Sponsor calls `createPool` N times (same agreement, same or different parameters). Each call costs ~300 000â€“400 000 gas (clone deploy + initialize) but is otherwise unconstrained.
3. `_poolsByAgreement[agreement]` now has N entries.
4. A third-party contract calls `factory.getPoolsByAgreement{gas: 100_000}(agreement)`.
5. Gas consumed: ~2 100 (length SLOAD) + N Ã— ~2 103 (element SLOAD + MSTORE) + call overhead. At N â‰ˆ 45 the budget is exhausted and the call reverts OOG.

**Failed guards:** None exist. `getPoolsByAgreement` has no gas check, no pagination, and no maximum-array-length enforcement. `poolCountByAgreement` exists as a separate view but provides no protection to callers of `getPoolsByAgreement`.

### Impact Explanation
Any on-chain contract that calls `getPoolsByAgreement` inside a gas-forwarded callback, a try/catch with a gas cap, or any context where the available gas is bounded (e.g., ERC-4337 validation, cross-chain message handlers, aggregator contracts) will revert with OOG once N is large enough. The DoS is permanent for that agreement: the array only grows, never shrinks, and there is no recovery path short of a factory upgrade. The core protocol (stake/resolve/claim) is unaffected, but any integration relying on this view function for on-chain routing or gating is bricked for the targeted agreement.

### Likelihood Explanation
The actor is the pool sponsor â€” a semi-trusted but unprivileged role that requires no moderator or owner cooperation. Prerequisites (valid agreement, allowlisted token) are the normal operating state of the protocol. The attack is repeatable and cheap relative to its effect: ~45 `createPool` transactions (each ~300 k gas) suffice to OOG a 100 k-gas callback. A sponsor acting in bad faith, or one who creates many pools for legitimate reasons, can trigger this unintentionally. The protocol comment at line 34 ("duplicates are allowed and curated off-chain") confirms the design anticipates many pools per agreement without bounding them.

### Recommendation
Add a paginated variant and document that `getPoolsByAgreement` is off-chain only:

```solidity
/// @notice Paginated pool list. Use off-chain; unbounded call may OOG.
function getPoolsByAgreement(address agreement, uint256 offset, uint256 limit)
    external view returns (address[] memory page)
{
    address[] storage all = _poolsByAgreement[agreement];
    uint256 total = all.length;
    if (offset >= total) return new address[](0);
    uint256 end = offset + limit > total ? total : offset + limit;
    page = new address[](end - offset);
    for (uint256 i = offset; i < end; i++) {
        page[i - offset] = all[i];
    }
}
```

Alternatively, enforce a per-agreement pool cap in `createPool` (e.g., `require(_poolsByAgreement[agreement].length < MAX_POOLS_PER_AGREEMENT)`). The existing `poolCountByAgreement` already lets callers check the count before paginating.
