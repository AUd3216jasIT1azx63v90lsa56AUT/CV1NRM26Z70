Audit Report

## Title
`mulDivUp` Arithmetic Overflow in `consumableUnits` for Buy Offers with Oversized `maxAssets` - (File: `src/periphery/ConsumableUnitsLib.sol`)

## Summary
When `consumableUnits` is called for a buy offer with `maxUnits == 0` and `maxAssets > type(uint256).max / WAD` (~1.157e59), the intermediate product `targetBuyerAssets * WAD` in `UtilsLib.mulDivUp` overflows under checked Solidity 0.8.x arithmetic, causing an unconditional revert. Because `MidnightBundles.supplyCollateralAndSellWithUnitsTarget` and `supplyCollateralAndSellWithAssetsTarget` call `ConsumableUnitsLib.consumableUnits` outside any `try/catch`, a single crafted offer in the `takes` array permanently DoS-es the entire bundle call.

## Finding Description

**Step 1 — `ConsumableUnitsLib.consumableUnits`:**
With `offer.maxUnits == 0` and `offer.buy == true`, the function enters the `else if` branch:
```solidity
return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
```
With `consumed == 0`, `targetBuyerAssets = offer.maxAssets`. [1](#0-0) 

**Step 2 — `TakeAmountsLib.buyerAssetsToUnits`:**
For a buy offer, line 29 executes:
```solidity
return offer.buy ? targetBuyerAssets.mulDivUp(WAD, buyerPrice) : ...
``` [2](#0-1) 

**Step 3 — `UtilsLib.mulDivUp`:**
The implementation has no `unchecked` block:
```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```
Under Solidity `^0.8.0`, `x * y` reverts on overflow when `x > type(uint256).max / y`. With `y = WAD = 1e18`, any `targetBuyerAssets > type(uint256).max / 1e18` (~1.157e59) causes an unconditional overflow revert. [3](#0-2) 

**Why existing checks fail:**
- `require(buyerPrice <= WAD, ...)` at line 28 of `TakeAmountsLib` only bounds `buyerPrice` from above; it does not constrain `targetBuyerAssets`. [4](#0-3) 
- `zeroFloorSub` only floors at zero; it does not cap `targetBuyerAssets`. [5](#0-4) 
- No validation on `offer.maxAssets` magnitude exists anywhere in the call chain.

**Propagation to `MidnightBundles`:**
`supplyCollateralAndSellWithUnitsTarget` (line 150) and `supplyCollateralAndSellWithAssetsTarget` (line 290) both call `ConsumableUnitsLib.consumableUnits` **before** the `try IMidnight(MIDNIGHT).take(...)` block. The `try/catch` does not protect against this revert. The contract's own NatSpec documents: *"Reverts if ConsumableUnitsLib reverts."* [6](#0-5) [7](#0-6) 

## Impact Explanation
Any call to `supplyCollateralAndSellWithUnitsTarget` or `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer with `maxUnits == 0` and `maxAssets > type(uint256).max / 1e18` reverts unconditionally. This permanently DoS-es those bundle functions for any `takes` array containing such an offer. Off-chain routing systems and aggregators that call `consumableUnits` to compute fill amounts are similarly broken for the affected offer class. The core `Midnight.take` is unaffected. [8](#0-7) 

## Likelihood Explanation
A maker needs only to sign an offer with `buy = true`, `maxUnits = 0`, and `maxAssets = type(uint256).max`. No special privilege, no oracle manipulation, and no existing state is required. The condition is trivially reachable on first call with zero prior consumption. Any routing system or aggregator that automatically ingests on-chain or off-chain offers can be fed this poisoned offer, causing bundle-level reverts for legitimate takers. [9](#0-8) 

## Recommendation
Replace the naive `mulDivUp` with an overflow-safe implementation using the 512-bit full-precision multiply (e.g., Solady's or OpenZeppelin's `Math.mulDiv`) that avoids intermediate overflow. Alternatively, add an explicit cap or revert guard in `buyerAssetsToUnits` before calling `mulDivUp`:
```solidity
require(targetBuyerAssets <= type(uint256).max / WAD, "targetBuyerAssets overflow");
```
The most robust fix is to replace `UtilsLib.mulDivUp` and `mulDivDown` with overflow-safe 512-bit implementations throughout, since the same pattern exists in `mulDivDown`. [10](#0-9)
