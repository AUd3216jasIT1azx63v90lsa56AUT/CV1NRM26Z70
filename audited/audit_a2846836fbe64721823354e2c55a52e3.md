Audit Report

## Title
Assets-mode buy offer with `offerPrice < WAD` allows unbounded credit inflation past `maxAssets` cap — (File: src/Midnight.sol)

## Summary
When a buy offer uses `maxAssets` mode and `offerPrice < WAD`, calling `take()` with `units` small enough that `units * buyerPrice < WAD` causes `buyerAssets` to round to zero via `mulDivDown`. The consumed-cap check increments by zero and passes even when the offer is fully consumed. Position state (`credit`, `debt`, `totalUnits`) is updated using `units` directly while zero tokens are transferred, allowing an attacker controlling two addresses to inflate lender credit without depositing any loan tokens and subsequently drain `withdrawable` tokens deposited by legitimate borrower repayments.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–364:**

When `offer.buy = true` and `units * buyerPrice < WAD`, `mulDivDown` returns 0:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD); // = 0
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD); // = 0
```

The consumed accumulator is then incremented by 0 at lines 367–369:

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets; // += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets()); // passes: consumed unchanged
}
```

Position accounting uses `units` directly at lines 382–417:

```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt); // = units if no debt
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt  += UtilsLib.toUint128(sellerDebtIncrease);
_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease;
```

Token transfer is zero at line 455:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // 0 - 0 = 0
```

The protocol acknowledges the gap at line 94:

```
/// @dev It is possible to give units to a fully consumed assets-based buy offer with price < 1.
```

The existing test `testBugBuyMaxAssetsBypass` at `test/TakeTest.sol` lines 857–889 explicitly demonstrates the behavior: sets `maxAssets = 1`, tick `MAX_TICK - 16` (offerPrice < WAD), pre-consumes the offer to `maxAssets`, then calls `take(1, borrower, lenderOffer)` and asserts `buyerAssets == 0`, `sellerAssets == 0`, token balances unchanged, but `creditOf(id, lender)` increased, `debtOf(id, borrower)` increased, `totalUnits(id)` increased.

**Exploit flow:**
1. Attacker controls address A (maker/buyer) and address B (taker/seller, authorized via `isAuthorized`).
2. A creates a buy offer: `maxAssets = 1`, tick chosen so `offerPrice < WAD` (e.g. `MAX_TICK - 16`).
3. A calls `setConsumed(group, 1, A)` to set `consumed[A][group] = maxAssets = 1`.
4. B supplies sufficient collateral to remain healthy under growing debt.
5. B calls `take(offer, ..., units=1, B, ...)` in a loop: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; consumed check passes; A's `credit += 1`, B's `debt += 1`, zero tokens transferred.
6. After N iterations, A has N inflated credit, B has N debt backed by collateral.
7. A calls `withdraw(N)`: `_position.credit -= N` succeeds (inflated), `_marketState.withdrawable -= N` succeeds if legitimate borrowers have repaid, `safeTransfer(loanToken, receiver, N)` drains their funds.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` only prevents `consumed` from exceeding the cap; it does not guard against `units > 0` when `buyerAssets == 0`.
- There is no `require(buyerAssets > 0)` or equivalent check anywhere in `take()`.
- `SelfTake()` at line 354 is bypassed by using two distinct attacker-controlled addresses.
- The `isHealthy` check at line 476 is satisfied as long as B has supplied adequate collateral (a separate token, not the loan token).

## Impact Explanation

The attacker inflates their `credit` by an arbitrary amount without depositing loan tokens. `withdraw()` at line 499 transfers `units` of `loanToken` per credit unit redeemed, decrementing `_marketState.withdrawable`. Any `withdrawable` balance accumulated from legitimate borrower repayments (via `repay()` at line 509) or liquidations can be fully drained. `totalUnits` grows without token backing, permanently breaking the core invariant that contract balances cover all credit redemptions. This constitutes direct theft of user funds — a critical in-scope impact.

## Likelihood Explanation

All preconditions are attacker-controlled: the attacker deploys two addresses, creates the offer themselves, and sets `consumed == maxAssets` with a single `setConsumed` call (no legitimate fill required). The tick range where `offerPrice < WAD` covers a large portion of valid ticks. The attack is repeatable in a single transaction via `multicall`, with no economic cost beyond gas and collateral (which is not the loan token and is not lost). The protocol's own comment at line 94 and the test `testBugBuyMaxAssetsBypass` document the behavior as a known, reachable edge case without a guard.

## Recommendation

Add a guard in `take()` immediately after computing `buyerAssets` and `sellerAssets` to reject zero-asset takes when `units > 0` and `maxAssets > 0`:

```solidity
if (offer.maxAssets > 0 && units > 0) {
    require(
        offer.buy ? buyerAssets > 0 : sellerAssets > 0,
        ZeroAssets()
    );
}
```

Alternatively, switch the consumed accumulator to track `units` instead of assets when `offerPrice < WAD` could cause rounding to zero, or enforce a minimum tick such that `offerPrice >= WAD` for `maxAssets`-mode buy offers.

## Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` at lines 857–889 is a complete, passing proof of concept. It demonstrates:
- `buyerAssets == 0`, `sellerAssets == 0` (zero tokens transferred)
- `consumed` unchanged at `maxAssets` (cap bypass confirmed)
- `creditOf(id, lender)` strictly increased (credit inflated without deposit)
- `debtOf(id, borrower)` strictly increased
- `totalUnits(id)` strictly increased

To demonstrate fund drainage, extend the test: have a third address repay debt (incrementing `withdrawable`), then call `withdraw(N)` from the inflated lender address and assert the third party's repaid tokens are transferred out.
