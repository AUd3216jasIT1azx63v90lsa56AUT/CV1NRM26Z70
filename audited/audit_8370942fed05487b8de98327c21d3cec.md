Audit Report

## Title
`sellerAssetsToUnits` reverts on division-by-zero when `offerPrice == settlementFee`, DoS-ing `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/TakeAmountsLib.sol)

## Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers. When `offerPrice == settlementFee`, `sellerPrice = 0`, and the subsequent `mulDivUp(WAD, 0)` call underflows on `(d - 1)` under Solidity 0.8+ checked arithmetic, reverting unconditionally. Because `supplyCollateralAndSellWithAssetsTarget` invokes `sellerAssetsToUnits` outside its `try/catch` block, the revert propagates to the caller, permanently DoS-ing that entry-point for any bundle whose `takes[]` array includes such an offer. The core `Midnight.take()` handles this case without reverting, making the DoS exclusive to the periphery.

## Finding Description

**Root cause — `TakeAmountsLib.sellerAssetsToUnits` (`src/periphery/TakeAmountsLib.sol` lines 44–46):**

When `offer.buy == true` and `offerPrice == settlementFee`, line 44 computes `sellerPrice = offerPrice - settlementFee = 0` without underflowing (equal values). Line 46 then calls `targetSellerAssets.mulDivUp(WAD, 0)`.

**`mulDivUp` underflows with `d = 0` (`src/libraries/UtilsLib.sol` lines 34–36):**

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```

With `d = 0`, the sub-expression `(d - 1)` = `(0 - 1)` underflows under Solidity 0.8+ checked arithmetic, reverting unconditionally before the division executes.

**Core `Midnight.take()` does NOT revert in the same scenario (`src/Midnight.sol` lines 361–364):**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;  // = 0
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...;  // = 0
```

`mulDivDown` is `(x * y) / d` with no `(d - 1)` term, so `mulDivDown(units, 0, WAD) = 0`. No underflow, no revert. The seller receives zero assets and the take succeeds.

**`supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside the `try/catch` (`src/periphery/MidnightBundles.sol` lines 285–300):**

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(   // <-- unguarded; revert propagates here
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) returns (...) { ... } catch {}
```

The `try/catch` only wraps `IMidnight.take()`. The `sellerAssetsToUnits` call preceding it is entirely unguarded.

**Existing protections are insufficient:**

The NatSpec at line 34 of `TakeAmountsLib.sol` documents only the `offerPrice < settlementFee` revert path (which underflows at line 44 itself). The `offerPrice == settlementFee` path silently passes line 44 with `sellerPrice = 0` and then reverts inside `mulDivUp`, which is undocumented and unguarded. The `try/catch` in `supplyCollateralAndSellWithAssetsTarget` provides no protection against this pre-`take` revert.

## Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` whose `takes[]` array contains a buy offer where `tickToPrice(offer.tick) == settlementFee` reverts entirely, including the collateral supply steps that precede the loop. An attacker can grief any taker who includes that offer, permanently blocking the periphery sell path for that bundle. Because the offer itself is valid and `Midnight.take()` would accept it (seller receives zero assets), the DoS is invisible to the taker until the bundle reverts. This constitutes a griefing/DoS of a core periphery entry-point with no capital cost to the attacker.

## Likelihood Explanation
Settlement fees are discrete governance-set values stored in CBP units (multiples of 1e-6 WAD). Tick prices are also discrete. The condition `tickToPrice(T) == settlementFee` is achievable whenever the fee coincides with any accessible tick price — a realistic scenario given the fee range (up to 50 bps = 5e15) and the tick price space. An unprivileged attacker can monitor the fee, identify the matching tick, and post a buy offer there at negligible cost (a buy offer with `maxUnits = 1` requires minimal capital). The attack is repeatable after each cancellation. The victim's off-chain routing may include this offer if it appears as the best or only available offer at that price level.

## Recommendation
Two complementary fixes should be applied:

1. **Guard `sellerAssetsToUnits` against `sellerPrice == 0`:** Add an explicit check or return a sentinel value (e.g., `type(uint256).max`) when `sellerPrice == 0`, so the caller can handle the degenerate case without reverting:
   ```solidity
   if (sellerPrice == 0) return type(uint256).max; // no finite units can yield nonzero sellerAssets
   ```

2. **Wrap the entire per-offer computation in `supplyCollateralAndSellWithAssetsTarget` in a `try/catch`:** Move the `sellerAssetsToUnits` and `ConsumableUnitsLib.consumableUnits` calls inside a helper that catches reverts, consistent with the function's documented intent to skip offers that revert. Alternatively, wrap the `unitsToTake` computation in a `try/catch` and skip the offer on revert, matching the behavior of `supplyCollateralAndSellWithUnitsTarget`.

## Proof of Concept
**Minimal manual steps:**
1. Deploy `Midnight` and `MidnightBundles` on a fork.
2. Set the settlement fee for a market such that `settlementFee(id, ttm) == tickToPrice(T)` for some tick `T`.
3. Have an attacker post a buy offer at tick `T` with `maxUnits = 1`.
4. Have a victim call `supplyCollateralAndSellWithAssetsTarget` with `takes[0]` pointing to that offer.
5. Observe the transaction reverts with an arithmetic underflow originating in `mulDivUp`.
6. Call `Midnight.take()` directly with the same offer and observe it succeeds (returning `sellerAssets = 0`), confirming the DoS is exclusive to the periphery path.

**Fuzz test plan:** Fuzz `offerPrice` and `settlementFee` over the valid CBP-aligned range; assert that whenever `Midnight.take()` succeeds, `supplyCollateralAndSellWithAssetsTarget` also does not revert due to `sellerAssetsToUnits`.
