### Title
Assets-mode buy offer with `offerPrice < WAD` allows unlimited unbacked credit/debt creation past `maxAssets` cap - (`src/Midnight.sol`)

### Summary
When a buy offer uses `maxAssets` mode and `offerPrice < WAD`, the consumed-cap check increments `consumed` by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`, which rounds to zero for small `units`. This allows a taker to call `take()` with `units > 0` on a fully-consumed offer, bypassing the cap entirely. Position state (`credit`, `debt`, `totalUnits`) is updated using `units` directly, creating unbacked credit and debt with zero token transfer.

### Finding Description

**Code path — `src/Midnight.sol` lines 363–418:**

```
buyerAssets = units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → 0 when units*buyerPrice < WAD
sellerAssets = units.mulDivDown(sellerPrice, WAD)  // also 0

// consumed accounting (line 368):
newConsumed = consumed[maker][group] += buyerAssets  // += 0, stays at maxAssets
require(newConsumed <= offer.maxAssets)              // maxAssets <= maxAssets → PASSES

// position accounting (lines 382–417) uses `units`, not `buyerAssets`:
buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)  // = units if buyer has no debt
buyerPos.credit += buyerCreditIncrease                     // lender credit grows
sellerPos.debt  += sellerDebtIncrease                      // borrower debt grows
totalUnits      += buyerCreditIncrease

// token transfer (line 455):
safeTransferFrom(loanToken, payer, this, buyerAssets - sellerAssets)  // = 0, no tokens move
```

**Preconditions (all attacker-controllable):**
1. `offer.buy = true`, `offer.maxAssets > 0`
2. `offer.tick` set so `offerPrice < WAD` (e.g. `MAX_TICK - 16`)
3. `consumed[maker][group] == maxAssets` (offer fully consumed — maker can self-set via `setConsumed`, or it was legitimately filled)
4. `units` chosen so `units * buyerPrice < WAD` (e.g. `units = 1` with `buyerPrice < WAD`)

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = 1`, `tick = MAX_TICK - 16` (price < 1e18)
2. Offer is fully consumed: `consumed[maker][group] = 1`
3. Unprivileged taker calls `take(offer, ..., units=1, taker, ...)`
4. `buyerAssets = 1.mulDivDown(buyerPrice, WAD) = 0` → consumed check passes
5. `buyerCreditIncrease = 1`, `sellerDebtIncrease = 1`
6. Lender's `credit += 1`, borrower's `debt += 1`, `totalUnits += 1`
7. Zero tokens transferred
8. Step 3–7 repeatable indefinitely

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` at line 369 only guards against `consumed` exceeding the cap; it does not guard against `units > 0` when `buyerAssets == 0`
- No check enforces `units == 0` when `buyerAssets == 0` in assets mode
- The protocol comment at line 94 explicitly acknowledges this: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1"* — confirming the gap is known but unguarded [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The maker (lender) accumulates `credit` units without depositing any loan tokens. Since `withdraw()` at line 499 transfers `units` of `loanToken` out of the contract per credit unit redeemed, the attacker can drain loan tokens deposited by other lenders. `totalUnits` grows without token backing, breaking the core invariant that contract balances cover credit redemption. The borrower simultaneously accumulates `debt` without receiving any loan tokens, creating phantom debt that can be used to manipulate health checks and liquidation thresholds. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

Preconditions are fully attacker-controlled: any taker can target any buy offer whose `offerPrice < WAD` (a large fraction of the tick range) once it reaches `consumed == maxAssets`. The offer need not be griefed first — the taker can partially fill it to `maxAssets` themselves, then immediately exploit. The attack is repeatable in a single transaction via multicall or a loop, with no gas-cost barrier beyond the per-call overhead. The protocol's own comment documents the behavior, indicating it is a known edge case without a guard. [3](#0-2) 

### Recommendation

In the `maxAssets` branch, add a guard that treats a zero-asset take as a no-op when `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    // Prevent units > 0 from bypassing the cap when price rounds assets to 0
    require(units == 0 || assetsDelta > 0, ConsumedAssets());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any non-trivial take (`units > 0`) in assets mode must produce a non-zero asset delta, closing the rounding bypass. [6](#0-5)
