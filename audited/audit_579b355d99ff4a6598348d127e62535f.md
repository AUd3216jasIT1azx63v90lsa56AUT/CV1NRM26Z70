### Title
Bitmap Oracle Loop Lacks Try/Catch, Blocking Liquidation When Any Non-Liquidated Collateral Oracle Reverts - (File: src/Midnight.sol)

### Summary
In `liquidate`, the bitmap loop at lines 607–618 calls `IOracle(_collateralParam.oracle).price()` (line 610) for **every** activated collateral in the borrower's position with no `try/catch` or low-level error handling. If any oracle in the bitmap reverts — including one for a collateral that is not being liquidated — the entire `liquidate` call reverts. This breaks the core invariant that unhealthy positions must remain liquidatable.

### Finding Description
**Code path:**

`liquidate` (Midnight.sol:581) → bitmap loop (607–618) → `IOracle(_collateralParam.oracle).price()` (line 610, bare external call, no error handling) → revert propagates up → `liquidate` reverts before the `NotLiquidatable` check at line 620.

**Root cause:** The loop unconditionally queries every oracle in `_position.collateralBitmap`, but only the oracle at `collateralIndex` is strictly required to compute `liquidatedCollatPrice` (line 611) and the seized/repaid amounts (lines 650–652). The other oracle prices are used to compute `maxDebt` and `badDebt`, but a revert in any one of them aborts the entire transaction.

**Attacker inputs / exploit flow:**

1. A borrower activates two collaterals (indices 0 and 1) in a market. Both oracles are healthy at deposit time.
2. The oracle for collateral[1] begins reverting (e.g., a Chainlink heartbeat/circuit-breaker triggers, or the oracle contract is paused — all common, designed behaviors in production oracles).
3. The borrower's position becomes unhealthy because collateral[0]'s price drops below the LLTV threshold.
4. A liquidator calls `liquidate(market, 0, 0, repaidUnits, borrower, false, receiver, address(0), '')` targeting collateral[0].
5. The bitmap loop reaches index 1, calls `IOracle(oracle[1]).price()`, which reverts.
6. The revert propagates; `liquidate` reverts with no state change.
7. The unhealthy position cannot be liquidated by anyone as long as oracle[1] reverts.

**Why existing checks do not stop it:** There is no `try/catch`, no fallback price, and no mechanism to skip a reverting oracle. The `liquidatorGate` check (lines 597–600) and `liquidationLocked` check (lines 620–624) are never reached. The loop has no per-oracle error isolation. [1](#0-0) 

### Impact Explanation
An unhealthy position with 2+ collaterals becomes permanently unliquidatable for as long as any non-liquidated collateral's oracle reverts. Bad debt accrues without being socialized, lenders cannot recover funds, and the protocol's solvency invariant is violated. The position is effectively frozen: collateral cannot be seized, debt cannot be repaid via liquidation, and the `badDebt` accounting path (lines 626–641) is also unreachable. [2](#0-1) [3](#0-2) 

### Likelihood Explanation
**Preconditions:**
- Borrower must have ≥ 2 activated collaterals (common in multi-collateral markets).
- At least one non-liquidated collateral's oracle must revert. Production oracles (Chainlink, Pyth, Uniswap TWAP) routinely revert on stale heartbeat, circuit-breaker triggers, or sequencer downtime — these are designed behaviors, not impossible failures.
- The position must be unhealthy (collateral[0] price drop), which is the exact scenario where liquidation is most needed.

**Feasibility:** High. The preconditions co-occur naturally during market stress: price drops that make positions unhealthy also tend to trigger oracle circuit-breakers on correlated assets. The window can persist for hours or days (e.g., during a sequencer outage on L2s).

**Repeatability:** The block is deterministic and repeatable — every liquidation attempt reverts identically until oracle[1] recovers.

### Recommendation
Wrap each oracle call in the bitmap loop with a `try/catch` (or low-level `staticcall`). If the oracle for the **liquidated** collateral (`i == collateralIndex`) reverts, revert the transaction (price is required). If the oracle for a **non-liquidated** collateral reverts, use a fallback value of `0` for that collateral's contribution to `maxDebt` and `badDebt` (conservative: treats that collateral as worthless, which can only make the position appear *more* unhealthy, never less). This preserves the liquidatability invariant without introducing under-collateralization risk.

```solidity
// Suggested pattern inside the bitmap loop:
uint256 price;
if (i == collateralIndex) {
    price = IOracle(_collateralParam.oracle).price(); // must succeed
} else {
    try IOracle(_collateralParam.oracle).price() returns (uint256 p) {
        price = p;
    } catch {
        price = 0; // conservative: collateral[i] contributes 0 to maxDebt
    }
}
``` [4](#0-3)
