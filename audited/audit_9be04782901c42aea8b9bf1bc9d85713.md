Audit Report

## Title
Zero-asset rounding bypasses `maxAssets` offer cap, creating unbacked credit and debt with no token transfer — (`src/Midnight.sol`, `src/libraries/UtilsLib.sol`)

## Summary
When a buy offer has `maxAssets > 0` and `offerPrice < WAD`, a taker can supply a `units` value small enough that `mulDivDown(units, offerPrice, WAD)` truncates to zero. Because `consumed` is incremented by `buyerAssets` (which is 0), the cap is never advanced and the offer is never exhausted. Meanwhile, position credit and debt are updated using `units` directly, so every such call mints unbacked credit for the maker and unbacked debt for the taker with zero token transfer. This is confirmed by the repository's own test `testBugBuyMaxAssetsBypass`.

## Finding Description

**Root cause — `UtilsLib.mulDivDown` integer truncation:**

`src/libraries/UtilsLib.sol` lines 29–31:
```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
```
When `x * y < d`, the result is 0.

**Vulnerable code path — `src/Midnight.sol` lines 358–373:**

For a buy offer, `buyerPrice = (offerPrice − fee) + fee = offerPrice`. When `units * offerPrice < WAD`:
```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // → 0
    : units.mulDivUp(buyerPrice, WAD);
uint256 sellerAssets = offer.buy
    ? units.mulDivDown(sellerPrice, WAD)  // → 0
    : units.mulDivUp(sellerPrice, WAD);
```
Both asset values are 0. The `consumed` tracking then does:
```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    //                                                              ^^^^^^^^^^^ += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // passes unchanged
}
```
`consumed` does not advance. The require passes because the value is unchanged.

**Position state mutates using `units`, not `buyerAssets` (lines 382–414):**
```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt);
uint256 sellerDebtIncrease  = units - sellerCreditDecrease;
...
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt  += UtilsLib.toUint128(sellerDebtIncrease);
```
Credit and debt change by `units`, not by `buyerAssets`.

**Token transfers are zero (lines 455–456):**
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // 0
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);                    // 0
```

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` only enforces an upper bound; it does not require that `consumed` actually increased.
- There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in `take`.
- The `maxUnits` branch (line 371) uses `+= units` and is immune; only the `maxAssets` branch is affected.

**Confirmed by existing test** `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol` lines 857–889): the test pre-fills `consumed` to `maxAssets`, then calls `take(1, borrower, lenderOffer)` and asserts `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.

**Developer acknowledgment:** The NatSpec at `src/Midnight.sol` line 94 states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* This documents awareness of the behavior; the test name `testBugBuyMaxAssetsBypass` confirms it is classified as a bug.

## Impact Explanation
Every call with `units = U > 0` and `buyerAssets = 0`:
- Grants the buyer (maker) `U` units of credit with zero token payment.
- Burdens the seller (taker) with `U` units of debt with zero token receipt.
- Leaves `consumed` unchanged, so the offer is never exhausted regardless of `maxAssets`.
- Inflates `totalUnits` without a corresponding increase in the protocol's token balance, directly violating the solvency invariant.

Repeated indefinitely, this creates unbounded credit for the maker and unbounded debt for the taker without any token backing. When the taker later repays real tokens, those tokens flow to the maker via `withdrawable`, constituting a direct transfer of value from the taker to the maker with no legitimate exchange. `totalUnits` inflation also corrupts loss-factor accounting, harming all lenders in the market.

## Likelihood Explanation
**Preconditions:**
- A buy offer with `maxAssets > 0` must exist (common configuration).
- The offer's tick must satisfy `offerPrice < WAD` — true for every valid tick below `MAX_TICK` (5820), i.e., essentially all real offers.
- `units * offerPrice < WAD` for some reachable `units > 0` — true for any tick where `offerPrice < WAD`; the required `units` is `floor((WAD − 1) / offerPrice)`, computable off-chain.

**Feasibility:** Any unprivileged taker can compute the required `units` off-chain and call `take` directly. No special permissions, oracle manipulation, or token owner cooperation is required. The taker need not hold any tokens.

**Repeatability:** Unlimited — `consumed` never advances, so the offer never closes. The attack can be repeated in every block.

## Recommendation
Add a guard in the `maxAssets` branch of `take` that rejects a non-zero `units` input whenever the resulting asset amount rounds to zero:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || (offer.buy ? buyerAssets : sellerAssets) > 0, ZeroAssets());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` value such that the computed asset amount is always nonzero when `maxAssets > 0`, or switch the `consumed` tracking to always use `units` (consistent with the `maxUnits` branch) so the cap is always advanced.

```text
    // Show that a buy offer with offerPrice < WAD can be taken with units > 0
    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);
        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
```
