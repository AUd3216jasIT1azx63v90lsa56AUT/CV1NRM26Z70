Audit Report

## Title
Pending `EcrecoverAuthorizer` Signature Survives Direct Midnight Revocation — (`src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer` maintains a per-authorizer nonce in its own storage that is only incremented by calls through `EcrecoverAuthorizer.setIsAuthorized`. When a victim revokes an attacker's authorization by calling `Midnight.setIsAuthorized` directly, the `EcrecoverAuthorizer` nonce is untouched, leaving any pending signed authorization valid. The attacker can submit the withheld signature to restore their authorization, fully defeating the revocation.

## Finding Description

**Root cause — nonce isolation:**

`EcrecoverAuthorizer.nonce` is a standalone mapping in `EcrecoverAuthorizer` storage: [1](#0-0) 

It is only incremented inside `EcrecoverAuthorizer.setIsAuthorized`: [2](#0-1) 

`Midnight.setIsAuthorized` is a completely separate entry point that writes `isAuthorized[onBehalf][authorized]` but has zero knowledge of, and makes no change to, `EcrecoverAuthorizer.nonce`: [3](#0-2) 

There is no `invalidateNonce()` or equivalent function anywhere in `EcrecoverAuthorizer` or its interface: [4](#0-3) 

**Exploit flow:**

1. Victim calls `Midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim)`. `isAuthorized[victim][ecrecoverAuthorizer] = true`. `EcrecoverAuthorizer.nonce[victim] = 0`.
2. Victim signs `Authorization{authorizer: victim, authorized: attacker, isAuthorized: true, nonce: 0, deadline: block.timestamp + 365 days}` and shares it with the attacker (off-chain delegation, intercepted mempool tx, or a previously issued permit).
3. Attacker deliberately withholds submission.
4. Victim decides to revoke and calls `Midnight.setIsAuthorized(attacker, false, victim)`. `isAuthorized[victim][attacker]` is now `false`. `EcrecoverAuthorizer.nonce[victim]` is still `0`.
5. Attacker submits the withheld signature to `EcrecoverAuthorizer.setIsAuthorized`:
   - Deadline check passes (far future).
   - Nonce check passes: `0 == nonce[victim]++` → `0 == 0` ✓.
   - Signature valid (victim signed it) → `signer == authorization.authorizer` ✓.
   - `EcrecoverAuthorizer` calls `Midnight.setIsAuthorized(attacker, true, victim)` with `msg.sender = ecrecoverAuthorizer`.
   - `Midnight` authorization check: `isAuthorized[victim][ecrecoverAuthorizer] == true` ✓ (victim never revoked the authorizer itself). [5](#0-4) 
6. `isAuthorized[victim][attacker]` is `true` again. Revocation is undone.

**Why existing checks fail:**

The nonce prevents *true replay* of an already-consumed signature, but it does not help the victim invalidate a *pending* signature. The victim's only escape is to also revoke `EcrecoverAuthorizer`'s own authorization via `Midnight.setIsAuthorized(ecrecoverAuthorizer, false, victim)`, but nothing in the protocol enforces or documents this as a required step. The AUTHORIZATIONS comment in `Midnight.sol` warns generically that "authorized accounts can authorize other accounts on behalf of the user," but does not address nonce invalidation: [6](#0-5) 

## Impact Explanation
An attacker holding any pending (signed, deadline-not-expired) `EcrecoverAuthorizer` authorization for `isAuthorized: true` can restore their own authorization at any time after the victim revokes it via the direct Midnight path. This gives the attacker persistent, victim-unrevocable access to all operations gated by `isAuthorized`: `take` on behalf, `repay` on behalf, `withdraw` on behalf, `setConsumed`, and `setIsAuthorized` chaining (the attacker can authorize further addresses on the victim's behalf). The impact is unauthorized state mutation and potential asset loss across all markets where the victim participates. [7](#0-6) [8](#0-7) [9](#0-8) 

## Likelihood Explanation
Preconditions are realistic: off-chain delegation flows routinely produce signed-but-not-yet-submitted authorizations. The attacker only needs to withhold submission until after the victim's revocation transaction confirms, then submit. The attack is repeatable for every pending signature the attacker holds, is not blocked by any on-chain mechanism, and requires no privileged access — only possession of a valid signed `Authorization` struct.

## Recommendation
Add a dedicated nonce-invalidation mechanism to `EcrecoverAuthorizer` that allows the authorizer to increment their own nonce without consuming an authorization. For example:

```solidity
function invalidateNonce(address authorizer) external {
    require(msg.sender == authorizer || IMidnight(MIDNIGHT).isAuthorized(authorizer, msg.sender));
    nonce[authorizer]++;
    emit NonceInvalidated(authorizer, nonce[authorizer]);
}
```

Alternatively, tie the nonce to the `Midnight.isAuthorized` state: when `Midnight.setIsAuthorized` is called to revoke an address, it should call back into `EcrecoverAuthorizer` to increment the nonce. However, the simpler and more robust fix is the standalone `invalidateNonce` function, which allows victims to proactively burn any pending signatures without revoking the entire authorizer.

## Proof of Concept

**Minimal manual steps:**

1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. As victim: call `midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim)`.
3. As victim: sign `Authorization{authorizer: victim, authorized: attacker, isAuthorized: true, nonce: 0, deadline: block.timestamp + 365 days}` — do not submit.
4. As victim: call `midnight.setIsAuthorized(attacker, false, victim)`.
5. Assert: `midnight.isAuthorized(victim, attacker) == false`. ✓
6. As attacker: submit the withheld signature to `ecrecoverAuthorizer.setIsAuthorized(authorization, signature)`.
7. Assert: `midnight.isAuthorized(victim, attacker) == true`. ✓ — revocation defeated.

**Forge test skeleton:**

```solidity
function testPendingSignatureSurvivesRevocation() public {
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);

    // victim signs nonce=0 authorization for attacker (deadline far future)
    Authorization memory auth = Authorization(victim, attacker, true, 0, block.timestamp + 365 days);
    Signature memory sig = _sign(victimKey, auth);

    // victim revokes attacker directly
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, false, victim);
    assertFalse(midnight.isAuthorized(victim, attacker));

    // attacker submits withheld signature
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // revocation is undone
    assertTrue(midnight.isAuthorized(victim, attacker));
}
```
