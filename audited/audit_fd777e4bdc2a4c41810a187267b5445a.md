### Title
`maxAssets` Cap Bypass on Buy Offers via Zero-Rounding of `buyerAssets` - (`src/Midnight.sol`)

### Summary

When `offer.buy == true` and `offer.maxAssets > 0`, the consumed tracking increments by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For any tick strictly below `MAX_TICK` (5820), `offerPrice < WAD`, so calling `take()` with `units = 1` yields `buyerAssets = 0`, meaning `consumed[maker][group]` never increases. The `require(newConsumed <= offer.maxAssets)` guard is permanently bypassed, allowing the taker to fill the offer an unbounded number of times beyond the maker's intended cap.

### Finding Description

**Exact code path** — `src/Midnight.sol` lines 363–373:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // ← rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

uint256 newConsumed;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // ← guard
}
```

**Root cause** — `tickToPrice(MAX_TICK) == WAD` (proven in `certora/specs/TickToPrice.spec` line 38–39) and `tickToPrice` is strictly less than WAD for every tick below `MAX_TICK` (`src/libraries/TickLib.sol` lines 44–52). Therefore, for any `offer.tick < MAX_TICK`:

```
buyerPrice = offerPrice (± settlementFee, but still < WAD for small fees)
mulDivDown(1, buyerPrice, WAD) = floor(buyerPrice / WAD) = 0
```

`consumed[maker][group] += 0` — the mapping is never updated, and `require(0 <= maxAssets)` trivially passes every time.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0` (any nonzero value)
- `offer.tick` = any tick < MAX_TICK where `buyerPrice < WAD` (e.g., `MAX_TICK - 16` as used in the existing test)
- `units = 1` on each call

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = 1e18`, `tick = MAX_TICK - 16`
2. Taker calls `take(offer, ..., 1, ...)` repeatedly
3. Each call: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; `consumed` stays at 0; guard passes
4. Each call still executes the full position accounting: maker credit += 1, taker debt += 1 (lines 408–414)
5. After N calls: `consumed[maker][group] == 0`, maker has N credit units, taker has N debt units — no assets transferred

**Why existing checks fail:**
- The `ConsumedAssets` guard at line 369 only fires when `newConsumed > maxAssets`, but `newConsumed` is permanently 0
- The `EcrecoverRatifier.isRatified` check (lines 33–46 of `src/ratifiers/EcrecoverRatifier.sol`) only verifies the maker's signature on the offer struct; it does not constrain how many times the same offer can be taken
- The code comment at `src/Midnight.sol` line 94 explicitly acknowledges: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1"* — confirming the bypass is known

**Confirming test** — `test/TakeTest.sol` lines 857–889, function `testBugBuyMaxAssetsBypass`:
- Sets `lenderOffer.tick = MAX_TICK - 16`, `lenderOffer.maxAssets = 1`
- Pre-fills consumed to `maxAssets` via `setConsumed`
- Calls `take(1, borrower, lenderOffer)` — succeeds despite offer being fully consumed
- Asserts `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged, but `creditOf(lender)` and `debtOf(borrower)` both increased [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

The maker's `maxAssets` cap — intended to bound total buyer-side asset exposure — is completely unenforced whenever `buyerPrice < WAD` and `units = 1`. A taker can call `take()` an arbitrary number of times, each time adding 1 unit of debt to themselves and 1 unit of credit to the maker, with zero asset transfer. The maker accumulates credit far beyond their intended cap; the taker accumulates debt without receiving any loan assets. The core invariant *"offers cannot be overfilled"* is violated. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

**Preconditions:**
- Buy offer with `maxAssets > 0` and any `tick < MAX_TICK` (the entire usable tick range except the single top tick)
- Taker supplies `units = 1` per call

**Feasibility:** Trivially reachable. Any tick below 5820 satisfies the condition. The `DEFAULT_TICK_SPACING = 4` means the highest accessible tick below MAX_TICK is 5816, which already gives `offerPrice < WAD`. Settlement fees only widen the gap. No special permissions, oracle manipulation, or admin action required — only a valid signed offer (standard precondition for any `take()` call).

**Repeatability:** Unlimited. Each call costs only gas. The taker is bounded only by the seller's collateral capacity and the market's maturity. [7](#0-6) [8](#0-7) 

### Recommendation

Add a check that rejects a take when `offer.maxAssets > 0`, `offer.buy == true`, and `buyerAssets == 0` but `units > 0`. Concretely, after computing `buyerAssets` at line 363, insert:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssetsWithMaxAssets());
}
```

This prevents the rounding-to-zero bypass while still allowing zero-unit no-op takes (which the protocol explicitly supports for callback purposes per line 93). [1](#0-0) 

### Proof of Concept

```solidity
function testMaxAssetsBypassUnbounded() public {
    // Setup: buy offer with maxAssets cap, tick just below MAX_TICK
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1e18;
    lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD → buyerAssets = 0 for units=1

    deal(address(loanToken), lender, 0);       // lender pays nothing
    collateralize(market, borrower, 1000);     // borrower has collateral for 1000 units of debt

    uint256 consumedBefore = midnight.consumed(lender, lenderOffer.group);

    // Taker (borrower) calls take() 1000 times with units=1
    for (uint256 i = 0; i < 1000; i++) {
        (uint256 ba, uint256 sa) = take(1, borrower, lenderOffer);
        assertEq(ba, 0, "buyerAssets must be 0");
        assertEq(sa, 0, "sellerAssets must be 0");
    }

    // Assertions:
    assertEq(midnight.consumed(lender, lenderOffer.group), consumedBefore,
        "consumed must not have increased — cap bypassed");
    assertEq(midnight.debtOf(id, borrower), 1000,
        "borrower debt increased by 1000 despite maxAssets cap");
    assertEq(midnight.creditOf(id, lender), 1000,
        "lender credit increased by 1000 beyond maxAssets cap");
}
```

Expected: all assertions pass, demonstrating that `maxAssets = 1e18` is completely unenforced and the offer is filled 1000× beyond the maker's intended cap. [9](#0-8) [1](#0-0)
