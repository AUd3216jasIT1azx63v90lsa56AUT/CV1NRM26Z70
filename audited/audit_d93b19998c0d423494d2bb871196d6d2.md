### Title
Division-by-Zero / Underflow in Bad Debt Socialization Permanently DoS-es Liquidation — (`File: src/Midnight.sol`)

---

### Summary

The `liquidate` function in `Midnight.sol` socializes bad debt by dividing by `_totalUnits`. Because `totalUnits` can legitimately reach zero while a borrower still carries outstanding debt (via a normal `take` flow that nets out credit against debt), any subsequent liquidation of that borrower that produces `badDebt > 0` will revert permanently due to an arithmetic underflow and division-by-zero. The borrower's bad-debt position becomes permanently frozen and unresolvable.

---

### Finding Description

**Vulnerable code — `liquidate`, bad-debt socialization block:** [1](#0-0) 

```solidity
if (badDebt > 0) {
    _position.debt -= uint128(badDebt);
    uint256 _totalUnits = _marketState.totalUnits;   // can be 0
    uint256 _lossFactor = _marketState.lossFactor;
    _marketState.lossFactor = UtilsLib.toUint128(
        type(uint128).max - (type(uint128).max - _lossFactor)
            .mulDivDown(_totalUnits - badDebt, _totalUnits)  // ← underflow + div/0
    );
    _marketState.totalUnits -= UtilsLib.toUint128(badDebt);  // ← also underflows
    ...
}
```

`mulDivDown` is a plain Solidity division: [2](#0-1) 

```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;   // reverts if d == 0
}
```

**How `totalUnits` reaches zero while debt remains:**

`totalUnits` is updated in `take` as: [3](#0-2) 

```solidity
_marketState.totalUnits =
    UtilsLib.toUint128(_marketState.totalUnits + buyerCreditIncrease - sellerCreditDecrease);
```

Where: [4](#0-3) 

```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt);
uint256 sellerCreditDecrease = UtilsLib.min(units, sellerPos.credit);
uint256 sellerDebtIncrease = units - sellerCreditDecrease;
```

If the **buyer already has debt ≥ units**, `buyerCreditIncrease = 0`. If the **seller has credit ≥ units**, `sellerCreditDecrease = units`. The net effect on `totalUnits` is `−units`, with **no corresponding reduction in any third borrower's debt**.

**Concrete state-transition path:**

| Step | Action | `totalUnits` | `Bob.debt` | `Charlie.debt` | `Alice.credit` |
|------|--------|-------------|-----------|---------------|---------------|
| 1 | Alice lends 100 to Bob | 100 | 100 | 0 | 100 |
| 2 | Charlie (debt=100) takes Alice's sell offer (100 units) | **0** | **100** | 0 | 0 |
| 3 | Bob's collateral drops; liquidator calls `liquidate(Bob)` | — | — | — | — |

After step 2: `totalUnits = 0`, `Bob.debt = 100`. Step 3 computes `badDebt > 0` and hits:
- `_totalUnits - badDebt` = `0 - badDebt` → **Solidity 0.8 underflow revert**
- Even if that were skipped, `mulDivDown(..., 0)` → **division-by-zero revert**

The liquidation reverts unconditionally. No code path bypasses the `if (badDebt > 0)` block once bad debt exists.

---

### Impact Explanation

- **Permanent DoS on liquidation**: Any borrower whose market reaches `totalUnits < badDebt` (including `totalUnits == 0`) can never be liquidated when they have bad debt. The `liquidate` call reverts every time.
- **Unresolvable bad debt**: The bad debt cannot be cleared, socialized, or written off. The borrower's position is permanently stuck.
- **Protocol accounting corruption**: `lossFactor` and `totalUnits` are never updated, so all downstream accounting (credit slashing via `updatePositionView`, `continuousFeeCredit` adjustments) is permanently stale for that market. [5](#0-4) 

---

### Likelihood Explanation

- **No privileged access required.** Any user with outstanding debt can take a sell offer from a lender, reducing `totalUnits` to zero as a normal protocol operation.
- **Can occur naturally** (Charlie innocently repaying their own debt by buying credit from Alice).
- **Can be triggered deliberately** by Bob (the at-risk borrower) who creates a second account (Charlie), borrows in the same market, then uses Charlie to drain `totalUnits` to zero before his own position becomes liquidatable — effectively immunizing himself from liquidation.
- The only precondition is that a borrower with debt ≥ units exists and a lender with credit ≥ units exists in the same market, which is the normal operating state of any active market.

---

### Recommendation

Add a guard before the `lossFactor` update to handle the case where `_totalUnits == 0` or `_totalUnits <= badDebt`:

```solidity
if (badDebt > 0) {
    _position.debt -= uint128(badDebt);
    uint256 _totalUnits = _marketState.totalUnits;
    uint256 _lossFactor = _marketState.lossFactor;

    if (_totalUnits <= badDebt) {
        // All remaining credit is wiped out; max out the loss factor.
        _marketState.lossFactor = type(uint128).max;
        _marketState.continuousFeeCredit = 0;
    } else {
        _marketState.lossFactor = UtilsLib.toUint128(
            type(uint128).max - (type(uint128).max - _lossFactor)
                .mulDivDown(_totalUnits - badDebt, _totalUnits)
        );
        _marketState.continuousFeeCredit = _lossFactor < type(uint128).max
            ? UtilsLib.toUint128(
                _marketState.continuousFeeCredit
                    .mulDivDown(type(uint128).max - _marketState.lossFactor, type(uint128).max - _lossFactor)
            )
            : 0;
    }
    _marketState.totalUnits -= UtilsLib.toUint128(badDebt);
}
```

This mirrors the existing `MarketLossFactorMaxedOut` guard in `take` and the `_lossFactor == type(uint128).max` branch already present in the `continuousFeeCredit` update. [6](#0-5)

### Proof of Concept

**Setup (same market, three actors):**

1. **Alice** (lender): supplies 100 loan tokens, receives 100 credit units. `Alice.credit = 100`, `totalUnits = 100`.
2. **Bob** (borrower): supplies collateral, borrows 100 units from Alice via `take`. `Bob.debt = 100`, `totalUnits = 100`.
3. **Charlie** (borrower): supplies collateral, borrows 100 units from a separate lender. `Charlie.debt = 100`, `totalUnits = 200`.

**Attack / natural trigger:**

4. Charlie posts a buy offer (tick = some price, `offer.buy = true`). Alice takes it for 100 units.
   - `buyerCreditIncrease = zeroFloorSub(100, Charlie.debt=100) = 0`
   - `sellerCreditDecrease = min(100, Alice.credit=100) = 100`
   - `totalUnits = 200 + 0 - 100 = 100`
   - `Charlie.debt -= 100 → 0`; `Alice.credit -= 100 → 0`

   Repeat with the second lender's credit to drain `totalUnits` to 0 while `Bob.debt = 100` remains.

5. Bob's collateral price drops. `badDebt = 100 > 0`, `_totalUnits = 0`.

6. Liquidator calls `liquidate(market, ..., Bob, ...)`:
   - Enters `if (badDebt > 0)` block.
   - Executes `_totalUnits - badDebt` = `0 - 100` → **arithmetic underflow → revert**.

7. Every subsequent call to `liquidate(Bob)` reverts identically. Bob's bad debt is permanently frozen. [1](#0-0)
