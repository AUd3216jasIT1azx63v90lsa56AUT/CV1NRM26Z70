### Title
Fee-on-Transfer Loan Token Inflates `withdrawable` Beyond Actual Balance in `repay()` - (File: src/Midnight.sol)

### Summary
`repay()` unconditionally increments `marketState[id].withdrawable` by the full `units` parameter before calling `safeTransferFrom`, but if the loan token charges a transfer fee, only `units * (1 - fee)` actually arrives at the contract. Because `touchMarket()` imposes no restriction on the loan token type, any unprivileged borrower can trigger this accounting divergence, causing `withdrawable` to permanently exceed the contract's real token balance and making the market insolvent for lenders.

### Finding Description
**Exact code path:**

In `src/Midnight.sol` lines 508–520:

```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);   // line 509: full units credited

address payer = callback != address(0) ? callback : msg.sender;
// ...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520: only units*(1-fee) arrives
```

`withdrawable` is incremented by `units` at line 509, but `safeTransferFrom` at line 520 only delivers `units * (1 - fee)` to `address(this)` when the loan token is fee-on-transfer. The delta `units * fee` is permanently missing from the contract balance while `withdrawable` reflects the full amount.

**Why no existing check stops it:**

`touchMarket()` at lines 755–791 validates only maturity, collateral count, collateral sorting, LLTV tiers, and maxLif values. There is no check that `market.loanToken` is a standard ERC20. Any address can call `touchMarket()` (it is `public`) with a fee-on-transfer token as `loanToken`, and the market is created without restriction.

The Certora solvency specification at `certora/specs/Solvency.spec` lines 31–33 explicitly assumes away this class of token:

```
// Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing...
```

So the formal solvency invariant is not proven for fee-on-transfer loan tokens.

**Attacker inputs and exploit flow:**

1. Attacker (borrower) creates or uses a market whose `loanToken` is a fee-on-transfer token (e.g., 1% fee).
2. Borrower acquires debt of `D` units via `take()`.
3. Borrower calls `repay(market, D, borrower, address(0), "")`.
4. Line 509 executes: `withdrawable += D`.
5. Line 520 executes: contract receives only `D * 0.99`; the fee `D * 0.01` is burned/redirected.
6. State after: `withdrawable == D`, `balanceOf(midnight) == D * 0.99`. Gap = `D * 0.01`.
7. Repeating across multiple repayments compounds the gap linearly.
8. When lenders call `withdraw()`, line 499 calls `safeTransfer(loanToken, receiver, units)`. The last lenders to withdraw will trigger a revert because the contract balance is exhausted before all `withdrawable` credit is redeemed.

### Impact Explanation
The contract's actual loan token balance falls short of `withdrawable` by `sum(units_i * fee)` across all repayments. Lenders who call `withdraw()` first drain the real balance; subsequent lenders receive a revert from `safeTransfer` because the contract holds fewer tokens than `withdrawable` promises. This is concrete insolvency: valid credit positions cannot be redeemed, and the shortfall grows with every repayment on a fee-on-transfer market.

### Likelihood Explanation
The precondition is a market whose `loanToken` is a fee-on-transfer token. Market creation is permissionless — any address can call `touchMarket()` with an arbitrary `loanToken`. Fee-on-transfer tokens exist in production (e.g., USDT with fee enabled, STA, PAXG). The exploit requires no special privilege: any borrower with outstanding debt can trigger it. It is repeatable on every `repay()` call, compounding the shortfall. The same pattern also applies to `liquidate()` at line 717, which has an identical `safeTransferFrom(market.loanToken, payer, address(this), repaidUnits)` after incrementing `withdrawable`.

### Recommendation
Measure the actual received amount by comparing `balanceOf(address(this))` before and after the `safeTransferFrom`, and use the measured delta — not the nominal `units` — to increment `withdrawable` and decrement `position.debt`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens by adding a balance-check guard in `touchMarket()` or in `repay()` itself (e.g., require `balanceOf` delta equals `units`). The same fix must be applied to `liquidate()`.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry fuzz/invariant test plan

contract FeeOnTransferLoanToken is ERC20 {
    uint256 public feePct; // e.g. 100 = 1%
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount * feePct / 10000;
        super.transferFrom(from, to, amount - fee); // only amount-fee arrives
        _burn(from, fee);                           // fee destroyed
        return true;
    }
}

function testRepayFeeOnTransferInflatesWithdrawable(uint256 units, uint256 feePct) public {
    feePct = bound(feePct, 1, 1000); // 0.01% to 10%
    units  = bound(units, 1e6, 1e24);

    FeeOnTransferLoanToken feeToken = new FeeOnTransferLoanToken(feePct);
    // Create market with feeToken as loanToken (permissionless)
    Market memory m = buildMarket(address(feeToken));
    bytes32 id = midnight.touchMarket(m);

    // Give borrower debt of `units` via take()
    setupBorrowerDebt(m, id, borrower, units);

    uint256 balBefore = feeToken.balanceOf(address(midnight));
    uint256 withdrawableBefore = midnight.withdrawable(id);

    deal(address(feeToken), borrower, units);
    vm.prank(borrower);
    feeToken.approve(address(midnight), units);
    vm.prank(borrower);
    midnight.repay(m, units, borrower, address(0), "");

    uint256 balAfter = feeToken.balanceOf(address(midnight));
    uint256 withdrawableAfter = midnight.withdrawable(id);

    uint256 actualReceived = balAfter - balBefore;
    uint256 withdrawableDelta = withdrawableAfter - withdrawableBefore;

    // KEY ASSERTION: withdrawable must not exceed actual balance increase
    // This will FAIL: withdrawableDelta == units, actualReceived == units*(1-fee)
    assertEq(withdrawableDelta, actualReceived, "withdrawable inflated beyond received tokens");

    // INVARIANT: sum of withdrawable <= balanceOf(midnight) for this token
    assertLe(midnight.withdrawable(id), feeToken.balanceOf(address(midnight)),
        "insolvency: withdrawable exceeds contract balance");
}
```

**Expected result:** Both assertions fail. `withdrawableDelta` equals `units` while `actualReceived` equals `units * (1 - feePct/10000)`, confirming the accounting gap and the insolvency invariant breach. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

        bytes32 id = toId(market);
        if (marketState[id].tickSpacing == 0) {

            require(market.collateralParams.length > 0, NoCollateralParams());
            require(market.collateralParams.length <= MAX_COLLATERALS, TooManyCollateralParams());
            address previousCollateralToken;
            for (uint256 i = 0; i < market.collateralParams.length; i++) {
                address collateralToken = market.collateralParams[i].token;
                require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
                uint256 lltv = market.collateralParams[i].lltv;
