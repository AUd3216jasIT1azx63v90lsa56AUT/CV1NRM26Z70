Audit Report

## Title
Liquidator-Controlled `callback` Parameter Designates Arbitrary Third-Party as `payer` Without Authorization Check - (File: src/Midnight.sol)

## Summary
In `Midnight.liquidate`, the `callback` address is supplied by the liquidator and unconditionally assigned as the `payer` for the loan repayment `safeTransferFrom` at line 679, with no check that the callback address has authorized the liquidator to act on its behalf. Any contract that implements `ILiquidateCallback`, returns `CALLBACK_SUCCESS` without validating the `caller` argument, and holds a `loanToken` approval to Midnight can be drained: the liquidator receives seized collateral at an attacker-controlled `receiver` while the third-party callback bears the full repayment cost.

## Finding Description

**Root cause — line 679:**
```solidity
address payer = callback != address(0) ? callback : msg.sender;
``` [1](#0-0) 

The liquidator freely designates any address as `payer`. There is no `require(callback == msg.sender || isAuthorized[callback][msg.sender])` or equivalent guard.

**Exploit flow:**

1. Attacker calls `liquidate(..., receiver=attacker, callback=thirdParty, ...)`.
2. Collateral is transferred to `receiver` (attacker) before the callback fires (line 696): [2](#0-1) 

3. `onLiquidate` is called on `thirdParty` with `msg.sender` (attacker) as the `caller` argument (lines 700–712). If `thirdParty` does not validate this argument and returns `CALLBACK_SUCCESS`, execution continues: [3](#0-2) 

4. Loan token is pulled from `payer = thirdParty` (line 717): [4](#0-3) 

**Why existing checks fail:**

- The `liquidatorGate` check (line 598) only verifies that `msg.sender` may liquidate; it says nothing about who the callback/payer is: [5](#0-4) 

- The Certora property `otherEntryPointsOnlyPullFromCaller` (line 119 of `OnlyExplicitPayerCanLoseTokens.spec`) proves only that the pull comes from a callback that returned `CALLBACK_SUCCESS` — it does not prove the callback consented to being the payer for this particular liquidator: [6](#0-5) 

- The `ILiquidateCallback.onLiquidate` interface exposes `caller` as the first argument (line 17 of `src/interfaces/ICallbacks.sol`), but the protocol never enforces that the callback validates it: [7](#0-6) 

**Contrast with `take` (line 422):** In `take`, the buyer callback is always set by the buyer (maker via `offer.callback`, taker via `takerCallback`), so the payer is always the party that set the callback. In `liquidate`, the liquidator sets the callback but the callback pays — a broken symmetry with no compensating authorization check: [8](#0-7) 

## Impact Explanation
An unprivileged liquidator profits from collateral seizure (`seizedAssets` sent to `receiver = attacker`) while a third-party contract bears the entire repayment cost (`repaidUnits` pulled from `thirdParty`). The invariant that liquidation repayment must come from the liquidator (or a party the liquidator is authorized to act for) is violated. The third party suffers a direct, unrecoverable loss of `loanToken` proportional to the liquidated debt. This constitutes unauthorized movement of assets from a third party — a direct theft impact.

## Likelihood Explanation
Preconditions are realistic:
1. An unhealthy borrower position exists (normal market condition).
2. The attacker passes any `liquidatorGate` check, or the market has none.
3. A `thirdParty` contract exists that: (a) implements `ILiquidateCallback` and returns `CALLBACK_SUCCESS` without validating the `caller` argument, and (b) has granted Midnight a `loanToken` allowance ≥ `repaidUnits`.

Condition 3 is realistic for liquidation-infrastructure contracts (flash-liquidation bots, vault integrations, keeper contracts) that pre-approve Midnight and implement the callback interface. Such contracts may trust Midnight to only invoke `onLiquidate` when they themselves initiated the liquidation, omitting the `caller` guard. The attack is repeatable as long as the third party's allowance and balance remain sufficient.

## Recommendation
Add an authorization check before assigning `payer`:
```solidity
if (callback != address(0)) {
    require(
        callback == msg.sender || isAuthorized[callback][msg.sender],
        Unauthorized()
    );
}
address payer = callback != address(0) ? callback : msg.sender;
```
This mirrors the authorization pattern already used elsewhere in the contract (e.g., `setConsumed` at line 724) and restores the invariant that the payer must have explicitly authorized the liquidator to act on its behalf. [9](#0-8) 

## Proof of Concept
1. Deploy `ThirdParty` implementing `ILiquidateCallback.onLiquidate` that returns `CALLBACK_SUCCESS` unconditionally (no `caller` validation). Pre-approve Midnight for `repaidUnits` of `loanToken`.
2. Create an undercollateralized borrower position in a market with no `liquidatorGate`.
3. Attacker calls:
   ```solidity
   midnight.liquidate(market, collateralIndex, seizedAssets, 0, borrower, false, attacker, address(thirdParty), "");
   ```
4. Observe: attacker's address receives `seizedAssets` of collateral token; `thirdParty`'s `loanToken` balance decreases by `repaidUnits`; attacker paid nothing.
