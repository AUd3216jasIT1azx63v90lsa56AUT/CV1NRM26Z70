### Title
Off-by-One in `CannotIncreaseDebtPostMaturity` Allows Debt Creation at Exact Maturity Timestamp - (File: src/Midnight.sol)

### Summary
The guard in `take()` uses `block.timestamp <= offer.market.maturity` to permit debt increases, meaning a call at the exact maturity second (`block.timestamp == maturity`) passes the check and allows `sellerDebtIncrease > 0`. The protocol's own stated core invariant explicitly forbids debt increase at the maturity boundary via timestamp equality, making this a direct violation of the invariant with no compensating check anywhere in the call path.

### Finding Description
**Code path:**

`take()` at `src/Midnight.sol:391`:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

`sellerDebtIncrease` is computed at line 384:
```solidity
uint256 sellerDebtIncrease = units - sellerCreditDecrease;
```
where `sellerCreditDecrease = min(units, sellerPos.credit)`. For any seller with zero existing credit, `sellerCreditDecrease == 0` and `sellerDebtIncrease == units`.

**Root cause:** The condition uses `<=` (non-strict). At `block.timestamp == offer.market.maturity`, the left side of the `||` evaluates to `true`, so the entire `require` passes unconditionally regardless of `sellerDebtIncrease`.

**Attacker inputs:**
- Taker (unprivileged) calls `take()` with any `units > 0`
- Seller (maker or taker) has zero existing credit so `sellerDebtIncrease = units`
- Transaction is mined in the block where `block.timestamp == offer.market.maturity`

**Why existing checks fail:** No other check in `take()` guards against debt increase at exact maturity. The `reduceOnly` flag is maker-controlled and optional. The enter gate is market-configured and optional. The only debt-at-maturity guard is line 391 itself, which is the defective check.

**All post-maturity tests** (`testBuy1PostMaturity`, `testSell1PostMaturity`, `testBuyPastMaturity`, `testSellPastMaturity`) use `market.maturity + 1` as the lower bound, leaving the `block.timestamp == maturity` case completely untested. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
A seller with no existing credit who is a party to a `take()` call at `block.timestamp == maturity` has their debt increased by `units`. This is unauthorized debt creation at the maturity boundary: the debt is immediately overdue (post-maturity liquidation applies for `block.timestamp > maturity`), the borrower owes loan tokens they did not have before, and the protocol's credit/debt accounting is corrupted relative to the invariant that every debt unit must be created only before maturity. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
Any block mined at exactly `block.timestamp == maturity` triggers this. Since `maturity` is a `uint32` unix timestamp set at market creation, and Ethereum block timestamps are miner/validator-influenced within ~12 seconds, a validator can trivially land a transaction in the maturity block. Even without validator collusion, any transaction naturally included in that block exploits this. The condition is repeatable for every market whose maturity falls on a block boundary, which is every market eventually. [6](#0-5) 

### Recommendation
Change the comparison from `<=` to `<` (strict less than):

```solidity
// Before (line 391):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After:
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns with the stated invariant: "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing." [1](#0-0) [7](#0-6) 

### Proof of Concept

```solidity
function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;

    // Warp to EXACT maturity (not maturity+1)
    vm.warp(market.maturity);

    borrowerOffer.expiry = market.maturity;
    borrowerOffer.maxUnits = units;
    deal(address(loanToken), lender, units);
    collateralize(market, borrower, units);

    uint256 debtBefore = midnight.debtOf(id, borrower);

    // This should revert with CannotIncreaseDebtPostMaturity but does NOT
    take(units, lender, borrowerOffer);

    uint256 debtAfter = midnight.debtOf(id, borrower);

    // Invariant assertion: debt must not increase at or after maturity
    assertEq(debtAfter, debtBefore, "debt must not increase at exact maturity");
}

// Fuzz variant
function testFuzzDebtIncreaseAtMaturityBoundary(uint256 units) public {
    units = bound(units, 1, maxAssets);
    uint256[3] memory timestamps = [market.maturity - 1, market.maturity, market.maturity + 1];

    for (uint256 i = 0; i < 3; i++) {
        // reset state per iteration
        vm.warp(timestamps[i]);
        borrowerOffer.expiry = timestamps[i];
        borrowerOffer.maxUnits = units;
        deal(address(loanToken), lender, units);
        collateralize(market, borrower, units);

        uint256 debtBefore = midnight.debtOf(id, borrower);
        if (timestamps[i] >= market.maturity) {
            vm.expectRevert(IMidnight.CannotIncreaseDebtPostMaturity.selector);
        }
        take(units, lender, borrowerOffer);

        if (timestamps[i] >= market.maturity) {
            assertEq(midnight.debtOf(id, borrower), debtBefore, "no debt increase at/after maturity");
        }
    }
}
```

Expected: the `i == 1` case (`block.timestamp == maturity`) succeeds and increases debt, failing the `assertEq`. With the fix (`<`), it reverts with `CannotIncreaseDebtPostMaturity`. [8](#0-7) [1](#0-0)
