Audit Report

## Title
Maker's `onBuy` callback can call `setConsumed` to inflate `consumed` and block subsequent fills - (File: `src/Midnight.sol`)

## Summary
In `take()`, `consumed[offer.maker][offer.group]` is incremented and the cap is enforced before the maker's `onBuy` callback is invoked. Because `setConsumed` only requires the caller to be authorized by `onBehalf`, a maker who pre-authorizes their callback contract can have that callback call `setConsumed(group, maxAssets, maker)` during `onBuy`, inflating `consumed` to the offer cap. Any subsequent fill of the same offer or any offer sharing the same group within the same transaction will then revert with `ConsumedAssets`/`ConsumedUnits`.

## Finding Description

**Root cause and code path:**

`take()` at lines 366–373 increments `consumed` and enforces the cap before any callback:

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
} else {
    newConsumed = consumed[offer.maker][offer.group] += units;
    require(newConsumed <= offer.maxUnits, ConsumedUnits());
}
```

For a buy offer, `buyerCallback` is set to `offer.callback` (maker-controlled) at line 420:

```solidity
address buyerCallback = offer.buy ? offer.callback : takerCallback;
```

The `onBuy` callback is invoked at lines 445–453, **after** `consumed` has already been written to storage. At this point, the callback can call back into `setConsumed`.

`setConsumed` at lines 723–728 only checks authorization and that the new amount is non-decreasing:

```solidity
function setConsumed(bytes32 group, uint256 amount, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    require(amount >= consumed[onBehalf][group], AlreadyConsumed());
    consumed[onBehalf][group] = amount;
    ...
}
```

**Exploit flow:**

1. Maker deploys `MaliciousCallback` implementing `IBuyCallback`.
2. Maker calls `setIsAuthorized(MaliciousCallback, true, maker)`.
3. Maker creates a buy offer with `offer.callback = MaliciousCallback`, `offer.maxAssets = M`, `offer.group = G`.
4. Taker calls `take(offer, ..., units1)` (partial fill, `units1 < M`):
   - `consumed[maker][G] += buyerAssets1` → e.g., 50. Check passes (50 ≤ 100).
   - `onBuy` is called on `MaliciousCallback`.
   - Inside `onBuy`, callback calls `setConsumed(G, 100, maker)`.
   - `setConsumed` checks: `isAuthorized[maker][MaliciousCallback]` = true ✓; `100 >= 50` ✓. Sets `consumed[maker][G] = 100`.
   - First `take()` completes successfully.
5. Taker (or bundler) calls `take(offer, ..., units2)` in the same transaction via `multicall`:
   - `consumed[maker][G] += buyerAssets2` → 100 + delta > 100. Reverts with `ConsumedAssets`.

**Why existing checks fail:**

- `AlreadyConsumed` in `setConsumed` only prevents *decreasing* consumed, not inflating it to the cap.
- There is no reentrancy guard preventing `setConsumed` from being called during `take`.
- The Certora `Consume.spec` rules (`takeConsumedDelta` at line 67, `takeConsumedBoundedByMax` at line 59) do not model callbacks re-entering `setConsumed`; the spec uses `NONDET` summaries for internal functions and does not configure reentrancy harnesses, so these invariants are not verified under the attack scenario.

## Impact Explanation
A maker can use their `onBuy` callback to inflate `consumed[maker][group]` to `maxAssets` after a partial fill. Any subsequent `take` on the same offer or any offer sharing the same group within the same transaction reverts with `ConsumedAssets`. In a `multicall` context this reverts the entire bundle; in a try/catch context the taker silently receives fewer units than intended. The maker can selectively allow partial fills while atomically blocking further fills, griefing takers who rely on filling the full offer in one transaction. No funds are directly stolen, but the integrity of the `consumed` accounting invariant is broken from the taker's perspective.

## Likelihood Explanation
All preconditions are fully attacker-controlled: the maker chooses `offer.callback`, `offer.group`, `offer.maxAssets`, and pre-authorizes the callback via `setIsAuthorized`. No special privileges, oracle manipulation, or external conditions are required. The attack is repeatable on every `take` that triggers the callback and requires no coordination beyond deploying a simple callback contract.

## Recommendation
Snapshot `consumed[offer.maker][offer.group]` before the callback and assert it is unchanged after the callback returns, or disallow `setConsumed` from being called during an active `take` by using a transient reentrancy lock (similar to the existing `LIQUIDATION_LOCK_SLOT` pattern) that blocks `setConsumed` for the maker's group while `take` is executing. Alternatively, read `consumed` after the callback and revert if it exceeds the value written during the current `take`.

## Proof of Concept
1. Deploy `MaliciousCallback` implementing `IBuyCallback.onBuy` that calls `Midnight.setConsumed(group, maxAssets, maker)`.
2. Maker calls `setIsAuthorized(address(MaliciousCallback), true, maker)`.
3. Maker creates a buy offer: `maxAssets = 100`, `group = G`, `callback = address(MaliciousCallback)`.
4. In a single `multicall`:
   - Call `take(offer, ..., units1)` where `buyerAssets1 = 50`. Expect success; `consumed` is now 100 (inflated by callback).
   - Call `take(offer, ..., units2)` where `buyerAssets2 > 0`. Expect revert with `ConsumedAssets`.
5. Confirm that without the malicious callback, both takes succeed with `consumed` correctly reflecting the sum.
