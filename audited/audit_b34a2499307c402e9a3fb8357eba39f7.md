### Title
Maker-controlled `offer.callback` designates arbitrary third-party as `payer` for buy offers, draining victim's loan tokens without maker payment - (`File: src/Midnight.sol`)

### Summary
In `Midnight.take`, when `offer.buy = true`, the `payer` for `buyerAssets` is derived directly from `offer.callback` — a field freely set by the maker — without any check that the callback address is authorized by the maker. An unprivileged maker can set `offer.callback` to any victim contract that has approved Midnight for the loan token and returns `CALLBACK_SUCCESS` from `onBuy`, causing the protocol to pull `buyerAssets` from the victim instead of the maker. The maker receives credit while paying nothing.

### Finding Description

**Code path:**

In `src/Midnight.sol`, lines 420–456:

```solidity
address buyerCallback = offer.buy ? offer.callback : takerCallback;   // L420
address payer = buyerCallback != address(0)
    ? buyerCallback
    : (offer.buy ? buyer : msg.sender);                                // L422

// ...

    require(
        IBuyCallback(buyerCallback).onBuy(...) == CALLBACK_SUCCESS,    // L448-451
        WrongBuyCallbackReturnValue()
    );
}
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // L455
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);                    // L456
```

When `offer.buy = true`:
- `buyerCallback = offer.callback` (maker-controlled field, no authorization check)
- `payer = offer.callback` (if non-zero)
- `onBuy` is called on `offer.callback`; if it returns `CALLBACK_SUCCESS`, both `safeTransferFrom` calls pull from `payer = offer.callback`

**Root cause:** There is no check analogous to `require(isAuthorized[offer.maker][offer.ratifier], RatifierUnauthorized())` for `offer.callback`. The ratifier is gated by maker authorization (L355), but the callback — which becomes the payer — is not.

**Attacker inputs:**
- `offer.buy = true`
- `offer.callback = victimContract` (any contract with Midnight approval + `IBuyCallback` returning `CALLBACK_SUCCESS`)
- `offer.maker = attacker`
- All other offer fields valid (ratifier authorized, tick valid, expiry in future)

**Exploit flow:**
1. Attacker identifies `victimContract`: a contract that (a) has approved Midnight for `loanToken` and (b) implements `IBuyCallback.onBuy()` returning `CALLBACK_SUCCESS`. This is any legitimate Midnight buy-callback integration (e.g., a lending vault, aggregator, or router).
2. Attacker creates a buy offer with `offer.callback = victimContract`.
3. Any taker calls `take(offer, ...)`.
4. `buyerCallback = offer.callback = victimContract`; `payer = victimContract`.
5. `victimContract.onBuy(...)` is called → returns `CALLBACK_SUCCESS`.
6. `safeTransferFrom(loanToken, victimContract, address(this), buyerAssets - sellerAssets)` executes.
7. `safeTransferFrom(loanToken, victimContract, receiver, sellerAssets)` executes.
8. `buyer = offer.maker = attacker` receives `buyerCreditIncrease` credit (L410). Attacker paid zero.

**Why existing checks fail:**
- L346: checks taker authorization — irrelevant to callback.
- L355: `isAuthorized[offer.maker][offer.ratifier]` — only gates the ratifier, not the callback.
- L356: ratifier `isRatified` — only validates the offer structure, not the callback identity.
- The Certora spec `OnlyExplicitPayerCanLoseTokens.spec` (L91–115) proves tokens are only pulled from "the buyerCallback that returned `CALLBACK_SUCCESS`" — but this is a weaker property than "the callback must be authorized by the maker." The spec's `callbackAllowed` flag is set to `true` for any address returning `CALLBACK_SUCCESS` (L62–64), regardless of whether the maker authorized it. The spec does not catch this class of attack. ### Impact Explanation

The maker gains `buyerCreditIncrease` credit in the market (L410) without transferring any loan tokens. The victim contract loses `buyerAssets` worth of loan tokens. This directly violates the core invariant that "every credit has matching debt or valid settled/loss state" and "the buyer must pay for credit received." The attacker can repeat this with any amount up to `offer.maxAssets` / `offer.maxUnits`, draining the victim's entire approved balance. [5](#0-4) 

### Likelihood Explanation

**Preconditions:**
1. Victim contract has approved Midnight for the loan token — this is a standard prerequisite for any contract that legitimately participates in Midnight as a buy-callback payer (e.g., a vault, aggregator, or router). Such contracts exist in any real deployment.
2. Victim contract implements `IBuyCallback.onBuy()` returning `CALLBACK_SUCCESS` — again, standard for any legitimate buy-callback integration.
3. Attacker is any unprivileged address that can create an offer (no special role required).
4. Any taker can trigger the exploit (the attacker can be the taker themselves, since `offer.maker != taker` is the only restriction, and the attacker is the maker).

The attack is fully repeatable up to the victim's approval limit and requires no oracle manipulation, no admin access, and no user mistake. [6](#0-5) 

### Recommendation

Add an authorization check for `offer.callback` when `offer.buy = true`, mirroring the existing ratifier check:

```solidity
require(
    !offer.buy || offer.callback == address(0) || isAuthorized[offer.maker][offer.callback],
    CallbackUnauthorized()
);
```

This ensures that when the maker's callback is used as the payer, the maker has explicitly authorized that address — consistent with the authorization model already applied to the ratifier. [7](#0-6)

```solidity
pragma solidity ^0.8.0;

import {Test} from "forge-std/Test.sol";
import {IMidnight, IBuyCallback, Offer, Market} from "src/interfaces/IMidnight.sol";
import {Midnight} from "src/Midnight.sol";
import {ERC20} from "test/erc20s/ERC20.sol";
// ... standard test imports

/// @dev Victim: has approved Midnight, returns CALLBACK_SUCCESS from onBuy.
contract VictimBuyCallback is IBuyCallback {
    bytes32 constant CALLBACK_SUCCESS = keccak256("IBuyCallback.onBuy");
    function onBuy(bytes32, Market memory, uint256, uint256, uint256, address, bytes memory)
        external pure returns (bytes32) { return CALLBACK_SUCCESS; }
}

contract PayerConfusionTest is Test {
    Midnight midnight;
    ERC20 loanToken;
    address attacker = makeAddr("attacker");
    address taker    = makeAddr("taker");
    VictimBuyCallback victim;

    function setUp() public {
        // deploy Midnight, loanToken, set up market, ratifier, etc.
        victim = new VictimBuyCallback();
        // Fund victim and approve Midnight
        deal(address(loanToken), address(victim), 1000e18);
        vm.prank(address(victim));
        loanToken.approve(address(midnight), type(uint256).max);
    }

    function testPayerConfusion() public {
        // Attacker creates a buy offer pointing to victim as callback
        Offer memory offer = /* valid buy offer */ Offer({
            buy: true,
            callback: address(victim),
            maker: attacker,
            // ... other fields: valid ratifier authorized by attacker, tick, expiry, etc.
        });

        uint256 units = 100e18;
        uint256 victimBalanceBefore = loanToken.balanceOf(address(victim));
        uint256 attackerCreditBefore = midnight.creditOf(id, attacker);

        // Taker (or attacker acting as taker via a different address) takes the offer
        vm.prank(taker);
        midnight.take(offer, hex"", units, taker, taker, address(0), hex"");

        // Assertions:
        // 1. Victim's loan token balance decreased by buyerAssets
        assertLt(loanToken.balanceOf(address(victim)), victimBalanceBefore, "victim paid");
        // 2. Attacker's credit increased
        assertGt(midnight.creditOf(id, attacker), attackerCreditBefore, "attacker got credit");
        // 3. Attacker's loan token balance unchanged (attacker paid nothing)
        assertEq(loanToken.balanceOf(attacker), 0, "attacker paid nothing");
    }
}
```

```text
/// Proves that in `take`, the only addresses whose tokens can be pulled are:
/// 1. msg.sender (when !offer.buy and buyerCallback == 0),
/// 2. the buyerCallback that returned CALLBACK_SUCCESS,
/// 3. the offer maker (when offer.buy and buyerCallback == 0, i.e. maker is the buyer with no callback).
rule takeOnlyExplicitPayer(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiverIfTakerIsSeller, address takerCallback, bytes takerCallbackData) {
    require e.msg.sender != currentContract, "only external calls";

    address buyerCallback = offer.buy ? offer.callback : takerCallback;

    msgSender = e.msg.sender;
    msgSenderAllowed = !offer.buy && buyerCallback == 0;
    callbackAllowed = false;
    maker = offer.maker;
    makerAllowed = offer.buy && buyerCallback == 0;

    buyCallbackAllowed = true;
    liquidateCallbackAllowed = false;
    repayCallbackAllowed = false;
    flashLoanCallbackAllowed = false;
    badPullSeen = false;

    take(e, offer, ratifierData, units, taker, receiverIfTakerIsSeller, takerCallback, takerCallbackData);

    assert !badPullSeen;
}
```
