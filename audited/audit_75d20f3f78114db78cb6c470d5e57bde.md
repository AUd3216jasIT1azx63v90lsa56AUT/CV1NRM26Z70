### Title
`consumableUnits` Returns Non-Zero Capacity When `lossFactor` Is Maxed, While `take` Always Reverts - (File: src/periphery/ConsumableUnitsLib.sol)

### Summary
`ConsumableUnitsLib.consumableUnits` computes fillable offer capacity using only `consumed`, `maxUnits`, `maxAssets`, and price/fee math, but never reads `marketState[id].lossFactor`. When an unprivileged liquidator drives `lossFactor` to `type(uint128).max` via repeated bad-debt realizations, `consumableUnits` continues to return a non-zero value while every subsequent `take` call on that market reverts with `MarketLossFactorMaxedOut`. This creates a persistent, attacker-reachable state where the capacity signal and the execution path are permanently diverged.

### Finding Description

**Code path â€” `consumableUnits`:** [1](#0-0) 

The function branches on `offer.maxUnits > 0`, `offer.buy`, and delegates to `TakeAmountsLib.buyerAssetsToUnits` / `sellerAssetsToUnits`. Neither branch reads `lossFactor` from `marketState`. [2](#0-1) [3](#0-2) 

Both helpers only query `settlementFee` and perform price arithmetic â€” no `lossFactor` check anywhere in the call chain.

**Code path â€” `take`:**

`Midnight.sol` contains the `MarketLossFactorMaxedOut` guard (2 matches confirmed). When `marketState[id].lossFactor == type(uint128).max`, `take` reverts unconditionally before any fill logic executes.

**Attacker path:**

1. Liquidator calls `liquidate` repeatedly on under-collateralized positions in market `id`, each time realizing bad debt that increments `marketState[id].lossFactor`.
2. After enough iterations, `lossFactor` reaches `type(uint128).max`.
3. Attacker (or any caller) calls `consumableUnits(midnight, id, offer)` for an assets-based sell offer (`offer.maxUnits == 0`, `offer.buy == false`). The function computes `sellerAssetsToUnits(...)` and returns a non-zero value.
4. Caller attempts `take(...)` on the same offer â€” reverts with `MarketLossFactorMaxedOut`.

**Why existing checks fail:**

`consumableUnits` is a pure view helper with no awareness of the `lossFactor` gate that `take` enforces. There is no shared pre-check or modifier that both paths consult. The divergence is structural.

### Impact Explanation

Any on-chain or off-chain system that uses `consumableUnits` to determine whether a `take` is executable â€” including `MidnightBundles` (14 call sites confirmed) â€” will receive a non-zero "fillable" signal and then have every execution attempt revert. This wastes gas, breaks bundle atomicity, and permanently misleads routing/aggregation logic for the affected market. The market is effectively bricked for takers while appearing open to capacity queries. [1](#0-0) 

### Likelihood Explanation

Preconditions are reachable by any unprivileged liquidator. Bad-debt markets exist whenever collateral value falls below debt value before liquidation. Repeated calls to `liquidate` are permissionless. Once `lossFactor` saturates at `type(uint128).max` it is permanent (no decrement path exists). The inconsistency then persists indefinitely for all offers on that market.

### Recommendation

Add a `lossFactor` guard at the top of `consumableUnits`:

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer)
    internal view returns (uint256)
{
    // Return 0 if the market's loss factor is maxed â€” take will revert.
    if (IMidnight(midnight).marketState(id).lossFactor == type(uint128).max) {
        return 0;
    }
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    ...
}
```

This mirrors the guard that `take` enforces and keeps the capacity signal consistent with execution reachability.

        internal
        view

    {
        uint256 offerPrice = TickLib.tickToPrice(offer.tick);

            IMidnight(midnight).settlementFee(id, UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp));
        // Mirrors Midnight's computation to revert if offerPrice < settlementFee in case of a buy offer.
        uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
        uint256 buyerPrice = sellerPrice + settlementFee;

        return offer.buy ? targetBuyerAssets.mulDivUp(WAD, buyerPrice) : targetBuyerAssets.mulDivDown(WAD, buyerPrice);
    }
