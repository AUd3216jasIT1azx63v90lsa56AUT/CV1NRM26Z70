### Title
Oracle overflow in `isHealthy` freezes all collateral withdrawals for a borrower with debt - (File: src/Midnight.sol)

### Summary
`isHealthy` iterates every activated collateral and calls `mulDivDown(collateral[i], price, ORACLE_PRICE_SCALE)`, where `mulDivDown` is a plain Solidity 0.8 multiplication that reverts on overflow. If any one activated oracle returns a price `P` such that `collateral[i] * P > type(uint256).max`, the entire `isHealthy` call reverts. Because `withdrawCollateral` calls `isHealthy` unconditionally when the borrower has debt, a borrower with two activated collaterals cannot withdraw the second (non-overflowing) collateral either, permanently freezing all withdrawals until the debt is fully repaid.

### Finding Description

**Code path:**

`withdrawCollateral` (line 568) calls `isHealthy` after optionally clearing the bit for the collateral being withdrawn (line 565). `isHealthy` (lines 949–957) loops over the borrower's `collateralBitmap`, calling `IOracle(collateralParam.oracle).price()` and then `_position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)` for every activated collateral. `mulDivDown` is `(x * y) / d` with no overflow guard (line 30 of `UtilsLib.sol`); Solidity 0.8 reverts on `x * y` overflow.

**Root cause:**

No on-chain constraint bounds oracle prices. `touchMarket` validates `lltv`, `maxLif`, token ordering, and maturity, but never the oracle address or any price range. The Certora `NoMultiplicationOverflow.spec` proves absence of overflow only under the assumption `price * max_uint128 + ORACLE_PRICE_SCALE - 1 <= max_uint256` — an integration assumption, not an enforced invariant.

**Overflow threshold:**

`collateral[i]` is `uint128` (max ≈ 3.4 × 10³⁸). Overflow occurs when `price > type(uint256).max / collateral[i]`. With `collateral[i] = type(uint128).max`, the threshold is `price > type(uint128).max ≈ 3.4 × 10³⁸`, i.e., `price > 340 * ORACLE_PRICE_SCALE` — meaning 1 collateral token = 340 loan tokens, a realistic ratio for many token pairs (e.g., ETH/stablecoin).

**Exploit flow:**

1. Borrower activates collateral[0] (oracle0) and collateral[1] (oracle1) in a market.
2. Borrower borrows, so `debt > 0`.
3. oracle0 returns price `P` such that `collateral[0] * P > type(uint256).max`.
4. Borrower calls `withdrawCollateral(market, 1, assets, ...)` to withdraw collateral[1].
5. `withdrawCollateral` clears bit 1 (if full withdrawal), then calls `isHealthy`.
6. `isHealthy` iterates the bitmap; bit 0 is still set (collateral[0] not withdrawn).
7. `mulDivDown(collateral[0], P, ORACLE_PRICE_SCALE)` reverts with arithmetic overflow.
8. `withdrawCollateral` reverts. Borrower cannot withdraw collateral[1].
9. Attempting to withdraw collateral[0] also calls `isHealthy`, which again overflows on collateral[0] * P before the bit is cleared (the bit is only cleared after the subtraction, not before the health check).

**Why existing checks fail:**

The Certora `oracleRevertCausesWithdrawCollateralRevert` rule documents that a reverting oracle freezes withdrawals — treating it as expected behavior. Overflow in `mulDivDown` is functionally identical to an oracle revert (both cause `isHealthy` to revert), but arises from a valid oracle return value, not an oracle failure. The `NoMultiplicationOverflow.spec` proof is conditional on bounded prices, which is not enforced on-chain.

### Impact Explanation

A borrower with debt and two activated collaterals cannot call `withdrawCollateral` for any collateral index as long as one oracle returns an overflow price. The position is frozen: the borrower cannot reduce exposure, cannot deactivate the overflowing collateral (since doing so also requires passing `isHealthy`), and cannot recover without first fully repaying all debt. This violates the invariant that valid withdrawals (those that leave the position healthy) must not be frozen.

### Likelihood Explanation

**Preconditions:**
- Market with ≥ 2 activated collaterals for the borrower.
- Borrower has nonzero debt.
- One oracle returns a price exceeding `type(uint256).max / collateral[i]`.

**Feasibility:** The overflow threshold is `price > type(uint128).max / (collateral[i] / type(uint128).max)`. For a borrower holding near-maximum collateral of a high-value token (e.g., a token worth 340× the loan token), a legitimate oracle price spike past the threshold triggers the condition. The `IOracle` interface imposes no upper bound; any `uint256` is a valid return. The condition is repeatable and persistent until debt is repaid.

### Recommendation

Add an on-chain price bound check in `isHealthy` (and `liquidate`) before the multiplication, or use a checked `mulDivDown` that returns 0 or `type(uint256).max` on overflow rather than reverting. Alternatively, enforce a maximum oracle price at market creation or at collateral supply time (e.g., require `price() * type(uint128).max <= type(uint256).max` at supply time). The simplest targeted fix is to replace the bare multiplication in `mulDivDown` with a saturating variant for the health-check path, or to cap the result at `type(uint256).max` when overflow would occur, so that an extreme price makes the position appear maximally healthy rather than reverting.

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;

/// If an activated collateral oracle reverts on price different than withdrawn collateral, withdrawCollateral reverts when the borrower has debt.
rule oracleRevertCausesWithdrawCollateralRevert(env e, Midnight.Market market, uint256 collateralIndex, uint256 assets, address onBehalf, address receiver, uint256 revertingCollateralIndex) {
    require singleRevertingOracle == market.collateralParams[revertingCollateralIndex].oracle, "oracle is reverting";
    require revertingCollateralIndex < 128, "clearBit produces a new bitmap whose summaryGetBit is unconstrained for indices >= 128";
    require revertingCollateralIndex != collateralIndex, "withdrawCollateral may clear the bit at collateralIndex before calling isHealthy";

    bytes32 id = summaryToId(market);
    uint128 bitmap = collateralBitmap(id, onBehalf);
    require summaryGetBit(bitmap, revertingCollateralIndex), "revertingCollateralIndex is activated";

    withdrawCollateral@withrevert(e, market, collateralIndex, assets, onBehalf, receiver);
    bool reverted = lastReverted;

    assert debtOf(id, onBehalf) > 0 => reverted;
}
