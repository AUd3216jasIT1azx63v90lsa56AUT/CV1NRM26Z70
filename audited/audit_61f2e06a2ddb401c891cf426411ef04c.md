Audit Report

## Title
Stale `consumed` Read in `ConsumableUnitsLib.consumableUnits` Enables Front-Run DoS on Bundler Take Loops - (`src/periphery/ConsumableUnitsLib.sol`)

## Summary
`ConsumableUnitsLib.consumableUnits` reads `consumed` at view time and computes `unitsToTake` calibrated to exactly exhaust the remaining capacity of an offer. Any concurrent fill of that offer between the view and the on-chain `take` causes `Midnight.take` to revert with `ConsumedAssets`. The bundler's `catch {}` silently discards this revert, and if no further offers cover the remaining `targetUnits`, the entire bundler transaction reverts with `OutOfOffers`. A malicious actor can front-run with a minimal fill (`units = 1`) at negligible cost to reliably trigger this on every retry.

## Finding Description
`ConsumableUnitsLib.consumableUnits` reads the on-chain `consumed` value at view time:

```solidity
uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
```

For a buy offer with `maxAssets > 0`, it computes:

```solidity
return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
```

`buyerAssetsToUnits` returns `targetBuyerAssets.mulDivUp(WAD, buyerPrice)`, i.e., `units = ceil((maxAssets - consumed_view) * WAD / buyerPrice)`. The test `testBuyerAssetsToUnitsBuyerIsLender` (line 110) confirms that `take(units, ...)` yields `buyerAssets == targetBuyerAssets == maxAssets - consumed_view` exactly.

In `Midnight.take`, the consumed accounting is:

```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

If any other transaction increases `consumed` by `delta > 0` between the view and the execute:

```
newConsumed = (consumed_view + delta) + (maxAssets - consumed_view)
            = maxAssets + delta  >  maxAssets  →  ConsumedAssets revert
```

The same arithmetic applies to the `maxUnits` path: `consumableUnits` returns `offer.maxUnits.zeroFloorSub(consumed)`, and any `delta > 0` causes `newConsumed = maxUnits + delta > maxUnits` → `ConsumedUnits` revert.

In `supplyCollateralAndSellWithUnitsTarget` (and all three other bundler functions), this revert is caught silently:

```solidity
try IMidnight(MIDNIGHT).take(...) returns (...) {
    filledUnits += unitsToTake;
    ...
} catch {}
```

The offer is skipped. If no further offers cover the remaining `targetUnits`, the function reverts:

```solidity
require(filledUnits == targetUnits, OutOfOffers());
```

This revert is atomic — the entire transaction rolls back, including any collateral supplied in the loop preceding the take loop. No funds are permanently stuck, but the user loses gas and must retry.

Existing guards are insufficient: the `catch {}` is intentional (documented at line 43–44 of `MidnightBundles.sol`) but makes no distinction between "offer genuinely exhausted" and "offer partially front-run." There is no safety margin, no retry, and no fallback to a partial fill.

## Impact Explanation
The user's bundler call reverts with `OutOfOffers` even though the targeted offer still has remaining capacity. The user loses gas on every attempt. A malicious actor can permanently block the user from successfully executing this bundler function for a given offer by front-running each retry at negligible cost (a single `Midnight.take` with `units = 1`). This is a targeted, repeatable griefing/DoS attack on the lending entry point.

## Likelihood Explanation
**Preconditions:** A buy (or sell) offer with `maxAssets > 0` (or `maxUnits > 0`) that is not yet fully consumed — a normal, expected state for any active offer. **Feasibility:** Any unprivileged taker calling `Midnight.take` directly on the same offer triggers this, even non-maliciously in a competitive market. A malicious actor can deliberately front-run with `units = 1` at minimal gas cost. **Repeatability:** The attack can be repeated on every retry, permanently blocking the victim from using the bundler for that offer.

## Recommendation
Replace the exact-exhaust computation with a conservative one that leaves a safety margin, or clamp `unitsToTake` to `floor((maxAssets - consumed_view) * WAD / buyerPrice)` (using `mulDivDown` instead of `mulDivUp`) so that `buyerAssets <= maxAssets - consumed_view`, tolerating any `delta` up to the rounding slack. Alternatively, re-read `consumed` inside the `try` block and recompute `unitsToTake` on-chain, or distinguish `ConsumedAssets`/`ConsumedUnits` reverts in the `catch` block and attempt a reduced fill rather than skipping the offer entirely.

## Proof of Concept
1. Deploy `Midnight` and `MidnightBundles`. Create a buy offer with `maxAssets = 1000e18`, `consumed = 0`.
2. Victim calls `supplyCollateralAndSellWithUnitsTarget` with `targetUnits = N` against this single offer. Off-chain simulation computes `unitsToTake = ceil(1000e18 * WAD / buyerPrice)`.
3. Attacker front-runs with `Midnight.take(offer, ..., 1, ...)`, setting `consumed = buyerAssets_for_1_unit > 0`.
4. Victim's bundler executes: `consumableUnits` reads stale `consumed = 0`, computes the same `unitsToTake`. `Midnight.take` computes `newConsumed = delta + 1000e18 > 1000e18`, reverts with `ConsumedAssets`. `catch {}` skips the offer. `filledUnits == 0 != targetUnits` → `OutOfOffers` revert.
