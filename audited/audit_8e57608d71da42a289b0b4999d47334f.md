### Title
`sellerAssetsToUnits` reverts with arithmetic underflow when `sellerPrice == 0` on a valid buy offer at the settlement fee boundary - (`File: src/periphery/TakeAmountsLib.sol`)

### Summary

`TakeAmountsLib.sellerAssetsToUnits` computes `targetSellerAssets.mulDivUp(WAD, sellerPrice)` where `sellerPrice = offerPrice - settlementFee` for buy offers. When `offerPrice == settlementFee`, `sellerPrice` is exactly `0`, and `mulDivUp` evaluates `(x * y + (d - 1)) / d` with `d = 0`, causing an arithmetic underflow on `(d - 1)` in Solidity 0.8+ checked arithmetic. `Midnight.take()` itself accepts this offer without reverting (it computes `units.mulDivDown(0, WAD) = 0`), so the boundary is valid in the core protocol but unhandled in the periphery inverse function. Because `supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside any `try/catch`, the revert propagates and DoSes the entire bundler call.

### Finding Description

**Exact code path:**

`UtilsLib.mulDivUp` is: [1](#0-0) 

With `d = 0`, the expression `(d - 1)` underflows (Solidity 0.8 checked arithmetic), producing an `arithmeticError` revert before any division occurs.

`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice` and then calls `mulDivUp(WAD, sellerPrice)`: [2](#0-1) 

When `offer.buy == true` and `tickToPrice(offer.tick) == settlementFee(id, ttm)`, line 44 yields `sellerPrice = offerPrice - offerPrice = 0`. Line 46 then calls `targetSellerAssets.mulDivUp(WAD, 0)` → arithmetic revert.

The comment on line 34 only documents the `offerPrice < settlementFee` revert (underflow in subtraction), not the `offerPrice == settlementFee` case (zero denominator in the inverse). [3](#0-2) 

**Why `Midnight.take()` accepts this offer:**

In the core protocol, `sellerAssets = units.mulDivDown(0, WAD) = 0` — no revert, the seller simply receives zero assets and all buyer payment goes to the settlement fee: [4](#0-3) 

**Why the bundler call reverts entirely:**

`supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` at line 286 **outside** the `try/catch` block that wraps `IMidnight.take()`. The `try/catch` at line 292 only catches reverts from `take()` itself, not from the preceding `sellerAssetsToUnits` call: [5](#0-4) 

The loop condition `filledSellerAssets < targetFilledSellerAssets` (line 282) guarantees `targetFilledSellerAssets - filledSellerAssets > 0` when `sellerAssetsToUnits` is called, so `targetSellerAssets > 0` always — the zero-numerator escape does not apply.

**Attacker-controlled inputs:**

The offer maker (attacker) freely chooses `offer.tick`. The settlement fee is a protocol-level piecewise-linear function of time-to-maturity. The attacker monitors `settlementFee(id, ttm)` and creates a buy offer at the tick where `tickToPrice(tick) == settlementFee`. Any victim taker who includes this offer in a `supplyCollateralAndSellWithAssetsTarget` call will have the entire transaction reverted.

**Existing tests confirm the gap:**

The existing fuzz tests in `TakeAmountsTest.sol` explicitly bound `tick` away from the `sellerPrice == 0` boundary: [6](#0-5) 

The `_maxTick` helper (line 70–74) only enforces `buyerPrice <= WAD`; it does not exclude `sellerPrice == 0`. The lower bound `tick >= 4` does not prevent `tickToPrice(tick) == settlementFee` for small settlement fee values. [7](#0-6) 

### Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee(id, ttm)` at call time reverts entirely. The taker's authorized bundler flow is completely blocked for that transaction. Because the settlement fee changes continuously with time-to-maturity, an attacker can craft an offer that hits this boundary at a predictable future time, or can monitor and submit the offer precisely when the condition holds.

### Likelihood Explanation

**Preconditions:**
1. A buy offer exists with `tickToPrice(offer.tick) == settlementFee(id, ttm)` at the time of the bundler call.
2. A victim taker has authorized the bundler (`isAuthorized[taker][bundler] == true`) and includes this offer in `takes[]`.

**Feasibility:** Settlement fees are set in multiples of `1e12` and tick prices are discrete values from `tickToPrice`. The attacker can enumerate tick prices and find one matching the current or near-future settlement fee. The condition is repeatable: the attacker can create a new offer each time the settlement fee drifts to a matching tick price. The attack is permissionless — no privileged access required.

### Recommendation

Add an explicit guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case when `offer.buy == true`. Since `sellerPrice == 0` means the seller receives zero assets for any number of units, the inverse is undefined for any positive `targetSellerAssets`. The function should revert with a descriptive error (mirroring the `buyerAssetsToUnits` pattern of explicit `require` guards), or return `type(uint256).max` to signal "unreachable target":

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
// Add this guard:
require(sellerPrice > 0, SellerPriceIsZero());
return offer.buy
    ? targetSellerAssets.mulDivUp(WAD, sellerPrice)
    : targetSellerAssets.mulDivDown(WAD, sellerPrice);
```

This mirrors the existing `require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne())` guard in `buyerAssetsToUnits`. [8](#0-7) 

### Proof of Concept

**Foundry fuzz test plan** targeting `test/TakeAmountsTest.sol`:

```solidity
function testSellerAssetsToUnitsRevertsAtSettlementFeeBoundary(
    uint256 settlementFee0,
    uint256 settlementFee1,
    uint256 targetSellerAssets
) public {
    // Set up a non-zero settlement fee.
    uint256 sf = _setSettlementFees(settlementFee0, settlementFee1);
    vm.assume(sf > 0);

    // Find a tick whose price exactly equals the settlement fee.
    // tickToPrice is monotone; search for tick where tickToPrice(tick) == sf.
    uint256 boundaryTick = TickLib.priceToTick(sf, 1);
    vm.assume(TickLib.tickToPrice(boundaryTick) == sf);

    targetSellerAssets = bound(targetSellerAssets, 1, 1e30);

    offer.buy = true;
    offer.tick = boundaryTick;

    // Assert: sellerAssetsToUnits MUST NOT revert for a valid offer Midnight accepts.
    // This will revert with arithmeticError, demonstrating the bug.
    uint256 units = TakeAmountsLib.sellerAssetsToUnits(
        address(midnight), id, offer, targetSellerAssets
    );
    // Expected: returns type(uint256).max or 0, not revert.
    // Actual: reverts with arithmetic underflow.
}
```

**Expected assertion failure:** `vm.expectRevert(stdError.arithmeticError)` passes, confirming the revert. After the fix, the call should either return a sentinel value or revert with a clean `SellerPriceIsZero()` error, and `supplyCollateralAndSellWithAssetsTarget` should skip or handle the offer gracefully.
