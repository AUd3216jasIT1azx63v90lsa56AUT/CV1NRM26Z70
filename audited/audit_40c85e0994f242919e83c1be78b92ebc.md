### Title
Zero-value `safeTransfer` on bad-debt-only liquidation path blocks bad debt realization for revert-on-zero collateral tokens - (File: src/Midnight.sol)

### Summary
When `liquidate` is called with `seizedAssets=0` and `repaidUnits=0` (the documented bad-debt-only path), the accounting block at line 643 is correctly skipped, but `SafeTransferLib.safeTransfer(collateralToken, receiver, 0)` at line 696 is still executed unconditionally. If the collateral token reverts on zero-value `transfer` calls — a known behavior in several ERC-20 implementations — the entire transaction reverts, making it impossible to realize bad debt for that market.

### Finding Description
**Code path:**
- Caller invokes `liquidate(market, collateralIndex, 0, 0, borrower, ...)`.
- Line 595: `atMostOneNonZero(0, 0)` passes (both zero is allowed).
- Line 596: `_position.debt > 0` passes (borrower has debt).
- Lines 607–618: oracle loop computes `badDebt > 0`.
- Lines 620–624: `NotLiquidatable()` check passes (position is unhealthy or post-maturity).
- Lines 626–641: bad debt is realized — `_position.debt` is reduced, `lossFactor` and `totalUnits` are updated. **State is already mutated here.**
- Line 643: `if (repaidUnits > 0 || seizedAssets > 0)` is `false` → entire seized/repaid block skipped; `seizedAssets` remains `0`.
- Line 696: `SafeTransferLib.safeTransfer(collateralToken, receiver, 0)` is called unconditionally.
- Inside `safeTransfer`: `extcodesize` check passes, then `token.call(transfer(receiver, 0))` is issued. If the token reverts on zero-value transfers, `success = false`, and the library propagates the revert (lines 16–19 of SafeTransferLib.sol).
- The entire transaction reverts, **undoing the bad debt accounting** that already happened in lines 626–641.

**Root cause:** The `safeTransfer` at line 696 is not guarded by `if (seizedAssets > 0)`, contradicting the documented invariant that `liquidate(0, 0)` realizes bad debt with zero tokens transferred.

**Attacker inputs:** Any unprivileged liquidator can call `liquidate` with `seizedAssets=0, repaidUnits=0` against any borrower with bad debt in a market whose collateral token reverts on zero-value transfers.

**Existing protections reviewed and insufficient:**
- `atMostOneNonZero` (line 595): explicitly permits both-zero.
- `if (repaidUnits > 0 || seizedAssets > 0)` (line 643): only guards accounting, not the transfer.
- No guard exists before line 696 to skip the call when `seizedAssets == 0`.

### Impact Explanation
Bad debt cannot be realized for any market whose collateral token reverts on zero-value `transfer` calls. The `lossFactor` and `totalUnits` updates that socialize losses among lenders are permanently blocked. Lenders cannot correctly redeem at the adjusted loss factor, and the protocol's invariant that "every credit has matching debt or valid settled/loss state" is violated because bad debt accumulates without being socialized.

### Likelihood Explanation
- Tokens that revert on zero-value transfers exist in production (e.g., LEND, some rebasing tokens, custom vault tokens).
- Any market creator can deploy a market with such a collateral token — market creation is permissionless.
- The bad-debt-only liquidation path (`liquidate(0,0)`) is explicitly documented and expected to be used.
- The condition is repeatable: every call to realize bad debt in such a market will fail.

### Recommendation
Guard the collateral transfer at line 696 with a zero-value check, matching the analogous guard already used for accounting:

```solidity
// line 696 — replace unconditional call with:
if (seizedAssets > 0) {
    SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
}
```

Apply the same guard to line 717 for `repaidUnits`:
```solidity
if (repaidUnits > 0) {
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
}
```

        require(token.code.length > 0, NoCode());
