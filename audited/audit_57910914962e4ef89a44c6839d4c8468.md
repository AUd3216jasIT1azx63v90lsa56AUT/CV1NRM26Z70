Audit Report

## Title
Division by Zero in `liquidate` When `collateralIndex` Not in Borrower's Bitmap with `repaidUnits > 0` - (File: src/Midnight.sol)

## Summary

In `Midnight.liquidate`, `liquidatedCollatPrice` is initialized to `0` and only assigned inside the bitmap loop when `i == collateralIndex`. If `collateralIndex` is absent from the borrower's `collateralBitmap`, the assignment is never reached and `liquidatedCollatPrice` remains `0`. When `repaidUnits > 0` and `seizedAssets == 0`, line 652 executes `mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice)` — a direct division by zero — causing an unconditional panic revert. The Certora spec `NoDivisionByZero.spec` explicitly sidesteps this with a proof assumption rather than a code-level guard, confirming the contract has no corresponding runtime check.

## Finding Description

`liquidatedCollatPrice` is declared as `uint256` (default `0`) at line 603: [1](#0-0) 

The bitmap loop at lines 607–618 iterates only over bits set in `_position.collateralBitmap` and assigns `liquidatedCollatPrice` only when `i == collateralIndex` at line 611: [2](#0-1) 

If `collateralIndex` is not set in `_position.collateralBitmap`, the loop never executes the assignment and `liquidatedCollatPrice` stays `0`. At line 652, when `repaidUnits > 0` and `seizedAssets == 0`: [3](#0-2) 

`mulDivDown` is `(x * y) / d` at line 30 of `src/libraries/UtilsLib.sol`, so `d = 0` triggers a Solidity division-by-zero panic revert: [4](#0-3) 

**No existing check prevents this:**
- Line 595: `atMostOneNonZero(repaidUnits, seizedAssets)` explicitly permits `repaidUnits > 0, seizedAssets == 0`.
- Line 596: `_position.debt > 0` requires a borrower, not that `collateralIndex` is active.
- Lines 620–624: the liquidatability check passes in post-maturity mode on `block.timestamp > market.maturity` alone, independent of which collaterals are active. [5](#0-4) 

There is no `require` that `collateralIndex` must be set in `_position.collateralBitmap` when `repaidUnits > 0`.

**The Certora spec confirms the gap.** `certora/specs/NoDivisionByZero.spec` lines 123–125 add this as a *proof assumption*, not a code invariant, with the comment "Otherwise, liquidate may revert with div by zero": [6](#0-5) 

**Exploit flow:**
1. A multi-collateral market exists with collaterals at indices 0 and 1.
2. A borrower supplies collateral only at index 0 (`collateralBitmap` has bit 0 set, bit 1 not set) and takes on debt.
3. The borrower becomes liquidatable — either post-maturity (`block.timestamp > market.maturity`) or unhealthy.
4. An unprivileged liquidator calls `liquidate(market, collateralIndex=1, seizedAssets=0, repaidUnits=X, borrower, ...)` with `X > 0`.
5. The bitmap loop runs over bit 0 only; `i == 1` is never true; `liquidatedCollatPrice` stays `0`.
6. Line 652 executes `mulDivDown(ORACLE_PRICE_SCALE, 0)` → panic revert.

## Impact Explanation

Any liquidator using the `repaidUnits`-input mode of `liquidate` with a `collateralIndex` not present in the borrower's bitmap receives an unexpected division-by-zero panic revert. This blocks the `repaidUnits` liquidation path for that collateral index entirely. In post-maturity mode, where liquidators may target collateral indices to realize bad debt, the call will always revert for any non-activated index. The borrower remains liquidatable via other activated collateral indices, so this is a partial DoS of the liquidation API rather than a complete freeze of the borrower's position.

## Likelihood Explanation

The preconditions are all reachable without privilege:
- A borrower whose `collateralBitmap` does not include `collateralIndex` is a normal, expected state (borrowers choose which collaterals to supply).
- The borrower being liquidatable (post-maturity or unhealthy) is the standard liquidation precondition.
- A liquidator passing `repaidUnits > 0` with a non-activated `collateralIndex` is a valid API call per the function signature; the NatSpec at lines 578–579 hints at the restriction only for the `(0, 0)` case and does not document it for non-zero inputs. [7](#0-6) 

No privileged action, oracle manipulation, or impossible state is required. The scenario is repeatable any time a liquidator targets a collateral index not in the borrower's bitmap with the `repaidUnits` input.

## Recommendation

Add a runtime guard before the `repaidUnits`/`seizedAssets` branch at line 643 to ensure `collateralIndex` is active in the borrower's bitmap whenever a non-zero liquidation amount is specified:

```solidity
if (repaidUnits > 0 || seizedAssets > 0) {
    require(
        _position.collateralBitmap.getBit(collateralIndex),
        CollateralNotActivated()
    );
    // ... rest of branch
}
```

Alternatively, add the check immediately after the bitmap loop and before line 643, or revert early at the top of `liquidate` when `repaidUnits > 0 && !_position.collateralBitmap.getBit(collateralIndex)`. The Certora spec assumption at line 125 should then be replaced with a verified invariant. [8](#0-7)
