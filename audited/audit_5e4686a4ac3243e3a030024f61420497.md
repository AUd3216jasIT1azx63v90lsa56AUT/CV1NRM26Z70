### Title
ERC2612 Permit Front-Run Causes Silent Allowance Failure, Reverting Entire Bundle Including Health-Maintaining Withdrawals - (File: src/periphery/MidnightBundles.sol)

### Summary
`MidnightBundles.pullToken()` wraps `IERC20Permit.permit()` in a `try/catch` to tolerate pre-consumed permits, but unconditionally calls `SafeTransferLib.safeTransferFrom()` immediately after. If an attacker front-runs the permit consumption and the user has no pre-existing allowance, `safeTransferFrom` reverts, propagating the revert through the entire bundle and cancelling any subsequent `withdrawCollateral()` calls the user needed to maintain position health.

### Finding Description
The exact code path is in `pullToken()`:

```solidity
// src/periphery/MidnightBundles.sol lines 379–384
if (permit.kind == PermitKind.ERC2612) {
    (uint256 deadline, uint8 v, bytes32 r, bytes32 s) =
        abi.decode(permit.data, (uint256, uint8, bytes32, bytes32));
    // Tolerate revert: a third party may have already consumed the permit.
    try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
    SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
``` [1](#0-0) 

The `try/catch` silently discards any revert from `permit()`. The comment acknowledges that a third party may have consumed the permit. However, the subsequent `safeTransferFrom` is unconditional. `SafeTransferLib.safeTransferFrom` propagates the token's revert if `transferFrom` fails: [2](#0-1) 

**Exploit flow:**
1. User constructs a bundle (e.g., `repayAndWithdrawCollateral` or `buyWithUnitsTargetAndWithdrawCollateral`) with a `TokenPermit` of kind `ERC2612`. The user has no pre-existing ERC20 allowance to the bundler — the permit is the sole authorization mechanism.
2. Attacker observes the pending transaction in the mempool, extracts `(deadline, v, r, s)` from calldata, and calls `token.permit(user, bundler, amount, deadline, v, r, s)` directly with higher gas, consuming the nonce.
3. User's bundle executes: `pullToken` → `permit()` reverts (nonce consumed) → silently caught → `safeTransferFrom(token, user, bundler, amount)` executes with zero allowance → token reverts → `safeTransferFrom` re-reverts via the assembly bubble-up path.
4. The entire bundle reverts. `withdrawCollateral()` calls queued after `pullToken` in `repayAndWithdrawCollateral` (lines 336–345) or `buyWithUnitsTargetAndWithdrawCollateral` (lines 91–100) never execute. [3](#0-2) [4](#0-3) 

The attacker's only required input is the permit signature visible in the pending transaction's calldata. No privileged access is needed.

### Impact Explanation
A user who relies on a bundled `repayAndWithdrawCollateral` or `buyWithUnitsTargetAndWithdrawCollateral` to atomically repay debt and withdraw collateral (or buy units and withdraw collateral) cannot complete the operation. The `withdrawCollateral` step — which may be necessary to keep the position healthy or to free funds — is silently blocked. The user's collateral remains locked in the protocol and cannot be retrieved via the bundle. The user must fall back to direct protocol calls, but in a time-sensitive liquidation window this delay can result in the position becoming liquidatable before the user can react.

### Likelihood Explanation
The preconditions are: (1) user submits a bundle with an ERC2612 permit and no prior allowance to the bundler — the standard and expected usage pattern; (2) the transaction is visible in the public mempool before inclusion. Both conditions are routine on any EVM chain with a public mempool. The attack is cheap (one `permit()` call), repeatable on every such bundle, and requires no capital. Any unprivileged address can execute it.

### Recommendation
After the `try/catch`, verify that the allowance is now sufficient before proceeding. Replace the unconditional `safeTransferFrom` with an allowance check:

```solidity
try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
require(IERC20(token).allowance(from, address(this)) >= amount, PermitFailed());
SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
```

This converts a silent mid-bundle revert into an explicit, early revert with a clear error, and prevents the attacker from causing the bundle to fail after partial state changes. Alternatively, skip the `safeTransferFrom` entirely when the allowance is insufficient and surface a dedicated error, allowing callers to detect and handle the front-run case.

### Proof of Concept
**Foundry unit test plan:**

```solidity
function test_frontRunPermitCausesWithdrawalRevert() public {
    // Setup: user has collateral supplied, has a borrow, needs to repay+withdraw
    // User signs an ERC2612 permit for the bundler (no prior approve)
    (uint8 v, bytes32 r, bytes32 s) = signPermit(user, address(bundles), repayAmount, deadline, userKey);
    TokenPermit memory permit = TokenPermit(PermitKind.ERC2612,
        abi.encode(deadline, v, r, s));

    // Attacker front-runs: consumes the permit
    vm.prank(attacker);
    loanToken.permit(user, address(bundles), repayAmount, deadline, v, r, s);

    // Assert: attacker consumed the nonce
    assertEq(loanToken.nonces(user), 1);

    // User's bundle reverts entirely (safeTransferFrom fails, withdrawCollateral never runs)
    vm.prank(user);
    vm.expectRevert(); // safeTransferFrom reverts due to zero allowance
    bundles.repayAndWithdrawCollateral(
        market, repayAmount, user, permit, withdrawals, user, 0, address(0)
    );

    // Assert: collateral was NOT withdrawn — user's position unchanged
    assertEq(midnight.collateral(marketId, user, 0), originalCollateral);
}
```

Expected assertions: the bundle call reverts; collateral balance is unchanged; the user's position remains in the same state as before the attack, confirming the withdrawal was blocked.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
