### Title
Zero-asset rounding bypass allows unlimited takes on fully-consumed buy offers - (File: src/Midnight.sol)

### Summary
When `offer.maxAssets > 0` and `offer.buy == true`, the consumed accounting increments by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. If `buyerPrice < WAD` (tick below a threshold) and `units` is small enough that `mulDivDown` rounds to zero, `consumed` does not increase, so the check `newConsumed <= offer.maxAssets` passes even when `consumed` is already at `maxAssets`. An unprivileged taker can therefore call `take` with non-zero `units` an unlimited number of times after the offer is fully consumed, mutating position state (credit, debt, `totalUnits`) without paying any assets and without advancing the consumed counter.

### Finding Description
**Code path** — `src/Midnight.sol`, lines 363 and 367–369:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [1](#0-0) 

**Root cause** — `consumed` is incremented by `buyerAssets`, not by `units`. `buyerAssets` is computed with `mulDivDown`, which truncates. When `buyerPrice < WAD` (i.e., `offerPrice < WAD`, reachable by choosing a tick below `MAX_TICK` by ≥16 ticks as shown in the test) and `units` is small (e.g., `units = 1`), `mulDivDown(1, buyerPrice, WAD) = 0`. The increment to `consumed` is therefore 0, and `newConsumed = consumed + 0 = consumed`. The guard `require(newConsumed <= offer.maxAssets)` is satisfied even when `consumed == maxAssets`.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.maxUnits = 0`
- `offer.tick` chosen so that `tickToPrice(tick) < WAD` (e.g., `MAX_TICK - 16`)
- `units = 1` (or any value where `mulDivDown(units, buyerPrice, WAD) = 0`)

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = M`, `tick` such that `buyerPrice < WAD`.
2. Taker (or anyone) fills the offer normally until `consumed[maker][group] == M`.
3. Taker calls `take(offer, ..., units=1)`. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. `newConsumed = M + 0 = M ≤ M` → passes. Position state mutates: `buyerPos.credit` increases, `sellerPos.debt` increases, `totalUnits` increases. No tokens are transferred.
4. Step 3 can be repeated indefinitely.

**Why existing checks fail** — The only guard against over-consumption is `require(newConsumed <= offer.maxAssets)`. Because `newConsumed` is not strictly greater than `maxAssets` when the increment is zero, the check never triggers. There is no separate guard requiring `units == 0` when `buyerAssets == 0`, and no check that `consumed < maxAssets` before the fill.

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) explicitly reproduces this exact scenario and asserts that position state changes while `consumed` stays at `maxAssets` and token balances are unchanged. [2](#0-1) 

### Impact Explanation
A fully-consumed buy offer (one where `consumed == maxAssets`) can be taken an unlimited number of times with `units > 0` as long as `buyerPrice < WAD` and `units` is small enough to round `buyerAssets` to zero. Each such take:
- Increases the buyer's (maker's) credit for free.
- Increases the seller's (taker's) debt without any asset payment.
- Increases `totalUnits`, corrupting market-wide accounting.
- Transfers zero tokens, so the maker pays nothing and the taker receives nothing — but the debt/credit ledger is permanently inflated.

This directly violates the core invariant: "offers cannot be replayed, overfilled, reused, or filled after cancel/deadline."

### Likelihood Explanation
**Preconditions:**
1. `offer.buy == true` and `offer.maxAssets > 0` — a standard buy offer with an asset cap.
2. `offerPrice < WAD` — requires `tick < MAX_TICK` by at least 16 ticks; this is a normal, reachable tick range.
3. `consumed == maxAssets` — the offer must first be fully consumed, which happens in normal operation.

All three preconditions are reachable by an unprivileged taker in normal protocol use. The attack is repeatable without limit once the offer is fully consumed, and requires no special permissions, no oracle manipulation, and no admin action.

### Recommendation
Add an explicit guard that rejects non-zero `units` when the asset increment is zero in `maxAssets` mode. In `src/Midnight.sol`, inside the `offer.maxAssets > 0` branch, add:

```solidity
uint256 assetIncrement = offer.buy ? buyerAssets : sellerAssets;
require(units == 0 || assetIncrement > 0, ZeroAssetFill());
newConsumed = consumed[offer.maker][offer.group] += assetIncrement;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

Alternatively, change the cap check to a strict less-than pre-check before incrementing:

```solidity
require(consumed[offer.maker][offer.group] < offer.maxAssets || units == 0, ConsumedAssets());
```

Either fix ensures that once `consumed == maxAssets`, only zero-unit (no-op) takes are permitted.

### Proof of Concept
The existing `testBugBuyMaxAssetsBypass` test in `test/TakeTest.sol` already serves as a unit-test PoC. A fuzz/invariant extension:

```solidity
// Fuzz test: for any buy offer with maxAssets > 0 and buyerPrice < WAD,
// after consumed reaches maxAssets, no take with units > 0 should succeed
// without reverting.
function testFuzz_noReuseAfterFullConsumption(uint256 units) public {
    vm.assume(units > 0);
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

    // Assert: consumed == maxAssets before take
    assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);

    uint256 creditBefore = midnight.creditOf(id, lender);
    uint256 debtBefore   = midnight.debtOf(id, borrower);
    uint256 totalBefore  = midnight.totalUnits(id);

    // This should revert with ConsumedAssets, but currently does not when
    // units.mulDivDown(buyerPrice, WAD) == 0
    take(units, borrower, lenderOffer);

    // Invariant assertions (should never be reached):
    assertEq(midnight.creditOf(id, lender),  creditBefore, "credit must not change");
    assertEq(midnight.debtOf(id, borrower),  debtBefore,   "debt must not change");
    assertEq(midnight.totalUnits(id),        totalBefore,  "totalUnits must not change");
}
```

Expected: the `take` call reverts with `ConsumedAssets`. Actual (unfixed): it succeeds and the three `assertEq` assertions fail, confirming the invariant violation.
