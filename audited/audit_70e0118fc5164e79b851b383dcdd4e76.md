Looking at the exact code path in `MidnightBundles.sol` and `Midnight.sol`:

### Title
Permit Front-Running Causes Repeatable DoS on Bundler Collateral Supply - (File: src/periphery/MidnightBundles.sol)

### Summary
In `MidnightBundles.pullToken`, when `PermitKind.ERC2612` is used, the permit call is wrapped in a silent `try/catch` but the subsequent `SafeTransferLib.safeTransferFrom` is unconditional. An unprivileged attacker can consume the victim's permit nonce by replaying the same `(owner, spender, value, deadline, v, r, s)` directly on the collateral token before the victim's bundler transaction lands, causing `safeTransferFrom` to revert due to insufficient allowance. Because `pullToken` is called before `supplyCollateral`, the entire `supplyCollateralAndSellWithUnitsTarget` transaction reverts with no state changes — the bitmap is never touched, remaining consistent, but the victim's operation is denied.

### Finding Description
**Exact code path:**

`supplyCollateralAndSellWithUnitsTarget` (lines 134–139) iterates over `collateralSupplies` and for each entry calls:

```
pullToken(token, msg.sender, collateralSupplies[i].assets, collateralSupplies[i].permit);  // line 136
forceApproveMax(token, MIDNIGHT);                                                           // line 137
IMidnight(MIDNIGHT).supplyCollateral(...);                                                  // lines 138-139
```

Inside `pullToken` (lines 379–384), for `PermitKind.ERC2612`:

```solidity
try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
``` [1](#0-0) 

The comment at line 382 explicitly acknowledges the front-run scenario ("a third party may have already consumed the permit"), but the design assumption is that the victim pre-approved the bundler via some other mechanism. If the victim has **no pre-existing allowance** and relies solely on the permit, the `safeTransferFrom` at line 384 will revert because `SafeTransferLib.safeTransferFrom` propagates the revert from the token contract. [2](#0-1) 

`supplyCollateral` in `Midnight.sol` updates collateral balance and bitmap **before** its own transfer: [3](#0-2) 

However, because `pullToken` (line 136) is called **before** `IMidnight(MIDNIGHT).supplyCollateral(...)` (line 138), the revert in `pullToken` unwinds the entire call — `supplyCollateral` is never entered, so neither the collateral balance nor the bitmap is modified.

**Attacker-controlled inputs:** The permit parameters `(owner, spender, value, deadline, v, r, s)` are visible in the mempool as calldata to `supplyCollateralAndSellWithUnitsTarget`. The attacker extracts them and calls `collateralToken.permit(owner, spender, value, deadline, v, r, s)` directly, consuming the ERC-2612 nonce.

**Why existing checks fail:** The `try/catch` at line 383 is the only protection, and it is intentionally designed to swallow the revert. There is no fallback check such as "if allowance is already sufficient, skip permit" before the unconditional `safeTransferFrom`.

### Impact Explanation
Any victim who submits `supplyCollateralAndSellWithUnitsTarget` (or `supplyCollateralAndSellWithAssetsTarget`) using `PermitKind.ERC2612` with no pre-existing allowance on the bundler can be griefed repeatedly. Each attempt can be front-run at negligible cost (only gas for the `permit` call). The victim cannot supply collateral through the bundler without switching to a pre-approval or Permit2 flow. The bitmap and all other protocol state remain consistent — this is a pure liveness break, not a corruption. [4](#0-3) 

### Likelihood Explanation
**Preconditions:** (1) Victim uses `PermitKind.ERC2612` in `CollateralSupply.permit`; (2) victim has no pre-existing ERC-20 allowance to the bundler for the collateral token; (3) attacker monitors the mempool (standard on public networks). All three are realistic. The attack is repeatable every time the victim generates a new permit signature, since each new signature produces a new nonce that can again be front-run. Cost to attacker is one `permit` call per grief attempt.

### Recommendation
Before calling `safeTransferFrom`, check whether the current allowance already covers the required amount and skip the transfer-from-revert path:

```solidity
try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
// Only proceed if allowance is now sufficient
if (IERC20(token).allowance(from, address(this)) < amount) revert PermitFailed();
SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
```

Alternatively, require callers to use `PermitKind.Permit2` (which uses unordered nonces and is replay-resistant) or a standard pre-approval (`PermitKind.None`) for collateral tokens, and remove ERC-2612 support from the collateral pull path. [5](#0-4)

    /// @dev Pulls `amount` of `token` from `from` to this bundler, optionally using ERC2612 or Permit2.
    function pullToken(address token, address from, uint256 amount, TokenPermit memory permit) internal {
        if (permit.kind == PermitKind.ERC2612) {
            (uint256 deadline, uint8 v, bytes32 r, bytes32 s) =
                abi.decode(permit.data, (uint256, uint8, bytes32, bytes32));
            // Tolerate revert: a third party may have already consumed the permit.
            try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
            SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
        } else if (permit.kind == PermitKind.Permit2) {
            (uint256 nonce, uint256 deadline, bytes memory signature) =
                abi.decode(permit.data, (uint256, uint256, bytes));
            IPermit2(PERMIT2)
                .permitTransferFrom(
                    IPermit2.PermitTransferFrom(IPermit2.TokenPermissions(token, amount), nonce, deadline),
                    IPermit2.SignatureTransferDetails(address(this), amount),
                    from,
                    signature
                );
        } else {
            SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
        }
    }

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
