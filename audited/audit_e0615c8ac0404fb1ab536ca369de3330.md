### Title
EcrecoverAuthorizer allows authorized operator to self-revoke via victim's nonce, invalidating pre-signed authorizations - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` permits any address that is authorized by `authorization.authorizer` in Midnight to sign and submit an `Authorization` struct on the authorizer's behalf, including one that revokes the signer's own authorization. Because the nonce for `authorization.authorizer` is incremented unconditionally on success, operatorA can consume victim's EcrecoverAuthorizer nonce while revoking their own Midnight authorization, invalidating any pre-signed authorizations victim has issued at that nonce.

### Finding Description
**Root cause:** `EcrecoverAuthorizer.setIsAuthorized` has no guard preventing `authorization.authorized == signer` when `authorization.isAuthorized == false`. The signer validity check at [1](#0-0)  accepts any address that satisfies `isAuthorized(authorization.authorizer, signer)` in Midnight — including `operatorA` itself.

**Exploit flow:**

1. **Preconditions:** `isAuthorized[victim][operatorA] == true` and `isAuthorized[victim][EcrecoverAuthorizer] == true` in Midnight.
2. operatorA constructs `Authorization{authorizer=victim, authorized=operatorA, isAuthorized=false, nonce=N, deadline=...}` where `N` is victim's current EcrecoverAuthorizer nonce.
3. operatorA signs the EIP-712 digest and calls `EcrecoverAuthorizer.setIsAuthorized(auth, sig)` (or any unprivileged third party submits it).
4. Line 26 increments `nonce[victim]` from `N` to `N+1`. [2](#0-1) 
5. The signer check at line 34 passes because `isAuthorized(victim, operatorA) == true`. [1](#0-0) 
6. Line 46-47 calls `midnight.setIsAuthorized(operatorA, false, victim)`. [3](#0-2) 
7. `Midnight.setIsAuthorized` passes its own check (`isAuthorized[victim][EcrecoverAuthorizer] == true`) and sets `isAuthorized[victim][operatorA] = false`. [4](#0-3) 

**Why existing checks fail:** The only authorization check in `EcrecoverAuthorizer` verifies that the signer is either the authorizer or someone the authorizer has authorized in Midnight. There is no check that the signer is not the same address as `authorization.authorized`, and no check that prevents a revocation (`isAuthorized=false`) from being signed by the very address being revoked.

### Impact Explanation
operatorA can grief victim by consuming victim's EcrecoverAuthorizer nonce at any value `N`. If victim has pre-signed `Authorization{authorizer=victim, authorized=operatorB, isAuthorized=true, nonce=N}` and distributed it off-chain (e.g., to a relayer or counterparty), operatorA can front-run the submission by consuming nonce `N` first, permanently invalidating operatorB's pre-signed authorization. Victim must re-sign at nonce `N+1`. This attack is repeatable: operatorA can keep consuming nonces as long as they remain authorized — but since the first call revokes their own authorization, a single execution is the limit per authorization grant. However, if victim re-authorizes operatorA (e.g., as part of normal protocol use), the attack can be repeated.

### Likelihood Explanation
Preconditions are standard protocol usage: victim authorizes EcrecoverAuthorizer (required to use the sig-based flow at all) and authorizes operatorA (normal delegation). The attack requires operatorA to act maliciously or be compromised. It is a low-cost, single-transaction griefing action with no financial requirement. The scenario is most dangerous when victim has distributed pre-signed authorizations off-chain (e.g., for gasless UX flows), which is the primary use case of EcrecoverAuthorizer.

### Recommendation
Add a check in `EcrecoverAuthorizer.setIsAuthorized` that prevents the signer from being the same address as `authorization.authorized` when `authorization.isAuthorized == false`:

```solidity
require(
    authorization.isAuthorized || signer != authorization.authorized,
    Unauthorized()
);
```

This ensures an operator cannot use EcrecoverAuthorizer to revoke their own authorization on behalf of the authorizer, while still allowing them to grant new authorizations or revoke other addresses' authorizations.

### Proof of Concept
```solidity
function testOperatorASelfRevocationConsumesVictimNonce() public {
    // Setup: victim authorizes EcrecoverAuthorizer and operatorA in Midnight
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);
    vm.prank(victim);
    midnight.setIsAuthorized(operatorA, true, victim);

    // Victim pre-signs authorization for operatorB at nonce 0
    Authorization memory victimAuth = Authorization({
        authorizer: victim,
        authorized: operatorB,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory victimSig = signAuthorization(victimAuth, victim);

    // operatorA self-revokes via EcrecoverAuthorizer, consuming victim's nonce 0
    Authorization memory selfRevoke = Authorization({
        authorizer: victim,
        authorized: operatorA,
        isAuthorized: false,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory operatorASig = signAuthorization(selfRevoke, operatorA);
    ecrecoverAuthorizer.setIsAuthorized(selfRevoke, operatorASig);

    // Assert: operatorA is no longer authorized
    assertFalse(midnight.isAuthorized(victim, operatorA));
    // Assert: victim's nonce advanced to 1
    assertEq(ecrecoverAuthorizer.nonce(victim), 1);
    // Assert: victim's pre-signed auth for operatorB at nonce 0 is now invalid (wrong nonce)
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(victimAuth, victimSig);
    assertFalse(midnight.isAuthorized(victim, operatorB));
}
```
