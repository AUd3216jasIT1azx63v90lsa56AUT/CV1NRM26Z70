Audit Report

## Title
Front-Running `consumableUnits` Snapshot Enables Griefing DoS on `buyWithUnitsTargetAndWithdrawCollateral` - (File: src/periphery/MidnightBundles.sol)

## Summary
`buyWithUnitsTargetAndWithdrawCollateral` computes each offer's fillable amount via a live `consumableUnits` read inside the execution loop, then enforces strict equality `require(filledUnits == targetUnits, OutOfOffers())`. Because `Midnight.take` is permissionless for the taker side and `EcrecoverRatifier` signatures do not bind the taker address or units amount, an attacker can extract the maker's reusable signature from the victim's pending mempool transaction and front-run it with a 1-unit take, reducing `consumableUnits` by 1 and guaranteeing `OutOfOffers()` revert at negligible net cost.

## Finding Description

**Code path — `buyWithUnitsTargetAndWithdrawCollateral`:**

The loop at lines 71–85 computes `unitsToTake` using a live call to `ConsumableUnitsLib.consumableUnits`: [1](#0-0) 

The strict equality check at line 88 requires exact fill: [2](#0-1) 

**`ConsumableUnitsLib.consumableUnits` reads `consumed[maker][group]` live:** [3](#0-2) 

**`Midnight.take` taker authorization:**

The only taker-side check is `taker == msg.sender`, which any attacker trivially satisfies: [4](#0-3) 

**`consumed[maker][group]` is incremented on every take:** [5](#0-4) 

**`EcrecoverRatifier` signature does not bind taker or units:**

The signature covers only the Merkle root of the offer tree — not the taker address, not the units amount, not a nonce: [6](#0-5) 

The `ratifierData` tuple `(Signature, root, leafIndex, proof)` is fully visible in the victim's pending mempool transaction and is reusable verbatim by any caller.

**Exploit flow:**

1. Maker creates a sell offer: `offer.maxUnits = N`, `offer.ratifier = EcrecoverRatifier`, signs the Merkle root.
2. Victim constructs `buyWithUnitsTargetAndWithdrawCollateral(targetUnits=N, ..., takes=[{offer, ratifierData=makerSig, units=N}])` and broadcasts.
3. Attacker observes the pending transaction, extracts `offer` and `ratifierData`.
4. Attacker calls `Midnight.take(offer, ratifierData, 1, attacker, address(0), address(0), "")` with higher gas. `consumed[maker][group]` becomes 1.
5. Victim's bundle executes: `consumableUnits(offer)` = N − 1; `unitsToTake` = N − 1; `take` succeeds; `filledUnits` = N − 1.
6. `require(N − 1 == N, OutOfOffers())` reverts.

**Why existing checks fail:**

- The `try/catch` at line 85 only catches reverts from `take` itself; it does not compensate for the shortfall in `filledUnits` caused by a reduced `consumableUnits`. [7](#0-6) 

- The strict equality at line 88 has zero tolerance for any shortfall.
- `TakerUnauthorized` is satisfied trivially by `taker = msg.sender`.
- The shared-group mechanism amplifies the attack: one 1-unit take against any offer sharing `(maker, group)` reduces `consumableUnits` for every other offer in the group simultaneously. [8](#0-7) 

## Impact Explanation
Any victim calling `buyWithUnitsTargetAndWithdrawCollateral` with a tight bundle (total offer capacity ≤ `targetUnits`, which is the normal usage pattern) can be indefinitely griefed. Every attempt reverts with `OutOfOffers()`. The victim's `maxBuyerAssets` are pulled and returned each time (no token loss), but the intended market action — acquiring exactly `targetUnits` of credit — is blocked for as long as the attacker continues the attack. This constitutes a concrete, sustained denial-of-service on a core periphery function.

## Likelihood Explanation
**Preconditions:**
- The offer uses `EcrecoverRatifier` (or any ratifier whose `isRatified` check the attacker can satisfy with data visible in the mempool).
- The victim's bundle has no slack (total offer capacity ≤ `targetUnits`), which is the expected usage.
- The attacker holds enough loan tokens to take ≥ 1 unit (recovered as credit; net cost is only the settlement fee on 1 unit).

**Feasibility:** High. The victim's `ratifierData` is fully visible in the mempool. The attack requires a single `Midnight.take` call with 1 unit. It is repeatable every block at near-zero net cost. The shared-group scenario makes it even cheaper: one 1-unit take can drain `consumableUnits` across all grouped offers simultaneously.

## Recommendation
Replace the strict equality check with a minimum-fill threshold:

```solidity
require(filledUnits >= minUnits, OutOfOffers());
```

where `minUnits` is a caller-supplied parameter (analogous to `minSellerAssets` in `supplyCollateralAndSellWithUnitsTarget`). This allows the bundle to succeed even if `consumableUnits` is reduced by a front-runner, as long as the fill meets the caller's minimum acceptable amount. Alternatively, take a snapshot of `consumableUnits` before the loop and use it as the cap, though this does not eliminate the race condition — only the tolerance parameter fully addresses it.

## Proof of Concept
**Minimal fork test steps:**

1. Deploy `Midnight` and `MidnightBundles` on a local fork.
2. Maker creates a sell offer with `maxUnits = N`, `ratifier = EcrecoverRatifier`, signs the Merkle root.
3. Victim calls `buyWithUnitsTargetAndWithdrawCollateral(targetUnits=N, ..., takes=[{offer, ratifierData, units=N}])`.
4. Before victim's tx is mined, attacker calls `Midnight.take(offer, ratifierData, 1, attacker, address(0), address(0), "")` in the same block with higher priority.
5. Assert: `consumed[maker][group] == 1` after attacker's tx.
6. Assert: victim's tx reverts with `OutOfOffers()`.
7. Assert: victim's `maxBuyerAssets` are returned (no token loss, but action blocked).

Repeat steps 3–7 to demonstrate the attack is indefinitely repeatable.
