### Title
Fee-on-Transfer `loanToken` in `repay` Inflates `withdrawable` Beyond Actual Balance - (`File: src/Midnight.sol`)

### Summary
The `repay` function unconditionally credits `marketState[id].withdrawable += units` before calling `SafeTransferLib.safeTransferFrom(..., units)`. When `market.loanToken` is a fee-on-transfer token, the contract receives only `units * (1 - fee_rate)` but records the full `units` as withdrawable. Because `touchMarket` imposes no restriction on the `loanToken` type, any unprivileged actor can create or interact with such a market, and the resulting accounting gap allows lenders to collectively withdraw more tokens than the contract holds.

### Finding Description
**Exact code path:**

`src/Midnight.sol` `repay` (lines 502–521):
```
position[id][onBehalf].debt -= units;          // line 508
marketState[id].withdrawable += units;          // line 509  ← credited in full
...
safeTransferFrom(market.loanToken, payer, address(this), units);  // line 520 ← only (1-fee)*units arrives
```

`src/libraries/SafeTransferLib.sol` `safeTransferFrom` (lines 24–34) checks only that `transferFrom` returns `true`; it performs no balance-delta check and cannot detect a fee deduction.

`touchMarket` (lines 755–791) validates collateral params, maturity, LLTV tiers, and maxLif — but places **no restriction on `loanToken`**. Any address with a valid ERC20 interface, including a fee-on-transfer token, is accepted.

**Attacker-controlled inputs:**
- `market.loanToken` = a fee-on-transfer ERC20 (fee rate `f` in [1, 10000] bps)
- `units` = any positive value ≤ borrower's outstanding debt

**Exploit flow:**
1. A market is created (permissionlessly via `touchMarket`) with a fee-on-transfer token as `loanToken`.
2. A lender supplies credit and a borrower accumulates debt through normal `take` interactions.
3. Borrower calls `repay(market, N, onBehalf, address(0), '')`.
4. Line 509: `withdrawable += N` — full `N` recorded.
5. Line 520: contract receives only `N * (1 - f)` tokens.
6. Gap = `N * f` tokens: `withdrawable` now exceeds the contract's actual token balance by this amount.
7. Step 3–6 can be repeated on every repayment, compounding the gap.
8. Lenders call `withdraw`; each `withdraw` sends `units` tokens (line 499) and decrements `withdrawable` by `units` (line 494). Early lenders are made whole; the last lender(s) find the contract insolvent and their `withdraw` call reverts (insufficient balance), permanently freezing their funds.

**Why existing checks fail:**
- `SafeTransferLib.safeTransferFrom` only validates the boolean return of `transferFrom`; it cannot observe the fee deduction.
- There is no post-transfer balance check in `repay`.
- `touchMarket` has no token allowlist or fee-on-transfer guard.
- The Certora `WithdrawableMonotonicity` spec rule `repayIncreasesWithdrawable` (line 16) asserts `withdrawableAfter == withdrawableBefore + units` — this rule is satisfied by the contract's accounting but does not model fee-on-transfer tokens, so it does not catch the invariant break against actual balances.

### Impact Explanation
The core invariant `token_balance(address(this)) >= withdrawable` is violated. Lenders who withdraw first drain the real token balance; subsequent lenders cannot withdraw their credited funds. The shortfall equals the cumulative fee deducted across all repayments, which grows with market usage. Funds are permanently frozen for the last lenders to attempt withdrawal.

### Likelihood Explanation
**Preconditions:** A market must exist with a fee-on-transfer `loanToken`. Market creation is permissionless — any actor can call `touchMarket` with an arbitrary `loanToken`. Fee-on-transfer tokens (e.g., tokens with a built-in transfer tax) are a well-known ERC20 variant. No privileged action is required. The bug is triggered on every `repay` call in such a market, making it repeatable and cumulative.

### Recommendation
After the `safeTransferFrom` call in `repay`, verify that the contract received at least `units` tokens by comparing balances before and after:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
require(
    IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore >= units,
    InsufficientTransferAmount()
);
```

Alternatively, apply the same pattern to all inbound transfers (`take`, `supplyCollateral`, `flashLoan` repayment) or document and enforce via `touchMarket` that fee-on-transfer tokens are not supported (e.g., by requiring a zero-fee transfer check at market creation time).

contract FeeToken is ERC20 {
    uint256 public feeBps;
    constructor(uint256 _feeBps) { feeBps = _feeBps; }
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {

        super.transferFrom(from, to, amount - fee); // only (amount - fee) arrives
        return true;
    }

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
