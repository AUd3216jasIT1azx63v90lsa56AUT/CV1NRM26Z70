Audit Report

## Title
Fee-on-Transfer Loan Token Causes Lenders to Receive Less Than Credit Entitlement on Withdraw - (File: src/Midnight.sol)

## Summary
The `withdraw` function decrements `credit`, `withdrawable`, and `totalUnits` by the full `units` amount before calling `SafeTransferLib.safeTransfer`, which only verifies call success and the boolean return value — not the receiver's actual balance delta. When the market's `loanToken` is a fee-on-transfer token, the receiver receives `units * (1 - fee_rate)` while the protocol's accounting is permanently reduced by the full `units`. Because `touchMarket` is permissionless and imposes no restrictions on `loanToken`, any user can create such a market, and every lender withdrawal in it results in an unrecoverable shortfall.

## Finding Description
**Root cause — `withdraw` in `src/Midnight.sol` lines 493–499:**

The accounting state is decremented by the full `units` before the transfer executes:

```solidity
_position.credit -= UtilsLib.toUint128(units);          // line 493
_marketState.withdrawable -= UtilsLib.toUint128(units); // line 494
_marketState.totalUnits -= UtilsLib.toUint128(units);   // line 495
// ...
SafeTransferLib.safeTransfer(market.loanToken, receiver, units); // line 499
``` [1](#0-0) 

**`SafeTransferLib.safeTransfer` (`src/libraries/SafeTransferLib.sol` lines 12–22)** only checks `success` and the ABI-decoded boolean return of `transfer`. It performs no balance-before/balance-after check and cannot detect that the receiver received fewer tokens than `units`. [2](#0-1) 

**`touchMarket` (`src/Midnight.sol` lines 755–791)** validates collateral params, maturity, LLTV, and `maxLif`, but imposes zero constraints on `market.loanToken`. Any ERC20 — including fee-on-transfer tokens — is accepted. [3](#0-2) 

**Compound accounting break across the full lifecycle:**

- On `repay` (line 520): `safeTransferFrom(market.loanToken, payer, address(this), units)` — the contract receives `units * (1 - fee_rate)` but credits the full `units` to `withdrawable` (line 509). The contract is immediately underfunded. [4](#0-3) 

- On `withdraw` (line 499): `safeTransfer(market.loanToken, receiver, units)` — the contract's balance decreases by `units` but the receiver only receives `units * (1 - fee_rate)`. The fee is silently absorbed by the token contract. [5](#0-4) 

**Exploit flow:**
1. Any user calls `touchMarket` with a fee-on-transfer token as `loanToken` (permissionless, no token restrictions).
2. A lender lends N units via `take`; the contract receives `buyerAssets * (1 - fee_rate)` but credits `buyerAssets` to the lender's position.
3. A borrower repays N units; the contract receives `N * (1 - fee_rate)` but credits N to `withdrawable`.
4. The lender calls `withdraw(market, N, lender, receiver)`:
   - `credit`, `withdrawable`, `totalUnits` all decrease by N (full amount).
   - `safeTransfer` succeeds (token returns `true`).
   - Receiver's balance increases by only `N * (1 - fee_rate)`.
5. No revert, no check, no recovery path.

**Why existing checks fail:**

`SafeTransferLib.safeTransfer` only checks `success` and the ABI-decoded boolean return — it cannot detect fee-on-transfer shortfalls. There is no balance-before/after guard anywhere in `withdraw`. `touchMarket` has no loan token allowlist or fee-on-transfer check. [6](#0-5) 

## Impact Explanation
Every lender withdrawal in a fee-on-transfer loan token market results in the receiver receiving `units * (1 - fee_rate)` tokens instead of `units`. The credit accounting is permanently reduced by the full `units`, so the lender cannot re-withdraw the shortfall. The loss is proportional to the fee rate and repeats on every withdrawal, draining lender value systematically. The contract's actual token balance also diverges from its tracked `withdrawable` state, eventually causing later withdrawals to revert due to insufficient balance, freezing remaining lenders' funds. This constitutes both direct theft of lender value and permanent fund freeze — both in-scope impact classes per `RESEARCHER.md`. [7](#0-6) 

## Likelihood Explanation
`touchMarket` is permissionless — any user can create a market with any ERC20 as the loan token. [8](#0-7) 
Fee-on-transfer tokens (deflationary tokens, tokens with protocol fees) exist in production. A lender who participates in such a market — whether knowingly or not — will suffer the loss on every `withdraw` call. The condition is fully attacker-controllable (market creation) and repeatable on every withdrawal. No privileged access is required. `SECURITY.md` contains no exclusion for fee-on-transfer token behavior. [9](#0-8) 

## Recommendation
Implement a balance-before/balance-after check in `withdraw` to ensure the receiver's actual received amount matches `units`, and revert if the delta is less than expected. Alternatively, document and enforce an explicit allowlist of supported `loanToken` types in `touchMarket`, rejecting tokens with transfer fees. The accounting decrements should only reflect the actual amount transferred, not the requested `units`.

## Proof of Concept
1. Deploy a mock ERC20 with a 1% transfer fee (fee deducted from receiver on every `transfer`/`transferFrom`).
2. Call `touchMarket` with this token as `loanToken` — succeeds with no restriction.
3. Lender calls `take` to lend 1000 units; contract receives 990, credits 1000 to lender position.
4. Borrower calls `repay(market, 1000, borrower, ...)`: contract receives 990, `withdrawable` increases by 1000.
5. Lender calls `withdraw(market, 1000, lender, receiver)`:
   - `credit` decremented by 1000 (now 0).
   - `withdrawable` decremented by 1000.
   - `safeTransfer` called for 1000; receiver gets 990; fee token returns `true`.
   - No revert. Lender lost 10 tokens with no recourse.
6. Repeat for all lenders until contract balance is exhausted; final lenders' `withdraw` calls revert due to insufficient balance.

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
    }
