Audit Report

## Title
Signature Malleability in `setIsAuthorized` Enables Front-Running to Consume Victim's Nonce - (File: `src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` passes raw `(v, r, s)` to `ecrecover` with no check that `s` is in the lower half of the secp256k1 curve order. Because the function is fully permissionless, an attacker can observe a pending `setIsAuthorized` transaction in the mempool, compute the malleable signature `(v', r, s')`, and front-run the victim to consume their nonce ‚Äî causing the victim's transaction to revert with `InvalidNonce()` and wasting their gas.

## Finding Description
**Root cause:** No low-`s` guard exists in `setIsAuthorized`. At line 31 of `src/periphery/EcrecoverAuthorizer.sol`:
```solidity
address signer = ecrecover(digest, signature.v, signature.r, signature.s);
```
There is no `require(uint256(signature.s) <= 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0)` guard. The `Signature` struct in `src/periphery/interfaces/IEcrecoverAuthorizer.sol` lines 5‚Äì9 is a plain `(uint8 v, bytes32 r, bytes32 s)` with no constraints.

**Permissionless entry:** `setIsAuthorized` never validates `msg.sender` against `authorization.authorizer`. The only `msg.sender` reference is in the emitted event at line 39. Any caller can submit any valid signed authorization.

**Exploit flow:**
1. Victim signs `Authorization{authorizer, authorized, isAuthorized, nonce=N, deadline}` producing `(v, r, s)` and broadcasts `setIsAuthorized(auth, {v, r, s})`.
2. Attacker observes the mempool, computes the malleable form: `v' = 55 - v` (flips 27‚Üî28), `s' = secp256k1_order - s`.
3. Attacker front-runs with `setIsAuthorized(auth, {v', r, s'})`.
4. Attacker's tx: line 26 nonce check passes (`nonce[authorizer] == N`), nonce incremented to N+1; `ecrecover(digest, v', r, s')` returns the same `authorizer` address; `InvalidSignature()` check at line 32 passes; authorization is written to Midnight.
5. Victim's tx: line 26 fails (`nonce[authorizer]` is now N+1 ‚âÝ N) ‚Üí reverts with `InvalidNonce()`.

**Why existing checks fail:** The `InvalidNonce()` check at line 26 and `InvalidSignature()` check at line 32 both pass for the malleable form ‚Äî the nonce is valid at the time of the attacker's front-run, and `ecrecover` returns a non-zero address identical to the original signer.

## Impact Explanation
The victim's `setIsAuthorized` transaction is griefed: it reverts with `InvalidNonce()` and the victim wastes gas. Because the `authorization` struct is fixed by the victim's signature, the attacker cannot alter the authorization's content ‚Äî the same authorization is applied. The concrete harms are: (1) victim's tx reverts and gas is lost; (2) the victim loses control over the timing and submission of their own signed message. The attack is repeatable for every new signature the victim produces, enabling sustained griefing. Note: after the attacker consumes nonce N, the on-chain nonce advances to N+1, so the victim's pre-signed nonce N+1 message (if any) remains valid.

## Likelihood Explanation
Preconditions are minimal: the attacker only needs to monitor the public mempool and perform one modular subtraction (`s' = n - s`) and one subtraction (`v' = 55 - v`). No special role, capital, or privileged access is required. The attack is repeatable for every `setIsAuthorized` submission and costs only the gas of a single transaction. On any chain without private mempools, this is trivially executable.

## Recommendation
Add a low-`s` guard before the `ecrecover` call:
```solidity
require(
    uint256(signature.s) <= 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0,
    InvalidSignature()
);
```
Alternatively, replace the raw `ecrecover` call with OpenZeppelin's `ECDSA.recover`, which enforces the low-`s` constraint and reverts on zero-address results, eliminating both the malleability and the zero-address check in one step.

## Proof of Concept
1. Deploy `EcrecoverAuthorizer` pointing to a mock `Midnight`.
2. Generate a valid `Authorization` with `nonce = 0` and sign it, obtaining `(v, r, s)`.
3. Compute `v' = 55 - v`, `s' = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141 - s`.
4. Call `setIsAuthorized(auth, {v', r, s'})` from the attacker address ‚Äî expect success, nonce advances to 1.
5. Call `setIsAuthorized(auth, {v, r, s})` from the victim address ‚Äî expect revert with `InvalidNonce()`.
6. Confirm `nonce[authorizer] == 1` and that the authorization state was set by the attacker's call.
