### Title
Zero-`buyerAssets` rounding allows unlimited replay of `maxAssets`-capped buy offers without consuming capacity — (`src/Midnight.sol`)

### Summary

When `offer.buy == true` and `offer.maxAssets > 0`, the `take` function increments `consumed` by `buyerAssets`, which is computed as `units.mulDivDown(buyerPrice, WAD)`. If `units * buyerPrice < WAD`, this rounds to zero, so `consumed` is never incremented. Because the `EcrecoverRatifier` is stateless (it re-validates the same Merkle proof and signature on every call), an unprivileged taker can call `take` an unbounded number of times with the same valid `ratifierData`, each time receiving non-zero credit/debt without consuming any of the offer's `maxAssets` capacity. The protocol itself acknowledges this in a comment and in a test named `testBugBuyMaxAssetsBypass`, confirming the behavior is real.

### Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

`mulDivDown` truncates: when `units * buyerPrice < WAD`, `buyerAssets == 0`. The `+=` adds zero to `consumed`, and `require(newConsumed <= offer.maxAssets)` trivially passes. No other guard prevents a zero-increment take from succeeding.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets = N > 0`
- `offer.tick` set to a tick where `tickToPrice(tick) < WAD` (e.g. `MAX_TICK - 16` as used in `testBugBuyMaxAssetsBypass`)
- `units = 1` (or any value where `units * offerPrice < WAD`)
- `ratifierData` = a valid EcrecoverRatifier Merkle proof + signature (identical on every call)

**Why the EcrecoverRatifier does not stop replay:**
`EcrecoverRatifier.isRatified` (`src/ratifiers/EcrecoverRatifier.sol` lines 33–46) is purely stateless — it checks the Merkle proof, the root-cancellation flag, and the ECDSA signature. It stores no per-take nonce or counter. The same `ratifierData` bytes pass validation on every invocation.

**Position state still mutates each call (lines 408–414):**
```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);
```
Even with `buyerAssets == 0`, `units` are processed in full: the buyer accumulates credit and the seller accumulates debt on every replay, with zero asset transfer and zero consumed increment.

**Existing test confirms the behavior** (`test/TakeTest.sol` lines 857–889, `testBugBuyMaxAssetsBypass`): the test pre-fills `consumed` to `maxAssets`, then calls `take(1, ...)` and asserts `buyerAssets == 0`, `consumed` unchanged, yet `creditOf` and `debtOf` strictly increase. The protocol comment at `src/Midnight.sol` line 94 also acknowledges: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*

### Impact Explanation

An offer with `maxAssets = N` is supposed to cap the total buyer assets transferred to `N`. With this bug, a taker can call `take` K times (K unbounded) with `units` chosen so `buyerAssets = 0`, accumulating K × `units` of credit for the buyer and K × `units` of debt for the seller, while `consumed` stays at its initial value. The `maxAssets` cap is completely bypassed. The EcrecoverRatifier's Merkle-tree authorization — intended to limit which offers can be taken — provides no per-take consumption guard, so the same signed offer is effectively replayable indefinitely.

### Likelihood Explanation

**Preconditions:**
1. `offer.buy == true` and `offer.maxAssets > 0` — a normal lender buy offer with an asset cap.
2. `offerPrice < WAD` — requires `tickToPrice(tick) < WAD`, achievable with any tick below the WAD-price tick; the test uses `MAX_TICK - 16`.
3. `units` small enough that `units * offerPrice < WAD` — with `offerPrice` near zero, even `units = 1` suffices.
4. A valid EcrecoverRatifier proof — the taker only needs to obtain the maker's signed Merkle root once; the same bytes work on every replay.

All preconditions are reachable by an unprivileged taker with no special access. The attack is repeatable in a single transaction via a callback or across multiple transactions.

### Recommendation

Add a guard that prevents a non-zero `units` take from producing a zero consumed increment in assets mode. Concretely, when `offer.maxAssets > 0` and `offer.buy == true`, require that `buyerAssets > 0` whenever `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetsTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` such that the resulting assets are always non-zero, or switch the consumed tracking to `units` (not assets) when `maxAssets > 0` and `offerPrice < WAD`.

### Proof of Concept

The existing `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) already demonstrates the single-take case. Extend it to a replay loop:

```solidity
function testMaxAssetsBuyOfferReplayViaZeroBuyerAssets() public {
    // Setup: buy offer with maxAssets = 1, tick such that offerPrice < WAD
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // tickToPrice < WAD

    deal(address(loanToken), lender, 0);
    collateralize(market, borrower, 1000);

    uint256 K = 100;
    for (uint256 i = 0; i < K; i++) {
        // units=1, buyerAssets=0 due to rounding; same ratifierData each time
        (uint256 buyerAssets,) = take(1, borrower, lenderOffer);
        assertEq(buyerAssets, 0);
    }

    // consumed must still be 0 — maxAssets cap was never enforced
    assertEq(midnight.consumed(lender, lenderOffer.group), 0);

    // But lender has accumulated K units of credit — offer was replayed K times
    assertGt(midnight.creditOf(id, lender), 0);

    // Invariant violation: offer with maxAssets=1 was filled K=100 times
    // Expected: test should FAIL (i.e., the loop should revert before K iterations)
    // Actual: test PASSES, demonstrating the bug
}
```

**Key assertions:**
- `consumed` remains 0 after K takes (never incremented) — violates the cap invariant.
- `creditOf(lender) > 0` after K takes — real position state changed K times.

    function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
        require(msg.sender == MIDNIGHT, NotMidnight());
        (Signature memory sig, bytes32 root, uint256 leafIndex, bytes32[] memory proof) =
            abi.decode(ratifierData, (Signature, bytes32, uint256, bytes32[]));
        require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
        require(!isRootCanceled[offer.maker][root], RootCanceled());
        bytes32 structHash = keccak256(abi.encode(HashLib.offerTreeTypeHash(proof.length), root));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, structHash));

        require(_signer != address(0), InvalidSignature());
        require(_signer == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer), Unauthorized());
