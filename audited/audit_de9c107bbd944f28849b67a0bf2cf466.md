### Title
Assets-mode offer consumed counter not incremented when `buyerAssets` rounds to zero, allowing unbounded credit accumulation beyond `maxAssets` cap - (`src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and `offerPrice < WAD`, calling `take()` with `units=1` produces `buyerAssets = floor(1 * buyerPrice / WAD) = 0`. The consumed counter is incremented by `buyerAssets` (zero), so `consumed[maker][group]` never advances and the `ConsumedAssets` guard never fires. Yet `units=1` is fully applied to positions: the maker gains 1 credit and the taker gains 1 debt on every such call. Any taker can repeat this indefinitely, giving the maker unbounded credit far beyond the `maxAssets / offerPrice` unit ceiling the maker intended.

### Finding Description
**Exact code path — `src/Midnight.sol` lines 363–373:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → 0 when units=1, buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // 0 <= maxAssets → always passes
}
``` [1](#0-0) 

Position accounting then runs unconditionally:

```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);  // +1 per call
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);  // +1 per call
``` [2](#0-1) 

**Preconditions for the attack:**
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.maxUnits = 0`
- `offerPrice < WAD` — achievable at any tick below `MAX_TICK/2` (tick < 2910); `tickToPrice` returns values in `[0, 1e18]`
- Attacker is any address that is not the maker (enforced by `SelfTake` check)
- Attacker has a valid ratifier signature (the signed offer is public once broadcast)

**Why existing checks fail:**
The `ConsumedAssets` guard compares `newConsumed` (which is `consumed + buyerAssets`) against `maxAssets`. When `buyerAssets = 0`, `newConsumed` equals the pre-call value of `consumed`, so the guard is trivially satisfied regardless of how many times `take` has been called. There is no separate check that `units > 0` implies `buyerAssets > 0`, and no floor on `buyerAssets` before the consumed update.

The codebase itself documents this gap at line 94: [3](#0-2) 

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) confirms the exact state change: `buyerAssets = 0`, `sellerAssets = 0`, consumed unchanged, yet `creditOf(lender)` and `debtOf(borrower)` both strictly increase. [4](#0-3) 

### Impact Explanation
The maker's credit grows without bound: each `take(units=1)` call adds 1 unit of credit to the maker while the consumed counter stays frozen. The maker's intended cap of `maxAssets / offerPrice` units of credit exposure is never enforced. A maker who set `maxAssets = 1000` at a tick where `offerPrice = 0.5e18` intended at most 2000 units of credit; an attacker can push that to any value. Because bad debt is socialized proportionally to credit held at the time of realization, the maker suffers losses proportional to their inflated credit balance, not their intended exposure.

### Likelihood Explanation
The precondition `offerPrice < WAD` is satisfied for any tick below 2910 (half of `MAX_TICK = 5820`). This covers the entire lower half of the tick range, which is the natural range for buy offers (buyers prefer lower prices). The attacker only needs a valid ratifier proof for the offer — for `EcrecoverRatifier`-based offers this is the maker's public signature, which is broadcast when the offer is first taken. The attack is repeatable with no per-iteration asset cost to the attacker beyond gas and the collateral needed to back the accumulated debt (which can be repaid at maturity or earlier).

### Recommendation
Before applying position changes, require that a non-zero `units` input produces a non-zero asset amount when `maxAssets > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetsTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, track consumed in units (not assets) even in `maxAssets` mode and convert the cap: `require(newConsumedUnits <= maxAssets.mulDivDown(WAD, offerPrice))`. Either fix closes the rounding gap that allows `units > 0` to contribute zero to the consumed counter.

### Proof of Concept
**Foundry unit test plan** (mirrors `testBugBuyMaxAssetsBypass` but driven by an unprivileged attacker, not `setConsumed`):

```solidity
function testAttackerOverfillsBuyOffer() public {
    // Setup: lender posts a buy offer with offerPrice < WAD and maxAssets = 10
    lenderOffer.maxUnits  = 0;
    lenderOffer.maxAssets = 10;
    lenderOffer.tick      = MAX_TICK - 16;  // offerPrice < WAD → buyerAssets rounds to 0 for units=1

    // Attacker (borrower) has collateral to back debt
    collateralize(market, borrower, 1000);

    uint256 creditBefore = midnight.creditOf(id, lender);

    // Attacker calls take(1) 500 times — consumed never advances
    for (uint256 i; i < 500; i++) {
        take(1, borrower, lenderOffer);
    }

    // Invariant assertions
    assertEq(midnight.consumed(lender, lenderOffer.group), 0,
        "consumed must stay 0 — bug: it does");
    assertGt(midnight.creditOf(id, lender), creditBefore + 10,
        "maker credit far exceeds maxAssets/offerPrice cap");
    assertLe(midnight.creditOf(id, lender) - creditBefore,
        lenderOffer.maxAssets * WAD / TickLib.tickToPrice(lenderOffer.tick),
        "INVARIANT VIOLATED: credit increase exceeds intended unit cap");
}
```

Expected result: the final `assertLe` fails, confirming the invariant is broken. `consumed` stays at 0 throughout while `creditOf(lender)` grows by 500 units.
