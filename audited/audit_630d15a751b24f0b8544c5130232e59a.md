### Title
Fee-on-Transfer Collateral Tokens Inflate Position Accounting, Enabling Undercollateralized Borrowing and Socialized Bad Debt — (`src/Midnight.sol`)

---

### Summary

`Midnight.sol` is a permissionless protocol where any address can create a market with any ERC20 token as collateral or loan token. The contract explicitly documents TOKEN SAFETY REQUIREMENTS (lines 133–140) but enforces none of them on-chain. When a fee-on-transfer token is used as collateral, `supplyCollateral()` credits the borrower's position with the full nominal `assets` amount while the contract receives only `assets - fee`. The resulting phantom collateral allows the borrower to take on debt that exceeds the real collateral backing, producing undercollateralized positions whose bad debt is then socialized across all lenders in the market.

---

### Finding Description

**Vulnerability class:** Accounting / token-balance discrepancy

**Root cause — `supplyCollateral()` (lines 524–546):**

```solidity
// line 533 — position credited with full nominal amount
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);

// line 545 — contract actually receives (assets - fee) for fee-on-transfer tokens
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

The state write at line 533 happens unconditionally before the transfer at line 545. No balance-before / balance-after check exists. For a token that deducts a percentage fee on every transfer, the contract's real balance increases by `assets × (1 - feeRate)`, but the position records `assets`. The gap is permanent phantom collateral.

**Downstream impact path:**

1. `isHealthy()` (line 944) reads `_position.collateral[i]` and multiplies by the oracle price and LLTV to compute `maxDebt`. Because `_position.collateral[i]` is inflated, `maxDebt` is inflated.
2. The borrower can take debt up to the inflated `maxDebt` via `take()`.
3. At liquidation time, `liquidate()` (line 696) calls `SafeTransferLib.safeTransfer(collateralToken, receiver, seizedAssets)`. The contract holds less collateral than recorded, so either the transfer reverts (DoS on liquidation) or the liquidator receives less than expected.
4. If the position cannot be fully liquidated, bad debt is realized and socialized via `lossFactor` (lines 631–634), slashing all lenders proportionally.

**Secondary accounting break — `repay()` (lines 502–521):**

```solidity
// line 509 — withdrawable inflated by full units
marketState[id].withdrawable += UtilsLib.toUint128(units);

// line 520 — contract receives (units - fee)
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
```

`withdrawable` is used by `withdraw()` (line 494) to gate lender redemptions. Inflated `withdrawable` means lenders can attempt to withdraw more than the contract holds; the last lenders to withdraw will find the contract insolvent.

**Tertiary drain — `flashLoan()` (lines 737–752):**

```solidity
SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);      // sends assets[i], callback gets assets[i]-fee
// ...
SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]); // pulls assets[i], gets assets[i]-fee
```

Each flash loan of a fee-on-transfer token costs the contract `2 × fee`. Any caller can repeat this to drain the contract's entire balance of that token.

---

### Impact Explanation

- **Direct loss of lender funds** via bad-debt socialization: a borrower who supplies 1,000 units of a 5% fee-on-transfer collateral token has 1,000 recorded but only 950 real. They borrow against 1,000, leaving a 50-unit undercollateralized gap that becomes bad debt slashed from all lenders' `credit`.
- **Lender insolvency on repay path**: inflated `withdrawable` allows early lenders to drain the contract, leaving later lenders with uncollectable claims.
- **Iterative balance drain via flash loans**: any actor can repeatedly flash-loan a fee-on-transfer token held by the contract, extracting the fee from the contract's balance each time with zero capital at risk.

Severity: **High** — direct, quantifiable loss of deposited funds with no privileged access required.

---

### Likelihood Explanation

- Market creation is fully permissionless (`touchMarket()`, lines 755–791). No on-chain token whitelist exists.
- Fee-on-transfer tokens are a well-known, deployed token class (e.g., STA, PAXG, tokens with reflection mechanics).
- An attacker can deploy a custom fee-on-transfer ERC20, create a market, and self-supply collateral to exploit the accounting gap immediately.
- Alternatively, an existing market that happens to use a fee-on-transfer token (e.g., one added to a frontend whitelist by mistake) is exploitable by any borrower without any special setup.

Likelihood: **Medium** — requires a market with a non-compliant token, which is trivially achievable by any permissionless actor.

---

### Recommendation

1. **Measure actual received amount** in `supplyCollateral()` and `repay()` using a balance-before / balance-after pattern, and credit only the delta:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Apply the same pattern in `repay()` before incrementing `withdrawable`, and in `flashLoan()` before asserting repayment.

2. **Alternatively**, enforce the TOKEN SAFETY REQUIREMENTS on-chain by maintaining a registry of approved tokens and reverting `touchMarket()` for unapproved tokens.

### Proof of Concept

**Setup:**
- Deploy `FeeToken` — an ERC20 that deducts 10% on every `transferFrom` (i.e., recipient receives 90% of the requested amount).
- Create a market in `Midnight` with `FeeToken` as `collateralParams[0].token`.

**Exploit steps:**

1. Attacker calls `supplyCollateral(market, 0, 1000e18, attacker)`.
   - Line 533 records `_position.collateral[0] = 1000e18`.
   - Line 545 transfers 1000e18 from attacker; contract receives `900e18` (10% fee deducted).
   - **Phantom collateral: 100e18 units.**

2. Attacker calls `take()` as seller against a buy offer, borrowing up to `maxDebt` computed from the inflated 1000e18 collateral. With LLTV = 0.8, `maxDebt = 800e18` units, but real backing supports only `720e18`.

3. Attacker defaults. Liquidator calls `liquidate()` with `seizedAssets = 1000e18`.
   - Line 696: `SafeTransferLib.safeTransfer(collateralToken, receiver, 1000e18)` — contract only holds `900e18`, transfer reverts or sends less.
   - Bad debt of `≥80e18` units is realized and socialized via `lossFactor` (lines 631–634), slashing all lenders.

4. **Flash loan drain (separate vector):** Attacker calls `flashLoan([feeToken], [900e18], attackerCallback, "")`.
   - Contract sends 900e18 → callback receives 810e18.
   - Contract pulls 900e18 → contract receives 810e18.
