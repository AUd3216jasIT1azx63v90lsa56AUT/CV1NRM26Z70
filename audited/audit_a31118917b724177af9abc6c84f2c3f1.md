Audit Report

## Title
Fee-on-transfer collateral token inflates stored balance, enabling undercollateralized borrow - (`src/Midnight.sol`, `src/libraries/SafeTransferLib.sol`)

## Summary
`supplyCollateral` writes the caller-supplied `assets` amount directly into `_position.collateral[collateralIndex]` before executing the token transfer, and `SafeTransferLib.safeTransferFrom` performs no balance-delta check. When the collateral token charges a transfer fee, only `assets - fee` arrives at the contract while the full `assets` is stored. `isHealthy` then computes `maxDebt` from the inflated stored value, allowing the borrower to take debt that exceeds the actual on-chain collateral, creating bad debt for lenders.

## Finding Description

**Root cause — write-before-transfer with no balance check:**

In `src/Midnight.sol` lines 531–545, `supplyCollateral` first writes the full `assets` to storage, then calls the transfer:

```solidity
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets); // line 533 — inflated write
...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545 — only assets-fee arrives
```

`SafeTransferLib.safeTransferFrom` in `src/libraries/SafeTransferLib.sol` lines 24–34 only verifies the boolean return value of `transferFrom`; it performs no before/after balance snapshot:

```solidity
(bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
...
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
```

**Health check reads inflated value:**

`isHealthy` in `src/Midnight.sol` lines 954–955 reads `_position.collateral[i]` (the inflated stored value) to compute `maxDebt`:

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

**Market creation is permissionless and imposes no token-type restriction:**

`touchMarket` in `src/Midnight.sol` lines 755–791 validates only maturity bound, non-empty/sorted collateral params, allowed LLTV tiers, and valid `maxLif`. Any ERC20 — including fee-on-transfer tokens — is accepted as a collateral token.

**Certora formal verification does not cover this:**

`certora/specs/CreatedMarkets.spec` line 33 explicitly models `SafeTransferLib.safeTransferFrom` as `NONDET`, bypassing any fee-on-transfer behavior in the formal model.

**Exploit flow:**
1. Attacker deploys or uses an existing fee-on-transfer ERC20 (e.g., 1% fee).
2. Attacker calls `touchMarket` with this token as `collateralParams[i].token`. No restriction prevents this.
3. Attacker calls `supplyCollateral(market, i, 1000e18, attacker)`. Storage records `_position.collateral[i] = 1000e18`, but only `990e18` arrives.
4. Attacker calls `take` (sell offer) to borrow. `isHealthy` computes `maxDebt` using the stored `1000e18`, not the actual `990e18`. The position passes the health check.
5. The attacker holds a position where `debt > actual_collateral_value * lltv / WAD`. Each additional `supplyCollateral` call compounds the discrepancy.

## Impact Explanation

The solvency invariant — that the contract's token balances cover all collateral obligations — is violated for the fee-on-transfer collateral token. If the borrower defaults, lenders absorb bad debt equal to `fee * lltv` per unit of fee-on-transfer collateral supplied. The discrepancy is permanent and compounds with each `supplyCollateral` call on the same index. This constitutes direct, concrete theft of lender funds via protocol accounting manipulation.

## Likelihood Explanation

**Preconditions:** A fee-on-transfer ERC20 exists (USDT historically had this; many DeFi tokens implement it). The attacker creates a market using such a token as collateral — fully permissionless, no admin approval required. **Feasibility:** High. The exploit is deterministic and repeatable. No oracle manipulation, no governance action, no flash loan, and no victim mistake is required. The attacker controls both market creation and collateral supply amounts.

## Recommendation

Replace the direct `assets` write with a balance-delta pattern: record the contract's token balance before and after the transfer, and store only the actual received amount:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as collateral by adding a validation check in `touchMarket` (though this is harder to enforce on-chain without a transfer probe). The balance-delta approach is the most robust fix.

## Proof of Concept

**Minimal Foundry test plan:**

1. Deploy a mock ERC20 that deducts a 1% fee on every `transferFrom`.
2. Deploy `Midnight` and call `touchMarket` with the fee token as the sole collateral param (valid LLTV, valid `maxLif`, valid maturity).
3. Approve and call `supplyCollateral(market, 0, 1000e18, attacker)`.
4. Assert `midnight.collateral(id, attacker, 0) == 1000e18` (stored) but `feeToken.balanceOf(address(midnight)) == 990e18` (actual).
5. Supply loan tokens from a lender via a buy offer and `take` a sell offer to borrow up to `maxDebt = 1000e18 * price / ORACLE_PRICE_SCALE * lltv / WAD`.
