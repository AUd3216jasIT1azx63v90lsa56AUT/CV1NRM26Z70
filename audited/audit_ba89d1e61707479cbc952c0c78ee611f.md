### Title
Stale `stake()` Transaction Executes After Registry Transitions to `UNDER_ATTACK`, Permanently Locking Staker With No Exit and Near-Zero Bonus - (File: src/ConfidencePool.sol)

### Summary
`stake()` accepts no caller-supplied deadline. A transaction submitted while the registry is in a pre-risk state (where `withdraw()` is still available) can sit in the mempool and execute after the registry has transitioned to `UNDER_ATTACK`. At that point `_observePoolState()` seals `riskWindowStart`, `withdraw()` is permanently disabled, and the staker's k=2 bonus is crushed to near-zero — all without the staker ever intending to commit under active-risk conditions.

### Finding Description
`stake(uint256 amount)` contains no deadline guard. [1](#0-0) 

When a staker submits the transaction while the registry is in `NOT_DEPLOYED`, `NEW_DEPLOYMENT`, or `ATTACK_REQUESTED` (pre-risk states where `withdraw()` is open), the transaction may remain pending in the mempool. If the registry transitions to `UNDER_ATTACK` before the transaction is mined, the call to `_observePoolState()` on line 227 observes the active-risk state and seals `riskWindowStart` via `_markRiskWindowStart()`. [2](#0-1) 

Because `UNDER_ATTACK` is intentionally not blocked by `_assertDepositsAllowed`, the stake proceeds. The staker's entry time is then floored at `riskWindowStart` (≈ `block.timestamp`): [3](#0-2) 

This collapses `(T − entryTime)²` to near-zero, yielding a negligible k=2 bonus share. Simultaneously, `withdraw()` is now permanently disabled because `riskWindowStart != 0`: [4](#0-3) 

The staker is locked in with no exit, having submitted the transaction under entirely different conditions than those that executed it.

`DESIGN.md §3` states "A staker who does not want this can read the live registry state before staking," but this only addresses a staker who *knowingly* stakes during `UNDER_ATTACK`. It does not address the mempool-delay scenario where the registry state changes *after* submission but *before* execution — the staker read the correct pre-risk state and acted on it, yet the transaction executed under active-risk conditions they never accepted.

### Impact Explanation
- The staker permanently loses the right to `withdraw()`, which was available at submission time.
- The staker earns near-zero k=2 bonus (entry floored at `riskWindowStart ≈ now`, so `(T − entry)² ≈ 0`).
- If the moderator subsequently flags `CORRUPTED` (bad-faith), the staker's principal is swept in full to `recoveryAddress` — a direct, unrecoverable financial loss on capital the staker never intended to expose to active-risk.

### Likelihood Explanation
No adversarial actor is required: ordinary network congestion or a low-gas-price submission is sufficient. A malicious validator or sequencer can also deliberately withhold the transaction until the registry transitions. The registry can move to `UNDER_ATTACK` at any time during the pool's lifetime, and the window between a staker's submission and mining can span minutes to hours.

### Recommendation
Add a `deadline` parameter to `stake()` (and symmetrically to `contributeBonus()`) and revert if the block timestamp exceeds it:

```solidity
function stake(uint256 amount, uint256 deadline) external nonReentrant whenPoolNotPaused {
    if (block.timestamp > deadline) revert DeadlineExpired();
    // ... existing checks
}
```

This gives stakers a trustless guarantee that their transaction executes only while the pool is in the state they observed at submission time, consistent with the pattern used by Uniswap and other time-sensitive DeFi protocols.

### Proof of Concept
1. Registry is in `NEW_DEPLOYMENT`. Alice calls `stake(1000e18)` with a low gas price; the transaction enters the mempool.
2. Before Alice's transaction is mined, the registry transitions to `UNDER_ATTACK`.
3. Alice's transaction is eventually included. `_observePoolState()` reads `UNDER_ATTACK` and calls `_markRiskWindowStart()`, sealing `riskWindowStart = block.timestamp`.
4. `_assertDepositsAllowed` passes (UNDER_ATTACK is not blocked by design). Alice's stake is recorded with `newEntry = riskWindowStart`.
5. Alice immediately tries `withdraw()` — it reverts `WithdrawsDisabled` because `riskWindowStart != 0`. She is permanently locked.
6. Her k=2 score is `amount × (T − riskWindowStart)²`, where `T − riskWindowStart` is the tiny interval between the attack start and resolution — effectively zero bonus.
7. The moderator later flags `CORRUPTED` (bad-faith). `claimCorrupted()` sweeps Alice's 1000e18 to `recoveryAddress`. Alice loses her entire principal despite having submitted her transaction under safe, pre-risk conditions.
