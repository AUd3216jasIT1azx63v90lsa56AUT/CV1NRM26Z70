Audit Report

## Title
Authorized-Party Nonce Burn via Delegated Signature Invalidates Authorizer's Pending Off-Chain Authorizations — (`src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` permits any party already authorized by `authorization.authorizer` in Midnight to construct and sign their own `Authorization` struct naming the victim as `authorizer`, consuming the victim's current nonce. Because the nonce is incremented unconditionally before the signature check and the delegation branch of the `Unauthorized` guard does not distinguish between "the authorizer signing their own nonce" and "a delegate consuming the authorizer's nonce on a struct the delegate constructed themselves," an authorized party can permanently invalidate any pending off-chain authorization the victim has already distributed, and repeat this indefinitely.

## Finding Description

**Code path** — `src/periphery/EcrecoverAuthorizer.sol` lines 24–48:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // nonce burned here

    bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
    bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
    bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
    address signer = ecrecover(digest, signature.v, signature.r, signature.s);
    require(signer != address(0), InvalidSignature());
    require(
        signer == authorization.authorizer
            || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer), // delegation branch
        Unauthorized()
    );
    ...
    IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
}
```

**Root cause** — The nonce at line 26 is keyed to `authorization.authorizer` (victim A) and is post-incremented unconditionally before any signature or authorization check. The `Unauthorized` guard at lines 33–36 accepts any signer for whom `isAuthorized[authorization.authorizer][signer] == true`. There is no requirement that the signer is the same party who produced the `Authorization` struct, nor that the struct was produced by or on behalf of the authorizer in any meaningful sense.

**Exploit flow:**

| Step | Action |
|------|--------|
| 0 | A authorizes `ecrecoverAuthorizer` and ATTACKER in Midnight (normal protocol setup). |
| 1 | A signs off-chain: `Authorization{authorizer=A, authorized=LEGITIMATE_OPERATOR, isAuthorized=true, nonce=N}` and distributes it to `LEGITIMATE_OPERATOR`. |
| 2 | ATTACKER independently constructs `Authorization{authorizer=A, authorized=ATTACKER, isAuthorized=true, nonce=N}` and signs it with ATTACKER's own private key. |
| 3 | ATTACKER calls `ecrecoverAuthorizer.setIsAuthorized(authAttacker, sigAttacker)`. |
| 4 | Line 26: `nonce[A]` advances N → N+1. `ecrecover` returns ATTACKER's address. `isAuthorized[A][ATTACKER] == true` → delegation branch passes. Call succeeds. `Midnight.setIsAuthorized(ATTACKER, true, A)` is called — a no-op since ATTACKER was already authorized. |
| 5 | `LEGITIMATE_OPERATOR` submits A's signed authorization → reverts `InvalidNonce` because `nonce[A]` is now N+1. |

**Why existing checks fail** — The `Unauthorized` check is satisfied by the delegation branch; it does not distinguish between "the authorizer is signing their own nonce" and "a delegate is consuming the authorizer's nonce on a struct the delegate constructed themselves." There is no guard preventing an authorized party from targeting `authorization.authorizer = A` with a self-constructed struct at a nonce they did not generate.

## Impact Explanation
A's off-chain signed authorization with nonce=N is permanently invalidated. `LEGITIMATE_OPERATOR` cannot submit it; A must re-sign with nonce=N+1 and redistribute. ATTACKER can repeat this indefinitely (each time submitting a no-op self-authorization `isAuthorized=true` for themselves at the current nonce), making it impossible for A to successfully delegate to any third party through `EcrecoverAuthorizer` as long as ATTACKER remains authorized. This constitutes a persistent, repeatable DoS on A's off-chain authorization mechanism — a permanent freeze of a user's ability to use the `EcrecoverAuthorizer` peripheral contract for delegation.

## Likelihood Explanation
**Preconditions:**
1. A has authorized `ecrecoverAuthorizer` in Midnight — required for any `EcrecoverAuthorizer` user.
2. A has authorized ATTACKER in Midnight at any prior point (e.g., a market maker, keeper, or liquidation bot).
3. A has a pending off-chain signed authorization that ATTACKER knows about (e.g., via mempool observation or public distribution).

All three are realistic for any active protocol user. `Midnight.sol` lines 101–110 document that "authorized accounts can authorize other accounts on behalf of the user," meaning users who authorize counterparties for position management do not expect those counterparties to also gain nonce-burning capability in `EcrecoverAuthorizer`. The attack is repeatable with zero cost beyond gas and requires no privileged access — only the attacker being a previously authorized party of the victim.

## Recommendation
Restrict the delegation branch so that only the `authorization.authorizer` themselves can sign an `Authorization` struct that names themselves as `authorizer`. Specifically, remove the delegation branch from `EcrecoverAuthorizer.setIsAuthorized` entirely, requiring `signer == authorization.authorizer`. If delegation is desired, it should be implemented by requiring the delegate to present a separate proof that the authorizer explicitly delegated nonce-signing authority (e.g., a separate on-chain flag or a countersignature from the authorizer), rather than reusing the general `isAuthorized` mapping which grants broad protocol-level permissions.

Alternatively, key the nonce to `msg.sender` or to the signer's address rather than to `authorization.authorizer`, so that a delegate consuming a nonce does not affect the authorizer's nonce space.
