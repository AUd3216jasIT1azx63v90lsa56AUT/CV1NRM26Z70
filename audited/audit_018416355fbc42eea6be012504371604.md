Audit Report

## Title
Off-by-one `<=` in maturity debt-increase guard allows forced debt creation at exact maturity timestamp - (File: `src/Midnight.sol`)

## Summary
The `take()` function at line 391 uses `block.timestamp <= offer.market.maturity` to guard against post-maturity debt increases, which permits `sellerDebtIncrease > 0` when `block.timestamp == offer.market.maturity`. The `liquidate()` function at line 622 uses the strict `block.timestamp > market.maturity` for post-maturity mode, creating a one-second window where debt is forced onto a seller who cannot yet be post-maturity liquidated. In the immediately following block, the position becomes liquidatable via post-maturity mode despite being collateral-healthy, causing the seller to lose collateral to a liquidator.

## Finding Description

**Root cause — asymmetric boundary operators:**

`take()`, line 391:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
When `block.timestamp == offer.market.maturity`, the left operand is `true`, so the `require` passes unconditionally regardless of `sellerDebtIncrease`. Debt is then written to storage at line 414:
```solidity
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
```

The end-of-take health check at line 476:
```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```
`isHealthy` (lines 944–960) returns `maxDebt >= debt` — a purely collateral-coverage check with no maturity component. A collateral-healthy seller passes this check even after debt is forced onto them at maturity.

`liquidate()`, lines 620–624, uses strict `>`:
```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```
At `block.timestamp == maturity` this evaluates to `false` for `postMaturityMode=true`, so `NotLiquidatable()` reverts. At `block.timestamp == maturity + 1` it evaluates to `true`.

This asymmetry is confirmed by the existing test at `test/LiquidationTest.sol` lines 170–177, which explicitly documents that post-maturity liquidation is unavailable at exact maturity but available one second later. No test covers the case of debt increase at exact maturity.

**Why existing checks fail:**
- `CannotIncreaseDebtPostMaturity` uses `<=` instead of `<`, creating a one-second window at the exact maturity timestamp.
- The `SellerIsLiquidatable` check at line 476 only enforces collateral health, not maturity-based liquidatability.
- The protocol's own invariant document (`live_context.json`, line 221) explicitly lists this as forbidden: `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"`.
- The `credit_debt` invariant (`live_context.json`, line 197) states `"debt must not increase after maturity"`. The maturity timestamp itself violates this invariant under the `<=` guard.

**Exploit flow:**
1. A sell offer exists with `expiry >= maturity` and the seller has no existing credit to absorb units (so `sellerDebtIncrease > 0`).
2. Attacker (taker) waits until `block.timestamp == offer.market.maturity`.
3. Taker calls `take()` with `units` such that `sellerDebtIncrease > 0`.
4. Line 391: `block.timestamp <= maturity` → `true` → passes. Seller's debt increases. Health check at line 476 passes (seller is collateral-healthy).
5. In the next block (`block.timestamp == maturity + 1`), taker calls `liquidate(..., postMaturityMode=true)`.
6. Line 622: `block.timestamp > market.maturity` → `true` → liquidation proceeds on a collateral-healthy position.
7. LIF at `maturity + 1` = `WAD + (maxLif - WAD) * 1 / TIME_TO_MAX_LIF` (just above 1, ramping over 15 minutes per line 646).

## Impact Explanation

A taker can force a collateral-healthy borrower to take on new debt at the exact maturity second. In the immediately following block, the borrower's position is liquidatable via post-maturity mode even though it is collateral-healthy (`live_context.json` line 88: `"overdue_debt_after_maturity_is_liquidatable_even_if_healthy": true`). The borrower has no opportunity to repay between debt creation and liquidation (consecutive blocks). The liquidator seizes collateral at LIF > 1 (even at `maturity + 1`, LIF = `WAD + (maxLif - WAD)/900`), causing direct loss of collateral value beyond the debt repaid. This constitutes unauthorized collateral seizure from a healthy account and matches the `liquidation` invariant in `live_context.json` line 213: `"healthy positions cannot be health-liquidated before maturity"` — and by extension, a healthy position that was never legitimately overdue should not be liquidatable post-maturity either.

## Likelihood Explanation

**Preconditions:**
- A sell offer with `expiry >= maturity` exists (common for offers meant to be filled up to maturity).
- The seller has no existing credit to absorb the units (`sellerCreditDecrease < units`).
- The seller has sufficient collateral to pass the health check after the debt increase.

**Feasibility:** On chains with 1-second block times (post-merge Ethereum, Arbitrum, Base), landing a transaction at exactly `block.timestamp == maturity` is achievable via MEV or simple block monitoring. The attacker (taker) controls the timing. The attack is repeatable for any market where a qualifying sell offer exists. No privileged access is required — the attacker is a normal unprivileged taker.

## Recommendation

Change the boundary operator in `take()` at line 391 from `<=` to `<`:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the debt-increase guard with the strict `>` used in `liquidate()`, closing the one-second window at the exact maturity timestamp. The fix is consistent with the protocol invariant at `live_context.json` line 221 and the `credit_debt` invariant at line 197.

## Proof of Concept

```solidity
// Minimal Foundry test sketch
function testForcedDebtAtMaturity() public {
    // 1. Setup: seller has no credit, has collateral, sell offer with expiry == maturity
    uint256 maturity = market.maturity;
    vm.warp(maturity); // land exactly at maturity

    // 2. Taker calls take() — sellerDebtIncrease > 0 passes the <= guard
    midnight.take(offer, ratifierData, units, taker, receiver, address(0), "");

    // 3. Verify seller has debt
    assertGt(midnight.debt(id, seller), 0);

    // 4. Advance one block
    vm.warp(maturity + 1);

    // 5. Liquidate collateral-healthy seller in post-maturity mode — succeeds
    midnight.liquidate(market, 0, seizedAssets, collateralIndex, seller, true, address(this), address(0), "");

    // 6. Verify seller lost collateral despite being collateral-healthy
    assertLt(midnight.collateral(id, seller, collateralIndex), initialCollateral);
}
```
