Audit Report

## Title
Unvalidated Non-Contract `liquidatorGate` Permanently Bricks Liquidations - (File: src/Midnight.sol)

## Summary

`touchMarket` accepts any `address` for `market.liquidatorGate` without validating it is `address(0)` or a deployed contract. An unprivileged attacker can create a market with `liquidatorGate` set to a codeless address. Every subsequent `liquidate()` call unconditionally reverts because Solidity 0.8.34's ABI decoder expects ≥ 32 bytes for the `bool` return of `ILiquidatorGate.canLiquidate` but receives 0 bytes from the codeless target, permanently disabling liquidations and enabling unbounded bad-debt accumulation at lenders' expense.

## Finding Description

**Root cause — no `liquidatorGate` validation in `touchMarket`:**

`touchMarket` (lines 755–791 of `src/Midnight.sol`) validates `maturity`, collateral count, token ordering, `lltv`, and `maxLif`, but performs zero validation on `market.liquidatorGate`. The `Market` struct accepts any `address` for `liquidatorGate` with no constraint (`src/interfaces/IMidnight.sol`, lines 5–12).

**Trigger — `liquidate()` calls the interface unconditionally on the non-contract address:**

```solidity
require(
    market.liquidatorGate == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
```
(`src/Midnight.sol`, lines 597–600)

When `liquidatorGate = address(0xdead)`:
1. `market.liquidatorGate == address(0)` → `false`; short-circuit does not fire.
2. `ILiquidatorGate(address(0xdead)).canLiquidate(msg.sender)` is evaluated.
3. The EVM CALL to a codeless address returns `(success=true, returndata=<empty>)`.
4. Solidity 0.8.34's ABI decoder expects ≥ 32 bytes for the `bool` return declared in `ILiquidatorGate.canLiquidate`; it receives 0 bytes → unconditional revert (ABI decoding error).
5. The `require` guard cannot catch this: a revert thrown *inside* the condition expression propagates unconditionally.

**Why the LIVENESS comment is insufficient:**

Line 152 states: *"If `liquidatorGate.canLiquidate` reverts or returns false, `liquidate` reverts."* This documents the behavior of a legitimate gate that may transiently revert (e.g., during a pause). It does not constitute acceptance of an adversary-created market with a permanently broken non-contract gate. No `extcodesize` or equivalent guard exists anywhere in the creation or liquidation path.

**`liquidatorGate` is immutable once set:** The market is stored via `IdLib.storeInCode` using CREATE2 (`src/libraries/IdLib.sol`, lines 35–41). There is no mechanism to update `liquidatorGate` post-creation.

**Exploit flow:**
1. Attacker calls `touchMarket(market)` with `market.liquidatorGate = address(0xdead)` and otherwise valid parameters. Market is created successfully.
2. Attacker seeds the market with competitive offers; lenders and borrowers interact normally (`take`, `repay`, `supplyCollateral` are unaffected).
3. Borrower's position becomes unhealthy (e.g., oracle price drop).
4. Any call to `liquidate(market, ...)` hits the `canLiquidate` call → ABI-decode revert → liquidation permanently impossible.
5. Bad debt accumulates; lenders cannot recover funds.

## Impact Explanation

Unhealthy borrowers in any market created with a non-contract, non-zero `liquidatorGate` can never be liquidated. Bad debt accumulates without bound. Lenders suffer permanent, unrecoverable fund loss. The core protocol invariant — "unhealthy positions remain liquidatable" — is permanently violated for the affected market. This constitutes direct, concrete loss of lender funds: a permanent freeze/loss of user assets.

## Likelihood Explanation

- **Privilege required:** None. `touchMarket` is public and permissionless.
- **Feasibility:** Trivial. One transaction suffices to create the malformed market.
- **Repeatability:** The market ID is deterministic from its parameters; once created, `liquidatorGate` is immutable (encoded via `IdLib.storeInCode`). The DoS is permanent.
- **Victim attraction:** The attacker can seed the market with competitive offers to draw in lenders and borrowers before positions become unhealthy.

## Recommendation

In `touchMarket`, validate that `liquidatorGate` is either `address(0)` or a deployed contract with code:

```solidity
require(
    market.liquidatorGate == address(0) || market.liquidatorGate.code.length > 0,
    InvalidLiquidatorGate()
);
```

This check should be placed alongside the existing collateral parameter validations in the market creation block. Additionally, consider applying the same guard to `enterGate`.

## Proof of Concept

**Minimal manual steps:**

1. Deploy `Midnight` on a local fork.
2. Call `touchMarket(market)` where `market.liquidatorGate = address(0xdead)` and all other parameters are valid (valid loanToken, collateralParams with allowed lltv/maxLif, maturity within range).
3. Have a lender and borrower interact via `take` to establish a borrow position.
4. Manipulate the oracle to make the borrower unhealthy.
5. Call `liquidate(market, ...)` from any address.
6. Observe unconditional revert (ABI decoding error, not `LiquidatorGatedFromLiquidating`).
7. Confirm no path exists to liquidate the position or recover lender funds.
