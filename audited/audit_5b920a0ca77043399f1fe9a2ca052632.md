### Title
Zero-asset take bypasses maxAssets cap on fully-consumed assets-mode buy offers with price < WAD - (File: src/Midnight.sol)

### Summary
When `offer.maxAssets > 0` and `offer.buy = true`, the consumed counter is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. If `buyerPrice < WAD` (i.e., `offerPrice < WAD`), a taker can choose `units` small enough that `buyerAssets` rounds down to zero, leaving `consumed[maker][group]` unchanged and passing the `require(newConsumed <= offer.maxAssets)` check even when the offer is already fully consumed. The take then proceeds with `units > 0`, causing real credit, debt, and `totalUnits` state changes with zero asset transfer.

### Finding Description
**Exact code path** — `src/Midnight.sol` lines 367–369:

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

**Root cause** — `buyerAssets` is computed as `units.mulDivDown(buyerPrice, WAD)` (line 363). When `buyerPrice < WAD` (achievable whenever `offerPrice < WAD`, i.e., any tick below `WAD`), a taker can supply `units` in the range `[1, WAD/buyerPrice - 1]` such that `mulDivDown` truncates to zero. The consumed delta added is therefore 0, `newConsumed` equals the pre-call value, and `require(newConsumed <= offer.maxAssets)` passes trivially even when `consumed == maxAssets`.

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = X` and `tick` such that `buyerPrice < WAD`.
2. Offer is legitimately filled until `consumed[maker][group] = X` (fully consumed).
3. Attacker (any taker) calls `take(offer, ..., units_small, ...)` where `units_small.mulDivDown(buyerPrice, WAD) == 0`.
4. Line 368: `consumed[maker][group] += 0` → still `X`.
5. Line 369: `require(X <= X)` → passes.
6. Execution continues: `buyerCreditIncrease`, `sellerDebtIncrease`, and `totalUnits` are updated using the non-zero `units_small` value (lines 382–384, 416–417).
7. Token transfers at lines 455–456 move `buyerAssets - sellerAssets = 0` tokens — no payment occurs.

**Why existing checks fail** — The only guard against over-consumption is `require(newConsumed <= offer.maxAssets)`. Because the assets-mode branch increments by `buyerAssets` (not `units`), and `buyerAssets` can be zero for non-zero `units` when `buyerPrice < WAD`, the check is ineffective. The units-mode branch (line 371) increments by `units` directly and cannot be bypassed this way. The protocol comment at line 94 acknowledges this gap but does not fix it.

**Confirmed by existing test** — `testBugBuyMaxAssetsBypass()` (test/TakeTest.sol lines 857–889) reproduces the exact scenario: pre-consumes the offer to `maxAssets = 1`, calls `take(1, borrower, lenderOffer)` with `tick = MAX_TICK - 16` (price < WAD), and asserts `buyerAssets == 0` while `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase.

### Impact Explanation
A fully-consumed assets-mode buy offer with `offerPrice < WAD` can be taken an unbounded number of times after full consumption. Each such take:
- Increases the maker's credit and the taker's debt by `units_small` with zero asset payment.
- Increases `totalUnits` and `claimableSettlementFee` (line 418: `buyerAssets - sellerAssets = 0`, so no fee accrual, but `totalUnits` grows).
- Allows a taker to force the maker into a larger credit position than the maker's `maxAssets` cap was intended to permit, violating the "offers cannot be overfilled" invariant.
- Allows a taker to take on unbounded debt at zero cost (self-harm, but exploitable in complex multi-step scenarios involving liquidation or bad-debt socialization).

### Likelihood Explanation
**Preconditions:**
1. `offer.buy = true` and `offer.maxAssets > 0` (assets-mode buy offer).
2. `offerPrice < WAD` — any tick strictly below the WAD tick satisfies this; the tick range is wide.
3. `consumed[maker][group] >= offer.maxAssets` — offer must be fully consumed; achievable by legitimate fills or by the maker calling `setConsumed`.

**Feasibility:** All preconditions are attacker-controllable or reachable through normal protocol use. The taker needs only to compute a `units` value in `[1, floor(WAD/buyerPrice) - 1]`. No privileged access, no oracle manipulation, no token owner action required. Repeatable indefinitely on the same offer.

### Recommendation
Add an explicit guard in the assets-mode branch that rejects non-zero `units` when `buyerAssets == 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0 || units == 0, ZeroAssetNonZeroUnits()); // new check
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, mirror the units-mode guard: if `consumed[maker][group] >= offer.maxAssets`, require `units == 0` before computing `buyerAssets`. This aligns with the Certora rule `fullyConsumedOfferRevertsOnNonTrivialTake` that already exists for units-mode but is intentionally absent for assets-mode.

### Proof of Concept
The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already constitutes a complete PoC. A minimal Foundry unit test plan:

```solidity
function testZeroAssetTakeOnFullyConsumedAssetsBuyOffer() public {
    // Setup: buy offer, maxAssets=1, tick such that buyerPrice < WAD
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    // Pre-consume fully
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, 1, lender);
    assertEq(midnight.consumed(lender, lenderOffer.group), 1); // == maxAssets

    uint256 creditBefore = midnight.creditOf(id, lender);
    uint256 debtBefore   = midnight.debtOf(id, borrower);

    // Take with units=1 → buyerAssets rounds to 0
    collateralize(market, borrower, 1);
    (uint256 ba, uint256 sa) = take(1, borrower, lenderOffer);

    // Assertions proving the bypass:
    assertEq(ba, 0);  // no assets paid
    assertEq(sa, 0);  // no assets received
    assertEq(midnight.consumed(lender, lenderOffer.group), 1); // cap unchanged
    assertGt(midnight.creditOf(id, lender),  creditBefore);    // FAIL: credit grew past cap
    assertGt(midnight.debtOf(id, borrower),  debtBefore);      // FAIL: debt grew past cap
}
```

/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
