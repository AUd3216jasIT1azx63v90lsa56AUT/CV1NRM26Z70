### Title
Zero-`buyerAssets` rounding allows unlimited free takes on assets-mode buy offers at sub-WAD price — (`src/Midnight.sol`)

### Summary
When a buy offer uses `maxAssets` mode and `buyerPrice` is small enough that `units * buyerPrice < WAD`, `mulDivDown` rounds `buyerAssets` to zero. The `consumed` mapping is incremented by zero, so the cap check `newConsumed <= offer.maxAssets` always passes — even on a fully-consumed offer — while position state (`credit`, `debt`, `totalUnits`) is updated using the raw `units` value and the maker's callback is invoked. No loan-token assets are transferred, yet the lender receives credit, violating the solvency invariant that every credit unit has a matching asset deposit.

### Finding Description

**Code path** — `src/Midnight.sol`, `take()`:

```
// line 363
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // ← rounds to 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

// lines 367-369

```

Position state is then updated with the raw `units` (not `buyerAssets`):

```
// lines 382, 408-410

buyerPos.debt  -= toUint128(units - buyerCreditIncrease);
buyerPos.credit += toUint128(buyerCreditIncrease);          // ← increases by units
```

Token transfer at line 455 moves `buyerAssets - sellerAssets = 0 - 0 = 0` tokens.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`
- `offer.tick` chosen so `tickToPrice(tick)` is small (e.g. `MAX_TICK - 16`, giving `offerPrice < WAD`)
- `units = 1` (or any value where `units * buyerPrice < WAD`)

**Preconditions:** The taker must be able to pass the ratifier check (e.g. the maker uses a permissive ratifier or a Merkle-tree ratifier that includes the offer). No privileged access is required.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — passes because `+= 0` never advances `newConsumed`.
- The Certora `Consume.spec` rule `fullyConsumedOfferRevertsOnNonTrivialTake` only covers `maxAssets == 0` (units mode); the assets-mode analogue is absent.
- The `Consume.spec` summaries use `NONDET` for `mulDivDown`/`mulDivUp`, so the prover never sees the zero-rounding case.
- The NatDev comment at line 94 explicitly acknowledges the gap: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) is a developer-written proof of the exact scenario: it pre-fills `consumed` to `maxAssets`, calls `take(1, borrower, lenderOffer)`, and asserts `buyerAssets == 0`, `consumed` unchanged, yet `lender.credit`, `borrower.debt`, and `totalUnits` all strictly increase.

### Impact Explanation

Each zero-asset take:
1. Advances the lender's `credit` by `units` without any loan-token deposit — the contract's balance no longer covers all credit redemptions (solvency invariant broken).
2. Advances the borrower's `debt` by `units` without the borrower receiving any assets.
3. Invokes the maker's `onBuy` callback with `buyerAssets = 0`, which may trigger unintended maker-side logic.
4. Never advances `consumed`, so the offer's `maxAssets` budget is never depleted — the attack is repeatable an unbounded number of times on the same offer.

### Likelihood Explanation

**Preconditions:** The offer must use `maxAssets` mode and be placed at a tick where `tickToPrice(tick) < WAD` (any tick below `MAX_TICK` satisfies this for small enough `units`). The minimum tick spacing is 1, so ticks near 0 (very low price) are reachable. The taker only needs to pass the ratifier — for common ratifiers (Merkle-tree, signature-based) this is a normal user action. The attack is repeatable in a single transaction via `multicall`.

### Recommendation

Add a guard that rejects a non-zero `units` take when the resulting tracked-asset delta is zero in assets mode:

```solidity
if (offer.maxAssets > 0) {
    uint256 trackedAssets = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || trackedAssets > 0, ZeroAssetTake());
    newConsumed = consumed[offer.maker][offer.group] += trackedAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` such that `units * buyerPrice >= WAD` when `maxAssets > 0` and `offer.buy == true`.

### Proof of Concept

```solidity
// Foundry unit test (extends TakeTest setup)
function testZeroAssetTakeBypassConsumed() public {
    // Buy offer, assets mode, low tick so tickToPrice < WAD
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 100;
    lenderOffer.tick = 1; // tickToPrice(1) is near-zero

    uint256 consumedBefore = midnight.consumed(lender, lenderOffer.group);
    uint256 creditBefore   = midnight.creditOf(id, lender);
    uint256 debtBefore     = midnight.debtOf(id, borrower);

    collateralize(market, borrower, 1);
    (uint256 buyerAssets,) = take(1, borrower, lenderOffer);

    // Root cause: buyerAssets rounds to 0
    assertEq(buyerAssets, 0, "buyerAssets must be 0");
    // Invariant violated: consumed did not advance despite non-zero units
    assertEq(midnight.consumed(lender, lenderOffer.group), consumedBefore,
        "consumed must not change");
    // Impact: positions changed without asset transfer
    assertGt(midnight.creditOf(id, lender),   creditBefore, "lender credit increased");
    assertGt(midnight.debtOf(id, borrower),   debtBefore,   "borrower debt increased");

    // Repeatability: take again — consumed still does not advance
    collateralize(market, borrower, 1);
    take(1, borrower, lenderOffer);
    assertEq(midnight.consumed(lender, lenderOffer.group), consumedBefore,
        "consumed still unchanged after second take");
}
```

/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
