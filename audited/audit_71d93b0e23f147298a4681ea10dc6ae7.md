Audit Report

## Title
Authorized caller can redirect all sell-side loan proceeds to an arbitrary address while debt accrues to the taker - (File: src/periphery/MidnightBundles.sol)

## Summary

`supplyCollateralAndSellWithUnitsTarget` and `supplyCollateralAndSellWithAssetsTarget` accept a caller-controlled `receiver` parameter that is never validated against `taker`. The authorization gate at line 127 only verifies the caller may act on the taker's behalf; it does not restrict where the resulting loan proceeds are sent. An attacker who holds `isAuthorized[victim][attacker] == true` can borrow against the victim's collateral and route 100% of the loan proceeds to themselves, leaving the victim with increased debt and no assets.

## Finding Description

**Exact code path:**

```solidity

```
Attacker passes `taker = victim`, `msg.sender = attacker`. Precondition `isAuthorized[victim][attacker] == true` satisfies this check. The `receiver` parameter (line 121) is accepted from the caller with no constraint that it equals `taker`.

**Step 8 ŌĆö Bundler forwards proceeds to attacker-controlled `receiver`** (line 168):
```solidity

```
`receiver` is the raw function parameter set by the attacker to their own address. No check constrains it to `taker`.

**Root cause:** The `receiver` parameter is never validated against `taker`. The `isAuthorized` gate grants the right to *initiate* actions on behalf of the taker; it does not grant the right to *redirect the taker's loan proceeds* to an arbitrary third party. The identical flaw exists in `supplyCollateralAndSellWithAssetsTarget` at line 307:
```solidity
SafeTransferLib.safeTransfer(loanToken, receiver, targetSellerAssets);
```

**Why existing checks fail:** The authorization check at line 127 is a binary gate ŌĆö it either passes or reverts. Once it passes, no subsequent check in the bundler constrains the `receiver` address. The health check in `Midnight.take` (line 476) only verifies the victim's collateral covers their new debt; it does not verify that proceeds went to the correct party.

## Impact Explanation

The victim's debt increases by the taken units and their collateral remains locked as backing. All loan proceeds (`filledSellerAssets`) are transferred to the attacker. This constitutes unauthorized borrowing with complete redirection of loan proceeds ŌĆö the victim is left with debt and no assets. This is a direct theft of value from the victim, enabled by a protocol-level contract (the bundler) that fails to enforce that proceeds belong to the taker.

## Likelihood Explanation

**Required preconditions:**
1. `isAuthorized[victim][bundler] == true` ŌĆö this is the normal operating precondition for any user of the bundler; without it, the bundler is unusable.
2. `isAuthorized[victim][attacker] == true` ŌĆö the victim must have authorized the attacker on Midnight. This is realistic: users commonly authorize portfolio manager contracts, keeper bots, or trusted EOAs for position management. The Midnight authorization model is explicitly designed to support this pattern (see `Midnight.sol` lines 101ŌĆō110).
3. Victim has sufficient pre-existing collateral in Midnight for the health check to pass.

All three conditions are realistic in normal protocol usage. The attack is repeatable as long as the authorization remains and collateral is present. The attacker does not need to supply any capital.

## Recommendation

In both `supplyCollateralAndSellWithUnitsTarget` and `supplyCollateralAndSellWithAssetsTarget`, enforce that when `msg.sender != taker`, the `receiver` must equal `taker`:

```solidity
if (msg.sender != taker) require(receiver == taker, InvalidReceiver());
```

Alternatively, remove the `receiver` parameter entirely from these functions and always use `taker` as the destination, since the taker is the party whose debt is being increased and who should receive the proceeds.
