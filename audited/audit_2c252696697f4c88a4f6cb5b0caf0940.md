Audit Report

## Title
Unprivileged liquidator can designate arbitrary third-party contract as `callback`/`payer`, draining its loanToken approval to Midnight - (File: src/Midnight.sol)

## Summary
`Midnight.liquidate` sets `payer = callback` at line 679 with no requirement that `callback == msg.sender` or that `callback` has authorized `msg.sender`. An attacker can supply any contract that implements `ILiquidateCallback` returning `CALLBACK_SUCCESS` and holds a loanToken approval to Midnight as the `callback`, causing the protocol to pull `repaidUnits` of loanToken from that victim contract while the attacker's chosen `receiver` collects the seized collateral for free.

## Finding Description
**Confirmed code path:**

`src/Midnight.sol` line 679:
```solidity
address payer = callback != address(0) ? callback : msg.sender;
```

`src/Midnight.sol` lines 696–717:
```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets); // L696
if (callback != address(0)) {
    require(
        ILiquidateCallback(callback).onLiquidate(msg.sender, ...) == CALLBACK_SUCCESS, // L700-714
        WrongLiquidateCallbackReturnValue()
    );
}
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits); // L717
```

**Root cause:** The only gate on `callback` is that it must return `CALLBACK_SUCCESS` from `onLiquidate`. The `liquidatorGate` check at lines 597–600 only restricts who can call `liquidate` (i.e., `msg.sender`), not what address is passed as `callback`. No `require(callback == msg.sender || isAuthorized[callback][msg.sender])` exists anywhere in `liquidate`.

**Exploit flow:**
1. `victimContract` implements `ILiquidateCallback`, its `onLiquidate` returns `CALLBACK_SUCCESS` (e.g., only checks `msg.sender == midnight`), and holds `loanToken.approve(midnight, N)`.
2. Attacker calls `midnight.liquidate(market, idx, 0, repaidUnits, borrower, false, attackerEOA, victimContract, data)`.
3. `payer = victimContract` (line 679).
4. `seizedAssets` of collateral transferred to `attackerEOA` (line 696).
5. `victimContract.onLiquidate(attacker, ...)` called — returns `CALLBACK_SUCCESS` (lines 700–714).
6. `safeTransferFrom(loanToken, victimContract, midnight, repaidUnits)` executes (line 717) — pulls from victim.
7. Net: attacker receives collateral worth `repaidUnits × LIF`, pays nothing; victim loses `repaidUnits` of loanToken.

**Existing checks and why they fail:** The `liquidatorGate` check (lines 597–600) only validates `msg.sender`'s permission to call `liquidate`, not the identity or authorization of `callback`. The `CALLBACK_SUCCESS` return value check (lines 699–714) only verifies the callback's return value, not that the callback authorized the caller to use it as payer. No other authorization check on `callback` exists in the function.

## Impact Explanation
Any contract that (a) implements `ILiquidateCallback` returning `CALLBACK_SUCCESS` and (b) holds a loanToken approval to Midnight can have its entire approved balance drained by an unprivileged attacker in a single `liquidate` call. The attacker receives seized collateral at no personal cost. This constitutes direct theft of funds from third-party contracts and violates the core protocol invariant (evidenced by `certora/specs/OnlyExplicitPayerCanLoseTokens.spec`) that only an explicitly consenting payer should lose tokens during liquidation.

## Likelihood Explanation
Preconditions are realistic: an unhealthy borrower is a routine market condition, and deployed liquidation bots or flash-liquidation helpers that pre-approve the lending protocol for loanToken and implement `ILiquidateCallback` with a simple `msg.sender == midnight` guard are common integration patterns. The attacker requires no special privilege, no oracle manipulation, and no governance access. The attack is repeatable as long as the victim's approval persists.

## Recommendation
Add an authorization check on `callback` before using it as `payer`. The fix should require either `callback == msg.sender` or `isAuthorized[callback][msg.sender]`:

```solidity
if (callback != address(0)) {
    require(
        callback == msg.sender || isAuthorized[callback][msg.sender],
        Unauthorized()
    );
}
address payer = callback != address(0) ? callback : msg.sender;
```

This mirrors the authorization pattern already used elsewhere in the protocol (e.g., `setConsumed` at line 724) and ensures that only a callback contract that has explicitly authorized the caller can be designated as payer.

## Proof of Concept
**Minimal Foundry test plan:**
1. Deploy a `VictimCallback` contract that implements `ILiquidateCallback`, returns `CALLBACK_SUCCESS` from `onLiquidate` when `msg.sender == address(midnight)`, and calls `loanToken.approve(address(midnight), type(uint256).max)` in its constructor.
2. Fund `VictimCallback` with loanToken.
3. Create a market, open a borrower position, and make it unhealthy (e.g., via oracle price drop).
4. From an attacker EOA (no special privileges), call:
   ```solidity
   midnight.liquidate(market, idx, 0, repaidUnits, borrower, false, attackerEOA, address(victimCallback), "");
   ```
5. Assert: attacker's collateral balance increased by `seizedAssets`; `VictimCallback`'s loanToken balance decreased by `repaidUnits`; attacker's loanToken balance is unchanged.
