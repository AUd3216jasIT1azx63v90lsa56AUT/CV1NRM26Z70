### Title
Liquidator-controlled `callback` parameter drains arbitrary victim's loan tokens while attacker receives seized collateral - (`src/Midnight.sol`)

### Summary

In `Midnight.liquidate`, the `payer` for loan token repayment is set to the caller-supplied `callback` address with no check that `callback` is `msg.sender` or authorized by `msg.sender`. An attacker can pass a victim contract as `callback`, causing the protocol to pull `repaidUnits` of loan tokens from the victim while the attacker's `receiver` address collects the seized collateral.

### Finding Description

**Exact code path** — `src/Midnight.sol` lines 679–717:

```solidity
// Line 679 — payer is set to callback, fully attacker-controlled
address payer = callback != address(0) ? callback : msg.sender;

// Line 696 — collateral sent to receiver (attacker-controlled)
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);

// Lines 698-714 — callback.onLiquidate() called; if it returns CALLBACK_SUCCESS, execution continues
if (callback != address(0)) {
    require(
        ILiquidateCallback(callback).onLiquidate(msg.sender, id, market, ...) == CALLBACK_SUCCESS,
        WrongLiquidateCallbackReturnValue()
    );
}

// Line 717 — loan tokens pulled from payer = callback = victim
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```

**Root cause**: There is no check that `callback == msg.sender` or `isAuthorized[callback][msg.sender]`. Any address can be supplied as `callback`. The only gate is that `callback.onLiquidate(...)` must return `CALLBACK_SUCCESS`.

**Attacker inputs**:
- `callback = victimContract` (a contract that implements `ILiquidateCallback` and returns `CALLBACK_SUCCESS` without validating `caller`)
- `receiver = attackerAddress` (receives seized collateral)
- `seizedAssets > 0`, `repaidUnits = 0` (input), computed internally from `seizedAssets`

**Exploit flow**:
1. Precondition: `victimContract` has approved Midnight for `≥ repaidUnits` of `market.loanToken`, holds those tokens, and its `onLiquidate` returns `CALLBACK_SUCCESS` unconditionally (or without checking `caller`).
2. Attacker identifies an unhealthy `borrower`.
3. Attacker calls `liquidate(market, collateralIndex, seizedAssets, 0, borrower, false, attackerAddress, victimContract, data)`.
4. `payer = victimContract` (line 679).
5. `seizedAssets` of collateral transferred to `attackerAddress` (line 696).
6. `victimContract.onLiquidate(attacker, ...)` called — returns `CALLBACK_SUCCESS` (line 700–712).
7. `repaidUnits` of loan tokens pulled from `victimContract` (line 717).
8. Attacker nets `seizedAssets` collateral; victim loses `repaidUnits` loan tokens.

**Why existing checks fail**:
- The `liquidatorGate` check (lines 597–600) only verifies `msg.sender` can liquidate; it says nothing about `callback`.
- The `CALLBACK_SUCCESS` return value check (lines 699–714) is the only gate, and it is satisfied by the victim's own implementation.
- The Certora spec `otherEntryPointsOnlyPullFromCaller` (line 119–135 of `certora/specs/OnlyExplicitPayerCanLoseTokens.spec`) explicitly permits pulling from any callback that returned `CALLBACK_SUCCESS` — the spec's invariant is too weak and does not require the callback to be authorized by the liquidator.
- There is no `isAuthorized[callback][msg.sender]` check analogous to the `onBehalf` authorization pattern used elsewhere in the protocol (e.g., `withdrawCollateral` line 556, `repay` line 505). [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

An unprivileged attacker profits from seized collateral at the victim's expense. The victim contract loses `repaidUnits` of loan tokens (transferred to Midnight's `withdrawable` pool, benefiting lenders) while the attacker receives `seizedAssets` of collateral. The attacker's net gain equals the liquidation incentive (LIF spread). This directly violates the invariant that "the liquidator must supply repayment tokens" and the invariant that "callbacks, ERC20 transfers, multicall, or reentrancy cannot corrupt partial state." [4](#0-3) 

### Likelihood Explanation

**Preconditions**:
1. An unhealthy borrower exists (normal market condition).
2. A victim contract implements `ILiquidateCallback` returning `CALLBACK_SUCCESS` without validating the `caller` argument passed by Midnight — realistic for any liquidation bot or protocol integration that uses flash-liquidation callbacks and does not defensively check `caller`.
3. The victim contract has approved Midnight for the loan token — required for any contract that itself calls `liquidate` with a callback, or that calls `repay` with a callback.
4. The victim holds sufficient loan tokens — typical for liquidation bots.

All four preconditions are simultaneously satisfiable in a live deployment. The attack is repeatable as long as the victim's approval and balance remain. [5](#0-4) 

### Recommendation

Add an authorization check on `callback` before using it as `payer`, mirroring the `onBehalf` pattern used elsewhere:

```solidity
require(
    callback == address(0) || callback == msg.sender || isAuthorized[callback][msg.sender],
    CallbackUnauthorized()
);
```

This ensures only the liquidator (or an address the liquidator has been authorized to act on behalf of) can be designated as the payer, while preserving the flash-liquidation use case where the liquidator passes their own contract as `callback`. [6](#0-5) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Market} from "src/Midnight.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";
import {ILiquidateCallback} from "src/interfaces/ICallbacks.sol";

/// @dev Victim: a liquidation bot that holds loan tokens, has approved Midnight,
///      and returns CALLBACK_SUCCESS without checking caller.
contract VictimLiquidationBot is ILiquidateCallback {
    function onLiquidate(
        address, bytes32, Market memory, uint256, uint256, uint256,
        address, address, bytes memory, uint256
    ) external returns (bytes32) {
        return CALLBACK_SUCCESS; // no caller validation
    }
}

contract PayerConfusionTest is Test {
    Midnight midnight;
    VictimLiquidationBot victim;
    address attacker = address(0xBEEF);

    function setUp() public {
        // Deploy Midnight, set up market, create unhealthy borrower position
        // Fund victim with loanToken, victim approves Midnight for max
        // Drop oracle price to make borrower unhealthy
    }

    function testPayerConfusion(uint256 seizedAssets) public {
        seizedAssets = bound(seizedAssets, 1, borrowerCollateral);

        uint256 victimBalanceBefore = loanToken.balanceOf(address(victim));
        uint256 attackerCollateralBefore = collateralToken.balanceOf(attacker);

        vm.prank(attacker);
        (uint256 seized, uint256 repaid) = midnight.liquidate(
            market,
            0,           // collateralIndex
            seizedAssets,
            0,           // repaidUnits (computed from seizedAssets)
            borrower,
            false,
            attacker,           // receiver: attacker gets collateral
            address(victim),    // callback = payer: victim pays loan tokens
            ""
        );

        // Assertions
        assertEq(collateralToken.balanceOf(attacker), attackerCollateralBefore + seized, "attacker received collateral");
        assertEq(loanToken.balanceOf(address(victim)), victimBalanceBefore - repaid, "victim paid loan tokens");
        assertGt(seized, 0, "nonzero collateral seized");
        assertGt(repaid, 0, "nonzero loan tokens drained from victim");
    }
}
```

**Expected assertions**: `attacker` balance of collateral token increases by `seized`; `victim` balance of loan token decreases by `repaid`; attacker spent zero loan tokens. [1](#0-0) [2](#0-1)
