### Title
Dust-unit griefing via `mulDivUp` rounding at minimum tick exhausts `maxAssets` offer budget one wei at a time - (`src/Midnight.sol`)

### Summary
When a sell offer has `maxAssets > 0` and `sellerPrice = 1e12` (the minimum non-zero tick price, `PRICE_ROUNDING_STEP`), calling `take()` with `units = 1` produces `sellerAssets = mulDivUp(1, 1e12, 1e18) = 1`. Because `consumed` is incremented by `sellerAssets` in the `maxAssets` branch, an unprivileged taker can exhaust the entire `maxAssets` budget in exactly `maxAssets` separate calls, each costing only 1 wei of loan token, while the maker's offer is permanently consumed and the maker receives essentially no loan proceeds.

### Finding Description

**Code path:**

In `src/Midnight.sol` lines 364 and 367–369:

```solidity
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

For a sell offer (`offer.buy = false`), `sellerAssets = units.mulDivUp(sellerPrice, WAD)`.

`mulDivUp` is defined in `src/libraries/UtilsLib.sol` line 35 as `(x * y + (d - 1)) / d`.

With `units = 1` and `sellerPrice = 1e12` (the value of `tickToPrice(2)`, confirmed by `test/TickLibTest.sol` line 17: `assertEq(TickLib.tickToPrice(2), 1e12, "first non-zero tick")`):

```
sellerAssets = (1 * 1e12 + (1e18 - 1)) / 1e18
             = (1e12 + 999999999999999999) / 1e18
             = 1000000999999999999 / 1e18
             = 1
```

So `consumed` increments by exactly 1 per call.

**Exploit flow:**

1. Maker (lender) posts a sell offer with `offer.buy = false`, `offer.tick = 2` (price = 1e12), `offer.maxAssets = N` (e.g., 10,000).
2. Attacker (taker) calls `take(offer, ..., units=1, ...)` N times.
3. Each call: `sellerAssets = 1`, `consumed += 1`, passes `require(newConsumed <= maxAssets)`.
4. After N calls: `consumed = N = maxAssets`, offer is fully exhausted.
5. Maker has taken on N units of debt and received only N wei of loan tokens (instead of the intended ~N loan tokens at the offer price).

**Why existing checks fail:**

- There is no `require(units > 0)` or `require(sellerAssets > 0)` guard in `take()`.
- The `ConsumedAssets` check only prevents exceeding `maxAssets`; it does not prevent dust increments.
- The `atMostOneNonZero` check on line 350 only validates that `maxAssets` and `maxUnits` are not both set; it does not enforce a minimum fill size.
- No minimum tick or minimum price constraint is enforced on the offer at take time.

### Impact Explanation
The maker's offer is exhausted by `maxAssets` dust-sized fills. The maker receives `maxAssets` wei of loan tokens (economically zero) while taking on `maxAssets` units of debt. The offer's `consumed` mapping reaches `maxAssets`, permanently blocking any legitimate fill. The maker's intended borrowing is griefed at a cost to the attacker of only `maxAssets` wei of loan token plus gas.

### Likelihood Explanation
Preconditions are fully attacker-controlled: the taker only needs to know the offer parameters (which are passed in calldata and not stored on-chain), set `units = 1`, and call `take()` repeatedly. The minimum tick price of `1e12` is a valid, accessible tick value (`tickToPrice(2) = 1e12`). No special permissions are required; any address can be a taker. The attack is repeatable and deterministic.

### Recommendation
Add a minimum fill size guard in the `maxAssets` branch. One approach: require that `sellerAssets > 0` (or `buyerAssets > 0` for buy offers) before incrementing `consumed`. A stronger fix is to require `units >= some_minimum` or to require that the assets increment is at least `PRICE_ROUNDING_STEP`. Concretely, in the `maxAssets` branch:

```solidity
uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
require(assetsDelta > 0, DustFill());
newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

This ensures that every fill that counts against `maxAssets` transfers at least 1 wei of assets, preventing zero-asset increments. A stronger bound (e.g., `>= PRICE_ROUNDING_STEP`) would prevent sub-rounding-step griefing entirely.

### Proof of Concept

```solidity
// Foundry unit test
function testDustGriefMaxAssets() public {
    uint256 N = 1000;
    // Maker posts sell offer at minimum tick (price = 1e12), maxAssets = N
    borrowerOffer.tick = 2; // tickToPrice(2) == 1e12
    borrowerOffer.maxUnits = 0;
    borrowerOffer.maxAssets = N;

    // Collateralize maker for N units of debt
    collateralize(market, borrower, N);
    deal(address(loanToken), lender, N); // attacker needs only N wei

    // Attacker calls take N times with units=1
    for (uint256 i = 0; i < N; i++) {
        take(1, lender, borrowerOffer);
    }

    // Assert: offer fully exhausted
    assertEq(midnight.consumed(borrowerOffer.maker, borrowerOffer.group), N);
    // Assert: maker received only N wei (not N loan tokens)
    assertEq(loanToken.balanceOf(borrower), N); // N wei, not N * 1e18
    // Assert: any further take reverts
    vm.expectRevert(IMidnight.ConsumedAssets.selector);
    take(1, lender, borrowerOffer);
}
```
