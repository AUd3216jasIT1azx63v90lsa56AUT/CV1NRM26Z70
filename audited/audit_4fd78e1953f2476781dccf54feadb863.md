### Title
Fee-on-Transfer Collateral Inflates Stored Balance, Enabling Undercollateralized Borrowing - (File: src/libraries/SafeTransferLib.sol)

### Summary
`supplyCollateral` in `src/Midnight.sol` credits `_position.collateral[collateralIndex]` with the full caller-supplied `assets` value before invoking `SafeTransferLib.safeTransferFrom`, which only verifies the boolean return of `transferFrom` and never checks the actual amount received. For a fee-on-transfer collateral token, the contract records `assets` but receives `assets - fee`, causing `isHealthy` to compute `maxDebt` against an inflated stored collateral balance, allowing a borrower to take debt that exceeds the real collateral value.

### Finding Description

**Code path:**

`supplyCollateral` (`src/Midnight.sol:524-546`):

```
...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545 — transfer happens after
```

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol:24-34`):

```
...
```

Only the boolean return value is checked. The actual balance delta is never measured.

`isHealthy` (`src/Midnight.sol:944-960`):
```
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(collateralParam.lltv, WAD);
```
Uses `_position.collateral[i]` — the stored (inflated) value — not the contract's real token balance.

**Root cause:** The check-effects-interactions ordering writes the full `assets` to storage before the transfer, and `safeTransferFrom` has no balance-before/after guard. For a fee-on-transfer token, `assets - fee` arrives but `assets` is stored.

**Market creation:** `touchMarket` (`src/Midnight.sol:755-791`) is permissionless and validates only: maturity bound, non-empty sorted collateral list, allowed LLTV tiers, and valid maxLif. There is no on-chain check excluding fee-on-transfer tokens. The Certora formal verification assumption "ERC20 tokens are assumed well-behaved" (`certora/README.md:112`) is a prover-level modeling convention, not an enforced on-chain invariant.

**Exploit flow:**
1. Attacker (market creator) calls `touchMarket` with a 3-collateral market where collateral at index `i` is a fee-on-transfer token (e.g., 1% fee). Two other collaterals are normal tokens.
2. Attacker (borrower) calls `supplyCollateral(market, i, 1000e18, attacker)`:
   - `_position.collateral[i]` is set to `1000e18` (line 533)
   - Contract receives only `990e18` (line 545, after 1% fee)
3. Attacker calls `supplyCollateral` for the two normal collaterals (correctly accounted).
4. Attacker takes debt via `take()`. `isHealthy` computes `maxDebt` using `1000e18` for the fee-on-transfer collateral, returning `true`.
5. Actual collateral value backing the position is `990e18 * price * lltv + ...` — less than the debt taken.
6. Position is undercollateralized from inception. The gap equals `fee * price * lltv / WAD / ORACLE_PRICE_SCALE` in loan token units.

**Why existing checks fail:** `safeTransferFrom` only checks the ERC20 boolean return; a fee-on-transfer token returns `true` while silently delivering fewer tokens. No balance snapshot is taken before or after the call. `isHealthy` has no access to actual contract balances — it reads only `_position.collateral[i]`.

### Impact Explanation
A borrower in a market containing at least one fee-on-transfer collateral token can open a position that is undercollateralized from the moment of creation. `isHealthy` returns `true` while actual collateral value < debt. The shortfall is `fee * oracle_price * lltv` per supply call, and it is permanent — the stored collateral value never self-corrects. When the position is eventually liquidated, the seized collateral is worth less than the debt repaid, realizing bad debt that is socialized across lenders via the loss factor mechanism.

### Likelihood Explanation
**Preconditions:** A market must exist with a fee-on-transfer token as one of its collateral tokens. Since `touchMarket` is permissionless and imposes no token-type restrictions, any unprivileged actor can create such a market. The attacker can be simultaneously the market creator and the borrower. The exploit is repeatable: each `supplyCollateral` call with a fee-on-transfer token inflates the stored balance by the fee amount. No special timing, oracle manipulation, or privileged access is required.

### Recommendation
In `supplyCollateral`, measure the contract's token balance before and after the transfer and credit only the actual received amount:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Move the storage write after the transfer and use `received` instead of `assets`. Alternatively, add explicit documentation and an on-chain guard (e.g., a token registry or a `require` that the token has no transfer fee) to prevent fee-on-transfer tokens from being registered as collateral in `touchMarket`.

```solidity
        super.transferFrom(from, to, amount - fee); // only amount-fee arrives

    }

        // Setup: 3-collateral market, collateral[1] is fee-on-transfer
        // ... (market creation, oracle setup at ORACLE_PRICE_SCALE)

        uint256 supplyAmount = 1000e18;
        feeCollateral.approve(address(midnight), supplyAmount);
        midnight.supplyCollateral(market, feeOnTransferIndex, supplyAmount, address(this));

        assertEq(feeCollateral.balanceOf(address(midnight)), 990e18, "actual balance is less");

            .mulDivDown(0.77e18, WAD);
        assertLt(actualMaxDebt, storedMaxDebt, "actual backing < stored maxDebt → bad debt");
```
