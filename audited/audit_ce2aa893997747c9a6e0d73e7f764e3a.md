### Title
Authorized operator can permanently freeze maker's entire offer group via `setConsumed(group, type(uint256).max, maker)` — (File: src/Midnight.sol)

### Summary
`setConsumed` in `src/Midnight.sol` accepts any address that `isAuthorized[onBehalf][msg.sender]` resolves to true, with no restriction on the `amount` value. Because the consumed mapping is strictly non-decreasing (enforced by `AlreadyConsumed()`), an authorized operator who passes `type(uint256).max` permanently saturates `consumed[maker][group]`, making every subsequent `take` and every lower `setConsumed` call on that group revert forever. The maker has no recovery path within the same group.

### Finding Description
**Exact code path — `src/Midnight.sol` lines 723–728:**

```solidity
/// @dev Passing type(uint256).max cancels all offers in the group (and never reverts).
function setConsumed(bytes32 group, uint256 amount, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    require(amount >= consumed[onBehalf][group], AlreadyConsumed());
    consumed[onBehalf][group] = amount;
    emit EventsLib.SetConsumed(msg.sender, group, amount, onBehalf);
}
```

The only two guards are:
- **Authorization**: `isAuthorized[onBehalf][msg.sender]` — any address the maker has ever authorized passes.
- **Monotonicity**: `amount >= consumed[onBehalf][group]` — `type(uint256).max` always satisfies this.

There is no guard that restricts the cancellation sentinel (`type(uint256).max`) to `msg.sender == onBehalf`.

**Exploit flow:**

| Step | Actor | Call |
|------|-------|------|
| 1 | maker | `setIsAuthorized(operator, true, maker)` — grants operator any authorization (e.g., to act as taker) |
| 2 | operator | `setConsumed(group, type(uint256).max, maker)` — passes both checks; sets `consumed[maker][group] = type(uint256).max` |
| 3 | maker | `setConsumed(group, X, maker)` for any `X < type(uint256).max` → **reverts `AlreadyConsumed()`** |
| 4 | any taker | `take(offer, ...)` where `offer.maker == maker && offer.group == group` → **reverts** |

For step 4, the `take` path at lines 371–372:
```solidity
newConsumed = consumed[offer.maker][offer.group] += units;
require(newConsumed <= offer.maxUnits, ConsumedUnits());
```
If `units > 0`, the `+=` overflows (Solidity 0.8 checked arithmetic → revert). If `units == 0`, `newConsumed == type(uint256).max` and `require(type(uint256).max <= offer.maxUnits)` fails for any finite `maxUnits`. Both branches are permanently blocked.

**Why existing checks fail:**
The `testSetConsumedAuthorization` test in `test/AuthorizationTest.sol` (lines 274–288) explicitly confirms that an authorized address can call `setConsumed` on behalf of the maker. The Certora `consumeNonDecreasing` rule in `certora/specs/Consume.spec` (lines 50–56) formally verifies that consumed never decreases — this is the invariant that makes the freeze irrecoverable. No check distinguishes between the maker calling `setConsumed` and an operator calling it with the maximum sentinel value.

### Impact Explanation
After step 2, `consumed[maker][group]` is permanently `type(uint256).max`. Every offer the maker has ever issued (or will issue) with that `group` value is irrecoverably cancelled: `take` reverts on overflow or `ConsumedUnits`/`ConsumedAssets`, and `setConsumed` with any lower value reverts `AlreadyConsumed()`. The maker must abandon the group entirely and migrate to a new `group` value, losing all partially-filled offer state and any off-chain coordination built around the old group identifier.

### Likelihood Explanation
The precondition — maker has authorized an operator — is the normal operating mode for any maker using a bundler, aggregator, or automated market-making bot. Authorization is all-or-nothing: granting it for `take` simultaneously grants it for `setConsumed`. The attack requires a single cheap transaction and is immediately permanent. A malicious or compromised operator, a front-running bot that observes a pending `setIsAuthorized`, or a griefing competitor can all trigger it. It is repeatable across every group the maker uses as long as authorization persists.

### Recommendation
Restrict the cancellation sentinel to the maker acting directly. The minimal fix is to add a guard in `setConsumed` that requires `msg.sender == onBehalf` when `amount == type(uint256).max`:

```solidity
function setConsumed(bytes32 group, uint256 amount, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    // Only the maker themselves may use the permanent-cancel sentinel.
    require(amount < type(uint256).max || onBehalf == msg.sender, Unauthorized());
    require(amount >= consumed[onBehalf][group], AlreadyConsumed());
    consumed[onBehalf][group] = amount;
    emit EventsLib.SetConsumed(msg.sender, group, amount, onBehalf);
}
```

Alternatively, introduce a dedicated `cancelGroup(bytes32 group)` function callable only by `msg.sender` (no `onBehalf` delegation), and remove the `type(uint256).max` shortcut from `setConsumed`.

### Proof of Concept
Foundry unit test plan (place in `test/`):

```solidity
function testOperatorPermanentGroupFreeze() public {
    address maker   = makeAddr("maker");
    address operator = makeAddr("operator");
    bytes32 group   = bytes32(uint256(1));

    // Precondition: maker authorizes operator for any reason.
    vm.prank(maker);
    midnight.setIsAuthorized(operator, true, maker);

    // Attack: operator saturates consumed to type(uint256).max.
    vm.prank(operator);
    midnight.setConsumed(group, type(uint256).max, maker);

    assertEq(midnight.consumed(maker, group), type(uint256).max);

    // Maker cannot reset: any lower value reverts AlreadyConsumed.
    vm.prank(maker);
    vm.expectRevert(IMidnight.AlreadyConsumed.selector);
    midnight.setConsumed(group, 0, maker);

    vm.prank(maker);
    vm.expectRevert(IMidnight.AlreadyConsumed.selector);
    midnight.setConsumed(group, type(uint256).max - 1, maker);

    // Any take on an offer in this group also reverts (overflow or ConsumedUnits).
    Offer memory offer;
    offer.maker  = maker;
    offer.group  = group;
    offer.maxUnits = type(uint256).max; // even with max cap, overflow on += units
    // ... configure remaining offer fields ...
    vm.expectRevert(); // arithmetic overflow or ConsumedUnits
    midnight.take(offer, hex"", 1, taker, taker, address(0), hex"");
}
```

**Expected assertions:**
- `consumed(maker, group) == type(uint256).max` after operator's call.
- `setConsumed(group, 0, maker)` reverts `AlreadyConsumed()`.
- `setConsumed(group, type(uint256).max - 1, maker)` reverts `AlreadyConsumed()`.
