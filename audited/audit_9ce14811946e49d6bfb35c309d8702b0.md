### Title
Rounding in `seizedAssets`-input path causes `_position.debt` underflow revert — (`src/Midnight.sol`)

### Summary

When `liquidate` is called with `seizedAssets > 0` and `repaidUnits = 0`, the computed `repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif)` uses ceiling rounding twice. With `price = ORACLE_PRICE_SCALE` the first `mulDivUp` is exact, but the second can round up by 1, producing `repaidUnits > _position.debt`. The subsequent unchecked subtraction `_position.debt -= UtilsLib.toUint128(repaidUnits)` then reverts with arithmetic underflow, even though the liquidator supplied a valid `seizedAssets ≤ available collateral`.

### Finding Description

**Code path** (`src/Midnight.sol` lines 649–676):

```solidity
// seizedAssets > 0 branch
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
                           .mulDivUp(WAD, lif);          // line 650 — ceiling rounding

// ... RCF check skipped in postMaturityMode ...

_position.debt -= UtilsLib.toUint128(repaidUnits);       // line 676 — no cap on repaidUnits
``` [1](#0-0) [2](#0-1) 

**Root cause:** There is no guard `repaidUnits <= _position.debt` before line 676. The `mulDivUp` on line 650 rounds up, so for any `seizedAssets` where `seizedAssets * WAD` is not divisible by `lif`, `repaidUnits` is inflated by 1. When that 1-unit inflation pushes `repaidUnits` above the remaining debt, the subtraction reverts.

**Concrete exploit flow:**

1. Market: `lltv = 0.77e18`, `price = ORACLE_PRICE_SCALE`, `maxLif ≈ 1.061e18` (cursor = 0.25e18).
2. Borrower state: `collateral = 2`, `debt = 1` (healthy pre-maturity; collateral value = 2 > debt = 1).
3. Time advances past `market.maturity` → position is post-maturity liquidatable.
4. Bad-debt computation: `badDebt = max(0, 1 − ceil(2·WAD/maxLif)) = max(0, 1−2) = 0`. Debt stays at 1.
5. Liquidator calls `liquidate(seizedAssets=2, repaidUnits=0, postMaturityMode=true)`.
6. `lif = WAD + (maxLif−WAD)·Δt/900 ≈ WAD` (just after maturity).
7. `repaidUnits = 2.mulDivUp(ORACLE_PRICE_SCALE, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif) = 2.mulDivUp(WAD, lif) = ceil(2·WAD/lif) = 2`.
8. `_position.debt -= 2` → `1 − 2` → arithmetic underflow → revert.

**Why existing checks fail:**

- The RCF check (lines 655–668) is **skipped** in `postMaturityMode`. [3](#0-2) 
- The collateral underflow check (line 670) passes because `seizedAssets = 2 = collateral`. [4](#0-3) 
- There is no `require(repaidUnits <= _position.debt)` anywhere in the function.
- The `testLiquidateCollateralInput` fuzz test manually bounds `seized` to avoid this, but the contract itself does not enforce that bound. [5](#0-4) 

### Impact Explanation

A liquidator providing a valid `seizedAssets` value (≤ available collateral) in post-maturity mode can trigger an unexpected arithmetic underflow revert. This blocks legitimate liquidations: the borrower's collateral cannot be seized and their debt cannot be cleared, violating the invariant that unhealthy/post-maturity positions remain liquidatable.

### Likelihood Explanation

The precondition is any post-maturity position where `collateral.mulDivUp(WAD, lif) > _position.debt` (after bad-debt subtraction). This is reachable whenever `seizedAssets * WAD` is not divisible by `lif` and the ceiling rounding pushes `repaidUnits` above remaining debt. It is repeatable by any unprivileged liquidator on any post-maturity market, requires no special oracle value (price = ORACLE_PRICE_SCALE is a normal value), and is more likely with small debt/collateral amounts where a 1-unit rounding error is significant relative to the debt.

### Recommendation

Cap `repaidUnits` at `_position.debt` after the bad-debt subtraction, before the collateral and debt updates:

```solidity
if (seizedAssets > 0) {
    repaidUnits = UtilsLib.min(
        seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif),
        _position.debt   // prevent underflow from ceiling rounding
    );
}
```

This preserves the semantics (liquidator seizes the requested collateral, repays at most the outstanding debt) and eliminates the underflow.

### Proof of Concept

```solidity
function testSeizedAssetsRoundingUnderflow() public {
    // Market: lltv=0.77e18, price=ORACLE_PRICE_SCALE, postMaturityMode
    // Borrower: collateral=2, debt=1
    // Setup: supply 2 collateral units, borrow 1 unit, advance past maturity

    uint256 COLLATERAL = 2;
    uint256 DEBT = 1;

    // Supply collateral and create debt=1
    collateralize(market, borrower, DEBT); // sets up collateral proportional to debt
    // Override collateral to 2 while keeping debt=1
    // (or use a market where collateral=2, debt=1 is achievable)
    setupMarket(market, DEBT);

    Oracle(market.collateralParams[0].oracle).setPrice(ORACLE_PRICE_SCALE);
    vm.warp(market.maturity + 1); // just past maturity, lif ≈ WAD

    // Assert: seizing all collateral (=2) should succeed but reverts
    vm.expectRevert(stdError.arithmeticError);
    midnight.liquidate(
        market, 0,
        COLLATERAL, // seizedAssets = 2
        0,          // repaidUnits = 0
        borrower, true, address(this), address(0), ""
    );
    // repaidUnits = ceil(2 * WAD / lif) = 2 > debt = 1 → underflow
}
```

**Expected assertion:** The call reverts with `arithmeticError` even though `seizedAssets ≤ collateral` and the position is legitimately liquidatable, confirming the bug.
