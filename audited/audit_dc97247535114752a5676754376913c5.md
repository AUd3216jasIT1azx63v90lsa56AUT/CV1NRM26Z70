### Title
Zero-buyerAssets Rounding Bypasses Fully-Consumed Check on Assets-Based Buy Offer, Triggering Maker Callback and Mutating Position State — (`src/Midnight.sol`)

### Summary

When a buy offer uses `maxAssets`-based consumption and `buyerPrice < WAD`, a taker can supply `units` small enough that `units.mulDivDown(buyerPrice, WAD) == 0`. Because the consumed accounting increments by `buyerAssets` (which is 0), the `newConsumed <= maxAssets` guard passes even when the offer is already fully consumed. The take executes in full: position credit/debt/totalUnits are mutated, the maker's `onBuy` callback is invoked with `buyerAssets=0, units=1`, and no token transfer occurs. The repo's own test `testBugBuyMaxAssetsBypass` confirms this path succeeds and documents the state corruption.

### Finding Description

**Code path — `src/Midnight.sol` lines 363–369:**

```
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // ← rounds to 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // ← maxAssets+0 <= maxAssets ✓
}
```

**Root cause:** The consumed counter is incremented by `buyerAssets`, not by `units`. When `buyerPrice < WAD` (any tick below `MAX_TICK`) and `units` is small (e.g., 1), `mulDivDown` truncates to 0. Adding 0 to an already-saturated counter leaves `newConsumed == maxAssets`, which satisfies the `<=` check.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets = N > 0`, `offer.tick` set so `offerPrice < WAD`
- `consumed[maker][group]` already equals `maxAssets` (fully consumed by prior legitimate fills or by `setConsumed`)
- `units = 1` (or any value where `units * buyerPrice < WAD`)

**Exploit flow:**
1. Precondition: `consumed[maker][group] == maxAssets`, `buyerPrice < WAD`.
2. Taker calls `take(offer, ..., units=1, ...)`.
3. `buyerAssets = 1.mulDivDown(buyerPrice, WAD) = 0`.
4. `newConsumed = maxAssets + 0 = maxAssets`; `require(maxAssets <= maxAssets)` passes.
5. Execution continues: `buyerCreditIncrease = zeroFloorSub(1, buyerPos.debt)` → 1 if buyer has no debt; `sellerDebtIncrease = 1`; positions mutated.
6. `buyerCallback = offer.callback` (line 420); if non-zero, `IBuyCallback(offer.callback).onBuy(..., buyerAssets=0, units=1, ...)` is called.
7. Token transfers: `safeTransferFrom(..., 0)` and `safeTransferFrom(..., 0)` — no tokens move.
8. Call is repeatable indefinitely; each iteration adds 0 to consumed and mutates state again.

**Why existing checks fail:**
- The `ConsumedAssets` guard at line 369 uses `<=`, not `<`, and compares `newConsumed` (which did not increase) against `maxAssets`. It cannot detect a zero-increment on a saturated counter.
- No check enforces `units == 0` when `buyerAssets == 0` for assets-based buy offers.
- The Certora spec rule `takeConsumedAtMaxUnchangedAssets` (line 88–96 of `Consume.spec`) requires `maxUnits == 0` but does not cover the case where `buyerAssets` rounds to zero, leaving the invariant unverified for this path. [1](#0-0) [2](#0-1) 

### Impact Explanation

The maker's `onBuy` callback is invoked with `buyerAssets=0` and `units=1` on an offer the maker considers fully consumed. Callback logic that gates on `buyerAssets > 0`, tracks cumulative fills, or assumes it will not be called after exhaustion will behave incorrectly. Beyond the callback, the buyer gains 1 unit of credit and the seller incurs 1 unit of debt per call, with zero token payment — a direct violation of the invariant that credit has matching debt funded by asset transfer. This is repeatable without limit. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

Preconditions are reachable by any unprivileged taker:
- `buyerPrice < WAD` holds for any tick below `MAX_TICK`, which is the common case for non-trivial markets.
- `maxAssets` can be any small value (e.g., 1) set by the maker.
- Full consumption of the offer is a normal end-state after legitimate fills.
- No privileged access, oracle manipulation, or user error is required.
- The attack is repeatable in the same transaction or across blocks.

The repo's own test `testBugBuyMaxAssetsBypass` (line 858) was written to demonstrate this exact scenario, confirming the developers identified it as a bug path. [5](#0-4) 

### Recommendation

Add an explicit guard that rejects a non-zero `units` take when the resulting `buyerAssets` (or `sellerAssets` for sell offers) is zero in assets-based mode:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetsTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, change the consumed check to `require(newConsumed < offer.maxAssets || units == 0)` so that a fully-saturated counter rejects any non-trivial take regardless of rounding. [6](#0-5) 

### Proof of Concept

The following Foundry unit test plan mirrors `testBugBuyMaxAssetsBypass` and extends it to assert callback invocation:

```solidity
function testFullyConsumedBuyOfferCallbackTriggered() public {
    // Setup: buy offer with maxAssets=1, tick below MAX_TICK so buyerPrice < WAD
    lenderOffer.buy = true;
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16;          // buyerPrice < WAD
    lenderOffer.callback = address(mockBuyCallback); // maker sets callback

    // Fully consume the offer legitimately
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, 1, lender);
    assertEq(midnight.consumed(lender, lenderOffer.group), 1);

    uint256 creditBefore = midnight.creditOf(id, lender);
    uint256 debtBefore   = midnight.debtOf(id, borrower);

    // Attacker (borrower) takes with units=1 on exhausted offer
    // Expected: REVERT. Actual: SUCCEEDS.
    (uint256 buyerAssets,) = take(1, borrower, lenderOffer);

    // Assertions that expose the bug:
    assertEq(buyerAssets, 0, "buyerAssets must be 0 due to rounding");
    assertEq(midnight.consumed(lender, lenderOffer.group), 1, "consumed unchanged");
    assertGt(midnight.creditOf(id, lender), creditBefore, "BUG: credit increased for free");
    assertGt(midnight.debtOf(id, borrower), debtBefore,   "BUG: debt increased for free");
    assertTrue(mockBuyCallback.wasCalled(), "BUG: callback invoked on exhausted offer");
    assertEq(mockBuyCallback.lastBuyerAssets(), 0, "callback received 0 assets");
    assertEq(mockBuyCallback.lastUnits(), 1,       "callback received 1 unit");
}
```

Expected assertion outcomes confirming the bug: `creditOf` and `debtOf` increase, callback fires with `buyerAssets=0`, and `consumed` stays at `maxAssets`. A correct implementation would revert before reaching the callback. [7](#0-6) [8](#0-7)
