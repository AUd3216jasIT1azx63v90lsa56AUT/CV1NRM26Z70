### Title
Token Allowlist Bypass via Direct Clone Initialization Enables Rogue Pool Deployment — (File: src/ConfidencePool.sol)

### Summary
`ConfidencePool.initialize()` is `external` with no restriction on who may call it. Because `poolImplementation` is a public state variable in the factory, any attacker can deploy their own minimal-proxy clone of the implementation and call `initialize()` directly with an arbitrary `stakeToken`, bypassing the factory's `allowedStakeToken` whitelist entirely.

### Finding Description
`ConfidencePoolFactory` enforces a token allowlist at pool-creation time: [1](#0-0) 

However, `ConfidencePool.initialize()` is declared `external initializer` with no check that `msg.sender` is the factory and no check that `stakeToken` is in the factory's allowlist: [2](#0-1) 

The factory's `poolImplementation` address is a public state variable: [3](#0-2) 

An attacker can read this address, deploy their own clone via `Clones.clone(poolImplementation)` (a standard OZ library callable by anyone), and then call `initialize()` on it with a malicious `stakeToken`. The only validation `initialize()` performs on the token is a zero-address check: [4](#0-3) 

The rogue pool passes all internal checks (valid agreement via registry, valid scope accounts via `IAgreement.isContractInScope`), but holds a non-whitelisted or malicious token. It will never appear in the factory's `_poolsByAgreement` mapping, but it is indistinguishable on-chain from a legitimate clone — same implementation bytecode, same interface, same agreement reference.

### Impact Explanation
A malicious token used as `stakeToken` can:
- Contain a backdoor allowing the token owner to drain the pool's balance after stakers deposit
- Implement a `transferFrom` that silently mints or redirects tokens, corrupting accounting
- Revert selectively on `safeTransfer` to permanently lock stakers' principal in the pool

The factory's allowlist exists precisely to exclude such tokens. Bypassing it defeats the primary defense against token-based fund loss documented in the factory: [5](#0-4) 

### Likelihood Explanation
The attack requires no special privilege. The attacker needs only:
1. A valid agreement address (any live agreement on the registry suffices — the attacker need not own it)
2. Any valid scope accounts for that agreement (readable from the agreement's public interface)
3. A call to `Clones.clone(factory.poolImplementation())` followed by `initialize()`

The factory's `poolImplementation` is public, the OZ `Clones` library is permissionless, and `initialize()` has no caller guard. The only friction is convincing stakers to interact with the rogue pool, which is feasible via a malicious frontend or off-chain social engineering.

### Recommendation
Add a caller restriction to `ConfidencePool.initialize()`. The simplest fix is to store the factory address at construction time and require `msg.sender == factory` in `initialize()`. Alternatively, pass the factory's `allowedStakeToken` mapping reference and validate the token inside `initialize()`. Either approach closes the bypass without changing the factory's existing flow, since the factory already calls `initialize()` immediately after `cloneDeterministic` in the same transaction. [6](#0-5) 

### Proof of Concept
```solidity
// Attacker (no agreement ownership required):
address impl = factory.poolImplementation();           // public — readable by anyone
address roguePool = Clones.clone(impl);                // permissionless OZ library

IConfidencePool(roguePool).initialize(
    validAgreement,          // any live agreement on the registry
    address(maliciousToken), // NOT in factory.allowedStakeToken — bypassed
    address(factory.safeHarborRegistry()),
    someOutcomeModerator,
    block.timestamp + 31 days,
    1 ether,
    attacker,                // recoveryAddress → attacker
    attacker,                // owner → attacker
    validScopeAccounts       // readable from IAgreement.isContractInScope
);

// roguePool is now a fully functional ConfidencePool clone with a malicious token.
// Stakers who deposit via stake() transfer maliciousToken to roguePool.
// Attacker drains via maliciousToken's backdoor or triggers selective-revert to lock funds.
```

The rogue pool is not in `factory.getPoolsByAgreement(validAgreement)`, but it shares the same implementation, ABI, and agreement reference as legitimate pools, making it convincing to users relying on bytecode or interface checks alone.

```text
    /// @inheritdoc IConfidencePool
    function initialize(
        address agreement_,
        address stakeToken_,
        address safeHarborRegistry_,
        address outcomeModerator_,
        uint256 expiry_,
        uint256 minStake_,
        address recoveryAddress_,
        address owner_,
        address[] calldata accounts
    ) external initializer {
        if (agreement_ == address(0)) revert ZeroAddress();
        if (stakeToken_ == address(0)) revert ZeroAddress();
        if (safeHarborRegistry_ == address(0)) revert ZeroAddress();
        if (outcomeModerator_ == address(0)) revert ZeroAddress();
        if (owner_ == address(0)) revert ZeroAddress();
        if (recoveryAddress_ == address(0)) revert InvalidRecoveryAddress();
        if (expiry_ < block.timestamp + _MIN_EXPIRY_LEAD) revert ExpiryTooSoon();
        if (expiry_ > type(uint32).max) revert ExpiryTooFar();
        if (minStake_ == 0) revert InvalidAmount();
        // aderyn-fp-next-line(reentrancy-state-change)
        if (!IBattleChainSafeHarborRegistry(safeHarborRegistry_).isAgreementValid(agreement_)) {
            revert InvalidAgreement();
        }

        agreement = agreement_;
        stakeToken = IERC20(stakeToken_);
        safeHarborRegistry = IBattleChainSafeHarborRegistry(safeHarborRegistry_);
        outcomeModerator = outcomeModerator_;
        // forge-lint: disable-next-line(unsafe-typecast)
        expiry = uint32(expiry_);
        minStake = minStake_;
        recoveryAddress = recoveryAddress_;
        outcome = PoolStates.Outcome.UNRESOLVED;

        _replaceScope(accounts);

        // Direct assignment (skipping Ownable2Step's two-step) so no `owner() == initializer-caller`
        // window exists between init and the new owner accepting. Two-step still applies to later transfers.
        _transferOwnership(owner_);
    }
```
