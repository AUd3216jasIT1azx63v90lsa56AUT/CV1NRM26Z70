### Title
SetterRatifier lacks chain-binding, enabling cross-chain replay of maker offer ratifications after a hard fork - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier` stores ratification as a plain `bool` in `isRootRatified[maker][root]` with no reference to `block.chainid`. After a hard fork, this storage state is identical on both chains, so any offer root ratified on the original chain is automatically valid on the forked chain. A taker (unprivileged attacker) can fill maker offers on the forked chain that the maker never intended to be active there, draining the maker's assets on that chain.

### Finding Description

**Root cause — missing chain-binding in `SetterRatifier`:**

`setIsRootRatified` writes a chain-agnostic boolean:

```solidity
// src/ratifiers/SetterRatifier.sol:18
mapping(address maker => mapping(bytes32 root => bool)) public isRootRatified;

// line 24-28
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    ...
}
```

`isRatified` checks only the Merkle proof and this boolean — no `block.chainid`:

```solidity
// line 30-37
function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
    require(msg.sender == MIDNIGHT, NotMidnight());
    ...
    require(isRootRatified[offer.maker][root], NotRatified());
    return CALLBACK_SUCCESS;
}

**Exploit flow:**

1. Maker calls `setIsRootRatified(maker, root, true)` on chain A (pre-fork). State: `isRootRatified[maker][root] = true`.
2. Hard fork occurs. Chain B inherits identical storage: `isRootRatified[maker][root] = true` on chain B.
3. Attacker (taker) on chain B calls `midnight.take(offer, abi.encode(root, leafIndex, proof), units, taker, ...)`.
4. `Midnight.take` calls `IRatifier(offer.ratifier).isRatified(offer, ratifierData)` → `SetterRatifier.isRatified`.
5. `isRatified` passes: Merkle proof is valid (same offer hash, same root), `isRootRatified[maker][root]` is `true`.
6. Take executes, draining maker's assets on chain B.

The attacker supplies only public data: `root` (observable on-chain), `leafIndex`, and `proof` (computable from the Merkle tree). No privileged access is needed. [8](#0-7) 

### Impact Explanation
After a hard fork, any maker using `SetterRatifier` has their offer roots automatically active on the forked chain. A taker can fill those offers on the forked chain, taking the maker's loan tokens or collateral without the maker's consent for that chain. The maker cannot preemptively cancel because `setIsRootRatified(maker, root, false)` on chain A does not affect chain B (the cancellation transaction would need to be replayed on chain B separately, and the attacker can front-run it).

### Likelihood Explanation
Hard forks are rare but have occurred on major EVM chains (ETH/ETC, ETH/ETH PoW). The precondition — a `SetterRatifier` deployment with active ratified roots at fork time — is the normal operating state for any maker using this ratifier. No special attacker capability is required beyond being a taker. The attack is repeatable for every ratified root that exists at fork time.

### Recommendation
Bind ratification to the chain at the time of `setIsRootRatified`, mirroring `EcrecoverRatifier`'s approach. Store `isRootRatified` keyed on `(maker, chainId, root)`:

```solidity
mapping(address maker => mapping(uint256 chainId => mapping(bytes32 root => bool))) public isRootRatified;
```

And in `isRatified`, verify:

```solidity
require(isRootRatified[offer.maker][block.chainid][root], NotRatified());
```

This ensures ratifications set on chain A are not automatically valid on chain B after a fork, consistent with the explicit protection already present in `EcrecoverRatifier`.

### Proof of Concept

```solidity
// Foundry unit test
function testSetterRatifierCrossChainReplay() public {
    // Setup: maker ratifies a root on the original chain
    Offer memory offer = makeOffer(lender);
    bytes32 root = HashLib.hashOffer(offer);

    vm.prank(lender);
    midnight.setIsAuthorized(address(setterRatifier), true, lender);
    vm.prank(lender);
    setterRatifier.setIsRootRatified(lender, root, true);

    // Assert ratified on original chain
    assertTrue(setterRatifier.isRootRatified(lender, root));

    // Simulate hard fork: change block.chainid
    uint64 forkChainId = block.chainid + 1;
    vm.chainId(forkChainId);

    // Assert: isRootRatified persists unchanged across chainid change
    assertTrue(setterRatifier.isRootRatified(lender, root));

    // Assert: isRatified succeeds on forked chain (attacker can take)
    vm.prank(address(midnight));
    bytes32 result = setterRatifier.isRatified(offer, abi.encode(root, 0, new bytes32[](0)));
    assertEq(result, CALLBACK_SUCCESS); // passes — no chain check

    // Contrast: EcrecoverRatifier would fail here because domainSeparator changes
    // bytes32 domSep = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(ecrecoverRatifier)));
    // The signature was made with the old chainId, so ecrecover would recover wrong address → revert
}
```

Expected assertions:
- `isRootRatified(lender, root)` returns `true` both before and after `vm.chainId(forkChainId)` — confirming no chain-binding.
- `isRatified` returns `CALLBACK_SUCCESS` on the forked chain — confirming the replay succeeds.
- The equivalent test with `EcrecoverRatifier` would revert with `Unauthorized` after the chainId change, confirming the asymmetry.
