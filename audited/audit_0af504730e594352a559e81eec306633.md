Audit Report

## Title
Midnight-Authorized Delegate Can Exhaust Victim's EcrecoverAuthorizer Nonces to Invalidate Pre-Signed Authorizations - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts signatures from any address that the victim has authorized on Midnight, not just the victim's own key. Because the nonce is keyed on `authorization.authorizer` and incremented unconditionally on success, an attacker holding a Midnight authorization from the victim can self-sign `Authorization` structs naming the victim as `authorizer` and submit them in a loop, permanently consuming `nonce[victim]`. Every pre-signed authorization the victim distributed to other agents becomes permanently invalid with `InvalidNonce`.

## Finding Description

**Exact code path — `EcrecoverAuthorizer.setIsAuthorized` (`src/periphery/EcrecoverAuthorizer.sol:24-48`):**

```solidity
// Line 26 — nonce keyed on authorization.authorizer, incremented unconditionally on success
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

// Lines 31-36 — signer check: authorizer OR any Midnight-authorized delegate
address signer = ecrecover(digest, signature.v, signature.r, signature.s);
require(signer != address(0), InvalidSignature());
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

**Root cause:** The second branch of the `Unauthorized` check — `IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)` — allows any Midnight-authorized delegate of the victim to sign an `Authorization` struct that names the victim as `authorizer`. The delegate's own ECDSA signature over a struct with `authorizer = victim` satisfies this check, and the nonce increment on line 26 fires against `nonce[victim]` and persists. There is no requirement that the signature originate from the authorizer's own key when consuming nonces.

**Exploit flow:**
1. Precondition: `IMidnight(MIDNIGHT).isAuthorized(victim, attacker) == true` — victim previously authorized attacker on Midnight (e.g., as a trading bot or router).
2. Victim pre-signs N authorizations with nonces 0…N-1 and distributes them to N agents for use in `take`, `repay`, `withdraw`, `liquidate`, or `claimFee` flows.
3. Attacker constructs `Authorization(authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=far_future)`, signs it with their own private key, and calls `EcrecoverAuthorizer.setIsAuthorized`.
4. Line 26 passes (`0 == nonce[victim]`), increments `nonce[victim]` to 1. The transaction succeeds.
5. Line 34 passes because `isAuthorized(victim, attacker) == true`.
6. Line 47 calls `IMidnight.setIsAuthorized(attacker, true, victim)` — a no-op since the authorization already exists.
7. Attacker repeats with nonce=1, 2, …, N-1.
8. All N pre-signed authorizations from the victim now revert with `InvalidNonce` when any agent tries to submit them.

**Why existing checks fail:**
- The `InvalidNonce` check (line 26) enforces sequential ordering only; it does not require that the authorizer themselves signed this particular nonce consumption.
- The `Unauthorized` check (lines 33-36) does not restrict who may consume the authorizer's nonces — it only requires the signer to be either the authorizer or a Midnight-authorized delegate. A delegate can therefore sign arbitrary `Authorization` structs naming the authorizer as `authorizer` and burn nonces at will.
- There is no `msg.sender == authorization.authorizer` guard or requirement that the signature originate from the authorizer's own key when consuming nonces.

## Impact Explanation
Any agent holding a Midnight authorization from the victim can unilaterally burn the victim's sequential nonces in `EcrecoverAuthorizer`. This permanently invalidates any batch of pre-signed `Authorization` structs the victim distributed off-chain. Every protocol action that relies on those pre-signed authorizations — `take`, `repay`, `withdraw`, `liquidate`, `claimFee` — reverts with `InvalidNonce` when submitted by the intended agents, causing a sustained and unrecoverable DoS on all those core market actions for the victim. The victim cannot "un-burn" consumed nonces; they must re-sign and redistribute all authorizations, which the attacker can immediately burn again as long as the Midnight authorization persists.

## Likelihood Explanation
**Precondition:** The attacker must hold `isAuthorized[victim][attacker] == true` on Midnight. This is a normal operational state — users routinely authorize bots, routers, or agents to act on their behalf. No tokens, collateral, or special role beyond this existing authorization is required.

**Feasibility:** The attack requires only repeated calls to `EcrecoverAuthorizer.setIsAuthorized` with self-signed structs. Each call costs one ECDSA signature (computed off-chain) and one transaction. On any L2 or low-fee chain, the cost per nonce burned is negligible. The attacker can front-run or race any legitimate submission.

**Repeatability:** The attack is indefinitely repeatable as long as the attacker remains authorized on Midnight. Even if the victim revokes the attacker's Midnight authorization, any nonces already burned are permanently consumed.

## Recommendation
Remove the delegate path from the `Unauthorized` check in `EcrecoverAuthorizer.setIsAuthorized`. The `EcrecoverAuthorizer` contract exists specifically to allow ECDSA-based authorization changes; the authorizer's own key should always be required to sign nonce-consuming operations. Delegates wishing to act on behalf of an authorizer should use the direct `IMidnight.setIsAuthorized` path, which does not involve sequential nonces.

```solidity
// Before (vulnerable):
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);

// After (fixed):
require(signer == authorization.authorizer, Unauthorized());
```

If delegate submission via `EcrecoverAuthorizer` is an intended feature, the nonce must be keyed on the signer (delegate), not on `authorization.authorizer`, so that a delegate's nonce consumption cannot affect the authorizer's nonce space.

## Proof of Concept

**Minimal manual steps:**
1. Deploy `EcrecoverAuthorizer` pointing to a mock `IMidnight` where `isAuthorized(victim, attacker) = true`.
2. Victim signs `Authorization(authorizer=victim, authorized=agentA, isAuthorized=true, nonce=0, deadline=far)` with victim's key and hands it to `agentA`.
3. Attacker constructs `Authorization(authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=far)`, signs with attacker's key, calls `EcrecoverAuthorizer.setIsAuthorized`.
4. Assert: `nonce[victim] == 1`. Call succeeds.
5. `agentA` attempts to submit victim's pre-signed authorization. Assert: reverts with `InvalidNonce`.

**Forge fuzz test sketch:**
```solidity
function testNonceBurnByDelegate(uint256 noncesToBurn) external {
    vm.assume(noncesToBurn > 0 && noncesToBurn < 100);
    // Setup: attacker is authorized on Midnight for victim
    midnight.setIsAuthorized(attacker, true, victim);
    for (uint256 i = 0; i < noncesToBurn; i++) {
        Authorization memory auth = Authorization(victim, attacker, true, i, type(uint256).max);
        bytes32 digest = computeDigest(auth);
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(attackerKey, digest);
        ecrecoverAuthorizer.setIsAuthorized(auth, Signature(v, r, s));
    }
    assertEq(ecrecoverAuthorizer.nonce(victim), noncesToBurn);
    // Victim's pre-signed auth with nonce=0 now reverts
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(victimPreSignedAuth, victimSignature);
}
```
