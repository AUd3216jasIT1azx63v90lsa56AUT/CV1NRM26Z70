Audit Report

## Title
`SafeTransferLib.safeTransfer` misinterprets 64-byte returndata with zero first word as `false`, enabling permanent lender fund freeze via malicious `loanToken` - (File: src/libraries/SafeTransferLib.sol)

## Summary
`SafeTransferLib.safeTransfer` uses `abi.decode(returndata, (bool))`, which reads only the first 32-byte word and silently ignores any trailing bytes. A token whose `transfer` returns 64 bytes with a zero first word is decoded as `false`, causing `TransferReturnedFalse()` to revert even though the call succeeded on-chain. Because `touchMarket` imposes no validation on `market.loanToken`, any unprivileged actor can create a market with such a token and permanently freeze all lender withdrawals.

## Finding Description

**Root cause — `src/libraries/SafeTransferLib.sol` line 21:**

```solidity
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
```

Solidity's `abi.decode(data, (bool))` reads the first 32-byte word of `data` as the boolean value and ignores any trailing bytes — it does not enforce that `data.length == 32`. When `returndata` is 64 bytes and the first word is `0x00…00` (e.g., `abi.encode(uint256(0), uint256(1))`), the decoded value is `false` and the `require` reverts with `TransferReturnedFalse()`, even though `success == true` and balances were updated on-chain. The `returndata.length == 0` short-circuit only bypasses the decode for zero-length returns; it provides no protection against oversized returns.

The existing test suite in `test/SafeTransferLibTest.sol` covers: no code, revert with reason, revert without reason, 32-byte `false`, 32-byte `true`, and zero-byte returns — but contains no test for 64-byte returndata with a zero first word, confirming this edge case is unexercised.

**No `loanToken` validation in `touchMarket` — `src/Midnight.sol` lines 755–791:**

`touchMarket` validates maturity, collateral count, collateral ordering, lltv, and maxLif. It performs zero checks on `market.loanToken`. Any address with `code.length > 0` is accepted as a loan token.

**Exploit flow:**

1. Attacker deploys `MaliciousToken`:
   - `transferFrom(from, to, amount)` → returns `abi.encode(true)` (32 bytes, first word non-zero) → `safeTransferFrom` passes.
   - `transfer(to, amount)` → returns `abi.encode(uint256(0), uint256(1))` (64 bytes, first word = 0) → `safeTransfer` reverts with `TransferReturnedFalse()`.
   - Both functions actually move balances on-chain.

2. Attacker calls `touchMarket` with `market.loanToken = address(MaliciousToken)`. Market is created without error.

3. Lender calls `take` → internally calls `safeTransferFrom(loanToken, …)` at line 455 → `transferFrom` returns 32-byte `true` → succeeds. Lender credit is recorded in `position[id][lender].credit`.

4. Lender calls `withdraw(market, units, lender, receiver)`:
   - Lines 493–495: state is mutated (`credit` decremented, `withdrawable` decremented, `totalUnits` decremented).
   - Line 499: `SafeTransferLib.safeTransfer(market.loanToken, receiver, units)` is called.
   - `transfer` returns 64 bytes with first word = 0 → `abi.decode(returndata, (bool))` = `false` → `require` reverts with `TransferReturnedFalse()`.
   - Entire transaction reverts; state is rolled back. Lender credit remains locked.

5. Every subsequent `withdraw` call follows the same path and reverts identically. Funds are permanently frozen with no recovery path.

**Why existing checks are insufficient:**
- The `returndata.length == 0` guard only bypasses decoding for zero-length returns; it does not protect against returns longer than 32 bytes.
- There is no allowlist, interface check, or code-size validation on `loanToken` in `touchMarket`.
- The Certora `Reverts.spec` summarizes `SafeTransferLib.safeTransfer` as a CVL stub (`CVL_safeTransfer`), meaning the 64-byte false-return edge case is not modeled in formal verification.

## Impact Explanation

All lender funds supplied to any market whose `loanToken` returns 64 bytes with a zero first word from `transfer` are permanently unwithdrawable. `withdraw` always reverts at line 499 with no recovery path. This constitutes a permanent, unrecoverable freeze of user funds — a direct in-scope impact per the RESEARCHER.md definition of "Permanent lock, freeze, or unrecoverable corruption of user/project state."

## Likelihood Explanation

All attacker preconditions are fully permissionless: deploying a contract requires no privileges, and `touchMarket` accepts any `loanToken`. The attack requires attracting lenders, but in a permissionless protocol with automated vaults or aggregators, a token that behaves normally during `transferFrom` (returning 32-byte `true`) but abnormally during `transfer` can attract real deposits before the `transfer` path is exercised. The condition is deterministic and repeatable: every `withdraw` call reverts with no exception.

## Recommendation

1. **Harden `SafeTransferLib`**: Replace the bare `abi.decode` with an explicit length check before decoding:
   ```solidity
   require(
       returndata.length == 0 ||
       (returndata.length == 32 && abi.decode(returndata, (bool))),
       TransferReturnedFalse()
   );
   ```
   This rejects any returndata that is neither empty nor exactly 32 bytes, closing the 64-byte edge case.

2. **Add `loanToken` validation in `touchMarket`**: Require that `market.loanToken` passes a basic sanity check (e.g., `code.length > 0` and optionally an allowlist or interface check) before a market is created.

3. **Add a regression test** to `SafeTransferLibTest.sol` covering a token that returns 64 bytes with a zero first word from both `transfer` and `transferFrom`.
