### Title
Fully-consumed assets-based buy offer with `buyerPrice < WAD` can be taken indefinitely with zero consumed increment - (`src/Midnight.sol`)

### Summary
When `offer.buy = true`, `offer.maxAssets > 0`, and `buyerPrice < WAD`, the consumed tracking in `take` increments by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For small `units`, this rounds down to zero, so `consumed` does not increase. The guard `require(newConsumed <= offer.maxAssets)` then passes even when `consumed == maxAssets`, allowing the offer to be taken repeatedly after full exhaustion. The protocol's own test suite (`testBugBuyMaxAssetsBypass`) confirms the bug is reachable and produces observable bad state.

### Finding Description

**Code path** — `src/Midnight.sol`, `take()`:

```
buyerAssets = units.mulDivDown(buyerPrice, WAD)   // line 363
// offer.buy == true, maxAssets > 0:
newConsumed = consumed[maker][group] += buyerAssets  // line 368
require(newConsumed <= offer.maxAssets)              // line 369
``` [1](#0-0) 

When `buyerPrice < WAD` (tick set below the WAD threshold, e.g. `MAX_TICK - 16`) and `units = 1`, `mulDivDown(1, buyerPrice, WAD)` evaluates to `0`. The `+=` adds nothing, so `consumed` stays at `maxAssets`. The `require` passes because `maxAssets <= maxAssets`.

The protocol itself documents this edge case at line 94:

> "It is possible to give units to a fully consumed assets-based buy offer with price < 1." [2](#0-1) 

**Attacker-controlled inputs:**
- `offer.buy = true`
- `offer.maxAssets > 0` (any nonzero value, e.g. `1`)
- `offer.tick` chosen so `buyerPrice < WAD` (e.g. `MAX_TICK - 16`)
- `units = 1` (or any value where `mulDivDown(units, buyerPrice, WAD) == 0`)
- `consumed[maker][group]` pre-set to `maxAssets` (either by prior legitimate fills or by the maker's own `setConsumed`)

**Exploit flow (confirmed by `testBugBuyMaxAssetsBypass`):**

1. Offer is fully consumed: `consumed[lender][group] == maxAssets == 1`
2. Attacker (borrower) calls `take(units=1, taker=borrower, offer=lenderOffer)`
3. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`
4. `consumed += 0` → still `== maxAssets`
5. `require(maxAssets <= maxAssets)` passes
6. Execution continues: `buyerPos.credit += 1` (lender gets credit), `sellerPos.debt += 1` (borrower gets debt), `totalUnits += 1`
7. No tokens transferred (`buyerAssets = sellerAssets = 0`)
8. Step 2–7 can be repeated indefinitely [3](#0-2) 

**Why existing checks fail:** The sole guard is `require(newConsumed <= offer.maxAssets)`. It is bypassed whenever `buyerAssets` rounds to zero, which is structurally guaranteed when `buyerPrice < WAD` and `units` is small. There is no separate check that `units == 0` when the offer is fully consumed.

### Impact Explanation

A fully exhausted buy offer (consumed == maxAssets) can be taken an unlimited number of times. Each call:
- Grants the maker (lender) additional credit units beyond their intended `maxAssets` cap, without the maker paying any loan-token assets (`buyerAssets = 0`)
- Forces the taker (seller/borrower) to absorb debt without receiving any assets (`sellerAssets = 0`)
- Inflates `totalUnits` without backing assets, corrupting the credit/debt accounting invariant

The maker's `maxAssets` cap — the primary mechanism for limiting offer exposure — is rendered ineffective. The offer is reused after exhaustion, and the maker is undercharged (receives credit for free).

### Likelihood Explanation

**Preconditions:**
1. `offer.buy = true` — standard buy offer, no privilege required
2. `offer.maxAssets > 0` — maker chose assets-based cap (common pattern)
3. `buyerPrice < WAD` — tick below the WAD threshold; the protocol explicitly supports ticks in this range
4. `consumed == maxAssets` — offer has been fully filled at least once (normal lifecycle)

All preconditions are reachable by any unprivileged taker. The attack is repeatable on every block with `units = 1`. No oracle manipulation, admin access, or token misbehavior is required.

### Recommendation

Add an explicit guard that rejects non-zero-unit takes when the offer is already fully consumed in assets mode. The minimal fix is to require that `buyerAssets > 0` whenever `units > 0` for a buy offer with `maxAssets > 0`, or equivalently, check that `consumed < maxAssets` before the take proceeds:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetNonZeroUnits()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce `require(consumed[offer.maker][offer.group] < offer.maxAssets || units == 0)` before computing amounts.

### Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` is a complete Foundry unit test that reproduces the bug: [3](#0-2) 

**Expected assertions (current, broken behavior):**
- `take(1, borrower, lenderOffer)` does **not** revert
- `consumed(lender, group) == maxAssets` (unchanged — bug)
- `creditOf(id, lender) > lenderCreditBefore` (lender gained credit for free — bug)
- `debtOf(id, borrower) > borrowerDebtBefore` (borrower gained debt with no assets — bug)
- `totalUnits(id) > totalUnitsBefore` (unbacked units created — bug)

**Correct behavior after fix:** `take(1, borrower, lenderOffer)` should revert (e.g. with `ConsumedAssets` or a new `ZeroAssetNonZeroUnits` error) when `consumed == maxAssets` and `buyerAssets` would round to zero.

A fuzz test should additionally sweep `units` in `[1, WAD/buyerPrice)` and assert revert for all values where `mulDivDown(units, buyerPrice, WAD) == 0` and `consumed == maxAssets`.

    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
