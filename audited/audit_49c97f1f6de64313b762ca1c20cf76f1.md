### Title
Referral-fee markup overflows refund subtraction in `buyWithUnitsTargetAndWithdrawCollateral` when fills consume all of `maxBuyerAssets` - (File: src/periphery/MidnightBundles.sol)

### Summary
In `MidnightBundles.buyWithUnitsTargetAndWithdrawCollateral`, the referral fee is computed with a markup formula (`filledBuyerAssets * referralFeePct / (WAD - referralFeePct)`) applied after fills complete. No check enforces that `filledBuyerAssets + referralFeeAssets <= maxBuyerAssets`, so when fills consume the full `maxBuyerAssets` budget and `referralFeePct > 0`, the refund subtraction `maxBuyerAssets - filledBuyerAssets - referralFeeAssets` underflows and reverts in Solidity 0.8.x. The user's transaction fails after gas is spent and no credit is acquired.

### Finding Description

**Exact code path** — `src/periphery/MidnightBundles.sol` lines 66–104:

```
Line 66:  pullToken(loanToken, msg.sender, maxBuyerAssets, loanTokenPermit);
Line 67:  forceApproveMax(loanToken, MIDNIGHT);
          // fills loop: filledBuyerAssets accumulates up to maxBuyerAssets
Line 102: uint256 referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
Line 103: if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
Line 104: SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - filledBuyerAssets - referralFeeAssets);
                                                              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                                              UNDERFLOW when filledBuyerAssets + referralFeeAssets > maxBuyerAssets
``` [1](#0-0) [2](#0-1) 

**Root cause** — `mulDivDown` is `(x * y) / d` with no overflow guard beyond Solidity's checked arithmetic. [3](#0-2) 

The markup formula produces:

```
referralFeeAssets = filledBuyerAssets * referralFeePct / (WAD - referralFeePct)
total cost        = filledBuyerAssets * WAD / (WAD - referralFeePct)
```

For any `referralFeePct > 0`, `total cost > filledBuyerAssets`. If `filledBuyerAssets = maxBuyerAssets` (fills consumed the entire approved budget), then `total cost > maxBuyerAssets`, and line 104 underflows.

**Attacker-controlled inputs:**
- `referralFeePct`: any value in `[1, WAD-1]` — the only guard is `require(referralFeePct < WAD)`.
- `maxBuyerAssets`: set by the caller as a slippage bound for fills.
- Offer prices: a maker can post an offer priced such that `filledBuyerAssets` equals exactly `maxBuyerAssets`. [4](#0-3) 

**Why existing checks fail:**
- `require(referralFeePct < WAD)` only prevents `referralFeePct >= WAD`; it does not bound the total cost relative to `maxBuyerAssets`.
- `require(filledUnits == targetUnits)` only checks unit completeness.
- There is no `require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets)` anywhere. [5](#0-4) 

**Concrete numeric example:**
- `referralFeePct = 0.5e18` (50%), `maxBuyerAssets = 100e18`, `filledBuyerAssets = 100e18`
- `referralFeeAssets = 100e18 * 0.5e18 / 0.5e18 = 100e18`
- Line 104: `100e18 - 100e18 - 100e18` → underflow → revert

Even with `referralFeePct = 1` (1 wei):
- `referralFeeAssets = 100e18 * 1 / (1e18 - 1) ≈ 100 wei`
- Line 104: `100e18 - 100e18 - 100` → underflow → revert

The NatSpec comment on line 46 states "The msg.sender will pay at most `maxBuyerAssets`", but this invariant is violated — the contract reverts instead of capping the cost. [6](#0-5) 

### Impact Explanation
A user calling `buyWithUnitsTargetAndWithdrawCollateral` with any `referralFeePct > 0` and a `maxBuyerAssets` that is tight (i.e., close to the expected fill cost) will have their transaction revert at line 104 after all fills have already succeeded on-chain. The user loses gas, acquires no credit, and receives no partial refund. The fills themselves are rolled back by the revert, but the user cannot retry without either setting `maxBuyerAssets` to an inflated value (defeating the slippage bound) or setting `referralFeePct = 0`.

### Likelihood Explanation
**Preconditions:**
1. `referralFeePct > 0` — any nonzero referral fee, which is the intended use case for the parameter.
2. `filledBuyerAssets` close to or equal to `maxBuyerAssets` — occurs whenever the user sets a tight slippage bound or the market price moves to the boundary between call submission and execution.

Both conditions are normal, expected usage. The bug is repeatable for any user who sets a referral fee and a non-infinite `maxBuyerAssets`. The existing fuzz test at line 534 avoids the issue by using `type(uint256).max` as `maxBuyerAssets`, which masks the bug. [7](#0-6) 

### Recommendation
Before the transfers, add a check that the total cost fits within `maxBuyerAssets`:

```solidity
uint256 referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets, BuyerAssetsTooHigh());
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - filledBuyerAssets - referralFeeAssets);
```

Alternatively, cap the fill budget upfront so the referral fee always fits:

```solidity
uint256 maxFilledBuyerAssets = maxBuyerAssets.mulDivDown(WAD - referralFeePct, WAD);
// use maxFilledBuyerAssets as the approval ceiling for MIDNIGHT
```

This mirrors the approach used in `buyWithAssetsTargetAndWithdrawCollateral` (line 200–201), which correctly pre-deducts the referral fee from `targetBuyerAssets` before fills. [8](#0-7) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity 0.8.34;

import {Test} from "forge-std/Test.sol";
// ... standard MidnightBundlesTest setup ...

contract BuyUnitsTargetReferralUnderflowTest is MidnightBundlesTest {

    /// @notice Fuzz: any referralFeePct in [1, WAD-1] with tight maxBuyerAssets triggers underflow.
    function testFuzz_buyUnitsTarget_referralFee_underflow(
        uint256 units,
        uint256 referralFeePct
    ) public {
        units = bound(units, 1, uint256(type(uint128).max) / 2);
        referralFeePct = bound(referralFeePct, 1, WAD - 1); // nonzero fee

        // Zero settlement fees so buyerPrice = price exactly.
        for (uint256 i; i <= 6; i++) midnight.setMarketSettlementFee(id, i, 0);

        uint256 price = TickLib.tickToPrice(MAX_TICK);
        // maxBuyerAssets set to EXACTLY the expected fill cost (tight slippage bound).
        uint256 maxBuyerAssets = units.mulDivUp(price, WAD); // == filledBuyerAssets

        offers[0].buy = false;
        offers[0].maker = borrower;
        offers[0].receiverIfMakerIsSeller = borrower;
        offers[0].maxUnits = type(uint256).max;
        collateralize(market, borrower, units);

        Take[] memory takes = new Take[](1);
        takes[0] = Take({offer: offers[0], units: type(uint256).max, ratifierData: hex""});

        address referrer = makeAddr("referrer");

        vm.prank(lender);
        // Expected: reverts with arithmetic underflow (Panic 0x11)
        // because filledBuyerAssets == maxBuyerAssets and referralFeeAssets > 0
        // => maxBuyerAssets - filledBuyerAssets - referralFeeAssets underflows
        vm.expectRevert(stdError.arithmeticError);
        midnightBundles.buyWithUnitsTargetAndWithdrawCollateral(
            units,
            maxBuyerAssets,   // tight: equals expected fill cost, no room for referral fee
            lender,
            _noPermit(),
            takes,
            new CollateralWithdrawal[](0),
            address(0),
            referralFeePct,   // nonzero
            referrer
        );

        // Assert: lender's credit is 0 (transaction reverted, no fill persisted)
        assertEq(midnight.creditOf(id, lender), 0, "no credit acquired");
        // Assert: lender's token balance unchanged (full refund via revert)
        // (gas cost aside, no token loss — but user wasted gas and failed to acquire credit)
    }

    /// @notice Unit test: concrete values demonstrating the underflow.
    function test_buyUnitsTarget_referralFee_underflow_concrete() public {
        uint256 units = 100e18;
        uint256 referralFeePct = 0.5e18; // 50%

        for (uint256 i; i <= 6; i++) midnight.setMarketSettlementFee(id, i, 0);

        uint256 price = TickLib.tickToPrice(MAX_TICK);
        uint256 maxBuyerAssets = units.mulDivUp(price, WAD);
        // filledBuyerAssets will equal maxBuyerAssets
        // referralFeeAssets = maxBuyerAssets * 0.5e18 / 0.5e18 = maxBuyerAssets
        // refund = maxBuyerAssets - maxBuyerAssets - maxBuyerAssets → underflow

        offers[0].buy = false;
        offers[0].maker = borrower;
        offers[0].receiverIfMakerIsSeller = borrower;
        offers[0].maxUnits = type(uint256).max;
        collateralize(market, borrower, units);

        Take[] memory takes = new Take[](1);
        takes[0] = Take({offer: offers[0], units: type(uint256).max, ratifierData: hex""});

        vm.prank(lender);
        vm.expectRevert(stdError.arithmeticError);
        midnightBundles.buyWithUnitsTargetAndWithdrawCollateral(
            units, maxBuyerAssets, lender, _noPermit(), takes,
            new CollateralWithdrawal[](0), address(0), referralFeePct, makeAddr("referrer")
        );
    }
}
```

**Expected assertions:**
- Both tests revert with `Panic(0x11)` (arithmetic underflow) at line 104 of `MidnightBundles.sol`.
- `midnight.creditOf(id, lender) == 0` after revert (no state persists).
- The fuzz test covers the full range `referralFeePct ∈ [1, WAD-1]` and confirms the underflow is universal whenever fills consume all of `maxBuyerAssets`.
