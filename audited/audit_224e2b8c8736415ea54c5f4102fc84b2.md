### Title
Fee-on-Transfer Collateral Token Inflates Position Accounting, Enabling Undercollateralized Borrowing — (File: src/Midnight.sol)

### Summary
`supplyCollateral()` credits `assets` to a borrower's position before the actual token transfer, with no check on the amount actually received. If the collateral token charges a transfer fee, the protocol records more collateral than it holds. The borrower can then borrow against the inflated collateral value, creating an undercollateralized position that socializes bad debt onto lenders. The same class of bug also affects `repay()`, where `withdrawable` is over-credited relative to actual received loan tokens.

### Finding Description

**Root cause — `supplyCollateral()`:**

In `src/Midnight.sol`, `supplyCollateral()` updates the position's collateral balance using the caller-supplied `assets` value, then calls `safeTransferFrom` for that same amount:

```solidity
// src/Midnight.sol:533
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
// ...
// src/Midnight.sol:545
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

`SafeTransferLib.safeTransferFrom` only checks that the call did not revert and that the return value (if any) is `true` — it does **not** verify the actual balance delta of the contract:

```solidity
// src/libraries/SafeTransferLib.sol:24-34
function safeTransferFrom(address token, address from, address to, uint256 value) internal {
    ...
    require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
}
```

If `collateralToken` is a fee-on-transfer ERC-20 (e.g., a token that deducts 1% on every transfer), the contract receives `assets * 0.99` but the position is credited with `assets`. The health check in `isHealthy()` and the liquidation logic both use `position.collateral[collateralIndex]` as the authoritative collateral amount, so the borrower can take on debt backed by phantom collateral.

**Secondary root cause — `repay()`:**

```solidity
// src/Midnight.sol:508-509
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);
// ...
// src/Midnight.sol:520
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
```

If the loan token charges a fee, `withdrawable` is over-credited by the fee amount. Lenders calling `withdraw()` can drain more tokens than the contract actually holds, breaking the solvency invariant proven in Certora (which explicitly assumes no fee-on-transfer tokens at `certora/specs/Solvency.spec:31`).

**Exploit path for `supplyCollateral()`:**

1. A market is created (permissionlessly) with a fee-on-transfer token as one of its collateral assets, or an existing collateral token (e.g., USDT) activates its fee mechanism.
2. Attacker calls `supplyCollateral(market, collateralIndex, 1000e18, attacker)`. The contract receives `990e18` (1% fee) but records `1000e18` in `position.collateral`.
3. Attacker calls `take()` as seller, borrowing the maximum amount permitted by the inflated collateral value.
4. Attacker walks away. The position is undercollateralized by the fee amount. When liquidated, only `990e18` of collateral can be seized, but the debt was sized for `1000e18`. The shortfall is socialized as bad debt via `lossFactor` onto all lenders in the market.

### Impact Explanation

- **Direct financial loss to lenders**: Bad debt from undercollateralized positions is socialized via `lossFactor`, permanently reducing the credit value of all lenders in the affected market.
- **Solvency invariant broken**: The Certora-proven invariant `tokenBalances[token][currentContract] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token)` is violated, as the contract's actual token balance falls below the sum of recorded obligations.
- **Severity**: High — direct, permanent loss of lender funds with no recovery path.

### Likelihood Explanation

- Markets are **permissionlessly created** with any ERC-20 as collateral. An attacker can deliberately choose a fee-on-transfer token.
- Tokens like USDT have a fee mechanism that is currently disabled but can be activated by the token owner at any time, retroactively affecting all existing markets using that token.
- No privileged access is required; any external user can call `supplyCollateral()`.

### Recommendation

Measure the actual received amount by comparing contract balance before and after the transfer, and use the delta for accounting:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Apply the same pattern to `repay()` for the loan token. Alternatively, explicitly document and enforce (e.g., via a registry or market creation check) that fee-on-transfer tokens are not permitted as loan or collateral tokens.

### Proof of Concept

```
Given: collateralToken charges 1% fee on every transfer.
       Market M uses collateralToken at collateralIndex=0.
       LLTV = 0.9 (90%).

1. attacker calls supplyCollateral(M, 0, 1000e18, attacker)
   → contract receives 990e18 tokens
   → position[id][attacker].collateral[0] = 1000e18  ← inflated

2. attacker calls take() as seller (borrower), borrowing up to
   floor(1000e18 * 0.9) = 900e18 units of loan token
   (health check passes because it reads 1000e18 from position)

3. attacker withdraws the 900e18 loan tokens and abandons the position.

4. Liquidator seizes collateral: only 990e18 tokens available.
   Debt was 900e18 units; at oracle price, collateral covers only
   990e18 * oraclePrice worth of debt. The shortfall is bad debt,
   socialized to all lenders via lossFactor.
```
