### Title
Silent No-Op `transferFrom` Bypasses Loan Token Payment in `liquidate()`, Breaking `withdrawable` Accounting Invariant - ([File: src/Midnight.sol])

### Summary

`SafeTransferLib.safeTransferFrom` accepts any token whose `transferFrom` returns empty calldata without reverting, because the guard `require(returndata.length == 0 || abi.decode(returndata, (bool)), ...)` short-circuits to `true` on empty return data. In `liquidate()`, all critical state mutations — collateral seizure, debt reduction, and `withdrawable` increment — are committed before the loan token pull at line 717. A market creator who sets `loanToken` to an ERC20NoRevert contract (one whose `transferFrom` is a no-op returning nothing) can therefore liquidate an unhealthy position, receive real collateral, and pay zero loan tokens, permanently inflating `withdrawable` beyond the protocol's actual loan token balance.

### Finding Description

**Root cause — `SafeTransferLib.safeTransferFrom`:** [1](#0-0) 

The check on line 33 passes whenever `returndata.length == 0`. A contract whose `transferFrom` body is empty (or simply `return`s with no value) satisfies this: the low-level call succeeds (`success = true`), `returndata` is zero-length, and the require is trivially satisfied. No tokens move.

**State mutations before the transfer in `liquidate()`:** [2](#0-1) 

`_position.collateral` is reduced, `_marketState.withdrawable` is incremented by `repaidUnits`, and `_position.debt` is decremented — all before any external call. Collateral is then sent to the receiver: [3](#0-2) 

Only after collateral is gone does the protocol attempt to pull loan tokens: [4](#0-3) 

There is no balance snapshot before/after this call, so a silent no-op is indistinguishable from a real transfer.

**Permissionless market creation — no `loanToken` validation:**

`touchMarket` validates `lltv`, `maxLif`, collateral ordering, and maturity, but imposes zero restrictions on `market.loanToken`. Any contract address with `code.length > 0` is accepted. An unprivileged market creator deploys `ERC20NoRevert` and registers a market with it as `loanToken`.

**Exploit flow:**

1. Attacker deploys `ERC20NoRevert` (contract with `transferFrom() {}` — no-op, no return value).
2. Attacker calls `touchMarket` with `loanToken = ERC20NoRevert` and a real collateral token.
3. A borrower supplies real collateral via `supplyCollateral` and opens a debt position (the borrow side also uses `safeTransferFrom` on the no-op token, so the borrower receives nothing real — but the debt is recorded on-chain).
4. Position becomes unhealthy (debt > maxDebt via oracle, or `block.timestamp > market.maturity` for `postMaturityMode`).
5. Attacker calls `liquidate(market, collateralIndex, seizedAssets > 0, 0, borrower, ...)`.
6. Lines 649–676 execute: `repaidUnits` is computed from `seizedAssets`, collateral is deducted, `withdrawable += repaidUnits`, `debt -= repaidUnits`.
7. Line 696: real collateral token transferred to attacker's `receiver`.
8. Line 717: `safeTransferFrom(ERC20NoRevert, payer, address(this), repaidUnits)` — no-op, succeeds, zero tokens received.

**Why existing checks fail:**

- `require(UtilsLib.atMostOneNonZero(repaidUnits, seizedAssets))` — satisfied; attacker passes `seizedAssets > 0, repaidUnits = 0`.
- `require(_position.debt > 0)` — satisfied; borrower has debt.
- `require(!liquidationLocked && originalDebt > maxDebt)` — satisfied; position is genuinely unhealthy.
- `liquidatorGate` — `address(0)` in attacker-created market.
- `SafeTransferLib` — passes on empty returndata.
- No post-transfer balance assertion exists anywhere in `liquidate()`.

### Impact Explanation

After the call: `_marketState.withdrawable` has increased by `repaidUnits` but `loanToken.balanceOf(address(this))` is unchanged. The invariant `withdrawable <= loanToken.balanceOf(midnight)` is broken. The attacker receives real collateral assets for free. Any lender who later calls `withdraw` against this market will find the protocol insolvent for the inflated `withdrawable` amount. The borrower's debt is erased without repayment.

### Likelihood Explanation

Market creation is fully permissionless — no governance approval is required. Deploying an ERC20NoRevert contract costs one transaction. The only precondition is an unhealthy position in the attacker-controlled market, which the attacker can engineer by controlling the oracle address (also unchecked in `touchMarket`) or by waiting for maturity. The attack is repeatable across any number of markets and positions.

### Recommendation

Add a before/after balance check around the loan token pull in `liquidate()`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
require(
    IERC20(market.loanToken).balanceOf(address(this)) >= balanceBefore + repaidUnits,
    InsufficientLoanTokenReceived()
);
```

Apply the same pattern to every `safeTransferFrom` that credits the protocol's internal accounting (`take`, `repay`, `supplyCollateral`, `flashLoan` repayment). Alternatively, restrict `market.loanToken` to a protocol-approved allowlist at market creation time.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../src/Midnight.sol";

/// @dev No-op transferFrom: returns nothing, does not revert.
contract ERC20NoRevert {
    // transferFrom selector: 0x23b872dd
    fallback() external {
        // return nothing — safeTransferFrom sees returndata.length == 0 → passes
    }
}

contract LiquidateNoOpTokenTest is Test {
    Midnight midnight;
    ERC20NoRevert loanToken;
    ERC20 realCollateral; // standard ERC20 with real balances

    function testLiquidateNoOpLoanToken() public {
        // Setup: deploy protocol, no-op loan token, real collateral token
        // Create market with loanToken = ERC20NoRevert
        // Borrower supplies real collateral, opens debt position
        // Make position unhealthy (e.g., postMaturityMode after warp)
        // Attacker calls liquidate(seizedAssets > 0)

        uint256 collateralBefore = realCollateral.balanceOf(attacker);
        uint256 loanBalBefore = loanToken.balanceOf(address(midnight)); // 0
        uint256 withdrawableBefore = midnight.marketState(id).withdrawable; // 0

        vm.prank(attacker);
        (uint256 seized, uint256 repaid) = midnight.liquidate(
            market, 0, SEIZED_ASSETS, 0, borrower, true, attacker, address(0), ""
        );

        // Assertions:
        assertGt(seized, 0, "attacker seized collateral");
        assertGt(repaid, 0, "repaidUnits computed non-zero");
        assertEq(
            realCollateral.balanceOf(attacker),
            collateralBefore + seized,
            "attacker received real collateral"
        );
        assertEq(
            loanToken.balanceOf(address(midnight)),
            loanBalBefore,
            "midnight loanToken balance unchanged — no tokens paid"
        );
        assertGt(
            midnight.marketState(id).withdrawable,
            withdrawableBefore,
            "withdrawable inflated without backing tokens"
        );
        // Invariant broken: withdrawable > loanToken.balanceOf(midnight)
    }
}
```

**Expected assertions:** all pass, confirming collateral is seized, `withdrawable` is inflated, and `loanToken.balanceOf(midnight)` is unchanged — a concrete violation of the protocol's solvency invariant.
