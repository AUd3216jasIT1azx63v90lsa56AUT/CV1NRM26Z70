### Title
Front-Running via Ratifier-Data Reuse Exhausts All Bundle Offers, Causing Permanent DoS - (`src/periphery/MidnightBundles.sol`)

### Summary

`buyWithUnitsTargetAndWithdrawCollateral` computes `unitsToTake` as the minimum of the remaining target, the per-offer cap, and `ConsumableUnitsLib.consumableUnits`. When an attacker front-runs the victim's pending bundle transaction by calling `Midnight.take` directly with the same offer structs and `ratifierData` extracted from the mempool, every offer's `consumableUnits` drops to 0. The bundler then calls `take(..., 0, ...)` for each offer, which succeeds as a no-op, `filledUnits` never advances, and the final `require(filledUnits == targetUnits, OutOfOffers())` reverts the entire bundle.

**Root cause — two compounding gaps:**

1. **`ratifierData` is not taker-bound.** The `Offer` struct contains no taker field. `Midnight.take` only checks `IRatifier(offer.ratifier).isRatified(offer, ratifierData)` against the offer parameters. Any address that observes a valid `(offer, ratifierData)` pair in the mempool can reuse it as their own taker call. [1](#0-0) 

2. **A 0-unit take is a silent no-op.** When `consumableUnits` returns 0, `unitsToTake = 0`. `Midnight.take` with `units = 0` increments `consumed` by 0, passes the cap check, and returns `(0, 0)`. The try/catch succeeds, `filledUnits += 0`, and the loop continues without advancing. After all offers are processed, `filledUnits == 0 != targetUnits` and `OutOfOffers()` reverts. [2](#0-1) 

**Exploit flow:**

1. Victim broadcasts a bundle with `takes[]` referencing N sell offers, each with maker-signed `ratifierData`, targeting `targetUnits` total.
2. Attacker observes the pending transaction in the mempool and extracts all `(offer, ratifierData)` pairs.
3. For each offer `i`, attacker calls `Midnight.take(takes[i].offer, takes[i].ratifierData, takes[i].offer.maxUnits - consumed_i, attacker, ...)`, exhausting `consumed[maker][group]` to `maxUnits` for every offer. [3](#0-2) 
4. Victim's bundle executes. For each offer, `consumableUnits` returns `maxUnits.zeroFloorSub(consumed) = 0`. [4](#0-3) 
5. `unitsToTake = min(..., 0) = 0`. The inner `take(0)` succeeds silently. `filledUnits` stays 0.
6. `require(filledUnits == targetUnits, OutOfOffers())` reverts. [5](#0-4) 

**Existing protections that do NOT stop this:**

- `require(offer.maker != taker, SelfTake())`: attacker ≠ maker, passes.
- `require(taker == msg.sender || isAuthorized[taker][msg.sender])`: attacker is their own taker, passes.
- The `try/catch` in the bundler: it catches reverts from `take`, but a 0-unit take does not revert — it returns `(0, 0)` successfully.
- `require(newConsumed <= offer.maxUnits)`: after exhaustion, `consumed += 0` still satisfies the cap.

### Impact Explanation

Every bundle call by the victim that references a fixed set of offer hashes reverts with `OutOfOffers()`. The victim's loan tokens are pulled and then refunded (line 104), so no funds are lost, but the intended borrow/buy action is permanently blocked as long as the attacker repeats the front-run. The attacker acquires credit in the market (recoverable at maturity or by selling through a buy offer), making the net attacker cost gas-only in liquid markets. [6](#0-5) 

### Likelihood Explanation

**Preconditions:**
- The victim's bundle transaction must be visible in a public mempool (standard on Ethereum mainnet and most L2s).
- The attacker must hold enough loan tokens to fill the remaining capacity of each offer in the victim's `takes[]` array (or use a flash loan to acquire them, then hold credit).
- The offers must use a ratifier whose `isRatified` check does not bind the taker address (e.g., a signature ratifier that signs only the `Offer` struct fields, which contain no taker).

**Feasibility:** High on chains with public mempools. The attack is repeatable every time the victim resubmits. The attacker's capital is not destroyed — it becomes credit redeemable at maturity — so the attack can be sustained indefinitely at gas cost only.

### Recommendation

1. **Bind `ratifierData` to the taker at the ratifier level.** The ratifier should sign over `(offer, taker)` so that a signature extracted from the victim's mempool transaction cannot be reused by a different taker. This is the most robust fix.

2. **Replace the exact-equality check with a minimum-fill check in the bundler.** Change `require(filledUnits == targetUnits, OutOfOffers())` to `require(filledUnits >= targetUnits, OutOfOffers())` and cap `unitsToTake` so it never overshoots. This alone does not prevent the DoS if all offers are exhausted, but combined with a larger offer array it reduces the attack surface.

3. **Allow the victim to supply a surplus of offers.** Document that callers should include more offers than strictly needed so that partial front-running does not exhaust the entire array.
