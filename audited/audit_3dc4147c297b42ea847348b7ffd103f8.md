Audit Report

## Title
Oracle Price Manipulated via `onSell` Callback Allows Undercollateralized Borrowing - (File: src/Midnight.sol)

## Summary
In `take()`, the seller's debt is committed to storage before the `onSell` callback fires. Because the callback is a regular `CALL` (not a `staticcall`), it can write to any external contract. When `isHealthy` is subsequently called, it invokes `IOracle.price()` via `staticcall`, which prevents the oracle from writing state but does not prevent it from reading state already written by the callback. An attacker who controls both the oracle (via permissionless market creation) and the seller callback can inflate the oracle price during the callback window, causing `isHealthy` to pass with a fabricated price and leaving the seller with undercollateralized debt.

## Finding Description
**Verified code path in `src/Midnight.sol`:**

1. `take()` is called (line 337).
2. `sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease)` (line 414) — seller's debt is committed to storage before any callback.
3. Liquidation lock is set for the seller (line 444).
4. `onSell` callback fires (lines 458–474) via a regular `CALL`. The callback can write to any external contract, including an attacker-controlled `StateContract`.
5. Liquidation lock is released (line 475).
6. `require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable())` (line 476).
7. Inside `isHealthy`, `IOracle(collateralParam.oracle).price()` is called (line 953). Because `IOracle.price()` is declared `external view` (`src/interfaces/IOracle.sol:6`), Solidity issues a `staticcall`. The `staticcall` prevents the oracle from writing state during its own execution, but it does not prevent the oracle from reading state already written by the callback in step 4. A malicious oracle that reads `StateContract.value` returns the inflated value written by the callback.
8. `maxDebt` is computed with the inflated price (line 954–955) and exceeds the seller's actual debt — health check passes.

**Attacker-controlled inputs:**
- `touchMarket` is permissionless (line 755). The attacker deploys a `StateContract`, a `MaliciousOracle` reading `StateContract.value`, and a `MaliciousCallback` writing to `StateContract.value`. They call `touchMarket` with `CollateralParams.oracle = address(MaliciousOracle)`.
- Seller callback: when `offer.buy == true`, the seller is the taker and controls `takerCallback`; when `offer.buy == false`, the seller is `offer.maker` and controls `offer.callback` (lines 420–421).

**Why existing checks fail:**
- The liquidation lock (lines 444/475) only prevents liquidation during callbacks; it does not prevent oracle price manipulation.
- `staticcall` on `price()` prevents the oracle from writing state during the health check, but the callback already wrote the manipulated value before `isHealthy` is called.
- No snapshot of the oracle price is taken before the callback to compare against.
- The Certora formal verification in `certora/specs/Healthiness.spec` lines 14–16 explicitly acknowledges this gap: `// Assumption: price does not change during rules.` and models `price()` as a `persistent ghost` returning a fixed value per oracle address. This assumption is not enforced on-chain.

## Impact Explanation
The seller can borrow an arbitrary number of units backed by negligible collateral. After `take()` completes, the seller's position is undercollateralized at the true oracle price. The attacker then calls `withdrawCollateral` (line 568: `require(isHealthy(market, id, onBehalf), UnhealthyBorrower())`), which calls the same manipulable oracle while `StateContract.value` remains inflated, and withdraws all collateral. Once collateral is withdrawn, the attacker resets `StateContract.value` to zero, leaving the protocol with bad debt socialized to lenders via the loss factor mechanism (lines 631–634). This directly violates the invariant that every debt unit must be backed by sufficient collateral at the oracle price. `live_context.json` explicitly lists `"bad debt creation"` and `"callback or multicall state corruption"` as highest-priority bug classes. `SECURITY.md` explicitly states oracle manipulation attacks are in scope.

## Likelihood Explanation
All preconditions are fully attacker-controlled: the attacker deploys three contracts, creates a market (permissionless), and supplies minimal collateral. No privileged access is required. `live_context.json` explicitly lists `"market creator"` and `"callback receiver"` as valid attacker profiles. The attack is repeatable across any number of takes. The only external dependency is a victim lender providing a buy offer in the malicious market, which is realistic in a permissionless protocol where lenders may not verify oracle contracts.

## Recommendation
1. **Snapshot the oracle price before the callback**: Read and cache `IOracle(collateralParam.oracle).price()` for each activated collateral before the `onSell` callback fires, then use the cached values in the post-callback health check.
2. **Alternatively, call `isHealthy` before the callback** (after debt is committed) and again after, rejecting if the post-callback check fails with a stricter bound.
3. **Document the oracle trust assumption explicitly**: Warn market creators and lenders that oracle contracts must not read state writable by callbacks.

## Proof of Concept
```solidity
// StateContract: holds a manipulable value
contract StateContract {
    uint256 public value;
    function set(uint256 v) external { value = v; }
}

// MaliciousOracle: reads StateContract.value as price
contract MaliciousOracle {
    StateContract public sc;
    constructor(StateContract _sc) { sc = _sc; }
    function price() external view returns (uint256) { return sc.value(); }
}

// MaliciousCallback: inflates StateContract.value during onSell
contract MaliciousCallback {
    StateContract public sc;
    uint256 public inflatedPrice;
    constructor(StateContract _sc, uint256 _p) { sc = _sc; inflatedPrice = _p; }
    function onSell(...) external returns (bytes32) {
        sc.set(inflatedPrice); // written before isHealthy is called
        return CALLBACK_SUCCESS;
    }
}

// Attack sequence:
// 1. Deploy StateContract, MaliciousOracle(sc), MaliciousCallback(sc, 1e54)
// 2. touchMarket with CollateralParams.oracle = address(MaliciousOracle)
// 3. Supply 1 wei collateral
// 4. As taker on a buy offer, pass takerCallback = address(MaliciousCallback)
//    → take() commits large debt, fires onSell (sc.value = 1e54),
//      isHealthy reads inflated price → passes
// 5. Call withdrawCollateral (sc.value still 1e54) → passes isHealthy → drains collateral
// 6. sc.set(0) → position is undercollateralized → bad debt realized on lenders
```
