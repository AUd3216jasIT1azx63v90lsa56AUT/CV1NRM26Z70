Audit Report

## Title
Attacker-Controlled Reverting Oracle Permanently DoS-es Liquidation for Any Borrower with That Collateral Activated - (File: src/Midnight.sol)

## Summary
The `liquidate` function iterates every activated collateral in a borrower's `collateralBitmap` and makes a bare external call to `IOracle(_collateralParam.oracle).price()` with no `try/catch` or error handling. Because `touchMarket` is permissionless and performs no validation on the oracle address, an attacker can deploy a market with a malicious oracle, wait for a borrower to activate that collateral, then permanently revert the oracle — blocking all liquidation attempts for that borrower. The same oracle loop in `isHealthy` also blocks `withdrawCollateral`, making self-rescue equally impossible.

## Finding Description
**Root cause:** `src/Midnight.sol` line 610 makes a bare external call inside the collateral bitmap loop with no error handling:

```solidity
uint256 price = IOracle(_collateralParam.oracle).price(); // no try/catch
```

The `NotLiquidatable` check at lines 620–624 is never reached if this call reverts.

**Permissionless market creation:** `touchMarket` (lines 755–791) validates only collateral token ordering, allowed LLTV tiers, and `maxLif` values. There is no validation of `collateralParams[i].oracle` — any address, including an attacker-controlled contract, is accepted.

**Self-rescue also blocked:** `isHealthy` (lines 949–957) contains the identical bare oracle call loop. Since `withdrawCollateral` calls `isHealthy` when the borrower has debt, the borrower cannot remove the malicious collateral to escape the DoS.

**Certora confirmation:** The formal verification rule `oracleRevertCausesLiquidateRevert` in `certora/specs/Reverts.spec` lines 182–193 explicitly proves this revert propagation as a verified property, confirming the path is real and unguarded.

**Exploit flow:**
1. Attacker deploys `MaliciousOracle` with a `shouldRevert` toggle; initially `price()` returns a valid value.
2. Attacker calls `touchMarket(market)` with `MaliciousOracle` as the oracle for collateral slot `k`.
3. Borrower calls `supplyCollateral(market, k, amount, borrower)`, setting bit `k` in `_position.collateralBitmap`.
4. Borrower takes on debt, making the position liquidatable when prices move.
5. Attacker calls `MaliciousOracle.setRevert(true)`.
6. All calls to `liquidate(market, ...)` for this borrower revert unconditionally at line 610.
7. All calls to `withdrawCollateral` for this borrower (with debt) also revert via `isHealthy` at line 953.

**Why existing checks fail:**
- `liquidatorGate` check (lines 597–600) runs before the loop and does not protect against oracle reverts.
- `liquidationLocked` / `NotLiquidatable` checks (lines 620–624) are after the loop and are never reached.

## Impact Explanation
Any borrower who has activated a collateral whose oracle is attacker-controlled is permanently unliquidatable and unable to self-rescue. The position accumulates bad debt that can never be realized, lenders cannot recover funds through liquidation, and the core protocol invariant — unhealthy positions must remain liquidatable — is broken for the affected borrower for the lifetime of the market. This constitutes a permanent freeze of user funds and protocol insolvency risk.

## Likelihood Explanation
Market creation is permissionless with no oracle validation, so the attacker only needs to deploy a contract implementing `IOracle` and call `touchMarket`. The borrower must supply collateral to that market slot, which is realistic in a permissionless ecosystem where borrowers may not audit oracle bytecode. The toggle can be flipped at any time after the borrower activates the collateral, including atomically in the same block the position becomes unhealthy. Cost is minimal: one oracle deployment and one `touchMarket` call.

## Recommendation
Wrap the oracle call in a `try/catch` block inside the bitmap loop in both `liquidate` and `isHealthy`. On revert, either skip the collateral (treating its price as zero for liquidation purposes) or treat the position as liquidatable by default. Additionally, consider adding an oracle validation step in `touchMarket` (e.g., a liveness check calling `price()` at creation time), though this alone would not prevent a toggle-based attack after creation.

## Proof of Concept
1. Deploy `MaliciousOracle` with `price()` returning `1e36` initially and a `stopOracle()` function that makes `price()` revert.
2. Call `touchMarket` with `MaliciousOracle` as the oracle for collateral index 0.
3. Call `supplyCollateral` for a borrower on that market, activating bit 0 in `collateralBitmap`.
4. Have the borrower take on debt (via `take`) to create a liquidatable position.
5. Call `MaliciousOracle.stopOracle()`.
6. Attempt `liquidate(market, 0, 0, repaidUnits, borrower, false, receiver, address(0), "")` — observe it reverts.
7. Attempt `withdrawCollateral(market, 0, amount, borrower, borrower)` as the borrower — observe it also reverts via `isHealthy`.

This mirrors the existing test pattern in `test/OtherFunctionsTest.sol` lines 312–333 (`testSupplyCollateralDoesNotCallOracle`) which already uses a `RevertingOracle` — the same setup applied to `liquidate` instead of `supplyCollateral` will reproduce the DoS.
