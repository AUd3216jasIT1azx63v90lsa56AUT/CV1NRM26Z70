Audit Report

## Title
`sellerAssetsToUnits` arithmetic revert at `offerPrice == settlementFee` blocks `supplyCollateralAndSellWithAssetsTarget` while `Midnight.take()` succeeds - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary

When a buy offer's tick price exactly equals the post-maturity settlement fee, `sellerPrice` computes to zero in `TakeAmountsLib.sellerAssetsToUnits`. The subsequent `mulDivUp(targetSellerAssets, WAD, 0)` call triggers an arithmetic underflow revert in Solidity 0.8+ (via the `d - 1` term), while `Midnight.take()` handles the same `sellerPrice = 0` case without reverting (producing `sellerAssets = 0`). Because `sellerAssetsToUnits` is called outside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, the entire bundler call reverts, blocking the assets-target sell path even when the underlying take would succeed.

## Finding Description

**Root cause — `TakeAmountsLib.sellerAssetsToUnits` (lines 44–46):** [1](#0-0) 

When `offerPrice == settlementFee`, `sellerPrice = 0`. `mulDivUp` is implemented as: [2](#0-1) 

With `d = 0`, the `d - 1` term underflows (Solidity 0.8 arithmetic revert), confirmed by `testMulDivUpDivisionByZero`: [3](#0-2) 

**`Midnight.take()` does not revert in the same case (lines 361–364):** [4](#0-3) 

`mulDivDown(units, 0, WAD) = (units * 0) / WAD = 0` — no division by zero.

**The comment in `sellerAssetsToUnits` (line 34) is inaccurate:** [5](#0-4) 

The comment claims "midnight reverts too" for `offerPrice < settlementFee`, but this is only true for strict inequality (underflow). When `offerPrice == settlementFee`, Midnight does **not** revert, but `sellerAssetsToUnits` does.

**Call site in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` (lines 285–300):** [6](#0-5) 

The `try/catch` wraps only `Midnight.take()`. A revert in `sellerAssetsToUnits` propagates uncaught, aborting the entire bundler call.

**Preconditions for the boundary condition:**

Tick prices are rounded to multiples of `PRICE_ROUNDING_STEP = 1e12 = CBP`: [7](#0-6) [8](#0-7) 

Settlement fees are also multiples of `CBP`: [9](#0-8) 

`MAX_SETTLEMENT_FEE_0_DAYS = 14e12`, so `settlementFeeCbp0` ranges from 0 to 14. Since tick prices are dense multiples of `1e12` in the low-price range, there exist valid ticks where `tickToPrice(tick) == settlementFeeCbp0 * CBP`. The fee setter can also raise the settlement fee post-offer-creation via `setMarketSettlementFee`: [10](#0-9) 

**Existing protections are insufficient:**

The `NoDivisionByZero.spec` Certora spec covers only `Midnight.sol`, not `TakeAmountsLib.sol`. The bundler documents "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts" (line 246), but this is documentation of the symptom, not a protection: [11](#0-10) 

## Impact Explanation

Any taker calling `supplyCollateralAndSellWithAssetsTarget` whose `takes` array contains a buy offer at the settlement-fee price boundary will have the entire call revert with an arithmetic error. The taker cannot use the atomic collateral-supply-and-sell bundler path to unwind at maturity, even though `Midnight.take()` would succeed. The taker must fall back to calling `Midnight.take()` directly, losing the atomic multi-offer aggregation the bundler provides. There is no loss of funds, but there is a concrete functional DoS of the bundler path.

## Likelihood Explanation

The condition requires `tickToPrice(offer.tick) == settlementFeeCbp0 * CBP`. Both quantities are multiples of `CBP = 1e12`, and tick prices are dense in the low-price range (up to `14e12`), making alignment achievable. Two independent paths trigger this:

1. An offer maker creates a buy offer at a tick whose price equals the current settlement fee — no admin action required.
2. The fee setter raises the settlement fee to match an existing offer's tick price — privileged but documented protocol behavior ("if the settlement fee gets increased, it might implicitly cancel offers with very low price").

The condition is repeatable across any market where `settlementFeeCbp0 * CBP` aligns with a tick price.

## Recommendation

Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case to match `Midnight.take()`'s behavior. When `sellerPrice == 0`, the forward direction yields `sellerAssets = 0` for any `units`, so the inverse should return `type(uint256).max` (or 0 if the intent is to skip the offer) rather than reverting:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
if (offer.buy && sellerPrice == 0) return type(uint256).max; // no units yield targetSellerAssets > 0
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

Also update the NatSpec comment on line 34 to accurately state that `sellerAssetsToUnits` reverts when `offerPrice == settlementFee` but `Midnight.take()` does not.

## Proof of Concept

1. Deploy `Midnight` and `MidnightBundles`.
2. Set `settlementFeeCbp0 = 1` (fee = `1e12`).
3. Find tick `T` such that `tickToPrice(T) == 1e12` (e.g., the highest tick, `MAX_TICK = 5820`, yields the lowest price).
4. Create a buy offer at tick `T`.
5. Warp to `block.timestamp >= offer.market.maturity` so `timeToMaturity = 0` and `settlementFee(id, 0) = 1e12`.
6. Call `supplyCollateralAndSellWithAssetsTarget` with `targetSellerAssets = 1` and this offer in `takes`.
7. Observe: the call reverts with `stdError.arithmeticError` from `mulDivUp(..., 0)` in `sellerAssetsToUnits`.
8. Call `Midnight.take()` directly with the same offer and `units = 1`.
9. Observe: succeeds, returning `sellerAssets = 0`.
