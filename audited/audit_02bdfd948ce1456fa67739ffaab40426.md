### Title
Lenders Can Front-Run Bad Debt Realization to Avoid Proportional Loss — (File: src/Midnight.sol)

---

### Summary

In Morpho Midnight, bad debt is socialized by updating the global `lossFactor` exclusively inside `liquidate()`. The `withdraw()` function applies only the *current* `lossFactor` at the moment of the call. A lender who monitors the chain for an unhealthy borrower and calls `withdraw()` before any liquidator calls `liquidate()` exits at full credit value, shifting their share of the loss entirely onto lenders who act later.

---

### Finding Description

**Vulnerability class**: Accounting / state-transition race (unequal loss distribution).

**Root cause — two-step loss realization**

Loss socialization is a two-step process:

1. **Step 1 — `liquidate()` updates `lossFactor`** (the only place this happens): [1](#0-0) 

```solidity
_marketState.lossFactor = UtilsLib.toUint128(
    type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
);
_marketState.totalUnits -= UtilsLib.toUint128(badDebt);
```

2. **Step 2 — `_updatePosition()` applies the current `lossFactor` to slash a lender's credit** (lazy, per-user): [2](#0-1) 

```solidity
uint256 postSlashCredit = _lastLossFactor < type(uint128).max
    ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
    : 0;
```

If `marketState[id].lossFactor == position[id][user].lastLossFactor` (i.e., no bad debt has been realized since the lender's last interaction), the formula reduces to `credit * 1 = credit` — **no slashing**.

**`withdraw()` applies only the current (pre-liquidation) `lossFactor`**: [3](#0-2) 

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    ...
    _updatePosition(market, id, onBehalf);   // applies current lossFactor — not yet updated
    ...
    _position.credit    -= UtilsLib.toUint128(units);
    _marketState.withdrawable -= UtilsLib.toUint128(units);
    _marketState.totalUnits   -= UtilsLib.toUint128(units);
    SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
}
```

There is no check that prevents a lender from withdrawing while unrealized bad debt exists in the market.

**Exploit path**

| Step | Actor | Action |
|------|-------|--------|
| 1 | — | Borrower's collateral value drops below debt; bad debt is visible on-chain via oracle prices |
| 2 | Attacker (lender) | Calls `withdraw(market, myCredit, self, self)` before any liquidator acts |
| 3 | — | `_updatePosition()` sees `lossFactor` unchanged → no slashing → attacker receives full `myCredit` tokens |
| 4 | Liquidator | Calls `liquidate()` → `lossFactor` is updated, `totalUnits` shrinks |
| 5 | Remaining lenders | Bear the full bad debt loss, distributed over a now-smaller `totalUnits` |

**Prerequisite**: `marketState[id].withdrawable >= units`. This is satisfied whenever any borrower has previously repaid (via `repay()`, which increments `withdrawable`), which is the normal operating state of an active market.

The protocol's own comment acknowledges the withdrawal race but only in the context of *post-repayment / post-liquidation* scenarios: [4](#0-3) 

> "When some assets become withdrawable before maturity (after a repayment or a liquidation), there is an incentive to take resting sell offers with price < 1 and withdraw instantly."

The pre-liquidation front-run is not addressed.

---

### Impact Explanation

A lender who front-runs bad debt realization withdraws at 1:1 (credit → loan token) and pays zero share of the loss. Every unit of credit they remove before `liquidate()` is called increases the loss-per-unit borne by remaining lenders:

- **Without front-run**: loss spread over `T` total units → each lender loses `badDebt / T` per unit.
- **With front-run** (attacker withdraws `W` units first): loss spread over `T - W` units → each remaining lender loses `badDebt / (T - W)` per unit.

The attacker's gain is exactly `W * badDebt / T` loan tokens at the expense of other lenders. In a market with significant bad debt and available `withdrawable` liquidity, this is a direct, measurable theft of value from slower lenders.

---

### Likelihood Explanation

- **No privileged access required**: any lender can call `withdraw()`.
- **Observable trigger**: bad debt is detectable off-chain by querying oracle prices and comparing to `position.debt` — standard MEV bot territory.
- **Condition on `withdrawable`**: active markets routinely have repaid debt, so `withdrawable > 0` is the common case, not an edge case.
- **Timing window**: the window between bad debt becoming visible on-chain and a liquidator calling `liquidate()` is non-zero and exploitable via standard front-running (higher gas price or private mempool).

---

### Recommendation

Realize bad debt eagerly rather than lazily. One approach: add a permissionless `realizeBadDebt(market, borrower)` function that updates `lossFactor` without requiring a liquidation, and/or call it atomically at the start of `withdraw()` for any borrower whose position is already in bad debt. Alternatively, track a "pending bad debt" accumulator that is deducted from `withdrawable` before any withdrawal is processed, ensuring lenders cannot exit ahead of a known loss.

### Proof of Concept

**Setup**

- Market with loan token `T`.
- 10 lenders, each with `credit = 1000` → `totalUnits = 10_000`.
- 5 borrowers have repaid → `withdrawable = 5_000`.
- Borrower B has `debt = 2_000`, collateral worth `500` at `maxLif` → `badDebt = 1_500` (unrealized).

**Attack**

1. Attacker (Lender 1) observes oracle price drop making Borrower B's position bad-debt.
2. Attacker calls `withdraw(market, 1000, lender1, lender1)`.
   - `_updatePosition`: `lossFactor` unchanged → `postSlashCredit = 1000` → no slashing.
   - `credit[lender1] -= 1000`, `withdrawable -= 1000`, `totalUnits -= 1000`.
   - Attacker receives **1000 T** tokens.
3. Liquidator calls `liquidate(market, ..., borrowerB, ...)`.
   - `badDebt = 1_500`, `totalUnits = 9_000`.
   - `lossFactor` updated: `MAX - (MAX - 0) * (9000 - 1500) / 9000 = MAX * 1500/9000`.
   - `totalUnits = 7_500`.
4. Each remaining lender's credit is slashed by `1500/9000 ≈ 16.7%` at their next interaction.
   - Each of the 9 remaining lenders loses `~167` units.

**Result**: Attacker avoided a `1000 * 1500/10000 = 150 T` loss. The 9 remaining lenders each bear `~167 T` loss instead of `~150 T` — a `~11%` excess loss caused by the attacker's exit.
