### Title
Validator-Manipulable `block.timestamp` in Post-Maturity LIF Calculation Allows Excess Collateral Seizure — (File: src/Midnight.sol)

---

### Summary

The `liquidate` function in `Midnight.sol` computes the post-maturity Liquidation Incentive Factor (LIF) using `block.timestamp` directly. Because `TIME_TO_MAX_LIF` is only 15 minutes, even a small validator-controlled timestamp shift represents a meaningful fraction of the full LIF range, allowing a validator-liquidator to seize more collateral from a borrower than the protocol intends.

---

### Finding Description

In `liquidate`, when `postMaturityMode == true`, the LIF is computed as:

```solidity
uint256 lif = postMaturityMode
    ? UtilsLib.min(_maxLif, WAD + (_maxLif - WAD) * (block.timestamp - market.maturity) / TIME_TO_MAX_LIF)
    : _maxLif;
``` [1](#0-0) 

`TIME_TO_MAX_LIF` is defined as:

```solidity
uint256 constant TIME_TO_MAX_LIF = 15 minutes;
``` [2](#0-1) 

The LIF linearly scales from `WAD` (1.0) at maturity to `maxLif` over exactly 900 seconds. A validator proposing a block can set `block.timestamp` up to ~12 seconds ahead of the previous block's timestamp (per Ethereum consensus rules, and potentially more on other EVM chains). This shifts the LIF upward by:

```
ΔLIF = (maxLif - WAD) × Δt / 900
```

The `maxLif` value depends on LLTV and liquidation cursor. For the lowest allowed LLTV (`LLTV_0 = 0.385e18`) with `LIQUIDATION_CURSOR_HIGH = 0.5e18`:

```
maxLif = WAD / (WAD - 0.5 × (WAD - 0.385e18)) ≈ 1.444e18
maxLif - WAD ≈ 0.444e18
``` [3](#0-2) 

A 12-second manipulation yields `ΔLIF ≈ 0.444e18 × 12/900 ≈ 0.00592e18` (~0.59% extra collateral seized per liquidation). On a $10M position this is ~$59,200 of excess collateral extracted from the borrower.

The seized collateral is computed from the inflated LIF:

```solidity
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
``` [4](#0-3) 

A higher `lif` directly increases `seizedAssets` for the same `repaidUnits`, transferring excess collateral to the liquidator at the borrower's expense.

---

### Impact Explanation

A validator-liquidator seizes more collateral than the protocol formula intends for a given repayment amount. The excess comes directly from the borrower's collateral balance. For large positions in low-LLTV markets, the dollar value of the excess can be substantial (e.g., ~$59K per $10M position per liquidation event). The borrower has no recourse once the transaction is included.

---

### Likelihood Explanation

The attacker must be an active Ethereum validator (or validator on any EVM chain where Midnight is deployed) and simultaneously operate a liquidation bot. This is a realistic profile: MEV-aware validator operators routinely combine block proposal rights with on-chain arbitrage/liquidation bots. No privileged protocol keys are required. The attack window is every post-maturity block the attacker proposes while a liquidatable position exists.

---

### Recommendation

Replace the direct `block.timestamp` dependency in the LIF ramp with a committed, manipulation-resistant timestamp. Options include:

1. **Snap to block boundaries**: Use `block.timestamp` only at coarse granularity (e.g., round down to the nearest minute), reducing the attacker's precision.
2. **Increase `TIME_TO_MAX_LIF`**: A longer ramp (e.g., 1 hour instead of 15 minutes) reduces the per-second LIF delta, making timestamp manipulation less profitable.
3. **Use a TWAP or committed timestamp**: Record the maturity-crossing timestamp on-chain at the first post-maturity interaction and use that fixed value for all subsequent LIF calculations, eliminating per-block manipulation entirely.

---

### Proof of Concept

**Setup:**
- Market with `loanToken = USDC`, `LLTV = 0.385e18`, `maxLif ≈ 1.444e18`, maturity = T.
- Borrower has 1,000,000 USDC worth of collateral and outstanding debt.
- Attacker is a validator and liquidation bot operator.

**Steps:**

1. Market maturity passes at timestamp T.
2. At T + 450 seconds (halfway through `TIME_TO_MAX_LIF`), the attacker's validator is selected to propose a block.
3. Honest LIF at T+450: `WAD + 0.444e18 × 450/900 = 1.222e18`
4. Attacker sets `block.timestamp = T + 462` (12 seconds ahead).
5. Manipulated LIF: `WAD + 0.444e18 × 462/900 = 1.2279e18`
6. Attacker calls `liquidate(..., repaidUnits = X, ...)`.
7. Honest `seizedAssets = X × 1.222e18 / WAD × ORACLE_PRICE_SCALE / price`
8. Manipulated `seizedAssets = X × 1.2279e18 / WAD × ORACLE_PRICE_SCALE / price`
9. **Extra collateral seized ≈ 0.59% of the liquidated collateral value**, extracted from the borrower with no protocol-level check preventing it. [5](#0-4) [2](#0-1)
