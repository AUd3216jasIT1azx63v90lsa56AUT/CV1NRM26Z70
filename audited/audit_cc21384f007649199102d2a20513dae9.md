Audit Report

## Title
Authorized caller can redirect all sell-side loan proceeds to an arbitrary address while debt accrues to the taker - (File: src/periphery/MidnightBundles.sol)

## Summary

`supplyCollateralAndSellWithUnitsTarget` and `supplyCollateralAndSellWithAssetsTarget` accept a caller-controlled `receiver` parameter that is never validated against `taker`. The authorization gate at line 127 only verifies the caller may act on the taker's behalf; it does not restrict where the resulting loan proceeds are sent. An attacker who holds `isAuthorized[victim][attacker] == true` can borrow against the victim's collateral and route 100% of the loan proceeds to themselves, leaving the victim with increased debt and no assets.

## Finding Description

**Exact code path:**

**Step 1 — Bundler authorization check** (`src/periphery/MidnightBundles.sol`, line 127):
```solidity
require(taker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(taker, msg.sender), Unauthorized());
```
Attacker passes `taker = victim`, `msg.sender = attacker`. Precondition `isAuthorized[victim][attacker] == true` satisfies this check. The `receiver` parameter (line 121) is accepted from the caller with no constraint that it equals `taker`.

**Step 2 — Take call** (lines 152–154):
```solidity
IMidnight(MIDNIGHT).take(
    takes[i].offer, takes[i].ratifierData, unitsToTake, taker, address(this), address(0), ""
)
```
`taker = victim`, `receiverIfTakerIsSeller = address(bundler)`.

**Step 3 — Midnight.take authorization** (`src/Midnight.sol`, line 346):
```solidity
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());
```
`msg.sender = bundler`. Requires `isAuthorized[victim][bundler] == true` (second precondition — normal operating condition for any bundler user).

**Step 4 — Seller identity and debt assignment** (lines 375, 414):
```solidity
(address buyer, address seller) = offer.buy ? (offer.maker, taker) : (taker, offer.maker);
// ...
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
```
Since `offer.buy == true` (enforced by bundler line 145), `seller = taker = victim`. Victim's debt grows.

**Step 5 — Receiver resolved inside Midnight.take** (line 423):
```solidity
address receiver = offer.buy ? receiverIfTakerIsSeller : offer.receiverIfMakerIsSeller;
```
`offer.buy == true`, so `receiver = address(bundler)`.

**Step 6 — sellerAssets transferred to bundler** (line 456):
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```
Loan proceeds land in the bundler contract.

**Step 7 — Health check** (line 476):
```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```
Passes because victim has pre-existing collateral.

**Step 8 — Bundler forwards proceeds to attacker-controlled `receiver`** (line 168):
```solidity
SafeTransferLib.safeTransfer(loanToken, receiver, filledSellerAssets - referralFeeAssets);
```
`receiver` is the raw function parameter set by the attacker to their own address. No check constrains it to `taker`.

**Root cause:** The `receiver` parameter is never validated against `taker`. The `isAuthorized` gate grants the right to *initiate* actions on behalf of the taker; it does not grant the right to *redirect the taker's loan proceeds* to an arbitrary third party. The identical flaw exists in `supplyCollateralAndSellWithAssetsTarget` at line 307:
```solidity
SafeTransferLib.safeTransfer(loanToken, receiver, targetSellerAssets);
```

**Why existing checks fail:** The authorization check at line 127 is a binary gate — it either passes or reverts. Once it passes, no subsequent check in the bundler constrains the `receiver` address. The health check in `Midnight.take` (line 476) only verifies the victim's collateral covers their new debt; it does not verify that proceeds went to the correct party.

## Impact Explanation

The victim's debt increases by the taken units and their collateral remains locked as backing. All loan proceeds (`filledSellerAssets`) are transferred to the attacker. This constitutes unauthorized borrowing with complete redirection of loan proceeds — the victim is left with debt and no assets. This is a direct theft of value from the victim, enabled by a protocol-level contract (the bundler) that fails to enforce that proceeds belong to the taker.

## Likelihood Explanation

**Required preconditions:**
1. `isAuthorized[victim][bundler] == true` — this is the normal operating precondition for any user of the bundler; without it, the bundler is unusable.
2. `isAuthorized[victim][attacker] == true` — the victim must have authorized the attacker on Midnight. This is realistic: users commonly authorize portfolio manager contracts, keeper bots, or trusted EOAs for position management. The Midnight authorization model is explicitly designed to support this pattern (see `Midnight.sol` lines 101–110).
3. Victim has sufficient pre-existing collateral in Midnight for the health check to pass.

All three conditions are realistic in normal protocol usage. The attack is repeatable as long as the authorization remains and collateral is present. The attacker does not need to supply any capital.

## Recommendation

In both `supplyCollateralAndSellWithUnitsTarget` and `supplyCollateralAndSellWithAssetsTarget`, enforce that the `receiver` of loan proceeds is the `taker`:

```solidity
// Option A: remove the receiver parameter entirely and always send to taker
SafeTransferLib.safeTransfer(loanToken, taker, filledSellerAssets - referralFeeAssets);

// Option B: validate the parameter
require(receiver == taker, InvalidReceiver());
```

Option A is preferable as it eliminates the attack surface entirely. The `receiver` parameter provides no legitimate use case that `taker` does not already cover — the taker is the party whose collateral is being used and whose debt is increasing, so they are the correct and only valid recipient of the proceeds.

## Proof of Concept

**Setup:**
1. Deploy Midnight and MidnightBundles. Create a market with a buy offer from `maker` at a valid tick.
2. `victim` supplies collateral to Midnight sufficient to back the target borrow.
3. `victim` calls `Midnight.setIsAuthorized(bundler, true, victim)` — normal bundler onboarding.
4. `victim` calls `Midnight.setIsAuthorized(attacker, true, victim)` — victim has authorized attacker as a keeper/manager.

**Attack:**
```solidity
// attacker calls:
MidnightBundles(bundler).supplyCollateralAndSellWithUnitsTarget(
    targetUnits,
    0,                  // minSellerAssets = 0 (no slippage protection needed)
    victim,             // taker = victim
    attacker,           // receiver = attacker's address  <-- exploit
    new CollateralSupply[](0),  // no collateral supplied; victim already has it
    takes,              // valid buy offer from maker
    0,
    address(0)
);
```

**Expected result:**
- `victim.debt` increases by `targetUnits`
- `attacker` receives `filledSellerAssets` of loan token
- `victim` receives nothing

**Verification:** Assert `IERC20(loanToken).balanceOf(attacker) == filledSellerAssets` and `Midnight.debtOf(id, victim) == targetUnits` after the call.
