### Title
Reentrant `supplyCollateral` during `onLiquidate` callback over-realizes bad debt, slashing lenders beyond actual undercollateralization - (File: src/Midnight.sol)

### Summary
In `liquidate()`, bad debt is computed and fully realized against lenders before the `onLiquidate` callback fires. Because there is no reentrancy guard and `supplyCollateral` only checks `isAuthorized`, a callback contract authorized by the borrower can supply new collateral for the borrower during the callback window, after bad debt has already been socialized. The new collateral is never counted in the `badDebt` computation, causing lenders to be slashed by more than the actual undercollateralization.

### Finding Description
The exact execution order in `liquidate()` is:

1. **Line 606** — bitmap snapshot taken: `uint128 _collateralBitmap = _position.collateralBitmap;`
2. **Lines 607–618** — `maxDebt` and `badDebt` computed from that snapshot, reading live `_position.collateral[i]` values.
3. **Lines 626–641** — bad debt **fully realized**: `_position.debt -= uint128(badDebt)`, `lossFactor` updated, `totalUnits` reduced. This is irreversible.
4. **Lines 643–677** — collateral seized, `repaidUnits` deducted from debt.
5. **Line 696** — seized collateral transferred to receiver.
6. **Lines 698–715** — `onLiquidate` callback fires. **No reentrancy guard exists anywhere in `Midnight.sol`** (confirmed: no `nonReentrant`, no transient lock for `liquidate`). The `liquidationLocked` transient mechanism at line 444 is set only inside `take`, not `liquidate`.
7. **Line 717** — loan tokens pulled from payer.

During step 6, the callback contract calls:
```solidity
IMidnight(midnight).supplyCollateral(market, newIndex, largeAmount, borrower);
```

`supplyCollateral` (lines 524–546) checks only:
```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
```

If `isAuthorized[borrower][callbackContract] == true`, this passes. The call sets a new bit in `_position.collateralBitmap` and increases `_position.collateral[newIndex]`. This new collateral was **not present** when `badDebt` was computed at step 2, and the bad debt realization at step 3 is already committed to storage.

**Attacker-controlled inputs:**
- Borrower pre-authorizes the callback contract: `setIsAuthorized(callbackContract, true, borrower)` (borrower calls this themselves).
- Callback contract is pre-funded with collateral tokens and has approved Midnight.
- Borrower (or any allowed liquidator) calls `liquidate(..., callback=callbackContract, ...)`.
- `callbackContract.onLiquidate(...)` calls `supplyCollateral(market, newIndex, largeAmount, borrower)`.

No privileged role is required. The borrower controls all preconditions.

### Impact Explanation
Bad debt is over-realized. Lenders' `lossFactor` is updated (line 631–634) and `totalUnits` is reduced (line 634) based on a `badDebt` value that does not account for the collateral supplied during the callback. If the new collateral's value (at `maxLif`) equals or exceeds the computed `badDebt`, lenders are slashed for a loss that did not actually exist at the time the transaction completed. The borrower's debt is permanently reduced by `badDebt` while they retain the newly supplied collateral, netting a direct economic gain at lenders' expense.

Concrete example: debt = 100, pre-callback collateral value (at `maxLif`) = 60 → `badDebt` = 40. Lenders slashed by 40. During callback, borrower supplies collateral worth 55 (at `maxLif`). Post-transaction: debt = 60, total collateral value = 115. Actual undercollateralization = 0. Lenders were slashed 40 units for a position that was fully collateralized by the end of the transaction.

### Likelihood Explanation
Preconditions are entirely borrower-controlled: authorization of the callback contract, pre-funding it with collateral tokens, and calling `liquidate` (self-liquidation is not blocked). The attack is repeatable as long as the borrower can manufacture a bad-debt state (e.g., by withdrawing collateral just below the bad-debt threshold before self-liquidating). It requires a single atomic transaction and no external cooperation.

### Recommendation
Snapshot the collateral bitmap and all collateral balances used in the `badDebt` computation into memory-only variables before any external call, and do not allow those values to be changed between computation and realization. The simplest fix is to move the `onLiquidate` callback invocation to **before** the bad-debt realization block (lines 626–641), or alternatively add a transient reentrancy lock (analogous to `LIQUIDATION_LOCK_SLOT` used in `take`) that is set at the entry of `liquidate` and checked in `supplyCollateral` and `withdrawCollateral` to revert if called while a liquidation is in progress for the same borrower.
