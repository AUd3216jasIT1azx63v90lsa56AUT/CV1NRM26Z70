### Title
Zero-asset rounding allows unbounded offer reuse with no consumed increment and no token transfer — (`src/Midnight.sol`, `src/libraries/UtilsLib.sol`)

### Summary

When `offer.maxAssets > 0` and `offer.buy == true`, the `consumed` mapping is incremented by `buyerAssets` rather than `units`. If `units` is chosen small enough that `units * buyerPrice < WAD`, `mulDivDown` returns zero, so `consumed` never advances. The offer can be taken indefinitely with zero token transfer while still mutating buyer credit and seller debt on every call, breaking the solvency invariant.

### Finding Description

**Exact code path — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    //                                                              ^^^^^^^^^^^ += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // passes unchanged
}
```

`mulDivDown` in `UtilsLib.sol` is `(x * y) / d` (integer division). When `units * buyerPrice < WAD`, the result is 0.

For a buy offer, `buyerPrice = offerPrice` (the settlement fee cancels: `sellerPrice = offerPrice − fee`, `buyerPrice = sellerPrice + fee`). So the condition is simply `units * offerPrice < WAD`. At a low tick (small `offerPrice`, e.g. `1e9`), any `units < WAD / offerPrice = 1e9` satisfies this.

**Attacker-controlled inputs:**
- `offer.maxAssets > 0`, `offer.buy = true` (maker-controlled, but any such offer is vulnerable)
- `units = U` where `U * offerPrice < WAD` (taker-controlled)

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = M > 0` at a low tick.
2. Taker calls `take(offer, ..., units=U)` with `U` small enough that `buyerAssets = 0`.
3. Line 368: `consumed += 0` → `consumed` unchanged.
4. Line 369: `require(consumed <= M)` passes (consumed never moved).
5. Lines 382–414: position state mutates — `buyerPos.credit += U` (if buyer has no debt), `sellerPos.debt += U` (if seller has no credit).
6. Lines 455–456: `safeTransferFrom(..., 0)` and `safeTransferFrom(..., 0)` — zero-amount ERC-20 transfers succeed on standard tokens.
7. Step 2–6 repeats indefinitely; `consumed` never reaches `M`.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` only enforces an upper bound; it does not require that `consumed` actually increased.
- There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in `take`.
- The `maxUnits` branch (line 371) correctly uses `+= units`, so it is immune; only the `maxAssets` branch is affected.

### Impact Explanation

Every call with `units = U > 0` and `buyerAssets = 0`:
- Grants the buyer (maker) `U` units of credit with zero token payment.
- Burdens the seller (taker) with `U` units of debt with zero token receipt.
- Leaves `consumed` unchanged, so the offer is never exhausted.

Repeated indefinitely, this creates unbounded credit for the maker and unbounded debt for the taker without any token backing, directly violating the solvency invariant: the contract's loan-token balance does not cover the credit it has issued.

### Likelihood Explanation

**Preconditions:**
- A buy offer with `maxAssets > 0` must exist (common configuration).
- The offer's tick must be low enough that `offerPrice < WAD` (always true) and `units * offerPrice < WAD` for some reachable `units` value (true for any tick where `offerPrice < WAD`, i.e., every valid tick below the maximum).

**Feasibility:** Any unprivileged taker can compute the required `units` off-chain as `U = floor((WAD − 1) / offerPrice)` and call `take` directly. No special permissions, oracle manipulation, or token owner cooperation is required.

**Repeatability:** Unlimited — `consumed` never advances, so the offer never closes.

### Recommendation

In the `maxAssets` branch, require that the consumed increment is non-zero whenever `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 consumedDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || consumedDelta > 0, ZeroAssetTake()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += consumedDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` such that the resulting asset amount is always at least 1, or switch the `maxAssets` branch to also track `units` internally (as the `maxUnits` branch does) and convert only for the cap comparison.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Offer} from "src/Midnight.sol";
import {TickLib} from "src/libraries/TickLib.sol";
import {WAD} from "src/libraries/ConstantsLib.sol";

contract ZeroAssetTakeTest is Test {
    // Setup: deploy Midnight, create market, fund maker and taker.
    // Use a low tick so offerPrice is small (e.g. offerPrice = 1e9).
    // Set settlementFee = 0 for simplicity.

    function testOfferReuseViaZeroAssetTake() public {
        // offerPrice at MIN_TICK ≈ 1e9 (illustrative)
        uint256 offerPrice = TickLib.tickToPrice(MIN_TICK);
        // Choose units such that units * offerPrice < WAD → buyerAssets = 0
        uint256 units = (WAD - 1) / offerPrice; // guaranteed to give buyerAssets = 0

        Offer memory offer = Offer({
            buy: true,
            maxAssets: 1e18,   // non-zero maxAssets mode
            maxUnits: 0,
            tick: MIN_TICK,
            // ... other fields
        });

        uint256 consumedBefore = midnight.consumed(offer.maker, offer.group);

        // First take: should succeed (no revert expected currently)
        (uint256 buyerAssets, uint256 sellerAssets) = midnight.take(offer, "", units, taker, receiver, address(0), "");

        // Assert both assets are zero (rounding)
        assertEq(buyerAssets, 0, "buyerAssets must be 0");
        assertEq(sellerAssets, 0, "sellerAssets must be 0");

        // Assert consumed did NOT change — the bug
        assertEq(midnight.consumed(offer.maker, offer.group), consumedBefore, "consumed must not change");

        // Assert the take can be repeated (offer reuse)
        midnight.take(offer, "", units, taker, receiver, address(0), "");
        assertEq(midnight.consumed(offer.maker, offer.group), consumedBefore, "consumed still unchanged after 2nd take");

        // After fix: both takes above should revert with ZeroAssetTake()
    }
}
```
