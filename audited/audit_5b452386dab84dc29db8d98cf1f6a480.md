### Title
Lender Can Frontrun Bad Debt Realization to Escape Proportional Loss, Shifting Burden to Remaining Lenders — (File: src/Midnight.sol)

---

### Summary

A lender with an existing credit position can monitor the mempool for a pending `liquidate` call that will realize bad debt, then frontrun it with `withdraw`. Because `withdraw` snapshots the current (pre-bad-debt) `lossFactor` via `_updatePosition`, the exiting lender receives their full credit in loan tokens. The subsequent `liquidate` then computes the new `lossFactor` over a smaller `totalUnits` pool, forcing remaining lenders to absorb a disproportionately larger share of the bad debt. This directly violates the protocol's stated invariant that bad debt reduces lender credit **proportionally**.

---

### Finding Description

**Vulnerability class**: Timing attack on shared accounting — same class as the external report (frontrunning an accounting update to gain/avoid value in a pooled system).

**Root cause — three interacting facts:**

**Fact 1.** `withdraw` applies the current `lossFactor` via `_updatePosition` before reducing credit. If `lossFactor` has not yet been updated by a pending bad debt liquidation, the lender exits at full face value. [1](#0-0) 

**Fact 2.** `_updatePosition` / `updatePositionView` computes the post-slash credit as:

```
postSlashCredit = credit * (MAX - currentLossFactor) / (MAX - lastLossFactor)
```

If `currentLossFactor == lastLossFactor` (no bad debt has been realized yet), `postSlashCredit == credit` — no slash occurs. [2](#0-1) 

**Fact 3.** The bad debt realization in `liquidate` computes the new `lossFactor` using `_totalUnits` **at the moment of the call**. If the frontrunning lender has already exited, `_totalUnits` is smaller, so the same `badDebt` amount produces a larger `lossFactor` increase, and remaining lenders are slashed more. [3](#0-2) 

**Exploit flow:**

1. Market has `totalUnits = 2000`, `withdrawable = 1000`, `lossFactor = 0`. Alice holds 1000 credit units; Bob holds 1000 credit units.
2. A borrower's position is undercollateralized. A liquidator submits `liquidate(...)` that will realize `badDebt = 200` units.
3. Alice sees the pending liquidation in the mempool. She frontruns with `withdraw(market, 1000, alice, alice)`.
   - `_updatePosition` runs: `lossFactor` is still 0, so `postSlashCredit = 1000`. Alice's `lastLossFactor` is set to 0.
   - `credit[alice] -= 1000`, `withdrawable -= 1000`, `totalUnits -= 1000`.
   - Alice receives 1000 loan tokens.
4. The liquidation executes. Now `_totalUnits = 1000` (only Bob remains).
   - `newLossFactor = MAX - (MAX - 0) * (1000 - 200) / 1000 = MAX * 200/1000 = MAX/5`
5. Bob's next `_updatePosition` computes: `postSlashCredit = 1000 * (MAX - MAX/5) / MAX = 800`.
   - Bob loses 200 units — the **entire** bad debt, despite having held only half the pool.

Without the frontrun, both Alice and Bob would each lose 100 units (proportional). Alice stole 100 loan tokens from Bob.

---

### Impact Explanation

**Direct theft of lender funds.** The frontrunning lender exits at full face value, transferring their proportional share of bad debt onto remaining lenders. In the worst case (single remaining lender), one lender absorbs 100% of bad debt that should have been shared. This violates the core protocol invariant:

> `"bad_debt_reduces_lenders_credit_proportionally": true` [4](#0-3) 

The loss is permanent and unrecoverable — once the frontrunner withdraws and the bad debt is realized, the remaining lenders' credit is irreversibly slashed beyond their fair share.

---

### Likelihood Explanation

**Realistic and repeatable.** The attacker requires:

1. An existing lender position in the market (normal user, no privilege required).
2. A pending `liquidate` transaction visible in the public mempool that will realize `badDebt > 0`.
3. `marketState[id].withdrawable >= attacker's credit` at the time of the frontrun.

Condition 3 is the binding constraint. It is satisfied whenever prior borrower repayments or partial liquidations have made loan tokens withdrawable — a normal occurrence in any active market. The attack is permissionless, requires no special role, and can be automated with a mempool watcher. It can be repeated across every bad debt event in every market.

The protocol's own NatSpec acknowledges the withdrawal race:

> *"Lenders (and the fee claimer) might also race to withdraw first."* [5](#0-4) 

---

### Recommendation

The root cause is that `withdraw` does not force the exiting lender to bear bad debt that is already economically certain (pending in the mempool) but not yet on-chain. Mitigations to consider:

1. **Withdrawal delay / timelock**: Require lenders to signal intent to withdraw N blocks in advance. This makes mempool-based frontrunning infeasible because the delay exceeds the window in which a liquidation can be censored.

2. **Snapshot-based accounting (TWAB)**: Track time-weighted average balances so that a lender's share of bad debt is computed over the period they actually held credit, not just at the moment of realization. This is the approach referenced in the original report (PoolTogether TWAB).

3. **Pessimistic exit**: Before allowing `withdraw`, check whether any position in the market is currently liquidatable and has bad debt. If so, require the lender to call `updatePosition` (which would apply any realized `lossFactor`) but also block exit until the bad debt is settled. This is the most conservative option.

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
