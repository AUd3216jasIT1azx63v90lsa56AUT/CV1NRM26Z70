### Title
Fee-on-Transfer Loan Token in `repay()` Creates Withdrawable/Balance Shortfall - (File: src/Midnight.sol)

### Summary

`repay()` unconditionally reduces `position[id][onBehalf].debt` and increases `marketState[id].withdrawable` by the caller-supplied `units` parameter, then calls `safeTransferFrom` for exactly `units`. If the loan token is a fee-on-transfer token, Midnight receives only `units - fee` but credits `units` to `withdrawable`, creating a permanent shortfall. Any market whose lender offer is ratified via `EcrecoverRatifier` is equally affected because the ratifier only validates offer signatures and imposes no restriction on token type.

### Finding Description

**Code path:**

`src/Midnight.sol` `repay()` (lines 502–521):

```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);   // L508
marketState[id].withdrawable += UtilsLib.toUint128(units);  // L509
...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // L520
```

`SafeTransferLib.safeTransferFrom` (lines 24–34 of `src/libraries/SafeTransferLib.sol`) only checks that `transferFrom` does not revert and does not return `false`. It does not measure the actual balance delta received.

**Root cause:** The accounting update (`+units` to `withdrawable`) is performed before and independently of the actual token receipt. No balance-before/after check exists.

**Attacker-controlled inputs:**
- `market.loanToken` — any address; no whitelist or token-type restriction in `touchMarket()` or anywhere else.
- `units` — caller-supplied repay amount.

**Exploit flow:**
1. Attacker (market creator or any participant) creates a market whose `loanToken` is a fee-on-transfer ERC20 (e.g., 1% fee on every transfer). The market's lender offer uses `EcrecoverRatifier` as its ratifier — this is irrelevant to the repay path but satisfies the scoped precondition.
2. Lender takes the borrower's sell offer (or borrower takes the lender's buy offer) via `take()`. Debt `D` is recorded for the borrower.
3. Borrower calls `repay(market, D, borrower, address(0), "")`.
4. Inside `repay()`:
   - `position[id][borrower].debt` is set to `0` (full debt cleared).
   - `marketState[id].withdrawable` is increased by `D`.
   - `safeTransferFrom(loanToken, borrower, midnight, D)` is called; the token contract transfers `D` from borrower but delivers only `D * (1 - fee_rate)` to Midnight.
5. `marketState[id].withdrawable == D` but `loanToken.balanceOf(midnight) == D * (1 - fee_rate)`.
6. When the lender calls `withdraw(market, D, lender, lender)`, `withdraw()` calls `safeTransfer(loanToken, lender, D)`. If the token also charges a fee on outbound transfers, the lender receives even less; regardless, Midnight's balance is insufficient to cover the full `withdrawable` amount, causing the last lender's `withdraw()` to revert.

**Why existing checks fail:**
- `SafeTransferLib` does not perform a balance snapshot; it only checks the boolean return value.
- `EcrecoverRatifier.isRatified()` only verifies the EIP-712 Merkle-tree signature of the offer; it has no awareness of token behavior.
- The Certora `Solvency.spec` explicitly models tokens as non-fee-taking (line 31: *"Assume ERC20 tokens transfer correctly: no fee taking"*), so the formal verification does not cover this case.
- `touchMarket()` validates collateral params, LLTV, and maturity but places no restriction on `loanToken`.

### Impact Explanation

`marketState[id].withdrawable` permanently exceeds `loanToken.balanceOf(address(midnight))` by `units * fee_rate` per repay call. Lenders holding credit equal to the full `withdrawable` amount cannot all withdraw; the shortfall grows with each repay. This directly breaks the core solvency invariant: *"contract token balances must cover withdrawable assets."*

### Likelihood Explanation

**Preconditions:**
- A fee-on-transfer token is used as `loanToken` (permissionless; any address can create a market with any token).
- A borrower repays any nonzero amount.

**Feasibility:** Trivially reachable by any borrower in any market using such a token. No privileged access required. Repeatable on every `repay()` call, compounding the shortfall.

### Recommendation

Measure the actual balance delta received in `repay()` (and symmetrically in `liquidate()` and `supplyCollateral()`) and use that delta — not the caller-supplied `units` — for the accounting update:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
require(received == units, FeeOnTransferNotSupported());
```

Alternatively, document that fee-on-transfer tokens are explicitly unsupported and enforce this at market creation time (e.g., a token registry or a creation-time transfer probe).

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry fuzz test
contract FeeOnTransferRepayTest is BaseTest {
    FeeOnTransferToken feeToken; // 1% fee on every transferFrom

    function setUp() public override {
        super.setUp();
        feeToken = new FeeOnTransferToken(100); // 1% = 100 bps
        market.loanToken = address(feeToken);
        // market.ratifier = address(ecrecoverRatifier) -- satisfies scoped precondition
    }

    function testFuzz_repayShortfall(uint256 units, uint256 feeBps) public {
        units = bound(units, 1e18, 1e24);
        feeBps = bound(feeBps, 1, 1000); // 0.01% to 10%

        // Setup: lender supplies credit, borrower accumulates debt D = units
        setupMarket(market, units);
        collateralize(market, borrower, units);
        // ... take() to create debt ...

        uint256 balanceBefore = feeToken.balanceOf(address(midnight));
        uint256 withdrawableBefore = midnight.withdrawable(id);

        vm.prank(borrower);
        midnight.repay(market, units, borrower, address(0), "");

        uint256 balanceAfter = feeToken.balanceOf(address(midnight));
        uint256 withdrawableAfter = midnight.withdrawable(id);

        uint256 actualReceived = balanceAfter - balanceBefore;
        uint256 withdrawableIncrease = withdrawableAfter - withdrawableBefore;

        // ASSERTION: withdrawable increase must not exceed actual tokens received
        assertLe(withdrawableIncrease, actualReceived,
            "withdrawable exceeds actual balance: shortfall created");

        // ASSERTION: lender can withdraw full credit
        vm.prank(lender);
        midnight.withdraw(market, units, lender, lender); // must not revert
    }
}
```

Expected result: `assertLe` fails because `withdrawableIncrease == units` but `actualReceived == units * (1 - fee_rate)`. The subsequent `withdraw()` reverts with an ERC20 transfer failure.
