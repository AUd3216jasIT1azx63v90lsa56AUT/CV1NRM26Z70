### Title
`mulDivDown` rounding in referral fee computation redirects 1 wei from referral recipient to caller - (`src/periphery/MidnightBundles.sol`)

### Summary
In `buyWithUnitsTargetAndWithdrawCollateral`, `referralFeeAssets` is computed with `mulDivDown`, which floors the result. Because the refund to `msg.sender` is `maxBuyerAssets - filledBuyerAssets - referralFeeAssets`, any fractional wei lost by the floor is silently added to the caller's refund rather than sent to the referral recipient. The referral recipient receives at most 1 wei less per call than the exact proportional fee.

### Finding Description
**Code path** — `src/periphery/MidnightBundles.sol` lines 102–104:

```solidity
uint256 referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - filledBuyerAssets - referralFeeAssets);
```

`mulDivDown` is `(x * y) / d` (plain Solidity integer division, floor). [1](#0-0) 

**Root cause** — When `filledBuyerAssets * referralFeePct` is not exactly divisible by `(WAD - referralFeePct)`, the floor discards the fractional wei. The exact fee is:

```
exactFee = filledBuyerAssets * referralFeePct / (WAD - referralFeePct)   [real number]
referralFeeAssets = floor(exactFee)  ≤  exactFee
```

The residual `exactFee - referralFeeAssets` (0 or 1 wei in integer arithmetic) is not sent to `referralFeeRecipient`; instead it inflates the refund to `msg.sender`. [2](#0-1) 

**Attacker inputs** — Any caller who picks `(filledBuyerAssets, referralFeePct)` such that `filledBuyerAssets * referralFeePct % (WAD - referralFeePct) != 0` triggers the rounding. No privileged access is required; the caller controls both `referralFeePct` (passed directly) and indirectly controls `filledBuyerAssets` through the offers they supply.

**Why existing checks do not stop it** — There is no check that `referralFeeAssets + refund == maxBuyerAssets - filledBuyerAssets` using ceiling arithmetic. The bundler-residual-zero invariant holds (the bundler sends out exactly what it holds), but the split between `referralFeeRecipient` and `msg.sender` is off by 1 wei whenever rounding occurs. The existing fuzz test (`testBuyUnitsTargetWithReferralFee`) computes `expectedFee` with the same `mulDivDown`, so it cannot detect the discrepancy. [3](#0-2) 

### Impact Explanation
The referral recipient receives `floor(filledBuyerAssets * referralFeePct / (WAD - referralFeePct))` instead of the ceiling. The difference is at most 1 wei per call. That 1 wei is transferred to `msg.sender` as part of the refund. No funds are locked or lost from the protocol; the impact is a minor, per-call precision loss to the referral recipient.

### Likelihood Explanation
The rounding condition (`filledBuyerAssets * referralFeePct % (WAD - referralFeePct) != 0`) is satisfied for the vast majority of `(filledBuyerAssets, referralFeePct)` pairs. It is trivially reachable on every call where the division is not exact, which is the common case. The caller can deliberately choose inputs to guarantee it. The finding is repeatable on every such call.

### Recommendation
Replace `mulDivDown` with `mulDivUp` for `referralFeeAssets` so that rounding favors the referral recipient rather than the caller:

```solidity
uint256 referralFeeAssets = filledBuyerAssets.mulDivUp(referralFeePct, WAD - referralFeePct);
```

This ensures `referralFeeAssets + refund <= maxBuyerAssets - filledBuyerAssets` (the bundler may retain at most 1 wei, which can be swept or accepted as dust), and the referral recipient never receives less than the exact proportional fee. Alternatively, compute the refund first with `mulDivDown` and derive the fee as the remainder, keeping the caller-favoring rounding explicit and documented.

### Proof of Concept
**Foundry fuzz test** targeting the invariant `referralFeeAssets + refund == maxBuyerAssets - filledBuyerAssets` with exact-proportional semantics:

```solidity
function testFuzz_referralRoundingRedirectsWei(
    uint256 filledBuyerAssets,
    uint256 referralFeePct
) public {
    filledBuyerAssets = bound(filledBuyerAssets, 1, 1e36);
    referralFeePct    = bound(referralFeePct, 1, WAD - 1);

    uint256 WAD = 1e18;
    uint256 denom = WAD - referralFeePct;

    uint256 feeDown = (filledBuyerAssets * referralFeePct) / denom;   // mulDivDown
    uint256 feeUp   = (filledBuyerAssets * referralFeePct + denom - 1) / denom; // mulDivUp

    // When rounding occurs, feeDown < feeUp: 1 wei goes to caller instead of referrer.
    if ((filledBuyerAssets * referralFeePct) % denom != 0) {
        assertEq(feeUp - feeDown, 1, "rounding gap must be exactly 1 wei");
        // The current implementation sends feeDown to referrer and
        // (maxBuyerAssets - filledBuyerAssets - feeDown) to caller,
        // giving the caller 1 extra wei vs. the exact proportional split.
    }
}
```

**Expected assertion**: `feeUp - feeDown == 1` whenever `filledBuyerAssets * referralFeePct % (WAD - referralFeePct) != 0`, confirming the 1-wei redirect. A concrete minimal example: `filledBuyerAssets = 1`, `referralFeePct = 1` (1 wei fee pct), `WAD - referralFeePct = 1e18 - 1`; `1 * 1 / (1e18 - 1) = 0` (floored), so the referrer receives 0 instead of the fractional-wei-rounded-up value of 1.
