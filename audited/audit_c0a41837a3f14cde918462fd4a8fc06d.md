### Title
Missing Receiver Address Validation Permanently Locks Withdrawn Funds in Contract — (`File: src/Midnight.sol`)

### Summary

The `withdraw` and `withdrawCollateral` functions in `Midnight.sol` accept a caller-supplied `receiver` address with no validation that it differs from `address(this)`. When `receiver` is set to the contract's own address, the accounting state is fully decremented (credit, withdrawable, totalUnits / collateral bitmap) before the token transfer executes, sending the tokens back into the contract with no corresponding accounting entry. Since Midnight has no token-rescue or sweep function, those tokens are permanently irrecoverable.

### Finding Description

Both functions follow the same pattern: decrement on-chain accounting first, then transfer tokens to the unvalidated `receiver`.

**`withdraw`** — `src/Midnight.sol` lines 481–500:

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    ...
    _position.credit        -= UtilsLib.toUint128(units);   // accounting decremented
    _marketState.withdrawable -= UtilsLib.toUint128(units); // accounting decremented
    _marketState.totalUnits   -= UtilsLib.toUint128(units); // accounting decremented
    ...
    SafeTransferLib.safeTransfer(market.loanToken, receiver, units); // receiver never validated
}

**`withdrawCollateral`** — `src/Midnight.sol` lines 549–573:

```solidity

If `receiver == address(this)` in either call:

1. All accounting entries are permanently decremented — the position can never be re-credited.
2. The ERC-20 `transfer` succeeds (the contract is a valid address), so no revert occurs.
3. The tokens land in the contract's own balance with no accounting entry tracking them.
4. No function in `Midnight.sol` can recover untracked token balances — `claimSettlementFee` and `claimContinuousFee` only drain their own tracked mappings/fields; there is no general sweep path. [3](#0-2) 

### Impact Explanation

**Permanent, unrecoverable loss of user funds.** Any loan tokens or collateral tokens sent to `address(this)` via a misrouted `withdraw` or `withdrawCollateral` call are locked forever. The user's on-chain position is fully consumed (credit zeroed, collateral cleared), so there is no accounting basis to re-issue the funds. Severity is high because the loss is total and irreversible for the affected amount.

### Likelihood Explanation

The scenario is realistic for several reasons:

- Integrators building smart-contract wrappers around Midnight may pass `address(this)` (their own contract) as `receiver`, then accidentally pass the Midnight contract address instead.
- Front-end bugs or copy-paste errors can substitute the contract address for the user's wallet address.
- Programmatic callers (bots, routers) that derive `receiver` dynamically can trivially produce `address(this)` under edge-case logic.

No privileged access is required; any authorized caller of `withdraw` or `withdrawCollateral` can trigger this path.

### Recommendation

Add a guard at the top of both functions rejecting `receiver == address(this)`:

```solidity
// in withdraw and withdrawCollateral
require(receiver != address(this), InvalidReceiver());
```

This mirrors the short-term fix recommended in the reference report and costs a single `SLOAD`-free comparison.

### Proof of Concept

**`withdraw` path:**

1. Alice holds `1000` credit units in a market with `loanToken = USDC`.
2. Alice calls `withdraw(market, 1000, alice, address(midnight))` — passing the Midnight contract as `receiver`.
3. The function decrements `alice.credit` by 1000, `withdrawable` by 1000, `totalUnits` by 1000.
4. `SafeTransferLib.safeTransfer(USDC, address(midnight), 1000)` succeeds — USDC balance of the contract increases by 1000.
5. Alice's position is now zero. The 1000 USDC sit in the contract with no accounting entry. No function can recover them.

**`withdrawCollateral` path:**

1. Bob holds `5 ETH` collateral in a market.
2. Bob calls `withdrawCollateral(market, 0, 5e18, bob, address(midnight))`.
3. `_position.collateral[0]` is decremented to 0; the collateral bitmap bit is cleared; `isHealthy` passes (no debt).
4. `SafeTransferLib.safeTransfer(WETH, address(midnight), 5e18)` succeeds.
5. Bob's collateral position is zeroed. The 5 WETH are permanently locked in the contract. [4](#0-3) [5](#0-4)

        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());

        uint128 pendingFeeDecrease;

        _marketState.totalUnits -= UtilsLib.toUint128(units);

    function withdrawCollateral(
        Market memory market,
        uint256 collateralIndex,
        uint256 assets,
        address onBehalf,
        address receiver
    ) external {
        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
        bytes32 id = touchMarket(market);
        address collateralToken = market.collateralParams[collateralIndex].token;

        Position storage _position = position[id][onBehalf];
        uint256 newCollateral = _position.collateral[collateralIndex] - assets;
        _position.collateral[collateralIndex] = UtilsLib.toUint128(newCollateral);

        if (newCollateral == 0 && assets > 0) {
            _position.collateralBitmap = _position.collateralBitmap.clearBit(collateralIndex);
        }

        require(isHealthy(market, id, onBehalf), UnhealthyBorrower());

        emit EventsLib.WithdrawCollateral(msg.sender, id, collateralToken, assets, onBehalf, receiver);

        SafeTransferLib.safeTransfer(collateralToken, receiver, assets);
    }
