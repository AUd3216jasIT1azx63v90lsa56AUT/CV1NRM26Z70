Audit Report

## Title
Front-Running Partial Fill Causes Exact-Equality DoS in `buyWithAssetsTargetAndWithdrawCollateral` and `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/MidnightBundles.sol)

## Summary

`ConsumableUnitsLib.consumableUnits` reads live `consumed[maker][group]` state at call time with no snapshot from bundle submission. An attacker who partially fills a target offer before the bundle executes reduces the returned unit capacity, causing `unitsToTake` to be capped below what is needed to reach `targetFilledBuyerAssets` / `targetFilledSellerAssets`. The `take` call succeeds but underfills, and the strict equality guards at lines 224 and 303 then revert the entire bundle. The victim's tokens are returned on revert, but the bundle is permanently blocked as long as the attacker repeats the front-run.

## Finding Description

**Root cause — live `consumed` read in `ConsumableUnitsLib.sol` lines 14–15:**
`consumableUnits` calls `IMidnight(midnight).consumed(offer.maker, offer.group)` at execution time. [1](#0-0) 

Any intervening `take` on the same offer increments `consumed[maker][group]`, reducing the returned unit capacity. The `consumed` mapping is confirmed non-decreasing by the Certora `consumeNonDecreasing` rule. [2](#0-1) 

**`unitsToTake` is capped by live capacity — `MidnightBundles.sol` lines 208–213:**
```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.buyerAssetsToUnits(..., targetFilledBuyerAssets - filledBuyerAssets),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // live state
);
``` [3](#0-2) 

If `consumableUnits` is reduced by a front-run, `unitsToTake` is smaller, `take` succeeds but fills fewer assets than needed, and `filledBuyerAssets` ends below `targetFilledBuyerAssets`.

**`try/catch` only handles reverts — lines 215–221:**
The catch block silently skips offers that revert, but does not handle the case where `take` succeeds with fewer assets than needed. There is no slippage tolerance or minimum-fill threshold. [4](#0-3) 

**Strict equality check — line 224:**
```solidity
require(filledBuyerAssets == targetFilledBuyerAssets, OutOfOffers());
``` [5](#0-4) 

**Identical pattern in `supplyCollateralAndSellWithAssetsTarget` — lines 285–290 and 303:** [6](#0-5) [7](#0-6) 

**No refund path in `buyWithAssetsTargetAndWithdrawCollateral`:**
The function pulls `targetBuyerAssets` at line 197 but only sends the referral fee at line 239 on success — there is no transfer back of unused tokens to `msg.sender`. This means relaxing the strict equality check requires simultaneously adding a refund path to avoid locking tokens in the bundler. [8](#0-7) [9](#0-8) 

**`take` is callable by any external address as their own taker:**
`Midnight.take` requires `taker == msg.sender || isAuthorized[taker][msg.sender]`. Any attacker sets `taker = msg.sender`, satisfying the check without any privilege. [10](#0-9) 

**Exploit flow:**
1. Victim submits `buyWithAssetsTargetAndWithdrawCollateral` with offer O (`maxAssets > 0`), `targetFilledBuyerAssets = X`. The offer and `ratifierData` are visible in the mempool.
2. Attacker front-runs with `Midnight.take(O, ratifierData, smallUnits, attacker, ...)` — permissionless, attacker sets `taker = attacker`.
3. `consumed[O.maker][O.group]` increases; `consumableUnits(O)` returns reduced capacity.
4. Victim's bundle executes: `unitsToTake` is capped at reduced value; `take` succeeds but `resBuyerAssets < X`.
5. Loop ends (no remaining offers or all exhausted); `filledBuyerAssets < targetFilledBuyerAssets`; `require` at line 224 reverts.
6. Victim's tokens are returned on revert. Attacker repeats next block.

## Impact Explanation

Any unprivileged actor can repeatedly grief a victim's `buyWithAssetsTargetAndWithdrawCollateral` or `supplyCollateralAndSellWithAssetsTarget` bundle by front-running with a minimal partial fill of the target offer. The victim's transaction reverts with `OutOfOffers` every time, permanently blocking the bundle from executing as long as the attacker repeats the front-run. This constitutes sustained, repeatable denial-of-service against a specific user's bundle execution, matching the "service unavailability or severe degradation under realistic attacker input" impact class in RESEARCHER.md. [11](#0-10) 

## Likelihood Explanation

Taking an offer is permissionless for any external address (attacker sets `taker = msg.sender`). The attacker needs only enough loan tokens to take 1 unit from the target offer, receiving a market credit position in return, so the net cost is gas plus any bid-ask spread. The `ratifierData` required to call `take` is visible in the victim's pending mempool transaction. The attack is repeatable every block with no cooldown. The only natural limit is the offer's `maxAssets` being fully consumed, but at that point the victim's bundle also cannot use the offer. Preconditions are minimal: victim uses an assets-target bundle with an offer that has `maxAssets > 0` and insufficient fallback offers.

## Recommendation

**For `buyWithAssetsTargetAndWithdrawCollateral`:**
1. Replace the strict equality `require(filledBuyerAssets == targetFilledBuyerAssets, OutOfOffers())` with a minimum-fill threshold (e.g., `require(filledBuyerAssets >= minBuyerAssets, OutOfOffers())`), where `minBuyerAssets` is a caller-supplied slippage parameter.
2. Add a refund path for unused tokens: `SafeTransferLib.safeTransfer(loanToken, msg.sender, targetFilledBuyerAssets - filledBuyerAssets)` after the loop, so partial fills do not strand tokens in the bundler.

**For `supplyCollateralAndSellWithAssetsTarget`:**
1. Replace the strict equality with a minimum-fill threshold.
2. Adjust the receiver transfer to use `filledSellerAssets - referralFeeAssets` instead of the fixed `targetSellerAssets`, so the receiver gets exactly what was filled.

Both fixes together eliminate the exact-equality DoS while preserving the slippage protection intent.

## Proof of Concept

**Minimal fork test plan:**
1. Deploy `Midnight` and `MidnightBundles`.
2. Create a market; maker posts a sell offer with `maxAssets = 1000`, `maxUnits = 0`.
3. Victim calls `buyWithAssetsTargetAndWithdrawCollateral` with a single-element `takes` array targeting the offer and `targetBuyerAssets` corresponding to 100 units.
4. Before victim's tx lands, attacker calls `Midnight.take(offer, ratifierData, 1, attacker, ...)` (1 unit), incrementing `consumed[maker][group]` by the corresponding asset amount.
5. Victim's bundle executes: `consumableUnits` returns capacity reduced by the front-run amount; `unitsToTake` is capped; `take` succeeds but `resBuyerAssets < targetFilledBuyerAssets`; `require` at line 224 reverts with `OutOfOffers`.
6. Assert victim's loan tokens are returned (no loss) but the bundle did not execute.
7. Repeat steps 4–6 to demonstrate sustained DoS.
