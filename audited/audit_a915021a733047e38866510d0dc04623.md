Audit Report

## Title
`sellerAssetsToUnits` division-by-zero when `offerPrice == settlementFee` causes unconditional revert in `supplyCollateralAndSellWithAssetsTarget` - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and then calls `mulDivUp(targetSellerAssets, WAD, sellerPrice)`. When `offerPrice == settlementFee(id, 0)` post-maturity, `sellerPrice = 0` and `mulDivUp` reverts via uint256 underflow in `d - 1`. Midnight's core `take()` handles this case correctly (computing `sellerAssets = 0`), so the periphery diverges from core. Because `sellerAssetsToUnits` is called outside the `try/catch` in `supplyCollateralAndSellWithAssetsTarget`, any `takes` array containing such an offer causes the entire transaction to revert unconditionally.

## Finding Description

**Root cause — `TakeAmountsLib.sellerAssetsToUnits`, lines 44–46:**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

When `offer.buy == true` and `offerPrice == settlementFee`, `sellerPrice = 0`. The call `targetSellerAssets.mulDivUp(WAD, 0)` executes `(x * y + (0 - 1)) / 0`, where `0 - 1` underflows in checked Solidity 0.8+ arithmetic, causing an arithmetic revert. This is confirmed by `testMulDivUpDivisionByZero`, which documents that `mulDivUp(x, y, 0)` reverts with `stdError.arithmeticError`.

**Why the condition is reachable:** `PRICE_ROUNDING_STEP = 1e12` means `tickToPrice(tick)` returns multiples of `1e12`. `MAX_SETTLEMENT_FEE_0_DAYS = 14e12` and `CBP = 1e12`, so `settlementFee(id, 0)` takes values in `{0, 1e12, 2e12, ..., 14e12}`. `tickToPrice(2) = 1e12`, so for `settlementFeeCbp0 = 1`, tick 2 satisfies `tickToPrice(tick) == settlementFee(id, 0)`. With tick spacing reduced to 1 (permitted by `tickSpacingSetter`), tick 2 is accessible.

**Why Midnight's `take()` succeeds:** In `Midnight.take()`, `sellerAssets = units.mulDivDown(0, WAD) = 0`, which is valid. The `CannotIncreaseDebtPostMaturity` check passes when `sellerDebtIncrease == 0` (seller has sufficient credit to cover the units).

**Why the periphery reverts:** `sellerAssetsToUnits` is called at line 286 **outside** the `try/catch` block that wraps `take()` (which starts at line 292). The NatSpec at line 175 explicitly acknowledges this: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." The `try/catch` only covers the `IMidnight(MIDNIGHT).take(...)` call.

**NatSpec divergence:** `sellerAssetsToUnits` documents "Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)." The `offerPrice == settlementFee` case is not covered by this comment, and Midnight does NOT revert in this case — only the periphery does.

**Exploit flow:**
1. Market passes maturity (`block.timestamp > offer.market.maturity`).
2. Attacker creates a buy offer at a tick where `tickToPrice(tick) == settlementFee(id, 0)` (e.g., `settlementFeeCbp0 = 1` → `settlementFee = 1e12`, tick 2 with `tickToPrice(2) = 1e12`).
3. Attacker broadcasts the offer off-chain. Off-chain routers include it in the `takes` array.
4. Victim calls `supplyCollateralAndSellWithAssetsTarget` with this offer in `takes`.
5. `sellerAssetsToUnits` is called with `targetSellerAssets > 0` and `sellerPrice = 0` → arithmetic revert.
6. Entire transaction reverts. The offer remains valid and the condition persists as long as the market is post-maturity with the same fee.

**Existing protections are insufficient:** `buyerAssetsToUnits` has a `require(buyerPrice <= WAD)` guard that incidentally catches `offerPrice < settlementFee` via underflow, but `sellerAssetsToUnits` has no analogous guard for `sellerPrice == 0`.

## Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee(id, 0)` post-maturity will revert unconditionally, regardless of `targetSellerAssets`. This is a permanent DoS of the sell-via-periphery path for post-maturity markets whenever such an offer appears in the routing set. The attacker's cost is only the gas to create the offer; no funds are at risk for the attacker.

## Likelihood Explanation
**Preconditions:**
- Market is post-maturity (inevitable for any market).
- `settlementFeeCbp0 >= 1` (any non-zero post-maturity fee, the common case).
- A tick exists where `tickToPrice(tick) == settlementFee(id, 0)` — achievable given `PRICE_ROUNDING_STEP = 1e12` and max fee of `14e12`.
- The attacker's offer is included in the victim's `takes` array via off-chain routing.

The condition is repeatable: the offer can be recreated after expiry, and the revert is deterministic. Any routing system that automatically aggregates available buy offers without filtering for `sellerPrice > 0` is vulnerable.

## Recommendation
Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case, mirroring how Midnight's core handles it. When `sellerPrice == 0`, the function should return `type(uint256).max` (or a sentinel indicating infinite units needed) so that `min(...)` in the bundler reduces `unitsToTake` to 0, and the offer is effectively skipped. Alternatively, add an explicit check:

```solidity
if (sellerPrice == 0) return type(uint256).max;
```

This aligns the periphery with core behavior: when `sellerPrice = 0`, any number of units yields `sellerAssets = 0`, so the offer cannot contribute to `targetSellerAssets > 0` and should be skipped.

## Proof of Concept

// 5. Observe revert with stdError.arithmeticError from mulDivUp(targetSellerAssets, WAD, 0).
