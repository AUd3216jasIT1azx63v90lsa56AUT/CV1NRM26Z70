### Title
`seizedAssets` path arithmetic underflow when oracle price > ORACLE_PRICE_SCALE causes post-maturity liquidation revert - (File: src/Midnight.sol)

### Summary
In `liquidate`, when `seizedAssets > 0`, `repaidUnits` is computed via two `mulDivUp` calls. In post-maturity mode there is no guard that `repaidUnits <= _position.debt`, so when the oracle price exceeds `ORACLE_PRICE_SCALE` and the borrower's debt is small, the computed `repaidUnits` can exceed `_position.debt`, causing an arithmetic underflow revert at `_position.debt -= UtilsLib.toUint128(repaidUnits)`. The `maxRepaid` guard that would catch this exists only in the `!postMaturityMode` branch.

**Root cause:** There is no `repaidUnits <= _position.debt` check in the `seizedAssets` path when `postMaturityMode == true`. The `maxRepaid` check at lines 655–668 is gated on `!postMaturityMode` and is never evaluated in the post-maturity branch. [4](#0-3) 

**Exploit flow (concrete numbers):**

- `ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`, `maxLif = WAD` (minimum allowed).
- Borrower has `debt = 1`, `collateral[0] = 1`, oracle price at borrow time = 1 (so `debt <= collateral * price * lltv / WAD` is satisfied).
- Oracle price rises to `2 * ORACLE_PRICE_SCALE = 2e36` (collateral appreciates 2×).
- Market matures; `block.timestamp > market.maturity` → post-maturity liquidation is permitted regardless of health.
- Bad-debt loop: `badDebt = zeroFloorSub(1, mulDivUp(1, 2e36, 1e36).mulDivUp(WAD, WAD)) = zeroFloorSub(1, 2) = 0`. `_position.debt` stays 1.
- Liquidator calls `liquidate(market, 0, 1, 0, borrower, true, ...)` (`seizedAssets=1`, `repaidUnits=0`).
- Line 650: `repaidUnits = mulDivUp(1, 2e36, 1e36).mulDivUp(WAD, WAD) = 2 * 1 = 2`.
- Line 676: `_position.debt -= toUint128(2)` → `1 - 2` → arithmetic underflow → **revert**.

**Why existing checks fail:**

1. The `maxRepaid` guard (lines 659–667) is inside `if (!postMaturityMode)` and is never reached in post-maturity mode. [5](#0-4) 
2. The `NotLiquidatable` check (line 622) passes because `block.timestamp > market.maturity` is sufficient in post-maturity mode — no health check is performed. [6](#0-5) 
3. `UtilsLib.toUint128` (line 676) only checks for `> type(uint128).max`, not for underflow of the subtraction. [7](#0-6) 
4. The `NoMultiplicationOverflow.spec` bounds price to `price * max_uint128 + ORACLE_PRICE_SCALE - 1 <= max_uint256`, which permits `price = 2e36` — well within the safe range, so no overflow protection triggers. [8](#0-7) 

**Attacker-controlled inputs:** `seizedAssets` (set to any value ≥ 1), `postMaturityMode = true`. The oracle price is not attacker-controlled but is a normal market condition (collateral appreciating above `ORACLE_PRICE_SCALE`).

### Impact Explanation
Any post-maturity liquidation call using the `seizedAssets` input path reverts with an arithmetic underflow whenever `mulDivUp(seizedAssets, price, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif) > _position.debt`. This blocks the `seizedAssets` liquidation interface entirely for positions where the collateral has appreciated above `ORACLE_PRICE_SCALE` and the remaining debt is small. The `repaidUnits` path also fails to seize collateral in this regime (it rounds `seizedAssets` down to 0), leaving the collateral effectively unreachable by liquidators via either interface.

### Likelihood Explanation
**Preconditions:**
1. Market is post-maturity (`block.timestamp > market.maturity`) — normal lifecycle event.
2. Oracle price > `ORACLE_PRICE_SCALE` — occurs whenever collateral appreciates significantly; `ORACLE_PRICE_SCALE = 1e36` is the neutral 1:1 price, and any collateral that doubles in value crosses this threshold.
3. Borrower's remaining debt is small relative to the collateral value — common after partial repayments or in markets with small minimum borrow sizes.

All three conditions are reachable through normal protocol usage without any privileged action. The condition is repeatable and deterministic given the oracle price.

### Recommendation
Add a cap on `repaidUnits` in the `seizedAssets` branch to prevent it from exceeding the current debt after bad-debt deduction:

```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
    repaidUnits = UtilsLib.min(repaidUnits, _position.debt); // cap at remaining debt
}
```

Alternatively, revert with a descriptive error (e.g., `RepaidExceedsDebt`) so callers receive a meaningful signal rather than a raw arithmetic underflow. The cap approach is preferable because it allows the liquidator to seize the maximum collateral that the remaining debt can cover, which is the natural intent of the `seizedAssets` interface.

### Proof of Concept
```solidity
function testSeizedAssetsUnderflowPostMaturity() public {
    // Setup: borrower has debt=1, collateral=1
    uint256 units = 1;
    collateralize(market, borrower, units);
    setupMarket(market, units);

    // Price rises to 2x ORACLE_PRICE_SCALE (collateral worth 2 debt units)
    uint256 highPrice = 2 * ORACLE_PRICE_SCALE;
    Oracle(market.collateralParams[0].oracle).setPrice(highPrice);

    // Advance past maturity
    vm.warp(market.maturity + TIME_TO_MAX_LIF);

    // Verify: repaidUnits computed from seizedAssets=1 would be 2 > debt=1
    uint256 lif = market.collateralParams[0].maxLif; // WAD at minimum
    uint256 computedRepaid = uint256(1).mulDivUp(highPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
    assertGt(computedRepaid, midnight.debtOf(id, borrower), "repaidUnits exceeds debt");

    // Assert: liquidation via seizedAssets=1 reverts with arithmetic underflow
    vm.expectRevert(stdError.arithmeticError);
    midnight.liquidate(market, 0, 1, 0, borrower, true, address(this), address(0), "");
}
```

**Expected assertions:**
- `computedRepaid > debtOf(id, borrower)` — confirms the overflow condition.
- `vm.expectRevert(stdError.arithmeticError)` — confirms the revert at `_position.debt -= toUint128(repaidUnits)`.
- A fuzz variant should sweep `price` over `[ORACLE_PRICE_SCALE + 1, type(uint128).max / max_uint128]` and `seizedAssets` over `[1, collateral]`, asserting either success or a meaningful revert (not arithmetic underflow).
