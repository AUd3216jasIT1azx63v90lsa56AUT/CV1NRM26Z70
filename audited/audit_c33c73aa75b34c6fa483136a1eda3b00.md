### Title
Fee-on-transfer loan token causes `repayAndWithdrawCollateral` to always revert - (`src/periphery/MidnightBundles.sol`)

### Summary
`repayAndWithdrawCollateral` computes `units` from the caller-supplied `assets` parameter before pulling tokens, without accounting for the fee deducted by fee-on-transfer loan tokens. After `pullToken`, the bundler holds `assets * (1 - f)` tokens, but `repay` attempts to pull `units = assets * (1 - referralFeePct/WAD)` from the bundler. When the transfer fee rate `f` exceeds `referralFeePct/WAD`, the bundler is underfunded and `Midnight.repay`'s `safeTransferFrom` reverts, making the function permanently unusable for any market whose loan token charges a transfer fee.

### Finding Description
**Code path:**

In `src/periphery/MidnightBundles.sol`, `repayAndWithdrawCollateral` (lines 315–348):

```
uint256 referralFeeAssets = assets.mulDivDown(referralFeePct, WAD);  // line 329
uint256 units = assets - referralFeeAssets;                           // line 330
pullToken(loanToken, msg.sender, assets, loanTokenPermit);            // line 331
forceApproveMax(loanToken, MIDNIGHT);                                 // line 332
IMidnight(MIDNIGHT).repay(market, units, onBehalf, address(0), "");  // line 334
```

`pullToken` (line 378–398) calls `SafeTransferLib.safeTransferFrom(token, from, address(this), assets)`. For a fee-on-transfer token with fee rate `f`, the bundler receives only `assets * (1 - f)`, not `assets`.

In `src/Midnight.sol`, `repay` (lines 502–521):
```
address payer = callback != address(0) ? callback : msg.sender;  // line 511
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);  // line 520
```
Since `callback = address(0)`, `payer = msg.sender = bundler`. Midnight pulls `units` from the bundler.

**Root cause:** `units` is derived from the input `assets`, not from the actual post-transfer balance. No balance snapshot is taken before/after `pullToken` to measure the real received amount.

**Exploit flow:**
1. Attacker (borrower) creates or uses a market whose `loanToken` is a fee-on-transfer token with fee rate `f > 0`.
2. Borrower calls `repayAndWithdrawCollateral(market, assets, onBehalf, permit, withdrawals, receiver, 0, address(0))` with `referralFeePct = 0`.
3. `units = assets - 0 = assets`.
4. `pullToken` transfers `assets` from borrower; bundler receives `assets * (1 - f) < assets`.
5. `repay(market, assets, ...)` triggers `safeTransferFrom(loanToken, bundler, midnight, assets)`.
6. Bundler balance is `assets * (1 - f) < assets` → transfer fails → entire tx reverts.

**Existing protections:** None. There is no balance-before/after check, no fee-on-transfer guard, and no slippage parameter on the received amount. The `forceApproveMax` sets allowance to `type(uint256).max`, so approval is not the bottleneck — the bundler simply lacks the tokens.

### Impact Explanation
`repayAndWithdrawCollateral` is permanently broken for any market with a fee-on-transfer loan token when `f > referralFeePct/WAD`. Borrowers in such markets cannot use the bundler to atomically repay and withdraw collateral. They must fall back to two separate direct calls to `Midnight.repay` and `Midnight.withdrawCollateral`, losing the atomicity guarantee the bundler is designed to provide. If the borrower's position requires atomic repay+withdraw (e.g., to avoid a health-check failure between the two steps), this constitutes a functional denial of service on the repayment path through the bundler.

### Likelihood Explanation
**Preconditions:** A market must exist with a fee-on-transfer loan token, and `referralFeePct` must be less than `f * WAD`. The simplest case is `referralFeePct = 0` with any nonzero fee. Fee-on-transfer tokens (e.g., tokens with deflationary mechanics or protocol fees) are a known token class. The condition is trivially met by any borrower calling the function with `referralFeePct = 0`. No privileged access is required; any borrower can trigger this. The revert is deterministic and repeatable on every call.

### Recommendation
Measure the bundler's actual received balance after `pullToken` and use that to compute `units`:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
pullToken(loanToken, msg.sender, assets, loanTokenPermit);
uint256 received = IERC20(loanToken).balanceOf(address(this)) - balanceBefore;

uint256 referralFeeAssets = received.mulDivDown(referralFeePct, WAD);
uint256 units = received - referralFeeAssets;
forceApproveMax(loanToken, MIDNIGHT);
IMidnight(MIDNIGHT).repay(market, units, onBehalf, address(0), "");
// transfer referralFeeAssets if > 0
```

This ensures `units` never exceeds the bundler's actual balance regardless of transfer fees.
