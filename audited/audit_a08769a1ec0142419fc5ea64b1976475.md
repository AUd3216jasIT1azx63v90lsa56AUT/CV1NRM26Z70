### Title
Fee-on-Transfer / Deflationary Token Accounting Desync in `supplyCollateral` and `repay` — (File: src/Midnight.sol)

---

### Summary

`supplyCollateral` and `repay` update internal accounting using the caller-supplied `assets`/`units` value **before** the actual ERC-20 transfer, and never verify the amount actually received. When a fee-on-transfer or deflationary token is used as a collateral or loan token, the protocol's recorded balances permanently exceed its real token holdings. This enables undercollateralized borrowing (bad debt creation harming lenders) and inflated `withdrawable` that cannot be fully redeemed.

---

### Finding Description

**Root cause — `supplyCollateral` (lines 533 and 545):**

```solidity
// src/Midnight.sol:533
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
// ...
// src/Midnight.sol:545
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

The position is credited with `assets` units of collateral before the transfer executes. `SafeTransferLib.safeTransferFrom` only checks that the call did not revert and that the return value (if any) is `true` — it does **not** measure the actual balance delta. [1](#0-0) [2](#0-1) [3](#0-2) 

If the collateral token charges a 1% transfer fee, supplying `100` tokens records `100` collateral but only `99` arrive. The borrower's `isHealthy` check uses the inflated `100` figure, allowing them to borrow against phantom collateral. [4](#0-3) 

**Root cause — `repay` (lines 508–509 and 520):**

```solidity
// src/Midnight.sol:508-509
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);
// ...
// src/Midnight.sol:520
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
```

`withdrawable` is increased by the full `units` before the transfer. If the loan token is deflationary, only `units − fee` arrives, but lenders can later call `withdraw` for the full `units`. [5](#0-4) [6](#0-5) 

The `live_context.json` audit context explicitly lists `"fee-on-transfer, rebasing, false-return, ERC777-like hooks, and non-standard decimals should be tested if not explicitly excluded"` under `core_invariants.external_calls`, and `"token charges fee"` / `"token rebases"` under `recommended_fuzz_axes.external_behavior`. The SECURITY.md contains no explicit exclusion for fee-on-transfer tokens. [7](#0-6) [8](#0-7) 

The protocol's `TOKEN SAFETY REQUIREMENTS` comment (lines 133–140) lists "Midnight must send/receive exactly the requested amount" as an **assumption**, not a documented exclusion from scope, and no on-chain enforcement of this assumption exists anywhere in the codebase. [9](#0-8) 

---

### Impact Explanation

**`supplyCollateral` path — bad debt / protocol insolvency:**

1. Attacker supplies `N` units of a fee-on-transfer collateral token; protocol records `N` but holds `N − fee`.
2. Attacker borrows against the inflated collateral record, obtaining more loan tokens than the real collateral supports.
3. Attacker defaults. Liquidator can only seize `N − fee` tokens, but the debt was sized against `N`.
4. Residual bad debt is socialized via `lossFactor`, slashing all lenders' credit — a direct loss of lender funds.

**`repay` path — withdrawable inflation / last-lender insolvency:**

1. Borrower repays `N` units of a deflationary loan token; `withdrawable` increases by `N` but only `N − fee` arrives.
2. Each repayment widens the gap between `withdrawable` and actual contract balance.
3. The final lender to call `withdraw` finds the contract balance insufficient and the transaction reverts, permanently locking their credit.

Both impacts fall squarely within the `live_context.json` highest-priority bug classes: `"protocol insolvency"`, `"bad debt creation"`, and `"direct loss of user funds"`. [10](#0-9) 

---

### Likelihood Explanation

- The protocol is **permissionless**: any address can call `touchMarket` with any ERC-20 as loan or collateral token. [11](#0-10) 
- Fee-on-transfer tokens exist in production (e.g., tokens with configurable transfer taxes). No privileged access is required to create a market with one.
- The attacker does **not** need to own or control the token contract — they only need to use an existing deflationary token.
- The `live_context.json` `reporting_bias.likely_invalid_if` list includes `"requires malicious token owner only"` as a disqualifier; this scenario does **not** require that. [12](#0-11) 

---

### Recommendation

Add before/after balance checks at every inbound transfer site to derive the **actual** received amount and use that for accounting:

```solidity
// supplyCollateral — replace lines 533 and 545
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);

// repay — replace lines 508-509 and 520
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
position[id][onBehalf].debt -= UtilsLib.toUint128(received);
marketState[id].withdrawable += UtilsLib.toUint128(received);
```

Apply the same pattern to the inbound transfer in `take` (line 455) and `liquidate` (line 717). Alternatively, enforce the token assumption on-chain by reverting if `received != requested`. [13](#0-12) [14](#0-13)

### Proof of Concept

**Setup:**
- Deploy `FeeToken` — an ERC-20 that deducts 1% on every `transferFrom` (i.e., transfers `value * 99 / 100` to `to`, burns the rest).
- Deploy `Midnight`.
- Create a market with `FeeToken` as the collateral token and a standard ERC-20 as the loan token.

**Attack flow (`supplyCollateral`):**

```
1. Attacker holds 10_000 FeeToken.
2. Attacker calls supplyCollateral(market, 0, 10_000, attacker).
   → position[id][attacker].collateral[0] = 10_000   (recorded)
   → Midnight actually receives 9_900 FeeToken        (1% fee burned)
3. Attacker calls take() to borrow against 10_000 collateral.
   → Borrow amount sized to 10_000 * oracle_price * LLTV.
   → Real collateral only supports 9_900 * oracle_price * LLTV.
4. Attacker does not repay.
5. Liquidator calls liquidate(); seizes only 9_900 FeeToken.
   → Residual ~100-unit debt becomes bad debt.
   → lossFactor increases; all lenders' credit is slashed.
```

**Attack flow (`repay`):**

```
1. Borrower repays 10_000 units of FeeToken loan token.
   → withdrawable += 10_000
   → Midnight receives 9_900
2. Repeat N times; gap grows to N * 100.
3. Lenders call withdraw(); last lender's withdraw() reverts
   because contract balance < withdrawable.
```

Expected outcome: protocol accounting permanently desynced from real token balances, leading to bad debt socialization or permanent fund lock for lenders. [15](#0-14) [16](#0-15)

    function touchMarket(Market memory market) public returns (bytes32) {
        bytes32 id = toId(market);
        if (marketState[id].tickSpacing == 0) {
            require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
            require(market.collateralParams.length > 0, NoCollateralParams());
            require(market.collateralParams.length <= MAX_COLLATERALS, TooManyCollateralParams());
            address previousCollateralToken;
            for (uint256 i = 0; i < market.collateralParams.length; i++) {
                address collateralToken = market.collateralParams[i].token;
                require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
                uint256 lltv = market.collateralParams[i].lltv;
                require(isLltvAllowed(lltv), LltvNotAllowed());
                require(
                    market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_LOW)

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
