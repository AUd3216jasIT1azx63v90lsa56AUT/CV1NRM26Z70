### Title
Off-by-one boundary mismatch: debt increase permitted at `block.timestamp == market.maturity` while post-maturity liquidation requires `block.timestamp > market.maturity` — (`src/Midnight.sol`)

### Summary
`take()` permits debt increase when `block.timestamp <= market.maturity` (inclusive), while `liquidate()` enables post-maturity mode only when `block.timestamp > market.maturity` (exclusive). At exactly `block.timestamp == market.maturity`, a seller can increase their debt via `take` but no liquidator can invoke `postMaturityMode=true` to liquidate that position. The two lifecycle transitions do not share the same boundary, violating the invariant that maturity cannot be abused to bypass liquidation.

### Finding Description
**Code path — `take()`:** [1](#0-0) 

`sellerDebtIncrease` is computed at line 384 as `units - sellerCreditDecrease`. When `block.timestamp == market.maturity`, the condition `block.timestamp <= offer.market.maturity` evaluates to `true`, so the `require` passes and `sellerPos.debt` is incremented at line 414. [2](#0-1) [3](#0-2) 

**Code path — `liquidate()`:** [4](#0-3) 

When `postMaturityMode == true` and `block.timestamp == market.maturity`, the sub-expression `block.timestamp > market.maturity` is `false`, so the entire `require` reverts with `NotLiquidatable()`. Normal-mode liquidation (`postMaturityMode=false`) is only available if `originalDebt > maxDebt`, i.e., the position is already unhealthy.

**Attacker-controlled inputs:** The seller (borrower) controls the `take` call and can choose to execute it in the block whose timestamp equals `market.maturity`. No privileged access is required; any unprivileged seller with sufficient collateral can do this.

**Exploit flow:**
1. Borrower has a healthy position with existing debt below `maxDebt`.
2. At `block.timestamp == market.maturity`, borrower (or an authorized operator) calls `take` against a lender offer, increasing `sellerPos.debt`. The check at line 391 passes because `block.timestamp <= market.maturity`.
3. Position remains healthy after the debt increase (oracle price is sufficient).
4. Liquidator attempts `liquidate(..., postMaturityMode=true)`. The check at line 622 evaluates `block.timestamp > market.maturity` → `false` → reverts with `NotLiquidatable`.
5. Liquidator attempts `liquidate(..., postMaturityMode=false)`. The check evaluates `originalDebt > maxDebt` → `false` (position is healthy) → also reverts.
6. The borrower's newly increased debt is unliquidatable for the duration of the block at exactly maturity.

**Why existing checks fail:** The `CannotIncreaseDebtPostMaturity` guard uses `<=` (inclusive), while the `NotLiquidatable` post-maturity guard uses `>` (exclusive). There is no check that reconciles these two boundaries. The test suite explicitly encodes this asymmetry as expected behavior: [5](#0-4) 

### Impact Explanation
At exactly `block.timestamp == market.maturity`, a borrower can increase their debt while post-maturity liquidation is simultaneously unavailable. If the position remains healthy after the debt increase, neither liquidation mode is accessible: post-maturity mode is blocked by `block.timestamp > market.maturity` being false, and normal mode is blocked by `originalDebt > maxDebt` being false. The borrower's overdue debt is immune to liquidation for the entire block at maturity. This directly violates the invariant "maturity cannot be abused to bypass repayment, settlement, or liquidation" and the protocol property `overdue_debt_after_maturity_is_liquidatable_even_if_healthy`. [1](#0-0) [6](#0-5) 

### Likelihood Explanation
The window is exactly one block (on Ethereum, ~12 seconds; on L2s, potentially 1–2 seconds). Exploiting it requires a transaction to land in the block whose `block.timestamp` equals `market.maturity` exactly. On L2s with deterministic block timestamps and short block times, this is straightforwardly achievable by monitoring the chain and submitting the `take` in the target block. On mainnet it is harder but not impossible, especially for a borrower who controls a validator or uses a block builder. The condition is repeatable for every market whose maturity is known in advance (all of them, since maturity is set at market creation).

### Recommendation
Align the two boundaries. The simplest fix is to change the post-maturity liquidation check to use `>=` so that post-maturity mode becomes available at the same moment debt increase is blocked:

```solidity
// liquidate(), line 622 — change > to >=
postMaturityMode ? block.timestamp >= market.maturity : originalDebt > maxDebt
```

Alternatively, change `take()` line 391 to use strict `<` so debt increase is blocked at maturity:

```solidity
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

Either change eliminates the 1-block gap. The `>=` fix in `liquidate` is preferable because it matches the protocol's stated invariant that overdue debt is liquidatable after maturity, and it does not restrict the last valid trading second.

### Proof of Concept

```solidity
function testBoundaryMismatchAtMaturity() public {
    // Preconditions: healthy borrower with existing debt
    uint256 units = 100;
    collateralize(market, borrower, units * 2); // over-collateralized, healthy
    setupMarket(market, units);                 // borrower has `units` debt

    // Warp to exactly maturity
    vm.warp(market.maturity);

    // Step 1: debt increase succeeds at block.timestamp == maturity
    // (block.timestamp <= market.maturity is TRUE)
    borrowerOffer.expiry = market.maturity;
    borrowerOffer.maxUnits = units;
    deal(address(loanToken), lender, units);
    take(units, lender, borrowerOffer); // must NOT revert

    // Assert debt increased
    assertGt(midnight.debtOf(id, borrower), units, "debt increased at maturity");

    // Step 2: post-maturity liquidation is NOT available at block.timestamp == maturity
    // (block.timestamp > market.maturity is FALSE)
    vm.expectRevert(IMidnight.NotLiquidatable.selector);
    midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");

    // Step 3: normal liquidation is also NOT available (position is healthy)
    vm.expectRevert(IMidnight.NotLiquidatable.selector);
    midnight.liquidate(market, 0, 0, 0, borrower, false, address(this), address(0), "");

    // Step 4: one second later, post-maturity liquidation becomes available
    vm.warp(market.maturity + 1);
    midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), ""); // must succeed
}
```

Expected assertions: steps 1 and 4 succeed; steps 2 and 3 revert with `NotLiquidatable`. This directly reproduces the 1-block window where increased overdue debt is immune to all liquidation modes.
