### Title
Missing bounds check on `collateralIndex` causes opaque panic revert in `liquidate()` - (File: src/Midnight.sol)

### Summary

`Midnight.liquidate()` accepts a caller-supplied `collateralIndex` but never validates it against `market.collateralParams.length` before unconditionally accessing `market.collateralParams[collateralIndex]` in the `emit` statement at line 684. When an unhealthy borrower exists and a liquidator passes `collateralIndex >= market.collateralParams.length`, the function passes all named `require` guards and then panics with a Solidity array out-of-bounds error (panic code `0x32`) rather than reverting with a descriptive custom error. The existing test suite explicitly documents this behavior via `vm.expectRevert(stdError.indexOOBError)`.

### Finding Description

**Code path and root cause:**

In `src/Midnight.sol`, `liquidate()` performs these checks in order:

1. `atMostOneNonZero(repaidUnits, seizedAssets)` — passes when both are 0. [1](#0-0) 
2. `_position.debt > 0` — passes for any borrower. [2](#0-1) 
3. `liquidatorGate` check — passes when no gate is set. [3](#0-2) 
4. The `while (_collateralBitmap != 0)` loop iterates only over bit-indices already set in the borrower's bitmap, so it only accesses `market.collateralParams[i]` for valid `i` values. `collateralIndex` is never touched here. [4](#0-3) 
5. `NotLiquidatable` check — passes when the position is genuinely unhealthy (`originalDebt > maxDebt`). [5](#0-4) 
6. `if (repaidUnits > 0 || seizedAssets > 0)` — **skipped** when both inputs are 0, so `market.collateralParams[collateralIndex].maxLif` at line 644 is never reached. [6](#0-5) 
7. The `emit EventsLib.Liquidate(...)` call at line 681 unconditionally reads `market.collateralParams[collateralIndex].token`. With `collateralIndex = 2` on a 2-element array, this is an **array out-of-bounds panic**. [7](#0-6) 

**Attacker inputs:**
- `collateralIndex` = `market.collateralParams.length` (e.g., `2` for a 2-collateral market)
- `seizedAssets` = 0, `repaidUnits` = 0 (both zero bypasses the guarded block at line 643)
- `borrower` = any address with `debt > 0` and `originalDebt > maxDebt`

**Why existing checks fail:** No check anywhere in `liquidate()` validates `collateralIndex < market.collateralParams.length`. The while-loop guard only protects bitmap-derived indices, not the caller-supplied `collateralIndex`.

The existing test suite confirms this path is reachable and produces `stdError.indexOOBError`:

```solidity
// test/LiquidationTest.sol:76-84
function testLiquidateInvalidCollateralIndex() public {
    uint256 units = 100e18;
    collateralize(market, borrower, units);
    setupMarket(market, units);
    Oracle(market.collateralParams[0].oracle).setPrice(1e36 - 1);

    vm.expectRevert(stdError.indexOOBError);
    midnight.liquidate(market, 2, 0, 0, borrower, false, address(this), address(0), "");
}
``` [8](#0-7) 

### Impact Explanation

An unprivileged liquidator calling `liquidate()` with an out-of-range `collateralIndex` receives a Solidity panic (`0x32`, array out-of-bounds) rather than a named custom error. Off-chain liquidation bots that distinguish transient failures (e.g., gas estimation errors, network issues) from permanent input errors by inspecting revert data may misclassify this panic as a transient failure and retry indefinitely with the same invalid index, delaying or preventing liquidation of the unhealthy position. The invariant that `liquidate()` must revert with a clear, distinguishable error for invalid inputs is violated.

### Likelihood Explanation

- **Preconditions:** An unhealthy borrower must exist (normal operating condition during market stress). The liquidator must supply an out-of-range `collateralIndex`, which can happen due to a bug in off-chain tooling, a misconfigured bot, or deliberate griefing.
- **Feasibility:** Trivially reachable by any caller with no special privileges. The market struct is caller-supplied calldata; the liquidator constructs it.
- **Repeatability:** Repeatable on every block as long as the unhealthy position exists and the caller keeps using the invalid index.

### Recommendation

Add an explicit bounds check at the top of `liquidate()`, immediately after the existing input validation, before any access to `market.collateralParams[collateralIndex]`:

```solidity
require(collateralIndex < market.collateralParams.length, InvalidCollateralIndex());
```

This should be placed after line 600 (the `liquidatorGate` check) and before the while loop, so that all `collateralIndex`-dependent accesses — at lines 611, 644, 656, 664, 670, 673, 684, and 696 — are guarded by a single, early, descriptive revert. [9](#0-8)
