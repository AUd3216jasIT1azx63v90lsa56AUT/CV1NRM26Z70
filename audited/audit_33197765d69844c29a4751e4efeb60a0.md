### Title
`maxAssets` Cap Bypass via Zero-Rounding `buyerAssets` on Buy Offers with `buyerPrice < WAD` - (`File: src/Midnight.sol`)

### Summary
When a buy offer uses `maxAssets > 0` and `buyerPrice < WAD` (true for all valid ticks since `tickToPrice(MAX_TICK) < WAD`), taking with `units = 1` computes `buyerAssets = units.mulDivDown(buyerPrice, WAD) = 0`. Because `consumed` is incremented by `buyerAssets`, it is incremented by zero, and the guard `newConsumed <= offer.maxAssets` passes trivially. A taker can repeat this indefinitely on a fully-consumed offer, accruing unbounded credit/debt positions with zero asset exchange, bypassing the maker's intended cap entirely.

### Finding Description
**Code path** (`src/Midnight.sol` lines 363–369):

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause**: For a buy offer, `consumed` is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. When `units = 1` and `buyerPrice < WAD` (which holds for every tick ≤ `MAX_TICK` since `tickToPrice(MAX_TICK) < 1e18`), the result is `floor(buyerPrice / WAD) = 0`. The increment is zero, so `newConsumed` is unchanged, and the cap check `newConsumed <= offer.maxAssets` is trivially satisfied even when `consumed` is already at `maxAssets`.

**Attacker inputs**: Any unprivileged taker calls `take(offer, ..., units=1, ...)` on a buy offer where `offer.maxAssets > 0` and `offer.tick < MAX_TICK`. No special permissions are required; the taker only needs to satisfy the ratifier check (e.g., a permissionless ratifier).

**Exploit flow**:
1. Maker creates a buy offer: `offer.buy = true`, `offer.maxAssets = M`, `offer.tick = MAX_TICK - 16` (so `buyerPrice < WAD`).
2. Offer reaches `consumed == M` (fully consumed via normal takes or `setConsumed`).
3. Taker calls `take(offer, ..., units=1, ...)` repeatedly.
4. Each call: `buyerAssets = 1 * buyerPrice / WAD = 0`, `consumed` unchanged, check passes, maker gains 1 unit of credit, taker gains 1 unit of debt, `totalUnits` increases by 1.
5. No asset transfer occurs (`buyerAssets = sellerAssets = 0`), but position state mutates unboundedly.

**Why checks fail**: The `ConsumedAssets` guard only checks `consumed (in assets) <= maxAssets`. It does not bound the number of units filled. The Certora spec `Consume.spec` rule `takeConsumedBoundedByMax` (line 62) only asserts `consumed <= offer.maxAssets` in asset terms — it does not assert any unit-level bound.

The protocol's own test `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol` lines 857–889) explicitly demonstrates and names this behavior as a bug: a fully-consumed offer (`consumed == maxAssets == 1`) is successfully taken with `units = 1`, `buyerAssets = 0`, `sellerAssets = 0`, while `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase.

### Impact Explanation
The maker's `maxAssets` cap — which is explicitly documented as capping "max buyer assets if offer.buy is true" (`src/Midnight.sol` line 91) — is bypassed in unit terms. A fully-consumed buy offer can be taken indefinitely with `units = 1`, creating unbounded credit for the maker and unbounded debt for the taker with zero asset exchange. This violates the core invariant "offers cannot be replayed, overfilled, reused, or filled after cancel/deadline." The maker's credit exposure grows without bound beyond their intended limit, and the taker can force the maker into unlimited credit positions. The `consumed` counter and emitted `EventsLib.Take.consumed` field remain frozen at `maxAssets` while unit fills accumulate without limit.

### Likelihood Explanation
Preconditions are minimal: any buy offer with `maxAssets > 0` and any tick below `MAX_TICK` (which is every practically useful tick) is affected. The taker needs no special privileges, only the ability to call `take()` with `units = 1`. The attack is repeatable in a single transaction via `multicall`. The condition `buyerPrice < WAD` holds for all ticks since `tickToPrice(MAX_TICK) < 1e18` by construction of the price formula. The protocol's own test suite already encodes this exact scenario.

### Recommendation
Track consumed in units when `buyerPrice < WAD` would cause zero-rounding, or enforce a minimum `buyerAssets >= 1` when `units > 0`. Concretely, add a guard before the `consumed` update:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
}
```

Alternatively, if the intent is to allow zero-asset takes for callback purposes only, document and gate this path explicitly (e.g., require `units == 0` when `consumed >= maxAssets` for buy offers with `buyerPrice < WAD`). Makers who need a hard unit cap should use `maxUnits` instead of `maxAssets` when `buyerPrice < WAD`.

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) already proves this. An extended fuzz/invariant test:

```solidity
function testFuzz_maxAssetsBypassUnbounded(uint256 N) public {
    N = bound(N, 1, 1000);
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    collateralize(market, borrower, N + 10);
    deal(address(loanToken), lender, 0);

    // Pre-consume to maxAssets
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

    uint256 debtBefore = midnight.debtOf(id, borrower);
    for (uint256 i = 0; i < N; i++) {
        take(1, borrower, lenderOffer);
    }

    // consumed never moved, but N units of debt were created
    assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
    assertEq(midnight.debtOf(id, borrower), debtBefore + N);
    // Assert violation: total units filled >> maxAssets / buyerPrice
    assertGt(midnight.debtOf(id, borrower) - debtBefore, lenderOffer.maxAssets);
}
```
