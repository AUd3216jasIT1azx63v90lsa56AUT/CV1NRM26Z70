### Title
Direct ERC20 Transfers to Pool Captured by Any Staker via Balance-Diff Accounting - (File: src/ConfidencePool.sol)

### Summary
`stake()` measures received tokens via a pre/post balance-diff rather than trusting the `amount` parameter. Any tokens sent directly to the pool contract (outside of `stake()` / `contributeBonus()`) inflate the next caller's `received` figure and are silently credited to that caller's `eligibleStake`. An attacker who monitors the pool's token balance can capture a victim's accidentally-transferred tokens by immediately calling `stake()` with only the `minStake` amount.

### Finding Description
`stake()` computes the credited amount as the raw balance increase across the `safeTransferFrom` call:

```solidity
uint256 balanceBefore = stakeToken.balanceOf(address(this));
stakeToken.safeTransferFrom(msg.sender, address(this), amount);
uint256 balanceAfter  = stakeToken.balanceOf(address(this));
uint256 received = balanceAfter > balanceBefore ? balanceAfter - balanceBefore : 0;
```

`received` is then unconditionally credited to the caller:

```solidity
eligibleStake[msg.sender] += received;
```

There is no assertion that `received == amount`, and no guard that the pre-call balance equals the pool's tracked liabilities (`totalEligibleStake + totalBonus`). Consequently, any tokens sitting in the contract that arrived via a direct `transfer()` — rather than through `stake()` or `contributeBonus()` — are silently absorbed into `received` and attributed to the next `stake()` caller. The pool has no record of who sent those tokens, so there is no mechanism to return them to the original sender.

The same balance-diff pattern exists in `contributeBonus()` (lines 275–282), where a direct transfer inflates `totalBonus` for all stakers rather than a single caller's `eligibleStake`; the `stake()` path is the more severe vector because it grants the attacker a direct, exclusive claim on the stolen principal.

### Impact Explanation
A victim who accidentally sends tokens directly to the pool address (or is socially-engineered into doing so) loses those tokens permanently. The attacker calls `stake(minStake)`, paying only `minStake` out of pocket, and receives `eligibleStake` credit for `minStake + victim_amount`. After any terminal resolution (SURVIVED, EXPIRED, or CORRUPTED-bounty path), the attacker withdraws the victim's principal plus any accrued bonus share. The victim has no recourse: the pool tracks no original-sender metadata for direct transfers, and no recovery function exists for this case.

### Likelihood Explanation
The scenario requires a user to send tokens directly to the pool address rather than calling `stake()`. This can occur through wallet UI mistakes (sending to a contract address), copy-paste errors, or deliberate social engineering ("send tokens here to register"). The attacker's side requires only mempool monitoring and a single `stake()` call, which is permissionless and cheap. The pool is a public, named contract whose address is deterministic and discoverable, making it a plausible target.

### Recommendation
Assert that the balance increase equals the declared `amount` parameter, rejecting any call where pre-existing untracked tokens would inflate `received`:

```solidity
uint256 received = balanceAfter - balanceBefore;
require(received == amount, "unexpected balance delta");
```

Because the factory allowlist is restricted to standard ERC20 tokens (no fee-on-transfer, no rebasing), `received` should always equal `amount` for legitimate tokens. The balance-diff guard against a post-allowlist token upgrade is still preserved — a fee-on-transfer token would produce `received < amount` and revert, which is the correct rejection. This change eliminates the attack surface without removing the intended defense-in-depth.

### Proof of Concept

1. Pool is deployed with `minStake = 100`, `stakeToken = TOKEN`, outcome `UNRESOLVED`, `block.timestamp < expiry`.
2. Alice mistakenly calls `TOKEN.transfer(pool, 1000)` directly (not `stake()`). Pool balance is now 1000; `totalEligibleStake` remains 0.
3. Attacker Bob observes the balance discrepancy (on-chain or via mempool).
4. Bob calls `stake(100)`:
   - `balanceBefore = 1000`
   - `safeTransferFrom(Bob, pool, 100)` → pool balance becomes 1100
   - `received = 1100 - 1000 = 1100` ✓ passes `>= minStake`
   - `eligibleStake[Bob] += 1100`; `totalEligibleStake = 1100`
5. Pool resolves SURVIVED. Bob calls `claimSurvived()` and receives 1100 tokens + bonus share.
6. Alice's 1000 tokens are gone; Bob paid only 100 tokens to steal them. [1](#0-0) [2](#0-1) [3](#0-2)
