Audit Report

## Title
Oracle Reentrancy into `supplyCollateral` Corrupts Stale Local Bitmap in Liquidation Loop, Causing Excess Bad Debt Socialization - (File: src/Midnight.sol)

## Summary

In `Midnight.liquidate`, the collateral bitmap is snapshotted into a local stack variable at line 606 before the oracle-calling loop. Because `IOracle.price()` is an unconstrained external call and `supplyCollateral` has no reentrancy guard, an attacker-controlled oracle can reenter `supplyCollateral` during that call, writing a new collateral bit into storage. The local bitmap never reflects this update, so the loop exits without accounting for the newly added collateral's value. The resulting `badDebt` is inflated and socialized among lenders via `lossFactor`, while the borrower retains the injected collateral with their debt zeroed.

## Finding Description

**Root cause — stale local bitmap vs. live storage bitmap:**

At `src/Midnight.sol:606`, the bitmap is snapshotted into a stack variable:

```solidity
uint128 _collateralBitmap = _position.collateralBitmap;   // local copy, line 606
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price();  // unconstrained external call, line 610
    uint256 _collateral = _position.collateral[i];
    maxDebt += _collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(_collateralParam.lltv, WAD);
    badDebt = badDebt.zeroFloorSub(
        _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
    );
    _collateralBitmap = _collateralBitmap.clearBit(i);  // only local cleared, line 617
}
```

The loop only ever clears bits from the local copy. Any bit written to `_position.collateralBitmap` in storage during the `price()` call is invisible to the loop.

**Reentrancy path — `supplyCollateral` writes to storage bitmap:**

`supplyCollateral` has no reentrancy guard and is callable by any account authorized for `onBehalf`:

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
```

When a new collateral slot is first activated, it writes directly to the storage bitmap:

```solidity
if (oldCollateral == 0 && assets > 0) {
    uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
    _position.collateralBitmap = newCollateralBitmap;   // storage write, line 537
    ...
}
```

**Exploit flow (all steps attacker-controlled):**

1. Attacker (market creator) deploys a market with collateral index `i` using a `ReentrantOracle`, and collateral index `j > i` with a normal token. Both pass `touchMarket` validation.
2. Attacker (as borrower) supplies collateral `i`, borrows `D` units, becomes unhealthy (oracle price drops).
3. Attacker calls `setIsAuthorized(reentrantOracle, true, borrower)` — authorizes the oracle to supply on their behalf.
4. Attacker pre-funds `ReentrantOracle` with collateral `j` tokens and approves Midnight.
5. Anyone calls `liquidate(market, i, 0, 0, borrower, false, ...)` (seizedAssets=0, repaidUnits=0 — pure bad-debt realization path, explicitly supported per line 577–579 comments).
6. Loop starts: local `_collateralBitmap = {i}`.
7. Loop calls `ReentrantOracle.price()` for index `i`.
8. Oracle reenters: `supplyCollateral(market, j, largeAmount, borrower)` → storage bitmap becomes `{i, j}`, `_position.collateral[j] = largeAmount`.
9. Oracle returns `0` (making `maxDebt += 0`, `badDebt` unchanged).
10. Loop clears bit `i` from local bitmap → local bitmap = `0`. Loop exits.
11. Bit `j` was never processed. `badDebt = D` (full original debt).
12. `_position.debt -= D` (borrower's debt zeroed), `lossFactor` updated to slash lenders by `D`.
13. Borrower retains collateral `j` (never seized, never counted) and freely withdraws it since debt is now 0.

**Why existing checks do not stop it:**

`liquidationLocked` is only set during `take` callbacks via transient storage (line 444: `UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`). It is never set during `liquidate`'s oracle calls, so it does not block `supplyCollateral` reentrancy.

The `NotLiquidatable` check at lines 620–624 uses the stale `maxDebt` computed without collateral `j`. Since the oracle returns `0`, `maxDebt = 0`, and `originalDebt > maxDebt` passes trivially even though the position would be healthy if `j` were counted.

The TOKEN SAFETY REQUIREMENTS (lines 133–139) explicitly restrict token reentrancy but make no equivalent restriction on oracle contracts. There is no `nonReentrant` modifier on `liquidate` or `supplyCollateral`.

## Impact Explanation

Lenders are slashed via `lossFactor` by the full `badDebt = D` instead of the correct `badDebt = D − value(collateral_j)/maxLif_j`. The attacker (borrower) has their debt zeroed and retains collateral `j` — effectively extracting `value(collateral_j)/maxLif_j` worth of loan tokens from lenders at no cost. The invariant "bad debt realization must account for all active collateral" is violated: collateral `j` is active in storage at the time of realization but excluded from the computation. This constitutes direct theft of lender funds via inflated loss socialization.

## Likelihood Explanation

All preconditions are reachable by a single unprivileged actor: market creation is permissionless via `touchMarket`, the attacker is both market creator and borrower, and `setIsAuthorized` is callable by any user for their own account. The attack is repeatable across any market the attacker creates. The attacker can size collateral `j` to maximize lender loss up to the full debt amount `D`. No gate or guard blocks the reentrancy path.

## Recommendation

Add a reentrancy guard to `liquidate` (and/or `supplyCollateral`) using a transient storage lock analogous to `liquidationLocked`, set before the oracle-calling loop and cleared after. Alternatively, re-read `_position.collateralBitmap` from storage after each `price()` call and union it with the local bitmap so newly added bits are not missed. A third option is to explicitly document and enforce that oracle contracts must not re-enter Midnight, analogous to the existing token safety requirements, and validate this at market creation time where possible.

## Proof of Concept

**Minimal manual steps:**

1. Deploy `ReentrantOracle` that, on `price()`, calls `Midnight.supplyCollateral(market, j, largeAmount, borrower)` then returns `0`.
2. Deploy a market with two collateral params: index `0` → `ReentrantOracle`, index `1` → normal token.
3. As borrower: call `supplyCollateral(market, 0, amount, borrower)`, then `take(...)` to borrow `D` units.
4. Drop oracle price to make position unhealthy (oracle returns `0` anyway).
5. Call `setIsAuthorized(reentrantOracle, true, borrower)`.
6. Fund `ReentrantOracle` with collateral `1` tokens; approve Midnight.
7. Call `liquidate(market, 0, 0, 0, borrower, false, address(0), address(0), "")`.
8. Observe: borrower's debt is 0, borrower still holds collateral `1` in storage, lenders' `lossFactor` reflects full `D` slashed.
9. Call `withdrawCollateral(market, 1, largeAmount, borrower, borrower)` — succeeds since debt is 0.

**Invariant fuzz test:** Assert that after any `liquidate` call, `sum over all bits in _position.collateralBitmap of (collateral[i] * price[i] / maxLif[i]) + _position.debt == originalDebt`. The reentrancy breaks this invariant by leaving collateral `j` in the bitmap with nonzero balance while debt is zeroed.
