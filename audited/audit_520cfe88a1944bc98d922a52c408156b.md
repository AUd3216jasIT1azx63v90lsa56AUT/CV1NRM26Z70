### Title
Front-run of sibling group offer inflates `consumableUnits` snapshot, causing bundler to skip the target offer and revert with `OutOfOffers` - (File: src/periphery/ConsumableUnitsLib.sol)

### Summary

`consumableUnits` reads `consumed[offer.maker][offer.group]` as a view-time snapshot. Because `consumed` is a shared group budget, a concurrent fill of any sibling offer (same maker, same group) between the snapshot and the actual `take()` call increases the on-chain counter. The bundler then passes a `unitsToTake` value that is larger than the remaining group budget, causing `take()` to revert with `ConsumedUnits` (or `ConsumedAssets`). The bundler's `catch {}` silently skips the offer without retrying at the corrected amount, and if no subsequent offer covers the shortfall, the bundler reverts with `OutOfOffers`.

### Finding Description

**Code path:**

`MidnightBundles.supplyCollateralAndSellWithUnitsTarget` (and the three other bundle entry-points) compute `unitsToTake` as:

```solidity
// MidnightBundles.sol lines 147-151
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
```

`consumableUnits` (lines 14-17 of `ConsumableUnitsLib.sol`) reads the group counter at view time:

```solidity
uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
if (offer.maxUnits > 0) {
    return offer.maxUnits.zeroFloorSub(consumed);   // snapshot: maxUnits - C
```

`take()` in `Midnight.sol` (lines 371-372) re-reads the same counter at execution time and enforces the cap:

```solidity
newConsumed = consumed[offer.maker][offer.group] += units;
require(newConsumed <= offer.maxUnits, ConsumedUnits());
```

**Root cause:** There is no atomicity guarantee between the view call in `consumableUnits` and the state mutation in `take()`. The group counter `consumed[maker][group]` is shared across all offers with the same `(maker, group)` pair (documented in `Midnight.sol` lines 84-87 as the "One cancels the other" mechanism).

**Exploit flow (units mode):**

1. Maker publishes two offers, A and B, with the same `(maker, group)` and `maxUnits = N`. `consumed[maker][group] = C` initially.
2. Victim submits a bundler transaction targeting offer A with `targetUnits = N - C`.
3. Attacker observes the pending transaction in the mempool and front-runs it by calling `Midnight.take(offerB, ..., X, ...)` for any `X > 0`. This is permissionless — any address can be the taker.
4. `consumed[maker][group]` is now `C + X` on-chain.
5. Victim's bundler executes: `consumableUnits(offerA)` returns `N - C` (stale snapshot from before step 3).
6. Bundler calls `take(offerA, N - C)`. Inside `take()`: `newConsumed = (C + X) + (N - C) = N + X > N`. The check `require(newConsumed <= maxUnits)` reverts with `ConsumedUnits`.
7. The bundler's `catch {}` (line 160) silently swallows the revert and does **not** retry with the corrected amount `N - C - X`.
8. `filledUnits` is not incremented. If offer A was the last offer in the list, `require(filledUnits == targetUnits, OutOfOffers())` (line 163) reverts the entire bundler transaction.

The same path applies to `maxAssets` mode (revert is `ConsumedAssets`) and to all four bundle entry-points.

**Why existing checks do not stop it:**

- `take()` correctly enforces the group cap — it is not the bug. The bug is that the bundler computes `unitsToTake` from a stale snapshot and then passes that stale value to `take()` without any retry logic.
- The `try/catch` is designed to handle asynchrony, but it skips the offer entirely on revert rather than re-reading the available capacity and retrying.
- `zeroFloorSub` in `consumableUnits` prevents underflow but does not prevent the snapshot from being stale.

### Impact Explanation

Any user relying on `MidnightBundles` to fill a target amount across a list of offers that share a group budget can have their transaction griefed. The bundler reverts with `OutOfOffers`, the user's transaction fails (gas is wasted), and the user cannot fill their intended order in that block. In `supplyCollateralAndSellWithUnitsTarget`, collateral is supplied before the take loop, so the revert unwinds the collateral supply too — but the user still loses gas and misses the market opportunity. The attack is repeatable every block at negligible cost to the attacker (only gas for the sibling fill, which is a legitimate trade).

### Likelihood Explanation

**Preconditions:**
- The maker must have at least two offers sharing the same `(maker, group)` pair — this is the documented and intended use of groups.
- The victim's bundler transaction must be visible in the mempool (standard for non-private RPC).
- The attacker must be able to call `take()` on the sibling offer (permissionless for any non-maker address).

**Feasibility:** High on chains with public mempools. The attacker does not need any special privilege, capital lock-up, or oracle manipulation. A partial fill of even 1 unit on the sibling offer is sufficient to trigger the revert if the bundler's snapshot was computed at `consumed = maxUnits - targetUnits` exactly.

**Repeatability:** The attack can be repeated every block as long as the maker's group has sibling offers with remaining capacity.

### Recommendation

Inside the bundler loop, after catching a revert from `take()`, re-read `consumableUnits` and retry the take with the updated (smaller) amount before moving to the next offer. Concretely, replace the silent `catch {}` with a retry branch:

```solidity
try IMidnight(MIDNIGHT).take(..., unitsToTake, ...) returns (...) {
    filledUnits += unitsToTake;
    ...
} catch {
    // Re-read available capacity and retry once with the corrected amount.
    uint256 retryUnits = min(
        targetUnits - filledUnits,
        takes[i].units,
        ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
    );
    if (retryUnits > 0) {
        try IMidnight(MIDNIGHT).take(..., retryUnits, ...) returns (...) {
            filledUnits += retryUnits;
            ...
        } catch {}
    }
}
```

Alternatively, pass `min(takes[i].units, targetUnits - filledUnits)` directly to `take()` and let the core contract enforce the group cap, then read back the actual units filled from the return values — but this requires a core change to return the capped amount.

### Proof of Concept

```solidity
// Foundry unit test (stateful)
function testFrontRunSiblingGroupGriefsBundler() public {
    // Setup: maker has two offers A and B in the same group, maxUnits = 100 each.
    offersA.maxUnits = 100;
    offersB.maxUnits = 100;
    // offersA.group == offersB.group (same maker, same group bytes32)

    collateralize(market, borrower, 100);

    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offersA, units: 100, ratifierData: hex""});

    // Attacker front-runs: partially fills sibling offer B for 1 unit.
    // consumed[maker][group] goes from 0 to 1.
    vm.prank(attacker);
    midnight.take(offersB, hex"", 1, attacker, address(0), address(0), "");

    // Victim's bundler transaction executes.
    // consumableUnits(offersA) returns 100 (stale: read before attacker's fill).
    // take(offersA, 100) reverts: newConsumed = 1 + 100 = 101 > 100.
    // Bundler catches, skips offersA, filledUnits = 0 < targetUnits = 100.
    vm.prank(borrower);
    vm.expectRevert(IMidnightBundles.OutOfOffers.selector);
    midnightBundles.supplyCollateralAndSellWithUnitsTarget(
        100, 0, borrower, borrower, new CollateralSupply[](0), takes, 0, address(0)
    );

    // Assert: consumed[maker][group] == 1 (only attacker's fill), victim got nothing.
    assertEq(midnight.consumed(offersA.maker, offersA.group), 1);
    assertEq(midnight.debtOf(id, borrower), 0);
}
```

**Expected assertions:**
- `vm.expectRevert(IMidnightBundles.OutOfOffers.selector)` passes — the bundler reverts.
- `consumed[maker][group] == 1` — only the attacker's fill landed.
