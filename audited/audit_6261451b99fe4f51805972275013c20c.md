Audit Report

## Title
Fully-Consumed Buy Offer (maxAssets Mode) Bypassable via Zero-Rounding Units — (File: src/Midnight.sol)

## Summary
In `take()`, when `offer.buy == true` and `offer.maxAssets > 0`, the consumed-cap increment uses `mulDivDown(units, buyerPrice, WAD)`. For any tick below `MAX_TICK`, `buyerPrice < WAD`, so `units = 1` yields `buyerAssets = 0`. The cap check `newConsumed <= maxAssets` becomes a no-op when `consumed` is already at `maxAssets`, yet position state (`credit`, `debt`, `totalUnits`) is still mutated with zero token transfer, bypassing the maker's intended exposure limit and inflating unbacked accounting state.

## Finding Description
**Root cause:** The consumed-cap guard at lines 367–369 increments `consumed` by `buyerAssets`, which rounds to zero for `units * buyerPrice < WAD`. There is no `require(buyerAssets > 0)` or `require(newConsumed > consumedBefore)` guard. The position-mutation block at lines 408–417 operates on `units` directly, not on `buyerAssets`, so state changes occur regardless of the zero asset amount.

**Code path:**
- Line 363: `buyerAssets = units.mulDivDown(buyerPrice, WAD)` — rounds to 0 when `units=1` and `buyerPrice < WAD`.
- Lines 367–369: `consumed[maker][group] += 0`; `require(newConsumed <= maxAssets)` passes trivially.
- Lines 382–384: `buyerCreditIncrease`, `sellerDebtIncrease` computed from `units=1`, not from `buyerAssets`.
- Lines 408–417: `buyerPos.credit += 1`, `totalUnits += 1` — state mutated with no asset backing.
- Line 455: `safeTransferFrom(..., buyerAssets - sellerAssets)` = `safeTransferFrom(..., 0)` — zero transfer.

**Exploit flow:**
1. Maker creates a buy offer: `maxAssets = M`, `maxUnits = 0`, tick below `MAX_TICK` (so `buyerPrice < WAD`).
2. Offer is legitimately filled until `consumed[maker][group] == M`.
3. Attacker calls `take(offer, ..., units=1)`. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. `consumed` stays at `M`. Cap check passes.
4. `buyerPos.credit += 1`, `totalUnits += 1` — with zero token transfer.
5. Repeatable indefinitely via `multicall`.

**Existing checks reviewed and found insufficient:**
- `require(newConsumed <= offer.maxAssets)` — passes because increment is 0.
- No `require(units > 0)` guard exists in `take()`.
- No `require(buyerAssets > 0)` guard exists.

The NatSpec at line 94 acknowledges: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` confirms the protocol team treats this as a reproducible bug, not an accepted design choice.

## Impact Explanation
- **maxAssets cap bypass:** The maker's explicit exposure limit is rendered meaningless; their credit position grows without bound beyond `maxAssets` with no corresponding asset inflow.
- **Unbacked totalUnits inflation:** `totalUnits` grows without any asset deposit, breaking the invariant that every credit unit has a matching asset payment. All lenders' withdrawable share is diluted, and the `lossFactor` bad-debt socialization computation (which divides by `totalUnits`) is corrupted.
- **Griefing at zero cost:** The attacker pays only gas. The maker is forced into credit positions they explicitly capped, with no recourse.

## Likelihood Explanation
All preconditions are attacker-controllable with no privileged access: `offer.buy == true` and `offer.maxAssets > 0` are standard buy offer parameters; `buyerPrice < WAD` holds for any tick below `MAX_TICK` (the vast majority of real offers); `consumed == maxAssets` is reachable by normal fills. The attack requires no flash loan, no oracle manipulation, and no privileged keys. It is repeatable in the same transaction via `multicall`.

## Recommendation
Add a guard before the position-mutation block when `offer.maxAssets > 0` and `offer.buy == true`:

```solidity
require(buyerAssets > 0 || units == 0, ZeroAssets());
```

This ensures that a take with `units > 0` against a maxAssets-capped buy offer always results in a nonzero asset transfer, preventing zero-cost state mutation. Alternatively, enforce `require(units * buyerPrice >= WAD)` when `offer.maxAssets > 0 && offer.buy`.

## Proof of Concept
1. Deploy a market with any tick `t < MAX_TICK` (so `tickToPrice(t) < WAD`).
2. Maker creates buy offer: `maxAssets = 100e18`, `maxUnits = 0`, `tick = t`.
3. Fill the offer normally until `consumed[maker][group] == 100e18`.
4. Attacker calls `take(offer, ..., units=1)`:
   - `buyerAssets = mulDivDown(1, tickToPrice(t), WAD) = 0`
   - `consumed` stays at `100e18`; cap check passes
   - `buyerPos.credit += 1`, `totalUnits += 1`
   - Token transfer = 0
5. Assert `midnight.creditOf(id, maker) > 100e18` and `midnight.totalUnits(id)` increased without asset deposit.
6. Repeat step 4 N times via `multicall` to amplify inflation.
