Audit Report

## Title
Missing payer authorization in `repay()` allows arbitrary callback designation to drain third-party loanToken - (`src/Midnight.sol`)

## Summary
`repay()` authorizes `msg.sender` to repay debt on behalf of `onBehalf` but imposes no authorization check on the `callback` parameter, which becomes the token payer. Any account authorized to repay on behalf of a debtor can designate an arbitrary victim address as `callback`, causing Midnight to call `victim.onRepay(...)` and then pull loanToken directly from the victim. This is exploitable from within the `onSell` callback during `take()`, where the seller's authorized callback contract can re-enter `repay()` with `callback=victim` to retire the seller's debt at the victim's expense.

## Finding Description

**Root cause — `repay()` sets payer from unchecked `callback` parameter:**

```solidity
// src/Midnight.sol:505
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
// ...
address payer = callback != address(0) ? callback : msg.sender;   // payer = victim
// ...
IRepayCallback(callback).onRepay(id, market, units, onBehalf, data);  // called on victim
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // pulls from victim
```

The only gate is that `msg.sender` must be authorized to act on behalf of `onBehalf`. There is no check that `callback` (the payer) has authorized `msg.sender` to use it as a payer.

**Exploit flow:**

1. Attacker deploys `maliciousContract` and creates a sell offer with `offer.callback = maliciousContract`.
2. Attacker (seller) calls `setIsAuthorized(maliciousContract, true, seller)` — standard precondition for any callback contract that needs to act on the seller's behalf.
3. Any taker calls `take(offer, ...)`.
4. Inside `take()`, the liquidation lock is set for the seller at line 444.
5. After the buyer-side token pull, `onSell` is invoked on `maliciousContract` at lines 458–473.
6. Inside `onSell`, `maliciousContract` calls `repay(market, units, seller, callback=victim, data)`:
   - `msg.sender = maliciousContract`; `isAuthorized[seller][maliciousContract] = true` → authorization check at line 505 passes.
   - `payer = victim` (line 511).
   - `IRepayCallback(victim).onRepay(...)` is called; if victim returns `CALLBACK_SUCCESS`, execution continues.
   - `safeTransferFrom(loanToken, victim, midnight, units)` pulls `units` loanToken from victim (line 520).
7. Seller's debt is reduced. The post-callback health check at line 476 passes because the debt was repaid.

**Why existing checks fail:**

- The `repay` authorization check (`isAuthorized[onBehalf][msg.sender]`) only verifies the right to repay on behalf of the seller; it says nothing about who may be designated as payer.
- The liquidation lock prevents the seller from being liquidated during the callback window but does not restrict what the callback may do to third parties.
- The Certora spec `OnlyExplicitPayerCanLoseTokens.spec` sets `repayCallbackAllowed = false` for `take` (line 108), which would flag a violation if the re-entrant `repay` call were modeled. However, `onSell` is abstracted with `HAVOC_ALL` (per the comment at line 13–14: "onRatify and onSell cannot authorize a payer, so we model them with a plain HAVOC_ALL"), which prevents the tool from exploring the re-entrant `repay` path from within `onSell`. The assumption that `onSell` cannot authorize a payer is incorrect.

## Impact Explanation

Direct theft of loanToken from any victim contract that (a) implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS`, (b) has approved Midnight to spend its loanToken, and (c) holds a loanToken balance. The victim loses `units` loanToken; the attacker's (seller's) debt is retired at zero cost. This is an unauthorized movement of assets from a third party, satisfying the critical impact class of direct theft.

## Likelihood Explanation

**Required preconditions:**
1. Seller authorizes their callback contract — standard practice for any callback that calls `supplyCollateral` or similar on the seller's behalf.
2. Victim contract implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS` without validating `onBehalf` — realistic for any Midnight-integrated aggregator or vault that uses repay callbacks for its own operations.
3. Victim has approved Midnight to spend loanToken and holds a loanToken balance — true for any active repay-callback contract.

All three preconditions are normal operational states for Midnight integrators. The attack is repeatable as long as the victim contract remains deployed and approved. The attacker controls the entire setup off-chain before deployment.

## Recommendation

Add an authorization check on the `callback` (payer) parameter in `repay()`, mirroring the existing `onBehalf` check:

```solidity
require(callback == address(0) || callback == msg.sender || isAuthorized[callback][msg.sender], Unauthorized());
```

This ensures that only an address that has explicitly authorized `msg.sender` can be designated as the payer. Alternatively, remove the `callback`-as-payer pattern entirely and require the caller to always be the payer, with the callback used only for notification/reentrancy purposes (as is done in `liquidate` where the collateral is sent first and the callback is used to source the repayment).

## Proof of Concept

**Minimal Foundry test outline:**

1. Deploy `MockLoanToken` (standard ERC-20).
2. Deploy `Midnight`.
3. Deploy `VictimRepayCallback` that implements `onRepay` returning `CALLBACK_SUCCESS` unconditionally, holds 1000e18 loanToken, and has approved Midnight for `type(uint256).max`.
4. Deploy `MaliciousSellerCallback` that implements `onSell` and, inside it, calls `midnight.repay(market, 500e18, seller, address(victimRepayCallback), "")`.
5. Seller creates a sell offer with `offer.callback = address(maliciousSellerCallback)`.
6. Seller calls `midnight.setIsAuthorized(address(maliciousSellerCallback), true, seller)`.
7. Seller supplies collateral and takes on debt.
8. Taker calls `take(offer, ...)`.
9. Assert: `victimRepayCallback.loanToken.balanceOf(victim)` decreased by 500e18; seller's debt decreased by 500e18.

Expected result: victim loses 500e18 loanToken; seller's debt is retired at victim's expense. The transaction completes without revert.
