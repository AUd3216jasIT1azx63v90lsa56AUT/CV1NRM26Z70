### Title
Reentrant `withdraw` during `onBuy` drains pre-existing `withdrawable` before payer tokens are pulled - (`src/Midnight.sol`)

### Summary

In `take`, buyer credit is committed to storage before `onBuy` is called, but `withdrawable` is never increased during `take`. Because `withdraw` has no reentrancy guard and no check preventing it from being called inside a callback, a malicious `buyerCallback` that holds `isAuthorized[buyer][callback]` can call `withdraw` on behalf of `buyer` during `onBuy`, draining pre-existing `withdrawable` using the freshly-granted credit. The payer transfer happens only after `onBuy` returns, and when `buyerPrice < WAD` the payer pays fewer tokens than were withdrawn, breaking the solvency invariant.

### Finding Description

**Exact code path:**

`take` in `src/Midnight.sol`:

1. Lines 408–410: buyer credit is written to storage (`buyerPos.credit += buyerCreditIncrease`). `withdrawable` is untouched.
2. Line 422: `payer = buyerCallback` when `buyerCallback != address(0)`.
3. Lines 445–453: `IBuyCallback(buyerCallback).onBuy(..., buyer, ...)` is called. `buyer` is passed; `payer` is not.
4. Lines 455–456: `safeTransferFrom(payer, ...)` pulls tokens — only after `onBuy` returns.

`withdraw` in `src/Midnight.sol` lines 481–499:
- Only checks `isAuthorized[onBehalf][msg.sender]` (line 482).
- Decrements `_position.credit` and `_marketState.withdrawable`, then transfers `units` tokens (lines 493–499).
- No reentrancy guard, no maturity check, no callback-context guard.

**Attacker inputs (sell-offer path, attacker is taker/buyer):**

- Attacker deploys `MaliciousCallback` implementing `IBuyCallback`.
- Attacker calls `setIsAuthorized(callback, true, attacker)` → `isAuthorized[attacker][callback] = true`.
- Market has pre-existing `withdrawable >= units` (from prior repayments or liquidations).
- Attacker calls `take(sellOffer, ..., units, attacker, ..., callback, ...)` where `sellOffer.tick` is low enough that `buyerPrice < WAD`.

**Exploit flow:**

1. `take` sets `buyerPos.credit += units` (assuming no prior debt, `buyerCreditIncrease = units`).
2. `take` calls `callback.onBuy(..., buyer=attacker, ...)`.
3. Inside `onBuy`, callback calls `midnight.withdraw(market, units, attacker, attacker)`:
   - `isAuthorized[attacker][callback]` = true → passes auth check.
   - `attacker.credit -= units` (credit just granted in step 1).
   - `withdrawable -= units` (drains pre-existing pool).
   - Transfers `units` tokens to attacker.
4. `onBuy` returns `CALLBACK_SUCCESS`.
5. `take` calls `safeTransferFrom(callback, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(callback, receiver, sellerAssets)` — total `buyerAssets` tokens pulled from callback.
6. Since `buyerPrice < WAD`: `buyerAssets = units * buyerPrice / WAD < units`.

**Why existing checks fail:**

- No `nonReentrant` modifier on `take` or `withdraw`.
- The Certora `OnlyExplicitPayerCanLoseTokens.spec` models `onBuy` as `HAVOC_ALL` but does not model the callback re-entering `withdraw`; the `ContinuousFee.spec` explicitly states "Assume no reentrancy."
- The `tokenBalanceCorrect` strong invariant in `Solvency.spec` is verified under the same no-reentrancy assumption.
- `withdraw` has no guard against being called while a `take` is in progress.

### Impact Explanation

Protocol token balance decreases by `units − buyerAssets > 0` per attack. Pre-existing `withdrawable` (belonging to other lenders) is drained at a discount. At tick = 0 with zero settlement fee, `buyerAssets = 0` and the attacker extracts the full `units` tokens for free. The core solvency invariant — contract balances cover withdrawable assets — is violated: `withdrawable` is decremented but the corresponding tokens are not replaced by the payer.

### Likelihood Explanation

**Preconditions:**
1. Market must have `withdrawable > 0` — satisfied in any market that has seen at least one repayment or liquidation.
2. Attacker must hold `isAuthorized[buyer][callback]` — trivially self-granted by the attacker before calling `take`.
3. A sell offer (or buy offer created by attacker) with `buyerPrice < WAD` must exist or be creatable — any tick below the WAD-equivalent tick satisfies this; tick = 0 is always accessible.

The attack is fully permissionless, requires no privileged role, and is repeatable as long as `withdrawable > 0`.

### Recommendation

Add a transient-storage reentrancy guard to `withdraw` (and other state-mutating functions) that reverts if called while `take` (or any other function with an open callback window) is executing. Alternatively, snapshot `withdrawable` before the `onBuy` call and assert it is unchanged after. A targeted fix is to record a "callback active" flag in transient storage at the start of the `onBuy` dispatch and clear it after, then revert in `withdraw` if that flag is set.
