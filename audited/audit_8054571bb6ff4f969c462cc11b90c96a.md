### Title
`supplyCollateralAndSellWithAssetsTarget` reverts `UnitsTooHigh` when `maxUnits` covers only `targetSellerAssets` and `referralFeePct > 0` - (File: src/periphery/MidnightBundles.sol)

### Summary
When `referralFeePct > 0`, the function inflates the fill target to `targetFilledSellerAssets = targetSellerAssets + referralFeeAssets`, requiring more units to be sold than a user would naturally budget for in `maxUnits`. A taker who sets `maxUnits` to cover exactly `targetSellerAssets` at the current price — the semantically correct value per the NatSpec guarantee "The taker will lose at most maxUnits" — will always hit the `UnitsTooHigh` revert, making the function unusable with any non-zero referral fee unless `maxUnits` is inflated by the caller to account for the fee.

### Finding Description

**Code path:** [1](#0-0) 

```solidity
uint256 referralFeeAssets = targetSellerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
uint256 targetFilledSellerAssets = targetSellerAssets + referralFeeAssets;
```

The fill loop at line 282 runs until `filledSellerAssets >= targetFilledSellerAssets`, so `filledUnits` accumulates units sufficient to produce `targetFilledSellerAssets` in seller proceeds — not just `targetSellerAssets`. [2](#0-1) 

The guard at line 304 then checks:

```solidity
require(filledUnits <= maxUnits, UnitsTooHigh());
```

**Root cause:** `filledUnits` is driven by `targetFilledSellerAssets` (which is `targetSellerAssets * WAD / (WAD - referralFeePct)`), but `maxUnits` is naturally set by the caller to cover only `targetSellerAssets`. The ratio between the two is exactly `WAD / (WAD - referralFeePct)`, so for any `referralFeePct > 0`, `filledUnits > maxUnits` and the revert is deterministic.

**Attacker-controlled inputs:** The taker is the caller. They supply `referralFeePct > 0` (any value in `[1, WAD-1]`) and `maxUnits = ceil(targetSellerAssets / sellerPrice)` — the value the NatSpec implies is correct.

**Why existing checks do not stop it:** The only relevant guard is `require(referralFeePct < WAD)` at line 263, which permits any non-zero fee. The `UnitsTooHigh` check itself is the mechanism that fires, not a protection against it.

**Existing test coverage gap:** The only referral-fee test for this function (`testSellSellerAssetsTargetWithReferralFee`, line 630–669) hard-codes `maxUnits = type(uint256).max`, which bypasses the issue entirely. [3](#0-2) 

### Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` with `referralFeePct > 0` and `maxUnits` set to the units needed to sell `targetSellerAssets` — the value implied by the NatSpec — reverts with `UnitsTooHigh`. The taker's collateral supply sub-calls have already executed (lines 269–275) before the revert, but since the entire transaction reverts, state is rolled back. The concrete impact is a guaranteed, repeatable denial-of-service: the function cannot be used as documented when a referral fee is active and `maxUnits` is set to its natural value.

### Likelihood Explanation

**Preconditions:** `referralFeePct > 0` (any non-zero value) and `maxUnits` set to `ceil(targetSellerAssets / sellerPrice)`. Both are the natural, intended inputs per the NatSpec. No special market state, oracle manipulation, or privileged access is required. Any taker using a referral integration and following the documented semantics will hit this on every call.

### Recommendation

The `maxUnits` check should compare against the units needed to fill only `targetSellerAssets`, not `targetFilledSellerAssets`. One concrete fix: scale the check so that the referral-fee units are excluded:

```solidity
// Instead of:
require(filledUnits <= maxUnits, UnitsTooHigh());

// Compute the units attributable to targetSellerAssets only:
uint256 netUnits = filledUnits.mulDivDown(targetSellerAssets, targetFilledSellerAssets);
require(netUnits <= maxUnits, UnitsTooHigh());
```

Alternatively, update the NatSpec to explicitly state that `maxUnits` must be set to cover `targetFilledSellerAssets = targetSellerAssets * WAD / (WAD - referralFeePct)`, not `targetSellerAssets`, and provide the formula in the interface.

### Proof of Concept

```solidity
function testSellAssetsTargetReferralFeeUnitsTooHigh(
    uint256 targetSellerAssets,
    uint256 referralFeePct
) public {
    targetSellerAssets = bound(targetSellerAssets, 1e18, uint256(type(uint128).max) / 4);
    referralFeePct = bound(referralFeePct, 1, WAD / 2); // non-zero fee

    offers[0].maxUnits = type(uint256).max;

    uint256 price = TickLib.tickToPrice(MAX_TICK);
    midnight.touchMarket(market);
    uint256 sellerPrice = price - midnight.settlementFee(id, market.maturity - vm.getBlockTimestamp());

    // maxUnits set to cover exactly targetSellerAssets — the natural/documented value
    uint256 maxUnits = targetSellerAssets.mulDivUp(WAD, sellerPrice);

    // Collateralize enough for the inflated fill (targetFilledSellerAssets)
    uint256 referralFeeAssets = targetSellerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
    uint256 inflatedUnits = (targetSellerAssets + referralFeeAssets).mulDivUp(WAD, sellerPrice);
    collateralize(market, borrower, inflatedUnits + 1);

    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offers[0], units: type(uint256).max, ratifierData: hex""});

    vm.prank(borrower);
    vm.expectRevert(IMidnightBundles.UnitsTooHigh.selector); // always fires when referralFeePct > 0
    midnightBundles.supplyCollateralAndSellWithAssetsTarget(
        targetSellerAssets,
        maxUnits,          // covers targetSellerAssets, not targetFilledSellerAssets
        borrower,
        borrower,
        new CollateralSupply[](0),
        takes,
        referralFeePct,
        makeAddr("referrer")
    );

    // Key assertion: filledUnits > maxUnits because the loop targets targetFilledSellerAssets
}
```

Expected result: the fuzz test passes (always reverts) for all `referralFeePct > 0`, confirming the invariant `filledUnits <= maxUnits` is broken whenever `maxUnits` is set to its semantically correct value.
