### Title
Consumed not incremented in assets-mode buy offer when `buyerAssets` rounds to zero, enabling indefinite offer reuse — (File: src/Midnight.sol)

### Summary
When `offer.maxAssets > 0` and `offer.buy == true`, the `consumed` counter is incremented by `buyerAssets` rather than by `units`. At tick=0, `tickToPrice(0)` evaluates to exactly 0 (due to `PRICE_ROUNDING_STEP = 1e12` truncation), making `buyerAssets = 0` for any `units` input. This means `consumed` never increments, the `ConsumedAssets` guard never triggers, and the offer can be taken an unlimited number of times. When the buyer (maker) carries existing debt ≥ units, each take silently reduces the maker's debt with zero token transfer and zero consumed increment.

### Finding Description

**Root cause — `src/Midnight.sol:367-369`:**

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

`consumed` is incremented by the *asset* amount, not by `units`. [1](#0-0) 

**Asset computation — `src/Midnight.sol:363`:**

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
```

`mulDivDown(units, buyerPrice, WAD) = (units * buyerPrice) / WAD`. When `buyerPrice = 0`, this is 0 for any `units`. [2](#0-1) 

**`tickToPrice(0) = 0` — `src/libraries/TickLib.sol:44-52`:**

```
1e36 / (1e18 + wExp(LN_ONE_PLUS_DELTA * 2910))
≈ 1e36 / 2e24
= 5e11
```

Then `divHalfDownUnchecked(5e11, PRICE_ROUNDING_STEP=1e12) = 0`, so `tickToPrice(0) = 0 * 1e12 = 0`. [3](#0-2) 

This is confirmed by the existing test `testPriceZeroNoSettlementFeeSell` which asserts `buyerAssets == 0` at tick=0 with `units = 1e18`. [4](#0-3) 

**Exploit flow (preconditions met):**

1. `offer.maxAssets > 0`, `offer.buy == true`, `offer.tick = 0`, `_settlementFee = 0`
2. Maker has `buyerPos.debt >= units`
3. Taker calls `take(offer, ..., units=U)` for any `U > 0`
4. `buyerPrice = tickToPrice(0) = 0` → `buyerAssets = 0`
5. `consumed[maker][group] += 0` — no change; `require(0 <= maxAssets)` trivially passes
6. `buyerCreditIncrease = zeroFloorSub(U, debt) = 0` (debt ≥ U)
7. `buyerPos.debt -= U` — maker's debt silently reduced
8. `sellerAssets = 0` → both `safeTransferFrom` calls transfer 0 tokens
9. Seller health check at line 476 must pass; taker must have collateral if taking on new debt

Steps 3–9 repeat indefinitely; `consumed` never reaches `maxAssets`. [5](#0-4) 

**Why existing checks fail:**

- `require(newConsumed <= offer.maxAssets)`: `newConsumed` is unchanged, so this always passes. [6](#0-5) 
- `require(liquidationLocked(id, seller) || isHealthy(...))`: constrains the taker but does not prevent the consumed accounting gap. [7](#0-6) 
- The Certora `takeConsumedDelta` rule only covers units mode (`require offer.maxAssets == 0`) and has no equivalent for assets mode. [8](#0-7) 

The protocol comment acknowledges "It is possible to give units to a fully consumed assets-based buy offer with price < 1" — but this only describes the already-at-max case; it does not sanction indefinite reuse of a non-fully-consumed offer. [9](#0-8) 

### Impact Explanation
The offer is never exhausted: `consumed` stays at its initial value regardless of how many times `take` is called with `units > 0`. Each call reduces the maker's debt by `units` with zero token transfer and zero consumed increment. A maker with debt can have that debt drained to zero by a cooperating taker (or a taker contract the maker controls via a separate authorized address) at no token cost to either party, violating the invariant that offers cannot be reused or overfilled.

### Likelihood Explanation
Preconditions are reachable without privilege:
- `offer.tick = 0` is a valid tick (tick spacing allows it; `tick % spacing == 0` passes for spacing=1 or any divisor).
- `offer.maxAssets > 0` is a normal offer configuration.
- `_settlementFee = 0` is the default state for a freshly created market.
- Maker having debt in the same market is a normal borrower state.
- The taker only needs to be a different address from the maker and must remain healthy.

The attack is repeatable in a loop within a single transaction via multicall or across multiple transactions.

### Recommendation
In assets mode, when `units > 0`, either:
1. **Require `buyerAssets > 0`** (or `sellerAssets > 0` for sell offers) when `units > 0` — i.e., add `require(units == 0 || (offer.buy ? buyerAssets : sellerAssets) > 0)` before the consumed update; or
2. **Increment `consumed` by `units` as a floor** when the asset amount rounds to zero — e.g., `consumed[...] += max(offer.buy ? buyerAssets : sellerAssets, units > 0 ? 1 : 0)`.

Option 1 is simpler and consistent with the existing revert on settlement-fee underflow for buy offers at tick=0 with non-zero fees.

### Proof of Concept

```solidity
// Foundry unit test
function testConsumedNotIncrementedOnZeroAssetsBuyOffer() public {
    // Setup: buy offer, assets mode, tick=0 (price=0), no settlement fee
    Offer memory offer;
    offer.buy = true;
    offer.maker = borrower;       // borrower has existing debt
    offer.tick = 0;
    offer.maxAssets = 1e18;       // assets mode
    offer.maxUnits = 0;
    offer.ratifier = address(dummyRatifier);
    offer.group = bytes32(0);

    // Give borrower debt first (e.g., via a prior take at a normal tick)
    // ... setup borrower with debt >= units ...

    uint256 consumedBefore = midnight.consumed(borrower, offer.group);
    uint256 debtBefore = midnight.debtOf(id, borrower);

    // Take with units=1; buyerAssets = mulDivDown(1, 0, WAD) = 0
    vm.prank(lender);
    midnight.take(offer, hex"", 1, lender, address(0), address(0), hex"");

    // EXPECTED (correct): consumed increments
    // ACTUAL (buggy): consumed unchanged
    assertGt(midnight.consumed(borrower, offer.group), consumedBefore, "consumed must increment");

    // Also assert debt decreased without token transfer
    assertLt(midnight.debtOf(id, borrower), debtBefore, "debt decreased");
    assertEq(loanToken.balanceOf(address(midnight)), /* unchanged */ 0, "no token transfer");

    // Demonstrate reuse: second take also succeeds and consumed still unchanged
    vm.prank(lender);
    midnight.take(offer, hex"", 1, lender, address(0), address(0), hex"");
    assertGt(midnight.consumed(borrower, offer.group), consumedBefore + 1, "consumed must increment again");
}
```

Expected assertion failure on `"consumed must increment"` demonstrates the bug. An invariant fuzz test parameterizing `units` over `[1, WAD-1]` at tick=0 with `maxAssets > 0` would confirm the same for all sub-WAD unit values.

/// After a successful take in units mode, the change in consumed equals the units taken.
rule takeConsumedDelta(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumed(offer.maker, offer.group) == consumedBefore + units;
}
