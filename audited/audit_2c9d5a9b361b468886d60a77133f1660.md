Audit Report

## Title
Buy Offer `maxAssets` Cap Bypassed via `mulDivDown` Rounding to Zero at Sub-WAD Prices - (File: `src/Midnight.sol`)

## Summary
When a buy offer uses `maxAssets` as its fill cap and `units * buyerPrice < WAD`, `mulDivDown` returns 0, causing the `consumed` counter to be incremented by zero. The `ConsumedAssets` guard passes trivially while the full `units` amount is still applied to `buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits`, creating phantom positions with no token backing. This is confirmed by an existing protocol test named `testBugBuyMaxAssetsBypass` and explicitly noted in the contract's NatDoc comments.

## Finding Description
**Root cause** — `src/Midnight.sol`, `take()`, lines 363–373:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// ...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For a buy offer, `buyerPrice = tickToPrice(tick) + settlementFee`. At any sub-WAD tick (the normal operating range), `buyerPrice < WAD`. When `units = 1`, `1 * buyerPrice < WAD` is trivially true, so `mulDivDown` returns 0. The `consumed` mapping is incremented by 0, and `require(newConsumed <= offer.maxAssets)` passes regardless of how many times `take` is called.

Despite the zero asset increment, lines 408–417 apply the full `units` to positions:

```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);
_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease;
```

The token transfer at line 455 (`safeTransferFrom(..., buyerAssets - sellerAssets)`) also transfers 0 tokens, so no loan tokens move while positions are mutated.

**Existing guard failure**: The `ConsumedAssets` guard is purely asset-denominated. When `buyerAssets = 0`, it is structurally bypassed and cannot enforce any unit-level limit.

**Protocol acknowledgment**: Line 94 of `src/Midnight.sol` states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test at `test/TakeTest.sol:858–889` (`testBugBuyMaxAssetsBypass`) pre-fills `consumed` to `maxAssets`, calls `take(units=1)`, and asserts that `consumed` stays at `maxAssets` and token balances are unchanged, while `creditOf`, `debtOf`, and `totalUnits` all strictly increase — confirming the exploit path.

## Impact Explanation
A maker who posts a buy offer with `maxAssets = N` intends to cap their total credit exposure at `N` loan tokens. An attacker can call `take(units=1)` in a loop, accumulating arbitrarily large credit/debt positions while the consumed counter never exceeds `N` and no tokens are transferred. The inflated `_marketState.totalUnits` directly corrupts the loss factor calculation in `liquidate()` (lines 631–634), where `lossFactor` is computed as a function of `_totalUnits`. An artificially large `totalUnits` dilutes the socialized loss per unit when bad debt is realized, harming all lenders in the market. Phantom credit positions also affect `continuousFeeCredit` accrual and settlement outcomes for all participants.

## Likelihood Explanation
Preconditions are minimal: any buy offer with `maxAssets > 0` and `buyerPrice < WAD` — satisfied at all sub-WAD ticks, which is the normal operating range. No special privileges are required; line 346 confirms `taker == msg.sender` is sufficient. The condition `units * buyerPrice < WAD` is trivially satisfied with `units = 1` at any realistic price. The attack is repeatable indefinitely at gas cost only, with no economic barrier.

## Recommendation
Track consumed exposure in units when `buyerAssets` rounds to zero. One approach: when `offer.maxAssets > 0` and `buyerAssets == 0`, either revert (preventing zero-asset fills against an asset-capped offer) or additionally enforce a unit-level sub-cap. Alternatively, use `mulDivUp` for the consumed increment on buy offers so that any nonzero `units` always increments `consumed` by at least 1, preventing the guard from being a no-op. The fix should ensure that `consumed` is incremented by a strictly positive amount whenever `units > 0` and `offer.maxAssets > 0`.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` at lines 858–889 is a complete, self-contained reproduction:

1. Set `lenderOffer.maxAssets = 1` and `lenderOffer.tick = MAX_TICK - 16` (sub-WAD price).
2. Pre-fill `consumed` to `maxAssets` via `midnight.setConsumed(...)`.
3. Call `take(units=1, borrower, lenderOffer)`.
4. Assert: `buyerAssets == 0`, `sellerAssets == 0`, `consumed == maxAssets` (unchanged), token balances unchanged.
5. Assert: `creditOf(id, lender) > lenderCreditBefore`, `debtOf(id, borrower) > borrowerDebtBefore`, `totalUnits(id) > totalUnitsBefore`.

All five assertions pass, confirming phantom position creation with no token movement and no consumed counter increment.
