Audit Report

## Title
Debt increase permitted at exact maturity timestamp due to `<=` boundary check - (File: src/Midnight.sol)

## Summary
The `take()` function at line 391 of `src/Midnight.sol` uses `block.timestamp <= offer.market.maturity` to guard against post-maturity debt increases. When `block.timestamp == offer.market.maturity`, this condition evaluates to `true`, causing the `||` short-circuit to bypass the `sellerDebtIncrease == 0` check entirely. This directly violates the explicit protocol invariant that "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing," and the core invariant that "debt must not increase after maturity."

## Finding Description
**Root cause:** Line 391 of `src/Midnight.sol`:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
When `block.timestamp == offer.market.maturity`, the left operand is `true`, making the entire `||` expression unconditionally `true`. The require never reverts regardless of `sellerDebtIncrease`.

**Exact code path:**
1. Line 359: `timeToMaturity = zeroFloorSub(offer.market.maturity, block.timestamp)` returns `0` at equality. All price/fee computations proceed normally.
2. Line 383: `sellerCreditDecrease = min(units, sellerPos.credit)`. If the seller has no existing credit, this is `0`.
3. Line 384: `sellerDebtIncrease = units - sellerCreditDecrease = units > 0`.
4. Line 391: Guard passes unconditionally because `block.timestamp <= offer.market.maturity` is `true`.
5. Line 414: `sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease)` — new debt is minted at maturity.

**Why existing checks fail:** The expiry check at line 353 (`block.timestamp <= offer.expiry`) is orthogonal. The `reduceOnly` check at line 392–395 only applies if the maker set that flag. The gate checks at lines 402–406 only apply if a gate is configured and do not substitute for the maturity guard. No other check in `take()` prevents debt increase at the exact maturity timestamp. All existing post-maturity tests in `test/TakeTest.sol` use `market.maturity + 1` as the warp target (lines 317, 330, 349, 391, 406, 667, 680), leaving the `block.timestamp == offer.market.maturity` case untested.

## Impact Explanation
A taker can cause `sellerPos.debt` to increase at the exact maturity block. This corrupts the debt/credit accounting invariant: a borrower who should be in a pure-repayment/settlement phase has new debt minted against them. This can push a previously healthy position into an unhealthy state, enabling liquidation of a position that should have been protected from new debt. The impact class is "credit/debt accounting corruption" and "bad debt creation," both listed as highest-priority bug classes in `live_context.json`. The violated invariants are explicitly stated in `live_context.json` lines 197 and 221.

## Likelihood Explanation
The precondition `block.timestamp == offer.market.maturity` is a single-block window that occurs exactly once per market. A taker monitoring the chain can submit a transaction in that specific block. The offer requires only a valid ratifier (e.g., a signature-based ratifier the maker already authorized) and a seller with no existing credit. No privileged access, oracle manipulation, or admin action is required. This is repeatable across every market whose maturity falls on a future block.

## Recommendation
Change `<=` to `<` at line 391 of `src/Midnight.sol`:
```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
This ensures that when `block.timestamp == offer.market.maturity`, the left operand is `false`, forcing the require to evaluate `sellerDebtIncrease == 0` and revert if any new debt would be minted. Add a Foundry test warping to exactly `market.maturity` and asserting `CannotIncreaseDebtPostMaturity` reverts for debt-increasing takes.

## Proof of Concept
```solidity
function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;
    // Warp to EXACT maturity (not maturity + 1)
    vm.warp(market.maturity);
    borrowerOffer.expiry = market.maturity;
    borrowerOffer.maxUnits = units;
    deal(address(loanToken), lender, units);
    collateralize(market, borrower, units);

    // This should revert with CannotIncreaseDebtPostMaturity but does NOT
    // under the current <= guard — it succeeds and mints debt at maturity.
    take(units, lender, borrowerOffer);

    // Debt was minted at maturity — invariant violated
    assertGt(midnight.debtOf(id, borrower), 0);
}
```
Run against the current code: the take succeeds and `debtOf(borrower) > 0` at `block.timestamp == market.maturity`. After changing `<=` to `<`, the take reverts with `CannotIncreaseDebtPostMaturity`.
