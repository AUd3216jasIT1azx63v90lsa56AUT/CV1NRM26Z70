Audit Report

## Title
Zero-unit `take` unconditionally dispatches `onBuy` to attacker-supplied `takerCallback` - (File: `src/Midnight.sol`)

## Summary
The `take` function in `src/Midnight.sol` dispatches the `onBuy` callback guarded only by `buyerCallback != address(0)`, with no check that `units > 0` or `buyerAssets > 0`. For a sell offer, `buyerCallback` is set directly from the caller-supplied `takerCallback` parameter. An unprivileged attacker can pass any `IBuyCallback`-implementing address as `takerCallback` and call `take` with `units=0`, causing `onBuy` to fire on that address with zero-value parameters while no tokens move and the offer's fill counter is never advanced, making the attack indefinitely repeatable.

## Finding Description

**Root cause:** Missing `units > 0` (or `buyerAssets > 0`) guard before the `onBuy` dispatch.

**Exact code path:** [1](#0-0) 

With `units=0`, both `buyerAssets` and `sellerAssets` resolve to `0` via `mulDivDown`/`mulDivUp`. [2](#0-1) 

The `consumed` accounting adds `0` in both branches (`buyerAssets=0` or `units=0`), so `newConsumed` is never advanced and the fill-limit `require` always passes. The attack is indefinitely repeatable against the same offer. [3](#0-2) 

For a sell offer (`offer.buy = false`), `buyerCallback` is assigned directly from the caller-controlled `takerCallback` parameter. [4](#0-3) 

The sole guard before dispatching `onBuy` is `buyerCallback != address(0)`. There is no `units > 0` or `buyerAssets > 0` check. [5](#0-4) 

Both `safeTransferFrom` calls execute with amount `0` — no tokens move.

**Exploit flow:**
1. Attacker calls `take(sellOffer, ratifierData, 0, taker, receiver, victimContract, data)`.
2. `buyerAssets = 0`, `sellerAssets = 0`; consumed counter unchanged.
3. `buyerCallback = victimContract` (attacker-supplied).
4. `IBuyCallback(victimContract).onBuy(id, market, 0, 0, 0, buyer, data)` is called.
5. Victim contract executes its full `onBuy` body (state mutations, approvals, nested calls, event emissions).
6. Both `safeTransferFrom` calls are no-ops; no tokens move.
7. Step 1–6 can be repeated indefinitely against the same offer.

**Why existing checks fail:**
- `require(taker == msg.sender || isAuthorized[taker][msg.sender])` validates taker identity only, not the callback target.
- The `consumed` accounting adds `0`, so fill limits are never reached.
- `WrongBuyCallbackReturnValue` only reverts if the victim returns the wrong magic value; any well-formed `IBuyCallback` returns `CALLBACK_SUCCESS` and the call succeeds.
- `require(UtilsLib.atMostOneNonZero(offer.maxAssets, offer.maxUnits))` permits both fields to be zero (unlimited offer), which also passes.

## Impact Explanation
Any contract implementing `IBuyCallback` that does not internally assert `units > 0` or `buyerAssets > 0` can have its `onBuy` logic triggered by an arbitrary unprivileged caller with zero-value parameters and attacker-controlled `callbackData`. This constitutes unauthorized state changes on third-party contracts: token approvals, storage mutations, event emissions, or reentrancy paths can all be triggered without any real trade occurring and without the victim contract's operator initiating or consenting to a trade. Because the attack is repeatable at zero cost (no tokens required, fill counter never advances), it can be used to spam callbacks, drain gas, or repeatedly trigger any side-effectful logic in victim contracts.

## Likelihood Explanation
Preconditions are minimal: a valid live sell offer (any public offer works), a victim contract address implementing `IBuyCallback` that returns `CALLBACK_SUCCESS`, and the ability to call `take` as `taker`. No privileged access, no oracle manipulation, no special role. The attack is fully repeatable because `units=0` never advances `consumed`.

## Recommendation
Add an explicit non-zero units guard at the top of `take`, before any state mutations or callback dispatches:

```solidity
require(units > 0, ZeroUnits());
```

Alternatively, add the guard immediately before the `onBuy` dispatch:

```solidity
if (buyerCallback != address(0) && units > 0) { ... }
```

The former is preferable as it also prevents zero-unit position state mutations and zero-amount `safeTransferFrom` calls.

## Proof of Concept
```solidity
// Minimal forge test
function testZeroUnitCallbackDispatch() public {
    // Deploy a victim IBuyCallback that records invocations
    VictimCallback victim = new VictimCallback();

    // Use any existing live sell offer (offer.buy = false)
    Offer memory sellOffer = _buildLiveSellOffer();

    // Call take with units=0 and takerCallback=victim
    vm.prank(attacker);
    midnight.take(sellOffer, ratifierData, 0, attacker, attacker, address(victim), "");

    // Victim's onBuy was called despite zero units and zero assets
    assertEq(victim.callCount(), 1);

    // Repeat — offer fill counter was not advanced
    vm.prank(attacker);
    midnight.take(sellOffer, ratifierData, 0, attacker, attacker, address(victim), "");
    assertEq(victim.callCount(), 2);
}
```

`VictimCallback.onBuy` should assert `units == 0` and `buyerAssets == 0` to confirm the zero-value invocation, and return `CALLBACK_SUCCESS` to allow the call to complete.
