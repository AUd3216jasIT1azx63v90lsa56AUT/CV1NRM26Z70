Audit Report

## Title
Fee-on-Transfer Loan Token Causes Permanent `claimableSettlementFee` Overstatement in `take` - (File: src/Midnight.sol)

## Summary
In `Midnight.take`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` at line 418 before the corresponding `safeTransferFrom` at line 455. When the loan token is fee-on-transfer, the contract receives only `(buyerAssets - sellerAssets) * (1 - f)` but records the full nominal amount, creating a permanent per-take shortfall. This violates the solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee`, and the shortfall accumulates linearly across repeated takes.

## Finding Description
**Root cause:** At `src/Midnight.sol:418`, `claimableSettlementFee` is incremented by the full nominal fee before the token transfer occurs. At lines 455ŌĆō456, `SafeTransferLib.safeTransferFrom` executes the actual pull but does not verify the amount received. With a fee-on-transfer token at rate `f`, the contract receives `(buyerAssets - sellerAssets) * (1 - f)` but records `buyerAssets - sellerAssets`, leaving a permanent shortfall of `(buyerAssets - sellerAssets) * f` per take.

**Why existing checks fail:** The Certora `Solvency.spec` explicitly assumes at line 31: *"no fee taking from sender or receiver, no rebasing."* The `CVL_transferFrom` summary at lines 50ŌĆō57 transfers the exact requested amount, so the `pendingFeeReceipt` ghost always reaches zero after the transfer. With a real fee-on-transfer token, the transfer only partially closes the `pendingFeeReceipt` gap, leaving a residual that the formal model never captures. Additionally, `multicall` is summarized as `HAVOC_ALL DELETE` (line 4 of Solvency.spec), meaning no multicall path is formally verified at all.

**Exploit path:** An unprivileged taker with sufficient balance and approval calls `take` (or batches N calls via `multicall` using distinct `offer.group` values) on a market whose loan token is fee-on-transfer. Each call increments `claimableSettlementFee` by the full nominal spread while the contract receives a fee-discounted amount. After N takes, `claimableSettlementFee` exceeds actual held tokens by `N * (buyerAssets - sellerAssets) * f`.

## Impact Explanation
The solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee` is violated. When the fee claimer calls `claimSettlementFee` for the full recorded amount, the transfer either reverts (insufficient balance) or drains tokens backing lender `withdrawable` credits, causing lender withdrawals to fail. The shortfall is permanent and non-reversible. The insolvency magnitude scales linearly with the number of takes and the fee rate, making it unbounded over time.

## Likelihood Explanation
Preconditions: the loan token must be fee-on-transfer; the taker needs sufficient balance and approval. No governance access, oracle manipulation, or privileged keys are required. Any taker can trigger this in a single transaction. The attack is repeatable across multiple transactions, accumulating unbounded shortfall. Fee-on-transfer tokens are a realistic and deployed token class (e.g., tokens with deflationary mechanics or protocol fees on transfer).

## Recommendation
After each `safeTransferFrom` to `address(this)`, measure the actual received amount by comparing pre- and post-transfer balances, and use the measured amount to increment `claimableSettlementFee` rather than the nominal `buyerAssets - sellerAssets`. Alternatively, explicitly document and enforce (e.g., via a market creation check or a known-token registry) that fee-on-transfer tokens are unsupported as loan tokens, and add a corresponding exclusion to SECURITY.md.

## Proof of Concept
1. Deploy a mock ERC20 with a 1% transfer fee (sender pays `amount`, recipient receives `amount * 0.99`).
2. Create a Midnight market with this token as `loanToken`.
3. Place a buy offer with `buyerAssets = 1000`, `sellerAssets = 990` (spread = 10).
4. Call `take` once: `claimableSettlementFee` increases by 10, but contract receives only 9.9 tokens for fees.
5. Assert `claimableSettlementFee[token] > tokenBalance[address(this)] - collateralSum - withdrawableSum`.
6. Repeat step 4 N times (or batch via `multicall`): shortfall grows to `N * 0.1`.
7. Call `claimSettlementFee` for the full recorded amount: transaction reverts or drains lender funds.
