Audit Report

## Title
Authorized-Signer Front-Run Advances Victim's Nonce and Invalidates Pending Signed Authorization - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` permits any address that `authorization.authorizer` has previously authorized in Midnight to sign and submit an `Authorization` struct with attacker-chosen `authorized` and `isAuthorized` values at the current nonce. Because the nonce is keyed solely on `authorization.authorizer` and increments on any valid delegated-signer call, an authorized attacker can front-run the victim's pending transaction, consume nonce N with a different struct, and permanently invalidate the victim's signed authorization while also writing arbitrary `isAuthorized` entries on the victim's behalf.

## Finding Description
**Root cause:**

The nonce in `EcrecoverAuthorizer` is keyed only on `authorization.authorizer` and increments on any valid call from any authorized signer. There is no binding between the signer's identity and the `authorized` field, and no constraint that a delegated signer may only submit a struct the authorizer themselves pre-approved.

```solidity
// src/periphery/EcrecoverAuthorizer.sol line 26
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

// lines 33-36
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

The `Authorization` struct (`src/periphery/interfaces/IEcrecoverAuthorizer.sol`) contains five independently attacker-controllable fields: `authorizer`, `authorized`, `isAuthorized`, `nonce`, `deadline`. A delegated signer can freely choose any values for `authorized` and `isAuthorized` — there is no check that the struct content was pre-approved by the authorizer.

**Exploit flow:**

1. **Precondition:** `IMidnight.isAuthorized(victim, attacker) == true` (victim previously authorized attacker as an operator/relayer — a routine, expected state). Additionally, `IMidnight.isAuthorized(victim, EcrecoverAuthorizer) == true` (required for the contract to call `Midnight.setIsAuthorized` on behalf of the victim — this is the standard setup for any user of `EcrecoverAuthorizer`).
2. Victim signs `Authorization{authorizer=victim, authorized=X, isAuthorized=true, nonce=N}` and broadcasts.
3. Attacker observes victim's tx in mempool; crafts `Authorization{authorizer=victim, authorized=attacker_controlled_addr, isAuthorized=true, nonce=N}` (same nonce, different `authorized`).
4. Attacker signs this struct with their own private key → `ecrecover(digest, ...) == attacker`.
5. Attacker submits with higher gas priority.
6. Contract execution:
   - Line 25: deadline check passes (attacker sets any valid future deadline).
   - Line 26: `N == nonce[victim]` → passes; nonce increments to N+1.
   - Line 31: `signer = attacker`.
   - Lines 33–36: `IMidnight.isAuthorized(victim, attacker) == true` → passes.
   - Line 47: `IMidnight.setIsAuthorized(attacker_controlled_addr, true, victim)` executes — attacker writes arbitrary authorization entries on victim's behalf.
7. Victim's tx arrives: `N != N+1` → reverts with `InvalidNonce()`.

**Why existing checks fail:**

- The nonce check prevents replay of the *same* signed message but does not prevent a *different* message at the same nonce from a delegated signer.
- The `isAuthorized` delegation path was designed to allow authorized agents to act on behalf of the authorizer, but it does not scope what struct content the agent may sign — the agent can sign any `authorized`/`isAuthorized` combination.
- There is no binding between the signer's identity and the `authorized` field in the struct.

## Impact Explanation
The victim's pending signed `Authorization` is permanently invalidated (DoS of delegation). The victim must sign a new message at nonce N+1, which the attacker can front-run again indefinitely. Additionally, the attacker can write arbitrary `isAuthorized` entries on behalf of the victim — authorizing malicious addresses or revoking existing authorizations — which constitutes unauthorized state mutation of the victim's authorization state in Midnight. This maps to: unauthorized state changes, privilege escalation from authorized-agent to full authorization-manager, and permanent service degradation for the victim's signature-gated delegation path.

## Likelihood Explanation
The precondition (`isAuthorized[victim][attacker] == true`) is a common, expected state — users routinely authorize operators, bots, and relayers in Midnight for trading and other operations. The second precondition (`isAuthorized[victim][EcrecoverAuthorizer] == true`) is required for any user of `EcrecoverAuthorizer` and is therefore always present when the victim is using the contract. Once both conditions are met, the attack requires only mempool monitoring and a higher-gas competing transaction, which is standard MEV infrastructure. The attack is trivially repeatable on every future signed authorization the victim broadcasts, making it a persistent griefing vector with no natural expiry.

## Recommendation
Bind the signer's identity to the `authorized` field in the struct. Specifically, when the signer is a delegated operator (not the authorizer themselves), enforce that `signer == authorization.authorized` — i.e., only the address being authorized/deauthorized may sign on behalf of the authorizer. Alternatively, require that delegated signers may only submit structs where `authorization.authorized == signer`, preventing them from writing arbitrary entries. A stricter fix is to remove the delegated-signer path entirely and require that only `authorization.authorizer` may sign the struct, which eliminates the attack surface while preserving the primary use case of off-chain signed authorizations.

## Proof of Concept
Minimal Foundry test extending `SetIsAuthorizedWithSigTest`:

```solidity
function testFrontRunInvalidatesVictimAuth() public {
    // Setup: victim authorizes ecrecoverAuthorizer and attacker in Midnight
    vm.prank(borrower); // borrower = victim
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);
    vm.prank(borrower);
    midnight.setIsAuthorized(lender, true, borrower); // lender = attacker

    // Victim signs Authorization{authorizer=victim, authorized=otherBorrower, isAuthorized=true, nonce=0}
    Authorization memory victimAuth = makeAuthorization(borrower, otherBorrower, true);
    Signature memory victimSig = signAuthorization(victimAuth, borrower);

    // Attacker crafts Authorization{authorizer=victim, authorized=lender, isAuthorized=true, nonce=0}
    Authorization memory attackerAuth = makeAuthorization(borrower, lender, true);
    Signature memory attackerSig = signAuthorization(attackerAuth, lender); // attacker signs with own key

    // Attacker front-runs: submits with higher gas
    ecrecoverAuthorizer.setIsAuthorized(attackerAuth, attackerSig);

    // Attacker's arbitrary entry is written
    assertEq(midnight.isAuthorized(borrower, lender), true);
    // Nonce is now 1
    assertEq(ecrecoverAuthorizer.nonce(borrower), 1);

    // Victim's tx now reverts with InvalidNonce
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(victimAuth, victimSig);

    // Victim's intended authorization was never applied
    assertEq(midnight.isAuthorized(borrower, otherBorrower), false);
}
```
