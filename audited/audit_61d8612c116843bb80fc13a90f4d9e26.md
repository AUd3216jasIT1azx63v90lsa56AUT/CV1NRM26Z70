### Title
Fee-on-transfer collateral token causes unconditional revert in `supplyCollateralAndSellWithUnitsTarget` - (File: src/periphery/MidnightBundles.sol)

### Summary
`supplyCollateralAndSellWithUnitsTarget` pulls `collateralSupplies[i].assets` from the caller into the bundler via `pullToken`, but a fee-on-transfer collateral token causes the bundler to receive fewer tokens than requested. The subsequent call to `Midnight.supplyCollateral` with the original (unreduced) `assets` value then attempts to pull the full amount from the bundler, which reverts due to insufficient balance. The same flaw exists identically in `supplyCollateralAndSellWithAssetsTarget`.

### Finding Description
**Code path:**

```
supplyCollateralAndSellWithUnitsTarget(...)
  → pullToken(token, msg.sender, collateralSupplies[i].assets, ...)
      → safeTransferFrom(token, user, bundler, assets)
      // bundler receives: assets - fee  (fee-on-transfer deduction)
  → forceApproveMax(token, MIDNIGHT)   // allowance = type(uint256).max, not the issue
  → IMidnight(MIDNIGHT).supplyCollateral(market, idx, assets, taker)
      // Midnight.sol line 545:
      → safeTransferFrom(token, bundler, midnight, assets)
      // bundler balance = assets - fee < assets  → REVERT
```

**Root cause:** The bundler passes the caller-supplied `collateralSupplies[i].assets` directly to `Midnight.supplyCollateral` without accounting for the actual amount received after `pullToken`. There is no balance snapshot (`balanceBefore`/`balanceAfter`) to compute the true received amount.

**Attacker inputs:** Any `collateralSupplies` entry whose `token` is a fee-on-transfer ERC20 (e.g., USDT with fee enabled, STA, PAXG, or any custom token with a transfer tax). The caller is the taker themselves — no third-party involvement is needed.

**Why checks fail:** `forceApproveMax` sets allowance to `type(uint256).max`, so allowance is not the bottleneck. The failure is purely a balance shortfall in the bundler. No guard in the bundler or in `Midnight.supplyCollateral` detects or tolerates this discrepancy.

### Impact Explanation
Any taker attempting to use `supplyCollateralAndSellWithUnitsTarget` (or `supplyCollateralAndSellWithAssetsTarget`) with a fee-on-transfer collateral token will have their transaction unconditionally revert at the `Midnight.supplyCollateral` call. The bundle is entirely unusable for such tokens. The taker's gas is wasted and the intended collateral supply + sell operation cannot be completed through the bundler.

### Likelihood Explanation
Requires a market whose `collateralParams` includes a fee-on-transfer token. Such tokens exist on mainnet (USDT with fee switch, PAXG, STA, etc.) and are realistic collateral candidates. The condition is deterministic — every call with such a token reverts, with no randomness or race condition involved. It is fully repeatable.

### Recommendation
Replace the hardcoded `collateralSupplies[i].assets` passed to `supplyCollateral` with the actual amount received by the bundler, measured via a balance snapshot:

```solidity
uint256 balanceBefore = IERC20(token).balanceOf(address(this));
pullToken(token, msg.sender, collateralSupplies[i].assets, collateralSupplies[i].permit);
uint256 received = IERC20(token).balanceOf(address(this)) - balanceBefore;
forceApproveMax(token, MIDNIGHT);
IMidnight(MIDNIGHT).supplyCollateral(
    market, collateralSupplies[i].collateralIndex, received, taker
);
```

Apply the same fix to `supplyCollateralAndSellWithAssetsTarget`. Alternatively, explicitly document that fee-on-transfer collateral tokens are unsupported and add a NatSpec `@dev` warning to both functions.
