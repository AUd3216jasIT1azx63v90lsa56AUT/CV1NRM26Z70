### Title
FOT loan token causes `supplyCollateralAndSellWithAssetsTarget` to always revert when `referralFeeAssets > 0` - (File: src/periphery/MidnightBundles.sol)

### Summary
`supplyCollateralAndSellWithAssetsTarget` tracks `filledSellerAssets` from the return value of `MIDNIGHT.take()`, which reflects the *intended* transfer amount. When the loan token is fee-on-transfer (FOT), `MIDNIGHT.take()` calls `safeTransferFrom(loanToken, payer, address(this), sellerAssets)` and the bundler receives `sellerAssets - F`. The function then attempts to transfer `referralFeeAssets + targetSellerAssets = targetFilledSellerAssets` out, but only holds `targetFilledSellerAssets - F`, causing the second `safeTransfer` to always revert. Since Solidity reverts are atomic, no partial state persists, but the function is completely unusable for FOT loan tokens whenever `referralFeePct > 0`.

**Attacker-controlled inputs:** `referralFeePct` (any value > 0, < WAD), `takes[]` (valid buy offers), `targetSellerAssets` (any positive value). The FOT token is the loan token of the market — a market creator can deploy a market with a FOT loan token, which is not gated.

**Existing protections reviewed:** `require(referralFeePct < WAD)` only prevents 100% fee. `require(filledSellerAssets == targetFilledSellerAssets)` checks the return-value accumulator, not the actual balance. No balance-before/after check exists. No FOT guard exists anywhere in the function.

### Impact Explanation

The function always reverts when the loan token is FOT and `referralFeePct > 0`. Because the revert is atomic, no partial state persists: the taker's debt is not created, the receiver receives nothing, and the referralFeeRecipient receives nothing. The concrete impact is complete denial of service for `supplyCollateralAndSellWithAssetsTarget` on any market whose loan token charges a transfer fee, whenever a referral fee is requested. The documented invariant "Total loan assets received by the receiver is `targetSellerAssets`" is violated — the receiver receives zero.

### Likelihood Explanation

**Preconditions:**
1. The market's `loanToken` is a FOT token (e.g., tokens with a built-in transfer tax). Market creation is permissionless, so any user can create such a market.
2. `referralFeePct > 0` — any non-zero value supplied by the caller.

Both conditions are reachable by an unprivileged user. The bug is deterministic and repeatable: every call to this function under these preconditions reverts. No oracle manipulation, governance action, or admin key is required.

### Recommendation

Replace the implicit balance assumption with an explicit balance-before/after measurement to determine the actual amount received from `MIDNIGHT.take()`:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
// ... take loop ...
uint256 actualReceived = IERC20(loanToken).balanceOf(address(this)) - balanceBefore;
require(actualReceived >= referralFeeAssets + targetSellerAssets, InsufficientReceived());
```

Alternatively, explicitly document and enforce that FOT tokens are not supported as loan tokens (e.g., add a check or NatSpec warning), consistent with how other Morpho products handle this class of tokens.
