Audit Report

## Title
Double `mulDivDown` Rounding Understates `maxDebt` by 1, Enabling Liquidation of Economically Healthy Positions - (File: src/Midnight.sol)

## Summary
`isHealthy` and `liquidate` both compute `maxDebt` via two sequential `mulDivDown` (floor division) calls. Each call independently discards a remainder, and the combined rounding error can understate `maxDebt` by exactly 1 relative to the true single-step floor. When a position sits at the exact health boundary, the protocol computes `maxDebt = debt − 1`, causing `isHealthy` to return `false` and the `liquidate` guard to pass, allowing seizure of collateral from an economically healthy borrower.

## Finding Description

**Confirmed code path:**

`mulDivDown` is plain integer floor division with no rounding protection (`src/libraries/UtilsLib.sol`, lines 29–31):
```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
```

`isHealthy` applies it twice in sequence (`src/Midnight.sol`, lines 954–955):
```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

`liquidate` uses the identical double-floor pattern (`src/Midnight.sol`, line 613):
```solidity
maxDebt += _collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(_collateralParam.lltv, WAD);
```

The liquidatability guard compares the understated `maxDebt` against `originalDebt` (`src/Midnight.sol`, line 622):
```solidity
(postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt)
```

**Root cause — double floor rounding:**

Let `V1 = floor(C × P / ORACLE_PRICE_SCALE)` and `V2 = floor(V1 × L / WAD)`. The exact single-step bound is `M = floor(C × P × L / (ORACLE_PRICE_SCALE × WAD))`. Because step 1 discards remainder `r = (C × P) mod ORACLE_PRICE_SCALE` before step 2 multiplies by `L`, the combined error satisfies `M − 1 ≤ V2 ≤ M`. The off-by-one (`V2 = M − 1`) is reachable whenever the two discarded remainders together exceed one full unit.

**Concrete counterexample** using confirmed protocol constants (`ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`, valid LLTV tier `LLTV_2 = 0.77e18` from `src/libraries/ConstantsLib.sol` line 31):

| Parameter | Value |
|---|---|
| `collateral` (C) | 10 |
| `price` (P) | `1.3e35` |
| `lltv` (L) | `0.77e18` |
| `debt` (D) | 1 |

- Step 1: `floor(10 × 1.3e35 / 1e36) = floor(1.3) = 1`
- Step 2: `floor(1 × 0.77e18 / 1e18) = floor(0.77) = 0`
- Exact: `10 × 1.3e35 × 0.77e18 / 1e54 = 1.001` → `floor = 1 = debt` → **healthy**
- Protocol sees: `maxDebt = 0 < 1 = debt` → **unhealthy** → liquidation guard passes

**Exploit flow:**

1. Borrower opens a position with C = 10 collateral units and borrows D = 1 debt unit at price P = 2e35. At that price: `V1 = floor(10 × 2e35 / 1e36) = 2`, `V2 = floor(2 × 0.77e18 / 1e18) = 1 ≥ 1 = debt` → `isHealthy` returns `true` at the `take` call site (line 476), so the borrow succeeds.
2. Oracle price drifts to exactly `1.3e35` through normal market movement. The position remains economically healthy: exact `maxDebt = floor(1.001) = 1 = debt`.
3. Liquidator calls `liquidate(market, collateralIndex, seizedAssets, 0, borrower, false, ...)`.
4. Protocol computes `maxDebt = 0`; guard `originalDebt (1) > maxDebt (0)` passes; collateral is seized from a healthy borrower.

**Why existing checks fail:**

There are no minimum debt or collateral dust guards anywhere in the codebase that would prevent positions with small integer values from being created. The `supplyCollateral` and `take` functions accept any nonzero amount.

The health check at the `take` path uses the same `isHealthy` function with the same double-floor computation, so it does not prevent the boundary condition from being reached after price movement.

The contract's own natspec (`src/Midnight.sol`, line 116) acknowledges `maxDebt` is rounded down but does not account for the compounding of two independent floor operations:
```
/// @dev maxDebt is rounded down in isHealthy and liquidate.
```

## Impact Explanation
A liquidator can seize collateral from a borrower whose position is economically healthy (exact `maxDebt ≥ debt`) purely due to the double floor-rounding understating `maxDebt` by 1. The borrower suffers an unwarranted collateral loss. The core protocol invariant — healthy positions cannot be liquidated — is violated. This is a direct, concrete loss of user funds triggered by an unprivileged external caller.

## Likelihood Explanation
No oracle manipulation is required. For any fixed collateral amount `C` and LLTV `L`, there exists a positive-measure interval of oracle prices `P` that triggers the off-by-one. A liquidator needs only to monitor positions and call `liquidate` when the price enters such an interval. The condition is passively reachable through normal price movement. The affected positions are dust-level (small integer collateral/debt amounts), but the protocol imposes no minimum size guards, so such positions are creatable by any user.

## Recommendation
Replace the two-step `mulDivDown` chain with a single fused computation that applies only one floor operation. Since `collateral` is `uint128` (max ~1.15e38) and `price` can be up to `ORACLE_PRICE_SCALE = 1e36`, the product `collateral * price` can reach ~1e74, which fits in `uint256` (max ~1.15e77). Multiplying further by `lltv` (max `1e18`) would overflow, so a 512-bit intermediate or a library such as Solady's `FullMath.mulDiv` should be used:

```solidity
// Replace:
maxDebt += collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);

// With (using a 512-bit fused muldiv):
maxDebt += FullMath.mulDiv(collateral * price, lltv, ORACLE_PRICE_SCALE * WAD);
// or equivalently, using two steps but compensating:
maxDebt += collateral.mulDivDown(price * lltv, ORACLE_PRICE_SCALE * WAD);
// (safe only if price * lltv doesn't overflow uint256)
```

Alternatively, add 1 to `maxDebt` after the double-floor computation as a conservative rounding correction, accepting that positions can be at most 1 unit above the true threshold.

## Proof of Concept
Minimal Foundry unit test:

```solidity
function test_doubleFloorLiquidatesHealthyPosition() public {
    // Setup: collateral=10, price=1.3e35, lltv=0.77e18, debt=1
    uint256 collateral = 10;
    uint256 price = 1.3e35;
    uint256 lltv = 0.77e18;
    uint256 debt = 1;

    // Protocol's double-floor computation
    uint256 step1 = (collateral * price) / 1e36;       // = 1
    uint256 maxDebt = (step1 * lltv) / 1e18;           // = 0

    // Exact single-step computation
    // 10 * 1.3e35 * 0.77e18 / 1e54 = 1.001 → floor = 1
    // maxDebt_exact = 1 >= debt = 1 → healthy

    assert(maxDebt < debt);          // Protocol says unhealthy
    // assert(maxDebt_exact >= debt) // Economically healthy
    // → liquidation guard passes on a healthy position
}
```
