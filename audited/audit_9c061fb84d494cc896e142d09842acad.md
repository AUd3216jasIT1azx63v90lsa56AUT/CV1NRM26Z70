### Title
Maker callback fires on `units=0` take of fully-consumed offer, allowing unprivileged taker to trigger callback without any fill - (File: src/Midnight.sol)

### Summary

`take()` in `src/Midnight.sol` dispatches `offer.callback` unconditionally whenever `offer.callback != address(0)`, with no guard on `units == 0`. Because the consumed-accounting check passes when `units=0` on an exactly-exhausted offer (`consumed == maxUnits`, so `consumed + 0 <= maxUnits` holds), an unprivileged taker can replay a fully-consumed offer's valid ratifier signature with `units=0` and cause the maker's callback to fire with `sellerAssets=0` / `buyerAssets=0` and `units=0`, without any actual position change or token transfer.

### Finding Description

**Consumed check — why it passes at `units=0`:**

In `src/Midnight.sol` lines 370–372:

```solidity
newConsumed = consumed[offer.maker][offer.group] += units;   // += 0
require(newConsumed <= offer.maxUnits, ConsumedUnits());     // maxUnits <= maxUnits → passes
``` [1](#0-0) 

When `units=0`, `newConsumed` equals the current `consumed` value. If the offer was filled exactly to `maxUnits`, the require is `maxUnits <= maxUnits`, which passes. The Certora spec at `certora/specs/Consume.spec` lines 99–111 explicitly acknowledges this: a fully-consumed offer allows `units=0` takes without reverting. [2](#0-1) 

**Asset computation — all zero:**

With `units=0`, both `buyerAssets` and `sellerAssets` compute to zero:

```solidity
uint256 buyerAssets  = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);  // 0
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD); // 0
``` [3](#0-2) 

**Callback dispatch — no `units > 0` guard:**

Lines 445–453 and 458–474 fire the buyer/seller callback solely on `!= address(0)`:

```solidity
if (buyerCallback != address(0)) {
    IBuyCallback(buyerCallback).onBuy(..., buyerAssets, units, ...);  // buyerAssets=0, units=0
}
// ...
if (sellerCallback != address(0)) {
    ISellCallback(sellerCallback).onSell(..., sellerAssets, units, ...); // sellerAssets=0, units=0
}
``` [4](#0-3) [5](#0-4) 

There is no `if (units == 0) return (0, 0)` or equivalent guard before callback dispatch.

**Exploit flow:**

1. Maker creates a sell offer with `offer.callback = address(BorrowCallback)`, `offer.callbackData = abi.encode(collateralIndex, collateralAmount, ...)`, `offer.maxUnits = N`.
2. Legitimate taker fills the offer to `N` units. `BorrowCallback.onSell` fires, decodes `collateralAmount` from `callbackData`, and calls `supplyCollateral(market, idx, collateralAmount, seller)`.
3. Attacker (any address ≠ maker) calls `take(offer, ratifierData, 0, attacker, ...)` with the identical offer struct and valid ratifier data.
4. Consumed check: `N + 0 = N <= N` → passes.
5. `BorrowCallback.onSell` fires again with `sellerAssets=0, units=0` but with the original `callbackData` still encoding `collateralAmount`. If the callback contract holds tokens (pre-funded for multiple fills, or re-funded externally), it calls `supplyCollateral` again — draining the callback contract's tokens without any corresponding fill.

The `BorrowCallback` pattern in the test suite decodes `amount` from `callbackData` (not from `sellerAssets`), making it exactly this class of callback: [6](#0-5) 

The `ISellCallback` interface passes `sellerAssets` and `units` as parameters, but callback implementations are not required to use them — any callback that acts on `callbackData`-encoded amounts is vulnerable to this trigger. [7](#0-6) 

**Why existing protections are insufficient:**

- The ratifier check (line 356) validates the offer struct but does not prevent replaying a valid signature with `units=0`.
- The `offer.maker != taker` check (line 354) only prevents self-takes; any third party can call this.
- The `isAuthorized` check (line 346) only gates the taker's authorization, not the callback trigger.
- The `liquidationLocked` / `isHealthy` check (line 476) fires after callbacks and does not prevent the callback from executing. [8](#0-7) 

### Impact Explanation

An unprivileged taker can trigger the maker's `offer.callback` with `units=0` and `sellerAssets=0`/`buyerAssets=0` on a fully-consumed offer. For callbacks that act on `callbackData`-encoded amounts rather than the passed `sellerAssets`/`units` (the canonical `BorrowCallback` pattern), this causes the callback to execute its side effects — including token transfers, collateral supply, or external protocol interactions — without any corresponding fill. This drains the callback contract's pre-funded tokens and corrupts the accounting invariant that callbacks only fire during real fills.

### Likelihood Explanation

**Preconditions:**
1. The offer must have `offer.callback != address(0)` — common for borrow/lend automation callbacks.
2. The offer must be fully consumed (`consumed == maxUnits`).
3. The attacker must possess the ratifier data (signature) for the offer — obtainable by observing the original fill transaction on-chain.
4. The callback contract must hold tokens or have side effects independent of `units`/`sellerAssets`.

Conditions 1–3 are routinely satisfied in normal protocol operation. Condition 4 applies to any callback pre-funded for multiple fills or holding residual balances. The attack is repeatable as long as the callback contract holds tokens, and requires no special privilege.

### Recommendation

Add an early return when `units == 0` before any callback dispatch and token transfers:

```solidity
if (units == 0) return (0, 0);
```

Insert this immediately after the consumed accounting block (after line 373) and before position updates. This preserves the existing behavior that `units=0` takes on fully-consumed offers do not revert (as modeled in the Certora spec), while ensuring callbacks and token transfers are skipped for true no-ops. [9](#0-8) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {TakeTest, BorrowCallback} from "./TakeTest.sol";
import {TickLib, MAX_TICK} from "../src/libraries/TickLib.sol";
import {UtilsLib} from "../src/libraries/UtilsLib.sol";

contract ZeroUnitExhaustedCallbackTest is TakeTest {
    using UtilsLib for uint256;

    function testZeroUnitTakeOnExhaustedOfferFiresCallback() public {
        uint256 units = 100e18;
        uint256 price = TickLib.tickToPrice(MAX_TICK);
        uint256 collateralAmount = units.mulDivUp(1e18, market.collateralParams[0].lltv);

        // Deploy a BorrowCallback pre-funded with collateral for TWO fills
        BorrowCallback cb = new BorrowCallback();
        deal(market.collateralParams[0].token, address(cb), collateralAmount * 2);

        borrowerOffer.callback     = address(cb);
        borrowerOffer.callbackData = abi.encode(0, collateralAmount, bytes(""));
        borrowerOffer.maxUnits     = units;
        borrowerOffer.tick         = MAX_TICK;

        vm.prank(borrower);
        midnight.setIsAuthorized(address(cb), true, borrower);

        // Step 1: fill offer to maxUnits (fully consumed)
        deal(address(loanToken), lender, units.mulDivUp(price, WAD));
        take(units, lender, borrowerOffer);

        // Assert: offer is fully consumed
        assertEq(midnight.consumed(borrower, borrowerOffer.group), units);
        // Callback supplied collateralAmount once
        assertEq(midnight.collateral(id, borrower, 0), collateralAmount);

        // Step 2: attacker calls take(units=0) on the exhausted offer
        address attacker = makeAddr("attacker");
        vm.prank(attacker);
        midnight.take(borrowerOffer, hex"", 0, attacker, attacker, address(0), hex"");

        // ASSERTION: callback fired again — collateral doubled without any fill
        // If bug exists: midnight.collateral(id, borrower, 0) == collateralAmount * 2
        // If fixed:      the call above should revert or collateral remains collateralAmount
        assertEq(
            midnight.collateral(id, borrower, 0),
            collateralAmount,          // expected: unchanged (no second fill)
            "callback must not fire on zero-unit take of exhausted offer"
        );
    }
}
```

**Expected assertions if bug is present:** `midnight.collateral(id, borrower, 0) == collateralAmount * 2` — the callback drained the pre-funded collateral and supplied it to the maker's position without any fill.

**Expected assertions after fix:** The `take(units=0)` returns `(0, 0)` immediately without invoking the callback, and collateral remains `collateralAmount`.

/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
