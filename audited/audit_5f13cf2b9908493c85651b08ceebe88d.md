Audit Report

## Title
Maker-controlled sell callback can transiently manipulate oracle price, causing `isHealthy` to return false and reverting valid taker transactions - (File: src/Midnight.sol)

## Summary
When a taker fills a sell offer (`offer.buy == false`), the maker's `offer.callback` is assigned as `sellerCallback` and invoked before the post-callback health check. A malicious maker callback that transiently manipulates a spot oracle during `onSell` causes `isHealthy(offer.market, id, seller)` to return `false` for a genuinely healthy seller, reverting the taker's transaction with `SellerIsLiquidatable`. Because the entire transaction reverts, the oracle manipulation also reverts, making the attack repeatable at gas cost only and permanently blocking takers from filling the poisoned offer.

## Finding Description

**Callback assignment** (`src/Midnight.sol`, line 421):
```solidity
address sellerCallback = offer.buy ? takerCallback : offer.callback;
```
When `offer.buy == false`, `sellerCallback = offer.callback` — fully controlled by the maker at offer-creation time with no protocol restriction. [1](#0-0) 

**Seller identity** (`src/Midnight.sol`, line 375):
```solidity
(address buyer, address seller) = offer.buy ? (offer.maker, taker) : (taker, offer.maker);
```
When `offer.buy == false`, `seller = offer.maker`. The post-callback health check at line 476 is therefore on the maker's own position. [2](#0-1) 

**Callback invocation** (`src/Midnight.sol`, lines 458–474):
`ISellCallback(sellerCallback).onSell(...)` is called with no restrictions on external state mutations. The callback can interact with any external contract, including a DEX pool whose reserves an oracle reads. [3](#0-2) 

**Lock reset and health check** (`src/Midnight.sol`, lines 475–476):
```solidity
if (!wasLocked) UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, false);
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```
In the normal case (`wasLocked == false`), the lock is reset to `false` before the `require`. Therefore `liquidationLocked(id, seller)` evaluates to `false`, and `isHealthy` must return `true` for the transaction to succeed. If the callback has shifted the oracle price, `isHealthy` reads the manipulated value and returns `false`. [4](#0-3) 

**`isHealthy` oracle read** (`src/Midnight.sol`, lines 953–955):
```solidity
uint256 price = IOracle(collateralParam.oracle).price();
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```
The oracle is called live with no snapshot or TWAP enforcement at the protocol level. [5](#0-4) 

The Certora formal verification spec (`certora/specs/Liquidate.spec`, line 13) explicitly assumes `PER_CALLEE_CONSTANT` for oracle prices — i.e., the price does not change during execution — which is exactly the invariant this attack violates. [6](#0-5) 

The comment at line 335 confirms the lock's intended scope: *"The seller cannot be liquidated during the callbacks of a take."* It does not protect the post-callback `isHealthy` evaluation from reading oracle state mutated during the callback. [7](#0-6) 

**Exploit flow:**
1. Maker deploys `MaliciousCallback` implementing `ISellCallback.onSell` that flash-loans tokens and swaps in the oracle's underlying pool, dropping the reported collateral price below the health threshold.
2. Maker creates a sell offer with `offer.callback = address(MaliciousCallback)`.
3. Taker calls `take()`. State is updated (seller debt increases), lock is set to `true` (line 444).
4. `MaliciousCallback.onSell` executes the pool swap; oracle price drops below the health threshold.
5. Lock is reset to `false` (line 475).
6. `isHealthy(offer.market, id, seller)` reads the manipulated price → returns `false`.
7. `require(false || false, SellerIsLiquidatable())` → transaction reverts.
8. The revert unwinds the swap and flash loan repayment — the attack costs only gas.

## Impact Explanation
Any taker attempting to fill a malicious maker's sell offer will have their transaction revert with `SellerIsLiquidatable` even though the seller's position was healthy at the pre-callback oracle price. The maker's offer is never consumed (all state changes revert), so the attack is repeatable at zero net cost to the maker (gas only, since the oracle manipulation also reverts). This permanently blocks market activity on any offer the maker chooses to poison and can be used to grief specific takers or deny liquidity to the entire market. The impact is a permanent, targeted denial-of-service on taker transactions. `SECURITY.md` explicitly states oracle manipulation/flash-loan attacks are not excluded. [8](#0-7) 

## Likelihood Explanation
**Preconditions:**
- Market uses an oracle whose `price()` reads mutable on-chain state (e.g., Uniswap v2/v3 spot price, any reserve-based oracle). TWAP oracles with short windows are also susceptible.
- Maker's position must be healthy at the normal oracle price but close enough to the liquidation threshold that a feasible pool swap can push it below.
- Maker must be able to execute the swap within the callback — trivially achievable via flash loan, which also reverts with the transaction.

**Feasibility:** High. The maker controls `offer.callback` unconditionally; no privileged access is required. The attack is atomic (single transaction), repeatable, and costs only gas. Any market with a spot oracle is vulnerable.

## Recommendation
Snapshot the oracle price(s) for the seller's collateral **before** invoking any callbacks, and use those cached values in the post-callback `isHealthy` check rather than re-querying the oracle live. Alternatively, pass the pre-callback price snapshot directly into a dedicated `isHealthyWithCachedPrices` variant. This eliminates the window in which callback-induced oracle state changes can affect the health evaluation.

## Proof of Concept
**Minimal Forge fork test plan:**
1. Deploy a mock spot oracle backed by a Uniswap v2 pool on a mainnet fork.
2. Deploy `MaliciousCallback` that, in `onSell`, flash-borrows the pool token and swaps to drop the oracle price below the maker's liquidation threshold, then repays.
3. Create a market using the mock oracle; fund the maker with collateral such that their position is healthy at the initial price but within reach of the manipulated price.
4. Maker creates a sell offer with `offer.callback = address(MaliciousCallback)`.
5. Taker calls `take()`; assert the call reverts with `SellerIsLiquidatable`.
6. Confirm the maker's position is still healthy by calling `isHealthy` in a separate transaction (oracle price has reverted), demonstrating the DoS is transient but repeatable.
