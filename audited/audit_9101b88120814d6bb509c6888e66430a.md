### Title
No User-Controlled Slippage Limits on Oracle-Dependent Transfer Amounts in `liquidate` — (File: src/Midnight.sol)

---

### Summary

The `liquidate` function in `Midnight.sol` computes either `repaidUnits` (loan tokens the liquidator pays) or `seizedAssets` (collateral the liquidator receives) entirely from a live oracle price fetched at execution time. No caller-supplied upper bound on `repaidUnits` or lower bound on `seizedAssets` exists. A pending oracle price update mined before the liquidation transaction causes the liquidator to pay more or receive less than they anticipated, with no on-chain protection.

---

### Finding Description

`liquidate` accepts at most one non-zero input: `seizedAssets` or `repaidUnits`.

**Path A — caller supplies `seizedAssets > 0`:** [1](#0-0) 

```solidity
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```

`repaidUnits` — the loan tokens pulled from the liquidator — is derived entirely from `liquidatedCollatPrice`, which is the oracle's live return value at execution time. There is no `maxRepaidUnits` parameter the caller can supply to cap this amount.

**Path B — caller supplies `repaidUnits > 0`:** [2](#0-1) 

```solidity
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
```

`seizedAssets` — the collateral transferred to the liquidator — is derived from the same oracle price. There is no `minSeizedAssets` parameter to guarantee a floor on what the liquidator receives.

The oracle price is fetched unconditionally from the external `IOracle` contract: [3](#0-2) 

In post-maturity mode, `lif` also grows with `block.timestamp`: [4](#0-3) 

making the computed amounts doubly unpredictable — both the oracle price and the incentive factor can shift between transaction submission and execution.

The final token pull uses the computed `repaidUnits` with no cap: [5](#0-4) 

---

### Impact Explanation

**Path A (seizedAssets input):** If the oracle price rises between submission and execution, `repaidUnits` increases proportionally. The liquidator is forced to pay more loan tokens than they budgeted for the same amount of collateral. In extreme oracle moves this can turn a profitable liquidation into a net loss.

**Path B (repaidUnits input):** If the oracle price falls between submission and execution, `seizedAssets` decreases. The liquidator pays the full `repaidUnits` but receives less collateral than expected, degrading or eliminating the liquidation incentive.

Both paths result in direct, uncontrolled financial loss to the liquidator with no on-chain recourse.

---

### Likelihood Explanation

Oracle price feeds update frequently. MEV bots and block builders actively monitor public mempools and can trivially reorder an oracle update transaction ahead of a pending liquidation. No privileged access is required — any actor with standard mempool visibility can trigger this. Liquidators using the `seizedAssets` path (the natural way to express "I want exactly X collateral") are the primary victims. The scenario is realistic and repeatable on any chain with a competitive block-building environment.

---

### Recommendation

**Short term:** Add a `maxRepaidUnits` parameter to `liquidate` and enforce it when `seizedAssets > 0`:

```solidity
require(repaidUnits <= maxRepaidUnits, RepaidUnitsTooHigh());
```

Symmetrically, add a `minSeizedAssets` parameter and enforce it when `repaidUnits > 0`:

```solidity
require(seizedAssets >= minSeizedAssets, SeizedAssetsTooLow());
```

**Long term:** As a general principle, any function that computes a transfer amount from an external oracle at execution time should expose caller-controlled bounds on that amount. This is the standard pattern used by Uniswap (`amountOutMin`, `amountInMax`) and is essential for any caller — especially smart-contract liquidators — that cannot predict the oracle's return value at the time of transaction construction.

---

### Proof of Concept

1. Collateral token price is `P` at the time the liquidator constructs the transaction. The liquidator calls `liquidate(..., seizedAssets = S, repaidUnits = 0, ...)` expecting to pay approximately `S * P / ORACLE_PRICE_SCALE / lif` loan tokens.

2. Before the liquidation is mined, an oracle update transaction sets the price to `P' > P` (e.g., a 10% price increase).

3. The liquidation executes. At line 650, `repaidUnits` is computed using `P'`:
   ```
   repaidUnits = S * P' / ORACLE_PRICE_SCALE * WAD / lif
   ```
   This is 10% higher than the liquidator anticipated.

4. At line 717, `repaidUnits` loan tokens are pulled from the liquidator with no cap. The liquidator has no on-chain mechanism to reject this outcome.

5. The liquidator receives exactly `S` collateral (worth `S * P'` at the new price) but paid `S * P' / lif` loan tokens — the economics may still be marginally profitable, but the liquidator had no ability to set a maximum acceptable cost, and a larger price move could make the liquidation unprofitable entirely.
