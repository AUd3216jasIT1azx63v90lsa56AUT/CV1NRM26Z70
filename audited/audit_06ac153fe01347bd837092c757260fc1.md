### Title
Borrower-Deployed Reverting Oracle Permanently Freezes Liquidation via Bitmap Loop - (File: src/Midnight.sol)

### Summary
The `liquidate` function iterates over every activated collateral in the borrower's bitmap and calls `IOracle.price()` on each one before the health check. Because market creation is permissionless and `supplyCollateral` never calls the oracle, an unprivileged borrower can create a market with a reverting oracle at one collateral slot, supply dust to that slot, borrow, and render their position permanently unliquidatable. The protocol's own NatDev comment at lines 34–36 explicitly acknowledges this attack surface.

### Finding Description

**Exact code path:**

`src/Midnight.sol` lines 607–618 — the `liquidate` while loop:

```solidity
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price(); // ← called for every activated collateral
    ...
    _collateralBitmap = _collateralBitmap.clearBit(i);
}
``` [1](#0-0) 

Every oracle in the bitmap is called unconditionally, including those for collaterals that are not being seized. If any single call reverts, the entire transaction reverts before the `NotLiquidatable` check is even reached.

**Why the attacker can reach this state:**

1. **Permissionless market creation** (`touchMarket`, lines 755–791): any caller can create a market. Validation checks only collateral token sort order, allowed LLTV tiers, and `maxLif` values. The oracle address is accepted without any liveness check. [2](#0-1) 

2. **`supplyCollateral` never calls the oracle** (lines 524–546): supplying collateral to a slot only sets the bitmap bit and transfers the token. No oracle call occurs, so a reverting oracle is silently activated. [3](#0-2) 

3. **Protocol explicitly acknowledges the gap** (lines 34–36):
   > "Liquidation reverts if any of the activated collaterals' oracle reverts (see LIVENESS)."
   > "Note that a borrower can activate a collateral once its oracle is reverting because the oracle is not called in supplyCollateral." [4](#0-3) 

**Exploit flow:**

1. Attacker deploys `RevertingOracle` (a contract whose `price()` always reverts — the test helper at `test/helpers/RevertingOracle.sol` is exactly this). [5](#0-4) 

2. Attacker calls `touchMarket` with 16 `CollateralParams` entries; slot 7 uses the reverting oracle address. All other slots use normal oracles. Market creation succeeds because no oracle liveness check exists.

3. Attacker calls `supplyCollateral` for all 16 slots (including slot 7). The bitmap now has 16 bits set. No oracle is called during supply.

4. Attacker borrows by taking a sell offer, creating debt.

5. Attacker (or market movement) makes the position unhealthy (e.g., other oracle prices drop).

6. Any liquidator calls `liquidate`. The while loop reaches slot 7, calls `IOracle(revertingOracle).price()`, which reverts. The entire `liquidate` call reverts. This is unconditional and permanent as long as the reverting oracle is in the bitmap.

**Why existing checks do not stop it:**

- The `liquidatorGate` check (lines 597–600) only gates *who* can liquidate, not *whether* the oracle loop can revert. [6](#0-5) 
- The `NotBorrower` and `InconsistentInput` guards fire before the loop but cannot prevent oracle reversion inside it.
- The Certora `Reverts.spec` rule `oracleRevertCausesLiquidateRevert` (lines 183–193) formally proves this revert propagation — confirming no mitigation exists. [7](#0-6) 
- `repay` is still callable (it does not call oracles), so the borrower can repay voluntarily, but a liquidator cannot force repayment.

### Impact Explanation

Any borrower who activates a collateral slot backed by a reverting oracle — which they can do unilaterally via permissionless market creation — permanently prevents liquidation of their position. The core invariant "unhealthy positions remain liquidatable" is broken for that borrower. Bad debt accumulates without any mechanism for liquidators to recover it, directly threatening lender solvency.

### Likelihood Explanation

**Preconditions:**
- Attacker must create a market (permissionless, zero privilege required).
- Attacker must supply a non-zero amount to the reverting-oracle slot (no minimum enforced).
- Attacker must have an open debt position in that market.

**Feasibility:** All three preconditions are reachable by any EOA with enough tokens to supply collateral and borrow. The attack is self-contained: the attacker creates the market, supplies, borrows, and the freeze is immediate. It is repeatable across any number of markets.

**Repeatability:** Once the reverting oracle is activated in the bitmap, every future `liquidate` call reverts deterministically. The freeze is permanent unless the borrower voluntarily repays (which they have no incentive to do if the position is underwater).

### Recommendation

The root cause is that `liquidate` calls every activated oracle unconditionally. Two targeted fixes:

1. **Wrap oracle calls in `try/catch`** inside the bitmap loop. If an oracle reverts, treat its price as 0 (worst-case for the borrower) or skip it for `maxDebt` computation while still allowing the seized collateral's oracle to be called normally. This preserves liquidatability even when non-seized oracles are broken.

2. **Alternatively, only call the oracle for the collateral being seized** for the `liquidatedCollatPrice` assignment, and use a separate, fault-tolerant path for the `maxDebt`/`badDebt` accumulation loop (e.g., skip reverting oracles, treating their contribution as 0).

Either approach ensures that a single broken oracle cannot block liquidation of the entire position.

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";

contract LiquidationFreezeTest is Test {
    Midnight midnight;
    // ... standard setup (loanToken, collateralTokens, lender offer)

    function testLiquidationFreezeViaRevertingOracle() public {
        // 1. Create 16 collateral tokens + oracles; slot 7 uses RevertingOracle
        RevertingOracle badOracle = new RevertingOracle();
        Market memory m = _buildMarketWith16Collaterals(badOracle, 7);

        // 2. Supply collateral to all 16 slots (no oracle called)
        for (uint256 i = 0; i < 16; i++) {
            midnight.supplyCollateral(m, i, 1e18, address(this));
        }
        // Assert: bitmap has 16 bits set
        assertEq(countBits(midnight.collateralBitmap(toId(m), address(this))), 16);

        // 3. Borrow (take a sell offer)
        _setupAndBorrow(m, 1e18);

        // 4. Make position unhealthy (drop prices on non-reverting oracles)
        _dropPrices(m, 7); // drop all except slot 7

        // 5. Assert liquidate always reverts
        vm.expectRevert(); // RevertingOracle.price() reverts
        midnight.liquidate(m, 0, 0, 1, address(this), false, address(this), address(0), "");

        // 6. Assert position is still unhealthy (invariant violated)
        assertFalse(midnight.isHealthy(m, toId(m), address(this)));
        // isHealthy also reverts due to same oracle loop — position is frozen
    }
}
```

**Expected assertions:**
- `liquidate` reverts on every call while slot 7 is activated.
- `isHealthy` also reverts, confirming the position is in a permanently unresolvable state.
- `debtOf(borrower) > 0` persists indefinitely, confirming bad debt accumulation with no liquidation path.
