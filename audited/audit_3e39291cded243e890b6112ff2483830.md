Audit Report

## Title
Assets-mode consumed counter not incremented when `buyerAssets` rounds to zero, bypassing `maxAssets` cap - (File: src/Midnight.sol)

## Summary
When a buy offer has `maxAssets > 0` and `offerPrice < WAD`, calling `take()` with `units=1` computes `buyerAssets = floor(1 * buyerPrice / WAD) = 0`. The consumed counter is incremented by zero, so the `ConsumedAssets` guard is trivially satisfied on every call. Position credit and debt are still updated unconditionally, allowing unbounded credit accumulation for the maker far beyond the `maxAssets` cap they intended.

## Finding Description
At [1](#0-0)  `buyerAssets` is computed with `mulDivDown`, which floors to zero when `units=1` and `buyerPrice < WAD`. The consumed counter is then incremented by this zero value, and the guard `require(newConsumed <= offer.maxAssets)` passes trivially since `newConsumed` equals the pre-call value of `consumed`. Position accounting at [2](#0-1)  runs unconditionally on `units`, not on `buyerAssets`, so `buyerPos.credit` and `sellerPos.debt` each increase by 1 on every such call. There is no check that `units > 0` implies `buyerAssets > 0`, and no floor enforced on `buyerAssets` before the consumed update. The protocol itself acknowledges this gap at [3](#0-2) .

The existing test `testBugBuyMaxAssetsBypass` at [4](#0-3)  directly confirms the exact state change: consumed stays frozen at `maxAssets`, token balances are unchanged, yet `creditOf(lender)` and `debtOf(borrower)` both strictly increase after a `take(units=1)` on a fully-consumed offer.

## Impact Explanation
The maker's credit grows without bound: each `take(units=1)` call adds 1 unit of credit to the maker while `consumed` stays frozen. The maker's intended cap of `maxAssets / offerPrice` units of credit exposure is never enforced. Because bad debt is socialized proportionally to credit held at realization, the maker suffers losses proportional to their inflated credit balance rather than their intended exposure. This constitutes a concrete accounting insolvency risk for the maker.

## Likelihood Explanation
The precondition `offerPrice < WAD` is satisfied for any tick below 2910 (the lower half of the full tick range `[0, MAX_TICK=5820]`), which is the natural range for buy offers. Once an offer is first taken, the ratifier signature is public, so any unprivileged address can repeat the attack with no per-iteration asset cost beyond gas. The attack is fully permissionless and repeatable.

## Recommendation
Before the consumed update, require that `buyerAssets > 0` when `units > 0` in assets-mode, or alternatively increment `consumed` by `units` (not `buyerAssets`) when `maxAssets > 0` and `buyerAssets` rounds to zero. A minimal fix is to add `require(buyerAssets > 0 || units == 0, ZeroBuyerAssets())` immediately after line 363, or to enforce a minimum of 1 on `buyerAssets` before the consumed increment when `units > 0`.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) is a complete, self-contained proof of concept. It pre-sets `consumed` to `maxAssets` (fully consumed state), calls `take(1, borrower, lenderOffer)` with a tick where `offerPrice < WAD`, and asserts that `consumed` is unchanged while `creditOf(lender)` and `debtOf(borrower)` both increase. Running `forge test --match-test testBugBuyMaxAssetsBypass` reproduces the issue.

    // Show that a buy offer with offerPrice < WAD can be taken with units > 0
    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);
        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
