### Title
Referral Fee Multiplication Overflow in `buyWithUnitsTargetAndWithdrawCollateral` Due to Unchecked Plain Arithmetic in `mulDivDown` - (File: src/periphery/MidnightBundles.sol)

### Summary
`UtilsLib.mulDivDown` is implemented as plain `(x * y) / d` with no 512-bit intermediate arithmetic. Under Solidity 0.8.34's checked arithmetic, the expression `filledBuyerAssets * referralFeePct` at line 102 reverts on overflow when both operands are large. The guard `require(referralFeePct < WAD)` does not bound the product, so any caller supplying a `referralFeePct` near `WAD - 1` and a `filledBuyerAssets` above `type(uint256).max / referralFeePct` will cause an unconditional revert for that configuration.

### Finding Description
**Root cause â€” `mulDivDown` uses plain multiplication:** [1](#0-0) 

`(x * y) / d` is a standard Solidity expression; in 0.8.x it reverts on overflow with no assembly escape hatch.

**Vulnerable line:** [2](#0-1) 

The only guard present is: [3](#0-2) 

`referralFeePct < WAD` keeps the denominator `WAD - referralFeePct` positive but says nothing about the numerator product `filledBuyerAssets * referralFeePct`.

**Overflow threshold:** With `referralFeePct = WAD - 1 = 999_999_999_999_999_999`, overflow occurs when:

```
filledBuyerAssets > type(uint256).max / (WAD - 1)
                  â‰ˆ 1.158 Ã— 10^59
```

**Exploit path:**
1. Attacker (market creator / taker) deploys or uses a loan token with a supply large enough to fund fills at this scale (a custom ERC20 with `type(uint256).max` supply suffices; market creation is permissionless).
2. Attacker sets up sell offers for a very large `units` amount and collateralises them.
3. Attacker calls `buyWithUnitsTargetAndWithdrawCollateral` with `referralFeePct = WAD - 1` and `maxBuyerAssets = type(uint256).max`.
4. After the fill loop, `filledBuyerAssets` is large; line 102 executes `filledBuyerAssets * (WAD - 1)` which overflows â†’ revert.

**Why existing checks fail:** `require(referralFeePct < WAD)` only prevents division by zero; it does not cap the product. There is no cap on `filledBuyerAssets` inside the function.

**Developer acknowledgement:** The fuzz test for this exact function artificially caps `units` at `type(uint128).max / 2` to stay below the overflow boundary, confirming the developers are aware the plain arithmetic breaks for larger values: [4](#0-3) 

### Impact Explanation
Any call to `buyWithUnitsTargetAndWithdrawCollateral` where `filledBuyerAssets * referralFeePct` exceeds `type(uint256).max` reverts unconditionally. For `referralFeePct` near `WAD - 1`, the threshold is ~`1.158 Ã— 10^59` raw token units â€” reachable with tokens that have large supplies (e.g., a custom ERC20 with `type(uint256).max` total supply). The bundler function becomes permanently unusable for that market/fee combination; no state is corrupted but the call always reverts, denying service to any taker relying on the bundler for that configuration.

### Likelihood Explanation
Requires a loan token with a very large raw supply and a high referral fee percentage simultaneously. Standard 18-decimal tokens with realistic supplies (even `type(uint128).max` units) stay below the threshold. However, market creation is permissionless, custom ERC20 tokens are trivially deployable, and the attacker controls both `referralFeePct` and the token supply, making the preconditions fully attacker-controlled. The condition is repeatable for any call matching those parameters.

### Recommendation
Replace the plain multiplication in `mulDivDown` (and `mulDivUp`) with a 512-bit intermediate multiplication that does not overflow for any `uint256` inputs â€” for example, Solady's `FullMath.mulDiv` or an inline assembly implementation using `mulmod`/`mulDiv512`. Alternatively, add an explicit pre-check before line 102:

```solidity
require(
    referralFeePct == 0 || filledBuyerAssets <= type(uint256).max / referralFeePct,
    ReferralFeeOverflow()
);
```

The cleaner fix is upgrading `mulDivDown` to use 512-bit arithmetic so the invariant holds for all `uint256` inputs without caller-side bounds.
