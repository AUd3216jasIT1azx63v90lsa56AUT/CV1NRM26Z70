### Title
Off-by-one in `CannotIncreaseDebtPostMaturity` guard allows debt increase at `block.timestamp == maturity` - (File: `src/Midnight.sol`)

### Summary
The maturity guard in `Midnight.take` uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` at exact equality, permitting a seller to increase debt at the maturity timestamp itself. The protocol's own documented invariant explicitly states "maturity boundary must not allow debt increase by rounding, **timestamp equality**, or callback sequencing." The liquidation post-maturity check uses strict `>`, so debt created at exactly `maturity` is immediately liquidatable one second later.

### Finding Description
**Root cause — line 391, `src/Midnight.sol`:**

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

The `<=` operator passes when `block.timestamp == offer.market.maturity`, allowing `sellerDebtIncrease > 0`.

**Exploit flow:**

1. Attacker (seller) has any valid sell offer ratified by a SetterRatifier (or any ratifier) — a normal protocol operation.
2. `block.timestamp` is warped/timed to exactly `market.maturity`.
3. Attacker calls `Midnight.take` with `units > sellerPos.credit`, so `sellerDebtIncrease = units - sellerCreditDecrease > 0`.
4. Line 391: `block.timestamp <= offer.market.maturity` → `maturity <= maturity` → **TRUE** → guard passes.
5. Line 476: `isHealthy` check passes because the seller has sufficient collateral (pre-maturity health check: `maxDebt >= debt`).
6. `sellerPos.debt += sellerDebtIncrease` executes — debt is created at maturity.
7. At `block.timestamp == maturity + 1`, the liquidation check at line 622 evaluates `postMaturityMode ? block.timestamp > market.maturity` → `maturity+1 > maturity` → **TRUE** → position is immediately liquidatable in post-maturity mode regardless of health.

**Why existing checks fail:**

- The `SellerIsLiquidatable` check at line 476 uses `isHealthy`, which is the pre-maturity health check (`maxDebt >= debt`). At `block.timestamp == maturity`, the position is healthy, so this check passes.
- The `CannotIncreaseDebtPostMaturity` guard uses `<=` instead of `<`, creating a one-second window where debt can be created that is immediately liquidatable in the next block.
- No existing test covers `block.timestamp == market.maturity` with `sellerDebtIncrease > 0`; all post-maturity tests use `market.maturity + 1`.

### Impact Explanation
Debt is created at exactly `block.timestamp == maturity`, violating the core invariant "debt must not increase after maturity" and the explicit maturity boundary invariant. At `block.timestamp == maturity + 1`, the position enters post-maturity mode and is immediately liquidatable regardless of collateral health. The seller has no opportunity to repay before liquidation in the next block, and the LIF at `maturity + 1` is `WAD` (1.0), meaning the liquidator seizes collateral at face value with no incentive — but the position is still liquidatable, and the debt was created in violation of protocol rules.

### Likelihood Explanation
Requires a transaction to land in the exact block where `block.timestamp == market.maturity`. On EVM chains with predictable block timestamps, a sophisticated attacker can monitor the mempool and time their transaction. The only precondition is a valid sell offer (ratified by any ratifier), which is a standard protocol operation. The attack is repeatable for any market whose maturity timestamp falls on a block boundary.

### Recommendation
Change the guard at line 391 from `<=` to `<`:

```solidity
// Before (buggy):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns with the liquidation post-maturity check, which uses strict `block.timestamp > market.maturity`, and with the documented invariant that timestamp equality must not allow debt increase.

### Proof of Concept
Foundry unit test (add to `TakeTest.sol`):

```solidity
function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;

    // Warp to exactly maturity (not maturity+1 as existing tests do)
    vm.warp(market.maturity);
    borrowerOffer.expiry = market.maturity; // offer still valid at maturity
    borrowerOffer.maxUnits = units;
    deal(address(loanToken), lender, units);
    collateralize(market, borrower, units);

    // SHOULD revert with CannotIncreaseDebtPostMaturity, but currently PASSES
    take(units, lender, borrowerOffer);

    // Invariant violation: debt was increased at maturity
    assertGt(midnight.debtOf(id, borrower), 0, "debt increased at maturity - invariant violated");

    // One second later: position is immediately liquidatable in post-maturity mode
    vm.warp(market.maturity + 1);
    // postMaturityMode=true: block.timestamp > market.maturity is now true
    midnight.liquidate(market, 0, 0, midnight.debtOf(id, borrower), borrower, true, address(this), address(0), "");
    assertEq(midnight.debtOf(id, borrower), 0, "debt liquidated immediately after maturity");
}
```

**Expected assertions with the bug present:**
- `take` at `block.timestamp == maturity` succeeds (no revert) — invariant violated
- `debtOf(id, borrower) > 0` after the take
- `liquidate` with `postMaturityMode = true` at `maturity + 1` succeeds
