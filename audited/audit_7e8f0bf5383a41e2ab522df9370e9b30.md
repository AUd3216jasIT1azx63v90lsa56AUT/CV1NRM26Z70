Audit Report

## Title
Incremental `withdraw` calls via `multicall` drain `pendingFee` super-proportionally through `mulDivUp` rounding accumulation - (File: `src/Midnight.sol`)

## Summary
In `withdraw`, `pendingFeeDecrease` is computed as `mulDivUp(_position.pendingFee, units, _position.credit)` per call. Because each call rounds up by at most 1 and both `pendingFee` and `credit` decrease after each call, splitting a withdrawal of N units into N calls of 1 unit each accumulates up to `pendingFee` extra units of decrease compared to a single call. A lender can drain their entire `pendingFee` obligation while withdrawing only a small fraction of their credit, permanently zeroing future continuous fee accrual on the remaining credit and causing direct, permanent loss of protocol fee revenue.

## Finding Description

**Code path:** `withdraw` → line 485 `_updatePosition` (no-op in same block: zero elapsed time, no slashing) → lines 489–493:

```solidity
if (_position.credit > 0) {
    pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
    _position.pendingFee -= pendingFeeDecrease;
}
_position.credit -= UtilsLib.toUint128(units);
```

`mulDivUp` is `(x * y + (d - 1)) / d` — ceiling division, rounding error ≤ 1 per call.

**Root cause:** After each call, both `_position.pendingFee` and `_position.credit` decrease. The ratio `pendingFee/credit` is not preserved across calls because each ceiling-rounding step removes slightly more `pendingFee` than the exact proportional share. The next call operates on a lower `pendingFee` but also a lower `credit`, and ceiling rounding fires again. Errors compound across calls.

**Exploit flow (concrete):**

State: `pendingFee = 3`, `credit = 10`, `withdrawable >= 5`.

| Call | `mulDivUp(pF, 1, cr)` | pF after | cr after |
|------|----------------------|----------|----------|
| 1 | `(3+9)/10 = 1` | 2 | 9 |
| 2 | `(2+8)/9 = 1` | 1 | 8 |
| 3 | `(1+7)/8 = 1` | 0 | 7 |
| 4 | 0 | 0 | 6 |
| 5 | 0 | 0 | 5 |
| **Total** | **3** | | |

Single `withdraw(units=5)`: `mulDivUp(3, 5, 10) = (15+9)/10 = 2`. Remaining `pendingFee = 1`.

The 5-call path drains 3 vs 2 — 1 extra unit, leaving `pendingFee = 0` on 5 remaining credit.

**Amplified case:** `pendingFee = 100`, `credit = 10000`, 100 calls of 1 unit each.
- Each call: `mulDivUp(pF_i, 1, cr_i) = 1` (since `pF_i + cr_i - 1 >= cr_i` while `pF_i > 0`).
- Total drain: 100. Remaining `pendingFee = 0`.
- Single `withdraw(100)`: `mulDivUp(100, 100, 10000) = 1`. Remaining `pendingFee = 99`.
- **100× amplification**: lender drains 100 units of `pendingFee` instead of 1, by withdrawing only 1% of their credit.

**Why existing checks fail:**

The Certora rule `pendingFeeDecreasesProportionallyOnWithdraw` in `certora/specs/ContinuousFee.spec` line 106 only asserts the single-call formula — it does not bound the sum of N calls. The Certora README explicitly states `multicall` is removed from verification, with the justification that "if all other functions respect an invariant, by induction `multicall` also respects it." This induction argument is sound for per-state invariants (e.g., `pendingFee <= credit`, which is verified in `pendingContinuousFeeBoundedByCredit`), but it is **not sound** for cross-call accounting properties like "total `pendingFee` decrease from N withdrawals of 1 unit equals the decrease from 1 withdrawal of N units." That property is not an invariant preserved per call; it is a global property about the composition of calls that the Certora setup explicitly does not verify.

## Impact Explanation

`pendingFee` is the sole source of future continuous fee accrual: `fee = pendingFee * elapsed / remaining_time` (lines 814–815 of `updatePositionView`). By draining `pendingFee` to 0 while retaining most of their credit, the lender permanently zeroes all future fee accrual on that credit. The protocol's `continuousFeeCredit` accumulates nothing further from that position, constituting a direct, permanent loss of continuous fee revenue proportional to the remaining credit and time-to-maturity. The amplification factor is approximately `credit / pendingFee`, which is large when the continuous fee rate or time-to-maturity is small.

## Likelihood Explanation

**Preconditions:**
- Lender has a position with `pendingFee > 0` and `credit > 0` — normal post-take state.
- `withdrawable >= N` where N is the number of 1-unit calls (N = `pendingFee` in the worst case).
- All calls in the same block — achievable atomically via `multicall`.

All preconditions are routine. The attack is repeatable at any time the lender holds credit with a non-zero `pendingFee`. `multicall` is confirmed present and callable by any user. The attack requires no privileged access and is executable by any lender.

## Recommendation

Replace the per-call proportional decrease with a formula that is sub-additive across calls. Two options:

1. **Track cumulative withdrawn units:** Store a `withdrawnUnits` field per position and compute `pendingFeeDecrease = mulDivUp(initialPendingFee, totalWithdrawnUnits, initialCredit) - alreadyDecreased`. This makes the total decrease path-independent.

2. **Use `mulDivDown` instead of `mulDivUp`:** Rounding down for the lender's `pendingFeeDecrease` means the lender can never drain more than the exact proportional share. The protocol would then favor itself on rounding, which is the correct direction for fee accounting.

Option 2 is the minimal fix: change line 490 from `mulDivUp` to `mulDivDown`. This ensures the total decrease across N calls of 1 unit is at most the decrease from a single call of N units (by the `mulDivAddDownDown` additivity property already proven in `certora/specs/MulDiv.spec` line 32).

## Proof of Concept

```solidity
// State: pendingFee = 100, credit = 10000, withdrawable >= 100
// Repay enough to make withdrawable >= 100

bytes[] memory calls = new bytes[](100);
for (uint i = 0; i < 100; i++) {
    calls[i] = abi.encodeCall(midnight.withdraw, (market, 1, lender, lender));
}
midnight.multicall(calls);

// After: pendingFee == 0, credit == 9900
// Expected (single call of 100): pendingFee == 99, credit == 9900
// Protocol lost 99 units of future fee revenue from this position
```

The test can be added to `test/ContinuousFeeTest.sol` alongside the existing `testWithdrawReducesPendingFee` fuzz test, with a fixed case using `withdrawAmount = 1` repeated 100 times vs a single `withdrawAmount = 100`, asserting the final `pendingFee` values differ.
