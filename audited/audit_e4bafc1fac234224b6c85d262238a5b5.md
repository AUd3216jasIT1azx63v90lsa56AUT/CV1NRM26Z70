Audit Report

## Title
Buy-offer `maxAssets` budget bypassed via zero-rounding of `buyerAssets` when `offerPrice < WAD` - (File: src/Midnight.sol)

## Summary
In `take()`, for a buy offer with `maxAssets > 0`, the consumed budget is incremented by `buyerAssets = mulDivDown(units, buyerPrice, WAD)`. Because `buyerPrice` algebraically equals `offerPrice` for all buy offers (the settlement fee cancels), and `offerPrice < WAD` for every tick below `MAX_TICK` (5819 out of 5820 ticks), calling `take()` with `units = 1` yields `buyerAssets = 0`. The `consumed` counter is never incremented, so the `ConsumedAssets` guard never triggers, and the offer can be filled an unlimited number of times even after the budget is fully exhausted.

## Finding Description
**Algebraic identity for buy offers (`src/Midnight.sol:361-362`):**
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;
// For buy offers: buyerPrice = (offerPrice - _settlementFee) + _settlementFee = offerPrice
```
The settlement fee cancels, so `buyerPrice = offerPrice` unconditionally for buy offers.

**Zero-rounding condition (`src/Midnight.sol:363`):**
```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;
// units=1, buyerPrice=offerPrice < 1e18 => mulDivDown(1, offerPrice, 1e18) = 0
```
`tickToPrice(MAX_TICK) = 1e18` and `tickToPrice(MAX_TICK - 2) = 1e18 - 1e12`, so every tick from 0 to 5819 yields `offerPrice < WAD`, making `buyerAssets = 0` for `units = 1`.

**Consumed accounting failure (`src/Midnight.sol:367-369`):**
```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    // += 0 when buyerAssets rounds to 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
    // passes: consumed + 0 <= maxAssets, even when consumed == maxAssets
}
```
`newConsumed` never increases, so `ConsumedAssets()` is never triggered. Each call with `units = 1`:
- Increments the maker's credit by 1 unit (or reduces debt by 1)
- Increments the taker's debt by 1 unit (or reduces credit by 1)
- Transfers 0 tokens (`buyerAssets - sellerAssets = 0`)
- Leaves `consumed` unchanged

The existing test `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol:858`) confirms this exactly: with `consumed` pre-set to `maxAssets` via `setConsumed`, `take(1, ...)` succeeds, `consumed` stays at `maxAssets`, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all increase.

## Impact Explanation
Any buy offer with `maxAssets > 0` and `tick < MAX_TICK` has its exposure cap completely bypassed. The maker accumulates unbounded credit and the taker accumulates unbounded debt, with zero token transfers per fill. The core invariant that offer fill amount cannot exceed remaining offer capacity is broken. A maker who sets `maxAssets` to limit their lending exposure receives no protection: their credit position grows without bound while the taker's debt grows without bound, with no corresponding token flow. This constitutes an accounting integrity failure with direct impact on position solvency.

## Likelihood Explanation
**Preconditions:**
1. A buy offer exists with `maxAssets > 0` and any `tick < MAX_TICK` — the common case, as `MAX_TICK` is the par/1:1 price and all economically meaningful discount prices are below it (5819 of 5820 ticks).
2. The taker is any unprivileged address that is not the maker.
3. The taker must pass `isHealthy` after each fill (requires sufficient collateral to absorb the debt increase).

No time-warp, special market state, or privileged access is required. The bug is present at any `block.timestamp` and is repeatable indefinitely in a single transaction via a loop.

## Recommendation
Track consumed budget in units rather than assets when `buyerAssets` can round to zero, or enforce a minimum `buyerAssets > 0` when `maxAssets > 0`. The most direct fix is to add a check that `units > 0 => buyerAssets > 0` when `offer.maxAssets > 0` and `offer.buy == true`:
```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(buyerAssets > 0, ZeroBuyerAssets());
}
```
Alternatively, use `units` as the consumed metric unconditionally (as is already done in the `maxUnits` branch), or require `units >= WAD / offerPrice` to guarantee `buyerAssets >= 1`.

## Proof of Concept
The existing test at `test/TakeTest.sol:858` (`testBugBuyMaxAssetsBypass`) is a complete, self-contained proof of concept:
1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (offerPrice < WAD).
2. Pre-consume the full budget: `midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender)`.
3. Call `take(1, borrower, lenderOffer)` — succeeds without revert.
4. Assert `consumed == maxAssets` (unchanged), `buyerAssets == 0`, `sellerAssets == 0`.
5. Assert `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.
