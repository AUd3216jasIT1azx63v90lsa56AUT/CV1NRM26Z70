Audit Report

## Title
Zero-`buyerAssets` rounding bypasses `maxAssets` consumption cap on buy offers — (`src/Midnight.sol`)

## Summary
When a buy offer sets `maxAssets > 0`, the consumed counter is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For any `tick < MAX_TICK`, `buyerPrice < 1e18`, so `units = 1` yields `buyerAssets = 0`. The consumed counter never advances, and a fully-consumed offer (consumed == maxAssets) remains indefinitely takeable. Each successful take still mutates position state — increasing the maker's credit and the taker's debt by 1 unit — while transferring zero tokens.

## Finding Description

**Root cause — `src/Midnight.sol` lines 358–373:**

`tickToPrice` returns values that are multiples of `PRICE_ROUNDING_STEP = 1e12` and is strictly less than `WAD = 1e18` for every tick below `MAX_TICK = 5820`. [1](#0-0) 

For `units = 1` and any `buyerPrice < 1e18`:

```
buyerAssets = units.mulDivDown(buyerPrice, WAD)
            = mulDivDown(1, buyerPrice, 1e18)
            = 0   // because 1 * buyerPrice < 1e18
``` [2](#0-1) 

The cap check then becomes:

```
newConsumed = consumed[maker][group] += 0;   // unchanged
require(newConsumed <= offer.maxAssets);      // trivially passes
``` [3](#0-2) 

Position state is updated unconditionally after the cap check — buyer credit and seller debt both increase by `units`: [4](#0-3) 

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N` at any `tick < MAX_TICK`.
2. Taker calls `take(offer, ..., units=1, ...)` repeatedly.
3. Each call: `buyerAssets = 0`, `consumed += 0`, cap check passes.
4. Each call: maker credit +1, taker debt +1, zero tokens transferred.
5. The offer is never consumed; the loop runs indefinitely.

**Why existing checks fail:** The sole guard against over-filling is `require(newConsumed <= offer.maxAssets)`. Because `newConsumed` is unchanged when `buyerAssets = 0`, this check is trivially satisfied forever. There is no `require(units == 0 || buyerAssets > 0)` guard.

## Impact Explanation
The `maxAssets` budget is completely bypassed for any buy offer at any tick in `[0, MAX_TICK)`. A maker's credit position grows without bound (one unit per call) while `consumed` stays frozen. The taker accumulates unbounded debt without receiving any tokens. The core protocol invariant — "offers cannot be replayed, overfilled, or reused" — is violated: a fully-consumed offer (consumed == maxAssets) remains indefinitely takeable. This constitutes unauthorized state change and accounting corruption directly in the core protocol contract.

## Likelihood Explanation
Preconditions are minimal and universally present: any buy offer with `maxAssets > 0` at any tick in `[0, MAX_TICK)` is affected — that is, 5819 out of 5820 valid ticks. The only immune tick is `MAX_TICK` itself (where `tickToPrice = 1e18` exactly, giving `buyerAssets = 1` for `units = 1`). The attack requires no special permissions, no oracle manipulation, and no unusual token behavior. It is repeatable in every block at negligible gas cost by any unprivileged address that is not the maker.

## Recommendation
Add a guard that rejects a non-zero `units` take when the resulting `buyerAssets` (or `sellerAssets` for sell offers) rounds to zero under `maxAssets` mode:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || (offer.buy ? buyerAssets : sellerAssets) > 0, ZeroAssets());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` such that the computed assets are always non-zero, or switch the consumed tracking to units-based when `buyerPrice < WAD` would cause truncation.

## Proof of Concept
The repository contains an explicit regression test that reproduces the exact scenario: [5](#0-4) 

The test (`testBugBuyMaxAssetsBypass`):
1. Sets `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16`.
2. Pre-sets `consumed = maxAssets` via `setConsumed` (fully consumed state).
3. Calls `take(1, borrower, lenderOffer)`.
4. Asserts `buyerAssets == 0`, `sellerAssets == 0`.
5. Asserts `consumed` is still equal to `maxAssets` (cap not advanced).
6. Asserts lender credit and borrower debt both strictly increased.

Run with: `forge test --match-test testBugBuyMaxAssetsBypass -vvv`
