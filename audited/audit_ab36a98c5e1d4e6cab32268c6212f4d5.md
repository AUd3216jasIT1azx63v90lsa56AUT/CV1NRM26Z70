Audit Report

## Title
Unconditional zero-value `safeTransfer` to receiver when `filledSellerAssets == 0` causes revert on zero-transfer-reverting `loanToken` - (File: src/periphery/MidnightBundles.sol)

## Summary
In `supplyCollateralAndSellWithUnitsTarget`, when `targetUnits == 0` the takes loop is skipped entirely, leaving `filledSellerAssets == 0`. The final `SafeTransferLib.safeTransfer(loanToken, receiver, 0)` at line 168 is called unconditionally — unlike the referral fee transfer at line 167 which is guarded by `if (referralFeeAssets > 0)`. On any `loanToken` that reverts on zero-value `transfer` calls, every invocation with `targetUnits == 0` and `minSellerAssets == 0` will revert, making the no-op path permanently unusable for such tokens despite the contract explicitly documenting that no-ops are allowed.

## Finding Description
`supplyCollateralAndSellWithUnitsTarget` accepts `targetUnits` with no lower-bound check. When `targetUnits == 0`, the loop condition `filledUnits < targetUnits` is immediately false (both start at 0), so the loop body never executes and `filledSellerAssets` remains 0. The post-loop `require(filledUnits == targetUnits)` passes trivially (0 == 0). `referralFeeAssets = 0.mulDivDown(referralFeePct, WAD) = 0`, so the slippage check `require(0 - 0 >= minSellerAssets)` passes when `minSellerAssets == 0`. The referral transfer at line 167 is correctly guarded by `if (referralFeeAssets > 0)`, but the receiver transfer at line 168 has no equivalent guard and unconditionally calls `SafeTransferLib.safeTransfer(loanToken, receiver, 0)`. `SafeTransferLib.safeTransfer` makes a raw `.call()` to `token.transfer(receiver, 0)` and propagates any revert from the token via the assembly block. If the token reverts on zero-value transfers, the entire call reverts. The contract-level NatDev comment (`/// @dev No-ops are allowed.`) confirms `targetUnits == 0` is an intended valid input, making this a real behavioral defect rather than an invalid-input edge case.

## Impact Explanation
Any call to `supplyCollateralAndSellWithUnitsTarget` with `targetUnits == 0` and `minSellerAssets == 0` on a market whose `loanToken` reverts on zero-value transfers will always revert at line 168. The transaction fails atomically — collateral supplies are rolled back — so no funds are permanently lost. However, the no-op path (explicitly documented as supported) is rendered permanently unusable for such tokens, and callers waste gas on every attempt. Severity is low: no fund loss, no state corruption, but a concrete and repeatable denial of a documented feature.

## Likelihood Explanation
All three preconditions are independently reachable by any unprivileged caller: (1) `targetUnits == 0` has no on-chain guard; (2) `minSellerAssets == 0` is the natural default for no slippage protection; (3) tokens that revert on zero-value transfers are uncommon but real (e.g., certain USDT deployments, custom ERC-20s). The scenario is deterministic and repeatable — every call with these inputs on such a token will revert.

## Recommendation
Add a zero-value guard on line 168 consistent with the guard already present on line 167:

```solidity
uint256 netSellerAssets = filledSellerAssets - referralFeeAssets;
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
if (netSellerAssets > 0) SafeTransferLib.safeTransfer(loanToken, receiver, netSellerAssets);
```

This makes the receiver transfer symmetric with the referral transfer and eliminates the zero-value call on tokens that revert for it.

## Proof of Concept
1. Deploy or fork a market whose `loanToken` is an ERC-20 that reverts on `transfer(to, 0)`.
2. Call `supplyCollateralAndSellWithUnitsTarget` with `targetUnits = 0`, `minSellerAssets = 0`, an empty `takes` array, and any valid `taker`/`receiver`.
3. Observe: `require(filledUnits == targetUnits)` passes (0 == 0); `require(0 >= 0)` passes; `safeTransfer(loanToken, receiver, 0)` is reached and the token reverts; the entire transaction reverts.
4. Confirm: replacing line 168 with `if (netSellerAssets > 0) safeTransfer(...)` causes the same call to succeed.

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));

                revert(add(returndata, 0x20), mload(returndata))
