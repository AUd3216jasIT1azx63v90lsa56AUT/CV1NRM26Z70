Audit Report

## Title
`maxAssets` cap bypassed on buy offers via zero-rounding of `buyerAssets` — (`File: src/Midnight.sol`)

## Summary
For a buy offer, `buyerPrice` algebraically equals `offerPrice` because the settlement fee is added back after being subtracted. With `units = 1` and any tick > 0 (i.e., `offerPrice < WAD`), `mulDivDown(1, offerPrice, WAD)` rounds to zero, so the `consumed` mapping is incremented by zero on every call. The `maxAssets` cap is therefore never reached, allowing the offer to be filled an unlimited number of times.

## Finding Description
**Root cause — fee cancellation:**
At lines 361–362 of `src/Midnight.sol`, for a buy offer:
```
sellerPrice = offerPrice - _settlementFee
buyerPrice  = sellerPrice + _settlementFee  // = offerPrice
```
`buyerPrice` is always `offerPrice`, regardless of `_settlementFee`.

**Root cause — rounding to zero:**
At line 363:
```
buyerAssets = units.mulDivDown(buyerPrice, WAD)
            = mulDivDown(1, offerPrice, WAD)
            = 0   (for any offerPrice < WAD, i.e. any tick > 0)
```

**Broken guard:**
At lines 367–369:
```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```
When `buyerAssets = 0`, `newConsumed` never increases. The `require` trivially passes on every call. No guard enforces `buyerAssets > 0` before accepting the fill.

**Position updates still execute:**
At lines 408–414, credit and debt are updated based on `units` (not `buyerAssets`), so the maker gains 1 unit of credit per call even though 0 tokens are exchanged.

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = X > 0` and any tick > 0.
2. Taker (holding credit in the market) calls `take(offer, …, 1, …)` in a loop (or via `multicall` in a single block).
3. Each call: `buyerAssets = 0`, `consumed` stays at 0, the cap check passes.
4. Each call: maker gains 1 unit of credit; taker loses 1 unit of credit; 0 tokens transferred.
5. After N calls the maker holds N units of credit backed by 0 deposited tokens.

## Impact Explanation
The `maxAssets` budget cap on buy offers is completely non-functional for any tick > 0. A maker accumulates unbounded credit redeemable for loan tokens without any corresponding token deposit, directly violating the solvency invariant that contract token balances cover all outstanding credit. This constitutes a critical accounting/solvency break.

## Likelihood Explanation
Preconditions are minimal: a buy offer with `maxAssets > 0` and tick > 0 (the common case), and a taker holding existing credit. No privileged access is required. The attack is fully executable by any unprivileged user, is repeatable within a single block via `multicall`, and is limited only by gas cost.

## Recommendation
Before accepting a fill, require that the computed asset amount is non-zero, or track consumed budget in `units` rather than in asset amounts when `maxAssets > 0`. A minimal fix:

```solidity
require(buyerAssets > 0, ZeroBuyerAssets());
```

placed before the `consumed` update, or alternatively enforce a minimum `units` such that `mulDivDown(units, offerPrice, WAD) >= 1`. A more robust fix is to track consumed in units uniformly and convert the `maxAssets` cap to a units cap at offer creation time.

## Proof of Concept
1. Deploy the protocol and create a market with tick spacing 1.
2. Maker creates a buy offer: `tick = 1`, `maxAssets = 1e18`, `buy = true`.
3. Taker (with credit ≥ 100 in the market) calls `take(offer, …, 1, …)` 100 times.
4. Assert: `consumed[maker][group] == 0` after all 100 calls (cap never advanced).
5. Assert: maker's credit position increased by 100 units; 0 loan tokens transferred to taker.
6. The `maxAssets` cap of `1e18` was never approached despite 100 fills.
