Audit Report

## Title
Unconditional `safeTransfer(collateralToken, receiver, 0)` in `liquidate()` blocks bad-debt-only liquidations on zero-revert collateral tokens - (File: src/Midnight.sol)

## Summary
`liquidate()` unconditionally calls `SafeTransferLib.safeTransfer` at line 696 with `seizedAssets`, which remains 0 in the bad-debt-only path (both inputs zero). The NatSpec at line 577 explicitly documents this path as valid and intends "0 token transferred," but the implementation always issues the low-level `transfer(receiver, 0)` call. For any collateral token that reverts on zero-amount transfers, every bad-debt realization attempt on that market permanently reverts, freezing `lossFactor` and `totalUnits` accounting and preventing lenders from socializing losses.

## Finding Description
**Root cause:** The guard at line 643 correctly skips collateral/debt accounting when both inputs are zero:

```solidity
if (repaidUnits > 0 || seizedAssets > 0) { ... }
```

After this block, `seizedAssets` is still 0. The bad-debt accounting block at lines 626–641 executes correctly, updating `_position.debt`, `_marketState.lossFactor`, `_marketState.totalUnits`, and `_marketState.continuousFeeCredit`. Then, unconditionally at line 696:

```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
```

`SafeTransferLib.safeTransfer` (lines 12–22 of `src/libraries/SafeTransferLib.sol`) makes a raw low-level call:

```solidity
(bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
if (!success) {
    assembly ("memory-safe") { revert(add(returndata, 0x20), mload(returndata)) }
}
```

If the collateral token reverts on `transfer(receiver, 0)`, the entire transaction reverts — including all bad-debt accounting already performed. The same issue applies to `SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits)` at line 717 when `repaidUnits = 0`.

**Why existing checks fail:**
- `atMostOneNonZero(repaidUnits, seizedAssets)` at line 595 explicitly permits both to be 0 — this is the intended bad-debt path.
- `require(_position.debt > 0, NotBorrower())` at line 596 is satisfied by any borrower with outstanding debt.
- There is no `if (seizedAssets > 0)` guard before the `safeTransfer` call at line 696.
- There is no `if (repaidUnits > 0)` guard before the `safeTransferFrom` call at line 717.

The NatSpec at line 577 directly contradicts the implementation: `"Passing both 0 for seizedAssets and repaidUnits allows to realize bad debt with 0 token transferred."` The implementation always issues the transfer call regardless.

## Impact Explanation
Bad-debt-only liquidations are permanently and repeatably blocked for any market whose collateral token reverts on `transfer(to, 0)`. The `lossFactor` and `totalUnits` state that socializes bad debt among lenders cannot be updated. Unhealthy positions with full bad debt remain stuck indefinitely. Lenders in affected markets cannot recover their proportional share of losses through the documented bad-debt mechanism. This is a concrete, permanent DoS on a core protocol invariant explicitly documented in the NatSpec.

## Likelihood Explanation
**Preconditions:**
1. A market is created with a collateral token that reverts on `transfer(to, 0)`. This behavior is present in several deployed ERC20 tokens (e.g., BNB, LEND, and various custom tokens). Market creation is permissionless in Midnight, so any user can deploy such a market.
2. A borrower's position reaches a state where `badDebt > 0` (collateral value falls below debt at `maxLif` pricing) — a normal market condition under adverse price movements.
3. Any unprivileged liquidator calls `liquidate(market, idx, 0, 0, borrower, ...)`.

No special privilege is required. The attack is repeatable on every bad-debt realization attempt. The attacker does not need to deploy the collateral token — they only need to create a market using an existing zero-revert token.

## Recommendation
Guard both transfer calls with zero-value checks:

```solidity
// Line 696
if (seizedAssets > 0) {
    SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
}

// Line 717
if (repaidUnits > 0) {
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
}
```

This aligns the implementation with the documented NatSpec invariant and ensures bad-debt-only liquidations (`seizedAssets = 0`, `repaidUnits = 0`) complete without issuing any token calls.

## Proof of Concept
1. Deploy a mock ERC20 collateral token that reverts on `transfer(to, 0)`.
2. Create a permissionless Midnight market using this token as collateral.
3. Open a borrow position and manipulate the oracle price so that `badDebt > 0` (collateral value at `maxLif` pricing falls below debt).
4. Call `liquidate(market, collateralIndex, 0, 0, borrower, false, receiver, address(0), "")`.
5. Observe: the call reverts at line 696 due to the zero-amount transfer revert from the collateral token, despite the bad-debt accounting at lines 626–641 having executed correctly in the same call frame.
6. Confirm: `_marketState.lossFactor` and `_marketState.totalUnits` are never updated; the borrower's bad debt is never socialized; every subsequent attempt to realize bad debt on this market also reverts.
