### Title
Off-by-one at maturity boundary allows debt increase while blocking post-maturity liquidation simultaneously - (`src/Midnight.sol`)

### Summary
The debt-increase guard in `take` uses `block.timestamp <= offer.market.maturity` (inclusive), while the post-maturity liquidation gate uses `block.timestamp > market.maturity` (exclusive). At the single timestamp where `block.timestamp == market.maturity`, a healthy borrower can increase debt and yet no liquidator can invoke post-maturity mode against that position in the same block. This directly violates the protocol invariants `debt_increase_after_maturity_forbidden` and `overdue_debt_after_maturity_is_liquidatable_even_if_healthy`.

### Finding Description
**Root cause — boundary mismatch between two independent checks:**

`src/Midnight.sol` line 391 (debt-increase guard in `take`):
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
`<=` means the condition is satisfied (no revert) when `block.timestamp == market.maturity`, so `sellerDebtIncrease > 0` is permitted.

`src/Midnight.sol` lines 620–624 (liquidatability check in `liquidate`):
```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```
`>` means post-maturity mode is rejected when `block.timestamp == market.maturity`.

**Preconditions (all attacker-controlled):**
1. `touchMarket` has no lower-bound check on maturity (`src/Midnight.sol` line 758: `require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar())`). Any unprivileged caller can create a market with `maturity = block.timestamp` or any past value.
2. The borrower must be healthy at the time of the `take` call (the `SellerIsLiquidatable` check at line 476 only blocks unhealthy sellers; a well-collateralised borrower passes it).

**Exploit flow:**
1. Attacker (market creator) calls `touchMarket` with `market.maturity = block.timestamp`. Market is created and immediately at its maturity timestamp.
2. Attacker (borrower) supplies collateral and calls `take` against a lender offer with `sellerDebtIncrease > 0`. The guard at line 391 evaluates `block.timestamp <= market.maturity` → `true` → no revert. Debt is written to the position.
3. In the same block, any liquidator calls `liquidate(..., postMaturityMode=true)`. The check at line 622 evaluates `block.timestamp > market.maturity` → `false` → reverts `NotLiquidatable`.
4. Normal-mode liquidation (`postMaturityMode=false`) also fails because the borrower is healthy (`originalDebt <= maxDebt`).

The existing test `testLiquidatePostMaturityModeBeforeMaturity` in `test/LiquidationTest.sol` lines 170–173 explicitly documents and asserts this behaviour:
```solidity
// At exact maturity: still not available (only valid strictly after maturity).
vm.warp(market.maturity);
vm.expectRevert(IMidnight.NotLiquidatable.selector);
midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");
```
This confirms the liquidation-side behaviour is intentional, making the debt-side `<=` the defect.

### Impact Explanation
At `block.timestamp == market.maturity` a borrower can open or increase a debt position in a market that is, by the protocol's own definition, expired. For that block, the position is immune to post-maturity liquidation and, if healthy, immune to normal-mode liquidation as well. The whitepaper invariants `debt_increase_after_maturity_forbidden` and `overdue_debt_after_maturity_is_liquidatable_even_if_healthy` are both violated simultaneously at this boundary. The window closes in the next block when `block.timestamp > market.maturity`, but the debt created during it persists and the borrower has borrowed against an expired market without the protocol's maturity enforcement applying.

### Likelihood Explanation
Requires no privilege: any address can call `touchMarket` with `maturity = block.timestamp`, and any address can be the borrower. The condition is reproducible deterministically on any block by setting `maturity` to the current `block.timestamp` at market creation time. It is repeatable across every block on every chain. The only constraint is that the borrower must be healthy at the moment of the `take`, which is the normal operating condition for any new borrow.

### Recommendation
Change the debt-increase guard in `take` from `<=` to `<` to make the maturity boundary exclusive and consistent with the liquidation check:

```solidity
// src/Midnight.sol line 391
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns both sides of the boundary: debt cannot increase at or after maturity (`<`), and post-maturity liquidation is available strictly after maturity (`>`). The existing test `testBuyPastMaturity` / `testSellPastMaturity` (which bound `timestamp` to `market.maturity + 1`) should be extended to also cover `timestamp == market.maturity`.

### Proof of Concept
```solidity
// Foundry unit test
function testDebtAtExactMaturityNotLiquidatable() public {
    // 1. Create market with maturity = block.timestamp
    uint256 ts = block.timestamp;
    market.maturity = ts;
    id = midnight.touchMarket(market);
    borrowerOffer.market = market;
    borrowerOffer.expiry = ts; // offer valid at this timestamp

    // 2. Borrower supplies collateral and takes debt at exactly maturity
    collateralize(market, borrower, 100);
    deal(address(loanToken), lender, 100);
    // Assert: take succeeds (debt increase allowed at block.timestamp == maturity)
    take(100, lender, borrowerOffer);
    assertGt(midnight.debtOf(id, borrower), 0, "debt created at maturity");

    // 3. Liquidator attempts post-maturity liquidation at same timestamp
    // Assert: reverts NotLiquidatable (block.timestamp > maturity is false)
    vm.expectRevert(IMidnight.NotLiquidatable.selector);
    midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");

    // 4. Confirm normal-mode liquidation also fails (borrower is healthy)
    vm.expectRevert(IMidnight.NotLiquidatable.selector);
    midnight.liquidate(market, 0, 0, 0, borrower, false, address(this), address(0), "");

    // 5. One second later, post-maturity liquidation becomes available
    vm.warp(ts + 1);
    midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), ""); // succeeds
}
```

Expected assertions: steps 2 passes (debt > 0), steps 3 and 4 revert with `NotLiquidatable`, step 5 succeeds — proving the invariant is violated at the exact maturity boundary.
