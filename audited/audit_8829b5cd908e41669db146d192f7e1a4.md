Audit Report

## Title
Zero-increment `consumed` bypass allows unlimited takes on fully-consumed buy offers — (File: `src/Midnight.sol`)

## Summary
When `offer.buy == true` and `offer.maxAssets > 0`, the consumed accounting increments by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For any tick below `MAX_TICK`, `buyerPrice < WAD`, so choosing `units = 1` yields `buyerAssets = 0` via integer truncation. The guard `require(newConsumed <= offer.maxAssets)` then passes even when `consumed == maxAssets`, because `maxAssets + 0 <= maxAssets`. All downstream position mutations still execute with the non-zero `units` value, producing real credit/debt changes with zero token transfer.

## Finding Description
**Root cause — `src/Midnight.sol` lines 363–373:**

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For buy offers, `consumed` advances by `buyerAssets`, not by `units`. `mulDivDown` performs `(units * buyerPrice) / WAD` with truncation. When `units * buyerPrice < WAD` — trivially satisfied with `units = 1` whenever `buyerPrice < WAD` — the result is `0`. The guard becomes `maxAssets + 0 <= maxAssets`, which always passes. [1](#0-0) 

Downstream state mutations at lines 382–417 all operate on `units` (not `buyerAssets`):
- `buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)` — non-zero
- `sellerDebtIncrease = units - sellerCreditDecrease` — non-zero
- `totalUnits` incremented by `buyerCreditIncrease - sellerCreditDecrease`
- `claimableSettlementFee` incremented by `buyerAssets - sellerAssets = 0` [2](#0-1) 

The token transfer at line 455 sends `buyerAssets - sellerAssets = 0` tokens, so no assets move. [3](#0-2) 

`tickToPrice` confirms that for any `tick < MAX_TICK`, the returned price is strictly less than `WAD` (the formula is `1e36 / (1e18 + wExp(...))`, which is `< WAD` for any positive `wExp` term). [4](#0-3) 

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` only catches cases where `buyerAssets > 0` pushes `newConsumed` over the cap. A zero increment never triggers it.
- The `reduceOnly` check (lines 392–395) only applies when `offer.reduceOnly` is set; it is not a general guard.
- There is no `require(units == 0 || buyerAssets > 0)` guard. [5](#0-4) 

## Impact Explanation
A fully-consumed buy offer (`consumed == maxAssets`) can be taken an unlimited number of times. Each such take:
- Increases the maker's (buyer's) credit and the taker's (seller's) debt by `units` with zero token payment.
- Increases `totalUnits` without a corresponding increase in `claimableSettlementFee` or token reserves, creating a permanent accounting mismatch between units outstanding and assets backing them.
- Violates the core protocol invariant that offers cannot be replayed, overfilled, or reused after being fully consumed.

The economic consequence is that the taker accumulates unbacked debt and the maker accumulates credit that is not backed by deposited assets, which can lead to insolvency of the market.

## Likelihood Explanation
Preconditions are minimal and common:
- `offer.buy == true`, `offer.maxAssets > 0` — standard buy offer configuration.
- `buyerPrice < WAD` — holds for every tick below `MAX_TICK`; this is the normal operating range.
- `units = 1` — trivially satisfies `1 * buyerPrice < WAD` whenever `buyerPrice < WAD`.

Any unprivileged taker can trigger this after any partial or full fill of a qualifying buy offer. No special permissions, oracle manipulation, or victim mistakes are required. The attack is repeatable without limit.

## Recommendation
Replace the consumed increment for buy offers with a value that is always non-zero when `units > 0`. The simplest fix is to use `mulDivUp` instead of `mulDivDown` when tracking consumed assets for buy offers, or to add an explicit guard:

```solidity
require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
```

Alternatively, track consumed in `units` (not assets) uniformly, or add a minimum-units check before the consumed accounting block.

## Proof of Concept
The repository contains a confirming test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (two references found by search). The test pre-sets `consumed = maxAssets = 1`, calls `take(1, borrower, lenderOffer)` with `tick = MAX_TICK - 16` (so `buyerPrice < WAD`), and asserts that `buyerAssets == 0`, `consumed` remains at `maxAssets`, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase — directly confirming the described exploit path.
