Audit Report

## Title
Unbounded `offer.maxAssets` causes arithmetic overflow in `buyerAssetsToUnits`, DoS-ing bundler calls - (`File: src/periphery/ConsumableUnitsLib.sol`)

## Summary
`ConsumableUnitsLib.consumableUnits` passes `offer.maxAssets.zeroFloorSub(consumed)` directly to `TakeAmountsLib.buyerAssetsToUnits`, which calls `UtilsLib.mulDivUp(targetBuyerAssets, WAD, buyerPrice)`. Since `mulDivUp` uses plain checked Solidity 0.8 arithmetic and `offer.maxAssets` is an unbounded `uint256`, any buy offer with `maxAssets > type(uint256).max / WAD` (~1.157×10⁵⁹) causes an arithmetic overflow revert. Because the `consumableUnits` call sits outside the bundler's `try/catch` block, the entire bundler transaction reverts, enabling a permissionless DoS against any bundler user who includes such an offer.

## Finding Description

**Code path:**

`MidnightBundles.supplyCollateralAndSellWithUnitsTarget` (and three other bundler functions) → `ConsumableUnitsLib.consumableUnits` → `TakeAmountsLib.buyerAssetsToUnits` → `UtilsLib.mulDivUp`

**Root cause — `UtilsLib.mulDivUp` uses plain multiplication in checked Solidity 0.8:**

```solidity
// src/libraries/UtilsLib.sol:34-35
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;   // reverts on overflow when x * y > type(uint256).max
}
``` [1](#0-0) 

The function is not wrapped in `unchecked`, so Solidity 0.8 checked arithmetic applies. With `y = WAD = 1e18`, any `x > type(uint256).max / 1e18` overflows. [2](#0-1) 

**`offer.maxAssets` is a raw `uint256` with no upper-bound validation:** [3](#0-2) 

**`consumableUnits` passes the raw remaining capacity into `buyerAssetsToUnits`:** [4](#0-3) 

**`buyerAssetsToUnits` calls `mulDivUp(targetBuyerAssets, WAD, buyerPrice)`:** [5](#0-4) 

The `require(buyerPrice <= WAD)` guard bounds only the denominator `d`, not the multiplicand `x`. It does not prevent `x * WAD` from overflowing.

**The `consumableUnits` call is outside the bundler's `try/catch` in all four bundler functions:**

In `supplyCollateralAndSellWithUnitsTarget`: [6](#0-5) 

In `buyWithUnitsTargetAndWithdrawCollateral`: [7](#0-6) 

In `buyWithAssetsTargetAndWithdrawCollateral`: [8](#0-7) 

In `supplyCollateralAndSellWithAssetsTarget`: [9](#0-8) 

The `try/catch` wraps only `IMidnight.take()`, not the preceding `consumableUnits` call. An overflow in `consumableUnits` propagates uncaught and reverts the entire transaction.

**Why existing checks fail:**
- `atMostOneNonZero(offer.maxAssets, offer.maxUnits)` checks mutual exclusivity only, not magnitude.
- `require(buyerPrice <= WAD)` bounds the denominator, not the multiplicand.
- The `consumableUnits` call is architecturally outside the error-isolation boundary.

## Impact Explanation
Any of the four bundler functions that includes a buy offer with `maxAssets > type(uint256).max / WAD` in its `takes[]` array will revert entirely. A malicious maker can publish such an offer on-chain to permanently DoS any automated bundler-based taker who discovers and includes it, without any ongoing cost. This disables the bundler's core functionality for affected users and offer sets. [10](#0-9) 

## Likelihood Explanation
Creating an offer requires no special privilege — any address can be a maker. Setting `maxAssets = type(uint256).max` is a valid `uint256` value accepted by `take()`. The overflow threshold (~1.157×10⁵⁹) is far below `type(uint256).max` (~1.157×10⁷⁷), so any value above ~1.16×10⁵⁹ triggers it. The attack is repeatable with no on-chain state change needed after offer creation, and costs only the gas to create the offer once. [3](#0-2) 

## Recommendation
Apply one or more of the following fixes:

1. **Cap `offer.maxAssets` at validation time** in `Midnight.take()` to `type(uint256).max / WAD` (or `type(uint128).max` to match other asset fields).
2. **Use `mulDivUp` with overflow-safe intermediate math** (e.g., via Solady/OpenZeppelin's `FullMath.mulDivUp` using 512-bit intermediates) in `UtilsLib`.
3. **Move `consumableUnits` inside the `try/catch`** in each bundler loop, or wrap it in its own `try/catch`, so a revert in `consumableUnits` skips the offer rather than reverting the entire transaction.

Option 3 is the most targeted fix for the bundler; options 1 or 2 address the root cause more broadly. [1](#0-0) 

## Proof of Concept
```solidity
// Minimal Foundry test sketch
function test_consumableUnits_overflow_dos() public {
    // 1. Attacker constructs a buy offer with maxUnits=0, maxAssets=type(uint256).max
    Offer memory maliciousOffer = Offer({
        ...,
        buy: true,
        maxUnits: 0,
        maxAssets: type(uint256).max,
        ...
    });

    // 2. Victim builds a takes[] array containing the malicious offer
    Take[] memory takes = new Take[](1);
    takes[0] = Take({ offer: maliciousOffer, units: 1, ratifierData: "" });

    // 3. Victim calls bundler — expect revert due to arithmetic overflow in mulDivUp
    vm.expectRevert(); // Panic: arithmetic overflow
    bundles.supplyCollateralAndSellWithUnitsTarget(
        1, 0, victim, victim, new CollateralSupply[](0), takes, 0, address(0)
    );
}
```

The revert occurs at `UtilsLib.mulDivUp` when computing `type(uint256).max * 1e18`, which exceeds `type(uint256).max`. No special setup beyond a valid market and the malicious offer is required. [11](#0-10)

        uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
        if (offer.maxUnits > 0) {
            return offer.maxUnits.zeroFloorSub(consumed);
        } else if (offer.buy) {
            return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
        } else {
            return TakeAmountsLib.sellerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
        }
    }
