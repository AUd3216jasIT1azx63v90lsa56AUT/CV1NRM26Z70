### Title
Fee-on-Transfer Loan Token Causes `claimableSettlementFee` Overcounting, Leading to Protocol Insolvency - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` before the ERC20 transfer executes. When the loan token charges a fee on transfer, `address(this)` receives strictly less than `buyerAssets - sellerAssets`, but the accounting records the full nominal amount. Repeated takes accumulate a deficit, causing `claimableSettlementFee` to exceed the contract's actual token balance and violating the core solvency invariant.

### Finding Description

**Exact code path** — `src/Midnight.sol`:

Line 418 (state write, before transfer):
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

Line 455 (ERC20 transfer to `address(this)`):
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
``` [2](#0-1) 

**Root cause**: The protocol assumes `safeTransferFrom` delivers exactly the requested amount. For a fee-on-transfer token with rate `f`, the contract receives `(buyerAssets - sellerAssets) * (1 - f)`, but `claimableSettlementFee` is incremented by the full `buyerAssets - sellerAssets`. There is no balance-before/balance-after check anywhere in `take()`.

**Attacker-controlled inputs**: The market creator (an explicitly listed unprivileged attacker profile per `live_context.json`) creates a market with a fee-on-transfer loan token. No further privileged action is required — the fee is a static, pre-existing property of the token, not a runtime action by the token owner. Any subsequent taker calling `take()` triggers the accounting drift automatically. [3](#0-2) 

**Exploit flow**:
1. Market creator calls `touchMarket` with a fee-on-transfer loan token (e.g., 1% fee per transfer).
2. Taker calls `take(offer, ..., units, ...)` with a valid offer.
3. `buyerAssets - sellerAssets` = 100 tokens (nominal spread).
4. Line 418 executes: `claimableSettlementFee[loanToken] += 100`.
5. Line 455 executes: contract receives only 99 tokens (1% fee deducted by token).
6. Net deficit per take: 1 token. After N takes: deficit = N tokens.
7. `claimableSettlementFee[loanToken]` = N × 100, actual balance increase = N × 99.

**Why existing checks fail**: The Certora rule `takeIncreasesClaimableSettlementFee` (line 75 of `certora/specs/WithdrawableMonotonicity.spec`) and the rule in `certora/specs/Midnight.spec` (line 75) both assert that `claimableSettlementFee` increases by exactly `buyerAssets - sellerAssets`. These rules are proven against an idealized ERC20 model that does not model fee-on-transfer behavior — they do not protect against real-world fee-on-transfer tokens. [4](#0-3) [5](#0-4) 

The `live_context.json` core invariants explicitly state: `"ERC20 transfer deltas must match accounting deltas"` and `"fee-on-transfer, rebasing, false-return, ERC777-like hooks, and non-standard decimals should be tested if not explicitly excluded"`. `SECURITY.md` does not exclude fee-on-transfer tokens. [6](#0-5) 

### Impact Explanation
`claimableSettlementFee[loanToken]` grows faster than the contract's actual token balance. When `claimSettlementFee` is called to withdraw the full recorded fee, it either reverts (if the contract has insufficient balance) or — more critically — drains tokens that belong to lenders' `withdrawable` assets or borrowers' collateral redemptions, causing direct loss of user funds and protocol insolvency. The deficit compounds with every `take()` call on the affected market. [7](#0-6) 

### Likelihood Explanation
**Preconditions**: A market must exist with a fee-on-transfer loan token. Market creation is permissionless. The attacker need only be a market creator (explicitly in scope) or operate in an existing such market. No privileged keys, oracle manipulation, or user mistakes are required. The bug triggers on every `take()` call — it is deterministic and repeatable. Fee-on-transfer tokens exist in production (e.g., tokens with configurable transfer fees).

### Recommendation
Replace the nominal-amount accounting with a balance-delta pattern for the settlement fee portion transferred to `address(this)`:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 received = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += received;
```

This ensures `claimableSettlementFee` only records tokens actually received. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens (e.g., via a market creation check or clear documentation), but this would be a weaker mitigation given the permissionless market creation model. [8](#0-7)

rule takeIncreasesClaimableSettlementFee(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiverIfTakerIsSeller, address takerCallback, bytes takerCallbackData, address anyToken) {
    uint256 before = claimableSettlementFee(anyToken);

    uint256 buyerAssets;
    uint256 sellerAssets;
    buyerAssets, sellerAssets = take(e, offer, ratifierData, units, taker, receiverIfTakerIsSeller, takerCallback, takerCallbackData);

    // We know that buyerAssets - sellerAssets >= 0, see rule settlementFeeSpreadBounds.
    assert anyToken == offer.market.loanToken => claimableSettlementFee(anyToken) == before + buyerAssets - sellerAssets;
    assert anyToken != offer.market.loanToken => claimableSettlementFee(anyToken) == before;
}
