Audit Report

## Title
`claimableSettlementFee` Over-Credited on Fee-on-Transfer Loan Token in `take` - (File: `src/Midnight.sol`)

## Summary

In `Midnight.take`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` at line 418, before the token transfer executes. When the loan token charges a transfer fee, the contract receives only `(buyerAssets - sellerAssets) * (1 - feeRate)` at line 455, but the accounting records the full nominal amount. Because `claimSettlementFee` performs no balance guard, the fee claimer can transfer more tokens than were ever received as fees, with the shortfall drawn from lenders' `withdrawable` balances, causing direct protocol insolvency.

## Finding Description

**Root cause — accounting before transfer, no received-amount check:**

At `src/Midnight.sol` line 418, the settlement fee accounting is updated unconditionally with the nominal spread:

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;  // line 418
```

The actual token transfer to the contract occurs later at line 455:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
```

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34) only checks for revert and a false return value. It performs no balance-before/balance-after check. With a fee-on-transfer token charging rate `f`, the contract receives only `(buyerAssets - sellerAssets) * (1 - f)`, while `claimableSettlementFee` was credited with the full `buyerAssets - sellerAssets`. The discrepancy per `take` call is `(buyerAssets - sellerAssets) * f`.

**`claimSettlementFee` has no balance guard:**

```solidity
function claimSettlementFee(address token, uint256 amount, address receiver) external {
    require(msg.sender == feeClaimer, OnlyFeeClaimer());
    claimableSettlementFee[token] -= amount;
    SafeTransferLib.safeTransfer(token, receiver, amount);  // line 309
}
```

The function transfers `amount` from the contract's unified token balance, which also covers lenders' `withdrawable` amounts. When the fee claimer calls this for the full (inflated) `claimableSettlementFee`, the shortfall is taken from lender funds.

**Why existing checks fail:**

- `SafeTransferLib` does not measure received amounts.
- `claimSettlementFee` has no invariant check comparing `claimableSettlementFee[token]` against actual token balance minus `withdrawable`.
- The Certora rule `takeIncreasesClaimableSettlementFee` (line 75 of `certora/specs/WithdrawableMonotonicity.spec`) models token transfers as balance-neutral, so it cannot detect the discrepancy introduced by a fee-on-transfer token.

**Exploit path:**

1. Attacker calls `touchMarket` (public, no access control, lines 755–791) with a fee-on-transfer token as `loanToken`.
2. Attacker (or any taker) calls `take` on a valid offer in that market.
3. Line 418 credits `claimableSettlementFee` with the full nominal spread.
4. Line 455 transfers only `(1 - f)` of that spread to the contract.
5. Repeated `take` calls accumulate the discrepancy.
6. When `feeClaimer` calls `claimSettlementFee` for the full `claimableSettlementFee` amount (normal intended behavior), `safeTransfer` draws the shortfall from lenders' `withdrawable` balances.
7. Lenders' subsequent `withdraw` calls revert or receive less than their credit entitles them to.

## Impact Explanation

Direct protocol insolvency: lenders lose funds proportional to `(buyerAssets - sellerAssets) * feeRate` per `take` call, compounding across all takes in fee-on-transfer markets. The `live_context.json` core invariant "ERC20 transfer deltas must match accounting deltas" is violated. This matches the highest-priority bug class "direct loss of user funds / protocol insolvency" listed in `live_context.json`.

## Likelihood Explanation

Market creation is permissionless (`touchMarket` is `public`). The protocol explicitly supports "arbitrary loan token" (`live_context.json`). Fee-on-transfer tokens are a well-known token class. No privileged access, oracle manipulation, or special permissions are required — any unprivileged market creator and taker can trigger this. The effect is repeatable and accumulates with every `take` in the affected market. The `live_context.json` explicitly lists "token charges fee" as a fuzz axis and "fee-on-transfer... should be tested if not explicitly excluded" as a core invariant concern; `SECURITY.md` contains no exclusion for fee-on-transfer tokens.

## Recommendation

Replace the pre-transfer accounting with a balance-before/balance-after pattern to measure the actual received amount:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 received = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += received;
```

Alternatively, document and enforce that fee-on-transfer tokens are not supported as loan tokens, and add a validation check in `touchMarket` (though this is difficult to enforce on-chain without a whitelist).

## Proof of Concept

**Minimal Foundry test plan:**

1. Deploy a mock ERC20 that deducts a 1% fee on every `transferFrom` to a non-owner address.
2. Create a market with this token as `loanToken` via `touchMarket`.
3. Set up a valid offer (maker with credit, taker with collateral).
4. Call `take`; record `claimableSettlementFee[token]` and the contract's actual token balance.
5. Assert: `claimableSettlementFee[token] > contract_balance_attributable_to_fees` (discrepancy = `spread * 0.01`).
6. Call `claimSettlementFee(token, claimableSettlementFee[token], attacker)`.
7. Assert: a lender's subsequent `withdraw` call reverts or returns less than their `withdrawable` credit, confirming fund drain from lender balances.
