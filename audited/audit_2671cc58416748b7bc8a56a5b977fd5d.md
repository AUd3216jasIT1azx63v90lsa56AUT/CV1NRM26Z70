### Title
Nonce-Race Front-Run on `EcrecoverAuthorizer.setIsAuthorized` Enables Unauthorized Withdrawal - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary

`EcrecoverAuthorizer.setIsAuthorized` is fully permissionless: any caller may submit any `Authorization` struct paired with a valid victim-signed signature. Because the nonce is sequential and consumed on first use, an attacker who holds a victim-signed grant for nonce=N can front-run the victim's pending revocation (also nonce=N) in the mempool, consuming the nonce and causing the revocation to revert with `InvalidNonce()`. The attacker is then `isAuthorized` in Midnight and can immediately call `Midnight.withdraw` to drain the victim's withdrawable assets.

### Finding Description

**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` (lines 24–48): [1](#0-0) 

The nonce check at line 26 increments `nonce[authorization.authorizer]` atomically on first use: [2](#0-1) 

The signature check at lines 33–36 only requires `ecrecover(digest) == authorization.authorizer` OR that the signer is already authorized by the authorizer: [3](#0-2) 

The function then calls `Midnight.setIsAuthorized` with the attacker-controlled `authorized` and `isAuthorized` fields: [4](#0-3) 

**Root cause:** The `Authorization` struct binds `isAuthorized` (grant vs. revoke) into the signed digest, so a grant `{victim, attacker, true, N, far}` and a revocation `{victim, attacker, false, N, ...}` are two distinct valid messages that both satisfy nonce=N. There is no mechanism to invalidate one without consuming the nonce. The function is permissionless — `msg.sender` is irrelevant — so the attacker can submit the grant before the victim's revocation lands.

**Exploit flow:**

1. Victim previously signed `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N, deadline=far}` off-chain (e.g., for a relayer) and shared the signature with the attacker but never submitted it.
2. Victim changes mind, signs `Authorization{authorizer=victim, authorized=attacker, isAuthorized=false, nonce=N}`, and broadcasts the revocation to the mempool.
3. Attacker observes the revocation in the mempool and front-runs it by submitting the old grant with the victim's valid signature.
4. Attacker's grant executes: `nonce[victim]` becomes N+1; `isAuthorized[victim][attacker] = true` in Midnight.
5. Victim's revocation reverts: `authorization.nonce == N` but `nonce[victim] == N+1` → `InvalidNonce()`.
6. Attacker calls `Midnight.withdraw(market, units, victim, attacker)`: [5](#0-4) 
   The check `isAuthorized[victim][attacker]` is `true`, so the call succeeds and tokens are transferred to the attacker.

**Why existing checks fail:**
- `InvalidNonce()`: prevents replay of the *same* message, but does not prevent two *different* signed messages (grant vs. revoke) from racing on the same nonce.
- `Unauthorized()`: the attacker's signature is genuinely from the victim (`ecrecover` returns `victim`), so this passes.
- `Expired()`: the attacker's grant carries a far deadline, so this passes.

### Impact Explanation

Once the attacker is `isAuthorized`, they can call `Midnight.withdraw(market, units, victim, attacker)` and receive the victim's withdrawable loan-token balance directly. This drains `_marketState.withdrawable` and `_position.credit` for the victim, reducing the contract's token balance available to cover legitimate lender redemptions — matching the scoped insolvency impact. [6](#0-5) 

### Likelihood Explanation

**Preconditions:**
- Victim must have signed a grant for nonce=N off-chain and shared it with the attacker (realistic in relayer/meta-transaction flows).
- Victim must then attempt to revoke using the same nonce=N via EcrecoverAuthorizer.
- Attacker must observe the mempool (standard on public chains).

**Feasibility:** High on any chain with a public mempool. The attacker needs only to submit a higher-gas transaction with the already-held valid signature. No special privileges required. Repeatable for each new nonce if the victim continues using EcrecoverAuthorizer for revocations.

### Recommendation

Add a direct nonce-invalidation function so the victim can burn nonce=N without executing any authorization:

```solidity
function invalidateNonce(uint256 expectedNonce) external {
    require(expectedNonce == nonce[msg.sender], InvalidNonce());
    nonce[msg.sender]++;
    emit NonceInvalidated(msg.sender, expectedNonce);
}
```

This lets the victim atomically advance their nonce past any outstanding signed-but-unsubmitted grants before broadcasting a revocation, eliminating the race window. [7](#0-6) 

### Proof of Concept

```solidity
// Foundry test — add to SetIsAuthorizedWithSigTest.sol
function testFrontRunRevocationViaNonceRace() public {
    // Setup: victim authorizes EcrecoverAuthorizer in Midnight
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);

    // Step 1: victim signs a grant (nonce=0) off-chain, gives to attacker
    Authorization memory grant = Authorization({
        authorizer: victim,
        authorized: attacker,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 365 days
    });
    Signature memory grantSig = signAuthorization(grant, victim);

    // Step 2: victim changes mind, signs revocation (nonce=0), broadcasts
    Authorization memory revoke = Authorization({
        authorizer: victim,
        authorized: attacker,
        isAuthorized: false,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory revokeSig = signAuthorization(revoke, victim);

    // Step 3: attacker front-runs with the grant
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(grant, grantSig);

    // Assert: attacker is now authorized
    assertEq(midnight.isAuthorized(victim, attacker), true);
    assertEq(ecrecoverAuthorizer.nonce(victim), 1);

    // Step 4: victim's revocation reverts
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(revoke, revokeSig);

    // Step 5: attacker drains victim's withdrawable assets
    // (assumes victim has credit in market from prior supply)
    vm.prank(attacker);
    midnight.withdraw(market, victimUnits, victim, attacker);

    // Assert: attacker received tokens, victim's credit is zero
    assertEq(loanToken.balanceOf(attacker), victimUnits);
    assertEq(midnight.creditOf(marketId, victim), 0);
}
```

**Expected assertions:** `InvalidNonce` revert on the victim's revocation; `loanToken.balanceOf(attacker) == victimUnits`; `midnight.isAuthorized(victim, attacker) == true` after the front-run.

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );
