### Title
`consumableUnits` reverts with `TickOutOfRange` for asset-capped offers, propagating outside the `try/catch` and reverting the entire bundle - (`File: src/periphery/ConsumableUnitsLib.sol`)

### Summary
`MidnightBundles` wraps each `IMidnight.take()` call in a `try/catch` to skip bad offers, but calls `ConsumableUnitsLib.consumableUnits()` unconditionally before the `try` block. When an offer has `maxUnits == 0` (i.e., `maxAssets > 0`), `consumableUnits` delegates to `TakeAmountsLib.buyerAssetsToUnits` or `sellerAssetsToUnits`, both of which call `TickLib.tickToPrice(offer.tick)`. If `offer.tick > MAX_TICK (5820)`, `tickToPrice` reverts with `TickOutOfRange`, which propagates through `consumableUnits` and out of the bundle function entirely, reverting the whole transaction instead of skipping the offer.

### Finding Description

**Exact code path:**

`MidnightBundles.buyWithUnitsTargetAndWithdrawCollateral` (and all other bundle entry points) execute the following loop body:

```solidity
// src/periphery/MidnightBundles.sol:74-85
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // ← NOT in try/catch
);
try IMidnight(MIDNIGHT)
    .take(takes[i].offer, ...) returns (...) {
    ...
} catch {}
```

`ConsumableUnitsLib.consumableUnits` branches on `offer.maxUnits`:

```solidity
// src/periphery/ConsumableUnitsLib.sol:16-22
if (offer.maxUnits > 0) {
    return offer.maxUnits.zeroFloorSub(consumed);   // safe — no tickToPrice
} else if (offer.buy) {
    return TakeAmountsLib.buyerAssetsToUnits(...);   // calls tickToPrice
} else {
    return TakeAmountsLib.sellerAssetsToUnits(...);  // calls tickToPrice
}
```

Both `TakeAmountsLib` functions call `TickLib.tickToPrice(offer.tick)` as their first statement:

```solidity
// src/periphery/TakeAmountsLib.sol:22 and 41
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
```

`tickToPrice` unconditionally reverts for any out-of-range tick:

```solidity
// src/libraries/TickLib.sol:44-45
function tickToPrice(uint256 tick) internal pure returns (uint256) {
    require(tick <= MAX_TICK, TickOutOfRange());
```

**Attacker-controlled inputs:** The `Offer` struct is passed entirely by the caller (taker) as calldata. There is no on-chain storage or validation of offer fields before `consumableUnits` is called. A malicious maker can sign (off-chain, at zero cost) an offer with `offer.tick = 5821` (or any value > 5820) and `offer.maxAssets > 0` (forcing `offer.maxUnits == 0`). When a taker's routing system includes this poisoned offer in a bundle alongside legitimate offers, `consumableUnits` reverts before the `try/catch` is ever reached.

**Why existing checks fail:** The NatSpec at `MidnightBundles.sol:43` explicitly documents "Reverts if ConsumableUnitsLib reverts," confirming the revert is not caught. The `try/catch` at line 79 only guards `IMidnight.take()`. There is no tick-range guard on `offer.tick` before `consumableUnits` is called in any bundle function. The `require(offer.tick % _marketState.tickSpacing == 0, TickNotAccessible())` check in `Midnight.take()` (line 351) is inside the `try` block and does not protect `consumableUnits`.

### Impact Explanation
A malicious maker poisons a taker's bundle by injecting a signed offer with `tick > 5820` and `maxAssets > 0`. The entire bundle reverts, causing all legitimate takes in the bundle to fail. In a time-sensitive context (near maturity, liquidation window, or volatile market), the taker suffers real financial loss from the failed execution. The attack is repeatable at zero cost to the maker (signing is free, no on-chain state required).

### Likelihood Explanation
Any maker can sign an offer with an out-of-range tick at zero cost. Takers using automated routing or aggregation systems that fetch offers from an off-chain order book are the primary target. The precondition is simply that the poisoned offer appears in the `takes` array with `maxAssets > 0`. The attack is trivially repeatable and requires no privileged access, no special market state, and no on-chain setup.

### Recommendation
Wrap the `consumableUnits` call inside the `try/catch`, or add a tick-range guard before calling `consumableUnits` in each bundle loop:

```solidity
// Option A: skip offers with out-of-range tick before consumableUnits
if (takes[i].offer.tick > MAX_TICK) continue;

// Option B: move consumableUnits inside try/catch (requires refactoring)
```

Alternatively, add a guard inside `consumableUnits` itself: if `offer.maxUnits == 0` and `offer.tick > MAX_TICK`, return 0 instead of reverting, so the offer is naturally skipped (0 units to take).

### Proof of Concept

```solidity
// Foundry unit test
function testPoisonedOfferRevertsBundle() public {
    // Setup: one legitimate offer and one poisoned offer
    Take[] memory takes = new Take[](2);

    // Legitimate offer: tick = MAX_TICK, maxAssets > 0
    takes[0].offer = legitimateOffer; // tick = 5820, maxAssets = 1e18

    // Poisoned offer: tick > MAX_TICK, maxAssets > 0 (maxUnits == 0)
    takes[1].offer = poisonedOffer;   // tick = 5821, maxAssets = 1e18, maxUnits = 0

    // Fund taker
    deal(loanToken, taker, 2e18);

    // Assert: the entire bundle reverts with TickOutOfRange
    vm.prank(taker);
    vm.expectRevert(TickLib.TickOutOfRange.selector);
    bundles.buyWithUnitsTargetAndWithdrawCollateral(
        targetUnits, maxBuyerAssets, taker, permit, takes, withdrawals, receiver, 0, address(0)
    );

    // Confirm: if poisoned offer is removed, bundle succeeds
    Take[] memory cleanTakes = new Take[](1);
    cleanTakes[0] = takes[0];
    vm.prank(taker);
    bundles.buyWithUnitsTargetAndWithdrawCollateral(...cleanTakes...); // succeeds
}
```

**Expected assertions:** First call reverts with `TickOutOfRange`; second call (without the poisoned offer) succeeds. This confirms the bundle cannot skip the bad offer and that the revert is not caught.
