### Title
Missing Zero Address Check for `oracle` in `touchMarket()` Causes Permanent DoS for Borrowers in Affected Markets — (File: src/Midnight.sol)

### Summary
The `touchMarket()` function in `Midnight.sol` validates collateral token addresses and LLTV/maxLif parameters but does not validate that `CollateralParams.oracle != address(0)`. A market can be created with a zero oracle address. Any borrower who enters such a market is permanently locked: `isHealthy()`, `liquidate()`, `withdrawCollateral()` (with debt), and `take()` (for sellers with debt) all revert when they attempt to call `IOracle(address(0)).price()`, because Solidity 0.8.x reverts on ABI-decoding a `uint256` from empty return data.

### Finding Description
In `touchMarket()`, the validation loop over `collateralParams` checks:

```solidity
require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
require(isLltvAllowed(lltv), LltvNotAllowed());
require(market.collateralParams[i].maxLif == ..., InvalidMaxLif());
``` [1](#0-0) 

There is **no check** that `market.collateralParams[i].oracle != address(0)`. The market is then permanently stored via `IdLib.storeInCode(market, INITIAL_CHAIN_ID)`. [4](#0-3) 

Downstream, every function that touches a borrower's health calls:

```solidity
uint256 price = IOracle(_collateralParam.oracle).price();
``` [5](#0-4) [6](#0-5) 

When `oracle == address(0)`, the EVM CALL to address(0) returns `(success=true, data=0x)`. Solidity 0.8.34's ABI decoder then reverts attempting to decode a `uint256` from 0 bytes of return data. This revert propagates through:

- `isHealthy()` → called by `withdrawCollateral()` and `take()` (for sellers with debt)
- `liquidate()` → directly calls `IOracle(oracle).price()` [7](#0-6) [8](#0-7) 

### Impact Explanation
Any borrower who supplies collateral and takes debt in a market where any activated collateral has `oracle == address(0)` is permanently frozen:
- Cannot withdraw collateral (health check reverts)
- Cannot be liquidated (liquidate reverts)
- Cannot sell via `take()` while holding debt (health check reverts)

Their collateral and debt are permanently locked in the contract with no recovery path. The market's `withdrawable` accounting is also corrupted since repayments can still be made but the resulting withdrawable units can never be claimed by lenders (since `take()` for sellers with debt reverts).

### Likelihood Explanation
Market creation via `touchMarket()` is fully permissionless — any caller can create a market with `oracle == address(0)` by passing a `CollateralParams` struct with a zero oracle. A realistic path:
1. An attacker or a buggy integration contract calls any entry-point (`take`, `supplyCollateral`, etc.) with a `Market` struct containing `collateralParams[i].oracle = address(0)`.
2. The market is created and stored immutably.
3. Any user who subsequently borrows against that collateral is permanently locked.

The protocol's own comment acknowledges: *"Zero checks are not systematically performed"* [9](#0-8) , confirming this is a known gap rather than an intentional design choice.

### Recommendation
Add an explicit non-zero oracle check inside the `touchMarket()` collateral validation loop:

```solidity
require(market.collateralParams[i].oracle != address(0), InvalidOracle());
``` [1](#0-0) 

### Proof of Concept
1. Deploy a mock ERC-20 as `loanToken` and `collateralToken`.
2. Call `touchMarket()` (or any entry-point that calls it) with:
   ```solidity
   Market({
       loanToken: <valid ERC-20>,
       collateralParams: [CollateralParams({
           token: <valid ERC-20>,
           lltv: <allowed value>,
           maxLif: <valid value>,
           oracle: address(0)   // ← zero oracle
       })],
       maturity: block.timestamp + 30 days,
       ...
   })
   ```
3. Observe: market is created successfully, no revert.
4. Have a user call `supplyCollateral()` and then become a seller via `take()` to acquire debt.
5. Attempt `withdrawCollateral()` → reverts at `isHealthy()` → `IOracle(address(0)).price()`.
6. Attempt `liquidate()` → reverts at `IOracle(address(0)).price()`.
7. The borrower's collateral and debt are permanently locked with no exit path.
