### Title
Rebasing Collateral Token Negative Rebase Inflates `_position.collateral` Accounting, Blocking Liquidation and Creating Bad Debt - (File: src/Midnight.sol)

### Summary
`supplyCollateral` records the caller-supplied `assets` value directly into `_position.collateral[collateralIndex]` without a balance-before/balance-after check, and no mechanism exists to reconcile stored accounting with actual token holdings after an external rebase event. When a rebasing collateral token undergoes a negative rebase, the stored `_position.collateral` value exceeds Midnight's actual token balance, causing `isHealthy()` to compute an inflated `maxDebt` that prevents liquidation and creates irrecoverable bad debt.

### Finding Description

**Root cause — `supplyCollateral` (src/Midnight.sol:533)** [1](#0-0) 

The accounting write at line 533 sets `_position.collateral[collateralIndex] = oldCollateral + assets` unconditionally. `assets` is a caller-controlled parameter; the actual tokens received are transferred at line 545 via `safeTransferFrom`. For a standard ERC-20 these are equal, but for a rebasing token the stored value diverges from the real balance the moment a rebase occurs — there is no snapshot, no balance-diff check, and no on-chain hook to update the stored value.

**No token-type restriction in `touchMarket` (src/Midnight.sol:755-791)** [2](#0-1) 

`touchMarket` validates LLTV tiers, `maxLif`, sorted order, and maturity, but imposes **no restriction on the ERC-20 type** of the collateral token. Any address satisfying the sort order is accepted, including rebasing tokens (stETH, aTokens, etc.).

**`isHealthy` uses stored accounting, not actual balance (src/Midnight.sol:944-960)** [3](#0-2) 

`maxDebt` is computed from `_position.collateral[i]` — the stale stored value — multiplied by the oracle price and LLTV. After a negative rebase the stored value is larger than the real balance, so `maxDebt` is overestimated and the position appears healthy.

**`liquidate` uses the same stale value (src/Midnight.sol:612-622)** [4](#0-3) 

The liquidatability check `originalDebt > maxDebt` uses the same inflated `maxDebt`. If the stored collateral is large enough to make `maxDebt >= debt`, the `NotLiquidatable()` revert fires and no liquidator can touch the position.

**Exploit flow:**
1. Market creator (unprivileged) calls `touchMarket` with a rebasing token (e.g., stETH) as `collateralParams[0].token` — no check prevents this.
2. Borrower calls `supplyCollateral(market, 0, C, borrower)` — `_position.collateral[0]` is set to `C`; Midnight holds `C` stETH.
3. Borrower borrows up to the LLTV limit: `debt = C * price * lltv / ORACLE_PRICE_SCALE`.
4. A negative rebase occurs (e.g., validator slashing): Midnight's actual stETH balance drops to `C' < C`; `_position.collateral[0]` remains `C`.
5. `isHealthy()` returns `true` because `maxDebt = C * price * lltv / ORACLE_PRICE_SCALE >= debt`.
6. Actual collateral value is only `C' * price * lltv / ORACLE_PRICE_SCALE < debt` — position is insolvent.
7. `liquidate` reverts with `NotLiquidatable()` because `debt <= maxDebt` (inflated).
8. Borrower walks away; lenders absorb the bad debt with no recourse.

**Existing protections are insufficient:** The Certora `Healthiness.spec` rules assume oracle prices do not change during a call (`summaryPrice` ghost), but they do not model external balance changes from rebasing. The `stayHealthy` rules are therefore silent on this class of token.

### Impact Explanation
The core invariant "contract balances cover collateral" is violated: `sum(_position.collateral[i]) > actualTokenBalance`. The borrower's position is reported healthy by `isHealthy()` while the actual collateral is insufficient to cover the debt. Liquidators are blocked by `NotLiquidatable()`. The shortfall becomes permanent bad debt absorbed by lenders, with no on-chain path to recover it.

### Likelihood Explanation
**Preconditions:**
- A market must be created with a rebasing token as collateral (any unprivileged actor can do this via `touchMarket`).
- A borrower must supply that collateral and take on debt near the LLTV limit.
- A negative rebase must occur (e.g., stETH slashing, aToken protocol loss).

All three preconditions are realistic and independent of privileged actors. Rebasing tokens are widely used as collateral in DeFi. The negative rebase is an external protocol event, not a malicious token-owner action. The scenario is repeatable for every market that lists a rebasing token.

### Recommendation
1. **Disallow rebasing tokens by design** (preferred): Document and enforce that only non-rebasing, non-fee-on-transfer ERC-20s may be used as collateral. Add a balance-before/balance-after check in `supplyCollateral` and revert if `received != assets`.
2. **Balance-diff accounting**: Replace the direct `assets` write with `balanceAfter - balanceBefore` to capture the actual received amount, which also handles fee-on-transfer tokens.
3. **Rebase reconciliation**: Provide an admin or permissionless function that reads the actual token balance and writes down `_position.collateral` proportionally, enabling liquidation after a negative rebase.

Option 1 is the simplest and most robust; options 2 and 3 are complementary.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Market, CollateralParams} from "src/Midnight.sol";

// Minimal rebasing ERC-20: owner can slash all holders proportionally
contract RebasingToken is ERC20 {
    uint256 public scalar = 1e18; // 1:1 initially
    mapping(address => uint256) public shares;
    uint256 public totalShares;

    function mint(address to, uint256 amount) external {
        uint256 s = amount * 1e18 / scalar;
        shares[to] += s;
        totalShares += s;
        _mint(to, amount);
    }

    function negativeRebase(uint256 newScalar) external {
        // Reduce scalar → all balances shrink
        scalar = newScalar;
    }

    function balanceOf(address a) public view override returns (uint256) {
        return shares[a] * scalar / 1e18;
    }

    function transfer(address to, uint256 amount) public override returns (bool) {
        uint256 s = amount * 1e18 / scalar;
        shares[msg.sender] -= s;
        shares[to] += s;
        return true;
    }

    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 s = amount * 1e18 / scalar;
        shares[from] -= s;
        shares[to] += s;
        _allowances[from][msg.sender] -= amount;
        return true;
    }
}

contract RebasingCollateralTest is Test {
    Midnight midnight;
    RebasingToken rebToken;
    Market market;

    function setUp() public {
        midnight = new Midnight();
        rebToken = new RebasingToken();
        // build market with rebasing token as collateral
        // oracle returns price = ORACLE_PRICE_SCALE (1:1)
        // lltv = 0.77e18
    }

    function testNegativeRebaseCreatesUnliquidatableBadDebt() public {
        uint256 C = 1000e18;
        rebToken.mint(borrower, C);
        vm.prank(borrower);
        rebToken.approve(address(midnight), C);
        vm.prank(borrower);
        midnight.supplyCollateral(market, 0, C, borrower);

        // borrow at LLTV limit
        uint256 debt = C * lltv / WAD; // simplified
        // ... take offer to create debt ...

        // Assert: position is healthy before rebase
        assertTrue(midnight.isHealthy(market, id, borrower));

        // Negative rebase: 10% slash
        rebToken.negativeRebase(0.9e18);

        // Midnight's actual balance is now 900e18, but _position.collateral[0] = 1000e18
        assertEq(rebToken.balanceOf(address(midnight)), 900e18);
        assertEq(midnight.collateral(id, borrower, 0), 1000e18); // stale

        // isHealthy still returns true (uses stale 1000e18)
        assertTrue(midnight.isHealthy(market, id, borrower));

        // Liquidation reverts: NotLiquidatable
        vm.expectRevert(IMidnight.NotLiquidatable.selector);
        midnight.liquidate(market, 0, 0, debt, borrower, false, address(this), address(0), "");

        // Invariant violated: actual balance < sum of position.collateral
        assertLt(rebToken.balanceOf(address(midnight)), midnight.collateral(id, borrower, 0));
    }
}
```

**Expected assertions:**
- `midnight.collateral(id, borrower, 0) == 1000e18` (stale, pre-rebase)
- `rebToken.balanceOf(address(midnight)) == 900e18` (actual, post-rebase)
- `midnight.isHealthy(market, id, borrower) == true` (incorrectly)
- `liquidate(...)` reverts with `NotLiquidatable()`
- Invariant: `actualBalance < storedCollateral` — violated
