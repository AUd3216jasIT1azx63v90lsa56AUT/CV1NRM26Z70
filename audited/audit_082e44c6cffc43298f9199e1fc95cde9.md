### Title
Fully-consumed assets-capped buy offer with `buyerPrice < WAD` can be taken indefinitely for free units — (`src/Midnight.sol`)

### Summary
When a buy offer uses `maxAssets` as its cap and the offer's `buyerPrice < WAD` (i.e., `offerPrice < WAD`), calling `take()` with a small `units` value (e.g., `units = 1`) produces `buyerAssets = units.mulDivDown(buyerPrice, WAD) = 0`. Because consumed accounting adds `buyerAssets` (zero) to the stored value, the `ConsumedAssets` check passes even when `consumed == maxAssets`, allowing the offer to be taken an unlimited number of times after it is fully consumed. Each such take mutates real position state (credit, debt, `totalUnits`) and triggers the maker callback, all without transferring any assets.

### Finding Description
**Code path** — `src/Midnight.sol`, `take()`:

```
// line 363
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;

// lines 367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause**: For a buy offer, `consumed` is incremented by `buyerAssets`, not by `units`. When `buyerPrice < WAD` (any tick where `offerPrice < 1e18`), `mulDivDown(1, buyerPrice, WAD)` truncates to `0`. Adding `0` to a `consumed` that is already at `maxAssets` leaves `newConsumed == maxAssets`, which satisfies `<= maxAssets`. The guard is bypassed.

**Exploit flow**:
1. Attacker observes a buy offer with `maxAssets > 0` and `offerPrice < WAD` (e.g., `tick = MAX_TICK - 16`).
2. The offer is fully consumed: `consumed[maker][group] == maxAssets` (reached legitimately or via `setConsumed`).
3. Attacker calls `take(units=1, ...)`. `buyerAssets = 1.mulDivDown(buyerPrice, WAD) = 0`.
4. `newConsumed = maxAssets + 0 = maxAssets`. Check `maxAssets <= maxAssets` passes.
5. Position accounting executes with `units = 1`: buyer gains 1 unit of credit (or debt reduced by 1), seller gains 1 unit of debt (or credit reduced by 1), `totalUnits` increases by 1.
6. Maker callback fires with `buyerAssets=0, units=1`.
7. No assets are transferred. Step 3–6 can be repeated indefinitely.

**Why existing checks fail**: The `ConsumedAssets` check at line 369 compares `newConsumed` (asset-denominated) against `maxAssets`, but the position mutation and callback use `units` (unit-denominated). There is no guard that rejects a take when `buyerAssets == 0` and `units > 0`.

The protocol's own test suite documents this exact behavior in `testBugBuyMaxAssetsBypass` (lines 857–889 of `test/TakeTest.sol`), and the NatDoc at line 94 of `src/Midnight.sol` explicitly states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*

### Impact Explanation
A fully-consumed buy offer (assets mode, `buyerPrice < WAD`) can be taken an unlimited number of times by any unprivileged taker. Each extra take:
- Increases the seller's debt and the buyer's credit by 1 unit at zero cost.
- Inflates `totalUnits` and `marketState` accounting.
- Fires the maker's `onBuy` callback with `buyerAssets=0, units=1`, which may trigger unintended maker-side logic.
- Violates the core invariant that `consumed == maxAssets` means the offer is exhausted. [1](#0-0) [2](#0-1) [3](#0-2) 

### Likelihood Explanation
**Preconditions**:
- Buy offer with `maxAssets > 0` and `offerPrice < WAD` (any tick below the WAD threshold, e.g., `MAX_TICK - 16`).
- Offer must be fully consumed (`consumed == maxAssets`), reachable through normal fills or `setConsumed`.

**Feasibility**: Both conditions are normal protocol states. The tick range `offerPrice < WAD` is explicitly supported and documented. Any unprivileged taker can call `take()` directly. The attack is repeatable in a single transaction via multicall or a loop. [4](#0-3) 

### Recommendation
Add an explicit guard that rejects a take when `maxAssets > 0`, `units > 0`, and the computed asset delta is zero:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any non-trivial take (`units > 0`) against an assets-capped offer must produce a non-zero asset movement, preventing the rounding bypass. [5](#0-4) 

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already serves as a complete PoC. A minimal Foundry unit test:

```solidity
function testOverfillFullyConsumedBuyOffer() public {
    // Setup: buy offer, offerPrice < WAD, maxAssets = 1
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    // Pre-consume to maxAssets
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, 1, lender);

    uint256 debtBefore = midnight.debtOf(id, borrower);
    uint256 totalBefore = midnight.totalUnits(id);

    // Second take with units=1 on a fully-consumed offer
    (uint256 buyerAssets,) = take(1, borrower, lenderOffer);

    // Assertions that prove the bug:
    assertEq(buyerAssets, 0, "no assets transferred");
    assertEq(midnight.consumed(lender, lenderOffer.group), 1, "consumed unchanged");
    assertGt(midnight.debtOf(id, borrower), debtBefore, "debt increased for free");
    assertGt(midnight.totalUnits(id), totalBefore, "totalUnits inflated");
    // Invariant that SHOULD hold but doesn't:
    // assertEq(midnight.debtOf(id, borrower), debtBefore, "offer exhausted => no state change");
}
```

Expected: the take reverts or `debtOf` is unchanged. Actual: take succeeds and debt increases by 1 unit at zero asset cost. [6](#0-5)
