### Title
Off-by-One in Maturity Guard Allows Debt Increase at Exact Maturity Timestamp - (File: src/Midnight.sol)

### Summary
The `take` function guards against post-maturity debt increases with `block.timestamp <= offer.market.maturity`, but this `<=` comparison permits debt creation when `block.timestamp == offer.market.maturity`. The protocol's own invariant documentation explicitly forbids debt increase at the maturity boundary via timestamp equality. A single `take` (or a `multicall`-batched sequence of takes) executed at the exact maturity block creates new overdue debt that is immediately liquidatable, constituting bad debt creation.

### Finding Description
**Root cause — off-by-one in the maturity guard:**

```solidity
// src/Midnight.sol:391
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

When `block.timestamp == offer.market.maturity` the left-hand side of the disjunction is `true`, so the require passes unconditionally regardless of `sellerDebtIncrease`. The check should be strict: `block.timestamp < offer.market.maturity`.

**Exploit path:**

1. Attacker (taker) waits for (or warps to) `block.timestamp == market.maturity`.
2. Attacker calls `take(sellOffer, ...)` where `offer.buy == false` (taker is buyer, maker/seller is the borrower). The seller has zero existing credit, so `sellerCreditDecrease = 0` and `sellerDebtIncrease = units > 0`.
3. Line 391 evaluates: `block.timestamp <= offer.market.maturity` → `maturity <= maturity` → `true`. Require passes.
4. Line 414 executes: `sellerPos.debt += sellerDebtIncrease`. New debt is written to the seller's position.
5. Because `block.timestamp == market.maturity`, this debt is immediately overdue and liquidatable in post-maturity mode.
6. Via `multicall` (lines 211–220), the attacker can atomically repeat this across multiple markets with the same maturity in a single transaction, each delegatecall sharing the same `block.timestamp`.

**Why existing checks do not stop it:**

- The `CannotIncreaseDebtPostMaturity` guard uses `<=` not `<`, so it is satisfied at equality.
- The `enterGate` check (lines 402–406) only applies if a gate is configured; ungated markets have no secondary protection.
- The `SellerIsLiquidatable` check (called inside `take` via `isHealthy`) only reverts if the seller is already unhealthy *before* the take; a healthy-but-undercollateralized-after-take seller is not caught.
- All existing post-maturity tests use `market.maturity + 1`, leaving the `== maturity` boundary completely untested. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
At the exact maturity block, a taker can force a seller (borrower) to take on new debt units. Because `block.timestamp == market.maturity`, this debt is immediately overdue and subject to post-maturity liquidation. If the seller's collateral is insufficient to cover the new debt at the post-maturity LIF, the shortfall is realized as bad debt and socialized across all lenders in the market. The attacker profits by buying units at the offer price while the seller absorbs debt they cannot repay, directly violating the invariant `"debt_increase_after_maturity_forbidden": true`. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
The precondition `block.timestamp == market.maturity` is a single specific block. On chains with 1-second block times (e.g., Ethereum post-merge, L2s) this is a predictable, reachable block. An attacker can monitor the chain and submit the transaction in the maturity block. The offer must be a valid sell offer (maker is borrower, `offer.buy == false`) with a ratifier that approves it; the attacker controls the taker side. No privileged access is required. The attack is repeatable across any market whose maturity falls on a future block. [6](#0-5) [7](#0-6) 

### Recommendation
Change the maturity guard from `<=` to `<` (strict less-than):

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the on-chain check with the documented invariant: "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing." [1](#0-0) [8](#0-7) 

### Proof of Concept
```solidity
// Foundry unit test
function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;

    // Setup: borrower has no existing credit; lender has funds
    collateralize(market, borrower, units);
    deal(address(loanToken), lender, units);

    // Warp to EXACT maturity (not maturity+1)
    vm.warp(market.maturity);

    // Extend offer expiry to cover this timestamp
    borrowerOffer.expiry = market.maturity;
    borrowerOffer.maxUnits = units;

    uint256 debtBefore = midnight.debtOf(id, borrower);

    // This should revert with CannotIncreaseDebtPostMaturity but does NOT
    take(units, lender, borrowerOffer);

    uint256 debtAfter = midnight.debtOf(id, borrower);

    // Assert: debt increased at maturity — invariant violated
    assertGt(debtAfter, debtBefore, "debt must not increase at maturity");

    // Multicall variant: two markets with same maturity
    bytes[] memory calls = new bytes[](2);
    calls[0] = abi.encodeCall(midnight.take, (borrowerOffer1, ratifierData, units, lender, address(0), address(0), ""));
    calls[1] = abi.encodeCall(midnight.take, (borrowerOffer2, ratifierData, units, lender, address(0), address(0), ""));
    midnight.multicall(calls); // both succeed, both increase debt at maturity
}
```

Expected assertions: `debtAfter > debtBefore` for both takes; neither reverts with `CannotIncreaseDebtPostMaturity`. After the fix (`<` instead of `<=`), both calls must revert. [9](#0-8) [3](#0-2)
