Audit Report

## Title
Fee-on-Transfer Loan Token Inflates `withdrawable` Beyond Actual Balance in `repay()` and `liquidate()` - (File: src/Midnight.sol)

## Summary
`repay()` credits `marketState[id].withdrawable` with the full `units` value before calling `safeTransferFrom`, but a fee-on-transfer loan token delivers only `units * (1 - fee)` to the contract. Because `touchMarket()` imposes no restriction on `loanToken` type and is permissionlessly callable, any user can create a market with a fee-on-transfer token, causing `withdrawable` to permanently exceed the contract's real token balance and making the market insolvent for lenders. The identical pattern exists in `liquidate()`.

## Finding Description
**`repay()` — lines 508–520:**
```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);   // line 509: full units credited
// ...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520: only units*(1-fee) arrives
```
`withdrawable` is incremented by `units` unconditionally at line 509, but `safeTransferFrom` at line 520 only delivers `units * (1 - fee)` when the loan token charges a transfer fee. The gap `units * fee` is permanently missing from the contract balance while `withdrawable` reflects the full amount.

**`liquidate()` — lines 675 and 717:**
```solidity
_marketState.withdrawable += UtilsLib.toUint128(repaidUnits);  // line 675
// ...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits); // line 717
```
The same accounting-before-transfer pattern exists in `liquidate()`, compounding the shortfall.

**Why existing checks fail:**
`touchMarket()` at lines 755–791 validates only maturity, collateral count, collateral sorting, LLTV tiers, and `maxLif` values — no check on `loanToken` type. The Certora solvency specification at `certora/specs/Solvency.spec` lines 31–33 explicitly assumes away this class of token (`// Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing...`), so the formal solvency invariant is not proven for fee-on-transfer loan tokens.

**Exploit flow:**
1. Attacker calls `touchMarket()` with a fee-on-transfer token (e.g., 1% fee) as `loanToken`.
2. Attacker acquires debt `D` via `take()`.
3. Attacker calls `repay(market, D, borrower, address(0), "")`.
4. Line 509: `withdrawable += D`.
5. Line 520: contract receives only `D * 0.99`; `D * 0.01` is burned/redirected.
6. State: `withdrawable == D`, `balanceOf(midnight) == D * 0.99`. Gap = `D * 0.01`.
7. Repeated repayments compound the gap linearly.
8. When lenders call `withdraw()`, line 499 calls `safeTransfer(loanToken, receiver, units)`. The last lenders trigger a revert because the contract balance is exhausted before all `withdrawable` credit is redeemed.

## Impact Explanation
The contract's actual loan token balance falls short of `withdrawable` by `sum(units_i * fee)` across all repayments and liquidations. Lenders who call `withdraw()` first drain the real balance; subsequent lenders receive a revert from `safeTransfer` because the contract holds fewer tokens than `withdrawable` promises. This is concrete, permanent insolvency: valid credit positions cannot be redeemed, and the shortfall grows with every `repay()` or `liquidate()` call on a fee-on-transfer market.

## Likelihood Explanation
Market creation is permissionless — any address can call `touchMarket()` with an arbitrary `loanToken`. Fee-on-transfer tokens exist in production (e.g., USDT with fee enabled, STA, PAXG). The exploit requires no special privilege: any borrower with outstanding debt can trigger it. It is repeatable on every `repay()` and `liquidate()` call, compounding the shortfall. The attacker need not be the lender — they only need to be a borrower in a market they created.

## Recommendation
1. **Measure actual received amount**: Record `balanceOf(address(this))` before and after `safeTransferFrom`, and use the delta (not `units`) to increment `withdrawable`.
2. **Alternatively, restrict `loanToken`**: Maintain an allowlist of approved loan tokens in `touchMarket()` and reject fee-on-transfer tokens at market creation time.
3. **Document the assumption**: If fee-on-transfer tokens are intentionally unsupported, enforce this at the `touchMarket()` level rather than relying only on a Certora assumption comment.

## Proof of Concept
1. Deploy a mock ERC20 with a 1% transfer fee.
2. Call `touchMarket()` with this token as `loanToken`.
3. Supply liquidity as a lender; borrow as a borrower via `take()`.
4. Call `repay()` for the full debt `D`.
5. Assert: `marketState[id].withdrawable == D` but `loanToken.balanceOf(midnight) == D * 0.99`.
6. Have the lender call `withdraw()` for `D`; observe revert due to insufficient balance.
7. Repeat step 4–6 multiple times to show linear compounding of the shortfall.
