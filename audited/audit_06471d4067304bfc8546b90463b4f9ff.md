### Title
Settlement Fee Accounting Desync via Fee-on-Transfer Loan Token in `take()` - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by `buyerAssets - sellerAssets` at line 418 **before** the actual token transfer at line 455. When `loanToken` is a fee-on-transfer token, the contract receives strictly less than `buyerAssets - sellerAssets`, creating a permanent gap between the recorded claimable fee and the actual token balance. The fee claimer cannot fully redeem the recorded settlement fee without drawing from collateral or withdrawable balances belonging to other users.

### Finding Description
**Code path:** [1](#0-0) 

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
```

This write happens unconditionally before the inbound transfer: [2](#0-1) 

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

**Root cause:** The protocol credits the full nominal `buyerAssets - sellerAssets` to `claimableSettlementFee` but relies on the ERC20 `transferFrom` delivering exactly that amount. A fee-on-transfer token deducts a fee in-flight, so the contract receives `(buyerAssets - sellerAssets) * (1 - fee_rate)` while the ledger records the full `buyerAssets - sellerAssets`.

**Attacker inputs / exploit flow:**
1. Market creator (unprivileged per audit rules) creates a market with a fee-on-transfer token as `loanToken`.
2. Maker signs an offer (ratified via `EcrecoverRatifier.isRatified`); `isRatified` performs only signature/Merkle checks — no token-type restriction. [3](#0-2) 

3. Taker calls `take()`. All precondition checks pass (tick, expiry, ratifier, health).
4. Line 418 increments `claimableSettlementFee` by the full spread.
5. Line 455 transfers the spread amount, but the token's internal fee mechanism delivers `spread - fee` to the contract.
6. `claimableSettlementFee[loanToken]` now exceeds the actual balance increase by exactly the token's transfer fee.

**Why existing checks fail:** There is no on-chain guard against fee-on-transfer `loanToken`. The Certora solvency proof explicitly assumes standard ERC20 behavior: [4](#0-3) 

The `pendingFeeReceiptZero` invariant and `tokenBalanceCorrect` strong invariant are only proven under this assumption. The `pendingFeeReceipt` ghost is cleared only when the exact expected amount arrives at the contract: [5](#0-4) 

With a fee-on-transfer token, the exact amount never arrives, so `pendingFeeReceipt` is never cleared and `tokenBalanceCorrect` is violated in the real execution environment.

### Impact Explanation
After each `take()` with a fee-on-transfer `loanToken`, `claimableSettlementFee[loanToken]` exceeds the contract's actual token balance attributable to settlement fees by the token's transfer fee amount. When the fee claimer calls `claimSettlementFee` for the full recorded amount, the transfer will either revert (if the contract has no surplus) or silently drain collateral/withdrawable balances belonging to lenders and borrowers, violating the core invariant that contract balances cover all claimable fees. [6](#0-5) 

### Likelihood Explanation
Preconditions: (1) a market exists with a fee-on-transfer `loanToken` — achievable by any market creator, who is an unprivileged actor; (2) a valid ratified offer exists in that market — achievable by any maker/signer. The exploit is triggered by any taker calling `take()` with non-zero units and a non-zero settlement fee. It is repeatable on every `take()` call and accumulates monotonically. Fee-on-transfer tokens are a well-known ERC20 variant (e.g., tokens with configurable transfer fees).

### Recommendation
Record the actual received amount rather than the nominal amount. Measure the contract's `loanToken` balance before and after the inbound transfer and use the delta to update `claimableSettlementFee`:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived;
```

Alternatively, explicitly document and enforce (via a registry or `require`) that only standard ERC20 tokens with no transfer fees are permitted as `loanToken`.

### Proof of Concept
**Foundry unit test plan:**

```solidity
// Deploy a fee-on-transfer ERC20 (e.g., 1% fee on every transferFrom).
// Create a Midnight market with this token as loanToken.
// Set a non-zero settlement fee.
// Maker signs a sell offer; taker calls take() with units > 0.
// Expected assertions:
uint256 spread = buyerAssets - sellerAssets;
uint256 tokenFee = spread * FEE_RATE / 1e18; // token's internal fee
uint256 actualReceived = spread - tokenFee;

assertEq(
    midnight.claimableSettlementFee(address(feeToken)),
    spread,                  // recorded: full spread
    "claimable fee recorded"
);
assertEq(
    feeToken.balanceOf(address(midnight)) - balanceBefore,
    actualReceived,          // actual: spread minus token fee
    "actual balance increase"
);
// Invariant violation: claimableSettlementFee > actual balance increase
assertTrue(
    midnight.claimableSettlementFee(address(feeToken)) >
    feeToken.balanceOf(address(midnight)) - balanceBefore
);
// Fee claimer cannot claim full amount without reverting or stealing from others
vm.prank(feeClaimer);
vm.expectRevert(); // or drains other users' funds
midnight.claimSettlementFee(address(feeToken), spread, receiver);
```
