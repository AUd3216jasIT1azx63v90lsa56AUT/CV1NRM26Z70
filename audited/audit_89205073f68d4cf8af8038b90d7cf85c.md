Audit Report

## Title
Authorized Delegate Can Front-Run Victim's `EcrecoverAuthorizer` Transaction by Signing a Different `Authorization` Struct with the Same Nonce - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` increments `nonce[authorization.authorizer]` unconditionally before verifying the signature, and the signer check accepts any address for which `IMidnight.isAuthorized(authorizer, signer)` returns `true`. An attacker who holds a Midnight authorization from the victim can construct a different `Authorization` struct using the victim's current nonce, sign it with their own key, and submit it ahead of the victim's pending transaction. The nonce is permanently consumed, causing the victim's in-flight transaction to revert with `InvalidNonce`, and the attacker's chosen authorization state is written instead.

## Finding Description

**Code path — `src/periphery/EcrecoverAuthorizer.sol` lines 24–48:**

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // (1) nonce consumed unconditionally

    bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
    bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
    bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
    address signer = ecrecover(digest, signature.v, signature.r, signature.s);
    require(signer != address(0), InvalidSignature());
    require(
        signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer), // (2)
        Unauthorized()
    );
    ...
    IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
}
```

**Root cause:** At (1), the nonce is incremented against `authorization.authorizer` before the signer is verified. At (2), the signer check accepts any address for which `Midnight.isAuthorized(authorizer, signer)` is `true` — not just the authorizer themselves. An attacker who is already authorized by the victim in Midnight can construct an entirely different `Authorization` struct (different `authorized` or `isAuthorized` fields) using the victim's current nonce `N`, sign it with their own key, and submit it. The EIP-712 digest is computed over the attacker's struct, so `ecrecover` returns the attacker's address, which passes check (2). The nonce is permanently consumed at (1), and the victim's signed message (which commits to a different struct) becomes invalid.

**Exploit flow:**

| Step | Actor | Action | State after |
|------|-------|--------|-------------|
| 0 | Victim | `midnight.setIsAuthorized(attacker, true, victim)` | `isAuthorized[victim][attacker] = true` |
| 1 | Victim | Signs `Authorization{authorizer=victim, authorized=operatorX, isAuthorized=true, nonce=N}`, broadcasts tx | Pending in mempool |
| 2 | Attacker | Observes mempool; constructs `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N}`, signs with attacker key | — |
| 3 | Attacker | Calls `setIsAuthorized(attackerAuth, attackerSig)` with higher gas | `nonce[victim] = N+1`; attacker's chosen auth set |
| 4 | Victim's tx | Lands: `nonce[victim]=N+1 != N` → reverts `InvalidNonce` | Victim's intended authorization never set |

**Why existing checks fail:** The deadline check (line 25) does not help — the attacker uses the same or a fresh deadline. The `InvalidNonce` check (line 26) is the mechanism being abused, not a guard against it. The `InvalidSignature` check (line 32) passes because the attacker's signature over their own struct is valid. The authorization check (lines 33–36) passes because `IMidnight(MIDNIGHT).isAuthorized(victim, attacker)` returns `true`.

## Impact Explanation
The victim's `EcrecoverAuthorizer`-based authorization flow is blocked for as long as the attacker remains authorized by the victim in Midnight. Every signed message the victim produces can be front-run and its nonce consumed. The victim cannot use `EcrecoverAuthorizer` to authorize their intended operator via off-chain signatures. Additionally, the attacker can write an arbitrary authorization state into Midnight on the victim's behalf via the same mechanism. The primary unique impact is a sustained, targeted DoS of the `EcrecoverAuthorizer` off-chain signature flow, combined with unauthorized state mutation of the victim's Midnight authorization mapping.

## Likelihood Explanation
**Preconditions:**
- The victim must have previously called `midnight.setIsAuthorized(attacker, true, victim)`. This is a normal protocol operation (authorizing an operator, bot, or contract).
- The attacker must observe the victim's pending `EcrecoverAuthorizer.setIsAuthorized` transaction in the mempool.

Both conditions are realistic on any public EVM chain with a visible mempool. The attack is repeatable: every time the victim signs a new message with nonce `N+1`, the attacker can front-run again. The victim's only escape is to revoke the attacker's Midnight authorization via a direct `midnight.setIsAuthorized(attacker, false, victim)` call (which does not go through `EcrecoverAuthorizer` and cannot be front-run in the same way).

## Recommendation
Move the nonce increment to after the signature and authorization checks, so that a failed authorization attempt does not consume the nonce:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer], InvalidNonce()); // check only, no increment

    bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
    bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
    bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
    address signer = ecrecover(digest, signature.v, signature.r, signature.s);
    require(signer != address(0), InvalidSignature());
    require(
        signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
        Unauthorized()
    );

    nonce[authorization.authorizer]++; // increment only after all checks pass
    ...
}
```

Additionally, consider whether the signer check should be restricted to `signer == authorization.authorizer` only, removing the delegate path entirely, since the delegate can already call `Midnight.setIsAuthorized` directly.

## Proof of Concept

**Minimal Foundry test plan:**

1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. Create victim and attacker EOAs.
3. Victim calls `midnight.setIsAuthorized(attacker, true, victim)`.
4. Victim signs `Authorization{authorizer=victim, authorized=operatorX, isAuthorized=true, nonce=0, deadline=far_future}` off-chain (EIP-712).
5. Attacker constructs `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=far_future}` and signs with attacker key.
6. Attacker calls `ecrecoverAuthorizer.setIsAuthorized(attackerAuth, attackerSig)` — assert it succeeds, `nonce[victim] == 1`, and `midnight.isAuthorized(victim, attacker) == true`.
7. Victim calls `ecrecoverAuthorizer.setIsAuthorized(victimAuth, victimSig)` — assert it reverts with `InvalidNonce`.
8. Assert `midnight.isAuthorized(victim, operatorX) == false` (victim's intended authorization was never set).
