### Title
Unchecked `takes[0]` array access causes opaque panic revert on empty `takes[]` input - (File: src/periphery/MidnightBundles.sol)

### Summary
`buyWithUnitsTargetAndWithdrawCollateral` unconditionally dereferences `takes[0]` at line 62 without first verifying `takes.length > 0`. Any authorized caller (including `taker == msg.sender`, which requires no special privilege) can pass an empty `takes[]` array, causing Solidity to emit a `panic(0x32)` (array out-of-bounds) instead of a named protocol error. The panic fires before any token transfer, so no funds are moved, but the call is permanently unserviceable for that input and the revert is opaque.

### Finding Description
**Execution order:**

1. Line 60 — authorization check passes (`taker == msg.sender` requires zero privilege).
2. Line 61 — `referralFeePct < WAD` passes with any valid fee.
3. **Line 62** — `address loanToken = takes[0].offer.market.loanToken;` — Solidity performs an implicit bounds check; with `takes.length == 0` this emits `panic(0x32)` and reverts.
4. Line 66 — `pullToken` is **never reached**; no tokens are transferred.

There is no `require(takes.length > 0, ...)` guard anywhere before line 62. The same unchecked pattern repeats at line 64 (`takes[0].offer.market` passed to `touchMarket`) and line 90 (`Market memory market = takes[0].offer.market`), so even if line 62 were fixed in isolation the panic would re-surface at line 64.

**Attacker-controlled inputs:** `takes` is a caller-supplied `Take[] memory` parameter with no minimum-length enforcement. Any caller for whom `taker == msg.sender` (the common self-taker case) satisfies the only pre-panic check.

**Why existing checks fail:** The two `require` statements at lines 60–61 guard authorization and fee range, not array length. Neither prevents the panic at line 62.

### Impact Explanation
Any caller who passes `takes.length == 0` receives an opaque `panic(0x32)` revert rather than a descriptive named error. Because the panic fires before `pullToken`, no funds are transferred or frozen; the concrete impact is a permanently unserviceable call path for empty-array inputs and a violation of the protocol invariant that bundle entry points must revert with descriptive errors rather than panics.

### Likelihood Explanation
Preconditions are trivially satisfiable: the caller need only be `taker == msg.sender` (no external authorization required) and supply `referralFeePct < WAD` and `takes = new Take[](0)`. The condition is repeatable on every call with an empty array and requires no special on-chain state.

### Recommendation
Add an explicit length guard immediately after the two existing `require` statements and before any `takes[0]` access:

```solidity
require(takes.length > 0, EmptyTakes());
```

Add `error EmptyTakes();` to `IMidnightBundles`. Apply the same guard to every other bundle entry point that dereferences `takes[0]` or `collateralSupplies[0]` unconditionally (`supplyCollateralAndSellWithUnitsTarget`, `buyWithAssetsTargetAndWithdrawCollateral`, `supplyCollateralAndSellWithAssetsTarget`).

### Proof of Concept
```solidity
function testBuyUnitsTargetEmptyTakesPanics() public {
    Take[] memory takes = new Take[](0); // empty array

    // taker == msg.sender satisfies the only pre-panic check
    vm.prank(lender);
    // Expect panic(0x32) — array out-of-bounds — NOT a named error
    vm.expectRevert(stdError.indexOOBError); // Foundry: bytes4(keccak256("Panic(uint256)")) with 0x32
    midnightBundles.buyWithUnitsTargetAndWithdrawCollateral(
        1,                          // targetUnits
        type(uint256).max,          // maxBuyerAssets
        lender,                     // taker == msg.sender
        _noPermit(),                // loanTokenPermit
        takes,                      // empty takes[]  ← triggers panic
        new CollateralWithdrawal[](0),
        address(0),
        0,
        address(0)
    );
    // Assertion: revert type is panic(0x32), not OutOfOffers() or any named error
}
```

Expected: `vm.expectRevert(stdError.indexOOBError)` passes, confirming the panic rather than a named revert. After the fix (`require(takes.length > 0, EmptyTakes())`), the same test should be updated to `vm.expectRevert(IMidnightBundles.EmptyTakes.selector)`. [1](#0-0) [2](#0-1) [3](#0-2)
