Audit Report

## Title
Missing Zero-Unit Guard in Bundler Loops Allows Attacker to Trigger Maker Callbacks on Fully-Consumed Offers - (File: src/periphery/MidnightBundles.sol)

## Summary
All four bundler loops in `MidnightBundles` compute `unitsToTake = min(..., consumableUnits(...))` which returns 0 for fully-consumed offers, but proceed to call `Midnight.take` with `units=0` without a guard. `Midnight.take` succeeds on a 0-unit call against a fully-consumed offer (the `consumed += 0` does not exceed `maxAssets`) and unconditionally fires `offer.callback` if set. An unprivileged attacker who includes a fully-consumed offer with a callback in the `takes` array — alongside enough valid offers to satisfy `targetUnits` — causes the maker's callback to execute with `units=0, assets=0` at a time the maker did not intend, with all state changes persisted.

## Finding Description

**Root cause — missing zero-unit guard in `MidnightBundles.sol`:**

In all four bundler functions (`buyWithUnitsTargetAndWithdrawCollateral`, `supplyCollateralAndSellWithUnitsTarget`, `buyWithAssetsTargetAndWithdrawCollateral`, `supplyCollateralAndSellWithAssetsTarget`), the pattern is:

```solidity
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // returns 0
);
try IMidnight(MIDNIGHT).take(takes[i].offer, takes[i].ratifierData, unitsToTake, ...) returns (...) {
    filledUnits += unitsToTake;   // += 0, no progress
    ...
} catch {}
```

There is no `if (unitsToTake == 0) continue;` guard. The `take` call proceeds with `units=0`.

**Why `take` succeeds with `units=0` on a fully-consumed offer:**

`ConsumableUnitsLib.consumableUnits` returns 0 when `consumed >= maxAssets` (via `zeroFloorSub`). Inside `Midnight.take`, when `units=0`: `buyerAssets = 0`, `sellerAssets = 0`, and `newConsumed = consumed[maker][group] += 0` — unchanged. `require(newConsumed <= offer.maxAssets)` passes because `consumed == maxAssets` satisfies `<=`. This is explicitly documented at `Midnight.sol` line 93: *"Midnight can call the callback of offers through a no-op take, even if those offers have consumed==max."*

**Callback fires unconditionally:**

After the consumed check, `take` assigns `buyerCallback = offer.buy ? offer.callback : takerCallback` and fires it if non-zero, with no guard on `units > 0`. The `try/catch` in the bundler only catches reverts; a successful `take` with a firing callback is not caught.

**Ratifier check — insufficient protection:**

`take` requires `isAuthorized[offer.maker][offer.ratifier]` and `IRatifier(offer.ratifier).isRatified(offer, ratifierData) == CALLBACK_SUCCESS`. For a signature-based ratifier, the signature covers the offer struct. The `consumed` mapping is not part of the offer struct, so a previously-valid signature remains valid after the offer is fully consumed. The attacker reuses the maker's offer struct and `ratifierData` (signature) from prior on-chain `Take` events.

**Attack sequence:**

1. Maker creates a sell offer with `offer.callback` set, `maxAssets = X`, signs it for a ratifier.
2. Legitimate takers fill the offer until `consumed[maker][group] == X`.
3. Attacker constructs a `takes` array: `[fullyConsumedOffer, validOffer1, ...]` where the valid offers together satisfy `targetUnits`.
4. Attacker calls `buyWithUnitsTargetAndWithdrawCollateral(targetUnits, ...)`.
5. Iteration 0: `consumableUnits` returns 0 → `unitsToTake=0` → `take(fullyConsumedOffer, sig, 0, ...)` succeeds → `offer.callback.onSell(units=0, sellerAssets=0, ...)` fires → `filledUnits += 0`.
6. Iterations 1+: valid offers fill `targetUnits`.
7. `require(filledUnits == targetUnits)` passes. Transaction succeeds. Callback state changes are permanent.

## Impact Explanation

The maker's `onSell` or `onBuy` callback executes with `units=0` and `assets=0` at a time the maker did not intend (offer fully consumed). Any state-mutating logic inside the callback — such as calling `repay`, `withdrawCollateral`, or `setIsAuthorized` on Midnight — executes and persists. For example, a callback that withdraws collateral upon a sell event would drain the maker's collateral position with zero assets received in return. The severity is high for makers whose callbacks perform non-trivial state mutations without a `units > 0` guard.

## Likelihood Explanation

Preconditions are low-friction: any offer with a callback that has been fully consumed is eligible. The attacker only needs the maker's previously-used offer struct and `ratifierData` (publicly observable on-chain from prior `Take` events), plus enough valid offers to satisfy `targetUnits`. No privileged access is required. The attack is repeatable as long as the offer has not expired.

## Recommendation

Add a zero-unit guard at the top of each bundler loop body, before the `try` call:

```solidity
if (unitsToTake == 0) continue;
```

This prevents `take` from being called with `units=0`, eliminating the spurious callback invocation on fully-consumed offers. Apply this fix to all four bundler functions.

## Proof of Concept

1. Deploy `Midnight` and `MidnightBundles`.
2. Create a market and a sell offer with `maxAssets = X` and `offer.callback` set to a contract that records invocations.
3. Fill the offer completely via direct `take` calls until `consumed[maker][group] == X`.
4. As an attacker, call `buyWithUnitsTargetAndWithdrawCollateral` with `takes = [fullyConsumedOffer, validOffer]` where `validOffer` alone satisfies `targetUnits`.
5. Observe that the callback contract records an invocation with `units=0, sellerAssets=0`, confirming the spurious callback execution.
6. Confirm the transaction succeeds and `filledUnits == targetUnits`.
