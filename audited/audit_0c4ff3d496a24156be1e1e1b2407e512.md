Audit Report

## Title
Assets-mode buy offer with `offerPrice < WAD` allows unbounded credit inflation past `maxAssets` cap — (`src/Midnight.sol`)

## Summary
When a buy offer uses `maxAssets` mode and `offerPrice < WAD`, calling `take()` with `units` small enough that `units * buyerPrice < WAD` causes `buyerAssets` to round to zero via `mulDivDown`. The consumed-cap check increments by zero and passes even when the offer is fully consumed. Position state (`credit`, `debt`, `totalUnits`) is updated using `units` directly while zero tokens are transferred, allowing an attacker who controls both the maker and taker addresses to inflate lender credit without depositing any loan tokens and subsequently drain `withdrawable` tokens deposited by other lenders.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

```solidity
// line 363
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;
// line 364
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...;

// lines 367–369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

When `offer.buy = true` and `units * buyerPrice < WAD`, `mulDivDown` returns 0. The consumed accumulator is incremented by 0, so `newConsumed == consumed` (already at `maxAssets`), and `require(newConsumed <= offer.maxAssets)` passes trivially.

**Position accounting uses `units` directly — lines 382–417:**

```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt); // = units if no debt
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt  += UtilsLib.toUint128(sellerDebtIncrease);
_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease;
```

**Token transfer is zero — line 455:**

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // 0 - 0 = 0
```

**Protocol acknowledges the gap — line 94:**

```
/// @dev It is possible to give units to a fully consumed assets-based buy offer with price < 1.
```

This comment confirms the behavior is known but no guard exists.

**Exploit flow:**
1. Attacker controls address A (maker) and address B (taker, authorized via `isAuthorized`).
2. A creates a buy offer: `maxAssets = 1`, `tick` chosen so `offerPrice < WAD` (e.g. `MAX_TICK - 16`).
3. B calls `take()` once legitimately to set `consumed[A][group] = 1 = maxAssets`.
4. B calls `take(offer, ..., units=1, B, ...)` again: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; consumed check passes; A's `credit += 1`, B's `debt += 1`, zero tokens transferred.
5. Step 4 is repeated in a loop or multicall. A's `credit` and `totalUnits` grow without bound; no tokens deposited.
6. A calls `withdraw()`: `_position.credit -= N` succeeds (inflated), `_marketState.withdrawable -= N` succeeds if other lenders have deposited tokens, and `safeTransfer(loanToken, receiver, N)` drains their funds.

**Why existing checks fail:**
- The `require(newConsumed <= offer.maxAssets)` guard only prevents `consumed` from exceeding the cap; it does not guard against `units > 0` when `buyerAssets == 0`.
- There is no `require(buyerAssets > 0)` or `require(units == 0 || buyerAssets > 0)` check anywhere in `take()`.
- `SelfTake()` is bypassed by using two attacker-controlled addresses.

## Impact Explanation

The attacker inflates their `credit` by an arbitrary amount without depositing loan tokens. `withdraw()` at line 499 transfers `units` of `loanToken` per credit unit redeemed. `totalUnits` grows without token backing, breaking the core invariant that contract balances cover all credit redemptions. Any `withdrawable` tokens deposited by legitimate lenders (via `supply()` or borrower repayments) can be fully drained. This constitutes direct theft of user funds — a critical in-scope impact.

## Likelihood Explanation

All preconditions are attacker-controlled: the attacker deploys two addresses, creates the offer themselves, and sets `consumed == maxAssets` with a single legitimate fill. No victim cooperation is required. The tick range where `offerPrice < WAD` covers a large portion of valid ticks. The attack is repeatable in a single transaction via a loop, with no economic cost beyond gas. The protocol's own comment at line 94 documents the behavior, confirming it is a reachable, known edge case without a guard.

## Recommendation

Add a guard in `take()` immediately after computing `buyerAssets`/`sellerAssets` to reject zero-asset trades when `maxAssets` mode is active:

```solidity
if (offer.maxAssets > 0) {
    require(offer.buy ? buyerAssets > 0 : sellerAssets > 0, ZeroAssets());
}
```

Alternatively, enforce `require(units == 0 || buyerAssets > 0)` unconditionally when `offer.buy = true` and `maxAssets > 0`. This closes the rounding gap without changing behavior for valid trades.

## Proof of Concept

**Minimal manual steps:**
1. Deploy a market with a loan token and `tickSpacing = 1`.
2. Legitimate lender L calls `supply(market, 1000, L, L)` — `withdrawable = 1000`.
3. Attacker deploys maker address A and taker address B; A authorizes B.
4. A creates offer: `buy=true`, `maxAssets=1`, `tick = MAX_TICK - 16` (so `offerPrice < WAD`), `group = G`.
5. B calls `take(offer, ..., units=1, B, ...)` — legitimate fill; `consumed[A][G] = 1 = maxAssets`.
6. B calls `take(offer, ..., units=1, B, ...)` again — `buyerAssets = 0`, check passes; `A.credit = 1`, `B.debt = 1`, 0 tokens transferred.
7. Repeat step 6 one thousand times (loop/multicall): `A.credit = 1001`.
8. A calls `withdraw(market, 1000, A, A)` — transfers 1000 loan tokens from the contract to A, draining L's deposit.

**Invariant fuzz test:** Assert `sum(position[id][u].credit for all u) <= totalSupplied + totalRepaid - totalWithdrawn` after any sequence of `take()` calls with `buyerAssets == 0`. The invariant will be violated after step 6 above. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
