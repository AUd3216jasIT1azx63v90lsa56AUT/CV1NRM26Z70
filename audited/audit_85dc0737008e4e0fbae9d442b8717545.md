### Title
Liquidator gate bypass via intermediary contract — (`src/Midnight.sol`)

### Summary
`Midnight.liquidate()` checks `canLiquidate(msg.sender)` where `msg.sender` is the immediate caller. An attacker blocked by the `liquidatorGate` can deploy or use any unblocked intermediary contract — including a flash loan callback — to call `liquidate()` on their behalf, because the gate never sees the original attacker's address. The flash loan path is one concrete route, but any unblocked contract calling `liquidate()` directly achieves the same bypass.

### Finding Description
**Root cause:** `liquidate()` at [1](#0-0)  checks `canLiquidate(msg.sender)`, where `msg.sender` is the immediate caller. There is no mechanism to check the ultimate beneficiary or the transaction originator.

**Flash loan path:**
1. Attacker (blocked) calls `Midnight.flashLoan([], [], callbackContract, data)` — empty token arrays are valid; the loop at lines 742–744 simply does not execute, but `onFlashLoan` is still invoked at line 746.
2. `Midnight` calls `callbackContract.onFlashLoan(attacker, [], [], data)` — `msg.sender` inside the callback is `Midnight`.
3. Inside `onFlashLoan`, `callbackContract` calls `Midnight.liquidate(..., receiver=attacker, ...)`.
4. Gate checks `canLiquidate(callbackContract)` — `callbackContract` is not in the blocklist → returns `true`.
5. Liquidation succeeds; seized collateral is sent to `receiver = attacker`.

**Simpler equivalent path (no flash loan needed):** The attacker deploys any new contract not in the blocklist and calls `liquidate()` from it directly. The flash loan is not a prerequisite — it is just one trigger mechanism.

The `flashLoan` function is at [2](#0-1) , and the gate check is at [1](#0-0) . The `ILiquidatorGate` interface only exposes `canLiquidate(address account)` with no context about the call origin. [3](#0-2) 

### Impact Explanation
A blocked liquidator seizes collateral from a liquidatable borrower by routing the call through any unblocked intermediary contract. The gate's intended access control is fully bypassed. Seized collateral is transferred to an attacker-controlled `receiver` address.

### Likelihood Explanation
Preconditions are low-friction: the attacker needs only to deploy a fresh contract (zero cost beyond gas) that is not yet in the blocklist, and a liquidatable borrower must exist. The attack is repeatable — each new contract address is unblocked by default. Any blocklist-based gate is trivially circumvented this way.

### Recommendation
The gate should check the ultimate beneficiary, not the immediate `msg.sender`. Two concrete options:

1. **Check `tx.origin`** in `canLiquidate` — simple but breaks contract-based liquidators.
2. **Pass the intended liquidator explicitly** and gate on that address: add a `liquidator` parameter to `liquidate()`, require `liquidator == msg.sender || isAuthorized[liquidator][msg.sender]`, and check `canLiquidate(liquidator)` instead of `canLiquidate(msg.sender)`. This mirrors how `take()` handles the `taker` parameter at [4](#0-3) .
