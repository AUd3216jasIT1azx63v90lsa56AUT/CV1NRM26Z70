### Title
Cross-chain signature replay via shared chainId allows unauthorized take on chain B - (File: src/ratifiers/EcrecoverRatifier.sol)

### Summary
`EcrecoverRatifier.isRatified` computes the EIP-712 domain separator dynamically as `keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)))`. When two chains share the same chain ID and the ratifier is deployed at the same address on both (e.g., via CREATE2 deterministic deployment), the domain separator is byte-for-byte identical. A taker can replay a signature produced on chain A on chain B, bypassing the intended chain-binding of the offer signature. If the maker has authorized the ratifier on chain B's Midnight contract, the take executes successfully on chain B with an offer the maker never intended for that chain.

### Finding Description
**Root cause — `EcrecoverRatifier.sol` line 40:**
```solidity
bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
``` [1](#0-0) 

The domain separator binds only to `block.chainid` and `address(this)`. If two chains share the same chain ID (a real-world scenario: some L2 testnets historically share chain IDs with their mainnet counterparts), and the `EcrecoverRatifier` is deployed at the same address on both chains, the domain separator is identical.

The `Offer` struct contains no chain-specific field: [2](#0-1) 

`HashLib.hashOffer` hashes the full offer without any chain-specific data: [3](#0-2) 

Therefore the full digest `keccak256(bytes.concat("\x19\x01", domainSeparator, structHash))` is identical on both chains, and a signature valid on chain A is cryptographically valid on chain B.

**Exploit flow (maker is seller, i.e., `offer.buy == false`):**

1. Maker uses the protocol on both chain A and chain B, authorizing the `EcrecoverRatifier` on both chains' Midnight contracts via `setIsAuthorized`.
2. Maker signs a sell offer on chain A (intending it only for chain A).
3. Taker observes the offer and signature on chain A.
4. Taker calls `Midnight.take` on chain B with the identical offer struct and signature.
5. `Midnight.take` line 355 checks `isAuthorized[offer.maker][offer.ratifier]` on chain B — passes, because the maker authorized the ratifier on chain B. [4](#0-3) 
6. `EcrecoverRatifier.isRatified` is called on chain B. The domain separator is identical to chain A's. `ecrecover` recovers the maker's address. The signature check passes. [5](#0-4) 
7. The take executes on chain B: debt is created for the maker (seller) on chain B in a market the maker never intended to participate in with that offer.
8. The `SellerIsLiquidatable` check at line 476 passes if the maker has collateral on chain B (which they do, since they are actively using the protocol there). [6](#0-5) 
9. The maker's position on chain B now carries debt from an offer they never intended for chain B. If oracle prices on chain B diverge, the position becomes undercollateralized and bad debt is realized.

**Why existing checks fail:**
- `isAuthorized[offer.maker][offer.ratifier]`: Not chain-specific; passes if the maker authorized the ratifier on chain B (which they did intentionally as a protocol user on chain B).
- `block.chainid` in the domain separator: Identical on both chains by precondition.
- `address(this)` in the domain separator: Identical on both chains by deterministic deployment.
- `offer.expiry` / `offer.start`: Timestamps, not chain-specific.
- `consumed[offer.maker][offer.group]`: Chain-specific storage, but only limits the total amount taken on chain B — it does not reject the replay.

**Note on `INITIAL_CHAIN_ID`:** `Midnight` captures `INITIAL_CHAIN_ID` at construction for market IDs to survive hard forks. However, `EcrecoverRatifier` uses `block.chainid` dynamically. When two chains share the same chain ID from genesis (not a hard fork), both `INITIAL_CHAIN_ID` and `block.chainid` are identical on both chains, so this design distinction provides no protection here. ### Impact Explanation
An unprivileged taker replays a maker's chain A signature on chain B, executing a take the maker never authorized for chain B. The maker (as seller) accrues debt on chain B in a market they did not intend to enter with that offer. If oracle prices on chain B differ from chain A (or the maker's collateral on chain B is insufficient relative to the replayed debt), the position becomes undercollateralized and bad debt is socialized among chain B lenders, violating the invariant that signatures bind the right user/market/action/amount.

### Likelihood Explanation
**Preconditions:**
1. Two chains share the same chain ID — a documented real-world occurrence (e.g., some L2 testnets vs. mainnets, or misconfigured chains).
2. `EcrecoverRatifier` deployed at the same address on both chains — standard with CREATE2 deterministic deployment, which is the norm for protocol deployments.
3. `Midnight` deployed at the same address on both chains — same reasoning.
4. Maker has authorized the `EcrecoverRatifier` on chain B's Midnight contract — required; the maker must be an active user on chain B. This is the primary limiting factor.

Given that Morpho-family protocols are typically deployed deterministically across chains, conditions 2 and 3 are near-certain. Condition 4 is the binding constraint: the attack is limited to makers who actively use the protocol on both chains. The replay is repeatable for any offer the maker signs on chain A, up to the `maxUnits`/`maxAssets` cap tracked in chain B's `consumed` mapping.

### Recommendation
Include the `MIDNIGHT` contract address in the domain separator so that even if two chains share the same chain ID and the `EcrecoverRatifier` is deployed at the same address, the domain separator differs if the Midnight deployments differ. Update `EIP712_DOMAIN_TYPEHASH` accordingly:

```solidity
// keccak256("EIP712Domain(uint256 chainId,address verifyingContract,address midnight)")
bytes32 constant EIP712_DOMAIN_TYPEHASH = ...;

bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this), MIDNIGHT));
```

Alternatively, add a `uint256 chainId` field to the `Offer` struct and verify it against `block.chainid` inside `isRatified`, so the offer itself is chain-bound independently of the domain separator.

### Proof of Concept
```solidity
// Foundry stateful test plan
// Setup: two fork environments configured with the same chainId (e.g., vm.chainId(999) on both)
// Contracts deployed at identical addresses via CREATE2 on both forks

function testCrossChainReplay() public {
    uint256 forkA = vm.createFork("rpcA"); // chainId = 999
    uint256 forkB = vm.createFork("rpcB"); // chainId = 999 (collision)

    // --- Chain A setup ---
    vm.selectFork(forkA);
    // Deploy Midnight + EcrecoverRatifier at deterministic addresses
    // Maker authorizes ratifier on chain A
    vm.prank(maker);
    midnight.setIsAuthorized(address(ecrecoverRatifier), true, maker);
    // Maker supplies collateral on chain A
    // Maker signs a sell offer on chain A
    Offer memory offer = makeSellOffer(maker);
    bytes32 root = HashLib.hashOffer(offer);
    Signature memory sig = signOffer(root, makerKey, address(ecrecoverRatifier));
    bytes memory ratifierData = abi.encode(sig, root, 0, new bytes32[](0));

    // --- Chain B setup ---
    vm.selectFork(forkB);
    // Same contracts at same addresses (deterministic deployment)
    // Maker has also authorized ratifier on chain B (active user on both chains)
    vm.prank(maker);
    midnight.setIsAuthorized(address(ecrecoverRatifier), true, maker);
    // Maker has collateral on chain B
    supplyCollateralForMaker();

    // --- Replay attack ---
    vm.prank(taker);
    // ASSERTION: this should revert but does NOT
    midnight.take(offer, ratifierData, units, taker, taker, address(0), hex"");

    // Assert unauthorized debt was created for maker on chain B
    assertGt(midnight.debtOf(toId(offer.market), maker), 0);
    // Assert maker's position on chain B is now at risk of bad debt
    // (oracle price manipulation or natural price movement can make it undercollateralized)
}
