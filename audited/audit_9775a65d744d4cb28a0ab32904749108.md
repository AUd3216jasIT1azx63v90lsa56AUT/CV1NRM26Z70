Audit Report

## Title
`buyWithAssetsTargetAndWithdrawCollateral` reverts for sell offers where `offerPrice + settlementFee > WAD` due to uncaught revert in `buyerAssetsToUnits` - (File: src/periphery/TakeAmountsLib.sol)

## Summary
`TakeAmountsLib.buyerAssetsToUnits` enforces `require(buyerPrice <= WAD)` at line 28, but for sell offers `buyerPrice = offerPrice + settlementFee`, which exceeds `WAD` at high ticks with any non-zero settlement fee. This call is placed outside the `try/catch` block in `MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` (lines 208–214), so the revert propagates and aborts the entire transaction. `IMidnight.take()` has no equivalent guard and would succeed on the same offer.

## Finding Description
**Root cause — `TakeAmountsLib.sol` line 28:**
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;  // line 26
uint256 buyerPrice  = sellerPrice + settlementFee;                           // line 27
require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());                   // line 28
```
For sell offers (`offer.buy = false`), `sellerPrice = offerPrice` and `buyerPrice = offerPrice + settlementFee`. At `tick = MAX_TICK = 5820`, `tickToPrice(5820)` evaluates to `WAD` (the `wExp(-14.5e18)` term is 0 in integer arithmetic, so `1e36 / 1e18 = WAD` after rounding to `PRICE_ROUNDING_STEP = 1e12`). Any non-zero `settlementFee` then makes `buyerPrice > WAD`, triggering the revert.

**Structural gap — `MidnightBundles.sol` lines 208–221:**
```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.buyerAssetsToUnits(...),   // outside try — revert propagates
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(...)
);
try IMidnight(MIDNIGHT).take(...) { ... } catch {}  // only take() is guarded
```
The NatSpec at line 175 explicitly documents this: *"Reverts if TakeAmountsLib or ConsumableUnitsLib reverts."*

**Absent guard in `Midnight.sol` lines 361–363:**
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
```
No `require(buyerPrice <= WAD)` exists. Core `take()` proceeds unconditionally with `buyerPrice > WAD`.

**Exploit flow:**
1. Maker creates a sell offer (`offer.buy = false`) at `tick = MAX_TICK = 5820` in any market with `settlementFeeCbp6 >= 1` (i.e., `settlementFee >= 1e12`).
2. Taker calls `buyWithAssetsTargetAndWithdrawCollateral` with this offer in `takes[]`.
3. Loop reaches this offer; `buyerAssetsToUnits` is called with `buyerPrice = WAD + settlementFee > WAD`.
4. `require(buyerPrice <= WAD)` reverts with `PriceGreaterThanOne()`.
5. Revert escapes the `min(...)` argument evaluation, bypasses the `try/catch`, and aborts the entire transaction including all other valid offers.
6. Calling `IMidnight.take()` directly on the same offer succeeds.

## Impact Explanation
`buyWithAssetsTargetAndWithdrawCollateral` is permanently unusable for any sell offer where `tickToPrice(offer.tick) + settlementFee(id, ttm) > WAD`. At `tick = MAX_TICK`, `tickToPrice` returns `WAD`, so the condition is triggered by any non-zero settlement fee. Any taker transaction that includes even one such offer in its `takes[]` array reverts entirely, including all other valid offers in the same call. Users are forced to call `IMidnight.take()` directly or use `buyWithUnitsTargetAndWithdrawCollateral`, defeating the purpose of the assets-target bundle.

## Likelihood Explanation
The condition is deterministically reachable without any special privileges, oracle manipulation, or admin action. A maker places a sell offer at a high tick — a normal protocol action — in any market with a non-zero settlement fee. `MAX_SETTLEMENT_FEE_360_DAYS = 0.005e18 = 5e15` dwarfs the sub-`1e12` gap between `tickToPrice(MAX_TICK)` and `WAD`. The condition holds for any `settlementFeeCbp6 >= 1`. No victim mistake is required; the taker's transaction simply reverts.

## Recommendation
Move `TakeAmountsLib.buyerAssetsToUnits(...)` inside the `try/catch` block, or add a pre-check that skips the offer when `buyerPrice > WAD` rather than reverting. Alternatively, remove the `require(buyerPrice <= WAD)` from `buyerAssetsToUnits` for sell offers and handle the `> WAD` case by returning 0 units (indicating the offer is untakeable at the target assets), consistent with how `ConsumableUnitsLib.consumableUnits` handles edge cases.

## Proof of Concept
1. Deploy Midnight with a market having `settlementFeeCbp6 = 1` (minimum non-zero fee).
2. Maker creates a sell offer at `tick = 5820` (`MAX_TICK`).
3. Call `buyWithAssetsTargetAndWithdrawCollateral` with `takes = [{offer: sellOfferAtMaxTick, units: 1, ...}]`.
4. Observe revert with `PriceGreaterThanOne()`.
5. Call `IMidnight.take(sellOfferAtMaxTick, ..., 1, taker, ...)` directly — observe success.
6. Fuzz variant: for all `tick` values where `tickToPrice(tick) + settlementFee(id, ttm) > WAD`, assert that `buyWithAssetsTargetAndWithdrawCollateral` reverts while `IMidnight.take()` succeeds.
