### Title
Debt Increase Permitted at Exact Maturity Timestamp via `<=` Boundary, Violating Core Invariant — (File: src/Midnight.sol)

---

### Summary

The `take` function guards against post-maturity debt increases using `block.timestamp <= offer.market.maturity`. This condition passes at exactly `block.timestamp == market.maturity`, allowing a taker to increase a seller's debt at the maturity block. The protocol's own stated core invariant explicitly forbids this: *"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."* This is the direct analog of the GovernorAlpha `>=` vs `>` boundary mismatch.

---

### Finding Description

In `take`, line 391:

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

The condition `block.timestamp <= offer.market.maturity` evaluates to `true` when `block.timestamp == market.maturity`. A taker can therefore submit a transaction in the maturity block that takes a sell offer, increasing the seller's debt.

The `live_context.json` core invariants section states:

> "maturity boundary must not allow debt increase by rounding, **timestamp equality**, or callback sequencing"

The implementation uses `<=` where the invariant requires `<`. This is the same class of off-by-one boundary mismatch described in the external GovernorAlpha report.

The asymmetry with the post-maturity liquidation check compounds the issue. The `liquidate` function uses a strict `>`:

```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
```

So at exactly `block.timestamp == market.maturity`:
- Debt **can** increase (via `take`, because `<=` passes).
- Post-maturity liquidation is **not yet available** (because `>` fails).

This creates a one-block window where new overdue debt exists but cannot be liquidated in post-maturity mode.

---

### Impact Explanation

1. A taker takes a sell offer at exactly `block.timestamp == market.maturity`, increasing the seller's debt.
2. In the next block (`block.timestamp > market.maturity`), the seller's debt is overdue and post-maturity liquidation becomes available.
3. The protocol invariant `"overdue_debt_after_maturity_is_liquidatable_even_if_healthy": true` means the seller's position is now liquidatable in post-maturity mode **even if it is fully collateralized and healthy**.
4. The post-maturity LIF starts at 1 and ramps to `maxLif` over 15 minutes (`TIME_TO_MAX_LIF`), so the liquidation incentive is initially minimal — but the position is still exposed to liquidation.

A seller who published a sell offer with `expiry >= maturity` (a natural default for offers intended to remain valid through maturity) can have their debt forcibly increased at the maturity block, immediately making them subject to post-maturity liquidation they did not anticipate.

---

### Likelihood Explanation

- Any unprivileged taker can execute this: no special role or key is required.
- The attacker only needs a valid sell offer with `expiry >= maturity` (the maker's own signed offer).
- On Ethereum (12-second blocks), the maturity block is fully predictable. The attacker monitors the chain and submits the `take` call in the maturity block.
- Offers that do not explicitly set `expiry < maturity` are vulnerable. This is a realistic default for makers who intend their offers to be valid up to and including maturity.

---

### Recommendation

Change `<=` to `<` in the maturity debt-increase guard:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the implementation with the stated core invariant and makes the maturity boundary consistent: debt cannot increase at or after maturity.

---

### Proof of Concept

1. Deploy a market with `maturity = T`.
2. Maker signs a sell offer with `expiry = T` (or any value `>= T`).
3. Warp to `block.timestamp = T`.
4. Taker calls `take(offer, ..., units > 0)` where `offer.buy = false` (sell offer), causing `sellerDebtIncrease > 0`.
5. The `require(block.timestamp <= offer.market.maturity ...)` check passes because `T <= T`.
6. Seller's `position[id][seller].debt` increases.
7. Warp to `block.timestamp = T + 1`.
8. Call `liquidate(..., postMaturityMode = true, borrower = seller)`.
9. The check `block.timestamp > market.maturity` passes (`T+1 > T`).
10. The seller is liquidated in post-maturity mode despite having been healthy before step 4.
