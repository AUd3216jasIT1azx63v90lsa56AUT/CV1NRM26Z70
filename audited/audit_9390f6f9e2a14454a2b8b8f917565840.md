### Title
Callback-Driven Bad Debt Over-Realization via `supplyCollateral` During `onLiquidate` - (`src/Midnight.sol`)

### Summary

In `liquidate`, bad debt is computed from a snapshot of `_collateralBitmap` and fully committed to state before the `onLiquidate` callback fires. Because `supplyCollateral` has no reentrancy guard and `liquidationLocked` is never set during a liquidate callback, a borrower-controlled callback can call `supplyCollateral` on behalf of the borrower after bad debt is already realized. This causes lenders to absorb the full original debt as bad debt even when the borrower possessed sufficient collateral to cover it, violating the invariant that bad debt equals the portion of debt not covered by any collateral.

### Finding Description

**Code path:**

`liquidate` reads a local snapshot of the bitmap at line 606:

```solidity
uint128 _collateralBitmap = _position.collateralBitmap;   // line 606
while (_collateralBitmap != 0) { ... }                     // lines 607-618
``` [1](#0-0) 

When `_position.collateralBitmap == 0`, the loop never executes, leaving `maxDebt = 0` and `badDebt = originalDebt`. The liquidatability check `originalDebt > maxDebt` passes trivially, and bad debt is fully committed to state before any external call:

```solidity
if (badDebt > 0) {
    _position.debt -= uint128(badDebt);          // line 628
    _marketState.lossFactor = ...;               // line 631
    _marketState.totalUnits -= ...;              // line 634
    ...
}
``` [2](#0-1) 

Only after this does the callback fire:

```solidity
if (callback != address(0)) {
    require(
        ILiquidateCallback(callback).onLiquidate(...) == CALLBACK_SUCCESS,
        ...
    );
}
``` [3](#0-2) 

`supplyCollateral` has no reentrancy guard and no lock that would block it during a liquidate callback:

```solidity
function supplyCollateral(Market memory market, uint256 collateralIndex, uint256 assets, address onBehalf)
    external
{
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    ...
}
``` [4](#0-3) 

`liquidationLocked` is only set during `take` callbacks (line 444) and is never set during `liquidate` callbacks, so it provides no protection here. [5](#0-4) 

**Attacker-controlled inputs and exploit flow:**

1. **Setup:** Borrower has `debt=D > 0`, `collateralBitmap=0` (reachable when a prior liquidation seized all collateral but left residual debt). Borrower holds X collateral tokens externally and calls `midnight.setIsAuthorized(callbackContract, true, borrower)`.

2. **Trigger:** Borrower calls `liquidate(market, 0, 0, 0, borrower, false, borrower, callbackContract, data)`.

3. **Bad debt realized:** Bitmap snapshot is 0 → `badDebt = D`, `maxDebt = 0`. Check `D > 0` passes. State committed: `_position.debt = 0`, `totalUnits -= D`, `lossFactor` updated.

4. **Callback:** `callbackContract.onLiquidate(...)` calls `supplyCollateral(market, idx, X, borrower)`. Authorization passes because `isAuthorized[borrower][callbackContract]` is true. Borrower's position now has `debt=0, collateral=X, bitmap≠0`.

5. **Recovery:** Borrower calls `withdrawCollateral`. Since `debt=0`, `isHealthy` returns true unconditionally, and X collateral tokens are returned.

**Net result:** Borrower's debt D is wiped at zero net cost (X collateral paid and recovered). Lenders absorb D units of bad debt.

**Why checks fail:** The `liquidationLocked` transient-storage mechanism is only written in `take` (line 444), not in `liquidate`. There is no equivalent lock set during the `onLiquidate` callback window. `supplyCollateral` has no guard against being called mid-liquidation. [6](#0-5) 

### Impact Explanation

Lenders' credit is reduced by the full original debt D via `lossFactor` socialization, even when the borrower possessed collateral sufficient to cover some or all of that debt. The invariant `badDebt = max(0, debt - Σ(collateral_i · price_i / maxLif_i))` is violated because the final collateral state (post-callback) is not reflected in the realized bad debt. The excess loss per attack is `min(D, X·price/maxLif)` units, which can equal D if the borrower supplies enough collateral to fully cover the debt.

### Likelihood Explanation

**Preconditions:**
- `collateralBitmap=0` with `debt>0` arises naturally after a partial liquidation that seizes all collateral without repaying all debt (a documented bad-debt scenario).
- The borrower must hold collateral tokens externally (purchasable on open market) and must have pre-authorized the callback contract (a single permissionless call).
- The borrower calls `liquidate` on themselves — permissionless, no gate required if `liquidatorGate == address(0)`.

The attack is repeatable across any market where the borrower can reach the zero-bitmap/nonzero-debt state, and requires no privileged access beyond the borrower's own authorization mapping.

### Recommendation

Set a transient-storage lock for the borrower at the start of `liquidate` (analogous to `liquidationLocked` in `take`) and check it in `supplyCollateral` to revert if the borrower's position is currently mid-liquidation. Alternatively, re-read `_position.collateralBitmap` and recompute `badDebt` after the callback returns, before finalizing state — though this is more invasive. The minimal fix is to extend the existing `liquidationLocked` mechanism to cover the `onLiquidate` callback window and enforce it in `supplyCollateral`.
