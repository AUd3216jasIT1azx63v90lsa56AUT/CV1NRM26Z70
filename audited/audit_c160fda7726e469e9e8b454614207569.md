### Title
Sponsor can redirect CORRUPTED sweep to attacker-controlled address by changing `recoveryAddress` after stakers deposit - (File: src/ConfidencePool.sol)

### Summary
`setRecoveryAddress` has no lifecycle guard preventing it from being called after stakers have deposited or after `flagOutcome` has been called. A malicious sponsor can front-run `claimCorrupted` by swapping `recoveryAddress` to their own address, stealing all staker principal and bonus under a bad-faith CORRUPTED resolution.

### Finding Description
Every other mutable sponsor parameter in `ConfidencePool` is protected by a one-way latch:

- `setExpiry` is blocked by `expiryLocked`, which is set on the first `stake()` call.
- `setPoolScope` is blocked by `scopeLocked`, which is set on the first `_observePoolState()` call past pre-attack staging.

`setRecoveryAddress` has no analogous guard:

```solidity
// src/ConfidencePool.sol:611-618
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    address oldRecoveryAddress = recoveryAddress;
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
```

It can be called at any time — including after stakers have deposited and after `flagOutcome` has been called — as long as `claimsStarted` is still false. `claimsStarted` is only set to `true` inside `claimCorrupted` itself (line 422), so the window between `flagOutcome` and the first `claimCorrupted` call is fully exploitable.

Under bad-faith CORRUPTED resolution, `claimCorrupted` sweeps the entire pool balance to `recoveryAddress`:

```solidity
// src/ConfidencePool.sol:423
stakeToken.safeTransfer(recoveryAddress, toSweep);
```

`toSweep` is the live `balanceOf(address(this))` — the full pool including all staker principal and bonus.

### Impact Explanation
A malicious sponsor can steal 100% of staker principal and bonus. Under bad-faith CORRUPTED, stakers already lose their principal by design (it goes to `recoveryAddress`). The defect allows the sponsor to redirect that sweep from the legitimate recovery address to themselves, converting a protocol-intended loss into a direct theft. The entire pool balance is at risk.

### Likelihood Explanation
The sponsor is not a trusted actor (only the moderator/DAO and factory owner are trusted per the contest scope). The attack requires:
1. The agreement to reach a terminal `CORRUPTED` registry state (a real-world event, not manufactured by the sponsor).
2. The moderator to honestly flag bad-faith CORRUPTED (the moderator is trusted and acts correctly).
3. The sponsor to call `setRecoveryAddress` in the block window between `flagOutcome` and `claimCorrupted`.

Step 3 is trivially achievable: the sponsor monitors the mempool for `flagOutcome`, then front-runs or races `claimCorrupted` with `setRecoveryAddress`. No special capability is required beyond being the pool owner.

### Recommendation
Add a stake-presence latch to `setRecoveryAddress`, mirroring the `expiryLocked` pattern used by `setExpiry`. Once any staker has deposited, `recoveryAddress` should be immutable:

```solidity
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (expiryLocked) revert RecoveryAddressLocked(); // expiryLocked == first stake received
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    ...
}
```

Alternatively, introduce a dedicated `recoveryAddressLocked` boolean set on the first `stake()` call, for clarity. Either way, the guard must be in place before any staker capital enters the pool.

### Proof of Concept
1. Sponsor deploys pool with `recoveryAddress = 0xLegitimate`.
2. Stakers deposit; `totalEligibleStake` grows; `expiryLocked` flips true.
3. The underlying agreement is breached; registry transitions to `CORRUPTED`.
4. Trusted moderator calls `flagOutcome(CORRUPTED, false, address(0))` (bad-faith). `claimsStarted` remains `false`.
5. Sponsor calls `setRecoveryAddress(0xSponsor)` — succeeds with no revert because there is no guard.
6. Anyone calls `claimCorrupted()`. The function reads `stakeToken.balanceOf(address(this))` (the full pool) and transfers it to `recoveryAddress`, which is now `0xSponsor`.
7. All staker principal and bonus is in the sponsor's wallet. Stakers receive nothing. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
