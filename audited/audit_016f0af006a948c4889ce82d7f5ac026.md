### Title
Buy-offer `maxAssets` cap bypassed via zero-`buyerAssets` rounding when `offerPrice < WAD` - (`src/Midnight.sol`)

### Summary

When a buy offer has `maxAssets > 0` and `offerPrice < WAD` (tick > `MAX_TICK/2`), calling `take()` with `units = 1` computes `buyerAssets = mulDivDown(1, offerPrice, WAD) = 0`. The consumed accounting increments by zero, so `consumed[maker][group]` never advances, the `ConsumedAssets` guard always passes, and the offer can be filled indefinitely — including after it is already fully consumed. The protocol's own test `testBugBuyMaxAssetsBypass` and the NatSpec comment at line 94 both confirm this is a real, reachable condition.

### Finding Description

**Code path** — `src/Midnight.sol`, `take()`:

```
Line 363: buyerAssets = units.mulDivDown(buyerPrice, WAD)
           = mulDivDown(1, offerPrice, WAD)   // buyerPrice == offerPrice for buy offers
           = (1 * offerPrice) / 1e18
           = 0                                // when offerPrice < 1e18
Line 368: consumed[maker][group] += buyerAssets   // += 0  → no change
Line 369: require(newConsumed <= offer.maxAssets)  // always passes
```

`buyerPrice` collapses to `offerPrice` because `sellerPrice = offerPrice - _settlementFee` and `buyerPrice = sellerPrice + _settlementFee`, so the fee cancels out.

**Root cause** — `consumed` is incremented by `buyerAssets`, not by `units`. For any tick > 2910 (`MAX_TICK/2 = 2910`), `tickToPrice(tick) < WAD`, so a single-unit take produces `buyerAssets = 0`. The cap check at line 369 is therefore a no-op.

**Attacker inputs** — unprivileged taker (e.g. borrower):
- `offer.buy = true`, `offer.maxAssets = N > 0`, `offer.maxUnits = 0`
- `offer.tick` any multiple of `tickSpacing` in `(MAX_TICK/2, MAX_TICK]` (e.g. `MAX_TICK - 16 = 5804`)
- `units = 1` per call, repeated in a loop

**Why existing checks fail** — the only guard is `require(newConsumed <= offer.maxAssets)` at line 369. Because `newConsumed` is unchanged (delta = 0), this check is trivially satisfied on every iteration regardless of how many times the offer has been taken. [1](#0-0) 

The protocol acknowledges this at line 94 but does not prevent it: [2](#0-1) 

### Impact Explanation

Each successful `take(units=1)` with `buyerAssets = 0`:
- Increases the buyer (maker/lender) credit by 1 unit at zero cost
- Increases the seller (taker/borrower) debt by 1 unit with zero loan-token transfer
- Leaves `consumed[maker][group]` unchanged

An attacker can loop this call to give the maker unbounded credit and themselves unbounded debt, or — if the attacker is the maker — to give themselves unbounded credit while forcing a counterparty into unbounded debt, all while the `maxAssets` cap is never enforced. The offer is overfilled indefinitely with no asset payment. [3](#0-2) 

### Likelihood Explanation

**Preconditions:**
1. A buy offer exists with `maxAssets > 0` and `tick > MAX_TICK/2` (i.e., `offerPrice < WAD`). This is a normal, valid offer configuration — the protocol supports prices below 1.
2. The attacker is any unprivileged taker; no special role or privilege is required.
3. The offer must not be expired and must pass ratifier checks (standard conditions).

**Feasibility:** Fully reachable with no oracle manipulation, no admin access, and no impossible preconditions. The tick range `(2910, 5820]` covers half of all valid ticks. The loop cost is only gas per iteration.

**Repeatability:** Unlimited — each call with `units=1` succeeds independently and leaves state ready for the next call. [4](#0-3) 

### Recommendation

In the `maxAssets` branch, also enforce that `units == 0` whenever `buyerAssets == 0` (for buy offers) or `sellerAssets == 0` (for sell offers). Concretely, add:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetsTake()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any non-trivial take (units > 0) against an assets-capped offer must produce a nonzero asset delta, making the cap meaningful for all price ranges.

### Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already proves the single-call case. Extend it to a loop to prove indefinite overfill:

```solidity
function testBugBuyMaxAssetsBypassLoop() public {
    collateralize(market, borrower, 1000);
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

    uint256 N = 500;
    for (uint256 i = 0; i < N; i++) {
        (uint256 buyerAssets,) = take(1, borrower, lenderOffer);
        assertEq(buyerAssets, 0, "buyerAssets must be 0");
        // consumed never advances past maxAssets
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
    }
    // After N iterations, lender has N units of credit, borrower has N units of debt
    assertEq(midnight.creditOf(id, lender), N, "lender credit overfilled");
    assertEq(midnight.debtOf(id, borrower), N, "borrower debt overfilled");
    // maxAssets cap was never enforced
    assertGt(midnight.totalUnits(id), lenderOffer.maxAssets);
}
```

**Expected assertions:** all pass, demonstrating that `consumed` is frozen at `maxAssets = 1` while credit/debt grow to `N`, violating the invariant that `maxAssets` bounds total fill. [5](#0-4) [6](#0-5)
