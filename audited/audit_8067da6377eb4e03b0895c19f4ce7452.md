### Title
Incremental `withdraw` calls accumulate super-proportional `pendingFee` drain via `mulDivUp` rounding - (`src/Midnight.sol`)

### Summary

In `withdraw`, the `pendingFeeDecrease` is computed as `mulDivUp(_position.pendingFee * units, _position.credit)` (line 490). Because `mulDivUp` rounds up by at most 1 per call, and both `pendingFee` and `credit` decrease after each call, splitting a withdrawal of N units into N calls of 1 unit each can accumulate up to `pendingFee - 1` extra units of decrease compared to a single call. This allows a lender to drain their entire `pendingFee` obligation while withdrawing only a small fraction of their credit, permanently suppressing future protocol continuous fee accrual on the remaining credit.

### Finding Description

**Code path:**

`withdraw` → line 485 `_updatePosition` (no-op in same block, same timestamp, no slash) → line 489–491:

```solidity
if (_position.credit > 0) {
    pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
    _position.pendingFee -= pendingFeeDecrease;
}
_position.credit -= UtilsLib.toUint128(units);
```

`mulDivUp` is defined as `(x * y + (d - 1)) / d` — ceiling division, rounding error ≤ 1 per call. [1](#0-0) [2](#0-1) 

**Root cause:** After each call, `_position.pendingFee` and `_position.credit` both decrease. The ratio `pendingFee/credit` is not preserved across calls because each rounding-up step removes slightly more `pendingFee` than the exact proportional share. The next call then operates on a lower `pendingFee` but also a lower `credit`, and the ceiling rounding fires again. The errors compound.

**Attacker inputs:** `units = 1`, repeated N times via `multicall` (confirmed present in `src/Midnight.sol`) in a single transaction, same block — so `_updatePosition` is a no-op (zero elapsed time, no slashing).

**Exploit flow (concrete):**

State: `pendingFee = 3`, `credit = 10`, `withdrawable >= 5`.

| Call | `mulDivUp(pF, 1, cr)` | pF after | cr after |
|------|----------------------|----------|----------|
| 1 | `(3+9)/10 = 1` | 2 | 9 |
| 2 | `(2+8)/9 = 1` | 1 | 8 |
| 3 | `(1+7)/8 = 1` | 0 | 7 |
| 4 | 0 | 0 | 6 |
| 5 | 0 | 0 | 5 |
| **Total** | **3** | | |

Single `withdraw(units=5)`: `mulDivUp(3, 5, 10) = (15+9)/10 = 2`. Remaining `pendingFee = 1`.

The 5-call path drains 3 vs 2 — 1 extra unit, and leaves `pendingFee = 0` on 5 remaining credit.

**Amplified case:** `pendingFee = 100`, `credit = 10000`, N = 100 calls of 1.

- Each call: `mulDivUp(pF_i, 1, cr_i) = 1` (since `pF_i < cr_i` but `pF_i + cr_i - 1 >= cr_i`).
- Total drain: 100. Remaining `pendingFee = 0`.
- Single `withdraw(100)`: `mulDivUp(100, 100, 10000) = 1`. Remaining `pendingFee = 99`.
- **100× amplification**: lender drains 100 units of `pendingFee` instead of 1, by withdrawing only 1% of their credit.

**Why existing checks fail:**

The Certora rule `pendingFeeDecreasesProportionallyOnWithdraw` only asserts the single-call formula — it does not bound the sum of N calls. [3](#0-2) 

No minimum `units` check, no per-block withdrawal limit, and `multicall` is available for atomic batching.

### Impact Explanation

`pendingFee` is the sole source of future continuous fee accrual: `fee = pendingFee * elapsed / remaining_time` (line 814–815 of `_updatePosition`/`updatePositionView`). By draining `pendingFee` to 0 while retaining most of their credit, the lender permanently zeroes out all future fee accrual on that credit. The protocol's `continuousFeeCredit` accumulates nothing further from that position, constituting a direct, permanent loss of continuous fee revenue proportional to the remaining credit and time-to-maturity. [4](#0-3) 

### Likelihood Explanation

**Preconditions:**
- Lender has a position with `pendingFee > 0` and `credit > 0` (normal post-take state).
- `withdrawable >= pendingFee` (enough liquidity to withdraw `pendingFee` units of credit).
- All calls in the same block (no time passes, `_updatePosition` is a no-op).

All preconditions are routine. The attack is repeatable at any time the lender holds credit with a non-zero `pendingFee`. The amplification factor is `pendingFee / mulDivUp(pendingFee, pendingFee, credit)` ≈ `credit / pendingFee`, which is large when the continuous fee rate or time-to-maturity is small (low `pendingFee` relative to `credit`). The protocol has `multicall`, making the N-call sequence atomic and gas-efficient.

### Recommendation

Replace `mulDivUp` with `mulDivDown` for `pendingFeeDecrease` in `withdraw`:

```solidity
pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivDown(units, _position.credit));
```

`mulDivDown` rounds the decrease down, so the remaining `pendingFee` is always ≥ the exact proportional share. Rounding errors then accumulate in the protocol's favor (lender retains a slightly higher `pendingFee` obligation), and the sum of N small-call decreases is bounded by the single large-call decrease. This matches the conservative rounding direction used for fee accrual (`mulDivDown` at line 815) and is consistent with the protocol collecting at least the proportional fee. [5](#0-4)

        assertGt(pFeeAfterSingle, 0, "single withdrawal leaves proportional pendingFee");

    }

rule pendingFeeDecreasesProportionallyOnWithdraw(env e, Midnight.Market market, uint256 units, address onBehalf, address receiver) {
    bytes32 id;
    uint128 postUpdateCredit;
    uint128 postUpdatePendingFee;

    postUpdateCredit, postUpdatePendingFee, _ = updatePositionView(e, market, id, onBehalf);

    withdraw(e, market, units, onBehalf, receiver);

    require id == lastId, "id should be derived from market";

    // When postUpdateCredit == 0, pendingFee(id, onBehalf) is unchanged on withdraw.
    assert postUpdateCredit == 0 ? pendingFee(id, onBehalf) == postUpdatePendingFee : pendingFee(id, onBehalf) == postUpdatePendingFee - (postUpdatePendingFee * units + postUpdateCredit - 1) / postUpdateCredit;
}
