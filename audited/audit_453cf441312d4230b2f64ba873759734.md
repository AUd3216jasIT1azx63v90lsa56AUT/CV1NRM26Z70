Audit Report

## Title
Reentrancy in `take()` buyer callback allows draining pre-existing `withdrawable` balance at a discount - (File: src/Midnight.sol)

## Summary
`take()` violates checks-effects-interactions by writing `buyerPos.credit` to storage at line 410 and then making an external call to the maker-controlled `IBuyCallback.onBuy` at lines 448–452, before pulling payment via `safeTransferFrom` at lines 455–456. Because `withdraw()` has no reentrancy guard and no transient lock protecting it during `take()` callbacks, a malicious `IBuyCallback` authorized by the maker can call `withdraw()` inside `onBuy`, consuming pre-existing `withdrawable` balance funded by other borrowers' repayments and using those tokens to fund the take's own payment. The attacker nets `U - buyerAssets` tokens per attack at the direct expense of other lenders' withdrawable pool.

## Finding Description

**Root cause:** `take()` violates CEI. Storage is mutated at lines 408–418 (including `buyerPos.credit += buyerCreditIncrease` at line 410), then an external call is made to `IBuyCallback(buyerCallback).onBuy(...)` at lines 448–452, and only then are tokens pulled via `safeTransferFrom` at lines 455–456. The `LIQUIDATION_LOCK_SLOT` set at line 444 only prevents liquidation of the seller; it does not protect `withdraw()`. `withdraw()` (lines 481–500) has no reentrancy guard and no check that it is being called during a `take()` callback.

**Exact code path:**

`take()` with `offer.buy == true`:
- Line 375: `buyer = offer.maker`, `seller = taker`
- Line 379: `if (hasCredit(id, buyer) || units > buyerPos.debt) _updatePosition(offer.market, id, buyer)` — since `buyerCreditIncrease > 0` implies `units > buyerPos.debt`, this always fires when credit is being granted; sets `lastLossFactor = lossFactor`, `lastAccrual = block.timestamp`
- Line 410: `buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease)` — credit written to storage before any external call
- Line 420: `buyerCallback = offer.callback` — fully maker-controlled
- Line 422: `payer = buyerCallback` — the callback is also the payer for the subsequent `safeTransferFrom`
- Line 444: `LIQUIDATION_LOCK_SLOT` set for seller only
- Lines 448–452: `IBuyCallback(buyerCallback).onBuy(...)` — external call before token transfer
- Lines 455–456: `safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, payer, receiver, sellerAssets)` — token pull happens after callback returns

`withdraw()` (lines 481–500):
- Line 482: `isAuthorized[onBehalf][msg.sender]` — satisfied if maker pre-authorized the callback
- Line 485: `_updatePosition(market, id, onBehalf)` — is a no-op in the same block because `lastLossFactor == lossFactor` and `lastAccrual == block.timestamp` (set at line 379 of `take()`), so `updatePositionView` returns `(credit, pendingFee, 0)` with `credit = U` unchanged
- Line 493: `_position.credit -= units` — succeeds because credit was just increased to `U` at line 410
- Line 494: `_marketState.withdrawable -= units` — succeeds if pre-existing `withdrawable ≥ units`
- Line 499: transfers `units` tokens to receiver

**`_updatePosition` no-op proof:** After `_updatePosition` runs at line 379, `_position.lastLossFactor = marketState[id].lossFactor` (line 843) and `_position.lastAccrual = block.timestamp` (line 845). When `withdraw()` calls `_updatePosition` again in the same block, `updatePositionView` computes `postSlashCredit = credit * (MAX - lossFactor) / (MAX - lossFactor) = credit` (no slash) and `fee = pendingFee * (block.timestamp - block.timestamp) / ... = 0` (no accrual). The call is a pure no-op on credit.

**Exploit flow:**

Precondition: market has pre-existing `withdrawable = W ≥ U` from prior borrower repayments (via `repay()` line 509 or `liquidate()` line 675).

1. Maker deploys `MaliciousCallback` implementing `IBuyCallback`; calls `setIsAuthorized(MaliciousCallback, true, maker)` so `isAuthorized[maker][MaliciousCallback] = true`.
2. Maker creates buy offer with `offer.callback = MaliciousCallback`, ratified via a maker-controlled ratifier.
3. Taker calls `take(offer, ratifierData, U, taker, receiver, address(0), hex"")`.
4. `take()` sets `buyerPos.credit += U` (line 410).
5. `take()` calls `MaliciousCallback.onBuy(...)` (line 448).
6. Inside `onBuy`, `MaliciousCallback` calls `Midnight.withdraw(market, U, maker, attacker)`:
   - Auth: `isAuthorized[maker][MaliciousCallback]` = true, `msg.sender` = MaliciousCallback ✓
   - Credit: `_position.credit = U ≥ U` ✓
   - Withdrawable: `_marketState.withdrawable = W ≥ U` ✓
   - Result: `_position.credit = 0`, `withdrawable = W - U`, attacker receives U tokens.
7. `MaliciousCallback` approves Midnight for `buyerAssets` tokens (funded by the U tokens just received; since `buyerPrice < WAD`, `buyerAssets < U`).
8. `onBuy` returns `CALLBACK_SUCCESS`.
9. `take()` calls `safeTransferFrom(loanToken, MaliciousCallback, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, MaliciousCallback, receiver, sellerAssets)` — both succeed.

**Why existing checks fail:**
- `LIQUIDATION_LOCK_SLOT` (line 444) only prevents liquidation of the seller; it does not block `withdraw()`.
- `withdraw()` has no transient lock, no reentrancy guard, and no check that it is not being called during a `take()` callback.
- There is no post-callback invariant check in `take()` that `withdrawable` was not consumed.
- The `_updatePosition` call inside `withdraw()` (line 485) is a no-op in the same block since `lastLossFactor` and `lastAccrual` were already synchronized for the buyer at line 379 of `take()`.

## Impact Explanation

Direct theft of tokens from the `withdrawable` pool. The attacker receives `U` tokens immediately from the pre-existing `withdrawable` pool while paying only `buyerAssets = U × buyerPrice / WAD < U` tokens back. Net profit per attack: `U × (1 - buyerPrice/WAD)`. The contract's token balance is short by `U - (buyerAssets - sellerAssets)` with no accounting correction. Other lenders who had repaid tokens available for withdrawal find `withdrawable` reduced by `U`, constituting direct, concrete theft of other lenders' withdrawable assets.

## Likelihood Explanation

Preconditions are realistic in any active market: (a) pre-existing `withdrawable` balance exists whenever any borrower has repaid; (b) the maker can freely set `offer.callback` and authorize it via `setIsAuthorized` — no privileged role required; (c) the ratifier only requires that the maker (or their authorized operator) ratify the offer root. The attack is repeatable as long as `withdrawable` is non-zero and is profitable whenever `buyerPrice < WAD` (the normal lending case). The taker does not need to be malicious or even aware of the attack.

## Recommendation

Apply one or more of the following mitigations:

1. **Move token pulls before callbacks (enforce CEI):** Restructure `take()` so that `safeTransferFrom` is called before `IBuyCallback.onBuy`. This eliminates the window in which the callback can exploit the already-mutated state.
2. **Add a transient reentrancy guard to `withdraw()`:** Use a transient storage slot (analogous to `LIQUIDATION_LOCK_SLOT`) that is set at the start of `take()` and checked in `withdraw()`, causing `withdraw()` to revert if called during a `take()` callback.
3. **Snapshot and verify `withdrawable` around the callback:** Record `withdrawable` before the callback and assert it has not decreased after the callback returns.

Option 1 is the cleanest fix and aligns with standard CEI discipline.

## Proof of Concept

**Minimal Foundry test outline:**

```solidity
// Setup: deploy Midnight, create market, have a borrower repay U tokens
// so marketState[id].withdrawable = W >= U.

// Attacker setup:
MaliciousCallback cb = new MaliciousCallback(midnight, market, U, attacker);
midnight.setIsAuthorized(address(cb), true, maker); // maker authorizes cb

// Offer: buy offer, offer.callback = address(cb), buyerPrice < WAD
Offer memory offer = Offer({
    buy: true, maker: maker, callback: address(cb), ...
});

// Taker triggers take():
vm.prank(taker);
midnight.take(offer, ratifierData, U, taker, taker, address(0), "");

// MaliciousCallback.onBuy() calls midnight.withdraw(market, U, maker, attacker)
// attacker receives U tokens, then approves midnight for buyerAssets < U tokens
// take() completes successfully

// Assert:
assertEq(token.balanceOf(attacker), U - buyerAssets); // net profit
assertEq(midnight.withdrawable(id), W - U);           // pool drained
```

The `MaliciousCallback.onBuy` implementation calls `midnight.withdraw(market, U, maker, attacker)`, then approves `midnight` for `buyerAssets` tokens, and returns `CALLBACK_SUCCESS`. No privileged access is required beyond normal maker/user operations.
