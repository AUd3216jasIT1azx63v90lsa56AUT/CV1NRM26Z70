Audit Report

## Title
`maxAssets` cap bypass via zero-rounding `buyerAssets` in buy offers with `offerPrice < WAD` — (`src/Midnight.sol`)

## Summary
When `offer.buy == true` and `offer.maxAssets > 0`, the `take` function computes `buyerAssets` using `mulDivDown`, which truncates to zero when `units * buyerPrice < WAD`. Because `consumed` is incremented by `buyerAssets`, a zero result leaves the cap permanently unconsumed. The `EcrecoverRatifier` is entirely stateless, so the same signed `ratifierData` passes validation on every call, enabling unbounded replay. The protocol explicitly acknowledges this behavior in a NatDev comment and in a test named `testBugBuyMaxAssetsBypass`.

## Finding Description
**Root cause — `src/Midnight.sol` lines 363–369:**
```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // truncates to 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```
When `units * buyerPrice < WAD`, `buyerAssets == 0`. The `+=` adds zero; `require(newConsumed <= offer.maxAssets)` trivially passes. No other guard blocks a zero-asset take from completing.

**Why the EcrecoverRatifier does not stop replay — `src/ratifiers/EcrecoverRatifier.sol` lines 33–46:**
`isRatified` is a `view` function that checks only the Merkle proof, root-cancellation flag, and ECDSA signature. It stores no per-take nonce, counter, or consumed state. The identical `ratifierData` bytes satisfy all checks on every invocation.

**Position state mutates on every replay — `src/Midnight.sol` lines 408–414:**
```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);
```
Even with `buyerAssets == 0`, `units` are processed in full: the buyer (lender/maker) accumulates credit and the seller (borrower/taker) accumulates debt on every replay, with zero asset transfer and zero consumed increment.

**Protocol acknowledgment — `src/Midnight.sol` line 94:**
> *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*

**Existing test confirms the behavior — `test/TakeTest.sol` lines 857–889 (`testBugBuyMaxAssetsBypass`):**
The test pre-fills `consumed` to `maxAssets`, calls `take(1, ...)`, and asserts `buyerAssets == 0`, `consumed` unchanged, yet `creditOf` and `debtOf` strictly increase.

## Impact Explanation
The `maxAssets` cap — the sole mechanism limiting total buyer-asset exposure for a capped buy offer — is completely bypassed. An attacker controlling both a maker address and a separate taker address (prohibited from being the same by the `SelfTake` check, but trivially split across two EOAs/contracts) can:
1. Create a buy offer with `maxAssets = N` and `tick` chosen so `tickToPrice(tick) < WAD`.
2. Replay `take(1, ...)` K times (K unbounded) with the same `ratifierData`.
3. Accumulate K × `units` of lender credit with zero asset outlay.
4. At market maturity, redeem that credit for real loan-token assets held by the protocol, constituting direct theft from the protocol's reserves.

Even without collusion, any taker can force unbounded debt onto themselves and unbounded credit onto the maker, breaking the accounting invariant that credit is always backed by a corresponding asset deposit.

## Likelihood Explanation
All preconditions are reachable by an unprivileged external user:
1. `offer.buy == true`, `offer.maxAssets > 0` — standard lender buy offer with an asset cap.
2. `tickToPrice(tick) < WAD` — any tick below the WAD-price tick; the existing test uses `MAX_TICK - 16`.
3. `units = 1` (or any value where `units * offerPrice < WAD`) — trivially satisfied when `offerPrice` is near zero.
4. A valid EcrecoverRatifier proof — obtained once from the maker's signed Merkle root; identical bytes replay indefinitely.

The attack is executable across multiple transactions or, via a callback loop, within a single transaction. No special access, governance keys, or oracle manipulation is required.

## Recommendation
Add a minimum-assets guard before the consumed accounting block. For buy offers, require that `buyerAssets > 0` before proceeding:
```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)
    : units.mulDivUp(buyerPrice, WAD);
require(buyerAssets > 0 || !offer.buy, ZeroBuyerAssets()); // or use mulDivUp for buy offers
```
Alternatively, switch buy offers to `mulDivUp` for `buyerAssets` (consistent with the seller-side rounding), which eliminates the zero-truncation path entirely. Additionally, consider adding a per-take nonce or consumed-delta check in the `EcrecoverRatifier` to prevent stateless replay of the same authorization.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol` lines 857–889) is a complete, self-contained reproduction:
1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (so `offerPrice < WAD`).
2. Pre-fill `consumed` to `maxAssets` via `midnight.setConsumed(...)`.
3. Call `take(1, borrower, lenderOffer)`.
4. Assert: `buyerAssets == 0`, `consumed` unchanged at `maxAssets`, yet `creditOf(lender)` and `debtOf(borrower)` both strictly increased.

To demonstrate unbounded replay, extend the test to call `take(1, ...)` in a loop N times and assert that `creditOf(lender)` grows by N × `creditIncrease` while `consumed` remains at `maxAssets` throughout.
