### Title
Systematic per-lender credit over-slash via `mulDivDown` rounding in `updatePositionView` — (File: src/Midnight.sol)

### Summary
In `updatePositionView`, `postSlashCredit` is computed with `mulDivDown`, which floors the result by up to 1 unit per lender. The `lossFactor` update in `liquidate` also uses `mulDivDown`, rounding the surviving fraction down and therefore rounding `lossFactor` up. Both roundings compound in the same direction. Across N lenders, the aggregate credit destroyed can exceed the actual bad debt by up to N units per event, permanently orphaning those units inside `totalUnits` with no lender able to claim them.

### Finding Description
**Step 1 — `lossFactor` update in `liquidate`** (`src/Midnight.sol:631-633`):

```solidity
_marketState.lossFactor = UtilsLib.toUint128(
    type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
);
```

Let `MAX = type(uint128).max`, `A = MAX - old_lossFactor`, `T = totalUnits`, `B = badDebt`.

`new_lossFactor = MAX − ⌊A·(T−B)/T⌋`

Because `mulDivDown` floors the surviving fraction, `new_lossFactor` is rounded **up** (≥ exact). Therefore `MAX − new_lossFactor = ⌊A·(T−B)/T⌋ ≤ A·(T−B)/T`.

**Step 2 — `postSlashCredit` in `updatePositionView`** (`src/Midnight.sol:805-806`):

```solidity
uint256 postSlashCredit = _lastLossFactor < type(uint128).max
    ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
    : 0;
```

For a lender synced before the bad-debt event (`_lastLossFactor = old_lossFactor`):

`postSlashCredit_i = ⌊c_i · ⌊A·(T−B)/T⌋ / A⌋`

This is a **double floor**: once from the `lossFactor` update, once from this `mulDivDown`. Both reduce credit.

**Aggregate error:** The Certora `mulDivAddDownDown` rule (`certora/specs/MulDiv.spec:30-33`) proves `mulDivDown(a1,b,d) + mulDivDown(a2,b,d) ≤ mulDivDown(a1+a2,b,d)`. Applying this inductively over N lenders:

`Σ postSlashCredit_i ≤ ⌊T · ⌊A·(T−B)/T⌋ / A⌋ ≤ T−B`

The lower bound is `Σ postSlashCredit_i ≥ (T−B) − N` (each floor loses at most 1 unit). So lenders collectively lose up to N extra units beyond the actual bad debt.

**Concrete minimal example:**
- 2 lenders, `credit_1 = credit_2 = 1`; `totalUnits = 2`, `badDebt = 1`, `old_lossFactor = 0`
- `new_lossFactor = MAX − ⌊MAX/2⌋ = MAX − (2^127−1) = 2^127`
- Each lender: `postSlashCredit = ⌊1·(2^127−1)/(2^128−1)⌋ = 0`
- `totalUnits` after bad debt = 1, but `sum(credit_i) = 0`
- 1 unit permanently orphaned — invariant violated

**Why existing checks do not stop it:** No on-chain check or Certora rule verifies `Σ credit_i ≥ totalUnits` after slashing. The specs verify per-lender credit monotonicity and `lastLossFactor` sync (`certora/specs/LossFactor.spec:63-70`, `certora/specs/BalanceEffects.spec:36-55`) but not the aggregate sum invariant. The existing test tolerance (`assertApproxEqAbs(..., 1)` at `test/LiquidationTest.sol:361`) acknowledges single-lender rounding but does not cover the multi-lender sum.

### Impact Explanation
Lenders collectively lose up to N extra units per bad-debt event (N = number of lenders with non-zero credit at the time of slashing). These units remain counted in `totalUnits` but no lender holds credit for them, making them permanently unclaimable. Over K bad-debt events with N lenders each time, the total orphaned value is up to N·K units. This is systematic, irreversible value leakage from lenders beyond their proportional share of bad debt, violating the invariant that `Σ credit_losses = total_bad_debt_realized`.

### Likelihood Explanation
Triggered by any bad-debt liquidation in a market with more than one lender — the standard operating condition. No privileged access, no oracle manipulation, and no special sequencing beyond the described call sequence is required. The effect is small per event (≤ N units) but accumulates monotonically over the market lifetime and is amplified by markets with many lenders or frequent bad-debt events.

### Recommendation
Round `lossFactor` in the direction that preserves aggregate lender credit. Specifically, use `mulDivUp` for the surviving-fraction computation in `liquidate` so that `MAX − new_lossFactor` is rounded **up**, ensuring `Σ postSlashCredit_i ≥ totalUnits − badDebt`:

```solidity
// In liquidate, replace:
_marketState.lossFactor = UtilsLib.toUint128(
    type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
);
// With:
_marketState.lossFactor = UtilsLib.toUint128(
    type(uint128).max - (type(uint128).max - _lossFactor).mulDivUp(_totalUnits - badDebt, _totalUnits)
);
```

This makes `new_lossFactor` round **down** (≤ exact), so `MAX − new_lossFactor` rounds up, and per-lender `postSlashCredit` values collectively cover `totalUnits − badDebt`. The `continuousFeeCredit` scaling on lines 635-640 should be reviewed for the same direction.

// Foundry fuzz/invariant test
function testSumCreditViolatesInvariantAfterBadDebt() public {
    // Setup: 2 lenders each with credit = 1, totalUnits = 2
    // Trigger bad debt of 1 (e.g., price = 0 liquidation)
    // updatePosition for both lenders
    uint256 sumCredit = midnight.creditOf(id, lender1) + midnight.creditOf(id, lender2);
    uint256 remainingUnits = midnight.totalUnits(id); // = 1
    // FAILS: sumCredit == 0, remainingUnits == 1

function testFuzzSumCreditInvariant(uint128 c1, uint128 c2, uint128 badDebt) public {
