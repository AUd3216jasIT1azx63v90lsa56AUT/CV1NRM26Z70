### Title
Fee-on-Transfer Collateral Token Inflates Recorded Collateral, Enabling Undercollateralized Borrowing - (File: src/Midnight.sol)

### Summary
`supplyCollateral()` records the caller-supplied `assets` parameter directly into `position[id][onBehalf].collateral[collateralIndex]` before calling `SafeTransferLib.safeTransferFrom()`, which only verifies the boolean return value of `transferFrom` and never measures the actual balance received. For a fee-on-transfer collateral token, the contract records the nominal `assets` but receives only `assets * (1 - fee)`. `isHealthy()` and the post-`take` health check both read the inflated stored value, allowing a borrower to carry debt that exceeds the value of collateral actually held by the protocol.

### Finding Description
**Root cause — no balance-before/after check in `supplyCollateral()`:**

`src/Midnight.sol:533` writes the nominal amount unconditionally:
```solidity
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
``` [1](#0-0) 

`src/Midnight.sol:545` then calls the transfer:
```solidity
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom()` only checks that `transferFrom` did not revert and returned `true`; it performs no balance delta measurement: [3](#0-2) 

A fee-on-transfer token's `transferFrom` returns `true` while silently delivering only `assets * (1 - fee_rate)` to the contract. The discrepancy is never detected.

**Health check reads the inflated stored value:**

`isHealthy()` computes `maxDebt` from `_position.collateral[i]` (the nominal, inflated value): [4](#0-3) 

The post-`take` health check at line 476 calls `isHealthy()` with the same inflated collateral: [5](#0-4) 

**Exploit flow:**

1. Attacker (market creator + borrower, both unprivileged roles) creates a market whose `collateralParams[0].token` is a fee-on-transfer ERC20 with fee rate `f` (e.g. 10%), oracle price `P`, and LLTV `L` chosen so that `assets * P * L / WAD >= debt` but `(1-f) * assets * P * L / WAD < debt`.
2. Attacker calls `supplyCollateral(market, 0, assets, borrower)`. The contract records `position[id][borrower].collateral[0] = assets` but holds only `(1-f) * assets` of the token.
3. Attacker (as maker of a sell offer) is taken, or takes a buy offer, accumulating `debt` units. The post-`take` `isHealthy()` call passes because it uses the nominal `assets`.
4. The position is now undercollateralized: actual collateral value is `(1-f) * assets * P * L / WAD < debt`.

**Why existing checks fail:**

- `SafeTransferLib` has no balance-delta guard.
- `isHealthy()` has no access to actual token balances; it reads only the stored accounting value.
- No whitelist restricts collateral tokens; market creation is permissionless via `touchMarket`.
- The Certora `supplyCollateralEffects` rule asserts `collateral == collateralBefore + assets` — this is proved under the assumption that token transfers are modelled as no-ops (see `Role.spec:46-47`), so the formal verification does not cover fee-on-transfer tokens. [6](#0-5) 

### Impact Explanation
The protocol holds less collateral than its accounting records. A borrower can carry debt that is not fully backed by actual token holdings. When the position becomes unhealthy and a liquidator attempts to seize `seizedAssets` tokens, `safeTransfer` will revert if `seizedAssets > (1-f) * assets` (insufficient contract balance), making the position partially or fully unliquidatable. The residual unrecoverable debt is socialized as bad debt among lenders via the loss-factor mechanism, causing direct lender losses without any price movement.

### Likelihood Explanation
- Any unprivileged user can create a market with a fee-on-transfer collateral token (market creation is permissionless).
- Fee-on-transfer tokens are a well-known ERC20 variant (e.g. STA, PAXG with fees enabled, deflationary tokens).
- The attack requires no special timing, no oracle manipulation, and no privileged access.
- It is repeatable: each `supplyCollateral` call with a fee-on-transfer token inflates the recorded balance by the fee amount.
- The only precondition is that the oracle price and LLTV are set such that the phantom collateral gap is exploitable, which the attacker controls at market creation time.

### Recommendation
Measure the actual balance received in `supplyCollateral()` using a balance-before/after pattern:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Alternatively, explicitly document that fee-on-transfer tokens are unsupported as collateral and add a check (e.g. via a token registry or a one-time balance verification at market creation) to enforce this invariant.

// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {Market, CollateralParams} from "src/interfaces/IMidnight.sol";

/// @dev ERC20 that takes a 10% fee on every transferFrom.
contract FeeOnTransferToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    uint256 public constant FEE = 10; // 10%

    function mint(address to, uint256 amount) external { balanceOf[to] += amount; }
    function approve(address spender, uint256 amount) external returns (bool) {
        allowance[msg.sender][spender] = amount; return true;
    }
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        allowance[from][msg.sender] -= amount;
        uint256 fee = amount * FEE / 100;
        balanceOf[from] -= amount;
        balanceOf[to] += amount - fee; // fee burned
        return true; // returns true despite delivering less
    }
    function transfer(address to, uint256 amount) external returns (bool) {
        balanceOf[msg.sender] -= amount; balanceOf[to] += amount; return true;
    }
}

contract FeeOnTransferCollateralTest is Test {
    Midnight midnight;
    FeeOnTransferToken collateral;
    // ... standard test setup ...

    function testPhantomCollateralBadDebt() public {
        // Setup: oracle price = 1e36 (ORACLE_PRICE_SCALE), LLTV = 0.9e18
        // assets = 1000, fee = 10% → received = 900
        // maxDebt (nominal) = 1000 * 1e36 / 1e36 * 0.9e18 / 1e18 = 900
        // maxDebt (actual)  = 900  * 1e36 / 1e36 * 0.9e18 / 1e18 = 810
        // We borrow 850 units: nominal check passes (900 >= 850), actual fails (810 < 850)

        uint256 assets = 1000;
        collateral.mint(address(this), assets);
        collateral.approve(address(midnight), assets);

        midnight.supplyCollateral(market, 0, assets, borrower);

        // Assert: recorded collateral = 1000, actual balance = 900
        assertEq(midnight.collateral(id, borrower, 0), 1000);
        assertEq(collateral.balanceOf(address(midnight)), 900);

        // Borrow 850 units via take (sell offer taken by lender)
        // isHealthy: 1000 * price * LLTV / WAD = 900 >= 850 → passes
        // Actual:     900 * price * LLTV / WAD = 810 < 850  → undercollateralized

        // After take, assert debt > actual collateral value
        uint256 debt = midnight.debtOf(id, borrower);
        uint256 actualCollateralValue = collateral.balanceOf(address(midnight)); // 900
        // actualCollateralValue * LLTV / WAD = 810 < debt = 850
        assertGt(debt, actualCollateralValue * 9 / 10); // 850 > 810

        // Liquidation attempt to seize full collateral fails if seizedAssets > 900
        // (contract only holds 900 but recorded 1000)
    }
}
```

**Expected assertions:**
- `midnight.collateral(id, borrower, 0) == 1000` (nominal, inflated)
- `collateral.balanceOf(address(midnight)) == 900` (actual)
- `midnight.debtOf(id, borrower) == 850` (passes health check using nominal value)
- `850 > 900 * LLTV / WAD` (position is undercollateralized relative to actual holdings)
- Liquidation attempting to seize `> 900` tokens reverts with insufficient balance

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];

                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)

                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
