### Title
Stale Signed Authorization Replay After Direct Revocation Re-grants Operator Access — (`src/periphery/EcrecoverAuthorizer.sol`)

### Summary
`EcrecoverAuthorizer` maintains its own per-authorizer `nonce` that is only incremented when `EcrecoverAuthorizer.setIsAuthorized` is called. When a victim revokes an operator directly via `Midnight.setIsAuthorized`, the `EcrecoverAuthorizer` nonce is never incremented, leaving any previously signed `Authorization` with the current nonce still valid and submittable by anyone. An attacker can replay the stale signature to re-grant the operator, who can then drain the victim's credit via `withdraw`.

### Finding Description

**Code path and root cause:**

`EcrecoverAuthorizer.setIsAuthorized` (line 26) enforces:
```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
``` [1](#0-0) 

The `nonce` mapping lives exclusively in `EcrecoverAuthorizer` and is only mutated here. [2](#0-1) 

`Midnight.setIsAuthorized` (the direct revocation path) only writes to `isAuthorized[onBehalf][authorized]` and emits an event — it has no knowledge of, and makes no call into, `EcrecoverAuthorizer.nonce`. [3](#0-2) 

**Exploit flow:**

1. Victim calls `midnight.setIsAuthorized(ecrecoverAuthorizer, true, victim)` — required so `EcrecoverAuthorizer` can act on the victim's behalf. `EcrecoverAuthorizer.nonce[victim] == 0`.
2. Victim signs `Authorization{authorizer=victim, authorized=operator, isAuthorized=true, nonce=0, deadline=now+1day}`.
3. Victim later revokes operator directly: `midnight.setIsAuthorized(operator, false, victim)`. This sets `isAuthorized[victim][operator] = false` in Midnight but **leaves `EcrecoverAuthorizer.nonce[victim] == 0` unchanged**.
4. Attacker (before deadline) calls `EcrecoverAuthorizer.setIsAuthorized(auth, sig)` with the old signed message.
5. Nonce check: `0 == nonce[victim]++` → passes; nonce becomes 1.
6. Signature verifies against victim's key → passes.
7. Authorization check at line 34: signer (`victim`) `== authorization.authorizer` → passes. [4](#0-3) 
8. `midnight.setIsAuthorized(operator, true, victim)` is called; `EcrecoverAuthorizer` is still authorized in Midnight, so the `onBehalf` check passes. [5](#0-4) 
9. `isAuthorized[victim][operator] = true` is restored.
10. Operator calls `midnight.withdraw(market, units, victim, attacker)` — authorization check passes, victim's credit is drained.

**Why existing checks fail:** The nonce check only prevents replay of a nonce that was already consumed *through `EcrecoverAuthorizer`*. A direct `Midnight.setIsAuthorized` revocation does not consume any nonce in `EcrecoverAuthorizer`, so the signed message at nonce 0 remains unconsumed and valid until its deadline.

### Impact Explanation
A victim who revokes an operator via the natural `Midnight.setIsAuthorized` path believes the revocation is final. An attacker holding any unexpired signed authorization from the victim can immediately re-grant the operator, who can then call `withdraw` (or any other authorized action) to steal the victim's credit. This is a direct, concrete theft of user funds.

### Likelihood Explanation
**Preconditions:**
- Victim must have authorized `EcrecoverAuthorizer` in Midnight (standard setup for any user of the signed-authorization flow).
- Victim must have signed at least one `Authorization` with `isAuthorized=true` whose deadline has not yet expired.
- Victim must have revoked the operator via `Midnight.setIsAuthorized` directly (the natural, documented revocation path) rather than via `EcrecoverAuthorizer`.

All three preconditions are realistic and likely for any user of the `EcrecoverAuthorizer` peripheral. The attack is permissionless (any address can submit the old signature), repeatable until the deadline expires, and requires no special privileges.

### Recommendation
The nonce in `EcrecoverAuthorizer` must be invalidated whenever the authorizer takes any direct action that changes authorization state. The cleanest fix is to expose a `invalidateNonce(address authorizer)` function on `EcrecoverAuthorizer` and have `Midnight.setIsAuthorized` call it, or — more practically — store the nonce inside `Midnight` itself so that any direct `setIsAuthorized` call atomically increments it. Alternatively, document that revocation **must** go through `EcrecoverAuthorizer` (signing a revocation message) and enforce this by removing the direct `Midnight.setIsAuthorized` path for users who have authorized `EcrecoverAuthorizer`. The simplest safe fix: move the nonce into `Midnight` and increment it on every `setIsAuthorized` call regardless of caller.

### Proof of Concept

```solidity
// Foundry unit test
function testStaleAuthReplayAfterDirectRevoke() public {
    // Step 1: victim authorizes EcrecoverAuthorizer in Midnight
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);

    // Step 2: victim signs Authorization{isAuthorized=true, operator, nonce=0, deadline=now+1day}
    Authorization memory auth = Authorization({
        authorizer: victim,
        authorized: operator,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory sig = signAuthorization(auth, victim);

    // Step 3: victim revokes operator directly via Midnight (nonce in EcrecoverAuthorizer stays 0)
    vm.prank(victim);
    midnight.setIsAuthorized(operator, false, victim);
    assertEq(midnight.isAuthorized(victim, operator), false);
    assertEq(ecrecoverAuthorizer.nonce(victim), 0); // nonce NOT incremented

    // Step 4: attacker replays old signed auth
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(auth, sig); // must not succeed

    // Step 5: assert operator is NOT re-authorized (this assertion FAILS, proving the bug)
    assertEq(midnight.isAuthorized(victim, operator), false, "operator must remain revoked");

    // Step 6 (demonstrating theft): operator withdraws victim's credit
    vm.prank(operator);
    midnight.withdraw(market, victimUnits, victim, attacker);
    // attacker receives victim's funds
}
```

**Expected (correct) behavior:** `ecrecoverAuthorizer.setIsAuthorized` reverts or the final `isAuthorized` assertion holds.
**Actual behavior:** The call succeeds, operator is re-authorized, and victim's funds are drained.
