Audit Report

## Title
`maxAssets` cap bypassed via zero-rounding `buyerAssets` on sub-WAD price buy offers — (`src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and its tick yields a price below WAD, the consumed-tracking logic at line 368 adds `mulDivDown(units, buyerPrice, WAD)` to `consumed[maker][group]`. For `units = 1` and any `buyerPrice < WAD`, integer division floors this to zero, so `consumed` never increments and the `require(newConsumed <= offer.maxAssets)` guard always passes. An unprivileged taker can call `take` with `units = 1` indefinitely, inflating the maker's credit and the taker's debt beyond the maker's intended cap at zero token cost.

## Finding Description
**Exact code path — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // floors to 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [1](#0-0) 

**Root cause:** `mulDivDown(1, buyerPrice, WAD)` computes `floor(buyerPrice / 1e18)`. For any tick below `MAX_TICK` (i.e., `buyerPrice < 1e18`), this is `0`. The `consumed` mapping is incremented by zero on every call, so `newConsumed` never advances and the cap check trivially passes regardless of how many times the offer has been filled.

**The protocol's own NatSpec at line 94 explicitly documents this behavior:**

> "It is possible to give units to a fully consumed assets-based buy offer with price < 1." [2](#0-1) 

**Exploit flow:**
1. Maker posts a buy offer: `offer.buy = true`, `offer.maxAssets = N`, `offer.tick` any value in `[2, MAX_TICK - 1]`.
2. Taker calls `take(offer, ..., units=1, ...)` in a loop (or via `multicall`).
3. Each iteration: `buyerAssets = 0`, `consumed += 0`, cap check passes.
4. Each iteration: `buyerPos.credit += 1`, `sellerPos.debt += 1`, `totalUnits += 1`.
5. No loan tokens are transferred in either direction (`buyerAssets = sellerAssets = 0`).

**Why existing checks fail:** The `require(newConsumed <= offer.maxAssets)` guard at line 369 is the sole enforcement mechanism for the asset cap. It is defeated because the addend is always zero for sub-WAD prices at `units = 1`. There is no secondary floor requiring `units == 0` when `buyerAssets == 0`, and no minimum-assets guard. [3](#0-2) 

## Impact Explanation
The maker's `maxAssets` cap — their primary tool for bounding credit/debt exposure — is rendered entirely ineffective for any buy offer placed below par (the overwhelming majority of real offers). The attacker accumulates unbounded debt at zero token cost. At maturity, this debt is unpayable (no loan tokens were ever received), causing bad debt that is socialized across all lenders in the market. `totalUnits` is also inflated, corrupting market-wide accounting. This constitutes unauthorized state mutation and potential direct loss of assets for the maker and other lenders via bad-debt socialization.

## Likelihood Explanation
Preconditions are trivially satisfiable by any unprivileged external address: any buy offer at any tick below `MAX_TICK` is vulnerable, which covers virtually all real-world offers (a tick at `MAX_TICK` corresponds to a zero-yield loan at price = 1.0 WAD). The attacker requires no capital, no oracle manipulation, and no special role. The attack is repeatable in a single transaction via `multicall`. The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` confirms reproducibility. [4](#0-3) 

## Recommendation
Add a guard that prevents a take from proceeding when `offer.maxAssets > 0` and `buyerAssets == 0` but `units > 0`. Concretely, after computing `buyerAssets` at line 363, require:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(buyerAssets > 0 || units == 0, ZeroAssetsTake());
}
```

Alternatively, use `mulDivUp` instead of `mulDivDown` for the buy-offer consumed tracking so that even a fractional asset rounds up to 1, ensuring `consumed` always advances by at least 1 per unit taken. A third option is to enforce a minimum `units` floor such that `units * buyerPrice >= WAD` before accepting the take.

## Proof of Concept
The test `testBugBuyMaxAssetsBypass` already exists in `test/TakeTest.sol` and reproduces the issue. Minimal manual steps:

1. Deploy Midnight; create a market with maturity in the future.
2. Maker calls `take` (or a setup function) to post a buy offer: `maxAssets = 100`, `tick = 2` (price ≈ `1e12`).
3. Pre-consume the offer to its cap by legitimate means (or skip if testing from zero).
4. Attacker calls `take(offer, taker=attacker, units=1, ...)`.
5. Assert: `consumed[maker][group]` is unchanged (still equals pre-attack value); `buyerPos.credit` increased by 1; `sellerPos.debt` increased by 1; no loan token transfer occurred.
6. Repeat step 4 arbitrarily many times — all calls succeed, credit/debt grow without bound.
