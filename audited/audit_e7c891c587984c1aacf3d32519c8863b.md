Audit Report

## Title
Zero-`buyerAssets` rounding bypasses `maxAssets` cap, allowing unlimited free position mutations - (File: src/Midnight.sol)

## Summary
In `take()`, when a buy offer has `maxAssets > 0`, the consumed counter is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For any tick below `MAX_TICK` with `units = 1`, integer truncation yields `buyerAssets = 0`, so `consumed` never advances. Because all position mutations (`credit`, `debt`, `totalUnits`) use `units` directly, a taker can call `take` repeatedly at zero token cost while the `maxAssets` cap is never enforced — even when `consumed` is already equal to `maxAssets`.

## Finding Description

**Root cause — denomination mismatch with no zero-asset guard:**

`src/Midnight.sol` line 363 computes:
```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
```
For a buy offer with `units = 1` and any `tick < MAX_TICK`, `buyerPrice < WAD = 1e18`, so `1 * buyerPrice / 1e18 = 0` by integer truncation. `PRICE_ROUNDING_STEP = 1e12` (confirmed at `src/libraries/TickLib.sol` line 8) and `MAX_TICK = 5820` (line 6), meaning 5819 out of 5820 valid ticks produce `buyerAssets = 0` for `units = 1`.

Lines 367–369:
```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```
When `buyerAssets == 0`, `consumed += 0`, so even if `consumed == maxAssets` already, `newConsumed == maxAssets <= maxAssets` trivially passes.

Lines 382–417 then mutate positions using `units`, not `buyerAssets`:
- `buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)` (line 382)
- `buyerPos.credit += buyerCreditIncrease` (line 410)
- `sellerPos.debt += sellerDebtIncrease` (line 414)
- `_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease` (lines 416–417)

Line 455 transfers `buyerAssets - sellerAssets = 0 - 0 = 0` tokens.

**Proof — existing test `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol` lines 857–889):**
The test sets `maxAssets = 1`, pre-fills `consumed` to `maxAssets` via `setConsumed`, then calls `take(1, borrower, lenderOffer)` with `tick = MAX_TICK - 16`. It asserts `buyerAssets == 0`, `consumed` unchanged, token balances unchanged — yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.

## Impact Explanation
A taker can fill a buy offer with `maxAssets > 0` an unlimited number of times at zero token cost. Each fill increases the maker's credit and the taker's debt by `units` without any token transfer and without advancing `consumed`. This violates the core accounting invariant that every credit increase must correspond to a valid asset transfer. The taker accrues unbounded debt and the maker gains unbounded credit with no economic exchange, corrupting market state and enabling manipulation of settlement fee accounting, downstream liquidation logic, and position health calculations.

## Likelihood Explanation
Preconditions are minimal: (1) a buy offer with `maxAssets > 0` exists — a standard offer type; (2) `tickToPrice(offer.tick) < WAD` — true for all 5819 valid ticks except `MAX_TICK`; (3) taker uses `units = 1`. No privileged access, oracle manipulation, or front-running is required. The attack is executable in a single transaction and is trivially repeatable with negligible gas cost per iteration.

## Recommendation
Add a guard immediately after computing `buyerAssets`/`sellerAssets` that rejects zero-asset fills when `maxAssets > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetAmount = offer.buy ? buyerAssets : sellerAssets;
    require(assetAmount > 0, ZeroAssetFill());
    newConsumed = consumed[offer.maker][offer.group] += assetAmount;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This closes the denomination mismatch by ensuring that any fill which would transfer zero assets is rejected outright, preventing position mutations without economic exchange.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) is a complete, self-contained PoC. Run:
```
forge test --match-test testBugBuyMaxAssetsBypass -vvvv
```
The test passes, confirming that `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all increase while `consumed` remains at `maxAssets` and zero tokens are transferred.
