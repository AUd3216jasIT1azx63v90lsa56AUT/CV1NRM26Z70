### Title
Division by Zero in `liquidate` When `collateralIndex` Not in Borrower's Bitmap with `repaidUnits > 0` - (File: src/Midnight.sol)

### Summary

In `Midnight.liquidate`, `liquidatedCollatPrice` is initialized to `0` and only set inside the bitmap loop when `i == collateralIndex`. If `collateralIndex` is not in the borrower's `collateralBitmap`, the loop never sets it, leaving it `0`. When `repaidUnits > 0` and `seizedAssets == 0`, the code executes `seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice)`, dividing by zero and reverting. The Certora spec `NoDivisionByZero.spec` explicitly requires the collateral to be activated as an assumption to avoid this, confirming the code has no corresponding runtime guard.

### Finding Description

**Root cause — `src/Midnight.sol` lines 603, 607–618, 651–652:**

`liquidatedCollatPrice` is declared as `uint256` (default `0`) at line 603. The bitmap loop at lines 607–618 only assigns it when `i == collateralIndex`:

```solidity
uint256 liquidatedCollatPrice;          // line 603 — initialized to 0
...
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    ...
    if (i == collateralIndex) liquidatedCollatPrice = price;   // line 611
    ...
}
```

If `collateralIndex` is absent from `_position.collateralBitmap`, the assignment at line 611 is never reached and `liquidatedCollatPrice` stays `0`.

Later, at lines 651–652, when `repaidUnits > 0` and `seizedAssets == 0`:

```solidity
seizedAssets = repaidUnits.mulDivDown(lif, WAD)
                           .mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice); // div by 0
```

`mulDivDown` is `(x * y) / d` (line 30 of `UtilsLib.sol`), so `d = 0` triggers a Solidity division-by-zero panic revert.

**No existing check prevents this.** The only input validation before this point is:
- Line 595: `atMostOneNonZero(repaidUnits, seizedAssets)` — allows `repaidUnits > 0, seizedAssets == 0`.
- Line 596: `_position.debt > 0` — requires a borrower, not that `collateralIndex` is active.
- Lines 620–624: liquidatability check — passes in post-maturity mode purely on `block.timestamp > market.maturity`, independent of which collaterals are active.

There is no `require` that `collateralIndex` must be set in `_position.collateralBitmap` when `repaidUnits > 0`.

**The Certora spec confirms the gap.** `certora/specs/NoDivisionByZero.spec` lines 123–125 explicitly add this as a *proof assumption*, not a code invariant:

```
// Assume that the collateral price is non-zero and the collateral is active.
// Otherwise, liquidate may revert with div by zero.
require ghostPrice(market.collateralParams[collateralIndex].oracle) > 0, ...;
require summaryGetBit(...collateralBitmap, collateralIndex), "Assumption: liquidated collateral was activated";
```

The spec sidesteps the revert with an assumption rather than the contract enforcing it.

**Exploit flow:**

1. Maker creates an offer in a multi-collateral market (e.g., collaterals at indices 0 and 1). `SetterRatifier.setIsRootRatified` is called to ratify the offer root.
2. A taker calls `take`, causing the maker/seller to accumulate debt (borrower position established).
3. The borrower's `collateralBitmap` has bit 0 set (collateral at index 0 supplied) but bit 1 **not** set (no collateral at index 1).
4. The borrower becomes liquidatable — either post-maturity (`block.timestamp > market.maturity`) or unhealthy (`debt > maxDebt` from index-0 collateral alone).
5. An unprivileged liquidator calls `liquidate(market, collateralIndex=1, seizedAssets=0, repaidUnits=X, borrower, ...)` with `X > 0`.
6. The bitmap loop runs over bit 0 only; `i == 1` is never true; `liquidatedCollatPrice` stays `0`.
7. Line 652 executes `mulDivDown(ORACLE_PRICE_SCALE, 0)` → panic revert.

### Impact Explanation

Any liquidator using the `repaidUnits`-input mode of `liquidate` with a `collateralIndex` not present in the borrower's bitmap receives an unexpected division-by-zero revert. This violates the invariant that unhealthy/post-maturity positions remain liquidatable: the `repaidUnits` liquidation path is completely blocked for that collateral index, even though the borrower is legitimately liquidatable. In post-maturity mode this is particularly relevant because the liquidator may specifically target a collateral index that the borrower never activated (e.g., to realize bad debt), and the call will always revert.

### Likelihood Explanation

**Preconditions:**
- A borrower with debt whose `collateralBitmap` does not include `collateralIndex` — a normal, reachable state (borrowers choose which collaterals to supply).
- The borrower is liquidatable (post-maturity or unhealthy) — standard liquidation precondition.
- The liquidator passes `repaidUnits > 0` with that `collateralIndex` — a valid API usage per the function signature and NatSpec.

No privileged action, oracle manipulation, or impossible state is required. The scenario is repeatable any time a liquidator targets a collateral index not in the borrower's bitmap with the `repaidUnits` input. The NatSpec comment at lines 578–579 ("Liquidations with both 0 for seizedAssets and repaidUnits can be done with a collateral that is not activated") implies non-zero liquidations should use an activated collateral, but this is undocumented and unenforced.

### Recommendation

Add an explicit check at the start of the `if (repaidUnits > 0 || seizedAssets > 0)` block (line 643) to require that `collateralIndex` is set in the borrower's bitmap:

```solidity
if (repaidUnits > 0 || seizedAssets > 0) {
    require(
        UtilsLib.getBit(_position.collateralBitmap, collateralIndex),
        CollateralNotActivated()
    );
    ...
}
```

This mirrors the assumption already present in `certora/specs/NoDivisionByZero.spec` line 125 and makes the invariant explicit in the contract rather than only in the formal spec.

**File:** certora/specs/NoDivisionByZero.spec (L114-128)
```text
// Show that liquidate does not cause a division by zero, in case the oracle price is non-zero and the collateral is active.
rule noDivisionByZeroLiquidate(env e, Midnight.Market market, uint256 collateralIndex, uint256 seizedAssets, uint256 repaidUnits, address borrower, address receiver, address callback, bytes data, bool postMaturityMode) {
    require equalsGlobalMarket(market);

    // Needed for the bitmap loop which calls mulDivUp(WAD, maxLif) for every activated collateral.
    require forall uint256 i. i < market.collateralParams.length => market.collateralParams[i].maxLif >= WAD(), "see maxLifIsAtLeastWad in ExactMath.spec";

    require market.collateralParams[collateralIndex].lltv < WAD() => to_mathint(market.collateralParams[collateralIndex].maxLif) * to_mathint(market.collateralParams[collateralIndex].lltv) <= to_mathint(WAD()) * (to_mathint(WAD()) - 1), "see lifTimesLltvStrictBound in ExactMath.spec";

    // Assume that the collateral price is non-zero and the collateral is active. Otherwise, liquidate may revert with div by zero.
    require ghostPrice(market.collateralParams[collateralIndex].oracle) > 0, "Assumption: the collateral price is not zero";
    require summaryGetBit(currentContract.position[globalId][borrower].collateralBitmap, collateralIndex), "Assumption: liquidated collateral was activated";

    liquidate(e, market, collateralIndex, seizedAssets, repaidUnits, borrower, postMaturityMode, receiver, callback, data);
    assert true;
```
