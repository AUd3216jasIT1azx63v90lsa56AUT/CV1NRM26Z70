### Title
Buy offer with `maxAssets > 0` and `buyerPrice < WAD` can be overfilled in unit terms via `mulDivDown` rounding to zero — (`src/Midnight.sol`)

### Summary
When `offer.buy = true`, `offer.maxAssets > 0`, and `buyerPrice < WAD` (tick > `MAX_TICK/2 = 2910`), the consumed tracking increments by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For small `units` values where `buyerAssets` rounds down to zero, the consumed counter does not increase, so the `require(newConsumed <= offer.maxAssets)` check always passes. This allows a taker to fill the offer with arbitrarily many additional units beyond `floor(maxAssets * WAD / buyerPrice)` at zero asset cost, overfilling the maker's buy offer in unit terms. The protocol team has documented and tested this exact behavior but has not fixed it.

### Finding Description
**Code path:** `src/Midnight.sol` lines 363–373.

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;  // line 363
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets; // line 368
    require(newConsumed <= offer.maxAssets, ConsumedAssets());                                  // line 369
}
``` [1](#0-0) 

**Root cause:** When `buyerPrice < WAD` (i.e., `tickToPrice(offer.tick) - settlementFee < WAD`, which holds for any tick above `MAX_TICK/2 = 2910`), `mulDivDown(units, buyerPrice, WAD)` truncates to zero whenever `units * buyerPrice < WAD`. The consumed counter is incremented by this zero value, so the cap check `newConsumed <= maxAssets` is trivially satisfied regardless of how many units have already been filled. [2](#0-1) 

**Exploit flow:**
1. Maker creates a buy offer: `offer.buy = true`, `offer.maxAssets = N`, `offer.tick > 2910` (so `buyerPrice < WAD`).
2. Taker fills the offer normally with `units = floor(N * WAD / buyerPrice)`, bringing `consumed = N`.
3. Taker calls `take(1, ...)`: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`, so `newConsumed = N + 0 = N ≤ N` — check passes. Lender gains 1 unit of credit; borrower gains 1 unit of debt; no assets transfer.
4. Step 3 repeats indefinitely. Total units filled grows without bound while `consumed` stays at `N`.

**Why existing checks fail:** The `ConsumedAssets` guard at line 369 only enforces an asset-denominated cap. It cannot detect unit-term overfilling when `buyerAssets` rounds to zero. There is no guard requiring `buyerAssets > 0` when `units > 0`. [3](#0-2) 

**Protocol acknowledgment:** The NatSpec at line 94 explicitly states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) demonstrates and asserts this exact behavior — confirming the team is aware but has not fixed it. [4](#0-3) [5](#0-4) 

### Impact Explanation
The maker's buy offer is overfilled in unit terms: after `consumed == maxAssets`, a taker can continue filling the offer with nonzero `units` at zero asset cost. The maker (lender) accrues credit without paying assets; the taker (borrower) accrues debt without receiving assets. `totalUnits` and individual credit/debt balances increase beyond what the maker's `maxAssets` cap was intended to allow. The `EventsLib.Take` event emits `consumed = maxAssets` on every such call, misleading off-chain systems into believing the offer is fully consumed and no further fills occurred. [6](#0-5) 

### Likelihood Explanation
Preconditions: `offer.buy = true`, `offer.maxAssets > 0`, `tick > MAX_TICK/2` (i.e., `buyerPrice < WAD`). These are normal, reachable market parameters — `MAX_TICK = 5820` and ticks above 2910 are valid and common. The taker must be willing to call `take()` with small `units`; since `buyerAssets = 0`, the taker pays nothing and receives nothing in assets but does accrue debt. The attack is repeatable with no gas-cost barrier beyond transaction fees. [2](#0-1) 

### Recommendation
Add a guard that prevents a nonzero `units` take from producing zero `buyerAssets` on a buy offer:

```solidity
// After computing buyerAssets / sellerAssets:
require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
```

Alternatively, when `offer.maxAssets > 0` and `offer.buy = true`, enforce the unit-denominated cap directly: `require(newConsumedUnits <= maxAssets.mulDivDown(WAD, buyerPrice))`, or require `buyerPrice >= WAD` for `maxAssets`-capped buy offers.

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already proves the finding. A minimal fuzz extension:

```solidity
function testFuzz_BuyMaxAssetsUnitOverfill(uint256 extraUnits) public {
    extraUnits = bound(extraUnits, 1, 1000);

    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    // Fully consume the offer in asset terms.
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, 1, lender);

    collateralize(market, borrower, extraUnits);

    uint256 debtBefore = midnight.debtOf(id, borrower);

    for (uint256 i; i < extraUnits; i++) {
        take(1, borrower, lenderOffer);
    }

    // Assert: total units filled > maxAssets / buyerPrice (which is 1 / price ≈ 1)
    assertGt(midnight.debtOf(id, borrower) - debtBefore, 0, "units overfilled");
    // consumed never exceeded maxAssets
    assertEq(midnight.consumed(lender, lenderOffer.group), 1);
}
```

Expected: test passes, demonstrating that `extraUnits` of debt are created while `consumed` stays at `maxAssets = 1`. [5](#0-4)
