### Title
Fee-on-Transfer Loan Token Causes `claimableSettlementFee` Overcount, Making Settlement Fee Unclaimable and Blocking Lender Withdrawals - (File: src/libraries/SafeTransferLib.sol / src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee` is incremented by the full nominal spread `buyerAssets - sellerAssets` at line 418 before the two `safeTransferFrom` calls execute. `SafeTransferLib.safeTransferFrom` only verifies the call succeeds and returns `true`; it does not check the amount actually credited to the destination. When the loan token is fee-on-transfer, the contract receives only `(buyerAssets - sellerAssets) * (1 - fee_rate)` from the first transfer, creating a permanent shortfall equal to `(buyerAssets - sellerAssets) * fee_rate` per `take`. This violates the core solvency invariant `contract_balance >= collateralSum + withdrawableSum + claimableSettlementFee`.

### Finding Description

**Exact code path:**

`src/Midnight.sol` line 418 — state write before transfers:
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

`src/Midnight.sol` lines 455–456 — the two transfers that fund the above increment:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
``` [2](#0-1) 

`src/libraries/SafeTransferLib.sol` lines 24–34 — `safeTransferFrom` only checks call success and return value, never the received balance delta: [3](#0-2) 

**Root cause:** The accounting increment at line 418 uses the nominal transfer amount, not the amount actually received. `SafeTransferLib` has no balance-before/balance-after guard.

**Attacker-controlled inputs:**
- Any unprivileged taker can call `take()` with an `offer.market.loanToken` that is a fee-on-transfer ERC-20. Markets are created permissionlessly via `touchMarket(offer.market)` inside `take()`, so no privileged setup is required.

**Exploit flow:**
1. Deploy or use an existing fee-on-transfer ERC-20 as `loanToken` (e.g., 1% fee on every transfer).
2. Construct an `Offer` referencing a market with that `loanToken`.
3. Call `take()` as an unprivileged taker.
4. At line 418: `claimableSettlementFee[token] += S` where `S = buyerAssets - sellerAssets`.
5. At line 455: contract receives `S * 0.99` (1% fee deducted by the token).
6. At line 456: `receiver` receives `sellerAssets * 0.99`; this does not affect the contract's balance.
7. Net: `claimableSettlementFee` overstates the contract's actual holdings by `S * 0.01` per `take`.
8. After N takes, the shortfall accumulates to `N * S * fee_rate`.

**Why existing checks fail:**
- `SafeTransferLib.safeTransferFrom` only asserts `success == true` and `returndata == true`; it cannot detect a fee-on-transfer deduction.
- The Certora `Solvency.spec` formal proof explicitly assumes standard (non-fee) tokens at line 31–33: *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver"* — so the `tokenBalanceCorrect` strong invariant is not proven for fee-on-transfer tokens.
- `claimSettlementFee` at line 307 performs `claimableSettlementFee[token] -= amount` then calls `safeTransfer(token, receiver, amount)`. If `amount` exceeds the actual contract balance, the `safeTransfer` reverts. [4](#0-3) [5](#0-4) 

### Impact Explanation
`claimableSettlementFee[token]` exceeds the contract's actual token balance attributable to fees. The `feeClaimer` calling `claimSettlementFee` for the full recorded amount will revert because the contract cannot transfer what it does not hold. If the accumulated shortfall also erodes the balance below `withdrawableSum`, lender `withdraw()` calls will revert as well, since both draw from the same token balance.

### Likelihood Explanation
**Preconditions:** The loan token must be fee-on-transfer. Such tokens exist in production (e.g., tokens with configurable transfer fees). Any market creator or taker can choose the loan token; no privileged role is required. The condition is repeatable: every `take()` on such a market deepens the shortfall. The fee rate need not be large — even 0.01% compounding over many takes produces an unclaimable gap.

### Recommendation
Record the actual received amount rather than the nominal amount. Before and after the first `safeTransferFrom` to `address(this)`, snapshot the contract's token balance and use the delta to increment `claimableSettlementFee`:

```solidity
uint256 balBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 received = IERC20(offer.market.loanToken).balanceOf(address(this)) - balBefore;
claimableSettlementFee[offer.market.loanToken] += received;
```

Alternatively, document that fee-on-transfer tokens are explicitly unsupported and add a revert guard (e.g., require `received == buyerAssets - sellerAssets`). The increment at line 418 must be moved to after the transfer and use the measured delta.

### Proof of Concept

**Foundry fuzz/invariant test plan:**

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";

// Minimal fee-on-transfer ERC-20: deducts `feeBps` basis points on every transferFrom/transfer.
contract FeeToken is ERC20 {
    uint256 public feeBps; // e.g. 100 = 1%
    constructor(uint256 _feeBps) ERC20("FeeToken","FT") { feeBps = _feeBps; }
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount * feeBps / 10000;
        super.transferFrom(from, to, amount - fee); // recipient gets amount - fee
        _burn(from, fee);                           // fee is burned / taken
        return true;
    }
}

contract FeeOnTransferSettlementFeeTest is Test {
    Midnight midnight;
    FeeToken token;

    function setUp() public { /* deploy midnight, token, create market, fund actors */ }

    // Fuzz: feeBps in [1, 1000], N takes in [1, 50]
    function testFuzz_settlementFeeShortfall(uint256 feeBps, uint256 N) public {
        feeBps = bound(feeBps, 1, 1000);
        N = bound(N, 1, 50);
        token = new FeeToken(feeBps);
        // ... setup market with token as loanToken, fund taker ...

        for (uint i = 0; i < N; i++) {
            midnight.take(/* offer, ... */);
        }

        uint256 claimable = midnight.claimableSettlementFee(address(token));
        uint256 actualBalance = token.balanceOf(address(midnight));

        // ASSERTION 1: invariant violated — contract holds less than claimable fee
        assertGt(claimable, actualBalance, "claimable > actual balance: invariant broken");

        // ASSERTION 2: feeClaimer cannot claim full amount
        vm.prank(feeClaimer);
        vm.expectRevert();
        midnight.claimSettlementFee(address(token), claimable, feeClaimer);
    }
}
```

**Expected assertions:**
- `claimableSettlementFee(token) > token.balanceOf(address(midnight))` after N takes with any `feeBps > 0`.
- `claimSettlementFee(token, claimable, receiver)` reverts for the full recorded amount.
- The shortfall scales linearly with `feeBps * N * spread`, confirming repeatability.
