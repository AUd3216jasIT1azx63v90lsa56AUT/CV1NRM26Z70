Audit Report

## Title
TOCTOU Stale `consumableUnits` Read Causes Bundler to Skip Partially-Filled Offers and Revert with `OutOfOffers` - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary
`ConsumableUnitsLib.consumableUnits` reads the live `consumed[maker][group]` mapping and computes remaining offer capacity as a snapshot. All four `MidnightBundles` entry-points use this snapshot to compute `unitsToTake` before calling `take()`. If any taker fills even 1 unit of the same offer between the snapshot read and the `take()` call, `unitsToTake` will exceed actual remaining capacity, causing `take()` to revert with `ConsumedUnits` or `ConsumedAssets`. The broad `catch {}` block silently drops the offer entirely — even though it retains remaining capacity — and if no subsequent offers cover the shortfall, the bundle reverts with `OutOfOffers`.

## Finding Description

**Root cause — stale snapshot in `ConsumableUnitsLib.consumableUnits`:**

`src/periphery/ConsumableUnitsLib.sol` lines 15–17 read the live `consumed` mapping and compute remaining capacity as a point-in-time snapshot:

```solidity
uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
if (offer.maxUnits > 0) {
    return offer.maxUnits.zeroFloorSub(consumed);   // snapshot, not atomic
``` [1](#0-0) 

`src/periphery/MidnightBundles.sol` lines 74–78 use this snapshot to compute `unitsToTake`:

```solidity
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
``` [2](#0-1) 

`src/Midnight.sol` lines 371–372 then increment `consumed` **before** enforcing the cap:

```solidity
newConsumed = consumed[offer.maker][offer.group] += units;
require(newConsumed <= offer.maxUnits, ConsumedUnits());
``` [3](#0-2) 

If `consumed` increased between the `consumableUnits` read and this line, `newConsumed` overshoots `maxUnits` and reverts. The bundler's `catch {}` at `MidnightBundles.sol` lines 85, 160, 221, and 300 swallows the revert and skips the offer with no retry: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

`filledUnits` is not incremented, and the loop moves on. If no remaining offers cover the shortfall, `MidnightBundles.sol` line 88 reverts:

```solidity
require(filledUnits == targetUnits, OutOfOffers());
``` [8](#0-7) 

**Exploit flow (`maxUnits = 100`, `consumed = 0`, `targetUnits = 100`, single offer):**

1. Victim submits bundle via `buyWithUnitsTargetAndWithdrawCollateral`.
2. `consumableUnits` reads `consumed = 0`, returns `100`. `unitsToTake = 100`.
3. Attacker (any unprivileged taker on an ungated offer) front-runs with `take(offer, ..., 1, ...)` → `consumed` becomes `1`.
4. Victim's `take(offer, ..., 100, ...)` executes: `newConsumed = 1 + 100 = 101 > 100` → reverts `ConsumedUnits`.
5. `catch {}` swallows the revert; `filledUnits` stays `0`.
6. Loop ends. `require(0 == 100, OutOfOffers())` → bundle reverts.

The offer still had 99 units of remaining capacity. The same TOCTOU applies to the `maxAssets` branch (`ConsumedAssets`) and to all four bundle entry-points. [9](#0-8) 

**Why existing checks do not stop it:**

The NatSpec on all four entry-points explicitly states: *"Skips every reason why take can revert (including ones that are not asynchrony related)"*, confirming the `catch {}` is intentionally broad. There is no retry with a refreshed `consumableUnits` value after a failed `take()`. The attacker only needs to fill **1 unit** to shift `consumed` enough that `unitsToTake` (computed from stale state) overshoots the remaining capacity. [10](#0-9) 

## Impact Explanation
Any unprivileged taker can grief a victim's bundle by front-running with a minimal fill of any offer in the bundle's `takes` array. The victim's transaction reverts with `OutOfOffers`, their intended borrow or lend does not execute, and gas is wasted. The attack is repeatable: the attacker can re-front-run every retry the victim makes, permanently blocking the operation as long as the attacker is willing to pay gas. This constitutes a concrete, repeatable denial-of-service against the bundler's core functionality across all four entry-points. [11](#0-10) 

## Likelihood Explanation
Preconditions are minimal: the attacker must observe the pending bundle transaction (standard mempool visibility) and must be authorized to call `take()` on the same offer. For offers with no `enterGate` (`address(0)`), any address qualifies — the only restriction is `offer.maker != taker`. The attacker's cost is one `take()` call for 1 unit. The victim's bundle fails entirely. The attack is repeatable at low cost and requires no privileged access. [12](#0-11) [13](#0-12) 

## Recommendation
After a failed `take()`, instead of skipping the offer entirely, the bundler should re-read `consumableUnits` and retry with the refreshed remaining capacity before moving to the next offer. Concretely, the `catch {}` block should re-invoke `ConsumableUnitsLib.consumableUnits` and, if the refreshed value is nonzero and still covers the shortfall, attempt `take()` again with the updated `unitsToTake`. Alternatively, the bundler can be redesigned to pass `type(uint256).max` as `units` and let `Midnight.sol` enforce the cap atomically, then read back the actual units filled from the return value — though this requires a protocol-level change to support partial fills. At minimum, the `catch {}` block should distinguish between `ConsumedUnits`/`ConsumedAssets` (retriable with refreshed state) and other terminal errors (non-retriable), rather than treating all reverts identically. [14](#0-13) 

## Proof of Concept
**Minimal fork test plan:**

1. Deploy `Midnight` and `MidnightBundles` on a local fork.
2. Maker creates a sell offer with `maxUnits = 100`, no `enterGate`.
3. Victim calls `buyWithUnitsTargetAndWithdrawCollateral` with `targetUnits = 100`, `takes = [{offer, units: 100}]`.
4. Before the victim's transaction is mined, attacker calls `take(offer, ..., 1, ...)` in the same block (simulate via `vm.prank` + ordering in a Foundry test).
5. Mine both transactions with attacker's `take` first.
6. Assert victim's transaction reverts with `OutOfOffers`.
7. Assert `consumed[maker][group] == 1` (offer still has 99 units remaining, confirming the DoS is not due to offer exhaustion).

The same test applies to `supplyCollateralAndSellWithUnitsTarget`, `buyWithAssetsTargetAndWithdrawCollateral`, and `supplyCollateralAndSellWithAssetsTarget` by substituting the appropriate entry-point and adjusting the target/fill parameters. [15](#0-14) [16](#0-15)

        uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
        if (offer.maxUnits > 0) {
            return offer.maxUnits.zeroFloorSub(consumed);
