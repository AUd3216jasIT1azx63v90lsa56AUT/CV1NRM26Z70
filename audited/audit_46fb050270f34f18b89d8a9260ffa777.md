Audit Report

## Title
Zero-`buyerAssets` rounding bypasses `maxAssets` cap, allowing unlimited free position mutations - (`src/Midnight.sol`)

## Summary
In `take()`, `consumed` is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`, which truncates to zero whenever `units * buyerPrice < WAD`. Because position state (`credit`, `debt`, `totalUnits`) is updated using `units` directly, a taker can call `take` repeatedly with small `units` values, mutating the maker's credit and taker's debt on every call while `consumed` never advances, bypassing the `maxAssets` cap entirely and transferring zero tokens.

## Finding Description
In `src/Midnight.sol`, the `take()` function computes:

- **Line 363**: `buyerAssets = units.mulDivDown(buyerPrice, WAD)` — truncates to `0` when `units * buyerPrice < WAD`.
- **Line 368**: `consumed[offer.maker][offer.group] += buyerAssets` — adds `0`, leaving `consumed` unchanged.
- **Line 369**: `require(newConsumed <= offer.maxAssets)` — trivially passes since `newConsumed == old`.
- **Line 382**: `buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)` — uses `units`, not `buyerAssets`.
- **Line 410**: `buyerPos.credit += buyerCreditIncrease` — maker credit grows by `units`.
- **Line 414**: `sellerPos.debt += sellerDebtIncrease` — taker debt grows by `units`.
- **Line 455**: `safeTransferFrom(..., buyerAssets - sellerAssets)` — transfers `0` tokens.

The root cause is a mismatch: `consumed` is asset-denominated but position mutations are unit-denominated. No guard enforces `buyerAssets > 0` before proceeding.

**Price conditions**: `tickToPrice` rounds to multiples of `PRICE_ROUNDING_STEP = 1e12`. For any tick where `price < WAD` (i.e., all ticks except `MAX_TICK`), choosing `units = 1` gives `1 * price / 1e18 = 0`. For `tick = MAX_TICK - 16`, price is just below `WAD`, so `units = 1` suffices.

**Existing checks that fail**:
- `require(newConsumed <= offer.maxAssets)` — passes because `+= 0`.
- `require(atMostOneNonZero(maxAssets, maxUnits))` — validates offer structure only, not fill size.
- No `require(units > 0)` or `require(buyerAssets > 0)` guard exists.

**Confirmed by `testBugBuyMaxAssetsBypass`** (`test/TakeTest.sol`, lines 857–889): the test pre-fills `consumed` to `maxAssets` via `setConsumed`, then calls `take(1, borrower, lenderOffer)`. It asserts `buyerAssets == 0`, `consumed` unchanged at `maxAssets`, token balances unchanged, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.

## Impact Explanation
A taker can fill a buy offer with `maxAssets > 0` an unlimited number of times at zero token cost. Each fill increases the maker's credit and the taker's debt by `units` without any token transfer and without advancing `consumed`. This violates the core accounting invariant that every credit increase must correspond to valid consumed accounting, and that offers cannot be overfilled or reused. The taker accrues unbounded debt obligations and the maker gains unbounded credit — both without any economic exchange — which can be used to manipulate market state, drain settlement fee accounting, or exploit downstream liquidation/settlement logic.

## Likelihood Explanation
Preconditions are minimal: (1) a buy offer with `maxAssets > 0` exists (standard offer type); (2) `tickToPrice(offer.tick) < WAD` — true for every valid tick except `MAX_TICK`; (3) taker uses `units = 1`. No privileged access, oracle manipulation, or front-running is required. The attack is executable in a single transaction via `multicall` and is trivially repeatable.

## Recommendation
Add a guard in `take()` that rejects fills where the computed asset amount rounds to zero:

```solidity
// After computing buyerAssets / sellerAssets:
if (offer.maxAssets > 0) {
    require(buyerAssets > 0, ZeroBuyerAssets());
}
```

Alternatively, enforce `require(units > 0)` and additionally require that `buyerAssets > 0` whenever `offer.maxAssets > 0`, ensuring the asset-denominated cap always advances on every fill.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) is a complete, passing proof of concept. It demonstrates:
1. `consumed` pre-filled to `maxAssets`.
2. `take(1, borrower, lenderOffer)` succeeds with `tick = MAX_TICK - 16`.
3. `buyerAssets == 0` (assertion line 878).
4. `consumed` still equals `maxAssets` — cap not advanced (assertion line 882).
5. Token balances unchanged (assertions lines 883–884).

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
