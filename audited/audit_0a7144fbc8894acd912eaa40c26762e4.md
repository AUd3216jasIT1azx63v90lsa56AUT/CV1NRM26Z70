### Title
`mulDivUp` rounding in `postSlashPendingFee` violates proportional net-credit invariant after bad-debt slash - (`src/Midnight.sol`)

### Summary
In `updatePositionView` at line 810, the formula `postSlashPendingFee = _pendingFee - _pendingFee.mulDivUp(credit - postSlashCredit, credit)` uses ceiling division for the fee reduction, which rounds the remaining `postSlashPendingFee` **down** by up to 1 unit below the exact proportional value `pendingFee * postSlashCredit / credit`. This causes the lender's net credit (`postSlashCredit - postSlashPendingFee`) to exceed `(credit - pendingFee) * postSlashCredit / credit` by up to 1 unit, violating the proportional-loss invariant. The lender avoids paying up to 1 unit of continuous fees per slash event, redeeming more net assets than economically owed.

### Finding Description
**Exact code path:**

`liquidate` at lines 631–634 increases `marketState[id].lossFactor`: [1](#0-0) 

`updatePositionView` at lines 805–810 then computes: [2](#0-1) 

- `postSlashCredit = credit.mulDivDown(MAX - newLossFactor, MAX - lastLossFactor)` — rounds **down**, correct.
- `postSlashPendingFee = pendingFee - pendingFee.mulDivUp(credit - postSlashCredit, credit)` — rounds **up** the subtracted term, so the remaining `postSlashPendingFee` is rounded **down**.

**Mathematical proof of invariant violation:**

Let `C = credit`, `P = pendingFee`, `S = postSlashCredit`.

Ideal proportional value: `postSlashPendingFee_ideal = P * S / C`

Since `mulDivUp` rounds up: `mulDivUp(P, C-S, C) >= P*(C-S)/C`

Therefore: `postSlashPendingFee_actual = P - mulDivUp(P, C-S, C) <= P*S/C = postSlashPendingFee_ideal`

Net credit post-slash: `S - postSlashPendingFee_actual >= S - P*S/C = (C-P)*S/C`

The invariant requires `S - postSlashPendingFee <= (C-P)*S/C`, but the actual value is **≥** this bound whenever `P*(C-S)` is not exactly divisible by `C`.

**Concrete counterexample (C=3, P=2, S=1):**
- `mulDivUp(2, 2, 3) = ceil(4/3) = 2`
- `postSlashPendingFee = 2 - 2 = 0`
- Net credit post-slash: `1 - 0 = 1`
- Economically owed: `(3-2)*1/3 = 0` (integer) or `1/3` (exact)
- Invariant: `1 <= 0`? **VIOLATED**

**Why existing checks do not stop it:**

The Certora invariant `pendingContinuousFeeBoundedByCredit` only checks `pendingFee <= credit` [3](#0-2) 

and `noRemainingContinuousFeeWithoutCredit` only checks `credit == 0 => pendingFee == 0`. [4](#0-3) 

Neither checks the proportional relationship between `postSlashPendingFee` and `postSlashCredit`. The `mulDivUp` is used deliberately to ensure `postSlashPendingFee <= postSlashCredit` (preventing underflow in `postSlashCredit - fee` at line 818), but this safety choice is what causes the invariant violation. [5](#0-4) 

The `UtilsLib.mulDivUp` implementation confirms it rounds up by exactly `(d-1)/d < 1`: [6](#0-5) 

### Impact Explanation
After a bad-debt slash, the lender's `postSlashPendingFee` is reduced by up to 1 unit more than proportional. This means the lender's net credit (`postSlashCredit - postSlashPendingFee`) exceeds the economically owed amount `(credit - pendingFee) * (1 - lossFactorDelta)` by up to 1 unit. Over the remaining market lifetime, the lender accrues fewer fees and redeems up to 1 unit more loan token than economically owed per slash event. The impact is bounded at **≤ 1 unit per slash event** — a rounding-level discrepancy, not an unbounded drain. The scoped invariant is concretely violated and a fuzz test will reproduce it, but the absolute asset leakage is capped at 1 unit.

### Likelihood Explanation
The violation occurs whenever `pendingFee * (credit - postSlashCredit)` is not exactly divisible by `credit`, which is the generic case for any non-trivial credit/pendingFee combination. Any lender with a non-zero `pendingFee` in a market that realizes bad debt will trigger this. It is repeatable across every slash event and requires no special attacker setup — the lender simply holds credit with a pending fee when a bad-debt liquidation occurs.

### Recommendation
Replace `mulDivUp` with `mulDivDown` for the fee reduction and clamp the result to `postSlashCredit` to prevent underflow:

```solidity
uint256 postSlashPendingFee = credit > 0
    ? UtilsLib.min(
        _pendingFee - _pendingFee.mulDivDown(credit - postSlashCredit, credit),
        postSlashCredit
      )
    : 0;
```

`mulDivDown` rounds the fee reduction **down**, so the remaining `postSlashPendingFee` is rounded **up** (≥ proportional), which favors the fee collector and preserves the invariant. The `min(..., postSlashCredit)` clamp handles the at-most-1-unit overshoot that `mulDivDown` can produce, preventing the underflow that motivated the original `mulDivUp`.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {UtilsLib} from "src/libraries/UtilsLib.sol";

contract PostSlashPendingFeeInvariantTest is Test {
    using UtilsLib for uint256;

    // Unit test: concrete counterexample
    function testInvariantViolationConcrete() public pure {
        uint256 C = 3;
        uint256 P = 2;
        uint256 S = 1; // postSlashCredit

        uint256 postSlashPendingFee = P - P.mulDivUp(C - S, C);
        uint256 netCreditPostSlash = S - postSlashPendingFee;
        uint256 economicallyOwed = (C - P) * S / C; // = 0 in integer arithmetic

        // Asserts: 1 <= 0 — FAILS, proving the invariant is violated
        assertLe(netCreditPostSlash, economicallyOwed, "invariant: net_post <= (C-P)*S/C");
    }

    // Fuzz test: finds all violating (C, P, S) triples
    function testFuzzInvariantViolation(uint128 C, uint128 P, uint128 S) public pure {
        vm.assume(C > 0 && P <= C && S <= C);

        uint256 postSlashPendingFee = P - uint256(P).mulDivUp(C - S, C);
        uint256 netCreditPostSlash = S - postSlashPendingFee;
        // Exact rational bound: (C-P)*S/C
        // In integer arithmetic, check: netCreditPostSlash * C <= (C-P) * S
        assertLe(
            netCreditPostSlash * uint256(C),
            uint256(C - P) * uint256(S),
            "invariant violated: lender net credit exceeds proportional bound"
        );
    }
}
```

Expected result: `testInvariantViolationConcrete` fails immediately; `testFuzzInvariantViolation` finds counterexamples such as `(C=3, P=2, S=1)`, `(C=5, P=3, S=2)`, etc., confirming the invariant is systematically violated whenever `P*(C-S) mod C != 0`.
