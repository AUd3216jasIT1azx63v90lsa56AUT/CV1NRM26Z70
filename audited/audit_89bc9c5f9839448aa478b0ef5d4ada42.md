### Title
Fee-on-Transfer `loanToken` Causes `claimableSettlementFee` Overcount Leading to Insolvency - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee` is incremented by the full face value `buyerAssets - sellerAssets` before the inbound transfer executes. When `loanToken` is a fee-on-transfer ERC20, only `(buyerAssets - sellerAssets) * (1 - fee%)` actually arrives at `address(this)`, creating a permanent shortfall. Subsequent calls to `claimSettlementFee()` will attempt to transfer more tokens than the contract holds, reverting and freezing the fee or draining funds from `withdrawable` reserves.

### Finding Description

**Exact code path:**

`src/Midnight.sol` line 418 records the full settlement fee before any transfer:

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

Then lines 455–456 execute the two transfers:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only checks for revert and false-return; it does not verify the amount actually credited to the recipient: [3](#0-2) 

**Root cause:** No balance-before/after check surrounds the inbound transfer on line 455. With a 1% fee-on-transfer token, the contract receives `(buyerAssets - sellerAssets) * 0.99` but `claimableSettlementFee` is incremented by the full `buyerAssets - sellerAssets`.

**Attacker-controlled inputs:**
- Any unprivileged actor (market creator / taker) deploys or uses an existing fee-on-transfer ERC20 as `loanToken`.
- The market is created permissionlessly via `touchMarket` (called inside `take()`).
- The settlement fee being nonzero is a precondition: it is set by the `feeSetter` for a given `loanToken` via `setDefaultSettlementFee`. Once set, any taker calling `take()` on a market using that token triggers the accounting gap. This is a standard privileged-setup → user-triggered exploit path.

**Why existing checks fail:**
- `SafeTransferLib` validates success/return value only, not received amount.
- The Certora `Solvency.spec` `tokenBalanceCorrect` invariant and `pendingFeeReceiptZero` weak invariant are proved under a CVL model where `CVL_transferFrom` delivers the full requested amount; fee-on-transfer behavior is not modelled, so the formal proof does not cover this case. [4](#0-3) 

### Impact Explanation

After a single `take()` with a 1% fee-on-transfer `loanToken` and nonzero settlement fee `F = buyerAssets - sellerAssets`:

- `claimableSettlementFee[loanToken]` increases by `F`
- Contract balance increases by `F * 0.99`
- Shortfall: `F * 0.01`

`claimSettlementFee()` subtracts from `claimableSettlementFee` and calls `safeTransfer` for the full recorded amount: [5](#0-4) 

If the shortfall is not covered by other deposits, `safeTransfer` reverts (fee claimer cannot claim), or it drains tokens that belong to `withdrawable` lenders, breaking the core solvency invariant: `balance >= collateralSum + withdrawableSum + claimableSettlementFee`.

### Likelihood Explanation

**Preconditions:**
1. `loanToken` is a fee-on-transfer ERC20 — such tokens exist in production (e.g., USDT with fee enabled, STA, PAXG).
2. `feeSetter` has set a nonzero settlement fee for that token — a routine protocol operation.
3. Any taker calls `take()` with nonzero `units` at a tick where `buyerAssets > sellerAssets`.

All three are reachable without any privileged attacker action after setup. The exploit is repeatable on every `take()` call, compounding the shortfall linearly.

### Recommendation

Measure the actual amount received by comparing the contract's balance before and after the inbound transfer on line 455, and credit only the delta to `claimableSettlementFee`:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 received = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += received; // move line 418 here, use `received`
```

Alternatively, explicitly document that fee-on-transfer tokens are unsupported as `loanToken` and add a token validation check at market creation time.

### Proof of Concept

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IERC20} from "src/interfaces/IERC20.sol";

// Minimal fee-on-transfer ERC20: charges 1% on every transferFrom
contract FeeToken is ERC20 {
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount / 100;
        super.transferFrom(from, to, amount - fee); // recipient gets 99%
        _burn(from, fee);                           // 1% destroyed
        return true;
    }
}

contract FeeOnTransferPoC is Test {
    Midnight midnight;
    FeeToken feeToken;
    address feeSetter = address(0xFEE5);
    address feeClaimer = address(0xC1A1);
    address lender = address(0xA);
    address borrower = address(0xB);

    function setUp() public {
        midnight = new Midnight();
        feeToken = new FeeToken();
        midnight.setFeeSetter(feeSetter);
        midnight.setFeeClaimer(feeClaimer);

        // feeSetter sets nonzero settlement fee for feeToken
        vm.prank(feeSetter);
        midnight.setDefaultSettlementFee(address(feeToken), 1, 1e13); // nonzero fee
    }

    function testFeeOnTransferOvercountsClaimableSettlementFee() public {
        // Setup market with feeToken as loanToken
        // ... (create market, collateralize borrower, fund lender)

        uint256 balBefore = feeToken.balanceOf(address(midnight));
        // taker calls take() — triggers the two safeTransferFrom calls
        // ... take(units, lender, borrowerOffer);

        uint256 claimable = midnight.claimableSettlementFee(address(feeToken));
        uint256 actualBalance = feeToken.balanceOf(address(midnight)) - balBefore;

        // ASSERTION: claimable > actual balance held for fees
        assertGt(claimable, actualBalance, "claimableSettlementFee overcounted");

        // ASSERTION: feeClaimer cannot claim full amount without draining withdrawable
        vm.prank(feeClaimer);
        vm.expectRevert(); // safeTransfer reverts if balance insufficient
        midnight.claimSettlementFee(address(feeToken), claimable, feeClaimer);
    }
}
```

Expected: `claimableSettlementFee(feeToken) > IERC20(feeToken).balanceOf(address(midnight))` after `take()`, and `claimSettlementFee()` for the full recorded amount reverts or drains lender-withdrawable funds.
