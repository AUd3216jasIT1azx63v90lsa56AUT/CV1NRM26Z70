### Title
Authorized Operator Can Redirect Seller's Loan Proceeds to Arbitrary Address via Unvalidated `receiverIfTakerIsSeller` - (File: src/Midnight.sol)

### Summary
In `Midnight.take()`, the `receiverIfTakerIsSeller` parameter is never validated against the taker's authorization. An operator authorized by a victim can call `take` on the victim's behalf with `receiverIfTakerIsSeller=attacker`, causing the victim's loan proceeds (`sellerAssets`) to be sent to the attacker while the victim's position accrues the corresponding debt.

### Finding Description
**Code path:**

`src/Midnight.sol` — `take()`:

- Line 346: `require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());`
  Only checks that `msg.sender` is authorized by `taker`. No check on `receiverIfTakerIsSeller`.

- Line 375: `(address buyer, address seller) = offer.buy ? (offer.maker, taker) : (taker, offer.maker);`
  When `offer.buy = true`, `taker` is the seller (borrower).

- Line 414: `sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);`
  Victim's debt is unconditionally increased.

- Line 422–423:
  ```solidity
  address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
  address receiver = offer.buy ? receiverIfTakerIsSeller : offer.receiverIfMakerIsSeller;
  ```
  `receiver` is set directly from the attacker-supplied `receiverIfTakerIsSeller`. `payer` is the buyer (lender/maker) when no buyerCallback.

- Line 456: `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);`
  `sellerAssets` (the loan proceeds owed to the seller/victim) are transferred to `receiver = attacker`.

**Exploit flow:**
1. Attacker obtains authorization from victim: `isAuthorized[victim][attacker] = true` (victim grants attacker as operator for any legitimate purpose).
2. A valid buy offer exists from a lender (`offer.buy = true`, `offer.maker = lender`).
3. Attacker calls `take(offer, ratifierData, units, taker=victim, receiverIfTakerIsSeller=attacker, address(0), hex"")`.
4. The `TakerUnauthorized` check passes because `isAuthorized[victim][attacker] = true`.
5. Victim's position gets `sellerDebtIncrease` added to debt (line 414).
6. `sellerAssets` (loan proceeds) are transferred from the lender to `attacker` (line 456).
7. Victim holds new debt with no corresponding assets received.

**Why existing checks fail:**
The only authorization check (line 346) validates that the caller is authorized to act as the taker. There is no check that `receiverIfTakerIsSeller` is the taker themselves or is authorized by the taker. The `receiverIfTakerIsSeller` parameter is consumed without any validation. ### Impact Explanation
A victim who has granted any operator authorization loses their loan proceeds: the operator forces the victim into debt while redirecting `sellerAssets` (the full loan amount minus settlement fee) to an attacker-controlled address. The victim's collateral remains locked against the new debt, and they receive nothing. This directly violates the invariant that user funds must not be stolen.

### Likelihood Explanation
**Preconditions:**
1. Victim has granted `isAuthorized[victim][attacker] = true` (e.g., for a DeFi integration, vault, or any other delegated operation).
2. A valid, ratified buy offer exists in the market (normal market condition).
3. Victim has sufficient collateral to pass the post-take health check (line 476), or the attacker uses `units` small enough that `sellerDebtIncrease == 0` (credit-reduction path via `sellerCreditDecrease`).

**Feasibility:** High. Authorization grants are a core protocol feature used by integrations. The attacker only needs to be authorized once and can drain proceeds on any subsequent take. Repeatable as long as the authorization holds and offers exist.

### Recommendation
Require that `receiverIfTakerIsSeller` is either the taker themselves or is explicitly authorized by the taker:

```solidity
require(
    receiverIfTakerIsSeller == taker || isAuthorized[taker][receiverIfTakerIsSeller],
    ReceiverUnauthorized()
);
```

This mirrors the pattern already used for `onBehalf`/`receiver` in `withdraw()` (line 482) and other functions, and ensures the taker controls where their proceeds land. [4](#0-3) 

### Proof of Concept

```solidity
function testOperatorStealsSellerProceeds() public {
    // Setup: victim has collateral, lender has a buy offer
    address victim   = makeAddr("victim");
    address attacker = makeAddr("attacker");
    uint256 units    = 1000e18;

    // Victim grants attacker as authorized operator
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, true, victim);

    // Victim supplies collateral so health check passes
    collateralize(market, victim, units);

    // Lender posts a buy offer
    Offer memory offer;
    offer.buy      = true;
    offer.maker    = lender;
    offer.ratifier = address(dummyRatifier);
    offer.maxUnits = units;
    offer.market   = market;
    offer.expiry   = block.timestamp + 200;
    offer.tick     = MAX_TICK;
    deal(address(loanToken), lender, units);
    vm.prank(lender);
    loanToken.approve(address(midnight), type(uint256).max);

    uint256 attackerBalanceBefore = loanToken.balanceOf(attacker);
    uint256 victimDebtBefore      = midnight.debtOf(id, victim);

    // Attacker takes the buy offer on behalf of victim,
    // redirecting sellerAssets to attacker
    vm.prank(attacker);
    midnight.take(offer, hex"", units, victim, attacker, address(0), hex"");

    // Assert: attacker received sellerAssets
    assertGt(loanToken.balanceOf(attacker), attackerBalanceBefore);
    // Assert: victim now holds debt
    assertGt(midnight.debtOf(id, victim), victimDebtBefore);
    // Assert: victim received nothing
    assertEq(loanToken.balanceOf(victim), 0);
}
```

**Expected assertions pass:** attacker's `loanToken` balance increases by `sellerAssets`; victim's debt increases by `units`; victim's token balance remains zero.
