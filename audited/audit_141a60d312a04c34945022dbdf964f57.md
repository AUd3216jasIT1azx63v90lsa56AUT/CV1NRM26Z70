### Title
Fee-on-Transfer Loan Token Causes `claimableSettlementFee` Overcount Leading to Insolvency - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee[loanToken]` is incremented by `buyerAssets - sellerAssets` before the inbound transfer executes. `SafeTransferLib.safeTransferFrom` only checks the boolean return value of `transferFrom`, not the actual amount received. If the loan token charges a transfer fee, the contract records more claimable settlement fee than it actually receives, violating the solvency invariant `tokenBalance >= collateralSum + withdrawableSum + claimableSettlementFee`.

### Finding Description

**Exact code path:**

`src/Midnight.sol` line 418 commits the accounting update unconditionally before the transfer:
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

Then lines 455–456 execute the actual transfers:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only validates that the call did not revert and that the return value (if any) is `true`. It performs no balance-before/balance-after check: [3](#0-2) 

For a fee-on-transfer token with fee rate `f`:
- `claimableSettlementFee[loanToken]` increases by `(buyerAssets - sellerAssets)`
- Contract actually receives only `(buyerAssets - sellerAssets) * (1 - f)`
- Overcount per take: `(buyerAssets - sellerAssets) * f`

**Why existing checks fail:**

The Certora formal verification of the solvency invariant `tokenBalanceCorrect` explicitly assumes tokens transfer correctly with no fee-taking:
```
// Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing...
function _.transferFrom(address src, address a, uint256 v) external with(env e) => CVL_transferFrom(...)
``` [4](#0-3) 

The `pendingFeeReceiptZero` weak invariant and `tokenBalanceCorrect` strong invariant are only proven under this assumption: [5](#0-4) 

The `claimSettlementFee` function transfers out whatever `claimableSettlementFee[token]` records, with no balance check: [6](#0-5) 

When the feeClaimer claims the overcounted amount, the shortfall is taken from tokens that back lender `withdrawable` balances, causing `withdraw()` to revert for lenders.

### Impact Explanation
The contract's actual loan token balance is less than `collateralSum + withdrawableSum + claimableSettlementFee`. When `claimSettlementFee` is called for the full overcounted amount, it transfers tokens that were deposited by lenders (via `repay`/`take`) to back their `withdrawable` credit. Subsequent lender `withdraw()` calls revert because the contract balance is insufficient, permanently freezing lender funds proportional to the cumulative fee shortfall.

### Likelihood Explanation
Preconditions: (1) a market exists with a fee-on-transfer loan token; (2) at least one `take()` is executed in that market. Fee-on-transfer tokens are a well-known ERC20 variant. The protocol imposes no restriction on which tokens can serve as `loanToken` in a market. Any taker can trigger the overcount on every `take()` call. The effect is cumulative and repeatable across multiple takes, amplifying the shortfall. The feeClaimer claiming fees is a routine protocol operation, not an adversarial act.

### Recommendation
Replace the two-transfer pattern with a balance-before/balance-after check for the fee portion transferred to `address(this)`:
```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived;
```
Alternatively, document and enforce at market creation that fee-on-transfer tokens are prohibited (e.g., via a whitelist or a creation-time balance check).

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Market, Offer} from "src/Midnight.sol";

contract FeeOnTransferToken {
    // 10% fee on transferFrom, returns true
    mapping(address => mapping(address => uint256)) public allowance;
    mapping(address => uint256) public balanceOf;
    uint256 constant FEE_BPS = 1000; // 10%

    function mint(address to, uint256 amount) external { balanceOf[to] += amount; }
    function approve(address spender, uint256 amount) external returns (bool) {
        allowance[msg.sender][spender] = amount; return true;
    }
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        allowance[from][msg.sender] -= amount;
        uint256 fee = amount * FEE_BPS / 10000;
        uint256 net = amount - fee;
        balanceOf[from] -= amount;
        balanceOf[to] += net;
        // fee burned / kept by token
        return true;
    }
    function transfer(address to, uint256 amount) external returns (bool) {
        balanceOf[msg.sender] -= amount;
        balanceOf[to] += amount;
        return true;
    }
}

contract FeeOnTransferTakeTest is Test {
    // Setup: deploy Midnight with FeeOnTransferToken as loanToken,
    // create market (admin sets tickSpacing), set up lender offer (buy=true),
    // set up borrower with collateral.

    function testClaimableSettlementFeeOvercounted() public {
        // ... setup omitted for brevity ...

        uint256 feeBefore = midnight.claimableSettlementFee(address(feeToken));
        uint256 balanceBefore = feeToken.balanceOf(address(midnight));

        // Taker (borrower) takes lender's buy offer
        vm.prank(taker);
        (uint256 buyerAssets, uint256 sellerAssets) = midnight.take(
            lenderBuyOffer, ratifierData, units, taker, taker, address(0), ""
        );

        uint256 feeAfter = midnight.claimableSettlementFee(address(feeToken));
        uint256 balanceAfter = feeToken.balanceOf(address(midnight));

        uint256 recorded = feeAfter - feeBefore;           // buyerAssets - sellerAssets
        uint256 actualReceived = balanceAfter - balanceBefore; // (buyerAssets - sellerAssets) * 0.9

        // ASSERTION 1: claimableSettlementFee is overcounted
        assertGt(recorded, actualReceived, "claimableSettlementFee overcounted");

        // ASSERTION 2: feeClaimer can claim overcounted amount (draining lender funds)
        vm.prank(feeClaimer);
        midnight.claimSettlementFee(address(feeToken), recorded, feeClaimer);

        // ASSERTION 3: lender withdraw now fails (insufficient balance)
        vm.prank(lender);
        vm.expectRevert();
        midnight.withdraw(lenderMarket, lenderCredit, lender, lender);
    }
}
```

**Expected assertions:**
- `recorded > actualReceived` — confirms overcount
- `claimSettlementFee` succeeds for the overcounted amount
- Lender `withdraw` reverts, confirming insolvency

// For any token, the pending settlement fee receipt after a transaction is 0: every claimableSettlementFee
// increment in take is paid back in by the same-function inbound transfer.
weak invariant pendingFeeReceiptZero(address token)
    pendingFeeReceipt[token] == 0;

// For any token, the balance of the contract is always greater than or equal to the sum of all collateral, withdrawable, and claimable settlement fee amounts for that token minus the flash loaned amount.
// Note: this invariant is strong, so it also holds before each external call.
strong invariant tokenBalanceCorrect(address token)
