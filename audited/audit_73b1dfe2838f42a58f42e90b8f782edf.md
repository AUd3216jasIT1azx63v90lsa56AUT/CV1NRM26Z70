Audit Report

## Title
Unconditional `safeTransfer` on destroyed collateral token permanently DoS-es all `liquidate()` paths - (`File: src/Midnight.sol` / `src/libraries/SafeTransferLib.sol`)

## Summary
`liquidate()` calls `SafeTransferLib.safeTransfer(collateralToken, receiver, seizedAssets)` at line 696 unconditionally — outside the `if (repaidUnits > 0 || seizedAssets > 0)` guard that closes at line 677. `SafeTransferLib.safeTransfer` checks `token.code.length > 0` before every call regardless of transfer value, so if the collateral token's bytecode has been removed, every invocation of `liquidate()` — including the documented zero-transfer bad-debt realization path — reverts with `NoCode()`. Because `touchMarket()` never validates that a collateral token has deployed code, an attacker can register a self-destructable token, open a borrow position, destroy the token, and permanently freeze all liquidation paths for every borrower in that market.

## Finding Description

**Root cause 1 — `touchMarket()` accepts any address as collateral token**

`touchMarket()` validates collateral tokens only for sort order, allowed LLTV tier, and valid `maxLif`: [1](#0-0) 

There is no `require(collateralToken.code.length > 0)` check. Any address — including one whose bytecode will later be erased — is accepted.

**Root cause 2 — `SafeTransferLib.safeTransfer` checks code existence unconditionally** [2](#0-1) 

The `require(token.code.length > 0, NoCode())` guard fires on every call, even when `value == 0`.

**Root cause 3 — The collateral transfer at line 696 is unconditional**

The `if (repaidUnits > 0 || seizedAssets > 0)` block closes at line 677: [3](#0-2) 

Line 696 is placed after this block and executes on every code path through `liquidate()`, including the documented "bad-debt realization with 0 tokens transferred" path (both inputs zero): [4](#0-3) 

**Exploit flow**

1. Attacker deploys a minimal ERC-20 with a `destroy()` function that calls `SELFDESTRUCT`.
2. Attacker calls `touchMarket()` with `collateralParams[0].token = address(selfDestructableToken)`. Market is created; no code check is performed.
3. Attacker (acting as both lender and borrower, or using a pre-existing lender offer) calls `supplyCollateral()` and `take()` to open a debt position. Token has code at this point, so `safeTransferFrom` succeeds.
4. Attacker calls `selfDestructableToken.destroy()`. On post-EIP-6780 chains, steps 1–4 must be atomic (token created and destroyed in the same transaction via a factory contract that deploys the token, creates the market, places a lender offer, supplies collateral, borrows, then self-destructs — all in one call). On pre-Cancun chains any `SELFDESTRUCT` removes the code.
5. `collateralToken.code.length == 0` from this point forward.
6. Any call to `liquidate(market, collateralIndex, *, *, borrower, *, *, *, *)` — regardless of `seizedAssets` and `repaidUnits` values — hits line 696 and reverts with `NoCode()`.

**Why existing checks fail**

- `touchMarket` has no `code.length` guard on collateral tokens.
- The `if (repaidUnits > 0 || seizedAssets > 0)` guard does not wrap line 696; the transfer is unconditional.
- There is no alternative liquidation path that skips the collateral `safeTransfer`.
- The `liquidationLocked` check and `NotLiquidatable` check are passed before the revert, confirming this is a hard transfer failure, not a false-positive health check.

## Impact Explanation
Every borrower in the affected market becomes permanently unliquidatable. Bad debt cannot be realized (the `seizedAssets = 0, repaidUnits = 0` path also reverts at line 696). Lender credit is permanently at risk because `withdrawable` can never be increased via liquidation and the `lossFactor` accounting for bad debt is never applied. The market is effectively bricked for all liquidation-dependent operations. The attacker retains the borrowed loan tokens while their position can never be force-closed.

## Likelihood Explanation
All preconditions are fully attacker-controlled: deploy a self-destructable token (no privilege required), call the permissionless `touchMarket`, supply collateral, borrow, then destroy the token. On post-EIP-6780 chains the self-destruct must occur in the same transaction as token creation, which is achievable with a single factory contract. The attack is one-shot and irreversible; once the token is destroyed the DoS is permanent. The attacker has direct financial incentive: they borrow against collateral they then render unliquidatable, keeping the borrowed assets while their position can never be closed by force.

## Recommendation
1. **Guard the collateral transfer**: wrap line 696 inside the existing `if (repaidUnits > 0 || seizedAssets > 0)` block, or add an explicit `if (seizedAssets > 0)` guard around the `safeTransfer` call. This ensures the zero-transfer bad-debt realization path does not attempt a token transfer.
2. **Validate collateral token code existence in `touchMarket()`**: add `require(collateralToken.code.length > 0)` inside the collateral params loop to prevent registration of non-contract addresses.

Both fixes are needed: fix (1) closes the immediate DoS vector for the zero-transfer path; fix (2) prevents the attack surface from being opened in the first place.

## Proof of Concept
**Minimal Foundry test plan:**
1. Deploy a `SelfDestructableERC20` with a `destroy()` function.
2. Call `midnight.touchMarket(market)` with this token as collateral.
3. Supply collateral and open a borrow position (using a pre-funded lender offer).
4. Call `selfDestructableToken.destroy()`.
5. Assert `address(selfDestructableToken).code.length == 0`.
6. Attempt `midnight.liquidate(market, 0, 0, 0, borrower, false, receiver, address(0), "")` — expect revert with `NoCode()`.
7. Attempt `midnight.liquidate(market, 0, 1e18, 0, borrower, false, receiver, address(0), "")` — expect revert with `NoCode()`.

Both calls revert, confirming permanent DoS of all liquidation paths including the zero-transfer bad-debt realization path documented at line 577–578. [5](#0-4)
