### Title
Permissionless `setIsAuthorized` Allows Attacker to Selectively Submit Pre-Signed Authorization Messages, Deauthorizing Repayment Bot - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` has no `msg.sender` restriction — any caller can submit a valid signed `Authorization`. Because nonces are strictly sequential, an attacker who holds multiple pre-signed `Authorization` messages (e.g., nonce=0 grant, nonce=1 revoke) can submit them in order and selectively stop at the revoke, leaving the victim's repayment bot deauthorized at a critical moment. The existing nonce, signature, and deadline checks do not prevent this because each individual message is cryptographically valid.

### Finding Description
**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` (lines 24–48):
- Line 25: `require(block.timestamp <= authorization.deadline, Expired())` — only checks expiry.
- Line 26: `require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce())` — enforces sequential nonces, increments on success.
- Lines 31–36: ecrecover + signer check — verifies the signature is from the authorizer (or their delegate).
- Line 47: calls `IMidnight.setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer)` — no restriction on who triggered this.

There is **no check on `msg.sender`**. This is confirmed by the existing test `testEcrecoverAuthorizerPermissionless` (line 74–86 of `test/SetIsAuthorizedWithSigTest.sol`), which explicitly documents "Anyone can submit — no caller auth needed."

**Root cause:** The permissionless submission model combined with sequential nonces means that any party holding N pre-signed `Authorization` messages can submit any prefix of them, choosing the final authorization state.

**Exploit flow:**
1. Victim pre-signs (via a relayer workflow or key delegation):
   - `Auth{nonce=0, isAuthorized=true}` — grants repayment bot
   - `Auth{nonce=1, isAuthorized=false}` — revokes repayment bot (intended for future use)
2. Attacker obtains both signed messages (e.g., from a public relayer mempool, off-chain storage, or intercepted broadcast).
3. Attacker submits nonce=0 (grant) → nonce=1 (revoke), stopping there.
4. `midnight.isAuthorized[victim][repaymentBot]` is now `false`.
5. Repayment bot calls `midnight.repay(market, units, victim, ...)` → reverts with `Unauthorized` (line 724 of `Midnight.sol`: `require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized())`).
6. Debt goes unrepaid; position accrues past maturity and becomes liquidatable.

**Why existing checks fail:**
- Nonce check: each message has the correct sequential nonce — passes.
- Signature check: each message is signed by the authorizer — passes.
- Deadline check: if deadlines are still valid — passes.
- No check exists to restrict who may submit a valid signed message.

### Impact Explanation
The repayment bot is deauthorized at a critical moment (e.g., near maturity). `midnight.repay` called by the bot reverts with `Unauthorized`. The borrower's debt goes unrepaid, the position crosses the health threshold, and it becomes liquidatable. The borrower can repay directly, but if the borrower is relying on an automated bot (the common use case for pre-signed authorizations), the position may be liquidated before manual intervention occurs.

### Likelihood Explanation
**Preconditions:**
1. Victim must have pre-signed multiple sequential `Authorization` messages (grant + revoke).
2. Attacker must have obtained both signed messages before the deadlines expire.
3. The victim must be relying on a bot for repayment rather than acting directly.

This is realistic in meta-transaction / relayer workflows where users pre-sign batches of authorization changes for future use. The attacker role is "signature user" — an unprivileged party who intercepts or is given signed messages. The attack is repeatable as long as unexpired pre-signed messages are available.

### Recommendation
Add a caller restriction so only the authorizer (or a pre-designated relayer) can submit the authorization:

```solidity
require(
    msg.sender == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, msg.sender),
    Unauthorized()
);
```

This preserves gasless delegation (the authorizer can designate a trusted relayer via `setIsAuthorized` on Midnight directly) while preventing arbitrary third parties from submitting pre-signed messages out of the victim's control.

Alternatively, add a `recipient` field to `Authorization` that restricts which address may submit the message, so the victim can bind each signed message to a specific trusted relayer.

### Proof of Concept

```solidity
// Foundry stateful test
function testGriefingAuthorizationOscillation() public {
    // Setup: borrower authorizes ecrecoverAuthorizer as a meta-tx relayer
    vm.prank(borrower);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

    // Victim pre-signs: nonce=0 (grant bot), nonce=1 (revoke bot)
    address bot = makeAddr("repaymentBot");
    Authorization memory grantAuth = Authorization({
        authorizer: borrower, authorized: bot,
        isAuthorized: true, nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Authorization memory revokeAuth = Authorization({
        authorizer: borrower, authorized: bot,
        isAuthorized: false, nonce: 1,
        deadline: block.timestamp + 1 days
    });
    Signature memory grantSig = signAuthorization(grantAuth, borrower);
    Signature memory revokeSig = signAuthorization(revokeAuth, borrower);

    // Attacker submits both — stops at revoke
    address attacker = makeAddr("attacker");
    vm.startPrank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(grantAuth, grantSig);   // nonce=0
    ecrecoverAuthorizer.setIsAuthorized(revokeAuth, revokeSig); // nonce=1
    vm.stopPrank();

    // Assert: bot is deauthorized despite victim intending to grant
    assertEq(midnight.isAuthorized(borrower, bot), false);
    assertEq(ecrecoverAuthorizer.nonce(borrower), 2);

    // Assert: bot cannot repay on behalf of borrower
    deal(address(loanToken), bot, 1000);
    vm.prank(bot);
    loanToken.approve(address(midnight), 1000);
    vm.prank(bot);
    vm.expectRevert(IMidnight.Unauthorized.selector);
    midnight.repay(market, 1000, borrower, address(0), hex"");
}
```

**Expected assertions:**
- `midnight.isAuthorized(borrower, bot) == false` after attacker submits both messages.
- `midnight.repay(...)` called by bot reverts with `Unauthorized`.
