Audit Report

## Title
Front-running of signature-based revocation via same-nonce grant submission â€” (`src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` is intentionally permissionless and enforces only sequential nonce ordering. When an authorizer A signs a grant `{nonce:N, isAuthorized:true}` and hands it to B off-chain, then later signs a revocation `{nonce:N, isAuthorized:false}` and broadcasts it, B can front-run the revocation by submitting the grant first. Whichever transaction lands first permanently consumes nonce N; the other reverts with `InvalidNonce`, leaving B authorized against A's explicit intent.

## Finding Description

**Root cause**

`setIsAuthorized` at `src/periphery/EcrecoverAuthorizer.sol` lines 24â€“26 has no `msg.sender` restriction and enforces only sequential nonce ordering:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

The `Authorization` struct at `src/periphery/interfaces/IEcrecoverAuthorizer.sol` lines 11â€“17 includes `isAuthorized` as a field in the EIP-712 hash, so `{nonce:N, isAuthorized:true}` and `{nonce:N, isAuthorized:false}` produce distinct digests requiring distinct signatures â€” but **both are valid for nonce N**. The permissionless design is confirmed by `testEcrecoverAuthorizerPermissionless` at `test/SetIsAuthorizedWithSigTest.sol` lines 74â€“86, which explicitly demonstrates that any caller can submit a valid signature.

**Exploit flow**

1. A signs `grant = {authorizer:A, authorized:B, isAuthorized:true, nonce:N, deadline:T}` and hands it to B for later submission. `nonce[A]` remains `N`.
2. A decides to revoke and signs `revoke = {authorizer:A, authorized:B, isAuthorized:false, nonce:N, deadline:T'}` and broadcasts it to the mempool.
3. B observes the mempool and front-runs with `grant` (nonce N, `isAuthorized:true`).
4. `grant` lands first: `nonce[A]` increments to `N+1`; `midnight.isAuthorized(A, B)` becomes `true`.
5. A's `revoke` executes: `authorization.nonce (N) != nonce[A] (N+1)` â†’ reverts `InvalidNonce`.

**Why existing checks fail**

- The deadline check (`block.timestamp <= authorization.deadline`) does not help: the grant's deadline is still in the future by precondition.
- The nonce check enforces ordering but does not prevent a competing same-nonce message from winning the race.
- There is no `msg.sender == authorization.authorizer` guard restricting who may submit a signature.

## Impact Explanation
B retains `midnight.isAuthorized(A, B) == true` after A explicitly signed and broadcast a revocation. Any protocol action gated on A's authorization of B remains available to B indefinitely. The attack is repeatable: each time A attempts revocation via a new signed message, B can front-run with the next unsubmitted grant for that nonce. This constitutes unauthorized privilege retention â€” matching the in-scope impact class "Authorization bypass leading to privileged action as unprivileged user" per RESEARCHER.md.

## Likelihood Explanation
Preconditions arise naturally from the contract's intended off-chain delegation flow: A signs a grant and gives it to B without submitting it, then later changes their mind. B is the natural attacker â€” they already hold the signed grant and have direct economic incentive to remain authorized. No privileged access, leaked keys, or external dependencies are required. The attack is executable by any party holding a valid unsubmitted grant signature, and is repeatable for every subsequent revocation attempt.

## Recommendation
Add a function allowing the authorizer to invalidate (skip) their current nonce directly, without requiring a competing signed message:

```solidity
function invalidateNonce() external {
    nonce[msg.sender]++;
}
```

This allows A to atomically consume nonce N before broadcasting a revocation, ensuring no prior-signed grant for nonce N can be submitted. Alternatively, adopt a bitmap-based nonce scheme (similar to Uniswap Permit2) that allows out-of-order invalidation of specific nonces, preventing any race between competing same-nonce messages.

## Proof of Concept
Extend `test/SetIsAuthorizedWithSigTest.sol` with the following test:

```solidity
function testFrontRunRevocation() public {
    vm.prank(borrower);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

    // Step 1: borrower signs grant (nonce 0) and gives to lender
    Authorization memory grant = makeAuthorization(borrower, lender, true);
    Signature memory grantSig = signAuthorization(grant, borrower);

    // Step 2: borrower signs revocation (same nonce 0) and broadcasts
    Authorization memory revoke = makeAuthorization(borrower, lender, false);
    Signature memory revokeSig = signAuthorization(revoke, borrower);

    // Step 3: lender front-runs by submitting grant first
    vm.prank(lender);
    ecrecoverAuthorizer.setIsAuthorized(grant, grantSig);
    assertEq(midnight.isAuthorized(borrower, lender), true);

    // Step 4: borrower's revocation fails with InvalidNonce
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(revoke, revokeSig);

    // lender remains authorized against borrower's explicit intent
    assertEq(midnight.isAuthorized(borrower, lender), true);
}
```
