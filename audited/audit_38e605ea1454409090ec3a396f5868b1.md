1. `safeTransfer(tokens[i], callback, assets[i])` — Midnight sends `A`, callback receives `A − fee_out`. Midnight balance: `B → B − A`.

3. `safeTransferFrom(tokens[i], callback, address(this), assets[i])` — callback sends `A`, Midnight receives `A − fee_in`. Midnight balance: `B − A → B − A + (A − fee_in) = B − fee_in`.

**Net loss to Midnight per flash loan = `fee_in` (the fee on the repayment leg), not `2*fee`.**

The `2*fee` claim in the question is incorrect from Midnight's balance perspective. The outgoing fee (`fee_out`) is borne by the callback (it receives less), not by Midnight. Midnight only loses `fee_in`.

For the `safeTransferFrom` in step 3 to succeed, the callback must hold `A` tokens. Since it only received `A − fee_out` in step 1, it must pre-fund `fee_out` tokens. This means the attacker pays `fee_out` per flash loan to drain `fee_in` from Midnight. If `fee_out = fee_in = fee` (symmetric fee), the attacker pays exactly as much as they drain — a pure griefing attack with no profit motive.

### Title
Flash loan repayment with fee-on-transfer token silently under-repays, draining Midnight's reserves — (`src/Midnight.sol`)

### Summary
`flashLoan` sends `assets[i]` tokens to the callback and then pulls `assets[i]` tokens back via `safeTransferFrom`, but performs no balance snapshot or post-repayment balance check. When the token charges a transfer fee, the repayment leg delivers `assets[i] − fee` to Midnight instead of `assets[i]`, causing a net loss of `fee` tokens per flash loan. Repeated calls drain Midnight's token reserves, eventually making lender withdrawals impossible.

### Finding Description
**Code path:** `src/Midnight.sol` `flashLoan` lines 737–752. [6](#0-5) 

**Root cause:** No pre/post balance invariant is enforced. The function only requires that `safeTransferFrom` does not revert; it never asserts `balanceAfter >= balanceBefore`. [7](#0-6) 

**Attacker inputs:**
- `tokens[0]` = a fee-on-transfer ERC20 (fee rate `r`, e.g. 1 %)
- `assets[0]` = `A` (any amount Midnight holds)
- `callback` = attacker-controlled contract implementing `IFlashLoanCallback`

**Exploit flow (one iteration, fee rate `r`, loan amount `A`):**

| Step | Action | Midnight balance | Callback balance |
|------|--------|-----------------|-----------------|
| Pre  | Attacker pre-funds callback with `A·r` | `B` | `A·r` |
| 1    | `safeTransfer(token, callback, A)` | `B − A` | `A·r + A·(1−r) = A` |
| 2    | `onFlashLoan` callback returns `CALLBACK_SUCCESS` | `B − A` | `A` |
| 3    | `safeTransferFrom(token, callback, midnight, A)` | `B − A + A·(1−r) = B − A·r` | `0` |

Midnight ends with `B − A·r`. The attacker spent `A·r` to drain `A·r` from Midnight (griefing). Repeating `N` times drains `N·A·r` from Midnight.

**Why checks fail:** `safeTransferFrom` succeeds (callback has `A` tokens), so no revert occurs. The Certora `tokenBalanceCorrect` invariant is proven only under the assumption of standard (non-fee) ERC20 tokens and does not apply here. [3](#0-2) 

### Impact Explanation
Midnight's token balance decreases by `fee_in = A·r` per flash loan call. Since `withdrawable`, `collateralSum`, and `claimableSettlementFee` accounting is unchanged, the actual balance falls below the sum of obligations tracked by the solvency invariant. Lenders attempting to `withdraw` will eventually receive reverts once the balance is exhausted.

### Likelihood Explanation
**Preconditions:** (1) A fee-on-transfer token is used as a loan token or collateral token in any Midnight market; (2) the attacker pre-funds `fee` tokens per iteration. Fee-on-transfer tokens exist in production (e.g. tokens with configurable fee switches). The attack is permissionless, requires no special role, and is repeatable indefinitely. The cost to the attacker equals the damage to Midnight (1:1 griefing ratio), making it economically viable for a motivated adversary.

### Recommendation
Record the balance of each token before the outgoing transfer and assert it is restored after the repayment:

```solidity
for (uint256 i = 0; i < tokens.length; i++) {
    uint256 balanceBefore = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
    // ... callback ...
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(IERC20(tokens[i]).balanceOf(address(this)) >= balanceBefore, InsufficientRepayment());
}
```

Alternatively, explicitly document and enforce (via a token allowlist or market creation check) that fee-on-transfer tokens are not supported.

    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    uint256 public feeBps; // e.g. 100 = 1%

    constructor(uint256 _feeBps) { feeBps = _feeBps; }

    function mint(address to, uint256 amt) external { balanceOf[to] += amt; }
    function approve(address sp, uint256 amt) external returns (bool) {
        allowance[msg.sender][sp] = amt; return true;
    }
    function transfer(address to, uint256 amt) external returns (bool) {
        uint256 fee = amt * feeBps / 10000;
        balanceOf[msg.sender] -= amt;
        balanceOf[to] += amt - fee;
        return true;
    }
    function transferFrom(address from, address to, uint256 amt) external returns (bool) {
        allowance[from][msg.sender] -= amt;
        uint256 fee = amt * feeBps / 10000;
        balanceOf[from] -= amt;
        balanceOf[to] += amt - fee;
        return true;
    }
}
