### Title
`mulDivUp` rounding in `sellerPendingFeeDecrease` drains `pendingFee` to zero while `credit > 0` across repeated unit-1 partial fills - (`src/Midnight.sol`)

### Summary

In `Midnight.take()`, the seller's `pendingFee` is reduced by `ceil(pendingFee * sellerCreditDecrease / credit)` using `mulDivUp`. When `sellerCreditDecrease = 1` and `pendingFee < credit`, the ceiling always rounds up to 1, deducting exactly 1 wei of `pendingFee` per fill regardless of the true proportional share. After exactly `pendingFee` such fills, `pendingFee` reaches 0 while `credit = original_credit - pendingFee > 0`, permanently zeroing the continuous fee obligation on remaining credit.

### Finding Description

**Exact code path** — `src/Midnight.sol` lines 387–389 and 412–413:

```solidity
uint128 sellerPendingFeeDecrease = sellerPos.credit > 0
    ? UtilsLib.toUint128(sellerPos.pendingFee.mulDivUp(sellerCreditDecrease, sellerPos.credit))
    : 0;
// ...
sellerPos.pendingFee -= sellerPendingFeeDecrease;
sellerPos.credit    -= UtilsLib.toUint128(sellerCreditDecrease);
```

`mulDivUp(x, y, d) = (x * y + d - 1) / d` (`src/libraries/UtilsLib.sol` lines 34–36).

**Root cause** — When `sellerCreditDecrease = 1` and `pendingFee < credit`:

```
sellerPendingFeeDecrease = ceil(pendingFee * 1 / credit) = ceil(pendingFee / credit)
```

Since `pendingFee < credit`, `pendingFee / credit < 1`, so `ceil(pendingFee / credit) = 1`. Each fill deducts exactly 1 from `pendingFee` regardless of the true proportional share `pendingFee / credit`.

**Concrete trace** — `credit = 1000`, `pendingFee = 10`:

| Fill | `sellerCreditDecrease` | `sellerPendingFeeDecrease` | `pendingFee` after | `credit` after |
|------|------------------------|----------------------------|--------------------|----------------|
| 1    | 1                      | ceil(10/1000) = 1          | 9                  | 999            |
| 2    | 1                      | ceil(9/999) = 1            | 8                  | 998            |
| …    | …                      | …                          | …                  | …              |
| 10   | 1                      | ceil(1/991) = 1            | **0**              | **990**        |

After 10 fills: `pendingFee = 0`, `credit = 990 > 0`.

**Why existing checks fail** — The Certora invariant `pendingContinuousFeeBoundedByCredit` (`certora/specs/Midnight.spec` lines 137–149) only proves `pendingFee ≤ credit`; it does not prevent `pendingFee = 0` with `credit > 0`. The rule `noRemainingContinuousFeeWithoutCredit` (lines 151–154) only proves `credit = 0 ⟹ pendingFee = 0`, not the converse. The Certora rule `pendingFeeDecreasesProportionallyForSeller` (`certora/specs/ContinuousFee.spec` lines 65–91) verifies only the single-step formula, not the multi-step ratio invariant.

**Exploit path** — Seller (address A) creates a sell offer ratified via `SetterRatifier.setIsRootRatified`. Address B (controlled by the same party, distinct from A to pass `require(offer.maker != taker, SelfTake())`) calls `take()` with `units = 1` exactly `pendingFee` times. After the loop, seller's `pendingFee = 0` while `credit > 0`. Subsequent calls to `updatePositionView` compute `fee = postSlashPendingFee.mulDivDown(...) = 0`, so no credit is ever deducted as continuous fees on the remaining balance.

### Impact Explanation

The seller permanently avoids paying continuous fees on `credit - pendingFee` units of remaining credit. The `continuousFeeCredit` pool accrues nothing from this position going forward, directly reducing the fee revenue claimable by the protocol's fee claimer. The magnitude of leakage equals the initial `pendingFee` value in credit units.

### Likelihood Explanation

**Preconditions**: seller holds credit with `pendingFee > 0` and `pendingFee < credit`; seller controls a second address to act as taker; sell offer has `maxUnits ≥ pendingFee`. These are all normal operating conditions.

**Feasibility**: The attack requires exactly `pendingFee` fills of size 1. For typical parameters (`continuousFee ≈ 1%/year`, `timeToMaturity = 360 days`, `credit = 10^18`), `pendingFee ≈ 10^16`, making the attack gas-prohibitive at scale. However, for small `pendingFee` values (e.g., positions with low fee rates, short TTM, or small credit), the number of fills is tractable. The attack is always theoretically profitable since `settlementFee < WAD` means each fill costs less than 1 unit of credit while saving 1 unit of `pendingFee`.

### Recommendation

After computing `sellerPendingFeeDecrease`, check whether the remaining credit is zero and if so zero out `pendingFee` entirely; alternatively, use a floor-then-clamp approach: if `sellerCreditDecrease == sellerPos.credit`, set `sellerPendingFeeDecrease = sellerPos.pendingFee` directly (full exit zeroes the fee). For partial fills, the current `mulDivUp` is correct in direction but the multi-fill accumulation should be bounded by asserting `sellerPos.pendingFee - sellerPendingFeeDecrease` is proportionally consistent, or by tracking the fee rate as a ratio rather than an absolute value.

contract PendingFeeRoundingTest is MidnightBaseTest {
    function testPendingFeeReachesZeroWhileCreditPositive() public {
        uint256 credit = 1000;
        uint256 feeRate = 1e14; // small but nonzero
        uint256 ttm = 360 days;

        setupLender(credit, feeRate, ttm);
        uint256 initialPendingFee = midnight.pendingFee(id, lender);
        // Precondition: pendingFee > 0 and pendingFee < credit
        vm.assume(initialPendingFee > 0 && initialPendingFee < credit);

        // Seller (lender) creates a sell offer; otherLender acts as taker (buyer)
        Offer memory offer = _makeBuyOffer(credit, keccak256("drain"));
        offer.maxUnits = initialPendingFee + 1;

        for (uint256 i = 0; i < initialPendingFee; i++) {
            deal(address(loanToken), otherLender, 1e18);
            take(1, lender, offer); // units = 1 each time
        }

        uint256 pendingFeeAfter = midnight.pendingFee(id, lender);
        uint256 creditAfter = midnight.creditOf(id, lender);

        // INVARIANT VIOLATION: pendingFee == 0 but credit > 0
        assertEq(pendingFeeAfter, 0, "pendingFee drained to zero");
        assertGt(creditAfter, 0, "credit still positive");

        // Confirm no fees accrue going forward
        vm.warp(block.timestamp + ttm / 2);
        midnight.updatePosition(market, lender);
        assertEq(midnight.pendingFee(id, lender), 0, "no fees accrue after drain");
        assertEq(midnight.creditOf(id, lender), creditAfter, "credit unchanged — zero fee leakage confirmed");
    }

    // Fuzz variant: assert pendingFee/credit ratio never drops to 0/positive across N fills
    function testFuzzPendingFeeRatioPreserved(uint256 credit, uint256 feeRate, uint256 ttm, uint256 fillSize, uint256 nFills) public {
        credit = bound(credit, 100, 1e6);
        feeRate = bound(feeRate, 1, MAX_CONTINUOUS_FEE);
        ttm = bound(ttm, 2, 360 days);
        setupLender(credit, feeRate, ttm);
        uint256 pf0 = midnight.pendingFee(id, lender);
        vm.assume(pf0 > 0);
        fillSize = bound(fillSize, 1, credit / 10);
        nFills = bound(nFills, 1, pf0);

        for (uint256 i = 0; i < nFills; i++) {
            if (midnight.creditOf(id, lender) < fillSize) break;
            deal(address(loanToken), otherLender, 1e18);
            take(fillSize, lender, _makeBuyOffer(credit, keccak256(abi.encode(i))));
        }

        uint256 pfAfter = midnight.pendingFee(id, lender);
        uint256 cAfter  = midnight.creditOf(id, lender);
        // Assert: if credit > 0, pendingFee must be > 0 (ratio preserved)
        if (cAfter > 0) {
            assertGt(pfAfter, 0, "pendingFee must not be zero while credit > 0");
        }
    }
}

// When a seller's credit decreases via a take, their pendingFee decreases by ceil(PendingFee * creditDelta / postUpdateCredit).
rule pendingFeeDecreasesProportionallyForSeller(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    address seller = offer.buy ? taker : offer.maker;

    bytes32 id;
    uint128 postUpdateCredit;
    uint128 postUpdatePendingFee;

    postUpdateCredit, postUpdatePendingFee, _ = updatePositionView(e, offer.market, id, seller);

    require postUpdateCredit > 0 || postUpdatePendingFee == 0, "See noRemainingContinuousFeeWithoutCredit in Midnight.spec";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    require id == lastId, "id should be derived from market";

    uint256 creditAfter = creditOf(id, seller);
    uint256 pendingFeeAfter = pendingFee(id, seller);

    require creditAfter > 0 || pendingFeeAfter == 0, "See noRemainingContinuousFeeWithoutCredit in Midnight.spec";

    mathint creditDelta = postUpdateCredit - creditAfter;

    // When postUpdateCredit == 0: noRemainingContinuousFeeWithoutCredit gives postUpdatePendingFee == 0; credit is non-increasing for a seller, therefore creditAfter == 0;
    // noRemainingContinuousFeeWithoutCredit gives pendingFeeAfter == 0; hence pendingFeeDelta == 0.
    assert postUpdateCredit == 0 ? postUpdatePendingFee == pendingFeeAfter : postUpdatePendingFee == pendingFeeAfter + (postUpdatePendingFee * creditDelta + postUpdateCredit - 1) / postUpdateCredit;
}
