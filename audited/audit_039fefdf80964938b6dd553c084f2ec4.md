Audit Report

## Title
Fee-on-Transfer Collateral Inflates `position.collateral` Accounting, Enabling Undercollateralized Borrowing - (File: src/Midnight.sol)

## Summary
`supplyCollateral` records the caller-supplied `assets` value into `_position.collateral[collateralIndex]` before executing the transfer, with no balance-delta verification. For fee-on-transfer collateral tokens, the protocol records more collateral than it actually receives. `isHealthy` then reads the inflated storage value, allowing a borrower's position to pass the health check while being undercollateralized from inception.

## Finding Description
In `supplyCollateral` (`src/Midnight.sol:533`), the accounting write unconditionally trusts the `assets` argument:

```solidity
// line 533 ŌĆö records full `assets` before transfer
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
// ...
// line 545 ŌĆö only delivers assets*(1-fee) for fee-on-transfer tokens
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol:24-34`) only checks the boolean return value; it performs no balance-before/after delta check. There is no snapshot of `balanceOf(address(this))` to verify the actual received amount.

`isHealthy` (`src/Midnight.sol:954-955`) reads `_position.collateral[i]` directly from storage:

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

`touchMarket` (`src/Midnight.sol:755-791`) validates only LLTV tiers, sorted collateral addresses, and `maxLif` ŌĆö no token-type restriction prevents a fee-on-transfer ERC20 from being used as collateral.

**Exploit path via `take` ŌåÆ `onSell` ŌåÆ `supplyCollateral`:**

1. Attacker deploys a fee-on-transfer ERC20 (e.g., 1% fee) and calls `touchMarket` to create a market with it as collateral. No admin action required.
2. A lender creates a buy offer ratified by `SetterRatifier`. The ratifier only verifies Merkle proof and root membership ŌĆö no collateral-token checks.
3. Attacker deploys a callback contract implementing `ISellCallback.onSell` that calls `supplyCollateral(market, index, amount, attacker)`. Attacker calls `setIsAuthorized(callbackContract, true, attacker)` so the authorization check at `src/Midnight.sol:527` passes.
4. Attacker calls `take(lenderOffer, ...)`. Inside `take`:
   - Debt state is updated: `sellerPos.debt += sellerDebtIncrease` (`src/Midnight.sol:414`).
   - Loan tokens are transferred to the attacker (`src/Midnight.sol:455-456`).
   - `onSell` is invoked on the callback contract (`src/Midnight.sol:458-473`).
   - Inside `onSell`, `supplyCollateral` is called: `_position.collateral[index] += amount` (records `amount`), but `safeTransferFrom` only delivers `amount * 0.99` to Midnight.
   - After the callback, `isHealthy` is called (`src/Midnight.sol:476`) using the inflated `_position.collateral[index] = amount`, not the actual `amount * 0.99` held.
5. Health check passes. The attacker now holds `units` of loan tokens backed by only `amount * 0.99` actual collateral, while the protocol believes it holds `amount`.

No existing check stops this: `supplyCollateral` has no balance-before/after guard; `SafeTransferLib` only checks the return boolean; `isHealthy` reads storage directly; `touchMarket` imposes no token-type constraints.

## Impact Explanation
The protocol's core solvency invariant ŌĆö contract token balances must cover all withdrawable assets, collateral claims, credit redemptions, and accrued fees ŌĆö is violated immediately upon the first `supplyCollateral` call with a fee-on-transfer token. The borrower's position is undercollateralized from inception. When the position is eventually liquidated, the seized collateral is less than the recorded amount, leaving unpayable bad debt socialized among lenders. This constitutes direct theft of lender funds via protocol accounting manipulation.

## Likelihood Explanation
Any permissionless market creator can deploy a market with a fee-on-transfer collateral token ŌĆö no admin action is required. The borrower only needs to authorize their own callback contract, which is standard self-authorization. `SECURITY.md` contains no exclusion for fee-on-transfer tokens. The exploit is repeatable on every `supplyCollateral` call with a fee-bearing token and scales linearly with the fee rate and borrowed amount.

## Recommendation
Replace the trust-the-argument pattern with a balance-delta check in `supplyCollateral`:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Move the accounting write after the transfer and use `received` instead of `assets`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as collateral by adding a token allowlist or a post-transfer balance assertion that reverts if `received < assets`.

## Proof of Concept
1. Deploy `FeeToken` ERC20 with a 1% transfer fee.
2. Call `touchMarket` with `FeeToken` as the sole collateral token.
3. Have a lender supply loan tokens and create a ratified buy offer.
4. Deploy `MaliciousCallback` implementing `ISellCallback.onSell` that calls `supplyCollateral(market, 0, 1000e18, attacker)`.
5. Call `setIsAuthorized(MaliciousCallback, true, attacker)`.
6. Call `take(lenderOffer, ..., MaliciousCallback, ...)`.
7. Assert: `position[id][attacker].collateral[0] == 1000e18` but `FeeToken.balanceOf(Midnight) == 990e18`.
8. Assert: `isHealthy(market, id, attacker) == true` despite the 10e18 shortfall.
9. Confirm: liquidating the position yields only 990e18 tokens, leaving 10e18 of bad debt unrecoverable.
