### Title
`sweepUnclaimedBonus` drains bonus between moderator re-flags, reducing named attacker's bounty entitlement - (File: src/ConfidencePool.sol)

### Summary
When the moderator corrects an initial SURVIVED flag to CORRUPTED (good-faith) before any claim, an unprivileged caller can call `sweepUnclaimedBonus` between the two `flagOutcome` calls to drain `totalBonus` to zero (when `riskWindowStart == 0`). The re-flag re-snapshots `snapshotTotalBonus = totalBonus` at the drained value, permanently reducing `bountyEntitlement` and causing the named whitehat attacker to lose the bonus portion of their entitled bounty.

### Finding Description
`flagOutcome` permits the moderator to re-flag the outcome before `claimsStarted` is set, re-snapshotting all accounting state on each call:

```solidity
// src/ConfidencePool.sol:354-362
outcome = newOutcome;
goodFaith = goodFaith_;
attacker = attacker_;
snapshotTotalStaked = totalEligibleStake;
snapshotTotalBonus = totalBonus;          // <-- re-snapshot on every flag
...
bountyEntitlement = willBeGoodFaithCorrupted ? snapshotTotalStaked + snapshotTotalBonus : 0;
```

Between the first flag (SURVIVED) and the correction (CORRUPTED good-faith), `sweepUnclaimedBonus` is callable by anyone with no access control. When `riskWindowStart == 0`, the sweep reserves only `totalEligibleStake` — no bonus is reserved for stakers because no risk window was observed:

```solidity
// src/ConfidencePool.sol:483-488
if (totalEligibleStake != 0) {
    reserved = totalEligibleStake;
    if (riskWindowStart != 0) {
        reserved += snapshotTotalBonus - claimedBonus;
    }
}
```

With `riskWindowStart == 0`, the entire `totalBonus` is above the reserve and is swept to `recoveryAddress`. Critically, `sweepUnclaimedBonus` intentionally does **not** set `claimsStarted` (to avoid blocking the moderator's re-flag window), so the re-flag remains open:

```solidity
// src/ConfidencePool.sol:503-506
// Intentionally does NOT set claimsStarted. A direct-transfer donation of as little as 1
// wei would otherwise let anyone flip the flag post-flagOutcome and block the moderator's
// documented pre-claim re-flag window.
stakeToken.safeTransfer(recoveryAddress, amount);
```

When the moderator then re-flags to CORRUPTED good-faith, `snapshotTotalBonus = totalBonus = 0`, so `bountyEntitlement = snapshotTotalStaked + 0`. The named attacker can only claim the stake portion.

### Impact Explanation
The named whitehat attacker loses the entire bonus portion of their bounty. Funds that the protocol intends to pay to the attacker (per `bountyEntitlement = snapshotTotalStaked + snapshotTotalBonus`) are instead permanently redirected to `recoveryAddress` via the sweep. This is a direct, irreversible fund misallocation caused by an unprivileged caller exploiting the interaction between `sweepUnclaimedBonus` and the re-flag snapshot.

### Likelihood Explanation
Three conditions are required:
1. `riskWindowStart == 0` at the time of the first flag — occurs when no pool interaction observed the registry in an active-risk state before it reached a terminal state.
2. Moderator makes a correction from SURVIVED to CORRUPTED good-faith — a realistic scenario where the moderator initially judges the breach out-of-scope, then corrects.
3. An unprivileged caller front-runs the re-flag with `sweepUnclaimedBonus` — trivially achievable by any on-chain observer watching the mempool.

Condition (1) is not rare: if the registry transitions quickly from `NEW_DEPLOYMENT` to `CORRUPTED` without anyone poking the pool, `riskWindowStart` stays zero. Conditions (2) and (3) are both realistic. The combination is a credible attack path.

### Recommendation
Prevent `sweepUnclaimedBonus` from executing during the moderator's re-flag window (i.e., when `outcome != UNRESOLVED && !claimsStarted`):

```solidity
function sweepUnclaimedBonus() external nonReentrant {
    if (outcome != PoolStates.Outcome.SURVIVED && outcome != PoolStates.Outcome.EXPIRED) {
        revert OutcomeNotEligibleForSweep();
    }
+   if (!claimsStarted) revert ClaimsNotStarted(); // block sweep during re-flag window
    ...
}
```

Alternatively, on re-flag, restore `totalBonus` from the previous `snapshotTotalBonus` if it was reduced by an intervening sweep, so the re-snapshot captures the correct pre-sweep value.

### Proof of Concept
1. Alice stakes 100 tokens. Carol contributes 50 tokens bonus. Registry transitions directly from `NEW_DEPLOYMENT` to `CORRUPTED` with no active-risk observation (`riskWindowStart == 0`, `riskWindowEnd` is set).
2. Moderator calls `flagOutcome(SURVIVED, false, address(0))` — initial (incorrect) flag. `snapshotTotalBonus = 50`.
3. Unprivileged caller (Eve) calls `sweepUnclaimedBonus()`. Since `riskWindowStart == 0`, `reserved = totalEligibleStake = 100`. Free balance = 150. `amount = 50`. Entire bonus swept to `recoveryAddress`. `totalBonus = 0`.
4. Moderator calls `flagOutcome(CORRUPTED, true, attackerAddr)` — correction. Re-snapshot: `snapshotTotalBonus = totalBonus = 0`. `bountyEntitlement = 100 + 0 = 100`.
5. Attacker calls `claimAttackerBounty()` — receives only 100 (stake), not 150 (stake + bonus). The 50-token bonus is permanently lost to `recoveryAddress`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
