### Title
Zero-Oracle Price Allows Free Collateral Seizure via `seizedAssets` Path — (`File: src/Midnight.sol`)

### Summary
When the oracle for the liquidated collateral returns `0`, the `seizedAssets > 0` branch in `liquidate` computes `repaidUnits = mulDivUp(seizedAssets, 0, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif) = 0` without reverting. The liquidator receives collateral while transferring zero loan tokens, violating the protocol's own stated invariant that a zero-returning oracle must block `liquidate`. The `repaidUnits > 0` branch correctly reverts (division by zero on `mulDivDown(ORACLE_PRICE_SCALE, 0)`), but the `seizedAssets > 0` branch has no equivalent guard.

### Finding Description

**Code path:**

In `src/Midnight.sol`, the oracle price loop sets `liquidatedCollatPrice`:

```solidity
uint256 price = IOracle(_collateralParam.oracle).price();
if (i == collateralIndex) liquidatedCollatPrice = price;
``` [1](#0-0) 

If the oracle returns `0`, `liquidatedCollatPrice = 0`. The position is still liquidatable because `maxDebt = 0` (zero-price collateral contributes nothing), so `originalDebt > maxDebt` passes. The entire debt is also classified as `badDebt` and written off:

```solidity
badDebt = badDebt.zeroFloorSub(
    _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
);
``` [2](#0-1) 

With `price = 0`, `zeroFloorSub(originalDebt, 0) = originalDebt`, so `badDebt = originalDebt` and `_position.debt` is zeroed at line 628.

Then in the seized-assets branch:

```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
``` [3](#0-2) 

`mulDivUp(seizedAssets, 0, ORACLE_PRICE_SCALE) = (seizedAssets * 0 + ORACLE_PRICE_SCALE - 1) / ORACLE_PRICE_SCALE = 0`. Then `mulDivUp(0, WAD, lif) = 0`. No division by zero occurs; no revert. `repaidUnits = 0`.

Execution continues:

```solidity
_marketState.withdrawable += UtilsLib.toUint128(repaidUnits);  // += 0
_position.debt -= UtilsLib.toUint128(repaidUnits);             // -= 0
``` [4](#0-3) 

```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
``` [5](#0-4) 

```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits); // 0
``` [6](#0-5) 

The liquidator receives `seizedAssets` collateral tokens and pays `0` loan tokens.

**Contrast with `repaidUnits > 0` path:**

```solidity
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
``` [7](#0-6) 

`mulDivDown(x, ORACLE_PRICE_SCALE, 0)` triggers division by zero → revert. This is why `oracleZeroCausesLiquidateWithRepaidRevert` holds in the Certora spec, but there is no corresponding rule for the `seizedAssets > 0` path. [8](#0-7) 

**Protocol's own invariant violated:**

The Certora README explicitly states:

> "A reverting or zero-returning collateral oracle blocks `liquidate`, `withdrawCollateral`, `isHealthy` and `take` whenever the borrower has debt." [9](#0-8) 

The `NoDivisionByZero.spec` also explicitly carves out the zero-price case as a known exception, confirming the protocol treats it as a realistic scenario rather than an impossible one: [10](#0-9) [11](#0-10) 

### Impact Explanation

The liquidator calls `liquidate(market, collateralIndex, seizedAssets=N, repaidUnits=0, borrower, ...)` when the collateral oracle returns `0`. The call succeeds: `N` collateral tokens are transferred to the liquidator's `receiver`, and `0` loan tokens are pulled from the liquidator. The borrower's entire debt is simultaneously written off as bad debt, socializing losses to lenders. The borrower loses collateral with no corresponding debt reduction (debt was already zeroed by bad debt accounting). The liquidator extracts real collateral value at zero cost.

### Likelihood Explanation

Preconditions: (1) a market collateral oracle returns `0` — explicitly modeled as realistic in the protocol's own Certora specs; (2) the borrower has debt and is liquidatable (trivially satisfied when oracle returns `0` since `maxDebt = 0`). The attack is permissionless, repeatable for any position in any market whose collateral oracle returns `0`, and requires no special role or privileged access. Any liquidator can trigger it.

### Recommendation

After computing `repaidUnits` in the `seizedAssets > 0` branch, add a guard:

```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
    require(repaidUnits > 0, ZeroRepaidUnits()); // guard against zero-price oracle
}
```

This mirrors the revert behavior already present in the `repaidUnits > 0` branch (which reverts via division by zero) and enforces the protocol's stated invariant that zero-returning oracles block `liquidate`.
