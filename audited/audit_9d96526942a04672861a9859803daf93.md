Audit Report

## Title
`SafeTransferLib` misinterprets 64-byte returndata with zero first word as `false`, enabling permanent lender fund freeze via malicious `loanToken` - (File: src/libraries/SafeTransferLib.sol)

## Summary
`SafeTransferLib.safeTransfer` uses `abi.decode(returndata, (bool))` which reads only the first 32-byte word regardless of total return length. A token returning 64 bytes where the first word is zero is decoded as `false`, causing `TransferReturnedFalse()` to revert even though the transfer succeeded on-chain. Because `touchMarket` imposes no validation on `market.loanToken`, any unprivileged actor can create a market with such a token and permanently freeze all lender withdrawals.

## Finding Description

**Root cause — `src/libraries/SafeTransferLib.sol` lines 21 and 33:**

```solidity
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
```

Solidity's `abi.decode(data, (bool))` reads the first 32-byte word of `data` as the boolean value, ignoring any trailing bytes. When `returndata` is 64 bytes and the first word is `0x00…00` (e.g., `abi.encode(uint256(0), uint256(1))`), the decoded value is `false` and the `require` reverts — even though `success == true` and balances were updated. The `returndata.length == 0` short-circuit only bypasses the decode for zero-length returns; it provides no protection against oversized returns.

**No `loanToken` validation in `touchMarket` — `src/Midnight.sol` lines 755–791:**

`touchMarket` validates maturity, collateral count, collateral ordering, lltv, and maxLif. It performs zero checks on `market.loanToken`. Any address with `code.length > 0` is accepted as a loan token.

**Exploit flow:**

1. Attacker deploys `MaliciousToken`:
   - `transferFrom(from, to, amount)` → returns `abi.encode(true)` (32 bytes, first word non-zero) → `safeTransferFrom` passes.
   - `transfer(to, amount)` → returns `abi.encode(uint256(0), uint256(1))` (64 bytes, first word = 0) → `safeTransfer` reverts with `TransferReturnedFalse()`.
   - Both functions actually move balances on-chain.

2. Attacker calls `touchMarket` with `market.loanToken = address(MaliciousToken)`. Market is created without error.

3. Lender calls `take` → internally calls `safeTransferFrom(loanToken, …)` → `transferFrom` returns 32-byte `true` → succeeds. Lender credit is recorded in `position[id][lender].credit`.

4. Lender calls `withdraw(market, units, lender, receiver)`:
   - Lines 493–495: state is mutated (`credit` decremented, `withdrawable` decremented, `totalUnits` decremented).
   - Line 499: `SafeTransferLib.safeTransfer(market.loanToken, receiver, units)` is called.
   - `transfer` returns 64 bytes with first word = 0 → `abi.decode(returndata, (bool))` = `false` → `require` reverts with `TransferReturnedFalse()`.
   - Entire transaction reverts; state is rolled back. Lender credit remains locked.

5. Every subsequent `withdraw` call follows the same path and reverts identically. Funds are permanently frozen with no recovery path.

**Why existing checks are insufficient:**
- The `returndata.length == 0` guard only bypasses decoding for zero-length returns; it does not protect against returns longer than 32 bytes.
- There is no allowlist, interface check, or code-size validation on `loanToken` in `touchMarket`.
- The Certora `BalanceEffects.spec` and `Reverts.spec` summarize `SafeTransferLib.safeTransfer` as `NONDET`, meaning the 64-byte false-return edge case is not modeled in formal verification.

## Impact Explanation

All lender funds supplied to any market whose `loanToken` returns 64 bytes with a zero first word from `transfer` are permanently unwithdrawable. `withdraw` always reverts at line 499 with no recovery path. This constitutes a permanent, unrecoverable freeze of user funds — a direct in-scope impact per the RESEARCHER.md definition of "Permanent lock, freeze, or unrecoverable corruption of user/project state."

## Likelihood Explanation

All attacker preconditions are fully permissionless: deploying a contract requires no privileges, and `touchMarket` accepts any `loanToken`. The attack requires attracting lenders, but in a permissionless protocol with automated vaults or aggregators, a token that behaves normally during `transferFrom` (returning 32-byte `true`) but abnormally during `transfer` can attract real deposits before the `transfer` path is exercised. The condition is deterministic and repeatable: every `withdraw` call reverts with no exception.

## Recommendation

Replace the bare `abi.decode` check with an explicit length-and-value check that rejects returndata of unexpected length:

```solidity
// Accept only: zero-length return (no-return tokens) OR exactly 32 bytes decoding to true
require(
    returndata.length == 0 ||
    (returndata.length == 32 && abi.decode(returndata, (bool))),
    TransferReturnedFalse()
);
```

This rejects any returndata that is not exactly 0 or 32 bytes, closing the 64-byte (and any other oversized) false-return vector. Apply the same fix to `safeTransferFrom`.

## Proof of Concept

```solidity
contract MaliciousToken {
    mapping(address => uint256) public balanceOf;

    function mint(address to, uint256 amount) external {
        balanceOf[to] += amount;
    }

    function approve(address, uint256) external returns (bool) { return true; }

    // transferFrom: returns standard 32-byte true → safeTransferFrom passes
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        balanceOf[from] -= amount;
        balanceOf[to] += amount;
        return true;
    }

    // transfer: returns 64 bytes with first word = 0 → safeTransfer reverts
    function transfer(address to, uint256 amount) external returns (bytes memory) {
        balanceOf[msg.sender] -= amount;
        balanceOf[to] += amount;
        // abi.decode(returndata, (bool)) reads first word = 0 → false → revert
        assembly {
            mstore(0x00, 0) // first word: 0 (decoded as false)
            mstore(0x20, 1) // second word: 1 (ignored by abi.decode)
            return(0x00, 0x40)
        }
    }
}
```

**Steps:**
1. Deploy `MaliciousToken`, mint tokens to lender.
2. Call `touchMarket` with `loanToken = address(MaliciousToken)` and valid collateral params.
3. Lender approves `MaliciousToken` to Midnight, calls `take` → succeeds, credit recorded.
4. Borrower repays → `safeTransferFrom` on `MaliciousToken` succeeds (32-byte `true`).
5. Lender calls `withdraw` → `safeTransfer` on `MaliciousToken` returns 64 bytes with first word 0 → reverts with `TransferReturnedFalse()`.
6. Repeat step 5 indefinitely — always reverts. Lender credit is permanently frozen.
