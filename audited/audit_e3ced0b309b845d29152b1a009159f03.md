### Title
Missing `LIQUIDATION_LOCK_SLOT` guard in `liquidate()` enables reentrant double collateral seizure - (File: src/Midnight.sol)

### Summary
`liquidate()` checks `!liquidationLocked(id, borrower)` as a precondition but **never sets** `LIQUIDATION_LOCK_SLOT` for the borrower at any point in its execution. Because all state changes (collateral seizure, debt reduction) occur before the external `onLiquidate` callback, a reentrant call to `liquidate()` during that callback will pass the lock check and, if the position remains liquidatable after the first partial seizure, will seize additional collateral — bypassing the per-call recovery close factor (`maxRepaid`) cap.

### Finding Description

**Root cause — `liquidate()` never writes `LIQUIDATION_LOCK_SLOT`:**

In `take()`, the lock is explicitly set before any external call and cleared after:

```solidity
// src/Midnight.sol line 444
bool wasLocked = UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true);
// ... callbacks ...
if (!wasLocked) UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, false);
``` [1](#0-0) 

In `liquidate()`, the lock is only **read**, never written:

```solidity
// src/Midnight.sol lines 620-624
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
``` [2](#0-1) 

**Execution order within `liquidate()`:**

1. Lines 626–677: state changes committed — `_position.collateral[collateralIndex]` reduced, `_position.debt` reduced.
2. Line 696: collateral transferred to `receiver`.
3. Lines 698–714: external `ILiquidateCallback(callback).onLiquidate(...)` called — **lock is still `false`**.
4. Line 717: loan token pulled from `payer`. [3](#0-2) 

**Exploit flow (non-postMaturity path):**

- Precondition: borrower has debt `D` and collateral `C` with `D > maxDebt` (unhealthy). Market has no `liquidatorGate`.
- Attacker calls `liquidate(borrower, seizedAssets=S1, callback=attacker_contract)`.
  - Lock check passes (`liquidationLocked == false`).
  - Collateral reduced to `C - S1`; debt reduced to `D - R1`.
  - `onLiquidate` fires on attacker contract.
  - Inside `onLiquidate`: attacker calls `liquidate(borrower, seizedAssets=S2)` again.
    - Lock check: `liquidationLocked(id, borrower)` is still `false` → **passes**.
    - New `maxDebt` computed from reduced collateral `C - S1` (lower than before).
    - If `D - R1 > new_maxDebt` → liquidatability check passes.
    - Second seizure: collateral reduced to `C - S1 - S2`; debt reduced to `D - R1 - R2`.
    - Can recurse further until debt is zero or gas runs out.
- Total seized across all reentrant calls: `S1 + S2 + ... >> maxRepaid` (the per-call RCF cap).

**postMaturity path is even simpler:** the liquidatability condition is only `block.timestamp > market.maturity`, which is timestamp-based and never changes during the transaction. As long as `_position.debt > 0` after each call, every reentrant call succeeds unconditionally. [4](#0-3) 

**Why existing checks fail:**

- The `maxRepaid` / RCF check (lines 659–667) is per-call only; it does not accumulate across reentrant calls within the same transaction.
- The `_position.debt > 0` check (line 596) is satisfied as long as the borrower still has remaining debt.
- The existing tests `testSellSellerCallbackLiquidateRevertsWhileLiquidationLocked` and `testSellNestedTakeLiquidateRevertsWhileLiquidationLocked` only cover reentrancy from within `take()`'s callback (where the lock IS set by `take()`), not from within `liquidate()`'s own callback. [5](#0-4) [6](#0-5) 

### Impact Explanation

An unprivileged liquidator can seize a borrower's collateral multiple times within a single transaction by reentering `liquidate()` through the `onLiquidate` callback. This bypasses the recovery close factor (`maxRepaid`) cap that is designed to limit how much collateral can be seized per liquidation event. In the `postMaturityMode` path, the attacker can drain the borrower's entire collateral in one transaction regardless of the RCF. This constitutes unauthorized excess collateral seizure beyond what the protocol's liquidation rules permit.

### Likelihood Explanation

- **Preconditions**: any unhealthy borrower (or any borrower post-maturity) with a non-zero `callback` parameter. No special privileges required — any address can call `liquidate()` on an unhealthy position when `liquidatorGate == address(0)`.
- **Feasibility**: trivially reachable; the attacker deploys a contract implementing `ILiquidateCallback` that reenters `liquidate()` inside `onLiquidate`. The attacker must supply the loan token for `repaidUnits` across all reentrant calls (received as collateral from the first seizure, or pre-funded).
- **Repeatability**: repeatable on every unhealthy position in every market without a `liquidatorGate`.

### Recommendation

Set `LIQUIDATION_LOCK_SLOT` for the borrower at the start of `liquidate()` (before any state changes or external calls) and clear it at the end, mirroring the pattern already used in `take()`:

```solidity
// At the top of liquidate(), after the NotLiquidatable check:
UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, borrower, true);
// ... existing logic ...
// At the end, before return:
UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, borrower, false);
```

This ensures that any reentrant call to `liquidate()` for the same `(id, borrower)` pair will hit `!liquidationLocked(id, borrower) == false` and revert with `NotLiquidatable`, exactly as the existing tests for `take()`-triggered reentrancy expect. [7](#0-6) [8](#0-7) 

### Proof of Concept

**Foundry unit test plan:**

```solidity
contract ReentrantLiquidateCallback is ILiquidateCallback {
    IMidnight midnight;
    Market market;
    uint256 collateralIndex;
    address borrower;
    uint256 reentrantSeizedAssets;
    bool public reentrantSucceeded;
    bytes4 public reentrantError;

    // fund with enough loanToken for both calls
    function onLiquidate(
        address, bytes32, Market memory _market,
        uint256 _collateralIndex, uint256, uint256,
        address _borrower, address, bytes memory, uint256
    ) external returns (bytes32) {
        // At this point liquidationLocked(id, borrower) is false
        // because liquidate() never sets it
        try midnight.liquidate(
            _market, _collateralIndex, reentrantSeizedAssets, 0,
            _borrower, false, address(this), address(0), ""
        ) {
            reentrantSucceeded = true;
        } catch (bytes memory err) {
            reentrantError = bytes4(err);
        }
        IERC20(market.loanToken).approve(address(midnight), type(uint256).max);
        return CALLBACK_SUCCESS;
    }
}

function testReentrantLiquidateDoubleSeizure() public {
    // Setup: borrower with debt D, collateral C, D > maxDebt (unhealthy)
    // ...
    uint256 collateralBefore = midnight.collateral(id, borrower, 0);
    uint256 debtBefore = midnight.debtOf(id, borrower);

    ReentrantLiquidateCallback cb = new ReentrantLiquidateCallback(...);
    deal(loanToken, address(cb), debtBefore); // fund for both repayments

    midnight.liquidate(market, 0, seizedAssets1, 0, borrower, false, address(cb), address(cb), "");

    // Assertions:
    assertTrue(cb.reentrantSucceeded(), "reentrant liquidate must succeed");
    // Total collateral seized > what a single maxRepaid-bounded call would allow
    assertLt(midnight.collateral(id, borrower, 0), collateralBefore - maxSingleSeize);
    // Debt reduced by both calls
    assertLt(midnight.debtOf(id, borrower), debtBefore - maxSingleRepaid);
}
```

Expected: `cb.reentrantSucceeded() == true`, total collateral seized exceeds the single-call RCF cap, demonstrating unauthorized double seizure.
