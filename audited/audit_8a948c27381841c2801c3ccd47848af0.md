### Title
First-Come-First-Serve Withdrawal Allows Lenders to Escape Bad Debt Socialization — (`src/Midnight.sol`)

---

### Summary

The `withdraw()` function in `Midnight.sol` allows lenders to withdraw their full credit value as long as `withdrawable >= units`, without accounting for unrealized bad debt. Because the `lossFactor` socialization mechanism is only updated when `liquidate()` is explicitly called, lenders who withdraw before bad debt is realized escape proportional loss entirely. This creates a first-come-first-serve race where early withdrawers receive 100% of their credit while late withdrawers are slashed to zero.

---

### Finding Description

**Vulnerability class**: Accounting — uneven loss distribution (first-come-first-serve withdrawal).

**Root cause**: The `lossFactor` is the protocol's mechanism for socializing bad debt among lenders. It is updated exclusively inside `liquidate()`: [1](#0-0) 

Until `liquidate()` is called, `lossFactor` remains stale and does not reflect any unrealized bad debt. The `withdraw()` function calls `_updatePosition()` before processing the withdrawal: [2](#0-1) 

`_updatePosition` slashes credit using the **current** `lossFactor`: [3](#0-2) 

If `lossFactor` has not yet been updated (bad debt unrealized), no slashing occurs and the lender withdraws at full face value. The only guard in `withdraw()` is the implicit underflow check on `withdrawable`: [4](#0-3) 

There is no proportional cap — a lender can drain the entire `withdrawable` pool before `liquidate()` is called, leaving nothing for other lenders who will later be slashed.

The protocol's own inline comment acknowledges this race: [5](#0-4) 

---

### Impact Explanation

**Concrete impact**: Lenders who withdraw before bad debt is realized receive 100% of their credit in tokens. Lenders who withdraw after `liquidate()` is called have their credit slashed proportionally and may find `withdrawable = 0`, receiving nothing. The intended invariant — that all lenders share losses proportionally via `lossFactor` — is violated. A lender with 50% of total credit can receive 100% of available tokens while another lender with 50% of total credit receives 0%.

**Severity**: High. Direct, permanent loss of lender funds with no recovery path.

---

### Likelihood Explanation

**Realistic feasibility**: No privileged access is required. Any lender can monitor on-chain collateral prices and detect when a borrower's position is undercollateralized before `liquidate()` is called. The window between a position becoming liquidatable and `liquidate()` being executed is non-zero (block latency, gas competition, liquidator incentive timing). A sophisticated lender (or MEV bot) can front-run the liquidation transaction with a `withdraw()` call, escaping the slash entirely.

---

### Recommendation

Apply the `lossFactor` slash to a lender's credit **before** allowing withdrawal, and additionally enforce that a lender can only withdraw their proportional share of `withdrawable` relative to `totalUnits`. Specifically, cap the withdrawable amount to:

```
maxWithdrawable = credit_after_slash * withdrawable / totalUnits
```

This ensures that even if bad debt has not yet been realized via `liquidate()`, lenders cannot withdraw more than their fair share of the currently available pool.

---

### Proof of Concept

**Setup**: Market with `loanToken = T`, two lenders (Alice, Bob), two borrowers (Charlie, Dave).

**Steps**:

1. Alice lends 50 units → `position[id][Alice].credit = 50`
2. Bob lends 50 units → `position[id][Bob].credit = 50`
3. `totalUnits = 100`, `withdrawable = 0`
4. Charlie borrows 50 units, Dave borrows 50 units
5. Charlie repays 50 units → `withdrawable = 50`
6. Dave's collateral price drops to 0 (bad debt). `liquidate()` has NOT been called yet. `lossFactor = 0`.
7. **Alice calls `withdraw(50)`**:
   - `_updatePosition`: `lossFactor = 0` → no slash → `credit` stays 50
   - `withdrawable` decrements: `50 - 50 = 0`
   - Alice receives **50 T tokens** (100% of her credit)
8. **Someone calls `liquidate(Dave)`**:
   - `badDebt = 50`, `totalUnits = 50` (Alice already withdrew her 50)
   - `lossFactor` updated to `type(uint128).max` (100% loss on remaining units)
   - `totalUnits = 0`, `withdrawable = 0`
9. **Bob calls `withdraw(50)`**:
   - `_updatePosition`: `lossFactor = type(uint128).max` → `postSlashCredit = 0`
   - `credit -= 50` underflows (credit is now 0) → **reverts**
   - Bob receives **0 T tokens**

**Result**: Alice receives 50 tokens (100%), Bob receives 0 tokens (0%).
**Expected fair distribution**: Alice 25 tokens (50%), Bob 25 tokens (50%). [6](#0-5) [7](#0-6) [8](#0-7)

    function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
        bytes32 id = touchMarket(market);
        MarketState storage _marketState = marketState[id];
        _updatePosition(market, id, onBehalf);

        Position storage _position = position[id][onBehalf];
        uint128 pendingFeeDecrease;
        if (_position.credit > 0) {
            pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
            _position.pendingFee -= pendingFeeDecrease;
        }
        _position.credit -= UtilsLib.toUint128(units);
        _marketState.withdrawable -= UtilsLib.toUint128(units);
        _marketState.totalUnits -= UtilsLib.toUint128(units);

        emit EventsLib.Withdraw(msg.sender, id, units, onBehalf, receiver, pendingFeeDecrease);

        SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
    }

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
