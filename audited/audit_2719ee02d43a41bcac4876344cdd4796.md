### Title
Free Credit Unit Accumulation via Zero-Asset Dust Takes on Buy Offers with `maxAssets` Cap - (`File: src/Midnight.sol`)

### Summary

When `offer.buy = true` and `offer.maxAssets > 0`, the `buyerAssets` amount is computed with `mulDivDown`, which rounds to zero for any `units < WAD / buyerPrice`. Because the `maxAssets` consumption counter is incremented by `buyerAssets` (not `units`), a zero-asset take leaves the counter unchanged, completely bypassing the cap. The MAKER (buyer) receives `units` credit for free while the two `safeTransferFrom` calls silently succeed with value 0.

### Finding Description

**Root cause — `src/Midnight.sol` lines 363, 367–369, 410, 455–456:**

```solidity
// line 363 — mulDivDown truncates to 0 when units * buyerPrice < WAD
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // ← can be 0
    : units.mulDivUp(buyerPrice, WAD);

// lines 367–369 — counter incremented by buyerAssets, not units
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group]
        += offer.buy ? buyerAssets : sellerAssets;  // += 0 → cap never advances
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}

// line 410 — credit is always units, regardless of buyerAssets
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease); // = units

// lines 455–456 — 0-value ERC20 transferFrom succeeds silently
SafeTransferLib.safeTransferFrom(..., buyerAssets - sellerAssets); // 0
SafeTransferLib.safeTransferFrom(..., sellerAssets);               // 0
```

**Preconditions:**
- `offer.buy = true`, `offer.maxAssets > 0`
- `buyerPrice` (= `tickToPrice(offer.tick) + settlementFee`) is small enough that `1 * buyerPrice < WAD` — satisfied by any low tick (tick near 0 gives `buyerPrice` ≪ WAD = 1e18)
- Attacker controls both MAKER address and a separate TAKER address (with collateral to pass the health check at line 476)

**Exploit flow:**
1. Attacker (as MAKER) calls `SetterRatifier.setIsRootRatified` to ratify a Merkle root containing a buy offer at a low tick with `maxAssets = X`.
2. Attacker (as TAKER, separate address with collateral) calls `Midnight.take` with `units = 1` (or any value where `units * buyerPrice < WAD`).
3. `buyerAssets = 1.mulDivDown(buyerPrice, WAD) = 0`; `sellerAssets = 0`.
4. `consumed[maker][group] += 0` → cap counter stays at 0 → `require(0 <= X)` passes.
5. `buyerCreditIncrease = 1`; MAKER's credit increases by 1 at zero token cost.
6. Both `safeTransferFrom` calls execute with value 0 — standard ERC20 accepts this.
7. TAKER receives 0 assets but gains 1 unit of debt (backed by their collateral).
8. Steps 2–7 repeat indefinitely; `maxAssets` cap is never consumed.

**Why existing checks fail:**
- No `require(buyerAssets > 0 || units == 0)` guard exists anywhere in `take`.
- The `maxAssets` branch (line 367) uses `buyerAssets` as the unit of consumption, not `units`, so the rounding gap is never closed.
- The `maxUnits` branch (line 371) would correctly track `units`, but it is not reached when `offer.maxAssets > 0`.
- The health check (line 476) applies to the *seller*, not the buyer, and is satisfied as long as the TAKER has sufficient collateral. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

The MAKER accumulates an unbounded number of credit units without transferring any loan tokens. Credit units are redeemable 1:1 against `withdrawable` via `withdraw` (line 494, `safeTransfer(loanToken, receiver, units)`). `withdrawable` is a shared pool funded by all borrowers' repayments (line 509) and liquidations (line 675). An attacker who has accumulated free credit units can drain `withdrawable` beyond their legitimate contribution, leaving other lenders unable to redeem their credit. This is a direct solvency violation: the contract's token balance no longer covers all outstanding credit. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

**Preconditions are easily met:**
- Any tick near 0 gives `buyerPrice` ≪ 1e18, making `units = 1` sufficient to trigger the rounding.
- The attacker only needs two addresses: one as MAKER (ratifies the offer via `SetterRatifier`) and one as TAKER (supplies collateral, takes the offer).
- The attack is repeatable in a single transaction via `multicall` or a loop, accumulating arbitrarily many free credit units before any `withdrawable` exists.
- No privileged role is required; any unprivileged user can create and ratify offers. [7](#0-6) 

### Recommendation

Add a guard immediately after computing `buyerAssets` and `sellerAssets` that rejects any take where `units > 0` but the corresponding asset amount rounds to zero:

```solidity
// After line 364, before line 366:
if (offer.buy) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
} else {
    require(units == 0 || sellerAssets > 0, ZeroSellerAssets());
}
```

Alternatively, switch the `maxAssets` consumption counter to always track `units` (not `buyerAssets`/`sellerAssets`) and separately enforce the asset cap by comparing `units.mulDivUp(buyerPrice, WAD) <= offer.maxAssets - consumed[...]` before the take proceeds. [8](#0-7)
