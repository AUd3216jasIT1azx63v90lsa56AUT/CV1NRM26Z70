### Title
Authorized Account Can Create Persistent Sub-Authorizations That Survive Revocation — (`src/Midnight.sol`)

### Summary

`setIsAuthorized` permits any currently-authorized account to grant further authorizations on behalf of the user. When the user later revokes the original authorized account, all sub-authorizations that account created remain active. There is no mechanism to enumerate or clear them, leaving the user's position permanently exposed.

### Finding Description

The authorization check in `setIsAuthorized` is:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    ...
}
``` [1](#0-0) 

The protocol's own NatSpec acknowledges this: *"authorized accounts can authorize other accounts on behalf of the user."* [2](#0-1) 

The test suite confirms an authorized account can successfully call `setIsAuthorized` on behalf of the user to grant a third party: [3](#0-2) 

**Exploit path:**

1. Alice calls `setIsAuthorized(Bob, true, Alice)` → `isAuthorized[Alice][Bob] = true`
2. Bob calls `setIsAuthorized(Charlie, true, Alice)` → passes the check (`isAuthorized[Alice][Bob]` is true), sets `isAuthorized[Alice][Charlie] = true`
3. Alice calls `setIsAuthorized(Bob, false, Alice)` → `isAuthorized[Alice][Bob] = false`
4. `isAuthorized[Alice][Charlie]` is **never cleared** — Charlie retains full authorization over Alice's position indefinitely

Charlie can now call any authorized function on behalf of Alice:
- `withdraw()` — drain Alice's credit [4](#0-3) 
- `withdrawCollateral()` — drain Alice's collateral [5](#0-4) 
- `take()` — open debt positions for Alice [6](#0-5) 
- `setIsAuthorized()` — grant further authorizations, creating an unbounded chain

The same attack vector exists through `EcrecoverAuthorizer`, where an authorized signer can submit a signed authorization granting a third party access on behalf of the user: [7](#0-6) 

### Impact Explanation

A malicious or compromised authorized account can silently plant a persistent backdoor (`isAuthorized[victim][attacker2] = true`) before the victim revokes it. After revocation, the attacker retains full control: they can withdraw all credit, drain all collateral, and open arbitrary debt positions on behalf of the victim. This is a direct, complete loss of funds for the victim with no on-chain recourse once the sub-authorization is set.

### Likelihood Explanation

The precondition is that Alice authorizes Bob (a malicious actor or a contract that becomes compromised). This is realistic because:
- Users routinely authorize periphery contracts (e.g., `MidnightBundles`, `EcrecoverAuthorizer`) for convenience
- A compromised or malicious periphery contract can silently plant a sub-authorization in a single transaction before the user notices
- The user has no way to enumerate all `isAuthorized[self][*]` entries on-chain (no enumerable mapping), so they cannot audit or clean up sub-authorizations

### Recommendation

Restrict `setIsAuthorized` so that only `onBehalf == msg.sender` (the account itself) can grant new authorizations. Authorized accounts should be able to act *as* the user but not *extend* the user's trust graph:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender, Unauthorized()); // only self can grant
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
}
```

Alternatively, if delegation of authorization is intentional, add a separate `revokeAll` mechanism or emit events that allow off-chain enumeration of all active sub-authorizations so users can clean up after revoking a compromised account.

### Proof of Concept

```solidity
// Alice authorizes Bob (e.g., a periphery contract)
vm.prank(alice);
midnight.setIsAuthorized(bob, true, alice);

// Bob (malicious) plants Charlie as a backdoor
vm.prank(bob);
midnight.setIsAuthorized(charlie, true, alice);

// Alice revokes Bob — believes she is safe
vm.prank(alice);
midnight.setIsAuthorized(bob, false, alice);

// Charlie still authorized — drains Alice's collateral
vm.prank(charlie);
midnight.withdrawCollateral(market, 0, aliceCollateral, alice, charlie);
// Charlie receives Alice's collateral; Alice has no recourse
