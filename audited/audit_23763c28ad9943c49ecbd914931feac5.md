Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via zero-rounding `buyerAssets` — (`src/Midnight.sol`)

## Summary
When a buy offer is taken with `maxAssets > 0` and any tick below `MAX_TICK`, integer division in `mulDivDown(units, buyerPrice, WAD)` floors `buyerAssets` to zero for small `units` values (e.g. `units = 1`). Because `consumed[maker][group]` is incremented by `buyerAssets`, it stays at its prior value, and the guard `require(newConsumed <= offer.maxAssets)` passes trivially on every call. The offer can be taken an unbounded number of times beyond the maker's intended cap, with the maker accumulating credit and the taker accumulating debt, but zero tokens transferred per fill.

## Finding Description

**Exact code path** — `src/Midnight.sol` lines 358–369:

```solidity
uint256 offerPrice  = TickLib.tickToPrice(offer.tick);                    // line 358
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice; // 361
uint256 buyerPrice  = sellerPrice + _settlementFee;                        // 362 → equals offerPrice for buy offers
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)                                    // 363 ← rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group]
        += offer.buy ? buyerAssets : sellerAssets;                         // 368 ← += 0 when buyerAssets == 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());             // 369 ← never fires
}
``` [1](#0-0) 

**Root cause** — `TickLib.tickToPrice` returns values strictly less than `WAD` (1e18) for every tick below `MAX_TICK = 5820`. The formula in `TickLib.sol` lines 44–52 computes `1e36 / (1e18 + wExp(...))`, which equals `WAD` only at `MAX_TICK` and is strictly smaller for all lower ticks (including `tickToPrice(0) = 0` after rounding to `PRICE_ROUNDING_STEP`). [2](#0-1) 

For a buy offer, `buyerPrice = offerPrice = tickToPrice(tick) < WAD`. With `units = 1`, `mulDivDown(1, buyerPrice, WAD) = (1 * buyerPrice) / WAD = 0` by integer division. The consumed mapping is incremented by 0, `newConsumed` stays at its prior value, and the cap check passes unconditionally.

**Token transfer** — Lines 455–456 call `safeTransferFrom(..., buyerAssets - sellerAssets)` and `safeTransferFrom(..., sellerAssets)`. When both are 0, these are zero-value transfers. The `SafeTransferLib` implementation (lines 24–34) forwards the call directly to the token's `transferFrom`; standard ERC-20 tokens accept zero-value transfers, so the take completes successfully. [3](#0-2) [4](#0-3) 

**Position mutation** — Lines 408–414 still execute with `units = 1`: the maker (buyer) gains 1 unit of credit and the taker (seller) gains 1 unit of debt, with no token movement. [5](#0-4) 

**Exploit flow:**
1. Maker posts a buy offer: `maxAssets = X`, any `tick < MAX_TICK`, valid ratifier, valid expiry.
2. Attacker calls `take(offer, ..., units=1, ...)` in a loop (or via `multicall`).
3. Each iteration: `buyerAssets = 0`; `consumed` unchanged; cap check passes; maker credit +1; taker debt +1; 0 tokens transferred.
4. After N iterations, maker holds N units of credit while `consumed[maker][group]` remains 0, far exceeding `maxAssets`.

**Why existing checks fail:**
- The `require(newConsumed <= offer.maxAssets)` guard is the sole enforcement of the cap and relies entirely on `buyerAssets` being non-zero — which is not guaranteed.
- `Consume.spec` line 68 scopes `takeConsumedDelta` to `maxAssets == 0` only; no analogous delta rule exists for assets mode. [6](#0-5) 

- `mulDivDown` and `mulDivUp` are summarized as `NONDET` (lines 11–12), so the formal proofs never model the rounding-to-zero case. [7](#0-6) 

- `takeConsumedBoundedByMax` (line 62) only asserts `consumed <= maxAssets` after a single take; it does not assert that consumed strictly increases when `units > 0`. [8](#0-7) 

## Impact Explanation
The `maxAssets` cap — the sole mechanism preventing a buy offer from being filled beyond the maker's stated limit — is completely bypassed. The maker accumulates unbounded credit exposure (claims on future repayment at maturity) without any corresponding token inflow per fill. This violates the core protocol invariant that offers cannot be overfilled. At maturity, the maker holds credit positions backed by taker debt that was created without token deposit, creating an accounting imbalance. The attacker's cost is the debt they accumulate (self-harmful griefing), but the maker's exposure grows without bound regardless.

## Likelihood Explanation
**Preconditions**: (1) A buy offer with `maxAssets > 0` — a normal maker configuration. (2) Any tick below `MAX_TICK` — satisfied by virtually every real offer since `MAX_TICK` is the par tick. (3) Attacker willing to accumulate debt at zero token cost per unit. No privileged access, no oracle manipulation, and no special token behavior is required. The attack is repeatable in a single transaction via `multicall` or across multiple blocks. [9](#0-8) [10](#0-9) 

## Recommendation
Add a guard ensuring the consumed increment is non-zero whenever `units > 0` in assets mode. The minimal fix is to require `buyerAssets > 0` before updating `consumed` for buy offers:

```solidity
if (offer.maxAssets > 0) {
    require(buyerAssets > 0 || units == 0, ZeroAssetIncrement());
    newConsumed = consumed[offer.maker][offer.group]
        += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, use `mulDivUp` (instead of `mulDivDown`) for the `buyerAssets` calculation when tracking consumed in assets mode, ensuring any non-zero `units` produces a non-zero increment. Additionally, the Certora `Consume.spec` should add a delta rule for assets mode analogous to `takeConsumedDelta`, and replace the `NONDET` summaries for `mulDivDown`/`mulDivUp` with concrete models when rounding behavior is security-relevant.

## Proof of Concept
**Minimal Foundry test plan:**

```solidity

    // 1. Deploy protocol, create market with tick spacing 1.

    // 3. Assert consumed[maker][group] == 0.
    // 4. Attacker calls take(offer, ..., units=1, ...) 10 times.
    // 5. Assert consumed[maker][group] == 0  (cap never incremented).
    // 6. Assert maker.credit == 10           (credit grew beyond maxAssets=1e18 in token terms).
    // 7. Assert taker.debt == 10             (attacker accumulated debt for free).
    // Confirm: buyerAssets returned by take == 0 on each call.

```

Key inputs: `offer.buy = true`, `offer.maxAssets = 1e18`, `offer.tick = 1`, `units = 1`. Expected: all 10 `take` calls succeed, `consumed` stays 0, maker credit = 10, taker debt = 10, 0 tokens transferred per call.

rule takeConsumedDelta(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumed(offer.maker, offer.group) == consumedBefore + units;
