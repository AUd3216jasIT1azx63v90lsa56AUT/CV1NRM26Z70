Audit Report

## Title
LiquidatorGate checks `msg.sender` but loan tokens are pulled from `callback` payer, enabling gate bypass - (File: src/Midnight.sol)

## Summary
The `liquidate` function enforces the `liquidatorGate` access control exclusively against `msg.sender`, but when `callback != address(0)` the actual loan token repayment is pulled from the `callback` address and collateral is sent to `receiver`. A whitelisted address A can call `liquidate` with `callback=B` and `receiver=B`, making gate-blocked address B the true economic actor (paying loan tokens, receiving collateral) while the gate never checks B.

## Finding Description
Three code locations combine to create the bypass:

**Gate check** (`src/Midnight.sol` lines 597–600): Only `msg.sender` is validated against the gate.
```solidity
require(
    market.liquidatorGate == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
```

**Payer assignment** (`src/Midnight.sol` line 679): When a callback is provided, the payer becomes the callback address, not `msg.sender`.
```solidity
address payer = callback != address(0) ? callback : msg.sender;
```

**Token pull** (`src/Midnight.sol` line 717): Loan tokens are pulled from `payer` (the callback address).
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```

**Collateral transfer** (`src/Midnight.sol` line 696): Collateral is sent to `receiver`, which is caller-controlled and can be set to B.
```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
```

**Exploit flow:**
1. Market has a `liquidatorGate` that whitelists A and blocks B.
2. B deploys a contract implementing `ILiquidateCallback`, pre-approves Midnight to pull loan tokens from it.
3. A (whitelisted) calls `midnight.liquidate(market, ..., receiver=B, callback=B, ...)`.
4. Gate calls `canLiquidate(A)` → `true` → passes.
5. Collateral is transferred to B (`receiver=B`).
6. `onLiquidate` is called on B; B returns `CALLBACK_SUCCESS`.
7. Loan tokens are pulled from B (`payer=B`).
8. B has paid the loan tokens and received the collateral — the full economic outcome of a liquidation — despite being blocked by the gate.

**Why existing checks fail:** The Certora rule `liquidatorGateBlocksLiquidation` (`certora/specs/Reverts.spec` lines 324–332) only verifies that `canLiquidate(msg.sender)` returning false causes a revert. It does not constrain the `callback` or `receiver` addresses against the gate. The `OnlyExplicitPayerCanLoseTokens.spec` rule (line 128) explicitly sets `liquidateCallbackAllowed = true` for `liquidate`, confirming the callback-as-payer path is intentional but the gate gap is unaddressed. The existing `GateTest.sol` liquidator gate tests (lines 269–300) only test with `callback=address(0)` and never exercise the callback bypass path.

## Impact Explanation
The `liquidatorGate` access control is fully bypassed for the actual economic actor. Any market relying on `liquidatorGate` to enforce KYC, regulatory, or trust-based restrictions on liquidators has those restrictions rendered ineffective. A gate-blocked address B can pay for and profit from liquidations (receiving seized collateral) in a market configured to exclude it. This is a direct authorization bypass with concrete asset movement consequences — B receives collateral it is not permitted to receive under the market's access control policy.

## Likelihood Explanation
The attack requires a whitelisted liquidator A to submit the transaction with `callback=B`. If A is a public/permissionless liquidation router contract (a common DeFi pattern), any external user can supply the `callback` argument directly, requiring zero cooperation from A. Even with a private EOA as A, the collusion is low-friction: B deploys a callback contract, pre-approves Midnight, and asks any whitelisted address to submit one transaction. The condition is repeatable on every liquidatable position in every gated market. The `live_context.json` test scenario `liquidator_gate_bypass` (lines 302–308) explicitly lists "alternate actor/callback/authorized address" as a bypass vector to test, indicating the protocol designers identified this surface but the gap remains unaddressed in both the implementation and the formal verification rules.

## Recommendation
Extend the gate check to also validate the `callback` address (and optionally `receiver`) when a callback is provided:
```solidity
require(
    market.liquidatorGate == address(0)
        || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
if (callback != address(0)) {
    require(
        market.liquidatorGate == address(0)
            || ILiquidatorGate(market.liquidatorGate).canLiquidate(callback),
        LiquidatorGatedFromLiquidating()
    );
}
```
Additionally, update the Certora rule `liquidatorGateBlocksLiquidation` to assert that when the gate blocks the callback address, `liquidate` with a non-zero callback also reverts.

## Proof of Concept
1. Deploy a `WhitelistGate` (as in `test/GateTest.sol`). Whitelist address A; do not whitelist address B.
2. Create a gated market with `liquidatorGate = address(gate)`.
3. Create an undercollateralized borrower position.
4. Deploy a callback contract at address B that implements `ILiquidateCallback.onLiquidate` returning `CALLBACK_SUCCESS`, and pre-approves Midnight to pull loan tokens from B.
5. From address A, call `midnight.liquidate(gatedMarket, 0, seizedAssets, 0, borrower, false, B, B, "")`.
6. Observe: the call succeeds (gate check passes for A), collateral is transferred to B, loan tokens are pulled from B.
7. Confirm: B has received collateral and paid loan tokens despite `gate.canLiquidate(B)` returning `false`.
