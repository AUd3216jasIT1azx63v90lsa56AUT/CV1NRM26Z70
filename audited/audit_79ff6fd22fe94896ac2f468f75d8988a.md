### Title
`onBuy` Callback Reentrancy Drains `withdrawable` Before `buyerAssets` Are Deposited - (`src/Midnight.sol`)

### Summary
In `take()`, `buyerPos.credit` is incremented at line 410 before the `onBuy` callback fires at line 448, and `safeTransferFrom` to pull `buyerAssets` does not execute until line 455 — after the callback returns. A malicious buyer-controlled callback can reenter `withdraw()` during `onBuy`, passing the authorization check because the buyer controls both the `onBehalf` address and the calling contract. This lets the callback drain `units` tokens from the market's pre-existing `withdrawable` balance before the corresponding `buyerAssets` have been deposited, breaking the invariant that credit is only redeemable once its backing assets are in the contract.

### Finding Description

**Exact code path:**

In `take()`, all state writes for the buyer complete before any external call:

```
buyerPos.credit += buyerCreditIncrease;   // line 410 — credit written
...
IBuyCallback(buyerCallback).onBuy(...)    // line 448 — external call, payment not yet pulled
...
safeTransferFrom(loanToken, payer, this, buyerAssets - sellerAssets);  // line 455 — payment pulled AFTER callback
``` [1](#0-0) 

`withdraw()` has no reentrancy guard and its only access control is:

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
``` [2](#0-1) 

**Attacker-controlled inputs and authorization bypass:**

For a buy offer (`offer.buy = true`): `buyer = offer.maker`, `buyerCallback = offer.callback`. The maker sets `offer.callback` to a contract they control. They either (a) make `offer.maker == offer.callback` (maker is a contract implementing `IBuyCallback`), or (b) call `setIsAuthorized(callback, true, buyer)` beforehand. Either satisfies `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` when the callback reenters `withdraw(market, units, buyer, attacker_receiver)`.

For a sell offer (`offer.buy = false`): `buyer = taker`, `buyerCallback = takerCallback`. The taker controls both and applies the same bypass.

**Exploit flow:**

1. Prior repayments have set `marketState[id].withdrawable = W ≥ units`.
2. Attacker calls `take(offer, ..., units, ...)` where `buyerCallback` is their malicious contract.
3. `take` writes `buyerPos.credit += units` (line 410).
4. `take` calls `onBuy` on the malicious callback (line 448).
5. Inside `onBuy`, callback calls `withdraw(market, units, buyer, attacker_receiver)`:
   - Authorization check passes.
   - `_position.credit -= units` → buyer credit returns to 0.
   - `_marketState.withdrawable -= units` → withdrawable = W − units.
   - `safeTransfer(loanToken, attacker_receiver, units)` → **units tokens leave the contract**.
6. `onBuy` returns `CALLBACK_SUCCESS`.
7. `take` resumes: `safeTransferFrom(loanToken, payer, address(this), buyerAssets − sellerAssets)` — only the settlement fee spread is deposited.

**Why no existing check stops it:**

- There is no transient reentrancy lock on `withdraw` (only `LIQUIDATION_LOCK_SLOT` guards liquidation during `take`). [3](#0-2) 
- The token safety requirements only prohibit token contracts from reentering Midnight on `transfer`/`transferFrom`; they say nothing about callbacks. [4](#0-3) 
- The Certora formal verification explicitly acknowledges this gap: *"Assumes no reentrancy: callbacks (onBuy, onSell) and token transfers are not modeled as re-entering Midnight, so re-entrant credit and debt changes are not covered."* [5](#0-4) 

### Impact Explanation

The contract transfers `units` loan tokens to the attacker's receiver from the existing `withdrawable` pool (funded by prior lender repayments), while only receiving `buyerAssets − sellerAssets` (the settlement fee spread) back. The deficit per attack is `units − (buyerAssets − sellerAssets) = units × sellerPrice / WAD`. At `tick = 0` (zero price, zero settlement fee), `buyerAssets = 0` and the attacker receives `units` tokens for free. At any tick where `buyerPrice < WAD`, the attacker profits directly. At any tick, the `withdrawable` pool is drained by more than the assets deposited, leaving other lenders unable to redeem their credit. This is a direct, concrete loss of funds belonging to other lenders.

### Likelihood Explanation

**Preconditions:**
1. `marketState[id].withdrawable ≥ units` — satisfied whenever any borrower has repaid in the market (normal operation).
2. Buyer controls `buyerCallback` — always true; the buyer sets `offer.callback` (buy offer) or `takerCallback` (sell offer).
3. Authorization: buyer pre-authorizes callback, or buyer IS the callback contract — trivially achievable by the attacker in a single setup transaction.

All three preconditions are under full attacker control and require no special market state beyond normal usage. The attack is repeatable across any market with repaid debt and is not blocked by any existing guard.

### Recommendation

Add a transient reentrancy lock that is set at the start of `take()` and checked at the start of `withdraw()` (and any other state-mutating function that should not be reenterable during a callback). Alternatively, move all state writes for the buyer's credit to after the `safeTransferFrom` calls (i.e., adopt strict checks-effects-interactions ordering where the external call precedes the credit write). The minimal targeted fix is to mirror the `LIQUIDATION_LOCK_SLOT` pattern with a `WITHDRAW_LOCK_SLOT` (or a global `REENTRANT_LOCK_SLOT`) that is set before `onBuy` fires and cleared after `safeTransferFrom` completes.
