Audit Report

## Title
Re-entrant `take()` via `onBuy()` Bypasses State-Dependent `enterGate.canIncreaseCredit` Check - (File: src/Midnight.sol)

## Summary
`Midnight.take()` writes `buyerPos.credit` to persistent storage at line 410 before invoking the buyer's `onBuy()` callback at lines 445–453. No reentrancy guard protects `take()` itself — only the seller's liquidation lock is set. A malicious buyer callback can re-enter `take()` for the same buyer in the same gated market; the inner call's `canIncreaseCredit` gate check executes against the already-mutated credit state, allowing a credit-threshold gate (e.g., `creditOf(id, buyer) < X`) to be bypassed and the buyer to accumulate credit beyond the gate's intended cap.

## Finding Description

**Exact code path in `src/Midnight.sol`:**

1. **Gate check (lines 397–401):** `canIncreaseCredit(buyer)` is called against pre-update state. [1](#0-0) 

2. **State mutation (line 410):** `buyerPos.credit` is incremented and committed to persistent storage. [2](#0-1) 

3. **Buyer callback (lines 445–453):** `IBuyCallback(buyerCallback).onBuy(...)` is invoked with credit already committed. [3](#0-2) 

4. **Reentrancy guard scope (line 444):** The only transient lock set before callbacks is `LIQUIDATION_LOCK_SLOT`, which exclusively prevents the *seller* from being liquidated during callbacks. It does not prevent re-entry into `take()` itself. [4](#0-3) 

**Who controls `buyerCallback`:**

When `offer.buy == false` (taker is buyer, maker is seller), `buyerCallback = takerCallback`, which is a fully attacker-supplied parameter with no privilege requirement. [5](#0-4) 

**Exploit flow:**

*Precondition:* A gated market exists where `canIncreaseCredit(buyer)` is implemented as `creditOf(id, buyer) < X` — a natural credit-cap design, explicitly permitted by the `external view` declaration in `IEnterGate`. [6](#0-5) 

1. Attacker (taker/buyer) calls outer `take()` on a sell offer with `takerCallback = maliciousContract`. Buyer credit = 0. Gate sees `0 < X` → passes.
2. Outer `take()` writes `buyerPos.credit = Y` (where `Y < X`) to storage.
3. Outer `take()` calls `maliciousContract.onBuy(...)`.
4. `maliciousContract` re-enters `take()` for the same buyer, targeting a different sell offer in the same market (different maker/group, so `consumed` check is independent).
5. Inner `take()`: gate calls `canIncreaseCredit(buyer)` → reads `creditOf = Y`. Since `Y < X`, returns `true`. Gate check passes.
6. Inner `take()` writes `buyerPos.credit = Y + Z` to storage.
7. Both calls complete. Final credit = `Y + Z > X`. Gate's cap is violated.

**Why existing checks fail:**

- `LIQUIDATION_LOCK_SLOT` only guards the seller against liquidation; it is not a general reentrancy guard for `take()`. [7](#0-6) 

- The `consumed` mapping prevents re-taking the *same* offer beyond its cap, but does not prevent taking a *different* offer in the same market during the callback. [8](#0-7) 

- The Certora spec (`certora/specs/Reverts.spec` lines 30–34) explicitly models all callbacks as pure (no state changes), with the comment: *"For gate rules, gate checks precede callbacks so re-entrant state changes cannot affect them."* This is only true for the outer call's gate check. The inner call's gate check executes *after* the outer call's state mutation, which the spec does not model. [9](#0-8) 

- The `ghostCanIncreaseCredit` ghost function returns a fixed value per gate address and cannot model state-dependent gate logic. [10](#0-9) 

## Impact Explanation
A buyer whose callback is a malicious contract can accumulate credit in a gated market beyond the gate's intended per-user cap. Any gate whose `canIncreaseCredit` logic reads Midnight's own `creditOf` (or any state that `take()` mutates before the callback) is vulnerable. This directly violates the protocol invariant that gate checks must apply to all entry paths and that callbacks must not corrupt partial state through reentrancy. The impact is unauthorized credit accumulation — a direct accounting integrity failure and unauthorized state change in a permissioned market.

## Likelihood Explanation
All preconditions are realistic and require no privilege:
1. A gated market with a credit-threshold gate is a natural design for permissioned lending pools.
2. The attacker is the taker and sets `takerCallback` to a malicious contract — a fully attacker-controlled input.
3. A second valid sell offer must exist in the same market; the attacker can arrange this trivially (e.g., as a separate address they control, subject to the `offer.maker != taker` check).

The attack is executable within a single transaction, is repeatable, and requires no special timing, oracle manipulation, or leaked credentials.

## Recommendation
Add a per-call reentrancy guard to `take()` using transient storage (consistent with the existing `tExchange`/`tGet` pattern). A simple boolean lock keyed on the market `id` (or globally) set before any state mutation and cleared after the function completes would prevent re-entry. Alternatively, restructure `take()` to follow checks-effects-interactions strictly by moving all callbacks after token transfers and after all state is finalized, and adding a transient lock to prevent nested `take()` calls on the same market within the same transaction.

## Proof of Concept

**Minimal test plan (Foundry fork test):**

1. Deploy a `CreditCapGate` implementing `canIncreaseCredit(address buyer) external view returns (bool)` as `IMidnight(midnight).creditOf(id, buyer) < CAP`.
2. Create a gated market with `enterGate = address(creditCapGate)`.
3. Deploy `MaliciousCallback` implementing `onBuy(...)`: inside `onBuy`, call `midnight.take(secondOffer, ..., takerCallback=address(0), ...)` for the same buyer against a second sell offer in the same market.
4. Attacker calls `midnight.take(firstOffer, ..., takerCallback=address(maliciousCallback))`.
5. Assert: after both calls complete, `midnight.creditOf(id, attacker) > CAP`. Without the reentrancy, the inner call would revert because `creditOf = Y >= CAP` after the outer call's state update; with reentrancy, `Y < CAP` so the inner call passes.
