### Title
`withdraw()` Can Drain `withdrawable` Below `continuousFeeCredit`, Permanently Blocking `claimContinuousFee()` — (File: `src/Midnight.sol`)

---

### Summary

The `withdraw()` function decrements `marketState.withdrawable` without reserving the fee claimer's accrued share (`continuousFeeCredit`). Because `claimContinuousFee()` also draws from `withdrawable`, any lender can drain `withdrawable` below `continuousFeeCredit`, causing `claimContinuousFee()` to revert with an arithmetic underflow. This is the direct analog of the external report: a normal user action (`withdraw`) silently corrupts a shared accounting variable in a way that blocks a key protocol function (`claimContinuousFee`).

---

### Finding Description

**Accounting variables involved:**

| Variable | Incremented by | Decremented by |
|---|---|---|
| `withdrawable` | `repay`, `liquidate` | `withdraw`, `claimContinuousFee` |
| `continuousFeeCredit` | `_updatePosition` | `claimContinuousFee`, `liquidate` (bad debt) |

**Root cause — two independent paths drain the same pool:**

`_updatePosition()` (called inside `withdraw()`) accrues the continuous fee into `continuousFeeCredit` but does **not** increment `withdrawable`: [1](#0-0) 

`claimContinuousFee()` then expects to draw `amount` from **both** `continuousFeeCredit` and `withdrawable`: [2](#0-1) 

Meanwhile, `withdraw()` decrements `withdrawable` with no awareness of the fee claimer's reserved share: [3](#0-2) 

There is no invariant enforcement that keeps `withdrawable >= continuousFeeCredit`. A lender can legally withdraw all of `withdrawable`, leaving `withdrawable < continuousFeeCredit`. The next call to `claimContinuousFee()` with `amount = continuousFeeCredit` hits the underflow at line 320 and reverts.

---

### Impact Explanation

`claimContinuousFee()` is the only mechanism for the protocol to collect continuous fee revenue. When `withdrawable < continuousFeeCredit`, the fee claimer cannot claim the full accrued amount — the call reverts. The fee claimer can pass a smaller `amount` (up to `withdrawable`), but the remainder is stuck until new repayments arrive. An attacker who front-runs every `repay()` with a matching `withdraw()` can sustain this DoS indefinitely across the market's lifetime, causing the protocol to lose all continuous fee revenue for that market. [4](#0-3) 

---

### Likelihood Explanation

Any lender holding credit in the market can trigger this with zero cost beyond gas — they are simply withdrawing their own funds. No privileged access, no flash loan, no oracle manipulation is required. The condition (`withdrawable < continuousFeeCredit`) arises naturally whenever continuous fees have accrued but lenders withdraw before the fee claimer acts. In active markets with many lenders, this race condition is highly probable even without a deliberate attacker. [5](#0-4) 

---

### Recommendation

Reserve the fee claimer's share before allowing lender withdrawals. The simplest fix is to enforce the invariant inside `withdraw()`:

```solidity
_marketState.withdrawable -= UtilsLib.toUint128(units);
require(_marketState.withdrawable >= _marketState.continuousFeeCredit, InsufficientWithdrawable());
```

Alternatively, track `continuousFeeCredit` in a separate balance that is incremented when `_updatePosition` accrues fees (mirroring the external report's fix of tracking withdrawn assets directly), so that `claimContinuousFee` draws from its own dedicated pool rather than competing with lenders over `withdrawable`.

---

### Proof of Concept

**Setup:**
- Market with 1 000 credit units (lender A) and 1 000 debt units (borrower B)
- `withdrawable = 0`, `continuousFeeCredit = 0`

**Step 1 — Borrower repays 200 units:**
```
repay(market, 200, borrowerB, ...)
→ withdrawable = 200
``` [6](#0-5) 

**Step 2 — Time passes; continuous fee accrues (e.g. 120 units) when lender A's position is updated:**
```
_updatePosition called (inside withdraw or take)
→ continuousFeeCredit = 120
→ lender A credit = 880
→ withdrawable unchanged = 200
``` [1](#0-0) 

**Step 3 — Lender A withdraws 150 units (valid: credit=880, withdrawable=200):**
```
withdraw(market, 150, lenderA, lenderA)
→ withdrawable = 200 − 150 = 50
→ totalUnits   = 1000 − 150 = 850
→ lender A credit = 730
``` [3](#0-2) 

**Step 4 — Fee claimer attempts to claim 120 (= continuousFeeCredit):**
```
claimContinuousFee(market, 120, feeReceiver)
→ continuousFeeCredit -= 120  ✓  (= 0)
→ totalUnits          -= 120  ✓  (= 730)
→ withdrawable        -= 120  ✗  50 − 120 → ARITHMETIC UNDERFLOW → REVERT
``` [2](#0-1) 

**Result:** `claimContinuousFee` is permanently blocked until at least 70 more units are repaid. An attacker repeating Step 3 after every repayment sustains the DoS indefinitely.

    function claimContinuousFee(Market memory market, uint256 amount, address receiver) external {
        bytes32 id = toId(market);
        MarketState storage _marketState = marketState[id];
        require(msg.sender == feeClaimer, OnlyFeeClaimer());
        require(_marketState.tickSpacing > 0, MarketNotCreated());

        _marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
        _marketState.totalUnits -= UtilsLib.toUint128(amount);
        _marketState.withdrawable -= UtilsLib.toUint128(amount);

        emit EventsLib.ClaimContinuousFee(msg.sender, id, amount, receiver);

        SafeTransferLib.safeTransfer(market.loanToken, receiver, amount);
    }
