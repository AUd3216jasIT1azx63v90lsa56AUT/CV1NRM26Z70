### Title
`buyerAssetsToUnits` unconditionally reverts for valid high-tick sell offers with nonzero settlement fees, DoS-ing `buyWithAssetsTargetAndWithdrawCollateral` - (File: src/periphery/TakeAmountsLib.sol)

### Summary
`TakeAmountsLib.buyerAssetsToUnits` contains a hard `require(buyerPrice <= WAD)` guard that fires for any sell offer where `tickToPrice(tick) + settlementFee > WAD`. `Midnight.take` has no equivalent guard for sell offers and succeeds in exactly this scenario. Because `MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` calls `buyerAssetsToUnits` without a try/catch, the entire lend-bundle transaction reverts for any sell offer at a high tick in a market with a nonzero settlement fee.

### Finding Description
**Code path in `buyerAssetsToUnits`** (`src/periphery/TakeAmountsLib.sol`, lines 22–28):
```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);          // = WAD when tick = MAX_TICK
uint256 settlementFee = IMidnight(midnight).settlementFee(...); // > 0 by precondition
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice; // sell: = offerPrice
uint256 buyerPrice  = sellerPrice + settlementFee;             // = WAD + settlementFee > WAD
require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne());     // REVERTS
```

**Code path in `Midnight.take`** (`src/Midnight.sol`, lines 358–364):
```solidity
uint256 offerPrice  = TickLib.tickToPrice(offer.tick);
uint256 _settlementFee = settlementFee(id, timeToMaturity);
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice; // sell: = offerPrice
uint256 buyerPrice  = sellerPrice + _settlementFee;            // = WAD + settlementFee > WAD
uint256 buyerAssets = offer.buy ? ... : units.mulDivUp(buyerPrice, WAD);   // NO revert guard
```

`take` has no `require(buyerPrice <= WAD)` for sell offers. It computes `buyerAssets` and continues normally.

**Exploit flow:**
1. A borrower posts a sell offer with `tick = MAX_TICK` (a valid, accessible tick).
2. The market has any nonzero settlement fee (normal protocol state).
3. A lender calls `MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` with that offer.
4. Inside the loop, `TakeAmountsLib.buyerAssetsToUnits` is called (line 209 of `MidnightBundles.sol`).
5. `buyerPrice = WAD + settlementFee > WAD` → `require` reverts with `PriceGreaterThanOne`.
6. The revert is not caught (no try/catch around `buyerAssetsToUnits`), so the entire bundle call reverts.
7. The lender cannot lend via the bundle against this offer, even though `Midnight.take` would succeed.

**Why existing checks fail:** The `require(buyerPrice <= WAD)` in `buyerAssetsToUnits` is intentional (documented: "not all buyerAssets are reachable then"), but it is overly conservative relative to `take`'s actual behavior for sell offers. `take` imposes no such restriction on sell offers and will succeed with `buyerPrice > WAD`.

### Impact Explanation
`MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` is permanently DoS'd for any sell offer where `tickToPrice(tick) + settlementFee > WAD`. At `tick = MAX_TICK`, this is triggered by any nonzero settlement fee. Lenders cannot use the bundle flow to lend against these offers; they must call `Midnight.take` directly, bypassing the bundle's collateral-withdrawal and referral-fee logic.

### Likelihood Explanation
`MAX_TICK = 5820` is a valid, accessible tick. Settlement fees are a standard protocol feature enabled by market creators. Any market with a nonzero settlement fee and a sell offer at or near `MAX_TICK` triggers this. No special permissions or unusual state are required. The condition is repeatable and deterministic.

### Recommendation
In `MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral`, wrap the `buyerAssetsToUnits` call in a try/catch and skip the offer on revert (consistent with how `take` itself is already wrapped in a try/catch). Alternatively, add a pre-check in the loop:

```solidity
// Skip offers where buyerPrice > WAD (take still works but units-to-assets inversion is undefined)
try TakeAmountsLib.buyerAssetsToUnits(...) returns (uint256 units_) {
    unitsToTake = min(units_, ...);
} catch {
    continue;
}
```

A deeper fix would be to handle `buyerPrice > WAD` in `buyerAssetsToUnits` by returning a snapped/clamped unit count rather than reverting, aligning its reachability contract with `take`'s actual behavior.
