### Title
Authorized delegate can MEV-sandwich ratify/unratify to DoS takers using `SetterRatifier` - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier.setIsRootRatified` is a public function with no time-lock or commit-reveal protection, allowing any address authorized by the maker to toggle `isRootRatified[maker][root]` freely within a single block. A malicious authorized delegate can ratify a root, let a taker's `take` transaction enter the mempool, then front-run it with an unratify call, causing `Midnight.take` to revert with `NotRatified` at the `isRatified` check. No existing guard prevents this ordering.

### Finding Description
**Code path:**
1. Precondition: `IMidnight.isAuthorized(maker, attacker) == true` (maker has authorized the attacker via `setIsAuthorized`).
2. Attacker calls `SetterRatifier.setIsRootRatified(maker, root, true)` → passes the `isAuthorized` check at line 25, sets `isRootRatified[maker][root] = true`.
3. Taker submits `Midnight.take(offer, ratifierData, ...)` to the mempool.
4. Attacker, observing the pending take, submits `SetterRatifier.setIsRootRatified(maker, root, false)` with higher priority (MEV bundle or higher gas).
5. Block ordering: attacker's unratify tx executes first → `isRootRatified[maker][root] = false`.
6. Taker's `take` executes: `Midnight.take` calls `IRatifier(offer.ratifier).isRatified(offer, ratifierData)` (line 356), which reaches `require(isRootRatified[offer.maker][root], NotRatified())` (line 35) and reverts.

**Root cause:** `setIsRootRatified` imposes no ordering constraint, time-lock, or minimum validity window. The `bool` is a live storage slot read atomically at `take` time. An authorized delegate has symmetric write access to both `true` and `false` states with no cooldown.

**Why checks fail:** The only guard is the authorization check (line 25), which the attacker passes by design. There is no check preventing a delegate from unratifying a root that was just ratified, nor any mechanism ensuring a ratified root remains valid for a minimum duration.

### Impact Explanation
A malicious authorized delegate can reliably cause any taker's `take` on a `SetterRatifier`-backed offer to revert with `NotRatified`, even after the taker has verified on-chain that the root is ratified. This is a targeted, repeatable MEV-based DoS: the taker wastes gas, the take fails, and the attacker can repeat the pattern indefinitely at low cost (two storage writes per grief).

### Likelihood Explanation
**Preconditions:**
- Maker must have authorized the attacker address (`isAuthorized[maker][attacker] == true`). This is realistic: makers may authorize smart contracts, bots, or third-party operators to manage their offer trees.
- Attacker must be able to influence tx ordering within a block (MEV searcher, block builder, or simply submitting with higher gas on networks without strict ordering).

**Feasibility:** Both conditions are achievable in production. The attack is cheap (two `SSTORE` operations), deterministic, and repeatable every block. The attacker does not need to hold any tokens or positions.

### Recommendation
Restrict `setIsRootRatified` so that only `maker == msg.sender` can set `newIsRootRatified = false` (unratify), while authorized delegates may only ratify (`true`). Alternatively, introduce a time-lock (e.g., a minimum block delay between ratification and unratification) or a commit-reveal scheme so that unratification cannot take effect in the same block as ratification. The simplest targeted fix:

```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    if (!newIsRootRatified) {
        // Only the maker themselves can unratify
        require(maker == msg.sender, Unauthorized());
    } else {
        require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    }
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
```
