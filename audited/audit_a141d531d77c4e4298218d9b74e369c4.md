### Title
Sponsor can update `recoveryAddress` post-stake to redirect all CORRUPTED funds to themselves - (File: src/ConfidencePool.sol)

### Summary
`setRecoveryAddress` carries no post-stake lock, so the pool sponsor can silently redirect the CORRUPTED sweep destination to their own address at any point after stakers have committed funds. Because `claimCorrupted` and `sweepUnclaimedCorrupted` read `recoveryAddress` at execution time rather than at snapshot time, the sponsor can steal the entire pool under any CORRUPTED resolution path — including the permissionless auto-CORRUPTED backstop that requires no moderator cooperation.

### Finding Description
`expiry` is explicitly locked after the first stake via the one-way `expiryLocked` flag, and pool scope is locked once the registry leaves pre-attack staging. `recoveryAddress` has no analogous guard:

```solidity
// src/ConfidencePool.sol L611-618
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    address oldRecoveryAddress = recoveryAddress;
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
```

No check on `expiryLocked`, `claimsStarted`, `outcome`, or any other state gate. The sponsor can call this function freely even after `flagOutcome` has been called and even after `claimsStarted` is set to `true` by `claimExpired`.

Both CORRUPTED sweep functions read `recoveryAddress` at call time:

```solidity
// claimCorrupted L423
stakeToken.safeTransfer(recoveryAddress, toSweep);

// sweepUnclaimedCorrupted L468
stakeToken.safeTransfer(recoveryAddress, amount);
```

The value is never snapshotted at `flagOutcome` or `claimExpired` resolution time.

### Impact Explanation
Under bad-faith CORRUPTED, `claimCorrupted` sweeps `stakeToken.balanceOf(address(this))` — the entire pool including all staker principal and bonus — to whatever `recoveryAddress` is at that moment. A sponsor who changes `recoveryAddress` to their own address between resolution and the sweep call receives all staker funds. Under good-faith CORRUPTED, `sweepUnclaimedCorrupted` similarly sweeps the remainder to `recoveryAddress` after the 180-day bounty window. The impact is total loss of staker principal and bonus.

### Likelihood Explanation
The auto-CORRUPTED backstop path (`claimExpired` after `expiry + MODERATOR_CORRUPTED_GRACE`) requires no moderator cooperation and is permissionlessly callable by the sponsor themselves. The sponsor can atomically: (1) call `claimExpired` to lock in bad-faith CORRUPTED, (2) call `setRecoveryAddress(sponsorAddress)`, (3) call `claimCorrupted()`. The only prerequisite is that the registry is `CORRUPTED` and `riskWindowStart != 0` — both observable on-chain. The moderator-flagged bad-faith path is equally exploitable: the sponsor changes `recoveryAddress` after the moderator's `flagOutcome` call but before anyone calls `claimCorrupted`.

### Recommendation
Lock `recoveryAddress` after the first stake, mirroring the `expiryLocked` pattern already used for `expiry`:

```solidity
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (expiryLocked) revert RecoveryAddressLocked(); // add analogous guard
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    ...
}
```

Alternatively, snapshot `recoveryAddress` into an immutable local variable inside `flagOutcome` / `claimExpired` at resolution time (alongside the other snapshot fields), and use that frozen value in all subsequent sweep calls.

### Proof of Concept
1. Sponsor creates pool with `recoveryAddress = 0xLegit` and `minStake = 1e18`.
2. Stakers verify `recoveryAddress` on-chain and deposit; `expiryLocked` flips `true`; `totalEligibleStake = S`.
3. Registry transitions to `UNDER_ATTACK`; `riskWindowStart` is sealed.
4. Registry transitions to `CORRUPTED`; `riskWindowEnd` is sealed.
5. Pool expires; moderator does not act for 180 days (`MODERATOR_CORRUPTED_GRACE`).
6. Sponsor calls `claimExpired()` → auto-CORRUPTED branch fires: `outcome = CORRUPTED`, `claimsStarted = true`, `corruptedReserve = S + bonus`.
7. Sponsor calls `setRecoveryAddress(sponsorAddress)` — succeeds with no revert (no lock on `claimsStarted` or `outcome`).
8. Sponsor calls `claimCorrupted()` → `stakeToken.safeTransfer(sponsorAddress, S + bonus)`.

All staker principal and bonus are now in the sponsor's wallet. No moderator involvement required. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
