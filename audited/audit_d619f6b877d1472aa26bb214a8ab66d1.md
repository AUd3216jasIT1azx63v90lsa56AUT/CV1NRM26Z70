### Title
Fee-on-Transfer Loan Token Inflates `marketState.withdrawable` Beyond Actual Balance in `liquidate` - (File: src/Midnight.sol)

### Summary
`Midnight.liquidate` increments `marketState.withdrawable` by the full `repaidUnits` before calling `SafeTransferLib.safeTransferFrom`, which only verifies the boolean return of `transferFrom` and does not measure the actual received balance. When the loan token charges a transfer fee, the contract receives `repaidUnits*(1-fee)` but records `repaidUnits` as withdrawable, creating a permanent balance shortfall that causes lenders to be unable to fully withdraw.

`SafeTransferLib.safeTransferFrom` only checks the boolean return value: [3](#0-2) 

It does not snapshot `balanceOf(address(this))` before and after the call, so a fee-on-transfer token silently delivers less than `repaidUnits`.

**Market creation has no loanToken validation:** [4](#0-3) 

`touchMarket` validates collateral params (LLTV, maxLif, sorting) but accepts any address as `loanToken`. An unprivileged market creator can deploy a market whose `loanToken` is a fee-on-transfer token (e.g., PAXG, or USDT with fees enabled).

**Exploit flow:**
1. Market creator calls `touchMarket` with a fee-on-transfer token as `loanToken` (fee = F%).
2. Lenders supply credit via `take`; borrowers borrow.
3. Borrower becomes unhealthy (oracle price drop or post-maturity).
4. Liquidator calls `liquidate(..., repaidUnits=R, ...)`.
5. `_marketState.withdrawable += R` and `_position.debt -= R` are written.
6. `safeTransferFrom` transfers `R` tokens but only `R*(1-F)` arrive in the contract.
7. `withdrawable` now exceeds `balanceOf(loanToken) - claimableSettlementFee` by `R*F`.
8. Repeated liquidations (or `repay` calls, which have the same pattern at lines 509/520) accumulate the shortfall.
9. When lenders call `withdraw`, the last lenders to withdraw receive nothing — the contract is insolvent.

**Why existing checks fail:**
- `SafeTransferLib` has no balance-delta check.
- `touchMarket` has no token whitelist or fee-detection check.
- The Certora solvency invariants explicitly assume non-fee tokens (Solvency.spec line 31–33), so they do not catch this. [5](#0-4) 

### Impact Explanation
`marketState.withdrawable` permanently exceeds the actual loan token balance held by the contract. Lenders calling `withdraw` will drain the real balance before all withdrawable units are redeemed, causing the last lenders to receive zero tokens. The protocol becomes insolvent for that market. The shortfall is proportional to `totalRepaidUnits * feeRate` and grows with every liquidation and repayment.

### Likelihood Explanation
- **Preconditions:** A market must be created with a fee-on-transfer loan token. Market creation is permissionless (`touchMarket` is callable by anyone). Real fee-on-transfer tokens exist (PAXG ~0.02%, USDT with fees enabled, various DeFi tokens).
- **Feasibility:** Any unprivileged market creator can set this up. The liquidator need not be the same actor — any liquidator calling `liquidate` on such a market triggers the shortfall.
- **Repeatability:** Every `liquidate` and `repay` call on the affected market compounds the shortfall. The attack is fully repeatable and deterministic.

### Recommendation
Add a balance-before/after check in `SafeTransferLib.safeTransferFrom`, or add a wrapper in `liquidate` and `repay` that asserts the received amount equals the expected amount:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
require(IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore == repaidUnits, FeeOnTransferToken());
```

Alternatively, add a token whitelist enforced in `touchMarket` that rejects fee-on-transfer tokens, or document and enforce via `SafeTransferLib` that only standard ERC20 tokens are supported and revert if the received amount differs.

        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
