### Title
Pre-signed EcrecoverAuthorizer nonce replay re-grants revoked operator authorization - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
When a maker revokes an operator via `Midnight.setIsAuthorized` directly, the EcrecoverAuthorizer's per-maker nonce is not incremented. If the maker previously signed an `Authorization` struct with a future nonce (e.g., nonce==1) and the `ecrecoverAuthorizer` contract remains authorized on Midnight, the operator can submit that pre-signed authorization to `EcrecoverAuthorizer.setIsAuthorized` and re-grant themselves authorization, bypassing the revocation. The invariant that revocation is permanent unless the maker explicitly re-grants is broken.

### Finding Description
The root cause is a nonce desync between the two authorization layers. `EcrecoverAuthorizer` maintains `mapping(address => uint256) public nonce` per authorizer, incremented only when `EcrecoverAuthorizer.setIsAuthorized` is called successfully. [1](#0-0) 

When the maker revokes the operator by calling `Midnight.setIsAuthorized(operator, false, maker)` directly: [2](#0-1) 

This sets `isAuthorized[maker][operator] = false` but leaves `nonce[maker]` in `EcrecoverAuthorizer` unchanged (still 1). The `ecrecoverAuthorizer` contract itself remains authorized on Midnight (`isAuthorized[maker][ecrecoverAuthorizer] = true`), since the maker only revoked the operator, not the authorizer contract.

The operator then submits the pre-signed `Authorization{authorizer: maker, authorized: operator, isAuthorized: true, nonce: 1}` to `EcrecoverAuthorizer.setIsAuthorized`. The checks at lines 25–36 all pass: [3](#0-2) 

- Deadline: not expired
- Nonce: `1 == nonce[maker]` (still 1, never incremented by the direct Midnight revocation)
- Signer: `maker` signed it, so `signer == authorization.authorizer` passes unconditionally

Then at line 46–47, `EcrecoverAuthorizer` calls `Midnight.setIsAuthorized(operator, true, maker)` as `msg.sender = ecrecoverAuthorizer`: [4](#0-3) 

Midnight's guard checks `isAuthorized[maker][ecrecoverAuthorizer]`, which is still `true`, so the call succeeds and `isAuthorized[maker][operator]` is set back to `true`.

### Impact Explanation
The revoked operator regains `isAuthorized[maker][operator] = true` on Midnight. They can then call `Midnight.setConsumed(group, amount, maker)` to manipulate the maker's consumed state for any offer group, enabling offer replay or cancellation of the maker's offers without consent. [5](#0-4) 

### Likelihood Explanation
Preconditions are realistic: the maker must have (a) authorized `ecrecoverAuthorizer` on Midnight, (b) used it to authorize the operator (nonce 0→1), and (c) pre-signed a nonce==1 authorization that the operator retained. The maker then revokes via `Midnight.setIsAuthorized` directly (the natural revocation path) without also revoking `ecrecoverAuthorizer`. The operator can replay the pre-signed authorization at any time before its deadline. The attack is repeatable for each pre-signed future nonce the operator holds.

### Recommendation
Provide a dedicated nonce-invalidation function on `EcrecoverAuthorizer` (e.g., `invalidateNonces(address authorizer)`) that increments `nonce[authorizer]` without granting any authorization, callable by the authorizer or anyone authorized by them on Midnight. Alternatively, document clearly that revoking via `Midnight.setIsAuthorized` does not invalidate pre-signed `EcrecoverAuthorizer` messages, and that makers must also revoke `ecrecoverAuthorizer` itself from Midnight or use `EcrecoverAuthorizer` to issue a revocation authorization (which increments the nonce).

```solidity
function testRevokedOperatorReauthorizesViaNonce1() public {
    // Setup: maker authorizes ecrecoverAuthorizer on Midnight
    vm.prank(maker);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, maker);

    // Maker pre-signs nonce==0 and nonce==1 authorizations for operator
    Authorization memory auth0 = Authorization({
        authorizer: maker, authorized: operator,
        isAuthorized: true, nonce: 0, deadline: block.timestamp + 1 days
    });
    Authorization memory auth1 = Authorization({
        authorizer: maker, authorized: operator,
        isAuthorized: true, nonce: 1, deadline: block.timestamp + 1 days
    });
    Signature memory sig0 = signAuthorization(auth0, makerKey);
    Signature memory sig1 = signAuthorization(auth1, makerKey); // operator retains this

    // Operator uses nonce==0 to get authorized (nonce becomes 1)
    ecrecoverAuthorizer.setIsAuthorized(auth0, sig0);
    assertEq(midnight.isAuthorized(maker, operator), true);

    // Maker revokes operator directly on Midnight (nonce in EcrecoverAuthorizer stays at 1)
    vm.prank(maker);
    midnight.setIsAuthorized(operator, false, maker);
    assertEq(midnight.isAuthorized(maker, operator), false);

    // Operator replays pre-signed nonce==1 Authorization — should revert but does not
    ecrecoverAuthorizer.setIsAuthorized(auth1, sig1);

    // Assertion: operator is re-authorized despite revocation — invariant violated
    assertEq(midnight.isAuthorized(maker, operator), true); // BUG: should be false
}
```

Expected assertion: `midnight.isAuthorized(maker, operator) == false` after revocation. Actual result: `true`, confirming the invariant violation.
