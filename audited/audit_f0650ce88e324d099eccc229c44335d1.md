Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via `mulDivDown` rounding to zero — (File: `src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and `tick < MAX_TICK`, a taker can call `take()` with a `units` value small enough that `units.mulDivDown(buyerPrice, WAD) == 0`. The `consumed` mapping is incremented by `buyerAssets` (which is 0), so the cap check `require(newConsumed <= offer.maxAssets)` always passes. Despite zero asset movement, `buyerPos.credit` and `sellerPos.debt` are still updated by the full `units` value, allowing the offer to be filled indefinitely beyond its stated cap.

## Finding Description

**Exact code path — `src/Midnight.sol` lines 363–369:**

```solidity
// line 363
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);

// lines 367–369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [1](#0-0) 

For buy offers, `consumed` is incremented by `buyerAssets`, which is computed with `mulDivDown`. When `units * buyerPrice < WAD`, integer truncation yields `buyerAssets = 0`, so `consumed` is never advanced.

**Root cause:** `tickToPrice` rounds prices to multiples of `PRICE_ROUNDING_STEP = 1e12`. [2](#0-1) [3](#0-2) 

At `MAX_TICK = 5820`, `tickToPrice(MAX_TICK) = WAD`. At any tick below `MAX_TICK`, `offerPrice < WAD`. For `units = 1`: `1 * (WAD - ε) / WAD = 0` in integer arithmetic. The bypass window spans `units ∈ [1, WAD/buyerPrice - 1]`, which can be millions of units wide at low ticks.

**Position effects still execute:** Even when `buyerAssets = 0`, lines 408–414 update positions based on `units`:

```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);  // based on units
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);  // based on units
``` [4](#0-3) 

No tokens transfer (`claimableSettlementFee += buyerAssets - sellerAssets = 0`), yet credit and debt grow. [5](#0-4) 

**Confirmed by existing test:** `testBugBuyMaxAssetsBypass()` in `test/TakeTest.sol` explicitly demonstrates the bypass — the offer is pre-consumed to `maxAssets`, then `take(1, ...)` succeeds, `consumed` stays at `maxAssets`, yet `creditOf(lender)` and `debtOf(borrower)` both increase. [6](#0-5) 

**Existing guard is insufficient:** The only protection is `require(newConsumed <= offer.maxAssets)`, which is trivially satisfied when the increment is 0. There is no floor check on `buyerAssets` or `units`. [7](#0-6) 

## Impact Explanation
The `maxAssets` cap on buy offers is completely unenforceable at any tick below `MAX_TICK`. A colluding maker-taker pair can give the maker unlimited credit at zero token cost: `buyerAssets = 0` means no tokens are transferred, yet `buyerPos.credit` grows without bound. The taker accumulates unbounded debt at zero cost. This violates the core protocol invariant that offers cannot be overfilled, and constitutes a direct accounting integrity failure enabling unauthorized credit creation.

## Likelihood Explanation
All preconditions are reachable by any unprivileged user with no special access:
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick < MAX_TICK` — all standard offer parameters
- `units < WAD / buyerPrice` — at tick `MAX_TICK - 1`, `units = 1` suffices

The attack is repeatable in a loop with no cooldown, no capital requirement (zero tokens transferred), and no rate limiting. The only cost is gas. The protocol's own test suite names and confirms the path as a bug.

## Recommendation
Add a zero-asset floor check before the consumed accounting for buy offers. The minimal fix is:

```solidity
require(buyerAssets > 0 || units == 0, RoundedToZero());
```

inserted after line 363 (after `buyerAssets` is computed) and before the `maxAssets` check. This ensures any non-zero `units` take produces a non-zero asset increment, making the cap enforceable. Alternatively, track `consumed` in `units` for buy offers (mirroring the `maxUnits` branch at line 371–372), or use `mulDivUp` for the consumed increment to guarantee a non-zero advance. [8](#0-7) 

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass()` in `test/TakeTest.sol` is a direct reproduction:

1. Create a buy offer with `maxAssets = N` and `tick < MAX_TICK`.
2. Pre-consume the offer to exactly `maxAssets` via normal takes.
3. Call `take(1, borrower, lenderOffer)` — `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`.
4. Assert: `consumed[maker][group]` remains at `maxAssets` (cap not advanced).
5. Assert: `creditOf(lender)` increased and `debtOf(borrower)` increased (positions mutated).
6. Repeat step 3 indefinitely — each call succeeds, credit/debt grow without bound. [6](#0-5)
