### Title
Fee-on-Transfer Loan Token Inflates `claimableSettlementFee` Beyond Real Balance, Breaking `claimSettlementFee` - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the nominal fee amount (`buyerAssets - sellerAssets`) before the actual `safeTransferFrom` call, and `safeTransferFrom` does not verify the actual balance delta received. When the loan token is a fee-on-transfer token, Midnight receives less than the nominal fee, but the accounting ledger is incremented by the full amount. This permanently inflates `claimableSettlementFee` above the real balance reserved for fees, causing `claimSettlementFee()` to either revert or drain funds belonging to other users (withdrawable/collateral).

### Finding Description

**Exact code path in `src/Midnight.sol`:**

Step 1 — accounting increment (line 418), before any transfer:
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

Step 2 — actual inbound transfer of the fee portion (line 455):
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only checks that the call does not revert and returns a truthy value; it does **not** measure the actual balance delta received by `address(this)`. [3](#0-2) 

With a fee-on-transfer loan token charging fee `f` per transfer, Midnight receives `(buyerAssets - sellerAssets) - f` instead of `buyerAssets - sellerAssets`. The ledger entry is `+= buyerAssets - sellerAssets` (full amount), so after the call:

```
claimableSettlementFee > actual_balance_reserved_for_fees  (by f)
```

**EcrecoverRatifier gate — why it is reachable by an unprivileged taker:**

`EcrecoverRatifier.isRatified` only verifies an EIP-712 signature over the offer tree root and checks `isRootCanceled`. It imposes no restriction on who the taker is and no restriction on the loan token type. [4](#0-3) 

Any unprivileged taker who obtains a valid signature + Merkle proof (e.g., from a public off-chain order book) can call `take()` with a fee-on-transfer loan token market. The `take()` entry-point only checks `taker == msg.sender || isAuthorized[taker][msg.sender]`, which is trivially satisfied by the taker acting for themselves. [5](#0-4) 

**Why existing checks do not stop it:**

The Certora formal verification explicitly assumes tokens transfer correctly with no fee-on-transfer behavior: [6](#0-5) 

The `pendingFeeReceiptZero` invariant tracks the gap between the `claimableSettlementFee` increment and the inbound transfer, and clears it only when `pendingFeeReceipt[token] == to_mathint(value)` — i.e., when the received amount exactly equals the nominal fee. With a fee-on-transfer token the received amount is `value - f`, so the ghost is never cleared to zero, and the invariant fails. [7](#0-6) 

### Impact Explanation

`claimableSettlementFee[loanToken]` permanently exceeds the real token balance reserved for fees by the cumulative transfer-fee deducted across all `take()` calls. When the fee claimer calls `claimSettlementFee(token, claimableSettlementFee[token], receiver)`:

- If the contract holds no surplus (no withdrawable or collateral for that token), `safeTransfer` reverts because the balance is insufficient — fees are permanently frozen.
- If the contract holds surplus funds (lender withdrawable assets or borrower collateral denominated in the same token), the transfer succeeds by consuming those funds, directly violating the `tokenBalanceCorrect` solvency invariant and draining lender/borrower assets. [8](#0-7) [9](#0-8) 

### Likelihood Explanation

**Preconditions:**
1. A market is created with a fee-on-transfer ERC-20 as `loanToken` (no protocol-level token whitelist exists).
2. A maker has posted an EcrecoverRatifier-gated offer (signed root + Merkle proof available to the taker).
3. A non-zero settlement fee is configured for the market.

**Feasibility:** Fully attacker-controlled. The taker supplies their own `taker` address, `units`, and valid `ratifierData`. No privileged role is needed. The discrepancy accumulates with every `take()` call and is irreversible.

**Repeatability:** Every `take()` on a fee-on-transfer market widens the gap by one transfer-fee amount. The effect is additive and permanent.

### Recommendation

Replace the nominal-amount accounting with a balance-before/after pattern for the inbound fee transfer:

```solidity
uint256 feeAmount = buyerAssets - sellerAssets;
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), feeAmount);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived;
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens (e.g., via a token registry or a `require` that `actualReceived == feeAmount`).

        return true;

function testFeeOnTransferInflatesClaimableSettlementFee() public {

    // 2. Configure non-zero settlement fee
    // 3. Maker signs an EcrecoverRatifier offer tree; taker obtains valid ratifierData
    // 4. Record balanceBefore = feeToken.balanceOf(address(midnight))
    // 5. Taker calls midnight.take(offer, ratifierData, units, ...)
    // 6. Record balanceAfter = feeToken.balanceOf(address(midnight))
    uint256 claimable = midnight.claimableSettlementFee(address(feeToken));

    // ASSERTION: claimable > actualDelta  (invariant violation)
    assertGt(claimable, actualDelta, "claimableSettlementFee exceeds real balance delta");
    // 7. Fee claimer attempts to claim full claimable amount
    vm.prank(feeClaimer);
    // ASSERTION: reverts OR drains other users' funds
    vm.expectRevert(); // if no surplus

    // OR assert lender withdrawable decreased if surplus exists

**Expected assertions:**
- `claimableSettlementFee[feeToken] > balanceOf(midnight) delta` after `take()` — `pendingFeeReceiptZero` invariant fails.
- `claimSettlementFee` reverts for the full `claimableSettlementFee` amount, or drains withdrawable/collateral funds.

// Settlement fee receipts pending settlement: claimableSettlementFee is incremented in take before
// the inbound fee transfer happens, so we track the gap and clear it in CVL_transferFrom.
persistent ghost mapping(address => mathint) pendingFeeReceipt {
    init_state axiom (forall address token. pendingFeeReceipt[token] == 0);
}

hook Sstore claimableSettlementFee[KEY address token] uint256 newVal (uint256 oldVal) {
    // Except for claimSettlementFee, the claimableSettlementFee is non-decreasing, see WithdrawableMonotonicity.spec.
    if (newVal > oldVal) {
        pendingFeeReceipt[token] = pendingFeeReceipt[token] + newVal - oldVal;
    }
}

/// INVARIANTS AND RULES ///

// For any token, the pending settlement fee receipt after a transaction is 0: every claimableSettlementFee
// increment in take is paid back in by the same-function inbound transfer.
weak invariant pendingFeeReceiptZero(address token)
    pendingFeeReceipt[token] == 0;
```
