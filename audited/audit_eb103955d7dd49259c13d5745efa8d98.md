### Title
Debt Increase Permitted at Exact Maturity Timestamp Due to Inclusive Boundary Check — (`File: src/Midnight.sol`)

### Summary

The `take()` function in `Midnight.sol` uses a `<=` comparison to guard against post-maturity debt increases. This allows a seller (borrower) to increase their debt at exactly `block.timestamp == market.maturity`, directly violating the protocol's own stated invariant: *"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."* This is the structural analog to the external report's boundary-condition vulnerability, where a strict-vs-inclusive comparison at a critical time boundary permits an action that should be forbidden.

---

### Finding Description

**Root Cause — Line 391, `src/Midnight.sol`:**

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

The condition `block.timestamp <= offer.market.maturity` evaluates to `true` when `block.timestamp == maturity`. This means a `take()` call that increases `sellerDebtIncrease > 0` is permitted at the exact maturity timestamp. [1](#0-0) 

The protocol's own invariant documentation explicitly names this case as forbidden: [2](#0-1) 

**Compounding effects at `block.timestamp == maturity`:**

1. `timeToMaturity = zeroFloorSub(maturity, block.timestamp) = 0`
2. `buyerPendingFeeIncrease = buyerCreditIncrease * continuousFee * 0 / WAD = 0` — the new lender accrues **zero** continuous fee obligation on credit issued at maturity.
3. Settlement fee is computed at the 0-day breakpoint (`settlementFeeCbp0 * CBP`), the minimum rate.
4. Post-maturity liquidation requires `block.timestamp > market.maturity` (strict `>`), so at exactly `block.timestamp == maturity`, post-maturity liquidation is **not yet available**. [3](#0-2) [4](#0-3) 

**Test gap confirmation:** All existing post-maturity tests use `market.maturity + 1` as the lower bound, explicitly skipping the boundary timestamp. No test covers `block.timestamp == maturity` with a debt-increasing `take`. [5](#0-4) 

---

### Impact Explanation

A borrower who executes `take()` at exactly `block.timestamp == maturity`:

- **Violates the core maturity invariant**: debt increases at a timestamp the protocol explicitly forbids.
- **Pays zero continuous fee**: `timeToMaturity == 0` makes `buyerPendingFeeIncrease == 0`, so the lender who provides credit at this boundary receives no continuous fee income on that credit.
- **Faces no immediate post-maturity liquidation**: since `block.timestamp > maturity` is false, the post-maturity liquidation path is unavailable for that block. Only health-based liquidation applies.
- **Lender is harmed**: a lender whose offer is filled at maturity issues credit with `pendingFee = 0`, receiving no continuous fee compensation for the credit extended.

The borrower still requires sufficient collateral to pass the health check at the end of `take()`, so this is not a zero-cost exploit, but it is a fee-avoidance and invariant-breaking path reachable by any unprivileged user. [6](#0-5) 

---

### Likelihood Explanation

- **No privileged access required**: any borrower with collateral can execute this.
- **Timing is deterministic**: `market.maturity` is a known, fixed value stored in the market struct. An attacker can monitor the chain and submit a transaction in the maturity block.
- **Realistic on low-block-time chains**: on chains with 1–2 second block times, landing a transaction at exactly `block.timestamp == maturity` is straightforward with standard MEV tooling or block-builder access.
- **Offer availability**: lenders routinely leave open offers; a standing sell offer with `expiry >= maturity` is sufficient.

---

### Recommendation

Change the boundary check from `<=` to `<` (strict less-than):

```solidity
// Before (buggy):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (correct):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the implementation with the stated invariant and matches the intent of the post-maturity tests, which all use `maturity + 1` as the earliest forbidden timestamp. [1](#0-0)

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {IMidnight} from "../src/interfaces/IMidnight.sol";

contract MaturityBoundaryPoC is BaseTest {
    function test_debtIncreaseAtExactMaturity() public {
        uint256 units = 100e18;

        // Setup: lender has an open buy offer, borrower has collateral
        collateralize(market, borrower, units);
        setupMarket(market, units);

        // Warp to EXACTLY maturity (not maturity + 1)
        vm.warp(market.maturity);

        // Prepare a new borrow offer expiring at maturity
        borrowerOffer.expiry = market.maturity;
        borrowerOffer.maxUnits = units;
        deal(address(loanToken), lender, units);

        uint256 debtBefore = midnight.debtOf(id, borrower);

        // This should revert with CannotIncreaseDebtPostMaturity, but it does NOT
        take(units, lender, borrowerOffer);

        uint256 debtAfter = midnight.debtOf(id, borrower);

        // Debt increased at maturity — invariant violated
        assertGt(debtAfter, debtBefore, "debt increased at exact maturity");

        // Continuous fee on the new credit is zero (timeToMaturity == 0)
        assertEq(midnight.pendingFee(id, lender), 0, "lender gets zero continuous fee");
    }
}
```

**Expected (correct) behavior**: revert with `CannotIncreaseDebtPostMaturity`.
**Actual behavior**: `take()` succeeds, debt increases, lender receives zero continuous fee.
