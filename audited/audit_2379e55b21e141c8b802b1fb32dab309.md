### Title
Referral fee not budgeted within `maxBuyerAssets`, causing revert that blocks a valid fill - (`File: src/periphery/MidnightBundles.sol`)

### Summary
In `buyWithUnitsTargetAndWithdrawCollateral`, the bundler pulls `maxBuyerAssets` from `msg.sender` and spends `filledBuyerAssets` of that in the take loop. The referral fee (`referralFeeAssets`) is then computed on top of `filledBuyerAssets` and transferred out of the bundler's remaining balance. When `maxBuyerAssets == filledBuyerAssets` and `referralFeePct > 0`, the bundler's post-loop balance is zero, so the `safeTransfer` to the referrer at line 103 reverts with an ERC20 insufficient-balance error, permanently blocking a fill that succeeded in every other respect. The arithmetic underflow at line 104 is the secondary failure mode for the same root cause; line 103 triggers first.

### Finding Description

**Code path:** [1](#0-0) 

The bundler pulls exactly `maxBuyerAssets` tokens from `msg.sender` and grants `type(uint256).max` approval to MIDNIGHT. [2](#0-1) 

The take loop calls `MIDNIGHT.take(...)`, which pulls `resBuyerAssets` from the bundler per iteration. After the loop, the bundler holds `maxBuyerAssets - filledBuyerAssets` tokens. [3](#0-2) 

`referralFeeAssets` is computed as `filledBuyerAssets * referralFeePct / (WAD - referralFeePct)`. When `referralFeePct > 0` and `filledBuyerAssets > 0`, this value is strictly positive. The function then attempts to transfer `referralFeeAssets` to the referrer from the bundler's balance. If `maxBuyerAssets == filledBuyerAssets`, the bundler's balance is zero at this point, so the ERC20 `transfer` call fails and `safeTransfer` propagates the revert: [4](#0-3) 

The subsequent subtraction `maxBuyerAssets - filledBuyerAssets - referralFeeAssets` at line 104 would also underflow (Solidity 0.8.34 checked arithmetic), but it is never reached because line 103 reverts first.

**Root cause:** No check enforces `maxBuyerAssets >= filledBuyerAssets + referralFeeAssets`. The NatSpec documents the total cost formula correctly: [5](#0-4) 

but the contract never validates that `maxBuyerAssets` is large enough to cover both the fill and the referral fee.

**Attacker-controlled inputs:** Any unprivileged taker acting as `msg.sender` can supply `referralFeePct ∈ [1, WAD-1]` and `maxBuyerAssets` set tightly to the expected `filledBuyerAssets` (e.g., computed off-chain from the offer price). No privileged role is required.

**Existing protections:** The only guards are `referralFeePct < WAD` and `taker` authorization: [6](#0-5) 

Neither prevents the underflow/ERC20-revert scenario.

### Impact Explanation
All offers are successfully taken (`filledUnits == targetUnits` passes at line 88), collateral withdrawals complete, but the transaction reverts at the referral-fee transfer (line 103). The user loses gas and cannot complete the fill in a single call even though the market state is valid and the fill is economically sound. Because the revert is deterministic for the given `(maxBuyerAssets, referralFeePct)` pair, the user is permanently blocked unless they either increase `maxBuyerAssets` or set `referralFeePct = 0`.

### Likelihood Explanation
Any taker who sets `maxBuyerAssets` equal to the expected fill cost (tight slippage, a common pattern) and passes a nonzero `referralFeePct` hits this revert. The precondition is trivially reachable by any unprivileged user with no special state required. It is repeatable across every call with these parameters.

### Recommendation
After computing `referralFeeAssets`, add an explicit check before the transfers:

```solidity
uint256 referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
uint256 totalCost = filledBuyerAssets + referralFeeAssets;
require(maxBuyerAssets >= totalCost, BuyerAssetsTooLow());
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - totalCost);
```

This makes the invariant "msg.sender pays at most `maxBuyerAssets`" explicit and reverts early with a meaningful error rather than an opaque ERC20 failure.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity 0.8.34;

// Foundry fuzz test
function testFuzz_buyUnitsTarget_referralFeeUnderflow(
    uint256 units,
    uint256 referralFeePct
) public {
    units = bound(units, 1, 1e24);
    referralFeePct = bound(referralFeePct, 1, WAD - 1); // nonzero fee

    // Setup: borrower posts a sell offer, collateralizes
    offers[0].buy = false;
    offers[0].maker = borrower;
    offers[0].receiverIfMakerIsSeller = borrower;
    offers[0].maxUnits = type(uint256).max;
    for (uint256 i; i <= 6; i++) midnight.setMarketSettlementFee(id, i, 0);
    collateralize(market, borrower, units);

    uint256 price = TickLib.tickToPrice(MAX_TICK);
    // Set maxBuyerAssets == expected filledBuyerAssets (tight, no room for fee)
    uint256 maxBuyerAssets = units.mulDivUp(price, WAD); // == filledBuyerAssets

    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offers[0], units: type(uint256).max, ratifierData: hex""});

    address referrer = makeAddr("referrer");

    vm.prank(lender);
    // Expected: reverts (ERC20 insufficient balance at referral transfer, or
    // arithmetic underflow at refund subtraction)
    vm.expectRevert();
    midnightBundles.buyWithUnitsTargetAndWithdrawCollateral(
        units,
        maxBuyerAssets,   // == filledBuyerAssets, no budget for referral fee
        lender,
        _noPermit(),
        takes,
        new CollateralWithdrawal[](0),
        address(0),
        referralFeePct,   // nonzero
        referrer
    );

    // Assert: bundler holds no residual (state is clean despite revert)
    assertEq(loanToken.balanceOf(address(midnightBundles)), 0);
}
```

**Expected assertion:** `vm.expectRevert()` passes for all `referralFeePct ∈ [1, WAD-1]` when `maxBuyerAssets == filledBuyerAssets`. The bundler balance assertion confirms no tokens are stranded. To confirm the fix, replace `maxBuyerAssets` with `filledBuyerAssets + referralFeeAssets` and assert the call succeeds.
