Audit Report

## Title
Reentrancy in `take()` buyer callback allows draining pre-existing `withdrawable` balance at a discount - (File: src/Midnight.sol)

## Summary
`take()` violates checks-effects-interactions by writing `buyerPos.credit` to storage at line 410 and then making an external call to the maker-controlled `IBuyCallback.onBuy` at lines 445ÔÇô453, before pulling payment via `safeTransferFrom` at lines 455ÔÇô456. Because `withdraw()` has no reentrancy guard and no transient lock protecting it during `take()` callbacks, a malicious `IBuyCallback` authorized by the maker can call `withdraw()` inside `onBuy`, consuming pre-existing `withdrawable` balance funded by other borrowers' repayments and using those tokens to fund the take's own payment. The attacker nets `U - buyerAssets` tokens per attack at the direct expense of other lenders' withdrawable pool.

## Finding Description

**Root cause:** `take()` violates CEI. Storage is mutated at lines 408ÔÇô418 (including `buyerPos.credit += buyerCreditIncrease` at line 410), then an external call is made to `IBuyCallback(buyerCallback).onBuy(...)` at lines 445ÔÇô453, and only then are tokens pulled via `safeTransferFrom` at lines 455ÔÇô456. The `LIQUIDATION_LOCK_SLOT` set at line 444 only prevents liquidation of the seller; it does not protect `withdraw()`. `withdraw()` (lines 481ÔÇô500) has no reentrancy guard and no check that it is being called during a `take()` callback.

**Exact code path:**

`take()` with `offer.buy == true`:
- Line 375: `buyer = offer.maker`, `seller = taker`
- Line 379: `_updatePosition(offer.market, id, buyer)` ÔÇö sets `lastLossFactor = lossFactor`, `lastAccrual = block.timestamp`
- Line 410: `buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease)` ÔÇö credit written to storage before any external call
- Line 420: `buyerCallback = offer.callback` ÔÇö fully maker-controlled
- Line 422: `payer = buyerCallback` ÔÇö the callback is also the payer for the subsequent `safeTransferFrom`
- Line 444: `LIQUIDATION_LOCK_SLOT` set for seller only
- Lines 445ÔÇô453: `IBuyCallback(buyerCallback).onBuy(...)` ÔÇö external call before token transfer
- Lines 455ÔÇô456: `safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, payer, receiver, sellerAssets)` ÔÇö token pull happens after callback returns

`withdraw()` (lines 481ÔÇô500):
- Line 482: `isAuthorized[onBehalf][msg.sender]` ÔÇö satisfied if maker pre-authorized the callback
- Line 485: `_updatePosition(market, id, onBehalf)` ÔÇö is a no-op in the same block because `lastLossFactor == lossFactor` and `lastAccrual == block.timestamp`, so `newCredit = U` unchanged
- Line 493: `_position.credit -= units` ÔÇö succeeds because credit was just increased to `U` at line 410
- Line 494: `_marketState.withdrawable -= units` ÔÇö succeeds if pre-existing `withdrawable Ôëą units`
- Line 499: transfers `units` tokens to receiver

**Exploit flow:**

Precondition: market has pre-existing `withdrawable = W Ôëą U` from prior borrower repayments (via `repay()` line 509 or `liquidate()` line 675).

1. Maker deploys `MaliciousCallback` implementing `IBuyCallback`; calls `setIsAuthorized(MaliciousCallback, true, maker)` so `isAuthorized[maker][MaliciousCallback] = true`.
2. Maker creates buy offer with `offer.callback = MaliciousCallback`, ratified via a maker-controlled ratifier.
3. Taker calls `take(offer, ratifierData, U, taker, receiver, address(0), hex"")`.
4. `take()` sets `buyerPos.credit += U` (line 410).
5. `take()` calls `MaliciousCallback.onBuy(...)` (line 448).
6. Inside `onBuy`, `MaliciousCallback` calls `Midnight.withdraw(market, U, maker, attacker)`:
   - Auth: `isAuthorized[maker][MaliciousCallback]` = true, `msg.sender` = MaliciousCallback Ôťô
   - Credit: `_position.credit = U Ôëą U` Ôťô
   - Withdrawable: `_marketState.withdrawable = W Ôëą U` Ôťô
   - Result: `_position.credit = 0`, `withdrawable = W - U`, attacker receives U tokens.
7. `MaliciousCallback` approves Midnight for `buyerAssets` tokens (funded by the U tokens just received; since `buyerPrice < WAD`, `buyerAssets < U`).
8. `onBuy` returns `CALLBACK_SUCCESS`.
9. `take()` calls `safeTransferFrom(loanToken, MaliciousCallback, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, MaliciousCallback, receiver, sellerAssets)` ÔÇö both succeed.

**Why existing checks fail:**
- `LIQUIDATION_LOCK_SLOT` (line 444) only prevents liquidation of the seller; it does not block `withdraw()`.
- `withdraw()` has no transient lock, no reentrancy guard, and no check that it is not being called during a `take()` callback.
- There is no post-callback invariant check in `take()` that `withdrawable` was not consumed.
- The `_updatePosition` call inside `withdraw()` (line 485) is a no-op in the same block since `lastLossFactor` and `lastAccrual` were already synchronized for the buyer at line 379 of `take()`.

## Impact Explanation

Direct theft of tokens from the `withdrawable` pool. The attacker receives `U` tokens immediately from the pre-existing `withdrawable` pool while paying only `buyerAssets = U ├Ś buyerPrice / WAD < U` tokens back. Net profit per attack: `U ├Ś (1 - buyerPrice/WAD)`. The contract's token balance is short by `U - (buyerAssets - sellerAssets)` with no accounting correction. Other lenders who had repaid tokens available for withdrawal find `withdrawable` reduced by `U`, constituting direct, concrete theft of other lenders' withdrawable assets.

## Likelihood Explanation

Preconditions are realistic in any active market: (a) pre-existing `withdrawable` balance exists whenever any borrower has repaid; (b) the maker can freely set `offer.callback` and authorize it via `setIsAuthorized` ÔÇö no privileged role required; (c) the ratifier only requires that the maker (or their authorized operator) ratify the offer root. The attack is repeatable as long as `withdrawable` is non-zero and is profitable whenever `buyerPrice < WAD` (the normal lending case). The taker does not need to be malicious or even aware of the attack.

## Recommendation

Apply a transient reentrancy lock in `withdraw()` that prevents it from being called during any active `take()` callback, or restructure `take()` to follow strict CEI: pull tokens from the payer before making any external callback. Alternatively, add a transient guard slot (analogous to `LIQUIDATION_LOCK_SLOT`) that is set at the start of `take()` and checked at the start of `withdraw()`, reverting if set. A simpler fix is to move the `safeTransferFrom` calls (lines 455ÔÇô456) to before the `onBuy` callback (line 445), so the payer must fund the take before the callback executes.

## Proof of Concept

1. Deploy `MaliciousCallback` implementing `IBuyCallback.onBuy` that calls `Midnight.withdraw(market, units, maker, attacker)` inside `onBuy`.
2. Maker calls `setIsAuthorized(address(MaliciousCallback), true, maker)`.
3. Seed market with `withdrawable = W` via a borrower calling `repay()`.
4. Maker creates a buy offer with `offer.callback = address(MaliciousCallback)`, `offer.tick` set to a price below WAD, and a maker-controlled ratifier.
5. Taker calls `take(offer, ratifierData, W, taker, takerReceiver, address(0), hex"")`.
6. Assert: attacker received `W` tokens; contract token balance decreased by `W - buyerAssets + sellerAssets`; `marketState[id].withdrawable == 0`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
