### Title
Oracle Price of Zero Enables Free Collateral Seizure via `seizedAssets` Path — (File: src/Midnight.sol)

---

### Summary

When a collateral oracle returns `price() == 0`, the `liquidate` function's `seizedAssets > 0` branch silently computes `repaidUnits = 0`, allowing a liquidator to seize the entire collateral balance while repaying nothing. Simultaneously, `isHealthy` treats any position with a zero-priced collateral as immediately unhealthy, making it liquidatable. The protocol's own LIVENESS documentation acknowledges the zero-price case only for the `repaidUnits > 0` branch (division-by-zero revert), leaving the `seizedAssets > 0` branch completely unguarded.

---

### Finding Description

**Vulnerability class:** Pricing/accounting — sentinel-value overloading (zero price treated as valid price rather than invalid/empty).

**Root cause in `liquidate` — `src/Midnight.sol` lines 607–653:**

The loop at lines 607–618 queries every activated collateral's oracle:

```solidity
uint256 price = IOracle(_collateralParam.oracle).price();
if (i == collateralIndex) liquidatedCollatPrice = price;
```

If the oracle for the liquidated collateral returns `0`, then `liquidatedCollatPrice = 0`.

The branch at lines 649–651 then executes:

```solidity
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```

Expanding `mulDivUp` (line 34–36 of `UtilsLib.sol`):

```
step 1: (seizedAssets * 0 + (ORACLE_PRICE_SCALE - 1)) / ORACLE_PRICE_SCALE
       = (1e36 - 1) / 1e36 = 0   (integer division)
step 2: (0 * WAD + (lif - 1)) / lif
       = (lif - 1) / lif = 0     (lif > WAD = 1e18)
```

`repaidUnits = 0`. The liquidator transfers zero loan tokens and receives the full `seizedAssets` collateral.

**Root cause in `isHealthy` — `src/Midnight.sol` lines 950–957:**

```solidity
uint256 price = IOracle(collateralParam.oracle).price();
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

When `price = 0`, the collateral contributes `0` to `maxDebt`. Any position with `debt > 0` and only this collateral activated becomes immediately liquidatable (`maxDebt = 0 < debt`), satisfying the `NotLiquidatable` guard at line 620–624.

**What the LIVENESS section misses:**

The contract comment at line 146 states:
> "If the liquidated collateral oracle returns 0 on price, liquidate with repaid input reverts."

This covers only the `repaidUnits > 0` branch (line 652), which divides by `liquidatedCollatPrice` and reverts on division-by-zero. The `seizedAssets > 0` branch (line 650) is not mentioned and does **not** revert — it silently produces `repaidUnits = 0`.

---

### Impact Explanation

A liquidator can call `liquidate(market, collateralIndex, seizedAssets=<full balance>, repaidUnits=0, borrower, ...)` and receive the borrower's entire collateral for zero loan tokens. The borrower loses all collateral with no debt reduction (beyond the `badDebt` socialization path, which is separate). This is a direct theft of borrower funds at the expense of the borrower and, indirectly, lenders who hold credit in the market.

---

### Likelihood Explanation

The trigger requires `IOracle.price()` to return `0` for the liquidated collateral. This can occur via:

1. **Oracle implementation bug** — a custom oracle that returns `0` during initialization, stale-data fallback, or edge-case price feed failure.
2. **Oracle manipulation** — the SECURITY.md explicitly keeps oracle manipulation/flash-loan attacks **in scope** ("This does not exclude oracle manipulation/flash-loan attacks"). A flash-loan-based manipulation of an on-chain TWAP or spot-price oracle to momentarily report `0` is a realistic attack vector.
3. **Permissionless market creation** — `touchMarket` imposes no validation that the oracle returns a non-zero price at creation time. A market creator can deploy a market whose oracle returns `0` under attacker-controlled conditions, then attract borrowers and exploit them.

---

### Recommendation

Add an explicit non-zero price guard before the seized/repaid computation in `liquidate`:

```solidity
require(liquidatedCollatPrice > 0, ZeroOraclePrice());
```

This mirrors the existing pattern in `isHealthy` where a zero price already causes silent misbehavior; both sites should be hardened. Optionally, add the same guard inside the `isHealthy` loop to prevent zero-price oracles from making healthy positions appear liquidatable.

---

### Proof of Concept

**Setup:**
1. Deploy a controllable oracle `MockOracle` whose `price()` returns `ORACLE_PRICE_SCALE` normally and `0` on demand.
2. Create a market with `collateralParams[0].oracle = address(mockOracle)`.
3. Borrower supplies 1000 collateral tokens and borrows 800 units (healthy at normal price).

**Attack:**
4. Attacker triggers `mockOracle` to return `0` (or flash-loan-manipulates a real oracle to `0`).
5. `isHealthy` now returns `false` for the borrower (maxDebt = 0 < 800 debt).
6. Attacker calls:
   ```solidity
   midnight.liquidate(
       market,
       0,               // collateralIndex
       1000,            // seizedAssets = full collateral
       0,               // repaidUnits (not used in this branch)
       borrower,
       false,
       attacker,
       address(0),
       ""
   );
   ```
7. Inside `liquidate`: `liquidatedCollatPrice = 0`, `repaidUnits = 0`.
8. `SafeTransferLib.safeTransfer(collateralToken, attacker, 1000)` executes.
9. `SafeTransferLib.safeTransferFrom(loanToken, attacker, address(this), 0)` is a no-op.

**Result:** Attacker receives 1000 collateral tokens, pays 0 loan tokens. Borrower's debt remains (or is partially socialized as bad debt). Borrower loses all collateral. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
