Audit Report

## Title
`sellerAssetsToUnits` arithmetic revert at `offerPrice == settlementFee` blocks `supplyCollateralAndSellWithAssetsTarget` while `Midnight.take()` succeeds - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary

When a buy offer's tick price exactly equals the settlement fee, `sellerPrice` computes to zero in `TakeAmountsLib.sellerAssetsToUnits`. The subsequent `mulDivUp(targetSellerAssets, WAD, 0)` call triggers an arithmetic underflow revert via the `d - 1` term in Solidity 0.8+, while `Midnight.take()` handles the same `sellerPrice = 0` case without reverting (producing `sellerAssets = 0`). Because `sellerAssetsToUnits` is called outside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, the entire bundler call reverts, blocking the assets-target sell path even when the underlying take would succeed.

## Finding Description

**Root cause — `TakeAmountsLib.sellerAssetsToUnits` (lines 44–46):** [1](#0-0) 

When `offer.buy == true` and `offerPrice == settlementFee`, `sellerPrice = offerPrice - settlementFee = 0`. The function then calls `targetSellerAssets.mulDivUp(WAD, 0)`.

**`mulDivUp` implementation (UtilsLib.sol line 35):** [2](#0-1) 

With `d = 0`, the expression `d - 1` underflows under Solidity 0.8 checked arithmetic, producing an `arithmeticError` revert — not a division-by-zero revert. This is explicitly confirmed by the test: [3](#0-2) 

**`Midnight.take()` does not revert in the same case (lines 361–364):** [4](#0-3) 

`units.mulDivDown(sellerPrice, WAD)` = `(units * 0) / WAD = 0`. The denominator is `WAD`, not zero — no revert.

**The comment in `sellerAssetsToUnits` (line 34) is inaccurate:** [5](#0-4) 

The comment claims "midnight reverts too" for `offerPrice < settlementFee`. This is only true for strict inequality. When `offerPrice == settlementFee`, Midnight does **not** revert, but `sellerAssetsToUnits` does.

**Call site in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` (lines 285–300):** [6](#0-5) 

`TakeAmountsLib.sellerAssetsToUnits` is called at lines 286–288, **outside** the `try/catch` block that wraps `Midnight.take()` at lines 292–300. A revert in `sellerAssetsToUnits` propagates uncaught, aborting the entire bundler call.

**Preconditions for the boundary condition:**

`PRICE_ROUNDING_STEP = 1e12 = CBP`: [7](#0-6) 

`MAX_SETTLEMENT_FEE_0_DAYS = 14e12`, so `settlementFeeCbp0` ranges from 0 to 14: [8](#0-7) 

`tickToPrice` rounds to multiples of `PRICE_ROUNDING_STEP`: [9](#0-8) 

Both tick prices and settlement fees are multiples of `1e12`. The post-maturity settlement fee ranges from `0` to `14e12`, and tick prices in the low-price range are dense multiples of `1e12`, so there exist valid ticks where `tickToPrice(tick) == settlementFeeCbp0 * CBP`.

**`NoDivisionByZero.spec` covers only `Midnight.sol`, not `TakeAmountsLib.sol`:** [10](#0-9) 

The bundler's own NatSpec at line 246 documents the symptom ("Reverts if TakeAmountsLib or ConsumableUnitsLib reverts") but provides no protection: [11](#0-10) 

## Impact Explanation

Any taker calling `supplyCollateralAndSellWithAssetsTarget` whose `takes` array contains a buy offer at the settlement-fee price boundary will have the entire call revert with an arithmetic error. The taker cannot use the atomic collateral-supply-and-sell bundler path to unwind at maturity, even though `Midnight.take()` would succeed. The taker must fall back to calling `Midnight.take()` directly, losing the atomic multi-offer aggregation the bundler provides. There is no loss of funds, but there is a concrete functional DoS of the bundler path.

## Likelihood Explanation

The condition requires `tickToPrice(offer.tick) == settlementFeeCbp0 * CBP`. Both quantities are multiples of `CBP = 1e12`, and tick prices are dense in the low-price range (up to `14e12`), making alignment achievable without any admin action: an offer maker creates a buy offer at a tick whose price equals the current settlement fee, and a taker calling the bundler with that offer triggers the revert. The fee setter can also raise the settlement fee to match an existing offer's tick price, which is documented protocol behavior. The condition is repeatable across any market where `settlementFeeCbp0 * CBP` aligns with a tick price.

## Recommendation

Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case, mirroring how `Midnight.take()` handles it. For example, return `type(uint256).max` (signaling that no finite unit count can yield a positive `sellerAssets`) or return `0` to cause the offer to be skipped. Alternatively, extend the `try/catch` in `supplyCollateralAndSellWithAssetsTarget` to also wrap the `TakeAmountsLib.sellerAssetsToUnits` call, so that offers at the boundary are skipped rather than aborting the entire bundler call. The inaccurate comment on line 34 of `TakeAmountsLib.sol` should also be corrected to reflect that the equality case (`offerPrice == settlementFee`) causes a revert in `sellerAssetsToUnits` but not in `Midnight.take()`.

## Proof of Concept

1. Deploy a market with `settlementFeeCbp0 = N` (e.g., `N = 14`, so post-maturity fee = `14e12`).
2. Create a buy offer at a tick `T` where `tickToPrice(T) == N * 1e12` (e.g., `tickToPrice(T) = 14e12`).
3. Warp `block.timestamp` past `offer.market.maturity` so `timeToMaturity = 0` and `settlementFee = settlementFeeCbp0 * CBP = 14e12`.
4. Call `supplyCollateralAndSellWithAssetsTarget` with `takes = [offer at tick T]` and any positive `targetSellerAssets`.
5. Observe: the call reverts with `stdError.arithmeticError` from `mulDivUp(targetSellerAssets, WAD, 0)` inside `sellerAssetsToUnits`.
6. Confirm: calling `Midnight.take(offer, ..., units, ...)` directly with the same offer succeeds and returns `sellerAssets = 0`.
