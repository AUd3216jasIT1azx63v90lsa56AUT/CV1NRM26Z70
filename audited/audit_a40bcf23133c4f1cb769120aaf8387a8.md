### Title
`IRepayCallback.onRepay` Does Not Pass the Initiator, Enabling Unauthorized Callback Triggering — (File: src/Midnight.sol, src/interfaces/ICallbacks.sol)

---

### Summary

The `repay()` function in `Midnight.sol` invokes `IRepayCallback.onRepay` without passing `msg.sender` (the initiator of the repay call). This is inconsistent with `onFlashLoan` and `onLiquidate`, which both correctly forward the caller. Any contract implementing `onRepay` as a payer-callback cannot authenticate who triggered the repay, making it impossible to safely gate token approvals or actions inside the callback without complex workarounds.

---

### Finding Description

In `repay()`, the callback is designated as the `payer` and is called before the token pull: [1](#0-0) 

The callback invocation is:

```solidity
IRepayCallback(callback).onRepay(id, market, units, onBehalf, data)
```

The `IRepayCallback` interface signature is: [2](#0-1) 

`msg.sender` (the initiator of `repay()`) is **not** forwarded. The callback only receives `onBehalf` — the account whose debt is being reduced — not who triggered the call.

By contrast, `onFlashLoan` and `onLiquidate` both correctly pass the caller as the first argument: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

The same omission exists in `onBuy` and `onSell` (neither receives `msg.sender`), but `onRepay` is the highest-impact case because the callback is the designated payer. [7](#0-6) 

---

### Impact Explanation

**Concrete attack path:**

1. A protocol deploys a `RepayCallback` contract that holds loan tokens, approves `Midnight` inside `onRepay`, and returns `CALLBACK_SUCCESS`. Because the initiator is not passed, the contract cannot distinguish a legitimate caller from an attacker.
2. Attacker calls `midnight.repay(market, units, attacker, address(repayCallback), data)` — valid because `onBehalf == msg.sender`.
3. `Midnight` calls `repayCallback.onRepay(id, market, units, attacker, data)`.
4. `repayCallback` cannot verify who initiated the call; it approves `Midnight` and returns `CALLBACK_SUCCESS`.
5. `Midnight` executes `safeTransferFrom(loanToken, repayCallback, address(this), units)` — pulling the victim callback's tokens.
6. The attacker's debt is repaid at the victim's expense. [8](#0-7) 

**Impact:** Direct theft of tokens held by or approved by any `onRepay` callback contract that does not implement a complex flag-based workaround. The attacker profits by having their debt cleared using another contract's funds.

---

### Likelihood Explanation

Any integrator building a repay-callback contract (e.g., a bundler, a smart wallet, a liquidation helper that repays and withdraws collateral atomically) faces this problem. The pattern of holding tokens in a callback contract and approving the core protocol inside the callback is standard practice (as seen in `MidnightBundles` itself). Without the initiator being passed, every such contract is vulnerable unless it implements a non-obvious flag mechanism. Given the number of real-world exploits from this exact pattern (cited in the original Astaria report), likelihood of exploitation is high once integrations exist.

---

### Recommendation

Add `address initiator` (= `msg.sender` of `repay()`) as the first parameter of `onRepay`, matching the pattern already used by `onFlashLoan` and `onLiquidate`:

```solidity
// ICallbacks.sol
interface IRepayCallback {
    function onRepay(address initiator, bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
}
```

```solidity
// Midnight.sol — repay()
IRepayCallback(callback).onRepay(msg.sender, id, market, units, onBehalf, data)
```

Apply the same fix to `onBuy` and `onSell` for consistency and to prevent the same class of issue in those callbacks.

---

### Proof of Concept

**Setup:**
- `RepayCallback` holds 1000 USDC, approves `Midnight` for `type(uint256).max` inside `onRepay`, and returns `CALLBACK_SUCCESS` unconditionally (cannot check initiator — it is not provided).
- Attacker has 500 units of debt in a USDC market.

**Steps:**
1. Attacker calls `midnight.repay(market, 500, attacker, address(repayCallback), "")`.
2. `Midnight` validates `onBehalf == msg.sender` ✓, decrements attacker's debt by 500.
3. `Midnight` calls `repayCallback.onRepay(id, market, 500, attacker, "")` — callback approves and returns `CALLBACK_SUCCESS`.
4. `Midnight` calls `safeTransferFrom(USDC, repayCallback, midnight, 500)` — 500 USDC leaves `repayCallback`.
5. Attacker's 500-unit debt is cleared. `repayCallback` lost 500 USDC.

**Expected result without fix:** Attacker's debt is repaid using `repayCallback`'s funds.
**Expected result with fix:** `repayCallback` checks `initiator == trustedCaller` and reverts, blocking the attack. [9](#0-8) [2](#0-1)
