Audit Report

## Title
Sequential `mulDivDown` in `isHealthy`/`liquidate` underestimates `maxDebt` by up to 16 units, enabling liquidation of healthy positions - (File: `src/Midnight.sol`)

## Summary
`isHealthy` and `liquidate` compute each collateral slot's contribution to `maxDebt` via two sequential `mulDivDown` calls — `floor(floor(c·p / ORACLE_PRICE_SCALE) · lltv / WAD)` — rather than a single combined division. Each sequential application can lose up to 1 unit of precision. With `MAX_COLLATERALS_PER_BORROWER = 16`, the aggregate `maxDebt` can be underestimated by up to 16 units, allowing a liquidator to liquidate a genuinely healthy position whose debt is within 16 units of its true `maxDebt`, and causing `withdrawCollateral` to revert for a healthy borrower.

## Finding Description

**Confirmed code path:**

`src/Midnight.sol` `isHealthy` lines 954–955:
```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```
Identical pattern in `liquidate` line 613. `withdrawCollateral` line 568 gates on `isHealthy`. `liquidate` line 622 gates on `originalDebt > maxDebt`.

**Root cause — sequential rounding loss:**

Let `a = c·p / ORACLE_PRICE_SCALE` (exact rational), `l = lltv`.

- Sequential result: `S = floor(floor(a) · l / WAD)`
- Ideal single-step: `I = floor(a · l / WAD)`

Since `floor(a) ≤ a`, we have `S ≤ I`. The loss is exactly 1 when `a·l/WAD` is an integer `n` and `floor(a) < a` (i.e., `c·p` is not divisible by `ORACLE_PRICE_SCALE`): then `floor(a) · l/WAD < n`, so `S ≤ n − 1`. And since `floor(a) ≥ a − 1` and `l ≤ WAD`, we get `floor(a) · l/WAD ≥ n − 1`, so `S = n − 1 = I − 1` exactly. Summed over 16 active slots: `sequential_maxDebt ≥ true_maxDebt − 16`.

**Exploit flow:**

1. Borrower opens a position with 16 active collateral slots (all bits set in `collateralBitmap`), each with `(collateral, price, lltv)` values chosen so that `c_i · p_i mod ORACLE_PRICE_SCALE ≠ 0` and `floor(c_i · p_i / ORACLE_PRICE_SCALE) · lltv mod WAD` produces `S_i = I_i − 1` for each slot.
2. Borrower's true `maxDebt = debt` (position exactly at the health boundary).
3. `isHealthy` computes `sequential_maxDebt = true_maxDebt − 16 = debt − 16`.
4. `liquidate` evaluates `originalDebt > maxDebt` → `debt > debt − 16` → **true** → liquidation proceeds on a healthy position.
5. Alternatively, `withdrawCollateral` calls `isHealthy`, gets `false`, and reverts with `UnhealthyBorrower()`.

**Why existing checks fail:**

There is no tolerance buffer in `isHealthy` (`maxDebt >= debt`) or `liquidate` (`originalDebt > maxDebt`). The strict inequality in `liquidate` means even a 1-unit underestimation of `maxDebt` on a boundary position is exploitable.

## Impact Explanation

An unprivileged liquidator can liquidate a genuinely healthy borrower whose position has up to 16 active collateral slots and whose debt is within 16 units of their true `maxDebt`. This directly violates the core protocol invariant that healthy positions are not liquidatable. Additionally, a healthy borrower with 16 active collateral slots cannot withdraw collateral even when their true health ratio permits it, because `withdrawCollateral` reverts via the same underestimated `isHealthy` check. The loss is bounded (at most 16 base units of the loan token per liquidation event), but the invariant violation is concrete and repeatable.

## Likelihood Explanation

Preconditions are specific but achievable: 16 active collateral slots (the protocol-enforced maximum per `MAX_COLLATERALS_PER_BORROWER = 16` in `ConstantsLib.sol` line 21), each slot's `(collateral, price, lltv)` producing a rounding loss of exactly 1 unit, and the position within 16 units of the health boundary. A liquidator monitoring on-chain positions can identify such configurations deterministically given fixed oracle prices. Oracle prices change continuously, so positions naturally drift through the boundary window. The 16-unit window is narrow but non-zero, and the condition is repeatable and deterministic.

## Recommendation

Replace the two sequential `mulDivDown` calls with a single combined division to eliminate the intermediate rounding step:

```solidity
// Instead of:
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);

// Use:
maxDebt += _position.collateral[i].mulDivDown(
    price * collateralParam.lltv,
    ORACLE_PRICE_SCALE * WAD
);
```

This computes `floor(c · p · lltv / (ORACLE_PRICE_SCALE · WAD))` in a single step, eliminating the intermediate floor and the associated per-slot precision loss. Verify that `price * collateralParam.lltv` cannot overflow `uint256` given the oracle price bounds enforced elsewhere (e.g., in `NoMultiplicationOverflow.spec`).

## Proof of Concept

**Minimal Foundry fuzz/unit test plan:**

1. Deploy a market with 16 collateral slots, each with a distinct oracle.
2. For each slot `i`, set oracle price `p_i` and collateral amount `c_i` such that `c_i * p_i % ORACLE_PRICE_SCALE == ORACLE_PRICE_SCALE - 1` (maximizing first-step rounding loss) and `floor(c_i * p_i / ORACLE_PRICE_SCALE) * lltv % WAD == WAD - 1` (maximizing second-step rounding loss).
3. Compute `true_maxDebt = sum_i floor(c_i * p_i * lltv / (ORACLE_PRICE_SCALE * WAD))`.
4. Set borrower's debt to `true_maxDebt` (position exactly at boundary).
5. Call `isHealthy` — assert it returns `false` (demonstrating the underestimation).
6. Call `liquidate` — assert it succeeds (demonstrating liquidation of a healthy position).
7. Separately, call `withdrawCollateral` with `assets = 0` — assert it reverts with `UnhealthyBorrower()`.
