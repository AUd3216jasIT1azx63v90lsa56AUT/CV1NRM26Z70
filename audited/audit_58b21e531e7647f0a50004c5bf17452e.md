### Title
Fee-on-Transfer Collateral Inflates `position.collateral` Accounting, Enabling Undercollateralized Borrowing - (File: src/Midnight.sol)

### Summary
`supplyCollateral` unconditionally increments `_position.collateral[collateralIndex]` by the caller-supplied `assets` parameter before calling `safeTransferFrom`, with no balance-delta check. When the collateral token charges a transfer fee, the protocol records more collateral than it actually receives, allowing a borrower to appear more collateralized than they are and borrow beyond what their real token balance supports.

### Finding Description
In `supplyCollateral` (`src/Midnight.sol:524-546`), the accounting write precedes the transfer and trusts the `assets` argument verbatim:

```solidity
// line 533 — records full `assets`
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
// ...
// line 545 — only delivers assets*(1-fee) to address(this) for fee-on-transfer tokens
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

There is no snapshot of `balanceOf(address(this))` before and after the transfer to verify the actual received amount. `SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol:24-34`) only checks the boolean return value; it does not validate the received delta.

`isHealthy` (`src/Midnight.sol:944-960`) then uses the inflated `_position.collateral[i]` directly:

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

**Exploit path via `take` → `onSell` → `supplyCollateral`:**

1. Attacker (borrower) deploys a fee-on-transfer ERC20 (e.g., 1% fee) and creates a market with it as collateral. Market creation (`touchMarket`, `src/Midnight.sol:755-791`) validates only LLTV tiers, sorted addresses, and maxLif — no token-type restriction.
2. A lender creates a buy offer ratified by `SetterRatifier`. The maker calls `setIsRootRatified` (`src/ratifiers/SetterRatifier.sol:24-28`) to ratify a Merkle root containing the offer. `isRatified` (`src/ratifiers/SetterRatifier.sol:30-37`) only verifies the Merkle proof and root membership — no collateral-token checks.
3. Borrower deploys a callback contract implementing `ISellCallback.onSell` that calls `supplyCollateral(market, index, amount, seller)`. Borrower calls `setIsAuthorized(callbackContract, true, borrower)` so the authorization check in `supplyCollateral` passes.
4. Borrower calls `take(lenderOffer, ratifierData, units, borrower, borrower, callbackContract, data)`. Inside `take`:
   - Debt state is updated: `sellerPos.debt += sellerDebtIncrease` (`src/Midnight.sol:414`).
   - Loan tokens are transferred to the borrower (`src/Midnight.sol:455-456`).
   - `onSell` is invoked on `callbackContract` (`src/Midnight.sol:458-473`).
   - Inside `onSell`, `supplyCollateral(market, index, amount, borrower)` is called: `_position.collateral[index] += amount` (records `amount`), then `safeTransferFrom` delivers only `amount * 0.99` to Midnight.
   - After the callback, `isHealthy` is called (`src/Midnight.sol:476`) using the inflated `_position.collateral[index] = amount`, not the actual `amount * 0.99` held.
5. Health check passes. The borrower now has `units` of debt backed by only `amount * 0.99` actual collateral tokens, while the protocol believes it holds `amount`.

No existing check stops this: `supplyCollateral` has no balance-before/after guard; `SafeTransferLib` only checks the return boolean; `isHealthy` reads storage directly; `SetterRatifier.isRatified` is purely offer-validity logic; `touchMarket` imposes no token-type constraints.

### Impact Explanation
The protocol's core solvency invariant — "contract token balances must cover withdrawable assets, collateral claims, credit redemptions, and accrued fees" — is violated immediately upon the first `supplyCollateral` call with a fee-on-transfer token. The borrower's position is undercollateralized from inception: `position.collateral[index]` exceeds the actual token balance held by Midnight for that collateral. When the position is eventually liquidated, the seized collateral is less than the recorded amount, leaving unpayable bad debt that is socialized among lenders.

### Likelihood Explanation
Any permissionless market creator can deploy a market with a fee-on-transfer collateral token — no admin action is required. The borrower only needs to authorize their own callback contract, which is a standard self-authorization. The `live_context.json` explicitly lists "fee-on-transfer … should be tested if not explicitly excluded" as an in-scope invariant, and no exclusion exists in the codebase. The Certora `Healthiness.spec` models `transferFrom` as `HAVOC` (non-deterministic), so the formal verification does not cover this case. The exploit is repeatable on every `supplyCollateral` call with a fee-bearing token.

### Recommendation
Replace the fixed-amount accounting in `supplyCollateral` with a balance-delta pattern:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

This ensures `position.collateral[index]` always equals the actual tokens received, regardless of token transfer mechanics. Alternatively, document and enforce at market creation that fee-on-transfer tokens are prohibited as collateral.

### Proof of Concept
```solidity
// Foundry unit test outline
contract FeeOnTransferCollateral is ERC20 {
    // 1% fee on every transferFrom
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount / 100;
        super.transferFrom(from, to, amount - fee); // delivers amount*0.99
        return true;
    }
}

contract FoTExploit is ISellCallback {
    function onSell(...) external returns (bytes32) {
        uint256 amount = 1000e18;
        FeeOnTransferCollateral(collateralToken).approve(msg.sender, amount);
        Midnight(msg.sender).supplyCollateral(market, 0, amount, seller);
        return CALLBACK_SUCCESS;
    }
}

function testFoTCollateralInflation() public {
    // Setup: deploy FeeOnTransferCollateral, create market, lender buy offer via SetterRatifier
    // Borrower authorizes FoTExploit callback
    uint256 amount = 1000e18;
    deal(address(fotToken), address(exploit), amount);

    uint256 balBefore = fotToken.balanceOf(address(midnight));
    midnight.take(lenderOffer, ratifierData, units, borrower, borrower, address(exploit), "");
    uint256 balAfter = fotToken.balanceOf(address(midnight));

    uint256 recorded = midnight.collateral(id, borrower, 0);
    uint256 actual = balAfter - balBefore;

    // Invariant violation: recorded > actual
    assertGt(recorded, actual); // recorded = 1000e18, actual = 990e18
    // Borrower has debt but is undercollateralized relative to accounting
    assertGt(midnight.debtOf(id, borrower), 0);
    assertTrue(midnight.isHealthy(market, id, borrower)); // passes on inflated collateral
}
```

**Expected assertions:** `recorded (1000e18) > actual (990e18)` — the invariant `position.collateral[index] == actual token balance held` is violated, confirming bad debt creation.
