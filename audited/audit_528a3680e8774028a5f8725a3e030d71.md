### Title
Unclaimed staker principal permanently locked after SURVIVED/EXPIRED resolution with no recovery path — (File: src/ConfidencePool.sol)

### Summary
After SURVIVED or EXPIRED resolution, each staker must individually call `claimSurvived()` or `claimExpired()` to recover their principal. If any staker never claims, their principal is permanently locked in the contract. Unlike the CORRUPTED path — which has `sweepUnclaimedCorrupted()` to drain all remaining funds to `recoveryAddress` after the claim deadline — the SURVIVED/EXPIRED path has no equivalent recovery mechanism. `sweepUnclaimedBonus()` explicitly reserves unclaimed principal and cannot touch it.

### Finding Description
`claimSurvived()` and `claimExpired()` are pull-only: each staker must individually call the function to receive their `eligibleStake[msg.sender]` plus bonus share. [1](#0-0) [2](#0-1)  `totalEligibleStake` is decremented only at claim time, so non-claimers keep it inflated indefinitely.

`sweepUnclaimedBonus()` computes a `reserved` floor that always includes `totalEligibleStake` (all unclaimed principal) and, when `riskWindowStart != 0`, also the remaining unclaimed bonus. It only sweeps `freeBalance − reserved`. [3](#0-2)  This means unclaimed principal is explicitly shielded from the sweep — but there is no other function that can move it.

Contrast this with the CORRUPTED path: `sweepUnclaimedCorrupted()` allows anyone to sweep the entire remaining balance to `recoveryAddress` once `corruptedClaimDeadline` has passed. [4](#0-3)  No analogous time-bounded sweep exists for unclaimed principal under SURVIVED or EXPIRED.

`docs/DESIGN.md` documents `sweepUnclaimedBonus` as recovering "excess over the remaining stakers' entitlement reserve" and lists the CORRUPTED bounty mechanics in §12, but nowhere accepts or acknowledges that unclaimed SURVIVED/EXPIRED principal is permanently irrecoverable. [5](#0-4) 

### Impact Explanation
Any staker who loses their private key, becomes permanently inactive, or simply never calls the claim function after resolution leaves their full principal locked in the contract forever. The sponsor, owner, moderator, and factory owner all have zero ability to recover these tokens. The contract balance will permanently exceed zero with no reachable code path to drain it. The longer the pool's lifetime (up to `expiry + 180 days` grace), the larger the window for stakers to become unreachable.

### Likelihood Explanation
Moderate. Pools can have lifetimes of months to years. Over that horizon, wallet loss, key compromise, or simple inactivity is a realistic outcome for at least a subset of stakers. The protocol has no minimum-claim incentive or deadline, so non-claiming stakers face no on-chain pressure to act. The scenario requires no attacker — ordinary staker inactivity is sufficient.

### Recommendation
Add a time-bounded sweep for unclaimed SURVIVED/EXPIRED principal, mirroring the CORRUPTED path's `sweepUnclaimedCorrupted()`. After a grace period past `outcomeFlaggedAt` (e.g. `expiry + SOME_GRACE`), allow anyone to call a `sweepUnclaimedPrincipal()` that transfers the remaining `stakeToken.balanceOf(address(this))` to `recoveryAddress` and zeroes `totalEligibleStake`. This is consistent with the existing pattern: the CORRUPTED path already accepts that unclaimed funds eventually route to `recoveryAddress`; SURVIVED/EXPIRED should have the same safety valve.

### Proof of Concept
1. Pool is created; `expiry` = T. Stakers A, B, C each stake 100 tokens. `totalEligibleStake` = 300.
2. Registry reaches PRODUCTION. Moderator calls `flagOutcome(SURVIVED, ...)`. `snapshotTotalStaked` = 300.
3. Staker A calls `claimSurvived()` → receives 100 + bonus share. `totalEligibleStake` = 200.
4. Staker B calls `claimSurvived()` → receives 100 + bonus share. `totalEligibleStake` = 100.
5. Staker C loses their private key. `eligibleStake[C]` = 100, `totalEligibleStake` = 100.
6. Anyone calls `sweepUnclaimedBonus()`: `reserved = totalEligibleStake = 100`. `freeBalance` = 100 (C's tokens). `amount = 100 − 100 = 0` → reverts `NothingToSweep`. [6](#0-5) 
7. No other function exists that can move C's 100 tokens. They are permanently locked.
8. For comparison: under CORRUPTED, `sweepUnclaimedCorrupted()` would have drained the entire balance to `recoveryAddress` after `corruptedClaimDeadline`. No equivalent exists here. [7](#0-6)

```markdown
For good-faith CORRUPTED, `bountyEntitlement` and `corruptedReserve` are both set to
`snapshotTotalStaked + snapshotTotalBonus` — i.e. the **entire pool** is the named attacker's
bounty (there is no separate "surplus" alongside an unclaimed bounty in the normal case).

- `claimAttackerBounty` pays `min(remaining, freeBalance)` — the *full* remaining entitlement, not
  a caller-chosen partial. With a normal token, one call satisfies it and unblocks `claimCorrupted`.
- The `MustClaimBountyFirst` gate (good-faith only) and the `sweepUnclaimedCorrupted` deadline are
  intentional: the pool is reserved for the named whitehat for `CORRUPTED_CLAIM_WINDOW` (180 days),
  after which anyone sweeps the remainder to `recoveryAddress`. A donation landing while the bounty
  is unclaimed is held until the attacker claims (then swept immediately) or the window elapses — a
  bounded, recoverable delay, not a lock. A true stuck-state would require a defective stake token
  that freezes the pool's own balance, which the factory allowlist exists to exclude.
```
