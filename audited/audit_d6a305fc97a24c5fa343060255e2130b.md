### Title
Stale `isRootRatified` Persists After Operator Deauthorization in SetterRatifier - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier.setIsRootRatified` checks `isAuthorized` only at write time and stores the result as persistent maker-keyed state. When the maker later revokes the operator's Midnight authorization, `isRootRatified[maker][root]` is not cleared, and `isRatified` never re-validates current authorization. Any taker can continue filling offers under the stale root indefinitely.

### Finding Description
The authorization check in `setIsRootRatified` is a **write-time** gate only:

```solidity
// src/ratifiers/SetterRatifier.sol:24-28
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    ...
}
``` [1](#0-0) 

At fill time, `isRatified` only reads the stored boolean — it never re-checks `isAuthorized`:

```solidity
// src/ratifiers/SetterRatifier.sol:30-37
function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
    require(msg.sender == MIDNIGHT, NotMidnight());
    ...
    require(isRootRatified[offer.maker][root], NotRatified());  // no isAuthorized re-check
    return CALLBACK_SUCCESS;
}
``` [2](#0-1) 

`Midnight.setIsAuthorized` only writes to its own `isAuthorized` mapping and has no knowledge of `SetterRatifier.isRootRatified`: [3](#0-2) 

**Exploit flow:**
1. Maker calls `Midnight.setIsAuthorized(setterRatifier, true, maker)` and `Midnight.setIsAuthorized(operator, true, maker)`.
2. Operator calls `setIsRootRatifier.setIsRootRatified(maker, root, true)` — passes the write-time check, sets `isRootRatified[maker][root] = true`.
3. Maker calls `Midnight.setIsAuthorized(operator, false, maker)` — intending to revoke all operator authority.
4. `isRootRatified[maker][root]` remains `true` in `SetterRatifier` storage.
5. Any taker calls `Midnight.take(offer, abi.encode(root, leafIndex, proof), ...)`. `Midnight.take` checks `isAuthorized[maker][setterRatifier]` (still true), then calls `setterRatifier.isRatified` which returns `CALLBACK_SUCCESS` because `isRootRatified[maker][root]` is still set. [4](#0-3) 

**Contrast with `EcrecoverRatifier`**, which re-checks `isAuthorized` at fill time and has an explicit test verifying revocation invalidates ratification: [5](#0-4) [6](#0-5) 

`SetterRatifier` has no equivalent test and no equivalent runtime check. [7](#0-6) 

### Impact Explanation
After the maker revokes the operator's Midnight authorization, all offers whose roots were set by that operator remain fillable. The maker cannot atomically invalidate them without knowing every root the operator registered. Any taker can fill those offers, causing the maker to take on credit or debt positions they intended to stop accepting.

### Likelihood Explanation
Preconditions are realistic: a maker delegates to an operator (e.g., an automated market-making bot), the operator registers one or more roots, and the maker later revokes the operator (e.g., due to a key compromise or strategy change). The maker calls `setIsAuthorized(operator, false, maker)` expecting full revocation. The stale roots persist until the maker explicitly calls `setIsRootRatified(maker, root, false)` for each one — which requires knowing every root the operator registered. The scenario is repeatable and requires no privileged access beyond the normal maker/operator relationship.

### Recommendation
In `isRatified`, record the address that set each root and re-validate their current authorization at fill time:

```solidity
mapping(address maker => mapping(bytes32 root => address setter)) public rootSetter;

function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    rootSetter[maker][root] = newIsRootRatified ? msg.sender : address(0);
    ...
}

function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
    require(msg.sender == MIDNIGHT, NotMidnight());
    ...
    require(isRootRatified[offer.maker][root], NotRatified());
    address setter = rootSetter[offer.maker][root];
    require(
        setter == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, setter),
        Unauthorized()
    );
    return CALLBACK_SUCCESS;
}
```

Alternatively, follow the `EcrecoverRatifier` pattern and re-check `isAuthorized` for the setter at fill time, or document explicitly that `SetterRatifier` roots are permanent maker-owned state and that makers must manually unratify roots when revoking operators.

### Proof of Concept
```solidity
function testStaleRootAfterRevocation() public {
    Offer memory offer = makeOffer(lender);           // offer.maker = lender
    bytes32 root = HashLib.hashOffer(offer);

    // 1. Maker authorizes setterRatifier and operator
    vm.prank(lender);
    midnight.setIsAuthorized(address(setterRatifier), true, lender);
    vm.prank(lender);
    midnight.setIsAuthorized(operator, true, lender);

    // 2. Operator sets root while authorized
    vm.prank(operator);
    setterRatifier.setIsRootRatified(lender, root, true);
    assertTrue(setterRatifier.isRootRatified(lender, root));

    // 3. Maker revokes operator
    vm.prank(lender);
    midnight.setIsAuthorized(operator, false, lender);
    assertFalse(midnight.isAuthorized(lender, operator));

    // 4. Root is still active — INVARIANT VIOLATION
    assertTrue(setterRatifier.isRootRatified(lender, root));

    // 5. Take still succeeds
    collateralize(offer.market, taker, 1000);
    deal(address(loanToken), lender, 1000);
    vm.prank(taker);
    midnight.take(offer, abi.encode(root, 0, new bytes32[](0)), 1000, taker, taker, address(0), hex"");
    // Assert taker acquired debt despite operator being deauthorized
    assertGt(midnight.debtOf(toId(offer.market), taker), 0);
}
```
