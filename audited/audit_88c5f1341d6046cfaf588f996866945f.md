### Title
Sponsor Can Redirect Full CORRUPTED Sweep by Swapping `recoveryAddress` After Staking — (File: src/ConfidencePool.sol)

### Summary
`setRecoveryAddress` carries no guard on staking status, risk-window state, or outcome state. The pool sponsor (an untrusted actor) can silently replace the CORRUPTED sweep destination at any moment — including after `flagOutcome(CORRUPTED, …)` is called and after withdrawals are permanently disabled — redirecting the entire pool balance to an attacker-controlled address. This is the direct structural analog of the oracle-swap vulnerability: a critical resolution parameter is freely mutable by one party after the counterparty has irrevocably committed.

### Finding Description
`expiry` is explicitly locked after the first stake via the one-way `expiryLocked` latch, with DESIGN.md §10 documenting the rationale ("protects staker reliance"). `recoveryAddress` carries no equivalent protection:

```solidity
// src/ConfidencePool.sol L611-L617
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    address oldRecoveryAddress = recoveryAddress;
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
```

No check on `expiryLocked`, `riskWindowStart`, `outcome`, or `claimsStarted`. The function is callable at any lifecycle stage.

`claimCorrupted` reads `recoveryAddress` live at call time:

```solidity
// src/ConfidencePool.sol L423
stakeToken.safeTransfer(recoveryAddress, toSweep);
```

`sweepUnclaimedCorrupted` does the same at L468. Neither snapshots the address at `flagOutcome` time.

**Concrete attack path (bad-faith CORRUPTED):**

1. Sponsor deploys pool with a legitimate `recoveryAddress` (e.g., a DAO multisig). Stakers verify the parameter and deposit.
2. `riskWindowStart` is set; `withdraw()` is permanently disabled for all stakers.
3. The agreement is breached; the trusted moderator calls `flagOutcome(CORRUPTED, false, address(0))`. The snapshot is taken; `corruptedReserve = snapshotTotalStaked + snapshotTotalBonus`.
4. Sponsor calls `setRecoveryAddress(sponsorControlledAddress)` — no guard reverts this.
5. Any caller (including the sponsor) calls `claimCorrupted`. The entire pool balance is transferred to `sponsorControlledAddress`.

The same path applies to `sweepUnclaimedCorrupted` on the good-faith branch after `corruptedClaimDeadline` elapses.

### Impact Explanation
All staker principal and all bonus tokens are drained to the sponsor's address. Stakers have no recourse: `withdraw()` is permanently disabled once `riskWindowStart != 0`, so they cannot exit after observing the address change. The loss is total and irreversible.

### Likelihood Explanation
The sponsor is the agreement owner and pool deployer — a role that is explicitly **not** listed as trusted in the contest actors ("trusted moderator/DAO" and "trusted factory owner" are trusted; "pool sponsor/agreement owner" is not). The attack requires only that the pool reach a CORRUPTED resolution, which is the normal outcome of an in-scope breach. The sponsor need only submit one transaction between `flagOutcome` and `claimCorrupted` — a trivially executable front-run on a public mempool, or a pre-planned call on a private relay. No special access beyond pool ownership is required.

### Recommendation
Apply the same "staker reliance" rationale that locks `expiry` to `recoveryAddress`. Add a one-way latch — either keyed on `expiryLocked` (lock on first stake) or on `outcome != UNRESOLVED` (lock on resolution) — that prevents `setRecoveryAddress` from executing once stakers have committed:

```solidity
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (expiryLocked) revert RecoveryAddressLocked(); // or a dedicated latch
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    ...
}
```

Alternatively, snapshot `recoveryAddress` into an immutable storage slot inside `flagOutcome` and use that snapshot in `claimCorrupted` / `sweepUnclaimedCorrupted`, mirroring how `snapshotTotalStaked` freezes the accounting at resolution time.

### Proof of Concept

1. Deploy `ConfidencePool` with `recoveryAddress = legitimateDAO`.
2. Stakers call `stake()`; `expiryLocked = true`; `riskWindowStart` is set; `withdraw()` permanently disabled.
3. Registry transitions to `CORRUPTED`; moderator calls `flagOutcome(CORRUPTED, false, address(0))`.
   - `snapshotTotalStaked` and `snapshotTotalBonus` are frozen. [1](#0-0) 
4. Sponsor calls `setRecoveryAddress(sponsorAddress)`. No guard prevents this. [2](#0-1) 
5. Any caller invokes `claimCorrupted()`. `stakeToken.safeTransfer(recoveryAddress, toSweep)` sends the full pool balance to `sponsorAddress`. [3](#0-2) 
6. All staker principal and bonus are lost. Stakers cannot withdraw because `riskWindowStart != 0`. [4](#0-3) 

The asymmetry with `expiry` — which is explicitly locked after first stake with documented rationale — confirms this is an implementation defect, not an accepted design trade-off. DESIGN.md §10 documents the `expiryLocked` latch to "protect staker reliance" but records no equivalent intent for `recoveryAddress`. [5](#0-4)
