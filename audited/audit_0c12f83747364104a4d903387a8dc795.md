Audit Report

## Title
Debt increase permitted at exact maturity timestamp due to `<=` boundary check - (File: src/Midnight.sol)

## Summary
The `take()` function uses `block.timestamp <= offer.market.maturity` at line 391 to guard against post-maturity debt increases. When `block.timestamp == offer.market.maturity`, this condition is `true`, causing the `||` to short-circuit and the `require` to pass unconditionally regardless of `sellerDebtIncrease`. This directly violates the protocol invariant explicitly stated in `live_context.json` line 221: "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."

## Finding Description
**Root cause:** Line 391 of `src/Midnight.sol`:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
When `block.timestamp == offer.market.maturity`, the left operand evaluates to `true`, making the entire `||` expression unconditionally `true`. The `require` never reverts regardless of `sellerDebtIncrease`.

**Exact code path at `block.timestamp == offer.market.maturity`:**
1. Line 359: `timeToMaturity = zeroFloorSub(offer.market.maturity, block.timestamp)` returns `0`. All price/fee computations proceed with TTM=0.
2. Line 383: `sellerCreditDecrease = min(units, sellerPos.credit)`. If the seller has no existing credit, this is `0`.
3. Line 384: `sellerDebtIncrease = units - 0 = units > 0`.
4. Line 391: Guard passes unconditionally because `block.timestamp <= offer.market.maturity` is `true`.
5. Line 414: `sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease)` — new debt is minted at maturity.

**Why existing checks fail:**
- The expiry check at line 353 (`block.timestamp <= offer.expiry`) is orthogonal; the offer can have `expiry >= maturity`.
- The `reduceOnly` check at lines 392–395 only applies if the maker explicitly set that flag.
- The gate checks at lines 402–406 only apply if a gate is configured and do not substitute for the maturity guard.
- All existing post-maturity tests in `test/TakeTest.sol` use `market.maturity + 1` as the warp target (lines 317, 330, 349, 391, 406, 667, 680), leaving the `block.timestamp == offer.market.maturity` case entirely untested.

## Impact Explanation
A taker can cause `sellerPos.debt` to increase at the exact maturity block, corrupting the debt/credit accounting invariant. A borrower who should be in a pure-repayment/settlement phase has new debt minted against them. This can push a previously healthy position into an unhealthy state at the moment maturity is reached, enabling immediate liquidation of a position that should have been protected from new debt. This constitutes credit/debt accounting corruption and bad debt creation — both listed as highest-priority bug classes in `live_context.json`. The violated invariants are explicitly stated at `live_context.json` lines 197 ("debt must not increase after maturity") and 221 ("maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing").

## Likelihood Explanation
The precondition `block.timestamp == offer.market.maturity` is a single-block window that occurs exactly once per market. A taker monitoring the chain can submit a transaction in that specific block. The offer requires only a valid ratifier (e.g., a signature-based ratifier the maker already authorized) and a seller with no existing credit. No privileged access, oracle manipulation, or admin action is required. This is repeatable across every market whose maturity falls on a future block.

## Recommendation
Change `<=` to `<` in the maturity guard at line 391:
```solidity
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
This ensures that at `block.timestamp == offer.market.maturity`, the left operand is `false`, forcing the `require` to check `sellerDebtIncrease == 0` and revert if new debt would be minted. Add a corresponding test warping to exactly `market.maturity` with a seller having no existing credit, asserting revert with `CannotIncreaseDebtPostMaturity`.

## Proof of Concept
```solidity
function testBuyAtExactMaturity() public {
    uint256 units = 100;
    uint256 timestamp = market.maturity; // exact boundary
    vm.warp(timestamp);
    borrowerOffer.expiry = timestamp;
    borrowerOffer.maxUnits = units;
    deal(address(loanToken), lender, units);
    collateralize(market, borrower, units);
    // borrower has no existing credit, so sellerDebtIncrease = units > 0
    // expect revert — but with current code, it does NOT revert
    vm.expectRevert(IMidnight.CannotIncreaseDebtPostMaturity.selector);
    take(units, lender, borrowerOffer);
    // Without the fix, the above expectRevert fails because take() succeeds,
    // and borrower.debt increases by `units` at the maturity block.
}
```
This mirrors the structure of `testBuy1PostMaturity` (line 315 of `test/TakeTest.sol`) but warps to `market.maturity` instead of `market.maturity + 1`, exposing the untested boundary.
