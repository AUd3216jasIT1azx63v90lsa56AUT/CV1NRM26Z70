Audit Report

## Title
Zero-price tick enables unbacked credit and debt creation with no loan token transfer - (`src/libraries/TickLib.sol`, `src/Midnight.sol`)

## Summary
`TickLib.tickToPrice(0)` returns `0`, confirmed by both the protocol's own test suite and a Certora formal verification rule. Because tick 0 always satisfies `0 % tickSpacing == 0` for any spacing, and `take` contains no guard requiring `offerPrice > 0`, a sell offer at tick 0 in a zero-settlement-fee market (the default) produces `buyerAssets = 0` and `sellerAssets = 0` while still granting the buyer real credit units and accumulating real debt on the seller. When `maxAssets > 0` is used, the consumed capacity is never incremented (since `sellerAssets = 0`), allowing the same offer to be taken an unlimited number of times. The attacker's accumulated credit can drain `withdrawable` funded by legitimate borrowers' repayments.

## Finding Description

**Root cause — `tickToPrice(0) == 0`:**

`TickLib.tickToPrice` computes a rounded price that is zero for ticks 0 and 1. This is explicitly confirmed by the protocol's own test:

```solidity
// test/TickLibTest.sol:16-17
assertEq(TickLib.tickToPrice(0), 0, "tick 0");
assertEq(TickLib.tickToPrice(2), 1e12, "first non-zero tick");
```

And by the Certora formal verification rule:

```
// certora/specs/TickToPrice.spec:34-36
rule tickToPriceIsZeroAtZero() {
    assert tickToPrice(0) == 0;
}
```

**Tick 0 always passes the spacing check:**

```solidity
// src/Midnight.sol:351
require(offer.tick % _marketState.tickSpacing == 0, TickNotAccessible());
```

`DEFAULT_TICK_SPACING = 4` and `0 % 4 == 0`, so tick 0 is always accessible regardless of spacing.

**Settlement fee defaults to zero:**

`defaultSettlementFeeCbp` is a storage mapping zero-initialized. `touchMarket` copies it into the market state, so all markets start with `_settlementFee = 0` unless the privileged `feeSetter` explicitly configures fees.

**No guard on zero price in `take`:**

```solidity
// src/Midnight.sol:358-364
uint256 offerPrice = TickLib.tickToPrice(offer.tick);   // == 0 for tick 0
uint256 _settlementFee = settlementFee(id, timeToMaturity); // == 0 by default
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice; // == 0
uint256 buyerPrice = sellerPrice + _settlementFee;  // == 0
uint256 buyerAssets = ... units.mulDivUp(buyerPrice, WAD);  // == 0
uint256 sellerAssets = ... units.mulDivUp(sellerPrice, WAD); // == 0
```

There is no `require(offerPrice > 0)` check anywhere in `take`.

**Credit and debt are updated regardless:**

```solidity
// src/Midnight.sol:408-417
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);   // += units
sellerPos.debt  += UtilsLib.toUint128(sellerDebtIncrease);    // += units
_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease;
```

**Zero tokens are transferred:**

```solidity
// src/Midnight.sol:455-456
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // 0
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);                   // 0
```

Standard ERC-20 tokens do not revert on zero-value transfers. The protocol's own token safety requirements explicitly state tokens "should not revert because a transfer is no-op."

**`maxAssets` consumption is never incremented:**

```solidity
// src/Midnight.sol:367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For a sell offer, `sellerAssets = 0`, so `consumed` never increases. `newConsumed` stays at 0, which always satisfies `<= offer.maxAssets`. The same offer can be taken an unlimited number of times.

**Health check does not prevent the invariant violation:**

```solidity
// src/Midnight.sol:944-959
function isHealthy(...) public view returns (bool) {
    ...
    return maxDebt >= debt;
}
```

`isHealthy` only verifies that collateral covers debt at oracle price. It does not verify that the protocol's loan token balance covers outstanding credit. A seller with sufficient collateral passes the check even though zero loan tokens entered the contract.

**Exploit flow (two-account attack):**

1. Attacker account A (maker/seller) supplies sufficient collateral and creates a sell offer at tick 0 with `maxAssets > 0`.
2. Attacker account B (taker/buyer) calls `take` with `units > 0`. Self-take is blocked by `require(offer.maker != taker, SelfTake())`, so two accounts are required.
3. B pays 0 loan tokens, receives `units` of credit. A accumulates `units` of debt, receives 0 loan tokens.
4. `_marketState.totalUnits` increases by `units` with no corresponding loan token deposit.
5. Step 2 is repeated arbitrarily (since `sellerAssets = 0` never increments `consumed`), bounded only by A's collateral-backed `maxDebt`.
6. When legitimate borrowers repay, `withdrawable` increases. B calls `withdraw` to drain those tokens, stealing from legitimate lenders.

## Impact Explanation

B holds credit claims unbacked by any deposited loan tokens. When legitimate borrowers repay debt, `withdrawable` increases and B can drain it before legitimate lenders. The `totalUnits == sumDebt + withdrawable` accounting invariant is maintained in ledger terms (A's debt equals B's credit), but the solvency invariant is violated: the contract's loan token balance does not cover all outstanding credit redemptions. This constitutes direct, concrete theft of legitimate lenders' deposited loan tokens. Severity: Critical.

## Likelihood Explanation

All preconditions are trivially satisfied by any unprivileged user: tick 0 is always accessible at any tick spacing, zero settlement fees are the default for all new markets, and the two-account requirement is trivial. No privileged access is required. The exploit is repeatable without bound when `maxAssets > 0` is used. Any market with default (zero) settlement fees is vulnerable from the moment of creation.

## Recommendation

Add a guard in `take` requiring a non-zero offer price:

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
require(offerPrice > 0, ZeroPrice());
```

Alternatively, enforce a minimum tick (e.g., `require(offer.tick >= 2)`) since ticks 0 and 1 both map to price 0. A defense-in-depth measure would also be to require `buyerAssets > 0` or `units == 0` when price is zero, preventing credit/debt creation with no token flow.

## Proof of Concept

Minimal Foundry test outline:

```solidity
// 1. Deploy market with default (zero) settlement fees.
// 2. Account A supplies collateral sufficient for N units of debt.
// 3. A creates sell offer: offer.buy=false, offer.tick=0, offer.maxAssets=1e18.
// 4. B calls take(offer, ..., units=N, taker=B, ...) — pays 0 tokens.
// 5. Assert: midnight.creditOf(id, B) == N, midnight.debtOf(id, A) == N,
//            loanToken.balanceOf(address(midnight)) unchanged.
// 6. Repeat step 4 (consumed never increments, offer never exhausted).
// 7. Legitimate lender L deposits M tokens via a buy offer at a real tick.
//    Legitimate borrower repays M units → withdrawable += M.
// 8. B calls withdraw(market, M, B, B) — drains L's tokens.
// 9. Assert: loanToken.balanceOf(B) == M, L cannot withdraw.
```

The Certora rule `tickToPriceIsZeroAtZero` and the unit test `testTickToPriceMinMax` already confirm the root cause. A complete Foundry integration test following the above steps would reproduce the full exploit.
