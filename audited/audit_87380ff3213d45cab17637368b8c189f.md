### Title
Zero-price tick allows debt accumulation and credit issuance with zero loan token transfer - (`src/libraries/TickLib.sol`, `src/Midnight.sol`)

### Summary
`TickLib.tickToPrice(0)` and `tickToPrice(1)` both return exactly `0`, as confirmed by the protocol's own test suite. The `take` function contains no guard requiring `offerPrice > 0`, so a sell offer placed at tick 0 or tick 1 causes `sellerAssets = 0` and `buyerAssets = 0` (when settlement fee is also zero). The seller accumulates real debt units while receiving zero loan tokens, and the buyer receives real credit units while paying zero loan tokens, breaking the invariant that contract loan-token balances cover all credit redemptions.

### Finding Description

**Root cause — `tickToPrice` returns 0 for ticks 0 and 1:** [1](#0-0) 

For tick = 0, the exponent argument is `LN_ONE_PLUS_DELTA * 2910 ≈ 14.51e18`, so `wExp(...)` ≈ `2e24`. The first `divHalfDownUnchecked` yields `1e36 / (1e18 + 2e24) ≈ 5e11`. The second `divHalfDownUnchecked(5e11, 1e12)` computes `(5e11 + 4.999...e11) / 1e12 = 999...e11 / 1e12 = 0` (integer division), so `0 * 1e12 = 0`. The protocol's own test confirms this: [2](#0-1) 

Ticks 0 and 1 both return price 0; tick 2 is the first non-zero price (`1e12`).

**Tick 0 is always accessible:** [3](#0-2) 

`0 % tickSpacing == 0` for any non-zero `tickSpacing`, so tick 0 always passes the accessibility check.

**Price and asset computation with no guard:** [4](#0-3) 

With `offerPrice = 0` and `_settlementFee = 0` (valid for markets with all fee breakpoints at zero):
- `sellerPrice = 0` (sell offer: `offer.buy = false`)
- `buyerPrice = 0`
- `buyerAssets = units.mulDivUp(0, WAD) = 0`
- `sellerAssets = units.mulDivUp(0, WAD) = 0`

**Debt and credit are updated regardless:** [5](#0-4) [6](#0-5) 

`sellerDebtIncrease = units` (assuming seller has no prior credit). `sellerPos.debt += units`. `buyerPos.credit += buyerCreditIncrease`. `_marketState.totalUnits` grows by `buyerCreditIncrease`.

**Zero tokens are transferred:** [7](#0-6) 

`safeTransferFrom(..., 0)` and `safeTransferFrom(..., 0)` — no loan tokens enter the contract or reach the seller.

**Health check does not prevent the invariant violation:** [8](#0-7) 

The health check only verifies the seller's collateral covers their debt at the oracle price. It does not verify that the protocol's loan token balance covers the newly issued credit. A seller with sufficient collateral passes the check even though they received zero loan tokens.

**Exploit flow (two-account attack):**
1. Attacker account A (maker/seller) creates a sell offer at tick 0 with sufficient collateral to remain healthy.
2. Attacker account B (taker/buyer) calls `take` with `units > 0`. Self-take is blocked, so two accounts are required.
3. Account B pays 0 loan tokens, receives `units` of credit.
4. Account A accumulates `units` of debt, receives 0 loan tokens.
5. `_marketState.totalUnits` increases by `units` with no corresponding loan token deposit.
6. When other lenders' repayments make `withdrawable > 0`, account B can drain those tokens via `withdraw`, stealing from legitimate lenders.

### Impact Explanation
The seller (account A) accumulates debt without receiving any loan assets — a textbook undercollateralized position from the protocol's accounting perspective. The buyer (account B) holds credit claims unbacked by any deposited loan tokens. `totalUnits` (the aggregate credit claim) exceeds the actual loan token balance held by the contract, violating the core invariant that contract balances cover credit redemption. Legitimate lenders' withdrawable balances can be drained by the attacker's credit position.

### Likelihood Explanation
Preconditions are easily satisfied: tick 0 is always accessible (0 % any spacing = 0), and markets with zero settlement fees are valid configurations. The two-account requirement is trivial for any attacker. The exploit is repeatable — it can be executed as many times as desired, accumulating unbacked credit. No privileged access is required.

### Recommendation
Add a check in `take` (or in offer validation) that `offerPrice > 0`:

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
require(offerPrice > 0, ZeroPrice()); // add this guard
```

Alternatively, enforce a minimum tick (e.g., `offer.tick >= 2`) so that only ticks with non-zero prices are accessible. A third option is to enforce `sellerAssets > 0` when `sellerDebtIncrease > 0`, directly protecting the debt-must-have-asset-transfer invariant.
