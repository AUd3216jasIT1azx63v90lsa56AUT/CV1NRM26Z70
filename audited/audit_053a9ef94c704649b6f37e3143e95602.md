### Title
Transitive Authorization Delegation Enables Unauthorized `cancelRoot` via Two-Hop Privilege Escalation - (File: src/ratifiers/EcrecoverRatifier.sol)

### Summary
`Midnight.setIsAuthorized` allows any address authorized by a maker to grant that same full authorization to arbitrary third parties, with no restriction on scope or depth. Because `EcrecoverRatifier.cancelRoot` accepts any address that `isAuthorized[maker][msg.sender]` returns true for, an attacker who holds any authorization from a maker can silently delegate to a second address and use it to cancel the maker's entire Merkle offer tree.

### Finding Description
**Root cause:** `Midnight.sol:732` checks `isAuthorized[onBehalf][msg.sender]` to gate `setIsAuthorized`, meaning any authorized delegate can grant new authorizations on behalf of the principal — including to addresses the principal has never interacted with. There is no depth limit, no scope restriction, and no separate permission required to sub-delegate. [1](#0-0) 

**Exploit flow:**

1. Maker legitimately calls `midnight.setIsAuthorized(attacker1, true, maker)` (e.g., to allow attacker1 to sign offers or manage positions).
2. Attacker1 calls `midnight.setIsAuthorized(attacker2, true, maker)`. The `isAuthorized[maker][attacker1]` check passes; `isAuthorized[maker][attacker2]` is now `true`.
3. Attacker2 calls `ecrecoverRatifier.cancelRoot(maker, root)`. The `IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender)` check passes; `isRootCanceled[maker][root]` is set to `true`. [2](#0-1) 

4. Any subsequent `take` call using an offer from that Merkle tree reverts with `RootCanceled`. [3](#0-2) 

**Why existing checks fail:** Both guards individually enforce "caller is maker or authorized by maker." Neither guard prevents an authorized address from sub-delegating, and neither `cancelRoot` nor `setIsAuthorized` checks whether the authorization was granted directly by the maker vs. transitively through a delegate.

### Impact Explanation
An attacker who holds any authorization from a maker (granted for any purpose) can cancel the maker's entire active offer batch in `EcrecoverRatifier` without the maker's knowledge or consent. All in-flight offers under the canceled root become permanently unfillable, constituting a complete denial of the maker's offer activity. The maker must re-sign and re-publish a new Merkle tree.

### Likelihood Explanation
**Preconditions:** The maker must have authorized at least one address (attacker1) for any reason — this is a normal operational action (e.g., authorizing a signing bot, a position manager, or the `EcrecoverRatifier` itself). Attacker1 need not be malicious at authorization time; the attack can be executed at any later point. The attack is two transactions, requires no funds, and is repeatable against every new root the maker publishes.

### Recommendation
Prevent sub-delegation in `setIsAuthorized`: only the principal themselves (`onBehalf == msg.sender`) should be allowed to grant new authorizations. Remove the `isAuthorized[onBehalf][msg.sender]` branch from the `setIsAuthorized` guard, or introduce a separate, non-delegatable `canDelegate` permission. Alternatively, scope authorizations by action type so that a delegate authorized for signing cannot also call `cancelRoot` or grant further authorizations. [4](#0-3)
