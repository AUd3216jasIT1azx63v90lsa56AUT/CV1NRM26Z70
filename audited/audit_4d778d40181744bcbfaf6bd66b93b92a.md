Audit Report

## Title
Continuous Fee Unclaimable When `withdrawable < continuousFeeCredit` Due to Missing `withdrawable` Increase in `_updatePosition` - (File: src/Midnight.sol)

## Summary
`_updatePosition` credits accrued fees to `continuousFeeCredit` but never increases `withdrawable`. Because `claimContinuousFee` unconditionally decrements `withdrawable` by `amount`, any claim attempt where `amount > withdrawable` reverts via checked-arithmetic underflow, even when `amount <= continuousFeeCredit`. The fee claimer has no unilateral ability to extract accrued continuous fees while debt remains outstanding.

## Finding Description

**Root cause — `_updatePosition` does not touch `withdrawable`:**

`_updatePosition` at `src/Midnight.sol:842-846` reduces the lender's `credit` by `accruedFee` and adds it to `continuousFeeCredit`, but leaves `withdrawable` unchanged:

```solidity
_position.credit = newCredit;
...
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
```

The Certora rule `withdrawableUnchanged` at `certora/specs/WithdrawableMonotonicity.spec:45-57` formally verifies this: `updatePosition` is not excluded from the "unchanged" filter, confirming it is proven to leave `withdrawable` unmodified. The only functions that increase `withdrawable` are `repay` (proven by `repayIncreasesWithdrawable`) and `liquidate` (proven by `liquidateIncreasesWithdrawable`).

**Revert path in `claimContinuousFee`:**

`claimContinuousFee` at `src/Midnight.sol:318-320` performs three unchecked decrements:

```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
_marketState.totalUnits          -= UtilsLib.toUint128(amount);
_marketState.withdrawable        -= UtilsLib.toUint128(amount);  // underflow if withdrawable < amount
```

There is no `require(amount <= withdrawable)` guard. The revert is a Solidity checked-arithmetic underflow on line 320.

**Exploit flow:**

1. Market created; lender supplies `C`, borrower takes debt `C`. State: `withdrawable = 0`, `continuousFeeCredit = 0`.
2. Time passes. Anyone calls `updatePosition(market, lender)`. State: `continuousFeeCredit = F > 0`, `withdrawable = 0` (unchanged).
3. Fee claimer calls `claimContinuousFee(market, F, receiver)`.
4. Line 320 executes `0 - F` → underflow revert.

**Test confirmation:**

`testClaimContinuousFee` at `test/ContinuousFeeTest.sol:420-423` explicitly calls `repay` before `claimContinuousFee` with the comment *"Repay so withdrawable covers the claim"*, directly acknowledging the dependency. The test `testClaimContinuousFeeExcessReverts` at `test/ContinuousFeeTest.sol:447-464` calls `updatePosition` then attempts to claim `feeAmount + 1` and expects a revert — but notably, even claiming exactly `feeAmount` (with `withdrawable = 0`) would revert at line 320, not line 318.

## Impact Explanation

The continuous fee is unclaimable whenever `withdrawable < continuousFeeCredit`. In any market with fully outstanding debt and no repayments, `withdrawable` remains 0 while `continuousFeeCredit` grows with every `updatePosition` call. The fee claimer cannot extract any accrued fees without borrower cooperation (repayment or liquidation). Fees are not permanently lost but are frozen for the entire duration that debt remains outstanding, which can span the full loan term.

## Likelihood Explanation

This condition is reachable in any market with outstanding debt and no repayments — the normal operating state of an active lending market. No attacker action is required: simply calling the permissionless `updatePosition` after time has elapsed creates `continuousFeeCredit > 0, withdrawable = 0`. The condition persists for the full duration of outstanding debt and is repeatable across all markets.

## Recommendation

In `_updatePosition`, increase `withdrawable` by `accruedFee` alongside the `continuousFeeCredit` increase:

```solidity
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
marketState[id].withdrawable        += UtilsLib.toUint128(accruedFee);
```

This maintains the invariant `continuousFeeCredit <= withdrawable` at all times and allows the fee claimer to unilaterally claim accrued fees. The Certora rule `withdrawableUnchanged` would need to be updated to exclude `updatePosition`.

## Proof of Concept

```solidity
function testClaimContinuousFeeBlockedWithoutRepay() public {
    uint256 credit = 1e18;
    uint256 feeRate = 0.1e18; // 10%
    uint256 ttm = 365 days;

    setupLender(credit, feeRate, ttm);

    // Advance time and accrue fees
    vm.warp(vm.getBlockTimestamp() + 100 days);
    midnight.updatePosition(market, lender);

    uint256 feeAmount = midnight.continuousFeeCredit(id);
    assertGt(feeAmount, 0, "fee accrued");
    assertEq(midnight.withdrawable(id), 0, "withdrawable still zero");

    // Fee claimer attempts to claim — reverts due to underflow on withdrawable
    vm.prank(feeClaimer);
    vm.expectRevert(); // underflow: 0 - feeAmount
    midnight.claimContinuousFee(market, feeAmount, feeClaimer);
}
```
