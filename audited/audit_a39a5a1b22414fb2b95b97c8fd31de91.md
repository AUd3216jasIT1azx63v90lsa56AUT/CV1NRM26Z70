### Title
Zero-value `safeTransferFrom` for settlement fee blocks valid takes on tokens that revert on zero transfers - (`src/Midnight.sol`)

### Summary

In `Midnight.take`, the settlement fee is transferred unconditionally via `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets)` at line 455. When `buyerAssets == sellerAssets` â€” which occurs whenever `_settlementFee == 0` or when `units * _settlementFee < WAD` causes both `mulDivUp`/`mulDivDown` calls to round to the same integer â€” this issues a zero-value `transferFrom` call. If the loan token reverts on zero-value transfers (a non-standard but real ERC20 behavior), the entire `take` reverts, blocking an otherwise valid trade.

### Finding Description

**Code path:**

`src/Midnight.sol`, `take` function:


`SafeTransferLib.safeTransferFrom` (lines 24â€“34 of `src/libraries/SafeTransferLib.sol`) calls `token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)))` with no zero-value guard. If the token's `transferFrom` reverts on `value == 0`, the assembly block at lines 28â€“31 re-reverts with the token's error data, propagating the revert up through `take`.

**Root cause:** No conditional guard on the settlement fee transfer. The amount `buyerAssets - sellerAssets` is zero in two reachable cases:

1. **`_settlementFee == 0`** â€” a fully valid configuration (default fees start at 0 and can be explicitly set to 0). In this case `buyerPrice == sellerPrice`, so `buyerAssets == sellerAssets` for every `units` value, blocking all takes on the market.

2. **Rounding absorbs the fee** â€” when `units * _settlementFee < WAD`, both ceiling (or floor) divisions produce the same integer. Concrete example for a sell offer (`!offer.buy`, uses `mulDivUp`):
   - `units = 1`, `_settlementFee = 1e12` (1 cbp, the minimum non-zero fee)
   - `offerPrice = 1e18 - 1e12 - 1` â†’ `buyerPrice = 1e18 - 1`
   - `sellerAssets = âŒˆ(1e18 - 1e12 - 1) / 1e18âŒ‰ = 1`
   - `buyerAssets  = âŒˆ(1e18 - 1) / 1e18âŒ‰ = 1`
   - `buyerAssets - sellerAssets = 0` â†’ zero-value transfer issued

**Attacker inputs:** Any taker calling `take` with `units = 1` (or any small value satisfying `units * _settlementFee < WAD`) on a market whose loan token reverts on zero-value `transferFrom`. No privileged access is required; the taker is unprivileged.

**Why existing checks fail:** There is no `require(buyerAssets > sellerAssets)` or `if (buyerAssets > sellerAssets)` guard anywhere in `take`. The Certora formal verification in `certora/specs/Role.spec` (line 22â€“24) explicitly assumes tokens never revert, so this case is outside the verified safety envelope.

### Impact Explanation

Any take where `buyerAssets == sellerAssets` reverts unconditionally on a zero-reverting loan token. When `_settlementFee == 0`, this blocks every single take on the affected market â€” sellers cannot reduce debt, buyers cannot acquire credit, and all resting offers become unfillable. When the fee is non-zero but small, takes with small `units` are selectively blocked. In both cases the taker's valid trade is denied with no recourse, and any seller who needs to reduce debt via take is stuck.

### Likelihood Explanation

**Precondition 1 (token reverts on zero transfer):** Non-standard but real. Several deployed ERC20s revert on zero-value transfers (e.g., some wrapped tokens, custom protocol tokens). The ERC20 standard does not mandate that zero-value transfers succeed.

**Precondition 2 (`buyerAssets == sellerAssets`):** Trivially satisfied whenever `_settlementFee == 0`, which is the default state before any fee is set and a valid explicit configuration. The rounding path is also easily reached with `units = 1` and any tick near par.

**Repeatability:** Deterministic and repeatable â€” every call to `take` under these conditions reverts. The condition is not race-dependent.

### Recommendation

Wrap the settlement fee transfer in a zero-value guard:

```solidity
if (buyerAssets > sellerAssets) {
    SafeTransferLib.safeTransferFrom(
        offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets
    );
}
```

This is consistent with how `claimableSettlementFee` is already updated unconditionally (line 418 correctly adds 0 when the spread is zero), and it mirrors the pattern used in other DeFi protocols that handle zero-fee markets.
