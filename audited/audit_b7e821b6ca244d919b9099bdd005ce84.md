Audit Report

## Title
Buy Offer `maxAssets` Cap Bypassed via `mulDivDown` Rounding to Zero in Unit Terms - (File: src/Midnight.sol)

## Summary
When a buy offer uses `maxAssets` as its cap, the consumed counter is incremented by `buyerAssets = mulDivDown(units, buyerPrice, WAD)`. For any tick where `buyerPrice < WAD` (the normal case), a taker can choose `units` small enough that `units * buyerPrice < WAD`, causing `buyerAssets = 0`. The consumed counter does not advance, yet position state (`credit`, `debt`, `totalUnits`) is updated by the full `units` amount, allowing repeated zero-cost fills that accumulate unbounded unit exposure beyond the maker's intended `maxAssets` cap.

## Finding Description
In `src/Midnight.sol`, `take()`, lines 363–369:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// ...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For a buy offer (`offer.buy = true`), `buyerPrice = tickToPrice(tick) + settlementFee`. Since `tickToPrice` returns values ≤ WAD and settlement fees are small, `buyerPrice < WAD` is the normal case for all sub-WAD ticks. When `units * buyerPrice < WAD`, `mulDivDown` returns 0. The consumed counter increments by 0, the `require` check trivially passes, but lines 408–417 still apply the full `units` to positions:

```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);
_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease;
```

The `ConsumedAssets` guard only checks the asset-denominated counter and has no awareness of unit-denominated exposure added to positions. When `buyerAssets = 0`, the guard is a no-op regardless of how many units are filled. The protocol's own comment at line 94 acknowledges this: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test `testBugBuyMaxAssetsBypass` (lines 858–889 of `test/TakeTest.sol`) explicitly demonstrates the scenario: consumed stays at `maxAssets`, token balances are unchanged, but `creditOf`, `debtOf`, and `totalUnits` all increase.

## Impact Explanation
A maker who posts a buy offer with `maxAssets = N` intends to cap their total credit exposure at `N` loan tokens. An attacker can call `take(units=1)` repeatedly, accumulating arbitrarily large credit/debt positions in the market while the consumed counter never exceeds `N`. The maker's unit-denominated exposure is unbounded despite the asset cap. This violates the core invariant that offers cannot be overfilled, constitutes unauthorized state change (unbounded position growth without corresponding asset transfer), and can result in the maker holding far more credit/debt than intended — a direct accounting integrity failure with concrete financial impact.

## Likelihood Explanation
Preconditions are minimal: any buy offer with `maxAssets > 0` and `buyerPrice < WAD` (the normal case for all sub-WAD ticks) is vulnerable. No special privileges are needed — any `msg.sender` can be the taker. The condition `units * buyerPrice < WAD` is trivially satisfied with `units = 1` at any realistic price. The attack is repeatable indefinitely and costs only gas.

## Recommendation
Add a unit-denominated floor check: require `buyerAssets > 0` (or equivalently `units * buyerPrice >= WAD`) before proceeding with a `maxAssets`-capped buy offer fill. Alternatively, track consumed in units when `buyerAssets` rounds to zero, or enforce a minimum `units` value such that `mulDivDown(units, buyerPrice, WAD) >= 1`. A combined guard would be:

```solidity
require(buyerAssets > 0 || offer.maxAssets == 0, ZeroAssetFill());
```

This ensures the consumed counter always advances by at least 1 for any fill that modifies position state under an asset cap.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) is a complete, minimal reproduction:
1. Deploy with a buy offer at `MAX_TICK - 16` (price < WAD), `maxAssets = 1`, `maxUnits = 0`.
2. Pre-consume the offer to `maxAssets` via `setConsumed`.
3. Call `take(units=1, borrower, lenderOffer)`.
4. Assert `buyerAssets == 0`, `consumed == maxAssets` (unchanged), token balances unchanged.
5. Assert `creditOf`, `debtOf`, and `totalUnits` all strictly increased.

All five assertions pass, confirming the bypass.

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);

        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
