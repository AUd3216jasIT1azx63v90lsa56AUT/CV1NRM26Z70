Audit Report

## Title
Taker-Controlled `takerCallback` Sets `payer` to Arbitrary Third Party on Sell Offers - (File: `src/Midnight.sol`)

## Summary
In `Midnight.take`, when processing a sell offer (`offer.buy = false`), the `payer` address is derived directly from the taker-supplied `takerCallback` with no check that `takerCallback` is the taker or authorized by the taker. An attacker can set `takerCallback = victimContract`, causing all loan token transfers to be pulled from `victimContract` rather than from the attacker. Any contract that implements `IBuyCallback`, returns `CALLBACK_SUCCESS` from `onBuy` (e.g., only checking `msg.sender == midnight`), and holds a token approval to Midnight can have its tokens drained.

## Finding Description

**Root cause — no authorization constraint on `takerCallback`:**

The only taker-side authorization check in `take` is:

```solidity
// src/Midnight.sol:346
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());
```

This verifies that `msg.sender` may act as `taker`, but places no constraint on `takerCallback`. The parameter is accepted as-is.

**Payer derivation for sell offers:**

```solidity
// src/Midnight.sol:420-422
address buyerCallback = offer.buy ? offer.callback : takerCallback;   // sell: = takerCallback (attacker-controlled)
address sellerCallback = offer.buy ? takerCallback : offer.callback;
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
// sell + non-zero takerCallback → payer = takerCallback
```

**`onBuy` is invoked on the attacker-chosen address before any transfer:**

```solidity
// src/Midnight.sol:445-452
if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback).onBuy(...) == CALLBACK_SUCCESS,
        WrongBuyCallbackReturnValue()
    );
}
```

**Transfers then pull from `payer = victimContract`:**

```solidity
// src/Midnight.sol:455-456
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

**Exploit flow:**
1. Maker signs a sell offer (`offer.buy = false`).
2. Attacker calls `take(offer, ratifierData, units, attacker, receiver, victimContract, data)`.
3. `buyerCallback = victimContract`, `payer = victimContract`.
4. `victimContract.onBuy(...)` is called by Midnight; if `victimContract` checks only `msg.sender == midnight` (a common pattern), it returns `CALLBACK_SUCCESS`.
5. `safeTransferFrom(loanToken, victimContract, ...)` pulls `buyerAssets` from `victimContract`.
6. The attacker's position is credited; the victim paid.

**Why existing checks fail:**
- The `TakerUnauthorized` check (line 346) only guards who may act as `taker`; it does not constrain `takerCallback`.
- The ratifier (`EcrecoverRatifier.isRatified`) validates the maker's signature over `offer` fields; `takerCallback` is not part of the signed offer and is not validated.
- The Certora spec `OnlyExplicitPayerCanLoseTokens.spec` explicitly models and proves that tokens may be pulled from any `buyerCallback` that returned `CALLBACK_SUCCESS` — confirming this path is reachable by design with no additional guard.

## Impact Explanation
Direct theft of ERC-20 tokens from any contract that (a) implements `IBuyCallback`, (b) returns `CALLBACK_SUCCESS` from `onBuy` without verifying it is the intended buyer in the specific trade, and (c) holds a token approval to Midnight. The victim never authorized the payment. This constitutes unauthorized movement of assets triggered by an unprivileged external caller.

## Likelihood Explanation
The attacker requires no privileged access — only the ability to call `take` as a taker. Any contract purpose-built to act as a buy callback in Midnight (flash-loan wrappers, routing contracts, aggregator adapters) will implement `IBuyCallback` and will commonly guard only `msg.sender == midnight`, making them susceptible. The attack is repeatable as long as the victim retains its approval and callback interface. The only precondition on the victim is that its `onBuy` does not verify the `buyer` argument matches itself or an address it has explicitly authorized for this specific trade.

## Recommendation
Add an authorization check requiring that `takerCallback` is either the taker themselves or has been explicitly authorized by the taker:

```solidity
require(
    takerCallback == address(0)
        || takerCallback == taker
        || isAuthorized[taker][takerCallback],
    TakerCallbackUnauthorized()
);
```

This mirrors the existing `TakerUnauthorized` pattern and ensures that only the taker or a contract the taker has pre-authorized can be designated as payer.

## Proof of Concept

**Minimal Foundry test outline:**

1. Deploy a `VictimCallback` contract that implements `IBuyCallback`, checks `msg.sender == address(midnight)`, and returns `CALLBACK_SUCCESS` unconditionally. Pre-approve Midnight for `buyerAssets` of `loanToken` and fund it.
2. Have a maker sign a sell offer (`offer.buy = false`) with any valid parameters.
3. As attacker (a fresh EOA with no tokens), call:
   ```solidity
   midnight.take(offer, ratifierData, units, attacker, receiver, address(victimCallback), "");
   ```
4. Assert that `victimCallback`'s `loanToken` balance decreased by `buyerAssets` and the attacker's position was credited — with the attacker having spent zero tokens.
