Audit Report

## Title
`sellerAssetsToUnits` reverts on zero `sellerPrice` while `Midnight.take()` succeeds, blocking bundler fills at the settlement-fee price boundary - (File: src/periphery/TakeAmountsLib.sol)

## Summary
When `offer.buy = true` and `tickToPrice(offer.tick) == settlementFee(id, ttm)`, `sellerPrice` computes to zero. `sellerAssetsToUnits` then calls `targetSellerAssets.mulDivUp(WAD, 0)`, which unconditionally reverts due to the `d - 1` underflow in `UtilsLib.mulDivUp`. Because `supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside its `try/catch` block, the entire bundler transaction reverts, while a direct `Midnight.take()` with the same inputs would succeed and return `sellerAssets = 0`.

## Finding Description

**Root cause — `mulDivUp` with denominator zero:**

`UtilsLib.mulDivUp` is implemented as: [1](#0-0) 

When `d = 0`, the sub-expression `d - 1` underflows under Solidity 0.8 checked arithmetic, reverting unconditionally. This behavior is explicitly confirmed by the existing test: [2](#0-1) 

**Code path in `sellerAssetsToUnits`:**

When `offer.buy = true` and `offerPrice == settlementFee`, `sellerPrice = offerPrice - settlementFee = 0`, and the subsequent `mulDivUp(WAD, sellerPrice)` call passes `0` as the denominator: [3](#0-2) 

**Contrast with `Midnight.take()`:**

In `Midnight.take()`, `sellerAssets` is computed as `units.mulDivDown(sellerPrice, WAD)`. The denominator is always `WAD` (never zero), so `sellerAssets = units * 0 / WAD = 0` — no revert: [4](#0-3) 

**Call site in `supplyCollateralAndSellWithAssetsTarget` — outside `try/catch`:**

`sellerAssetsToUnits` is invoked at line 286, before the `try` block at line 292. Its revert propagates unconditionally to the caller: [5](#0-4) 

**Wrong NatSpec comment:**

The comment at line 34 of `TakeAmountsLib.sol` states "Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)." At the boundary `offerPrice == settlementFee`, `Midnight.take()` does **not** revert — only the periphery helper does: [6](#0-5) 

## Impact Explanation

Any taker calling `supplyCollateralAndSellWithAssetsTarget` against a buy offer where `tickToPrice(offer.tick) == settlementFee(id, ttm)` receives an unconditional revert from the bundler. The fill is completely blocked through the periphery path. No funds are lost (the entire transaction rolls back), but the bundler is rendered non-functional for this specific price boundary, creating a persistent DoS of the `supplyCollateralAndSellWithAssetsTarget` path for affected offers.

## Likelihood Explanation

The condition requires exact equality between `tickToPrice(offer.tick)` and `settlementFee(id, ttm)`. Settlement fees are multiples of `CBP = 1e12`; tick prices are derived from `1e36 / (1e18 + wExp(...))`. Exact equality is possible but not guaranteed for arbitrary ticks. The most accessible trigger is at `ttm = 0` where the fee is the constant `settlementFeeCbp0 * CBP`. If the fee setter configures this value to match a specific tick price — even inadvertently — the condition is met for the entire post-maturity window and is repeatable for every call against that offer. Likelihood is low but non-zero and persistent once the alignment exists.

## Recommendation

Add a zero-check guard in `sellerAssetsToUnits` before calling `mulDivUp`. When `sellerPrice == 0`, the correct inverse is undefined (any number of units yields `sellerAssets = 0`), so the function should either return `type(uint256).max` (signaling "take as many units as possible") or revert with a descriptive error. The call site in `supplyCollateralAndSellWithAssetsTarget` should also be moved inside the `try/catch` block, or the result should be guarded, to match the documented intent of skipping offers that cannot be taken.

## Proof of Concept

1. Deploy `Midnight` and `MidnightBundles` in a test environment.
2. Set `settlementFeeCbp0` such that `settlementFeeCbp0 * CBP == tickToPrice(T)` for some tick `T`.
3. Create a buy offer at tick `T` with `offer.buy = true`.
4. Warp to or past maturity (`ttm = 0`) so `settlementFee(id, 0) == tickToPrice(T)`.
5. Call `supplyCollateralAndSellWithAssetsTarget` with `targetSellerAssets > 0` and the above offer.
6. Observe revert with `stdError.arithmeticError` from `mulDivUp(WAD, 0)`.
7. Call `IMidnight.take(offer, ..., units, ...)` directly with the same inputs — observe success with `sellerAssets = 0`.
