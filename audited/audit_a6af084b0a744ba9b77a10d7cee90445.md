### Title
Maker-Controlled `offer.callback` Enables Gas Griefing and Arbitrary State Changes at Taker's Expense — (File: src/Midnight.sol)

---

### Summary

In `take()`, the maker embeds `offer.callback` inside the signed `Offer` struct. This callback is invoked via an uncapped regular `CALL` (not `STATICCALL`) during every execution of `take()`. A malicious maker can deploy a callback that either exhausts all forwarded gas (causing the taker's transaction to revert) or performs arbitrary gas-intensive state-changing work at the taker's expense. The taker has no way to modify or bypass this callback without invalidating the maker's signature.

---

### Finding Description

**Root cause — maker controls an uncapped external call inside `take()`:**

In `src/Midnight.sol`, the `take()` function resolves the maker's callback as follows:

```solidity
address buyerCallback  = offer.buy ? offer.callback : takerCallback;
address sellerCallback = offer.buy ? takerCallback  : offer.callback;
``` [1](#0-0) 

Regardless of `offer.buy`, `offer.callback` is always the maker-controlled side. It is then invoked with no gas cap:

```solidity
if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback)
            .onBuy(id, offer.market, buyerAssets, units, buyerPendingFeeIncrease, buyer, buyerCallbackData)
        == CALLBACK_SUCCESS,
        WrongBuyCallbackReturnValue()
    );
}
``` [2](#0-1) 

```solidity
if (sellerCallback != address(0)) {
    require(
        ISellCallback(sellerCallback)
            .onSell(id, offer.market, sellerAssets, units, sellerPendingFeeDecrease, seller, receiver, sellerCallbackData)
        == CALLBACK_SUCCESS,
        WrongSellCallbackReturnValue()
    );
}
``` [3](#0-2) 

**Why the taker cannot escape it:**

Before any callback is reached, the ratifier validates the maker's signature over the *entire* `Offer` struct, including `offer.callback`:

```solidity
require(IRatifier(offer.ratifier).isRatified(offer, ratifierData) == CALLBACK_SUCCESS, RatifierFail());
``` [4](#0-3) 

The `Offer` struct explicitly includes `callback` and `callbackData` as maker-set fields:

```solidity
struct Offer {
    ...
    address callback;
    bytes callbackData;
    ...
}
``` [5](#0-4) 

Because the offer is signed by the maker over all fields, the taker cannot substitute a different callback address without breaking the signature check. The taker is forced to execute whatever contract the maker placed at `offer.callback`.

**Two concrete abuse paths:**

1. **Gas griefing**: The maker deploys a callback that runs an infinite loop or burns all forwarded gas (EIP-150 forwards 63/64 of remaining gas). The taker's `take()` call reverts, wasting the taker's gas. The maker pays nothing.

2. **Gas stealing**: The maker deploys a callback that performs useful, gas-intensive work — e.g., updating state in another protocol, executing a complex computation, or triggering a chain of external calls — all paid for by the taker's gas budget. Because this is a regular `CALL` (not `STATICCALL`), full state mutation is permitted.

**Secondary surface — ratifier (STATICCALL, gas griefing only):**

The ratifier is declared `external view`:

```solidity
interface IRatifier {
    function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32);
}
``` [6](#0-5) 

The compiler emits a `STATICCALL`, so state changes are blocked. Gas griefing (burning all forwarded gas before returning) remains possible, but the impact is limited to transaction reversion.

---

### Impact Explanation

- **Gas griefing**: Any taker attempting to fill a malicious maker's offer has their transaction reverted. The taker loses gas with no recourse. Repeated across many takers or bundled offers (e.g., via `MidnightBundles`), this constitutes a sustained DoS against takers.
- **Gas stealing**: The maker performs arbitrary state-changing work at the taker's expense. This is a direct financial transfer: the taker pays ETH (gas) for the maker's computation. The maker can use this to subsidize expensive on-chain operations (e.g., oracle updates, liquidity rebalancing) at zero cost to themselves.
- The `MidnightBundles` periphery wraps `take()` calls in `try/catch`, so griefed individual takes are silently skipped — but the gas is still consumed and the taker's `targetUnits` may not be filled, causing the bundle to revert at `require(filledUnits == targetUnits, OutOfOffers())`. [7](#0-6) 

---

### Likelihood Explanation

- **No privileged access required.** Any address can create an offer with a malicious callback.
- **Cheap to execute.** The attacker only needs to deploy a single malicious callback contract once and reference it in all their offers.
- **Automated takers are the primary victims.** MEV bots, aggregators, and the `MidnightBundles` periphery do not inspect callback bytecode before executing `take()`.
- **Takers have no on-chain defense.** There is no allowlist, no gas cap, and no way to strip the callback from a signed offer.

---

### Recommendation

1. **Cap gas forwarded to maker callbacks**: Forward a fixed, bounded amount of gas (e.g., `call{gas: 100_000}(...)`) to `offer.callback`. This prevents both gas griefing and gas stealing beyond the cap.
2. **Callback allowlist**: Maintain a registry of approved callback contracts. Reject offers whose `callback` is not on the list. This limits flexibility but eliminates the attack surface entirely.
3. **Documentation**: At minimum, prominently document that `offer.callback` is maker-controlled and executes with the taker's gas, so taker implementations can perform their own callback vetting.

### Proof of Concept

```solidity
// Malicious callback deployed by the maker
contract GasGriefCallback is IBuyCallback {
    function onBuy(
        bytes32, Market memory, uint256, uint256, uint256, address, bytes memory
    ) external returns (bytes32) {
        // Burn all forwarded gas — causes taker's take() to revert
        assembly { invalid() }
        // OR: perform useful work at taker's expense
        // expensiveProtocol.doWork();
    }
}

// Attack flow:
// 1. Maker deploys GasGriefCallback.
// 2. Maker creates Offer with offer.callback = address(GasGriefCallback).
// 3. Maker signs the offer (EcrecoverRatifier signs over the full Offer struct including callback).
// 4. Taker calls Midnight.take(offer, ratifierData, units, taker, ...).
// 5. Midnight validates the signature (passes — callback is part of the signed offer).
// 6. Midnight calls GasGriefCallback.onBuy(...) with no gas cap.
// 7. GasGriefCallback burns all gas → take() reverts → taker loses gas.
//    OR GasGriefCallback does useful work → taker subsidizes maker's computation.
```
