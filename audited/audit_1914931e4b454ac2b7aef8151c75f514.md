### Title
`_clampUserSums` Incorrectly Floors Post-Risk Deposits to `riskWindowStart`, Inflating k=2 Bonus Score for Mixed-Deposit Stakers — (File: src/ConfidencePool.sol)

---

### Summary

`_clampUserSums` uses a single aggregate inequality to detect whether a user's per-user sums need to be promoted to `riskWindowStart`. When a staker holds both pre-risk deposits (entry time < `riskWindowStart`) and post-risk deposits (entry time ≥ `riskWindowStart`), the aggregate check can still fire and incorrectly overwrites the entire `userSumStakeTime` / `userSumStakeTimeSq` with values that treat **all** stake as entering at `riskWindowStart` — including the post-risk portion that already recorded the correct, later entry time. The result is an inflated k=2 score for that staker, stealing bonus share from every other participant.

---

### Finding Description

`_clampUserSums` is designed to floor pre-risk deposit entry times to `riskWindowStart` so that pre-risk capital earns no time weight:

```solidity
// src/ConfidencePool.sol : 681-684
if (userSumStakeTime[u] < stake_ * start) {
    userSumStakeTime[u] = stake_ * start;
    userSumStakeTimeSq[u] = stake_ * start * start;
}
``` [1](#0-0) 

The condition `userSumStakeTime[u] < eligibleStake[u] * riskWindowStart` is correct for a pure pre-risk staker: all their entry times are below `riskWindowStart`, so their aggregate sum is necessarily below `eligibleStake * riskWindowStart`. The clamp promotes them to the floor.

However, staking during `UNDER_ATTACK` is explicitly permitted by design (DESIGN.md §3). When a staker has:

- Pre-risk stake **A** deposited at time **t₁ < riskWindowStart (t₂)**
- Post-risk stake **B** deposited at time **t₃ ≥ t₂** (during `UNDER_ATTACK`)

their aggregate state is:

```
eligibleStake        = A + B
userSumStakeTime     = A·t₁ + B·t₃
userSumStakeTimeSq   = A·t₁² + B·t₃²
```

The clamp fires whenever:

```
A·t₁ + B·t₃  <  (A + B)·t₂
⟺  A·(t₂ − t₁)  >  B·(t₃ − t₂)
```

This is satisfied whenever the pre-risk stake is large relative to the post-risk stake, or the pre-risk deposit was made long before `riskWindowStart`. When it fires, the function overwrites:

```
userSumStakeTime   ← (A + B)·t₂      [was: A·t₂ + B·t₃]
userSumStakeTimeSq ← (A + B)·t₂²     [was: A·t₂² + B·t₃²]
``` [2](#0-1) 

This treats B as if it entered at `t₂` instead of `t₃`. The user's k=2 score in `_bonusShare` becomes:

```
Incorrect: (A + B)·(T − t₂)²
Correct:    A·(T − t₂)² + B·(T − t₃)²
``` [3](#0-2) 

Since `t₃ > t₂`, `(T − t₂)² > (T − t₃)²`, so the incorrect score is strictly larger. The global snapshot (`snapshotSumStakeTime`, `snapshotSumStakeTimeSq`) was captured correctly at `flagOutcome` time and reflects B entering at `t₃`, so the denominator is correct while the numerator is inflated — the staker receives a larger fraction of `snapshotTotalBonus` than they are entitled to. [4](#0-3) 

The global accumulators are eagerly reset in `_markRiskWindowStart` to treat all pre-existing stakers as entering at `t₂`, and post-risk stakes are added at their actual wall-clock time: [5](#0-4) [6](#0-5) 

So the snapshot denominator correctly encodes `A·t₂ + B·t₃`, while the per-user numerator is incorrectly inflated to `(A+B)·t₂`. The mismatch is the source of the bonus theft.

---

### Impact Explanation

The attacker receives a k=2 bonus score of `(A+B)·(T−t₂)²` instead of the correct `A·(T−t₂)² + B·(T−t₃)²`. The excess score `B·[(T−t₂)² − (T−t₃)²]` is taken from the denominator that other stakers' shares are computed against, directly reducing every other staker's bonus payout. No principal is at risk and no tokens are created; the total bonus disbursed is unchanged, but the distribution is wrong. The magnitude scales with B and the square of the risk-window duration: a staker who adds a large post-risk deposit early in a long risk window can nearly double their effective score.

---

### Likelihood Explanation

The trigger condition `A·(t₂−t₁) > B·(t₃−t₂)` is satisfied in the common case where a staker deposited well before `riskWindowStart` and then tops up during `UNDER_ATTACK`. No privileged access is required. Any unprivileged staker who (a) staked before the risk window and (b) stakes again during `UNDER_ATTACK` — which is explicitly allowed — will silently receive an inflated bonus share. The condition is not a corner case; it is the normal operating pattern for a rational staker who wants to increase their position after observing the risk window open.

---

### Recommendation

The aggregate clamp cannot distinguish pre-risk from post-risk stake once both are mixed into `eligibleStake`. The fix is to track the pre-risk and post-risk contributions separately. One approach: when `_markRiskWindowStart` fires, snapshot `sumStakeTime` and `sumStakeTimeSq` as the "pre-risk baseline" for each user (or store a per-user `preRiskStake` counter). In `_clampUserSums`, only promote the pre-risk portion to `riskWindowStart` and leave the post-risk contribution untouched:

```solidity
// Pseudocode
uint256 preRisk = preRiskStake[u];          // stake held at riskWindowStart
uint256 postRiskTime   = userSumStakeTime[u]   - preRisk * t1_avg; // post-risk contribution
uint256 postRiskTimeSq = userSumStakeTimeSq[u] - preRisk * t1_sq_avg;
userSumStakeTime[u]   = preRisk * start + postRiskTime;
userSumStakeTimeSq[u] = preRisk * start * start + postRiskTimeSq;
```

A simpler alternative: record `eligibleStake[u]` at the moment `riskWindowStart` is sealed (a per-user `stakeAtRiskWindowStart` mapping), and use that value — not the current `eligibleStake[u]` — as the multiplier in the clamp, leaving the post-risk contribution in the sums untouched.

### Proof of Concept

**Setup:**
- `riskWindowStart` (t₂) = 2 000 s, `riskWindowEnd` / T = 3 000 s
- Alice stakes **A = 1 000 tokens** at t₁ = 1 000 s (pre-risk)
- Registry enters `UNDER_ATTACK`; `_markRiskWindowStart` sets `riskWindowStart = 2000`
- Alice stakes **B = 1 000 tokens** at t₃ = 2 999 s (post-risk, during `UNDER_ATTACK`)

**State after second stake:**
```
eligibleStake[Alice]      = 2 000
userSumStakeTime[Alice]   = 1000·1000 + 1000·2999 = 3 999 000
userSumStakeTimeSq[Alice] = 1000·1000² + 1000·2999² = 1 000 000 000 + 8 994 001 000
                          = 9 994 001 000
```

**Clamp check at claim time:**
```
userSumStakeTime[Alice] = 3 999 000
eligibleStake * start   = 2000 · 2000 = 4 000 000
3 999 000 < 4 000 000  → clamp fires
```

**After incorrect clamp:**
```
userSumStakeTime[Alice]   = 2000 · 2000 = 4 000 000
userSumStakeTimeSq[Alice] = 2000 · 2000² = 8 000 000 000
```

**Score comparison (T = 3 000):**
```
Incorrect score = 2000·(3000−2000)² = 2 000 · 1 000 000 = 2 000 000 000
Correct score   = 1000·(3000−2000)² + 1000·(3000−2999)²
               = 1 000 000 000 + 1 000 = 1 000 001 000
```

Alice's score is **~2× what it should be**, stealing bonus from every other staker proportionally. The global snapshot denominator correctly encodes `1000·2000 + 1000·2999 = 4 999 000` for `snapshotSumStakeTime`, so the mismatch between numerator and denominator is real and not self-cancelling.         uint256 start = riskWindowStart;
        uint256 stake_ = eligibleStake[u];

    function _bonusShare(address u, uint256 userEligible) internal view returns (uint256) {
        if (snapshotTotalBonus == 0) return 0;
        // No observable risk → no bonus (see contract natspec).
        if (riskWindowStart == 0) return 0;
        uint256 T = outcomeFlaggedAt;

        // Underflow guards on both subtractions: globally the sum of squares is nonneg, but
        // truncation/rounding pathologies could push individual terms over.
        uint256 userPlus = T * T * userEligible + userSumStakeTimeSq[u];
        uint256 userMinus = 2 * T * userSumStakeTime[u];
        uint256 userScore = userPlus > userMinus ? userPlus - userMinus : 0;

        uint256 plus = T * T * snapshotTotalStaked + snapshotSumStakeTimeSq;
        uint256 minus = 2 * T * snapshotSumStakeTime;
        uint256 globalScore = plus > minus ? plus - minus : 0;

        if (globalScore == 0) {
            // No time elapsed in the risk window for anyone → fallback to amount-weighted.
            if (snapshotTotalStaked == 0) return 0;
            return Math.mulDiv(userEligible, snapshotTotalBonus, snapshotTotalStaked);
        }
        // mulDiv handles the final multiply-then-divide via 512-bit intermediates, so a very
        // large `snapshotTotalBonus` cannot push the numerator over uint256 before division.
        return Math.mulDiv(userScore, snapshotTotalBonus, globalScore);
    }
```

    function _markRiskWindowStart() internal {
        // Cap at expiry: accrual is bounded by the pool's lifecycle. Without the cap, a late
        // observation could pin riskWindowStart > expiry, and `_clampUserSums` would record every
        // pre-risk deposit as entering past T = expiry (EXPIRED path). The k=2 sums then encode
        // post-deadline "at-risk" time that never actually existed.
        uint256 t = block.timestamp;
        if (t > expiry) t = expiry;
        // Cast is truncation-safe: `t` is capped at `expiry`, which is itself a uint32.
        // forge-lint: disable-next-line(unsafe-typecast)
        riskWindowStart = uint32(t);
        // Eagerly reset the global accumulators so every currently-eligible staker is treated as
        // entering at `t`. Per-user sums stay stale until `_clampUserSums` runs on the next op
        // touching that user.
        sumStakeTime = totalEligibleStake * t;
        sumStakeTimeSq = totalEligibleStake * t * t;
        emit RiskWindowStarted(t);
```
