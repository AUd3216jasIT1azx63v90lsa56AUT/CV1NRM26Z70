Audit Report

## Title
Borrower Self-Activates Reverting-Oracle Collateral to Permanently Block Own Liquidation - (File: src/Midnight.sol)

## Summary
`supplyCollateral` sets the collateral bitmap bit without querying the oracle, allowing a borrower to activate a collateral slot whose oracle is already reverting. `liquidate` iterates every activated collateral and calls `IOracle(...).price()` unconditionally with no error handling, so a single reverting oracle causes the entire liquidation call to revert. Because `withdrawCollateral` also calls `isHealthy` — which iterates all activated oracles when debt > 0 — the borrower cannot self-clear the poisoned slot while carrying debt, making the position permanently unliquidatable and leaving lenders exposed to unrecoverable bad debt.

## Finding Description

**Root cause 1 — `supplyCollateral` (lines 524–546):**
The function gates on `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` and sets the bitmap bit when `oldCollateral == 0 && assets > 0`, but never calls the oracle:

```solidity
if (oldCollateral == 0 && assets > 0) {
    uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
    _position.collateralBitmap = newCollateralBitmap;
    require(
        UtilsLib.countBits(newCollateralBitmap) <= MAX_COLLATERALS_PER_BORROWER, TooManyActivatedCollaterals()
    );
}
``` [1](#0-0) 

The authorization guard prevents a *third party* from poisoning a victim's bitmap, but it does not prevent the borrower from poisoning their *own* bitmap with a reverting-oracle collateral index.

**Root cause 2 — `liquidate` (lines 607–618):**
Every set bit is visited and `IOracle(_collateralParam.oracle).price()` is called unconditionally with no try/catch:

```solidity
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price();
    ...
    _collateralBitmap = _collateralBitmap.clearBit(i);
}
``` [2](#0-1) 

A single reverting oracle unwinds the entire call.

**Root cause 3 — `withdrawCollateral` / `isHealthy` (lines 568, 944–960):**
`withdrawCollateral` calls `isHealthy` unconditionally: [3](#0-2) 

`isHealthy` iterates all activated oracles when `debt > 0`:

```solidity
if (debt > 0) {
    uint128 _collateralBitmap = _position.collateralBitmap;
    while (_collateralBitmap != 0) {
        uint256 i = UtilsLib.msb(_collateralBitmap);
        CollateralParams memory collateralParam = market.collateralParams[i];
        uint256 price = IOracle(collateralParam.oracle).price();
        ...
    }
}
``` [4](#0-3) 

This closes the escape hatch: the borrower cannot self-remove the reverting slot while carrying debt.

**Protocol's own documentation confirms both halves:** [5](#0-4) [6](#0-5) 

**Exploit flow:**
1. Borrower identifies a market containing at least one collateral whose oracle is reverting or will revert.
2. Borrower calls `supplyCollateral` for up to 15 normal collateral indices and 1 reverting-oracle index (≥ 1 wei each). The bitmap reaches `MAX_COLLATERALS_PER_BORROWER` (16 bits set).
3. Borrower borrows against the healthy collaterals via `take`.
4. Position becomes unhealthy (price drop or borrow-to-limit).
5. Any liquidator calls `liquidate(...)`. The while loop reaches the reverting-oracle index; `IOracle(...).price()` reverts; the entire call reverts.
6. `withdrawCollateral` also reverts (via `isHealthy`) as long as debt > 0, so the borrower cannot self-clear the slot.
7. The position is permanently unliquidatable.

**Why `MAX_COLLATERALS_PER_BORROWER` does not help:**
The cap is reached exactly at 16 — it does not block the reverting slot, it merely prevents adding a 17th. [7](#0-6) 

## Impact Explanation
The unhealthy borrower's position becomes permanently unliquidatable. Lenders in the market cannot recover their principal; bad debt accumulates and is socialized across all lenders via the loss-factor mechanism. The borrower retains all non-reverting collateral and is never forced to repay. This directly violates the core protocol invariant that unhealthy positions must remain liquidatable, and constitutes a permanent freeze of lender funds — a critical in-scope impact.

## Likelihood Explanation
Preconditions are fully attacker-controlled and negligible in cost. Supplying 1 wei of the reverting-oracle collateral token is sufficient to set the bitmap bit. The oracle need not be malicious: any deprecated, paused, or temporarily broken oracle qualifies; the borrower activates it while it is reverting, which `supplyCollateral` permits. The attack is one-time and permanent: once the bitmap is full with the reverting slot, no external party can clear it. It is repeatable across every market that contains at least one such collateral parameter. No privileged access is required; any borrower in any eligible market can execute this.

## Recommendation
1. **In `supplyCollateral`**: Call `IOracle(market.collateralParams[collateralIndex].oracle).price()` when activating a new collateral slot (i.e., when `oldCollateral == 0 && assets > 0`). If the oracle reverts, revert the supply. This prevents activation of a reverting-oracle slot without breaking any existing healthy flow.
2. **In `liquidate` and `isHealthy`**: Wrap each `IOracle(...).price()` call in a try/catch. If an oracle reverts, treat the collateral price as 0 (conservative for health checks) or skip it with a flag, ensuring liquidation can always proceed for unhealthy positions.
3. **Defense-in-depth**: Consider allowing a permissionless `clearCollateralSlot` function that removes a zero-balance, reverting-oracle slot from the bitmap without requiring a health check, giving liquidators or third parties a recovery path.

## Proof of Concept
**Minimal Foundry test plan:**
1. Deploy `Midnight` and a mock oracle that reverts on `price()`.
2. Create a market with two collateral params: index 0 (healthy oracle) and index 1 (reverting oracle).
3. As `attacker`, call `supplyCollateral(market, 0, 1e18, attacker)` and `supplyCollateral(market, 1, 1, attacker)` — both succeed.
4. As `attacker`, call `take(...)` to borrow against index 0 collateral.
5. Manipulate the healthy oracle price downward so the position is unhealthy.
6. As `liquidator`, call `liquidate(market, 0, 0, repaidUnits, attacker, false, ...)` — assert it reverts due to the reverting oracle at index 1.
7. As `attacker`, call `withdrawCollateral(market, 1, 1, attacker, attacker)` — assert it also reverts via `isHealthy`.
8. Confirm the position is permanently unliquidatable and the debt is unrecoverable.
