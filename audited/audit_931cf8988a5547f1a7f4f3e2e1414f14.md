Audit Report

## Title
ECDSA Signature Malleability Enables Nonce-Griefing Front-Run on `setIsAuthorized` - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` invokes raw `ecrecover` at line 31 without enforcing a low-s constraint, allowing any observer to derive a second valid signature `(v^1, r, secp256k1_order − s)` for the same message and signer. An attacker who front-runs a victim's pending `setIsAuthorized` transaction with the malleable counterpart consumes the victim's nonce, causing the victim's original transaction to revert with `InvalidNonce`. If the victim submitted the call as part of an atomic multicall bundle (e.g., authorize-then-invoke a protective action), the entire bundle fails and must be re-submitted, introducing a timing gap exploitable in liquidation-sensitive contexts.

## Finding Description
**Exact code path — `EcrecoverAuthorizer.setIsAuthorized` (src/periphery/EcrecoverAuthorizer.sol, lines 24–48):**

- Line 26: `require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());` — nonce is checked and incremented atomically; the first caller to reach this line wins.
- Line 31: `address signer = ecrecover(digest, signature.v, signature.r, signature.s);` — raw `ecrecover` with no low-s guard.
- No `msg.sender` restriction anywhere in the function; any address may call it with any valid-looking signature.

**Root cause:** secp256k1 ECDSA produces two valid `(v, r, s)` pairs per `(message, key)`: the canonical `(v, r, s)` and the malleable `(v^1, r, n − s)` where `n` is the curve order. Both pass `ecrecover` and return the identical signer address. Without `require(uint256(s) <= 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0)`, both are accepted.

**Exploit flow:**
1. Victim signs `Authorization{authorizer=V, authorized=X, isAuthorized=true, nonce=N, deadline=D}` and broadcasts `setIsAuthorized(auth, sig)`.
2. Attacker observes the pending transaction, extracts `(v, r, s)`, computes `s' = n − s`, `v' = v ^ 1`.
3. Attacker submits `setIsAuthorized(auth, Signature{v', r, s'})` with higher gas.
4. Attacker's transaction executes first: `nonce[V] == N` passes, nonce becomes `N+1`, `ecrecover` returns `V`, authorization is set to the victim's intended value.
5. Victim's transaction executes: `nonce[V] == N+1` but `auth.nonce == N` → `InvalidNonce()` revert.

**Why existing checks fail:**
- `require(signer != address(0))` — both signatures recover the same non-zero address; passes.
- `require(signer == authorization.authorizer || isAuthorized(...))` — same signer recovered; passes.
- The nonce is already consumed before the victim's transaction executes.

**Important nuance on "repeated" griefing:** After the first front-run, the authorization state is already set to the victim's intended value (the attacker's transaction executed the same `setIsAuthorized` call on Midnight). The victim does not need to re-call `setIsAuthorized` for the same authorization. The attacker cannot repeat the attack indefinitely for the same authorization intent. However, if the victim submitted the call inside an atomic multicall bundle (authorize + protective action in one transaction), the entire bundle reverts and the victim must resubmit — the protective action is delayed by at least one block.

## Impact Explanation
The concrete impact is a one-time forced revert of any atomic bundle that includes `setIsAuthorized` as its first step. In a liquidation-protection scenario where a user bundles `EcrecoverAuthorizer.setIsAuthorized` + a `MidnightBundles` protective call (repay debt, withdraw collateral) into a single multicall transaction, the attacker's front-run causes the entire bundle to fail. The victim must detect the failure, resubmit (now without the `setIsAuthorized` step, since authorization is already set), and race against the liquidation deadline. Even a single-block delay can be sufficient for a liquidator to execute. The attacker pays only gas; the victim bears the liquidation loss.

## Likelihood Explanation
**Preconditions:** Victim broadcasts a `setIsAuthorized` transaction (routine operation). Attacker monitors the public mempool (standard, zero-cost capability on Ethereum mainnet and most L2s with public mempools). No special role, funds, or credentials required beyond gas.

**Feasibility:** Computing `s' = n − s` and `v' = v ^ 1` is trivial off-chain arithmetic, executable in milliseconds. The attack window is the standard mempool propagation window. The attacker pays gas but requires no upfront capital.

## Recommendation
Add a low-s constraint immediately after computing the digest, before calling `ecrecover`:

```solidity
require(
    uint256(signature.s) <= 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0,
    InvalidSignature()
);
```

This is the same guard used by OpenZeppelin's `ECDSA.recover`. Alternatively, use OpenZeppelin's `ECDSA` library directly, which enforces this constraint internally. This eliminates the malleable counterpart and ensures each `(message, key)` pair maps to exactly one accepted signature.
