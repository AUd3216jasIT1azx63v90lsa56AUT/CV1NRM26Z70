Audit Report

## Title
Unchecked `callback` payer designation in `repay()` enables theft of loanToken from any `IRepayCallback`-implementing contract - (`src/Midnight.sol`)

## Summary
`repay()` designates the `callback` parameter as the token payer without any authorization check verifying that `callback` has consented to being used as a payer by `msg.sender`. An attacker who is authorized to repay on behalf of a borrower (e.g., a seller's callback contract authorized via `isAuthorized`) can call `repay(..., callback=victim, ...)` to pull loanToken from any victim contract that implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS` and holds a Midnight approval. The seller's debt is retired at the victim's expense.

## Finding Description

**Root cause — `repay()` lines 502–521 (`src/Midnight.sol`):**

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
// ...
address payer = callback != address(0) ? callback : msg.sender;   // payer = victim
// ...
IRepayCallback(callback).onRepay(id, market, units, onBehalf, data);  // called on victim
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // pulls from victim
```

The sole authorization gate checks that `msg.sender` may act on behalf of `onBehalf`. There is no check that `callback` (the payer) has authorized `msg.sender` to designate it as a payer. Any address passed as `callback` becomes the payer.

**Exploit flow:**

1. Attacker deploys `maliciousContract` and creates a sell offer with `offer.callback = maliciousContract`.
2. Attacker calls `setIsAuthorized(maliciousContract, true, seller)` — standard practice for callback contracts that need to act on the seller's behalf.
3. Taker calls `take(offer, ...)`.
4. Inside `take()`, the liquidation lock is set for the seller (line 444).
5. After buyer-side token pulls (lines 455–456), `onSell` is invoked on `maliciousContract` (lines 458–473).
6. Inside `onSell`, `maliciousContract` calls `repay(market, units, seller, callback=victim, data)`:
   - `msg.sender = maliciousContract`; `isAuthorized[seller][maliciousContract] = true` → authorization check passes.
   - `payer = victim`.
   - `IRepayCallback(victim).onRepay(...)` is called; if victim returns `CALLBACK_SUCCESS`, execution continues.
   - `safeTransferFrom(loanToken, victim, midnight, units)` pulls `units` loanToken from victim.
7. Seller's debt is reduced. The post-callback health check at line 476 passes because the debt was repaid.

**Why existing checks fail:**

- The `repay` authorization check (`isAuthorized[onBehalf][msg.sender]`) only verifies the right to repay on behalf of the seller; it says nothing about who may be designated as payer.
- The liquidation lock prevents the seller from being liquidated during the callback window but does not restrict what the callback may do to third parties.
- The Certora spec `OnlyExplicitPayerCanLoseTokens.spec` sets `repayCallbackAllowed = false` for `take` (line 108) and models `onSell` with `HAVOC_ALL` (it is not listed in the `methods` block, unlike `onBuy`, `onRepay`, `onLiquidate`, `onFlashLoan`). The comment at line 13–14 states "onSell cannot authorize a payer," but this is incorrect — `onSell` can re-enter `repay` with an arbitrary `callback`. The `HAVOC_ALL` abstraction for `onSell` prevents the prover from tracking the re-entrant `repay` call and the resulting `badPullSeen` flag, so the formal tool does not detect the violation.

## Impact Explanation

Victim loses loanToken equal to `units` (the repaid amount). The seller's debt is retired at the victim's expense with no cost to the attacker beyond gas. This is a direct, unauthorized transfer of assets from a third party, violating the invariant that only explicit payers can lose tokens. The impact is **High**: direct theft of funds from any integrated contract that implements `IRepayCallback`.

## Likelihood Explanation

**Required preconditions:**
1. Seller authorizes their callback contract — standard practice for any callback that calls `supplyCollateral` or similar on the seller's behalf.
2. Victim contract implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS` without validating `onBehalf` — realistic for any Midnight-integrated aggregator, vault, or router that uses repay callbacks for its own operations and does not defensively check the `onBehalf` argument.
3. Victim has approved Midnight to spend loanToken and holds a loanToken balance — true for any active repay-callback contract.

The attack is repeatable as long as the victim contract remains deployed and approved. The attacker can craft the offer and callback contract entirely off-chain before deployment. **Likelihood: Medium** (requires a victim contract with the above properties, which is realistic for any protocol built on top of Midnight).

## Recommendation

Add an authorization check in `repay()` requiring that `callback` has authorized `msg.sender` to use it as a payer, analogous to the `onBehalf` check:

```solidity
require(
    callback == address(0) || callback == msg.sender || isAuthorized[callback][msg.sender],
    Unauthorized()
);
```

Alternatively, restrict the payer to always be `msg.sender` and require the callback to pull funds itself and transfer them to Midnight, removing the ability to designate an arbitrary third-party payer entirely.

## Proof of Concept

**Minimal Foundry test outline:**

1. Deploy a `MockVictim` contract that:
   - Implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS` unconditionally.
   - Holds loanToken and has approved Midnight for `type(uint256).max`.

2. Deploy `MaliciousCallback` that:
   - Implements `ISellCallback.onSell`.
   - Inside `onSell`, calls `midnight.repay(market, units, seller, address(mockVictim), "")`.

3. Setup:
   - Seller calls `setIsAuthorized(address(maliciousCallback), true, seller)`.
   - Seller creates a sell offer with `offer.callback = address(maliciousCallback)`.
   - Seller has debt in the market.

4. Taker calls `take(offer, ...)`.

5. Assert:
   - `mockVictim`'s loanToken balance decreased by `units`.
   - Seller's debt decreased by `units`.
   - No tokens were pulled from the seller or taker for the repayment.
