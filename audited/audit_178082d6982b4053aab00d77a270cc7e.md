### Title
Buy offer `maxAssets` cap bypassed via zero-rounding when `buyerPrice < WAD` — (`src/Midnight.sol` / `src/libraries/TickLib.sol`)

### Summary
When a buy offer (`offer.buy = true`) sets `offer.maxAssets > 0` and the tick yields a `buyerPrice < WAD`, calling `take` with `units = 1` produces `buyerAssets = units.mulDivDown(buyerPrice, WAD) = 0`. Because the consumed counter increments by zero, the guard `newConsumed <= offer.maxAssets` never trips, and the offer can be filled an unlimited number of times — each time mutating lender credit, borrower debt, and `totalUnits` — while paying zero assets and never advancing the consumption counter.

### Finding Description
**Code path** — `src/Midnight.sol` `take()`:

```
// line 363
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // ← floors to 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

// lines 367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // ← += 0, always passes
}
``` [1](#0-0) 

`tickToPrice` is proven to return `≤ WAD` for every valid tick. [2](#0-1) 

For any tick where `tickToPrice(tick) < WAD` — which is every tick except the very top of the range — passing `units = 1` makes `1 * buyerPrice < WAD`, so `mulDivDown` floors to zero. The consumed mapping is incremented by zero, `newConsumed` stays at its prior value, and the cap check passes unconditionally regardless of how many times the offer has already been filled.

**Attacker inputs**: `offer.buy = true`, `offer.maxAssets = X > 0`, any tick where `tickToPrice < WAD` (e.g. `MAX_TICK - 16`), `units = 1` per call.

**State after each fill**: `consumed` unchanged; `lender.credit += 1`; `borrower.debt += 1`; `totalUnits += 1`; zero tokens transferred. The offer is structurally "fully consumed" yet accepts unlimited further fills.

**Why existing checks fail**: The `require(newConsumed <= offer.maxAssets)` check is the only guard on the assets-mode cap. It relies entirely on `buyerAssets > 0` to make progress. No floor-of-one or minimum-units check exists. The sell-offer path is safe because it uses `mulDivUp`, which returns at least 1 for any `units > 0` and `sellerPrice > 0`. [3](#0-2) 

The protocol's own NatSpec acknowledges the consequence: [4](#0-3) 

### Impact Explanation
The `maxAssets` cap — the sole mechanism preventing a buy offer from being overfilled in assets mode — is completely ineffective for any tick where `tickToPrice < WAD`. An offer intended to consume at most `X` assets of buyer credit can be replayed indefinitely: the maker accumulates unbounded credit units while paying zero loan tokens, and the taker accumulates unbounded debt while receiving zero loan tokens. This directly violates the core invariant that "offers cannot be replayed, overfilled, reused, or filled after cancel/deadline."

### Likelihood Explanation
`tickToPrice` returns a value strictly less than WAD for every tick below the maximum. The condition `units = 1` is always caller-controlled. No special oracle state, token behavior, or admin action is required. The existing test `testBugBuyMaxAssetsBypass` in the repository reproduces the exact scenario with `tick = MAX_TICK - 16` and `maxAssets = 1`, confirming the path is reachable with standard setup. [5](#0-4) 

### Recommendation
Add a minimum-assets floor in the `maxAssets` branch: require that the computed assets value is at least 1 when `units > 0`, or require `units` to be large enough that `buyerAssets > 0` before updating consumed. Concretely, before line 368, add:

```solidity
if (offer.maxAssets > 0) {
    uint256 delta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || delta > 0, ZeroAssetFill()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += delta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` such that `units.mulDivDown(buyerPrice, WAD) >= 1` before accepting the fill in assets mode.

### Proof of Concept
The repository already contains a passing Foundry test that proves the bug:

```solidity
// test/TakeTest.sol — testBugBuyMaxAssetsBypass (lines 858-889)
function testBugBuyMaxAssetsBypass() public {
    deal(address(loanToken), lender, 0);       // lender pays 0
    collateralize(market, borrower, 100);

    lenderOffer.maxUnits  = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick      = MAX_TICK - 16;     // offerPrice < WAD

    // Pre-consume to the cap.
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

    (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

    assertEq(buyerAssets, 0);                  // zero assets transferred
    assertEq(sellerAssets, 0);
    assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets); // cap unchanged
    assertGt(midnight.creditOf(id, lender), 0);   // but credit grew
    assertGt(midnight.debtOf(id, borrower), 0);   // and debt grew
}
``` [6](#0-5) 

A fuzz extension would loop `take(1, borrower, lenderOffer)` N times and assert `midnight.creditOf(id, lender) == N` while `midnight.consumed(lender, lenderOffer.group) == lenderOffer.maxAssets` throughout, confirming unbounded replay.
