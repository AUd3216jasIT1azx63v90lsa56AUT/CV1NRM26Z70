### Title
Unvalidated Oracle Address in Permissionless Market Creation Allows Attacker to Steal Lender Funds — (File: `src/Midnight.sol`)

---

### Summary

`touchMarket()` in `Midnight.sol` validates LLTV tiers and `maxLif` values for each collateral entry but performs **no validation whatsoever on the `oracle` address** inside `CollateralParams`. Because market creation is fully permissionless, an attacker can deploy a malicious oracle contract, create a market referencing it, attract lenders with favorable offers, and then flip the oracle's behavior to drain collateral and leave lenders with unrecoverable bad debt.

---

### Finding Description

**Vulnerability class:** Unvalidated attacker-controlled external contract used for critical accounting (pricing / health checks).

**Root cause — `touchMarket()` skips oracle validation:** [1](#0-0) 

The loop inside `touchMarket()` checks:
- `collateralToken > previousCollateralToken` (sort order)
- `isLltvAllowed(lltv)` (whitelist)
- `maxLif == maxLif(lltv, cursor)` (formula check)

It never calls or inspects `market.collateralParams[i].oracle`. Any address — including a contract the attacker fully controls — is silently accepted.

**Oracle is used for all health and liquidation accounting:**

`isHealthy()` calls `IOracle(collateralParam.oracle).price()` for every activated collateral: [2](#0-1) 

`liquidate()` does the same: [3](#0-2) 

`isHealthy()` is the sole guard in `withdrawCollateral()`: [4](#0-3) 

and the final safety check at the end of `take()`: [5](#0-4) 

**Why the oracle address cannot be changed post-creation but the oracle contract can:**

The market id is a `keccak256` over the full `Market` struct (including every oracle address): [6](#0-5) 

Changing the oracle address would produce a different `marketId` and a different market. However, the attacker does not need to change the address — they only need to change the **behavior** of the contract at that address (e.g., via an owner-controlled flag). The address stays the same; the price returned flips.

---

### Impact Explanation

**Concrete theft path:**

1. Attacker deploys `MaliciousOracle` with a hidden owner flag. Phase 1: returns a realistic price (e.g., `1e36` for 1:1). Phase 2: returns `type(uint256).max` or reverts.
2. Attacker calls `touchMarket()` with a `Market` struct that uses `MaliciousOracle` for collateral C. Market is created; `tickSpacing` is set.
3. Attacker posts attractive sell offers (high implied yield). Victims take these offers, paying loan tokens and receiving credit units. Victims' funds now sit in the contract.
4. Attacker supplies a small amount of collateral C and borrows (sells debt). Health check at line 476 passes because `MaliciousOracle` still returns a legitimate price.
5. **Attacker flips the oracle flag** → `price()` now returns `type(uint256).max`.
6. Attacker calls `withdrawCollateral()`. `isHealthy()` computes `maxDebt = collateral * (type(uint256).max / 1e36) * lltv / 1e18` which overflows or is astronomically large → `maxDebt >= debt` → health check passes. Attacker withdraws all collateral.
7. Attacker flips oracle to revert (or keeps returning `type(uint256).max` so `originalDebt > maxDebt` is false).
8. Liquidators call `liquidate()` → oracle reverts → entire call reverts (documented in LIVENESS). Position cannot be liquidated.
9. Bad debt is socialized. Lenders' credit is slashed proportionally. Victims lose their principal.

**Severity:** High — direct theft of lender principal with no recovery path once collateral is withdrawn and oracle is bricked.

---

### Likelihood Explanation

- **No privileged access required.** Any EOA can deploy a contract and call `touchMarket()`.
- **Realistic social-engineering surface.** In DeFi, users routinely interact with markets based on APY, UI listings, or community links. A market with a malicious oracle is indistinguishable on-chain from a legitimate one until the attack fires.
- **Attack timing is fully attacker-controlled.** The attacker waits until TVL is maximized before flipping the flag, maximizing profit.
- **No existing on-chain defense.** There is no oracle whitelist, no oracle interface validation, and no admin veto on market creation.

---

### Recommendation

Maintain a protocol-level whitelist of approved oracle addresses, enforced inside `touchMarket()`:

```solidity
mapping(address oracle => bool) public isOracleAllowed;

// In touchMarket(), inside the collateral loop:
require(isOracleAllowed[market.collateralParams[i].oracle], OracleNotAllowed());
```

Only the `roleSetter` (or a dedicated `oracleSetter`) should be able to add oracles to the whitelist. This mirrors the LLTV whitelist pattern already present in `isLltvAllowed()`. [7](#0-6) 

Alternatively, require oracles to be deployed by a factory contract that enforces immutability and a known implementation, preventing post-deployment behavior changes.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import {IOracle} from "src/interfaces/IOracle.sol";

/// Attacker-controlled oracle with a hidden flip flag.
contract MaliciousOracle is IOracle {
    address public owner;
    uint8 public phase; // 0 = normal, 1 = max, 2 = revert

    constructor() { owner = msg.sender; }

    function setPhase(uint8 p) external { require(msg.sender == owner); phase = p; }

    function price() external view returns (uint256) {
        if (phase == 0) return 1e36;          // legitimate 1:1 price
        if (phase == 1) return type(uint256).max; // inflated → withdraw collateral
        revert("oracle bricked");              // phase 2 → liquidation impossible
    }
}
```

**Steps:**

1. Deploy `MaliciousOracle` (phase 0).
2. Call `midnight.touchMarket(market)` where `market.collateralParams[0].oracle = address(maliciousOracle)`, valid LLTV, valid `maxLif`. Market is created.
3. Victim calls `midnight.take(offer, ...)` — lends 10,000 USDC, receives 10,000 credit units.
4. Attacker calls `midnight.supplyCollateral(market, 0, 1 ether, attacker)` — supplies 1 ETH (worth 1e36 * 1 / 1e36 = 1 loan token at phase-0 price; adjust amounts so health holds).
5. Attacker calls `midnight.take(victimOffer, ...)` — borrows USDC. Health check at line 476 passes (phase 0 price).
6. `maliciousOracle.setPhase(1)` — price becomes `type(uint256).max`.
7. Attacker calls `midnight.withdrawCollateral(market, 0, 1 ether, attacker, attacker)`. `isHealthy()` returns `true` (overflow makes `maxDebt` huge). Collateral withdrawn.
8. `maliciousOracle.setPhase(2)` — oracle reverts.
9. Any call to `liquidate()` reverts at line 610 (`IOracle(...).price()` reverts).
10. Victim's 10,000 USDC credit becomes bad debt. Loss is socialized. Victim's `credit` is slashed to near zero on next `_updatePosition`. [8](#0-7) [9](#0-8) [10](#0-9)
