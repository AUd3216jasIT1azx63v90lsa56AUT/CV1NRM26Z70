### Title
Zero-amount `safeTransferFrom` on fee transfer reverts for tokens that reject zero-value transfers in zero-fee markets - (File: src/Midnight.sol)

### Summary
In `take()`, the settlement fee spread `buyerAssets - sellerAssets` is transferred to the contract via `safeTransferFrom` unconditionally. When the market's settlement fee is 0 (the default state for all newly created markets), `buyerAssets == sellerAssets` for both buy and sell offers, making this a zero-amount transfer. Loan tokens that revert on zero-amount `transferFrom` calls will cause every `take()` invocation in such a market to revert permanently.

### Finding Description
**Code path:**

1. `settlementFee(id, timeToMaturity)` (`src/Midnight.sol:963–980`) returns `0` when all seven `settlementFeeCbp` breakpoints are 0 — the default uninitialized storage value for any freshly created market.
2. In `take()` (lines 360–364), with `_settlementFee = 0`: `buyerPrice == sellerPrice == offerPrice`, so `buyerAssets == sellerAssets` regardless of `offer.buy` direction (both use the same price in both `mulDivDown`/`mulDivUp` branches).
3. Line 418: `claimableSettlementFee[loanToken] += 0` — no revert here.
4. Line 455: `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), 0)` — called unconditionally with value `0`.
5. `SafeTransferLib.safeTransferFrom` (lines 24–34) issues a raw `token.call` with `transferFrom(from, to, 0)`. If the token reverts, the assembly block propagates the revert, causing `take()` to revert.

**No existing check stops this:** there is no `if (buyerAssets > sellerAssets)` guard, no minimum-fee enforcement, and `safeTransferFrom` has no zero-value short-circuit.

**Attacker-controlled inputs:** The market creator selects the `loanToken`. A token that reverts on zero-amount `transferFrom` (a known pattern in several real ERC-20 implementations) combined with the default zero-fee state is sufficient. No privileged action is required after market creation.

### Impact Explanation
Every call to `take()` in any market whose loan token reverts on zero-amount `transferFrom` and whose settlement fee is 0 will revert. Since the default fee is 0, this affects all such markets from creation. The second transfer on line 456 (`sellerAssets`, which is non-zero for non-zero `units`) is unaffected; only the fee-collection transfer on line 455 is the failure point. The result is a permanent, complete DoS of `take()` for the affected market.

### Likelihood Explanation
The precondition (settlement fee = 0) is the **default** state for every new market — no special setup is needed. The only additional requirement is a loan token that reverts on zero-amount transfers. This is a documented behavior of several deployed ERC-20 tokens (e.g., tokens following strict EIP-20 interpretations or custom guard logic). A market creator — an unprivileged role — selects the loan token, making this trivially reachable. The DoS is repeatable on every `take()` call and cannot be self-healed by users.

### Recommendation
Guard the fee transfer with a zero-amount check:

```solidity
if (buyerAssets > sellerAssets) {
    SafeTransferLib.safeTransferFrom(
        offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets
    );
}
```

Apply the same guard to `sellerAssets` on line 456 for consistency (covers the `units = 0` edge case).

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));

                revert(add(returndata, 0x20), mload(returndata))
