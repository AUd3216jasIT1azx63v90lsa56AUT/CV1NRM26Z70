### Title
Lenders Can Front-Run Bad Debt Socialization to Avoid Losses - (File: src/Midnight.sol)

### Summary

Morpho Midnight socializes bad debt by lazily updating a global `lossFactor` in `marketState` during `liquidate()`. Because the slash is only applied to a lender's credit at their next `_updatePosition` call, a lender who calls `withdraw()` before a bad debt liquidation executes avoids the loss entirely. This concentrates the full loss on remaining lenders and also amplifies their proportional loss because the withdrawing lender reduces `totalUnits` before the bad debt denominator is computed.

### Finding Description

**Root cause — lazy slashing via `lossFactor`:**

When bad debt is realized in `liquidate()`, the global `lossFactor` is updated atomically: [1](#0-0) 

The slash is **not** applied to any individual lender at this point. Each lender's credit is only reduced when `_updatePosition` is called for them, which computes: [2](#0-1) 

The slash ratio is `(MAX - currentLossFactor) / (MAX - lastLossFactor)`. A lender whose `lastLossFactor` equals `currentLossFactor` (i.e., they interacted after the bad debt event) suffers no slash.

**`withdraw()` calls `_updatePosition` before the slash:** [3](#0-2) 

`_updatePosition` at line 485 applies the **current** `lossFactor`. If the lender front-runs the liquidation, `lossFactor` has not yet been updated, so `_updatePosition` applies zero slash. The lender then exits cleanly.

**Amplified loss for remaining lenders:**

`withdraw()` also decrements `totalUnits` (line 495). The bad debt loss factor computation uses `totalUnits` as the denominator: [4](#0-3) 

If a lender withdraws `W` units before bad debt `B` is realized from a pool of `T` total units:
- Without front-run: each remaining lender loses `B / T` of their credit.
- With front-run: `totalUnits = T - W`, so each remaining lender loses `B / (T - W)` of their credit — a strictly larger fraction.

The protocol's own developer comment acknowledges a race-to-withdraw dynamic: [5](#0-4) 

This confirms the withdrawal race is a known design consideration, but the bad debt front-running variant is a distinct, exploitable attack that causes direct financial harm to remaining lenders.

**Exploit flow:**

1. Attacker is a lender (credit holder) in a market where a borrower's position is heading toward bad debt.
2. Attacker monitors the mempool for a `liquidate()` call targeting that borrower.
3. Attacker submits `withdraw(units)` with higher gas, where `units <= withdrawable` (available from prior repayments or liquidations in the same market).
4. Attacker's `withdraw` executes first: `_updatePosition` applies the old `lossFactor` (no slash), credit is reduced, `totalUnits` decreases.
5. `liquidate()` executes: `lossFactor` increases based on the now-smaller `totalUnits`, amplifying the loss for remaining lenders.
6. Attacker has exited with full value; remaining lenders bear a disproportionately larger loss.

### Impact Explanation

A malicious lender can systematically avoid bad debt losses by front-running liquidations. The remaining lenders bear both the full bad debt loss and an amplified proportional loss due to the reduced `totalUnits` denominator. In markets with significant bad debt events, this can result in material, unfair redistribution of losses from the attacker to honest lenders. The impact is direct financial harm — theft of loss-avoidance value at the expense of other users.

### Likelihood Explanation

The precondition is `withdrawable > 0` at the time of front-running. This is realistic whenever:
- Any borrower in the same market has previously repaid debt (line 509: `marketState[id].withdrawable += units`), or
- A prior partial liquidation has added to `withdrawable` (line 675).

Front-running is straightforward on any EVM chain with a public mempool. The attacker needs only to be a lender in the market — no privileged access is required. In active markets with multiple borrowers, `withdrawable > 0` is the common case, making this attack highly feasible.

### Recommendation

The core issue is that `withdraw()` allows a lender to exit before `lossFactor` is updated, using the stale (pre-bad-debt) loss state. Possible mitigations:

1. **Snapshot-based slashing:** Before processing a withdrawal, check whether any pending bad debt exists (e.g., by checking if the borrower is liquidatable) and apply a prospective slash. This is complex to implement correctly.
2. **Withdrawal delay / lock:** Introduce a short withdrawal delay or lock withdrawals when a liquidatable position exists in the market. This is the most robust fix but changes UX.
3. **Pro-rata loss on exit:** When a lender withdraws, compute their share of any unrealized bad debt (from positions already liquidatable) and deduct it from the withdrawal amount.

### Proof of Concept

**Setup:**
- Market with `loanToken`, one collateral, `lltv = 0.8e18`.
- Alice (lender): supplies 1000 units of credit via `take`.
- Bob (borrower): borrows 800 units, supplies collateral worth 1000 at oracle price.
- Carol (repayer): repays 200 units → `withdrawable = 200`.
- Oracle drops: Bob's collateral is now worth 700, making him liquidatable with bad debt.

**Attack:**
1. Liquidator submits `liquidate(borrower=Bob, repaidUnits=0, seizedAssets=0)` to realize bad debt (or a full liquidation with bad debt).
2. Alice front-runs with `withdraw(200)` (using the 200 `withdrawable` from Carol's repayment).
3. Alice's `_updatePosition` applies old `lossFactor` (no slash). Alice receives 200 loan tokens.
4. Liquidation executes: `totalUnits` is now `800` (not `1000`), so `lossFactor` increases by `badDebt / 800` instead of `badDebt / 1000`.
5. Alice's remaining 800 credit is slashed at her next interaction, but she already extracted 200 units at full value.
6. Remaining lenders (if any) bear the amplified loss.

**Expected output (analogous to the Panoptic PoC):**

```
totalUnits before liquidation:  1000
withdrawable before liquidation: 200
Alice withdraws before liquidation: 200 (full value, no slash)
totalUnits after Alice's withdrawal: 800
badDebt realized against 800 totalUnits instead of 1000
Remaining lenders' loss factor: badDebt/800 > badDebt/1000
Alice avoided loss on 200 units; remaining lenders bear amplified loss
```
