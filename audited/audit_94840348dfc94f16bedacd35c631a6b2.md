### Title
Fee-on-Transfer Collateral Token Inflates Position Accounting, Enabling Undercollateralized Borrowing - (File: src/Midnight.sol)

### Summary
`supplyCollateral` credits `position[id][onBehalf].collateral[collateralIndex]` with the caller-supplied `assets` value before performing the ERC20 transfer, with no balance-before/balance-after reconciliation. If the collateral token charges a transfer fee, Midnight receives fewer tokens than the amount recorded in the position. Because `isHealthy` and the liquidation health check both read the stored (inflated) collateral value rather than the actual token balance, the borrower can borrow against phantom collateral, creating bad debt.

### Finding Description
**Code path:**

`supplyCollateral` (src/Midnight.sol lines 524–546):

```
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets); // line 533 — credits full `assets`
...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545 — receives assets*(1-fee)
``` [1](#0-0) 

The state write (line 533) happens unconditionally before the transfer (line 545), and there is no `balanceBefore`/`balanceAfter` guard anywhere in the function. `SafeTransferLib.safeTransferFrom` only checks that the call does not revert and that a non-false return value is returned; it does not verify the net amount received.

**Market creation is permissionless.** `touchMarket` validates lltv tiers, maxLif values, and sorted collateral token addresses, but imposes no restriction on the ERC20 token type. [2](#0-1) 

Any unprivileged actor can deploy a fee-on-transfer ERC20, register it as a collateral token in a new market, and immediately call `supplyCollateral`.

**Health check uses stored values.** `isHealthy` and the liquidation eligibility check in `liquidate` both iterate over `_position.collateral[i]` (the stored, inflated value) to compute `maxDebt`. [3](#0-2) 

The oracle-weighted collateral sum is computed from the inflated stored value, so the position appears healthier than it actually is.

**Exploit flow:**
1. Attacker deploys `FeeToken` with a 2% transfer fee and a matching oracle.
2. Attacker calls `touchMarket` with `FeeToken` as the sole collateral token — succeeds because no token-type check exists.
3. Attacker calls `supplyCollateral(market, 0, 1e18, borrower)`.
   - Line 533 sets `position[id][borrower].collateral[0] = 1e18`.
   - Line 545 transfers `1e18` from the attacker; Midnight receives `0.98e18`.
4. Attacker (as borrower) takes a lender's buy offer. `isHealthy` computes `maxDebt` using `1e18` collateral, so the borrower can borrow up to `lltv * price * 1e18 / WAD` units — 2% more than the actual collateral backing supports.
5. If the oracle price drops to the true collateral value, the position is undercollateralized but `isHealthy` may still return true, blocking liquidation.
6. On liquidation, `safeTransfer` of `seizedAssets` collateral tokens also loses 2% to the fee, so the liquidator receives less than `seizedAssets`, and the accounting diverges further.

**Existing protections are insufficient.** There is no whitelist of approved collateral tokens, no balance-delta check, and no fee-on-transfer detection anywhere in the protocol.

### Impact Explanation
A borrower using a fee-on-transfer collateral token has their position credited with more collateral than Midnight actually holds. This directly violates the core invariant that "contract balances cover collateral." The inflated collateral value allows the borrower to take on more debt than the real collateral can cover, creating undercollateralized positions. When prices move, these positions may not be liquidatable in time (or at all at full value), resulting in bad debt socialized to lenders.

### Likelihood Explanation
Market creation is fully permissionless — any address can register a fee-on-transfer token as collateral. The attacker needs only to deploy a compliant ERC20 with a fee mechanism and fund the initial supply. The attack is repeatable across any number of markets and positions. No privileged access, oracle manipulation, or user error is required.

### Recommendation
Add a balance-before/balance-after check in `supplyCollateral` and credit only the net received amount:

```solidity
uint256 balanceBefore = ERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = ERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Alternatively, maintain a per-market, per-token tracked balance and reject any call where `received < assets`. The same pattern should be applied to `repay` and `flashLoan` repayment paths for the loan token.

import "forge-std/Test.sol";
import {Midnight, Market, CollateralParams} from "src/Midnight.sol";
import {ERC20} from "solmate/tokens/ERC20.sol";

contract FeeToken is ERC20("Fee", "FEE", 18) {
    uint256 constant FEE_BPS = 200; // 2%
    function mint(address to, uint256 amt) external { _mint(to, amt); }
    function transfer(address to, uint256 amt) public override returns (bool) {
        uint256 fee = amt * FEE_BPS / 10_000;
        return super.transfer(to, amt - fee);
    }
    function transferFrom(address from, address to, uint256 amt) public override returns (bool) {
        uint256 fee = amt * FEE_BPS / 10_000;
        return super.transferFrom(from, to, amt - fee);
    }
}

contract FeeOnTransferCollateralTest is Test {
    Midnight midnight;
    FeeToken feeToken;
    // ... setup oracle, loanToken, etc.

    function testFeeOnTransferInflatesCollateral() public {
        // 1. Deploy fee token and create market
        feeToken = new FeeToken();
        // ... build market with feeToken as collateral, valid lltv/maxLif/oracle

        // 2. Supply 1e18 collateral
        feeToken.mint(address(this), 1e18);
        feeToken.approve(address(midnight), 1e18);
        midnight.supplyCollateral(market, 0, 1e18, borrower);

        // 3. Assert accounting mismatch
        assertEq(midnight.collateral(id, borrower, 0), 1e18);           // stored: 1e18
        assertEq(feeToken.balanceOf(address(midnight)), 0.98e18);        // actual: 0.98e18

        // 4. Borrow against inflated collateral — should succeed
        // take a lender buy offer for units = lltv * price * 1e18 / WAD
        // (would revert if accounting used actual 0.98e18)

        // 5. Assert isHealthy returns true despite real undercollateralization
        assertTrue(midnight.isHealthy(market, id, borrower));
        // Manual check: real collateral value = 0.98e18 * price / ORACLE_PRICE_SCALE * lltv / WAD < debt
    }
}

**Expected assertions:**
- `position.collateral[0] == 1e18` (inflated)
- `feeToken.balanceOf(midnight) == 0.98e18` (actual)
- Borrower can take debt up to `lltv * price * 1e18 / WAD`, which exceeds what `0.98e18` real collateral supports
- `isHealthy` returns `true` for a position that is actually undercollateralized by 2%
