### Title
Authorized Operator Can Freeze All Maker Offers via `setIsRootRatified(false)` - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier.setIsRootRatified` uses the Midnight core `isAuthorized` mapping as its sole authorization gate, with no scope restriction. Any address the maker has authorized in Midnight â€” for any purpose â€” can call `setIsRootRatified(maker, root, false)`, atomically freezing every offer under that Merkle root. Because the authorization is coarse-grained and flat, a maker who grants an operator access for collateral management or repayment implicitly grants them the power to destroy all active offer ratification state.

### Finding Description
**Code path:**

`SetterRatifier.setIsRootRatified` (src/ratifiers/SetterRatifier.sol, line 24â€“28):
```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
```

The check `IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender)` delegates entirely to the flat Midnight authorization mapping (src/Midnight.sol, line 732â€“734):
```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
```

`isAuthorized` is a single boolean per `(maker, operator)` pair â€” there is no scope, no action whitelist, and no distinction between "authorized to manage collateral" and "authorized to mutate ratification state."

**Exploit flow:**
1. Maker calls `midnight.setIsAuthorized(operator, true, maker)` â€” legitimately, e.g., to let the operator manage collateral or repay on their behalf.
2. Operator (malicious or compromised) calls `setterRatifier.setIsRootRatified(maker, root, false)`.
3. The `require` passes: `isAuthorized[maker][operator] == true`.
4. `isRootRatified[maker][root]` is set to `false`.
5. Every subsequent `take` call for any offer under `root` reaches `isRatified`, which calls `require(isRootRatified[offer.maker][root], NotRatified())` (src/ratifiers/SetterRatifier.sol, line 35) and reverts.

**Why existing checks fail:**
There is no additional check. The `setIsRootRatified` function accepts `false` as a valid value for `newIsRootRatified` and applies it unconditionally once the flat authorization check passes. The existing test `testIsRatifiedAuthorizedSetterCanRatifyOnBehalf` (test/SetterRatifierTest.sol, line 48â€“61) only covers the `true` direction; no test guards the `false` direction against an operator acting outside the maker's intent.

### Impact Explanation
All offers whose Merkle root is `root` become permanently un-takeable until the maker re-calls `setIsRootRatified(maker, root, true)`. If the operator is malicious and also revokes their own authorization (or the maker's key is unavailable), recovery may be delayed or impossible. Lenders with active buy offers lose all fills; borrowers with active sell offers lose access to credit. This matches the scoped impact: all offers in the Merkle tree are frozen without maker's intent.

### Likelihood Explanation
**Preconditions:**
- Maker has called `midnight.setIsAuthorized(operator, true, maker)` for any reason (collateral management, repayment delegation, multicall batching, etc.).
- Operator is malicious or compromised.

**Feasibility:** The Midnight authorization model is explicitly coarse-grained (confirmed by Certora spec `onlyAuthorizedCanChangeCollateralExceptLiquidate`, `onlyAuthorizedCanChangeConsumedExceptTake`, etc. â€” all use the same flat `isAuthorized` check). Granting an operator for one purpose is a normal, documented workflow. The `SetterRatifier` silently inherits this scope. The attack is a single external call with no capital requirement, no time constraint, and is repeatable for every root the maker has ever ratified.

### Recommendation
Separate the `SetterRatifier` authorization from the Midnight core `isAuthorized` mapping. Two concrete options:

1. **Restrict `false` to maker-only:** Allow any authorized operator to ratify (`true`) but require `maker == msg.sender` to de-ratify (`false`), since de-ratification is a destructive, irreversible-until-corrected action.
2. **Dedicated ratifier authorization:** Maintain a separate `isRatifierAuthorized[maker][operator]` mapping inside `SetterRatifier`, requiring the maker to explicitly grant ratifier-management rights independently of Midnight core authorization.

Option 1 is the minimal fix and aligns with the principle of least privilege without breaking the `testIsRatifiedAuthorizedSetterCanRatifyOnBehalf` use case.

### Proof of Concept
```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {SetterRatifier} from "../src/ratifiers/SetterRatifier.sol";
import {ISetterRatifier} from "../src/ratifiers/interfaces/ISetterRatifier.sol";
import {BaseTest} from "./BaseTest.sol";

contract SetterRatifierFreezeTest is BaseTest {
    SetterRatifier internal setterRatifier;

    function setUp() public override {
        super.setUp();
        setterRatifier = new SetterRatifier(address(midnight));
    }

    /// @notice Fuzz: any authorized operator can freeze maker's root
    function testFuzz_AuthorizedOperatorCanFreezeRoot(address operator) public {
        vm.assume(operator != lender);
        vm.assume(operator != address(0));

        bytes32 root = keccak256("root");

        // Maker ratifies root
        vm.prank(lender);
        setterRatifier.setIsRootRatified(lender, root, true);
        assertTrue(setterRatifier.isRootRatified(lender, root));

        // Maker authorizes operator for unrelated purpose (e.g., collateral mgmt)
        vm.prank(lender);
        midnight.setIsAuthorized(operator, true, lender);

        // Malicious/compromised operator freezes the root
        vm.prank(operator);
        setterRatifier.setIsRootRatified(lender, root, false); // must revert â€” currently does NOT

        // Assert: root is now frozen â€” all takes under root will revert NotRatified
        assertFalse(
            setterRatifier.isRootRatified(lender, root),
            "operator froze maker root without maker intent"
        );
    }
}
```
