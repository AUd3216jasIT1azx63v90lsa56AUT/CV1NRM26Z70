Audit Report

## Title
Fee-on-Transfer `loanToken` Overcredits `claimableSettlementFee` Relative to Actual Balance - (File: `src/Midnight.sol`)

## Summary
In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` at line 418, before the actual `safeTransferFrom` at line 455. When `loanToken` is a fee-on-transfer token, only `(buyerAssets - sellerAssets) * (1 - fee%)` tokens arrive at the contract, creating a permanent and compounding gap between the accounting entry and the real balance. Because `touchMarket` imposes no restriction on `loanToken`, any unprivileged actor can trigger this condition.

## Finding Description
**Exact code path:**

`src/Midnight.sol` line 418 writes the accounting entry unconditionally: [1](#0-0) 

`src/Midnight.sol` lines 455–456 perform the actual transfers afterward: [2](#0-1) 

For a fee-on-transfer token with fee rate `f`, the contract receives `(buyerAssets - sellerAssets) * (1 - f)` but `claimableSettlementFee` is credited the full `buyerAssets - sellerAssets`. The deficit per `take()` call is `(buyerAssets - sellerAssets) * f`, and it compounds across every subsequent `take()` in the market's lifetime.

**Why existing checks fail:**

`touchMarket()` (lines 755–791) validates maturity, collateral params, LLTV tiers, and collateral sort order, but performs zero validation on `loanToken` — any address is accepted: [3](#0-2) 

There is no token whitelist, no `transferFrom`-and-verify pattern, and no balance-before/after check anywhere in the `take()` flow.

**Formal verification scope:**

The `tokenBalanceCorrect` strong invariant and `pendingFeeReceiptZero` weak invariant in `certora/specs/Solvency.spec` are both proven under the assumption that `safeTransferFrom` delivers exactly `amount` tokens: [4](#0-3) 

The `pendingFeeReceipt` ghost tracks the `claimableSettlementFee` increment and expects it to be zeroed out by the corresponding inbound transfer — an assumption that is structurally false for fee-on-transfer tokens. The formal proofs provide no protection here.

**`claimSettlementFee` transfer path:** [5](#0-4) 

When `feeClaimer` calls `claimSettlementFee(feeToken, claimableSettlementFee[feeToken], receiver)`, the `safeTransfer` at line 309 attempts to send the full overcredited amount.

## Impact Explanation
Two concrete outcomes:

1. **Fee claimer DoS / loss of protocol revenue:** If the contract's total `feeToken` balance is less than `claimableSettlementFee[feeToken]`, the `safeTransfer` at line 309 reverts. The fee claimer cannot withdraw the full recorded amount; accumulated protocol fees are permanently unclaimable for that token.

2. **Solvency violation / theft from other users:** If other `feeToken` balance exists in the contract (e.g., from lenders' deposits or borrowers' collateral denominated in the same token across any market), the transfer succeeds by consuming tokens that belong to other users, directly breaking the core invariant `balance >= collateral + withdrawable + claimableSettlementFee`.

Both outcomes constitute protocol insolvency for the affected token. The second outcome is direct theft of user funds.

## Likelihood Explanation
- **Preconditions:** Deploy any fee-on-transfer ERC20 (trivial, no privileged access required) and call `touchMarket(market)` with `market.loanToken = address(feeToken)` — fully permissionless.
- **Trigger:** Any taker interacting with an offer in that market calls `take()`. No victim mistake, no oracle manipulation, no governance action required.
- **Repeatability:** Every `take()` call widens the deficit by `spread * fee_rate`. The effect is deterministic and compounds indefinitely.
- **Attacker profile:** Unprivileged external user. No leaked keys, no privileged roles.

Neither `SECURITY.md` nor `RESEARCHER.md` excludes fee-on-transfer tokens from scope: [6](#0-5) 

## Recommendation
Apply one or more of the following mitigations:

1. **Balance-before/after check in `take()`:** Record `balanceBefore = IERC20(loanToken).balanceOf(address(this))` before the transfer at line 455, then assert `balanceAfter - balanceBefore == buyerAssets - sellerAssets`. Revert if the received amount is less than expected. Only credit `claimableSettlementFee` by the actual received amount.

2. **Move the accounting write after the transfer:** Restructure `take()` so that `claimableSettlementFee` is incremented only after the transfer completes, using the actual received delta rather than the nominal amount.

3. **Token whitelist on `touchMarket()`:** Restrict `loanToken` to a protocol-approved list of standard ERC20 tokens, explicitly excluding fee-on-transfer and rebasing tokens.

Option 1 or 2 is preferred as it is the most surgical fix and preserves the permissionless market creation model.

## Proof of Concept
**Minimal Foundry test plan:**

1. Deploy a mock ERC20 with a 1% transfer fee (deducted from the received amount on every `transferFrom`).
2. Call `touchMarket(market)` with `market.loanToken = address(feeToken)` — no privileged role needed.
3. Create a buy offer and call `take()` with `buyerAssets - sellerAssets = 1000e18`.
4. Assert: `claimableSettlementFee[feeToken] == 1000e18` (overcredited).
5. Assert: `feeToken.balanceOf(address(midnight)) == 990e18` (actual received, 1% fee deducted).
6. Call `claimSettlementFee(feeToken, 1000e18, receiver)` — assert it reverts due to insufficient balance, or if other `feeToken` balance exists, assert it drains from collateral/withdrawable buckets.

The deficit of `10e18` per `take()` call is deterministic and reproducible with any fee-on-transfer ERC20.
