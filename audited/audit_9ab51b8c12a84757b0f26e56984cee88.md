### Title
Missing deadline in `stake()` allows stale transaction to execute in `UNDER_ATTACK` state, permanently locking staker principal - (File: src/ConfidencePool.sol)

### Summary
`stake()` accepts no deadline or maximum-registry-state guard. A transaction submitted while the registry is in a pre-attack state (e.g. `ATTACK_REQUESTED`, where withdrawal is still permitted) can sit in the mempool, be included only after the registry has transitioned to `UNDER_ATTACK`, and permanently lock the staker's principal with no ability to exit.

### Finding Description
`stake()` calls `_observePoolState()` at execution time and uses `block.timestamp` as the deposit entry time. There is no caller-supplied deadline that would cause the transaction to revert if it is included too late.

The critical path:

1. The registry is in `ATTACK_REQUESTED` — withdrawals are still open.
2. A staker submits `stake(amount)`, intending to participate while retaining the right to exit.
3. The transaction is delayed in the mempool.
4. The registry transitions to `UNDER_ATTACK`.
5. The transaction is included. `_observePoolState()` observes `UNDER_ATTACK` and calls `_markRiskWindowStart()`, setting `riskWindowStart = block.timestamp`.
6. `_assertDepositsAllowed()` does **not** block `UNDER_ATTACK` — the stake is accepted.
7. Inside `stake()`, `newEntry = block.timestamp = riskWindowStart`, so the deposit is recorded at the floor of the risk window.
8. From this point, `withdraw()` permanently reverts with `WithdrawsDisabled` because `riskWindowStart != 0`.

The staker is now irrevocably committed to the resolution path — SURVIVED, EXPIRED, or CORRUPTED — with no exit, despite having submitted the transaction in a state where exit was still possible. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
The staker loses the withdrawal option they expected to retain. Their principal is locked until resolution — potentially months away — with no recourse. Under a CORRUPTED (bad-faith) outcome, the entire pool including their principal is swept to `recoveryAddress`, resulting in total loss of principal. Even under SURVIVED/EXPIRED, the staker bears the full duration lock they did not consent to. The DESIGN.md §3 states the mitigation is "A staker who does not want this can read the live registry state before staking," but this advice is ineffective against mempool delays: the staker did read the state before submitting, and it was safe. [4](#0-3) 

### Likelihood Explanation
BattleChain is an EVM-compatible L2 where block proposers are known in advance (analogous to PoS Ethereum). A validator or MEV searcher can observe a pending `stake()` transaction and a concurrent registry transition to `UNDER_ATTACK`, then sequence the stake inclusion after the transition. Even without active manipulation, organic mempool congestion during a high-activity period (e.g., the moment an attack is publicly announced and the registry is updated) can produce the same outcome. The window is narrow but the consequence is permanent.

### Recommendation
Add a `deadline` parameter to `stake()` and revert if `block.timestamp > deadline`:

```solidity
function stake(uint256 amount, uint256 deadline) external nonReentrant whenPoolNotPaused {
    if (block.timestamp > deadline) revert DeadlineExpired();
    // ... rest of function
}
```

Optionally, also accept a `maxState` parameter so the staker can specify the highest registry state they are willing to stake into, reverting if `_observePoolState()` returns a state beyond it. Never use `block.timestamp` as the deadline value.

### Proof of Concept

```
State: registry = ATTACK_REQUESTED, riskWindowStart = 0

1. Alice calls stake(1000e18) — registry is ATTACK_REQUESTED, withdraw() is open.
2. Transaction enters mempool. Registry transitions to UNDER_ATTACK.
3. Transaction is included in a later block.
   - _observePoolState() → UNDER_ATTACK → _markRiskWindowStart() sets riskWindowStart = block.timestamp
   - _assertDepositsAllowed(UNDER_ATTACK) → passes (UNDER_ATTACK is not blocked)
   - newEntry = block.timestamp = riskWindowStart
   - eligibleStake[Alice] = 1000e18
4. Alice calls withdraw() → riskWindowStart != 0 → WithdrawsDisabled revert.
5. Alice's 1000e18 is locked until resolution.
   - If CORRUPTED (bad-faith): swept to recoveryAddress. Alice loses 1000e18.
   - If SURVIVED/EXPIRED: returned, but Alice bore months of illiquidity she did not consent to.
``` [5](#0-4) [6](#0-5) [7](#0-6)

    function stake(uint256 amount) external nonReentrant whenPoolNotPaused {
        if (amount == 0) revert InvalidAmount();
        if (amount < minStake) revert BelowMinStake();
        if (outcome != PoolStates.Outcome.UNRESOLVED) revert OutcomeAlreadySet();
        if (block.timestamp >= expiry) revert StakingClosed();
        _assertDepositsAllowed(_observePoolState());

        if (!expiryLocked) {
            expiryLocked = true;
        }

        // Balance-diff defense-in-depth against the factory allowlist admitting a fee-on-transfer
        // or rebasing token (governance error, or a proxy-token upgrade post-allowlist).
        // aderyn-fp-next-line(reentrancy-state-change)
        uint256 balanceBefore = stakeToken.balanceOf(address(this));
        stakeToken.safeTransferFrom(msg.sender, address(this), amount);
        // aderyn-fp-next-line(reentrancy-state-change)
        uint256 balanceAfter = stakeToken.balanceOf(address(this));
        uint256 received = balanceAfter > balanceBefore ? balanceAfter - balanceBefore : 0;
        if (received == 0) revert NoTokensReceived();
        if (received < minStake) revert BelowMinStake();

        _clampUserSums(msg.sender);

        // Pre-risk deposits use wall clock and get promoted to riskWindowStart later via
        // `_clampUserSums`; post-risk deposits go in at wall clock (already ≥ riskWindowStart).
        uint256 newEntry = block.timestamp;
        uint256 start = riskWindowStart;
        if (start != 0 && newEntry < start) newEntry = start;

        uint256 contribTime = received * newEntry;
        uint256 contribTimeSq = received * newEntry * newEntry;

        eligibleStake[msg.sender] += received;
        userSumStakeTime[msg.sender] += contribTime;
        userSumStakeTimeSq[msg.sender] += contribTimeSq;
        totalEligibleStake += received;
        sumStakeTime += contribTime;
        sumStakeTimeSq += contribTimeSq;

        emit Staked(msg.sender, received);
    }

    function _observePoolState() internal returns (IAttackRegistry.ContractState state) {
        state = _getAgreementState();
        if (
            !scopeLocked && state != IAttackRegistry.ContractState.NOT_DEPLOYED
                && state != IAttackRegistry.ContractState.NEW_DEPLOYMENT
        ) {
            scopeLocked = true;
            emit ScopeLocked(block.timestamp);
        }
        if (riskWindowStart == 0 && _isActiveRiskState(state)) {
            _markRiskWindowStart();
        }
        if (riskWindowEnd == 0 && _isTerminalState(state)) {

    function _markRiskWindowStart() internal {
        // Cap at expiry: accrual is bounded by the pool's lifecycle. Without the cap, a late
        // observation could pin riskWindowStart > expiry, and `_clampUserSums` would record every
        // pre-risk deposit as entering past T = expiry (EXPIRED path). The k=2 sums then encode
        // post-deadline "at-risk" time that never actually existed.
        uint256 t = block.timestamp;
        if (t > expiry) t = expiry;
        // Cast is truncation-safe: `t` is capped at `expiry`, which is itself a uint32.
        // forge-lint: disable-next-line(unsafe-typecast)
        riskWindowStart = uint32(t);
        // Eagerly reset the global accumulators so every currently-eligible staker is treated as
        // entering at `t`. Per-user sums stay stale until `_clampUserSums` runs on the next op
        // touching that user.
        sumStakeTime = totalEligibleStake * t;
        sumStakeTimeSq = totalEligibleStake * t * t;
        emit RiskWindowStarted(t);
```
