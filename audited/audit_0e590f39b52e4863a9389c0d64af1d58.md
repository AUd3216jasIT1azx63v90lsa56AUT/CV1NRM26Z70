### Title
Unchecked overflow in `mulDivDown` when `offer.maxAssets = type(uint256).max` and `offer.buy = false` causes `consumableUnits` to always revert, DoSing sell-side bundle flows - (File: src/periphery/ConsumableUnitsLib.sol)

### Summary
`UtilsLib.mulDivDown` is implemented in plain Solidity 0.8+ checked arithmetic as `return (x * y) / d`, so the intermediate product `x * y` reverts on overflow. When a maker crafts a sell offer with `maxAssets = type(uint256).max` and `maxUnits = 0`, `consumableUnits` passes `type(uint256).max` as `targetSellerAssets` into `sellerAssetsToUnits`, which calls `mulDivDown(type(uint256).max, WAD, sellerPrice)`. The product `type(uint256).max * 1e18` overflows uint256 and reverts. Because `consumableUnits` is called outside the `try/catch` block in every bundle loop, the entire bundle transaction reverts.

### Finding Description

**Code path:**

`ConsumableUnitsLib.consumableUnits` (line 21): [1](#0-0) 

With `offer.maxUnits = 0` and `offer.buy = false`, execution reaches:
```
TakeAmountsLib.sellerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed))
```
`zeroFloorSub(type(uint256).max, 0)` returns `type(uint256).max` (assembly, no overflow). [2](#0-1) 

`sellerAssetsToUnits` (line 46) for `offer.buy = false`:
```
targetSellerAssets.mulDivDown(WAD, sellerPrice)
``` [3](#0-2) 

`mulDivDown` is plain checked Solidity 0.8+:
```solidity
return (x * y) / d;   // x = type(uint256).max, y = WAD = 1e18 → overflow → revert
``` [4](#0-3) 

**Why existing checks do not stop it:**

- `tickToPrice` is bounded to `<= WAD` (proven in Certora spec), so `sellerPrice` is always a small positive value — it does not prevent the overflow in the numerator. [5](#0-4) 

- The `try/catch` in the bundle loop wraps only the `take()` call. `consumableUnits` is called on line 77, **before** the `try` on line 79. The NatSpec explicitly documents this: *"Reverts if ConsumableUnitsLib reverts."* [6](#0-5) 

- There is no cap or validation on `offer.maxAssets` anywhere in `consumableUnits`, `sellerAssetsToUnits`, or the bundle entry points.

**Attacker inputs:** `offer.maxUnits = 0`, `offer.buy = false`, `offer.maxAssets = type(uint256).max`, any valid `offer.tick` (0–5820).

### Impact Explanation
Any bundle function that iterates over a `takes[]` array containing the attacker's malicious sell offer will revert unconditionally when `consumableUnits` is reached for that offer. This DoSes `buyWithUnitsTargetAndWithdrawCollateral`, `buyWithAssetsTargetAndWithdrawCollateral`, and `supplyCollateralAndSellWithAssetsTarget` for any user whose off-chain routing or aggregator includes the poisoned offer. The core `Midnight.take()` is unaffected, but the periphery bundle flows — the primary UX path for takers — are fully blocked for any transaction that touches the malicious offer. [7](#0-6) 

### Likelihood Explanation
**Preconditions:** Attacker must be able to post a signed sell offer with `maxAssets = type(uint256).max` and `maxUnits = 0`. There is no on-chain validation preventing this. The offer just needs to appear in the order book or be passed as calldata by any routing layer that auto-selects offers. **Cost:** zero (no funds required, only a signed message). **Repeatability:** permanent — the offer remains poisonous until the maker cancels it or the deadline passes. Any taker or aggregator that includes it in a bundle is DoSed.

### Recommendation
Add an overflow guard before calling `mulDivDown` in `sellerAssetsToUnits`. The simplest safe fix is to cap `targetSellerAssets` before the multiplication:

```solidity
// In sellerAssetsToUnits, before the return:
// Safe: if targetSellerAssets > type(uint256).max / WAD, the result would exceed
// any realistic unit count; return type(uint256).max as a sentinel.
if (targetSellerAssets > type(uint256).max / WAD) return type(uint256).max;
```

Alternatively, implement `mulDivDown` using full 512-bit intermediate arithmetic (e.g., via `mulmod`/assembly) so it never reverts on overflow, or add a `maxAssets` cap in the `Offer` validation layer. The same fix should be applied symmetrically to `buyerAssetsToUnits` for the `mulDivUp` path. [8](#0-7)

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {ConsumableUnitsLib} from "src/periphery/ConsumableUnitsLib.sol";
import {TakeAmountsLib} from "src/periphery/TakeAmountsLib.sol";
import {Offer} from "src/interfaces/IMidnight.sol";

contract ConsumableUnitsOverflowTest is Test {
    // Mock midnight: consumed() returns 0, settlementFee() returns a small value.
    MockMidnight midnight;
    bytes32 id;

    function setUp() public {
        midnight = new MockMidnight();
        id = bytes32(0);
    }

    function testConsumableUnitsOverflowSellSide() public {
        Offer memory offer;
        offer.maxUnits  = 0;
        offer.buy       = false;
        offer.maxAssets = type(uint256).max;  // attacker-controlled
        offer.tick      = 0;                  // any valid tick

        // Expected: arithmetic overflow revert from mulDivDown(type(uint256).max, 1e18, sellerPrice)
        vm.expectRevert(stdError.arithmeticError);
        ConsumableUnitsLib.consumableUnits(address(midnight), id, offer);
    }

    function testFuzz_consumableUnitsOverflow(uint256 maxAssets) public {
        // Any maxAssets > type(uint256).max / WAD triggers overflow
        maxAssets = bound(maxAssets, type(uint256).max / 1e18 + 1, type(uint256).max);

        Offer memory offer;
        offer.maxUnits  = 0;
        offer.buy       = false;
        offer.maxAssets = maxAssets;
        offer.tick      = 0;

        vm.expectRevert(stdError.arithmeticError);
        ConsumableUnitsLib.consumableUnits(address(midnight), id, offer);
    }
}
```

**Expected assertions:**
- `vm.expectRevert(stdError.arithmeticError)` passes — confirming the revert is an arithmetic overflow, not a logic revert.
- Fuzz test confirms the overflow threshold is `maxAssets > type(uint256).max / WAD` (~1.157e59), well below `type(uint256).max`. [4](#0-3) [9](#0-8) [3](#0-2)
