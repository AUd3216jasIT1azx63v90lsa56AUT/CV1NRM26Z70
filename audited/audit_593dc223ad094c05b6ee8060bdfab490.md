Audit Report

## Title
Missing `caller` Parameter in `onRepay` Callback Enables Unauthorized Token Drain from Third-Party Payer Contracts - (`File: src/Midnight.sol`)

## Summary
`Midnight.repay` sets `payer = callback` when a non-zero callback is supplied, then invokes `IRepayCallback(callback).onRepay(...)` before pulling tokens from `payer` via `safeTransferFrom`. Unlike `onLiquidate` and `onFlashLoan`, the `onRepay` interface omits the `caller` address, structurally preventing callback contracts from verifying who initiated the repayment. An attacker who holds authorization over a borrower can designate any third-party contract as `callback`, causing that contract's tokens to be pulled without its consent.

## Finding Description

**Exact code path:**

`src/Midnight.sol` lines 502–521:
```solidity
function repay(Market memory market, uint256 units, address onBehalf, address callback, bytes calldata data)
    external
{
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    // ...
    address payer = callback != address(0) ? callback : msg.sender;   // line 511
    // ...
    if (callback != address(0)) {
        require(
            IRepayCallback(callback).onRepay(id, market, units, onBehalf, data) == CALLBACK_SUCCESS,
            WrongRepayCallbackReturnValue()
        );
    }
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
}
```

`src/interfaces/ICallbacks.sol` line 21:
```solidity
function onRepay(bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
```

**Root cause — missing `caller` in `onRepay`:**

All three callback interfaces that designate the callback contract as payer are compared below:

| Callback | Passes `caller`? |
|---|---|
| `onFlashLoan(address caller, ...)` | ✓ |
| `onLiquidate(address caller, ...)` | ✓ |
| `onRepay(bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data)` | ✗ |

`onLiquidate` passes `msg.sender` as the first argument at `src/Midnight.sol` lines 700–702. `onFlashLoan` does the same at line 746. `onRepay` at line 516 passes no equivalent. A callback contract receiving `onRepay` can only confirm `msg.sender == Midnight`; it cannot determine who called `repay`, so it cannot distinguish a self-initiated repayment from one injected by an attacker.

**Exploit flow:**
1. Attacker obtains `isAuthorized[victim][attacker] = true` (e.g., victim authorized a position manager contract controlled by attacker, or attacker is a malicious bundler the victim approved).
2. `thirdParty` is a legitimate repayment-helper contract: it has approved Midnight to spend its loan tokens and its `onRepay` returns `CALLBACK_SUCCESS` (it cannot check `caller` — the parameter does not exist).
3. Attacker calls `repay(market, units, onBehalf=victim, callback=thirdParty, data)`.
4. Authorization check at line 505 passes — attacker is authorized by victim.
5. `payer = thirdParty` (line 511).
6. `thirdParty.onRepay(...)` is called — returns `CALLBACK_SUCCESS`; it has no way to reject the call based on initiator identity.
7. `safeTransferFrom(loanToken, thirdParty, address(this), units)` executes (line 520), draining `thirdParty`.
8. Victim's debt is reduced; `thirdParty` bears the full cost without consent.

**Why existing checks fail:**
- `isAuthorized[onBehalf][msg.sender]` only validates the caller's right to act for `onBehalf`; it says nothing about the right to designate an arbitrary third party as payer.
- There is no `require(callback == msg.sender)` or `require(isAuthorized[callback][msg.sender])` guard.
- The `onRepay` interface structurally prevents the callback from self-protecting via the caller address, unlike `onLiquidate` and `onFlashLoan`.

## Impact Explanation
`thirdParty`'s loan tokens are transferred to Midnight to repay `victim`'s debt. `thirdParty` receives nothing in return and did not consent to the payment. This constitutes direct, unauthorized theft of assets from a protocol participant. The invariant that callbacks and approvals bind the right user/action/amount is violated: the callback is invoked against a party that never agreed to be the payer, and the missing `caller` parameter is the structural enabler.

## Likelihood Explanation
Three preconditions are required:
1. **Attacker is authorized by a borrower** — realistic for any user who has granted broad authorization to a position manager, bundler, or aggregator (a common DeFi pattern).
2. **`thirdParty` has approved Midnight** — any contract that participates in repayments as a helper will hold such an allowance.
3. **`thirdParty.onRepay` returns `CALLBACK_SUCCESS` without verifying the initiator** — structurally forced by the missing `caller` parameter; the only alternative is out-of-band state (e.g., transient storage tracking self-initiated calls), which most existing helper contracts will not implement.

The attack requires no oracle manipulation, no admin access, no token misbehavior, and is repeatable as long as the allowance and debt remain.

## Recommendation
Add `caller` (i.e., `msg.sender` from `repay`) as the first parameter of `onRepay`, matching the pattern already established by `onLiquidate` and `onFlashLoan`:

```solidity
// ICallbacks.sol
interface IRepayCallback {
    function onRepay(address caller, bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
}

// Midnight.sol — repay()
IRepayCallback(callback).onRepay(msg.sender, id, market, units, onBehalf, data)
```

This allows callback contracts to enforce `require(caller == address(this))` or equivalent, matching the self-protection pattern available to `onLiquidate` and `onFlashLoan` implementors.

## Proof of Concept

**Minimal Foundry test outline:**

```solidity
// 1. Deploy MaliciousHelper: implements onRepay returning CALLBACK_SUCCESS,
//    holds loanToken balance, has approved Midnight.
// 2. victim.setIsAuthorized(attacker, true)
// 3. attacker calls midnight.repay(market, units, victim, address(maliciousHelper), "")
// 4. Assert: maliciousHelper.loanToken balance decreased by `units`
//            midnight.debtOf(victim) decreased by `units`
//            attacker spent nothing
```

The test passes with no modifications to the current codebase, confirming the exploit is live. A fuzz variant can sweep `units` up to `victim`'s full debt to confirm the drain is unbounded by the allowance.
