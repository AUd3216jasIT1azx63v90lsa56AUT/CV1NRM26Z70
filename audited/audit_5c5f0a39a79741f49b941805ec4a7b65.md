### Title
Zero-asset take bypasses full-consumption guard on buy offers with `buyerPrice < WAD` - (`src/Midnight.sol`)

### Summary

When `offer.buy == true` and `offer.maxAssets > 0`, the consumed accounting adds `buyerAssets = mulDivDown(units * buyerPrice, WAD)` and checks `newConsumed <= offer.maxAssets`. Because `mulDivDown` truncates to zero for small `units` when `buyerPrice < WAD`, a taker can call `take` with `units > 0` after the offer is fully consumed (`consumed == maxAssets`), pass the cap check with `newConsumed = maxAssets + 0 = maxAssets`, and mutate position state (credit, debt, `totalUnits`) without transferring any assets. The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` explicitly demonstrates and names this as a bug.

### Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // truncates to 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // <= not <
}
``` [1](#0-0) 

`mulDivDown` is `(x * y) / d` — plain integer division: [2](#0-1) 

**Exploit flow (three-step sequence):**

1. Taker calls `take(offer, ..., U1)` where `mulDivDown(U1 * buyerPrice, WAD) = maxAssets - 1` → `consumed = maxAssets - 1`.
2. Taker calls `take(offer, ..., U2)` where `mulDivDown(U2 * buyerPrice, WAD) = 1` → `consumed = maxAssets`. Offer is now fully consumed.
3. Taker calls `take(offer, ..., 1)` where `mulDivDown(1 * buyerPrice, WAD) = 0` (feasible when `buyerPrice < WAD`, i.e. `offerPrice < WAD`). `newConsumed = maxAssets + 0 = maxAssets ≤ maxAssets` → **check passes**. `units = 1 > 0` so position state mutates: `buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` all increase, but zero tokens are transferred.

**Why existing checks fail:** The guard uses `<=` rather than enforcing that `buyerAssets > 0` when `units > 0`. There is no floor check preventing a zero-asset take from proceeding when the offer is already at capacity.

**Attacker-controlled inputs:** `units` (any value satisfying `mulDivDown(units * buyerPrice, WAD) = 0`), `offer.tick` (controls `buyerPrice`; any tick where `offerPrice < WAD` suffices). Both are fully attacker-controlled as an unprivileged taker.

### Impact Explanation

A fully-consumed buy offer (`consumed == maxAssets`) can be taken an unbounded number of additional times with `units > 0` and `buyerAssets = 0`. Each such take:
- Increases the buyer's (maker's) credit and the seller's (taker's) debt by `units` without any token payment.
- Increases `totalUnits` by `units`.
- Leaves `consumed` unchanged at `maxAssets`, so the cap is never exceeded.

This violates the core invariant that `consumed == maxAssets` means the offer is fully consumed and no further fills are allowed. The maker's offer is reused after full consumption to manufacture unbacked credit/debt.

### Likelihood Explanation

**Preconditions:**
- `offer.buy == true`, `offer.maxAssets > 0` — standard offer configuration.
- `buyerPrice < WAD` — requires `offerPrice < WAD`, i.e. a tick below zero. The test uses `MAX_TICK - 16`; any sufficiently low tick works.
- `consumed == maxAssets` — reachable by the attacker themselves via prior legitimate fills, or by the maker via `setConsumed`.

All preconditions are reachable by an unprivileged taker. The attack is repeatable indefinitely on the same offer after it reaches full consumption.

### Recommendation

Add a guard that prevents a non-zero-unit take from contributing zero assets in the `maxAssets` path. The minimal fix is to require `buyerAssets > 0` whenever `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroAssetTake());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce `consumed[offer.maker][offer.group] < offer.maxAssets` before adding, so a fully-consumed offer always reverts regardless of rounding.

### Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already proves the single-step variant (step 3 alone, with `setConsumed` used to pre-fill): [3](#0-2) 

It asserts `buyerAssets == 0`, `sellerAssets == 0`, token balances unchanged, but `creditOf`, `debtOf`, and `totalUnits` all strictly increased — confirming unbacked position mutation on a fully-consumed offer.

**Extended fuzz test plan:**

```solidity
function testFuzz_OfferReuseAfterFullConsumption(
    uint256 maxAssets, uint256 buyerPrice, uint256 units3
) public {
    // Preconditions: offer.buy=true, buyerPrice < WAD, maxAssets > 0
    vm.assume(maxAssets > 0 && buyerPrice > 0 && buyerPrice < WAD);
    // Set consumed == maxAssets (fully consumed)
    vm.prank(maker);
    midnight.setConsumed(offer.group, maxAssets, maker);
    // Choose units3 such that mulDivDown(units3 * buyerPrice, WAD) == 0
    units3 = bound(units3, 1, (WAD - 1) / buyerPrice);
    // Third take must revert — assert it does
    vm.expectRevert(IMidnight.ConsumedAssets.selector);
    take(units3, taker, offer);
}
```

Expected assertion: the third `take` reverts with `ConsumedAssets`. Currently it does not — it succeeds and mutates position state, confirming the bug.
