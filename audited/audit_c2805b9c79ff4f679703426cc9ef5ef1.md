Audit Report

## Title
Authorized-party nonce front-run invalidates victim's pending EcrecoverAuthorizer authorization - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` increments the authorizer's nonce on line 26 as a side-effect of the `require` check, before the signer identity is verified on lines 33–36. Because the signer check accepts any address for which `isAuthorized[authorizer][signer]` is true on Midnight, an attacker who holds a prior Midnight authorization from the victim can craft an arbitrary `Authorization` struct with the victim's current nonce, sign it with their own key, and submit it first — consuming the nonce and causing the victim's pending transaction to revert with `InvalidNonce`.

## Finding Description

**Confirmed code path:** `src/periphery/EcrecoverAuthorizer.sol`, `setIsAuthorized`, lines 24–47.

```solidity
// line 26 — nonce consumed BEFORE signer is checked
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

// lines 33-36 — any Midnight-authorized party is accepted as signer
require(
    signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

**Root cause:** The post-increment `nonce[authorization.authorizer]++` on line 26 commits the nonce consumption as a side-effect of the `require` passing. The subsequent signer check on lines 33–36 accepts not only the authorizer themselves but also any address for which `IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)` returns true. There is no binding between a nonce slot and a specific authorization payload from the authorizer's own intent; any Midnight-authorized party can craft an arbitrary `Authorization` struct with the victim's current nonce and a valid signature from their own key.

**Exploit flow:**
1. Precondition: `isAuthorized[victim][attacker] == true` on Midnight (attacker was previously authorized by victim). `isAuthorized[victim][EcrecoverAuthorizer] == true` on Midnight (required for `EcrecoverAuthorizer` to call `Midnight.setIsAuthorized` on victim's behalf).
2. Victim signs `Authorization{authorizer=victim, authorized=X, isAuthorized=true, nonce=N, deadline=T}` and broadcasts to `EcrecoverAuthorizer.setIsAuthorized`.
3. Attacker observes the mempool and front-runs with `Authorization{authorizer=victim, authorized=attacker_addr, isAuthorized=true, nonce=N, deadline=T'}` signed by the attacker's own key.
4. Attacker's tx executes first: line 26 passes (`N == N`), nonce becomes `N+1`; lines 33–36 pass because `IMidnight(MIDNIGHT).isAuthorized(victim, attacker)` is true; attacker's chosen authorization is written to Midnight via lines 46–47.
5. Victim's tx executes: line 26 fails (`N != N+1`) → `InvalidNonce` revert.

**Why existing checks fail:** The `InvalidNonce` check only prevents replay of an already-consumed nonce; it does not prevent a different authorized party from consuming the nonce first with a different payload. The `Unauthorized` check is the intended delegation feature but is the enabler of the attack.

## Impact Explanation

The victim's pending authorization is permanently invalidated for nonce N, forcing re-signing. The attacker's chosen authorization — which can target any `authorized` address with any `isAuthorized` value — takes effect on Midnight via lines 46–47. The attacker can repeat this for every subsequent nonce as long as they remain authorized, making the `EcrecoverAuthorizer` path permanently unusable for the victim without first revoking the attacker's Midnight authorization via the direct `Midnight.setIsAuthorized` call. This constitutes unauthorized state changes (payload substitution) and permanent degradation of the `EcrecoverAuthorizer` service for the victim. Note: because `isAuthorized[victim][attacker]` being true already permits the attacker to call `Midnight.setIsAuthorized` directly on the victim's behalf, the incremental capability is primarily the nonce-consumption griefing and the ability to substitute a different payload than what the victim signed — the victim cannot prevent this substitution through `EcrecoverAuthorizer` alone.

## Likelihood Explanation

The precondition — attacker holds a Midnight authorization from the victim — is reachable via any prior `setIsAuthorized` call (direct or via `EcrecoverAuthorizer`). Ratifiers, liquidators, and other protocol participants are commonly authorized, as noted in the Midnight contract's AUTHORIZATIONS section (lines 101–110 of `src/Midnight.sol`). Front-running is straightforward on any chain with a public mempool. The attack is repeatable at zero cost beyond gas.

## Recommendation

Move the nonce increment to after the signer verification, so that the nonce is only consumed once the signer is confirmed to be the authorizer or a legitimately authorized party for the specific payload:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    // Do NOT increment nonce here; only check it
    require(authorization.nonce == nonce[authorization.authorizer], InvalidNonce());

    bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
    bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
    bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
    address signer = ecrecover(digest, signature.v, signature.r, signature.s);
    require(signer != address(0), InvalidSignature());
    require(
        signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
        Unauthorized()
    );

    // Increment nonce only after signer is verified
    nonce[authorization.authorizer]++;

    // ... rest of function
}
```

Additionally, consider whether the delegation feature (accepting any Midnight-authorized party as signer) is intentional. If the intent is that only the authorizer themselves can sign EcrecoverAuthorizer messages, remove the `IMidnight(MIDNIGHT).isAuthorized(...)` branch from the signer check entirely.

## Proof of Concept

**Minimal manual steps:**
1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. As victim: call `Midnight.setIsAuthorized(attacker, true, victim)` — grants attacker Midnight authorization.
3. As victim: call `Midnight.setIsAuthorized(EcrecoverAuthorizer, true, victim)` — grants EcrecoverAuthorizer Midnight authorization.
4. As victim: sign `Authorization{authorizer=victim, authorized=someAddress, isAuthorized=true, nonce=0, deadline=T}` with victim's private key. Broadcast to mempool but do not mine yet.
5. As attacker: observe the mempool. Craft `Authorization{authorizer=victim, authorized=attackerAddress, isAuthorized=true, nonce=0, deadline=T'}` and sign with attacker's private key. Submit with higher gas.
6. Attacker's tx mines first: `nonce[victim]` becomes 1; `isAuthorized[victim][attackerAddress]` is set on Midnight.
7. Victim's tx mines: `require(0 == 1, InvalidNonce())` → reverts.
8. Verify: victim's intended authorization was never written; attacker's substituted authorization is live on Midnight; `nonce[victim] == 1`.

**Fuzz/invariant test plan:** Write an invariant asserting that for any `Authorization` struct signed by a key other than `authorization.authorizer`, the resulting Midnight state must match what the authorizer themselves would have produced — this invariant will be violated by the above scenario.
