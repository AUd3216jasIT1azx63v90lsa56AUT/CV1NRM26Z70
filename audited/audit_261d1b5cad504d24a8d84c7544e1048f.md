### Title
`totalUnits` Near-Max Overflow in `take` Blocks Valid Net-Credit-Increasing Takes — (`src/Midnight.sol`)

### Summary

In `take()`, the expression `_marketState.totalUnits + buyerCreditIncrease - sellerCreditDecrease` is computed in `uint256` and then narrowed to `uint128` via `UtilsLib.toUint128`. When `_marketState.totalUnits` is near `type(uint128).max` and `buyerCreditIncrease > sellerCreditDecrease` (net credit increase), the sum exceeds `type(uint128).max` and `toUint128` reverts with `CastOverflow`, blocking an otherwise valid take. No existing check prevents this.

### Finding Description

**Exact code path:**

`src/Midnight.sol` lines 382–384 compute:
```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt);
uint256 sellerCreditDecrease = UtilsLib.min(units, sellerPos.credit);
``` [1](#0-0) 

Then lines 416–417 write:
```solidity
_marketState.totalUnits =
    UtilsLib.toUint128(_marketState.totalUnits + buyerCreditIncrease - sellerCreditDecrease);
``` [2](#0-1) 

`UtilsLib.toUint128` reverts with `CastOverflow` if the argument exceeds `type(uint128).max`:
```solidity
function toUint128(uint256 x) internal pure returns (uint128) {
    require(x <= type(uint128).max, CastOverflow());
    return uint128(x);
}
``` [3](#0-2) 

**Attacker-controlled inputs / preconditions:**

- `_marketState.totalUnits` is near `type(uint128).max`. The protocol explicitly allows this: `testTakeMaxAmount()` reaches exactly `type(uint128).max`, and the NatSpec states "The max amount of totalUnits… is `type(uint128).max`". [4](#0-3) [5](#0-4) 
- A buy offer is taken where the buyer has zero debt (`buyerCreditIncrease = units`) and the seller has zero credit (`sellerCreditDecrease = 0`), i.e., a new lender takes a buy offer from a new borrower.
- `units` is small (e.g., 100), so no individual position overflows.

**Concrete overflow:**

```
totalUnits = type(uint128).max - 50
buyerCreditIncrease = 100
sellerCreditDecrease = 0
sum = type(uint128).max - 50 + 100 = type(uint128).max + 50  → CastOverflow
```

**Why existing checks fail:**

Scanning all `require` statements in `take()` — loss-factor check, tick check, timestamp checks, ratifier, reduceOnly, gate checks — none bound `totalUnits + buyerCreditIncrease - sellerCreditDecrease` to `type(uint128).max`. [6](#0-5) 

The individual position writes at lines 408–414 do not overflow because `buyerPos.credit` and `sellerPos.debt` are each bounded by `type(uint128).max` independently; the overflow is only in the market-level aggregate. [7](#0-6) 

### Impact Explanation

Once `totalUnits` reaches `type(uint128).max - k` for small `k`, any take where `buyerCreditIncrease - sellerCreditDecrease > k` reverts with `CastOverflow`. This means a valid lender cannot take any buy offer that would increase net credit in the market, even for economically sound, small-unit takes. Existing lenders and borrowers are unaffected (withdraw and repay still work), but new lending is permanently blocked — a partial fund freeze on new entry.

### Likelihood Explanation

**Precondition:** `totalUnits` must be within ~`k` of `type(uint128).max` (~3.4 × 10^38). For an 18-decimal token this is ~3.4 × 10^20 tokens — far beyond any realistic market TVL today. The protocol's own test suite confirms `type(uint128).max` is reachable in a single take (`testTakeMaxAmount`), but reaching it organically through accumulated takes requires an astronomically large market. Likelihood is therefore **very low** in practice, though the code path is unconditionally reachable and the revert is not documented as expected behavior for this incremental scenario (only the single-take-above-max case is tested in `testTakeAboveMaxAmountReverts`). [8](#0-7) 

### Recommendation

Before the `toUint128` cast, add an explicit cap check or revert with a descriptive error:

```solidity
uint256 newTotalUnits = _marketState.totalUnits + buyerCreditIncrease - sellerCreditDecrease;
require(newTotalUnits <= type(uint128).max, TotalUnitsOverflow());
_marketState.totalUnits = uint128(newTotalUnits);
```

Alternatively, document this as a hard protocol limit and add a pre-take check that rejects takes that would push `totalUnits` over `type(uint128).max`, returning a clear error rather than the opaque `CastOverflow`.
