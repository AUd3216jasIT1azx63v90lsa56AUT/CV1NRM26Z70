Audit Report

## Title
Unconditional zero-value `safeTransfer` to receiver when `filledSellerAssets == 0` causes revert on zero-transfer-reverting `loanToken` - (File: src/periphery/MidnightBundles.sol)

## Summary
In `supplyCollateralAndSellWithUnitsTarget`, when `targetUnits == 0` the takes loop is skipped entirely, leaving `filledSellerAssets == 0`. The final `SafeTransferLib.safeTransfer(loanToken, receiver, 0)` at line 168 executes unconditionally — unlike the referral fee transfer at line 167 which is guarded by `if (referralFeeAssets > 0)`. On any `loanToken` that reverts on zero-value `transfer` calls, every invocation with `targetUnits == 0` and `minSellerAssets == 0` will revert, making the no-op path permanently unusable for such tokens despite the contract explicitly documenting that no-ops are allowed.

## Finding Description
`supplyCollateralAndSellWithUnitsTarget` accepts `targetUnits` with no lower-bound check. When `targetUnits == 0`, the loop condition `filledUnits < targetUnits` at line 144 is immediately false (both start at 0), so the loop body never executes and `filledSellerAssets` remains 0. [1](#0-0) 

The post-loop `require(filledUnits == targetUnits, OutOfOffers())` at line 163 passes trivially (0 == 0). [2](#0-1) 

`referralFeeAssets = 0.mulDivDown(referralFeePct, WAD) = 0`, so the slippage check `require(0 - 0 >= minSellerAssets)` passes when `minSellerAssets == 0`. The referral transfer at line 167 is correctly guarded by `if (referralFeeAssets > 0)`, but the receiver transfer at line 168 has no equivalent guard and unconditionally calls `SafeTransferLib.safeTransfer(loanToken, receiver, 0)`. [3](#0-2) 

`SafeTransferLib.safeTransfer` makes a raw `.call()` to `token.transfer(receiver, 0)` and propagates any revert from the token via the assembly block. [4](#0-3) 

If the token reverts on zero-value transfers, the entire call reverts. The contract-level NatDev comment at line 25 (`/// @dev No-ops are allowed.`) confirms `targetUnits == 0` is an intended valid input, making this a real behavioral defect rather than an invalid-input edge case. [5](#0-4) 

## Impact Explanation
Any call to `supplyCollateralAndSellWithUnitsTarget` with `targetUnits == 0` and `minSellerAssets == 0` on a market whose `loanToken` reverts on zero-value transfers will always revert at line 168. The transaction fails atomically — collateral supplies are rolled back — so no funds are permanently lost. However, the no-op path (explicitly documented as supported) is rendered permanently unusable for such tokens, and callers waste gas on every attempt. Severity is low: no fund loss, no state corruption, but a concrete and repeatable denial of a documented feature.

## Likelihood Explanation
All three preconditions are independently reachable by any unprivileged caller: (1) `targetUnits == 0` has no on-chain guard; (2) `minSellerAssets == 0` is the natural default for no slippage protection; (3) tokens that revert on zero-value transfers are uncommon but real (e.g., certain USDT deployments, custom ERC-20s). The scenario is deterministic and repeatable — every call with these inputs on such a token will revert.

## Recommendation
Add a zero-value guard on the receiver transfer, mirroring the existing guard on the referral fee transfer:

```solidity
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
uint256 receiverAssets = filledSellerAssets - referralFeeAssets;
if (receiverAssets > 0) SafeTransferLib.safeTransfer(loanToken, receiver, receiverAssets);
```

This is consistent with the pattern already used for `referralFeeAssets` at line 167 and eliminates the zero-transfer call when `filledSellerAssets == 0`. [6](#0-5) 

## Proof of Concept
1. Deploy or fork a network with a `loanToken` that reverts on `transfer(to, 0)` (e.g., a custom ERC-20 with `require(amount > 0)`).
2. Create a Midnight market using that token as `loanToken`.
3. Call `supplyCollateralAndSellWithUnitsTarget` with:
   - `targetUnits = 0`
   - `minSellerAssets = 0`
   - `collateralSupplies = []` (empty, to avoid collateral side effects)
   - `takes` = any single valid offer entry (needed to avoid `takes[0]` out-of-bounds at line 129)
   - `referralFeePct = 0`
4. Observe the call reverts at line 168 due to the zero-value `safeTransfer` to `receiver`.
5. Confirm that the identical call on a standard ERC-20 (which accepts zero-value transfers) succeeds, proving the root cause is the unconditional zero transfer rather than any other guard.
