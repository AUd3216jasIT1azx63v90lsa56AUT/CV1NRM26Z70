### Title
Missing Zero-Address Validation on `receiver` in `liquidate()` Causes Permanent Loss of Seized Collateral — (File: src/Midnight.sol)

---

### Summary

The `liquidate` function in `Midnight.sol` accepts a `receiver` address parameter with no zero-address guard. If a liquidator mistakenly passes `receiver = address(0)`, the seized collateral is transferred to the zero address (burned) while the liquidator still pays the full `repaidUnits` of loan tokens to the protocol. This is a direct analog to the Gearbox `CreditManager` finding: a liquidator loses assets by calling the function with a zero-address receiver.

---

### Finding Description

In `liquidate()`, the seized collateral is unconditionally transferred to `receiver` at line 696:

```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
``` [1](#0-0) 

No check exists anywhere in the function that `receiver != address(0)`. [2](#0-1) 

`SafeTransferLib.safeTransfer` performs no zero-address validation — it calls `token.transfer(to, value)` directly: [3](#0-2) 

The protocol's own TOKEN SAFETY REQUIREMENTS (lines 133–140) explicitly accommodate tokens that "should not revert because a transfer is no-op," meaning the protocol is designed to work with tokens that may silently succeed on a transfer to `address(0)`. For any such collateral token, the seized assets are permanently burned.

After the collateral transfer, the liquidator's loan token payment is pulled unconditionally: [4](#0-3) 

Net result: the liquidator pays `repaidUnits` of loan tokens to the protocol and receives zero collateral in return.

The same missing check exists in two other user-facing functions:

- **`withdraw()`** — loan tokens sent to `receiver` with no zero-address check: [5](#0-4) 

- **`withdrawCollateral()`** — collateral tokens sent to `receiver` with no zero-address check: [6](#0-5) 

---

### Impact Explanation

A liquidator calling `liquidate(..., receiver = address(0), ...)` on a market whose collateral token does not revert on transfer to `address(0)`:

1. Has their `repaidUnits` of loan tokens pulled from their wallet (line 717).
2. Receives zero collateral — it is burned at `address(0)` (line 696).
3. The borrower's debt is reduced and the protocol's `withdrawable` increases normally (lines 675–676), so the protocol state is updated correctly — only the liquidator suffers.

This is a **direct, unrecoverable loss of funds** for the liquidator. The magnitude equals the full value of `seizedAssets` of collateral. [7](#0-6) 

---

### Likelihood Explanation

- `liquidate` is a **permissionless** function — any address can call it.
- Liquidator bots are commonly built with configurable `receiver` parameters; a misconfiguration or off-by-one in calldata encoding can produce `address(0)`.
- The impact is conditional on the collateral token not reverting on transfer to `address(0)`. Standard OpenZeppelin ERC-20 tokens do revert, but the protocol explicitly supports non-standard tokens (TOKEN SAFETY REQUIREMENTS, lines 133–140), and many custom or wrapped tokens do not include this guard.
- No privileged access is required; any liquidator can trigger this by mistake.

---

### Recommendation

Add a zero-address guard at the top of `liquidate`, `withdraw`, and `withdrawCollateral`:

```solidity
require(receiver != address(0), ZeroAddress());
```

This is the exact fix recommended in the analogous Gearbox report and eliminates the entire vulnerability class across all three functions.

---

### Proof of Concept

**Preconditions:**
- A market exists with a collateral token that does not revert on `transfer(address(0), amount)` (e.g., a custom ERC-20 without zero-address validation).
- A borrower has an unhealthy position with `N` units of collateral and `D` units of debt.

**Steps:**
1. Liquidator calls:
   ```solidity
   midnight.liquidate(
       market,
       collateralIndex,
       N,          // seizedAssets
       0,          // repaidUnits (computed from seizedAssets)
       borrower,
       false,
       address(0), // receiver — mistakenly zero
       address(0), // no callback
       ""
   );
   ```
2. Protocol computes `repaidUnits` from `seizedAssets` (line 650), reduces borrower's debt, increases `withdrawable`.
3. Line 696: `safeTransfer(collateralToken, address(0), N)` — collateral burned.
4. Line 717: `safeTransferFrom(loanToken, liquidator, address(this), repaidUnits)` — liquidator pays loan tokens.
5. **Result:** Liquidator's loan tokens are taken; seized collateral is permanently burned. The liquidator loses the full value of the seized collateral. [8](#0-7) [9](#0-8)

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
    }
