### Title
Buy-offer assets-mode consumed never increments when `units.mulDivDown(buyerPrice, WAD) == 0`, allowing unbounded unit fills - (`src/Midnight.sol`)

### Summary

When a buy offer is created with `maxAssets > 0` and any tick below `MAX_TICK`, calling `take` with `units = 1` produces `buyerAssets = 1 * tickToPrice(tick) / WAD = 0` due to integer truncation. The consumed mapping is incremented by `buyerAssets` (zero), so the `ConsumedAssets` cap is never reached and the offer can be filled with an unlimited number of units across repeated calls.

### Finding Description

**Exact code path — `src/Midnight.sol` lines 358–373:**

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);          // < WAD for any tick < MAX_TICK
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;            // == offerPrice for buy offer
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // (1 * offerPrice) / 1e18 == 0 when offerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    // += 0 → newConsumed unchanged
    require(newConsumed <= offer.maxAssets, ConsumedAssets());  // 0 <= maxAssets always passes
}
```

**Root cause:** In assets mode the consumed mapping is incremented by the *asset amount*, not by `units`. `mulDivDown` is plain integer division (`(x * y) / d`, `src/libraries/UtilsLib.sol` line 30), so for any `units < WAD / buyerPrice` the result truncates to zero. For `units = 1` this holds for every tick except `MAX_TICK` (where `tickToPrice == WAD`).

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick` ∈ `[0, MAX_TICK-1]` (any tick the maker chose)
- `units = 1` per call (or any value < `WAD / buyerPrice`)

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N` and tick < `MAX_TICK` (e.g., tick = 2, `offerPrice = 1e12`).
2. Taker calls `take(offer, ..., units=1, ...)` in a loop.
3. Each iteration: `buyerAssets = (1 * 1e12) / 1e18 = 0`; `consumed[maker][group] += 0`; cap check `0 <= N` passes.
4. Position accounting still executes with `units = 1`: maker's credit grows by 1, taker's debt grows by 1 (line 410, 414).
5. Token transfers are `safeTransferFrom(..., 0)` — succeed silently (lines 455–456).
6. Health check on the seller (taker) at line 476 is the only gate; taker needs collateral, but the maker's cap is irrelevant to it.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — `newConsumed` is always 0, so this never fires.
- No `require(units > 0)` guard.
- No `require(buyerAssets > 0)` guard.
- The Certora `Consume.spec` rule `takeConsumedDelta` (line 67) explicitly scopes itself to units mode (`require offer.maxAssets == 0`) and does not assert a minimum increment in assets mode. `mulDivDown` is also summarized as `NONDET` in that spec (line 11), so the prover never reasons about truncation-to-zero. ### Impact Explanation

The `maxAssets` cap on a buy offer is completely bypassed. A taker can fill the offer with an unbounded number of units (each costing 0 assets), forcing the maker to accumulate credit far beyond their intended limit. The maker's credit grows without bound while `consumed` stays at zero, violating the core invariant that "offers cannot be replayed, overfilled, reused, or filled after cancel/deadline." Additionally, the taker accumulates debt without receiving any loan tokens per fill, which is economically incoherent and breaks the expected asset-to-unit correspondence. [5](#0-4) 

### Likelihood Explanation

**Preconditions:**
- Any buy offer with `maxAssets > 0` at any tick except `MAX_TICK` (5820) — this covers the vast majority of realistic offer ticks since `MAX_TICK` corresponds to price = WAD (par), and most lending offers are below par.
- Taker must have enough collateral to remain healthy after accumulating debt (line 476). This is a self-imposed cost, not a protocol barrier.
- Settlement fee must satisfy `offerPrice >= _settlementFee` (otherwise the buy offer reverts on underflow at line 361). With zero or small settlement fees this is trivially satisfied.

**Feasibility:** Fully reachable with no special privileges. Any address can be a taker. The attack is repeatable in a single transaction via `multicall`. [6](#0-5) [7](#0-6) 

### Recommendation

In assets mode, enforce that each fill contributes at least 1 to `consumed` when `units > 0`. Two concrete options:

1. **Minimum-assets guard:** After computing `buyerAssets`/`sellerAssets`, add:
   ```solidity
   require(units == 0 || (offer.buy ? buyerAssets : sellerAssets) > 0, ZeroAssets());
   ```
2. **Hybrid increment:** In assets mode, increment `consumed` by `max(assetAmount, units > 0 ? 1 : 0)` so that every non-zero-unit fill advances the counter by at least 1.

Option 1 is simpler and matches the existing comment ("All sellerAssets are reachable with the units input, and all buyerAssets are reachable only if buyerPrice <= WAD") — it makes the zero-asset case an explicit revert rather than a silent no-op. [8](#0-7) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {TickLib, MAX_TICK} from "../src/libraries/TickLib.sol";
import {IMidnight} from "../src/interfaces/IMidnight.sol";

contract ConsumedZeroIncrementFuzzTest is BaseTest {
    // Fuzz: any tick in [2, MAX_TICK-1] (tick 0 gives price 0, tick 1 gives price 0 too)
    function testBuyOfferMaxAssetsConsumedNeverIncrements(uint256 tick) public {
        tick = bound(tick, 2, MAX_TICK - 1);
        // Align to default tick spacing
        tick = (tick / DEFAULT_TICK_SPACING) * DEFAULT_TICK_SPACING;
        if (tick == 0) tick = DEFAULT_TICK_SPACING;

        // Zero settlement fee so buyerPrice == offerPrice < WAD
        midnight.setDefaultSettlementFee(address(loanToken), 1, 0);

        uint256 offerPrice = TickLib.tickToPrice(tick);
        // Confirm the truncation: 1 * offerPrice / WAD == 0
        assertEq((1 * offerPrice) / 1e18, 0, "precondition: buyerAssets rounds to 0");

        // Maker posts buy offer with maxAssets cap
        borrowerOffer.tick = tick;
        borrowerOffer.maxAssets = 1e18; // non-zero cap
        borrowerOffer.maxUnits = 0;

        // Taker needs collateral for debt
        collateralize(market, borrower, 1000);
        deal(address(loanToken), lender, 1000);

        uint256 consumedBefore = midnight.consumed(borrowerOffer.maker, borrowerOffer.group);

        // First fill: units=1, buyerAssets=0 → consumed should NOT stay at 0
        take(1, lender, borrowerOffer);

        uint256 consumedAfter = midnight.consumed(borrowerOffer.maker, borrowerOffer.group);

        // ASSERTION: consumed must increase by at least 1 when units > 0
        // This assertion FAILS, proving the bug
        assertGt(consumedAfter, consumedBefore, "consumed must increase per fill");

        // Demonstrate unbounded fill: repeat 100 times, consumed stays 0
        for (uint256 i = 0; i < 99; i++) {
            take(1, lender, borrowerOffer);
        }
        assertEq(midnight.consumed(borrowerOffer.maker, borrowerOffer.group), 0,
            "consumed is still 0 after 100 fills — maxAssets cap fully bypassed");
    }
}
```

**Expected assertions:**
- `assertGt(consumedAfter, consumedBefore)` **fails** — consumed stays at 0 after a `units=1` fill.
- `assertEq(consumed, 0)` after 100 fills **passes** — the cap is never approached.
- The maker's credit increases by 100 units while `consumed` remains 0, confirming unbounded overfill. [9](#0-8) [10](#0-9)
