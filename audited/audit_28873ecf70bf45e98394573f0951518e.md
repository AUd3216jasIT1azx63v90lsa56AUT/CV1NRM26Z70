### Title
Unconditional zero-value `safeTransferFrom` in `take()` causes permanent DoS for zero-settlement-fee markets with zero-revert loan tokens - (`src/Midnight.sol`)

### Summary
When `settlementFee(id, timeToMaturity) == 0` (the protocol default for all new markets), `buyerAssets` and `sellerAssets` are computed with identical prices, making their difference exactly zero. `take()` unconditionally calls `SafeTransferLib.safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets)` with value `0` at line 455, with no guard. For any ERC20 loan token that reverts on zero-value transfers, every `take()` call on such a market reverts, making the market permanently unusable until a privileged fee setter intervenes.

### Finding Description

**Code path:**

In `src/Midnight.sol`, `take()` computes:

```
uint256 _settlementFee = settlementFee(id, timeToMaturity);   // line 360
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;  // line 361
uint256 buyerPrice  = sellerPrice + _settlementFee;            // line 362
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)
    : units.mulDivUp(buyerPrice, WAD);                         // line 363
uint256 sellerAssets = offer.buy
    ? units.mulDivDown(sellerPrice, WAD)
    : units.mulDivUp(sellerPrice, WAD);                        // line 364
```

When `_settlementFee == 0`, `buyerPrice == sellerPrice == offerPrice`, so `buyerAssets == sellerAssets` for every value of `units` and every rounding direction. The difference `buyerAssets - sellerAssets` is always exactly `0`.

Later, unconditionally:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);  // line 455
```

`SafeTransferLib.safeTransferFrom` (in `src/libraries/SafeTransferLib.sol`) makes a raw `.call` to the token and propagates any revert verbatim:

```solidity
(bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
if (!success) {
    assembly ("memory-safe") {
        revert(add(returndata, 0x20), mload(returndata))
    }
}
```

There is no `if (buyerAssets > sellerAssets)` guard anywhere before this call.

**Root cause:** Missing zero-amount guard on the settlement-fee transfer leg.

**Attacker-controlled inputs:**
- Any unprivileged actor (taker, market creator) can deploy or use a loan token that reverts on zero-value `transferFrom` calls.
- Market creation is permissionless (`touchMarket`); any actor can create a market with such a token.
- Settlement fees default to `0` for all breakpoints (stored as `uint16 settlementFeeCbp*` fields, zero-initialized) unless the privileged `feeSetter` explicitly sets them.

**Exploit flow:**
1. Deploy a loan token whose `transferFrom` reverts when `value == 0`.
2. Call `touchMarket(market)` with that token as `loanToken` — no fee setter action needed; all breakpoints default to `0`.
3. Any taker calls `take(offer, ...)` with any valid `units > 0`.
4. `settlementFee(id, timeToMaturity) == 0` → `buyerAssets == sellerAssets` → `buyerAssets - sellerAssets == 0`.
5. Line 455 calls `safeTransferFrom(loanToken, payer, address(this), 0)`.
6. Token reverts → entire `take()` reverts (all state changes rolled back by EVM).
7. No `take()` can ever succeed on this market.

**Why existing checks fail:** There is no check on the value passed to `safeTransferFrom` at line 455. The `SafeTransferLib` itself has no zero-amount guard. The `ERC20RevertToZero` token in the test suite (`test/erc20s/ERC20RevertToZero.sol`) reverts on transfers *to* `address(0)`, not on zero-amount transfers — so this specific failure mode is untested.

### Impact Explanation
Any market whose loan token reverts on zero-value `transferFrom` is completely bricked for `take()` as long as the settlement fee is zero (the default). No offer can be filled, no lending or borrowing can occur through `take()`. The DoS is low-cost (just deploying a token and creating a market) and requires no ongoing attacker action.

### Likelihood Explanation
Settlement fee is zero by default for every new market — no admin action is needed to reach the vulnerable state. Zero-revert-on-zero-transfer tokens exist in production (e.g., tokens following strict ERC20 interpretations). The precondition is trivially reachable by any unprivileged actor who controls the loan token. The condition is persistent until the `feeSetter` sets a non-zero fee for every breakpoint, which is a privileged out-of-band fix.

### Recommendation
Add a zero-amount guard before the settlement-fee transfer in `take()`:

```solidity
// src/Midnight.sol, after line 454
uint256 feeAmount = buyerAssets - sellerAssets;
if (feeAmount > 0) {
    SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), feeAmount);
}
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

Apply the same guard to the `sellerAssets` transfer at line 456 for completeness, since `sellerAssets` can also be zero when `units * sellerPrice < WAD`.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
