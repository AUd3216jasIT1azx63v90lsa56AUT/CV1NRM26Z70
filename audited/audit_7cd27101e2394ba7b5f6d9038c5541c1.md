### Title
Zero-value `safeTransferFrom` in `take()` blocks valid takers on loan tokens that revert on zero-value transfers - (File: `src/Midnight.sol`)

### Summary
When `offer.buy = true` and `units` is small enough that `units * buyerPrice < WAD`, the expression `units.mulDivDown(buyerPrice, WAD)` at line 363 rounds to zero. Both subsequent `safeTransferFrom` calls at lines 455–456 are then invoked with `value = 0`, and `SafeTransferLib.safeTransferFrom` contains no zero-value guard, so any loan token that reverts on zero-value `transferFrom` will cause the entire `take()` to revert, permanently blocking that taker/units combination.

### Finding Description
**Exact code path:**

`src/Midnight.sol` line 363 — for a buy offer (`offer.buy = true`), `buyerAssets` is computed with `mulDivDown`: [1](#0-0) 

`mulDivDown` is plain integer division `(x * y) / d`: [2](#0-1) 

For `units = 1` and any `buyerPrice < WAD` (e.g. `buyerPrice = 0.5e18`): `(1 * 0.5e18) / 1e18 = 0`. Since `buyerPrice >= sellerPrice`, `sellerAssets` also rounds to 0 by the same arithmetic.

Both transfers at lines 455–456 are then called with `value = 0`: [3](#0-2) 

`SafeTransferLib.safeTransferFrom` has no `if (value == 0) return` guard — it unconditionally calls `token.transferFrom(from, to, 0)`: [4](#0-3) 

If the loan token reverts on zero-value transfers (a known pattern in tokens such as LEND, BNB, and various fee-on-transfer tokens), the `!success` branch triggers the assembly revert, propagating the token's revert reason and unwinding the entire `take()` call.

**No existing check stops this.** There is no `require(units > 0)` or `require(buyerAssets > 0)` guard anywhere in `take()`. The `consumed` tracking at lines 367–373 only enforces an upper bound, not a lower bound: [5](#0-4) 

The `WAD` constant is `1e18`: [6](#0-5) 

For a 0-decimal loan token, 1 unit = 1 whole token. With any `buyerPrice < 1e18` (i.e., any price below par, which is the normal case for a discount bond), a taker supplying `units = 1` produces `buyerAssets = 0`. This is not a contrived edge case — it is the normal operating range for small-denomination takes on low-decimal markets.

### Impact Explanation
Any taker attempting a small-units take on a market whose loan token reverts on zero-value `transferFrom` will have their `take()` call revert unconditionally. The taker cannot split the take into smaller pieces (the minimum is already `units = 1`), and cannot increase `units` without changing the economic terms of the trade. The market is effectively frozen for that granularity of take, blocking debt entry and credit exit for affected participants.

### Likelihood Explanation
Preconditions are: (1) a loan token that reverts on zero-value `transferFrom` — a documented non-standard but real-world behavior; (2) `units` small enough that `units * buyerPrice < WAD`, which is trivially satisfied for `units = 1` at any sub-par price; (3) `offer.buy = true` (buy offer, taker is seller). All three are independently reachable by an unprivileged market creator choosing the loan token and an unprivileged taker choosing `units`. The condition is repeatable and deterministic once the market is created with such a token.

### Recommendation
Add a zero-value guard before each `safeTransferFrom` call in `take()`, or add a single guard in `SafeTransferLib.safeTransferFrom`:

```solidity
// In SafeTransferLib.safeTransferFrom:
if (value == 0) return;
```

Alternatively, guard in `take()` directly:

```solidity
if (buyerAssets - sellerAssets > 0)
    SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
if (sellerAssets > 0)
    SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

        require(amt > 0, "zero transfer rejected");

        balanceOf[to] += amt;
        return true;
    }

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
