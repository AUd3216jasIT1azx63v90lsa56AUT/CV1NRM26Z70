### Title
Fee-on-transfer token in `flashLoan` causes guaranteed revert on repayment leg - ([File: src/Midnight.sol])

### Summary
The `flashLoan` function in `src/Midnight.sol` sends exactly `assets[i]` to the callback via `safeTransfer`, then unconditionally demands exactly `assets[i]` back via `safeTransferFrom`. For any fee-on-transfer token, the callback receives only `assets[i] * (1 - fee_rate)`, making it impossible to repay the full `assets[i]` without pre-existing surplus, so the repayment leg always reverts. No balance-before/after guard exists to detect or accommodate the shortfall.

### Finding Description
The reachable code path is `src/Midnight.sol` lines 737–752:

```
safeTransfer(tokens[i], callback, assets[i])   // callback receives assets[i]*(1-fee)
onFlashLoan(...)                                // callback executes
safeTransferFrom(tokens[i], callback, this, assets[i])  // demands full assets[i] back → revert
``` [1](#0-0) 

`SafeTransferLib.safeTransfer` and `safeTransferFrom` are thin wrappers that propagate any token-level revert directly: [2](#0-1) 

There is no `balanceBefore` snapshot and no `balanceAfter >= balanceBefore` invariant check. The function uses the caller-supplied `assets[i]` value as both the outbound amount and the required inbound amount. For a fee-on-transfer token the callback's balance after receipt is `assets[i] - fee`, so `safeTransferFrom(..., assets[i])` will revert either because the token itself reverts on insufficient balance, or because the token's `transferFrom` returns `false` (caught by `TransferFromReturnedFalse`).

Attacker-controlled inputs: `tokens[]` (any fee-on-transfer ERC20), `assets[]` (any non-zero amount), `callback` (any address implementing `onFlashLoan`). No privileged role is required; `flashLoan` is `external` with no access control.

`TickLib.sol` / `tickToPrice` are pure arithmetic functions with zero ERC20 interaction and are not part of this exploit path. [3](#0-2) 

### Impact Explanation
Every call to `flashLoan` with a fee-on-transfer token reverts at the `safeTransferFrom` repayment step. Flash loan functionality is completely non-functional for such tokens; no flash loan can ever succeed regardless of callback logic or pre-funded balances (unless the callback independently sources extra tokens to cover the fee, which is not a protocol guarantee).

### Likelihood Explanation
Precondition: the token passed to `flashLoan` charges a transfer fee. This is a property of the token, not of any privileged configuration. Any unprivileged caller can trigger this by passing such a token. The revert is deterministic and repeatable on every invocation. The only mitigating factor is that fee-on-transfer tokens are relatively uncommon among high-value assets.

### Recommendation
Replace the exact-amount repayment pattern with a balance-delta check:

```solidity
uint256[] memory balancesBefore = new uint256[](tokens.length);
for (uint256 i = 0; i < tokens.length; i++) {
    balancesBefore[i] = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
}
IFlashLoanCallback(callback).onFlashLoan(msg.sender, tokens, assets, data);
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(IERC20(tokens[i]).balanceOf(address(this)) >= balancesBefore[i], RepaymentShort());
}
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are unsupported (e.g., via a token allowlist), so callers receive a clear error rather than an opaque revert.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";
import {IFlashLoanCallback} from "src/interfaces/ICallbacks.sol";

contract FeeToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    uint256 constant FEE_BPS = 100; // 1%

    function mint(address to, uint256 amt) external { balanceOf[to] += amt; }
    function approve(address sp, uint256 amt) external { allowance[msg.sender][sp] = amt; }

    function transfer(address to, uint256 amt) external returns (bool) {
        uint256 fee = amt * FEE_BPS / 10000;
        balanceOf[msg.sender] -= amt;
        balanceOf[to] += amt - fee; // fee burned
        return true;
    }
    function transferFrom(address from, address to, uint256 amt) external returns (bool) {
        allowance[from][msg.sender] -= amt;
        uint256 fee = amt * FEE_BPS / 10000;
        balanceOf[from] -= amt;
        balanceOf[to] += amt - fee;
        return true;
    }
}

contract FeeFlashTest is Test, IFlashLoanCallback {
    Midnight midnight;
    FeeToken token;

    function setUp() public {
        midnight = new Midnight(address(this));
        token = new FeeToken();
        token.mint(address(midnight), 1e18);
    }

    function onFlashLoan(address, address[] calldata tokens, uint256[] calldata amounts, bytes calldata)
        external returns (bytes32)
    {
        // approve repayment of full amount (callback only has amounts[0]*(1-fee))
        FeeToken(tokens[0]).approve(msg.sender, amounts[0]);
        return CALLBACK_SUCCESS;
    }

    function testFlashLoanFeeTokenAlwaysReverts() public {
        address[] memory tokens = new address[](1);
        tokens[0] = address(token);
        uint256[] memory amounts = new uint256[](1);
        amounts[0] = 1e18;

        // Assert: always reverts because callback received only 0.99e18 but must repay 1e18
        vm.expectRevert();
        midnight.flashLoan(tokens, amounts, address(this), "");
    }
}
```

Expected assertion: `vm.expectRevert()` passes unconditionally — the `safeTransferFrom` on the repayment leg reverts because the callback's balance (`assets * 0.99`) is less than the demanded repayment (`assets`), confirming the flash loan is permanently broken for fee-on-transfer tokens.
