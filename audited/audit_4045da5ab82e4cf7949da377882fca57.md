### Title
MCOPY Opcode Incompatibility Breaks Pool Creation and Array-Returning Functions on Non-Cancun BattleChain - (File: src/ConfidencePoolFactory.sol)

### Summary
`foundry.toml` specifies Solidity 0.8.26 with no `evm_version` override, so the compiler defaults to the Cancun EVM target. Solidity ≥ 0.8.25 emits the `MCOPY` opcode for `abi.encode()` calls and functions returning dynamic memory arrays. If BattleChain (an EVM-compatible L2, chainId 627) has not upgraded to Cancun, every call to `createPool()` will revert unconditionally, making the entire protocol non-deployable/non-functional.

### Finding Description
`foundry.toml` sets `solc = "0.8.26"` and `via_ir = true` but contains no `evm_version` key. A project-wide grep confirms zero occurrences of `evm_version` anywhere in the repository. Solidity 0.8.25 changed the default EVM compilation target from Shanghai to **Cancun**, which introduces the `MCOPY` (0x5e) opcode for memory-copy operations. Solidity 0.8.26 inherits this default.

The following code paths in the two scoped contracts will emit `MCOPY` in the compiled bytecode:

1. **`ConfidencePoolFactory.createPool()`** — `abi.encode(agreement, _poolsByAgreement[agreement].length)` at line 86 is the salt computation for deterministic clone deployment. `abi.encode()` is one of the operations Solidity 0.8.25+ compiles to `MCOPY` by default.

2. **`ConfidencePool.getScopeAccounts()`** — returns `address[] memory` (a dynamic memory array copy) at line 644.

3. **`ConfidencePoolFactory.getPoolsByAgreement()`** — returns `address[] memory` at line 117.

`MCOPY` is an invalid opcode on any EVM that has not upgraded to Cancun. Executing it causes an immediate revert with no error data (treated as an invalid instruction).

### Impact Explanation
`createPool()` is the sole entry point for deploying pool clones. Because `abi.encode()` is called unconditionally inside it (line 86), every invocation reverts on a non-Cancun chain. No pool can ever be created, making the entire protocol non-functional. Stakers cannot stake, sponsors cannot create pools, and no funds can enter or exit the system. This is a complete protocol-level denial of service, not merely a partial degradation.

Additionally, `getScopeAccounts()` and `getPoolsByAgreement()` would revert, breaking any off-chain tooling or integrations that query pool scope or enumerate pools.

### Likelihood Explanation
BattleChain is explicitly described in `README.md` as an "EVM-compatible L2." L2 chains routinely lag behind mainnet EVM upgrades; Cancun support requires explicit hardfork coordination by the L2 operator. The external report was filed precisely because this class of chain (custom/L2, chainId 627) is the common deployment target that has not yet adopted Cancun. The absence of any `evm_version` setting in `foundry.toml` is a confirmed fact, not a speculation.

### Recommendation
Add `evm_version = "paris"` to `foundry.toml` under `[profile.default]`. Paris predates both `PUSH0` (Shanghai) and `MCOPY` (Cancun), giving the broadest L2 compatibility. Alternatively, validate that BattleChain mainnet and testnet have both activated the Cancun hardfork before deploying with the current settings. If Shanghai (`PUSH0`) is confirmed supported, `evm_version = "shanghai"` is also acceptable.

```toml
[profile.default]
solc = "0.8.26"
via_ir = true
optimizer = true
optimizer_runs = 200
evm_version = "paris"   # add this line
```

### Proof of Concept
1. Confirm no `evm_version` is set: `grep -r "evm_version" .` returns zero results.
2. Compile: `forge build` — Solidity 0.8.26 defaults to Cancun, emitting `MCOPY`.
3. Inspect bytecode: `forge inspect ConfidencePoolFactory bytecode | grep -o '5e'` — `0x5e` (MCOPY) will appear.
4. Deploy `ConfidencePoolFactory` on a non-Cancun BattleChain node.
5. Call `createPool(agreement, stakeToken, expiry, minStake, recoveryAddress, accounts)`.
6. The transaction reverts at the `abi.encode(agreement, _poolsByAgreement[agreement].length)` line (line 86) because the EVM encounters `MCOPY` and treats it as an invalid opcode. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
