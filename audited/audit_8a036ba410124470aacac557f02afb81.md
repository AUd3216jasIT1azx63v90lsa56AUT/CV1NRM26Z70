### Title
Fee-on-Transfer `loanToken` in `repay()` Inflates `withdrawable` Beyond Actual Contract Balance - (File: src/Midnight.sol)

### Summary
`repay()` unconditionally credits `marketState[id].withdrawable` by the caller-supplied `units` before and regardless of how many tokens actually arrive at the contract. When `market.loanToken` silently deducts a transfer fee, `SafeTransferLib.safeTransferFrom` delivers only `units * (1 - fee_rate)` to `address(this)`, while `withdrawable` is inflated by the full `units`. Repeated repayments accumulate a growing gap between `withdrawable` and the contract's real token balance, allowing lenders to collectively drain more tokens than the contract holds.

### Finding Description
**Exact code path:**

`repay()` at `src/Midnight.sol` lines 502–521:

```
position[id][onBehalf].debt      -= toUint128(units);   // line 508
marketState[id].withdrawable     += toUint128(units);   // line 509
// ...
SafeTransferLib.safeTransferFrom(
    market.loanToken, payer, address(this), units       // line 520
);
```

State is mutated first (lines 508–509), then the transfer is executed (line 520). There is no pre/post `balanceOf` check anywhere in the function. `safeTransferFrom` is called with the nominal `units` amount; if the token deducts a fee in-flight, the contract receives `units - fee` but `withdrawable` has already been increased by `units`.

**Market creation — no `loanToken` validation:**

`touchMarket()` (lines 754–791) validates collateral params, maturity, LLTV, and maxLif, but performs **zero validation on `market.loanToken`**. Any address is accepted. `touchMarket` is `public` and callable by any unprivileged user.

**Documented assumption, not enforced on-chain:**

The NatSpec at lines 133–140 states: *"Midnight must send/receive exactly the requested amount on transfers."* This is a comment-level assumption with no corresponding `require`, no balance snapshot, and no revert path. The Certora solvency spec (line 31 of `certora/specs/Solvency.spec`) also explicitly models tokens as non-fee-taking — meaning the formal verification does not cover this case.

**Exploit flow:**

1. Attacker (unprivileged market creator) deploys a fee-on-transfer ERC-20 with a 1% transfer fee and calls `touchMarket(market)` where `market.loanToken = feeToken`.
2. A lender supplies credit into the market (via `take` as buyer), establishing `withdrawable = 0` and `totalUnits = D`.
3. A borrower (attacker or any user) accumulates debt `D` in the market.
4. Borrower calls `repay(market, D, onBehalf, address(0), "")`:
   - Line 508: `debt[onBehalf] -= D` → debt zeroed.
   - Line 509: `withdrawable += D` → withdrawable = D.
   - Line 520: contract receives `D * 0.99` (fee deducted by token).
   - Net: `withdrawable = D`, `loanToken.balanceOf(midnight) = D * 0.99`.
5. Lender calls `withdraw(market, D, lender, lender)`:
   - Line 494: `withdrawable -= D` → withdrawable = 0.
   - Line 499: `safeTransfer(loanToken, lender, D)` → **reverts** if balance < D, or if another lender already withdrew, the last lender gets nothing.
6. After N repayments: `withdrawable = N * D`, `loanToken.balanceOf(midnight) = N * D * 0.99`. The shortfall `N * D * 0.01` is unrecoverable — lenders collectively cannot withdraw the full credited amount.

**Why existing checks fail:**

- The authorization check (`require(onBehalf == msg.sender || isAuthorized[...])`) only gates who can repay on behalf of whom; it does not validate token behavior.
- `SafeTransferLib.safeTransferFrom` only checks that the call does not revert and (if present) returns `true`; it does not verify the received amount.
- No invariant check, no balance snapshot, no post-transfer assertion exists anywhere in `repay()`.

### Impact Explanation
Every `repay()` call with a fee-on-transfer `loanToken` permanently inflates `marketState[id].withdrawable` beyond the contract's actual token balance by `units * fee_rate`. Lenders calling `withdraw()` will either revert (if the shortfall is large enough) or drain tokens belonging to other markets sharing the same token, causing protocol-wide insolvency. The invariant *"contract token balances cover withdrawable assets"* is broken irreversibly with each repayment.

### Likelihood Explanation
`touchMarket` is permissionless — any address can create a market with any `loanToken`. Fee-on-transfer tokens (e.g., STA, PAXG, tokens with configurable fees) exist on mainnet. The exploit requires no privileged access, no oracle manipulation, and no special timing. It is repeatable: every `repay()` call widens the gap. A single market with a fee-on-transfer loanToken is sufficient to trigger the insolvency condition.

### Recommendation
Measure the actual received amount using a pre/post `balanceOf` snapshot in `repay()` and credit only the delta to `withdrawable`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
require(received == units, FeeOnTransferToken());
```

Alternatively, add an on-chain token allowlist or enforce the no-fee-on-transfer invariant at market creation time in `touchMarket()` by performing a probe transfer and balance check. The current comment-only assumption at lines 133–140 provides no runtime protection.

    function touchMarket(Market memory market) public returns (bytes32) {
        bytes32 id = toId(market);
        if (marketState[id].tickSpacing == 0) {
            require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
            require(market.collateralParams.length > 0, NoCollateralParams());
            require(market.collateralParams.length <= MAX_COLLATERALS, TooManyCollateralParams());
            address previousCollateralToken;
            for (uint256 i = 0; i < market.collateralParams.length; i++) {
                address collateralToken = market.collateralParams[i].token;
                require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
                uint256 lltv = market.collateralParams[i].lltv;
                require(isLltvAllowed(lltv), LltvNotAllowed());
                require(
                    market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_LOW)

                    InvalidMaxLif()
