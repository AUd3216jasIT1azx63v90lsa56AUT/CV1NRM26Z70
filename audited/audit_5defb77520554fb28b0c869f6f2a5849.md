### Title
Fee-on-Transfer Loan Token Inflates `withdrawable`, Causing Last Lender's `withdraw()` to Revert - (`src/Midnight.sol` / `src/libraries/SafeTransferLib.sol`)

### Summary
In `liquidate()` and `repay()`, `_marketState.withdrawable` is incremented by the nominal `repaidUnits`/`units` before `safeTransferFrom` is called. `SafeTransferLib.safeTransferFrom` only verifies the call succeeded and returned `true`; it does not measure the actual balance delta. When `market.loanToken` is a fee-on-transfer token, the contract receives `repaidUnits - fee` but records `repaidUnits` as withdrawable, creating a permanent shortfall. Repeated calls accumulate the gap until the last lender's `withdraw()` reverts due to insufficient token balance.

### Finding Description

**Exact code path — `liquidate()`:**

`src/Midnight.sol` lines 675–676 update state unconditionally before the transfer:
```solidity
_marketState.withdrawable += UtilsLib.toUint128(repaidUnits);   // L675
_position.debt -= UtilsLib.toUint128(repaidUnits);              // L676
``` [1](#0-0) 

Then, at line 717, the actual pull happens:
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only checks call success and the boolean return value — it never measures `balanceOf` before/after: [3](#0-2) 

The identical pattern exists in `repay()`: [4](#0-3) 

**Market creation has no loan-token type restriction.** `touchMarket` validates collateral ordering, LLTV tiers, and maxLif, but imposes no constraint on `market.loanToken`: [5](#0-4) 

**`withdraw()` sends the full recorded units:**
```solidity
_marketState.withdrawable -= UtilsLib.toUint128(units);
SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
``` [6](#0-5) 

**Exploit flow:**
1. Attacker (or anyone) creates a market with a fee-on-transfer token as `loanToken` (fee = `f` per transfer).
2. Lender supplies credit; borrower takes debt.
3. Borrower becomes unhealthy (oracle price drop).
4. Liquidator calls `liquidate(market, 0, 0, R, borrower, false, receiver, address(0), "")`.
   - `withdrawable += R` (recorded)
   - Contract receives only `R - f` (actual)
   - Gap = `f`
5. Step 4 is repeatable by any liquidator on any unhealthy position, or by any borrower via `repay()`.
6. After `N` such calls, `withdrawable` exceeds `loanToken.balanceOf(midnight)` by `N * f`.
7. Lenders call `withdraw()` in order; the last lender(s) whose cumulative credit exceeds the actual balance receive a revert from `safeTransfer`.

**No existing check stops this.** The Certora spec `liquidateIncreasesWithdrawable` asserts `withdrawableAfter == withdrawableBefore + repaidResult`, which is satisfied by the accounting — but the spec models `safeTransferFrom` as `NONDET`, explicitly ignoring actual token behavior: [7](#0-6) 

### Impact Explanation
`marketState[id].withdrawable` permanently exceeds `loanToken.balanceOf(address(midnight))` for the affected market. Every `withdraw()` call drains the real balance; the last lender(s) whose aggregate credit exceeds the remaining balance will have their `withdraw()` revert (or, if the token silently short-transfers, receive fewer tokens than their credit). This is protocol insolvency scoped to any market whose `loanToken` charges a transfer fee.

### Likelihood Explanation
Markets are permissionlessly created via `touchMarket` — any address can deploy a market with a fee-on-transfer loan token. Fee-on-transfer ERC20s (e.g., tokens with deflationary mechanics or explicit transfer taxes) are a well-known token class. The preconditions (unhealthy borrower, liquidator with balance) are normal operating conditions. The bug is triggered by every `liquidate` or `repay` call and is therefore repeatable and cumulative.

### Recommendation
Measure the actual received amount using a balance-before/after pattern in both `liquidate()` and `repay()`, and use the measured delta (not the nominal `repaidUnits`/`units`) for all accounting updates:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
// use `received` instead of `repaidUnits` for withdrawable and debt updates
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens by adding a validation in `touchMarket` (e.g., a round-trip transfer check).

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry stateful fuzz / invariant test sketch
contract FeeOnTransferLoanToken is ERC20 {
    uint256 public constant FEE_BPS = 100; // 1% fee
    function _transfer(address from, address to, uint256 amount) internal override {
        uint256 fee = amount * FEE_BPS / 10000;
        super._transfer(from, to, amount - fee);
        // fee stays with `from` (burned or kept)
    }
}

contract InvariantFeeOnTransfer is Test {
    Midnight midnight;
    FeeOnTransferLoanToken loanToken;
    Market market;

    function setUp() public {
        loanToken = new FeeOnTransferLoanToken();
        // deploy midnight, create market with loanToken, supply collateral, take debt
        // make borrower unhealthy via oracle price drop
    }

    // Invariant: contract balance must always cover withdrawable
    function invariant_balanceCoversWithdrawable() public {
        bytes32 id = midnight.toId(market);
        assertGe(
            loanToken.balanceOf(address(midnight)),
            midnight.withdrawable(id),
            "balance < withdrawable: insolvency"
        );
    }

    // Fuzz: call liquidate with repaidUnits > 0, assert invariant holds
    function testFuzz_liquidateFeeOnTransfer(uint256 repaidUnits) public {
        repaidUnits = bound(repaidUnits, 1, borrowerDebt);
        uint256 balBefore = loanToken.balanceOf(address(midnight));
        uint256 wBefore = midnight.withdrawable(id);
        midnight.liquidate(market, 0, 0, repaidUnits, borrower, false, address(this), address(0), "");
        uint256 balAfter = loanToken.balanceOf(address(midnight));
        uint256 wAfter = midnight.withdrawable(id);
        // This assertion FAILS: wAfter - wBefore > balAfter - balBefore by the fee amount
        assertEq(wAfter - wBefore, balAfter - balBefore, "withdrawable delta != balance delta");
    }
}
```

**Expected assertion failure:** `wAfter - wBefore` equals `repaidUnits` but `balAfter - balBefore` equals `repaidUnits * 99 / 100`, confirming the `fee`-sized gap accumulates with each call.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
