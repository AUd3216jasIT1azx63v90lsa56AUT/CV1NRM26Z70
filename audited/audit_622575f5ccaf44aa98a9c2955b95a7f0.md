### Title
Maker Front-Running Can Cause `OutOfOffers()` DoS in `MidnightBundles` Bundle Functions — (File: `src/periphery/MidnightBundles.sol`)

---

### Summary

All four bundle functions in `MidnightBundles.sol` use `try/catch` to skip individual failed `take()` calls, but enforce a hard exact-equality target check at the end (`filledUnits == targetUnits` or equivalent). A malicious maker can front-run a bundle transaction by canceling their offer on-chain, causing the `try/catch` to silently skip their offer and the final target check to revert with `OutOfOffers()` — denying the taker's transaction entirely.

---

### Finding Description

**Root cause — exact target enforcement after silent skipping:**

All four bundle functions follow the same pattern:

```solidity
// try/catch silently skips failed takes
try IMidnight(MIDNIGHT).take(...) returns (...) {
    filledUnits += unitsToTake;
    ...
} catch {}

// hard exact-equality check at the end
require(filledUnits == targetUnits, OutOfOffers());
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

**Attacker-controlled cancellation vectors:**

A maker can invalidate their in-flight offer via any of the following permissionless calls:

1. **`setConsumed(group, type(uint256).max, maker)`** — sets consumed to `type(uint256).max`, causing `take()` to revert with `ConsumedAssets()` / `ConsumedUnits()` (caught by `try/catch`, offer silently skipped): [5](#0-4) 

2. **`cancelRoot(maker, root)`** in `EcrecoverRatifier` — marks the Merkle root as canceled, causing `isRatified` to revert with `RootCanceled()` (caught by `try/catch`): [6](#0-5) 

3. **`setIsAuthorized(ratifier, false, maker)`** — revokes the ratifier's authorization, causing `take()` to revert with `RatifierUnauthorized()` at: [7](#0-6) 

**Exploit flow:**

1. Taker constructs a bundle with `N` offers, exactly enough to reach `targetUnits`.
2. Maker observes the pending transaction in the mempool.
3. Maker front-runs with `setConsumed(group, type(uint256).max, maker)` or `cancelRoot(maker, root)`.
4. Bundle executes: the maker's `take()` reverts, `try/catch` silently skips it, `filledUnits` falls short of `targetUnits`.
5. `require(filledUnits == targetUnits, OutOfOffers())` reverts the entire transaction.

---

### Impact Explanation

The taker's bundle transaction is reverted entirely. Any collateral supplied earlier in the same transaction (e.g., in `supplyCollateralAndSellWithUnitsTarget`) is also reverted, so no funds are permanently lost. However:

- The taker wastes gas on every failed attempt.
- A malicious maker can **repeatedly** front-run to permanently block a specific taker from executing trades through the bundler.
- In time-sensitive scenarios (e.g., a borrower trying to sell units to avoid liquidation), repeated failures can have indirect financial consequences.

---

### Likelihood Explanation

- **No privileged access required**: any maker whose offer appears in a bundle can execute this attack.
- **Trivially front-runnable**: `setConsumed` and `cancelRoot` are cheap, single-call transactions that can be submitted with higher gas to reliably land before the bundle.
- **Repeatable**: the attacker can re-post an offer, wait for the taker to include it in a new bundle, and cancel again indefinitely.
- The attack is most effective when the taker's bundle has tight liquidity (few backup offers), which is common in low-liquidity markets.

---

### Recommendation

Replace the exact-equality target check with a minimum-fill check, and accept partial fills:

```solidity
// Instead of:
require(filledUnits == targetUnits, OutOfOffers());

// Use:
require(filledUnits >= minUnits, OutOfOffers());
```

Alternatively, expose a `minFilled` parameter to the caller so they can specify acceptable slippage, rather than requiring exact fulfillment. This mirrors the recommendation from the original CrabNetting report: orders that fail for any reason should be skipped, and the transaction should succeed as long as a minimum acceptable fill is achieved.

---

### Proof of Concept

**Setup**: A market exists. Maker M has signed an offer included in a bundle. The bundle has exactly enough offers to reach `targetUnits`, with M's offer providing the last `X` units.

**Steps**:
1. Taker T calls `buyWithUnitsTargetAndWithdrawCollateral(targetUnits, maxBuyerAssets, taker, ..., takes, ...)`.
2. Maker M observes the pending tx and front-runs: `midnight.setConsumed(offer.group, type(uint256).max, M)`.
3. Bundle executes: all other offers fill `targetUnits - X` units. M's `take()` reverts with `ConsumedUnits()`, caught by `try/catch`. `filledUnits = targetUnits - X`.
4. `require(filledUnits == targetUnits, OutOfOffers())` → **reverts**.
5. T's transaction fails. M repeats on every retry. [8](#0-7) [9](#0-8)
