### Title
Unconditional zero-value `safeTransfer` refund reverts entire bundle when loan token rejects zero transfers - (`src/periphery/MidnightBundles.sol`)

### Summary

In `buyWithUnitsTargetAndWithdrawCollateral`, after all takes succeed, the bundler unconditionally calls `SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - filledBuyerAssets - referralFeeAssets)` with no zero-value guard. When a taker sets `maxBuyerAssets` exactly equal to `filledBuyerAssets + referralFeeAssets`, the refund amount is zero, and any loan token that reverts on zero-value ERC-20 transfers will cause the entire transaction to revert — undoing all successful takes. The referral fee transfer on the immediately preceding line is correctly guarded with `if (referralFeeAssets > 0)`, making the omission on the refund line an inconsistency and a real bug.

### Finding Description

**Code path:**

`MidnightBundles.sol` lines 102–104:
```solidity
uint256 referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets); // guarded
SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - filledBuyerAssets - referralFeeAssets); // NOT guarded
```

`SafeTransferLib.safeTransfer` (lines 12–22) unconditionally issues a low-level `token.call(abi.encodeCall(IERC20.transfer, (to, value)))`. If `value == 0` and the token reverts on zero-value transfers, `success` is `false`, and the assembly block re-reverts with the token's revert data, propagating the revert up through the entire call.

**Attacker-controlled inputs:**
- `maxBuyerAssets`: caller-supplied; the taker sets this to exactly the amount they expect to spend.
- `referralFeePct`: caller-supplied; can be 0 (making `referralFeeAssets = 0`) or any value such that `filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct)` equals `maxBuyerAssets - filledBuyerAssets`.

**Exploit flow:**
1. Taker calls `buyWithUnitsTargetAndWithdrawCollateral` with a loan token that reverts on zero-value transfers.
2. Taker sets `maxBuyerAssets` = exact cost of the takes (e.g., `filledBuyerAssets` when `referralFeePct == 0`).
3. All takes succeed; `filledBuyerAssets` accumulates correctly.
4. `referralFeeAssets` is computed; the guarded transfer on line 103 is skipped (or executes if fee > 0).
5. Line 104 computes `maxBuyerAssets - filledBuyerAssets - referralFeeAssets = 0` and calls `safeTransfer(..., 0)`.
6. The token reverts; the entire transaction reverts, undoing all takes.

**Why existing checks do not stop it:**
- There is no `> 0` guard on line 104, unlike line 103.
- The contract NatSpec on line 26 states "Zero checks are not systematically performed," but the referral fee line contradicts this by having a guard, making the missing guard on the refund line a defect rather than a design choice.
- `SafeTransferLib.safeTransfer` has no zero-value short-circuit; it always issues the external call.

### Impact Explanation

When `maxBuyerAssets == filledBuyerAssets + referralFeeAssets`, the entire bundle transaction reverts despite all takes having succeeded. The taker loses gas and cannot complete the intended buy-and-withdraw operation. For time-sensitive operations (e.g., filling before a deadline, capturing a specific price), the revert has concrete economic cost beyond gas. The bug is triggered by normal, intended usage (setting `maxBuyerAssets` to the exact expected cost).

### Likelihood Explanation

**Preconditions:**
1. The loan token reverts on zero-value transfers. This is a known behavior in several ERC-20 tokens (e.g., tokens with explicit `require(amount > 0)` guards).
2. The taker sets `maxBuyerAssets` exactly equal to the total cost. This is the natural behavior of any frontend or contract that computes the exact cost before calling.

Both preconditions are independently reachable without any privileged access, oracle manipulation, or user mistake. The scenario is repeatable on every call with a matching token and exact `maxBuyerAssets`.

### Recommendation

Add a zero-value guard on the refund transfer, mirroring the guard already present on line 103:

```solidity
uint256 refund = maxBuyerAssets - filledBuyerAssets - referralFeeAssets;
if (refund > 0) SafeTransferLib.safeTransfer(loanToken, msg.sender, refund);
```

This is consistent with the existing pattern on line 103 and eliminates the revert for the zero-refund case.

        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
