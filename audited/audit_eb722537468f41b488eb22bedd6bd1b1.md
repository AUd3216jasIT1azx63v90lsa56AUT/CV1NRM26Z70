Audit Report

## Title
Missing authorization check on `callback` parameter allows arbitrary third-party payer designation - (File: src/Midnight.sol)

## Summary
`Midnight.liquidate` sets `payer = callback` when `callback != address(0)` at line 679 with no requirement that `callback == msg.sender` or that `callback` has authorized `msg.sender` via `isAuthorized`. An attacker can designate any contract implementing `ILiquidateCallback` that returns `CALLBACK_SUCCESS` and holds a loanToken approval to Midnight as the `callback`/`payer`, causing the protocol to pull `repaidUnits` of loanToken from that victim contract while the attacker's `receiver` collects the seized collateral.

## Finding Description
**Exact code path:**

`src/Midnight.sol` line 679:
```solidity
address payer = callback != address(0) ? callback : msg.sender;
```
`src/Midnight.sol` lines 698–717:
```solidity
if (callback != address(0)) {
    require(
        ILiquidateCallback(callback).onLiquidate(msg.sender, ...) == CALLBACK_SUCCESS,
        WrongLiquidateCallbackReturnValue()
    );
}
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```

**Root cause:** No check of the form `require(callback == msg.sender || isAuthorized[callback][msg.sender])` exists anywhere in `liquidate`. The only gate on `callback` is that it must return `CALLBACK_SUCCESS` from `onLiquidate`, which is satisfied by any conforming callback contract regardless of who invoked `liquidate`.

**Exploit flow:**
1. Precondition: `victimContract` has called `loanToken.approve(midnight, N)` and its `onLiquidate` returns `CALLBACK_SUCCESS` (e.g., it checks only `msg.sender == midnight`, which is satisfied since Midnight is the caller).
2. Attacker calls `midnight.liquidate(market, idx, 0, repaidUnits, borrower, false, attackerEOA, victimContract, data)`.
3. `payer = victimContract` (line 679).
4. `seizedAssets` of collateral transferred to `attackerEOA` (line 696).
5. `victimContract.onLiquidate(attacker, ...)` called — returns `CALLBACK_SUCCESS` (lines 700–714).
6. `safeTransferFrom(loanToken, victimContract, midnight, repaidUnits)` executes (line 717) — pulls from victim.
7. Net: attacker receives collateral worth `repaidUnits × LIF`, pays nothing; victim loses `repaidUnits` of loanToken.

**Why existing checks fail:**
- The `liquidatorGate` check (line 598) gates who can call `liquidate`, not what `callback` is.
- The Certora spec `otherEntryPointsOnlyPullFromCaller` (line 128) sets `liquidateCallbackAllowed = true` unconditionally for any address passed as callback to `liquidate`. It proves the weaker property "tokens only leave from a callback that returned `CALLBACK_SUCCESS`," not "callback must be `msg.sender` or authorized by `msg.sender`."
- The `isAuthorized` mapping is used in `setConsumed` (line 724) and `setIsAuthorized` (line 732) but is never consulted in `liquidate` for the `callback` parameter.

## Impact Explanation
Any contract that (a) implements `ILiquidateCallback` returning `CALLBACK_SUCCESS` and (b) holds a loanToken approval to Midnight can have its entire approved balance drained by an unprivileged attacker in a single `liquidate` call. The attacker receives seized collateral at zero personal cost. This directly violates the protocol invariant that liquidation must not transfer assets from unauthorized parties, and constitutes unauthorized token loss for the victim.

## Likelihood Explanation
Preconditions are routinely satisfied in production: (1) unhealthy or post-maturity borrowers exist as a normal market condition; (2) deployed flash-liquidation bots and callback contracts pre-approve Midnight for loanToken to enable the pull after `onLiquidate`, and many check only `msg.sender == midnight` without validating that the collateral receiver is themselves or that the liquidation is profitable for them. The attacker requires no special privilege, no oracle manipulation, and no governance access. The attack is repeatable as long as the victim's approval persists.

## Recommendation
Add an authorization check on `callback` before using it as `payer`:

```solidity
require(
    callback == address(0) || callback == msg.sender || isAuthorized[callback][msg.sender],
    Unauthorized()
);
address payer = callback != address(0) ? callback : msg.sender;
```

This mirrors the pattern already used in `setConsumed` and `setIsAuthorized`, ensuring that only the caller or an address the caller has been explicitly authorized to act on behalf of can be designated as the loanToken payer.

## Proof of Concept
1. Deploy `VictimCallback` implementing `ILiquidateCallback`: `onLiquidate` checks `msg.sender == midnight` only and returns `CALLBACK_SUCCESS`; constructor calls `loanToken.approve(midnight, type(uint256).max)`.
2. Fund `VictimCallback` with loanToken.
3. Create an unhealthy borrower position in any market.
4. Call `midnight.liquidate(market, idx, 0, repaidUnits, borrower, false, attackerEOA, address(victimCallback), "")` from an unprivileged EOA.
