### Title
Buy-offer `maxAssets` cap bypassed when `buyerAssets` rounds to zero, minting maker credit without token inflow - (`src/Midnight.sol`)

### Summary

When `offer.buy == true` and `offer.maxAssets > 0`, the consumed-offer cap is tracked by adding `buyerAssets` to `consumed[maker][group]`. Because `buyerAssets` is computed with `mulDivDown` and can round to zero for small `units` at a low tick, the consumed counter never advances, the cap check always passes, and the take executes full position state changes — including a `buyerCreditIncrease` of `units` — while transferring zero tokens. The maker's credit is minted without any corresponding token inflow, and the offer can be replayed indefinitely.

### Finding Description

**Exact code path — `src/Midnight.sol` `take`:**

```
Line 363: buyerAssets = units.mulDivDown(buyerPrice, WAD)   // rounds to 0 when units*buyerPrice < WAD
Line 367-369:
    if (offer.maxAssets > 0) {
        newConsumed = consumed[maker][group] += offer.buy ? buyerAssets : sellerAssets;
        // += 0 → newConsumed unchanged → passes even if already at cap
        require(newConsumed <= offer.maxAssets, ConsumedAssets());
    }
Line 382: buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)
          // = units when buyerPos.debt == 0
Line 410: buyerPos.credit += buyerCreditIncrease   // credit minted
Line 455: safeTransferFrom(token, payer, this, buyerAssets - sellerAssets)  // transfers 0
Line 456: safeTransferFrom(token, payer, receiver, sellerAssets)            // transfers 0
```

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0` (any positive value, including 1)
- `offer.tick` set to a very low value (e.g., `tick = 0` → `offerPrice = 1 wei`) so that `units * buyerPrice < WAD`
- `units` chosen such that `units.mulDivDown(buyerPrice, WAD) == 0` (e.g., `units = 1` at `tick = 0` with no settlement fee)
- `buyerPos.debt == 0` (maker has no existing debt, so `buyerCreditIncrease = units`)

**Why existing checks fail:**
- The `ConsumedAssets` check at line 369 compares `newConsumed <= offer.maxAssets`. When `buyerAssets == 0`, `newConsumed` is unchanged, so the check passes regardless of how many times the offer has been taken or whether it was already fully consumed.
- There is no `require(buyerAssets > 0 || units == 0)` guard.
- There is no check that `buyerCreditIncrease == 0` when `buyerAssets == 0`.

**The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) explicitly demonstrates this:** the offer is pre-consumed to its full `maxAssets = 1` cap, then `take(1, borrower, lenderOffer)` succeeds with `buyerAssets = 0`, `sellerAssets = 0`, zero token transfers, yet `creditOf(id, lender)` increases and `debtOf(id, borrower)` increases.

### Impact Explanation

The maker (lender) accumulates credit — a claim on the protocol's loan token pool — without depositing any tokens. Every unit of unbacked credit is a direct shortfall: when the maker later calls `withdraw`, they drain tokens that belong to other depositors. The offer can be replayed an unlimited number of times (consumed counter never advances), so the unbacked credit can be inflated to any amount. This is a direct loss to the protocol's solvency.

### Likelihood Explanation

**Preconditions:**
1. A buy offer exists with `offer.maxAssets > 0` and a tick low enough that `units * buyerPrice < WAD` (tick = 0 with no settlement fee is sufficient; `buyerPrice = 1 wei`).
2. The maker has no existing debt (`buyerPos.debt == 0`), which is the normal state for a lender.
3. Any unprivileged taker can call `take` directly.

All three conditions are easily satisfied and require no special privileges. The attack is repeatable in a single transaction via multicall or a loop, and is not blocked by any slippage, health, or authorization check.

### Recommendation

Add a guard that prevents a take from proceeding with non-zero `units` when `buyerAssets == 0` in assets-cap mode:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetsTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, track consumed in units (not assets) when `offer.maxAssets > 0` and convert the cap to units at offer creation, or require `units == 0` whenever both `buyerAssets` and `sellerAssets` are zero and `offer.maxAssets > 0`.

### Proof of Concept

```solidity
function testBuyMaxAssetsBypassCreditMintedWithoutTransfer() public {
    // Setup: buy offer (maker=lender), maxAssets=1, tick=0 (price=1 wei), no settlement fee
    midnight.setMarketSettlementFee(id, 1, 0); // ensure fee=0
    lenderOffer.buy = true;
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = 0; // buyerPrice = 1 wei → units*1/1e18 = 0 for units < 1e18

    collateralize(market, borrower, 100);
    deal(address(loanToken), lender, 0); // lender pays nothing

    uint256 creditBefore = midnight.creditOf(id, lender);
    uint256 balBefore = loanToken.balanceOf(lender);

    // Take with units=1: buyerAssets = 1*1/1e18 = 0 (rounds down)
    (uint256 buyerAssets,) = take(1, borrower, lenderOffer);

    // Assertions:
    assertEq(buyerAssets, 0, "buyerAssets must be 0");
    assertEq(loanToken.balanceOf(lender), balBefore, "no token transferred");
    assertGt(midnight.creditOf(id, lender), creditBefore, "credit minted without payment");
    // Offer cap not consumed — can repeat:
    assertEq(midnight.consumed(lender, lenderOffer.group), 1); // unchanged from pre-consume
    // Repeat take — offer cap still not exhausted
    take(1, borrower, lenderOffer);
    assertGt(midnight.creditOf(id, lender), creditBefore + 1, "credit keeps growing");
}
```
