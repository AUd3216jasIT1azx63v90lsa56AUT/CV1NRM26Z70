### Title
Buy offer `maxAssets` cap bypassed via zero-rounding `consumed` accounting when `buyerPrice < WAD` — (`src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and `offerPrice < WAD` (any tick below `MAX_TICK`), calling `take()` with `units = 1` produces `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. Because `consumed` is incremented by `buyerAssets`, it never increases, the `ConsumedAssets` guard always passes, and an unprivileged taker can call `take()` indefinitely. Each call still mutates position state — the maker accumulates credit and the taker accumulates debt — with no asset transfer and no cap enforcement.

### Finding Description
**Exact code path** — `src/Midnight.sol` lines 358–373:

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);          // line 358
// For offer.buy: buyerPrice == offerPrice (settlement fee cancels)
uint256 buyerPrice = sellerPrice + _settlementFee;             // line 362
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)                        // line 363 — rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group]
        += offer.buy ? buyerAssets : sellerAssets;             // line 368 — += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets()); // line 369 — always passes
}
```

`mulDivDown` is `(x * y) / d` (plain integer division, rounds toward zero). [1](#0-0) 

For a buy offer, `buyerPrice = offerPrice = TickLib.tickToPrice(offer.tick)`. `tickToPrice` returns values in the range `(0, WAD]`; only tick `MAX_TICK = 5820` yields exactly `WAD`. Every other valid tick yields `offerPrice < WAD`. [2](#0-1) 

Therefore, for any tick `< MAX_TICK`:
```
mulDivDown(1, buyerPrice, WAD) = (1 * buyerPrice) / 1e18 = 0
```
`consumed` is incremented by 0, `newConsumed` is unchanged, the `require` passes, and the take succeeds. Position state is still fully updated: [3](#0-2) 

```solidity
buyerPos.credit += toUint128(buyerCreditIncrease);   // lender gains 1 unit of credit
sellerPos.debt   += toUint128(sellerDebtIncrease);   // borrower gains 1 unit of debt
```

No assets are transferred (`buyerAssets = sellerAssets = 0`), no settlement fee is collected, and `consumed` stays frozen. The taker can repeat this call an unbounded number of times.

**Attacker-controlled inputs:** `units = 1`, any valid buy offer with `tick < MAX_TICK` and `maxAssets > 0`. No special role required — any address that is not `offer.maker` can be the taker.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — never triggers because `newConsumed` never increases. [4](#0-3) 
- There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in `take()`.
- The protocol's own NatSpec at line 94 acknowledges the symptom: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* [5](#0-4) 

The existing test `testBugBuyMaxAssetsBypass` (line 858) explicitly reproduces the scenario — it pre-fills `consumed` to `maxAssets`, then calls `take(1, borrower, lenderOffer)` and asserts that `creditOf(lender)` and `debtOf(borrower)` both increase while `consumed` stays at `maxAssets` and no tokens move. [6](#0-5) 

### Impact Explanation
The `maxAssets` cap — the maker's primary mechanism to bound their total exposure — is rendered ineffective for any buy offer at a tick below `MAX_TICK`. A taker can:
1. Drive the maker's credit position to an arbitrary size without the maker spending any assets.
2. Accumulate their own debt position of arbitrary size without paying any assets.
3. Violate the core invariant that `consumed[maker][group]` must increase by at least the fill's contribution on every successful non-zero-unit take.

The maker's credit exposure grows unboundedly, increasing their susceptibility to bad-debt socialization losses beyond what they consented to. The taker gains debt-denominated positions for free, which could be used to extract collateral or manipulate market state.

### Likelihood Explanation
The precondition is trivially satisfied: any buy offer with `maxAssets > 0` and any tick in `[0, MAX_TICK)` — which is the entire practical tick range — is vulnerable. The attack requires no privileged access, no oracle manipulation, and no special token behavior. It is repeatable in a single transaction via `multicall`. Gas cost on cheap chains makes griefing or position inflation economically viable.

### Recommendation
Add a minimum-assets guard before the consumed update for the `maxAssets` branch. When `units > 0` and `offer.maxAssets > 0`, require that the computed assets value is at least 1:

```solidity
uint256 assetsForConsumed = offer.buy ? buyerAssets : sellerAssets;
if (offer.maxAssets > 0) {
    require(units == 0 || assetsForConsumed > 0, ZeroAssetsOnNonZeroUnits());
    newConsumed = consumed[offer.maker][offer.group] += assetsForConsumed;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, use `mulDivUp` instead of `mulDivDown` when computing `buyerAssets` for the consumed accounting path (keeping `mulDivDown` for the actual transfer amount), so that `consumed` always increases by at least 1 for any non-zero `units` fill.

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already constitutes a passing unit-test PoC. A complementary fuzz/invariant test:

```solidity
// Foundry fuzz test
function testFuzz_consumedAlwaysIncreasesOnNonZeroUnitsBuyMaxAssets(
    uint256 tick,
    uint256 units
) public {
    tick = bound(tick, 0, MAX_TICK - 1); // any tick below MAX_TICK
    units = bound(units, 1, 1e6);

    lenderOffer.tick = tick;
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = type(uint128).max;

    uint256 consumedBefore = midnight.consumed(lender, lenderOffer.group);
    deal(address(loanToken), lender, type(uint128).max);
    collateralize(market, borrower, units);

    take(units, borrower, lenderOffer);

    uint256 consumedAfter = midnight.consumed(lender, lenderOffer.group);
    // ASSERTION: consumed must increase by at least 1 on every successful non-zero take
    assertGt(consumedAfter, consumedBefore, "consumed did not increase");
}
```

**Expected result (current code):** assertion fails for any `tick` where `units * tickToPrice(tick) < WAD` (i.e., `tick < MAX_TICK` with `units = 1`).
**Expected result (after fix):** either the assertion holds, or `take()` reverts with `ZeroAssetsOnNonZeroUnits`.
