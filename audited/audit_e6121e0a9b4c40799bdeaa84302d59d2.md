Audit Report

## Title
`liquidatorGate` access control bypassed via `callback`/`receiver` payer substitution - (File: src/Midnight.sol)

## Summary
The `liquidate` function checks `liquidatorGate.canLiquidate` only against `msg.sender`, but when `callback != address(0)` the actual loan token repayment is pulled from the `callback` address and collateral is sent to the caller-controlled `receiver`. A whitelisted address A can call `liquidate(callback=B, receiver=B)`, causing gate-blocked address B to be the true economic actor — paying loan tokens and receiving seized collateral — while the gate never inspects B.

## Finding Description
**Gate check** (`src/Midnight.sol` lines 597–600): only `msg.sender` is evaluated by `canLiquidate`: [1](#0-0) 

**Payer substitution** (line 679): when `callback` is non-zero, the payer becomes the callback address, not `msg.sender`: [2](#0-1) 

**Collateral transfer** (line 696): collateral is sent to caller-supplied `receiver` with no gate check: [3](#0-2) 

**Loan token pull** (line 717): tokens are pulled from `payer` (the callback address B), not `msg.sender`: [4](#0-3) 

**Exploit flow:**
1. Market has `liquidatorGate` that whitelists A and blocks B.
2. B deploys a contract implementing `ILiquidateCallback`, pre-approves Midnight to pull loan tokens from it.
3. A calls `midnight.liquidate(market, ..., receiver=B, callback=B, ...)`.
4. Gate calls `canLiquidate(A)` → `true` → passes.
5. Collateral transferred to B (`receiver`).
6. `onLiquidate` called on B; B returns `CALLBACK_SUCCESS`.
7. Loan tokens pulled from B (`payer=B`).
8. B has fully executed the liquidation despite being gate-blocked.

**Why existing checks fail:** The Certora rule `liquidatorGateBlocksLiquidation` (`certora/specs/Reverts.spec` lines 325–332) only proves that a call reverts when `ghostCanLiquidate(market.liquidatorGate)` is false — it does not constrain what address is passed to `canLiquidate`, and does not model the callback address at all: [5](#0-4) 

`OnlyExplicitPayerCanLoseTokens.spec` line 128 explicitly sets `liquidateCallbackAllowed = true` for `liquidate`, confirming the callback-as-payer path is intentional but the gate gap is unaddressed: [6](#0-5) 

## Impact Explanation
The `liquidatorGate` access control is fully circumvented for the actual economic actor. Any market relying on the gate for KYC, regulatory compliance, or trust-based liquidator restrictions has those restrictions rendered ineffective. Address B receives seized collateral and bears the loan repayment cost — the full economic outcome of a liquidation — without ever being checked by the gate. This constitutes unauthorized movement of assets (collateral) to a gate-blocked address and unauthorized participation in a gated liquidation market.

## Likelihood Explanation
Requires whitelisted address A to cooperate with blocked address B, or for A to be a public/permissionless liquidation router contract (common in DeFi). In the router case, any external user can supply `callback=B` and `receiver=B` as arguments with no collusion needed from A's operator. Even with a cooperative EOA, the setup is low-friction: B deploys a callback contract, pre-approves Midnight, and asks any whitelisted address to submit the transaction. The condition is repeatable on every liquidatable position in every gated market.

## Recommendation
Extend the gate check to also validate the `callback` and `receiver` addresses when a `liquidatorGate` is set:

```solidity
require(
    market.liquidatorGate == address(0)
        || (ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender)
            && (callback == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(callback))
            && (receiver == msg.sender || ILiquidatorGate(market.liquidatorGate).canLiquidate(receiver))),
    LiquidatorGatedFromLiquidating()
);
```

Alternatively, restrict `receiver` and `callback` to `msg.sender` when a gate is active, or require the gate to explicitly approve all three addresses.

## Proof of Concept
**Minimal fork test plan:**
1. Deploy a mock `ILiquidatorGate` that returns `true` for address A and `false` for address B.
2. Create a market with `liquidatorGate` set to the mock gate.
3. Supply collateral and borrow to create a liquidatable position.
4. Deploy a callback contract at address B that implements `onLiquidate` returning `CALLBACK_SUCCESS` and pre-approves Midnight for loan token transfer.
5. Call `liquidate(market, ..., receiver=B, callback=B, ...)` from address A.
6. Assert: call succeeds, B receives seized collateral, loan tokens are pulled from B — despite `canLiquidate(B)` returning `false`.
