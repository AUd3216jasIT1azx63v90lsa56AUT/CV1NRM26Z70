Audit Report

## Title
Fee-on-Transfer `loanToken` in `repay()` Inflates `withdrawable` Beyond Actual Contract Balance - (File: src/Midnight.sol)

## Summary
`repay()` unconditionally increments `marketState[id].withdrawable` by the full `units` parameter before calling `SafeTransferLib.safeTransferFrom`, which only verifies the call did not revert and returned `true` — never the actual tokens received. When a fee-on-transfer ERC20 is used as `loanToken`, the contract receives `units * (1 - fee_rate)` while recording `units` as withdrawable, permanently inflating accounting. Because `touchMarket()` imposes no restriction on `loanToken` type, any unprivileged actor can create such a market and trigger cumulative insolvency.

## Finding Description
**Root cause:** `repay()` at lines 508–509 of `src/Midnight.sol` performs:
```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);
```
before the actual token transfer at line 520:
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
```
`SafeTransferLib.safeTransferFrom` (lines 24–34 of `src/libraries/SafeTransferLib.sol`) calls `token.transferFrom(from, to, value)` and only checks that the call did not revert and that the return value is `true`. It performs no balance-before/after comparison. A fee-on-transfer token satisfies both conditions while delivering only `units * (1 - f)`.

**Exploit flow:**
1. Attacker calls `touchMarket(market)` with a fee-on-transfer ERC20 as `market.loanToken`. `touchMarket` (lines 755–791) validates only maturity, collateral count, LLTV tiers, and `maxLif` — no check on `loanToken` properties.
2. A borrower acquires debt in that market via `take()`.
3. The borrower calls `repay(market, units, onBehalf, address(0), '')`. `withdrawable` increases by `units`; the contract receives only `units * (1 - f)`. Deficit per call: `units * f`.
4. After N repayments, cumulative deficit is `N * units * f`. When lenders call `withdraw()` (line 499: `SafeTransferLib.safeTransfer(market.loanToken, receiver, units)`), the contract reverts for the last lenders once the real balance is exhausted while `withdrawable` still shows a positive value.

**Existing checks found insufficient:**
- `require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender])` — authorization only, irrelevant to token accounting.
- `SafeTransferLib` — checks revert and boolean return, not received amount.
- No allowlist/denylist on `loanToken` in `touchMarket`.
- No balance-before/after guard anywhere in `repay()`.

## Impact Explanation
Protocol insolvency scoped to the affected market: `marketState[id].withdrawable` permanently exceeds `loanToken.balanceOf(address(midnight))` by the cumulative fee amount. Lenders who call `withdraw()` last receive a revert due to insufficient token balance, resulting in permanent, unrecoverable loss of their entitled loan tokens. The invariant `withdrawable <= actual loanToken balance` is broken and cannot self-correct.

## Likelihood Explanation
Preconditions are minimal and fully unprivileged: (1) any actor can create a market with a fee-on-transfer `loanToken` via `touchMarket()`; (2) any borrower with debt calls `repay()` — normal protocol usage. No governance action, oracle manipulation, or user mistake is required. The deficit is cumulative and widens with every repayment in the affected market.

## Recommendation
Adopt a balance-before/after pattern in `repay()` to measure actual tokens received:
```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
require(received == units, InsufficientTokensReceived());
```
Alternatively, credit `withdrawable` only by `received` instead of `units`. A similar audit should be applied to `take()` and any other function that calls `safeTransferFrom` and credits internal accounting by the requested amount rather than the received amount.

## Proof of Concept
1. Deploy a fee-on-transfer ERC20 with a 1% transfer fee.
2. Call `touchMarket(market)` with this token as `loanToken`.
3. Supply liquidity and have a borrower take debt.
4. Borrower calls `repay(market, 1000e18, borrower, address(0), '')`.
   - `marketState[id].withdrawable` increases by `1000e18`.
   - Contract receives `990e18` (1% fee deducted).
5. Repeat step 4 ten times. `withdrawable` = `10000e18`; actual balance = `9900e18`.
6. Lenders call `withdraw()` sequentially. The last lender's call reverts with insufficient balance while `withdrawable` still shows a positive value, confirming permanent fund loss.
