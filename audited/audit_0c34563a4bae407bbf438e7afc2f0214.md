Audit Report

## Title
Buy-offer `maxAssets` budget bypassed via zero-rounding of `buyerAssets` when `offerPrice < WAD` - (File: src/Midnight.sol)

## Summary
In `take()`, for a buy offer with `maxAssets > 0`, the consumed budget is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `buyerPrice` algebraically equals `offerPrice` for all buy offers, and `offerPrice < WAD` for every tick below `MAX_TICK` (5819 out of 5820 ticks), calling `take()` with `units = 1` yields `buyerAssets = 0`. The `consumed` mapping is never incremented, so the `ConsumedAssets` guard never triggers, allowing unlimited fills past the maker's intended budget cap. The existing test `testBugBuyMaxAssetsBypass` at `test/TakeTest.sol:858` explicitly confirms this behavior.

## Finding Description
**Algebraic identity for buy offers:**

In `src/Midnight.sol`:
- `sellerPrice = offerPrice - _settlementFee` [1](#0-0) 
- `buyerPrice = sellerPrice + _settlementFee = offerPrice` (settlement fee cancels) [2](#0-1) 

**Zero-rounding condition:**

`buyerAssets = units.mulDivDown(buyerPrice, WAD)` for buy offers. [3](#0-2) 

With `units = 1` and `buyerPrice = offerPrice < 1e18`, `mulDivDown(1, offerPrice, 1e18) = 0`.

`MAX_TICK = 5820` and `tickToPrice(MAX_TICK)` returns exactly `1e18`. [4](#0-3)  Every tick from 0 to 5819 yields `offerPrice < WAD`, making `buyerAssets = 0` for `units = 1`. [5](#0-4) 

**Consumed accounting failure:**

```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
// += 0 when buyerAssets rounds to 0
require(newConsumed <= offer.maxAssets, ConsumedAssets());
// passes: consumed + 0 <= maxAssets, even when consumed == maxAssets
``` [6](#0-5) 

Each `take(offer, ..., 1, ...)` call with a fully-consumed offer:
- Increments `buyerPos.credit` and `sellerPos.debt` by 1 unit [7](#0-6) 
- Increments `_marketState.totalUnits` [8](#0-7) 
- Transfers 0 tokens (`buyerAssets - sellerAssets = 0`) [9](#0-8) 
- Leaves `consumed` unchanged

**Test confirmation:**

`testBugBuyMaxAssetsBypass` pre-sets `consumed` to `maxAssets`, calls `take(1, ...)`, and asserts `buyerAssets == 0`, `consumed` unchanged, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased. [10](#0-9) 

## Impact Explanation
Any buy offer with `maxAssets > 0` and `tick < MAX_TICK` can be filled an unlimited number of times with `units = 1` after the budget is fully consumed. The maker's intended exposure cap is completely bypassed: the maker accumulates unbounded credit and the taker accumulates unbounded debt, with zero token transfers per fill. `totalUnits` grows without bound, corrupting market-wide accounting. The core invariant "offer fill amount cannot exceed remaining offer capacity" is broken.

## Likelihood Explanation
The condition `offerPrice < WAD` holds for 5819 out of 5820 possible ticks — essentially all economically meaningful prices below par. No special market state, time manipulation, or privileged access is required. Any unprivileged taker with sufficient collateral to pass `isHealthy` checks can trigger this in a loop within a single transaction. The bug is present at any `block.timestamp` and is repeatable indefinitely.

## Recommendation
Replace the `mulDivDown` rounding with `mulDivUp` for the buy-offer branch of `buyerAssets` when used for consumed accounting, or enforce a minimum increment of 1 when `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 consumedDelta = offer.buy ? buyerAssets : sellerAssets;
    require(consumedDelta > 0 || units == 0, ZeroAssetFill());
    newConsumed = consumed[offer.maker][offer.group] += consumedDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, use `mulDivUp` consistently for the consumed-accounting path (separate from the transfer-amount path) to ensure any nonzero `units` always increments `consumed` by at least 1.

## Proof of Concept
The existing test at `test/TakeTest.sol:858` is a complete, self-contained proof of concept. [10](#0-9)  It:
1. Sets `lenderOffer.maxAssets = 1` and `tick = MAX_TICK - 16` (so `offerPrice < WAD`)
2. Pre-fills `consumed` to `maxAssets` via `setConsumed`
3. Calls `take(1, borrower, lenderOffer)` — succeeds despite the offer being fully consumed
4. Asserts `buyerAssets == 0`, `consumed` unchanged, but credit/debt/totalUnits all increased

    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);
        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
