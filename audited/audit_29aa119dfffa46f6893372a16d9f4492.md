### Title
Fee-on-transfer loan token inflates `claimableSettlementFee`, draining lenders' withdrawable - (`src/libraries/SafeTransferLib.sol` / `src/Midnight.sol`)

### Summary
`take()` increments `claimableSettlementFee[loanToken]` by the full spread (`buyerAssets - sellerAssets`) at line 418 before calling `safeTransferFrom` at line 455. `SafeTransferLib.safeTransferFrom` only validates the boolean return value of `transferFrom` and never checks the actual amount received. When `loanToken` is a fee-on-transfer token, the contract receives `spread - tokenFee` but `claimableSettlementFee` is credited the full `spread`, permanently inflating it by `tokenFee` per `take()`. Subsequent `claimSettlementFee` calls drain the shortfall from lenders' `withdrawable` balance, violating the `tokenBalanceCorrect` solvency invariant.

### Finding Description

**Exact code path:**

`touchMarket` (`src/Midnight.sol:755-791`) imposes no restriction on `market.loanToken` — it only validates maturity, collateral params, LLTV, and maxLif. Any ERC20, including a fee-on-transfer token, is accepted as a loan token. [1](#0-0) 

Inside `take()`, the accounting write happens first: [2](#0-1) 

Then, after callbacks, the inbound transfer is executed: [3](#0-2) 

`SafeTransferLib.safeTransferFrom` calls `transferFrom(from, to, value)` and only checks that the call did not revert and returned `true`. It performs no balance-before/after check and cannot detect that a fee-on-transfer token silently delivered `value - tokenFee` instead of `value`: [4](#0-3) 

**Attacker-controlled inputs:**
- `offer.market.loanToken` — set at market creation by any unprivileged market creator via `touchMarket`.
- `offer.buy = true` — taker is the seller; payer is `offer.maker` (the buyer). The taker triggers the call without paying the settlement fee themselves.
- `units` — controls the magnitude of `buyerAssets - sellerAssets` and thus the per-call inflation.

**Exploit flow (`offer.buy == true`, no buyer callback):**
1. Attacker (market creator) deploys a fee-on-transfer ERC20 and calls `touchMarket` with it as `loanToken`.
2. Legitimate lenders supply credit; `marketState[id].withdrawable` accumulates.
3. A buy offer exists (maker = lender/buyer). Taker calls `take()`.
4. Line 418: `claimableSettlementFee[loanToken] += spread` (e.g., `spread = 1000`).
5. Line 455: `safeTransferFrom(loanToken, payer, address(this), 1000)` — token charges 10 fee; contract receives 990.
6. Post-`take()`: `claimableSettlementFee = 1000`, actual fee balance held = 990. Deficit = 10.
7. Repeated `take()` calls accumulate the deficit linearly.
8. `claimSettlementFee(claimableSettlementFee)` transfers the full inflated amount from the contract, consuming 10 units that belonged to lenders' `withdrawable`.
9. Lenders calling `withdraw()` receive less than their credited amount; the last lenders to withdraw find the contract insolvent.

**Why existing checks fail:**
- `SafeTransferLib` has no received-amount verification.
- `touchMarket` has no token allowlist or fee-on-transfer guard.
- The Certora `pendingFeeReceiptZero` weak invariant and `tokenBalanceCorrect` strong invariant are proven under the assumption that `CVL_transferFrom` delivers exactly the requested amount. The spec comment at line 140-141 explicitly acknowledges the ordering gap but models it as fully closed by the transfer — an assumption that breaks for fee-on-transfer tokens. [5](#0-4) 

### Impact Explanation
Every `take()` on a market whose `loanToken` is fee-on-transfer inflates `claimableSettlementFee` by `tokenFee`. When `claimSettlementFee` is subsequently called (a routine privileged operation), it transfers the inflated amount from the contract, consuming funds that back lenders' `withdrawable`. The `tokenBalanceCorrect` invariant — `balance >= collateralSum + withdrawableSum + claimableSettlementFee` — is violated after each such `take()`. Lenders suffer credit destruction: their on-chain `withdrawable` credit is valid but the underlying token balance is insufficient to honour it, causing insolvency proportional to the cumulative fee-on-transfer deductions.

### Likelihood Explanation
**Preconditions:**
1. `loanToken` is a fee-on-transfer ERC20 — deployable by any unprivileged market creator at zero cost.
2. `offer.buy == true` and settlement fee > 0 — the default settlement fee schedule is non-zero for non-zero TTM markets.
3. A buy offer exists with an approved payer — normal market operation.

All three preconditions are reachable without any privileged action. The attack is repeatable on every `take()` call and scales with trade volume. No oracle manipulation, admin key, or user mistake is required.

### Recommendation
Add a balance-before/after check in `take()` around the inbound settlement-fee transfer to ensure the contract actually received the expected amount:

```solidity
// Before line 455
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
require(
    IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore == buyerAssets - sellerAssets,
    FeeOnTransferNotSupported()
);
```

Alternatively, document that fee-on-transfer tokens are unsupported and enforce this at `touchMarket` by requiring a zero-fee transfer check on `loanToken` at market creation time.

```text
// Settlement fee receipts pending settlement: claimableSettlementFee is incremented in take before
// the inbound fee transfer happens, so we track the gap and clear it in CVL_transferFrom.
persistent ghost mapping(address => mathint) pendingFeeReceipt {
    init_state axiom (forall address token. pendingFeeReceipt[token] == 0);
}

hook Sstore claimableSettlementFee[KEY address token] uint256 newVal (uint256 oldVal) {
    // Except for claimSettlementFee, the claimableSettlementFee is non-decreasing, see WithdrawableMonotonicity.spec.
    if (newVal > oldVal) {
        pendingFeeReceipt[token] = pendingFeeReceipt[token] + newVal - oldVal;
    }
}

/// INVARIANTS AND RULES ///

// For any token, the pending settlement fee receipt after a transaction is 0: every claimableSettlementFee
// increment in take is paid back in by the same-function inbound transfer.
weak invariant pendingFeeReceiptZero(address token)
    pendingFeeReceipt[token] == 0;

// For any token, the balance of the contract is always greater than or equal to the sum of all collateral, withdrawable, and claimable settlement fee amounts for that token minus the flash loaned amount.
// Note: this invariant is strong, so it also holds before each external call.
strong invariant tokenBalanceCorrect(address token)
    tokenBalances[token][currentContract] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token) - flashloans[token] - pendingFeeReceipt[token]
```
