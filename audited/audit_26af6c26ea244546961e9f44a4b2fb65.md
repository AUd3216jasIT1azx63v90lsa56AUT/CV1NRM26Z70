### Title
Continuous Fee Accrual Precision Loss Allows Lenders to Avoid Paying Protocol Fees — (`src/Midnight.sol`)

### Summary

The continuous fee accrual in `updatePositionView()` / `_updatePosition()` suffers from precision loss when the loan token has low decimals (e.g., USDC/USDT at 6 decimals). Because `_position.lastAccrual` is unconditionally updated to `block.timestamp` on every call to `_updatePosition()`, frequent interactions reset the accrual checkpoint even when zero fee was collected. This allows a lender to eliminate 100% of their continuous fee obligation by triggering `_updatePosition()` at intervals shorter than the rounding threshold.

---

### Finding Description

**Root cause — fee formula rounds to zero for short elapsed windows:**

In `updatePositionView()`, the accrued continuous fee is computed as:

```solidity
uint128 fee = _lastAccrual < market.maturity
    ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
    : 0;
``` [1](#0-0) 

This is integer division: `fee = pendingFee * elapsed / (maturity - lastAccrual)`. For a 6-decimal loan token with a small `pendingFee`, the numerator `pendingFee * elapsed` can be smaller than the denominator `(maturity - lastAccrual)`, causing `fee` to round to zero.

**Root cause — checkpoint always advances regardless of fee collected:**

In `_updatePosition()`, `lastAccrual` is unconditionally set to `block.timestamp`:

```solidity
_position.lastAccrual = uint128(block.timestamp);
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
``` [2](#0-1) 

When `accruedFee == 0` due to rounding, the checkpoint still advances. The elapsed time that produced zero fee is permanently discarded and cannot be recovered on the next call.

**Trigger paths — `_updatePosition` is called on every `take()` and `withdraw()`:**

```solidity
if (hasCredit(id, buyer) || units > buyerPos.debt) _updatePosition(offer.market, id, buyer);
if (hasCredit(id, seller)) _updatePosition(offer.market, id, seller);
``` [3](#0-2) 

```solidity
_updatePosition(market, id, onBehalf);
``` [4](#0-3) 

Additionally, `updatePosition()` is a public, permissionless function callable by anyone on any user's position:

```solidity
function updatePosition(Market memory market, address user) external returns (uint128, uint128, uint128) {
    bytes32 id = toId(market);
    require(marketState[id].tickSpacing > 0, MarketNotCreated());
    return _updatePosition(market, id, user);
}
``` [5](#0-4) 

**Numeric example with USDC (6 decimals):**

- `creditIncrease = 1000e6` (1000 USDC)
- `continuousFee = MAX_CONTINUOUS_FEE = 0.01e18 / 365 days ≈ 317` (per-second WAD-scaled rate)
- `timeToMaturity = 30 days = 2,592,000 seconds`
- `pendingFeeIncrease = 1000e6 * 317 * 2,592,000 / 1e18 ≈ 821,664` (≈ 0.82 USDC) [6](#0-5) 

The fee accrued per call is `821,664 * elapsed / 2,592,000`. For `fee > 0`, `elapsed` must exceed `2,592,000 / 821,664 ≈ 3.15 seconds`. Any call to `_updatePosition` within a 3-second window produces `fee = 0` while still advancing `lastAccrual`.

The `pendingFee` is set at credit-increase time in `take()`:

```solidity
uint128 buyerPendingFeeIncrease =
    UtilsLib.toUint128(buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD));
``` [7](#0-6) 

---

### Impact Explanation

The continuous fee is borne by lenders (it reduces their credit). If the fee rounds to zero on every call, the protocol's `continuousFeeCredit` accumulates nothing while lenders retain full credit. A lender with a 1000 USDC position in a 30-day USDC market at the maximum continuous fee rate (1% annualized) should pay ≈ 0.82 USDC in continuous fees. By calling `updatePosition()` every ≤3 seconds, the attacker eliminates 100% of this obligation. The loss scales linearly with position size and number of affected positions. On L2 chains where gas is negligible, the attack is essentially free.

---

### Likelihood Explanation

- No privileged access is required. `updatePosition()` is public and permissionless.
- The attacker is the lender themselves (or anyone calling on their behalf), acting in their own financial interest.
- USDC and USDT are the most common loan tokens in DeFi lending markets; 6-decimal markets are the expected common case.
- The attack can be automated with a simple keeper script calling `updatePosition()` every few seconds.
- Even without deliberate attack, normal market activity (frequent `take()` or `withdraw()` calls) can cause the same precision loss unintentionally.

---

### Recommendation

Do not advance `lastAccrual` when the accrued fee is zero due to rounding. Only update the checkpoint when a non-zero fee is collected, or accumulate elapsed time separately and only reset it when a fee is actually minted:

```solidity
// Only advance lastAccrual if fee > 0
if (accruedFee > 0) {
    _position.lastAccrual = uint128(block.timestamp);
}
```

Alternatively, scale `pendingFee` internally to 18 decimals regardless of the loan token's native decimals, and convert back only at withdrawal/claim time. This is the same fix recommended in M-18.

---

### Proof of Concept

1. Deploy Midnight with USDC (6 decimals) as the loan token.
2. Create a market with `maturity = block.timestamp + 30 days` and `continuousFee = MAX_CONTINUOUS_FEE`.
3. Lender A takes a buy offer, acquiring `credit = 1000e6`. Their `pendingFee ≈ 821,664`.
4. Every 2 seconds, call `midnight.updatePosition(market, lenderA)`.
5. Each call computes `fee = 821664 * 2 / 2592000 = 0` (rounds to zero).
6. `lastAccrual` advances by 2 seconds each call; `continuousFeeCredit` never increases.
7. After 30 days (1,296,000 calls at 2-second intervals), the protocol has collected 0 continuous fee instead of ≈ 821,664 USDC units (≈ 0.82 USDC per 1000 USDC position).
8. Scale to many lenders and larger positions for proportionally larger losses. [8](#0-7) [9](#0-8)

    function updatePositionView(Market memory market, bytes32 id, address user)
        public
        view
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        uint128 credit = _position.credit;
        uint128 _lastLossFactor = _position.lastLossFactor;
        uint256 postSlashCredit = _lastLossFactor < type(uint128).max
            ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
            : 0;
        uint128 _pendingFee = _position.pendingFee;
        uint256 postSlashPendingFee =
            credit > 0 ? _pendingFee - _pendingFee.mulDivUp(credit - postSlashCredit, credit) : 0;
        uint256 accrualEnd = UtilsLib.min(block.timestamp, market.maturity);
        uint128 _lastAccrual = _position.lastAccrual;
        // forge-lint: disable-next-item(unsafe-typecast) as fee <= pending <= credit which are uint128 position fields
        uint128 fee = _lastAccrual < market.maturity
            ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
            : 0;
        // forge-lint: disable-next-item(unsafe-typecast) as credit and pending are <= uint128 position fields
        return (uint128(postSlashCredit) - fee, uint128(postSlashPendingFee) - fee, fee);
    }

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
```
