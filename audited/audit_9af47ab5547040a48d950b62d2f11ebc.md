### Title
`stake()` Lacks Registry-State Guard, Reorg Can Permanently Lock Staker Into UNDER_ATTACK With Full CORRUPTED Exposure - (File: src/ConfidencePool.sol)

### Summary
`stake()` accepts no maximum-acceptable-registry-state parameter. A chain reorg that re-orders the staker's transaction to execute after the registry has transitioned to `UNDER_ATTACK` permanently locks the staker's principal, strips their bonus to near-zero, and exposes them to full CORRUPTED loss — an outcome they never consented to.

### Finding Description
`stake()` calls `_assertDepositsAllowed(_observePoolState())` to gate on the live registry state at execution time. [1](#0-0) 

`_assertDepositsAllowed` permits `UNDER_ATTACK` deposits by design (DESIGN.md §3), so a transaction that executes in `UNDER_ATTACK` succeeds silently. [2](#0-1) 

Inside `_observePoolState`, the first observation of an active-risk state calls `_markRiskWindowStart`, which sets the one-way `riskWindowStart` latch. [3](#0-2) 

Once `riskWindowStart != 0`, `withdraw()` is permanently disabled for every staker. [4](#0-3) 

A staker who submitted the transaction while the registry was in `NEW_DEPLOYMENT` or `ATTACK_REQUESTED` (where withdrawal is still possible) has no on-chain mechanism to express "revert if the state has advanced past X." If a reorg re-orders their transaction to a block where the registry is already `UNDER_ATTACK`, the call succeeds, `riskWindowStart` is sealed, and the staker is irrevocably committed.

### Impact Explanation
The staker suffers three simultaneous harms they did not consent to:

1. **Permanent withdrawal lock** — `riskWindowStart != 0` disables `withdraw()` forever; the staker cannot exit regardless of subsequent registry behavior.
2. **Near-zero bonus** — entry time is floored at `riskWindowStart ≈ block.timestamp`, so `(T − entryTime)² ≈ 0` and the k=2 bonus share collapses to essentially nothing. [5](#0-4) 
3. **Full CORRUPTED exposure** — if the agreement is later flagged CORRUPTED, the staker loses their entire principal to `recoveryAddress` or the named attacker. [6](#0-5) 

### Likelihood Explanation
BattleChain is an EVM-compatible L2. L2 sequencers can experience reorgs during sequencer failover, forced-inclusion windows, or cross-chain state lag. The `UNDER_ATTACK` transition is a discrete, observable on-chain event; a reorg that re-orders a stake transaction to land after that transition is a realistic scenario on any chain that is not Ethereum mainnet. The protocol's README explicitly scopes deployment to BattleChain, and the original kairos report was raised precisely because the contracts were intended for any EVM chain. [7](#0-6) 

### Recommendation
Add an optional `maxState` (or `requireState`) parameter to `stake()` (and symmetrically to `contributeBonus()`) that reverts if the observed registry state exceeds the caller's declared tolerance:

```solidity
function stake(uint256 amount, IAttackRegistry.ContractState maxAllowedState) external nonReentrant whenPoolNotPaused {
    ...
    IAttackRegistry.ContractState state = _observePoolState();
    if (uint8(state) > uint8(maxAllowedState)) revert StateExceedsMaxAllowed();
    _assertDepositsAllowed(state);
    ...
}
```

This mirrors the fix applied in kairos-contracts PR #50 (adding a max-price parameter to `useLoan`) and gives stakers a trustless guarantee that their transaction reverts rather than silently committing them to an unintended risk level.

### Proof of Concept

1. Registry is in `ATTACK_REQUESTED`. Alice checks state off-chain, sees she can still withdraw, and submits `stake(1000e18)`.
2. Before Alice's transaction is included, the registry transitions to `UNDER_ATTACK` (e.g., a whitehat calls `markUnderAttack`).
3. A reorg removes Alice's original block and re-includes her transaction in a later block where the registry is already `UNDER_ATTACK`.
4. `_observePoolState()` returns `UNDER_ATTACK`; `_assertDepositsAllowed` does **not** revert (UNDER_ATTACK is explicitly permitted). [2](#0-1) 
5. `_markRiskWindowStart()` fires, setting `riskWindowStart = block.timestamp`. [8](#0-7) 
6. Alice's 1000e18 tokens are transferred in; `eligibleStake[Alice] = 1000e18`.
7. Alice immediately calls `withdraw()` — it reverts with `WithdrawsDisabled` because `riskWindowStart != 0`. [4](#0-3) 
8. The agreement is later flagged CORRUPTED. Alice's entire 1000e18 is swept to `recoveryAddress` via `claimCorrupted`, with zero bonus and zero principal returned. [6](#0-5)

```text
    function _markRiskWindowStart() internal {
        // Cap at expiry: accrual is bounded by the pool's lifecycle. Without the cap, a late
        // observation could pin riskWindowStart > expiry, and `_clampUserSums` would record every
        // pre-risk deposit as entering past T = expiry (EXPIRED path). The k=2 sums then encode
        // post-deadline "at-risk" time that never actually existed.
        uint256 t = block.timestamp;
        if (t > expiry) t = expiry;
        // Cast is truncation-safe: `t` is capped at `expiry`, which is itself a uint32.
        // forge-lint: disable-next-line(unsafe-typecast)
        riskWindowStart = uint32(t);
        // Eagerly reset the global accumulators so every currently-eligible staker is treated as
        // entering at `t`. Per-user sums stay stale until `_clampUserSums` runs on the next op
        // touching that user.
        sumStakeTime = totalEligibleStake * t;
        sumStakeTimeSq = totalEligibleStake * t * t;
        emit RiskWindowStarted(t);
```
