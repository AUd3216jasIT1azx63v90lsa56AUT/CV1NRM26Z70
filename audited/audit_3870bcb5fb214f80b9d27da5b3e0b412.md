### Title
Reentrant `supplyCollateral` in `onLiquidate` callback causes `badDebt` over-realization, slashing lenders beyond actual undercollateralization - (File: src/Midnight.sol)

### Summary

In `liquidate()`, `badDebt` is computed from a snapshot of the collateral bitmap and storage collateral values, then immediately applied to storage (reducing `_position.debt` and updating `lossFactor`/`totalUnits`), all before the `onLiquidate` callback fires. A liquidator whose callback contract is authorized by the borrower can reenter `supplyCollateral` during the callback, adding collateral that would have reduced or eliminated `badDebt`, but which is never counted. Lenders are socialized a loss larger than the actual undercollateralization.

### Finding Description

**Exact code path:**

`liquidate()` at `src/Midnight.sol` lines 602–618 takes a local copy of the bitmap and iterates over it, reading `_position.collateral[i]` from storage for each active index: [1](#0-0) 

`badDebt` and `maxDebt` are fully computed from this pre-callback state. Then, at lines 626–641, `badDebt` is immediately committed to storage — `_position.debt` is reduced, `lossFactor` is increased, and `totalUnits` is decreased: [2](#0-1) 

Only after all of this does the `onLiquidate` callback fire at lines 698–715: [3](#0-2) 

`supplyCollateral` has no reentrancy guard and its only authorization check is: [4](#0-3) 

**Attacker inputs and exploit flow:**

Preconditions:
- `isAuthorized[borrower][callbackContract] == true` (borrower previously authorized the callback contract — realistic in DeFi position-management setups)
- The borrower's position has `badDebt > 0` (debt exceeds max-LIF-weighted collateral value)
- `liquidatorGate == address(0)` or the callback contract passes the gate

Exploit sequence:
1. Callback contract calls `liquidate(market, collateralIndex, 0, 0, borrower, postMaturityMode, receiver, callbackContract, data)`
2. Protocol computes `badDebt = originalDebt − Σ(collateral[i] × price[i] / ORACLE_PRICE_SCALE × WAD / maxLif[i])` from the current storage values
3. Protocol applies `badDebt`: reduces `_position.debt`, increases `lossFactor`, decreases `totalUnits` — lenders are slashed
4. Protocol calls `callbackContract.onLiquidate(...)`
5. Inside the callback, `callbackContract` calls `supplyCollateral(market, newIndex, largeAmount, borrower)` — this succeeds because `isAuthorized[borrower][callbackContract]` is true
6. `_position.collateral[newIndex]` and `_position.collateralBitmap` are updated in storage
7. Callback returns `CALLBACK_SUCCESS`
8. `liquidate()` completes

The `badDebt` that was applied in step 3 did not account for the collateral added in step 5. The attack also works for an existing bitmap index: the loop reads `_position.collateral[i]` from storage at loop time; supplying more to an already-active index after the loop also escapes the `badDebt` computation.

**Why existing checks fail:**

There is no reentrancy guard on `liquidate()` or `supplyCollateral()`. The Certora formal verification explicitly excludes this scenario — `OnlyAuthorizedCanChange.spec` line 73 states: *"Assumes no reentrancy: callbacks and token transfers are not modeled as re-entering Midnight, so re-entrant collateral changes are not covered."* [5](#0-4) 

### Impact Explanation

`badDebt` is over-realized by `Δ = additionalCollateral × price / ORACLE_PRICE_SCALE × WAD / maxLif`. This excess is socialized to all lenders in the market via an inflated `lossFactor` and reduced `totalUnits`. Lenders suffer a credit loss larger than the actual undercollateralization. The borrower's debt is reduced by more than warranted, and the supplied collateral remains in the borrower's position and can be withdrawn later once the position is healthy, making the net cost to the attacker near zero.

### Likelihood Explanation

Preconditions are realistic: authorization of a callback/manager contract by a borrower is standard DeFi UX. The position must have bad debt, which occurs whenever collateral value falls below the max-LIF-weighted debt. The attack is repeatable across any market with `liquidatorGate == address(0)` (unrestricted) or where the callback passes the gate. The borrower and liquidator can be the same entity (borrower authorizes their own liquidation contract).

### Recommendation

Apply a reentrancy lock around the `badDebt` computation and application, or move the `onLiquidate` callback to before the `badDebt` is committed to storage, or snapshot and re-verify `badDebt` after the callback returns. The simplest fix is a transient-storage reentrancy guard (`tstore`/`tload`, already available given the contract targets Cancun/Osaka) that prevents `supplyCollateral` and `withdrawCollateral` from executing while `liquidate()` is in progress.

import {Midnight, Market, CollateralParams} from "src/Midnight.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";

contract ReentrantLiquidateSupplyCallback {
    Midnight public midnight;
    Market public market;
    address public borrower;
    uint256 public newCollateralIndex;
    uint256 public supplyAmount;

    constructor(address _midnight, Market memory _market, address _borrower,
                uint256 _newIdx, uint256 _amount) {
        midnight = Midnight(_midnight);
        market = _market;
        borrower = _borrower;
        newCollateralIndex = _newIdx;
        supplyAmount = _amount;
    }

    function onLiquidate(
        address, bytes32, Market memory _market, uint256, uint256, uint256,
        address _borrower, address, bytes memory, uint256 /*badDebt*/
    ) external returns (bytes32) {
        // Reenter supplyCollateral on behalf of borrower
        // Requires isAuthorized[borrower][address(this)] == true
        IERC20(_market.collateralParams[newCollateralIndex].token)
            .approve(msg.sender, supplyAmount);
        Midnight(msg.sender).supplyCollateral(
            _market, newCollateralIndex, supplyAmount, _borrower
        );
        return CALLBACK_SUCCESS;
    }
}
