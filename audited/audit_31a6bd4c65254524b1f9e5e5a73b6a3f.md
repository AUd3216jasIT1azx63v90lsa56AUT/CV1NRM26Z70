Audit Report

## Title
Front-running `repay` with a partial liquidation causes arithmetic underflow DoS - (`src/Midnight.sol`)

## Summary
The `repay` function subtracts `units` from `position[id][onBehalf].debt` with no guard that `units ≤ current debt`. Because `liquidate` can reduce the same debt field in a preceding transaction, an attacker can front-run any `repay(market, D, onBehalf, …)` call with a partial liquidation that lowers debt below `D`, causing the borrower's repay to revert with a Solidity 0.8 arithmetic underflow. The attacker earns the liquidation incentive on every front-run, making the attack self-funding and repeatable.

## Finding Description

**Root cause — missing bounds check in `repay`:**

`repay` at `src/Midnight.sol:508` performs:
```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
``` [1](#0-0) 

`UtilsLib.toUint128` only guards against a cast overflow (`x <= type(uint128).max`); it does not enforce `units ≤ debt`. [2](#0-1) 

The `-=` on a `uint128` storage slot is Solidity 0.8 checked arithmetic and reverts on underflow. The only other check in `repay` is the authorization guard at line 505, which is irrelevant here. [3](#0-2) 

**State-changing path in `liquidate`:**

`liquidate` writes to the same `_position.debt` slot via two paths before any external call:
- Bad-debt path (line 628): `_position.debt -= uint128(badDebt);`
- Normal path (line 676): `_position.debt -= UtilsLib.toUint128(repaidUnits);` [4](#0-3) [5](#0-4) 

Both state changes are committed before the liquidator's token transfer, so they are visible to any subsequent transaction in the same block.

**`liquidatorGate` does not protect permissionless markets:**

The gate check at lines 597–600 is only enforced when `market.liquidatorGate != address(0)`. Markets with `liquidatorGate == address(0)` (the default) allow any caller to liquidate. [6](#0-5) 

**RCF does not prevent a 1-unit liquidation:**

In normal mode, `maxRepaid` is computed as `(debt - maxDebt) * WAD² / (WAD² - lif*lltv)`. For any unhealthy position, `debt > maxDebt`, so `maxRepaid ≥ 1`. The attacker can always liquidate at least 1 unit. [7](#0-6) 

**Exploit flow:**
1. Borrower has unhealthy position: `debt = D`, position is liquidatable (`originalDebt > maxDebt`).
2. Borrower broadcasts `repay(market, D, onBehalf, …)` to repay the full balance.
3. Attacker sees the transaction in the mempool and front-runs with `liquidate(market, collateralIndex, 0, 1, borrower, false, …)`.
4. Liquidation succeeds: `_position.debt` is now `D - 1`.
5. Borrower's `repay` lands: `(D-1) - D` underflows → **revert**.
6. Attacker received collateral worth `1 * lif / WAD` in step 3, paying only 1 loan token — net profit from the liquidation incentive.
7. Attacker repeats on every retry, permanently blocking repayment.

**Test suite gap:**

The `testRepay` fuzz test always bounds `repaid ≤ units`, so the underflow path is never exercised. [8](#0-7) 

## Impact Explanation
Any borrower whose position is unhealthy can be permanently denied the ability to repay via `repay`. Every retry is front-runnable with a 1-unit liquidation. The borrower is forced into full liquidation, losing the liquidation incentive premium on their collateral rather than repaying at par. This constitutes a permanent, targeted freeze of the `repay` function for any unhealthy borrower in any market without a `liquidatorGate`, which is the default configuration.

## Likelihood Explanation
**Preconditions:** (a) borrower's position is unhealthy (price dropped below LLTV threshold) — a routine on-chain state; (b) market has no `liquidatorGate` — the default. **Feasibility:** front-running is standard on EVM chains with a public mempool; the attacker submits a higher-gas-price `liquidate` with `repaidUnits = 1`. **Profitability:** the attacker earns the liquidation incentive (`lif > WAD`) on every front-run, making the attack net-positive and self-funding. **Repeatability:** the attack can be repeated on every retry until the borrower's collateral is fully seized.

## Recommendation
Add an explicit cap in `repay` so that `units` is silently clamped to the current debt, or add a `require` guard:

```solidity
uint128 currentDebt = position[id][onBehalf].debt;
uint128 repayUnits = UtilsLib.toUint128(units);
require(repayUnits <= currentDebt, RepayExceedsDebt());
position[id][onBehalf].debt = currentDebt - repayUnits;
```

Alternatively, expose a "repay-all" variant that reads the current debt atomically and repays exactly that amount, eliminating the race window entirely.

## Proof of Concept
Minimal Foundry test outline:

```solidity
function testFrontRunRepayUnderflow() public {
    // 1. Setup: borrower has debt = D, position is unhealthy
    uint256 D = 1000e18;
    collateralize(market, borrower, D);
    setupMarket(market, D);
    // Drop oracle price to make position unhealthy
    Oracle(market.collateralParams[0].oracle).setPrice(unhealthyPrice);

    // 2. Attacker front-runs with a 1-unit liquidation
    deal(address(loanToken), attacker, 1);
    vm.prank(attacker);
    midnight.liquidate(market, 0, 0, 1, borrower, false, attacker, address(0), "");
    // debt is now D - 1

    // 3. Borrower's repay of D now underflows and reverts
    deal(address(loanToken), borrower, D);
    vm.prank(borrower);
    vm.expectRevert(stdError.arithmeticError);
    midnight.repay(market, D, borrower, address(0), "");
}
```
