### Title
Zero-buyerAssets rounding allows unlimited takes without advancing consumed counter on buy offers with maxAssets cap - (File: src/Midnight.sol)

### Summary
When a buy offer has `offer.maxAssets > 0`, the consumed counter is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `mulDivDown` is plain integer division, any `units` satisfying `units * buyerPrice < WAD` yields `buyerAssets = 0`, so `consumed[maker][group] += 0` and the offer capacity is never decremented. No guard in `take()` prevents this, so a taker can call `take()` an unbounded number of times against the same buy offer without ever exhausting its `maxAssets` cap.

### Finding Description

**Code path — `src/Midnight.sol` lines 356–369:**

The ratifier is checked first: [1](#0-0) 

Then `buyerAssets` is computed with a rounding-down division: [2](#0-1) 

And the consumed counter is updated only by `buyerAssets`: [3](#0-2) 

`mulDivDown` is `(x * y) / d` with no rounding-up or minimum-one guard: [4](#0-3) 

**Root cause:** There is no `require(units > 0)` and no `require(buyerAssets > 0)` (or equivalent) anywhere in `take()`. When `units * buyerPrice < WAD`, integer division truncates to zero, `consumed` is unchanged, and the `require(newConsumed <= offer.maxAssets)` check trivially passes because `newConsumed` equals its prior value.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0` (any positive cap)
- `offer.tick = 0` → `tickToPrice(0) = 0` → `buyerPrice = 0` → `buyerAssets = 0` for every `units` (confirmed by the zero-price test suite)
- Alternatively, any tick whose price satisfies `price < WAD / units` (e.g., minimum `PRICE_ROUNDING_STEP = 1e12` with `units < 1e6`)

**Why existing checks fail:**
- The ratifier (`IRatifier.isRatified`) is `external view`; it validates offer parameters, not the `units` value, and is called before the consumed accounting. [5](#0-4) 
- `safeTransferFrom` with `value = 0` does not revert on standard ERC20 tokens. [6](#0-5) 
- The `require(newConsumed <= offer.maxAssets)` check passes because `newConsumed` is unchanged.
- `tickSpacing` allows tick=0 (`0 % spacing == 0`). [7](#0-6) 

**Exploit flow (concrete):**
1. Maker authorizes a ratifier and posts a buy offer: `offer.buy = true`, `offer.maxAssets = 1e18`, `offer.tick = 0`, no settlement fee.
2. Taker calls `take(offer, ratifierData, 1e18, taker, receiver, address(0), "")`.
3. `offerPrice = tickToPrice(0) = 0`; `buyerPrice = 0`; `buyerAssets = mulDivDown(1e18, 0, 1e18) = 0`.
4. `consumed[maker][group] += 0` → still 0; `require(0 <= 1e18)` passes.
5. Position state may change (credit/debt for non-zero `units`), but consumed is untouched.
6. Taker repeats step 2 indefinitely; `consumed` never reaches `maxAssets`.

The same path works for any `buyerPrice > 0` that is small enough: with `PRICE_ROUNDING_STEP = 1e12` as the minimum representable price, any `units < 1e6` yields `buyerAssets = 0`. [8](#0-7) 

### Impact Explanation
The invariant "every successful take advances the offer's consumed state" is broken. A buy offer with `maxAssets` cap can be taken an unlimited number of times without the cap ever being reached. The ratifier is invoked on every call, and all other take-side effects (position updates, events, health checks) execute normally, but the capacity gate is permanently bypassed. This is confirmed by the protocol's own zero-price test suite, which shows zero-asset takes succeed without reverting. [9](#0-8) 

### Likelihood Explanation
**Preconditions:** A buy offer with `offer.maxAssets > 0` must exist at a tick whose price is zero or small enough that `units * buyerPrice < WAD`. Tick=0 is always accessible (it is a multiple of any `tickSpacing`), and `tickToPrice(0) = 0` is a documented, tested value. Settlement fee must be zero (or the offer reverts on underflow at `offerPrice - settlementFee`), which is the default state for markets with no fee set. [10](#0-9) 

Any unprivileged taker can trigger this; no special role or leaked key is required. The attack is freely repeatable in the same transaction or across blocks.

### Recommendation
Add a guard that prevents a successful take from leaving `consumed` unchanged when `offer.maxAssets > 0`. The minimal fix is to require the asset-denominated consumed increment to be non-zero:

```solidity
if (offer.maxAssets > 0) {
    uint256 consumedDelta = offer.buy ? buyerAssets : sellerAssets;
    require(consumedDelta > 0, ConsumedZero()); // <-- add this
    newConsumed = consumed[offer.maker][offer.group] += consumedDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, add `require(units > 0)` unconditionally at the top of `take()`, which also prevents the degenerate `units = 0` no-op path.

### Proof of Concept

```solidity
function testZeroBuyerAssetsDoesNotAdvanceConsumed() public {
    // Setup: buy offer with maxAssets cap, tick=0 (price=0), no settlement fee
    lenderOffer.buy = true;
    lenderOffer.tick = 0;
    lenderOffer.maxAssets = 1e18;
    lenderOffer.maxUnits = 0;

    uint256 units = 1e18;
    collateralize(market, borrower, units * 3);

    uint256 consumedBefore = midnight.consumed(lender, lenderOffer.group);

    // First take: buyerAssets = mulDivDown(1e18, 0, 1e18) = 0
    vm.prank(borrower);
    midnight.take(lenderOffer, hex"", units, borrower, borrower, address(0), "");

    uint256 consumedAfter = midnight.consumed(lender, lenderOffer.group);

    // ASSERTION: consumed must have advanced by at least 1 on a successful take
    assertGt(consumedAfter, consumedBefore, "consumed did not advance");

    // Second take: should revert if maxAssets is truly enforced, but it succeeds
    vm.prank(borrower);
    midnight.take(lenderOffer, hex"", units, borrower, borrower, address(0), "");
    // consumed is still 0 — offer capacity was never consumed
    assertEq(midnight.consumed(lender, lenderOffer.group), 0, "consumed still zero after two takes");
}
```

Expected: the first `assertGt` fails (consumed remains 0), demonstrating the invariant violation. The second take also succeeds, confirming unbounded repetition.
