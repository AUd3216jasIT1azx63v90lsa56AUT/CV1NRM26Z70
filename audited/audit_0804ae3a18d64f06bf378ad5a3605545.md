### Title
Consumed-offer accounting not advanced when `buyerAssets` rounds to zero, allowing buy offers to be overfilled indefinitely - (`src/Midnight.sol`)

### Summary
When a buy offer has `offer.maxAssets > 0` and a low tick (e.g., tick = 2, `buyerPrice = 1e12`), a taker can call `take()` with `units` in `[1, WAD/buyerPrice - 1]` such that `units.mulDivDown(buyerPrice, WAD) == 0`. Because `consumed[offer.maker][offer.group]` is incremented by `buyerAssets` (which is 0), the cap check `require(newConsumed <= offer.maxAssets)` never triggers, yet the maker's credit and the taker's debt positions are updated by `units` on every call. The offer can be filled an unbounded number of times with zero asset transfer per fill.

### Finding Description
In `src/Midnight.sol`, `take()` computes:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// ...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For a buy offer, `buyerPrice = tickToPrice(offer.tick)`. Since `tickToPrice` returns values in `[0, WAD]` and uses `PRICE_ROUNDING_STEP = 1e12`, the minimum non-zero price is `1e12` (tick = 2). With `buyerPrice = 1e12` and `units = 1`, `mulDivDown(1, 1e12, 1e18) = 0`. Consumed is incremented by 0, so `newConsumed` stays at its prior value and the cap check always passes. Meanwhile, the position update block executes unconditionally with `units = 1`:

```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt);
uint256 sellerDebtIncrease = units - sellerCreditDecrease;
```

The maker (buyer) gains `units` of credit and the taker (seller) gains `units` of debt, but both `buyerAssets` and `sellerAssets` are 0, so the two `safeTransferFrom` calls transfer nothing. No existing check (`require(units > 0)` is absent; the ratifier check is maker-controlled; the health check applies to the seller/taker) blocks this path.

**Exploit flow:**
1. Maker creates a buy offer: `offer.buy = true`, `offer.maxAssets = X`, `offer.tick = 2` (`buyerPrice = 1e12`).
2. Taker calls `take(offer, ..., units=1, ...)` — `buyerAssets = 0`, consumed unchanged, maker credit +1, taker debt +1, zero assets transferred.
3. Taker repeats step 2 arbitrarily many times. `consumed` stays at 0; the cap `X` is never reached.

### Impact Explanation
The maker's `maxAssets` cap is completely bypassed. The maker accumulates credit far beyond their intended limit with no corresponding asset inflow. The core invariant "offers cannot be overfilled" is violated: `consumed[offer.maker][offer.group]` never advances despite real position changes occurring on every fill.

### Likelihood Explanation
Preconditions are easily met: any buy offer with `offer.maxAssets > 0` and `offer.tick < MAX_TICK` (i.e., `buyerPrice < WAD`) is vulnerable. The taker only needs to choose `units < WAD / buyerPrice` (e.g., `units = 1` for tick = 2). The taker must remain healthy (collateral or existing credit covers the debt increase), but this is a standard operational state. The attack is repeatable across any number of transactions at negligible cost since zero assets are transferred per call.

### Recommendation
Add a minimum-assets guard in the `maxAssets` branch: require that the computed asset delta is nonzero whenever `units > 0`, or alternatively track consumed in units (not assets) even in the `maxAssets` mode, converting the cap at check time. The simplest targeted fix:

```solidity
if (offer.maxAssets > 0) {
    uint256 delta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || delta > 0, ZeroAssetFill());
    newConsumed = consumed[offer.maker][offer.group] += delta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

### Proof of Concept
```solidity
// Foundry fuzz test
function testConsumedDoesNotAdvanceOnZeroAssetFill(uint256 units) public {
    // buyerPrice = tickToPrice(2) = 1e12; WAD/buyerPrice = 1e6
    units = bound(units, 1, 1e6 - 1); // units s.t. mulDivDown(units, 1e12, 1e18) == 0

    // Setup: buy offer, maxAssets = 1e18, tick = 2
    offer.buy = true;
    offer.tick = 2;
    offer.maxAssets = 1e18;
    offer.maxUnits = 0;
    // maker = lender, taker = borrower (with sufficient collateral)
    collateralize(market, borrower, units * 1000);

    uint256 consumedBefore = midnight.consumed(offer.maker, offer.group);
    take(units, borrower, offer); // taker is seller
    uint256 consumedAfter = midnight.consumed(offer.maker, offer.group);

    // BUG: consumed did not advance despite units > 0
    assertEq(consumedAfter, consumedBefore, "consumed delta == 0");
    // But credit increased
    assertGt(midnight.creditOf(id, offer.maker), 0, "maker credit increased");
}
```

Expected: `consumedAfter == consumedBefore` (bug confirmed) while `creditOf(maker) > 0`. A correct implementation would require `consumedAfter > consumedBefore` or revert.
