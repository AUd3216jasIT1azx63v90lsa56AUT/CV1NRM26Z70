Audit Report

## Title
Bad Debt Over-Realization via `onLiquidate` Callback Supplying Collateral After Bitmap Snapshot - (File: `src/Midnight.sol`)

## Summary
The `liquidate` function snapshots `_position.collateralBitmap` into a local variable at line 606, computes `badDebt` exclusively from that snapshot, and fully applies the bad debt (modifying `_position.debt`, `_marketState.lossFactor`, and `_marketState.totalUnits`) at lines 626–641 before invoking the `onLiquidate` callback at lines 698–715. A borrower-controlled smart contract can use the callback to call `supplyCollateral` for a previously-inactive collateral index, depositing real collateral value that was never subtracted from `badDebt`, causing the protocol to over-realize bad debt and over-slash lenders.

## Finding Description

**Root cause:** The local bitmap variable `_collateralBitmap` (line 606) is never refreshed from storage after the callback. Any collateral index activated during the callback is invisible to the `badDebt` computation.

**Exact code path:**

- `src/Midnight.sol` line 606: `uint128 _collateralBitmap = _position.collateralBitmap;` — snapshot taken.
- Lines 607–618: `while (_collateralBitmap != 0)` loop computes `badDebt` only from indices present in the snapshot.
- Lines 626–641: `badDebt` is irreversibly applied — `_position.debt -= uint128(badDebt)`, `_marketState.lossFactor` is ratcheted up, `_marketState.totalUnits` is reduced — all before any external call.
- Lines 698–715: `ILiquidateCallback(callback).onLiquidate(...)` is invoked after full realization.
- No reentrancy guard exists anywhere in `Midnight.sol` (confirmed by absence of `nonReentrant`, `ReentrancyGuard`, or equivalent).

**Exploit flow:**

1. Borrower deploys a smart contract with a position where all existing collateral is worth less than debt at `maxLif` prices (genuine bad debt).
2. Borrower contract calls `liquidate(market, 0, 0, 0, address(this), false, receiver, address(this), "")` — `seizedAssets = repaidUnits = 0`, `callback = address(this)`.
3. `liquidate` snapshots `_collateralBitmap` (line 606), loops over existing collateral, computes `badDebt = originalDebt − Σ(collateral_i × price_i / maxLif_i)` (positive).
4. `liquidate` applies `badDebt`: reduces `_position.debt`, increases `_marketState.lossFactor`, reduces `_marketState.totalUnits` (lines 626–641) — irreversibly.
5. `liquidate` calls `onLiquidate` on the borrower contract (line 700).
6. Inside `onLiquidate`, the borrower contract calls `supplyCollateral(market, newIndex, largeAmount, address(this))`. Authorization passes trivially because `onBehalf == msg.sender` (both are the borrower contract). This sets `_position.collateral[newIndex]` and sets the corresponding bit in `_position.collateralBitmap` in storage.
7. `liquidate` returns. The new collateral is now in the position, but `badDebt` was already applied without accounting for it.

**Why existing checks fail:**

- No reentrancy guard on `liquidate` or `supplyCollateral`.
- No post-callback re-computation or validation of `badDebt`.
- The local `_collateralBitmap` variable is never refreshed from storage after the callback.
- `supplyCollateral`'s authorization check (`onBehalf == msg.sender`) is satisfied trivially when the borrower contract calls on behalf of itself.

## Impact Explanation

`_marketState.lossFactor` is set higher than warranted and `_marketState.totalUnits` is reduced by more than the true uncollateralized shortfall. Every subsequent lender `updatePosition` call slashes lender credit proportionally to the inflated loss factor — lenders bear losses they should not bear. The borrower's debt is reduced by a `badDebt` amount that exceeds the true shortfall, effectively forgiving debt that was actually backed by the newly supplied collateral. This constitutes direct, unauthorized theft of lender value.

## Likelihood Explanation

Preconditions are minimal and realistic: (1) a position with genuine bad debt — a normal liquidation scenario; (2) the borrower is a smart contract — standard in DeFi; (3) the borrower passes itself as the callback — requires no external authorization. The attack requires no privileged access, no oracle manipulation, no flash loan (the borrower can pre-fund the collateral), and is repeatable on any market with bad debt.

## Recommendation

Apply one or more of the following:

1. **Reentrancy guard:** Add a `nonReentrant` modifier to both `liquidate` and `supplyCollateral` (and any other state-mutating function callable during a callback).
2. **Post-callback re-validation:** After the `onLiquidate` callback returns, re-read `_position.collateralBitmap` from storage and recompute the true bad debt. If the re-computed value is less than the applied `badDebt`, revert or adjust the loss factor accordingly.
3. **Snapshot-then-lock:** Before invoking the callback, record the current `collateralBitmap` in storage and reject any `supplyCollateral` call that would set a bit not present in the pre-callback snapshot for the duration of the liquidation.

        // Supply collateral for a new index AFTER badDebt has been applied
