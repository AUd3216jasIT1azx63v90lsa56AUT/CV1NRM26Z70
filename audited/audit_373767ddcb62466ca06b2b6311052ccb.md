### Title
Debt increase permitted at exactly `block.timestamp == maturity` due to non-strict `<=` boundary check - (`src/Midnight.sol`)

### Summary
In `Midnight.take()`, the guard against post-maturity debt increase uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` at exactly `block.timestamp == maturity`. This directly contradicts the protocol's own stated invariant that "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing." An unprivileged taker (or an authorized operator acting as taker) can take a sell offer at the exact maturity timestamp, increasing the seller's debt at a moment when repayment is no longer possible.

### Finding Description
**Code path:**

`src/Midnight.sol:391`
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

At `block.timestamp == offer.market.maturity`, the left operand `block.timestamp <= offer.market.maturity` is `true`, so the entire `require` passes regardless of `sellerDebtIncrease`. The protocol then executes:

```solidity
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);  // line 414
```

**Ratifier path via SetterRatifier:**

`src/ratifiers/SetterRatifier.sol:24-27` — an authorized operator (`isAuthorized[maker][operator] == true`) can call `setIsRootRatified(maker, root, true)` to ratify any Merkle root on behalf of the maker. A root containing a sell offer (where `offer.buy == false`, making `offer.maker` the seller) can be ratified at any time before the take.

**Exploit flow:**
1. Maker creates a sell offer (maker is seller, `offer.buy = false`). Operator calls `setIsRootRatified(maker, root, true)` to ratify the Merkle root containing this offer.
2. Maker has no existing credit in the market, so `sellerCreditDecrease = 0` and `sellerDebtIncrease = units`.
3. Attacker (operator or any taker) warps/waits to `block.timestamp == offer.market.maturity`.
4. Attacker calls `take(offer, ratifierData, units, taker, ...)` with a valid Merkle proof.
5. Check at line 391: `maturity <= maturity` → `true` → passes.
6. `sellerPos.debt += units` executes, increasing maker's debt at exactly maturity.

**Why existing checks fail:**
- `block.timestamp <= offer.expiry` (line 353): passes if `offer.expiry >= maturity`.
- `isHealthy` check (line 476): only checks health after the take, does not prevent the debt increase itself.
- All existing post-maturity tests (`testBuy1PostMaturity`, `testSell1PostMaturity`, `testBuyPastMaturity`, `testSellPastMaturity`) use `market.maturity + 1`, leaving the exact boundary untested.

### Impact Explanation
The maker's debt is increased at exactly `block.timestamp == maturity`. Post-maturity, no new debt can be created (by the invariant), so this debt is immediately overdue and liquidatable but cannot be repaid through normal borrow/repay flows that also enforce the same boundary. The maker holds debt that was created at the moment repayment windows close, creating an incompatible position state: debt exists but the normal pre-maturity repayment path is gone, and the debt is immediately subject to overdue liquidation.

### Likelihood Explanation
**Preconditions:**
- A sell offer exists with a ratified root in `SetterRatifier` (normal protocol usage).
- The offer's `expiry >= maturity` (common for long-lived offers).
- The maker has no existing credit to absorb the units (i.e., `sellerCreditDecrease < units`).

**Feasibility:** Any taker can observe the maturity timestamp on-chain and submit the transaction at exactly that block. On EVM chains, `block.timestamp` is set by the block proposer, making exact-maturity execution straightforward. The condition is repeatable for any market whose sell offers have `expiry >= maturity`.

### Recommendation
Change the boundary check from `<=` to `<` (strict less-than):

```solidity
// Before (line 391):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After:
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns with the protocol invariant at `live_context.json:221`: "maturity boundary must not allow debt increase by rounding, **timestamp equality**, or callback sequencing."

### Proof of Concept
```solidity
function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;

    // Precondition: maker (borrower) has no existing credit
    // Precondition: sell offer ratified via SetterRatifier
    collateralize(market, borrower, units);

    // Warp to EXACTLY maturity (not maturity + 1)
    vm.warp(market.maturity);
    borrowerOffer.expiry = market.maturity; // offer valid at maturity
    borrowerOffer.maxUnits = units;

    uint256 debtBefore = midnight.debtOf(id, borrower);

    // This should revert with CannotIncreaseDebtPostMaturity but does NOT
    take(units, lender, borrowerOffer);

    // Assert: debt increased at maturity — invariant violated
    assertGt(midnight.debtOf(id, borrower), debtBefore, "debt increased at maturity");

    // Assert: debt is immediately overdue/liquidatable post-maturity
    vm.warp(market.maturity + 1);
    assertTrue(midnight.isLiquidatable(market, id, borrower), "immediately liquidatable");
}
```

**Expected assertion behavior with the bug present:** `take()` succeeds (no revert), `debtOf(borrower)` increases, and the position is immediately liquidatable at `maturity + 1`.

**Expected assertion behavior after fix:** `take()` reverts with `CannotIncreaseDebtPostMaturity`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
