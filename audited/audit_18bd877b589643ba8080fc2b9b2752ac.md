Audit Report

## Title
Missing payer authorization in `repay()` allows arbitrary callback designation to drain third-party loanToken - (File: src/Midnight.sol)

## Summary
`repay()` authorizes `msg.sender` to repay debt on behalf of `onBehalf` but imposes no authorization check on the `callback` parameter, which becomes the token payer. Any account authorized to repay on behalf of a debtor can designate an arbitrary victim address as `callback`, causing Midnight to invoke `victim.onRepay(...)` and then pull loanToken directly from the victim. This is exploitable from within the `onSell` callback during `take()`, where the seller's authorized callback contract can re-enter `repay()` with `callback=victim` to retire the seller's debt at the victim's expense.

## Finding Description

**Root cause — `repay()` sets payer from unchecked `callback` parameter:**

```solidity
// src/Midnight.sol:505
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
// ...
address payer = callback != address(0) ? callback : msg.sender;   // payer = victim (line 511)
// ...
IRepayCallback(callback).onRepay(id, market, units, onBehalf, data);  // called on victim (line 516)
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // pulls from victim (line 520)
```

The only gate is that `msg.sender` must be authorized to act on behalf of `onBehalf`. There is no check that `callback` (the payer) has authorized `msg.sender` to use it as a payer.

**Exploit flow:**

1. Attacker deploys `maliciousContract` and creates a sell offer with `offer.callback = maliciousContract`.
2. Seller calls `setIsAuthorized(maliciousContract, true, seller)` — standard precondition for any callback that needs to act on the seller's behalf.
3. Any taker calls `take(offer, ...)`.
4. Inside `take()`, `sellerCallback = offer.callback = maliciousContract` (line 421), and the liquidation lock is set for the seller (line 444).
5. After the buyer-side token pull (lines 455–456), `onSell` is invoked on `maliciousContract` (lines 458–473).
6. Inside `onSell`, `maliciousContract` calls `repay(market, units, seller, victim, data)`:
   - `msg.sender = maliciousContract`; `isAuthorized[seller][maliciousContract] = true` → authorization check at line 505 passes.
   - `payer = victim` (line 511).
   - `IRepayCallback(victim).onRepay(...)` is called; if victim returns `CALLBACK_SUCCESS`, execution continues.
   - `safeTransferFrom(loanToken, victim, midnight, units)` pulls `units` loanToken from victim (line 520).
7. Seller's debt is reduced. The post-callback health check at line 476 passes because the debt was repaid.

**Why existing checks fail:**

- The `repay` authorization check (`isAuthorized[onBehalf][msg.sender]`) only verifies the right to repay on behalf of the seller; it says nothing about who may be designated as payer.
- The liquidation lock (line 444) prevents the seller from being liquidated during the callback window but does not restrict what the callback may do to third parties.
- The Certora spec `OnlyExplicitPayerCanLoseTokens.spec` sets `repayCallbackAllowed = false` for `take` (line 108), which would flag a violation if the re-entrant `repay` call were modeled. However, `onSell` is not listed in the methods block and defaults to `HAVOC_ALL` (per the comment at lines 13–14: "onRatify and onSell cannot authorize a payer, so we model them with a plain HAVOC_ALL"). `HAVOC_ALL` havoces storage variables but does not update the persistent ghost variables (`repayCallbackAllowed`, `badPullSeen`), so the tool never explores the re-entrant `repay` path from within `onSell`. The assumption that `onSell` cannot authorize a payer is incorrect.

## Impact Explanation

Direct theft of loanToken from any victim contract that (a) implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS`, (b) has approved Midnight to spend its loanToken, and (c) holds a loanToken balance. The victim loses `units` loanToken; the attacker's (seller's) debt is retired at zero cost. This constitutes unauthorized movement of assets from a third party, satisfying the critical impact class of direct theft.

## Likelihood Explanation

All three required preconditions are normal operational states for Midnight integrators:
1. Seller authorizes their callback contract — standard practice for any callback that calls `supplyCollateral` or similar on the seller's behalf.
2. Victim contract implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS` without validating `onBehalf` — realistic for any Midnight-integrated aggregator or vault that uses repay callbacks for its own operations.
3. Victim has approved Midnight to spend loanToken and holds a loanToken balance — true for any active repay-callback contract.

The attack is repeatable as long as the victim contract remains deployed and approved. The attacker controls the entire setup off-chain before deployment and requires no privileged access.

## Recommendation

Add an explicit authorization check in `repay()` requiring that `callback` has authorized `msg.sender` (or `onBehalf`) to use it as a payer. For example:

```solidity
if (callback != address(0)) {
    require(
        callback == msg.sender || isAuthorized[callback][msg.sender],
        UnauthorizedPayer()
    );
}
```

Alternatively, restrict `callback` to only be `msg.sender` itself (i.e., disallow third-party payer designation entirely), or require `callback == onBehalf`. Additionally, update the Certora spec to model `onSell` as capable of re-entering `repay()` rather than assuming it cannot authorize a payer.

## Proof of Concept

**Minimal test steps:**

1. Deploy `MaliciousCallback` implementing `ISellCallback.onSell` and `IRepayCallback.onRepay`.
2. Deploy `VictimRepayCallback` implementing `IRepayCallback.onRepay` that returns `CALLBACK_SUCCESS` unconditionally, with a loanToken balance and Midnight approval.
3. Attacker (seller) creates a sell offer with `offer.callback = MaliciousCallback`.
4. Seller calls `setIsAuthorized(MaliciousCallback, true)`.
5. Taker calls `take(offer, ...)`.
6. `MaliciousCallback.onSell(...)` is triggered; inside it calls `midnight.repay(market, units, seller, VictimRepayCallback, data)`.
7. `repay()` calls `VictimRepayCallback.onRepay(...)` (returns `CALLBACK_SUCCESS`), then calls `safeTransferFrom(loanToken, VictimRepayCallback, midnight, units)`.
8. Assert: seller's debt is reduced by `units`; `VictimRepayCallback`'s loanToken balance is reduced by `units`; seller paid nothing.
