### Title
Missing deadline parameter in `stake()` allows mempool-delayed transactions to execute post-risk, permanently locking stakers — (File: src/ConfidencePool.sol)

### Summary
`stake()` accepts no deadline parameter. A transaction submitted while the registry is in a pre-risk state can sit in the mempool, execute after the registry transitions to `UNDER_ATTACK`, and permanently strip the staker of their withdrawal right — locking them into a near-zero-bonus, full-CORRUPTED-exposure position they never intended.

### Finding Description
`stake()` in `ConfidencePool.sol` takes only an `amount` argument: [1](#0-0) 

Inside, it calls `_assertDepositsAllowed(_observePoolState())`. `_observePoolState()` reads the live registry state and, if it is `UNDER_ATTACK`, calls `_markRiskWindowStart()`, which sets the one-way latch `riskWindowStart = block.timestamp`: [2](#0-1) [3](#0-2) 

`_assertDepositsAllowed` explicitly permits `UNDER_ATTACK` deposits: [4](#0-3) 

`withdraw()` is permanently disabled once `riskWindowStart != 0`: [5](#0-4) 

**Attack path:** A staker submits `stake()` while the registry is in `NEW_DEPLOYMENT` or `ATTACK_REQUESTED`, expecting to retain the right to `withdraw()` if conditions change. The transaction sits in the mempool (network congestion, low gas price). The registry transitions to `UNDER_ATTACK` before the transaction is mined. When the staker's `stake()` finally executes, `_observePoolState()` reads `UNDER_ATTACK`, sets `riskWindowStart`, and the deposit succeeds — but `withdraw()` is now permanently disabled for every staker, including this one. The staker's entry time is floored at `riskWindowStart ≈ now`, collapsing their `(T − entry)²` bonus score to near zero.

DESIGN.md §3 states "A staker who does not want this can read the live registry state before staking," but this only addresses a staker who *knowingly* submits during `UNDER_ATTACK`. It does not protect against a state transition occurring between transaction submission and execution — the exact gap a deadline parameter closes. [6](#0-5) 

### Impact Explanation
A staker who intended to stake pre-risk with the option to exit loses:
1. **Withdrawal right** — permanently disabled by the `riskWindowStart != 0` one-way latch.
2. **Bonus entitlement** — entry time floored at `riskWindowStart ≈ now` makes `(T − entry)² ≈ 0`, yielding a near-zero share of the bonus pool.
3. **Exit optionality** — principal is now locked until resolution (SURVIVED/EXPIRED/CORRUPTED), with full CORRUPTED exposure.

Principal is not immediately lost, but the staker is forced into a risk position they explicitly did not choose, with no recourse.

### Likelihood Explanation
Medium. The trigger is a registry state transition to `UNDER_ATTACK` occurring while a `stake()` transaction is pending in the mempool — a realistic scenario during network congestion or when a staker uses a conservative gas price. The BattleChain registry transition is an external, unpredictable event. No adversarial actor is required; ordinary mempool latency is sufficient.

### Recommendation
Add a `deadline` parameter to `stake()` and revert if `block.timestamp > deadline`:

```solidity
function stake(uint256 amount, uint256 deadline) external nonReentrant whenPoolNotPaused {
    if (block.timestamp > deadline) revert DeadlineExpired();
    // ... existing logic
}
```

This allows stakers to bound the execution window of their transaction, ensuring a pre-risk submission cannot silently execute post-risk and strip them of their withdrawal right.

### Proof of Concept
1. Registry is in `NEW_DEPLOYMENT`. Alice submits `stake(1000e18)` with a moderate gas price, intending to retain the right to `withdraw()`.
2. Network congestion delays Alice's transaction in the mempool.
3. The BattleChain registry transitions to `UNDER_ATTACK` (off-chain event).
4. Alice's `stake()` is mined. `_observePoolState()` reads `UNDER_ATTACK` and calls `_markRiskWindowStart()`, setting `riskWindowStart = block.timestamp`. `_assertDepositsAllowed` passes (UNDER_ATTACK is allowed). The stake succeeds.
5. Alice calls `withdraw()`. It reverts `WithdrawsDisabled` because `riskWindowStart != 0`.
6. Alice is permanently locked in with `(T − riskWindowStart)² ≈ 0` bonus score and full CORRUPTED exposure — the exact outcome she submitted her transaction to avoid. [7](#0-6) [8](#0-7)

```text
    function _markRiskWindowStart() internal {
        // Cap at expiry: accrual is bounded by the pool's lifecycle. Without the cap, a late
        // observation could pin riskWindowStart > expiry, and `_clampUserSums` would record every
        // pre-risk deposit as entering past T = expiry (EXPIRED path). The k=2 sums then encode
        // post-deadline "at-risk" time that never actually existed.
        uint256 t = block.timestamp;
        if (t > expiry) t = expiry;
        // Cast is truncation-safe: `t` is capped at `expiry`, which is itself a uint32.
        // forge-lint: disable-next-line(unsafe-typecast)
        riskWindowStart = uint32(t);
```

```markdown

- **`UNDER_ATTACK`:** a depositor takes on real, visible, voluntary risk. Their bonus share is
  crushed to ~zero by the k=2 weighting — entry is floored at `riskWindowStart ≈ now`, so
  `(T − entry)² ≈ 0` — and the deposit self-locks into the resolution path (`withdraw` is
  disabled once `riskWindowStart != 0`). There is no late-join *advantage* to gate against and
  no trap: it is voluntary, near-zero-reward risk capital. A staker who does not want this can
  read the live registry state before staking.
- **`PROMOTION_REQUESTED`:** the agreement has requested to exit toward `PRODUCTION`, so the risk
  window is about to close (`riskWindowEnd` imminent). Blocking deposits here stops a late join
  in the final stretch — entering with almost no remaining risk-bearing time before resolution.
  The state is still attackable; it is the *closing window*, not assured survival, that justifies
  the block.

**Why this is not a "trap" finding:** the withdraw-lock from `UNDER_ATTACK` onward is the *same*
intended commitment every staker accepts once risk materializes, and the late entrant earns
negligible bonus. Principal remains recoverable through resolution.
```
