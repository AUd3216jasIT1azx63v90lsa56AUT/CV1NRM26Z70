### Title
`ConsumableUnitsLib.consumableUnits` overflows outside try/catch, reverting bundler transactions for buy offers with large `maxAssets` - (File: src/periphery/ConsumableUnitsLib.sol)

### Summary
When a buy offer has `maxUnits == 0` and `maxAssets` large enough that `remaining * WAD` overflows `uint256`, `ConsumableUnitsLib.consumableUnits` panics inside `TakeAmountsLib.buyerAssetsToUnits`. Because `consumableUnits` is called **outside** the try/catch block in every bundler loop, the panic propagates and reverts the entire transaction. Tokens pulled from the user before the loop are returned by the revert, but gas is permanently lost.

### Finding Description

**Code path:**

`MidnightBundles.buyWithUnitsTargetAndWithdrawCollateral` (and the other three bundler entry-points) pull tokens from the caller, then enter a loop:

```solidity
// src/periphery/MidnightBundles.sol:66-85
pullToken(loanToken, msg.sender, maxBuyerAssets, loanTokenPermit);   // tokens pulled first
...
for (...) {
    uint256 unitsToTake = min(
        targetUnits - filledUnits,
        takes[i].units,
        ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // ← OUTSIDE try/catch
    );
    try IMidnight(MIDNIGHT).take(...) { ... } catch {}   // only take() is guarded
}
```

`ConsumableUnitsLib.consumableUnits` takes the `offer.buy == true`, `offer.maxUnits == 0` branch:

```solidity
// src/periphery/ConsumableUnitsLib.sol:18-19
} else if (offer.buy) {
    return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
```

With `consumed == 0` and `offer.maxAssets = type(uint256).max`, `remaining = type(uint256).max`.

`TakeAmountsLib.buyerAssetsToUnits` then executes:

```solidity
// src/periphery/TakeAmountsLib.sol:29
return offer.buy ? targetBuyerAssets.mulDivUp(WAD, buyerPrice) : ...
```

Which calls:

```solidity
// src/libraries/UtilsLib.sol:34-36
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;   // checked arithmetic — no unchecked block
}
```

`type(uint256).max * WAD` (`≈ 1.157e77 × 1e18`) overflows `uint256`. Solidity 0.8.x checked arithmetic panics. The panic propagates through `consumableUnits` → `min(...)` → the loop body, all of which are **outside** the try/catch. The entire transaction reverts.

**Overflow threshold:** any `offer.maxAssets > type(uint256).max / WAD ≈ 1.157e59` triggers the overflow. `type(uint256).max` is the natural "no cap" sentinel a maker might use.

**Attacker-controlled inputs:** The `takes[]` array is supplied by the caller. The `offer` struct within each take is maker-signed but the maker is an unprivileged user who can freely set `maxAssets` to any `uint256` value. No on-chain validation caps `maxAssets`.

**Why existing checks fail:** The `require(buyerPrice <= WAD, ...)` guard in `buyerAssetsToUnits` runs after the multiplication, so it never executes. The try/catch in the bundler loop only wraps `IMidnight.take`, not the `min(...)` computation that calls `consumableUnits`.

### Impact Explanation
Any taker who routes a buy offer with `maxUnits == 0` and `maxAssets > ~1.157e59` through any of the four bundler entry-points will have their transaction revert. Gas is permanently lost. Tokens are returned by the revert, so there is no token loss. The bundler is rendered unusable for such offers, which includes the natural "no asset cap" pattern (`maxAssets = type(uint256).max`).

### Likelihood Explanation
A maker legitimately setting `maxAssets = type(uint256).max` as an "unlimited" sentinel is a plausible and common pattern. Any taker who then attempts to fill that offer via the bundler triggers the revert. A malicious maker can also deliberately craft such an offer to grief specific takers who rely on the bundler. The condition is easy to reach, requires no special privileges, and is repeatable.

### Recommendation
Cap `remaining` before passing it to `buyerAssetsToUnits` inside `consumableUnits`, or use a saturating/safe multiply in `mulDivUp`. The simplest targeted fix is in `ConsumableUnitsLib`:

```solidity
uint256 remaining = offer.maxAssets.zeroFloorSub(consumed);
// Prevent overflow in mulDivUp: cap remaining so remaining * WAD fits in uint256
uint256 safeRemaining = remaining > type(uint256).max / WAD ? type(uint256).max / WAD : remaining;
return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, safeRemaining);
```

Alternatively, replace `mulDivUp` with a 512-bit full-precision implementation (e.g., Solady's `FullMath.mulDivUp`) so arbitrarily large asset values are handled correctly.
