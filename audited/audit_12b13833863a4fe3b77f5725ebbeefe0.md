### Title
`setIsAuthorized` Race Condition Enables Front-Running of Authorization Revocation, Including Persistent Backdoor Injection - (File: src/Midnight.sol)

### Summary
`Midnight.sol`'s `setIsAuthorized` function is the direct structural analog of the `setSignatureValidatorApproval` function described in the external report. If a user authorizes a contract that is later compromised, an attacker controlling that contract can observe the user's pending revocation transaction in the mempool and front-run it — draining funds or, critically, injecting a persistent secondary authorization that survives the revocation.

### Finding Description

The `setIsAuthorized` function grants an `authorized` address full operator-level access over the `onBehalf` account:

```solidity
// src/Midnight.sol lines 731-735
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
}
``` [1](#0-0) 

An authorized address can call every state-mutating function on behalf of the user:

- `withdraw()` — drain credit [2](#0-1) 
- `withdrawCollateral()` — drain collateral [3](#0-2) 
- `setConsumed()` — cancel all offers [4](#0-3) 
- **`setIsAuthorized()` itself** — authorize additional accounts on behalf of the user [5](#0-4) 

The last capability is the critical amplifier absent from the external report's scenario. Because `setIsAuthorized` checks `isAuthorized[onBehalf][msg.sender]`, a compromised authorized contract can call `setIsAuthorized(AttackerContract, true, Alice)` before Alice's revocation is mined. Alice's revocation of the original contract does **not** remove the newly injected `AttackerContract` authorization — the backdoor persists indefinitely.

The `EcrecoverAuthorizer` periphery contract also calls through to `Midnight.setIsAuthorized`, and the `EcrecoverRatifier` and `SetterRatifier` both re-use the same `isAuthorized` mapping for their own authorization checks, widening the blast radius: [6](#0-5) [7](#0-6) [8](#0-7) 

### Impact Explanation

**Immediate (single-block window):**
- The compromised contract calls `withdraw(market, credit, Alice, attacker)` or `withdrawCollateral(...)` to steal Alice's lent assets or collateral before her revocation lands.

**Persistent (survives revocation):**
- The compromised contract calls `setIsAuthorized(AttackerContract, true, Alice)`. Alice's subsequent `setIsAuthorized(CompromisedContract, false, Alice)` only removes the original contract; `AttackerContract` retains full operator access permanently, enabling continued theft at any future time.

**Cascading (ratifier abuse):**
- Because `EcrecoverRatifier.isRatified` and `SetterRatifier.setIsRootRatified` both gate on `IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer)`, a backdoored authorization also allows the attacker to ratify arbitrary offers signed by any key they control, executing trades against Alice's position at attacker-chosen prices.

### Likelihood Explanation

- Front-running pending Ethereum transactions is a well-documented, tooled, and routinely exploited technique (MEV bots, Flashbots bundles).
- The attacker requires control of the previously authorized contract — a user-level delegation, not a governance or admin key. Smart-contract operators, ratifiers, and bundlers are routinely authorized by users; any bug or malicious upgrade in such a contract satisfies this precondition.
- The `MidnightBundles` periphery explicitly documents that users must authorize it: `"The taker must have authorized this bundler"` — making this a realistic, expected authorization pattern. [9](#0-8) 

### Recommendation

**Short term:** Document in `setIsAuthorized` and in user-facing materials that:
1. Revoking an authorization does **not** protect against front-running in the revocation window.
2. An authorized contract can grant further authorizations; users must audit every contract they authorize.
3. If a contract is suspected compromised, users should also audit whether it has injected secondary authorizations before assuming revocation is sufficient.

**Long term:** Consider emitting a `SetIsAuthorized` event that off-chain monitors can watch to detect unexpected secondary authorization grants. Optionally, introduce a time-locked revocation path (commit-reveal or delay) so that revocations cannot be front-run.

### Proof of Concept

```
1. Alice calls setIsAuthorized(OperatorContract, true, Alice).
   → isAuthorized[Alice][OperatorContract] = true

2. OperatorContract is compromised (bug, malicious upgrade, etc.).

3. Alice submits setIsAuthorized(OperatorContract, false, Alice) to revoke.
   → Transaction is visible in the public mempool, not yet mined.

4. Attacker observes Alice's pending revocation.

5. Attacker (controlling OperatorContract) submits with higher gas:
   a. OperatorContract calls setIsAuthorized(AttackerEOA_or_Contract, true, Alice)
      → isAuthorized[Alice][AttackerContract] = true  ← backdoor planted
   b. OperatorContract calls withdraw(market, Alice.credit, Alice, attacker)
      → Alice's credit drained to attacker

6. Alice's revocation mines:
   → isAuthorized[Alice][OperatorContract] = false
   → But isAuthorized[Alice][AttackerContract] = true  ← backdoor survives

7. Attacker later calls withdrawCollateral(..., Alice, attacker) via AttackerContract,
   draining remaining collateral at leisure.
```

Root cause: `setIsAuthorized` is a single-step, immediately-effective state write with no revocation delay, and the authorized address has recursive authorization-granting power over the same mapping. [1](#0-0)
