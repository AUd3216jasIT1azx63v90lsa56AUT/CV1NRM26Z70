### Title
Liquidator-controlled `callback` parameter designates arbitrary third-party as payer, draining its `loanToken` balance while liquidator receives seized collateral - (`src/Midnight.sol`)

### Summary
In `Midnight.liquidate`, the `callback` parameter is supplied by the liquidator (`msg.sender`) and is unconditionally set as the `payer` for the loan repayment `safeTransferFrom`. There is no check that the callback address has authorized the liquidator to act on its behalf. Any contract that implements `ILiquidateCallback`, returns `CALLBACK_SUCCESS` without verifying the caller identity, and holds a `loanToken` approval to Midnight can be exploited: the liquidator receives seized collateral at `receiver` while the third-party callback bears the full repayment cost.

### Finding Description

**Exact code path** in `src/Midnight.sol`:

```
liquidate(market, collateralIndex, seizedAssets, repaidUnits, borrower, false, receiver=attacker, callback=thirdParty, data)
```

**Step 1 — payer is set to callback with no authorization check** (line 679):
```solidity
address payer = callback != address(0) ? callback : msg.sender;
```
There is no `require(callback == msg.sender || isAuthorized[callback][msg.sender])` or equivalent. The liquidator freely designates any address as payer.

**Step 2 — collateral is transferred to `receiver` (attacker) before the callback fires** (line 696):
```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
```
At this point the attacker already holds the seized collateral.

**Step 3 — `onLiquidate` is called on `thirdParty`** (lines 698–714):
```solidity
ILiquidateCallback(callback).onLiquidate(msg.sender, id, market, ...) == CALLBACK_SUCCESS
```
The callback receives `msg.sender` (the liquidator) as its first argument. If `thirdParty` does not validate this argument and returns `CALLBACK_SUCCESS`, execution continues.

**Step 4 — loan token is pulled from `payer` = `thirdParty`** (line 717):
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```
`thirdParty`'s `loanToken` balance is debited for the full repayment.

**Why existing checks do not stop it:**
- The `liquidatorGate` check (line 598) only verifies that `msg.sender` is permitted to liquidate; it says nothing about who the callback/payer is.
- The Certora property `OnlyExplicitPayerCanLoseTokens.spec` (line 80) only asserts that the pull comes from the callback that returned `CALLBACK_SUCCESS` — it does not assert that the callback consented to being the payer for this particular liquidator.
- `ILiquidateCallback.onLiquidate` (line 17 of `src/interfaces/ICallbacks.sol`) exposes `caller` as the first argument, but the protocol never enforces that the callback validates it.

**Contrast with `take`** (line 422 of `src/Midnight.sol`):
```solidity
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
```
Here the callback is always set by the party that will be the payer (maker sets `offer.callback`, taker sets `takerCallback`). In `liquidate` the liquidator sets the callback but the callback pays — a broken symmetry with no compensating authorization check.

### Impact Explanation
An unprivileged liquidator profits from collateral seizure (`seizedAssets` sent to `receiver = attacker`) while a third-party contract bears the entire repayment cost (`repaidUnits` pulled from `thirdParty`). The invariant that liquidation repayment must come from the liquidator (or a party the liquidator is authorized to act for) is violated. The third party suffers a direct, unrecoverable loss of `loanToken` proportional to the liquidated debt.

### Likelihood Explanation
**Preconditions:**
1. An unhealthy borrower position exists (normal market condition).
2. The attacker passes any `liquidatorGate` check (or the market has none).
3. A `thirdParty` contract exists that: (a) implements `ILiquidateCallback` and returns `CALLBACK_SUCCESS` without validating the `caller` argument, and (b) has granted Midnight a `loanToken` allowance ≥ `repaidUnits`.

Condition 3 is realistic for any liquidation-infrastructure contract (flash-liquidation bots, vault integrations, keeper contracts) that has pre-approved Midnight and implements the callback interface without a caller guard. The attack is repeatable as long as the third party's allowance and balance remain sufficient.

### Recommendation
Add an authorization check in `liquidate` before accepting an arbitrary `callback` as payer:

```solidity
require(
    callback == address(0) || callback == msg.sender || isAuthorized[callback][msg.sender],
    CallbackUnauthorized()
);
```

This mirrors the pattern used for `taker` in `take` (line 346) and for `onBehalf` in `withdrawCollateral` (line 556), ensuring the callback/payer has explicitly authorized the liquidator to act on its behalf.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract MaliciousLiquidator {
    // Attacker calls this; thirdParty is a deployed contract that:
    //   - implements ILiquidateCallback, returns CALLBACK_SUCCESS unconditionally
    //   - has approved Midnight for loanToken with sufficient allowance
    function exploit(
        Midnight midnight,
        Market calldata market,
        uint256 collateralIndex,
        uint256 seizedAssets,
        address borrower,
        address thirdParty,
        address attacker
    ) external {
        // receiver = attacker, callback = thirdParty
        midnight.liquidate(
            market, collateralIndex, seizedAssets, 0,
            borrower, false, attacker, thirdParty, ""
        );
        // Assert: attacker.collateralBalance increased by seizedAssets
        // Assert: thirdParty.loanTokenBalance decreased by repaidUnits
    }
}

contract VulnerableCallback is ILiquidateCallback {
    // Simulates a liquidation-infrastructure contract that has pre-approved Midnight
    // and does NOT validate the caller argument in onLiquidate.
    function onLiquidate(
        address, /* caller — NOT checked */
        bytes32, Market memory market,
        uint256, uint256, uint256 repaidUnits,
        address, address, bytes memory, uint256
    ) external returns (bytes32) {
        // Approve Midnight to pull repaidUnits (already done via ERC20.approve in setup)
        return CALLBACK_SUCCESS;
    }
}
```
