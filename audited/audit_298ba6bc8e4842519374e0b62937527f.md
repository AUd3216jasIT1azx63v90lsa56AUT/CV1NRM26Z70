### Title
Zero-Cost Credit/Debt Creation via `mulDivDown` Truncation on Buy Offer with `units=1` and `buyerPrice < WAD` - (File: src/Midnight.sol)

### Summary
When a buy offer is taken with `units=1` and `buyerPrice < WAD`, the expression `units.mulDivDown(buyerPrice, WAD)` truncates to zero, yielding `buyerAssets = 0` and `sellerAssets = 0`. The protocol still executes full credit/debt accounting for `units` worth of position state while performing zero token transfers. The protocol's own test suite names and demonstrates this exact scenario in `testBugBuyMaxAssetsBypass`.

### Finding Description
In `src/Midnight.sol` lines 363–364, for a buy offer:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD);
```

`mulDivDown` in `src/libraries/UtilsLib.sol` line 30 is `(x * y) / d`. With `x=1`, `y=buyerPrice < WAD`, `d=WAD`, integer division yields `0`. Since `sellerPrice ≤ buyerPrice < WAD`, `sellerAssets = 0` as well.

The consumed-cap check at line 368 then adds `buyerAssets = 0` to `consumed`:

```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

Because `consumed` does not increase, the cap check passes even when the offer is already fully consumed. Position accounting at lines 382–414 still executes in full: `buyerCreditIncrease = zeroFloorSub(1, buyerPos.debt)` and `sellerDebtIncrease = 1 - sellerCreditDecrease`, updating `buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` by 1 unit. The two `safeTransferFrom` calls at lines 455–456 transfer `0` tokens each.

**Exploit flow:**
1. Lender (maker) creates a buy offer at any tick where `tickToPrice(tick) < WAD` (any tick < `MAX_TICK` with zero settlement fee, or appropriate tick/fee combination). This is the normal operating range.
2. Attacker (taker/borrower) calls `take(offer, ..., units=1, ...)`.
3. `buyerAssets = 0`, `sellerAssets = 0` — zero tokens transferred.
4. Lender gains `+1` credit unit; borrower gains `+1` debt unit — for free.
5. If `offer.maxAssets > 0` and already fully consumed, the cap is bypassed entirely since `consumed += 0`.
6. Step 2–5 can be repeated arbitrarily.

No existing check prevents this. The Certora spec `takeInputOutputConsistency` in `certora/specs/Midnight.spec` line 72 only asserts `unitsInput == 0 => buyerAssetsOutput == 0`, not the converse. There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in `take`.

### Impact Explanation
An attacker can accumulate unbounded credit units at zero token cost by repeatedly calling `take` with `units=1` on any buy offer with `buyerPrice < WAD`. Each call grants the lender/maker one free credit unit redeemable via `withdraw`, draining loan tokens from the pool that were deposited by other lenders. Simultaneously, the colluding borrower accumulates debt units without receiving any loan tokens, corrupting the credit/debt invariant that every credit must have a matching funded debt or settled state. The `maxAssets`-based offer cap is also fully bypassed, allowing unlimited takes on a nominally exhausted offer.

### Likelihood Explanation
The precondition — a buy offer at any tick below `MAX_TICK` with zero or low settlement fee — describes the normal operating range of the protocol. `tickToPrice(tick) < WAD` for all ticks below `MAX_TICK`. The attacker needs only to be the taker on such an offer (unprivileged). The attack is repeatable in a single transaction via `multicall`. The protocol's own test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` lines 857–889 explicitly demonstrates and asserts this exact state: `buyerAssets == 0`, `sellerAssets == 0`, token balances unchanged, yet `creditOf > before` and `debtOf > before`.

### Recommendation
Add a guard in `take` that requires either `units == 0` or both computed asset amounts are positive for the relevant side. Specifically, for buy offers, require `units == 0 || buyerAssets > 0`. Equivalently, enforce a minimum `units` such that `units * buyerPrice >= WAD` before proceeding with position accounting. A simpler global fix: `require(units == 0 || (buyerAssets > 0 && sellerAssets > 0) || settlementFee == 0)` — or more precisely, revert when `units > 0 && buyerAssets == 0` for buy offers and `units > 0 && sellerAssets == 0` for sell offers.

### Proof of Concept
```solidity
// Foundry unit test mirroring testBugBuyMaxAssetsBypass
function testZeroCostCreditDebt() public {
    // Setup: buy offer (lender is maker) at tick < MAX_TICK so buyerPrice < WAD
    lenderOffer.buy = true;
    lenderOffer.tick = MAX_TICK - 16; // tickToPrice < WAD
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1; // assets-based cap, already consumed below

    // Pre-consume the offer fully
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, 1, lender);

    uint256 creditBefore = midnight.creditOf(id, lender);
    uint256 debtBefore   = midnight.debtOf(id, borrower);
    uint256 unitsBefore  = midnight.totalUnits(id);

    collateralize(market, borrower, 100);
    (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

    // Assertions proving the bug:
    assertEq(buyerAssets, 0,  "buyerAssets must be 0 (truncation)");
    assertEq(sellerAssets, 0, "sellerAssets must be 0");
    // Cap not consumed — offer bypass confirmed
    assertEq(midnight.consumed(lender, lenderOffer.group), 1, "consumed unchanged");
    // But position state changed — free credit/debt
    assertGt(midnight.creditOf(id, lender),   creditBefore, "lender gained free credit");
    assertGt(midnight.debtOf(id, borrower),   debtBefore,   "borrower gained free debt");
    assertGt(midnight.totalUnits(id),          unitsBefore,  "totalUnits increased");
}
```
