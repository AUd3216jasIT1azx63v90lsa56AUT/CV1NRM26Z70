### Title
`mulDivDown` fee rounding makes `targetFilledSellerAssets` strictly less than exact rational target, causing minor effective-fee-rate divergence - (File: src/periphery/MidnightBundles.sol)

### Summary
In `supplyCollateralAndSellWithAssetsTarget`, the referral fee is computed as `targetSellerAssets.mulDivDown(referralFeePct, WAD - referralFeePct)`, which floors the result. This makes `targetFilledSellerAssets` at most 1 wei less than the exact rational value `targetSellerAssets * WAD / (WAD - referralFeePct)` whenever `targetSellerAssets * referralFeePct` is not divisible by `WAD - referralFeePct`. The referrer receives at most 1 wei less than the exact fee, and the effective fee rate is slightly below the intended `referralFeePct / WAD`.

### Finding Description
The relevant code is at `src/periphery/MidnightBundles.sol` lines 277–278:

```solidity
uint256 referralFeeAssets = targetSellerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
uint256 targetFilledSellerAssets = targetSellerAssets + referralFeeAssets;
```

Let `T = targetSellerAssets`, `p = referralFeePct`, `D = WAD - p`.

The exact rational fill target is `T * WAD / D = T + T*p/D`. The computed value is `T + floor(T*p/D)`. Whenever `T*p mod D != 0`, `floor(T*p/D) < T*p/D`, so `targetFilledSellerAssets < T * WAD / D` by up to `(D-1)/D`, i.e., strictly 1 wei short.

The fill loop enforces `require(filledSellerAssets == targetFilledSellerAssets)` (line 303), so the taker fills exactly the rounded-down target. The receiver always gets `targetSellerAssets` (line 307), but the referrer gets `floor(T*p/D)` instead of the exact rational fee. No existing check prevents this: `require(referralFeePct < WAD)` (line 263) only guards against 100% fee.

Any unprivileged taker (borrower acting as seller) can trigger this by calling the function with any `(targetSellerAssets, referralFeePct)` pair where `targetSellerAssets * referralFeePct % (WAD - referralFeePct) != 0`. This is the common case for arbitrary inputs.

The existing fuzz test `testSellSellerAssetsTargetWithReferralFee` (test/MidnightBundlesTest.sol line 639) uses the same `mulDivDown` formula for `expectedFee`, so it mirrors the implementation's rounding and does not detect the divergence from the exact rational value.

### Impact Explanation
The referrer receives at most 1 wei less than the exact fee per call. The effective fee rate `referralFeeAssets / targetFilledSellerAssets` is slightly below the intended `referralFeePct / WAD`. The receiver is unaffected (always receives exactly `targetSellerAssets`). No funds are locked or lost beyond the 1-wei rounding shortfall to the referrer.

### Likelihood Explanation
Triggered on every call where `targetSellerAssets * referralFeePct % (WAD - referralFeePct) != 0`, which is the generic case for arbitrary `uint256` inputs. No special preconditions, permissions, or state are required. Any taker can trigger it on every invocation with a non-zero `referralFeePct`.

### Recommendation
Replace the two-step fee derivation with a single `mulDivUp` on the total fill target, then derive the fee as the remainder:

```solidity
uint256 targetFilledSellerAssets = targetSellerAssets.mulDivUp(WAD, WAD - referralFeePct);
uint256 referralFeeAssets = targetFilledSellerAssets - targetSellerAssets;
```

This computes `ceil(T * WAD / D)`, ensuring the total filled is at least the exact rational value. The receiver still receives exactly `targetSellerAssets`; the referrer receives `ceil(T*WAD/D) - T`, which rounds in the referrer's favor by at most 1 wei and keeps the effective fee rate at or above `referralFeePct / WAD`.

### Proof of Concept
```solidity
function testFuzz_SellAssetsTargetFeeRoundingDivergence(
    uint256 targetSellerAssets,
    uint256 referralFeePct
) public {
    targetSellerAssets = bound(targetSellerAssets, 1, type(uint128).max);
    referralFeePct = bound(referralFeePct, 1, WAD - 1);

    uint256 D = WAD - referralFeePct;

    // Computed values (implementation)
    uint256 referralFeeAssets = targetSellerAssets.mulDivDown(referralFeePct, D);
    uint256 targetFilledSellerAssets = targetSellerAssets + referralFeeAssets;

    // Exact rational target (scaled to avoid fractions)
    // targetSellerAssets * WAD / D  =>  targetFilledSellerAssets * D vs targetSellerAssets * WAD
    uint256 lhs = targetFilledSellerAssets * D;   // computed * D
    uint256 rhs = targetSellerAssets * WAD;        // exact numerator (no overflow for uint128 inputs)

    // Assert: whenever rounding occurs, computed is strictly below exact
    if (targetSellerAssets * referralFeePct % D != 0) {
        assertLt(lhs, rhs, "targetFilledSellerAssets is below exact rational target");
        // Effective fee rate is below intended
        // referralFeeAssets / targetFilledSellerAssets < referralFeePct / WAD
        assertLt(
            referralFeeAssets * WAD,
            referralFeePct * targetFilledSellerAssets,
            "effective fee rate below referralFeePct/WAD"
        );
    } else {
        assertEq(lhs, rhs, "exact division: no divergence expected");
    }
}
```
