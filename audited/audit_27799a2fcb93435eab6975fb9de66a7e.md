Audit Report

## Title
Reentrant `withdraw` via `onBuy` callback drains pre-existing `withdrawable` before payer tokens are pulled - (File: `src/Midnight.sol`)

## Summary
In `take`, `buyerPos.credit` is incremented at line 410 before the `onBuy` callback fires at lines 448–452, while `_marketState.withdrawable` is never increased during `take`. Because `withdraw` has no reentrancy guard and only checks `isAuthorized`, a malicious `buyerCallback` (self-authorized by the attacker) can call `withdraw` on behalf of the buyer inside `onBuy`, consuming the freshly-granted credit to drain pre-existing `withdrawable` tokens. The payer `safeTransferFrom` executes only after `onBuy` returns (lines 455–456), and when `buyerPrice < WAD` the protocol receives fewer tokens than it sent out, permanently breaking the solvency invariant.

## Finding Description

**Root cause — storage ordering in `take`:**
`buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease)` is committed to storage at line 410 before any external call. `_marketState.withdrawable` is never touched inside `take` (only `totalUnits` at line 416 and `claimableSettlementFee` at line 418 are updated).

**Callback fires before token pull:**
`IBuyCallback(buyerCallback).onBuy(...)` is called at lines 448–452. The `safeTransferFrom` calls that pull tokens from `payer` happen only at lines 455–456, after `onBuy` returns.

**`withdraw` has no reentrancy guard:**
`withdraw` only checks `isAuthorized[onBehalf][msg.sender]` (line 482). It then decrements `_position.credit` (line 493), `_marketState.withdrawable` (line 494), and transfers tokens (line 499). There is no guard preventing it from being called while a `take` is in progress.

**The `LIQUIDATION_LOCK_SLOT` does not help:**
The transient lock set at line 444 protects only the *seller* from liquidation during callbacks. It does not block reentrant calls to `withdraw` for the buyer.

**`buyerPrice < WAD` is reachable:**
`tickToPrice(0)` evaluates to approximately `5e11` (well below `WAD = 1e18`), since `wExp(LN_ONE_PLUS_DELTA * 2910) ≈ 2e6 * 1e18`, making `1e36 / (1e18 + 2e24) ≈ 5e11`. Tick 0 is accessible because `DEFAULT_TICK_SPACING = 4` and `0 % 4 == 0`. For a sell offer, `buyerPrice = offerPrice + _settlementFee ≈ 5e11 + small << WAD`.

**`_updatePosition` inside `withdraw` is a no-op during the same block:**
`take` calls `_updatePosition` for the buyer at line 379 (before incrementing credit), setting `lastLossFactor = marketState[id].lossFactor` and `lastAccrual = block.timestamp`. When `withdraw` calls `_updatePosition` again inside `onBuy`, `updatePositionView` computes zero slashing (same `lossFactor`) and zero fee accrual (`accrualEnd - lastAccrual = 0`), so credit remains exactly `buyerCreditIncrease = units`.

**Exploit flow:**
1. Attacker deploys `MaliciousCallback` implementing `IBuyCallback`, pre-funded with `buyerAssets` tokens approved to Midnight.
2. Attacker calls `setIsAuthorized(callback, true)` → `isAuthorized[attacker][callback] = true`.
3. A second attacker-controlled address creates a sell offer at tick 0 (bypassing the `offer.maker != taker` self-take check at line 354).
4. Attacker calls `take(sellOffer, ..., units, attacker, ..., callback, ...)`.
5. `take` sets `buyerPos.credit += units` (line 410). `withdrawable` is unchanged.
6. `take` calls `callback.onBuy(...)` (lines 448–452).
7. Inside `onBuy`, callback calls `midnight.withdraw(market, units, attacker, attacker)`:
   - Auth check passes (`isAuthorized[attacker][callback]` = true).
   - `_updatePosition` is a no-op (same block).
   - `attacker.credit -= units` (zeroes the credit just granted, line 493).
   - `withdrawable -= units` (drains pre-existing pool, line 494).
   - Transfers `units` tokens to attacker (line 499).
8. `onBuy` returns `CALLBACK_SUCCESS`.
9. `take` calls `safeTransferFrom(callback, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(callback, receiver, sellerAssets)` — total `buyerAssets = units * buyerPrice / WAD << units` tokens pulled from callback.
10. Net protocol loss: `units − buyerAssets > 0` tokens per execution.

**Why existing checks fail:**
- No `nonReentrant` modifier on `take` or `withdraw`.
- The token safety requirement ("should not re-enter Midnight on transfer") covers token callbacks only, not `onBuy` callbacks.
- The `isHealthy` check at line 476 applies to the seller, not the buyer.
- The `LIQUIDATION_LOCK_SLOT` only guards seller liquidation.

## Impact Explanation
The protocol's token balance decreases by `units − buyerAssets` per attack. Pre-existing `withdrawable` (belonging to other lenders) is drained at a discount. At tick 0 with zero settlement fee, `buyerAssets ≈ 0` and the attacker extracts the full `units` tokens for free. The core solvency invariant — contract token balance covers all withdrawable assets — is violated: `withdrawable` is decremented but the corresponding tokens are not replaced by the payer. This constitutes direct theft of lender funds.

## Likelihood Explanation
All three preconditions are trivially satisfiable by an unprivileged attacker: (1) `withdrawable > 0` — satisfied in any market that has seen at least one repayment or liquidation; (2) `isAuthorized[attacker][callback]` — self-granted by the attacker in a single transaction before calling `take`; (3) a sell offer with `buyerPrice < WAD` — the attacker can use a second controlled address to create one at tick 0. The attack is fully permissionless, requires no privileged role, and is repeatable as long as `withdrawable > 0`.

## Recommendation
Apply a reentrancy guard to `withdraw` (and ideally to all state-mutating external entry points), or restructure `take` so that the payer token pull occurs before the `onBuy` callback fires. A minimal fix is to add a transient-storage reentrancy lock (consistent with the existing `LIQUIDATION_LOCK_SLOT` pattern) that is set at the start of `take` and checked at the start of `withdraw`, preventing `withdraw` from being called while a `take` is in progress.
