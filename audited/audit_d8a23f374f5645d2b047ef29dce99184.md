Audit Report

## Title
Taker-Supplied `proof.length > 20` Causes `TreeTooHigh` to Propagate Through `Midnight.take` Instead of `RatifierFail` - (File: src/ratifiers/EcrecoverRatifier.sol)

## Summary
Any taker calling `Midnight.take` with an offer whose `ratifier` is `EcrecoverRatifier` can supply `ratifierData` encoding a 21-element Merkle proof. This causes `HashLib.offerTreeTypeHash(21)` to revert with `TreeTooHigh` inside `EcrecoverRatifier.isRatified`. Because `Midnight.take` wraps the ratifier call with a plain `require`, the revert is not caught and re-thrown as `RatifierFail`; it propagates verbatim as `TreeTooHigh`, breaking the protocol's error-surface contract for all callers.

## Finding Description

**`Midnight.take` line 356** uses a plain `require` to call the ratifier:

```solidity
require(IRatifier(offer.ratifier).isRatified(offer, ratifierData) == CALLBACK_SUCCESS, RatifierFail());
``` [1](#0-0) 

This pattern converts only a *wrong return value* into `RatifierFail`. A revert originating inside `isRatified` bypasses this conversion entirely.

**Inside `EcrecoverRatifier.isRatified`**, the execution order is:

1. **Line 37** — `HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof)` — all four arguments are taker-controlled via `ratifierData`.
2. **Line 38** — `isRootCanceled[offer.maker][root]` — taker controls `root`.
3. **Line 39** — `HashLib.offerTreeTypeHash(proof.length)` — **reverts with `TreeTooHigh()` when `proof.length > 20`**, before any signature validation. [2](#0-1) 

**`HashLib.offerTreeTypeHash`** has no branch for values above 20 and unconditionally reverts: [3](#0-2) 

**`HashLib.isLeaf` does not block this**: it only checks `leafIndex >> proof.length == 0`. With `leafIndex = 0` and `proof.length = 21`, the guard passes (`0 >> 21 == 0`). An attacker precomputes a valid 21-level Merkle root by starting from `HashLib.hashOffer(offer)` and applying 21 chosen sibling hashes; `isLeaf` returns `true`. [4](#0-3) 

**The signature check does not block this**: `offerTreeTypeHash` is called at line 39, before `ecrecover` at line 42. The revert fires before any signature validation, so no knowledge of the maker's private key is required. [5](#0-4) 

## Impact Explanation
`Midnight.take` reverts with `HashLib.TreeTooHigh` instead of `IMidnight.RatifierFail`. Any smart-contract caller (router, aggregator, multicall wrapper) that uses `try/catch` or selector-based error handling expecting only documented `Midnight` errors will encounter an unhandled revert. The protocol's implicit invariant — that all ratifier-related failures surface as `RatifierFail` — is broken for `EcrecoverRatifier`.

## Likelihood Explanation
The precondition is minimal: the taker must call `Midnight.take` with an offer whose `ratifier` is `EcrecoverRatifier` and supply `ratifierData` encoding a 21-element proof array. No privileged access, no special state, and no valid signature are required. The trigger is deterministic and repeatable on every call.

## Recommendation
Add an explicit upper-bound check on `proof.length` inside `EcrecoverRatifier.isRatified` before calling `offerTreeTypeHash`, and revert with a ratifier-specific error (e.g., `InvalidProof`) when `proof.length > 20`. This ensures all invalid-proof conditions are caught and surfaced as a known ratifier error before reaching `offerTreeTypeHash`.

```solidity
require(proof.length <= 20, InvalidProof());
```

This check should be placed before the `isLeaf` call at line 37, or at minimum before the `offerTreeTypeHash` call at line 39.

## Proof of Concept
1. Deploy `EcrecoverRatifier` and `Midnight` on a fork/test environment.
2. Create a valid `Offer` with `ratifier = address(ecrecoverRatifier)`.
3. Authorize the ratifier via `setIsAuthorized`.
4. Build `ratifierData` with `proof = new bytes32[](21)`, `leafIndex = 0`, and `root` computed by applying 21 rounds of `HashLib.hashNode` starting from `HashLib.hashOffer(offer)` with arbitrary siblings. Any signature values suffice (they are never reached).
5. Call `Midnight.take` with this `ratifierData`.
6. Observe the transaction reverts with `HashLib.TreeTooHigh` selector, not `IMidnight.RatifierFail`.

The existing test `testOfferTreeTypeHashInvalidHeight` in `test/HashLibTest.sol` already confirms `offerTreeTypeHash` reverts with `TreeTooHigh` for `height >= 21`. [6](#0-5)

    function isLeaf(bytes32 root, bytes32 leafHash, uint256 leafIndex, bytes32[] memory proof)
        internal
        pure
        returns (bool)
    {
        require(leafIndex >> proof.length == 0, LeafIndexOutOfRange());
        bytes32 currentHash = leafHash;
        for (uint256 i = 0; i < proof.length; i++) {
            currentHash = (leafIndex >> i) & 1 == 0 ? hashNode(currentHash, proof[i]) : hashNode(proof[i], currentHash);
        }
        return currentHash == root;
    }
