### Title
Buy-offer `maxAssets` cap bypassed via zero-rounding of `buyerAssets` in `consumed` accounting — (`src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and `tick < MAX_TICK` (i.e., `offerPrice < WAD`), calling `take()` with `units = 1` produces `buyerAssets = 1.mulDivDown(offerPrice, WAD) = 0` because `offerPrice < WAD`. The consumed accounting branch adds `buyerAssets` (zero) to `consumed[maker][group]`, so the cap check `require(newConsumed <= offer.maxAssets)` always passes and the counter never advances. The offer can therefore be taken an unlimited number of times, each time granting the maker one unit of credit and imposing one unit of debt on the taker, with no token transfer and no cap consumption.

### Finding Description
**Exact code path — `src/Midnight.sol` `take()`, lines 358–369:**

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);          // line 358
// For offer.buy == true: buyerPrice == offerPrice (settlement fee cancels)
uint256 buyerAssets = units.mulDivDown(buyerPrice, WAD);       // line 363

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());  // line 369
}
```

**Price range:** `tickToPrice(0) == 0`, `tickToPrice(2) == 1e12`, `tickToPrice(MAX_TICK) == 1e18 = WAD`. Every tick below `MAX_TICK` yields `offerPrice < WAD`.

**Zero-rounding condition:** With `units = 1` and any `tick < MAX_TICK`:
```
buyerAssets = 1 * offerPrice / WAD = 0   (integer division, offerPrice < WAD)
```

**Consumed delta:** `consumed[maker][group] += 0` — the cap is never consumed.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick` any value `< MAX_TICK`
- `units = 1` (or any value where `units * offerPrice < WAD`)
- Ratifier must pass (maker controls this, but permissive ratifiers are the common case)

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — passes because `newConsumed == consumed` (unchanged)
- `require(UtilsLib.atMostOneNonZero(offer.maxAssets, offer.maxUnits))` — passes
- No check enforces `buyerAssets > 0` when `units > 0`

**Position state still mutates:** Even with `buyerAssets = 0`, the position update at lines 408–414 runs unconditionally: `buyerPos.credit += 1`, `sellerPos.debt += 1`, `totalUnits += 1`. The test `testBugBuyMaxAssetsBypass` (line 858–889) explicitly confirms this: credit, debt, and totalUnits all increase while `consumed` stays at `maxAssets` and no tokens move.

The protocol's own NatSpec at line 94 acknowledges the fully-consumed variant: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The not-yet-consumed case is the same root cause: `consumed` never reaches `maxAssets` because every increment is zero.

### Impact Explanation
The `maxAssets` cap — the maker's sole mechanism to bound total credit issued under a buy offer — is permanently bypassed for any `tick < MAX_TICK`. A taker can call `take(1, ...)` in a loop, incrementing the maker's credit by 1 unit per call with zero asset cost to the maker and zero asset receipt to the taker. The offer is never marked consumed, violating the invariant *"offers cannot be replayed, overfilled, reused, or filled after cancel/deadline."*

### Likelihood Explanation
**Preconditions:**
1. Buy offer with `maxAssets > 0` — a standard cap configuration.
2. `tick < MAX_TICK` — satisfied by every tick except the single maximum tick (5820); the vast majority of real offers use lower ticks.
3. `_settlementFee <= offerPrice` — almost always true; max settlement fee is 0.5% for 360-day markets, while even `tick = 4` gives `offerPrice > 0`.
4. Permissive ratifier (e.g., `DummyRatifier`) — common in practice.

The condition is met by essentially all real buy offers with an asset cap. The attack requires no special privilege, no oracle manipulation, and no flash loan. It is repeatable indefinitely within a single transaction via `multicall`.

### Recommendation
Add a guard that prevents a non-zero-unit take from contributing zero to the consumed counter when `maxAssets > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 consumedDelta = offer.buy ? buyerAssets : sellerAssets;
    // Prevent zero-rounding from bypassing the cap on non-trivial takes.
    require(units == 0 || consumedDelta > 0, ZeroAssetsTake());
    newConsumed = consumed[offer.maker][offer.group] += consumedDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, round the consumed delta up (`mulDivUp`) so that every unit taken advances the counter by at least 1, consistent with how sell-offer taker assets are already rounded up at line 364.

### Proof of Concept
The existing `testBugBuyMaxAssetsBypass` (line 858) already proves the mechanism for a fully-consumed offer. The following extends it to the not-yet-consumed case:

```solidity
function testMaxAssetsBypassNotYetConsumed() public {
    // Buy offer: maker=lender, taker=borrower. tick < MAX_TICK → offerPrice < WAD.
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1000;
    lenderOffer.tick = MAX_TICK - 4; // offerPrice < WAD

    collateralize(market, borrower, 500); // enough collateral for 500 units of debt

    // Each take with units=1 should give buyerAssets=0 and consumed+=0.
    for (uint256 i = 0; i < 500; i++) {
        (uint256 buyerAssets,) = take(1, borrower, lenderOffer);
        assertEq(buyerAssets, 0, "buyerAssets must be 0");
        // INVARIANT: consumed must not exceed maxAssets, but also must advance.
        uint256 c = midnight.consumed(lender, lenderOffer.group);
        assertEq(c, 0, "consumed never advances — cap bypassed");
    }

    // After 500 takes the maker has 500 units of credit, cap was 1000 assets.
    // consumed is still 0, so the offer appears completely unfilled.
    assertEq(midnight.creditOf(id, lender), 500);
    assertEq(midnight.consumed(lender, lenderOffer.group), 0); // BUG: should be > 0
}
```
