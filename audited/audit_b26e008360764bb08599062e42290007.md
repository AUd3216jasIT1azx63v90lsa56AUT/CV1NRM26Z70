Audit Report

## Title
Zero-asset rounding bypass allows unlimited takes on fully-consumed buy offers - (File: src/Midnight.sol)

## Summary
When a buy offer has `maxAssets > 0` and `offerPrice < WAD`, the consumed-assets increment uses `mulDivDown`, which truncates to zero for small `units`. Because `consumed` does not advance, the guard `newConsumed <= offer.maxAssets` passes even when the offer is already fully consumed. An unprivileged taker can repeat this indefinitely, inflating credit, debt, and `totalUnits` without transferring any tokens.

## Finding Description

**Exact code path** — `src/Midnight.sol` lines 363 and 367–369:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [1](#0-0) 

**Root cause** — `consumed` is incremented by `buyerAssets`, not by `units`. `buyerAssets` is computed with `mulDivDown`. When `buyerPrice < WAD` (achievable by setting `tick` below `MAX_TICK` by ≥16 ticks, a normal reachable range) and `units = 1`, `mulDivDown(1, buyerPrice, WAD) = 0`. The increment is zero, so `newConsumed = consumed + 0 = consumed`. The guard `require(newConsumed <= offer.maxAssets)` is satisfied even when `consumed == maxAssets`.

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = M`, `tick = MAX_TICK - 16` (so `buyerPrice < WAD`).
2. Offer is filled normally until `consumed[maker][group] == M`.
3. Taker calls `take(offer, ..., units=1)`. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. `newConsumed = M + 0 = M ≤ M` → passes.
4. Position state mutates: `buyerPos.credit` increases, `sellerPos.debt` increases, `totalUnits` increases. Zero tokens are transferred (line 455: `safeTransferFrom(..., 0 - 0)`).
5. Step 3–4 repeats without limit.

**Why existing checks fail** — The only guard against over-consumption is `require(newConsumed <= offer.maxAssets)`. There is no check requiring `buyerAssets > 0` when `units > 0`, and no strict `consumed < maxAssets` pre-check before the fill. [2](#0-1) 

## Impact Explanation
Each zero-cost take permanently inflates the buyer's (maker's) credit, the seller's (taker's) debt, and the market-wide `totalUnits` counter — all without any token payment. This directly violates the core protocol invariant that offers cannot be overfilled or replayed after full consumption. The corrupted accounting affects solvency calculations, health checks, and settlement for all participants in the market. [3](#0-2) 

## Likelihood Explanation
All three preconditions are reachable by any unprivileged user in normal protocol operation: (1) buy offers with `maxAssets > 0` are standard; (2) ticks below `MAX_TICK - 16` are a normal, accessible tick range; (3) full consumption of an offer happens in ordinary use. No special permissions, oracle manipulation, or admin action is required. The attack is repeatable without limit once the offer is fully consumed.

## Recommendation
Add a check that rejects a take when `units > 0` but the computed asset amount rounds to zero, preventing zero-increment consumed accounting:

```solidity
require(units == 0 || buyerAssets > 0, ZeroAssets());
```

Alternatively, enforce a strict pre-check `require(consumed[offer.maker][offer.group] < offer.maxAssets)` before the increment, so a fully-consumed offer cannot be taken at all regardless of rounding.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` lines 858–889 explicitly reproduces this scenario. It sets `consumed` to `maxAssets` via `setConsumed` (simulating a fully consumed offer), then calls `take(1, borrower, lenderOffer)` and asserts: `buyerAssets == 0`, `consumed` remains at `maxAssets`, token balances are unchanged, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase.     function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);
        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
