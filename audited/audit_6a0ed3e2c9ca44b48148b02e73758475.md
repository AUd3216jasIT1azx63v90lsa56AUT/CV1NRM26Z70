### Title
Division by Zero in `sellerAssetsToUnits` When `sellerPrice == 0` Causes Unexpected Revert in `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/TakeAmountsLib.sol)

### Summary

`TakeAmountsLib.sellerAssetsToUnits` computes `targetSellerAssets.mulDivUp(WAD, sellerPrice)` when `offer.buy == true`, but performs no guard against `sellerPrice == 0`. When `offerPrice == settlementFee` exactly, `sellerPrice` is zero and the division panics. Unlike the `offerPrice < settlementFee` case (which underflows in `Midnight.take()` and reverts there too), the `offerPrice == settlementFee` case is silently valid in `Midnight.take()` — it simply yields `sellerAssets = 0` — making the revert in `sellerAssetsToUnits` unexpected and unhandled.

### Finding Description

**Code path:**

In `TakeAmountsLib.sellerAssetsToUnits` ( [1](#0-0) ):

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
uint256 settlementFee =
    IMidnight(midnight).settlementFee(id, UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp));
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

When `offer.buy == true` and `offerPrice == settlementFee`, `sellerPrice = 0`. The subsequent `mulDivUp(targetSellerAssets, WAD, 0)` is a division by zero, triggering a Solidity 0.8 panic revert.

**Why `Midnight.take()` does NOT revert in this case:**

In `Midnight.take()` ( [2](#0-1) ):

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice = sellerPrice + _settlementFee;
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...;
```

When `sellerPrice == 0`, `units.mulDivDown(0, WAD) == 0` — no division, no revert. The take succeeds with `sellerAssets = 0`. The NatSpec comment at line 331 only warns about `offerPrice < settlementFee` (which underflows), not the equal case. [3](#0-2) 

**Where the revert propagates:**

`supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside the `try/catch` block that wraps `Midnight.take()`: [4](#0-3) 

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(   // <-- NOT in try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    ...
);
try IMidnight(MIDNIGHT).take(...) ...   // only this is guarded
```

The division-by-zero panic propagates out of `sellerAssetsToUnits`, bypasses the `try/catch`, and reverts the entire `supplyCollateralAndSellWithAssetsTarget` call.

**Contrast with `buyerAssetsToUnits`:**

`buyerAssetsToUnits` divides by `buyerPrice = sellerPrice + settlementFee`. When `sellerPrice == 0`, `buyerPrice == settlementFee > 0`, so no division by zero occurs there. The asymmetry is the root cause. [5](#0-4) 

**Attacker-controlled inputs:**

- A maker creates a buy offer (`offer.buy = true`) with `offer.tick` chosen such that `tickToPrice(offer.tick) == settlementFee(id, timeToMaturity)`.
- The settlement fee is a piecewise-linear function of time-to-maturity stored in CBP units (multiples of `1e12`). As time passes, the fee value changes continuously and can pass through a value that exactly equals a tick price. A maker can also deliberately select a tick whose price matches the current settlement fee.
- A taker then calls `supplyCollateralAndSellWithAssetsTarget` with `targetSellerAssets > 0` and this offer in the `takes` array.

**Existing protections reviewed and found insufficient:**

The `sellerAssetsToUnits` NatSpec says "Reverts if `offerPrice < settlementFee`" but provides no guard for `offerPrice == settlementFee`. [6](#0-5)  No `require(sellerPrice > 0)` or equivalent check exists in the function.

### Impact Explanation

`MidnightBundles.supplyCollateralAndSellWithAssetsTarget` reverts with a division-by-zero panic whenever any offer in the `takes` array has `tickToPrice(offer.tick) == settlementFee` and `offer.buy == true`. The taker's entire transaction fails — collateral supply, all takes, and the loan asset transfer — even though `Midnight.take()` itself would succeed for that offer (yielding `sellerAssets = 0`). This is a concrete, reproducible DoS on the bundler's sell-with-assets-target flow.

### Likelihood Explanation

The settlement fee is a piecewise-linear function of time-to-maturity. As time elapses within a market, the fee value changes and can coincide with a tick price. Tick prices are discrete but dense enough (especially at low ticks) that coincidence is plausible. A maker can also deliberately pick a tick whose price matches the current fee. The condition is repeatable: any taker who includes such an offer in `supplyCollateralAndSellWithAssetsTarget` triggers the revert. No privileged action is required beyond the normal market and offer creation flow.

### Recommendation

Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case, mirroring the intent of the existing `offerPrice < settlementFee` revert. For example:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
require(!offer.buy || sellerPrice > 0, /* e.g. SellerPriceIsZero() */);
```

Alternatively, return `type(uint256).max` when `sellerPrice == 0` (since no finite unit count can yield positive seller assets), so the `min(...)` in `supplyCollateralAndSellWithAssetsTarget` naturally caps to `takes[i].units` or `consumableUnits`, and the offer is skipped via the `try/catch`.

### Proof of Concept

```solidity
// Foundry unit test
function testSellerAssetsToUnitsZeroSellerPriceDivisionByZero() public {
    // 1. Set settlement fee to a known value matching a tick price.
    uint256 targetFee = TickLib.tickToPrice(someAccessibleTick); // e.g. tick = 4 * tickSpacing
    midnight.setMarketSettlementFee(id, /* breakpoint */ 1, targetFee);
    midnight.touchMarket(market);

    // Confirm sellerPrice == 0
    uint256 fee = midnight.settlementFee(id, market.maturity - block.timestamp);
    uint256 price = TickLib.tickToPrice(someAccessibleTick);
    assertEq(price, fee, "sellerPrice must be zero");

    // 2. Construct buy offer at that tick.
    offers[0].buy = true;
    offers[0].tick = someAccessibleTick;
    offers[0].maxUnits = type(uint256).max;

    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offers[0], units: type(uint256).max, ratifierData: hex""});

    // 3. Call supplyCollateralAndSellWithAssetsTarget — expect division-by-zero panic revert.
    vm.prank(borrower);
    vm.expectRevert(); // Panic: division by zero
    midnightBundles.supplyCollateralAndSellWithAssetsTarget(
        1e18,              // targetSellerAssets > 0
        type(uint256).max,
        borrower,
        borrower,
        new CollateralSupply[](0),
        takes,
        0,
        address(0)
    );

    // 4. Confirm Midnight.take() itself would NOT revert (sellerAssets = 0 is valid).
    collateralize(market, borrower, 1e18);
    (uint256 buyerAssets, uint256 sellerAssets) =
        midnight.take(offers[0], hex"", 1e18, borrower, borrower, address(0), hex"");
    assertEq(sellerAssets, 0, "take succeeds with sellerAssets=0");
    assertTrue(buyerAssets > 0, "buyer still pays fee");
}
```

**Expected assertions:** The `supplyCollateralAndSellWithAssetsTarget` call reverts with a division-by-zero panic, while the direct `Midnight.take()` call succeeds with `sellerAssets == 0`.
