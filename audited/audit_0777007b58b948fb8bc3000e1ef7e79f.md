Audit Report

## Title
Unchecked oracle price overflow in `mulDivDown` permanently freezes `withdrawCollateral`, `liquidate`, and `take` for all borrowers in a market - (File: src/Midnight.sol)

## Summary
`UtilsLib.mulDivDown` uses plain Solidity 0.8 checked arithmetic `(x * y) / d` with no `unchecked` block, causing an arithmetic panic revert whenever `collateral * price > type(uint256).max`. Because `isHealthy` is called unconditionally by `withdrawCollateral`, `liquidate`, and `take` for any borrower with debt, a single oracle returning an extreme price permanently freezes all three operations for every borrower in that market. Market creation is fully permissionless and imposes no bound on oracle return values.

## Finding Description

**Root cause — `mulDivDown` is plain checked arithmetic:**

`UtilsLib.mulDivDown` at `src/libraries/UtilsLib.sol:29-31` computes `(x * y) / d` with no `unchecked` block. Solidity 0.8 checked arithmetic reverts with an arithmetic panic on any `x * y > type(uint256).max`. [1](#0-0) 

This is not a latent assumption — `test/UtilsLibTest.sol:65-71` explicitly tests and expects this revert behavior. [2](#0-1) 

**Overflow site — `isHealthy`:**

`src/Midnight.sol:953-955` calls `_position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)`. Collateral is stored as `uint128` (max ≈ 3.4×10³⁸). For `collateral = type(uint128).max`, overflow occurs when `price > 2^128 ≈ 3.4×10³⁸`. For smaller collateral amounts the threshold is proportionally higher but still reachable. [3](#0-2) 

**`withdrawCollateral` calls `isHealthy` unconditionally for any borrower with debt:** [4](#0-3) 

**`liquidate` has the identical overflow site in its own price loop:** [5](#0-4) 

**`take` calls `isHealthy` for the seller post-callback:** [6](#0-5) 

**No oracle price cap is enforced at market creation.** `touchMarket` validates `lltv` tiers, `maxLif`, sorted collaterals, maturity, and collateral count, but places no bound on the oracle address or the value it may return. [7](#0-6) 

**The formal verification spec explicitly treats price boundedness as an assumption, not a proven invariant.** `NoMultiplicationOverflow.spec:46-50` introduces `boundedPrice` as a `require`-based assumption scoped only to calls where the product fits in `uint256`. The `noMultiplicationOverflow` rule and `noMultiplicationOverflowIsHealthy` rule both operate under this assumption. [8](#0-7) [9](#0-8) 

**Exploit flow:**
1. Attacker deploys an oracle with `price()` and `setPrice(uint256)`, initially returning a normal price.
2. Attacker calls `touchMarket` (permissionless, `public`) with this oracle — no price validation occurs.
3. Borrowers supply collateral and take debt in the market.
4. Attacker calls `oracle.setPrice(type(uint256).max)` (or any value exceeding `type(uint256).max / collateral`).
5. Every subsequent call to `withdrawCollateral`, `liquidate`, or `take` for any borrower with debt reverts with an arithmetic panic inside `mulDivDown`.
6. Borrowers cannot recover their collateral; liquidators cannot liquidate; the market is permanently frozen.

## Impact Explanation

All borrowers in the affected market are permanently frozen: `withdrawCollateral` reverts at the `isHealthy` call, `liquidate` reverts at the identical `mulDivDown` inside its own price loop, and `take` reverts at its post-callback `isHealthy` check. Collateral is permanently locked in the contract with no recovery path. This directly violates the core protocol invariant that collateral cannot be permanently locked outside health/liquidation rules — here the rules themselves become unreachable. The freeze is irreversible once triggered.

## Likelihood Explanation

Market creation is fully permissionless — `touchMarket` is `public` with no access control and no oracle validation. The attacker needs only to deploy a two-function oracle contract, create a market, and wait for borrowers. The price threshold for overflow is not astronomically high: for any borrower with non-trivial collateral, a price in the range of `1e38`–`1e72` suffices. The attack is repeatable across any number of markets and requires no special privileges.

## Recommendation

Replace the plain checked multiplication in `mulDivDown` and `mulDivUp` with an `unchecked` block combined with an explicit overflow guard using full-width arithmetic (e.g., via assembly `mulmod` or a 512-bit multiply), or cap oracle prices at market creation time in `touchMarket` to enforce `price * type(uint128).max <= type(uint256).max`. The `NoMultiplicationOverflow.spec` assumption `boundedPrice` should become an enforced on-chain invariant rather than a verification-time assumption.
