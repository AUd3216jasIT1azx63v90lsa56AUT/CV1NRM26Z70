Audit Report

## Title
Zero-increment `consumed` bypass allows unlimited takes on fully-consumed buy offers — (File: `src/Midnight.sol`)

## Summary
When `offer.buy == true` and `offer.maxAssets > 0`, the consumed accounting increments by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For any tick below `MAX_TICK`, `buyerPrice < WAD`, so `units = 1` yields `buyerAssets = 0` via integer truncation. The guard `require(newConsumed <= offer.maxAssets)` then passes even when `consumed == maxAssets`, because `maxAssets + 0 <= maxAssets`. All downstream position mutations still execute with the non-zero `units` value, producing real credit/debt changes with zero token transfer.

## Finding Description
**Root cause — `src/Midnight.sol` lines 363–373:**

`tickToPrice(MAX_TICK) == 1e18 == WAD` (confirmed by `TickLibTest.sol` line 19 and `TickToPrice.spec` line 39), and `tickToPrice(tick) < WAD` for all `tick < MAX_TICK` (confirmed by the monotonicity test at `TickLibTest.sol` lines 34–38). Therefore, for any normally operating tick, `buyerPrice < WAD`. [1](#0-0) 

With `units = 1` and `buyerPrice < WAD`: `mulDivDown(1, buyerPrice, WAD) = buyerPrice / WAD = 0`. The storage update becomes `consumed += 0`, so `newConsumed == consumed`. The `require(newConsumed <= offer.maxAssets)` passes unconditionally even when `consumed == maxAssets`. [2](#0-1) 

Downstream at lines 382–417, all mutations use `units` (non-zero): [3](#0-2) 

The token transfer at line 455 sends `buyerAssets - sellerAssets = 0` tokens, so no assets move. There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in the function. [4](#0-3) 

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` only catches cases where `buyerAssets > 0` pushes `newConsumed` over the cap. A zero increment never triggers it.
- The `reduceOnly` check (lines 392–395) only applies when `offer.reduceOnly` is set.
- No minimum-units or minimum-assets guard exists.

**Confirmed by the test suite:** `test/TakeTest.sol` lines 857–889 contain a test explicitly named `testBugBuyMaxAssetsBypass` that reproduces the exact scenario: a fully-consumed buy offer (`consumed == maxAssets == 1`) at `tick = MAX_TICK - 16` is taken with `units = 1`, resulting in `buyerAssets == 0`, no token transfer, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase. [5](#0-4) 

## Impact Explanation
A fully-consumed buy offer (`consumed == maxAssets`) can be taken an unlimited number of times. Each such take increases the maker's credit and the taker's debt by `units` with zero token payment, and inflates `totalUnits` without a corresponding increase in `claimableSettlementFee` or token reserves. This creates a permanent accounting mismatch: units outstanding are not backed by deposited assets. At settlement, the maker can claim tokens that do not exist in the contract, leading to insolvency of the market. This violates the core protocol invariant that offers cannot be replayed or reused after being fully consumed.

## Likelihood Explanation
Preconditions are minimal:
- `offer.buy == true`, `offer.maxAssets > 0` — standard buy offer configuration.
- `buyerPrice < WAD` — holds for every tick below `MAX_TICK`; this is the normal operating range.
- `units = 1` — trivially satisfies `1 * buyerPrice < WAD` whenever `buyerPrice < WAD`.

Any unprivileged taker can trigger this after any partial or full fill of a qualifying buy offer. No special permissions, oracle manipulation, or victim mistakes are required. The attack is repeatable without limit and requires only the ability to call `take`.

## Recommendation
Add a guard before or within the `maxAssets` branch that rejects a take when `units > 0` but the asset increment rounds to zero:

```solidity
// src/Midnight.sol, after line 364
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets()); // add this guard
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum tick or minimum units such that `units * buyerPrice >= WAD` always holds when `offer.maxAssets > 0`. A complementary invariant test asserting `totalUnits * minPrice <= totalAssets` would catch regressions.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` is a minimal, self-contained reproduction:

1. Create a buy offer with `maxAssets = 1`, `tick = MAX_TICK - 16` (so `offerPrice < WAD`).
2. Pre-set `consumed[maker][group] = maxAssets` via `setConsumed` (simulating a fully-consumed offer).
3. Call `take(units=1, borrower, lenderOffer)`.
4. Assert `buyerAssets == 0` and `sellerAssets == 0` (no tokens transferred).
5. Assert `consumed` is still `== maxAssets` (cap not advanced).
6. Assert `creditOf(lender) > lenderCreditBefore`, `debtOf(borrower) > borrowerDebtBefore`, `totalUnits > totalUnitsBefore` (position state mutated without payment). [6](#0-5)
