Audit Report

## Title
Persistent `isRootRatified` State Survives Operator Authorization Revocation - (File: src/ratifiers/SetterRatifier.sol)

## Summary
`SetterRatifier.setIsRootRatified` performs a one-time authorization check before writing permanent on-chain state. `SetterRatifier.isRatified`, called by `Midnight` on every `take`, reads only the stored boolean and never re-checks whether the address that wrote it remains authorized. Revoking an operator via `Midnight.setIsAuthorized` has no effect on previously written `isRootRatified` entries, leaving any root ratified by a now-revoked operator permanently valid for `take` calls against the maker's position.

## Finding Description
**Root cause**

`SetterRatifier.setIsRootRatified` (lines 24–28) performs a point-in-time authorization check and writes persistent state:

```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
```

`SetterRatifier.isRatified` (lines 30–36) reads only the stored boolean with no re-check of the original writer's current authorization:

```solidity
require(isRootRatified[offer.maker][root], NotRatified());
```

`Midnight.take` (lines 355–356) checks only that the *ratifier contract* is authorized (`isAuthorized[offer.maker][offer.ratifier]`), then calls `isRatified`. The operator who wrote the root is never re-examined.

`Midnight.setIsAuthorized` writes only to `isAuthorized[onBehalf][authorized]` and has no knowledge of, and makes no changes to, `SetterRatifier.isRootRatified`.

**Contrast with `EcrecoverRatifier`**

`EcrecoverRatifier.isRatified` (line 44) re-checks `IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer)` on every invocation, so revocation immediately blocks future takes. The test `testIsRatifiedRevokeAuthorizationInvalidates` (lines 185–205 of `test/EcrecoverRatifierTest.sol`) explicitly verifies this property. No equivalent test or mechanism exists in `SetterRatifierTest.sol`.

**Exploit flow**

1. Maker grants operator authorization: `midnight.setIsAuthorized(operator, true, maker)`.
2. Maker authorizes `SetterRatifier` as ratifier: `midnight.setIsAuthorized(setterRatifier, true, maker)`.
3. Operator calls `setterRatifier.setIsRootRatified(maker, maliciousRoot, true)`. Authorization check passes; `isRootRatified[maker][maliciousRoot] = true` is written.
4. Maker revokes operator: `midnight.setIsAuthorized(operator, false, maker)`. Only `isAuthorized[maker][operator]` changes.
5. `isRootRatified[maker][maliciousRoot]` remains `true` — untouched by revocation.
6. Any party calls `midnight.take(maliciousOffer, abi.encode(maliciousRoot, leafIndex, proof), ...)`. `SetterRatifier.isRatified` returns `CALLBACK_SUCCESS`; the take executes against the maker's position without their current consent.

The front-running scenario (step 3 racing step 4 in the mempool) is one path, but the vulnerability is broader: any operator who called `setIsRootRatified` at any prior time retains that root's validity indefinitely, with no race condition required.

**Why existing checks fail**

The authorization guard in `setIsRootRatified` is a gate on the write, not on the read. Once written, `isRatified` has no mechanism to detect that the writer's authorization was subsequently revoked. The maker cannot enumerate which roots the operator ratified without scanning all `SetIsRootRatified` events off-chain, and cannot selectively block a specific malicious root without knowing its exact value. Revoking the `SetterRatifier` contract's authorization entirely would block all takes — including legitimate ones — and is not a targeted remedy.

## Impact Explanation
After the maker's revocation is mined, any `take` supplying a valid Merkle proof against `maliciousRoot` passes `SetterRatifier.isRatified` and executes against the maker's position. This can drain credit, consume offer capacity, or create debt without the maker's current consent and contrary to their explicit revocation. The maker has no targeted on-chain remedy unless they independently discover the exact root value. Impact is direct, unauthorized movement of assets/value from the maker's position.

## Likelihood Explanation
**Preconditions**: The operator must have been authorized at some point — the normal, intended use case for `SetterRatifier`. No leaked keys or external dependencies are required. **Front-running path**: The operator observes the revocation transaction in the mempool and submits `setIsRootRatified` with higher gas; standard on Ethereum. **Non-front-running path**: Any operator who called `setIsRootRatified` with any root during their authorization window retains that root's validity indefinitely, with no race condition required. **Repeatability**: The operator can ratify multiple roots in a single block. Each malicious root can be used for takes at any future time.

## Recommendation
`SetterRatifier.isRatified` should record the address that wrote each root and re-check their current authorization on every call, mirroring `EcrecoverRatifier`'s pattern. Concretely:

1. Change `isRootRatified` mapping to store the writer address: `mapping(address maker => mapping(bytes32 root => address writer)) public rootWriter`.
2. In `setIsRootRatified`, store `rootWriter[maker][root] = msg.sender` (or `address(0)` to un-ratify).
3. In `isRatified`, after verifying the Merkle proof, re-check: `address writer = rootWriter[offer.maker][root]; require(writer != address(0) && (writer == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, writer)), Unauthorized())`.

This ensures revocation of an operator immediately invalidates all roots they wrote, consistent with the behavior of `EcrecoverRatifier`.

## Proof of Concept
Add the following test to `test/SetterRatifierTest.sol`:

```solidity
function testIsRatifiedRevokeAuthorizationDoesNotInvalidate() public {
    Offer memory offer = makeOffer(lender);
    bytes32 _root = HashLib.hashOffer(offer);

    // Authorize setterRatifier and borrower as operator
    vm.prank(lender);
    midnight.setIsAuthorized(address(setterRatifier), true, lender);
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, true, lender);

    // Operator writes root while authorized
    vm.prank(borrower);
    setterRatifier.setIsRootRatified(lender, _root, true);

    // Maker revokes operator
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, false, lender);

    // Root remains valid — isRatified still returns CALLBACK_SUCCESS
    vm.prank(address(midnight));
    bytes32 result = setterRatifier.isRatified(offer, abi.encode(_root, 0, new bytes32[](0)));
    assertEq(result, CALLBACK_SUCCESS); // passes, demonstrating the vulnerability
}
```

Running this test will confirm that revocation of the operator has no effect on the previously written root, and `isRatified` continues to return `CALLBACK_SUCCESS` for the malicious root.
