### Title
Zero Settlement Fee Collected for Non-Zero-Units Buy-Offer Take Due to Rounding — (File: src/Midnight.sol)

### Summary

When `offer.buy == true` and `units * offerPrice < WAD`, the expression `units.mulDivDown(buyerPrice, WAD)` evaluates to 0, making both `buyerAssets` and `sellerAssets` zero. The line `claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets` therefore adds 0, while `units` of credit/debt are still created in the position accounting. No settlement fee is collected despite a non-zero settlement fee being configured and non-zero units being transacted.

```solidity

uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...; // line 364
```

`mulDivDown(x, y, d)` is `(x * y) / d` (floor division). With `WAD = 1e18`, `PRICE_ROUNDING_STEP = 1e12`, and `CBP = 1e12`:

- Minimum non-zero tick price: `tickToPrice(4) = 1e12` (first multiple of `PRICE_ROUNDING_STEP` reachable with `DEFAULT_TICK_SPACING = 4`).
- Minimum non-zero settlement fee: `1 * CBP = 1e12`.

**Concrete exploit parameters:**
- `offer.buy = true`, `offer.tick = 4` → `offerPrice = 1e12`
- `_settlementFee = 1e12` → `sellerPrice = 0`, `buyerPrice = 1e12`
- `units = 1` → `buyerAssets = floor(1 * 1e12 / 1e18) = 0`, `sellerAssets = floor(1 * 0 / 1e18) = 0`

**Result at line 418:**
```solidity
claimableSettlementFee[offer.market.loanToken] += 0 - 0;  // += 0
```

Meanwhile, lines 382–414 still execute the full position update: `buyerCreditIncrease`, `sellerDebtIncrease`, and `_marketState.totalUnits` all change by `units = 1`. The protocol creates real credit/debt but collects zero fee.

**Consumed accounting bypass (compounding issue):** When `offer.maxAssets > 0` and `offer.buy == true`, line 368 does:
```solidity
newConsumed = consumed[offer.maker][offer.group] += buyerAssets; // += 0
require(newConsumed <= offer.maxAssets, ConsumedAssets());       // 0 <= maxAssets, always passes
```
The consumed counter never advances, so the same offer can be taken an unbounded number of times with `units` in `[1, 1e6 - 1]`, each time creating credit/debt with zero fee.

**Why existing checks fail:**
- No `require(units > 0)` guard.
- No `require(buyerAssets > 0)` guard.
- The Certora spec `settlementFeeSpreadBounds` (line 60) asserts `buyerAssets - sellerAssets >= (units * fee) / WAD()`. When `units * fee < WAD`, the RHS is 0, so the assertion is trivially satisfied and does not catch this case.
- The protocol comment at line 113–114 acknowledges "the settlement fee has no defined rounding direction, which could lead to fees manipulations on chains with very cheap gas," but does not prevent or bound the zero-fee case.

### Impact Explanation

The protocol collects zero settlement fee for every take where `units * offerPrice < WAD`, even though real credit/debt positions are created and the settlement fee rate is non-zero. On low-gas chains (L2s), an attacker can repeat this cheaply and accumulate large credit/debt balances while the protocol accrues no fee revenue. The `claimableSettlementFee` mapping is permanently under-credited relative to the actual volume transacted.

### Likelihood Explanation

**Preconditions:**
1. A buy offer exists (or is created by the attacker acting as maker) at a low tick where `offerPrice` is small (e.g., tick = 4, `offerPrice = 1e12`).
2. The market has a non-zero settlement fee (`_settlementFee >= 1e12`).
3. The taker chooses `units` in `[1, floor(WAD / offerPrice) - 1]`.

All three are fully attacker-controlled. The attacker can be both maker (creating the low-tick buy offer) and taker (using a second address). The attack is repeatable in a loop with no state that resets between calls. On L2s with sub-cent gas, the cost per iteration is negligible.

### Recommendation

Add a minimum-assets guard before the consumed and position updates:

```solidity
// After computing buyerAssets and sellerAssets:
if (units > 0) {
    require(offer.buy ? buyerAssets > 0 : sellerAssets > 0, ZeroAssets());
}
```

Alternatively, compute the settlement fee directly from `units` and `_settlementFee` (rounding up) rather than deriving it as `buyerAssets - sellerAssets`, so the fee is always at least `mulDivUp(units, _settlementFee, WAD)` when `units > 0` and `_settlementFee > 0`.
