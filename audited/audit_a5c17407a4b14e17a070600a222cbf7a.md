Audit Report

## Title
`SetterRatifier` Lacks Chain Binding, Enabling Cross-Fork Offer Replay - (File: src/ratifiers/SetterRatifier.sol)

## Summary
`SetterRatifier` stores ratification state as `mapping(address maker => mapping(bytes32 root => bool)) public isRootRatified` with no chain discriminator. After a hard fork that changes `block.chainid`, storage is copied verbatim to the forked chain. Because neither `SetterRatifier.isRatified` nor `HashLib.hashOffer` includes any chain-specific value, and because `Midnight.toId` uses the immutable `INITIAL_CHAIN_ID` (intentionally stable across forks), a taker can fill a maker's offers on the forked chain using ratification state the maker set only on the original chain, modifying the maker's position without post-fork consent.

## Finding Description

**Root cause â€” `SetterRatifier` has no chain binding:**

`setIsRootRatified` writes to a storage key of `(maker, root)` only: [1](#0-0) 

`isRatified` checks only that same key, with no `block.chainid` or `INITIAL_CHAIN_ID` guard: [2](#0-1) 

**`HashLib.hashOffer` is chain-agnostic:**
The offer hash covers market fields, maker, expiry, tick, ratifier, etc. but never `block.chainid` or any chain-specific value: [3](#0-2) 

**`Midnight.toId` uses `INITIAL_CHAIN_ID` (immutable), not `block.chainid`:**
The protocol explicitly and intentionally uses `INITIAL_CHAIN_ID` to keep market IDs stable across hard forks: [4](#0-3) [5](#0-4) 

After a fork, `block.chainid` changes but `INITIAL_CHAIN_ID` does not, so `toId` returns the **same** market ID on both chains, meaning `touchMarket` finds the already-existing market and proceeds directly to the ratifier check. This is confirmed by the explicit test `testToIdStableAcrossHardfork`: [6](#0-5) 

**Contrast with `EcrecoverRatifier`**, which explicitly binds to the current chain via `block.chainid` in the EIP-712 domain separator: [7](#0-6) [8](#0-7) 

Its NatSpec explicitly states that a `block.chainid` change invalidates previously signed offers. `SetterRatifier` has no equivalent protection.

**Full exploit path:**
1. Pre-fork: maker calls `setIsRootRatified(maker, root, true)` â†’ `isRootRatified[maker][root] = true`.
2. Hard fork: `block.chainid` changes; all storage is copied; `isRootRatified[maker][root]` remains `true` on the forked chain.
3. Post-fork: taker calls `midnight.take(offer, abi.encode(root, leafIndex, proof), units, taker, ...)` on the forked chain.
4. `touchMarket` resolves the same market ID (via unchanged `INITIAL_CHAIN_ID`).
5. `isAuthorized[offer.maker][offer.ratifier]` â€” also copied from original chain, still `true`.
6. `isRatified` verifies the Merkle proof (chain-agnostic) and checks `isRootRatified[maker][root]` â€” `true`.
7. Take executes: maker's credit/debt position is modified on the forked chain without any post-fork authorization act by the maker.

The `take` function in `Midnight.sol` confirms the full call chain: [9](#0-8) 

## Impact Explanation
A taker can fill any `SetterRatifier`-backed offer on the forked chain using ratification state the maker set only on the original chain. The maker's position (credit increased or debt incurred) is modified on the forked chain without post-fork consent. This constitutes unauthorized state change and unauthorized movement of value â€” both in-scope impact classes per RESEARCHER.md. The asymmetry with `EcrecoverRatifier` (which is explicitly protected) demonstrates this is an unintended gap, not a design choice.

## Likelihood Explanation
Preconditions: (1) a contentious hard fork that changes `block.chainid` while keeping contract addresses identical (ETH/ETC-style), and (2) at least one maker using `SetterRatifier` with an active, non-expired offer. Both are realistic â€” hard forks are historical precedent, and the codebase explicitly anticipates the hard fork scenario (NatSpec comment, dedicated test). The attack requires no special privilege, is executable by any taker, and is repeatable for every ratified root not explicitly revoked post-fork.

## Recommendation
Add a chain binding to `SetterRatifier` analogous to `EcrecoverRatifier`. Options:

1. **Include `block.chainid` in the storage key**: Change the mapping to `mapping(address maker => mapping(uint256 chainId => mapping(bytes32 root => bool)))` and key on `block.chainid` in both `setIsRootRatified` and `isRatified`.
2. **Include `block.chainid` in the ratification check**: In `isRatified`, require `block.chainid == SOME_CAPTURED_CHAIN_ID` or incorporate chain ID into the root derivation.
3. **Document as known/accepted risk**: If cross-fork replay is intentionally accepted for `SetterRatifier` (e.g., because makers can revoke roots post-fork), add explicit NatSpec warning analogous to `EcrecoverRatifier`'s comment â€” but this does not eliminate the risk.

Option 1 is the cleanest fix, as it makes pre-fork and post-fork ratification state independent.

## Proof of Concept
**Fork test plan (Foundry):**
```solidity
function testSetterRatifierCrossForkReplay() public {
    // 1. Setup: maker ratifies a root on chain ID 1
    vm.chainId(1);
    Midnight midnight = new Midnight(); // INITIAL_CHAIN_ID = 1
    SetterRatifier ratifier = new SetterRatifier(address(midnight));
    // ... setup market, offer, merkle tree with root R ...
    vm.prank(maker);
    ratifier.setIsRootRatified(maker, root, true);
    vm.prank(maker);
    midnight.setAuthorization(address(ratifier), true);

    // 2. Simulate hard fork: change block.chainid
    vm.chainId(2); // fork chain ID

    // 3. Post-fork: taker fills offer â€” should revert but does not
    vm.prank(taker);
    midnight.take(offer, abi.encode(root, leafIndex, proof), units, taker, address(0), address(0), "");
    // Assert: maker's position was modified without post-fork consent
    assertGt(midnight.debtOf(id, maker), 0);
}
```
The test passes (take succeeds) because `INITIAL_CHAIN_ID` remains 1, market ID is unchanged, `isRootRatified[maker][root]` is still `true`, and the Merkle proof is chain-agnostic.

    function testToIdStableAcrossHardfork(Market memory _market, Market memory otherMarket, uint64 newChainId) public {
        vm.assume(_market.collateralParams.length > 0);
        vm.assume(newChainId != block.chainid);
        _market = validMarket(_market);

        bytes32 idBefore = midnight.touchMarket(_market);
        uint256 capturedChainId = midnight.INITIAL_CHAIN_ID();

        vm.chainId(newChainId);

        assertEq(midnight.INITIAL_CHAIN_ID(), capturedChainId, "INITIAL_CHAIN_ID changed");
        assertEq(midnight.toId(_market), idBefore, "toId changed");
        Market memory roundTrip = midnight.toMarket(idBefore);
        assertEq(keccak256(abi.encode(roundTrip)), keccak256(abi.encode(_market)), "stored market lost");

        otherMarket = validMarket(otherMarket);
        bytes32 otherId = midnight.touchMarket(otherMarket);
        Market memory otherRoundTrip = midnight.toMarket(otherId);
        assertEq(keccak256(abi.encode(otherRoundTrip)), keccak256(abi.encode(otherMarket)), "stored market lost");
    }
