### Title
Buy-offer `maxAssets` cap bypassed via `mulDivDown` rounding to zero — (`src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and `offerPrice < WAD` (any tick below `MAX_TICK`), a taker can call `take()` with a `units` value small enough that `units.mulDivDown(buyerPrice, WAD) == 0`. Because `consumed[maker][group]` is incremented by `buyerAssets` (which is 0), the cap is never advanced, yet the maker's credit and the taker's debt both increase. This allows the offer to be filled indefinitely without ever consuming the cap. The codebase itself documents this as a confirmed bug in `testBugBuyMaxAssetsBypass()`.

### Finding Description

**Code path — `src/Midnight.sol` lines 358–373:**

```
offerPrice  = tickToPrice(offer.tick)          // < WAD for any tick < MAX_TICK
buyerPrice  = offerPrice                        // for buy offer: sellerPrice + settlementFee = offerPrice
buyerAssets = units.mulDivDown(buyerPrice, WAD) // = 0 when units * buyerPrice < WAD

if (offer.maxAssets > 0) {
    newConsumed = consumed[maker][group] += buyerAssets;  // += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets()); // always passes
}
``` [1](#0-0) 

**Root cause:** The consumed accounting for buy offers uses `buyerAssets` (a rounded-down asset amount) rather than `units`. Because `mulDivDown` truncates, any `units` satisfying `units * buyerPrice < WAD` produces `buyerAssets = 0`, so `consumed` is never incremented.

**Concrete arithmetic:** `tickToPrice` rounds prices to multiples of `PRICE_ROUNDING_STEP = 1e12`. [2](#0-1) 

At tick `MAX_TICK - 16`, `offerPrice < WAD`. For `units = 1`: `1 * (WAD - ε) / WAD = 0` in integer arithmetic. The condition `units * buyerPrice < WAD` is satisfied for `units ∈ [1, WAD/buyerPrice - 1]`, a range that can be millions of units wide at low ticks.

**Exploit flow:**
1. Maker creates a buy offer: `offer.buy = true`, `maxAssets = N`, `tick < MAX_TICK` (any sub-par tick).
2. Taker calls `take(units, ...)` where `units < WAD / buyerPrice`.
3. `buyerAssets = 0` → `consumed` unchanged → `require(newConsumed <= maxAssets)` passes.
4. `buyerPos.credit += buyerCreditIncrease` and `sellerPos.debt += sellerDebtIncrease` still execute with the full `units` value.
5. Repeat indefinitely: maker accumulates unbounded credit, offer cap is never consumed.

**Existing checks are insufficient:** The only guard is `require(newConsumed <= offer.maxAssets)`, which is trivially satisfied when the increment is 0. There is no `require(units == 0 || buyerAssets > 0)` or equivalent floor check. [3](#0-2) 

**Confirmed by existing test:** `testBugBuyMaxAssetsBypass()` in `test/TakeTest.sol` explicitly demonstrates the bypass: the offer is pre-consumed to `maxAssets`, then `take(1, borrower, lenderOffer)` succeeds, `consumed` stays at `maxAssets`, yet `creditOf(lender)` and `debtOf(borrower)` both increase. [4](#0-3) 

### Impact Explanation

The `maxAssets` cap on buy offers is completely unenforceable at any tick below `MAX_TICK`. A maker's credit can be inflated without bound regardless of the cap they set, violating the invariant that "offers cannot be replayed, overfilled, or reused." The maker receives credit without paying tokens (`buyerAssets = 0`), and the taker accumulates debt without receiving tokens (`sellerAssets = 0`). A colluding maker-taker pair can exploit this to give the maker unlimited credit at zero cost, or a griefing actor can force a maker to hold more credit exposure than their `maxAssets` limit intended.

### Likelihood Explanation

**Preconditions:**
- `offer.buy = true` (buy offer)
- `offer.maxAssets > 0`
- `offer.tick < MAX_TICK` (any sub-par price — the vast majority of practical ticks)
- `units < WAD / buyerPrice` (e.g., at tick `MAX_TICK - 16`, any `units = 1` suffices)

All preconditions are reachable by any unprivileged user. The attack is repeatable in a loop with no cooldown. The only cost is gas. The codebase's own test suite confirms the path is live.

### Recommendation

Add a floor check that rejects a non-zero `units` take when the resulting asset amount rounds to zero. In the `maxAssets` branch, require that the asset increment is positive whenever `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, track consumed in `units` for all offer types and convert the `maxAssets` cap to a units ceiling at offer creation time, eliminating the rounding surface entirely.

### Proof of Concept

```solidity
// Foundry unit test — mirrors testBugBuyMaxAssetsBypass() and extends it to show indefinite reuse.
function testMaxAssetsBypassIndefiniteReuse() public {
    // Setup: buy offer at any sub-par tick, small maxAssets cap.
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 50;
    lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD → buyerAssets rounds to 0 for units=1

    deal(address(loanToken), lender, 0);
    collateralize(market, borrower, 1000);

    uint256 consumedBefore = midnight.consumed(lender, lenderOffer.group);

    // Take 100 times with units=1; each call should be blocked after maxAssets is consumed,
    // but instead every call succeeds and consumed never advances.
    for (uint256 i = 0; i < 100; i++) {
        take(1, borrower, lenderOffer);
    }

    // ASSERTION: consumed must have advanced (invariant).
    // This assertion FAILS, proving the bug.
    assertGt(midnight.consumed(lender, lenderOffer.group), consumedBefore, "consumed never advanced");

    // ASSERTION: credit must not exceed what maxAssets permits.
    // This assertion also FAILS.
    assertLe(midnight.creditOf(id, lender), lenderOffer.maxAssets, "credit exceeds cap");
}

// Fuzz variant: assert consumed increases by at least 1 on every non-zero-unit take.
function testFuzzConsumedMonotonicOnNonZeroTake(uint256 units, uint256 tick) public {
    units = bound(units, 1, 1e6);
    tick = bound(tick, 2, MAX_TICK - 1); // sub-par ticks only

    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = type(uint128).max;
    lenderOffer.tick = tick;

    deal(address(loanToken), lender, type(uint256).max);
    collateralize(market, borrower, units);

    uint256 consumedBefore = midnight.consumed(lender, lenderOffer.group);
    take(units, borrower, lenderOffer);
    uint256 consumedAfter = midnight.consumed(lender, lenderOffer.group);

    // FAILS when units * tickToPrice(tick) < WAD.
    assertGt(consumedAfter, consumedBefore, "consumed did not increase on non-zero take");
}
```

**Expected assertions:** Both `assertGt(consumed, consumedBefore)` calls fail, confirming that `consumed` is not incremented when `buyerAssets` rounds to zero, and that the `maxAssets` cap is bypassed indefinitely.
