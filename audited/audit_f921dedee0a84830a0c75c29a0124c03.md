### Title
Transitive Authorization Delegation Allows Authorized Attacker to Grant Full Control to Third Party - (File: src/Midnight.sol)

### Summary
`setIsAuthorized` permits any address that has been authorized by a victim to call `setIsAuthorized(attacker2, true, victim)`, writing `isAuthorized[victim][attacker2] = true` without the victim's knowledge or consent. Because every privileged action in the protocol (withdraw, withdrawCollateral, repay, setConsumed, etc.) uses the same `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` guard, attacker2 immediately inherits unrestricted control over the victim's positions and can further chain the delegation indefinitely.

### Finding Description
The root cause is in `src/Midnight.sol` at line 732:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    ...
}
``` [1](#0-0) 

The guard only checks that `msg.sender` is itself authorized by `onBehalf`; it does not require `msg.sender == onBehalf`. Therefore:

1. Victim calls `setIsAuthorized(attacker, true, victim)` — legitimate, `isAuthorized[victim][attacker] = true`.
2. Attacker calls `setIsAuthorized(attacker2, true, victim)` — `onBehalf = victim`, `msg.sender = attacker`. The check evaluates `isAuthorized[victim][attacker]` which is `true`, so it passes. `isAuthorized[victim][attacker2]` is now `true`.
3. Attacker2 calls `withdrawCollateral(market, 0, amount, victim, attacker2)` — the guard at line 482 (`isAuthorized[victim][attacker2]`) passes, draining victim's collateral. [2](#0-1) 

The same pattern applies to `withdraw`, `repay`, `setConsumed`, and `setIsAuthorized` itself (enabling unbounded chaining). No other check exists to block this path. The Certora spec `onlyAuthorizedCanChangeIsAuthorized` at line 103 explicitly defines `authorizerIsAuthorized` as `authorizer == e.msg.sender || isAuthorized(authorizer, e.msg.sender)`, meaning the spec itself treats a delegate as authorized to mutate the mapping — the spec does not catch this as a violation. [3](#0-2) 

### Impact Explanation
Attacker2 gains `isAuthorized[victim][attacker2] == true`, giving it the same rights as the victim over all markets: withdraw credit, withdraw collateral, repay debt (draining attacker2's own funds to victim's account), cancel offers via `setConsumed`, and further authorize additional addresses. The victim's entire on-chain state is compromised without any action by the victim beyond the original, single authorization of attacker.

### Likelihood Explanation
The precondition — victim having authorized any address — is the normal operating state for users who delegate to operator contracts, bots, or periphery contracts (e.g., `EcrecoverAuthorizer` itself is authorized by users as shown in `SetIsAuthorizedWithSigTest.sol` line 56). Any such authorized contract or EOA that is itself compromised, malicious, or has a callback vulnerability can immediately escalate to full victim control. The attack is a single external call with no special tokens, no oracle manipulation, and no timing dependency, making it trivially repeatable. [4](#0-3) 

### Recommendation
Restrict `setIsAuthorized` so that only the account whose mapping is being modified can call it — i.e., require `onBehalf == msg.sender` unconditionally:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender, Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    ...
}
```

Delegation of the *ability to authorize* is a separate, higher-privilege action than delegation of operational rights and should require an explicit, separate mechanism (e.g., a signed EIP-712 message via `EcrecoverAuthorizer`) rather than being implicitly inherited by every authorized address.

pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {IMidnight, Market, CollateralParams} from "../src/interfaces/IMidnight.sol";

contract TransitiveAuthorizationTest is BaseTest {
    function testTransitiveAuthorizationExploit() public {
        // Setup: victim has collateral in a market
        address victim  = makeAddr("victim");
        address attacker  = makeAddr("attacker");
        address attacker2 = makeAddr("attacker2");

        Market memory market;
        market.loanToken = address(loanToken);
        market.maturity  = vm.getBlockTimestamp() + 100;
        market.collateralParams.push(CollateralParams({
            token: address(collateralToken1),
            lltv: 0.77e18,
            maxLif: maxLif(0.77e18, 0.25e18),
            oracle: address(oracle1)
        }));

        uint256 collateralAmount = 1000;
        deal(address(collateralToken1), victim, collateralAmount);
        vm.startPrank(victim);
        collateralToken1.approve(address(midnight), collateralAmount);
        midnight.supplyCollateral(market, 0, collateralAmount, victim);
        // Victim legitimately authorizes attacker (e.g., an operator contract)
        midnight.setIsAuthorized(attacker, true, victim);
        vm.stopPrank();

        // Attacker exploits: delegates victim's authorization to attacker2
        vm.prank(attacker);
        midnight.setIsAuthorized(attacker2, true, victim);

        // Assert transitive authorization was granted
        assertTrue(midnight.isAuthorized(victim, attacker2));

        // attacker2 drains victim's collateral without victim's consent
        vm.prank(attacker2);
        midnight.withdrawCollateral(market, 0, collateralAmount, victim, attacker2);

        assertEq(collateralToken1.balanceOf(attacker2), collateralAmount);
        assertEq(midnight.collateral(toId(market), victim, 0), 0);
    }
}
```

Expected assertions all pass: `isAuthorized[victim][attacker2] == true`, `attacker2` balance equals `collateralAmount`, victim collateral is zero.
