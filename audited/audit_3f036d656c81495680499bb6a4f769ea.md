### Title
Fee-on-Transfer loanToken Inflates `withdrawable` Beyond Actual Balance in `liquidate()` - (File: src/Midnight.sol)

### Summary

`liquidate()` credits `_marketState.withdrawable` with the full `repaidUnits` at line 675 before the inbound transfer at line 717, but `SafeTransferLib.safeTransferFrom` only verifies call success and a boolean return value — it does not measure the actual tokens received. When the market's `loanToken` is a fee-on-transfer token, the contract receives `repaidUnits * (1 - fee_rate)` while `withdrawable` is inflated by the full `repaidUnits`, permanently breaking the invariant `withdrawable <= loanToken.balanceOf(address(this))`.

### Finding Description

**Code path:**

`liquidate()` in `src/Midnight.sol`:

1. Lines 675–676: accounting is updated unconditionally before any inbound transfer: [1](#0-0) 

2. Line 717: the actual inbound transfer uses `SafeTransferLib.safeTransferFrom`, which only checks `success` and the ERC20 boolean return — it never compares received balance to `repaidUnits`: [2](#0-1) 

3. `SafeTransferLib.safeTransferFrom` implementation confirms it performs no balance-delta check: [3](#0-2) 

**Root cause:** The protocol credits `withdrawable` with the nominal transfer amount rather than the amount actually received. For a standard ERC20 this is equivalent, but for a fee-on-transfer token the two diverge by `repaidUnits * fee_rate` per liquidation call.

**Preconditions and exploit flow:**

1. *Market creation (permissionless):* Any unprivileged actor calls `touchMarket()` with a fee-on-transfer ERC20 as `loanToken`. No check in `touchMarket()` restricts token type; the design explicitly supports "arbitrary loan token": [4](#0-3) 

2. Lenders supply credit and borrowers take loans in this market (both paths also affected by the same pattern, but the question is scoped to `liquidate()`).

3. A borrower becomes unhealthy. An unprivileged liquidator calls `liquidate()` with `repaidUnits = R`.

4. At line 675, `_marketState.withdrawable += R` (full nominal amount).

5. At line 717, the contract receives only `R * (1 - fee_rate)` tokens.

6. Net effect per liquidation: `withdrawable` is overstated by `R * fee_rate`.

7. Repeated across N liquidations: `withdrawable` exceeds `loanToken.balanceOf(address(this))` by `N * R * fee_rate`.

**Why existing checks do not stop it:**

- `SafeTransferLib` checks only call success and the ERC20 `bool` return, not the received amount delta.
- `touchMarket()` imposes no restriction on `loanToken` type.
- `SECURITY.md` does not list fee-on-transfer tokens as out of scope.
- The same pattern exists identically in `repay()` (lines 508–520), confirming this is a systemic design assumption, not an isolated oversight: [5](#0-4) 

### Impact Explanation

Each `liquidate()` call in a fee-on-transfer loanToken market inflates `_marketState.withdrawable` beyond the contract's actual token balance. When lenders subsequently call `withdraw()`, the contract executes `safeTransfer(market.loanToken, receiver, units)` against an insufficient balance. The last lenders to withdraw will have their calls revert, permanently freezing their funds. The shortfall compounds with every liquidation and every `repay()` call, making insolvency progressive and irreversible within the market.

### Likelihood Explanation

Market creation is permissionless — any actor can deploy a market with a fee-on-transfer loanToken. Lenders who supply to such a market (possibly unaware of the token's fee behavior) are exposed. The liquidator triggering the bug need not be malicious; any liquidation of an unhealthy position in the market is sufficient. The bug is repeatable on every `liquidate()` and `repay()` call, so insolvency compounds deterministically.

### Recommendation

Replace the nominal-amount accounting pattern with a balance-delta measurement. Before and after each inbound `safeTransferFrom`, snapshot `IERC20(loanToken).balanceOf(address(this))` and use the actual delta to update `withdrawable` and reduce `debt`. Apply this fix consistently to both `liquidate()` (line 717) and `repay()` (line 520). Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as `loanToken` by adding a validation step in `touchMarket()` (e.g., a round-trip transfer check during market creation).

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../src/Midnight.sol";

// Fee-on-transfer token: deducts 1% on every transferFrom
contract FeeToken is ERC20 {
    function transferFrom(address from, address to, uint256 amount)
        public override returns (bool)
    {
        uint256 fee = amount / 100;
        super.transferFrom(from, to, amount - fee); // delivers amount*(1-0.01)
        return true;
    }
}

contract FeeOnTransferLiquidateTest is Test {
    Midnight midnight;
    FeeToken loanToken;
    // ... setup collateral token, oracle, market params

    function test_liquidateInflatesWithdrawable() public {
        // 1. Create market with FeeToken as loanToken (permissionless)
        // 2. Lender supplies credit; borrower takes loan
        // 3. Warp time / move oracle to make borrower unhealthy
        // 4. Liquidator calls liquidate() with repaidUnits = R

        uint256 withdrawableBefore = midnight.withdrawable(id);
        uint256 balanceBefore = loanToken.balanceOf(address(midnight));

        vm.prank(liquidator);
        midnight.liquidate(market, collateralIndex, 0, R, borrower,
                           false, liquidator, address(0), "");

        uint256 withdrawableAfter = midnight.withdrawable(id);
        uint256 balanceAfter = loanToken.balanceOf(address(midnight));

        // withdrawable increased by R (full nominal)
        assertEq(withdrawableAfter, withdrawableBefore + R);
        // but balance only increased by R * 0.99
        assertEq(balanceAfter, balanceBefore + R * 99 / 100);

        // KEY ASSERTION: withdrawable > actual balance — invariant broken
        assertGt(withdrawableAfter, balanceAfter);

        // Repeat N times to compound insolvency, then assert lender withdraw reverts
    }

    function invariant_withdrawableLeBalanceOf() public {
        // Stateful invariant: should never hold if fee-on-transfer token is used
        assertLe(midnight.withdrawable(id), loanToken.balanceOf(address(midnight)));
    }
}
```

**Expected assertions:** `withdrawable > loanToken.balanceOf(address(midnight))` after each liquidation; lender `withdraw()` reverts for the last lender due to insufficient balance.
