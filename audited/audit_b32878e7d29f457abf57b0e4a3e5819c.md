Looking at the Morpho Midnight codebase for division-before-multiplication precision loss analogous to the external report.

**Vulnerability class:** Pricing/rounding — division before multiplication causing unnecessary precision loss in fee computation.

**Candidate sites investigated:**

**1. `settlementFee()` → `take()` chain** (`src/Midnight.sol` lines 963–979, 360–364)

```solidity
// settlementFee() — division at end of weighted average
return (feeLower * (end - timeToMaturity) + feeUpper * (timeToMaturity - start)) / (end - start);

// take() — truncated fee then multiplied by units
uint256 _settlementFee = settlementFee(id, timeToMaturity);
uint256 buyerPrice = sellerPrice + _settlementFee;
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
```

Truncation bound: `feeLower = a * CBP`, `feeUpper = b * CBP` (`CBP = 1e12`). The numerator is always a multiple of `CBP`, so the truncation is at most `CBP − 1 = 1e12 − 1` WAD units of fee. For a 1e24-unit trade this yields at most `1e24 * 1e12 / 1e18 = 1e18` wei (1 token) of lost protocol fee — but this is not attacker-controllable and is not compounding.
