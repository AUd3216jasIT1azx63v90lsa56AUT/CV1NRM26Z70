### Title
Fee-on-Transfer Collateral Inflates `_position.collateral[i]`, Causing `badDebt` Under-Computation and `lossFactor` Under-Update in `liquidate()` - (File: src/libraries/SafeTransferLib.sol)

### Summary

`supplyCollateral` records `oldCollateral + assets` into `_position.collateral[collateralIndex]` before calling `SafeTransferLib.safeTransferFrom`, which does not verify the actual received amount. For a fee-on-transfer collateral token, the contract receives `assets * (1 - fee_rate)` but the position stores `assets`, inflating the accounting by `assets * fee_rate`. When `liquidate()` iterates `collateralBitmap` to compute `badDebt`, it reads the inflated `_position.collateral[i]`, causing `badDebt` to be under-computed (or zeroed by `zeroFloorSub`), so `lossFactor` is under-updated and lenders are under-slashed for the true loss.

### Finding Description

**Root cause — `supplyCollateral` (src/Midnight.sol:531–545):**

State is written before the transfer:
```solidity
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets); // line 533
// ...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545
``` [1](#0-0) 

`SafeTransferLib.safeTransferFrom` only checks that the call succeeded and returned `true`; it does not compare pre/post balances: [2](#0-1) 

For a fee-on-transfer token with fee rate `f`, the contract receives `assets * (1 - f)` but `_position.collateral[i]` stores `assets`. Inflation per call = `assets * f`.

**No token-type restriction in `touchMarket`:** The market creation validation checks only LLTV tiers, `maxLif`, sorted collateral addresses, and count limits — no restriction on fee-on-transfer tokens: [3](#0-2) 

**`badDebt` computation in `liquidate()` uses the inflated value (src/Midnight.sol:612–616):**
```solidity
uint256 _collateral = _position.collateral[i];   // inflated
badDebt = badDebt.zeroFloorSub(
    _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
);
``` [4](#0-3) 

Because `_collateral` is inflated, the subtracted term is larger than the true collateral value, so `badDebt` is smaller than the true uncovered debt. With a large enough fee or collateral amount, `zeroFloorSub` floors `badDebt` to 0.

**`lossFactor` update is gated on `badDebt > 0` (src/Midnight.sol:626–641):** [5](#0-4) 

If `badDebt` is zeroed by the inflation, `lossFactor` is not updated at all. If `badDebt` is merely reduced, `lossFactor` is under-updated proportionally.

**Exploit flow:**
1. Unprivileged market creator calls `touchMarket` with a fee-on-transfer token at `collateralParams[j]`.
2. Borrower calls `supplyCollateral(market, j, assets, borrower)` → `_position.collateral[j]` = `assets`; contract holds `assets*(1-f)`.
3. Borrower takes debt via `take`.
4. Oracle price drops to create true bad debt: `debt > collateral_actual * price / maxLif`.
5. Liquidator calls `liquidate(market, j, 0, 0, borrower, false, receiver, address(0), "")`.
6. `badDebt` is computed using `assets` (inflated) instead of `assets*(1-f)` (actual) → `badDebt_computed < badDebt_true`.
7. `lossFactor` is under-updated; lenders' credit is under-slashed.

**Existing checks that fail to stop it:**
- `SafeTransferLib` only checks call success, not received delta.
- `touchMarket` has no token-type whitelist.
- The Certora `supplyCollateralEffects` rule (line 217) asserts `collateral == collateralBefore + assets`, which is exactly the inflated invariant — the formal spec models the bug as correct behavior. [6](#0-5) 

### Impact Explanation

Lenders are under-slashed: `lossFactor` is set lower than the true loss ratio, so when lenders call `updatePosition`, their credit is reduced by less than the actual bad debt. The gap between `badDebt_true` and `badDebt_computed` is permanently unaccounted for. The protocol's solvency invariant — "bad debt must reduce lender credit exactly once and proportionally" — is violated. Additionally, the contract's actual collateral token balance is less than `sum(_position.collateral[i])` for that token, breaking the balance-coverage invariant.

### Likelihood Explanation

**Preconditions:**
- A market with a fee-on-transfer collateral token must exist (any unprivileged address can create one via `touchMarket`).
- The borrower must supply that collateral token (standard user action).
- The oracle price must drop enough to create true bad debt.

**Feasibility:** Several real ERC20 tokens charge transfer fees (e.g., PAXG, STA, tokens with configurable fees). The market creator and borrower can be the same address. The attack is repeatable on every `supplyCollateral` call with a fee-on-transfer token. The `live_context.json` explicitly lists fee-on-transfer tokens as a concern that "should be tested if not explicitly excluded," confirming no protocol-level exclusion exists. [7](#0-6) 

### Recommendation

In `supplyCollateral`, measure the actual received amount by comparing pre/post balances and record that instead of `assets`:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as collateral (e.g., add a token registry check in `touchMarket`). The state write must either move after the transfer or use the balance-delta pattern.
