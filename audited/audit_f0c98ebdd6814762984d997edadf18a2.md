Audit Report

## Title
Fee-on-Transfer Token Flash Loan Drains Protocol Reserves Without Balance Invariant Check - (File: src/Midnight.sol)

## Summary
The `flashLoan` function in `src/Midnight.sol` (lines 737–752) disburses `assets[i]` tokens and reclaims `assets[i]` tokens with no pre/post balance snapshot. When a fee-on-transfer token is used, the protocol receives only `assets[i]*(1-f)` on repayment, losing `assets[i]*f` per call. The only formal exclusion of this token class is a prover-level assumption in `certora/specs/Solvency.spec`, not an on-chain guard.

## Finding Description
**Exact code path:**

`flashLoan` at lines 737–752 of `src/Midnight.sol`:
- Outbound: `SafeTransferLib.safeTransfer(tokens[i], callback, assets[i])` (line 743) — protocol balance decreases by `assets[i]`.
- Inbound: `SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i])` (line 750) — protocol requests `assets[i]` but receives only `assets[i]*(1-f)` due to the transfer fee.

**Root cause:** A grep across `src/Midnight.sol` confirms zero occurrences of `balanceOf`, `balanceBefore`, or `balanceAfter`. There is no `require(balanceAfter >= balanceBefore)` guard. The function also has no token allowlist, no role gate, and no market-membership check on supplied token addresses.

**Why the Certora proof does not protect against this:** The prover-level assumption at line 31 of `certora/specs/Solvency.spec` explicitly states: *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits."* The `CVL_transferFrom` ghost function models lossless transfers (exact value deducted from `src`, exact value added to `dest`), making the `tokenBalanceCorrect` invariant unsound for fee-on-transfer tokens in production.

**Why existing checks fail:** The only post-inbound check is that `safeTransferFrom` does not revert. It succeeds as long as the callback holds `assets[i]` tokens (satisfiable by pre-funding with `assets[i]*f`). No balance invariant is enforced on-chain.

## Impact Explanation
The protocol's token balance for any fee-on-transfer loan token decreases by `assets*f` per flash loan call. Repeated calls drain the protocol's reserves for that token, directly violating the core solvency invariant that contract balances must cover collateral, withdrawable, and claimable settlement fee amounts. Lenders and borrowers whose positions are denominated in that token face insolvency as backing reserves are eroded. This constitutes direct, concrete loss of protocol-held assets. Neither `SECURITY.md` nor `RESEARCHER.md` excludes fee-on-transfer tokens or griefing attacks from scope.

## Likelihood Explanation
**Preconditions:**
- A fee-on-transfer token must be used as a market's loan token (USDT has a dormant fee-on-transfer switch; many DeFi-native tokens carry transfer fees by design).
- The attacker must pre-fund their callback with `assets*f` tokens per iteration.

**Feasibility:** `flashLoan` is callable by any unprivileged address with no token allowlist, no role gate, and no market-membership check on the supplied token addresses. This is a 1:1 griefing attack — the attacker spends `assets*f` tokens per iteration and the protocol loses the same amount. It is not directly profitable, but it is a credible griefing/DoS vector against any market whose loan token carries a transfer fee, and the cost to the attacker is bounded while the damage to protocol solvency is cumulative and permanent.

## Recommendation
Add a pre/post balance snapshot around the flash loan disbursement and repayment:

```solidity
uint256[] memory balancesBefore = new uint256[](tokens.length);
for (uint256 i = 0; i < tokens.length; i++) {
    balancesBefore[i] = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
}
// ... callback ...
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(
        IERC20(tokens[i]).balanceOf(address(this)) >= balancesBefore[i],
        InsufficientRepayment()
    );
}
```

Alternatively, maintain a token allowlist restricting flash loans to tokens known to be non-fee-on-transfer.

## Proof of Concept
1. Deploy a fee-on-transfer ERC20 token with fee rate `f` (e.g., 1%).
2. Create a market using this token as the loan token via `touchMarket`.
3. Deploy a malicious callback contract that pre-funds itself with `assets*f` tokens and implements `onFlashLoan` to do nothing (just hold the received tokens and approve repayment).
4. Call `flashLoan([token], [assets], callback, "")` repeatedly.
5. After each call, observe that `token.balanceOf(address(midnight))` decreases by `assets*f`.
6. After `1/f` calls, the protocol's entire reserve for that token is drained.

A Foundry fork test can confirm this by forking mainnet, deploying a mock fee-on-transfer token, and asserting the protocol balance after each `flashLoan` call. [1](#0-0) [2](#0-1) [3](#0-2)
