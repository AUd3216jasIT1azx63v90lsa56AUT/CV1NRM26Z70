### Title
Factory enforcement bypassed via direct `initialize()` — attacker-controlled moderator drains staker funds - (`File: src/ConfidencePool.sol`)

### Summary
`ConfidencePool.initialize()` is `external` with only the `initializer` re-entrancy guard and no factory-only access control. The two critical enforcement checks present in `ConfidencePoolFactory.createPool()` — the stake-token allowlist (`allowedStakeToken`) and the agreement-owner authorization (`IAgreement(agreement).owner() != msg.sender`) — are absent from `initialize()`. Because `poolImplementation` is a public variable and OpenZeppelin `Clones` is a permissionless library, any attacker can deploy a clone of the implementation and call `initialize()` directly, supplying themselves as both `outcomeModerator_` and `owner_` with a malicious `recoveryAddress_`. The factory's enforcement is the only gate; the pool contract itself has none.

### Finding Description
`ConfidencePoolFactory.createPool()` enforces two invariants before deploying a pool:

```solidity
// ConfidencePoolFactory.sol line 77
if (!allowedStakeToken[stakeToken]) revert StakeTokenNotAllowed();
// line 82
if (IAgreement(agreement).owner() != msg.sender) revert UnauthorizedCreator();
```

Neither check exists in `ConfidencePool.initialize()`. The function signature is:

```solidity
// ConfidencePool.sol line 179
function initialize(
    address agreement_,
    address stakeToken_,
    address safeHarborRegistry_,
    address outcomeModerator_,   // ← attacker-supplied
    uint256 expiry_,
    uint256 minStake_,
    address recoveryAddress_,    // ← attacker-supplied
    address owner_,              // ← attacker-supplied
    address[] calldata accounts
) external initializer {
```

The `initializer` modifier (OpenZeppelin `Initializable`) only prevents re-initialization of the same proxy; it does not restrict who may call the function. A fresh clone's storage is zeroed, so `_initialized == 0` and the guard passes for any caller.

`poolImplementation` is a public state variable on the factory, readable by anyone. Using `Clones.clone(poolImplementation)` or `Clones.cloneDeterministic(poolImplementation, anySalt)` — both permissionless library calls — an attacker deploys a legitimate clone of the implementation and calls `initialize()` with:
- `outcomeModerator_` = attacker address (bypasses `defaultOutcomeModerator`)
- `owner_` = attacker address (bypasses the agreement-owner check)
- `recoveryAddress_` = attacker address (CORRUPTED sweep destination)
- `stakeToken_` = any token, including fee-on-transfer (bypasses `allowedStakeToken`)

The rogue pool passes `safeHarborRegistry.isAgreementValid(agreement_)` and `IAgreement(agreement).isContractInScope(account)` for any valid agreement and scope, making it indistinguishable from a factory-created pool to an on-chain observer who does not cross-reference `_poolsByAgreement`.

### Impact Explanation
Once stakers deposit into the rogue pool, the attacker — holding both the `onlyOwner` and `onlyModerator` roles — can:

1. **Steal all funds via bad-faith CORRUPTED:** when the registry reaches `CORRUPTED` state for the target agreement, the attacker calls `flagOutcome(CORRUPTED, false, address(0))` (moderator role) and then `claimCorrupted()`, sweeping `stakeToken.balanceOf(address(this))` to their controlled `recoveryAddress`. All staker principal and bonus is lost.

2. **Steal via auto-CORRUPTED backstop:** if the pool expires while the registry is `CORRUPTED` and `riskWindowStart != 0`, the attacker calls `claimExpired()` after `expiry + MODERATOR_CORRUPTED_GRACE` to auto-resolve as bad-faith CORRUPTED, then `claimCorrupted()` — same outcome, no moderator action needed.

3. **Introduce fee-on-transfer token grief:** with a non-allowlisted fee-on-transfer token, every `stake()` and `contributeBonus()` call receives fewer tokens than `amount`, but the balance-diff check only reverts on `received == 0` or `received < minStake`; a small fee passes silently, causing every `claimSurvived`/`claimExpired` payout to under-deliver relative to `eligibleStake[msg.sender]`, eventually reverting for the last claimers when the balance is exhausted.

### Likelihood Explanation
The attack requires stakers to interact with the rogue pool rather than a factory-created one. However:
- The rogue pool is on-chain, passes all registry validity checks, and holds a legitimate agreement and scope.
- Any off-chain indexer or UI that discovers pools by scanning `initialize` events (rather than the factory's `PoolCreated` event or `_poolsByAgreement`) will surface the rogue pool as legitimate.
- The attacker can create the rogue pool for a high-value agreement and wait; no time pressure exists until stakers deposit.
- The factory's `_poolsByAgreement` mapping is internal and only exposed via `getPoolsByAgreement` — there is no on-chain mechanism that prevents a rogue pool from being presented as legitimate.

### Recommendation
Add a factory-only access control check to `ConfidencePool.initialize()`. The simplest approach is to store the factory address in the implementation constructor (immutable) and require `msg.sender == factory` in `initialize()`. Alternatively, replicate the `allowedStakeToken` and agreement-owner checks inside `initialize()` itself so the pool is self-enforcing regardless of how it is deployed.

### Proof of Concept
```solidity
// Attacker script (no privileged access required)
address implAddr = factory.poolImplementation();          // public var
address roguePool = Clones.clone(implAddr);               // permissionless

address[] memory scope = new address[](1);
scope[0] = validScopeAccount;                             // passes isContractInScope

IConfidencePool(roguePool).initialize(

    allowedToken,            // any token; allowedStakeToken NOT checked here

    attacker,                // owner_ — bypasses agreement-owner check

// ... time passes, registry reaches CORRUPTED for validAgreement ...

IConfidencePool(roguePool).claimCorrupted();
