Audit Report

## Title
Authorized-Party Nonce Burn via Delegated Signature Invalidates Pending Off-Chain Authorizations — (`src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts a valid signature from any party already authorized by `authorization.authorizer` in Midnight, not only from the authorizer themselves. Because the nonce is consumed on every successful call, an already-authorized party can sign and submit a self-referential `Authorization` struct using the victim's current nonce value, permanently burning any pending off-chain authorization the victim has already signed with that same nonce. The attack is repeatable at gas cost only, making it possible to indefinitely block the victim from delegating through `EcrecoverAuthorizer`.

## Finding Description
**Code path** — `src/periphery/EcrecoverAuthorizer.sol` lines 24–48:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // nonce consumed here

    bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
    bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
    bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
    address signer = ecrecover(digest, signature.v, signature.r, signature.s);
    require(signer != address(0), InvalidSignature());
    require(
        signer == authorization.authorizer
            || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer), // delegation branch
        Unauthorized()
    );
    ...
    IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
}
```

**Root cause** — The `Unauthorized` guard at line 33–36 has two branches: (1) the authorizer signs themselves, or (2) any party already authorized by the authorizer in Midnight signs. The second branch allows an authorized party (`ATTACKER`) to produce a valid EIP-712 signature over an `Authorization` struct where `authorizer = A`, consuming `nonce[A]` in the process. There is no guard preventing an authorized party from targeting `authorization.authorizer = A` with a nonce they did not generate.

**Exploit flow:**

| Step | Action |
|------|--------|
| 0 | A calls `midnight.setIsAuthorized(ecrecoverAuthorizer, true, A)` (standard setup) and `midnight.setIsAuthorized(ATTACKER, true, A)` (e.g., authorizing a keeper or market maker). |
| 1 | A signs off-chain: `Authorization{authorizer=A, authorized=LEGITIMATE_OPERATOR, isAuthorized=true, nonce=N}` and hands the signature to `LEGITIMATE_OPERATOR`. |
| 2 | ATTACKER signs: `Authorization{authorizer=A, authorized=ATTACKER, isAuthorized=true, nonce=N}` with ATTACKER's own private key. |
| 3 | ATTACKER calls `ecrecoverAuthorizer.setIsAuthorized(authAttacker, sigAttacker)`. |
| 4 | Line 26: `nonce[A]` advances N → N+1. Line 34: `isAuthorized(A, ATTACKER) == true` → passes. Call succeeds. |
| 5 | `LEGITIMATE_OPERATOR` submits A's signed authorization → reverts `InvalidNonce` because `nonce[A]` is now N+1. |

**Why existing checks fail** — The `Unauthorized` check is satisfied by the delegation branch; it does not distinguish between "the authorizer is signing their own nonce" and "a delegate is consuming the authorizer's nonce on a struct the authorizer never signed." The nonce increment at line 26 occurs before the signature check, and on revert the increment is rolled back — but on success via the delegation branch, it is permanently consumed.

Note: `Midnight.sol` lines 101–110 do warn that "authorized accounts can authorize other accounts on behalf of the user," but this does not address the nonce-burning attack vector specific to `EcrecoverAuthorizer`.

## Impact Explanation
A's off-chain signed authorization with nonce=N is permanently invalidated. `LEGITIMATE_OPERATOR` cannot submit it; A must re-sign with nonce=N+1 and redistribute the new signature. ATTACKER can repeat this indefinitely (each time signing a no-op self-authorization `isAuthorized=true` for themselves), making it impossible for A to successfully delegate to any third party through `EcrecoverAuthorizer` as long as ATTACKER remains authorized. A's only escape is to revoke ATTACKER's authorization via a direct on-chain call to `midnight.setIsAuthorized`, but ATTACKER can front-run that revocation with another nonce burn first. This constitutes a permanent, repeatable griefing/freeze of the off-chain authorization mechanism for any affected user.

## Likelihood Explanation
**Preconditions:**
1. A has authorized `ecrecoverAuthorizer` in Midnight — required for the contract to act on A's behalf, standard setup for any `EcrecoverAuthorizer` user.
2. A has authorized ATTACKER in Midnight at any prior point (e.g., a market maker, keeper, or liquidation bot).
3. A has a pending off-chain signed authorization that ATTACKER knows about (e.g., via mempool observation or social engineering).

All three are realistic for any user who has ever granted authorization to a counterparty. The attack requires no privileged access beyond what A voluntarily granted. It is repeatable with zero cost beyond gas, and the front-running window for the revocation escape is exploitable on any chain with a public mempool.

## Recommendation
Restrict the signer check so that only `authorization.authorizer` themselves can sign an `Authorization` struct that consumes `nonce[authorization.authorizer]`. Remove the delegation branch from `EcrecoverAuthorizer.setIsAuthorized`:

```solidity
require(signer == authorization.authorizer, Unauthorized());
```

If delegation is intentionally desired, scope it so that a delegate can only sign structs where `authorization.authorizer == signer` (i.e., the delegate authorizes on their own behalf), not on behalf of a third party whose nonce they do not control. Alternatively, key the nonce on the signer rather than on `authorization.authorizer` when the signer is a delegate.
