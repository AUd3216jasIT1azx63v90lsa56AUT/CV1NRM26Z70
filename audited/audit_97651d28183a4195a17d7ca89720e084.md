### Title
Fee-on-Transfer Loan Token Causes `claimableSettlementFee` Over-Crediting in `take()` - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee` is incremented by `buyerAssets - sellerAssets` before the actual `safeTransferFrom` to the contract. `SafeTransferLib.safeTransferFrom` does not measure the actual balance delta received. When the loan token is a fee-on-transfer token, the contract receives fewer tokens than the credited amount, creating a persistent accounting surplus in `claimableSettlementFee` that exceeds the contract's real token balance attributable to settlement fees.

### Finding Description
**Code path:**

`src/Midnight.sol` line 418 credits the settlement fee accounting: [1](#0-0) 

Then line 455 performs the actual transfer to the contract: [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only checks that the call succeeded and returned `true`; it does not compare `balanceOf(address(this))` before and after: [3](#0-2) 

**Root cause:** The accounting delta (`buyerAssets - sellerAssets`) is written to storage unconditionally before the transfer, with no balance-before/after reconciliation.

**Attacker inputs and exploit flow:**
1. Market creation is permissionless and accepts an arbitrary `loanToken` (confirmed by `live_context.json` and `touchMarket` which imposes no token restrictions): [4](#0-3) 
2. An attacker (market creator or any taker) uses a market whose `loanToken` charges a transfer fee `f` on every `transferFrom`.
3. Taker calls `take()`. At line 418, `claimableSettlementFee[loanToken] += (buyerAssets - sellerAssets)`.
4. At line 455, `safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets)` executes. The token deducts fee `f`, so the contract receives only `(buyerAssets - sellerAssets) - f`.
5. `claimableSettlementFee` is now over-credited by `f` relative to actual tokens held.
6. Repeated `take()` calls accumulate the discrepancy.
7. When the `feeClaimer` calls `claimSettlementFee` (a routine protocol operation, not an attack), it transfers the full credited amount, draining tokens that belong to lenders' withdrawable balances or repaid loan pools.

**Why existing checks fail:** There are no checks on the token type, no balance snapshots, and no post-transfer reconciliation anywhere in `take()`. The Certora spec rule `takeIncreasesClaimableSettlementFee` models `safeTransferFrom` as an exact-amount transfer (see `cvlSafeTransferFrom` in `Role.spec`), so the formal verification does not cover fee-on-transfer behavior: [5](#0-4) 

### Impact Explanation
`claimableSettlementFee[loanToken]` exceeds the actual settlement fee tokens held by the contract by the cumulative transfer fees paid. When `claimSettlementFee` is subsequently called (a normal protocol operation), it transfers out more tokens than were received as settlement fees, directly draining funds that back lenders' credit redemptions or borrowers' repayments. This is a direct loss of user funds.

### Likelihood Explanation
**Preconditions:** A market must exist with a fee-on-transfer loan token. Market creation is permissionless and accepts any `loanToken` with no whitelist or token behavior checks. The `live_context.json` explicitly lists "token charges fee" as a considered external behavior. Any taker can trigger `take()` on such a market. The condition is repeatable on every `take()` call, accumulating the discrepancy linearly.

### Recommendation
Measure the actual balance received by comparing `balanceOf(address(this))` before and after the `safeTransferFrom` call, and use the measured delta (not the nominal `buyerAssets - sellerAssets`) to update `claimableSettlementFee`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens (e.g., via a token registry or a balance-check guard in `touchMarket`).

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
