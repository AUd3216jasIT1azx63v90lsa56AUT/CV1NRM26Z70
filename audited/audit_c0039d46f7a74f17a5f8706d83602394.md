Audit Report

## Title
Zero-asset rounding enables unbounded free credit/debt creation via `maxAssets` offer reuse â€” (`src/Midnight.sol`, `src/libraries/UtilsLib.sol`)

## Summary
When a buy offer has `maxAssets > 0`, the `consumed` mapping is incremented by `buyerAssets`, which is computed via `mulDivDown(units, buyerPrice, WAD)`. For any tick below `MAX_TICK`, `offerPrice < WAD`, so calling `take` with `units = 1` always yields `buyerAssets = 0`. The `consumed` value never advances, the offer is never exhausted, yet position credit and debt are mutated using `units` directly â€” with zero token transfer on every call.

## Finding Description

**Root cause â€” `mulDivDown` truncates to zero for sub-WAD prices:** [1](#0-0) 

`(x * y) / d` returns 0 whenever `x * y < d`. Since `WAD = 1e18` and `tickToPrice` returns values in `[0, WAD]`, any tick below `MAX_TICK` yields `offerPrice < WAD`. With `units = 1`: `mulDivDown(1, buyerPrice, WAD) = buyerPrice / WAD = 0`.

**Confirmed price range:** [2](#0-1) 

`tickToPrice(0) = 0`, `tickToPrice(2) = 1e12`, `tickToPrice(MAX_TICK) = 1e18`. Every valid tick below `MAX_TICK = 5820` produces `offerPrice < WAD`, making `units = 1` sufficient to trigger the zero-asset path.

**Vulnerable code path â€” `consumed` incremented by zero:** [3](#0-2) 

For a buy offer, `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. Line 368 adds 0 to `consumed[offer.maker][offer.group]`, so `newConsumed` is unchanged. The `require(newConsumed <= offer.maxAssets)` check passes trivially â€” it enforces an upper bound but does not require that `consumed` actually increased.

**Position state mutates using `units` directly, not `buyerAssets`:** [4](#0-3) 

`buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)` and `sellerDebtIncrease = units - sellerCreditDecrease` both use `units` (= 1), not `buyerAssets` (= 0). If the buyer has no existing debt and the seller has no existing credit, each call grants the buyer 1 unit of credit and burdens the seller with 1 unit of debt.

**Token transfers are zero:** [5](#0-4) 

Both `buyerAssets - sellerAssets = 0` and `sellerAssets = 0`, so no tokens move.

**No guard exists:** There is no `require(buyerAssets > 0)`, no `require(units == 0 || buyerAssets > 0)`, and no check that `consumed` actually increased before proceeding.

## Impact Explanation
Each call with `units = 1` and `buyerAssets = 0` grants the buyer (maker) 1 unit of credit and assigns the seller (taker) 1 unit of debt with zero token backing. Repeated indefinitely, this creates unbounded credit and debt without any loan-token deposit, directly violating the solvency invariant: the contract's loan-token balance does not cover the credit it has issued. The maker can accumulate arbitrary credit redeemable at maturity against a contract balance that was never funded.

## Likelihood Explanation
**Preconditions:**
- A buy offer with `maxAssets > 0` must exist â€” a standard configuration.
- The offer's tick must be below `MAX_TICK` â€” true for every practical offer since `MAX_TICK` corresponds to a price of exactly 1 (par), which is the least favorable buy price.
- The settlement fee must not exceed `offerPrice` (otherwise `offerPrice - _settlementFee` underflows and reverts). This is satisfied whenever the settlement fee is 0 or the tick is high enough that `offerPrice >= settlementFee`.

**Feasibility:** Any unprivileged taker can call `take(offer, ..., 1, ...)` directly. No oracle manipulation, special permissions, or victim cooperation is required. The required `units` value is trivially computed as any value satisfying `units * buyerPrice < WAD`.

**Repeatability:** Unlimited â€” `consumed` never advances, so the offer never closes.

## Recommendation
In the `maxAssets` branch, require that the asset amount actually increased before accepting the call:

```solidity
uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
require(units == 0 || assetsDelta > 0, ZeroAssetTake());
newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

Alternatively, enforce a minimum `units` value such that `units * minPrice >= WAD`, or add a global `require(units > 0)` guard combined with `require(buyerAssets > 0 || sellerAssets > 0)`.

## Proof of Concept
**Minimal manual steps:**

1. Deploy Midnight and create a market with any valid parameters and settlement fee = 0.
2. Maker creates a buy offer at tick 2 (price = `1e12`) with `maxAssets = 1e18` and `maxUnits = 0`.
3. Taker calls `take(offer, ..., units=1, taker, ...)` â€” no token approval needed.
4. Observe: `creditOf(id, maker) = 1`, `debtOf(id, taker) = 1`, `consumed[maker][group] = 0`, zero tokens transferred.
5. Repeat step 3 one million times: `creditOf(id, maker) = 1_000_000`, `debtOf(id, taker) = 1_000_000`, `consumed` still 0, contract loan-token balance still 0.
