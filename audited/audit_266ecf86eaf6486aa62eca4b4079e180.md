Audit Report

## Title
Buy-offer `mulDivDown` rounding allows zero-asset takes that mint credit units for free — (`src/Midnight.sol`)

## Summary
In `Midnight.take`, when `offer.buy == true` and `offer.maxAssets > 0`, `buyerAssets` is computed with `mulDivDown`, which truncates to zero for any `units` value where `units * buyerPrice < WAD`. Because the consumed-cap accounting increments by `buyerAssets` (zero), the cap check always passes regardless of how many times the offer has been taken. An attacker controlling two addresses can repeatedly call `take` with `units = 1`, minting one credit unit per call at zero asset cost, indefinitely diluting the redemption pool of legitimate lenders.

## Finding Description

**Root cause**: `mulDivDown` integer truncation at line 363 produces `buyerAssets = 0` whenever `units * buyerPrice < WAD`. The consumed-cap accounting at line 368 adds this zero to `consumed[maker][group]`, so the cap check at line 369 (`newConsumed <= offer.maxAssets`) never advances toward the limit. Meanwhile, credit and debt accounting at lines 410 and 414 operate on `units` directly, not on `buyerAssets`, so 1 credit unit is minted for the buyer and 1 debt unit is created for the seller per call, with zero tokens transferred.

**Exact code path**:
```
line 363: buyerAssets = units.mulDivDown(buyerPrice, WAD)  // = 0 when units*buyerPrice < WAD
line 368: consumed[maker][group] += buyerAssets            // += 0, cap never advances
line 369: require(newConsumed <= offer.maxAssets)          // trivially passes
line 382: buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)  // = units = 1
line 410: buyerPos.credit += toUint128(buyerCreditIncrease)         // +1 credit, free
line 414: sellerPos.debt   += toUint128(sellerDebtIncrease)         // +1 debt, no collateral
line 455: safeTransferFrom(..., buyerAssets - sellerAssets)         // transfers 0
line 456: safeTransferFrom(..., sellerAssets)                       // transfers 0
```

**Preconditions**:
- `offer.buy = true`, `offer.maxAssets > 0`, any tick below `MAX_TICK` (giving `buyerPrice < WAD`)
- `units = 1` (always satisfies `1 * buyerPrice / WAD = 0` for any sub-WAD price)
- Attacker controls two distinct addresses (`makerAddr` ≠ `takerAddr`) to bypass the `SelfTake` check at line 354

**Exploit flow**:
1. `makerAddr` creates a buy offer: `buy = true`, `maxAssets = N`, tick chosen so `buyerPrice < WAD`
2. `makerAddr` calls `SetterRatifier.setIsRootRatified(makerAddr, root, true)` — permissionless
3. `takerAddr` calls `Midnight.take(offer, ..., units=1, takerAddr, ...)` in a loop (or via `multicall`)
4. Each iteration: `buyerAssets = 0`, `consumed` unchanged, 1 credit unit minted to `makerAddr`, 1 debt unit created on `takerAddr`, 0 tokens transferred
5. Step 3 is repeatable indefinitely; `consumed` never reaches `maxAssets`

**Why existing checks fail**:
- `require(newConsumed <= offer.maxAssets)` only checks the cap is not exceeded; it does not require `buyerAssets > 0`
- No `require(units == 0 || buyerAssets > 0)` guard exists anywhere in `take`
- `SelfTake` check (`offer.maker != taker`) is bypassed by using two distinct controlled addresses
- The protocol's own NatDoc at line 94 acknowledges the gap without marking it as acceptable behavior

## Impact Explanation

The attacker accumulates an unbounded number of credit units at zero asset cost. At maturity (or whenever `withdrawable` is funded by legitimate borrower repayments), those credit units are redeemable for real assets from the pool via `withdraw`. The corresponding debt on `takerAddr` is abandoned as bad debt, which is socialized across all legitimate lenders via the `lossFactor` slashing mechanism, diluting their redemption value. The attacker's net gain equals the assets redeemed minus zero paid, funded entirely by other lenders' capital. The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` confirms the state change is real: `creditOf`, `debtOf`, and `totalUnits` all increase while token balances remain unchanged.

## Likelihood Explanation

All preconditions are trivially satisfiable by any unprivileged user: any tick below `MAX_TICK` gives `buyerPrice < WAD`, and `units = 1` always satisfies the rounding condition. No oracle manipulation, no admin access, and no special market state is required. The attacker only needs two EOAs and a call to the permissionless `setIsRootRatified`. The attack is repeatable in a single transaction via `multicall`, making it gas-efficient at scale on low-fee chains.

## Recommendation

Add a guard in `take` that requires a nonzero asset transfer whenever `units > 0` and `offer.maxAssets > 0`:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(units == 0 || buyerAssets > 0, ZeroAssetTake());
}
```

Alternatively, enforce a minimum `units` value such that `units * buyerPrice >= WAD` when `offer.maxAssets > 0` and `offer.buy == true`. A complementary fix is to also require `buyerAssets > 0` before updating `consumed`, ensuring the cap accounting always advances on a non-trivial take.

## Proof of Concept

The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already provides a reproducible proof path. Minimal manual steps:

1. Deploy `Midnight` on a local fork
2. Create a buy offer: `buy = true`, `maxAssets = 1e18`, tick = 1 (so `buyerPrice < WAD`)
3. Ratify the offer root via `SetterRatifier.setIsRootRatified`
4. Call `take(offer, ..., units=1, takerAddr, ...)` 1000 times via `multicall`
5. Assert: `creditOf(id, makerAddr) == 1000`, `debtOf(id, takerAddr) == 1000`, `consumed[makerAddr][group] == 0`, token balances unchanged
6. At maturity, after any repayment funds `withdrawable`, call `withdraw` from `makerAddr` to extract real assets
