Audit Report

## Title
Flash Loan Repayment Uses Nominal Amount Without Balance Check, Allowing Fee-on-Transfer Tokens to Drain Protocol Reserves - (File: src/Midnight.sol)

## Summary
The `flashLoan` function dispatches tokens outbound and pulls back exactly the same nominal amount inbound via `safeTransferFrom`, with no pre/post balance snapshot. For fee-on-transfer tokens, the inbound `safeTransferFrom` call succeeds (returns `true`) but Midnight receives only `assets[i] - fee`, permanently reducing the protocol's reserve by `fee` per invocation. The Certora `Solvency.spec` explicitly assumes standard ERC20 semantics and does not cover this case.

## Finding Description
**Exact code path** (`src/Midnight.sol`, lines 737–751):

```solidity
function flashLoan(address[] calldata tokens, uint256[] calldata assets, address callback, bytes calldata data)
    external
{
    require(tokens.length == assets.length, InconsistentInput());
    emit EventsLib.FlashLoan(msg.sender, tokens, assets, callback);
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);   // (A)
    }
    require(
        IFlashLoanCallback(callback).onFlashLoan(msg.sender, tokens, assets, data) == CALLBACK_SUCCESS,
        WrongFlashLoanCallbackReturnValue()
    );
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]); // (B)
    }
}
```

**Root cause:** Repayment verification is purely nominal. The protocol checks that `transferFrom(callback, this, assets[i])` does not revert and returns `true`, but never verifies that `balanceOf(address(this))` is restored to its pre-loan value.

**Exploit flow (per token, per call):**

| Step | Action | Midnight balance | Callback balance |
|------|--------|-----------------|-----------------|
| Start | — | B | 0 |
| (A) `safeTransfer(token, callback, assets[i])` | Fee deducted outbound | B − assets[i] | assets[i] − fee |
| `onFlashLoan` | Callback tops up `fee` from own reserves; approves Midnight for `assets[i]` | B − assets[i] | assets[i] |
| (B) `safeTransferFrom(token, callback, this, assets[i])` | Fee deducted inbound | **B − fee** | 0 |

The call at (B) does not revert — `transferFrom` returns `true` — but Midnight receives only `assets[i] − fee`.

**Why existing checks fail:**
- `SafeTransferLib.safeTransferFrom` only checks the boolean return value, not the actual received delta.
- There is no `balanceBefore` / `balanceAfter` guard anywhere in `flashLoan`.
- `certora/specs/Solvency.spec` lines 31–33 explicitly state: `// Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver` — the `tokenBalanceCorrect` invariant is proven only under this assumption and does not protect against fee-on-transfer tokens.
- `live_context.json` line 233 lists fee-on-transfer tokens as requiring testing "if not explicitly excluded," and no such exclusion exists in the contract or `SECURITY.md`.

## Impact Explanation
For every `flashLoan` call using a fee-on-transfer token, the protocol's token balance decreases by exactly `fee`. This directly erodes the reserves backing lender withdrawals, collateral claims, and fee distributions. Repeated calls compound the loss linearly. The invariant that contract balances cover all obligations is violated, constituting direct unauthorized movement of protocol assets — an in-scope critical impact per `RESEARCHER.md`.

## Likelihood Explanation
**Preconditions:**
1. A fee-on-transfer token must be present in Midnight's balance (as a loan token or collateral token in any market, or via direct transfer).
2. The attacker must control a callback contract that can top up `fee` tokens and approve the repayment.
3. No privileged access is required; `flashLoan` is fully permissionless and imposes no token whitelist.

**Feasibility:** Fee-on-transfer tokens are a real, deployed token class (e.g., STA, PAXG, tokens with deflationary mechanics). The attack is repeatable indefinitely at the cost of `fee` tokens per call. If the attacker can source the fee tokens cheaply, the cost approaches zero relative to the damage inflicted on protocol solvency.

## Recommendation
Add a balance snapshot before the outbound transfer and assert full restoration after the inbound transfer:

```solidity
for (uint256 i = 0; i < tokens.length; i++) {
    uint256 balanceBefore = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
    // store balanceBefore for post-repayment check
    _flashBalances[i] = balanceBefore;
}
// ... callback ...
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(
        IERC20(tokens[i]).balanceOf(address(this)) >= _flashBalances[i],
        InsufficientRepayment()
    );
}
```

Alternatively, document and enforce a token whitelist that excludes fee-on-transfer tokens from use in flash loans, and add a corresponding check at the entry point.

## Proof of Concept
**Minimal fork test plan:**
1. Deploy a mock ERC20 with a 1% transfer fee.
2. Seed Midnight with 1000 tokens of this ERC20.
3. Deploy an attacker callback contract that: (a) receives the flash loan, (b) tops up 10 tokens (the fee), (c) approves Midnight for 1000 tokens, (d) returns `CALLBACK_SUCCESS`.
4. Call `flashLoan([feeToken], [1000], attackerCallback, "")`.
5. Assert `feeToken.balanceOf(address(midnight)) == 990` (reduced by 10, the 1% fee on the inbound transfer).
6. Repeat 100 times; assert Midnight balance approaches 0 while attacker's net cost is only the fee tokens supplied per round.
