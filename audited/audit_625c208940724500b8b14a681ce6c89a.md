Audit Report

## Title
Negative-Rebasing Loan Token Breaks Solvency Invariant via Missing Balance-Delta Verification in `safeTransferFrom` - (File: src/libraries/SafeTransferLib.sol)

## Summary
`SafeTransferLib.safeTransferFrom` verifies only call success and ERC20 return value, never the actual balance delta received by the contract. Because `touchMarket` imposes no restriction on the loan token type, an unprivileged attacker can deploy a negative-rebasing ERC20 as the loan token, establish `withdrawable` and `claimableSettlementFee` balances through normal protocol interactions, then trigger a rebase that silently reduces `balanceOf(midnight, loanToken)` while all accounting state remains unchanged, permanently violating the solvency invariant and making full redemption impossible for lenders and fee claimers.

## Finding Description

**Root cause — `SafeTransferLib.safeTransferFrom` (lines 24–34):**

`safeTransferFrom` calls `transferFrom` and checks only that the call did not revert and did not return `false`. No `balanceOf(address(this))` snapshot is taken before or after the call. Any transfer that succeeds nominally is accepted regardless of the actual balance delta received. [1](#0-0) 

**Permissionless market creation with unrestricted loan token — `touchMarket` (lines 755–791):**

`touchMarket` validates maturity, collateral params, LLTV tiers, and maxLif values, but places **no restriction on `market.loanToken`**. Any address with deployed code is accepted. An unprivileged caller can create a market whose loan token is a contract they fully control. [2](#0-1) 

**Accounting increment without delta verification — `repay` (lines 508–520):**

`withdrawable` is incremented by the nominal `units` value at line 509 before the transfer at line 520. The transfer is never checked against an actual balance delta. [3](#0-2) 

**Same pattern in `take` (lines 418 and 455):**

`claimableSettlementFee` is incremented at line 418 by `buyerAssets - sellerAssets`, and the corresponding inbound transfer occurs at line 455 with no balance delta verification. [4](#0-3) [5](#0-4) 

**Formal verification explicitly excludes rebasing tokens — `Solvency.spec` (line 31):**

The spec states: *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits."* The `tokenBalanceCorrect` strong invariant (lines 162–163), which asserts `balanceOf(midnight, token) >= collateralSum + withdrawableSum + claimableSettlementFee - flashloans`, is therefore **not proven** for rebasing tokens. [6](#0-5) [7](#0-6) 

**Protocol's own open-concern list confirms no on-chain exclusion — `live_context.json` (line 233):**

The project's `live_context.json` lists *"fee-on-transfer, rebasing, false-return, ERC777-like hooks, and non-standard decimals should be tested if not explicitly excluded"* as an open concern, confirming the team is aware of the gap and that no on-chain guard exists. [8](#0-7) 

**Exploit flow:**
1. Attacker deploys `NegativeRebaseToken` — a standard ERC20 with an owner-callable `rebase(uint256 pct)` that multiplies every balance by `(WAD - pct) / WAD`.
2. Attacker calls `midnight.touchMarket(market)` with `market.loanToken = address(NegativeRebaseToken)`. No check blocks this.
3. Attacker acts as both lender (buy-offer maker) and borrower (sell-offer taker) in the same market to self-fund state. After settlement, borrower has `debt = D`, lender has `credit = D`.
4. Borrower calls `repay(market, D, ...)`. At line 509, `withdrawable += D`. At line 520, `safeTransferFrom` pulls `D` tokens. Contract balance = `D`. Invariant holds.
5. Settlement fee spread from `take` means `claimableSettlementFee[loanToken] = F > 0`.
6. Attacker calls `NegativeRebaseToken.rebase(X)`. Every balance, including `balanceOf(midnight)`, drops by `X%`. Contract balance becomes `D * (1 - X/100)`.
7. `withdrawable + claimableSettlementFee = D + F > D * (1 - X/100) = balanceOf(midnight, loanToken)`. Solvency invariant violated.
8. The first lender to call `withdraw` drains the remaining balance; subsequent lenders and the fee claimer receive zero.

**Existing checks are insufficient:** `SECURITY.md` contains no explicit exclusion for rebasing or non-standard tokens. The formal verification assumption in `Solvency.spec` is a prover-level constraint, not an on-chain guard. `touchMarket` enforces no token allowlist or interface check.

## Impact Explanation

The solvency invariant `balanceOf(midnight, loanToken) >= withdrawableSum + claimableSettlementFee` is concretely and permanently violated. Lenders whose `credit` maps to `withdrawable` units and the fee claimer whose `claimableSettlementFee` is non-zero cannot fully redeem their entitlements. The shortfall is permanent and proportional to the rebase magnitude and the total outstanding withdrawable plus fee balance. This constitutes direct, irreversible loss of funds for affected lenders and fee claimers — a critical in-scope impact.

## Likelihood Explanation

**Preconditions:**
- Attacker deploys a negative-rebasing ERC20 (~50 lines of Solidity, trivial).
- Attacker calls `touchMarket` (permissionless, gas cost only).
- At least one repayment or settlement-fee-generating `take` must have occurred. The attacker can self-fund this state by acting as both lender and borrower, or can attract third-party users to a market with a token that superficially resembles a legitimate asset.
- Attacker triggers `rebase()` (one transaction).

The attack requires no privileged access, no oracle manipulation, and no victim cooperation to establish the vulnerable state (attacker can self-fund). Harm to third parties requires attracting users to the malicious market, which is feasible in a permissionless protocol where any market can be created. The attack is repeatable across any number of markets.

## Recommendation

1. **Validate loan token on market creation:** In `touchMarket`, require `market.loanToken.code.length > 0` and optionally maintain an allowlist of approved loan tokens (as Morpho Blue does with its factory pattern), or at minimum document the exclusion prominently in contract-level NatSpec.
2. **Balance-delta verification:** For inbound transfers in `repay`, `take`, `supplyCollateral`, and `liquidate`, snapshot `balanceOf(address(this))` before and after `safeTransferFrom` and require the delta equals the nominal `value`. This is the canonical defense against both fee-on-transfer and rebasing tokens.
3. **Formal verification alignment:** If the protocol intends to support only non-rebasing tokens, enforce this on-chain so the `Solvency.spec` assumption is backed by a real guard rather than an unverified precondition.

## Proof of Concept

**Minimal Foundry test plan:**

```solidity
// 1. Deploy NegativeRebaseToken with owner = attacker.
//    rebase(pct) multiplies all balances by (1e18 - pct) / 1e18.

// 2. Create market with loanToken = NegativeRebaseToken.
midnight.touchMarket(market); // succeeds, no token check

// 3. Attacker is lender: place buy offer, attacker is borrower: take sell offer.
//    After take: borrower.debt = D, lender.credit = D, claimableSettlementFee = F.

// 4. Borrower repays.
midnight.repay(market, D, borrower, address(0), "");
assertEq(midnight.withdrawable(id), D);
assertEq(NegativeRebaseToken.balanceOf(address(midnight)), D);

// 5. Trigger 50% negative rebase.
NegativeRebaseToken.rebase(0.5e18);
assertEq(NegativeRebaseToken.balanceOf(address(midnight)), D / 2);

// 6. Invariant violated.
assert(midnight.withdrawable(id) + midnight.claimableSettlementFee(address(NegativeRebaseToken)) > D / 2);

// 7. First lender withdraws D/2, second lender gets 0.
midnight.withdraw(market, D, lender1, lender1); // receives D/2, not D
midnight.withdraw(market, D, lender2, lender2); // reverts or receives 0
```

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }

    function touchMarket(Market memory market) public returns (bytes32) {
        bytes32 id = toId(market);
        if (marketState[id].tickSpacing == 0) {
            require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
            require(market.collateralParams.length > 0, NoCollateralParams());
            require(market.collateralParams.length <= MAX_COLLATERALS, TooManyCollateralParams());
            address previousCollateralToken;
            for (uint256 i = 0; i < market.collateralParams.length; i++) {
                address collateralToken = market.collateralParams[i].token;
                require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
                uint256 lltv = market.collateralParams[i].lltv;
                require(isLltvAllowed(lltv), LltvNotAllowed());
                require(
                    market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_LOW)
                        || market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_HIGH),
                    InvalidMaxLif()
                );
                previousCollateralToken = collateralToken;
            }

            MarketState storage _marketState = marketState[id];
            _marketState.tickSpacing = DEFAULT_TICK_SPACING;

            _marketState.settlementFeeCbp0 = _defaultSettlementFeeCbp[0];
            _marketState.settlementFeeCbp1 = _defaultSettlementFeeCbp[1];
            _marketState.settlementFeeCbp2 = _defaultSettlementFeeCbp[2];

            IdLib.storeInCode(market, INITIAL_CHAIN_ID);

            emit EventsLib.MarketCreated(market, id);
        }
        return id;
