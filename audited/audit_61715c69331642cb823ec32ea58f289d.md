Audit Report

## Title
Fee-on-transfer loan token silently overcounts `claimableSettlementFee` in `take`, enabling protocol insolvency - (`File: src/Midnight.sol`)

## Summary
In `take`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal `buyerAssets - sellerAssets` before the inbound `safeTransferFrom` executes. Because `SafeTransferLib.safeTransferFrom` only checks the boolean return value of `transferFrom` with no balance-delta accounting, a fee-on-transfer loan token causes only `(buyerAssets - sellerAssets - transferFee)` to actually arrive while the full nominal amount is credited. The deficit accumulates across every `take`, and when `feeClaimer` calls `claimSettlementFee` against the overcounted balance, it drains tokens backing lender `withdrawable` credits or borrower collateral, causing direct protocol insolvency.

## Finding Description

**Exact code path:**

1. `src/Midnight.sol` line 418: `claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;` — accounting state written before any transfer occurs. [1](#0-0) 

2. `src/Midnight.sol` line 455: `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);` — for a fee-on-transfer token, only `(buyerAssets - sellerAssets - transferFee)` arrives. [2](#0-1) 

3. `src/libraries/SafeTransferLib.sol` lines 24–34: `safeTransferFrom` calls `transferFrom` and checks only the boolean return value. No balance delta is measured. A fee-on-transfer token returns `true` while silently withholding the fee, so no revert occurs. [3](#0-2) 

4. `src/Midnight.sol` lines 305–309: `claimSettlementFee` decrements `claimableSettlementFee[token]` by `amount` and calls `safeTransfer(token, receiver, amount)`. If `claimableSettlementFee` exceeds actual tokens held for that purpose, this transfer pulls from tokens backing lender `withdrawable` or borrower collateral. [4](#0-3) 

**Root cause:** The protocol credits `claimableSettlementFee` with the nominal transfer amount rather than the actual received amount. `SafeTransferLib` provides no mechanism to detect the shortfall.

**Why existing checks fail:**

`certora/specs/Solvency.spec` line 31 explicitly states: *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver."* [5](#0-4) 

The `CVL_transferFrom` ghost (lines 43–59) moves the full `value` between ghost balances unconditionally, so the `tokenBalanceCorrect` invariant (lines 162–177) and `pendingFeeReceiptZero` invariant (lines 157–158) are proven only under this assumption. [6](#0-5) [7](#0-6) 

The `pendingFeeReceipt` ghost (lines 142–151) tracks the gap between when `claimableSettlementFee` is incremented and when the transfer arrives, and `pendingFeeReceiptZero` proves this gap closes to zero after each transaction — but only because the ghost model assumes the full `value` arrives. With a fee-on-transfer token, the ghost clears `pendingFeeReceipt` even though in reality less arrived. [8](#0-7) 

**No loan token validation exists:** `touchMarket` (lines 755–791) validates collateral params, maturity, and LLTV but performs zero validation on `market.loanToken`. [9](#0-8) 

The protocol design explicitly states "arbitrary loan token" (`live_context.json` line 17), and `live_context.json` line 233 states fee-on-transfer tokens "should be tested if not explicitly excluded" — they are not excluded anywhere in the codebase. [10](#0-9) [11](#0-10) 

## Impact Explanation

Each `take` with a fee-on-transfer loan token creates a deficit: `claimableSettlementFee[token]` exceeds actual tokens held for that purpose by `transferFee` per call. The deficit accumulates across every take in the affected market. When `feeClaimer` calls `claimSettlementFee` (acting normally, not maliciously), `safeTransfer` is executed against the overcounted balance. Once the cumulative shortfall exceeds the slack in the contract's token balance, the transfer pulls from tokens backing lender `withdrawable` credits or borrower collateral. This breaks the core solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee`, causing lenders to be unable to withdraw and/or collateral to be unrecoverable — direct protocol insolvency and loss of user funds. [12](#0-11) 

## Likelihood Explanation

**Preconditions:**
- A market exists (or is created by the attacker) with a fee-on-transfer loan token.
- The attacker (as taker) has approved Midnight for the loan token and holds sufficient balance.

Market creation is permissionless with no loan token validation. The attacker can be simultaneously the token deployer, market creator, offer maker, and taker — requiring no cooperation from other users. `live_context.json` explicitly lists "token charges fee" as an external behavior to test (line 389) and states fee-on-transfer tokens "should be tested if not explicitly excluded" (line 233); they are not excluded anywhere in the codebase. [13](#0-12) 

The attack is repeatable on every `take` call, accumulating the deficit until `feeClaimer` claims fees. Fee-on-transfer tokens (tax tokens, some rebasing tokens) are a well-known ERC20 variant. `SECURITY.md` contains no exclusion for fee-on-transfer tokens or non-standard ERC20 behavior. [14](#0-13) 

## Recommendation

Use balance-before/after accounting to measure the actual received amount and credit only that amount to `claimableSettlementFee`:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived;
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens by adding a validation check in `touchMarket`. If the protocol intends to support arbitrary loan tokens (as stated in `live_context.json`), the balance-delta approach is required.

## Proof of Concept

**Minimal Foundry test plan:**

1. Deploy a mock ERC20 with a 1% transfer fee (deducted from the received amount on every `transferFrom`).
2. Call `touchMarket` to create a market with this token as `loanToken`.
3. Create a buy offer (maker) and take it (taker) for `N` units, where `buyerAssets - sellerAssets = F` (the settlement fee).
4. Assert: `claimableSettlementFee[feeToken] == F` but `feeToken.balanceOf(address(midnight)) < F` (only `F * 0.99` arrived).
5. Repeat `take` `K` times to accumulate deficit `K * F * 0.01`.
6. Ensure lenders have deposited enough to cover `withdrawable` and collateral is posted.
7. Call `claimSettlementFee(feeToken, claimableSettlementFee[feeToken], receiver)`.
8. Assert: the transfer succeeds by draining tokens that back lender `withdrawable` or borrower collateral, and subsequent lender `withdraw` or collateral retrieval reverts — demonstrating insolvency.

```text
function CVL_transferFrom(env e, address token, address src, address dest, uint256 value) returns bool {
    if (tokenBalances[token][src] < value || tokenBalances[token][dest] + value >= 2 ^ 256) {
        revert();
    }

    // Non-deterministically set success, which allows to simulate permissions.
    bool success;
    if (success) {
        tokenBalances[token][src] = assert_uint256(tokenBalances[token][src] - value);
        tokenBalances[token][dest] = assert_uint256(tokenBalances[token][dest] + value);
    
        // Settle pending settlement fee receipts only on the exact fee transfer expected by take().
        if (dest == currentContract && pendingFeeReceipt[token] == to_mathint(value)) {
            pendingFeeReceipt[token] = 0;
        }
    }
    return success;
```
