### Title
Division by Zero in `liquidate` When `collateralIndex` Not in Borrower's Bitmap and `repaidUnits > 0` - (File: src/Midnight.sol)

### Summary
In `Midnight.liquidate`, `liquidatedCollatPrice` is initialized to `0` and only set inside the bitmap loop when `i == collateralIndex`. If the borrower has no collateral at `collateralIndex` (bit not set in their bitmap), the loop never assigns the price, leaving `liquidatedCollatPrice == 0`. When `repaidUnits > 0`, the subsequent `mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice)` at line 652 divides by zero, causing an unconditional revert. The Certora formal verification spec explicitly acknowledges this gap by requiring the bitmap bit to be set as a precondition rather than proving it holds unconditionally.

### Finding Description

**Root cause — unguarded `liquidatedCollatPrice` initialization:** [1](#0-0) 

`liquidatedCollatPrice` starts at `0`. The loop only assigns it on `if (i == collateralIndex)`. If `collateralIndex` is absent from `_position.collateralBitmap`, the assignment never executes.

**Division by zero site:** [2](#0-1) 

When `repaidUnits > 0` (the `else` branch), `mulDivDown(ORACLE_PRICE_SCALE, 0)` triggers a Solidity division-by-zero panic.

**No existing check prevents this.** The only input guards are: [3](#0-2) 

Neither `atMostOneNonZero` nor `_position.debt > 0` requires `collateralIndex` to be in the bitmap. The `NotLiquidatable` check at line 620–624 is satisfied by the position being unhealthy based on *other* collateral indices.

**Certora spec explicitly assumes the bit is set — it does not prove it:** [4](#0-3) 

The comment reads: *"Assume that the collateral price is non-zero and the collateral is active. Otherwise, liquidate may revert with div by zero."* This is a `require` assumption in the prover, not a `require` in the contract — the gap is unguarded on-chain.

**Exploit flow:**
1. Market has collateral params at indices 0 and 1.
2. Borrower supplies collateral only at index 0 (bitmap = `0b01`), borrows, and the position becomes unhealthy (price of index-0 collateral drops so `debt > maxDebt`).
3. Liquidator calls `liquidate(market, collateralIndex=1, seizedAssets=0, repaidUnits=N, borrower, false, ...)` with `N > 0`.
4. Bitmap loop iterates only over bit 0; `liquidatedCollatPrice` is never set (stays `0`).
5. `NotLiquidatable` check passes because `originalDebt > maxDebt`.
6. Execution reaches line 652: `seizedAssets = N.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, 0)` → **panic: division by zero**.

The protocol's own NatSpec acknowledges the asymmetry: [5](#0-4) 

Zero-amount liquidations are explicitly permitted on non-activated collateral (for bad-debt realization), but non-zero liquidations are not guarded against it.

### Impact Explanation
Any liquidator passing a `collateralIndex` absent from the borrower's bitmap with `repaidUnits > 0` receives an unconditional division-by-zero revert. This violates the core invariant that *unhealthy positions remain liquidatable*: the position is genuinely liquidatable, but the call reverts unexpectedly. A borrower who has never activated a particular collateral index is effectively immune to liquidation attempts targeting that index with a `repaidUnits` input, even though the protocol exposes no explicit guard communicating this restriction.

### Likelihood Explanation
The precondition is straightforward to reach: any market with more than one collateral param, where the borrower has only activated a subset of them. A liquidator (or an automated bot) that iterates over all `market.collateralParams` indices and calls `liquidate` with `repaidUnits > 0` for each will revert on every index not in the borrower's bitmap. The borrower cannot manufacture this state maliciously (they cannot remove a collateral index while holding debt, since `withdrawCollateral` requires `isHealthy`), but the state arises naturally whenever a borrower uses fewer collateral types than the market supports.

### Recommendation
Add a bitmap membership check before the price-dependent arithmetic. When `repaidUnits > 0 || seizedAssets > 0`, require that `collateralIndex` is set in the borrower's bitmap:

```solidity
if (repaidUnits > 0 || seizedAssets > 0) {
    require(
        _position.collateralBitmap & (1 << collateralIndex) != 0,
        CollateralNotActivated()
    );
    // ... existing lif / seizedAssets / repaidUnits logic
}
```

Alternatively, after the bitmap loop, assert `liquidatedCollatPrice > 0` when a non-zero liquidation is requested. Either fix mirrors the assumption already encoded in the Certora spec and aligns with the existing NatSpec that zero-amount liquidations are the only permitted path for non-activated collateral.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity 0.8.34;

// Foundry unit test — expected to demonstrate division-by-zero revert.
// Setup: two-collateral market; borrower activates only index 0.

function testLiquidateDivByZeroCollateralNotInBitmap() public {
    // 1. Create market with two collateral params (index 0 and index 1).
    //    Both oracles return a non-zero price.
    Market memory mkt = _twoCollateralMarket();
    oracle0.setPrice(ORACLE_PRICE_SCALE);   // index 0 price = 1
    oracle1.setPrice(ORACLE_PRICE_SCALE);   // index 1 price = 1

    // 2. Borrower supplies collateral ONLY at index 0 and borrows.
    uint256 units = 100;
    _supplyCollateral(mkt, borrower, /*collateralIndex=*/0, units);
    _borrow(mkt, borrower, units);

    // 3. Drop index-0 price to make position unhealthy.
    oracle0.setPrice(0);  // maxDebt = 0 < debt = units → unhealthy

    // Precondition: bit 1 is NOT set in borrower's bitmap.
    assertEq(midnight.collateralBitmap(id, borrower) & (1 << 1), 0);

    // 4. Liquidator calls liquidate targeting index 1 with repaidUnits > 0.
    //    Expect division-by-zero panic (0x12).
    vm.expectRevert(stdError.divisionError);
    midnight.liquidate(
        mkt,
        /*collateralIndex=*/ 1,
        /*seizedAssets=*/    0,
        /*repaidUnits=*/     1,
        borrower,
        /*postMaturityMode=*/ false,
        address(this),
        address(0),
        ""
    );
    // Assert: the position is still unhealthy (liquidatable) after the revert,
    // confirming the invariant "unhealthy positions remain liquidatable" is broken
    // for this call pattern.
    assertFalse(midnight.isHealthy(mkt, id, borrower));
}
```

**Expected assertions:**
- The call reverts with a division-by-zero panic.
- `midnight.isHealthy(mkt, id, borrower)` returns `false` after the revert, confirming the position was genuinely liquidatable but the call failed unexpectedly.
