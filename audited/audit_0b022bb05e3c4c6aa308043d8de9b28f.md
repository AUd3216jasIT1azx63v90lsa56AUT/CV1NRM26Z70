Audit Report

## Title
Debt Increase Permitted at Exact Maturity Timestamp via `<=` Boundary — (File: `src/Midnight.sol`)

## Summary
The maturity guard in `take` uses `block.timestamp <= offer.market.maturity` (line 391), allowing `sellerDebtIncrease > 0` when `block.timestamp == maturity`. The liquidation guard uses the strict `block.timestamp > market.maturity` (line 622) for post-maturity mode. This one-second asymmetry permits debt creation at exactly `T` while making that debt immediately subject to health-agnostic post-maturity liquidation at `T+1`, directly violating the protocol's own invariant at `live_context.json` line 221.

## Finding Description
**Root cause — asymmetric boundary operators:**

`src/Midnight.sol` line 391 uses `<=`:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
At `block.timestamp == T`, `T <= T` is `true`, so the require passes unconditionally. `sellerDebtIncrease` is computed at line 384 and written to state at line 414:
```solidity
uint256 sellerDebtIncrease = units - sellerCreditDecrease;
...
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
```

`src/Midnight.sol` lines 620–622 use strict `>`:
```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```
At `T`, `T > T` is `false` — post-maturity liquidation is blocked. At `T+1`, `T+1 > T` is `true` — any borrower with debt is liquidatable regardless of health, per `live_context.json` line 88: `"overdue_debt_after_maturity_is_liquidatable_even_if_healthy": true`.

**Protocol invariants explicitly violated:**
- `live_context.json` line 221: `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"` — names this exact case.
- `live_context.json` line 86: `"debt_increase_after_maturity_forbidden": true`.

**Exploit flow:**
1. A market with `maturity = T` is public on-chain. Attacker (or cooperating validator) targets the block with `block.timestamp == T`.
2. Attacker submits `take` with a sell offer where `sellerDebtIncrease > 0`. The check `T <= T` passes; debt is written to the seller's position.
3. At `T+1`, liquidator calls `liquidate(..., postMaturityMode=true)`. The check `T+1 > T` passes. The position is liquidatable even if fully collateralized.
4. LIF grows from ~1 at `T+1` to `maxLif` over 15 minutes, at which point the liquidator seizes collateral at full incentive from a position that was healthy at creation.

If the victim is a maker (seller) whose offer is taken at exactly `T` by a front-running taker, the victim had no ability to prevent the take, and their collateral becomes seizable within 15 minutes.

## Impact Explanation
Unauthorized collateral seizure from a healthy position. A borrower whose debt is created at exactly `T` is immediately subject to post-maturity liquidation at `T+1` with no health-based protection. This falls directly under the protocol's listed best bug classes: `"unauthorized collateral seizure"` and `"healthy-account liquidation"`. The borrower suffers a concrete loss of collateral from a position that should never have had debt created at that timestamp per the protocol's own invariant.

## Likelihood Explanation
Requires `block.timestamp == market.maturity` exactly. Market maturities are public on-chain parameters, making the target timestamp known in advance. On proof-of-stake Ethereum, validators control block timestamps within the allowed drift window (~12 seconds), making exact timestamp targeting feasible for a validator-attacker. For non-validators, the condition requires a transaction landing in the exact maturity block, which is probabilistically achievable by monitoring the mempool and submitting with appropriate gas. The condition is repeatable across any market whose maturity falls on a future block.

## Recommendation
Change the boundary operator at `src/Midnight.sol` line 391 from `<=` to `<`:
```solidity
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
This ensures debt cannot be created at exactly `T`, making the take-side guard consistent with the liquidation-side guard's strict `>` operator. Both boundaries then agree: debt creation is forbidden at `T` and beyond, and post-maturity liquidation is available strictly after `T`.

## Proof of Concept
**Minimal fork test plan:**
1. Deploy a market with `maturity = block.timestamp + 1`.
2. Advance time to `block.timestamp == maturity` (e.g., `vm.warp(maturity)`).
3. Call `take` with a sell offer where `sellerDebtIncrease > 0`. Assert it succeeds and `position[id][seller].debt > 0`.
4. Advance time by 1 second (`vm.warp(maturity + 1)`).
5. Call `liquidate(..., postMaturityMode=true)` on the seller's position. Assert it succeeds despite the position being fully collateralized.
6. Confirm collateral is seized from a position that was healthy at debt creation.

**Invariant fuzz test:** Fuzz `block.timestamp` values in `[maturity - 1, maturity, maturity + 1]` and assert that `sellerDebtIncrease > 0` is only possible when `block.timestamp < maturity`, and that post-maturity liquidation is only possible when `block.timestamp > maturity`.
