Audit Report

## Title
Fee-on-Transfer Loan Token in `repay` Inflates `withdrawable` Beyond Actual Balance — (`src/Midnight.sol`)

## Summary
The `repay` function unconditionally credits `marketState[id].withdrawable` by the caller-supplied `units` before executing the token transfer, while `SafeTransferLib.safeTransferFrom` only validates the boolean return value and never measures the actual balance delta. When the market's `loanToken` is a fee-on-transfer token, the contract receives `units - fee` but records `units` as withdrawable, permanently breaking the solvency invariant and enabling direct loss of lender principal.

## Finding Description

**Exact code path:**

`repay` in `src/Midnight.sol` lines 508–520:
```solidity
position[id][onBehalf].debt      -= UtilsLib.toUint128(units);  // L508
marketState[id].withdrawable     += UtilsLib.toUint128(units);  // L509
...
SafeTransferLib.safeTransferFrom(
    market.loanToken, payer, address(this), units               // L520
);
``` [1](#0-0) 

`SafeTransferLib.safeTransferFrom` in `src/libraries/SafeTransferLib.sol` lines 24–34 checks only `token.code.length > 0`, call success, and the boolean return value. No balance-before/balance-after measurement is performed. [2](#0-1) 

**Root cause:** The accounting update at lines 508–509 is unconditionally based on the argument `units`, not on the actual tokens received. For a fee-on-transfer token, `transferFrom(payer, address(this), units)` returns `true` and emits a transfer event for `units`, but only `units - fee` arrives at the contract. The protocol has no balance-delta check.

**Market creation has no token validation:** `touchMarket` validates collateral params, maturity, and LLTV, but performs zero validation on `loanToken` properties. Market creation is permissionless. [3](#0-2) 

**Exploit flow:**
1. Attacker creates a market with a fee-on-transfer ERC-20 as `loanToken` (permissionless, no token validation).
2. Borrower takes a loan of `D` units via normal protocol flow.
3. Borrower calls `repay(market, D, onBehalf, address(0), "")`.
4. Protocol executes: `debt[onBehalf] -= D` and `withdrawable += D`.
5. `safeTransferFrom` transfers `D` tokens from borrower; only `D * (1 - feeRate)` arrives at `address(midnight)`.
6. `withdrawable` now exceeds actual balance by `D * feeRate`.
7. Lenders call `withdraw` — protocol executes `withdrawable -= D` and `safeTransfer(loanToken, receiver, D)`.
8. The `safeTransfer` of `D` drains tokens belonging to other lenders or other markets sharing the same token, causing a shortfall.

**Why existing checks fail:** `SafeTransferLib` is a return-value-only wrapper with no balance accounting. There is no market-creation validation rejecting fee-on-transfer tokens. The state mutation precedes the transfer, so no reentrancy guard would help. The Certora solvency spec explicitly assumes tokens transfer correctly (no fee-on-transfer), meaning formal verification does not cover this case. [4](#0-3) 

## Impact Explanation
Every `repay` call with a fee-on-transfer loan token permanently inflates `marketState[id].withdrawable` relative to the contract's actual token balance. The core invariant explicitly stated in `live_context.json` — "ERC20 transfer deltas must match accounting deltas" — is violated after the first such repayment. [5](#0-4) 

Lenders who withdraw after such repayments receive tokens that were deposited by other lenders or belong to other markets, causing direct, irreversible loss of lender principal. This is a direct theft of assets impact class.

## Likelihood Explanation
Any unprivileged actor can create a market with a fee-on-transfer loan token — market creation is permissionless and has no token validation. The borrower needs only a standard borrow-then-repay flow. The exploit is repeatable: each `repay` call widens the deficit by `units * feeRate`. Tokens with owner-configurable fees (e.g., USDT's historical fee switch) make this reachable without any malicious token-owner action at exploit time — the fee just needs to be active when `repay` is called. Fee-on-transfer tokens are explicitly listed as a test target in `live_context.json` and are not excluded by `SECURITY.md`. [6](#0-5) 

## Recommendation
Measure the actual balance delta in `repay` rather than trusting the caller-supplied `units`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;

position[id][onBehalf].debt      -= UtilsLib.toUint128(received);
marketState[id].withdrawable     += UtilsLib.toUint128(received);
```

Alternatively, add an explicit check `require(received == units)` to reject fee-on-transfer tokens at the call site. The same pattern should be applied to any other inbound transfer path (e.g., `take`, `liquidate`, `supplyCollateral`) that credits accounting before verifying receipt.

## Proof of Concept

**Minimal Foundry test plan:**
1. Deploy a mock ERC-20 with a configurable transfer fee (e.g., 1% deducted from recipient on every `transferFrom`).
2. Create a Midnight market with this token as `loanToken`.
3. Lender supplies liquidity; borrower takes a loan of `D` units.
4. Enable the fee on the mock token.
5. Borrower calls `repay(market, D, borrower, address(0), "")`.
6. Assert: `marketState[id].withdrawable == D` (inflated).
7. Assert: `IERC20(loanToken).balanceOf(address(midnight)) < D` (actual balance is `D * 0.99`).
8. Lender calls `withdraw(market, D, lender, lender)`.
9. Assert: the `safeTransfer` either reverts (if no other funds) or drains tokens belonging to another depositor, confirming the shortfall.
