### Title
Re-entrant `take()` via `onBuy()` Bypasses State-Dependent `enterGate.canIncreaseCredit` Check - (File: src/Midnight.sol)

### Summary

`Midnight.take()` updates the buyer's credit position in persistent storage before invoking the `onBuy()` buyer callback, but has no reentrancy guard preventing a re-entrant `take()` call during that callback. A malicious buyer callback can re-enter `take()` for the same buyer in the same gated market; the inner call's `canIncreaseCredit` gate check executes against the already-mutated intermediate credit state, allowing a gate with credit-threshold logic to be bypassed and the buyer to accumulate credit beyond the gate's intended cap.

### Finding Description

**Exact code path:**

In `src/Midnight.sol`, `take()` executes in this order:

1. Gate check (lines 397–401) — `canIncreaseCredit(buyer)` called with pre-update state.
2. State update (lines 408–410) — `buyerPos.credit += buyerCreditIncrease` written to storage.
3. Buyer callback (lines 445–453) — `IBuyCallback(buyerCallback).onBuy(...)` called with credit already committed.

```
// gate check — BEFORE state update
require(
    offer.market.enterGate == address(0) || buyerCreditIncrease == 0
        || IEnterGate(offer.market.enterGate).canIncreaseCredit(buyer),
    BuyerGatedFromIncreasingCredit()
);                                                          // lines 397-401

buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease); // line 410

// callback — AFTER state update, NO reentrancy guard
IBuyCallback(buyerCallback).onBuy(...);                     // lines 448-452
```

There is no transient-storage lock (or any other guard) preventing `take()` from being re-entered during `onBuy()`. The only transient lock in the function is `LIQUIDATION_LOCK_SLOT`, which protects only the seller from liquidation.

**Attacker-controlled inputs:**

- Taker (or authorized account) sets `takerCallback` to a malicious contract implementing `IBuyCallback`.
- For a buy offer, the maker sets `offer.callback` to the malicious contract.
- The malicious contract re-enters `take()` during `onBuy()` with a second valid offer for the same buyer in the same gated market.

**Exploit flow (concrete example):**

Precondition: gate implements `canIncreaseCredit(buyer)` as `creditOf(id, buyer) < X` (reads Midnight state; a natural credit-cap design).

1. Outer `take()`: buyer credit = 0. Gate sees 0 < X → returns `true`. Gate check passes.
2. Outer `take()` writes `buyerPos.credit = Y` (where Y < X) to storage.
3. Outer `take()` calls `onBuy()` on malicious callback.
4. Malicious callback re-enters `take()` for same buyer, different offer, same market.
5. Inner `take()`: gate calls `canIncreaseCredit(buyer)` → reads `creditOf = Y`. Since Y < X, returns `true`. Gate check passes.
6. Inner `take()` writes `buyerPos.credit = Y + Z` to storage.
7. Both calls complete. Final credit = Y + Z > X. Gate's cap is violated.

**Why existing checks fail:**

- The Certora spec (`certora/specs/Reverts.spec` lines 30–34) explicitly assumes no reentrancy for gate rules, justifying it with "gate checks precede callbacks so re-entrant state changes cannot affect them." This is only true for the *outer* call's gate check. The *inner* call's gate check executes after the outer call's state mutation, so it sees intermediate state.
- The `enterGateBlocksCreditIncrease` rule (lines 295–307) uses a ghost function returning a fixed value — it does not model state-dependent gate logic and therefore does not catch this scenario.
- No `nonReentrant` modifier or transient-storage lock guards `take()` against re-entry.

### Impact Explanation

A buyer whose callback is a malicious contract can accumulate credit in a gated market beyond the gate's intended per-user cap. Any gate whose `canIncreaseCredit` logic reads Midnight's own `creditOf` (or any state that `take()` mutates before the callback) is vulnerable. This directly violates the protocol invariant "gate checks must apply to all entry paths" and "callbacks must not corrupt partial state through reentrancy."

### Likelihood Explanation

Preconditions are realistic: (1) a gated market exists with a credit-threshold gate (a natural and common design), (2) the attacker is the taker and sets `takerCallback` to a malicious contract (fully attacker-controlled, no privilege required), (3) a second valid offer exists in the same market (easily arranged by the attacker as maker). The attack is repeatable within a single transaction and requires no special timing or oracle manipulation.

### Recommendation

Add a per-market reentrancy guard on `take()` using transient storage (consistent with the existing `LIQUIDATION_LOCK_SLOT` pattern). Before the first callback, set a market-scoped `TAKE_LOCK` slot; revert if it is already set on entry. Clear it after the final callback. Alternatively, move the gate checks to after all callbacks complete — but this changes semantics and may break exit-path guarantees. The transient-lock approach is minimal and consistent with the existing codebase pattern.

contract CreditThresholdGate is IEnterGate {
    IMidnight immutable midnight;
    bytes32 immutable id;
    uint256 immutable cap; // e.g. 100e18

    function canIncreaseCredit(address buyer) external view returns (bool) {
        return midnight.creditOf(id, buyer) < cap;
    }
    function canIncreaseDebt(address) external view returns (bool) { return true; }
}

contract MaliciousBuyCallback is IBuyCallback {
    // Stores a second offer to fill on re-entry
    function onBuy(bytes32, Market memory market, uint256, uint256, uint256, address buyer, bytes memory) external returns (bytes32) {
        // Re-enter take() with second offer, same buyer, same market
        IMidnight(msg.sender).take(secondOffer, "", innerUnits, buyer, buyer, address(0), "");
        return CALLBACK_SUCCESS;

}

    // Setup: gated market with cap = 100e18
    // Outer take: units = 60e18 → buyer credit becomes 60e18 (< 100e18, gate passes)
    // onBuy re-enters: units = 60e18 → gate sees 60e18 < 100e18, passes
