### Title
Fee-on-Transfer Token Causes `claimableSettlementFee` Overstatement in `take`, Amplified Linearly by `multicall` Batching - (File: src/Midnight.sol)

### Summary
In `Midnight.take`, `claimableSettlementFee` is incremented by the full nominal spread `buyerAssets - sellerAssets` before the corresponding `safeTransferFrom` to `address(this)`. When the loan token is fee-on-transfer, Midnight receives less than the recorded amount, creating a per-take shortfall. Batching N such takes via `multicall` accumulates a shortfall of `N * (buyerAssets - sellerAssets) * fee_rate`, violating the solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee`.

### Finding Description

**Exact code path:**

At `src/Midnight.sol:418`, `claimableSettlementFee` is incremented by the full nominal settlement fee:

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

Then at lines 455–456, the actual token pull happens:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
``` [2](#0-1) 

`safeTransferFrom` does not verify the amount actually received. With a fee-on-transfer token charging rate `f`, Midnight receives only `(buyerAssets - sellerAssets) * (1 - f)` but records `buyerAssets - sellerAssets` in `claimableSettlementFee`. The per-take shortfall is `(buyerAssets - sellerAssets) * f`.

**`multicall` amplification:**

`multicall` uses `delegatecall` in a loop with no inter-call balance checks:

```solidity
function multicall(bytes[] calldata calls) external {
    for (uint256 i = 0; i < calls.length; i++) {
        (bool success, bytes memory returnData) = address(this).delegatecall(calls[i]);
        ...
    }
}
``` [3](#0-2) 

Each delegated `take` call independently increments `claimableSettlementFee` by the full nominal amount and independently receives a fee-discounted transfer. After N takes, the total overstatement is `N * (buyerAssets - sellerAssets) * f`.

**Attacker inputs:** Any unprivileged taker constructs N `take` calldata entries (using distinct `offer.group` values to avoid `ConsumedUnits`/`ConsumedAssets` reverts) and passes them to `multicall`. No special privilege is required; `taker == msg.sender` satisfies the authorization check at line 346. [4](#0-3) 

**Why existing checks fail:** The Certora `tokenBalanceCorrect` solvency invariant and `pendingFeeReceiptZero` invariant are verified under the assumption that `SafeTransferLib` delivers exactly the requested amount. The `pendingFeeReceipt` ghost tracks the ordering gap (fee recorded before transfer) but assumes the transfer closes the gap exactly. [5](#0-4) 

With fee-on-transfer, the transfer only partially closes the gap, leaving a permanent residual shortfall per take that the formal model does not capture.

### Impact Explanation

After N takes via `multicall`, `claimableSettlementFee[loanToken]` exceeds the actual tokens held for fees by `N * (buyerAssets - sellerAssets) * f`. The solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee` is violated. When the fee claimer calls `claimSettlementFee` for the full recorded amount, the transfer either reverts (insufficient balance) or drains tokens that back lender `withdrawable` credits, causing lender withdrawals to fail. The insolvency magnitude scales linearly with N and the fee rate.

### Likelihood Explanation

**Preconditions:** The loan token must be fee-on-transfer. The taker must have sufficient approval and balance to pay `buyerAssets` per take. Multiple distinct offer groups must be available (or the maker must set `maxUnits`/`maxAssets` large enough). These are all normal operating conditions for a fee-on-transfer market.

**Feasibility:** Any taker can trigger this in a single transaction. No governance action, oracle manipulation, or privileged access is required. The shortfall is permanent and non-reversible once the takes execute.

**Repeatability:** The attack can be repeated across multiple transactions, accumulating unbounded shortfall over time.

### Recommendation

Record `claimableSettlementFee` based on the actual received amount rather than the nominal amount. Measure the contract's balance before and after the `safeTransferFrom` to `address(this)` and use the delta:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived;
```

Move the `claimableSettlementFee` increment to after the transfer and use `actualReceived`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens (e.g., via a market creation check or a known-token allowlist).

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {FeeOnTransferERC20} from "test/mocks/FeeOnTransferERC20.sol"; // 1% fee

contract FeeOnTransferMulticallTest is Test {
    Midnight midnight;
    FeeOnTransferERC20 feeToken; // 1% fee on every transfer

    function setUp() public {
        midnight = new Midnight();
        feeToken = new FeeOnTransferERC20(100); // 1% = 100 bps
        // create market with feeToken as loanToken, set up lender/borrower positions
    }

    function testFuzz_multicallFeeOnTransferDesync(uint8 N) public {
        N = uint8(bound(N, 1, 20));
        // Setup: lender has buy offers, borrower has collateral
        // ...

        uint256 feePerTake = /* buyerAssets - sellerAssets per take */ 1e16;
        uint256 balanceBefore = feeToken.balanceOf(address(midnight));
        uint256 claimableBefore = midnight.claimableSettlementFee(address(feeToken));

        // Build N take calls with distinct groups
        bytes[] memory calls = new bytes[](N);
        for (uint256 i = 0; i < N; i++) {
            offer.group = bytes32(i);
            calls[i] = abi.encodeCall(midnight.take, (offer, hex"", units, taker, taker, address(0), hex""));
        }
        vm.prank(taker);
        midnight.multicall(calls);

        uint256 claimableAfter = midnight.claimableSettlementFee(address(feeToken));
        uint256 balanceAfter = feeToken.balanceOf(address(midnight));

        uint256 nominalFeeAccrued = claimableAfter - claimableBefore;
        uint256 actualFeeReceived = balanceAfter - balanceBefore;

        // Assert: shortfall grows linearly with N
        assertGt(nominalFeeAccrued, actualFeeReceived, "claimable overstated");
        // shortfall ~= N * feePerTake * 0.01
        assertApproxEqRel(nominalFeeAccrued - actualFeeReceived, N * feePerTake / 100, 0.01e18);

        // Assert invariant violation: balance < claimable + withdrawable
        uint256 withdrawable = midnight.withdrawable(id);
        assertLt(balanceAfter, claimableAfter + withdrawable, "solvency invariant violated");
    }
}
```

**Expected assertions:** `nominalFeeAccrued > actualFeeReceived` with the gap scaling as `N * fee_rate * (buyerAssets - sellerAssets)`; the solvency invariant `balance >= claimableSettlementFee + withdrawable` fails.
