### Title
Fee-on-Transfer `loanToken` Inflates `claimableSettlementFee` Beyond Actual Balance in `take()` - (File: src/Midnight.sol)

### Summary

In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by `buyerAssets - sellerAssets` at line 418 before the actual `safeTransferFrom` at line 455. `SafeTransferLib.safeTransferFrom` only validates the boolean return value of `transferFrom`, not the actual balance delta. When `loanToken` is a fee-on-transfer token, the contract receives fewer tokens than the amount credited to `claimableSettlementFee`, permanently inflating the accounting beyond the real balance.

### Finding Description

**Exact code path:** [1](#0-0) 

This line executes unconditionally before any transfer. Then: [2](#0-1) 

`payer` is resolved as `buyerCallback` when `buyerCallback != address(0)`: [3](#0-2) 

`SafeTransferLib.safeTransferFrom` only checks the `transferFrom` return value — no balance-before/after check: [4](#0-3) 

**Root cause:** The accounting write at line 418 assumes the transfer at line 455 delivers exactly `buyerAssets - sellerAssets` tokens to `address(this)`. For a fee-on-transfer token, the contract receives `(buyerAssets - sellerAssets) * (1 - fee_rate)`, but `claimableSettlementFee` is inflated by the full `buyerAssets - sellerAssets`.

**Attacker inputs and exploit flow:**

1. Attacker deploys a fee-on-transfer ERC20 (e.g., 1% fee on every transfer) and calls `touchMarket()` with it as `loanToken`. `touchMarket()` has no restriction on token type: [5](#0-4) 

2. Attacker (as maker) creates a buy offer on this market with `offer.callback` set to a contract implementing `IBuyCallback` that holds the fee-on-transfer token and approves `Midnight` in `onBuy`.

3. An unprivileged taker calls `take()`. The `buyerCallback` is `offer.callback` (since `offer.buy == true`). The callback fires, returns `CALLBACK_SUCCESS`, then:
   - Line 455: `safeTransferFrom(loanToken, buyerCallback, address(this), buyerAssets - sellerAssets)` — token's `transferFrom` returns `true` but only delivers `(buyerAssets - sellerAssets) * 0.99` to the contract.
   - Line 456: `safeTransferFrom(loanToken, buyerCallback, receiver, sellerAssets)` — similarly delivers less.

4. After `take()` returns: `claimableSettlementFee[loanToken]` is `buyerAssets - sellerAssets`, but the contract's actual balance increase is `(buyerAssets - sellerAssets) * 0.99`. The gap is `(buyerAssets - sellerAssets) * 0.01`.

**Why existing checks fail:**

The Certora `Solvency.spec` formal verification explicitly assumes well-behaved ERC20 tokens with no fee-on-transfer: [6](#0-5) 

The `pendingFeeReceipt` ghost in `Solvency.spec` tracks the gap between the accounting write and the transfer, but clears it only when the exact expected amount arrives — it is never proven sound for fee-on-transfer tokens: [7](#0-6) 

The protocol's own `live_context.json` lists fee-on-transfer tokens as a concern that "should be tested if not explicitly excluded" — they are not excluded anywhere in the codebase.

### Impact Explanation

`claimableSettlementFee[loanToken]` permanently exceeds the contract's actual token balance attributable to fees. When `feeClaimer` calls `claimSettlementFee(token, amount, receiver)`: [8](#0-7) 

The `safeTransfer` at line 309 will revert if the contract lacks sufficient balance (DoS on fee claiming), or if the contract holds other tokens from lender deposits, it drains `withdrawable` funds that lenders depend on for `withdraw()`, breaking the core solvency invariant: [9](#0-8) 

### Likelihood Explanation

**Preconditions:**
- `loanToken` is a fee-on-transfer token (attacker-deployable, no admin action required)
- `buyerCallback != address(0)` (attacker-controlled via `offer.callback` on a buy offer, or `takerCallback` on a sell offer)
- Settlement fee > 0 so `buyerAssets - sellerAssets > 0`

All three preconditions are fully attacker-controlled. The attack is repeatable on every `take()` call, compounding the inflation. No privileged role is needed. The `touchMarket()` function is permissionless.

### Recommendation

Record the contract's token balance before and after the inbound transfer and use the actual delta to update `claimableSettlementFee`, rather than the computed `buyerAssets - sellerAssets`. Concretely, move the `claimableSettlementFee` increment to after line 455 and compute it as `balanceAfter - balanceBefore`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as `loanToken` by adding a validation in `touchMarket()` (e.g., a round-trip transfer check), and add a corresponding invariant test.
