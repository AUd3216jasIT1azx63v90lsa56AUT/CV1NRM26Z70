## Title
Fee-on-Transfer Loan Token Causes `withdrawable` Overcounting, Enabling Lender Fund Freeze or Cross-User Drain - (File: src/Midnight.sol)

## Summary

`repay()` unconditionally increments `marketState[id].withdrawable` by the caller-supplied `units` before invoking `SafeTransferLib.safeTransferFrom`, which performs no balance-delta check. When the market's `loanToken` is a fee-on-transfer token, only `units - fee` tokens arrive at the contract while `withdrawable` is inflated by the full `units`. A subsequent `withdraw(units)` by a lender then attempts to transfer `units` out of a contract that holds only `units - fee`, causing either a revert (fund freeze) or a drain of tokens belonging to other users.

## Finding Description

**Root cause — `repay()`** (`src/Midnight.sol` lines 508–520):

```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);   // (1) +units
...
SafeTransferLib.safeTransferFrom(                             // (2) +(units-fee) received
    market.loanToken, payer, address(this), units
);
``` [1](#0-0) 

`SafeTransferLib.safeTransferFrom` only checks the boolean return value; it performs no before/after balance comparison and cannot detect that fewer tokens than `units` were deposited. The gap `fee` accumulates in `withdrawable` with every `repay` call.

**Drain path — `withdraw()`** (`src/Midnight.sol` lines 494–499):

```solidity
_marketState.withdrawable -= UtilsLib.toUint128(units);   // (3) passes: inflated value
...
SafeTransferLib.safeTransfer(market.loanToken, receiver, units);  // (4) tries to send units
``` [2](#0-1) 

The underflow check at step (3) passes because `withdrawable` was inflated in step (1). Step (4) then attempts to transfer `units` when the contract holds only `units - fee`, causing a revert (fund freeze) or consuming tokens belonging to other users (cross-user drain).

**Market creation is permissionless with arbitrary loan token.** `touchMarket()` imposes no whitelist or token-type validation on `market.loanToken`: [3](#0-2) 

The protocol's own `live_context.json` confirms "permissionless market creation" and "arbitrary loan token" as design properties, and explicitly lists "market creator" as an attacker role and "token charges fee" as an in-scope external behavior test case. [4](#0-3) [5](#0-4) 

**Existing formal proofs are insufficient.** `Solvency.spec` line 31 explicitly assumes "no fee taking from sender or receiver, no rebasing" — the `tokenBalanceCorrect` invariant is not proven for fee-on-transfer tokens: [6](#0-5) 

`WithdrawableMonotonicity.spec` rule `repayIncreasesWithdrawable` asserts `withdrawableAfter == withdrawableBefore + units`, which is exactly the broken invariant under a fee-on-transfer token: [7](#0-6) 

**Exploit flow:**

1. Attacker (market creator) calls `touchMarket` with a fee-on-transfer ERC20 (e.g., 1% fee) as `loanToken`. No access control blocks this.
2. A lender supplies credit; attacker (borrower) takes `D` units of debt via `take()`.
3. Attacker calls `repay(market, D, borrower, address(0), "")`:
   - `withdrawable += D` (accounting: full amount).
   - `safeTransferFrom` moves only `D * 0.99` tokens into the contract.
   - Gap: `withdrawable` overstates actual balance by `D * 0.01`.
4. Lender calls `withdraw(market, D, lender, lender)`:
   - `withdrawable -= D` passes (inflated value).
   - `safeTransfer(loanToken, lender, D)` is attempted.
   - Contract holds only `D * 0.99` → **revert (fund freeze)**, or if other users' deposits cover the shortfall → **cross-user drain / theft**.

The shortfall compounds with every `repay` call.

## Impact Explanation

A lender whose credit is backed by repayments made with a fee-on-transfer loan token cannot withdraw their entitled funds. `withdraw()` either reverts permanently (fund freeze until the shortfall is externally covered) or succeeds by consuming tokens belonging to other lenders or fee claimers, constituting direct theft. The core solvency invariant `tokenBalance >= collateralSum + withdrawableSum + claimableSettlementFee` is violated. This matches the "direct loss of user funds," "protocol insolvency," and "permanent or long-term fund freeze" impact classes listed in `live_context.json`. [8](#0-7) [9](#0-8) 

## Likelihood Explanation

- Market creation is permissionless; any address can be a market creator (explicitly listed attacker role) and supply an arbitrary `loanToken` with no whitelist check.
- Fee-on-transfer ERC20s are a well-known, deployed token class (e.g., USDT with fee enabled, STA, PAXG with fee).
- No admin action is required after market creation; any borrower can trigger the repay step.
- The shortfall accumulates with every `repay` call, making the impact repeatable and compounding.
- The protocol's own test matrix explicitly includes "token charges fee" as an external behavior to handle, confirming this is an expected scenario. [10](#0-9) 

## Recommendation

Measure the actual balance delta received in `repay()` and use that — not the caller-supplied `units` — to increment `withdrawable`. A standard pattern:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
marketState[id].withdrawable += UtilsLib.toUint128(received);
```

Apply the same pattern to `liquidate()`, which has an identical structure at line 717. [11](#0-10) 

Alternatively, document and enforce that fee-on-transfer tokens are not supported as loan tokens, and add a revert guard in `touchMarket` (though this is difficult to enforce on-chain without a balance-delta check at market creation time).
