### Title
`sellerAssetsToUnits` panics on `sellerPrice == 0`, unconditionally reverting `supplyCollateralAndSellWithAssetsTarget` - (`src/periphery/TakeAmountsLib.sol`)

### Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `targetSellerAssets.mulDivUp(WAD, sellerPrice)` where `sellerPrice = offerPrice - settlementFee` for a buy offer. When `offerPrice == settlementFee`, `sellerPrice` is exactly `0`, and `mulDivUp(x, y, 0)` evaluates `(x * y + (0 - 1)) / 0`, triggering an arithmetic underflow panic in Solidity 0.8+. This call is made outside any try/catch in `supplyCollateralAndSellWithAssetsTarget`, so the panic propagates and reverts the entire bundler transaction. Crucially, `Midnight.take` itself does **not** revert for `sellerPrice == 0` — it simply computes `sellerAssets = units.mulDivDown(0, WAD) = 0` — creating a divergence between the core and the periphery.

### Finding Description

**Root cause — `TakeAmountsLib.sellerAssetsToUnits`:** [1](#0-0) 

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : targetSellerAssets.mulDivDown(WAD, sellerPrice);
```

When `offer.buy == true` and `offerPrice == settlementFee`, `sellerPrice = 0`. The `mulDivUp` implementation is: [2](#0-1) 

```solidity
return (x * y + (d - 1)) / d;
```

With `d = 0`, the sub-expression `(d - 1)` underflows (Solidity 0.8 checked arithmetic), producing an arithmetic panic **before** any division. The existing test suite confirms this exact behavior: [3](#0-2) 

**Divergence from `Midnight.take`:**

In the core contract, the same `sellerPrice = 0` path computes: [4](#0-3) 

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD);
```

`units.mulDivDown(0, WAD) = (units * 0) / WAD = 0` — no revert. The core proceeds normally; the periphery panics.

**Unprotected call site in the bundler:** [5](#0-4) 

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(          // ← NOT in try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) returns (...) { // ← only take is guarded
    ...
} catch {}
```

The bundler's own NatSpec acknowledges this: [6](#0-5) 

**Exploit flow:**
1. Attacker observes `settlementFee(id, ttm)` for a target market.
2. Attacker finds tick `T` such that `tickToPrice(T) == settlementFee` (both are multiples of `PRICE_ROUNDING_STEP = 1e12`, so exact equality is achievable).
3. Attacker publishes a buy offer (`offer.buy = true`, `offer.maxAssets > 0`) at tick `T`. Alternatively, attacker publishes at a tick where `tickToPrice(T) > settlementFee` now, knowing the fee will rise to meet the price as TTM decreases (deterministic piecewise-linear schedule).
4. An off-chain aggregator or routing layer includes this offer in a victim's `takes` array.
5. Victim calls `supplyCollateralAndSellWithAssetsTarget`; the loop reaches the poisoned offer, `sellerAssetsToUnits` panics, and the entire transaction reverts.

**Why existing checks do not stop it:**

The comment on `sellerAssetsToUnits` states "Reverts if `offerPrice < settlementFee`" — but the `==` case is not mentioned and is not guarded. The subtraction `offerPrice - settlementFee` does not underflow when they are equal; it silently produces `0`, which then propagates to the division. [7](#0-6) 

### Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee(id, ttm)` at call time reverts unconditionally. This blocks the sell-side bundle flow for borrowers who rely on the bundler to supply collateral and borrow in one transaction. The DoS is repeatable: the attacker can maintain one or more poisoned offers in the order book, and any aggregator-routed transaction that picks them up will fail.

### Likelihood Explanation
**Preconditions:**
- Settlement fees and tick prices are both multiples of `1e12`; exact equality is achievable at specific TTM values.
- The settlement fee schedule is fully deterministic and public, so the attacker can predict the exact block at which a given tick's price equals the fee.
- The attacker only needs to publish one offer (permissionless, no stake required beyond gas).
- The victim must include the offer in their `takes` array — realistic when using an off-chain aggregator that sorts by price and does not pre-simulate `sellerAssetsToUnits`.

**Repeatability:** The attacker can create new offers whenever the fee schedule crosses a tick price, or maintain a portfolio of offers at multiple ticks to cover a range of future fee values. The cost is gas for offer creation only.

### Recommendation
Add a zero-guard in `sellerAssetsToUnits` before the division. When `sellerPrice == 0`, no finite number of units can yield a positive `targetSellerAssets`, so returning `type(uint256).max` is semantically correct (it will be clamped by `takes[i].units` and `consumableUnits` in the `min`):

```solidity
function sellerAssetsToUnits(...) internal view returns (uint256) {
    uint256 offerPrice = TickLib.tickToPrice(offer.tick);
    uint256 settlementFee = IMidnight(midnight).settlementFee(...);
    uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
    if (sellerPrice == 0) return type(uint256).max; // add this guard
    return offer.buy
        ? targetSellerAssets.mulDivUp(WAD, sellerPrice)
        : targetSellerAssets.mulDivDown(WAD, sellerPrice);
}
```

This aligns with `midnight.take`'s behavior (which does not revert for `sellerPrice == 0`) and prevents the panic from propagating through the bundler.
