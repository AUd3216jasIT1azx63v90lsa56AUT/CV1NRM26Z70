### Title
Nonce Rollback on Reverted Submission Allows Stale Authorization Replay - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` increments the authorizer's nonce at line 26 before making an external call to `Midnight.setIsAuthorized` at line 46–47. If `EcrecoverAuthorizer` is not yet authorized in Midnight (`isAuthorized[authorizer][EcrecoverAuthorizer] == false`), the external call reverts, rolling back the entire transaction including the nonce increment. A signed authorization with nonce N therefore remains valid and replayable until the deadline expires, regardless of how many failed submission attempts occurred.

### Finding Description
**Exact code path:**

In `EcrecoverAuthorizer.setIsAuthorized`: [1](#0-0) 

The nonce is post-incremented at line 26 (`nonce[authorization.authorizer]++`). If the `require` passes, the nonce storage slot is written. However, this write is part of the same transaction.

Later, at lines 46–47: [2](#0-1) 

This calls `Midnight.setIsAuthorized` with `msg.sender = EcrecoverAuthorizer` and `onBehalf = authorization.authorizer`. Inside Midnight: [3](#0-2) 

The check `onBehalf == msg.sender` is false (authorizer ≠ EcrecoverAuthorizer), so it falls to `isAuthorized[onBehalf][msg.sender]` = `isAuthorized[authorization.authorizer][EcrecoverAuthorizer]`. If this is `false`, the call reverts with `Unauthorized()`. This revert propagates up and rolls back **all** state changes in the transaction, including the nonce increment at line 26.

**Attacker-controlled inputs:** The attacker holds the signed `(Authorization, Signature)` tuple observed from a prior failed transaction's calldata (publicly visible on-chain).

**Exploit flow:**
1. Authorizer signs `Authorization{authorizer=A, authorized=X, isAuthorized=true, nonce=0, deadline=T}` and submits (or the attacker submits it) while `isAuthorized[A][EcrecoverAuthorizer] == false`.
2. `Midnight.setIsAuthorized` reverts → entire tx reverts → `nonce[A]` remains `0`.
3. Authorizer later calls `midnight.setIsAuthorized(ecrecoverAuthorizer, true, A)` to enable EcrecoverAuthorizer.
4. Attacker replays the original `(Authorization, Signature)` with `nonce=0`. All checks pass: deadline still valid, nonce matches, signature valid, EcrecoverAuthorizer now authorized.
5. `isAuthorized[A][X] = true` is written in Midnight.

**Why existing checks fail:** The deadline check (line 25) and nonce check (line 26) both pass because the nonce was never durably consumed. The signer check at lines 33–36 passes because the authorizer signed it themselves (`signer == authorization.authorizer`). There is no check that detects or blocks a replay of a previously-failed submission. [4](#0-3) 

### Impact Explanation
An attacker who observed a signed authorization from a failed submission can replay it at any future time before the deadline, once the authorizer has enabled EcrecoverAuthorizer in Midnight. This grants `isAuthorized[authorizer][authorized] = true` in Midnight without the authorizer's current consent, enabling the newly-authorized address to act on the authorizer's behalf (withdraw collateral, reduce debt, seize collateral, etc.).

### Likelihood Explanation
**Preconditions:** (1) Authorizer signs an authorization before enabling EcrecoverAuthorizer in Midnight — a natural ordering since EcrecoverAuthorizer is a periphery contract that users may not pre-authorize. (2) The signed authorization is submitted (by anyone, since the function is permissionless) and fails. (3) The deadline has not expired. (4) The authorizer later enables EcrecoverAuthorizer. All four conditions are realistic and require no privileged access. The function is permissionless, so the attacker does not need to be the original submitter. [5](#0-4) 

### Recommendation
Move the nonce increment to occur only after the external call succeeds, or — preferably — use a separate `nonceUsed` mapping that is written before the external call and checked on entry, so a failed submission permanently consumes the nonce. Alternatively, restructure the function so the external call to Midnight is the first state-changing operation, and the nonce is incremented only after it returns successfully. A simpler mitigation is to check upfront that `IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, address(this))` is true before incrementing the nonce, reverting early with a clear error rather than allowing the nonce increment to be silently rolled back.

### Proof of Concept
```solidity
// Foundry unit test
function testStaleAuthorizationReplay() public {
    // Step 1: borrower signs authorization with nonce 0, but does NOT yet
    //         authorize ecrecoverAuthorizer in Midnight.
    Authorization memory auth = Authorization({
        authorizer: borrower,
        authorized: attacker,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 365 days
    });
    Signature memory sig = signAuthorization(auth, borrower);

    // Step 2: Submit while EcrecoverAuthorizer is NOT authorized → reverts.
    vm.expectRevert();
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // Assert nonce was NOT consumed (rolled back).
    assertEq(ecrecoverAuthorizer.nonce(borrower), 0);

    // Step 3: Authorizer enables EcrecoverAuthorizer in Midnight.
    vm.prank(borrower);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

    // Step 4: Attacker replays the stale authorization.
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // Assert: attacker is now authorized — unauthorized grant via stale sig.
    assertTrue(midnight.isAuthorized(borrower, attacker));
    assertEq(ecrecoverAuthorizer.nonce(borrower), 1);
}
```
Expected assertions: the replay at step 4 succeeds and `midnight.isAuthorized(borrower, attacker)` returns `true`, demonstrating that a signed authorization submitted before EcrecoverAuthorizer was enabled in Midnight can be replayed after it is enabled.
