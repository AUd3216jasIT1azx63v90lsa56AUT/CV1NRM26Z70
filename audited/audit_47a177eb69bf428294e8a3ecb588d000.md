Audit Report

## Title
Phantom Collateral via Fee-on-Transfer Token in `supplyCollateral` Enables Undercollateralized Borrowing - (File: src/Midnight.sol)

## Summary
`supplyCollateral` records collateral and sets the bitmap bit before executing the token transfer. Because `SafeTransferLib.safeTransferFrom` only checks for revert and a boolean return value — not the actual amount received — a fee-on-transfer collateral token causes the protocol to record `assets` phantom collateral while receiving 0 tokens. The attacker can then borrow real loan tokens against this phantom collateral, creating bad debt borne by lenders.

## Finding Description
**Root cause — CEI violation with no received-amount guard:**

In `supplyCollateral` (`src/Midnight.sol` lines 532–545), state mutation precedes the transfer:

```solidity
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets); // line 533 — written first
// bitmap set at lines 535–537
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545 — transfer last
```

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34) performs only a revert-check and a boolean-return-value decode. It does not measure the contract's balance before and after the call, so it cannot detect that fewer than `assets` tokens arrived.

`touchMarket` (`src/Midnight.sol` lines 762–773) validates only token ordering, LLTV tier membership, and `maxLif` — no restriction on token type exists, and market creation is permissionless.

`isHealthy` (`src/Midnight.sol` lines 954–955) computes `maxDebt` directly from `_position.collateral[i]`, the stored (phantom) value, not from the contract's actual token balance.

Certora specs summarize `safeTransferFrom` as `NONDET` (`certora/specs/CreatedMarkets.spec` line 33), so formal verification does not model fee-on-transfer behavior and provides no coverage here.

**Exploit flow:**
1. Attacker deploys an ERC20 with a 100% transfer fee (all transferred tokens go to a fee recipient; the recipient receives 0).
2. Attacker calls `touchMarket` with this token as `collateralParams[i].token`. The call succeeds.
3. Attacker calls `supplyCollateral(market, i, assets, self)` with any `assets > 0`.
4. Protocol writes `_position.collateral[i] = assets` and sets the bitmap bit. `safeTransferFrom` succeeds (fee token returns `true`) but delivers 0 tokens to the contract.
5. `isHealthy` computes a non-zero `maxDebt` from the phantom collateral value.
6. Attacker borrows up to `maxDebt` in real loan tokens against collateral the contract does not hold.
7. Loan is never repaid; bad debt is socialized across all lenders in the market.

## Impact Explanation
The protocol records `assets` phantom collateral for a borrower who deposited 0 tokens. `isHealthy` computes a non-zero `maxDebt` from this phantom collateral, allowing the borrower to take a fully uncollateralized loan. When the loan is not repaid, the shortfall becomes bad debt borne by lenders. The core solvency invariant — that recorded collateral is backed by actual token holdings — is broken. This constitutes direct theft of lender funds, a critical in-scope impact.

## Likelihood Explanation
Market creation is permissionless; the attacker deploys the fee-on-transfer token and creates the market themselves with no admin access required. No whitelist or token-type restriction exists in `touchMarket`. The only practical dependency is attracting lenders to the attacker-created market, which is achievable by mimicking a legitimate market configuration (same loan token, valid LLTV, plausible oracle). The attack is trivially repeatable: the attacker controls the token contract and can set fee = 100%. No special privileges, no oracle manipulation, and no reentrancy are required.

## Recommendation
Apply a balance-before/after guard in `supplyCollateral`: record `balanceBefore = IERC20(collateralToken).balanceOf(address(this))` before the transfer, execute the transfer, then compute `received = balanceAfter - balanceBefore` and use `received` (not `assets`) as the amount to credit. Alternatively, move all state mutations after the transfer (strict CEI) and use the measured received amount. Additionally, consider restricting collateral tokens to a whitelist of known-good ERC20s to prevent fee-on-transfer and rebasing token abuse.

## Proof of Concept
1. Deploy `FeeToken` — an ERC20 where `transferFrom` sends 100% of `amount` to a fee sink and returns `true`.
2. Deploy a market via `touchMarket` using `FeeToken` as the sole collateral token with a valid LLTV and a mock oracle returning a non-zero price.
3. Supply loan tokens to the market as a victim lender.
4. Call `supplyCollateral(market, 0, 1e18, attacker)` — `_position.collateral[0]` is set to `1e18`; contract receives 0 `FeeToken`.
5. Call `borrow(...)` up to `maxDebt` computed by `isHealthy` — succeeds because `_position.collateral[0] = 1e18` yields non-zero `maxDebt`.
6. Verify attacker holds real loan tokens; verify contract holds 0 `FeeToken`; verify lender position is now undercollateralized bad debt.
