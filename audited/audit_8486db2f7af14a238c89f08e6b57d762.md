### Title
Referral fee can exceed remaining budget causing DoS revert in `buyWithUnitsTargetAndWithdrawCollateral` - (File: src/periphery/MidnightBundles.sol)

### Summary
`buyWithUnitsTargetAndWithdrawCollateral` pulls `maxBuyerAssets` from the caller, fills takes until `targetUnits` is reached (consuming up to `maxBuyerAssets` in fill cost), then computes `referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct)` on the already-spent balance. No check enforces that `filledBuyerAssets + referralFeeAssets <= maxBuyerAssets`, so when the fill consumes a large fraction of `maxBuyerAssets` and `referralFeePct > 0`, the bundler's remaining balance is insufficient to pay the referral fee. The function reverts at the `safeTransfer` to `referralFeeRecipient` (line 103) due to insufficient token balance, and if the bundler holds any pre-existing token balance the Solidity 0.8 subtraction at line 104 underflows instead.

### Finding Description

**Exact code path** — `src/periphery/MidnightBundles.sol` lines 66–104:

```
pullToken(loanToken, msg.sender, maxBuyerAssets, loanTokenPermit);   // L66
forceApproveMax(loanToken, MIDNIGHT);                                 // L67
// ... fill loop accumulates filledBuyerAssets (L71-86) ...
// bundler balance is now: maxBuyerAssets - filledBuyerAssets
uint256 referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct); // L102
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets); // L103 ← REVERT
SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - filledBuyerAssets - referralFeeAssets); // L104 ← UNDERFLOW
``` [1](#0-0) 

**Root cause** — The fill loop has no budget ceiling that accounts for the referral fee. The loop runs until `filledUnits == targetUnits` with no constraint on `filledBuyerAssets` relative to `maxBuyerAssets * (WAD - referralFeePct) / WAD`. After the loop, the bundler holds `maxBuyerAssets - filledBuyerAssets` tokens. The referral fee formula `filledBuyerAssets * referralFeePct / (WAD - referralFeePct)` can exceed that remainder whenever:

```
filledBuyerAssets > maxBuyerAssets * (WAD - referralFeePct) / WAD
``` [2](#0-1) 

**Contrast with the sibling function** — `buyWithAssetsTargetAndWithdrawCollateral` correctly pre-deducts the referral fee before filling:

```solidity
uint256 referralFeeAssets = targetBuyerAssets.mulDivDown(referralFeePct, WAD);
uint256 targetFilledBuyerAssets = targetBuyerAssets - referralFeeAssets;
``` [3](#0-2) 

`buyWithUnitsTargetAndWithdrawCollateral` has no equivalent pre-deduction.

**Attacker-controlled inputs** — The caller (an unprivileged taker) controls `maxBuyerAssets`, `referralFeePct`, and the `takes` array. The actual `filledBuyerAssets` is determined by offer prices and settlement fees at execution time. A caller who sets `maxBuyerAssets` to the expected fill cost (without headroom for the referral fee) triggers the revert.

**Concrete numeric example**:
- `maxBuyerAssets = 3e18`
- `referralFeePct = 0.5e18` (50%)
- `filledBuyerAssets = 2e18` (takes consumed 2/3 of the budget — fully valid)
- `referralFeeAssets = 2e18 * 0.5e18 / 0.5e18 = 2e18`
- Bundler balance after takes: `3e18 - 2e18 = 1e18`
- Line 103 attempts `safeTransfer(..., 2e18)` with only `1e18` available → **revert**

**Why existing checks are insufficient**:
- `require(referralFeePct < WAD)` (line 61) only prevents `referralFeePct = WAD`; any value in `[1, WAD-1]` is accepted. [4](#0-3) 
- `require(filledUnits == targetUnits)` (line 88) only checks unit completeness, not budget adequacy. [5](#0-4) 
- There is no `require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets)` anywhere.

The NatSpec comment on line 46 states "The msg.sender will pay at most `maxBuyerAssets`" — this invariant is violated because the function reverts rather than capping the fill. [6](#0-5) 

### Impact Explanation
Any call to `buyWithUnitsTargetAndWithdrawCollateral` with `referralFeePct > 0` where `filledBuyerAssets > maxBuyerAssets * (WAD - referralFeePct) / WAD` reverts unconditionally. The caller's transaction fails (gas lost, no fill executed, no funds lost due to revert). The function is effectively unusable with referral fees unless the caller sets `maxBuyerAssets` to `expectedFill * WAD / (WAD - referralFeePct)` — a non-obvious requirement not enforced or clearly documented. This is a concrete DoS of `buyWithUnitsTargetAndWithdrawCollateral` for any caller using referral fees with a tight `maxBuyerAssets`.

### Likelihood Explanation
Preconditions: `referralFeePct > 0` (any non-zero referral fee) and `filledBuyerAssets` large relative to `maxBuyerAssets`. This is easily triggered: a caller who sets `maxBuyerAssets` to the expected fill cost (a natural choice given the parameter name and the "pay at most" NatSpec) will hit this whenever `referralFeePct > 0`. The condition is repeatable and deterministic given fixed offer prices and settlement fees. No privileged access or special market state is required.

### Recommendation
Mirror the pattern from `buyWithAssetsTargetAndWithdrawCollateral`: pre-compute the referral fee budget before the fill loop and cap fills to `maxBuyerAssets - referralFeeReserve`:

```solidity
// Reserve referral fee headroom before filling
uint256 referralFeeReserve = maxBuyerAssets.mulDivDown(referralFeePct, WAD);
uint256 maxFilledBuyerAssets = maxBuyerAssets - referralFeeReserve;

// ... fill loop, but stop when filledBuyerAssets would exceed maxFilledBuyerAssets ...

uint256 referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct);
require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets, ExceedsMaxBuyerAssets());
```

Alternatively, add a post-fill guard: `require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets, ExceedsMaxBuyerAssets())` immediately after line 102.
