Audit Report

## Title
Division-by-Zero in `sellerAssetsToUnits` When `tickToPrice(tick) == settlementFee` Causes DoS of `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/TakeAmountsLib.sol)

## Summary

In `TakeAmountsLib.sellerAssetsToUnits`, when `offer.buy == true` and `tickToPrice(offer.tick) == settlementFee`, `sellerPrice` is computed as `0`. The subsequent `mulDivUp(targetSellerAssets, WAD, 0)` call underflows at `(d - 1)` with `d = 0` under Solidity 0.8 checked arithmetic, causing an unconditional revert. Because `Midnight.take()` does not revert in this same state (it computes `sellerAssets = units * 0 / WAD = 0` via `mulDivDown`), the NatDoc invariant "midnight reverts too" is false for the equality case. Since `supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside its `try/catch` block, any `takes[]` array containing such an offer causes the entire bundler call to revert unconditionally.

## Finding Description

**Root cause — `sellerAssetsToUnits` (`src/periphery/TakeAmountsLib.sol` lines 44–46):**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...
```

When `offer.buy == true` and `offerPrice == settlementFee`, `sellerPrice = 0`. [1](#0-0) 

**`mulDivUp` with `d = 0` reverts (`src/libraries/UtilsLib.sol` lines 34–36):**

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```

With `d = 0`, the expression `(d - 1)` underflows under Solidity 0.8 checked arithmetic before any division occurs, causing an unconditional revert. [2](#0-1) 

**Why `Midnight.take()` does NOT revert in the same state:**

`Midnight.take()` computes `sellerAssets = units.mulDivDown(sellerPrice, WAD)`. With `sellerPrice = 0`, this is `(units * 0) / WAD = 0` — `mulDivDown` divides by `WAD` (1e18), not by `sellerPrice`, so no division-by-zero occurs. The NatDoc comment at line 34 of `TakeAmountsLib.sol` states "Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)" — this is incorrect for the equality case: `midnight` does not revert, but `sellerAssetsToUnits` does. [3](#0-2) 

**Exploit path — `supplyCollateralAndSellWithAssetsTarget` (`src/periphery/MidnightBundles.sol` lines 285–291):**

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(   // ← OUTSIDE try/catch; reverts here
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) returns (...) { ... } catch {}  // never reached
```

The `sellerAssetsToUnits` call is not wrapped in the `try/catch`. Its revert propagates unconditionally to the caller. [4](#0-3) 

**Structural reachability of `offerPrice == settlementFee`:**

`tickToPrice` rounds all tick prices to multiples of `PRICE_ROUNDING_STEP = 1e12`. [5](#0-4) [6](#0-5) 

Settlement fees are also public view functions. An attacker reads the current settlement fee, calls `TickLib.priceToTick(settlementFee, tickSpacing)` to find the matching tick, and posts a buy offer at that tick. No privileged access is required.

## Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer at the settlement-fee price point in its `takes[]` array reverts entirely. An aggregator or UI that enumerates open buy offers and routes through the malicious offer will have the victim's entire bundler transaction reverted, blocking the sell-via-periphery path for affected takers for as long as the offer exists and the settlement fee remains at a level with a matching tick. This is a targeted, low-cost, repeatable DoS of a core periphery function.

## Likelihood Explanation

- Settlement fees are public and readable on-chain; the attacker trivially computes the matching tick using `TickLib.priceToTick`.
- Both tick prices and settlement fees are multiples of `1e12`, making exact equality structurally guaranteed to be reachable for any non-zero settlement fee within the tick range.
- The attacker bears only the gas cost of posting one buy offer; no capital is at risk.
- The DoS persists until the offer is cancelled (by the attacker, who has no incentive to do so) or the settlement fee changes to a value with no matching tick.
- The attack is repeatable: if the fee changes, the attacker posts a new offer at the new matching tick.

## Recommendation

Two complementary fixes:

1. **In `sellerAssetsToUnits`**: Add an explicit guard before calling `mulDivUp`. When `sellerPrice == 0`, `Midnight.take()` will always yield `sellerAssets = 0` regardless of units, so no finite number of units can satisfy a nonzero `targetSellerAssets`. Return `type(uint256).max` (or revert with a descriptive error) when `sellerPrice == 0 && targetSellerAssets > 0`, and `0` when `targetSellerAssets == 0`.

2. **In `supplyCollateralAndSellWithAssetsTarget`**: Wrap the `sellerAssetsToUnits` (and `consumableUnits`) calls in a `try/catch` or use a low-level call, consistent with how `IMidnight.take()` is already handled, so that a single malicious offer in the `takes[]` array causes that offer to be skipped rather than reverting the entire transaction.

## Proof of Concept

1. Deploy Midnight with a non-zero settlement fee `F` that is a multiple of `1e12`.
2. Call `TickLib.priceToTick(F, tickSpacing)` to find tick `T` such that `tickToPrice(T) == F`.
3. Post a buy offer at tick `T` (attacker-controlled, no capital at risk beyond gas).
4. Construct a `takes[]` array containing this offer and call `supplyCollateralAndSellWithAssetsTarget`.
5. Observe that the call reverts unconditionally at the `sellerAssetsToUnits` call due to underflow in `mulDivUp(targetSellerAssets, WAD, 0)`.
6. Confirm that calling `Midnight.take()` directly with the same offer and any nonzero `units` succeeds (returning `sellerAssets = 0`), proving the divergence between `sellerAssetsToUnits` and `take()`.
