Audit Report

## Title
Authorized Midnight Delegate Can Consume Authorizer's EcrecoverAuthorizer Nonce, Permanently Invalidating Pre-Signed Authorization Messages - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` increments `nonce[authorization.authorizer]` at line 26 before the signer is recovered or validated. Because the signer check at lines 33–36 explicitly accepts any address for which `IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)` is true, any Midnight-level delegate of Alice can submit a self-signed `Authorization` struct with `authorizer=Alice` and Alice's current nonce, permanently advancing it and invalidating all of Alice's pre-signed messages carrying that nonce. The attack is repeatable with no bound.

## Finding Description
**Root cause — `src/periphery/EcrecoverAuthorizer.sol`, lines 24–36:**

```solidity
// Line 26: nonce permanently advanced before signer identity is known
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

// Lines 31–36: signer check evaluated after nonce is already consumed
address signer = ecrecover(digest, signature.v, signature.r, signature.s);
require(signer != address(0), InvalidSignature());
require(
    signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

The post-increment `nonce[authorization.authorizer]++` at line 26 permanently advances the nonce for any transaction that does not revert. The signer check at lines 33–36 explicitly permits any address `s` for which `isAuthorized[authorization.authorizer][s]` is `true` in the core `Midnight` contract (`Midnight.sol` line 192). Because the nonce is incremented before the signer is identified, and because the signer check accepts authorized delegates, a delegate can consume the authorizer's nonce without the authorizer's consent.

**Exploit flow:**
1. Alice calls `midnight.setIsAuthorized(Bob, true, Alice)` — `isAuthorized[Alice][Bob] = true` (`Midnight.sol` lines 731–734).
2. Alice off-chain pre-signs `Authorization(authorizer=Alice, authorized=Dave, isAuthorized=true, nonce=0, deadline=T)`.
3. Bob constructs `Authorization(authorizer=Alice, authorized=Charlie, isAuthorized=true, nonce=0, deadline=T')` and signs it with his own key.
4. Bob calls `ecrecoverAuthorizer.setIsAuthorized(auth_bob, sig_bob)`.
5. Line 26: `nonce[Alice]` is 0; check passes; nonce advances to 1.
6. Lines 33–36: `signer == Bob`; `Midnight.isAuthorized(Alice, Bob) == true`; check passes; transaction succeeds.
7. Alice's pre-signed message `(authorizer=Alice, nonce=0)` now reverts with `InvalidNonce()` on any future submission.
8. Bob immediately repeats with nonce=1, then 2, etc.

**Why existing checks fail:**
- The nonce check at line 26 only verifies the value matches and then increments — it does not restrict *who* may consume it.
- The signer check at lines 33–36 is evaluated *after* the nonce is already consumed; even if reordered, it still explicitly permits authorized delegates.
- `Midnight.sol` lines 101–110 acknowledge that "other contracts might re-use Midnight's authorization mapping too (e.g ratifiers and authorizers)" and that "authorized accounts can authorize other accounts on behalf of the user," but do not document nonce-consumption in `EcrecoverAuthorizer` as an accepted risk. The warning targets smart-contract authorizations and does not cover the adversarial nonce-griefing scenario described here.

## Impact Explanation
Any address Alice has authorized in Midnight can consume Alice's current `EcrecoverAuthorizer` nonce by signing and submitting an `Authorization` struct with `authorizer=Alice` and the current nonce value. This permanently invalidates all of Alice's pre-signed authorization messages carrying that nonce. Alice must re-sign with the new nonce, but the attacker can immediately repeat the attack, creating a sustained, repeatable DoS on Alice's ability to use gasless/pre-signed delegations via `EcrecoverAuthorizer`. The impact is a permanent, repeatable freeze of a core protocol feature for any user with at least one Midnight delegate.

## Likelihood Explanation
Preconditions: Alice must have authorized Bob in Midnight — a normal, expected protocol operation requiring no special privileges. Bob must act adversarially or be compromised. No funds are at risk directly, but the DoS is persistent and requires active remediation (revoking Bob's Midnight authorization). The attack is repeatable: after each nonce consumption, Bob can immediately consume the next nonce. No admin keys, leaked credentials, or special on-chain privileges are required.

## Recommendation
Restrict nonce consumption in `EcrecoverAuthorizer.setIsAuthorized` to the authorizer themselves. Move the signer recovery and validation *before* the nonce increment, and only permit `signer == authorization.authorizer` (not delegates) to consume the nonce. Alternatively, maintain a separate nonce namespace for `EcrecoverAuthorizer` that is not consumable by delegates, or add an explicit check that `msg.sender == authorization.authorizer` before advancing the nonce. If delegate-driven nonce consumption is intentional, it must be explicitly documented as an accepted risk in both `EcrecoverAuthorizer` and the `Midnight.sol` NatSpec.

## Proof of Concept
```solidity
// 1. Alice authorizes Bob in Midnight
midnight.setIsAuthorized(bob, true, alice); // called by alice

// 2. Alice pre-signs Authorization(authorizer=alice, authorized=dave, isAuthorized=true, nonce=0, deadline=T)
// (off-chain, stored for later submission)

// 3. Bob constructs and signs a different Authorization with authorizer=alice, nonce=0
Authorization memory authBob = Authorization({
    authorizer: alice,
    authorized: charlie,
    isAuthorized: true,
    nonce: 0,
    deadline: block.timestamp + 1 hours
});
// Bob signs authBob with his own key -> sigBob

// 4. Bob submits
ecrecoverAuthorizer.setIsAuthorized(authBob, sigBob); // succeeds: nonce[alice] -> 1

// 5. Alice's pre-signed message now reverts
ecrecoverAuthorizer.setIsAuthorized(alicePreSigned, aliceSig); // reverts: InvalidNonce()

// 6. Bob repeats with nonce=1, 2, ... indefinitely
```
A Foundry fuzz test can confirm by fuzzing `authorization.authorized` and `authorization.deadline` with `authorization.authorizer = alice` and `authorization.nonce = nonce[alice]`, signed by Bob, verifying that each call succeeds and Alice's corresponding pre-signed message reverts.
