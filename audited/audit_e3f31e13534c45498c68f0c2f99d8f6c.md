### Title
Transitive Delegation via EcrecoverAuthorizer Allows Any Midnight Delegate to Grant Arbitrary Authorizations - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts a signature from any address that `isAuthorized[authorizer][signer]` is true in Midnight, not just the authorizer themselves. Because Midnight's `isAuthorized` is a general-purpose delegation mapping used for all protocol actions (withdraw, repay, collateral management, etc.), any delegate B of A can sign an `Authorization` struct naming an arbitrary third party C and submit it to `EcrecoverAuthorizer`, causing `isAuthorized[A][C]` to be set to `true` without A ever signing anything related to C.

### Finding Description
The vulnerable check is at `src/periphery/EcrecoverAuthorizer.sol` lines 33–36:

```solidity
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
``` [1](#0-0) 

`IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)` queries the same `isAuthorized` mapping that `EcrecoverAuthorizer` itself writes to. Any address B that A has authorized for *any* Midnight operation satisfies this check.

**Exact exploit path:**

1. A calls `midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, A)` — required so `EcrecoverAuthorizer` can later call `midnight.setIsAuthorized` on A's behalf. [2](#0-1) 
2. A calls `midnight.setIsAuthorized(B, true, A)` — authorizes B for any normal protocol operation (e.g., repay, withdraw). Now `isAuthorized[A][B] == true`. [2](#0-1) 
3. B signs `Authorization{authorizer: A, authorized: C, isAuthorized: true, nonce: 0, deadline: future}` with B's private key.
4. Anyone calls `ecrecoverAuthorizer.setIsAuthorized(auth, sig_by_B)`.
5. `ecrecover` returns B. The check `IMidnight(MIDNIGHT).isAuthorized(A, B)` → `true` → passes. [3](#0-2) 
6. `IMidnight(MIDNIGHT).setIsAuthorized(C, true, A)` is called. Inside `Midnight.setIsAuthorized`, `msg.sender` is `ecrecoverAuthorizer`, and `isAuthorized[A][ecrecoverAuthorizer] == true`, so the call succeeds. [4](#0-3) 
7. `isAuthorized[A][C] == true`. C can now act on behalf of A for all Midnight operations.

The root cause is that the signer-validity check conflates "authorized to act in Midnight" with "authorized to sign off-chain authorization structs on behalf of the authorizer." These are distinct trust levels: A may authorize B to repay a loan without intending to give B the power to delegate A's full account to an arbitrary C.

### Impact Explanation
C gains full Midnight authorization on behalf of A. C can call `midnight.withdraw(market, amount, A, C)`, `midnight.withdrawCollateral(market, index, amount, A, C)`, and any other `onBehalf`-gated function, draining A's credit and collateral. This is unauthorized seizure/withdrawal of A's assets.

### Likelihood Explanation
Preconditions are realistic: A must have authorized both `ecrecoverAuthorizer` and B. Authorizing `ecrecoverAuthorizer` is the standard setup for any user who wants to use signature-based authorization. Authorizing B is a normal operational pattern (e.g., a keeper, a UI wallet, a co-signer). Once both are true, B can execute the attack unilaterally with no further cooperation from A. The attack is repeatable (nonce increments, but B can sign new structs for new targets) and requires no special on-chain state beyond the two `setIsAuthorized` calls A already made.

### Recommendation
Restrict the signer check in `EcrecoverAuthorizer.setIsAuthorized` to only accept the authorizer themselves. Remove the `isAuthorized` delegation fallback entirely from this function:

```solidity
require(signer == authorization.authorizer, Unauthorized());
```

If delegated signing is intentionally desired, it should require a separate, explicit opt-in mechanism (e.g., a dedicated `signingDelegate` mapping distinct from the general `isAuthorized` mapping) so that granting operational access does not implicitly grant the power to issue new authorizations.

### Proof of Concept
```solidity
function testDelegateCanGrantArbitraryAuthorization() public {
    // Setup: A authorizes ecrecoverAuthorizer and B
    vm.prank(A);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, A);
    vm.prank(A);
    midnight.setIsAuthorized(B, true, A);

    // B signs Authorization{authorizer: A, authorized: C, isAuthorized: true}
    Authorization memory auth = Authorization({
        authorizer: A,
        authorized: C,
        isAuthorized: true,
        nonce: ecrecoverAuthorizer.nonce(A), // 0
        deadline: block.timestamp + 1 days
    });
    // B signs with B's private key
    Signature memory sig = signAuthorization(auth, B);

    // Anyone submits
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // Assert: C is now authorized on behalf of A — A never signed anything about C
    assertTrue(midnight.isAuthorized(A, C));
    // C can now withdraw A's assets
}
```

Expected assertion: `midnight.isAuthorized(A, C) == true`. The test passes, confirming the vulnerability.
