### Title
Buyer callback can call `withdraw()` before token transfer in `take()`, draining pre-existing `withdrawable` balance at a discount - (`src/Midnight.sol`)

### Summary
In `take()`, the buyer's credit is increased and `onBuy` is called before the `safeTransferFrom` that settles the payment. Because `withdraw()` has no reentrancy guard and no check that it is being called during a `take()` callback, a malicious `IBuyCallback` authorized by the maker can call `withdraw()` during `onBuy`, consuming pre-existing `withdrawable` balance (backed by other borrowers' repayments) and using those tokens to fund the take's own payment. The attacker receives `units` tokens while paying only `buyerAssets < units`, netting `units - buyerAssets` tokens at the expense of other lenders' withdrawable pool.

### Finding Description

**Exact code path:**

`take()` with `offer.buy == true`:

1. Line 410: `buyerPos.credit += buyerCreditIncrease` — buyer's credit is written to storage **before** any external call.
2. Lines 420–422: `buyerCallback = offer.callback` (maker-controlled); `payer = buyerCallback` since callback is non-zero.
3. Lines 445–453: `IBuyCallback(buyerCallback).onBuy(...)` is called — **external call before token transfer**.
4. Lines 455–456: `safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, payer, receiver, sellerAssets)` — token transfer happens **after** the callback returns.

`withdraw()` (lines 481–500):
- Checks `isAuthorized[onBehalf][msg.sender]` (line 482) — satisfied if maker pre-authorized the callback.
- Decrements `_position.credit -= units` (line 493) — succeeds because credit was just increased.
- Decrements `_marketState.withdrawable -= units` (line 494) — succeeds if pre-existing withdrawable ≥ units.
- Transfers `units` tokens to receiver (line 499).
- **No guard against being called during `onBuy`.**

`withdrawable` is only increased by `repay()` (line 509) and `liquidate()` (line 675). It is **never increased by `take()`**. The `LIQUIDATION_LOCK_SLOT` mechanism (line 444) only prevents liquidation of the seller during callbacks; it does not protect `withdraw()`.

**Attacker-controlled inputs:**
- `offer.callback` = attacker-deployed `MaliciousCallback` implementing `IBuyCallback`
- `isAuthorized[maker][MaliciousCallback] = true` (set by maker via `setIsAuthorized`)
- Offer ratified via `SetterRatifier.setIsRootRatified(maker, root, true)` by maker or authorized operator

**Exploit flow:**

Precondition: market has pre-existing `withdrawable` balance W ≥ U from prior borrower repayments.

1. Maker deploys `MaliciousCallback`, authorizes it: `isAuthorized[maker][MaliciousCallback] = true`.
2. Maker creates buy offer with `offer.callback = MaliciousCallback`, ratifies root via `SetterRatifier`.
3. Taker calls `take(offer, ratifierData, U, taker, receiver, address(0), hex"")`.
4. `take()` sets `buyerPos.credit += U` (line 410).
5. `take()` calls `MaliciousCallback.onBuy(...)`.
6. Inside `onBuy`, `MaliciousCallback` calls `Midnight.withdraw(market, U, maker, attacker)`:
   - Auth: `isAuthorized[maker][MaliciousCallback]` = true ✓
   - Credit: `_position.credit = U ≥ U` ✓
   - Withdrawable: `_marketState.withdrawable = W ≥ U` ✓
   - Result: `_position.credit = 0`, `withdrawable = W - U`, attacker receives U tokens.
7. `MaliciousCallback` approves Midnight for `buyerAssets` tokens (funded by the U tokens just received; since `buyerPrice < WAD`, `buyerAssets < U`).
8. `onBuy` returns `CALLBACK_SUCCESS`.
9. `take()` calls `safeTransferFrom(loanToken, MaliciousCallback, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, MaliciousCallback, receiver, sellerAssets)` — both succeed.

**Why checks fail:** There is no reentrancy guard on `withdraw()`, no transient lock protecting `withdrawable` during `take()`, and no post-callback invariant check that `withdrawable` was not consumed.

### Impact Explanation

The attacker (maker) receives `U` tokens immediately from the pre-existing `withdrawable` pool while paying only `buyerAssets = U × buyerPrice / WAD < U` tokens. The net profit per attack is `U - buyerAssets = U × (1 - buyerPrice/WAD)`. Other lenders who had repaid tokens available for withdrawal find `withdrawable` reduced by `U`, temporarily (until the taker repays) or permanently (if the taker defaults). The contract's token balance decreases by `U - (buyerAssets - sellerAssets)` net, which is not reflected in any accounting correction. This directly violates the invariant that withdrawable assets must not be accessible before the take settles.

### Likelihood Explanation

Preconditions are realistic in any active market: (a) pre-existing `withdrawable` balance exists whenever any borrower has repaid; (b) the maker can freely set `offer.callback` and authorize it; (c) the `SetterRatifier` requires only that the maker (or their authorized operator) call `setIsRootRatified`. The attack is repeatable as long as `withdrawable` is non-zero and is profitable whenever `buyerPrice < WAD` (the normal lending case). No privileged role is required beyond the maker's own authorization.

### Recommendation

Add a transient reentrancy guard to `withdraw()` that reverts if called during a `take()` callback. Concretely, use a transient storage slot (analogous to `LIQUIDATION_LOCK_SLOT`) set to `true` at the start of `take()` and cleared at the end, and add `require(!takeLocked(), WithdrawDuringTake())` at the top of `withdraw()`. Alternatively, move all position state writes in `take()` to after the token transfer (checks-effects-interactions), so that `buyerPos.credit` is not yet increased when `onBuy` fires.

### Proof of Concept

```solidity
// Foundry unit test plan
contract MaliciousCallback is IBuyCallback {
    Midnight midnight;
    Market market;
    address attacker;

    constructor(address _midnight, Market memory _market, address _attacker) {
        midnight = Midnight(_midnight);
        market = _market;
        attacker = _attacker;
    }

    function onBuy(
        bytes32, Market memory, uint256 buyerAssets, uint256 units,
        uint256, address buyer, bytes memory
    ) external returns (bytes32) {
        // Reenter withdraw() before safeTransferFrom in take() fires
        midnight.withdraw(market, units, buyer, attacker);
        // Approve buyerAssets from the withdrawn tokens (units > buyerAssets when price < WAD)
        ERC20(market.loanToken).approve(msg.sender, buyerAssets);
        return CALLBACK_SUCCESS;
    }
}

function testReentrantWithdrawDuringOnBuy() public {
    uint256 U = 1e18;
    // Setup: lender has credit U, borrower repays U → withdrawable = U
    addCredit(lender, U);
    deal(address(loanToken), borrower, U);
    vm.prank(borrower); midnight.repay(market, U, borrower, address(0), hex"");
    assertEq(midnight.withdrawable(id), U);

    // Attacker (maker2) creates buy offer with malicious callback
    MaliciousCallback cb = new MaliciousCallback(address(midnight), market, attacker);
    vm.prank(maker2); midnight.setIsAuthorized(address(cb), true, maker2);
    lenderOffer.maker = maker2;
    lenderOffer.callback = address(cb);
    // ratify offer via SetterRatifier...

    uint256 attackerBalBefore = loanToken.balanceOf(attacker);
    // Taker (borrower2) takes the offer
    collateralize(market, borrower2, U);
    vm.prank(borrower2); midnight.take(lenderOffer, ratifierData, U, borrower2, borrower2, address(0), hex"");

    // Assertions:
    // 1. Attacker received U tokens but paid only buyerAssets < U
    assertGt(loanToken.balanceOf(attacker) - attackerBalBefore, 0);
    // 2. withdrawable was drained (other lenders cannot withdraw)
    assertLt(midnight.withdrawable(id), U);
    // 3. maker2's credit is 0 (increased then immediately withdrawn)
    assertEq(midnight.creditOf(id, maker2), 0);
}
```
