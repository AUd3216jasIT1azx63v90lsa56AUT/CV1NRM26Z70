Audit Report

## Title
`onLiquidate` callback enables `supplyCollateral` after bad-debt accounting is finalized, causing lenders to be over-slashed - (File: src/Midnight.sol)

## Summary
In `liquidate()`, bad-debt is fully computed from a collateral bitmap snapshot and immediately committed to `lossFactor`/`totalUnits` before the `onLiquidate` callback fires. A callback contract authorized for the borrower can call `supplyCollateral` during this window, injecting collateral that was never counted in the bad-debt formula. Lenders are permanently slashed for a shortfall larger than the true post-liquidation shortfall, while the borrower retains the newly supplied collateral.

## Finding Description
**Exact code path in `src/Midnight.sol`:**

1. **Line 606** — collateral bitmap is snapshotted into a local variable: `uint128 _collateralBitmap = _position.collateralBitmap;`

2. **Lines 607–618** — `badDebt` is computed by iterating only over indices present in the snapshot. Any collateral added after this point is invisible to the formula.

3. **Lines 626–641** — `badDebt` is immediately and irrevocably committed: `_position.debt -= uint128(badDebt)`, `_marketState.lossFactor` is updated, `_marketState.totalUnits` is reduced. Lenders are slashed here, before any external call.

4. **Lines 698–715** — `onLiquidate` callback fires. All bad-debt accounting is already final at this point.

5. Inside the callback, `supplyCollateral` is called for `borrower`. The authorization check at **line 527** is `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. If `callback == borrower` (borrower is a smart contract implementing `ILiquidateCallback`), `msg.sender == borrower` satisfies the check with no additional setup. Alternatively, the borrower pre-authorizes the callback contract via `setIsAuthorized`.

6. `supplyCollateral` writes `_position.collateral[newIndex]` and sets the corresponding bit in `_position.collateralBitmap` (**lines 533–537**). Neither write is visible to the already-completed bad-debt loop.

7. **Line 717** — `safeTransferFrom` pulls loan tokens (0 if `repaidUnits == 0`). No re-check of collateral or bad debt occurs.

**Why existing checks do not stop it:**

- `liquidationLocked` is a transient-storage flag set only inside `take()` at **line 444** via `UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`. It is never set inside `liquidate()`. It does not block `supplyCollateral` from being called during the `onLiquidate` callback.
- There is no reentrancy guard on `liquidate()` or `supplyCollateral()`.
- The Certora `OnlyAuthorizedCanChange.spec` invariant at **line 73** explicitly acknowledges: *"Assumes no reentrancy: callbacks and token transfers are not modeled as re-entering Midnight, so re-entrant collateral changes are not covered."* This is a known gap in the formal verification, not a design guarantee of safety.

## Impact Explanation
`badDebt` realized = `D − Σ(collateral_i · price_i / maxLif_i)` (pre-callback collateral only).
True shortfall after callback = `D − Σ(collateral_i · price_i / maxLif_i) − A · price_new / maxLif_new`.
Over-slashing per attack = `A · price_new / maxLif_new` units, permanently subtracted from `totalUnits` and added to `lossFactor`. Every lender's effective credit is permanently reduced by this excess amount when they call `updatePosition`. The attacker (borrower) retains the newly supplied collateral in their position and can withdraw it (partially or fully) once the position is healthy, recovering most or all of the supplied tokens at near-zero net cost. This constitutes direct, permanent theft of lender credit.

## Likelihood Explanation
**Required preconditions:**
1. Borrower is a smart contract (or has pre-authorized the callback) — realistic for any protocol-integrated borrower or DeFi vault.
2. Position has genuine bad debt (oracle price drop) — the normal trigger for bad-debt realization.
3. Callback contract holds collateral tokens to supply — sourceable from a flash loan within the same callback, since `supplyCollateral` pulls from `msg.sender` and no repayment is required within `liquidate()`.

Self-liquidation (`borrower == liquidator`) requires no third-party cooperation. The attack is repeatable on any market with bad debt and a callback-capable borrower. The attacker controls `seizedAssets = 0`, `repaidUnits = 0`, `callback`, and the collateral amount, requiring no privileged access.

## Recommendation
Set the `liquidationLocked` transient flag for the borrower at the start of `liquidate()` (analogous to how `take()` sets it for the seller at line 444), and clear it at the end. This prevents any re-entrant call to `supplyCollateral`, `withdrawCollateral`, or other position-mutating functions during the `onLiquidate` callback. Alternatively, snapshot and re-validate the full collateral state after the callback before finalizing bad-debt accounting, though the lock approach is simpler and consistent with the existing pattern.

## Proof of Concept
**Minimal Foundry test outline:**

1. Deploy a market with two collateral types (indices 0 and 1).
2. Borrower (a smart contract) supplies collateral[0] and borrows. Drop oracle price so the position has bad debt.
3. Borrower-contract implements `ILiquidateCallback.onLiquidate`: inside the callback, flash-loan collateral[1] tokens, approve Midnight, call `midnight.supplyCollateral(market, 1, largeAmount, borrower)`, return `CALLBACK_SUCCESS`.
4. Call `midnight.liquidate(market, 0, 0, 0, borrower, false, borrower, borrower, "")` (self-liquidation, pure bad-debt path).
5. Assert: `marketState.lossFactor` after the call is strictly greater than it would be without the callback-supplied collateral (i.e., greater than the value computed from collateral[0] alone).
6. Assert: `position.collateral[1]` equals `largeAmount` (collateral was accepted).
7. Call `midnight.updatePosition` for a lender and assert their credit decreased by more than the true shortfall (collateral[0] value / maxLif[0] subtracted from debt).
