Audit Report

## Title
`sellerAssetsToUnits` division-by-zero when `offerPrice == settlementFee` causes unconditional revert in `supplyCollateralAndSellWithAssetsTarget` - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and calls `mulDivUp(targetSellerAssets, WAD, sellerPrice)`. When `offerPrice == settlementFee` post-maturity, `sellerPrice = 0` and `mulDivUp` reverts via uint256 underflow on `d - 1`. Because `sellerAssetsToUnits` is invoked at line 286 of `MidnightBundles.sol` **outside** the `try/catch` wrapping `take()`, any `takes` array containing such an offer causes the entire `supplyCollateralAndSellWithAssetsTarget` transaction to revert unconditionally, while Midnight's core `take()` would handle the case correctly.

## Finding Description

**Root cause — `TakeAmountsLib.sellerAssetsToUnits`, lines 44–46:**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

When `offer.buy == true` and `offerPrice == settlementFee`, `sellerPrice = 0`. The call `targetSellerAssets.mulDivUp(WAD, 0)` executes `(x * y + (0 - 1)) / 0`, where `0 - 1` underflows in checked Solidity 0.8+ arithmetic, causing an arithmetic revert. [1](#0-0) 

This is confirmed by `testMulDivUpDivisionByZero`, which documents that `mulDivUp(x, y, 0)` reverts with `stdError.arithmeticError` due to the `d - 1` underflow: [2](#0-1) 

`UtilsLib.mulDivUp` implementation: [3](#0-2) 

**Why the condition is reachable:** `PRICE_ROUNDING_STEP = 1e12` means `tickToPrice(tick)` returns multiples of `1e12`. `CBP = 1e12` and `MAX_SETTLEMENT_FEE_0_DAYS = 0.000014e18 = 14e12`, so `settlementFee(id, 0)` takes values in `{0, 1e12, 2e12, ..., 14e12}`. Since `tickToPrice` also produces multiples of `1e12`, there exist ticks where `tickToPrice(tick) == settlementFee(id, 0)` for any non-zero fee setting. [4](#0-3) [5](#0-4) 

**Why the periphery reverts unconditionally:** `sellerAssetsToUnits` is called at line 286, **outside** the `try/catch` block that starts at line 292. The loop requires `takes[i].offer.buy == true` (line 283), so every iteration through this loop for a buy offer calls `sellerAssetsToUnits` before the protected `take()` call: [6](#0-5) 

**NatSpec divergence:** `sellerAssetsToUnits` documents "Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)." The `offerPrice == settlementFee` case is not covered, and Midnight's core does NOT revert in that case — only the periphery does. [7](#0-6) 

**Contrast with `buyerAssetsToUnits`:** When `offerPrice == settlementFee`, `buyerAssetsToUnits` computes `buyerPrice = 0 + settlementFee = settlementFee`, which is far below `WAD`, so the `require(buyerPrice <= WAD)` guard passes and `mulDivUp(WAD, buyerPrice)` succeeds. `sellerAssetsToUnits` has no analogous guard for `sellerPrice == 0`. [8](#0-7) 

**Note on `ConsumableUnitsLib.consumableUnits`:** When `offer.buy = true` and `offer.maxUnits == 0`, `consumableUnits` calls `buyerAssetsToUnits` (line 19), not `sellerAssetsToUnits` (line 21). The vulnerability is triggered solely by the direct call to `sellerAssetsToUnits` at line 286 of `MidnightBundles.sol`. [9](#0-8) 

## Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee(id, 0)` post-maturity reverts unconditionally, regardless of `targetSellerAssets`. This is a permanent DoS of the sell-via-periphery path for post-maturity markets whenever such an offer appears in the routing set. No funds are at risk for the attacker; the cost is only the gas to create the offer.

## Likelihood Explanation
**Preconditions:**
- Market is post-maturity (inevitable for any market).
- `settlementFeeCbp0 >= 1` (any non-zero post-maturity fee, the common case).
- A tick exists where `tickToPrice(tick) == settlementFee(id, 0)` — achievable given `PRICE_ROUNDING_STEP = 1e12` and max fee of `14e12`.
- The attacker's offer is included in the victim's `takes` array via off-chain routing.

The condition is repeatable: the offer can be recreated after expiry, and the revert is deterministic. Any routing system that automatically aggregates available buy offers without filtering for `sellerPrice > 0` is vulnerable.

## Recommendation
Add a guard in `sellerAssetsToUnits` analogous to the `require(buyerPrice <= WAD)` check in `buyerAssetsToUnits`. Specifically, before calling `mulDivUp`, require `sellerPrice > 0` when `offer.buy == true`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
if (offer.buy) require(sellerPrice > 0, SellerPriceIsZero());
return offer.buy
    ? targetSellerAssets.mulDivUp(WAD, sellerPrice)
    : targetSellerAssets.mulDivDown(WAD, sellerPrice);
```

Alternatively, update the NatSpec to explicitly document the `offerPrice == settlementFee` revert case, and add a pre-filter in `supplyCollateralAndSellWithAssetsTarget` to skip offers where `sellerPrice == 0` (treating them as fully consumed).

## Proof of Concept
1. Deploy Midnight with `settlementFeeCbp0 = 1` (so `settlementFee(id, 0) = 1e12`).
2. Create a buy offer at tick 2 (verify `tickToPrice(2) == 1e12` matches the settlement fee).
3. Advance `block.timestamp` past `offer.market.maturity`.
4. Call `supplyCollateralAndSellWithAssetsTarget` with `takes = [<the above offer>]` and any `targetSellerAssets > 0`.
5. Observe the transaction reverts with `stdError.arithmeticError` from `mulDivUp` in `sellerAssetsToUnits`.
6. Confirm that calling `IMidnight.take(offer, ..., units, ...)` directly with the same offer succeeds (computing `sellerAssets = units * 0 / WAD = 0`).
