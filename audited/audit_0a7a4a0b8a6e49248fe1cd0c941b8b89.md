### Title
`ConsumableUnitsLib.consumableUnits` returns 0 for uncapped offers (maxAssets=0, maxUnits=0), causing bundler to skip them entirely - (File: src/periphery/ConsumableUnitsLib.sol)

### Summary
When a maker creates an offer with both `maxAssets=0` and `maxUnits=0` (the documented "no cap" state), `ConsumableUnitsLib.consumableUnits` falls through to the assets-based branch and computes `zeroFloorSub(0, consumed) = 0`, then passes `0` to `TakeAmountsLib.buyerAssetsToUnits`/`sellerAssetsToUnits`, which returns `0`. Every bundler function that calls `ConsumableUnitsLib.consumableUnits` then computes `unitsToTake = min(..., 0) = 0`, issuing a no-op take and skipping the offer entirely. Uncapped offers are permanently unfillable through `MidnightBundles`.

### Finding Description

**Root cause — `ConsumableUnitsLib.consumableUnits`** [1](#0-0) 

The function has three branches:
1. `maxUnits > 0` → units-capped path
2. `else if (offer.buy)` → assets-capped buy path: `buyerAssetsToUnits(..., maxAssets.zeroFloorSub(consumed))`
3. `else` → assets-capped sell path: `sellerAssetsToUnits(..., maxAssets.zeroFloorSub(consumed))`

There is **no branch for `maxAssets == 0 && maxUnits == 0`** (the uncapped case). The function falls into branch 2 or 3 and evaluates:

```
offer.maxAssets.zeroFloorSub(consumed)
= zeroFloorSub(0, consumed)
= mul(gt(0, consumed), sub(0, consumed))   // UtilsLib assembly
= mul(0, ...)
= 0
```

`zeroFloorSub` returns `0` whenever `x <= y`; since `consumed >= 0` always, `zeroFloorSub(0, consumed)` is always `0`. [2](#0-1) 

`TakeAmountsLib.buyerAssetsToUnits` and `sellerAssetsToUnits` with `targetAssets = 0` return `0 * WAD / price = 0`. [3](#0-2) 

**Bundler impact** — all four bundler functions call:

```solidity
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // returns 0
);
```

`min(..., 0) = 0`, so `take` is called with `units = 0` (a documented no-op), `filledUnits` is not incremented, and the offer is skipped. [4](#0-3) 

**Protocol semantics confirm the bug** — Midnight.sol explicitly documents: *"If maxAssets > 0, assets are capped to maxAssets, otherwise units are capped to maxUnits."* When both are 0, there is no cap. `consumableUnits` misinterprets `maxAssets = 0` as "zero asset cap" rather than "no cap." [5](#0-4) 

### Impact Explanation
Any maker who publishes an uncapped offer (`maxAssets=0`, `maxUnits=0`) — the natural default for a maker who wants unlimited fills — will have that offer permanently skipped by all four `MidnightBundles` entry points. The bundler loop reaches `OutOfOffers()` revert even when valid liquidity exists, breaking bundler routing for uncapped offers.

### Likelihood Explanation
Uncapped offers (`maxAssets=0`, `maxUnits=0`) are the zero-value default for both fields and the natural choice for makers who do not want to limit their exposure. The bug triggers deterministically for every such offer on every bundler call — no special precondition, no attacker action, no race condition required. It is 100% reproducible.

### Recommendation
Add an explicit early-return for the uncapped case at the top of `consumableUnits`:

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    if (offer.maxUnits == 0 && offer.maxAssets == 0) {
        return type(uint256).max; // no cap: offer is always fully consumable
    }
    if (offer.maxUnits > 0) {
        return offer.maxUnits.zeroFloorSub(consumed);
    } else if (offer.buy) {
        return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
    } else {
        return TakeAmountsLib.sellerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
    }
}
```

// Foundry unit test
function testConsumableUnitsUncappedOfferReturnsZero() public {
    // Arrange: offer with maxAssets=0, maxUnits=0 (uncapped)
    Offer memory offer = buildOffer({ maxAssets: 0, maxUnits: 0, buy: false, ... });
    // consumed[maker][group] = 0 (fresh offer)

    uint256 result = ConsumableUnitsLib.consumableUnits(address(midnight), id, offer);

    // BUG: asserts 0, should be type(uint256).max or a large value
    assertEq(result, 0); // demonstrates the bug

    // Bundler consequence: unitsToTake = min(targetUnits, takes[i].units, 0) = 0
    // => take(offer, ..., 0, ...) is a no-op => offer skipped => OutOfOffers revert
}

function testBundlerSkipsUncappedOffer() public {
    // Arrange: single uncapped sell offer with sufficient liquidity
    // Act: call buyWithUnitsTargetAndWithdrawCollateral with targetUnits > 0
    // Assert: reverts with OutOfOffers() even though the offer can fill the target
    vm.expectRevert(IMidnightBundles.OutOfOffers.selector);
    bundles.buyWithUnitsTargetAndWithdrawCollateral(targetUnits, maxBuyerAssets, taker, permit, takes, ...);
}
