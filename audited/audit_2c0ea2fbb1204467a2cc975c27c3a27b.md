Audit Report

## Title
Fully-consumed buy offer with `buyerPrice < WAD` allows unlimited zero-cost re-takes via rounding bypass - (`src/Midnight.sol`)

## Summary
When a buy offer uses assets-mode (`maxAssets > 0`) and `buyerPrice < WAD`, calling `take` with `units` small enough that `units * buyerPrice < WAD` causes `buyerAssets = mulDivDown(units, buyerPrice, WAD) = 0`. Once `consumed == maxAssets`, the consumed increment adds 0, the `newConsumed <= maxAssets` check trivially passes, and the take executes fully — modifying buyer credit and seller debt — without any token payment. This is repeatable indefinitely.

## Finding Description

**Root cause:**

In `src/Midnight.sol` line 363, `buyerAssets` for a buy offer is computed as:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
```

`mulDivDown` in `src/libraries/UtilsLib.sol` line 30 is `(x * y) / d` (integer division). When `units * buyerPrice < WAD`, the result is 0. [1](#0-0) [2](#0-1) 

At lines 367–369, the consumed accounting increments by `buyerAssets`:

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

When `buyerAssets = 0`, `consumed` is unchanged. If `consumed == maxAssets`, then `newConsumed = maxAssets` and `require(maxAssets <= maxAssets)` passes. [3](#0-2) 

**`buyerPrice < WAD` is always reachable:** `tickToPrice` returns values in `[0, WAD]`, with `tickToPrice(MAX_TICK) = WAD` and any tick below `MAX_TICK` (5820) yielding a strictly smaller price. For a buy offer, `buyerPrice = offerPrice`, so any tick below `MAX_TICK` satisfies the precondition. [4](#0-3) 

**Position state is modified based on `units`, not `buyerAssets`:** Lines 382–414 compute `buyerCreditIncrease`, `sellerDebtIncrease`, etc. from `units` directly. With `units = 1`, `buyerPos.credit += 1` and `sellerPos.debt += 1` execute regardless of `buyerAssets`. [5](#0-4) 

**Token transfers are zero:** Lines 455–456 transfer `buyerAssets - sellerAssets = 0` and `sellerAssets = 0`, both of which succeed as valid ERC20 zero-value transfers. [6](#0-5) 

**Certora spec gap:** `fullyConsumedOfferRevertsOnNonTrivialTake` (lines 99–111) only covers `maxAssets == 0` (units mode) — there is no equivalent rule for assets mode. `takeConsumedAtMaxUnchangedAssets` (lines 88–97) only asserts that `consumed` does not change, which is vacuously satisfied when `buyerAssets = 0`, but does not assert that `units == 0` or that the take reverts. [7](#0-6) 

## Impact Explanation
A fully-consumed buy offer with any tick below `MAX_TICK` can be taken an unlimited number of times with `units > 0` and zero token cost. Each call increases the buyer's credit and the seller's debt by `units` without any corresponding loan token payment. This allows a taker to inflate credit/debt positions for free, bypassing the offer's intended fill cap entirely. The invariant that offers cannot be overfilled or replayed after exhaustion is violated, enabling unbounded artificial debt creation against any seller who has posted such an offer.

## Likelihood Explanation
The precondition `buyerPrice < WAD` is satisfied by any tick below `MAX_TICK` (5820), which covers the vast majority of all valid ticks. No special permissions are required — any unprivileged taker can call `take` with `units = 1`. The offer does not need to be maliciously crafted; any legitimate buy offer with a sub-par price is vulnerable once `consumed` reaches `maxAssets`. The exploit is repeatable within a single transaction or across blocks at negligible gas cost.

## Recommendation
Add a guard that rejects takes where the computed asset amount rounds to zero but `units > 0`. For buy offers in assets mode, require:

```solidity
require(buyerAssets > 0 || units == 0, ZeroAssets());
```

Alternatively, enforce a minimum `units` such that `units * buyerPrice >= WAD` when `maxAssets > 0` and `offer.buy == true`. The Certora spec should also be extended with a rule analogous to `fullyConsumedOfferRevertsOnNonTrivialTake` for assets mode, asserting that a successful take with `consumedBefore >= maxAssets` implies `units == 0`.

## Proof of Concept

1. Deploy a market with any tick `t < MAX_TICK` (e.g., `t = MAX_TICK - 2`, giving `buyerPrice = 1e18 - 1e12`).
2. Maker creates a buy offer with `maxAssets = 1e18` and `group = G`.
3. Legitimately fill the offer until `consumed[maker][G] == 1e18`.
4. Call `take(offer, ratifierData, 1, taker, ...)`. Verify: `mulDivDown(1, 1e18 - 1e12, 1e18) = 0`, so `consumed` stays at `1e18`, check passes.
5. Assert `buyerPos.credit` increased by 1 and `sellerPos.debt` increased by 1 with zero token transfer.
6. Repeat step 4 N times; assert credit/debt grow by N with no tokens paid.

/// If consumed is already at or above maxAssets before a take in assets mode, it remains unchanged.
rule takeConsumedAtMaxUnchangedAssets(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxUnits == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumedBefore >= offer.maxAssets => consumed(offer.maker, offer.group) == consumedBefore;
}

/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
