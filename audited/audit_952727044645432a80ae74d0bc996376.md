### Title
Blacklisted staker's principal permanently frozen with no recovery path - (File: src/ConfidencePool.sol)

### Summary
If a staker's address is blacklisted by the stake token (e.g., USDC) after depositing, their principal is permanently locked in the pool. `claimSurvived()` and `claimExpired()` always transfer to `msg.sender` with no recipient override, and `sweepUnclaimedBonus()` explicitly reserves the non-claimer principal, leaving no escape valve.

### Finding Description
`claimSurvived()` and `claimExpired()` both unconditionally transfer the payout to `msg.sender`: [1](#0-0) [2](#0-1) 

If the staker's address is blacklisted by the stake token after depositing, every call to either function reverts at the `safeTransfer` step. The staker cannot redirect the claim to a clean address because the claim entitlement is keyed to `eligibleStake[msg.sender]` and there is no recipient parameter.

`sweepUnclaimedBonus()` cannot rescue the principal either. It explicitly reserves `totalEligibleStake` (the sum of all non-claimer principals) and only sweeps the excess above that reserve: [3](#0-2) 

So the blacklisted staker's principal sits in `totalEligibleStake` indefinitely, protected from any sweep, with no admin function to redirect it.

The factory allowlist comment enumerates only fee-on-transfer and rebasing tokens as excluded: [4](#0-3) 

USDC is a standard ERC20 (no transfer fees, no rebasing) and would pass the allowlist check. The DESIGN.md §12 statement that "a true stuck-state would require a defective stake token that freezes the pool's own balance" refers to the pool contract address being blacklisted — a distinct scenario from a *user* being blacklisted. User-blacklisting is not acknowledged or excluded anywhere in the design documents.

### Impact Explanation
A staker whose address is blacklisted by the stake token after depositing loses their full principal permanently. The funds remain in the contract balance but are unreachable: claim functions revert, `sweepUnclaimedBonus()` reserves the amount, and no admin redirect exists. This is a full, irrecoverable principal loss for the affected staker.

### Likelihood Explanation
Blacklisting is a low-probability event (requires a compliance action by the token issuer against a specific address), but USDC is a widely used standard ERC20 that the factory allowlist is not documented to exclude. The combination of low probability and full principal loss places this at medium severity.

### Recommendation
Add an optional `recipient` parameter to `claimSurvived()` and `claimExpired()` (and `withdraw()`), defaulting to `msg.sender` when not provided, so a staker can direct their payout to a non-blacklisted address they control. For example:

```solidity
function claimSurvived(address recipient) external nonReentrant {
    if (recipient == address(0)) recipient = msg.sender;
    // ... existing logic ...
    stakeToken.safeTransfer(recipient, payout);
}
```

Alternatively, restrict the allowlist to tokens with no blacklist capability, and document that restriction explicitly.

### Proof of Concept
1. Factory owner allowlists USDC (a standard ERC20, no fees, no rebasing — passes the allowlist check).
2. Staker Alice deposits 10,000 USDC via `stake()`. `eligibleStake[Alice] = 10_000e6`.
3. Pool resolves as SURVIVED. `snapshotTotalStaked = 10_000e6`.
4. USDC issuer blacklists Alice's address (e.g., due to a compliance order).
5. Alice calls `claimSurvived()`. The call reaches `stakeToken.safeTransfer(msg.sender, payout)` and reverts because USDC's `transfer` checks the blacklist.
6. Alice has no alternative: she cannot pass a different recipient, and no admin function exists to redirect her claim.
7. Anyone calls `sweepUnclaimedBonus()`. The function computes `reserved = totalEligibleStake = 10_000e6` and finds `freeBalance - reserved = 0`, so it reverts with `NothingToSweep`.
8. Alice's 10,000 USDC is permanently locked in the contract. [5](#0-4) [6](#0-5)
