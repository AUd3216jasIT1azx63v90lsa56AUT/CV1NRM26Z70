### Title
`mulDivUp` Rounding in `pendingFeeDecrease` Allows Users to Eliminate Continuous Fee Obligation via Repeated Small Withdrawals — (File: src/Midnight.sol)

---

### Summary

The `withdraw` function and the `take` function (seller path) both compute `pendingFeeDecrease` using `mulDivUp`, rounding the fee reduction **up in favor of the user**. By making many small withdrawals of 1 unit at a time — batched cheaply via `multicall` — a lender can eliminate their entire `pendingFee` obligation while only withdrawing a small fraction of their credit, causing the protocol to collect far less continuous fee revenue than it is owed.

---

### Finding Description

**Root cause — `withdraw`:** [1](#0-0) 

```solidity
if (_position.credit > 0) {
    pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
    _position.pendingFee -= pendingFeeDecrease;
}
```

**Root cause — `take` (seller side):** [2](#0-1) 

```solidity
uint128 sellerPendingFeeDecrease = sellerPos.credit > 0
    ? UtilsLib.toUint128(sellerPos.pendingFee.mulDivUp(sellerCreditDecrease, sellerPos.credit))
    : 0;
```

`mulDivUp` is defined as: [3](#0-2) 

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```

**Mechanics of the exploit:**

When `units = 1` and `credit` is large relative to `pendingFee`, the correct proportional decrease is `pendingFee / credit < 1`, which floors to **0**. But `mulDivUp` returns **1**. Each single-unit withdrawal therefore reduces `pendingFee` by 1 regardless of how small the proportional share is.

A user with `credit = N` and `pendingFee = F` can call `withdraw(units=1)` exactly `F` times. After `F` calls:
- `credit` decreases from `N` to `N − F`
- `pendingFee` decreases from `F` to **0**

The correct proportional reduction for withdrawing `F` units would be `F * F / N`. For realistic parameters (e.g. `N = 1,000,000`, `F = 10,000`), the correct reduction is only **100 units**, but the attacker eliminates all **10,000 units** of pending fee — avoiding ~99% of their fee obligation.

**Batching via `multicall`:** [4](#0-3) 

All `F` withdrawals can be packed into one or a few `multicall` transactions. Because `_updatePosition` is called at the start of each `withdraw` and all calls share the same `block.timestamp`, `accrualEnd − _lastAccrual = 0` for every call after the first, so no additional fee accrues mid-batch. The rounding manipulation is fully contained within a single atomic transaction.

The developers acknowledge the direction of this rounding at line 115 of `Midnight.sol`: [5](#0-4) 

> `pendingFee updates are rounded in favor of the user. It could lead to fees manipulations too.`

However, the comment does not quantify the exploitability or bound the loss, and no mitigation is applied.

---

### Impact Explanation

The continuous fee is borne by lenders and flows to `continuousFeeCredit`, from which the `feeClaimer` withdraws via `claimContinuousFee`. [6](#0-5) 

By zeroing `pendingFee` before it accrues, the attacker causes `accruedFee = 0` on every subsequent `_updatePosition` call, so nothing is ever added to `continuousFeeCredit`. The protocol permanently loses the fee revenue on the attacker's position. The maximum loss per position is bounded by `MAX_CONTINUOUS_FEE` (1% annualised): [7](#0-6) 

For a 1 M-token position in a 1-year market, this is up to **10,000 tokens** of fee revenue lost per attacker per market. The attack is repeatable across all markets and all positions.

---

### Likelihood Explanation

**Preconditions (all attacker-controlled or market-normal):**
1. The attacker is a lender — a normal, permissionless role.
2. `withdrawable ≥ F` — borrowers must have repaid enough. This is a normal market condition, not a special setup.
3. Cheap gas — on any L2 (Arbitrum, Base, Optimism) the gas cost of `F` delegatecalls inside one `multicall` is negligible compared to the fee avoided on a meaningful position.

No privileged access, no oracle manipulation, and no external dependency is required. Any lender can execute this atomically.

---

### Recommendation

Replace `mulDivUp` with `mulDivDown` for `pendingFeeDecrease` in both `withdraw` and `take`, rounding the fee reduction **against** the user (in favour of the protocol):

```solidity
// withdraw (line 490)
pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivDown(units, _position.credit));

// take (line 387-388)
uint128 sellerPendingFeeDecrease = sellerPos.credit > 0
    ? UtilsLib.toUint128(sellerPos.pendingFee.mulDivDown(sellerCreditDecrease, sellerPos.credit))
    : 0;
```

This mirrors the fix described in the external report (switching from ceil to floor rounding) and ensures that partial withdrawals never reduce `pendingFee` by more than the proportional share.

---

### Proof of Concept

**Setup:** Market with `continuousFee = MAX_CONTINUOUS_FEE` (≈ 317 per second per WAD), maturity = 1 year from now, 1 token = 1 unit.

**Steps:**

1. Attacker calls `take` as buyer, acquiring `credit = 1,000,000`, receiving `pendingFee = 10,000` (1% of 1 M).
2. Attacker constructs a `multicall` payload of **10,000** calls to `withdraw(units=1, onBehalf=attacker, receiver=attacker)`.
3. **Call 1:** `_updatePosition` accrues 0 fee (just created). `pendingFeeDecrease = mulDivUp(10000, 1, 1000000) = 1`. State: `credit=999999, pendingFee=9999`.
4. **Call 2:** `_updatePosition` accrues 0 (same block). `pendingFeeDecrease = mulDivUp(9999, 1, 999999) = 1`. State: `credit=999998, pendingFee=9998`.
5. … (repeat) …
6. **Call 10,000:** `pendingFeeDecrease = mulDivUp(1, 1, 990001) = 1`. State: `credit=990000, pendingFee=0`.

**Result:** Attacker holds 990,000 credit with **zero pending fee**. All future `_updatePosition` calls accrue `fee = 0`. The protocol collects **0** continuous fee from this position instead of the expected ~9,900 tokens (the ~100 tokens correctly attributable to the 10,000 units withdrawn are also lost). The attacker can optionally re-lend the 10,000 withdrawn tokens at a much later time, incurring only a tiny new `pendingFee` proportional to the remaining time to maturity.
