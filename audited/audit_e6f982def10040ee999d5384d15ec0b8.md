### Title
Fee-on-Transfer Loan Token Inflates `claimableSettlementFee` Beyond Actual Contract Balance, Permanently Bricking `claimSettlementFee()` - (File: src/Midnight.sol)

### Summary

In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the nominal spread `buyerAssets - sellerAssets` before the inbound transfer executes. Because `SafeTransferLib.safeTransferFrom` does not verify the amount actually received, a fee-on-transfer loan token causes the contract to receive only `(buyerAssets - sellerAssets) * (1 - fee_rate)` while the accounting counter grows by the full nominal value. After N takes the counter permanently exceeds the contract's real token balance for that bucket, causing `claimSettlementFee()` to revert on any attempt to drain the full accumulated amount.

### Finding Description

**Exact code path:**

`take()` at [1](#0-0)  writes:
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
```
This happens unconditionally before any token movement. The actual inbound transfer for the fee portion occurs later at [2](#0-1) :
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
```

`SafeTransferLib.safeTransferFrom` only checks that the call did not revert and that the return value is truthy; it does not compare pre/post balances: [3](#0-2) 

`touchMarket()` validates collateral params, LLTV tiers, and maturity, but imposes **no restriction on `loanToken`**: [4](#0-3) 

Market creation is explicitly permissionless and supports "arbitrary loan token" (confirmed by `live_context.json` line 17).

**Attacker-controlled inputs:**
- Attacker deploys a fee-on-transfer ERC20 (e.g., 1% fee on every `transferFrom`) and sets it as `offer.market.loanToken`.
- Attacker (or any user) calls `take()` repeatedly on this market.

**Exploit flow:**
1. Attacker deploys `FeeToken` with `fee_rate = 1%` and creates a market with it as `loanToken`.
2. Each `take()` call with spread `S = buyerAssets - sellerAssets`:
   - `claimableSettlementFee[FeeToken] += S` (nominal)
   - Contract receives `S * 0.99` (actual)
   - Shortfall per call: `S * 0.01`
3. After N calls: `claimableSettlementFee[FeeToken]` = `N * S`, contract balance for fees = `N * S * 0.99`.
4. `feeClaimer` calls `claimSettlementFee(FeeToken, N*S, receiver)`:
   - `claimableSettlementFee[FeeToken] -= N*S` succeeds (no underflow)
   - `SafeTransferLib.safeTransfer(FeeToken, receiver, N*S)` **reverts** — contract holds only `N*S*0.99`.

**Why existing checks fail:**
- `SafeTransferLib` performs no balance delta check. [3](#0-2) 
- The Certora `Solvency.spec` `tokenBalanceCorrect` invariant is only proven under the explicit assumption `"no fee taking from sender or receiver"` — fee-on-transfer tokens are outside the verified model. [5](#0-4) 
- The `pendingFeeReceiptZero` weak invariant in `Solvency.spec` tracks the gap between the `claimableSettlementFee` increment and the inbound transfer, but the CVL token summary credits the full `value` to `dest` regardless of fee, so the invariant holds in the model but not on-chain with a fee token. [6](#0-5) 

### Impact Explanation

`claimSettlementFee()` permanently reverts for any `amount` exceeding the real balance attributable to fees. [7](#0-6)  The shortfall grows monotonically with every `take()` on the affected market. The `feeClaimer` can never recover the full `claimableSettlementFee` balance; partial claims are possible only up to the real balance, but the counter remains inflated, making the discrepancy irrecoverable without an admin rescue mechanism that does not exist in the contract.

### Likelihood Explanation

**Preconditions:** Attacker deploys any ERC20 with a transfer fee (trivial, no privilege required) and calls `touchMarket` or `take` with it as `loanToken` (permissionless). No oracle manipulation, no admin key, no user mistake required. The fee-on-transfer property is set at token deployment and applies to all transfers uniformly — it is not a post-deployment owner action.

**Feasibility:** Extremely low cost. A single Solidity contract deployment plus one `take()` call is sufficient to begin accumulating the shortfall. The effect is permanent and cumulative.

**Repeatability:** Every subsequent `take()` on the market widens the gap. The DoS on `claimSettlementFee()` is triggered the first time the claimer attempts to withdraw the full accumulated amount.

### Recommendation

Replace the nominal-amount accounting with a balance-delta pattern for the fee bucket. Before and after the inbound transfer at line 455, snapshot `IERC20(loanToken).balanceOf(address(this))` and credit `claimableSettlementFee` only by the actual delta:

```solidity
uint256 balBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived; // replaces line 418
```

Move the `claimableSettlementFee` increment to after the transfer and use `actualReceived`. Alternatively, add an explicit token whitelist or a deployment-time check that rejects tokens whose `transferFrom` delivers less than the requested amount.

import {Market, Offer, CollateralParams} from "src/interfaces/IMidnight.sol";

// Minimal fee-on-transfer ERC20: deducts 1% on every transferFrom to address(midnight)
contract FeeToken is ERC20 {
    address public midnight;
    constructor(address _midnight) ERC20("Fee","FEE") { midnight = _midnight; }
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = to == midnight ? amount / 100 : 0;
        super.transferFrom(from, to, amount - fee); // only amount-fee arrives
        if (fee > 0) _burn(from, fee);              // fee is destroyed
        return true;
    }
}

contract FeeOnTransferSettlementFeeTest is Test {
    Midnight midnight;
    FeeToken feeToken;
    // ... standard test setup (market, offer, collateral) ...

    function testClaimableSettlementFeeExceedsBalance() public {
        // Setup: create market with feeToken as loanToken, fund buyer, collateralize seller
        // ...

        uint256 N = 10;
        for (uint256 i = 0; i < N; i++) {
            // take() with spread S each time
            // claimableSettlementFee grows by S, contract receives S*0.99
        }

        uint256 claimable = midnight.claimableSettlementFee(address(feeToken));
        uint256 balance   = feeToken.balanceOf(address(midnight));

        // ASSERTION 1: counter exceeds real balance
        assertGt(claimable, balance, "claimable > balance");

        // ASSERTION 2: claiming full amount reverts
        vm.prank(feeClaimer);
        vm.expectRevert();
        midnight.claimSettlementFee(address(feeToken), claimable, address(this));
    }
}

- `claimableSettlementFee[feeToken] > feeToken.balanceOf(address(midnight))` after N takes.
- `claimSettlementFee(feeToken, claimable, receiver)` reverts with an ERC20 transfer failure.
- A fuzz variant over `fee_rate ∈ [1, 50]%` and `N ∈ [1, 100]` should always satisfy both assertions for any non-zero fee rate.

// Settlement fee receipts pending settlement: claimableSettlementFee is incremented in take before
// the inbound fee transfer happens, so we track the gap and clear it in CVL_transferFrom.
persistent ghost mapping(address => mathint) pendingFeeReceipt {
    init_state axiom (forall address token. pendingFeeReceipt[token] == 0);
}

hook Sstore claimableSettlementFee[KEY address token] uint256 newVal (uint256 oldVal) {
    // Except for claimSettlementFee, the claimableSettlementFee is non-decreasing, see WithdrawableMonotonicity.spec.
    if (newVal > oldVal) {
        pendingFeeReceipt[token] = pendingFeeReceipt[token] + newVal - oldVal;
    }
}

/// INVARIANTS AND RULES ///

// For any token, the pending settlement fee receipt after a transaction is 0: every claimableSettlementFee
// increment in take is paid back in by the same-function inbound transfer.
weak invariant pendingFeeReceiptZero(address token)
    pendingFeeReceipt[token] == 0;
