Audit Report

## Title
`IRepayCallback.onRepay` Missing `caller` Parameter Enables Authorized Operator to Drain Tokens from Arbitrary Callback Contracts - (File: src/Midnight.sol)

## Summary
The `repay()` function unconditionally sets `payer = callback` when a non-zero callback is supplied, with no requirement that `callback == msg.sender`. Because `IRepayCallback.onRepay` does not receive the original `msg.sender` (unlike `ILiquidateCallback.onLiquidate` and `IFlashLoanCallback.onFlashLoan`, which both pass `address caller`), a victim contract implementing the interface cannot distinguish a legitimate self-initiated repay from an attacker-initiated one. Any address authorized for any borrower — including a borrower the attacker themselves controls — can exploit this to pull loan tokens from an arbitrary contract that implements `onRepay()` and holds a Midnight approval.

## Finding Description

**Root cause — two cooperating flaws:**

**Flaw 1:** `repay()` in `src/Midnight.sol` lines 502–521 sets `payer = callback` with no constraint that `callback` is the caller:

```solidity
address payer = callback != address(0) ? callback : msg.sender;  // line 511
...
IRepayCallback(callback).onRepay(id, market, units, onBehalf, data) // line 516
...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
```

The authorization check on line 505 only verifies `msg.sender` is authorized for `onBehalf`. There is no check that `callback == msg.sender` or that `callback` is related to the caller in any way.

**Flaw 2:** `IRepayCallback.onRepay` in `src/interfaces/ICallbacks.sol` line 21 does not receive the original caller:

```solidity
function onRepay(bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
```

Compare with `ILiquidateCallback` (line 17) and `IFlashLoanCallback` (line 25), which both include `address caller` as their first parameter. A victim contract implementing `onRepay()` only knows Midnight called it — it cannot identify the original `msg.sender` and therefore cannot reject an attacker-initiated call.

**Exploit flow:**

1. Attacker creates their own borrower account (`selfBorrower`) and calls `setIsAuthorized(attacker, true, selfBorrower)` — no external privilege required.
2. Attacker calls `repay(market, units, selfBorrower, victimContract, data)`.
3. Authorization check passes: `isAuthorized[selfBorrower][attacker] == true`.
4. `payer = victimContract`.
5. `victimContract.onRepay(id, market, units, selfBorrower, data)` is called with `msg.sender == Midnight`. The victim contract cannot see the attacker's address and returns `CALLBACK_SUCCESS`.
6. `safeTransferFrom(loanToken, victimContract, Midnight, units)` drains `units` of loan tokens from `victimContract`.

**Why existing checks fail:**

The Certora spec `certora/specs/OnlyExplicitPayerCanLoseTokens.spec` lines 117–136 proves only that tokens are pulled from "a callback that returned `CALLBACK_SUCCESS`." This is a weaker property than "the callback consented to this specific repayment by this specific caller." The spec explicitly models `repayCallbackAllowed = true` for any callback returning success, confirming the protocol does not enforce caller-callback identity.

The `testRepayCallback` test at `test/OtherFunctionsTest.sol` lines 133–156 demonstrates the design: `caller` (authorized operator) and `callback` (payer) are explicitly different addresses, and no test verifies they must be the same.

## Impact Explanation
An attacker with authorization over any borrower (including a self-controlled one) can drain loan tokens from any contract that (a) implements `IRepayCallback.onRepay()` returning `CALLBACK_SUCCESS` and (b) has granted Midnight a token approval. The victim contract's balance decreases to repay a debt it did not choose to repay. This constitutes unauthorized movement of assets from a third-party contract, violating the protocol's stated invariant that "authorization must only allow intended account delegation."

## Likelihood Explanation
Preconditions are fully attacker-controlled: the attacker needs no external privilege, only a self-owned borrower account with self-authorization. Any production contract built as a repay-callback integration (holding Midnight approval and implementing `onRepay()`) is a valid target. The attack is repeatable as long as the victim retains approval and token balance. The asymmetry between `onRepay` (no `caller`) and `onLiquidate`/`onFlashLoan` (both pass `caller`) means no compliant victim contract can defend itself using the interface alone.

## Recommendation
**Primary fix:** Add `address caller` as the first parameter to `IRepayCallback.onRepay`, matching the pattern of `ILiquidateCallback` and `IFlashLoanCallback`:

```solidity
interface IRepayCallback {
    function onRepay(address caller, bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data) external returns (bytes32);
}
```

Pass `msg.sender` as `caller` at the call site in `repay()`. This allows victim contracts to verify the initiator and reject unauthorized calls.

**Secondary fix (defense-in-depth):** Require `callback == msg.sender` in `repay()`, or at minimum document that `callback` is an arbitrary address whose tokens will be pulled, so integrators understand the trust model.

## Proof of Concept

```solidity
// Minimal Foundry test demonstrating the drain

contract VictimRepayCallback is IRepayCallback {
    // Implements onRepay, returns CALLBACK_SUCCESS unconditionally
    // (cannot identify caller — only knows msg.sender == Midnight)
    function onRepay(bytes32, Market memory, uint256, address, bytes memory)
        external pure returns (bytes32) { return CALLBACK_SUCCESS; }
}

function testDrainVictimCallback() public {
    // 1. Attacker self-authorizes for their own borrower account
    address attacker = makeAddr("attacker");
    address selfBorrower = makeAddr("selfBorrower");
    vm.prank(selfBorrower);
    midnight.setIsAuthorized(attacker, true, selfBorrower);

    // 2. Give selfBorrower some debt to repay
    // (setup: collateralize + take to create debt of `units`)
    uint256 units = 1000;
    // ... standard market setup ...

    // 3. Deploy victim contract, fund it, give it Midnight approval
    VictimRepayCallback victim = new VictimRepayCallback();
    deal(address(loanToken), address(victim), units);
    vm.prank(address(victim));
    loanToken.approve(address(midnight), units);

    // 4. Attacker calls repay with victim as callback
    vm.prank(attacker);
    midnight.repay(market, units, selfBorrower, address(victim), "");

    // 5. Victim's tokens are gone; selfBorrower's debt is reduced
    assertEq(loanToken.balanceOf(address(victim)), 0);
    assertEq(midnight.debtOf(id, selfBorrower), 0);
}
```
