Looking at the exact code path in `buyWithUnitsTargetAndWithdrawCollateral`:

### Title
Arithmetic underflow in refund computation reverts `buyWithUnitsTargetAndWithdrawCollateral` when referral fee is nonzero and fills consume near-full budget - (File: src/periphery/MidnightBundles.sol)

### Summary
In `buyWithUnitsTargetAndWithdrawCollateral`, the contract pulls `maxBuyerAssets` from the caller, runs fills that can consume up to `maxBuyerAssets` in `filledBuyerAssets`, then computes `referralFeeAssets` as an *additional* amount on top of `filledBuyerAssets`. The refund subtraction `maxBuyerAssets - filledBuyerAssets - referralFeeAssets` at line 104 underflows whenever `filledBuyerAssets + referralFeeAssets > maxBuyerAssets`, causing the transaction to revert. The contract's own NatSpec comment ("The msg.sender will pay at most `maxBuyerAssets`") directly contradicts this behavior.

### Finding Description
**Exact code path:**

Line 66 pulls `maxBuyerAssets` from the caller: [1](#0-0) 

Lines 71â€“86 accumulate `filledBuyerAssets` via `take()` calls. The only bound on `filledBuyerAssets` is the bundler's token balance, which is exactly `maxBuyerAssets`. There is no cap that reserves headroom for the referral fee: [2](#0-1) 

Line 102 computes the referral fee using the asymmetric gross-up formula `filledBuyerAssets * referralFeePct / (WAD - referralFeePct)`, which is an *additional* amount on top of `filledBuyerAssets`: [3](#0-2) 

**Root cause:** The fill loop has no awareness of the referral fee. It can consume up to `maxBuyerAssets` in `filledBuyerAssets`. The referral fee is then computed on top, making the total cost `filledBuyerAssets * WAD / (WAD - referralFeePct)`, which exceeds `maxBuyerAssets` whenever `filledBuyerAssets > maxBuyerAssets * (WAD - referralFeePct) / WAD`.

**Contradictory NatSpec:** The comment at lines 46â€“48 simultaneously claims "The msg.sender will pay at most `maxBuyerAssets`" and that the total is `filledBuyerAssets + filledBuyerAssets * referralFeePct / (WAD - referralFeePct)`. These are only consistent if `filledBuyerAssets <= maxBuyerAssets * (WAD - referralFeePct) / WAD`, a constraint the code never enforces: [4](#0-3) 

**Concrete numeric example:**
- `WAD = 1e18`, `referralFeePct = 1e17` (10%)
- `maxBuyerAssets = 1000`, fills succeed and `filledBuyerAssets = 1000`
- `referralFeeAssets = 1000 * 1e17 / 9e17 = 111` (rounded down by `mulDivDown`)
- Refund = `1000 - 1000 - 111` â†’ arithmetic underflow â†’ revert in Solidity 0.8.34

**`mulDivDown` rounding direction does not help:** Rounding down makes `referralFeeAssets` smaller, which *reduces* the chance of underflow slightly, but does not prevent it. The underflow is structural: any `filledBuyerAssets > maxBuyerAssets * (WAD - referralFeePct) / WAD` triggers it, and the fill loop has no such cap. [5](#0-4) 

**Existing protections reviewed:**
- `require(referralFeePct < WAD, PctExceeded())` at line 61 only prevents `referralFeePct == WAD` (which would cause division by zero). It does not prevent the underflow for any `referralFeePct` in `[1, WAD-1]`.
- There is no `require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets)` check anywhere.
- The `try/catch` around `take()` only skips failed individual takes; it does not bound `filledBuyerAssets`.

### Impact Explanation
Any call to `buyWithUnitsTargetAndWithdrawCollateral` with a nonzero `referralFeePct` where the fills consume more than `maxBuyerAssets * (WAD - referralFeePct) / WAD` will revert. Since the revert unwinds all state changes including the token pull, the user loses only gas â€” but the transaction cannot complete. A user who sets `maxBuyerAssets` to their total intended budget (the documented behavior) will have every such transaction revert, permanently blocking the intended operation unless they set `maxBuyerAssets` significantly higher than the actual fill cost â€” which is not documented and contradicts the NatSpec.

### Likelihood Explanation
Preconditions: (1) caller passes any `referralFeePct > 0`; (2) the fill loop succeeds in filling `targetUnits` while consuming `filledBuyerAssets > maxBuyerAssets * (WAD - referralFeePct) / WAD`. Condition (2) is easily met: if `referralFeePct = 10%` and fills consume 95% of `maxBuyerAssets`, the total cost is `~105.5%` of `maxBuyerAssets`, causing underflow. This is a normal usage scenario â€” a user sets `maxBuyerAssets` as their total budget and passes a referral fee. The bug is repeatable and deterministic for any such parameter combination.

### Recommendation
Before the fill loop, compute the effective fill budget that leaves room for the referral fee:

```solidity
uint256 fillBudget = maxBuyerAssets.mulDivDown(WAD - referralFeePct, WAD);
```

Then use `fillBudget` as the cap for fills (e.g., stop the loop when `filledBuyerAssets >= fillBudget`), and compute `referralFeeAssets` from `filledBuyerAssets` as before. This guarantees `filledBuyerAssets + referralFeeAssets <= maxBuyerAssets` by construction, consistent with the documented invariant. Alternatively, add an explicit post-fill check:

```solidity
require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets, BudgetExceeded());
```
