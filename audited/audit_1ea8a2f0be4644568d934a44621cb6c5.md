### Title
`claimableSettlementFee` Over-Credited Before Transfer Enables Drain via Fee-on-Transfer Loan Token - (`src/Midnight.sol`)

### Summary
In `take()`, `claimableSettlementFee` is incremented by the full spread `buyerAssets - sellerAssets` at line 418, before the actual `safeTransferFrom` at line 455. If the loan token charges a fee on transfer, the contract receives strictly less than `buyerAssets - sellerAssets`, but the accounting is already committed. The Certora solvency spec explicitly acknowledges this ordering gap and papers over it by assuming tokens transfer correctly with no fee-taking — an assumption that is not enforced on-chain.

### Finding Description
**Exact code path:** [1](#0-0) 

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
```

This write happens unconditionally before any external call. The actual inbound transfer of the fee portion occurs later: [2](#0-1) 

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```

`SafeTransferLib.safeTransferFrom` only checks that the call does not revert and returns a truthy value — it does not verify the contract's balance delta. For a fee-on-transfer token with rate `f`, the contract receives `(buyerAssets - sellerAssets) * (1 - f)` but `claimableSettlementFee` is credited the full `buyerAssets - sellerAssets`.

**Why existing checks do not stop it:**

The Certora solvency spec explicitly models this gap with a `pendingFeeReceipt` ghost and clears it only when the transfer delivers exactly the expected amount: [3](#0-2) 

```
// Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits.
function _.transfer(address a, uint256 v) external with(env e) => CVL_transferFrom(...)
function _.transferFrom(address src, address a, uint256 v) external with(env e) => CVL_transferFrom(...)
``` [4](#0-3) 

The `pendingFeeReceiptZero` weak invariant and the `tokenBalanceCorrect` strong invariant both depend on this assumption. Neither invariant holds when a fee-on-transfer token is used. There is no on-chain guard that rejects fee-on-transfer tokens as loan tokens — markets are permissionlessly created with any `loanToken`.

**Attacker inputs / exploit flow:**

1. Attacker (market creator, or any participant) creates a market whose `loanToken` is a fee-on-transfer ERC20 (e.g., 1% fee).
2. Any taker calls `take()` against an offer in that market.
3. At line 418, `claimableSettlementFee[loanToken] += spread` (full amount).
4. At line 455, the contract receives `spread * 0.99`; the 1% is burned/redirected by the token.
5. After N takes, `claimableSettlementFee` exceeds the actual token balance attributable to fees by `N * spread * 0.01`.
6. `feeClaimer` calls `claimSettlementFee(token, claimableSettlementFee, receiver)`, which subtracts from the inflated counter and transfers the full recorded amount. [5](#0-4) 

Because `claimableSettlementFee` exceeds the real fee balance, the transfer at line 309 pulls tokens that belong to lenders (`withdrawable`) or collateral holders, breaking the `tokenBalanceCorrect` invariant.

### Impact Explanation
`feeClaimer` can withdraw more loan tokens than the protocol actually received as settlement fees. The shortfall is covered by tokens belonging to lenders and collateral suppliers in the same loan token, directly violating the core invariant that contract balances cover collateral, credit redemption, fees, and withdrawable assets. Lenders or borrowers may be unable to withdraw or redeem their full entitlement.

### Likelihood Explanation
Preconditions: a fee-on-transfer ERC20 must be used as a market's `loanToken`. Markets are permissionlessly created, so any participant can deploy such a market. Fee-on-transfer tokens exist in production (e.g., tokens with configurable transfer taxes). The inflation compounds with every `take()` call in the affected market, making it repeatable and cumulative. No privileged setup is required beyond the token itself existing.

### Recommendation
Replace the pre-transfer accounting increment with a balance-delta pattern:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 received = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += received;
```

Alternatively, document and enforce (via a registry or market creation check) that only standard ERC20 tokens with no transfer fees are permitted as loan tokens, consistent with the Certora assumption at line 31 of `Solvency.spec`.

### Proof of Concept
```solidity
// Foundry test outline
contract FeeOnTransferToken is ERC20 {
    uint256 public feeBps = 100; // 1%
    function _transfer(address from, address to, uint256 amount) internal override {
        uint256 fee = amount * feeBps / 10000;
        super._transfer(from, to, amount - fee);
        // fee is burned / kept by token
    }
}

function testClaimableSettlementFeeInflation() public {
    FeeOnTransferToken fot = new FeeOnTransferToken();
    // Create market with fot as loanToken, set up offer, collateral, etc.
    uint256 contractBalBefore = fot.balanceOf(address(midnight));
    take(units, taker, offer); // offer.market.loanToken = fot
    uint256 contractBalAfter = fot.balanceOf(address(midnight));
    uint256 actualFeeReceived = contractBalAfter - contractBalBefore;
    uint256 recorded = midnight.claimableSettlementFee(address(fot));
    // Assertion: recorded > actualFeeReceived (inflated by 1%)
    assertGt(recorded, actualFeeReceived, "claimableSettlementFee inflated");
    // feeClaimer drains recorded amount, pulling from lender withdrawable
    vm.prank(feeClaimer);
    midnight.claimSettlementFee(address(fot), recorded, feeClaimer);
    // Assert feeClaimer received more than actual fee tokens deposited
    assertEq(fot.balanceOf(feeClaimer), recorded);
    assertGt(recorded, actualFeeReceived);
}
```

Expected: `claimableSettlementFee > actual fee token balance`, and `claimSettlementFee` succeeds by draining tokens belonging to other users.

// Settlement fee receipts pending settlement: claimableSettlementFee is incremented in take before
// the inbound fee transfer happens, so we track the gap and clear it in CVL_transferFrom.
persistent ghost mapping(address => mathint) pendingFeeReceipt {
    init_state axiom (forall address token. pendingFeeReceipt[token] == 0);
}

hook Sstore claimableSettlementFee[KEY address token] uint256 newVal (uint256 oldVal) {
    // Except for claimSettlementFee, the claimableSettlementFee is non-decreasing, see WithdrawableMonotonicity.spec.
    if (newVal > oldVal) {
        pendingFeeReceipt[token] = pendingFeeReceipt[token] + newVal - oldVal;
    }
}

/// INVARIANTS AND RULES ///

// For any token, the pending settlement fee receipt after a transaction is 0: every claimableSettlementFee
// increment in take is paid back in by the same-function inbound transfer.
weak invariant pendingFeeReceiptZero(address token)
    pendingFeeReceipt[token] == 0;
