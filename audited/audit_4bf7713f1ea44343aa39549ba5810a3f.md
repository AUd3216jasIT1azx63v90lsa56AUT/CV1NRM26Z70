Audit Report

## Title
Unconditional zero-amount `safeTransfer` in `liquidate()` permanently blocks bad-debt-only liquidations for zero-revert collateral tokens - (File: src/Midnight.sol)

## Summary
`liquidate()` unconditionally calls `SafeTransferLib.safeTransfer(collateralToken, receiver, seizedAssets)` at line 696 and `SafeTransferLib.safeTransferFrom(loanToken, payer, address(this), repaidUnits)` at line 717, even when both `seizedAssets` and `repaidUnits` are zero. For any market whose collateral or loan token reverts on zero-amount `transfer()` calls, the bad-debt-only liquidation path (`seizedAssets = 0, repaidUnits = 0`) is permanently DoS'd. This directly contradicts the NatSpec guarantee at line 577 and prevents `lossFactor` updates and lender loss socialization.

## Finding Description

**Verified code path:**

The NatSpec at line 577 explicitly documents the zero-zero path as valid: [1](#0-0) 

The guard at line 595 uses `atMostOneNonZero`, which returns `true` when both inputs are zero, so the call is not rejected: [2](#0-1) 

The collateral/repayment block at line 643 is gated and is entirely skipped when both inputs are zero, leaving `seizedAssets = 0`: [3](#0-2) 

Bad-debt accounting at lines 626–641 executes correctly and modifies `_position.debt` and `_marketState`: [4](#0-3) 

Execution then falls through unconditionally to the `safeTransfer` call with `seizedAssets = 0`: [5](#0-4) 

And then unconditionally to `safeTransferFrom` with `repaidUnits = 0`: [6](#0-5) 

`SafeTransferLib.safeTransfer` has no zero-amount guard — it always issues the low-level `token.call` regardless of `value`: [7](#0-6) 

For a collateral token that reverts on zero-amount `transfer()`, the entire transaction reverts, rolling back all state changes including the bad-debt accounting performed at lines 626–641.

**Root cause:** Missing `if (seizedAssets > 0)` guard before line 696 and missing `if (repaidUnits > 0)` guard before line 717.

## Impact Explanation

Bad-debt-only liquidations (`seizedAssets = 0, repaidUnits = 0`) are permanently blocked for any market whose collateral token reverts on zero-amount transfers. The consequences are:

- `_marketState.lossFactor` is never updated — lenders cannot recover proportional losses
- `_position.debt` is never reduced — the borrower's bad debt persists indefinitely
- The protocol invariant "bad debt must reduce lender credit exactly once and proportionally" (stated in `live_context.json`) is violated
- The protocol's own documented guarantee at line 577 is broken

Market configuration is immutable post-deployment, so the collateral token cannot be replaced. The DoS is permanent. This matches the "permanent or long-term fund freeze," "bad debt creation," and "credit/debt accounting corruption" impact classes listed in `live_context.json`. [8](#0-7) 

## Likelihood Explanation

**Preconditions:**
1. A market exists with a collateral token that reverts on zero-amount `transfer()` calls. Several deployed ERC20 tokens exhibit this behavior. Market creation is permissionless — the market creator role is explicitly listed as an unprivileged attacker in scope. [9](#0-8) 
2. A borrower's position reaches `badDebt > 0` (collateral value insufficient to cover debt at `maxLif`), which is a normal market outcome under adverse price movement.

Non-standard token behavior is explicitly listed as in-scope under `external_calls` invariants and `recommended_fuzz_axes`: [10](#0-9) [11](#0-10) 

Once such a market exists and a position reaches bad-debt state, the DoS is triggered by any liquidator on every attempt. It is repeatable and permanent.

## Recommendation

Add zero-amount guards before both unconditional transfer calls in `liquidate()`:

```solidity
// Line 696 — guard the collateral transfer
if (seizedAssets > 0) {
    SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
}

// Line 717 — guard the loan token transfer
if (repaidUnits > 0) {
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
}
```

This aligns the implementation with the NatSpec guarantee at line 577 and ensures bad-debt-only liquidations succeed without any token transfer for markets with zero-revert tokens. [12](#0-11) 

## Proof of Concept

**Minimal Foundry test plan:**

1. Deploy a mock ERC20 collateral token whose `transfer()` reverts when `value == 0`.
2. Create a Midnight market using this token as collateral.
3. Supply collateral and borrow (take an offer) to create a position with `debt > 0`.
4. Manipulate the oracle price so that `badDebt > 0` (collateral value at `maxLif` is less than debt).
5. Call `liquidate(market, collateralIndex, 0, 0, borrower, false, receiver, address(0), "")`.
6. **Expected (per NatSpec):** Transaction succeeds; `_marketState.lossFactor` is updated; `_position.debt` is reduced.
7. **Actual:** Transaction reverts at line 696 due to the zero-amount `transfer()` call on the collateral token; all state changes are rolled back; bad debt is never realized.

        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));

                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
