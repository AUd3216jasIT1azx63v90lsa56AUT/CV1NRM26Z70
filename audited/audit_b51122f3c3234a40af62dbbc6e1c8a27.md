### Title
Attacker Permanently Locks `expiryLocked` via Stake-and-Withdraw, Griefing Sponsor's `setExpiry()` — (File: src/ConfidencePool.sol)

### Summary
An unprivileged attacker can permanently prevent the pool sponsor from calling `setExpiry()` by staking the minimum required amount and immediately withdrawing. The `expiryLocked` flag is set on the first `stake()` call and is never reset — even after full withdrawal — so the attacker recovers all tokens at the cost of gas alone while permanently closing the sponsor's expiry-adjustment window.

### Finding Description
`stake()` unconditionally sets `expiryLocked = true` on the first call:

```solidity
if (!expiryLocked) {
    expiryLocked = true;
}
``` [1](#0-0) 

`setExpiry()` gates entirely on this flag and reverts if it is set:

```solidity
function setExpiry(uint256 newExpiry) external onlyOwner {
    if (expiryLocked) revert ExpiryLocked();
``` [2](#0-1) 

`withdraw()` clears all stake accounting (`eligibleStake`, `userSumStakeTime`, `totalEligibleStake`) but never touches `expiryLocked`: [3](#0-2) 

`withdraw()` is fully available in pre-attack states (`NOT_DEPLOYED`, `NEW_DEPLOYMENT`, `ATTACK_REQUESTED`) when `riskWindowStart == 0`: [4](#0-3) 

An attacker therefore executes the following two-step grief at net-zero token cost:

1. Call `stake(minStake)` → `expiryLocked` is permanently set to `true`.
2. Call `withdraw()` → full principal is returned; `expiryLocked` remains `true`.

Any subsequent `setExpiry()` call by the sponsor reverts with `ExpiryLocked`, even though no legitimate staker has ever deposited and `totalEligibleStake` is back to zero.

DESIGN.md §10 documents that the latch does not reset on withdrawal ("resetting it would let the sponsor move `expiry` during an all-stakers-exited moment and harm the next cohort"), but it does not acknowledge or accept the distinct griefing vector where an adversary — not a legitimate staker — triggers the latch and immediately exits, leaving the sponsor with no recourse. [5](#0-4) 

### Impact Explanation
The sponsor permanently loses the ability to adjust the pool's expiry before any legitimate staker has deposited. This can prevent the sponsor from extending the pool's lifetime to attract more stakers, correcting an expiry set too short, or aligning the deadline with market conditions. The only recourse is to deploy an entirely new pool, wasting gas and potentially losing early staker interest. The pool itself continues to function, but the sponsor's documented pre-stake configuration right is irreversibly foreclosed.

### Likelihood Explanation
Medium. The attacker must hold `minStake` tokens of the allowlisted stake token, but recovers them in full via `withdraw()`, so the net cost is gas only. The attack can be executed in the same block as pool creation. Any actor who can acquire `minStake` tokens — a trivially low bar for most ERC-20 tokens — can execute this before the sponsor calls `setExpiry()`.

### Recommendation
Decouple the expiry-lock signal from the raw first-stake event. One approach: replace the boolean `expiryLocked` with a check against a monotonically increasing `totalStakeEver` counter (incremented in `stake()`, never decremented). Gate `setExpiry()` on `totalStakeEver > 0` and allow the expiry to be changed only while `totalEligibleStake == 0 && outcome == UNRESOLVED` — i.e., when no stake is currently held and the pool is unresolved. This preserves the design's stated goal (sponsor cannot move expiry while stakers are present or after resolution) while closing the griefing path where a stake-and-withdraw with zero net position permanently forecloses the sponsor's configuration window.

### Proof of Concept
```solidity
// Pool created: expiry = block.timestamp + 30 days, minStake = 1e18

// Attacker (any address with minStake tokens):
stakeToken.approve(address(pool), minStake);
pool.stake(minStake);   // expiryLocked = true; attacker's tokens held

pool.withdraw();        // attacker recovers all tokens; expiryLocked stays true

// Sponsor attempts to extend expiry:
pool.setExpiry(block.timestamp + 60 days);
// ↑ reverts: ExpiryLocked
// Sponsor has no further recourse; must deploy a new pool.
```
