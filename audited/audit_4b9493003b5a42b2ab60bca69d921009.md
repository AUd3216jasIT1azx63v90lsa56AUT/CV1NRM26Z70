### Title
Phantom Collateral via Silent No-Op `transferFrom` in `supplyCollateral` - (`File: src/Midnight.sol`)

### Summary
`supplyCollateral` credits `_position.collateral[collateralIndex]` before calling `SafeTransferLib.safeTransferFrom`. The library only checks for revert and a non-`false` return value; it does not verify that the protocol's actual token balance increased. A market creator who deploys a collateral token whose `transferFrom` silently no-ops (returns `true` or returns nothing) can supply phantom collateral, pass `isHealthy()`, and borrow real `loanToken` against zero actual collateral.

### Finding Description

**Code path:**

`supplyCollateral` (`src/Midnight.sol`, lines 524–546):

```
line 533: _position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
...
line 545: SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
``` [1](#0-0) 

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol`, lines 24–34):

```
require(token.code.length > 0, NoCode());
(bool success, bytes memory returndata) = token.call(...transferFrom...);
if (!success) { revert(...) }
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
``` [2](#0-1) 

**Root cause:** The library's three checks — code existence, no-revert, non-false return — are all satisfied by a token whose `transferFrom` is a no-op that returns `true` (or returns empty bytes). There is no balance-before/balance-after guard anywhere in `supplyCollateral`.

**Attacker inputs and exploit flow:**

1. Attacker deploys `ERC20NoRevert`: a contract with code, whose `transferFrom(from, to, amount)` does nothing and returns `true` (or returns nothing).
2. Market creation is permissionless (`touchMarket` has no token whitelist). Attacker calls `touchMarket(market)` with `collateralParams[0].token = address(ERC20NoRevert)` and a valid `lltv`/`maxLif`/`oracle`. [3](#0-2) 

3. Attacker calls `supplyCollateral(market, 0, 1e18, attacker)`.
   - Line 533 executes: `_position.collateral[0] += 1e18` — phantom credit recorded.
   - Line 545 executes: `safeTransferFrom(ERC20NoRevert, attacker, midnight, 1e18)` — no-op, returns `true`, all three library checks pass. Zero tokens transferred.
4. Attacker takes an existing sell offer (or one placed by a colluding/deceived lender) in this market. `isHealthy` reads `_position.collateral[0] == 1e18` and the oracle price, concludes the position is healthy, and the take succeeds. [4](#0-3) 

5. Attacker now holds real `loanToken` debt backed by zero actual collateral tokens in the contract.

**Why existing checks fail:**
- `SafeTransferLib` has no balance-delta check; it trusts the token's return value.
- `supplyCollateral` has no `balanceOf(address(this))` snapshot before/after.
- `touchMarket` validates `lltv`, `maxLif`, and collateral sort order, but does **not** validate that the token is a legitimate ERC20. [5](#0-4) 

### Impact Explanation
The attacker obtains phantom `position.collateral[i]` credit without depositing any tokens. `isHealthy()` computes `maxDebt` from this phantom collateral value and oracle price, so the position appears healthy. The attacker borrows real `loanToken` from lenders. When the loan matures or the attacker defaults, there is zero collateral to seize, creating unrecoverable bad debt socialized across all lenders in the market. The invariant `position.collateral[i] <= collateralToken.balanceOf(midnight)` is directly violated.

### Likelihood Explanation
**Preconditions:**
- Attacker deploys a no-op ERC20 (trivial, ~10 lines of Solidity).
- Market creation is permissionless; no admin approval needed.
- A lender must have a live buy offer in the market. This requires either a deceived lender or a colluding one (attacker can be both lender and borrower in separate accounts, though the economic incentive requires a real victim lender to extract value).

**Feasibility:** High — all preconditions are attacker-controlled except finding a victim lender. In practice, aggregators or UI tools that auto-fill offers across markets could be targeted. The attack is repeatable with any `assets` value up to `uint128` max.

### Recommendation
Add a balance-before/balance-after check in `supplyCollateral`:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
require(IERC20(collateralToken).balanceOf(address(this)) - balanceBefore == assets, BadTransfer());
```

This also catches fee-on-transfer tokens. Alternatively, enforce a collateral token whitelist at market creation time in `touchMarket`.
