Audit Report

## Title
Division-by-Zero in `sellerAssetsToUnits` When `tickToPrice(tick) == settlementFee` Causes DoS of `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/TakeAmountsLib.sol)

## Summary

In `TakeAmountsLib.sellerAssetsToUnits`, when `offer.buy == true` and `tickToPrice(offer.tick) == settlementFee`, `sellerPrice` is computed as `0`. The subsequent `mulDivUp(targetSellerAssets, WAD, 0)` reverts due to Solidity 0.8 checked-arithmetic underflow at `(d - 1)` with `d = 0`. Because `Midnight.take()` does not revert in this same state, the NatDoc invariant "midnight reverts too" is false for the equality case. Since `supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside its `try/catch` block, any `takes[]` array containing such an offer causes the entire bundler call to revert unconditionally.

## Finding Description

**Root cause — `sellerAssetsToUnits` (`src/periphery/TakeAmountsLib.sol` lines 44–46):**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...
```

When `offer.buy == true` and `offerPrice == settlementFee`, `sellerPrice = 0`. [1](#0-0) 

`mulDivUp` in `src/libraries/UtilsLib.sol` is:

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```

With `d = 0`, the expression `(d - 1)` underflows under Solidity 0.8 checked arithmetic, causing an unconditional revert. [2](#0-1) 

**Why `take()` does NOT revert in the same state:**

In `Midnight.take()`, `sellerAssets` is computed as `units.mulDivDown(sellerPrice, WAD)`, which divides by `WAD` (1e18), not by `sellerPrice`. With `sellerPrice = 0`, this yields `sellerAssets = 0` — no revert occurs. This directly contradicts the NatDoc at line 34 of `TakeAmountsLib.sol`:

> "Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)"

The comment is incorrect for the equality case: `midnight` does not revert, but `sellerAssetsToUnits` does. [3](#0-2) 

**Exploit path in `supplyCollateralAndSellWithAssetsTarget` (`src/periphery/MidnightBundles.sol` lines 285–300):**

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(   // ← OUTSIDE try/catch; reverts here
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) returns (...) { ... } catch {}  // never reached
```

The `sellerAssetsToUnits` call is not wrapped in the `try/catch`. Its revert propagates unconditionally to the caller. The NatDoc for this function explicitly acknowledges this: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." [4](#0-3) [5](#0-4) 

**Structural reachability of `offerPrice == settlementFee`:**

`tickToPrice` rounds prices to multiples of `PRICE_ROUNDING_STEP = 1e12`. [6](#0-5) [7](#0-6) 

Settlement fees are also multiples of `1e12`, making exact equality structurally guaranteed to be reachable. An attacker reads the current settlement fee, calls `TickLib.priceToTick(settlementFee, tickSpacing)` to find the matching tick, and posts a buy offer at that tick. No privileged access is required. [8](#0-7) 

## Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer at the settlement-fee price point in its `takes[]` array reverts entirely. An aggregator or UI that enumerates open buy offers and routes through the malicious offer will have the victim's entire bundler transaction reverted. This blocks the sell-via-periphery path for affected takers for as long as the offer exists and the settlement fee remains at a level with a matching tick. The impact is a targeted, low-cost, repeatable DoS of a core periphery function.

## Likelihood Explanation

- Settlement fees are public and readable on-chain; the attacker trivially computes the matching tick using `TickLib.priceToTick`.
- Both tick prices and settlement fees are multiples of `1e12`, making exact equality structurally guaranteed to be reachable for any non-zero settlement fee within the tick range.
- The attacker bears only the gas cost of posting one buy offer; no capital is at risk.
- The DoS persists until the offer is cancelled (by the attacker, who has no incentive to do so) or the settlement fee changes to a value with no matching tick.
- The attack is repeatable: if the fee changes, the attacker posts a new offer at the new matching tick.

## Recommendation

Guard against `sellerPrice == 0` in `sellerAssetsToUnits`. Two options:

1. **Add an explicit check** before calling `mulDivUp`:
   ```solidity
   require(sellerPrice > 0, ZeroSellerPrice());
   ```
   This makes the revert condition explicit and matches the NatDoc (update the comment to `<=` instead of `<`).

2. **Mirror the NatDoc precisely**: update the comment to state "Reverts if `offerPrice <= settlementFee`" and add the guard, so callers know to filter out such offers before passing them to `supplyCollateralAndSellWithAssetsTarget`.

Additionally, consider wrapping the `sellerAssetsToUnits` and `consumableUnits` calls inside the `try/catch` in `supplyCollateralAndSellWithAssetsTarget`, or pre-filtering offers where `tickToPrice(offer.tick) <= settlementFee` before the loop.

## Proof of Concept

1. Deploy Midnight with a non-zero settlement fee `F` that is a multiple of `1e12`.
2. Compute `tick = TickLib.priceToTick(F, tickSpacing)` such that `tickToPrice(tick) == F`.
3. Post a buy offer at `tick` (attacker-controlled, minimal collateral).
4. Call `supplyCollateralAndSellWithAssetsTarget` with `takes = [attackerOffer, ...]`.
5. Observe the entire call reverts at `TakeAmountsLib.sellerAssetsToUnits` due to underflow in `mulDivUp(targetSellerAssets, WAD, 0)`.

A Foundry fuzz test can confirm: for any `settlementFee` that is a multiple of `1e12` and within the tick price range, `sellerAssetsToUnits` with `offerPrice == settlementFee` and `targetSellerAssets > 0` always reverts.
