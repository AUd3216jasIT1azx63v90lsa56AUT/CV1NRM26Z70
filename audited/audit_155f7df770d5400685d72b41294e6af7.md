Audit Report

## Title
Flash loan repayment with fee-on-transfer token silently under-repays, draining Midnight's reserves — (File: `src/Midnight.sol`)

## Summary
`flashLoan` transfers `assets[i]` out to the callback and then pulls `assets[i]` back via `safeTransferFrom`, but never snapshots or asserts the contract balance before and after. When the token charges a transfer fee, the repayment leg delivers `assets[i] − fee` to Midnight instead of `assets[i]`, causing a net loss of `fee` tokens per call. Repeated calls drain Midnight's token reserves until lender withdrawals revert.

## Finding Description
**Code path:** `src/Midnight.sol` lines 737–752.

```solidity
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);          // out
}
require(IFlashLoanCallback(callback).onFlashLoan(...) == CALLBACK_SUCCESS, ...);
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]); // in
}
``` [1](#0-0) 

**Root cause:** No pre/post balance invariant is enforced. The function only requires that `safeTransferFrom` does not revert; it never asserts `balanceAfter >= balanceBefore`.

**Exploit flow** (fee rate `r`, loan amount `A`, Midnight balance `B`):

| Step | Action | Midnight balance | Callback balance |
|------|--------|-----------------|-----------------|
| Pre  | Attacker pre-funds callback with `A·r` | `B` | `A·r` |
| 1    | `safeTransfer(token, callback, A)` — callback receives `A·(1−r)` | `B − A` | `A·r + A·(1−r) = A` |
| 2    | `onFlashLoan` returns `CALLBACK_SUCCESS` | `B − A` | `A` |
| 3    | `safeTransferFrom(token, callback, midnight, A)` — Midnight receives `A·(1−r)` | `B − A·r` | `0` |

Midnight ends with `B − A·r`. Repeating `N` times drains `N·A·r`.

**Why existing checks fail:** `safeTransferFrom` succeeds because the callback holds exactly `A` tokens (pre-funded `A·r` + received `A·(1−r)`), so no revert occurs. The Certora `tokenBalanceCorrect` strong invariant in `certora/specs/Solvency.spec` is proven only under the explicit assumption of standard (non-fee) ERC20 tokens and provides no protection here. [2](#0-1) [3](#0-2) 

The `live_context.json` core invariants explicitly flag this gap: "fee-on-transfer, rebasing, false-return, ERC777-like hooks, and non-standard decimals should be tested if not explicitly excluded." [4](#0-3) 

## Impact Explanation
Midnight's actual token balance decreases by `A·r` per flash loan call while `withdrawable`, `collateralSum`, and `claimableSettlementFee` accounting remains unchanged. The actual balance falls below the sum of obligations tracked by the solvency invariant. Lenders attempting to `withdraw` will receive reverts once the balance is exhausted. This maps directly to the in-scope impact class of protocol insolvency and permanent fund freeze for lenders.

## Likelihood Explanation
**Preconditions:** (1) A fee-on-transfer token is used as a loan token in any Midnight market. The protocol explicitly supports "arbitrary loan token" and permissionless market creation, so an attacker can create such a market without any privilege. (2) The attacker pre-funds `A·r` tokens per iteration. The attack is permissionless, requires no special role, and is repeatable indefinitely. The cost to the attacker equals the damage to Midnight (1:1 griefing ratio), making it economically viable for a motivated adversary (e.g., a competitor or malicious actor targeting a specific market). [5](#0-4) 

## Recommendation
Add a balance snapshot before the outgoing transfer and assert the balance is fully restored after the repayment:

```solidity
for (uint256 i = 0; i < tokens.length; i++) {
    uint256 balanceBefore = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
    // ... callback ...
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(IERC20(tokens[i]).balanceOf(address(this)) >= balanceBefore, FlashLoanUnderRepaid());
}
```

Alternatively, document and enforce that fee-on-transfer tokens are not supported as loan or collateral tokens, and add a check or registry that rejects such tokens at market creation time.

## Proof of Concept
**Foundry fork test plan:**

1. Deploy a mock ERC20 with a configurable transfer fee (e.g., 1% deducted from receiver on every transfer).
2. Create a Midnight market using this token as the loan token.
3. Deposit `B` tokens into Midnight as a lender (establishing `withdrawable = B`).
4. Deploy an attacker contract that:
   - Pre-funds itself with `A·r` tokens.
   - Calls `flashLoan(tokens=[feeToken], assets=[A], callback=self, data="")`.
   - In `onFlashLoan`, does nothing and returns `CALLBACK_SUCCESS`.
5. Assert `feeToken.balanceOf(midnight) == B − A·r` after one call.
6. Repeat `N` times until `feeToken.balanceOf(midnight) < withdrawable`.
7. Call `withdraw` as the lender and assert it reverts.

Expected result: each iteration reduces Midnight's balance by `A·r` while accounting is unchanged, eventually making `withdraw` impossible. [6](#0-5)
