Audit Report

## Title
Division by Zero in `sellerAssetsToUnits` via `tickToPrice(0) = 0` Causes DoS of Bundle Flows - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary

`tickToPrice(0)` returns `0` by design due to double rounding by `PRICE_ROUNDING_STEP`. When `consumableUnits` is called on a sell offer with `offer.maxUnits == 0` and `offer.tick == 0`, it delegates to `sellerAssetsToUnits`, which sets `sellerPrice = 0` and calls `mulDivDown(targetSellerAssets, WAD, 0)`, triggering an unconditional division-by-zero revert. Because bundle functions in `MidnightBundles` call `consumableUnits` outside any `try/catch`, the entire bundle transaction reverts, enabling a zero-cost griefing attack by any unprivileged maker.

## Finding Description

**Confirmed zero-price fact:**

`tickToPrice(0) = 0` is explicitly tested and passes in `test/TickLibTest.sol`: [1](#0-0) 

It is also formally verified in `certora/specs/TickToPrice.spec`: [2](#0-1) 

The root cause is the double rounding in `tickToPrice`: for tick=0, the intermediate result of `1e36 / (1e18 + wExp(...))` is smaller than `PRICE_ROUNDING_STEP = 1e12`, so `divHalfDownUnchecked(..., PRICE_ROUNDING_STEP) * PRICE_ROUNDING_STEP` collapses to `0`: [3](#0-2) 

**Reachable division-by-zero path:**

`consumableUnits` reaches `sellerAssetsToUnits` when `offer.maxUnits == 0` and `offer.buy == false`: [4](#0-3) 

Inside `sellerAssetsToUnits`, for a sell offer (`offer.buy = false`), `sellerPrice` is set directly to `offerPrice = tickToPrice(0) = 0`, then `mulDivDown(targetSellerAssets, WAD, sellerPrice)` is called with denominator `0`. Even if `targetSellerAssets = 0`, Solidity's `0 / 0` is still a division-by-zero revert: [5](#0-4) 

**Bundle functions propagate the revert:**

`buyWithUnitsTargetAndWithdrawCollateral` calls `consumableUnits` at line 77 outside any `try/catch`; only the subsequent `take` call is wrapped. The NatSpec explicitly documents this: *"Reverts if ConsumableUnitsLib reverts."* [6](#0-5) [7](#0-6) 

`buyWithAssetsTargetAndWithdrawCollateral` has the same pattern at lines 208–213: [8](#0-7) [9](#0-8) 

`supplyCollateralAndSellWithAssetsTarget` also calls `consumableUnits` at line 290 outside any `try/catch` (for buy offers with tick=0 and zero settlement fee): [10](#0-9) 

**Why existing checks fail:**

- `tickToPrice` only guards `tick > MAX_TICK`; tick=0 is in range.
- `consumableUnits` has no guard for `sellerPrice == 0`.
- `sellerAssetsToUnits` has no guard for `sellerPrice == 0`.
- The `NoDivisionByZero.spec` Certora spec covers only `Midnight.sol` and uses `NONDET` for `tickToPrice`, so it does not catch this periphery path. The comment claims `tickToPrice` is "checked manually to not cause a division by zero," but this manual check does not extend to `TakeAmountsLib.sellerAssetsToUnits`: [11](#0-10) 

## Impact Explanation

Any bundle transaction that calls `consumableUnits` on a sell offer with `tick=0` and `maxUnits=0` reverts unconditionally. Because `MidnightBundles` does not wrap `consumableUnits` in a `try/catch`, the entire bundle call fails. This is a concrete DoS of bundle flows: any automated system (aggregator, router, front-end) that selects offers from an on-chain order book and includes a malicious offer in the `takes` array will have its entire bundle transaction griefed. Legitimate takers lose gas and fail to execute their intended trades.

## Likelihood Explanation

The preconditions are trivially satisfiable by any unprivileged maker: `tick=0` is always a valid, accessible tick (it satisfies `tick <= MAX_TICK` and `0 % spacing == 0` for any spacing), and setting `maxUnits=0` with any `maxAssets > 0` is a normal asset-capped offer configuration. The attack is zero-cost (no funds required beyond gas for offer creation), repeatable on every block, and requires no special role or privilege. The attack is most effective against automated bundle-construction systems that pull offers from an on-chain order book without filtering for zero-price ticks.

## Recommendation

Add a zero-price guard in `sellerAssetsToUnits` before the division:

```solidity
require(sellerPrice > 0, ZeroSellerPrice());
```

Alternatively, add the guard in `consumableUnits` before delegating to `sellerAssetsToUnits`:

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
if (offerPrice == 0) return 0; // or revert, depending on desired semantics
```

A third option is to add a zero-price check in `tickToPrice` itself or enforce a minimum tick in offer creation, but this would be a more invasive change. The most targeted fix is in `sellerAssetsToUnits` and `buyerAssetsToUnits` in `TakeAmountsLib.sol`.

## Proof of Concept

```solidity
// Minimal Foundry test demonstrating the DoS
function testDivisionByZeroGrief() public {
    // Attacker creates a sell offer with tick=0, maxUnits=0, maxAssets=1e18
    Offer memory maliciousOffer;
    maliciousOffer.buy = false;
    maliciousOffer.tick = 0;
    maliciousOffer.maxUnits = 0;
    maliciousOffer.maxAssets = 1e18;
    maliciousOffer.maker = attacker;
    maliciousOffer.market = market;
    // ... (set ratifier, group, etc.)

    // Verify tickToPrice(0) == 0
    assertEq(TickLib.tickToPrice(0), 0);

    // consumableUnits reverts: sellerAssetsToUnits called with sellerPrice=0
    vm.expectRevert(); // division by zero
    ConsumableUnitsLib.consumableUnits(address(midnight), id, maliciousOffer);

    // Bundle call including this offer reverts entirely
    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: maliciousOffer, units: type(uint256).max, ratifierData: hex""});
    vm.prank(victim);
    vm.expectRevert();
    midnightBundles.buyWithUnitsTargetAndWithdrawCollateral(
        1e18, type(uint256).max, victim, _noPermit(), takes,
        new CollateralWithdrawal[](0), address(0), 0, address(0)
    );
}
```
