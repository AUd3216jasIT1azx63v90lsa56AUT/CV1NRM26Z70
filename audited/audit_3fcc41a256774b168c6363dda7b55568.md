### Title
Arithmetic underflow in refund calculation causes opaque revert when `referralFeeAssets` pushes total cost above `maxBuyerAssets` - (File: src/periphery/MidnightBundles.sol)

### Summary

`buyWithUnitsTargetAndWithdrawCollateral` pulls `maxBuyerAssets` from the caller, fills orders accumulating `filledBuyerAssets`, then computes `referralFeeAssets = filledBuyerAssets * referralFeePct / (WAD - referralFeePct)` and attempts the refund `maxBuyerAssets - filledBuyerAssets - referralFeeAssets`. No guard exists to verify `filledBuyerAssets + referralFeeAssets <= maxBuyerAssets` before this subtraction. When a caller sets `maxBuyerAssets` to cover only the fill cost (not the additional referral fee on top), the subtraction underflows in Solidity 0.8.34, reverting the entire transaction with an arithmetic panic and violating the NatSpec invariant "The msg.sender will pay at most maxBuyerAssets."

### Finding Description

**Exact code path** — `src/periphery/MidnightBundles.sol` lines 66–104:

1. Line 66: `pullToken(loanToken, msg.sender, maxBuyerAssets, ...)` — bundler holds `maxBuyerAssets`.
2. Lines 71–86: fills loop accumulates `filledBuyerAssets` from actual `take()` results; no cap relative to `maxBuyerAssets` minus the future fee.
3. Line 102: `referralFeeAssets = filledBuyerAssets.mulDivDown(referralFeePct, WAD - referralFeePct)` — fee is computed *after* fills are complete.
4. Line 104: `SafeTransferLib.safeTransfer(loanToken, msg.sender, maxBuyerAssets - filledBuyerAssets - referralFeeAssets)` — unchecked subtraction; underflows when `filledBuyerAssets + referralFeeAssets > maxBuyerAssets`.

**Root cause** — The function design requires `maxBuyerAssets` to cover `filledBuyerAssets * WAD / (WAD - referralFeePct)`, but this is never validated. The fills loop has no upper bound derived from the fee-adjusted budget, and there is no `require` before line 104.

**Attacker-controlled inputs** — Any unprivileged taker (borrower acting as taker, or any `msg.sender` authorized by the taker) can trigger this by:
- Setting `referralFeePct` to any value in `[1, WAD-1]` (only `referralFeePct < WAD` is checked at line 61).
- Setting `maxBuyerAssets` to a value that covers `filledBuyerAssets` but not `filledBuyerAssets + referralFeeAssets`.

**Concrete exploit flow:**
```
referralFeePct = 0.5e18  (50%)
filledBuyerAssets = 100e18  (actual fill cost at market price)
referralFeeAssets = mulDivDown(100e18, 0.5e18, 0.5e18) = 100e18
maxBuyerAssets = 100e18  (user set it to their expected fill cost)

Line 104: 100e18 - 100e18 - 100e18  →  underflow → revert
```

Even with a modest fee, e.g. `referralFeePct = 0.01e18` (1%):
```
filledBuyerAssets = 100e18
referralFeeAssets = mulDivDown(100e18, 0.01e18, 0.99e18) ≈ 1.01e18
maxBuyerAssets = 100e18
→ 100e18 - 100e18 - 1.01e18  →  underflow → revert
```

**Why existing checks fail** — Line 61 only checks `referralFeePct < WAD`. The fills loop has no fee-adjusted budget cap. There is no `require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets, ...)` anywhere before line 104. The existing fuzz test `testBuyUnitsTargetWithReferralFee` (test line 534) always passes `type(uint256).max` as `maxBuyerAssets`, which never triggers the underflow and leaves this path untested.

### Impact Explanation

Every call to `buyWithUnitsTargetAndWithdrawCollateral` with `referralFeePct > 0` and `maxBuyerAssets` set to the expected fill cost (without the fee premium) reverts with an arithmetic underflow panic. The user cannot complete their intended borrow. Because the revert rolls back all state including `pullToken`, no funds are permanently lost, but the transaction is a failed no-op. The NatSpec guarantee "The msg.sender will pay at most maxBuyerAssets" is violated: instead of succeeding or reverting with a meaningful error, the function panics opaquely.

### Likelihood Explanation

Any taker who passes a nonzero `referralFeePct` and sets `maxBuyerAssets` to the expected fill cost (a natural interpretation of the parameter name and the "at most" NatSpec guarantee) will hit this. The condition `filledBuyerAssets * WAD / (WAD - referralFeePct) > maxBuyerAssets` is trivially satisfied whenever `maxBuyerAssets == filledBuyerAssets` and `referralFeePct > 0`. This is repeatable and deterministic for any such input combination.

### Recommendation

Before the refund transfer on line 104, add an explicit check:

```solidity
require(filledBuyerAssets + referralFeeAssets <= maxBuyerAssets, BuyerAssetsTooHigh());
```

Alternatively, cap the fills loop so that `filledBuyerAssets` never exceeds `maxBuyerAssets * (WAD - referralFeePct) / WAD`, mirroring the approach used in `buyWithAssetsTargetAndWithdrawCollateral` (lines 200–201) where the fee is deducted from `targetBuyerAssets` before fills begin.

### Proof of Concept

```solidity
function testFuzz_buyUnitsTarget_referralFeeUnderflow(
    uint256 referralFeePct
) public {
    referralFeePct = bound(referralFeePct, 1, WAD - 1);

    uint256 units = 100e18;
    // Zero settlement fees so filledBuyerAssets == units exactly.
    for (uint256 i; i <= 6; i++) midnight.setMarketSettlementFee(id, i, 0);

    uint256 price = TickLib.tickToPrice(MAX_TICK);
    uint256 filledBuyerAssets = units.mulDivUp(price, WAD);
    // maxBuyerAssets covers only the fill, not the referral fee on top.
    uint256 maxBuyerAssets = filledBuyerAssets;

    offers[0].buy = false;
    offers[0].maker = borrower;
    offers[0].receiverIfMakerIsSeller = borrower;
    offers[0].maxUnits = units;
    collateralize(market, borrower, units);

    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offers[0], units: units, ratifierData: hex""});

    vm.prank(lender);
    // Expect arithmetic underflow (panic 0x11) on line 104.
    vm.expectRevert(stdError.arithmeticError);
    midnightBundles.buyWithUnitsTargetAndWithdrawCollateral(
        units,
        maxBuyerAssets,
        lender,
        _noPermit(),
        takes,
        new CollateralWithdrawal[](0),
        address(0),
        referralFeePct,
        makeAddr("referrer")
    );
}
```

**Expected assertion:** `vm.expectRevert(stdError.arithmeticError)` passes for all `referralFeePct` in `[1, WAD-1]`, confirming the underflow. After the fix, the assertion should be replaced with a check that the call succeeds (or reverts with the new meaningful error) and `loanToken.balanceOf(address(midnightBundles)) == 0`.
