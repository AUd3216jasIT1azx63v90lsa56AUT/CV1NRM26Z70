Audit Report

## Title
Authorized Operator Can Consume Authorizer's Nonce via Self-Revocation in EcrecoverAuthorizer - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` permits any address that satisfies `isAuthorized[authorizer][signer] == true` in Midnight to sign and submit an `Authorization` struct on behalf of the authorizer, including one that revokes the signer's own authorization (`authorized == signer`, `isAuthorized == false`). Because the nonce increment at line 26 is unconditional, this allows an already-authorized operator (`operatorA`) to advance the victim's nonce in `EcrecoverAuthorizer` without the victim's consent, permanently invalidating any pre-signed authorization the victim has distributed at that nonce value.

## Finding Description
**Root cause:** The signer validity check at lines 33–36 of `EcrecoverAuthorizer.sol` only verifies that the signer has authority to act on behalf of `authorization.authorizer`; it does not restrict what the signer may do with that authority:

```solidity
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

There is no guard preventing `signer == authorization.authorized && authorization.isAuthorized == false` (self-revocation). The nonce is incremented at line 26 before the signer check, and the call to `Midnight.setIsAuthorized` at lines 46–47 executes unconditionally after the check passes.

**Exploit flow:**

Preconditions (normal operating state):
- `isAuthorized[victim][EcrecoverAuthorizer] == true` in Midnight (victim uses the authorizer).
- `isAuthorized[victim][operatorA] == true` in Midnight.
- Victim has pre-signed and distributed `Authorization{authorizer=victim, authorized=operatorB, isAuthorized=true, nonce=N}`.

Attack:
1. `operatorA` constructs `Authorization{authorizer=victim, authorized=operatorA, isAuthorized=false, nonce=N, deadline=future}` and signs it with their own key.
2. Any caller submits this to `EcrecoverAuthorizer.setIsAuthorized`.
3. Line 26: `nonce[victim]` is incremented from `N` to `N+1`. ✓
4. Lines 33–36: `signer = operatorA`; `isAuthorized[victim][operatorA] == true` → check passes. ✓
5. Lines 46–47: `Midnight.setIsAuthorized(operatorA, false, victim)` executes — `operatorA` is revoked in Midnight.
6. Victim's pre-signed authorization to `operatorB` at nonce `N` is now permanently invalid; it will revert with `InvalidNonce()` if submitted.

**Why existing checks fail:** The `Unauthorized()` guard only confirms the signer has delegation authority. It does not distinguish between a signer acting to benefit the authorizer versus acting to consume the authorizer's nonce for the signer's own purposes. `Midnight.setIsAuthorized` (line 731–735 of `Midnight.sol`) also passes because `EcrecoverAuthorizer` is itself authorized by the victim.

## Impact Explanation
The primary concrete harm is **nonce griefing**: an authorized operator can front-run or race any pre-signed authorization the victim has distributed, permanently invalidating it by consuming the nonce. The victim must re-sign and redistribute a new authorization at the incremented nonce. In time-sensitive flows (e.g., a pre-signed grant to `operatorB` that must be submitted within a deadline), this can cause the authorization to expire before it can be resubmitted. The secondary effect — self-revocation — is already achievable by `operatorA` calling `Midnight.setIsAuthorized` directly and does not represent a new capability; the novel harm is exclusively the unauthorized nonce consumption in `EcrecoverAuthorizer`.

## Likelihood Explanation
Preconditions are the normal operating state of the protocol. No special privilege beyond being an already-authorized operator is required. `operatorA` can generate the self-revoking signature at any time and submit it (or have any third party submit it) without victim interaction. The attack is repeatable at every nonce increment if `operatorA` is re-authorized. It is front-runnable: if the victim broadcasts a pre-signed authorization, `operatorA` can observe it in the mempool and submit the self-revocation first.

## Recommendation
Add a guard in `EcrecoverAuthorizer.setIsAuthorized` that prevents a signer from submitting an authorization that revokes their own access when they are acting as a delegate (not as the authorizer themselves):

```solidity
require(
    signer == authorization.authorizer
        || authorization.isAuthorized
        || authorization.authorized != signer,
    Unauthorized()
);
```

This ensures that only the authorizer themselves (not a delegate) can submit a self-revocation, preserving the integrity of the victim's nonce for their own pre-signed authorizations. Alternatively, restrict nonce-consuming calls to `signer == authorization.authorizer` only, and provide a separate path for authorized delegates that does not touch the nonce.
