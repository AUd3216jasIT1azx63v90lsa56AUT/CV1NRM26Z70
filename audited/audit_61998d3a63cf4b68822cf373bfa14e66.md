Audit Report

## Title
Unbounded zero-unit take loop in bundle functions allows gas griefing via fully-consumed group front-run - (File: src/periphery/MidnightBundles.sol)

## Summary
All four bundle functions in `MidnightBundles.sol` dispatch `IMidnight.take()` with `units=0` when `ConsumableUnitsLib.consumableUnits()` returns 0 for a fully-consumed group, because no guard skips the `try` call when `unitsToTake == 0`. Since `Midnight.take()` explicitly permits no-op takes on fully-consumed offers (line 93 comment), each such call succeeds and returns `(0, 0)`, leaving `filledUnits` unchanged. An attacker who controls the maker address can front-run a taker's bundle transaction by calling `setConsumed(group, type(uint256).max, maker)`, converting every offer in the `takes[]` array into a zero-unit no-op and forcing the taker to burn O(N × per-take gas) before the transaction reverts with `OutOfOffers()`.

## Finding Description

**Root cause:** No `if (unitsToTake == 0) continue;` guard exists before the `try` call in any of the four bundle loops.

**Exact code path in `buyWithUnitsTargetAndWithdrawCollateral` (lines 74–85):**

`unitsToTake` is computed as the three-way minimum: [1](#0-0) 

`ConsumableUnitsLib.consumableUnits()` returns `offer.maxUnits.zeroFloorSub(consumed)`, which is 0 when `consumed >= maxUnits`: [2](#0-1) 

`zeroFloorSub` returns 0 when `x <= y`: [3](#0-2) 

So `unitsToTake = min(targetUnits - filledUnits, takes[i].units, 0) = 0`. The `try` is dispatched unconditionally with `units=0`: [4](#0-3) 

Inside `Midnight.take()`, with `units=0` and `consumed == maxUnits`, the consumed check evaluates `newConsumed = maxUnits + 0 = maxUnits`, so `require(maxUnits <= maxUnits)` passes: [5](#0-4) 

The protocol comment at line 93 explicitly confirms this is intentional behavior: [6](#0-5) 

Before the consumed check, the ratifier external call executes unconditionally: [7](#0-6) 

`_updatePosition` is called for buyer and/or seller if they hold credit: [8](#0-7) 

A `Take` event is emitted, a TSTORE liquidation lock is set, two `safeTransferFrom` calls with amount 0 execute, and an `isHealthy` call (potentially invoking an oracle) runs: [9](#0-8) 

`take()` returns `(0, 0)`. Back in the bundle, `filledUnits += 0`, the loop condition `filledUnits < targetUnits` remains true, and the next iteration begins. After all N iterations the loop exits by exhausting `takes.length`, and `require(filledUnits == targetUnits, OutOfOffers())` reverts the entire transaction: [10](#0-9) 

**`setConsumed` is callable by the maker (any unprivileged user acting as their own `onBehalf`):** [11](#0-10) 

**Why existing checks fail:**
- The `try/catch` at lines 79–85 is designed to tolerate asynchrony (e.g., an offer taken by someone else between block construction and execution). It silently swallows both reverts and zero-return successes, providing no protection against this case.
- The loop condition `filledUnits < targetUnits` does not terminate early because `filledUnits` never advances when all takes return 0.
- `ConsumableUnitsLib.consumableUnits()` correctly returns 0, but the bundle does not act on this signal before dispatching the take.

The same missing guard exists identically in all four bundle functions:
- `buyWithUnitsTargetAndWithdrawCollateral` (lines 74–85): [12](#0-11) 
- `supplyCollateralAndSellWithUnitsTarget` (lines 147–160): [13](#0-12) 
- `buyWithAssetsTargetAndWithdrawCollateral` (lines 208–221): [14](#0-13) 
- `supplyCollateralAndSellWithAssetsTarget` (lines 285–300): [15](#0-14) 

## Impact Explanation
Every zero-unit take in `Midnight.take()` executes: one external ratifier call (line 356), up to two `_updatePosition` calls with multiple SLOADs/SSTOREs (lines 379–380), position arithmetic touching storage slots (lines 408–414), a `claimableSettlementFee` SLOAD+SSTORE (line 418), a `Take` event emission (lines 425–442), a TSTORE liquidation lock (line 444), two `safeTransferFrom` calls with amount 0 (lines 455–456), and an `isHealthy` call that may invoke an oracle (line 476). Conservatively, each zero-unit take costs on the order of 30,000–80,000 gas depending on position state and oracle complexity. For N = 100 offers, the taker's transaction burns 3,000,000–8,000,000 gas before reverting with `OutOfOffers()`. The taker loses all gas; no funds are transferred because the transaction reverts. The attack is a repeatable, targeted DoS against any user of the bundle functions.

## Likelihood Explanation
**Preconditions:**
1. Attacker controls a maker address — any unprivileged user qualifies.
2. Attacker creates N offers sharing the same `(maker, group)` pair — no cost beyond gas.
3. Attacker calls `setConsumed(group, type(uint256).max, maker)` — one cheap transaction, callable by the maker on their own behalf.
4. A victim taker calls a bundle function with those N offers in `takes[]`.

The front-run variant is straightforward on any chain with a public mempool. The social-engineering variant (listing N exhausted offers in an off-chain order book) requires no front-running at all. The attack is repeatable: the attacker can re-exhaust a group after any partial fill, or create new groups. Likelihood is medium-high given that bundle functions are designed for use with aggregators and order-book integrations that fetch offer lists off-chain.

## Recommendation
Add an early-continue guard immediately after computing `unitsToTake` in each of the four bundle loops:

```solidity
if (unitsToTake == 0) continue;
```

This single-line fix, applied before each `try IMidnight(MIDNIGHT).take(...)` call, ensures that fully-consumed offers are skipped without dispatching an external call, eliminating the gas-griefing vector entirely. The fix is consistent with the existing design intent: the `try/catch` already handles the case where an offer is taken by a third party between block construction and execution; skipping zero-unit takes is the analogous guard for the fully-consumed case.

## Proof of Concept
**Minimal manual steps:**
1. Deploy `Midnight` and `MidnightBundles`.
2. As attacker (maker), create N offers in the same group with `maxUnits = 1e18`.
3. As attacker, call `midnight.setConsumed(group, type(uint256).max, maker)` — all N offers are now fully consumed.
4. As victim (taker), call `midnightBundles.buyWithUnitsTargetAndWithdrawCollateral(targetUnits, ..., takes)` where `takes[]` contains the N exhausted offers.
5. Observe: the bundle loop iterates N times, each dispatching `take(..., 0, ...)` which succeeds and returns `(0, 0)`. After N iterations, `require(filledUnits == targetUnits)` reverts with `OutOfOffers()`. The victim's gas is consumed; no tokens are transferred.

**Fuzz/invariant test plan:** Fuzz `targetUnits` and `takes.length` with all offers pre-exhausted via `setConsumed`. Assert that the transaction always reverts with `OutOfOffers()` and that gas consumed scales linearly with `takes.length`, confirming the unbounded loop execution.
