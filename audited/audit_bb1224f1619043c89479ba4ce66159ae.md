### Title
Debt Increase Permitted at Exact Maturity Timestamp Due to Off-by-One in `take()` — (File: src/Midnight.sol)

---

### Summary

The `take()` function in `Midnight.sol` guards against post-maturity debt increases with `block.timestamp <= offer.market.maturity`. At exactly `block.timestamp == market.maturity`, this condition is `true`, allowing a seller's debt to increase. The protocol's own documented invariant explicitly forbids this: *"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."*

---

### Finding Description

**Root cause — off-by-one in the maturity guard:**

In `take()`, the check that prevents post-maturity debt increases is:

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [1](#0-0) 

When `block.timestamp == market.maturity`, the left-hand side evaluates to `true`, so the `|| sellerDebtIncrease == 0` short-circuits and the require passes unconditionally — even when `sellerDebtIncrease > 0`.

**Simultaneously, post-maturity liquidation is unavailable at the same timestamp:**

```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
``` [2](#0-1) 

`block.timestamp > market.maturity` is `false` when `block.timestamp == market.maturity`, so post-maturity liquidation is blocked. Normal-mode liquidation still requires `originalDebt > maxDebt` (i.e., the position must already be unhealthy).

**The result:** At the exact maturity second, a seller can increase their debt (which the protocol forbids), while the post-maturity liquidation path that would make even healthy overdue borrowers liquidatable is not yet open.

**Documented invariants violated:**

The `live_context.json` core invariants state:

> *"debt must not increase after maturity"*
> *"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"* [3](#0-2) 

The whitepaper constants confirm: `"debt_increase_after_maturity_forbidden": true`. [4](#0-3) 

---

### Impact Explanation

At `block.timestamp == market.maturity`:

1. A taker executes a sell offer against a maker, causing `sellerDebtIncrease > 0`.
2. `timeToMaturity = zeroFloorSub(maturity, block.timestamp) = 0`, so `buyerPendingFeeIncrease = 0` — the buyer receives credit with no continuous-fee obligation crystallised.
3. The settlement fee is computed at the 0-day breakpoint (≤ 0.14 bps), the lowest possible rate.
4. The seller now holds debt that should never have been created post-maturity.
5. Post-maturity liquidation (`block.timestamp > maturity`) is unavailable in this same block; normal-mode liquidation only fires if the position is already unhealthy.

The concrete broken invariant is: **debt increased at the maturity boundary in violation of the protocol's explicit rule**. A healthy borrower can take on new debt at maturity with near-zero fees and face no immediate post-maturity liquidation exposure until the next block. [5](#0-4) 

---

### Likelihood Explanation

- Any taker can call `take()` at any block. Targeting `block.timestamp == market.maturity` requires either a validator/builder relationship or simply submitting a transaction and having it land in the right block — a realistic MEV/timing scenario.
- Markets have fixed, publicly known maturities, so the target timestamp is known in advance.
- No privileged access is required; any unprivileged taker can execute this.

---

### Recommendation

Change `<=` to `<` in the maturity guard inside `take()`:

```solidity
// Before (buggy):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (correct):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [1](#0-0) 

This makes the maturity boundary exclusive, consistent with the documented invariant and with the `block.timestamp > market.maturity` check used in `liquidate()` for post-maturity mode.

---

### Proof of Concept

```solidity
// Foundry test sketch
function testDebtIncreaseAtExactMaturity() public {
    uint40 maturity = uint40(block.timestamp + 30 days);
    // ... create market with maturity T ...

    vm.warp(maturity); // block.timestamp == market.maturity exactly

    // Seller (borrower) has no existing credit; sellerDebtIncrease will be > 0
    // take() passes because block.timestamp <= maturity is TRUE
    uint256 units = 1e18;
    (uint256 buyerAssets, uint256 sellerAssets) = midnight.take(
        sellOffer, ratifierData, units, taker, address(0), address(0), ""
    );

    // Assert: seller's debt increased at maturity — invariant violated
    assertGt(midnight.debtOf(id, seller), 0);

    // Assert: post-maturity liquidation is NOT available at this timestamp
    vm.expectRevert(NotLiquidatable.selector);
    midnight.liquidate(market, 0, 0, units, seller, true, receiver, address(0), "");
}
```

**Expected outcome:** `take()` succeeds and increases the seller's debt at `block.timestamp == maturity`, while `liquidate()` with `postMaturityMode = true` reverts — confirming the two checks are inconsistent at the boundary and the invariant is broken.
