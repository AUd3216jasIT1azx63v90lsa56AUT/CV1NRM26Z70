### Title
Maker's `offer.callback` can grief takers in `take()` via front-running — (File: src/Midnight.sol)

---

### Summary

The `take()` function in `Midnight.sol` calls the maker's `offer.callback` (as either `buyerCallback` or `sellerCallback`) using a bare `require(...)` with no try/catch. A malicious maker can deploy a callback contract that succeeds during off-chain simulation but is toggled to revert on-chain by front-running the taker's transaction, causing the taker's `take()` to revert and the taker to lose gas with no trade executed.

---

### Finding Description

In `take()`, the callback address routing is:

```solidity
address buyerCallback = offer.buy ? offer.callback : takerCallback;
address sellerCallback = offer.buy ? takerCallback : offer.callback;
``` [1](#0-0) 

When `offer.buy = true`, `buyerCallback = offer.callback` (the maker's callback). When `offer.buy = false`, `sellerCallback = offer.callback` (the maker's callback). In both cases the maker's callback is invoked with a hard `require`:

```solidity
require(
    IBuyCallback(buyerCallback)
        .onBuy(id, offer.market, buyerAssets, units, buyerPendingFeeIncrease, buyer, buyerCallbackData)
    == CALLBACK_SUCCESS,
    WrongBuyCallbackReturnValue()
);
``` [2](#0-1) 

```solidity
require(
    ISellCallback(sellerCallback)
        .onSell(...)
    == CALLBACK_SUCCESS,
    WrongSellCallbackReturnValue()
);
``` [3](#0-2) 

If either callback reverts or returns a non-`CALLBACK_SUCCESS` value, the entire `take()` transaction reverts. The maker fully controls the `offer.callback` contract and its internal state. A malicious maker can:

1. Deploy a callback contract with a `shouldRevert` flag, initially `false`.
2. Post a signed offer with `offer.callback` pointing to this contract.
3. Monitor the mempool for a taker's `take()` transaction.
4. Front-run the taker by calling `toggle()` on the callback contract (setting `shouldRevert = true`).
5. The taker's `take()` is included after the toggle and reverts at the `require(IBuyCallback(...).onBuy(...) == CALLBACK_SUCCESS, ...)` check.

The `Offer` struct embeds `callback` as a field: [4](#0-3) 

The callback address is fixed in the signed offer and cannot be changed, but the **state** of the callback contract is fully mutable by the maker at any time.

---

### Impact Explanation

The taker loses gas on every griefed `take()` call with no trade executed. Because the offer passes all pre-callback validation checks (ratifier, tick, expiry, consumed limits), off-chain simulation succeeds, giving the taker no warning. A malicious maker can create a persistent "honeypot" offer: it appears valid and simulates successfully, but every on-chain `take()` attempt is front-run and reverted. This constitutes targeted, repeatable service degradation for any taker attempting to fill the offer.

---

### Likelihood Explanation

Front-running is realistic on any public mempool. The maker only needs to:
- Monitor the mempool for `take()` calls referencing their offer.
- Submit a single state-change transaction to the callback contract with a higher gas price.

No privileged access is required. The maker is a normal, unprivileged protocol participant. The attack requires no capital at risk beyond the gas cost of the toggle transaction.

---

### Recommendation

Wrap the maker callback invocations in try/catch blocks so that a reverting maker callback does not revert the taker's transaction. Instead, the trade can proceed (or the callback failure can be logged/emitted) without penalizing the taker:

```solidity
if (buyerCallback != address(0)) {
    try IBuyCallback(buyerCallback).onBuy(...) returns (bytes32 result) {
        require(result == CALLBACK_SUCCESS, WrongBuyCallbackReturnValue());
    } catch { /* handle or emit */ }
}
```

This mirrors the fix recommended in the analogous Connext `PromiseRouter` finding.

---

### Proof of Concept

```solidity
contract MaliciousCallback is IBuyCallback {
    bool public shouldRevert;
    function toggle() external { shouldRevert = !shouldRevert; }
    function onBuy(...) external returns (bytes32) {
        require(!shouldRevert, "griefed");
        return CALLBACK_SUCCESS;
    }
}
```

1. Maker deploys `MaliciousCallback` with `shouldRevert = false`.
2. Maker signs an `Offer` with `offer.callback = address(maliciousCallback)` and `offer.buy = true`.
3. Taker simulates `take(offer, ...)` — simulation succeeds.
4. Taker broadcasts `take(offer, ...)`.
5. Maker observes the pending transaction and broadcasts `maliciousCallback.toggle()` with higher gas.
6. Maker's toggle is mined first; `shouldRevert` becomes `true`.
7. Taker's `take()` is mined next; `IBuyCallback(buyerCallback).onBuy(...)` reverts.
8. `take()` reverts at line 447–452 of `src/Midnight.sol`; taker loses gas, trade does not execute. [5](#0-4)
