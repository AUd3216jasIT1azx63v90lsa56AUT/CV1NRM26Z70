Audit Report

## Title
Unvalidated `liquidatorGate` Allows Market Creation with Non-Implementing Address, Permanently Blocking All Liquidations - (File: src/Midnight.sol)

## Summary
`touchMarket` performs no validation on `market.liquidatorGate`, allowing any unprivileged caller to create a market with `liquidatorGate` set to an arbitrary address (e.g., the loan token) that does not implement `ILiquidatorGate`. Because `liquidate` unconditionally performs an external call to `ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender)` for any non-zero `liquidatorGate`, and a standard ERC20 has no matching selector and no fallback, every liquidation call in that market reverts permanently. A malicious borrower who creates such a market renders their position immune to liquidation, causing unrecoverable losses for lenders.

## Finding Description
**Root cause — missing validation in `touchMarket`:**

`touchMarket` validates maturity, collateral count, sort order, LLTV tiers, and `maxLif` values, but places no constraint on `market.liquidatorGate`:

```solidity
// src/Midnight.sol:755-791
function touchMarket(Market memory market) public returns (bytes32) {
    bytes32 id = toId(market);
    if (marketState[id].tickSpacing == 0) {
        require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
        require(market.collateralParams.length > 0, NoCollateralParams());
        require(market.collateralParams.length <= MAX_COLLATERALS, TooManyCollateralParams());
        // collateral sort / LLTV / maxLif checks ...
        // ← no check on liquidatorGate whatsoever
    }
    return id;
}
```

**Trigger in `liquidate`:**

```solidity
// src/Midnight.sol:597-600
require(
    market.liquidatorGate == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
```

`ILiquidatorGate` is defined as:
```solidity
// src/interfaces/IGate.sol:10-12
interface ILiquidatorGate {
    function canLiquidate(address account) external view returns (bool);
}
```

When `liquidatorGate` is set to a standard ERC20 address (e.g., `loanToken`), the external call to `canLiquidate` targets a contract with no matching selector and no fallback, causing the call to revert. Solidity 0.8.x propagates this revert through the `require`, so `liquidate` always reverts for every caller. There is no `try/catch` around the `canLiquidate` call and no interface-existence check at market creation time.

The protocol's own LIVENESS documentation at line 152 acknowledges: *"If liquidatorGate.canLiquidate reverts or returns false, liquidate reverts."* However, this documents the behavior of a legitimate gate contract reverting — it does not sanction or exclude the attack vector of deliberately setting `liquidatorGate` to a non-implementing address at market creation time. The GATES section (lines 127–132) describes the gate as a mechanism to restrict specific liquidators, not to permanently block all liquidations via a broken address.

**Exploit flow:**
1. Attacker calls `touchMarket` with `market.liquidatorGate = market.loanToken` — passes all existing checks.
2. Attacker (as borrower) opens a position in this market (lenders may not carefully inspect every field of the market struct).
3. Oracle price drops, making the position unhealthy (`debt > maxDebt`).
4. Any liquidator calls `liquidate`; execution reaches line 598, calls `ILiquidatorGate(loanToken).canLiquidate(liquidator)`, which reverts (no matching selector, no fallback on a standard ERC20).
5. `liquidate` reverts for every caller, forever.

**Why existing checks fail:** The `liquidatorGate == address(0)` short-circuit is the only bypass, but the attacker sets it to a non-zero address. No `try/catch` exists around the `canLiquidate` call, and no interface-existence check is performed at market creation time. Market parameters are immutable after creation, so the broken gate cannot be corrected.

## Impact Explanation
All liquidations in the affected market permanently revert. Unhealthy borrower positions accumulate unbounded bad debt that can never be realized or cleared, directly violating the core protocol invariant that unhealthy positions remain liquidatable. Lenders in the market suffer unrecoverable credit losses with no mechanism to force repayment or seize collateral. This constitutes a permanent freeze of lender funds and is a critical integrity failure.

## Likelihood Explanation
The precondition is trivially satisfiable: any unprivileged address can call `touchMarket` with `liquidatorGate = loanToken`. The attacker needs only to be the borrower in the market they create. The attack is repeatable across any loan token that lacks a `canLiquidate` function (i.e., every standard ERC20). It is permanent once the market is created and a position is opened, because market parameters are immutable after creation. Lenders may be deceived into providing offers for what appears to be a legitimately gated market, as the `liquidatorGate` field being set to the loan token address is not immediately obvious as malicious.

## Recommendation
In `touchMarket`, validate that `market.liquidatorGate` either is `address(0)` or implements `ILiquidatorGate` by performing an ERC-165 supportsInterface check, or by making a low-level static call to `canLiquidate` and requiring it to succeed without reverting. Alternatively, restrict market creation to trusted/whitelisted addresses, or add an on-chain registry of approved gate contracts. The simplest fix is to add a check in the market creation block:

```solidity
if (market.liquidatorGate != address(0)) {
    // Verify the gate implements canLiquidate by doing a low-level call
    (bool success,) = market.liquidatorGate.staticcall(
        abi.encodeCall(ILiquidatorGate.canLiquidate, (address(this)))
    );
    require(success, InvalidLiquidatorGate());
}
```

## Proof of Concept
1. Deploy a standard ERC20 token as `loanToken`.
2. Call `touchMarket` with a valid market struct where `liquidatorGate = address(loanToken)`. Observe it succeeds.
3. As the attacker/borrower, supply collateral and take a sell offer to accumulate debt in this market.
4. Manipulate or wait for the oracle price to drop so `debt > maxDebt`.
5. As any liquidator, call `liquidate` on the borrower's position. Observe it reverts with no selector match on the ERC20's `canLiquidate` call.
6. Confirm no liquidator can ever succeed — the position is permanently immune to liquidation.
