Audit Report

## Title
`sellerAssetsToUnits` division-by-zero when `offerPrice == settlementFee` on a buy offer, freezing bundler sell flows for affected offers - (`File: src/periphery/TakeAmountsLib.sol`)

## Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and immediately uses it as the denominator in `mulDivUp(targetSellerAssets, WAD, sellerPrice)` with no guard against `sellerPrice == 0`. When the feeSetter sets the index-6 settlement fee to `MAX_SETTLEMENT_FEE_360_DAYS = 0.005e18` and a buy offer exists at the tick whose price equals exactly `0.005e18`, `sellerPrice` becomes zero and every call to `sellerAssetsToUnits` for that offer panics. Because this call in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` sits at line 286, *before* the `try/catch` block at line 292, the panic propagates and reverts the entire bundler transaction for any user attempting to sell against that offer through the bundler.

## Finding Description

**Root cause — missing zero-denominator guard in `sellerAssetsToUnits`:** [1](#0-0) 

`sellerPrice` is computed as `offerPrice - settlementFee` for buy offers and used directly as the denominator. There is no `require(sellerPrice > 0)` guard. The sibling function `buyerAssetsToUnits` computes the same `sellerPrice` at line 26 and would revert on underflow if `offerPrice < settlementFee`, but the equal-to case (`offerPrice == settlementFee`) silently produces `sellerPrice = 0` in both functions. [2](#0-1) 

**Why `sellerPrice == 0` is reachable:**

- `MAX_SETTLEMENT_FEE_360_DAYS = 0.005e18` is a valid, settable value for index 6. [3](#0-2) 

- `setMarketSettlementFee` enforces `newSettlementFee <= maxSettlementFee(index)`, so `0.005e18` passes the `<=` check. [4](#0-3) 

- For TTM ≥ 360 days, `settlementFee` returns `settlementFeeCbp6 * CBP` directly. [5](#0-4) 

- `tickToPrice` rounds to multiples of `PRICE_ROUNDING_STEP = 1e12`. Since `0.005e18 = 5e15` is a multiple of `1e12`, there exists a tick `t*` (approximately tick 1847, well within `[0, MAX_TICK=5820]`) where `tickToPrice(t*) = 0.005e18` exactly. [6](#0-5) 

- Therefore `sellerPrice = 0.005e18 − 0.005e18 = 0`.

**Why `Midnight.take()` does NOT protect against this:**

In `Midnight.take()`, `sellerAssets = units.mulDivDown(0, WAD) = 0` — no revert. The NatDoc comment "Taking buy offers with price < settlement fee will revert" is accurate only for strict less-than (arithmetic underflow in checked arithmetic); the equal-to case silently produces `sellerAssets = 0` and succeeds. [7](#0-6) 

**Why the bundler `try/catch` does NOT protect against this:**

`TakeAmountsLib.sellerAssetsToUnits` is called at line 286, *before* the `try { IMidnight(MIDNIGHT).take(...) } catch {}` block at line 292. The Solidity 0.8 arithmetic panic from `mulDivUp(..., 0)` is thrown at line 286 and propagates directly to the caller, reverting the entire bundler call. [8](#0-7) 

## Impact Explanation
Every call to `supplyCollateralAndSellWithAssetsTarget` that includes the affected buy offer reverts unconditionally. Users cannot use the bundler to sell against that offer. Direct `Midnight.take()` still works (sellerAssets = 0, though economically useless), but the bundler sell path — the primary interface for asset-targeted selling — is frozen for all affected offers. This is a targeted, repeatable DoS of a core bundler function with no loss of funds but complete denial of service for the affected offer routing.

## Likelihood Explanation
The feeSetter setting the 360-day fee to its maximum (`0.005e18`) is a routine governance action with no indication it is unsafe. The tick whose price equals exactly `0.005e18` is a valid, accessible tick (price is a multiple of `PRICE_ROUNDING_STEP`). Any maker who placed a buy offer at that tick is affected. The condition is deterministic and repeatable: every bundler call including that offer will panic until the fee is lowered or the offer expires. No attacker action is required beyond the normal governance fee-setting and a maker placing an offer at the affected tick.

## Recommendation
Add a zero-denominator guard in `sellerAssetsToUnits` mirroring the intent of the NatDoc comment already present on the function:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
require(sellerPrice > 0, SellerPriceIsZero()); // guard against offerPrice == settlementFee
return offer.buy
    ? targetSellerAssets.mulDivUp(WAD, sellerPrice)
    : targetSellerAssets.mulDivDown(WAD, sellerPrice);
```

This mirrors the existing behavior of `buyerAssetsToUnits` which already reverts on underflow when `offerPrice < settlementFee`, making the equal-to case consistently handled. Alternatively, move the `sellerAssetsToUnits` call inside the `try/catch` in `MidnightBundles`, though the library fix is preferable as it protects all callers.

## Proof of Concept

**Minimal manual steps:**

1. Deploy `Midnight` and `MidnightBundles`.
2. Create a market with TTM ≥ 360 days.
3. feeSetter calls `setMarketSettlementFee(id, 6, 0.005e18)` — passes all validation.
4. Maker places a buy offer at tick `t*` where `tickToPrice(t*) == 0.005e18`.
5. Any unprivileged user calls `supplyCollateralAndSellWithAssetsTarget(...)` with `takes[0]` pointing to that offer.
6. Execution reaches `TakeAmountsLib.sellerAssetsToUnits` → `sellerPrice = 0` → `mulDivUp(target, WAD, 0)` → Solidity 0.8 arithmetic panic → entire bundler call reverts.

**Fuzz test plan:** Fuzz `(tick, settlementFee6)` pairs where `tickToPrice(tick) == settlementFee6` and assert that `sellerAssetsToUnits` does not panic. The invariant `tickToPrice(tick) > settlementFee(id, ttm)` for all valid buy offers should be enforced at the library level.
