Audit Report

## Title
Permissionless Nonce Consumption Allows Suppression of Authorizer's Revocation Signature - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` enforces a sequential monotonic nonce with no caller restriction, allowing any party holding a valid `isAuthorized=true` signature to submit it and consume nonce N even after the authorizer has signed a same-nonce `isAuthorized=false` revocation. The revocation becomes permanently unsubmittable via the EcrecoverAuthorizer, leaving `isAuthorized[A][X] = true` against the authorizer's most recent expressed intent, and exposing the authorizer's assets to unauthorized access until a direct on-chain recovery is performed.

## Finding Description
**Root cause:**

`EcrecoverAuthorizer.setIsAuthorized` at line 26 enforces a purely sequential nonce with no restriction on who may submit:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

The permissionless submission path is confirmed by `testEcrecoverAuthorizerPermissionless` (lines 74–86 of `test/SetIsAuthorizedWithSigTest.sol`), which explicitly demonstrates any address (`otherLender`) can submit a valid signature on behalf of the authorizer. There is no mechanism to mark a nonce as superseded, no ordering preference for later-signed messages, and no way for the authorizer to advance their nonce counter without submitting a full authorization.

**Exploit flow:**

1. Authorizer A (nonce=0) signs `auth_true = {authorizer=A, authorized=X, isAuthorized=true, nonce=0, deadline=T}` and distributes it to a relayer or counterparty.
2. Before `auth_true` is submitted, A changes their mind and signs `auth_false = {..., isAuthorized=false, nonce=0, deadline=T+1}` (same nonce, since `nonce[A]` is still 0).
3. Attacker (holding `auth_true`) calls `ecrecoverAuthorizer.setIsAuthorized(auth_true, sig_true)`:
   - Line 25: deadline check passes (T is still valid).
   - Line 26: `0 == nonce[A]++` passes; `nonce[A]` becomes 1.
   - Lines 46–47: `IMidnight(MIDNIGHT).setIsAuthorized(X, true, A)` executes → `isAuthorized[A][X] = true`.
4. A (or anyone) attempts `ecrecoverAuthorizer.setIsAuthorized(auth_false, sig_false)`:
   - Line 26: `0 != 1` → reverts `InvalidNonce`.
5. Final state: `isAuthorized[A][X] = true` despite A's most recent signed intent being revocation.

**Why existing checks fail:**

- **Deadline check (line 25):** Does not help — `auth_true`'s deadline T is still valid at submission time.
- **Signature check (lines 31–36):** Does not help — `auth_true` is a legitimately signed message from A.
- **No supersession mechanism:** There is no way to mark nonce 0 as "cancelled" or to prefer the later-signed message.

## Impact Explanation
An unprivileged attacker forces `isAuthorized[A][X] = true` against A's explicit revocation intent. X thereby retains the ability to act on A's behalf across all `onBehalf`-gated entry points in `Midnight.sol`: `withdraw` (line 482), `withdrawCollateral` (line 556), `repay` (line 505), `setConsumed` (line 724), `setIsAuthorized` (line 732), and `take` (line 346) — enabling unauthorized movement of A's assets (funds, collateral) until A submits an on-chain direct revocation via `Midnight.setIsAuthorized`. During the window between the forced authorization and A's on-chain recovery, X can drain A's deposited funds or collateral.

## Likelihood Explanation
Preconditions are realistic in any relayer or meta-transaction workflow: (1) A signed and distributed an `isAuthorized=true` authorization that has not yet been submitted on-chain; (2) A subsequently signed a same-nonce `isAuthorized=false` revocation. The attacker needs only to have received or intercepted the first signature — a normal outcome when signatures are handed to relayers or counterparties. The attack requires no special privilege, no capital, and is repeatable for every nonce slot. The `testEcrecoverAuthorizerPermissionless` test confirms the permissionless submission path is an intentional design feature, making this a realistic attack surface.

## Recommendation
Add a timestamp or sequence number to the `Authorization` struct so that later-signed messages can be identified and preferred. Alternatively, implement a nonce cancellation mechanism (e.g., `invalidateNonce(uint256 upToNonce)`) that allows the authorizer to advance their nonce counter without submitting a full authorization, rendering any pending signatures with lower nonces invalid. A simpler mitigation is to restrict `setIsAuthorized` so that only the authorizer (or an already-authorized account) can submit the transaction, eliminating the permissionless front-running vector — though this trades off the meta-transaction UX benefit.

## Proof of Concept
Minimal Foundry test extending `EcrecoverAuthorizerTest`:

```solidity
function testNonceSuppressionAttack() public {
    vm.prank(borrower);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

    // Step 1: borrower signs auth_true (nonce=0)
    Authorization memory authTrue = makeAuthorization(borrower, lender, true);
    Signature memory sigTrue = signAuthorization(authTrue, borrower);

    // Step 2: borrower signs auth_false (nonce=0, same nonce — not yet consumed)
    Authorization memory authFalse = makeAuthorization(borrower, lender, false);
    Signature memory sigFalse = signAuthorization(authFalse, borrower);

    // Step 3: attacker (otherLender) front-runs with auth_true
    vm.prank(otherLender);
    ecrecoverAuthorizer.setIsAuthorized(authTrue, sigTrue);
    assertEq(midnight.isAuthorized(borrower, lender), true);
    assertEq(ecrecoverAuthorizer.nonce(borrower), 1);

    // Step 4: auth_false is now unsubmittable — nonce consumed
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(authFalse, sigFalse);

    // Final state: isAuthorized=true despite borrower's revocation intent
    assertEq(midnight.isAuthorized(borrower, lender), true);
}
```
