Audit Report

## Title
Unprivileged Attacker Can Drain Any Approved `IBuyCallback` Contract via Unvalidated `takerCallback` Payer Assignment - (File: src/Midnight.sol)

## Summary
In `Midnight.take`, when processing a sell offer (`offer.buy = false`), the `takerCallback` parameter is assigned directly as `buyerCallback` and consequently as `payer` with no check that it equals `msg.sender`, `taker`, or is authorized by either. An attacker can supply any victim contract as `takerCallback`; if that contract has approved Midnight for the loan token and returns `CALLBACK_SUCCESS` from `onBuy`, Midnight will pull `buyerAssets` from the victim and credit the attacker's position at zero personal cost.

## Finding Description

**Root cause — `src/Midnight.sol` lines 420–456:**

When `offer.buy = false`:

```solidity
address buyerCallback = offer.buy ? offer.callback : takerCallback; // line 420: fully attacker-supplied
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender); // line 422: payer = victim
```

The only authorization check in `take` is at line 346:

```solidity
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());
```

This validates the caller's right to act as `taker`, but says nothing about `takerCallback`. After the callback returns `CALLBACK_SUCCESS`, tokens are pulled from `payer` (the victim):

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // line 455
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets); // line 456
```

**Exploit flow:**
1. Victim contract `V` has `loanToken.approve(midnight, type(uint256).max)` and implements `IBuyCallback`, returning `CALLBACK_SUCCESS` from `onBuy` for any call originating from Midnight (checking only `msg.sender == midnight`, not `buyer`).
2. Attacker calls `midnight.take(sellOffer, ratifierData, units, attacker, receiver, address(V), callbackData)`.
3. `buyerCallback = address(V)`, `payer = address(V)`.
4. Midnight calls `V.onBuy(id, market, buyerAssets, units, pendingFeeIncrease, taker, callbackData)` → `V` returns `CALLBACK_SUCCESS`.
5. Midnight executes `safeTransferFrom(loanToken, V, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, V, receiver, sellerAssets)`.
6. `V` loses `buyerAssets` in loan tokens; attacker's position gains credit for `units` at zero personal cost.

**Why existing checks fail:**

The `onBuy` interface (`src/interfaces/ICallbacks.sol` line 9) passes `buyer` as a parameter, but the protocol does not require the callback contract to verify `buyer == address(this)` or any other ownership assertion. A contract that only guards `msg.sender == midnight` — the natural trust boundary for a Midnight integrator — cannot distinguish a legitimate invocation from an attacker-injected one. The protocol itself is the only place where this authorization can be enforced, and it does not enforce it.

## Impact Explanation
Direct theft of loan tokens from any contract that (a) has approved Midnight for the loan token and (b) implements `IBuyCallback` returning `CALLBACK_SUCCESS` without verifying it was legitimately designated as the callback. The attacker acquires protocol credit (`buyerCreditIncrease` units) for free. The victim's token balance decreases by `buyerAssets = units.mulDivUp(buyerPrice, WAD)`, unbounded up to the victim's balance and approval. The loss is immediate and unrecoverable.

## Likelihood Explanation
All three preconditions are realistic in a live ecosystem: (1) any router, aggregator, or vault interacting with Midnight will have approved the loan token; (2) any legitimate Midnight buy-callback integrator will return `CALLBACK_SUCCESS` from `onBuy` when called by Midnight, as the protocol is the expected trust anchor; (3) valid sell offers exist in any active market. The attack requires no special privilege, is repeatable up to the victim's balance and approval, and costs the attacker only gas.

## Recommendation
Add an authorization check on `takerCallback` before using it as `payer`. Specifically, when `offer.buy = false`, require that `takerCallback == address(0)` or `takerCallback == msg.sender` or `isAuthorized[taker][takerCallback]`. This mirrors the existing `TakerUnauthorized` guard and ensures the designated payer has consented to the role. Alternatively, always default `payer` to `msg.sender` when `takerCallback` is supplied by the taker, and only use the callback address as payer when it is the maker's own callback (`offer.callback`).

## Proof of Concept
1. Deploy `VictimCallback` implementing `IBuyCallback`: `onBuy` checks `msg.sender == midnight` and returns `CALLBACK_SUCCESS`; the contract holds loan tokens and has `loanToken.approve(midnight, type(uint256).max)`.
2. Deploy a valid sell offer from `maker` with `offer.buy = false`.
3. Attacker calls `midnight.take(sellOffer, ratifierData, units, attacker, attacker, address(VictimCallback), "")`.
4. Observe: `VictimCallback` loses `buyerAssets` loan tokens; attacker's `position[id][attacker].credit` increases by `buyerCreditIncrease`; attacker paid nothing.
5. Repeat until `VictimCallback`'s balance or approval is exhausted.
