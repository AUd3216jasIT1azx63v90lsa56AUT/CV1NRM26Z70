### Title
Fee-on-Transfer Loan Token in `repay()` Inflates `withdrawable` Without Matching Token Balance, Destroying Last Lender's Credit — (`src/libraries/SafeTransferLib.sol` / `src/Midnight.sol`)

### Summary
`repay()` unconditionally increments `marketState[id].withdrawable` by the caller-supplied `units` parameter, then calls `SafeTransferLib.safeTransferFrom` which only checks the ERC-20 boolean return value and never measures the actual tokens received. When the loan token charges a transfer fee, the contract receives `units - fee` per call but credits `units` to `withdrawable`, creating a cumulative shortfall of `N * fee` after N repayments. When lenders subsequently call `withdraw()`, the contract attempts to transfer the full face-value amount out; the last lender's call reverts because the real token balance is exhausted, permanently destroying their credit without any `lossFactor` update.

**Exploit flow:**

1. Market creator creates a market whose `loanToken` is a fee-on-transfer token with fee rate `f` (e.g., 1%).
2. N borrowers each hold `D` debt units (acquired via `take()`).
3. Each borrower calls `repay(market, D, self, address(0), "")`. Per call: `withdrawable += D`, but contract receives only `D*(1-f)`. After N calls: `withdrawable = N*D`, actual balance = `N*D*(1-f)`, shortfall = `N*D*f`.
4. All borrowers' debt = 0; `totalUnits = withdrawable = N*D` (invariant holds in accounting).
5. Lenders call `withdraw()` in order. Each successful call sends `D` tokens out. After `floor(N*(1-f))` successful withdrawals, the contract balance is zero.
6. The remaining lenders' `withdraw()` calls revert inside `safeTransfer` (ERC-20 transfer of more than balance fails), even though their `credit` and `withdrawable` accounting show they are owed tokens.
7. No `lossFactor` is ever updated; the destroyed credit is not reflected anywhere in protocol state.

**Why existing checks fail:**

- `SafeTransferLib.safeTransferFrom` passes as long as the token returns `true` — fee-on-transfer tokens do exactly that.
- The Certora rule `repayIncreasesWithdrawable` asserts `withdrawableAfter == withdrawableBefore + units` — this holds in the EVM state but the rule's model does not model fee-on-transfer tokens; the prover assumes a standard ERC-20.
- The `totalUnitsEqualsSumNegativeDebtPlusWithdrawable` invariant holds in accounting but the real token balance is not part of the invariant.
- There is no `require(balanceAfter == balanceBefore + units)` guard anywhere in `repay()`.

### Impact Explanation
Lenders whose `credit > 0` and whose share of `withdrawable` exceeds the remaining real token balance cannot redeem their credit. Their tokens are permanently locked (the contract has no recovery path) and no `lossFactor` update compensates them. This is concrete credit destruction and fund freeze proportional to `N * fee`, where N is the number of repayments and `fee` is the per-transfer fee.

### Likelihood Explanation
Preconditions: (1) a fee-on-transfer ERC-20 is used as `loanToken` — this is a known token pattern (e.g., tokens with buy/sell taxes); (2) at least one borrower repays. The market creator role is unprivileged and permissionless. No special timing, oracle manipulation, or admin key is required. The bug is deterministic and repeatable: every repayment call with a fee-on-transfer token widens the shortfall by exactly `fee`.

### Recommendation
In `repay()`, measure the actual tokens received by comparing balance before and after the transfer, and use that delta — not the caller-supplied `units` — to increment `withdrawable`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
// use `received` instead of `units` for withdrawable, or revert if received != units
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported by adding a post-transfer balance assertion (`require(received == units, FeeOnTransferNotSupported())`), which at minimum prevents silent accounting corruption.

The same pattern should be audited in `liquidate()` which also calls `safeTransferFrom` and increments `withdrawable`. [4](#0-3) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
// FeeOnTransferToken: transferFrom deducts 1% fee from recipient

contract FeeOnTransferRepayTest is Test {
    Midnight midnight;
    FeeOnTransferToken loanToken; // 1% fee on every transfer
    Market memory market;
    bytes32 id;

    address lender1; address lender2;
    address borrower1; address borrower2;
    uint256 constant UNITS = 1000e18;

    function setUp() public {
        // deploy Midnight, create market with loanToken = FeeOnTransferToken
        // lender1 and lender2 each supply UNITS credit via take()
        // borrower1 and borrower2 each take UNITS debt
    }

    function testFeeOnTransferInflatesWithdrawable() public {
        // Both borrowers repay their full debt
        vm.prank(borrower1);
        midnight.repay(market, UNITS, borrower1, address(0), "");
        vm.prank(borrower2);
        midnight.repay(market, UNITS, borrower2, address(0), "");

        // Accounting: withdrawable == 2*UNITS, totalUnits == 2*UNITS
        assertEq(midnight.withdrawable(id), 2 * UNITS);
        // Actual balance: 2*UNITS*(1-0.01) = 1980e18
        assertLt(loanToken.balanceOf(address(midnight)), 2 * UNITS);

        // lender1 withdraws successfully
        vm.prank(lender1);
        midnight.withdraw(market, UNITS, lender1, lender1);

        // lender2's withdraw reverts — contract balance insufficient
        vm.prank(lender2);
        vm.expectRevert(); // safeTransfer fails: balance < UNITS
        midnight.withdraw(market, UNITS, lender2, lender2);

        // Assert: lender2 still has credit but cannot redeem it
        assertEq(midnight.creditOf(id, lender2), UNITS);
        // Assert: no lossFactor update occurred
        assertEq(midnight.lossFactor(id), 0);
    }
}
```

**Expected assertions:**
- `withdrawable` after 2 repays = `2 * UNITS` (inflated)
- `loanToken.balanceOf(midnight)` < `2 * UNITS` (real shortfall = `2 * fee`)
- `lender2.withdraw()` reverts
- `creditOf(id, lender2)` = `UNITS` (credit not zeroed, not compensated)
- `lossFactor(id)` = 0 (no bad-debt accounting triggered)
