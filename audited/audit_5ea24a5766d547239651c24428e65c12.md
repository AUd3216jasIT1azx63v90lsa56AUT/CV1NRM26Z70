Audit Report

## Title
Fee-on-Transfer Collateral Token Inflates Stored Collateral, Enabling Uncollateralized Borrowing - (File: src/Midnight.sol)

## Summary
`supplyCollateral` writes phantom collateral to storage before executing the token transfer, and `SafeTransferLib.safeTransferFrom` performs no balance-delta check. A fee-on-transfer token that returns `true` while delivering zero tokens leaves the protocol with inflated `_position.collateral` values. `isHealthy` computes `maxDebt` from this phantom stored value, allowing an attacker to borrow real loan tokens against zero actual collateral, creating immediate bad debt at the expense of lenders.

## Finding Description
**CEI violation in `supplyCollateral` (`src/Midnight.sol` lines 531–545):**

State is mutated before the transfer executes. `_position.collateral[collateralIndex]` is set to `oldCollateral + assets` at line 533, and the `collateralBitmap` bit is set at lines 535–537. The actual token pull via `SafeTransferLib.safeTransferFrom` occurs at line 545, after all storage writes. [1](#0-0) 

**`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34):**

Only two failure conditions are checked: the low-level call reverted, or the return value decoded to `false`. There is no balance-delta check. A 100%-fee-on-transfer token returns `true` and emits a `Transfer` event for `assets` while moving 0 tokens to `address(this)`. Both checks pass silently. [2](#0-1) 

**`isHealthy` (`src/Midnight.sol` lines 944–959):**

`_position.collateral[i]` is read directly from storage — the phantom value — and multiplied by the oracle price and LLTV to produce `maxDebt`. With a valid oracle and any allowed LLTV, `maxDebt` is positive and `isHealthy` returns `true`. [3](#0-2) 

**Health gate in `take` (`src/Midnight.sol` line 476):**

This is the only post-borrow health check for the seller. It passes because `isHealthy` uses the phantom stored collateral. [4](#0-3) 

**`touchMarket` (`src/Midnight.sol` lines 755–791):**

Permissionless market creation validates only LLTV tier, `maxLif`, sort order, and collateral count. No token whitelist or fee-on-transfer guard exists. [5](#0-4) 

**Exploit flow:**
1. Attacker deploys `FeeToken` — `transferFrom` returns `true`, transfers 0 tokens.
2. Attacker calls `touchMarket(market)` with `FeeToken` as the sole collateral token and a valid LLTV/maxLif pair.
3. A victim lender places a buy offer (supplies loan tokens) in this market.
4. Attacker calls `supplyCollateral(market, 0, 1e18, attacker)`: `_position.collateral[0]` becomes `1e18`, bit 0 of `collateralBitmap` is set, `FeeToken.transferFrom` returns `true`, 0 tokens arrive.
5. Attacker takes the lender's buy offer: `sellerPos.debt += units`, `isHealthy` returns `true` (phantom collateral covers debt), borrow succeeds.
6. Attacker walks away with loan tokens; position has `debt > 0`, `collateral[0] = 1e18` in storage, `token.balanceOf(midnight) = 0`.

## Impact Explanation
The attacker holds real loan tokens with zero actual collateral backing. The position is immediately insolvent: `token.balanceOf(midnight)` for the collateral is 0, so liquidation seizes nothing. The full debt becomes bad debt, socialised across all lenders in the market via the `lossFactor` mechanism. This constitutes direct theft of lender funds with no capital at risk for the attacker.

## Likelihood Explanation
No privileged access is required. Deploying a fee-on-transfer token is trivial, and `touchMarket` is permissionless with no token whitelist. The practical barrier is attracting a victim lender, which can be overcome by seeding initial liquidity to attract other lenders, or by deploying an upgradeable proxy collateral token that activates the fee only after market creation and initial liquidity is deposited. The attack is repeatable across any number of markets.

## Recommendation
1. **Enforce CEI**: Move `SafeTransferLib.safeTransferFrom` before any storage writes in `supplyCollateral`, or use a balance-delta check: record `balanceBefore`, execute the transfer, then assert `balanceAfter - balanceBefore >= assets` and use the actual delta as the credited amount.
2. **Balance-delta accounting**: Credit `_position.collateral[collateralIndex]` with the actual received amount (`balanceAfter - balanceBefore`) rather than the requested `assets`, which also handles partial-fee tokens.
3. **Optional token whitelist**: Governance-controlled whitelisting of collateral tokens would provide defense-in-depth, though the balance-delta fix alone closes the vulnerability.

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];
                uint256 price = IOracle(collateralParam.oracle).price();
                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                    .mulDivDown(collateralParam.lltv, WAD);
                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
