### Title
Unconditional zero-value `safeTransfer` on collateral token blocks bad-debt realization when token reverts on zero transfers - (File: src/Midnight.sol)

### Summary

`Midnight.liquidate` explicitly supports calling with `seizedAssets=0` and `repaidUnits=0` to realize bad debt without transferring tokens (documented in NatSpec at line 577). However, `SafeTransferLib.safeTransfer(collateralToken, receiver, 0)` is called unconditionally at line 696 regardless of whether `seizedAssets` is zero. If the collateral token reverts on zero-value transfers â€” a known behavior in several non-standard ERC20s â€” the entire liquidation transaction reverts, permanently blocking bad-debt realization for that borrower.

**Root cause:** No `if (seizedAssets > 0)` guard before the `safeTransfer` call at line 696, contradicting the NatSpec promise at lines 577â€“579 that `(0, 0)` inputs realize bad debt "with 0 token transferred."

**Attacker-controlled inputs:** Any unprivileged caller can supply `seizedAssets=0, repaidUnits=0`. The collateral token is fixed at market creation; no attacker action is needed to make it revert on zero â€” it is an intrinsic property of the token.

**Existing protections reviewed and insufficient:** The `atMostOneNonZero` check (line 595) explicitly permits both-zero. The `if (repaidUnits > 0 || seizedAssets > 0)` guard (line 643) correctly skips accounting but does not guard the transfer. No other check prevents the unconditional `safeTransfer(token, receiver, 0)`.

### Impact Explanation

Bad-debt realization via `liquidate(0, 0, ...)` is permanently blocked for any market whose collateral token reverts on zero-value transfers. The `lossFactor` and `totalUnits` are never updated, so lenders' credit is never written down to reflect the loss. The borrower's position remains in a state where it is technically liquidatable but every attempt to realize the bad debt reverts. This violates the core invariant: *unhealthy positions remain liquidatable*.

### Likelihood Explanation

Preconditions: (1) a market is created with a collateral token that reverts on `transfer(addr, 0)` â€” a documented behavior of several deployed ERC20s; (2) a borrower accumulates bad debt in that market (collateral value at `maxLif` < debt). Both conditions are reachable without any privileged action. Once the collateral token is set at market creation it cannot be changed, making the block permanent and repeatable on every liquidation attempt.

### Recommendation

Guard both external transfer calls with non-zero checks, consistent with the `if (repaidUnits > 0 || seizedAssets > 0)` pattern already used for accounting:

```solidity
// line 696
if (seizedAssets > 0) {
    SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
}

// line 717
if (repaidUnits > 0) {
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
}
```

This matches the documented intent ("0 token transferred") and eliminates the revert path for zero-value calls.
