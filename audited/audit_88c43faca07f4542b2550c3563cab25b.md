### Title
Fee-on-transfer collateral token causes receiver to receive less than accounting deduction in `withdrawCollateral` - (File: src/Midnight.sol)

### Summary
`withdrawCollateral` deducts exactly `assets` from `position[id][onBehalf].collateral[collateralIndex]` and then calls `SafeTransferLib.safeTransfer(collateralToken, receiver, assets)`. When the collateral token is a fee-on-transfer token with fee rate `f`, the receiver receives only `assets*(1-f)` while the accounting records a full `assets` deduction. The same asymmetry exists on `supplyCollateral`, where the protocol records `assets` credited but only receives `assets*(1-f)` tokens, meaning the contract's actual token balance is already less than its accounting, and the withdrawal compounds the loss to the user.

### Finding Description
**Exact code path:**

`src/Midnight.sol` `withdrawCollateral` (lines 549–573):
```
_position.collateral[collateralIndex] = UtilsLib.toUint128(newCollateral); // deducts `assets`
require(isHealthy(...), UnhealthyBorrower());                               // health check on reduced accounting
SafeTransferLib.safeTransfer(collateralToken, receiver, assets);            // sends `assets`, fee taken in-flight
```

`src/libraries/SafeTransferLib.sol` `safeTransfer` (lines 12–22):
```solidity
(bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
```
The library only checks that `transfer` returns `true`; it performs no balance-before/balance-after check and cannot detect that the recipient received fewer tokens than `value`.

**Root cause:** Neither `withdrawCollateral` nor `SafeTransferLib.safeTransfer` verifies the actual token delta at `receiver`. The protocol assumes a 1:1 relationship between the nominal `assets` argument and the tokens that arrive at `receiver`.

**Attacker-controlled inputs:** `assets` (any nonzero value), `receiver` (any address). No privileged role required; the caller need only be `onBehalf` or an authorized operator.

**Exploit flow:**
1. A market is created with a fee-on-transfer collateral token (fee rate `f > 0`). Market creation (`touchMarket`) validates only LLTV, `maxLif`, sort order, and address non-zero — no token-type restriction.
2. Borrower calls `supplyCollateral(market, index, assets, onBehalf)`. Protocol records `assets` in accounting but receives only `assets*(1-f)` tokens (supply-side loss, not the scoped impact but relevant to solvency).
3. Borrower calls `withdrawCollateral(market, index, assets, onBehalf, receiver)`.
   - Accounting: `position[id][onBehalf].collateral[index] -= assets` ✓
   - Health check: passes (collateral was reduced by `assets` in accounting) ✓
   - Transfer: `safeTransfer(collateralToken, receiver, assets)` — token contract deducts `f*assets` as fee; receiver receives `assets*(1-f)`.
4. Result: accounting shows `assets` withdrawn; receiver holds `assets*(1-f)`; `f*assets` is permanently lost.

**Why existing checks fail:** `isHealthy` operates on the accounting value, not on actual token balances, so it cannot detect the discrepancy. `SafeTransferLib.safeTransfer` only checks the boolean return of `transfer`, not the received amount. There is no balance snapshot or received-amount assertion anywhere in the withdrawal path.

### Impact Explanation
For every `withdrawCollateral` call with a fee-on-transfer collateral token at fee rate `f`, the receiver receives `assets*(1-f)` tokens while their position is debited `assets`. The shortfall `f*assets` is irrecoverable — it is neither held by the protocol nor returned to the user. Over repeated withdrawals the cumulative loss scales linearly with `f` and total withdrawn volume. The protocol's internal collateral accounting also diverges from its actual token balance (compounded by the supply-side under-receipt), eventually making the contract insolvent for that collateral token.

### Likelihood Explanation
**Preconditions:**
- A market must use a fee-on-transfer ERC20 as a collateral token. Market creation imposes no restriction on token type.
- The borrower must have a healthy position with nonzero collateral of that token.

**Feasibility:** Any unprivileged user can create a market with a fee-on-transfer collateral token (via `touchMarket`). The borrower triggering the loss is the same user who suffers it — no adversarial third party is required. The condition is met whenever such a market exists and a borrower withdraws collateral. It is repeatable on every `withdrawCollateral` call.

### Recommendation
In `withdrawCollateral`, record the receiver's token balance before the transfer and assert the delta equals `assets`:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(receiver);
SafeTransferLib.safeTransfer(collateralToken, receiver, assets);
require(IERC20(collateralToken).balanceOf(receiver) - balanceBefore == assets, UnexpectedTokenDelta());
```

Alternatively, apply the same balance-check pattern in `supplyCollateral` (using `address(this)` as the measured recipient) and document that fee-on-transfer tokens are unsupported, reverting on any discrepancy at deposit time so the bad state is never entered.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {Market, CollateralParams} from "src/interfaces/IMidnight.sol";

// Minimal fee-on-transfer ERC20: charges 10% on every transfer/transferFrom
contract FeeToken is ERC20 {
    constructor() ERC20("FeeToken", "FT") { _mint(msg.sender, 1e24); }
    function _transfer(address from, address to, uint256 amount) internal override {
        uint256 fee = amount / 10;
        super._transfer(from, address(0xdead), fee);   // burn fee
        super._transfer(from, to, amount - fee);
    }
}

contract FeeOnTransferCollateralTest is Test {
    Midnight midnight;
    FeeToken feeToken;
    ERC20    loanToken;
    address  borrower = address(0xB0B);
    address  receiver = address(0xBEEF);

    function setUp() public { /* deploy midnight, loanToken, oracle, create market */ }

    function testWithdrawCollateralFeeOnTransfer() public {
        uint256 assets = 1000e18;
        // 1. Supply collateral: protocol records `assets`, receives assets*0.9
        vm.startPrank(borrower);
        feeToken.approve(address(midnight), assets);
        midnight.supplyCollateral(market, 0, assets, borrower);

        // 2. Withdraw collateral
        uint256 receiverBalBefore = feeToken.balanceOf(receiver);
        midnight.withdrawCollateral(market, 0, assets, borrower, receiver);
        uint256 receiverBalAfter  = feeToken.balanceOf(receiver);

        uint256 received = receiverBalAfter - receiverBalBefore;
        // Accounting deducted `assets`, but receiver got assets*(1-0.1) = assets*0.9
        assertLt(received, assets, "receiver got less than assets");
        assertEq(received, assets * 9 / 10, "receiver got assets*(1-f)");
        // Collateral accounting shows 0 remaining, but receiver lost f*assets
        assertEq(midnight.collateral(id, borrower, 0), 0, "accounting fully deducted");
    }
}
```

**Expected assertions:**
- `received < assets` — receiver got fewer tokens than accounting deducted.
- `received == assets * 9 / 10` — shortfall equals exactly `f * assets`.
- `midnight.collateral(id, borrower, 0) == 0` — accounting is fully decremented, confirming the discrepancy is not a rounding artifact. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
