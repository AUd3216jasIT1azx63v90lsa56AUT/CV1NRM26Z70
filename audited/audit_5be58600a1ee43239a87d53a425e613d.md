### Title
Allowlisted Proxy Token Upgrade Post-Allowlist Breaks Pool Accounting and Permanently Locks Staker Principal - (File: src/ConfidencePoolFactory.sol)

### Summary
The `allowedStakeToken` allowlist in `ConfidencePoolFactory` is enforced only at `createPool` time. If an allowlisted upgradeable (proxy) ERC20 token is later upgraded to introduce negative rebasing or fee-on-outgoing-transfer, existing pools are permanently affected. The balance-diff defense in `stake()` and `contributeBonus()` only protects at deposit time and does not guard against post-deposit token behavior changes, leaving staker principal at risk of being locked or lost.

### Finding Description
`ConfidencePoolFactory.createPool()` enforces the allowlist with a single point-in-time check:

```solidity
if (!allowedStakeToken[stakeToken]) revert StakeTokenNotAllowed();
```

The factory's own natspec acknowledges this is a snapshot check: *"Checked only at `createPool` time, so de-listing a token does not affect existing pools."*

`ConfidencePool.stake()` contains a balance-diff guard described as *"defense-in-depth against the factory allowlist admitting a fee-on-transfer or rebasing token (governance error, or a proxy-token upgrade post-allowlist)"*:

```solidity
uint256 balanceBefore = stakeToken.balanceOf(address(this));
stakeToken.safeTransferFrom(msg.sender, address(this), amount);
uint256 balanceAfter = stakeToken.balanceOf(address(this));
uint256 received = balanceAfter > balanceBefore ? balanceAfter - balanceBefore : 0;
```

This defense is **incomplete**. It correctly records the actual received amount at deposit time, but it provides no protection against token behavior changes introduced *after* deposits are made:

1. **Negative rebasing post-deposit**: The pool's `stakeToken.balanceOf(address(this))` decreases over time while `totalEligibleStake`, `eligibleStake[u]`, and `snapshotTotalStaked` remain at their original values. All claim and sweep paths (`claimSurvived`, `claimExpired`, `claimCorrupted`, `claimAttackerBounty`, `sweepUnclaimedCorrupted`, `sweepUnclaimedBonus`) transfer based on tracked accounting, not live balance. When the live balance falls below tracked liabilities, `safeTransfer` reverts for later claimants.

2. **Fee-on-outgoing-transfer post-deposit**: Every `safeTransfer` out of the pool silently underpays the recipient relative to their tracked `eligibleStake`, with no revert and no accounting correction.

Neither scenario is caught by the deposit-time balance-diff check, and neither is documented as an accepted trade-off in `docs/DESIGN.md`. Section 12 of DESIGN.md states *"A true stuck-state would require a defective stake token that freezes the pool's own balance, which the factory allowlist exists to exclude"* — but the allowlist does not exclude proxy tokens that become defective after allowlisting.

### Impact Explanation
If an allowlisted proxy token is upgraded post-allowlist to introduce negative rebasing:

- The pool's actual token balance decreases continuously while all tracked accounting (`totalEligibleStake`, `snapshotTotalStaked`, `eligibleStake[u]`) remains at original deposit values.
- First claimants drain the remaining balance; later claimants' `safeTransfer` calls revert permanently.
- Stakers who cannot claim lose their entire principal with no recovery path — the pool has no mechanism to detect or respond to a balance shortfall.

For fee-on-outgoing-transfer: every claimant receives less than their tracked entitlement, with the shortfall silently absorbed, and no revert to signal the accounting error.

### Likelihood Explanation
Many widely-used ERC20 tokens (USDC, USDT, and others) are proxy-upgradeable. A factory owner acting in good faith may allowlist such a token. The token's own upgrade authority (a separate entity from the factory owner) can then introduce behavior changes that affect all existing pools. The code itself acknowledges this exact scenario in the `stake()` comment, confirming it is a realistic operational path, not a theoretical one.

### Recommendation
1. Add a warning in `setStakeTokenAllowed` natspec explicitly prohibiting proxy/upgradeable tokens from being allowlisted, mirroring the external report's recommendation.
2. Consider adding a code-level check at `createPool` time to detect whether the token address contains an EIP-1967 implementation slot (a heuristic proxy indicator), reverting if so.
3. Document in `docs/DESIGN.md` that the balance-diff defense covers only deposit-time fee-on-transfer and does not protect against post-deposit token upgrades, so the allowlist governance process must treat proxy tokens as categorically ineligible.

### Proof of Concept

1. Factory owner calls `setStakeTokenAllowed(proxyUSDC, true)` — allowlists a UUPS-upgradeable ERC20.
2. Sponsor calls `createPool(agreement, proxyUSDC, expiry, minStake, recoveryAddress, accounts)` — pool is created.
3. Ten stakers each call `stake(1000e18)`. The balance-diff check passes (no fee yet); `eligibleStake[u] = 1000e18` for each, `totalEligibleStake = 10_000e18`, pool balance = `10_000e18`.
4. Token owner upgrades `proxyUSDC` to introduce a 1% daily negative rebase.
5. After 30 days, pool's `stakeToken.balanceOf(pool) ≈ 7_397e18` (compounded 1% daily loss), while `totalEligibleStake` remains `10_000e18`.
6. First ~7 stakers call `claimSurvived()` / `claimExpired()` and drain the remaining balance.
7. The remaining ~3 stakers call `claimSurvived()` / `claimExpired()`: `stakeToken.safeTransfer(msg.sender, 1000e18)` reverts because the pool balance is 0.
8. Those stakers permanently lose their `1000e18` principal with no recovery path in the scoped contracts. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
