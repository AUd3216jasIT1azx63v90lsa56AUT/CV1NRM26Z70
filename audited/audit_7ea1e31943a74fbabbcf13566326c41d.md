Audit Report

## Title
Taker-Supplied `takerCallback` Becomes Unchecked `payer` for Sell Offers, Enabling Token Drain from Victim Callback Contracts - (File: src/Midnight.sol)

## Summary
In `take()`, for sell offers (`offer.buy == false`), the `payer` for `safeTransferFrom` calls is derived directly from the attacker-controlled `takerCallback` parameter. No authorization check binds `takerCallback` to the taker or to `msg.sender`. Any contract that has approved Midnight for the loan token and implements `IBuyCallback.onBuy` returning `CALLBACK_SUCCESS` without validating the `buyer` argument can be designated as `payer` by an attacker, causing its tokens to be drained.

## Finding Description

**Payer derivation (src/Midnight.sol lines 420–422):**
```solidity
address buyerCallback = offer.buy ? offer.callback : takerCallback;
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
```
For `offer.buy == false` and `takerCallback != address(0)`: `buyerCallback = takerCallback` and `payer = takerCallback` — both fully attacker-controlled.

**Token pulls (lines 455–456):**
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
```
Tokens are pulled from `payer = takerCallback` with no consent check beyond the ERC-20 allowance.

**Callback gate (lines 445–452):**
```solidity
if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback).onBuy(..., buyer, buyerCallbackData) == CALLBACK_SUCCESS,
        WrongBuyCallbackReturnValue()
    );
}
```
The callback is invoked on the attacker-supplied address. If the victim contract only gates on `msg.sender == midnight` (a common and reasonable pattern), it returns `CALLBACK_SUCCESS` and the subsequent `safeTransferFrom` drains it.

**`hashOffer` does not commit `takerCallback` (src/ratifiers/libraries/HashLib.sol lines 118–138):**
```solidity
function hashOffer(Offer memory offer) internal pure returns (bytes32) {
    return keccak256(abi.encode(
        OFFER_TYPEHASH, hashMarket(offer.market), offer.buy, offer.maker,
        offer.start, offer.expiry, offer.tick, offer.group,
        offer.callback, keccak256(offer.callbackData),
        offer.receiverIfMakerIsSeller, offer.ratifier, offer.reduceOnly,
        offer.maxUnits, offer.maxAssets
    ));
}
```
`takerCallback` and `takerCallbackData` are runtime parameters of `take()` and are never included in the signed hash, so the ratifier cannot detect substitution.

**Only authorization check in `take()` (line 346):**
```solidity
require(taker == msg.sender || isAuthorized[taker][msg.sender], TakerUnauthorized());
```
There is no corresponding check that `takerCallback` is authorized by `taker` or equals `msg.sender`. The taker authorization check does not extend to the callback address.

**Exploit flow:**
1. Maker signs a valid sell offer with `offer.callback = address(0)`.
2. Attacker identifies `victimContract`: a Midnight-aware periphery contract that (a) has approved Midnight to spend its loan tokens, and (b) implements `IBuyCallback.onBuy` returning `CALLBACK_SUCCESS` gated only on `msg.sender == midnight`.
3. Attacker calls `take(offer, ratifierData, units, attacker, receiver, victimContract, takerCallbackData)`.
4. `buyerCallback = victimContract`, `payer = victimContract`.
5. Midnight calls `victimContract.onBuy(...)`. Victim checks `msg.sender == midnight` → passes → returns `CALLBACK_SUCCESS`.
6. Midnight executes `safeTransferFrom(loanToken, victimContract, address(this), buyerAssets - sellerAssets)` and `safeTransferFrom(loanToken, victimContract, receiver, sellerAssets)`, draining `buyerAssets` from `victimContract`.

## Impact Explanation
Tokens are pulled via `safeTransferFrom` from an address (`victimContract`) that is not committed in the maker's signed offer hash and not authorized by the taker. The victim loses `buyerAssets` of the loan token with no recourse. This constitutes direct, unauthorized theft of assets from a third-party contract — a critical in-scope impact.

## Likelihood Explanation
Preconditions are realistic: (1) valid sell offers exist in normal protocol operation; (2) periphery contracts, aggregator routers, and vault wrappers built on Midnight are natural targets — they routinely approve Midnight for token spending and implement `IBuyCallback` gated only on `msg.sender == midnight`. The attack is permissionless, requires no privileged access, is repeatable across any valid sell offer, and can be executed by any external user.

## Recommendation
Add an authorization check binding `takerCallback` to the taker, analogous to the existing taker check:
```solidity
require(
    takerCallback == address(0) || takerCallback == msg.sender || isAuthorized[taker][takerCallback],
    TakerCallbackUnauthorized()
);
```
Alternatively, document explicitly in `IBuyCallback` that implementors **must** validate the `buyer` argument against an allowlist, and enforce this at the protocol level rather than relying on callback implementors.

## Proof of Concept
1. Deploy a `VictimCallback` contract that implements `IBuyCallback.onBuy` returning `CALLBACK_SUCCESS` with only a `require(msg.sender == midnight)` guard, and approve Midnight for `loanToken`.
2. Have a maker sign a sell offer (`offer.buy = false`, `offer.callback = address(0)`).
3. As attacker, call:
   ```solidity
   midnight.take(offer, ratifierData, units, attacker, attacker, address(victimCallback), "");
   ```
4. Observe that `loanToken.balanceOf(victimCallback)` decreases by `buyerAssets` and the attacker's `receiver` receives `sellerAssets` worth of tokens — all pulled from `victimCallback` without its consent. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
