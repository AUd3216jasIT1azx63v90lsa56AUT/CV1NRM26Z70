### Title
Missing zero-address guard on `referralFeeRecipient` with non-zero `referralFeePct` causes token-dependent behavior divergence - (`src/periphery/MidnightBundles.sol`)

### Summary
`buyWithUnitsTargetAndWithdrawCollateral` accepts caller-controlled `referralFeePct` and `referralFeeRecipient` with no check that `referralFeeRecipient != address(0)` when `referralFeePct > 0`. When `referralFeeAssets > 0`, `SafeTransferLib.safeTransfer(loanToken, address(0), referralFeeAssets)` executes unconditionally, and the outcome is entirely token-dependent: standard ERC20 tokens revert, while no-return or non-standard tokens may silently burn the fee assets.

### Finding Description
**Code path:**

In `MidnightBundles.sol`, the only input guards are: [1](#0-0) 

Neither check constrains `referralFeeRecipient`. After all takes complete and collateral withdrawals execute, the fee transfer fires unconditionally when `referralFeeAssets > 0`: [2](#0-1) 

`SafeTransferLib.safeTransfer` calls `token.transfer(address(0), referralFeeAssets)` with no pre-check on the recipient: [3](#0-2) 

**Attacker inputs:** Any caller who is the taker (or is authorized by the taker) passes `referralFeePct = 0.01e18` and `referralFeeRecipient = address(0)`. No privileged role is required.

**Exploit flow:**
1. Caller invokes `buyWithUnitsTargetAndWithdrawCollateral(targetUnits, maxBuyerAssets, taker, ..., 0.01e18, address(0))`.
2. `referralFeePct < WAD` passes; no zero-address check exists.
3. Takes fill successfully; `filledBuyerAssets > 0`; `referralFeeAssets = filledBuyerAssets * referralFeePct / (WAD - referralFeePct) > 0`.
4. `SafeTransferLib.safeTransfer(loanToken, address(0), referralFeeAssets)` executes.
5. Token-dependent outcome: OZ-style ERC20 reverts (entire tx rolls back); no-return or permissive tokens silently transfer to `address(0)`, burning the fee.

**Why existing checks fail:** `require(referralFeePct < WAD)` only bounds the percentage magnitude. The contract comment `/// @dev Zero checks are not systematically performed.` confirms the omission is intentional at the design level, but no compensating check exists at the fee-transfer site. [4](#0-3) 

### Impact Explanation
Behavior is token-dependent: on standard ERC20 the transaction reverts after all takes and collateral withdrawals have been attempted (wasted gas, no state change due to revert); on no-return or permissive tokens the referral fee is silently burned to `address(0)`, permanently destroying the caller's assets. This is a concrete non-critical behavior divergence across token types for the same input combination.

### Likelihood Explanation
Any taker (or their authorized agent) can trigger this with a single call — no special state, no oracle manipulation, no admin action required. It is repeatable on every invocation where `referralFeePct > 0` and `referralFeeRecipient = address(0)`. The most likely scenario is accidental misconfiguration by an integrator or front-end, but a malicious caller can also trigger it deliberately.

### Recommendation
Add a guard at the top of the function (and symmetrically in `supplyCollateralAndSellWithUnitsTarget`, `buyWithAssetsTargetAndWithdrawCollateral`, `supplyCollateralAndSellWithAssetsTarget`, and `repayAndWithdrawCollateral`):

```solidity
require(referralFeePct == 0 || referralFeeRecipient != address(0), InvalidReferralRecipient());
```

This makes the behavior consistent (always revert) regardless of token type when a fee is specified without a valid recipient.

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
    }
