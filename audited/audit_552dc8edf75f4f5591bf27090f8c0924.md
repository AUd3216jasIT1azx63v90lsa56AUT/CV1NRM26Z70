Audit Report

## Title
`sellerAssetsToUnits` Division-by-Zero When `sellerPrice == 0` DoS's Periphery Bundler - (File: src/periphery/TakeAmountsLib.sol)

## Summary
In `TakeAmountsLib.sellerAssetsToUnits`, when `offer.buy = true` and `settlementFee == tickToPrice(offer.tick)`, `sellerPrice` computes to exactly `0`, causing an unconditional Solidity 0.8 division-by-zero panic on `mulDivUp(WAD, 0)`. The core `Midnight.take()` does not revert in this state — it computes `sellerAssets = 0` via `mulDivDown(0, WAD)` — creating a core/periphery divergence. Because `sellerAssetsToUnits` is called outside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, the panic propagates and permanently DoS's that bundler function for any offer in this state.

## Finding Description

**Root cause — `TakeAmountsLib.sol` lines 44–46:** [1](#0-0) 

`sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice`. When `offer.buy = true` and `settlementFee == offerPrice`, `sellerPrice = 0`. The return statement immediately calls `targetSellerAssets.mulDivUp(WAD, 0)` — a Solidity 0.8 panic. There is no zero-guard on `sellerPrice`.

The NatSpec at line 34 states *"Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)"* — but this only covers the strict underflow case. For the equal case (`offerPrice == settlementFee`), midnight does **not** revert, while the periphery panics. [2](#0-1) 

**Core does not revert — `Midnight.sol` lines 361–364:** [3](#0-2) 

`sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD)`. With `sellerPrice = 0`, this returns `0` without reverting. `take()` proceeds normally, transferring 0 assets to the seller.

**Bundler call site — `MidnightBundles.sol` lines 285–300:** [4](#0-3) 

`TakeAmountsLib.sellerAssetsToUnits(...)` is called at line 286, **outside** the `try/catch` block that wraps `IMidnight(MIDNIGHT).take(...)` at lines 292–300. The panic propagates uncaught and reverts the entire `supplyCollateralAndSellWithAssetsTarget` call.

**Exploit flow:**
1. Governance sets `settlementFee` to value `S` (privileged prerequisite, but also reachable via natural time-based interpolation as the piecewise-linear fee drifts through tick price values over time).
2. An unprivileged lender creates a buy offer at tick `T` where `tickToPrice(T) == S`.
3. Any caller invokes `supplyCollateralAndSellWithAssetsTarget` with `takes` containing that offer and `targetSellerAssets > 0`.
4. `sellerAssetsToUnits` panics with division-by-zero; the entire bundler transaction reverts.

**Why existing checks fail:** The NatSpec guard only documents the `offerPrice < settlementFee` case (underflow revert in both core and periphery). The equal case is undocumented and unguarded. Both `tickToPrice` prices and settlement fees are rounded to `PRICE_ROUNDING_STEP = 1e12` (CBP), making exact equality structurally achievable. [5](#0-4) [6](#0-5) 

## Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee` and `targetSellerAssets > 0` reverts unconditionally. The bundler is DoS'd for that offer. Because the `try/catch` only wraps `take()` and not `sellerAssetsToUnits`, there is no graceful fallback. Funds are not frozen in the core (`take()` still works directly), but the periphery path is completely blocked for affected offers. The divergence between core and periphery is a broken accounting/safety assumption: the NatSpec contract that "midnight reverts too" is violated for the equal case.

## Likelihood Explanation
**Preconditions:**
- Settlement fee must equal `tickToPrice(T)` for some tick `T`. Both quantities are multiples of `1e12`, so exact equality is structurally possible.
- The settlement fee is a piecewise-linear function of TTM that changes continuously with time. Even without malicious governance action, the interpolated fee value naturally drifts and can pass through a tick price value.
- Once the condition holds, any unprivileged lender can create a buy offer at tick `T`, and any caller of the bundler triggers the DoS.
- `targetSellerAssets > 0` — always true for any real bundler call.

**Repeatability:** Deterministic — every bundler call with that offer reverts until the offer expires or the fee changes.

## Recommendation
Add a zero-guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case. Since `sellerPrice = 0` means any number of units yields 0 seller assets, `targetSellerAssets > 0` is unsatisfiable; the function should revert with a descriptive error (e.g., `SellerPriceIsZero()`):

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
require(sellerPrice > 0, SellerPriceIsZero());
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

This aligns the periphery with the core's documented behavior and prevents the panic. The NatSpec comment should also be updated to cover the equal case explicitly.

## Proof of Concept
**Minimal manual steps:**
1. Deploy `Midnight` and `MidnightBundles`.
2. Create a market with a buy offer at tick `T` where `tickToPrice(T) = F` for some fee value `F`.
3. Set `settlementFee` (via `setMarketSettlementFee`) to exactly `F` (a multiple of `CBP = 1e12`).
4. Call `supplyCollateralAndSellWithAssetsTarget` with `takes[0].offer` pointing to that buy offer and `targetSellerAssets = 1`.
5. Observe: transaction reverts with a Solidity panic (division by zero) at `TakeAmountsLib.sellerAssetsToUnits`.
6. Call `IMidnight.take()` directly with the same offer and `units = 1`: observe it succeeds with `sellerAssets = 0`, confirming the core/periphery divergence.
