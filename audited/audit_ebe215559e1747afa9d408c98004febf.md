### Title
Buy offer `maxAssets` cap bypassed via zero-rounding `buyerAssets` on sub-WAD price - (`File: src/Midnight.sol`)

### Summary
When `offer.buy=true` and `offer.maxAssets > 0`, the consumed counter is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. If `units * buyerPrice < WAD`, integer division rounds `buyerAssets` to zero, so `consumed[maker][group]` never advances. An unprivileged taker can call `take` with `units=1` on a fully-consumed buy offer at any tick below `MAX_TICK`, bypassing the cap entirely while still mutating lender credit and borrower debt each call.

### Finding Description
**Code path** — `src/Midnight.sol` lines 358–369:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → can be 0
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy
        ? buyerAssets   // += 0 when units * buyerPrice < WAD
        : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause** — `mulDivDown` truncates. For any tick strictly below `MAX_TICK`, `buyerPrice < WAD`. With `units = 1`, `1 * buyerPrice / WAD = 0` in integer arithmetic. The consumed accumulator therefore does not advance, and the cap check `newConsumed <= offer.maxAssets` trivially passes even when `consumed` is already at `maxAssets`.

**Attacker-controlled inputs**:
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick` any value `< MAX_TICK` (i.e., `buyerPrice < WAD`)
- `units = 1` (or any value where `units * buyerPrice < WAD`)
- Taker is any unprivileged borrower

**Exploit flow**:
1. Maker (lender) posts a buy offer with `maxAssets = N` and `tick < MAX_TICK`.
2. Offer is fully consumed (legitimately or via `setConsumed`).
3. Attacker (borrower) calls `take(offer, ..., units=1, ...)` repeatedly.
4. Each call: `buyerAssets = 0`, `consumed += 0`, cap check passes.
5. Each call: lender's `credit += 1`, borrower's `debt += 1`, `totalUnits += 1`, zero tokens transferred.
6. Repeated indefinitely — lender accumulates unbounded credit beyond `maxAssets`.

**Why existing checks fail** — The only guard is `require(newConsumed <= offer.maxAssets)`. When `buyerAssets = 0`, `newConsumed` equals the pre-call value, so the check is a no-op. The NatSpec at line 94 explicitly acknowledges this: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The existing test `testBugBuyMaxAssetsBypass` (line 857–889) confirms the behavior is reachable and observable. ### Impact Explanation
The maker's `maxAssets` cap — intended to bound the total buyer-side asset exposure of a buy offer — is violated. An unprivileged taker can force the lender's credit to grow without bound beyond `maxAssets`, at zero token cost. Credit created without a matching token deposit breaks the balance invariant: when the lender later redeems that excess credit by taking a sell offer, the protocol pays out loan tokens that were never deposited for those units. Additionally, the borrower accumulates unbounded debt without receiving any assets, which can be used to manufacture undercollateralized positions if the attacker controls both sides (e.g., via a second account as maker). [4](#0-3) 

### Likelihood Explanation
**Preconditions**: (1) a buy offer exists with `maxAssets > 0` and `tick < MAX_TICK`; (2) the offer is at or near its consumed cap. Condition (1) is the normal operating mode for any lender using asset-denominated caps. Condition (2) is reachable by the attacker themselves via legitimate fills. The attack requires no special privilege, no oracle manipulation, and no token quirks. It is repeatable in a single transaction via `multicall` or a loop, and works on any market. The only cost is gas and the collateral needed to keep the borrower position healthy. [5](#0-4) 

### Recommendation
Track consumed in **units** when `buyerAssets` rounds to zero, or enforce a minimum `units` floor such that `units.mulDivDown(buyerPrice, WAD) >= 1` whenever `units > 0`. The simplest targeted fix is to add a guard in the `maxAssets` branch:

```solidity
if (offer.maxAssets > 0) {
    uint256 delta = offer.buy ? buyerAssets : sellerAssets;
    // Prevent zero-delta fills from bypassing the cap when units > 0
    require(units == 0 || delta > 0, ZeroDeltaOnNonZeroUnits());
    newConsumed = consumed[offer.maker][offer.group] += delta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, track consumed in units unconditionally and convert `maxAssets` to a units ceiling at offer creation time, eliminating the rounding gap entirely. [6](#0-5) 

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` already constitutes a passing unit-test PoC. An extended fuzz/invariant version:

```solidity
// Foundry fuzz test
function testFuzz_BuyMaxAssetsBypassUnbounded(uint256 fillCount) public {
    fillCount = bound(fillCount, 1, 1000);

    // Setup: buy offer, price < WAD, maxAssets = 1 (fully consumed)
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD → 1 unit → buyerAssets = 0

    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, 1, lender);

    collateralize(market, borrower, fillCount); // borrower needs collateral

    uint256 creditBefore = midnight.creditOf(id, lender);

    for (uint256 i; i < fillCount; i++) {
        take(1, borrower, lenderOffer); // each call: consumed unchanged, credit grows
    }

    // ASSERTION: credit grew beyond maxAssets worth of units
    assertGt(midnight.creditOf(id, lender), creditBefore + 0);
    // Invariant: consumed must equal maxAssets (cap was never exceeded in the counter)
    assertEq(midnight.consumed(lender, lenderOffer.group), 1);
    // But actual units filled = fillCount >> maxAssets / price
    assertEq(midnight.debtOf(id, borrower), midnight.creditOf(id, lender) - creditBefore);
}
```

**Expected assertions**: `consumed` stays at `1` (cap counter frozen), while `creditOf(lender)` grows by `fillCount` units — demonstrating unbounded bypass of the `maxAssets = 1` cap. [3](#0-2) [1](#0-0)
