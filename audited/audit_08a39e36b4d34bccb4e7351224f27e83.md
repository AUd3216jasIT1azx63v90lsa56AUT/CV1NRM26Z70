Audit Report

## Title
Fee-on-Transfer Loan Token Causes Lenders to Receive Less Than Credit Entitlement on Withdraw - (File: src/Midnight.sol)

## Summary
The `withdraw` function decrements `credit`, `withdrawable`, and `totalUnits` by the full `units` amount before calling `SafeTransferLib.safeTransfer`, which only verifies call success and the boolean return value — not the receiver's actual balance delta. When the market's `loanToken` is a fee-on-transfer token, the receiver receives `units * (1 - fee_rate)` while the protocol's accounting is permanently reduced by the full `units`. Because `touchMarket` is permissionless and imposes no restrictions on `loanToken`, any user can create such a market, and every lender withdrawal in it results in an unrecoverable shortfall.

**Compound accounting break across the full lifecycle:**

- On `repay` (line 520): `safeTransferFrom(market.loanToken, payer, address(this), units)` — the contract receives `units * (1 - fee_rate)` but credits the full `units` to `withdrawable` (line 509). The contract is immediately underfunded. [4](#0-3) 
- On `withdraw` (line 499): `safeTransfer(market.loanToken, receiver, units)` — the contract's balance decreases by `units` but the receiver only receives `units * (1 - fee_rate)`. The fee is silently absorbed by the token contract. [1](#0-0) 

**Exploit flow:**
1. Any user calls `touchMarket` with a fee-on-transfer token as `loanToken` (permissionless, no token restrictions).
2. A lender lends N units via `take`; the contract receives `buyerAssets * (1 - fee_rate)` but credits `buyerAssets` to the lender's position.
3. A borrower repays N units; the contract receives `N * (1 - fee_rate)` but credits N to `withdrawable`.
4. The lender calls `withdraw(market, N, lender, receiver)`:
   - `credit`, `withdrawable`, `totalUnits` all decrease by N (full amount).
   - `safeTransfer` succeeds (token returns `true`).
   - Receiver's balance increases by only `N * (1 - fee_rate)`.
5. No revert, no check, no recovery path.

**Why existing checks fail:**

`SafeTransferLib.safeTransfer` only checks `success` and the ABI-decoded boolean return — it cannot detect fee-on-transfer shortfalls. [5](#0-4) 
There is no balance-before/after guard anywhere in `withdraw`. [6](#0-5) 
`touchMarket` has no loan token allowlist or fee-on-transfer check. [7](#0-6) 

## Impact Explanation
Every lender withdrawal in a fee-on-transfer loan token market results in the receiver receiving `units * (1 - fee_rate)` tokens instead of `units`. The credit accounting is permanently reduced by the full `units`, so the lender cannot re-withdraw the shortfall. The loss is proportional to the fee rate and repeats on every withdrawal, draining lender value systematically. The contract's actual token balance also diverges from its tracked `withdrawable` state, eventually causing later withdrawals to revert due to insufficient balance, freezing remaining lenders' funds. This constitutes both direct theft of lender value and permanent fund freeze — both in-scope impact classes per `RESEARCHER.md`. [8](#0-7) 

## Likelihood Explanation
`touchMarket` is permissionless — any user can create a market with any ERC20 as the loan token. [9](#0-8) 
Fee-on-transfer tokens (deflationary tokens, tokens with protocol fees) exist in production. A lender who participates in such a market — whether knowingly or not — will suffer the loss on every `withdraw` call. The condition is fully attacker-controllable (market creation) and repeatable on every withdrawal. No privileged access is required.

## Recommendation
1. **Balance-delta check in `withdraw`**: Record `balanceBefore = IERC20(market.loanToken).balanceOf(receiver)` before the transfer and verify `balanceAfter - balanceBefore == units` after, reverting if not.
2. **Alternatively, restrict `touchMarket`**: Validate that the loan token does not take a fee by performing a test transfer during market creation, or maintain an allowlist of approved loan tokens.
3. **Apply the same fix to `repay`**: Verify that the contract's balance increased by exactly `units` after `safeTransferFrom`, or adjust `withdrawable` to reflect the actual received amount. [10](#0-9) 

## Proof of Concept
**Minimal Foundry fork test plan:**
1. Deploy a mock ERC20 with a 1% fee-on-transfer (i.e., `transfer` and `transferFrom` send `amount * 99 / 100` to the recipient and burn the rest).
2. Call `touchMarket` with this token as `loanToken` (permissionless).
3. Lender calls `take` to lend 100 units; assert `position[id][lender].credit == 100` but `token.balanceOf(address(midnight)) < 100`.
4. Borrower calls `repay(market, 100, borrower, ...)`: assert `marketState[id].withdrawable == 100` but `token.balanceOf(address(midnight))` increased by only 99.
5. Lender calls `withdraw(market, 100, lender, receiver)`: assert `position[id][lender].credit == 0` and `marketState[id].withdrawable == 0`, but `token.balanceOf(receiver)` increased by only 99.
6. Confirm: lender lost 1 token per 100 withdrawn, with no revert and no recovery path. [6](#0-5)

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
    }
