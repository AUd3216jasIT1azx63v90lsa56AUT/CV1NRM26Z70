### Title
Maker-controlled sell callback can transiently manipulate oracle price, causing `isHealthy` to return false and reverting valid taker transactions - (File: src/Midnight.sol)

### Summary

When a taker fills a sell offer (`offer.buy == false`), the maker's `offer.callback` is used as the `sellerCallback` and is invoked before the post-callback health check at line 476. Because the `LIQUIDATION_LOCK_SLOT` is reset to `false` before `isHealthy` is evaluated, a malicious maker callback that transiently depresses an on-chain oracle (e.g., a spot/TWAP oracle reading pool reserves) causes `isHealthy(offer.market, id, seller)` to return `false` for a genuinely healthy seller, reverting the taker's transaction with `SellerIsLiquidatable`.

### Finding Description

**Callback assignment (line 421):**

```solidity
address sellerCallback = offer.buy ? takerCallback : offer.callback;
```

When `offer.buy == false`, `sellerCallback = offer.callback` â€” fully controlled by the maker at offer-creation time. [1](#0-0) 

**Seller identity (line 375):**

```solidity
(address buyer, address seller) = offer.buy ? (offer.maker, taker) : (taker, offer.maker);
```

When `offer.buy == false`, `seller = offer.maker`. The health check at line 476 is therefore on the maker's own position. [2](#0-1) 

**Callback invocation (lines 458â€“474):**

`ISellCallback(sellerCallback).onSell(...)` is called with no restrictions on what the callback may do to external state (e.g., pool reserves). [3](#0-2) 

**Lock reset and health check (lines 475â€“476):**

```solidity
if (!wasLocked) UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, false);
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```

In the normal case (`wasLocked == false`), the lock is reset to `false` before the `require`. Therefore `liquidationLocked(id, seller)` evaluates to `false`, and `isHealthy` must return `true` for the transaction to succeed. If the callback has shifted the oracle price, `isHealthy` reads the manipulated value and can return `false`. [4](#0-3) 

**Exploit flow:**

1. Maker deploys `MaliciousCallback` implementing `ISellCallback.onSell` that swaps tokens in the oracle's underlying pool, dropping the reported collateral price.
2. Maker creates a sell offer with `offer.callback = address(MaliciousCallback)`.
3. Taker calls `take()`. State is updated (seller debt increases), lock is set to `true`.
4. `MaliciousCallback.onSell` executes the pool swap; oracle price drops below the health threshold.
5. Lock is reset to `false` (line 475, `!wasLocked`).
6. `isHealthy(offer.market, id, seller)` reads the manipulated price â†’ returns `false`.
7. `require(false || false, SellerIsLiquidatable())` â†’ transaction reverts.

**Why existing protections are insufficient:**

The `LIQUIDATION_LOCK_SLOT` mechanism (lines 444, 475) is designed to prevent reentrant liquidation calls during callbacks â€” it does not protect the post-callback `isHealthy` evaluation from reading oracle state that was mutated during the callback. The comment at line 335 ("The seller cannot be liquidated during the callbacks of a take") confirms the intent is only to block reentrant liquidation, not to guard the health check against oracle manipulation. [5](#0-4) 

### Impact Explanation

Any taker attempting to fill a malicious maker's sell offer will have their transaction revert with `SellerIsLiquidatable` even though the seller's position was healthy at the pre-callback oracle price. The maker's offer is never consumed (all state changes revert), so the attack is repeatable at zero cost to the maker. This permanently blocks market activity on any offer the maker chooses to poison, and can be used to grief specific takers or deny liquidity to the entire market.

### Likelihood Explanation

**Preconditions:**
- Market uses an oracle whose `price()` reads mutable on-chain state (e.g., Uniswap v2/v3 spot price, any reserve-based oracle). TWAP oracles with short windows are also susceptible.
- Maker's position must be healthy before the callback (so the offer passes normal validation) but close enough to the liquidation threshold that a feasible pool swap can push it below.
- Maker must be able to afford the swap cost (flash loan makes this essentially free).

**Feasibility:** High. The maker controls `offer.callback` unconditionally; no privileged access is required. The attack is atomic (single transaction), repeatable, and costs only gas plus any swap slippage (recoverable via flash loan). Any market with a spot oracle is vulnerable.

### Recommendation

Move the `isHealthy` check to **before** the seller callback is invoked, using the oracle price captured at that point, or snapshot the oracle price before the callback and use the snapshot for the post-callback health check. Alternatively, prohibit state-mutating external calls from the seller callback by reading and caching the oracle price before any external call and asserting the cached value is used in `isHealthy`. A simpler mitigation is to require that `isHealthy` passes both before and after the callback, so a callback that worsens health causes an immediate revert with a clear error rather than a misleading `SellerIsLiquidatable`.

### Proof of Concept

```solidity

pragma solidity ^0.8.0;

contract MockOracle {
    uint256 public price = 1e18; // healthy price
    function setPrice(uint256 p) external { price = p; }
}

contract MaliciousCallback is ISellCallback {
    MockOracle oracle;
    constructor(MockOracle _o) { oracle = _o; }
    function onSell(...) external returns (bytes32) {
        oracle.setPrice(1); // drop price to near-zero
        return CALLBACK_SUCCESS;
    }
}

contract PoCTest is Test {
    function test_oracleManipulationGriefsTaker() public {
        // 1. Deploy MockOracle, set market to use it
        // 2. Fund maker with collateral; maker position is healthy at price=1e18
        // 3. Maker creates sell offer with callback = MaliciousCallback
        // 4. Taker calls take()
        // 5. Assert: take() reverts with SellerIsLiquidatable
        // 6. Assert: oracle.price() == 1 (manipulated) at revert time
        // 7. Assert: resetting oracle.price() to 1e18 and retrying succeeds
        vm.expectRevert(SellerIsLiquidatable.selector);
        midnight.take(offer, ...);
    }
}
```

**Expected assertions:**
- `take()` reverts with `SellerIsLiquidatable` despite seller being healthy at pre-callback price.
- Resetting the oracle to its original price and replaying the identical `take()` call succeeds.
- Fuzz variant: for any oracle price `p` such that `isHealthy` returns `true` at `p` and `false` at `p - delta`, a callback that sets price to `p - delta` always causes revert.
