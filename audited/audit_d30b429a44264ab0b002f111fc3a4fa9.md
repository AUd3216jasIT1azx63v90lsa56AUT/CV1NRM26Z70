### Title
Unconditional zero-value `safeTransferFrom` for settlement fee blocks valid takes on tokens that revert on zero transfers - (File: src/Midnight.sol)

### Summary
In `Midnight.take`, the settlement fee portion (`buyerAssets - sellerAssets`) is transferred to `address(this)` unconditionally via `SafeTransferLib.safeTransferFrom` at line 455, with no guard for the zero case. When `units` is small and `_settlementFee` is small, integer rounding causes `buyerAssets == sellerAssets`, making the transfer amount exactly 0. If the market's `loanToken` is any ERC20 that reverts on zero-value transfers (a known real-world pattern), the entire `take` call reverts, permanently blocking that (tick, units) combination.

### Finding Description

**Code path:**

`src/Midnight.sol` lines 361–364 compute assets: [1](#0-0) 

For a **sell offer** (`offer.buy = false`):
- `sellerPrice = offerPrice`
- `buyerPrice = offerPrice + _settlementFee`
- `buyerAssets = mulDivUp(units, buyerPrice, WAD)`
- `sellerAssets = mulDivUp(units, sellerPrice, WAD)`

Both use `mulDivUp`. When `units * _settlementFee < WAD`, the ceiling of both expressions is identical, so `buyerAssets - sellerAssets == 0`.

**Concrete arithmetic proof:**
- `units = 1`, `offerPrice = WAD - 1` (tick near price=1), `_settlementFee = 1`
- `buyerAssets = mulDivUp(1, WAD, WAD) = 1`
- `sellerAssets = mulDivUp(1, WAD - 1, WAD) = ceil((WAD-1)/WAD) = 1`
- `buyerAssets - sellerAssets = 0`

Line 418 silently accepts this (adding 0 to `claimableSettlementFee`): [2](#0-1) 

Then line 455 calls `safeTransferFrom` unconditionally with amount 0: [3](#0-2) 

`SafeTransferLib.safeTransferFrom` has no zero-value guard — it calls `token.transferFrom(from, to, 0)` directly: [4](#0-3) 

If the token's `transferFrom` reverts on a zero amount, the assembly block at lines 28–31 re-bubbles the revert, causing the entire `take` to fail: [5](#0-4) 

**No existing protection stops this.** There is no `require(buyerAssets > sellerAssets)` or `if (fee > 0)` guard anywhere in `take` before the transfer.

### Impact Explanation
Any taker attempting a valid take on a market whose `loanToken` reverts on zero-value transfers, with `(tick, units)` parameters that cause `buyerAssets == sellerAssets`, receives a revert. All state mutations (position updates, `consumed` increments, `claimableSettlementFee` update) have already occurred before the transfer, but the transaction reverts atomically, so no state is actually corrupted — the take simply cannot execute. The affected `(tick, units)` combinations are permanently inaccessible for that token, constituting a targeted denial-of-service on valid near-par takes.

### Likelihood Explanation
- **Token precondition:** Tokens that revert on zero-value `transferFrom` exist in production (e.g., LEND, some rebasing or fee-on-transfer tokens). The protocol places no restriction on `loanToken` choice.
- **Rounding precondition:** Triggered whenever `units * _settlementFee < WAD`. With `_settlementFee` at its minimum (1 cbp = 1e12 in WAD terms) and `units = 1`, the condition holds trivially. Near-par ticks (price ≈ WAD) are the most common trading range.
- **Repeatability:** Every call with the same `(token, tick, units)` combination will fail identically. The taker cannot work around it without increasing `units` enough that `units * _settlementFee >= WAD`, which may not be economically desirable or possible within offer limits.

### Recommendation
Guard both settlement-fee and seller-assets transfers against zero:

```solidity
uint256 fee = buyerAssets - sellerAssets;
if (fee > 0) {
    SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), fee);
}
if (sellerAssets > 0) {
    SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
}
```

This mirrors the pattern used in Morpho Blue and other Morpho-family contracts that skip zero-value ERC20 calls.

// Mock ERC20 that reverts on zero-value transferFrom
contract ZeroRevertToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;

    function mint(address to, uint256 amt) external { balanceOf[to] += amt; }
    function approve(address spender, uint256 amt) external returns (bool) {
        allowance[msg.sender][spender] = amt; return true;
    }
    function transferFrom(address from, address to, uint256 amt) external returns (bool) {
        require(amt > 0, "zero transfer");          // <-- the trigger
        allowance[from][msg.sender] -= amt;
        balanceOf[from] -= amt;
        balanceOf[to] += amt;
        return true;
    }
    function transfer(address to, uint256 amt) external returns (bool) {
        require(amt > 0, "zero transfer");
        balanceOf[msg.sender] -= amt;
        balanceOf[to] += amt;
        return true;
    }
}

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
