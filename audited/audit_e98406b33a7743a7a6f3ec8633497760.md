### Title
Zero-buyerAssets Take Bypasses Consumed Cap on Fully-Exhausted Assets-Based Buy Offer, Triggering Maker Callback Indefinitely - (File: src/Midnight.sol)

### Summary

When a buy offer uses `maxAssets`-based accounting and `buyerPrice < WAD`, a taker can call `take()` with `units=1` on a fully consumed offer (`consumed == maxAssets`). Because `buyerAssets = units.mulDivDown(buyerPrice, WAD) = 0`, the consumed increment is zero, `newConsumed == maxAssets <= maxAssets` passes, and the take executes in full — mutating position state and invoking the maker's callback — with no token transfer. Since consumed never advances past `maxAssets`, this is repeatable without bound.

### Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

```solidity
// line 363
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;

// lines 367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

When `offer.buy == true`, `buyerPrice < WAD`, and `units` is small enough that `units * buyerPrice / WAD` truncates to zero, `buyerAssets == 0`. The consumed mapping is incremented by zero, so `newConsumed` stays at its pre-call value. If that value is already `maxAssets`, the `require` passes trivially.

**Exploit flow:**

1. Maker creates a buy offer with `maxAssets = N` and `tick` such that `buyerPrice < WAD` (any tick below the WAD-price tick, e.g. `MAX_TICK - 16`).
2. Offer is legitimately filled to `consumed == maxAssets` (fully exhausted).
3. Unprivileged taker calls `take(offer, ..., units=1, ...)`.
4. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`.
5. `consumed[maker][group] += 0` → still `maxAssets`; `require(maxAssets <= maxAssets)` passes.
6. Execution continues: `buyerCreditIncrease = 1`, `sellerDebtIncrease = 1`, `totalUnits += 1`.
7. Maker's `offer.callback` is invoked via `IBuyCallback.onBuy(... buyerAssets=0, units=1 ...)`.
8. Token transfers are `safeTransferFrom(..., 0)` — no-ops.
9. `consumed` is unchanged at `maxAssets`, so step 3–8 can be repeated indefinitely.

**Existing checks that fail to stop it:**

- `require(newConsumed <= offer.maxAssets)` — passes because the increment is zero.
- No check that `units == 0` when `buyerAssets == 0` for assets-based offers.
- No check that `consumed < maxAssets` before proceeding.

The protocol's own NatDoc at line 94 acknowledges this: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The existing test `testBugBuyMaxAssetsBypass()` (lines 857–889) explicitly demonstrates and labels this as a bug. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The maker's callback (`offer.callback`) is invoked via `IBuyCallback.onBuy` with `buyerAssets=0` and `units=1` on an offer the maker considers fully consumed. Callback logic that guards on `buyerAssets > 0`, tracks cumulative fills, releases collateral, or performs accounting based on the assumption that the offer is exhausted will receive unexpected calls. Simultaneously, each such call silently inflates `buyerPos.credit` by 1, `sellerPos.debt` by 1, and `_marketState.totalUnits` by 1 — all without any corresponding token transfer — breaking the invariant that contract balances cover credit redemption. [4](#0-3) [5](#0-4) [6](#0-5) 

### Likelihood Explanation

Preconditions are all attacker-controllable or naturally occurring:

- `offer.buy == true` and `offer.maxAssets > 0`: standard buy offer configuration.
- `buyerPrice < WAD`: any tick below the WAD-price threshold; the test uses `MAX_TICK - 16`.
- `consumed == maxAssets`: the offer has been legitimately filled; this is the normal end-state of any fully filled offer.
- `units = 1`: trivially chosen by the taker.

No privileged access, no oracle manipulation, and no special timing is required. The attack is repeatable in every block at negligible cost (only gas), since `consumed` never advances. [7](#0-6) 

### Recommendation

Add an explicit guard that rejects a take when the offer is assets-based, `offer.buy == true`, and `buyerAssets == 0` but `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    // Prevent zero-asset takes from bypassing the consumed cap.
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce `require(units == 0 || buyerAssets > 0)` unconditionally for buy offers in assets mode, mirroring the existing rounding note that "not all buyerAssets are reachable" when `buyerPrice > WAD`. [8](#0-7) 

### Proof of Concept

The existing Foundry test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already constitutes a complete PoC. Its assertions confirm the bug:

```solidity
// Setup: buy offer, maxAssets=1, tick such that buyerPrice < WAD, consumed pre-set to maxAssets.
lenderOffer.maxAssets = 1;
lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD
midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender); // consumed == maxAssets

// Attack: take with units=1
(uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

// Assertions proving the bug:
assertEq(buyerAssets, 0);                                                    // no assets transferred
assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets); // consumed unchanged → repeatable
assertGt(midnight.creditOf(id, lender), lenderCreditBefore);                // credit inflated
assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);                // debt inflated
assertGt(midnight.totalUnits(id), totalUnitsBefore);                        // totalUnits inflated
```

A fuzz extension should loop the attack N times and assert that `creditOf(lender)` grows by exactly N while `consumed` stays fixed at `maxAssets`, confirming unbounded repeatability. [3](#0-2)
