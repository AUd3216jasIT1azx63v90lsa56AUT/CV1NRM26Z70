Audit Report

## Title
`mulDivUp` Arithmetic Overflow in `consumableUnits` for Buy Offers with Oversized `maxAssets` - (File: `src/periphery/ConsumableUnitsLib.sol`)

## Summary
`UtilsLib.mulDivUp` performs `(x * y + (d - 1)) / d` under checked Solidity 0.8.x arithmetic. When `consumableUnits` is called for a buy offer with `maxUnits == 0` and `maxAssets > type(uint256).max / WAD` (~1.157e59), the intermediate product `targetBuyerAssets * WAD` overflows and the call reverts unconditionally. Because `MidnightBundles` explicitly propagates this revert ("Reverts if ConsumableUnitsLib reverts"), any bundle that includes such a crafted offer is permanently DoS-ed.

## Finding Description

**Step 1 — `ConsumableUnitsLib.consumableUnits`:**
With `offer.maxUnits == 0` and `offer.buy == true`, the function falls into the `else if` branch and calls:
```solidity
TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
```
With `consumed == 0`, `targetBuyerAssets = offer.maxAssets`. [1](#0-0) 

**Step 2 — `TakeAmountsLib.buyerAssetsToUnits`:**
For a buy offer (`offer.buy == true`), line 29 calls:
```solidity
targetBuyerAssets.mulDivUp(WAD, buyerPrice)
``` [2](#0-1) 

**Step 3 — `UtilsLib.mulDivUp`:**
The implementation is:
```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```
There is no `unchecked` block. Under Solidity `^0.8.0`, `x * y` reverts on overflow when `x > type(uint256).max / y`. [3](#0-2) 

**Why existing checks fail:**
- `require(buyerPrice <= WAD, ...)` (line 28 of `TakeAmountsLib`) only bounds `buyerPrice` from above; it does not bound `targetBuyerAssets`.
- `zeroFloorSub` only floors at zero; it does not cap `targetBuyerAssets`.
- `atMostOneNonZero` in `Midnight.take` (line 350) only prevents both `maxAssets > 0` and `maxUnits > 0` simultaneously; it imposes no magnitude constraint on `maxAssets`. [4](#0-3) 

**Propagation to `MidnightBundles`:**
`supplyCollateralAndSellWithUnitsTarget` (line 150) and `supplyCollateralAndSellWithAssetsTarget` (line 290) both call `ConsumableUnitsLib.consumableUnits` for buy offers outside any `try/catch`. The contract's own NatSpec documents: *"Reverts if ConsumableUnitsLib reverts."* A single poisoned offer in the `takes` array causes the entire bundle to revert. [5](#0-4) [6](#0-5) 

## Impact Explanation
Any call to `consumableUnits` for a buy offer where `offer.maxAssets > type(uint256).max / 1e18` reverts unconditionally. This permanently DoS-es `supplyCollateralAndSellWithUnitsTarget` and `supplyCollateralAndSellWithAssetsTarget` in `MidnightBundles` for any bundle that includes such an offer. Off-chain integrations and routing systems that rely on `consumableUnits` to compute fill amounts are similarly broken for the affected offer class. The core `Midnight.take` is unaffected.

## Likelihood Explanation
A maker needs only to sign an offer with `buy = true`, `maxUnits = 0`, and `maxAssets` set to any value above ~1.157e59 (e.g., `type(uint256).max`). No special privilege, no oracle manipulation, and no existing state is required. The condition is trivially reachable on first call with zero prior consumption. Any routing system or aggregator that automatically ingests on-chain or off-chain offers can be fed this poisoned offer, causing bundle-level reverts for legitimate takers.

## Recommendation
Replace the naïve checked multiplication in `mulDivUp` (and `mulDivDown`) with an `unchecked` block or a full 512-bit overflow-safe `mulDiv` (e.g., Solmate's or a Muldiv512 implementation). At minimum, wrap the multiplication in `unchecked`:
```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    unchecked {
        return x == 0 ? 0 : (x * y + (d - 1)) / d;
    }
}
```
Alternatively, add an explicit upper-bound check on `offer.maxAssets` in `consumableUnits` or in `buyerAssetsToUnits` before the multiplication, or cap `maxAssets` to `type(uint128).max` at offer creation time (consistent with the test suite's `MAX_AMOUNT = type(uint128).max` convention). [3](#0-2) 

## Proof of Concept
```solidity
// Foundry test (no fork required)
function testConsumableUnitsOverflow() public {
    Offer memory offer;
    offer.buy = true;
    offer.maxUnits = 0;
    offer.maxAssets = type(uint256).max; // > type(uint256).max / 1e18
    offer.tick = MAX_TICK; // valid tick, buyerPrice <= WAD
    offer.maker = address(0xdead);
    offer.ratifier = address(dummyRatifier);
    offer.market = market;
    offer.expiry = type(uint256).max;

    // consumableUnits reverts with arithmetic overflow
    vm.expectRevert(); // Panic: arithmetic overflow
    ConsumableUnitsLib.consumableUnits(address(midnight), id, offer);
}
```
The revert propagates through `supplyCollateralAndSellWithUnitsTarget` when the poisoned offer is included in the `takes` array, causing the entire bundle to fail.
