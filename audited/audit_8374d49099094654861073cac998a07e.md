### Title
Buy offer `maxAssets` cap bypassed via zero-rounding of `buyerAssets` when `buyerPrice < WAD` - (`File: src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and `buyerPrice < WAD`, the consumed accounting increments by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`, which rounds to zero for sufficiently small `units`. Because the cap check `newConsumed <= offer.maxAssets` operates on this rounded-down asset value, a taker can call `take()` with small `units` repeatedly — each time contributing 0 to `consumed` — filling the maker's credit position arbitrarily beyond the intended `maxAssets` cap while paying zero loan tokens. The protocol explicitly acknowledges this in its NatSpec and a dedicated test named `testBugBuyMaxAssetsBypass` confirms the exact exploit path.

### Finding Description
**Code path:** `src/Midnight.sol` lines 363–369.

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// ...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause:** For a buy offer, `consumed` is incremented by `buyerAssets`, not by `units`. When `buyerPrice < WAD` (any tick below `MAX_TICK` with zero settlement fee, or any tick where `offerPrice - settlementFee < WAD`), choosing `units` such that `units * buyerPrice < WAD` causes `mulDivDown` to return 0. The cap check then passes unconditionally — even when `consumed` is already at `maxAssets`.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick` set so `buyerPrice < WAD`
- `units` chosen small enough that `units * buyerPrice < WAD` (e.g., `units = 1` when `buyerPrice < WAD`)

**Exploit flow:**
1. Maker (lender) posts a buy offer: `offer.buy = true`, `maxAssets = N`, tick such that `buyerPrice < WAD`.
2. Taker (borrower) calls `take(offer, ..., units=1, ...)` repeatedly.
3. Each call: `buyerAssets = 1 * buyerPrice / WAD = 0` (rounds down). `consumed` stays unchanged. Check `newConsumed <= maxAssets` passes.
4. Each call: `buyerCreditIncrease = 1`, `sellerDebtIncrease = 1`. Maker's credit grows by 1 unit, taker's debt grows by 1 unit. Zero loan tokens are transferred.
5. `totalUnits` increases by 1 per call with no backing asset deposit.
6. `EventsLib.Take` emits `consumed = maxAssets` (unchanged) and `buyerAssets = 0` on every iteration, confirming the cap is never enforced.

**Why existing checks fail:** The `ConsumedAssets` check (`newConsumed <= offer.maxAssets`) is the only guard. It operates on the asset-denominated `consumed` value, which never increases when `buyerAssets = 0`. There is no floor check requiring `buyerAssets > 0` when `units > 0`, and no unit-denominated secondary cap for the `maxAssets` branch.

### Impact Explanation
The maker's buy offer is overfilled in unit terms without bound. The maker accumulates credit far exceeding `maxAssets / buyerPrice` units while paying zero assets per extra unit. Simultaneously, `totalUnits` grows without a matching asset deposit, creating a protocol-level insolvency gap: when the maker later withdraws credit, the contract lacks the backing loan tokens. This directly violates the core invariant that contract balances cover credit redemption and that offers cannot be overfilled.

### Likelihood Explanation
Preconditions are easily met: any buy offer with `buyerPrice < WAD` (any tick below `MAX_TICK` when settlement fee is zero, or any tick where `offerPrice > settlementFee` but `offerPrice < WAD`) and `maxAssets > 0`. The taker needs no special privilege — any address can call `take()`. The attack is repeatable in a single transaction via multicall or a loop in a callback. The protocol's own NatSpec at line 94 states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1,"* and `testBugBuyMaxAssetsBypass` (line 858) confirms the path is reachable with `units = 1`, `maxAssets = 1`, `tick = MAX_TICK - 16`.

### Recommendation
Add a require that `buyerAssets > 0` whenever `units > 0` for buy offers in the `maxAssets` branch, or alternatively enforce a unit-denominated secondary cap alongside the asset cap:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets()); // prevent rounding bypass
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any non-zero unit fill contributes a non-zero amount to `consumed`, making the cap enforceable in both asset and unit terms.

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol:858`) already proves the path. An extended fuzz/invariant test:

```solidity
function testFuzz_BuyMaxAssetsBypassUnbounded(uint256 numTakes) public {
    numTakes = bound(numTakes, 1, 1000);
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    collateralize(market, borrower, numTakes);
    deal(address(loanToken), lender, 0);

    uint256 totalUnitsBefore = midnight.totalUnits(id);
    for (uint256 i; i < numTakes; i++) {
        take(1, borrower, lenderOffer);
    }

    // Assert: consumed never exceeded maxAssets (asset cap "held")
    assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
    // Assert: but units filled far exceed maxAssets / buyerPrice
    assertEq(midnight.totalUnits(id), totalUnitsBefore + numTakes);
    // Assert: no loan tokens were transferred (maker paid nothing)
    assertEq(loanToken.balanceOf(address(midnight)), 0); // no deposit backing the credit
    // Assert: maker has credit with no backing assets — insolvency
    assertGt(midnight.creditOf(id, lender), 0);
}
```
