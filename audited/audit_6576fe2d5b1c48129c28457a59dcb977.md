### Title
Fully-consumed assets-mode buy offer with `buyerPrice < WAD` allows unbounded credit/debt creation with zero token transfer â€” (`src/Midnight.sol`)

### Summary
When `offer.buy = true`, `offer.maxAssets > 0`, and `buyerPrice < WAD`, the consumed-cap check in `take()` increments `consumed` by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`, which rounds down to `0` for small `units`. This means the check `newConsumed <= offer.maxAssets` always passes even when `consumed[maker][group]` is already at `maxAssets`, because `0` is added. However, position accounting uses `units` directly, so `buyerPos.credit` and `sellerPos.debt` increase by `units` while zero tokens are transferred. The attack is repeatable without limit.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets = N` (any nonzero value), `offer.tick` set to any tick where `tickToPrice(tick) < WAD` (i.e., any tick below `MAX_TICK`)
- `units` chosen small enough that `units.mulDivDown(buyerPrice, WAD) == 0` (e.g., `units = 1` when `buyerPrice < WAD`)
- `consumed[maker][group]` pre-set to `maxAssets` via `setConsumed` (callable by the maker themselves, or the maker may have already legitimately consumed the offer)

**Why existing checks fail:**
- The `ConsumedAssets` guard at line 369 checks `newConsumed <= maxAssets`. Since `buyerAssets = 0`, `newConsumed` is unchanged and the check trivially passes.
- There is no guard requiring `buyerAssets > 0` when `units > 0`.
- The `reduceOnly` flag does not apply here (it guards against credit increase for the maker, but the maker is the buyer whose credit is increasing â€” and `reduceOnly` only blocks the take if `buyerCreditIncrease == 0` is violated, which is the opposite direction).
- The protocol itself documents this at line 94: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1"* â€” confirming the path is reachable and not blocked. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The maker (lender) sets `maxAssets = N` to cap their total credit exposure. After the offer is fully consumed (`consumed == N`), an unprivileged taker can call `take(units=1, ...)` repeatedly. Each call:
1. Adds `0` to `consumed` (cap check passes)
2. Increases `creditOf(maker)` by `units`
3. Increases `debtOf(taker)` by `units`
4. Transfers `0` tokens

The maker's credit grows without bound beyond `maxAssets` with no token backing. The contract's loan-token balance does not increase to cover the new credit, breaking the core invariant that contract balances cover credit redemption. At maturity, the maker can attempt to withdraw credit that has no corresponding token deposit, causing insolvency or failed withdrawals for other lenders. [4](#0-3) 

### Likelihood Explanation

**Preconditions:**
1. `offer.buy = true` â€” standard lender offer
2. `offer.maxAssets > 0` â€” maker uses assets-mode cap (common pattern)
3. `offer.tick < MAX_TICK` â€” `buyerPrice < WAD` (any tick below the maximum; the test uses `MAX_TICK - 16`)
4. `units` small enough that `units.mulDivDown(buyerPrice, WAD) == 0` â€” trivially satisfied with `units = 1` at any sub-WAD price
5. `consumed[maker][group] >= maxAssets` â€” offer is fully consumed (either legitimately or via `setConsumed`)

All preconditions are reachable by an unprivileged taker with no special access. The attack is repeatable indefinitely since `consumed` never advances past `maxAssets`. Any taker who observes a fully-consumed buy offer at a sub-WAD tick can execute this. [5](#0-4) 

### Recommendation

Add a guard that reverts when `units > 0` but the computed asset amount is `0` in assets mode. Specifically, after computing `buyerAssets` and `sellerAssets`, require:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    // Prevent zero-asset takes from bypassing the consumed cap
    require(units == 0 || assetsDelta > 0, ZeroAssetsTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any take with `units > 0` must produce a nonzero asset delta, so the consumed counter always advances and the cap is enforced. Alternatively, the consumed check could be based on `units` even in assets mode when `buyerPrice < WAD` causes rounding to zero, but the above targeted guard is the minimal fix. [5](#0-4)
