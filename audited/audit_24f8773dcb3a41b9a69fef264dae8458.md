### Title
Off-by-one `<=` in maturity debt-increase guard allows debt creation at exact maturity timestamp - (`src/Midnight.sol`)

### Summary

The guard on line 391 of `src/Midnight.sol` uses `block.timestamp <= offer.market.maturity`, permitting `sellerDebtIncrease > 0` when `block.timestamp == offer.market.maturity`. The post-maturity liquidation gate on line 622 uses the strict `block.timestamp > market.maturity`, meaning post-maturity liquidation is available starting at `maturity + 1`. A taker can therefore fill a sell offer at the exact maturity second, increasing the seller's debt, and then liquidate the seller in the very next block via post-maturity mode even though the position is collateral-healthy. This directly violates the protocol's own stated invariant that "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."

### Finding Description

**Exact code path:**

`src/Midnight.sol`, `take()`, line 391:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [1](#0-0) 

When `block.timestamp == offer.market.maturity`, the left side of the `||` is `true`, so the require passes unconditionally regardless of `sellerDebtIncrease`. Debt is then written to storage at line 414:
```solidity
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
``` [2](#0-1) 

The end-of-take health check (line 476) only calls `isHealthy`, which checks collateral coverage, not maturity: [3](#0-2) 

So the take succeeds as long as the seller is collateral-healthy after the debt increase.

**Post-maturity liquidation gate** (line 622) uses strict `>`:
```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
``` [4](#0-3) 

This means at `block.timestamp == maturity`, post-maturity liquidation reverts with `NotLiquidatable`. At `block.timestamp == maturity + 1`, it is accepted. This is confirmed by the existing test: [5](#0-4) 

**Exploit flow:**

1. Borrower (seller/maker) signs a sell offer with `expiry >= maturity`.
2. Taker waits until `block.timestamp == offer.market.maturity`.
3. Taker calls `take()` with `units` such that `sellerDebtIncrease > 0` (seller has no existing credit to absorb the units).
4. Line 391 check: `block.timestamp <= maturity` → `true` → passes.
5. Seller's debt increases; health check at line 476 passes (seller is collateral-healthy).
6. In the next block (`block.timestamp == maturity + 1`), taker calls `liquidate(..., postMaturityMode=true)`.
7. Line 622: `block.timestamp > market.maturity` → `true` → liquidation proceeds on a collateral-healthy position.
8. Taker seizes collateral at LIF = 1 (at maturity+1, LIF starts at 1 and ramps to maxLif over 15 minutes), repaying the debt just created.

**Why existing checks fail:**

- The `CannotIncreaseDebtPostMaturity` guard uses `<=` instead of `<`, creating a one-second window where debt can increase but post-maturity liquidation is not yet available to the borrower to defend against.
- The `SellerIsLiquidatable` check at line 476 only enforces collateral health, not maturity-based liquidatability.
- The protocol's own invariant document (`live_context.json`) explicitly lists "timestamp equality" as a forbidden bypass: "maturity boundary must not allow debt increase by rounding, **timestamp equality**, or callback sequencing." [6](#0-5) 

### Impact Explanation

A taker can force a borrower to take on new debt at the exact maturity second. In the immediately following block, the borrower's position is liquidatable via post-maturity mode even though it is collateral-healthy. The borrower has no opportunity to repay between the debt creation and the liquidation (both happen in consecutive blocks). The borrower's collateral is seized at LIF ≥ 1, causing direct loss of collateral value beyond the debt repaid. This matches the scoped impact: the borrower's position is immediately liquidatable post-maturity due to debt created at the maturity boundary.

### Likelihood Explanation

**Preconditions:**
- A sell offer exists with `expiry >= maturity` (common for offers meant to be filled up to maturity).
- The seller has no existing credit to absorb the units (i.e., `sellerCreditDecrease < units`, so `sellerDebtIncrease > 0`).
- The seller has sufficient collateral to pass the health check after the debt increase.

**Feasibility:** On chains with 1-second block times (e.g., post-merge Ethereum, Arbitrum, Base), landing a transaction at exactly `block.timestamp == maturity` is achievable via MEV or simple monitoring. The attacker (taker) controls the timing. The attack is repeatable for any market where a sell offer with `expiry >= maturity` exists.

### Recommendation

Change the debt-increase guard from `<=` to `<`:

```solidity
// Before (line 391):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After:
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [1](#0-0) 

This aligns the debt-increase boundary with the liquidation boundary: debt cannot increase at or after maturity, and post-maturity liquidation is available strictly after maturity. The existing tests `testBuyPastMaturity` and `testSellPastMaturity` use `maturity + 1` as the lower bound and would need a companion test at `block.timestamp == maturity` to cover this boundary. [7](#0-6) 

### Proof of Concept

```solidity
function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;
    // Warp to exact maturity
    vm.warp(market.maturity);

    // Seller offer with expiry == maturity (or higher)
    lenderOffer.expiry = market.maturity;
    lenderOffer.maxUnits = units;
    collateralize(market, borrower, units); // borrower has no credit, so sellerDebtIncrease == units

    // Take succeeds at block.timestamp == maturity: sellerDebtIncrease > 0 passes the <= check
    take(units, borrower, lenderOffer);

    // Assert: debt increased at maturity (violates invariant)
    assertEq(midnight.debtOf(id, borrower), units, "debt increased at maturity");

    // Warp one second forward
    vm.warp(market.maturity + 1);

    // Assert: position is immediately liquidatable via post-maturity mode, even though healthy
    assertTrue(midnight.isHealthy(market, id, borrower), "position is healthy");
    // This should succeed (post-maturity liquidation on a healthy position):
    midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");
    // Expected: succeeds (bug confirmed) — with fix applied, the take above should revert with CannotIncreaseDebtPostMaturity
}
```

**Expected assertions with bug present:** `take()` succeeds, `debtOf(borrower) == units`, `isHealthy == true`, `liquidate(postMaturityMode=true)` succeeds.

**Expected assertions with fix applied:** `take()` reverts with `CannotIncreaseDebtPostMaturity`.
