### Title
Anyone Can Trigger a Flash Loan Callback on Behalf of Arbitrary Receiver Contracts - (File: src/Midnight.sol)

### Summary

The `flashLoan` function in `Midnight.sol` accepts a fully attacker-controlled `callback` address and unconditionally sends tokens to it and invokes `onFlashLoan` on it. Any contract implementing `IFlashLoanCallback` can be forced into executing its `onFlashLoan` logic with attacker-supplied `data`. While `msg.sender` (the attacker's address) is forwarded as the `caller` argument, there is no protocol-level enforcement that `callback == msg.sender`, and no documentation or secure reference implementation exists to guide integrators in validating this field.

### Finding Description

The vulnerable entry point is `flashLoan` in `src/Midnight.sol`:

```solidity
function flashLoan(address[] calldata tokens, uint256[] calldata assets, address callback, bytes calldata data)
    external
{
    require(tokens.length == assets.length, InconsistentInput());
    emit EventsLib.FlashLoan(msg.sender, tokens, assets, callback);
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);   // tokens sent to attacker-chosen address
    }
    require(
        IFlashLoanCallback(callback).onFlashLoan(msg.sender, tokens, assets, data) == CALLBACK_SUCCESS,
        WrongFlashLoanCallbackReturnValue()
    );
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]); // pulled back from callback
    }
}
``` [1](#0-0) 

The `callback` parameter is never validated against `msg.sender`. The `IFlashLoanCallback` interface is:

```solidity
interface IFlashLoanCallback {
    function onFlashLoan(address caller, address[] memory tokens, uint256[] memory assets, bytes memory data) external returns (bytes32);
}
``` [2](#0-1) 

The attack flow is:

1. Attacker calls `midnight.flashLoan(tokens, assets, victimContract, maliciousData)` where `victimContract` is any deployed contract implementing `IFlashLoanCallback`.
2. Midnight transfers `assets[i]` of each token to `victimContract`.
3. Midnight calls `victimContract.onFlashLoan(attacker, tokens, assets, maliciousData)`.
4. If `victimContract` does not validate that `caller == address(victimContract)` (i.e., that it initiated the loan itself) or does not validate `caller` against a trusted set, it executes its flash loan logic with fully attacker-controlled `data`.
5. Midnight then pulls `assets[i]` back from `victimContract` via `safeTransferFrom` â€” requiring `victimContract` to have approved Midnight, which any legitimate flash loan receiver must do.

The `caller` field is the only signal available to the callback to detect this attack. The protocol provides no documentation, no enforcement, and no reference implementation showing how to use it safely. The only available example, `test/FlashloanTest.sol`, does not validate `caller` at all:

```solidity
function onFlashLoan(address caller, address[] memory tokens, uint256[] memory amounts, bytes memory data)
    external
    returns (bytes32)
{
    recordedTokens = tokens;
    recordedAmounts = amounts;
    recordedCaller = caller;   // recorded but never validated
    recordedData = data;
    ...
    return CALLBACK_SUCCESS;
}
``` [3](#0-2) 

### Impact Explanation

Any contract implementing `IFlashLoanCallback` that does not explicitly validate `caller` can be forced to execute its `onFlashLoan` with attacker-controlled `data`. Depending on the callback's logic, this can result in:

- **Unauthorized token approvals**: the callback approves attacker-controlled addresses based on `data`.
- **Unauthorized swaps or protocol interactions**: the callback executes trades or deposits using attacker-supplied parameters.
- **Draining of token balances**: if the callback holds tokens beyond the flash-loaned amount and its logic transfers them based on `data`, those tokens can be stolen.

The repayment pull (`safeTransferFrom(tokens[i], callback, address(this), assets[i])`) means the callback itself does not lose the flash-loaned principal â€” but any side-effects of `onFlashLoan` executing with malicious `data` are fully attacker-controlled.

### Likelihood Explanation

The likelihood is **medium**. The preconditions are:

1. A deployed contract implementing `IFlashLoanCallback` that does not validate `caller` â€” a realistic condition given the absence of any documentation or secure reference implementation in the protocol.
2. The attacker needs no privileged access, no capital at risk, and no special permissions. Any EOA can call `flashLoan`.

The protocol's only example implementation (`test/FlashloanTest.sol`) does not validate `caller`, establishing a dangerous precedent for integrators.

### Recommendation

Enforce that `callback == msg.sender` at the protocol level, eliminating the ability to trigger flash loans on behalf of third-party contracts:

```solidity
function flashLoan(address[] calldata tokens, uint256[] calldata assets, address callback, bytes calldata data)
    external
{
    require(callback == msg.sender, CallbackMustBeCaller()); // add this
    ...
}
```

If triggering flash loans on behalf of other contracts is an intentional design feature, then:

1. Add explicit documentation in `ICallbacks.sol` and `IFlashLoanCallback` warning that `caller` must be validated.
2. Provide a secure reference implementation demonstrating the required `require(caller == address(this), ...)` guard.

### Proof of Concept

**Setup**: Deploy `VulnerableReceiver` implementing `IFlashLoanCallback` that does not validate `caller`, holds 1000 USDC, and has approved Midnight to spend its tokens.

```solidity
contract VulnerableReceiver is IFlashLoanCallback {
    function onFlashLoan(address caller, address[] memory tokens, uint256[] memory assets, bytes memory data)
        external returns (bytes32)
    {
        // No check: require(caller == address(this), "unauthorized");
        address target = abi.decode(data, (address));
        IERC20(tokens[0]).transfer(target, IERC20(tokens[0]).balanceOf(address(this)) - assets[0]);
        IERC20(tokens[0]).approve(msg.sender, assets[0]); // approve repayment
        return CALLBACK_SUCCESS;
    }
}
```

**Attack**:
```solidity
address[] memory tokens = new address[](1);
tokens[0] = address(usdc);
uint256[] memory amounts = new uint256[](1);
amounts[0] = 1; // minimal flash loan

midnight.flashLoan(
    tokens,
    amounts,
    address(vulnerableReceiver),
    abi.encode(attacker) // data instructs receiver to send its balance to attacker
);
// Result: attacker receives VulnerableReceiver's entire USDC balance minus 1
```

The attacker calls `flashLoan` with `callback = vulnerableReceiver`, forcing it to execute `onFlashLoan` with `data = abi.encode(attacker)`. The receiver transfers its balance to the attacker, repays the 1-unit flash loan, and Midnight completes successfully. The attacker has drained the receiver. [1](#0-0) [2](#0-1)
