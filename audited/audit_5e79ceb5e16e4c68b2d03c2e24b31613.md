### Title
Stale `isRootRatified` in `SetterRatifier` persists after authorization revocation, enabling unauthorized offer execution - (File: src/ratifiers/SetterRatifier.sol)

---

### Summary

`SetterRatifier` stores offer-tree approvals in `isRootRatified[maker][root]` and allows any account authorized by the maker to set them. When the maker later revokes that account's authorization via `Midnight.setIsAuthorized`, the `isRootRatified` entry is **not cleared**. Because `SetterRatifier.isRatified()` only checks the stored boolean and never re-validates the setter's current authorization, offers approved by a now-revoked account remain executable indefinitely.

---

### Finding Description

**Root cause — `SetterRatifier.isRatified()` performs no live authorization check:**

`setIsRootRatified` gates writes behind a live `isAuthorized` check:

```solidity
// src/ratifiers/SetterRatifier.sol:24-28
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    ...
}
```

But `isRatified()` only reads the stored boolean — it never re-checks whether the account that set the root is still authorized:

```solidity
// src/ratifiers/SetterRatifier.sol:30-37
function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
    require(msg.sender == MIDNIGHT, NotMidnight());
    ...
    require(isRootRatified[offer.maker][root], NotRatified());   // ← no live authz check
    return CALLBACK_SUCCESS;
}
```

**Contrast with `EcrecoverRatifier`**, which re-validates the signer's authorization at `take()` time:

```solidity
// src/ratifiers/EcrecoverRatifier.sol:44
require(_signer == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer), Unauthorized());
```

Revoking authorization immediately invalidates all `EcrecoverRatifier` signatures. No equivalent protection exists in `SetterRatifier`.

**`Midnight.take()` only checks that the ratifier itself is authorized, not the root setter:**

```solidity
// src/Midnight.sol:355-356
require(isAuthorized[offer.maker][offer.ratifier], RatifierUnauthorized());
require(IRatifier(offer.ratifier).isRatified(offer, ratifierData) == CALLBACK_SUCCESS, RatifierFail());
```

So as long as the maker still authorizes the `SetterRatifier` contract (which they need for their own legitimate offers), any root set by a previously-authorized-but-now-revoked account remains valid.

---

### Impact Explanation

A maker who revokes a malicious operator's authorization believes all of that operator's actions are void. In reality, every Merkle root the operator set via `setIsRootRatified` remains active. Any taker can call `Midnight.take()` against those offers and the call succeeds:

- **Sell offers** (maker is seller): maker is forced into debt at terms the operator chose — potentially at a very low tick (high borrow rate), draining collateral or creating an immediately-liquidatable position.
- **Buy offers** (maker is buyer): maker is forced to lend at unfavorable terms, locking up loan tokens.

The maker cannot easily recover without knowing every root the operator set (requires off-chain event scanning), and must call `setIsRootRatified(maker, root, false)` for each one individually. During the window between revocation and manual cleanup, any taker can exploit the stale roots.

---

### Likelihood Explanation

- Authorizing a third-party account or smart contract to manage offers is a documented, intended use case (see `AUTHORIZATIONS` section in `Midnight.sol` and `testIsRatifiedAuthorizedSetterCanRatifyOnBehalf` in `SetterRatifierTest.sol`).
- A malicious operator can set a harmful root in the same block as (or just before) the maker's revocation transaction, making atomic protection impossible.
- The maker has no on-chain view of which roots were set on their behalf; discovery requires off-chain event monitoring of `SetIsRootRatified` logs.
- No privileged access is required — any account the maker has ever authorized can execute this.

---

### Recommendation

`SetterRatifier.isRatified()` should record **who** set each root and re-verify their authorization at execution time, mirroring `EcrecoverRatifier`'s pattern:

```diff
- mapping(address maker => mapping(bytes32 root => bool)) public isRootRatified;
+ mapping(address maker => mapping(bytes32 root => address)) public rootSetter;

  function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
      require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
-     isRootRatified[maker][root] = newIsRootRatified;
+     rootSetter[maker][root] = newIsRootRatified ? msg.sender : address(0);
      ...
  }

  function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
      ...
-     require(isRootRatified[offer.maker][root], NotRatified());
+     address setter = rootSetter[offer.maker][root];
+     require(setter != address(0), NotRatified());
+     require(
+         setter == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, setter),
+         Unauthorized()
+     );
      return CALLBACK_SUCCESS;
  }
```

This ensures that revoking an operator's authorization in `Midnight` immediately invalidates all roots that operator set, consistent with the behavior of `EcrecoverRatifier`.

### Proof of Concept

**Preconditions:** Alice is a maker using `SetterRatifier`. Bob is a malicious operator.

1. Alice calls `midnight.setIsAuthorized(address(setterRatifier), true, alice)` — authorizes the ratifier.
2. Alice calls `midnight.setIsAuthorized(bob, true, alice)` — authorizes Bob to manage offers.
3. Bob calls `setterRatifier.setIsRootRatified(alice, root, true)` where `root` is a Merkle root containing a sell offer at a very low tick (high borrow rate, unfavorable to Alice). This succeeds because `isAuthorized[alice][bob] == true`.
4. Alice discovers Bob is malicious and calls `midnight.setIsAuthorized(bob, false, alice)`.
5. Alice believes she is safe. `isAuthorized[alice][bob]` is now `false`.
6. **`isRootRatified[alice][root]` is still `true`** — it was never cleared.
7. Any taker calls `midnight.take(aliceSellOffer, ratifierData, units, taker, ...)`.
   - Line 355: `isAuthorized[alice][setterRatifier]` → `true` ✓
   - `SetterRatifier.isRatified()` line 35: `isRootRatified[alice][root]` → `true` ✓
   - Take succeeds. Alice is now a borrower at terms she never approved. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
