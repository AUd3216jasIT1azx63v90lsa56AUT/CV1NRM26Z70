Audit Report

## Title
Missing payer authorization in `repay()` allows arbitrary callback designation to drain third-party loanToken - (File: src/Midnight.sol)

## Summary
`repay()` verifies only that `msg.sender` is authorized to act on behalf of `onBehalf` (the debtor), but imposes no authorization check on the `callback` parameter, which becomes the token payer. Any account authorized to repay on behalf of a debtor can therefore designate an arbitrary victim address as `callback`, causing Midnight to invoke `victim.onRepay(...)` and then pull loanToken directly from the victim. This is concretely reachable from within the `onSell` callback during `take()`, where a seller's authorized callback contract can re-enter `repay()` with `callback=victim` to retire the seller's debt at the victim's expense.

## Finding Description

**Root cause — `repay()` sets payer from unchecked `callback` parameter:**

```solidity
// src/Midnight.sol:505
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
// ...
address payer = callback != address(0) ? callback : msg.sender;   // payer = victim
// ...
IRepayCallback(callback).onRepay(id, market, units, onBehalf, data);  // called on victim
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // pulls from victim
``` [1](#0-0) 

The only gate is that `msg.sender` must be authorized to act on behalf of `onBehalf`. There is no check that `callback` (the payer) has authorized `msg.sender` to use it as a payer.

**Exploit flow:**

1. Attacker deploys `maliciousContract` and creates a sell offer with `offer.callback = maliciousContract`.
2. Attacker (seller) calls `setIsAuthorized(maliciousContract, true, seller)` — standard precondition for any callback that needs to act on the seller's behalf.
3. Any taker calls `take(offer, ...)`.
4. Inside `take()`, `sellerCallback = offer.callback = maliciousContract` (line 421), and the liquidation lock is set for the seller (line 444). [2](#0-1) 

5. After the buyer-side token pull, `onSell` is invoked on `maliciousContract` (lines 458–473). [3](#0-2) 

6. Inside `onSell`, `maliciousContract` calls `repay(market, units, seller, victim, data)`:
   - `msg.sender = maliciousContract`; `isAuthorized[seller][maliciousContract] = true` → authorization check at line 505 passes.
   - `payer = victim` (line 511).
   - `IRepayCallback(victim).onRepay(...)` is called; if victim returns `CALLBACK_SUCCESS`, execution continues.
   - `safeTransferFrom(loanToken, victim, midnight, units)` pulls `units` loanToken from victim (line 520).
7. Seller's debt is reduced. The post-callback health check at line 476 passes because the debt was repaid. [4](#0-3) 

**Why existing checks fail:**

- The `repay` authorization check (`isAuthorized[onBehalf][msg.sender]`) only verifies the right to repay on behalf of the seller; it says nothing about who may be designated as payer.
- The liquidation lock prevents the seller from being liquidated during the callback window but does not restrict what the callback may do to third parties.
- The Certora spec `OnlyExplicitPayerCanLoseTokens.spec` sets `repayCallbackAllowed = false` for `take` (line 108), which would flag a violation if the re-entrant `repay` call were modeled. However, `onSell` is not listed in the methods block and defaults to `HAVOC_ALL` per the comment at lines 13–14: *"onRatify and onSell cannot authorize a payer, so we model them with a plain HAVOC_ALL"*. `HAVOC_ALL` havoces storage variables but does not update the persistent ghost variables (`repayCallbackAllowed`, `badPullSeen`), so the tool never explores the re-entrant `repay` path from within `onSell`. The assumption that `onSell` cannot authorize a payer is incorrect. [5](#0-4) [6](#0-5) 

## Impact Explanation

Direct theft of loanToken from any victim contract that (a) implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS`, (b) has approved Midnight to spend its loanToken, and (c) holds a loanToken balance. The victim loses `units` loanToken; the attacker's (seller's) debt is retired at zero cost. This is an unauthorized movement of assets from a third party, satisfying the critical impact class of direct theft.

## Likelihood Explanation

**Required preconditions:**
1. Seller authorizes their callback contract — standard practice for any callback that calls `supplyCollateral` or similar on the seller's behalf.
2. Victim contract implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS` without validating `onBehalf` — realistic for any Midnight-integrated aggregator or vault that uses repay callbacks for its own operations.
3. Victim has approved Midnight to spend loanToken and holds a loanToken balance — true for any active repay-callback contract.

All three preconditions are normal operational states for Midnight integrators. The attack is repeatable as long as the victim contract remains deployed and approved. The attacker controls the entire setup off-chain before deployment.

## Recommendation

Add an authorization check on the `callback` parameter in `repay()`, requiring that either `callback == msg.sender` or `isAuthorized[callback][msg.sender]` is true before designating `callback` as the payer. This mirrors the pattern already used for `onBehalf` authorization and ensures that only a callback that has explicitly authorized the caller can be used as a payer.

```solidity
if (callback != address(0)) {
    require(callback == msg.sender || isAuthorized[callback][msg.sender], Unauthorized());
    // ...
}
```

Additionally, update the Certora spec `OnlyExplicitPayerCanLoseTokens.spec` to model `onSell` with a summary that can set `repayCallbackAllowed = true`, rather than `HAVOC_ALL`, so that re-entrant `repay` calls from within `onSell` are properly explored.

## Proof of Concept

**Minimal manual steps:**

1. Deploy `MaliciousCallback` implementing `ISellCallback.onSell` and `IRepayCallback.onRepay` (returns `CALLBACK_SUCCESS`).
2. Deploy `Victim` implementing `IRepayCallback.onRepay` (returns `CALLBACK_SUCCESS`) with Midnight approved to spend its loanToken balance.
3. Attacker (seller) calls `setIsAuthorized(MaliciousCallback, true, seller)`.
4. Attacker creates a sell offer with `offer.callback = MaliciousCallback`.
5. Any taker calls `take(offer, ...)`.
6. Inside `MaliciousCallback.onSell`, call `midnight.repay(market, units, seller, Victim, data)`.
7. Observe: `units` loanToken pulled from `Victim`; seller's debt reduced by `units`; `Victim` balance decreases.

**Invariant/fuzz test plan:** Write a Foundry invariant that asserts no address other than `msg.sender` or an explicitly self-authorizing callback loses loanToken balance across any sequence of `take` + `repay` calls. The invariant will be violated by the above sequence.
