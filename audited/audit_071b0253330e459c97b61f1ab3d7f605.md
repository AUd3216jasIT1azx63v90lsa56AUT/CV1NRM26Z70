Audit Report

## Title
Post-maturity bad-debt liquidation reverts with arithmetic underflow when `lif < maxLif` and `seizedAssets > 0` - (File: src/Midnight.sol)

## Summary
When a position carries bad debt and is liquidated in post-maturity mode during the 15-minute `TIME_TO_MAX_LIF` ramp window, the bad-debt write-down uses `maxLif` to reduce `_position.debt`, but `repaidUnits` is then computed with the smaller ramped `lif`. Because dividing by a smaller denominator yields a strictly larger `repaidUnits`, the subsequent unchecked subtraction at line 676 underflows and reverts, making the position permanently un-liquidatable via the `seizedAssets` path for the entire ramp window.

## Finding Description

**Root cause:** Two different LIF values are used in the same liquidation call without reconciliation.

**Step 1 — Bad-debt write-down uses `maxLif` (lines 614–616):** [1](#0-0) 

After the loop, if `badDebt > 0`, `_position.debt` is reduced at line 628: [2](#0-1) 

The resulting `_position.debt ≈ Σ collateral[i].mulDivUp(price_i, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif_i)`.

**Step 2 — `lif` is computed as a linear ramp (lines 645–647):** [3](#0-2) 

`TIME_TO_MAX_LIF = 15 minutes` per `ConstantsLib.sol`: [4](#0-3) 

When `block.timestamp - market.maturity < TIME_TO_MAX_LIF`, `lif < maxLif`.

**Step 3 — `repaidUnits` computed with smaller `lif` (line 650):** [5](#0-4) 

Dividing by `lif < maxLif` yields `repaidUnits > _position.debt`.

**Step 4 — Unchecked subtraction reverts (line 676):** [6](#0-5) 

**Why existing guards fail:** The `maxRepaid`/RCF check that would otherwise cap `repaidUnits` is gated behind `if (!postMaturityMode)` and is entirely skipped in the post-maturity branch: [7](#0-6) 

There is no equivalent upper-bound guard for `repaidUnits` in the post-maturity path.

**Concrete numeric example** (`ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`, single collateral):

| Parameter | Value |
|---|---|
| `collateral` | 1 000 |
| `price` | `ORACLE_PRICE_SCALE` |
| `maxLif` | `1.1e18` |
| `lif` (early ramp) | `1.05e18` |
| `originalDebt` | 1 000 |

- `badDebt = 1000 − ⌈1000/1.1⌉ = 1000 − 910 = 90`
- `_position.debt` after line 628 = **910**
- `repaidUnits = ⌈1000/1.05⌉ = 953`
- Line 676: `910 − 953` → **arithmetic underflow → revert**

## Impact Explanation
Any liquidator calling `liquidate` with `seizedAssets > 0` on a position with bad debt during the 15-minute post-maturity ramp window receives a guaranteed revert. The position cannot be liquidated via the `seizedAssets` path for the entire ramp window, violating the core protocol invariant that unhealthy positions must always remain liquidatable. This constitutes a temporary but deterministic freeze of the liquidation mechanism for affected positions, with direct risk of protocol insolvency if bad-debt positions cannot be cleared.

## Likelihood Explanation
All three preconditions are permissionless and time-based: market maturity is reached naturally, bad debt arises from normal price movements, and the 15-minute window is precisely when liquidation incentives are lowest (lif is near WAD) and liquidators are most likely to seize all available collateral via `seizedAssets`. The bug is deterministic and repeatable for any parameter set satisfying `badDebt > 0` and `lif < maxLif`.

## Recommendation
In the post-maturity `seizedAssets` branch, cap `repaidUnits` to `_position.debt` before the subtraction, or compute `repaidUnits` using `maxLif` (consistent with the bad-debt write-down) rather than the ramped `lif`. The simplest safe fix:

```solidity
// After line 650:
if (repaidUnits > _position.debt) repaidUnits = _position.debt;
```

Alternatively, use `_maxLif` instead of `lif` when computing `repaidUnits` in the post-maturity bad-debt case, so both computations use the same denominator.

## Proof of Concept
Deploy a market with a single collateral (`maxLif = 1.1e18`, `lltv` any allowed value). Create a borrower position with `debt = 1000` and `collateral = 1000` at `price = ORACLE_PRICE_SCALE`. Advance time past `market.maturity` but less than `market.maturity + 15 minutes`. Call `liquidate` with `postMaturityMode = true`, `seizedAssets = 1000`. The transaction reverts with an arithmetic underflow at line 676. The same call succeeds if `block.timestamp >= market.maturity + 15 minutes` (where `lif == maxLif`), confirming the ramp window as the trigger.
