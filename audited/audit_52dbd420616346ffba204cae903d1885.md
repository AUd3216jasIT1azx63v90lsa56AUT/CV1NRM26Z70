Audit Report

## Title
Taker-Controlled `takerCallback` Can Coerce Maker to Fund Buyer's Payment on Sell Offers - (File: src/Midnight.sol)

## Summary
In `take`, for a sell offer (`offer.buy = false`), `buyerCallback` is assigned directly from the unconstrained taker-supplied `takerCallback` parameter. Setting `takerCallback = offer.maker` causes `payer` to be set to `offer.maker`, and both token transfer calls pull from the maker rather than the taker. The taker receives a credited position at zero cost while the maker's token balance is drained.

## Finding Description

**Root cause:** `takerCallback` is an unconstrained taker-supplied parameter. At line 420, for sell offers, it is assigned directly to `buyerCallback`:

```solidity
address buyerCallback = offer.buy ? offer.callback : takerCallback;  // line 420
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);  // line 422
```

Setting `takerCallback = offer.maker` (non-zero) therefore sets `payer = offer.maker`.

**Role assignment:** At line 375, for sell offers, `buyer = taker` and `seller = offer.maker`. The taker's position receives credit (line 410), and the maker's position receives debt (line 414) — the normal sell-offer accounting.

**Callback gate:** Lines 447–452 call `IBuyCallback(buyerCallback).onBuy(...)` passing `buyer = taker` (not `offer.maker`). If `offer.maker`'s `onBuy` implementation does not assert `buyer == address(this)`, it returns `CALLBACK_SUCCESS` and the gate passes. After this, `callbackAllowed = true` and `callback = offer.maker` in the Certora spec's ghost state.

**Token pulls:** Lines 455–456 then pull the full `buyerAssets` from `payer = offer.maker`:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);  // line 455
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);  // line 456
```

The taker's credit is increased by `buyerCreditIncrease` (line 410) with zero token outflow from the taker.

**Why existing checks fail:**
- `require(offer.maker != taker, SelfTake())` at line 354 does not prevent `takerCallback = offer.maker`.
- There is no check that `takerCallback` must be the taker's own address or authorized by the taker.
- The `onBuy` return-value check is the only gate, but it is satisfied by any `IBuyCallback` implementation that does not verify `buyer == address(this)`.
- The Certora `takeOnlyExplicitPayer` rule in `certora/specs/OnlyExplicitPayerCanLoseTokens.spec` (lines 95–115) explicitly classifies this as valid: after `onBuy` returns `CALLBACK_SUCCESS`, it sets `callbackAllowed = true` and `callback = offer.maker`, so `CVL_transferFrom` permits the pull at line 80–82. The spec proves tokens are only pulled from "explicit payers" but does not prove the maker cannot be coerced into that role by the taker supplying `takerCallback = offer.maker`.

## Impact Explanation
A taker taking a sell offer with `takerCallback = offer.maker` causes the maker's tokens to be pulled as the buyer's payment. The taker receives a credited position worth approximately `units` loan tokens for free. The maker suffers a net token loss equal to `buyerAssets - sellerAssets` (the settlement fee portion burned to Midnight), while `sellerAssets` is sent to `offer.receiverIfMakerIsSeller`. Additionally, the maker's debt is increased by `sellerDebtIncrease` units as part of normal sell-offer accounting. This constitutes direct theft of maker funds by an unprivileged taker.

## Likelihood Explanation
**Required preconditions:**
1. `offer.maker` is a contract (EOA has no code; `onBuy` call reverts).
2. `offer.maker` implements `IBuyCallback.onBuy` and returns `CALLBACK_SUCCESS` without asserting `buyer == address(this)`.
3. `offer.maker` holds a token allowance to the Midnight contract.

These conditions are realistic for protocol-integrated vault or market-maker contracts that both post sell offers and participate as buyers (e.g., a lending vault with a standing approval). The attack requires no privileged access, is repeatable against any qualifying maker contract, and is executable by any external caller.

## Recommendation
Constrain `takerCallback` so it cannot be set to an address that is not the taker or an address authorized by the taker. Specifically, add a check such as:

```solidity
require(takerCallback == address(0) || takerCallback == taker || isAuthorized[taker][takerCallback], InvalidTakerCallback());
```

This ensures the taker can only designate their own callback, preventing them from nominating the maker's address as the payer. Alternatively, validate inside `onBuy` that `buyer` matches the callback contract's own address, but this is a per-integrator mitigation and not a protocol-level fix.

## Proof of Concept
**Setup:**
- Deploy a mock `MakerVault` contract that implements both `IBuyCallback` (for buy offers it posts) and posts sell offers. Its `onBuy` returns `CALLBACK_SUCCESS` without checking `buyer == address(this)`. It has a token allowance to Midnight.
- `MakerVault` posts a sell offer with `offer.buy = false`.

**Attack steps:**
1. Attacker calls `take(offer, ratifierData, units, attacker, attackerReceiver, offer.maker, "")`.
2. `buyerCallback = takerCallback = offer.maker` (line 420).
3. `payer = offer.maker` (line 422).
4. `buyer = attacker`, `seller = offer.maker` (line 375).
5. `attacker`'s credit increases by `buyerCreditIncrease` (line 410).
6. `IBuyCallback(offer.maker).onBuy(..., buyer=attacker, ...)` is called; `MakerVault.onBuy` returns `CALLBACK_SUCCESS` without checking `buyer`.
7. `safeTransferFrom(loanToken, offer.maker, address(this), buyerAssets - sellerAssets)` — maker pays settlement fee (line 455).
8. `safeTransferFrom(loanToken, offer.maker, offer.receiverIfMakerIsSeller, sellerAssets)` — maker pays seller assets to their own receiver (line 456).
9. Attacker holds `buyerCreditIncrease` units of credit, withdrawable for loan tokens, at zero cost.

**Expected result:** Attacker's credit balance increases; `MakerVault`'s token balance decreases by `buyerAssets`; `MakerVault` receives `sellerAssets` back to its receiver; net theft = taker's credit gain funded entirely by maker.
