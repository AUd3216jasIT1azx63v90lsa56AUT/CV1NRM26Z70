Audit Report

## Title
`maxAssets` cap permanently bypassed via zero-rounding of `buyerAssets` on buy offers with `offerPrice < WAD` - (File: `src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and `offerPrice < WAD`, calling `take()` with `units = 1` computes `buyerAssets = mulDivDown(1, offerPrice, WAD) = 0`. Because `consumed` is incremented by `buyerAssets` (which is 0), the cap check at line 369 never advances and always passes, allowing the offer to be taken an unlimited number of times. A taker can exploit this to accumulate unbounded debt at zero token cost, then default at maturity to force bad-debt socialization onto all lenders via the slashing mechanism.

## Finding Description

**Root cause — `src/Midnight.sol`, `take()`, lines 363–369:**

```solidity
// Line 363
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // mulDivDown(1, offerPrice, WAD) = 0 when offerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

// Lines 367–369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    //                                                              ^^^^^^^^^^^ += 0 every call
    require(newConsumed <= offer.maxAssets, ConsumedAssets());  // trivially passes forever
}
```

For any tick where `offerPrice < WAD` (i.e., tick > `MAX_TICK/2`), `mulDivDown(1, offerPrice, 1e18) = 0`. The `consumed` mapping never advances, so `maxAssets` is never enforced regardless of how many times the offer is taken.

**Why existing checks fail:** The sole guard is `require(newConsumed <= offer.maxAssets)` at line 369. Since `newConsumed` is unchanged on every iteration (incremented by 0), this check trivially passes indefinitely.

**Downstream consequence:** With `buyerAssets = sellerAssets = 0`, the token transfers at lines 455–456 move zero tokens. However, the taker's (seller's) debt is still incremented by `sellerDebtIncrease` at line 414 (`sellerPos.debt += sellerDebtIncrease`). The taker accumulates unbounded on-chain debt with zero loan-token receipt. At maturity, this unpaid debt is realized as bad debt and socialized across all lenders via proportional credit slashing (NatSpec lines 80–82).

**Protocol acknowledgment:** The NatSpec at line 94 explicitly states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` confirms this is a recognized bug, not an intentional design choice.

## Impact Explanation

An attacker (taker) can loop `take(units=1)` against any qualifying buy offer at zero token cost, accumulating arbitrarily large debt positions with no corresponding loan-token receipt. At maturity, the entire unpaid debt is realized as bad debt and socialized among all lenders in the market via proportional credit slashing. This constitutes direct, unbounded loss to lenders — a critical accounting integrity failure. The `maxAssets` cap, the protocol's primary mechanism for limiting offer exposure, is rendered completely ineffective for the affected tick range (any tick > `MAX_TICK/2`).

## Likelihood Explanation

**Preconditions:**
- A buy offer exists with `maxAssets > 0` and any tick in `(MAX_TICK/2, MAX_TICK]` (i.e., `offerPrice < WAD`). This is a normal, valid configuration covering half of all valid ticks.
- The attacker is any unprivileged taker — no special role, no admin access, no oracle manipulation required.
- The offer must not be expired and must pass ratifier checks (standard conditions).

**Feasibility:** Fully reachable on-chain. The loop cost is gas only. Each iteration is independent and leaves state ready for the next call. The condition is permanent until the offer expires or is cancelled. The existence of `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` provides a ready-made reproduction path.

## Recommendation

Replace the rounding mode used for consumed tracking on buy offers, or add an explicit floor of 1 when `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 consumedDelta = offer.buy
        ? units.mulDivUp(buyerPrice, WAD)   // use mulDivUp so 1 unit always consumes ≥ 1 asset
        : sellerAssets;
    require(consumedDelta > 0 || units == 0, ZeroAssetsTaken());
    newConsumed = consumed[offer.maker][offer.group] += consumedDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, add a pre-check `require(buyerAssets > 0 || units == 0)` before the cap logic to revert any take that would contribute zero assets to a `maxAssets`-capped offer.

## Proof of Concept

The test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already encodes the reproduction path. Minimal manual steps:

1. Deploy a market with a buy offer at any tick where `offerPrice < WAD` (e.g., tick = `MAX_TICK`), with `maxAssets = 1e18` and `maxUnits = 0`.
2. As an unprivileged taker, call `take(offer, 1, taker, ...)` in a loop (e.g., 1000 iterations).
3. Observe: `consumed[maker][group]` remains 0 after every call; `sellerPos.debt` of the taker increases by 1 per call; zero loan tokens are transferred.
