Audit Report

## Title
`maxAssets` Cap Bypass via `mulDivDown` Zero-Rounding on Buy Offers with `buyerPrice < WAD` Allows Unbounded Credit Accumulation Without Asset Transfer - (File: `src/Midnight.sol`)

## Summary
When `offer.buy = true` and `offer.maxAssets > 0`, the `take` function increments `consumed[maker][group]` by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `tickToPrice` is formally proven to return `<= WAD` for all valid ticks, any tick below `MAX_TICK` yields `buyerPrice < WAD`, causing `buyerAssets = 0` for `units = 1`. Each such fill leaves `consumed` unchanged while still mutating `credit`, `debt`, and `totalUnits`, allowing the `maxAssets` cap to be bypassed entirely across arbitrarily many fills — including fills against an already fully-consumed offer.

## Finding Description

**Root cause — `src/Midnight.sol` lines 358–373:**

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);          // <= WAD always
uint256 buyerPrice = sellerPrice + _settlementFee;             // == offerPrice for buy offers
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)                        // rounds DOWN → 0 when units < WAD/buyerPrice
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());  // trivially passes when buyerAssets == 0
}
```

The Certora rule `tickToPriceAtMostWad` (`certora/specs/TickToPrice.spec` line 48) formally proves `tickToPrice(tick) <= 10^18` for all ticks. For any tick below `MAX_TICK`, `offerPrice < WAD`, so `mulDivDown(1, buyerPrice, WAD) = floor(buyerPrice / WAD) = 0`. The `consumed` increment is 0, the cap check `require(0 <= maxAssets)` trivially passes, yet the position mutation at lines 408–417 proceeds unconditionally: `buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` all increase by the full `units` amount.

**Exploit flow:**
1. Maker creates a buy offer: `offer.buy = true`, `offer.maxAssets = X`, `offer.tick < MAX_TICK` (any non-par price).
2. Attacker (maker + separate taker address, bypassing `SelfTake`) calls `take(offer, ..., units=1, ...)` in a loop N times.
3. Each iteration: `buyerAssets = 0`, `consumed += 0`, cap check passes. Maker gains 1 unit of credit; taker gains 1 unit of debt. Zero loan tokens transferred (`SafeTransferLib.safeTransferFrom(..., 0)` is a no-op).
4. After N iterations: `consumed` is unchanged, but the maker holds N units of unbacked credit and the taker holds N units of debt with no corresponding loan token deposit.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` cannot fire when the increment is 0.
- `SelfTake` blocks `offer.maker == taker` but not a maker-controlled second address.
- The `makerFavorableRounding` Certora rule (`certora/specs/SettlementFeeSpread.spec` line 45: `assert offer.buy => buyerAssets * WAD() <= units * offerPrice`) explicitly permits `buyerAssets = 0` as favorable to the maker, so no formal property catches this.
- The protocol's own NatSpec at `src/Midnight.sol` line 94 documents the behavior: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1"* — confirming awareness but not remediation.

**Test proof — `test/TakeTest.sol` lines 857–889** (`testBugBuyMaxAssetsBypass`): pre-fills `consumed` to `maxAssets`, calls `take(1, borrower, lenderOffer)`, then asserts `buyerAssets == 0`, `consumed` unchanged, yet `creditOf`, `debtOf`, and `totalUnits` all strictly increased.

## Impact Explanation

The maker (buyer/lender) accumulates an unbounded credit position without depositing any loan tokens. When the maker later redeems credit, the contract will be short the corresponding loan tokens, constituting a direct theft of other depositors' funds. The `claimableSettlementFee` accounting is also corrupted: `buyerAssets - sellerAssets = 0` is added per fill, so settlement fees are never collected despite real unit-level exposure being created. This is a concrete accounting integrity failure with direct asset-theft potential, not a theoretical issue.

## Likelihood Explanation

Preconditions are trivially satisfied: `offer.buy = true` and any tick below `MAX_TICK` (the entire usable price range except the single par tick). The two-address requirement (maker + separate taker) is a minimal bar — two EOAs or an EOA plus a contract suffice. The taker must have collateral for the debt, but the maker controls the taker and can pre-fund it. The attack is repeatable indefinitely within a single block via `multicall`, making the unbounded accumulation achievable atomically.

## Recommendation

Replace the `mulDivDown` rounding for buy-offer `consumed` tracking with `mulDivUp`, or add a `require(units == 0 || buyerAssets > 0, ZeroAssets())` guard before the `consumed` increment. Alternatively, track `consumed` in units (not assets) for buy offers, converting to assets only for the cap comparison. The fix must ensure that any fill with `units > 0` produces a strictly positive `consumed` increment so the cap is enforceable.

        lenderOffer.maxAssets = 1;

        vm.prank(lender);

        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
