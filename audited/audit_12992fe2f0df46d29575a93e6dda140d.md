### Title
Fully-consumed buy offer with `buyerPrice < WAD` can be re-taken indefinitely via zero-`buyerAssets` rounding bypass - (`src/Midnight.sol`)

### Summary
When `offer.buy == true`, `offer.maxAssets > 0`, and `buyerPrice < WAD`, the consumed-accounting increment adds `mulDivDown(units, buyerPrice, WAD)` to `consumed`. For sufficiently small `units` (specifically `units * buyerPrice < WAD`), this rounds down to zero. Once `consumed == maxAssets`, any subsequent `take` with such small `units` adds 0 to `consumed`, passes the `newConsumed <= maxAssets` check, and executes fully — modifying position credit/debt state — without being blocked. The offer is reusable an unlimited number of times after it is nominally exhausted.

### Finding Description

**Code path and root cause:**

In `src/Midnight.sol` lines 363 and 368–369:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// ...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For a buy offer, `buyerAssets = mulDivDown(units, buyerPrice, WAD)` (from `UtilsLib.mulDivDown`, line 29–31 of `src/libraries/UtilsLib.sol`). When `buyerPrice < WAD` — which is reachable by any maker setting a tick such that `tickToPrice(tick) < WAD` — and `units < WAD / buyerPrice`, the result is 0. The consumed increment becomes `consumed += 0`, so `newConsumed` is unchanged. If `consumed` was already at `maxAssets`, the check `require(maxAssets <= maxAssets)` trivially passes and the take proceeds.

**Attacker-controlled inputs:**
- `offer.tick`: set low enough that `tickToPrice(tick) < WAD` (maker-controlled at offer creation; any taker can exploit any such offer)
- `units`: any value satisfying `units * buyerPrice < WAD` (e.g., `units = 1` when `buyerPrice < WAD`)

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = N` and `tick` such that `buyerPrice < WAD`.
2. Offer is legitimately filled until `consumed == maxAssets` (or maker calls `setConsumed` directly).
3. Taker calls `take(offer, ..., units=1)`. Since `mulDivDown(1, buyerPrice, WAD) == 0`, `consumed` stays at `maxAssets`, the check passes, and the take executes.
4. Position state changes: `buyerPos.credit` and `sellerPos.debt` increase by `units = 1` each call.
5. No tokens are transferred (`buyerAssets = sellerAssets = 0`), so the taker pays nothing.
6. Step 3–5 can be repeated indefinitely.

**Why existing checks fail:**
The `ConsumedAssets` check at line 369 only guards against `newConsumed > maxAssets`. It does not guard against `units > 0` when `buyerAssets == 0`. The Certora spec `fullyConsumedOfferRevertsOnNonTrivialTake` (line 99–111 of `certora/specs/Consume.spec`) only covers `maxAssets == 0` (units mode); there is no equivalent rule for assets mode. The `takeConsumedAtMaxUnchangedAssets` rule (lines 88–97) is vacuously satisfied because consumed indeed does not change — but the take still executes with `units > 0`.

### Impact Explanation
A fully-consumed buy offer (consumed == maxAssets) with `buyerPrice < WAD` can be taken an unlimited number of times with `units > 0`. Each such take increases the buyer's credit and the seller's debt by `units` without any token payment. This allows a taker to inflate credit/debt positions for free, bypassing the offer's intended fill cap entirely. The invariant "offers cannot be replayed, overfilled, reused, or filled after cancel/deadline" is violated.

### Likelihood Explanation
The precondition `buyerPrice < WAD` is reachable whenever `offerPrice < WAD`, which is a normal and common configuration (any tick below the WAD-price tick). The offer does not need to be maliciously crafted — any legitimate buy offer with a low enough price is vulnerable once consumed reaches `maxAssets`. The exploit requires only a single `take` call with `units = 1` and no special permissions. It is repeatable indefinitely in the same transaction or across blocks.

### Recommendation
Add a guard that prevents a non-zero-unit take from proceeding when the assets contribution rounds to zero. Concretely, in the `offer.maxAssets > 0` branch, add:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetsTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any take with `units > 0` must contribute at least 1 asset unit to consumed, making it impossible to bypass the cap via rounding.

### Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) is a direct PoC. It:

1. Sets `lenderOffer.maxAssets = 1` and `lenderOffer.tick = MAX_TICK - 16` (so `buyerPrice < WAD`).
2. Pre-fills consumed to `maxAssets` via `setConsumed`.
3. Calls `take(1, borrower, lenderOffer)` — `units = 1`, `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`.
4. Asserts `buyerAssets == 0`, `sellerAssets == 0`, `consumed == maxAssets` (unchanged), but `creditOf(lender) > lenderCreditBefore` and `debtOf(borrower) > borrowerDebtBefore`.

A fuzz extension:

```solidity
function testFuzz_BuyMaxAssetsBypassRepeated(uint256 reps) public {
    reps = bound(reps, 1, 1000);
    // setup: lenderOffer.maxAssets = 1, tick s.t. buyerPrice < WAD

    // Assert: consumed never exceeded maxAssets
    assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
}
```

```text
/// If consumed is already at or above maxAssets before a take in assets mode, it remains unchanged.
rule takeConsumedAtMaxUnchangedAssets(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxUnits == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumedBefore >= offer.maxAssets => consumed(offer.maker, offer.group) == consumedBefore;
}
```

```text
/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
```
