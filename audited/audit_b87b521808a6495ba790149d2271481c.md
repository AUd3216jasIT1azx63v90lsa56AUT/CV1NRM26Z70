Audit Report

## Title
Missing bounds check in `repay` enables front-run underflow DoS for unhealthy borrowers - (`src/Midnight.sol`)

## Summary
The `repay` function performs a checked subtraction `position[id][onBehalf].debt -= UtilsLib.toUint128(units)` with no guard that `units ≤ debt`. An attacker can front-run any `repay(market, D, …)` call with a partial `liquidate` that reduces the borrower's debt below `D`, causing the repay to revert via Solidity 0.8 checked-arithmetic underflow. The attacker earns the liquidation incentive on every front-run, making the attack self-funding and repeatable, permanently denying unhealthy borrowers the ability to repay.

## Finding Description

**Root cause — missing bounds check in `repay`:**

`repay` performs a direct checked subtraction with no prior validation:

```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
``` [1](#0-0) 

`UtilsLib.toUint128` only guards against cast overflow (`x <= type(uint128).max`); it does not enforce `units ≤ debt`:

```solidity
function toUint128(uint256 x) internal pure returns (uint128) {
    require(x <= type(uint128).max, CastOverflow());
    return uint128(x);
}
``` [2](#0-1) 

**State-modifying paths in `liquidate` that write to the same debt slot:**

Both the bad-debt path and the normal repaid-units path reduce `_position.debt` before any external call, committing the debt reduction to storage:

```solidity
_position.debt -= uint128(badDebt);          // line 628
...
_position.debt -= UtilsLib.toUint128(repaidUnits);  // line 676
``` [3](#0-2) [4](#0-3) 

**Exploit flow:**
1. Borrower has unhealthy position: `debt = D`, position is liquidatable.
2. Borrower broadcasts `repay(market, D, onBehalf, …)`.
3. Attacker sees the transaction in the mempool and front-runs with `liquidate(…, repaidUnits=1, borrower, …)`.
4. Liquidation succeeds: `_position.debt` is now `D − 1`.
5. Borrower's `repay` lands: `(D−1) − D` underflows → **revert**.
6. Attacker received collateral worth `1 * lif / WAD` in step 3, paying only 1 loan token — net profit from the liquidation incentive.
7. Attacker repeats on every retry, continuously front-running with `repaidUnits = 1`.

**Why existing checks do not stop it:**

The only check in `repay` is the authorization guard, which is irrelevant to the underflow: [5](#0-4) 

The `liquidatorGate` check in `liquidate` is only present when the market creator sets a non-zero gate; permissionless markets have `liquidatorGate == address(0)`, bypassing this guard entirely: [6](#0-5) 

The `liquidationLocked` check uses transient storage scoped to a single transaction's callback reentrancy — it does not prevent a separate front-running transaction: [7](#0-6) 

**Test gap confirmed:**

`testRepay` always bounds `repaid ≤ units`, so the underflow path is never exercised by the existing test suite:

```solidity
repaid = bound(repaid, 0, units);
``` [8](#0-7) 

## Impact Explanation
Any borrower whose position is unhealthy (liquidatable) can be permanently denied the ability to repay via `repay`. Every `repay(market, D, onBehalf, …)` call reverts as long as the attacker front-runs with even a 1-unit liquidation. The borrower is forced into full liquidation, losing the liquidation incentive premium on their collateral rather than repaying at par. This constitutes a concrete, repeatable DoS against the `repay` function for any unhealthy borrower in any market without a `liquidatorGate`, with direct financial loss to the victim.

## Likelihood Explanation
**Preconditions:** (a) borrower's position is unhealthy (price dropped below LLTV threshold), (b) market has no `liquidatorGate` (the default for permissionless markets). Both are routine on-chain states. **Feasibility:** front-running is standard on EVM chains with a public mempool; the attacker needs only to submit a higher-gas-price `liquidate` with `repaidUnits = 1`. **Profitability:** the attacker earns the liquidation incentive (`lif > WAD`) on every front-run, so the attack is net-positive and self-funding. **Repeatability:** the attack can be repeated on every retry until the borrower's collateral is fully seized.

## Recommendation
Add a bounds check in `repay` to cap `units` at the borrower's current debt, or use `zeroFloorSub` to make the subtraction saturating. For example:

```solidity
uint128 currentDebt = position[id][onBehalf].debt;
uint128 repayUnits = UtilsLib.toUint128(UtilsLib.min(units, currentDebt));
position[id][onBehalf].debt = currentDebt - repayUnits;
marketState[id].withdrawable += repayUnits;
```

Alternatively, add an explicit require before the subtraction:

```solidity
require(units <= position[id][onBehalf].debt, RepayExceedsDebt());
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
```

## Proof of Concept
Minimal Foundry test plan:

```solidity
function testRepayUnderflowFrontRun() public {
    uint256 units = 1e18;
    // 1. Set up borrower with unhealthy position (debt = units)
    collateralize(market, borrower, units);
    setupMarket(market, units);
    // 2. Drop oracle price to make position unhealthy
    oracle1.setPrice(ORACLE_PRICE_SCALE / 2);
    // 3. Attacker front-runs with repaidUnits = 1
    deal(address(loanToken), attacker, 1);
    vm.prank(attacker);
    midnight.liquidate(market, 0, 0, 1, borrower, false, attacker, address(0), "");
    // 4. Borrower's repay of full debt now reverts
    deal(address(loanToken), borrower, units);
    vm.prank(borrower);
    vm.expectRevert(); // arithmetic underflow
    midnight.repay(market, units, borrower, address(0), "");
}
```
