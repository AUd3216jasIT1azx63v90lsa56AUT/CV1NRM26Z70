### Title
Maker-Controlled `offer.market` Passed Unverified to `onBuy` Callback Enables Gate Spoofing — (`src/interfaces/ICallbacks.sol` / `src/Midnight.sol`)

### Summary
In `Midnight.take()`, the `market` parameter forwarded to `IBuyCallback.onBuy` is taken directly from `offer.market`, which is entirely maker-controlled. No check verifies that `offer.market` matches any stored canonical `MarketState` or previously committed market configuration. A malicious maker can therefore craft a sell offer embedding an arbitrary `market.enterGate`, and any taker callback that treats `market.enterGate` as an authoritative gate will execute under a false market context.

### Finding Description

**Code path:**

`Midnight.take()` (line 347) derives the market id from the offer's embedded market struct: [1](#0-0) 

`IdLib.toId` hashes the full `Market` struct including `enterGate`, so a different `enterGate` produces a different `id`: [2](#0-1) 

`touchMarket` creates the market on first use with whatever `enterGate` the maker embedded — no check against any pre-existing canonical market: [3](#0-2) 

For a sell offer (`offer.buy == false`), the buyer callback is the **taker's** callback: [4](#0-3) 

`onBuy` is then called with `offer.market` verbatim — no verification against stored `MarketState`: [5](#0-4) 

The `Market` struct exposes `enterGate` as a plain address field: [6](#0-5) 

**Attacker inputs:**
- `offer.market.enterGate` = address of a malicious contract implementing `IEnterGate` that always returns `true` for `canIncreaseCredit` / `canIncreaseDebt`, or that mimics a trusted gate's interface.
- All other offer fields are valid (correct `loanToken`, valid `collateralParams`, valid `tick`, etc.) so the offer passes all protocol-level checks.

**Exploit flow:**
1. Malicious maker posts a sell offer off-chain (via a ratifier/signature) with `offer.market.enterGate = MaliciousGate`.
2. A taker whose `takerCallback` implements `IBuyCallback` and reads `market.enterGate` to make trust decisions (e.g., "is this market KYC-gated?", "which vault should I route funds to?") calls `take()`.
3. `take()` computes `id = toId(offer.market)` — a fresh market id for the spoofed market — and calls `touchMarket`, creating the market if needed.
4. Protocol gate checks at lines 397–406 use `offer.market.enterGate`, so they call `MaliciousGate.canIncreaseCredit/canIncreaseDebt` — both return `true`, no revert.
5. `onBuy(id, offer.market, ...)` is invoked on the taker's callback. The callback reads `market.enterGate`, sees `MaliciousGate`, and — under the precondition — treats it as authoritative.
6. The callback executes under false assumptions: it may route `buyerAssets` to a wrong receiver, approve tokens to an unintended contract, or skip safety checks it would otherwise apply.

**Why existing checks fail:**
- Lines 397–406 only check that the gate allows credit/debt increases; they do not verify that `offer.market.enterGate` is the gate of any "real" or previously known market. [7](#0-6) 
- `touchMarket` creates a new market for any valid `Market` struct, so the spoofed market is silently instantiated.
- The `id` passed to `onBuy` is consistent with `offer.market` (both derived from the same struct), so a callback cannot detect the spoof by comparing `id` to `market` alone — it would need to call `midnight.toMarket(id)` and compare, which naive implementations do not do.

### Impact Explanation
A taker callback that reads `market.enterGate` from the `onBuy` parameter and uses it to make routing or trust decisions will execute under a maker-injected false context. Concretely: the callback may send `buyerAssets` to an address determined by querying `MaliciousGate`, approve tokens to an attacker-controlled contract, or bypass internal safety checks, resulting in direct loss of the taker's funds for the amount of `buyerAssets` involved in the take.

### Likelihood Explanation
**Preconditions:**
- Maker is unprivileged; any address can post a sell offer with arbitrary `market.enterGate`.
- The taker must use a callback that reads `market.enterGate` from the callback parameter — a natural and documented pattern for periphery integrators who want to respect market access controls.
- The taker (or an automated bot) must call `take()` against the malicious offer without independently verifying `offer.market.enterGate` off-chain.

**Feasibility:** High for automated takers / aggregators. The offer can be made to look identical to a legitimate offer in all fields except `enterGate`, which is not typically surfaced in UI order books. The attack is repeatable for every taker using a vulnerable callback pattern.

### Recommendation
In `take()`, after computing `id = touchMarket(offer.market)`, verify that `offer.market` matches the stored canonical market by decoding it from the SSTORE2 slot and comparing, or by re-deriving the id from the stored market and asserting equality. Alternatively, document explicitly in `IBuyCallback` and `ISellCallback` that the `market` parameter is **maker-supplied and unverified**, and that callbacks must not use `market.enterGate` (or any other `market` field) for trust decisions without independently fetching the market via `midnight.toMarket(id)`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IBuyCallback} from "src/interfaces/ICallbacks.sol";
import {Market, Offer} from "src/interfaces/IMidnight.sol";
import {IEnterGate} from "src/interfaces/IGate.sol";

// Malicious gate: always allows entry, used to spoof market.enterGate
contract MaliciousGate is IEnterGate {
    function canIncreaseCredit(address) external pure returns (bool) { return true; }
    function canIncreaseDebt(address) external pure returns (bool) { return true; }
}

// Vulnerable callback: trusts market.enterGate as authoritative
contract VulnerableCallback is IBuyCallback {
    address public routedTo;
    address public trustedGate;
    constructor(address _trustedGate) { trustedGate = _trustedGate; }

    function onBuy(bytes32, Market memory market, uint256, uint256, uint256, address, bytes memory)
        external returns (bytes32)
    {
        // BUG: trusts market.enterGate from maker-controlled offer
        routedTo = market.enterGate; // e.g., used to route funds
        // assert: if market.enterGate == trustedGate, proceed; else revert
        // MaliciousGate mimics trustedGate interface, so check passes
        return keccak256("IBuyCallback.onBuy");
    }
}

contract GateSpoofTest is Test {
    Midnight midnight;
    MaliciousGate maliciousGate;
    VulnerableCallback callback;

    function setUp() public {
        midnight = new Midnight();
        maliciousGate = new MaliciousGate();
        callback = new VulnerableCallback(address(maliciousGate));
    }

    function test_gateSpoof() public {
        // Maker crafts sell offer with spoofed enterGate
        Market memory spoofedMarket = /* valid market params */ Market({
            loanToken: address(token),
            collateralParams: validCollateralParams,
            maturity: block.timestamp + 30 days,
            rcfThreshold: 0,
            enterGate: address(maliciousGate), // <-- spoofed
            liquidatorGate: address(0)
        });

        Offer memory offer = Offer({
            market: spoofedMarket,
            buy: false, // sell offer: taker is buyer, taker's callback = buyerCallback
            maker: maker,
            // ... valid fields ...
            callback: address(0),
            // ...
        });

        // Taker calls take() with vulnerable callback
        midnight.take(offer, ratifierData, units, taker, address(0), address(callback), "");

        // Assert: callback executed with spoofed gate
        assertEq(callback.routedTo(), address(maliciousGate));
        // Assert: callback did NOT receive the legitimate market's enterGate
        // Assert: funds were routed based on maliciousGate, not the real gate
    }
}
```

**Expected assertions:**
- `callback.routedTo() == address(maliciousGate)` — callback executed under spoofed gate context.
- The `id` passed to `onBuy` corresponds to the spoofed market, not any legitimate market the taker intended to interact with.
- No revert occurs anywhere in `take()` — all protocol checks pass because `MaliciousGate` returns `true`.
