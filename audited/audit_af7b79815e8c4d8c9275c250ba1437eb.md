Audit Report

## Title
Off-by-One in Maturity Guard Permits Debt Increase at Exact Maturity Timestamp - (File: `src/Midnight.sol`)

## Summary
The guard at `src/Midnight.sol:391` uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` when `block.timestamp == market.maturity`, allowing `sellerDebtIncrease > 0` at that exact timestamp. This directly violates the protocol's own invariant that explicitly names timestamp equality as a forbidden path for debt increase. The fee calculation simultaneously treats this timestamp as post-maturity (`timeToMaturity = 0`), creating an internal inconsistency where debt is increased with zero fee accrual and the resulting position is immediately liquidatable in the next block.

## Finding Description

**Root cause — `<=` instead of `<` at line 391:** [1](#0-0) 

When `block.timestamp == market.maturity`, the condition `block.timestamp <= offer.market.maturity` is `true`, so the `require` passes unconditionally regardless of `sellerDebtIncrease`. The correct boundary should be `block.timestamp < offer.market.maturity`.

**Fee calculation already treats exact maturity as post-maturity:** [2](#0-1) 

`zeroFloorSub(maturity, maturity)` returns `0`, so `timeToMaturity = 0` and `buyerPendingFeeIncrease = 0`. The fee path treats `block.timestamp == maturity` identically to any post-maturity block, but the debt guard does not enforce the same boundary.

**Exploit flow:**
1. A sell offer exists (`offer.buy == false`, maker = borrower) with `offer.expiry >= market.maturity`.
2. The seller has zero credit (`sellerPos.credit == 0`), so `sellerCreditDecrease = 0` and `sellerDebtIncrease = units`.
3. Taker waits for `block.timestamp == market.maturity`.
4. Taker calls `take(sellOffer, ..., units, taker, ...)`.
5. Line 391: `maturity <= maturity` → `true` → passes.
6. Line 414: `sellerPos.debt += units` — debt is increased at maturity with zero fee accrual. [3](#0-2) 

**Why existing checks do not stop it:**
- `block.timestamp <= offer.expiry` (line 353) passes if `expiry >= maturity`.
- The maturity guard (line 391) uses `<=` not `<`.
- Post-maturity liquidation (line 622) requires `block.timestamp > market.maturity` (strict `>`), so the newly created debt cannot be liquidated in post-maturity mode in the same block. [4](#0-3) 

This is confirmed by the existing liquidation test, which explicitly asserts that post-maturity liquidation is rejected at exact maturity and only accepted at `maturity + 1`: [5](#0-4) 

**Violated invariants (from `live_context.json`):**
- `"debt must not increase after maturity"` (credit_debt invariants)
- `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"` (maturity invariants) [6](#0-5) [7](#0-6) 

**Existing tests only cover `maturity + 1`, not `maturity` exactly:**

All post-maturity take tests use `market.maturity + 1` as the warp target, and the fuzz variants bound `timestamp` to `[market.maturity + 1, type(uint32).max]`, leaving the exact maturity timestamp untested. [8](#0-7) 

## Impact Explanation
A borrower's debt is increased at `block.timestamp == market.maturity` with zero fee accrual, violating the core credit/debt accounting invariant. The position is immediately subject to post-maturity liquidation in the next block (`block.timestamp == maturity + 1`) with no recourse for the borrower. This constitutes credit/debt accounting corruption and can be used to force a borrower into an immediately-liquidatable state, enabling unauthorized collateral seizure — both of which are explicitly listed in-scope impact classes.

## Likelihood Explanation
Any block whose `block.timestamp` equals a market's maturity is sufficient. On PoS chains, validators can influence `block.timestamp` within a small window; a taker can also simply submit the transaction and have it land in the correct block. The precondition — a sell offer with `expiry >= maturity` — is normal and expected protocol behavior. The attack is repeatable for every market whose maturity falls on a block timestamp, and requires no special privileges beyond being an unprivileged taker.

## Recommendation
Change `<=` to `<` at line 391:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the debt guard with the fee calculation boundary (`timeToMaturity = 0` at equality) and with the post-maturity liquidation boundary (`block.timestamp > market.maturity`), making all three boundaries consistent.

## Proof of Concept
Minimal Foundry test (add to `TakeTest.sol`):

```solidity
function testSellDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;
    // Warp to exact maturity (not maturity + 1)
    vm.warp(market.maturity);
    lenderOffer.expiry = market.maturity;
    lenderOffer.maxUnits = units;
    deal(address(loanToken), lender, units);
    collateralize(market, borrower, units);

    uint256 debtBefore = midnight.debtOf(id, borrower);
    // This should revert with CannotIncreaseDebtPostMaturity but does NOT
    take(units, borrower, lenderOffer);
    // Debt was increased at maturity — invariant violated
    assertGt(midnight.debtOf(id, borrower), debtBefore);

    // Next block: borrower is immediately liquidatable in post-maturity mode
    vm.warp(market.maturity + 1);
    midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");
}
```
