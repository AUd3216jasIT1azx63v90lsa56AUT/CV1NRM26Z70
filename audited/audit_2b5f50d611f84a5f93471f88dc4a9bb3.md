### Title
Any Authorized Address Can Irreversibly Cancel Another Authorized Address's Signed Offer Root — (`src/ratifiers/EcrecoverRatifier.sol`)

---

### Summary

`EcrecoverRatifier.cancelRoot` allows any address authorized by a maker to permanently cancel a Merkle root, including roots that were signed by a *different* authorized address. Because `isRootCanceled` can only be set to `true` (never reversed), a malicious co-authorized address can permanently invalidate a legitimate maker's signed offer tree before any taker executes it. The same class of issue exists in `SetterRatifier.setIsRootRatified`, where an authorized address can deactivate a root ratified by another authorized address.

---

### Finding Description

**Root cause — `EcrecoverRatifier.cancelRoot`**

```solidity
// src/ratifiers/EcrecoverRatifier.sol  lines 27-31
function cancelRoot(address maker, bytes32 root) external {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootCanceled[maker][root] = true;
    emit CancelRoot(msg.sender, maker, root);
}
```

The guard only checks that `msg.sender` is *any* address authorized by `maker`. It does not verify that `msg.sender` was the address that originally signed the root. [1](#0-0) 

**Ratification path — `isRatified`**

During `take`, the ratifier verifies that the EIP-712 digest was signed by either the maker or *any* authorized address:

```solidity
// src/ratifiers/EcrecoverRatifier.sol  lines 44
require(_signer == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer), Unauthorized());
``` [2](#0-1) 

This means Alice (authorized by maker) can legitimately sign a root and distribute offers. But the `cancelRoot` guard uses the same broad `isAuthorized` check, so Bob (also authorized by maker) can cancel Alice's root even though Bob never signed it.

**Cancellation is irreversible.** `isRootCanceled` is a `mapping(address => mapping(bytes32 => bool))` with no setter that can flip it back to `false`. [3](#0-2) 

**Same class in `SetterRatifier.setIsRootRatified`**

```solidity
// src/ratifiers/SetterRatifier.sol  lines 24-28
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    ...
}
```

Any authorized address can call `setIsRootRatified(maker, root, false)` to deactivate a root that a different authorized address activated. [4](#0-3) 

**Authorization model in `Midnight.sol`**

The core contract's `isAuthorized` mapping is a flat `authorizer → authorized → bool` structure with no role scoping:

```solidity
mapping(address authorizer => mapping(address authorized => bool)) public isAuthorized;
``` [5](#0-4) 

Both ratifiers reuse this mapping without any additional constraint on *which* authorized address may cancel a root.

---

### Impact Explanation

- A malicious co-authorized address (Bob) can **permanently** cancel every Merkle root signed by a legitimate authorized address (Alice) for the same maker.
- All takers who received Alice's signed offers can no longer execute them — the offers are dead with no recovery path.
- This is a **permanent, irreversible griefing** of a maker's entire offer book for any root Bob knows about (roots are observable in calldata or off-chain distribution channels).
- In `SetterRatifier`, the attack is reversible by the maker, but the window between deactivation and reactivation is enough to front-run pending `take` transactions.

**Impact: Medium** — no direct theft, but permanent disruption of legitimate market-making activity and loss of expected trade execution.

---

### Likelihood Explanation

- Requires the maker to have authorized at least two addresses (a common pattern for market-making bots, multi-sig setups, or delegated trading).
- The attacker needs to know the root value. Roots appear in `take` calldata (mempool-visible) or in off-chain offer distribution; a co-authorized address is likely already in the same operational context.
- No privileged access, no leaked keys, no economic cost beyond gas.

**Likelihood: Medium**

---

### Recommendation

Bind root cancellation to the address that signed (or set) the root, not to the broad `isAuthorized` set.

**Option A — store the signer at ratification time:**
```solidity
mapping(address maker => mapping(bytes32 root => address signer)) public rootSigner;

function cancelRoot(address maker, bytes32 root) external {
    address _signer = rootSigner[maker][root];
    require(
        msg.sender == maker
            || (msg.sender == _signer && IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender)),
        Unauthorized()
    );
    isRootCanceled[maker][root] = true;
}
```

**Option B — restrict cancellation to `maker == msg.sender` only**, mirroring the principle that only the account owner can invalidate a signature they delegated.

Apply the same fix to `SetterRatifier.setIsRootRatified` for the `false` direction.

---

### Proof of Concept

```
Setup:
  maker  = 0xMAKER
  alice  = 0xALICE  (isAuthorized[maker][alice] = true)
  bob    = 0xBOB    (isAuthorized[maker][bob]   = true)

Step 1: Alice signs an EIP-712 digest over root R containing 100 offers.
        Alice distributes the signed offers to takers off-chain.

Step 2: Bob (monitoring the mempool or the off-chain channel) observes root R.

Step 3: Bob calls:
          EcrecoverRatifier.cancelRoot(maker, R)
        Guard passes because isAuthorized[maker][bob] == true.
        isRootCanceled[maker][R] is set to true permanently.

Step 4: Any taker who calls Midnight.take() with an offer from root R
        hits the check:
          require(!isRootCanceled[offer.maker][root], RootCanceled());
        and reverts.

Result: All 100 of Alice's offers are permanently dead.
        Bob spent only gas. No recovery is possible.
``` [6](#0-5)
