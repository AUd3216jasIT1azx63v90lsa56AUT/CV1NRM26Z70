### Title
Debt Increase Permitted at Exact Maturity Timestamp via Off-by-One in `take()` Guard - (`src/Midnight.sol`)

### Summary
`take()` guards debt increases with `block.timestamp <= offer.market.maturity`, which evaluates to `true` when `block.timestamp == maturity`, allowing `sellerDebtIncrease > 0` to pass at the exact maturity boundary. The post-maturity liquidation path in `liquidate()` uses the strict `block.timestamp > market.maturity`, so the newly created debt is immediately liquidatable in the very next block. The protocol's own invariant document explicitly names "timestamp equality" as a forbidden path for debt increase.

### Finding Description
**Exact code path:**

`src/Midnight.sol` line 391:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
When `block.timestamp == offer.market.maturity`, the left operand `block.timestamp <= offer.market.maturity` evaluates to `true`, short-circuiting the entire `require`. Any value of `sellerDebtIncrease` passes.

`src/Midnight.sol` line 359:
```solidity
uint256 timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp);
```
At `block.timestamp == maturity`, `timeToMaturity = 0`. This is used for `buyerPendingFeeIncrease` (which becomes 0) and for the settlement fee (which uses the 0d breakpoint), so neither of these sub-computations blocks the call.

`src/Midnight.sol` line 476:
```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```
`isHealthy` checks only collateral vs. debt, not maturity. A well-collateralized borrower passes this check, so the take completes and `sellerPos.debt += sellerDebtIncrease` is committed at line 414.

**Why the post-maturity liquidation check does not save it:**

`src/Midnight.sol` line 622:
```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
```
Post-maturity liquidation requires `block.timestamp > market.maturity` (strictly greater). At `block.timestamp == maturity` the condition is `false`, so the newly created debt cannot be post-maturity liquidated in the same block. In the very next block (`block.timestamp == maturity + 1`), the strict inequality flips and the position is immediately post-maturity liquidatable regardless of health.

**Attacker-controlled inputs:**
- A sell offer (maker is lender, `offer.buy == false`) with `offer.expiry >= maturity` and `offer.start <= maturity`.
- Taker is the borrower with sufficient collateral to pass `isHealthy`.
- `units > sellerPos.credit` so that `sellerDebtIncrease = units - sellerCreditDecrease > 0`.
- Transaction submitted in the block where `block.timestamp == offer.market.maturity`.

**Existing protections reviewed and found insufficient:**
- `block.timestamp <= offer.expiry` — passes since `expiry >= maturity`.
- `CannotIncreaseDebtPostMaturity` guard — uses `<=`, not `<`; passes at equality.
- `SellerIsLiquidatable` guard — health-only, not maturity-aware.
- `SetterRatifier.isRatified` — validates the Merkle proof against the offer struct; it has no maturity-time restriction and passes if the root is ratified.

### Impact Explanation
A borrower takes a sell offer at `block.timestamp == maturity`, increasing their debt. The debt is committed on-chain. In the next block the position is post-maturity liquidatable (even if fully collateralized), meaning the borrower has created debt that is immediately subject to forced liquidation with a growing LIF ramp. This violates the core invariant "debt must not increase after maturity" and the explicit protocol rule "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."

### Likelihood Explanation
Any borrower who monitors the chain can submit a debt-increasing take in the exact block where `block.timestamp == maturity`. Block timestamps are miner/validator-controlled within a small window, making this reliably reachable. The precondition (sufficient collateral, a live sell offer with `expiry >= maturity`) is normal operating state. The attack is repeatable on every market at its maturity block.

### Recommendation
Change the guard on line 391 from `<=` to `<`:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the debt-increase guard with the post-maturity liquidation condition (`block.timestamp > market.maturity`) so that `block.timestamp == maturity` is treated as post-maturity for debt creation purposes.

### Proof of Concept

function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;
    // Warp to exactly maturity (not maturity + 1)
    vm.warp(market.maturity);

    // Lender posts a sell offer expiring at maturity
    lenderOffer.expiry = market.maturity;
    lenderOffer.maxUnits = units;
    deal(address(loanToken), lender, units);
    collateralize(market, borrower, units); // borrower has zero existing credit

    uint256 debtBefore = midnight.debtOf(id, borrower);

    // This should revert with CannotIncreaseDebtPostMaturity but does NOT

    uint256 debtAfter = midnight.debtOf(id, borrower);

    // Assert: debt increased at maturity — invariant violated
    assertGt(debtAfter, debtBefore, "debt must not increase at maturity");

    // Assert: position is immediately post-maturity liquidatable in next block
    vm.warp(market.maturity + 1);

    // liquidate should succeed even though position is healthy
    midnight.liquidate(market, 0, 0, debtAfter, borrower, true, address(this), address(0), "");
    assertEq(midnight.debtOf(id, borrower), 0, "debt liquidated post-maturity");
}
