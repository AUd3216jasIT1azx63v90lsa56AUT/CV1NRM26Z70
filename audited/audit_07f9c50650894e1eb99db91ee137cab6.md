### Title
Borrower Can Frontrun Liquidation to Trigger `RecoveryCloseFactorConditionsViolated` Revert — (`src/Midnight.sol`)

### Summary

In `Midnight.sol`, the `liquidate()` function reverts with `RecoveryCloseFactorConditionsViolated` when `repaidUnits > maxRepaid` and the residual collateral exceeds `rcfThreshold`. A borrower can exploit this by frontrunning a liquidation transaction with a small `supplyCollateral()` call, which increases `maxDebt`, shrinks the unhealthy gap, and lowers `maxRepaid` below the liquidator's pre-computed `repaidUnits`, causing the liquidation to revert without the position becoming healthy.

### Finding Description

**Root cause — the RCF check reverts instead of capping:**

In `liquidate()`, after computing `repaidUnits` from the caller's input, the normal-mode RCF check is: [1](#0-0) 

`maxRepaid` is derived from the current on-chain state at execution time: [2](#0-1) 

The formula is `(debt − maxDebt) × WAD² / (WAD² − lif × lltv)`. Any increase in `maxDebt` (from additional collateral) shrinks `debt − maxDebt` and therefore shrinks `maxRepaid`.

**Attack path:**

1. Alice has an unhealthy position: `debt > maxDebt` with `lltv < WAD`.
2. Bob (liquidator) reads the chain state, computes `maxRepaid` off-chain, and submits `liquidate(..., repaidUnits = maxRepaid, ...)`.
3. Alice observes Bob's pending transaction and frontruns it with `supplyCollateral()` depositing a small amount to the same or any other active collateral index. [3](#0-2) 

4. `supplyCollateral` has no health check and succeeds unconditionally (Alice's position is still unhealthy after the small deposit).
5. Bob's transaction executes. The new `maxDebt` is higher, so `maxRepaid` is now smaller than Bob's `repaidUnits`. The residual collateral value `collateral × price / lif − maxRepaid` is larger (more collateral was added), so it exceeds `rcfThreshold`.
6. Both conditions of the `require` fail → revert with `RecoveryCloseFactorConditionsViolated`.
7. Alice's position remains unhealthy. Alice can repeat this indefinitely at low cost.

**Why existing checks do not prevent this:**

- `supplyCollateral` has no restriction on unhealthy borrowers adding collateral.
- The `liquidate` function reads `_position.debt` and recomputes `maxDebt` fresh at execution time, so any state change between Bob's off-chain computation and on-chain execution invalidates his `repaidUnits`.
- The `NotLiquidatable` guard only checks `debt > maxDebt`, which remains true after a tiny collateral top-up, so Alice's position is still liquidatable in principle — but the specific `repaidUnits` value Bob chose now exceeds the new `maxRepaid`. [4](#0-3) 

### Impact Explanation

A borrower facing liquidation can indefinitely delay it by spending small amounts of collateral or debt repayment to shift `maxRepaid` below any liquidator's pre-computed `repaidUnits`. The position remains unhealthy and accrues bad debt risk for lenders. This is a temporary but repeatable DoS on the liquidation mechanism for all markets with `lltv < WAD`.

### Likelihood Explanation

Any borrower with an unhealthy position has a direct financial incentive to execute this attack. The cost is minimal (a small collateral deposit or partial repayment that does not restore health). The attack requires only a standard frontrun (mempool observation + higher gas), which is realistic on any EVM chain with a public mempool. It applies to all markets where `lltv < WAD` (all allowed LLTV tiers except `LLTV_8 = 1e18`). [5](#0-4) 

### Recommendation

Instead of reverting when `repaidUnits > maxRepaid`, cap `repaidUnits` at `maxRepaid` and recompute `seizedAssets` accordingly. This mirrors the recommended fix from the Taurus audit: the liquidation proceeds with the maximum currently-allowed amount rather than reverting entirely.

```solidity
// In the !postMaturityMode branch, after computing maxRepaid:
if (repaidUnits > maxRepaid) {
    bool dustExempt = _position.collateral[collateralIndex]
        .mulDivDown(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
        .mulDivDown(WAD, lif)
        .zeroFloorSub(maxRepaid) < market.rcfThreshold;
    require(dustExempt, RecoveryCloseFactorConditionsViolated());
    repaidUnits = maxRepaid;
    seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
}
```

### Proof of Concept

**Setup:**
- Market with `lltv = 0.86e18`, `maxLif = maxLif(0.86e18, 0.5e18)`, `rcfThreshold = 1e18`.
- Alice supplies 100 units of collateral (oracle price = `ORACLE_PRICE_SCALE`, so collateral value = 100).
- Alice borrows 90 units of debt (healthy at `lltv = 0.86`, `maxDebt = 86`).
- Oracle price drops: collateral value = 85, so `maxDebt = 73.1`, `debt = 90 > maxDebt`. Alice is unhealthy.

**Liquidation attempt:**
- Bob computes `maxRepaid = (90 − 73.1) × WAD² / (WAD² − lif × lltv)` ≈ some value X.
- Bob submits `liquidate(..., repaidUnits = X, ...)`.

**Frontrun:**
- Alice calls `supplyCollateral(..., assets = 1 wei, ...)` — cost: 1 wei of collateral.
- New `maxDebt` increases slightly → new `maxRepaid` = X − ε < X.
- Bob's `repaidUnits = X > X − ε = new maxRepaid`.
- Residual collateral is still large (100 + 1 wei), so `collateral × price / lif − maxRepaid >> rcfThreshold`.
- Bob's `liquidate` reverts with `RecoveryCloseFactorConditionsViolated`.

Alice repeats each time Bob resubmits, permanently delaying liquidation at negligible cost. [6](#0-5) [7](#0-6)
