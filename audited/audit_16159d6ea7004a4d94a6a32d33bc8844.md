Audit Report

## Title
Maker-controlled sell callback can transiently manipulate oracle price, causing `isHealthy` to return false and reverting valid taker transactions - (File: src/Midnight.sol)

## Summary
When a taker fills a sell offer (`offer.buy == false`), the maker's `offer.callback` is assigned as `sellerCallback` and invoked before the post-callback health check at line 476. A malicious maker can implement `onSell` to transiently depress a spot/reserve-based oracle, causing `isHealthy(offer.market, id, seller)` to return `false` for a position that was healthy at pre-callback prices. The transaction reverts with `SellerIsLiquidatable`, all state changes are undone, and the attack is repeatable at near-zero cost.

## Finding Description

**Callback assignment (line 421):**
```solidity
address sellerCallback = offer.buy ? takerCallback : offer.callback;
```
When `offer.buy == false`, `sellerCallback = offer.callback` — fully controlled by the maker at offer-creation time. No validation of `offer.callback` exists anywhere in `take()`.

**Seller identity (line 375):**
```solidity
(address buyer, address seller) = offer.buy ? (offer.maker, taker) : (taker, offer.maker);
```
When `offer.buy == false`, `seller = offer.maker`. The health check at line 476 is therefore on the maker's own position.

**State mutation before callback (lines 408–414):**
The maker's debt is increased (`sellerPos.debt += sellerDebtIncrease`) before the callback fires, so `isHealthy` will call `IOracle(collateralParam.oracle).price()` (line 953) against a position that now carries more debt.

**Callback invocation (lines 458–474):**
`ISellCallback(sellerCallback).onSell(...)` is called with no restrictions on external state mutations (pool reserves, oracle state, etc.).

**Lock reset and health check (lines 475–476):**
```solidity
if (!wasLocked) UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, false);
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```
In the normal case (`wasLocked == false`), the lock is reset to `false` before the `require`. `liquidationLocked(id, seller)` therefore evaluates to `false`, and `isHealthy` must return `true`. If the callback has shifted the oracle price, `isHealthy` reads the manipulated value and returns `false`.

**`isHealthy` oracle read (line 953):**
```solidity
uint256 price = IOracle(collateralParam.oracle).price();
```
This is a live call to the oracle with no snapshot or TWAP enforcement at the protocol level.

**Exploit flow:**
1. Maker deploys `MaliciousCallback` implementing `ISellCallback.onSell` that flash-borrows tokens and swaps them in the oracle's underlying pool, dropping the reported collateral price.
2. Maker creates a sell offer with `offer.callback = address(MaliciousCallback)` and sufficient collateral that the position is healthy at the pre-callback price.
3. Taker calls `take()`. Maker's debt increases (line 414), lock is set to `true` (line 444).
4. `MaliciousCallback.onSell` executes the pool swap; oracle price drops below the health threshold.
5. Lock is reset to `false` (line 475).
6. `isHealthy(offer.market, id, seller)` reads the manipulated price → returns `false`.
7. `require(false || false, SellerIsLiquidatable())` → transaction reverts; flash loan is repaid atomically.

**Why existing protections are insufficient:**
The `LIQUIDATION_LOCK_SLOT` mechanism (lines 444, 475) is designed solely to prevent reentrant liquidation calls during callbacks — confirmed by the comment at line 335: *"The seller cannot be liquidated during the callbacks of a take."* It does not protect the post-callback `isHealthy` evaluation from reading oracle state that was mutated during the callback. The Certora `Healthiness.spec` (line 14–16) explicitly assumes `price()` does not change during rules (`function _.price() external => summaryPrice(calledContract)`), meaning this attack vector is outside the formal verification's scope.

## Impact Explanation
Any taker attempting to fill a malicious maker's sell offer will have their transaction revert with `SellerIsLiquidatable`. No funds are lost (revert undoes all state changes), but the offer is never consumed, making the attack repeatable indefinitely. The maker can permanently block market activity on any offer they choose to poison, grief specific takers, or deny liquidity to an entire market. This constitutes service unavailability and severe degradation of market function under realistic attacker input.

## Likelihood Explanation
**Preconditions:**
- Market uses an oracle whose `price()` reads mutable on-chain state (spot price, reserve-based oracle, short-window TWAP). This is the common case for DeFi markets.
- Maker's position must be healthy before the callback (so the offer is valid) but the oracle must be manipulable enough to push it below the health threshold — achievable with a flash loan.

**Feasibility:** High. The maker controls `offer.callback` unconditionally; no privileged access is required. The attack is atomic (single transaction), repeatable, and costs only gas plus any swap slippage (recoverable via flash loan). Any market with a spot oracle is vulnerable.

## Recommendation
1. **Snapshot oracle prices before callbacks:** Read and cache all oracle prices needed for the health check before invoking any callback. Use the cached values in the post-callback `isHealthy` evaluation rather than re-querying live oracles.
2. **Alternatively, move the health check before the sell callback:** Evaluate `isHealthy` immediately after state mutation (lines 408–414) and before `sellerCallback` is invoked. This eliminates the window for oracle manipulation entirely.
3. **Validate `offer.callback`:** Consider restricting `offer.callback` to a whitelist or requiring it to be the zero address for sell offers where the maker is the seller, if the callback is not strictly necessary for the maker's use case.

## Proof of Concept
```solidity
// MaliciousCallback.sol
contract MaliciousCallback is ISellCallback {
    IUniswapV2Pair pair;
    IERC20 collateralToken;
    IERC20 loanToken;

    function onSell(
        bytes32, Market memory, uint256, uint256, uint256, address, address, bytes memory
    ) external returns (bytes32) {
        // Flash-swap: dump collateral into the pool, depressing collateral price
        uint256 amount = collateralToken.balanceOf(address(pair)) * 99 / 100;
        collateralToken.transfer(address(pair), amount);
        pair.swap(0, loanToken.balanceOf(address(pair)) * 99 / 100, address(this), "");
        return CALLBACK_SUCCESS;
    }
}

// 3. Taker calls take() — MaliciousCallback.onSell fires, dumps collateral into pool.
// 4. isHealthy() reads depressed spot price → returns false.
// 5. Transaction reverts with SellerIsLiquidatable.
// 6. Repeat for every taker attempt — offer is never consumed.
```
