Audit Report

## Title
Zero-increment `consumed` bypass allows unlimited takes on fully-consumed buy offers — (File: `src/Midnight.sol`)

## Summary
When `offer.buy == true` and `offer.maxAssets > 0`, the consumed accounting increments by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For any tick below `MAX_TICK`, `buyerPrice < WAD`, so `units = 1` yields `buyerAssets = 0` via integer truncation. The guard `require(newConsumed <= offer.maxAssets)` then passes even when `consumed == maxAssets`, because `maxAssets + 0 <= maxAssets`. All downstream position mutations still execute with the non-zero `units` value, producing real credit/debt changes with zero token transfer.

## Finding Description
**Root cause — `src/Midnight.sol` lines 363–373:**

For a buy offer, `buyerPrice = offerPrice = tickToPrice(offer.tick)`. The Certora spec (`TickToPrice.spec` line 39) and the test suite (`TickLibTest.sol` line 19) both confirm `tickToPrice(MAX_TICK) == 1e18 == WAD`, and the monotonicity proof confirms `tickToPrice(tick) < WAD` for all `tick < MAX_TICK`. Therefore, for any normally operating tick, `buyerPrice < WAD`.

```solidity
// src/Midnight.sol:363
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// ...
// src/Midnight.sol:367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

With `units = 1` and `buyerPrice < WAD`: `mulDivDown(1, buyerPrice, WAD) = buyerPrice / WAD = 0`. The storage update becomes `consumed += 0`, so `newConsumed == consumed`. The require passes unconditionally even when `consumed == maxAssets`.

Downstream at lines 382–417, all mutations use `units` (non-zero):
- `buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)` — non-zero if buyer has no debt
- `sellerDebtIncrease = units - sellerCreditDecrease` — non-zero if seller has no credit
- `_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease`
- `claimableSettlementFee[...] += buyerAssets - sellerAssets = 0`

The token transfer at line 455 sends `buyerAssets - sellerAssets = 0` tokens, so no assets move. There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in the function.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` only catches cases where `buyerAssets > 0` pushes `newConsumed` over the cap. A zero increment never triggers it.
- The `reduceOnly` check (lines 392–395) only applies when `offer.reduceOnly` is set.
- No minimum-units or minimum-assets guard exists.

## Impact Explanation
A fully-consumed buy offer (`consumed == maxAssets`) can be taken an unlimited number of times. Each such take increases the maker's credit and the taker's debt by `units` with zero token payment, and inflates `totalUnits` without a corresponding increase in `claimableSettlementFee` or token reserves. This creates a permanent accounting mismatch: units outstanding are not backed by deposited assets. At settlement, the maker can claim tokens that do not exist in the contract, leading to insolvency of the market. This violates the core protocol invariant that offers cannot be replayed or reused after being fully consumed.

## Likelihood Explanation
Preconditions are minimal:
- `offer.buy == true`, `offer.maxAssets > 0` — standard buy offer configuration.
- `buyerPrice < WAD` — holds for every tick below `MAX_TICK`; this is the normal operating range.
- `units = 1` — trivially satisfies `1 * buyerPrice < WAD` whenever `buyerPrice < WAD`.

Any unprivileged taker can trigger this after any partial or full fill of a qualifying buy offer. No special permissions, oracle manipulation, or victim mistakes are required. The attack is repeatable without limit and requires only the ability to call `take`.

## Recommendation
Add a guard that rejects takes where `units > 0` but `buyerAssets == 0` (for buy offers) or `sellerAssets == 0` (for sell offers with `maxAssets > 0`):

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0 || units == 0, ZeroAssetsTake());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` value such that `units.mulDivDown(buyerPrice, WAD) >= 1`, or switch the consumed accounting for buy offers to track `units` instead of `buyerAssets` (consistent with the `maxUnits` branch).

## Proof of Concept
1. Deploy the protocol and create a market with any tick `t < MAX_TICK` (e.g., `MAX_TICK - 2`, giving `buyerPrice = 1e18 - 1e12`).
2. Maker creates a buy offer with `maxAssets = N` (e.g., `N = 1e18`).
3. Taker fills the offer normally until `consumed[maker][group] == N`.
4. Taker calls `take` again with `units = 1`. Verify: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`, `newConsumed = N + 0 = N`, require passes.
5. Observe: `buyerPos.credit += 1`, `sellerPos.debt += 1`, `totalUnits += 1`, zero tokens transferred.
6. Repeat step 4 arbitrarily many times, accumulating unbacked credit/debt and inflating `totalUnits`.

A Foundry fuzz test asserting `consumed[maker][group] <= maxAssets` after any sequence of `take` calls will falsify the invariant with `units = 1` on a fully-consumed offer.
