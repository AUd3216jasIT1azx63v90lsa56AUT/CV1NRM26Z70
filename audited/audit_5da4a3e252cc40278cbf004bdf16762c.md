Audit Report

## Title
`sellerAssetsToUnits` Division-by-Zero DoS When `offerPrice == settlementFee` for Buy Offers - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary

`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and passes it as the divisor to `mulDivUp`. When `offerPrice == settlementFee`, `sellerPrice = 0`, and `mulDivUp`'s `d - 1` expression underflows under Solidity 0.8 checked arithmetic, causing a revert. `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside any try/catch, so the entire bundler call reverts. `Midnight.take` itself handles `sellerPrice = 0` gracefully via `mulDivDown`, so the offer remains directly takeable but not through the bundler.

## Finding Description

**Root cause — `sellerAssetsToUnits`:**

In `src/periphery/TakeAmountsLib.sol` lines 44–46:
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...
```
When `offerPrice == settlementFee`, `sellerPrice = 0`, and `mulDivUp(targetSellerAssets, WAD, 0)` is called.

**`mulDivUp` underflow:**

In `src/libraries/UtilsLib.sol` lines 34–36:
```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```
`d - 1` with `d = 0` underflows in Solidity 0.8 checked arithmetic → arithmetic revert. This is confirmed by the existing test `testMulDivUpDivisionByZero` in `test/UtilsLibTest.sol` (line 80–83), which expects `stdError.arithmeticError` (not `divisionError`) precisely because the underflow in `d - 1` fires before the division.

**Reachability of `sellerPrice = 0`:**

`CBP = 1e12` (`src/libraries/ConstantsLib.sol` line 10) and `PRICE_ROUNDING_STEP = 1e12` (`src/libraries/TickLib.sol` line 8) share the same granularity. Settlement fees are stored and validated as multiples of `CBP`, so the minimum non-zero fee is `1e12`. `MAX_SETTLEMENT_FEE_1_DAY = 0.000014e18 = 14000e12` (`src/libraries/ConstantsLib.sol` line 12), confirming `1e12` is a valid fee. Tick prices are rounded to multiples of `PRICE_ROUNDING_STEP = 1e12` (`src/libraries/TickLib.sol` line 50), and `tickToPrice(0) == 0` (proven in `certora/specs/TickToPrice.spec` line 34–36), so the minimum non-zero price is `1e12`, achievable at the lowest non-zero tick. A buy offer at that tick with settlement fee set to `1e12` gives `sellerPrice = 0`. Both values are multiples of `1e12`, so exact equality is achievable.

**Why `Midnight.take` does NOT revert:**

In `src/Midnight.sol` lines 361–364:
```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice = sellerPrice + _settlementFee;
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...
```
`mulDivDown(units, 0, WAD) = (units * 0) / WAD = 0` — no underflow, no division by zero. The offer is takeable directly.

**Where the bundler reverts (not caught):**

In `src/periphery/MidnightBundles.sol` lines 285–291:
```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(   // ← reverts here, NOT in try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
```
The NatDoc at line 175 explicitly states: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." The `try/catch` at lines 292–300 only wraps `IMidnight(MIDNIGHT).take(...)`, not the `min(...)` argument evaluation.

**`ConsumableUnitsLib` does not independently revert for buy offers:**

In `src/periphery/ConsumableUnitsLib.sol` lines 16–19: for buy offers with `maxUnits > 0`, `consumableUnits` returns `offer.maxUnits.zeroFloorSub(consumed)` without calling `sellerAssetsToUnits`. For buy offers with `maxAssets > 0`, it calls `buyerAssetsToUnits` (not `sellerAssetsToUnits`), which uses `buyerPrice = sellerPrice + settlementFee = settlementFee > 0` as divisor — no revert. The revert originates solely from the direct `sellerAssetsToUnits` call.

**Asymmetry with `buyerAssetsToUnits`:**

In `src/periphery/TakeAmountsLib.sol` lines 26–29: `buyerAssetsToUnits` uses `buyerPrice = sellerPrice + settlementFee` as divisor. When `sellerPrice = 0`, `buyerPrice = settlementFee > 0`, so no division by zero. Only `sellerAssetsToUnits` is unguarded for the `offerPrice == settlementFee` edge case.

## Impact Explanation

`MidnightBundles.supplyCollateralAndSellWithAssetsTarget` permanently reverts for any buy offer where `offerPrice == settlementFee`, regardless of the number of units requested. Borrowers relying on the bundler for atomic collateral supply + borrow cannot execute their borrow flow for such offers. The offer remains takeable via `Midnight.take` directly, so the DoS is scoped to the bundler's assets-target sell path. This constitutes service unavailability for borrowers who depend on the bundler's atomic flow.

## Likelihood Explanation

**Preconditions:**
1. The admin (feeSetter) sets the settlement fee to `1e12` (1 CBP) — a valid, minimum non-zero fee well within `MAX_SETTLEMENT_FEE_1_DAY = 14000e12`.
2. A lender (unprivileged) creates a buy offer at the lowest non-zero tick (`offerPrice = 1e12`).
3. A borrower calls `supplyCollateralAndSellWithAssetsTarget` with `targetSellerAssets > 0` against that offer.

The lender does not need privileged access. The admin setting a fee of `1e12` is a legitimate protocol action. The lender can observe the current fee and deliberately place an offer at the matching tick. The condition is an exact equality between two `1e12`-granularity values, achievable at the minimum non-zero tick with the minimum non-zero fee. The condition is repeatable and persistent until the fee or offer changes.

## Recommendation

Guard `sellerAssetsToUnits` against `sellerPrice == 0` for buy offers. The simplest fix mirrors the NatDoc's stated intent — revert explicitly when `sellerPrice == 0` (i.e., `offerPrice <= settlementFee`), consistent with how `buyerAssetsToUnits` already reverts via the underflow on `offerPrice - settlementFee` when `offerPrice < settlementFee`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
require(!offer.buy || sellerPrice > 0, ZeroSellerPrice());
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...
```

Alternatively, add a zero-check before calling `mulDivUp` and return a sentinel value (e.g., `type(uint256).max`) when `sellerPrice == 0`, so the bundler skips the offer rather than reverting. The NatDoc should also be updated to cover the equality case (`offerPrice <= settlementFee`, not just `offerPrice < settlementFee`).

## Proof of Concept

**Minimal Forge test:**

```solidity
function testSellerAssetsToUnitsRevertWhenSellerPriceZero() public {
    // Set settlement fee to 1e12 (minimum non-zero, 1 CBP)
    midnight.setMarketSettlementFee(id, 1, 1e12);

    // Find the lowest tick whose price == 1e12
    // tickToPrice(0) == 0, so iterate upward until price == 1e12
    uint256 targetTick;
    for (uint256 t = 1; t <= MAX_TICK; t++) {
        if (TickLib.tickToPrice(t) == 1e12) { targetTick = t; break; }
    }

    // Create a buy offer at that tick
    Offer memory offer = lenderOffer; // buy offer
    offer.tick = targetTick;
    offer.maxUnits = type(uint256).max;

    // sellerAssetsToUnits should revert with arithmetic error
    vm.expectRevert(stdError.arithmeticError);
    TakeAmountsLib.sellerAssetsToUnits(address(midnight), id, offer, 1e18);
}
```
