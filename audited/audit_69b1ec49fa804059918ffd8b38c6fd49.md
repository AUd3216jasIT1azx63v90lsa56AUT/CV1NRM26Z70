Audit Report

## Title
`maxAssets` Cap Bypass via `mulDivDown` Zero-Rounding on Buy Offers - (File: `src/Midnight.sol`)

## Summary
When `offer.buy == true` and `offer.maxAssets > 0`, the consumed tracker is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `mulDivDown` truncates toward zero, any call where `units * buyerPrice < WAD` produces `buyerAssets = 0`, leaving `consumed[offer.maker][offer.group]` unchanged. A taker can therefore call `take` repeatedly with small-unit fills — each passing the `ConsumedAssets` guard — while the maker's credit and the taker's debt grow without bound, completely defeating the `maxAssets` cap.

## Finding Description
**Root cause** — `src/Midnight.sol` lines 363–369:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → can be 0
    : units.mulDivUp(buyerPrice, WAD);

uint256 newConsumed;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy
        ? buyerAssets   // ← 0 when units * buyerPrice < WAD
        : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

`tickToPrice` returns values in `[0, WAD]`, confirmed by `test/TickLibTest.sol` line 17: `tickToPrice(2) == 1e12` and `tickToPrice(MAX_TICK) == 1e18`. For any tick below `MAX_TICK`, `buyerPrice < WAD`, so choosing `units < WAD / buyerPrice` (e.g., `units = 1` when `buyerPrice = 1e12`) makes `units * buyerPrice < WAD`, causing `mulDivDown` to return 0.

**Exploit flow**:
1. Maker posts a buy offer with `offer.buy = true`, `offer.maxAssets = M`, and any `tick < MAX_TICK`.
2. Taker calls `take` with `units = 1` (or any value where `units * buyerPrice < WAD`).
3. `buyerAssets = 0` → `consumed` is unchanged → `require(newConsumed <= maxAssets)` always passes, even when `consumed` was already at `maxAssets`.
4. Position update still executes: maker's `credit` increases by `units`, taker's `debt` increases by `units`, `totalUnits` grows.
5. Repeated indefinitely: total units transferred = N, total `consumed` = 0 (or whatever it was before), maker's credit exposure is unbounded.

**Existing checks fail**: The `ConsumedAssets` guard at line 369 only fires when `newConsumed > maxAssets`. Since `newConsumed` never increases (because `buyerAssets = 0`), the guard is permanently inert for these fills. There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in the path.

**Protocol acknowledgment**: The NatSpec at `src/Midnight.sol` line 94 explicitly states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test suite contains `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol` lines 858–889) which directly confirms: after taking 1 unit against a fully-consumed buy offer at `tick = MAX_TICK - 16`, `buyerAssets == 0`, `consumed` is unchanged, yet `creditOf(lender)` increases, `debtOf(borrower)` increases, and `totalUnits` increases.

## Impact Explanation
The `maxAssets` cap on buy offers is completely ineffective when `buyerPrice < WAD` and the taker uses sub-WAD unit fills. The maker accumulates unbounded credit and the taker accumulates unbounded debt at zero token cost per fill, since `buyerAssets = 0` means no loan tokens are transferred per fill. This violates the core protocol invariant that "offers cannot be overfilled" and constitutes an unauthorized state change causing unbounded position growth beyond the maker's intended exposure limit.

## Likelihood Explanation
- **Preconditions**: Any buy offer with `maxAssets > 0` and `tick < MAX_TICK` (i.e., `buyerPrice < WAD`). This covers virtually all real buy offers, since `MAX_TICK` (price = 1.0) is the par-value boundary.
- **Attacker**: Any unprivileged taker; no special role or capital required (token cost per fill is 0).
- **Repeatability**: Unlimited; each fill is independent and the guard never trips.
- **Feasibility**: Trivial — the taker passes `units = 1` on each call.

## Recommendation
Add a guard that prevents non-zero unit fills from producing zero `buyerAssets` when `maxAssets` mode is active for a buy offer:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// Add: require(units == 0 || buyerAssets > 0 || !offer.buy || offer.maxAssets == 0, ZeroBuyerAssets());
```

Alternatively, use `mulDivUp` for the consumed tracking on buy offers (tracking the ceiling rather than the floor), ensuring any non-zero fill always increments `consumed` by at least 1. A third option is to enforce a minimum `units` such that `units * buyerPrice >= WAD` when `offer.maxAssets > 0`.
