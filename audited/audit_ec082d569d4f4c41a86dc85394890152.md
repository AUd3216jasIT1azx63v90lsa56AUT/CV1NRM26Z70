Audit Report

## Title
`buyerAssetsToUnits` unconditionally reverts with `PriceGreaterThanOne` for sell offers at `MAX_TICK` with non-zero `settlementFee`, causing `buyWithAssetsTargetAndWithdrawCollateral` to DoS - (`File: src/periphery/TakeAmountsLib.sol`)

## Summary

`TakeAmountsLib.buyerAssetsToUnits` applies `require(buyerPrice <= WAD)` unconditionally for both buy and sell offers. For a sell offer at `tick = MAX_TICK`, `tickToPrice(MAX_TICK) = WAD` exactly, so `buyerPrice = WAD + settlementFee > WAD` whenever `settlementFee > 0`, triggering the revert. `Midnight.take` has no equivalent guard and executes successfully for the same inputs. Because `MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` calls `buyerAssetsToUnits` outside any `try/catch`, the entire bundler transaction reverts for any sell offer at `MAX_TICK` in a market with a non-zero 0-day settlement fee at or after maturity.

## Finding Description

**`tickToPrice(MAX_TICK) = WAD` — confirmed from `TickLib.tickToPrice`:**

At `tick = MAX_TICK = 5820`, the exponent `LN_ONE_PLUS_DELTA * (int256(2910) - int256(5820))` is a large negative integer. `wExp(large_negative)` returns `1e36 / wExp(large_positive)`, which integer-divides to `0`. The denominator becomes `1e18 + 0 = 1e18`, and `divHalfDownUnchecked(1e36, 1e18) = 1e18 = WAD`. After `PRICE_ROUNDING_STEP` rounding, the result is still `WAD`. [1](#0-0) 

**Revert path in `buyerAssetsToUnits` for a sell offer:**

```
offerPrice = tickToPrice(MAX_TICK) = WAD
sellerPrice = offerPrice              // offer.buy = false
buyerPrice  = sellerPrice + settlementFee = WAD + settlementFee > WAD
require(buyerPrice <= WAD, ...)       // ← REVERTS
``` [2](#0-1) 

**`Midnight.take` has no equivalent guard:**

`take` computes the identical `buyerPrice = WAD + settlementFee` and proceeds directly to `units.mulDivUp(buyerPrice, WAD)` without any `require(buyerPrice <= WAD)` check. [3](#0-2) 

**Bundler propagation — `buyWithAssetsTargetAndWithdrawCollateral`:**

`TakeAmountsLib.buyerAssetsToUnits` is called at line 209 outside any `try/catch`. The NatSpec at line 175 explicitly documents: *"Reverts if TakeAmountsLib or ConsumableUnitsLib reverts."* The revert propagates to the top-level call, making the entire function unusable for this offer configuration. [4](#0-3) [5](#0-4) 

**Test suite independently confirms the gap:**

`_maxTick(settlementFee)` computes the highest tick where `tickToPrice(tick) + settlementFee <= WAD`. All four fuzz tests bound `tick` strictly below `MAX_TICK` when `settlementFee > 0`. The `testSnappedBuyerAssets*` tests handle the `MAX_TICK` case by computing `targetUnits` directly and calling `take` directly — deliberately bypassing `buyerAssetsToUnits`. [6](#0-5) [7](#0-6) [8](#0-7) 

## Impact Explanation

`MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` is completely unusable for any sell offer at `MAX_TICK` in a market where `settlementFeeCbp0 > 0` once `block.timestamp >= offer.market.maturity`. The entire transaction reverts; because `pullToken` (line 197) is part of the same transaction, the EVM revert returns the user's tokens — no funds are permanently frozen. The concrete impact is a **DoS of the bundler's assets-target buy path** for a valid, reachable market state. Users must fall back to calling `Midnight.take` directly with a manually computed `units` value. [9](#0-8) 

## Likelihood Explanation

All three preconditions are routine and can coexist in production:

1. **`settlementFeeCbp0 > 0`**: `MAX_SETTLEMENT_FEE_0_DAYS = 0.000014e18 = 14 * CBP`; any value from `1 * CBP` to `14 * CBP` triggers the bug. The fee setter can configure this at any time. [10](#0-9) 

2. **`tick = MAX_TICK = 5820`**: A valid tick (5820 is a multiple of `DEFAULT_TICK_SPACING = 4`). Makers offering at the highest price naturally place offers here. [11](#0-10) 

3. **`block.timestamp >= offer.market.maturity`**: Normal post-maturity state, reached by every market.

The bug is repeatable for every call to `buyWithAssetsTargetAndWithdrawCollateral` that includes a sell offer at `MAX_TICK` under these conditions. No attacker action is required — the revert is triggered by the user's own valid transaction.

## Recommendation

In `buyerAssetsToUnits`, the `require(buyerPrice <= WAD)` guard is only semantically necessary for buy offers (where the inversion `mulDivUp(WAD, buyerPrice)` is undefined for `buyerPrice > WAD` in the context of the forward formula). For sell offers, `buyerPrice > WAD` is arithmetically valid in `Midnight.take`. The fix should either:

1. **Restrict the guard to buy offers only**: `require(offer.buy || buyerPrice <= WAD, ...)` — but this still leaves the inversion undefined for sell offers at `MAX_TICK` with `settlementFee > 0`.
2. **Preferred fix**: In `buyWithAssetsTargetAndWithdrawCollateral`, add a pre-check before calling `buyerAssetsToUnits`. If `tickToPrice(offer.tick) + settlementFee > WAD`, fall back to computing `unitsToTake` via `mulDivUp(remainingAssets, WAD, buyerPrice)` directly (mirroring the `testSnappedBuyerAssets*` test pattern), rather than calling `buyerAssetsToUnits`.
3. **Alternative**: Wrap the `buyerAssetsToUnits` call in a `try/catch` in the bundler and handle the `PriceGreaterThanOne` revert as a special case. [12](#0-11) 

## Proof of Concept

Minimal Foundry test extending `TakeAmountsTest`:

```solidity
function testBuyWithAssetsTargetRevertsAtMaxTickWithFee() public {
    // Set non-zero 0-day settlement fee (e.g., 1 CBP)
    midnight.touchMarket(market);
    midnight.setMarketSettlementFee(id, 0, 1e12); // 1 * CBP

    // Warp to maturity so timeToMaturity = 0
    vm.warp(market.maturity);

    // Confirm tickToPrice(MAX_TICK) = WAD
    assertEq(TickLib.tickToPrice(MAX_TICK), WAD);

    // Confirm settlementFee > 0
    uint256 fee = midnight.settlementFee(id, 0);
    assertGt(fee, 0);

    // buyerAssetsToUnits must revert with PriceGreaterThanOne
    offer.tick = MAX_TICK;
    vm.expectRevert(TickLib.PriceGreaterThanOne.selector);
    TakeAmountsLib.buyerAssetsToUnits(address(midnight), id, offer, 1e18);
}
```

This directly exercises the revert path in `TakeAmountsLib.buyerAssetsToUnits` at line 28 with the exact preconditions described. The bundler DoS follows because `buyWithAssetsTargetAndWithdrawCollateral` calls this function unconditionally at line 209. [13](#0-12) [14](#0-13)
