### Title
Post-maturity bad-debt liquidation reverts with arithmetic underflow when `lif < maxLif` and `seizedAssets > 0` - (File: src/Midnight.sol)

### Summary
When a position has bad debt (`badDebt > 0`) and is liquidated in post-maturity mode during the ramp-up window (`block.timestamp - market.maturity < TIME_TO_MAX_LIF`), the effective `lif` is strictly less than `maxLif`. Bad debt is realized using `maxLif`, leaving `_position.debt = Σ collateral[i].mulDivUp(price_i, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif_i)`. But `repaidUnits` is then computed with the smaller `lif`, yielding a larger value that can exceed the already-reduced `_position.debt`, causing an arithmetic underflow at line 676 and reverting the entire liquidation.

### Finding Description

**Code path:**

1. **Lines 605–618** — bad debt loop uses `maxLif` for every collateral:
   ```solidity
   badDebt = badDebt.zeroFloorSub(
       _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
   );
   ```
   After the loop, if `badDebt > 0`, `_position.debt` after line 628 equals exactly `Σ collateral[i].mulDivUp(price_i, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif_i)`.

2. **Lines 645–647** — `lif` is computed as a linear ramp in post-maturity mode:
   ```solidity
   uint256 lif = postMaturityMode
       ? UtilsLib.min(_maxLif, WAD + (_maxLif - WAD) * (block.timestamp - market.maturity) / TIME_TO_MAX_LIF)
       : _maxLif;
   ```
   When `block.timestamp - market.maturity < TIME_TO_MAX_LIF` (i.e., within the 15-minute ramp), `lif < maxLif`.

3. **Line 650** — `repaidUnits` is computed with the smaller `lif`:
   ```solidity
   repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
   ```
   Because `lif < maxLif`, dividing by a smaller denominator yields a strictly larger `repaidUnits` than the same expression with `maxLif`.

4. **Line 676** — unchecked subtraction:
   ```solidity
   _position.debt -= UtilsLib.toUint128(repaidUnits);
   ```
   No guard ensures `repaidUnits <= _position.debt`. Solidity 0.8 checked arithmetic reverts.

**Why existing checks do not prevent this:**
- The `maxRepaid` / RCF guard (lines 655–668) is wrapped in `if (!postMaturityMode)` and is entirely skipped in post-maturity mode.
- There is no upper bound on `repaidUnits` relative to `_position.debt` in the post-maturity branch.

**Concrete numeric example (single collateral, `ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`):**

| Parameter | Value |
|---|---|
| `collateral` | 1 000 |
| `price` | `ORACLE_PRICE_SCALE` (= 1) |
| `maxLif` | `1.1e18` |
| `lif` (early post-maturity) | `1.05e18` |
| `originalDebt` | 1 000 |

- `collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif)` = `⌈1000/1.1⌉` = **910**
- `badDebt` = 1000 − 910 = **90**; `_position.debt` after line 628 = **910**
- `repaidUnits` = `⌈1000/1.05⌉` = **953**
- Line 676: `910 − 953` → **arithmetic underflow → revert**

**Attacker inputs:** `postMaturityMode = true`, `seizedAssets = collateral` (or any value large enough that `repaidUnits > debtAfterBadDebt`), any oracle price that produces `badDebt > 0`.

### Impact Explanation
Any liquidator calling `liquidate` with `seizedAssets > 0` on a position that has bad debt during the 15-minute post-maturity ramp window will receive an arithmetic underflow revert. The position cannot be fully liquidated via the `seizedAssets` input path during this window, violating the core invariant that unhealthy positions must always remain liquidatable.

### Likelihood Explanation
The preconditions are:
- Market has passed maturity (permissionless, time-based).
- Position has bad debt (oracle price dropped below the `maxLif`-weighted collateral value — a normal market event).
- Liquidation is attempted within 15 minutes of maturity with `seizedAssets > 0`.

All three conditions are reachable without any privileged action. The 15-minute window is short but is precisely the window where liquidation incentives are lowest and liquidators are most likely to use the `seizedAssets` input to seize all available collateral. The bug is deterministic and repeatable for any matching parameter set.

### Recommendation
Before computing `repaidUnits` from `seizedAssets` in the post-maturity branch, cap `repaidUnits` to `_position.debt` (after bad debt realization), or equivalently cap `seizedAssets` to the amount that would produce `repaidUnits == _position.debt`. A minimal fix at line 650:

```solidity
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
// Add: cap repaidUnits to remaining debt to prevent underflow when lif < maxLif and badDebt > 0
if (repaidUnits > _position.debt) repaidUnits = _position.debt;
```

Alternatively, compute `seizedAssets` from `_position.debt` (as the maximum repayable) and use that as the cap on `seizedAssets` before computing `repaidUnits`.

    setupMarket(market, units);

    vm.expectRevert(stdError.arithmeticError);
    midnight.liquidate(
        market, 0, allCollateral, 0, borrower, true, address(this), address(0), ""
    );
}
