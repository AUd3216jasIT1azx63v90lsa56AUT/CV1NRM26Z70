### Title
`sellerAssetsToUnits` Division-by-Zero When `sellerPrice == 0` on Buy Offer DoS's Periphery Bundler - (File: src/periphery/TakeAmountsLib.sol)

### Summary
When `offer.buy = true` and the market's `settlementFee` equals `tickToPrice(offer.tick)`, `sellerPrice` computes to exactly `0`. `sellerAssetsToUnits` then unconditionally executes `targetSellerAssets.mulDivUp(WAD, 0)`, which panics with division-by-zero. The core `Midnight.take()` does not revert in this state (it computes `sellerAssets = 0` via `mulDivDown(0, WAD)`), so the periphery and core diverge, and any bundler call routing through `sellerAssetsToUnits` is permanently DoS'd for that offer.

### Finding Description

**Root cause — `TakeAmountsLib.sol` lines 44–46:**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

When `offer.buy = true` and `settlementFee == offerPrice`, `sellerPrice = 0`. The return statement immediately calls `mulDivUp(WAD, 0)` — a Solidity 0.8 panic (division by zero). There is no guard for this edge case. [1](#0-0) 

**Core does not revert in the same state — `Midnight.sol` lines 361–364:**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
// sellerPrice = 0 when settlementFee == offerPrice
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...;
// mulDivDown(0, WAD) = 0 — no revert
```

`mulDivDown` with a zero numerator returns 0 without reverting, so `take()` succeeds with `sellerAssets = 0`. [2](#0-1) 

**Bundler call site — `MidnightBundles.sol` lines 285–291:**

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(          // <-- panics here
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) { ... } catch {}  // try/catch is AFTER the panic
```

The `sellerAssetsToUnits` call is **outside** the `try/catch` block. The panic propagates and reverts the entire `supplyCollateralAndSellWithAssetsTarget` call. [3](#0-2) 

**Exploit flow:**
1. Governance sets `settlementFee` to value `S` (privileged, but creates the exploitable state).
2. An unprivileged lender creates a buy offer at tick `T` where `tickToPrice(T) == S` — fully within user control.
3. A borrower (or any caller) invokes `supplyCollateralAndSellWithAssetsTarget` with `takes` containing that offer and `targetSellerAssets > 0`.
4. `sellerAssetsToUnits` is called with `sellerPrice = 0`, panics, and the entire bundler transaction reverts.

Alternatively, if the fee was 0 when the offer was created (so `sellerPrice > 0`), and governance later raises the fee to equal `offerPrice`, the same DoS is triggered retroactively for any bundler call referencing that offer.

**Why existing checks fail:**
The NatSpec comment on `sellerAssetsToUnits` states *"Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)"* — but this only covers the strict-less-than case (underflow revert in both). The equal case (`offerPrice == settlementFee`) is not covered: the core succeeds, the periphery panics. [4](#0-3) 

### Impact Explanation
Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `tickToPrice(offer.tick) == settlementFee` and `targetSellerAssets > 0` reverts unconditionally. The bundler is DoS'd for that offer. Because the `try/catch` only wraps `take()` and not `sellerAssetsToUnits`, there is no graceful fallback. Funds are not frozen in the core (take() still works directly), but the periphery path — which is the primary UX entry point for borrowers — is completely blocked. [5](#0-4) 

### Likelihood Explanation
**Preconditions:**
- Settlement fee must equal `tickToPrice(T)` for some tick `T`. This is a governance-set value, but once set, any lender can create a buy offer at tick `T`.
- The fee can also drift into equality retroactively if governance raises the fee after an offer is placed at a low tick.
- `targetSellerAssets > 0` — always true for any real bundler call.

**Feasibility:** Low-to-medium. Requires the settlement fee to land exactly on a tick price. Given that `tickToPrice` is a discrete function over a finite tick range, and settlement fees are set in fixed increments (rounded to `1e12`), exact equality is achievable. The scenario is more likely after a fee increase that crosses a tick boundary.

**Repeatability:** Deterministic — every bundler call with that offer reverts until the offer expires or the fee changes.

### Recommendation
Add a zero-check for `sellerPrice` in `sellerAssetsToUnits`. When `sellerPrice == 0` and `targetSellerAssets > 0`, the function should revert with a descriptive error (e.g., `SellerPriceIsZero`) rather than panicking. When `targetSellerAssets == 0`, return 0 directly. This mirrors the existing guard in `buyerAssetsToUnits` for `buyerPrice > WAD`.

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
if (targetSellerAssets == 0) return 0;
require(sellerPrice > 0, SellerPriceIsZero());
return offer.buy
    ? targetSellerAssets.mulDivUp(WAD, sellerPrice)
    : targetSellerAssets.mulDivDown(WAD, sellerPrice);
``` [6](#0-5)
