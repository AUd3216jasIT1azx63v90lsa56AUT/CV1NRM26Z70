### Title
Sponsor can redirect CORRUPTED sweep to arbitrary address by changing `recoveryAddress` post-outcome — (File: src/ConfidencePool.sol)

### Summary
`setRecoveryAddress()` has no guard preventing it from being called after `flagOutcome()` resolves the pool as CORRUPTED. A malicious sponsor can silently redirect the entire pool balance — stakers' principal plus bonus — to an attacker-controlled address by calling `setRecoveryAddress()` in the window between `flagOutcome()` and `claimCorrupted()`.

### Finding Description
`setRecoveryAddress()` is gated only by `onlyOwner` and a zero-address check: [1](#0-0) 

There is no guard on `outcome`, `claimsStarted`, or any post-resolution latch. After the moderator calls `flagOutcome(CORRUPTED, false, address(0))`, the pool's entire balance is destined for `recoveryAddress` via `claimCorrupted()`: [2](#0-1) 

`claimCorrupted()` reads `recoveryAddress` at call time — it is not snapshotted at `flagOutcome()`. The sponsor therefore controls the sweep destination up until the moment `claimCorrupted()` executes.

The analog to `StETHOracle.setPrice()` is exact: just as the oracle owner could write `lastGoodPrice` directly to produce an arbitrary price consumed by `fetchPrice_view()`, the pool sponsor can write `recoveryAddress` directly to produce an arbitrary sweep destination consumed by `claimCorrupted()`.

By contrast, `expiry` is protected by a one-way `expiryLocked` latch once the first stake arrives: [3](#0-2) 

No equivalent post-flagging latch exists for `recoveryAddress`. `DESIGN.md §10` acknowledges the sponsor controls `recoveryAddress` but does not document post-outcome mutability as intentional, and the `expiryLocked` pattern shows the codebase already applies one-way latches to protect stakers from sponsor manipulation of critical parameters. [4](#0-3) 

### Impact Explanation
Under bad-faith CORRUPTED resolution, `claimCorrupted()` sweeps `stakeToken.balanceOf(address(this))` — the full pool — to `recoveryAddress`. If the sponsor has replaced `recoveryAddress` with an attacker-controlled address, every staker loses their entire principal and bonus with no recourse. The loss is total and irreversible.

### Likelihood Explanation
The sponsor is not a trusted actor (the prompt explicitly distinguishes "pool sponsor/agreement owner" from "trusted moderator/DAO" and "trusted factory owner"). The attack requires only that the sponsor act maliciously in the window between `flagOutcome()` and `claimCorrupted()`. That window is permissionless and unbounded — no one is forced to call `claimCorrupted()` immediately. A sponsor who created the pool with a legitimate `recoveryAddress` to attract stakers, then switches it post-flag, is a realistic adversarial scenario.

### Recommendation
Add an outcome guard to `setRecoveryAddress()` that prevents mutation once the pool is resolved:

```solidity
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (outcome != PoolStates.Outcome.UNRESOLVED) revert OutcomeAlreadySet();
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    address oldRecoveryAddress = recoveryAddress;
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
```

Alternatively, snapshot `recoveryAddress` into a local variable inside `flagOutcome()` (analogous to how `snapshotTotalStaked` freezes accounting at resolution time) and use the snapshot in all post-resolution transfer paths.

### Proof of Concept

1. Sponsor deploys pool with `recoveryAddress = legitimateAddress`. Stakers observe this and deposit.
2. The agreement is breached; the registry transitions to `CORRUPTED`.
3. Moderator calls `flagOutcome(PoolStates.Outcome.CORRUPTED, false, address(0))` (bad-faith). `outcome` is now `CORRUPTED`, `corruptedReserve = snapshotTotalStaked + snapshotTotalBonus`.
4. Before anyone calls `claimCorrupted()`, sponsor calls `setRecoveryAddress(attackerAddress)`. The call succeeds — no revert.
5. Any caller (or the sponsor themselves) calls `claimCorrupted()`. The function reads the live `recoveryAddress`:
   ```solidity
   stakeToken.safeTransfer(recoveryAddress, toSweep); // recoveryAddress == attackerAddress
   ```
6. The entire pool balance is transferred to `attackerAddress`. All stakers lose principal and bonus.
