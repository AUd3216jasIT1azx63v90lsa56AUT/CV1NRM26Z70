### Title
Reentrancy via `onLiquidate` callback allows attacker to grant themselves authorization over borrower and cancel all sell offers - (File: src/periphery/EcrecoverAuthorizer.sol / src/Midnight.sol)

### Summary

`Midnight.liquidate` issues an external `onLiquidate` callback to an attacker-controlled address before pulling the repayment token, with no reentrancy guard present anywhere in the contract. During that callback window, an attacker who holds a valid pre-signed `Authorization` from the borrower can submit it to `EcrecoverAuthorizer.setIsAuthorized`, gain `isAuthorized[borrower][attacker] = true`, and immediately call `Midnight.setConsumed(group, type(uint256).max, borrower)` to cancel every sell offer the borrower has in that group.

### Finding Description

**Code path and root cause**

`Midnight.liquidate` (src/Midnight.sol lines 698–717) follows this ordering:

1. All position state is mutated (debt reduced, collateral seized, loss factor updated). [1](#0-0) 
2. Seized collateral is transferred to `receiver`. [2](#0-1) 
3. `ILiquidateCallback(callback).onLiquidate(...)` is called — **attacker-controlled external call**. [3](#0-2) 
4. Repayment token is pulled from `payer`. [4](#0-3) 

There is **no reentrancy guard** anywhere in `Midnight.sol` (confirmed: zero matches for `nonReentrant`, `ReentrancyGuard`, `_locked`, `_entered`). The Certora formal specs explicitly acknowledge this gap: *"Assumes no reentrancy: callbacks and token transfers are not modeled as re-entering Midnight, so re-entrant consumed changes are not covered."* [5](#0-4) 

**`EcrecoverAuthorizer.setIsAuthorized` accepts any caller**

`EcrecoverAuthorizer.setIsAuthorized` is permissionless — any address can submit a pre-signed `Authorization` struct. It verifies the EIP-712 signature, increments the authorizer's nonce, and calls `IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer)`. [6](#0-5) 

**`Midnight.setIsAuthorized` and `setConsumed` only check `isAuthorized`**

```solidity
// setIsAuthorized
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
isAuthorized[onBehalf][authorized] = newIsAuthorized;

// setConsumed
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
consumed[onBehalf][group] = amount;
``` [7](#0-6) [8](#0-7) 

Neither function has any lock or re-entrancy check. Once `isAuthorized[borrower][attacker]` is set to `true` inside the callback, `setConsumed` passes immediately.

**Full exploit flow**

Preconditions:
- Borrower has called `Midnight.setIsAuthorized(EcrecoverAuthorizer, true, borrower)` (required to use EcrecoverAuthorizer at all).
- Borrower previously signed `Authorization{authorizer: borrower, authorized: attacker, isAuthorized: true, nonce: N, deadline: T}` (e.g., for a trading bot or operator) and the nonce `N` is still valid.
- Borrower has active sell offers in group `G` with `consumed[borrower][G] < type(uint256).max`.
- Borrower is liquidatable.

Attack sequence:
1. Attacker calls `Midnight.liquidate(market, ..., borrower, ..., callback=attacker, ...)`.
2. Midnight mutates position state, transfers collateral to `receiver`, then calls `attacker.onLiquidate(...)`.
3. Inside `onLiquidate`, attacker calls `EcrecoverAuthorizer.setIsAuthorized(authorization, signature)`.
4. EcrecoverAuthorizer verifies the borrower's signature, increments `nonce[borrower]` to `N+1`, and calls `Midnight.setIsAuthorized(attacker, true, borrower)`.
5. `isAuthorized[borrower][attacker]` is now `true`.
6. Attacker calls `Midnight.setConsumed(G, type(uint256).max, borrower)`.
7. `consumed[borrower][G] = type(uint256).max` — all of the borrower's sell offers in group `G` are permanently cancelled.
8. `onLiquidate` returns `CALLBACK_SUCCESS`; Midnight pulls the repayment token. Liquidation completes normally.

**Why existing checks fail**

- `liquidationLocked` only prevents liquidation of the *seller* during `take` callbacks; it does not block `setConsumed` or `setIsAuthorized` calls during `liquidate` callbacks. [9](#0-8) 
- `setConsumed` only checks `isAuthorized[onBehalf][msg.sender]`, which the attacker just set to `true` in step 5. [10](#0-9) 
- The Certora `onlyAuthorizedCanChangeConsumedExceptTake` rule explicitly excludes reentrancy from its scope. [11](#0-10) 

### Impact Explanation

`consumed[borrower][G]` is set to `type(uint256).max` during the liquidation callback. Every sell offer the borrower has in group `G` is effectively cancelled: any subsequent `take` call will hit `require(newConsumed <= offer.maxUnits/maxAssets, ConsumedUnits/Assets())` and revert. [12](#0-11) 

The borrower cannot use those offers to unwind their remaining debt post-liquidation. The attacker can target all groups the borrower uses by repeating the attack across multiple liquidations (if the borrower remains liquidatable) or by holding multiple pre-signed authorizations with sequential nonces.

### Likelihood Explanation

**Preconditions:**
- Borrower must have authorized `EcrecoverAuthorizer` in Midnight — this is the standard setup for any user of the periphery contract, so it is common.
- Attacker must hold a valid pre-signed `Authorization` from the borrower. This is realistic when: (a) the borrower previously authorized an operator and the operator turns adversarial; (b) the borrower signed an authorization for a contract that was later compromised; (c) the borrower revoked authorization in Midnight directly but forgot to invalidate the EcrecoverAuthorizer nonce.
- Borrower must be liquidatable — a normal market condition.

The attack is repeatable as long as the attacker holds unused pre-signed authorizations with valid sequential nonces.

### Recommendation

1. **Add a reentrancy guard** to `Midnight.sol` (e.g., OpenZeppelin `ReentrancyGuard` or a transient-storage lock) covering all state-mutating external entry points. This is the root fix.
2. Alternatively, **snapshot and assert `consumed` state** around the `onLiquidate` callback, reverting if it changed.
3. **Document** that users who authorize `EcrecoverAuthorizer` should invalidate outstanding signed authorizations (by self-submitting a dummy authorization to advance the nonce) before revoking an operator in Midnight.

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {EcrecoverAuthorizer, Authorization, Signature} from "src/periphery/EcrecoverAuthorizer.sol";
import {ILiquidateCallback} from "src/interfaces/ICallbacks.sol";

contract AttackerCallback is ILiquidateCallback {
    EcrecoverAuthorizer ecrecover_auth;
    Midnight midnight;
    Authorization auth;
    Signature sig;
    bytes32 group;

    constructor(address _midnight, address _ecrecover, Authorization memory _auth, Signature memory _sig, bytes32 _group) {
        midnight = Midnight(_midnight);
        ecrecover_auth = EcrecoverAuthorizer(_ecrecover);
        auth = _auth;
        sig = _sig;
        group = _group;
    }

    function onLiquidate(...) external returns (bytes32) {
        // Step 1: submit pre-signed authorization during callback
        ecrecover_auth.setIsAuthorized(auth, sig);
        // Step 2: now authorized over borrower, cancel all offers
        midnight.setConsumed(group, type(uint256).max, auth.authorizer);
        return CALLBACK_SUCCESS;
    }
}

        Authorization memory auth = Authorization({authorizer: borrower, authorized: attacker, isAuthorized: true, nonce: 0, deadline: block.timestamp + 1 days});
        Signature memory sig = Signature({v: v, r: r, s: s});

        // Borrower creates sell offers in group G
        // ... (setup offers, consumed[borrower][G] = 0)

        // Make borrower liquidatable (drop oracle price)
        // ...

        uint256 consumedBefore = midnight.consumed(borrower, group);

        // Attacker deploys callback and initiates liquidation
        AttackerCallback cb = new AttackerCallback(address(midnight), address(ecrecoverAuth), auth, sig, group);

**Expected assertions:**
- `consumed[borrower][group]` equals `type(uint256).max` after the liquidation call, up from its pre-liquidation value.
- Any subsequent `midnight.take(offer, ...)` where `offer.maker == borrower && offer.group == group` reverts with `ConsumedUnits` or `ConsumedAssets`.
- `isAuthorized[borrower][attacker]` is `true` after the call, despite the borrower never directly authorizing the attacker in Midnight.

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
