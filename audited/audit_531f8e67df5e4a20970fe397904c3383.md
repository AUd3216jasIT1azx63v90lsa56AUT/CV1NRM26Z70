Looking at the exact code path in `HashLib.isLeaf` and `EcrecoverRatifier.isRatified`:

### Title
`HashLib.LeafIndexOutOfRange` surfaces instead of `IEcrecoverRatifier.InvalidProof` when `proof.length == 0` and `leafIndex != 0` - (File: src/ratifiers/EcrecoverRatifier.sol)

### Summary
In `EcrecoverRatifier.isRatified`, the call `require(HashLib.isLeaf(...), InvalidProof())` only substitutes `InvalidProof()` when `isLeaf` returns `false`. When `isLeaf` itself reverts internally — as it does when `leafIndex >> proof.length != 0` — the inner revert propagates directly to the caller, bypassing the `InvalidProof()` wrapper entirely. A taker supplying `proof.length == 0` with `leafIndex == 1` triggers this divergence.

### Finding Description
**Code path:**

`EcrecoverRatifier.isRatified` (line 37):
```solidity
require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
```

`HashLib.isLeaf` (line 58):
```solidity
require(leafIndex >> proof.length == 0, LeafIndexOutOfRange());
```

With `proof.length == 0` and `leafIndex == 1`: `1 >> 0 == 1`, which is `!= 0`, so `HashLib.isLeaf` reverts with `HashLib.LeafIndexOutOfRange()`. Because this is an internal revert (not a `false` return), Solidity's `require` at line 37 never gets to evaluate its error argument — the `LeafIndexOutOfRange` revert propagates directly up the call stack.

**Attacker inputs:** Any taker calls `midnight.take()` with `ratifierData` ABI-encoded as `(sig, root, 1, new bytes32[](0))`. The `isRatified` guard `require(msg.sender == MIDNIGHT)` is satisfied because Midnight is the caller of `isRatified`; the taker reaches this path via `midnight.take()`.

**Why existing checks fail:** The `NotMidnight` guard only prevents direct calls to `isRatified`; it does not prevent a taker from routing crafted `ratifierData` through `midnight.take()`. There is no pre-validation of `leafIndex` vs `proof.length` in `EcrecoverRatifier` before delegating to `HashLib.isLeaf`.

### Impact Explanation
The transaction reverts in both the expected and actual cases — no funds are at risk and no state is corrupted. The divergence is purely in the error selector: callers and off-chain tooling that catch `IEcrecoverRatifier.InvalidProof` will not match `HashLib.LeafIndexOutOfRange`, causing unexpected error-handling behavior in integrations that distinguish revert reasons.

### Likelihood Explanation
Trivially reachable by any taker with zero preconditions: supply `leafIndex = 1` and an empty proof array. Repeatable on every call. No special state or privilege required.

### Recommendation
Add an explicit pre-check in `EcrecoverRatifier.isRatified` before calling `HashLib.isLeaf`, using the same `InvalidProof` error:

```solidity
require(leafIndex >> proof.length == 0, InvalidProof());
require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
```

This ensures all invalid-proof conditions — whether `leafIndex` out of range or hash mismatch — surface uniformly as `InvalidProof` to callers.
