Audit Report

## Title
Fee-on-transfer collateral token inflates stored collateral, enabling undercollateralized borrowing - (File: src/Midnight.sol)

## Summary
`supplyCollateral` writes the caller-supplied `assets` value into `_position.collateral[collateralIndex]` before invoking `SafeTransferLib.safeTransferFrom`, which performs no balance-delta check. For a fee-on-transfer collateral token the contract receives `assets - fee` but records `assets`, inflating stored collateral. `isHealthy` computes `maxDebt` from the inflated stored value, allowing the borrower to take debt that exceeds the actual on-chain collateral, creating unrecoverable bad debt.

## Finding Description
**Root cause — credits before transfer, no balance-delta check:**

In `supplyCollateral` (`src/Midnight.sol:533`):
```solidity
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
```
The position is credited with the full `assets` parameter. The transfer occurs later at line 545:
```solidity
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```
`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol:27-33`) calls `transferFrom` and checks only the boolean return value — it does not snapshot the contract balance before and after to verify the actual received amount. For a fee-on-transfer token with fee rate `f`, the contract receives `assets * (1 - f)` but `_position.collateral[collateralIndex]` stores `assets`.

**`isHealthy` uses the inflated stored value:**
Per `live_context.json` the max-debt formula is `sum(collateral_amount_i * oracle_price_i * LLTV_i)`, reading `_position.collateral[i]` directly. The resulting `maxDebt` is computed from `assets` rather than the actual `assets * (1 - f)` held by the contract.

**Exploit flow:**
1. Attacker calls `touchMarket` with a market whose `collateralParams[i].token` is a fee-on-transfer ERC20. `touchMarket` is permissionless and validates only LLTV tiers, maxLif, and sorted token addresses — no token-type restriction.
2. Attacker calls `supplyCollateral(market, i, assets, borrower)`. Contract records `assets` in `_position.collateral[i]` but receives only `assets - fee`.
3. Attacker calls `take` to increase debt. At line 476, `isHealthy` is called with the inflated stored collateral, returning `true` for debt up to `assets * lltv` instead of the correct `(assets - fee) * lltv`.
4. Position is now undercollateralized: `debt > actual_collateral_value`.

**Why existing checks fail:**
- `SafeTransferLib` checks only `transferFrom` success, not balance delta.
- `isHealthy` reads `_position.collateral[i]` directly with no access to actual token balance.
- `touchMarket` imposes no restriction on fee-on-transfer tokens.
- The TOKEN SAFETY REQUIREMENTS comment (`src/Midnight.sol:139`) is documentation only; it is not enforced on-chain.

## Impact Explanation
An undercollateralized position is created. The gap between recorded collateral and actual collateral is `fee * assets` per `supplyCollateral` call. For a 1% fee token with LLTV 0.77, supplying 1000 units records 1000 but holds 990; the borrower can take 770 units of debt while the actual collateral supports only 762.3 units. When the position is liquidated, the seized collateral is worth less than the repaid debt, realizing bad debt that is socialized to lenders — a direct match for the "bad debt creation" and "protocol insolvency" impact classes listed in `live_context.json`.

## Likelihood Explanation
`touchMarket` is permissionless — any address can create a market with a fee-on-transfer collateral token without governance approval, whitelist, or privileged role. The attacker can simultaneously be market creator, borrower, and lender. Fee-on-transfer tokens exist on mainnet. The attack is repeatable on every `supplyCollateral` call, compounding the inflation with each deposit. `live_context.json` (`core_invariants.external_calls`) explicitly states that "fee-on-transfer … should be tested if not explicitly excluded," and no explicit exclusion exists in `SECURITY.md` or anywhere else in the repository.

## Recommendation
Replace the credits-before-transfer pattern with a balance-delta check:
```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```
Alternatively, enforce the token safety requirement on-chain by reverting if `received != assets`. Either approach closes the gap between recorded and actual collateral for fee-on-transfer tokens.

## Proof of Concept
**Foundry fork test outline:**
1. Deploy a mock ERC20 that deducts a 1% fee on every `transferFrom`.
2. Call `touchMarket` with this token as `collateralParams[0].token`, valid LLTV, and a real oracle.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
