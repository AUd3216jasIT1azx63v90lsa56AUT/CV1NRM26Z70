### Title
Fee-on-transfer `loanToken` causes `withdrawable` over-accounting, freezing last lenders' funds - (`File: src/Midnight.sol`)

### Summary
`repay` unconditionally adds `units` to `marketState[id].withdrawable` and then calls `safeTransferFrom` for exactly `units` tokens, but with a fee-on-transfer `loanToken` the contract receives only `units * (1 - f)`. Because `withdraw` later sends the full `units` out, the contract's real token balance falls short of its internal `withdrawable` accounting. Once early lenders drain the actual balance, every subsequent `withdraw` call reverts, permanently freezing the remaining lenders' credit.

### Finding Description
**Root cause — `repay` (lines 508–520):**

```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
marketState[id].withdrawable += UtilsLib.toUint128(units);   // ← records full `units`
// ...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
// ← with FOT token, contract receives units*(1-f), not units
```

`safeTransferFrom` only verifies the call did not revert and returned `true`; it performs no before/after balance check. [1](#0-0) [2](#0-1) 

**Consumption path — `withdraw` (lines 493–499):**

```solidity
_position.credit -= UtilsLib.toUint128(units);
_marketState.withdrawable -= UtilsLib.toUint128(units);
// ...
SafeTransferLib.safeTransfer(market.loanToken, receiver, units);  // ← sends full `units`
```

`withdraw` sends the full `units` amount, trusting that the balance covers `withdrawable`. With a FOT token it does not. [3](#0-2) 

**No exclusion of FOT tokens:** `touchMarket` accepts any address as `loanToken` with no token-type restriction. [4](#0-3) 

The Certora solvency invariant (`tokenBalanceCorrect`) is proven only under the explicit assumption that tokens transfer without fees: [5](#0-4) 

The project's own `live_context.json` lists "fee-on-transfer … should be tested if not explicitly excluded" — confirming no explicit exclusion exists in the protocol. [6](#0-5) 

**Exploit flow (N lenders, fee rate `f > 0`):**

1. Any user creates a market whose `loanToken` is a fee-on-transfer token (fee rate `f`).
2. Lenders A … N each acquire `units_i` credit via `take`.
3. Borrower calls `repay(T)` where `T = Σ units_i`. Contract records `withdrawable += T` but receives only `T*(1-f)` tokens.
4. Lender A calls `withdraw(units_A)` → succeeds; contract balance drops to `T*(1-f) - units_A`.
5. Lender B calls `withdraw(units_B)` → `safeTransfer` attempts to send `units_B` but contract holds only `T*(1-f) - units_A < T - units_A = units_B + … + units_N`. Reverts.
6. All lenders from B onward are permanently unable to withdraw.

No privileged role is required. The market creator, borrower, and lenders are all unprivileged roles.

### Impact Explanation
Every lender whose `withdraw` call is ordered after the contract's real balance is exhausted receives a hard revert. Their credit is non-zero and `withdrawable` is non-zero, yet the ERC-20 transfer fails. The funds are permanently frozen inside the contract with no recovery path — no admin function exists to top up the balance or adjust `withdrawable` downward.

### Likelihood Explanation
**Preconditions:** (a) a market exists with a fee-on-transfer `loanToken`; (b) at least two lenders hold credit; (c) a borrower repays. All three are normal, intended protocol operations. Any user can create such a market permissionlessly. Fee-on-transfer tokens (e.g., tokens with a built-in burn or redistribution fee) are common on-chain. The bug is deterministic and repeatable: every repayment in such a market widens the deficit by `repaid * f`.

### Recommendation
In `repay`, measure the actual received amount with a before/after balance check and credit only the delta to `withdrawable`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
marketState[id].withdrawable += UtilsLib.toUint128(received);
position[id][onBehalf].debt -= UtilsLib.toUint128(received);
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as `loanToken` (e.g., via a registry or a deployment-time check), and add an invariant test that asserts `balanceOf(address(this)) >= withdrawable + claimableSettlementFee + collateralSum` after every operation.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Market, ...} from "src/Midnight.sol";

// FeeOnTransferToken: charges 10% on every transferFrom
contract FeeOnTransferToken is ERC20 {
    function transferFrom(address from, address to, uint256 amount)
        public override returns (bool)
    {
        uint256 fee = amount / 10;
        super.transferFrom(from, address(0xdead), fee); // burn fee
        return super.transferFrom(from, to, amount - fee);
    }
}

contract FotInsolvencyTest is Test {
    Midnight midnight;
    FeeOnTransferToken loanToken;
    Market market;

    function setUp() public { /* deploy midnight, create market with loanToken */ }

    function testFotMultiLenderInsolvency() public {
        uint256 units = 1000e18;
        // lenderA and lenderB each hold `units` credit
        // borrower repays `2 * units` → contract receives 1800e18, withdrawable = 2000e18

        // lenderA withdraws `units` → succeeds, contract balance = 800e18
        vm.prank(lenderA);
        midnight.withdraw(market, units, lenderA, lenderA);
        assertEq(loanToken.balanceOf(lenderA), units);

        // lenderB withdraws `units` → MUST revert (contract only has 800e18 < 1000e18)
        vm.prank(lenderB);
        vm.expectRevert();  // ERC20 transfer fails: insufficient balance
        midnight.withdraw(market, units, lenderB, lenderB);

        // Assert: lenderB's credit is non-zero but funds are frozen
        assertGt(midnight.creditOf(id, lenderB), 0);
        assertLt(loanToken.balanceOf(address(midnight)), units);
    }
}
```

**Expected assertions:**
- `lenderA.withdraw` succeeds.
- `lenderB.withdraw` reverts.
- `midnight.creditOf(id, lenderB) > 0` (credit exists but is unredeemable).
- `loanToken.balanceOf(address(midnight)) < units` (balance insufficient to cover remaining `withdrawable`).

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
