### Title
Phantom Collateral via Fee-on-Transfer Token in `supplyCollateral` Enables Undercollateralized Borrowing - (File: src/Midnight.sol)

### Summary
`supplyCollateral` writes `_position.collateral[collateralIndex] = oldCollateral + assets` and sets the bitmap bit before calling `SafeTransferLib.safeTransferFrom`, which only verifies the call succeeded and returned `true` — it never checks the actual amount received. A 100%-fee-on-transfer collateral token causes 0 tokens to arrive while the protocol records `assets` worth of phantom collateral with the bitmap bit set, allowing the borrower to borrow against collateral the contract does not hold.

### Finding Description
**Exact code path** (`src/Midnight.sol` lines 531–545):

```
uint256 oldCollateral = _position.collateral[collateralIndex];          // line 532
_position.collateral[collateralIndex] = toUint128(oldCollateral + assets); // line 533  ← state written first

if (oldCollateral == 0 && assets > 0) {                                 // line 535
    _position.collateralBitmap = _position.collateralBitmap.setBit(collateralIndex); // line 536-537
    require(countBits(...) <= MAX_COLLATERALS_PER_BORROWER, ...);        // line 538-540
}

SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545
```

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34) only checks `success == true` and that the return value decodes to `true`. It performs no balance-before/after comparison and does not verify that `assets` tokens were actually received.

**Attacker inputs:**
- Deploy an ERC20 token that charges a 100% transfer fee (all transferred tokens go to a fee recipient; recipient receives 0).
- Create a market (permissionless via `touchMarket`) with this token as `collateralParams[i].token`.
- Call `supplyCollateral(market, i, assets, self)` with any `assets > 0`.

**Resulting state:**
- `_position.collateral[i]` = `assets` (phantom value)
- `_position.collateralBitmap` has bit `i` set
- `token.balanceOf(address(midnight))` unchanged (0 tokens received)

**Health check consequence** (`src/Midnight.sol` lines 944–960):
`isHealthy` iterates the bitmap and computes:
```
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)
         = assets * price / ORACLE_PRICE_SCALE * lltv / WAD   // non-zero phantom value
```
The position appears healthy with `maxDebt = assets * price * lltv / (ORACLE_PRICE_SCALE * WAD)` despite the contract holding 0 collateral tokens. The borrower can then take a loan up to this phantom `maxDebt`.

**Note on the question's framing:** The question states `maxDebt += 0 * price = 0`; this is incorrect. The stored collateral is `assets`, not 0. The actual impact is worse: phantom collateral inflates `maxDebt`, enabling real undercollateralized borrowing rather than merely wasting a bitmap slot.

**Why existing checks fail:**
- `safeTransferFrom` has no received-amount check.
- `touchMarket` validates LLTV tiers and token ordering but imposes no restriction on token type (no fee-on-transfer guard).
- The Certora `Healthiness.spec` summarizes `transferFrom` as `NONDET` (line 27), so formal verification does not model fee-on-transfer behavior.
- The `nonZeroCollateralsAreActivated` invariant in `CollateralBitmap.spec` (line 39–40) holds on the stored value but the stored value diverges from actual holdings.

### Impact Explanation
The protocol records `assets` phantom collateral for a borrower who deposited 0 tokens. `isHealthy` computes a non-zero `maxDebt` from this phantom collateral, allowing the borrower to take a loan backed by nothing. When the loan is not repaid, the shortfall becomes bad debt socialized across all lenders in the market. The contract's core invariant — "contract balances cover collateral" — is broken: `sum(_position.collateral[i]) > token.balanceOf(midnight)`.

### Likelihood Explanation
**Preconditions:**
1. Market creation is permissionless; the attacker deploys the fee-on-transfer token and creates the market themselves.
2. No whitelist or token-type restriction exists.
3. The attacker needs only to call `supplyCollateral` once with `assets > 0` and `oldCollateral == 0`.

**Feasibility:** Trivially repeatable. The attacker controls the token contract and can set fee = 100%. The attack requires no special privileges, no oracle manipulation, and no reentrancy.

**Repeatability:** Can be executed on any new market the attacker creates. Each call inflates phantom collateral by `assets`.

### Recommendation
Add a received-amount check in `supplyCollateral` using a balance-before/after pattern:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
require(received == assets, FeeOnTransferNotSupported());
```

Alternatively, use `received` instead of `assets` for the collateral accounting update, and only set the bitmap bit if `received > 0`. This mirrors the pattern used by Morpho Blue and Uniswap v2/v3 for fee-on-transfer token safety.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// FeeOnTransferToken: transferFrom sends 0 to recipient, all to fee sink
contract FeeOnTransferToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    address feeSink;
    constructor(address _feeSink) { feeSink = _feeSink; }
    function mint(address to, uint256 amt) external { balanceOf[to] += amt; }
    function approve(address sp, uint256 amt) external returns (bool) {
        allowance[msg.sender][sp] = amt; return true;
    }
    function transferFrom(address from, address, uint256 amt) external returns (bool) {
        allowance[from][msg.sender] -= amt;
        balanceOf[from] -= amt;
        balanceOf[feeSink] += amt; // 100% fee: recipient gets 0
        return true;
    }
    function transfer(address to, uint256 amt) external returns (bool) {
        balanceOf[msg.sender] -= amt; balanceOf[to] += amt; return true;
    }
}

// Foundry test
function testPhantomCollateralFeeOnTransfer() public {
```
