### Title
Empty `takes[]` causes undocumented OOB panic inconsistent with "No-ops are allowed" contract invariant - (File: src/periphery/MidnightBundles.sol)

### Summary
All four bundle functions unconditionally dereference `takes[0]` immediately after the two `require` guards, before any length check. When called with `takes = new Take[](0)`, every function panics with Solidity panic code `0x32` (array out-of-bounds). This directly contradicts the contract-level NatSpec `/// @dev No-ops are allowed.` and is undocumented in the per-function comments, which only state "This function should only be called with the same market for all takes."

### Finding Description
The four affected functions share an identical pattern. After the authorization and `referralFeePct` guards, each unconditionally reads `takes[0]`:

- `buyWithUnitsTargetAndWithdrawCollateral` — line 62: `address loanToken = takes[0].offer.market.loanToken;`
- `supplyCollateralAndSellWithUnitsTarget` — line 129: `address loanToken = takes[0].offer.market.loanToken;`
- `buyWithAssetsTargetAndWithdrawCollateral` — line 193: `address loanToken = takes[0].offer.market.loanToken;`
- `supplyCollateralAndSellWithAssetsTarget` — line 264: `address loanToken = takes[0].offer.market.loanToken;` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

No state changes occur before this dereference — no token transfers, no approvals, no Midnight calls. The panic is the first meaningful side-effect after the two `require` guards. The contract-level NatSpec explicitly states `/// @dev No-ops are allowed.`, which a caller can reasonably interpret as meaning an empty `takes` array is safe to pass. [5](#0-4) 

The per-function comment "The collateral transfers always use the first offer's market" references `takes[0]` semantically but does not document that an empty array causes a panic rather than a graceful no-op or a descriptive revert. [6](#0-5) 

### Impact Explanation
Any unprivileged caller (taker, lender, borrower) can trigger the panic by passing `takes = new Take[](0)`. The revert carries Solidity panic code `0x32` rather than any of the defined custom errors (`OutOfOffers`, `InconsistentMarket`, etc.). No funds are lost and no protocol state is corrupted — the sole concrete impact is: (1) the revert type is undocumented and inconsistent with the "No-ops are allowed" invariant, and (2) no test in the suite covers this path, leaving the discrepancy undetected. [7](#0-6) 

### Likelihood Explanation
Preconditions are trivially satisfiable: any authorized caller, any valid `referralFeePct < WAD`, and `takes = new Take[](0)`. No special state is required. The call is repeatable at zero cost (it reverts before any token transfer). Integrators relying on the "No-ops are allowed" contract comment are the most likely to hit this path.

### Recommendation
Add an explicit length guard at the top of each function, before the `takes[0]` dereference, and document the behavior:

```solidity
if (takes.length == 0) {
    require(targetUnits == 0, OutOfOffers()); // or simply revert OutOfOffers()
    return;
}
```

Alternatively, if empty `takes` should always revert with a descriptive error, replace the implicit OOB panic with:

```solidity
require(takes.length > 0, OutOfOffers());
```

and update the per-function NatSpec to document this precondition explicitly, resolving the contradiction with `/// @dev No-ops are allowed.`

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity 0.8.34;

import "forge-std/Test.sol";
import {MidnightBundles, Take, CollateralWithdrawal, CollateralSupply, TokenPermit, PermitKind}
    from "src/periphery/MidnightBundles.sol";

contract EmptyTakesPanicTest is Test {
    MidnightBundles bundles;

    function setUp() public {
        // deploy with a mock midnight address; panic fires before any Midnight call
        bundles = new MidnightBundles(address(this));
        // authorize self
    }

    function testBuyUnitsTargetEmptyTakesPanics() public {
        Take[] memory takes = new Take[](0);
        // expect panic 0x32 (array out-of-bounds), NOT OutOfOffers()
        vm.expectRevert(stdError.indexOOBError);
        bundles.buyWithUnitsTargetAndWithdrawCollateral(
            0, 0, address(this),
            TokenPermit({kind: PermitKind.None, data: ""}),
            takes, new CollateralWithdrawal[](0), address(0), 0, address(0)
        );
    }

    function testSellUnitsTargetEmptyTakesPanics() public {
        Take[] memory takes = new Take[](0);
        vm.expectRevert(stdError.indexOOBError);
        bundles.supplyCollateralAndSellWithUnitsTarget(
            0, 0, address(this), address(this),
            new CollateralSupply[](0), takes, 0, address(0)
        );
    }

    function testBuyAssetsTargetEmptyTakesPanics() public {
        Take[] memory takes = new Take[](0);
        vm.expectRevert(stdError.indexOOBError);
        bundles.buyWithAssetsTargetAndWithdrawCollateral(
            0, 0, address(this),
            TokenPermit({kind: PermitKind.None, data: ""}),
            takes, new CollateralWithdrawal[](0), address(0), 0, address(0)
        );
    }

    function testSellAssetsTargetEmptyTakesPanics() public {
        Take[] memory takes = new Take[](0);
        vm.expectRevert(stdError.indexOOBError);
        bundles.supplyCollateralAndSellWithAssetsTarget(
            0, type(uint256).max, address(this), address(this),
            new CollateralSupply[](0), takes, 0, address(0)
        );
    }
}
```

All four tests assert `stdError.indexOOBError` (panic `0x32`). A correct fix would make them revert with `OutOfOffers()` instead, or return silently when `targetUnits == 0` and `takes` is empty, consistent with `/// @dev No-ops are allowed.`
