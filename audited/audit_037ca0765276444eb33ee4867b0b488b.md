### Title
Front-running liquidation causes arithmetic underflow revert in `repay`, DoS-ing borrower repayment - (File: src/Midnight.sol)

### Summary
The `repay` function performs an unchecked subtraction `position[id][onBehalf].debt -= UtilsLib.toUint128(units)` with no guard that `units ≤ current debt`. Because any unprivileged liquidator can reduce a borrower's debt between the borrower's debt-read and their repay landing on-chain, a front-running liquidation can make `units > debt`, triggering a Solidity 0.8 arithmetic underflow revert. The borrower's repay is denied for the duration of active liquidation pressure, and the attacker profits from the liquidation itself.

### Finding Description
**Code path:**

`repay` (`src/Midnight.sol:502-521`):
```
position[id][onBehalf].debt -= UtilsLib.toUint128(units);   // line 508
```
No pre-check that `units ≤ position[id][onBehalf].debt` exists anywhere in the function. The only guard is the authorization check at line 505, which is irrelevant to the debt amount.

`liquidate` (`src/Midnight.sol:626-677`) has two independent paths that reduce `_position.debt` before `repay` can land:
- Bad-debt path (line 628): `_position.debt -= uint128(badDebt);`
- Normal repay path (line 676): `_position.debt -= UtilsLib.toUint128(repaidUnits);`

**Exploit flow:**

1. **Precondition**: Borrower holds debt `D` in an unhealthy position (`debt > maxDebt`). Borrower reads `D` off-chain and constructs `repay(market, D, onBehalf, address(0), "")`.
2. **Attacker action**: Attacker (any address, no special privilege) submits `liquidate(market, collateralIndex, 0, X, borrower, false, receiver, address(0), "")` with `0 < X ≤ D` and higher gas, landing before the repay.
3. **State after liquidation**: `position[id][borrower].debt = D - X`.
4. **Trigger**: Borrower's `repay(market, D, ...)` executes line 508: `(D - X) - D` → arithmetic underflow → revert.
5. **Why checks fail**: There is no `require(units <= position[id][onBehalf].debt)` in `repay`. The Solidity 0.8 underflow check is the only protection, and it manifests as the DoS revert rather than preventing it.

The attacker does not need to liquidate the full debt — any `X ≥ 1` is sufficient to cause the revert if the borrower specified `units = D`.

### Impact Explanation
Any borrower with an unhealthy position who attempts to repay their full (or any amount of) outstanding debt can have their `repay` transaction reverted by a front-running liquidation. The borrower must retry with the post-liquidation debt value, but the attacker can repeat the pattern on every retry as long as the position remains partially unhealthy. There is no "repay max" or "repay all" path in the protocol that would allow the borrower to atomically repay whatever debt remains; the `units` parameter must be specified exactly. This constitutes a concrete, repeatable DoS against the `repay` action for any borrower under active liquidation.

### Likelihood Explanation
**Preconditions**: The borrower's position must be unhealthy (or post-maturity), which is exactly the state where a borrower most urgently needs to repay. The attacker needs no special role — any address can call `liquidate` on an unhealthy position. The attacker is economically incentivized (they receive collateral at a discount), so the "attack" is a profitable liquidation whose side effect is the DoS. The attack is repeatable on every repay attempt until the position is fully liquidated or the borrower successfully repays with a precisely timed transaction.

### Recommendation
Add a debt cap in `repay` so that `units` is silently clamped to the current debt, or add an explicit require:

```solidity
// Option A: cap (allows "repay all" semantics)
uint128 currentDebt = position[id][onBehalf].debt;
uint128 repayUnits = units > currentDebt ? currentDebt : UtilsLib.toUint128(units);
position[id][onBehalf].debt = currentDebt - repayUnits;
marketState[id].withdrawable += repayUnits;
// use repayUnits for the transfer and event

// Option B: explicit guard (preserves exact-units semantics, reverts on over-repay)
require(units <= position[id][onBehalf].debt, RepayExceedsDebt());
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
```

Option A is preferred because it eliminates the front-running DoS vector entirely and gives borrowers a safe "repay all" path.

### Proof of Concept
```solidity
function testFrontRunLiquidationDosesRepay() public {
    uint256 D = 100e18;

    // 1. Setup: borrower has debt D, position is unhealthy
    collateralize(market, borrower, D);
    setupMarket(market, D);                          // lender fills D units
    Oracle(market.collateralParams[0].oracle).setPrice(ORACLE_PRICE_SCALE / 2); // drop price → unhealthy

    assertGt(midnight.debtOf(id, borrower), 0);

    // 2. Attacker front-runs: liquidate 1 unit, reducing debt to D-1
    deal(address(loanToken), attacker, 1);
    vm.prank(attacker);
    midnight.liquidate(market, 0, 0, 1, borrower, false, attacker, address(0), "");

    uint256 debtAfterLiq = midnight.debtOf(id, borrower);
    assertEq(debtAfterLiq, D - 1);

    // 3. Borrower's repay(D) now reverts with arithmetic underflow
    deal(address(loanToken), borrower, D);
    vm.prank(borrower);
    vm.expectRevert(stdError.arithmeticError);
    midnight.repay(market, D, borrower, address(0), "");

    // Expected assertion: repay reverts; borrower cannot repay their original debt amount.
    // Fuzz variant: fuzz X in [1, D] for liquidate repaidUnits, assert repay(D) always reverts.
}
```
