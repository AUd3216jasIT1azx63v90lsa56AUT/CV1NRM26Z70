### Title
Fee-on-Transfer Collateral Token Inflates Stored Collateral, Enabling Undercollateralized Borrowing - (File: src/Midnight.sol)

### Summary
`supplyCollateral` credits `_position.collateral[collateralIndex]` with the caller-supplied `assets` value before executing the token transfer, and `SafeTransferLib.safeTransferFrom` only validates the boolean return value — not the actual amount received. Because market creation is permissionless and no on-chain check enforces the documented "exact-amount" token assumption, an attacker can deploy a fee-on-transfer ERC20 as collateral, causing the stored collateral to exceed the real token balance held by Midnight, and then borrow against the inflated figure to create an undercollateralized position.

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34) only checks the boolean return of `transferFrom`; it never measures the balance delta: [3](#0-2) 

`isHealthy` (`src/Midnight.sol` lines 944–959) computes `maxDebt` directly from the stored `_position.collateral[i]`, which is the inflated figure: [4](#0-3) 

`touchMarket` (`src/Midnight.sol` lines 755–791) is permissionless and validates only LLTV tier, maxLif, sort order, and length — it performs no check on whether the collateral token is fee-on-transfer: [5](#0-4) 

The protocol documents the exact-amount requirement as a comment-level assumption only: [6](#0-5) 

There is no on-chain enforcement.

**Exploit flow:**

1. Attacker deploys `FeeToken` — a standard ERC20 that deducts a percentage fee on every `transferFrom` (e.g., 1 %).
2. Attacker calls `touchMarket` with `FeeToken` as the collateral token. This succeeds because `touchMarket` only validates LLTV, maxLif, sort order, and length.
3. An innocent lender places a buy offer in this market, providing loan-token liquidity.
4. Attacker calls `supplyCollateral(market, 0, 1000e18, attacker)`. Storage is updated: `position[id][attacker].collateral[0] = 1000e18`. Then `safeTransferFrom` transfers `FeeToken` — only `990e18` arrive at Midnight (1 % fee). No revert occurs; the boolean return is `true`.
5. Attacker calls `take` on the lender's buy offer, acting as seller. `isHealthy` is called and computes `maxDebt = 1000e18 * price * lltv / WAD / ORACLE_PRICE_SCALE` — using the inflated `1000e18` figure. The position passes the health check.
6. The attacker's actual collateral backing is only `990e18` tokens. The position is undercollateralized by the fee amount times the oracle price times LLTV from the moment of creation.
7. A small oracle price drop (or repeated fee-on-transfer supplies) widens the gap until the position becomes liquidatable, realizing bad debt that is socialized among lenders.

**Why existing checks fail:**

- `safeTransferFrom` checks only the boolean return, not the received amount.
- `isHealthy` and `liquidate` both read `_position.collateral[i]` (the stored, inflated value), not the actual token balance.
- `touchMarket` has no token allowlist or fee-on-transfer detection.
- The Certora `supplyCollateralEffects` rule (`certora/specs/BalanceEffects.spec` line 217) asserts `collateral == collateralBefore + assets` — this is proved against a model where `safeTransferFrom` is summarized as `NONDET`, so it does not catch the real-token discrepancy. [7](#0-6) 

### Impact Explanation

The stored collateral exceeds the real token balance held by Midnight. `isHealthy` returns `true` for a position that is already undercollateralized at inception. The borrower can extract loan tokens whose value exceeds the liquidation value of the actual collateral. When the position is eventually liquidated, the seized collateral is worth less than the repaid debt, and the shortfall is realized as bad debt and socialized among lenders via the loss-factor mechanism — a direct, concrete loss of lender funds.

### Likelihood Explanation

Market creation is fully permissionless; any address can deploy a fee-on-transfer ERC20 and register it as a collateral token. No admin approval, whitelist, or governance vote is required. The attack is repeatable across any number of markets and collateral indices. The only precondition is the presence of a lender willing to supply liquidity to the market, which can be achieved by social engineering or by the attacker seeding initial liquidity themselves and then attracting third-party lenders.

### Recommendation

Add a balance-before / balance-after check in `supplyCollateral` and require that the received amount equals `assets`:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
require(IERC20(collateralToken).balanceOf(address(this)) - balanceBefore == assets, BadTransferAmount());
```

Apply the same pattern to `repay` (loan token) and the flash-loan repayment path. Alternatively, document and enforce a token allowlist at the market-creation level so that only pre-vetted, non-fee-on-transfer tokens can be registered as collateral.
