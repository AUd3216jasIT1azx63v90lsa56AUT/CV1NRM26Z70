Audit Report

## Title
Callback Address Used as Payer Without Caller Authorization, Enabling Theft of Loan Tokens from Contracts Implementing `onLiquidate` - (File: src/Midnight.sol)

## Summary
In `liquidate()`, `payer` is unconditionally set to `callback` when `callback != address(0)` at line 679, with no requirement that `callback` authorized `msg.sender` to use it as payer. Any contract implementing `ILiquidateCallback` that returns `CALLBACK_SUCCESS` without verifying it initiated the call, and holds a Midnight approval for the loan token, can be drained by an unprivileged attacker who passes that contract's address as `callback` while directing seized collateral to themselves via `receiver`.

## Finding Description
**Root cause â€” `src/Midnight.sol` lines 679 and 717:**

```solidity
address payer = callback != address(0) ? callback : msg.sender;  // line 679
```
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits); // line 717
```

The only gate between the `payer = callback` assignment and the token pull is the `onLiquidate` return-value check at lines 698â€“714:

```solidity
require(
    ILiquidateCallback(callback).onLiquidate(msg.sender, ...) == CALLBACK_SUCCESS,
    WrongLiquidateCallbackReturnValue()
);
```

Returning `CALLBACK_SUCCESS` is treated as implicit consent to pay `repaidUnits` of loan token. There is no check that `callback == msg.sender`, that `callback` authorized `msg.sender` to use it as payer, or that `callback` initiated the liquidation.

The contrast with `repay()` at line 505 is direct: `repay()` requires `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` before allowing a third party to be used as payer. `liquidate()` has no analogous constraint on `callback`.

The `liquidatorGate` check at lines 597â€“600 only gates whether `msg.sender` can liquidate at all; it does not constrain the `callback` parameter.

**Exploit flow:**

1. Victim contract `V` implements `onLiquidate`, returns `CALLBACK_SUCCESS` without verifying `caller == address(this)`, and has approved Midnight for the loan token (standard flash-liquidation bot pattern).
2. Attacker identifies a liquidatable borrower `B`.
3. Attacker calls:
   ```solidity
   midnight.liquidate(market, idx, seizedAssets, 0, B, false, attacker, V, "");
   ```
   with `callback = V`, `receiver = attacker`.
4. Execution:
   - Line 679: `payer = V`
   - Line 696: `safeTransfer(collateralToken, attacker, seizedAssets)` â€” attacker receives collateral.
   - Lines 698â€“714: `V.onLiquidate(attacker, ...)` is called; `V` does not check `caller == address(this)`, performs its normal logic, returns `CALLBACK_SUCCESS`.
   - Line 717: `safeTransferFrom(loanToken, V, address(this), repaidUnits)` â€” loan tokens pulled from `V`.
5. Net result: attacker gains `seizedAssets` of collateral at zero personal cost; `V` loses `repaidUnits` of loan token.

The test contract at `test/LiquidationTest.sol` lines 985â€“1009 itself demonstrates the vulnerable pattern: it implements `onLiquidate`, records `_caller` but does not assert `_caller == address(this)`, and returns `CALLBACK_SUCCESS` unconditionally.

## Impact Explanation
Direct theft of loan tokens from any contract that (a) implements `ILiquidateCallback` returning `CALLBACK_SUCCESS` without verifying `caller == address(this)` and (b) holds a Midnight approval for the loan token. The attacker simultaneously receives seized collateral at no personal cost. The loss is concrete, repeatable, and constitutes unauthorized movement of assets outside valid protocol rules. Severity: Critical.

## Likelihood Explanation
All three preconditions are simultaneously satisfied by the primary intended users of the callback feature â€” flash-liquidation bots and protocol integrations that implement `onLiquidate` for their own liquidations and pre-approve Midnight for loan tokens. The `caller` check is non-obvious: the protocol does not document it as a required security invariant, and the reference test implementation omits it. A liquidatable borrower is a normal market condition. The attack requires no privileged access and is executable by any external user.

## Recommendation
Add an authorization check in `liquidate()` analogous to the one in `repay()`. Before setting `payer = callback`, require either `callback == msg.sender` or that `callback` has explicitly authorized `msg.sender` via `isAuthorized`:

```solidity
if (callback != address(0)) {
    require(callback == msg.sender || isAuthorized[callback][msg.sender], Unauthorized());
}
address payer = callback != address(0) ? callback : msg.sender;
```

Alternatively, document `caller == address(this)` as a mandatory security invariant in `ILiquidateCallback` and enforce it at the protocol level rather than relying on implementors. The protocol-level check is strongly preferred.

## Proof of Concept
Deploy a victim contract `V` that:
- Implements `onLiquidate` returning `CALLBACK_SUCCESS` without checking `_caller == address(this)` (matching the pattern in `test/LiquidationTest.sol` lines 985â€“1009).
- Pre-approves `midnight` for `repaidUnits` of loan token and holds that balance.

Then, as an unprivileged attacker:
```solidity
// Setup: liquidatable borrower B exists
// V holds loanToken balance and has approved midnight

uint256 attackerCollateralBefore = collateralToken.balanceOf(attacker);
uint256 victimLoanBefore = loanToken.balanceOf(address(V));

midnight.liquidate(market, collateralIndex, seizedAssets, 0, B, false, attacker, address(V), "");

assert(collateralToken.balanceOf(attacker) == attackerCollateralBefore + seizedAssets);
assert(loanToken.balanceOf(address(V)) == victimLoanBefore - repaidUnits);
```

Expected result: attacker's collateral balance increases by `seizedAssets`; `V`'s loan token balance decreases by `repaidUnits`. The attacker expends no loan tokens of their own.
