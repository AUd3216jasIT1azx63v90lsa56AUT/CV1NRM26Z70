### Title
Unconditional zero-value `safeTransferFrom` in `take()` DoS for tokens reverting on zero transfers - (File: src/Midnight.sol)

### Summary
`take()` unconditionally calls `SafeTransferLib.safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets)` at line 455, with no guard for the case where `buyerAssets == sellerAssets`. When the settlement fee is zero — a valid, explicitly supported protocol state — this difference is zero, and any loanToken that reverts on zero-amount `transferFrom` calls will cause every `take()` on that market to revert, permanently bricking the function for that market.

### Finding Description
**Code path:**

In `src/Midnight.sol`, `take()` computes: [1](#0-0) 

`buyerPrice = sellerPrice + _settlementFee` and `sellerPrice = buyerPrice - _settlementFee`, so `buyerAssets - sellerAssets` is exactly the settlement-fee spread. When `_settlementFee == 0`, `buyerPrice == sellerPrice`, and both `mulDivDown`/`mulDivUp` yield the same value, making `buyerAssets - sellerAssets == 0`.

The transfer is then issued unconditionally: [2](#0-1) 

`SafeTransferLib.safeTransferFrom` has no zero-value guard — it always issues the low-level call: [3](#0-2) 

If the token reverts on `transferFrom(payer, address(this), 0)`, the assembly block at lines 28–31 re-bubbles the revert, causing `take()` to revert.

**When is `_settlementFee == 0`?**

Post-maturity: `timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp) == 0`, so `_settlementFee = settlementFee(id, 0)` — the 0d breakpoint fee. This breakpoint is explicitly allowed to be 0 (max is ~0.14 bps, min is 0), and the protocol's own tests set it to 0 routinely. [4](#0-3) 

Pre-maturity: if all breakpoints are set to 0, `_settlementFee == 0` at any TTM.

**Additional rounding path (broader scope):** Even with `_settlementFee > 0`, if `units * _settlementFee < WAD`, integer division rounds `buyerAssets - sellerAssets` to 0, triggering the same zero-transfer for any small-unit take.

**Existing protections:** None. There is no `if (buyerAssets > sellerAssets)` guard before line 455, and `SafeTransferLib` has no zero-skip logic. The `CannotIncreaseDebtPostMaturity` check at line 391 only restricts debt increases post-maturity; it does not prevent credit-reducing takes (e.g., a lender reducing their position), which are valid post-maturity operations. [5](#0-4) 

**Attacker-controlled inputs:** Any unprivileged market creator can deploy a loanToken that reverts on zero-amount transfers and call `touchMarket` to create a market. Alternatively, if a legitimate market's loanToken has this property (e.g., LEND, some older ERC20s), any taker triggering a zero-fee take is the victim, not the attacker — the DoS is structural.

### Impact Explanation
All `take()` calls on any market whose loanToken reverts on zero-amount transfers are permanently bricked whenever `buyerAssets - sellerAssets == 0`. This includes: all post-maturity takes when the 0d settlement fee is 0, all takes at any TTM when all breakpoints are 0, and all small-unit takes where rounding collapses the spread to 0. Lenders cannot reduce credit, borrowers cannot reduce debt via take, and the market is effectively frozen for those conditions.

### Likelihood Explanation
The precondition — a loanToken that reverts on zero-amount `transferFrom` — is a known real-world behavior (e.g., the LEND token, some older ERC20 implementations). The protocol places no documented restriction on loanToken behavior regarding zero transfers. The 0d settlement fee being 0 is a valid and common configuration (the protocol's own test suite sets all fees to 0 in multiple tests). Post-maturity takes with zero 0d fee are a normal operational scenario. The combination is fully reachable without any privileged action.

### Recommendation
Add a zero-value guard before the settlement-fee transfer in `take()`:

```solidity
if (buyerAssets > sellerAssets) {
    SafeTransferLib.safeTransferFrom(
        offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets
    );
}
```

Similarly guard the `sellerAssets` transfer for tokens that also revert on zero `transfer`. This pattern is consistent with how `claimSettlementFee` already guards with `amount > 0` semantics implicitly via the subtraction underflow check.

```solidity
pragma solidity ^0.8.0;

import "forge-std/Test.sol";

// Token that reverts on zero-amount transferFrom
contract ZeroRevertToken is ERC20 {
    function transferFrom(address from, address to, uint256 amount) 
        public override returns (bool) 
    {
        require(amount > 0, "zero transfer");
        return super.transferFrom(from, to, amount);
    }
}

contract ZeroFeeDoSTest is MidnightBaseTest {
    function testZeroSettlementFeeDoS() public {
        // 1. Deploy ZeroRevertToken as loanToken
        ZeroRevertToken zrt = new ZeroRevertToken();
        
        // 2. Create market with zrt as loanToken, set ALL settlement fees to 0
        Market memory mkt = _buildMarket(address(zrt));
        for (uint256 i = 0; i <= 6; i++) {
            midnight.setDefaultSettlementFee(address(zrt), i, 0);
        }
        bytes32 id = midnight.touchMarket(mkt);
        
        // 3. Warp to maturity (timeToMaturity = 0, settlementFee = 0d breakpoint = 0)
        vm.warp(mkt.maturity);
        
        // 4. Set up a valid sell offer (borrower has credit, no debt increase post-maturity)
        //    Lender has credit from a prior take pre-maturity; now reduces it post-maturity
        //    (sellerDebtIncrease == 0 satisfied)
        Offer memory offer = _buildSellOffer(mkt, lender); // lender has credit
        
        // 5. Call take() — expect revert due to zero-amount transferFrom
        vm.prank(borrower);
        vm.expectRevert("zero transfer");
        midnight.take(offer, hex"", units, borrower, borrower, address(0), hex"");
        
        // Assert: take reverted even though all protocol invariants were satisfied
        // (valid offer, valid taker, valid units, zero fee is legitimate)
    }
}
```

**Expected assertion:** `take()` reverts with `"zero transfer"` (the token's revert reason), propagated through `SafeTransferLib`'s assembly re-bubble at `src/libraries/SafeTransferLib.sol:29-31`, despite the take being otherwise fully valid. The same test with a standard ERC20 (which allows zero transfers) passes without revert.
