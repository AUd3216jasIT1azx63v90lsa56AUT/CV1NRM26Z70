All three root causes are confirmed in the actual code. Let me verify the exploit path step by step:

**Root Cause 1 confirmed** — [1](#0-0)  The post-increment `nonce[authorization.authorizer]++` inside the `require` permanently advances the nonce before signature verification on lines 28–36. If the nonce check passes, the nonce is already N+1 regardless of whether the signature is valid.

**Exploit path is valid:**
- At Step 4, the delegate is still authorized (`isAuthorized[maker][delegate] == true`) because the maker's revocation tx has not yet landed. The delegate's call to `EcrecoverAuthorizer.setIsAuthorized` with nonce N succeeds, consuming nonce N and authorizing the attacker.

- At Step 6, the attacker (now authorized for maker) calls `setConsumed(group, type(uint256).max, maker)`, permanently setting `consumed[maker][group] = type(uint256).max`.

Audit Report

## Title
Malicious delegate front-runs maker's EcrecoverAuthorizer revocation to permanently grief offer groups - (File: `src/periphery/EcrecoverAuthorizer.sol`)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` increments the maker's nonce before verifying the signature, and accepts signatures from any existing delegate of the maker. A malicious delegate exploits this ordering to consume the maker's current nonce with a crafted authorization, permanently invalidating the maker's pending revocation. The delegate then uses the newly authorized attacker address to call `Midnight.setConsumed(group, type(uint256).max, maker)`, irreversibly cancelling all of the maker's offers in that group.

## Finding Description

**Root cause 1 — Nonce consumed before signature verification:**

In `EcrecoverAuthorizer.setIsAuthorized`, line 26 increments `nonce[authorization.authorizer]` atomically before the signature is checked:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

Once any transaction consuming nonce N lands, the maker's signed message using nonce N is permanently invalid — even if the consuming transaction was submitted by a delegate, not the maker.

**Root cause 2 — Delegates can sign authorizations on behalf of the maker:**

Lines 33–36 accept a signature from any address that `IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)` returns true for:

```solidity
require(
    signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

A delegate can construct and sign an `Authorization` struct with `authorization.authorizer = maker` and arbitrary `authorized`/`isAuthorized` values, and the contract will accept it as if the maker signed it.

**Root cause 3 — `setConsumed` with `type(uint256).max` is irreversible:**

`setConsumed` enforces strict monotonicity via `AlreadyConsumed`, and `consumed` can never decrease. Setting it to `type(uint256).max` permanently blocks every subsequent `take` on any offer in that group.

**Full exploit path:**

| Step | Actor | Action | Effect |
|------|-------|--------|--------|
| 0 | maker | `Midnight.setIsAuthorized(ecrecoverAuthorizer, true, maker)` | EcrecoverAuthorizer becomes a delegate |
| 1 | maker | `Midnight.setIsAuthorized(delegate, true, maker)` | delegate is authorized |
| 2 | maker | Signs `Authorization{maker, delegate, false, N, deadline}`, broadcasts | Pending in mempool |
| 3 | delegate | Signs `Authorization{maker, attacker, true, N, deadline2}` | Off-chain |
| 4 | delegate | `EcrecoverAuthorizer.setIsAuthorized(auth3, sig3)` (front-run) | `nonce[maker]→N+1`, `isAuthorized[maker][attacker]=true` |
| 5 | maker | Maker's tx lands | Reverts `InvalidNonce` |
| 6 | attacker | `Midnight.setConsumed(group, type(uint256).max, maker)` | `consumed[maker][group]=type(uint256).max` — permanent |

At Step 4, `isAuthorized[maker][delegate]` is still `true` (the revocation has not landed), so the delegate's signature passes the authorization check. The maker can subsequently call `Midnight.setIsAuthorized(delegate, false, maker)` directly, but `consumed[maker][group]` is already `type(uint256).max` and cannot be lowered.

## Impact Explanation

All of the maker's offers sharing the griefed `group` are permanently cancelled — every `take` on them reverts with `ConsumedUnits` or `ConsumedAssets`. The maker cannot recover the group budget because `setConsumed` enforces strict monotonicity. The maker must redeploy offers under a new group identifier, losing any off-chain routing or aggregator integrations that referenced the old group. This constitutes permanent, irreversible corruption of user state — a concrete in-scope impact class per RESEARCHER.md ("Permanent lock, freeze, or unrecoverable corruption of user/project state").

## Likelihood Explanation

**Preconditions:**
1. Maker has opted into `EcrecoverAuthorizer` — required for any signature-based authorization flow, common in production setups.
2. Maker has at least one active delegate — standard for automated market-making.
3. Maker attempts to revoke the delegate via a signed message rather than a direct `Midnight.setIsAuthorized` call — plausible when the maker is a smart contract or uses a meta-transaction relay.

The front-run requires no special knowledge beyond the delegate's own private key and the current nonce (publicly readable). The attack is repeatable — every time the maker tries to revoke via `EcrecoverAuthorizer`, the delegate can repeat the nonce-burn. Only one successful front-run is needed to cause permanent damage via `setConsumed`.

## Recommendation

1. **Move nonce increment after signature verification.** Store the current nonce in a local variable, verify the signature first, then increment:
   ```solidity
   uint256 currentNonce = nonce[authorization.authorizer];
   require(authorization.nonce == currentNonce, InvalidNonce());
   // ... verify signature ...
   nonce[authorization.authorizer] = currentNonce + 1;
   ```
   This ensures a failed signature check does not consume the nonce.

2. **Restrict who can sign on behalf of the authorizer.** Consider requiring that only `authorization.authorizer` itself (not delegates) can sign an `Authorization` struct. Delegates can already call `Midnight.setIsAuthorized` directly; the EcrecoverAuthorizer's purpose is to allow the authorizer to act via signed messages without gas, not to allow delegates to issue signed authorizations on the authorizer's behalf.

3. **Alternatively, scope delegate signing to non-revocation actions.** If delegate signing is intentional, at minimum prevent delegates from signing authorizations that would authorize new third parties, limiting them to the specific `authorized` addresses they are already permitted to manage.

## Proof of Concept

**Minimal manual steps:**
1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. As `maker`: call `Midnight.setIsAuthorized(ecrecoverAuthorizer, true, maker)` and `Midnight.setIsAuthorized(delegate, true, maker)`.
3. As `maker`: sign `Authorization{authorizer: maker, authorized: delegate, isAuthorized: false, nonce: 0, deadline: far_future}` and submit to a public mempool (or simulate pending tx).
4. As `delegate`: sign `Authorization{authorizer: maker, authorized: attacker, isAuthorized: true, nonce: 0, deadline: far_future}` and call `EcrecoverAuthorizer.setIsAuthorized` with higher gas price.
5. Observe: `nonce[maker] == 1`, `isAuthorized[maker][attacker] == true`, maker's tx reverts with `InvalidNonce`.
6. As `attacker`: call `Midnight.setConsumed(group, type(uint256).max, maker)`.
7. Observe: `consumed[maker][group] == type(uint256).max`. Any subsequent `take` on maker's offers in `group` reverts with `ConsumedUnits`/`ConsumedAssets`.

**Forge test outline:**
```solidity
function testDelegateFrontRunsRevocation() public {
    vm.prank(maker); midnight.setIsAuthorized(address(ecrecoverAuth), true, maker);
    vm.prank(maker); midnight.setIsAuthorized(delegate, true, maker);

    // delegate front-runs with nonce 0, authorizing attacker
    Authorization memory auth = Authorization(maker, attacker, true, 0, block.timestamp + 1 days);
    Signature memory sig = _sign(delegateKey, auth);
    ecrecoverAuth.setIsAuthorized(auth, sig);

    // maker's revocation with nonce 0 now reverts
    Authorization memory revokeAuth = Authorization(maker, delegate, false, 0, block.timestamp + 1 days);
    Signature memory revokeSig = _sign(makerKey, revokeAuth);
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuth.setIsAuthorized(revokeAuth, revokeSig);

    // attacker permanently griefs the group
    vm.prank(attacker);
    midnight.setConsumed(group, type(uint256).max, maker);
    assertEq(midnight.consumed(maker, group), type(uint256).max);
}
```
