Audit Report

## Title
Missing `caller` Parameter in `IRepayCallback.onRepay()` Enables Unauthorized Token Drain from Callback Contracts - (File: src/Midnight.sol)

## Summary
The `repay()` function accepts an arbitrary `callback` address and sets it as the token `payer` without requiring `callback == msg.sender`. Because `IRepayCallback.onRepay()` omits the `address caller` parameter that both `ILiquidateCallback.onLiquidate` and `IFlashLoanCallback.onFlashLoan` include, any contract implementing `onRepay` cannot identify who initiated the repay call. An unprivileged attacker with any debt position can drain loan tokens from any contract that implements `onRepay` and holds a Midnight approval.

## Finding Description
In `src/Midnight.sol`, `repay()` performs the following sequence:

1. **Line 505**: Checks only `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` — no constraint that `callback == msg.sender` or that `callback` is related to the caller in any way. [1](#0-0) 

2. **Line 511**: Sets `payer = callback` when `callback != address(0)`. [2](#0-1) 

3. **Lines 514–518**: Calls `IRepayCallback(callback).onRepay(id, market, units, onBehalf, data)` — without forwarding `msg.sender`. [3](#0-2) 

4. **Line 520**: Executes `safeTransferFrom(loanToken, payer, address(this), units)`, pulling tokens from `callback`. [4](#0-3) 

The `IRepayCallback` interface omits `address caller`: [5](#0-4) 

While both `ILiquidateCallback.onLiquidate` and `IFlashLoanCallback.onFlashLoan` include it: [6](#0-5) 

Inside `onRepay`, `msg.sender` is always `address(Midnight)` — the callback contract has no way to identify the initiating user. The reference test implementation `RepayCallback` confirms this: it checks only that the market ID is well-formed, then returns `CALLBACK_SUCCESS` unconditionally, with no check on who called `repay()`. [7](#0-6) 

**Exploit flow:**
1. Attacker acquires any debt position in any market (borrows via `take`), so `onBehalf == msg.sender` is trivially satisfied.
2. Attacker calls `repay(market, units, attacker, victimContract, data)`. Authorization passes: `onBehalf == attacker == msg.sender`.
3. Midnight calls `victimContract.onRepay(id, market, units, attacker, data)`. Inside `onRepay`, `msg.sender == Midnight`. The victim cannot identify the attacker.
4. `victimContract` returns `CALLBACK_SUCCESS` (as any compliant implementation must, having no way to reject based on initiator).
5. Midnight executes `safeTransferFrom(loanToken, victimContract, Midnight, units)`. Attacker's debt is repaid using the victim's tokens.

**Why the formal model does not catch this:** `OnlyExplicitPayerCanLoseTokens.spec` treats `CALLBACK_SUCCESS` as informed consent — it proves tokens are only pulled from a callback that returned `CALLBACK_SUCCESS`. This is correct at the protocol level but the model assumption breaks down when the callback structurally cannot determine who triggered the call. [8](#0-7) [9](#0-8) 

## Impact Explanation
Any contract that (a) implements `IRepayCallback.onRepay()` returning `CALLBACK_SUCCESS` and (b) holds a Midnight approval for loan tokens can have its entire approved balance drained by an unprivileged attacker. This is direct, concrete theft of assets from third-party integration contracts (flash-repay routers, bundlers, aggregators) — the intended integration targets of the repay callback feature. The victim's token balance decreases to repay a debt it did not choose to repay, violating the protocol's authorization model.

## Likelihood Explanation
Preconditions are minimal: the attacker only needs an existing debt position in any market, obtainable by borrowing with no cooperation from any other party. Contracts implementing `IRepayCallback` and holding Midnight approval are the intended integration targets of this feature. The attack is repeatable as long as the victim retains Midnight approval and loan token balance. The missing `caller` parameter makes it structurally impossible for callback implementors to defend using only the provided interface — unlike `onLiquidate` and `onFlashLoan`, which both supply `caller` and allow implementors to enforce `require(caller == address(this))` or equivalent.

## Recommendation
Add `address caller` (the `msg.sender` of `repay()`) as the first parameter of `IRepayCallback.onRepay()`, mirroring the pattern already used in `ILiquidateCallback.onLiquidate` and `IFlashLoanCallback.onFlashLoan`. Update `repay()` to pass `msg.sender` as the first argument when invoking the callback. This allows callback implementors to enforce `require(caller == address(this))` or any equivalent access-control check, making uninvited repay-callback invocations revert.

## Proof of Concept
Deploy a `VictimRepayCallback` contract that:
- Implements `onRepay` returning `CALLBACK_SUCCESS` (the only compliant behavior given the interface).
- Holds loan tokens and has approved Midnight.

Deploy an `Attacker` contract that:
1. Borrows `units` from any market (so `position[id][attacker].debt >= units`).
2. Calls `midnight.repay(market, units, attacker, address(victimRepayCallback), "")`.

Assert after the call:
- `loanToken.balanceOf(victimRepayCallback)` decreased by `units`.
- `midnight.debtOf(id, attacker)` decreased by `units`.
- `loanToken.balanceOf(attacker)` is unchanged (attacker paid nothing).

This directly demonstrates that the attacker's debt was repaid using the victim's tokens without the victim's informed consent.
