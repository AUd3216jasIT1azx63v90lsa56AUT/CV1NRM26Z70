### Title
Double `mulDivDown` in `maxDebt` Computation Allows Liquidation of Truly Healthy Positions - (File: src/Midnight.sol)

### Summary

The `maxDebt` computation in both `liquidate()` and `isHealthy()` applies `mulDivDown` twice in sequence — first dividing by `ORACLE_PRICE_SCALE`, then by `WAD`. Each truncation discards a fractional remainder, and when both remainders are large enough, their combined loss exceeds 1 unit, causing `computed_maxDebt` to be exactly 1 less than `floor(true_maxDebt)`. A position where `debt == floor(true_maxDebt)` is mathematically healthy (true maxDebt ≥ debt), yet the protocol computes `maxDebt = debt − 1`, satisfying `originalDebt > maxDebt` and allowing an unprivileged liquidator to seize collateral.

### Finding Description

**Code path:**

`liquidate()` at `src/Midnight.sol` line 613: [1](#0-0) 

```solidity
maxDebt += _collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(_collateralParam.lltv, WAD);
```

`isHealthy()` at `src/Midnight.sol` lines 954–955: [2](#0-1) 

`mulDivDown` is plain integer division: [3](#0-2) 

```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
```

**Root cause — double truncation:**

Let:
- `r1 = (collateral × price) mod ORACLE_PRICE_SCALE`
- `step1 = ⌊collateral × price / ORACLE_PRICE_SCALE⌋`
- `r2 = (step1 × lltv) mod WAD`

Then:

```
computed_maxDebt = (step1 × lltv − r2) / WAD
true_maxDebt     = computed_maxDebt + r2/WAD + r1×lltv/(ORACLE_PRICE_SCALE×WAD)
```

The discrepancy `r2/WAD + r1×lltv/(ORACLE_PRICE_SCALE×WAD)` reaches ≥ 1 when:

```
r2 × ORACLE_PRICE_SCALE + r1 × lltv  ≥  ORACLE_PRICE_SCALE × WAD
```

This is achievable with allowed LLTV tiers. Concrete example using `LLTV_0 = 0.385e18`:

| Parameter | Value |
|---|---|
| `collateral` | `187020000000000000000` (≈187.02 tokens, 18 dec) |
| `price` | `1000000000000000000` (= 10^18) |
| `lltv` | `385000000000000000` (= 0.385e18, allowed tier) |
| `debt` | `72` |

Step-by-step:
- `collateral × price = 187020000000000000000 × 10^18 = 187.02 × 10^36`
- `step1 = ⌊187.02⌋ = 187`, `r1 = 0.02 × 10^36 = 2×10^34`
- `step1 × lltv = 187 × 385×10^15 = 71995×10^15`
- `r2 = 71995×10^15 mod 10^18 = 995×10^15`
- `computed_maxDebt = ⌊71995×10^15 / 10^18⌋ = 71`
- `true_maxDebt = 187.02 × 10^36 × 385×10^15 / 10^54 = 72.0027`

Position with `debt = 72` is **truly healthy** (72.0027 ≥ 72), but `computed_maxDebt = 71 < 72`.

**Liquidatability check** at line 622: [4](#0-3) 

```solidity
require(
    !liquidationLocked(id, borrower)
        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
    NotLiquidatable()
);
```

`originalDebt (72) > maxDebt (71)` → `true` → `NotLiquidatable` is **not** thrown → liquidation proceeds.

**Why Certora specs do not catch this:**

The `Healthiness.spec` replaces `mulDivDown` with a ghost summary `summaryMulDivDown` axiomatized only for monotonicity and zero-input behavior: [5](#0-4) 

These axioms do not model the concrete truncation behavior of the double `mulDivDown`, so the formal verification cannot detect the 1-unit underestimation.

### Impact Explanation

An unprivileged liquidator can call `liquidate()` on a position that is mathematically healthy (true collateral value ≥ debt), seizing the borrower's collateral and forcing repayment of debt the borrower was not obligated to repay. This is unauthorized collateral seizure from a solvent borrower.

### Likelihood Explanation

The condition `r2 × ORACLE_PRICE_SCALE + r1 × lltv ≥ ORACLE_PRICE_SCALE × WAD` is satisfied for a dense set of `(collateral, price)` pairs at every allowed LLTV tier. Oracle prices are continuous and change over time; for any given borrower position, there will exist price values at which the rounding boundary is crossed. A liquidation bot monitoring for these conditions can exploit them systematically. The precondition requires no privileged access, no oracle manipulation, and no user error — only a specific oracle price value that the market reaches naturally.

### Recommendation

Replace the chained double `mulDivDown` with a single full-precision computation that avoids intermediate truncation. The cleanest fix is a 512-bit `mulDiv` (as used in Uniswap v3's `FullMath`):

```solidity
// In both liquidate() and isHealthy():
maxDebt += FullMath.mulDiv(
    _collateral * price,   // or use mulDiv for the outer product too if overflow is possible
    _collateralParam.lltv,
    ORACLE_PRICE_SCALE * WAD
);
```

If 512-bit arithmetic is undesirable, the minimum safe fix is to use `mulDivUp` for the first step, giving the borrower the benefit of the doubt on the first rounding:

```solidity
maxDebt += _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivDown(_collateralParam.lltv, WAD);
```

This ensures `computed_maxDebt ≥ floor(true_maxDebt)`, so a truly healthy position is never liquidatable due to rounding. The same fix must be applied consistently in both `liquidate()` and `isHealthy()`. [1](#0-0) [2](#0-1) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IMidnight, Market, CollateralParams} from "src/interfaces/IMidnight.sol";
import {ORACLE_PRICE_SCALE, WAD, LLTV_0} from "src/libraries/ConstantsLib.sol";

contract RoundingLiquidationTest is Test {
    Midnight midnight;
    address borrower = address(0xB0);
    address liquidator = address(0xL1);

    function setUp() public {
        midnight = new Midnight();
        // deploy loan token, collateral token, oracle — standard test setup
    }

    function testHealthyPositionLiquidatedDueToRounding() public {
        // Parameters chosen so double-mulDivDown underestimates maxDebt by 1
        uint256 collateralAmount = 187_020000000000000000; // 187.02e18
        uint256 oraclePrice      = 1e18;                   // price = 1e18
        uint256 lltv             = LLTV_0;                 // 0.385e18 (allowed tier)
        uint256 debtUnits        = 72;                     // floor(true_maxDebt) = 72

        // Step 1: borrower supplies collateral
        vm.prank(borrower);
        midnight.supplyCollateral(market, 0, collateralAmount, borrower, "");

        // Step 2: borrower borrows 72 units (at a price where computed maxDebt >= 72)
        // (use a slightly higher price for the borrow, then drop to boundary)
        // ... take() call omitted for brevity ...

        // Step 3: oracle price set to boundary value
        oracle.setPrice(oraclePrice);

        // Verify: true maxDebt = 72.0027 >= 72 → position IS healthy
        // computed maxDebt = 71 < 72 → isHealthy() returns false
        assertFalse(midnight.isHealthy(market, id, borrower), "protocol wrongly reports unhealthy");

        // Step 4: liquidator liquidates a truly healthy position
        vm.prank(liquidator);
        // Assert: this should revert with NotLiquidatable, but it does NOT
        midnight.liquidate(market, 0, 0, 1, borrower, false, liquidator, address(0), "");

        // Invariant violated: borrower lost collateral despite being healthy
        // Assert debt decreased (liquidation succeeded)
        assertLt(midnight.debtOf(id, borrower), debtUnits, "liquidation succeeded on healthy position");
    }

    // Fuzz variant: find all (collateral, price, lltv) triples where rounding causes false unhealthy
    function testFuzz_roundingCausesSpuriousLiquidation(
        uint128 collateral,
        uint128 price,
        uint8 lltvIndex
    ) public {
        uint256[9] memory lltvsAllowed = [
            0.385e18, 0.625e18, 0.77e18, 0.86e18,
            0.915e18, 0.945e18, 0.965e18, 0.98e18, 1e18
        ];
        uint256 lltv = lltvsAllowed[lltvIndex % 9];
        uint256 step1 = uint256(collateral) * uint256(price) / ORACLE_PRICE_SCALE;
        uint256 computedMaxDebt = step1 * lltv / WAD;
        // true maxDebt (using 512-bit math)
        uint256 trueMaxDebt = FullMath.mulDiv(uint256(collateral) * uint256(price), lltv, ORACLE_PRICE_SCALE * WAD);
        if (trueMaxDebt > computedMaxDebt) {
            // Rounding discrepancy found: set debt = trueMaxDebt, assert liquidate reverts
            // ... setup position with debt = trueMaxDebt ...
            vm.expectRevert(IMidnight.NotLiquidatable.selector);
            midnight.liquidate(market, 0, 0, 1, borrower, false, liquidator, address(0), "");
        }
    }
}
```

**Expected assertion:** `liquidate()` must revert with `NotLiquidatable` when `true_maxDebt ≥ debt`. The test demonstrates it does not, confirming the invariant "healthy positions are not liquidatable" is violated.
