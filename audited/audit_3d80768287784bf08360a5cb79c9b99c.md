### Title
Fee-on-Transfer Collateral Token Inflates `collateral[i]` Causing Undercollateralized Borrowing - (File: src/Midnight.sol)

### Summary
`supplyCollateral` records the caller-supplied `assets` value directly into `_position.collateral[collateralIndex]` before executing the token transfer. When the collateral token charges a fee on transfer, the contract receives fewer tokens than recorded, permanently overstating the borrower's collateral. `isHealthy` then computes `maxDebt` from the inflated on-chain value, allowing the borrower to take on more debt than the actual collateral can support.

### Finding Description
**Root cause — `src/Midnight.sol` lines 533 and 545:**

```solidity
// Line 533 — state written with caller-supplied `assets`
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
// ...
// Line 545 — transfer executes AFTER state is committed
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
``` [1](#0-0) 

The accounting uses the input parameter `assets`, not the actual balance delta. No balance-before/after check exists anywhere in `supplyCollateral`.

**`isHealthy` — `src/Midnight.sol` lines 944–959:**

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
``` [2](#0-1) 

`isHealthy` reads `_position.collateral[i]` — the inflated recorded value — to compute the borrower's maximum allowed debt. It has no knowledge of the actual token balance held by the contract.

**Market creation — `touchMarket` (`src/Midnight.sol` lines 755–791):**

`touchMarket` validates only LLTV tier, `maxLif`, and sorted collateral addresses. There is no check that the collateral token is not a fee-on-transfer token. [3](#0-2) 

**`IERC20` interface — `src/interfaces/IERC20.sol` lines 1–11:**

The interface is a minimal standard ERC20 with no mechanism to detect or reject fee-on-transfer behavior. [4](#0-3) 

**Exploit flow:**

1. Attacker deploys a fee-on-transfer ERC20 (1% fee) and creates a permissionless market with it as collateral.
2. Attacker calls `supplyCollateral(market, 0, 100e18, attacker)`.
   - `_position.collateral[0]` is set to `100e18`.
   - Contract actually receives `99e18` tokens.
3. Attacker calls `take(...)` as seller (borrower), borrowing at LLTV based on `100e18` collateral.
   - `isHealthy` computes `maxDebt = 100e18 * price / ORACLE_PRICE_SCALE * lltv / WAD`.
   - Actual backing is only `99e18` tokens.
4. Position is immediately undercollateralized by the fee percentage. The attacker has extracted `lltv * fee_amount` in excess loan assets.
5. Repeated calls amplify the discrepancy. On `withdrawCollateral`, the contract attempts to transfer the recorded amount but may lack the balance, causing a revert or draining other users' collateral.

**Existing checks and why they fail:**

- The `isHealthy` check inside `withdrawCollateral` (line 568) uses the same inflated `_position.collateral[i]`, so it does not catch the discrepancy.
- The Certora `supplyCollateralEffects` rule (line 217 of `certora/specs/BalanceEffects.spec`) asserts `collateral == collateralBefore + assets`, which is exactly the buggy behavior — it models the input, not the actual received amount. [5](#0-4) 

### Impact Explanation
An attacker can borrow against overstated collateral, creating an immediately undercollateralized position. The excess debt is unrecoverable bad debt socialized to lenders. On `withdrawCollateral`, the contract may attempt to transfer more tokens than it holds, reverting or draining collateral belonging to other borrowers in the same market. This directly violates the core invariant: "contract balances cover collateral."

### Likelihood Explanation
The protocol is fully permissionless — any address can create a market with any ERC20 as collateral token. Fee-on-transfer tokens (e.g., tokens with deflationary mechanics, tokens with protocol fees) are a well-known ERC20 variant. No precondition requires admin action. The attack is repeatable on every `supplyCollateral` call and scales linearly with the fee rate and supplied amount.

### Recommendation
Replace the input-based accounting with a balance-delta measurement in `supplyCollateral`:

```solidity
function supplyCollateral(...) external {
    // ...
    uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
    SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
    uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
    _position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
    // ...
}
```

Alternatively, document that fee-on-transfer tokens are unsupported and add a validation hook in `touchMarket` that detects non-standard transfer behavior, or enforce a token allowlist.

### Proof of Concept
**Foundry unit test plan:**

```solidity
contract FeeOnTransferERC20 is ERC20 {
    uint256 public constant FEE_BPS = 100; // 1%
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount * FEE_BPS / 10000;
        super.transferFrom(from, to, amount - fee);
        // fee stays with `from` (burned or sent to fee collector)
        return true;
    }
}

function testFeeOnTransferCollateralInflatesAccounting() public {
    // 1. Deploy fee-on-transfer token and create market
    FeeOnTransferERC20 feeToken = new FeeOnTransferERC20();
    Market memory m = buildMarket(address(feeToken), lltv=0.77e18);
    midnight.touchMarket(m);

    // 2. Supply 100e18 collateral
    uint256 supplied = 100e18;
    feeToken.mint(attacker, supplied);
    vm.prank(attacker);
    feeToken.approve(address(midnight), supplied);
    vm.prank(attacker);
    midnight.supplyCollateral(m, 0, supplied, attacker);

    // 3. Assert: recorded collateral > actual balance increase
    uint256 recorded = midnight.collateral(id, attacker, 0);
    assertEq(recorded, 100e18);                          // inflated
    assertEq(feeToken.balanceOf(address(midnight)), 99e18); // actual
    assertGt(recorded, feeToken.balanceOf(address(midnight)), "collateral overstated");

    // 4. Assert: isHealthy returns true for a borrow sized to 100e18 collateral
    //    but position is actually undercollateralized relative to 99e18
    assertTrue(midnight.isHealthy(m, id, attacker));
    // maxDebt computed from 100e18 * price * lltv > actual backing
}
```

**Expected assertions:** `recorded > actual_balance` and `isHealthy == true` despite the position being undercollateralized by the fee amount.
