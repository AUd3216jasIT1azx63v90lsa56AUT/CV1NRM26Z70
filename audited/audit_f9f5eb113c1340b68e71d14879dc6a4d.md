### Title
Lender Can Withdraw Full `credit` Without Accounting for `pendingFee`, Stealing Continuous Fee Revenue — (File: `src/Midnight.sol`)

---

### Summary

The `withdraw` function in `Midnight.sol` allows a lender to withdraw up to their full `credit` balance without enforcing that the withdrawal is bounded by `credit - pendingFee`. The `pendingFee` is the lender's future continuous-fee obligation — the "locked" portion of credit that should accrue to the protocol over time. By withdrawing their full `credit`, a lender zeroes out their `pendingFee` without it ever being transferred to `continuousFeeCredit`, causing the protocol to lose that fee revenue entirely.

---

### Finding Description

**Root cause — the missing bound in `withdraw`:**

The `withdraw` function calls `_updatePosition` first (which accrues the fee from `lastAccrual` to `now`, reducing both `credit` and `pendingFee` by the elapsed portion). After that call, the position holds:

- `credit` = gross credit remaining
- `pendingFee` = future fee obligation from now until maturity

The protocol's own invariant, stated in the contract comments, is:

> "In the absence of bad debt realizations, the face value of a lender's position is `credit - pendingFee`." [1](#0-0) 

Yet the `withdraw` function imposes no such bound. It only reverts if `units > credit` (Solidity 0.8 underflow) or `units > withdrawable`:

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    ...
    _updatePosition(market, id, onBehalf);

    Position storage _position = position[id][onBehalf];
    uint128 pendingFeeDecrease;
    if (_position.credit > 0) {
        pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
        _position.pendingFee -= pendingFeeDecrease;
    }
    _position.credit -= UtilsLib.toUint128(units);          // only reverts if units > credit
    _marketState.withdrawable -= UtilsLib.toUint128(units); // only reverts if units > withdrawable
    ...
    SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
}
``` [2](#0-1) 

There is no `require(units + _position.pendingFee <= _position.credit)` guard.

**What happens when `units = credit`:**

`pendingFeeDecrease = pendingFee.mulDivUp(credit, credit) = pendingFee` — the entire future fee obligation is silently zeroed out. It is never routed to `continuousFeeCredit`. The lender receives `credit` tokens but should only receive `credit - pendingFee` tokens.

Compare with `_updatePosition`, which correctly routes the accrued portion to `continuousFeeCredit`:

```solidity
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
``` [3](#0-2) 

The `withdraw` path has no equivalent routing for the cancelled `pendingFee`.

**How `pendingFee` is set (the "locked" amount):**

When a lender acquires credit via `take()`, their `pendingFee` is set proportional to `continuousFee * timeToMaturity`:

```solidity
uint128 buyerPendingFeeIncrease =
    UtilsLib.toUint128(buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD));
``` [4](#0-3) 

For a market with a meaningful continuous fee and long time to maturity, `pendingFee` can be a substantial fraction of `credit`.

---

### Impact Explanation

A lender can call `withdraw(units = credit)` whenever `withdrawable >= credit` (e.g., after borrowers repay). The lender receives `credit` tokens instead of the correct `credit - pendingFee` tokens. The difference — `pendingFee` — is fee revenue that should flow to the protocol's `continuousFeeCredit` but instead goes to the lender. This is a direct theft of protocol fee revenue, proportional to the lender's `pendingFee` balance. For markets with high continuous fees or long maturities, this can be significant.

---

### Likelihood Explanation

Any lender can trigger this with no privileged access. The only precondition is that `withdrawable >= credit`, which is a normal market state after borrowers repay. The attacker simply calls `withdraw` with `units = position.credit` before maturity. No flash loan or oracle manipulation is required.

---

### Recommendation

Add a check in `withdraw` that bounds `units` to the lender's net credit (gross credit minus pending fee):

```solidity
require(
    units + _position.pendingFee <= _position.credit,
    "Withdrawal exceeds net credit after pending fee"
);
```

This mirrors the fix in the referenced RPL report: subtract the "locked" amount (`pendingFee`) from the total balance (`credit`) before allowing withdrawal.

Alternatively, when a lender withdraws, route the proportional cancelled `pendingFee` to `continuousFeeCredit` rather than zeroing it out silently, so the protocol still collects the fee.

---

### Proof of Concept

1. Market is created with `continuousFee > 0` and maturity 1 year away.
2. Lender calls `take()` acquiring `credit = 1000e18`. Their `pendingFee` is set to `continuousFee * timeToMaturity * credit / WAD`, e.g. `50e18` (5%).
3. Borrower repays their debt; `withdrawable` increases to ≥ `1000e18`.
4. Lender calls `withdraw(units = 1000e18)`.
   - `_updatePosition` accrues a small elapsed fee (say `1e18`), reducing `credit` to `999e18` and `pendingFee` to `49e18`.
   - `pendingFeeDecrease = 49e18.mulDivUp(999e18, 999e18) = 49e18` — all remaining `pendingFee` is zeroed.
   - Lender receives `999e18` tokens.
   - `continuousFeeCredit` receives `0` for the cancelled `49e18` future fee.
5. Protocol loses `49e18` in fee revenue. Lender extracted `49e18` more than their net position entitled them to. [5](#0-4)
