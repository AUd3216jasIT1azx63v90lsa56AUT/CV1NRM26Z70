### Title
Fee-on-Transfer Loan Token Causes `withdrawable` Overstatement in `liquidate`, Leading to Protocol Insolvency - (`src/Midnight.sol`)

### Summary
In `liquidate()`, `_marketState.withdrawable` is incremented by `repaidUnits` before the actual `safeTransferFrom` call. `SafeTransferLib.safeTransferFrom` only verifies the call did not revert and returned `true`; it does not measure the actual balance delta. When `market.loanToken` is a fee-on-transfer token — which `touchMarket` does not prohibit — the contract receives `repaidUnits - fee` but records `repaidUnits` as withdrawable, permanently overstating the lender-redeemable pool.

### Finding Description

**Exact code path:**

`src/Midnight.sol` `liquidate()`: [1](#0-0) 

```solidity
_marketState.withdrawable += UtilsLib.toUint128(repaidUnits);   // line 675
_position.debt -= UtilsLib.toUint128(repaidUnits);              // line 676
```

Then, after the collateral transfer and optional callback: [2](#0-1) 

```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```

`SafeTransferLib.safeTransferFrom` only checks call success and boolean return value — it does not compare pre/post balances: [3](#0-2) 

`touchMarket` validates LLTV tiers, collateral sort order, and maturity bounds, but performs **no validation on `market.loanToken`** — any ERC20 address, including a fee-on-transfer token, is accepted: [4](#0-3) 

**Exploit flow:**

1. Attacker (market creator, unprivileged) calls `touchMarket(market)` where `market.loanToken` is a fee-on-transfer ERC20 (e.g., 1% fee on every transfer).
2. A borrower supplies collateral and borrows via `take`. The same fee-on-transfer issue affects the supply side, but the attacker's goal is the liquidation path.
3. The position becomes unhealthy (oracle price drop or post-maturity).
4. Liquidator calls `liquidate(..., repaidUnits = X, ...)`.
5. At line 675: `withdrawable += X`.
6. At line 717: `safeTransferFrom(loanToken, payer, address(this), X)` — only `X - fee` arrives.
7. `withdrawable` is now overstated by `fee` per liquidation call.
8. Repeated liquidations accumulate the deficit. Lenders calling `withdraw()` drain the real balance before all credits are redeemed.

**Why existing checks fail:**

- `safeTransferFrom` in `SafeTransferLib` is not balance-aware; it trusts the return value of `transferFrom`, which fee-on-transfer tokens return as `true`.
- No pre/post balance snapshot exists anywhere in `liquidate`.
- `touchMarket` imposes no restriction on the loan token type.

### Impact Explanation

`marketState[id].withdrawable` exceeds the actual loan token balance held by the contract. Lenders who call `withdraw()` — which decrements `withdrawable` and calls `safeTransfer(loanToken, receiver, units)` — will cause the last withdrawers to receive a revert (insufficient balance) or receive fewer tokens than their credit entitles them to, constituting direct, irreversible loss of lender funds and protocol insolvency for that market. [5](#0-4) 

### Likelihood Explanation

**Preconditions:**
- A market must be created with a fee-on-transfer loan token. `touchMarket` is permissionless; any address can do this.
- At least one borrower must take a position and become liquidatable.
- A liquidator must call `liquidate` with `repaidUnits > 0`.

All three preconditions are reachable by unprivileged actors. Fee-on-transfer tokens (e.g., tokens with deflationary mechanics or protocol fees) exist in production. The bug is repeatable: every liquidation with `repaidUnits > 0` compounds the deficit. No special timing, oracle manipulation, or admin action is required.

### Recommendation

Replace the fixed-amount accounting with a balance-delta pattern in `liquidate` (and symmetrically in `repay`):

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
uint256 actualReceived = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
// Use actualReceived instead of repaidUnits for withdrawable and debt updates
_marketState.withdrawable += UtilsLib.toUint128(actualReceived);
_position.debt -= UtilsLib.toUint128(actualReceived);
```

Alternatively, explicitly disallow fee-on-transfer tokens by asserting `actualReceived == repaidUnits` and reverting otherwise, which preserves the existing accounting model while rejecting incompatible tokens.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {Market, CollateralParams} from "src/interfaces/IMidnight.sol";
import {IERC20} from "src/interfaces/IERC20.sol";

/// @dev Minimal fee-on-transfer ERC20: deducts 1% on every transferFrom.
contract FeeOnTransferToken { /* standard ERC20 + 1% fee in transferFrom */ }

contract FeeOnTransferLiquidateTest is Test {
    Midnight midnight;
    FeeOnTransferToken loanToken;
    ERC20 collateralToken;
    Oracle oracle;
    Market market;

    function setUp() public {
        // Deploy protocol and fee-on-transfer loan token
        // Create market with loanToken = address(feeOnTransferToken)
        // touchMarket succeeds — no validation on loanToken
    }

    function testWithdrawableOverstatedAfterLiquidation() public {
        uint256 units = 1e18;
        // 1. Lender supplies, borrower borrows `units`
        // 2. Drop oracle price to make position unhealthy
        // 3. Record contract balance before liquidation
        uint256 balanceBefore = loanToken.balanceOf(address(midnight));
        uint256 withdrawableBefore = midnight.withdrawable(id);

        // 4. Liquidate with repaidUnits = units
        midnight.liquidate(market, 0, 0, units, borrower, false, address(this), address(0), "");

        uint256 balanceAfter = loanToken.balanceOf(address(midnight));
        uint256 withdrawableAfter = midnight.withdrawable(id);

        uint256 actualReceived = balanceAfter - balanceBefore;
        uint256 withdrawableIncrease = withdrawableAfter - withdrawableBefore;

        // KEY ASSERTIONS:
        // withdrawable increased by repaidUnits, but contract only received repaidUnits - fee
        assertGt(withdrawableIncrease, actualReceived, "withdrawable overstated");
        // Deficit equals the fee taken
        assertEq(withdrawableIncrease - actualReceived, units / 100, "deficit equals 1% fee");

        // 5. Lender tries to withdraw full credit — last lender reverts (insufficient balance)
        vm.prank(lender);
        vm.expectRevert(); // contract balance < withdrawable
        midnight.withdraw(market, units, lender, lender);
    }
}
```

**Expected assertions:** `withdrawableIncrease > actualReceived` confirms the invariant `withdrawable increase == actual tokens received` is broken. The final `withdraw` revert confirms lenders cannot redeem their full entitlement, proving protocol insolvency.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
