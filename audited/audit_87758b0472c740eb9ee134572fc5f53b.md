### Title
Zero Settlement Fee Collected on Unit-Granularity Buy-Offer Takes Due to Integer Rounding — (`src/Midnight.sol`)

### Summary
When `offer.buy == true` and `offer.maxAssets > 0`, a taker can call `take` with `units = 1` on any offer whose tick price is strictly less than `WAD` (1e18). Because `buyerAssets` and `sellerAssets` are both computed with `mulDivDown` and `offerPrice < WAD`, both round to zero. The line `claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets` therefore adds zero, collecting no settlement fee despite the take creating a real unit of credit and debt. As a compounding effect, the `consumed` counter also fails to advance (it increments by `buyerAssets = 0`), so the offer's `maxAssets` cap is never reached and the offer can be taken an unlimited number of times.

### Finding Description

**Exact code path (`src/Midnight.sol`):**

```
take(offer, ..., units=1)
  offerPrice = TickLib.tickToPrice(offer.tick)          // any tick < MAX_TICK → offerPrice < WAD
  _settlementFee = settlementFee(id, timeToMaturity)    // > 0 by precondition
  sellerPrice = offerPrice - _settlementFee             // ≥ 0 (required, else underflow revert)
  buyerPrice  = sellerPrice + _settlementFee            // = offerPrice algebraically
  buyerAssets  = mulDivDown(1, offerPrice, WAD)         // = 0  (offerPrice < WAD)
  sellerAssets = mulDivDown(1, sellerPrice, WAD)        // = 0  (sellerPrice < offerPrice < WAD)

  // consumed branch (offer.maxAssets > 0, offer.buy == true):
  consumed[maker][group] += buyerAssets                 // += 0 → counter unchanged
  require(0 <= offer.maxAssets)                         // always passes

  // position accounting:
  buyerCreditIncrease = zeroFloorSub(1, buyerPos.debt)  // = 1 (fresh buyer)
  sellerDebtIncrease  = 1 - sellerCreditDecrease        // = 1 (fresh seller)
  buyerPos.credit  += 1                                 // real credit created
  sellerPos.debt   += 1                                 // real debt created

  // fee accounting:
  claimableSettlementFee[loanToken] += 0 - 0            // zero fee collected

  // token transfers:
  safeTransferFrom(loanToken, payer, this,     0)       // no tokens move
  safeTransferFrom(loanToken, payer, receiver, 0)       // no tokens move
```

**Root cause:** `buyerAssets` and `sellerAssets` are computed with `mulDivDown` (lines 363–364). Since `tickToPrice` always returns a value ≤ `WAD` (all ticks 0–5820 produce prices in `(0, WAD]`), multiplying by `units = 1` and dividing by `WAD` yields zero for any tick where `offerPrice < WAD`. The settlement fee line (line 418) then adds `0 - 0 = 0`. No guard checks that `buyerAssets > 0` before proceeding.

**Attacker-controlled inputs:**
- `offer.buy = true`
- `offer.maxAssets > 0` (any positive value)
- `offer.tick` = any valid tick where `tickToPrice(tick) < WAD` (ticks 0 through at least 5819)
- `units = 1`
- Market must have `_settlementFee > 0`

**Why existing checks fail:**
- No `require(units > 0)` or `require(buyerAssets > 0)` guard exists.
- The `ConsumedAssets` check (`require(newConsumed <= offer.maxAssets)`) passes trivially because `newConsumed` never advances.
- The code comment at line 113–114 acknowledges "fees manipulations on chains with very cheap gas" but provides no on-chain enforcement.
- The `sellerPrice = offerPrice - _settlementFee` underflow protection only prevents `offerPrice < _settlementFee`; it does not prevent `offerPrice < WAD`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation
For every `take` with `units = 1` on a buy offer at any sub-WAD tick price, the protocol collects exactly zero settlement fee while simultaneously recording one unit of credit for the buyer and one unit of debt for the seller. Because the `consumed` counter also stays at zero, the same offer can be taken an unlimited number of times. On low-gas chains (L2s), an attacker or colluding maker/taker pair can drain the protocol's expected settlement fee revenue across any number of takes at negligible cost. The invariant "settlement fee must increase for every non-zero-units take with non-zero settlement fee" is violated. [5](#0-4) 

### Likelihood Explanation
**Preconditions:**
1. Market has a non-zero settlement fee (default or per-market) — normal operating condition.
2. Offer tick is any value in `[0, MAX_TICK)` — covers essentially the entire valid tick range since `tickToPrice(MAX_TICK) = tickToPrice(5820)` is still slightly below `WAD` after `PRICE_ROUNDING_STEP` rounding.
3. `offer.buy = true`, `offer.maxAssets > 0` — standard buy-offer configuration.
4. Taker passes `units = 1`.

All preconditions are reachable by any unprivileged taker without any special state. The attack is repeatable an unlimited number of times on the same offer (since `consumed` never advances). On cheap-gas chains the cost per zero-fee take is only the base transaction cost. [6](#0-5) [7](#0-6) 

### Recommendation
Add a minimum-assets guard before the consumed and fee accounting. Two complementary fixes:

1. **Require `buyerAssets > 0` (or `units == 0`) for non-zero-units takes:**
   ```solidity
   require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
   ```
   This prevents any take where rounding collapses the asset amount to zero.

2. **Switch the consumed branch to track `units` instead of assets when `maxAssets > 0` and the asset amount rounds to zero**, or enforce a minimum `units` granularity per tick (e.g., `units >= WAD / offerPrice`).

3. **Alternatively, round `buyerAssets` up (`mulDivUp`) for buy offers** so the buyer always pays at least 1 asset per unit, ensuring the fee is always positive. This is consistent with the existing comment that assets are "rounded against the taker." [8](#0-7) [9](#0-8)
