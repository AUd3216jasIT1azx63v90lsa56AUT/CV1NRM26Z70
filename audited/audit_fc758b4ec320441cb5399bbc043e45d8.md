Audit Report

## Title
Double `mulDivDown` Sequential Floor Rounding Understates `maxDebt` by 1, Enabling Liquidation of Economically Healthy Positions - (File: src/Midnight.sol)

## Summary
`isHealthy` and `liquidate` both compute `maxDebt` via two sequential `mulDivDown` (floor division) calls. Each call independently discards a remainder, and the combined rounding error can understate `maxDebt` by exactly 1 relative to the true single-step floor. When a position sits at the exact health boundary, the protocol computes `maxDebt = debt âˆ’ 1`, causing `isHealthy` to return `false` and the `liquidate` guard to pass, allowing seizure of collateral from an economically healthy borrower.

## Finding Description

**Confirmed code path:**

`mulDivDown` is plain integer floor division with no rounding protection (`src/libraries/UtilsLib.sol` lines 29â€“31):
```solidity

    return (x * y) / d;
}
```

`isHealthy` applies it twice in sequence (`src/Midnight.sol` lines 954â€“955):
```solidity

```

`liquidate` uses the identical double-floor pattern (`src/Midnight.sol` line 613):
```solidity

```

The liquidatability guard compares the understated `maxDebt` against `originalDebt` (`src/Midnight.sol` lines 620â€“624):
```solidity

        && (postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt),
```

**Root cause â€” double floor rounding:**

Let `V1 = floor(C Ã— P / ORACLE_PRICE_SCALE)` and `V2 = floor(V1 Ã— L / WAD)`. The exact single-step bound is `M = floor(C Ã— P Ã— L / (ORACLE_PRICE_SCALE Ã— WAD))`. Because step 1 discards remainder `r = (C Ã— P) mod ORACLE_PRICE_SCALE` before step 2 multiplies by `L`, the combined error satisfies `M âˆ’ 1 â‰¤ V2 â‰¤ M`. The off-by-one (`V2 = M âˆ’ 1`) is reachable whenever `r Ã— L / WAD â‰¥ 1`, i.e., `r â‰¥ WAD / L`.

**Concrete counterexample** using confirmed protocol constants (`ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`, valid LLTV tier `LLTV_2 = 0.77e18`):

| Parameter | Value |
|---|---|
| `collateral` (C) | 10 |
| `price` (P) | `1.3e35` |
| `lltv` (L) | `0.77e18` |
| `debt` (D) | 1 |

- Step 1: `(10 Ã— 1.3e35) / 1e36 = 1` (floor of 1.3)
- Step 2: `(1 Ã— 0.77e18) / 1e18 = 0` (floor of 0.77)
- Exact: `10 Ã— 1.3e35 Ã— 0.77e18 / 1e54 = 1.001` â†’ `floor = 1 = debt` â†’ **healthy**
- Protocol sees: `maxDebt = 0 < 1 = debt` â†’ **unhealthy** â†’ liquidation guard passes

**Exploit flow:**

1. Borrower opens a position with C = 10 collateral units and borrows D = 1 debt unit at price P = 2e35. At that price: `V1 = floor(10 Ã— 2e35 / 1e36) = 2`, `V2 = floor(2 Ã— 0.77e18 / 1e18) = 1 â‰¥ 1 = debt` â†’ `isHealthy` returns `true` at the `take` call site (line 476), so the borrow succeeds.
2. Oracle price drifts to exactly `1.3e35` through normal market movement. The position remains economically healthy: exact `maxDebt = floor(1.001) = 1 = debt`.
3. Liquidator calls `liquidate(market, collateralIndex, seizedAssets, 0, borrower, false, ...)`.
4. Protocol computes `maxDebt = 0`; guard `originalDebt (1) > maxDebt (0)` passes; collateral is seized from a healthy borrower.

**Why existing checks fail:**

There are no minimum debt or collateral dust guards anywhere in the codebase that would prevent positions with small integer values from being created. The health check at the `take` path uses the same `isHealthy` function with the same double-floor computation, so it does not prevent the boundary condition from being reached after price movement. The contract's own natspec at line 116 acknowledges `maxDebt` is rounded down but does not account for the compounding of two independent floor operations producing an additional unit of error beyond a single-step floor.

## Impact Explanation
A liquidator can seize collateral from a borrower whose position is economically healthy (exact `maxDebt â‰¥ debt`) purely due to the double floor-rounding understating `maxDebt` by 1. The borrower suffers an unwarranted collateral loss. The core protocol invariant â€” healthy positions cannot be liquidated â€” is violated. This is a direct, concrete loss of user funds triggered by an unprivileged external caller.

## Likelihood Explanation
No oracle manipulation is required. For any fixed collateral amount `C` and LLTV `L`, there exists a positive-measure interval of oracle prices `P` satisfying `(C Ã— P) mod ORACLE_PRICE_SCALE â‰¥ WAD / L` that triggers the off-by-one. A liquidator needs only to monitor positions and call `liquidate` when the price enters such an interval. With many positions across many markets, the probability of at least one being in the vulnerable range at any given time is non-negligible. The condition is passively reachable through normal price movement.

## Recommendation
Replace the two sequential `mulDivDown` calls with a single combined multiplication before dividing, computing `maxDebt` as:
```solidity
maxDebt += _collateral.mulDivDown(price * collateralParam.lltv, ORACLE_PRICE_SCALE * WAD);
```
This performs a single floor operation on the exact numerator `C Ã— P Ã— L`, eliminating the intermediate rounding. Care must be taken to ensure `price * collateralParam.lltv` does not overflow `uint256`; given `price` is bounded by oracle design and `lltv â‰¤ WAD = 1e18`, the product fits within `uint256` for all realistic oracle prices. Alternatively, use a 512-bit intermediate (e.g., via a `mulDiv` with full-precision numerator) to avoid any overflow concern.
