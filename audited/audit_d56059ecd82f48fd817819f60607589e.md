### Title
Dust-take on buy offer in assets mode accrues unbounded debt to victim with zero asset transfer - (`src/Midnight.sol`)

### Summary

When `take()` is called on a buy offer with `units=1` and `offerPrice < WAD`, both `sellerAssets` and `buyerAssets` round to zero via `mulDivDown`, yet `sellerDebtIncrease` is still 1. In assets mode (`maxAssets > 0`), the consumed counter tracks `buyerAssets`; since `buyerAssets = 0`, consumed never advances, making the offer infinitely re-takeable. An attacker who is authorized for a victim can therefore call `take()` with `taker=victim` in a tight loop, each iteration adding 1 unit of debt to the victim while transferring zero assets, until the victim's health is degraded to the liquidation boundary.

### Finding Description

**Code path:**

In `Midnight.take()`:

```
sellerPrice = offerPrice - _settlementFee          // line 361
sellerAssets = units.mulDivDown(sellerPrice, WAD)  // line 364  → 0 when units=1, sellerPrice < WAD
buyerAssets  = units.mulDivDown(buyerPrice,  WAD)  // line 363  → 0 when units=1, buyerPrice  < WAD
```

For any buy offer whose tick is below the maximum, `offerPrice < WAD`, so `buyerPrice = offerPrice < WAD` and `sellerPrice = offerPrice - settlementFee < WAD`. With `units = 1`, both `mulDivDown` calls return `floor(price / WAD) = 0`.

**Consumed counter does not advance (assets mode):**

```solidity
// lines 367-369
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
// buyerAssets == 0  →  consumed unchanged  →  require(newConsumed <= offer.maxAssets) always passes
```

Because `buyerAssets = 0`, `consumed[maker][group]` is never incremented. The offer can be taken an unlimited number of times.

**Debt accrues with no asset transfer:**

```solidity
sellerDebtIncrease = units - sellerCreditDecrease;   // = 1 - 0 = 1  (victim has no credit)
sellerPos.debt += sellerDebtIncrease;                 // line 414: victim debt += 1
SafeTransferLib.safeTransferFrom(..., receiver, sellerAssets);  // line 456: transfers 0
```

**Exploit flow:**

1. Attacker deploys a buy offer with `offer.buy = true`, `offer.maxAssets > 0`, any tick below maximum (e.g., tick 4), and a valid ratifier.
2. Victim has authorized the attacker: `isAuthorized[victim][attacker] = true` (via `setIsAuthorized` or `EcrecoverAuthorizer.setIsAuthorized` with a signed `Authorization`).
3. Attacker calls `take(buyOffer, ratifierData, 1, victim, receiver, address(0), "")` in a loop.
4. Each call: `sellerAssets = 0`, `buyerAssets = 0`, `sellerDebtIncrease = 1`. Victim's debt grows by 1 per iteration; no tokens move; consumed stays fixed.
5. The health check at line 476 (`require(... || isHealthy(...))`) stops each individual call from making the victim immediately liquidatable, but the attacker can iterate until the victim is at the liquidation boundary.

**Why existing checks fail:**

- **Authorization check (line 346)**: `require(taker == msg.sender || isAuthorized[taker][msg.sender])` — satisfied because the victim authorized the attacker.
- **Health check (line 476)**: Prevents each single call from crossing the liquidation threshold, but does not prevent the cumulative degradation across many calls.
- **No minimum-assets guard**: There is no `require(sellerAssets > 0 || sellerDebtIncrease == 0)` or equivalent.
- **`setConsumed` pre-consumption**: Not required; the consumed counter never advances regardless of its initial value when `buyerAssets = 0`.

### Impact Explanation

The victim (seller/taker) accrues 1 unit of debt per dust-take while receiving zero loan-token assets. Repeated calls degrade the victim's collateral health monotonically toward the liquidation threshold. Once at the boundary, any small oracle price movement makes the victim liquidatable, allowing the attacker (or a third party) to seize collateral. The attacker's cost is gas only; no loan tokens are required.

### Likelihood Explanation

**Preconditions:**
1. Attacker must be authorized for the victim (`isAuthorized[victim][attacker] = true`). This is achievable if the victim has authorized a smart contract the attacker controls (e.g., a malicious bundler), or if the victim was phished into signing an `Authorization` struct consumed via `EcrecoverAuthorizer.setIsAuthorized`.
2. A buy offer with `offerPrice < WAD` and `maxAssets > 0` must exist (or the attacker creates one as maker).
3. The victim must have an active position with collateral.

**Feasibility:** Condition 1 is the main barrier, but it is realistic given the `EcrecoverAuthorizer` signature flow and the common pattern of users authorizing bundler contracts. Conditions 2 and 3 are trivially satisfiable by the attacker. The attack is repeatable at zero token cost per iteration.

### Recommendation

Add a guard in `take()` that rejects any call where debt would increase but the seller receives zero assets:

```solidity
require(sellerDebtIncrease == 0 || sellerAssets > 0, DustTakeCreatesDebt());
```

This enforces the invariant that every unit of new seller debt must be backed by a non-zero asset transfer. Alternatively, enforce `require(units == 0 || sellerAssets > 0 || sellerCreditDecrease == units)` to allow credit-reducing takes while blocking debt-creating dust takes.

### Proof of Concept

```solidity
// Foundry fuzz/unit test
function testDustTakeDebtWithZeroAssets(uint256 tick) public {
    // Bound tick to any valid tick below maximum (offerPrice < WAD)
    tick = bound(tick, MIN_TICK, MAX_TICK - tickSpacing) / tickSpacing * tickSpacing;

    // Setup: attacker is maker of buy offer, victim is taker
    Offer memory buyOffer = Offer({
        buy: true,
        maker: attacker,
        maxAssets: 1e30,   // assets mode
        maxUnits: 0,
        tick: tick,
        // ... other fields
    });

    // Victim authorizes attacker
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, true, victim);

    // Collateralize victim so health check passes initially
    collateralize(market, victim, 1000e18);

    uint256 debtBefore = midnight.debtOf(id, victim);

    // Single dust take
    vm.prank(attacker);
    (uint256 buyerAssets, uint256 sellerAssets) =
        midnight.take(buyOffer, ratifierData, 1, victim, victim, address(0), "");

    // Assertions
    assertEq(sellerAssets, 0, "sellerAssets must be 0 (rounding)");
    assertEq(buyerAssets,  0, "buyerAssets must be 0 (rounding)");
    assertEq(midnight.debtOf(id, victim), debtBefore + 1, "victim debt increased");

    // Consumed did not advance — offer is infinitely re-takeable
    assertEq(midnight.consumed(attacker, buyOffer.group), 0, "consumed unchanged");

    // Invariant assertion (should FAIL without fix):
    // sellerAssets > 0 whenever sellerDebtIncrease > 0
    assertTrue(sellerAssets > 0, "INVARIANT VIOLATED: debt created with zero asset transfer");
}
```

**Expected result without fix:** `sellerAssets = 0`, `victim.debt = debtBefore + 1`, `consumed = 0` (unchanged). The invariant assertion fails, confirming the bug. With the recommended fix, the `take()` call reverts with `DustTakeCreatesDebt`.
