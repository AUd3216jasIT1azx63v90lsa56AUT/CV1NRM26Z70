### Title
Dust-withdrawal `mulDivUp` rounding drains `pendingFee` to zero before `credit` reaches zero, enabling continuous-fee evasion - (File: src/Midnight.sol)

### Summary
In `withdraw()`, `pendingFeeDecrease` is computed as `pendingFee.mulDivUp(units, credit)`. When `units = 1` and `pendingFee < credit`, `mulDivUp` always rounds up to exactly 1, so each dust call removes 1 from `pendingFee` but only 1 from `credit`. After `pendingFee` such calls the lender holds `credit - pendingFee` credit with `pendingFee == 0`, paying no further continuous fee on that remaining balance. The Certora spec `pendingFeeDecreasesProportionallyOnWithdraw` only verifies a single call and does not catch the cumulative drift.

### Finding Description

**Exact code path** — `src/Midnight.sol` lines 489–491:

```solidity
if (_position.credit > 0) {
    pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
    _position.pendingFee -= pendingFeeDecrease;
}
_position.credit -= UtilsLib.toUint128(units);
```

`mulDivUp` is defined in `src/libraries/UtilsLib.sol` line 35 as `(x * y + (d - 1)) / d`.

With `units = 1`:

```
mulDivUp(P, 1, C) = (P + C - 1) / C
```

When `0 < P < C` (the normal operating range, since `pendingFee ≤ credit` is a protocol invariant), this evaluates to exactly `1` for every call, regardless of the true ratio `P/C`.

**Cumulative drift over N = P calls:**

| Call k | credit before | pendingFee before | decrease | credit after | pendingFee after |
|--------|--------------|-------------------|----------|--------------|-----------------|
| 1 | C | P | 1 | C−1 | P−1 |
| 2 | C−1 | P−1 | 1 | C−2 | P−2 |
| … | … | … | 1 | … | … |
| P | C−P+1 | 1 | 1 | C−P | **0** |

After exactly `P` calls: `credit = C − P > 0`, `pendingFee = 0`.

**Why existing checks fail:**
- No minimum `units` floor in `withdraw()`.
- Authorization check (`onBehalf == msg.sender`) is satisfied by the lender acting for themselves.
- The Certora rule `pendingFeeDecreasesProportionallyOnWithdraw` (ContinuousFee.spec line 94–107) asserts the formula for a **single** call only; it does not constrain the ratio `pendingFee / credit` across multiple calls.
- `_updatePosition` is called at the top of each `withdraw()`, but it only accrues time-based fees; it does not restore the `pendingFee / credit` ratio. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

After the attack, `credit = C − P > 0` with `pendingFee = 0`. `updatePositionView` computes the accrued fee as `pendingFee.mulDivDown(elapsed, ttm)`, which is `0` for all future time. The lender retains `C − P` credit and owes zero continuous fee on it for the remainder of the market's life. The protocol loses the fee revenue it was owed on that balance. The lender also recovers `P` units of loan token from the withdrawals, so the net gain equals the full original `pendingFee` value. [4](#0-3) 

### Likelihood Explanation

**Preconditions:** lender has `credit = C` and `pendingFee = P` with `0 < P < C`; enough `withdrawable` liquidity exists (borrower has repaid ≥ P units). Both are normal post-`take` states.

**Feasibility:** The lender calls `withdraw(market, 1, lender, lender)` exactly `P` times. No privileged role is required. The call sequence is fully attacker-controlled.

**Gas constraint:** Each call costs roughly 30 000–50 000 gas. On Ethereum mainnet the gas cost per call typically exceeds the value of 1 loan-token unit for standard 6- or 18-decimal tokens, making the attack unprofitable there. On low-fee L2 deployments (Arbitrum, Base, etc.) or for high-value-per-unit tokens the break-even shifts and the attack becomes economically rational. The invariant violation itself is unconditional regardless of profitability.

### Recommendation

Replace `mulDivUp` with `mulDivDown` for the `pendingFeeDecrease` in `withdraw()`, and add an explicit full-exit guard to ensure a complete withdrawal zeroes `pendingFee`:

```solidity
if (_position.credit > 0) {
    pendingFeeDecrease = units >= _position.credit
        ? _position.pendingFee
        : UtilsLib.toUint128(_position.pendingFee.mulDivDown(units, _position.credit));
    _position.pendingFee -= pendingFeeDecrease;
}
```

`mulDivDown` never over-reduces `pendingFee` for partial withdrawals, and the explicit guard preserves the existing invariant that a full withdrawal zeroes the fee. The same fix should be applied symmetrically to the `sellerPendingFeeDecrease` computation in `take()` at line 388 if the same attack vector applies there. [5](#0-4) 

### Proof of Concept

```solidity
pragma solidity ^0.8.0;

import {ContinuousFeeTest} from "./ContinuousFeeTest.sol";

contract DustWithdrawFeeEvasionTest is ContinuousFeeTest {
    function testDustWithdrawDrainsPendingFee() public {
        uint256 credit   = 1_000;   // C
        uint256 feeRate  = 1e14;    // non-zero continuous fee
        uint256 ttm      = 100 days;

        setupLender(credit, feeRate, ttm);

        uint256 P = midnight.pendingFee(id, lender);
        // Precondition: 0 < P < C
        require(P > 0 && P < credit, "precondition");

        // Repay so withdrawable >= P
        deal(address(loanToken), borrower, credit);
        vm.prank(borrower);
        midnight.repay(market, credit, borrower, address(0), hex"");

        // Attack: P calls of withdraw(1)
        vm.startPrank(lender);
        for (uint256 i = 0; i < P; i++) {
            midnight.withdraw(market, 1, lender, lender);
        }
        vm.stopPrank();

        uint256 creditAfter     = midnight.creditOf(id, lender);
        uint256 pendingFeeAfter = midnight.pendingFee(id, lender);

        // Invariant violated: pendingFee == 0 while credit > 0
        assertEq(pendingFeeAfter, 0,        "pendingFee drained to zero");
        assertGt(creditAfter,     0,        "credit still positive");
        // Ratio pendingFee/credit should be ~P/C but is 0
        // No future fee will accrue on creditAfter
    }
}
```

Expected assertions: `pendingFeeAfter == 0` and `creditAfter == credit - P > 0`, demonstrating that the lender retains positive credit with zero fee obligation.

```text
// When credit decreases via withdraw, pendingFee decreases by ceil(pendingFee * units / postUpdateCredit).
rule pendingFeeDecreasesProportionallyOnWithdraw(env e, Midnight.Market market, uint256 units, address onBehalf, address receiver) {
    bytes32 id;
    uint128 postUpdateCredit;
    uint128 postUpdatePendingFee;

    postUpdateCredit, postUpdatePendingFee, _ = updatePositionView(e, market, id, onBehalf);

    withdraw(e, market, units, onBehalf, receiver);

    require id == lastId, "id should be derived from market";

    // When postUpdateCredit == 0, pendingFee(id, onBehalf) is unchanged on withdraw.
    assert postUpdateCredit == 0 ? pendingFee(id, onBehalf) == postUpdatePendingFee : pendingFee(id, onBehalf) == postUpdatePendingFee - (postUpdatePendingFee * units + postUpdateCredit - 1) / postUpdateCredit;
}
```
