### Title
Buy-offer `maxAssets` cap bypassed via `mulDivDown` rounding to zero in `take()` â€” (`src/Midnight.sol`)

### Summary

In `take()`, when `offer.buy == true` and `offer.maxAssets > 0`, the consumed counter is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. When `units * buyerPrice < WAD`, this rounds to zero, so `consumed[maker][group]` never advances. The `require(newConsumed <= offer.maxAssets)` check passes trivially, allowing the offer to be filled an unbounded number of times beyond the maker's intended cap. The codebase already contains `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858â€“889), which is a named-bug regression test that explicitly confirms this behavior.

### Finding Description

**Exact code path** â€” `src/Midnight.sol` lines 363â€“369:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}

**Root cause**: `mulDivDown(units, buyerPrice, WAD)` returns `0` whenever `units * buyerPrice < WAD`. Because `tickToPrice` is proven to return values `<= WAD` (see `NoMultiplicationOverflow.spec` line 55), `buyerPrice <= WAD` always holds. A taker can therefore choose any `units` satisfying `units < WAD / buyerPrice` to produce `buyerAssets = 0`.

**Attacker-controlled inputs**:
- `units`: chosen so that `units * buyerPrice < WAD` (e.g., `units = 1` when `buyerPrice < WAD`)
- `offer`: any valid buy offer with `maxAssets > 0` and `buyerPrice < WAD` (any tick below the maximum)

**Exploit flow**:
1. Maker posts a buy offer: `offer.buy = true`, `maxAssets = M`, tick chosen so `buyerPrice < WAD`.
2. Taker calls `take(units, ...)` repeatedly with `units` satisfying `units * buyerPrice < WAD`.
3. Each call: `buyerAssets = 0`, `consumed` += 0, check `0 <= M` passes.
4. Maker's credit (`creditOf(id, maker)`) grows by `units` per call; `consumed` stays at its pre-call value.
5. After N calls the maker has delivered `N * units` credit while `consumed` has not advanced past its starting value.

**Why existing checks fail**: The only guard is `require(newConsumed <= offer.maxAssets)`. Since `newConsumed` is incremented by `buyerAssets = 0`, the check is trivially satisfied regardless of how many units have actually been delivered. There is no `require(buyerAssets > 0 || units == 0)` guard.

The Certora rule `takeConsumedBoundedByMax` (line 62 of `certora/specs/Consume.spec`) only verifies `consumed <= maxAssets` â€” it does not verify that total units delivered are bounded by what `maxAssets` implies, so the formal verification does not catch this. [2](#0-1)

### Impact Explanation

A taker can fill a buy offer beyond the maker's `maxAssets` limit without advancing the consumed counter. The maker (lender) accumulates credit (units) in excess of what `maxAssets` was intended to cap, violating the core invariant that "offers cannot be replayed, overfilled, reused, or filled after cancel/deadline." Concretely: with `maxAssets = 1` and `buyerPrice < WAD`, the offer can be taken for arbitrarily many units at zero token cost, giving the maker unbounded credit exposure they did not consent to.

### Likelihood Explanation

**Preconditions**: `offer.buy == true`, `offer.maxAssets > 0`, and `buyerPrice < WAD`. The last condition holds for every tick below the maximum (i.e., virtually all real offers, since `tickToPrice` returns `<= WAD` and equals `WAD` only at the very top tick). The taker simply needs to pass `units = 1` (or any value `< WAD / buyerPrice`). No special privilege, no oracle manipulation, and no front-running is required. The attack is repeatable in a single transaction via `multicall`.

### Recommendation

Add a guard that rejects takes where `units > 0` but `buyerAssets == 0` under assets-mode accounting:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssetsWithNonZeroUnits());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, use `mulDivUp` for the consumed increment on buy offers so that rounding always favors the cap rather than the taker, consistent with how sell-offer assets are computed.

### Proof of Concept

The existing `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858â€“889) already reproduces the single-step bypass. A fuzz extension to prove cumulative overfill:

```solidity
function testFuzz_BuyMaxAssets_CumulativeOverfill(uint256 buyerPrice, uint256 units, uint8 numTakes) public {
    // Preconditions
    vm.assume(buyerPrice > 0 && buyerPrice < WAD);
    vm.assume(units > 0 && units * buyerPrice < WAD); // ensures mulDivDown == 0
    vm.assume(numTakes > 0 && numTakes <= 100);

    uint256 maxAssets = 1; // any small cap
    // set up lenderOffer with maxAssets, tick matching buyerPrice
    // fund and collateralize appropriately

    uint256 creditBefore = midnight.creditOf(id, lender);
    for (uint256 i; i < numTakes; i++) {
        take(units, borrower, lenderOffer);
    }

    uint256 totalUnitsDelivered = midnight.creditOf(id, lender) - creditBefore;
    uint256 impliedMaxUnits = maxAssets * WAD / buyerPrice; // floor

    // INVARIANT: must hold but does NOT
    assertLe(totalUnitsDelivered, impliedMaxUnits, "offer overfilled");
    // consumed never advanced, so all takes passed
    assertEq(midnight.consumed(lender, lenderOffer.group), 0);
}
```

Expected: `assertLe` fails because `totalUnitsDelivered = numTakes * units >> impliedMaxUnits` while `consumed` remains `0`.
