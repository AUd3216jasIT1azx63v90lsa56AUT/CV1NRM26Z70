### Title
Settlement Fee Overcrediting via Fee-on-Transfer Loan Token with BuyerCallback Payer - (File: src/Midnight.sol)

### Summary

In `take()`, `claimableSettlementFee` is incremented by the nominal `buyerAssets - sellerAssets` before any ERC20 transfers occur. When the loan token is fee-on-transfer and a `buyerCallback` is set (making `payer = buyerCallback`), two separate `safeTransferFrom` calls each deduct a transfer fee, so Midnight receives less than the credited amount. The overcredited `claimableSettlementFee` can then be claimed by the `feeClaimer`, draining funds that belong to lenders.

### Finding Description

**Code path:**

In `src/Midnight.sol`, `take()` executes in this order:

1. Line 418 — accounting update (before any transfer):
   ```solidity
   claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
   ```

2. Lines 420–422 — payer resolution:
   ```solidity
   address buyerCallback = offer.buy ? offer.callback : takerCallback;
   address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
   ```
   When `offer.buy == false` (sell offer) and the taker supplies a non-zero `takerCallback`, `buyerCallback = takerCallback` and `payer = buyerCallback`.

3. Lines 445–452 — callback invocation (attacker returns `CALLBACK_SUCCESS`).

4. Lines 455–456 — two separate transfers from `payer`:
   ```solidity
   SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
   SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
   ```

**Root cause:** `SafeTransferLib.safeTransferFrom` (lines 24–34 of `src/libraries/SafeTransferLib.sol`) only checks that `transferFrom` returns `true`; it does not verify the amount actually received. With a fee-on-transfer token at rate `r`:
- Midnight receives `(buyerAssets - sellerAssets) × (1 − r)` instead of `buyerAssets - sellerAssets`.
- `claimableSettlementFee` was already incremented by the full `buyerAssets - sellerAssets`.
- Overcrediting per take: `(buyerAssets - sellerAssets) × r`.

**Attacker inputs:**
- Taker takes any sell offer (`offer.buy == false`) with a fee-on-transfer loan token.
- Supplies `takerCallback` pointing to an attacker-controlled contract implementing `IBuyCallback` that holds the fee-on-transfer token and returns `CALLBACK_SUCCESS`.
- No privileged role is required; `take()` is permissionless for any authorized taker.

**Why existing checks fail:**
- The Certora formal verification (`certora/specs/Solvency.spec`, line 31) explicitly assumes "no fee taking from sender or receiver" — this is a prover simplification, not an on-chain enforcement.
- The protocol accepts "arbitrary loan token" (per `live_context.json`, line 17) with no on-chain restriction against fee-on-transfer tokens.
- `live_context.json` line 233 lists fee-on-transfer tokens as something that "should be tested if not explicitly excluded" — confirming no explicit exclusion exists in the source.
- There is no balance snapshot before/after the transfers to verify received amounts.

### Impact Explanation

`claimableSettlementFee[loanToken]` is overcredited by `(buyerAssets - sellerAssets) × feeRate` per exploited take. When `feeClaimer` calls `claimSettlementFee(token, amount, receiver)` (line 305–310), it decrements `claimableSettlementFee` and calls `safeTransfer(token, receiver, amount)`. Since the contract's actual balance is short by the accumulated overcrediting, the claim drains tokens that belong to lenders (their `withdrawable` credit). Repeated takes compound the deficit. The solvency invariant — contract token balances must cover withdrawable assets, collateral claims, credit redemptions, and accrued fees — is broken.

### Likelihood Explanation

**Preconditions:**
1. A market exists with a fee-on-transfer loan token (no on-chain restriction prevents this).
2. A sell offer exists in that market (any borrower can publish one).
3. The attacker deploys a minimal `IBuyCallback` contract holding the fee-on-transfer token.

All three are attacker-reachable without any privileged action. The attack is repeatable across multiple takes and compounds with each execution. The only external dependency is the existence of a fee-on-transfer token as the loan token, which is a realistic market configuration given the protocol's permissionless market creation.

### Recommendation

After the two `safeTransferFrom` calls, verify that the contract's actual balance increased by at least `buyerAssets - sellerAssets`. Concretely, snapshot the contract's token balance before the transfers and assert the delta post-transfer:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
require(
    IERC20(offer.market.loanToken).balanceOf(address(this)) >= balanceBefore + (buyerAssets - sellerAssets),
    InsufficientReceivedAmount()
);
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens (e.g., via a registry or a deployment-time check), and add this as a hard invariant in the Certora specs.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
