### Title
Fee-on-transfer loanToken causes `repayAndWithdrawCollateral` to always revert - (`src/periphery/MidnightBundles.sol`)

### Summary
`MidnightBundles.repayAndWithdrawCollateral` pulls `assets` from `msg.sender` and then calls `Midnight.repay(market, units, ...)` where `units = assets - referralFeeAssets`. With a fee-on-transfer `loanToken`, `pullToken` delivers only `assets*(1-fee)` to MidnightBundles, but `repay` attempts to pull the full `units` from MidnightBundles. When `referralFeePct = 0` (or when the transfer fee rate exceeds `referralFeePct`), MidnightBundles holds fewer tokens than `units`, causing `safeTransferFrom` inside `repay` to revert. This permanently DoS-es the bundler's repay path for any market whose `loanToken` is fee-on-transfer.

### Finding Description
**Code path:**

1. `MidnightBundles.repayAndWithdrawCollateral` (line 329â€“334):
   ```solidity
   uint256 referralFeeAssets = assets.mulDivDown(referralFeePct, WAD);  // line 329
   uint256 units = assets - referralFeeAssets;                           // line 330
   pullToken(loanToken, msg.sender, assets, loanTokenPermit);            // line 331
   forceApproveMax(loanToken, MIDNIGHT);                                 // line 332
   IMidnight(MIDNIGHT).repay(market, units, onBehalf, address(0), "");  // line 334
   ```

2. `pullToken` calls `SafeTransferLib.safeTransferFrom(token, from, address(this), amount)` (line 396), which succeeds but delivers only `assets*(1-fee_rate)` to MidnightBundles due to the token's transfer fee.

3. `Midnight.repay` (line 511): `payer = msg.sender` (MidnightBundles, since `callback == address(0)`).

4. `Midnight.repay` (line 520): `SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units)` â€” attempts to pull `units` from MidnightBundles.

**Root cause:** No balance-delta accounting after `pullToken`. The function assumes `pullToken(assets)` delivers exactly `assets` to MidnightBundles, but fee-on-transfer tokens deliver `assets*(1-fee_rate)`.

**Attacker inputs:** Any borrower calls `repayAndWithdrawCollateral` on a market where `loanToken` is fee-on-transfer. No special permissions needed â€” only requires the borrower to have debt and have approved MidnightBundles.

**Why checks fail:** The only guards are `require(referralFeePct < WAD)` and the authorization check. Neither accounts for token transfer fees. The discrepancy between received balance and `units` is never validated.

**Worst case:** With `referralFeePct = 0`, `units = assets`. MidnightBundles holds `assets*(1-fee_rate)`. The `safeTransferFrom` in `repay` reverts unconditionally for any non-zero fee rate.

**General case:** Even with `referralFeePct > 0`, the revert occurs whenever `fee_rate > referralFeePct/WAD`, i.e., whenever the transfer fee exceeds the referral fee.

### Impact Explanation
`repayAndWithdrawCollateral` is permanently broken for any market whose `loanToken` is fee-on-transfer. Every call reverts at the `safeTransferFrom` inside `Midnight.repay`. Borrowers cannot use the bundler to repay debt and simultaneously withdraw collateral in a single transaction. They must call `Midnight.repay` directly (without the collateral withdrawal convenience), fragmenting the UX and potentially leaving collateral locked if the borrower relies on the atomic bundle. No funds are permanently lost since direct repayment remains available, but the bundler's repay path is fully DoS-ed for this token class.

### Likelihood Explanation
**Preconditions:**
- Market's `loanToken` is fee-on-transfer (e.g., USDT with fee enabled, STA, PAXG, or any custom token)
- Borrower has outstanding debt in that market
- Borrower calls `repayAndWithdrawCollateral` via MidnightBundles

**Feasibility:** Moderate. Fee-on-transfer tokens are uncommon but exist in production (USDT has a fee switch, several DeFi tokens use transfer fees). Market creation in Midnight is permissionless, so any user can create a market with a fee-on-transfer loanToken. The bug is deterministic â€” it triggers on every call, not probabilistically.

**Repeatability:** 100% â€” every call to `repayAndWithdrawCollateral` with a fee-on-transfer loanToken and `referralFeePct = 0` (or `fee_rate > referralFeePct/WAD`) reverts.

### Recommendation
After `pullToken`, measure the actual received balance and use it to compute `units`:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
pullToken(loanToken, msg.sender, assets, loanTokenPermit);
uint256 received = IERC20(loanToken).balanceOf(address(this)) - balanceBefore;
uint256 referralFeeAssets = received.mulDivDown(referralFeePct, WAD);
uint256 units = received - referralFeeAssets;
```

This ensures `units` is derived from the actual post-fee balance, not the nominal `assets` input, so `Midnight.repay` can always pull exactly `units` from MidnightBundles.
