### Title
Fully-consumed buy offer (maxAssets mode) can be re-taken with zero-asset units, mutating position state without payment - (File: src/Midnight.sol)

### Summary
When `offer.buy == true` and `offer.maxAssets > 0`, the consumed counter is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. If `units * buyerPrice < WAD`, `buyerAssets` rounds to zero, so `consumed` does not increase and the cap check `require(newConsumed <= offer.maxAssets)` passes even when `consumed` is already equal to `maxAssets`. A taker can therefore call `take` with `units > 0` on a fully-consumed buy offer, successfully mutating credit, debt, and `totalUnits` state with zero token transfer.

### Finding Description
**Code path:**

In `src/Midnight.sol` `take()`:

```
// line 363
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

// lines 367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [1](#0-0) 

`mulDivDown` is `(x * y) / d` (plain integer division): [2](#0-1) 

**Root cause:** When `units * buyerPrice < WAD`, `mulDivDown` returns 0. The consumed mapping is incremented by 0, so `newConsumed` stays at its current value. If `consumed[maker][group]` is already `maxAssets`, the check `newConsumed <= maxAssets` is trivially satisfied and the take proceeds.

**Exploit flow:**

1. Maker creates a buy offer with `maxAssets = M`, `maxUnits = 0`, and a tick where `buyerPrice < WAD` (any tick below `MAX_TICK`).
2. Offer is legitimately filled until `consumed[maker][group] == M` (or maker calls `setConsumed` directly).
3. Attacker (any taker) calls `take(offer, ..., units=1)`. Since `1 * buyerPrice < WAD`, `buyerAssets = 0`. `consumed` stays at `M`. Check passes.
4. The take continues: `buyerPos.credit` increases, `sellerPos.debt` increases, `totalUnits` increases — all with zero token transfer (`buyerAssets - sellerAssets = 0`).
5. Step 3–4 can be repeated indefinitely.

**Why existing checks fail:** The only guard against over-filling is `require(newConsumed <= offer.maxAssets)`. When the increment is 0, this check is a no-op. There is no `require(units == 0 || buyerAssets > 0)` guard and no `require(newConsumed > consumedBefore)` guard.

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already reproduces this exactly: [3](#0-2) 

It asserts `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged at `maxAssets`, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase.

### Impact Explanation
A fully-consumed buy offer is not blocked from further fills when `units * buyerPrice < WAD`. An attacker can:
- Repeatedly take the offer, accruing unbounded credit for the maker and debt for themselves (or any target seller) at zero cost.
- Inflate `totalUnits` and `pendingFee` state without any corresponding asset transfer, breaking the invariant that every credit unit has a matching asset payment.
- Grief the maker by forcing unwanted credit/debt positions onto them after their offer cap was reached.

### Likelihood Explanation
**Preconditions:**
- `offer.buy == true`, `offer.maxAssets > 0`, `offer.maxUnits == 0` — standard buy offer configuration.
- `buyerPrice < WAD` — true for any tick below `MAX_TICK`; the vast majority of real offers.
- `consumed[maker][group] >= maxAssets` — reachable by normal fills or by the maker calling `setConsumed`.
- `units` chosen so `units * buyerPrice < WAD` — e.g. `units = 1` when `buyerPrice < WAD`.

All preconditions are attacker-controllable with no privileged access. The attack is repeatable in the same transaction via multicall or across blocks.

### Recommendation
Add a guard that prevents a take from proceeding when `units > 0` but the asset increment is zero in maxAssets mode:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetIncrement = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetIncrement > 0, ZeroAssetIncrement());
    newConsumed = consumed[offer.maker][offer.group] += assetIncrement;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` granularity such that `units * buyerPrice >= WAD` always holds when `units > 0` in maxAssets mode.

### Proof of Concept
The existing Foundry test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) is a complete, passing reproduction. A complementary fuzz test:

```solidity
function testFuzz_BuyMaxAssetsBypassAfterFullConsumption(uint256 units) public {
    // Setup: buy offer, maxAssets = 1, low tick so buyerPrice < WAD
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16;

    // Pre-consume to cap
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, 1, lender);

    // Bound units so buyerAssets rounds to 0
    units = bound(units, 1, WAD / TickLib.tickToPrice(MAX_TICK - 16));

    collateralize(market, borrower, units);
    uint256 debtBefore = midnight.debtOf(id, borrower);

    take(units, borrower, lenderOffer); // must NOT succeed

    // Assertion: debt must not have increased
    assertEq(midnight.debtOf(id, borrower), debtBefore);
}
```

Expected: test **fails** (debt increases despite consumed == maxAssets), confirming the bug. After the fix, the take reverts and the assertion holds.
