### Title
Debt Increase Permitted at Exact Maturity Timestamp via Off-by-One in `take` Boundary Check - (`src/Midnight.sol`)

### Summary
The guard at `src/Midnight.sol:391` uses `block.timestamp <= offer.market.maturity`, which passes at exact equality, allowing `sellerDebtIncrease > 0` when `block.timestamp == market.maturity`. The protocol's own invariant explicitly forbids debt increase at the maturity boundary by timestamp equality. This is reachable by any unprivileged taker with a direct `take()` call (multicall is an irrelevant delivery wrapper).

### Finding Description
**Root cause — off-by-one in the maturity guard:**

```solidity
// src/Midnight.sol:391
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

When `block.timestamp == market.maturity`, the left operand is `true`, so the `require` passes unconditionally regardless of `sellerDebtIncrease`.

**`zeroFloorSub` behaviour at equality:**

```solidity
// src/libraries/UtilsLib.sol:22-26
function zeroFloorSub(uint256 x, uint256 y) internal pure returns (uint256 z) {
    assembly { z := mul(gt(x, y), sub(x, y)) }
}
```

`gt(maturity, maturity)` evaluates to `0`, so `timeToMaturity = 0` at `block.timestamp == maturity` — identical to the post-maturity case. The settlement fee is therefore the 0-day fee, and `buyerPendingFeeIncrease = 0`. The protocol is already treating this timestamp as post-maturity for fee purposes, but the debt-increase guard does not.

**Exploit flow:**

1. A sell offer exists (maker = borrower) with `offer.expiry >= market.maturity`.
2. The seller has zero credit (`sellerPos.credit == 0`), so `sellerCreditDecrease = 0` and `sellerDebtIncrease = units`.
3. Taker waits for (or mines) `block.timestamp == market.maturity`.
4. Taker calls `take(sellOffer, ..., units, taker, ...)`.
5. Line 391 check: `maturity <= maturity` → `true` → passes.
6. Line 414: `sellerPos.debt += units` — debt is increased at maturity.

**Why existing checks do not stop it:**

- `block.timestamp <= offer.expiry` (line 353): passes if `expiry >= maturity`.
- The maturity guard (line 391) uses `<=` not `<`.
- Post-maturity liquidation (line 622) requires `block.timestamp > market.maturity` (strict), so the newly increased debt cannot be liquidated in post-maturity mode in the same block.

### Impact Explanation
A borrower's debt is increased at exactly `block.timestamp == market.maturity`, violating the protocol invariant `"debt_increase_after_maturity_forbidden": true` and the explicit boundary rule `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"`. The seller's debt grows at maturity without the continuous-fee accrual that would normally accompany pre-maturity borrowing, and the position becomes immediately subject to post-maturity liquidation in the next block with no recourse.

### Likelihood Explanation
Any block with `block.timestamp == market.maturity` is sufficient. On PoS chains validators can influence `block.timestamp` within a small window; a taker can also simply submit the transaction and have it land in the correct block. The precondition (a sell offer with `expiry >= maturity`) is normal — makers routinely set expiry to maturity or beyond. The attack is repeatable for every market whose maturity falls on a block timestamp.

### Recommendation
Change the guard from `<=` to `<`:

```solidity
// src/Midnight.sol:391
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns with the post-maturity liquidation check at line 622 (`block.timestamp > market.maturity`), which already uses strict inequality, and with `zeroFloorSub`'s own strict-greater-than semantics.

### Proof of Concept

```solidity
function testDebtIncreaseAtExactMaturity() public {
    uint256 units = 100e18;

    // Seller (borrower) has no existing credit.
    collateralize(market, borrower, units);

    // Sell offer with expiry == maturity.
    borrowerOffer.maxUnits = units;
    borrowerOffer.expiry   = market.maturity;

    // Warp to exactly maturity.
    vm.warp(market.maturity);

    uint256 debtBefore = midnight.debtOf(id, borrower);

    // Should revert with CannotIncreaseDebtPostMaturity — but does NOT.
    take(units, lender, borrowerOffer);

    uint256 debtAfter = midnight.debtOf(id, borrower);

    // This assertion FAILS under the current code, proving the bug.
    assertEq(debtAfter, debtBefore, "debt must not increase at maturity boundary");
}
```
