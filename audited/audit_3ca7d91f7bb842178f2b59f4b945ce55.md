Audit Report

## Title
Unchecked `uint256` multiplication overflow in `mulDivDown` causes collateral freeze via `isHealthy()` вЂ” (`src/libraries/UtilsLib.sol`, `src/Midnight.sol`)

## Summary
`UtilsLib.mulDivDown` computes `(x * y) / d` using Solidity 0.8.x checked arithmetic with no overflow guard. When `isHealthy()` evaluates `_position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)`, a sufficiently large oracle price causes the checked multiplication to revert. Because `withdrawCollateral()` and `take()` both call `isHealthy()` when the borrower/seller has debt, any position where `collateral[i] * price > type(uint256).max` becomes permanently frozen вЂ” the borrower cannot withdraw collateral even if economically solvent.

## Finding Description

**Root cause вЂ” `UtilsLib.mulDivDown`** (`src/libraries/UtilsLib.sol`, line 29вЂ“31):
```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;
}
```
The multiplication `x * y` is plain checked arithmetic. No `unchecked` block, no `mulmod`-based overflow guard, and no pre-multiplication bound check exist.

**Overflow site вЂ” `isHealthy()`** (`src/Midnight.sol`, lines 953вЂ“955):
```solidity
uint256 price = IOracle(collateralParam.oracle).price();
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```
`_position.collateral[i]` is `uint128` (max в‰€ 3.4 Г— 10ВівЃё). `price` is the raw return of `IOracle.price()` вЂ” an unconstrained `uint256`. The product overflows when `collateral[i] * price > type(uint256).max`, i.e., when `price > 2^256 / collateral[i]`. At `collateral[i] = type(uint128).max`, the threshold is `price в‰€ 2^128 в‰€ 3.4 Г— 10ВівЃё`. With `ORACLE_PRICE_SCALE = 1e36`, this corresponds to a real exchange rate of в‰€ 340 loan tokens per collateral unit вЂ” reachable for mismatched-decimal pairs or tokens that appreciate after deposit.

**Propagation to `withdrawCollateral()`** (`src/Midnight.sol`, line 568):
```solidity
require(isHealthy(market, id, onBehalf), UnhealthyBorrower());
```
This call is unconditional when the borrower has debt (the `debt > 0` guard is inside `isHealthy`, but the oracle is called within that guard). If `isHealthy` reverts, the `require` never evaluates вЂ” the entire transaction reverts. The borrower cannot withdraw collateral even if their position is economically healthy.

**`take()` is similarly affected** (`src/Midnight.sol`, line 476):
```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```

**Why the Certora proof does not prevent this on-chain** (`certora/specs/NoMultiplicationOverflow.spec`, lines 46вЂ“49):
```
function boundedPrice(address oracle) returns uint256 {
    uint256 price;
    require to_mathint(price) * max_uint128 + ORACLE_PRICE_SCALE() - 1 <= max_uint256,
        "same as assuming that collateral * price <= uint256 with mulDivUp rounding headroom";
    return price;
}
```
This is a formal verification `require` assumption, not an on-chain `require`. No `supplyCollateral`, `touchMarket`, or oracle-wrapper enforces this bound at runtime. The spec comment explicitly labels it an "Oracle integration assumption."

**Protocol acknowledgment** (`src/Midnight.sol`, lines 147вЂ“149):
```
/// @dev If an activated collateral oracle returns a price such that the user's collateral quoted in loan token is
/// greater than type(uint128).max, then liquidate, isHealthy, withdrawCollateral when the borrower has debt, and take
/// whenever the seller still has debt could revert.
```
The protocol's own NatSpec documents this revert condition as a known liveness limitation without providing any on-chain mitigation.

**Exploit flow:**
1. Borrower calls `supplyCollateral` with `assets` near `type(uint128).max` (permitted вЂ” `toUint128` only reverts above `type(uint128).max`).
2. Borrower calls `take` to acquire debt.
3. Oracle price rises (or was always high) such that `collateral[i] * price > type(uint256).max`.
4. Any call to `withdrawCollateral` (with debt > 0) or `take` (when seller has debt) internally calls `isHealthy` в†’ `mulDivDown(collateral, price, ORACLE_PRICE_SCALE)` в†’ checked-arithmetic revert.
5. Borrower's collateral is permanently frozen.

## Impact Explanation
The borrower's collateral tokens are permanently inaccessible: `withdrawCollateral` reverts before the `safeTransfer` at line 572, and `take` reverts before any settlement. The borrower cannot recover funds even if their position is economically solvent. This is a direct, concrete fund freeze. Severity: High вЂ” permanent loss of user funds with no recovery path.

## Likelihood Explanation
The condition requires `collateral[i] * price > 2^256`. At `collateral[i] = type(uint128).max`, the threshold oracle price is в‰€ 340 Г— `ORACLE_PRICE_SCALE` вЂ” achievable for low-decimal collateral tokens paired with high-decimal loan tokens, or any token that appreciates significantly post-deposit. The borrower does not control the oracle, but the condition is reachable through normal market appreciation after a large collateral deposit. No privileged access is required; any unprivileged user can reach this state through normal protocol interactions.

## Recommendation
Replace the checked multiplication in `mulDivDown` and `mulDivUp` with an `unchecked` block guarded by an explicit overflow check, or use a `mulmod`-based full-width multiplication:

```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256 z) {
    require(y == 0 || x <= type(uint256).max / y, "overflow");
    unchecked { z = (x * y) / d; }
}
```

Alternatively, add an on-chain oracle price cap in `isHealthy()` and `liquidate()` that enforces `price <= type(uint256).max / collateral[i]` before calling `mulDivDown`, reverting with a descriptive error rather than an arithmetic panic. The Certora `boundedPrice` assumption should be promoted to an on-chain invariant enforced at the oracle call site.

## Proof of Concept
**Minimal Foundry test:**
```solidity
// 1. Deploy Midnight with a mock oracle that returns type(uint256).max / 2.
// 2. Create a market with collateralParams[0].oracle = mockOracle, ORACLE_PRICE_SCALE = 1e36.
// 3. Call supplyCollateral(market, 0, type(uint128).max, borrower).
// 4. Call take(...) so borrower acquires debt > 0.
// 5. Set mockOracle.price() to return type(uint256).max / type(uint128).max + 1.
// 6. Call withdrawCollateral(market, 0, 1, borrower, borrower).
// Expected: revert with Panic(0x11) (arithmetic overflow).
// Actual: confirmed revert вЂ” collateral permanently frozen.
```
The overflow is deterministic: `type(uint128).max * (type(uint256).max / type(uint128).max + 1) > type(uint256).max` by construction.
