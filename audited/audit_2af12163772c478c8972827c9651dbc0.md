Audit Report

## Title
`consumableUnits` called outside `try/catch` allows out-of-range tick to DoS entire bundle - (File: `src/periphery/MidnightBundles.sol`)

## Summary
All four bundle entry points in `MidnightBundles` evaluate `ConsumableUnitsLib.consumableUnits()` as an argument to `min(...)` before the `try` keyword. When an offer has `maxUnits == 0` and `maxAssets > 0`, `consumableUnits` delegates to `TakeAmountsLib`, which calls `TickLib.tickToPrice(offer.tick)` unconditionally. Any tick exceeding `MAX_TICK = 5820` causes `tickToPrice` to revert with `TickOutOfRange`, propagating outside the `try/catch` and reverting the entire bundle transaction, DoS-ing all legitimate takes.

## Finding Description

**`consumableUnits` is evaluated before `try` in all four bundle functions:**

In `buyWithUnitsTargetAndWithdrawCollateral`, the `min(...)` call at lines 74–78 is a Solidity expression evaluated before the `try` at line 79. Any revert inside it propagates to the caller, not into the `catch` block. [1](#0-0) 

The same pattern appears in `supplyCollateralAndSellWithUnitsTarget` (lines 147–151), `buyWithAssetsTargetAndWithdrawCollateral` (lines 208–214), and `supplyCollateralAndSellWithAssetsTarget` (lines 285–291). The NatSpec at line 43 explicitly acknowledges this: `"Reverts if ConsumableUnitsLib reverts"`. [2](#0-1) 

**`consumableUnits` branches into `TakeAmountsLib` when `maxUnits == 0`:**

When `offer.maxUnits == 0`, `consumableUnits` calls `TakeAmountsLib.buyerAssetsToUnits` or `TakeAmountsLib.sellerAssetsToUnits`. [3](#0-2) 

**Both `TakeAmountsLib` functions call `tickToPrice` unconditionally as their first statement:** [4](#0-3) [5](#0-4) 

**`tickToPrice` reverts for any tick exceeding `MAX_TICK = 5820`:** [6](#0-5) [7](#0-6) 

**Existing guards are insufficient:**

The only tick validation in the protocol (`require(offer.tick % _marketState.tickSpacing == 0, TickNotAccessible())`) lives inside `IMidnight.take()`, which is inside the `try` block. It does not execute before `consumableUnits` is called. There is no on-chain validation of `offer.tick` prior to the `min(...)` expression.

**Exploit flow:**
1. Attacker (maker) signs an offer off-chain with `offer.tick = 5821` (any value > 5820) and `offer.maxAssets > 0` (ensuring `offer.maxUnits == 0`).
2. The poisoned offer is included anywhere in a taker's `takes[]` array alongside legitimate offers.
3. When the loop reaches the poisoned offer, `consumableUnits` → `buyerAssetsToUnits`/`sellerAssetsToUnits` → `tickToPrice(5821)` reverts with `TickOutOfRange`.
4. The revert propagates outside the `try/catch`, reverting the entire bundle transaction and failing all legitimate takes.

## Impact Explanation
Complete DoS of any bundle transaction containing the poisoned offer. All legitimate takes in the bundle fail atomically. In time-sensitive contexts (near maturity, liquidation windows, volatile markets), the taker suffers real financial loss from failed execution. The attack is repeatable at zero cost to the attacker.

## Likelihood Explanation
Any maker can sign an offer with an out-of-range tick at zero cost (off-chain signing, no gas, no on-chain state, no privileged access). The only precondition is that the poisoned offer appears in the `takes` array with `maxAssets > 0`. Automated routing or aggregation systems that fetch offers from an off-chain order book are the primary target and cannot distinguish a poisoned offer from a legitimate one without pre-validating `offer.tick` client-side. The attack is trivially repeatable.

## Recommendation
Wrap the `consumableUnits` (and `TakeAmountsLib`) calls inside the `try/catch` block, or add a pre-check that skips any offer whose `tick > MAX_TICK` before calling `consumableUnits`. For example, in each loop iteration, guard with:

```solidity
if (takes[i].offer.tick > MAX_TICK) continue;
```

before the `min(...)` expression, so a poisoned offer is silently skipped rather than reverting the bundle. Alternatively, restructure the code so that `consumableUnits` is called inside the `try` block itself.

## Proof of Concept
1. Deploy `MidnightBundles` pointing to a live or forked `Midnight` instance.
2. Construct a `takes[]` array with two entries: one legitimate offer and one poisoned offer with `offer.tick = 5821` and `offer.maxAssets = 1` (so `maxUnits == 0`).
3. Call `buyWithUnitsTargetAndWithdrawCollateral` with this array.
4. Observe the transaction reverts with `TickOutOfRange` rather than skipping the poisoned offer and executing the legitimate one.
5. Confirm by removing the poisoned offer: the legitimate take succeeds.
