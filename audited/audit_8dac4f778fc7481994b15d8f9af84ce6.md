### Title
Missing Zero Address Check on `collateralParams[i].oracle` Enables Permanent Collateral Lock and Broken Liquidations — (File: src/Midnight.sol)

---

### Summary

The `touchMarket` function in `src/Midnight.sol` validates several `CollateralParams` fields (token sort order, lltv, maxLif) but never checks that `collateralParams[i].oracle != address(0)`. Because market creation is fully permissionless, any actor can register a market with a zero oracle address. Once a borrower opens a position in such a market, every code path that calls `IOracle(address(0)).price()` reverts due to empty returndata ABI-decoding failure, permanently locking the borrower's collateral and making liquidation impossible.

---

### Finding Description

**Root cause — `touchMarket`, lines 762–773:**

```solidity
for (uint256 i = 0; i < market.collateralParams.length; i++) {
    address collateralToken = market.collateralParams[i].token;
    require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
    uint256 lltv = market.collateralParams[i].lltv;
    require(isLltvAllowed(lltv), LltvNotAllowed());
    require(
        market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_LOW)
            || market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_HIGH),
        InvalidMaxLif()
    );
    previousCollateralToken = collateralToken;
    // ← no require(market.collateralParams[i].oracle != address(0))
}
``` [1](#0-0) 

The `token` field is implicitly protected (must be `> address(0)` due to the sort check), but `oracle` has no analogous guard.

**Downstream failure — `isHealthy`, line 953:**

```solidity
uint256 price = IOracle(collateralParam.oracle).price();
``` [2](#0-1) 

In Solidity 0.8.34, a high-level external call to `address(0)` succeeds at the EVM level (no code → CALL returns success with empty returndata), but the ABI decoder then attempts to decode 0 bytes as `uint256`. Because the returndata is shorter than the expected 32 bytes, the decoder reverts unconditionally.

The same revert occurs in `liquidate` at line 610:

```solidity
uint256 price = IOracle(_collateralParam.oracle).price();
``` [3](#0-2) 

**Affected call paths once a borrower has non-zero debt:**

| Function | Oracle call site | Effect |
|---|---|---|
| `isHealthy` | line 953 | reverts |
| `liquidate` | line 610 | reverts |
| `withdrawCollateral` | calls `isHealthy` → line 568 | reverts |
| `take` (seller with debt) | calls `isHealthy` → line 476 | reverts | [4](#0-3) [5](#0-4) 

The only escape for a trapped borrower is to repay the full debt (which does not call the oracle), then withdraw collateral. If the borrower cannot repay — e.g., they already spent the loan tokens — the collateral is permanently locked with no liquidation path.

---

### Impact Explanation

- **Permanent collateral lock**: A borrower who cannot repay in full loses access to their collateral forever; no liquidation can recover it.
- **Liquidation DoS**: The entire liquidation mechanism for the market is bricked; bad debt cannot be realized, and lenders' credit cannot be redeemed through the normal liquidation flow.
- **Lender credit stranded**: With liquidations reverting and no withdrawable balance accruing, lenders in the market cannot recover their principal.

Severity: **High** — direct, unrecoverable loss of user funds with no admin override (the protocol is immutable).

---

### Likelihood Explanation

Market creation is fully permissionless; no privileged role is required. An attacker can:

1. Deploy a market with `oracle = address(0)` at zero cost beyond gas.
2. Post a competitive buy offer (lend) to attract borrowers.
3. Once a victim borrows and supplies collateral, the trap is set.

The attacker does not need any keys, governance access, or leaked credentials. The only friction is convincing users to interact with the malicious market, which is realistic given that off-chain UIs and aggregators may not surface raw oracle addresses.

---

### Recommendation

Add an explicit non-zero check for `oracle` inside the `touchMarket` validation loop, mirroring the implicit non-zero guarantee already enforced on `token`:

```solidity
require(market.collateralParams[i].oracle != address(0), ZeroOracle());
```

Place it alongside the existing `lltv` and `maxLif` checks at lines 765–771 of `src/Midnight.sol`. [6](#0-5) 

---

### Proof of Concept

**Setup:**
1. Deploy `Midnight`.
2. Construct a `Market` with `collateralParams[0].oracle = address(0)`, valid `lltv`, valid `maxLif`, and a real `loanToken`.
3. Call `touchMarket(market)` — succeeds, market is created.

**Trap:**
4. Attacker posts a buy offer (lend) in this market.
5. Victim calls `take(offer, ...)` as seller (borrows), supplying collateral. `take` succeeds because `isHealthy` is only called at the end for the seller, and at that point the seller has debt — `IOracle(address(0)).price()` is called and **reverts**.

Wait — `take` itself reverts at line 476 (`require(... || isHealthy(...))`), so the victim cannot even open a position via `take`. However, the victim **can** open a position by calling `repay` with `units = 0` to touch the market, then having the maker side of a sell offer execute. More directly: if the victim is the **maker** of a sell offer (i.e., `offer.buy = true`, victim is seller/maker), the `isHealthy` check at line 476 applies to the seller (victim). This reverts immediately.

**Revised direct impact path:**
- Any borrower who already has debt in the market (e.g., entered before the oracle was zeroed out, or via a multi-step flow) cannot call `withdrawCollateral`, cannot be `liquidate`d, and cannot participate in further `take`s as seller.
- The permanent lock and liquidation DoS are the concrete, irreversible impacts. [5](#0-4) [7](#0-6)
