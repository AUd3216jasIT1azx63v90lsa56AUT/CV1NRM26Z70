Audit Report

## Title
Zero-rounding in `maxAssets` branch allows unbounded offer replay with free credit minting and no token transfer — (`src/Midnight.sol`, `src/libraries/UtilsLib.sol`)

## Summary
When a buy offer has `maxAssets > 0` and the taker supplies `units` small enough that `units * buyerPrice < WAD`, `mulDivDown` returns zero and `consumed` is incremented by zero. The `require(newConsumed <= offer.maxAssets)` check passes unchanged on every call, so the offer is never exhausted. Meanwhile, position state mutates normally — the maker accrues credit and the taker accrues debt — with zero token transfers, creating unbacked credit and violating the protocol solvency invariant.

## Finding Description

**Root cause:** `mulDivDown(units, buyerPrice, WAD)` performs integer division `(units * buyerPrice) / WAD`. When `units * buyerPrice < WAD`, the quotient is 0. The `maxAssets` branch increments `consumed` by this zero value, so the cap check `require(newConsumed <= offer.maxAssets)` is trivially satisfied on every call.

**Exact code path:**

`src/libraries/UtilsLib.sol` lines 29–31:
```solidity
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;   // returns 0 when x*y < d
}
```

`src/Midnight.sol` lines 363–369:
```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // → 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    //                                                              ^^^^^^^^^^^ += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // passes unchanged
}
```

Lines 382–414 compute `buyerCreditIncrease` and `sellerDebtIncrease` from `units` (not `buyerAssets`), so position state mutates with non-zero values even when `buyerAssets = 0`.

Lines 455–456 execute `safeTransferFrom(..., 0)` — zero-amount transfers succeed on standard ERC-20 tokens.

**Exploit flow:**
1. Attacker controls address A (maker) and address B (taker, satisfying `maker != taker`).
2. A posts a buy offer with `maxAssets = M > 0` at a low tick (e.g., tick 0, `offerPrice ≈ 1e12`).
3. B calls `take(offer, ..., units = U)` where `U < WAD / offerPrice` (e.g., `U = 999_999` at `offerPrice = 1e12`).
4. `buyerAssets = mulDivDown(999_999, 1e12, 1e18) = 0`.
5. `consumed += 0` → `consumed` unchanged; `require(consumed <= M)` passes.
6. A's credit increases by `U`; B's debt increases by `U`.
7. Both token transfers are for 0 tokens.
8. Steps 3–7 repeat indefinitely; `consumed` never reaches `M`.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` enforces an upper bound but does not require `consumed` to have increased.
- There is no `require(units == 0 || buyerAssets > 0)` guard.
- The `maxUnits` branch (line 371) uses `+= units` and is immune; only the `maxAssets` branch is affected.

## Impact Explanation
Each iteration grants the maker (A) `U` units of credit with zero token payment and burdens the taker (B) with `U` units of debt with zero token receipt. Repeated indefinitely, A accumulates unbounded credit not backed by any token deposit. When A redeems this credit, the protocol pays out tokens deposited by legitimate users, directly draining the contract's loan-token balance. This is a direct protocol insolvency / theft-of-funds impact: the contract's token balance no longer covers outstanding credit obligations.

## Likelihood Explanation
**Preconditions:** A buy offer with `maxAssets > 0` must exist at any tick below the maximum (true for virtually all real offers). The attacker needs two addresses (trivially created). No admin access, oracle manipulation, or victim cooperation is required.

**Feasibility:** The required `units` value is computed off-chain as `floor((WAD - 1) / offerPrice)`. At `offerPrice = 1e12` (tick 0), `U = 999_999`. At higher ticks the window narrows but remains non-zero for any tick where `offerPrice < WAD` (i.e., all valid ticks).

**Repeatability:** Unlimited — `consumed` never advances, so the offer never closes.

## Recommendation
Add a guard in the `maxAssets` branch requiring that the asset increment is non-zero whenever `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetIncrement = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetIncrement > 0, ZeroAssetIncrement());
    newConsumed = consumed[offer.maker][offer.group] += assetIncrement;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce a minimum `units` value such that `units * min(buyerPrice, sellerPrice) >= WAD`, or switch the `maxAssets` branch to use `mulDivUp` for the consumed increment (matching the rounding direction used for the seller in non-buy offers).

## Proof of Concept
**Minimal Foundry test outline:**
```solidity
// Setup: market with loanToken, tick = 0 (offerPrice ≈ 1e12)
// Maker A: posts buy offer, maxAssets = 1e18, maxUnits = 0
// Taker B: calls take(offer, ..., units = 999_999) in a loop N times
// Assert after N iterations:
//   consumed[A][group] == 0  (never advanced)
//   position[id][A].credit == N * 999_999  (unbounded free credit)
//   position[id][B].debt   == N * 999_999  (unbounded debt, no tokens received)
//   loanToken.balanceOf(address(midnight)) unchanged  (no tokens transferred)
// Then A withdraws credit → protocol pays out tokens it never received
```
