### Title
Fee-on-transfer collateral token inflates stored balance, enabling undercollateralized borrow - (`src/libraries/SafeTransferLib.sol`, `src/Midnight.sol`)

### Summary
`supplyCollateral` records the caller-supplied `assets` value directly into `_position.collateral[collateralIndex]` before calling `SafeTransferLib.safeTransferFrom`, which performs no balance-before/after check. When the collateral token is fee-on-transfer, only `assets - fee` arrives at the contract while the full `assets` is stored. `isHealthy` then computes `maxDebt` from the inflated stored value, allowing the borrower to take debt that exceeds the actual on-chain collateral value.

**Market creation is permissionless.** `touchMarket` validates only: maturity bound, non-empty/sorted collateral params, allowed LLTV tiers, and valid `maxLif`. There is no restriction on the token type — any ERC20, including fee-on-transfer tokens, is accepted: [4](#0-3) 

The Certora formal verification explicitly models tokens as "well-behaved" and does not cover fee-on-transfer behavior: [5](#0-4) 

**Exploit flow:**

1. Attacker (unprivileged market creator/borrower) calls `touchMarket` with a market whose `collateralParams[i].token` is a fee-on-transfer ERC20 (e.g., 1% fee). This succeeds because `touchMarket` has no token-type restriction.
2. Attacker calls `supplyCollateral(market, i, 1000e18, attacker)`. Storage records `_position.collateral[i] = 1000e18`, but only `990e18` arrives (1% fee).
3. Attacker repeats for other collateral indices (normal tokens) to satisfy the multi-collateral setup.
4. Attacker calls `take` (sell offer) to borrow. `isHealthy` is called internally and computes `maxDebt` using the stored `1000e18` for the fee-on-transfer collateral, not the actual `990e18`. The position passes the health check.
5. The attacker has borrowed against `10e18` of phantom collateral value per supply call. The contract's actual token balance is less than what the accounting implies.

### Impact Explanation

The borrower holds a position where `debt > actual_collateral_value * lltv / WAD`. The solvency invariant — that the contract's token balances cover all collateral obligations — is violated for the fee-on-transfer collateral token. If the borrower defaults, lenders absorb a loss equal to `fee * lltv` per unit of fee-on-transfer collateral supplied, constituting concrete bad debt. The discrepancy compounds with each additional `supplyCollateral` call on the same index.

### Likelihood Explanation

**Preconditions:**
- A fee-on-transfer ERC20 token exists (USDT historically had this; many DeFi tokens implement it).
- The attacker creates or uses a market that includes such a token as a collateral param.
- Market creation is fully permissionless; no admin approval is required.

**Feasibility:** High. The attacker controls market creation and collateral supply amounts. The exploit is deterministic and repeatable — each `supplyCollateral` call with `assets = X` on a fee-on-transfer token inflates stored collateral by exactly `fee(X)`. No oracle manipulation, no governance action, and no user mistake is required.

### Recommendation

In `supplyCollateral`, measure the actual received amount using a balance-before/after pattern and credit only the delta:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

The state write must move to after the transfer. Alternatively, document that fee-on-transfer tokens are explicitly unsupported and add a revert guard (e.g., require `received == assets`), which is simpler and avoids the ordering change.

### Proof of Concept

```solidity
// Foundry unit test sketch
function testFeeOnTransferCollateralInflatesPosition() public {
    // 1. Deploy a 1%-fee-on-transfer ERC20 as collateral token.
    FeeToken feeToken = new FeeToken(100); // 1% fee

    // 2. Create market with feeToken as collateral[0], normal token as collateral[1].
    Market memory m = buildMarket(address(feeToken), address(normalToken));
    bytes32 id = midnight.touchMarket(m);

    // 3. Supply 1000e18 of feeToken; only 990e18 arrives.
    feeToken.mint(attacker, 1000e18);
    vm.prank(attacker);
    feeToken.approve(address(midnight), 1000e18);
    vm.prank(attacker);
    midnight.supplyCollateral(m, 0, 1000e18, attacker);

    // 4. Assert stored collateral is inflated vs actual balance.
    assertEq(midnight.collateral(id, attacker, 0), 1000e18);           // stored: 1000e18
    assertEq(feeToken.balanceOf(address(midnight)), 990e18);            // actual: 990e18

    // 5. Borrow up to maxDebt computed from stored (inflated) value.
    uint256 maxDebt = 1000e18 * price / ORACLE_PRICE_SCALE * lltv / WAD;
    collateralize(m, attacker, maxDebt); // supply normal collateral to pass health check for debt
    // Take debt = maxDebt (which includes phantom 10e18 * lltv contribution).

    // 6. Assert isHealthy returns true but actual collateral value < debt.
    assertTrue(midnight.isHealthy(m, id, attacker));
    uint256 actualMaxDebt = 990e18 * price / ORACLE_PRICE_SCALE * lltv / WAD
                          + normalCollateral * normalPrice / ORACLE_PRICE_SCALE * lltv / WAD;
    assertGt(midnight.debtOf(id, attacker), actualMaxDebt); // undercollateralized
}
```

**Expected assertions:** `midnight.collateral(id, attacker, 0) > feeToken.balanceOf(address(midnight))` and `debtOf(id, attacker) > actualMaxDebt` both hold, confirming the undercollateralized position with `isHealthy` returning `true`.
