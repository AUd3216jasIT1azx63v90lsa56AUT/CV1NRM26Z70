Audit Report

## Title
Rebasing Collateral Token Negative Rebase Inflates `_position.collateral` Accounting, Blocking Liquidation and Creating Bad Debt - (File: src/Midnight.sol)

## Summary
`supplyCollateral` records the caller-supplied `assets` value directly into `_position.collateral[collateralIndex]` without a balance-before/after reconciliation. When a rebasing collateral token undergoes a negative rebase, the stored accounting permanently exceeds the protocol's actual token holdings. Both `isHealthy` and `liquidate` read the stale stored value, causing insolvent positions to appear healthy and blocking liquidators with `NotLiquidatable()`, while lenders absorb irrecoverable bad debt.

## Finding Description

**Root cause — `supplyCollateral` (src/Midnight.sol:533, 545)**

At line 533, `_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets)` unconditionally records the caller-supplied `assets` parameter. The actual token transfer occurs at line 545 via `safeTransferFrom`. For standard ERC-20 tokens these are equal, but for rebasing tokens the stored value permanently diverges from the real balance after any negative rebase event — there is no snapshot, no balance-diff check, and no hook to reconcile the stored value.

**No token-type restriction in `touchMarket` (src/Midnight.sol:755-791)**

`touchMarket` validates LLTV tiers, `maxLif`, sorted order, and maturity, but imposes no restriction on the ERC-20 type of the collateral token. Any address satisfying the sort order is accepted, including rebasing tokens. Market creation is permissionless — any unprivileged actor can create a market with a rebasing collateral token.

**`isHealthy` uses stored accounting, not actual balance (src/Midnight.sol:944-960)**

`maxDebt` is computed from `_position.collateral[i]` (line 954) — the stale stored value — multiplied by the oracle price and LLTV. After a negative rebase the stored value exceeds the real balance, so `maxDebt` is overestimated and the position appears healthy when it is actually insolvent.

**`liquidate` uses the same stale value (src/Midnight.sol:612-624)**

The liquidatability check `originalDebt > maxDebt` at line 622 uses `_position.collateral[i]` read at line 612 to compute `maxDebt`. If the stored collateral is large enough to make `maxDebt >= debt`, the `NotLiquidatable()` revert fires and no liquidator can touch the position.

**Exploit flow:**
1. Unprivileged actor calls `touchMarket` with a rebasing token (e.g., stETH) as `collateralParams[0].token` — no check prevents this.
2. Borrower calls `supplyCollateral(market, 0, C, borrower)` — `_position.collateral[0]` is set to `C`; Midnight holds `C` stETH.
3. Borrower borrows up to the LLTV limit: `debt = C * price * lltv / ORACLE_PRICE_SCALE`.
4. A negative rebase occurs (e.g., validator slashing): Midnight's actual stETH balance drops to `C' < C`; `_position.collateral[0]` remains `C`.
5. `isHealthy()` returns `true` because `maxDebt = C * price * lltv / ORACLE_PRICE_SCALE >= debt`.
6. Actual collateral value is only `C' * price * lltv / ORACLE_PRICE_SCALE < debt` — position is insolvent.
7. `liquidate` reverts with `NotLiquidatable()` because `debt <= maxDebt` (inflated).
8. Borrower walks away; lenders absorb the bad debt with no recourse.

**Existing protections are insufficient:** `SECURITY.md` excludes "depegging of an external stablecoin," but a negative rebase on stETH from validator slashing is a distinct mechanism — it is a protocol-level balance reduction applied to a liquid staking token, not a stablecoin peg failure. No documentation in the repository restricts rebasing tokens from being used as collateral, and no on-chain guard enforces such a restriction. The Certora formal verification setup notes "ERC20 tokens are assumed well-behaved" as a modeling convention (`certora/README.md:112`), but this is a verification assumption, not an enforced on-chain invariant.

## Impact Explanation
The core accounting invariant `sum(_position.collateral[i]) <= actualTokenBalance` is violated. The borrower's position is reported healthy by `isHealthy()` while the actual collateral is insufficient to cover the debt. Liquidators are permanently blocked by `NotLiquidatable()`. The shortfall becomes irrecoverable bad debt absorbed by lenders, constituting direct, permanent loss of funds for an in-scope user class.

## Likelihood Explanation
All three preconditions are realistic and require no privileged access: (1) market creation with a rebasing token is permissionless via `touchMarket`; (2) borrowing near the LLTV limit is normal user behavior; (3) negative rebases on stETH (validator slashing) and aTokens (protocol losses) are documented, recurring external events. The scenario is repeatable for every market that lists a rebasing token and is independent of any malicious actor — a passive borrower benefits from the accounting divergence without taking any exploitative action.

## Recommendation
Enforce a balance-before/after reconciliation in `supplyCollateral`: record `balanceAfter - balanceBefore` as the actual deposited amount rather than the caller-supplied `assets` parameter. Alternatively, maintain a per-market, per-token aggregate collateral accounting and rebase it against the actual contract balance on each interaction. A complementary measure is to add an explicit on-chain allowlist or denylist for collateral token types in `touchMarket`, rejecting known rebasing tokens. Document clearly that rebasing tokens are unsupported if the balance-diff approach is not adopted.

## Proof of Concept
1. Deploy a mock rebasing ERC-20 token (`MockRebasingToken`) whose `totalSupply` and all balances can be scaled down by a privileged `slash(uint256 factor)` call (simulating validator slashing).
2. Call `touchMarket` with `MockRebasingToken` as the sole collateral, a standard loan token, and a valid LLTV tier.
3. Call `supplyCollateral(market, 0, 1000e18, borrower)` — `_position.collateral[0]` is set to `1000e18`.
4. Call `borrow` to take debt up to the LLTV limit.
5. Call `MockRebasingToken.slash(10)` to reduce all balances by 10% — Midnight's actual balance drops to `900e18`, but `_position.collateral[0]` remains `1000e18`.
6. Assert `isHealthy(market, id, borrower)` returns `true` (it will, because maxDebt is computed from the stale `1000e18`).
7. Attempt `liquidate(...)` — assert it reverts with `NotLiquidatable()`.
8. Confirm the position is economically insolvent: actual collateral value `< debt`, yet liquidation is blocked.
