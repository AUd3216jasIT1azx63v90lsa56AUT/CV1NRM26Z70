### Title
Rounding-to-zero `buyerAssets` allows unlimited re-fills of a fully-consumed assets-capped buy offer — (`src/Midnight.sol`)

### Summary
When a buy offer uses `maxAssets` as its cap and `buyerPrice < WAD`, calling `take` with `units = 1` produces `buyerAssets = 1.mulDivDown(buyerPrice, WAD) = 0`. Because consumed is incremented by `buyerAssets` (not `units`), the cap check `newConsumed <= maxAssets` passes even when the offer is already fully consumed, and the maker receives 1 additional credit unit per call with no asset transfer. The protocol acknowledges this edge case in a NatSpec comment and an existing test named `testBugBuyMaxAssetsBypass` confirms the exact state change.

### Finding Description
**Root cause — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // passes: maxAssets + 0 <= maxAssets
}
``` [1](#0-0) 

The consumed accounting for assets-capped buy offers tracks `buyerAssets`, not `units`. When `buyerPrice < WAD` (any tick below `WAD` after subtracting the settlement fee), a taker can supply `units = 1` and produce `buyerAssets = 0`. The cap check then becomes `maxAssets + 0 <= maxAssets`, which always passes regardless of how many times it is called.

**Exploit flow:**

1. Maker creates a buy offer with `maxAssets = N` and `tick` such that `buyerPrice < WAD` (e.g., `MAX_TICK - 16` as used in the existing test).
2. Offer is legitimately filled to `consumed[maker][group] == N` (fully consumed).
3. Attacker (any unprivileged taker ≠ maker) calls `take(offer, ..., units=1, ...)`.
4. `buyerAssets = 1.mulDivDown(buyerPrice, WAD) = 0`.
5. `consumed += 0`; check `N <= N` passes.
6. Position accounting proceeds with `units = 1`: maker's credit increases by 1, taker's debt increases by 1, `totalUnits` increases by 1.
7. No tokens are transferred (`buyerAssets = sellerAssets = 0`).
8. Step 3–7 can be repeated indefinitely.

**Why existing checks fail:**

The `require(newConsumed <= offer.maxAssets)` guard is the only cap enforcement for assets-mode offers. It is bypassed whenever `buyerAssets` rounds to zero. The Certora spec rule `takeConsumedAtMaxUnchangedAssets` only asserts that `consumed` does not change — it does not assert that position state (credit/debt/totalUnits) is also unchanged, so the formal verification does not catch this. [2](#0-1) 

The protocol's own NatSpec at line 94 explicitly notes: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* [3](#0-2) 

### Impact Explanation
A fully-consumed assets-capped buy offer can be re-filled an unlimited number of times with `units = 1` at zero asset cost. Each call increases the maker's credit by 1 unit and the taker's debt by 1 unit without any token transfer. The maker accumulates credit beyond their intended `maxAssets` budget; the taker accumulates debt without receiving assets. This directly violates the invariant that *offers cannot be overfilled* and breaks the accounting guarantee that every credit unit has a matching asset payment.

### Likelihood Explanation
**Preconditions:**
- A buy offer exists with `maxAssets > 0` and a tick such that `buyerPrice < WAD` (any tick below the WAD threshold, which is a normal and reachable tick range).
- The offer has been fully consumed (reachable through normal fills).

**Feasibility:** Any unprivileged taker can trigger this with a single `take` call. No special permissions, oracle manipulation, or governance action is required. The taker does incur 1 unit of debt per call with no asset receipt, which is self-harmful, but a griefing attacker or a smart contract indifferent to its own debt can repeat this indefinitely. The attack is also reachable by the maker themselves if they want to inflate their own credit beyond the cap (they cannot be the taker due to `SelfTake`, but a colluding address can act as taker).

**Repeatability:** Unlimited — `consumed` never increases past `maxAssets`, so the bypass condition is permanent once the offer is fully consumed.

### Recommendation
Add a guard that prevents a non-zero `units` take from proceeding when `buyerAssets` rounds to zero for assets-capped buy offers. Concretely, inside the `offer.maxAssets > 0` branch, require that `buyerAssets > 0` whenever `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetNonZeroUnits()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any take with `units > 0` must move the consumed counter forward, making it impossible to bypass the cap through rounding.

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already constitutes a complete, passing Foundry unit test that reproduces the bug: [4](#0-3) 

**Key assertions that confirm the bug:**
- `assertEq(buyerAssets, 0)` — no assets transferred.
- `assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets)` — consumed unchanged, cap not enforced.
- `assertGt(midnight.creditOf(id, lender), lenderCreditBefore)` — maker credit increased beyond cap.
- `assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore)` — taker debt increased with no asset receipt.
- `assertGt(midnight.totalUnits(id), totalUnitsBefore)` — protocol-wide unit supply inflated.

A complementary fuzz/invariant test should assert: for all buy offers with `maxAssets > 0`, after any sequence of `take` calls, `creditOf(maker) - creditBefore <= maxAssets / buyerPrice * WAD` (i.e., credit increase is bounded by the asset cap).

/// If consumed is already at or above maxAssets before a take in assets mode, it remains unchanged.
rule takeConsumedAtMaxUnchangedAssets(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxUnits == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumedBefore >= offer.maxAssets => consumed(offer.maker, offer.group) == consumedBefore;

    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:

        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);

    }
