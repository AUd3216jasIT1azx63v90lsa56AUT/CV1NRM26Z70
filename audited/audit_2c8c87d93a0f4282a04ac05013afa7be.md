Audit Report

## Title
Persistent `isRootRatified` State Survives Operator Authorization Revocation - (File: src/ratifiers/SetterRatifier.sol)

## Summary
`SetterRatifier.setIsRootRatified` gates the write on a point-in-time authorization check but stores the result as permanent on-chain state. `SetterRatifier.isRatified` reads only that stored boolean and never re-checks whether the address that wrote it is still authorized. Revoking an operator's authorization via `Midnight.setIsAuthorized` has no effect on previously written `isRootRatified` entries, so any root ratified by a now-revoked operator remains permanently valid for `take` calls.

## Finding Description
**Root cause**

`SetterRatifier.setIsRootRatified` (lines 24–28) performs a one-time authorization check and writes persistent state:
```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;   // permanent write
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
```

`SetterRatifier.isRatified` (lines 30–37), called by `Midnight` on every `take`, reads only the stored boolean:
```solidity
require(isRootRatified[offer.maker][root], NotRatified());  // no authorization re-check
```

`Midnight.setIsAuthorized` (lines 731–735) writes only to `isAuthorized[onBehalf][authorized]` and has no knowledge of, and makes no changes to, `SetterRatifier.isRootRatified`.

`Midnight.take` (line 355) checks `isAuthorized[offer.maker][offer.ratifier]` — verifying the *ratifier contract* is authorized, not the operator who wrote the root — then calls `isRatified` (line 356). Once a root is written, the ratifier contract check is the only gate, and it is independent of the operator's current authorization status.

**Contrast with `EcrecoverRatifier`**

`EcrecoverRatifier.isRatified` (line 44) re-checks `IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer)` on every call, so revocation immediately blocks future takes. The test `testIsRatifiedRevokeAuthorizationInvalidates` (lines 185–205 of `test/EcrecoverRatifierTest.sol`) explicitly verifies this property. No equivalent test or mechanism exists for `SetterRatifier`.

**Exploit flow**

1. Maker grants operator authorization: `midnight.setIsAuthorized(operator, true, maker)`.
2. Maker also authorizes `SetterRatifier` as ratifier (required for `take` to pass line 355).
3. Operator calls `setterRatifier.setIsRootRatified(maker, maliciousRoot, true)`. Authorization check passes; `isRootRatified[maker][maliciousRoot] = true` is written.
4. Maker revokes operator: `midnight.setIsAuthorized(operator, false, maker)`. Only `isAuthorized[maker][operator]` changes.
5. `isRootRatified[maker][maliciousRoot]` remains `true` — untouched by revocation.
6. Operator (or any colluding party) calls `midnight.take(maliciousOffer, abi.encode(maliciousRoot, leafIndex, proof), ...)`. `SetterRatifier.isRatified` returns `CALLBACK_SUCCESS`; the take executes against the maker's position.

The front-running scenario (step 3 racing step 4 in the mempool) is one path, but the vulnerability is broader: any operator who was ever authorized and wrote a root at any prior time achieves the same persistent impact without any race condition.

**Why existing checks fail**

The authorization guard in `setIsRootRatified` is a gate on the write, not on the read. Once written, `isRatified` has no mechanism to detect that the writer's authorization was subsequently revoked. The maker cannot enumerate which roots the operator ratified without scanning all `SetIsRootRatified` events off-chain, and cannot selectively block a specific malicious root without knowing its exact value and calling `setIsRootRatified(maker, maliciousRoot, false)` themselves. Revoking the `SetterRatifier` contract's authorization entirely would block all takes — including legitimate ones — and is not a targeted remedy.

## Impact Explanation
After the maker's revocation is mined, any `take` supplying a valid Merkle proof against `maliciousRoot` passes `SetterRatifier.isRatified` and executes against the maker's position. This can drain credit, consume offer capacity, or create debt without the maker's current consent and contrary to their explicit revocation. The maker has no targeted on-chain remedy unless they independently discover the exact root value. Impact is direct, unauthorized movement of assets/value from the maker's position.

## Likelihood Explanation
**Preconditions**: The operator must have been authorized at some point — the normal, intended use case for `SetterRatifier`. No leaked keys or external dependencies are required. **Front-running path**: The operator observes the revocation transaction in the mempool and submits `setIsRootRatified` with higher gas; standard on Ethereum. **Non-front-running path**: Any operator who called `setIsRootRatified` with any root during their authorization window retains that root's validity indefinitely, with no race condition required. **Repeatability**: The operator can ratify multiple roots in a single block. The malicious root can be used for takes at any future time.

## Recommendation
Re-check the authorization of the address that originally ratified the root inside `isRatified`, analogous to `EcrecoverRatifier`. One approach: store `(maker => root => ratifier_address)` instead of a plain boolean, then in `isRatified` verify `IMidnight(MIDNIGHT).isAuthorized(offer.maker, storedRatifier)` before returning `CALLBACK_SUCCESS`. Alternatively, add a hook in `Midnight.setIsAuthorized` (or a separate `invalidateRatifiedRoots` function) that clears `isRootRatified` entries when authorization is revoked, though this requires enumerability. The simplest fix matching `EcrecoverRatifier`'s model is to record the authorizing address and re-validate it on every `isRatified` call.

## Proof of Concept
Extend `test/SetterRatifierTest.sol` with:

```solidity
function testIsRatifiedRevokeAuthorizationDoesNotInvalidate() public {
    Offer memory offer = makeOffer(lender);
    bytes32 _root = HashLib.hashOffer(offer);

    // Authorize borrower as operator
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, true, lender);

    // Operator ratifies root while authorized
    vm.prank(borrower);
    setterRatifier.setIsRootRatified(lender, _root, true);

    // Maker revokes operator
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, false, lender);

    // isRatified still succeeds — revocation had no effect
    vm.prank(address(midnight));
    bytes32 result = setterRatifier.isRatified(offer, abi.encode(_root, 0, new bytes32[](0)));
    assertEq(result, CALLBACK_SUCCESS); // passes, demonstrating the vulnerability
}
```

This test will pass (demonstrating the bug) where the analogous `testIsRatifiedRevokeAuthorizationInvalidates` in `EcrecoverRatifierTest.sol` (lines 185–205) expects a revert.
