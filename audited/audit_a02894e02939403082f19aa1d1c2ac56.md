Audit Report

## Title
Authorized party can escalate authorization to arbitrary third addresses and erase own trace — (`src/periphery/EcrecoverAuthorizer.sol` / `src/Midnight.sol`)

## Summary
After a victim-signed `Authorization` message is processed through `EcrecoverAuthorizer`, the newly authorized party satisfies `Midnight`'s `isAuthorized[onBehalf][msg.sender]` guard and can call `Midnight.setIsAuthorized` directly — without any further signature — to grant authorization to an arbitrary second address and then revoke their own. The victim's signed message authorized only the attacker, but the protocol allows that authorization to be used as a key to modify the entire authorization state, violating the invariant that authorization changes must reflect only the authorizer's intent.

## Finding Description

**Root cause:** `Midnight.setIsAuthorized` (line 731–735) guards access with only `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. Any currently-authorized party can therefore call this function to grant or revoke authorization for any address on behalf of the authorizer, with no restriction to the scope expressed in the original signed message.

**Code path:**

1. **Precondition:** Victim calls `Midnight.setIsAuthorized(EcrecoverAuthorizer, true, victim)` to enable the periphery. `isAuthorized[victim][EcrecoverAuthorizer] = true`.

2. **Attacker calls** `EcrecoverAuthorizer.setIsAuthorized(Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N}, sig)` with a valid victim-signed message.
