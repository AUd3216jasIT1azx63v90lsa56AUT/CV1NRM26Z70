### Title
`consumableUnits` reverts unconditionally for buy offers at tick > 2910 with `maxAssets > 0`, causing full bundle DoS — (`src/periphery/TakeAmountsLib.sol`, `src/periphery/ConsumableUnitsLib.sol`, `src/periphery/MidnightBundles.sol`)

### Summary

`TakeAmountsLib.buyerAssetsToUnits` contains a hard `require(buyerPrice <= WAD)` guard that reverts for any buy offer where `tickToPrice(tick) > WAD` (i.e., tick > 2910). `ConsumableUnitsLib.consumableUnits` calls this function unconditionally for buy offers with `maxAssets > 0`, and all `MidnightBundles` functions that process buy offers call `consumableUnits` outside any `try/catch`. An attacker who publishes a buy offer at tick > 2910 with `maxAssets > 0` can cause any bundle that includes that offer to revert entirely, while the core `Midnight.take` imposes no such restriction.

### Finding Description

**Tick range producing `buyerPrice > WAD`:**

`TickLib.tickToPrice` computes:
```
price = 1e36 / (1e18 + wExp(LN_ONE_PLUS_DELTA * (MAX_TICK/2 - tick)))
```
For `tick > MAX_TICK/2 = 2910`, the exponent is negative, `wExp(negative) < 1e18`, so `1e18 + wExp(...) < 2e18`, giving `price > 1e36/2e18 = 5e17`. More precisely, for `tick > 2910`, `wExp(LN_ONE_PLUS_DELTA*(2910-tick)) < 1e18`, which means `price > WAD`. The valid tick range is `[0, 5820]`, so ticks 2911–5820 all produce `tickToPrice > WAD`. [1](#0-0) [2](#0-1) 

**Core `Midnight.take` does not reject these ticks:**

The only tick validation in `take` is `require(offer.tick % _marketState.tickSpacing == 0, TickNotAccessible())`. There is no `buyerPrice <= WAD` check. The NatSpec even acknowledges: "All sellerAssets are reachable with the units input, and all buyerAssets are reachable only if buyerPrice <= WAD." So buy offers at tick > 2910 are valid and takeable in the core. [3](#0-2) [4](#0-3) 

**`buyerAssetsToUnits` reverts unconditionally for these ticks:**

For a buy offer, `sellerPrice = offerPrice - settlementFee` and `buyerPrice = sellerPrice + settlementFee = offerPrice`. The `require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne())` fires for any tick > 2910, regardless of `targetBuyerAssets` (even 0). [5](#0-4) 

**`consumableUnits` propagates the revert without guarding:**

When `offer.maxUnits == 0` and `offer.buy == true`, `consumableUnits` calls `buyerAssetsToUnits` directly. There is no try/catch, no pre-check on `buyerPrice`, and no fallback to returning 0. [6](#0-5) 

**Bundle functions do not catch `consumableUnits` reverts:**

In `supplyCollateralAndSellWithUnitsTarget` (and `supplyCollateralAndSellWithAssetsTarget`), `ConsumableUnitsLib.consumableUnits` is called bare inside the loop, while only the subsequent `IMidnight(MIDNIGHT).take(...)` call is wrapped in `try/catch`. The NatSpec explicitly states: "Reverts if ConsumableUnitsLib reverts." [7](#0-6) [8](#0-7) 

**Exploit flow:**
1. Attacker signs a buy offer with `tick = 5820` (or any tick > 2910), `maxAssets > 0`, `maxUnits = 0`, and publishes it to an off-chain order book.
2. A victim or aggregator building a `supplyCollateralAndSellWithUnitsTarget` bundle includes this offer (it appears to offer the highest price, making it attractive for a seller).
3. The bundle loop reaches this offer and calls `ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)`.
4. `consumableUnits` calls `buyerAssetsToUnits(..., offer.maxAssets.zeroFloorSub(consumed))`.
5. `buyerAssetsToUnits` hits `require(buyerPrice <= WAD)` → reverts with `PriceGreaterThanOne`.
6. The revert propagates out of the loop, out of the bundle function — the entire transaction reverts.

The attacker's cost is negligible (a signed off-chain message). The victim loses gas and fails to execute their intended market action.

### Impact Explanation

Any call to `supplyCollateralAndSellWithUnitsTarget` or `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer at tick > 2910 with `maxAssets > 0` will revert entirely. Since the core protocol allows such offers and they appear economically attractive (high price for the seller), aggregators may automatically include them. This is a concrete, repeatable DoS of the periphery's bundle-based take path for any bundle containing such an offer.

### Likelihood Explanation

The preconditions are fully attacker-controlled: any unprivileged user can sign a buy offer at tick > 2910 with `maxAssets > 0` at zero on-chain cost. The offer is valid in the core. Aggregators that sort offers by price will preferentially include high-tick buy offers, making the poisoning reliable. The attack is repeatable and cheap to sustain.

### Recommendation

In `ConsumableUnitsLib.consumableUnits`, guard the `buyerAssetsToUnits` call with a `buyerPrice <= WAD` pre-check and return 0 if the condition fails (treating the offer as non-consumable via the assets path):

```solidity
} else if (offer.buy) {
    uint256 remaining = offer.maxAssets.zeroFloorSub(consumed);
    uint256 offerPrice = TickLib.tickToPrice(offer.tick);
    uint256 _settlementFee = IMidnight(midnight).settlementFee(id, ...);
    uint256 buyerPrice = offerPrice; // for buy offers
    if (buyerPrice > WAD) return 0;
    return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, remaining);
}
```

Alternatively, wrap the `consumableUnits` call in the bundle functions with a try/catch that returns 0 on revert, consistent with the existing pattern for `take`.

### Proof of Concept

```solidity
// Foundry unit test
function test_consumableUnitsRevertsHighTickBuyOffer() public {
    // Setup: create market, deploy MidnightBundles
    // Create buy offer at tick = 5820 (MAX_TICK), maxAssets = 1e18, maxUnits = 0
    Offer memory offer = Offer({
        market: market,
        buy: true,
        maker: attacker,
        tick: 5820, // tickToPrice(5820) > WAD
        maxAssets: 1e18,
        maxUnits: 0,
        // ... other fields
    });

    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offer, units: 1e18, ratifierData: ""});

    // Assert: bundle reverts with PriceGreaterThanOne
    vm.expectRevert(TickLib.PriceGreaterThanOne.selector);
    bundles.supplyCollateralAndSellWithUnitsTarget(
        1e18, 0, victim, receiver, new CollateralSupply[](0), takes, 0, address(0)
    );

    // Also assert: consumableUnits alone reverts
    vm.expectRevert(TickLib.PriceGreaterThanOne.selector);
    ConsumableUnitsLib.consumableUnits(address(midnight), id, offer);
}
```

Expected assertions: both calls revert with `PriceGreaterThanOne`; the core `midnight.take` with the same offer and a units-based input does NOT revert (confirming the periphery-only nature of the bug).
