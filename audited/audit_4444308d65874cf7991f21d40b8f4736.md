Audit Report

## Title
Taker-Controlled `takerCallback` Can Coerce Maker to Fund Buyer's Payment on Sell Offers - (File: src/Midnight.sol)

## Summary
In `take`, for a sell offer (`offer.buy = false`), `buyerCallback` is derived directly from the taker-supplied `takerCallback` parameter. If a taker passes `takerCallback = offer.maker`, then `payer` is set to `offer.maker`, and both token transfers pull from the maker rather than the taker. The taker acquires a credited position in the protocol at zero cost, provided `offer.maker` is a contract implementing `IBuyCallback` that returns `CALLBACK_SUCCESS` without asserting `buyer == address(this)`.

## Finding Description

**Root cause:** `takerCallback` is an unconstrained taker-supplied parameter. For sell offers, it is assigned directly to `buyerCallback`, which then determines `payer`. [1](#0-0) 

Line 420: `buyerCallback = takerCallback` (for `offer.buy = false`).
Line 422: `payer = buyerCallback` whenever `buyerCallback != address(0)`.

Setting `takerCallback = offer.maker` therefore sets `payer = offer.maker`.

**Callback gate:** Lines 445–452 call `IBuyCallback(buyerCallback).onBuy(...)` and require `CALLBACK_SUCCESS`. [2](#0-1) 

The `onBuy` call passes `buyer = taker` (not `offer.maker`). If `offer.maker`'s `onBuy` implementation does not assert `buyer == address(this)`, it returns `CALLBACK_SUCCESS` and the gate passes.

**Token pulls:** Lines 455–456 then pull `buyerAssets` from `payer = offer.maker`. [3](#0-2) 

The taker's position is credited for `units` with zero token outflow. The maker's token balance is drained by `buyerAssets`.

**Why existing checks fail:**
- `require(offer.maker != taker, SelfTake())` at line 354 does not prevent `takerCallback = offer.maker`.
- The `onBuy` return-value check is the only gate, but it is satisfied by any `IBuyCallback` implementation that does not verify `buyer == address(this)`.
- The Certora `takeOnlyExplicitPayer` rule in `certora/specs/OnlyExplicitPayerCanLoseTokens.spec` explicitly classifies this as valid: it sets `callbackAllowed = true` after `onBuy` returns `CALLBACK_SUCCESS` and `callback = offer.maker`, so `CVL_transferFrom` permits the pull. The spec proves tokens are only pulled from "explicit payers" but does not prove the maker cannot be coerced into that role by the taker. [4](#0-3) 

## Impact Explanation
A taker taking a sell offer with `takerCallback = offer.maker` causes the maker's tokens to be pulled as the buyer's payment. The taker receives a credited position for free. The maker suffers a token loss equal to `buyerAssets` (settlement fee portion burned, `sellerAssets` sent to `offer.receiverIfMakerIsSeller`, but the maker's balance is drained by `buyerAssets` rather than receiving `sellerAssets`). This is a direct theft of maker funds by an unprivileged taker, constituting a high-severity loss-of-funds impact.

## Likelihood Explanation
**Required preconditions:**
1. `offer.maker` is a contract (EOA has no code; `onBuy` call reverts).
2. `offer.maker` implements `IBuyCallback.onBuy` and returns `CALLBACK_SUCCESS` without asserting `buyer == address(this)`.
3. `offer.maker` holds a token allowance to the Midnight contract.

These conditions are realistic for protocol-integrated vault or market-maker contracts that both post sell offers and participate as buyers (e.g., a lending vault with a standing approval). The attack requires no privileged access, is repeatable against any qualifying maker contract, and is executable by any external caller.

## Recommendation
Add a validation in `take` that prevents the taker from setting `takerCallback` to `offer.maker` for sell offers:

```solidity
// For sell offers, takerCallback becomes buyerCallback and payer.
// Prevent the taker from coercing the maker into funding the buyer's payment.
require(!(!offer.buy && takerCallback == offer.maker), TakerCallbackIsMaker());
```

Alternatively, decouple `payer` from `buyerCallback` for sell offers: always require the taker (`msg.sender`) to be the payer unless an explicit, taker-owned callback is used, and add a check that `takerCallback != offer.maker`.

## Proof of Concept

**Minimal Foundry test outline:**

1. Deploy a `MakerVault` contract that:
   - Implements `IBuyCallback.onBuy` and returns `CALLBACK_SUCCESS` unconditionally (no `buyer == address(this)` check).
   - Holds `buyerAssets` worth of `loanToken` and has approved Midnight for the full amount.
2. Create a sell offer (`offer.buy = false`) signed by `MakerVault` as `offer.maker`.
3. As an unprivileged taker, call:
   ```solidity
   midnight.take(offer, ratifierData, units, taker, receiver, address(makerVault), "");
   //                                                              ^^^^^^^^^^^^^^^^
   //                                                              takerCallback = offer.maker
   ```
4. Assert:
   - `loanToken.balanceOf(address(makerVault))` decreased by `buyerAssets`.
   - `midnight.creditOf(id, taker)` increased by `units`.
   - `loanToken.balanceOf(taker)` is unchanged (taker paid nothing).
