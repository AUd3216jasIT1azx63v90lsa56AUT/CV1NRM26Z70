### Title
Buy offer `maxAssets` cap bypassed via zero-rounding `mulDivDown` when `buyerPrice < WAD` - (`File: src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and the offer's `buyerPrice < WAD` (achievable by choosing a tick where `tickToPrice(tick) < WAD`), calling `take()` with `units=1` after the cap is fully consumed computes `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. The consumed tracking adds zero, the cap check passes, but the full `units=1` position mutation still executes — granting the buyer unbounded extra credit and the seller unbounded extra debt with no asset transfer. This is confirmed by the existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol`.

### Finding Description

**Code path** (`src/Midnight.sol`):

```
take()
  buyerAssets = units.mulDivDown(buyerPrice, WAD)   // line 363
  consumed[maker][group] += buyerAssets              // line 368
  require(newConsumed <= offer.maxAssets)            // line 369
  ...
  buyerPos.credit += buyerCreditIncrease             // line 410  (units=1 → +1)
  sellerPos.debt   += sellerDebtIncrease             // line 414  (units=1 → +1)
  totalUnits       += buyerCreditIncrease            // line 416-417
```

**Root cause:** The consumed guard tracks `buyerAssets` (an asset-denominated quantity), but `mulDivDown(1, buyerPrice, WAD)` returns **0** whenever `buyerPrice < WAD`, because `1 × buyerPrice < WAD`. The position mutation, however, always operates on the raw `units` value, not on `buyerAssets`. These two quantities are decoupled when rounding produces zero.

**Attacker-controlled inputs:**
- `offer.tick` — chosen so that `tickToPrice(tick) < WAD` (e.g., `MAX_TICK - 16` as used in the test).
- `units = 1` — the minimal value that rounds `buyerAssets` to zero.
- `offer.maxAssets` — any positive value; the attack works once `consumed` reaches `maxAssets`.

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = N`, `tick` s.t. `buyerPrice < WAD`, ratified via `SetterRatifier` (`isRootRatified[maker][root] = true`).
2. Taker (or maker via `setConsumed`) fills the offer to exactly `consumed = N`.
3. Taker calls `take(units=1, ...)`:
   - `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`
   - `newConsumed = N + 0 = N` → passes `require(newConsumed <= maxAssets)`
   - `buyerPos.credit += 1`, `sellerPos.debt += 1`, `totalUnits += 1`
4. Step 3 is repeatable indefinitely; each call adds 1 unit of credit/debt at zero asset cost.

**Why existing checks fail:** The only guard is `require(newConsumed <= offer.maxAssets)` at line 369. Because `buyerAssets = 0`, `newConsumed` never exceeds `maxAssets`, so the check never triggers regardless of how many times the call is repeated. [1](#0-0) [2](#0-1) 

### Impact Explanation

An unprivileged taker can accumulate an arbitrary amount of credit (and impose an arbitrary amount of debt on the seller) beyond the maker's intended `maxAssets` cap, at zero token cost. The buyer's credit balance grows without any matching asset inflow, breaking the invariant that every credit has a matching debt or valid settled/loss state and that offers cannot be overfilled. [3](#0-2) 

### Likelihood Explanation

**Preconditions:**
- A buy offer exists with `maxAssets > 0` and a tick where `tickToPrice(tick) < WAD` (the comment at line 333–334 of `Midnight.sol` explicitly acknowledges that `buyerPrice <= WAD` is the normal operating range).
- The offer's consumed value has reached `maxAssets` (achievable by the taker themselves in a prior call, or by the maker via `setConsumed`).

Both conditions are easily reachable by any unprivileged taker. The attack is repeatable in a loop within a single transaction via `multicall`. No special privileges, oracle manipulation, or admin action is required. [4](#0-3) 

### Recommendation

Replace the consumed increment with a `units`-based guard when `buyerAssets` rounds to zero, or require `buyerAssets > 0` before processing any position mutation when `maxAssets > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assets = offer.buy ? buyerAssets : sellerAssets;
    require(assets > 0, ZeroAssets()); // prevent zero-rounding bypass
    newConsumed = consumed[offer.maker][offer.group] += assets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, track consumed in `units` even for asset-capped offers and convert the cap to units at offer creation, eliminating the rounding gap entirely. [5](#0-4) 

### Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already constitutes a passing PoC. A fuzz/invariant extension:

```solidity
// Invariant: consumed[maker][group] tracks all units taken, not just asset-rounded ones.
// Fuzz test plan:
function testFuzz_maxAssetsBypassRepeat(uint256 repeatCount) public {
    repeatCount = bound(repeatCount, 1, 1000);

    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

    uint256 creditBefore = midnight.creditOf(id, lender);

    for (uint256 i = 0; i < repeatCount; i++) {
        take(1, borrower, lenderOffer);
    }

    // Assert: consumed must not exceed maxAssets
    assertLe(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
    // Assert: credit must not have increased (offer was already full)
    assertEq(midnight.creditOf(id, lender), creditBefore);
    // Both assertions FAIL with the current code, confirming the bug.
}
```

Expected: both assertions fail, demonstrating unbounded credit accumulation past the `maxAssets` cap. [2](#0-1) [1](#0-0)
