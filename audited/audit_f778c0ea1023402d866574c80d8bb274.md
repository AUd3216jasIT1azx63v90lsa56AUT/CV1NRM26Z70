### Title
Fee-on-Transfer loanToken Causes `claimableSettlementFee` Overstatement Leading to Protocol Insolvency - (File: src/Midnight.sol)

### Summary
In `Midnight.take`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full nominal spread `buyerAssets - sellerAssets` before the inbound transfer, but `SafeTransferLib.safeTransferFrom` does not verify the amount actually received. When `loanToken` is a fee-on-transfer (FOT) token, Midnight receives less than the recorded amount on every take, causing `claimableSettlementFee` to permanently exceed the contract's actual token balance. Repeated takes accumulate the deficit until `claimSettlementFee` drains tokens belonging to other accounting buckets (withdrawable, collateral), breaking the core solvency invariant.

### Finding Description

**Exact code path:**

`src/Midnight.sol` line 418 increments the fee counter by the full spread: [1](#0-0) 

Then line 455 performs the inbound transfer: [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only checks the boolean return value of `transferFrom`; it does not measure the actual balance delta received: [3](#0-2) 

**Market creation has no loanToken validation.** `touchMarket` validates collateral params, LLTV tiers, and maxLif, but imposes zero constraints on `market.loanToken`: [4](#0-3) 

Any address can call `touchMarket` with an arbitrary `loanToken`, including a FOT token.

**Exploit flow:**

1. Attacker (market creator) calls `touchMarket` with a FOT loanToken (e.g., 1% transfer fee).
2. Maker calls `SetterRatifier.setIsRootRatified` to ratify a Merkle root containing a sell offer (`offer.buy = false`) in that market: [5](#0-4) 
3. Taker calls `Midnight.take` with the sell offer.
4. At line 418: `claimableSettlementFee[loanToken] += (buyerAssets - sellerAssets)` — full nominal spread recorded.
5. At line 455: FOT token transfers `(buyerAssets - sellerAssets) * 0.99` to Midnight — 1% is burned/taken as fee.
6. Per-take deficit: `(buyerAssets - sellerAssets) * 0.01`.
7. After N takes: `claimableSettlementFee` exceeds actual tokens held for fees by `N * (buyerAssets - sellerAssets) * 0.01`.
8. `feeClaimer` calls `claimSettlementFee(token, claimableSettlementFee, receiver)`: [6](#0-5) 
   This `safeTransfer` either reverts (if balance is insufficient) or drains tokens that belong to `withdrawable` or collateral accounting.

**Why existing checks fail:**

The Certora `tokenBalanceCorrect` solvency invariant is proven with `SafeTransferLib.safeTransferFrom` modeled as `NONDET`, explicitly bypassing FOT behavior: [7](#0-6) 

The `pendingFeeReceiptZero` weak invariant assumes the inbound transfer always covers the full `claimableSettlementFee` increment: [8](#0-7) 

With a FOT token, `pendingFeeReceipt[token]` is never fully cleared, and `tokenBalanceCorrect` is violated.

### Impact Explanation
After repeated takes, `Midnight.balanceOf(loanToken) < claimableSettlementFee[loanToken]`. When `feeClaimer` attempts to claim the full recorded fee, the `safeTransfer` at line 309 either reverts (fee is unclaimable) or — if the contract holds additional loanToken from repayments — drains tokens that back `withdrawable` positions, preventing lenders from withdrawing their credit. This directly violates the core invariant: contract balances must cover collateral, withdrawable assets, and claimable fees.

### Likelihood Explanation
**Preconditions:** Any unprivileged address can create a market with a FOT loanToken via the public `touchMarket`. The maker only needs to ratify a root via `SetterRatifier.setIsRootRatified`. No admin action is required. FOT tokens are a real, deployed token class (e.g., tokens with configurable transfer fees). The deficit accumulates on every single take — it is not a one-time edge case. The attack is fully repeatable and permissionless.

### Recommendation
After the inbound `safeTransferFrom` at line 455, measure the actual balance delta received and use that value to increment `claimableSettlementFee` instead of the pre-computed nominal `buyerAssets - sellerAssets`. Alternatively, record the balance before and after the transfer and cap the fee increment to the actual received amount:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived; // move line 418 here, use actualReceived
```

Or, document that FOT tokens are explicitly unsupported and add a revert guard (e.g., a balance-delta check) in `touchMarket` or `take`.

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {SetterRatifier} from "src/ratifiers/SetterRatifier.sol";

// FOT token: deducts `feeBps` basis points on every transferFrom
contract FeeOnTransferToken is ERC20 {
    uint256 public feeBps; // e.g. 100 = 1%
    constructor(uint256 _feeBps) ERC20("FOT","FOT") { feeBps = _feeBps; }
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount * feeBps / 10000;
        super.transferFrom(from, to, amount - fee); // deliver less
        _burn(from, fee);                           // burn the fee
        return true;
    }
}

contract FotSettlementFeeTest is Test {
    Midnight midnight;
    SetterRatifier ratifier;
    FeeOnTransferToken fotToken;

    function testFotClaimableSettlementFeeOverstated(uint256 feeBps, uint256 units) public {
        feeBps = bound(feeBps, 1, 500); // 0.01% – 5%
        units  = bound(units, 1e15, 1e22);

        // Setup: create market with FOT loanToken
        fotToken = new FeeOnTransferToken(feeBps);
        // ... create market, ratify offer, fund payer ...

        uint256 balBefore = fotToken.balanceOf(address(midnight));
        // Execute take (sell offer, maker is lender, taker is borrower)
        (uint256 buyerAssets, uint256 sellerAssets) = midnight.take(...);

        uint256 spread = buyerAssets - sellerAssets;
        uint256 actualReceived = fotToken.balanceOf(address(midnight)) - balBefore;

        // Core assertion: claimableSettlementFee is overstated
        assertEq(midnight.claimableSettlementFee(address(fotToken)), spread);
        assertLt(actualReceived, spread); // FOT delivered less
        // Invariant violation: balance < claimableSettlementFee
        assertLt(fotToken.balanceOf(address(midnight)), midnight.claimableSettlementFee(address(fotToken)));

        // Attempting to claim the full recorded fee must either revert or drain other funds
        vm.prank(feeClaimer);
        vm.expectRevert(); // insufficient balance
        midnight.claimSettlementFee(address(fotToken), spread, address(this));
    }
}

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
