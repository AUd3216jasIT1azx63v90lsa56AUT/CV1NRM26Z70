Audit Report

## Title
Reentrancy in `repay` callback allows authorized operator to drain borrower collateral via transient debt zeroing - (File: src/Midnight.sol)

## Summary
The `repay` function decrements `position[id][onBehalf].debt` at line 508 before invoking the `IRepayCallback.onRepay` external callback at line 516 and before pulling loan tokens at line 520. During the callback window the position's debt is zero while no loan tokens have been received. An authorized operator acting as the callback contract can re-enter `withdrawCollateral`, which passes the `isHealthy` check because `debt == 0` causes the oracle loop to be skipped and `0 >= 0` returns `true`. The attacker funds the repayment from the proceeds of the drained collateral, completing the attack atomically with zero upfront capital.

## Finding Description

**Root cause:** `repay` writes `position[id][onBehalf].debt -= units` before the external callback and before `safeTransferFrom`. The health invariant â€” "collateral cannot be withdrawn unless debt is backed" â€” is violated because `isHealthy` evaluates against a transiently zeroed debt that has not yet been backed by received loan tokens.

**`repay` code path (`src/Midnight.sol:502-521`):**
- Line 505: auth check passes if `isAuthorized[onBehalf][msg.sender]`
- Line 508: `position[id][onBehalf].debt -= units` â€” debt zeroed **before** callback or payment
- Line 516: `IRepayCallback(callback).onRepay(...)` â€” external callback fires with debt already zero
- Line 520: `safeTransferFrom(loanToken, payer, address(this), units)` â€” loan tokens pulled **last**

**`withdrawCollateral` code path (`src/Midnight.sol:549-573`):**
- Line 556: auth check passes (X is authorized for A)
- Line 568: `require(isHealthy(...))` passes because `debt == 0` at this point
- Line 572: collateral transferred to attacker

**`isHealthy` short-circuit (`src/Midnight.sol:944-960`):**
When `debt == 0`, the `if (debt > 0)` guard at line 948 prevents the oracle loop from executing, `maxDebt` stays 0, and `return maxDebt >= debt` evaluates to `0 >= 0 = true` regardless of collateral state. This behavior is documented ("does not call any oracle if debt is 0") and is correct for a genuinely repaid position, but is exploitable during the transient callback window.

**No reentrancy guard on `repay`:** The `LIQUIDATION_LOCK_SLOT` transient lock is set only during `take` callbacks at line 444 for the seller. It is never set during `repay`. The TOKEN SAFETY REQUIREMENTS exclude reentrancy only through token `transfer`/`transferFrom` calls; the `onRepay` callback is an explicit protocol-level callback, so this exclusion does not apply.

**Exploit flow:**
1. Victim A authorizes malicious contract X via `setIsAuthorized` or via the permissionless `EcrecoverAuthorizer.setIsAuthorized` relay (any party can submit a signed authorization).
2. X calls `repay(market, D, A, X, data)` â€” X is both `msg.sender` and `callback`.
3. Line 508 zeros A's debt. Line 516 calls `X.onRepay(...)`.
4. Inside `onRepay`, X re-enters `withdrawCollateral(market, idx, C_total, A, X)`:
   - Auth check passes (X is authorized for A).
   - Health check passes (debt is 0).
   - All of A's collateral is transferred to X.
5. X swaps collateral for D loan tokens via a DEX, returns `CALLBACK_SUCCESS`.
6. Line 520: `safeTransferFrom(loanToken, X, midnight, D)` succeeds â€” funded by the swap.

**Net result:** A's debt is 0 and A's collateral is 0. X extracted `value(C) âˆ’ D` profit with zero upfront capital.

**Why existing checks fail:**
- Authorization check in `withdrawCollateral` passes because X is legitimately authorized.
- `isHealthy` passes because debt was zeroed at line 508 before any loan tokens were received.
- `LIQUIDATION_LOCK_SLOT` is not set during `repay`, only during `take`.
- Token reentrancy guard does not cover the explicit `onRepay` callback.

## Impact Explanation
An authorized operator can drain 100% of a borrower's overcollateralized position in a single atomic transaction with zero upfront capital. The invariant "collateral cannot be withdrawn outside health/liquidation rules" is broken because the health check is evaluated against a transiently zeroed debt that has not yet been backed by received loan tokens. This constitutes direct theft of user assets. The overcollateralization premium (`value(C) âˆ’ D`) is extracted entirely by the attacker.

## Likelihood Explanation
Preconditions are routine: (1) A must have authorized X â€” common for automated repayment helpers, position managers, or integrators; (2) A must have a healthy, overcollateralized position; (3) the collateral token must be swappable for loan tokens within one transaction, which is true for any token with DEX liquidity. The `EcrecoverAuthorizer.setIsAuthorized` relay is permissionless â€” any party can submit a signed authorization on behalf of a user who has signed off-chain. The attack is repeatable across all markets and all authorized operators. A malicious contract deployed as a "repayment helper" can exploit this against any user who grants it authorization.

## Recommendation
Apply a reentrancy guard to `repay` that prevents re-entry into `withdrawCollateral` (and other state-mutating functions) during the callback window. The simplest fix is to extend the existing `LIQUIDATION_LOCK_SLOT` mechanism to also lock the borrower's position during the `onRepay` callback, mirroring the pattern already used in `take` at line 444. Alternatively, restructure `repay` to follow the checks-effects-interactions pattern: pull loan tokens from the payer **before** decrementing debt, or decrement debt only **after** the callback and token transfer complete. A dedicated transient reentrancy guard scoped to the `onBehalf` address during `repay` would also be sufficient.

## Proof of Concept
**Minimal Foundry test outline:**
1. Deploy `Midnight` and a mock oracle/collateral/loan token setup.
2. Deploy `MaliciousOperator` implementing `IRepayCallback.onRepay`.
3. Have victim A supply collateral C and borrow D loan tokens (healthy, overcollateralized position).
4. Have A authorize `MaliciousOperator` via `setIsAuthorized`.
5. Call `MaliciousOperator.attack()`:
   - Calls `midnight.repay(market, D, A, address(this), data)`.
   - Inside `onRepay`: calls `midnight.withdrawCollateral(market, 0, C_total, A, address(this))` â€” succeeds because `debt == 0`.
   - Swaps C for D loan tokens on a DEX fork.
   - Returns `CALLBACK_SUCCESS`.
6. Assert: A's debt == 0, A's collateral == 0, `MaliciousOperator` holds `value(C) âˆ’ D` profit.

**Expected invariant violation:** After the transaction, A's position has zero debt and zero collateral, meaning the overcollateralization premium was extracted without A's consent and without the protocol receiving loan tokens before the collateral was released.

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];
                uint256 price = IOracle(collateralParam.oracle).price();
                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                    .mulDivDown(collateralParam.lltv, WAD);
                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
    }
