### Title
`setPoolImplementation` Accepts Undeployed Address, Silently Producing Broken Pool Clones — (File: src/ConfidencePoolFactory.sol)

### Summary
`ConfidencePoolFactory.setPoolImplementation` only guards against `address(0)` but performs no `extcodesize` check. If the owner sets `poolImplementation` to an address at which no contract is deployed (e.g., a pre-computed CREATE2 address whose deployment transaction failed silently), every subsequent `createPool` call will deploy a minimal proxy clone that delegatecalls into empty bytespace. All calls to such clones succeed silently and return empty data, producing ghost pools that are permanently registered as legitimate.

### Finding Description
`setPoolImplementation` in `ConfidencePoolFactory` accepts any non-zero address without verifying that a contract exists there:

```solidity
function setPoolImplementation(address newPoolImplementation) external onlyOwner {
    if (newPoolImplementation == address(0)) revert ZeroAddress();   // only guard
    address old = poolImplementation;
    poolImplementation = newPoolImplementation;
    emit PoolImplementationUpdated(old, newPoolImplementation);
}
```

`createPool` then calls `Clones.cloneDeterministic(poolImplementation, salt)`, which deploys a minimal EIP-1167 proxy that hard-codes the implementation address in its bytecode without any code-existence check of its own. The subsequent high-level call `IConfidencePool(pool).initialize(...)` targets the newly deployed clone (which has code — the proxy bytecode), so Solidity's implicit `extcodesize` check on the *callee* passes. Inside the proxy, a `DELEGATECALL` to the empty implementation address returns `(success=1, returndata=0 bytes)` per EVM semantics. Because `initialize` has no return value, the Solidity ABI decoder accepts the empty returndata and the call appears to succeed. The pool is then pushed into `_poolsByAgreement` and a `PoolCreated` event is emitted, making it indistinguishable from a legitimate pool.

### Impact Explanation
Every pool created after the bad `setPoolImplementation` call is a ghost contract. All state-mutating calls (`stake`, `contributeBonus`, `withdraw`, `flagOutcome`, `claimSurvived`, `claimExpired`, etc.) delegatecall into empty bytespace, return success with empty data, and execute no code. No storage is ever written, no tokens are ever transferred. Users who call `stake()` receive a success receipt but their tokens remain in their own wallet and their `eligibleStake` stays zero; users who later call `claimSurvived()` or `claimExpired()` also receive success but receive nothing. The clone is non-upgradeable by design, so there is no recovery path once it is deployed. Any tokens directly sent to the pool address (e.g., via `contributeBonus` through a direct ERC-20 `transfer`) are permanently locked. The factory's `_poolsByAgreement` registry is permanently polluted with broken entries.

### Likelihood Explanation
The factory README and architecture describe a UUPS-upgradeable factory that deploys non-upgradeable clones, a pattern that naturally motivates future implementation upgrades. The CREATE2 deployment pattern (compute address ahead of time, set implementation, then deploy) is the standard upgrade workflow for deterministic addresses and is explicitly referenced in the analogous BeaconDS report. A failed or front-run deployment transaction that goes unnoticed is a realistic operational error, not a contrived scenario. The missing guard is a single-line omission with no compensating check anywhere in the call chain.

### Recommendation
Add an `extcodesize` check inside `setPoolImplementation` before accepting the new address:

```solidity
function setPoolImplementation(address newPoolImplementation) external onlyOwner {
    if (newPoolImplementation == address(0)) revert ZeroAddress();
    uint256 size;
    assembly { size := extcodesize(newPoolImplementation) }
    if (size == 0) revert ImplementationNotDeployed();
    address old = poolImplementation;
    poolImplementation = newPoolImplementation;
    emit PoolImplementationUpdated(old, newPoolImplementation);
}
```

Apply the same guard to the `initialize` path for `poolImplementation_` for defense-in-depth.

### Proof of Concept

1. Owner pre-computes a CREATE2 address `X` for the next `ConfidencePool` implementation.
2. Owner calls `setPoolImplementation(X)`. Only the `address(0)` check runs; it passes. `poolImplementation = X`.
3. The deployment transaction for the new implementation at `X` fails (e.g., out-of-gas, nonce mismatch). The failure is not noticed. `X` has `extcodesize == 0`.
4. A sponsor calls `createPool(agreement, token, expiry, minStake, recovery, accounts)`.
5. `Clones.cloneDeterministic(X, salt)` deploys a minimal proxy `P` whose bytecode hard-codes `X`. `P` has code; `X` does not.
6. `IConfidencePool(P).initialize(...)` is called. Solidity checks `extcodesize(P) > 0` ✓. The proxy `DELEGATECALL`s to `X` (no code) → `(success=1, returndata=0x)`. `initialize` has no return value → ABI decode succeeds. Call returns normally.
7. `_poolsByAgreement[agreement].push(P)` and `PoolCreated` event emitted. Pool `P` appears legitimate.
8. Staker calls `P.stake(amount)`. Proxy `DELEGATECALL`s to `X` → `(success=1, returndata=0x)`. `stake` has no return value → call succeeds. No `safeTransferFrom` executes; no storage written. Staker's tokens remain in their wallet; `eligibleStake[staker] == 0`.
9. Staker calls `P.claimSurvived()`. Same silent success. Zero tokens transferred. Pool is permanently broken with no recovery path.
