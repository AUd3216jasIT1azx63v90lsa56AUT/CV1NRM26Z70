### Title
Maturity Boundary `<=` Allows Debt Increase at Exact Maturity Timestamp via Multicall - (File: src/Midnight.sol)

### Summary
The maturity guard in `take` uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` when `block.timestamp == market.maturity`. This means a seller's debt can be increased at the exact maturity block, violating the protocol invariant that explicitly forbids debt increase by timestamp equality at the maturity boundary. A taker can compose two debt-increasing sell-side takes in a single `multicall` at this timestamp, and both succeed.

### Finding Description
**Code path:**

`src/Midnight.sol` `take()` line 391:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

`src/Midnight.sol` `multicall()` lines 211-220:
```solidity
function multicall(bytes[] calldata calls) external {
    for (uint256 i = 0; i < calls.length; i++) {
        (bool success, bytes memory returnData) = address(this).delegatecall(calls[i]);
        ...
    }
}
```

**Root cause:** The guard uses `<=` (not `<`). At `block.timestamp == market.maturity`, the left-hand side of the `||` is `true`, so `sellerDebtIncrease` is never checked. Any sell-side take that would increase the seller's debt passes unconditionally.

**Exploit flow:**

1. Attacker (taker) warps/waits for a block where `block.timestamp == market.maturity`.
2. Attacker has sufficient collateral deposited for the seller (or is the seller themselves taking a buy offer).
3. Attacker calls `midnight.multicall([encode(take(sellOffer, ratifierData, units1)), encode(take(sellOffer2, ratifierData, units2))])`.
4. **First take:** `sellerDebtIncrease = units1 > 0`. Check: `block.timestamp (== maturity) <= maturity` → `true`. Passes. `sellerPos.debt += units1`.
5. **Second take:** `sellerPos.credit` is 0 (or reduced), so `sellerCreditDecrease = 0`, `sellerDebtIncrease = units2 > 0`. Check: same `block.timestamp <= maturity` → still `true`. Passes. `sellerPos.debt += units2`.
6. Both takes succeed; seller's debt has increased at the exact maturity timestamp.

**Why existing checks fail:** The `CannotIncreaseDebtPostMaturity` guard is the only protection. It uses `<=` so equality is not caught. No other check in `take` re-evaluates the maturity condition between the two delegatecalls. `block.timestamp` is constant within a transaction, so both calls see the same timestamp.

### Impact Explanation
At `block.timestamp == market.maturity`, a taker can open or increase a borrower's debt position. This directly violates the core invariant: "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing." The newly created debt is immediately overdue (post-maturity liquidatable), creating bad debt risk and breaking the accounting invariant that debt must not increase after maturity.

### Likelihood Explanation
The precondition `block.timestamp == market.maturity` is reachable in any block whose timestamp equals the market's stored maturity. Since `maturity` is a `uint32` set at market creation and block timestamps are miner/validator-influenceable within ~12 seconds on Ethereum (and freely settable on L2s), this is a realistic and repeatable condition. Any unprivileged taker with a valid signed offer and sufficient collateral can trigger it. It is not a one-time race: any market whose maturity falls on a block timestamp is permanently vulnerable for that block.

### Recommendation
Change the guard from `<=` to `<`:

```solidity
// Before (line 391):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After:
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This ensures that at the exact maturity timestamp, debt-increasing takes are rejected, consistent with the stated invariant.

### Proof of Concept
```solidity
function testMulticallDebtIncreaseAtExactMaturity() public {
    uint256 units = 100;
    // Warp to exact maturity (not maturity + 1)
    vm.warp(market.maturity);
    borrowerOffer.expiry = market.maturity;
    borrowerOffer.maxUnits = units * 2;

    deal(address(loanToken), lender, units * 2);
    collateralize(market, borrower, units * 2);

    uint256 debtBefore = midnight.debtOf(id, borrower);

    // Compose two debt-increasing takes in one multicall
    bytes[] memory calls = new bytes[](2);
    calls[0] = abi.encodeCall(midnight.take, (borrowerOffer, ratifierData, units, lender, address(0), address(0), ""));
    calls[1] = abi.encodeCall(midnight.take, (borrowerOffer, ratifierData, units, lender, address(0), address(0), ""));

    vm.prank(lender);
    midnight.multicall(calls); // Both succeed — should revert on first take

    uint256 debtAfter = midnight.debtOf(id, borrower);

    // Assert: debt increased at maturity — invariant violated
    assertGt(debtAfter, debtBefore, "debt must not increase at maturity");
}
```

Expected assertion failure (demonstrating the bug): `debtAfter > debtBefore` passes, proving debt was increased at `block.timestamp == maturity`.
