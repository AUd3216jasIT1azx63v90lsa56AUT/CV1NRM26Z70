### Title
`sellerAssetsToUnits` panics on division-by-zero when `offerPrice == settlementFee` for a buy offer, causing DoS of `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/TakeAmountsLib.sol)

### Summary

`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and then calls `mulDivUp(targetSellerAssets, WAD, sellerPrice)`. When `offerPrice == settlementFee`, `sellerPrice` is exactly `0`, and `mulDivUp` panics via arithmetic underflow on `d - 1` before the division. Unlike `Midnight.take()`, which handles `sellerPrice == 0` gracefully (yielding `sellerAssets = 0`), `sellerAssetsToUnits` has no guard for this boundary case, causing `supplyCollateralAndSellWithAssetsTarget` to revert unconditionally for any buy offer at the settlement-fee-boundary tick.

### Finding Description

**Code path:**

`MidnightBundles.supplyCollateralAndSellWithAssetsTarget` (line 285–291) calls `TakeAmountsLib.sellerAssetsToUnits` outside the `try/catch` block:

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(          // <-- not inside try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) returns (...) { ... } catch {}
``` [1](#0-0) 

Inside `sellerAssetsToUnits`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
``` [2](#0-1) 

When `offerPrice == settlementFee`, `sellerPrice = 0`. Then `mulDivUp(targetSellerAssets, WAD, 0)` is called:

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;   // d=0: (d-1) underflows → Panic(0x11)
}
``` [3](#0-2) 

The `d - 1` term underflows before the division, producing `Panic(0x11)` (arithmetic underflow), not `Panic(0x12)` as the question states — but the revert is identical in effect. This is confirmed by the existing test:

```solidity
function testMulDivUpDivisionByZero(uint256 x, uint256 y) public {
    // because there is d-1.
    vm.expectRevert(stdError.arithmeticError);
    this.mulDivUp(x, y, 0);
}
``` [4](#0-3) 

**Contrast with `Midnight.take()`:** The core protocol handles `sellerPrice == 0` without reverting — it simply yields `sellerAssets = 0` via `units.mulDivDown(0, WAD) = 0`. The take succeeds. This is confirmed by `testPriceZeroNoSettlementFeeSell`. [5](#0-4) 

**The NatSpec discrepancy:** `sellerAssetsToUnits` documents "Reverts if `offerPrice < settlementFee` in case of a buy offer (midnight reverts too)." But it also reverts when `offerPrice == settlementFee`, a case where Midnight does **not** revert. This is the missing guard. [6](#0-5) 

**Attacker inputs:** The taker provides the `takes[]` array to `supplyCollateralAndSellWithAssetsTarget`. They can include any offer struct with `offer.buy = true` and `offer.tick` set to the tick whose price equals the current `settlementFee`. No ratification is needed for `sellerAssetsToUnits` to execute and panic — the revert occurs before the `try IMidnight.take(...)` call.

**Existing checks insufficient:** `buyerAssetsToUnits` has a `require(buyerPrice <= WAD)` guard but `sellerAssetsToUnits` has no analogous `require(sellerPrice > 0)` guard. [7](#0-6) 

### Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `TickLib.tickToPrice(offer.tick) == IMidnight.settlementFee(id, ttm)` will revert unconditionally with an arithmetic panic. Since the revert propagates outside the `try/catch`, the entire bundler call fails. The taker cannot use this bundler function to fill such an offer. Direct calls to `Midnight.take()` are unaffected.

### Likelihood Explanation

**Preconditions:**
1. A buy offer exists with `offer.tick` such that `tickToPrice(offer.tick)` equals the current settlement fee.
2. Both tick prices and settlement fees are multiples of `PRICE_ROUNDING_STEP = 1e12`, so exact equality is achievable.
3. The settlement fee can change after an offer is placed (via `setMarketSettlementFee`), moving an existing offer to the boundary without the maker's knowledge.
4. Any taker who includes such an offer in `supplyCollateralAndSellWithAssetsTarget` triggers the revert.

The condition is narrow but reachable, especially as settlement fees are tunable by governance and tick prices are discrete multiples of `1e12`.

### Recommendation

Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case, mirroring the behavior of `Midnight.take()` (which returns 0 seller assets). For example:

```solidity
if (sellerPrice == 0) return type(uint256).max; // or revert with a descriptive error
```

Alternatively, add `require(sellerPrice > 0, SellerPriceIsZero())` to make the boundary condition explicit and consistent with the NatSpec documentation. The fix should align the revert condition of `sellerAssetsToUnits` with that of `Midnight.take()`: revert only when `offerPrice < settlementFee` (underflow), not when `offerPrice == settlementFee`.

### Proof of Concept

```solidity
function testSellerAssetsToUnitsZeroSellerPrice() public {
    // Set settlement fee to a value that equals a tick price.
    // tickToPrice(0) = 0 (rounded to PRICE_ROUNDING_STEP), so use fee = tickToPrice(someSmallTick).
    uint256 boundaryTick = 4; // choose tick s.t. tickToPrice(tick) > 0
    uint256 boundaryPrice = TickLib.tickToPrice(boundaryTick);
    // Set settlement fee equal to the tick price.
    midnight.touchMarket(market);
    midnight.setMarketSettlementFee(id, 1, boundaryPrice);

    // Confirm Midnight.take() does NOT revert at this boundary (sellerAssets = 0).
    lenderOffer.tick = boundaryTick; // buy offer
    lenderOffer.maxUnits = 1e18;
    collateralize(market, borrower, 1e18);
    // Direct take succeeds:
    (uint256 buyerAssets, uint256 sellerAssets) = take(1e18, borrower, lenderOffer);
    assertEq(sellerAssets, 0, "direct take: sellerAssets should be 0");

    // Now confirm sellerAssetsToUnits panics:
    vm.expectRevert(stdError.arithmeticError); // Panic(0x11), not Panic(0x12)
    TakeAmountsLib.sellerAssetsToUnits(address(midnight), id, lenderOffer, 1e18);

    // Confirm supplyCollateralAndSellWithAssetsTarget reverts:
    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: lenderOffer, units: 1e18, ratifierData: hex""});
    vm.prank(borrower);
    vm.expectRevert(stdError.arithmeticError);
    midnightBundles.supplyCollateralAndSellWithAssetsTarget(
        1e18, type(uint256).max, borrower, borrower,
        new CollateralSupply[](0), takes, 0, address(0)
    );
}
```
