### Title
Zero-asset free fill on low-price buy offers bypasses `maxAssets` cap and transfers zero settlement fee - (`src/Midnight.sol`)

### Summary
In `Midnight.take()`, for a buy offer (`offer.buy = true`) at any tick below `MAX_TICK`, `buyerAssets = units.mulDivDown(buyerPrice, WAD)` truncates to zero when `units * buyerPrice < WAD`. Because the `maxAssets` consumed counter increments by `buyerAssets` (zero), the cap check always passes, allowing the offer to be taken an unlimited number of times. Each such take mutates position state (buyer gains credit, seller gains debt) while transferring zero assets and collecting zero settlement fee. The codebase itself documents and tests this exact behavior as a known bug.

### Finding Description
In `src/Midnight.sol` lines 363–364, for a buy offer:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : units.mulDivUp(sellerPrice, WAD);
```

`mulDivDown` truncates toward zero. For `units = 1` and any `buyerPrice < WAD` (i.e., any tick < `MAX_TICK = 5820`), `1 * buyerPrice / WAD = 0`. `tickToPrice(2) = 1e12` and `tickToPrice(MAX_TICK - 2) = 1e18 - 1e12`, so the condition holds for virtually the entire tick range.

In `maxAssets` mode (lines 367–369):

```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

`consumed += 0` on every call. `newConsumed` never advances past its pre-call value, so the cap check `newConsumed <= offer.maxAssets` passes unconditionally, even when the offer is already fully consumed. The offer can be taken an unbounded number of times.

Downstream, line 418:

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets; // += 0
```

And lines 455–456:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets); // transfer(0)
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets); // transfer(0)
```

Zero tokens move. Yet lines 408–414 unconditionally update `buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` by `units = 1` per call.

**Exploit flow:**
1. Attacker (lender/maker) creates a buy offer at any tick < `MAX_TICK` with `maxAssets = N` (large).
2. Any taker (or an operator authorized via `EcrecoverAuthorizer.setIsAuthorized`) calls `Midnight.take(offer, ..., units=1, taker, ...)` in a loop.
3. Each iteration: `buyerAssets = 0`, `sellerAssets = 0`, `consumed` unchanged, maker gains 1 unit of credit, taker gains 1 unit of debt, zero tokens transferred, zero fee accrued.
4. Loop runs indefinitely; maker accumulates unbounded credit for free.

There is no `require(units == 0 || buyerAssets > 0)` guard anywhere in `take()`. The code comment at line 94 explicitly acknowledges: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*

### Impact Explanation
- **Offer cap bypass**: In `maxAssets` mode, the consumed counter never advances, so the offer is never exhausted regardless of how many times it is taken.
- **Free credit for maker**: The lender accumulates credit (a claim on future repayments) without depositing any loan assets.
- **Free debt for taker**: The borrower accumulates debt without receiving any assets; if an authorized operator drives the loop, the victim's position is silently degraded.
- **Zero settlement fee**: `claimableSettlementFee` is never incremented; the protocol collects nothing for these fills.
- **Solvency skew**: `totalUnits` grows without a matching increase in `withdrawable` or contract token balance, violating the accounting invariant that every credit unit is backed by a deposited or repaid asset.

### Likelihood Explanation
- Precondition: any buy offer at any tick < `MAX_TICK` (5819 out of 5820 ticks qualify). Default tick spacing is 4, so the lowest accessible tick is 0 (price = 0) and the second-lowest is 4 (price = `1e12`).
- `units = 1` is always a valid input; no minimum-units check exists.
- Authorization via `EcrecoverAuthorizer` requires only a valid EIP-712 signature from the taker; once granted, the operator can loop `take()` without further interaction from the taker.
- Repeatable indefinitely at negligible gas cost on L2 deployments.

### Recommendation
Add a guard in `take()` immediately after computing `buyerAssets` and `sellerAssets`:

```solidity
require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
```

Alternatively, round `buyerAssets` up for buy offers (`mulDivUp`) so that any non-zero `units` always produces a non-zero asset charge. This also aligns with the stated rounding policy ("rounded against the taker") since the taker is the seller in a buy offer, and rounding up `buyerAssets` increases the cost to the buyer/maker, not the taker — so the correct fix is the explicit guard above.

### Proof of Concept

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) already proves the behavior. A minimal extension to prove the infinite-loop variant:

```solidity
function testFreeFillLoop() public {
    // Buy offer at tick MAX_TICK - 4 (price < WAD), maxAssets mode
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1e18; // large cap
    lenderOffer.tick = MAX_TICK - 4; // buyerPrice < WAD

    collateralize(market, borrower, 1000);
    deal(address(loanToken), lender, 0); // lender pays nothing

    uint256 creditBefore = midnight.creditOf(id, lender);
    uint256 feeBefore = midnight.claimableSettlementFee(address(loanToken));

    // Loop 100 times with units=1
    for (uint256 i; i < 100; i++) {
        (uint256 ba, uint256 sa) = take(1, borrower, lenderOffer);
        assertEq(ba, 0, "buyerAssets must be 0");
        assertEq(sa, 0, "sellerAssets must be 0");
    }

    // consumed never advanced past 0
    assertEq(midnight.consumed(lender, lenderOffer.group), 0);
    // fee never collected
    assertEq(midnight.claimableSettlementFee(address(loanToken)), feeBefore);
    // but maker gained 100 units of free credit
    assertEq(midnight.creditOf(id, lender), creditBefore + 100);
    // and taker gained 100 units of free debt
    assertEq(midnight.debtOf(id, borrower), 100);
}
```
