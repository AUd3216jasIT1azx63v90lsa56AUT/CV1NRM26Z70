### Title
Unconditional zero-value `safeTransfer` to receiver when `filledSellerAssets == 0` causes revert on zero-transfer-reverting `loanToken` - (`File: src/periphery/MidnightBundles.sol`)

### Summary
In `supplyCollateralAndSellWithUnitsTarget`, when `targetUnits == 0` the takes loop is skipped entirely, leaving `filledSellerAssets == 0`. The final `SafeTransferLib.safeTransfer(loanToken, receiver, 0)` at line 168 is called unconditionally — unlike the referral fee transfer at line 167 which is guarded by `if (referralFeeAssets > 0)`. If the `loanToken` reverts on zero-value transfers, the entire bundler call reverts even though all takes vacuously succeeded and the `require(filledUnits == targetUnits)` check passed.

### Finding Description

**Exact code path:**

`supplyCollateralAndSellWithUnitsTarget` accepts `targetUnits` with no lower-bound check. [1](#0-0) 

When `targetUnits == 0`, the loop condition `filledUnits < targetUnits` is immediately false (both start at 0), so the loop body never executes: [2](#0-1) 

After the loop, `require(filledUnits == targetUnits)` passes (0 == 0): [3](#0-2) 

`referralFeeAssets = 0.mulDivDown(referralFeePct, WAD) == 0`. With `minSellerAssets == 0`, the slippage check also passes. The referral transfer is correctly guarded, but the receiver transfer is not: [4](#0-3) 

`SafeTransferLib.safeTransfer` makes a raw `.call()` to `token.transfer(receiver, 0)` and propagates any revert from the token: [5](#0-4) 

**Attacker-controlled inputs:** `targetUnits = 0`, `minSellerAssets = 0`. The taker is an unprivileged user calling the bundler on their own behalf.

**Why existing checks fail:**
- No `require(targetUnits > 0)` exists anywhere in the function.
- `require(filledUnits == targetUnits)` is satisfied trivially (0 == 0).
- `require(filledSellerAssets - referralFeeAssets >= minSellerAssets)` is satisfied when `minSellerAssets == 0`.
- Line 167's `if (referralFeeAssets > 0)` guard is present but line 168 has no equivalent guard.

### Impact Explanation
If the market's `loanToken` reverts on zero-value `transfer` calls (a known behavior in some ERC-20 tokens), any call to `supplyCollateralAndSellWithUnitsTarget` with `targetUnits == 0` and `minSellerAssets == 0` will always revert at line 168. The transaction fails atomically — collateral supplies and takes are rolled back — so no funds are permanently lost, but the user cannot execute the intended bundler operation and wastes gas. The function is rendered unusable for this input combination on such tokens.

### Likelihood Explanation
**Preconditions:**
1. `targetUnits == 0` — no on-chain guard prevents this; reachable by any caller.
2. `minSellerAssets == 0` — the natural default when no slippage protection is desired.
3. `loanToken` reverts on zero-value transfers — uncommon but real (e.g., USDT on some deployments, custom tokens).

All three preconditions are independently reachable by an unprivileged taker. The scenario is repeatable: every call with these inputs will revert deterministically.

### Recommendation
Add a zero-value guard on the receiver transfer, consistent with the existing guard on the referral fee transfer at line 167:

```solidity
uint256 netSellerAssets = filledSellerAssets - referralFeeAssets;
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
if (netSellerAssets > 0) SafeTransferLib.safeTransfer(loanToken, receiver, netSellerAssets);
```

Alternatively, add `require(targetUnits > 0)` at the top of the function to reject the degenerate case entirely.

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
