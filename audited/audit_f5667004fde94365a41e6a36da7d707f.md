Audit Report

## Title
Debt Increase Permitted at Exact Maturity Boundary via `<=` Operator in `CannotIncreaseDebtPostMaturity` Check - (File: src/Midnight.sol)

## Summary
The guard at `src/Midnight.sol` line 391 uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` when `block.timestamp == maturity`, permitting `sellerDebtIncrease > 0` to execute. This is inconsistent with the post-maturity liquidation check at line 622, which uses strict `block.timestamp > market.maturity`, creating a one-block window where a borrower can increase debt at exactly maturity while post-maturity liquidation is simultaneously blocked. The new lender credit accrues zero continuous fee because `timeToMaturity = 0` at that timestamp.

## Finding Description

**Root cause — `UtilsLib.zeroFloorSub` (`src/libraries/UtilsLib.sol` line 24):**
```solidity
z := mul(gt(x, y), sub(x, y))
```
When `block.timestamp == maturity`, `gt(maturity, block.timestamp) = 0`, so `timeToMaturity = 0`. [1](#0-0) 

**Guard at `src/Midnight.sol` line 391:**
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
At `block.timestamp == maturity`, the left operand is `true`, so the `require` passes unconditionally regardless of `sellerDebtIncrease`. The debt increase at line 414 (`sellerPos.debt += sellerDebtIncrease`) executes. [2](#0-1) [3](#0-2) 

**`timeToMaturity` computation at line 359:**
```solidity
uint256 timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp);
```
Yields `0` at exactly maturity. [4](#0-3) 

**Zero continuous-fee consequence (lines 385–386):**
```solidity
uint128 buyerPendingFeeIncrease =
    UtilsLib.toUint128(buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD));
```
With `timeToMaturity = 0`, `buyerPendingFeeIncrease = 0`. The lender receives no continuous-fee obligation for credit extended at maturity. [5](#0-4) 

**Post-maturity liquidation guard at line 622:**
```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
```
Uses strict `>`. At `block.timestamp == maturity`, post-maturity liquidation is unavailable. Normal-mode liquidation requires `originalDebt > maxDebt` (unhealthy position). A borrower who passes the health check at line 476 after increasing debt at exactly maturity is immune to both liquidation paths for that block. [6](#0-5) [7](#0-6) 

**Exploit flow:**
1. Attacker identifies a market whose `maturity` falls on a reachable block timestamp.
2. At `block.timestamp == maturity`, attacker submits a `take` transaction with `sellerDebtIncrease > 0`.
3. Line 391 passes (`block.timestamp <= maturity` is `true`).
4. Debt is increased; new lender credit carries `pendingFee = 0`.
5. Health check at line 476 passes (position is healthy).
6. Post-maturity liquidation (line 622) is blocked for this block (`block.timestamp > maturity` is `false`).
7. Normal liquidation is also blocked (position is healthy).
8. Next block, post-maturity liquidation becomes available, but the fee-free credit has already been recorded.

## Impact Explanation
The protocol invariant that debt increase is forbidden at or after maturity is concretely violated. The lender providing credit at maturity receives zero continuous-fee compensation (`buyerPendingFeeIncrease = 0`), constituting a direct accounting loss to the lender. The borrower gains a one-block window of liquidation immunity at a timestamp where the position is immediately overdue. The state inconsistency is permanent: the fee-free credit record persists after the block.

## Likelihood Explanation
The precondition is a single deterministic timestamp (`block.timestamp == maturity`). Any borrower or MEV searcher with standard transaction submission capability can target this block. No privileged access is required. The offer need only have a non-expired `expiry` covering that timestamp and sufficient collateral pre-supplied. The condition is repeatable for every market whose maturity falls on a reachable block timestamp.

## Recommendation
Change the guard at line 391 from `<=` to `<` to make it consistent with the post-maturity liquidation check:

```solidity
// Before
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This ensures debt increase is forbidden at and after maturity, consistent with the strict `>` used in the liquidation path at line 622.
