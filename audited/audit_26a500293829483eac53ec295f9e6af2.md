Audit Report

## Title
Buy Offer `maxAssets` Cap Bypassed via `mulDivDown` Rounding to Zero in Unit Terms - (File: src/Midnight.sol)

## Summary
When a buy offer uses `maxAssets` as its cap, the consumed counter is incremented by `buyerAssets = mulDivDown(units, buyerPrice, WAD)`. For any tick where `buyerPrice < WAD`, a taker can choose `units` small enough that `units * buyerPrice < WAD`, causing `buyerAssets = 0`. The consumed counter does not advance, yet position state (`credit`, `debt`, `totalUnits`) is updated by the full `units` amount, allowing repeated zero-cost fills that accumulate unbounded unit exposure beyond the maker's intended `maxAssets` cap.

## Finding Description
In `src/Midnight.sol`, `take()`:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
``` [1](#0-0) 

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [2](#0-1) 

For a buy offer, `buyerPrice = sellerPrice + _settlementFee` where `sellerPrice = offerPrice - _settlementFee`, so `buyerPrice = offerPrice`. Since `tickToPrice` returns values ≤ WAD, `buyerPrice < WAD` is the normal case for all sub-WAD ticks. When `units * buyerPrice < WAD`, `mulDivDown` returns 0. The consumed counter increments by 0, the `require` check trivially passes, but position state is still updated by the full `units`:

```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);
_marketState.totalUnits = UtilsLib.toUint128(_marketState.totalUnits + buyerCreditIncrease - sellerCreditDecrease);
``` [3](#0-2) 

The protocol's own NatSpec at line 94 acknowledges this: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* [4](#0-3) 

The test `testBugBuyMaxAssetsBypass` explicitly demonstrates the scenario: consumed stays at `maxAssets`, token balances are unchanged, but `creditOf`, `debtOf`, and `totalUnits` all increase. [5](#0-4) 

## Impact Explanation
A maker who posts a buy offer with `maxAssets = N` intends to cap their total credit exposure at `N` loan tokens. An attacker can call `take(units=1)` repeatedly, accumulating arbitrarily large credit/debt positions in the market while the consumed counter never exceeds `N`. The maker's unit-denominated exposure is unbounded despite the asset cap. This violates the core invariant that offers cannot be overfilled, constitutes unauthorized state change (unbounded position growth without corresponding asset transfer), and results in the maker holding far more credit/debt than intended — a direct accounting integrity failure with concrete financial impact.

## Likelihood Explanation
Preconditions are minimal: any buy offer with `maxAssets > 0` and `buyerPrice < WAD` (the normal case for all sub-WAD ticks) is vulnerable. No special privileges are needed — any `msg.sender` can be the taker. The condition `units * buyerPrice < WAD` is trivially satisfied with `units = 1` at any realistic price. The attack is repeatable indefinitely and costs only gas.

## Recommendation
Track consumed exposure in units when `buyerAssets` rounds to zero, or add a secondary guard that rejects fills where `buyerAssets == 0` but `units > 0` under a `maxAssets`-capped offer. Alternatively, enforce a minimum `units` floor such that `units * buyerPrice >= WAD` before accepting the fill.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) is a complete, self-contained proof of concept: [5](#0-4) 

1. Set `maxAssets = 1`, `tick = MAX_TICK - 16` (so `offerPrice < WAD`).
2. Pre-set `consumed` to `maxAssets` via `setConsumed` (fully consumed offer).
3. Call `take(units=1, borrower, lenderOffer)`.
4. Assert `buyerAssets == 0`, `consumed` unchanged, token balances unchanged.
5. Assert `creditOf`, `debtOf`, and `totalUnits` all strictly increased.

    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);
        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
