### Title
Operator-ratified roots persist after authorization revocation, allowing indefinite unauthorized offer fills - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier.setIsRootRatified` checks authorization only at the time of the call and writes a persistent boolean flag `isRootRatified[maker][root]`. When the maker later revokes the operator's authorization via `setIsAuthorized(operator, false)`, the stored flag is not cleared. `SetterRatifier.isRatified` only reads this flag without re-checking whether the entity that set it is still authorized, so any taker can continue filling offers from the ratified root indefinitely after revocation.

### Finding Description
**Code path:**

- `Midnight.take` (`src/Midnight.sol:355-356`): checks `isAuthorized[offer.maker][offer.ratifier]` (i.e., maker authorized the `SetterRatifier` contract — unaffected by operator revocation), then calls `IRatifier(offer.ratifier).isRatified(offer, ratifierData)`.
- `SetterRatifier.isRatified` (`src/ratifiers/SetterRatifier.sol:35`): only checks `isRootRatified[offer.maker][root]` — no live authorization re-check.
- `SetterRatifier.setIsRootRatified` (`src/ratifiers/SetterRatifier.sol:25-26`): authorization checked once at write time; the boolean persists in storage forever unless explicitly cleared.

**Root cause:** Authorization is a snapshot check at write time, not a live check at take time. There is no mechanism to atomically invalidate all roots set by a revoked operator.

**Exploit flow:**
1. Maker calls `midnight.setIsAuthorized(address(setterRatifier), true, maker)` — authorizes the ratifier contract.
2. Maker calls `midnight.setIsAuthorized(operator, true, maker)` — authorizes operator.
3. Operator calls `setterRatifier.setIsRootRatified(maker, root, true)` — authorization passes, `isRootRatified[maker][root] = true` stored. The root can contain offers with terms unfavorable to the maker (e.g., lending at near-zero interest).
4. Maker calls `midnight.setIsAuthorized(operator, false, maker)` — revokes operator. `isRootRatified[maker][root]` is **not** cleared.
5. Any taker calls `midnight.take(offer, ratifierData, units, taker, ...)` with an offer from the ratified root.
6. `take` passes: `isAuthorized[maker][setterRatifier]` is still `true` (line 355).
7. `setterRatifier.isRatified` passes: `isRootRatified[maker][root]` is still `true` (line 35).
8. Take executes; maker's assets are consumed at the operator-chosen terms.

**Why existing checks fail:** The Certora spec rule `takeRequiresMakerConsent` only verifies `isAuthorized(offer.maker, offer.ratifier)` (the ratifier contract), not the operator who set the root. The `SetterRatifier` has no live authorization re-check analogous to `EcrecoverRatifier`'s line 44 check.

### Impact Explanation
A malicious or compromised operator can ratify a Merkle root containing offers with terms unfavorable to the maker (e.g., lend at 0% interest, or borrow at maximum rate). After the maker revokes the operator, those offers remain fillable indefinitely. Takers can drain the maker's lending capacity or force the maker into debt positions at adversarial rates, causing direct asset loss to the maker. The maker cannot easily discover which roots the operator ratified without monitoring all `SetIsRootRatified` events.

### Likelihood Explanation
Requires: (1) maker has authorized an operator (common delegation pattern), (2) operator acts maliciously or is compromised and ratifies an adversarial root, (3) maker revokes the operator but does not know to also call `setIsRootRatified(maker, root, false)` for each root the operator set. Steps 1–3 are realistic in any protocol that uses operator delegation. The attack is repeatable: the operator can ratify multiple roots before being revoked, each remaining active.

### Recommendation
In `SetterRatifier.isRatified`, store the address that ratified each root and re-check their authorization at take time, mirroring `EcrecoverRatifier`:

```solidity
// Store ratifier address alongside the flag
mapping(address maker => mapping(bytes32 root => address)) public rootRatifier;

function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    rootRatifier[maker][root] = newIsRootRatified ? msg.sender : address(0);
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}

// In isRatified:
require(isRootRatified[offer.maker][root], NotRatified());
address setter = rootRatifier[offer.maker][root];
require(
    setter == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, setter),
    Unauthorized()
);
```

Alternatively, add a test analogous to `testIsRatifiedRevokeAuthorizationInvalidates` from `EcrecoverRatifierTest` to document the intended behavior and enforce it.
