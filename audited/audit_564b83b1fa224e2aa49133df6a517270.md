### Title
Lender can drain `withdrawable` below `continuousFeeCredit`, permanently DoS-ing `claimContinuousFee()` - (File: src/Midnight.sol)

### Summary

`claimContinuousFee()` decrements `_marketState.withdrawable` by `amount` with no prior guard that `withdrawable >= amount`. Because `continuousFeeCredit` grows from lender-credit reductions (fee accrual) without any corresponding increase to `withdrawable`, and because any lender can freely drain `withdrawable` via `withdraw()`, an unprivileged lender can reduce `withdrawable` below `continuousFeeCredit` and cause every subsequent `claimContinuousFee()` call to revert with an arithmetic underflow.

### Finding Description

**Accounting divergence — root cause**

`withdrawable` increases only on `repay` and `liquidate` (debt repaid by borrowers): [1](#0-0) 

`continuousFeeCredit` increases inside `_updatePosition` when fee is accrued from a lender's credit — with no matching `withdrawable` increase: [2](#0-1) 

`withdraw()` decrements `withdrawable` by the full withdrawn amount: [3](#0-2) 

`claimContinuousFee()` then blindly subtracts `amount` from `withdrawable` with no guard: [4](#0-3) 

**Exploit flow**

Preconditions: market exists, fees have accrued (`continuousFeeCredit = F > 0`), some repayments have been made so `withdrawable = W >= F`.

1. Attacker (lender with credit ≥ `W − F + 1`) calls `withdraw(market, W − F + 1, attacker, attacker)`.
   - `withdrawable` drops to `F − 1`.
   - This is fully authorized — the attacker is withdrawing their own credit.
2. feeClaimer calls `claimContinuousFee(market, F, receiver)`.
   - Line 318: `continuousFeeCredit -= F` — succeeds (F ≤ continuousFeeCredit).
   - Line 319: `totalUnits -= F` — succeeds.
   - Line 320: `withdrawable -= F` → `(F−1) − F` → **arithmetic underflow → revert**.

The attack is repeatable: every time a borrower repays and `withdrawable` rises back above `continuousFeeCredit`, the attacker can drain it again with a single `withdraw()` call.

**Why existing checks are insufficient**

`claimContinuousFee()` only checks `msg.sender == feeClaimer` and `tickSpacing > 0`. [5](#0-4) 

There is no `require(withdrawable >= amount)`. The Certora liveness rule at line 264 of `Role.spec` documents that the function reverts when `amount > withdrawable`, but this is the underflow itself — not an intentional guard. [6](#0-5) 

The protocol's own test confirms the gap: it explicitly repays before claiming to ensure `withdrawable` covers the fee, acknowledging that without repayment the claim would fail: [7](#0-6) 

### Impact Explanation

`claimContinuousFee()` is permanently DoS-able at negligible cost. Any lender can block the feeClaimer from collecting all accrued continuous-fee revenue for a market. The feeClaimer can only succeed after a borrower repays enough to restore `withdrawable ≥ continuousFeeCredit`, at which point the attacker can immediately drain it again. Protocol fee revenue is effectively held hostage.

### Likelihood Explanation

- **Preconditions**: any market with a non-zero continuous fee and at least one lender who has credit to withdraw. This is the normal operating state of any active market.
- **Attacker cost**: zero beyond gas — the attacker withdraws their own credit and receives their own tokens back.
- **Repeatability**: unlimited; the attacker can re-execute after every repayment.
- **No special role required**: any lender (or anyone authorized by a lender) qualifies.

### Recommendation

Add an explicit guard in `claimContinuousFee()` before the three decrements, or restructure the accounting so that fee accrual also increments `withdrawable`. The minimal fix is:

```solidity
require(_marketState.withdrawable >= UtilsLib.toUint128(amount), InsufficientWithdrawable());
```

placed immediately after the `MarketNotCreated` check at line 316. A more robust fix is to increment `withdrawable` by `accruedFee` inside `_updatePosition` at the same time `continuousFeeCredit` is incremented, maintaining the invariant `withdrawable >= continuousFeeCredit` at all times.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry unit test — expected to pass, demonstrating the DoS

function testClaimContinuousFeeDoSByWithdraw() public {
    uint256 credit = 1_000e18;
    uint256 feeRate = MAX_CONTINUOUS_FEE;
    uint256 ttm = 365 days;

    // 1. Setup: lender supplies credit, borrower takes debt
    setupLender(credit, feeRate, ttm);

    // 2. Borrower repays in full so withdrawable = credit
    deal(address(loanToken), borrower, credit);
    vm.prank(borrower);
    midnight.repay(market, credit, borrower, address(0), hex"");
    // withdrawable == credit, continuousFeeCredit == 0

    // 3. Time passes; fee accrues
    vm.warp(block.timestamp + ttm / 2);
    midnight.updatePosition(market, lender);
    uint256 feeCredit = midnight.continuousFeeCredit(id);
    // feeCredit > 0; withdrawable still == credit - feeCredit (lender credit reduced)

    // 4. Attacker (lender) drains withdrawable to feeCredit - 1
    uint256 w = midnight.withdrawable(id);
    uint256 drainAmount = w - feeCredit + 1;
    vm.prank(lender);
    midnight.withdraw(market, drainAmount, lender, lender);
    assertLt(midnight.withdrawable(id), feeCredit, "withdrawable now below feeCredit");

    // 5. feeClaimer attempts to claim — must revert
    vm.prank(feeClaimer);
    vm.expectRevert(); // arithmetic underflow on withdrawable -= feeCredit
    midnight.claimContinuousFee(market, feeCredit, feeClaimer);
}
```

**Expected assertions**: `vm.expectRevert()` passes, confirming `claimContinuousFee` reverts due to underflow at `_marketState.withdrawable -= UtilsLib.toUint128(amount)`. [8](#0-7)

    function claimContinuousFee(Market memory market, uint256 amount, address receiver) external {
        bytes32 id = toId(market);
        MarketState storage _marketState = marketState[id];
        require(msg.sender == feeClaimer, OnlyFeeClaimer());
        require(_marketState.tickSpacing > 0, MarketNotCreated());

        _marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
        _marketState.totalUnits -= UtilsLib.toUint128(amount);
        _marketState.withdrawable -= UtilsLib.toUint128(amount);

        emit EventsLib.ClaimContinuousFee(msg.sender, id, amount, receiver);

    }
