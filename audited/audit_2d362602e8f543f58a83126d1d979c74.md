### Title
ERC2612 permit failure silently swallowed, `safeTransferFrom` emits misleading allowance revert - (File: src/periphery/MidnightBundles.sol)

### Summary
In `pullToken`, when `permit.kind == PermitKind.ERC2612`, the `permit()` call is wrapped in a bare `try {} catch {}` that discards all revert data. If the permit is invalid (expired deadline, already-consumed nonce, wrong signature), the revert is swallowed and `safeTransferFrom` is called unconditionally. Without a prior allowance, the token reverts with an insufficient-allowance error rather than the true permit failure reason, producing a misleading diagnostic for the caller.

### Finding Description
The exact path in `MidnightBundles.sol`:

```
pullToken (line 378)
  └─ permit.kind == PermitKind.ERC2612 (line 379)
       ├─ try IERC20Permit(token).permit(...) {} catch {}   // line 383 – revert discarded
       └─ SafeTransferLib.safeTransferFrom(token, from, address(this), amount)  // line 384
```

The comment at line 382 documents the intent: *"Tolerate revert: a third party may have already consumed the permit."* The design assumes that if the permit was front-run, the allowance is already set and `safeTransferFrom` will succeed. However, the same catch block also silently absorbs permit failures caused by:
- An expired `deadline` (e.g., `block.timestamp > deadline`)
- A nonce that was already consumed by the owner in a prior transaction
- A malformed or mismatched signature

In all these cases, no allowance is set, `safeTransferFrom` reverts with the token's own insufficient-allowance error (e.g., OpenZeppelin's `ERC20InsufficientAllowance`), and the true cause (expired permit, bad nonce) is invisible to the caller. The attacker-controlled input is the `TokenPermit.data` field, which is fully caller-supplied and decoded without validation before the try/catch.

### Impact Explanation
The transaction reverts in all failure cases, so no funds are lost and no protocol state is corrupted. The concrete impact is a non-critical behavior divergence: a caller who supplies an expired or replayed ERC2612 permit receives an opaque allowance error from the token contract instead of a permit-specific revert reason, making debugging and off-chain simulation misleading.

### Likelihood Explanation
Reachable by any unprivileged caller (borrower, lender, taker, etc.) on every public entry point that accepts a `TokenPermit` (`buyWithUnitsTargetAndWithdrawCollateral`, `buyWithAssetsTargetAndWithdrawCollateral`, `repayAndWithdrawCollateral`, `supplyCollateralAndSellWithUnitsTarget` via `CollateralSupply.permit`). Triggerable any time a permit has expired or its nonce was already consumed, which is a normal operational condition (e.g., a user retries a failed transaction after the deadline passes). Repeatable on every call.

### Recommendation
Capture the permit revert and re-throw it only when the allowance is still insufficient, so the true cause is preserved:

```solidity
bool permitSucceeded;
try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {
    permitSucceeded = true;
} catch {}
// Only proceed if the permit succeeded OR a sufficient allowance already exists
if (!permitSucceeded) {
    require(IERC20(token).allowance(from, address(this)) >= amount, PermitFailed());
}
SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
```

This preserves the front-running tolerance while surfacing a meaningful error when neither the permit nor a pre-existing allowance is sufficient.
