Audit Report

## Title
`claimContinuousFee` reverts via arithmetic underflow when bad-debt liquidation front-runs the feeClaimer - (File: `src/Midnight.sol`)

## Summary
`claimContinuousFee` performs a Solidity 0.8 checked subtraction on `continuousFeeCredit` at line 318 with no guard that `amount <= continuousFeeCredit`. An unprivileged attacker can call `liquidate` with `seizedAssets=0, repaidUnits=0` at zero token cost to realize existing bad debt, atomically reducing `continuousFeeCredit` before the feeClaimer's pending transaction executes, causing it to revert with an arithmetic underflow. The attack is repeatable across distinct bad-debt positions and constitutes a sustained denial-of-service against protocol fee collection.

## Finding Description
**Root cause — `claimContinuousFee`** (`src/Midnight.sol:318`):
```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
```
No `require(amount <= continuousFeeCredit)` guard exists. Solidity 0.8 checked arithmetic reverts on underflow. The Certora spec at `certora/specs/Role.spec:264` encodes this revert condition explicitly:
```
assert !reverted <=> ... && amount <= continuousFeeCreditBefore;
```
confirming `amount <= continuousFeeCredit` is a necessary non-revert condition with no on-chain enforcement.

**Bad-debt slash — `liquidate`** (`src/Midnight.sol:635-640`):
```solidity
_marketState.continuousFeeCredit = _lossFactor < type(uint128).max
    ? UtilsLib.toUint128(
        _marketState.continuousFeeCredit
            .mulDivDown(type(uint128).max - _marketState.lossFactor,
                        type(uint128).max - _lossFactor)
      )
    : 0;
```
Because `newLossFactor > oldLossFactor`, the ratio `(MAX − newLF) / (MAX − oldLF) < 1`, so `continuousFeeCredit` is strictly reduced on every bad-debt event.

**Zero-cost liquidation path**: The protocol explicitly supports `seizedAssets=0, repaidUnits=0` calls (`src/Midnight.sol:577`). The `atMostOneNonZero` check at line 595 uses `or(iszero(x), iszero(y))` — when both are zero this evaluates to `or(1,1) = 1 = true`, so the check passes. The token-transfer block at line 643 (`if (repaidUnits > 0 || seizedAssets > 0)`) is skipped entirely. The only cost to the attacker is gas.

**Exploit flow**:
1. A borrower position exists with `badDebt > 0` (collateral worth less than debt at `maxLif`). `continuousFeeCredit = X`. feeClaimer broadcasts `claimContinuousFee(market, X, receiver)`.
2. Attacker front-runs with `liquidate(market, 0, 0, 0, borrower, false, receiver, address(0), "")` — zero tokens transferred. Bad debt is realized; `continuousFeeCredit` drops to `X' < X`.
3. feeClaimer's transaction executes: `X' -= X` → arithmetic underflow → revert.

**Why existing checks fail**: The `liquidatorGate` check (`src/Midnight.sol:597-600`) only applies when `market.liquidatorGate != address(0)`; markets with `liquidatorGate == address(0)` allow any caller. No minimum-amount or slippage guard exists in `claimContinuousFee`. The `_position.debt > 0` check at line 596 is a precondition on the target borrower, not a blocker for the attacker.

## Impact Explanation
The feeClaimer's `claimContinuousFee` transactions are persistently griefed and revert whenever the submitted `amount` equals the current `continuousFeeCredit` (the natural "claim all" pattern). Protocol fee collection is disrupted: the feeClaimer must re-read on-chain state and resubmit on every attempt. The attack is zero-cost (gas only) and can be replayed across multiple bad-debt positions or repeated price-drop events, constituting a sustained denial-of-service against a core protocol revenue function. No funds are permanently lost, but fee collection is indefinitely delayable as long as liquidatable bad-debt positions exist.

## Likelihood Explanation
**Attacker cost**: gas only — no tokens required. **Preconditions**: a liquidatable position with `badDebt > 0` must exist (routine in any lending market experiencing price volatility) and the feeClaimer must submit `amount` equal to the current `continuousFeeCredit`. **Repeatability**: each distinct bad-debt position or subsequent price drop creates a new front-run opportunity. **Access**: any unprivileged external address can call `liquidate` on markets without a `liquidatorGate`.

## Recommendation
Add a guard in `claimContinuousFee` to cap the claimed amount at the available credit, or revert with a clear error when `amount > continuousFeeCredit`:

```solidity
// Option A: explicit guard
require(amount <= _marketState.continuousFeeCredit, InsufficientContinuousFeeCredit());
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);

// Option B: claim at most available (slippage-tolerant)
uint128 claimable = UtilsLib.toUint128(Math.min(amount, _marketState.continuousFeeCredit));
_marketState.continuousFeeCredit -= claimable;
// adjust totalUnits and withdrawable accordingly, transfer claimable
```

Option A preserves the feeClaimer's intent (revert if less than expected); Option B makes the call idempotent under front-running. The feeClaimer should also be updated to read `continuousFeeCredit` atomically and pass it as the `amount` argument rather than using a cached value.
