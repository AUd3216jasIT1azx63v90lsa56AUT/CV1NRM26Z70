Audit Report

## Title
Permissionless Nonce Consumption Allows Suppression of Authorizer's Revocation Signature - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` uses a sequential monotonic nonce with no caller restriction. An attacker holding a previously distributed `isAuthorized=true` signature can submit it after the authorizer has signed a same-nonce `isAuthorized=false` revocation (while the first signature is still pending), permanently consuming nonce slot N and rendering the revocation unsubmittable. The authorizer's most recently expressed off-chain intent is silently overridden, leaving an unauthorized party with active `isAuthorized=true` access to all `onBehalf`-gated entry points.

## Finding Description
**Root cause:**

`EcrecoverAuthorizer.setIsAuthorized` at line 26 enforces a purely sequential nonce with no restriction on who may submit:

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
``` [1](#0-0) 

The permissionless submission is confirmed by `testEcrecoverAuthorizerPermissionless`, which explicitly demonstrates any address (`otherLender`) can submit a valid signature on behalf of the authorizer: [2](#0-1) 

There is no mechanism to mark a nonce as superseded, no ordering preference for later-signed messages, and no way for the authorizer to advance their nonce counter without submitting a full authorization.

**Exploit flow:**

1. Authorizer A (nonce=0) signs `auth_true = {authorizer=A, authorized=X, isAuthorized=true, nonce=0, deadline=T}` and distributes it to a relayer or counterparty.
2. Before `auth_true` is submitted, A changes their mind and signs `auth_false = {..., isAuthorized=false, nonce=0, deadline=T+1}` (same nonce, since nonce[A] is still 0).
3. Attacker (holding `auth_true`) calls `ecrecoverAuthorizer.setIsAuthorized(auth_true, sig_true)`:
   - Line 25: deadline check passes (T is still valid).
   - Line 26: `0 == nonce[A]++` passes; `nonce[A]` becomes 1.
   - Lines 46–47: `IMidnight(MIDNIGHT).setIsAuthorized(X, true, A)` executes → `isAuthorized[A][X] = true`.
4. A (or anyone) attempts `ecrecoverAuthorizer.setIsAuthorized(auth_false, sig_false)`:
   - Line 26: `0 != 1` → reverts `InvalidNonce`.
5. Final state: `isAuthorized[A][X] = true` despite A's most recent signed intent being revocation. [3](#0-2) 

**Why existing checks fail:**

- **Deadline check (line 25):** Does not help — `auth_true`'s deadline T is still valid at submission time.
- **Signature check (lines 31–36):** Does not help — `auth_true` is a legitimately signed message from A.
- **No supersession mechanism:** There is no way to mark nonce 0 as "cancelled" or to prefer the later-signed message. [4](#0-3) 

## Impact Explanation
An unprivileged attacker forces `isAuthorized[A][X] = true` against A's explicit revocation intent. X thereby retains the ability to act on A's behalf across all `onBehalf`-gated entry points: `withdraw`, `withdrawCollateral`, `repay`, `setConsumed`, `setIsAuthorized`, and `take` — enabling unauthorized movement of A's assets (funds, collateral) until A submits an on-chain direct revocation via `Midnight.setIsAuthorized`. During the window between the forced authorization and A's on-chain recovery, X can drain A's deposited funds or collateral. [5](#0-4) [6](#0-5) 

## Likelihood Explanation
Preconditions are realistic in any relayer or meta-transaction workflow: (1) A signed and distributed an `isAuthorized=true` authorization that has not yet been submitted on-chain; (2) A subsequently signed a same-nonce `isAuthorized=false` revocation. The attacker needs only to have received or intercepted the first signature — a normal outcome when signatures are handed to relayers or counterparties. The attack requires no special privilege, no capital, and is repeatable for every nonce slot. The `testEcrecoverAuthorizerPermissionless` test confirms the permissionless submission path is an intentional design feature, making this a realistic attack surface. [2](#0-1) 

## Recommendation
Add a `cancelNonce(address authorizer)` or `incrementNonce()` function that allows the authorizer (or an authorized party) to advance their own nonce counter directly, invalidating all pending signatures at the current nonce without requiring a full authorization submission. This is the standard pattern used in protocols like Morpho Blue. Alternatively, include a `minValidNonce` mapping that the authorizer can advance on-chain, and validate `authorization.nonce >= minValidNonce[authorization.authorizer]` in addition to the current sequential check. This allows the authorizer to atomically invalidate any distributed-but-unsubmitted signatures by bumping their minimum valid nonce before an attacker can front-run. [7](#0-6) 

## Proof of Concept
Minimal Foundry test (extends `BaseTest`):

```solidity
function testNonceRaceSuppressesRevocation() public {
    // Setup: authorize ecrecoverAuthorizer
    vm.prank(borrower);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

    // Step 1: borrower signs isAuthorized=true at nonce=0
    Authorization memory authTrue = Authorization({
        authorizer: borrower, authorized: lender,
        isAuthorized: true, nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory sigTrue = signAuthorization(authTrue, borrower);

    // Step 2: borrower changes mind, signs isAuthorized=false at same nonce=0
    Authorization memory authFalse = Authorization({
        authorizer: borrower, authorized: lender,
        isAuthorized: false, nonce: 0,
        deadline: block.timestamp + 2 days
    });
    Signature memory sigFalse = signAuthorization(authFalse, borrower);

    // Step 3: attacker submits the stale isAuthorized=true first
    address attacker = makeAddr("attacker");
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(authTrue, sigTrue);
    assertEq(midnight.isAuthorized(borrower, lender), true);

    // Step 4: borrower's revocation is now unsubmittable
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(authFalse, sigFalse);

    // Final state: isAuthorized=true despite borrower's revocation intent
    assertEq(midnight.isAuthorized(borrower, lender), true);
}

        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
