### Title
Buy-offer `maxAssets` budget bypassed via zero-rounding of `buyerAssets` when `offerPrice < WAD` - (File: src/Midnight.sol)

### Summary
In `take()`, when `offer.buy = true` and `offer.maxAssets > 0`, the consumed budget is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. For a buy offer, `buyerPrice` algebraically equals `offerPrice` regardless of settlement fee, and for any tick below `MAX_TICK` (5820), `offerPrice < WAD`. With `units = 1`, `mulDivDown(1, offerPrice, WAD) = 0`, so `consumed` is never incremented. An unprivileged taker can fill the offer indefinitely with `units = 1` even after the budget is fully consumed, violating the invariant that offer fill amount cannot exceed remaining offer capacity.

### Finding Description
**Algebraic identity for buy offers:**

```
sellerPrice = offerPrice - _settlementFee   (Midnight.sol:361)
buyerPrice  = sellerPrice + _settlementFee  (Midnight.sol:362)
           = offerPrice                     // settlement fee cancels
```

`buyerPrice = offerPrice` always for buy offers, independent of `timeToMaturity` or `_settlementFee`.

**Zero-rounding condition:**

```
buyerAssets = units.mulDivDown(buyerPrice, WAD)   (Midnight.sol:363)
            = mulDivDown(1, offerPrice, 1e18)
            = 0   when offerPrice < 1e18
```

`tickToPrice(MAX_TICK) = 1e18` and `tickToPrice(MAX_TICK - 2) = 1e18 - 1e12`, so every tick below `MAX_TICK` yields `offerPrice < WAD`, making `buyerAssets = 0` for `units = 1`.

**Consumed accounting failure:**

```solidity
// Midnight.sol:367-369
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    // += 0 when buyerAssets rounds to 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
    // passes: consumed + 0 <= maxAssets, even when consumed == maxAssets
}
```

The `ConsumedAssets` guard is never triggered because `newConsumed` never increases. The taker can call `take(offer, ..., 1, ...)` in an unbounded loop, each time:
- incrementing the maker's credit by 1 unit (or reducing debt by 1)
- incrementing the taker's debt by 1 unit (or reducing credit by 1)
- transferring 0 tokens (`buyerAssets - sellerAssets = 0`)
- leaving `consumed` unchanged

The existing test at `test/TakeTest.sol:858` (`testBugBuyMaxAssetsBypass`) explicitly confirms this: with `consumed` pre-set to `maxAssets`, `take(1, ...)` succeeds, `consumed` stays at `maxAssets`, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all increase. [1](#0-0) [2](#0-1) 

### Impact Explanation
Any buy offer with `maxAssets > 0` and `tick < MAX_TICK` (which covers all economically meaningful prices below par) can be filled an unlimited number of times with `units = 1` after the budget is fully consumed. The maker's intended exposure cap is completely bypassed: the maker accumulates unbounded credit and the taker accumulates unbounded debt, with zero token transfers per fill. The core invariant "offer fill amount cannot exceed remaining offer capacity" is broken. [3](#0-2) 

### Likelihood Explanation
**Preconditions:**
1. A buy offer exists with `maxAssets > 0` and any `tick < MAX_TICK` (the common case; `MAX_TICK` is the par/1:1 price).
2. The taker is any unprivileged address that is not the maker.
3. The taker must pass the `isHealthy` check after each fill (needs collateral for debt increases, or existing credit to absorb the fill).

The condition `offerPrice < WAD` holds for every tick from 0 to 5819 out of 5820 possible ticks. No time-warp or special market state is required; the bug is present at any `block.timestamp`. It is repeatable indefinitely in a single transaction via a loop. [4](#0-3) [5](#0-4) 

### Recommendation
Add a guard that prevents a non-zero `units` take from contributing zero to the consumed counter in assets mode:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetsDelta());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures every non-trivial fill advances the consumed counter, making the budget enforceable. Alternatively, track consumed in `units` even in assets mode and enforce the assets cap separately, though this changes the semantics of `setConsumed`. [3](#0-2) 

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already serves as a unit PoC. An extended invariant/fuzz test:

```solidity
function testBuyMaxAssetsBypassLoop(uint256 N) public {
    N = bound(N, 1, 1000);
    collateralize(market, borrower, N + 100);

    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;          // budget = 1 asset
    lenderOffer.tick = MAX_TICK - 16;   // offerPrice < WAD → buyerAssets rounds to 0

    deal(address(loanToken), lender, 0);

    for (uint256 i = 0; i < N; i++) {
        (uint256 ba, uint256 sa) = take(1, borrower, lenderOffer);
        assertEq(ba, 0, "buyerAssets must be 0");
        assertEq(sa, 0, "sellerAssets must be 0");
        // consumed never advances past maxAssets
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
    }

    // Invariant violated: N units of debt created despite maxAssets = 1
    assertEq(midnight.debtOf(id, borrower), N);
    assertEq(midnight.creditOf(id, lender), N);
}
```

Expected assertions: all `N` iterations succeed, `consumed` stays at 1, `debtOf(borrower) == N`, `creditOf(lender) == N` — proving the offer was filled `N` times beyond its capacity. [2](#0-1) [6](#0-5)
