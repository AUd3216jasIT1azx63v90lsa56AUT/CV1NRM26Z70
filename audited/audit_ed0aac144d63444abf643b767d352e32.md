### Title
Lenders Can Front-Run Bad Debt Realization to Avoid Proportional Loss — (File: src/Midnight.sol)

---

### Summary

When a liquidation that will realize bad debt is pending in the mempool, any lender can front-run it by calling `withdraw()` before the liquidation executes. Because `lossFactor` is only updated at liquidation time, the front-running lender exits at full credit value. The remaining lenders then absorb a disproportionately larger share of the socialized bad debt. This is the direct analog of the PoolTogether H01 vulnerability: state that determines loss distribution can be manipulated by an unprivileged actor before the critical state transition executes.

---

### Finding Description

**Vulnerability class**: State manipulation / front-running before a critical state transition (accounting, bad-debt socialization).

**Root cause — how bad debt is socialized:**

In `liquidate()`, when `badDebt > 0`, the market-wide `lossFactor` is updated and `totalUnits` is reduced: [1](#0-0) 

The `lossFactor` is a lazy accumulator. It is only applied to an individual lender's `credit` when `_updatePosition` is called for that lender: [2](#0-1) 

**Root cause — how `withdraw` interacts with this:**

`withdraw()` calls `_updatePosition` first (line 485), which snapshots the lender's credit against the *current* (pre-bad-debt) `lossFactor`. The lender then receives tokens equal to their full, unslashed credit: [3](#0-2) 

**Exploit path:**

1. Market has lenders Alice (100 credit) and Bob (100 credit); `totalUnits = 200`.
2. A borrower has 50 units of uncollateralized bad debt. A prior repayment of 100 units means `withdrawable = 100`.
3. A liquidator submits a transaction to liquidate the borrower, which will realize 50 bad debt and increase `lossFactor` proportionally (each lender should lose 25 credit).
4. Alice observes the pending liquidation in the mempool and front-runs it with `withdraw(100)`:
   - `_updatePosition` applies the current `lossFactor` (0) → Alice's credit stays at 100.
   - Alice receives 100 tokens. `totalUnits = 100`, `withdrawable = 0`.
5. The liquidation executes. `badDebt = 50`, `_totalUnits = 100`:
   - `lossFactor` is set to `type(uint128).max / 2` (50% loss).
   - `totalUnits = 50`.
6. Bob calls `withdraw(50)` → his credit is slashed from 100 to 50; he receives 50 tokens.

**Result**: Alice received 100 (zero loss). Bob received 50 (lost 50). Without front-running, both would have received 75 each.

**Note on the acknowledged race condition:** The protocol comment at lines 27–29 acknowledges that lenders race to withdraw *after* assets become withdrawable (post-repayment/liquidation). That is a different scenario — racing for already-available assets. The attack here is front-running a *pending* bad debt realization to avoid the proportional loss entirely, which is not covered by that comment. [4](#0-3) 

---

### Impact Explanation

A sophisticated lender (or MEV bot) can consistently avoid all bad debt socialization by monitoring the mempool and front-running every liquidation that will realize bad debt. The full loss is shifted onto passive lenders who do not front-run. In markets with concentrated bad debt and sufficient prior repayments (providing `withdrawable` liquidity), this can result in complete loss of principal for remaining lenders while the front-runner exits at par.

---

### Likelihood Explanation

- **No privileged access required.** Any lender can call `withdraw()`.
- **Mempool visibility is standard.** MEV bots routinely monitor pending transactions.
- **Precondition (`withdrawable >= credit`) is realistic.** In any market where some borrowers repay normally and others default, there will be withdrawable assets available before the bad debt liquidation is processed.
- **Gas price advantage is standard MEV capability.** Front-running a single liquidation transaction is a well-understood MEV strategy.

---

### Recommendation

Before bad debt is realized in `liquidate()`, snapshot or freeze the credit distribution so that lenders who withdraw in the same block as (or just before) the liquidation still bear their proportional share of the loss. One approach: record a "pending bad debt" amount and apply it retroactively to any `withdraw` or `_updatePosition` call that occurs in the same block. Alternatively, introduce a short withdrawal delay or a per-block withdrawal cap, analogous to the freeze mechanism added in the PoolTogether fix.

### Proof of Concept

```
State setup:
  - Alice: credit = 100, lastLossFactor = 0
  - Bob:   credit = 100, lastLossFactor = 0
  - totalUnits = 200, withdrawable = 100 (from a prior repayment)
  - Borrower: debt = 50, collateral = 0 (pure bad debt)

Step 1 — Liquidator submits tx T_liq to liquidate borrower (pending in mempool).

Step 2 — Alice front-runs T_liq with withdraw(market, 100, alice, alice):
  _updatePosition: lossFactor=0, lastLossFactor=0 → postSlashCredit = 100 (no change)
  alice.credit    = 100 - 100 = 0
  withdrawable    = 100 - 100 = 0
  totalUnits      = 200 - 100 = 100
  Alice receives 100 loan tokens.

Step 3 — T_liq executes: liquidate(borrower, badDebt=50):
  _totalUnits = 100
  lossFactor  = MAX - (MAX - 0) * (100 - 50) / 100 = MAX/2
  totalUnits  = 100 - 50 = 50
  withdrawable += 0 (no repaid units, pure bad debt realization)

Step 4 — Bob calls withdraw(market, 50, bob, bob):
  _updatePosition: lossFactor=MAX/2, lastLossFactor=0
    postSlashCredit = 100 * (MAX - MAX/2) / (MAX - 0) = 50
  bob.credit   = 50 - 50 = 0
  withdrawable = 0 - 50 → REVERTS (no withdrawable left)

  Bob can only withdraw 0 tokens. His 100 credit is now worth 50 units,
  but withdrawable = 0 so he cannot redeem until new repayments arrive.
  His effective loss = 50 tokens (the entire bad debt).

Expected (no front-run): Alice loses 25, Bob loses 25.
Actual (with front-run): Alice loses 0, Bob loses 50.

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
