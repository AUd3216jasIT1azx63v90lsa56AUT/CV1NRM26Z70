### Title
Continuous Fee Permanently Unclaimable After Full Lender Withdrawal With Partial Borrower Repayment - (File: src/Midnight.sol)

### Summary

`_updatePosition` increases `continuousFeeCredit` by the accrued fee `F` but never increases `withdrawable`. When a lender subsequently calls `withdraw(credit)` to exit their full remaining position, `withdrawable` is reduced by the full post-fee credit amount. If the borrower has only partially repaid (enough to cover the withdrawal but not the fee), `withdrawable` drops below `continuousFeeCredit`, causing `claimContinuousFee` to revert permanently via uint128 underflow.

### Finding Description

**Code path:**

`_updatePosition` at `src/Midnight.sol:846` adds `accruedFee` to `continuousFeeCredit` but leaves `withdrawable` unchanged:

```solidity
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
``` [1](#0-0) 

`withdraw` at `src/Midnight.sol:494` decreases `withdrawable` by the full `units` withdrawn, with no floor relative to `continuousFeeCredit`:

```solidity
_marketState.withdrawable -= UtilsLib.toUint128(units);
``` [2](#0-1) 

`claimContinuousFee` at `src/Midnight.sol:318-320` performs three unchecked-underflow subtractions; it reverts if `amount > withdrawable`:

```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
_marketState.totalUnits -= UtilsLib.toUint128(amount);
_marketState.withdrawable -= UtilsLib.toUint128(amount);
``` [3](#0-2) 

The Certora spec confirms the revert condition: `claimContinuousFee` succeeds iff `amount <= withdrawable && amount <= continuousFeeCredit`: [4](#0-3) 

**Exploit flow (concrete numbers):**

| Step | Action | `withdrawable` | `continuousFeeCredit` | lender `credit` |
|---|---|---|---|---|
| 0 | Initial | 0 | 0 | 0 |
| 1 | `take(C)` | 0 | 0 | C |
| 2 | `repay(C - F)` (partial) | C - F | 0 | C |
| 3 | `updatePosition` (accrues F) | C - F | F | C - F |
| 4 | `withdraw(C - F)` | 0 | F | 0 |
| 5 | `claimContinuousFee(F)` | **REVERT** (0 < F) | F | 0 |

At step 4, `withdraw(C - F)` succeeds because `withdrawable = C - F >= C - F`. At step 5, `claimContinuousFee(F)` reverts because `withdrawable = 0 < F`.

The lender is an unprivileged actor who can freely choose when to call `withdraw`. The borrower only needs to repay exactly `C - F` — the minimum to allow the lender's full withdrawal — which is a natural and common scenario (e.g., borrower repays their net obligation after fee accrual has reduced the lender's credit).

**Why existing checks fail:**

There is no invariant enforcement anywhere that `withdrawable >= continuousFeeCredit`. The `withdraw` function has no lower-bound guard on `withdrawable` relative to `continuousFeeCredit`. The existing test `testClaimContinuousFee` always performs a full repay of `credit` (not `credit - fee`) before claiming, which masks the bug. [5](#0-4) 

### Impact Explanation

The accrued continuous fee `F` stored in `continuousFeeCredit` becomes permanently unclaimable: `claimContinuousFee(F)` always reverts because `withdrawable = 0 < F`. The fee claimer can never recover these funds. The tokens representing the fee remain locked in the contract with no mechanism to release them, since no other function increases `withdrawable` without a corresponding new repayment.

### Likelihood Explanation

This requires only: (1) a market with a non-zero continuous fee, (2) time elapsed so that `F > 0`, and (3) the borrower repaying exactly `C - F` (or any amount in `[C - F, C)`). Condition 3 is not adversarial — it is the natural repayment amount when a borrower repays their net obligation after fee accrual. The scenario is reachable by any lender calling `withdraw` with `units == credit` after `updatePosition`, which is the standard exit path. It is repeatable across any market with continuous fees.

### Recommendation

In `withdraw`, after decreasing `withdrawable`, enforce that `withdrawable` does not fall below `continuousFeeCredit`:

```solidity
_marketState.withdrawable -= UtilsLib.toUint128(units);
require(_marketState.withdrawable >= _marketState.continuousFeeCredit, WithdrawableUnderflow());
```

Alternatively, redesign the accounting so that `_updatePosition` also increases `withdrawable` by the accrued fee (treating the fee as immediately redeemable from the pool's repaid balance), or cap the withdrawable amount a lender can take to `credit - proportional_continuousFeeCredit`.

### Proof of Concept

```solidity
function testContinuousFeeUnclaimableAfterFullWithdraw() public {
    uint256 C = 1e18;
    uint256 feeRate = MAX_CONTINUOUS_FEE; // maximize fee accrual
    uint256 ttm = 365 days;
    uint256 elapsed = ttm / 2;

    // Step 1: setup lender with credit C
    setupLender(C, feeRate, ttm);

    // Step 2: warp and accrue fee F
    vm.warp(vm.getBlockTimestamp() + elapsed);
    midnight.updatePosition(market, lender);

    uint256 F = midnight.continuousFeeCredit(id);
    uint256 creditAfterFee = midnight.creditOf(id, lender);
    assertGt(F, 0, "fee must be positive");
    assertEq(creditAfterFee, C - F, "credit reduced by fee");

    // Step 3: borrower repays exactly creditAfterFee (not full C)
    deal(address(loanToken), borrower, creditAfterFee);
    vm.prank(borrower);
    midnight.repay(market, creditAfterFee, borrower, address(0), hex"");

    assertEq(midnight.withdrawable(id), creditAfterFee, "withdrawable == creditAfterFee");

    // Step 4: lender withdraws all credit
    vm.prank(lender);
    midnight.withdraw(market, creditAfterFee, lender, lender);

    assertEq(midnight.withdrawable(id), 0, "withdrawable is now 0");
    assertEq(midnight.continuousFeeCredit(id), F, "continuousFeeCredit still F");

    // Step 5: fee claimer cannot claim — REVERTS
    vm.prank(feeClaimer);
    vm.expectRevert(); // uint128 underflow: withdrawable(0) - F
    midnight.claimContinuousFee(market, F, feeClaimer);
}
```

Expected assertions: `continuousFeeCredit == F > 0`, `withdrawable == 0`, `claimContinuousFee(F)` reverts. The fee is permanently locked.
