Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via `mulDivDown` rounding to zero — (File: `src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and `tick < MAX_TICK`, a taker can supply a `units` value small enough that `units.mulDivDown(buyerPrice, WAD) == 0`. The `consumed` mapping is incremented by zero, so `require(newConsumed <= offer.maxAssets)` always passes. Despite zero token movement, `buyerPos.credit` and `sellerPos.debt` are updated by the full `units` value, allowing the offer to be filled indefinitely beyond its stated cap.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // truncates to 0 when units * buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [1](#0-0) 

`tickToPrice` in `src/libraries/TickLib.sol` defines `MAX_TICK = 5820` and `PRICE_ROUNDING_STEP = 1e12`. At `MAX_TICK`, `tickToPrice` returns exactly `WAD`. At any tick below `MAX_TICK`, `buyerPrice < WAD`. [2](#0-1) [3](#0-2) 

For `units = 1` at any tick below `MAX_TICK`: `1 * (WAD - ε) / WAD = 0` in integer arithmetic. The bypass window spans `units ∈ [1, WAD/buyerPrice - 1]`, which grows to millions of units at low ticks.

**Position effects still execute at lines 408–414:**

```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);  // based on units
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);  // based on units
``` [4](#0-3) 

At line 455, the token transfer is `buyerAssets - sellerAssets = 0 - 0 = 0`, so no tokens move. [5](#0-4) 

**Existing guard is insufficient:** The only protection is `require(newConsumed <= offer.maxAssets)`, which is trivially satisfied when the increment is 0. There is no floor check on `buyerAssets` or `units`. [6](#0-5) 

**Confirmed by existing test:** `testBugBuyMaxAssetsBypass()` in `test/TakeTest.sol` explicitly names and demonstrates this bypass path. [7](#0-6) 

## Impact Explanation
The `maxAssets` cap on buy offers is completely unenforceable at any tick below `MAX_TICK`. A colluding maker-taker pair can give the maker unlimited credit at zero token cost: `buyerAssets = 0` means no tokens are transferred, yet `buyerPos.credit` grows without bound. The taker accumulates unbounded debt at zero cost. This violates the core protocol invariant that offers cannot be overfilled and constitutes a direct accounting integrity failure enabling unauthorized credit creation.

## Likelihood Explanation
All preconditions are reachable by any unprivileged user: `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick < MAX_TICK` are standard offer parameters. At `MAX_TICK - 1`, `units = 1` suffices to trigger the bypass. The attack is repeatable in a loop with no cooldown, no capital requirement, and no rate limiting. The only cost is gas.

## Recommendation
Add a check that `buyerAssets > 0` whenever `units > 0` for buy offers, or enforce a minimum `units` value such that `units.mulDivDown(buyerPrice, WAD) >= 1`. Alternatively, use `mulDivUp` when computing the `consumed` increment for buy offers so that even a fractional asset rounds up to 1, preventing zero-increment fills.

## Proof of Concept
The protocol's own test suite contains `testBugBuyMaxAssetsBypass()` in `test/TakeTest.sol`, which pre-consumes an offer to `maxAssets`, then calls `take(1, ...)` and asserts that `consumed` stays at `maxAssets` while `creditOf(lender)` and `debtOf(borrower)` both increase — directly confirming the bypass.
