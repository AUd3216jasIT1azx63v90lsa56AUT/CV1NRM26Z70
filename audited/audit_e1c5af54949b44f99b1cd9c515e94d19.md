### Title
Authorized agent can deauthorize co-authorized agents via `setIsAuthorized`, causing DoS on `withdrawCollateral` - (File: src/periphery/interfaces/IEcrecoverAuthorizer.sol / src/Midnight.sol)

### Summary
`Midnight.setIsAuthorized` permits any authorized agent of a principal to modify the principal's authorization mapping, including setting `isAuthorized[principal][otherAgent] = false`. A malicious co-authorized agent can exploit this to deauthorize a collateral manager agent, causing all subsequent `withdrawCollateral` calls by that agent to revert with `Unauthorized()`. The same path is reachable through `EcrecoverAuthorizer.setIsAuthorized` since it accepts a signature from any currently-authorized agent of the authorizer.

### Finding Description

**Root cause — `Midnight.setIsAuthorized` (src/Midnight.sol:731-735):**

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    ...
}
```

The guard `isAuthorized[onBehalf][msg.sender]` passes for any currently-authorized agent, not only the principal. There is no restriction preventing an agent from writing `false` into another agent's slot.

**`EcrecoverAuthorizer.setIsAuthorized` (src/periphery/EcrecoverAuthorizer.sol:33-36) has the same permissiveness:**

```solidity
require(
    signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

A signer who is merely an authorized agent of `authorization.authorizer` satisfies this check and can submit an `Authorization{isAuthorized: false}` for any other agent.

**`withdrawCollateral` authorization check (src/Midnight.sol:556):**

```solidity
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
```

Once `isAuthorized[borrower][collateralManager]` is flipped to `false`, every call by `collateralManager` with `onBehalf = borrower` reverts.

**Exploit flow:**

1. Borrower authorizes both `collateralManager` and `attacker`:
   - `isAuthorized[borrower][collateralManager] = true`
   - `isAuthorized[borrower][attacker] = true`
2. Borrower has an active debt position (collateral locked by health check).
3. Attacker calls `midnight.setIsAuthorized(collateralManager, false, borrower)` directly (or via `EcrecoverAuthorizer` with a self-signed `Authorization{authorizer: borrower, authorized: collateralManager, isAuthorized: false}`).
4. Guard passes: `isAuthorized[borrower][attacker] == true`.
5. State: `isAuthorized[borrower][collateralManager] = false`.
6. `collateralManager` calls `withdrawCollateral(market, idx, assets, borrower, receiver)`.
7. Check at line 556 fails: `borrower != collateralManager` and `isAuthorized[borrower][collateralManager] == false` → reverts `Unauthorized()`.

The existing test `testSetIsAuthorizedAuthorization` in `test/AuthorizationTest.sol:290-304` explicitly confirms that an authorized agent can call `setIsAuthorized` on behalf of the principal, demonstrating the behavior is present and unconstrained.

### Impact Explanation
Any authorized agent can permanently (until the borrower re-authorizes) block a collateral manager from calling `withdrawCollateral` on the borrower's behalf. If the borrower is a smart contract that routes all collateral management through the designated agent and has no independent execution path to call `withdrawCollateral` directly, the collateral is effectively trapped for as long as the debt position exists (health check prevents withdrawal). Even for EOA borrowers, the attack forces the borrower to re-authorize the manager in a separate transaction, which can be front-run repeatedly at negligible cost (single `setIsAuthorized` call).

### Likelihood Explanation
**Preconditions:**
- Attacker must be an authorized agent of the borrower — a realistic scenario for any protocol that grants multiple agents (e.g., a keeper, a bundler, a ratifier) authorization over the same account.
- Borrower must have an active debt position (otherwise `withdrawCollateral` would succeed anyway via the health check).

**Feasibility:** The attack requires a single on-chain call with no capital. It is repeatable: every time the borrower re-authorizes the collateral manager, the attacker can immediately deauthorize it again. The `EcrecoverAuthorizer` path requires signing an `Authorization` struct, but the nonce is a public storage variable (`nonce(authorizer)`), so the attacker can always construct a valid message.

### Recommendation
Restrict `setIsAuthorized` so that only the principal itself (`onBehalf == msg.sender`) can revoke authorizations (`newIsAuthorized == false`). Authorized agents should only be permitted to grant new authorizations, not revoke existing ones:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    if (!newIsAuthorized) {
        require(onBehalf == msg.sender, Unauthorized()); // only principal can revoke
    } else {
        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    }
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
}
```

Apply the same restriction in `EcrecoverAuthorizer.setIsAuthorized` by requiring `signer == authorization.authorizer` when `authorization.isAuthorized == false`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {IMidnight} from "../src/interfaces/IMidnight.sol";

contract DeauthorizeDoSTest is BaseTest {
    function testAttackerDeauthorizesCollateralManager() public {
        address collateralManager = makeAddr("collateralManager");
        address attacker          = makeAddr("attacker");
        uint256 collateralAmount  = 1000;
        address collateralToken   = market.collateralParams[0].token;

        // 1. Borrower authorizes both collateralManager and attacker
        vm.startPrank(borrower);
        midnight.setIsAuthorized(collateralManager, true, borrower);
        midnight.setIsAuthorized(attacker, true, borrower);
        vm.stopPrank();

        // 2. Borrower supplies collateral (no debt, so health check passes for withdrawal)
        deal(collateralToken, borrower, collateralAmount);
        vm.prank(borrower);
        ERC20(collateralToken).approve(address(midnight), collateralAmount);
        vm.prank(borrower);
        midnight.supplyCollateral(market, 0, collateralAmount, borrower);

        // 3. Attacker deauthorizes collateralManager on behalf of borrower
        vm.prank(attacker);
        midnight.setIsAuthorized(collateralManager, false, borrower);

        // 4. Assert: collateralManager's withdrawCollateral now reverts
        vm.prank(collateralManager);
        vm.expectRevert(IMidnight.Unauthorized.selector);
        midnight.withdrawCollateral(market, 0, collateralAmount, borrower, collateralManager);

        // 5. Assert: borrower's authorization mapping is corrupted
        assertFalse(midnight.isAuthorized(borrower, collateralManager));
    }
}
```

- `vm.expectRevert(IMidnight.Unauthorized.selector)` passes — `collateralManager` cannot withdraw.
