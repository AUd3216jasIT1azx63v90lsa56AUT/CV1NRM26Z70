### Title
`liquidate` Permanently Reverts When `badDebt > totalUnits`, Locking Borrower Collateral — (File: src/Midnight.sol)

### Summary

The `liquidate` function in `Midnight.sol` computes `_totalUnits - badDebt` inside the bad-debt realization block. When `badDebt` exceeds `totalUnits` — a reachable state when lenders have withdrawn more credit than the remaining borrower's outstanding debt — this subtraction underflows and reverts in Solidity 0.8+. The result is a permanent DoS on liquidation for those positions, locking the borrower's collateral in the contract with no recovery path.

### Finding Description

In `Midnight.sol`, the `liquidate` function contains the following bad-debt realization block: [1](#0-0) 

```solidity
if (badDebt > 0) {
    _position.debt -= uint128(badDebt);
    uint256 _totalUnits = _marketState.totalUnits;
    uint256 _lossFactor = _marketState.lossFactor;
    _marketState.lossFactor = UtilsLib.toUint128(
        type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
    );
    _marketState.totalUnits -= UtilsLib.toUint128(badDebt);
    ...
}
```

The expression `_totalUnits - badDebt` is a checked `uint256` subtraction. If `badDebt > _totalUnits`, it underflows and reverts. There is no guard against this condition.

`totalUnits` tracks total lender credit. `badDebt` is bounded by `originalDebt` (the borrower's `uint128` debt field). The protocol allows lenders to withdraw credit independently of outstanding borrower debt — `withdraw` only checks `withdrawable`, not whether debt remains: [2](#0-1) 

This means `totalUnits` can fall below a single borrower's debt when other borrowers repay and lenders withdraw the resulting `withdrawable`. Once `totalUnits < badDebt`, every call to `liquidate` for that borrower reverts permanently.

**Concrete reachable path:**

1. Lender A takes a buy offer for 200 units → `totalUnits = 200`, Borrower B gets 100 debt, Borrower C gets 100 debt.
2. Borrower C repays 100 → `withdrawable += 100`.
3. Lender A withdraws 150 → `totalUnits = 50`, `withdrawable = 0`.
4. Now `totalUnits = 50`, Borrower B still has `debt = 100`.
5. Borrower B's collateral price drops; `badDebt = 100 > totalUnits = 50`.
6. `liquidate` is called → `_totalUnits - badDebt = 50 - 100` → **underflow revert**.

The `liquidate` function has no check preventing this: [3](#0-2) 

Only `_position.debt > 0` is required; there is no `require(badDebt <= totalUnits)` guard.

### Impact Explanation

- **Liquidation is permanently broken** for any position where `badDebt > totalUnits`. Every call to `liquidate` for that borrower reverts.
- **Borrower's collateral is permanently locked** in the contract. The borrower cannot `withdrawCollateral` (they have debt, so `isHealthy` fails). Liquidators cannot seize it. The only exit is the borrower repaying their full debt — impossible if they are insolvent.
- This is a **permanent fund freeze** matching the highest-priority bug class in `live_context.json`: "permanent or long-term fund freeze." [4](#0-3) 

### Likelihood Explanation

The precondition (`badDebt > totalUnits`) is reachable through normal, unprivileged protocol usage:

- Multiple borrowers in one market (common in a permissionless protocol).
- One borrower repays, enabling a lender to withdraw more credit than the remaining borrower's debt.
- The remaining borrower becomes insolvent.

No privileged access, oracle manipulation, or exotic token behavior is required. Any market with at least two borrowers and one lender is susceptible.

### Recommendation

Before computing `_totalUnits - badDebt`, clamp `badDebt` to `_totalUnits`:

```solidity
if (badDebt > 0) {
    _position.debt -= uint128(badDebt);
    uint256 _totalUnits = _marketState.totalUnits;
    uint256 _lossFactor = _marketState.lossFactor;
    uint256 effectiveBadDebt = badDebt > _totalUnits ? _totalUnits : badDebt;
    _marketState.lossFactor = effectiveBadDebt < _totalUnits
        ? UtilsLib.toUint128(
            type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - effectiveBadDebt, _totalUnits)
          )
        : type(uint128).max;
    _marketState.totalUnits -= UtilsLib.toUint128(effectiveBadDebt);
    ...
}
```

When `badDebt >= totalUnits`, set `lossFactor = type(uint128).max` (all remaining credit is wiped) and `totalUnits = 0`. This mirrors the existing `_lastLossFactor == type(uint128).max` guard in `updatePositionView`: [5](#0-4) 

### Proof of Concept

```solidity
// Setup:
// - Market with loanToken = USDC, one collateral
// - Lender A takes buy offer: 200 units credit, totalUnits = 200
// - Borrower B takes sell offer: 100 units debt
// - Borrower C takes sell offer: 100 units debt
// - Borrower C repays 100: withdrawable = 100
// - Lender A withdraws 150: totalUnits = 50, withdrawable = 0
// - Oracle drops: Borrower B collateral value < debt, badDebt = 100

// Call:
midnight.liquidate(market, 0, 0, 0, borrowerB, false, receiver, address(0), "");
// Result: REVERT (arithmetic underflow at `_totalUnits - badDebt = 50 - 100`)
// Borrower B's collateral is permanently locked.
``` [1](#0-0)
