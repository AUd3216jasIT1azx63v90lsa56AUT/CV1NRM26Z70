### Title
Fee-on-transfer loanToken causes `claimableSettlementFee` overstatement, leading to settlement fee undercollection - (File: src/Midnight.sol)

### Summary
In `Midnight.take`, `claimableSettlementFee` is incremented by the full `buyerAssets - sellerAssets` at line 418 before any token transfer occurs. When the `loanToken` is a fee-on-transfer token and `offer.callback` is set (making the callback contract the payer), `safeTransferFrom` at line 455 delivers only `(buyerAssets - sellerAssets) * (1 - fee_rate)` to `address(this)`, while the accounting records the full nominal amount. No post-transfer balance check exists to detect the shortfall.

### Finding Description
**Code path:**

1. Line 418: `claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets` — full nominal fee credited to accounting.
2. Line 420: `buyerCallback = offer.buy ? offer.callback : takerCallback` — for a buy offer, the maker-controlled `offer.callback` becomes `buyerCallback`.
3. Line 422: `payer = buyerCallback != address(0) ? buyerCallback : ...` — callback contract is designated payer.
4. Lines 445–453: `onBuy` is called on `buyerCallback`; it returns `CALLBACK_SUCCESS`.
5. Line 455: `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets)` — with a fee-on-transfer token, Midnight receives only `(buyerAssets - sellerAssets) * (1 - fee_rate)`.

**Root cause:** `claimableSettlementFee` is updated before the transfer, and `SafeTransferLib.safeTransferFrom` only validates the boolean return of `transferFrom` — it never measures the actual balance increase at `address(this)`.

**Attacker inputs:**
- Deploy a fee-on-transfer ERC20 token (e.g., 10% fee on every transfer).
- Call `touchMarket` with this token as `loanToken` (permissionless).
- Construct a buy offer (`offer.buy = true`) with `offer.callback` pointing to an attacker-controlled contract that: (a) holds fee-on-transfer tokens, (b) has approved Midnight, (c) returns `CALLBACK_SUCCESS` from `onBuy`.
- Use a second address as taker (to satisfy `offer.maker != taker`).
- Call `take`.

**Why existing checks fail:**
- `SafeTransferLib` checks only the `transferFrom` return value, not the received amount.
- `touchMarket` imposes no restriction on `loanToken` type.
- The Certora formal verification explicitly assumes standard ERC20 behavior and does not cover this case.
- No post-transfer invariant check (e.g., `balanceOf(address(this))` delta) exists anywhere in `take`.

### Impact Explanation
After each such `take`, `claimableSettlementFee[token]` exceeds the actual token balance held by Midnight for that token by `(buyerAssets - sellerAssets) * fee_rate`. Repeated calls accumulate the discrepancy. When `claimSettlementFee` is eventually called, it attempts to `safeTransfer` the full recorded amount, which exceeds the actual balance, causing a revert. The settlement fee is permanently undercollected and unclaimable, constituting protocol insolvency for that token's fee accounting.

### Likelihood Explanation
Market creation is fully permissionless — any address can call `touchMarket` with an arbitrary `loanToken`. The attacker controls both the fee-on-transfer token and the callback contract. No special privilege, oracle manipulation, or user mistake is required. The exploit is repeatable across multiple `take` calls to accumulate the discrepancy. Any existing market whose `loanToken` happens to be a fee-on-transfer token (e.g., tokens with built-in transfer taxes) is also affected without any attacker-deployed token.

### Recommendation
Add a balance check around the settlement fee transfer to enforce that Midnight actually received the expected amount:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
require(
    IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore == buyerAssets - sellerAssets,
    FeeOnTransferNotSupported()
);
```

Alternatively, document and enforce that fee-on-transfer tokens are not supported as `loanToken` by adding a token registry or a `require` in `touchMarket` that validates the token does not take a fee (e.g., via a round-trip transfer check at market creation time).
