### Title
`buyerAssetsToUnits` reverts with `PriceGreaterThanOne` for valid sell offers near MAX_TICK when settlement fee is non-zero, permanently blocking `buyWithAssetsTargetAndWithdrawCollateral` - (`File: src/periphery/TakeAmountsLib.sol`)

### Summary

`TakeAmountsLib.buyerAssetsToUnits` enforces `require(buyerPrice <= WAD)` as a periphery-only guard, but `Midnight.take()` has no such check and succeeds for any `buyerPrice`. When a sell offer is placed at a tick near `MAX_TICK` and the market has a non-zero settlement fee, `offerPrice + settlementFee > WAD` is reachable, causing `buyerAssetsToUnits` to revert with `PriceGreaterThanOne`. Because this call is outside the try/catch in `buyWithAssetsTargetAndWithdrawCollateral`, the entire bundle transaction reverts, permanently blocking the taker from using the periphery for those offers even though the core `take()` would succeed.

### Finding Description

**Code path and root cause:**

In `TakeAmountsLib.buyerAssetsToUnits` for a sell offer (`offer.buy = false`):

```
sellerPrice = offerPrice          // line 26
buyerPrice  = offerPrice + settlementFee  // line 27
require(buyerPrice <= WAD, PriceGreaterThanOne())  // line 28 — REVERTS
``` [1](#0-0) 

In `Midnight.take()` for the same sell offer, the identical computation is performed but **no `buyerPrice <= WAD` check exists**:

```
sellerPrice = offerPrice          // line 361
buyerPrice  = offerPrice + settlementFee  // line 362
buyerAssets = units.mulDivUp(buyerPrice, WAD)  // line 363 — proceeds without check
``` [2](#0-1) 

The core protocol itself documents this asymmetry: "All sellerAssets are reachable with the units input, and all buyerAssets are reachable only if buyerPrice <= WAD" — meaning `take()` is intentionally designed to work even when `buyerPrice > WAD`. [3](#0-2) 

**Exploit flow in `buyWithAssetsTargetAndWithdrawCollateral`:**

The `buyerAssetsToUnits` call at lines 209–211 is **outside** the try/catch block. If it reverts, the entire transaction reverts. The `take()` call at lines 215–221 is inside try/catch and would be skipped on failure — but it is never reached. [4](#0-3) 

The natspec at line 175 explicitly acknowledges this: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." [5](#0-4) 

**Attacker-controlled inputs:** The maker (an unprivileged user) places a sell offer at any tick T where `tickToPrice(T) + settlementFee(id, ttm) > WAD`. With `MAX_TICK = 5820`, `tickToPrice(MAX_TICK)` is just below WAD; any non-zero settlement fee pushes `buyerPrice` above WAD. The taker then calls `buyWithAssetsTargetAndWithdrawCollateral` with that offer in `takes[]`.

**Existing protections are insufficient:** The only protection is the try/catch around `take()` itself, which does not cover `buyerAssetsToUnits`. `ConsumableUnitsLib.consumableUnits` also calls `buyerAssetsToUnits` (line 19) and would propagate the same revert. [6](#0-5) 

**Test suite confirms the divergence:** `testSnappedBuyerAssetsBuyerIsLender` and `testSnappedBuyerAssetsBuyerIsBorrower` explicitly test `take()` with `buyerPrice = tickToPrice(MAX_TICK) + settlementFee > WAD` and assert it succeeds — bypassing `buyerAssetsToUnits` entirely. [7](#0-6) 

`testBuyBuyerAssetsTarget` resets all settlement fees to 0 before calling `buyWithAssetsTargetAndWithdrawCollateral`, with the comment "Reset settlement fees so buyerPrice = price <= WAD at MAX_TICK" — confirming the authors knew non-zero fees at high ticks would cause `buyerAssetsToUnits` to revert. [8](#0-7) 

### Impact Explanation

Any taker calling `buyWithAssetsTargetAndWithdrawCollateral` with a sell offer at tick T where `tickToPrice(T) + settlementFee > WAD` will have the entire bundle revert. The taker cannot use this periphery function for those offers at all, even though `Midnight.take()` would accept them. This is a permanent, unconditional DoS of the periphery take path for all sell offers near `MAX_TICK` whenever the market settlement fee is non-zero — a normal market condition.

### Likelihood Explanation

**Preconditions:**
1. Market has a non-zero settlement fee at the current TTM breakpoint — a routine governance configuration.
2. A maker places a sell offer at tick T where `tickToPrice(T) + settlementFee > WAD`. With any non-zero fee, this is achievable at `MAX_TICK` or ticks just below it.
3. A taker calls `buyWithAssetsTargetAndWithdrawCollateral` with that offer.

All three preconditions are normal market operations. No privileged action is required from the attacker. The DoS is repeatable and deterministic: every call with such an offer will revert.

### Recommendation

Remove the `require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne())` check from `buyerAssetsToUnits` for sell offers, or handle the `buyerPrice > WAD` case by returning `targetBuyerAssets.mulDivDown(WAD, buyerPrice)` (which is mathematically valid even when `buyerPrice > WAD`). Alternatively, wrap the `buyerAssetsToUnits` and `consumableUnits` calls in `buyWithAssetsTargetAndWithdrawCollateral` in a try/catch so that offers where `buyerPrice > WAD` are skipped rather than causing a full revert — consistent with how `take()` failures are already handled.

### Proof of Concept

```solidity
// Foundry fuzz test
function testBuyerAssetsToUnitsDoS(uint256 settlementFee0) public {
    // Bound to a non-zero settlement fee
    settlementFee0 = bound(settlementFee0, 1e12, maxSettlementFee(0));
    settlementFee0 = settlementFee0 / 1e12 * 1e12;
    midnight.setMarketSettlementFee(id, 0, settlementFee0);
    midnight.touchMarket(market);

    uint256 settlementFee = midnight.settlementFee(id, market.maturity - block.timestamp);
    uint256 offerPrice = TickLib.tickToPrice(MAX_TICK);

    // Precondition: buyerPrice > WAD
    vm.assume(offerPrice + settlementFee > WAD);

    offer.tick = MAX_TICK;
    offer.maker = borrower;
    offer.receiverIfMakerIsSeller = borrower;
    offer.maxUnits = type(uint256).max;

    uint256 units = 1e18;
    collateralize(market, borrower, units);

    // Assert: core take() succeeds
    deal(address(loanToken), lender, type(uint256).max);
    (uint256 buyerAssets,) = take(units, lender, offer);
    assertGt(buyerAssets, 0, "core take should succeed");

    // Assert: buyerAssetsToUnits reverts with PriceGreaterThanOne
    vm.expectRevert(TickLib.PriceGreaterThanOne.selector);
    TakeAmountsLib.buyerAssetsToUnits(address(midnight), id, offer, 1e18);

    // Assert: buyWithAssetsTargetAndWithdrawCollateral reverts (not just skips)
    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offer, units: type(uint256).max, ratifierData: hex""});
    deal(address(loanToken), lender, type(uint256).max);
    vm.prank(lender);
    vm.expectRevert(TickLib.PriceGreaterThanOne.selector);
    midnightBundles.buyWithAssetsTargetAndWithdrawCollateral(
        1e18, 0, lender, _noPermit(), takes,
        new CollateralWithdrawal[](0), address(0), 0, address(0)
    );
}
```

**Expected assertions:**
- `Midnight.take()` with `units` succeeds and returns non-zero `buyerAssets`.
- `TakeAmountsLib.buyerAssetsToUnits(...)` reverts with `PriceGreaterThanOne`.
- `midnightBundles.buyWithAssetsTargetAndWithdrawCollateral(...)` reverts with `PriceGreaterThanOne` (not `OutOfOffers`), confirming the DoS originates in `buyerAssetsToUnits` before any `take()` is attempted.
