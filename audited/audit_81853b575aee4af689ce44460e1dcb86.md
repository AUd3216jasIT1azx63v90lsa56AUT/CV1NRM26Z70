### Title
Callback payer confusion in `repay()` allows attacker to drain loanToken from any contract that has approved Midnight and implements `onRepay()` - (File: src/Midnight.sol)

### Summary
In `repay()`, the `payer` is unconditionally set to `callback` when `callback != address(0)` (line 511), with no check that `callback` is `msg.sender` or has authorized `msg.sender` to act on its behalf. An attacker who controls `onBehalf` (set to themselves) can supply any victim contract as `callback`, causing `safeTransferFrom` at line 520 to pull repayment tokens from the victim rather than the attacker, clearing the attacker's debt at the victim's expense.

### Finding Description
**Code path:**

`src/Midnight.sol` → `repay()` lines 502–521:

```
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized()); // line 505
...
address payer = callback != address(0) ? callback : msg.sender;                         // line 511
...
IRepayCallback(callback).onRepay(id, market, units, onBehalf, data) == CALLBACK_SUCCESS // line 516
...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);        // line 520
```

**Root cause:** The authorization check at line 505 guards only `onBehalf`, not `callback`. There is no `require(callback == msg.sender || isAuthorized[callback][msg.sender])` or equivalent. The `callback` address is fully attacker-controlled, and when non-zero it becomes `payer` — the address from which loanToken is pulled.

**Attacker inputs:**
- `onBehalf = msg.sender` (attacker's own address — passes auth check trivially)
- `callback = victimContract` (any contract with Midnight approval and a compliant `onRepay()`)
- `units` = attacker's outstanding debt (or any amount ≤ victim's balance/allowance)

**Exploit flow:**
1. Attacker has existing debt of `units` in a market.
2. Victim contract (`victimContract`) has previously approved Midnight for ≥ `units` loanToken and holds ≥ `units` loanToken balance. It implements `IRepayCallback.onRepay()` returning `CALLBACK_SUCCESS` (e.g., any legitimate repay-callback helper, including the pattern shown in `test/OtherFunctionsTest.sol` lines 709–734).
3. Attacker calls `midnight.repay(market, units, attacker, victimContract, data)`.
4. Line 505: `onBehalf == msg.sender` → passes.
5. Line 511: `payer = victimContract`.
6. Lines 514–518: `victimContract.onRepay(...)` is called → returns `CALLBACK_SUCCESS`.
7. Line 520: `safeTransferFrom(loanToken, victimContract, address(this), units)` — pulls `units` loanToken from `victimContract`.
8. Attacker's debt is zeroed; victim's balance decreases by `units`.

**Why existing checks fail:** The only authorization check (`isAuthorized[onBehalf][msg.sender]`) is scoped to `onBehalf`. There is no analogous check binding `callback` to `msg.sender`. The callback's `onRepay()` return value is verified, but a victim contract that returns `CALLBACK_SUCCESS` unconditionally (or after logic that doesn't detect the abuse) cannot prevent the subsequent `safeTransferFrom`.

### Impact Explanation
Any contract that (a) holds loanToken, (b) has approved Midnight, and (c) implements `onRepay()` returning `CALLBACK_SUCCESS` can have its entire approved loanToken balance drained by an unprivileged borrower in a single transaction. The attacker's debt is simultaneously cleared, constituting a direct theft of funds and a violation of the invariant that user funds must not be stolen via callbacks.

### Likelihood Explanation
**Preconditions:**
1. Victim contract has approved Midnight for loanToken — satisfied by any legitimate repay-callback contract (e.g., bundlers, routers, or helper contracts that pre-approve Midnight).
2. Victim implements `onRepay()` returning `CALLBACK_SUCCESS` — satisfied by any contract conforming to `IRepayCallback`.
3. Attacker has any outstanding debt — trivially achievable by borrowing.

The attack is repeatable, requires no special privilege, and can be executed in a single transaction. Any deployed repay-callback helper (such as `MidnightBundles` if it ever holds a loanToken approval and exposes `onRepay`) is a candidate victim.

### Recommendation
Bind `callback` to `msg.sender`. The simplest fix is to require that the caller is the callback when a non-zero callback is supplied:

```solidity
if (callback != address(0)) {
    require(callback == msg.sender, Unauthorized());
    // ... existing onRepay call ...
}
```

Alternatively, mirror the `onBehalf` pattern and require `isAuthorized[callback][msg.sender]`, so only explicitly authorized callers may designate a callback as payer. This matches the design intent that the callback contract is the entity funding the repayment on its own behalf.
