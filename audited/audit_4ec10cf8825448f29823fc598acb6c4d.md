### Title
Unbounded `_poolsByAgreement` array causes `getPoolsByAgreement` OOG DoS in low-gas contexts — (`File: src/ConfidencePoolFactory.sol`)

### Summary
`getPoolsByAgreement` copies the entire `_poolsByAgreement[agreement]` storage array into memory with no pagination or size cap. Because `createPool` imposes no per-agreement pool limit, a sponsor (the agreement owner) can inflate the array to an arbitrary length, making any caller that invokes `getPoolsByAgreement` inside a gas-constrained context (e.g. a 100 000-gas callback) revert with out-of-gas. No trusted-role compromise is required beyond the normal operational prerequisite that at least one stake token is allowlisted.

### Finding Description
`createPool` pushes every new clone address into `_poolsByAgreement[agreement]` (line 103) with no upper bound on the array length. The only per-call guards are: valid agreement, allowlisted token, expiry lead-time, and `IAgreement.owner() == msg.sender`. None of these limit how many times the same sponsor may call `createPool` for the same agreement. The code comment at line 34–35 explicitly acknowledges "An agreement may back many pools … duplicates are allowed and curated off-chain," confirming there is no on-chain cap.

`getPoolsByAgreement` (lines 117–119) is a single-statement `return _poolsByAgreement[agreement]` that copies the full storage array into a freshly allocated memory array. The EVM charges:
- 2 100 gas (cold SLOAD) for the array length slot
- 2 100 gas per element slot (cold) or 100 gas (warm) for each of the N address words
- Memory expansion proportional to N

At cold-slot pricing, ~47 elements exhaust 100 000 gas before the function returns. At warm-slot pricing (e.g. after the length slot warms the base slot), the threshold rises to roughly 500–1 000 elements, still reachable by a motivated sponsor. Each `createPool` call costs on the order of 300 000–500 000 gas (clone deployment + `initialize`), so inflating to the OOG threshold is expensive but one-time and permanent.

No existing guard in `getPoolsByAgreement` checks remaining gas, imposes a return-size limit, or offers pagination. `poolCountByAgreement` (line 122–124) is safe because it reads only the length slot, but callers that need the actual addresses have no safe alternative.

### Impact Explanation
Any on-chain contract that calls `getPoolsByAgreement` inside a gas-limited context — a callback, a `try/catch` with a forwarded gas stipend, a multicall aggregator, or a keeper bot — will revert with OOG once the array is large enough. The DoS is permanent for that agreement: the array can only grow, never shrink. Core protocol functions (`stake`, `claim`, `withdraw`, resolution paths) are unaffected, so the impact is scoped to integrators and off-chain tooling that rely on the view function for on-chain logic.

### Likelihood Explanation
The actor is the pool sponsor — the agreement owner — a semi-trusted but unprivileged role relative to the factory owner and moderator. The preconditions are: (a) own a valid agreement (normal operational state), (b) at least one token is allowlisted (required for the factory to be usable at all). The sponsor pays gas per pool but the cost is one-time; the DoS persists indefinitely. A griefing sponsor or a sponsor whose agreement becomes contested has clear incentive to inflate the array to neutralize integrators that gate on `getPoolsByAgreement`.

### Recommendation
Add a per-agreement pool cap in `createPool` (e.g. `require(_poolsByAgreement[agreement].length < MAX_POOLS_PER_AGREEMENT, PoolLimitReached())`), or replace the full-array return in `getPoolsByAgreement` with a paginated variant:

```solidity
function getPoolsByAgreement(address agreement, uint256 offset, uint256 limit)
    external view returns (address[] memory page)
{
    address[] storage all = _poolsByAgreement[agreement];
    uint256 end = offset + limit > all.length ? all.length : offset + limit;
    page = new address[](end - offset);
    for (uint256 i = offset; i < end; i++) page[i - offset] = all[i];
}
```

Both mitigations are in scope (`ConfidencePoolFactory.sol`). The paginated approach is preferable because it does not break existing pools and allows callers to bound their own gas usage.

### Proof of Concept

```solidity
pragma solidity 0.8.26;

import "forge-std/Test.sol";
import {ConfidencePoolFactory} from "src/ConfidencePoolFactory.sol";
// ... standard test setup imports

contract PoolConsumer {
    IConfidencePoolFactory public factory;
    constructor(address f) { factory = IConfidencePoolFactory(f); }

    /// Called with a forwarded gas limit to simulate a callback context.
    function process(address agreement) external returns (address[] memory) {
        return factory.getPoolsByAgreement(agreement);
    }
}

contract GetPoolsByAgreementOOGTest is Test {
    ConfidencePoolFactory factory;
    PoolConsumer consumer;
    address sponsor;
    address agreement; // mock agreement owned by sponsor

    function setUp() public {
        // deploy factory, mock registry, mock agreement, allowlist token
        // sponsor = agreement owner
    }

    function test_OOGAfterNPools() public {
        uint256 N = 60; // tune to threshold; ~47 at cold pricing for 100k gas
        for (uint256 i = 0; i < N; i++) {
            vm.prank(sponsor);
            factory.createPool(agreement, token, expiry, 0, recovery, accounts);
        }

        // Assert that getPoolsByAgreement succeeds with ample gas
        address[] memory all = factory.getPoolsByAgreement(agreement);
        assertEq(all.length, N);

        // Assert that PoolConsumer.process() OOGs with 100 000 gas
        (bool ok,) = address(consumer).call{gas: 100_000}(
            abi.encodeCall(PoolConsumer.process, (agreement))
        );
        assertFalse(ok, "expected OOG revert");
    }
}
```
