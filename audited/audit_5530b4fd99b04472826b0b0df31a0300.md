### Title
Inconsistent Liquidation Behavior When Oracle Price Is Zero Allows Free Collateral Seizure — (`src/Midnight.sol`)

### Summary

In `Midnight.liquidate()`, when the oracle for the liquidated collateral returns `0`, the two input paths behave inconsistently: passing `seizedAssets > 0` succeeds and computes `repaidUnits = 0` (free seizure), while passing `repaidUnits > 0` reverts with a division-by-zero. This is the direct analog of the reported `UniswapAnchoredView` inconsistency, transposed to Midnight's liquidation engine.

### Finding Description

In `liquidate()`, after the collateral loop resolves `liquidatedCollatPrice`, the two branches at lines 649–653 are:

```solidity
// Branch A: seizedAssets > 0
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);

// Branch B: repaidUnits > 0
seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
``` [1](#0-0) 

When `liquidatedCollatPrice == 0`:

**Branch A** (`seizedAssets > 0, repaidUnits = 0` as input):
```
mulDivUp(seizedAssets, 0, ORACLE_PRICE_SCALE)
  = (seizedAssets * 0 + (ORACLE_PRICE_SCALE - 1)) / ORACLE_PRICE_SCALE
  = 0
mulDivUp(0, WAD, lif) = (lif - 1) / lif = 0
→ repaidUnits = 0
```
The liquidator seizes `seizedAssets` tokens and repays **zero** debt units. The call **succeeds**.

**Branch B** (`repaidUnits > 0, seizedAssets = 0` as input):
```
mulDivDown(ORACLE_PRICE_SCALE, 0) → division by zero → REVERT
``` [2](#0-1) 

The protocol's own LIVENESS comment at line 146 acknowledges only the revert case ("If the liquidated collateral oracle returns 0 on price, liquidate with repaid input reverts") but does not address the free-seizure case in Branch A. [3](#0-2) 

**State after Branch A executes with price = 0:**
- `_position.collateral[collateralIndex]` is reduced by `seizedAssets` (line 670–671)
- `_position.debt` is unchanged (`repaidUnits = 0`, line 676)
- `_marketState.withdrawable` is unchanged (line 675) [4](#0-3) 

The borrower loses real collateral while their debt obligation is entirely preserved.

### Impact Explanation

A liquidator can drain a borrower's collateral for zero repaid units whenever the oracle for that collateral returns `0`. The borrower suffers a total loss of the seized collateral while retaining the full debt, leaving them in a worse position than before the liquidation. If the oracle is manipulated (e.g., via a flash-loan attack on a TWAP-based oracle) to transiently return `0`, an attacker can steal the collateral of any borrower who has that collateral activated and whose position is liquidatable (unhealthy or post-maturity). The protocol's own accounting is not directly harmed (no bad debt is realized), but individual borrowers suffer direct asset loss.

### Likelihood Explanation

The trigger requires the oracle to return `0` for the liquidated collateral. This can occur:
1. **Legitimately** — if the collateral genuinely loses all value (rare but possible).
2. **Via oracle manipulation** — a flash-loan attack on a TWAP or spot-price oracle can transiently push the reported price to zero. The SECURITY.md explicitly keeps oracle manipulation in scope ("This does not exclude oracle manipulation/flash-loan attacks"). [5](#0-4) 

No privileged access is required. Any external caller can invoke `liquidate()` with `seizedAssets > 0` once the oracle returns `0` and the position is liquidatable.

### Recommendation

Add an explicit guard at the start of the `repaidUnits > 0 || seizedAssets > 0` block to revert when `liquidatedCollatPrice == 0`, enforcing a single consistent behavior (disallow liquidation of a zero-priced collateral entirely):

```solidity
if (repaidUnits > 0 || seizedAssets > 0) {
    require(liquidatedCollatPrice > 0, ZeroOraclePrice()); // <-- add this
    ...
}
```

This mirrors the approach suggested in the external report: define one behavior for the zero-price case rather than allowing one path to silently succeed with a zero repayment.

### Proof of Concept

**Setup:**
- Market with one collateral whose oracle is manipulable to return `0`.
- Borrower has `debt = 1000` units and `collateral[0] = 500e18` tokens.
- Oracle is manipulated to return `price = 0`.
- Position becomes liquidatable because `maxDebt = 0 < debt = 1000`.

**Attack call:**
```solidity
midnight.liquidate(
    market,
    0,              // collateralIndex
    500e18,         // seizedAssets = full collateral
    0,              // repaidUnits = 0 (input)
    borrower,
    false,          // normal mode
    attacker,
    address(0),
    ""
);
```

**Execution trace:**
1. `liquidatedCollatPrice = 0` (from oracle)
2. `maxDebt = 0`, `originalDebt = 1000` → `originalDebt > maxDebt` → liquidatable ✓
3. Branch A: `repaidUnits = mulDivUp(500e18, 0, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif) = 0`
4. `_position.collateral[0] -= 500e18` → borrower loses all collateral
5. `_position.debt -= 0` → borrower's debt unchanged at 1000
6. `SafeTransferLib.safeTransfer(collateralToken, attacker, 500e18)` → attacker receives 500e18 tokens
7. `SafeTransferLib.safeTransferFrom(loanToken, attacker, address(this), 0)` → attacker pays nothing

**Result:** Attacker receives 500e18 collateral tokens for free. Borrower retains 1000 units of debt with zero collateral. [6](#0-5)

    function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
