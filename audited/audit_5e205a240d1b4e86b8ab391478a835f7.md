### Title
Double `mulDivDown` Rounding Understates `maxDebt` by 1, Enabling Liquidation of Economically Healthy Positions - (File: src/Midnight.sol)

### Summary
Both `isHealthy` and `liquidate` compute `maxDebt` via two sequential `mulDivDown` calls: `collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)`. This double floor-rounding can produce a result that is 1 unit less than `floor(collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD))`, the single-step exact integer bound. When a position sits exactly at the health boundary (exact maxDebt = debt), the computed `maxDebt` equals `debt − 1`, causing `isHealthy` to return `false` and `liquidate`'s guard `originalDebt > maxDebt` to pass, allowing a liquidator to seize collateral from an economically healthy borrower.

**Root cause — double floor rounding:**

Let `r = (C × P) mod ORACLE_PRICE_SCALE` (the remainder discarded in step 1) and `s = (V1 × L) mod WAD` (the remainder discarded in step 2), where `V1 = floor(C × P / ORACLE_PRICE_SCALE)`.

The computed value is `V2 = floor(V1 × L / WAD)`.  
The single-step exact integer bound is `M = floor(C × P × L / (ORACLE_PRICE_SCALE × WAD))`.

`V2 < M` whenever `s × ORACLE_PRICE_SCALE + r × L ≥ ORACLE_PRICE_SCALE × WAD`, i.e., the two rounding remainders together exceed one unit. Since `r < ORACLE_PRICE_SCALE` and `L ≤ WAD`, the combined error is strictly less than 2, so `V2 ≥ M − 1`. The off-by-one is therefore the worst case and is reachable.

**Concrete counterexample** (using actual protocol constants `ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`, allowed `lltv = 0.77e18`):

| Parameter | Value |
|---|---|
| `collateral` (C) | 10 |
| `price` (P) | `1.3e35` |
| `lltv` (L) | `0.77e18` |
| `debt` (D) | 1 |

- Step 1: `V1 = floor(10 × 1.3e35 / 1e36) = floor(1.3) = 1`
- Step 2: `V2 = floor(1 × 0.77e18 / 1e18) = floor(0.77) = 0`
- Exact: `10 × 1.3e35 × 0.77e18 / 1e54 = 1.001` → `floor = 1 = debt` → **healthy**
- Protocol sees: `maxDebt = 0 < 1 = debt` → **unhealthy** → liquidation allowed

**Exploit flow:**

1. Borrower supplies 10 units of collateral and borrows D = 1 unit of debt when the oracle price is above `1.3e35` (so the computed `maxDebt ≥ 1` and `take` passes its `isHealthy` check at line 476).
2. Oracle price drifts to `1.3e35` — the position is economically healthy (exact maxDebt = 1 = debt).
3. Liquidator calls `liquidate(market, collateralIndex, seizedAssets, 0, borrower, false, ...)`.
4. Inside `liquidate`, `maxDebt` is computed as 0; the guard `originalDebt (1) > maxDebt (0)` passes; collateral is seized. [4](#0-3) [5](#0-4) 

**Why existing protections fail:**

The Certora `Healthiness.spec` proofs use abstract ghost summaries for `mulDivDown` with only monotonicity axioms — they do not model the concrete integer rounding behavior, so the off-by-one is invisible to the formal verifier. [6](#0-5) 

No buffer or tolerance is added anywhere in the health check path.

### Impact Explanation
A liquidator can seize collateral from a borrower whose position is economically healthy (exact `maxDebt ≥ debt`) purely because the double floor-rounding understates `maxDebt` by 1. The borrower suffers an unwarranted collateral loss and the core invariant — *healthy positions cannot be liquidated* — is violated.

### Likelihood Explanation
The condition requires the oracle price to land at a value where `(C × P) mod ORACLE_PRICE_SCALE ≠ 0` and the two remainders together exceed `ORACLE_PRICE_SCALE × WAD`. This is not a measure-zero set: for any fixed `C` and `L`, there is a positive-measure interval of prices `P` that triggers the off-by-one. A liquidator needs only to monitor positions and call `liquidate` when the price enters such an interval — no oracle manipulation is required. With many positions across many markets the probability of at least one being in the vulnerable range at any given time is non-negligible.

### Recommendation
Replace the two-step computation with a single `mulDivDown` over the combined denominator to eliminate the intermediate rounding:

```solidity
// Instead of:
maxDebt += collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);

// Use (requires 512-bit intermediate or a checked mulDiv library):
maxDebt += mulDiv(collateral, price * lltv, ORACLE_PRICE_SCALE * WAD); // single floor
```

`price × lltv ≤ 1e36 × 1e18 = 1e54` and `collateral ≤ 2^128`, so `collateral × price × lltv ≤ ~3.4e38 × 1e54 = 3.4e92`, which exceeds `uint256`; a 512-bit `mulDiv` (e.g., Solady's `FullMath.mulDiv`) is needed. Alternatively, add a 1-unit grace margin to the health check (`maxDebt + 1 >= debt`) in both `isHealthy` and `liquidate`, accepting that positions 1 unit underwater are not liquidatable — a negligible economic concession that restores the invariant.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {UtilsLib} from "src/libraries/UtilsLib.sol";
import {ORACLE_PRICE_SCALE, WAD} from "src/libraries/ConstantsLib.sol";

contract DoubleRoundingPoC is Test {
    using UtilsLib for uint256;

    // Unit test: demonstrate the off-by-one with concrete values
    function testDoubleRoundingOffByOne() public pure {
        uint256 collateral = 10;
        uint256 price      = 1.3e35;   // 0.13 * ORACLE_PRICE_SCALE
        uint256 lltv       = 0.77e18;  // LLTV_2, an allowed tier
        uint256 debt       = 1;

        // Two-step (protocol's current computation)
        uint256 computed = collateral.mulDivDown(price, ORACLE_PRICE_SCALE)
                                     .mulDivDown(lltv, WAD);

        // Single-step exact integer bound (using 256-bit; safe here because
        // collateral * price = 1.3e36 fits in uint256)
        uint256 exact = (collateral * price * lltv) / (ORACLE_PRICE_SCALE * WAD);

        // Assert the off-by-one exists
        assertEq(computed, 0,    "computed maxDebt should be 0 (rounded down twice)");
        assertEq(exact,    1,    "exact maxDebt should be 1");
        assertGe(exact,    debt, "position is economically healthy");
        assertLt(computed, debt, "but protocol sees it as unhealthy -> liquidatable");
    }

    // Fuzz test: find boundary cases across all allowed LLTV tiers
    function testFuzzDoubleRoundingBoundary(
        uint128 collateral,
        uint256 price,
        uint8   lltvIndex
    ) public pure {
        uint256[9] memory lltvs = [
            uint256(0.385e18), 0.625e18, 0.77e18, 0.86e18,
            0.915e18, 0.945e18, 0.965e18, 0.98e18, 1e18
        ];
        uint256 lltv = lltvs[lltvIndex % 9];
        price = bound(price, 1, ORACLE_PRICE_SCALE);

        uint256 computed = uint256(collateral).mulDivDown(price, ORACLE_PRICE_SCALE)
                                              .mulDivDown(lltv, WAD);

        // Exact single-step (may overflow for large collateral*price; skip those)
        if (collateral > 0 && price <= type(uint256).max / collateral) {
            uint256 numerator = uint256(collateral) * price;
            if (numerator <= type(uint256).max / lltv) {
                uint256 exact = (numerator * lltv) / (ORACLE_PRICE_SCALE * WAD);
                // Assert: double-rounding never understates by more than 1
                assertGe(computed + 1, exact, "off-by-more-than-one detected");
                // Log cases where the off-by-one actually occurs
                if (computed < exact) {
                    // debt == exact makes the position healthy but computed < debt
                    // -> liquidatable. This assertion should FAIL (proving the bug):
                    // assertGe(computed, exact); // uncomment to see failure
                }
            }
        }
    }
}
```

**Expected assertions:**
- `testDoubleRoundingOffByOne` passes, confirming `computed = 0 < 1 = exact = debt`.
- The fuzz test finds inputs where `computed < exact`, confirming the off-by-one is reachable across real LLTV tiers.
- An end-to-end integration test setting up a position with the above parameters, dropping the oracle to `1.3e35`, and calling `liquidate` would succeed (no revert), demonstrating the invariant violation.
