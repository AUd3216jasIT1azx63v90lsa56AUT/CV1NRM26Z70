### Title
Arithmetic Underflow in `take()` When `settlementFee > tickToPrice(offer.tick)` for Low-Tick Buy Offers - (File: src/Midnight.sol)

### Summary
When a maker signs a buy offer at a very low tick (e.g., tick=4 with `DEFAULT_TICK_SPACING=4`), `tickToPrice(4)` evaluates to `1e12`. The settlement fee for any market with a non-trivial fee schedule (even `MAX_SETTLEMENT_FEE_1_DAY = 1.4e13`) exceeds this value. Because Solidity 0.8.34 reverts on arithmetic underflow and there is no guard before the subtraction `sellerPrice = offerPrice - _settlementFee` at Midnight.sol:361, any taker attempting to fill such an offer will receive a revert, permanently DoS-ing `take()` for that offer.

### Finding Description

**Exact code path:** [1](#0-0) 

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);          // line 358
uint256 timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp);
uint256 _settlementFee = settlementFee(id, timeToMaturity);
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice; // line 361 — UNDERFLOW
```

There is no `require(offerPrice >= _settlementFee)` or equivalent guard anywhere between lines 346–361.

**Root cause — `tickToPrice(4)` is smaller than any non-trivial settlement fee:**

`tickToPrice` is defined as: [2](#0-1) 

With `tick=4`, `MAX_TICK=5820`, `LN_ONE_PLUS_DELTA=0.004987541511039073e18`:

- `wExp(0.004987541511039073e18 × (2910 − 4))` = `wExp(14.4987…e18)` ≈ `1.98e24`
- `divHalfDownUnchecked(1e36, 1e18 + 1.98e24)` ≈ `5.05e11`
- `divHalfDownUnchecked(5.05e11, 1e12) × 1e12` = `1 × 1e12` = **`1e12`**

So `tickToPrice(4) = 1e12`.

**Settlement fee constants:** [3](#0-2) 

- `MAX_SETTLEMENT_FEE_1_DAY  = 0.000014e18 = 1.4e13`
- `MAX_SETTLEMENT_FEE_360_DAYS = 0.005e18   = 5e15`

Even the minimum non-zero breakpoint (`1.4e13`) is **14× larger** than `tickToPrice(4) = 1e12`. The 360-day maximum (`5e15`) is **5000× larger**.

**Tick accessibility check passes:** [4](#0-3) 

`4 % DEFAULT_TICK_SPACING(4) == 0` — tick=4 is a valid, accessible tick.

**Attacker-controlled inputs:**
- Maker (unprivileged) signs an `Offer` struct with `offer.tick = 4` and `offer.buy = true`.
- The signature is valid off-chain; no on-chain state is needed to create it.
- Precondition: the market's settlement fee is set to any value above `1e12` (satisfied by any market with even a 1-day max fee).

**Exploit flow:**
1. Market is created/touched with `tickSpacing = 4` and settlement fee ≥ `1.4e13` (any 1-day+ market at max fee).
2. Attacker (maker) signs a buy offer at `tick=4`, broadcasts it.
3. Any taker calls `take(offer, ...)`.
4. Line 358: `offerPrice = 1e12`.
5. Line 360: `_settlementFee = 1.4e13` (or higher).
6. Line 361: `1e12 - 1.4e13` → **arithmetic underflow → revert** in Solidity 0.8.34.
7. Every taker for this offer is permanently blocked.

**Why existing checks fail:** The only tick-related check is `offer.tick % tickSpacing == 0` (line 351), which passes for tick=4. There is no check that `offerPrice >= _settlementFee` anywhere in `take()`.

### Impact Explanation

Any taker attempting to fill a buy offer at tick=4 (or any tick where `tickToPrice(tick) < _settlementFee`) will have their transaction revert unconditionally. The offer is permanently unfillable. If the maker has already broadcast the offer and the settlement fee is raised (or was always high), the offer cannot be taken, and the maker cannot force-cancel it on-chain (offers are off-chain signatures). This constitutes a targeted, low-cost DoS of `take()` for affected offers.

### Likelihood Explanation

**Preconditions:**
1. Market settlement fee must exceed `tickToPrice(4) = 1e12`. This is true for any market with `MAX_SETTLEMENT_FEE_1_DAY` or higher — i.e., virtually every real market.
2. Maker signs a buy offer at tick=4 (the lowest accessible tick with `DEFAULT_TICK_SPACING=4`).

Both conditions are trivially achievable. The attacker cost is zero (signing is free). The condition is permanent — once the offer is broadcast, every `take()` attempt reverts. The attack is repeatable across any market with a non-trivial settlement fee.

### Recommendation

Add a guard before line 361 to ensure the offer price covers the settlement fee for buy offers:

```solidity
if (offer.buy) require(offerPrice >= _settlementFee, SettlementFeeExceedsPrice());
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
```

Alternatively, enforce a minimum tick for buy offers such that `tickToPrice(minTick) >= maxSettlementFee(maxIndex)` at market creation, or use `zeroFloorSub` and propagate a zero `sellerPrice` as an error condition.

    }

    function test_take_underflow_with_low_tick_buy_offer() public {
        // Deploy market with settlement fee = MAX_SETTLEMENT_FEE_1_DAY
        // Maker signs buy offer at tick=4
        // Taker calls take() → expect revert(Panic: arithmetic underflow)
        // Expected: vm.expectRevert(stdError.arithmeticError)
        // Assertion: sellerPrice = tickToPrice(4) - settlementFee underflows
        uint256 offerPrice = TickLib.tickToPrice(4); // 1e12

        uint256 sellerPrice = offerPrice - fee;       // underflow
