Audit Report

## Title
ERC-2612 Permit Front-Running in `pullToken` Causes Unconditional `safeTransferFrom` Revert When Victim Has No Pre-Existing Allowance - (File: src/periphery/MidnightBundles.sol)

## Summary
`pullToken` in `MidnightBundles` wraps the ERC-2612 `permit` call in a bare `try/catch` that silently discards any revert, then unconditionally calls `safeTransferFrom`. An unprivileged attacker can front-run the victim's pending transaction by extracting the permit signature from the mempool and calling `permit` directly, consuming the nonce. When the victim's transaction executes, the `safeTransferFrom` reverts because no allowance exists, permanently invalidating the victim's signed permit and blocking position opening.

## Finding Description
The vulnerable code is in `pullToken` at lines 378–384 of `src/periphery/MidnightBundles.sol`:

```solidity
if (permit.kind == PermitKind.ERC2612) {
    (uint256 deadline, uint8 v, bytes32 r, bytes32 s) =
        abi.decode(permit.data, (uint256, uint8, bytes32, bytes32));
    // Tolerate revert: a third party may have already consumed the permit.
    try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
    SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
}
``` [1](#0-0) 

The comment on line 382 reveals the design assumption: the `try/catch` is intentional, meant to tolerate a permit already consumed by a third party. However, the assumption that the user has a pre-existing allowance in that case is never enforced or verified. The `safeTransferFrom` on line 384 executes unconditionally regardless of whether the permit succeeded.

`pullToken` is called for each collateral supply entry in both `supplyCollateralAndSellWithUnitsTarget` (line 136) and `supplyCollateralAndSellWithAssetsTarget`: [2](#0-1) 

**Exploit flow:**
1. Victim signs an ERC-2612 permit granting `MidnightBundles` an allowance of `assets` for the collateral token. Victim has 0 pre-existing allowance to `MidnightBundles`.
2. Victim submits `supplyCollateralAndSellWithAssetsTarget` (or `supplyCollateralAndSellWithUnitsTarget`) with `CollateralSupply.permit.kind == PermitKind.ERC2612`.
3. Attacker observes the pending transaction in the mempool, extracts `(deadline, v, r, s)` from calldata, and calls `IERC20Permit(collateralToken).permit(victim, bundler, amount, deadline, v, r, s)` with higher gas, consuming the nonce.
4. Victim's transaction executes: `try permit(...) {} catch {}` silently swallows the "invalid nonce" revert.
5. `SafeTransferLib.safeTransferFrom(token, victim, bundler, amount)` reverts because `allowance(victim, bundler) == 0`.
6. The entire bundle reverts. The victim's permit nonce is permanently consumed.

**Why existing checks fail:** The `try/catch` is the only protection and it explicitly discards the failure. There is no post-permit allowance check. The Permit2 path (lines 385–394) is not vulnerable because `permitTransferFrom` atomically verifies the signature and executes the transfer in a single call. [3](#0-2) 

## Impact Explanation
The victim cannot open a collateral-backed position via the bundler using ERC-2612 permits. The signed permit nonce is permanently invalidated — it cannot be reused. The attacker can repeat this for every new permit the victim signs, creating an indefinite, cost-free DoS against any user relying solely on ERC-2612 permits for collateral tokens. This constitutes permanent freeze/unavailability of a core protocol function for targeted users, triggered by an unprivileged external actor with no capital requirement.

## Likelihood Explanation
The attack is feasible on any chain with a public mempool (Ethereum mainnet, most L2s). The permit parameters `(deadline, v, r, s)` are fully visible in pending calldata. The front-run call is a single cheap `permit()` call requiring no capital. The only preconditions are: (1) victim uses `PermitKind.ERC2612` for a collateral token, and (2) victim has no pre-existing allowance to `MidnightBundles` — the common case when relying on a permit. The attack is repeatable indefinitely at negligible cost.

## Recommendation
After the `try/catch`, verify that sufficient allowance exists before calling `safeTransferFrom`. If the allowance is insufficient (meaning the permit failed and no pre-existing allowance covers the transfer), revert with a descriptive error rather than proceeding to a guaranteed `safeTransferFrom` revert:

```solidity
try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
require(IERC20(token).allowance(from, address(this)) >= amount, InsufficientAllowance());
SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
```

This preserves the intended tolerance for already-consumed permits (when the user has a pre-existing allowance) while failing fast and with a clear error when neither the permit nor a pre-existing allowance covers the transfer. Alternatively, migrate collateral permit flows to `PermitKind.Permit2`, which is immune to this class of attack by design.

## Proof of Concept
**Minimal fork test steps:**
1. Deploy `MidnightBundles` against a forked environment with a standard ERC-2612 collateral token.
2. Fund `victim` with `amount` of the collateral token. Confirm `allowance(victim, bundler) == 0`.
3. Have `victim` sign an ERC-2612 permit for `(spender=bundler, amount, deadline)`.
4. As `attacker`, call `IERC20Permit(token).permit(victim, bundler, amount, deadline, v, r, s)` directly. Confirm nonce is consumed and `allowance(victim, bundler) == amount`.
5. As `attacker`, call `IERC20Permit(token).permit(victim, bundler, amount, deadline, v, r, s)` again to reset the allowance back to 0 (or simply have the victim approve 0 — the point is the allowance is 0 when the victim's tx lands).
6. Submit `victim`'s `supplyCollateralAndSellWithAssetsTarget` transaction. Observe it reverts at `safeTransferFrom` despite the `try/catch` on the permit call.

The test confirms: (a) the `try/catch` silently swallows the nonce-consumed revert, (b) `safeTransferFrom` reverts with zero allowance, and (c) the victim's nonce is permanently consumed.
