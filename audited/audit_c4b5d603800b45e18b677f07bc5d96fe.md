### Title
`seizedAssets` path in `liquidate` reverts via `mulDivUp` overflow for realistic high oracle prices while `repaidUnits` path succeeds - (`src/libraries/UtilsLib.sol` / `src/Midnight.sol`)

### Summary

`UtilsLib.mulDivUp` uses plain Solidity 0.8.x multiplication (`x * y`) which reverts on overflow. In `liquidate`, the `seizedAssets` input path computes `seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE)` before any bound check on `seizedAssets`, and the oracle price is unbounded on-chain. For realistic token pairs (e.g., WBTC/USDC), the oracle price can exceed `type(uint128).max`, causing the product to overflow `uint256` and revert, while the `repaidUnits` path is structurally immune to the same overflow.

### Finding Description

**`mulDivUp` implementation** (`src/libraries/UtilsLib.sol` line 34–36):
```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;  // plain multiplication, reverts on overflow in 0.8.x
}
```

**`seizedAssets` path** (`src/Midnight.sol` line 649–650):
```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```

**`repaidUnits` path** (`src/Midnight.sol` line 651–652):
```solidity
} else {
    seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice);
```

**Execution order matters:** The `mulDivUp` at line 650 executes before the `toUint128(seizedAssets)` cast at line 670. There is no on-chain bound on `liquidatedCollatPrice` before line 650.

**Overflow condition:**
- Collateral is stored as `uint128`, so `seizedAssets` is at most `type(uint128).max ≈ 3.4e38`.
- `ORACLE_PRICE_SCALE = 1e36`.
- For overflow: `seizedAssets * liquidatedCollatPrice > type(uint256).max ≈ 1.157e77`.
- With `seizedAssets = type(uint128).max`, overflow occurs when `liquidatedCollatPrice > type(uint128).max ≈ 3.4e38`.

**Realistic oracle price example (WBTC/USDC):**
- WBTC: 8 decimals, USDC: 6 decimals, BTC price: $100,000
- `price = 100,000 × (10^6 / 10^8) × 1e36 = 1e39`
- `1e39 > type(uint128).max ≈ 3.4e38` ✓

**Overflow arithmetic:**
- `seizedAssets × liquidatedCollatPrice = 3.4e38 × 1e39 = 3.4e77 > type(uint256).max ≈ 1.157e77` → **REVERTS**

**`repaidUnits` path is immune:**
- `repaidUnits ≤ type(uint128).max` (bounded by `_position.debt` which is `uint128`)
- `lif ≤ 2 × WAD = 2e18`
- Step 1: `repaidUnits.mulDivDown(lif, WAD)` → max `3.4e38 × 2e18 / 1e18 = 6.8e38` ✓
- Step 2: `.mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice)` → `6.8e38 × 1e36 / 1e39 = 6.8e35` ✓ — no overflow

**Existing protections are insufficient:** `certora/specs/NoMultiplicationOverflow.spec` explicitly documents the oracle price bound as an **"Oracle integration assumption"** (`price * max_uint128 + ORACLE_PRICE_SCALE - 1 <= max_uint256`), but this assumption is **not enforced on-chain** anywhere — not in `touchMarket`, not in `liquidate`, not in `supplyCollateral`. Any market using a WBTC/USDC-style oracle violates this assumption.

### Impact Explanation

A liquidator calling `liquidate` with `seizedAssets > 0` on a market where `liquidatedCollatPrice > type(uint256).max / seizedAssets` (achievable with realistic WBTC/USDC oracle prices and large collateral positions) receives an arithmetic overflow revert. The same unhealthy position remains liquidatable via the `repaidUnits` path. The `seizedAssets` input path — a first-class API — is silently broken for an entire class of realistic markets, forcing liquidators to use only the `repaidUnits` path and preventing exact collateral-amount-based liquidations.

### Likelihood Explanation

- **Preconditions:** Market uses a collateral token with significantly more decimal precision than the loan token (e.g., WBTC 8 dec / USDC 6 dec), or any pair where the oracle price exceeds `~3.4e38`. This is a common real-world configuration.
- **Feasibility:** Any unprivileged liquidator triggers this by passing `seizedAssets` near the borrower's collateral balance (which is `uint128`-bounded) and `repaidUnits = 0`. No special setup required beyond the market existing with such an oracle.
- **Repeatability:** Deterministic — every call to the `seizedAssets` path on such a market reverts. Not dependent on timing, randomness, or race conditions.

### Recommendation

Replace the bare multiplication in `mulDivUp` (and `mulDivDown`) with a 512-bit overflow-safe implementation (e.g., Uniswap v3's `FullMath.mulDiv`), or add an explicit on-chain oracle price bound check in `touchMarket` enforcing `price * type(uint128).max + ORACLE_PRICE_SCALE - 1 <= type(uint256).max` at market creation time. The Certora assumption in `NoMultiplicationOverflow.spec` should be promoted to an enforced invariant.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {UtilsLib} from "src/libraries/UtilsLib.sol";
import {ORACLE_PRICE_SCALE, WAD} from "src/libraries/ConstantsLib.sol";

contract OverflowLiquidateTest is Test {
    using UtilsLib for uint256;

    // Fuzz: seizedAssets near type(uint128).max, price near type(uint256).max / type(uint128).max
    function testFuzz_seizedAssetsPathOverflows_repaidUnitsPathSucceeds(
        uint256 seizedAssets,
        uint256 price
    ) public {
        // Bound seizedAssets to realistic collateral range (uint128)
        seizedAssets = bound(seizedAssets, type(uint128).max / 2, type(uint128).max);
        // Bound price to the overflow-triggering range: price > type(uint256).max / seizedAssets
        uint256 overflowThreshold = type(uint256).max / seizedAssets;
        price = bound(price, overflowThreshold + 1, type(uint256).max);

        // seizedAssets path: expect revert due to mulDivUp overflow
        vm.expectRevert(); // arithmetic overflow
        uint256 result = seizedAssets.mulDivUp(price, ORACLE_PRICE_SCALE);

        // repaidUnits path: same price, repaidUnits bounded by uint128 debt
        uint256 repaidUnits = type(uint128).max;
        uint256 lif = 2 * WAD; // maxLif upper bound
        // Step 1: repaidUnits * lif / WAD — no overflow (3.4e38 * 2e18 / 1e18 = 6.8e38 < uint256.max)
        uint256 step1 = repaidUnits.mulDivDown(lif, WAD);
        // Step 2: step1 * ORACLE_PRICE_SCALE / price — no overflow (numerator = 6.8e38 * 1e36 = 6.8e74 < uint256.max)
        uint256 step2 = step1.mulDivDown(ORACLE_PRICE_SCALE, price);
        // Assert repaidUnits path succeeds and produces a valid (possibly zero) result
        assertLe(step2, type(uint256).max);
    }
}
```

**Expected assertions:**
- `seizedAssets.mulDivUp(price, ORACLE_PRICE_SCALE)` reverts with arithmetic overflow for `price > type(uint256).max / seizedAssets`.
- `repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, price)` completes without revert for the same `price`.
- A full integration test deploying Midnight with a WBTC/USDC oracle returning `1e39` and a borrower with `type(uint128).max / 2` collateral would confirm `liquidate(..., seizedAssets, 0, ...)` reverts while `liquidate(..., 0, repaidUnits, ...)` succeeds.
