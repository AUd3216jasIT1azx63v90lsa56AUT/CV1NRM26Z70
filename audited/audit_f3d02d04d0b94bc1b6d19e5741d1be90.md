Audit Report

## Title
Off-by-one in `CannotIncreaseDebtPostMaturity` guard allows debt creation at exactly `block.timestamp == market.maturity` - (File: `src/Midnight.sol`)

## Summary
The guard at `src/Midnight.sol:391` uses `<=` instead of `<`, permitting `sellerDebtIncrease > 0` when `block.timestamp` equals `market.maturity` exactly. This directly contradicts the protocol's own stated invariant in `live_context.json:221`: `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"`. The resulting seller debt is immediately liquidatable, and the corresponding buyer credit can drain pre-existing `withdrawable` balances that other lenders depend on.

## Finding Description
**Root cause — `src/Midnight.sol:391`:**
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
When `block.timestamp == market.maturity`, the first disjunct is `true` and the require passes regardless of `sellerDebtIncrease`. Lines 413–414 then execute unconditionally:
```solidity
sellerPos.credit -= UtilsLib.toUint128(sellerCreditDecrease);
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);   // executes at maturity
```
And line 410 grants the buyer credit:
```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);
```

**`take()` does not touch `withdrawable`.** Confirmed at lines 375–430: there is no `_marketState.withdrawable` increment anywhere in `take()`. `withdrawable` is only incremented by `repay()` (line 509) and by `liquidate()`. So the buyer's newly minted credit is backed by pre-existing `withdrawable` accumulated from other borrowers' repayments.

**`withdraw()` at line 494** decrements `_marketState.withdrawable` directly:
```solidity
_marketState.withdrawable -= UtilsLib.toUint128(units);
```
If `withdrawable >= units` at the time of the attack, the buyer can immediately drain those tokens, which belong to other lenders awaiting their own `withdraw()` calls.

**Exploit flow:**
1. A sell offer exists (seller = `offer.maker`, signed by the seller; taker = attacker as `msg.sender`).
2. `vm.warp(market.maturity)` — or attacker lands tx in the block where `block.timestamp == market.maturity`.
3. Attacker calls `take(sellOffer, ..., units, ...)` — line 391 passes because `block.timestamp <= maturity` is `true`.
4. `sellerPos.debt += units`; `buyerPos.credit += units`; `_marketState.withdrawable` unchanged.
5. If `_marketState.withdrawable >= units`, attacker calls `withdraw(market, units, buyer, receiver)` — line 494 decrements `withdrawable` and transfers tokens out, draining the pool.

**Why existing checks fail:** The `CannotIncreaseDebtPostMaturity` check is the sole guard. It fails at the exact boundary due to `<=`. No secondary check prevents debt creation at maturity. All existing tests for this error use `market.maturity + 1` (24 matches in `test/TakeTest.sol`); the `block.timestamp == market.maturity` boundary is untested.

## Impact Explanation
- **Seller:** Placed into an immediately-liquidatable debt position at maturity (post-maturity debt is liquidatable regardless of health).
- **Buyer/attacker:** Receives credit redeemable against any pre-existing `withdrawable` in the market, directly stealing funds that other lenders are owed.
- **Other lenders:** Their `withdraw()` calls revert or are underfunded because `withdrawable` was drained.
- Impact class: direct theft of lender funds / permanent accounting desync. Concrete and in-scope.

## Likelihood Explanation
- `block.timestamp == market.maturity` is achievable by any block proposer, or by any attacker who monitors the mempool and submits a transaction timed to land in the maturity block. On chains with predictable block times (e.g., Ethereum post-merge, 12s slots), this is straightforward to target.
- A sell offer is a normal user action; no privileged access is required. The attacker is the taker (`msg.sender`), not the offer maker.
- `withdrawable > 0` is a realistic condition in any active market that has seen prior repayments.
- The invariant violation (debt at maturity) occurs even without `withdrawable > 0`; the fund-loss step additionally requires `withdrawable > 0`.

## Recommendation
Change `<=` to `<` at line 391:
```solidity
// Before (buggy):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (correct):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```
This aligns the guard with the stated invariant: debt increase is only permitted strictly before maturity. Add a test case with `vm.warp(market.maturity)` (exact equality) asserting the revert.

## Proof of Concept
```solidity
function test_takeAtExactMaturity_debtCreatedAndWithdrawableDrained() public {
    // Setup: active market with prior repayments so withdrawable > 0
    // ... (standard market/offer setup) ...

    // Warp to exact maturity
    vm.warp(market.maturity);

    uint256 withdrawableBefore = midnight.marketState(id).withdrawable;
    assertGt(withdrawableBefore, 0);

    // Attacker takes sell offer at exact maturity — should revert but doesn't
    vm.prank(attacker);
    midnight.take(sellOffer, ratifierData, units, attacker, attacker, address(0), "");

    // Seller has new debt at maturity (immediately liquidatable)
    assertGt(midnight.position(id, seller).debt, 0);

    // Attacker withdraws against pre-existing withdrawable
    vm.prank(attacker);
    midnight.withdraw(market, units, attacker, attacker);

    // withdrawable drained from other lenders
    assertEq(midnight.marketState(id).withdrawable, withdrawableBefore - units);
}
