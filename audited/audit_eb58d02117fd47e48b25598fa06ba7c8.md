Audit Report

## Title
Buy offer `maxAssets` cap bypassed via `mulDivDown` rounding to zero when `buyerPrice < WAD` — (`src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and its tick is set such that `tickToPrice(tick) < WAD`, the expression `units.mulDivDown(buyerPrice, WAD)` evaluates to zero for any `units` where `units * buyerPrice < WAD`. Because `consumed` is incremented by this zero value, the `ConsumedAssets` guard never triggers, allowing a taker to push unbounded units through a fully-consumed offer at zero token cost. The protocol acknowledges this behavior in a code comment and has a dedicated regression test confirming it.

## Finding Description
**Code path:** `src/Midnight.sol` lines 363–369.

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause:** For a buy offer, `consumed` is incremented by `buyerAssets`, not by `units`. When `buyerPrice < WAD` and `units` is small enough that `units * buyerPrice < WAD`, `mulDivDown` returns 0. `consumed` is unchanged, the cap check passes, and the take succeeds — but `units` of credit and debt are still allocated to the buyer (maker) and seller (taker) respectively at lines 408–414.

**Why existing checks fail:** The `ConsumedAssets` guard at line 369 only checks `newConsumed <= offer.maxAssets`. Since `consumed += 0`, `newConsumed` never exceeds `maxAssets` regardless of how many additional units are pushed through. The `maxUnits` path (line 371–372) is not reached because `maxAssets > 0` takes the first branch.

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets > 0` and `tick` such that `tickToPrice(tick) < WAD` (standard for discount lending markets).
2. Offer is at or near its `maxAssets` limit (via prior fills or `setConsumed`).
3. Taker calls `take(offer, ..., units=1, ...)`.
4. `buyerAssets = 1.mulDivDown(buyerPrice, WAD) = 0`.
5. `consumed += 0` → `newConsumed == maxAssets` → `require(newConsumed <= maxAssets)` passes.
6. Position state is updated: maker's `credit` increases by 1, taker's `debt` increases by 1, `totalUnits` increases, maker's `pendingFee` increases.
7. No loan tokens are transferred (`buyerAssets = 0`).
8. Steps 3–7 repeat indefinitely.

**Protocol acknowledgment:** The behavior is documented at `src/Midnight.sol` line 94: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` lines 857–889 explicitly asserts `buyerAssets = 0`, `sellerAssets = 0`, yet `creditOf` and `debtOf` both increase, confirming the bypass is reproducible.

## Impact Explanation
A malicious taker can grief any buy-offer maker with a sub-WAD tick by forcing unbounded unit accumulation beyond the maker's intended `maxAssets` cap. The maker's `credit`, `pendingFee`, and the market's `totalUnits` grow without bound. The taker pays zero loan tokens and only accepts self-inflicted debt, making this a low-cost, repeatable grief vector. This constitutes an unauthorized state change and integrity failure in the offer cap accounting invariant.

## Likelihood Explanation
**Preconditions:** (1) A buy offer exists with `maxAssets > 0` and a tick where `tickToPrice < WAD`. (2) The offer is at or near its consumed limit. Both are normal operating conditions — sub-WAD ticks are standard for discount lending markets. **Feasibility:** Any unprivileged taker can execute this with `units = 1` in a single transaction. **Repeatability:** Unlimited; each call adds 0 to `consumed` and 1 to the maker's credit/pendingFee.

## Recommendation
Track cap consumption in `units` rather than in `buyerAssets` for buy offers, or add a minimum-units guard. Concretely, when `offer.maxAssets > 0` and `offer.buy == true`, require `buyerAssets > 0` before proceeding (i.e., `require(buyerAssets > 0 || units == 0)`), or alternatively increment `consumed` by `units` and compare against a `maxUnits`-equivalent derived from `maxAssets`. A simpler fix is to add `require(units == 0 || buyerAssets > 0)` immediately after computing `buyerAssets` to reject zero-asset fills when units are nonzero.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` lines 857–889 is a complete, passing PoC:
- Sets `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (price < WAD).
- Pre-consumes the offer to its limit via `setConsumed`.
- Calls `take(1, borrower, lenderOffer)` and asserts `buyerAssets == 0`, `sellerAssets == 0`, yet `creditOf(lender)` and `debtOf(borrower)` both increase and `consumed` remains at `maxAssets`.
- Run with: `forge test --match-test testBugBuyMaxAssetsBypass`.
