### Title
Fee-on-Transfer Loan Token Causes `repayAndWithdrawCollateral` to DoS Repay Flow - (`File: src/periphery/MidnightBundles.sol`)

### Summary

`repayAndWithdrawCollateral` computes `units = assets - referralFeeAssets` from the caller-supplied nominal `assets` value, then calls `pullToken(loanToken, msg.sender, assets, ...)` which performs a raw `safeTransferFrom` without checking the amount actually received. When the loan token charges a transfer fee `f`, the bundler receives only `assets*(1-f)` tokens. Midnight's `repay` subsequently calls `safeTransferFrom(loanToken, bundler, address(this), units)` where `units = assets - referralFeeAssets`, and if `f*assets > referralFeeAssets` the bundler's balance is insufficient, causing the entire transaction to revert.

### Finding Description

**Exact code path:**

In `src/periphery/MidnightBundles.sol`, `repayAndWithdrawCollateral` (lines 315–348):

```
referralFeeAssets = assets.mulDivDown(referralFeePct, WAD)   // line 329
units             = assets - referralFeeAssets                // line 330
pullToken(loanToken, msg.sender, assets, loanTokenPermit)     // line 331
forceApproveMax(loanToken, MIDNIGHT)                          // line 332
IMidnight(MIDNIGHT).repay(market, units, ...)                 // line 334
```

`pullToken` resolves to `SafeTransferLib.safeTransferFrom(token, from, address(this), amount)` (line 384 of `MidnightBundles.sol`, line 27 of `SafeTransferLib.sol`). This call requests `assets` tokens but with a fee-on-transfer token the bundler actually receives `assets - f*assets`.

Inside `Midnight.repay` (line 520 of `src/Midnight.sol`):
```
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
```
where `payer = msg.sender = bundler` (callback is `address(0)`, so line 511 sets `payer = msg.sender`).

**Root cause:** No received-amount check after `pullToken`. The bundler's actual balance is `assets*(1-f)`, but `units` is computed from the nominal `assets`. When `f*assets > referralFeeAssets` (equivalently, `f > referralFeePct/WAD`), the bundler holds fewer tokens than `units`, and Midnight's `safeTransferFrom` reverts.

**Attacker-controlled inputs:**
- Any borrower can call `repayAndWithdrawCollateral` with a market whose `loanToken` is a fee-on-transfer token.
- `referralFeePct` is caller-supplied; setting it to `0` makes `referralFeeAssets = 0` and `units = assets`, so any nonzero fee rate `f > 0` triggers the revert.
- No privileged access required.

**Exploit flow:**
1. Market exists with a fee-on-transfer loan token (fee rate `f > 0`).
2. Borrower calls `repayAndWithdrawCollateral(market, assets, onBehalf, ..., referralFeePct=0, ...)`.
3. `units = assets`, `pullToken` pulls `assets` nominal but bundler receives `assets*(1-f)`.
4. `repay(market, assets, ...)` → Midnight calls `safeTransferFrom(loanToken, bundler, midnight, assets)`.
5. Bundler balance = `assets*(1-f) < assets` → ERC20 transfer reverts → entire tx reverts.

**Existing checks are insufficient:**
- `require(referralFeePct < WAD)` (line 326) only bounds the referral percentage; it does not guard against fee-on-transfer discrepancy.
- No balance check or received-amount accounting exists anywhere in the path.

### Impact Explanation

Any call to `repayAndWithdrawCollateral` with a fee-on-transfer loan token where `f*assets > referralFeeAssets` is permanently DoS'd. The borrower cannot repay debt or withdraw collateral through the bundler. With `referralFeePct = 0` (the common case), the condition reduces to `f > 0`, meaning every call fails for any fee-on-transfer token. The borrower must interact with `Midnight.repay` directly, bypassing the bundler entirely, losing the atomic repay-and-withdraw-collateral functionality.

### Likelihood Explanation

**Preconditions:**
1. A market's `loanToken` implements a transfer fee (e.g., tokens with configurable fees, deflationary tokens).
2. The fee rate `f` exceeds `referralFeePct/WAD` — trivially satisfied when `referralFeePct = 0`.

**Feasibility:** Any borrower in such a market triggers this by calling the function as documented. No special setup, no admin action, no oracle manipulation required. The condition is deterministic and repeatable on every call.

### Recommendation

After `pullToken`, measure the actual received amount and use it to recompute `units`:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
pullToken(loanToken, msg.sender, assets, loanTokenPermit);
uint256 received = IERC20(loanToken).balanceOf(address(this)) - balanceBefore;
uint256 referralFeeAssets = received.mulDivDown(referralFeePct, WAD);
uint256 units = received - referralFeeAssets;
```

This ensures `units` is always ≤ the bundler's actual balance regardless of token fee behavior.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";

// FeeToken: charges feeBps on every transferFrom
contract FeeToken is ERC20 {
    uint256 public feeBps; // e.g. 100 = 1%
    constructor(uint256 _feeBps) ERC20("FEE","FEE") { feeBps = _feeBps; }
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount * feeBps / 10_000;
        super.transferFrom(from, to, amount - fee); // recipient gets amount-fee
        _burn(from, fee);                           // fee is burned
        return true;
    }
}

contract FeeOnTransferRepayTest is Test {
    // Setup: deploy Midnight + MidnightBundles with FeeToken as loanToken
    // Create market, collateralize borrower, establish debt via sell

    function testFuzz_repayDoS(uint256 feeBps, uint256 assets, uint256 referralFeePct) public {
        feeBps = bound(feeBps, 1, 500);           // 0.01% – 5% fee
        assets = bound(assets, 1e18, 1e24);
        referralFeePct = bound(referralFeePct, 0, feeBps * 1e14); // referralFeePct < f*WAD

        // precondition: f*assets > referralFeeAssets
        // i.e. feeBps/10000 * assets > assets * referralFeePct / WAD
        // i.e. feeBps * WAD > referralFeePct * 10000  (always true when referralFeePct < feeBps*1e14)

        // deal borrower `assets` fee tokens, approve bundler
        deal(address(feeToken), borrower, assets);
        vm.prank(borrower); feeToken.approve(address(bundles), assets);

        // expect revert on repay because bundler holds assets*(1-f) < units=assets-referralFeeAssets
        vm.prank(borrower);
        vm.expectRevert(); // ERC20 insufficient balance
        bundles.repayAndWithdrawCollateral(
            market, assets, borrower, _noPermit(),
            new CollateralWithdrawal[](0), address(0),
            referralFeePct, address(0xFEE)
        );
    }
}
```

**Expected assertion:** `vm.expectRevert()` passes — the call always reverts when `f*assets > referralFeeAssets`, confirming the DoS. A complementary invariant test can assert `bundler.balanceOf(loanToken) == 0` after any successful call, which would fail mid-execution when the fee causes an underflow in the transfer.     function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
