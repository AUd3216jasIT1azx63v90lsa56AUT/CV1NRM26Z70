### Title
Out-of-bounds `collateralIndex` in `CollateralSupply` causes Solidity panic instead of meaningful bundler error - (File: src/periphery/MidnightBundles.sol)

### Summary
Both `supplyCollateralAndSellWithUnitsTarget` and `supplyCollateralAndSellWithAssetsTarget` access `market.collateralParams[collateralSupplies[i].collateralIndex].token` without first checking that `collateralSupplies[i].collateralIndex < market.collateralParams.length`. An unprivileged caller who is the `taker` (or is authorized by the taker) can pass a `CollateralSupply` with an out-of-bounds `collateralIndex`, causing the bundler to revert with a Solidity panic (error code `0x32`) rather than a defined bundler error.

### Finding Description
**Code path:**

`supplyCollateralAndSellWithUnitsTarget` — `src/periphery/MidnightBundles.sol` lines 133–135:
```solidity
Market memory market = takes[0].offer.market;
for (uint256 i; i < collateralSupplies.length; i++) {
    address token = market.collateralParams[collateralSupplies[i].collateralIndex].token;
``` [1](#0-0) 

`supplyCollateralAndSellWithAssetsTarget` — lines 268–270:
```solidity
Market memory market = takes[0].offer.market;
for (uint256 i; i < collateralSupplies.length; i++) {
    address token = market.collateralParams[collateralSupplies[i].collateralIndex].token;
``` [2](#0-1) 

**Root cause:** There is no `require(collateralSupplies[i].collateralIndex < market.collateralParams.length, ...)` guard before the array access. Both `market` (taken from `takes[0].offer.market`) and `collateralSupplies[i].collateralIndex` are fully caller-controlled inputs.

**Exploit flow:**
1. Attacker is the `taker` (or is authorized by the taker — the only access check present). [3](#0-2) 
2. Attacker constructs a `CollateralSupply` with `collateralIndex = market.collateralParams.length` (e.g., `1` for a single-collateral market).
3. Calls `supplyCollateralAndSellWithUnitsTarget` or `supplyCollateralAndSellWithAssetsTarget`.
4. The bundler executes `market.collateralParams[1]` on a length-1 array — Solidity emits a panic `0x4e487b71` with code `0x32` (array out-of-bounds) rather than any named bundler error.

**Existing protections:** None. The only pre-loop checks are the `Unauthorized` and `PctExceeded` guards, neither of which validates `collateralIndex`. [4](#0-3) 

Note: `Midnight.supplyCollateral` itself also panics on an out-of-bounds index (line 529), so the panic would occur in the bundler before any external call is made. [5](#0-4) 

### Impact Explanation
The transaction reverts with a raw Solidity panic (`0x4e487b71` / `0x32`) instead of a named bundler error. Off-chain tooling, integrators, and wallets that decode revert data against the `IMidnightBundles` error ABI will receive an unrecognized revert reason. This is a non-critical behavior divergence: no funds are lost or frozen, but the error surface is inconsistent with the rest of the bundler's error handling.

### Likelihood Explanation
Trivially reachable by any caller who is the `taker` or is authorized by the taker — no special state is required. The attacker only needs to supply a `CollateralSupply` struct with `collateralIndex >= market.collateralParams.length`. Repeatable at will.

### Recommendation
Add an explicit bounds check before the array access in both functions:

```solidity
require(
    collateralSupplies[i].collateralIndex < market.collateralParams.length,
    InvalidCollateralIndex()
);
```

Add `error InvalidCollateralIndex();` to `IMidnightBundles`. [6](#0-5) 

### Proof of Concept
```solidity
function testSupplyCollateralOOBIndexPanics() public {
    // market has 1 collateral (index 0 only)
    // collateralIndex = 1 is out of bounds
    CollateralSupply[] memory supplies = new CollateralSupply[](1);
    supplies[0] = CollateralSupply({
        collateralIndex: market.collateralParams.length, // OOB
        assets: 1e18,
        permit: _noPermit()
    });

    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offers[0], units: 1e18, ratifierData: hex""});

    vm.prank(borrower);
    // Assert it does NOT revert with a panic (0x4e487b71)
    // but with a named bundler error instead.
    // Currently this assertion FAILS — it reverts with stdError.indexOOBError.
    vm.expectRevert(IMidnightBundles.InvalidCollateralIndex.selector);
    midnightBundles.supplyCollateralAndSellWithUnitsTarget(
        1e18, 0, borrower, borrower, supplies, takes, 0, address(0)
    );
}
```

Expected (after fix): revert with `InvalidCollateralIndex`.
Actual (before fix): revert with `stdError.indexOOBError` (Solidity panic `0x32`), as demonstrated by the analogous `testLiquidateInvalidCollateralIndex` pattern already present in the test suite. [7](#0-6)
