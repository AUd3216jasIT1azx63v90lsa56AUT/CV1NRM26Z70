### Title
`sellerAssetsToUnits` division-by-zero at `offerPrice == settlementFee` blocks `supplyCollateralAndSellWithAssetsTarget` while `Midnight.take()` succeeds - (`File: src/periphery/TakeAmountsLib.sol`)

### Summary

When a buy offer's tick price exactly equals the post-maturity settlement fee (`tickToPrice(offer.tick) == settlementFeeCbp0 * CBP`), `sellerPrice` computes to zero in `sellerAssetsToUnits`. The subsequent `mulDivUp(WAD, 0)` call triggers an arithmetic underflow/division-by-zero revert in Solidity 0.8+, while `Midnight.take()` handles the same `sellerPrice = 0` case without reverting (producing `sellerAssets = 0`). Because `sellerAssetsToUnits` is called outside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, the entire bundler call reverts, blocking the assets-target sell path even when the underlying take would succeed.

### Finding Description

**Root cause — `TakeAmountsLib.sellerAssetsToUnits` (lines 44–46):**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

When `offerPrice == settlementFee`, `sellerPrice = 0`. `mulDivUp` is implemented as `(x * y + (d - 1)) / d`; with `d = 0`, the `d - 1` term underflows (Solidity 0.8 arithmetic revert), confirmed by `testMulDivUpDivisionByZero`.

**`Midnight.take()` does not revert in the same case (lines 361–364):**

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
...
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...;
```

`mulDivDown(units, 0, WAD) = (units * 0) / WAD = 0` — no division by zero.

**The comment in `sellerAssetsToUnits` (line 34) is inaccurate:**
> "Reverts if offerPrice < settlementFee in case of a buy offer (midnight reverts too)."

This is only true for `offerPrice < settlementFee` (underflow). When `offerPrice == settlementFee`, Midnight does **not** revert, but `sellerAssetsToUnits` does.

**Call site in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` (lines 285–300):**

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(   // ← reverts here, OUTSIDE try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) returns (...) {   // ← try/catch only here
    ...
} catch {}
```

The `try/catch` wraps only `Midnight.take()`. A revert in `sellerAssetsToUnits` propagates uncaught, aborting the entire bundler call.

**Preconditions for the boundary condition:**

- `MAX_SETTLEMENT_FEE_0_DAYS = 0.000014e18` (14 CBP units). Settlement fees are multiples of `CBP = 1e12`. Tick prices span the full range including values in `[1e12, 14e12]`, so there exist valid ticks where `tickToPrice(tick) == settlementFeeCbp0 * CBP`.
- The settlement fee can also be raised post-offer-creation (the protocol explicitly documents this: "if the settlement fee gets increased, it might implicitly cancel offers with very low price"), creating the boundary condition for existing offers.

**Exploit flow:**

1. A buy offer exists with `tickToPrice(offer.tick) == F` for some valid fee value `F`.
2. `settlementFeeCbp0` is set (or raised) so that `settlementFeeCbp0 * CBP == F`.
3. Market reaches maturity (`block.timestamp >= offer.market.maturity`), so `timeToMaturity = 0` and `settlementFee(id, 0) = F`.
4. Taker calls `supplyCollateralAndSellWithAssetsTarget` with `targetSellerAssets > 0` and this offer in the `takes` array.
5. `sellerAssetsToUnits` is called with `sellerPrice = 0` and `targetSellerAssets > 0`, triggering arithmetic revert.
6. The entire bundler call reverts. Direct `Midnight.take()` with the same offer succeeds (returning `sellerAssets = 0`).

**Existing protections are insufficient:**

- The `NoDivisionByZero.spec` Certora spec covers only `Midnight.sol`, not `TakeAmountsLib.sol`.
- The bundler documents "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts" (line 246), but this is a documentation of the symptom, not a protection against the root cause.
- No guard for `sellerPrice == 0` exists in `sellerAssetsToUnits`.

### Impact Explanation

Any taker calling `supplyCollateralAndSellWithAssetsTarget` whose `takes` array contains a buy offer at the settlement-fee price boundary will have the entire call revert with an arithmetic error. If the settlement fee was raised to match an existing offer's tick price, the taker cannot use the assets-target bundler path to unwind at maturity, even though `Midnight.take()` would succeed. The taker must fall back to calling `Midnight.take()` directly, losing the atomic collateral-supply-and-sell convenience and any multi-offer aggregation the bundler provides.

### Likelihood Explanation

**Preconditions:**
- `settlementFeeCbp0 * CBP` must equal `tickToPrice(some_tick)`. Given `MAX_SETTLEMENT_FEE_0_DAYS = 14e12` and tick prices are dense in the low-price range, this is achievable. The feeSetter can set the fee to match a tick price, or an offer maker can choose a tick matching the current fee.
- The settlement fee can be raised post-offer-creation, converting a previously valid offer into a boundary-condition offer without any action by the taker.
- The condition is repeatable: any market where `settlementFeeCbp0 * CBP` aligns with a tick price will exhibit this for all offers at that tick.

### Recommendation

Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case. When `sellerPrice == 0`, no finite number of units can yield `targetSellerAssets > 0`, so the function should return `type(uint256).max` (signaling "infinite units needed") or revert with a descriptive error rather than a raw arithmetic panic. The `min(...)` call in the bundler would then clamp to `takes[i].units`, and the subsequent take would return `sellerAssets = 0`, allowing the loop to continue to the next offer:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
if (offer.buy && sellerPrice == 0) return type(uint256).max;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

The corresponding NatSpec comment should be corrected: the function currently says "midnight reverts too" for `offerPrice == settlementFee`, but Midnight does not revert in that case.
