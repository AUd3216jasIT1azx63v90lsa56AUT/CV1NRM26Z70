### Title
Fee-on-Transfer Loan Token Allows `repay` to Inflate `withdrawable` Beyond Actual Token Balance, Socializing Bad Debt Among Lenders - (File: src/libraries/SafeTransferLib.sol / src/Midnight.sol)

### Summary

The `repay` function in `src/Midnight.sol` decrements `position[id][onBehalf].debt` and increments `marketState[id].withdrawable` by `units` before calling `SafeTransferLib.safeTransferFrom`, which does not verify the actual amount of tokens received. When the market's `loanToken` is a fee-on-transfer token, only `units - fee` tokens arrive at the contract while `withdrawable` is credited with the full `units`, permanently breaking the invariant that the contract's token balance covers all withdrawable amounts and socializing the shortfall as bad debt among lenders.

### Finding Description

**Exact code path:**

In `src/Midnight.sol` lines 508–520:

```solidity
position[id][onBehalf].debt -= UtilsLib.toUint128(units);      // L508
marketState[id].withdrawable += UtilsLib.toUint128(units);      // L509
// ...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // L520
```

`SafeTransferLib.safeTransferFrom` in `src/libraries/SafeTransferLib.sol` lines 24–34 only checks that `transferFrom` does not revert and returns `true` (or no return value). It performs no balance-before/balance-after check and cannot detect that a fee-on-transfer token delivered fewer tokens than `units`.

**Root cause:** Missing received-amount verification after `safeTransferFrom`. The accounting update (lines 508–509) is unconditionally based on the caller-supplied `units` parameter, not on the tokens actually received.

**Attacker-controlled inputs:**
- Market creator (unprivileged): creates a market with a fee-on-transfer ERC-20 as `market.loanToken`. No validation of `loanToken` exists anywhere in the codebase; `Market` is a plain struct with an `address loanToken` field.
- Borrower (unprivileged, may be the same actor): calls `repay(market, units, onBehalf, address(0), "")` with any `units ≤ position.debt`.

**Exploit flow:**
1. Market creator deploys a market whose `loanToken` charges a percentage fee `f` on every `transferFrom`.
2. Lender supplies liquidity; borrower borrows `D` units via `take`.
3. Borrower calls `repay(market, D, borrower, address(0), "")`.
4. Protocol executes: `debt -= D`, `withdrawable += D`, then `safeTransferFrom(..., D)`.
5. Token contract transfers `D` from borrower but delivers only `D - fee` to Midnight.
6. `withdrawable` is now `D` but `loanToken.balanceOf(midnight)` is only `D - fee`.
7. Lenders collectively hold credit redeemable for `D` units but the contract holds `D - fee` tokens.
8. The first `N-1` lenders who call `withdraw` succeed; the last lender's `withdraw` call triggers `safeTransfer(loanToken, receiver, units)` which reverts because the balance is exhausted.

**Why existing checks fail:**
- `SafeTransferLib.safeTransferFrom` checks only revert/return-value, not received quantity.
- No balance snapshot is taken before or after the transfer.
- No documentation or code restricts `loanToken` to non-fee-on-transfer tokens.
- The Certora spec `repayIncreasesWithdrawable` (line 16 of `certora/specs/WithdrawableMonotonicity.spec`) asserts `withdrawableAfter == withdrawableBefore + units` — this is proven only under the assumption that the token model is standard; it does not model fee-on-transfer behavior.

### Impact Explanation

`marketState[id].withdrawable` permanently exceeds `loanToken.balanceOf(address(midnight))` by exactly the cumulative fee deducted across all `repay` calls. Every subsequent `withdraw` drains the real balance until it is exhausted. The last lender(s) holding valid credit positions cannot redeem them; their loss is proportional to the fee rate and total repaid volume. This is concrete, irreversible bad debt socialized across all lenders in the affected market.

### Likelihood Explanation

**Preconditions:**
- A fee-on-transfer token must be used as `loanToken`. This requires either (a) an existing fee-on-transfer token (e.g., tokens with configurable transfer fees) or (b) a market creator deploying a custom token — both are permissionless actions.
- The borrower need only call `repay` normally; no special privileges or timing are required.

**Feasibility:** High — market creation is permissionless, the exploit requires no flash loans, no oracle manipulation, and no privileged access. It is repeatable on every `repay` call, compounding the deficit.

### Recommendation

Record the contract's token balance before and after `safeTransferFrom` in `repay` and use the actual received amount for accounting:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
require(received == units, InsufficientTokensReceived());
```

Alternatively, explicitly document and enforce (via a registry or deployment check) that only standard ERC-20 tokens with no transfer fees are permitted as `loanToken`. Apply the same fix to `liquidate` (line 717) and `take` (lines 455–456) for consistency.

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";

// Fee-on-transfer token: deducts feeRate (in bps) on every transferFrom
contract FeeToken is ERC20 {
    uint256 public feeRate; // basis points, e.g. 100 = 1%
    address public feeRecipient;
    constructor(uint256 _feeRate, address _feeRecipient) ERC20("FEE","FEE") {
        feeRate = _feeRate; feeRecipient = _feeRecipient;
    }
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount * feeRate / 10000;
        super.transferFrom(from, to, amount - fee);
        if (fee > 0) super.transferFrom(from, feeRecipient, fee);
        return true;
    }
}

contract FeeOnTransferRepayTest is Test {
    Midnight midnight;
    FeeToken loanToken;
    address lender = makeAddr("lender");
    address borrower = makeAddr("borrower");

    function setUp() public { /* deploy midnight, create market with loanToken = FeeToken */ }

    function testFeeOnTransferRepayDrainLenders(uint256 feeRate, uint256 repayAmount) public {
        feeRate = bound(feeRate, 1, 1000); // 0.01% to 10%
        // Setup: lender supplies liquidity, borrower borrows repayAmount units
        // ...

        uint256 balanceBefore = loanToken.balanceOf(address(midnight));
        uint256 withdrawableBefore = midnight.withdrawable(id);

        vm.prank(borrower);
        midnight.repay(market, repayAmount, borrower, address(0), "");

        uint256 balanceAfter = loanToken.balanceOf(address(midnight));
        uint256 withdrawableAfter = midnight.withdrawable(id);
        uint256 fee = repayAmount * feeRate / 10000;

        // Core invariant violation: withdrawable grew by repayAmount but balance grew by repayAmount - fee
        assertEq(withdrawableAfter, withdrawableBefore + repayAmount, "withdrawable inflated");
        assertEq(balanceAfter, balanceBefore + repayAmount - fee, "actual balance");
        assertGt(withdrawableAfter - withdrawableBefore, balanceAfter - balanceBefore, "INVARIANT BROKEN");

        // Last lender cannot withdraw
        vm.prank(lender);
        vm.expectRevert(); // safeTransfer reverts: insufficient balance
        midnight.withdraw(market, repayAmount, lender, lender);
    }
}

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
