### Title
Sponsor can redirect the entire pool to themselves by changing `recoveryAddress` after bad-faith CORRUPTED is flagged - (File: src/ConfidencePool.sol)

### Summary
`setRecoveryAddress` imposes no lifecycle gate. The pool sponsor can call it at any point — including after the moderator has flagged a bad-faith CORRUPTED outcome and before `claimCorrupted` executes — to silently redirect the full pool sweep (stakers' principal + bonus) to an address they control.

### Finding Description
`setRecoveryAddress` is gated only by `onlyOwner` and a zero-address check:

```solidity
// src/ConfidencePool.sol L611-L618
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    address oldRecoveryAddress = recoveryAddress;
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
```

There is no check against `outcome`, `claimsStarted`, `expiryLocked`, or any other lifecycle latch. By contrast, `expiry` is explicitly locked after the first stake (`expiryLocked`) and scope is locked once the registry leaves pre-attack staging (`scopeLocked`). Both locks are documented in DESIGN.md §10 as protecting "staker reliance." `recoveryAddress` carries no equivalent protection.

`claimCorrupted` reads `recoveryAddress` at call time:

```solidity
// src/ConfidencePool.sol L423
stakeToken.safeTransfer(recoveryAddress, toSweep);
```

So any change made between `flagOutcome` and the first `claimCorrupted` call takes full effect. The same late-change window applies to `sweepUnclaimedCorrupted` (L468) and `sweepUnclaimedBonus` (L506), both of which also transfer to the live `recoveryAddress`.

### Impact Explanation
Under bad-faith CORRUPTED the entire pool — stakers' principal plus the bonus — is swept to `recoveryAddress`. A sponsor who changes that address to one they control after the moderator flags the outcome captures 100 % of staker funds. Stakers have no recourse: `withdraw` is permanently disabled once `riskWindowStart != 0`, and the claim paths all revert for CORRUPTED. The loss is total and irreversible.

### Likelihood Explanation
The sponsor is a single EOA or multisig (not labeled "trusted" in the actor model, unlike the factory owner and moderator). The attack requires only one privileged transaction inserted between `flagOutcome` and `claimCorrupted` — a window that can be days or weeks long. A compromised or malicious sponsor key is sufficient; no on-chain coordination or flash-loan is needed. The attack is also front-runnable: the sponsor can watch the mempool for `flagOutcome` and respond in the same block.

### Recommendation
Apply the same "staker reliance" latch used for `expiry`: lock `recoveryAddress` on the first stake (or at the latest on `flagOutcome`). A minimal fix adds a check at the top of `setRecoveryAddress`:

```solidity
if (expiryLocked) revert RecoveryAddressLocked();
```

A stricter fix locks it independently at `flagOutcome` time (setting a dedicated `recoveryAddressLocked` flag), so the sponsor cannot change it even in the pre-stake window once an outcome is imminent.

### Proof of Concept
1. Sponsor creates a pool with `recoveryAddress = daoTreasury`.
2. Stakers deposit; `riskWindowStart` seals; `withdraw` is permanently disabled.
3. Registry reaches `CORRUPTED`; moderator calls `flagOutcome(CORRUPTED, false, address(0))` (bad-faith).
4. Sponsor calls `setRecoveryAddress(sponsorEOA)` — succeeds with no revert.
5. Any caller invokes `claimCorrupted()`; the entire pool balance transfers to `sponsorEOA`.
6. Stakers receive nothing; `hasClaimed` is never set for them; no recovery path exists.
