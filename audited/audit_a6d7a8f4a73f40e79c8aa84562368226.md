Audit Report

## Title
`tickToPrice(0) == 0` Causes `maxAssets` Cap Bypass on Buy Offers at Tick 0 — (`src/libraries/TickLib.sol` / `src/Midnight.sol`)

## Summary

`tickToPrice(0)` evaluates to exactly `0` because the intermediate WAD-scaled price (~4.975 × 10¹¹) is smaller than `PRICE_ROUNDING_STEP` (10¹²), causing the final rounding step to produce zero. Any buy offer placed at `tick = 0` with `maxAssets = M > 0` on a market with zero settlement fee (the protocol default) will compute `buyerAssets = 0` on every `take` call, meaning `consumed` never advances and the `maxAssets` cap is never reached. The offer is fillable an unlimited number of times, `totalUnits` is inflated without token backing, and any maker callback is invocable without bound.

## Finding Description

**Root cause — `tickToPrice(0) == 0`** (`src/libraries/TickLib.sol` lines 44–52):

```
exponent = LN_ONE_PLUS_DELTA * (2910 - 0) ≈ 14.514e18
wExp(14.514e18) ≈ 2.009e24
1e36 / (1e18 + 2.009e24) ≈ 4.975e11
divHalfDownUnchecked(4.975e11, 1e12) = (4.975e11 + 499999999999) / 1e12 = 0
0 * 1e12 = 0
```

`tickToPrice(0)` returns `0`. [1](#0-0) 

**Tick validity** (`src/Midnight.sol` line 351): `DEFAULT_TICK_SPACING = 4` and `0 % 4 == 0`, so `tick = 0` passes the accessibility check. [2](#0-1) [3](#0-2) 

**Settlement fee precondition** (`src/Midnight.sol` line 361): `sellerPrice = offerPrice - _settlementFee`. With `offerPrice = 0`, any non-zero `_settlementFee` causes an arithmetic underflow revert. The default settlement fee is zero (uninitialized mapping), so this path is reachable on every newly created market before the fee setter acts. [4](#0-3) 

**Asset computation** (`src/Midnight.sol` lines 363–364): With `buyerPrice = 0`, `buyerAssets = units.mulDivDown(0, WAD) = 0` and `sellerAssets = 0` for any `units`. [5](#0-4) 

**Consumed accounting** (`src/Midnight.sol` lines 367–369): `consumed[maker][group] += 0` → `newConsumed` stays at 0 → `require(0 <= M)` is trivially satisfied forever. The `maxAssets` cap is never reached. [6](#0-5) 

**State mutation with zero token flow**: Despite `buyerAssets = sellerAssets = 0`, the position accounting still executes. `buyerCreditIncrease = units` (if maker has no debt), `sellerDebtIncrease = units` (if taker has no credit), and `_marketState.totalUnits += buyerCreditIncrease`. Zero loan tokens are transferred. `withdrawable` is not increased. The result is phantom credit for the maker (unwithdrawable) and phantom debt for the taker (unrecompensed), with `totalUnits` inflated without token backing. [7](#0-6) 

**Exploit flow**:
1. Maker creates a buy offer: `tick = 0`, `maxAssets = M > 0`, on a market with zero settlement fee (default).
2. Attacker (taker/seller) calls `take(offer, ..., units = N, ...)` repeatedly (or in a single `multicall`).
3. Each call: `consumed` stays at 0, `totalUnits` grows by `N`, maker accumulates phantom credit, taker accumulates unrecompensed debt, zero tokens transferred.
4. If `offer.callback != address(0)`, the maker's `onBuy` callback is invoked on every iteration without bound.

## Impact Explanation

Three concrete impacts:

1. **`maxAssets` cap invariant broken**: The sole mechanism limiting a buy offer's asset consumption is rendered inoperative. Any offer at tick 0 with `maxAssets > 0` is effectively uncapped.

2. **`totalUnits` inflation without token backing**: `totalUnits` is used as the denominator in bad-debt loss-factor calculations (`_marketState.lossFactor`). Inflating it without depositing tokens dilutes the loss factor, reducing the per-unit credit slash applied to all lenders when bad debt is socialized. An attacker who is also a lender can exploit this to shield their credit from bad-debt haircuts.

3. **Unbounded callback invocation**: A maker whose offer includes a callback (`offer.callback`) will have that callback triggered on every `take` call. If the callback has side effects (state changes, token transfers, reentrancy vectors), this is a griefing or exploitation amplifier.

## Likelihood Explanation

- `tick = 0` is valid on every market (`0 % DEFAULT_TICK_SPACING == 0`).
- Zero settlement fee is the protocol default until a privileged fee setter acts; most markets start in this state.
- No special role, capital, or prior state is required. Any unprivileged address can be the taker.
- The attack is repeatable in a single transaction via `multicall`.
- The only cost to the attacker is gas and the unrecompensed debt they accumulate (which they may simply never repay).

## Recommendation

Two complementary fixes:

1. **Reject zero-price offers in `take`**: Add `require(offerPrice > 0, ZeroPrice())` immediately after computing `offerPrice` at line 358. This prevents any offer at a tick whose price rounds to zero from being taken.

2. **Reject zero-asset trades when `maxAssets > 0`**: Add `require(offer.maxAssets == 0 || (offer.buy ? buyerAssets : sellerAssets) > 0, ZeroAssets())` before the consumed accounting block. This ensures the cap counter always advances when an assets-based cap is in effect.

Either fix alone is sufficient to close the bypass; both together provide defense in depth.

## Proof of Concept

Minimal Foundry test outline:

```solidity
// 1. Create market with zero settlement fee (default).
// 2. Maker signs buy offer: tick=0, maxAssets=1e18, group=bytes32(0).
// 3. Assert tickToPrice(0) == 0.
// 4. Taker calls take(offer, ..., units=1e18, ...) ten times.
// 5. Assert consumed[maker][group] == 0 after all calls (cap never advanced).
// 6. Assert position[id][maker].credit == 10e18 (phantom credit).
// 7. Assert marketState[id].totalUnits == 10e18 (inflated without token backing).
// 8. Assert no loan tokens were transferred (balance unchanged).
```
