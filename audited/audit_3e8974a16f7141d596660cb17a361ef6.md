### Title
Near-zero oracle price causes `seizedAssets` to exceed borrower collateral, reverting valid liquidations with arithmetic underflow - (File: src/Midnight.sol)

### Summary
When the liquidated collateral's oracle returns a very small non-zero price (e.g., 1), the formula `seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice)` produces a value orders of magnitude larger than the borrower's actual collateral balance. The subsequent unchecked subtraction at line 670 underflows, reverting the transaction. No existing guard prevents this: the recovery-close-factor check silently passes because the collateral's value at price=1 rounds to zero, satisfying the RCF condition for any `repaidUnits`.

### Finding Description

**Exact code path:**

1. Liquidator calls `liquidate(market, collateralIndex, 0, repaidUnits, borrower, postMaturityMode, ...)` with `repaidUnits > 0`.
2. The oracle loop (lines 607–618) computes `liquidatedCollatPrice = 1` and `maxDebt ≈ 0`, so `originalDebt > maxDebt` → `NotLiquidatable` check passes.
3. Bad-debt branch (lines 626–641): with price=1, `_collateral.mulDivUp(1, 1e36).mulDivUp(WAD, maxLif) = 1`, so `badDebt = originalDebt - 1` and `_position.debt` is reduced to 1.
4. Line 652 executes: `seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, 1)`. For `repaidUnits = 1`, `lif = WAD = 1e18`: `seizedAssets = 1e36`.
5. Lines 662–667 (only in `!postMaturityMode`): `_position.collateral[collateralIndex].mulDivDown(1, 1e36) = 0`, so `0.zeroFloorSub(maxRepaid) = 0 < rcfThreshold` → require passes unconditionally.
6. Line 670: `uint128 newCollateral = _position.collateral[collateralIndex] - UtilsLib.toUint128(seizedAssets)`. If collateral is any realistic value (e.g., 1e18 tokens), `1e18 - 1e36` underflows → arithmetic revert.

**Root cause:** There is no upper bound enforced on `seizedAssets` relative to `_position.collateral[collateralIndex]` before the subtraction. The RCF check that is supposed to limit `repaidUnits` is bypassed when `liquidatedCollatPrice` is near zero, because the collateral's value expression rounds to zero, making the RCF condition trivially true.

**Attacker-controlled inputs:** `repaidUnits` (any value ≥ 1), `collateralIndex` (pointing to the near-zero-price collateral). The oracle price is a market parameter, not attacker-controlled, but a legitimate price crash to 1 is a realistic market event. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

An unhealthy borrower position with a near-zero oracle price cannot be liquidated via the `repaidUnits` input path. Every call to `liquidate` with `repaidUnits > 0` reverts with an arithmetic underflow (or `CastOverflow` from `toUint128` if `seizedAssets > type(uint128).max`). This violates the core invariant that "unhealthy positions remain liquidatable." The position accumulates bad debt that cannot be resolved, and lenders cannot recover funds through the normal liquidation mechanism. [4](#0-3) 

### Likelihood Explanation

**Preconditions:**
- A market exists with a collateral whose oracle price can drop to a very small value (e.g., 1). This is a realistic scenario for any volatile or depreciating asset.
- A borrower has an active debt position collateralized by that asset.
- The position is unhealthy (price drop makes `originalDebt > maxDebt`).

**Feasibility:** Any unprivileged liquidator can trigger this by calling `liquidate` with `repaidUnits = 1`. No special permissions are required. The condition is repeatable for as long as the oracle price remains near zero.

**Both modes affected:** In `postMaturityMode`, the `maxRepaid` check is entirely absent, making the bug unconditional. In normal mode, the RCF bypass makes it equally reachable. [5](#0-4) [6](#0-5) 

### Recommendation

Add an explicit cap on `seizedAssets` before the subtraction at line 670:

```solidity
seizedAssets = UtilsLib.min(seizedAssets, _position.collateral[collateralIndex]);
```

This mirrors the intent of the `seizedAssets` input path (where the caller explicitly bounds the seized amount) and ensures the subtraction never underflows. Alternatively, compute `seizedAssets` as `min(repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice), _position.collateral[collateralIndex])` and recompute `repaidUnits` from the capped `seizedAssets` using the `mulDivUp` inverse formula, consistent with the `seizedAssets > 0` branch. [7](#0-6)
