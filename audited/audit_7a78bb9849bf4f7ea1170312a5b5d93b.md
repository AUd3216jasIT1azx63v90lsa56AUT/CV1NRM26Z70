Audit Report

## Title
`onLiquidate` callback enables `supplyCollateral` after bad-debt accounting is finalized, causing lenders to be over-slashed - (File: src/Midnight.sol)

## Summary
In `liquidate()`, bad debt is computed from a local snapshot of `_position.collateralBitmap` and irrevocably committed to `lossFactor`, `totalUnits`, and `_position.debt` before the `onLiquidate` callback fires. A callback contract authorized for the borrower can call `supplyCollateral` during this window, injecting collateral that was invisible to the bad-debt formula. Lenders are permanently slashed for a shortfall larger than the true post-liquidation shortfall, while the borrower retains and can freely withdraw the newly supplied collateral.

## Finding Description
**Exact code path:**

1. **Line 606** вЂ” `uint128 _collateralBitmap = _position.collateralBitmap;` snapshots the bitmap into a local variable. Any collateral added after this point is invisible to the bad-debt loop.

2. **Lines 607вЂ“618** вЂ” `badDebt` is computed by iterating only over indices present in the snapshot. Bits are cleared from the local copy, not from storage.

3. **Lines 626вЂ“641** вЂ” `badDebt` is irrevocably committed before any external call: `_position.debt -= uint128(badDebt)`, `_marketState.lossFactor` is updated, and `_marketState.totalUnits` is reduced. Lenders are slashed here.

4. **Lines 698вЂ“715** вЂ” `onLiquidate` callback fires. All bad-debt accounting is already final at this point.

5. Inside the callback, `supplyCollateral` is called for `borrower`. The authorization check at **line 527** (`onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`) is satisfied if the callback contract is the borrower (self-liquidation) or if the borrower pre-authorized the callback contract via `setIsAuthorized`.

6. `supplyCollateral` writes `_position.collateral[collateralIndex]` and sets the corresponding bit in `_position.collateralBitmap` at **lines 533вЂ“537**. Neither write was visible to the already-completed bad-debt loop.

7. **Line 717** вЂ” `safeTransferFrom` pulls loan tokens. No re-check of collateral or bad debt occurs.

**Why existing checks do not stop it:**

- `liquidationLocked` is a transient-storage flag set only inside `take()` at **line 444** via `UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)`. It is never set inside `liquidate()`. It does not block `supplyCollateral` from being called during the `onLiquidate` callback.
- There is no reentrancy guard on `liquidate()` or `supplyCollateral()`.
- The Certora `OnlyAuthorizedCanChange.spec` at **line 73** explicitly acknowledges: *"Assumes no reentrancy: callbacks and token transfers are not modeled as re-entering Midnight, so re-entrant collateral changes are not covered."* This is a documented gap in the formal verification, not a design guarantee of safety.

## Impact Explanation
Bad debt realized = `D в€’ ОЈ(collateral_i В· price_i / maxLif_i)` (pre-callback collateral only). True shortfall after callback = `D в€’ ОЈ(collateral_i В· price_i / maxLif_i) в€’ A В· price_new / maxLif_new`. Over-slashing per attack = `A В· price_new / maxLif_new` units, permanently subtracted from `totalUnits` and added to `lossFactor`. Every lender's effective credit is permanently reduced by this excess amount when they call `updatePosition`. In the full-bad-debt case, `_position.debt` is set to zero before the callback, so the borrower can immediately withdraw all collateral including the newly supplied amount, recovering the flash-loaned tokens at near-zero net cost. This constitutes direct, permanent theft of lender credit with no privileged access required.

## Likelihood Explanation
Required preconditions: (1) Borrower is a smart contract or has pre-authorized the callback вЂ” realistic for any protocol-integrated borrower or DeFi vault. (2) Position has genuine bad debt (oracle price drop) вЂ” the normal trigger for bad-debt realization. (3) Callback contract holds collateral tokens to supply вЂ” sourceable from a flash loan within the same callback, since `supplyCollateral` pulls from `msg.sender` and no repayment is required within `liquidate()`. Self-liquidation (`borrower == liquidator == callback`) requires no third-party cooperation. The attack is repeatable on any market with bad debt and a callback-capable borrower. The attacker controls `seizedAssets = 0`, `repaidUnits = 0`, `callback`, and the collateral amount, requiring no privileged access.

## Recommendation
Move the `onLiquidate` callback invocation to **before** the bad-debt accounting block, or re-snapshot `_position.collateralBitmap` from storage after the callback returns and recompute `badDebt` using the post-callback collateral state. Alternatively, set a transient reentrancy lock at the entry of `liquidate()` (analogous to `LIQUIDATION_LOCK_SLOT` in `take()`) that blocks `supplyCollateral` for the borrower's position until `liquidate()` returns. The simplest safe fix is to add a `nonReentrant`-style transient lock on `liquidate()` that prevents any state-mutating call on the same position during execution.

## Proof of Concept
1. Deploy a malicious borrower contract `AttackBorrower` that implements `ILiquidateCallback.onLiquidate`.
2. `AttackBorrower` borrows from a market and lets the position fall into bad debt (oracle price drop).
3. `AttackBorrower` calls `liquidate()` on itself with `callback = address(this)`, `seizedAssets = 0`, `repaidUnits = 0`.
4. Inside `onLiquidate`, `AttackBorrower` flash-loans collateral token amount `A` and calls `supplyCollateral(market, newIndex, A, address(AttackBorrower))`.
5. After `liquidate()` returns, `_position.debt == 0` (full bad debt case). `AttackBorrower` calls `withdrawCollateral` to recover all collateral including `A`.
6. Observe: `_marketState.lossFactor` and `_marketState.totalUnits` reflect a bad-debt amount larger than the true shortfall by `A В· price_new / maxLif_new`. All lenders' effective credit is permanently reduced by this excess.
