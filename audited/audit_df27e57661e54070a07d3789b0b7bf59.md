### Title
Fee-on-transfer `loanToken` in `repay()` inflates `withdrawable` beyond actual contract balance, causing lender fund freeze - (`File: src/Midnight.sol`)

### Summary

`repay()` unconditionally credits `marketState[id].withdrawable` by the full `units` parameter at line 509, then calls `SafeTransferLib.safeTransferFrom` at line 520, which only validates the boolean return value of `transferFrom` — not the actual token amount received. Because `touchMarket()` imposes no restriction on `market.loanToken` and the protocol explicitly supports permissionless market creation with arbitrary loan tokens, a market creator can deploy a fee-on-transfer token as `loanToken`. Every `repay()` call then inflates `withdrawable` by `units` while the contract receives only `units × (1 − fee_rate)`, permanently breaking the invariant `withdrawable ≤ loanToken.balanceOf(address(midnight))` and leaving the last lenders unable to withdraw.

### Finding Description

**Exact code path:**

`repay()` in `src/Midnight.sol`: [1](#0-0) 

State is mutated at lines 508–509 before the external call at line 520: [2](#0-1) 

`SafeTransferLib.safeTransferFrom` only checks `success` and the boolean return value; it performs no balance-delta check: [3](#0-2) 

`touchMarket()` validates collateral params, LLTV tiers, and maturity, but places **zero restriction on `market.loanToken`**: [4](#0-3) 

The protocol design explicitly declares "permissionless market creation" and "arbitrary loan token": [5](#0-4) 

**Attacker-controlled inputs:**
- `market.loanToken` — set to an attacker-deployed fee-on-transfer ERC20 (e.g., 1% fee on every `transferFrom`)
- `units` — any value in `[1, position.debt]`
- `onBehalf` — the borrower's own address (no special privilege needed)
- `callback = address(0)`, `data = ''`

**Exploit flow:**
1. Attacker deploys `FeeToken` with a 1% fee deducted from the recipient on every `transferFrom`.
2. Attacker calls `touchMarket(market)` with `market.loanToken = address(feeToken)` — succeeds with no loanToken validation.
3. A lender supplies credit (via `take()`); the market now has lenders with `credit` and `totalUnits`.
4. Borrower accumulates `debt` via `take()`.
5. Borrower calls `repay(market, units, borrower, address(0), '')`:
   - Line 508: `position[id][borrower].debt -= units` ✓
   - Line 509: `marketState[id].withdrawable += units` ← inflated by full `units`
   - Line 520: `safeTransferFrom(feeToken, borrower, address(this), units)` → contract receives `units × 0.99`
6. After each repayment: `withdrawable` exceeds actual `feeToken.balanceOf(address(midnight))` by `units × 0.01`.
7. After N repayments: `withdrawable = N × units`, `balance = N × units × 0.99`. The gap grows linearly.
8. When lenders call `withdraw()`, line 499 executes `safeTransfer(loanToken, receiver, units)`. The last `N × units × 0.01` worth of lender withdrawals revert because the contract is insolvent.

**Why existing checks fail:**
- `SafeTransferLib` checks only the ERC20 boolean return, not the received amount.
- `touchMarket()` has no loanToken allowlist or fee-detection logic.
- There is no balance-before/after guard anywhere in `repay()`.
- The authorization check (`onBehalf == msg.sender`) is satisfied trivially by the borrower repaying their own debt.

### Impact Explanation

`marketState[id].withdrawable` permanently exceeds `feeToken.balanceOf(address(midnight))` for the affected market. Lenders who call `withdraw()` first drain the real balance; the remaining lenders' `withdraw()` calls revert with an ERC20 transfer failure. The shortfall equals `totalRepaid × fee_rate` and grows with every repayment. Lender principal is permanently frozen — this is concrete protocol insolvency scoped to the affected market.

### Likelihood Explanation

- **Preconditions**: Attacker must deploy a fee-on-transfer ERC20 and create a market with it. Both are permissionless and require no special role.
- **Feasibility**: Trivial. The attacker can be both the market creator and a borrower (or recruit a borrower). No oracle manipulation, no admin key, no impossible state required.
- **Repeatability**: Every single `repay()` call compounds the shortfall. The bug is triggered on the very first repayment and worsens monotonically.

### Recommendation

In `repay()`, measure the actual token amount received by snapshotting the balance before and after the transfer, and use the delta — not `units` — to update `withdrawable` and `debt`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;

position[id][onBehalf].debt -= UtilsLib.toUint128(received);
marketState[id].withdrawable += UtilsLib.toUint128(received);
```

This must also be applied consistently in `liquidate()`, `take()`, and `flashLoan()` for the same class of token. Alternatively, document that fee-on-transfer tokens are unsupported and add a loanToken allowlist enforced in `touchMarket()`.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
