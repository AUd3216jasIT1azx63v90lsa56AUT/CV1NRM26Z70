### Title
Borrower Can Permanently Block Liquidation by Activating a Reverting-Oracle Collateral — (`File: src/Midnight.sol`)

---

### Summary

`liquidate()` iterates over every activated collateral and calls its oracle with no error handling. `supplyCollateral()` never calls the oracle. A borrower facing liquidation can supply 1 wei of a collateral whose oracle is reverting, activating it in their bitmap, and every subsequent `liquidate()` call against them will revert — shielding them from liquidation for as long as the oracle is broken.

---

### Finding Description

**Vulnerability class:** DoS on a critical protocol function via a reverting external call embedded in the execution path.

**Root cause — `liquidate()` oracle loop (no try/catch):**

```solidity
// src/Midnight.sol lines 607–618
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price();   // ← bare call, no try/catch
    ...
    _collateralBitmap = _collateralBitmap.clearBit(i);
}
``` [1](#0-0) 

If **any** oracle in the loop reverts, the entire `liquidate()` transaction reverts.

**Entry point — `supplyCollateral()` never calls the oracle:**

```solidity
// src/Midnight.sol lines 524–545
function supplyCollateral(...) external {
    ...
    _position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);
    if (oldCollateral == 0 && assets > 0) {
        _position.collateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
        ...
    }
    SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
}
``` [2](#0-1) 

No oracle is consulted, so a borrower can activate any collateral slot — including one whose oracle is currently reverting.

The protocol itself documents this exact combination as a known liveness risk:

> *"Liquidation reverts if any of the activated collaterals' oracle reverts (see LIVENESS). Note that a borrower can activate a collateral once its oracle is reverting because the oracle is not called in supplyCollateral."* [3](#0-2) 

**Exploit flow:**

1. Market `M` has collaterals `[C0, C1]`. `C1`'s oracle has a circuit-breaker / pause mechanism (common in Chainlink, Pyth, etc.).
2. Borrower `B` borrows against `C0` and their position becomes unhealthy (or they anticipate it will).
3. `C1`'s oracle trips its circuit-breaker and starts reverting.
4. `B` calls `supplyCollateral(M, 1 /*C1 index*/, 1 wei, B)` — succeeds, sets bit 1 in `B`'s `collateralBitmap`.
5. Any liquidator calling `liquidate(M, ..., B, ...)` now hits the oracle loop, calls `C1.oracle.price()`, and reverts.
6. `B` is immune to liquidation for the entire duration of the oracle outage.
7. If prices recover during the outage, `B` exits healthy; if not, the protocol accumulates unresolvable bad debt.

---

### Impact Explanation

- **Liquidation is completely bricked** for the targeted borrower for the duration of the oracle outage.
- Bad debt that should have been socialized via `lossFactor` is not realized, leaving lenders exposed to a larger eventual loss.
- In post-maturity mode the same loop runs, so even the time-based liquidation path is blocked.
- The borrower retains their collateral and debt position, effectively getting a free option on price recovery at the protocol's expense.

---

### Likelihood Explanation

- Multi-collateral markets are a core feature; any market with ≥ 2 collaterals is potentially affected.
- Circuit-breaker oracles (Chainlink sequencer uptime feeds, Pyth confidence-interval guards, custom staleness checks) are standard in production deployments and revert under stress — exactly when liquidations are most needed.
- The attacker needs only: (a) an existing debt position, (b) 1 wei of the collateral token with the reverting oracle, and (c) knowledge of the oracle's state. No privileged access is required.
- The cost is negligible (1 wei + gas); the benefit is avoiding liquidation of an arbitrarily large position.

---

### Recommendation

Wrap each oracle call in a `try/catch` inside the collateral loop. If an oracle reverts, treat that collateral's contribution as zero for `maxDebt` (conservative for the borrower) and zero for `badDebt` reduction (conservative for the protocol), or revert only when the **liquidated** collateral's oracle fails. This mirrors the fix recommended in the Telcoin report: decouple the critical path from the potentially-failing external calls.

```solidity
try IOracle(_collateralParam.oracle).price() returns (uint256 price) {
    if (i == collateralIndex) liquidatedCollatPrice = price;
    maxDebt += ...;
    badDebt = badDebt.zeroFloorSub(...);
} catch {
    // oracle unavailable: treat collateral as contributing 0 to maxDebt
    // (safe: makes position appear more underwater, not less)
}
```

Alternatively, require that the **liquidated** collateral's oracle must succeed (it is needed for price computation), but allow other activated collaterals' oracle failures to be skipped with a zero contribution.

---

### Proof of Concept

**Preconditions:**
- Market with collaterals `[WETH (C0, healthy oracle), TOKEN_X (C1, oracle with circuit-breaker)]`.
- Borrower `B` has 100 units of debt backed by WETH; position is unhealthy (WETH price dropped).
- `TOKEN_X` oracle is currently reverting (circuit-breaker triggered).

**Steps:**
1. `B` holds 1 wei of `TOKEN_X` (trivially obtainable on any DEX).
2. `B` calls `supplyCollateral(market, 1, 1, B)` → succeeds; `collateralBitmap` now has bits `{0, 1}` set.
3. Liquidator calls `liquidate(market, 0, 0, repaidUnits, B, false, receiver, address(0), "")`.
4. Inside `liquidate`, the while-loop hits `i=1`, calls `IOracle(C1.oracle).price()` → **reverts**.
5. Entire `liquidate()` call reverts. Borrower `B` is not liquidated.
6. Repeat for any liquidation attempt until `C1`'s oracle recovers. [1](#0-0) [4](#0-3)
