Audit Report

## Title
`consumableUnits` ignores `offer.reduceOnly`, causing bundler to silently skip permanently-reverting offers and revert with `OutOfOffers` - (File: src/periphery/ConsumableUnitsLib.sol)

## Summary
`ConsumableUnitsLib.consumableUnits` computes fillable units solely from `consumed[maker][group]` vs `offer.maxUnits`/`offer.maxAssets`, with no awareness of `offer.reduceOnly` or the maker's current `credit`/`debt`. When a `reduceOnly` sell offer's maker holds zero credit (or a `reduceOnly` buy offer's maker holds zero debt), `consumableUnits` returns a positive value, but every subsequent `Midnight.take()` call will deterministically revert with `MakerCreditOrDebtIncreased`. The bundler's `catch {}` silently absorbs the revert, `filledUnits` is never incremented, and the bundler ultimately reverts with `OutOfOffers`, griefing any taker who routes through it.

## Finding Description

**Root cause — missing `reduceOnly` guard in `consumableUnits`:**

`src/periphery/ConsumableUnitsLib.sol` lines 14–23 read only the `consumed` counter and `offer.maxUnits`/`offer.maxAssets`:

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    if (offer.maxUnits > 0) {
        return offer.maxUnits.zeroFloorSub(consumed);   // ← no reduceOnly check
    } else if (offer.buy) {
        return TakeAmountsLib.buyerAssetsToUnits(...);
    } else {
        return TakeAmountsLib.sellerAssetsToUnits(...);
    }
}
```

`offer.reduceOnly` and the maker's on-chain `credit`/`debt` are never consulted.

**`take()` enforces the constraint that `consumableUnits` ignores:**

`src/Midnight.sol` lines 382–395:

```solidity
uint256 sellerCreditDecrease = UtilsLib.min(units, sellerPos.credit);
uint256 sellerDebtIncrease   = units - sellerCreditDecrease;
...
require(
    !offer.reduceOnly || (offer.buy ? buyerCreditIncrease == 0 : sellerDebtIncrease == 0),
    MakerCreditOrDebtIncreased()
);
```

For a `reduceOnly` sell offer where `sellerPos.credit == 0`: `sellerCreditDecrease = 0`, `sellerDebtIncrease = units`. The require evaluates to `false || (units == 0)`, which always fails for any `units > 0`.

**Bundler silently absorbs the revert:**

`src/periphery/MidnightBundles.sol` lines 74–88:

```solidity
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // returns N > 0
);
try IMidnight(MIDNIGHT).take(..., unitsToTake, ...) returns (...) {
    filledUnits += unitsToTake;
} catch {}   // MakerCreditOrDebtIncreased swallowed here
```

`filledUnits` is never incremented. After the loop, `require(filledUnits == targetUnits, OutOfOffers())` fires.

**Exploit flow:**

1. Maker (attacker) signs a `reduceOnly` sell offer with `maxUnits = N` while holding `credit = 0` in the target market (or withdraws all credit after signing).
2. Taker (victim) constructs a bundler call (e.g., `buyWithUnitsTargetAndWithdrawCollateral`) with `targetUnits = N` and `takes = [attackerOffer]`.
3. Bundler calls `consumableUnits` → returns `N` (positive; no `reduceOnly` check).
4. Bundler calls `take(offer, ..., N, ...)` → `sellerDebtIncrease = N`, `reduceOnly` check fails → reverts with `MakerCreditOrDebtIncreased`.
5. `catch {}` swallows the revert; `filledUnits` stays 0.
6. Loop ends; `require(0 == N, OutOfOffers())` → entire bundler call reverts.

The same path applies to a `reduceOnly` buy offer where the maker's `debt == 0` (`buyerCreditIncrease = units > 0` → revert). All four bundle functions (`buyWithUnitsTargetAndWithdrawCollateral`, `supplyCollateralAndSellWithUnitsTarget`, `buyWithAssetsTargetAndWithdrawCollateral`, `supplyCollateralAndSellWithAssetsTarget`) are affected.

**Why existing protections are insufficient:**

The bundler comment at line 42 of `MidnightBundles.sol` explicitly states it "Skips every reason why take can revert (including ones that are not asynchrony related)." The `try/catch` is intentional, but `consumableUnits` is supposed to pre-filter offers that cannot be filled. It fails to do so for `reduceOnly` offers whose maker has no position to reduce.

## Impact Explanation

A maker can publish a `reduceOnly` offer with no existing credit/debt. Any taker who routes through the bundler with this offer as the sole (or gap-filling) entry will have their transaction revert with `OutOfOffers`. The taker loses gas and may miss time-sensitive operations (e.g., liquidation-adjacent collateral withdrawal, rate-sensitive borrowing). The bundler's `pullToken`/`forceApproveMax` sequence executes before the loop, so the taker's tokens are temporarily pulled and then returned on revert — no permanent fund loss, but the transaction is reliably and repeatably griefable. This constitutes service unavailability/severe degradation under realistic attacker input, which is an in-scope impact per RESEARCHER.md.

## Likelihood Explanation

**Preconditions:** Maker creates a `reduceOnly` offer with `maxUnits > 0` and `credit == 0` (sell) or `debt == 0` (buy). This is a valid, signable offer state — no privileged access required. The `Offer` struct confirms `reduceOnly` is a standard field (`src/interfaces/IMidnight.sol` line 33).

**Feasibility:** Trivially reachable. A maker can sign such an offer at any time. Off-chain order books and aggregators rely on `consumableUnits` to filter offers; a positive return value will cause them to include the offer.

**Repeatability:** The maker's offer remains valid (not consumed, not expired) after the bundler reverts, so the same offer can grief repeated bundler calls until the taker manually excludes it.

## Recommendation

In `ConsumableUnitsLib.consumableUnits`, add an early-exit guard for `reduceOnly` offers by reading the maker's on-chain position:

```solidity
if (offer.reduceOnly) {
    Position memory pos = IMidnight(midnight).position(id, offer.maker);
    if (offer.buy ? pos.debt == 0 : pos.credit == 0) return 0;
}
```

This should be applied before the `maxUnits`/`maxAssets` branch so that a `reduceOnly` offer with no reducible position is treated as fully consumed (returns 0), causing the bundler to skip it cleanly rather than attempting a doomed `take()`.

Alternatively, the bundler could check `offer.reduceOnly` and the maker's position inline before computing `unitsToTake`, but fixing it in `ConsumableUnitsLib` is preferable since it is the canonical pre-filter used by all four bundle functions.

## Proof of Concept

**Minimal Foundry test outline:**

1. Deploy `Midnight` and `MidnightBundles`.
2. Create a market; do not give the maker any credit or debt.
3. Construct a `reduceOnly` sell offer (`offer.reduceOnly = true`, `offer.maxUnits = 100`, `offer.buy = false`) signed by the maker (credit = 0).
4. Call `buyWithUnitsTargetAndWithdrawCollateral` with `targetUnits = 100` and `takes = [attackerOffer]`.
5. Assert the call reverts with `OutOfOffers`.
6. Verify that calling `ConsumableUnitsLib.consumableUnits` for the same offer returns 100 (not 0), confirming the missing guard.

The existing test `testReduceOnlySellRevert` in `test/TakeTest.sol` (lines 519–529) already confirms that `take()` reverts with `MakerCreditOrDebtIncreased` when `exitUnits > existingUnits` (i.e., when the maker lacks sufficient credit). The bundler PoC extends this by wrapping the same scenario in a bundle call and observing the `OutOfOffers` revert.
