### Title
Fee-on-transfer loan token inflates `claimableSettlementFee` beyond actual contract balance in `Midnight.take` (pre- and post-maturity) - (File: src/Midnight.sol)

### Summary
In `Midnight.take`, `claimableSettlementFee` is incremented by the nominal `buyerAssets - sellerAssets` at line 418 before the actual `safeTransferFrom` at line 455. `SafeTransferLib.safeTransferFrom` only checks the boolean return value and never verifies the amount actually received. With a fee-on-transfer loan token, the contract receives less than accounted, creating a permanent deficit between `claimableSettlementFee` and the real token balance. This applies to all takes including post-maturity unwinding where `sellerDebtIncrease == 0` is enforced but a non-zero 0-day settlement fee still generates a fee spread.

### Finding Description
**Exact code path:**

`take()` computes `buyerAssets - sellerAssets` as the settlement fee spread using the 0-day breakpoint post-maturity (`timeToMaturity = zeroFloorSub(maturity, block.timestamp) = 0`): [1](#0-0) 

The accounting is updated before any transfer: [2](#0-1) 

Then the actual transfer occurs: [3](#0-2) 

`SafeTransferLib.safeTransferFrom` only validates the boolean return, never the received amount: [4](#0-3) 

**Root cause:** No balance-before/after check surrounds the `safeTransferFrom` call. With a fee-on-transfer token charging rate `f`, the contract receives `(buyerAssets - sellerAssets) * (1 - f)` but `claimableSettlementFee` is incremented by the full `buyerAssets - sellerAssets`. The deficit `(buyerAssets - sellerAssets) * f` is permanent and accumulates across every take.

**Post-maturity path:** Line 391 enforces `sellerDebtIncrease == 0`, restricting takes to credit-to-credit unwinding. However, the 0-day settlement fee breakpoint can be non-zero (set by `setDefaultSettlementFee` or `setMarketSettlementFee`), so `buyerAssets - sellerAssets > 0` is reachable post-maturity, and the same inflation occurs. [5](#0-4) 

**Attacker inputs:** Market creator (permissionless) deploys a market with a fee-on-transfer loan token. Any taker then calls `take` with a valid offer. No privileged access is required beyond market creation, which is open.

**Why existing checks fail:** The Certora solvency spec explicitly models the gap between the `claimableSettlementFee` increment and the inbound transfer using `pendingFeeReceipt`, and proves `pendingFeeReceiptZero` holds — but this proof assumes `transferFrom` delivers exactly the requested amount. With fee-on-transfer tokens, `pendingFeeReceipt` is never fully cleared, permanently violating `tokenBalanceCorrect`: [6](#0-5) 

### Impact Explanation
`claimableSettlementFee[loanToken]` exceeds the actual token balance held by the contract. When `claimSettlementFee` is called, it attempts to transfer the full accounted amount: [7](#0-6) 

This either reverts (fee claimer cannot claim), or — if the contract holds surplus from other sources — silently drains funds backing `withdrawable` (lender redemptions). Post-maturity, where the protocol is in pure unwinding mode and no new debt is created, every take with a non-zero 0-day settlement fee widens the deficit, accelerating insolvency for lenders attempting to withdraw.

### Likelihood Explanation
**Preconditions:**
1. Market created with a fee-on-transfer loan token (permissionless market creation).
2. Non-zero settlement fee at the 0-day breakpoint (configurable, default may be non-zero).
3. At least one take executed post-maturity with `sellerCreditDecrease > 0` (i.e., a lender unwinding).

All three preconditions are reachable without any privileged action beyond market creation. The attack is repeatable: each post-maturity take with a non-zero fee spread widens the deficit additively.

### Recommendation
Replace the two-transfer pattern with a balance-check wrapper that measures actual received amounts, or explicitly document and enforce that fee-on-transfer tokens are unsupported (e.g., add a market creation check or a received-amount assertion after each `safeTransferFrom` into the contract):

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
require(
    IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore == buyerAssets - sellerAssets,
    FeeOnTransferNotSupported()
);
```

Alternatively, use the actual received delta to update `claimableSettlementFee` instead of the nominal `buyerAssets - sellerAssets`.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {

// For any token, the pending settlement fee receipt after a transaction is 0: every claimableSettlementFee
// increment in take is paid back in by the same-function inbound transfer.
weak invariant pendingFeeReceiptZero(address token)
    pendingFeeReceipt[token] == 0;

// For any token, the balance of the contract is always greater than or equal to the sum of all collateral, withdrawable, and claimable settlement fee amounts for that token minus the flash loaned amount.
// Note: this invariant is strong, so it also holds before each external call.
strong invariant tokenBalanceCorrect(address token)
    tokenBalances[token][currentContract] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token) - flashloans[token] - pendingFeeReceipt[token]
