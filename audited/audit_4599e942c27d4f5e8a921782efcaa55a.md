### Title
Missing zero-unit skip guard in bundler loops causes unnecessary `take` calls and gas waste for fully consumed buy offers in maxAssets mode - (File: src/periphery/MidnightBundles.sol)

### Summary
When a buy offer has `maxAssets > 0` and `consumed[maker][group] == maxAssets`, `ConsumableUnitsLib.consumableUnits` returns 0 via `buyerAssetsToUnits(..., 0)`. The four bundler loops in `MidnightBundles` have no `if (unitsToTake == 0) continue;` guard, so they call `Midnight.take` with 0 units. Because `take` with 0 units is an explicitly documented no-op that succeeds even when consumed equals the cap, `filledUnits` does not advance, wasting an external call per fully consumed offer. If every offer in the array is fully consumed, the bundler exhausts the array and reverts with `OutOfOffers`.

### Finding Description

**Code path — `ConsumableUnitsLib.consumableUnits`:**

`ConsumableUnitsLib.sol` lines 14–23:
```
if (offer.maxUnits > 0) { ... }
else if (offer.buy) {
    return TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer,
               offer.maxAssets.zeroFloorSub(consumed));   // → 0 when consumed == maxAssets
}
``` [1](#0-0) 

`zeroFloorSub(maxAssets, maxAssets)` = 0 (assembly: `mul(gt(x,y), sub(x,y))` is 0 when x == y). [2](#0-1) 

`buyerAssetsToUnits(..., 0)` = `mulDivUp(0, WAD, buyerPrice)` = `(0 + buyerPrice − 1) / buyerPrice` = 0 for any `buyerPrice ≥ 1`. [3](#0-2) 

**Code path — bundler loop (all four functions share the same pattern):**

`MidnightBundles.sol` lines 147–160 (`supplyCollateralAndSellWithUnitsTarget`):
```solidity
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // → 0
);
try IMidnight(MIDNIGHT).take(takes[i].offer, ..., unitsToTake, ...)   // called with 0
    returns (...) {
    filledUnits += unitsToTake;   // += 0, no progress
    ...
} catch {}
``` [4](#0-3) 

There is no `if (unitsToTake == 0) continue;` guard before the `try` block in any of the four bundler functions.

**Why `take(units=0)` succeeds when consumed == maxAssets:**

`Midnight.sol` lines 363–369:
```solidity
uint256 buyerAssets = units.mulDivDown(buyerPrice, WAD);  // 0
newConsumed = consumed[offer.maker][offer.group] += buyerAssets;  // += 0 → still maxAssets
require(newConsumed <= offer.maxAssets, ConsumedAssets());  // maxAssets <= maxAssets → passes
``` [5](#0-4) 

This is explicitly documented in the protocol: [6](#0-5) 

The Certora spec `fullyConsumedOfferRevertsOnNonTrivialTake` only covers the `maxUnits` mode (`require offer.maxAssets == 0`); there is no analogous rule for the `maxAssets` mode, confirming the gap. [7](#0-6) 

**Exploit flow:**
1. Attacker (any taker) front-runs victim's bundler call and takes the offer, setting `consumed == maxAssets`.
2. Victim calls `supplyCollateralAndSellWithUnitsTarget` (or any of the three other bundler functions) with the now-fully-consumed buy offer in the `takes` array.
3. `consumableUnits` returns 0; `unitsToTake = min(..., 0) = 0`.
4. Bundler calls `Midnight.take(..., 0, ...)` — succeeds as a no-op.
5. `filledUnits += 0`; loop advances to `i+1`.
6. If no remaining offers can fill `targetUnits`, the loop exits and `require(filledUnits == targetUnits, OutOfOffers())` reverts. [8](#0-7) 

**Existing protections are insufficient:** The `try/catch` only catches reverts; it does not skip zero-unit calls. The loop condition `filledUnits < targetUnits` does not prevent iterating over exhausted offers because `i` still increments.

### Impact Explanation
Every fully consumed buy offer (maxAssets mode) in the `takes` array causes one unnecessary cross-contract `take` call, wasting gas. If the maker's offer carries a callback (`offer.callback != address(0)`), the callback is also invoked with 0 assets — an unintended side-effect. If all offers in the array are fully consumed, the bundler reverts with `OutOfOffers` after burning gas on every wasted call. The scoped impact — gas waste and griefing of the taker after full consumption — is concrete and reproducible.

### Likelihood Explanation
Preconditions: a buy offer with `maxAssets > 0` must exist and be fully consumed before the victim's bundler transaction lands. Any taker (including a griefing attacker) can achieve this by front-running. The attack is cheap (one `take` to consume the offer), repeatable, and requires no special privileges. It applies identically to all four bundler functions that use `ConsumableUnitsLib.consumableUnits`.

### Recommendation
Add an explicit zero-unit skip guard before each `try` block in all four bundler loops:

```solidity
if (unitsToTake == 0) continue;
```

This ensures fully consumed offers are skipped without an external call, matching the intent of `consumableUnits` returning 0 as a "nothing left to take" signal.

### Proof of Concept

```solidity
// Foundry unit test
function testFullyConsumedBuyOfferWastesGas() public {
    // Setup: buy offer with maxAssets = X, consumed = X (fully consumed)
    offer.buy = true;
    offer.maxAssets = 100e18;
    offer.maxUnits = 0;
    vm.prank(offer.maker);
    midnight.setConsumed(offer.group, offer.maxAssets, offer.maker);

    // Assert consumableUnits returns 0
    uint256 cu = ConsumableUnitsLib.consumableUnits(address(midnight), id, offer);
    assertEq(cu, 0, "consumableUnits must be 0 for fully consumed offer");

    // Assert take(0) succeeds (no-op) even when consumed == maxAssets
    uint256 consumedBefore = midnight.consumed(offer.maker, offer.group);
    (uint256 ba, uint256 sa) = midnight.take(offer, "", 0, taker, address(0), address(0), "");
    assertEq(ba, 0);
    assertEq(sa, 0);
    assertEq(midnight.consumed(offer.maker, offer.group), consumedBefore, "consumed unchanged");

    // Assert bundler wastes a take call and reverts OutOfOffers
    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offer, units: 100e18, ratifierData: ""});
    vm.expectRevert(IMidnightBundles.OutOfOffers.selector);
    midnightBundles.supplyCollateralAndSellWithUnitsTarget(
        1e18, 0, taker, taker, new CollateralSupply[](0), takes, 0, address(0)
    );
    // Expected: bundler called take(0) once (gas wasted), then reverted
}
```

Key assertions:
- `consumableUnits == 0` for a fully consumed buy offer in maxAssets mode.
- `take(units=0)` returns `(0, 0)` and does not revert.
- `consumed` is unchanged after the 0-unit take.
- The bundler reverts with `OutOfOffers` after the wasted call, confirming the griefing path.

/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
