Audit Report

## Title
Authorized-Signer Front-Run Advances Victim's Nonce and Invalidates Pending Signed Authorization - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` permits any address that `authorization.authorizer` has previously authorized in Midnight to sign an arbitrary `Authorization` struct — including one with a different `authorized` or `isAuthorized` value — at the same nonce. Because the nonce is per-authorizer and increments on any valid call, an authorized attacker can front-run the victim's pending transaction, consume nonce N, and cause the victim's transaction to revert with `InvalidNonce()`. The attacker can repeat this indefinitely and can also set arbitrary `isAuthorized` entries on behalf of the victim.

## Finding Description
**Root cause:**

In `EcrecoverAuthorizer.setIsAuthorized` ( [1](#0-0) ), the nonce is keyed only on `authorization.authorizer` and the signer check accepts any address for which `IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)` returns `true`. There is no binding between the signer's identity and the `authorized` field, and no constraint that the signer may only submit a struct the authorizer themselves pre-approved.

```solidity
// line 26
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
// lines 33-36
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

The `Authorization` struct contains five independently attacker-controllable fields: `authorizer`, `authorized`, `isAuthorized`, `nonce`, `deadline`. [2](#0-1) 

**Exploit flow:**

1. Precondition: `isAuthorized[victim][attacker] == true` in Midnight (victim previously authorized attacker as an operator/relayer).
2. Victim signs `Authorization{authorizer=victim, authorized=X, isAuthorized=true, nonce=N}` and broadcasts.
3. Attacker observes victim's tx in mempool; crafts `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N}` (same nonce, different `authorized`).
4. Attacker signs this struct with their own private key → `ecrecover(digest, ...) == attacker`.
5. Attacker submits with higher gas priority.
6. Contract execution:
   - Line 25: deadline check passes (attacker sets any valid future deadline).
   - Line 26: `N == nonce[victim]` → passes; nonce increments to N+1.
   - Line 31: `signer = attacker`.
   - Lines 33–36: `isAuthorized[victim][attacker] == true` → passes.
   - Line 47: `Midnight.setIsAuthorized(attacker, true, victim)` executes. [3](#0-2) 
7. Victim's tx arrives: `N != N+1` → reverts with `InvalidNonce()`.

**Why existing checks fail:**

- The nonce check prevents replay of the same signed message but does not prevent a *different* message at the same nonce from a delegated signer.
- The `isAuthorized` delegation path was designed to allow authorized agents to act on behalf of the authorizer, but it does not scope what struct content the agent may sign — the agent can sign any `authorized`/`isAuthorized` combination.
- There is no binding between the signer's identity and the `authorized` field in the struct.

## Impact Explanation
The victim's pending signed `Authorization` is permanently invalidated (DoS of delegation). The victim must sign a new message at nonce N+1, which the attacker can front-run again indefinitely. Additionally, the attacker can set arbitrary `isAuthorized` entries on behalf of the victim — authorizing malicious addresses or revoking existing authorizations — which constitutes unauthorized state mutation of the victim's authorization state in Midnight. This maps to: unauthorized state changes, privilege escalation from authorized-agent to full authorization-manager, and permanent service degradation for the victim's signature-gated delegation path.

## Likelihood Explanation
The precondition (`isAuthorized[victim][attacker] == true`) is a common, expected state — users routinely authorize operators, bots, and relayers. Once met, the attack requires only mempool monitoring and a higher-gas competing transaction, which is standard MEV infrastructure. The attack is trivially repeatable on every future signed authorization the victim broadcasts, making it a persistent griefing vector with no natural expiry.

## Recommendation
Restrict the delegation path so that a delegated signer may only relay a struct that the `authorizer` themselves signed, rather than signing an independent struct. Concretely:

1. **Require the authorizer's own signature always.** Remove the `isAuthorized` delegation path from `EcrecoverAuthorizer` entirely. Authorized agents who need to submit on behalf of a user should hold the user's pre-signed message, not sign their own.
2. **Alternatively, bind the signer to the `authorized` field.** If delegation must be preserved, add a constraint that `signer == authorization.authorized` when the signer is not the authorizer, preventing a delegated signer from naming an arbitrary `authorized` address.
3. **Use per-signer nonces.** Key the nonce on `(authorizer, signer)` rather than `authorizer` alone, so a delegated signer consuming a nonce does not invalidate the authorizer's own pending messages.

## Proof of Concept
```
1. Deploy Midnight + EcrecoverAuthorizer on a local fork.
2. victim.call(midnight.setIsAuthorized(attacker, true, victim))
   → isAuthorized[victim][attacker] = true
3. victim signs Authorization{authorizer=victim, authorized=someAddr, isAuthorized=true, nonce=0, deadline=future}
   → victimSig
4. attacker crafts Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=future}
   → attackerSig (attacker's own ECDSA signature over this struct)
5. attacker submits ecrecoverAuthorizer.setIsAuthorized(attackerStruct, attackerSig) with higher gas
   → succeeds; nonce[victim] = 1; isAuthorized[victim][attacker] = true (re-confirmed or escalated)
6. victim submits ecrecoverAuthorizer.setIsAuthorized(victimStruct, victimSig)
   → reverts: InvalidNonce() (nonce is now 1, struct has nonce=0)
```
A Foundry fuzz test can confirm this by fuzzing `authorized` and `isAuthorized` in the attacker's struct while holding `authorizer` and `nonce` fixed to the victim's values.
