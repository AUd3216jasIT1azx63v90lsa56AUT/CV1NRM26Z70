### Title
Buy-offer `consumed` accounting silently rounds to zero for any tick below MAX_TICK, allowing unlimited overfill beyond `maxAssets` - (`src/Midnight.sol`)

### Summary

When a buy offer has `maxAssets > 0`, the `consumed` mapping is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because `tickToPrice` returns values in `[0, 1e18]` and `PRICE_ROUNDING_STEP = 1e12`, any tick below `MAX_TICK` yields `buyerPrice < 1e18`. For `units` small enough that `units * buyerPrice < WAD`, `mulDivDown` truncates to zero, so `consumed` is never incremented. The `require(newConsumed <= offer.maxAssets)` guard then passes unconditionally on every such call, and the offer can be filled with arbitrarily many units while `consumed` stays at zero.

### Finding Description

**Exact code path** — `src/Midnight.sol` lines 366–369:

```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

`buyerAssets` is computed at line 363:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
```

`mulDivDown` is plain integer division: `(units * buyerPrice) / 1e18`.

**Price range** — confirmed by `test/TickLibTest.sol` line 17: `tickToPrice(2) == 1e12` (the first non-zero price). For any tick in `[2, MAX_TICK-1]`, `buyerPrice ∈ [1e12, 1e18 - 1e12]`. For `units ≤ 999_999` with `buyerPrice = 1e12`: `(999_999 × 1e12) / 1e18 = 0`. So `buyerAssets = 0` and `consumed` is unchanged.

**Attacker-controlled inputs**:
- `offer.buy = true`, `offer.maxAssets = X > 0`, `offer.tick` set to any tick < `MAX_TICK`
- `units` chosen so that `units * buyerPrice < 1e18` (e.g., `units = 999_999` at tick 2)
- Valid ratifier data (EcrecoverRatifier Merkle proof whose root is not canceled)

**Exploit flow**:
1. Maker posts a buy offer: `maxAssets = 100`, tick = 2 (`buyerPrice = 1e12`), `offer.buy = true`
2. Taker calls `take(offer, ratifierData, 999_999, taker, ...)` — `buyerAssets = 0`, `consumed` stays 0, check `0 ≤ 100` passes; maker gains 999_999 units of credit, taker gains 999_999 units of debt, 0 tokens transferred
3. Taker repeats step 2 indefinitely — `consumed` never reaches `maxAssets`, the offer is filled without bound

**Why existing checks fail**:
- `require(newConsumed <= offer.maxAssets)` — passes every time because `newConsumed` is unchanged
- No `require(units == 0 || buyerAssets > 0)` guard exists; the codebase explicitly states "Zero checks are not systematically performed"
- The Certora `Consume.spec` rule `takeConsumedBoundedByMax` is verified with `mulDivDown` summarized as `NONDET`, so the prover never sees the rounding-to-zero case
- The EcrecoverRatifier uses a Merkle root that is not consumed per-call; since `consumed` never advances, the same proof is valid on every repeat call [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

The `maxAssets` cap on a buy offer is completely bypassed. A taker can fill the offer with an unbounded number of units while `consumed` remains at zero. The maker accumulates credit far beyond their intended exposure limit. The core invariant "offers cannot be overfilled" is violated. The taker self-inflicts debt (0 assets received per fill), making this a griefing vector: a malicious taker can force the maker into an arbitrarily large credit position, exposing them to bad-debt risk they did not consent to. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

**Preconditions**:
1. A buy offer exists with `maxAssets > 0` and any tick in `[2, MAX_TICK-1]` — this is the normal operating range; tick = MAX_TICK (price = 1e18) is the only tick immune
2. The taker has sufficient collateral to absorb the debt units
3. The EcrecoverRatifier root is not canceled

All three are routine conditions. The tick constraint covers essentially the entire usable price range. No privileged access is required. The attack is repeatable in a single transaction via `multicall`. [7](#0-6) [8](#0-7) 

### Recommendation

Add a guard in the `maxAssets` branch that rejects a non-zero `units` take that produces zero assets:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsIncrement = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsIncrement > 0, ZeroAssetsIncrement());
    newConsumed = consumed[offer.maker][offer.group] += assetsIncrement;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that every non-trivial take in assets mode advances `consumed` by at least 1, preserving the invariant that `consumed` strictly increases with each fill. [9](#0-8)

import {Test} from "forge-std/Test.sol";
import {Midnight, Offer} from "src/Midnight.sol";
import {TickLib} from "src/libraries/TickLib.sol";

contract ConsumedRoundingPoC is Test {
    Midnight midnight;
    address maker = address(0x1);
    address taker = address(0x2);

    function setUp() public {
        midnight = new Midnight();
        // ... standard market setup, collateral, ratifier authorization
    }

    function testConsumedNeverIncreasesOnZeroAssetsTake() public {
        // Offer: buy=true, maxAssets=100, tick=2 (price=1e12)
        Offer memory offer = _buildBuyOffer({maxAssets: 100, tick: 2});

        uint256 consumedBefore = midnight.consumed(maker, offer.group);

        // units=999_999: buyerAssets = (999_999 * 1e12) / 1e18 = 0
        vm.prank(taker);
        midnight.take(offer, ratifierData, 999_999, taker, address(0), address(0), "");

        uint256 consumedAfter = midnight.consumed(maker, offer.group);

        // ASSERTION: consumed must increase by at least 1 per non-trivial take
        // This assertion FAILS, proving the bug
        assertGt(consumedAfter, consumedBefore, "consumed did not increase");

        // Secondary assertion: repeat 200 times, maker credit >> maxAssets worth of units
        for (uint256 i = 0; i < 200; i++) {
            vm.prank(taker);
            midnight.take(offer, ratifierData, 999_999, taker, address(0), address(0), "");
        }
        // consumed is still 0, but maker has 200*999_999 units of credit
        assertEq(midnight.consumed(maker, offer.group), 0, "consumed stayed zero");
        assertGt(midnight.creditOf(id, maker), 100, "maker credit exceeds maxAssets");
    }
}
