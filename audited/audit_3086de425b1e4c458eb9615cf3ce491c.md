### Title
Lack of Signed Authorization Cancellation in `EcrecoverAuthorizer` Allows Forced Execution of Revoked Intent — (File: `src/periphery/EcrecoverAuthorizer.sol`)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` allows users to sign `Authorization` messages off-chain to gaslessly grant or revoke protocol-level authorization. However, there is no mechanism to cancel a signed-but-unexecuted `Authorization`. Once signed, any party holding the signature can execute it at any time before the `deadline`, even if the signer has changed their mind. This is in direct contrast to `EcrecoverRatifier`, which provides a `cancelRoot()` function for analogous off-chain signed offers.

### Finding Description

`EcrecoverAuthorizer` uses a sequential nonce to prevent replay: [1](#0-0) [2](#0-1) 

The nonce is auto-incremented only when `setIsAuthorized` is successfully called. There is no function to increment the nonce independently (i.e., no "cancel" or "invalidate" path). The only time-bound protection is the `deadline` field in the `Authorization` struct: [3](#0-2) 

If a user signs an `Authorization` with `isAuthorized: true` and a far-future deadline, then changes their mind, they have no on-chain recourse to invalidate that specific signed message. The submitter (or any third party who obtained the signature) can execute it at any time before the deadline.

By contrast, `EcrecoverRatifier` explicitly provides `cancelRoot()`: [4](#0-3) 

No equivalent exists in `EcrecoverAuthorizer`.

The test suite confirms the contract is designed for permissionless submission — anyone can call `setIsAuthorized` with a valid signature, not just the authorizer: [5](#0-4) 

### Impact Explanation

A user who signs an `Authorization` granting a third party the ability to act on their behalf in the Midnight protocol (e.g., execute trades, manage positions) cannot revoke that specific signed message before its deadline. If the user's intent changes — or if the signature is leaked — the third party (or any holder of the signature) can execute the authorization, resulting in an unauthorized change to the user's `isAuthorized` state in `Midnight.sol`. This can lead to unauthorized actors gaining protocol-level delegation over a victim's account.

**Impact: Medium** — unauthorized state change to authorization mapping; no direct asset theft but enables further exploitation via the newly granted authorization.

### Likelihood Explanation

**Likelihood: High** — The `EcrecoverAuthorizer` is explicitly designed for gasless, off-chain signing workflows where the signer and submitter are different parties. Users routinely sign messages in advance and distribute them. The scenario where a user wants to cancel a signed-but-unsubmitted authorization (key compromise, change of intent, front-running) is realistic and common in gasless authorization systems. No privileged access is required; any holder of the signature can trigger the issue.

### Recommendation

Add a `cancelNonce` (or `incrementNonce`) function to `EcrecoverAuthorizer` that allows the authorizer (or their authorized delegate) to advance their nonce, invalidating all previously signed `Authorization` messages with the old nonce:

```solidity
function cancelNonce(address authorizer) external {
    require(msg.sender == authorizer || IMidnight(MIDNIGHT).isAuthorized(authorizer, msg.sender), Unauthorized());
    nonce[authorizer]++;
    emit NonceCanceled(msg.sender, authorizer, nonce[authorizer] - 1);
}
```

This mirrors the `cancelRoot` pattern already present in `EcrecoverRatifier`. [4](#0-3) 

### Proof of Concept

1. Alice (authorizer) signs an `Authorization` granting Bob (`authorized`) `isAuthorized = true` with `nonce = 0` and `deadline = block.timestamp + 365 days`.
2. Alice distributes the signed message to Bob off-chain.
3. Alice changes her mind and wants to cancel the authorization before Bob submits it.
4. Alice has no on-chain function to increment her nonce or cancel the signed message.
5. Bob (or any third party who obtained the signature) calls `ecrecoverAuthorizer.setIsAuthorized(auth, sig)`.
6. `nonce[Alice]` advances to 1, and `IMidnight.setIsAuthorized(Bob, true, Alice)` is called — granting Bob full delegation over Alice's account against Alice's current intent. [6](#0-5)
