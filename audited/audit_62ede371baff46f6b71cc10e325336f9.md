Audit Report

## Title
Missing zero-amount guard before `safeTransfer` in `liquidate()` blocks bad-debt-only liquidations for tokens that revert on zero transfers - (`File: src/Midnight.sol`)

## Summary
`liquidate()` unconditionally calls `SafeTransferLib.safeTransfer(collateralToken, receiver, seizedAssets)` at line 696 even when `seizedAssets = 0`, directly contradicting the NatSpec at line 577 which guarantees "Passing both 0 for seizedAssets and repaidUnits allows to realize bad debt with 0 token transferred." For any market whose collateral token reverts on zero-amount transfers, the bad-debt-only liquidation path is permanently DoS'd, preventing `lossFactor` from ever being updated and leaving bad debt unrealized.

## Finding Description

The NatSpec at line 577 explicitly documents the zero-zero path as valid: [1](#0-0) 

The guard at line 595 permits both inputs being zero — `atMostOneNonZero` returns `or(iszero(x), iszero(y))`, which is `true` when both are zero: [2](#0-1) [3](#0-2) 

The block at line 643 is skipped entirely when both inputs are zero, leaving `seizedAssets = 0`: [4](#0-3) 

Execution then falls through unconditionally to the `safeTransfer` call: [5](#0-4) 

`SafeTransferLib.safeTransfer` has no zero-amount guard — it always issues the low-level `token.call`: [6](#0-5) 

If the collateral token reverts on a zero-amount `transfer()`, the entire transaction reverts, rolling back the bad-debt accounting at lines 626–641 (the `_position.debt` reduction and `_marketState.lossFactor` update): [7](#0-6) 

The Certora rule `liquidateLossFactorDoesNotRevert` at line 104 of `LossFactor.spec` does not catch this because `SafeTransferLib.safeTransfer` is summarized as `NONDET` at line 21 of the same spec, abstracting away the external ERC20 call entirely: [8](#0-7) [9](#0-8) 

The same structural issue exists at line 717 for `safeTransferFrom(loanToken, payer, address(this), repaidUnits)` when `repaidUnits = 0`: [10](#0-9) 

## Impact Explanation

Bad-debt-only liquidations are permanently blocked for any market whose collateral token reverts on zero-amount transfers. The `_marketState.lossFactor` is never updated, lenders cannot recover proportional losses through the loss socialization mechanism, and the borrower's debt is never reduced. This violates the protocol's own documented guarantee and the core accounting invariant. The impact is a permanent freeze of the bad-debt realization path — a concrete, non-hypothetical loss of protocol functionality.

## Likelihood Explanation

The market creator role is unprivileged. A market can be created with any ERC20 as collateral, including tokens that revert on zero-amount transfers (a known real-world pattern present in several deployed tokens). Once such a market exists and any borrower's position reaches `badDebt > 0`, every liquidation attempt on the bad-debt-only path reverts. The DoS is repeatable, permanent (the collateral token cannot be changed post-deployment), and requires no special timing, oracle manipulation, or victim mistakes.

## Recommendation

Add a zero-amount guard before the unconditional `safeTransfer` call at line 696:

```solidity
if (seizedAssets > 0) {
    SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
}
```

Apply the same guard at line 717:

```solidity
if (repaidUnits > 0) {
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
}
```

This aligns the implementation with the NatSpec guarantee and ensures the bad-debt-only path (`seizedAssets = 0, repaidUnits = 0`) completes without any token interaction.

## Proof of Concept

1. Deploy a collateral token that reverts on zero-amount `transfer()` calls.
2. Create a Midnight market using this token as collateral.
3. Open a borrower position and allow it to reach a state where `badDebt > 0` (all collateral value is below the `maxLif`-adjusted threshold, e.g., collateral price drops to zero or collateral bitmap is empty).
4. Call `liquidate(market, collateralIndex, 0, 0, borrower, false, receiver, address(0), "")`.
5. Observe the transaction reverts at the `safeTransfer` call at line 696 because the collateral token reverts on `transfer(receiver, 0)`.
6. Confirm that `_marketState.lossFactor` remains unchanged and `_position.debt` is not reduced.

The Certora rule `liquidateLossFactorDoesNotRevert` in `LossFactor.spec` does not catch this because `SafeTransferLib.safeTransfer` is summarized as `NONDET` (line 21), making the prover assume the call always succeeds.
