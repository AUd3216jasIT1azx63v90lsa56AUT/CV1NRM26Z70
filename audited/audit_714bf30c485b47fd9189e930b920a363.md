### Title
Zero-price sell offer in assets mode allows unlimited reuse — consumed never increments - (File: src/Midnight.sol)

### Summary

When a sell offer (`offer.buy == false`) is created with `tick ∈ {0, 1}` (both yield `tickToPrice == 0`) and `offer.maxAssets > 0`, the assets-mode consumed accounting adds `sellerAssets = mulDivUp(units * 0, WAD) = 0` to `consumed` on every call to `take`. Because `consumed` never increments, the `require(newConsumed <= offer.maxAssets)` guard is permanently satisfied and the offer can be taken an unlimited number of times, bypassing the `maxAssets` cap entirely.

### Finding Description

**Exact code path — `src/Midnight.sol`:**

```
sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;   // line 361
                                                                        // sell offer → sellerPrice = offerPrice
sellerAssets = units.mulDivUp(sellerPrice, WAD);                       // line 364
                                                                        // mulDivUp(1 * 0, WAD) = (0 + WAD-1)/WAD = 0

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += sellerAssets;  // line 368 → += 0
    require(newConsumed <= offer.maxAssets, ConsumedAssets());          // line 369 → 0 <= maxAssets, always passes
}
```

**Root cause:** `UtilsLib.mulDivUp` is defined as `(x * y + (d - 1)) / d`. With `x=1, y=0, d=WAD`: `(0 + WAD-1)/WAD = 0` by integer division. [1](#0-0) 

**`tickToPrice(0) == 0` is a protocol invariant**, confirmed by the existing test suite: [2](#0-1) 

Ticks 0 and 1 both produce price 0 (tick 2 is the first non-zero price at `1e12`). Both pass the tick-accessibility check `offer.tick % tickSpacing == 0` since `0 % n == 0` for any `n`. [3](#0-2) 

**Attacker-controlled inputs:**
- `offer.tick = 0` (or 1), `offer.buy = false`, `offer.maxAssets > 0` (maker-set, but taker exploits)
- `units = 1` per call (any positive value works)

**Exploit flow:**
1. Maker (borrower) creates a sell offer with `tick=0`, `maxAssets=N` (any positive value)
2. Taker calls `take(offer, ..., units=1, ...)` repeatedly
3. Each call: `sellerAssets = 0`, `consumed += 0`, check `0 <= N` passes
4. Each call: `sellerPos.debt += 1`, `buyerPos.credit += 1`
5. After K calls: `consumed[maker][group] = 0`, `debtOf(maker) = K`

**Existing protections reviewed and insufficient:**
- `require(newConsumed <= offer.maxAssets)` — always passes since `newConsumed = 0` [4](#0-3) 
- Post-take health check `require(... || isHealthy(...), SellerIsLiquidatable())` — limits per-take debt increase to the seller's current health headroom, but does **not** fix the consumed accounting; the taker can take up to the health limit, and as collateral value fluctuates the window reopens [5](#0-4) 
- No check that `sellerAssets > 0` before updating consumed
- No check that `tick > 0` or `offerPrice > 0` for assets-mode offers

The existing zero-price tests all use `maxUnits` mode, not `maxAssets` mode, so this path is untested: [6](#0-5) 

### Impact Explanation

The `maxAssets` cap — the sole mechanism preventing an assets-mode offer from being overfilled — is permanently bypassed. The offer is reused indefinitely: `consumed` stays at zero while the maker's debt grows without bound (bounded only by the per-call health check). The maker's `maxAssets` intent is completely unenforceable. As the maker's debt accumulates and collateral value fluctuates, the position becomes liquidatable and any shortfall becomes bad debt socialized across all lenders in the market.

### Likelihood Explanation

**Preconditions:**
1. A sell offer exists with `tick ∈ {0, 1}` and `offer.maxAssets > 0` — tick 0 is always accessible (`0 % tickSpacing == 0`), and a maker may set it as a near-zero-rate offer
2. Taker is any unprivileged address (not the maker itself due to `SelfTake` check)

**Feasibility:** Fully reachable with no special privileges. The taker pays at most `mulDivUp(1, _settlementFee, WAD)` per call (1 wei if settlement fee > 0, 0 if fee = 0). Repeatable in a loop or across blocks. The Certora `Consume.spec` rules that verify consumed accounting use `NONDET` summaries for `mulDivUp` and `tickToPrice`, meaning this concrete zero-price path is not covered by the formal verification. [7](#0-6) 

### Recommendation

Add a guard in the assets-mode branch of `take` that requires the computed asset amount to be nonzero before updating consumed, or require `sellerAssets > 0` (and symmetrically `buyerAssets > 0` for buy offers) when `offer.maxAssets > 0`:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0, ZeroAssetsDelta()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, disallow `offer.maxAssets > 0` when `tickToPrice(offer.tick) == 0` at offer-validation time, or enforce a minimum tick for assets-mode offers.

### Proof of Concept

```solidity
function testConsumedZeroSellOfferAssetsMode() public {
    // Setup: sell offer at tick=0 (price=0), assets mode
    Offer memory offer = /* borrower's sell offer */;
    offer.tick = 0;
    offer.buy = false;
    offer.maxAssets = 1000e18;  // maker intends to cap at 1000 assets
    offer.maxUnits = 0;

    // Provide collateral so health check passes per-take
    collateralize(market, borrower, 100);

    uint256 consumedBefore = midnight.consumed(borrower, offer.group);

    // Taker takes 1 unit
    vm.prank(taker);
    midnight.take(offer, hex"", 1, taker, address(0), address(0), hex"");

    // ASSERTION: consumed must have incremented — but it hasn't
    assertGt(midnight.consumed(borrower, offer.group), consumedBefore, "consumed must increment");
    // ^ This assertion FAILS: consumed stays at 0

    // Taker takes again — offer is reused
    vm.prank(taker);
    midnight.take(offer, hex"", 1, taker, address(0), address(0), hex"");

    // Debt has grown but consumed is still 0
    assertEq(midnight.consumed(borrower, offer.group), 0, "consumed is still zero");
    assertEq(midnight.debtOf(id, borrower), 2, "debt grew without consumed tracking");
}
```

Expected: `consumed > 0` after first take. Actual: `consumed == 0`, offer reusable indefinitely.
