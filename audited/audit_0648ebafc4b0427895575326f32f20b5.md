### Title
maxAssets Cap Bypass via Zero-Rounding buyerAssets on Buy Offers with buyerPrice < WAD - (File: src/Midnight.sol)

### Summary
When a buy offer has `maxAssets > 0` and `buyerPrice < WAD`, a taker can call `take` with `units=1` (or any small `units` where `units * buyerPrice < WAD`) and `mulDivDown` rounds `buyerAssets` to zero. Because `consumed` is incremented by `buyerAssets`, the cap check `newConsumed <= maxAssets` passes even after the offer is fully consumed, allowing unlimited additional fills that mutate position state (credit/debt/totalUnits) without any token transfer. The codebase itself documents this as a bug in `testBugBuyMaxAssetsBypass`.

### Finding Description
**Code path** — `src/Midnight.sol` lines 363–369:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

`mulDivDown(units, buyerPrice, WAD)` is `(units * buyerPrice) / WAD`. When `units * buyerPrice < WAD` (e.g., `units=1` and `buyerPrice < WAD`), the result is `0`. The consumed accumulator is then incremented by `0`, so `newConsumed` equals the pre-call value of `consumed`, and the cap check trivially passes regardless of how many times this is repeated.

**Attacker inputs** — Any unprivileged taker can:
1. Fill the offer normally until `consumed = maxAssets - 1` (or wait for it to reach `maxAssets` via `setConsumed`).
2. Repeatedly call `take(offer, ratifierData, 1, taker, ...)` with `units=1`.

**State changes per zero-asset fill** (confirmed by `testBugBuyMaxAssetsBypass`, lines 876–888):
- `buyerAssets = 0`, `sellerAssets = 0` → no loan token transfer occurs.
- `consumed` is unchanged (cap check passes).
- `buyerPos.credit += 1` (maker/lender gains 1 credit unit for free).
- `sellerPos.debt += 1` (taker/borrower gains 1 debt unit, receiving 0 tokens).
- `totalUnits += 1`.

**Why existing checks fail** — The only guard is `require(newConsumed <= offer.maxAssets)`. Since `buyerAssets = 0`, `newConsumed` never exceeds `maxAssets`, so the check never triggers. There is no guard of the form `require(units == 0 || buyerAssets > 0)`.

**Ratifier** — `SetterRatifier.isRatified` and `EcrecoverRatifier.isRatified` both validate the offer struct (Merkle proof + root), not the fill amount. They pass on every call as long as the offer is valid, so they provide no protection against repeated zero-asset fills.

### Impact Explanation
The maker's `maxAssets` cap — intended to bound total loan-token exposure — is bypassed. The maker accumulates unbounded credit units without paying the corresponding loan tokens, violating the invariant that every credit unit has matching debt backed by a real token transfer. The taker accumulates matching debt units without receiving tokens, which can be used to grief a specific maker by inflating their credit position beyond their intended limit, potentially affecting downstream health checks, fee accrual (`buyerPendingFeeIncrease` is also computed per fill), and settlement.

### Likelihood Explanation
**Preconditions:**
- `offer.buy = true` (buy offer, maker is lender).
- `offer.maxAssets > 0`.
- `buyerPrice < WAD` — satisfied for any tick below `MAX_TICK` once the settlement fee is subtracted; `tickToPrice` is proven ≤ WAD for all valid ticks.
- `units` small enough that `units * buyerPrice < WAD` — `units=1` always works when `buyerPrice < WAD`.

These conditions are routine in normal protocol usage. The attack is permissionless, requires no special role, and is repeatable indefinitely within a single transaction via `multicall`. The existing test `testBugBuyMaxAssetsBypass` (lines 857–889) reproduces it exactly with `maxAssets=1` and `tick = MAX_TICK - 16`.

### Recommendation
Add a guard immediately after computing `buyerAssets` for buy offers in assets mode:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
}
```

Alternatively, track consumed in units (not assets) and convert only for the cap comparison, or require `buyerPrice <= WAD` to be enforced such that `buyerAssetsToUnits` is always the inverse (already done in `TakeAmountsLib` but not enforced in `take` itself).

function testFuzz_maxAssetsBypassInvariant(uint256 units1, uint8 extraFills) public {
    // Setup: buy offer, buyerPrice < WAD, maxAssets = N
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 100;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    collateralize(market, borrower, type(uint128).max);

    // Fill 1: consume up to maxAssets - 1
    // (find units1 such that buyerAssets = maxAssets - 1)
    // Fill 2..N: take(units=1) repeatedly
    for (uint256 i = 0; i < extraFills; i++) {

    assertLe(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
    // Invariant: lender credit must not exceed maxAssets units

}

    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);
