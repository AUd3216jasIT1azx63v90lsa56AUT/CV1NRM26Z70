### Title
`consumableUnits` ignores `reduceOnly` + maker debt, causing bundler to skip always-reverting buy offers and revert with `OutOfOffers()` - (File: src/periphery/ConsumableUnitsLib.sol)

### Summary
`ConsumableUnitsLib.consumableUnits` computes fillable units purely from the `consumed` counter vs `maxUnits`/`maxAssets`, with no awareness of `offer.reduceOnly` or the maker's live debt. For a `reduceOnly` buy offer whose maker has zero debt, `consumableUnits` returns a positive value, but every call to `Midnight.take` with those units reverts with `MakerCreditOrDebtIncreased()`. The bundler's `catch {}` silently discards the revert without crediting `filledUnits`, so if the taker's `takes` array relies on that offer to reach `targetUnits`, the outer `require(filledUnits == targetUnits, OutOfOffers())` fires and the entire bundler call reverts.

### Finding Description

**Root cause — `consumableUnits` (lines 14–23, `src/periphery/ConsumableUnitsLib.sol`):**

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    if (offer.maxUnits > 0) {
        return offer.maxUnits.zeroFloorSub(consumed);   // ← no reduceOnly / debt check
    } ...
}
``` [1](#0-0) 

The function never reads `offer.reduceOnly` or the maker's `position[id][maker].debt`.

**`take()` enforcement (lines 382–395, `src/Midnight.sol`):**

For `offer.buy = true`, `buyer = offer.maker`. After the optional `_updatePosition`:

```solidity
uint256 buyerCreditIncrease = UtilsLib.zeroFloorSub(units, buyerPos.debt);
...
require(
    !offer.reduceOnly || (offer.buy ? buyerCreditIncrease == 0 : sellerDebtIncrease == 0),
    MakerCreditOrDebtIncreased()
);
``` [2](#0-1) 

If `buyerPos.debt == 0` and `units > 0`, then `buyerCreditIncrease = units > 0`, and the `reduceOnly` guard reverts unconditionally.

**Bundler loop (lines 147–163, `src/periphery/MidnightBundles.sol`):**

```solidity
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // returns > 0
);
try IMidnight(MIDNIGHT).take(..., unitsToTake, ...) returns (...) {
    filledUnits += unitsToTake;
    ...
} catch {}   // revert swallowed; filledUnits unchanged
``` [3](#0-2) 

`consumableUnits` returns, say, 100. `unitsToTake = 100`. `take` reverts with `MakerCreditOrDebtIncreased`. `catch {}` swallows it. `filledUnits` stays at 0. After the loop: `require(filledUnits == targetUnits, OutOfOffers())` reverts.

**Exploit flow:**

1. Maker (attacker or asynchrony victim) holds a `reduceOnly` buy offer with `maxUnits = N`, `consumed = 0`, and `buyerPos.debt = 0`.
2. Taker calls `supplyCollateralAndSellWithUnitsTarget(targetUnits = N, ..., takes = [that offer])`.
3. `consumableUnits` returns `N`; `unitsToTake = N`.
4. `take(N)` reverts: `buyerCreditIncrease = N > 0` violates `reduceOnly`.
5. `catch {}` discards the revert; `filledUnits = 0`.
6. Loop ends; `require(0 == N, OutOfOffers())` reverts the entire bundler call.

The same path applies to `buyWithAssetsTargetAndWithdrawCollateral` and `supplyCollateralAndSellWithAssetsTarget`, which all use `ConsumableUnitsLib.consumableUnits` and the same `try/catch` pattern. [4](#0-3) [5](#0-4) 

### Impact Explanation

A taker's bundler call reverts with `OutOfOffers()` whenever the `takes` array contains a `reduceOnly` buy offer whose maker currently has zero (or insufficient) debt. The taker's collateral supply is already executed before the loop, so the taker wastes gas and the transaction is a no-op. No funds are lost from the protocol, but the taker's fill is completely blocked. A malicious maker can deliberately maintain zero debt while keeping a `reduceOnly` buy offer live to reliably grief any taker who routes through that offer.

### Likelihood Explanation

**Preconditions:**
- A `reduceOnly` buy offer exists with `consumed < maxUnits` (appears fillable).
- The maker's debt is 0 at execution time (either always, or after repaying between tx construction and mining).

**Feasibility:** Both conditions are reachable without any privileged action. The maker can sign a `reduceOnly` buy offer and then repay all debt, or simply never have had debt. The taker has no on-chain way to atomically verify the maker's debt before the bundler loop runs. The asynchrony window (debt repaid between off-chain simulation and on-chain execution) makes this repeatable and non-exotic.

### Recommendation

In `consumableUnits`, when `offer.reduceOnly && offer.buy`, cap the returned units by the maker's current debt:

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    uint256 units;
    if (offer.maxUnits > 0) {
        units = offer.maxUnits.zeroFloorSub(consumed);
    } else if (offer.buy) {
        units = TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
    } else {
        units = TakeAmountsLib.sellerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
    }
    // Cap by maker's debt for reduceOnly buy offers; cap by maker's credit for reduceOnly sell offers.
    if (offer.reduceOnly) {
        (uint128 credit,,,, uint128 debt,) = IMidnight(midnight).position(id, offer.maker);
        units = offer.buy ? units.min(debt) : units.min(credit);
    }
    return units;
}
```

This makes `consumableUnits` return 0 when the maker has no debt (buy) or no credit (sell), so the bundler calls `take(0)` — a no-op that succeeds — rather than a positive-unit take that always reverts.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {MidnightBundles} from "src/periphery/MidnightBundles.sol";
import {IMidnightBundles, Take, CollateralSupply} from "src/periphery/interfaces/IMidnightBundles.sol";
import {IMidnight, Offer, Market} from "src/interfaces/IMidnight.sol";

contract ReduceOnlyBuyBundlerDoSTest is Test {
    // Setup: deploy Midnight + MidnightBundles, create market, fund users.
    // maker  = lender who has NEVER borrowed (debt == 0).
    // taker  = borrower who wants to sell (borrow) via the bundler.

    function testReduceOnlyBuyNoDebtCausesOutOfOffers() public {
        // 1. maker signs a reduceOnly buy offer (maxUnits = 100, reduceOnly = true, buy = true).
        //    maker has debt == 0 at all times.
        Offer memory offer = _buildReduceOnlyBuyOffer(maker, 100);

        // 2. taker calls supplyCollateralAndSellWithUnitsTarget targeting 100 units.
        Take[] memory takes = new Take[](1);
        takes[0] = Take({offer: offer, units: 100, ratifierData: hex""});

        vm.prank(taker);
        // ASSERTION: bundler reverts with OutOfOffers even though consumableUnits returned 100.
        vm.expectRevert(IMidnightBundles.OutOfOffers.selector);
        midnightBundles.supplyCollateralAndSellWithUnitsTarget(
            100, 0, taker, taker, new CollateralSupply[](0), takes, 0, address(0)
        );

        // SECONDARY ASSERTION: consumed counter is still 0 (no state was changed).
        assertEq(midnight.consumed(maker, offer.group), 0, "consumed must be 0");
        // SECONDARY ASSERTION: taker has no debt (transaction was fully reverted).
        assertEq(midnight.debtOf(id, taker), 0, "taker debt must be 0");
    }
}
```

**Expected assertions:**
- `OutOfOffers()` is thrown despite `consumableUnits` returning 100.
- `consumed[maker][group]` remains 0 (the `take` revert rolled back the consumed increment).
- Taker's debt remains 0.
