### Title
`sellerAssetsToUnits` Division-by-Zero Revert at Settlement Fee Boundary Causes Unrecoverable DoS of `supplyCollateralAndSellWithAssetsTarget` - (File: src/periphery/TakeAmountsLib.sol)

### Summary

When a buy offer's tick price exactly equals the current settlement fee, `sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee = 0` and then calls `mulDivUp(targetSellerAssets, WAD, 0)`. Because `mulDivUp` is not in an `unchecked` block, the expression `d - 1` with `d = 0` triggers an arithmetic underflow (Panic(0x11)) before any division occurs. This revert propagates directly through `supplyCollateralAndSellWithAssetsTarget` because the `sellerAssetsToUnits` call sits outside the `try/catch` that only wraps `IMidnight.take()`. The underlying `Midnight.take()` would succeed at this boundary (yielding `sellerAssets = 0`), so the bundler is strictly more restrictive than the core protocol.

### Finding Description

**Code path:**

`supplyCollateralAndSellWithAssetsTarget` (MidnightBundles.sol:285-291) calls `TakeAmountsLib.sellerAssetsToUnits` unconditionally outside the try/catch:

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(          // ← NOT in try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
try IMidnight(MIDNIGHT).take(...) ... catch {}   // ← only take() is guarded
```

Inside `sellerAssetsToUnits` (TakeAmountsLib.sol:41-46):

```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
uint256 settlementFee = IMidnight(midnight).settlementFee(id, ...);
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...;
```

When `offerPrice == settlementFee`: `sellerPrice = 0`, then `mulDivUp(targetSellerAssets, WAD, 0)` is called.

`mulDivUp` (UtilsLib.sol:34-36) is:

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```

With `d = 0` and no `unchecked` block: `d - 1 = 0 - 1` underflows → **Panic(0x11)**. The loop condition `filledSellerAssets < targetFilledSellerAssets` guarantees `targetSellerAssets > 0`, so the underflow is unconditional.

**Why existing checks fail:**

- `Midnight.take()` at this boundary computes `sellerAssets = units.mulDivDown(0, WAD) = 0` and succeeds — no revert in the core.
- The NatSpec of `sellerAssetsToUnits` documents revert only for `offerPrice < settlementFee`, not `offerPrice == settlementFee`.
- The try/catch in `supplyCollateralAndSellWithAssetsTarget` only guards `IMidnight.take()`, not the `TakeAmountsLib` call.
- `ConsumableUnitsLib.consumableUnits` for a buy offer with `maxUnits > 0` returns without calling `sellerAssetsToUnits`, so it does not independently guard against this.

**Attacker inputs:** An unprivileged taker observes the current `settlementFee(id, ttm)` (public state), finds tick `t` such that `TickLib.tickToPrice(t) == settlementFee`, and calls `supplyCollateralAndSellWithAssetsTarget` with `takes[0].offer.buy = true` and `takes[0].offer.tick = t`. Both prices and fees are multiples of `PRICE_ROUNDING_STEP = CBP = 1e12`, making exact alignment achievable across the 5821-tick range against any of the ~5000 valid fee values.

### Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer whose tick price equals the current settlement fee reverts unconditionally with Panic(0x11). Because the settlement fee can change (fee setter is privileged but the resulting state is public), a previously valid offer can become boundary-priced after a fee update, permanently breaking the bundler path for that offer. The core `Midnight.take()` remains functional; only the assets-target bundler variant is affected.

### Likelihood Explanation

Preconditions: (1) a buy offer exists at a tick where `tickToPrice(tick) == settlementFee(id, ttm)`, and (2) a taker attempts to fill it via `supplyCollateralAndSellWithAssetsTarget`. Both prices and fees are multiples of `1e12`, so exact alignment is structurally possible. A fee setter update can create this condition for existing offers without any action by the taker. The condition is repeatable: every call with such an offer reverts.

### Recommendation

Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case, consistent with the documented behavior for `offerPrice < settlementFee`:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
// offerPrice == settlementFee: sellerPrice = 0, no finite units yield positive sellerAssets.
require(!offer.buy || sellerPrice > 0, /* e.g. SellerPriceZero() */);
```

Alternatively, align the NatSpec to document the `offerPrice == settlementFee` revert case, and update callers to handle it (e.g., skip the offer or use `supplyCollateralAndSellWithUnitsTarget`).

```solidity
    uint256 boundaryTick = /* tick where tickToPrice == fee */ 1;
    require(TickLib.tickToPrice(boundaryTick) == fee, "tick not at boundary");

    // 3. Construct a buy offer at the boundary tick.
    Offer memory buyOffer = /* standard buy offer */ ...;
    buyOffer.buy = true;
    buyOffer.tick = boundaryTick;
    buyOffer.maxUnits = type(uint256).max;

        midnight.take(buyOffer, hex"", 1, borrower, borrower, address(0), hex"");

```
