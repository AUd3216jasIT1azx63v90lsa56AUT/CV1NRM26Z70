### Title
Buy-offer `maxAssets` cap bypassed via rounding-to-zero `buyerAssets` on small-unit takes - (`src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and `buyerPrice < WAD` (tick > `MAX_TICK/2`), `buyerAssets` is computed with `mulDivDown`, which rounds to zero for sufficiently small `units`. Because `consumed` is incremented by `buyerAssets`, a zero-asset take leaves `consumed` unchanged, passes the `ConsumedAssets` check, yet still creates real credit/debt and increases `totalUnits`. This can be repeated without bound, even after the offer is fully consumed in asset terms. The protocol explicitly acknowledges this in its NatSpec and has a test named `testBugBuyMaxAssetsBypass` that confirms the behavior.

### Finding Description
**Code path:**

`src/Midnight.sol` lines 363–369:

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause:** For a buy offer, `consumed` is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. When `buyerPrice < WAD` (i.e., `tick > MAX_TICK/2`, so `tickToPrice(tick) < 1e18`) and `units < WAD / buyerPrice`, the product `units * buyerPrice` is less than `WAD`, so `mulDivDown` returns 0. The cap check `newConsumed <= offer.maxAssets` then passes trivially because `consumed` did not increase.

**Attacker inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick > MAX_TICK/2` (so `buyerPrice < WAD`)
- `units = 1` (or any value satisfying `units * buyerPrice < WAD`)

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = N` and `tick > MAX_TICK/2`.
2. Taker (or anyone) calls `take()` with `units = 1` repeatedly.
3. Each call: `buyerAssets = 1 * buyerPrice / WAD = 0` (rounds down). `consumed` stays unchanged. Check passes. Lender credit, borrower debt, and `totalUnits` all increase by 1 unit per call.
4. After `N` such calls, the offer is "fully consumed" in asset terms (`consumed == maxAssets`), but the taker can continue indefinitely because each subsequent call still produces `buyerAssets = 0`.

**Why existing checks fail:** The only guard is `require(newConsumed <= offer.maxAssets)`. Since `newConsumed` never increases when `buyerAssets = 0`, this check is permanently satisfied regardless of how many unit-level fills have occurred.

The protocol's own NatSpec at line 94 states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* The test `testBugBuyMaxAssetsBypass` (lines 857–889 of `test/TakeTest.sol`) confirms: after pre-consuming the offer to `maxAssets`, a `take(1, ...)` succeeds, `consumed` stays at `maxAssets`, but `creditOf`, `debtOf`, and `totalUnits` all strictly increase.

### Impact Explanation
The maker's `maxAssets` cap is intended to bound total economic exposure in asset terms. Due to rounding, an attacker can fill an unbounded number of units beyond what `maxAssets / buyerPrice` would imply, creating unbounded credit and debt on behalf of the maker. The maker's offer is overfilled in unit terms while the asset-denominated `consumed` counter never exceeds `maxAssets`, so no revert ever occurs. This violates the core invariant: *"offers cannot be replayed, overfilled, reused, or filled after cancel/deadline."*

### Likelihood Explanation
Preconditions are easily satisfied: any buy offer at a tick above `MAX_TICK/2` (i.e., `buyerPrice < WAD`) with `maxAssets > 0` is vulnerable. The attacker needs no special privilege — any address can call `take()` as the taker. The attack is repeatable in a single transaction via a callback or across multiple transactions. Ticks above `MAX_TICK/2` are common for lenders offering below-par rates.

### Recommendation
Add a check that reverts when `units > 0` but `buyerAssets == 0` for buy offers in `maxAssets` mode:

```solidity
if (offer.maxAssets > 0) {
    // Prevent unit-level overfill when rounding produces zero assets
    require(units == 0 || buyerAssets > 0, ZeroAssetTake());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, when `buyerAssets == 0` and `units > 0`, fall back to unit-based consumed tracking to ensure the cap is enforced in unit terms as well.

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` already serves as a unit-test PoC. A fuzz/invariant extension:

```solidity
function testFuzz_BuyMaxAssetsBypassRepeated(uint256 numTakes) public {
    numTakes = bound(numTakes, 1, 1000);
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1;
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    collateralize(market, borrower, numTakes * 2);
    deal(address(loanToken), borrower, numTakes * 2);

    uint256 totalUnitsFilled;
    for (uint256 i; i < numTakes; i++) {
        take(1, borrower, lenderOffer);
        totalUnitsFilled += 1;
    }

    // consumed never exceeded maxAssets (1), yet units filled is unbounded
    assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
    // This assertion FAILS — units filled >> maxAssets / buyerPrice
    assertLe(totalUnitsFilled, lenderOffer.maxAssets * WAD / TickLib.tickToPrice(lenderOffer.tick));
}
```

        // Fully consume the offer before the take.
