### Title
Unconditional `safeTransfer(collateralToken, receiver, 0)` in `liquidate()` blocks bad-debt-only liquidations on zero-revert collateral tokens - (`src/Midnight.sol`)

### Summary
`liquidate()` calls `SafeTransferLib.safeTransfer` unconditionally at line 696 with `seizedAssets` as the value, even when both `seizedAssets` and `repaidUnits` are 0. The NatSpec at line 577 explicitly documents that passing both as 0 "allows to realize bad debt with 0 token transferred," but the implementation contradicts this: the transfer is always issued. If the collateral token reverts on zero-amount transfers, the entire bad-debt liquidation reverts, permanently blocking bad-debt realization for that market.

### Finding Description

**Code path:**

In `liquidate()`, the guard at line 643 correctly skips collateral/debt accounting when both inputs are zero:

```solidity
if (repaidUnits > 0 || seizedAssets > 0) {
    // ... compute seizedAssets / repaidUnits, update position ...
}
``` [1](#0-0) 

After this block, `seizedAssets` is still 0. Then, unconditionally:

```solidity
SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransfer` makes a raw low-level call to `token.transfer(receiver, 0)` and propagates any revert from the token:

```solidity
(bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
if (!success) {
    assembly ("memory-safe") { revert(add(returndata, 0x20), mload(returndata)) }
}
``` [3](#0-2) 

If the collateral token reverts on `transfer(receiver, 0)`, the entire `liquidate()` call reverts — including the bad-debt accounting already performed in the `if (badDebt > 0)` block at lines 626–641. [4](#0-3) 

The same issue exists for `SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits)` at line 717 when `repaidUnits = 0`. [5](#0-4) 

**Attacker-controlled inputs:** `seizedAssets = 0`, `repaidUnits = 0`, any valid `borrower` with bad debt, any `collateralIndex`.

**Why existing checks don't stop it:**
- `atMostOneNonZero(repaidUnits, seizedAssets)` at line 595 explicitly allows both to be 0 — this is the intended bad-debt path.
- `require(_position.debt > 0, NotBorrower())` at line 596 is satisfied by any borrower with debt.
- There is no guard of the form `if (seizedAssets > 0)` before the `safeTransfer` call. [6](#0-5) 

The NatSpec comment directly contradicts the implementation:

> "Passing both 0 for seizedAssets and repaidUnits allows to realize bad debt with 0 token transferred." [7](#0-6) 

### Impact Explanation

Bad-debt-only liquidations (`seizedAssets = 0`, `repaidUnits = 0`) are permanently blocked for any market whose collateral token reverts on zero-amount `transfer` calls. The `lossFactor` and `totalUnits` accounting that socializes bad debt among lenders cannot be executed. Unhealthy positions with full bad debt remain stuck, and lenders cannot recover their proportional share of the loss. This is a concrete, scoped DoS on a documented core invariant of the protocol.

### Likelihood Explanation

**Preconditions:**
1. A market is created with a collateral token that reverts on `transfer(to, 0)`. This is a known behavior in several deployed ERC20 tokens (e.g., LEND, BNB, and various custom tokens). Market creation is permissionless in Midnight, so any user can create such a market.
2. A borrower's position reaches a state where `badDebt > 0` (collateral value falls below debt at `maxLif` pricing).
3. Any unprivileged liquidator calls `liquidate(market, idx, 0, 0, borrower, ...)`.

The attack is repeatable: every call to realize bad debt on such a market will revert. No special privilege is required. The attacker does not need to deploy the collateral token — they only need to create a market using an existing zero-revert token.

### Recommendation

Guard both transfer calls with a non-zero check:

```solidity
// Line 696
if (seizedAssets > 0) {
    SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
}

// Line 717
if (repaidUnits > 0) {
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
}
```

This aligns the implementation with the documented invariant that passing both inputs as 0 realizes bad debt with zero tokens transferred.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../src/Midnight.sol";

// Collateral token that reverts on zero-amount transfers
contract ZeroRevertCollateral is ERC20 {
    function transfer(address to, uint256 amount) public override returns (bool) {
        require(amount > 0, "zero transfer");
        return super.transfer(to, amount);
    }
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        require(amount > 0, "zero transfer");
        return super.transferFrom(from, to, amount);
    }
}

contract BadDebtLiquidationDoSTest is Test {
    Midnight midnight;
    ZeroRevertCollateral collateral;
    // ... standard market setup ...

    function testBadDebtLiquidationReverts() public {
        // 1. Deploy ZeroRevertCollateral, create market with it
        // 2. Borrower supplies collateral, takes debt (via take())
        // 3. Oracle price drops so badDebt > 0 (collateral value < debt at maxLif)
        // 4. Liquidator calls liquidate(market, 0, 0, 0, borrower, false/true, receiver, address(0), "")
        // 5. Assert: call reverts with "zero transfer" from collateral token
        // 6. Assert: lossFactor and totalUnits are UNCHANGED (bad debt not realized)
        
        vm.expectRevert("zero transfer");
        midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");
        
        // Confirm bad debt state is stuck
        assertEq(midnight.debtOf(id, borrower), originalDebt, "debt unchanged: bad debt not realized");
    }
}
```

**Expected assertions:**
- The `liquidate()` call reverts with the token's zero-transfer error.
- `marketState[id].lossFactor` and `totalUnits` are unchanged after the revert.
- The borrower's debt remains at its pre-call value.
- Repeating the call always reverts — the DoS is permanent for this market.
