### Title
Unbounded oracle gas consumption in `liquidate()` collateral bitmap loop enables liquidation DoS - (File: src/Midnight.sol)

### Summary
The `liquidate()` function iterates over all activated collateral slots (up to `MAX_COLLATERALS_PER_BORROWER = 16`) and calls `IOracle(_collateralParam.oracle).price()` for each with no gas stipend or limit. Because `touchMarket()` is permissionless and performs no validation on oracle addresses, a borrower can create a market with 16 attacker-controlled oracles each consuming near-block-gas-limit gas per `price()` call, making any liquidation of their position revert due to out-of-gas, permanently blocking liquidation of an undercollateralized position.

### Finding Description

**Market creation is permissionless.** `touchMarket()` at `src/Midnight.sol:755-791` validates only that collateral tokens are sorted, LLTV is an allowed tier, and `maxLif` matches a cursor formula. **Oracle addresses are never validated.** [2](#0-1) 

**`supplyCollateral()` does not call the oracle.** A borrower activates a collateral slot simply by supplying a non-zero amount. The oracle is never invoked during supply, so gas-consuming oracles are invisible at activation time. [3](#0-2) 

**`IOracle` imposes no gas restriction.** The interface is a single `price()` view function with no gas stipend. [4](#0-3) 

**`MAX_COLLATERALS_PER_BORROWER = 16`** bounds loop iterations but not gas per iteration. [5](#0-4) 

**Exploit flow:**
1. Attacker deploys 16 oracle contracts whose `price()` each consume ~1.9M gas (e.g., a tight SLOAD loop) but return a valid non-zero price.
2. Attacker calls `touchMarket()` (via `supplyCollateral`) with a `Market` struct whose 16 `collateralParams` each reference one of these oracles. All other market parameters are valid (sorted tokens, allowed LLTV, correct `maxLif`).
3. Attacker calls `supplyCollateral()` 16 times (once per collateral index), activating all 16 bits in `collateralBitmap`. Oracle is never called here.
4. Attacker borrows by taking a sell offer (or self-lends from a second address to fund the market).
5. Attacker's own oracles return a low price, making the position unhealthy.
6. Any liquidator calling `liquidate()` triggers the while loop: 16 × ~1.9M gas ≈ 30M gas → out-of-gas revert.

**Why existing checks fail:**
- `MAX_COLLATERALS_PER_BORROWER = 16` only bounds iteration count, not gas per call.
- The `liquidatorGate` check (line 597-600) runs before the loop and cannot prevent the OOG.
- The protocol's own LIVENESS section (lines 142-158) acknowledges reverting oracles block liquidation but does not address gas-consuming oracles. [6](#0-5) 

### Impact Explanation
An undercollateralized borrower position with 16 gas-consuming oracles activated cannot be liquidated by any liquidator regardless of gas price or block gas limit. The core invariant "unhealthy positions remain liquidatable" is violated. Bad debt accrues and is never realized, harming all lenders in the market. The DoS is permanent as long as the borrower does not voluntarily withdraw collateral (which they have no incentive to do).

### Likelihood Explanation
**Preconditions:**
- Attacker must deploy 16 gas-consuming oracle contracts (cheap, one-time cost).
- Attacker must create a market with these oracles (permissionless, no admin approval).
- Attacker must activate all 16 collateral slots (requires 16 `supplyCollateral` calls with non-zero amounts of 16 distinct ERC-20 tokens — the main practical cost).
- Attacker must have debt in the market (requires a lender; attacker can self-lend from a second address).

**Feasibility:** Moderate. The token-diversity requirement (16 distinct collateral tokens, sorted by address) is the main barrier. An attacker can deploy 16 minimal ERC-20 tokens at chosen addresses to satisfy the sort order. The attack is fully repeatable and the DoS is indefinite.

### Recommendation
Apply a gas stipend to each oracle call using a low-level call with explicit gas forwarding, and treat a gas-exhausting oracle the same as a reverting oracle (i.e., revert or skip). Alternatively, enforce a per-oracle gas cap at the `IOracle` call site:

```solidity
(bool success, bytes memory data) = _collateralParam.oracle.call{gas: MAX_ORACLE_GAS}(
    abi.encodeCall(IOracle.price, ())
);
require(success && data.length == 32, OracleFailed());
uint256 price = abi.decode(data, (uint256));
```

A complementary mitigation is to validate oracle gas consumption at market creation time in `touchMarket()`, though this is harder to enforce on-chain.

        bytes32 id = toId(market);
        if (marketState[id].tickSpacing == 0) {
            require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
            require(market.collateralParams.length > 0, NoCollateralParams());
            require(market.collateralParams.length <= MAX_COLLATERALS, TooManyCollateralParams());
            address previousCollateralToken;
            for (uint256 i = 0; i < market.collateralParams.length; i++) {
                address collateralToken = market.collateralParams[i].token;
                require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());
                uint256 lltv = market.collateralParams[i].lltv;
                require(isLltvAllowed(lltv), LltvNotAllowed());
                require(
                    market.collateralParams[i].maxLif == maxLif(lltv, LIQUIDATION_CURSOR_LOW)

                    InvalidMaxLif()
