### Title
Zero Oracle Price Allows Free Collateral Seizure in Liquidation — (File: src/Midnight.sol)

### Summary

The `liquidate` function in `Midnight.sol` does not validate that the oracle price for the liquidated collateral is non-zero before using it in repayment calculations. When a collateral oracle returns `0`, the `seizedAssets`-input path silently computes `repaidUnits = 0`, allowing a liquidator to seize collateral without repaying any debt. The protocol's own LIVENESS documentation acknowledges the zero-price issue only for the `repaidUnits`-input path (which reverts due to division by zero), leaving the `seizedAssets`-input path unaddressed.

---

### Finding Description

**Vulnerability class:** Pricing/accounting — unvalidated zero oracle price in liquidation calculation.

**Root cause:**

In `liquidate`, `liquidatedCollatPrice` is declared as `uint256` (default value `0`) and is only assigned if the loop over activated collaterals encounters `i == collateralIndex`: [1](#0-0) 

If the oracle at `market.collateralParams[collateralIndex].oracle` returns `0`, then `liquidatedCollatPrice` remains `0` after the loop.

The value is then used in the repayment calculation: [2](#0-1) 

**Arithmetic trace when `seizedAssets > 0` and `liquidatedCollatPrice == 0`:**

`mulDivUp` is defined as: [3](#0-2) 

So:
```
seizedAssets.mulDivUp(0, ORACLE_PRICE_SCALE)
= (seizedAssets * 0 + (1e36 - 1)) / 1e36
= (1e36 - 1) / 1e36
= 0   ← integer division, no revert
```
Then `0.mulDivUp(WAD, lif) = (lif - 1) / lif = 0`.

Result: **`repaidUnits = 0`** — the liquidator seizes `seizedAssets` collateral tokens while repaying nothing.

**Simultaneous bad-debt path:**

When `price == 0` for all activated collaterals, `maxDebt = 0` and `badDebt = originalDebt` (the full debt is socialized): [4](#0-3) 

The borrower's debt is zeroed out via bad-debt realization before the seizure executes, so `_position.debt -= 0` does not revert. The net effect is:

1. All borrower debt is socialized onto lenders (`lossFactor` increases).
2. Liquidator seizes all collateral for free.
3. Lenders are double-harmed: they absorb the bad debt **and** receive no collateral recovery.

**The protocol's own LIVENESS comment only documents the revert case:** [5](#0-4) 

The `seizedAssets`-input path with zero price is not acknowledged as a known/accepted behavior.

---

### Impact Explanation

A liquidator can call `liquidate(market, collateralIndex, seizedAssets=X, repaidUnits=0, ...)` against any borrower in a market whose collateral oracle returns `0`. The liquidator receives `X` collateral tokens while transferring `0` loan tokens. Lenders in the market simultaneously absorb the full bad debt via `lossFactor` socialization. This constitutes direct theft of collateral value from the protocol's lenders.

---

### Likelihood Explanation

Requires the collateral oracle to return `0`. This is reachable via:

- An uninitialized oracle contract (returns `0` by default before any price is set).
- An oracle that is upgraded/replaced and temporarily returns `0` during migration.
- Oracle manipulation (explicitly in scope per `SECURITY.md`: *"This does not exclude oracle manipulation/flash-loan attacks"*). [6](#0-5) 

Any of these conditions, combined with an existing borrower in the affected market, is sufficient to trigger the exploit with no privileged access required.

---

### Recommendation

Add an explicit non-zero check on `liquidatedCollatPrice` before it is used in the repayment calculation, analogous to the external report's recommendation:

```solidity
// After the collateral loop, before the repayment block:
if (repaidUnits > 0 || seizedAssets > 0) {
    require(liquidatedCollatPrice > 0, "oracle price not initialized");
    ...
}
```

This mirrors the fix recommended in the reference report (`assert old > 0, "oracle price not init"`) and closes the gap left by the existing LIVENESS documentation.

---

### Proof of Concept

**Preconditions:**
- Market `M` exists with a single collateral whose oracle can be made to return `0`.
- Borrower `B` has `debt = D > 0` and `collateral[0] = C > 0` in market `M`.

**Steps:**
1. Attacker causes oracle for collateral `0` to return `0` (e.g., via manipulation or uninitialized state).
2. Attacker calls:
   ```solidity
   midnight.liquidate(
       market,          // market M
       0,               // collateralIndex
       C,               // seizedAssets = full collateral
       0,               // repaidUnits = 0 (input)
       B,               // borrower
       false,           // postMaturityMode
       attacker,        // receiver
       address(0),      // no callback
       ""
   );
   ```
3. Inside `liquidate`:
   - Loop: `price = oracle.price() = 0` → `maxDebt = 0`, `badDebt = D`.
   - Bad debt branch: `_position.debt -= D` → debt zeroed, `lossFactor` increased.
   - `seizedAssets = C > 0` branch: `repaidUnits = C.mulDivUp(0, 1e36).mulDivUp(WAD, lif) = 0`.
   - `_position.collateral[0] -= C` → borrower loses all collateral.
   - `SafeTransferLib.safeTransfer(collateralToken, attacker, C)` → attacker receives `C` tokens.
   - `SafeTransferLib.safeTransferFrom(loanToken, attacker, address(this), 0)` → attacker pays nothing.

**Outcome:** Attacker receives `C` collateral tokens for free. Lenders absorb `D` units of bad debt. [7](#0-6)
