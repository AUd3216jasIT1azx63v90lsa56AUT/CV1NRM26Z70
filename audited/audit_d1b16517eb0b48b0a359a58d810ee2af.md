### Title
Fee-on-Transfer `loanToken` in `repay()` Breaks Solvency Invariant, Causing Last-Lender Loss - (File: src/Midnight.sol)

### Summary

`repay()` increments `marketState[id].withdrawable` by the caller-supplied `units` before executing `safeTransferFrom`, but with a fee-on-transfer `loanToken` the contract receives only `units * (1 - fee)`. The resulting gap — `withdrawable` exceeds actual token balance — means the last lenders to call `withdraw()` either receive fewer tokens than their credit entitlement or have their call revert entirely. No check in `touchMarket()` or `repay()` prevents fee-on-transfer tokens from being used as `loanToken`, and the Certora solvency proof explicitly assumes standard ERC-20 transfer semantics.

### Finding Description

**Exact code path:**

`repay()` in `src/Midnight.sol` lines 502–521:

```
position[id][onBehalf].debt -= UtilsLib.toUint128(units);          // line 508
marketState[id].withdrawable += UtilsLib.toUint128(units);          // line 509
...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
``` [1](#0-0) 

The state write at line 509 uses the caller-supplied `units` value unconditionally. The transfer at line 520 requests `units` from the payer, but a fee-on-transfer token deducts a fee in-flight, so the contract receives `units * (1 - fee)`. There is no pre/post balance check to reconcile the actual received amount against the accounting delta.

**Market creation — no restriction on fee-on-transfer tokens:**

`touchMarket()` validates collateral params, LLTV, maxLif, and maturity, but imposes no constraint on `market.loanToken`. Any address, including a fee-on-transfer ERC-20, is accepted. [2](#0-1) 

**Certora solvency proof scope:**

The `tokenBalanceCorrect` strong invariant in `certora/specs/Solvency.spec` is proven only under the explicit assumption that tokens transfer without fees:

> "Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits." [3](#0-2) 

The invariant itself: [4](#0-3) 

This proof therefore provides no protection for fee-on-transfer tokens.

**`live_context.json` acknowledges the risk but does not exclude it:** [5](#0-4) 

**Exploit flow:**

1. Attacker (borrower) calls `touchMarket()` with a fee-on-transfer `loanToken` (e.g., 1% transfer fee). Market is created with no restriction.
2. Lender provides credit via `take()` — lender's `position.credit` and `marketState.totalUnits` are set to `U`.
3. Borrower calls `repay(market, U, borrower, address(0), "")`:
   - `marketState[id].withdrawable` becomes `U`
   - Contract receives only `U * 0.99` (after 1% fee)
4. Invariant is now broken: `withdrawable = U` but `loanToken.balanceOf(midnight) = U * 0.99`
5. First lender calls `withdraw(market, U * 0.99, lender, lender)` — succeeds, draining the balance.
6. Second lender calls `withdraw(market, U * 0.01, lender2, lender2)` — `safeTransfer` reverts (insufficient balance) or, if multiple lenders share the shortfall, each receives proportionally less than their credit.

The same structural flaw exists in `liquidate()` at line 675 (`withdrawable += repaidUnits` before the liquidator's `safeTransferFrom`) and in `take()` at line 418 (`claimableSettlementFee += buyerAssets - sellerAssets` before the fee transfer at line 455), but those are outside the stated scope. ### Impact Explanation

The solvency invariant `loanToken.balanceOf(midnight) >= marketState.withdrawable + claimableSettlementFee[loanToken]` is violated after every `repay()` call with a fee-on-transfer `loanToken`. The shortfall accumulates with each repayment. Lenders who call `withdraw()` after the balance is exhausted receive fewer tokens than their `credit` entitlement — a direct, unrecoverable loss of principal. The loss is proportional to the token's transfer fee and the total repaid volume.

### Likelihood Explanation

**Preconditions:** A fee-on-transfer ERC-20 must be used as `loanToken`. This is permissionless — any user can call `touchMarket()` with such a token. No admin action is required.

**Feasibility:** The bug is triggered by the first `repay()` call in any such market. It is deterministic, repeatable, and requires no special timing or oracle manipulation.

**Repeatability:** Every subsequent `repay()` widens the gap. The shortfall compounds linearly with repayment volume and the token's fee rate.

### Recommendation

After the `safeTransferFrom` in `repay()`, measure the actual received amount via a pre/post balance check and use that value — not the caller-supplied `units` — for the `withdrawable` increment:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
marketState[id].withdrawable += UtilsLib.toUint128(received);
position[id][onBehalf].debt -= UtilsLib.toUint128(received);
```

Note this requires moving the state updates **after** the transfer (breaking the current checks-effects-interactions ordering), which in turn requires a reentrancy guard on `repay()`. Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as `loanToken` by adding a validation check in `touchMarket()` (e.g., a balance-delta probe during market creation). The same fix must be applied to `liquidate()` and `take()`.
