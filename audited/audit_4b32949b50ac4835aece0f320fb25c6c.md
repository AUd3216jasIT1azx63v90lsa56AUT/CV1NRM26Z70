Audit Report

## Title
Bitmap Oracle Loop Lacks Try/Catch, Blocking Liquidation When Any Non-Liquidated Collateral Oracle Reverts - (File: src/Midnight.sol)

## Summary
The `liquidate` function iterates over every collateral in a borrower's `collateralBitmap` and calls `IOracle(_collateralParam.oracle).price()` as a bare external call with no `try/catch` or fallback. If any oracle in the bitmap reverts — including one for a collateral not being liquidated — the entire `liquidate` transaction reverts. This makes unhealthy multi-collateral positions permanently unliquidatable for as long as any one oracle is non-functional, violating the protocol's core solvency invariant.

## Finding Description
**Code path:** `liquidate` (Midnight.sol:581) → bitmap loop (607–618) → `IOracle(_collateralParam.oracle).price()` (line 610, bare external call, no error handling) → revert propagates → `liquidate` reverts before the `NotLiquidatable` check at line 620.

**Root cause:** The loop at lines 607–618 unconditionally queries every oracle in `_position.collateralBitmap`. Only the oracle at `collateralIndex` is strictly required to compute `liquidatedCollatPrice` (line 611) and the seized/repaid amounts (lines 650–652). The remaining oracle prices are used to compute `maxDebt` and `badDebt`, but a revert in any one of them aborts the entire transaction with no state change.

**Exploit flow:**
1. Borrower activates two collaterals (indices 0 and 1) in a market. Both oracles are healthy at deposit time.
2. The oracle for collateral[1] begins reverting (e.g., Chainlink heartbeat/circuit-breaker triggers, Pyth staleness check, or sequencer downtime — all standard production behaviors).
3. The borrower's position becomes unhealthy because collateral[0]'s price drops below the LLTV threshold.
4. A liquidator calls `liquidate(market, 0, 0, repaidUnits, borrower, false, receiver, address(0), '')` targeting collateral[0].
5. The bitmap loop reaches index 1, calls `IOracle(oracle[1]).price()`, which reverts.
6. The revert propagates; `liquidate` reverts with no state change.
7. The unhealthy position cannot be liquidated by anyone as long as oracle[1] reverts.

**Why existing checks fail:** The `liquidatorGate` check (lines 597–600) and `liquidationLocked`/`NotLiquidatable` check (lines 620–624) are never reached. There is no per-oracle error isolation, no fallback price, and no mechanism to skip a reverting oracle.

## Impact Explanation
An unhealthy position with 2+ collaterals becomes permanently unliquidatable for as long as any non-liquidated collateral's oracle reverts. Bad debt accrues without being socialized via the `lossFactor` path (lines 626–641), which is also unreachable. Lenders cannot recover funds, collateral cannot be seized, and the protocol's solvency invariant is violated. This constitutes a permanent freeze of user/protocol state and a critical integrity failure in the liquidation state machine.

## Likelihood Explanation
**Preconditions:** Borrower must have ≥ 2 activated collaterals (standard in multi-collateral markets) and at least one non-liquidated collateral's oracle must revert. Production oracles (Chainlink, Pyth, Uniswap TWAP) routinely revert on stale heartbeat, circuit-breaker triggers, or L2 sequencer downtime — these are designed behaviors, not edge cases. The preconditions co-occur naturally during market stress: price drops that make positions unhealthy also tend to trigger oracle circuit-breakers on correlated assets. The block is deterministic and repeatable — every liquidation attempt reverts identically until oracle[1] recovers, which can persist for hours or days.

## Recommendation
Wrap each oracle call in a `try/catch` block. If the oracle for the collateral being liquidated (`collateralIndex`) reverts, revert the transaction (the price is strictly required). For all other collaterals, use a fallback (e.g., last known price, zero, or skip the contribution to `maxDebt`/`badDebt`) so that a single non-critical oracle failure does not block liquidation. Example:

```solidity
try IOracle(_collateralParam.oracle).price() returns (uint256 price) {
    if (i == collateralIndex) liquidatedCollatPrice = price;
    maxDebt += ...;
    badDebt = badDebt.zeroFloorSub(...);
} catch {
    require(i != collateralIndex, OracleReverted());
    // skip contribution from this collateral to maxDebt/badDebt
}
```

This preserves the invariant that the liquidated collateral's price is always available while allowing liquidation to proceed when non-liquidated collateral oracles are temporarily unavailable.

## Proof of Concept
**Minimal Foundry test plan:**
1. Deploy a mock market with two collaterals, each backed by a mock oracle.
2. Have a borrower deposit both collaterals and borrow against them.
3. Drop collateral[0]'s price below the LLTV threshold (position becomes unhealthy).
4. Set oracle[1] to revert on `price()`.
5. Call `liquidate(market, 0, 0, repaidUnits, borrower, false, receiver, address(0), '')`.
6. Assert the call reverts (confirming the bug).
7. Restore oracle[1] and assert the same call succeeds (confirming the fix path).
