Audit Report

## Title
`maxAssets` cap on buy offers bypassed via zero-rounding when `buyerPrice < WAD` - (File: `src/Midnight.sol`)

## Summary
When a buy offer sets `maxAssets > 0`, the `consumed` mapping is incremented by `buyerAssets = mulDivDown(units, buyerPrice, WAD)`. For any tick where `buyerPrice < WAD`, a taker submitting `units = 1` causes `buyerAssets` to floor to zero, leaving `consumed` permanently at zero and the `ConsumedAssets` guard permanently satisfied. The offer can be filled an unlimited number of times, inflating the maker's credit, the taker's debt, and `totalUnits` without bound and without any token transfer. The protocol's own NatDoc and a dedicated test named `testBugBuyMaxAssetsBypass` confirm the path is reachable and the state change is real.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // floor(units * buyerPrice / WAD)
    : units.mulDivUp(buyerPrice, WAD);

uint256 newConsumed;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For a buy offer (`offer.buy == true`) with `maxAssets > 0`, the consumed delta is `buyerAssets`. When `buyerPrice < WAD` (true for every tick strictly below `MAX_TICK`) and `units = 1`, `mulDivDown(1, buyerPrice, WAD) = 0`. The guard `newConsumed <= maxAssets` is trivially satisfied because `newConsumed` never changes from its pre-call value.

**NatDoc acknowledgment — `src/Midnight.sol` line 94:**
> "It is possible to give units to a fully consumed assets-based buy offer with price < 1."

This confirms the path is known and reachable, not gated by any external precondition.

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = M > 0`, `maxUnits = 0`, and any `tick < MAX_TICK` (so `buyerPrice < WAD`).
2. Taker calls `take(offer, ..., units=1, ...)` repeatedly.
3. Each call: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; `consumed` stays at its current value; guard passes; `buyerPos.credit += 1` (line 410); `sellerPos.debt += 1` (line 414); `totalUnits += 1` (line 416–417); zero tokens transferred (line 455).
4. After N calls: `consumed[maker][group]` is unchanged, maker's credit has grown by N, taker's debt has grown by N, `totalUnits` has grown by N.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — `newConsumed` is always 0 (or whatever it was before), so this never fires regardless of `maxAssets`.
- `require(UtilsLib.atMostOneNonZero(offer.maxAssets, offer.maxUnits))` — satisfied; only `maxAssets` is set.
- No minimum-units or minimum-assets guard exists anywhere in the `take` path.

**Test confirmation — `test/TakeTest.sol` lines 857–889:**
The test pre-sets `consumed` to `maxAssets` (fully consumed), then calls `take(1, ...)` and asserts: `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.

## Impact Explanation
The `maxAssets` cap on buy offers is completely ineffective whenever `buyerPrice < WAD` and the taker submits `units` small enough that `units * buyerPrice < WAD`. An unprivileged taker can:
- Bypass the maker's intended fill cap entirely, forcing the maker's credit and the taker's debt to grow without bound.
- Inflate `totalUnits` without limit, corrupting the protocol-wide accounting invariant that `consumed` must faithfully track and cap total fill activity.
- Execute each fill at zero token cost (`buyerAssets = sellerAssets = 0`), meaning the only constraint is gas and the taker's collateral backing the debt they accumulate.

This constitutes an unauthorized state change and accounting integrity failure: the `maxAssets` cap, a core offer parameter, is rendered meaningless for the most common tick range.

## Likelihood Explanation
**Preconditions:**
1. A buy offer exists with `maxAssets > 0` and any `tick < MAX_TICK` — the normal operating range for virtually all offers.
2. The taker is any unprivileged address; no special role, oracle manipulation, or admin action is required.
3. The taker must supply collateral to back the debt they accumulate, but the per-fill token cost is zero.

**Feasibility:** Trivially repeatable. The condition `buyerPrice < WAD` holds for all ticks below `MAX_TICK`. The only cost is gas per call. No external dependency or timing constraint is involved.

## Recommendation
Enforce a minimum consumed increment of 1 when `maxAssets > 0` for buy offers. The simplest targeted fix is to use ceiling division (`mulDivUp`) for the `consumed` increment (not necessarily for the asset transfer amount), or to add an explicit guard:

```solidity
if (offer.maxAssets > 0) {
    uint256 consumedDelta = offer.buy ? buyerAssets : sellerAssets;
    // Prevent zero-rounding bypass: require at least 1 unit of consumed per fill
    require(consumedDelta > 0 || units == 0, ZeroConsumedDelta());
    newConsumed = consumed[offer.maker][offer.group] += consumedDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, use `mulDivUp` for the consumed delta on buy offers so that any nonzero `units` always advances `consumed` by at least 1, preserving the cap's effectiveness across all tick ranges.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) is a complete, self-contained proof:
- Sets `maxAssets = 1`, `tick = MAX_TICK - 16` (ensuring `buyerPrice < WAD`).
- Pre-sets `consumed` to `maxAssets` via `setConsumed` (simulating a fully consumed offer).
- Calls `take(1, borrower, lenderOffer)`.
- Asserts `buyerAssets == 0`, `sellerAssets == 0`, `consumed` unchanged, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.

    // Show that a buy offer with offerPrice < WAD can be taken with units > 0
    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:

        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
