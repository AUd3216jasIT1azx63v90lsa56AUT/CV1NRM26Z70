### Title
Pool Sponsor Can Pause Staking and Bonus Contributions to Execute a Targeted DoS - (File: src/ConfidencePool.sol)

### Summary
`ConfidencePool` inherits OpenZeppelin's `Pausable` and exposes `pause()` / `unpause()` exclusively to the pool owner — the sponsor (agreement owner). The `stake()` and `contributeBonus()` functions are gated by `whenPoolNotPaused`. A sponsor who has staked in their own pool can pause it after `riskWindowStart` is sealed — at which point withdrawals are already permanently disabled — locking all existing stakers in while blocking any new capital or bonus from entering. There is no timelock, no decentralization mechanism, and `docs/DESIGN.md` contains no acceptance of this behavior.

### Finding Description
`ConfidencePool` inherits `Pausable` and declares two owner-only entrypoints:

```solidity
// src/ConfidencePool.sol:662-670
function pause() external onlyOwner whenPoolNotPaused { _pause(); }
function unpause() external onlyOwner whenPoolPaused  { _unpause(); }
```

The pool owner is the sponsor, assigned unconditionally at initialization:

```solidity
// src/ConfidencePool.sol:218
_transferOwnership(owner_);   // owner_ == msg.sender from createPool()
```

`createPool()` enforces `IAgreement(agreement).owner() != msg.sender`, so the pool owner is always the agreement owner (sponsor).

Both deposit-side entrypoints carry the pause gate:

```solidity
// src/ConfidencePool.sol:222
function stake(uint256 amount) external nonReentrant whenPoolNotPaused { … }
// src/ConfidencePool.sol:266
function contributeBonus(uint256 amount) external nonReentrant whenPoolNotPaused { … }
```

`withdraw()` (line 288) carries **no** `whenPoolNotPaused` guard, but withdrawals are already permanently disabled once `riskWindowStart != 0` (the one-way latch sealed on the first `UNDER_ATTACK` observation). The claim paths (`claimSurvived`, `claimExpired`, `claimCorrupted`, etc.) are also unguarded, so existing stakers can still exit post-resolution — but they cannot do so during the active-risk window.

**Concrete attack path:**

1. Sponsor deploys a pool via `createPool()` and stakes into it via `stake()`.
2. The registry transitions to `UNDER_ATTACK`; any pool interaction (or a permissionless `pokeRiskWindow()`) seals `riskWindowStart`, permanently disabling `withdraw()` for all stakers.
3. Sponsor calls `pause()`. The pool is now paused with no timelock.
4. Every subsequent `stake()` call reverts `PoolPaused`; every `contributeBonus()` call reverts `PoolPaused`.
5. Existing stakers are locked in with no exit until resolution; no new capital can dilute the sponsor's share; no additional bonus can be contributed to the pool.
6. Sponsor's k=2 score — and therefore their fraction of `snapshotTotalBonus` — is frozen at its current maximum relative weight.

The sponsor can `unpause()` at will (e.g., to admit a friendly staker), making the mechanism selectively gating rather than a blunt shutdown.

### Impact Explanation
Once `riskWindowStart` is sealed, existing stakers have no exit. A paused pool traps them while simultaneously preventing any new staker from joining and any bonus contributor from adding to the pool. The sponsor — who controls both the pause lever and their own stake position — can freeze the composition of the pool at the moment most favorable to their own bonus share. The bonus pool is not stolen outright, but its distribution is manipulated: the sponsor earns a larger fraction of `snapshotTotalBonus` than they would under open competition, at the direct expense of stakers who would have joined or bonus contributors who would have increased the pool.

### Likelihood Explanation
The sponsor is the agreement owner and the natural first party to create and seed a pool. Staking in one's own pool is economically rational (early entry maximizes k=2 score). The pause call is a single, cheap transaction with no on-chain prerequisite beyond pool ownership. The window of opportunity is the entire active-risk interval — potentially days to weeks — giving the sponsor ample time to act. No external condition or oracle is required.

### Recommendation
Remove `Pausable` inheritance, the `pause()` / `unpause()` functions, and the `whenPoolNotPaused` modifier from `ConfidencePool` entirely, mirroring the fix applied in the referenced Primitive Protocol PR #20. The factory already exposes a `pause()` that gates new pool *creation*; a per-pool pause controlled by the sponsor adds no legitimate safety valve that isn't already covered by the registry-driven state machine and the moderator's `flagOutcome`. If an emergency pause is genuinely needed, restrict it to the `defaultOutcomeModerator` or the factory owner — both explicitly trusted actors — and document the accepted behavior in `docs/DESIGN.md`.

```solidity

pragma solidity 0.8.26;

// Assume pool is deployed, riskWindowStart has been sealed (UNDER_ATTACK observed).
// Sponsor holds eligibleStake[sponsor] > 0.

// Step 1 — sponsor pauses the pool (single tx, no timelock)
pool.pause();   // onlyOwner passes; PoolPaused event emitted

// Step 2 — honest staker attempts to join
vm.prank(honestStaker);

vm.prank(bonusContributor);
pool.contributeBonus(500e18);   // reverts: PoolPaused

// Step 4 — honest staker attempts to exit (already blocked by riskWindowStart latch)
vm.prank(honestStaker);

// Result: pool composition frozen; sponsor's share of snapshotTotalBonus is maximized
// at resolution because no diluting stakes or bonus additions were admitted.
