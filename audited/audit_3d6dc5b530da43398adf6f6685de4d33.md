Audit Report

## Title
Fee-on-Transfer Loan Token in `repay` Inflates `withdrawable` Beyond Actual Balance ŌĆö (File: `src/Midnight.sol`)

## Summary
The `repay` function credits `marketState[id].withdrawable` using the caller-supplied `units` argument before executing the token transfer. `SafeTransferLib.safeTransferFrom` performs no balance-delta measurement. When the market's `loanToken` is a fee-on-transfer token, the contract receives `units - fee` but records `units` as withdrawable, permanently breaking the solvency invariant and enabling direct loss of lender principal.

## Finding Description

**Exact code path:**

In `src/Midnight.sol`, `repay` at lines 508ŌĆō509 unconditionally increments `withdrawable` by `units` before the transfer executes: [1](#0-0) 

The transfer at line 520 delegates to `SafeTransferLib.safeTransferFrom`, which only checks `token.code.length > 0`, call success, and the boolean return value ŌĆö no balance-before/balance-after measurement is performed: [2](#0-1) 

**Root cause:** For a fee-on-transfer token, `transferFrom(payer, address(this), units)` returns `true` and emits a transfer event for `units`, but only `units - fee` arrives at the contract. The protocol has no balance-delta check.

**Market creation has no token validation:** `touchMarket` validates maturity, collateral ordering, LLTV, and `maxLif`, but performs zero validation on `loanToken` properties. Market creation is permissionless: [3](#0-2) 

**Exploit flow:**
1. Attacker creates a market with a fee-on-transfer ERC-20 as `loanToken` (permissionless, no token validation in `touchMarket`).
2. Borrower takes a loan of `D` units via normal protocol flow.
3. Borrower calls `repay(market, D, onBehalf, address(0), "")`.
4. Protocol executes: `position[id][onBehalf].debt -= D` and `marketState[id].withdrawable += D`.
5. `safeTransferFrom` transfers `D` tokens from borrower; only `D * (1 - feeRate)` arrives at `address(this)`.
6. `withdrawable` now exceeds actual balance by `D * feeRate`.
7. Lenders call `withdraw` ŌĆö protocol executes `withdrawable -= D` and `safeTransfer(loanToken, receiver, D)`.
8. The `safeTransfer` of `D` drains tokens belonging to other lenders or other markets sharing the same token, causing a shortfall.

**Why existing checks fail:** `SafeTransferLib` is a return-value-only wrapper with no balance accounting. There is no market-creation validation rejecting fee-on-transfer tokens. The state mutation precedes the transfer, so no reentrancy guard would help.

## Impact Explanation
Every `repay` call with a fee-on-transfer loan token permanently inflates `marketState[id].withdrawable` relative to the contract's actual token balance. Lenders who withdraw after such repayments receive tokens deposited by other lenders or belonging to other markets, causing direct, irreversible loss of lender principal. This is a direct theft of assets impact class. `SECURITY.md` contains no exclusion for fee-on-transfer tokens. [4](#0-3) 

## Likelihood Explanation
Any unprivileged actor can create a market with a fee-on-transfer loan token ŌĆö market creation is permissionless and `touchMarket` performs no validation on `loanToken` properties. The borrower needs only a standard borrow-then-repay flow. The exploit is repeatable: each `repay` call widens the deficit by `units * feeRate`. Tokens with owner-configurable fees (e.g., USDT's historical fee switch) make this reachable without any malicious token-owner action at exploit time.

## Recommendation
Measure the actual balance delta in `repay` (and any other function that calls `safeTransferFrom` and credits internal accounting by the nominal amount). Before the transfer, record `balanceBefore = IERC20(market.loanToken).balanceOf(address(this))`; after the transfer, compute `received = balanceAfter - balanceBefore`; use `received` instead of `units` for the `withdrawable` increment. Alternatively, document and enforce at market creation that fee-on-transfer tokens are not supported as `loanToken`, by adding a validation check in `touchMarket`.

## Proof of Concept
1. Deploy a standard ERC-20 with a configurable transfer fee (e.g., 1%).
2. Call `touchMarket` with this token as `loanToken` ŌĆö it succeeds with no revert.
3. Execute a normal lend/borrow flow to create a position with `D` units of debt.
4. Call `repay(market, D, borrower, address(0), "")`.
5. Assert: `marketState[id].withdrawable` increased by `D`, but `IERC20(loanToken).balanceOf(address(midnight))` increased by only `D * 0.99`.
6. Have a second lender attempt to `withdraw` their full credit ŌĆö the `safeTransfer` will either revert (if the contract is underfunded) or drain tokens belonging to other depositors, demonstrating the solvency break.
