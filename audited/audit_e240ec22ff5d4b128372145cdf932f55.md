### Title
Continuous Fee Accrual in `_updatePosition` Causes Full-Credit Withdrawals to Revert — (`src/Midnight.sol`)

### Summary

The `withdraw` function calls `_updatePosition` before subtracting the user-supplied `units` from `_position.credit`. Because `_updatePosition` reduces `credit` by the accrued continuous fee, any user who passes `units` equal to their pre-call credit balance will trigger an arithmetic underflow revert. This makes it impossible to withdraw a full credit balance in a single transaction without a helper contract, mirroring the exact accounting ordering flaw described in the external report.

### Finding Description

In `withdraw` (`src/Midnight.sol` lines 481–500):

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    ...
    _updatePosition(market, id, onBehalf);          // (1) credit reduced by accrued fee

    Position storage _position = position[id][onBehalf];
    ...
    _position.credit -= UtilsLib.toUint128(units);  // (2) reverts if units > post-fee credit
    _marketState.withdrawable -= UtilsLib.toUint128(units);
    _marketState.totalUnits -= UtilsLib.toUint128(units);
    ...
}
```

Step (1): `_updatePosition` calls `updatePositionView`, which computes:

```
fee = pendingFee * (block.timestamp - lastAccrual) / (maturity - lastAccrual)
newCredit = credit - fee
```

and writes `_position.credit = newCredit` (line 842).

Step (2): `_position.credit -= units` is then executed. If the caller passed `units = credit` (the pre-call value), the subtraction becomes `newCredit - credit = (credit - fee) - credit`, which underflows and reverts in Solidity 0.8+.

The `_updatePosition` path is unconditional — it always runs before the credit subtraction, and there is no "withdraw all" shortcut that reads the post-update credit automatically.

**Concrete scenario:**

| Variable | Value |
|---|---|
| `_position.credit` at call time | `1000` |
| `_position.pendingFee` | `100` |
| Elapsed since `lastAccrual` | 50% of remaining TTM |
| `fee` computed by `_updatePosition` | `50` |
| `_position.credit` after `_updatePosition` | `950` |
| User-supplied `units` | `1000` |
| `_position.credit -= 1000` | **underflow → revert** |

The same underflow occurs when bad-debt slashing (also applied inside `_updatePosition`) reduces credit before the subtraction.

### Impact Explanation

Any lender who attempts to withdraw their entire credit balance — the most common withdrawal pattern — will have their transaction revert whenever any continuous fee has accrued since their last `updatePosition` call. Because fee accrual is time-continuous, even a single block of delay between reading the credit balance and submitting the transaction is sufficient to trigger the revert. The user's funds are not permanently locked (they can withdraw a slightly smaller amount), but the inability to perform a full withdrawal in one transaction is a reliable, repeatable DoS on the standard exit flow.

### Likelihood Explanation

Likelihood is high. The continuous fee accrues every second the market is active. Any lender who has not called `updatePosition` in the same block as their `withdraw` call will have a non-zero `fee` deducted, making `units = credit` always too large. Frontends that read `creditOf` and submit a withdrawal for that exact amount will fail on every attempt unless the user happens to be in the same block as a prior `updatePosition`. This is the default failure mode for full withdrawals, not an edge case.

### Recommendation

Record the post-`_updatePosition` credit and use it as the effective withdrawal ceiling, analogous to the fix described in the external report. One approach:

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    bytes32 id = touchMarket(market);
    MarketState storage _marketState = marketState[id];
    _updatePosition(market, id, onBehalf);

    Position storage _position = position[id][onBehalf];
    // Clamp units to post-fee credit so a "withdraw all" intent never reverts
    if (units == type(uint256).max) units = _position.credit;

    uint128 pendingFeeDecrease;
    if (_position.credit > 0) {
        pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
        _position.pendingFee -= pendingFeeDecrease;
    }
    _position.credit -= UtilsLib.toUint128(units);
    ...
}
```

Alternatively, document and enforce that callers must use `updatePositionView` to obtain the post-fee credit in the same block, and expose a `withdrawMax` helper in the periphery.

### Proof of Concept

**Setup** (mirrors `testWithdrawReducesPendingFee` in `test/ContinuousFeeTest.sol`):

1. Lender enters a market with `credit = 1000`, `pendingFee = 100`, `continuousFee > 0`, TTM = 100 days.
2. 50 days pass (`elapsed = 50 days`). No `updatePosition` is called.
3. Lender reads `midnight.creditOf(id, lender)` → returns stale value `1000` (not yet updated).
4. Lender submits `midnight.withdraw(market, 1000, lender, lender)`.

**Execution inside `withdraw`:**

- `_updatePosition` runs: `fee = 100 * 50/100 = 50`, sets `_position.credit = 950`.
- `_position.credit -= 1000` → `950 - 1000` → **arithmetic underflow → revert**.

**Expected result:** Transaction reverts. Lender cannot exit their full position.

**Workaround that succeeds:** Lender submits `midnight.withdraw(market, 949, lender, lender)` (any value ≤ 950) — succeeds, but leaves residual credit that requires a second transaction to clear. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function updatePositionView(Market memory market, bytes32 id, address user)
        public
        view
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        uint128 credit = _position.credit;
        uint128 _lastLossFactor = _position.lastLossFactor;
        uint256 postSlashCredit = _lastLossFactor < type(uint128).max
            ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
            : 0;
        uint128 _pendingFee = _position.pendingFee;
        uint256 postSlashPendingFee =
            credit > 0 ? _pendingFee - _pendingFee.mulDivUp(credit - postSlashCredit, credit) : 0;
        uint256 accrualEnd = UtilsLib.min(block.timestamp, market.maturity);
        uint128 _lastAccrual = _position.lastAccrual;
        // forge-lint: disable-next-item(unsafe-typecast) as fee <= pending <= credit which are uint128 position fields
        uint128 fee = _lastAccrual < market.maturity
            ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
            : 0;
        // forge-lint: disable-next-item(unsafe-typecast) as credit and pending are <= uint128 position fields
        return (uint128(postSlashCredit) - fee, uint128(postSlashPendingFee) - fee, fee);
    }
```

```text
    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
```

```text
    function testWithdrawReducesPendingFee(
        uint256 credit,
        uint256 withdrawAmount,
        uint256 feeRate,
        uint256 ttm,
        uint256 elapsed
    ) public {
        credit = bound(credit, 1, MAX_CREDIT);
        feeRate = bound(feeRate, 0, MAX_CONTINUOUS_FEE);
        ttm = bound(ttm, 2, 360 days);
        elapsed = bound(elapsed, 0, ttm - 1);

        setupLender(credit, feeRate, ttm);

        vm.warp(vm.getBlockTimestamp() + elapsed);

        uint256 remaining = midnight.pendingFee(id, lender);
        uint256 feeUnits = remaining.mulDivDown(elapsed, ttm);
        uint256 creditAfterAccrual = credit - feeUnits;
        uint256 remainingAfterAccrual = remaining - feeUnits;

        withdrawAmount = bound(withdrawAmount, 0, creditAfterAccrual);

        deal(address(loanToken), borrower, credit);
        vm.prank(borrower);
        midnight.repay(market, credit, borrower, address(0), hex"");

        uint256 pendingFeeDecrease =
            creditAfterAccrual > 0 ? remainingAfterAccrual.mulDivUp(withdrawAmount, creditAfterAccrual) : 0;

        vm.expectEmit();
        emit EventsLib.UpdatePosition(
            id, lender, credit - creditAfterAccrual, remaining - remainingAfterAccrual, feeUnits
        );
        vm.expectEmit();
        emit EventsLib.Withdraw(lender, id, withdrawAmount, lender, lender, pendingFeeDecrease);
        vm.prank(lender);
        midnight.withdraw(market, withdrawAmount, lender, lender);

        uint256 expectedRemaining = creditAfterAccrual > 0 ? remainingAfterAccrual - pendingFeeDecrease : 0;

        assertEq(midnight.creditOf(id, lender), creditAfterAccrual - withdrawAmount, "credit after withdraw");
        assertApproxEqAbs(midnight.pendingFee(id, lender), expectedRemaining, 1, "remaining after withdraw");

        if (withdrawAmount == creditAfterAccrual) {
            assertEq(midnight.pendingFee(id, lender), 0, "full withdraw zeroes remaining");
            midnight.updatePosition(market, lender);
            assertEq(midnight.pendingFee(id, lender), 0, "full withdraw stays at zero");
        }
```
