### Title
LiquidatorGate checks `msg.sender` but loan token is pulled from `callback` payer, allowing gate bypass - (File: src/Midnight.sol)

### Summary
In `Midnight.liquidate`, the `liquidatorGate.canLiquidate` check is performed against `msg.sender`, but when `callback != address(0)` the actual loan token repayment is pulled from the `callback` address (the payer). A whitelisted liquidator A can call `liquidate` with `callback=B` where B is a gate-blocked address, causing B to be the economic actor (payer and collateral receiver) while the gate only ever checks A.

### Finding Description
The exact code path in `src/Midnight.sol`:

**Step 1 — Gate check (line 597–600):** Only `msg.sender` is checked.
```solidity
require(
    market.liquidatorGate == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender),
    LiquidatorGatedFromLiquidating()
);
```

**Step 2 — Payer assignment (line 679):** When `callback != address(0)`, the payer becomes the callback address, not `msg.sender`.
```solidity
address payer = callback != address(0) ? callback : msg.sender;
```

**Step 3 — Token pull (line 717):** Loan tokens are pulled from `payer` (the callback), not from `msg.sender`.
```solidity
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
```

**Exploit flow:**
- Preconditions: market has a `liquidatorGate` that whitelists address A and blocks address B. B is a contract implementing `ILiquidateCallback` that has pre-approved Midnight to pull loan tokens from it.
- A (whitelisted) calls `midnight.liquidate(market, ..., receiver=B, callback=B, ...)`.
- Gate calls `canLiquidate(A)` → returns `true` → passes.
- Collateral is sent to `receiver=B`.
- `onLiquidate` is called on B; B returns `CALLBACK_SUCCESS`.
- Loan tokens are pulled from B (`payer=B`).
- B has effectively performed the liquidation (paid the loan tokens, received the collateral) despite being blocked by the gate.

**Why existing checks fail:** The `liquidatorGateBlocksLiquidation` Certora rule in `certora/specs/Reverts.spec` (line 324–329) only verifies that if `canLiquidate(msg.sender)` returns false the call reverts. It does not verify that the callback address is also gate-authorized. The `OnlyExplicitPayerCanLoseTokens.spec` rule (line 128) explicitly permits the callback as a payer for `liquidate`, confirming the callback-as-payer path is intentional but the gate gap is unaddressed.

### Impact Explanation
The `liquidatorGate` access control is fully bypassed for the actual economic actor. A gate-blocked address B can pay for and profit from liquidations (receiving seized collateral) in a market that was configured to exclude it. Any market relying on `liquidatorGate` to enforce KYC, regulatory, or trust-based restrictions on liquidators has those restrictions rendered ineffective whenever a whitelisted liquidator cooperates (or is a public/permissionless contract that B can trigger).

### Likelihood Explanation
Requires a whitelisted liquidator A to call `liquidate` with `callback=B`. This is trivially achievable if A is a public liquidation router contract (anyone can supply the `callback` argument). Even with a cooperative EOA, the collusion is low-friction: B deploys a callback contract, pre-approves Midnight, and asks any whitelisted address to submit the transaction. The condition is repeatable on every liquidatable position in the gated market.

### Recommendation
When `callback != address(0)`, also gate-check the callback address:
```solidity
require(
    market.liquidatorGate == address(0)
        || (ILiquidatorGate(market.liquidatorGate).canLiquidate(msg.sender)
            && (callback == address(0) || ILiquidatorGate(market.liquidatorGate).canLiquidate(callback))),
    LiquidatorGatedFromLiquidating()
);
```
This ensures that both the initiator and the actual payer/callback are authorized by the gate, closing the bypass.
