### Title
Fee-on-Transfer Collateral Token Breaks Withdrawal Accounting, Causing Receiver to Receive Less Than Debited Amount - (File: src/Midnight.sol)

### Summary
`withdrawCollateral` debits `assets` from `position[id][onBehalf].collateral[collateralIndex]` and then calls `SafeTransferLib.safeTransfer(collateralToken, receiver, assets)`, which only verifies the boolean return value of `transfer()` but does not check the actual amount received by `receiver`. When the collateral token is a fee-on-transfer token, the receiver receives `assets * (1 - fee_rate)` while the internal accounting is reduced by the full `assets`, creating a permanent divergence between recorded and actual collateral. The same flaw exists symmetrically in `supplyCollateral`, where the contract records `assets` deposited but only receives `assets * (1 - fee_rate)`, meaning the contract's actual token balance is already less than its internal accounting before any withdrawal occurs.

### Finding Description

**Code path:**

`supplyCollateral` (`src/Midnight.sol` line 533): [1](#0-0)

```

```
The contract records `assets` but receives only `assets * (1 - fee_rate)` — internal accounting is immediately inflated.

`withdrawCollateral` (`src/Midnight.sol` lines 561–572): [3](#0-2) 
1. Debits `assets` from internal accounting.
2. Passes `isHealthy` (which uses the inflated internal collateral value).
3. Calls `SafeTransferLib.safeTransfer(collateralToken, receiver, assets)`.

`SafeTransferLib.safeTransfer` (`src/libraries/SafeTransferLib.sol` lines 12–22): [4](#0-3) 
Only checks that `transfer()` returns `true` (or no return). It does **not** compare pre/post balances of `receiver` or the contract. A fee-on-transfer token's `transfer()` returns `true` while silently delivering only `assets * (1 - fee_rate)` to `receiver`.

**Root cause:** No balance-before/balance-after guard in either `supplyCollateral` or `withdrawCollateral`. Market creation (`touchMarket`) validates LLTV tiers, sorted collateral params, and maxLif, but imposes no restriction on the token type. [5](#0-4) 

**Attacker-controlled inputs:** Any borrower who creates or uses a market whose `collateralParams[i].token` is a fee-on-transfer ERC20. Market creation is permissionless.

**Exploit flow:**
1. Borrower creates a market with a fee-on-transfer collateral token (fee rate `f`).
2. Calls `supplyCollateral(market, 0, 1000, borrower)`:
   - Internal accounting records `1000`.
   - Contract receives `1000 * (1 - f)` (e.g., 900 for `f = 10%`).
3. Calls `withdrawCollateral(market, 0, 900, borrower, receiver)`:
   - Internal accounting decreases by 900 → records 100.
   - `isHealthy` passes (uses inflated internal value).
   - `safeTransfer` sends 900 from the contract; receiver gets `900 * (1 - f)` = 810.
   - Contract balance: 0. Internal accounting: 100. Receiver: 810 instead of 900.
4. Attempting to withdraw the remaining 100 recorded units reverts — the contract has no tokens left.

**Why existing checks fail:** The health check operates on `position[id][onBehalf].collateral[collateralIndex]`, which is the inflated internal value, not the actual token balance. `SafeTransferLib.safeTransfer` only validates the boolean return, not the delivered amount. [6](#0-5) 

### Impact Explanation

Every withdrawal of a fee-on-transfer collateral token delivers `assets * (1 - fee_rate)` to `receiver` while debiting the full `assets` from the borrower's internal position. The borrower permanently loses `assets * fee_rate` per withdrawal to the token's fee collector. Additionally, because `supplyCollateral` also inflates accounting, the contract's actual token balance is always less than the sum of recorded collateral positions, meaning the last borrowers to withdraw will find the contract insolvent for their recorded collateral — they cannot recover what the accounting says they own.

### Likelihood Explanation

Markets are permissionless: any address can call `touchMarket` with any ERC20 as collateral. Fee-on-transfer tokens (e.g., tokens with deflationary mechanics or explicit transfer taxes) are a known token category. No on-chain check prevents their use as collateral. The condition is fully attacker-controlled and repeatable on every supply/withdraw cycle.

### Recommendation

Add balance-before/balance-after checks in both `supplyCollateral` and `withdrawCollateral` to measure the actual transferred amount and use that for accounting, similar to how Morpho Blue handles this. Alternatively, explicitly document and enforce (via `touchMarket` validation or a registry) that fee-on-transfer tokens are not supported as collateral or loan tokens, and add a corresponding integration test with a fee-on-transfer mock.

Concretely for `supplyCollateral`:
```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

### Proof of Concept

```solidity
// Foundry unit test
contract FeeOnTransferCollateralTest is BaseTest {
    // Deploy a fee-on-transfer ERC20 that takes 10% on every transfer
    FeeOnTransferERC20 feeToken; // 10% fee, returns true

    function setUp() public override {
        super.setUp();
        feeToken = new FeeOnTransferERC20(10); // 10% fee
    }

    function testFeeOnTransferWithdrawalLoss() public {
        // Build market with fee-on-transfer collateral
        Market memory mkt;
        mkt.loanToken = address(loanToken);
        mkt.maturity = block.timestamp + 100;
        CollateralParams[] memory cp = new CollateralParams[](1);
        cp[0] = CollateralParams({
            token: address(feeToken),
            lltv: 0.77e18,
            maxLif: maxLif(0.77e18, 0.25e18),
            oracle: address(oracle1)
        });
        mkt.collateralParams = cp;

        // Supply 1000 tokens; contract receives 900, records 1000
        deal(address(feeToken), address(this), 1000);
        feeToken.approve(address(midnight), 1000);
        midnight.supplyCollateral(mkt, 0, 1000, address(this));

        bytes32 id = toId(mkt);
        // Internal accounting says 1000
        assertEq(midnight.collateral(id, address(this), 0), 1000);
        // Actual contract balance is 900
        assertEq(feeToken.balanceOf(address(midnight)), 900);

        // Withdraw 900 (max the contract can send)
        address receiver = makeAddr("receiver");
        midnight.withdrawCollateral(mkt, 0, 900, address(this), receiver);

        // Receiver gets 810 (900 * 0.9), not 900
        assertEq(feeToken.balanceOf(receiver), 810); // FAILS: expected 900
        // Internal accounting still shows 100 but contract has 0 tokens
        assertEq(midnight.collateral(id, address(this), 0), 100);
        assertEq(feeToken.balanceOf(address(midnight)), 0);
        // Attempting to withdraw remaining 100 will revert (insufficient balance)
    }
}
```

Expected assertion failure: `feeToken.balanceOf(receiver) == 810`, not `900`, confirming the receiver loses `fee_rate` of every withdrawal amount. The residual 100 units in internal accounting are permanently unrecoverable.

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
    }
