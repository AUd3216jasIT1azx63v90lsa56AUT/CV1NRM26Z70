Audit Report

## Title
Bad Debt Over-Realization via `supplyCollateral` Reentrancy in `onLiquidate` Callback - (File: src/Midnight.sol)

## Summary

`liquidate()` snapshots the borrower's collateral bitmap, computes `badDebt`, and irreversibly commits it to `_marketState.lossFactor` before invoking the `onLiquidate` external callback. Because no reentrancy guard protects `liquidate()`, an authorized callback contract can call `supplyCollateral` for the borrower mid-execution, injecting collateral that was invisible to the snapshot. Lenders are permanently slashed by a `badDebt` that exceeds the true undercollateralization at callback-completion time, while the attacker recovers the supplied collateral at a fixed net cost independent of the slashing magnitude.

## Finding Description

**Exact code path:**

**Line 606** — local bitmap snapshot taken: [1](#0-0) 

**Lines 607–618** — `maxDebt` and `badDebt` computed by iterating only over bits present in the snapshot; any collateral added after this point is invisible: [2](#0-1) 

**Lines 626–641** — `badDebt` irreversibly written to storage: `_position.debt -= uint128(badDebt)` and `_marketState.lossFactor` updated (lenders slashed) before any external call: [3](#0-2) 

**Lines 698–715** — `ILiquidateCallback(callback).onLiquidate(...)` is called **after** the bad-debt write; no reentrancy lock is set before this call: [4](#0-3) 

**Line 444** — `liquidationLocked` / `tExchange` is set exclusively inside `take()`, never inside `liquidate()`: [5](#0-4) 

**Lines 524–546** — `supplyCollateral` only checks `isAuthorized[onBehalf][msg.sender]`; it sets a new bit in `_position.collateralBitmap` and increases `_position.collateral[newIndex]` with no reentrancy guard: [6](#0-5) 

**Root cause:** The protocol assumes collateral state is frozen between the `badDebt` computation and the external callback. This assumption is broken because `supplyCollateral` is callable by any authorized address at any time, including during `onLiquidate`, and `liquidate()` has no reentrancy protection.

**Exploit flow:**

1. Attacker holds debt `D` and collateral `C₀` at index 0 only. Position has genuine bad debt: `D > C₀ · price₀ / ORACLE_PRICE_SCALE · lltv₀ / WAD`.
2. Attacker (as borrower) calls `setIsAuthorized(callbackContract, true, borrower)`.
3. Attacker calls `liquidate(market, 0, 0, 0, borrower, false, receiver, callbackContract, data)` — both `seizedAssets` and `repaidUnits` are 0, which satisfies `atMostOneNonZero`.
4. `badDebt = D − C₀ · price₀ / ORACLE_PRICE_SCALE · WAD / maxLif₀` computed from snapshot (index 0 only).
5. `_position.debt -= badDebt`; `_marketState.lossFactor` updated — lenders slashed by `badDebt`.
6. `onLiquidate` fires. Callback calls `supplyCollateral(market, 1, largeAmount, borrower)`. Bit 1 set in `_position.collateralBitmap`; `_position.collateral[1] = largeAmount`. This collateral was absent from the snapshot and therefore not subtracted from `badDebt`.
7. Callback returns `CALLBACK_SUCCESS`. `repaidUnits = 0`, so `SafeTransferLib.safeTransferFrom(..., 0)` pulls nothing.
8. Post-liquidation: `_position.debt = C₀ · price₀ / ORACLE_PRICE_SCALE · WAD / maxLif₀`; `_position.collateral[1] = largeAmount`.
9. Attacker repays fixed `repay_amount = C₀ · price₀ / ORACLE_PRICE_SCALE · (WAD/maxLif₀ − lltv₀/WAD)` to satisfy the `isHealthy` check in `withdrawCollateral`, then withdraws the full `largeAmount` of collateral[1], recovering it entirely.

**Why existing checks fail:**

The `liquidationLocked` guard (line 621) checks whether the borrower is locked during a `take()` call — it is not a reentrancy guard on `liquidate()` itself and is never set during `liquidate()`. [7](#0-6) 

`supplyCollateral` has no awareness of an in-progress liquidation. [8](#0-7) 

`withdrawCollateral` enforces `isHealthy` after withdrawal (line 568), but the attacker satisfies this by repaying a fixed amount before withdrawing. [9](#0-8) 

There is no post-liquidation health check in `liquidate()` itself. [10](#0-9) 

## Impact Explanation

Lenders suffer a permanent, unwarranted credit loss. `_marketState.lossFactor` is updated with a `badDebt` value larger than the true undercollateralization at callback-completion time. The excess slashing equals approximately `largeAmount · price₁ / ORACLE_PRICE_SCALE · WAD / maxLif₁` loan-token units. Because the attacker recovers the supplied collateral (net cost ≈ fixed `repay_amount` + gas, independent of `largeAmount`), the excess slashing can be made arbitrarily large. `lossFactor` is monotonically non-decreasing, so the loss is irreversible. [11](#0-10) 

## Likelihood Explanation

All preconditions are satisfiable by a single actor who is simultaneously borrower and liquidator. `liquidate()` contains no check preventing `msg.sender == borrower`. No oracle manipulation, governance action, or victim mistake is required. The attack is applicable to any multi-collateral market with at least one unused collateral index for the borrower, which is the common case. It is repeatable and permissionless given the self-authorization path. [12](#0-11) 

## Recommendation

Apply a reentrancy guard to `liquidate()` that prevents re-entry into `supplyCollateral` (and other state-mutating functions) for the affected borrower during execution. The simplest approach is to set the `liquidationLocked` flag for the borrower at the start of `liquidate()` (before the bad-debt computation) and clear it at the end, mirroring the pattern already used in `take()`. Alternatively, move the `badDebt` computation and `lossFactor` update to **after** the `onLiquidate` callback, re-reading `_position.collateralBitmap` from storage at that point so that any collateral supplied during the callback is included in the computation. [13](#0-12) 

## Proof of Concept

**Minimal Foundry test outline:**

1. Deploy a two-collateral market (tokens T0, T1; oracles O0, O1; `lltv₀ = 0.8e18`, `maxLif₀ = 1.1e18`).
2. Lender supplies `L` loan tokens.
3. Borrower supplies `C₀` of T0, borrows `D` such that `D > C₀ · price₀ · lltv₀` (genuinely unhealthy) but `D < C₀ · price₀ / maxLif₀` is false — i.e., `badDebt > 0`.
4. Borrower deploys `MaliciousCallback` that, in `onLiquidate`, calls `supplyCollateral(market, 1, LARGE, borrower)`.
5. Borrower calls `setIsAuthorized(MaliciousCallback, true, borrower)`.
6. Borrower calls `liquidate(market, 0, 0, 0, borrower, false, borrower, MaliciousCallback, "")`.
7. Assert `_marketState.lossFactor` increased by more than the true bad debt (i.e., more than `D − C₀ · price₀ / maxLif₀`).
8. Borrower calls `repay(market, repay_amount, borrower, address(0), "")`.
9. Borrower calls `withdrawCollateral(market, 1, LARGE, borrower, borrower)` — succeeds, recovering `LARGE` of T1.
10. Assert net loan-token cost to attacker = `repay_amount` (fixed), while lender loss = `badDebt` (scales with `LARGE`).
