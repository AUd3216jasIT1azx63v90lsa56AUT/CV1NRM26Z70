### Title
Fee-on-transfer loan token drains bundler residual balance in `repayAndWithdrawCollateral` - (File: src/periphery/MidnightBundles.sol)

### Summary
`repayAndWithdrawCollateral` computes outflows (`units` to Midnight, `referralFeeAssets` to recipient) based on the nominal `assets` parameter, but `pullToken` uses `safeTransferFrom` which does not measure the actual balance delta. When the loan token charges a transfer fee `f`, the bundler receives only `assets*(1-f)` yet must pay out `assets` total, drawing the shortfall `assets*f` from any pre-existing residual balance `R` held by the bundler.

### Finding Description
**Exact code path:**

`repayAndWithdrawCollateral` (`src/periphery/MidnightBundles.sol`, lines 315–348):

```
line 329: referralFeeAssets = assets * referralFeePct / WAD
line 330: units = assets - referralFeeAssets
line 331: pullToken(loanToken, msg.sender, assets, ...)   // inflow: assets*(1-f) credited to bundler
line 332: forceApproveMax(loanToken, MIDNIGHT)
line 334: IMidnight(MIDNIGHT).repay(market, units, ...)   // outflow: bundler pays units to Midnight
line 347: safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets) // outflow: bundler pays referralFeeAssets
```

`pullToken` (`src/periphery/MidnightBundles.sol`, lines 378–398) calls `SafeTransferLib.safeTransferFrom(token, from, address(this), amount)`.

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol`, lines 24–34) only checks that the call succeeds and returns `true`; it does **not** compare balance before/after. With a fee-on-transfer token, the bundler's balance increases by `assets*(1-f)`, not `assets`.

`Midnight.repay` (`src/Midnight.sol`, lines 502–521): since `callback == address(0)`, `payer = msg.sender` (the bundler), and line 520 executes `SafeTransferLib.safeTransferFrom(market.loanToken, bundler, address(this), units)`, pulling the full nominal `units` from the bundler.

**Token balance accounting:**

| Step | Bundler balance |
|---|---|
| Before call | `R` |
| After `pullToken(assets)` | `R + assets*(1-f)` |
| After `repay(units)` | `R + assets*(1-f) - units` |
| After `safeTransfer(referralFeeAssets)` | `R + assets*(1-f) - units - referralFeeAssets` = `R - assets*f` |

Total outflow = `units + referralFeeAssets = assets`. Total inflow = `assets*(1-f)`. Shortfall = `assets*f` is drawn from `R`.

**Attacker-controlled inputs:** `assets` (any positive value), `referralFeePct` (0 to WAD-1), `market.loanToken` (a fee-on-transfer token). The attacker is an ordinary borrower with an existing debt position.

**Why existing checks fail:** The only guards are `require(referralFeePct < WAD)` and the authorization check. Neither `pullToken` nor `SafeTransferLib.safeTransferFrom` verifies the actual received amount. There is no end-of-function balance invariant check.

### Impact Explanation
Any residual balance `R` of the fee-on-transfer loan token held by the bundler (deposited by prior users who have not yet been refunded, or from rounding dust) is drained at rate `f` per `assets` repaid. A borrower calling `repayAndWithdrawCollateral` with `assets = R/f` (rounded) can drain the entire residual in one transaction, stealing funds belonging to other users.

### Likelihood Explanation
Preconditions: (1) the market's `loanToken` is a fee-on-transfer ERC-20; (2) the bundler holds a non-zero residual balance `R` of that token. Condition (1) is a protocol-level market parameter set at market creation and is not under the attacker's control, but fee-on-transfer tokens are a known token class. Condition (2) arises naturally from rounding dust, failed refunds, or concurrent bundler usage. The exploit is repeatable across any call to `repayAndWithdrawCollateral` with such a token and is fully permissionless.

### Recommendation
Replace the nominal-amount accounting with a balance-delta pattern in `pullToken`, or add a post-`pullToken` balance check:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
pullToken(loanToken, msg.sender, assets, loanTokenPermit);
uint256 received = IERC20(loanToken).balanceOf(address(this)) - balanceBefore;
uint256 referralFeeAssets = received.mulDivDown(referralFeePct, WAD);
uint256 units = received - referralFeeAssets;
```

This ensures all downstream outflows are bounded by what was actually received, eliminating the residual-drain vector.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

                revert(add(returndata, 0x20), mload(returndata))

        }
