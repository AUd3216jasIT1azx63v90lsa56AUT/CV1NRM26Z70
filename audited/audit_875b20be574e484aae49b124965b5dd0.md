### Title
Fee-on-transfer loan token inflates `claimableSettlementFee` beyond actual tokens received, violating the solvency invariant - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee` is incremented by the full computed spread `buyerAssets - sellerAssets` before the token transfer, but `SafeTransferLib.safeTransferFrom` delivers only `(buyerAssets - sellerAssets) * (1 - fee_rate)` to the contract when the loan token is fee-on-transfer. Each call creates a gap of `(buyerAssets - sellerAssets) * fee_rate`, and after N takes the cumulative shortfall causes `tokenBalance[loanToken][Midnight] < collateralSum + withdrawableSum + claimableSettlementFee`, violating the core solvency invariant.

### Finding Description
**Exact code path:**

`src/Midnight.sol` line 418 increments the accounting variable unconditionally:
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

Then line 455 performs the inbound transfer:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` calls `transferFrom` and checks only that it does not revert and does not return `false`. It does **not** verify that `address(this).balance` increased by exactly `value`. [3](#0-2) 

**Root cause:** There is no post-transfer balance check. With a fee-on-transfer loan token the contract receives `(buyerAssets - sellerAssets) * (1 - r)` where `r` is the token's fee rate, but `claimableSettlementFee` is credited the full `buyerAssets - sellerAssets`.

**Per-take gap:** `Δ = (buyerAssets - sellerAssets) * r`

**After N takes:** cumulative shortfall = `N * Δ`

**Solvency invariant (from `Solvency.spec`):**
```
tokenBalances[token][Midnight] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token)
``` [4](#0-3) 

This invariant is violated after the first take with a non-zero settlement fee and a fee-on-transfer loan token.

**Why existing checks do not stop it:**
- The Certora `Solvency.spec` proof explicitly assumes well-behaved ERC20 tokens: `"no fee taking from sender or receiver, no rebasing"`. [5](#0-4) [6](#0-5) 
- There is no `require` or balance-delta check anywhere in `take()` after the transfer.
- No market creation check excludes fee-on-transfer tokens as `loanToken`.
- The `pendingFeeReceipt` ghost in the spec tracks the gap only in the CVL model; it has no on-chain counterpart. [7](#0-6) 

**Attacker-controlled inputs:** Any unprivileged taker can call `take()` with a valid offer on a market whose `loanToken` is fee-on-transfer. No special role is required. The taker only needs to be the `msg.sender` or authorized. [8](#0-7) 

### Impact Explanation
After sufficient trading volume the contract's loan token balance falls below `collateralSum + withdrawableSum + claimableSettlementFee`. The `feeClaimer` cannot claim the full accrued settlement fee (the `claimSettlementFee` transfer will revert once the shortfall exceeds the surplus). In the extreme case where the shortfall exceeds `claimableSettlementFee` itself (impossible in a single market but possible across markets sharing the same token), lender withdrawals and collateral redemptions could also be impaired. The solvency invariant is concretely and permanently broken.

### Likelihood Explanation
**Preconditions:**
1. A market is created with a fee-on-transfer ERC20 as `loanToken` — permissionless via `touchMarket`.
2. At least one take is executed with a non-zero settlement fee (settlement fee is set by the `feeSetter` but defaults can be non-zero).
3. No privileged action is required from the attacker.

**Feasibility:** High — any taker executing normal trades on such a market triggers the gap. The gap is proportional to `fee_rate * spread * volume`, so it accumulates monotonically and is irreversible. The attack is repeatable by any market participant.

### Recommendation
After each inbound `safeTransferFrom` in `take()`, verify the actual balance increase equals the expected amount:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
require(
    IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore == buyerAssets - sellerAssets,
    FeeOnTransferToken()
);
```

Alternatively, explicitly document and enforce (via `touchMarket` validation or a registry) that fee-on-transfer tokens are not supported as `loanToken`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IMidnight} from "src/interfaces/IMidnight.sol";

/// @dev ERC20 that takes a 1% fee on every transferFrom.
contract FeeOnTransferToken is ERC20 {
    uint256 constant FEE_BPS = 100; // 1%
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount * FEE_BPS / 10000;
        super.transferFrom(from, to, amount - fee); // deliver less
        _burn(from, fee);                           // burn the fee
        return true;
    }
}

contract FeeOnTransferSolvencyTest is Test {
    Midnight midnight;
    FeeOnTransferToken loanToken;
    // ... standard test setup ...

    function testStatefulFuzzSolvencyViolation(uint256 N, uint256 units) public {
        N = bound(N, 1, 100);
        units = bound(units, 1e18, 1e24);

        // Setup: create market with FeeOnTransferToken as loanToken,
        // fund maker (lender) and taker (borrower), supply collateral.

        for (uint256 i = 0; i < N; i++) {
            midnight.take(offer, "", units, taker, receiver, address(0), "");

            uint256 balance = loanToken.balanceOf(address(midnight));
            uint256 required = collateralSum() + withdrawableSum() + midnight.claimableSettlementFee(address(loanToken));

            // ASSERTION: solvency invariant must hold after every take
            assertGe(balance, required, "solvency invariant violated");
        }
    }
}
```

**Expected result:** The assertion fails on the first iteration (`N=1`) whenever `settlementFee > 0` and `fee_rate > 0`, because `claimableSettlementFee` is incremented by the full spread but the contract receives only `(1 - fee_rate)` of it. The gap grows linearly with `N`.
