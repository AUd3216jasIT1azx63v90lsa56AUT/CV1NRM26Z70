### Title
ERC2612 Permit Front-Running Causes Persistent DoS on `repayAndWithdrawCollateral` - (File: src/periphery/MidnightBundles.sol)

### Summary
`MidnightBundles.repayAndWithdrawCollateral` calls `pullToken` with an ERC2612 permit before any Midnight state changes. The `pullToken` implementation wraps `permit()` in a `try/catch` that silently swallows reverts, then unconditionally calls `safeTransferFrom`. A front-runner who consumes the permit on-chain leaves the borrower with zero allowance, causing `safeTransferFrom` to revert and the entire transaction to fail. Because `pullToken` executes at line 331 before `repay` (line 334) or any `withdrawCollateral` (lines 336–345), no Midnight storage — including the collateral bitmap — is ever written, so the revert is fully atomic with no partial state.

### Finding Description
**Code path:**

`repayAndWithdrawCollateral` (MidnightBundles.sol:315–348): [1](#0-0) 

`pullToken` for `PermitKind.ERC2612` (MidnightBundles.sol:379–384): [2](#0-1) 

**Root cause:** The `try/catch` around `permit()` is intentional ("Tolerate revert: a third party may have already consumed the permit"), but the unconditional `safeTransferFrom` on line 384 still requires a live ERC20 allowance. If the borrower granted allowance exclusively via the permit signature (no prior `approve`), and a front-runner consumes that permit, the allowance is 0 and `safeTransferFrom` reverts.

**Exploit flow:**
1. Borrower signs an ERC2612 permit authorising `MidnightBundles` to spend `assets` of the loan token; no separate `approve` is issued.
2. Borrower broadcasts `repayAndWithdrawCollateral(market, assets, onBehalf, loanTokenPermit, withdrawals, receiver, feePct, feeRecipient)`.
3. Front-runner observes the mempool, extracts `(deadline, v, r, s)` from calldata, and calls `loanToken.permit(borrower, midnightBundles, assets, deadline, v, r, s)` directly, consuming the nonce.
4. Borrower's transaction executes: `try permit(...)` reverts (nonce replayed) → caught silently → `safeTransferFrom(loanToken, borrower, address(this), assets)` reverts (allowance = 0).
5. Entire call reverts. `repay` and `withdrawCollateral` are never reached.

**Why existing checks fail:** The `try/catch` is the only protection; it prevents the permit revert from propagating but does not ensure the subsequent `safeTransferFrom` has sufficient allowance. There is no fallback to check or require a pre-existing allowance. [3](#0-2) 

**Bitmap consistency:** `withdrawCollateral` (Midnight.sol:564–565) clears the bitmap bit only when collateral reaches zero. [4](#0-3) 
Since the revert occurs before line 334, neither `repay` nor `withdrawCollateral` is ever called, so the bitmap is never touched. Transaction atomicity holds; there is no partial state update.

### Impact Explanation
The borrower cannot atomically repay debt and withdraw collateral whenever they rely on an ERC2612 permit for the loan token transfer. A front-runner can repeat this attack for every new permit the borrower signs, creating a persistent, low-cost DoS on the `repayAndWithdrawCollateral` flow. The collateral bitmap is not corrupted (the revert is fully atomic), but the borrower is blocked from using the bundler's permit-based repay path until they switch to a pre-approved allowance.

### Likelihood Explanation
**Preconditions:** (1) The loan token implements ERC2612. (2) The borrower passes `PermitKind.ERC2612` without a prior `approve`. Both are the expected usage pattern for the permit path. **Feasibility:** Mempool permit extraction is trivial on any non-private mempool; the front-runner only needs to call `permit()` with the extracted signature before the borrower's transaction is included. **Repeatability:** The attack can be repeated for every new permit the borrower signs, making the DoS persistent at negligible cost to the attacker.

### Recommendation
After the `try/catch`, check whether the resulting allowance is sufficient before calling `safeTransferFrom`, and if not, revert with a descriptive error rather than letting `safeTransferFrom` revert opaquely. Alternatively, document that callers must always issue a separate `approve` before using the ERC2612 path, and enforce this with a require:

```solidity
// After try/catch:
if (IERC20(token).allowance(from, address(this)) < amount) {
    revert PermitFrontRan();
}
SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
```

This converts a silent DoS into a clear, user-actionable error and prevents the front-runner from causing an opaque revert.

```solidity
// Foundry fork test (or unit test with mock ERC20Permit)
function testPermitFrontRunDoS() public {
    // 1. Borrower has debt; no prior approve to midnightBundles
    uint256 assets = 100e18;
    deal(address(loanToken), borrower, assets);

    // 2. Borrower signs ERC2612 permit
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(borrowerKey,
        _permitHash(address(loanToken), borrower, address(midnightBundles), assets, 0, block.timestamp + 1));

    // 3. Front-runner consumes the permit
    vm.prank(attacker);
    loanToken.permit(borrower, address(midnightBundles), assets, block.timestamp + 1, v, r, s);

    // 4. Borrower's repayAndWithdrawCollateral reverts
    TokenPermit memory permit = TokenPermit({
        kind: PermitKind.ERC2612,
        data: abi.encode(block.timestamp + 1, v, r, s)
    });
    vm.prank(borrower);
    vm.expectRevert(); // safeTransferFrom fails: allowance = 0
    midnightBundles.repayAndWithdrawCollateral(
        market, assets, borrower, permit, new CollateralWithdrawal[](0), borrower, 0, address(0)
    );

    // 5. Assert no state change
    assertEq(midnight.debtOf(id, borrower), initialDebt, "debt unchanged");
    assertEq(midnight.position(id, borrower).collateralBitmap, initialBitmap, "bitmap unchanged");
}
```

**Expected assertions:** `debtOf` and `collateralBitmap` are identical to pre-call values, confirming full atomicity and no bitmap corruption. The revert itself is the DoS evidence.
