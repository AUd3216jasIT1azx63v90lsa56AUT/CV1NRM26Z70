### Title
Flash Loan Fee-on-Transfer Token Drains Protocol Balance Without Repayment Check - (File: src/Midnight.sol)

### Summary

The `flashLoan` function in `src/Midnight.sol` sends `assets[i]` tokens out via `safeTransfer` and then pulls exactly `assets[i]` tokens back via `safeTransferFrom`, with no pre/post balance snapshot to verify the protocol's net position. When a fee-on-transfer token is used, the pull-back leg delivers only `assets[i] - fee_in` to the protocol, permanently reducing the protocol's balance by `fee_in` per call. No existing check in the function detects or prevents this shortfall.

### Finding Description

**Code path:**

`flashLoan` (lines 737–752, `src/Midnight.sol`):

```
line 743: SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
          // protocol balance: B → B - assets[i]
          // callback receives: assets[i] - fee_out  (fee deducted by token)

line 750: SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
          // callback debited: assets[i]
          // protocol receives: assets[i] - fee_in   (fee deducted by token)
          // protocol balance: B - assets[i] + (assets[i] - fee_in) = B - fee_in
```

`SafeTransferLib.safeTransfer` and `safeTransferFrom` only verify the boolean return value of the ERC20 call; they do not snapshot or compare balances. [1](#0-0) 

The `flashLoan` function has no `balanceBefore`/`balanceAfter` guard anywhere in its body. [2](#0-1) 

**Attacker-controlled inputs:**
- `tokens[i]`: any fee-on-transfer ERC20 the protocol already holds (e.g., deposited as collateral or loan token).
- `assets[i]`: any amount ≤ protocol balance.
- `callback`: attacker-controlled contract that (a) pre-funds itself with `fee_out` tokens so the repayment `transferFrom` does not revert, and (b) approves Midnight for `assets[i]`.

**Why existing checks fail:**
- `require(tokens.length == assets.length)` — only validates array lengths.
- `require(onFlashLoan(...) == CALLBACK_SUCCESS)` — only validates the callback's return value, not the token balance.
- `safeTransferFrom` succeeds as long as the callback has ≥ `assets[i]` tokens and has granted allowance; it does not verify how many tokens Midnight actually received.

**Net accounting per call:**

| Party | Δ balance |
|---|---|
| Protocol (Midnight) | **−fee_in** |
| Callback (attacker) | −fee_out |
| Fee sink (token contract) | +(fee_in + fee_out) |

The protocol's loss is `fee_in`; the attacker's loss is `fee_out`. The question's stated total of `fee_out + fee_in` is the combined burn, but the protocol-specific loss is `fee_in` per call.

### Impact Explanation

Every `flashLoan` call with a fee-on-transfer token permanently reduces `IERC20(token).balanceOf(address(midnight))` by `fee_in`, violating the invariant that the protocol's balance after a flash loan must be ≥ its balance before. Repeated calls drain the protocol's holdings of that token, reducing the assets available to cover collateral, credit redemption, and fees. [3](#0-2) 

### Likelihood Explanation

**Preconditions:**
1. The protocol holds a fee-on-transfer token (possible whenever such a token is used as a loan or collateral token in any market).
2. The attacker's callback pre-funds itself with `fee_out` tokens and approves Midnight for `assets[i]`.

**Feasibility:** Any unprivileged address can call `flashLoan`; no role or permission is required. [4](#0-3) 

**Repeatability:** The call can be repeated indefinitely. Each call costs the attacker `fee_out` tokens (a griefing cost), but drains the protocol by `fee_in` per iteration. If the attacker is indifferent to their own loss (e.g., a competitor or adversarial actor), the attack is fully repeatable until the protocol's balance of that token is exhausted.

### Recommendation

Record the protocol's token balance before the outbound transfer and assert it is restored after the inbound transfer:

```solidity
for (uint256 i = 0; i < tokens.length; i++) {
    uint256 balanceBefore = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
    // ... callback ...
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(
        IERC20(tokens[i]).balanceOf(address(this)) >= balanceBefore,
        InsufficientRepayment()
    );
}
```

This pattern is used by Morpho Blue and Aave v3 flash loan implementations and is the standard defense against fee-on-transfer token shortfalls.

contract FeeFlashLoanTest is Test, IFlashLoanCallback {

    FeeOnTransferToken token;

    function testFuzz_feeOnTransferDrainsProtocol(uint256 feeBps, uint256 assets) public {
        feeBps = bound(feeBps, 1, 500);   // 0.01% – 5%
        assets = bound(assets, 1e18, 1e24);

        token = new FeeOnTransferToken(feeBps);
        token.mint(address(midnight), assets);
        uint256 balBefore = token.balanceOf(address(midnight));

        // pre-fund callback so repayment transferFrom doesn't revert
        uint256 feeOut = assets * feeBps / 10000;
        token.mint(address(this), feeOut);
        token.approve(address(midnight), assets);

        address[] memory tokens = new address[](1);
        tokens[0] = address(token);
        uint256[] memory amounts = new uint256[](1);
        amounts[0] = assets;

        midnight.flashLoan(tokens, amounts, address(this), "");

        uint256 balAfter = token.balanceOf(address(midnight));

        assertGe(balAfter, balBefore, "protocol balance drained by fee_in");
        // Expected deficit: assets * feeBps / 10000 (fee_in)
    }

    function onFlashLoan(address, address[] memory, uint256[] memory amounts, bytes memory)

        // approve repayment; callback has (amounts[0] - fee_out) + fee_out pre-funded

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
