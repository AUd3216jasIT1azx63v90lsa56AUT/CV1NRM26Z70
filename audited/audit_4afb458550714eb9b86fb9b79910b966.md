Audit Report

## Title
`consumableUnits` called outside `try/catch` reverts entire bundle on out-of-range tick - (`File: src/periphery/MidnightBundles.sol`)

## Summary
All four bundle entry points in `MidnightBundles` call `ConsumableUnitsLib.consumableUnits()` unconditionally before the `try/catch` block that guards `IMidnight.take()`. When an offer has `maxUnits == 0` and `maxAssets > 0`, `consumableUnits` delegates to `TakeAmountsLib`, which calls `TickLib.tickToPrice(offer.tick)`. If `offer.tick > MAX_TICK (5820)`, `tickToPrice` reverts with `TickOutOfRange`, propagating through `consumableUnits` and out of the bundle function entirely, reverting the whole transaction instead of skipping the offer.

## Finding Description

All four bundle entry points (`buyWithUnitsTargetAndWithdrawCollateral`, `supplyCollateralAndSellWithUnitsTarget`, `buyWithAssetsTargetAndWithdrawCollateral`, `supplyCollateralAndSellWithAssetsTarget`) share the same pattern: `ConsumableUnitsLib.consumableUnits()` is evaluated as an argument to `min(...)` before the `try` keyword. [1](#0-0) 

The NatSpec at line 43 explicitly acknowledges this: `"Reverts if ConsumableUnitsLib reverts"`. [2](#0-1) 

`ConsumableUnitsLib.consumableUnits` branches on `offer.maxUnits`. When `maxUnits == 0` (i.e., the offer is asset-capped), it calls `TakeAmountsLib.buyerAssetsToUnits` or `TakeAmountsLib.sellerAssetsToUnits`: [3](#0-2) 

Both `TakeAmountsLib` functions call `TickLib.tickToPrice(offer.tick)` as their very first statement: [4](#0-3) [5](#0-4) 

`tickToPrice` unconditionally reverts for any tick exceeding `MAX_TICK = 5820`: [6](#0-5) 

The `Offer` struct is passed entirely as caller-supplied calldata. There is no on-chain validation of `offer.tick` before `consumableUnits` is invoked. The `require(offer.tick % _marketState.tickSpacing == 0, TickNotAccessible())` check in `Midnight.take()` is inside the `try` block and does not protect `consumableUnits`.

**Exploit flow:**
1. Attacker (maker) signs an offer off-chain with `offer.tick = 5821` (or any value > 5820) and `offer.maxAssets > 0` (ensuring `offer.maxUnits == 0`).
2. The poisoned offer is included in a taker's `takes[]` array alongside legitimate offers.
3. When the loop reaches the poisoned offer, `consumableUnits` → `buyerAssetsToUnits`/`sellerAssetsToUnits` → `tickToPrice(5821)` reverts with `TickOutOfRange`.
4. The revert propagates outside the `try/catch`, reverting the entire bundle transaction.

## Impact Explanation
Complete DoS of the bundle transaction: all legitimate takes in the bundle fail. In time-sensitive contexts (near maturity, liquidation windows, volatile markets), the taker suffers real financial loss from failed execution. The attack is repeatable at zero cost to the attacker (off-chain signing requires no gas, no on-chain state, no privileged access).

## Likelihood Explanation
Any maker can sign an offer with an out-of-range tick at zero cost. Takers using automated routing or aggregation systems that fetch offers from an off-chain order book are the primary target. The only precondition is that the poisoned offer appears in the `takes` array with `maxAssets > 0`. The attack is trivially repeatable and requires no special market state or on-chain setup.

## Recommendation
Wrap the `consumableUnits` call inside the `try/catch` block, or add a tick-range guard before calling `consumableUnits`. The simplest fix is to move the `unitsToTake` computation inside the `try` block (or use a separate `try/catch` around `consumableUnits` itself) so that a revert from an invalid tick causes the offer to be skipped rather than the entire bundle to revert. Alternatively, validate `offer.tick <= MAX_TICK` before calling `consumableUnits` and skip the offer if the check fails.

## Proof of Concept
1. Deploy `MidnightBundles` pointing to a valid `Midnight` instance.
2. Construct a `Take` with `offer.tick = 5821`, `offer.maxUnits = 0`, `offer.maxAssets = 1e18`, and any valid `offer.market`.
3. Include this `Take` alongside one or more valid takes in a call to `buyWithUnitsTargetAndWithdrawCollateral`.
4. Observe the transaction reverts with `TickOutOfRange` rather than skipping the poisoned offer and proceeding with the valid ones.

A fuzz test can confirm by fuzzing `offer.tick` over `[MAX_TICK+1, type(uint256).max]` with `offer.maxUnits = 0` and `offer.maxAssets > 0`, asserting that the bundle always reverts rather than skipping the offer.
