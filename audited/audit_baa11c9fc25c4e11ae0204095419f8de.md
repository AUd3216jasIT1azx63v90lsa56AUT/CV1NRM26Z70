### Title
Arbitrary Third-Party Payer Designation in `liquidate()` Allows Attacker to Drain Victim Callback Contract - (File: src/Midnight.sol)

### Summary
In `liquidate()`, the `payer` for the loan token repayment is unconditionally set to `callback` when a callback address is provided, with no check that `msg.sender` controls or has been authorized by that callback contract. An attacker can designate any third-party contract that implements `onLiquidate()` returning `CALLBACK_SUCCESS` and has pre-approved Midnight as the payer, receiving seized collateral at zero personal cost while the victim contract's loan tokens are drained.

### Finding Description

**Code path:**

At [1](#0-0)  the payer is set unconditionally:

```solidity
address payer = callback != address(0) ? callback : msg.sender;
```

There is no check that `callback == msg.sender`, that `msg.sender` is authorized by `callback`, or that `callback` is the liquidator's own contract. The full sequence is:

1. [2](#0-1)  — borrower's collateral is decremented and `_marketState.withdrawable` is incremented (state committed before any external call).
2. [3](#0-2)  — seized collateral is transferred to `receiver` (attacker-controlled address).
3. [4](#0-3)  — `ILiquidateCallback(callback).onLiquidate(...)` is called on the victim contract; if it returns `CALLBACK_SUCCESS`, execution continues.
4. [5](#0-4)  — `safeTransferFrom(market.loanToken, payer, address(this), repaidUnits)` pulls loan tokens from `payer = callback = victimContract`.

**Attacker inputs:**
- `callback = victimContract` (any contract implementing `ILiquidateCallback` that returns `CALLBACK_SUCCESS`)
- `receiver = attacker` (receives seized collateral)
- `repaidUnits` = desired amount to drain from victim

**Existing checks that do NOT stop this:**
- The `liquidatorGate` check at [6](#0-5)  only gates `msg.sender`, not the callback.
- The `NotLiquidatable` check at [7](#0-6)  only verifies the borrower's position is unhealthy — a legitimate precondition the attacker can satisfy by using any real unhealthy position.
- The Certora spec `OnlyExplicitPayerCanLoseTokens.spec` at [8](#0-7)  proves tokens are only pulled from "the callback that returned `CALLBACK_SUCCESS`" — but this invariant does not verify that the callback is the caller's own contract, so it does not prevent this attack.

Compare with `repay()`, which has an authorization guard at [9](#0-8)  (`require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`). No equivalent guard exists in `liquidate()` for the callback/payer relationship.

**Exploit flow:**
1. Identify `victimContract`: a deployed liquidation callback contract that (a) implements `onLiquidate()` returning `CALLBACK_SUCCESS` without reverting when called unexpectedly, (b) has approved Midnight for `loanToken`, and (c) holds `repaidUnits` of `loanToken`.
2. Find or create an unhealthy borrower position in any valid market.
3. Call `liquidate(market, collateralIndex, 0, repaidUnits, borrower, false, receiver=attacker, callback=victimContract, data)`.
4. Midnight sets `payer = victimContract`, sends seized collateral to `attacker`, calls `victimContract.onLiquidate()` (returns `CALLBACK_SUCCESS`), then pulls `repaidUnits` of `loanToken` from `victimContract`.
5. Attacker receives seized collateral; `victimContract` loses `repaidUnits` of loan tokens; accounting is internally consistent (Midnight's books balance), but victim's funds are stolen.

### Impact Explanation
A victim liquidation callback contract that has approved Midnight and holds loan token balance loses those tokens to an attacker who receives seized collateral at zero personal cost. The attacker profits by `seizedAssets` (collateral value at liquidation incentive) while paying nothing. The protocol's accounting remains internally consistent (withdrawable increases, debt decreases), but user funds are stolen from the victim contract — violating the invariant that "callbacks, ERC20 transfers, multicall, or reentrancy cannot corrupt partial state" and that "user funds must not be stolen."

### Likelihood Explanation
**Preconditions:**
1. An unhealthy borrower position must exist (common in any active market during price drops).
2. `victimContract` must implement `onLiquidate()` returning `CALLBACK_SUCCESS` without reverting when called by an unexpected liquidation — realistic for any liquidation bot with a permissive callback implementation.
3. `victimContract` must have approved Midnight for `loanToken` — standard for any liquidation infrastructure contract.
4. `victimContract` must hold `repaidUnits` of `loanToken` — realistic for bots that pre-fund their callback contracts.

Any liquidation bot operator who deploys a callback contract with a simple `onLiquidate` implementation (e.g., one that just returns `CALLBACK_SUCCESS` after doing its swap logic, without checking that `receiver == address(this)`) is vulnerable. The attack is repeatable and permissionless.

### Recommendation
Add a check that `msg.sender` controls the callback, either by requiring `callback == msg.sender`, or by requiring that the callback has explicitly authorized `msg.sender` via the existing `isAuthorized` mapping:

```solidity
require(
    callback == address(0) || callback == msg.sender || isAuthorized[callback][msg.sender],
    Unauthorized()
);
```

This mirrors the authorization pattern already used in `repay()` and `supplyCollateral()` and ensures that only the liquidator's own callback contract can be designated as payer.
