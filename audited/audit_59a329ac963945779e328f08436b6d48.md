### Title
Assets-mode buy offer cap bypassed via zero-rounding when `buyerPrice < WAD` - (`src/Midnight.sol`)

### Summary

When a buy offer uses `maxAssets > 0` as its cap, the `consumed` mapping is incremented by `buyerAssets = mulDivDown(units, buyerPrice, WAD)`. If `buyerPrice < WAD` and `units` is small enough that `units * buyerPrice < WAD`, this floors to zero, so `consumed` never advances. The `ConsumedAssets` guard then never triggers, and the offer can be filled an unlimited number of times. The protocol's own test suite names this `testBugBuyMaxAssetsBypass` and the NatDoc at line 94 explicitly acknowledges the edge case, confirming the path is reachable and the impact is real.

### Finding Description

**Exact code path — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // floor(units * buyerPrice / WAD)
    : units.mulDivUp(buyerPrice, WAD);

uint256 newConsumed;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // guard
}
```

**Root cause:** For a buy offer (`offer.buy == true`) with `maxAssets > 0`, the consumed delta is `buyerAssets`. When `buyerPrice < WAD` and `units = 1`, `mulDivDown(1, buyerPrice, WAD) = 0`. The guard `newConsumed <= maxAssets` is trivially satisfied because `newConsumed` never changes.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets = M > 0`, `offer.maxUnits = 0`
- `offer.tick` chosen so that `buyerPrice = offerPrice - settlementFee + settlementFee < WAD` (any tick below `MAX_TICK` satisfies this; the test uses `MAX_TICK - 16`)
- `units = 1` per call

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = M`, `tick` such that `buyerPrice < WAD`.
2. Taker calls `take(offer, ..., units=1, ...)` N times.
3. Each call: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; `consumed` stays at 0; guard passes; `buyerPos.credit += 1` (line 410).
4. After N calls: `consumed[maker][group] == 0`, maker's credit has grown by N, taker's debt has grown by N, with zero asset transfer (`buyerAssets - sellerAssets = 0`, line 455).

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — `newConsumed` is always 0, so this never fires regardless of `maxAssets`.
- `require(UtilsLib.atMostOneNonZero(offer.maxAssets, offer.maxUnits))` — satisfied; only `maxAssets` is set.
- No minimum-units or minimum-assets guard exists.

The NatDoc at `src/Midnight.sol` line 94 states: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."* This confirms the path is known and unguarded. The test `testBugBuyMaxAssetsBypass` (lines 858–889) proves the state change occurs even when `consumed` is already at `maxAssets`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The `maxAssets` cap on a buy offer is completely ineffective whenever `buyerPrice < WAD` and the taker submits `units` small enough that `units * buyerPrice < WAD`. An unprivileged taker can:

- Fill the offer an unlimited number of times, each time increasing the maker's credit and the taker's debt by 1 unit at zero token cost (`buyerAssets = sellerAssets = 0`).
- Inflate `totalUnits` and the maker's credit position without bound, violating the invariant that `consumed` must track and cap total fill activity.
- Effectively grief the maker by forcing unbounded debt onto the taker-controlled counterparty, or exploit the maker's credit position in downstream interactions. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

**Preconditions:**
1. A buy offer exists with `maxAssets > 0` and `tick` such that `buyerPrice < WAD`. Any tick strictly below `MAX_TICK` with zero settlement fee satisfies this; with nonzero settlement fee the window is even wider.
2. The taker is any unprivileged address (no special role required).
3. The taker must supply collateral for the debt they take on (or have an existing credit position to absorb it), but the token cost per fill is zero.

**Feasibility:** Trivially repeatable. The only cost is gas. The condition `buyerPrice < WAD` is the normal operating range for most ticks (all ticks below `MAX_TICK`). No oracle manipulation or admin action is required. [6](#0-5) 

### Recommendation

When `offer.maxAssets > 0` and `offer.buy == true`, revert if `units > 0` but `buyerAssets == 0`:

```solidity
if (offer.maxAssets > 0) {
    // Prevent zero-rounding bypass: a non-trivial fill must advance consumed.
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
    newConsumed = consumed[offer.maker][offer.group] += buyerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, track consumed in units (not assets) even in `maxAssets` mode, converting the cap via `mulDivUp(maxAssets, WAD, buyerPrice)` to a unit ceiling. This is consistent with how `ConsumableUnitsLib.buyerAssetsToUnits` already inverts the relationship. [7](#0-6) 

### Proof of Concept

```solidity
// Foundry unit test
function testConsumedBypassUnlimitedFill() public {
    // Setup: buy offer, buyerPrice < WAD (tick below MAX_TICK, zero settlement fee)
    for (uint256 i; i <= 6; i++) midnight.setMarketSettlementFee(id, i, 0);

    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1e18;          // finite cap
    lenderOffer.tick = MAX_TICK - 16;      // offerPrice < WAD → buyerAssets = 0 for units=1

    uint256 N = 500;
    collateralize(market, borrower, N);    // borrower supplies collateral for N units of debt
    deal(address(loanToken), lender, 0);   // lender pays nothing

    uint256 creditBefore = midnight.creditOf(id, lender);

    for (uint256 i; i < N; i++) {
        take(1, borrower, lenderOffer);    // each fill: buyerAssets=0, consumed unchanged
    }

    // Assertions
    assertEq(midnight.consumed(lender, lenderOffer.group), 0,
        "consumed must be 0: cap never advanced");
    assertEq(midnight.creditOf(id, lender), creditBefore + N,
        "maker credit grew by N despite consumed==0");
    assertEq(midnight.debtOf(id, borrower), N,
        "taker debt grew by N");
    // The offer's maxAssets cap was completely bypassed
    assertTrue(N > lenderOffer.maxAssets,
        "filled more units than maxAssets would allow");
}
```

**Expected assertions:** all pass, demonstrating that `consumed` stays at 0 while the maker accumulates N units of credit, violating the `maxAssets` cap invariant. [8](#0-7) [9](#0-8)
