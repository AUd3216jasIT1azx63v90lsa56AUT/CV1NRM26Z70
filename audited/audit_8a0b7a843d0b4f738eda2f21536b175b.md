Audit Report

## Title
Off-by-one `<=` in maturity debt-increase guard allows forced debt creation at exact maturity timestamp - (File: `src/Midnight.sol`)

## Summary
The `take()` function guards against post-maturity debt increases using `block.timestamp <= offer.market.maturity` (line 391), which permits `sellerDebtIncrease > 0` when `block.timestamp == offer.market.maturity`. The post-maturity liquidation gate uses the strict `block.timestamp > market.maturity` (line 622), so at the exact maturity second the borrower's position can have debt forced onto it but cannot yet be liquidated in post-maturity mode. One block later, the position is liquidatable via post-maturity mode even though it is collateral-healthy, causing the borrower to lose collateral to a liquidator.

## Finding Description

**Root cause — asymmetric boundary operators:**

`take()`, line 391:
```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [1](#0-0) 

When `block.timestamp == offer.market.maturity` the left operand is `true`, so the `require` passes unconditionally regardless of `sellerDebtIncrease`. Debt is then written to storage:
```solidity
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);
``` [2](#0-1) 

The end-of-take health check (line 476) only enforces collateral coverage, not maturity:
```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
``` [3](#0-2) 

`isHealthy` returns `maxDebt >= debt` — purely a collateral-coverage check with no maturity component. [4](#0-3) 

`liquidate()`, line 622, uses strict `>`:
```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
``` [5](#0-4) 

At `block.timestamp == maturity` this evaluates to `false`, so `NotLiquidatable()` reverts. At `block.timestamp == maturity + 1` it evaluates to `true`.

**Exploit flow:**
1. A sell offer exists with `expiry >= maturity` and the seller has no existing credit to absorb units (so `sellerDebtIncrease > 0`).
2. Taker waits until `block.timestamp == offer.market.maturity`.
3. Taker calls `take()` with `units` such that `sellerDebtIncrease > 0`.
4. Line 391: `block.timestamp <= maturity` → `true` → passes. Seller's debt increases. Health check at line 476 passes (seller is collateral-healthy).
5. In the next block (`block.timestamp == maturity + 1`), taker calls `liquidate(..., postMaturityMode=true)`.
6. Line 622: `block.timestamp > market.maturity` → `true` → liquidation proceeds on a collateral-healthy position.
7. LIF at `maturity + 1` = `WAD + (maxLif - WAD) * 1 / TIME_TO_MAX_LIF` (just above 1, ramping over 15 minutes). [6](#0-5) 

**Why existing checks fail:**
- `CannotIncreaseDebtPostMaturity` uses `<=` instead of `<`, creating a one-second window.
- The `SellerIsLiquidatable` check at line 476 only enforces collateral health, not maturity-based liquidatability.
- The protocol's own invariant document explicitly lists this as forbidden: `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"`. [7](#0-6) 

The `credit_debt` invariant also states `"debt must not increase after maturity"`. [8](#0-7) 

## Impact Explanation

A taker can force a collateral-healthy borrower to take on new debt at the exact maturity second. In the immediately following block, the borrower's position is liquidatable via post-maturity mode even though it is collateral-healthy (`live_context.json` confirms: `"overdue debt after maturity is liquidatable even if healthy"`). [9](#0-8) 

The borrower has no opportunity to repay between debt creation and liquidation (consecutive blocks). The liquidator seizes collateral at LIF > 1 (even at `maturity + 1`, LIF = `WAD + (maxLif - WAD)/900`), causing direct loss of collateral value beyond the debt repaid. This matches the `best_bug_classes` entry `"healthy-account liquidation"` and `"unauthorized collateral seizure"`. [10](#0-9) 

## Likelihood Explanation

**Preconditions:**
- A sell offer with `expiry >= maturity` exists (common for offers meant to be filled up to maturity).
- The seller has no existing credit to absorb the units (`sellerCreditDecrease < units`).
- The seller has sufficient collateral to pass the health check after the debt increase.

**Feasibility:** On chains with 1-second block times (post-merge Ethereum, Arbitrum, Base), landing a transaction at exactly `block.timestamp == maturity` is achievable via MEV or simple block monitoring. The attacker (taker) controls the timing. The attack is repeatable for any market where a qualifying sell offer exists. No privileged access is required — the attacker is a normal unprivileged taker.

## Recommendation

Change `<=` to `<` on line 391 of `src/Midnight.sol`:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [1](#0-0) 

This aligns the debt-increase guard with the post-maturity liquidation gate (which uses strict `>`), closing the one-second window and satisfying the protocol's stated invariant that timestamp equality must not permit debt increase.

## Proof of Concept

Minimal Foundry test outline:

```solidity
// 1. Setup: seller has collateral, no existing credit. Sell offer with expiry == maturity.
// 2. vm.warp(market.maturity);
// 3. taker calls take() with units s.t. sellerDebtIncrease > 0
//    → assert: take() succeeds, seller.debt > 0
// 4. vm.warp(market.maturity + 1);
// 5. taker calls liquidate(market, 0, 0, repaid, seller, true, ...)
//    → assert: liquidate() succeeds despite seller being collateral-healthy
//    → assert: seller.collateral decreased, taker received collateral
```

The existing test suite already warps to `market.maturity + TIME_TO_MAX_LIF + delay` for post-maturity liquidation tests. [11](#0-10) 

A targeted test at `vm.warp(market.maturity)` for the `take()` step and `vm.warp(market.maturity + 1)` for the `liquidate()` step would reproduce the exploit deterministically.

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];
                uint256 price = IOracle(collateralParam.oracle).price();
                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                    .mulDivDown(collateralParam.lltv, WAD);
                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
