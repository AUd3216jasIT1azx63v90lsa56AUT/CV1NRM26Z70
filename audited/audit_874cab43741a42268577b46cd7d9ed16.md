Audit Report

## Title
Missing zero-amount guard before `safeTransfer` in `liquidate()` blocks bad-debt-only liquidations for tokens that revert on zero transfers - (`File: src/Midnight.sol`)

## Summary
`liquidate()` unconditionally calls `SafeTransferLib.safeTransfer(collateralToken, receiver, seizedAssets)` at line 696 even when `seizedAssets = 0`, directly contradicting the NatSpec at line 577 which guarantees "Passing both 0 for seizedAssets and repaidUnits allows to realize bad debt with 0 token transferred." For any market whose collateral token reverts on zero-amount transfers, the bad-debt-only liquidation path is permanently DoS'd, preventing `lossFactor` from ever being updated and leaving bad debt unrealized.

## Finding Description

The NatSpec at line 577 explicitly documents the zero-zero path as valid: [1](#0-0) 

The guard at line 595 permits both inputs being zero (`atMostOneNonZero` returns true when both are zero): [2](#0-1) 

The block at line 643 is skipped entirely when both inputs are zero, leaving `seizedAssets = 0`: [3](#0-2) 

Execution then falls through unconditionally to: [4](#0-3) 

`SafeTransferLib.safeTransfer` has no zero-amount guard — it always issues the low-level `token.call`: [5](#0-4) 

If the collateral token reverts on a zero-amount `transfer()`, the entire transaction reverts, rolling back the bad-debt accounting at lines 626–641 (the `_position.debt` reduction and `_marketState.lossFactor` update): [6](#0-5) 

The Certora rule `liquidateLossFactorDoesNotRevert` at line 104 of `LossFactor.spec` does not catch this because `SafeTransferLib.safeTransfer` is summarized as `NONDET` in the Certora specs (as seen in `EmptyOffer.spec` line 11), abstracting away the external ERC20 call entirely: [7](#0-6) 

Note: line 717 has the same structural issue for `safeTransferFrom(loanToken, payer, address(this), repaidUnits)` when `repaidUnits = 0`, though loan tokens that revert on zero transfers are less common. [8](#0-7) 

## Impact Explanation

Bad-debt-only liquidations are permanently blocked for any market whose collateral token reverts on zero-amount transfers. The `_marketState.lossFactor` is never updated, lenders cannot recover proportional losses through the loss socialization mechanism, and the borrower's debt is never reduced. This violates the protocol's own documented guarantee and the core accounting invariant. The impact is a permanent freeze of the bad-debt realization path — a concrete, non-hypothetical loss of protocol functionality.

## Likelihood Explanation

The market creator role is unprivileged. A market can be created with any ERC20 as collateral, including tokens that revert on zero-amount transfers (a known real-world pattern present in several deployed tokens). Once such a market exists and any borrower's position reaches `badDebt > 0`, every liquidation attempt on the bad-debt-only path reverts. The DoS is repeatable, permanent (the collateral token cannot be changed post-deployment), and requires no special timing, oracle manipulation, or victim mistakes.

## Recommendation

Add a zero-amount guard before the `safeTransfer` call at line 696:

```solidity
if (seizedAssets > 0) {
    SafeTransferLib.safeTransfer(market.collateralParams[collateralIndex].token, receiver, seizedAssets);
}
```

Apply the same guard to the `safeTransferFrom` call at line 717:

```solidity
if (repaidUnits > 0) {
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), repaidUnits);
}
```

## Proof of Concept

1. Deploy a collateral token that reverts on zero-amount `transfer()` calls.
2. Create a market with this token as `collateralParams[0].token`.
3. Have a borrower supply collateral and borrow; let the collateral price drop so `badDebt > 0` (e.g., set collateral price to 0 via oracle).
4. Call `liquidate(market, 0, 0, 0, borrower, false, receiver, address(0), "")`.
5. Observe: the transaction reverts at line 696 due to the zero-amount transfer revert from the collateral token.
6. Confirm: `_marketState.lossFactor` is unchanged and `_position.debt` is unchanged — bad debt is never realized.

A minimal Foundry test can implement this with a mock ERC20 that adds `require(value > 0)` to its `transfer()` function, using the existing test infrastructure in the repository.

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
