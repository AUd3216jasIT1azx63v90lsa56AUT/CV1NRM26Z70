### Title
Fee-on-Transfer Token Inflates `claimableSettlementFee` Relative to Actual Protocol Receipts - (File: src/Midnight.sol / src/libraries/SafeTransferLib.sol)

### Summary

In `Midnight.sol`'s `take` function, `claimableSettlementFee` is incremented by the nominal `buyerAssets - sellerAssets` at line 418 before any external call. The subsequent `SafeTransferLib.safeTransferFrom` at line 455 only verifies that `transferFrom` does not revert and returns `true`; it performs no balance-before/balance-after check. When the market's `loanToken` is a fee-on-transfer token and the `payer` is a `buyerCallback` contract, the contract receives strictly less than the nominal amount, permanently inflating `claimableSettlementFee` relative to actual token holdings.

   ...
   require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
   ``` [5](#0-4) 

**Root cause:** `SafeTransferLib.safeTransferFrom` does not verify the actual amount received by `address(this)`. A fee-on-transfer token deducts a fee silently while returning `true`, so the call succeeds but the contract receives `(buyerAssets - sellerAssets) * (1 - fee_rate)` instead of the full nominal amount. Because `claimableSettlementFee` was already incremented by the full nominal amount at line 418, the accounting is permanently wrong.

**Attacker-controlled inputs:**
- For a sell offer (`offer.buy = false`): `buyerCallback = takerCallback` (line 420), which is a taker-supplied parameter. The taker deploys a contract implementing `IBuyCallback` that holds fee-on-transfer tokens and approves Midnight in `onBuy`. The taker passes this contract as `takerCallback`.
- The market's `loanToken` must be a fee-on-transfer token (set at market creation; no protocol restriction prevents this).

**Why existing checks fail:** The only checks on the transfer are revert-propagation and the boolean return value. There is no `balanceOf(address(this))` snapshot before/after the transfer. The `claimableSettlementFee` increment at line 418 is unconditional and occurs before the transfer.

### Impact Explanation

After each such `take`, `claimableSettlementFee[loanToken]` exceeds the contract's actual token balance attributable to settlement fees by `(buyerAssets - sellerAssets) * fee_rate`. When the protocol owner calls `claimSettlementFee`, it will attempt to transfer more tokens than the contract holds for that purpose, either reverting (DoS on fee claiming) or — if other token sources (withdrawable assets, collateral) are co-mingled in the same ERC-20 balance — silently draining lender-withdrawable funds to cover the shortfall. The core invariant "contract balances cover collateral, credit redemption, fees, and withdrawable assets" is violated.

### Likelihood Explanation

**Preconditions:**
1. A market exists whose `loanToken` is a fee-on-transfer token (e.g., USDT with fee enabled, or any custom ERC-20 with a transfer tax). No protocol check prevents this.
2. A taker submits a sell offer take with `takerCallback` pointing to a contract that implements `IBuyCallback`, holds the fee-on-transfer token, and approves Midnight in `onBuy`.

Both preconditions are fully attacker-controllable (market creation is permissionless; `takerCallback` is a free parameter). The attack is repeatable on every `take` call in such a market, compounding the discrepancy.

### Recommendation

After the `safeTransferFrom` call at line 455, verify the actual amount received using a balance snapshot:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
require(
    IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore == buyerAssets - sellerAssets,
    UnexpectedTokenFee()
);
```

Alternatively, explicitly document and enforce (via market creation validation) that fee-on-transfer tokens are not supported as `loanToken`.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
