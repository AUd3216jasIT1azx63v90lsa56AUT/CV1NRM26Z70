### Title
Authorized-signer front-run advances victim's nonce and invalidates pending signed Authorization - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts a signature from any address already authorized by `authorization.authorizer` via the `isAuthorized` delegation path. Because the signer is not constrained to sign only the struct the authorizer intended, an already-authorized attacker can craft a different `Authorization` struct (same `authorizer`, same `nonce`, different `authorized`/`isAuthorized` fields), sign it with their own key, and submit it before the victim's pending tx. The nonce increments on the attacker's call, causing the victim's tx to revert with `InvalidNonce()`.

The `Authorization` struct contains five fields: `authorizer`, `authorized`, `isAuthorized`, `nonce`, `deadline`. [3](#0-2) 

The signature check at lines 33–36 only verifies that the recovered signer is either the `authorizer` themselves **or** any address that `authorizer` has previously authorized in Midnight. It does **not** verify that the signer is constrained to sign only the struct the authorizer intended — the signer is free to sign any struct with any `authorized` and `isAuthorized` values.

**Attacker-controlled inputs:**
- `authorization.authorized` — arbitrary address (attacker-chosen)
- `authorization.isAuthorized` — arbitrary bool
- `authorization.nonce` — set to victim's current nonce N
- `authorization.deadline` — any valid future timestamp
- `signature` — attacker's own valid ECDSA signature over the crafted struct

**Exploit flow:**
1. Precondition: `isAuthorized[victim][attacker] == true` (attacker was previously authorized by victim via `Midnight.setIsAuthorized`)
2. Victim signs `Authorization{authorizer=victim, authorized=X, isAuthorized=true, nonce=N}` and broadcasts
3. Attacker observes victim's tx in mempool; crafts `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N}` (same nonce, different `authorized`)
4. Attacker signs this struct with their own private key → `ecrecover(digest, ...) == attacker`
5. Attacker submits with higher gas priority
6. Contract execution:
   - Line 25: deadline check passes
   - Line 26: `N == nonce[victim]` → passes; nonce increments to N+1
   - Line 31: `signer = attacker`
   - Lines 33–36: `isAuthorized[victim][attacker] == true` → passes
   - Line 47: `Midnight.setIsAuthorized(attacker, true, victim)` executes (attacker grants themselves authorization, or any other address)
7. Victim's tx arrives: `N != N+1` → reverts with `InvalidNonce()`

**Why existing checks fail:**
- The nonce check only prevents replay of the exact same signed message; it does not prevent a different message at the same nonce from a delegated signer.
- The `isAuthorized` delegation path was designed to allow authorized agents to act on behalf of the authorizer, but it does not scope what struct content the agent may sign — the agent can sign any `authorized`/`isAuthorized` combination.
- There is no binding between the signer's identity and the `authorized` field in the struct.

### Impact Explanation
The victim's pending signed `Authorization` is permanently invalidated (DoS of delegation). The victim must sign a new message at nonce N+1, which the attacker can front-run again indefinitely. Additionally, the attacker can set arbitrary `isAuthorized` entries on behalf of the victim (e.g., authorize a malicious address or revoke existing authorizations), which can block the victim's ability to use any signature-gated delegation path through `EcrecoverAuthorizer` for as long as the attacker remains authorized.

### Likelihood Explanation
**Preconditions:** The attacker must already hold `isAuthorized[victim][attacker] == true`. This is a common state — users routinely authorize operators, bots, or relayers. Once this precondition is met, the attack is trivially repeatable on every future signed authorization the victim broadcasts. The attacker needs only to monitor the mempool and submit a competing tx with higher gas, which is standard MEV infrastructure.

### Recommendation
Remove the `isAuthorized` delegation path from `EcrecoverAuthorizer.setIsAuthorized` entirely. Only the `authorizer` themselves should be permitted to sign an `Authorization` struct, since the struct's `authorized` and `isAuthorized` fields represent the authorizer's explicit intent about who to grant or revoke access to. The delegation path creates an asymmetry: the authorizer signs one thing, but a delegated signer can substitute different content at the same nonce.

```solidity
// Replace lines 33-36 with:
require(signer == authorization.authorizer, Unauthorized());
```

If delegation is a desired feature, scope it by requiring the signer to only be able to sign structs where `authorization.authorized == signer` (i.e., a delegated signer may only grant/revoke their own authorization, not arbitrary third parties).

### Proof of Concept
```solidity
function testFrontRunNonceAdvance() public {
    address victim  = borrower;   // has privateKey[borrower]
    address attacker = lender;    // has privateKey[lender]
    address intendedTarget = makeAddr("X");

    // Setup: victim authorizes ecrecoverAuthorizer and attacker in Midnight
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, true, victim);  // precondition

    // Victim signs Authorization{authorizer=victim, authorized=intendedTarget, nonce=0}
    Authorization memory victimAuth = Authorization({
        authorizer: victim,
        authorized: intendedTarget,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory victimSig = signAuthorization(victimAuth, victim);

    // Attacker crafts different struct with same nonce, signs with attacker key
    Authorization memory attackerAuth = Authorization({
        authorizer: victim,
        authorized: attacker,   // different authorized address
        isAuthorized: true,
        nonce: 0,               // same nonce
        deadline: block.timestamp + 1 days
    });
    Signature memory attackerSig = signAuthorization(attackerAuth, attacker);

    // Attacker front-runs
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(attackerAuth, attackerSig);

    // Assert: nonce advanced, attacker's chosen authorization applied
    assertEq(ecrecoverAuthorizer.nonce(victim), 1);
    assertEq(midnight.isAuthorized(victim, attacker), true);
    assertEq(midnight.isAuthorized(victim, intendedTarget), false); // victim's intent never applied

    // Assert: victim's tx now reverts
    vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
    ecrecoverAuthorizer.setIsAuthorized(victimAuth, victimSig);
}
```

Expected assertions: nonce is 1 after attacker's call; `isAuthorized[victim][intendedTarget]` remains false; victim's tx reverts with `InvalidNonce`.
