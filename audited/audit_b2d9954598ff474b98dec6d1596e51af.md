Audit Report

## Title
Partial `withdraw(units = credit - 1)` zeroes `pendingFee` on remaining credit via `mulDivUp` rounding, enabling full continuous-fee evasion - (File: src/Midnight.sol)

## Summary

In `Midnight.withdraw`, `pendingFeeDecrease` is computed as `mulDivUp(pendingFee, units, credit)`. When `units = credit - 1` and `pendingFee < credit` — a condition guaranteed in normal operation by the `pendingContinuousFeeBoundedByCredit` invariant — ceiling rounding causes the entire `pendingFee` to be attributed to the withdrawn portion. The remaining 1-unit position is left with `pendingFee = 0`, permanently eliminating its fee obligation and allowing the lender to recover the full fee amount the protocol was owed.

## Finding Description

**Exact code path:**

`src/Midnight.sol:489-493`:
```solidity
if (_position.credit > 0) {
    pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
    _position.pendingFee -= pendingFeeDecrease;
}
_position.credit -= UtilsLib.toUint128(units);
```

`src/libraries/UtilsLib.sol:34-35`:
```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
```

**Arithmetic proof (verified against actual code):**

Let `C = credit`, `F = pendingFee`, `units = C - 1`.

```
mulDivUp(F, C-1, C) = floor((F*(C-1) + (C-1)) / C)
                    = floor((C-1)*(F+1) / C)
                    = floor((CF + C - F - 1) / C)
                    = F + floor((C - F - 1) / C)
```

When `F < C`: `0 ≤ C - F - 1 < C`, so `floor((C - F - 1) / C) = 0`, giving `mulDivUp(F, C-1, C) = F`.

The `pendingContinuousFeeBoundedByCredit` invariant at `certora/specs/Midnight.spec:137-138` guarantees `F ≤ C`. The strict inequality `F < C` holds in all normal operating states (any lender who entered before maturity with a non-zero fee rate). Therefore `pendingFeeDecrease = F` whenever `units = C - 1` and `F < C`.

**State after the call:**
- `_position.pendingFee = F - F = 0`
- `_position.credit = C - (C-1) = 1`

The remaining 1-unit position carries zero fee obligation. In `_updatePosition` → `updatePositionView` (`src/Midnight.sol:814-816`), `fee = postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual)`. With `postSlashPendingFee = 0`, `fee = 0` always, so `continuousFeeCredit` is never incremented for this position.

Critically, the `pendingFeeDecrease` in `withdraw` is **not** added to `continuousFeeCredit` — it is simply removed from the position's pending fee. The protocol permanently loses the fee that would have accrued on the remaining credit.

**Exploit flow:**

1. Lender has `credit = C`, `pendingFee = F > 0`, `F < C` (normal operating state).
2. Borrower repays so `withdrawable >= C - 1`.
3. Lender calls `withdraw(market, C-1, lender, lender)`. Authorization check at line 482 passes trivially since `msg.sender == onBehalf`.
4. `pendingFeeDecrease = mulDivUp(F, C-1, C) = F` — the full `pendingFee` is cancelled.
5. State: `credit = 1`, `pendingFee = 0`.
6. No fee ever accrues on the remaining unit. Lender later calls `withdraw(market, 1, lender, lender)`.
7. Total `loanToken` received: `(C-1) + 1 = C`. Without the exploit, the lender would receive only `C - F` after the continuous fee deducts `F` from credit over time.

**Why existing checks do not stop it:**

- The Certora rule `pendingFeeDecreasesProportionallyOnWithdraw` at `certora/specs/ContinuousFee.spec:94-107` encodes the `mulDivUp` formula as the *expected* behaviour. It verifies the code is self-consistent but does not assert that the fee rate on the remaining credit is non-decreasing. The rule passes for the exploiting input.
- The fuzz test `testWithdrawReducesPendingFee` at `test/ContinuousFeeTest.sol:337-338` computes `pendingFeeDecrease` using the same `mulDivUp` formula, so it passes for the exploiting input and does not catch the invariant violation.
- The `withdrawEffects` rule at `certora/specs/BalanceEffects.spec:77` only asserts `continuousFeeCredit(id) == feeAmountBefore + userFee` where `userFee` comes from `_updatePosition` — it does not check that the remaining `pendingFee` is correctly proportional.
- There is no minimum-remaining-credit guard and no floor on the post-withdrawal fee rate.

## Impact Explanation

The lender recovers `F` extra units of `loanToken` — the full continuous fee the protocol was owed. `continuousFeeCredit` for the market is never incremented by the fee on the remaining credit, so the fee claimer cannot claim it. For a lender with `credit = 1e18` and a 10% annual fee rate at 1-year TTM, `F ≈ 1e17` tokens are permanently lost to the protocol per position. The exploit is repeatable across every position where `pendingFee > 0` and `credit > 1`, constituting direct, unbounded theft of protocol fee revenue. This is a direct theft impact, in scope per SECURITY.md and RESEARCHER.md.

## Likelihood Explanation

**Preconditions:** (a) lender has `credit > 1` and `pendingFee > 0` — satisfied for any lender who entered before maturity with a non-zero fee rate; (b) `withdrawable >= credit - 1` — satisfied once the borrower repays. Both are routine operating conditions. **Attacker:** the lender themselves; no privileged role, no oracle manipulation, no signature forgery required. **Repeatability:** every lender position is independently exploitable with a single `withdraw` call.

## Recommendation

Replace the direct computation of `pendingFeeDecrease` with a computation that first determines the remaining position's fee (using ceiling rounding to favour the protocol) and derives the decrease from that:

```solidity
if (_position.credit > 0) {
    uint128 remainingCredit = _position.credit - UtilsLib.toUint128(units);
    uint128 remainingFee = remainingCredit > 0
        ? UtilsLib.toUint128(_position.pendingFee.mulDivUp(remainingCredit, _position.credit))
        : 0;
    pendingFeeDecrease = _position.pendingFee - remainingFee;
    _position.pendingFee = remainingFee;
}
```

This ensures the remaining position always retains `ceil(pendingFee * remainingCredit / credit)`, preventing rounding from zeroing the fee obligation on a non-zero remaining credit. The fuzz test `testWithdrawReducesPendingFee` and the Certora rule `pendingFeeDecreasesProportionallyOnWithdraw` must be updated to reflect the new formula.

## Proof of Concept

Minimal Foundry test (add to `test/ContinuousFeeTest.sol`):

```solidity
function testWithdrawCreditMinusOneZeroesPendingFee() public {
    uint256 credit = 1e18;
    uint256 feeRate = 0.1e18; // 10% annual
    uint256 ttm = 365 days;

    setupLender(credit, feeRate, ttm);

    uint256 pendingFee = midnight.pendingFee(id, lender);
    assertGt(pendingFee, 0, "pendingFee must be positive");
    assertLt(pendingFee, credit, "pendingFee must be less than credit (normal state)");

    // Borrower repays so withdrawable >= credit - 1
    deal(address(loanToken), borrower, credit);
    vm.prank(borrower);
    midnight.repay(market, credit, borrower, address(0), hex"");

    // Exploit: withdraw credit - 1
    vm.prank(lender);
    midnight.withdraw(market, credit - 1, lender, lender);

    // pendingFee is now 0 on the remaining 1-unit position
    assertEq(midnight.pendingFee(id, lender), 0, "pendingFee zeroed on remaining credit");
    assertEq(midnight.creditOf(id, lender), 1, "1 unit of credit remains");

    // No fee ever accrues: updatePosition returns 0 accrued fee
    (,, uint128 accruedFee) = midnight.updatePositionView(market, id, lender);
    assertEq(accruedFee, 0, "no fee accrues on remaining credit");

    // Lender withdraws the last unit with no fee deducted
    vm.prank(lender);
    midnight.withdraw(market, 1, lender, lender);

    // continuousFeeCredit was never incremented by the evaded fee
    assertEq(midnight.continuousFeeCredit(id), 0, "protocol received no fee");
}
```
