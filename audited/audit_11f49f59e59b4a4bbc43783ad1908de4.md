Audit Report

## Title
Fee-on-Transfer Loan Token in `repay` Inflates `withdrawable` Beyond Actual Balance — (File: `src/Midnight.sol`)

## Summary
The `repay` function credits `marketState[id].withdrawable` by the caller-supplied `units` before executing the token transfer. `SafeTransferLib.safeTransferFrom` only validates the boolean return value and never measures the actual balance delta. When the market's `loanToken` is a fee-on-transfer token, the contract receives `units - fee` but records `units` as withdrawable, permanently breaking the solvency invariant and enabling direct loss of lender principal.

## Finding Description

**Exact code path:**

`repay` in `src/Midnight.sol` lines 508–520: [1](#0-0) 

The state mutation at lines 508–509 unconditionally uses the argument `units`, not the actual tokens received. `SafeTransferLib.safeTransferFrom` in `src/libraries/SafeTransferLib.sol` lines 24–34 checks only `token.code.length > 0`, call success, and the boolean return value — no balance-before/balance-after measurement is performed: [2](#0-1) 

**Root cause:** For a fee-on-transfer token, `transferFrom(payer, address(this), units)` returns `true` and emits a transfer event for `units`, but only `units - fee` arrives at the contract. The protocol has no balance-delta check.

**Market creation has no token validation:** `touchMarket` validates collateral params, maturity, and LLTV, but performs zero validation on `loanToken` properties. Market creation is permissionless: [3](#0-2) 

**Exploit flow:**
1. Attacker creates a market with a fee-on-transfer ERC-20 as `loanToken` (permissionless, no token validation).
2. Borrower takes a loan of `D` units via normal protocol flow.
3. Borrower calls `repay(market, D, onBehalf, address(0), "")`.
4. Protocol executes: `debt[onBehalf] -= D` and `withdrawable += D`.
5. `safeTransferFrom` transfers `D` tokens from borrower; only `D * (1 - feeRate)` arrives at `address(midnight)`.
6. `withdrawable` now exceeds actual balance by `D * feeRate`.
7. Lenders call `withdraw` — protocol executes `withdrawable -= D` and `safeTransfer(loanToken, receiver, D)`.
8. The `safeTransfer` of `D` drains tokens belonging to other lenders or other markets sharing the same token, causing a shortfall.

**Why existing checks fail:** `SafeTransferLib` is a return-value-only wrapper with no balance accounting. There is no market-creation validation rejecting fee-on-transfer tokens. The state mutation precedes the transfer, so no reentrancy guard would help. The Certora specs explicitly summarize `SafeTransferLib.safeTransferFrom` as `NONDET`, meaning formal verification does not cover actual token behavior: [4](#0-3) 

## Impact Explanation
Every `repay` call with a fee-on-transfer loan token permanently inflates `marketState[id].withdrawable` relative to the contract's actual token balance. Lenders who withdraw after such repayments receive tokens deposited by other lenders or belonging to other markets, causing direct, irreversible loss of lender principal. This is a direct theft of assets impact class. `SECURITY.md` contains no exclusion for fee-on-transfer tokens. [5](#0-4) 

## Likelihood Explanation
Any unprivileged actor can create a market with a fee-on-transfer loan token — market creation is permissionless and `touchMarket` performs no validation on `loanToken` properties. The borrower needs only a standard borrow-then-repay flow. The exploit is repeatable: each `repay` call widens the deficit by `units * feeRate`. Tokens with owner-configurable fees (e.g., USDT's historical fee switch) make this reachable without any malicious token-owner action at exploit time.

## Recommendation
Measure the actual balance delta in `repay` by recording `balanceBefore = IERC20(market.loanToken).balanceOf(address(this))` before the transfer and computing `received = balanceAfter - balanceBefore` after it. Use `received` (not `units`) to update `position[id][onBehalf].debt` and `marketState[id].withdrawable`. Alternatively, document fee-on-transfer tokens as explicitly out of scope and enforce this at market creation by requiring `loanToken` to pass a balance-delta check during `touchMarket`.

## Proof of Concept
**Minimal Foundry test plan:**
1. Deploy a mock ERC-20 that deducts a 1% fee on every `transferFrom`.
2. Create a market with this token as `loanToken` (no validation blocks it).
3. Have a lender supply liquidity; have a borrower take a loan of `D` units.
4. Borrower calls `repay(market, D, borrower, address(0), "")`.
5. Assert `marketState[id].withdrawable == D` (recorded) while `IERC20(feeToken).balanceOf(address(midnight)) == D * 99 / 100` (actual).
6. Have the lender call `withdraw` for `D` units — the call succeeds by draining tokens from another depositor, proving the shortfall.
