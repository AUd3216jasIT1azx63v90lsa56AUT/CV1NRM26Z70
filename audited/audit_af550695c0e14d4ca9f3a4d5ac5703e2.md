### Title
Fee-on-Transfer loanToken Causes `claimableSettlementFee` to Exceed Actual Balance, Enabling feeClaimer to Drain Lender Funds - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by the full `buyerAssets - sellerAssets` at line 418 before the inbound transfer at line 455. When `loanToken` is a fee-on-transfer token, Midnight receives only `(buyerAssets - sellerAssets) * (1 - transferFee)`, creating a per-take deficit of `(buyerAssets - sellerAssets) * transferFee` between the accounting ledger and the actual balance. After N takes this deficit accumulates, and when feeClaimer calls `claimSettlementFee` up to the inflated `claimableSettlementFee` ceiling, it drains tokens that back lender credits (`withdrawable`), causing insolvency.

### Finding Description

**Exact code path:**

`src/Midnight.sol` line 418 — state update before transfer:
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

`src/Midnight.sol` line 455 — inbound transfer of the settlement fee portion:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
``` [2](#0-1) 

`SafeTransferLib.safeTransferFrom` checks only for revert/return-value success; it does not verify the amount actually credited to `address(this)`. With a fee-on-transfer `loanToken`, Midnight receives `(buyerAssets - sellerAssets) * (1 - f)` where `f` is the token's transfer fee rate, while `claimableSettlementFee` was already incremented by the full `buyerAssets - sellerAssets`.

**Market creation is permissionless.** `touchMarket()` imposes no restriction on `loanToken` type — any address is accepted: [3](#0-2) 

**No existing check stops it.** There is no balance-before/after guard, no fee-on-transfer exclusion, and no received-amount verification anywhere in `take()`.

**The Certora solvency spec explicitly models this ordering gap** via `pendingFeeReceipt` and asserts it clears to zero via `CVL_transferFrom` — but that model assumes a standard ERC20 that delivers exactly the requested amount. With fee-on-transfer tokens the `pendingFeeReceiptZero` invariant is violated and `tokenBalanceCorrect` breaks: [4](#0-3) 

**Exploit flow:**
1. Attacker (market creator, unprivileged) calls `touchMarket()` with a fee-on-transfer token as `loanToken` and `settlementFee > 0`.
2. Any taker calls `take()` repeatedly with valid offers.
3. Each call: `claimableSettlementFee` += `D` (full spread), Midnight receives `D * (1 - f)`. Deficit per take = `D * f`.
4. After N takes: `claimableSettlementFee` exceeds actual balance by `N * D * f`.
5. feeClaimer calls `claimSettlementFee(token, claimableSettlementFee, receiver)` — this transfers tokens that back lender `withdrawable` balances, causing lenders to be unable to `withdraw()`.

### Impact Explanation
`claimableSettlementFee[token]` grows faster than Midnight's actual token balance. The feeClaimer can drain up to `N * D * f` tokens that belong to lenders (backed by `withdrawable`). Lenders calling `withdraw()` receive `SafeTransferLib.safeTransfer` which will revert once the balance is exhausted. This is concrete insolvency: lender credit redemption fails while feeClaimer has already extracted the shortfall.

### Likelihood Explanation
**Preconditions:** (1) `loanToken` is fee-on-transfer — market creation is permissionless so any actor can create such a market; (2) `settlementFee > 0` — this is the normal operating state set by the feeSetter. Both preconditions are reachable without any privileged action by the attacker. The exploit is repeatable on every `take()` call and the deficit compounds linearly with volume.

### Recommendation
Add a balance-before/after check in `take()` to verify the actual amount received equals `buyerAssets - sellerAssets`:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
require(
    IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore == buyerAssets - sellerAssets,
    FeeOnTransferToken()
);
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as `loanToken` by adding a check in `touchMarket()` (e.g., a whitelist or a probe transfer).

// Settlement fee receipts pending settlement: claimableSettlementFee is incremented in take before
// the inbound fee transfer happens, so we track the gap and clear it in CVL_transferFrom.
persistent ghost mapping(address => mathint) pendingFeeReceipt {
    init_state axiom (forall address token. pendingFeeReceipt[token] == 0);
}

hook Sstore claimableSettlementFee[KEY address token] uint256 newVal (uint256 oldVal) {
    // Except for claimSettlementFee, the claimableSettlementFee is non-decreasing, see WithdrawableMonotonicity.spec.
    if (newVal > oldVal) {
        pendingFeeReceipt[token] = pendingFeeReceipt[token] + newVal - oldVal;
    }
}

/// INVARIANTS AND RULES ///

// For any token, the pending settlement fee receipt after a transaction is 0: every claimableSettlementFee
// increment in take is paid back in by the same-function inbound transfer.
weak invariant pendingFeeReceiptZero(address token)
    pendingFeeReceipt[token] == 0;

// For any token, the balance of the contract is always greater than or equal to the sum of all collateral, withdrawable, and claimable settlement fee amounts for that token minus the flash loaned amount.
// Note: this invariant is strong, so it also holds before each external call.
strong invariant tokenBalanceCorrect(address token)
    tokenBalances[token][currentContract] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token) - flashloans[token] - pendingFeeReceipt[token]
```
