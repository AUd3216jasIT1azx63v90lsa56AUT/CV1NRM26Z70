### Title
Continuous-Fee Undercharge via Repeated `mulDivDown` Rounding in Sequential `updatePosition` Calls - (File: src/Midnight.sol)

### Summary

Each call to `updatePositionView` computes the accrued fee as `postSlashPendingFee.mulDivDown(accrualEnd - lastAccrual, maturity - lastAccrual)`, flooring the result. Because `lastAccrual` is reset to `block.timestamp` after every `_updatePosition` call, splitting one time window into N sub-windows compounds the floor rounding: each sub-window can independently round down by up to 1 unit, so N steps can undercharge the fee by up to N−1 units compared to a single equivalent step. An unprivileged lender can exploit this to retain more credit than they are economically owed, withdrawing the excess at the expense of the protocol's `continuousFeeCredit`.

### Finding Description

**Exact code path:**

`updatePositionView` in `src/Midnight.sol` lines 814–815:
```solidity
uint128 fee = _lastAccrual < market.maturity
    ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
    : 0;
``` [1](#0-0) 

`_updatePosition` then writes `_position.lastAccrual = uint128(block.timestamp)` and adds `accruedFee` to `continuousFeeCredit`, but does **not** adjust `withdrawable` or `totalUnits`. [2](#0-1) 

`mulDivDown` is a plain integer division (floor): [3](#0-2) 

**Root cause:** The formula `floor(P_i × dt_i / (M − t_i))` is applied independently at each sub-window. Because `P_i` is reduced by the floored fee after each step, the denominator shrinks faster than the numerator, but the floor still discards fractional units. The sum of N floored values is strictly less than the single-step floor for certain parameter combinations.

**Concrete counterexample (no slashing, no bad debt):**

Let `pendingFee = P = 7`, `maturity = M = 9`, three steps at `t = 1, 2, 3`:

| Step | Formula | Result | P after |
|------|---------|--------|---------|
| t=1 | floor(7 × 1 / 9) | 0 | 7 |
| t=2 | floor(7 × 1 / 8) | 0 | 7 |
| t=3 | floor(7 × 1 / 7) | 1 | 6 |

Total fee charged = **1**. Single-step equivalent: `floor(7 × 3 / 9) = 2`. The lender retains 1 extra unit of credit.

**Exploit flow:**

1. Lender takes a buy offer (`take`), acquiring `credit = C`, `pendingFee = P`.
2. Lender (or anyone) calls `updatePosition` N times, each after a small `warp(+dt)`, choosing `dt` values where `floor(P_i × dt / (M − t_i)) = 0` for the first N−1 steps.
3. After N steps, total fee charged = `sum_floor < floor(P × T / M)` (the single-step ideal).
4. Lender calls `withdraw(credit)` — the `withdraw` function calls `_updatePosition` internally, then transfers `credit` tokens to the lender.
5. Lender receives `C − sum_floor` tokens instead of the economically correct `C − floor(P × T / M)`.

**Why existing checks do not stop it:**

- `withdraw` enforces `withdrawable -= units` (reverts if insufficient), but `withdrawable` is not reduced by `updatePosition` — it is only reduced by `withdraw` and `claimContinuousFee`. The excess credit the lender retains is backed by `withdrawable` that should have flowed to `continuousFeeCredit`.
- The Certora invariant `totalUnitsEqualsSumNegativeDebtPlusWithdrawable` (`totalUnits == sumDebt + withdrawable`) is maintained because `updatePosition` touches neither field.
- The solvency invariant `tokenBalanceCorrect` is maintained: the contract balance covers `withdrawable`, but the split between lender share and protocol fee share is wrong.
- The existing fuzz test `testMultipleAccrualsSumCorrectly` uses `assertApproxEqAbs(..., 2)` — a tolerance of 2 units for 2 accruals — and does not test N up to 100. [4](#0-3) 

### Impact Explanation

The lender withdraws up to N−1 extra loan-token units compared to the single-step ideal, where N is the number of `updatePosition` calls made before withdrawal. These units are taken from `withdrawable` that would otherwise be claimable by the protocol via `claimContinuousFee`. For N = 100, the maximum fee undercharge is 99 units per lender per market. For small `pendingFee` values (e.g., `pendingFee = 100`), this can represent nearly the entire fee obligation. The solvency invariant is not broken (total tokens in = total tokens out), but the protocol's continuous-fee revenue is reduced by the rounding shortfall.

### Likelihood Explanation

- **Preconditions:** Market with nonzero `continuousFee`, lender with nonzero `pendingFee`, time before maturity. All are normal operating conditions.
- **Feasibility:** `updatePosition` is a public, permissionless function callable by anyone including the lender themselves. No special role or signature is required.
- **Repeatability:** The attack is repeatable every time a lender enters a new position. Using `multicall`, all N calls can be batched in a single transaction, minimizing gas overhead.
- **Optimal parameters:** Small `pendingFee` relative to `maturity − lastAccrual` maximizes the number of zero-fee steps. The worst case is `pendingFee < maturity − lastAccrual` for the first N−1 steps, then `pendingFee = maturity − lastAccrual` for the last step.

### Recommendation

Replace `mulDivDown` with `mulDivUp` in the fee accrual line so that each sub-window charges at least the ceiling of the proportional fee:

```solidity
// Before (src/Midnight.sol line 815):
uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))

// After:
uint128(postSlashPendingFee.mulDivUp(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
```

This ensures `sum(fee_i) >= floor(P × T / M)` across any number of sub-windows, eliminating the undercharge. The lender may pay at most 1 extra unit per step (bounded, and symmetric with the existing ceiling rounding used for `pendingFeeDecrease` on withdraw). [1](#0-0) [5](#0-4) 

### Proof of Concept

```solidity
// Foundry stateful fuzz test
function testFeeUnderchargeViaSplitAccruals(
    uint256 credit,
    uint256 feeRate,
    uint256 ttm,
    uint8 N  // number of steps, bound to [2, 100]
) public {
    credit = bound(credit, 1, MAX_CREDIT);
    feeRate = bound(feeRate, 1, MAX_CONTINUOUS_FEE);
    ttm = bound(ttm, N + 1, 360 days);
    N = uint8(bound(N, 2, 100));

    setupLender(credit, feeRate, ttm);
    uint256 initialPendingFee = midnight.pendingFee(id, lender);
    vm.assume(initialPendingFee > 0);

    uint256 totalElapsed = ttm / 2; // stay pre-maturity
    uint256 dt = totalElapsed / N;
    vm.assume(dt > 0);

    // N-step path
    uint256 snap = vm.snapshotState();
    uint256 totalAccruedFeeNSteps;
    for (uint8 i = 0; i < N; i++) {
        vm.warp(vm.getBlockTimestamp() + dt);
        (, , uint128 stepFee) = midnight.updatePosition(market, lender);
        totalAccruedFeeNSteps += stepFee;
    }
    uint256 creditNSteps = midnight.creditOf(id, lender);
    vm.revertToState(snap);

    // Single-step path
    vm.warp(vm.getBlockTimestamp() + dt * N);
    (, , uint128 singleFee) = midnight.updatePosition(market, lender);
    uint256 creditSingleStep = midnight.creditOf(id, lender);

    // Key assertions:
    // 1. N-step total fee must be >= single-step fee (FAILS = vulnerability confirmed)
    assertGe(totalAccruedFeeNSteps, singleFee,
        "N-step fee undercharges vs single step");

    // 2. N-step credit must be <= single-step credit (FAILS = lender gets excess)
    assertLe(creditNSteps, creditSingleStep,
        "N-step credit exceeds single-step credit");

    // 3. Rounding error bounded by N-1
    assertLe(creditNSteps - creditSingleStep, N - 1,
        "excess credit exceeds N-1 bound");
}
```

**Expected result:** Assertion 1 and 2 fail for inputs such as `pendingFee = 7`, `maturity − lastAccrual = 9`, `N = 3`, `dt = 1`, demonstrating that the lender retains 1 extra unit of credit (6 vs 5) and the protocol collects 1 fewer unit of fee (1 vs 2).
