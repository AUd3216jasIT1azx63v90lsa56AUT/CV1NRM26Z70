Audit Report

## Title
Missing `caller` Parameter in `IRepayCallback.onRepay()` Enables Authorized Operator to Pull Funds from Arbitrary Callback Contracts - (File: src/Midnight.sol)

## Summary
The `repay()` function sets `payer = callback` when a non-zero callback is provided, with no constraint that `callback == msg.sender`. Because `IRepayCallback.onRepay()` does not pass the original caller (unlike `ILiquidateCallback.onLiquidate` and `IFlashLoanCallback.onFlashLoan`, which both include an `address caller` parameter), a callback contract cannot distinguish a legitimate self-initiated repay from one triggered by an arbitrary authorized operator. Any user who is authorized for any borrower (including one they control themselves) can drain loan tokens from any contract that implements `onRepay()` and holds a Midnight approval.

## Finding Description
In `src/Midnight.sol` lines 502–521:

```solidity
function repay(Market memory market, uint256 units, address onBehalf, address callback, bytes calldata data)
    external
{
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized()); // line 505
    // ...
    address payer = callback != address(0) ? callback : msg.sender;  // line 511
    // ...
    if (callback != address(0)) {
        require(
            IRepayCallback(callback).onRepay(id, market, units, onBehalf, data) == CALLBACK_SUCCESS,
            WrongRepayCallbackReturnValue()
        );
    }
    SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units); // line 520
}
```

The authorization check on line 505 only verifies that `msg.sender` is authorized for `onBehalf`. There is no check that `callback == msg.sender`. The `callback` address is freely supplied by the caller and becomes the `payer` on line 520.

The `IRepayCallback` interface is:

```solidity
interface IRepayCallback {
    function onRepay(bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data)
        external returns (bytes32);
}
```

Critically, `onRepay` does **not** receive the original `msg.sender` (the initiator of `repay()`). Compare with the other callback interfaces in `src/interfaces/ICallbacks.sol`:

- `ILiquidateCallback.onLiquidate(address caller, ...)` — passes caller ✓
- `IFlashLoanCallback.onFlashLoan(address caller, ...)` — passes caller ✓
- `IRepayCallback.onRepay(bytes32 id, ...)` — **does not pass caller** ✗

This asymmetry means a callback contract implementing `onRepay` cannot verify who initiated the repay. When `onRepay` is invoked, `msg.sender` is always Midnight, not the original attacker.

**Exploit flow:**

1. Attacker creates or controls a borrower account `B` and calls `setIsAuthorized(attacker, true, B)` so `isAuthorized[B][attacker] == true`.
2. Attacker calls `repay(market, units, B, victimContract, data)`.
3. Authorization check passes: attacker is authorized for `B`.
4. `payer = victimContract`.
5. Midnight calls `victimContract.onRepay(id, market, units, B, data)`. Inside `onRepay`, `msg.sender == address(Midnight)`. The victim cannot identify the attacker.
6. If `victimContract` returns `CALLBACK_SUCCESS` (as any compliant repay callback must), Midnight executes `safeTransferFrom(loanToken, victimContract, Midnight, units)`.

The victim contract's `onRepay` has no way to distinguish this from a legitimate self-initiated repay, because the caller identity is not forwarded. The standard defense used in other callbacks (checking `caller == address(this)` or a known initiator) is impossible here.

The `OnlyExplicitPayerCanLoseTokens.spec` formally models "callback returned `CALLBACK_SUCCESS`" as consent, but this model is broken when the callback cannot determine who triggered the call.

## Impact Explanation
Any contract that (a) implements `IRepayCallback.onRepay()` returning `CALLBACK_SUCCESS` and (b) holds a Midnight approval for loan tokens can have its entire approved balance drained by any user who is authorized for any borrower — including a borrower they themselves control. The victim's token balance decreases to repay a debt it did not choose to repay. This violates the protocol's stated invariant that authorization only allows intended account delegation, not pulling funds from arbitrary third parties.

## Likelihood Explanation
The preconditions are realistic and low-cost to satisfy. Any user can create their own borrower account and self-authorize, requiring no cooperation from any other party. Contracts implementing `IRepayCallback` and holding Midnight approval are the intended integration target of the repay callback feature (e.g., flash-repay routers, bundlers with callback-based repay flows). The attack is repeatable as long as the victim retains Midnight approval and loan token balance. The missing `caller` parameter makes it structurally impossible for callback implementors to defend against this without out-of-band state (e.g., transient storage reentrancy guards keyed to `msg.sender`), which the interface does not require or document.

## Recommendation
Add the original caller to the `onRepay` callback signature, consistent with `onLiquidate` and `onFlashLoan`:

```solidity
interface IRepayCallback {
    function onRepay(address caller, bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data)
        external returns (bytes32);
}
```

And in `repay()`, pass `msg.sender` as the first argument:

```solidity
IRepayCallback(callback).onRepay(msg.sender, id, market, units, onBehalf, data)
```

This allows callback contracts to verify that the repay was initiated by themselves (or another trusted address), matching the protection available in `onLiquidate` and `onFlashLoan`.

## Proof of Concept
Minimal Foundry test outline:

```solidity
// VictimCallback: implements onRepay, approves Midnight, holds loan tokens
contract VictimCallback is IRepayCallback {
    function onRepay(bytes32, Market memory market, uint256 units, address, bytes memory)
        external returns (bytes32)
    {
        // Cannot check who initiated this — msg.sender is always Midnight
        ERC20(market.loanToken).approve(msg.sender, units); // approve Midnight
        return CALLBACK_SUCCESS;
    }
}

function testCallbackPayerConfusion() public {
    // Setup: attacker self-authorizes for a borrower they control
    vm.prank(attacker);
    midnight.setIsAuthorized(attacker, true, attacker); // attacker authorized for themselves

    // Give victim loan tokens and set up victim as a repay callback
    VictimCallback victim = new VictimCallback();
    deal(address(loanToken), address(victim), repayAmount);

    // Attacker drains victim by supplying victim as callback
    vm.prank(attacker);
    midnight.repay(market, repayAmount, attacker, address(victim), "");

    // Victim's tokens are gone, attacker's (or onBehalf's) debt is reduced
    assertEq(loanToken.balanceOf(address(victim)), 0);
}
```
