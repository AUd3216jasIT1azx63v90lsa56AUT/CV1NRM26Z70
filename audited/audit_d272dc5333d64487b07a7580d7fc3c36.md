### Title
Front-running `repay` with a partial liquidation causes arithmetic underflow DoS - (`src/Midnight.sol`)

### Summary
The `repay` function subtracts `units` from `position[id][onBehalf].debt` with no guard that `units ≤ debt`. Because `liquidate` can reduce the same debt field in a preceding transaction, an attacker can front-run any `repay(market, D, onBehalf, …)` call with a partial liquidation that lowers debt below `D`, causing the borrower's repay to revert with an arithmetic underflow. The attacker earns the liquidation incentive on every front-run, making the attack self-funding.

### Finding Description
**Code path:**

`repay` (`src/Midnight.sol:502-521`):
```
position[id][onBehalf].debt -= UtilsLib.toUint128(units);   // line 508
```
`UtilsLib.toUint128` (`src/libraries/UtilsLib.sol:38-42`) only guards against cast overflow; it does not enforce `units ≤ debt`. The `-=` on a `uint128` storage slot is Solidity 0.8 checked arithmetic and reverts on underflow.

`liquidate` (`src/Midnight.sol:626-677`) writes to the same slot via:
- `_position.debt -= uint128(badDebt);` (line 628) — bad-debt path
- `_position.debt -= UtilsLib.toUint128(repaidUnits);` (line 676) — normal path

Both execute before any external call, so the state change is committed before the borrower's pending `repay` is included.

**Exploit flow:**
1. Borrower has unhealthy position: `debt = D`, position is liquidatable (price dropped below LLTV threshold).
2. Borrower broadcasts `repay(market, D, onBehalf, …)` to repay the full balance.
3. Attacker sees the transaction in the mempool and front-runs with `liquidate(market, collateralIndex, 0, X, borrower, …)` where `1 ≤ X ≤ D-1`.
4. Liquidation succeeds: `_position.debt` is now `D - X` (or `D - badDebt - X` if bad debt is also realized).
5. Borrower's `repay` lands: `position[id][onBehalf].debt -= toUint128(D)` → `(D-X) - D` underflows → **revert**.
6. Attacker received collateral worth `X * lif / WAD` in step 3, paying only `X` loan tokens — net profit from the liquidation incentive.
7. Attacker repeats on every retry, continuously front-running with a minimal `repaidUnits = 1` if needed.

**Why existing checks do not stop it:**
- The only check in `repay` is the authorization guard (`onBehalf == msg.sender || isAuthorized`), which is irrelevant here.
- There is no `require(units <= position[id][onBehalf].debt)` in `repay`.
- The `liquidatorGate` check in `liquidate` (line 598-600) is only present when the market creator sets a non-zero gate; permissionless markets have `liquidatorGate == address(0)`.
- The `testRepay` fuzz test in `test/OtherFunctionsTest.sol:115-131` always bounds `repaid ≤ units`, so the underflow path is never exercised by the existing test suite.

### Impact Explanation
Any borrower whose position is unhealthy (liquidatable) can be permanently denied the ability to repay via `repay`. Every `repay(market, D, onBehalf, …)` call reverts as long as the attacker front-runs with even a 1-unit liquidation. The borrower is forced into full liquidation, losing the liquidation incentive premium on their collateral rather than repaying at par. This is a concrete, repeatable DoS against the `repay` function for any unhealthy borrower in any market without a `liquidatorGate`.

### Likelihood Explanation
**Preconditions:** (a) borrower's position is unhealthy (price has dropped below LLTV), (b) market has no `liquidatorGate` (the default). Both are routine on-chain states. **Feasibility:** front-running is standard on EVM chains with a public mempool; the attacker needs only to submit a higher-gas-price `liquidate` with `repaidUnits = 1`. **Profitability:** the attacker earns the liquidation incentive (`lif > WAD`) on every front-run, so the attack is net-positive. **Repeatability:** the attack can be repeated on every retry until the borrower's collateral is fully seized.

### Recommendation
Add a debt cap in `repay` so that `units` is silently clamped to the current debt, or add an explicit guard:

```solidity
// Option A – silent cap (matches common lending protocol patterns):
uint128 _debt = position[id][onBehalf].debt;
uint128 repaidUnits = units > _debt ? _debt : UtilsLib.toUint128(units);
position[id][onBehalf].debt = _debt - repaidUnits;
marketState[id].withdrawable += repaidUnits;
// … use repaidUnits for the token transfer

// Option B – explicit revert with a clear error:
require(units <= position[id][onBehalf].debt, RepayExceedsDebt());
position[id][onBehalf].debt -= UtilsLib.toUint128(units);
```

Option A is preferable because it lets a borrower pass `type(uint256).max` to repay their full balance atomically, which itself eliminates the front-running surface.

### Proof of Concept

```solidity
pragma solidity 0.8.34;

import {BaseTest} from "./BaseTest.sol";
import {IMidnight} from "../src/interfaces/IMidnight.sol";
import {stdError} from "forge-std/StdError.sol";

contract RepayFrontRunDoSTest is BaseTest {
    function testFrontRunRepayWithLiquidation(uint256 units, uint256 frontRunUnits) public {
        // Setup: borrower takes a position of `units`
        units = bound(units, 10, MAX_TEST_AMOUNT / 2);
        frontRunUnits = bound(frontRunUnits, 1, units - 1);

        collateralize(market, borrower, units);
        setupMarket(market, units);          // lender fills the offer; borrower.debt = units

        // Make position unhealthy so liquidation is possible
        Oracle(market.collateralParams[0].oracle).setPrice(ORACLE_PRICE_SCALE / 2);

        // Attacker front-runs with a partial liquidation
        deal(address(loanToken), address(this), frontRunUnits);
        midnight.liquidate(market, 0, 0, frontRunUnits, borrower, false, address(this), address(0), "");

        // Borrower's debt is now units - frontRunUnits (or less if bad debt was realized)
        uint256 debtAfterLiquidation = midnight.debtOf(toId(market), borrower);
        assert(debtAfterLiquidation < units);

        // Borrower tries to repay original `units` — must revert with arithmetic underflow
        deal(address(loanToken), borrower, units);
        vm.prank(borrower);
        vm.expectRevert(stdError.arithmeticError);
        midnight.repay(market, units, borrower, address(0), hex"");

        // Assert: borrower still has outstanding debt — DoS confirmed
        assertGt(midnight.debtOf(toId(market), borrower), 0);
    }
}
```

**Expected assertions:**
- `vm.expectRevert(stdError.arithmeticError)` passes — `repay` reverts on underflow.
- `assertGt(debtOf, 0)` passes — borrower cannot clear their debt.
- Fuzz over `(units, frontRunUnits)` confirms the underflow for all valid combinations where `frontRunUnits ∈ [1, units-1]`.
