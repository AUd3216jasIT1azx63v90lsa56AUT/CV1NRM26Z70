### Title
`claimableSettlementFee` Over-Credited on Fee-on-Transfer Loan Token in `take` - (`src/Midnight.sol`)

### Summary

In `Midnight.take`, `claimableSettlementFee[loanToken]` is incremented by the nominal spread `buyerAssets - sellerAssets` before the token transfer executes. When the loan token is a fee-on-transfer token, Midnight only receives `(buyerAssets - sellerAssets) * (1 - feeRate)`, but the accounting records the full nominal amount. The `claimSettlementFee` function has no balance guard, so the fee claimer can withdraw more tokens than were ever received as fees, draining funds that belong to lenders.

### Finding Description

**Exact code path:**

In `src/Midnight.sol`, the `take` function at line 418 updates the fee accounting before the transfer:

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;  // line 418
```

Then at lines 455–456, the actual transfers occur:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);  // line 455
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);                     // line 456
```

`SafeTransferLib.safeTransferFrom` (in `src/libraries/SafeTransferLib.sol` lines 24–34) performs no balance-before/after check — it only checks for revert and false return. With a fee-on-transfer token charging rate `f`, Midnight receives only `(buyerAssets - sellerAssets) * (1 - f)` from line 455, while `claimableSettlementFee` was credited with the full `buyerAssets - sellerAssets`.

**`claimSettlementFee` (lines 305–310):**

```solidity
function claimSettlementFee(address token, uint256 amount, address receiver) external {
    require(msg.sender == feeClaimer, OnlyFeeClaimer());
    claimableSettlementFee[token] -= amount;
    SafeTransferLib.safeTransfer(token, receiver, amount);
}
```

No balance check. It transfers `amount` from Midnight's unified token balance, which also covers lenders' `withdrawable` amounts. The inflated `claimableSettlementFee` allows the fee claimer to drain tokens belonging to lenders.

**Attacker-controlled inputs:**

Market creation is permissionless (`touchMarket` is `public`, lines 755–791, no access control). The protocol explicitly supports "arbitrary loan token" (confirmed in `live_context.json`). Any unprivileged market creator can deploy a market with a fee-on-transfer token as `loanToken`. The taker then executes `take` normally — no special inputs required beyond a valid offer.

**Why existing checks fail:**

There is no check anywhere in `take` or `claimSettlementFee` that compares the actual received balance against the credited amount. The Certora spec rule `takeIncreasesClaimableSettlementFee` (in `certora/specs/WithdrawableMonotonicity.spec` line 75) models token transfers as no-ops (balance-neutral), so it does not catch this discrepancy. The `SafeTransferLib` does not perform a received-amount check.

### Impact Explanation

After each `take` with a fee-on-transfer loan token, `claimableSettlementFee[token]` exceeds the actual fee tokens held by Midnight by `(buyerAssets - sellerAssets) * feeRate`. When `claimSettlementFee` is called for the full `claimableSettlementFee` amount, the shortfall is taken from lenders' `withdrawable` balances. Lenders who attempt to `withdraw` after the fee claim will receive less than their credit entitles them to, or the `withdraw` call will revert entirely. This is direct protocol insolvency.

### Likelihood Explanation

**Preconditions:** A fee-on-transfer token is used as `loanToken`. Market creation is permissionless, so any user can create such a market. Fee-on-transfer tokens are a well-known token class (e.g., tokens with built-in tax mechanisms). The protocol makes no exclusion for them.

**Feasibility:** Fully reachable by any unprivileged taker. No special permissions, no oracle manipulation, no admin action required. Repeatable: every `take` in such a market compounds the discrepancy.

**Repeatability:** Each `take` call inflates `claimableSettlementFee` by `(buyerAssets - sellerAssets) * feeRate`. The effect accumulates across multiple takes.

### Recommendation

Record the actual received amount rather than the nominal amount. Before and after the transfer to `address(this)`, snapshot the contract's token balance and credit only the delta:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualReceived;
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan tokens (e.g., via a market creation check or NatSpec warning), accepting that such markets are out of scope.

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";

// Fee-on-transfer token: charges 1% on every transferFrom
contract FeeOnTransferToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    uint256 public constant FEE_BPS = 100; // 1%

    function mint(address to, uint256 amount) external { balanceOf[to] += amount; }
    function approve(address spender, uint256 amount) external returns (bool) {
        allowance[msg.sender][spender] = amount; return true;
    }
    function transfer(address to, uint256 amount) external returns (bool) {
        balanceOf[msg.sender] -= amount; balanceOf[to] += amount; return true;
    }
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        allowance[from][msg.sender] -= amount;
        uint256 fee = amount * FEE_BPS / 10000;
        balanceOf[from] -= amount;
        balanceOf[to] += amount - fee; // fee burned/kept
        return true;
    }
}

contract FeeOnTransferTakeTest is Test {
    Midnight midnight;
    FeeOnTransferToken loanToken;
    address feeClaimer = address(0xFEE);
    address lender = address(0xA);
    address borrower = address(0xB);

    function setUp() public {
        midnight = new Midnight();
        loanToken = new FeeOnTransferToken();
        midnight.setFeeClaimer(feeClaimer);
        // ... set up market, ratifier, collateral, offers (standard test setup)
    }

    function testClaimableSettlementFeeInflatedByFeeOnTransfer() public {
        // 1. Create market with loanToken as fee-on-transfer token
        // 2. Lender supplies credit (lender is buyer/maker of buy offer)
        // 3. Borrower takes the offer (taker is seller)
        // 4. Record Midnight's actual token balance before take
        uint256 midnightBalanceBefore = loanToken.balanceOf(address(midnight));

        // 5. Execute take — standard call, no special inputs
        // take(offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData)

        uint256 midnightBalanceAfter = loanToken.balanceOf(address(midnight));
        uint256 actualFeeReceived = midnightBalanceAfter - midnightBalanceBefore;
        uint256 claimable = midnight.claimableSettlementFee(address(loanToken));

        // ASSERTION 1: claimableSettlementFee > actual tokens received
        assertGt(claimable, actualFeeReceived, "claimable exceeds actual received");

        // 6. Fee claimer claims full claimableSettlementFee
        vm.prank(feeClaimer);
        midnight.claimSettlementFee(address(loanToken), claimable, feeClaimer);

        // ASSERTION 2: Midnight's balance went below what lenders are owed

        uint256 lenderWithdrawable = midnight.withdrawable(/* marketId */bytes32(0));
        assertGt(lenderWithdrawable, midnightFinalBalance, "insolvency: lenders owed more than held");

        // ASSERTION 3: Lender withdraw reverts or underflows
        vm.expectRevert();
        midnight.withdraw(/* market */, lenderWithdrawable, lender, lender);
