### Title
Attacker-Controlled Reverting Oracle Permanently DoS-es Liquidation for Any Borrower with That Collateral Activated - (File: src/Midnight.sol)

### Summary
The `liquidate` function iterates every activated collateral in the borrower's `collateralBitmap` and calls `IOracle(_collateralParam.oracle).price()` on each one with no error handling. Because market creation is permissionless, an attacker can deploy a market whose oracle is a contract they control, wait for a borrower to activate that collateral, then make the oracle revert permanently, blocking all liquidation attempts for that borrower and violating the invariant that unhealthy positions must remain liquidatable.

### Finding Description
**Exact code path** — `src/Midnight.sol` lines 606–618:

```solidity
uint128 _collateralBitmap = _position.collateralBitmap;
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price();   // ← bare external call
    ...
    _collateralBitmap = _collateralBitmap.clearBit(i);
}
``` [1](#0-0) 

There is no `try/catch`, no return-value guard, and no way to skip a single collateral entry. If `IOracle.price()` reverts for any activated collateral, the entire `liquidate` call reverts before the `NotLiquidatable` check is even reached.

**Attacker-controlled inputs:**
- `market.collateralParams[i].oracle` — the oracle address is part of the market struct, which is supplied by the caller and hashed to produce `id`. Market creation via `touchMarket` is permissionless; any address can be used as an oracle.

**Exploit flow:**
1. Attacker deploys `MaliciousOracle` with a `shouldRevert` toggle; initially `price()` returns a valid value.
2. Attacker calls `touchMarket(market)` with `MaliciousOracle` as the oracle for collateral slot `k`.
3. Borrower calls `supplyCollateral(market, k, amount, borrower)`, setting bit `k` in `_position.collateralBitmap`.
4. Borrower takes on debt (via `take` or any credit-increasing path), making the position liquidatable when prices move.
5. Attacker calls `MaliciousOracle.setRevert(true)`.
6. Any call to `liquidate(market, ...)` for this borrower now reverts unconditionally inside the bitmap loop at line 610.

**Why existing checks do not stop it:**
- `liquidatorGate` check (lines 597–600) runs before the loop but does not protect against oracle reverts.
- `liquidationLocked` / `NotLiquidatable` checks (lines 620–624) are after the loop and are never reached.
- The Certora formal verification spec itself proves this revert propagation as a verified property (`oracleRevertCausesLiquidateRevert` in `certora/specs/Reverts.spec` lines 182–193), confirming the path is real and unguarded. [2](#0-1) 

### Impact Explanation
Any borrower who has activated a collateral whose oracle is attacker-controlled is permanently unliquidatable. The position accumulates bad debt that can never be realized, lenders cannot recover funds through liquidation, and the core protocol invariant — "unhealthy positions remain liquidatable" — is broken for the affected borrower for the lifetime of the market.

### Likelihood Explanation
**Preconditions:**
- Market creation is permissionless; the attacker only needs to deploy a contract that implements `IOracle` and call `touchMarket`.
- The borrower must supply collateral to that market slot, activating the bit. This is realistic: borrowers may not audit oracle bytecode, especially in a permissionless ecosystem where many markets exist.

**Feasibility:** Low cost — one oracle deployment, one `touchMarket` call. The toggle can be flipped at any time after the borrower activates the collateral, including atomically in the same block the position becomes unhealthy.

**Repeatability:** Permanent until the borrower manually withdraws all collateral for that slot (which itself requires `isHealthy`, which also calls the same oracle loop — see `src/Midnight.sol` lines 949–957 — making self-rescue equally impossible while the oracle reverts). [3](#0-2) 

### Recommendation
Wrap each oracle call in the bitmap loop with a `try/catch`. If an oracle reverts, treat the price as `0` (or skip that collateral's contribution to `maxDebt` and `badDebt`). This preserves liquidatability even when one oracle is broken, and is consistent with the existing comment in `NoDivisionByZero.spec` that already handles the zero-price case:

```solidity
try IOracle(_collateralParam.oracle).price() returns (uint256 price) {
    // existing accounting
} catch {
    // treat price as 0: collateral contributes nothing to maxDebt/badDebt
}
```

Alternatively, allow liquidators to pass a subset bitmap of collaterals to evaluate, so a single bad oracle does not block liquidation of the remaining collateral.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Market, CollateralParams} from "src/Midnight.sol";

contract ToggleRevertOracle {
    bool public shouldRevert;
    uint256 public fixedPrice = 1e18;
    function setRevert(bool v) external { shouldRevert = v; }
    function price() external view returns (uint256) {
        require(!shouldRevert, "oracle disabled");
        return fixedPrice;
    }
}

contract LiquidateOracleDoSTest is Test {
    Midnight midnight;
    ToggleRevertOracle maliciousOracle;
    address borrower = address(0xB0);
    address lender  = address(0xA0);

    function setUp() public { /* deploy Midnight, tokens, fund actors */ }

    function testPermanentLiquidateDoS() public {
        maliciousOracle = new ToggleRevertOracle();

        // 1. Attacker creates market with malicious oracle at slot 0
        Market memory market = _buildMarket(address(maliciousOracle));
        midnight.touchMarket(market);

        // 2. Borrower supplies collateral (activates bit 0) and borrows
        vm.startPrank(borrower);
        midnight.supplyCollateral(market, 0, 1e18, borrower);
        // ... take / borrow to create debt ...
        vm.stopPrank();

        // 3. Price drops → position is unhealthy (assert isHealthy == false)
        maliciousOracle.fixedPrice = 1; // drop price
        assertFalse(midnight.isHealthy(market, midnight.toId(market), borrower));

        // 4. Attacker toggles oracle to revert
        maliciousOracle.setRevert(true);

        // 5. Any liquidator call reverts → DoS confirmed
        vm.expectRevert("oracle disabled");
        midnight.liquidate(market, 0, 0, 1, borrower, false, address(this), address(0), "");

        // 6. Self-rescue also blocked: withdrawCollateral calls isHealthy which hits same loop
        vm.prank(borrower);
        vm.expectRevert();
        midnight.withdrawCollateral(market, 0, 1e18, borrower, borrower);
    }
}
```

**Expected assertions:**
- `isHealthy` returns `false` before the toggle (position is genuinely unhealthy).
- `liquidate` reverts with the oracle's revert reason after the toggle.
- `withdrawCollateral` also reverts, confirming the borrower cannot self-rescue.
- Toggling the oracle back makes `liquidate` succeed, proving the root cause is solely the oracle revert.

/// If any activated collateral oracle reverts on price, liquidate reverts.
rule oracleRevertCausesLiquidateRevert(env e, Midnight.Market market, uint256 collateralIndex, uint256 seizedAssets, uint256 repaidUnits, address borrower, address receiver, address callback, bytes data, uint256 revertingCollateralIndex, bool postMaturityMode) {
    require singleRevertingOracle == market.collateralParams[revertingCollateralIndex].oracle, "oracle is reverting";

    bytes32 id = summaryToId(market);
    uint128 bitmap = collateralBitmap(id, borrower);
    require summaryGetBit(bitmap, revertingCollateralIndex), "revertingCollateralIndex is activated";

    liquidate@withrevert(e, market, collateralIndex, seizedAssets, repaidUnits, borrower, postMaturityMode, receiver, callback, data);

    assert lastReverted;
}
