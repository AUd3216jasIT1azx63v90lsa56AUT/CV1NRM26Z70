### Title
Continuous Fee Accrual Rounds to Zero via Frequent `updatePosition` Calls — (File: src/Midnight.sol)

### Summary

The continuous fee accrual formula in `updatePositionView` uses integer division (`mulDivDown`) to compute the fee owed per update interval. When the elapsed time since the last accrual is small relative to the remaining time to maturity, the fee rounds to zero. Because `updatePosition` is permissionless — callable by anyone for any user — a lender can call it every block to eliminate or drastically reduce the continuous fee they owe to the protocol.

### Finding Description

The fee accrual formula is in `updatePositionView`: [1](#0-0) 

```solidity
uint256 accrualEnd = UtilsLib.min(block.timestamp, market.maturity);
uint128 _lastAccrual = _position.lastAccrual;
uint128 fee = _lastAccrual < market.maturity
    ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
    : 0;
```

`mulDivDown` is plain integer division: [2](#0-1) 

The formula is:

```
fee = postSlashPendingFee * (accrualEnd - _lastAccrual) / (market.maturity - _lastAccrual)
```

For this to be non-zero, `postSlashPendingFee * interval >= (market.maturity - _lastAccrual)`.

**`pendingFee` magnitude.** At position creation, `pendingFee` is set as: [3](#0-2) 

```solidity
buyerPendingFeeIncrease =
    buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD);
```

`MAX_CONTINUOUS_FEE` is: [4](#0-3) 

```solidity
uint32 constant MAX_CONTINUOUS_FEE = uint32(uint256(0.01e18) / uint256(365 days)); // ≈ 3.17e8
```

For a lender with `credit = 1e6` (1 USDC, 6-decimal token), max fee, 1-year maturity:

```
pendingFee = 1e6 * 3.17e8 * 31_536_000 / 1e18 ≈ 9_990
```

Fee per 1-second interval:
```
fee = 9_990 * 1 / 31_536_000 ≈ 0.000317  →  rounds to 0
```

Fee per 12-second Ethereum block:
```
fee = 9_990 * 12 / 31_536_000 ≈ 0.0038  →  rounds to 0
```

The fee only becomes non-zero when the interval exceeds `31_536_000 / 9_990 ≈ 3_156 seconds (~52 minutes)`. Any lender calling `updatePosition` more frequently than once per 52 minutes pays **zero** continuous fee for a 1 USDC position.

For larger positions (e.g., 1000 USDC, `pendingFee ≈ 9_990_000`), calling every 4 seconds (L2 block time) results in each call accruing `floor(1.27) = 1` instead of `1.27`, a ~21% fee reduction over the year.

**The entry path is permissionless.** `updatePosition` has no authorization check: [5](#0-4) 

```solidity
function updatePosition(Market memory market, address user) external returns (uint128, uint128, uint128) {
    bytes32 id = toId(market);
    require(marketState[id].tickSpacing > 0, MarketNotCreated());
    return _updatePosition(market, id, user);
}
```

Any lender can call this on their own position every block.

### Impact Explanation

Lenders can partially or fully evade the continuous fee:

- **Small positions** (e.g., ≤ 1000 USDC on Ethereum, any size on sub-second L2s): fee rounds to 0 on every block-level call, resulting in **complete fee evasion** for the entire market lifetime.
- **Large positions on L2s**: systematic rounding loss of 10–30% of expected fee revenue per position.
- The accrued fee flows to `continuousFeeCredit` and is claimable by the protocol's `feeClaimer`. Zero accrual means direct, permanent protocol revenue loss. [6](#0-5) 

### Likelihood Explanation

- `updatePosition` is permissionless and costs only gas.
- On L2s (Arbitrum, Base, Optimism) with block times of 0.25–2 seconds and near-zero gas costs, running a bot to call `updatePosition` every block is trivially cheap.
- Even on Ethereum mainnet, small positions (< ~3,000 USDC) have a `pendingFee` so small that the fee rounds to 0 on every 12-second block call, making the attack gas-profitable for any position where the saved fee exceeds gas cost.
- The attack requires no privileged access, no special setup, and no coordination.

### Recommendation

Replace the linear interpolation with a formula that accumulates fee per unit time at higher precision, or enforce a minimum accrual interval (e.g., 1 day) so that `interval` is always large enough to produce a non-zero fee. Alternatively, track fee accrual in a higher-precision accumulator (e.g., scaled by `WAD`) and only convert to units on withdrawal, preventing dust-level rounding from compounding across many calls.

### Proof of Concept

**Setup:**
- Loan token: USDC (6 decimals)
- Market maturity: 1 year from now (`maturity = block.timestamp + 31_536_000`)
- `continuousFee = MAX_CONTINUOUS_FEE ≈ 3.17e8`
- Lender takes a buy offer for `credit = 1_000e6` (1000 USDC worth of units)

**Computed `pendingFee` at position creation:**
```
pendingFee = 1_000e6 * 3.17e8 * 31_536_000 / 1e18 ≈ 9_990_000
```

**Attack loop (bot, every Ethereum block = 12 s):**
```
fee = 9_990_000 * 12 / 31_536_000 = 119_880_000 / 31_536_000 = 3 (floor)
expected = 3.8
rounding loss per block = 0.8 units
```

Over 1 year (`2_628_000` blocks): `0.8 * 2_628_000 = 2_102_400` units lost to rounding ≈ **21% of the total fee evaded**.

On a 1-second L2:
```
fee = 9_990_000 * 1 / 31_536_000 = 0 (floor)
```
**100% of the continuous fee is evaded** for the entire year at negligible gas cost.
