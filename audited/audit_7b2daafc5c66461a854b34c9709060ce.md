### Title
Missing registry-state guard in `stake()` permanently locks users into the pool unexpectedly - (File: src/ConfidencePool.sol)

### Summary
`stake()` accepts no parameter letting the caller bound the registry state they are willing to enter. A user who intends to stake only while withdrawals remain open (pre-risk state) can have their transaction execute after the registry has transitioned to `UNDER_ATTACK`, permanently locking their principal with ~zero bonus entitlement and no exit path — with no on-chain protection against this outcome.

### Finding Description
`stake()` calls `_observePoolState()` on every invocation. [1](#0-0) 

If the registry has transitioned to `UNDER_ATTACK` by the time the transaction executes, `_observePoolState()` seals `riskWindowStart` via `_markRiskWindowStart()`. [2](#0-1) 

`_assertDepositsAllowed()` explicitly permits deposits during `UNDER_ATTACK`, so the stake succeeds. [3](#0-2) 

However, once `riskWindowStart != 0`, `withdraw()` is permanently disabled — the one-way latch is checked before any other logic. [4](#0-3) 

Additionally, because the user's entry time is recorded as `block.timestamp ≈ riskWindowStart`, their k=2 score `amount × (T − entry)²` collapses to ~zero, yielding a negligible bonus share. [5](#0-4) 

There is no parameter (e.g. `maxRegistryState`) that would allow the caller to say "revert if the registry state is beyond `NEW_DEPLOYMENT`." The only documented mitigation is the off-chain advice in DESIGN.md §3: "A staker who does not want this can read the live registry state before staking." This does not protect against the race between observation and execution.

A malicious actor can exploit this deliberately: `pokeRiskWindow()` is permissionless and callable by anyone once the registry is in `UNDER_ATTACK`. [6](#0-5) 

Calling it just before a victim's pending `stake()` transaction ensures `riskWindowStart` is sealed before the victim's stake lands, locking them in.

### Impact Explanation
A user who staked expecting to retain the withdrawal option instead finds their principal irrevocably committed to the resolution path. Their bonus entitlement is ~zero (entry time ≈ `riskWindowStart`). If the pool resolves `CORRUPTED`, their entire principal is swept to `recoveryAddress` or the named attacker — a total loss. This is materially worse than the outcome the user expected when they submitted the transaction.

### Likelihood Explanation
The scenario arises in normal operation whenever the registry transitions to `UNDER_ATTACK` between a user's off-chain state check and their transaction's inclusion. This is a routine lifecycle event. The deliberate front-running path (calling `pokeRiskWindow()` ahead of a victim's `stake()`) requires no special privilege and costs only gas, making it trivially executable by any adversary who monitors the mempool.

### Recommendation
Add a `maxRegistryState` (or equivalent `requireWithdrawable: bool`) parameter to `stake()`. Before accepting the deposit, compare the observed state against the caller's bound and revert if it is exceeded:

```solidity
function stake(uint256 amount, IAttackRegistry.ContractState maxState) external nonReentrant whenPoolNotPaused {
    ...
    IAttackRegistry.ContractState state = _observePoolState();
    if (uint8(state) > uint8(maxState)) revert RegistryStateTooAdvanced();
    _assertDepositsAllowed(state);
    ...
}
```

This mirrors the slippage-protection pattern recommended in the referenced Size finding and gives callers an on-chain guarantee that their stake only executes under the conditions they evaluated off-chain.

### Proof of Concept

```
Scenario 1 — normal race (no malice):
  1. Registry is in NEW_DEPLOYMENT.
  2. Alice reads state off-chain, decides to stake 100 tokens expecting withdraw() to remain available.
  3. Registry transitions to UNDER_ATTACK (normal lifecycle).
  4. Alice's stake() executes: _observePoolState() seals riskWindowStart = block.timestamp.
  5. _assertDepositsAllowed passes (UNDER_ATTACK is allowed).
  6. Alice's eligibleStake += 100; entry time = block.timestamp = riskWindowStart.
  7. Alice calls withdraw() → reverts WithdrawsDisabled (riskWindowStart != 0).
  8. Alice's k=2 score = amount × (T − riskWindowStart)² ≈ 0 → ~zero bonus.
  9. If outcome = CORRUPTED, Alice loses 100 tokens to recoveryAddress.

Scenario 2 — deliberate front-run:
  1. Registry is already UNDER_ATTACK; riskWindowStart is still 0 (no one has poked yet).
  2. Alice submits stake(100) while observing riskWindowStart == 0 in the pool.
  3. Mallory sees Alice's pending tx in the mempool.
  4. Mallory calls pokeRiskWindow() with higher gas, sealing riskWindowStart.
  5. Alice's stake() executes: riskWindowStart already set; stake accepted; withdraw locked.
  6. Same outcome as Scenario 1.
```
