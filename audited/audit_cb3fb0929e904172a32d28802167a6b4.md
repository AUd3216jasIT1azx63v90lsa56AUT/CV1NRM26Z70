### Title
Front-Running `consumableUnits` Stale Read Causes Bundler to Skip Partially-Filled Offers and Revert with `OutOfOffers` - (File: src/periphery/ConsumableUnitsLib.sol)

### Summary
`ConsumableUnitsLib.consumableUnits` reads `consumed[maker][group]` from on-chain state and returns a unit count representing remaining offer capacity. In `MidnightBundles`, this value is used to compute `unitsToTake` before calling `take()`. If another taker fills any portion of the same offer between the `consumableUnits` read and the `take()` call, `unitsToTake` will exceed the actual remaining capacity, causing `take()` to revert with `ConsumedUnits` or `ConsumedAssets`. The `catch {}` block then silently skips the offer entirely — even though it still has remaining capacity — and if no subsequent offers can cover the shortfall, the bundler reverts with `OutOfOffers`.

Then `take()` is called with `unitsToTake`. Inside `Midnight.take()`:

The `consumed` mapping is incremented **before** the cap check. If a front-runner increased `consumed` between the `consumableUnits` read and this line, `newConsumed` will exceed `maxUnits` and revert.

**Exploit flow (units mode, `maxUnits = 100`, `consumed = 0`):**

1. Victim submits a bundle with `targetUnits = 100`, one offer with `maxUnits = 100`.
2. `consumableUnits` reads `consumed = 0`, returns `100`. `unitsToTake = 100`.
3. Attacker (any unprivileged taker) front-runs with `take(offer, ..., 1, ...)` → `consumed` becomes `1`.
4. Victim's `take(offer, ..., 100, ...)` executes: `newConsumed = 1 + 100 = 101 > 100` → reverts `ConsumedUnits`.
5. `catch {}` silently swallows the revert; `filledUnits` stays `0`.
6. Loop ends. `require(0 == 100, OutOfOffers())` → bundle reverts.

The offer still had 99 units of remaining capacity, but the bundler never retried with a smaller amount. The same TOCTOU applies to the `maxAssets` branch (`ConsumedAssets`) and to all four bundle entry-points.

**Why existing checks do not stop it:**

- The `catch {}` is intentionally broad ("Skips every reason why take can revert"), so `ConsumedUnits`/`ConsumedAssets` are caught and the offer is dropped.
- There is no retry with a refreshed `consumableUnits` value after a failed `take()`.
- The attacker only needs to fill **1 unit** to shift `consumed` enough that `unitsToTake` (computed from stale state) overshoots the remaining capacity.

### Impact Explanation
Any unprivileged taker can grief a victim's bundle by front-running with a minimal fill of any offer in the bundle's `takes` array. The victim's transaction reverts with `OutOfOffers`, their intended borrow or lend does not execute, and gas is wasted. The attack is repeatable: the attacker can re-front-run every retry the victim makes, permanently blocking the operation as long as the attacker is willing to pay gas.

### Likelihood Explanation
Preconditions are minimal: the attacker must be able to observe the pending bundle transaction (standard mempool visibility) and must be authorized to call `take()` on the same offer (any address can be a taker if the offer has no gate). The attacker's cost is one `take()` call for 1 unit. The victim's bundle fails entirely. The attack is repeatable at low cost and requires no privileged access.

### Recommendation
After a failed `take()`, do not skip the offer entirely. Instead, re-read `consumableUnits` with fresh state and retry with the updated value before advancing to the next offer. A minimal fix inside the loop:

```solidity
try IMidnight(MIDNIGHT).take(..., unitsToTake, ...) returns (...) {
    filledUnits += unitsToTake;
    ...
} catch {
    // Retry once with a fresh consumed read.
    uint256 retryUnits = min(
        targetUnits - filledUnits,
        takes[i].units,
        ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
    );
    if (retryUnits > 0) {
        try IMidnight(MIDNIGHT).take(..., retryUnits, ...) returns (...) {
            filledUnits += retryUnits;
            ...
        } catch {}
    }
}
```

Alternatively, remove `consumableUnits` from the pre-check `min()` entirely and rely solely on the `catch {}` to handle capacity exhaustion, accepting that zero-unit takes will be attempted for fully-consumed offers.
