### Title
Buy-offer `maxAssets` cap bypassed via zero-rounding `buyerAssets` at sub-WAD price — (`src/Midnight.sol`)

### Summary
When a buy offer has `maxAssets > 0` and its tick yields a price strictly below WAD, the consumed-tracking branch at line 368 adds `buyerAssets = units.mulDivDown(buyerPrice, WAD)` to `consumed[maker][group]`. For `units = 1` and any `buyerPrice < WAD`, integer division floors this to zero, so `consumed` never increments and the `require(newConsumed <= offer.maxAssets)` guard always passes — even when the offer is already fully consumed. The codebase itself confirms this: the NatSpec at line 94 documents it and the test `testBugBuyMaxAssetsBypass` reproduces it, showing that position state (credit, debt, totalUnits) grows while consumed and token balances stay frozen.

### Finding Description
**Code path:**

`src/Midnight.sol` lines 363 and 367–369:
```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

**Root cause:** `mulDivDown(units, buyerPrice, WAD)` computes `floor(units * buyerPrice / WAD)`. For `units = 1` and any `buyerPrice < WAD` (i.e., any tick < `MAX_TICK = 5820`), the result is `floor(buyerPrice / WAD) = 0`. The consumed mapping is incremented by zero, so `newConsumed == consumed_before`, and the cap check trivially passes regardless of how many times the offer has already been filled.

**Attacker inputs:**
- `offer.buy = true`, `offer.maxAssets = N > 0`, `offer.tick` any value in `[2, MAX_TICK - 1]` (price in `[1e12, WAD - 1e12]`)
- `units = 1` per call (or any value where `units * buyerPrice < WAD`)
- Taker is any unprivileged address that is not the maker

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N` at any tick < `MAX_TICK`.
2. Taker calls `take(offer, ..., units=1, ...)` repeatedly.
3. Each call: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; `consumed += 0`; `require(consumed <= N)` passes.
4. Each call: `buyerPos.credit += 1` (maker gains 1 unit of credit), `sellerPos.debt += 1` (taker gains 1 unit of debt), `totalUnits += 1`.
5. No loan tokens are transferred in either direction (`buyerAssets = sellerAssets = 0`).

**Why existing checks fail:** The `require(newConsumed <= offer.maxAssets)` guard is the only cap enforcement. It is defeated because the addend is zero. There is no secondary check requiring `units == 0` when `buyerAssets == 0`, and no minimum-assets floor guard.

### Impact Explanation
The maker's credit exposure grows unboundedly beyond `maxAssets` with zero token cost to the taker. The maker (lender) accumulates credit they did not intend to issue; the taker (borrower) accumulates debt without receiving any loan assets. `totalUnits` inflates, distorting market accounting. The `maxAssets` cap — the maker's primary tool for limiting their credit/debt exposure — is rendered ineffective for any buy offer placed below par.

### Likelihood Explanation
Preconditions are trivially satisfiable: any buy offer at any tick below `MAX_TICK` (the overwhelming majority of real offers, since `MAX_TICK` corresponds to price = 1.0 WAD, i.e., a zero-yield loan) is vulnerable. The taker needs no special privilege, no capital, and no oracle manipulation. The attack is repeatable in a single transaction via `multicall`. The codebase's own test suite (`testBugBuyMaxAssetsBypass`) reproduces it with a single `take(1, ...)` call after pre-consuming the offer to its cap.

### Recommendation
Add a guard that rejects non-zero unit fills that produce zero `buyerAssets` when `maxAssets > 0` and `offer.buy == true`:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroAssetFill()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, fall back to units-based consumed tracking when `buyerAssets` rounds to zero, or enforce a minimum `units` such that `units * buyerPrice >= WAD` when `maxAssets > 0`.

### Proof of Concept
The existing Foundry test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (line 858) is a complete, passing unit test that proves the bug. A fuzz/invariant extension:

```solidity
function testFuzz_BuyMaxAssetsBypassRepeated(uint256 tick, uint256 reps) public {
    tick = bound(tick, 2, MAX_TICK - 1); // any sub-WAD price tick
    reps = bound(reps, 1, 1000);

    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1; // tiny cap
    lenderOffer.tick = tick;

    collateralize(market, borrower, reps + 1);

    for (uint256 i = 0; i < reps; i++) {
        take(1, borrower, lenderOffer);
    }

    // Invariant: consumed must not exceed maxAssets
    assertLe(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
    // Invariant: credit must not exceed maxAssets (violated — this assertion FAILS)
    assertLe(midnight.creditOf(id, lender), lenderOffer.maxAssets);
}
```

**Expected assertions:** `consumed` stays at its pre-fill value (zero, since `buyerAssets = 0` each iteration), `creditOf(lender)` equals `reps`, and `debtOf(borrower)` equals `reps` — all far exceeding `maxAssets = 1`. The credit/debt invariant assertion fails, confirming unbounded overfill. [1](#0-0) [2](#0-1) [3](#0-2)

        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
