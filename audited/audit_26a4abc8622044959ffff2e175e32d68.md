Audit Report

## Title
`sellerAssetsToUnits` Division-by-Zero DoS When `offerPrice == settlementFee` for Buy Offers - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary

`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and passes it as the divisor to `mulDivUp`. When `offerPrice == settlementFee`, `sellerPrice = 0`, and `mulDivUp`'s `d - 1` expression underflows under Solidity 0.8 checked arithmetic, causing a revert. Because `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside any try/catch, the entire bundler call reverts. `Midnight.take` itself handles `sellerPrice = 0` gracefully via `mulDivDown`, so the offer remains directly takeable but not through the bundler.

## Finding Description

**Root cause — `sellerAssetsToUnits`:** [1](#0-0) 

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...
```

When `sellerPrice = 0`, `mulDivUp(targetSellerAssets, WAD, 0)` is called.

**`mulDivUp` underflow:** [2](#0-1) 

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```

`d - 1` with `d = 0` underflows in Solidity 0.8 checked arithmetic → arithmetic revert.

**Reachability of `sellerPrice = 0`:**

`CBP = 1e12` and `PRICE_ROUNDING_STEP = 1e12` are the same granularity unit. [3](#0-2) [4](#0-3) 

The minimum non-zero settlement fee is `1 * CBP = 1e12`. `MAX_SETTLEMENT_FEE_1_DAY = 0.000014e18 = 14000e12`, so `1e12` is a valid fee value. [5](#0-4) 

`tickToPrice(2)` produces `1e12` (the first non-zero price, as prices are rounded to multiples of `PRICE_ROUNDING_STEP`). A buy offer at tick=2 with settlement fee set to `1e12` gives `sellerPrice = 0`. Both values are multiples of `1e12`, so exact equality is achievable.

**Why `Midnight.take` does NOT revert:** [6](#0-5) 

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...
```

`mulDivDown(units, 0, WAD) = (units * 0) / WAD = 0` — no division by zero. The offer is takeable directly.

**Where the bundler reverts (not caught):** [7](#0-6) 

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(   // ← reverts here, NOT in try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
```

The NatDoc at line 175 explicitly states: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." [8](#0-7) 

**`ConsumableUnitsLib` does not independently revert for buy offers:**

For buy offers with `maxUnits > 0`, `consumableUnits` returns `offer.maxUnits.zeroFloorSub(consumed)` without calling `sellerAssetsToUnits`. For buy offers with `maxAssets > 0`, it calls `buyerAssetsToUnits` (not `sellerAssetsToUnits`), which uses `buyerPrice` (= `sellerPrice + settlementFee = settlementFee > 0`) as divisor — no revert. The revert originates solely from the direct `sellerAssetsToUnits` call. [9](#0-8) 

**Asymmetry with `buyerAssetsToUnits`:**

`buyerAssetsToUnits` uses `buyerPrice = sellerPrice + settlementFee` as divisor. When `sellerPrice = 0`, `buyerPrice = settlementFee > 0`, so no division by zero occurs. Only `sellerAssetsToUnits` is unguarded for the `offerPrice == settlementFee` edge case. [10](#0-9) 

## Impact Explanation

`MidnightBundles.supplyCollateralAndSellWithAssetsTarget` permanently reverts for any buy offer where `offerPrice == settlementFee`, regardless of the number of units requested. Borrowers relying on the bundler for atomic collateral supply + borrow cannot execute their borrow flow for such offers. The offer remains takeable via `Midnight.take` directly, so the DoS is scoped to the bundler's assets-target sell path. This constitutes service unavailability for borrowers who depend on the bundler's atomic flow.

## Likelihood Explanation

**Preconditions:**
1. The admin (feeSetter) sets the settlement fee to `1e12` (1 CBP) — a valid, minimum non-zero fee well within `MAX_SETTLEMENT_FEE_1_DAY = 14000e12`.
2. A lender (unprivileged) creates a buy offer at tick=2 (`offerPrice = 1e12`).
3. A borrower calls `supplyCollateralAndSellWithAssetsTarget` with `targetSellerAssets > 0` against that offer.

The attacker (lender) does not need privileged access. The admin setting a fee of `1e12` is a legitimate protocol action. The lender can observe the current fee and deliberately place an offer at the matching tick. The condition is an exact equality between two `1e12`-granularity values, achievable at the minimum tick (tick=2) with the minimum non-zero fee. The condition is repeatable and persistent until the fee or offer changes.

## Recommendation

Add a guard in `sellerAssetsToUnits` analogous to the implicit guard in `buyerAssetsToUnits`. When `sellerPrice == 0`, either revert explicitly (mirroring `Midnight.take`'s behavior of treating such offers as having zero seller proceeds) or return a sentinel value. The simplest fix:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
require(sellerPrice > 0 || targetSellerAssets == 0, ZeroSellerPrice());
return offer.buy
    ? (sellerPrice == 0 ? type(uint256).max : targetSellerAssets.mulDivUp(WAD, sellerPrice))
    : targetSellerAssets.mulDivDown(WAD, sellerPrice);
```

Alternatively, add an explicit `require(sellerPrice > 0)` check with a descriptive error, consistent with the NatDoc's documented revert conditions.

## Proof of Concept

1. Deploy `Midnight` and `MidnightBundles` on a local fork.
2. Admin calls `setMarketSettlementFee(id, 0, 1e12)` (sets the 0-day breakpoint fee to 1 CBP = 1e12).
3. Lender creates a buy offer at tick=2 (price = `tickToPrice(2) = 1e12`) with `maxUnits > 0`.
4. Borrower supplies collateral and calls `supplyCollateralAndSellWithAssetsTarget` with `targetSellerAssets = 1` and the lender's offer.
5. Observe: the call reverts with an arithmetic underflow inside `mulDivUp` (via `sellerAssetsToUnits`).
6. Confirm: calling `Midnight.take` directly with the same offer and `units = 1` succeeds and returns `sellerAssets = 0`.
