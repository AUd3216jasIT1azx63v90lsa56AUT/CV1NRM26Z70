### Title
Buy offer `maxAssets` cap bypassed via dust rounding when `buyerPrice < WAD` - (`src/Midnight.sol`)

### Summary
When a buy offer (`offer.buy == true`) uses `maxAssets` as its fill cap and the offer's `buyerPrice < WAD`, a taker can call `take` with `units = 1` after the offer is fully consumed. Because `buyerAssets = 1.mulDivDown(buyerPrice, WAD) = 0`, the `ConsumedAssets` check passes (`newConsumed = maxAssets + 0 <= maxAssets`), yet `units = 1` is still processed, creating real credit and debt with zero asset transfer. This can be repeated indefinitely.

### Finding Description
In `src/Midnight.sol`, the `take` function computes:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
``` [1](#0-0) 

Then enforces the cap:

```solidity
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
``` [2](#0-1) 

When `buyerPrice < WAD` (tick set below `MAX_TICK`) and `units = 1`, `mulDivDown(1, buyerPrice, WAD)` rounds down to `0`. If `consumed[maker][group]` already equals `offer.maxAssets`, then `newConsumed = maxAssets + 0 = maxAssets`, which satisfies `<= maxAssets`. The check passes, but the rest of `take` still processes `units = 1`, updating `buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` by 1 unit each, with no token transfer (`buyerAssets = sellerAssets = 0`).

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = N` and `tick < MAX_TICK` (so `buyerPrice < WAD`).
2. Taker (or anyone) fills the offer normally until `consumed[maker][group] == N`.
3. Taker calls `take(offer, ..., units=1, ...)` repeatedly.
4. Each call: `buyerAssets = 0`, `newConsumed = N + 0 = N <= N` → passes. Lender gains +1 credit, borrower gains +1 debt, `totalUnits` increases by 1. No assets move.
5. Step 4 is unbounded — the taker can repeat it indefinitely.

The existing test `testBugBuyMaxAssetsBypass` in the test suite explicitly documents and confirms this behavior: [3](#0-2) 

The test asserts `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all increase despite `consumed` staying at `maxAssets` and both `buyerAssets` and `sellerAssets` being zero.

### Impact Explanation
The `maxAssets` cap on buy offers is completely ineffective for dust fills when `buyerPrice < WAD`. An unprivileged taker can force the maker (lender) to accumulate unbounded credit beyond their stated limit, and force the taker's own debt to grow without paying any assets. This violates the core invariant that "offers cannot be overfilled" and that "every credit has matching debt or valid settled/loss state" (credit is created without a corresponding asset payment).

### Likelihood Explanation
The precondition `buyerPrice < WAD` is reachable whenever `tick < MAX_TICK`, which is the common case for any offer priced below par. The attacker only needs to: (a) know the offer parameters, (b) fill the offer to `maxAssets`, and (c) call `take` with `units = 1`. No special privileges are required. The attack is repeatable in a single transaction via multicall or a loop, making it trivially automatable.

### Recommendation
Add a guard that rejects a fill where `units > 0` but `buyerAssets == 0` (for buy offers) or `sellerAssets == 0` (for sell offers with `maxAssets`). Concretely, after computing `buyerAssets`/`sellerAssets` and before the consumed check:

```solidity
require(units == 0 || (offer.buy ? buyerAssets > 0 : sellerAssets > 0), ZeroAssets());
```

This ensures that any non-zero unit fill always contributes a non-zero amount to the consumed counter, making the cap enforceable.

### Proof of Concept
The existing Foundry unit test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) is a complete PoC. Key assertions that confirm the bug:

```solidity
// consumed does not increase (0 added), so cap check passes:
assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
// No tokens moved:
assertEq(loanToken.balanceOf(lender), lenderBalBefore);
assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
// But position state changed — invariant broken:
assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
assertGt(midnight.totalUnits(id), totalUnitsBefore);
``` [4](#0-3) 

A fuzz extension would call `take(units=1, ...)` in a loop `N` times after full consumption and assert `creditOf(lender) <= maxAssets` — this assertion will fail, confirming unbounded bypass.
