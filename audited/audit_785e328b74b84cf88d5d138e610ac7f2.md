### Title
Buy-offer `maxAssets` cap bypassed via `mulDivDown` zero-rounding when `buyerPrice < WAD` - (File: src/Midnight.sol)

### Summary
When a buy offer sets `maxAssets > 0` and the offer's `buyerPrice < WAD` (tick below `MAX_TICK`), a taker can call `take()` with `units` small enough that `mulDivDown(units, buyerPrice, WAD) = 0`. The consumed counter does not advance, yet credit, debt, and `totalUnits` all increase. The offer can be filled indefinitely in unit terms regardless of `maxAssets`.

### Finding Description
**Code path:**

In `src/Midnight.sol` lines 363 and 367–369:
```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause:** `buyerAssets` is computed with `mulDivDown`, which truncates. When `buyerPrice < WAD` (any tick strictly below `MAX_TICK`) and `units` is chosen so that `units * buyerPrice < WAD`, the result is `buyerAssets = 0`. The consumed accumulator adds 0, so `newConsumed` never exceeds `maxAssets`. The `ConsumedAssets` guard is permanently satisfied regardless of how many times `take()` is called.

**Attacker inputs:**
- `offer.buy = true`, `offer.maxAssets = N` (any N ≥ 1), `offer.tick < MAX_TICK` (so `buyerPrice < WAD`)
- `units = 1` (or any value where `units * buyerPrice < WAD`)

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N` and a tick where `buyerPrice < WAD`.
2. Attacker (taker, or an operator authorized via `EcrecoverAuthorizer.setIsAuthorized`) calls `take(units=1)` repeatedly.
3. Each call: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; consumed stays at its current value; `ConsumedAssets` check passes.
4. Each call still executes the full position update: `buyerPos.credit += buyerCreditIncrease`, `sellerPos.debt += sellerDebtIncrease`, `_marketState.totalUnits` increases.
5. The offer is filled unboundedly in unit terms while `consumed` never reaches `maxAssets`.

**Why existing checks fail:** The only guard on unit-fill quantity under `maxAssets` mode is `newConsumed <= offer.maxAssets`. Because `buyerAssets` rounds to 0, this check is trivially satisfied on every call. There is no secondary check bounding total units filled against `maxAssets`.

The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) explicitly reproduces this: the offer is pre-consumed to `maxAssets = 1`, yet `take(1)` succeeds, `consumed` stays at 1, and credit/debt/totalUnits all increase.

### Impact Explanation
A buy offer maker who intends to cap their total asset exposure at `maxAssets` has that cap completely bypassed. An attacker can fill the offer for an unbounded number of units at zero asset cost (since `buyerAssets = 0`, no tokens are transferred), accruing credit and forcing debt onto the seller side without any payment. This violates the core invariant that "offers cannot be overfilled" and that "contract balances cover credit redemption."

### Likelihood Explanation
**Preconditions:**
- Buy offer with `maxAssets > 0` and `tick < MAX_TICK` (i.e., `buyerPrice < WAD`). This is the common case — `MAX_TICK` corresponds to price = 1.0 and most lending markets operate below par.
- Attacker needs to be a valid taker: either `msg.sender == taker` (direct) or authorized via `isAuthorized[taker][msg.sender]` (e.g., through `EcrecoverAuthorizer.setIsAuthorized`).

**Feasibility:** Trivially reachable. No special state is required beyond a normally posted buy offer. The attacker simply calls `take()` with `units = 1` in a loop (or via multicall). No oracle manipulation or admin access needed.

**Repeatability:** Unlimited — each call adds 0 to consumed, so the condition never closes.

### Recommendation
Replace the `mulDivDown` consumed accounting for buy offers with a unit-denominated cap check, or add a secondary guard. Concretely, when `offer.maxAssets > 0` and `offer.buy = true`, also enforce that `buyerAssets > 0` before accepting the fill (i.e., revert if `buyerAssets == 0` and `units > 0`). Alternatively, track consumed in units (not assets) and convert `maxAssets` to a unit ceiling at offer creation, or use `mulDivUp` for the consumed accounting so that rounding always favors the cap rather than the taker.

function testMaxAssetsBypassZeroBuyerAssets() public {
    // Setup: buy offer, buyerPrice < WAD (tick below MAX_TICK)
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = 1e6; // e.g. 1 USDC-equivalent
    lenderOffer.tick = MAX_TICK - 16; // buyerPrice < WAD

    collateralize(market, borrower, 1000);
    deal(address(loanToken), borrower, 0);

    uint256 creditBefore = midnight.creditOf(id, lender);

    // Each call: buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0
    // consumed never advances, cap never triggers
    for (uint256 i = 0; i < 500; i++) {
        take(1, borrower, lenderOffer);
    }

    // Assert: consumed still 0 (cap not consumed), but 500 units filled
    assertEq(midnight.consumed(lender, lenderOffer.group), 0);
    assertEq(midnight.creditOf(id, lender), creditBefore + 500);

    // Invariant violated: units_filled * buyerPrice / WAD >> maxAssets
    // 500 * buyerPrice / WAD >> 1e6 is false only if buyerPrice is tiny,
    // but the point is units were filled with zero asset payment.
}
