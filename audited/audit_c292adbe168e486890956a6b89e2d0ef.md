### Title
Debt Increase Permitted at Exact Maturity Boundary via `<=` Operator in `CannotIncreaseDebtPostMaturity` Check - (File: src/Midnight.sol)

### Summary

`Midnight.take` computes `timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp)`, which returns `0` when `block.timestamp == maturity` because `zeroFloorSub` uses `gt(x, y)` (strictly greater) in assembly. The guard against post-maturity debt increase at line 391 uses `block.timestamp <= offer.market.maturity`, which evaluates to `true` at the exact equality point, allowing `sellerDebtIncrease > 0` to pass. This violates the protocol's stated invariant that "maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing."

### Finding Description

**Root cause — `src/Midnight.sol` lines 359 and 391:**

```solidity
// line 359
uint256 timeToMaturity = UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp);
// line 391
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

`UtilsLib.zeroFloorSub` is implemented as `mul(gt(x, y), sub(x, y))` (`src/libraries/UtilsLib.sol` line 24). When `x == y`, `gt(x, y) = 0`, so the result is `0`. At `block.timestamp == maturity`, `timeToMaturity = 0`.

The guard on line 391 uses `<=`, so at `block.timestamp == maturity` the condition `block.timestamp <= offer.market.maturity` is `true`, and the `require` passes unconditionally regardless of `sellerDebtIncrease`.

**Downstream effects at `timeToMaturity = 0`:**

- `buyerPendingFeeIncrease = buyerCreditIncrease.mulDivDown(continuousFee * 0, WAD) = 0` (line 385–386). New lender credit carries zero continuous-fee obligation.
- `settlementFee(id, 0)` falls into the `timeToMaturity < 1 days` branch (line 971), interpolating to exactly `settlementFeeCbp0 * CBP` — the 0-day breakpoint fee. This is the same fee used post-maturity, so fee accounting is internally consistent but the debt creation itself should not have been permitted.

**Asymmetry with post-maturity liquidation:**

Post-maturity liquidation (line 622) requires `block.timestamp > market.maturity` (strictly greater). At `block.timestamp == maturity`, post-maturity liquidation is unavailable. Normal-mode liquidation requires the position to be unhealthy. Therefore, a healthy borrower who increases debt at exactly maturity holds a position that is neither health-liquidatable nor post-maturity-liquidatable at that timestamp — a one-block window of immunity.

**Exploit path:**

1. Borrower (or any taker acting as seller) pre-positions collateral so the resulting position will be healthy.
2. Warp/wait to `block.timestamp == offer.market.maturity`.
3. Call `take` on a sell offer (`offer.buy = false`, `offer.maker = borrower`) with `units > sellerPos.credit` so `sellerDebtIncrease > 0`.
4. Line 391 passes because `maturity <= maturity` is `true`.
5. `sellerPos.debt += sellerDebtIncrease` executes (line 414).
6. Health check at line 476 passes if collateral is sufficient.
7. Borrower holds new debt at maturity; post-maturity liquidation is unavailable for this block.

### Impact Explanation

The core invariant `"debt_increase_after_maturity_forbidden": true` and the explicit maturity boundary invariant `"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"` are both violated. A borrower can open or increase a debt position at exactly maturity — a timestamp at which the position is immediately overdue — while post-maturity liquidation is blocked for that block. The new lender credit carries `buyerPendingFeeIncrease = 0`, meaning no continuous fee is crystallized for the lender on this credit, consistent with TTM=0 but only valid if the trade itself were legitimate. The position is properly collateralized (health check enforced), so direct fund loss requires the borrower to subsequently default, but the invariant breach is concrete and reproducible.

### Likelihood Explanation

The precondition is a single specific `block.timestamp == maturity`. An attacker who controls transaction submission (e.g., a borrower or MEV searcher) can target this block deterministically. The offer need only have a non-expired `expiry` covering that timestamp and sufficient collateral pre-supplied. The condition is repeatable for every market whose maturity falls on a reachable block timestamp. No privileged access is required.

### Recommendation

Change the `CannotIncreaseDebtPostMaturity` guard from `<=` to `<`:

```solidity
// src/Midnight.sol line 391
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns with the post-maturity liquidation condition (`block.timestamp > market.maturity`) and closes the one-second boundary gap. The `timeToMaturity` computation via `zeroFloorSub` is correct and does not need to change; the fix is solely in the guard predicate.

### Proof of Concept

```solidity
function testDebtIncreaseAtExactMaturity() public {
    // Setup: market with maturity = now + 100 (from TakeTest setUp)
    uint256 units = 50;
    // Pre-supply collateral so health check passes after debt increase
    collateralize(market, borrower, units);

    // Warp to exactly maturity
    vm.warp(market.maturity);

    // Extend offer expiry to cover this timestamp
    borrowerOffer.expiry = market.maturity;
    borrowerOffer.maxUnits = units;

    deal(address(loanToken), lender, units);

    // This should revert with CannotIncreaseDebtPostMaturity but does NOT
    // because block.timestamp <= maturity is true at equality
    take(units, lender, borrowerOffer);

    // Assert: debt was increased at maturity — invariant violated
    assertGt(midnight.debtOf(id, borrower), 0, "debt increased at maturity");

    // Assert: post-maturity liquidation is NOT yet available at this timestamp
    vm.expectRevert(IMidnight.NotLiquidatable.selector);
    midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");
}
```

Expected assertions: `debtOf(id, borrower) > 0` (debt created at maturity) and `liquidate` with `postMaturityMode=true` reverts with `NotLiquidatable` — both confirming the invariant breach.
