### Title
Fee-on-transfer collateral token inflates recorded collateral, enabling undercollateralized borrowing - (File: src/Midnight.sol)

### Summary

`supplyCollateral` records `assets` as the deposited collateral amount before calling `safeTransferFrom`, with no balance-before/after reconciliation. When the collateral token is fee-on-transfer, the contract receives `assets - fee` but credits the borrower with `assets`, overstating their collateral. The health check in `isHealthy` then computes `maxDebt` from the inflated stored value, allowing the borrower to take on more debt than their actual on-chain collateral supports.

`SafeTransferLib.safeTransferFrom` only verifies the call succeeded and returned `true`; it does not measure the actual balance delta. [2](#0-1) 

`isHealthy` then computes `maxDebt` directly from `_position.collateral[i]`, which is the inflated `assets` value:

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
``` [3](#0-2) 

**Root cause:** Effects (state write) precede the interaction (transfer), and no balance reconciliation is performed after the transfer. The recorded amount is always `assets`, regardless of what the token actually delivers.

**Attacker inputs and exploit flow:**

1. Attacker (market creator, unprivileged) calls `touchMarket` with a fee-on-transfer token as the collateral token. Market creation is permissionless — `touchMarket` has no token-type restrictions. [4](#0-3) 

2. Attacker (as borrower) calls `supplyCollateral(market, i, assets, onBehalf)`. The protocol writes `collateral[i] = assets`. The token delivers `assets - fee`. The bitmap bit `i` is set.

3. Attacker calls into the borrow path (via `take` as a sell-offer maker). `isHealthy` is called, computing `maxDebt` from the inflated `collateral[i] = assets` rather than the true `assets - fee`. The borrower passes the health check and receives more debt than their real collateral supports.

4. The position is undercollateralized from inception. When the oracle price drops even slightly, the position becomes liquidatable with insufficient collateral to cover the debt, realizing bad debt.

**Why existing checks fail:**

- `SafeTransferLib` checks only call success and return value, not the received amount.
- `touchMarket` validates LLTV tiers, sorted collateral addresses, and `maxLif` — but imposes no restriction on token type (no fee-on-transfer exclusion).
- The Certora `Solvency.spec` models `safeTransferFrom` as `NONDET`, so it does not catch this discrepancy under fee-on-transfer semantics. [5](#0-4) 

### Impact Explanation

The borrower's `collateral[i]` is overstated by `fee`. `maxDebt` is overstated by `fee * price * lltv / (ORACLE_PRICE_SCALE * WAD)`. The borrower can draw debt exceeding the value of their actual on-chain collateral. Upon liquidation, the seized collateral is worth less than the repaid debt, producing bad debt that is socialized across lenders via the `lossFactor` mechanism. The solvency invariant — contract balance covers all recorded collateral — is broken from the moment of supply.

### Likelihood Explanation

**Preconditions:**
- A fee-on-transfer token must be used as a collateral token in a market. Market creation is permissionless; the attacker can deploy their own fee-on-transfer ERC20 and create a market around it.
- The attacker must find or create a lender willing to supply loan tokens to that market (or be the lender themselves in a self-dealing scenario targeting other lenders who join later).

**Feasibility:** High — no privileged access is required. The attacker controls the collateral token contract and the market creation call. The exploit is repeatable on every `supplyCollateral` call where `oldCollateral == 0` (first supply, triggering bitmap set) or on any subsequent supply (collateral is always overstated by the fee amount).

### Recommendation

Measure the actual received amount using a balance-before/balance-after pattern in `supplyCollateral`, and use the delta as the credited amount:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;

_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
if (oldCollateral == 0 && received > 0) { ... }
```

Alternatively, document that fee-on-transfer tokens are unsupported and add a runtime check (e.g., require `received == assets`), reverting if a fee was taken.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
