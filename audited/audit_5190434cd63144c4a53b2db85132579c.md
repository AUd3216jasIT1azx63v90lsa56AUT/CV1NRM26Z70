### Title
Unchecked `takerCallback` as `payer` enables draining any approved `IBuyCallback` contract via sell-offer take - (`src/Midnight.sol`)

### Summary
When a taker takes a sell offer (`offer.buy = false`), `takerCallback` is an attacker-supplied parameter that becomes both `buyerCallback` and `payer` with no authorization check against the address it names. Any contract that has approved Midnight for `loanToken` and implements `IBuyCallback` returning `CALLBACK_SUCCESS` can be drained of `buyerAssets` in a single `take` call by an unprivileged taker.

### Finding Description

**Exact code path:**

`offer.buy = false` (sell offer). At line 375: [1](#0-0) 

`buyer = taker` (attacker), `seller = offer.maker`.

At lines 420–422: [2](#0-1) 

- `buyerCallback = takerCallback` (attacker-controlled, set to `victimAddress`)
- `sellerCallback = offer.callback` (= `address(0)` per precondition, no seller callback)
- `payer = buyerCallback = victimAddress` (since `buyerCallback != address(0)`)

At lines 445–453, `IBuyCallback(victimAddress).onBuy(...)` is called: [3](#0-2) 

Then at lines 455–456, tokens are pulled from `payer = victimAddress`: [4](#0-3) 

Total pulled from victim: `buyerAssets` (= `buyerAssets - sellerAssets` to Midnight + `sellerAssets` to `receiver`).

**Root cause — missing authorization check on `takerCallback`:**

The only authorization check in `take` is: [5](#0-4) 

This checks that `msg.sender` is authorized for `taker`, not for `takerCallback`. There is no check of the form `require(takerCallback == address(0) || takerCallback == taker || isAuthorized[takerCallback][msg.sender])`.

**Why `EcrecoverRatifier` does not stop it:**

`EcrecoverRatifier.isRatified` validates only the maker's signed `Offer` struct — Merkle proof, root cancellation, and ECDSA signature: [6](#0-5) 

`takerCallback` is not a field of `Offer` and is never signed. The ratifier has no visibility into it whatsoever.

**Attacker-controlled inputs:**
- `takerCallback = victimAddress` — fully attacker-chosen
- `takerCallbackData` — arbitrary bytes passed to `onBuy`
- `units` — controls `buyerAssets` magnitude
- `receiver` — attacker-chosen destination for `sellerAssets`

**Why the victim's `onBuy` returning `CALLBACK_SUCCESS` is a realistic precondition:**
Any contract designed to participate in Midnight as a buyer callback (e.g., a smart-contract aggregator, vault, or router) will: (a) hold a max approval to Midnight for `loanToken`, and (b) return `CALLBACK_SUCCESS` from `onBuy`. If `onBuy` does not verify that the contract itself initiated the take (e.g., by checking an internal flag set before calling `take`), it is exploitable. Many such contracts will not have this guard because the protocol gives no indication that `takerCallback` requires it.

### Impact Explanation
Direct, complete loss of `buyerAssets` worth of `loanToken` from the victim contract per exploit call. The attacker pays nothing (they are the `taker`/`buyer` whose position receives credit; the victim pays as `payer`). The attacker can repeat this for any amount up to the victim's balance and approval, and can direct `sellerAssets` to an arbitrary `receiver`.

### Likelihood Explanation
**Preconditions:**
1. A valid sell offer exists (signed via `EcrecoverRatifier`, `offer.callback = address(0)`) — normal protocol usage.
2. A victim contract exists with `loanToken.approve(midnight, type(uint256).max)` and `onBuy` returning `CALLBACK_SUCCESS` — any Midnight-integrated buyer callback contract satisfies this.
3. Attacker is any unprivileged taker — no special role required.

The attack is fully repeatable, requires no flash loan, no oracle manipulation, and no governance action. It is executable in a single transaction.

### Recommendation
Add an authorization check on `takerCallback` before it can be used as `payer`. The fix should mirror the existing `taker` authorization pattern:

```solidity
require(
    takerCallback == address(0) || takerCallback == taker || isAuthorized[takerCallback][msg.sender],
    TakerCallbackUnauthorized()
);
```

This ensures that only the taker themselves, or an address that has explicitly authorized `msg.sender`, can be designated as the buyer callback/payer. Alternatively, restrict `payer` to always be `msg.sender` or `taker` when `takerCallback` is set, and require the callback to pull funds internally.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {EcrecoverRatifier} from "src/ratifiers/EcrecoverRatifier.sol";
import {IBuyCallback, CALLBACK_SUCCESS} from "src/interfaces/ICallbacks.sol";
import {Offer, Market} from "src/interfaces/IMidnight.sol";

// Victim: a legitimate buyer-callback contract with max approval
contract VictimBuyerCallback is IBuyCallback {
    address public loanToken;
    address public midnight;
    constructor(address _loanToken, address _midnight) {
        loanToken = _loanToken;
        midnight = _midnight;
        IERC20(_loanToken).approve(_midnight, type(uint256).max);
    }
    function onBuy(...) external returns (bytes32) {
        return CALLBACK_SUCCESS; // no internal-initiation guard
    }
}

contract TakerCallbackPayerConfusionTest is Test {
    function test_drainVictimViaTakerCallback() public {
        // Setup: deploy Midnight, EcrecoverRatifier, mock loanToken
        // Fund victim with loanToken; victim approves Midnight (max)
        // Maker signs a valid sell offer (offer.buy=false, offer.callback=address(0))
        // Attacker calls:
        //   midnight.take(offer, ratifierData, units, attacker, receiver, victimAddress, "")
        
        uint256 victimBalanceBefore = loanToken.balanceOf(address(victim));
        
        vm.prank(attacker);
        (uint256 buyerAssets,) = midnight.take(
            offer, ratifierData, units, attacker, attacker, address(victim), ""
        );
        
        uint256 victimBalanceAfter = loanToken.balanceOf(address(victim));
        
        // Assert: victim lost exactly buyerAssets
        assertEq(victimBalanceBefore - victimBalanceAfter, buyerAssets);
        // Assert: attacker's position received credit (units)
        assertEq(midnight.creditOf(id, attacker), units); // or debt reduced
    }
}
```

**Expected assertions:**
- `victimBalanceBefore - victimBalanceAfter == buyerAssets` — victim drained
- Attacker paid zero from their own balance
- Fuzz over `units` and `victimAddress` (any address with nonzero approval and `onBuy` returning `CALLBACK_SUCCESS`) to confirm generality
