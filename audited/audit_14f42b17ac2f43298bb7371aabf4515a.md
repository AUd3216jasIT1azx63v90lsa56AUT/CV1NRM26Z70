Audit Report

## Title
Fully-Consumed Assets-Based Buy Offer With `buyerPrice < WAD` Bypasses Consumption Guard, Allowing Unbounded Position Mutation - (`src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and its effective `buyerPrice < WAD`, the consumed-assets guard can be bypassed indefinitely after the offer is fully consumed. Supplying `units = 1` produces `buyerAssets = mulDivDown(1 * buyerPrice, WAD) = 0`, so `consumed` does not increase and the `require(newConsumed <= maxAssets)` check passes. All downstream position mutations — buyer credit, seller debt, and `totalUnits` — still execute against `units = 1`, corrupting the credit/debt accounting invariant without any corresponding asset transfer.

## Finding Description
**Exact code path — `src/Midnight.sol`:**

At line 363, `buyerAssets` is computed with `mulDivDown`, which truncates toward zero: [1](#0-0) 

At lines 367–369, the consumed guard increments by `buyerAssets` (which is `0`) and checks `<= maxAssets`: [2](#0-1) 

When `consumed == maxAssets` already and `buyerAssets == 0`, the addition is a no-op and the `require` passes. Execution continues unconditionally to the position mutation block at lines 408–417, which operates on `units`, not `buyerAssets`: [3](#0-2) 

**Root cause:** The guard enforces an asset cap but has no corresponding check that `units == 0` when `consumed >= maxAssets`, and no check that `buyerAssets > 0` when `units > 0`. The rounding mode (`mulDivDown`) for buy offers allows `units > 0` to produce `buyerAssets = 0` whenever `buyerPrice < WAD`.

**Why existing checks fail:**
- `require(newConsumed <= offer.maxAssets)` — passes because `maxAssets + 0 ≤ maxAssets`.
- No guard enforces `units == 0` when the offer is fully consumed.
- No guard enforces `buyerAssets > 0` when `units > 0` on a buy offer.

**Protocol acknowledgment:** The codebase explicitly documents this behavior at line 94: [4](#0-3) 

**Confirmed by test:** `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` reproduces the exact scenario — zero asset transfers, unchanged `consumed`, but strictly increased `creditOf`, `debtOf`, and `totalUnits`: [5](#0-4) 

## Impact Explanation
Each post-consumption call with `units = 1`:
- Increases the maker's (buyer's) `credit` by `buyerCreditIncrease` derived from `units = 1`, with zero loan tokens paid.
- Increases the taker's (seller's) `debt` by `sellerDebtIncrease` derived from `units = 1`, with zero loan tokens received.
- Increases `totalUnits` by `buyerCreditIncrease` with no asset backing.

The maker's credit grows without bound at negligible gas cost. Because `totalUnits` is inflated without corresponding assets, legitimate lenders' pro-rata claims at maturity are diluted. An attacker controlling both the maker address and a throwaway taker address (with no collateral) can inflate the maker's credit arbitrarily; the taker's resulting bad debt is socialized among all lenders, directly reducing their recoverable assets. This constitutes unauthorized state change, accounting corruption, and indirect theft from other lenders.

## Likelihood Explanation
All preconditions are attacker-reachable without any privileged access:
1. `offer.buy = true`, `offer.maxAssets > 0` — standard offer configuration.
2. `buyerPrice < WAD` — achieved by setting `offer.tick` to any value where `tickToPrice(tick) + settlementFee < WAD` (e.g., `MAX_TICK - 16` as used in the test).
3. Offer fully consumed — the attacker can self-consume as the first taker, or wait for organic fills.

The attack is repeatable every block with no token transfers, making it essentially free beyond gas. Any external account can act as the taker.

## Recommendation
Add a guard that prevents `units > 0` from being processed when `buyerAssets` rounds to zero on a buy offer with an asset cap. The minimal fix is to require that `buyerAssets > 0` whenever `units > 0` and `offer.buy == true` with `offer.maxAssets > 0`:

```solidity
if (offer.maxAssets > 0 && offer.buy && units > 0) {
    require(buyerAssets > 0, ZeroBuyerAssets());
}
```

Alternatively, enforce that the offer is not yet fully consumed before allowing any non-zero `units` fill:

```solidity
if (offer.maxAssets > 0) {
    uint256 prevConsumed = consumed[offer.maker][offer.group];
    require(units == 0 || prevConsumed < offer.maxAssets, ConsumedAssets());
    newConsumed = prevConsumed + (offer.buy ? buyerAssets : sellerAssets);
    consumed[offer.maker][offer.group] = newConsumed;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This preserves the intentional no-op (`units = 0`) callback path documented at line 93 while closing the bypass for `units > 0`.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` at `test/TakeTest.sol:858` is a complete, passing reproduction: [6](#0-5) 

Steps:
1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.tick = MAX_TICK - 16` (ensures `buyerPrice < WAD`).
2. Pre-consume the offer to `consumed == maxAssets` via `setConsumed`.
3. Call `take(1, borrower, lenderOffer)`.
4. Assert `buyerAssets == 0`, `sellerAssets == 0`, token balances unchanged, `consumed` unchanged.
5. Assert `creditOf(lender) > lenderCreditBefore`, `debtOf(borrower) > borrowerDebtBefore`, `totalUnits > totalUnitsBefore`.

All five assertions pass, confirming the invariant violation.
