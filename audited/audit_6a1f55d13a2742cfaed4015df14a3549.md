### Title
Lenders Can Front-Run Bad-Debt Liquidations to Escape Loss Socialization — (`File: src/Midnight.sol`)

---

### Summary

`Midnight.sol` socializes bad debt by increasing `marketState[id].lossFactor` inside `liquidate()`. The `withdraw()` function applies the current `lossFactor` at call time via `_updatePosition()`. A lender who observes a pending liquidation that will realize bad debt can front-run it, withdrawing at the pre-loss factor and receiving full face-value tokens, while the remaining lenders absorb a disproportionately larger share of the loss.

---

### Finding Description

**Vulnerability class**: Front-running / TOCTOU on a global accounting variable (direct analog to front-running a negative rebase).

**Step 1 — Bad debt is realized inside `liquidate()`.**

When `badDebt > 0`, the function increases `lossFactor` and decreases `totalUnits`:

```solidity
// src/Midnight.sol lines 626–634
if (badDebt > 0) {
    _position.debt -= uint128(badDebt);
    uint256 _totalUnits = _marketState.totalUnits;
    uint256 _lossFactor = _marketState.lossFactor;
    _marketState.lossFactor = UtilsLib.toUint128(
        type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
    );
    _marketState.totalUnits -= UtilsLib.toUint128(badDebt);
    ...
}
```

`lossFactor` is the protocol's analog to `balancePerShare`: it encodes the cumulative proportional loss every lender must absorb. [1](#0-0) 

**Step 2 — `withdraw()` applies `lossFactor` at call time.**

```solidity
// src/Midnight.sol lines 481–499
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    ...
    _updatePosition(market, id, onBehalf);   // applies current lossFactor
    ...
    _position.credit -= UtilsLib.toUint128(units);
    _marketState.withdrawable -= UtilsLib.toUint128(units);
    ...
    SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
}
``` [2](#0-1) 

**Step 3 — `_updatePosition` / `updatePositionView` computes post-slash credit.**

```solidity
// src/Midnight.sol lines 805–806
uint256 postSlashCredit = _lastLossFactor < type(uint128).max
    ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
    : 0;
```

If `withdraw()` is called **before** the liquidation, `marketState[id].lossFactor == _lastLossFactor`, so `postSlashCredit == credit` — no reduction. If called **after**, `lossFactor` is higher and `postSlashCredit < credit`. [3](#0-2) 

**Exploit path:**

1. Market has `withdrawable > 0` (from prior repayments by other borrowers).
2. Attacker is a lender with credit `C` in the same market.
3. A liquidation transaction is broadcast that will realize `badDebt = D` against `totalUnits = T`.
4. Attacker front-runs with `withdraw(market, C, attacker, attacker)`.
5. `_updatePosition` sees the old `lossFactor` → `postSlashCredit = C` → attacker receives `C` tokens.
6. Liquidation executes: `lossFactor` increases, `totalUnits` drops to `T - D`.
7. Remaining lenders' credit is now slashed by `D / (T - C)` instead of `D / T`, bearing the attacker's share of the loss.

---

### Impact Explanation

The loss-socialization invariant is broken: a lender who front-runs a bad-debt liquidation exits at full face value, shifting their proportional share of the bad debt (`C * D / T`) onto the remaining lenders. In a market with large bad debt and significant withdrawable liquidity, this can represent a material transfer of value from honest lenders to the front-runner. The `withdrawable` constraint limits the maximum extractable amount per attack but does not prevent the attack.

**Impact: High** (direct, quantifiable theft from other lenders via bypassed loss socialization).

---

### Likelihood Explanation

Preconditions required:
- Attacker holds credit in a market (`take()` as buyer — permissionless).
- `withdrawable > 0` in that market (common when some borrowers have repaid while others have bad debt).
- A pending `liquidate()` transaction is visible in the mempool that will realize `badDebt > 0`.
- Attacker can submit a higher-gas-price `withdraw()` to front-run it.

All preconditions are reachable by an unprivileged actor with mempool access. MEV bots routinely monitor for liquidation opportunities; the same infrastructure detects and front-runs them.

**Likelihood: Low** (requires the specific co-occurrence of withdrawable funds and a pending bad-debt liquidation, but the attack is fully permissionless and mechanically straightforward for a MEV bot).

---

### Recommendation

Apply the updated `lossFactor` **before** allowing withdrawal, which `_updatePosition` already does — the gap is that `lossFactor` is not yet updated when the liquidation is still pending. Possible mitigations:

1. **Withdrawal delay / epoch-based settlement**: Introduce a minimum time between credit acquisition and withdrawal, making front-running economically unattractive.
2. **Snapshot-based loss application**: Record a "pending bad debt" flag or a pre-committed `lossFactor` update that `withdraw()` must respect even before the liquidation transaction is mined (e.g., a two-phase commit for bad debt realization).
3. **Withdrawal fee**: A small fee on `withdraw()` makes the marginal profit from front-running a single liquidation negative in most cases.

### Proof of Concept

**Setup:**
- Market with `loanToken = USDC`, two lenders A (attacker, credit = 500) and B (honest, credit = 500), `totalUnits = 1000`, `withdrawable = 400` (from a prior repayment by Borrower X).
- Borrower Y has `debt = 200`, collateral worth 100 → `badDebt = 100`.

**Without front-running:**
- Liquidation executes: `lossFactor` increases by `100/1000 = 10%`.
- A's effective credit: `500 * 0.9 = 450`. B's effective credit: `450`.
- Both lenders share the 100-unit bad debt equally.

**With front-running:**
1. Liquidator broadcasts `liquidate(market, ..., borrowerY, ...)`.
2. Attacker sees it in the mempool, broadcasts `withdraw(market, 400, attacker, attacker)` with higher gas.
3. Attacker's `_updatePosition`: `lossFactor` unchanged → `postSlashCredit = 500` → attacker withdraws 400 tokens (limited by `withdrawable`).
4. Liquidation executes: `totalUnits = 600`, `lossFactor` increases by `100/600 ≈ 16.7%`.
5. B's effective credit: `500 * (1 - 0.167) = 416.5` — B absorbs ~83.5 units of loss instead of 50.
6. Attacker's remaining credit: `100 * (1 - 0.167) = 83.5`.

Attacker avoided `~16.5` units of loss that B now bears. Scaled to realistic market sizes (millions in credit, tens of thousands in bad debt), the extracted value is significant.
