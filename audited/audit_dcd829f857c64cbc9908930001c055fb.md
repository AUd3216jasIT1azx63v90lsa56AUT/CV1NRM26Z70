### Title
Zero settlement fee causes unconditional zero-value `safeTransferFrom` that reverts on non-standard loan tokens - (`src/Midnight.sol`)

### Summary
When all settlement fee breakpoints are zero, `buyerAssets - sellerAssets` evaluates to exactly `0`. `take()` unconditionally calls `SafeTransferLib.safeTransferFrom(loanToken, payer, address(this), 0)` at line 455 with no zero-value guard. Loan tokens that revert on zero-value `transferFrom` calls will cause every `take()` in that market to revert, permanently blocking valid takers.

### Finding Description
**Code path:**

In `src/Midnight.sol`, `take()` computes prices at lines 360–364:

```solidity
uint256 _settlementFee = settlementFee(id, timeToMaturity);   // returns 0 when all cbps == 0
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 buyerPrice  = sellerPrice + _settlementFee;            // == sellerPrice when fee == 0
uint256 buyerAssets = ...mulDiv...(buyerPrice, WAD);
uint256 sellerAssets = ...mulDiv...(sellerPrice, WAD);         // == buyerAssets when fee == 0
``` [1](#0-0) 

Because `buyerPrice == sellerPrice` when `_settlementFee == 0`, `buyerAssets - sellerAssets == 0` exactly (same rounding direction per offer side).

Line 418 silently adds 0 to `claimableSettlementFee` — no revert there: [2](#0-1) 

Then, unconditionally at line 455:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
``` [3](#0-2) 

`SafeTransferLib.safeTransferFrom` has no zero-value guard — it always issues the low-level `transferFrom` call: [4](#0-3) 

When `buyerCallback != address(0)`, `payer = buyerCallback` (line 422), so the zero-value pull is from the callback contract: [5](#0-4) 

**Attacker inputs / exploit flow:**
1. Deploy a loan token that reverts on zero-value `transferFrom` (a known real-world pattern).
2. Create a market using that token with all `settlementFeeCbp` values at 0 (the default initial state — `touchMarket` copies `defaultSettlementFeeCbp` which is 0 unless explicitly set).
3. Set a `buyerCallback` on the offer (or use `takerCallback`).
4. Call `take()` — execution reaches line 455, `safeTransferFrom(..., 0)` is issued, the token reverts, the entire `take()` reverts.

**Why existing checks fail:**
There is no `if (buyerAssets > sellerAssets)` guard before line 455. The `claimableSettlementFee` accounting at line 418 already handles the zero case silently, but the transfer is not similarly guarded.

### Impact Explanation
Any market whose loan token reverts on zero-value transfers is permanently DoS'd for `take()` whenever the settlement fee is zero. All state changes made before line 455 (position updates, consumed tracking, credit/debt accounting) are reverted, so no funds are lost — but valid takes are completely blocked. This is a fund-freeze / liveness failure for affected markets.

### Likelihood Explanation
- Zero settlement fee is the **default** state: `touchMarket` copies `defaultSettlementFeeCbp[loanToken]`, which is `0` for any token not explicitly configured by the fee setter.
- Tokens that revert on zero-value transfers exist in production (e.g., LEND, some rebasing/fee-on-transfer tokens). The protocol places no restriction on which ERC-20 can be a loan token.
- The condition is fully repeatable: every `take()` call in the affected market will revert until the settlement fee is raised above zero.
- No privileged action by the attacker is required; any taker triggers it.

### Recommendation
Guard the settlement-fee transfer with a zero check, mirroring the pattern already used in `MidnightBundles.sol` for referral fees:

```solidity
// src/Midnight.sol, line 455
if (buyerAssets > sellerAssets) {
    SafeTransferLib.safeTransferFrom(
        offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets
    );
}
```

The same guard should be applied to line 456 for `sellerAssets` if `sellerAssets` can also be zero (e.g., tick == 0).

        require(amt > 0, "zero transfer");

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
