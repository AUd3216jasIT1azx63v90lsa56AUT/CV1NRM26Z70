### Title
Debt Increase Allowed at Exact Maturity Timestamp Due to Off-by-One in Boundary Check — (`src/Midnight.sol`)

### Summary

The `take` function in `src/Midnight.sol` uses `block.timestamp <= offer.market.maturity` to guard against post-maturity debt increases. This `<=` operator allows debt to be increased at the exact maturity timestamp (`block.timestamp == market.maturity`), violating the protocol's own stated invariant that "maturity boundary must not allow debt increase by rounding, **timestamp equality**, or callback sequencing."

### Finding Description

**Root cause — line 391:**

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
``` [1](#0-0) 

The condition `block.timestamp <= offer.market.maturity` evaluates to `true` when `block.timestamp == market.maturity`, so a taker can execute a sell offer that increases `sellerDebtIncrease` at the exact maturity second. This debt is immediately overdue.

**Conflicting check — post-maturity liquidation (line 622):**

```solidity
postMaturityMode ? block.timestamp > market.maturity : originalDebt > maxDebt
``` [2](#0-1) 

Post-maturity liquidation requires strict `>`, so at `block.timestamp == market.maturity` it is **not yet available**. The test suite explicitly confirms this:

```solidity
// At exact maturity: still not available (only valid strictly after maturity).
vm.warp(market.maturity);
vm.expectRevert(IMidnight.NotLiquidatable.selector);
midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");
``` [3](#0-2) 

**Violated invariant (live_context.json):**

```json
"maturity boundary must not allow debt increase by rounding, timestamp equality, or callback sequencing"
``` [4](#0-3) 

**Exploit path:**

1. A borrower (or an authorized operator) monitors the mempool for the block whose `block.timestamp == market.maturity`.
2. They submit a `take` against a sell offer in that block, increasing `sellerDebtIncrease > 0`.
3. The check `block.timestamp <= market.maturity` passes (equality), so the debt increase is accepted.
4. The newly created debt is immediately overdue, but post-maturity liquidation is unavailable at that exact timestamp.
5. In the next block (`block.timestamp == market.maturity + 1`), post-maturity liquidation becomes available, but the LIF is `WAD + (maxLif - WAD) * 1 / TIME_TO_MAX_LIF` — nearly 1.0 — providing minimal liquidation incentive for the first ~15 minutes. [5](#0-4) 

### Impact Explanation

A borrower can create overdue debt at exactly the maturity timestamp. This debt enters the post-maturity ramp window with a near-zero liquidation incentive (`lif ≈ WAD`), meaning liquidators have little economic motivation to act immediately. The protocol's core invariant — that debt must not increase at or after maturity — is violated. In markets with large positions, this creates a window where overdue debt is not profitably liquidatable, potentially leaving the protocol exposed to bad debt if collateral prices move adversely during the ramp window.

### Likelihood Explanation

Requires a transaction to land at exactly `block.timestamp == market.maturity`. On Ethereum mainnet, block timestamps are set by validators and can be manipulated within ~12 seconds. A sophisticated borrower (or a validator who is also a borrower) can reliably target this boundary. The attack is permissionless and requires no privileged access.

### Recommendation

Change `<=` to `<` on line 391:

```solidity
// Before (vulnerable):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (fixed):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the debt-increase guard with the protocol's stated invariant and closes the one-second window at exact maturity.

### Proof of Concept

```solidity
// At block.timestamp == market.maturity:
// 1. Debt increase is allowed (block.timestamp <= maturity passes)
vm.warp(market.maturity);
// sellerDebtIncrease > 0 is accepted — no revert
take(units, lender, borrowerOffer); // succeeds, debt increased

// 2. Post-maturity liquidation is NOT yet available
vm.expectRevert(IMidnight.NotLiquidatable.selector);
midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");

// 3. One second later, liquidation is available but LIF ≈ WAD (minimal incentive)
vm.warp(market.maturity + 1);
// lif = WAD + (maxLif - WAD) * 1 / 900 ≈ WAD
midnight.liquidate(market, 0, 0, 0, borrower, true, address(this), address(0), "");
```
