Audit Report

## Title
Unchecked multiplication in `UtilsLib.mulDivDown`/`mulDivUp` causes permanent position freeze when `collateral × price` overflows uint256 - (File: src/libraries/UtilsLib.sol)

## Summary
`UtilsLib.mulDivDown` and `mulDivUp` perform bare `x * y` multiplication under Solidity 0.8.34's checked arithmetic. When a borrower's `collateral[i] × oracle_price` exceeds `type(uint256).max`, both `isHealthy` and `liquidate` revert with an arithmetic panic. Any position crossing this threshold is permanently frozen: collateral cannot be withdrawn and the position cannot be liquidated, violating the protocol's core health and liquidation invariants.

## Finding Description

**Root cause — `UtilsLib.mulDivDown` and `mulDivUp`:**

```solidity
// src/libraries/UtilsLib.sol L29-31
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y) / d;   // checked mul — panics on overflow
}

// src/libraries/UtilsLib.sol L34-36
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;   // same issue
}
```

Both functions use plain Solidity multiplication. Under `pragma solidity 0.8.34` (confirmed at `src/Midnight.sol` L3), any overflow causes an unconditional `Panic(0x11)` revert.

**Call site 1 — `isHealthy` (src/Midnight.sol L954-955):**

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
               .mulDivDown(collateralParam.lltv, WAD);
```

`_position.collateral[i]` is `uint128` (capped by `toUint128`). `price` is an unconstrained `uint256` from `IOracle(collateralParam.oracle).price()`. No bounds check on `price` exists anywhere in the protocol.

**Call site 2 — `liquidate` (src/Midnight.sol L613, L615):**

```solidity
maxDebt += _collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(_collateralParam.lltv, WAD);
badDebt = badDebt.zeroFloorSub(
    _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
);
```

Both `mulDivDown` and `mulDivUp` are hit before the `NotLiquidatable` check at L620-624, so the revert occurs before any liquidatability determination.

**Overflow threshold:**

With `collateral = type(uint128).max = 2^128 − 1`, overflow occurs when:
```
price > (2^256 − 1) / (2^128 − 1) = 2^128 + 1
```
i.e., `price ≥ 2^128 + 2 ≈ 3.4 × 10^38`.

**Realistic trigger — WBTC (8 dec) / USDC (6 dec):**

With `ORACLE_PRICE_SCALE = 1e36` (standard Morpho convention) and BTC at $100,000:
```
price = 100,000 × 1e36 × 10^(8−6) = 1e43
```
`1e43 >> 3.4e38`. Even at a modest collateral amount well below `type(uint128).max`, the product overflows.

**Exploit flow:**

1. Market creator deploys a market with a legitimate oracle returning `price ≥ 2^128 + 2` (e.g., any WBTC/USDC-style pair).
2. Borrower calls `supplyCollateral` — this stores collateral as `uint128` via `toUint128` and does **not** call the oracle. No revert.
3. Borrower takes a loan (`debt > 0`), activating the oracle path in `isHealthy`.
4. Any subsequent call to `isHealthy`, `withdrawCollateral` (L568), or `liquidate` (L613/L615) hits `mulDivDown(collateral, price, ORACLE_PRICE_SCALE)` and panics.

**Existing protections — insufficient:**

- `toUint128` caps collateral at `type(uint128).max` but does not prevent `collateral × price` from overflowing uint256.
- Market creation validates `lltv`, `maxLif`, collateral sorting, and count — but imposes no constraint on oracle return values.
- The comment at `src/Midnight.sol` L34 explicitly acknowledges "Liquidation reverts if any of the activated collaterals' oracle reverts" but does not address arithmetic overflow from valid oracle values.

## Impact Explanation

Any borrower whose `collateral[i] × oracle_price > type(uint256).max` has their position permanently bricked:

- `withdrawCollateral` reverts (calls `isHealthy` at L568).
- `liquidate` reverts at L613/L615 before the liquidatability check at L620.
- `isHealthy` itself reverts, making the position appear neither healthy nor unhealthy to any caller.

This constitutes a **permanent freeze of user funds** (collateral locked forever) and **permanent denial of liquidation** (bad debt cannot be realized), directly violating the protocol's core health and liquidation invariants. Severity: **Critical**.

## Likelihood Explanation

**Required preconditions:**

1. Oracle returns `price ≥ 2^128 + 2 ≈ 3.4e38` — satisfied by any token pair where the collateral unit is worth ≥ ~340 loan token units at `ORACLE_PRICE_SCALE = 1e36` (e.g., WBTC/USDC, ETH/USDC with standard decimal configurations).
2. Borrower supplies collateral such that `collateral × price > type(uint256).max` — with `price = 1e43`, the threshold is `collateral > type(uint256).max / 1e43 ≈ 1.16e34`, which is far below `type(uint128).max ≈ 3.4e38`.

Both conditions are reachable by unprivileged actors using entirely legitimate inputs. The state is permanent once the position carries debt. This is not "incorrect data supplied by third-party oracles" (excluded by SECURITY.md) — the oracle is returning a correct, expected price; the bug is in the protocol's own arithmetic.

## Recommendation

Replace the bare multiplications in `mulDivDown` and `mulDivUp` with overflow-safe 512-bit intermediate arithmetic, as used in Uniswap v3's `FullMath` library or OpenZeppelin's `Math.mulDiv`:

```solidity
// src/libraries/UtilsLib.sol
function mulDivDown(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return Math.mulDiv(x, y, d); // OZ Math — uses 512-bit intermediates
}

function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return Math.mulDiv(x, y, d, Math.Rounding.Ceil);
}
```

Alternatively, implement Solady's `FullMath.mulDivUp`/`mulDivDown` inline to avoid the dependency. This fix must be applied to **both** functions since `mulDivUp` at L34-36 has the identical `x * y` overflow exposure.

## Proof of Concept

**Minimal Foundry test:**

```solidity
function test_overflow_freeze() public {
    // 1. Deploy market with WBTC/USDC oracle returning price = 1e43
    MockOracle oracle = new MockOracle(1e43); // > 2^128+2 ≈ 3.4e38
    Market memory market = _buildMarket(oracle);
    bytes32 id = midnight.touchMarket(market);

    // 2. Borrower supplies collateral (no oracle call here — succeeds)
    uint256 collateral = 1e18; // 1e18 * 1e43 = 1e61 >> type(uint256).max
    wbtc.mint(borrower, collateral);
    vm.prank(borrower);
    midnight.supplyCollateral(market, 0, collateral, borrower);

    // 3. Borrower takes a loan
    vm.prank(borrower);
    midnight.borrow(market, 1e6, borrower, borrower); // any nonzero debt

    // 4. isHealthy reverts with arithmetic panic
    vm.expectRevert(stdError.arithmeticError);
    midnight.isHealthy(market, id, borrower);

    // 5. withdrawCollateral reverts
    vm.prank(borrower);
    vm.expectRevert(stdError.arithmeticError);
    midnight.withdrawCollateral(market, 0, 1, borrower, borrower);

    // 6. liquidate reverts
    vm.expectRevert(stdError.arithmeticError);
    midnight.liquidate(market, 0, 0, 1, borrower, false, address(this), address(0), "");
}
```

**Fuzz variant:** Fuzz `price` over `[2^128+2, type(uint256).max]` and `collateral` over `[type(uint256).max / price + 1, type(uint128).max]` — every combination should trigger the arithmetic panic at `mulDivDown`.
