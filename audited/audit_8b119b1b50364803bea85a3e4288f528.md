### Title
maxAssets cap bypass via dust units on buy offers with price < WAD — (`src/Midnight.sol`)

### Summary
In `Midnight.take()`, when a buy offer uses `maxAssets`-based consumption tracking and `buyerPrice < WAD`, a taker can supply `units` in the range `[1, WAD/buyerPrice - 1]` such that `mulDivDown(units, buyerPrice, WAD) = 0`. The consumed counter increments by zero, so the `maxAssets` cap is never advanced, yet the buyer's credit and seller's debt each increase by `units` per call. This allows unbounded repetition with zero asset transfer.

### Finding Description
**Root cause:** `src/Midnight.sol` line 368 increments `consumed` by `buyerAssets`, not by `units`. When `buyerAssets` rounds to zero via `mulDivDown`, the cap check at line 369 (`require(newConsumed <= offer.maxAssets)`) is trivially satisfied regardless of how many times the offer has been taken.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.maxUnits = 0`
- `offer.tick` set to any tick where `tickToPrice(tick) < WAD` (all ticks satisfy this per the Certora proof)
- `units = 1` (or any value `< WAD / buyerPrice`)

**Exploit flow:**
1. Maker posts a buy offer with `maxAssets = N` and any tick (all prices are `< WAD`)
2. Attacker calls `take(offer, ratifierData, 1, attacker, ...)` in a loop
3. Each iteration: `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`; `consumed` unchanged; maker credit `+1`; attacker debt `+1`; zero tokens transferred
4. After `N` iterations, `consumed` is still 0; the offer appears entirely unconsumed; the maker has `N` units of credit they did not pay for

**Why existing checks fail:** The only guard is `require(newConsumed <= offer.maxAssets)`. When the increment is zero, this is always satisfied. There is no `require(units == 0 || buyerAssets > 0)` guard.

### Impact Explanation
The `maxAssets` cap — intended to bound the maker's total lending exposure in asset terms — is rendered ineffective for buy offers at any tick below MAX_TICK. An attacker can inflate the maker's credit and their own debt without limit and without transferring any assets, violating the invariant that offers cannot be overfilled.

### Likelihood Explanation
Preconditions are minimal: any buy offer with `maxAssets > 0` and any tick (since all `tickToPrice` values are `< WAD`). The attacker needs enough collateral to pass the health check at line 476 for the accumulated debt. The attack is repeatable in a single transaction via `multicall`. Gas cost per iteration is the only practical bound.

### Recommendation
Add a guard that rejects non-zero unit takes that produce zero assets in `maxAssets` mode:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, DustTakeInAssetsMode());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This preserves the existing no-op (`units == 0`) path used for callback-only takes while closing the dust bypass.

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass()` in `test/TakeTest.sol` (lines 858–889) is a complete unit-test PoC. A fuzz extension:

/// If consumed is already at or above maxAssets before a take in assets mode, it remains unchanged.
rule takeConsumedAtMaxUnchangedAssets(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxUnits == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumedBefore >= offer.maxAssets => consumed(offer.maker, offer.group) == consumedBefore;
