### Title
Taker-Controlled `receiverIfTakerIsSeller` Accepts `address(this)`, Permanently Locking `sellerAssets` in Midnight With No Accounting Entry - (File: src/Midnight.sol)

### Summary
In `take()`, when `offer.buy = true` the taker is the seller and fully controls the `receiverIfTakerIsSeller` parameter, which is used verbatim as the destination for the `sellerAssets` transfer. No check prevents this address from being `address(this)` (the Midnight contract itself). When set to `address(this)`, `sellerAssets` tokens are transferred into the contract but are not credited to any accounting field (`withdrawable`, `claimableSettlementFee`, `credit`, or any position), permanently locking them and violating the invariant that every token inflow has a corresponding accounting entry.

### Finding Description
**Code path and root cause:**

`take()` resolves the receiver at line 423:

```solidity
address receiver = offer.buy ? receiverIfTakerIsSeller : offer.receiverIfMakerIsSeller;
``` [1](#0-0) 

When `offer.buy = true` (maker is buyer/lender, taker is seller/borrower), `receiver` is set directly to the taker-supplied `receiverIfTakerIsSeller` with no validation. The payer is resolved as:

```solidity
address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);
``` [2](#0-1) 

With no `buyerCallback` and `offer.buy = true`, `payer = buyer = offer.maker`. The two transfers then execute:

```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
``` [3](#0-2) 

If `receiver = address(this)`, the second transfer sends `sellerAssets` tokens to the Midnight contract itself. The only accounting update for token inflows is:

```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [4](#0-3) 

The `sellerAssets` portion that lands in the contract is not added to `claimableSettlementFee`, `withdrawable`, any position's `credit`, or any other tracked field. No check of the form `require(receiver != address(this))` exists anywhere in `src/`.

**Exploit flow:**
1. Attacker is an unprivileged taker (borrower).
2. A maker has posted a buy offer (`offer.buy = true`) with sufficient approval.
3. Attacker calls `midnight.take(offer, ratifierData, units, taker, address(midnight), address(0), "")`.
4. All pre-checks pass (authorization, ratifier, tick, expiry, etc.).
5. Seller (taker) position accounting updates: `sellerPos.debt += sellerDebtIncrease`, `sellerPos.credit -= sellerCreditDecrease`.
6. `claimableSettlementFee` increases by `buyerAssets - sellerAssets` only.
7. `safeTransferFrom(loanToken, offer.maker, address(midnight), sellerAssets)` executes successfully.
8. `sellerAssets` tokens are now in the contract with no corresponding accounting entry.

### Impact Explanation
`sellerAssets` tokens are permanently locked inside the Midnight contract. No function (`withdraw`, `claimSettlementFee`, `claimContinuousFee`, `flashLoan`) can recover them because none of these operate on unaccounted surplus balances. The contract's token balance exceeds the sum of all tracked liabilities (`withdrawable + claimableSettlementFee + sum of credits`) by exactly `sellerAssets`. The taker simultaneously accrues debt without receiving loan proceeds, and the locked tokens are irrecoverable.

### Likelihood Explanation
The precondition is simply a valid buy offer with a willing maker — a normal market condition. The taker needs no special privilege, no oracle manipulation, and no reentrancy. The call is a single atomic transaction. It is repeatable across any market and any buy offer. The only constraint is that `sellerAssets > 0`, which holds for any offer with `sellerPrice > 0` (i.e., `offerPrice > settlementFee`).

### Recommendation
Add a zero-address and self-address guard when resolving `receiver` in `take()`:

```solidity
address receiver = offer.buy ? receiverIfTakerIsSeller : offer.receiverIfMakerIsSeller;
require(receiver != address(0) && receiver != address(this), InvalidReceiver());
```

The same guard should be applied to `offer.receiverIfMakerIsSeller` for the symmetric `offer.buy = false` case, and analogously to `payer` if a callback address is used.

### Proof of Concept
```solidity
function testSellerAssetsLockedInMidnight() public {
    // Setup: maker posts a buy offer (offer.buy = true), maker has approved midnight
    uint256 units = 1e18;
    Offer memory offer = /* standard buy offer with maker = lender */;

    uint256 contractBalanceBefore = loanToken.balanceOf(address(midnight));
    uint256 claimableBefore = midnight.claimableSettlementFee(address(loanToken));

    // Taker (borrower) sets receiver to address(midnight)
    vm.prank(taker);
    (uint256 buyerAssets, uint256 sellerAssets) =
        midnight.take(offer, hex"", units, taker, address(midnight), address(0), hex"");

    uint256 contractBalanceAfter = loanToken.balanceOf(address(midnight));
    uint256 claimableAfter = midnight.claimableSettlementFee(address(loanToken));

    // Contract balance increased by buyerAssets (settlement fee + sellerAssets)
    assertEq(contractBalanceAfter, contractBalanceBefore + buyerAssets);

    // But only (buyerAssets - sellerAssets) is accounted for
    assertEq(claimableAfter, claimableBefore + buyerAssets - sellerAssets);

    // sellerAssets are locked: no withdrawable, no credit, no claimable covers them
    // Invariant violated: contractBalance > claimable + withdrawable + sum(credits)
    assertGt(sellerAssets, 0);
    assertEq(contractBalanceAfter - contractBalanceBefore - (claimableAfter - claimableBefore), sellerAssets);

    // Taker has debt but no proceeds
    bytes32 id = midnight.toId(offer.market);
    assertGt(midnight.position(id, taker).debt, 0);
}
```

Expected assertions: `token.balanceOf(address(midnight))` increases by `buyerAssets`, but `claimableSettlementFee` only increases by `buyerAssets - sellerAssets`, leaving `sellerAssets` unaccounted and irrecoverable.
