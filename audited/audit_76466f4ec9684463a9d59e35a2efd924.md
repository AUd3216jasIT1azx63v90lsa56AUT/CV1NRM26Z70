### Title
`Midnight::liquidate` Can Be DoSed via Front-Running with Dust Amount — (File: src/Midnight.sol)

---

### Summary

The `liquidate` function accepts a caller-specified `repaidUnits` (or `seizedAssets`) parameter but does not cap it against the borrower's current on-chain debt (or collateral). Because `liquidate` is permissionless, any party — including the borrower via `repay` — can front-run a pending liquidation with a dust-sized transaction that reduces the borrower's debt by 1 unit. The original liquidator's transaction then reverts with an arithmetic underflow at line 676, since `repaidUnits > _position.debt`. This is the exact same vulnerability class as M-13 in the reference report.

---

### Finding Description

**Root cause**: `liquidate` performs an unchecked subtraction `_position.debt -= UtilsLib.toUint128(repaidUnits)` at line 676 with no prior guard that `repaidUnits <= _position.debt` (after bad-debt reduction at line 628). Solidity 0.8.x checked arithmetic causes an underflow revert when the invariant is violated.

**Two permissionless front-run vectors exist**:

**Vector 1 — Borrower calls `repay` with 1 unit**:
`repay` is restricted to `onBehalf == msg.sender || isAuthorized[...]`, so the borrower can call `repay(market, 1, borrower, ...)` to reduce their own debt by 1. A liquidator who read `debt = D` and submitted `repaidUnits = D` now faces `D > D - 1`, causing underflow revert at: [1](#0-0) 

**Vector 2 — Any party calls `liquidate` with `repaidUnits = 1`**:
`liquidate` has no authorization check beyond the optional `liquidatorGate`. Any address can call `liquidate(market, collateralIndex, 0, 1, borrower, ...)` to reduce the borrower's debt by 1 at minimal cost. The original liquidator's `repaidUnits = D` then reverts identically. [2](#0-1) 

**Normal-mode secondary revert path**: Even before reaching line 676, in normal (pre-maturity) mode the `maxRepaid` check at lines 659–667 uses `_position.debt - maxDebt`. After a front-run reduces `_position.debt` by 1, `maxRepaid` shrinks, and the original liquidator's `repaidUnits` may exceed the new `maxRepaid`, reverting with `RecoveryCloseFactorConditionsViolated`: [3](#0-2) 

**Parallel path for `seizedAssets` input**: The same DoS applies when the liquidator specifies `seizedAssets` instead of `repaidUnits`. A front-runner seizing 1 unit of collateral reduces `_position.collateral[collateralIndex]` by 1, causing the original liquidator's `seizedAssets` to underflow at line 670: [4](#0-3) 

The existing test suite explicitly confirms the underflow revert path (`testCannotRepayMoreThanDebt`, line 310: `vm.expectRevert(stdError.arithmeticError)`): [5](#0-4) 

---

### Impact Explanation

Liquidations can be indefinitely DoSed against any borrower. Unhealthy positions remain open, accumulating bad debt that is socialized across all lenders in the market. In post-maturity mode (where the RCF is deactivated and the LIF grows over time), delayed liquidation directly increases the bad-debt loss absorbed by lenders. The borrower's cost to execute this attack is 1 loan-token unit per block (via `repay`), or a third party can do it for the same cost via a dust `liquidate` call that may even be profitable (they receive `seizedAssets` proportional to `lif > 1`).

---

### Likelihood Explanation

- **Borrower motivation is strong**: avoiding liquidation is a direct financial incentive.
- **Cost is negligible**: 1 unit of loan token (e.g., 1 wei of USDC) per front-run attempt.
- **No privileged access required**: `repay` is callable by the borrower; `liquidate` is permissionless.
- **Feasibility**: requires same-block ordering, which is standard MEV on any chain with a public mempool. On L2s with a centralized sequencer the borrower can still submit a transaction first if they observe the liquidator's pending tx.
- **Repeatability**: the attack can be repeated every block at negligible cost.

---

### Recommendation

Cap `repaidUnits` to the remaining debt (after bad-debt reduction) before the subtraction, mirroring the M-13 fix of using `min(deficit, amount)`:

```solidity
// After bad-debt block (line 641), before line 676:
repaidUnits = UtilsLib.min(repaidUnits, _position.debt);
```

Similarly, cap `seizedAssets` to the available collateral before line 670:

```solidity
seizedAssets = UtilsLib.min(seizedAssets, _position.collateral[collateralIndex]);
// then recompute repaidUnits from the capped seizedAssets
```

This ensures a liquidator who specifies "full liquidation" always succeeds even if the debt was partially reduced by a concurrent transaction.

### Proof of Concept

```solidity
function testLiquidateFrontRunDoS() public {
    // Setup: borrower has debt = D, position is liquidatable post-maturity
    uint256 D = 1000e18;
    // ... (collateralize, borrow D units, warp past maturity) ...

    uint256 debtBefore = midnight.debtOf(id, borrower);

    // Step 1: Attacker (or borrower) front-runs with repaidUnits = 1
    // Cost: 1 loan token unit; attacker receives seizedAssets proportional to lif
    midnight.liquidate(market, 0, 0, 1, borrower, true, attacker, address(0), "");

    // Step 2: Original liquidator's tx — repaidUnits = debtBefore — now reverts
    // because _position.debt = debtBefore - 1 < debtBefore = repaidUnits
    vm.expectRevert(stdError.arithmeticError);
    midnight.liquidate(market, 0, 0, debtBefore, borrower, true, liquidator, address(0), "");

    // Borrower's debt reduced by only 1 unit; position remains liquidatable
    assertEq(midnight.debtOf(id, borrower), debtBefore - 1);
}
```

        units = bound(units, 10, MAX_UNITS - 1);
        collateralize(market, borrower, units);
        setupMarket(market, units);
        vm.warp(market.maturity + TIME_TO_MAX_LIF); // Warp to post-maturity for full LIF.

        uint256 _maxLif = market.collateralParams[0].maxLif;
        uint256 collateral = midnight.collateral(id, borrower, 0);

        // Price must be high enough that seized assets for (units + 1) don't exceed available collateral.
        uint256 minPrice = (units + 1).mulDivUp(_maxLif, WAD).mulDivUp(ORACLE_PRICE_SCALE, collateral);

        Oracle(market.collateralParams[0].oracle).setPrice(liquidationOraclePrice);
