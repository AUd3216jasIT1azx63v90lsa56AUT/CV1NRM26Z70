Audit Report

## Title
Zero-`buyerAssets` rounding bypasses `maxAssets` cap, allowing unlimited free position mutations - (File: src/Midnight.sol)

## Summary
In `take()`, when `offer.maxAssets > 0`, the consumed counter is incremented by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`, which truncates to zero whenever `units * buyerPrice < WAD`. Because position state (`credit`, `debt`, `totalUnits`) is updated using `units` directly rather than `buyerAssets`, a taker can call `take` repeatedly with `units = 1` at any tick below `MAX_TICK`, mutating maker credit and taker debt on every call while `consumed` never advances and zero tokens are transferred. The `maxAssets` cap is completely bypassed.

## Finding Description
In `src/Midnight.sol`, the `take()` function computes asset amounts and consumed tracking as follows:

**Line 363**: `buyerAssets = units.mulDivDown(buyerPrice, WAD)` — for a buy offer with `units = 1` and any `tick < MAX_TICK`, `buyerPrice < WAD = 1e18`, so `1 * buyerPrice / 1e18 = 0` by integer truncation. `PRICE_ROUNDING_STEP = 1e12` (confirmed in `src/libraries/TickLib.sol` line 8), and `tickToPrice(MAX_TICK) = 1e18` while all lower ticks yield strictly smaller values (confirmed by `testTickToPriceMinMax` in `test/TickLibTest.sol` line 15–20).

**Lines 367–369**:
```solidity
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```
When `buyerAssets == 0`, `consumed` is incremented by zero. Even if `consumed` was already equal to `maxAssets`, `newConsumed == maxAssets <= maxAssets` trivially passes.

**Lines 382–414**: All position mutations use `units`, not `buyerAssets`:
- `buyerCreditIncrease = zeroFloorSub(units, buyerPos.debt)` (line 382)
- `buyerPos.credit += buyerCreditIncrease` (line 410)
- `sellerPos.debt += sellerDebtIncrease` (line 414)
- `_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease` (line 416–417)

**Line 455**: `safeTransferFrom(..., buyerAssets - sellerAssets)` transfers `0 - 0 = 0` tokens.

The root cause is a denomination mismatch: `consumed` is asset-denominated but position mutations are unit-denominated, with no guard enforcing `buyerAssets > 0` before proceeding.

**Confirmed by `testBugBuyMaxAssetsBypass`** (`test/TakeTest.sol`, lines 857–889): the test sets `maxAssets = 1`, pre-fills `consumed` to `maxAssets` via `setConsumed` (called as the maker/lender, a self-service function), then calls `take(1, borrower, lenderOffer)` with `tick = MAX_TICK - 16`. It asserts `buyerAssets == 0`, `consumed` unchanged at `maxAssets`, token balances unchanged, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increased.

**Existing checks that fail**:
- `require(newConsumed <= offer.maxAssets)` — passes because `+= 0`
- `require(atMostOneNonZero(maxAssets, maxUnits))` — validates offer structure only, not fill size
- No `require(units > 0)` or `require(buyerAssets > 0)` guard exists

## Impact Explanation
A taker can fill a buy offer with `maxAssets > 0` an unlimited number of times at zero token cost. Each fill increases the maker's credit and the taker's debt by `units` without any token transfer and without advancing `consumed`. This violates the core accounting invariant that every credit increase must correspond to a valid asset transfer and consumed accounting. The taker accrues unbounded debt obligations and the maker gains unbounded credit — both without any economic exchange — which corrupts market state and can be exploited to manipulate settlement fee accounting, downstream liquidation logic, and position health calculations. This constitutes unauthorized state change and critical integrity failure in the protocol's accounting model.

## Likelihood Explanation
Preconditions are minimal: (1) a buy offer with `maxAssets > 0` exists — this is a standard offer type; (2) `tickToPrice(offer.tick) < WAD` — true for every valid tick except `MAX_TICK` (i.e., 5819 out of 5820 valid ticks); (3) taker uses `units = 1`. No privileged access, oracle manipulation, or front-running is required. The attack is executable in a single transaction via `multicall` and is trivially repeatable with negligible gas cost per iteration.

## Recommendation
Add a guard at the start of the asset-denominated branch to reject zero-asset fills:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0, ZeroAssets()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This ensures that any fill tracked against an asset-denominated cap must move at least one unit of the loan token, preserving the invariant that credit/debt mutations correspond to real economic exchange.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) is a direct reproduction:

1. Deploy with a buy offer at `tick = MAX_TICK - 16` (price < WAD) and `maxAssets = 1`.
2. Pre-fill `consumed` to `maxAssets` (simulating a fully consumed offer).
3. Call `take(1, borrower, lenderOffer)`.
4. Observe: `buyerAssets == 0`, `consumed` unchanged, token balances unchanged, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all increased.
