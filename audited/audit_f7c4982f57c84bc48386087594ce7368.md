### Title
Attacker-controlled `takerCallback` used as unchecked `payer` drains `loanToken` from victim contracts — (`src/Midnight.sol`)

### Summary
In `take()`, for a sell offer (`offer.buy = false`), the `takerCallback` parameter supplied by the caller is unconditionally promoted to `payer` for all `loanToken` `safeTransferFrom` calls. There is no check that `takerCallback` has authorized the caller to act on its behalf. Any contract that implements `IBuyCallback.onBuy()` returning `CALLBACK_SUCCESS` and holds a `loanToken` approval to Midnight can be drained. The scoped invariant is confirmed: only `loanToken` approval enables the drain; `collateralToken`-only approval causes a revert.

### Finding Description
**Code path — sell offer (`offer.buy = false`):** [1](#0-0) 

```
buyerCallback = takerCallback          // attacker-supplied, line 420
payer         = buyerCallback          // victimContract, line 422
``` [2](#0-1) 

The only authorization gate is `taker == msg.sender || isAuthorized[taker][msg.sender]`. There is **no** corresponding check that `takerCallback` (which becomes `payer`) has authorized `msg.sender` or `taker` to spend on its behalf.

**Callback gate (sole protection):** [3](#0-2) 

`victimContract.onBuy()` must return `CALLBACK_SUCCESS`. This is the only barrier. Any contract that legitimately implements `IBuyCallback` and returns `CALLBACK_SUCCESS` passes this check.

**Token pull from payer:** [4](#0-3) 

Both transfers use `offer.market.loanToken` from `payer` (= `victimContract`). The ERC-20 `transferFrom` checks the `loanToken` allowance of `victimContract` to Midnight — not the `collateralToken` allowance. Therefore:

- `victimContract` has `collateralToken.approve(midnight, max)`, `loanToken.approve(midnight, 0)` → `safeTransferFrom` for `loanToken` reverts → **attack fails**.
- `victimContract` has `loanToken.approve(midnight, max)` → transfers succeed → **victimContract's `loanToken` is drained**.

**Root cause:** `takerCallback` is an unchecked attacker input that is silently elevated to `payer` with no `isAuthorized` or consent verification.

### Impact Explanation
Any contract that (a) implements `IBuyCallback.onBuy()` returning `CALLBACK_SUCCESS` and (b) holds a `loanToken` allowance to Midnight can have its entire `loanToken` balance drained by an unprivileged attacker in a single `take()` call. The scoped invariant holds exactly: `collateralToken`-only approval is insufficient and causes a revert; `loanToken` approval is the precise requirement.

### Likelihood Explanation
Realistic victim contracts include: taker-side callback helpers that pre-approve Midnight for `loanToken` to fund their own legitimate takes, or any protocol integration that grants Midnight a standing `loanToken` allowance. The attacker needs only a valid (non-expired, non-consumed) sell offer signed by any maker — no privileged access required. The attack is repeatable as long as the victim contract retains its `loanToken` approval.

### Recommendation
Add an authorization check before accepting `takerCallback` as `payer`. Specifically, require that `takerCallback == msg.sender` or `isAuthorized[takerCallback][msg.sender]` (mirroring the existing `taker` authorization pattern). Alternatively, require `takerCallback == taker` so the payer is always the authenticated taker, not an arbitrary third-party address.

```solidity
// proposed guard before line 422
require(
    takerCallback == address(0)
        || takerCallback == msg.sender
        || isAuthorized[takerCallback][msg.sender],
    TakerCallbackUnauthorized()
);
```

### Proof of Concept
**Foundry unit test plan:**

```solidity
// Setup
MockBuyCallback victimContract = new MockBuyCallback(); // returns CALLBACK_SUCCESS from onBuy()
deal(loanToken, address(victimContract), 1000e18);

// Case 1: only collateralToken approval → revert
collateralToken.approve(address(midnight), type(uint256).max); // from victimContract
// loanToken allowance remains 0
vm.expectRevert();
midnight.take(sellOffer, ratifierData, units, attacker, receiver, address(victimContract), "");

// Case 2: loanToken approval → drain succeeds
vm.prank(address(victimContract));
loanToken.approve(address(midnight), type(uint256).max);

uint256 balanceBefore = loanToken.balanceOf(address(victimContract));
midnight.take(sellOffer, ratifierData, units, attacker, receiver, address(victimContract), "");
uint256 balanceAfter = loanToken.balanceOf(address(victimContract));

assertLt(balanceAfter, balanceBefore); // victimContract drained
assertEq(balanceBefore - balanceAfter, buyerAssets); // exact drain amount
```

**Expected assertions:**
- Case 1 reverts (insufficient `loanToken` allowance; `collateralToken` approval is irrelevant).
- Case 2 succeeds; `victimContract`'s `loanToken` balance decreases by exactly `buyerAssets`.
