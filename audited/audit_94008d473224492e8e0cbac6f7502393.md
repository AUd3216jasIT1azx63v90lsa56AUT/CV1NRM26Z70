### Title
Fee-on-transfer collateral token causes receiver to receive less than accounting deducts in `withdrawCollateral` - (File: src/Midnight.sol)

### Summary
`withdrawCollateral` deducts exactly `assets` from `position[id][onBehalf].collateral[collateralIndex]` and then calls `SafeTransferLib.safeTransfer(collateralToken, receiver, assets)`. `safeTransfer` only validates the boolean return value of `transfer`; it never checks the actual token balance delta at `receiver`. When the collateral token is fee-on-transfer with fee rate `f`, the receiver receives `assets*(1-f)` while the accounting deduction is the full `assets`, producing a permanent `f*assets` shortfall per withdrawal.

### Finding Description
**Exact code path:**

`withdrawCollateral` at `src/Midnight.sol:561-562` subtracts `assets` from storage unconditionally: [1](#0-0) 

The health check at line 568 passes because it reads the already-reduced accounting value: [2](#0-1) 

Then `safeTransfer` is called with the nominal `assets` amount: [3](#0-2) 

`SafeTransferLib.safeTransfer` calls `token.transfer(to, value)` and only checks the boolean return; it performs no balance-before/after comparison: [4](#0-3) 

**Root cause:** Neither `withdrawCollateral` nor `safeTransfer` measures the actual tokens received by `receiver`. For a fee-on-transfer token with fee rate `f`, `transfer(receiver, assets)` delivers `assets*(1-f)` to `receiver` and routes `assets*f` to the fee collector. The protocol's balance decreases by the full `assets`, but `receiver` only receives `assets*(1-f)`.

**Supply-side compounding:** `supplyCollateral` has the symmetric flaw — it credits `assets` to accounting but only receives `assets*(1-f)` from the user: [5](#0-4) [6](#0-5) 

This means the protocol's actual token balance is already `f*assets` short after supply. On withdrawal, the protocol must cover the outgoing `assets` transfer from other users' deposits, draining the shared pool by `f*assets` per round-trip.

**Attacker-controlled inputs:** `assets` (withdrawal amount), `receiver` (destination). No privileged role required; any authorized borrower can call `withdrawCollateral`.

**Why existing checks do not stop it:** The only guards are the authorization check (`onBehalf == msg.sender || isAuthorized`) and `isHealthy`. Neither inspects token balances or enforces that `receiver` receives exactly `assets`. `safeTransfer` only enforces a non-reverting, `true`-returning call — a fee-on-transfer token satisfies both conditions while silently delivering less.

### Impact Explanation
For every `withdrawCollateral` call with a fee-on-transfer collateral token of fee rate `f`:
- Accounting deducts `assets` from `position[id][onBehalf].collateral[collateralIndex]`
- `receiver` receives only `assets*(1-f)` tokens
- The gap `f*assets` is permanently lost to the fee collector
- The protocol's collateral token balance falls below what the aggregate accounting claims, meaning subsequent withdrawers or liquidators drawing on the same pool face insolvency for that shortfall

### Likelihood Explanation
**Preconditions:**
1. A market is created with a fee-on-transfer token as a `collateralParams[i].token` — no market-creation validation prevents this.
2. A borrower supplies and then withdraws collateral through the normal flow.

**Feasibility:** Fee-on-transfer tokens are a real, deployed token class. Market creation is permissionless; any market creator (an unprivileged role per the rules) can instantiate such a market. The exploit requires no special timing, oracle manipulation, or privileged access. It is repeatable on every `withdrawCollateral` call.

### Recommendation
Record the receiver's token balance before and after the transfer and require the delta equals `assets`:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(receiver);
SafeTransferLib.safeTransfer(collateralToken, receiver, assets);
require(IERC20(collateralToken).balanceOf(receiver) - balanceBefore == assets, FeeOnTransferToken());
```

Apply the same guard symmetrically in `supplyCollateral` (measure `address(this)` balance delta after `safeTransferFrom`) to ensure the accounting credit equals the tokens actually received.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";

// Minimal fee-on-transfer ERC20: charges 1% on every transfer/transferFrom
contract FeeToken is ERC20 {
    uint256 constant FEE_BPS = 100; // 1%
    address public feeCollector;
    constructor(address _fc) ERC20("FeeToken","FT") { feeCollector = _fc; }
    function _transfer(address from, address to, uint256 amount) internal override {
        uint256 fee = amount * FEE_BPS / 10_000;
        super._transfer(from, feeCollector, fee);
        super._transfer(from, to, amount - fee);
    }
}

contract FeeOnTransferCollateralTest is Test {
    Midnight midnight;
    FeeToken feeToken;
    address feeCollector = address(0xFEE);
    address borrower   = address(0xB0B);
    address receiver   = address(0xBEEF);

    function setUp() public {
        // deploy Midnight and build a market whose collateral is feeToken
        feeToken = new FeeToken(feeCollector);
        // ... (standard market setup with feeToken as collateralParams[0].token)
    }

    function testFeeOnTransferWithdrawCollateralLoss() public {
        uint256 assets = 1000e18;
        // Fund borrower and supply collateral
        deal(address(feeToken), borrower, assets * 2);
        vm.startPrank(borrower);
        feeToken.approve(address(midnight), type(uint256).max);
        midnight.supplyCollateral(market, 0, assets, borrower);
        vm.stopPrank();

        // Record state before withdrawal
        uint256 collateralBefore = midnight.collateral(id, borrower, 0);
        uint256 receiverBalBefore = feeToken.balanceOf(receiver);

        // Borrower withdraws
        vm.prank(borrower);
        midnight.withdrawCollateral(market, 0, assets, borrower, receiver);

        uint256 collateralAfter = midnight.collateral(id, borrower, 0);
        uint256 receiverBalAfter = feeToken.balanceOf(receiver);

        // Accounting deducted full `assets`
        assertEq(collateralBefore - collateralAfter, assets, "accounting deducted assets");

        // Receiver received less than assets (fee taken)
        uint256 received = receiverBalAfter - receiverBalBefore;
        assertLt(received, assets, "receiver got less than assets");

        // Loss equals f * assets (1% of assets)
        uint256 expectedLoss = assets * 100 / 10_000;
        assertEq(assets - received, expectedLoss, "loss equals fee * assets");
    }
}
```

**Expected assertions:** `accounting deducted assets` passes; `receiver got less than assets` passes; `loss equals fee * assets` passes — confirming the invariant violation.
