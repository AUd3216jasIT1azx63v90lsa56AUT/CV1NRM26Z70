### Title
Market Insolvency and "Run on the Bank" via `lossFactor` Saturation Permanently Freezes `take()` — (File: src/Midnight.sol)

---

### Summary

Morpho Midnight's bad-debt socialization mechanism uses a monotonically increasing `lossFactor`. Once it saturates at `type(uint128).max`, `take()` is permanently and irrecoverably blocked for that market. Lenders whose credit has not yet been slashed to zero race to call `withdraw()` before their credit is zeroed out by `_updatePosition`, creating a "run on the bank" where the last lenders to interact lose all their funds. This is the direct analog of the Holdefi M06 finding.

---

### Finding Description

**Vulnerability class:** Accounting / State Transition — bad-debt socialization with a hard saturation ceiling.

**Root cause — `lossFactor` saturation in `liquidate()`:**

Every bad-debt realization in `liquidate()` updates `lossFactor` with a multiplicative formula:

```solidity
_marketState.lossFactor = UtilsLib.toUint128(
    type(uint128).max - (type(uint128).max - _lossFactor).mulDivDown(_totalUnits - badDebt, _totalUnits)
);
``` [1](#0-0) 

`mulDivDown` rounds down, so `lossFactor` rounds **up** on every bad-debt event. When `badDebt == _totalUnits` (or when accumulated rounding causes the numerator to underflow to zero), `lossFactor` is set to exactly `type(uint128).max`.

**Permanent freeze of `take()`:**

```solidity
require(_marketState.lossFactor < type(uint128).max, MarketLossFactorMaxedOut());
``` [2](#0-1) 

Once saturated, `take()` reverts unconditionally and forever. No new trading, no position unwinding, no debt reduction through the offer book. The contract's own comments acknowledge this:

> "If a market loses almost all of its value to bad debt over its lifetime, then the accounting of the loss may become extremely imprecise (against the user), potentially leading to a total loss. Note that the take function reverts when the loss factor is maxed out." [3](#0-2) 

**Credit zeroed to zero on next interaction:**

In `updatePositionView()`, when `marketState[id].lossFactor == type(uint128).max`:

```solidity
uint256 postSlashCredit = _lastLossFactor < type(uint128).max
    ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
    : 0;
``` [4](#0-3) 

`type(uint128).max - marketState[id].lossFactor == 0`, so `postSlashCredit = 0` for every lender whose `lastLossFactor < type(uint128).max`.

**`withdraw()` calls `_updatePosition` first:**

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    ...
    _updatePosition(market, id, onBehalf);   // ← slashes credit to 0
    ...
    _position.credit -= UtilsLib.toUint128(units);  // ← underflows → revert
``` [5](#0-4) 

Any lender who calls `withdraw()` after `lossFactor` is maxed out has their credit zeroed by `_updatePosition`, then the subtraction underflows and reverts. They cannot recover any funds.

**The "run on the bank" race:**

Lenders who call `withdraw()` *before* `lossFactor` is maxed out have their credit slashed proportionally (not to zero) and can still withdraw the remainder. Lenders who call `withdraw()` *after* saturation lose everything. This creates a first-mover advantage: every lender is incentivized to withdraw the moment they observe bad debt accumulating, accelerating the bank run.

**Exploit flow:**

1. Market has lenders with credit and borrowers with debt.
2. Collateral prices drop sharply during network congestion (or oracle latency), preventing timely liquidations.
3. Bad debt accumulates across multiple liquidation calls; each call increases `lossFactor`.
4. Eventually `lossFactor == type(uint128).max`.
5. `take()` is permanently blocked — no new trading or unwinding.
6. Lenders who call `withdraw()` now have credit zeroed by `_updatePosition` and cannot withdraw.
7. Lenders who withdrew earlier recovered proportionally slashed (but non-zero) funds.
8. Last lenders out lose 100% of their remaining credit.

---

### Impact Explanation

- **Direct loss of user funds**: Lenders who interact with the market after `lossFactor` saturation lose all remaining credit. There is no recovery path.
- **Permanent market freeze**: `take()` is irrecoverably blocked; the market cannot be unwound through normal trading.
- **"Run on the bank"**: Creates a structural first-mover advantage that incentivizes panic withdrawals, amplifying the insolvency spiral.
- **Severity**: High — direct, permanent, total loss of lender funds with no protocol-level recovery mechanism.

---

### Likelihood Explanation

- **Realistic trigger**: Any significant collateral price drop during network congestion (high gas prices delaying liquidation transactions) can cause bad debt to accumulate faster than liquidators can act. This scenario occurred historically with Maker's pricing oracles in March 2020.
- **No privileged attacker required**: The trigger is a market condition (price volatility + liquidation delay), not an admin action or oracle compromise.
- **Compounding rounding**: The `mulDivDown` rounding in the `lossFactor` formula means `lossFactor` can reach `type(uint128).max` even when a small residual of non-bad-debt units remains, making saturation reachable slightly earlier than the "all units are bad debt" threshold.
- **Confirmed by existing tests**: The test suite already demonstrates the full saturation path:

```solidity
Oracle(market.collateralParams[0].oracle).setPrice(0);
midnight.liquidate(market, 0, 0, 0, borrower, false, address(this), address(0), "");
assertEq(midnight.lossFactor(id), type(uint128).max, "loss factor");
midnight.updatePosition(market, lender);
assertEq(midnight.creditOf(id, lender), 0, "credit after slashing");
``` [6](#0-5) 

---

### Recommendation

1. **Pro-rata withdrawal cap**: When `lossFactor` approaches saturation, enforce that each lender can only withdraw their proportional share of `withdrawable`, preventing early withdrawers from draining the pool at the expense of late withdrawers.
2. **Circuit breaker**: Add a threshold (e.g., `lossFactor > INSOLVENCY_THRESHOLD`) that halts new borrowing and triggers a wind-down mode with pro-rata distribution.
3. **Targeted tests**: Add integration tests that simulate partial bad-debt accumulation across multiple liquidation events and verify that lender recovery is proportional regardless of withdrawal order.
4. **User-facing disclosure**: Clearly communicate to lenders that markets can become insolvent and that `take()` can be permanently frozen, with total loss possible for late withdrawers.

### Proof of Concept

```solidity
// 1. Setup: lender supplies 1000 units of credit; borrower borrows 1000 units against collateral.
// 2. Oracle price drops to 0 (simulating rapid price crash during congestion).
// 3. Anyone calls liquidate with seizedAssets=0, repaidUnits=0 to realize bad debt:
midnight.liquidate(market, 0, 0, 0, borrower, false, receiver, address(0), "");
// → lossFactor == type(uint128).max
// → take() now permanently reverts with MarketLossFactorMaxedOut

// 4. Lender A (withdrew before saturation): recovered proportionally slashed credit.
// 5. Lender B (withdraws now):
midnight.withdraw(market, 1000, lenderB, lenderB);
// → _updatePosition sets lenderB.credit = 0
// → _position.credit -= 1000 underflows → REVERT
// → Lender B loses 100% of funds.
```
