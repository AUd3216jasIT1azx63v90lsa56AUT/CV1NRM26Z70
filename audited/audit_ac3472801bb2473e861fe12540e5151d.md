### Title
Fee-on-Transfer Collateral Token Causes Over-Recorded Collateral, Enabling Under-Collateralized Borrowing and Permanent Liquidation DoS — (File: src/Midnight.sol)

---

### Summary

`supplyCollateral()` credits a borrower's position with the full `assets` parameter without verifying the actual amount received by the contract. When a fee-on-transfer token is used as collateral, the contract records more collateral than it holds. This allows a borrower to borrow against phantom collateral, and when the position becomes liquidatable, the liquidation permanently reverts because the contract cannot transfer the recorded collateral amount, leaving lenders with unrecoverable bad debt.

---

### Finding Description

**Vulnerability class:** Accounting — balance-vs-recorded-amount discrepancy.

**Root cause:** `supplyCollateral()` updates the position's collateral balance using the caller-supplied `assets` value, then calls `safeTransferFrom`. No balance snapshot is taken before or after the transfer to verify the actual received amount.

```solidity
// src/Midnight.sol  lines 532-545
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);   // ← recorded
...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // ← actual < assets
``` [1](#0-0) 

`SafeTransferLib.safeTransferFrom` only checks the boolean return value; it does not verify that `balanceOf(address(this))` increased by exactly `assets`. [2](#0-1) 

The same pattern appears in `repay()`, where `withdrawable` is incremented by `units` before the transfer, and in `take()`, where credit/debt accounting is settled against `buyerAssets`/`sellerAssets` before the pull: [3](#0-2) [4](#0-3) 

**Why the documented requirement does not mitigate this:** The protocol's `TOKEN SAFETY REQUIREMENTS` comment states "Midnight must send/receive exactly the requested amount on transfers," but this is an unenforced assumption, not a runtime check. Markets are created permissionlessly via `touchMarket()`, so any token — including fee-on-transfer tokens — can be used as collateral or loan token without any gate. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Collateral path (highest severity):**

1. Borrower calls `supplyCollateral()` with a 10%-fee-on-transfer collateral token, `assets = 1000`.
2. Contract receives 900 tokens; position records 1000.
3. Borrower borrows against the full 1000-unit collateral value and withdraws loan tokens.
4. Borrower defaults. Liquidator calls `liquidate()` specifying `seizedAssets = 1000`.
5. `_position.collateral[collateralIndex] -= 1000` succeeds in storage, but `SafeTransferLib.safeTransfer(collateralToken, receiver, 1000)` reverts because the contract only holds 900 tokens.
6. Liquidation is permanently blocked. The position cannot be unwound; bad debt cannot be socialized. Lenders suffer a total loss on the affected units. [7](#0-6) [8](#0-7) 

**Repay path:** If the loan token is fee-on-transfer, `repay()` inflates `withdrawable` beyond the tokens actually held. Early withdrawers drain the contract; later lenders receive nothing. [9](#0-8) 

---

### Likelihood Explanation

- Market creation is fully permissionless; no whitelist or token validation exists.
- Fee-on-transfer tokens are deployed on mainnet (e.g., STA, PAXG, and USDT with its dormant fee switch). A malicious actor can also deploy a custom fee-on-transfer token and create a market around it to attract liquidity.
- No privileged access is required. Any externally owned account can execute the full exploit path.

---

### Recommendation

**Short term:** Record a pre-transfer balance snapshot and compute the actual received delta; use that delta — not the caller-supplied `assets` — to update the position:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Apply the same pattern to `repay()` and the `take()` pull leg.

**Long term:** Apply the Token Integration Checklist to every token accepted as a loan token or collateral token, and consider adding an on-chain token validator called during `touchMarket()` that rejects known fee-on-transfer or rebasing tokens.

### Proof of Concept

```
Setup:
  - Deploy FeeToken (ERC-20, 10% fee on every transferFrom)
  - Create market: loanToken = DAI, collateralParams[0].token = FeeToken

Step 1 — Inflate collateral:
  attacker.supplyCollateral(market, 0, 1000e18, attacker)
  → FeeToken.transferFrom(attacker, Midnight, 1000e18) delivers 900e18
  → position[id][attacker].collateral[0] == 1000e18   ← phantom 100e18

Step 2 — Borrow against phantom collateral:
  attacker takes a sell offer, receiving DAI proportional to 1000e18 collateral

Step 3 — Default and attempt liquidation:
  liquidator.liquidate(..., seizedAssets=1000e18, ...)
  → _position.collateral[0] -= 1000e18  (storage OK)
  → SafeTransferLib.safeTransfer(FeeToken, liquidator, 1000e18)
  → REVERTS: contract balance is 900e18 < 1000e18

Result:
  Position is permanently illiquidatable.
  Lenders cannot recover their loan tokens.
  Bad debt is never socialized; totalUnits accounting is corrupted.
```

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
