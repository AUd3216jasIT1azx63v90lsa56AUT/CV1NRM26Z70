### Title
Two-step `mulDivDown` in `maxDebt` computation allows liquidation of economically healthy positions - (File: src/Midnight.sol)

### Summary
The `maxDebt` computation in both `isHealthy` and `liquidate` applies two sequential `mulDivDown` operations: `collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)`. Two chained floor divisions can produce a result strictly less than `floor(collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD))`, the true single-step floor. When a borrower's position sits exactly at the health boundary (true `maxDebt == debt`), the computed `maxDebt` can equal `debt - 1`, causing `originalDebt > maxDebt` to pass in `liquidate` and `isHealthy` to return `false`, both incorrectly.

### Finding Description

**Code path:**

`isHealthy` (`src/Midnight.sol:944-960`) and `liquidate` (`src/Midnight.sol:602-624`) both compute:

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
               .mulDivDown(collateralParam.lltv, WAD);
```

`mulDivDown` is defined in `src/libraries/UtilsLib.sol:29-31` as `(x * y) / d` (Solidity integer division, i.e., floor).

**Root cause — two-step floor vs. one-step floor:**

Let `C = collateral`, `P = price`, `L = lltv`. The two-step result is:

```
v1 = floor(C * P / ORACLE_PRICE_SCALE)
computed_maxDebt = floor(v1 * L / WAD)
```

The true single-step floor is:

```
true_maxDebt = floor(C * P * L / (ORACLE_PRICE_SCALE * WAD))
```

Whenever `C * P mod ORACLE_PRICE_SCALE > 0`, `v1 < C*P/ORACLE_PRICE_SCALE` (exact), and the subsequent multiplication by `L` can push the result below the true floor by exactly 1.

**Concrete counterexample with real scales** (`ORACLE_PRICE_SCALE = 1e36`, `WAD = 1e18`):

- `C = 3`, `price = 4e35`, `lltv = 9e17`
- Step 1: `v1 = floor(3 * 4e35 / 1e36) = floor(1.2) = 1`
- Step 2: `computed_maxDebt = floor(1 * 9e17 / 1e18) = floor(0.9) = 0`
- True value: `floor(3 * 4e35 * 9e17 / 1e54) = floor(1.08) = 1`

With `debt = 1`: true `maxDebt = 1 ≥ debt` → healthy; computed `maxDebt = 0 < debt` → liquidatable.

**Exploit flow:**

1. Borrower has `debt = D`, `collateral = C`, oracle `price = P` such that `floor(C*P*L/(ORACLE_PRICE_SCALE*WAD)) = D` (true boundary).
2. Two-step computation yields `computed_maxDebt = D - 1`.
3. Any liquidator calls `liquidate(market, collateralIndex, seizedAssets, repaidUnits, borrower, false, ...)`.
4. The check at line 622 evaluates `originalDebt > maxDebt` → `D > D-1` → `true` → `NotLiquidatable()` is not thrown.
5. Liquidation proceeds; borrower loses collateral.

**Why existing checks do not stop it:**

- `liquidate` has no tolerance buffer in the health check; it uses the identical two-step `mulDivDown` as `isHealthy`.
- The Certora `Healthiness.spec` uses abstract ghost summaries for `mulDivDown` (lines 23-24 of `Healthiness.spec`) with only monotonicity axioms — it does not prove the concrete two-step rounding behavior and therefore does not cover this case.
- There is no `+1` guard or single-step combined computation anywhere in the health path.

### Impact Explanation

A borrower whose position is economically healthy (true `maxDebt ≥ debt`) can be liquidated by any unprivileged liquidator when the two-step rounding yields `computed_maxDebt = debt - 1`. The liquidator can repay up to `maxRepaid = (debt - computed_maxDebt).mulDivUp(WAD², WAD² - lif*lltv)` units; when `debt - computed_maxDebt = 1` this is a small number (≈ 1–6 units for typical parameters). However, if `market.rcfThreshold` is set to `type(uint256).max` (or any value above the residual collateral value), the RCF bypass at lines 662-667 allows full liquidation of the position, causing the borrower to lose all collateral despite being economically solvent.

### Likelihood Explanation

The precondition requires the oracle price to be at a value where the two-step rounding error fires while the position is exactly at the health boundary. This is a measure-zero condition for any fixed `(C, D, lltv)` triple, making spontaneous occurrence rare. However, it is fully deterministic and repeatable: a liquidator monitoring mempool prices can identify the exact price tick at which the rounding diverges and submit the liquidation atomically. The borrower has no way to prevent this once the price reaches the critical value. The condition is more likely to arise for small collateral amounts (low `C`) and non-round prices, which are common in practice.

### Recommendation

Replace the two chained `mulDivDown` calls with a single combined computation to eliminate the intermediate rounding step:

```solidity
// Instead of:
maxDebt += collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);

// Use:
maxDebt += mulDiv(collateral, price * lltv, ORACLE_PRICE_SCALE * WAD); // single floor
```

Alternatively, add a `+1` tolerance to the liquidatability check so that a position is only liquidatable when `originalDebt > maxDebt + ROUNDING_TOLERANCE`, where `ROUNDING_TOLERANCE` equals the number of activated collaterals (at most `MAX_COLLATERALS_PER_BORROWER = 16`). The single-step approach is preferable as it eliminates the error entirely rather than masking it.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {UtilsLib} from "src/libraries/UtilsLib.sol";
import {ORACLE_PRICE_SCALE, WAD} from "src/libraries/ConstantsLib.sol";

contract TwoStepRoundingPoC is Test {
    using UtilsLib for uint256;

    /// @dev Unit test: demonstrate the two-step mulDivDown gives strictly less
    ///      than the true single-step floor for concrete real-scale values.
    function testTwoStepRoundingError() public pure {
        uint256 C     = 3;
        uint256 price = 4e35;          // 0.4 * ORACLE_PRICE_SCALE
        uint256 lltv  = 9e17;          // 0.9 * WAD

        uint256 twoStep = C.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);
        // floor(3 * 4e35 / 1e36) = floor(1.2) = 1
        // floor(1 * 9e17 / 1e18) = floor(0.9) = 0  <-- computed maxDebt

        uint256 trueFloor = (C * price / ORACLE_PRICE_SCALE * lltv) / WAD;
        // Alternatively: floor(3 * 4e35 * 9e17 / 1e54) = floor(1.08) = 1

        assertEq(twoStep, 0,         "two-step gives 0");
        assertEq(trueFloor, 1,       "true floor gives 1");
        assertLt(twoStep, trueFloor, "two-step < true floor: rounding error confirmed");
    }

    /// @dev Fuzz test: find (collateral, price, lltv) triples where two-step < true floor
    ///      and debt == true floor, asserting isHealthy must return true.
    function testFuzzRoundingBoundary(uint128 C, uint128 price, uint64 lltv) public {
        vm.assume(lltv > 0 && lltv <= WAD);
        vm.assume(price > 0 && price <= ORACLE_PRICE_SCALE);
        vm.assume(C > 0);

        uint256 twoStep  = uint256(C).mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);
        uint256 oneStep  = (uint256(C) * uint256(price) / ORACLE_PRICE_SCALE * uint256(lltv)) / WAD;

        // When the rounding error fires and debt == oneStep:
        if (twoStep < oneStep) {
            uint256 debt = oneStep; // true maxDebt == debt → healthy
            // Protocol computed maxDebt = twoStep < debt → appears unhealthy
            bool protocolSaysHealthy = (twoStep >= debt);
            assertFalse(protocolSaysHealthy,
                "protocol incorrectly reports unhealthy for a truly healthy position");
            // Expected assertion: this assertFalse should PASS (confirming the bug),
            // meaning the protocol would allow liquidation of this healthy position.
        }
    }
}
```

**Expected assertions:**
- `testTwoStepRoundingError`: passes, confirming the two-step result (0) is strictly less than the true floor (1) for the given inputs.
- `testFuzzRoundingBoundary`: the fuzzer finds inputs where `twoStep < oneStep`; for those inputs, `protocolSaysHealthy` is `false` even though `debt == oneStep` (true `maxDebt == debt`), confirming the invariant violation. An integration test wrapping a full `liquidate()` call with these parameters would confirm the liquidation succeeds on a truly healthy position.
