### Title
Dust credit position (credit=1) fully wiped by any lossFactor increase due to mulDivDown rounding to zero - (File: src/Midnight.sol)

### Summary
In `updatePositionView`, the slashing formula `credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)` with `credit=1` always evaluates to `floor((MAX - currentLossFactor) / (MAX - lastLossFactor))`. Since `currentLossFactor > lastLossFactor` after any bad debt event, the numerator is strictly less than the denominator, so the result floors to 0, destroying the entire 1-unit credit position. The proportional loss for `credit=1` with a tiny bad debt (e.g., `badDebt=1` out of `totalUnits=1e18`) should round to 0 slash (lender retains 1 unit), but the formula instead wipes it entirely. `updatePosition` has no access control, so any caller can trigger this for any victim.

### Finding Description

**Code path:**

`updatePositionView` at `src/Midnight.sol:805-806`:
```solidity
uint256 postSlashCredit = _lastLossFactor < type(uint128).max
    ? credit.mulDivDown(type(uint128).max - marketState[id].lossFactor, type(uint128).max - _lastLossFactor)
    : 0;
```

`mulDivDown(a, b, c) = floor(a * b / c)`. With `credit = 1`:

```
postSlashCredit = floor((MAX - currentLossFactor) / (MAX - lastLossFactor))
```

Because `currentLossFactor >= lastLossFactor` (monotonically increasing, enforced by Certora rule `marketLossFactorMonotonicallyIncreases`), the ratio is always `<= 1`. It floors to **0** whenever `currentLossFactor > lastLossFactor` — i.e., after any bad debt event, no matter how small.

**Concrete numeric example:**
- `totalUnits = 1e18`, `badDebt = 1`, `credit = 1`, `lastLossFactor = 0`
- New `lossFactor = MAX - (MAX * (1e18 - 1) / 1e18) = MAX / 1e18`
- `postSlashCredit = floor(1 * (MAX - MAX/1e18) / MAX) = floor((1e18 - 1) / 1e18) = 0`
- Proportional loss should be `floor(1 * 1/1e18) = 0` → lender should retain 1 unit
- Actual result: lender retains 0 units — full wipe

**Exploit flow:**
1. Lender holds `credit = 1` (dust position, reachable via small initial take or prior slashing/fee accrual reducing credit to 1)
2. Any liquidation with bad debt occurs (attacker can be the liquidator, or simply wait for one), causing `marketState[id].lossFactor` to increase above the lender's `lastLossFactor`
3. Attacker (any address) calls `updatePosition(market, victim)` — no authorization required:

```solidity
function updatePosition(Market memory market, address user) external returns (uint128, uint128, uint128) {
    bytes32 id = toId(market);
    require(marketState[id].tickSpacing > 0, MarketNotCreated());
    return _updatePosition(market, id, user);  // no msg.sender check
}
```

4. `_updatePosition` calls `updatePositionView`, which computes `postSlashCredit = 0`, sets `_position.credit = 0`

**Why existing checks fail:**
- The Certora spec `updatePositionEffects` only asserts `creditOf(id, user) <= creditBefore` — it does not assert proportionality
- `LossFactor.spec:updatePositionDoesNotRevert` only checks non-reversion, not correctness of the slash amount
- No floor/minimum-credit guard exists in `_updatePosition` or `updatePositionView`
- The `EcrecoverAuthorizer` is irrelevant here — `updatePosition` is fully permissionless

### Impact Explanation
Any lender whose credit has been reduced to 1 (the minimum non-zero value) will have their entire credit destroyed by the next `updatePosition` call after any bad debt event, regardless of how small the bad debt is relative to `totalUnits`. The lender loses 1 unit of credit when the proportional loss formula should yield 0 slash (retain 1 unit). This violates the protocol's stated invariant ("credit is slashed proportionally") for dust positions. The maximum loss per victim is 1 unit of credit; no funds are transferred to the attacker.

### Likelihood Explanation
**Preconditions:**
1. A lender must hold `credit = 1`. This is reachable: a lender who starts with a small position, or whose credit has been reduced to 1 through accumulated slashing and continuous fee accrual over time.
2. At least one bad debt event must have occurred since the lender's last `updatePosition` (i.e., `marketState[id].lossFactor > position[id][lender].lastLossFactor`). Bad debt events are a normal part of the protocol lifecycle.

**Feasibility:** Both preconditions are organic and require no special privileges. The trigger (`updatePosition`) is permissionless and callable by anyone. The scenario is repeatable across any market with bad debt history.

### Recommendation
Add a minimum-credit floor in `updatePositionView` to prevent rounding from destroying a non-zero credit position entirely when the proportional loss is less than 1 unit:

```solidity
// After computing postSlashCredit:
if (credit > 0 && postSlashCredit == 0 && marketState[id].lossFactor < type(uint128).max) {
    postSlashCredit = 1; // retain at least 1 unit if not a full-loss market
}
```

Alternatively, use a rounding formula that preserves 1 unit when the proportional loss is sub-unit: `postSlashCredit = max(1, credit.mulDivDown(...))` when `lossFactor < MAX`.

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
