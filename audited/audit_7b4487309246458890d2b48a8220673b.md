### Title
Off-by-one in `CannotIncreaseDebtPostMaturity` guard allows debt creation at exactly `block.timestamp == market.maturity` - (`src/Midnight.sol`)

### Summary
The guard `require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity())` uses `<=`, permitting `sellerDebtIncrease > 0` when `block.timestamp` equals `market.maturity` exactly. The protocol's own invariant explicitly states the maturity boundary must not allow debt increase by timestamp equality. This creates seller debt that is immediately liquidatable and buyer credit redeemable against any pre-existing `withdrawable` pool.

### Finding Description
**Code path:** `src/Midnight.sol`, `take()`, line 391.

```solidity
// line 384
uint256 sellerDebtIncrease = units - sellerCreditDecrease;
// ...
// line 391 — BUG: <= should be <
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
// line 414
sellerPos.debt += UtilsLib.toUint128(sellerDebtIncrease);  // executes when timestamp == maturity
```

**Root cause:** `<=` instead of `<`. When `block.timestamp == market.maturity`, the first disjunct is satisfied and `sellerDebtIncrease > 0` is not blocked.

**Attacker inputs:**
- Sell offer (`offer.buy == false`) where `offer.maker` is the seller (authorized the attacker via `EcrecoverAuthorizer` or direct `setIsAuthorized`)
- Seller has zero credit (`sellerPos.credit == 0`), so `sellerCreditDecrease = 0` and `sellerDebtIncrease = units`
- `block.timestamp == market.maturity`

**Exploit flow:**
1. `vm.warp(market.maturity)` — set timestamp to exactly maturity
2. Attacker calls `take(sellOffer, ratifierData, units, buyer, receiver, ...)` — `sellerDebtIncrease = units`, check at line 391 passes because `block.timestamp <= maturity`
3. `sellerPos.debt += units` (line 414); `buyerPos.credit += buyerCreditIncrease` (line 410)
4. `_marketState.withdrawable` is **not** changed by `take()`
5. If `_marketState.withdrawable >= units` (from prior repayments by other borrowers), buyer calls `withdraw(market, units, buyer, receiver)` — line 494 decrements `withdrawable` by `units` and transfers tokens out

**Why checks fail:** The `CannotIncreaseDebtPostMaturity` check is the only guard against this. It fails at the boundary because of `<=`. No other check prevents debt creation at exactly maturity.

### Impact Explanation
New seller debt is created at maturity — a timestamp at which all such debt is immediately liquidatable (post-maturity mode). The corresponding buyer credit can be redeemed against any pre-existing `withdrawable` in the market (accumulated from other borrowers' repayments), draining the pool that other lenders depend on for their own `withdraw()` calls. The seller is placed into an immediately-liquidatable position without their consent (the attacker is authorized, but the authorization model is coarse-grained and not action-scoped).

### Likelihood Explanation
**Preconditions:**
- `block.timestamp == market.maturity` — achievable by any block proposer or by a transaction landing in the exact block; also reachable in any EVM environment where the caller controls timing
- A sell offer exists (or the attacker constructs one as the maker)
- Attacker is authorized for the seller (via `EcrecoverAuthorizer` or direct authorization)
- `withdrawable > 0` in the market (required for the `withdraw()` step; not required for the invariant violation itself)

The invariant violation (debt at maturity) occurs regardless of `withdrawable`. The fund-loss step requires pre-existing `withdrawable`. Both conditions are realistic in an active market near maturity.

### Recommendation
Change `<=` to `<` on line 391:

```solidity
// Before (buggy):
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());

// After (correct):
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the on-chain check with the stated invariant: debt must not increase **at or after** maturity.
