Audit Report

## Title
Authorized agent can grind victim's nonce in `EcrecoverAuthorizer` to permanently invalidate all pending pre-signed authorizations - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` increments `nonce[authorization.authorizer]` on every successful call, but the authorization check accepts any `signer` for whom `IMidnight.isAuthorized(authorizer, signer)` returns `true`, not only the authorizer's own key. An attacker who holds a valid on-chain authorization from a victim can submit an unbounded sequence of no-op re-authorizations of themselves, each consuming one nonce slot at gas cost only, permanently invalidating every pre-signed authorization the victim distributed offline.

## Finding Description
**Exact code path â€” `src/periphery/EcrecoverAuthorizer.sol` lines 24â€“48:**

The nonce is keyed on `authorization.authorizer` and incremented unconditionally on success: [1](#0-0) 

The authorization check accepts the authorizer's own key **or** any on-chain delegate: [2](#0-1) 

The downstream call to `Midnight.setIsAuthorized` succeeds whenever `isAuthorized[victim][EcrecoverAuthorizer]` is true: [3](#0-2) [4](#0-3) 

**Root cause.** The nonce slot is keyed on `authorization.authorizer` (the victim), but the signer check is satisfied by any on-chain delegate. There is no requirement that `signer == authorization.authorizer`. The nonce increment at line 26 is committed whenever the full call succeeds, including no-op writes.

**Exploit flow (N iterations).**

Preconditions:
1. `isAuthorized[victim][EcrecoverAuthorizer] == true` â€” required for `Midnight.setIsAuthorized` (called at line 47) to not revert; this is the normal operational state for any user of the contract.
2. `isAuthorized[victim][attacker] == true` â€” victim has granted the attacker on-chain authorization for any purpose.

For `i = 0 â€¦ N-1`:
1. Attacker constructs `Authorization(authorizer=victim, authorized=attacker, isAuthorized=true, nonce=i, deadline=future)`.
2. Attacker signs the EIP-712 digest with their own private key â†’ `signer = attacker`.
3. Attacker calls `EcrecoverAuthorizer.setIsAuthorized(auth, sig)`.
4. Line 25: deadline check passes.
5. Line 26: `i == nonce[victim]` â†’ passes; `nonce[victim]` becomes `i+1`.
6. Lines 33â€“36: `attacker != victim`; `isAuthorized[victim][attacker] == true` â†’ passes.
7. Line 47: `Midnight.setIsAuthorized(attacker, true, victim)` â†’ no-op write (attacker already authorized).

After N calls: `nonce[victim] == N`. Every pre-signed authorization the victim distributed with `nonce < N` reverts with `InvalidNonce()`.

**Why existing checks fail.** The `Unauthorized` guard is satisfied by the attacker's own valid ECDSA signature because the delegation branch is true. The nonce check enforces sequential ordering only; it does not restrict which party may consume nonce slots. No check requires `signer == authorization.authorizer`.

## Impact Explanation
All pending pre-signed authorizations the victim has distributed offline (for `take`, `repay`, `withdraw`, `withdrawCollateral`, `liquidate`, `setConsumed`, or any other action gated on `isAuthorized`) become permanently invalid. Counterparties holding those signatures cannot execute them. The attacker can immediately repeat the attack after any re-signing, creating a sustained, low-cost denial-of-service against every signature-based action in the protocol. This constitutes permanent corruption of user state and severe service degradation under realistic attacker input.

## Likelihood Explanation
Both preconditions are normal operational states for any user relying on the signature-based authorization flow: authorizing `EcrecoverAuthorizer` is required to use it, and authorizing other agents (e.g., smart contracts, keepers, counterparties) is the protocol's intended delegation model, explicitly documented at: [5](#0-4) 

The attacker requires no capital, no special role, and no privileged access beyond the on-chain authorization the victim already granted. The attack costs only gas and is repeatable indefinitely. The attacker can front-run any revocation attempt by submitting additional nonce-consuming transactions.

## Recommendation
Restrict the signer check in `EcrecoverAuthorizer.setIsAuthorized` to require `signer == authorization.authorizer` only, removing the delegation branch entirely. The nonce is a replay-protection mechanism for the authorizer's own key; allowing delegates to consume it breaks the invariant that only the authorizer controls their nonce sequence. If delegation is desired, it should be implemented without touching the authorizer's nonce (e.g., key the nonce on the signer, not the authorizer, or require a separate nonce namespace per delegate).

## Proof of Concept
Minimal Foundry test:
1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. As `victim`: call `Midnight.setIsAuthorized(EcrecoverAuthorizer, true, victim)` and `Midnight.setIsAuthorized(attacker, true, victim)`.
3. As `victim`: sign an `Authorization(authorizer=victim, authorized=someThirdParty, isAuthorized=true, nonce=0, deadline=future)` and distribute it offline.
4. As `attacker` (EOA): loop N times â€” construct `Authorization(authorizer=victim, authorized=attacker, isAuthorized=true, nonce=i, deadline=future)`, sign with attacker key, call `EcrecoverAuthorizer.setIsAuthorized`.
5. Assert `nonce[victim] == N`.
6. Attempt to submit the victim's pre-signed authorization from step 3 â€” assert it reverts with `InvalidNonce()`.
