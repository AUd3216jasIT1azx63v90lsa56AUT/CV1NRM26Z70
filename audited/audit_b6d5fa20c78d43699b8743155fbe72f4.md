### Title
Continuous Fee Unclaimable When `withdrawable < continuousFeeCredit` Due to Missing Invariant in `claimContinuousFee` - (`src/Midnight.sol`)

### Summary
`claimContinuousFee` decrements `_marketState.withdrawable` by `amount` without first checking that `withdrawable >= amount`. Because `_updatePosition` increases `continuousFeeCredit` without ever increasing `withdrawable`, the two counters diverge in any market where borrowers have not repaid. The feeClaimer cannot claim legitimately accrued fees, violating the invariant that accrued continuous fees are always claimable.

### Finding Description

**Root cause — `claimContinuousFee` (`src/Midnight.sol:312-325`):**

```solidity
function claimContinuousFee(Market memory market, uint256 amount, address receiver) external {
    bytes32 id = toId(market);
    MarketState storage _marketState = marketState[id];
    require(msg.sender == feeClaimer, OnlyFeeClaimer());
    require(_marketState.tickSpacing > 0, MarketNotCreated());

    _marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
    _marketState.totalUnits -= UtilsLib.toUint128(amount);
    _marketState.withdrawable -= UtilsLib.toUint128(amount);   // ← no guard
    ...
}
``` [1](#0-0) 

The only pre-conditions checked are `msg.sender == feeClaimer` and `tickSpacing > 0`. There is no `require(amount <= _marketState.withdrawable)`. Solidity 0.8 checked arithmetic means the subtraction on line 320 reverts with an underflow whenever `withdrawable < amount`.

**How `continuousFeeCredit` grows without `withdrawable` growing:**

`_updatePosition` (`src/Midnight.sol:832-851`) increases `continuousFeeCredit` by `accruedFee` but does not touch `withdrawable`:

```solidity
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
``` [2](#0-1) 

`withdrawable` only increases in `repay` (+`units`) and `liquidate` (+`repaidUnits`): [3](#0-2) [4](#0-3) 

The Certora spec `withdrawableUnchanged` explicitly confirms that `updatePosition` (and `take`) leave `withdrawable` unchanged: [5](#0-4) 

**Exploit path (no privileged action needed to reach the bad state):**

1. Lender provides `C` units of credit via `take`; borrower receives `C` units of debt. `withdrawable = 0`, `continuousFeeCredit = 0`.
2. Time passes. Anyone calls `updatePosition(market, lender)`. Lender's credit decreases by `fee`; `continuousFeeCredit += fee`. `withdrawable` remains `0`.
3. Borrower does **not** repay. State: `continuousFeeCredit = fee > 0`, `withdrawable = 0`.
4. feeClaimer calls `claimContinuousFee(market, fee, receiver)` with `fee <= continuousFeeCredit`.
5. Line 318: `continuousFeeCredit -= fee` → OK.
6. Line 319: `totalUnits -= fee` → OK.
7. Line 320: `withdrawable -= fee` → **arithmetic underflow, revert**.

The Certora liveness rule for `claimContinuousFee` in `certora/specs/Role.spec:264` independently confirms the revert condition — it asserts success requires `amount <= withdrawableBefore`, a constraint the code never enforces: [6](#0-5) 

The existing test `testClaimContinuousFee` works around this silently by calling `repay` before claiming, with the comment *"Repay so withdrawable covers the claim"*: [7](#0-6) 

### Impact Explanation
Accrued continuous fees are permanently unclaimable whenever `withdrawable < continuousFeeCredit`. This is the normal state for any market with outstanding borrower debt and accrued fees. The feeClaimer receives zero revenue from such markets until borrowers repay enough to bring `withdrawable >= continuousFeeCredit`, which may never happen (e.g., defaulted borrowers, post-maturity markets with bad debt).

### Likelihood Explanation
The condition `continuousFeeCredit > withdrawable` is the **default state** of any active lending market: fees accrue continuously from the moment a position is opened, while `withdrawable` only grows on repayment or liquidation. No special preconditions, attacker inputs, or oracle manipulation are required. The state is reached by ordinary protocol usage and persists for the entire life of the loan.

### Recommendation
In `_updatePosition`, when `continuousFeeCredit` is increased by `accruedFee`, also increase `withdrawable` by `accruedFee` (and decrease `totalUnits` by `accruedFee` to preserve the `totalUnits = sumDebt + withdrawable` invariant). Alternatively, add an explicit guard in `claimContinuousFee`:

```solidity
require(amount <= _marketState.withdrawable, InsufficientWithdrawable());
```

and separately ensure the fee accrual path keeps `continuousFeeCredit <= withdrawable` at all times. The deeper fix is the former: mirror the credit-to-fee transfer in `withdrawable`/`totalUnits` at accrual time so the claim path never encounters an underflow.

### Proof of Concept

```solidity
// Foundry unit test
function testContinuousFeeUnclaimableWithoutRepay(
    uint256 credit, uint256 feeRate, uint256 ttm, uint256 elapsed
) public {
    credit  = bound(credit,  1, MAX_CREDIT);
    feeRate = bound(feeRate, 1, MAX_CONTINUOUS_FEE);
    ttm     = bound(ttm,     2, 360 days);
    elapsed = bound(elapsed, 1, ttm - 1);

    // 1. Lender provides credit; borrower takes debt.
    setupLender(credit, feeRate, ttm);

    // 2. Time passes; fee accrues into continuousFeeCredit.
    vm.warp(vm.getBlockTimestamp() + elapsed);
    midnight.updatePosition(market, lender);

    uint256 feeAmount = midnight.continuousFeeCredit(id);
    vm.assume(feeAmount > 0);

    // 3. Borrower does NOT repay → withdrawable == 0.
    assertEq(midnight.withdrawable(id), 0, "withdrawable must be 0");
    assertGt(feeAmount, 0, "continuousFeeCredit must be > 0");

    // 4. feeClaimer attempts to claim a valid amount (≤ continuousFeeCredit).
    //    Expected: REVERT due to withdrawable underflow.
    vm.prank(feeClaimer);
    vm.expectRevert(); // arithmetic underflow on withdrawable -= amount
    midnight.claimContinuousFee(market, feeAmount, feeClaimer);
}
```

**Expected assertion:** the call reverts. If the protocol is fixed (e.g., `withdrawable` is increased at accrual time), the call should succeed and `continuousFeeCredit` should equal zero after the claim.
