Audit Report

## Title
Authorized Delegate Can Perpetually Front-Run Revocations via EcrecoverAuthorizer Delegate-Signing Loop - (File: src/periphery/EcrecoverAuthorizer.sol)

## Summary
`EcrecoverAuthorizer.setIsAuthorized` accepts EIP-712 signatures from any address already authorized in Midnight on behalf of the `authorizer`, not just the `authorizer` itself. An attacker who controls an authorized operator key can perpetually consume every revocation nonce the victim broadcasts by front-running with a fresh re-grant signed by the operator key, consuming the nonce before the victim's revocation lands. A smart-contract victim with no direct `Midnight.setIsAuthorized` call path is permanently unable to revoke the operator's authorization.

## Finding Description
**Root cause — `src/periphery/EcrecoverAuthorizer.sol` lines 33–36:**

```solidity
require(
    signer == authorization.authorizer
        || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
    Unauthorized()
);
```

Any address `signer` for which `isAuthorized[victim][signer] == true` in Midnight can produce a valid `Authorization` struct with `authorizer = victim` and any future nonce. This creates a circular dependency: granting authorization to an operator gives that operator the power to sign further authorizations on the victim's behalf — including re-grants that consume the exact nonce the victim is attempting to use for revocation.

**Nonce mechanism — line 26:**

```solidity
require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
```

Nonces are strictly sequential. Consuming nonce `N` with a re-grant forces the victim's revocation (also targeting nonce `N`) to fail with `InvalidNonce`.

**Exploit path:**

1. Victim (smart contract, no direct `Midnight.setIsAuthorized` call path) previously authorized `operator` via `EcrecoverAuthorizer`. State: `isAuthorized[victim][operator] = true`, `nonce[victim] = 1`.
2. Victim signs `Authorization{authorizer=victim, authorized=operator, isAuthorized=false, nonce=1, deadline=T}` and broadcasts.
3. Attacker (controlling `operator`) observes the pending tx in the mempool. Attacker signs `Authorization{authorizer=victim, authorized=operator, isAuthorized=true, nonce=1, deadline=T+1}` with `operator`'s key. The line 34 check passes because `isAuthorized[victim][operator] == true`. Attacker front-runs: `nonce[victim] = 2`, authorization remains `true`.
4. Victim's revocation lands with `nonce=1`, fails `InvalidNonce`.
5. Steps 2–4 repeat for every nonce increment. The attacker needs only one EOA key and a mempool watcher.

**Why existing checks are insufficient:**
- The nonce check (line 26) prevents replay of the *same* signature but does not prevent the attacker from consuming the *next* nonce with a freshly signed re-grant.
- The deadline check (line 25) is irrelevant; the attacker can use any future deadline.
- `Midnight.setIsAuthorized` (lines 731–735) would allow direct revocation when `onBehalf == msg.sender`, but this requires the victim contract to have a code path that calls it directly — which is precisely the scenario excluded by precondition 1.

## Impact Explanation
The operator retains permanent `isAuthorized[victim][operator] = true` in Midnight. With this authorization the operator can call `withdraw`, `withdrawCollateral`, `repay`, `take`, `setConsumed`, and any other `onBehalf`-gated entry point on behalf of the victim. The victim's funds (collateral, withdrawable credit, pending fees) are fully accessible to the attacker with no on-chain revocation path available, constituting a permanent fund drain / fund freeze. This is a **Critical** severity impact: direct theft of assets and permanent loss of access control.

## Likelihood Explanation
**Required preconditions:**
1. Victim is a smart contract whose code has no direct call to `Midnight.setIsAuthorized` — realistic for protocol integrations, smart-contract wallets, or multisigs that route all Midnight interaction through `EcrecoverAuthorizer`.
2. Victim signed at least one `isAuthorized=true` authorization and the attacker obtained the operator key (the attacker *is* the operator).
3. Attacker controls the `authorized` address (operator) and can sign new `Authorization` structs with it.

All three preconditions require no privileged protocol access, oracle manipulation, or user mistake beyond the initial authorization issuance. The front-running loop is repeatable every block at negligible gas cost for the attacker. MEV infrastructure makes same-block front-running trivially reliable.

## Recommendation
Restrict the signer check for revocations (`isAuthorized=false`) to the `authorizer` only. Delegated signers should only be permitted to issue grants, not revocations:

```solidity
require(
    signer == authorization.authorizer
        || (authorization.isAuthorized && IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer)),
    Unauthorized()
);
```

This ensures that only the authorizer themselves can revoke an authorization, while still allowing authorized operators to grant further authorizations on behalf of the user (preserving the documented design intent from `Midnight.sol` lines 101–109).

## Proof of Concept
**Minimal Foundry test outline:**

1. Deploy `Midnight` and `EcrecoverAuthorizer`.
2. Create victim smart contract (no `setIsAuthorized` call path). Generate operator EOA key pair.
3. Victim signs `Authorization{authorizer=victim, authorized=operator, isAuthorized=true, nonce=0}` → call `EcrecoverAuthorizer.setIsAuthorized`. Confirm `isAuthorized[victim][operator] == true`, `nonce[victim] == 1`.
4. Victim signs `Authorization{authorizer=victim, authorized=operator, isAuthorized=false, nonce=1}`.
5. Before submitting victim's tx, attacker signs `Authorization{authorizer=victim, authorized=operator, isAuthorized=true, nonce=1}` with operator key → call `EcrecoverAuthorizer.setIsAuthorized`. Confirm succeeds: `nonce[victim] == 2`, `isAuthorized[victim][operator] == true`.
6. Submit victim's revocation (nonce=1) → confirm revert with `InvalidNonce`.
7. Repeat steps 4–6 for nonce=2, 3, … to demonstrate the loop is unbounded.
8. Assert `isAuthorized[victim][operator] == true` after N iterations, confirming permanent inability to revoke.
