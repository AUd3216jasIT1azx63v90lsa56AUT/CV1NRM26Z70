### Title
Zero-`buyerAssets` rounding allows unlimited free fills of a buy offer with `maxAssets` cap, mutating credit/debt without advancing `consumed` - (`src/Midnight.sol`)

### Summary
When a buy offer sets `maxAssets > 0`, the `consumed` mapping is incremented by `buyerAssets`, computed as `units.mulDivDown(buyerPrice, WAD)`. Integer division truncates this to zero whenever `units * buyerPrice < WAD`. Because position state (credit, debt, `totalUnits`) is updated using `units` directly—not `buyerAssets`—a taker can repeatedly call `take` with small `units` values, mutating the maker's credit and the taker's debt on every call while `consumed` never advances, bypassing the `maxAssets` cap entirely.

**Root cause** — `consumed` tracks asset-denominated fill progress, but position mutations are unit-denominated. When `units * buyerPrice < WAD`, `mulDivDown` returns 0, so `consumed` is unchanged. No check enforces `buyerAssets > 0` before proceeding.

**Price system** — `tickToPrice(0) = 0`, `tickToPrice(2) = 1e12` (first non-zero price, `PRICE_ROUNDING_STEP = 1e12`). With `price = 1e12` and `units < 1e6`, `units * 1e12 / 1e18 = 0`. With `price` near `WAD` and `units = 1`, `1 * (WAD - ε) / WAD = 0`. [3](#0-2) [4](#0-3) 

**Attacker-controlled inputs** — `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick` set to any valid tick (including `MAX_TICK - 16` or lower), `units` chosen so that `units * buyerPrice < WAD`. No privileged access required; any taker can call `take`.

**Exploit flow**:
1. Maker posts a buy offer with `maxAssets = N`, any tick where `tickToPrice(tick) < WAD`.
2. Taker calls `take(offer, ..., units=1, ...)` repeatedly.
3. Each call: `buyerAssets = 1 * price / WAD = 0` → `consumed` unchanged → `require(0 <= N)` passes.
4. Each call: `buyerCreditIncrease = units = 1` → maker's credit grows by 1, taker's debt grows by 1.
5. Taker pays 0 tokens (`safeTransferFrom(..., 0)` is a no-op).
6. Repeat indefinitely; `consumed` never reaches `maxAssets`.

**Existing checks that fail to stop it**:
- `require(newConsumed <= offer.maxAssets)` — passes because `newConsumed = old + 0 = old`.
- `require(atMostOneNonZero(maxAssets, maxUnits))` — only validates offer structure, not fill size.
- No `require(units > 0)` or `require(buyerAssets > 0)` guard exists.

**Confirmed by existing test** — `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 857–889) already demonstrates this exact behavior: `buyerAssets = 0`, `consumed` unchanged at `maxAssets`, yet `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all increase. [5](#0-4) 

### Impact Explanation
A taker can fill a buy offer with `maxAssets > 0` an unlimited number of times at zero token cost. Each fill increases the maker's credit and the taker's debt by `units` without any token transfer and without advancing `consumed`. The maker's offer is effectively filled for free: the `maxAssets` cap is never consumed, the taker incurs debt without paying, and the maker gains credit without receiving assets. This directly violates the invariant that every credit increase must correspond to valid consumed accounting and that offers cannot be overfilled or reused.

### Likelihood Explanation
**Preconditions**: (1) a buy offer exists with `maxAssets > 0`; (2) `tickToPrice(offer.tick) < WAD` (true for all ticks except `MAX_TICK`); (3) taker chooses `units` small enough that `units * buyerPrice < WAD` (e.g., `units = 1` works for any price below WAD). Condition (2) is satisfied by virtually every valid offer. Condition (3) requires only `units = 1`. The attack is trivially repeatable in a loop within a single transaction via `multicall`, and requires no special role, no oracle manipulation, and no front-running.

### Recommendation
Add a guard that rejects a fill when the asset amount rounds to zero. For a buy offer with `maxAssets > 0`, require `buyerAssets > 0` before proceeding:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(assetsDelta > 0 || units == 0, ZeroAssetFill());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, enforce `require(units == 0 || buyerAssets > 0)` unconditionally for buy offers, mirroring the intent that `maxAssets` tracks real economic throughput.
