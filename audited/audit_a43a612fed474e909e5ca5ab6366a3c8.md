### Title
Fee-on-transfer loanToken causes `repayAndWithdrawCollateral` to always revert due to bundler balance shortfall - (File: src/periphery/MidnightBundles.sol)

### Summary

In `repayAndWithdrawCollateral`, the repay unit count is derived from the nominal `assets` parameter, but `pullToken` delivers only `assets - transferFee` to the bundler when the loan token charges a transfer fee. When `Midnight.repay` subsequently pulls `units` from the bundler, the bundler's actual balance is insufficient, causing the call to revert. No existing check reconciles the received amount against the computed `units`.

### Finding Description

**Exact code path** (`src/periphery/MidnightBundles.sol` lines 328–334):

```solidity
address loanToken = market.loanToken;
uint256 referralFeeAssets = assets.mulDivDown(referralFeePct, WAD);  // L329
uint256 units = assets - referralFeeAssets;                           // L330
pullToken(loanToken, msg.sender, assets, loanTokenPermit);            // L331
forceApproveMax(loanToken, MIDNIGHT);                                 // L332

IMidnight(MIDNIGHT).repay(market, units, onBehalf, address(0), "");  // L334
```

`pullToken` resolves to `SafeTransferLib.safeTransferFrom(token, from, address(this), amount)` with `amount = assets`. For a fee-on-transfer token with fee rate `f`, the bundler receives `assets - f` tokens, not `assets`.

`units` is computed as `assets - referralFeeAssets` — a value derived from the nominal input, not the actual received balance.

Inside `Midnight.repay` (src/Midnight.sol line 511, 520):

```solidity
address payer = callback != address(0) ? callback : msg.sender;  // callback=address(0) → payer=bundler
...
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
```

Midnight pulls exactly `units` from the bundler. The bundler holds `assets - f`. The pull fails whenever:

```
units > assets - f
⟺  assets - referralFeeAssets > assets - f
⟺  f > referralFeeAssets
```

When `referralFeePct = 0` (the common case), `referralFeeAssets = 0` and `units = assets`, so the condition reduces to `f > 0` — true for any fee-on-transfer token regardless of fee magnitude. Even with a non-zero referral fee, any token whose per-transfer fee exceeds `referralFeeAssets` triggers the revert.

**Attacker-controlled inputs:** `assets` (any positive value), `referralFeePct` (0 is sufficient), `market.loanToken` pointing to a fee-on-transfer token.

**Existing protections reviewed:** The only token-type disclaimer in `MidnightBundles.sol` (lines 23–24) excludes tokens that revert on `approve(0); approve(max)`. Fee-on-transfer tokens are not excluded. No balance snapshot or received-amount reconciliation exists between `pullToken` and the `repay` call.

### Impact Explanation

Every call to `repayAndWithdrawCollateral` with a fee-on-transfer loan token reverts at the `safeTransferFrom` inside `Midnight.repay` due to the bundler's insufficient balance. The borrower cannot repay debt or withdraw collateral through the bundler, permanently blocking the intended repay-and-withdraw flow for that token type.

### Likelihood Explanation

The precondition is that the market's `loanToken` charges a transfer fee. Tokens with configurable fees (e.g., tokens where the owner can enable fees) or tokens that are inherently fee-on-transfer satisfy this. The borrower triggers the path by calling `repayAndWithdrawCollateral` with `referralFeePct = 0` (or any value where `f > referralFeeAssets`). No privileged action is required; any borrower with an open position in such a market can trigger the revert. The condition is deterministic and repeatable on every call.

### Recommendation

After `pullToken`, measure the actual received balance and use it to derive `units`:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
pullToken(loanToken, msg.sender, assets, loanTokenPermit);
uint256 received = IERC20(loanToken).balanceOf(address(this)) - balanceBefore;

uint256 referralFeeAssets = received.mulDivDown(referralFeePct, WAD);
uint256 units = received - referralFeeAssets;
```

This ensures `units` never exceeds the bundler's actual balance before the `repay` call.

    function pullToken(address token, address from, uint256 amount, TokenPermit memory permit) internal {
        if (permit.kind == PermitKind.ERC2612) {
            (uint256 deadline, uint8 v, bytes32 r, bytes32 s) =
                abi.decode(permit.data, (uint256, uint8, bytes32, bytes32));
            // Tolerate revert: a third party may have already consumed the permit.
            try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
            SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
        } else if (permit.kind == PermitKind.Permit2) {
            (uint256 nonce, uint256 deadline, bytes memory signature) =
                abi.decode(permit.data, (uint256, uint256, bytes));

                    IPermit2.SignatureTransferDetails(address(this), amount),
                    from,
                    signature
                );
        } else {
            SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
        }
