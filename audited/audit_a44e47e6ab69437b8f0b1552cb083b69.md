Audit Report

## Title
Zero-`buyerAssets` rounding bypasses `maxAssets` cap, enabling unbounded phantom credit creation on buy offers with sub-WAD price — (File: `src/Midnight.sol`)

## Summary
When a buy offer's tick maps to a price below WAD, `mulDivDown(1, offerPrice, WAD)` rounds to zero, causing the `consumed` counter to never advance regardless of how many times the offer is taken. Because `EcrecoverRatifier` is entirely stateless, the same `ratifierData` passes `isRatified` on every call. A maker controlling two addresses can replay the take indefinitely, accumulating phantom credit (lender position) without transferring any loan tokens, then collect repayment at maturity.

## Finding Description

**Root cause — `src/Midnight.sol:363–369`**

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

For a buy offer at `tick = MAX_TICK − 16`, `offerPrice = tickToPrice(tick) < WAD`. With `units = 1`: `buyerAssets = mulDivDown(1, offerPrice, WAD) = 0`. The consumed increment is 0; `require(0 ≤ maxAssets)` is unconditionally satisfied. The protocol's own NatSpec at line 94 acknowledges: *"It is possible to give units to a fully consumed assets-based buy offer with price < 1."*

**EcrecoverRatifier is stateless — `src/ratifiers/EcrecoverRatifier.sol:33–46`**

The ratifier performs exactly three checks: Merkle proof membership, root-not-canceled, and ECDSA signature validity. There is no per-take nonce, counter, or bitmap. The identical `(sig, root, leafIndex, proof)` tuple passes `isRatified` on every invocation.

**Position state is mutated despite zero asset transfer — `src/Midnight.sol:408–417`**

Credit/debt accounting uses `units` directly, not `buyerAssets`. Even when `buyerAssets = 0`, `buyerCreditIncrease`, `sellerDebtIncrease`, and `totalUnits` are all updated based on `units = 1`. The test `testBugBuyMaxAssetsBypass` (line 858–889) confirms this: after a take with `buyerAssets = 0`, `creditOf(lender)` increases, `debtOf(borrower)` increases, and `totalUnits` increases, while token balances and the consumed counter remain unchanged.

**Exploit flow**

1. Maker creates a buy offer: `maxAssets = N`, `tick = MAX_TICK − 16`, signs a Merkle root via EcrecoverRatifier.
2. Maker's second address (taker) calls `take(offer, validProof, units=1, ...)`.
3. `buyerAssets = mulDivDown(1, offerPrice, WAD) = 0`.
4. `consumed[maker][group] += 0` → consumed stays at 0; `require(0 ≤ N)` passes.
5. Maker's credit increases by 1 unit; taker's debt increases by 1 unit; 0 tokens transferred.
6. Steps 2–5 repeat K times. Consumed never advances; the offer is never exhausted.
7. At maturity, maker redeems phantom credit for real loan tokens from the protocol pool.

## Impact Explanation
The maker accumulates an unbounded lending position (credit) without depositing any loan tokens. At maturity, this phantom credit entitles the maker to receive repayment from the protocol's settlement pool, constituting direct theft of assets from other participants. The `maxAssets` cap — the sole on-chain mechanism limiting offer fill depth — is rendered completely ineffective for any buy offer at a sub-WAD tick.

## Likelihood Explanation
Preconditions are minimal: any tick in roughly the upper half of the tick range satisfies `offerPrice < WAD`; `units = 1` always produces `buyerAssets = 0`; the maker needs only one additional address to act as taker. The attack is executable in a single transaction via multicall, requires no privileged access, and costs only gas. The EcrecoverRatifier's statelessness means no additional setup is needed after the initial signature.

## Recommendation
Add a guard in `take` that rejects a non-zero `units` value when the resulting asset amount rounds to zero:

```solidity
if (offer.maxAssets > 0 && units > 0) {
    require(offer.buy ? buyerAssets > 0 : sellerAssets > 0, ZeroAssets());
}
```

Alternatively, enforce a minimum tick such that `tickToPrice(tick) >= WAD` for asset-capped buy offers, or switch the consumed accounting to track `units` instead of `buyerAssets` when `offer.buy == true`.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` at `test/TakeTest.sol:858–889` is a direct reproduction:
- Sets `maxAssets = 1`, `tick = MAX_TICK − 16`
- Pre-fills consumed to `maxAssets` (fully consumed state)
- Calls `take(1, borrower, lenderOffer)` and asserts `buyerAssets == 0`
- Asserts consumed remains at `maxAssets` (not incremented)
- Asserts `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all increased

To demonstrate unbounded replay, extend the test to call `take` in a loop K times and assert that consumed remains 0 while credit grows by K units with zero token cost.
