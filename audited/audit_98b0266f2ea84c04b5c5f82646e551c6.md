### Title
Permissionless `setIsAuthorized` Allows Attacker to Front-Run Revocation by Selectively Submitting a Stale Same-Nonce Authorization - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` is fully permissionless: any caller can submit any validly-signed `Authorization` struct. Because the nonce is strictly sequential and only one of two same-nonce authorizations can ever succeed, an attacker who holds a signed `auth_true{nonce=N}` can front-run the authorizer's `auth_false{nonce=N}` revocation, consuming nonce N with the grant and permanently invalidating the revocation for that nonce slot.

### Finding Description
**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` (lines 24–48, `src/periphery/EcrecoverAuthorizer.sol`):

```solidity
function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
    require(block.timestamp <= authorization.deadline, Expired());
    require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
    // ... ecrecover, then:
    IMidnight(MIDNIGHT).setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
}
```

There is no restriction on `msg.sender`. [1](#0-0) 

The nonce check `authorization.nonce == nonce[authorization.authorizer]++` is a strict sequential counter: exactly one authorization per nonce slot can ever be executed. [2](#0-1) 

**Exploit flow:**

1. Authorizer A signs `auth_true{authorizer=A, authorized=X, isAuthorized=true, nonce=0, deadline=T}` and shares or broadcasts it (e.g., to a relayer, or it is intercepted).
2. A changes their mind. Because nonce is still 0 (auth_true was never submitted), A signs `auth_false{authorizer=A, authorized=X, isAuthorized=false, nonce=0, deadline=T+1}` and submits or broadcasts it.
3. Attacker observes `auth_false` in the mempool and front-runs by calling `ecrecoverAuthorizer.setIsAuthorized(auth_true, sig_true)`.
4. `auth_true` passes all checks: `block.timestamp <= T` ✓, `nonce[A] == 0` ✓, `ecrecover == A` ✓. Nonce increments to 1. `isAuthorized[A][X]` is set to `true`.
5. A's `auth_false{nonce=0}` now reverts with `InvalidNonce` because `nonce[A] == 1`.
6. Final state: `isAuthorized[A][X] = true` despite A's most recently expressed intent being revocation.

**Why existing checks fail:**

- `Expired()`: auth_true's deadline `T` is still in the future at attack time.
- `InvalidNonce()`: auth_true carries nonce=0, which matches `nonce[A]` at the moment of submission.
- `Unauthorized()`: the signature is genuinely from A.
- No `msg.sender` guard exists anywhere in the function. [3](#0-2) 

**Compounding factor:** Even if A directly calls `Midnight.setIsAuthorized(X, false, A)` on-chain to revoke, the attacker can still submit `auth_true{nonce=0}` through `EcrecoverAuthorizer` (nonce=0 is still unconsumed), which calls `Midnight.setIsAuthorized(X, true, A)` and overrides A's direct revocation. [4](#0-3) 

### Impact Explanation
An unprivileged attacker who possesses any previously signed `Authorization` can permanently prevent the authorizer from revoking it via the same nonce, leaving `isAuthorized[A][X] = true` against A's will. Because `isAuthorized` gates `withdraw`, `withdrawCollateral`, `repay`, `setConsumed`, and `setIsAuthorized` on behalf of A, the unauthorized operator X retains full delegated control over A's funds and positions. [5](#0-4) 

### Likelihood Explanation
**Preconditions:**
- Attacker must possess a valid signed `Authorization` from A (e.g., obtained from a relayer, off-chain signing flow, or prior broadcast).
- auth_true's deadline must not have expired at attack time.
- A must attempt to revoke using the same nonce (the only option when auth_true was never submitted).

**Feasibility:** Front-running is standard on any EVM chain without private mempools. The attacker needs only to watch the mempool for `auth_false` and submit `auth_true` with higher gas. Repeatable as long as the attacker holds any unsubmitted signed authorization. [1](#0-0) 

### Recommendation
Add a self-callable nonce-invalidation function that lets the authorizer atomically burn the current nonce without granting any authorization:

```solidity
function invalidateNonce() external {
    nonce[msg.sender]++;
    emit NonceInvalidated(msg.sender, nonce[msg.sender] - 1);
}
```

This allows A to consume nonce=0 themselves, rendering any attacker-held `auth_true{nonce=0}` permanently invalid before the attacker can submit it. This is the pattern used by Uniswap Permit2 and EIP-4494. [6](#0-5) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {EcrecoverAuthorizerTest} from "./SetIsAuthorizedWithSigTest.sol";
import {IEcrecoverAuthorizer, Authorization, Signature} from "../src/periphery/interfaces/IEcrecoverAuthorizer.sol";

contract NonceRaceTest is EcrecoverAuthorizerTest {
    function testAttackerFrontRunsRevocation() public {
        // Prerequisite: borrower authorizes EcrecoverAuthorizer to act on their behalf
        vm.prank(borrower);
        midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

        // Step 1: borrower signs auth_true{nonce=0}
        Authorization memory authTrue = Authorization({
            authorizer: borrower,
            authorized: lender,
            isAuthorized: true,
            nonce: 0,
            deadline: block.timestamp + 1 days
        });
        Signature memory sigTrue = signAuthorization(authTrue, borrower);

        // Step 2: borrower changes mind, signs auth_false{nonce=0} (nonce still 0)
        Authorization memory authFalse = Authorization({
            authorizer: borrower,
            authorized: lender,
            isAuthorized: false,
            nonce: 0,
            deadline: block.timestamp + 2 days
        });
        Signature memory sigFalse = signAuthorization(authFalse, borrower);

        // Step 3: attacker front-runs by submitting auth_true
        address attacker = makeAddr("attacker");
        vm.prank(attacker);
        ecrecoverAuthorizer.setIsAuthorized(authTrue, sigTrue);

        // Step 4: borrower's revocation now reverts with InvalidNonce
        vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector);
        ecrecoverAuthorizer.setIsAuthorized(authFalse, sigFalse);

        // Step 5: assert bad final state
        assertEq(midnight.isAuthorized(borrower, lender), true);  // WRONG: should be false
        assertEq(ecrecoverAuthorizer.nonce(borrower), 1);
    }
}
```

Expected assertions: `InvalidNonce` revert on `auth_false`, and `isAuthorized[borrower][lender] == true` despite borrower's intent to revoke. [7](#0-6)

```text
    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
```
