### Title
Fee-on-Transfer Collateral Token Inflates Stored Collateral, Enabling Uncollateralized Borrowing - (File: src/Midnight.sol)

### Summary

`supplyCollateral` writes `_position.collateral[collateralIndex]` and sets the `collateralBitmap` bit **before** calling `SafeTransferLib.safeTransferFrom`, and `safeTransferFrom` only validates the boolean return value — it never checks the actual balance received. A 100%-fee-on-transfer token returns `true` from `transferFrom` while delivering zero tokens, leaving the protocol with phantom collateral in storage. `isHealthy` then computes `maxDebt` from that phantom stored value, allowing the attacker to borrow real loan tokens against zero actual collateral.

### Finding Description

**Code path — `supplyCollateral` (`src/Midnight.sol` lines 531–545):**

```
_position.collateral[collateralIndex] = toUint128(oldCollateral + assets);  // line 533 — state written
if (oldCollateral == 0 && assets > 0) {
    _position.collateralBitmap = _position.collateralBitmap.setBit(collateralIndex); // line 536 — bitmap set
}
// ... emit ...
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets); // line 545 — transfer last
```

**`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34):**

```
(bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, ...));
// only checks: !success → revert, returndata == false → revert
// NO balance-delta check
```

A token that deducts 100% as a fee returns `true` and emits a `Transfer` event for `assets` but moves 0 tokens to `address(this)`. Both checks pass; the phantom collateral persists.

**`isHealthy` (`src/Midnight.sol` lines 944–959):**

```
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    uint256 price = IOracle(collateralParam.oracle).price();
    maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                                      .mulDivDown(collateralParam.lltv, WAD);
    _collateralBitmap = _collateralBitmap.clearBit(i);
}
return maxDebt >= debt;
```

`_position.collateral[i]` is the phantom value. With a valid oracle price and any allowed LLTV, `maxDebt` is positive, so `isHealthy` returns `true`.

**`take` health gate (`src/Midnight.sol` line 476):**

```
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```

This is the only post-borrow health check for the seller. It passes because `isHealthy` uses phantom collateral.

**`touchMarket` (`src/Midnight.sol` lines 755–791):** Permissionless; validates only LLTV tier, maxLif, sort order, and count. No token whitelist or fee-on-transfer guard exists.

**Exploit flow:**
1. Attacker deploys `FeeToken` — `transferFrom` returns `true`, transfers 0 tokens.
2. Attacker calls `touchMarket(market)` with `FeeToken` as the sole collateral token and a valid LLTV/maxLif pair.
3. A victim lender places a buy offer (supplies loan tokens) in this market.
4. Attacker calls `supplyCollateral(market, 0, 1e18, attacker)`: `_position.collateral[0]` becomes `1e18`, bit 0 of `collateralBitmap` is set, `FeeToken.transferFrom` returns `true`, 0 tokens arrive.
5. Attacker takes the lender's buy offer: `sellerPos.debt += units`, `isHealthy` returns `true` (phantom collateral covers debt), borrow succeeds.
6. Attacker walks away with loan tokens; position has `debt > 0`, `collateral[0] = 1e18` in storage, `token.balanceOf(midnight) = 0`.

### Impact Explanation

The attacker holds real loan tokens with zero actual collateral backing. The position is immediately insolvent: `token.balanceOf(midnight)` for the collateral is 0, so liquidation seizes nothing. The full debt becomes bad debt, socialised across all lenders in the market via the `lossFactor` mechanism. This is pure bad debt creation with no collateral.

### Likelihood Explanation

**Preconditions:**
- Attacker must deploy a fee-on-transfer token (trivial, no privilege required).
- Attacker must create a market with that token (permissionless via `touchMarket`).
- At least one lender must place a buy offer in that market.

The third precondition is the practical barrier: a sophisticated lender would inspect the collateral token. However, the token can be disguised (e.g., upgradeable proxy that activates the fee after market creation and initial liquidity is deposited), or the attacker can seed initial liquidity themselves to attract other lenders. The vulnerability is repeatable and requires no privileged access.

### Recommendation

Add a balance-before/after check in `supplyCollateral` to enforce that the protocol actually received `assets` tokens:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
require(IERC20(collateralToken).balanceOf(address(this)) - balanceBefore == assets, InsufficientCollateralReceived());
```

Alternatively, document that fee-on-transfer tokens are explicitly unsupported as collateral and enforce this at the `touchMarket` level (though on-chain enforcement is not straightforward without a token registry).
