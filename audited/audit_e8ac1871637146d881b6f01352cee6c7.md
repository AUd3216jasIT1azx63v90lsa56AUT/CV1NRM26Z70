### Title
Settlement Fee Accounting Desync via Fee-on-Transfer Loan Token in `take()` - (File: src/Midnight.sol)

### Summary
In `take()`, `claimableSettlementFee[offer.market.loanToken]` is incremented by `buyerAssets - sellerAssets` at line 418 before the corresponding `safeTransferFrom` at line 455. When `loanToken` is a fee-on-transfer token, the contract receives fewer tokens than the recorded amount, permanently desynchronizing `claimableSettlementFee` from the actual token balance. The fee claimer will be unable to claim the full recorded settlement fee.

### Finding Description
**Exact code path:**

`src/Midnight.sol` line 418 executes unconditionally before the transfer:
```solidity
claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;
``` [1](#0-0) 

Then at line 455, the settlement fee is pulled from `payer`:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
``` [2](#0-1) 

**Root cause:** The accounting write precedes the transfer with no balance-delta check. For a standard ERC20, the two are equivalent. For a fee-on-transfer token, the contract receives `(buyerAssets - sellerAssets) * (1 - fee_rate)` but records the full `buyerAssets - sellerAssets` in `claimableSettlementFee`.

**Attacker-controlled inputs:**
- Market creator (permissionless) creates a market with a fee-on-transfer token as `loanToken`. `touchMarket()` imposes no restriction on token type. [3](#0-2) 
- Any maker creates a ratified offer (e.g., via `EcrecoverRatifier`) for that market. [4](#0-3) 
- Any taker calls `take()` with valid parameters.

**Exploit flow:**
1. Market with fee-on-transfer `loanToken` (e.g., 1% transfer fee) is created permissionlessly.
2. Taker calls `take()` resulting in `buyerAssets - sellerAssets = 100`.
3. Line 418: `claimableSettlementFee[loanToken] += 100`.
4. Line 455: `safeTransferFrom(loanToken, payer, address(this), 100)` — token delivers only 99 to the contract.
5. `claimableSettlementFee[loanToken]` records 100; actual balance increased by 99. Gap = 1 per trade, accumulates indefinitely.
6. Fee claimer calls `claimSettlementFee(loanToken, 100, receiver)` — `safeTransfer` reverts because the contract lacks sufficient balance. [5](#0-4) 

**Why existing checks fail:**
The Certora `Solvency.spec` `tokenBalanceCorrect` invariant and `pendingFeeReceiptZero` invariant are proven only under the explicit assumption: *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits."* [6](#0-5) 

The formal verification is therefore unsound for fee-on-transfer tokens. There is no on-chain guard in `take()` or `touchMarket()` that rejects such tokens. The protocol's own `live_context.json` lists "fee-on-transfer" under `external_calls` core invariants as something that "should be tested if not explicitly excluded," and "token charges fee" as a recommended fuzz axis — confirming no explicit exclusion exists. [7](#0-6) 

### Impact Explanation
`claimableSettlementFee[loanToken]` permanently exceeds the contract's actual token balance attributable to settlement fees. The fee claimer cannot claim the full recorded settlement fee; any attempt to claim the full amount reverts. The shortfall accumulates with every `take()` call in the affected market. This violates the core solvency invariant: *"contract token balances must cover withdrawable assets, collateral claims, credit redemptions, and accrued fees."* [8](#0-7) 

### Likelihood Explanation
- Market creation is permissionless and accepts arbitrary loan tokens; no admin action is required.
- Fee-on-transfer tokens (e.g., tokens with configurable transfer fees) are a well-known token class.
- The condition is triggered by any ordinary `take()` call in such a market — no special timing, reentrancy, or privileged access needed.
- The desync is permanent and cumulative: every subsequent `take()` in the same market widens the gap.
- The `live_context.json` explicitly identifies this as a priority test axis, confirming the protocol does not document fee-on-transfer tokens as unsupported.

### Recommendation
After the `safeTransferFrom` call for the settlement fee, measure the actual balance delta and use that to update `claimableSettlementFee` instead of the nominal `buyerAssets - sellerAssets`. Concretely:

```solidity
uint256 balanceBefore = IERC20(offer.market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);
uint256 actualFeeReceived = IERC20(offer.market.loanToken).balanceOf(address(this)) - balanceBefore;
claimableSettlementFee[offer.market.loanToken] += actualFeeReceived;
```

This requires moving the `claimableSettlementFee` increment to after the transfer. Alternatively, explicitly document and enforce (via `touchMarket`) that fee-on-transfer tokens are not supported as loan tokens.

### Proof of Concept
**Foundry unit test plan:**

```solidity
// Deploy a fee-on-transfer ERC20 (1% fee on every transfer)
FeeOnTransferToken fot = new FeeOnTransferToken(1); // 1% fee

// Create market with fot as loanToken (permissionless)
Market memory market = ...; market.loanToken = address(fot);
midnight.touchMarket(market);

// Setup: lender has fot, borrower has collateral, ratified offer exists
// Record balances before take
uint256 balanceBefore = fot.balanceOf(address(midnight));
uint256 claimableBefore = midnight.claimableSettlementFee(address(fot));

// Taker executes take()
(uint256 buyerAssets, uint256 sellerAssets) = midnight.take(offer, ratifierData, units, taker, ...);
uint256 nominalFee = buyerAssets - sellerAssets;

uint256 balanceAfter = fot.balanceOf(address(midnight));
uint256 claimableAfter = midnight.claimableSettlementFee(address(fot));

// Assert: claimable increased by full nominal fee
assertEq(claimableAfter - claimableBefore, nominalFee);

// Assert: actual balance increased by less (fee-on-transfer took 1%)
uint256 actualIncrease = balanceAfter - balanceBefore;
assertLt(actualIncrease, nominalFee); // FAILS invariant: claimable > actual balance increase

// Assert: fee claimer cannot claim the full recorded fee
vm.prank(feeClaimer);
vm.expectRevert(); // safeTransfer reverts — insufficient balance
midnight.claimSettlementFee(address(fot), claimableAfter - claimableBefore, feeClaimer);
```

Expected: the third assertion demonstrates the fee claimer is permanently unable to claim the full recorded settlement fee, confirming the accounting desync.

        require(msg.sender == MIDNIGHT, NotMidnight());
