Looking at the exact code paths in `EcrecoverAuthorizer.sol` and `Midnight.sol`:

### Title
Signed `Authorization` replay bypasses direct revocation via `Midnight.setIsAuthorized` — (`src/periphery/EcrecoverAuthorizer.sol`)

### Summary
`Midnight.setIsAuthorized` updates `isAuthorized[onBehalf][authorized]` but never touches `EcrecoverAuthorizer.nonce`. If a victim signed an `Authorization{isAuthorized=true}` for an attacker and then revokes via `Midnight.setIsAuthorized` directly, the `EcrecoverAuthorizer` nonce remains unchanged, allowing the attacker to submit the old signed authorization and re-grant themselves access. The revocation invariant — that a revocation is permanent and not replayable — is broken.

### Finding Description

**Exact code path:**

`EcrecoverAuthorizer.setIsAuthorized` (line 26) enforces replay protection solely through its own `nonce` mapping: [1](#0-0) 

`Midnight.setIsAuthorized` (lines 731–735) only writes to `isAuthorized` and emits an event — it has no interaction with `EcrecoverAuthorizer.nonce`: [2](#0-1) 

**Preconditions:**
1. Victim calls `Midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim)` — required so `EcrecoverAuthorizer` can act on victim's behalf.
2. Victim signs `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=0, deadline=T+future}` and the attacker obtains this signature (e.g., from a broadcast, mempool, or prior off-chain exchange).

**Exploit flow:**
1. Victim calls `Midnight.setIsAuthorized(attacker, false, victim)` to revoke. `isAuthorized[victim][attacker]` becomes `false`. `EcrecoverAuthorizer.nonce[victim]` remains `0`.
2. Attacker calls `EcrecoverAuthorizer.setIsAuthorized(auth, sig)` with the old signed struct:
   - Line 25: deadline still in the future → passes.
   - Line 26: `auth.nonce (0) == nonce[victim] (0)` → passes; nonce incremented to 1.
   - Lines 33–36: `signer == auth.authorizer` (victim signed it) → passes.
   - Line 46–47: calls `Midnight.setIsAuthorized(attacker, true, victim)`; `isAuthorized[victim][ecrecoverAuthorizer]` is still `true` → passes.
3. Result: `isAuthorized[victim][attacker]` is `true` again. [3](#0-2) 

**Why existing checks fail:** The nonce check on line 26 only prevents reuse of a signature that was already submitted through `EcrecoverAuthorizer`. A direct call to `Midnight.setIsAuthorized` is invisible to the `EcrecoverAuthorizer` nonce, so any signed-but-not-yet-submitted authorization remains valid after a direct revocation.

### Impact Explanation
An attacker who holds a signed `Authorization{isAuthorized=true}` can permanently re-authorize themselves on the victim's account even after the victim explicitly revokes. This grants the attacker the ability to act as the victim in any context that checks `isAuthorized` — including `take` (acting as taker), `setConsumed` (cancelling victim's offers), `withdrawCollateral`, and further `setIsAuthorized` calls — constituting a full account takeover for all protocol actions. [4](#0-3) 

### Likelihood Explanation
**Preconditions:** Victim must have (a) authorized `EcrecoverAuthorizer` in `Midnight`, and (b) previously signed an `Authorization{isAuthorized=true}` for the attacker that has not yet been submitted. Both are normal usage patterns. The attacker only needs to hold the signed message and wait for the victim to attempt a direct revocation. The attack is repeatable as long as the deadline has not expired and the victim does not revoke `EcrecoverAuthorizer` itself from `Midnight`.

### Recommendation
Add a nonce-increment function to `EcrecoverAuthorizer` callable by the authorizer (or anyone `Midnight`-authorized by them) that invalidates all pending signatures:

```solidity
function invalidateNonce(address authorizer) external {
    require(
        authorizer == msg.sender || IMidnight(MIDNIGHT).isAuthorized(authorizer, msg.sender),
        Unauthorized()
    );
    nonce[authorizer]++;
}
```

Additionally, document clearly that revocation of a previously signed-but-unsubmitted authorization **must** go through `EcrecoverAuthorizer` (either by submitting a signed `isAuthorized=false` message or calling `invalidateNonce`), not through `Midnight.setIsAuthorized` directly.

### Proof of Concept

```solidity
function testReplayAfterDirectRevocation() public {
    // Step 1: victim authorizes EcrecoverAuthorizer in Midnight
    vm.prank(borrower); // borrower = victim
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, borrower);

    // Step 2: victim signs Authorization{isAuthorized=true} for attacker (nonce=0)
    Authorization memory auth = Authorization({
        authorizer: borrower,
        authorized: lender, // lender = attacker
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory sig = signAuthorization(auth, borrower);

    // Step 3: victim revokes directly via Midnight (does NOT increment EcrecoverAuthorizer nonce)
    vm.prank(borrower);
    midnight.setIsAuthorized(lender, false, borrower);
    assertEq(midnight.isAuthorized(borrower, lender), false);
    assertEq(ecrecoverAuthorizer.nonce(borrower), 0); // nonce still 0

    // Step 4: attacker replays the old signature — expect revert, but it SUCCEEDS
    // vm.expectRevert(IEcrecoverAuthorizer.InvalidNonce.selector); // this does NOT revert
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // Assertion: revocation was bypassed
    assertEq(midnight.isAuthorized(borrower, lender), true); // BUG: should be false
    assertEq(ecrecoverAuthorizer.nonce(borrower), 1);
}
```

Expected: the `ecrecoverAuthorizer.setIsAuthorized` call reverts with `InvalidNonce`. Actual: it succeeds and re-grants authorization, demonstrating the replay.
