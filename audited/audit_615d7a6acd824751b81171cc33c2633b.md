Audit Report

## Title
Authorized address can de-ratify maker's offer root, freezing all takes - (`src/ratifiers/SetterRatifier.sol`)

## Summary
`SetterRatifier.setIsRootRatified` applies an identical `isAuthorized` guard for both ratifying (`true`) and de-ratifying (`false`) a Merkle root. Any address that a maker has authorized in Midnight â€” for any reason â€” can call `setIsRootRatified(maker, root, false)`, setting `isRootRatified[maker][root] = false` and causing every subsequent `take` on that maker's offer tree to revert with `NotRatified`. The maker cannot atomically revoke the authorization and re-ratify without a race condition.

## Finding Description
**Root cause:** `SetterRatifier.sol` lines 24â€“26 apply a single, undifferentiated guard:

```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
```

The `isAuthorized` mapping in Midnight (`IMidnight.sol` line 123) is a flat boolean with no action-scoping. Granting authorization for any purpose â€” e.g., delegating take authority to a keeper â€” simultaneously grants the power to write `false` to `isRootRatified`.

**Downstream enforcement:** `SetterRatifier.sol` line 35:
```solidity
require(isRootRatified[offer.maker][root], NotRatified());
```
hard-reverts every `take` call if the root is de-ratified.

**Exploit flow:**
1. `makerB` calls `midnight.setIsAuthorized(makerA, true, makerB)` for any legitimate reason (e.g., keeper delegation).
2. `makerA` calls `setterRatifier.setIsRootRatified(makerB, root, false)`.
3. Guard passes: `IMidnight(MIDNIGHT).isAuthorized(makerB, makerA) == true`.
4. `isRootRatified[makerB][root]` is set to `false`.
5. All `midnight.take(offer, ...)` calls where `offer.maker == makerB` and ratifier data encodes `root` revert `NotRatified`.

**Why existing checks fail:** The `Unauthorized()` revert only fires if `msg.sender` is neither the maker nor an authorized address. It does not restrict what value an authorized address may write. There is no check on `newIsRootRatified`.

The test `testIsRatifiedAuthorizedSetterCanRatifyOnBehalf` (`test/SetterRatifierTest.sol` lines 48â€“61) confirms the authorized-setter path is fully reachable; the same path is equally reachable for `false`.

## Impact Explanation
`makerB`'s entire offer tree (all offers whose Merkle root is `root`) is permanently frozen for takes. Lenders/borrowers whose positions depend on those offers being fillable cannot execute. `makerB` can re-ratify, but `makerA` can immediately de-ratify again in the same block, making recovery impossible while the authorization persists. This constitutes a permanent, repeatable freeze of user state â€” an in-scope impact per `RESEARCHER.md` ("Permanent lock, freeze, or unrecoverable corruption of user/project state").

## Likelihood Explanation
Precondition: `makerB` must have previously called `setIsAuthorized(makerA, true, makerB)`. This is a normal, expected protocol operation (keeper, co-signer, automated taker). The authorization is coarse-grained, so any such grant is sufficient. The attack is a single external call, requires no funds, and is immediately repeatable. The attacker need not be the maker; any address that has been granted authorization qualifies.

## Recommendation
Restrict de-ratification to the maker only. Authorized addresses should only be permitted to write `true`:

```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    if (newIsRootRatified) {
        require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    } else {
        require(maker == msg.sender, Unauthorized());
    }
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
```

This preserves the delegation-of-ratification use case while ensuring only the maker can revoke their own root.

## Proof of Concept
Extend `SetterRatifierTest.sol` with:

```solidity
function testAuthorizedAddressCanDeRatify() public {
    Offer memory offer = makeOffer(lender);
    bytes32 _root = HashLib.hashOffer(offer);

    // lender ratifies root
    vm.prank(lender);
    setterRatifier.setIsRootRatified(lender, _root, true);
    assertTrue(setterRatifier.isRootRatified(lender, _root));

    // lender authorizes borrower for any purpose
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, true, lender);

    // borrower de-ratifies lender's root
    vm.prank(borrower);
    setterRatifier.setIsRootRatified(lender, _root, false);
    assertFalse(setterRatifier.isRootRatified(lender, _root));

    // take now reverts NotRatified
    vm.prank(address(midnight));
    vm.expectRevert(ISetterRatifier.NotRatified.selector);
    setterRatifier.isRatified(offer, abi.encode(_root, 0, new bytes32[](0)));
}
```

Expected: test passes, confirming the de-ratification path is fully reachable by an authorized-but-non-maker address.
