Audit Report

## Title
Dust-collateral rounding in `isHealthy` and `liquidate` enables liquidation of economically healthy positions - (File: src/Midnight.sol)

## Summary
Chained `mulDivDown` calls in `isHealthy` and `liquidate` can round `maxDebt` to 0 when a position holds exactly 1 wei of collateral. Because `withdrawCollateral` only clears the bitmap bit when collateral reaches exactly 0, a borrower can deliberately leave 1 wei, pass the health check at a high oracle price, and then be liquidated after a routine price drop even though the position remains overcollateralized in real arithmetic.

## Finding Description

**Constants confirmed** in `src/libraries/ConstantsLib.sol`: [1](#0-0) 

**Root cause — chained floor division in `isHealthy`:** [2](#0-1) 

**Same path in `liquidate`:** [3](#0-2) 

Both compute `collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)`. With `collateral[i] = 1` wei and `price = 1.5e36`:
- Step 1: `mulDivDown(1, 1.5e36, 1e36) = floor(1.5) = 1`
- Step 2: `mulDivDown(1, 0.77e18, 1e18) = floor(0.77) = 0`
- `maxDebt = 0`

In real arithmetic: `1 × 1.5e36 / 1e36 × 0.77 = 1.155 > 1` — the position is overcollateralized.

**Bitmap not cleared on partial withdrawal:** [4](#0-3) 

The bit is only cleared when `newCollateral == 0`, so leaving 1 wei keeps the collateral tracked and the health check active.

**`withdrawCollateral` health check is point-in-time only:** [5](#0-4) 

At withdrawal time with `price = 2e36`: `mulDivDown(1, 2e36, 1e36) = 2`; `mulDivDown(2, 0.77e18, 1e18) = 1`. `maxDebt = 1 >= debt = 1` → passes. No minimum collateral floor exists.

**`liquidate` liquidatability check:** [6](#0-5) 

After the price drops to 1.5e36, `originalDebt (1) > maxDebt (0)` passes, and liquidation proceeds.

**Exploit path (LLTV_2 = 0.77e18):**
1. Borrower supplies collateral, borrows 1 unit of debt, then calls `withdrawCollateral` with `assets = initial_collateral − 1`, leaving 1 wei.
2. At withdrawal time with `price = 2e36`: `maxDebt = 1 >= debt = 1` → `isHealthy` returns true → withdrawal succeeds.
3. Oracle price drops to 1.5e36 (~25% drop): `maxDebt = 0`.
4. Liquidator calls `liquidate`. `originalDebt (1) > maxDebt (0)` → `NotLiquidatable` check passes.
5. Liquidator seizes 1 wei of collateral worth 1.5 loan-token units by repaying 1 unit of debt.

**Why existing checks fail:** `withdrawCollateral` calls `isHealthy` only at withdrawal time; there is no minimum-collateral floor, no dust guard, and no protection against subsequent price drops causing rounding collapse. The `liquidate` path recomputes `maxDebt` with the identical two-step `mulDivDown`, so the same rounding error recurs unconditionally. The vulnerability applies to all LLTV tiers except `LLTV_8 = 1e18` (where `mulDivDown(x, 1e18, 1e18) = x`). [7](#0-6) 

## Impact Explanation
A liquidator seizes 1 wei of collateral worth 1.5 loan-token units by repaying only 1 unit of debt, netting ~0.5 loan-token units of profit at the borrower's expense. The core protocol invariant — that positions with `collateral_value × lltv > debt` in real arithmetic cannot be liquidated — is concretely violated. This constitutes direct theft of user assets caused by a protocol accounting bug, not an external dependency.

## Likelihood Explanation
No privileged access is required. Any borrower can reach the dust state by calling `withdrawCollateral` with `assets = current_collateral − 1`. The required oracle price at withdrawal (`≥ ceil(WAD / lltv) × ORACLE_PRICE_SCALE`, i.e., `≥ 2e36` for LLTV_2) is a normal market condition. The subsequent price drop into the rounding-collapse range (~25–33% depending on LLTV tier) is a routine market event. The dust state can also arise unintentionally from partial withdrawals. The liquidator is any unprivileged external account. The attack is repeatable across all markets using LLTV tiers 0–7.

## Recommendation
**Option 1 — Single-step formula:** Replace the two chained `mulDivDown` calls with a single combined division:
```solidity
maxDebt += _position.collateral[i].mulDivDown(price * collateralParam.lltv, ORACLE_PRICE_SCALE * WAD);
```
This eliminates the intermediate rounding step. Note: verify no overflow risk for `price * lltv` given `price ≤ ORACLE_PRICE_SCALE` and `lltv ≤ WAD`.

**Option 2 — Minimum collateral floor:** In `withdrawCollateral`, require that after withdrawal, if the borrower has debt, the remaining collateral at any activated index is either 0 or above a protocol-defined dust threshold sufficient to avoid rounding collapse.

**Option 3 — Rounding-aware health check:** In `isHealthy`, use `mulDivUp` for the collateral-value computation (conservative for the borrower) and `mulDivDown` for debt, ensuring rounding always favors the borrower's health.

## Proof of Concept
Minimal Foundry test (add to `LiquidationTest.sol`):

```solidity
function testDustCollateralRoundingLiquidation() public {
    // Setup: use LLTV_2 = 0.77e18 market (already configured in setUp)
    uint256 lltv = 0.77e18;
    // Price at which 1 wei collateral gives maxDebt = 1: need price >= 2e36
    Oracle(market.collateralParams[0].oracle).setPrice(2e36);

    // Supply enough collateral to borrow 1 unit, then withdraw all but 1 wei
    uint256 initialCollateral = 2; // 2 wei collateral
    deal(address(collateralToken1), borrower, initialCollateral);
    vm.startPrank(borrower);
    collateralToken1.approve(address(midnight), initialCollateral);
    midnight.supplyCollateral(market, 0, initialCollateral, borrower);
    vm.stopPrank();

    // Borrow 1 unit
    setupMarket(market, 1);

    // Withdraw all but 1 wei
    vm.prank(borrower);
    midnight.withdrawCollateral(market, 0, initialCollateral - 1, borrower, borrower);

    // Verify position is healthy at price = 2e36
    assertTrue(midnight.isHealthy(market, id, borrower), "should be healthy at price 2e36");

    // Drop price to 1.5e36 (~25% drop)
    Oracle(market.collateralParams[0].oracle).setPrice(1.5e36);

    // Verify real arithmetic: collateral_value * lltv = 1.5 * 0.77 = 1.155 > 1 (debt)
    // But on-chain maxDebt = 0 due to rounding

    // Liquidation succeeds despite position being overcollateralized in real arithmetic
    deal(address(loanToken), address(this), 1);
    midnight.liquidate(market, 0, 0, 1, borrower, false, address(this), address(0), "");

    // Borrower lost their collateral despite being economically healthy
    assertEq(midnight.debtOf(id, borrower), 0);
    assertEq(midnight.collateral(id, borrower, 0), 0);
}
```
