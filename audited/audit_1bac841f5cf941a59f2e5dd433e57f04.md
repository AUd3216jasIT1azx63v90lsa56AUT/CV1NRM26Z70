Audit Report

## Title
FOT loan token causes `supplyCollateralAndSellWithAssetsTarget` to always revert when `referralFeeAssets > 0` - (File: src/periphery/MidnightBundles.sol)

## Summary
`supplyCollateralAndSellWithAssetsTarget` accumulates `filledSellerAssets` from the return value of `MIDNIGHT.take()`, which reflects the *intended* transfer amount. When the loan token is fee-on-transfer (FOT), the bundler receives `sellerAssets - F` per take, but the return-value accumulator reaches `targetFilledSellerAssets` exactly. The subsequent outbound transfers then attempt to send `referralFeeAssets + targetSellerAssets = targetFilledSellerAssets` while the bundler only holds `targetFilledSellerAssets - F_total`, causing the final `safeTransfer` to always revert. The function is completely unusable for FOT loan tokens whenever `referralFeePct > 0`.

## Finding Description

**Root cause:** The bundler trusts the return value of `MIDNIGHT.take()` as a proxy for its actual token balance increase, but with FOT tokens these diverge.

**Exact code path:**

In `Midnight.sol`, when `offer.buy == true` (taker is seller), `take()` resolves:
```
receiver = receiverIfTakerIsSeller  // = address(this), the bundler
```
and executes at line 456:
```solidity
SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);
``` [1](#0-0) 

The return value is `(buyerAssets, sellerAssets)` at line 478 — the *requested* amount, not the amount actually received. [2](#0-1) 

Back in `supplyCollateralAndSellWithAssetsTarget`, the bundler passes `address(this)` as `receiverIfTakerIsSeller` and accumulates the return value:
```solidity
try IMidnight(MIDNIGHT).take(
    takes[i].offer, takes[i].ratifierData, unitsToTake, taker, address(this), address(0), ""
) returns (uint256, uint256 resSellerAssets) {
    filledSellerAssets += resSellerAssets;
}
``` [3](#0-2) 

With a FOT token, the bundler's actual balance after the loop is `targetFilledSellerAssets - F_total`. The check at line 303 passes because it compares the return-value accumulator, not the actual balance:
```solidity
require(filledSellerAssets == targetFilledSellerAssets, OutOfOffers());
``` [4](#0-3) 

Then the two outbound transfers execute:
```solidity
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets); // line 306
SafeTransferLib.safeTransfer(loanToken, receiver, targetSellerAssets);                                       // line 307
``` [5](#0-4) 

After line 306 succeeds (assuming `F_total < targetSellerAssets`), the bundler holds `targetSellerAssets - F_total`. Line 307 requests `targetSellerAssets`, which exceeds the available balance by exactly `F_total` → `safeTransfer` reverts.

**Why existing checks fail:**
- `require(referralFeePct < WAD)` only prevents 100% fee — does not address FOT.
- `require(filledSellerAssets == targetFilledSellerAssets)` checks the return-value accumulator, not the actual balance delta.
- No balance-before/after check exists anywhere in the function. [6](#0-5) 

## Impact Explanation
The function always reverts when the loan token is FOT and `referralFeePct > 0`. Because the revert is atomic, no partial state persists: the taker's debt is not created, the receiver receives nothing, and the referralFeeRecipient receives nothing. The NatSpec invariant at line 249 — *"Total loan assets received by the receiver is `targetSellerAssets`"* — is violated: the receiver receives zero. [7](#0-6) 

This is a complete, deterministic denial of service for `supplyCollateralAndSellWithAssetsTarget` on any market whose loan token charges a transfer fee, whenever a referral fee is requested.

## Likelihood Explanation
**Preconditions:**
1. The market's `loanToken` is a FOT token. Market creation is permissionless, so any user can create such a market.
2. `referralFeePct > 0` — any non-zero value supplied by the caller.

Both conditions are reachable by an unprivileged user with no special keys or governance access. The bug is deterministic and repeatable: every call under these preconditions reverts. No oracle manipulation or admin action is required.

## Recommendation
Replace the return-value accumulator pattern with a balance-before/balance-after check to measure actual tokens received:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
// ... take loop ...
uint256 actualReceived = IERC20(loanToken).balanceOf(address(this)) - balanceBefore;
require(actualReceived >= targetFilledSellerAssets, OutOfOffers());
// distribute from actualReceived, not from filledSellerAssets
```

Alternatively, explicitly document and enforce that FOT tokens are not supported as loan tokens for this bundler function, and add a guard or note in the NatSpec.

## Proof of Concept
**Minimal fork test plan:**
1. Deploy a mock ERC20 with a 1% transfer fee (FOT token).
2. Create a Midnight market with the FOT token as `loanToken`.
3. Create a buy offer on the market.
4. Call `supplyCollateralAndSellWithAssetsTarget` with `referralFeePct = 0.01e18` (1%), valid `takes[]`, and sufficient collateral.
5. Observe: the call reverts at line 307 (`safeTransfer` to `receiver`) because the bundler's balance is `targetSellerAssets - fee` but the transfer requests `targetSellerAssets`.
6. Confirm: calling the same function with `referralFeePct = 0` succeeds (the final transfer is `filledSellerAssets - 0 = actualReceived` which the bundler holds).
