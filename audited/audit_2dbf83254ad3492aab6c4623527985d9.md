### Title
Permit2 `pullToken` path reverts on front-run nonce consumption while ERC2612 path silently tolerates it — (`src/periphery/MidnightBundles.sol`)

### Summary
`MidnightBundles.pullToken` wraps the ERC2612 `permit` call in a `try/catch` with an explicit comment acknowledging that a third party may have already consumed the permit, then falls through to `safeTransferFrom`. The Permit2 path makes no equivalent accommodation: `IPermit2.permitTransferFrom` is called bare with no try/catch and no fallback transfer, so any front-run that consumes the nonce causes the entire bundler transaction to revert. This creates a demonstrable, undocumented behavioral asymmetry between the two permit paths that is untested.

### Finding Description
In `MidnightBundles.pullToken` (lines 378–398):

**ERC2612 path (lines 379–384):**
```solidity
// Tolerate revert: a third party may have already consumed the permit.
try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
```
The permit call is swallowed on failure; `safeTransferFrom` still executes and succeeds if the user holds a pre-existing ERC20 allowance to the bundler.

**Permit2 path (lines 385–394):**
```solidity
IPermit2(PERMIT2)
    .permitTransferFrom(
        IPermit2.PermitTransferFrom(IPermit2.TokenPermissions(token, amount), nonce, deadline),
        IPermit2.SignatureTransferDetails(address(this), amount),
        from,
        signature
    );
```
No try/catch, no fallback. `IPermit2.permitTransferFrom` is atomic: it validates the signature, marks the nonce as used, and transfers tokens in one call. If the nonce is already consumed, it reverts with `InvalidNonce` and the entire bundler transaction reverts.

**Front-run exploit path:**
1. User submits a bundler call (e.g., `buyWithUnitsTargetAndWithdrawCollateral`) with `PermitKind.Permit2` and a valid signature visible in the mempool.
2. Attacker extracts `(nonce, deadline, signature)` from calldata and calls `Permit2.permitTransferFrom` directly with `to = bundler` (the signature binds the `to` address, so the attacker cannot redirect tokens to themselves).
3. Permit2 transfers the user's tokens to the bundler and marks the nonce consumed — in the attacker's transaction.
4. The user's bundler transaction executes, reaches `pullToken`, calls `permitTransferFrom` with the same nonce → reverts `InvalidNonce`.
5. The bundler transaction reverts entirely. The tokens transferred in step 3 remain in the bundler (no sweep/recovery function exists in `MidnightBundles`).

The attacker gains nothing financially but can grief any user relying on `PermitKind.Permit2` and, in the worst case, lock the user's tokens in the bundler contract.

No existing check in `MidnightBundles.sol` or `IPermit2.sol` prevents or documents this asymmetry for the Permit2 path.

### Impact Explanation
Any bundler call using `PermitKind.Permit2` can be front-run to force a revert. If the attacker's front-run transaction succeeds first, the user's tokens are transferred to the bundler but the bundler transaction reverts, leaving tokens stranded with no recovery path. The ERC2612 path is immune to the same attack because of its explicit try/catch + fallback `safeTransferFrom`. The asymmetry is undocumented and untested.

### Likelihood Explanation
Preconditions: user submits a bundler call with `PermitKind.Permit2` through a public mempool. The attacker only needs to read the calldata and replay the Permit2 call with the same parameters before the bundler call is mined. This is repeatable on every such transaction. The attacker pays gas but incurs no other cost; the attack is feasible for any griefing-motivated actor (e.g., a competing lender or taker).

### Recommendation
Wrap the Permit2 `permitTransferFrom` call in a try/catch and, on failure, fall back to `SafeTransferLib.safeTransferFrom` (requiring the user to have pre-approved the bundler via a standard ERC20 `approve` or Permit2's allowance path). Alternatively, document explicitly that `PermitKind.Permit2` does **not** degrade gracefully on nonce front-running, unlike `PermitKind.ERC2612`, so callers understand they must use a fresh nonce or accept revert risk. At minimum, add a natspec comment on the Permit2 branch mirroring the one on line 382.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import "forge-std/Test.sol";
import {MidnightBundles} from "src/periphery/MidnightBundles.sol";
import {IPermit2} from "src/periphery/interfaces/IPermit2.sol";
import {PermitKind, TokenPermit} from "src/periphery/interfaces/IMidnightBundles.sol";

contract Permit2FrontRunTest is Test {
    address constant PERMIT2 = 0x000000000022D473030F116dDEE9F6B43aC78BA3;

    function test_permit2FrontRunCausesRevert_erc2612DoesNot() public {
        // Setup: deploy token with both ERC2612 and Permit2 support,
        // fund user, have user sign a Permit2 signature with nonce=0.

        // Step 1: Attacker calls Permit2.permitTransferFrom directly
        // with to=bundler, same nonce, same signature.
        // Assert: succeeds, nonce 0 is now consumed, tokens in bundler.

        // Step 2: Call bundler function with PermitKind.Permit2 and same nonce.
        // Assert: REVERTS (InvalidNonce from Permit2).
        vm.expectRevert();
        // bundler.buyWithUnitsTargetAndWithdrawCollateral(..., permit2Permit, ...);

        // Step 3: Repeat with PermitKind.ERC2612 where a third party
        // has already consumed the ERC2612 permit nonce.
        // Assert: SUCCEEDS (try/catch swallows permit failure,
        // safeTransferFrom uses pre-existing allowance).
        // bundler.buyWithUnitsTargetAndWithdrawCollateral(..., erc2612Permit, ...);

        // Final assertion: ERC2612 path returns normally,
        // Permit2 path reverted — asymmetry confirmed.
    }
}
```

Expected assertions:
- `PermitKind.Permit2` call reverts after nonce is front-run.
- `PermitKind.ERC2612` call succeeds after permit nonce is front-run (given pre-existing allowance).
