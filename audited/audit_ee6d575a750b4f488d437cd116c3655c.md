### Title
Silent pool initialization failure when `poolImplementation` has no code — (`src/ConfidencePoolFactory.sol`)

### Summary
`ConfidencePoolFactory.setPoolImplementation` validates only against `address(0)`, not against code existence. If `poolImplementation` is set to an address that contains no bytecode, `createPool` silently produces a registered but permanently uninitialized clone: the EIP-1167 minimal proxy's implicit `delegatecall` to the empty implementation returns `(true, "")` per EVM spec, `initialize()` (which returns void) appears to succeed, and the pool is pushed into `_poolsByAgreement` and emits `PoolCreated` — with every state variable at its zero default.

### Finding Description
`setPoolImplementation` guards only against the zero address:

```solidity
function setPoolImplementation(address newPoolImplementation) external onlyOwner {
    if (newPoolImplementation == address(0)) revert ZeroAddress();   // only guard
    poolImplementation = newPoolImplementation;
    ...
}
``` [1](#0-0) 

`createPool` then deploys an EIP-1167 minimal proxy whose entire bytecode is a `delegatecall` trampoline to the hardcoded `poolImplementation` address, and immediately calls `initialize()` on it:

```solidity
pool = Clones.cloneDeterministic(poolImplementation, salt);
IConfidencePool(pool).initialize(...);   // delegatecalls into poolImplementation
``` [2](#0-1) 

The Solidity documentation warns: *"The low-level functions `call`, `delegatecall` and `staticcall` return `true` as their first return value if the account called is non-existent, as part of the design of the EVM."* Because `initialize()` returns `void`, the high-level call to the clone sees a successful call with empty return data and does not revert. The `Initializable._initialized` flag in the clone's storage is never written (no code ran), so the clone is not marked initialized.

### Impact Explanation
Every state variable in the clone remains at its zero default: `stakeToken = address(0)`, `expiry = 0`, `minStake = 0`, `outcomeModerator = address(0)`, `owner = address(0)`, `outcome = UNRESOLVED`. The pool is permanently non-functional:

- `stake()` immediately reverts with `StakingClosed` because `block.timestamp >= expiry` (expiry is 0). Stakers cannot deposit.
- `flagOutcome()` reverts with `NotModerator` because `outcomeModerator` is `address(0)`.
- The clone's bytecode permanently delegates to the original empty address; a later `setPoolImplementation` to a valid contract does not fix existing broken clones.
- Because `_initialized` is never set, anyone can call `initialize()` on the broken clone — but since `poolImplementation` still has no code, the call again silently succeeds without writing any state. If a contract is later deployed at the original empty `poolImplementation` address, a malicious actor could call `initialize()` on the broken clone with attacker-controlled parameters (arbitrary `owner`, `recoveryAddress`, etc.).
- The deterministic salt for that agreement index is consumed; the broken pool occupies a permanent slot in `_poolsByAgreement`.

### Likelihood Explanation
Requires the factory owner to set `poolImplementation` to an address with no code — an EOA, a pre-computed CREATE2 address before deployment, or a contract that was subsequently self-destructed. This is an honest operational mistake (wrong address copy-paste, deployment order error) that the zero-address guard does not catch. Once set, every `createPool` call by any agreement owner (an unprivileged caller relative to the factory) produces a broken pool until the owner corrects the implementation address.

### Recommendation
Add a code-existence check in `setPoolImplementation` (and defensively in `initialize` as well):

```solidity
function setPoolImplementation(address newPoolImplementation) external onlyOwner {
    if (newPoolImplementation == address(0)) revert ZeroAddress();
    if (newPoolImplementation.code.length == 0) revert NoCode();
    ...
}
```

Optionally, add the same guard in `createPool` before `Clones.cloneDeterministic` as defense-in-depth.

### Proof of Concept
1. Factory owner calls `setPoolImplementation(0xDeAdBeEf...)` where the address is an EOA (no code). Only the zero-address check runs; the call succeeds.
2. Agreement owner calls `createPool(agreement, stakeToken, expiry, minStake, recoveryAddress, accounts)`.
3. `Clones.cloneDeterministic(0xDeAdBeEf..., salt)` deploys a minimal proxy clone. The clone has code (the EIP-1167 trampoline); deployment succeeds.
4. `IConfidencePool(pool).initialize(...)` calls the clone. The clone executes `delegatecall` to `0xDeAdBeEf...`. Since that address has no code, the EVM returns `(true, "")`. `initialize()` returns void; the high-level call does not revert.
5. `_poolsByAgreement[agreement].push(pool)` and `emit PoolCreated(...)` execute. `createPool` returns the pool address.
6. The clone's storage: `_initialized = 0`, `stakeToken = address(0)`, `expiry = 0`, `owner = address(0)` — all zero defaults.
7. Any call to `stake()` reverts with `StakingClosed`; the pool is permanently unusable. The agreement's index-0 slot is consumed.
