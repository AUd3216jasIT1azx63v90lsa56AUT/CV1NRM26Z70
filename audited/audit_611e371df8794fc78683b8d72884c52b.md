### Title
Zero-rounding in assets-mode buy offer bypasses `maxAssets` consumed accounting — (`src/Midnight.sol`)

### Summary
When `offer.maxAssets > 0` and `offer.buy == true`, the `consumed` mapping is incremented by `buyerAssets` rather than by `units`. Because `buyerAssets` is computed with `mulDivDown`, a taker can choose `units` small enough that `units * buyerPrice < WAD`, making `buyerAssets = 0`. The consumed counter never advances, the `ConsumedAssets` cap is never reached, and the offer can be taken an unbounded number of times with zero token transfer per call.

### Finding Description
The relevant code path is in `src/Midnight.sol`:

```
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // line 363 – rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group]
        += offer.buy ? buyerAssets : sellerAssets;  // line 368
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [1](#0-0) 

`mulDivDown` returns `(units * buyerPrice) / WAD`. Whenever `units * buyerPrice < WAD` the result is 0. Because `sellerPrice ≤ buyerPrice`, `sellerAssets` is also 0 in the same condition. There is no guard of the form `require(units == 0 || buyerAssets > 0)` anywhere in `take`.

**Exploit flow (buy offer, assets mode):**

1. Maker publishes a buy offer with `offer.buy = true`, `offer.maxAssets = X > 0`, and any tick whose `buyerPrice < WAD` (e.g. `tick` such that `tickToPrice(tick) = 1e15`).
2. Attacker (taker, a different address) calls `take(offer, ..., units = U)` where `U * buyerPrice < WAD` (e.g. `U = 999` with `buyerPrice = 1e15` gives `999e15 < 1e18`).
3. Line 363: `buyerAssets = mulDivDown(999, 1e15, 1e18) = 0`.
4. Line 368: `consumed[maker][group] += 0` — consumed is unchanged.
5. Line 369: `require(unchanged_consumed <= X)` — always passes.
6. Lines 455–456: `safeTransferFrom(..., 0)` — zero-value transfers succeed on standard ERC-20s.
7. Line 476: `isHealthy` check passes provided the taker has sufficient collateral.
8. The call succeeds. Repeat indefinitely; `consumed` never reaches `maxAssets`.

The existing `testPriceZeroNoSettlementFeeSell` test already demonstrates that `buyerAssets = 0` and `sellerAssets = 0` with `units > 0` is a reachable, non-reverting outcome (for sell offers in units mode). [2](#0-1) 

The Certora `takeConsumedDelta` rule that asserts `consumed` advances by `units` is explicitly scoped to `offer.maxAssets == 0` (units mode) and does not cover this path. [3](#0-2) 

### Impact Explanation
The `maxAssets` cap — the maker's intended upper bound on total assets consumed — is never reached. An attacker can take the offer an unlimited number of times, each time advancing the maker's credit and the taker's debt by `units` with zero token transfer. The maker accumulates credit far beyond `maxAssets`; when other borrowers repay, the maker can withdraw those repayments. The core invariant "offers cannot be replayed, overfilled, or reused" is violated.

### Likelihood Explanation
The precondition `units * buyerPrice < WAD` is trivially satisfiable: any tick below `WAD` (i.e. any tick below the maximum) combined with `units = 1` satisfies it. The attacker needs only a valid taker address (different from the maker), sufficient collateral to pass the `isHealthy` check, and a buy offer in assets mode — all of which are unprivileged, attacker-controlled inputs. The attack is repeatable in a single transaction via `multicall`.

### Recommendation
Add a guard immediately after computing `buyerAssets`/`sellerAssets` that reverts when `units > 0` but the asset amount that would be added to `consumed` is zero:

```solidity
if (offer.maxAssets > 0) {
    uint256 assetsDelta = offer.buy ? buyerAssets : sellerAssets;
    require(units == 0 || assetsDelta > 0, ZeroAssetDelta());
    newConsumed = consumed[offer.maker][offer.group] += assetsDelta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This mirrors the existing behaviour of units mode, where `consumed += units` always advances by a non-zero amount when `units > 0`.

### Proof of Concept
```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight, Offer} from "src/Midnight.sol";
// ... standard test setup (market, loanToken, collateralToken, ratifier)

function testMaxAssetsBypassViaZeroRounding() public {
    // Arrange: buy offer in assets mode, low tick so buyerPrice = 1e15
    uint256 lowTick = /* tick such that tickToPrice(tick) == 1e15 */ ...;
    Offer memory offer;
    offer.buy        = true;
    offer.maker      = maker;
    offer.maxAssets  = 1e18;   // maker intends to cap at 1e18 tokens
    offer.tick       = lowTick;
    offer.ratifier   = address(dummyRatifier);
    offer.expiry     = type(uint256).max;
    // units chosen so units * buyerPrice < WAD
    uint256 units = 999; // 999 * 1e15 = 999e15 < 1e18

    // taker needs collateral to stay healthy
    collateralize(market, taker, units * 1000); // generous headroom

    uint256 consumedBefore = midnight.consumed(maker, offer.group);

    vm.prank(taker);
    (uint256 buyerAssets, uint256 sellerAssets) =
        midnight.take(offer, hex"", units, taker, address(0), address(0), hex"");

    // Assert: both asset amounts are zero
    assertEq(buyerAssets,  0, "buyerAssets must be 0");
    assertEq(sellerAssets, 0, "sellerAssets must be 0");

    // Assert: consumed did NOT advance — maxAssets cap is bypassed
    assertEq(midnight.consumed(maker, offer.group), consumedBefore,
        "consumed must not change when buyerAssets == 0");

    // Assert: credit/debt DID change (state mutation without token transfer)
    assertEq(midnight.creditOf(id, maker), units, "maker credit increased");
    assertEq(midnight.debtOf(id, taker),   units, "taker debt increased");

    // Assert: calling take again succeeds (offer reuse)
    vm.prank(taker);
    midnight.take(offer, hex"", units, taker, address(0), address(0), hex"");
    assertEq(midnight.consumed(maker, offer.group), consumedBefore,
        "consumed still unchanged after second take");
}
```

Expected: both `assertEq(consumed, consumedBefore)` assertions pass, demonstrating that the `maxAssets` limit is never enforced and the offer is reusable indefinitely.
