### Title
Persistent `isRootRatified` State Survives Operator Revocation, Allowing Stale Offer Fills - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier.isRatified` only checks the stored `isRootRatified[maker][root]` boolean at fill time, without re-verifying whether the address that originally set the ratification is still authorized by the maker. Once an authorized operator calls `setIsRootRatified(maker, root, true)`, that state persists permanently even after the maker revokes the operator via `midnight.setIsAuthorized(operator, false, maker)`, allowing any taker to continue filling offers under that root against the maker's current intent.

### Finding Description
**Root cause — `SetterRatifier.isRatified` (line 35):**

```solidity
require(isRootRatified[offer.maker][root], NotRatified());
```

This is a pure storage read. It does not check who set the flag or whether that setter is still authorized. Compare with `EcrecoverRatifier.isRatified` (line 44), which performs a live authorization check at every fill:

```solidity
require(_signer == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, _signer), Unauthorized());
```

**Exact call sequence:**

1. Maker calls `midnight.setIsAuthorized(address(setterRatifier), true, maker)` — required for `Midnight.take` line 355 to pass.
2. Maker calls `midnight.setIsAuthorized(operator, true, maker)` — grants operator delegation.
3. Operator calls `setterRatifier.setIsRootRatified(maker, root, true)` — line 25 passes (operator is authorized), line 26 writes `isRootRatified[maker][root] = true`.
4. Maker calls `midnight.setIsAuthorized(operator, false, maker)` — revokes operator. `isRootRatified[maker][root]` is **not touched**.
5. Any taker calls `midnight.take(offer, abi.encode(root, leafIndex, proof), ...)`.
   - Line 355: `isAuthorized[maker][setterRatifier]` → still `true` (maker only revoked the operator, not the ratifier contract) → **passes**.
   - Line 356: `SetterRatifier.isRatified` → line 35 checks `isRootRatified[maker][root]` → still `true` → **passes**.
6. Take executes successfully against the maker's revoked intent.

**Why existing checks fail:**
- `Midnight.take` line 355 checks `isAuthorized[offer.maker][offer.ratifier]` — this is the maker→SetterRatifier authorization, which is separate from and unaffected by the maker→operator revocation.
- `SetterRatifier.isRatified` has no live authorization check whatsoever; it only reads the stored boolean.
- The maker has no enumeration mechanism to discover all roots an operator ratified on their behalf, making manual cleanup via `setIsRootRatified(maker, root, false)` impractical.

The protocol's own test suite confirms the intended behavior for `EcrecoverRatifier`: `testIsRatifiedRevokeAuthorizationInvalidates` (lines 185–205 of `test/EcrecoverRatifierTest.sol`) explicitly asserts that revoking an operator's authorization causes `isRatified` to revert. No equivalent test exists for `SetterRatifier`, and the behavior is the opposite.

### Impact Explanation
Any taker can fill offers under a Merkle root that was ratified by a now-revoked operator. The maker's credit/debt position is modified (units consumed, credit/debt balances changed per `Midnight.take` lines 368–384) without the maker's current consent. In a restricted market (gated by the ratifier), this bypasses the access control the maker intended to enforce by revoking the operator.

### Likelihood Explanation
The preconditions are a normal operational pattern: a maker delegates to an operator (e.g., a keeper or automated system), the operator ratifies a root, and the maker later rotates or revokes that operator (key compromise, end of service, etc.). The stale ratification persists indefinitely with no automatic cleanup. Any taker — including an attacker who monitors `SetIsRootRatified` events — can exploit the window between revocation and manual cleanup. The attack is repeatable for every root the operator ratified.

### Recommendation
Store the address that ratified each root and re-verify their authorization at fill time:

```solidity
mapping(address maker => mapping(bytes32 root => address ratifier)) public rootRatifier;

function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    rootRatifier[maker][root] = newIsRootRatified ? msg.sender : address(0);
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}

function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
    require(msg.sender == MIDNIGHT, NotMidnight());
    (bytes32 root, uint256 leafIndex, bytes32[] memory proof) =
        abi.decode(ratifierData, (bytes32, uint256, bytes32[]));
    require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
    require(isRootRatified[offer.maker][root], NotRatified());
    address setter = rootRatifier[offer.maker][root];
    require(setter == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, setter), Unauthorized());
    return CALLBACK_SUCCESS;
}
```

This mirrors `EcrecoverRatifier`'s live authorization check and ensures revocation immediately invalidates all roots set by the revoked operator.

function testRevokeAuthorizationDoesNotInvalidateSetterRatification() public {

    bytes32 _root = HashLib.hashOffer(offer);

    // Step 1: lender authorizes setterRatifier contract and borrower as operator
    vm.prank(lender);
    midnight.setIsAuthorized(address(setterRatifier), true, lender);
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, true, lender);

    // Step 2: operator ratifies root on behalf of lender
    vm.prank(borrower);
    setterRatifier.setIsRootRatified(lender, _root, true);
    assertTrue(setterRatifier.isRootRatified(lender, _root));

    // Step 3: lender revokes operator
    vm.prank(lender);
    midnight.setIsAuthorized(borrower, false, lender);
    assertFalse(midnight.isAuthorized(lender, borrower));

    // Step 4: isRatified still passes — BUG: should revert
    vm.prank(address(midnight));
    bytes32 result = setterRatifier.isRatified(offer, abi.encode(_root, 0, new bytes32[](0)));

    deal(address(loanToken), lender, 1e18);
    vm.prank(borrower);
