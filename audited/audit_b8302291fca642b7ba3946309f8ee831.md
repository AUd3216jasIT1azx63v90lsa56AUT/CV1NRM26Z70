Audit Report

## Title
Buy-offer `maxAssets` cap bypassed via rounding-to-zero `buyerAssets` on small-unit takes - (File: `src/Midnight.sol`)

## Summary
When a buy offer has `maxAssets > 0` and `buyerPrice < WAD`, `buyerAssets` is computed via `mulDivDown` and rounds to zero for sufficiently small `units`. Because `consumed` is incremented by `buyerAssets`, a zero-asset take leaves `consumed` unchanged and trivially passes the `ConsumedAssets` cap check, yet still mutates `credit`, `debt`, and `totalUnits`. This can be repeated without bound, even after the offer is fully consumed in asset terms.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363 and 368:**

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
// ...
newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
require(newConsumed <= offer.maxAssets, ConsumedAssets());
```

When `offer.buy == true` and `buyerPrice < WAD` (tick above `MAX_TICK/2`), any `units` satisfying `units * buyerPrice < WAD` produces `buyerAssets = 0`. The `consumed` accumulator is incremented by zero, so `newConsumed` is unchanged and the cap check `newConsumed <= offer.maxAssets` always passes — even when `consumed` is already at `maxAssets`.

**Position state still mutates despite zero assets — lines 410, 414, 416–417:**

`buyerPos.credit`, `sellerPos.debt`, and `_marketState.totalUnits` all increase by `units` regardless of whether `buyerAssets` is zero.

**Protocol's own acknowledgment — `src/Midnight.sol` line 94:**

```
/// @dev It is possible to give units to a fully consumed assets-based buy offer with price < 1.
```

**Exploit flow:**
1. Maker creates a buy offer: `maxAssets = N`, tick chosen so `buyerPrice < WAD`, `maxUnits = 0`.
2. Attacker calls `take(1, ...)` repeatedly (or in a loop via callback).
3. Each call: `buyerAssets = 1 * buyerPrice / WAD = 0`. `consumed` stays at its current value. Cap check passes. Lender credit and borrower debt each increase by 1 unit.
4. After `N` such calls, `consumed == maxAssets` but the offer remains permanently takeable; every subsequent call still produces `buyerAssets = 0` and passes the cap check.

**Why existing checks fail:** The sole guard is `require(newConsumed <= offer.maxAssets)`. Since `newConsumed` never increases when `buyerAssets = 0`, this check is permanently satisfied regardless of how many unit-level fills have occurred.

## Impact Explanation
The maker's `maxAssets` cap is intended to bound total economic exposure in asset terms. Due to rounding, an attacker can fill an unbounded number of units beyond what `maxAssets / buyerPrice` would imply. The lender (maker) accumulates credit redeemable at maturity for real assets, and the taker (borrower) accumulates debt that must be repaid — both without any asset transfer occurring at fill time. This violates the core protocol invariant that offers cannot be overfilled beyond their stated cap, resulting in unbounded, unintended lender credit and borrower debt accrual.

## Likelihood Explanation
Preconditions are minimal: any buy offer at a tick above `MAX_TICK/2` with `maxAssets > 0` and `maxUnits = 0` is vulnerable. No privileged access is required — any address can call `take()` as the taker. The attack is repeatable within a single transaction via callback or across multiple transactions. Ticks above `MAX_TICK/2` correspond to below-par lending rates, which are a normal and expected use case.

## Recommendation
When `offer.maxAssets > 0` and `offer.buy == true`, enforce a minimum of 1 asset per non-zero unit take, or revert when `units > 0` but `buyerAssets == 0`. Alternatively, track consumption in units rather than assets for buy offers, or add a guard:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(units == 0 || buyerAssets > 0, RoundedToZero());
}
```

This ensures that any non-trivial take always advances the `consumed` accumulator.

## Proof of Concept
The protocol's own test suite contains a direct reproduction at `test/TakeTest.sol:858`, function `testBugBuyMaxAssetsBypass`:

1. Set `lenderOffer.maxAssets = 1`, `lenderOffer.maxUnits = 0`, `lenderOffer.tick = MAX_TICK - 16` (so `offerPrice < WAD`).
2. Pre-consume the offer to its cap via `midnight.setConsumed(lenderOffer.group, 1, lender)`.
3. Call `take(1, borrower, lenderOffer)`.
4. Observe: `buyerAssets == 0`, `consumed` unchanged at `maxAssets`, but `creditOf(lender)`, `debtOf(borrower)`, and `totalUnits` all strictly increase.
