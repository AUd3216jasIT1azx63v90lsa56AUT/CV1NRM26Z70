Audit Report

## Title
`SetterRatifier` Lacks Chain Binding, Enabling Cross-Fork Offer Replay - (File: src/ratifiers/SetterRatifier.sol)

## Summary
`SetterRatifier` stores ratification state as `mapping(address maker => mapping(bytes32 root => bool)) public isRootRatified` with no chain discriminator. After a hard fork that changes `block.chainid`, this storage is copied verbatim to the forked chain. Because neither `SetterRatifier.isRatified` nor `HashLib.hashOffer` includes any chain-specific value, a taker can fill a maker's offers on the forked chain using ratification state the maker set only on the original chain, modifying the maker's position without post-fork consent.

## Finding Description
**Root cause — `SetterRatifier` has no chain binding:**

`setIsRootRatified` writes to a storage key of `(maker, root)` only:
```solidity
isRootRatified[maker][root] = newIsRootRatified;   // SetterRatifier.sol:26
```
`isRatified` checks only that same key, with no `block.chainid` or `INITIAL_CHAIN_ID` guard:
```solidity
require(isRootRatified[offer.maker][root], NotRatified());   // SetterRatifier.sol:35
```

**`HashLib.hashOffer` is chain-agnostic:**
The offer hash covers market fields, maker, expiry, tick, ratifier, etc. but never `block.chainid` or any chain-specific value (`HashLib.sol:118-138`). The same offer struct produces the same leaf hash on both chains.

**`Midnight.toId` uses `INITIAL_CHAIN_ID` (immutable), not `block.chainid`:**
`IdLib.toId` takes `chainId` as a parameter (`IdLib.sol:25`). After a fork, `block.chainid` changes but `INITIAL_CHAIN_ID` does not, so `toId` returns the **same** market ID on both chains, meaning `touchMarket` finds the already-existing market and proceeds directly to the ratifier check.

**Contrast with `EcrecoverRatifier`**, which explicitly binds to the current chain via:
```solidity
bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
// EcrecoverRatifier.sol:40
```
Its NatSpec explicitly states: *"If block.chainid changes (hard fork), the EIP-712 domain separator changes and previously signed offers are no longer valid."* (`EcrecoverRatifier.sol:10-11`). `SetterRatifier` has no equivalent protection.

**Full exploit path:**
1. Pre-fork: maker calls `setIsRootRatified(maker, root, true)` → `isRootRatified[maker][root] = true`.
2. Hard fork: `block.chainid` changes; all storage is copied; `isRootRatified[maker][root]` remains `true` on the forked chain.
3. Post-fork: taker calls `midnight.take(offer, abi.encode(root, leafIndex, proof), units, taker, ...)` on the forked chain.
4. `touchMarket` resolves the same market ID (via unchanged `INITIAL_CHAIN_ID`).
5. `isAuthorized[offer.maker][offer.ratifier]` — also copied from original chain, still `true`.
6. `isRatified` verifies the Merkle proof (chain-agnostic) and checks `isRootRatified[maker][root]` — `true`.
7. Take executes: maker's credit/debt position is modified on the forked chain without any post-fork authorization act by the maker.

## Impact Explanation
A taker can fill any `SetterRatifier`-backed offer on the forked chain using ratification state the maker set only on the original chain. The maker's position (credit increased or debt incurred) is modified on the forked chain without post-fork consent. This violates the core invariant that ratifiers bind the right user/market/action/chain. The impact is direct unauthorized modification of a user's financial position.

## Likelihood Explanation
Preconditions: (1) a contentious hard fork that changes `block.chainid` while keeping contract addresses identical (ETH/ETC-style), and (2) at least one maker using `SetterRatifier` with an active, non-expired offer. Both are realistic — hard forks are historical precedent, and `SetterRatifier` is a first-party ratifier. The attack is repeatable for every ratified root not explicitly revoked post-fork, requires no special privilege, and is executable by any taker.

## Recommendation
Bind `SetterRatifier` ratification state to the chain, mirroring `EcrecoverRatifier`'s approach. Options:
1. Include `block.chainid` in the storage key: `isRootRatified[block.chainid][maker][root]`.
2. Store `INITIAL_CHAIN_ID` as an immutable at construction and require `block.chainid == INITIAL_CHAIN_ID` in `isRatified`, invalidating all state on a fork.
3. Include `chainId` in the Merkle leaf hash via `hashOffer`, though this would require changes to `HashLib` affecting all ratifiers.

Option 1 or 2 is the minimal targeted fix for `SetterRatifier` specifically.

## Proof of Concept
Fork test plan:
1. Deploy `Midnight` and `SetterRatifier` on a local fork at `chainId = 1`.
2. Maker calls `setIsRootRatified(maker, root, true)`.
3. Use `vm.chainId(2)` (Foundry cheatcode) to simulate a hard fork changing `block.chainid` to `2`.
4. Confirm `isRootRatified[maker][root]` is still `true` (storage unchanged).
5. Confirm `Midnight.toId(market)` returns the same bytes32 (since `INITIAL_CHAIN_ID` is immutable).
6. Taker calls `midnight.take(offer, ratifierData, units, taker, ...)` — confirm it succeeds and modifies the maker's position without any post-fork action by the maker.
