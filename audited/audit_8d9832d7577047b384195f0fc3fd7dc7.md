### Title
`claimContinuousFee` Is Permanently Blocked When `withdrawable` Is Zero Despite Accrued `continuousFeeCredit` — (File: src/Midnight.sol)

### Summary
The `claimContinuousFee` function unconditionally decreases `marketState.withdrawable` when the fee claimer collects accrued continuous fees. However, the mechanism that accrues those fees — `_updatePosition` — only increases `continuousFeeCredit` with no corresponding increase to `withdrawable`. This creates a persistent state where `continuousFeeCredit > 0` but `withdrawable == 0`, causing every call to `claimContinuousFee` to revert via underflow, making continuous fee collection impossible until borrowers independently repay or are liquidated.

### Finding Description

**Root cause — fee accrual does not increase `withdrawable`:**

In `_updatePosition`, the accrued continuous fee is credited to the protocol: [1](#0-0) 

```solidity
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
```

`withdrawable` is never touched here.

**Consumption side — `claimContinuousFee` decreases `withdrawable`:** [2](#0-1) 

```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
_marketState.totalUnits -= UtilsLib.toUint128(amount);
_marketState.withdrawable -= UtilsLib.toUint128(amount);   // ← underflows when withdrawable == 0
```

**The only sources that increase `withdrawable`:** [3](#0-2) [4](#0-3) 

```solidity
// repay():
marketState[id].withdrawable += UtilsLib.toUint128(units);

// liquidate():
_marketState.withdrawable += UtilsLib.toUint128(repaidUnits);
```

There is no path by which fee accrual itself increases `withdrawable`. The fee claimer's claim is therefore not backed by any corresponding increase in the withdrawable pool.

**Exploit path:**
1. Market is created with a non-zero `continuousFee`.
2. A lender takes a buy offer, gaining credit; a borrower takes a sell offer, gaining debt. `withdrawable` remains `0`.
3. Time passes; `_updatePosition` is called (e.g., inside a subsequent `take`). `continuousFeeCredit` grows; `withdrawable` stays `0`.
4. Fee claimer calls `claimContinuousFee(market, amount, receiver)` with any `amount > 0`.
5. Execution reaches `_marketState.withdrawable -= amount` → arithmetic underflow → revert.

This is structurally identical to the referenced report: a function (`claimContinuousFee`) depends on a state variable (`withdrawable`) that is never updated by the mechanism (`_updatePosition`) that logically should update it.

### Impact Explanation
Protocol continuous-fee revenue is permanently uncollectable for the entire duration of any market in which no early repayment or liquidation has occurred. In fixed-rate markets where borrowers rationally hold to maturity, `withdrawable` stays at zero throughout the market's life, meaning `continuousFeeCredit` accumulates but can never be claimed. Even after some repayments occur, lenders racing to `withdraw` can drain `withdrawable` back to zero before the fee claimer acts, re-creating the same DoS. Protocol revenue is effectively frozen.

### Likelihood Explanation
High. Fixed-rate borrowers have no incentive to repay early (they locked in a rate). A market running for months with zero early repayments is the normal case, not an edge case. No attacker action is required; the DoS arises from ordinary protocol usage.

### Recommendation
Remove the `withdrawable` decrement from `claimContinuousFee`. The continuous fee is accrued from lenders' credit (a unit-accounting entry), not from liquid tokens already in the contract. The fee claimer's claim should be settled by transferring tokens directly from the contract's balance — which always holds at least `totalUnits` worth of loan tokens — without touching `withdrawable`, which tracks only early-repaid assets. Alternatively, increase `withdrawable` inside `_updatePosition` by `accruedFee` to keep the accounting consistent.

### Proof of Concept

```
1. Deploy Midnight; set feeClaimer = Alice; set continuousFee > 0 for loanToken.
2. Bob (lender) takes a sell offer: Bob gains 1000 credit units; Carol (borrower) gains 1000 debt units.
   → withdrawable = 0, continuousFeeCredit = 0
3. Advance time by 30 days.
4. Call updatePosition(market, Bob).
   → continuousFeeCredit = X  (X > 0, proportional to fee rate × time)
   → withdrawable = 0  (unchanged)
5. Alice (feeClaimer) calls claimContinuousFee(market, X, Alice).
   → Execution: _marketState.withdrawable -= X
   → Reverts: arithmetic underflow (0 - X)
6. Alice can never collect fees until Carol independently calls repay().
``` [5](#0-4) [6](#0-5)

    function claimContinuousFee(Market memory market, uint256 amount, address receiver) external {
        bytes32 id = toId(market);
        MarketState storage _marketState = marketState[id];
        require(msg.sender == feeClaimer, OnlyFeeClaimer());
        require(_marketState.tickSpacing > 0, MarketNotCreated());

        _marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
        _marketState.totalUnits -= UtilsLib.toUint128(amount);
        _marketState.withdrawable -= UtilsLib.toUint128(amount);

        emit EventsLib.ClaimContinuousFee(msg.sender, id, amount, receiver);

        SafeTransferLib.safeTransfer(market.loanToken, receiver, amount);
    }

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
