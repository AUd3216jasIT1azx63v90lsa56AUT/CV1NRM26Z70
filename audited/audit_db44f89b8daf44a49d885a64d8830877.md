### Title
ERC721 Tokens Can Be Permanently Trapped as Collateral or Loan Token Due to Shared `transferFrom` Selector — (File: src/Midnight.sol, src/libraries/SafeTransferLib.sol)

### Summary
`ERC721.transferFrom(address,address,uint256)` and `ERC20.transferFrom(address,address,uint256)` share the identical 4-byte selector `0x23b872dd`. Because `touchMarket` performs no on-chain validation that `loanToken` or any `collateralParams[i].token` is an ERC-20 contract, an ERC-721 token can be silently accepted by `safeTransferFrom` on the way in, yet permanently trapped because `safeTransfer` calls `IERC20.transfer(to,value)` (selector `0xa9059cbb`), a function that does not exist on ERC-721 contracts.

### Finding Description

**Root cause — no token-type guard at market creation**

`touchMarket` validates collateral ordering, LLTV, and maxLif, but never checks that `market.loanToken` or any `market.collateralParams[i].token` implements ERC-20 rather than ERC-721: [1](#0-0) 

**Inbound path succeeds (same selector)**

`SafeTransferLib.safeTransferFrom` encodes the call as `IERC20.transferFrom(from, to, value)`: [2](#0-1) 

Because `ERC721.transferFrom(address,address,uint256)` has the identical selector `0x23b872dd`, the call dispatches correctly on an ERC-721 contract and the NFT (whose token-id equals the `assets` argument) is transferred into `Midnight`.

**Outbound path reverts (missing selector)**

`SafeTransferLib.safeTransfer` encodes the call as `IERC20.transfer(to, value)` (selector `0xa9059cbb`): [3](#0-2) 

ERC-721 contracts have no `transfer(address,uint256)` function. The low-level call reverts, and `safeTransfer` propagates the revert. Every exit path that calls `safeTransfer` on the trapped token is permanently bricked.

**Affected exit paths in `Midnight.sol`**

| Function | Line | Token |
|---|---|---|
| `withdrawCollateral` | 572 | collateral token |
| `withdraw` | 499 | loan token |
| `claimSettlementFee` | 309 | loan token |
| `claimContinuousFee` | 324 | loan token |
| `liquidate` (seize) | 696 | collateral token | [4](#0-3) [5](#0-4) [6](#0-5) 

### Impact Explanation

Any ERC-721 token transferred into `Midnight` as collateral or as a loan-token payment is **permanently irrecoverable**. There is no admin escape hatch, no rescue function, and no upgrade mechanism. The NFT is locked in the singleton contract forever. Depending on the NFT's value, this constitutes a direct, irreversible loss of user funds.

### Likelihood Explanation

Market creation is **permissionless** — any address can call `touchMarket` with an ERC-721 address as `loanToken` or as a `collateralParams[i].token`. A malicious actor can:

1. Deploy or reuse an existing ERC-721 contract.
2. Create a Midnight market listing it as the collateral token.
3. Advertise the market (e.g., via a forked UI or off-chain offer distribution).
4. Wait for a victim to call `supplyCollateral` with their NFT.

The victim's NFT is accepted silently (no revert), and the victim only discovers the loss when `withdrawCollateral` reverts. No privileged key is required. The only social-engineering barrier is convincing a user to interact with the crafted market.

### Recommendation

Add an on-chain check in `touchMarket` that rejects any token address that returns `true` for ERC-165 interface ID `0x80ac58cd` (ERC-721), or alternatively maintain a protocol-level whitelist of approved ERC-20 tokens for both `loanToken` and collateral tokens. At minimum, document clearly in the interface that ERC-721 addresses must never be used and add a revert guard:

```solidity
// In touchMarket, before accepting loanToken or collateralToken:
require(!_isERC721(market.loanToken), InvalidToken());
for (uint256 i = 0; i < market.collateralParams.length; i++) {
    require(!_isERC721(market.collateralParams[i].token), InvalidToken());
}
```

where `_isERC721` calls `supportsInterface(0x80ac58cd)` and returns `true` only if the call succeeds and returns `true`.

### Proof of Concept

1. Deploy a standard ERC-721 contract (`MockNFT`). Mint token ID `1e18` to Alice.
2. Call `touchMarket` with `market.collateralParams[0].token = address(MockNFT)`.
3. Alice approves `Midnight` and calls `supplyCollateral(market, 0, 1e18, alice)`.
   - `SafeTransferLib.safeTransferFrom(MockNFT, alice, Midnight, 1e18)` dispatches to `ERC721.transferFrom(alice, Midnight, 1e18)` — **succeeds**, NFT now held by `Midnight`.
4. Alice calls `withdrawCollateral(market, 0, 1e18, alice, alice)`.
   - `SafeTransferLib.safeTransfer(MockNFT, alice, 1e18)` dispatches to `MockNFT.transfer(alice, 1e18)` — **reverts** (no such function on ERC-721).
5. Alice's NFT is permanently locked. No other function can recover it.

```text
        bytes32 id = toId(market);
        if (marketState[id].tickSpacing == 0) {

            require(market.collateralParams.length > 0, NoCollateralParams());

            address previousCollateralToken;
```

```text
    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
    }
```
