Audit Report

## Title
`sellerAssetsToUnits` arithmetic revert at `offerPrice == settlementFee` blocks `supplyCollateralAndSellWithAssetsTarget` while `Midnight.take()` succeeds - (File: `src/periphery/TakeAmountsLib.sol`)

## Summary

When a buy offer's tick price exactly equals the settlement fee, `sellerPrice` computes to zero in `TakeAmountsLib.sellerAssetsToUnits`. The subsequent `mulDivUp(targetSellerAssets, WAD, 0)` call triggers an arithmetic underflow revert via the `d - 1` term under Solidity 0.8 checked arithmetic. Because `sellerAssetsToUnits` is invoked outside the `try/catch` block in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget`, the entire bundler call reverts, while `Midnight.take()` itself would succeed with `sellerAssets = 0`.

## Finding Description

**Root cause — `TakeAmountsLib.sellerAssetsToUnits` (lines 44–46):** [1](#0-0) 

When `offer.buy == true` and `offerPrice == settlementFee`, the subtraction `offerPrice - settlementFee` yields `sellerPrice = 0`. The function then calls `targetSellerAssets.mulDivUp(WAD, 0)`.

**`mulDivUp` implementation (`UtilsLib.sol` line 35):** [2](#0-1) 

With `d = 0`, the expression `d - 1` underflows under Solidity 0.8 checked arithmetic, producing an `arithmeticError` revert — not a division-by-zero revert.

**`Midnight.take()` does not revert in the same case:**

`Midnight.take()` computes `sellerAssets = units.mulDivDown(sellerPrice, WAD)` = `(units * 0) / WAD = 0`. The denominator is `WAD` (non-zero), so no revert occurs.

**The comment at line 34 of `sellerAssetsToUnits` is inaccurate:** [3](#0-2) 

The comment claims "midnight reverts too" for `offerPrice < settlementFee`. This is only true for strict inequality. When `offerPrice == settlementFee`, Midnight does **not** revert, but `sellerAssetsToUnits` does.

**Call site in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` (lines 285–300):** [4](#0-3) 

`TakeAmountsLib.sellerAssetsToUnits` is called at lines 286–288, **outside** the `try/catch` block that wraps `Midnight.take()` at lines 292–300. A revert in `sellerAssetsToUnits` propagates uncaught, aborting the entire bundler call.

**Preconditions are realistic:**

`PRICE_ROUNDING_STEP = CBP = 1e12`: [5](#0-4) [6](#0-5) 

`tickToPrice` rounds to multiples of `PRICE_ROUNDING_STEP`: [7](#0-6) 

Both tick prices and settlement fees are multiples of `1e12`. The post-maturity settlement fee ranges from `0` to `14e12`, and tick prices in the low-price range are dense multiples of `1e12`, so valid ticks exist where `tickToPrice(tick) == settlementFee`.

**`NoDivisionByZero.spec` covers only `Midnight.sol`, not `TakeAmountsLib.sol`:** [8](#0-7) 

The spec verifies `Midnight` contract methods only; `MidnightBundles` and `TakeAmountsLib` are not in scope.

**The bundler's own NatSpec at line 246 documents the symptom but provides no protection:** [9](#0-8) 

## Impact Explanation

Any taker calling `supplyCollateralAndSellWithAssetsTarget` whose `takes` array contains a buy offer at the settlement-fee price boundary will have the entire call revert with an arithmetic error. The taker cannot use the atomic collateral-supply-and-sell bundler path to unwind at maturity, even though `Midnight.take()` would succeed. The taker must fall back to calling `Midnight.take()` directly, losing the atomic multi-offer aggregation the bundler provides. There is no loss of funds, but there is a concrete functional DoS of the bundler path.

## Likelihood Explanation

The condition requires `tickToPrice(offer.tick) == settlementFee`. Both quantities are multiples of `CBP = 1e12`, and tick prices are dense in the low-price range (up to `14e12`), making alignment achievable without any admin action: an offer maker creates a buy offer at a tick whose price equals the current settlement fee, and a taker calling the bundler with that offer triggers the revert. The fee setter can also raise the settlement fee to match an existing offer's tick price, which is documented protocol behavior. The condition is repeatable across any market where `settlementFeeCbp0 * CBP` aligns with a tick price.

## Recommendation

Guard against `sellerPrice == 0` in `sellerAssetsToUnits`. One approach: if `sellerPrice == 0`, return `type(uint256).max` (or a sentinel indicating "infinite units needed"), so the `min(...)` call in the bundler reduces `unitsToTake` to zero and the `try/catch` block is entered with zero units (which `Midnight.take()` handles gracefully). Alternatively, move the `sellerAssetsToUnits` call inside the `try/catch` block, or add an explicit `require(sellerPrice > 0)` with a descriptive revert that the bundler can catch.

## Proof of Concept

1. Deploy a market with `settlementFeeCbp0 = k` for some `k` in `[1, 14]`.
2. Create a buy offer at the tick `t` such that `tickToPrice(t) == k * 1e12`.
3. After maturity, call `supplyCollateralAndSellWithAssetsTarget` with that offer in the `takes` array.
4. Observe the call reverts with an arithmetic error from `mulDivUp(targetSellerAssets, WAD, 0)` in `TakeAmountsLib.sellerAssetsToUnits`.
5. Confirm that calling `Midnight.take()` directly with the same offer succeeds and returns `sellerAssets = 0`.
