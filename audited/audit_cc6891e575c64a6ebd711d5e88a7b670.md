### Title
SetterRatifier Root Transition Vulnerable to Frontrun Double-Consumption - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier.setIsRootRatified` is a bidirectional boolean setter that allows a maker to both activate and deactivate offer roots. When a maker transitions from an old root (R1) to a new root (R2), an attacker can frontrun the deactivation of R1, consume its full capacity, and then consume R2's full capacity after activation — forcing the maker into N+M units of positions instead of the intended M.

### Finding Description
`SetterRatifier.setIsRootRatified` sets `isRootRatified[maker][root]` to any boolean value:

```solidity
// src/ratifiers/SetterRatifier.sol:24-28
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}
``` [1](#0-0) 

When `isRatified` is called during `take`, it checks this mapping:

```solidity
// src/ratifiers/SetterRatifier.sol:35-36
require(isRootRatified[offer.maker][root], NotRatified());
return CALLBACK_SUCCESS;
``` [2](#0-1) 

**Exploit path:**
1. Maker has root R1 ratified with sell offers totaling N units of debt capacity.
2. Maker wants to update to root R2 with M units. They broadcast `setIsRootRatified(maker, R1, false)`.
3. Attacker sees this in the mempool, frontruns it by calling `take()` against R1 offers, consuming all N units. Maker now has N units of debt.
4. Maker's deactivation of R1 lands. Maker then calls `setIsRootRatified(maker, R2, true)`.
5. Attacker calls `take()` against R2 offers, consuming all M units. Maker now has N+M units of debt.
6. Maker intended to carry M units of debt; they now carry N+M.

The `consumed` tracking in `take` enforces per-offer caps but does not prevent consuming across two different roots sequentially:

```solidity
// src/Midnight.sol:367-373
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
} else {
    newConsumed = consumed[offer.maker][offer.group] += units;
    require(newConsumed <= offer.maxUnits, ConsumedUnits());
}
``` [3](#0-2)

### Impact Explanation
For a **sell offer maker** (maker is borrower): the maker is forced into N+M units of debt instead of M. This can push the position past its LLTV threshold, making it immediately liquidatable and causing direct loss of collateral.

For a **buy offer maker** (maker is lender): the maker is forced to deploy N+M units of loan tokens instead of M, draining more capital than intended.

In both cases, the maker suffers a concrete, quantifiable loss proportional to N (the capacity of the old root).

### Likelihood Explanation
- Offer data is distributed off-chain (e.g., via API), so the attacker can observe R1 offers before the root is deactivated.
- Mempool monitoring for `setIsRootRatified(..., false)` transactions is standard MEV infrastructure.
- The maker must have sufficient capital to cover both N and M (otherwise the second take reverts), but this is a realistic condition for active market makers.
- No privileged access is required; any external actor can call `take()`.

### Recommendation
Replace the bidirectional setter with two separate monotonic functions, mirroring `EcrecoverRatifier`'s design:

```solidity
function activateRoot(address maker, bytes32 root) external { ... }   // sets true only
function deactivateRoot(address maker, bytes32 root) external { ... } // sets false only
```

Alternatively, document that makers **must** atomically switch roots using `multicall` (deactivate R1 and activate R2 in a single transaction) to eliminate the frontrun window. However, the contract-level fix is strongly preferred.

### Proof of Concept
1. Maker calls `setIsRootRatified(maker, R1, true)` — R1 contains sell offers for N units at tick T.
2. Maker broadcasts `setIsRootRatified(maker, R1, false)` to switch to R2.
3. Attacker observes the pending tx, submits `take(offerFromR1, ..., N, attacker, ...)` with higher gas — lands first. Maker's position now has N units of debt.
4. Maker's deactivation lands. Maker calls `setIsRootRatified(maker, R2, true)` — R2 contains sell offers for M units.
5. Attacker calls `take(offerFromR2, ..., M, attacker, ...)`. Maker's position now has N+M units of debt.
6. If N+M exceeds the maker's `maxDebt` (computed via LLTV and oracle price), `isHealthy` returns false and the maker's collateral is immediately liquidatable. [5](#0-4) [6](#0-5)
