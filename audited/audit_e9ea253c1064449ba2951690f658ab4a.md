Audit Report

## Title
Bad Debt Over-Realization via `onLiquidate` Callback Supplying Collateral After Bitmap Snapshot - (File: `src/Midnight.sol`)

## Summary
The `liquidate` function snapshots `_position.collateralBitmap` into a local variable before computing `badDebt`, irreversibly applies the bad debt socialization to `_marketState.lossFactor` and `_marketState.totalUnits`, and only then invokes the `onLiquidate` callback. A borrower-controlled smart contract can exploit this ordering by calling `supplyCollateral` inside the callback window, depositing real collateral that was invisible to the bad debt computation, causing the protocol to over-socialize losses onto lenders.

## Finding Description

**Bitmap snapshot before bad debt computation:**

At line 606, `_collateralBitmap` is copied into a local stack variable. The while loop at lines 607–618 iterates only over this local copy. Any collateral index not set in the bitmap at snapshot time is invisible to the `badDebt` computation. [1](#0-0) 

**Bad debt irreversibly applied before callback:**

Lines 626–641 reduce `_position.debt`, ratchet up `_marketState.lossFactor`, and reduce `_marketState.totalUnits` — all before any external call is made. [2](#0-1) 

**Callback invoked after all state mutations:**

The `onLiquidate` callback is called at lines 698–715, after all accounting state has been finalized. [3](#0-2) 

**`supplyCollateral` authorization trivially satisfied:**

`supplyCollateral` checks `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]`. When `BorrowerContract` calls `supplyCollateral(..., address(this))` inside `onLiquidate`, `msg.sender == BorrowerContract` and `onBehalf == address(this) == BorrowerContract`, so the check passes unconditionally. [4](#0-3) 

`supplyCollateral` then sets the new bit in the storage bitmap, making the new collateral visible in the position going forward but not retroactively to the already-applied `badDebt`. [5](#0-4) 

**No reentrancy guard on `liquidate` or `supplyCollateral`:**

The `liquidationLocked` mechanism via `tExchange` is only set during `take` callbacks for the seller (line 444). The `liquidate` function never sets a lock for the borrower during its own callback window. [6](#0-5) 

There is no `nonReentrant` modifier or equivalent transient lock on `liquidate` or `supplyCollateral`.

**Exploit flow:**
1. `BorrowerContract` holds a position with genuine bad debt (all existing collateral worth less than debt at `maxLif` prices).
2. `BorrowerContract` calls `liquidate(market, 0, 0, 0, address(this), false, receiver, address(this), "")` — `seizedAssets = repaidUnits = 0`, `callback = address(this)`. This is explicitly supported per the comment at line 577–578.
3. `liquidate` snapshots `_collateralBitmap` (line 606), computes `badDebt` from existing collateral only.
4. `badDebt` is applied: `_position.debt` reduced, `_marketState.lossFactor` increased, `_marketState.totalUnits` reduced (lines 626–641).
5. `onLiquidate` is called on `BorrowerContract` (line 700).
6. Inside `onLiquidate`, `BorrowerContract` calls `supplyCollateral(market, newIndex, largeAmount, address(this))`, depositing real collateral and setting a new bit in `_position.collateralBitmap` in storage.
7. `liquidate` returns. The new collateral is in the position, but `badDebt` was already applied without accounting for it. The borrower's debt was forgiven by an amount actually backed by the newly supplied collateral.

## Impact Explanation
`_marketState.lossFactor` is set higher than warranted, and `_marketState.totalUnits` is reduced by more than the true uncollateralized shortfall. Every lender who subsequently calls `updatePosition` has their credit slashed proportionally to the inflated loss factor — a direct, concrete loss of lender funds. The borrower's debt is forgiven by an amount that was actually backed by the newly supplied collateral, constituting an illegitimate debt reduction at lenders' expense. This is a high-severity accounting integrity violation with direct fund loss to lenders.

## Likelihood Explanation
Preconditions are minimal: (1) a position with genuine bad debt — a normal liquidation scenario; (2) the borrower is a smart contract — standard in DeFi; (3) the borrower passes itself as the callback — requires no external authorization, no oracle manipulation, no privileged access, and no victim mistakes. The attack is repeatable on any market with bad debt and is fully self-contained.

## Recommendation
Apply one or more of the following mitigations:

1. **Re-read the bitmap from storage after the callback** and recompute `badDebt`, reverting if it differs from the pre-callback value.
2. **Set a transient reentrancy lock on `liquidate`** (analogous to the existing `tExchange` lock used in `take`) that prevents `supplyCollateral` from being called for the borrower during the liquidation callback window.
3. **Move the `onLiquidate` callback before bad debt realization**, so any collateral supplied during the callback is visible to the `badDebt` computation.
4. **Snapshot and re-validate** `_position.collateralBitmap` from storage after the callback returns, reverting if it has changed.

## Proof of Concept

```solidity
contract BorrowerContract is ILiquidateCallback {
    Midnight midnight;
    Market market;
    uint256 newCollateralIndex;
    uint256 largeAmount;

    // Setup: BorrowerContract has genuine bad debt in `market`.
    // All existing collateral is worth less than debt at maxLif prices.

    function triggerExploit() external {
        // Call liquidate on own position with seizedAssets=repaidUnits=0, callback=this
        midnight.liquidate(market, 0, 0, 0, address(this), false, address(this), address(this), "");
    }

    function onLiquidate(
        address, bytes32, Market memory _market, uint256, uint256, uint256, address, address, bytes calldata, uint256
    ) external returns (bytes4) {
        // At this point, badDebt has already been applied to lossFactor/totalUnits.
        // Supply real collateral for a new index not in the original bitmap.
        IERC20(market.collateralParams[newCollateralIndex].token).approve(address(midnight), largeAmount);
        midnight.supplyCollateral(_market, newCollateralIndex, largeAmount, address(this));
        // New collateral is now in storage, but badDebt was computed without it.
        return CALLBACK_SUCCESS;
    }
}
```

After `triggerExploit()` returns, verify that `marketState[id].lossFactor` is higher than it would be if the new collateral had been present during bad debt computation, and that lender credits are slashed by the inflated amount on their next `updatePosition` call.
