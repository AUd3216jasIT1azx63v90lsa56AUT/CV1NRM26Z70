Audit Report

## Title
Fully-consumed assets-based buy offer with `buyerPrice < WAD` allows unbounded credit inflation and theft from the shared withdrawable pool - (File: `src/Midnight.sol`)

## Summary
When a buy offer with `maxAssets > 0` is fully consumed and `buyerPrice < WAD`, the consumed guard at line 368–369 can be bypassed indefinitely by calling `take` with `units = 1`, because `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0` and `consumed + 0 <= maxAssets` always passes. All position mutations at lines 408–417 execute on `units = 1` with zero token transfer, inflating the maker's `credit` and `totalUnits` without any corresponding asset deposit. An attacker who controls the maker address can then drain the shared `withdrawable` pool funded by real borrower repayments, stealing from legitimate lenders.

## Finding Description

**Root cause — lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN → 0 when buyerPrice < WAD and units = 1
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // maxAssets + 0 ≤ maxAssets ✓
}
```

`tickToPrice` is proven `<= WAD` for all ticks (Certora spec `NoMultiplicationOverflow.spec` line 52–56). For any tick where `tickToPrice(tick) < WAD`, a buy offer has `buyerPrice = tickToPrice(tick) < WAD`, so `mulDivDown(1, buyerPrice, WAD) = 0`. Once `consumed == maxAssets`, adding 0 satisfies the guard unconditionally.

**Position mutations — lines 408–417 — execute on `units = 1`:**

```solidity
buyerPos.credit += UtilsLib.toUint128(buyerCreditIncrease);   // +1 per call if buyer has no debt
sellerPos.debt   += UtilsLib.toUint128(sellerDebtIncrease);   // +1 per call if seller has no credit
_marketState.totalUnits += buyerCreditIncrease - sellerCreditDecrease;
```

**Token transfers — lines 455–456 — are zero:**

```solidity
SafeTransferLib.safeTransferFrom(..., buyerAssets - sellerAssets);  // 0 - 0 = 0
SafeTransferLib.safeTransferFrom(..., sellerAssets);                // 0
```

**Concrete theft path:**

1. Attacker (address A) creates a buy offer: `offer.buy = true`, `maxAssets = X`, tick where `tickToPrice < WAD`.
2. Attacker (address B ≠ A) takes the offer legitimately until `consumed[A][group] == maxAssets`. Real tokens are transferred; A accumulates `credit_real` credit.
3. Attacker (address C ≠ A) calls `take(offer, units=1)` N times. Each call: A gains +1 credit, C gains +1 debt, zero tokens transferred. After N calls A has `credit_real + N` credit.
4. Real borrowers repay their debt → `marketState.withdrawable += repaid_units`.
5. A calls `withdraw(units = credit_real + N)`. The `withdraw` function at line 493–499 decrements `_position.credit` and `_marketState.withdrawable` and transfers `units` tokens to A.

A deposited tokens worth `credit_real` units but withdraws `credit_real + N` units, stealing N tokens from the shared `withdrawable` pool that real lenders are entitled to.

**Why existing checks fail:**

- The consumed guard (line 369) checks `newConsumed <= maxAssets` but `newConsumed` is incremented by `buyerAssets`, not `units`. When `buyerAssets = 0`, the guard is permanently satisfied regardless of how many times `take` is called.
- There is no check enforcing `units == 0` when `consumed >= maxAssets`.
- There is no check enforcing `buyerAssets > 0` when `units > 0`.
- The `withdraw` function (lines 481–500) only checks `_marketState.withdrawable` sufficiency; it has no mechanism to detect that the credit being withdrawn was not backed by a real deposit.
- The protocol acknowledges this at line 94 ("It is possible to give units to a fully consumed assets-based buy offer with price < 1") but does not bound or reject the resulting state mutations.

## Impact Explanation

An attacker can inflate their `credit` by an arbitrary amount N at negligible gas cost (zero token transfers per iteration). When real borrowers repay, the attacker withdraws N extra tokens from the shared `withdrawable` pool, directly stealing from legitimate lenders who are entitled to those repayments. `totalUnits` is also inflated by N, corrupting all downstream accounting that depends on it (settlement, fee distribution, loss factor). The impact is direct theft of lender funds and permanent accounting corruption, both critical in-scope impacts.

## Likelihood Explanation

All preconditions are reachable by an unprivileged attacker:
- `offer.buy = true`, `maxAssets > 0`: standard offer parameters, no privilege required.
- `buyerPrice < WAD`: true for any tick where `tickToPrice(tick) < WAD`, which is the case for essentially all non-trivial ticks (proven by the Certora spec that `tickToPrice <= WAD`).
- Offer fully consumed: attacker uses a second address (B) to consume the offer; the `SelfTake` check (line 354) only blocks `offer.maker == taker`, not a distinct attacker-controlled address.
- Repeated `take` calls: zero token cost per call, repeatable every block, bounded only by gas.

The attack is fully self-contained, requires no victim cooperation, and scales linearly with the number of calls.

## Recommendation

Add a guard that rejects a `take` call when the offer is assets-capped, the offer is a buy offer, and `buyerAssets == 0` but `units > 0`. The minimal targeted fix, inserted immediately after line 364:

```solidity
if (offer.maxAssets > 0 && offer.buy) {
    require(units == 0 || buyerAssets > 0, ZeroAssetsTake());
}
```

Alternatively, change the consumed increment for buy offers to track `units` instead of `buyerAssets` when `maxAssets > 0`, or add a secondary check `require(newConsumed < offer.maxAssets || units == 0, ConsumedAssets())` so that a fully-consumed offer cannot accept any further non-zero `units`.

## Proof of Concept

```
Setup:
  - Deploy market with loanToken, tick T where tickToPrice(T) = WAD/2 (buyerPrice = WAD/2 < WAD)
  - Attacker address A creates buy offer: buy=true, maxAssets=100, tick=T, group=G
  - Attacker address B (≠ A) calls take(offer, units=200) → buyerAssets = mulDivDown(200, WAD/2, WAD) = 100
    → consumed[A][G] = 100 = maxAssets; A has 200 credit; B has 200 debt; 100 tokens transferred to contract

Attack:
  - Attacker address C (≠ A) calls take(offer, units=1) 1000 times
    → each call: buyerAssets = mulDivDown(1, WAD/2, WAD) = 0
    → consumed[A][G] = 100 + 0 = 100 ≤ 100 ✓ (guard passes every time)
    → A.credit += 1 per call → A.credit = 1200 after 1000 calls
    → C.debt += 1 per call → C.debt = 1000
    → zero tokens transferred each call

Theft:
  - Real borrower R repays 1000 units → withdrawable += 1000
  - A calls withdraw(units=1000) → A receives 1000 tokens
  - A deposited 0 tokens in the attack phase; net theft = 1000 tokens from the withdrawable pool

Assert: loanToken.balanceOf(A) increased by 1000; legitimate lenders cannot withdraw their share.
```
