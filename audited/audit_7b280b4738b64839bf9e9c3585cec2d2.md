### Title
Double `mulDivDown` in `isHealthy` Produces False-Negative at Exact Health Boundary, Causing `take` to Revert with `SellerIsLiquidatable()` - (File: src/Midnight.sol)

### Summary
The `isHealthy` function computes `maxDebt` using two sequential `mulDivDown` calls: `collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)`. The intermediate rounding in the first call can cause the final result to be strictly less than `floor(collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD))` — the exact single-step floor. When a taker fills a sell offer that brings the seller's debt to exactly the true mathematical maxDebt, the protocol's computed `maxDebt` can be `debt - 1`, causing `isHealthy` to return `false` and the `take` to revert with `SellerIsLiquidatable()` even though the position is genuinely healthy.

### Finding Description

**Code path:**

`take()` at `src/Midnight.sol:414` writes `sellerPos.debt += sellerDebtIncrease`, then at line 476 calls:

```solidity
require(liquidationLocked(id, seller) || isHealthy(offer.market, id, seller), SellerIsLiquidatable());
```

`isHealthy` at `src/Midnight.sol:944–959` computes:

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
// ...
return maxDebt >= debt;
```

`mulDivDown` is `(x * y) / d` (plain integer division, rounds down).

**Root cause — two-step rounding loss:**

Let `A = floor(C * P / ORACLE_PRICE_SCALE)`. The protocol computes `maxDebt = floor(A * lltv / WAD)`. The exact single-step floor is `floor(C * P * lltv / (ORACLE_PRICE_SCALE * WAD))`. Because `A` already lost up to `(ORACLE_PRICE_SCALE - 1) / ORACLE_PRICE_SCALE` in the first division, the second division can lose an additional unit, making `maxDebt` equal to `exact_floor - 1`.

**Concrete numerical example** (using simplified scale = 10 for clarity, same structure holds with 1e36/1e18):

- `C = 7`, `P = 7 * ORACLE_PRICE_SCALE / 10`, `lltv = 7 * WAD / 10`
- Step 1: `floor(7 * 7e35 / 1e36) = floor(4.9) = 4`
- Step 2: `floor(4 * 7e17 / 1e18) = floor(2.8) = 2`
- Exact single-step: `floor(7 * 7e35 * 7e17 / (1e36 * 1e18)) = floor(3.43) = 3`

If `debt = 3` after the take, `isHealthy` returns `2 >= 3 = false` → revert, but the position is truly healthy (3.43 ≥ 3).

**Attacker-controlled inputs:** The taker controls `units`, which directly sets `sellerDebtIncrease`. A taker (or the seller themselves acting as taker on a buy offer) can choose `units` such that the seller's resulting debt lands at exactly `exact_floor`, triggering the false negative.

**Why existing checks fail:** The Certora `Healthiness.spec` `stayHealthy` rule explicitly excludes `take` from its scope:

```
rule stayHealthy(...) filtered { 
    f -> f.selector != sig:take(...).selector 
    ...
}
```

No formal proof covers the case that a healthy seller remains healthy after a `take`. The `liquidate` function uses the same double-`mulDivDown` for its `originalDebt > maxDebt` check, so the position would also be liquidatable under the protocol's own accounting — the protocol is internally consistent but diverges from the true mathematical boundary by 1 unit.

### Impact Explanation
A legitimate `take` that would leave the seller's position exactly at the true mathematical health boundary reverts with `SellerIsLiquidatable()`. The seller cannot borrow up to their exact mathematical capacity; they are effectively limited to `computed_maxDebt` rather than `exact_floor`. This is a temporary, repeatable denial of service at the 1-unit boundary — no funds are lost, but the seller's borrowing is artificially capped by 1 loan-token unit.

### Likelihood Explanation
Requires the taker to choose `units` such that `sellerDebtIncrease` lands the seller's debt at exactly `exact_floor` while `computed_maxDebt = exact_floor - 1`. This requires `C * P` to be non-divisible by `ORACLE_PRICE_SCALE` (so step-1 rounding occurs) and the fractional loss to propagate through step 2. With arbitrary oracle prices and collateral amounts this is reachable; it is not a contrived overflow or impossible-oracle scenario. It is repeatable for any `(collateral, price, lltv)` triple satisfying the condition.

### Recommendation
Replace the two-step `mulDivDown` chain with a single combined-denominator computation:

```solidity
maxDebt += _position.collateral[i].mulDivDown(
    price * collateralParam.lltv,
    ORACLE_PRICE_SCALE * WAD
);
```

This eliminates the intermediate rounding step and makes the computed `maxDebt` equal to `floor(C * P * lltv / (ORACLE_PRICE_SCALE * WAD))`, matching the exact single-step floor. Overflow safety must be verified (price is bounded by the oracle, lltv ≤ WAD, collateral is uint128; the product `price * lltv` fits in uint256 under the existing overflow bounds proven in `NoMultiplicationOverflow.spec`).

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {UtilsLib} from "src/libraries/UtilsLib.sol";
import {ORACLE_PRICE_SCALE, WAD} from "src/libraries/ConstantsLib.sol";

contract RoundingFalseNegativeTest is Test {
    using UtilsLib for uint256;

    // Fuzz over (collateral, price, lltv, debtIncrease) near the health boundary.
    // Assert: if exact_floor >= debt, then computed_maxDebt >= debt (no false negative).
    function testFuzz_isHealthyNoFalseNegative(
        uint128 collateral,
        uint256 price,
        uint256 lltv,
        uint256 debt
    ) public pure {
        // Bound to realistic protocol ranges
        price = bound(price, 1, type(uint128).max); // oracle price bounded
        lltv  = bound(lltv,  1, WAD);               // lltv <= WAD
        vm.assume(collateral > 0);
        // Avoid overflow: collateral * price must fit uint256
        vm.assume(uint256(collateral) <= type(uint256).max / price);

        // Two-step computed maxDebt (protocol's actual computation)
        uint256 step1       = uint256(collateral).mulDivDown(price, ORACLE_PRICE_SCALE);
        uint256 computedMax = step1.mulDivDown(lltv, WAD);

        // Single-step exact floor
        // collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD)
        // Use 512-bit math or bound inputs to avoid overflow
        uint256 exactMax;
        unchecked {
            // Only valid when collateral * price * lltv doesn't overflow uint256
            if (price <= type(uint256).max / uint256(collateral) &&
                uint256(collateral) * price <= type(uint256).max / lltv) {
                exactMax = (uint256(collateral) * price * lltv) / (ORACLE_PRICE_SCALE * WAD);
            } else {
                return; // skip overflow cases
            }
        }

        // Bound debt to the exact boundary
        debt = bound(debt, 0, exactMax);

        // ASSERTION: if position is truly healthy (exactMax >= debt),
        // the protocol's computed maxDebt must also be >= debt.
        // A failure here is a false negative.
        if (exactMax >= debt) {
            assertGe(
                computedMax,
                debt,
                "FALSE NEGATIVE: isHealthy would return false for a truly healthy position"
            );
        }
    }
}
```

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];
                uint256 price = IOracle(collateralParam.oracle).price();
                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                    .mulDivDown(collateralParam.lltv, WAD);
                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
    }
