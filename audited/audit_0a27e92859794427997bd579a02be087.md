### Title
Authorized attacker creates victim debt with zero asset transfer via `mulDivDown` rounding on buy offer dust-take â€” (`src/Midnight.sol`)

### Summary
In `Midnight.take()`, for a buy offer, `sellerAssets` is computed as `units.mulDivDown(sellerPrice, WAD)`. When `units = 1` and `sellerPrice < WAD` (true for every tick below `MAX_TICK`), this rounds to zero. However, `sellerDebtIncrease` is computed independently from `units` and can be 1. No guard prevents `sellerDebtIncrease > 0` while `sellerAssets == 0`, so an authorized attacker can force the victim (taker/seller) to accrue debt while receiving zero loan-token assets.

### Finding Description

**Exact code path:**

`sellerAssets` is computed at line 364: [1](#0-0) 

For a buy offer (`offer.buy = true`):
- `sellerPrice = offerPrice - _settlementFee`
- `sellerAssets = units.mulDivDown(sellerPrice, WAD)`

When `units = 1` and `sellerPrice < WAD` (i.e., any tick below `MAX_TICK`), `floor(1 * sellerPrice / 1e18) = 0`.

`sellerDebtIncrease` is computed independently at line 384: [2](#0-1) 

If the victim has no credit, `sellerCreditDecrease = 0` and `sellerDebtIncrease = units = 1`.

The debt is written to the victim's position at line 414: [3](#0-2) 

The asset transfer to the victim at line 456 sends `sellerAssets = 0`: [4](#0-3) 

**Exploit flow (pre-consume variant, tick = MAX_TICK, fee > 0):**

1. Attacker is authorized for victim via `isAuthorized[victim][attacker] = true`.
2. Attacker creates a buy offer: `offer.maker = attacker`, `offer.buy = true`, `offer.tick = MAX_TICK`, `offer.maxAssets = N`, settlement fee > 0.
   - `buyerPrice = WAD`, so `buyerAssets = 1.mulDivDown(WAD, WAD) = 1`.
   - `sellerPrice = WAD - fee < WAD`, so `sellerAssets = 1.mulDivDown(WAD - fee, WAD) = 0`.
3. Attacker calls `setConsumed(group, N-1, attacker)` â€” valid since attacker calls on own behalf. [5](#0-4) 
4. Attacker calls `take(buyOffer, hex"", 1, victim, victim, address(0), hex"")`.
   - Authorization check passes: `isAuthorized[victim][attacker]` is true. [6](#0-5) 
   - `offer.maker != taker` passes: `attacker != victim`.
   - `newConsumed = (N-1) + buyerAssets = (N-1) + 1 = N â‰¤ N` â€” consumed check passes. [7](#0-6) 
   - `sellerDebtIncrease = 1`, `sellerAssets = 0`.
   - Victim's debt increases by 1; victim receives 0 tokens.

**Unlimited variant (tick = 0, fee = 0, `maxAssets > 0`):**

`tickToPrice(0) = 0` (proven by Certora rule `tickToPriceIsZeroAtZero` and `testTickToPriceMinMax`): [8](#0-7) 

With `sellerPrice = 0`, `buyerAssets = 0` per take. For `maxAssets > 0` mode, `consumed += buyerAssets = 0` â€” the consumed counter never advances: [7](#0-6) 

The attacker can call `take` with `units = 1` indefinitely, accumulating 1 unit of debt on the victim per call at zero cost to the attacker.

**Why the health check does not stop it:**

The post-take health check at line 476 only reverts if the victim becomes unhealthy: [9](#0-8) 

As long as the victim has sufficient collateral, each individual dust-take leaves them healthy. The attacker repeats until the accumulated debt degrades health to the liquidation threshold.

**Protocol's own note acknowledges the rounding gap:** [10](#0-9) 

This note covers the consumed-cap bypass but does not address the debt-without-assets invariant break.

### Impact Explanation
The victim (seller/taker) accrues 1 unit of debt per dust-take while receiving zero loan-token assets. The attacker (buyer/maker) receives 1 unit of credit per dust-take while paying zero assets (tick=0 variant) or 1 token (MAX_TICK variant). Repeated dust-takes degrade the victim's collateral health ratio until the position becomes liquidatable, at which point the victim's collateral can be seized. The attacker's free credit is redeemed against the victim's forced debt at maturity.

### Likelihood Explanation
**Preconditions:**
- Attacker must hold `isAuthorized[victim][attacker] = true`. This is a meaningful barrier but is routinely granted to smart-contract integrations (bundlers, routers, vaults). A malicious contract that requests authorization before revealing its behavior can satisfy this precondition.
- Victim must have collateral (otherwise the first dust-take makes them immediately unhealthy and reverts).
- Settlement fee must be > 0 for the MAX_TICK variant; no fee required for the tick=0 variant.

**Repeatability:** The tick=0 / `maxAssets > 0` variant is unlimited â€” consumed never advances. The MAX_TICK variant requires a new offer per attack cycle but is still repeatable.

### Recommendation
Add a guard that prevents debt creation when the corresponding asset transfer rounds to zero:

```solidity
require(
    sellerDebtIncrease == 0 || sellerAssets > 0,
    ZeroAssetDebtCreation()
);
```

Place this check after `sellerDebtIncrease` is computed (after line 384) and before position state is mutated. Alternatively, enforce a minimum `units` such that `units.mulDivDown(sellerPrice, WAD) >= 1` whenever `sellerDebtIncrease > 0`, or require `units >= WAD` when `sellerPrice > 0` to guarantee at least 1 wei of asset transfer per unit of debt.
