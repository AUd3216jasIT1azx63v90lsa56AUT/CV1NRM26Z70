Audit Report

## Title
Zero-asset rounding bypass allows unlimited takes on fully-consumed buy offers - (File: src/Midnight.sol)

## Summary
When a buy offer has `maxAssets > 0` and its tick is set such that `buyerPrice < WAD`, calling `take` with a small `units` value causes `mulDivDown(units, buyerPrice, WAD)` to round to zero. Because `consumed` is incremented by this zero value, the guard `newConsumed <= offer.maxAssets` never triggers even when the offer is already fully consumed. An unprivileged taker can repeat this indefinitely, inflating credit, debt, and `totalUnits` without transferring any tokens.

## Finding Description
**Code path** — `src/Midnight.sol`, lines 363 and 367–369:

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : units.mulDivUp(buyerPrice, WAD);
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause** — `consumed` is incremented by `buyerAssets` (an asset-denominated value), not by `units`. `buyerAssets` is computed with `mulDivDown`, which truncates. When `buyerPrice < WAD` (achievable at any tick below `MAX_TICK` by a sufficient margin, e.g. `MAX_TICK - 16 = 5804`) and `units = 1`, `mulDivDown(1, buyerPrice, WAD) = 0`. The increment is zero, so `newConsumed = consumed + 0 = consumed`. The guard `require(newConsumed <= offer.maxAssets)` passes even when `consumed == maxAssets`.

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = M`, `tick = MAX_TICK - 16` (so `buyerPrice < WAD`), `maxUnits = 0`.
2. Offer is fully consumed through normal fills until `consumed[maker][group] == M`.
3. Taker calls `take(offer, ..., units=1)`. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`. `newConsumed = M + 0 = M ≤ M` → passes. Position state mutates: `buyerPos.credit` increases, `sellerPos.debt` increases, `totalUnits` increases. Zero tokens are transferred.
4. Step 3 repeats indefinitely.

**Why existing checks fail** — The only guard against over-consumption is `require(newConsumed <= offer.maxAssets)`. There is no check requiring `units == 0` when `buyerAssets == 0`, and no strict `consumed < maxAssets` pre-check before the fill.

## Impact Explanation
Each zero-asset take permanently inflates the buyer's (maker's) credit, the seller's (taker's) debt, and `totalUnits` without any token movement. This corrupts the core accounting invariants of the protocol: credit/debt ledgers become permanently inflated, `totalUnits` diverges from actual economic exposure, and the offer replay protection is bypassed. This constitutes unauthorized state mutation and accounting integrity failure — a concrete in-scope impact.

## Likelihood Explanation
All three preconditions are reachable by any unprivileged user in normal protocol operation: (1) buy offers with `maxAssets > 0` are standard; (2) ticks below `MAX_TICK` by ≥16 steps are a normal, reachable range; (3) full consumption of an offer happens in ordinary use. No admin access, oracle manipulation, or special permissions are required. The attack is repeatable without limit once the offer is fully consumed.

## Recommendation
Add a strict pre-check before the consumed increment to reject takes that would produce zero asset movement on a `maxAssets`-bounded offer:

```solidity
if (offer.maxAssets > 0) {
    require(buyerAssets > 0 || units == 0, ZeroAssetTake());
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, change the consumed guard to `require(newConsumed < offer.maxAssets || units == 0)` so that a fully-consumed offer cannot be taken with any non-zero `units`, regardless of rounding.

## Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` (lines 858–889) is a complete, self-contained reproduction. It:
1. Sets `lenderOffer.tick = MAX_TICK - 16` to ensure `buyerPrice < WAD`.
2. Pre-sets `consumed` to `maxAssets` (simulating full consumption).
3. Calls `take(1, borrower, lenderOffer)`.
4. Asserts `buyerAssets == 0`, `consumed` unchanged, token balances unchanged, but `creditOf`, `debtOf`, and `totalUnits` all strictly increased.

        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        assertEq(loanToken.balanceOf(lender), lenderBalBefore);

        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
