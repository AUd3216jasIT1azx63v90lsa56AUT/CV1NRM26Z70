### Title
Two-step `mulDivDown` rounding in health check allows liquidation of truly-healthy positions - (`src/Midnight.sol`)

### Summary

`isHealthy` and `liquidate` both compute the maximum debt capacity as `collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD)`, applying two sequential floor divisions. This two-step floor can undercount the true capacity by exactly 1 unit compared to the mathematically correct single-step `floor(collateral * price * lltv / (ORACLE_PRICE_SCALE * WAD))`. When the oracle price lands in the narrow band where the two-step result is 1 less than the single-step result and the borrower's debt equals the single-step result, the position is truly healthy but the protocol treats it as liquidatable.

### Finding Description

**Code path:**

`isHealthy` at [1](#0-0)  and `liquidate` at [2](#0-1)  both compute:

```solidity
maxDebt += _collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);
```

`mulDivDown` is a plain integer floor division: [3](#0-2) 

The liquidation gate at [4](#0-3)  fires when `originalDebt > maxDebt`.

**Root cause — two-step floor loses up to 1 unit:**

Let `A = C * P`, `q = floor(A / S)`, `r = A mod S` (with `S = 1e36`, `W = 1e18`, `L = lltv`).

- Two-step result: `floor(q * L / W)`
- Single-step result: `floor(A * L / (S * W)) = floor(q*L/W + r*L/(S*W))`

The difference equals 1 exactly when `{q*L/W} + r*L/(S*W) >= 1`. Since `r < S` and `L <= W`, the term `r*L/(S*W) < 1`, so the maximum discrepancy is **exactly 1**.

**Concrete witness (LLTV_7 = 0.98e18):**

Choose `C = 1`, `L = 0.98e18`, `P = 51 * 1e36 + 2.1e34`:
- `q = floor((51e36 + 2.1e34) / 1e36) = 51`
- Two-step: `floor(51 * 0.98) = floor(49.98) = 49`
- Single-step: `floor((51e36 + 2.1e34) * 0.98 / 1e36) = floor(50.00058) = 50`

If `debt = 50`: two-step gives `maxDebt = 49`, so `originalDebt (50) > maxDebt (49)` → liquidation proceeds. Single-step gives `maxDebt = 50 >= debt` → position is truly healthy.

**How the borrower reaches `debt = 50`:** At a prior price `P' = 52e36`, both formulas give `maxDebt = floor(52 * 0.98) = 50`. The borrower borrows 50 units. The oracle then moves to `P = 51e36 + 2.1e34`. No existing check prevents the liquidation: `liquidationLocked` is a separate mechanism, and the `NotLiquidatable` guard fails because `50 > 49` is true.

### Impact Explanation

A liquidator monitoring oracle prices can call `liquidate()` the moment the oracle price enters the narrow band where the two-step formula underestimates by 1. The borrower loses collateral (amplified by the LIF) despite being genuinely solvent at the oracle price. This directly violates the core invariant: *healthy positions are not liquidatable*.

### Likelihood Explanation

The oracle price must land in a specific narrow band (width ≈ `S*W / (L * C)` price units) where `{q*L/W} + r*L/(S*W) >= 1`. This band exists for every allowed LLTV tier except `LLTV_8 = WAD` (where `L/W = 1` and the two-step collapses to a single step). With high-value collateral (`C` large), the band widens proportionally. The condition is passively observable — no attacker manipulation of the oracle is required — and repeatable across any market using a non-WAD LLTV.

### Recommendation

Replace the two sequential `mulDivDown` calls with a single fused operation that avoids the intermediate floor:

```solidity
// Instead of:
maxDebt += collateral.mulDivDown(price, ORACLE_PRICE_SCALE).mulDivDown(lltv, WAD);

// Use:
maxDebt += mulDivDown(collateral * price, lltv, ORACLE_PRICE_SCALE * WAD);
// or equivalently, using a 512-bit intermediate via a library such as PRBMath/FullMath:
maxDebt += FullMath.mulDiv(collateral, price * lltv, ORACLE_PRICE_SCALE * WAD);
```

This computes `floor(C * P * L / (S * W))` in one step, eliminating the intermediate floor that causes the 1-unit undercount. Note that `ORACLE_PRICE_SCALE * WAD = 1e54` fits in a `uint256` (< 2^256), but `collateral * price` may overflow `uint256` for large values, so a 512-bit multiply-then-divide (e.g., Solady's `FullMath.mulDiv`) is required.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {ORACLE_PRICE_SCALE, WAD, LLTV_7} from "src/libraries/ConstantsLib.sol";
// ... standard test setup imports

contract TwoStepRoundingPoC is Test {
    // Setup: single-collateral market with LLTV_7 = 0.98e18
    // Borrower deposits C = 1 unit of collateral
    // Oracle initially at P_initial = 52e36 (maxDebt_two_step = 50)
    // Borrower borrows debt = 50 units

    function testHealthyBorrowerLiquidated() public {
        // [setup market, deposit collateral=1, borrow 50 units at price 52e36]

        // Move oracle to the rounding-gap price
        uint256 gapPrice = 51 * ORACLE_PRICE_SCALE + 21e33; // 51e36 + 2.1e34

        // Verify two-step gives 49 (unhealthy by protocol)
        uint256 twoStep = (1 * gapPrice / ORACLE_PRICE_SCALE) * LLTV_7 / WAD;
        assertEq(twoStep, 49);

        // Verify single-step gives 50 (truly healthy)
        // (requires 512-bit math; use Python/chisel to confirm: 
        //  floor(1 * (51e36+2.1e34) * 0.98e18 / 1e54) == 50)

        oracle.setPrice(gapPrice);

        // isHealthy returns false despite true health
        assertFalse(midnight.isHealthy(market, id, borrower));

        // Liquidation succeeds — should revert with NotLiquidatable but does not
        // vm.expectRevert(IMidnight.NotLiquidatable.selector); // <-- this is what SHOULD happen
        midnight.liquidate(market, 0, 0, 1, borrower, false, address(this), address(0), "");
        // Assert borrower lost collateral despite being truly solvent
        assertLt(midnight.collateral(id, borrower, 0), 1, "borrower lost collateral unfairly");
    }
}
```

Expected assertion: `liquidate` should revert with `NotLiquidatable` but instead succeeds, seizing collateral from a truly-healthy borrower.
