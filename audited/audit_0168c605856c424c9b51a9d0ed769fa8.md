### Title
`seizedAssets` path computes `repaidUnits > _position.debt` with no cap, causing arithmetic underflow revert in post-maturity liquidations - (`src/Midnight.sol`)

### Summary
In `liquidate`, when `seizedAssets > 0`, `repaidUnits` is computed via two `mulDivUp` calls and then subtracted from `_position.debt` with no guard that `repaidUnits <= _position.debt`. The `maxRepaid` cap that exists in normal mode is entirely absent in post-maturity mode. When the oracle price exceeds `ORACLE_PRICE_SCALE * _position.debt / seizedAssets`, the computed `repaidUnits` exceeds the borrower's actual debt, and the Solidity 0.8 checked subtraction at line 676 reverts, permanently blocking the `seizedAssets` liquidation path for that position.

### Finding Description

**Exact code path:** [1](#0-0) 

```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
}
```

Both `mulDivUp` calls round up. With `seizedAssets = 1` and `price = 2 * ORACLE_PRICE_SCALE`:

```
step1 = mulDivUp(1, 2e36, 1e36) = 2
step2 = mulDivUp(2, WAD, WAD)   = 2   → repaidUnits = 2
```

If `_position.debt = 1`, the subtraction at line 676 underflows: [2](#0-1) 

The only guard against `repaidUnits > _position.debt` is the `maxRepaid` check, but it is gated on `!postMaturityMode`: [3](#0-2) 

In post-maturity mode that entire block is skipped. There is no equivalent cap for the `seizedAssets` path.

**Attacker-controlled inputs:**
- `seizedAssets = 1` (liquidator-supplied)
- `postMaturityMode = true` (liquidator-supplied)
- Oracle price is read from the oracle contract; prices above `ORACLE_PRICE_SCALE` are valid and tested in the existing test suite (e.g., `testLiquidateNormalModeRequiresUnhealthy` uses up to `10 * ORACLE_PRICE_SCALE`, `testLiquidatePostMaturityPartialLIF` uses up to `10 * ORACLE_PRICE_SCALE`) [4](#0-3) 

**Bad-debt block does not help:** With a high price, `badDebt = zeroFloorSub(debt, mulDivUp(collateral, price, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif)) = 0`, so `_position.debt` is unchanged before the subtraction. [5](#0-4) 

**Liquidatability check passes:** In post-maturity mode the only requirement is `block.timestamp > market.maturity`, regardless of health. [6](#0-5) 

**`UtilsLib.toUint128` does not help:** It only checks `repaidUnits <= type(uint128).max`; `repaidUnits = 2` passes that check, but `uint128(1) - uint128(2)` reverts under Solidity 0.8 checked arithmetic. [7](#0-6) 

### Impact Explanation

Any liquidator calling `liquidate` with `seizedAssets > 0` in post-maturity mode on a position where `price > ORACLE_PRICE_SCALE * _position.debt / seizedAssets` receives an arithmetic underflow revert. The `seizedAssets` liquidation path is completely blocked for that position under those oracle conditions. The `repaidUnits` path remains available but yields `seizedAssets = 0` (rounds down to zero collateral) when the price is high, making it economically irrational. This matches the scoped impact: "withdrawals/liquidations to revert unexpectedly."

### Likelihood Explanation

**Preconditions:**
1. Market is post-maturity — normal lifecycle event.
2. Borrower has small debt (e.g., 1 unit) — achievable by any borrower who partially repaid or was partially liquidated.
3. Oracle price > `ORACLE_PRICE_SCALE` — explicitly tested in the existing test suite as a valid price range; occurs naturally for high-value collateral assets.
4. Borrower has ≥ 1 unit of collateral at `collateralIndex` — required to pass the collateral subtraction at line 670.

All four preconditions are independently reachable without any privileged action. The condition is repeatable: every call to `liquidate` with `seizedAssets = 1` under these parameters reverts.

### Recommendation

Cap `repaidUnits` at `_position.debt` immediately after computing it in the `seizedAssets` branch, mirroring the intent of the `maxRepaid` guard:

```solidity
if (seizedAssets > 0) {
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
+   if (repaidUnits > _position.debt) repaidUnits = _position.debt;
}
```

Alternatively, apply `UtilsLib.min(repaidUnits, _position.debt)`. This is consistent with the semantics of the `seizedAssets` path: the liquidator specifies how much collateral to take, and the protocol computes the corresponding debt repayment, which cannot logically exceed the outstanding debt.

### Proof of Concept

```solidity
function testSeizedAssetsUnderflowPostMaturity() public {
    // Borrower borrows 1 unit.
    uint256 units = 1;
    collateralize(market, borrower, units);
    setupMarket(market, units);

    // Set oracle price to 2 * ORACLE_PRICE_SCALE (valid, tested range).
    Oracle(market.collateralParams[0].oracle).setPrice(2 * ORACLE_PRICE_SCALE);

    // Warp past maturity.
    vm.warp(market.maturity + 1);

    // seizedAssets = 1:
    //   repaidUnits = mulDivUp(1, 2e36, 1e36).mulDivUp(WAD, WAD) = 2
    //   _position.debt = 1
    //   1 - 2 → arithmetic underflow → revert
    vm.expectRevert(stdError.arithmeticError);
    midnight.liquidate(market, 0, 1, 0, borrower, true, address(this), address(0), "");

    // Assert: repaidUnits must never exceed _position.debt.
    // (This assertion is never reached due to the revert above.)
    // assertLe(repaidUnits, midnight.debtOf(id, borrower));
}
```

**Expected assertion:** `vm.expectRevert(stdError.arithmeticError)` passes, confirming the underflow. A fuzz variant should sweep `price` over `[ORACLE_PRICE_SCALE + 1, type(uint128).max]` with `units = 1`, `seizedAssets = 1`, `postMaturityMode = true`, and assert the call does not revert (which it currently does, proving the bug).
