### Title
Liquidation Reverts Due to Underflow When `repaidUnits` Exceeds Remaining Debt After Bad-Debt Socialization in Post-Maturity Mode — (`src/Midnight.sol`)

---

### Summary

In `liquidate()`, bad debt is socialized by reducing `_position.debt` using `maxLif` as the coverage denominator. However, `repaidUnits` is computed using the actual `lif`, which is strictly less than `maxLif` during the 15-minute ramp window after maturity. This mismatch causes `repaidUnits > _position.debt` for any liquidator who attempts to seize all (or too much) collateral, producing an arithmetic underflow revert at line 676 and permanently preventing full collateral seizure in that window.

---

### Finding Description

**Vulnerability type**: Accounting underflow (unsigned integer subtraction where computed amount exceeds remaining balance).

**Root cause — two-step debt reduction with mismatched LIF denominators:**

**Step 1 — Bad debt socialization uses `maxLif`** (lines 607–628):

```solidity
// src/Midnight.sol:614-616
badDebt = badDebt.zeroFloorSub(
    _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
);
```

After the loop, `badDebt` is subtracted from `_position.debt`:

```solidity
// src/Midnight.sol:628
_position.debt -= uint128(badDebt);
```

For a single-collateral borrower, the remaining debt after this step is:

```
_position.debt ≈ collateral × price / ORACLE_PRICE_SCALE × WAD / maxLif
```

**Step 2 — `repaidUnits` is computed using actual `lif`** (line 650):

```solidity
// src/Midnight.sol:650
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```

In post-maturity mode, `lif` ramps linearly from `WAD` (= 1) at maturity to `maxLif` over `TIME_TO_MAX_LIF = 15 minutes`:

```solidity
// src/Midnight.sol:645-647
uint256 lif = postMaturityMode
    ? UtilsLib.min(_maxLif, WAD + (_maxLif - WAD) * (block.timestamp - market.maturity) / TIME_TO_MAX_LIF)
    : _maxLif;
```

When `lif < maxLif` (i.e., within the first 15 minutes after maturity), and `seizedAssets = _position.collateral[collateralIndex]` (all collateral):

```
repaidUnits = collateral × price / ORACLE_PRICE_SCALE × WAD / lif
            > collateral × price / ORACLE_PRICE_SCALE × WAD / maxLif
            = _position.debt   ← remaining after bad-debt subtraction
```

**Step 3 — Underflow at line 676:**

```solidity
// src/Midnight.sol:676
_position.debt -= UtilsLib.toUint128(repaidUnits);  // REVERTS: repaidUnits > _position.debt
```

Solidity 0.8 checked arithmetic causes this to revert. There is no cap on `repaidUnits` relative to `_position.debt` in post-maturity mode — the RCF check at lines 655–667 is explicitly skipped (`if (!postMaturityMode)`). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

### Impact Explanation

**Primary — liquidation reverts**: Any liquidator who passes `seizedAssets` large enough that `repaidUnits > _position.debt` (after bad-debt subtraction) receives a revert. For a borrower with bad debt liquidated at or near maturity (lif ≈ 1), seizing even the full collateral amount triggers this.

**Secondary — borrower retains collateral despite bad debt**: The maximum safe `seizedAssets` is:

```
seizedAssets_max = _position.debt × lif / WAD × ORACLE_PRICE_SCALE / price
                 = collateral × lif / maxLif   (< collateral when lif < maxLif)
```

After a successful liquidation at this cap, the borrower's debt is zero and they retain:

```
collateral × (1 − lif/maxLif)
```

of collateral, which they can immediately withdraw via `withdrawCollateral`. For example, with LLTV = 0.86 and cursor = 0.5 (`maxLif ≈ 1.075`), a liquidation at maturity (lif = 1) leaves the borrower with ≈ 7% of their collateral despite having had bad debt that was already socialized onto lenders.

This violates the core invariant: *"liquidation must not leave accounting desynced between borrower debt, lender credit, protocol balances, and seized collateral."* Lenders absorb the full bad debt loss while the borrower escapes with residual collateral. [6](#0-5) 

---

### Likelihood Explanation

**Conditions required** (all realistic, no privileged access needed):

1. Market is past maturity (`block.timestamp > market.maturity`) — normal protocol lifecycle.
2. `block.timestamp < market.maturity + TIME_TO_MAX_LIF` — within the 15-minute ramp window.
3. Borrower has bad debt: `originalDebt > sum_i(collateral[i] × price[i] / ORACLE_PRICE_SCALE × WAD / maxLif[i])`.
4. Liquidator passes `seizedAssets` such that `repaidUnits > _position.debt` after bad-debt subtraction.

Condition 4 is the natural behavior of any liquidator trying to seize all collateral in one call. The 15-minute window is short but is precisely the window where liquidation incentives are lowest and bad-debt positions are most likely to be processed. Any liquidation bot that does not specifically account for this mismatch will fail. [5](#0-4) [7](#0-6) 

---

### Recommendation

Cap `repaidUnits` at the remaining `_position.debt` after bad-debt subtraction, before the collateral and debt state mutations:

```solidity
// After line 653, before line 655:
repaidUnits = UtilsLib.min(repaidUnits, _position.debt);
// Recompute seizedAssets consistently if repaidUnits was capped:
if (seizedAssets > 0) {
    seizedAssets = UtilsLib.min(seizedAssets, _position.collateral[collateralIndex]);
}
```

This mirrors the fix applied in the analogous SYMM-IO report (capping the deduction to the available balance) and prevents the underflow while preserving the liquidation incentive semantics. [8](#0-7)

### Proof of Concept

**Setup** (LLTV = 0.86, cursor = 0.5 → `maxLif ≈ 1.0753e18`):

```
collateral = 1000 tokens, price = ORACLE_PRICE_SCALE (1:1 with loan token)
debt       = 1000 loan tokens
```

**At maturity** (`block.timestamp = market.maturity`, `lif = WAD = 1e18`):

1. Bad debt loop computes:
   `badDebt = 1000 − 1000 × 1e36/1e36 × 1e18/1.0753e18 ≈ 1000 − 930 = 70`

2. `_position.debt = 1000 − 70 = 930`

3. Liquidator calls `liquidate(..., seizedAssets=1000, repaidUnits=0, postMaturityMode=true)`:
   `repaidUnits = 1000 × 1e36/1e36 × 1e18/1e18 = 1000`

4. Line 676: `930 − 1000` → **underflow revert**.

5. Liquidator retries with `seizedAssets = 930 × 1e18/1e18 × 1e36/1e36 = 930`:
   `repaidUnits = 930 × 1e36/1e36 × 1e18/1e18 = 930 = _position.debt` → succeeds.
   Borrower retains `1000 − 930 = 70` tokens of collateral with zero debt → withdraws freely. [9](#0-8) [10](#0-9)
