### Title
Zero-unit take on fully-consumed buy offer invokes maker's `IBuyCallback.onBuy` with zero assets — (`File: src/Midnight.sol`)

### Summary
When `offer.maxAssets > 0` and `offer.buy == true`, the `ConsumedAssets` guard uses a post-increment `<=` check. Because adding `buyerAssets = 0` (from `units = 0`) leaves `consumed` unchanged at `maxAssets`, the check `maxAssets <= maxAssets` passes, and execution continues to invoke the maker's `IBuyCallback.onBuy` with `buyerAssets = 0` and `units = 0`. No existing check prevents this on a fully-consumed offer.

### Finding Description

**Root cause — `src/Midnight.sol` lines 363–369 and 445–453:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // units=0 → buyerAssets=0
    : units.mulDivUp(buyerPrice, WAD);

uint256 newConsumed;
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    // consumed += 0  →  newConsumed == maxAssets
    require(newConsumed <= offer.maxAssets, ConsumedAssets());   // maxAssets <= maxAssets → PASSES
}
``` [1](#0-0) 

Then, unconditionally:

```solidity
if (buyerCallback != address(0)) {
    require(
        IBuyCallback(buyerCallback)
            .onBuy(id, offer.market, buyerAssets, units, buyerPendingFeeIncrease, buyer, buyerCallbackData)
        == CALLBACK_SUCCESS, ...
    );
}
``` [2](#0-1) 

For `offer.buy == true`, `buyerCallback = offer.callback` (line 420), so the **maker's** registered callback is invoked. [3](#0-2) 

**Exploit flow:**

1. Maker creates a buy offer: `offer.buy = true`, `offer.maxAssets = M > 0`, `offer.callback = <MakerCallbackContract>`.
2. Legitimate taker fills it exactly: `take(offer, ..., units=U)` where `buyerAssets = M`. Now `consumed[maker][group] = M`.
3. Attacker (any address ≠ maker) calls `take(offer, ..., units=0)`.
   - `buyerAssets = 0`
   - `newConsumed = M + 0 = M ≤ M` → `ConsumedAssets` check passes
   - All position-delta checks pass trivially (`buyerCreditIncrease = 0`, `sellerDebtIncrease = 0`)
   - `IBuyCallback(offer.callback).onBuy(..., buyerAssets=0, units=0, ...)` is called

**Why existing checks fail:**

- No `require(units > 0)` guard anywhere in `take`.
- The `ConsumedAssets` check is `newConsumed <= maxAssets`, not `consumed_before < maxAssets`.
- The Certora spec `fullyConsumedOfferRevertsOnNonTrivialTake` (line 99–111) only covers the `maxUnits` branch (`require offer.maxAssets == 0`); there is no equivalent invariant for the `maxAssets` branch. [4](#0-3) 

The analogous spec `takeConsumedAtMaxUnchangedAssets` (line 88–97) only asserts that `consumed` does not change — it does not assert that callbacks are not invoked. [5](#0-4) 

### Impact Explanation
An unprivileged taker can invoke the maker's `IBuyCallback.onBuy` with `buyerAssets = 0` and `units = 0` on an already fully-consumed buy offer, an unlimited number of times. Maker callback contracts that track fills, update internal accounting, emit events, or execute conditional logic on each `onBuy` invocation will process a spurious zero-asset callback they cannot distinguish from a legitimate one. This violates the invariant that a fully-consumed offer must not allow any further interaction.

### Likelihood Explanation
Preconditions are low-friction: any buy offer with `maxAssets > 0` and a non-zero `offer.callback` is vulnerable once fully consumed. The attacker needs no special privilege — only the ability to call `take` as any address that is not the maker. The attack is repeatable indefinitely at the cost of gas only.

### Recommendation
Replace the post-increment `<=` check with a pre-increment strict `<` check in the `maxAssets` branch:

```solidity
if (offer.maxAssets > 0) {
    uint256 delta = offer.buy ? buyerAssets : sellerAssets;
    require(consumed[offer.maker][offer.group] < offer.maxAssets, ConsumedAssets()); // strict pre-check
    newConsumed = consumed[offer.maker][offer.group] += delta;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

This causes any take on a fully-consumed offer (including `units = 0`) to revert, matching the behaviour already enforced in the `maxUnits` branch and closing the Certora spec gap. Alternatively, add a symmetric Certora rule for the `maxAssets` case mirroring `fullyConsumedOfferRevertsOnNonTrivialTake`.

### Proof of Concept

```solidity
function testZeroUnitTakeOnFullyConsumedBuyOfferCallsCallback() public {
    // Setup: buy offer with maxAssets and a tracking callback
    TrackingBuyCallback cb = new TrackingBuyCallback();
    lenderOffer.buy = true;
    lenderOffer.maxUnits = 0;
    lenderOffer.maxAssets = expectedBuyerAssets; // exact fill amount
    lenderOffer.callback = address(cb);
    lenderOffer.tick = MAX_TICK;

    // Step 1: legitimate full fill
    deal(address(loanToken), address(cb), expectedBuyerAssets);
    collateralize(market, borrower, units);
    take(units, borrower, lenderOffer); // consumed == maxAssets now

    uint256 callCountBefore = cb.callCount();

    // Step 2: attacker calls take with units=0 on fully-consumed offer
    vm.prank(attacker);
    midnight.take(lenderOffer, hex"", 0, attacker, attacker, address(0), hex"");

    // Assertion: callback must NOT have been invoked again
    assertEq(cb.callCount(), callCountBefore, "onBuy must not fire on fully-consumed offer");
}
```

`TrackingBuyCallback.onBuy` increments a counter and returns `CALLBACK_SUCCESS`. The test asserts the counter is unchanged after the zero-unit take. Under the current code the assertion fails, confirming the bug.

/// If consumed is already at or above maxAssets before a take in assets mode, it remains unchanged.
rule takeConsumedAtMaxUnchangedAssets(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxUnits == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumedBefore >= offer.maxAssets => consumed(offer.maker, offer.group) == consumedBefore;
}

/// A fully-consumed offer in units mode only allows no-op takes.
rule fullyConsumedOfferRevertsOnNonTrivialTake(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxAssets == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    require offer.maxUnits > 0 && consumedBefore >= offer.maxUnits, "assume the offer is fully consumed";

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    // If take does not revert, its input has to be zero.
    assert units == 0;
}
