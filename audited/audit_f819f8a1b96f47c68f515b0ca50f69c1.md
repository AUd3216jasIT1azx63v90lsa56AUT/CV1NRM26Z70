### Title
Authorized Accounts Can Create Persistent Sub-Authorizations That Survive Revocation, Enabling Position Drainage After Trust Is Revoked — (`src/Midnight.sol`)

---

### Summary

`setIsAuthorized` in `Midnight.sol` allows any currently-authorized account to grant further authorizations on behalf of the user. When the user later revokes the original authorization, all sub-authorizations created by that account remain active. A malicious authorized contract can silently plant a persistent backdoor before being revoked, then drain the victim's credit and collateral at any time.

---

### Finding Description

**Root cause — `setIsAuthorized` has no delegation depth limit:** [1](#0-0) 

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
}
```

The only check is that `msg.sender` is currently authorized for `onBehalf`. There is no restriction preventing an authorized account from granting further authorizations. The protocol itself acknowledges this in its developer comments: [2](#0-1) 

> "In particular, authorized accounts can authorize other accounts on behalf of the user."

**Exploit flow:**

1. User A authorizes a DApp contract B: `setIsAuthorized(B, true, A)` → `isAuthorized[A][B] = true`
2. B (malicious) atomically calls `setIsAuthorized(Attacker, true, A)` → `isAuthorized[A][Attacker] = true`
3. User A revokes B: `setIsAuthorized(B, false, A)` → `isAuthorized[A][B] = false`
4. `isAuthorized[A][Attacker]` **remains `true`** — the sub-authorization is invisible to A and survives revocation
5. Attacker calls `withdraw(market, A.credit, A, attacker_address)` → drains all credit
6. Attacker calls `withdrawCollateral(market, index, A.collateral, A, attacker_address)` → drains all collateral [3](#0-2) [4](#0-3) 

Both `withdraw` and `withdrawCollateral` only check `isAuthorized[onBehalf][msg.sender]`, so the attacker passes freely.

The same persistent-state issue extends to `SetterRatifier.setIsRootRatified` and `EcrecoverRatifier.cancelRoot`, both of which also accept calls from any `isAuthorized` account on behalf of the maker: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

A malicious authorized contract can silently plant a hidden address with full position control. After the user revokes the contract, the hidden address retains the ability to:

- Drain all credit via `withdraw`
- Drain all collateral via `withdrawCollateral` (if debt is zero or after repayment)
- Grant further authorizations via `setIsAuthorized`
- Cancel or ratify offer roots via `EcrecoverRatifier`/`SetterRatifier`

This is a **complete, irreversible loss of all user funds** with no on-chain signal to the victim.

---

### Likelihood Explanation

Authorizing a third-party contract is the standard DeFi interaction pattern (e.g., using `MidnightBundles`). A user who authorizes any contract — even a legitimate-looking one — is exposed if that contract is malicious or upgradeable. The sub-authorization is a single silent call that can be embedded in any callback or bundled transaction. The victim has no practical way to enumerate all `isAuthorized` entries set on their behalf. [7](#0-6) 

The `isAuthorized` mapping is not enumerable, so a user cannot audit what sub-authorizations exist.

---

### Recommendation

Restrict `setIsAuthorized` so that only the direct account owner (`onBehalf == msg.sender`) can grant new authorizations. Authorized accounts should be able to act on behalf of the user for position operations, but should not be able to extend the authorization graph:

```solidity
function setIsAuthorized(address authorized, bool newIsAuthorized, address onBehalf) external {
    require(onBehalf == msg.sender, Unauthorized()); // remove delegation of authorization
    isAuthorized[onBehalf][authorized] = newIsAuthorized;
    emit EventsLib.SetIsAuthorized(msg.sender, authorized, newIsAuthorized, onBehalf);
}
```

If delegation of authorization is intentional, add a depth flag (e.g., `canDelegate`) so users can grant limited vs. full authorizations.

---

### Proof of Concept

```
Preconditions:
  - User A has credit and collateral in market M
  - Attacker deploys MaliciousContract B

Step 1: A calls Midnight.setIsAuthorized(B, true, A)
        → isAuthorized[A][B] = true

Step 2: B (in its constructor or any function) calls:
        Midnight.setIsAuthorized(AttackerEOA, true, A)
        → isAuthorized[A][AttackerEOA] = true
        (This call succeeds because isAuthorized[A][B] == true)

Step 3: A calls Midnight.setIsAuthorized(B, false, A)
        → isAuthorized[A][B] = false
        → isAuthorized[A][AttackerEOA] = true  ← PERSISTS

Step 4: AttackerEOA calls:
        Midnight.withdraw(M, A.credit, A, AttackerEOA)
        → succeeds: isAuthorized[A][AttackerEOA] == true
        → A loses all credit

Step 5: AttackerEOA calls:
        Midnight.withdrawCollateral(M, 0, A.collateral[0], A, AttackerEOA)
        → succeeds (no debt check if debt == 0)
        → A loses all collateral

Result: A's position is fully drained. A has no recourse.
```
