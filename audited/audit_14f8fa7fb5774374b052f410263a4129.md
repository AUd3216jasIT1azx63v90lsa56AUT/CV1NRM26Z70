### Title
Bad debt realized before liquidation callback allows borrower to slash lenders while retaining collateral - (File: src/Midnight.sol)

### Summary
In `Midnight.liquidate`, bad debt is computed from a local bitmap snapshot and immediately committed to storage (reducing `_position.debt`, updating `lossFactor`, reducing `totalUnits`) before the `onLiquidate` callback is invoked. Because there is no reentrancy guard, the callback can call `supplyCollateral` on behalf of the borrower to deposit collateral at a new index k, collateral that — had it been deposited before the call — would have prevented bad debt realization entirely. Lenders are slashed for a bad debt that the borrower had the means to avoid.

### Finding Description
**Code path:**

`liquidate()` at `src/Midnight.sol`:

- Line 606: `uint128 _collateralBitmap = _position.collateralBitmap;` — snapshot of storage bitmap into a local variable. Collateral index k is absent.
- Lines 607–618: while-loop over the local `_collateralBitmap` computes `maxDebt` and `badDebt` without k.
- Lines 620–624: `require(!liquidationLocked(id, borrower) && originalDebt > maxDebt)` — passes because k is excluded.
- Lines 626–641: `badDebt > 0` branch executes — `_position.debt -= uint128(badDebt)`, `_marketState.lossFactor` updated, `_marketState.totalUnits` reduced. **All written to storage.**
- Lines 643–677: skipped entirely when `seizedAssets == 0 && repaidUnits == 0`.
- Line 696: `safeTransfer(collateralToken, receiver, 0)` — no-op.
- Lines 698–715: `ILiquidateCallback(callback).onLiquidate(...)` — callback fires.
  - Inside callback: `midnight.supplyCollateral(market, k, largeAmount, borrower)` executes successfully because `isAuthorized[borrower][callbackContract] == true` (set up beforehand).
  - `_position.collateral[k]` updated, bit k set in `_position.collateralBitmap`.
- Line 717: `safeTransferFrom(loanToken, payer, address(this), 0)` — no-op.

**Root cause:** Bad debt realization (lines 626–641) is ordered before the external callback (lines 698–715), and no reentrancy guard or state lock prevents `supplyCollateral` from being called during the callback window.

**Attacker-controlled inputs:**
- `seizedAssets = 0`, `repaidUnits = 0` (explicitly supported per the NatSpec at line 577)
- `callback = callbackContract` (borrower-controlled contract)
- `data` = arbitrary

**Preconditions:**
1. Borrower calls `setIsAuthorized(callbackContract, true, borrower)` before the attack.
2. Borrower funds `callbackContract` with collateral k tokens.
3. Borrower's position is unhealthy (`originalDebt > maxDebt` over existing collateral C, excluding k).
4. `value(k) / maxLif >= badDebt` — i.e., if k had been deposited, `badDebt` would have been zero.

**Why existing checks fail:**
- `liquidationLocked` only blocks liquidation during `take` callbacks; it does not block `supplyCollateral`.
- No `nonReentrant` modifier exists on `liquidate` or `supplyCollateral`.
- The authorization check in `supplyCollateral` is satisfied by the pre-set `isAuthorized` mapping.
- The `atMostOneNonZero(0, 0)` check passes (both zero is explicitly allowed).

### Impact Explanation
Bad debt is realized and `lossFactor` is increased — slashing all lenders proportionally — even though the borrower possessed sufficient collateral (k) to cover the debt. The borrower's debt is reduced by `badDebt` at lenders' expense, while the borrower retains collateral k in the protocol (deposited during the callback). The borrower effectively converts a debt obligation into a lender loss without spending loan tokens.

### Likelihood Explanation
Requires the borrower to control the callback contract and pre-authorize it. No oracle manipulation, admin access, or impossible conditions are needed. The borrower must have an unhealthy position and hold collateral k tokens externally. This is a deliberate, repeatable, single-transaction exploit available to any borrower in any market with `liquidatorGate == address(0)`.

### Recommendation
Move bad debt realization to **after** the callback, or add a reentrancy guard (transient storage lock) that prevents `supplyCollateral` (and other position-mutating functions) from being called while `liquidate` is executing. Alternatively, re-read `_position.collateralBitmap` from storage after the callback and re-evaluate `badDebt` before committing it. The simplest fix is a transient-storage reentrancy lock set at the top of `liquidate` and cleared at the bottom, blocking re-entrant calls to `supplyCollateral`.
