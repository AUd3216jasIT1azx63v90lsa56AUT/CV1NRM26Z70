### Title
`mulDivUp` Arithmetic Overflow in `consumableUnits` for Large-`maxAssets` Buy Offers - (`File: src/periphery/ConsumableUnitsLib.sol`)

### Summary
`UtilsLib.mulDivUp` computes `(x * y + (d - 1)) / d` in checked Solidity 0.8.x arithmetic with no `unchecked` block. When `consumableUnits` is called for a buy offer with `maxUnits == 0` and `maxAssets > type(uint256).max / WAD` (≈ 1.157e59), the intermediate product `targetBuyerAssets * WAD` overflows `uint256` and the call reverts. Because `maxAssets` is an unconstrained `uint256` in the `Offer` struct and the core `take` function imposes no upper bound on it, any maker can craft such an offer, permanently DoS-ing `consumableUnits` for that offer.

### Finding Description

**Code path:**

1. `ConsumableUnitsLib.consumableUnits` (line 18–19): with `offer.maxUnits == 0` and `offer.buy == true`, calls:
   ```solidity
   TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
   ```
   With `consumed == 0`, `targetBuyerAssets = offer.maxAssets`.

2. `TakeAmountsLib.buyerAssetsToUnits` (line 29): since `offer.buy == true`, calls:
   ```solidity
   targetBuyerAssets.mulDivUp(WAD, buyerPrice)
   ```

3. `UtilsLib.mulDivUp` (lines 34–36):
   ```solidity
   function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
       return (x * y + (d - 1)) / d;
   }
   ```
   There is **no `unchecked` block**. In Solidity `^0.8.0`, `x * y` reverts on overflow.

**Root cause:** `mulDivUp` uses naïve checked multiplication. When `x = targetBuyerAssets > type(uint256).max / 1e18 ≈ 1.157e59`, the product `x * WAD` overflows `uint256` and the EVM reverts.

**Attacker-controlled inputs:**
- `offer.maxAssets` is `uint256` with no upper-bound validation anywhere in `Midnight.sol`'s `take` path (lines 367–369 only check `newConsumed <= offer.maxAssets`, not the magnitude of `maxAssets` itself).
- `offer.maxUnits` set to `0` (valid per the `atMostOneNonZero` check).
- `offer.buy` set to `true`.
- `offer.tick` set to any valid tick (e.g., `MAX_TICK = 5820`) giving `buyerPrice ≥ PRICE_ROUNDING_STEP = 1e12`.

**Why existing checks fail:**
- `require(buyerPrice <= WAD, ...)` (line 28 of `TakeAmountsLib`) only bounds `buyerPrice` from above; it does not bound `targetBuyerAssets`.
- `zeroFloorSub` (line 19 of `ConsumableUnitsLib`) only floors at zero; it does not cap `targetBuyerAssets`.
- The `MultipleNonZero` check in `Midnight.sol` only prevents both `maxUnits > 0` and `maxAssets > 0` simultaneously; it does not constrain the magnitude of `maxAssets`.

### Impact Explanation
Any call to `consumableUnits` for a buy offer where `offer.maxAssets > type(uint256).max / 1e18` reverts unconditionally. Every periphery contract or off-chain integration that relies on `consumableUnits` to compute fill amounts for such an offer is permanently broken for that offer. The core `Midnight.take` is unaffected (it never calls `consumableUnits`), but the periphery helper is fully DoS-ed for the affected offer class.

### Likelihood Explanation
A maker needs only to sign an offer with `buy = true`, `maxUnits = 0`, and `maxAssets` set to any value above `≈ 1.157e59` (e.g., `type(uint256).max`). No special privilege, no oracle manipulation, and no existing state is required. The condition is trivially reachable on first call with zero prior consumption. It is repeatable for every such offer.

### Recommendation
Replace the naïve `mulDivUp` in `UtilsLib.sol` with an overflow-safe implementation (e.g., using `mulmod` for the rounding correction, as in Solmate's `FixedPointMathLib`):

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256 z) {
    assembly {
        // Revert if d == 0 or if the result overflows uint256.
        if iszero(mul(d, iszero(iszero(x)))) { revert(0, 0) }
        z := add(div(mul(x, y), d), gt(mulmod(x, y, d), 0))
    }
}
```

Alternatively, add a pre-condition guard in `buyerAssetsToUnits` before calling `mulDivUp`:

```solidity
require(targetBuyerAssets <= type(uint256).max / WAD, "overflow");
```

The same fix should be applied to `mulDivDown` for consistency.

```solidity
import "forge-std/Test.sol";
import {ConsumableUnitsLib} from "src/periphery/ConsumableUnitsLib.sol";
import {TakeAmountsLib} from "src/periphery/TakeAmountsLib.sol";
import {UtilsLib} from "src/libraries/UtilsLib.sol";
import {WAD} from "src/libraries/ConstantsLib.sol";
import {MAX_TICK} from "src/libraries/TickLib.sol";
import {Offer} from "src/interfaces/IMidnight.sol";

contract MulDivUpOverflowTest is Test {
    // Fuzz: any maxAssets above the overflow threshold causes revert
    function testFuzz_consumableUnitsOverflow(uint256 maxAssets) public {
        // Overflow threshold: type(uint256).max / WAD
        vm.assume(maxAssets > type(uint256).max / WAD);

        Offer memory offer;
        offer.buy = true;
        offer.maxUnits = 0;
        offer.maxAssets = maxAssets;
        offer.tick = MAX_TICK; // small but valid buyerPrice >= PRICE_ROUNDING_STEP

        // Direct call to mulDivUp to confirm overflow
        // targetBuyerAssets = maxAssets (consumed = 0)
        // mulDivUp(maxAssets, WAD, buyerPrice) must NOT revert for valid offer params
        vm.expectRevert(); // asserts the invariant is violated: it DOES revert
        UtilsLib.mulDivUp(maxAssets, WAD, 1e12 /* min buyerPrice */);
    }

    // Unit: exact boundary
    function test_mulDivUpOverflowBoundary() public {
        uint256 threshold = type(uint256).max / WAD;
        // At threshold: should not overflow
        UtilsLib.mulDivUp(threshold, WAD, 1e12); // passes

        // One above threshold: overflows
        vm.expectRevert();
        UtilsLib.mulDivUp(threshold + 1, WAD, 1e12);
    }
}
```
