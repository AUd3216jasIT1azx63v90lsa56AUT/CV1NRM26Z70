### Title
`sellerAssetsToUnits` divides by zero when `offer.buy=true` and `offerPrice==settlementFee`, causing `supplyCollateralAndSellWithAssetsTarget` to revert uncaught - (`File: src/periphery/TakeAmountsLib.sol`)

### Summary

When a buy offer exists where `offerPrice == settlementFee`, `sellerPrice` computes to exactly zero. `sellerAssetsToUnits` then calls `mulDivUp(targetSellerAssets, WAD, 0)`, which triggers an arithmetic underflow on `d-1` (Solidity 0.8 checked math) and reverts. This revert occurs outside the `try/catch` block in `supplyCollateralAndSellWithAssetsTarget`, causing the entire bundler call to revert rather than skipping the offer.

### Finding Description

**Root cause — `sellerAssetsToUnits`:** [1](#0-0) 

```
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...
```

When `offer.buy = true` and `offerPrice == settlementFee`, `sellerPrice = 0`. The call becomes `mulDivUp(targetSellerAssets, WAD, 0)`.

**`mulDivUp` implementation:** [2](#0-1) 

```solidity
function mulDivUp(uint256 x, uint256 y, uint256 d) internal pure returns (uint256) {
    return (x * y + (d - 1)) / d;
}
```

With `d = 0`, the expression `d - 1` underflows (Solidity 0.8 checked arithmetic), reverting with `stdError.arithmeticError`. The existing test suite confirms this exact behavior: [3](#0-2) 

**Uncaught revert path in `supplyCollateralAndSellWithAssetsTarget`:** [4](#0-3) 

The `sellerAssetsToUnits` call at line 286 is evaluated as an argument to `min(...)` **before** the `try` block at line 292. The `try/catch` only wraps `IMidnight(MIDNIGHT).take(...)`. The revert from `sellerAssetsToUnits` propagates uncaught, reverting the entire function.

**Precondition reachability:**

The settlement fee is a piecewise-linear function of time-to-maturity, set by the `feeSetter` admin across 7 breakpoints: [5](#0-4) 

A maker creates a buy offer at a tick where `tickToPrice(tick)` equals the current settlement fee. This is a valid offer — Midnight's own `take` only reverts when `offerPrice < settlementFee` (underflow on `offerPrice - settlementFee`); the equality case `offerPrice == settlementFee` passes Midnight's checks (sellerAssets = 0 is valid). Alternatively, the admin raises the settlement fee post-offer-creation to match an existing offer's price — a normal operational action.

**`ConsumableUnitsLib.consumableUnits` is not the direct trigger:** [6](#0-5) 

When `offer.buy = true`, `consumableUnits` calls `buyerAssetsToUnits` (not `sellerAssetsToUnits`), so `consumableUnits` itself does not revert. The revert originates from the direct `sellerAssetsToUnits` call at `MidnightBundles.sol:286`.

### Impact Explanation

Any call to `supplyCollateralAndSellWithAssetsTarget` that includes a buy offer where `offerPrice == settlementFee` reverts entirely. The bundler cannot skip the offer (no try/catch around `sellerAssetsToUnits`). Borrowers relying on this bundler function to sell units and manage their debt position are blocked for the duration that the condition holds. If the settlement fee is raised to match multiple outstanding offers simultaneously, the bundler is bricked for all of them at once.

### Likelihood Explanation

The condition `offerPrice == settlementFee` is reachable in two ways: (1) a maker deliberately places a buy offer at the tick whose price equals the current settlement fee; (2) the `feeSetter` raises the settlement fee to a value that coincidentally equals an existing offer's price. The settlement fee is a piecewise-linear interpolation over discrete `CBP`-granularity values, and tick prices are also discrete — exact equality is achievable. The condition is transient (it resolves as time-to-maturity changes the interpolated fee), but it can be re-triggered repeatedly.

### Recommendation

Guard against `sellerPrice == 0` in `sellerAssetsToUnits` before dividing. When `sellerPrice == 0` and `offer.buy == true`, the offer yields zero seller assets for any number of units, so no finite unit count can satisfy a positive `targetSellerAssets`. Return `type(uint256).max` in this case so that `min(type(uint256).max, takes[i].units, consumableUnits(...))` collapses to the offer's own unit cap and the offer is skipped naturally:

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
if (offer.buy && sellerPrice == 0) return type(uint256).max;
return offer.buy
    ? targetSellerAssets.mulDivUp(WAD, sellerPrice)
    : targetSellerAssets.mulDivDown(WAD, sellerPrice);
```
