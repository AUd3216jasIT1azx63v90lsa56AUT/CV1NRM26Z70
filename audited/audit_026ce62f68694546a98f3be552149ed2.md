### Title
`consumableUnits` returns non-zero for post-maturity sell offers that always revert in `take`, causing bundles to revert with `OutOfOffers` - (`src/periphery/ConsumableUnitsLib.sol`)

### Summary
`ConsumableUnitsLib.consumableUnits` only checks the consumed-vs-cap accounting to determine if an offer can be taken, but does not check whether the offer maker has sufficient credit to satisfy the `CannotIncreaseDebtPostMaturity` guard in `Midnight.take`. Post-maturity, a sell offer whose maker has zero credit will always revert in `take`, yet `consumableUnits` returns a non-zero value. The bundle's `try/catch` silently swallows the revert, `filledUnits` is never incremented, and the bundle terminates with `OutOfOffers`.

### Finding Description

**Code path — `consumableUnits` for a sell offer with `maxAssets > 0`:**

`consumableUnits` dispatches to `sellerAssetsToUnits` for sell offers: [1](#0-0) 

`sellerAssetsToUnits` computes TTM as `zeroFloorSub(maturity, block.timestamp)` = 0 post-maturity, fetches `settlementFee(id, 0)` = `settlementFeeCbp0 * CBP`, and for a sell offer uses `sellerPrice = offerPrice` (settlement fee does not enter the sell-side price), returning: [2](#0-1) 

This is non-zero whenever `offer.maxAssets > consumed[maker][group]`. No credit check is performed.

**Code path — `take` post-maturity:**

`take` computes: [3](#0-2) 

and then enforces: [4](#0-3) 

For a sell offer, `seller = offer.maker`. If the maker has zero credit, `sellerCreditDecrease = 0` and `sellerDebtIncrease = units`. Post-maturity this check reverts unconditionally with `CannotIncreaseDebtPostMaturity`.

**Bundle execution:** [5](#0-4) 

`consumableUnits` returns non-zero → `unitsToTake > 0` → `take` reverts → `catch {}` swallows → `filledUnits` unchanged → `require(filledUnits == targetUnits, OutOfOffers())` reverts.

**Root cause:** `consumableUnits` is documented as returning "a number of units such that it fully consumes the offer," but it only validates the consumed-vs-cap invariant. It does not validate the maker's credit against the `CannotIncreaseDebtPostMaturity` guard, creating a permanent inconsistency post-maturity for any sell offer whose maker has insufficient credit.

**Attacker inputs:** The maker creates a sell offer with `offer.buy = false`, `maxAssets > 0`, and zero prior credit. No special privilege is required. After maturity passes, the offer is permanently untakeable but permanently appears consumable.

### Impact Explanation
Any call to `buyWithUnitsTargetAndWithdrawCollateral` (or `buyWithAssetsTargetAndWithdrawCollateral`) that includes such an offer in its `takes[]` array will have the offer silently skipped by the `catch {}` block. If the skipped offer was needed to reach `targetUnits`, the bundle reverts with `OutOfOffers`, wasting the taker's gas and blocking their fill. A malicious maker can deliberately engineer this state to grief any taker whose routing logic relies on `consumableUnits` to pre-filter offers.

### Likelihood Explanation
Preconditions are reachable without privilege: (1) a sell offer exists with `maxAssets > consumed`, (2) `block.timestamp > offer.market.maturity`, (3) the maker's on-chain credit is less than the units being taken. This arises naturally (maker withdrew credit before maturity, or never had credit) and can be deliberately engineered. The condition is permanent and repeatable for the lifetime of the offer post-maturity.

### Recommendation
In `consumableUnits`, add a post-maturity credit cap for sell offers. After computing the candidate units from the assets/units cap, if `block.timestamp > offer.market.maturity` and `offer.buy == false`, fetch the maker's current credit via `updatePositionView` (to account for slashing and fee accrual) and cap the returned units at that credit value. If the maker's credit is zero, return 0.

```solidity
// After computing candidateUnits for a sell offer:
if (!offer.buy && block.timestamp > offer.market.maturity) {
    (uint128 makerCredit,,) = IMidnight(midnight).updatePositionView(offer.market, id, offer.maker);
    candidateUnits = UtilsLib.min(candidateUnits, makerCredit);
}
return candidateUnits;
```

Alternatively, expose a `creditOf` view that returns the post-slash, post-fee credit and use it here to avoid the state-mutating `updatePosition` call.

### Proof of Concept
```solidity
function test_consumableUnitsPostMaturitySellNoCredit() public {
    // 1. Create market, maturity = block.timestamp + 1 days
    // 2. Maker signs/creates sell offer: buy=false, maxAssets=1000e18, maxUnits=0
    //    Maker has zero credit (never lent)
    // 3. Warp past maturity
    vm.warp(market.maturity + 1);

    // 4. consumableUnits returns non-zero — BUG
    uint256 units = ConsumableUnitsLib.consumableUnits(address(midnight), id, offer);
    assertGt(units, 0); // passes — incorrectly

    // 5. take reverts with CannotIncreaseDebtPostMaturity
    vm.expectRevert(IMidnight.CannotIncreaseDebtPostMaturity.selector);
    midnight.take(offer, ratifierData, units, taker, address(0), address(0), "");

    // 6. Bundle reverts with OutOfOffers
    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: offer, ratifierData: ratifierData, units: units});
    vm.expectRevert(IMidnightBundles.OutOfOffers.selector);
    bundles.buyWithUnitsTargetAndWithdrawCollateral(
        units, type(uint256).max, taker, emptyPermit, takes,
        new CollateralWithdrawal[](0), taker, 0, address(0)
    );
}
```

Expected assertions: `consumableUnits > 0`, `take` reverts with `CannotIncreaseDebtPostMaturity`, bundle reverts with `OutOfOffers`.

    function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
        uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);

            return offer.maxUnits.zeroFloorSub(consumed);
