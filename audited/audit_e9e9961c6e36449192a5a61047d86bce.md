Audit Report

## Title
Off-by-one in `CannotIncreaseDebtPostMaturity` guard permits debt creation at `block.timestamp == maturity` - (File: `src/Midnight.sol`)

## Summary
The maturity guard at line 391 of `src/Midnight.sol` uses `<=`, which passes when `block.timestamp == offer.market.maturity`, allowing `sellerDebtIncrease > 0` at the exact maturity boundary. The post-take health check (`isHealthy`) is a pure collateral-ratio function with no maturity awareness and cannot block this. One block later, the liquidation path at line 622 evaluates `block.timestamp > market.maturity` as `true`, making the position immediately liquidatable regardless of collateral health.

## Finding Description

**Root cause — line 391, `src/Midnight.sol`:**

```solidity
require(block.timestamp <= offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

`<=` passes when `block.timestamp == offer.market.maturity`, so `sellerDebtIncrease > 0` is not blocked at the maturity boundary. The guard name `CannotIncreaseDebtPostMaturity` and the symmetric liquidation check (strict `>`) both indicate the intended invariant is `block.timestamp < maturity`.

**Exploit flow:**
1. Attacker holds any valid sell offer (standard, unprivileged protocol operation).
2. Transaction lands in the block where `block.timestamp == market.maturity`.
3. `take` is called with `units > sellerPos.credit`, so `sellerDebtIncrease = units - sellerCreditDecrease > 0`.
4. Line 391: `maturity <= maturity` → `true` → guard passes.
5. `timeToMaturity` at line 359 evaluates to `zeroFloorSub(maturity, maturity) = 0`, so `buyerPendingFeeIncrease = 0`; no revert from fee math.
6. Line 414: `sellerPos.debt += sellerDebtIncrease` executes — debt is written to state at maturity.
7. Line 476: `isHealthy` (lines 944–960) checks only `maxDebt >= debt` (collateral ratio); at this timestamp the position can be collateral-healthy, so the check passes.
8. At `block.timestamp == maturity + 1`, a liquidator calls `liquidate` with `postMaturityMode = true`. Line 622: `block.timestamp > market.maturity` → `true` → `NotLiquidatable` does not revert → position is immediately liquidatable regardless of collateral health.

**Why existing checks fail:**
- The `CannotIncreaseDebtPostMaturity` guard uses `<=` instead of `<`, creating a one-block window.
- `isHealthy` (line 944) is a pure collateral-ratio function (`maxDebt >= debt`) with no maturity awareness; it cannot block debt creation at maturity.
- The liquidation guard at line 622 uses strict `>`, so it is consistent with `<` being the correct boundary for debt creation — confirming the asymmetry is a bug, not a design choice.
- No existing test exercises `block.timestamp == market.maturity` with `sellerDebtIncrease > 0`; all `CannotIncreaseDebtPostMaturity` tests warp to `maturity + 1`.

## Impact Explanation
Debt is created in violation of the core protocol invariant that debt must not increase at or after maturity. The affected position enters post-maturity mode in the very next block and becomes immediately liquidatable regardless of collateral health. The seller has no opportunity to repay before liquidation. This constitutes an accounting integrity failure and an unauthorized state change directly caused by the protocol's own guard logic.

## Likelihood Explanation
Requires a transaction to land in the exact block where `block.timestamp == market.maturity`. On post-Merge Ethereum with deterministic 12-second slots, a validator or MEV searcher using Flashbots can guarantee block inclusion at a target timestamp. A regular attacker can monitor the mempool and submit with appropriate timing. The only precondition is a valid sell offer, which is a normal, unprivileged protocol operation. The attack is repeatable for every market whose maturity timestamp coincides with a block boundary.

## Recommendation
Change the guard at line 391 from `<=` to `<`:

```solidity
require(block.timestamp < offer.market.maturity || sellerDebtIncrease == 0, CannotIncreaseDebtPostMaturity());
```

This aligns the debt-creation boundary with the liquidation boundary at line 622 (`block.timestamp > market.maturity`), closing the one-block window.

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];
                uint256 price = IOracle(collateralParam.oracle).price();
                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                    .mulDivDown(collateralParam.lltv, WAD);
                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
