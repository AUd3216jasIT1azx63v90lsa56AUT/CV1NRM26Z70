Audit Report

## Title
Arithmetic Underflow in `liquidate` When Seizing All Collateral at Price Just Above Bad-Debt Threshold - (File: `src/Midnight.sol`)

## Summary

When a liquidator passes `seizedAssets = collateral` at a price exactly one unit above `badDebtPriceDown`, the bad-debt block is skipped (`badDebt = 0`, so `_position.debt` remains `originalDebt`), but `repaidUnits`, computed with the identical ceiling expression (`mulDivUp`), can equal `originalDebt + 1`. The checked subtraction `_position.debt -= UtilsLib.toUint128(repaidUnits)` at line 676 then reverts with an arithmetic underflow, permanently blocking the `seizedAssets` input path at this price boundary for any unprivileged liquidator.

## Finding Description

**Root cause — rounding asymmetry between bad-debt check and `repaidUnits` computation.**

The bad-debt amount is zeroed out via ceiling rounding (lines 614–616):

```solidity
badDebt = badDebt.zeroFloorSub(
    _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, _collateralParam.maxLif)
);
```

`badDebt = 0` whenever `_collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif) >= originalDebt`. When `badDebt = 0`, the subtraction at line 628 is skipped, so `_position.debt` remains `originalDebt`.

When the liquidator passes `seizedAssets = collateral`, `repaidUnits` is computed at line 650 with the **identical ceiling expression**:

```solidity
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
```

In pre-maturity mode, `lif = _maxLif = maxLif`, so `repaidUnits = _collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif)`. This is the same expression as in the bad-debt computation. Therefore, when `badDebt = 0`, `repaidUnits >= originalDebt`. When the real-valued result of `_collateral * price / ORACLE_PRICE_SCALE * WAD / maxLif` lies strictly in `(originalDebt, originalDebt + 1)`, the ceiling rounds it up to `originalDebt + 1`, so `repaidUnits = originalDebt + 1 > _position.debt`. The checked subtraction at line 676 then reverts:

```solidity
_position.debt -= UtilsLib.toUint128(repaidUnits); // underflow: repaidUnits > _position.debt
```

**Why the RCF check does not prevent this.**

The RCF check at lines 655–668 computes `maxRepaid` as a multiple of `(_position.debt - maxDebt)`. For a position modestly underwater, `maxRepaid` can be a large multiple of `debt`, so `repaidUnits = debt + 1 <= maxRepaid` is satisfied. Markets with `rcfThreshold = type(uint256).max` trivially pass the second condition. Neither path checks `repaidUnits <= _position.debt`.

**Exploit flow:**

1. Market: single collateral, any LLTV < WAD, normal pre-maturity mode.
2. Borrower has `debt = units`, `collateral = units.mulDivUp(WAD, lltv)`.
3. Oracle set to `price = badDebtPriceDown(units) + 1`.
4. At this price: `badDebt = 0` (position unhealthy but no bad debt).
5. Liquidator calls `liquidate(seizedAssets = collateral, repaidUnits = 0, postMaturityMode = false)`.
6. `repaidUnits = collateral.mulDivUp(price, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif) = units + 1`.
7. RCF check passes.
8. `_position.debt -= UtilsLib.toUint128(repaidUnits)` → arithmetic underflow → revert.

## Impact Explanation

The `seizedAssets = collateral` liquidation path reverts unconditionally at this price boundary. The position is genuinely unhealthy and the liquidator is entitled to seize collateral, but the transaction always reverts. This is a service availability failure: a valid, documented liquidation input mode is permanently broken at a deterministic price value. The `repaidUnits = debt` input path (which uses `mulDivDown` at line 652) is unaffected, so the position is not permanently unliquidatable, but the full-collateral-seizure path is blocked.

## Likelihood Explanation

The price window is a single integer value (`badDebtPriceDown + 1`), which is narrow in relative terms but deterministic and repeatable. Any oracle returning integer prices will hit it with nonzero probability. No privileged access is required. Markets with `rcfThreshold = type(uint256).max` are unconditionally vulnerable whenever the oracle returns this exact price. The condition is reachable by any liquidator observing the oracle price.

## Recommendation

After computing `repaidUnits` from `seizedAssets` at line 650, cap it to `_position.debt`:

```solidity
repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
if (repaidUnits > _position.debt) repaidUnits = _position.debt;
```

This ensures the subtraction at line 676 never underflows when the liquidator seizes all collateral at the bad-debt price boundary, while preserving the economic intent (liquidator repays at most the full outstanding debt).

## Proof of Concept

1. Deploy a market with a single collateral, `lltv = 0.8e18`, `maxLif = 1.1e18`, `ORACLE_PRICE_SCALE = 1e36`.
2. Borrower supplies `collateral = C` and borrows `debt = D` such that `D = C.mulDivUp(WAD, lltv)`.
3. Find `price = P` such that `C.mulDivUp(P, ORACLE_PRICE_SCALE).mulDivUp(WAD, maxLif) = D + 1` (i.e., `P = badDebtPriceDown(D) + 1`).
4. Set oracle to return `P`.
5. Call `liquidate(seizedAssets = C, repaidUnits = 0, postMaturityMode = false)`.
6. Observe revert with arithmetic underflow at line 676.
7. Confirm that calling `liquidate(seizedAssets = 0, repaidUnits = D, postMaturityMode = false)` succeeds (the `mulDivDown` path is unaffected). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
