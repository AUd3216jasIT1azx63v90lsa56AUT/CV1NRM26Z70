### Title
Continuous Fee Accrual Rounds to Zero for Low-Decimal Loan Tokens, Allowing Lenders to Avoid Paying Protocol Fees ŌĆö (File: src/Midnight.sol)

---

### Summary

The continuous fee accrual formula in `updatePositionView` performs integer division that rounds to zero when the elapsed time between updates is small relative to the remaining time-to-maturity divided by `pendingFee`. For low-decimal loan tokens (e.g., USDC with 6 decimals), a lender with a small position can call the permissionless `updatePosition` every block, causing every fee accrual to round to zero. When the lender subsequently withdraws, the `pendingFee` obligation is silently zeroed out in `withdraw` without the protocol ever receiving the fee via `continuousFeeCredit`.

---

### Finding Description

**Root cause ŌĆö the accrual formula:**

In `updatePositionView`, the continuous fee accrued per call is:

```solidity
uint128 fee = _lastAccrual < market.maturity
    ? uint128(postSlashPendingFee.mulDivDown(accrualEnd - _lastAccrual, market.maturity - _lastAccrual))
    : 0;
``` [1](#0-0) 

This is `fee = pendingFee ├Ś elapsed / (maturity ŌłÆ lastAccrual)`. Integer division truncates to zero whenever:

```
pendingFee ├Ś elapsed  <  (maturity ŌłÆ lastAccrual)
```

**How `pendingFee` is set:**

When a buyer acquires credit in `take`, their `pendingFee` is:

```solidity
uint128 buyerPendingFeeIncrease =
    UtilsLib.toUint128(buyerCreditIncrease.mulDivDown(_marketState.continuousFee * timeToMaturity, WAD));
``` [2](#0-1) 

`MAX_CONTINUOUS_FEE = 0.01e18 / 365 days Ōēł 317,097,919` (representing 1 % annual fee). [3](#0-2) 

For a lender with **1,000 USDC credit** (= `1e9` raw units, 6 decimals) in a 1-year market at max fee:

```
pendingFee = 1e9 ├Ś 317097919 ├Ś 31536000 / 1e18 Ōēł 1e7   (10 USDC)
```

Fee per 2-second block:

```
fee = 1e7 ├Ś 2 / 31536000 = 0   ŌåÉ rounds to zero
```

The rounding-to-zero condition holds for any position where `credit < ~1,577 USDC` at max fee (or proportionally larger positions at lower fees).

**Why `updatePosition` is the trigger:**

`updatePosition` is fully permissionless ŌĆö any caller can invoke it for any user:

```solidity
function updatePosition(Market memory market, address user) external returns (uint128, uint128, uint128) {
    bytes32 id = toId(market);
    require(marketState[id].tickSpacing > 0, MarketNotCreated());
    return _updatePosition(market, id, user);
}
``` [4](#0-3) 

The lender can also trigger `_updatePosition` indirectly through `take` (small trades) or `withdraw`.

**How the fee obligation is silently erased on withdrawal:**

Inside `withdraw`, after `_updatePosition` (which accrues 0 fee due to rounding), the remaining `pendingFee` is proportionally zeroed out without transferring anything to `continuousFeeCredit`:

```solidity
_updatePosition(market, id, onBehalf);          // accrues 0 fee
...
pendingFeeDecrease = UtilsLib.toUint128(_position.pendingFee.mulDivUp(units, _position.credit));
_position.pendingFee -= pendingFeeDecrease;     // pendingFee ŌåÆ 0, no protocol credit
_position.credit -= UtilsLib.toUint128(units);
``` [5](#0-4) 

The `continuousFeeCredit` state variable ŌĆö the only source of claimable protocol fee ŌĆö is only incremented inside `_updatePosition`:

```solidity
marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);
``` [6](#0-5) 

If `accruedFee = 0` on every call, `continuousFeeCredit` never increases, and the fee is permanently lost when the lender withdraws.

---

### Impact Explanation

A lender with a small USDC (or USDT) position can avoid paying the entire continuous fee owed to the protocol. The protocol's `continuousFeeCredit` for that market never increases, so the fee claimer receives nothing. For a 1,000 USDC position at 1 % annual fee over 6 months, the protocol loses ~5 USDC per position. Scaled across many small lenders in active markets, the cumulative loss of fee revenue is material. The lender withdraws their full credit as if no fee were owed.

---

### Likelihood Explanation

The attack requires only that the lender:
1. Hold a position in a low-decimal loan token market (USDC/USDT are the primary candidates).
2. Call the permissionless `updatePosition` (or perform small `take` operations) every block.
3. Withdraw when liquidity becomes available (borrower repayment or post-maturity).

On L2 networks (Arbitrum, Base, Optimism) where gas costs are negligible, calling `updatePosition` every block is economically rational for any position where the avoided fee exceeds gas cost. No privileged access is required. The `updatePosition` function has no access control.

---

### Recommendation

Introduce a precision multiplier (e.g., `1e12`) in the fee accrual calculation to prevent rounding to zero for low-decimal tokens, then divide when writing back to storage:

```solidity
// Scale up during computation
uint256 feeScaled = postSlashPendingFee * PRECISION_MULTIPLIER * (accrualEnd - _lastAccrual)
                    / (market.maturity - _lastAccrual);
uint128 fee = uint128(feeScaled / PRECISION_MULTIPLIER);
```

Alternatively, enforce a minimum elapsed time before `_updatePosition` has any effect (e.g., require at least `MIN_ACCRUAL_INTERVAL` seconds since `lastAccrual`). This mirrors the mitigation suggested in the referenced Sherlock report for SymmStaking.

### Proof of Concept

**Setup:**
- Loan token: USDC (6 decimals)
- Market maturity: 1 year from now (`T = 31,536,000 s`)
- `continuousFee = MAX_CONTINUOUS_FEE Ōēł 317,097,919`
- Lender buys `1,000 USDC` credit ŌåÆ `pendingFee = 1e7` (10 USDC)

**Step 1 ŌĆö Verify rounding to zero:**
```
fee = 1e7 ├Ś 2 / 31,536,000 = 20,000,000 / 31,536,000 = 0  Ō£ō
```

**Step 2 ŌĆö Lender calls `updatePosition` every block for 6 months:**
- Each call: `accruedFee = 0`, `continuousFeeCredit` unchanged, `lastAccrual` advances by 2 s.
- After 6 months: `pendingFee` still `1e7`, `continuousFeeCredit` still 0.

**Step 3 ŌĆö Borrower repays at 6 months; lender calls `withdraw(units = 1e9)`:**
- `_updatePosition` inside `withdraw`: `elapsed = 2 s`, `maturity ŌłÆ lastAccrual = 6 months + 2 s Ōēł 15,768,002`
  ```
  fee = 1e7 ├Ś 2 / 15,768,002 = 0  Ō£ō
  ```
- `pendingFeeDecrease = 1e7 ├Ś 1e9 / 1e9 = 1e7` ŌåÆ `pendingFee` zeroed, no `continuousFeeCredit` increase.
- Lender receives `1,000 USDC` in full; protocol receives `0 USDC` fee.

**Expected outcome:** Protocol should have received ~5 USDC (50 % of 10 USDC for 6-month hold). Actual: 0 USDC.
