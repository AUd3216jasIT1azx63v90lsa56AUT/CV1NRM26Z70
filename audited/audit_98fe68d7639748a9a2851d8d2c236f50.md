### Title
Front-running victim's EcrecoverAuthorizer revocation with pre-signed authorization grants unauthorized Midnight control - (File: src/periphery/EcrecoverAuthorizer.sol)

### Summary
`EcrecoverAuthorizer.setIsAuthorized` is permissionless: any caller can submit a valid signed authorization. When a victim attempts to cancel a pending signed authorization by revoking `EcrecoverAuthorizer`'s Midnight permission, an attacker can front-run that revocation with the victim's pre-signed authorization in the same block, causing the authorization to land before the revocation and leaving the victim's intended cancellation ineffective.

### Finding Description

**Code path:**

`EcrecoverAuthorizer.setIsAuthorized` (lines 24–48):

```
Line 25: require(block.timestamp <= authorization.deadline, Expired());
Line 26: require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());
Lines 31–36: signer = ecrecover(...); require(signer == authorization.authorizer || ...);
Line 47: IMidnight(MIDNIGHT).setIsAuthorized(authorized, isAuthorized, authorizer);
```

`Midnight.setIsAuthorized` (line 732):

```
require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
```

**Root cause:** There is no check in `EcrecoverAuthorizer.setIsAuthorized` that `EcrecoverAuthorizer` is currently authorized in Midnight at the time of submission. The only Midnight-level gate is inside `Midnight.setIsAuthorized` at line 732, which evaluates `isAuthorized[victim][EcrecoverAuthorizer]` at call time. Because the function is permissionless (any `msg.sender` can submit a valid signature), an attacker who possesses the victim's signed authorization can submit it before the victim's revocation lands.

**Exploit flow:**

1. Victim signs `Authorization{authorizer=victim, authorized=X, isAuthorized=true, nonce=0, deadline=block.timestamp+365 days}` and the signature is known to the attacker (e.g., shared off-chain, seen in mempool from a prior submission attempt, or emitted via a dApp).
2. Victim decides to cancel and broadcasts `midnight.setIsAuthorized(address(ecrecoverAuthorizer), false, victim)`.
3. Attacker observes the revocation in the mempool and front-runs it with `ecrecoverAuthorizer.setIsAuthorized(auth, sig)`.
4. At the time of the attacker's call: deadline passes (365 days away); nonce=0 matches; `signer == authorization.authorizer` is true (victim signed it, so this branch short-circuits — no Midnight authorization check on the signer path); `isAuthorized[victim][EcrecoverAuthorizer]` is still `true` in `Midnight.setIsAuthorized` because the revocation has not yet executed. All checks pass.
5. `isAuthorized[victim][X] = true` is written in Midnight.
6. Victim's revocation executes: `isAuthorized[victim][EcrecoverAuthorizer] = false`. But `isAuthorized[victim][X]` remains `true`.

**Why existing checks fail:**

- The nonce check (line 26) passes because nonce=0 matches the current state. [1](#0-0) 
- The deadline check (line 25) passes because the deadline is 365 days away. [2](#0-1) 
- The signer check (lines 33–36) passes via the `signer == authorization.authorizer` branch — the victim signed it, so this is true regardless of whether `EcrecoverAuthorizer` is authorized in Midnight. [3](#0-2) 
- The Midnight gate (line 732) passes because `isAuthorized[victim][EcrecoverAuthorizer]` is still `true` at the moment the attacker's transaction executes. [4](#0-3) 
- There is no self-cancellation or nonce-invalidation mechanism the victim can use atomically without going through `EcrecoverAuthorizer` (which itself requires Midnight authorization to complete). [5](#0-4) 

### Impact Explanation
X receives `isAuthorized[victim][X] = true` in Midnight, granting X full Midnight control over the victim (ability to take offers, set consumed, withdraw collateral, etc.) even though the victim believed they had cancelled the authorization. The victim's revocation transaction succeeds but is rendered meaningless because the authorization was already applied.

### Likelihood Explanation
**Preconditions:** (1) Victim has a signed authorization with a future deadline in circulation (common in dApp flows where signatures are shared off-chain or submitted to a relayer). (2) Victim attempts to cancel by revoking `EcrecoverAuthorizer`'s Midnight permission rather than burning the nonce. (3) Attacker has the signed authorization (from mempool, off-chain, or prior on-chain event). All three are realistic. The attack is repeatable by any MEV searcher or attacker monitoring the mempool, and requires no special privilege.

### Recommendation
Add a self-service nonce-invalidation function to `EcrecoverAuthorizer` that the victim can call directly (without requiring Midnight authorization) to advance their nonce and atomically invalidate all pending signed authorizations:

```solidity
function invalidateNonce() external {
    nonce[msg.sender]++;
    emit NonceInvalidated(msg.sender, nonce[msg.sender]);
}
```

This allows the victim to atomically burn the pending nonce in a single transaction, making any pre-signed authorization with the old nonce permanently invalid regardless of transaction ordering. The victim can then safely revoke `EcrecoverAuthorizer`'s Midnight authorization in a separate transaction.

### Proof of Concept

```solidity
function testFrontRunRevocation() public {
    // Setup: victim authorizes EcrecoverAuthorizer in Midnight
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);

    // Victim signs authorization granting X control, nonce=0, deadline=+365 days
    Authorization memory auth = Authorization({
        authorizer: victim,
        authorized: X,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 365 days
    });
    Signature memory sig = signAuthorization(auth, victim);

    // Victim decides to cancel — broadcasts revocation of EcrecoverAuthorizer
    // Attacker front-runs with victim's pre-signed authorization
    // (same block, attacker tx first)
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // Victim's revocation executes after
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), false, victim);

    // Assertions: victim's revocation was ineffective
    assertFalse(midnight.isAuthorized(victim, address(ecrecoverAuthorizer)));
    // X still has full Midnight control — invariant violated
    assertTrue(midnight.isAuthorized(victim, X));
}
```

Expected: `isAuthorized[victim][X]` is `true` after the victim's revocation, demonstrating the invariant violation. [6](#0-5)

    function setIsAuthorized(Authorization memory authorization, Signature calldata signature) external {
        require(block.timestamp <= authorization.deadline, Expired());
        require(authorization.nonce == nonce[authorization.authorizer]++, InvalidNonce());

        bytes32 hashStruct = keccak256(abi.encode(AUTHORIZATION_TYPEHASH, authorization));
        bytes32 domainSeparator = keccak256(abi.encode(EIP712_DOMAIN_TYPEHASH, block.chainid, address(this)));
        bytes32 digest = keccak256(bytes.concat("\x19\x01", domainSeparator, hashStruct));
        address signer = ecrecover(digest, signature.v, signature.r, signature.s);
        require(signer != address(0), InvalidSignature());
        require(
            signer == authorization.authorizer || IMidnight(MIDNIGHT).isAuthorized(authorization.authorizer, signer),
            Unauthorized()
        );

        emit SetIsAuthorized(
            msg.sender,
            authorization.authorizer,
            authorization.authorized,
            authorization.isAuthorized,
            authorization.nonce
        );

        IMidnight(MIDNIGHT)
            .setIsAuthorized(authorization.authorized, authorization.isAuthorized, authorization.authorizer);
    }
