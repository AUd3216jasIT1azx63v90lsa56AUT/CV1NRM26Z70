### Title
Authorized Operator Can Re-Enable Maker's Revoked Root in SetterRatifier - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier.setIsRootRatified` accepts a bidirectional `bool` parameter and grants any `isAuthorized` operator the same power as the maker to toggle `isRootRatified` in either direction. When a maker calls `setIsRootRatified(maker, root, false)` to revoke a root, any operator still holding `isAuthorized[maker][operator] == true` can immediately call `setIsRootRatified(maker, root, true)` to re-enable it, causing offers the maker intended to cancel to become fillable again.

### Finding Description
The authorization guard in `SetterRatifier` at line 25 is:

```solidity
require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
isRootRatified[maker][root] = newIsRootRatified;
``` [1](#0-0) 

The function accepts any `bool` value for `newIsRootRatified` and applies no restriction on which direction (enable vs. disable) an operator may toggle the flag. The authorization check treats enabling and disabling identically.

Contrast this with `EcrecoverRatifier.cancelRoot`, which is hardcoded to only ever set `isRootCanceled[maker][root] = true` — making cancellation permanent and operator-proof:

```solidity
isRootCanceled[maker][root] = true;
``` [2](#0-1) 

The `isRatified` check in `SetterRatifier` simply reads the current flag value:

```solidity
require(isRootRatified[offer.maker][root], NotRatified());
``` [3](#0-2) 

**Exploit path:**
1. Maker calls `midnight.setIsAuthorized(operator, true, maker)` — operator is now authorized.
2. Maker calls `setterRatifier.setIsRootRatified(maker, root, true)` — root is live.
3. Maker calls `setterRatifier.setIsRootRatified(maker, root, false)` — maker intends to stop all fills under this root.
4. Operator calls `setterRatifier.setIsRootRatified(maker, root, true)` — passes the authorization check because `isAuthorized[maker][operator] == true`; `isRootRatified[maker][root]` is set back to `true`.
5. Taker calls `midnight.take(offer, ratifierData, ...)` — `isRatified` passes, fill executes against maker's explicit revocation intent.

No existing check prevents step 4. The operator's authorization was granted for a different purpose (e.g., managing positions), not to override the maker's explicit revocation decision.

### Impact Explanation
A maker's explicit revocation of a Merkle root via `setIsRootRatified(..., false)` can be silently undone by any currently-authorized operator. Offers the maker believed were cancelled remain fillable, exposing the maker to unwanted loan positions, collateral obligations, or credit deployments they did not consent to at the time of the fill.

### Likelihood Explanation
Preconditions: (1) maker has previously authorized at least one operator (`isAuthorized[maker][operator] == true`), and (2) maker calls `setIsRootRatified(maker, root, false)` without atomically revoking the operator's authorization in the same transaction. This is a realistic operational pattern — makers may authorize operators for position management while separately managing offer roots. The operator need not be malicious; a compromised operator key is sufficient. The attack is repeatable as long as the operator's authorization is not revoked.

### Recommendation
Restrict the disable direction to the maker only. Split the function into two: one that only enables (callable by maker or operator) and one that only disables (callable by maker only):

```solidity
function enableRoot(address maker, bytes32 root) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = true;
    emit SetIsRootRatified(msg.sender, maker, root, true);
}

function revokeRoot(address maker, bytes32 root) public {
    require(maker == msg.sender, Unauthorized()); // only maker can revoke
    isRootRatified[maker][root] = false;
    emit SetIsRootRatified(msg.sender, maker, root, false);
}
```

Alternatively, mirror `EcrecoverRatifier`'s one-way design: make `setIsRootRatified` only accept `true` from operators, and require `maker == msg.sender` to set `false`.

### Proof of Concept

```solidity
function testOperatorReEnablesRevokedRoot() public {
    // Setup
    address maker = makeAddr("maker");
    address operator = makeAddr("operator");
    address taker = makeAddr("taker");

    Offer memory offer = makeOffer(maker); // ratifier = setterRatifier
    bytes32 root = HashLib.hashOffer(offer);
    bytes memory ratifierData = abi.encode(root, 0, new bytes32[](0));

    // Maker authorizes operator and enables root
    vm.prank(maker);
    midnight.setIsAuthorized(operator, true, maker);
    vm.prank(maker);
    midnight.setIsAuthorized(address(setterRatifier), true, maker);
    vm.prank(maker);
    setterRatifier.setIsRootRatified(maker, root, true);

    // Maker revokes root — intends to stop all fills
    vm.prank(maker);
    setterRatifier.setIsRootRatified(maker, root, false);
    assertFalse(setterRatifier.isRootRatified(maker, root));

    // Operator re-enables root without maker's consent
    vm.prank(operator);
    setterRatifier.setIsRootRatified(maker, root, true);
    assertTrue(setterRatifier.isRootRatified(maker, root)); // re-enabled

    // Taker fills offer — succeeds against maker's intent
    deal(address(loanToken), maker, offer.maxUnits);
    collateralize(offer.market, taker, offer.maxUnits);
    vm.prank(taker);
    midnight.take(offer, ratifierData, 1, taker, taker, address(0), hex"");

    // Assert: fill succeeded — maker's revocation was bypassed
    assertGt(midnight.debtOf(marketId(offer.market), taker), 0);
}
```

Expected assertion: `debtOf > 0`, confirming the fill succeeded after the maker's explicit revocation, proving the invariant is broken.
