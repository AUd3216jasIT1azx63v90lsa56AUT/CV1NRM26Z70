### Title
Stale EcrecoverAuthorizer Signature Replayed After Direct Midnight Revocation ‚Äî (`src/periphery/EcrecoverAuthorizer.sol`)

### Summary
`EcrecoverAuthorizer` maintains its own `nonce` mapping that is only incremented when `EcrecoverAuthorizer.setIsAuthorized` is called. A victim who revokes an attacker by calling `Midnight.setIsAuthorized` directly does not increment this nonce, leaving any previously signed `Authorization{nonce=N, isAuthorized=true}` replayable through `EcrecoverAuthorizer` for as long as its deadline has not expired. An unprivileged attacker holding such a signature can re-authorize themselves on Midnight after the victim believed they had been revoked.

### Finding Description

**State before exploit:**
- `EcrecoverAuthorizer.nonce[victim] = N`
- `Midnight.isAuthorized[victim][ecrecoverAuthorizer] = true` (victim authorized the periphery contract to act on their behalf ‚Äî required for the signed flow to have been useful at all)
- Victim previously signed `Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N, deadline=T}` with `T > block.timestamp`
- Victim calls `Midnight.setIsAuthorized(attacker, false, victim)` directly, setting `Midnight.isAuthorized[victim][attacker] = false`
- `EcrecoverAuthorizer.nonce[victim]` remains `N` ‚Äî Midnight's direct call does not touch it

**Attacker's call:**
```
EcrecoverAuthorizer.setIsAuthorized(
    Authorization{authorizer=victim, authorized=attacker, isAuthorized=true, nonce=N, deadline=T},
    sig  // victim's original signature
)
```

**Check-by-check trace in `EcrecoverAuthorizer.setIsAuthorized`:**

1. `block.timestamp <= authorization.deadline` ‚Üí `block.timestamp <= T` ‚Üí **passes** (deadline still valid) [5](#0-4) 
2. `authorization.nonce == nonce[authorization.authorizer]++` ‚Üí `N == N` ‚Üí **passes**, nonce becomes `N+1` [6](#0-5) 
3. `ecrecover(digest, ...)` ‚Üí returns `victim` (valid signature) [7](#0-6) 
4. `signer == authorization.authorizer` ‚Üí `victim == victim` ‚Üí **passes** (no Midnight state consulted here) [3](#0-2) 
5. `IMidnight(MIDNIGHT).setIsAuthorized(attacker, true, victim)` is called with `msg.sender = ecrecoverAuthorizer`; Midnight checks `isAuthorized[victim][ecrecoverAuthorizer] == true` ‚Üí **passes** ‚Üí sets `isAuthorized[victim][attacker] = true` [8](#0-7) 

**Result:** `Midnight.isAuthorized[victim][attacker]` is `true` again. The victim's revocation is silently undone.

No existing check stops this. The nonce check only prevents the same EcrecoverAuthorizer signature from being submitted twice through EcrecoverAuthorizer; it has no awareness of direct Midnight calls. The signer check does not consult current Midnight authorization state for the case where the signer is the authorizer themselves.

### Impact Explanation
The attacker regains full delegated access on Midnight ‚Äî they can call `withdraw`, `withdrawCollateral`, `repay`, `setConsumed`, `setIsAuthorized`, and `take` (as taker) on behalf of the victim. This directly violates the invariant that a Midnight-level revocation must prevent all further delegated actions. Concretely, the attacker can drain the victim's lender credit via `withdraw` or seize their collateral via `withdrawCollateral`.

### Likelihood Explanation
**Preconditions:**
1. Victim must have authorized `EcrecoverAuthorizer` on Midnight (common ‚Äî required to use the signed flow at all).
2. Victim must have signed an `Authorization{isAuthorized=true}` with a future deadline that has not yet been submitted.
3. Victim must revoke the attacker via direct `Midnight.setIsAuthorized` rather than through `EcrecoverAuthorizer`.

Condition 3 is the most natural revocation path a user would take ‚Äî calling Midnight directly is simpler and more obvious than constructing and submitting a new signed authorization. The window of exploitability is the remaining lifetime of the signed authorization's deadline. This is repeatable: the attacker can wait until the victim revokes and then immediately replay.

### Recommendation
Add a `invalidateNonce(address authorizer)` function to `EcrecoverAuthorizer` that increments `nonce[authorizer]` when called by `authorizer` or by an address authorized on Midnight to act on their behalf. This allows a victim to atomically invalidate all pending signed authorizations without needing to construct a new signature.

Alternatively, document explicitly (and enforce via NatSpec or a wrapper) that revocation must always go through `EcrecoverAuthorizer.setIsAuthorized` with `isAuthorized=false` ‚Äî never directly through `Midnight.setIsAuthorized` ‚Äî when the EcrecoverAuthorizer periphery is in use. However, this is an operational control, not a code-level fix, and is insufficient on its own.

### Proof of Concept

```solidity
// Foundry unit test
function testStaleAuthReplayAfterDirectRevocation() public {
    // Setup: victim authorizes EcrecoverAuthorizer on Midnight
    vm.prank(victim);
    midnight.setIsAuthorized(address(ecrecoverAuthorizer), true, victim);

    // Victim signs Authorization{nonce=0, authorized=attacker, isAuthorized=true, deadline=+1day}
    Authorization memory auth = Authorization({
        authorizer: victim,
        authorized: attacker,
        isAuthorized: true,
        nonce: 0,
        deadline: block.timestamp + 1 days
    });
    Signature memory sig = signAuthorization(auth, victimPrivKey);

    // Victim revokes attacker directly on Midnight (does NOT increment EcrecoverAuthorizer nonce)
    vm.prank(victim);
    midnight.setIsAuthorized(attacker, false, victim);

    // Assert: attacker is revoked
    assertFalse(midnight.isAuthorized(victim, attacker));
    // Assert: EcrecoverAuthorizer nonce is still 0
    assertEq(ecrecoverAuthorizer.nonce(victim), 0);

    // Attacker replays the stale signed authorization
    vm.prank(attacker);
    ecrecoverAuthorizer.setIsAuthorized(auth, sig);

    // Assert: attacker is re-authorized ‚Äî INVARIANT VIOLATED
    assertTrue(midnight.isAuthorized(victim, attacker));
    // Assert: nonce is now 1 (replay consumed)
    assertEq(ecrecoverAuthorizer.nonce(victim), 1);
}
```

**Expected assertion failure:** `assertTrue(midnight.isAuthorized(victim, attacker))` should fail if the protocol correctly enforces that a Midnight-level revocation prevents replay. Instead it passes, confirming the bug.

A stateful fuzz test can generalize this by fuzzing call orderings of `Midnight.setIsAuthorized(attacker, false, victim)` and `EcrecoverAuthorizer.setIsAuthorized(auth, sig)` and asserting that after any sequence containing a revocation, `midnight.isAuthorized(victim, attacker)` is `false`.
