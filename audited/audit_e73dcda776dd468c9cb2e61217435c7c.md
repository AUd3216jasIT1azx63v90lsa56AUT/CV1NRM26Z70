Audit Report

## Title
Missing `caller` Parameter in `IRepayCallback.onRepay()` Enables Unauthorized Token Drain from Callback Contracts - (File: src/Midnight.sol)

## Summary
The `repay()` function accepts an arbitrary `callback` address and sets it as the `payer` without requiring `callback == msg.sender`. Because `IRepayCallback.onRepay()` does not forward the original `msg.sender` (unlike `ILiquidateCallback` and `IFlashLoanCallback`, which both include `address caller`), any callback contract implementing `onRepay` cannot distinguish a legitimate self-initiated repay from one triggered by an attacker. An attacker with any debt in a market can drain loan tokens from any contract that implements `onRepay` and holds a Midnight approval.

## Finding Description
In `src/Midnight.sol` lines 502–521, the `repay()` function:

1. Checks only that `msg.sender` is authorized for `onBehalf` (line 505) — no check that `callback == msg.sender`.
2. Sets `payer = callback` when `callback != address(0)` (line 511).
3. Calls `IRepayCallback(callback).onRepay(id, market, units, onBehalf, data)` (line 516) — without passing the original caller.
4. Executes `safeTransferFrom(loanToken, payer, address(this), units)` (line 520), pulling tokens from `callback`.

The `IRepayCallback` interface (`src/interfaces/ICallbacks.sol` line 21) omits `address caller`, while both `ILiquidateCallback.onLiquidate` (line 17) and `IFlashLoanCallback.onFlashLoan` (line 25) include it. Inside `onRepay`, `msg.sender` is always `address(Midnight)` — the victim cannot identify the initiating user.

**Exploit flow:**
1. Attacker borrows from a market to acquire debt (or uses any existing debt position where `onBehalf == msg.sender`).
2. Attacker calls `repay(market, units, attacker, victimContract, data)`. Authorization check passes: `onBehalf == msg.sender`.
3. Midnight calls `victimContract.onRepay(id, market, units, attacker, data)`. Inside `onRepay`, `msg.sender == Midnight`. The victim cannot identify the attacker.
4. If `victimContract` returns `CALLBACK_SUCCESS` (as any compliant implementation must), Midnight executes `safeTransferFrom(loanToken, victimContract, Midnight, units)`.
5. Attacker's debt is repaid using the victim's tokens. The victim's balance is reduced without its informed consent.

The `OnlyExplicitPayerCanLoseTokens.spec` formal model treats `CALLBACK_SUCCESS` as consent, but this model is broken when the callback cannot determine who triggered the call.

## Impact Explanation
Any contract that (a) implements `IRepayCallback.onRepay()` returning `CALLBACK_SUCCESS` and (b) holds a Midnight approval for loan tokens can have its entire approved balance drained. This is direct theft of assets from third-party integration contracts (flash-repay routers, bundlers, aggregators) — a concrete, in-scope impact. The victim's token balance decreases to repay a debt it did not choose to repay, violating the protocol's authorization model.

## Likelihood Explanation
Preconditions are low-cost: the attacker only needs existing debt in any market (obtainable by borrowing), which requires no cooperation from any other party. Contracts implementing `IRepayCallback` and holding Midnight approval are the intended integration targets of the repay callback feature. The attack is repeatable as long as the victim retains Midnight approval and loan token balance. The missing `caller` parameter makes it structurally impossible for callback implementors to defend without out-of-band mechanisms (e.g., transient storage reentrancy guards), which the interface does not require or document.

## Recommendation
Add `address caller` to `IRepayCallback.onRepay()` and pass `msg.sender` at the call site, consistent with `ILiquidateCallback` and `IFlashLoanCallback`:

```solidity
// ICallbacks.sol
interface IRepayCallback {
    function onRepay(address caller, bytes32 id, Market memory market, uint256 units, address onBehalf, bytes memory data)
        external returns (bytes32);
}

// Midnight.sol
IRepayCallback(callback).onRepay(msg.sender, id, market, units, onBehalf, data)
```

This allows callback contracts to verify `caller == address(this)` or another trusted initiator, matching the defense pattern available in all other callback interfaces.

## Proof of Concept
1. Deploy a token and a Midnight market.
2. Deploy `VictimRepayCallback` implementing `onRepay` that returns `CALLBACK_SUCCESS` if `msg.sender == address(Midnight)` (standard compliant implementation). Grant Midnight approval for `units` loan tokens.
3. Attacker borrows `units` loan tokens from the market (establishing debt).
4. Attacker calls `repay(market, units, attacker, address(VictimRepayCallback), "")`.
5. Observe: attacker's debt is cleared; `VictimRepayCallback`'s loan token balance decreases by `units`; attacker retains the originally borrowed tokens.
