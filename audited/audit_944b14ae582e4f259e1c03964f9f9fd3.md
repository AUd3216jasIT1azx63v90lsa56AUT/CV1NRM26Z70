### Title
Buy offer `maxAssets` cap bypassed in units when `buyerPrice < WAD` via `mulDivDown` rounding to zero — (`src/Midnight.sol`)

### Summary
When a buy offer uses `maxAssets > 0` as its cap and the tick is set such that `buyerPrice < WAD`, the expression `units.mulDivDown(buyerPrice, WAD)` evaluates to zero for any `units < WAD / buyerPrice`. Because `consumed` is incremented by `buyerAssets` (which is zero), the `ConsumedAssets` guard passes even on a fully-consumed offer, and the taker transfers real units of credit/debt to the maker at zero asset cost. This is a differential from the `buyerPrice = WAD` case, where the same small fill correctly increments `consumed` by a nonzero amount and reverts.

### Finding Description
**Code path:** `src/Midnight.sol` lines 363–369.

```solidity
uint256 buyerAssets = offer.buy ? units.mulDivDown(buyerPrice, WAD) : ...;
...
if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

**Root cause:** For a buy offer, `consumed` is incremented by `buyerAssets`, not by `units`. When `buyerPrice < WAD` and `units` is small enough that `units * buyerPrice < WAD`, `mulDivDown` returns 0. `consumed` is unchanged, the cap check passes, and the take succeeds — but `units` of credit and debt are still allocated to the buyer (maker) and seller (taker) respectively.

**Attacker-controlled inputs:**
- `offer.buy = true`, `offer.maxAssets > 0`, `offer.tick` chosen so `tickToPrice(tick) < WAD`
- `units = 1` (or any value where `units * buyerPrice < WAD`)

**Exploit flow:**
1. Offer is at or near its `maxAssets` limit (either by prior fills or by the maker's own `setConsumed`).
2. Taker calls `take(offer, ..., units=1, ...)` where `buyerPrice < WAD`.
3. `buyerAssets = 1.mulDivDown(buyerPrice, WAD) = 0`.
4. `consumed += 0` → `newConsumed == maxAssets` → `require(newConsumed <= maxAssets)` passes.
5. Position state is updated: maker's `credit` increases by 1 unit, taker's `debt` increases by 1 unit, `totalUnits` increases, maker's `pendingFee` increases.
6. No loan tokens are transferred (`buyerAssets = 0`).
7. Step 2–6 can be repeated indefinitely.

**Why checks fail:** The `ConsumedAssets` guard only checks `consumed <= maxAssets`. Because `consumed` is incremented by the rounded-down `buyerAssets` (which is 0), the guard never triggers regardless of how many additional units are pushed through.

**Contrast at `buyerPrice = WAD`:** `1.mulDivDown(WAD, WAD) = 1`, so `consumed` increments by 1 and the check reverts on the second fill. The differential is exact and deterministic.

**Protocol acknowledgment:** The behavior is documented at `src/Midnight.sol` line 94 ("It is possible to give units to a fully consumed assets-based buy offer with price < 1") and is demonstrated by the existing test `testBugBuyMaxAssetsBypass` in `test/TakeTest.sol` lines 857–889, which asserts `buyerAssets = 0`, `sellerAssets = 0`, yet `creditOf` and `debtOf` both increase.

**Clarification on the question's invariant framing:** The claim that "`consumed` must equal total `buyerAssets` transferred" is not violated — `consumed` does equal the sum of all `buyerAssets`. The invariant that IS violated is "offers cannot be overfilled," specifically that the maker's unit exposure is not bounded by `maxAssets` when `buyerPrice < WAD`.

### Impact Explanation
A malicious taker can grief a buy-offer maker by forcing unbounded unit accumulation (credit, `pendingFee`, position size) beyond the maker's intended `maxAssets` cap, at zero token cost to the taker (the taker absorbs the corresponding debt). The maker's `pendingFee` grows without bound, their credit position grows beyond their intended limit, and `totalUnits` for the market inflates. The taker pays only gas and accepts self-inflicted debt, making this a viable grief vector against any maker with a buy offer at a sub-WAD tick.

### Likelihood Explanation
**Preconditions:** (1) A buy offer exists with `maxAssets > 0` and a tick where `tickToPrice < WAD`. (2) The offer is at or near its consumed limit. Both are normal operating conditions — sub-WAD ticks are standard for discount lending markets. **Feasibility:** Any unprivileged taker can execute this with `units = 1` in a single transaction. **Repeatability:** Unlimited; each call adds 0 to `consumed` and 1 to the maker's credit/pendingFee.

### Recommendation
Add a guard in the `maxAssets` branch that rejects takes where `buyerAssets == 0` but `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets()); // new guard
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, track `consumed` in units (not assets) even in `maxAssets` mode, and convert `maxAssets` to a unit cap at offer creation time using the tick price. This eliminates the rounding gap entirely.

### Proof of Concept
The existing test `testBugBuyMaxAssetsBypass` (`test/TakeTest.sol:858`) already proves the core behavior. A differential Foundry test to confirm the price=WAD vs price<WAD gap:

```solidity
function testDifferentialConsumedAtWADVsSubWAD() public {
    // Setup: buy offer, maxAssets = 1e18, settlement fee = 0
    // Tick A: tickToPrice == WAD  → buyerPrice = WAD
    // Tick B: tickToPrice == WAD-1 → buyerPrice = WAD-1

    // --- price = WAD ---
    lenderOffer.tick = TICK_AT_WAD;
    lenderOffer.maxAssets = 1e18;
    // Fill to cap: units = 1e18 → buyerAssets = 1e18 → consumed = 1e18
    take(1e18, borrower, lenderOffer);
    assertEq(midnight.consumed(lender, lenderOffer.group), 1e18);
    // Second fill: units = 1 → buyerAssets = 1 → newConsumed = 1e18+1 → REVERT
    vm.expectRevert(IMidnight.ConsumedAssets.selector);
    take(1, borrower, lenderOffer);

    // --- price < WAD ---
    lenderOffer.tick = TICK_BELOW_WAD; // tickToPrice < WAD
    lenderOffer.maxAssets = 1;
    vm.prank(lender);
    midnight.setConsumed(lenderOffer.group, 1, lender); // consumed = maxAssets = 1
    uint256 creditBefore = midnight.creditOf(id, lender);
    // Second fill: units = 1 → buyerAssets = 0 → consumed unchanged → SUCCEEDS
    (uint256 ba,) = take(1, borrower, lenderOffer);
    assertEq(ba, 0);                                          // zero assets transferred
    assertEq(midnight.consumed(lender, lenderOffer.group), 1); // consumed unchanged
    assertGt(midnight.creditOf(id, lender), creditBefore);    // but credit grew — INVARIANT BROKEN
}
```

**Expected assertions:** `ba == 0`, `consumed == maxAssets` (unchanged), `creditOf > creditBefore`. These three together prove the overfill: the cap was not exceeded in asset terms, yet real unit-denominated state changed. [1](#0-0) [2](#0-1) [3](#0-2)
