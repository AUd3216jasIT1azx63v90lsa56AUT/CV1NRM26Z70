### Title
`consumableUnits` ignores `offer.reduceOnly`, causing bundler to silently skip permanently-reverting offers and revert with `OutOfOffers` - (File: src/periphery/ConsumableUnitsLib.sol)

### Summary
`ConsumableUnitsLib.consumableUnits` computes fillable units solely from `consumed[maker][group]` vs `offer.maxUnits`/`offer.maxAssets`, with no awareness of `offer.reduceOnly` or the maker's current credit/debt. When a `reduceOnly` offer's maker has zero credit (sell) or zero debt (buy), `consumableUnits` returns a positive value, but `Midnight.take()` will always revert with `MakerCreditOrDebtIncreased`. The bundler's `try/catch` silently swallows this revert, the offer is skipped, and if no remaining offers cover `targetUnits`, the bundler reverts with `OutOfOffers`.

### Finding Description

**Root cause — missing `reduceOnly` guard in `consumableUnits`:**

`src/periphery/ConsumableUnitsLib.sol` lines 14–23:
```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    if (offer.maxUnits > 0) {
        return offer.maxUnits.zeroFloorSub(consumed);   // ← no reduceOnly check
    } else if (offer.buy) {
        return TakeAmountsLib.buyerAssetsToUnits(...);
    } else {
        return TakeAmountsLib.sellerAssetsToUnits(...);
    }
}
```

The function never reads `offer.reduceOnly` or the maker's on-chain `credit`/`debt`.

**`take()` enforces the constraint that `consumableUnits` ignores:**

`src/Midnight.sol` lines 382–395:
```solidity
uint256 sellerCreditDecrease = UtilsLib.min(units, sellerPos.credit);
uint256 sellerDebtIncrease   = units - sellerCreditDecrease;
...
require(
    !offer.reduceOnly || (offer.buy ? buyerCreditIncrease == 0 : sellerDebtIncrease == 0),
    MakerCreditOrDebtIncreased()
);
```

For a `reduceOnly` sell offer where `sellerPos.credit == 0`: `sellerDebtIncrease = units`, so the require always fails regardless of `units > 0`.

**Bundler silently absorbs the revert:**

`src/periphery/MidnightBundles.sol` lines 74–88:
```solidity
uint256 unitsToTake = min(
    targetUnits - filledUnits,
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)  // returns N > 0
);
try IMidnight(MIDNIGHT).take(..., unitsToTake, ...) returns (...) {
    filledUnits += unitsToTake;
} catch {}   // MakerCreditOrDebtIncreased swallowed here
```

`filledUnits` is never incremented. After the loop, `require(filledUnits == targetUnits, OutOfOffers())` fires.

**Exploit flow (maker as attacker, taker as victim):**

1. Maker (attacker) signs a `reduceOnly` sell offer with `maxUnits = N`, while holding `credit = 0` in the target market (or withdraws all credit after signing).
2. Taker (victim) constructs a bundler call to `buyWithUnitsTargetAndWithdrawCollateral` (or any of the four bundle functions) with `targetUnits = N` and `takes = [attackerOffer]`.
3. Bundler calls `consumableUnits` → returns `N` (positive; no `reduceOnly` check).
4. Bundler calls `take(offer, ..., N, ...)` → `sellerDebtIncrease = N`, `reduceOnly` check fails → reverts with `MakerCreditOrDebtIncreased`.
5. `catch {}` swallows the revert; `filledUnits` stays 0.
6. Loop ends; `require(0 == N, OutOfOffers())` → entire bundler call reverts.

The same path applies to a `reduceOnly` buy offer where the maker's `debt == 0` (`buyerCreditIncrease = units > 0` → revert).

**Why existing protections are insufficient:**

The bundler comment explicitly states it "Skips every reason why take can revert (including ones that are not asynchrony related)." The `try/catch` is intentional, but `consumableUnits` is supposed to pre-filter offers that cannot be filled. It fails to do so for `reduceOnly` offers whose maker has no position to reduce.

### Impact Explanation

A maker can publish a `reduceOnly` offer with no existing credit/debt. Any taker who routes through the bundler with this offer as the sole (or gap-filling) entry will have their transaction revert with `OutOfOffers`. The taker loses gas and may miss time-sensitive operations (e.g., liquidation-adjacent collateral withdrawal, rate-sensitive borrowing). The bundler's `pullToken` / `forceApproveMax` sequence executes before the loop, so the taker's tokens are temporarily pulled and then returned on revert — no permanent fund loss, but the transaction is reliably griefable.

### Likelihood Explanation

**Preconditions:**
- Maker creates a `reduceOnly` offer with `maxUnits > 0` and `credit == 0` (sell) or `debt == 0` (buy). This is a valid, signable offer state — no privileged access required.
- Taker includes this offer in a bundler call. Off-chain order books and aggregators rely on `consumableUnits` to filter offers; a positive return value will cause them to include the offer.

**Feasibility:** Trivially reachable. A maker can sign such an offer at any time. The attack is repeatable across any market and any bundler function that uses `ConsumableUnitsLib`.

**Repeatability:** The maker's offer remains valid (not consumed, not expired) after the bundler reverts, so the same offer can grief repeated bundler calls until the taker manually excludes it.

### Recommendation

In `consumableUnits`, when `offer.reduceOnly` is set, cap the returned units by the maker's actual reducible position:

```solidity
function consumableUnits(address midnight, bytes32 id, Offer memory offer) internal view returns (uint256) {
    uint256 consumed = IMidnight(midnight).consumed(offer.maker, offer.group);
    uint256 units;
    if (offer.maxUnits > 0) {
        units = offer.maxUnits.zeroFloorSub(consumed);
    } else if (offer.buy) {
        units = TakeAmountsLib.buyerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
    } else {
        units = TakeAmountsLib.sellerAssetsToUnits(midnight, id, offer, offer.maxAssets.zeroFloorSub(consumed));
    }
    if (offer.reduceOnly) {
        // For a buy offer the maker is the buyer; reduceOnly requires buyerCreditIncrease == 0,
        // i.e. units <= maker's debt. For a sell offer the maker is the seller; reduceOnly
        // requires sellerDebtIncrease == 0, i.e. units <= maker's credit.
        (uint128 makerCredit,,,,uint128 makerDebt,) = IMidnight(midnight).position(id, offer.maker);
        uint256 cap = offer.buy ? makerDebt : makerCredit;
        units = units.zeroFloorSub(units > cap ? units - cap : 0); // i.e. min(units, cap)
    }
    return units;
}
```

This ensures `consumableUnits` returns 0 when `take()` would always revert due to the `reduceOnly` constraint, preventing the bundler from wasting a slot on an unfillable offer.
