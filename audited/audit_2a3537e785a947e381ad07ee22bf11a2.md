Audit Report

## Title
Unconditional zero-value `safeTransfer` to receiver when `filledSellerAssets == 0` causes revert on zero-transfer-reverting `loanToken` - (File: src/periphery/MidnightBundles.sol)

## Summary
In `supplyCollateralAndSellWithUnitsTarget`, when `targetUnits == 0` the takes loop is skipped entirely, leaving `filledSellerAssets == 0`. The final `SafeTransferLib.safeTransfer(loanToken, receiver, 0)` at line 168 executes unconditionally — unlike the referral fee transfer at line 167 which is guarded by `if (referralFeeAssets > 0)`. On any `loanToken` that reverts on zero-value `transfer` calls, every invocation with `targetUnits == 0` and `minSellerAssets == 0` will revert, making the no-op path permanently unusable for such tokens despite the contract explicitly documenting that no-ops are allowed.

## Finding Description
`supplyCollateralAndSellWithUnitsTarget` accepts `targetUnits` with no lower-bound check. When `targetUnits == 0`, the loop condition `filledUnits < targetUnits` is immediately false (both start at 0), so the loop body never executes and `filledSellerAssets` remains 0. The post-loop `require(filledUnits == targetUnits, OutOfOffers())` passes trivially (0 == 0). `referralFeeAssets = 0.mulDivDown(referralFeePct, WAD) = 0`, so the slippage check `require(0 - 0 >= minSellerAssets)` passes when `minSellerAssets == 0`. The referral transfer at line 167 is correctly guarded by `if (referralFeeAssets > 0)`, but the receiver transfer at line 168 has no equivalent guard and unconditionally calls `SafeTransferLib.safeTransfer(loanToken, receiver, 0)`. `SafeTransferLib.safeTransfer` makes a raw `.call()` to `token.transfer(receiver, 0)` and propagates any revert from the token via the assembly block. If the token reverts on zero-value transfers, the entire call reverts. The contract-level NatDev comment (`/// @dev No-ops are allowed.`) confirms `targetUnits == 0` is an intended valid input, making this a real behavioral defect rather than an invalid-input edge case.

## Impact Explanation
Any call to `supplyCollateralAndSellWithUnitsTarget` with `targetUnits == 0` and `minSellerAssets == 0` on a market whose `loanToken` reverts on zero-value transfers will always revert at line 168. The transaction fails atomically — collateral supplies are rolled back — so no funds are permanently lost. However, the no-op path (explicitly documented as supported) is rendered permanently unusable for such tokens, and callers waste gas on every attempt. Severity is low: no fund loss, no state corruption, but a concrete and repeatable denial of a documented feature.

## Likelihood Explanation
All three preconditions are independently reachable by any unprivileged caller: (1) `targetUnits == 0` has no on-chain guard; (2) `minSellerAssets == 0` is the natural default for no slippage protection; (3) tokens that revert on zero-value transfers are uncommon but real (e.g., certain USDT deployments, custom ERC-20s). The scenario is deterministic and repeatable — every call with these inputs on such a token will revert.

## Recommendation
Add a zero-value guard on line 168 matching the pattern already used on line 167:

```solidity
if (referralFeeAssets > 0) SafeTransferLib.safeTransfer(loanToken, referralFeeRecipient, referralFeeAssets);
uint256 receiverAssets = filledSellerAssets - referralFeeAssets;
if (receiverAssets > 0) SafeTransferLib.safeTransfer(loanToken, receiver, receiverAssets);
```

This makes the receiver transfer consistent with the referral transfer and eliminates the zero-value call on the no-op path.

## Proof of Concept
1. Deploy or fork a market whose `loanToken` reverts on `transfer(to, 0)`.
2. Call `supplyCollateralAndSellWithUnitsTarget` with `targetUnits = 0`, `minSellerAssets = 0`, an empty or irrelevant `takes` array (with at least one entry to avoid index-out-of-bounds at line 129), and empty `collateralSupplies`.
3. Observe: the takes loop is skipped, `filledSellerAssets == 0`, the `require` at line 163 passes, `referralFeeAssets == 0` so line 167 is skipped, but line 168 calls `safeTransfer(loanToken, receiver, 0)` which propagates the token's revert.
4. The entire transaction reverts despite `targetUnits == 0` being an explicitly documented valid no-op input.
