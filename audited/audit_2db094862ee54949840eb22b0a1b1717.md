Audit Report

## Title
Unbounded `tokens[]` array in `flashLoan` enables block-stuffing via linear gas growth - (File: src/Midnight.sol)

## Summary
`Midnight.flashLoan` accepts caller-supplied `tokens[]` and `assets[]` arrays with no upper-bound length cap. The function iterates both arrays twice, issuing an unconditional external ERC20 `call` per element via `SafeTransferLib`, with no zero-amount short-circuit. An unprivileged caller can supply an arbitrarily large array of a single attacker-deployed ERC20 with zero amounts — requiring no token balance — to consume near-block-gas-limit gas in a single transaction, enabling repeated block stuffing on L2 deployments at negligible cost.

## Finding Description
**Root cause:** `flashLoan` (lines 737–752, `src/Midnight.sol`) enforces only a length-equality check (`tokens.length == assets.length`, line 740) with no upper-bound cap. No `MAX_TOKENS` or equivalent constant exists anywhere in `src/` (confirmed: `ConstantsLib.sol` defines no such bound).

**Code path:**
1. Line 740: `require(tokens.length == assets.length, InconsistentInput())` — equality only, no cap.
2. Lines 742–744: first loop — `SafeTransferLib.safeTransfer(tokens[i], callback, assets[i])` for every `i`.
3. Lines 745–748: callback invocation passing full unbounded arrays.
4. Lines 749–751: second loop — `SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i])` for every `i`.

`SafeTransferLib.safeTransfer` (line 15, `src/libraries/SafeTransferLib.sol`) and `safeTransferFrom` (line 27) unconditionally issue an external `token.call(...)` regardless of `value`; there is no zero-amount short-circuit path.

**Exploit flow:**
1. Attacker deploys a minimal ERC20 that accepts 0-amount transfers and a callback contract that returns `CALLBACK_SUCCESS`.
2. Attacker calls `midnight.flashLoan(tokens_N, zeros_N, callback, "")` where `tokens_N` is N copies of the attacker ERC20 and `zeros_N` is N zeros.
3. The contract executes 2N unconditional external calls (~2,100 gas cold CALL + ERC20 overhead each).
4. At N = 1,000: ~2,000 external calls × ~3,000 gas ≈ 6M gas, approaching or exceeding typical L2 block gas limits.
5. The attacker pays only gas; no token balance is required.
6. The attack is repeatable every block.

**Why existing checks fail:** The sole guard is the length-equality check at line 740. There is no `MAX_TOKENS` constant, no `require(tokens.length <= MAX_TOKENS)`, and no zero-amount early exit in `SafeTransferLib`.

## Impact Explanation
An unprivileged caller can submit a single `flashLoan` transaction consuming gas proportional to `tokens.length` with no protocol-enforced ceiling. On L2 networks (Arbitrum, Optimism, Base) where gas prices are fractions of a cent, the cost to the attacker is negligible while the block-stuffing effect is concrete: legitimate user transactions are delayed or excluded for the duration of the attack. This constitutes service unavailability and severe degradation of protocol usability, a valid impact class per RESEARCHER.md ("Service unavailability or severe degradation under realistic attacker input").

## Likelihood Explanation
- **Preconditions:** None beyond deploying a trivial ERC20 (~50 lines) and a callback returning `CALLBACK_SUCCESS`. No funds, no position, no authorization required.
- **Feasibility:** Fully reachable on mainnet and all L2 deployments. Zero-amount transfers are universally accepted by standard ERC20 implementations.
- **Cost:** On L2s, gas costs are negligible; the attacker can sustain the attack indefinitely.
- **Repeatability:** The attack can be repeated every block at near-zero marginal cost on L2s.

## Recommendation
Add a `MAX_FLASH_LOAN_TOKENS` constant to `src/libraries/ConstantsLib.sol` (e.g., `uint256 constant MAX_FLASH_LOAN_TOKENS = 16`) and enforce it at the top of `flashLoan`:

```solidity
require(tokens.length <= MAX_FLASH_LOAN_TOKENS, TooManyTokens());
```

This bounds the worst-case gas consumption of a single `flashLoan` call to a fixed multiple of per-token cost, eliminating the linear-growth attack surface while preserving all legitimate multi-token flash loan use cases.

## Proof of Concept
**Minimal Foundry test plan:**
1. Deploy `MockERC20` with `transfer` and `transferFrom` returning `true` for any amount including 0.
2. Deploy `MockCallback` returning `CALLBACK_SUCCESS` from `onFlashLoan`.
3. Build `tokens_N` as an array of N copies of `MockERC20` address; `assets_N` as N zeros.
4. Call `midnight.flashLoan(tokens_N, assets_N, address(mockCallback), "")` and measure `gasleft()` delta.
5. Assert gas consumed scales linearly with N and that at N = 2,000 the transaction approaches the L2 block gas limit.
6. Confirm no token balance is required for the call to succeed.
