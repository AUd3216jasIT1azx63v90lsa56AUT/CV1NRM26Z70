### Title
Missing Deadline on `stake()` Allows Mempool-Delayed Execution to Permanently Lock Staker Funds with Zero Bonus - (File: src/ConfidencePool.sol)

### Summary
`stake()` in `ConfidencePool.sol` has no `deadline` parameter. A transaction submitted while the registry is in a pre-risk state (where withdrawal is still possible) can sit in the mempool and execute only after the registry has transitioned to `UNDER_ATTACK`, permanently disabling the staker's withdrawal rights and collapsing their bonus share to approximately zero — an outcome the staker never consented to.

### Finding Description
`stake()` records `block.timestamp` as the deposit entry time and calls `_observePoolState()` / `_assertDepositsAllowed()` at execution time, not at submission time. There is no `deadline` parameter to bound how stale the transaction may be when it lands. [1](#0-0) 

The concrete attack path:

1. Alice checks the registry: it is in `NEW_DEPLOYMENT`. Withdrawals are open. She submits `stake(amount)` with a low gas price.
2. The transaction sits in the mempool. The registry transitions to `UNDER_ATTACK`. Any pool interaction (or a permissionless `pokeRiskWindow()` call) seals `riskWindowStart = block.timestamp`.
3. Alice's transaction is eventually included. `_assertDepositsAllowed` passes — `UNDER_ATTACK` is explicitly allowed. [2](#0-1) 

4. Inside `stake()`, `riskWindowStart != 0`, so Alice's entry time is floored to `riskWindowStart` via `_clampUserSums`: [3](#0-2) 

5. Because Alice's entry is floored to `riskWindowStart ≈ T`, her k=2 score `(T − entry)² ≈ 0`, giving her a near-zero bonus share.
6. `withdraw()` is now permanently disabled for Alice because `riskWindowStart != 0`: [4](#0-3) 

Alice's principal is locked until pool resolution (potentially months away), she earns no bonus, and she had no mechanism to prevent this.

### Impact Explanation
A staker who submitted a transaction in good faith — having verified the registry was in a safe pre-risk state — can have their funds permanently locked with zero bonus and no withdrawal option. This is materially worse than the Portal.sol analog: Portal.sol at least provides `_minReceived` slippage protection; `stake()` provides no equivalent guard. The staker loses: (a) the ability to withdraw principal until resolution, (b) their entire expected bonus share, and (c) any opportunity cost over the resolution period.

### Likelihood Explanation
The scenario requires a registry state transition from a pre-risk state to `UNDER_ATTACK` while a `stake()` transaction is pending. This is a realistic condition: BattleChain agreements can transition at any time, and low-gas transactions routinely sit in mempools for extended periods. No privileged actor is required — the registry transition is a normal protocol event. The permissionless `pokeRiskWindow()` function means any third party can seal `riskWindowStart` the moment the registry transitions, maximizing the window in which a pending `stake()` lands in the wrong state. [5](#0-4) 

### Recommendation
Add a `deadline` parameter to `stake()` (and symmetrically to `contributeBonus()`) and revert if the transaction is executed after the deadline:

```diff
- function stake(uint256 amount) external nonReentrant whenPoolNotPaused {
+ function stake(uint256 amount, uint256 deadline) external nonReentrant whenPoolNotPaused {
+     if (block.timestamp > deadline) revert DeadlineExpired();
```

This gives stakers a trustless guarantee that their transaction cannot execute in a registry state they did not intend to accept.

    function _bonusShare(address u, uint256 userEligible) internal view returns (uint256) {
        if (snapshotTotalBonus == 0) return 0;
        // No observable risk → no bonus (see contract natspec).
        if (riskWindowStart == 0) return 0;
        uint256 T = outcomeFlaggedAt;

        // Underflow guards on both subtractions: globally the sum of squares is nonneg, but
        // truncation/rounding pathologies could push individual terms over.
        uint256 userPlus = T * T * userEligible + userSumStakeTimeSq[u];
        uint256 userMinus = 2 * T * userSumStakeTime[u];
        uint256 userScore = userPlus > userMinus ? userPlus - userMinus : 0;

        uint256 plus = T * T * snapshotTotalStaked + snapshotSumStakeTimeSq;
        uint256 minus = 2 * T * snapshotSumStakeTime;
        uint256 globalScore = plus > minus ? plus - minus : 0;

        if (globalScore == 0) {
            // No time elapsed in the risk window for anyone → fallback to amount-weighted.
            if (snapshotTotalStaked == 0) return 0;
            return Math.mulDiv(userEligible, snapshotTotalBonus, snapshotTotalStaked);
        }
        // mulDiv handles the final multiply-then-divide via 512-bit intermediates, so a very
        // large `snapshotTotalBonus` cannot push the numerator over uint256 before division.
        return Math.mulDiv(userScore, snapshotTotalBonus, globalScore);
    }
