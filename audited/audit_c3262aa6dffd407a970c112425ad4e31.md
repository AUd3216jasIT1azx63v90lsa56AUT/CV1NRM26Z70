Looking at the external report's root cause — instantaneous value distribution that is JIT-capturable — I need to find an analog in `ConfidencePool.sol`. The relevant mechanism is the `globalScore == 0` amount-weighted fallback in `_bonusShare`, combined with the ability to stake during `UNDER_ATTACK`.

### Title
JIT Bonus Capture via `globalScore == 0` Amount-Weighted Fallback When `riskWindowStart == riskWindowEnd` — (`src/ConfidencePool.sol`)

### Summary

When `riskWindowStart` and `riskWindowEnd` are both sealed in the same block, `globalScore` collapses to zero and `_bonusShare` falls back to a pure amount-weighted distribution. An MEV attacker can engineer this condition by sandwiching a registry `markProduction()` call: front-running with a large `stake()` (which sets `riskWindowStart`) and back-running with `pokeRiskWindow()` (which sets `riskWindowEnd`). Because both timestamps equal `block.timestamp`, every staker's effective `(T − entry)² = 0`, the k=2 weighting is neutralised, and the attacker captures a proportional share of the entire bonus pool — directly contradicting the DESIGN.md §3 guarantee that `UNDER_ATTACK` deposits earn "~zero bonus."

### Finding Description

The k=2 bonus formula computes each staker's score as `amount × (T − entryTime)²`. `_markRiskWindowStart()` eagerly resets the global accumulators so all pre-existing stakers are treated as entering at `riskWindowStart`:

```solidity
// ConfidencePool.sol L814-L815
sumStakeTime    = totalEligibleStake * t;
sumStakeTimeSq  = totalEligibleStake * t * t;
```

A staker who calls `stake()` while the registry is `UNDER_ATTACK` in the same block that `riskWindowStart` is first set records `newEntry = block.timestamp = riskWindowStart`. [1](#0-0) 

If `riskWindowEnd` is also sealed in the same block (e.g., via a back-run `pokeRiskWindow()` after `markProduction()` executes), then `T = outcomeFlaggedAt = riskWindowEnd = riskWindowStart`. The global score:

```
globalScore = T² × snapshotTotalStaked − 2T × snapshotSumStakeTime + snapshotSumStakeTimeSq
            = (T − riskWindowStart)² × snapshotTotalStaked
            = 0
```

`_bonusShare` then falls back to the amount-weighted path: [2](#0-1) 

```solidity
if (globalScore == 0) {
    if (snapshotTotalStaked == 0) return 0;
    return Math.mulDiv(userEligible, snapshotTotalBonus, snapshotTotalStaked);
}
```

The attacker's share is `received / snapshotTotalStaked × snapshotTotalBonus`. With sufficient capital the attacker can make this ratio arbitrarily close to 1.

**Attack flow (same-block sandwich):**

1. Attacker monitors the mempool for a `markProduction()` call on the registry while the registry is `UNDER_ATTACK`.
2. **Front-run:** attacker calls `stake(large_amount)`. `_observePoolState()` sees `UNDER_ATTACK`, calls `_markRiskWindowStart()` setting `riskWindowStart = block.timestamp`. The stake is recorded with `newEntry = block.timestamp`. [3](#0-2) 
3. `markProduction()` executes; registry transitions to `PRODUCTION`.
4. **Back-run:** attacker calls `pokeRiskWindow()`. `_observePoolState()` sees `PRODUCTION`, calls `_markRiskWindowEnd()` setting `riskWindowEnd = block.timestamp`. [4](#0-3) 
5. Now `riskWindowStart == riskWindowEnd == block.timestamp`. `globalScore = 0` for all stakers.
6. Moderator calls `flagOutcome(SURVIVED, ...)` → `outcomeFlaggedAt = riskWindowEnd`.
7. Attacker calls `claimSurvived()`: receives full stake back plus `received / totalStaked × totalBonus`. [5](#0-4) 

The attacker's principal is fully returned; the only cost is opportunity cost of locking capital until resolution. Legitimate long-term stakers who held through the entire risk window receive a diluted bonus share.

### Impact Explanation

Legitimate stakers who deposited early and bore the full risk window earn the same per-token bonus as a JIT attacker who entered in the final block. The entire bonus pool — contributed by the sponsor to incentivise long-term confidence — can be captured by a single MEV actor. This directly undermines the economic purpose of the k=2 time-weighting mechanism and degrades LP incentives.

### Likelihood Explanation

The condition requires `riskWindowStart == riskWindowEnd`, achievable whenever the registry transitions from `UNDER_ATTACK` to `PRODUCTION` (or `CORRUPTED`) within a single block. On EVM-compatible chains with public mempools, a `markProduction()` call is visible before inclusion, making the sandwich straightforward. The `pokeRiskWindow()` function is permissionless and callable by anyone, so the attacker controls both legs of the sandwich without any privileged access. [6](#0-5) 

The DESIGN.md §3 explicitly states that `UNDER_ATTACK` deposits have "no late-join *advantage* to gate against" — but this is precisely what the `globalScore == 0` fallback creates when engineered via a same-block sandwich. The stated guarantee ("bonus share is crushed to ~zero") is violated.

### Recommendation

When `globalScore == 0`, instead of falling back to amount-weighted distribution, treat the bonus as unclaimable (return 0) and let it sweep to `recoveryAddress` via `sweepUnclaimedBonus()`. This is consistent with the "no observable risk → no bonus" rule already applied when `riskWindowStart == 0`: a risk window of zero duration is economically equivalent to no observable risk window.

Alternatively, require a minimum elapsed time between `riskWindowStart` and `riskWindowEnd` before the amount-weighted fallback activates, or cap the fallback to exclude stakes entered in the same block as `riskWindowStart`.

### Proof of Concept

```solidity

// - Pool has existing stakers with totalEligibleStake = T_existing
// - totalBonus = B (contributed by sponsor)
// - Registry is UNDER_ATTACK; riskWindowStart == 0

pool.stake(LARGE_AMOUNT);

// sumStakeTime    = (T_existing + LARGE_AMOUNT) * riskWindowStart
// sumStakeTimeSq  = (T_existing + LARGE_AMOUNT) * riskWindowStart²

// Block N, Tx 2: markProduction() on registry → state = PRODUCTION

// Block N, Tx 3 (attacker back-runs):
pool.pokeRiskWindow();
// _observePoolState() → PRODUCTION → _markRiskWindowEnd() → riskWindowEnd = block.timestamp
// riskWindowStart == riskWindowEnd == block.timestamp

// Later: moderator flags SURVIVED
// outcomeFlaggedAt = riskWindowEnd = riskWindowStart

// At claimSurvived():
// T = outcomeFlaggedAt = riskWindowStart
// globalScore = (T - riskWindowStart)² × snapshotTotalStaked = 0
// → amount-weighted fallback
// attacker bonus = LARGE_AMOUNT / (T_existing + LARGE_AMOUNT) × B
// → approaches B as LARGE_AMOUNT >> T_existing
// attacker also recovers full LARGE_AMOUNT principal
```
