### Title
Zero-`buyerAssets` rounding in `maxAssets`-mode buy-offer consumed accounting allows indefinite offer reuse — (`src/Midnight.sol`)

### Summary
When a buy offer is configured with `offer.maxAssets > 0`, the `take` function tracks capacity consumption by incrementing `consumed[maker][group]` by `buyerAssets = units.mulDivDown(buyerPrice, WAD)`. Because every valid tick price satisfies `buyerPrice < WAD`, filling with `units = 1` always produces `buyerAssets = 0`, so `consumed` never advances. The `ConsumedAssets` guard is therefore never triggered, the offer can be filled an unlimited number of times, and the maker accumulates credit units without transferring any loan tokens.

### Finding Description

**Root cause — `src/Midnight.sol` lines 363 and 367–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds DOWN
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
``` [1](#0-0) 

`buyerPrice` equals `offerPrice = TickLib.tickToPrice(offer.tick)`. From `TickLib.sol`:

```solidity
uint256 constant PRICE_ROUNDING_STEP = 1e12;
uint256 constant MAX_TICK = 5820;
``` [2](#0-1) 

The `tickToPrice` formula produces values in the range `[0, ~9.99999e17]`; the maximum representable price at `tick = MAX_TICK` is approximately `9.99999e17 < 1e18 = WAD`. [3](#0-2) 

Therefore, for **any** valid tick and `units = 1`:

```
buyerAssets = 1 * buyerPrice / 1e18 = 0   (integer division, buyerPrice < 1e18)
```

`mulDivDown` is `(x * y) / d` with no rounding-up: [4](#0-3) 

**Exploit flow:**

1. Maker creates a buy offer: `offer.buy = true`, `offer.maxAssets = M` (any positive value), `offer.tick` = any valid tick (e.g., `MAX_TICK`), valid `expiry`, authorized `ratifier`.
2. Colluding taker (a second address controlled by the maker, or any willing counterparty) calls `take(offer, ..., units=1, ...)` in a loop N times.
3. Each iteration:
   - `buyerAssets = 1.mulDivDown(buyerPrice, 1e18) = 0`
   - `consumed[maker][group] += 0` → stays at 0
   - `require(0 <= M)` → always passes
   - `buyerPos.credit += 1` (maker gains 1 unit of credit)
   - `sellerPos.debt += 1` (taker incurs 1 unit of debt)
   - `safeTransferFrom(..., 0)` and `safeTransferFrom(..., 0)` → no tokens move
4. After N iterations: `consumed = 0`, maker credit = N, taker debt = N, loan tokens paid = 0.

**Why existing checks fail:**

- `require(newConsumed <= offer.maxAssets)` — `newConsumed` is always 0, so this never fires.
- No `require(buyerAssets > 0)` guard exists anywhere in the path.
- No minimum `units` check exists.
- The taker must pass the health check at line 476 (`isHealthy`), so they must supply collateral — but this only limits the taker's debt capacity, not the number of fills. [5](#0-4) 

**Certora spec gap:** The `takeConsumedDelta` rule that verifies `consumed` advances by the fill amount is scoped exclusively to `maxUnits` mode (`require offer.maxAssets == 0`). No equivalent rule exists for `maxAssets` mode. Additionally, `mulDivDown` is summarized as `NONDET` in the spec, so rounding to zero is never modeled. [6](#0-5) 

### Impact Explanation

The `maxAssets` capacity limit on buy offers is completely unenforced when `units = 1`. A maker can accumulate an unbounded amount of credit (redeemable for loan tokens at maturity) while paying zero loan tokens upfront. The `consumed` mapping — the sole mechanism preventing offer overfill — stays permanently at 0, violating the core invariant that "offers cannot be replayed, overfilled, or reused."

### Likelihood Explanation

**Preconditions:**
- Any buy offer with `maxAssets > 0` at any valid tick (the condition holds for all 5821 possible ticks).
- A colluding taker (the maker's own second address suffices) who supplies enough collateral to remain healthy under the accumulated debt.
- No special permissions, governance access, or oracle manipulation required.

**Feasibility:** Trivially repeatable in a loop. The only cost is gas and the collateral the taker must lock. The maker's credit gain is unbounded.

### Recommendation

Enforce a minimum non-zero asset transfer per fill when `offer.maxAssets > 0` and `offer.buy = true`. The simplest targeted fix is to require that `buyerAssets > 0` whenever `units > 0`:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || buyerAssets > 0, ZeroBuyerAssets());
    newConsumed = consumed[offer.maker][offer.group] += buyerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());
}
```

Alternatively, use `mulDivUp` for the consumed increment in buy-offer mode so that even a 1-unit fill consumes at least 1 asset from the budget:

```solidity
newConsumed = consumed[offer.maker][offer.group] +=
    offer.buy ? units.mulDivUp(buyerPrice, WAD) : sellerAssets;
```

This mirrors the existing `mulDivUp` already used for sell offers and ensures the consumed counter is always a conservative (over-)estimate of assets used.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.34;

import "forge-std/Test.sol";
import {Midnight, IMidnight, Offer, Market} from "src/Midnight.sol";
import {TickLib, MAX_TICK} from "src/libraries/TickLib.sol";

contract BuyOfferConsumedZeroTest is Test {
    Midnight midnight;
    address maker  = address(0xA);
    address taker  = address(0xB);   // colluding second address
    MockERC20 loan;
    MockRatifier ratifier;
    Market market;
    bytes32 group = keccak256("g");

    function setUp() public {
        // deploy protocol, create market with 0 settlement fees
        // collateralize taker so health check passes
    }

    function testConsumedStaysZero() public {
        uint256 maxAssets = 1000;
        Offer memory offer = Offer({
            market:   market,
            buy:      true,
            maker:    maker,
            tick:     MAX_TICK,          // price ≈ 9.99999e17 < WAD
            maxAssets: maxAssets,
            maxUnits: 0,
            group:    group,
            expiry:   block.timestamp + 1 days,
            start:    0,
            ratifier: address(ratifier),
            // ... other fields
        });

        // Taker fills 1000 times with units=1
        for (uint256 i; i < 1000; i++) {
            vm.prank(taker);
            midnight.take(offer, "", 1, taker, taker, address(0), "");
        }

        // Key assertions
        assertEq(midnight.consumed(maker, group), 0,    "consumed must be 0");
        assertEq(midnight.creditOf(id, maker),    1000, "maker credit must be 1000");
        assertEq(midnight.debtOf(id, taker),      1000, "taker debt must be 1000");
        assertEq(loan.balanceOf(address(midnight)), 0,  "no tokens transferred");
    }
}
```

**Expected assertions:** `consumed == 0` after 1000 fills, `creditOf(maker) == 1000`, `debtOf(taker) == 1000`, zero loan-token transfers — confirming the offer was filled 1000× beyond its intended `maxAssets = 1000` budget while the maker paid nothing.
