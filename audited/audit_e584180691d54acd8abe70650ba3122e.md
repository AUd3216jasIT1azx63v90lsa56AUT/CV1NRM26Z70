### Title
`sellerAssetsToUnits` Division-by-Zero DoS When `offerPrice == settlementFee` for Buy Offers - (`src/periphery/TakeAmountsLib.sol`)

### Summary

`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and then calls `mulDivUp(targetSellerAssets, WAD, sellerPrice)`. When `offerPrice == settlementFee`, `sellerPrice = 0`, and `mulDivUp` reverts with an arithmetic error (underflow on `d - 1` when `d = 0`). Because `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` calls `sellerAssetsToUnits` outside any try/catch, the entire bundler call reverts. Crucially, `Midnight.take` itself handles `sellerPrice = 0` gracefully (computing `sellerAssets = 0`), so the offer remains takeable directly but not through the bundler.

### Finding Description

**Exact code path:**

`TakeAmountsLib.sellerAssetsToUnits` (lines 36–47): [1](#0-0) 

```
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : ...
```

`UtilsLib.mulDivUp` (line 35): [2](#0-1) 

```
return (x * y + (d - 1)) / d;
```

When `d = 0`, `d - 1` underflows (Solidity 0.8 checked arithmetic) → arithmetic revert.

**Reachability of `sellerPrice = 0`:**

- `tickToPrice(0) = 0` (proven by spec rule `tickToPriceIsZeroAtZero`)
- `tickToPrice(2) = 1e12` (first non-zero price, confirmed by `testTickToPriceMinMax`)
- Settlement fee is stored as `uint16 * CBP` where `CBP = 1e12`, so the minimum non-zero fee is `1e12`
- `MAX_SETTLEMENT_FEE_1_DAY = 0.000014e18 = 14000e12`, so `1e12` is a valid fee [3](#0-2) [4](#0-3) 

A buy offer at tick=2 (`offerPrice = 1e12`) with settlement fee set to `1e12` gives `sellerPrice = 0`. Both values are multiples of `1e12`, so exact equality is achievable.

**Why `Midnight.take` does NOT revert:** [5](#0-4) 

```
uint256 sellerPrice = offer.buy ? offerPrice - _settlementFee : offerPrice;
uint256 sellerAssets = offer.buy ? units.mulDivDown(sellerPrice, WAD) : ...
```

`mulDivDown(0, WAD) = 0` — no division by zero. The test `testPriceZeroWithSettlementFeeSell` confirms that zero `sellerAssets` is a valid outcome. [6](#0-5) 

**Where the bundler reverts (not caught):** [7](#0-6) 

```solidity
uint256 unitsToTake = min(
    TakeAmountsLib.sellerAssetsToUnits(   // ← reverts here, NOT in try/catch
        MIDNIGHT, id, takes[i].offer, targetFilledSellerAssets - filledSellerAssets
    ),
    takes[i].units,
    ConsumableUnitsLib.consumableUnits(MIDNIGHT, id, takes[i].offer)
);
```

The NatDoc explicitly warns: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts." [8](#0-7) 

Note: `ConsumableUnitsLib.consumableUnits` for a buy offer calls `buyerAssetsToUnits` (not `sellerAssetsToUnits`), so it does not revert independently. The revert comes solely from the direct `sellerAssetsToUnits` call. [9](#0-8) 

**Attacker inputs:** A lender (unprivileged maker) creates a buy offer at tick=2. If the settlement fee is `1e12` (a valid, normal admin setting), any borrower calling `supplyCollateralAndSellWithAssetsTarget` with that offer will trigger the revert.

**Existing protections insufficient:** `buyerAssetsToUnits` has a guard (`offerPrice - settlementFee` underflows when `offerPrice < settlementFee`), but `sellerAssetsToUnits` has no guard for the `offerPrice == settlementFee` edge case. The NatDoc for `sellerAssetsToUnits` only documents the `offerPrice < settlementFee` revert, not the `offerPrice == settlementFee` case. [10](#0-9) 

### Impact Explanation

`MidnightBundles.supplyCollateralAndSellWithAssetsTarget` permanently reverts for any buy offer where `offerPrice == settlementFee`, regardless of how many units are requested. Borrowers who rely on the bundler for atomic collateral supply + borrow cannot execute their borrow flow for such offers. The offer remains takeable via `Midnight.take` directly, so the DoS is scoped to the bundler. This constitutes a fund freeze for borrowers who depend on the bundler's atomic flow (e.g., for collateral supply + borrow in one transaction).

### Likelihood Explanation

**Preconditions:**
1. A buy offer exists at a tick where `tickToPrice(tick) == settlementFee` (e.g., tick=2, price=1e12, fee=1e12).
2. The offer uses `maxAssets > 0` (so `consumableUnits` calls `buyerAssetsToUnits`) or `maxUnits > 0` (so `consumableUnits` returns normally), but the direct `sellerAssetsToUnits` call still reverts.
3. A borrower calls `supplyCollateralAndSellWithAssetsTarget` with `targetSellerAssets > 0`.

The condition is an exact equality between two `1e12`-granularity values. It is achievable at the minimum tick (tick=2, price=1e12) with the minimum non-zero fee (1e12). The fee setter can set this legitimately. A lender can deliberately place an offer at this tick to DoS the bundler for other borrowers. The condition is repeatable and permanent until the fee or offer changes.

### Recommendation

Add a guard in `sellerAssetsToUnits` for the `sellerPrice == 0` case. When `sellerPrice == 0`, any nonzero `targetSellerAssets` is unreachable (since `sellerAssets = units * 0 / WAD = 0` for all units). Return `type(uint256).max` to signal "unreachable target" (the bundler's `min(...)` will then bound it by `takes[i].units` and `consumableUnits`, and the offer will be skipped via the `OutOfOffers` path):

```solidity
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
if (sellerPrice == 0) return type(uint256).max; // sellerAssets = 0 for any units; target unreachable
return offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : targetSellerAssets.mulDivDown(WAD, sellerPrice);
```

Alternatively, mirror the `buyerAssetsToUnits` approach and add a `require(sellerPrice > 0 || targetSellerAssets == 0)` guard with a descriptive error.

### Proof of Concept

```solidity
function testSellerAssetsToUnitsZeroSellerPriceReverts() public {
    // Set settlement fee to 1e12 (1 CBP, valid fee)
    midnight.touchMarket(market);
    midnight.setMarketSettlementFee(id, 1, 1e12);

    // Buy offer at tick=2: tickToPrice(2) = 1e12 == settlementFee → sellerPrice = 0
    lenderOffer.tick = 2;
    lenderOffer.buy = true;
    lenderOffer.maxUnits = type(uint256).max;

    // Assert sellerAssetsToUnits reverts (division by zero / arithmetic error)
    vm.expectRevert(stdError.arithmeticError);
    TakeAmountsLib.sellerAssetsToUnits(address(midnight), id, lenderOffer, 1e18);

    // Assert Midnight.take with the same offer and nonzero units does NOT revert
    collateralize(market, borrower, 1e18);
    vm.prank(borrower);
    (uint256 buyerAssets, uint256 sellerAssets) = midnight.take(
        lenderOffer, hex"", 1e18, borrower, borrower, address(0), hex""
    );
    assertEq(sellerAssets, 0, "sellerAssets should be 0");
    assertGt(buyerAssets, 0, "buyerAssets should be nonzero");

    // Assert supplyCollateralAndSellWithAssetsTarget reverts (DoS)
    Take[] memory takes = new Take[](1);
    takes[0] = Take({offer: lenderOffer, units: type(uint256).max, ratifierData: hex""});
    collateralize(market, borrower, 1e18);
    vm.prank(borrower);
    vm.expectRevert(stdError.arithmeticError);
    midnightBundles.supplyCollateralAndSellWithAssetsTarget(
        1e18, type(uint256).max, borrower, borrower,
        new CollateralSupply[](0), takes, 0, address(0)
    );
}
```

**Expected assertions:**
- `sellerAssetsToUnits` reverts with arithmetic error
- `Midnight.take` succeeds with `sellerAssets = 0`
- `supplyCollateralAndSellWithAssetsTarget` reverts (DoS confirmed)
