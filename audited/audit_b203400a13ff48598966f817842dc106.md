### Title
Borrower Can Permanently Block Their Own Liquidation by Activating a Reverting Oracle's Collateral — (`src/Midnight.sol`)

### Summary

`liquidate()` and `isHealthy()` both loop over every activated collateral in a borrower's `collateralBitmap` and call `IOracle(oracle).price()` for each. `supplyCollateral()` does not call any oracle. A borrower can therefore activate a collateral whose oracle is reverting by supplying even 1 wei of it, permanently blocking all liquidation attempts against their position.

### Finding Description

**Root cause — asymmetric oracle call coverage between `supplyCollateral` and `liquidate`/`isHealthy`:**

`supplyCollateral()` activates a collateral index in the borrower's bitmap with no oracle call: [1](#0-0) 

`liquidate()` then iterates over every bit in `_collateralBitmap` and calls `IOracle(oracle).price()` unconditionally for each: [2](#0-1) 

`isHealthy()` has the identical loop structure: [3](#0-2) 

`isHealthy()` is called at the end of `take()` for the seller and inside `withdrawCollateral()`: [4](#0-3) [5](#0-4) 

**Exploit flow:**

1. A multi-collateral market exists with collateral index `j` whose oracle can revert (circuit breaker, stale-data guard, upstream feed failure, etc.).
2. A borrower's position is approaching insolvency.
3. The borrower calls `supplyCollateral(market, j, 1, self)` — 1 wei of the collateral token — setting bit `j` in their `collateralBitmap`. No oracle is consulted.
4. The oracle for index `j` is (or becomes) reverting.
5. Every subsequent call to `liquidate()` for this borrower hits the loop, reaches index `j`, calls `IOracle(oracle).price()`, and reverts.
6. `isHealthy()` also reverts for this borrower, so `take()` (when the borrower is the seller with debt) and `withdrawCollateral()` also revert.
7. The borrower's position is permanently unliquidatable for as long as the oracle reverts.

The protocol's own LIVENESS section and MULTI-COLLATERAL MARKETS section explicitly document this gap: [6](#0-5) [7](#0-6) 

### Impact Explanation

A borrower with an underwater position avoids liquidation indefinitely. Their debt remains on the books while their collateral value falls. When the oracle eventually recovers (or the market matures), the position may be deeply insolvent, forcing bad-debt socialization via `lossFactor` onto all lenders in the market. In the worst case — many borrowers exploiting this simultaneously or a long oracle outage — the market's `lossFactor` can be driven toward `type(uint128).max`, at which point `take()` reverts for the entire market (`MarketLossFactorMaxedOut`), freezing all lending activity. [8](#0-7) 

### Likelihood Explanation

The preconditions are:
- A multi-collateral market (standard design; markets support up to 128 collateral types).
- At least one collateral oracle that can revert — realistic for any oracle with a circuit breaker, a TWAP deviation guard, or a dependency on an external feed that can go stale.
- The borrower must act before being liquidated (one transaction after the oracle starts reverting).

No privileged access is required. The borrower is a normal user executing a standard protocol flow (`supplyCollateral`). The `CollateralParams.oracle` field is an arbitrary `address` set at market creation; any oracle implementation that can revert satisfies the condition. [9](#0-8) [10](#0-9) 

### Recommendation

Wrap the `IOracle(oracle).price()` call inside `liquidate()` and `isHealthy()` in a `try/catch`. On revert, either skip that collateral (treating its contribution as zero, conservative for the borrower) or treat the position as immediately liquidatable. A simpler alternative is to allow liquidators to pass an explicit subset of collateral indices to evaluate, bypassing broken oracles for the purpose of seizing other healthy collaterals.

### Proof of Concept

```
Setup:
  market.collateralParams = [
    { token: WETH, oracle: healthyOracle, lltv: 0.8e18, ... },   // index 0
    { token: EXOTIC, oracle: breakableOracle, lltv: 0.5e18, ... } // index 1
  ]

Step 1: Borrower supplies WETH (index 0) and borrows via take().
        position[id][borrower].collateralBitmap = 0b01  (only index 0 active)

Step 2: breakableOracle.price() begins reverting (e.g., circuit breaker trips).

Step 3: Borrower calls supplyCollateral(market, 1, 1 wei, borrower).
        → No oracle call in supplyCollateral.
        → position[id][borrower].collateralBitmap = 0b11  (both indices active)

Step 4: Borrower's WETH collateral drops; position becomes unhealthy.

Step 5: Liquidator calls liquidate(market, 0, 0, repaidUnits, borrower, ...).
        → Loop hits i=1, calls breakableOracle.price() → REVERT.
        → Entire liquidate() reverts.

Step 6: Borrower is unliquidatable for the duration of the oracle outage.
        Bad debt accrues; lossFactor increases; lenders are slashed.
``` [11](#0-10) [12](#0-11)

    function supplyCollateral(Market memory market, uint256 collateralIndex, uint256 assets, address onBehalf)
        external
    {
        require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
        bytes32 id = touchMarket(market);
        address collateralToken = market.collateralParams[collateralIndex].token;

        Position storage _position = position[id][onBehalf];
        uint256 oldCollateral = _position.collateral[collateralIndex];
        _position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);

        if (oldCollateral == 0 && assets > 0) {
            uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
            _position.collateralBitmap = newCollateralBitmap;
            require(
                UtilsLib.countBits(newCollateralBitmap) <= MAX_COLLATERALS_PER_BORROWER, TooManyActivatedCollaterals()
            );
        }
