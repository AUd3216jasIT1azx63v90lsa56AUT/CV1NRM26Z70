### Title
Lenders Can Front-Run Bad Debt Realization to Avoid Socialized Losses — (File: src/Midnight.sol)

---

### Summary

Morpho Midnight socializes bad debt among all lenders by updating `marketState.lossFactor` inside `liquidate()`. Because credit units are freely tradeable via `take()` at any time, a lender who observes a pending bad-debt liquidation in the mempool can front-run it — selling their credit units before `lossFactor` is updated. The buyer receives credit units that are immediately slashed by the subsequent liquidation, while the seller exits at the pre-loss price. This violates the protocol's core invariant that bad debt reduces all lenders' credit proportionally.

---

### Finding Description

**Root cause**: `lossFactor` is written in `liquidate()` and read lazily in `_updatePosition()`. There is no atomicity guarantee between a bad-debt realization and a concurrent credit sale via `take()`.

**Code path**:

`liquidate()` computes and writes the new `lossFactor` only when it executes: [1](#0-0) 

`take()` calls `_updatePosition()` for the seller, which reads the *current* (pre-liquidation) `lossFactor` and stamps `lastLossFactor` to that value: [2](#0-1) 

Inside `_updatePosition()` → `updatePositionView()`, the slash is computed as:

```
postSlashCredit = credit * (MAX - currentLossFactor) / (MAX - lastLossFactor)
``` [3](#0-2) 

If `currentLossFactor` has not yet been updated (liquidation is still pending), the seller's credit is not slashed. After `_updatePosition()` stamps `lastLossFactor = currentLossFactor` (pre-loss value): [4](#0-3) 

…the seller's position is permanently marked as "already accounted for" at the pre-loss factor. When the liquidation then executes and raises `lossFactor`, only the *buyer's* position (whose `lastLossFactor` was also set to the pre-loss value at purchase time) will be slashed at their next `_updatePosition()` call.

**Exploit flow**:

1. Borrower B has bad debt: collateral value < debt, so `badDebt > 0` in `liquidate()`.
2. Liquidator L broadcasts `liquidate(B, ...)` — visible in the public mempool.
3. Lender A (holding 100 credit units, `lastLossFactor = 0`) observes the pending tx.
4. Lender A front-runs by calling `take()` to sell 100 credit units to Buyer C (who has a resting buy offer at price ≈ 1:1).
5. Inside `take()`, `_updatePosition()` runs for Lender A: `currentLossFactor = 0`, so `postSlashCredit = 100`. Lender A's `lastLossFactor` is stamped to `0`. Lender A receives 100 loan tokens.
6. Buyer C's position: `credit += 100`, `lastLossFactor = 0`.
7. Liquidation executes: `lossFactor` is updated to reflect `badDebt = 10` out of `totalUnits = 100`.
8. Buyer C calls `updatePosition()`: `postSlashCredit = 100 * (MAX - newLossFactor) / (MAX - 0) ≈ 90`. Buyer C loses 10 loan tokens.
9. Lender A has exited at full value; the loss is entirely borne by Buyer C.

---

### Impact Explanation

Lender A avoids the bad debt loss entirely and transfers it to Buyer C. The protocol's stated invariant — `bad_debt_reduces_lenders_credit_proportionally: true` — is violated for the exiting lender. Buyer C suffers a direct, immediate loss of funds equal to the bad debt proportional to their newly acquired credit. The loss is not distributed among all lenders as intended; it is concentrated on the uninformed buyer of the resting offer. [5](#0-4) 

---

### Likelihood Explanation

The attack requires no privileged access:

- The attacker only needs to monitor the public mempool for `liquidate()` transactions where `badDebt > 0` (detectable by simulating the call).
- A resting buy offer (maker) is a normal market condition in any active lending market.
- The attacker submits `take()` with a higher gas price to front-run the liquidation.
- The attack is profitable whenever `badDebt_loss_to_attacker > gas_cost_of_front_run`, which is true for any non-trivial bad debt event.
- The attacker profile is an unprivileged lender — explicitly listed as a valid attacker in `live_context.json`. [6](#0-5) 

---

### Recommendation

This is an inherent tension between the offer-book model (freely tradeable credit units) and the loss-socialization mechanism. Mitigations in increasing order of protocol change:

1. **Off-chain monitoring / UI warnings**: Warn makers with resting buy offers when a bad-debt liquidation is pending for the market.
2. **Offer-level `minLossFactor` guard**: Allow makers to embed a `maxLossFactor` assertion in their offer's ratifier data. The ratifier rejects the fill if `marketState[id].lossFactor` exceeds the maker's tolerance, preventing uninformed buyers from being front-run.
3. **Snapshot `lossFactor` at trade time for the buyer**: When a buyer acquires credit units, set their `lastLossFactor` to the *post-trade* `lossFactor` rather than the pre-trade value, so any loss realized in the same block is immediately reflected. (This is a more invasive change to `_updatePosition` ordering.)

// Foundry test sketch
function test_frontRunBadDebt() public {
    // 1. Setup: lenderA buys 100 credit units; borrowerB borrows 100 units with collateral
    // 2. Oracle price drops: borrowerB's collateral now worth 90 → badDebt = 10
    // 3. LiquidatorL prepares liquidate() tx (not yet submitted)

    // 4. LenderA front-runs: sells 100 credit units to buyerC via take()
    vm.prank(lenderA);
    midnight.take(sellOffer, ratifierData, 100, lenderA, lenderA, address(0), "");
    // lenderA receives 100 loan tokens; buyerC.credit = 100, lastLossFactor = 0

    // 5. Liquidation executes
    vm.prank(liquidatorL);
    midnight.liquidate(market, 0, 0, 0, borrowerB, false, liquidatorL, address(0), "");
    // lossFactor updated: reflects 10/100 bad debt

    // 6. BuyerC's credit is now slashed
    (uint128 newCredit,,) = midnight.updatePositionView(market, id, buyerC);
    assertLt(newCredit, 100); // buyerC lost ~10 units
    // lenderA kept all 100 loan tokens — loss fully transferred
}
