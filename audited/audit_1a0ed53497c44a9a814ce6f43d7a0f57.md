Audit Report

## Title
Fee-on-Transfer Collateral Token Inflates Recorded Collateral Beyond Actual Balance - (File: src/Midnight.sol)

## Summary
`supplyCollateral` records the full caller-supplied `assets` amount into `_position.collateral[collateralIndex]` at line 533 before executing the token transfer at line 545, with no pre/post balance measurement. When a fee-on-transfer ERC20 is used as collateral, the protocol records `assets` but receives only `assets * (1 - fee)`, creating phantom collateral. `isHealthy` uses the inflated recorded value to compute `maxDebt`, allowing the attacker to open debt positions that exceed the actual token balance held by the contract, resulting in immediate undercollateralization and bad debt socialized across lenders.

## Finding Description
**Root cause — `src/Midnight.sol` lines 524–546:**

```solidity
// line 533: records full `assets` unconditionally, before transfer
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + assets);

// line 545: transfer may deliver less than `assets` if token charges a fee
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
```

`SafeTransferLib.safeTransferFrom` only verifies the call did not revert and the return value (if any) is `true`; it performs no pre/post balance comparison.

**`touchMarket` imposes no token-type restriction** (`src/Midnight.sol` lines 755–791): validation is limited to LLTV tier membership, sorted collateral addresses, and `maxLif` values. Any ERC20 address — including a fee-on-transfer token — is accepted as a valid collateral token.

**`isHealthy` uses the recorded (inflated) collateral value directly** (`src/Midnight.sol` lines 953–955):

```solidity
maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
    .mulDivDown(collateralParam.lltv, WAD);
```

**Formal verification explicitly excludes this case.** `certora/specs/Solvency.spec` line 31 states: *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits."* The `CVL_transferFrom` summary (lines 43–59) transfers the exact amount specified, so the `tokenBalanceCorrect` solvency invariant is not verified for fee-on-transfer tokens. This is an internal prover assumption, not a documented protocol restriction visible to market creators.

**Exploit flow:**
1. Attacker deploys a fee-on-transfer ERC20 (e.g., 10% fee) and calls `touchMarket` to create a permissionless Midnight market with it as collateral.
2. Attacker calls `supplyCollateral(market, 0, 1000e18, attacker)`.
3. `_position.collateral[0]` is set to `1000e18`; Midnight receives only `900e18`.
4. `isHealthy` computes `maxDebt` using `1000e18`, allowing the attacker to borrow up to `1000e18 * price * lltv / WAD`.
5. Actual collateral backing is only `900e18 * price * lltv / WAD` — the position is immediately undercollateralized.
6. On liquidation, the protocol attempts to seize up to `1000e18` tokens but only holds `900e18`, causing a shortfall and bad debt socialized across lenders via the loss-factor mechanism.

**Existing checks are insufficient:** `SECURITY.md` contains no exclusion for fee-on-transfer tokens. `touchMarket` has no token allowlist or denylist. `supplyCollateral` has no balance-delta guard.

## Impact Explanation
The solvency invariant `tokenBalances[token][currentContract] >= collateralSum(token) + withdrawableSum(token) + claimableSettlementFee(token)` is violated. Phantom collateral allows the attacker to open debt positions exceeding the actual token balance held by the contract. On liquidation, the seized amount cannot be fully transferred out, leaving bad debt that is socialized across lenders via the loss-factor mechanism. This constitutes direct theft of lender funds and protocol insolvency for the affected market.

## Likelihood Explanation
Market creation is permissionless — any address can call `touchMarket` with a fee-on-transfer token as collateral. The attacker only needs to be the `onBehalf` address (or authorized for it), which is trivially satisfied by self-authorization (`isAuthorized[onBehalf][msg.sender]`). The attack is repeatable: each `supplyCollateral` call inflates collateral by `fee * assets`. Fee-on-transfer tokens exist in production (e.g., USDT with fee enabled, STA, PAXG). No admin action is required.

## Recommendation
Measure the actual received amount using pre/post balance checks in `supplyCollateral`:

```solidity
uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;
_position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
```

Alternatively, document and enforce (via `touchMarket` validation or a token allowlist) that fee-on-transfer tokens are not supported as collateral.

## Proof of Concept
1. Deploy a mock ERC20 with a 10% transfer fee.
2. Call `touchMarket` with the fee token as collateral (any LLTV tier, valid `maxLif`).
3. Call `supplyCollateral(market, 0, 1000e18, attacker)`.
4. Assert `position[id][attacker].collateral[0] == 1000e18` and `feeToken.balanceOf(address(midnight)) == 900e18`.
5. Call `borrow` up to the `maxDebt` computed from `1000e18`; assert the borrow succeeds.
6. Assert `isHealthy` returns `true` despite the position being undercollateralized by 100e18 tokens.
7. Attempt liquidation and observe the seized-amount transfer fails or leaves a shortfall, confirming bad debt.
