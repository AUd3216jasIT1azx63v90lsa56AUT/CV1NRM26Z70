### Title
Fee-on-transfer loan token in `repay()` inflates `withdrawable` unchecked, enabling `claimContinuousFee` to drain contract below lender redemption threshold - (File: `src/libraries/SafeTransferLib.sol`)

### Summary
`repay()` unconditionally increments `marketState[id].withdrawable` by the caller-supplied `units` parameter, then delegates the inbound transfer to `SafeTransferLib.safeTransferFrom`, which only verifies the boolean return value and never measures the actual balance delta. With a fee-on-transfer loan token every repay call silently inflates `withdrawable` by the fee amount. When the `feeClaimer` subsequently calls `claimContinuousFee`, it passes both the `continuousFeeCredit` and the inflated `withdrawable` guards and transfers real tokens out, leaving the contract's balance below what lenders can redeem.

`withdrawable` is incremented by the nominal `units` value before the transfer executes. [1](#0-0) 

**`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24‚Äď34):**

The library calls `transferFrom` and checks only the boolean return value. It performs no pre/post balance comparison. With a fee-on-transfer token the contract receives `units ‚ąí fee` but the check passes and returns normally. [2](#0-1) 

**`claimContinuousFee` (`src/Midnight.sol` lines 312‚Äď325):**


Both guards are underflow-revert checks. `continuousFeeCredit` is correctly bounded by accrued fees (set only in `_updatePosition` line 846). `withdrawable`, however, is inflated by the cumulative FOT delta, so guard 2 passes even when the contract does not hold enough tokens to satisfy both the fee claim and outstanding lender credits. **`_updatePosition` ‚ÄĒ `continuousFeeCredit` source (`src/Midnight.sol` line 846):**

```solidity

```

`continuousFeeCredit` grows only from lender-credit accrual, never from repayments. It is not inflated by FOT tokens. [4](#0-3) 

**Exploit flow (concrete numbers, 1 % FOT fee):**

| Step | Action | `withdrawable` | `continuousFeeCredit` | Contract balance |
|------|--------|---------------|----------------------|-----------------|
| 0 | Lender deposits 1 000 via `take()` | 0 | 0 | 0 |
| 1 | Fee accrues | 0 | 100 | 0 |
| 2 | Borrower `repay(1000)` with FOT | **1 000** | 100 | **990** |
| 3 | `feeClaimer` `claimContinuousFee(100)` | **900** | 0 | **890** |
| 4 | Lender `withdraw(900)` | would reach 0 | ‚ÄĒ | 890 < 900 ‚Üí **REVERT** |

Without step 3, the lender's `withdraw(900)` succeeds (balance 990 ‚Č• 900). The fee claim is what tips the balance below the lender's entitlement.

**Why existing checks do not stop it:**

- `claimContinuousFee` requires `msg.sender == feeClaimer` ‚ÄĒ the `feeClaimer` acts normally, not maliciously.
- The `continuousFeeCredit` underflow guard is correct but insufficient: it only ensures the fee claimer does not over-claim fees; it does not ensure the contract holds enough tokens to cover both the fee and lender withdrawals.
- The `withdrawable` underflow guard is the broken check: it is supposed to confirm token availability but is inflated by the FOT delta.
- The Certora `tokenBalanceCorrect` invariant (`certora/specs/Solvency.spec` line 162‚Äď163) explicitly assumes standard ERC-20 transfers (line 31‚Äď33: *"no fee taking from sender or receiver"*), so the formal proof does not cover this path. [5](#0-4) [6](#0-5) 

### Impact Explanation
Every `repay()` call with a fee-on-transfer loan token silently under-funds the contract relative to `withdrawable`. When `claimContinuousFee` subsequently transfers real tokens to the fee receiver, the contract's balance falls below the sum of outstanding lender credits. Lenders who call `withdraw()` after the fee claim receive a revert, permanently destroying their credit redemption rights. The invariant `balance ‚Č• collateralSum + withdrawableSum + claimableSettlementFee` is violated, constituting protocol insolvency for the affected market.

### Likelihood Explanation
**Preconditions:** (1) the market's loan token charges a transfer fee ‚ÄĒ a property of the token, not a user action; (2) at least one borrower repays; (3) the `feeClaimer` claims accrued fees (routine operation). All three are expected in normal protocol operation once a FOT token is used. The attack is deterministic, repeatable with each repay, and requires no special timing or oracle manipulation.

### Recommendation
In `repay()`, measure the actual received amount by snapshotting the contract's loan-token balance before and after `safeTransferFrom`, and increment `withdrawable` only by the observed delta:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units);
uint256 received = IERC20(market.loanToken).balanceOf(address(this)) - balanceBefore;
marketState[id].withdrawable += UtilsLib.toUint128(received);
```

Apply the same pattern to `liquidate()` and `flashLoan()` repayment legs. Alternatively, explicitly document and enforce that only standard (non-FOT, non-rebasing) ERC-20 tokens are permitted as loan tokens, and add a deployment-time or market-creation check.
