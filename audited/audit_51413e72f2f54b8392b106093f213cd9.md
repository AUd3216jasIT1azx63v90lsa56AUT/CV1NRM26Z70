Audit Report

## Title
Fee-on-transfer loanToken causes `repayAndWithdrawCollateral` to always revert - (File: src/periphery/MidnightBundles.sol)

## Summary
`MidnightBundles.repayAndWithdrawCollateral` computes `units = assets - referralFeeAssets` based on the nominal `assets` parameter, then calls `pullToken` which delivers only `assets*(1-fee_rate)` to MidnightBundles when `loanToken` is fee-on-transfer. The subsequent `Midnight.repay` call attempts to pull the full `units` from MidnightBundles, which holds fewer tokens than required, causing an unconditional revert on every invocation for such markets.

## Finding Description
**Verified code path:**

1. `repayAndWithdrawCollateral` (`src/periphery/MidnightBundles.sol`, lines 329–334):
   - Line 329: `referralFeeAssets = assets.mulDivDown(referralFeePct, WAD)`
   - Line 330: `units = assets - referralFeeAssets`
   - Line 331: `pullToken(loanToken, msg.sender, assets, loanTokenPermit)` — with a fee-on-transfer token, MidnightBundles receives only `assets*(1-fee_rate)`, not `assets`.
   - Line 334: `IMidnight(MIDNIGHT).repay(market, units, onBehalf, address(0), "")` — passes `callback = address(0)`.

2. `pullToken` fallback (`src/periphery/MidnightBundles.sol`, line 396): `SafeTransferLib.safeTransferFrom(token, from, address(this), amount)` — succeeds but delivers a deflated balance.

3. `Midnight.repay` payer resolution (`src/Midnight.sol`, line 511): `address payer = callback != address(0) ? callback : msg.sender` — since `callback == address(0)`, `payer = MidnightBundles`.

4. `Midnight.repay` transfer (`src/Midnight.sol`, line 520): `SafeTransferLib.safeTransferFrom(market.loanToken, payer, address(this), units)` — attempts to pull `units` from MidnightBundles. When `referralFeePct == 0`, `units == assets` but MidnightBundles holds only `assets*(1-fee_rate) < assets`. Reverts unconditionally.

**Root cause:** No balance-delta accounting after `pullToken`. The function assumes `pullToken(assets)` delivers exactly `assets` to MidnightBundles, which is false for fee-on-transfer tokens.

**Why existing checks fail:** The only guards are `require(onBehalf == msg.sender || isAuthorized(...))` and `require(referralFeePct < WAD)` (lines 325–326). Neither validates the actual received balance against `units`. The `forceApproveMax` on line 332 grants unlimited allowance to Midnight, but the transfer still fails because MidnightBundles lacks sufficient balance.

**Token safety note:** `MidnightBundles.sol` line 23 states `/// @dev Inherits the token safety requirements of Midnight (see Midnight.sol)`, but no fee-on-transfer exclusion exists anywhere in `Midnight.sol`, `SECURITY.md`, or `RESEARCHER.md`.

## Impact Explanation
`repayAndWithdrawCollateral` is permanently and deterministically broken for any market whose `loanToken` is fee-on-transfer. Every call reverts at the `safeTransferFrom` inside `Midnight.repay`. Borrowers cannot use the bundler to atomically repay debt and withdraw collateral in such markets. No funds are permanently lost since direct `Midnight.repay` remains available, but the bundler's atomic repay-and-withdraw path is fully DoS-ed for this token class. This constitutes service unavailability for a deployed, in-scope contract function, matching the impact class in `RESEARCHER.md` ("Service unavailability or severe degradation under realistic attacker input").

## Likelihood Explanation
Market creation in Midnight is permissionless and supports arbitrary loan tokens (confirmed by `live_context.json`). Any user can create a market with a fee-on-transfer loanToken (e.g., USDT with fee enabled, STA, PAXG, or any custom ERC20 with transfer fees). The bug is deterministic — it triggers on every call, not probabilistically. No special permissions are required beyond having debt in such a market and having approved MidnightBundles. Repeatability is 100%.

## Recommendation
After `pullToken`, measure the actual received balance using a before/after balance check and use that as the basis for `units`:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
pullToken(loanToken, msg.sender, assets, loanTokenPermit);
uint256 received = IERC20(loanToken).balanceOf(address(this)) - balanceBefore;
uint256 referralFeeAssets = received.mulDivDown(referralFeePct, WAD);
uint256 units = received - referralFeeAssets;
```

This ensures `units` never exceeds the actual balance held by MidnightBundles, preventing the revert in `Midnight.repay`.

## Proof of Concept
1. Deploy a mock ERC20 with a 1% transfer fee.
2. Create a Midnight market with this token as `loanToken`.
3. Borrower supplies collateral, takes a sell offer to acquire debt.
4. Borrower approves MidnightBundles for `assets` of the fee token.
5. Borrower calls `repayAndWithdrawCollateral` with `referralFeePct = 0`.
6. `pullToken` transfers `assets` from borrower; MidnightBundles receives `assets * 0.99`.
7. `Midnight.repay` attempts `safeTransferFrom(loanToken, MidnightBundles, Midnight, assets)`.
8. MidnightBundles balance is `assets * 0.99 < assets` → revert.
9. Confirm: direct `Midnight.repay(market, units, ...)` from borrower succeeds, proving the core protocol is unaffected but the bundler path is broken.
