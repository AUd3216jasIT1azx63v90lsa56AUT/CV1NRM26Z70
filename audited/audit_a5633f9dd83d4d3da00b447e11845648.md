### Title
Continuous fee accrual without withdrawable backing makes `continuousFeeCredit` unclaimable - (File: src/Midnight.sol)

### Summary
`_updatePosition` crystallises accrued fees into `continuousFeeCredit` but never increments `withdrawable`. Because `claimContinuousFee` decrements all three fields (`continuousFeeCredit`, `totalUnits`, `withdrawable`) by the same `amount`, any claim where `amount > withdrawable` reverts on the Solidity 0.8 underflow guard. Whenever outstanding debt exists and fees have accrued without a matching repayment, the full `continuousFeeCredit` balance is unclaimable.

### Finding Description

**Invariant from Certora spec:**
`totalUnits == sumDebt + withdrawable` [1](#0-0) 

This means `withdrawable = totalUnits − sumDebt`. While debt is outstanding, `withdrawable < totalUnits`.

**Fee accrual path — `_updatePosition`:** [2](#0-1) 

Line 846 increments `continuousFeeCredit` by `accruedFee`. Neither `withdrawable` nor `totalUnits` is touched. So after accrual:

```
continuousFeeCredit += accruedFee
withdrawable        unchanged
```

**Claim path — `claimContinuousFee`:** [3](#0-2) 

Lines 318–320 perform three unchecked subtractions:
```solidity
_marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
_marketState.totalUnits          -= UtilsLib.toUint128(amount);
_marketState.withdrawable        -= UtilsLib.toUint128(amount);   // ← reverts if amount > withdrawable
```

There is no explicit `require(amount <= withdrawable)`. The Solidity 0.8 underflow on the third line is the only guard.

**Exploit flow:**

1. Lender supplies 1 000 units of credit; borrower takes on 1 000 units of debt.
   State: `totalUnits = 1000`, `sumDebt = 1000`, `withdrawable = 0`.
2. Time passes; anyone calls `updatePosition(market, lender)`.
   State: `continuousFeeCredit = F > 0`, `withdrawable = 0` (unchanged).
3. feeClaimer calls `claimContinuousFee(market, F, receiver)`.
   - `continuousFeeCredit -= F` → succeeds.
   - `totalUnits -= F` → succeeds.
   - `withdrawable -= F` → **underflow revert** (0 − F).

The Certora liveness rule for `claimContinuousFee` explicitly encodes this constraint: [4](#0-3) 

`amount <= withdrawableBefore` is a hard precondition for success, confirming the revert path is real.

The existing test `testClaimContinuousFee` works around this by explicitly repaying the full debt before claiming: [5](#0-4) 

The comment `// Repay so withdrawable covers the claim` confirms the protocol team is aware that without repayment the claim fails.

### Impact Explanation

The feeClaimer cannot extract any portion of `continuousFeeCredit` that exceeds `withdrawable`. Fees crystallised by `updatePosition` while debt remains outstanding are locked until borrowers repay. If borrowers default (bad debt), `continuousFeeCredit` is scaled down by the loss factor during liquidation, but `withdrawable` is only increased by the actually repaid units — so a portion of accrued fees may be permanently unclaimable. This directly violates the invariant that all accrued continuous fees are claimable.

### Likelihood Explanation

The condition `continuousFeeCredit > withdrawable` arises in every market where:
- at least one lender position has been updated (fees accrued), and
- the full outstanding debt has not yet been repaid.

This is the normal operating state of any active lending market. No attacker action is required; the condition is reached by ordinary protocol use. It is repeatable and persistent for the lifetime of any loan.

### Recommendation

`claimContinuousFee` should not require `withdrawable` to back the claim. The continuous fee is already accounted for as a reduction in lender credit; the protocol holds the corresponding tokens implicitly as part of the outstanding loan principal. Two options:

1. **Remove the `withdrawable` decrement from `claimContinuousFee`** and instead track a separate `claimableContinuousFee` counter that is incremented by `_updatePosition` alongside `continuousFeeCredit`, backed by a matching token balance invariant.
2. **Decouple `continuousFeeCredit` from `withdrawable`** entirely: when fees accrue, also increment a dedicated `continuousFeeWithdrawable` field (separate from the lender-facing `withdrawable`), and decrement only that field in `claimContinuousFee`.

### Proof of Concept

```solidity
function testContinuousFeeUnclaimableWithoutRepay(
    uint256 credit, uint256 feeRate, uint256 ttm, uint256 elapsed
) public {
    credit   = bound(credit,   1, MAX_CREDIT);
    feeRate  = bound(feeRate,  1, MAX_CONTINUOUS_FEE);
    ttm      = bound(ttm,      2, 360 days);
    elapsed  = bound(elapsed,  1, ttm - 1);

    // 1. Lender supplies credit; borrower takes on matching debt.
    setupLender(credit, feeRate, ttm);

    // 2. Accrue fees without any repayment.
    vm.warp(vm.getBlockTimestamp() + elapsed);
    midnight.updatePosition(market, lender);

    uint256 feeAmount = midnight.continuousFeeCredit(id);
    vm.assume(feeAmount > 0);

    // 3. Assert precondition: withdrawable is 0 (no repayments).
    assertEq(midnight.withdrawable(id), 0, "withdrawable must be 0");

    // 4. feeClaimer attempts to claim the full accrued fee — must revert.
    vm.prank(feeClaimer);
    vm.expectRevert();                          // underflow on withdrawable
    midnight.claimContinuousFee(market, feeAmount, feeClaimer);

    // 5. Even claiming 1 unit reverts.
    vm.prank(feeClaimer);
    vm.expectRevert();
    midnight.claimContinuousFee(market, 1, feeClaimer);
}
```

Expected assertions:
- Both `claimContinuousFee` calls revert.
- `continuousFeeCredit(id) == feeAmount` (unchanged — fees are stuck).
- `withdrawable(id) == 0` (unchanged).

    function claimContinuousFee(Market memory market, uint256 amount, address receiver) external {
        bytes32 id = toId(market);
        MarketState storage _marketState = marketState[id];
        require(msg.sender == feeClaimer, OnlyFeeClaimer());
        require(_marketState.tickSpacing > 0, MarketNotCreated());

        _marketState.continuousFeeCredit -= UtilsLib.toUint128(amount);
        _marketState.totalUnits -= UtilsLib.toUint128(amount);
        _marketState.withdrawable -= UtilsLib.toUint128(amount);

        emit EventsLib.ClaimContinuousFee(msg.sender, id, amount, receiver);

        SafeTransferLib.safeTransfer(market.loanToken, receiver, amount);
    }

    function _updatePosition(Market memory market, bytes32 id, address user)
        internal
        returns (uint128, uint128, uint128)
    {
        Position storage _position = position[id][user];
        (uint128 newCredit, uint128 newPendingFee, uint128 accruedFee) = updatePositionView(market, id, user);

        uint128 creditDecrease = _position.credit - newCredit;
        uint128 pendingFeeDecrease = _position.pendingFee - newPendingFee;

        _position.credit = newCredit;
        _position.lastLossFactor = marketState[id].lossFactor;
        _position.pendingFee = newPendingFee;
        _position.lastAccrual = uint128(block.timestamp);
        marketState[id].continuousFeeCredit += UtilsLib.toUint128(accruedFee);

        emit EventsLib.UpdatePosition(id, user, creditDecrease, pendingFeeDecrease, accruedFee);

        return (newCredit, newPendingFee, accruedFee);
    }
