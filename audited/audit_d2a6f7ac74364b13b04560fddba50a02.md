### Title
Fee-on-Transfer Collateral Inflates Position Accounting, Enabling Phantom Collateral Borrowing and Bad Debt - (File: src/Midnight.sol)

### Summary
`supplyCollateral` credits `position[id][onBehalf].collateral[collateralIndex]` with the full `assets` parameter before calling `SafeTransferLib.safeTransferFrom`, but `safeTransferFrom` only verifies the call succeeded and returned `true` — it does not measure the actual tokens received. For a fee-on-transfer collateral token, the protocol records `assets` units of collateral while only receiving `assets × (1 − fee_rate)` tokens, creating phantom collateral. `isHealthy` then computes `maxDebt` from the inflated accounting value, allowing the borrower to borrow against collateral that does not exist in the contract, producing irrecoverable bad debt.

### Finding Description

**Exact code path:**

`supplyCollateral` in `src/Midnight.sol`: [1](#0-0) 

Line 533 writes `oldCollateral + assets` to storage unconditionally: [2](#0-1) 

Line 545 then calls `safeTransferFrom`, which only checks call success and boolean return — it never compares `balanceOf(address(this))` before and after: [3](#0-2) [4](#0-3) 

**Root cause:** No balance-before/after check. The accounting delta is `assets`; the actual received delta is `assets × (1 − fee_rate)`. The gap is `assets × fee_rate` of phantom collateral.

**Market creation — no token type restriction:**

`touchMarket` validates LLTV tiers, `maxLif`, and collateral sort order, but imposes no constraint on the ERC20 implementation of the collateral token: [5](#0-4) 

Any address (unprivileged market creator) can deploy a market with a fee-on-transfer token as collateral. Alternatively, a well-known fee-on-transfer token (e.g., USDT with fee enabled, or a rebasing token) could be used in a market created by anyone.

**Health check uses inflated accounting:**

`isHealthy` reads `_position.collateral[i]` directly — the storage value, not the actual token balance: [6](#0-5) 

`maxDebt` is therefore computed on phantom collateral, allowing the borrower to pass the health check in `take` at line 476: [7](#0-6) 

**Exploit flow:**

1. A market exists (or is created by any caller) with a fee-on-transfer collateral token (fee rate `f`).
2. Borrower calls `supplyCollateral(market, 0, 1000e18, borrower)`.
   - Storage: `position[id][borrower].collateral[0] = 1000e18`
   - Actual tokens received: `1000e18 × (1 − f)`
3. Borrower places a sell offer and calls `take(...)` to borrow up to `1000e18 × price × lltv / WAD` units.
   - `isHealthy` passes because it uses the inflated `1000e18` accounting value.
4. The position is undercollateralized by `1000e18 × f × price × lltv / WAD` units from the moment of supply.
5. When the position is liquidated, `safeTransfer(collateralToken, receiver, seizedAssets)` also incurs the fee, so the liquidator receives less than `seizedAssets`, reducing or eliminating the liquidation incentive and leaving residual bad debt socialized to lenders.

**Why existing checks fail:**

- Authorization check (line 527) only prevents unauthorized `onBehalf` — irrelevant here.
- `SafeTransferLib.safeTransferFrom` checks `success` and `returndata` boolean, not received amount.
- `touchMarket` has no token allowlist or fee-on-transfer guard.
- Certora `BalanceEffects.spec` rule `supplyCollateralEffects` asserts `collateral == collateralBefore + assets` as a tautology of the code, not a balance invariant — it does not model fee-on-transfer tokens. [8](#0-7) 

### Impact Explanation
The borrower holds a position where `position.collateral[i] > IERC20(collateralToken).balanceOf(address(midnight))` attributable to that borrower. The excess collateral value (`assets × fee_rate × price × lltv`) is phantom borrowing capacity. Debt taken against it cannot be fully covered by seizing collateral, producing bad debt that is socialized to all lenders in the market via the loss-factor mechanism.

### Likelihood Explanation
**Preconditions:** A market must use a fee-on-transfer ERC20 as a collateral token. This is permissionless — any caller can create such a market via `touchMarket`. Fee-on-transfer tokens exist in production (e.g., STA, PAXG with fee enabled, or any custom ERC20). The attack is repeatable: every `supplyCollateral` call with a fee-on-transfer token inflates accounting by `assets × fee_rate`. No privileged action is required after market creation.

### Recommendation
Add a balance-before/after check in `supplyCollateral` and credit only the actual received amount:

```solidity
function supplyCollateral(...) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
    bytes32 id = touchMarket(market);
    address collateralToken = market.collateralParams[collateralIndex].token;

    uint256 balanceBefore = IERC20(collateralToken).balanceOf(address(this));
    SafeTransferLib.safeTransferFrom(collateralToken, msg.sender, address(this), assets);
    uint256 received = IERC20(collateralToken).balanceOf(address(this)) - balanceBefore;

    Position storage _position = position[id][onBehalf];
    uint256 oldCollateral = _position.collateral[collateralIndex];
    _position.collateral[collateralIndex] = UtilsLib.toUint128(oldCollateral + received);
    // ... bitmap logic using received instead of assets
}
```

Alternatively, document that fee-on-transfer tokens are explicitly unsupported as collateral and add a revert guard in `touchMarket` (though this is not enforceable on-chain without a token registry).

    function isHealthy(Market memory market, bytes32 id, address borrower) public view returns (bool) {
        Position storage _position = position[id][borrower];
        uint256 debt = _position.debt;
        uint256 maxDebt;
        if (debt > 0) {
            uint128 _collateralBitmap = _position.collateralBitmap;
            while (_collateralBitmap != 0) {
                uint256 i = UtilsLib.msb(_collateralBitmap);
                CollateralParams memory collateralParam = market.collateralParams[i];
                uint256 price = IOracle(collateralParam.oracle).price();
                maxDebt += _position.collateral[i].mulDivDown(price, ORACLE_PRICE_SCALE)
                    .mulDivDown(collateralParam.lltv, WAD);
                _collateralBitmap = _collateralBitmap.clearBit(i);
            }
        }
        return maxDebt >= debt;
    }

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());

/// supplyCollateral increases onBehalf's collateral by exactly assets,
/// and only changes position[id][onBehalf].collateral[collateralIndex].
rule supplyCollateralEffects(env e, Midnight.Market market, uint256 collateralIndex, uint256 assets, address onBehalf, bytes32 anyId, address anyUser, uint256 anyIndex) {
    bytes32 id = toId(e, market);

    uint256 collateralBefore = collateral(id, onBehalf, collateralIndex);
    uint256 otherCollateralBefore = collateral(anyId, anyUser, anyIndex);

    supplyCollateral(e, market, collateralIndex, assets, onBehalf);

    assert collateral(id, onBehalf, collateralIndex) == collateralBefore + assets;
    assert anyUser != onBehalf || anyId != id || anyIndex != collateralIndex => collateral(anyId, anyUser, anyIndex) == otherCollateralBefore;
