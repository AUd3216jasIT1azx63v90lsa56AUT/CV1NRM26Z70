Audit Report

## Title
Borrower Self-Activates Reverting-Oracle Collateral to Permanently Block Own Liquidation - (File: src/Midnight.sol)

## Summary
`supplyCollateral` sets the collateral bitmap bit without ever querying the oracle, allowing a borrower to activate a collateral slot whose oracle is already reverting. `liquidate` iterates every activated collateral and calls `IOracle(...).price()` with no try/catch; a single reverting oracle causes the entire liquidation transaction to revert. Because `withdrawCollateral` also calls `isHealthy` — which iterates all oracles — the borrower with outstanding debt cannot remove the poisoned slot, making the position permanently unliquidatable and leaving lenders exposed to unrecoverable bad debt.

## Finding Description

**Root cause — `supplyCollateral` (lines 524–546):**
The function gates on `onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender]` and sets the bitmap bit when `oldCollateral == 0 && assets > 0`, but never calls the oracle: [1](#0-0) 

The authorization guard prevents a *third party* from poisoning a victim's bitmap, but it does not prevent the borrower from poisoning their *own* bitmap.

**Root cause — `liquidate` (lines 607–618):**
Every set bit is visited and `IOracle(_collateralParam.oracle).price()` is called unconditionally with no error handling: [2](#0-1) 

A single reverting oracle unwinds the entire call.

**Root cause — `withdrawCollateral` / `isHealthy` (lines 568, 944–960):**
`withdrawCollateral` calls `isHealthy`, which also iterates all activated oracles when `debt > 0`: [3](#0-2) [4](#0-3) 

This closes the escape hatch: the borrower cannot self-remove the reverting slot while carrying debt.

**Protocol's own documentation confirms both halves:** [5](#0-4) [6](#0-5) 

**Exploit flow:**
1. Borrower identifies (or deploys) a market containing at least one collateral whose oracle is reverting or will revert.
2. Borrower calls `supplyCollateral` for up to 15 normal collateral indices and 1 reverting-oracle index (≥ 1 wei each). The bitmap reaches `MAX_COLLATERALS_PER_BORROWER` (16 bits set); the `TooManyActivatedCollaterals` guard is satisfied exactly.
3. Borrower borrows against the healthy collaterals via `take`.
4. Position becomes unhealthy (price drop or borrow-to-limit).
5. Any liquidator calls `liquidate(...)`. The while loop reaches the reverting-oracle index; `IOracle(...).price()` reverts; the entire call reverts.
6. `withdrawCollateral` also reverts (via `isHealthy`) as long as debt > 0, so the borrower cannot self-clear the slot.
7. The position is permanently unliquidatable.

**Why `MAX_COLLATERALS_PER_BORROWER` does not help:**
The cap is reached exactly at 16 — it does not block the reverting slot, it merely prevents adding a 17th. [7](#0-6) 

## Impact Explanation
The unhealthy borrower's position becomes permanently unliquidatable. Lenders in the market cannot recover their principal; bad debt accumulates and is socialized across all lenders via the loss-factor mechanism. The borrower retains all non-reverting collateral and is never forced to repay. This directly violates the core protocol invariant that unhealthy positions must remain liquidatable, and constitutes a permanent freeze of lender funds — a critical in-scope impact.

## Likelihood Explanation
Preconditions are fully attacker-controlled and negligible in cost:
- Supplying 1 wei of the reverting-oracle collateral token is sufficient to set the bitmap bit.
- The oracle need not be malicious: any deprecated, paused, or temporarily broken oracle qualifies; the borrower activates it while it is reverting, which `supplyCollateral` permits.
- The attack is one-time and permanent: once the bitmap is full with the reverting slot, no external party can clear it.
- It is repeatable across every market that contains at least one such collateral parameter.
- No privileged access is required; any borrower in any eligible market can execute this.

## Recommendation
1. **Add try/catch in the liquidation oracle loop:** Wrap `IOracle(_collateralParam.oracle).price()` in a try/catch inside the `liquidate` while-loop. On revert, treat the price as 0 (worst-case for the borrower) or skip the collateral's contribution to `maxDebt`, ensuring liquidation can always proceed.
2. **Alternatively, validate the oracle at activation time:** In `supplyCollateral`, call the oracle and require it to return a non-reverting, non-zero price before setting the bitmap bit. This prevents activation of broken oracles entirely.
3. **Allow forced deactivation of a reverting-oracle slot:** Provide a permissionless function (or extend `liquidate`) that can clear a bitmap bit for a collateral whose oracle is reverting, without requiring a health check against that oracle.

Option 1 is the minimal fix and preserves the most flexibility; option 2 is the most conservative but may block legitimate use cases where oracles are temporarily paused.

## Proof of Concept
**Minimal Foundry test outline:**
1. Deploy a mock oracle `RevertingOracle` whose `price()` always reverts.
2. Create a market with 16 collateral params: indices 0–14 use a normal oracle, index 15 uses `RevertingOracle`.
3. As `borrower`, call `supplyCollateral` for indices 0–15 (1 wei each). Assert bitmap has 16 bits set.
4. As `borrower`, call `take` to borrow against indices 0–14 until the position is at the health boundary.
5. Manipulate the normal oracle prices downward so `originalDebt > maxDebt`.
6. As `liquidator`, call `liquidate(market, 0, 0, repaidUnits, borrower, false, ...)`. Assert the call reverts (due to `RevertingOracle.price()` at index 15).
7. Assert `borrower.debt > 0` and no liquidation has succeeded — demonstrating permanent unliquidatability.
