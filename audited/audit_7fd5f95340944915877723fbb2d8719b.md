### Title
Zero-value `safeTransferFrom` on rounded settlement fee blocks valid takes for tokens reverting on zero transfers - (File: src/Midnight.sol)

### Summary
In `take()`, the settlement fee spread (`buyerAssets - sellerAssets`) is transferred unconditionally to `address(this)` via `SafeTransferLib.safeTransferFrom`. When `units * _settlementFee < WAD`, integer rounding collapses `buyerAssets == sellerAssets`, producing a zero-value transfer. Loan tokens that revert on zero-value `transferFrom` calls cause the entire `take` to revert, permanently blocking a taker from a valid near-par take.

### Finding Description
**Exact code path:**

`take()` computes assets at lines 363–364: [1](#0-0) 

For a sell offer (`offer.buy = false`):
- `sellerPrice = offerPrice`
- `buyerPrice = offerPrice + _settlementFee`
- `buyerAssets = mulDivUp(units, buyerPrice, WAD)`
- `sellerAssets = mulDivUp(units, sellerPrice, WAD)`

When `units * _settlementFee < WAD`, both `mulDivUp` calls produce the same ceiling value, so `buyerAssets == sellerAssets`. The minimum non-zero settlement fee is 1 CBP = `1e12` (one centi-basis-point): [2](#0-1) 

With `units = 1` and any fee ≤ `0.005e18` (the maximum at 360 days), `1 * fee < 1e18 = WAD` always holds, so the rounding always collapses the spread to zero.

The unconditional transfer at line 455 then calls `transferFrom(payer, address(this), 0)`: [3](#0-2) 

`SafeTransferLib.safeTransferFrom` has no zero-value guard: [4](#0-3) 

If the token reverts on `transferFrom(..., 0)`, the revert propagates and the entire `take` fails.

**Concrete numeric example (sell offer):**
- `units = 1`, `offerPrice = 0.5e18`, `_settlementFee = 1e12`
- `buyerAssets = mulDivUp(1, 500001000000000000, 1e18) = 1`
- `sellerAssets = mulDivUp(1, 500000000000000000, 1e18) = 1`
- `buyerAssets - sellerAssets = 0` → `safeTransferFrom(token, payer, address(this), 0)` → token reverts

**Why existing checks fail:** No minimum `units` check, no `require(buyerAssets > sellerAssets)`, and no zero-value skip in `SafeTransferLib`. The `claimableSettlementFee` increment at line 418 adds 0, which is harmless but confirms the protocol proceeds to the transfer unconditionally: [5](#0-4) 

### Impact Explanation
A taker is permanently blocked from executing a valid take at small `units` on any market whose loan token reverts on zero-value `transferFrom`. All state mutations before line 455 (position updates, `claimableSettlementFee`) are reverted, so no state corruption occurs, but the take cannot complete. The taker cannot work around this without increasing `units` to at least `WAD / _settlementFee` (e.g., `1e6` units for a 1e12 fee), which changes the economics of the intended trade.

### Likelihood Explanation
Preconditions:
1. Loan token has `require(amount > 0)` in `transferFrom` — a non-standard but real pattern used by some tokens for gas efficiency or dust prevention.
2. `units * _settlementFee < WAD` — trivially satisfied with `units = 1` and any fee below `1e18`; the maximum fee is `0.005e18`, so this holds for all `units < 200`.
3. Taker calls `take` with small `units`.

The condition is repeatable and deterministic for any affected market. The taker has no on-chain recourse other than using a larger `units` value.

### Recommendation
Add a zero-value guard before the first `safeTransferFrom` call in `take()`:

```solidity
if (buyerAssets > sellerAssets) {
    SafeTransferLib.safeTransferFrom(
        offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets
    );
}
```

This is consistent with the protocol's existing pattern of skipping no-op operations and does not affect accounting since `claimableSettlementFee` is already incremented by the correct (possibly zero) spread.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
    }
