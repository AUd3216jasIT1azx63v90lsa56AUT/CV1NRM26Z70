### Title
Fee-on-transfer loan token silently underfunds `claimableSettlementFee` in `take` - (`src/Midnight.sol`)

### Summary
In `take`, `claimableSettlementFee[loanToken]` is incremented by `buyerAssets - sellerAssets` on line 418 before the inbound `safeTransferFrom` on line 455. `SafeTransferLib.safeTransferFrom` only checks the boolean return value of `transferFrom` and performs no balance-before/after accounting. When the loan token is a fee-on-transfer ERC20, only `(buyerAssets - sellerAssets - transferFee)` actually arrives at `address(this)`, while `claimableSettlementFee` is credited the full nominal amount. This breaks the solvency invariant `balance >= collateralSum + withdrawableSum + claimableSettlementFee` and enables the fee claimer to drain tokens belonging to lenders or collateral providers.

### Finding Description
**Exact code path:**

1. `src/Midnight.sol` line 418: `claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;` — state written before any transfer.
2. Line 420: `address buyerCallback = offer.buy ? offer.callback : takerCallback;` — for a sell offer (`offer.buy = false`), `buyerCallback = takerCallback`.
3. Line 422: `address payer = buyerCallback != address(0) ? buyerCallback : (offer.buy ? buyer : msg.sender);` — if `takerCallback != address(0)`, `payer = takerCallback`.
4. Lines 445–452: `IBuyCallback(buyerCallback).onBuy(...)` is called; the callback contract returns `CALLBACK_SUCCESS` without transferring tokens (payment is expected via `safeTransferFrom`).
5. Line 455: `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);` — for a fee-on-transfer token, only `(buyerAssets - sellerAssets - transferFee)` arrives.

**Root cause:** `SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34) checks only the `bool` return of `transferFrom`. No balance delta is measured. There is no validation of the loan token type in `touchMarket` (`src/Midnight.sol` lines 755–791); the protocol explicitly supports "arbitrary loan token" per `live_context.json`.

**Attacker-controlled inputs:**
- `takerCallback`: any contract the taker deploys that returns `CALLBACK_SUCCESS` from `onBuy` and has pre-approved Midnight for the fee-on-transfer loan token.
- `offer.market.loanToken`: a fee-on-transfer ERC20. Since market creation is permissionless and accepts any loan token, the attacker (as market creator) can deploy such a market, or exploit any existing market whose loan token charges a transfer fee.

**Why existing checks fail:** The Certora `pendingFeeReceiptZero` invariant (`certora/specs/Solvency.spec` lines 157–158) and `tokenBalanceCorrect` (`lines 162–177`) are proven under the assumption that `CVL_transferFrom` delivers the full nominal amount. Fee-on-transfer tokens violate this assumption silently — the `safeTransferFrom` call succeeds (returns `true`), so no revert occurs, but the actual balance increase is less than the credited amount.

### Impact Explanation
After each `take` with a fee-on-transfer loan token, `claimableSettlementFee[loanToken]` exceeds the actual tokens held for that purpose. The deficit accumulates across every take. When `claimSettlementFee` (`src/Midnight.sol` lines 305–310) is called, it executes `safeTransfer(token, receiver, amount)` against the overcounted balance. If the cumulative shortfall is large enough, this transfer drains tokens that back lender `withdrawable` credits or borrower collateral, causing direct insolvency: lenders cannot withdraw and/or collateral cannot be returned.

### Likelihood Explanation
**Preconditions:**
- A market exists (or is created by the attacker) with a fee-on-transfer loan token.
- The `takerCallback` contract has approved Midnight and holds enough of the loan token for `safeTransferFrom` to succeed.

Market creation is permissionless with no loan token validation. The attack is repeatable on every `take` call in the affected market. Fee-on-transfer tokens (tax tokens, some rebasing tokens) are a known ERC20 variant. The attacker can be the market creator and taker simultaneously, requiring no cooperation from other users.

### Recommendation
Choose one of:
1. **Measure actual received amount**: Before line 455, record `uint256 balBefore = IERC20(offer.market.loanToken).balanceOf(address(this));`. After the transfer, compute `uint256 actualFee = balanceAfter - balBefore` and use that value to increment `claimableSettlementFee` instead of the nominal `buyerAssets - sellerAssets`.
2. **Explicitly exclude fee-on-transfer tokens**: Add a check in `touchMarket` that performs a self-transfer probe and reverts if the received amount differs from the sent amount, or document and enforce this restriction at the integration layer.

### Proof of Concept
```solidity
// Foundry unit test sketch
contract FeeOnTransferLoanToken is ERC20 {
    // 1% fee burned on every transferFrom
    function transferFrom(address from, address to, uint256 amount) public override returns (bool) {
        uint256 fee = amount / 100;
        _burn(from, fee);
        return super.transferFrom(from, to, amount - fee);
    }
}

function testFeeOnTransferOvercountsSettlementFee() public {
    // 1. Deploy fee-on-transfer token, create market with it
    // 2. Collateralize borrower, set up sell offer
    // 3. Deploy takerCallback that returns CALLBACK_SUCCESS and pre-approves Midnight
    uint256 balBefore = feeToken.balanceOf(address(midnight));
    vm.prank(taker);
    midnight.take(sellOffer, hex"", units, taker, borrower, address(takerCallback), hex"");
    uint256 actualReceived = feeToken.balanceOf(address(midnight)) - balBefore;
    uint256 claimable = midnight.claimableSettlementFee(address(feeToken));
    // Invariant violated: claimable > actual tokens received for fees
    assertGt(claimable, actualReceived);
    // Fee claimer can now drain lender withdrawable tokens
}
```

**Expected assertion:** `claimableSettlementFee > actualReceived` after `take`, with the gap equal to `(buyerAssets - sellerAssets) * feeRate`. Repeated takes widen the gap until `claimSettlementFee` causes an insolvency revert or drains lender/collateral funds.
