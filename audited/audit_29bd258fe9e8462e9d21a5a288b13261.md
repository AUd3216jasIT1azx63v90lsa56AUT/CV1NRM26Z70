### Title
SafeTransferLib `abi.decode(returndata, (bool))` misinterprets 64-byte returndata with zero first word as `false`, enabling permanent fund freeze via malicious loan token - (File: src/libraries/SafeTransferLib.sol)

### Summary
`SafeTransferLib.safeTransfer` and `safeTransferFrom` use `abi.decode(returndata, (bool))` to validate ERC20 return values. Solidity's `abi.decode` reads only the first 32 bytes when decoding a `bool`, so a token returning 64 bytes where the first word is zero (e.g., `abi.encode(uint256(0), uint256(1))`) is decoded as `false`, triggering `TransferReturnedFalse()` even though the transfer succeeded on-chain. Because `touchMarket` imposes no validation on `market.loanToken`, any unprivileged actor can create a market with such a token, lure lenders into supplying credit, and permanently freeze all withdrawals.

### Finding Description

**Root cause — `SafeTransferLib.sol` lines 21 and 33:**

```solidity
// safeTransfer
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());

// safeTransferFrom
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferFromReturnedFalse());
```

`abi.decode(returndata, (bool))` reads the first 32-byte word of `returndata` regardless of total length. When `returndata` is 64 bytes and the first word is `0x00…00`, the decoded value is `false` and the `require` reverts — even though `success == true` and balances were updated.

**No loanToken validation in `touchMarket` (lines 755–791):**

`touchMarket` checks maturity, collateral count, collateral ordering, lltv, and maxLif. It performs **zero checks on `market.loanToken`**. Any address with `code.length > 0` is accepted.

**Exploit flow:**

1. Attacker deploys `MaliciousToken`:
   - `transferFrom(from, to, amount)` → returns `abi.encode(true)` (32 bytes, first word non-zero) → `safeTransferFrom` passes.
   - `transfer(to, amount)` → returns `abi.encode(uint256(0), uint256(1))` (64 bytes, first word = 0) → `safeTransfer` reverts.
   - Both functions actually move balances (the transfer is real).

2. Attacker calls `touchMarket` with `market.loanToken = address(MaliciousToken)`. Market is created without error.

3. Attacker (or any lender) calls `take` → internally calls `SafeTransferLib.safeTransferFrom(loanToken, …)` → `transferFrom` returns 32-byte `true` → succeeds. Lender credit is recorded in `position[id][lender].credit`.

4. Lender calls `withdraw(market, units, lender, receiver)`:
   - Lines 493–495: state is mutated (`credit` decremented, `withdrawable` decremented, `totalUnits` decremented).
   - Line 499: `SafeTransferLib.safeTransfer(market.loanToken, receiver, units)` is called.
   - `transfer` returns 64 bytes with first word = 0 → `abi.decode(returndata, (bool))` = `false` → `require` reverts with `TransferReturnedFalse()`.
   - Entire transaction reverts; state is rolled back. Lender credit remains locked.

5. Every subsequent `withdraw` call follows the same path and reverts identically. Funds are permanently frozen.

**Why existing checks are insufficient:**
- The `returndata.length == 0` short-circuit only bypasses the decode for zero-length returns; it does not protect against oversized returns.
- There is no allowlist or interface check on `loanToken`.
- The Certora `Reverts.spec` models `safeTransfer` as a CVL stub that either reverts or succeeds cleanly; it does not model the 64-byte false-return edge case.

### Impact Explanation

All lender funds supplied to any market whose `loanToken` exhibits this return pattern are permanently unwithdrawable. `withdraw` always reverts at line 499. If `transfer` is the only affected function (while `transferFrom` works), repayments succeed but lenders still cannot redeem credit. If the attacker also makes `transferFrom` return the same 64-byte pattern after an initial period, repayments and liquidations also freeze, trapping borrower collateral as well.

### Likelihood Explanation

Preconditions are fully attacker-controlled: deploying a contract is permissionless, and `touchMarket` accepts any `loanToken`. The attack requires social engineering to attract lenders, but in a permissionless lending protocol with aggregators or automated vaults, a convincingly named token with normal initial behavior (returning `true` from `transferFrom`) can attract real deposits before the `transfer` path is exercised. The condition is deterministic and repeatable: every `withdraw` call reverts with no recovery path.

### Recommendation

Tighten the return-data check in both `safeTransfer` and `safeTransferFrom` to reject any return data that is neither empty nor exactly 32 bytes encoding `true`:

```solidity
// Accept: no returndata (non-standard tokens like USDT on mainnet)
// Accept: exactly 32 bytes decoding to true
// Reject: anything else, including 64-byte returns whose first word is false
require(
    returndata.length == 0 || (returndata.length == 32 && abi.decode(returndata, (bool))),
    TransferReturnedFalse()
);
```

This matches the ERC-20 spec (single `bool` return) while preserving compatibility with the common no-return-data pattern. Tokens returning non-standard lengths are rejected at market creation time rather than silently accepted and later causing irreversible freezes.

contract MaliciousLoanToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;

    function mint(address to, uint256 amount) external { balanceOf[to] += amount; }
    function approve(address spender, uint256 amount) external returns (bool) {
        allowance[msg.sender][spender] = amount; return true;

    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        balanceOf[from] -= amount; balanceOf[to] += amount; return true; // standard 32-byte true

    function transfer(address to, uint256 amount) external {
        balanceOf[msg.sender] -= amount; balanceOf[to] += amount;
        // Return 64 bytes: first word = 0 (false), second word = 1
        assembly { mstore(0x00, 0) mstore(0x20, 1) return(0x00, 0x40) }
