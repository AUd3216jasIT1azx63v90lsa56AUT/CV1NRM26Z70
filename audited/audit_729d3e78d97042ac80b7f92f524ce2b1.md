### Title
Borrower Self-Immunizes Against Liquidation by Activating a Reverting-Oracle Collateral - ([File: src/interfaces/IOracle.sol])

### Summary
`supplyCollateral` never calls the oracle, so a borrower can activate any collateral slot—including one whose oracle always reverts—at any time, even after becoming unhealthy. `liquidate` unconditionally iterates every bit in `collateralBitmap` and calls `IOracle.price()` on each; if any single call reverts, the entire liquidation reverts. A borrower who supplies 1 wei of a collateral backed by a reverting oracle therefore becomes permanently unliquidatable, violating the invariant that unhealthy positions must remain liquidatable.

### Finding Description

**Root cause — asymmetric oracle access between `supplyCollateral` and `liquidate`.**

`supplyCollateral` (lines 524–546) performs no oracle call. Its only guard when activating a new slot is the 16-collateral cap:

```solidity
if (oldCollateral == 0 && assets > 0) {
    uint128 newCollateralBitmap = _position.collateralBitmap.setBit(collateralIndex);
    _position.collateralBitmap = newCollateralBitmap;
    require(
        UtilsLib.countBits(newCollateralBitmap) <= MAX_COLLATERALS_PER_BORROWER,
        TooManyActivatedCollaterals()
    );
}
``` [1](#0-0) 

`liquidate` (lines 607–618) iterates every set bit in `collateralBitmap` and calls `price()` on each oracle with no error handling:

```solidity
while (_collateralBitmap != 0) {
    uint256 i = UtilsLib.msb(_collateralBitmap);
    CollateralParams memory _collateralParam = market.collateralParams[i];
    uint256 price = IOracle(_collateralParam.oracle).price();   // ← reverts if oracle reverts
    ...
    _collateralBitmap = _collateralBitmap.clearBit(i);
}
``` [2](#0-1) 

`isHealthy` has the identical unchecked loop: [3](#0-2) 

The protocol's own NatSpec explicitly acknowledges both halves of the problem:

> "Liquidation reverts if any of the activated collaterals' oracle reverts (see LIVENESS)."
> "Note that a borrower can activate a collateral once its oracle is reverting because the oracle is not called in supplyCollateral." [4](#0-3) 

**Exploit flow:**

1. A multi-collateral market exists with `collateralParams[0]` (normal oracle, e.g. `oracle1`) and `collateralParams[1]` (oracle that reverts, e.g. a Chainlink feed with a triggered circuit-breaker, or a purpose-deployed `AlwaysRevertOracle`).
2. Borrower calls `supplyCollateral(market, 0, largeAmount, borrower)` and then `take(sellOffer, ...)` to borrow against idx0.
3. Price of idx0 drops; borrower is now unhealthy (`debt > maxDebt`).
4. Borrower calls `supplyCollateral(market, 1, 1, borrower)`. This succeeds: no oracle is called, 1 wei of the idx1 token is transferred in, and bit 1 is set in `collateralBitmap`.
5. Any subsequent call to `liquidate(market, 0, ...)` enters the bitmap loop, hits `IOracle(collateralParams[1].oracle).price()`, which reverts, and the entire transaction reverts.
6. `isHealthy` also reverts for the same reason, so `take` (when the borrower is the seller) and `withdrawCollateral` also revert—but those are secondary; the critical failure is liquidation.

**Why existing checks do not stop it:**

- The `Unauthorized` check in `supplyCollateral` is satisfied because the borrower calls on their own behalf.
- `TooManyActivatedCollaterals` only fires at 17 active slots; a borrower with fewer than 16 active collaterals can always add one more.
- There is no health check in `supplyCollateral`; the function is explicitly designed to never call an oracle.
- There is no try/catch or skip-on-revert logic anywhere in the `liquidate` bitmap loop.

### Impact Explanation
An unhealthy borrower with outstanding debt becomes permanently unliquidatable. Their debt remains on the books, `marketState.totalUnits` is never reduced via bad-debt socialization, and lenders cannot recover their funds. Every lender in the market suffers a loss proportional to the bad debt that cannot be realized. The `withdrawable` pool is never replenished, so lenders who try to withdraw after maturity also find insufficient assets.

### Likelihood Explanation
**Preconditions:**
- A multi-collateral market where at least one collateral's oracle can revert. This is realistic: Chainlink aggregators revert when the sequencer is down (L2s), when the answer is stale beyond a configured threshold, or when a circuit-breaker fires. Custom oracles may revert on price deviation. The market need not be created by the attacker.
- The borrower must hold (or be able to acquire) at least 1 wei of the idx1 collateral token. For any ERC-20 with non-zero supply this is trivially achievable.
- The borrower must act before being liquidated, i.e., between becoming unhealthy and the first successful liquidation call. Given that oracle outages are often sustained (minutes to hours), this window is realistic.

**Repeatability:** The attack is permanent once executed. The borrower cannot be forced to withdraw the 1-wei collateral (only they or an authorized address can call `withdrawCollateral`, and that function also reverts when the borrower has debt and any oracle reverts). The state is irreversible without a protocol upgrade.

### Recommendation
Wrap each oracle call in the `liquidate` (and `isHealthy`) bitmap loop with a low-level try/catch. If an oracle reverts, treat the collateral's contribution as zero (conservative: makes the position appear maximally unhealthy, never blocking a valid liquidation):

```solidity
uint256 price;
try IOracle(_collateralParam.oracle).price() returns (uint256 p) {
    price = p;
} catch {
    price = 0; // treat reverting oracle as zero-value collateral
}
```

This preserves the existing liveness guarantee for `supplyCollateral` while ensuring that a reverting oracle on any non-liquidated collateral slot cannot block liquidation of the position. Alternatively, allow the liquidator to pass a bitmask of collateral indices to include in the health computation, skipping slots with known-reverting oracles.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {Market, CollateralParams} from "src/interfaces/IMidnight.sol";
// ... standard test imports

contract AlwaysRevertOracle {
    function price() external pure returns (uint256) {
        revert("oracle down");
    }
}

contract SelfImmunizationTest is Test {
    Midnight midnight;
    // loanToken, collateralToken0, collateralToken1, oracle0 (normal), oracle1 (AlwaysRevertOracle)
    // borrower, lender addresses

    function testBorrowerSelfImmunizesAgainstLiquidation() public {
        // 1. Deploy AlwaysRevertOracle for idx1.
        AlwaysRevertOracle revertOracle = new AlwaysRevertOracle();

        // 2. Create market with two collaterals:
        //    collateralParams[0] = (token0, lltv=0.77e18, oracle0=normalOracle)
        //    collateralParams[1] = (token1, lltv=0.77e18, oracle1=revertOracle)

        // 3. Borrower supplies collateral idx0 and borrows (take sell offer).
        //    Assert: debtOf(borrower) > 0

        // 4. Drop oracle0 price so borrower is unhealthy.
        //    Assert: isHealthy(..., borrower) == false  [while idx1 not yet activated]

        // 5. Borrower supplies 1 wei of token1 (idx1).
        //    vm.prank(borrower);
        //    midnight.supplyCollateral(market, 1, 1, borrower);
        //    Assert: collateralBitmap has bit 1 set

        // 6. Attempt liquidation — must revert.
        vm.expectRevert();
        midnight.liquidate(market, 0, 0, 0, borrower, false, address(this), address(0), "");

        // 7. Assert borrower's debt is unchanged (bad debt not realized, lenders not made whole).
        assertGt(midnight.debtOf(id, borrower), 0, "debt persists: borrower immune to liquidation");
    }
}
```

**Expected assertions:**
- Step 4: `isHealthy` returns `false` (before idx1 activation, oracle0 is callable).
- Step 5: `supplyCollateral` succeeds with no revert.
- Step 6: `liquidate` reverts (oracle1 call bubbles up).
- Step 7: `debtOf(borrower) > 0` and `totalUnits` unchanged, confirming bad debt cannot be socialized.
