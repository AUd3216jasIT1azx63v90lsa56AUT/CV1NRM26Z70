Audit Report

## Title
Zero Oracle Price Enables Free Collateral Seizure via `seizedAssets` Path — (`File: src/Midnight.sol`)

## Summary
When a liquidated collateral's oracle returns `0`, calling `liquidate` with `seizedAssets > 0` (and `repaidUnits = 0`) computes `repaidUnits = 0` without reverting, allowing the liquidator to receive real collateral tokens while transferring zero loan tokens. The `repaidUnits > 0` path correctly reverts via division-by-zero, but the `seizedAssets > 0` path has no equivalent guard. The protocol's own Certora README claims zero-returning oracles block `liquidate`, but this is only true for one of the two input modes.

## Finding Description

**Oracle price loop — lines 607–618:**

When `price = 0` for the liquidated collateral:
- `liquidatedCollatPrice = 0` (line 611)
- `maxDebt += mulDivDown(_collateral, 0, ORACLE_PRICE_SCALE) = 0` (line 613)
- `badDebt = zeroFloorSub(originalDebt, mulDivUp(_collateral, 0, ORACLE_PRICE_SCALE)) = originalDebt` (lines 614–616) [1](#0-0) 

**Liquidatability check — line 622:**

`originalDebt > maxDebt` → `originalDebt > 0` → `true`. The position is liquidatable. [2](#0-1) 

**Bad debt zeroes the debt — line 628:**

`_position.debt -= uint128(badDebt)` = `_position.debt -= originalDebt` → `_position.debt = 0`. [3](#0-2) 

**`seizedAssets > 0` branch — line 650:**

```
repaidUnits = mulDivUp(seizedAssets, 0, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif)
            = mulDivUp(seizedAssets * 0 + ORACLE_PRICE_SCALE - 1, 1, ORACLE_PRICE_SCALE) ... 
```

More precisely: `mulDivUp(x, 0, d) = (x*0 + d - 1)/d = (d-1)/d = 0` (integer division). Then `mulDivUp(0, WAD, lif) = 0`. No division by zero; no revert. `repaidUnits = 0`. [4](#0-3) 

**RCF check — lines 659–667:**

After bad debt, `_position.debt = 0` and `maxDebt = 0`, so `maxRepaid = mulDivUp(0, ...) = 0`. The check `repaidUnits <= maxRepaid` → `0 <= 0` → passes silently. [5](#0-4) 

**Token transfers — lines 696 and 717:**

`safeTransfer(collateralToken, receiver, seizedAssets)` — liquidator receives `seizedAssets` collateral tokens.
`safeTransferFrom(loanToken, payer, address(this), 0)` — liquidator pays zero loan tokens. [6](#0-5) 

**Contrast with `repaidUnits > 0` path — line 652:**

`seizedAssets = repaidUnits.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice)` → `mulDivDown(x, ORACLE_PRICE_SCALE, 0)` → division by zero → revert. This is why `oracleZeroCausesLiquidateWithRepaidRevert` holds in `Reverts.spec`, but there is no corresponding rule for the `seizedAssets > 0` path. [7](#0-6) [8](#0-7) 

**Protocol's own invariant violated:**

The `certora/README.md` states: *"A reverting or zero-returning collateral oracle blocks `liquidate`, `withdrawCollateral`, `isHealthy` and `take` whenever the borrower has debt."* This is false for `liquidate` with `seizedAssets > 0`. [9](#0-8) 

The `NoDivisionByZero.spec` explicitly requires `ghostPrice > 0` as a precondition for the `noDivisionByZeroLiquidate` rule, confirming the zero-price case is a known gap, not an impossible scenario. [10](#0-9) 

The `LiquidationProfitability.spec` rule `liquidationLifSeizedAssets` asserts `seizedResult * price > (repaidResult - 1) * ORACLE_PRICE_SCALE()`. With `price = 0` and `repaidResult = 0`, this becomes `0 > -ORACLE_PRICE_SCALE()`, which is vacuously satisfied — the profitability spec does not catch this case. [11](#0-10) 

## Impact Explanation

An unprivileged liquidator calls `liquidate(market, collateralIndex, seizedAssets=N, repaidUnits=0, borrower, ...)` when the collateral oracle returns `0`. The call succeeds: `N` collateral tokens are transferred to the liquidator's `receiver` at zero cost. The borrower's entire debt is simultaneously written off as bad debt, socializing losses to lenders. The borrower loses their collateral with no corresponding debt reduction (debt was already zeroed by bad debt accounting). The liquidator extracts real collateral value — whatever the collateral is worth on the open market — while paying nothing. This constitutes direct theft of assets from the borrower and indirect loss to lenders.

## Likelihood Explanation

Preconditions: (1) a market collateral oracle returns `0` — explicitly modeled as a realistic scenario in the protocol's own Certora specs (`NoDivisionByZero.spec` carves it out as a known exception); (2) the borrower has debt and is liquidatable — trivially satisfied when the oracle returns `0` since `maxDebt = 0`. The attack is permissionless, requires no special role or privileged access, is repeatable for any position in any market whose collateral oracle returns `0`, and can be executed atomically in a single transaction.

## Recommendation

Add an explicit guard in the `seizedAssets > 0` branch to revert when `liquidatedCollatPrice == 0`, mirroring the natural revert that already occurs in the `repaidUnits > 0` branch:

```solidity
if (seizedAssets > 0) {
    require(liquidatedCollatPrice > 0, ZeroOraclePrice());
    repaidUnits = seizedAssets.mulDivUp(liquidatedCollatPrice, ORACLE_PRICE_SCALE).mulDivUp(WAD, lif);
}
```

This aligns the `seizedAssets` path with the documented invariant and with the behavior of the `repaidUnits` path. Additionally, update the `oracleZeroCausesLiquidateWithRepaidRevert` Certora rule to also cover the `seizedAssets > 0` case, and correct the `certora/README.md` claim. [4](#0-3) 

## Proof of Concept

**Minimal manual steps:**

1. Deploy a market with a single collateral whose oracle can be set to return `0`.
2. Have a borrower supply collateral and take on debt (e.g., 1000 collateral tokens, 500 loan units of debt).
3. Set the oracle to return `0`.
4. As an unprivileged attacker, call:
   ```solidity
   midnight.liquidate(
       market,
       collateralIndex,
       /* seizedAssets = */ 1000,   // all collateral
       /* repaidUnits  = */ 0,
       borrower,
       false,                        // normal mode
       attackerAddress,
       address(0),
       ""
   );
   ```
5. Observe: attacker receives 1000 collateral tokens; `safeTransferFrom(loanToken, attacker, contract, 0)` transfers nothing; borrower's debt is zeroed via bad debt; lenders bear the full loss.

**Invariant/fuzz test plan:** Write a Foundry fuzz test that, for any `seizedAssets > 0` and `oracle.price() == 0`, asserts that `liquidate` reverts. The test will currently fail, confirming the vulnerability.
