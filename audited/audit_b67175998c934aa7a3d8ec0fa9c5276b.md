### Title
Stale `isRootRatified` State Persists After Authorization Revocation in `SetterRatifier` — (File: src/ratifiers/SetterRatifier.sol)

---

### Summary

In `SetterRatifier`, any account authorized by a maker can call `setIsRootRatified()` to activate offer roots on the maker's behalf. When the maker later revokes that account's authorization on `Midnight`, the previously activated roots remain ratified in `SetterRatifier` and can still be used to execute offers against the maker's position. This is the direct analog of M-13: state set by a previously-authorized party persists and is consumed without checking whether the setter is still authorized.

---

### Finding Description

**Root cause — `setIsRootRatified` writes state that `isRatified` never re-validates against current authorization:**

`SetterRatifier.setIsRootRatified()` allows any currently-authorized party to write `isRootRatified[maker][root] = true`: [1](#0-0) 

`SetterRatifier.isRatified()` later consumes that state with no check on who set it or whether they are still authorized: [2](#0-1) 

The only guard in `Midnight.take()` is that the ratifier itself is still authorized by the maker: [3](#0-2) 

This check passes as long as Alice still uses `SetterRatifier` for her legitimate offers — it does **not** verify that the specific root was set by a currently-authorized party.

**End-to-end exploit path:**

1. Alice (maker) calls `setIsAuthorized(SetterRatifier, true, Alice)` and `setIsAuthorized(Bob, true, Alice)` on `Midnight`.
2. Bob crafts a Merkle tree (`maliciousRoot`) containing offers where Alice is the maker, e.g., selling credit at a price far below market.
3. Bob calls `setIsRootRatified(Alice, maliciousRoot, true)` on `SetterRatifier` — this is permitted because `isAuthorized[Alice][Bob]` is `true` at this moment.
4. Alice discovers Bob is malicious and calls `setIsAuthorized(Bob, false, Alice)` on `Midnight`.
5. Alice does not know which roots Bob activated (she would need to parse `SetIsRootRatified` events off-chain).
6. `isRootRatified[Alice][maliciousRoot]` remains `true` — it is never cleared by the authorization revocation.
7. Bob (or any third party holding the Merkle proof) submits `take()` with `offer.maker = Alice`, `offer.ratifier = SetterRatifier`, and `ratifierData` pointing to `maliciousRoot`.
8. `take()` passes: `isAuthorized[Alice][SetterRatifier]` is still `true`, and `isRootRatified[Alice][maliciousRoot]` is still `true`.
9. The malicious offer executes against Alice's position. [4](#0-3) 

---

### Impact Explanation

A previously-authorized malicious party can pre-activate roots containing offers unfavorable to the maker (e.g., selling credit at a near-zero price, or taking on debt at an extreme rate) before being revoked. After revocation, those offers remain fully executable. The maker suffers direct financial loss — credit sold below market value or debt taken on at exploitative terms — with no on-chain mechanism to atomically revoke all roots set by the revoked party.

---

### Likelihood Explanation

- The maker must have authorized a malicious party (Bob), which is a realistic scenario for DeFi automation, smart-contract integrations, or compromised operator keys.
- Bob only needs to call `setIsRootRatified` once before being revoked; the window is any time during the authorization period.
- Alice has no atomic "revoke all roots set by Bob" operation; she must manually identify and cancel each root by calling `setIsRootRatified(Alice, root, false)` herself, which requires off-chain event monitoring.
- The attack is passive after setup: Bob (or anyone with the proof) can execute at any time after revocation.

---

### Recommendation

Track the address that activated each root and re-validate authorization at execution time:

```solidity
// Add to SetterRatifier storage:
mapping(address maker => mapping(bytes32 root => address setter)) public rootSetter;

function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRatified[maker][root] = newIsRootRatified;
    rootSetter[maker][root] = msg.sender; // record who set it
    emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
}

function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
    require(msg.sender == MIDNIGHT, NotMidnight());
    ...
    require(isRootRatified[offer.maker][root], NotRatified());
    // Re-validate that the setter is still authorized
    address setter = rootSetter[offer.maker][root];
    require(
        setter == offer.maker || IMidnight(MIDNIGHT).isAuthorized(offer.maker, setter),
        Unauthorized()
    );
    return CALLBACK_SUCCESS;
}
```

Alternatively, document clearly that revoking an authorized party does **not** cancel roots they set, and provide a batch-cancel helper so makers can atomically clear all roots associated with a revoked party.

### Proof of Concept

**Preconditions:** Alice is a maker using `SetterRatifier`. Bob is an address Alice has authorized on `Midnight`.

**Steps:**

1. Bob constructs `maliciousRoot` — a Merkle tree of offers where `offer.maker = Alice`, `offer.buy = false` (Alice sells credit), `offer.tick` set to a very low price tick.
2. Bob calls `SetterRatifier.setIsRootRatified(Alice, maliciousRoot, true)`.
3. Alice calls `Midnight.setIsAuthorized(Bob, false, Alice)` to revoke Bob.
4. Verify: `IMidnight.isAuthorized(Alice, Bob)` returns `false`. ✓
5. Verify: `SetterRatifier.isRootRatified(Alice, maliciousRoot)` returns `true`. ✓ (stale state)
6. An attacker calls `Midnight.take(offer, ratifierData, units, attacker, ...)` where:
   - `offer.maker = Alice`, `offer.ratifier = SetterRatifier`, `offer.buy = false`, `offer.tick = lowPriceTick`
   - `ratifierData = abi.encode(maliciousRoot, leafIndex, proof)`
7. `take()` succeeds: Alice's credit is sold at the attacker-chosen low price, transferring value from Alice to the attacker.

**Expected outcome:** Alice loses credit/assets at an unfavorable price set by the previously-revoked Bob, despite having revoked his authorization.

```text
contract SetterRatifier is ISetterRatifier {
    address public immutable MIDNIGHT;

    mapping(address maker => mapping(bytes32 root => bool)) public isRootRatified;

    constructor(address _midnight) {
        MIDNIGHT = _midnight;
    }

    function setIsRootRatified(address maker, bytes32 root, bool newIsRootRatified) public {
        require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
        isRootRatified[maker][root] = newIsRootRatified;
        emit SetIsRootRatified(msg.sender, maker, root, newIsRootRatified);
    }

    function isRatified(Offer memory offer, bytes memory ratifierData) external view returns (bytes32) {
        require(msg.sender == MIDNIGHT, NotMidnight());
        (bytes32 root, uint256 leafIndex, bytes32[] memory proof) =
            abi.decode(ratifierData, (bytes32, uint256, bytes32[]));
        require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
        require(isRootRatified[offer.maker][root], NotRatified());
        return CALLBACK_SUCCESS;
    }
```
