### Title
Fee-Free Flash Loans Enable Theft of External Rewards on Protocol-Held Tokens — (File: src/Midnight.sol)

---

### Summary

The `flashLoan` function in `Midnight.sol` charges no fee and only verifies that the exact original token amounts are returned. Any caller can temporarily borrow the full balance of any token held by the protocol. If a loan token or collateral token has an external reward mechanism (e.g., an airdrop claimable by the current holder), an attacker can claim those rewards during the callback and return only the principal, stealing value that belongs to lenders and borrowers who deposited those tokens.

---

### Finding Description

The `flashLoan` function transfers tokens to the callback, invokes `onFlashLoan`, then pulls back exactly `assets[i]` per token: [1](#0-0) 

There is no fee deducted, no `claimableSettlementFee` update, and no accounting adjustment of any kind. The only invariant enforced is that the same nominal amount is returned. This is structurally identical to the Caviar `wrap()`/`unwrap()` cycle: in Caviar, a user could wrap one NFT ID and unwrap a different one at zero cost; here, a user can borrow the protocol's entire token balance at zero cost.

The protocol holds two categories of assets on behalf of users:
- **Loan tokens** — deposited by lenders via `take()` (credit positions)
- **Collateral tokens** — deposited by borrowers via `supplyCollateral()` [2](#0-1) 

If any of these tokens implement an external reward mechanism (e.g., `getAirdrop(tokenId)`, staking reward claims, governance reward distributions), an attacker can:

1. Call `flashLoan` requesting the full protocol balance of the target token.
2. Inside `onFlashLoan`, call the token's reward-claim function — rewards are sent to the callback contract, not to the protocol.
3. Return exactly the original `assets[i]` amount to the protocol.
4. Keep all claimed rewards.

The protocol's balance of the principal token is unchanged after the call, so no check fails. The TOKEN SAFETY REQUIREMENTS comment acknowledges that the protocol assumes "Midnight's balance of the token should only decrease on transfer and transferFrom": [3](#0-2) 

This assumption is not violated by the attack — the balance does decrease on the outbound transfer and increase on the inbound `transferFrom`. The airdrop accrues to the callback contract as a separate token or side-effect, entirely outside the protocol's accounting.

---

### Impact Explanation

- **Direct theft**: External rewards (airdrops, staking yields, governance distributions) accruing to tokens held by the protocol are diverted to the attacker in a single transaction.
- **Victims**: All lenders holding credit in a market whose loan token has external rewards, and all borrowers whose collateral token has external rewards.
- **Scale**: The attacker can flash-loan the entire protocol balance in one call, so the theft is not bounded by the attacker's own capital.
- **No fee compensation**: Because `flashLoan` charges zero fee, lenders and borrowers receive nothing in exchange for the temporary loss of ownership of their assets. [4](#0-3) 

---

### Likelihood Explanation

- **Attacker preconditions**: None beyond being able to call `flashLoan` (permissionless) and deploy a callback contract.
- **Token preconditions**: The target token must have an external reward mechanism callable by the current holder. This is common: BAYC/MAYC ApeCoin staking, Lido stETH rebasing, governance tokens with claimable distributions, and NFT-backed tokens with airdrop functions are all real examples.
- **Detection**: The attack is atomic and leaves no on-chain trace distinguishable from a normal flash loan.
- **Frequency**: Any time a reward event occurs for a token held by the protocol, the attack is profitable.

---

### Recommendation

1. **Charge a flash loan fee** and distribute it to lenders/borrowers (analogous to Aave's flash loan fee model). This at minimum compensates depositors for the temporary loss of ownership.
2. **Document the limitation explicitly** in the TOKEN SAFETY REQUIREMENTS: tokens with claimable external rewards (airdrops, staking, governance) should be considered incompatible with the protocol, and market creators should be warned.
3. **Consider restricting flash-loanable tokens** to only the loan token of a specific market rather than any token held by the contract, reducing the attack surface.

---

### Proof of Concept

**Setup**: A market exists with `loanToken = BAYC_WRAPPED` (a token that allows holders to call `claimApeCoinReward()`). Lenders have deposited 1,000 BAYC_WRAPPED into the protocol via `take()`.

**Attack**:

```solidity
contract AirdropThief is IFlashLoanCallback {
    IMidnight midnight;
    address loanToken;
    address rewardToken;

    function attack() external {
        address[] memory tokens = new address[](1);
        tokens[0] = loanToken;
        uint256[] memory amounts = new uint256[](1);
        amounts[0] = IERC20(loanToken).balanceOf(address(midnight)); // full balance
        midnight.flashLoan(tokens, amounts, address(this), "");
    }

    function onFlashLoan(address, address[] calldata tokens, uint256[] calldata amounts, bytes calldata)
        external returns (bytes32)
    {
        // Temporarily hold all protocol loan tokens → claim external reward
        IBaycWrapped(tokens[0]).claimApeCoinReward(); // reward sent to address(this)
        // Return exactly the original amount — no fee required
        IERC20(tokens[0]).approve(address(midnight), amounts[0]);
        return CALLBACK_SUCCESS;
    }
}
```

**Result**: The attacker receives all ApeCoin rewards that should have accrued to the 1,000 BAYC_WRAPPED held on behalf of lenders. The protocol's `loanToken` balance is unchanged; no revert occurs. The lenders receive nothing. [1](#0-0)

    function flashLoan(address[] calldata tokens, uint256[] calldata assets, address callback, bytes calldata data)
        external
    {
        require(tokens.length == assets.length, InconsistentInput());
        emit EventsLib.FlashLoan(msg.sender, tokens, assets, callback);
        for (uint256 i = 0; i < tokens.length; i++) {
            SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
        }
        require(
            IFlashLoanCallback(callback).onFlashLoan(msg.sender, tokens, assets, data) == CALLBACK_SUCCESS,
            WrongFlashLoanCallbackReturnValue()
        );
        for (uint256 i = 0; i < tokens.length; i++) {
            SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
        }
    }
