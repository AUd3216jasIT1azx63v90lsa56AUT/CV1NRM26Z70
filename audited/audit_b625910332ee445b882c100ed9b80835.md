### Title
Division by Zero in `liquidate` When `collateralIndex` Is Not in Borrower's Bitmap and `repaidUnits > 0` - (File: src/Midnight.sol)

### Summary

In `Midnight.liquidate`, `liquidatedCollatPrice` is initialized to `0` and only assigned inside the bitmap loop when `i == collateralIndex`. If `collateralIndex` is not set in the borrower's `collateralBitmap`, the loop never assigns it, leaving it `0`. When `repaidUnits > 0`, the subsequent `mulDivDown(ORACLE_PRICE_SCALE, liquidatedCollatPrice)` at line 652 divides by zero, causing an unexpected panic revert. The `NoDivisionByZero.spec` formal verification explicitly assumes the collateral is activated (line 125), confirming this case is unguarded in the Solidity code.

### Finding Description

**Root cause — `src/Midnight.sol` lines 602–652:** [1](#0-0) 

`liquidatedCollatPrice` is declared as `uint256` (default `0`) and is only written inside the while-loop when `i == collateralIndex`: [2](#0-1) 

The loop iterates only over bits set in `_position.collateralBitmap`. If `collateralIndex` is not set in that bitmap, the assignment `liquidatedCollatPrice = price` is never reached.

After the loop, when `repaidUnits > 0`: [3](#0-2) 

`mulDivDown(ORACLE_PRICE_SCALE, 0)` is `(repaidUnits * lif / WAD * ORACLE_PRICE_SCALE) / 0` — a division-by-zero panic.

**No existing check prevents this.** The only pre-conditions checked before entering the `if (repaidUnits > 0 || seizedAssets > 0)` block are:

- `atMostOneNonZero(repaidUnits, seizedAssets)` — does not constrain `collateralIndex` vs. bitmap [4](#0-3) 
- `_position.debt > 0` — only checks debt exists [5](#0-4) 
- `NotLiquidatable` — checks unhealthiness or post-maturity, not bitmap membership [6](#0-5) 

The NatSpec at line 578 explicitly states that zero-amount liquidations can use a non-activated collateral, implying non-zero liquidations should not — but no Solidity guard enforces this: [7](#0-6) 

The Certora spec acknowledges the gap by requiring the collateral to be activated as a precondition for the no-division-by-zero proof: [8](#0-7) 

**Exploit flow:**

1. Borrower has debt and collateral only at index 0 (bitmap bit 0 set, bit 1 not set).
2. Position becomes liquidatable (unhealthy or post-maturity).
3. Liquidator calls `liquidate(market, collateralIndex=1, seizedAssets=0, repaidUnits=X)` with `X > 0`.
4. Bitmap loop processes index 0 only; `liquidatedCollatPrice` stays `0`.
5. `NotLiquidatable` passes (position is genuinely unhealthy/post-maturity).
6. Line 652: `seizedAssets = X.mulDivDown(lif, WAD).mulDivDown(ORACLE_PRICE_SCALE, 0)` → panic division by zero.

The `SetterRatifier` path mentioned in the question (`setIsRootRatified → take → liquidate`) is one valid way to create the borrower's debt position, but the vulnerability is independent of how the position was created.

### Impact Explanation

Any call to `liquidate` with `repaidUnits > 0` and a `collateralIndex` not present in the borrower's `collateralBitmap` reverts with a division-by-zero panic instead of a clean, descriptive error. This violates the protocol invariant that liquidations must not revert unexpectedly. In post-maturity mode, every borrower with debt is liquidatable, so any liquidator can trigger this revert on any borrower by choosing an unactivated `collateralIndex`. In normal mode, the same applies to any unhealthy borrower. The position is not permanently unliquidatable (a correct `collateralIndex` still works), but the unexpected revert breaks tooling, off-chain liquidation bots, and any contract that does not anticipate a panic revert from this path.

### Likelihood Explanation

**Preconditions:**
- Borrower has debt (`_position.debt > 0`) — standard state.
- Position is liquidatable (unhealthy or post-maturity) — required for `NotLiquidatable` to pass.
- Liquidator passes a `collateralIndex` not in the borrower's bitmap with `repaidUnits > 0`.

This is trivially reachable in post-maturity mode (no price condition needed) and in normal mode whenever a borrower is unhealthy. A liquidation bot that iterates over all market `collateralParams` indices without checking the borrower's bitmap will hit this. The condition is repeatable and deterministic.

### Recommendation

Add an explicit check before the `if (repaidUnits > 0 || seizedAssets > 0)` block (or inside it) that requires `collateralIndex` to be set in the borrower's bitmap whenever a non-zero amount is being liquidated:

```solidity
if (repaidUnits > 0 || seizedAssets > 0) {
    require(
        _position.collateralBitmap.getBit(collateralIndex),
        CollateralNotActivated()
    );
    // ... rest of block
}
```

This matches the documented intent (NatSpec line 578–579) and closes the gap left open by the formal verification assumption at `NoDivisionByZero.spec` line 125.

    vm.warp(market.maturity + 1);

    // Expected: revert with Panic(0x12) division by zero.
    vm.expectRevert(); // panic: division by zero
    midnight.liquidate(
        market,
        1,          // collateralIndex NOT in bitmap
        0,          // seizedAssets
        1,          // repaidUnits > 0
        borrower,
        true,       // postMaturityMode
        address(this),
        address(0),
        ""
    );

    // A correct fix would revert with a named error like CollateralNotActivated().
