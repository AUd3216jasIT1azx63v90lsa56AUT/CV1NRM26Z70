### Title
Front-running `setRecoveryAddress` with permissionless `claimCorrupted` diverts entire pool to compromised address - (File: src/ConfidencePool.sol)

### Summary
`setRecoveryAddress` imposes no guard once the outcome is `CORRUPTED`, and `claimCorrupted` is fully permissionless. An attacker who controls the current (compromised) `recoveryAddress` can observe a sponsor's pending `setRecoveryAddress` transaction in the mempool and front-run it with `claimCorrupted`, sweeping the entire pool to the old address before the update lands.

### Finding Description
`setRecoveryAddress` allows the pool owner (sponsor) to update the sweep destination at any time, including after the moderator has already called `flagOutcome(CORRUPTED, false, address(0))`:

```solidity
function setRecoveryAddress(address newRecoveryAddress) external onlyOwner {
    if (newRecoveryAddress == address(0)) revert InvalidRecoveryAddress();
    address oldRecoveryAddress = recoveryAddress;
    recoveryAddress = newRecoveryAddress;
    emit RecoveryAddressUpdated(oldRecoveryAddress, newRecoveryAddress);
}
``` [1](#0-0) 

There is no check that `outcome == PoolStates.Outcome.UNRESOLVED` or that `claimsStarted == false`. Meanwhile, `claimCorrupted` is entirely permissionless — any caller can invoke it the moment the outcome is `CORRUPTED`:

```solidity
function claimCorrupted() external nonReentrant {
    if (outcome != PoolStates.Outcome.CORRUPTED) revert OutcomeNotSet();
    if (goodFaith && bountyClaimed < bountyEntitlement) revert MustClaimBountyFirst();
    ...
    stakeToken.safeTransfer(recoveryAddress, toSweep);
}
``` [2](#0-1) 

`recoveryAddress` is read at execution time, not at flag time. The race window opens the moment `flagOutcome` is mined and closes only when `claimCorrupted` is mined. Any actor who controls the current `recoveryAddress` can exploit this window.

`DESIGN.md §10` documents `recoveryAddress` as mutable but does not acknowledge or accept this race condition: [3](#0-2) 

### Impact Explanation
Under bad-faith CORRUPTED, `claimCorrupted` sweeps `stakeToken.balanceOf(address(this))` — the entire pool including all stakers' principal and the full bonus — to `recoveryAddress`. If the attacker controls the old `recoveryAddress` and successfully front-runs the sponsor's update, every token in the pool is permanently redirected to the compromised address. There is no recovery path: `claimsStarted` is set to `true` by `claimCorrupted`, which blocks any moderator re-flag, and the balance is zero so no subsequent sweep is possible.

### Likelihood Explanation
Three conditions must align: (1) the moderator flags bad-faith CORRUPTED, (2) the current `recoveryAddress` is compromised and controlled by an attacker, and (3) the sponsor broadcasts `setRecoveryAddress` before `claimCorrupted` is called. Condition (1) is a normal protocol event; condition (2) is the trigger that motivates the sponsor's update; condition (3) is the natural sponsor response. On a public mempool chain the attacker has a clear, low-cost front-run vector (a single permissionless call). Likelihood is low-to-medium given the prerequisite of a compromised `recoveryAddress`, but the exploit requires no special privilege once those conditions hold.

### Recommendation
**Short term:** Document that `recoveryAddress` should be treated as immutable once the pool is live and that sponsors must ensure it is a secure, non-compromisable address before pool creation. Warn that updating it after `flagOutcome(CORRUPTED, ...)` is unsafe.

**Long term:** Add a guard in `setRecoveryAddress` that reverts if `outcome != PoolStates.Outcome.UNRESOLVED`, preventing any post-resolution change. Alternatively, snapshot `recoveryAddress` into the resolution state at `flagOutcome` time (analogous to how `snapshotTotalStaked` is captured) so the sweep destination is frozen at flag time and immune to subsequent updates.

### Proof of Concept
1. Pool is deployed with `recoveryAddress = 0xCompromised`. The attacker later gains control of `0xCompromised`.
2. The moderator calls `flagOutcome(PoolStates.Outcome.CORRUPTED, false, address(0))`. Outcome is now bad-faith CORRUPTED; `claimsStarted` is still `false`.
3. The sponsor observes that `0xCompromised` is under attacker control and broadcasts `setRecoveryAddress(0xSafe)` with standard gas.
4. The attacker sees the pending `setRecoveryAddress` transaction in the mempool and immediately submits `claimCorrupted()` with a higher gas price.
5. `claimCorrupted()` is mined first. It reads `recoveryAddress == 0xCompromised`, sweeps the entire pool balance to `0xCompromised`, and sets `claimsStarted = true`.
6. `setRecoveryAddress(0xSafe)` is mined next — the state update succeeds, but the pool balance is already zero and `claimsStarted` is `true`, permanently blocking any moderator re-flag or further sweep.
7. All stakers' principal and the full bonus pool are in the attacker's hands with no on-chain recourse.
