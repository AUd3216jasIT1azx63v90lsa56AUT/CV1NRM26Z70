### Title
`buyWithAssetsTargetAndWithdrawCollateral` DoS for sell offers where `offerPrice + settlementFee > WAD` due to uncaught revert in `buyerAssetsToUnits` - (File: src/periphery/TakeAmountsLib.sol)

### Summary
For sell offers where `offerPrice + settlementFee > WAD`, `buyerAssetsToUnits` reverts with `PriceGreaterThanOne()` at line 28 of `TakeAmountsLib.sol`. This call is not wrapped in a try/catch inside `buyWithAssetsTargetAndWithdrawCollateral`, so the entire transaction reverts. Midnight's `take()` has no equivalent `buyerPrice <= WAD` guard and would accept the same offer, creating a strict divergence between periphery and core revert behavior.

### Finding Description
**Root cause:** `require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne())` at `TakeAmountsLib.sol:28`. For sell offers, `buyerPrice = offerPrice + settlementFee`. This check is absent in Midnight's `take()`.

**Code path:**

1. `buyWithAssetsTargetAndWithdrawCollateral` (MidnightBundles.sol:208-214) calls `TakeAmountsLib.buyerAssetsToUnits(...)` directly inside `min(...)`, outside any try/catch.
2. `buyerAssetsToUnits` (TakeAmountsLib.sol:22-28) computes `buyerPrice = offerPrice + settlementFee` for sell offers and executes `require(buyerPrice <= WAD)`.
3. Midnight's `take()` (Midnight.sol:361-363) computes the same `buyerPrice` but has no such require — it proceeds to `units.mulDivUp(buyerPrice, WAD)` unconditionally.

**Attacker inputs:**
- Create a sell offer (`offer.buy = false`) with `offer.tick = MAX_TICK = 5820` (or any tick where `tickToPrice(tick) > WAD - settlementFee`).
- Market must have any non-zero settlement fee (e.g., `settlementFeeCbp6 = 1`, giving `settlementFee = CBP = 1e12`).

**Exploit flow:**
1. Maker (unprivileged) creates a sell offer at `tick = MAX_TICK` in a market with non-zero settlement fee.
2. Taker calls `buyWithAssetsTargetAndWithdrawCollateral` including this offer in `takes[]`.
3. Loop iteration hits this offer; `buyerAssetsToUnits` is called with `buyerPrice ≈ WAD + 5e15 > WAD`.
4. `require(buyerPrice <= WAD)` reverts with `PriceGreaterThanOne()`.
5. Since this is outside try/catch, the entire transaction reverts.
6. Calling `IMidnight.take()` directly on the same offer would succeed — no equivalent check exists.

**Why existing protections fail:** The try/catch at MidnightBundles.sol:215-221 only wraps the `take()` call, not the preceding `buyerAssetsToUnits` call. The comment at line 175 ("Reverts if TakeAmountsLib or ConsumableUnitsLib reverts") documents this behavior but does not justify it as correct — it is a design gap.

### Impact Explanation
`buyWithAssetsTargetAndWithdrawCollateral` is permanently unusable for any sell offer where `tickToPrice(offer.tick) + settlementFee(id, ttm) > WAD`. For markets with the maximum 360-day settlement fee of `0.005e18`, this affects all sell offers with `tick ≥ ~5819`. Users are forced to either call Midnight's `take()` directly or use `buyWithUnitsTargetAndWithdrawCollateral`, defeating the purpose of the assets-target bundle.

### Likelihood Explanation
The condition is deterministically reachable: `tickToPrice(MAX_TICK) ≈ WAD - 5e11`, and `MAX_SETTLEMENT_FEE_360_DAYS = 5e15 >> 5e11`. Any market with a non-zero settlement fee at any breakpoint will trigger this for offers at or near `tick = MAX_TICK`. No special oracle values, admin actions, or user mistakes are required — a maker simply places a sell offer at a high tick, which is a normal protocol action.

### Recommendation
Remove the `require(buyerPrice <= WAD)` guard for the sell-offer branch in `buyerAssetsToUnits`, since Midnight's `take()` imposes no such restriction. For sell offers, the inverse is well-defined: `units = targetBuyerAssets.mulDivDown(WAD, buyerPrice)` regardless of whether `buyerPrice > WAD`. Alternatively, wrap the `buyerAssetsToUnits` call in a try/catch in `buyWithAssetsTargetAndWithdrawCollateral` and skip the offer on revert, consistent with how `take()` failures are handled.
