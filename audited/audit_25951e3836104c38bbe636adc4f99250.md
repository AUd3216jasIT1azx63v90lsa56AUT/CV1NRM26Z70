Audit Report

## Title
Missing `units > 0` guard allows zero-cost callback invocation and transient liquidation lock manipulation - (`src/Midnight.sol`)

## Summary
`take()` contains no guard against `units == 0`. When called with `units=0`, `offer.buy=True`, and a non-zero `offer.callback`, all accounting deltas are zero yet the function unconditionally sets the seller's liquidation lock in transient storage (line 444) and dispatches `buyerCallback.onBuy(...)` with `buyerAssets=0` (lines 445–453). An unprivileged taker can trigger the maker's callback at zero token cost, violating the invariant that a zero-unit take is a no-op.

## Finding Description

**Root cause**: No `require(units > 0)` exists anywhere in `take()`. Confirmed by grep returning zero matches for `require.*units` in `src/Midnight.sol`.

**Exact code path** (`src/Midnight.sol`):

- Lines 363–364: `buyerAssets = units.mulDivDown(buyerPrice, WAD) = 0`; `sellerAssets = 0`.
- Lines 367–369: `consumed[offer.maker][offer.group] += 0`; `require(newConsumed <= offer.maxAssets)` passes trivially and is repeatable indefinitely since `newConsumed` never advances.
- Lines 382–389: All credit/debt/fee deltas are zero; position state is unchanged.
- Line 444: `bool wasLocked = UtilsLib.tExchange(LIQUIDATION_LOCK_SLOT, id, seller, true)` — executed **unconditionally**, regardless of `units`.
- Lines 445–453: `if (buyerCallback != address(0))` — `buyerCallback = offer.callback` for a buy offer. `IBuyCallback(buyerCallback).onBuy(id, market, 0, 0, 0, buyer, data)` is dispatched with no guard on `buyerAssets > 0`.
- Lines 455–456: `safeTransferFrom(..., 0)` executes silently (standard ERC20 does not revert on zero).
- Line 476: Health check passes because `sellerDebtIncrease == 0`.

**Why existing checks fail**:
- `require(UtilsLib.atMostOneNonZero(offer.maxAssets, offer.maxUnits))` — checks mutual exclusivity only, not that `units > 0`.
- `require(newConsumed <= offer.maxAssets)` — `newConsumed` is unchanged when `buyerAssets=0`, so this always passes and the same offer can be targeted indefinitely.
- `require(offer.maker != taker)` — prevents self-take but any third party satisfies this.

## Impact Explanation

1. **Callback invoked at zero economic cost**: `offer.callback.onBuy(id, market, 0, 0, 0, buyer, data)` executes. Any side effects inside the callback (state writes, external calls, event emissions, interactions with other protocols) run for free. The callback must return `CALLBACK_SUCCESS` but is not required to transfer any tokens since `buyerAssets - sellerAssets = 0`.
2. **Liquidation lock observable during callback**: At the moment `onBuy` executes, `liquidationLocked(id, seller) == true` in transient storage. The callback can read this via `IMidnight.liquidationLocked(id, taker)` and branch on it or relay it to other contracts.
3. **Unlimited repeatability**: Since `consumed` never advances on a zero-unit take, the same offer can be targeted in an unbounded loop at only gas cost, enabling sustained griefing of any callback-bearing buy offer.

## Likelihood Explanation

**Preconditions**: A live buy offer with `offer.callback != address(0)` and `offer.maxAssets > 0` must exist — standard for lend-side offers using flash-lend callbacks. The taker must not equal `offer.maker` (trivially satisfied by any third party). No privileged access, oracle manipulation, or special token behavior required.

**Feasibility**: Fully reachable with a single external call. The existing fuzz test `testBuyBuyerCallback` in `test/TakeTest.sol` bounds `units` from 0, confirming the protocol does not reject `units=0` at the test level.

## Recommendation

Add a guard at the top of `take()` after the existing checks:

```solidity
require(units > 0, ZeroUnits());
```

This ensures all downstream accounting, callback dispatch, and lock-setting are only reached when a meaningful trade is occurring, restoring the invariant that a zero-unit take is a no-op.

## Proof of Concept

1. Deploy a buy offer with `offer.buy=True`, `offer.maxAssets > 0`, `offer.callback = <any contract returning CALLBACK_SUCCESS>`.
2. Call `take(offer, ratifierData, 0, taker, receiver, address(0), "")` from any address that is not `offer.maker`.
3. Observe: `onBuy` is called with `buyerAssets=0`; `liquidationLocked(id, taker) == true` during the callback; both `safeTransferFrom` calls transfer 0 tokens without reverting; the health check passes.
4. Repeat step 2 indefinitely — `consumed` never advances, so `require(newConsumed <= offer.maxAssets)` always passes. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
