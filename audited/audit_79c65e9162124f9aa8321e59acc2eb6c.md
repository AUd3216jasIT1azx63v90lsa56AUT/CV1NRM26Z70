Audit Report

## Title
Zero-`buyerAssets` Rounding Bypasses Fully-Consumed Guard on Assets-Based Buy Offer, Enabling Unbounded Position Mutation Without Payment — (`src/Midnight.sol`)

## Summary
When a buy offer uses `maxAssets`-based consumption and `buyerPrice < WAD`, a taker can supply `units` small enough that `units.mulDivDown(buyerPrice, WAD) == 0`. The consumed counter increments by zero, so the `newConsumed <= maxAssets` guard passes even when the offer is already fully consumed. Position credit, debt, and `totalUnits` are mutated and the maker's `onBuy` callback is invoked — all with zero token transfer — and the call is repeatable without limit. The repository's own test `testBugBuyMaxAssetsBypass` confirms and documents this exact state corruption.

## Finding Description

**Root cause — `src/Midnight.sol` lines 363–369:**

```solidity
uint256 buyerAssets = offer.buy
    ? units.mulDivDown(buyerPrice, WAD)   // rounds to 0 when units*buyerPrice < WAD
    : units.mulDivUp(buyerPrice, WAD);

if (offer.maxAssets > 0) {
    newConsumed = consumed[offer.maker][offer.group] += offer.buy ? buyerAssets : sellerAssets;
    require(newConsumed <= offer.maxAssets, ConsumedAssets());  // maxAssets+0 <= maxAssets ✓
}
```

The consumed counter is incremented by `buyerAssets`, not by `units`. When `buyerPrice < WAD` (any tick below `MAX_TICK`) and `units` is small (e.g., 1), `mulDivDown` truncates to 0. Adding 0 to a saturated counter leaves `newConsumed == maxAssets`, satisfying the `<=` check.

**Exploit flow:**
1. Maker creates a buy offer with `maxAssets = N`, `tick < MAX_TICK` (so `buyerPrice < WAD`).
2. Offer is fully consumed: `consumed[maker][group] == maxAssets` (via legitimate fills or `setConsumed`).
3. Taker calls `take(offer, ..., units=1, ...)`.
4. `buyerAssets = mulDivDown(1, buyerPrice, WAD) = 0`.
5. `newConsumed = maxAssets + 0 = maxAssets`; `require(maxAssets <= maxAssets)` passes.
6. Execution continues at lines 382–417: `buyerCreditIncrease = 1`, `sellerDebtIncrease = 1`, `totalUnits` incremented.
7. `onBuy` callback invoked at line 449 with `buyerAssets=0, units=1`.
8. Token transfers at lines 455–456: `safeTransferFrom(..., 0)` — no tokens move.
9. Step 3–8 is repeatable indefinitely.

**Why existing checks fail:**
- The `ConsumedAssets` guard uses `<=`, not `<`, and compares against a counter that did not increase. It cannot detect a zero-increment on a saturated counter.
- No check enforces `units == 0` when `buyerAssets == 0` for assets-based buy offers.
- The Certora spec rule `takeConsumedAtMaxUnchangedAssets` (lines 88–96 of `certora/specs/Consume.spec`) is rendered vacuous by the `NONDET` summary for `mulDivDown` (line 11 of the same file): the prover never explores the concrete zero-rounding path, so the invariant is unverified for this case.

## Impact Explanation
Each call grants the buyer 1 unit of credit and burdens the seller with 1 unit of debt, with zero token payment — a direct violation of the invariant that credit is funded by asset transfer. The maker's `onBuy` callback is invoked on an offer the maker considers exhausted; callback logic tracking cumulative fills or gating on `buyerAssets > 0` will behave incorrectly. The attack is unbounded and repeatable, enabling unlimited inflation of credit/debt positions and `totalUnits` at zero cost. This constitutes unauthorized state mutation and accounting integrity failure.

## Likelihood Explanation
All preconditions are reachable by any unprivileged taker: `buyerPrice < WAD` holds for every tick below `MAX_TICK` (the common case); full consumption is a normal end-state after legitimate fills; `units=1` is a valid input. No privileged access, oracle manipulation, or victim error is required. The attack is executable in a single transaction and repeatable across blocks.

## Recommendation
Add a guard immediately after computing `buyerAssets` (and `sellerAssets`) that rejects a non-zero `units` input when the computed asset amount rounds to zero:

```solidity
if (offer.maxAssets > 0) {
    require(units == 0 || (offer.buy ? buyerAssets : sellerAssets) > 0, ZeroAssets());
}
```

Alternatively, change the consumed-check comparison from `<=` to `<` and separately require `buyerAssets > 0 || units == 0` for assets-based offers. The core invariant to enforce is: *a non-trivial take (units > 0) on an assets-based offer must always produce a non-zero asset amount*.

## Proof of Concept
The repository contains a complete, passing proof at `test/TakeTest.sol` lines 857–889 (`testBugBuyMaxAssetsBypass`):

1. Set `maxAssets = 1`, `tick = MAX_TICK - 16` (so `offerPrice < WAD`).
2. Pre-saturate: `midnight.setConsumed(group, 1, lender)`.
3. Call `take(1, borrower, lenderOffer)`.
4. Assert `buyerAssets == 0`, `sellerAssets == 0` (no payment).
5. Assert `consumed` unchanged at `maxAssets`.

```text
    // Show that a buy offer with offerPrice < WAD can be taken with units > 0
    function testBugBuyMaxAssetsBypass() public {
        deal(address(loanToken), lender, 0); // lender pays 0
        collateralize(market, borrower, 100);

        lenderOffer.maxUnits = 0;
        lenderOffer.maxAssets = 1;
        lenderOffer.tick = MAX_TICK - 16; // offerPrice < WAD

        // Fully consume the offer before the take.
        vm.prank(lender);
        midnight.setConsumed(lenderOffer.group, lenderOffer.maxAssets, lender);

        uint256 lenderCreditBefore = midnight.creditOf(id, lender);
        uint256 borrowerDebtBefore = midnight.debtOf(id, borrower);
        uint256 totalUnitsBefore = midnight.totalUnits(id);
        uint256 lenderBalBefore = loanToken.balanceOf(lender);
        uint256 borrowerBalBefore = loanToken.balanceOf(borrower);

        (uint256 buyerAssets, uint256 sellerAssets) = take(1, borrower, lenderOffer);

        assertEq(buyerAssets, 0);
        assertEq(sellerAssets, 0);

        // Nothing observable to the cap or token balances changed:
        assertEq(midnight.consumed(lender, lenderOffer.group), lenderOffer.maxAssets);
        assertEq(loanToken.balanceOf(lender), lenderBalBefore);
        assertEq(loanToken.balanceOf(borrower), borrowerBalBefore);
        // But position state strictly changed:
        assertGt(midnight.creditOf(id, lender), lenderCreditBefore);
        assertGt(midnight.debtOf(id, borrower), borrowerDebtBefore);
        assertGt(midnight.totalUnits(id), totalUnitsBefore);
    }
```

```text
/// If consumed is already at or above maxAssets before a take in assets mode, it remains unchanged.
rule takeConsumedAtMaxUnchangedAssets(env e, Midnight.Offer offer, bytes ratifierData, uint256 units, address taker, address receiver, address takerCallback, bytes takerCallbackData) {
    require offer.maxUnits == 0;

    uint256 consumedBefore = consumed(offer.maker, offer.group);

    take(e, offer, ratifierData, units, taker, receiver, takerCallback, takerCallbackData);

    assert consumedBefore >= offer.maxAssets => consumed(offer.maker, offer.group) == consumedBefore;
}
```
