### Title
Authorized Operator Can Re-Enable a Maker-Revoked Root in SetterRatifier - (File: src/ratifiers/SetterRatifier.sol)

### Summary
`SetterRatifier.setIsRootRatified` permits any address holding `isAuthorized[maker][operator]=true` to set `isRootRatified[maker][root]` to either `true` or `false` with no distinction between enabling and revoking. After a maker explicitly revokes a root by calling `setIsRootRatified(maker, root, false)`, a still-authorized operator can immediately call `setIsRootRatified(maker, root, true)`, restoring fillability of every offer in that Merkle tree. This violates the maker's intent to cancel those offers.

### Finding Description
The sole guard in `setIsRootRatified` is:

```solidity
require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
isRootRatified[maker][root] = newIsRootRatified;
``` [1](#0-0) 

The check is identical whether `newIsRootRatified` is `true` or `false`. There is no "revocation is final" path. Contrast this with `EcrecoverRatifier.cancelRoot`, which only ever sets `isRootCanceled[maker][root] = true` and provides no inverse function, making cancellation permanently one-way. [2](#0-1) 

**Exploit flow:**

1. Maker calls `setIsRootRatified(maker, root, true)` — offers in the tree become fillable.
2. Maker calls `setIsRootRatified(maker, root, false)` — intending to cancel all offers in the tree.
3. Operator (still holding `isAuthorized[maker][operator]=true`) calls `setIsRootRatified(maker, root, true)`.
4. `isRootRatified[maker][root]` is now `true` again.
5. Any taker calls `midnight.take(offer, abi.encode(root, leafIndex, proof), ...)`.
6. `isRatified` passes because `isRootRatified[offer.maker][root]` is `true`. [3](#0-2) 

The existing authorization check does not stop this: it only verifies the caller is the maker or an authorized delegate, not whether the action being taken is a re-enable after a maker-initiated revoke. No nonce, no timestamp, no "revoked" flag, and no one-way latch exist.

The existing test suite confirms operators can ratify on behalf of makers but does not test the revoke-then-re-enable sequence. [4](#0-3) 

### Impact Explanation
Offers the maker believed permanently cancelled become fillable again without the maker's knowledge or consent. A taker (or the operator themselves acting as taker) can drain the maker's approved loan tokens by filling offers the maker had revoked. The maker has no on-chain signal that the revocation was undone unless they monitor events.

### Likelihood Explanation
Preconditions are realistic and common: maker delegates management of offer trees to an operator (e.g., a keeper bot or a co-signer contract), then decides to cancel a batch of offers by revoking the root. If the maker does not atomically revoke the operator's authorization in the same transaction, the window is open. The operator can re-enable the root in the very next block. The attack is repeatable as long as `isAuthorized[maker][operator]` remains `true`.

### Recommendation
Make revocation one-way, mirroring `EcrecoverRatifier`. Replace the bidirectional setter with a dedicated revoke function that only ever sets the flag to `false` and cannot be reversed:

```solidity
mapping(address maker => mapping(bytes32 root => bool)) public isRootRevoked;

function revokeRoot(address maker, bytes32 root) external {
    require(maker == msg.sender || IMidnight(MIDNIGHT).isAuthorized(maker, msg.sender), Unauthorized());
    isRootRevoked[maker][root] = true;
    emit RevokeRoot(msg.sender, maker, root);
}
```

Then in `isRatified`, check `!isRootRevoked[offer.maker][root]` instead of `isRootRatified[offer.maker][root]`. Alternatively, restrict the ability to set `newIsRootRatified = true` to `maker == msg.sender` only, while allowing operators to set it to `false`.

### Proof of Concept

```solidity
function testOperatorReEnablesRevokedRoot() public {
    // Setup
    address maker = lender;
    address operator = borrower;
    Offer memory offer = makeOffer(maker);
    bytes32 root = HashLib.hashOffer(offer);

    // Maker authorizes operator
    vm.prank(maker);
    midnight.setIsAuthorized(operator, true, maker);

    // Maker ratifies root
    vm.prank(maker);
    setterRatifier.setIsRootRatified(maker, root, true);
    assertTrue(setterRatifier.isRootRatified(maker, root));

    // Maker revokes root (intends to cancel offers)
    vm.prank(maker);
    setterRatifier.setIsRootRatified(maker, root, false);
    assertFalse(setterRatifier.isRootRatified(maker, root));

    // Operator re-enables root — maker's revocation is NOT final
    vm.prank(operator);
    setterRatifier.setIsRootRatified(maker, root, true);
    assertTrue(setterRatifier.isRootRatified(maker, root)); // BUG: should revert or remain false

    // isRatified now succeeds — offer is fillable again
    vm.prank(address(midnight));
    bytes32 result = setterRatifier.isRatified(offer, abi.encode(root, 0, new bytes32[](0)));
    assertEq(result, CALLBACK_SUCCESS); // BUG: take would succeed against maker's intent
}
```

Expected assertion: the re-enable call should revert, or `isRatified` should revert with `NotRatified`. Instead both pass, confirming the invariant is broken.

        require(msg.sender == MIDNIGHT, NotMidnight());
        (bytes32 root, uint256 leafIndex, bytes32[] memory proof) =
            abi.decode(ratifierData, (bytes32, uint256, bytes32[]));
        require(HashLib.isLeaf(root, HashLib.hashOffer(offer), leafIndex, proof), InvalidProof());
        require(isRootRatified[offer.maker][root], NotRatified());
        return CALLBACK_SUCCESS;
    }
