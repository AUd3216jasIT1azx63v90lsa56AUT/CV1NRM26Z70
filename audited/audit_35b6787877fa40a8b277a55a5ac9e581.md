Audit Report

## Title
Negative-Rebasing Loan Token Breaks Solvency Invariant via Missing Balance-Delta Verification in `safeTransferFrom` - (File: src/libraries/SafeTransferLib.sol)

## Summary
`SafeTransferLib.safeTransferFrom` verifies only call success and ERC20 return value, never the actual balance delta received. Because `touchMarket` imposes no restriction on the loan token type, an unprivileged attacker can deploy a negative-rebasing ERC20 as the loan token, establish `withdrawable` and `claimableSettlementFee` balances through normal protocol interactions, then trigger a rebase that silently reduces `balanceOf(midnight, loanToken)` while all accounting state remains unchanged, permanently violating the solvency invariant and making full redemption impossible for lenders and fee claimers.

## Finding Description

**Root cause — `SafeTransferLib.safeTransferFrom` (lines 24–34):**

`safeTransferFrom` calls `transferFrom` and checks only that the call did not revert and did not return `false`. No `balanceOf(address(this))` snapshot is taken before or after the call. Any transfer that succeeds nominally is accepted regardless of the actual balance delta received. [1](#0-0) 

**Permissionless market creation with unrestricted loan token — `touchMarket` (lines 755–791):**

`touchMarket` validates maturity, collateral params, LLTV tiers, and maxLif values, but places no restriction on `market.loanToken`. Any address with deployed code is accepted. An unprivileged caller can create a market whose loan token is a contract they fully control. [2](#0-1) 

**Accounting increment without delta verification — `repay` (lines 508–520):**

`withdrawable` is incremented by the nominal `units` value at line 509 before the transfer at line 520. The transfer is never checked against an actual balance delta. [3](#0-2) 

**Same pattern in `take` (lines 418 and 455):**

`claimableSettlementFee` is incremented at line 418 by `buyerAssets - sellerAssets`, and the corresponding inbound transfer occurs at line 455 with no balance delta verification. [4](#0-3) [5](#0-4) 

**Formal verification explicitly excludes rebasing tokens — `Solvency.spec` (line 31):**

The spec states: *"Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits."* The `tokenBalanceCorrect` strong invariant (lines 162–163), which asserts `tokenBalances[token][currentContract] >= collateralSum + withdrawableSum + claimableSettlementFee - flashloans`, is therefore **not proven** for rebasing tokens. This is a prover-level assumption, not an on-chain guard. [6](#0-5) [7](#0-6) 

**Protocol's own NatSpec documents the assumption without enforcing it on-chain — `Midnight.sol` (lines 133–139):**

The contract NatSpec lists token safety requirements including "Midnight's balance of the token should only decrease on transfer and transferFrom" and "Midnight must send/receive exactly the requested amount on transfers." These are documentation-level constraints with no corresponding on-chain enforcement. [8](#0-7) 

**Protocol's own open-concern list confirms no on-chain exclusion — `live_context.json` (line 233):**

The project's `live_context.json` lists *"fee-on-transfer, rebasing, false-return, ERC777-like hooks, and non-standard decimals should be tested if not explicitly excluded"* as an open concern, confirming the team is aware of the gap and that no on-chain guard exists. [9](#0-8) 

**`SECURITY.md` contains no explicit exclusion for rebasing or non-standard tokens:**

The exclusion list covers oracle failures, governance attacks, liquidity impacts, Sybil attacks, and centralization risks. Rebasing tokens are not listed as out of scope. [10](#0-9) 

**Exploit flow:**
1. Attacker deploys `NegativeRebaseToken` — a standard ERC20 with an owner-callable `rebase(uint256 pct)` that multiplies every balance by `(WAD - pct) / WAD`.
2. Attacker calls `midnight.touchMarket(market)` with `market.loanToken = address(NegativeRebaseToken)`. No check blocks this.
3. Attacker acts as both lender (buy-offer maker) and borrower (sell-offer taker) in the same market to self-fund state. After settlement, borrower has `debt = D`, lender has `credit = D`.
4. Borrower calls `repay(market, D, ...)`. At line 509, `withdrawable += D`. At line 520, `safeTransferFrom` pulls `D` tokens. Contract balance = `D`. Invariant holds.
5. Settlement fee spread from `take` means `claimableSettlementFee[loanToken] = F > 0`.
6. Attacker calls `NegativeRebaseToken.rebase(X)`. Every balance, including `balanceOf(midnight)`, drops by `X%`. Contract balance becomes `D * (1 - X/100)`.
7. `withdrawable + claimableSettlementFee = D + F > D * (1 - X/100) = balanceOf(midnight, loanToken)`. Solvency invariant violated.
8. The first lender to call `withdraw` drains the remaining balance; subsequent lenders and the fee claimer receive zero.

**Existing checks are insufficient:** `SECURITY.md` contains no explicit exclusion for rebasing or non-standard tokens. The formal verification assumption in `Solvency.spec` is a prover-level constraint, not an on-chain guard. `touchMarket` enforces no token allowlist or interface check.

## Impact Explanation

The solvency invariant `balanceOf(midnight, loanToken) >= withdrawableSum + claimableSettlementFee` is concretely and permanently violated. Lenders whose `credit` maps to `withdrawable` units and the fee claimer whose `claimableSettlementFee` is non-zero cannot fully redeem their entitlements. The shortfall is permanent and proportional to the rebase magnitude and the total outstanding withdrawable plus fee balance. This constitutes direct, irreversible loss of funds for affected lenders and fee claimers.

## Likelihood Explanation

**Preconditions:**
- Attacker deploys a negative-rebasing ERC20 (~50 lines of Solidity, trivial).
- Attacker calls `touchMarket` (permissionless, gas cost only).
- At least one repayment or settlement-fee-generating `take` must have occurred. The attacker can self-fund this state by acting as both lender and borrower, or can attract third-party users to a market with a token that superficially resembles a legitimate asset.
- Attacker triggers `rebase()` (one transaction).

The attack requires no privileged access, no oracle manipulation, and no victim cooperation to establish the vulnerable state (attacker can self-fund). Harm to third parties requires attracting users to the malicious market, which is feasible in a permissionless protocol where any market can be created. The attack is repeatable across any number of markets.

## Recommendation

1. **Add a balance-delta check in `safeTransferFrom`**: Record `balanceOf(address(this))` before and after the call and require the delta equals `value`.
2. **Alternatively, add a token allowlist or interface check in `touchMarket`**: Restrict `market.loanToken` to a protocol-maintained allowlist of vetted tokens, or require the token to pass a standard interface check.
3. **Document the restriction on-chain**: If rebasing tokens are intended to be unsupported, enforce this at the `touchMarket` level rather than relying on documentation-level NatSpec assumptions.

```solidity
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../src/Midnight.sol";

contract NegativeRebaseToken is ERC20 {
    address public owner;
    constructor() ERC20("NRT", "NRT") { owner = msg.sender; }
    function mint(address to, uint256 amount) external { _mint(to, amount); }
    function rebase(uint256 pct) external {
        // Reduce every balance by pct/WAD
        // Implementation: iterate known holders or use a scalar
        // Simplified: burn pct% from midnight's balance directly
        require(msg.sender == owner);
        uint256 midnightBalance = balanceOf(address(midnight));
        _burn(address(midnight), midnightBalance * pct / 1e18);
    }
}

contract RebaseExploit is Test {
    Midnight midnight;
    NegativeRebaseToken token;

    function testExploit() public {
        // 1. Deploy malicious token
        token = new NegativeRebaseToken();

        // 2. Create market with malicious loan token (permissionless)
        Market memory market = /* valid market params with loanToken = address(token) */;
        midnight.touchMarket(market);

        // 3. Self-fund: act as lender and borrower
        // ... (take offer, repay D units)
        // After repay: withdrawable = D, balanceOf(midnight) = D

        // 4. Trigger rebase: reduce midnight's balance by 50%
        token.rebase(0.5e18);

        // 5. Invariant violated: withdrawable = D > balanceOf(midnight) = D/2
        assert(midnight.withdrawable(id) > token.balanceOf(address(midnight)));

        // 6. First lender drains remaining balance; second lender gets zero
    }
}
```

**Minimal manual steps:**
1. Deploy `NegativeRebaseToken` with owner-callable `rebase`.
2. Call `midnight.touchMarket(market)` with `loanToken = address(NegativeRebaseToken)`.
3. As lender, post a buy offer. As borrower, take it and repay `D` units.
4. Verify `midnight.withdrawable(id) == D` and `token.balanceOf(address(midnight)) == D`.
5. Call `token.rebase(0.5e18)`.
6. Verify `token.balanceOf(address(midnight)) == D/2` while `midnight.withdrawable(id) == D`.
7. Call `midnight.withdraw(market, D, lender, lender)` — reverts or drains all remaining balance, leaving subsequent claimants with zero.

```text
    function touchMarket(Market memory market) public returns (bytes32) {
        bytes32 id = toId(market);
        if (marketState[id].tickSpacing == 0) {
            require(market.maturity <= block.timestamp + 100 * 365 days, MaturityTooFar());
            require(market.collateralParams.length > 0, NoCollateralParams());
            require(market.collateralParams.length <= MAX_COLLATERALS, TooManyCollateralParams());
            address previousCollateralToken;
            for (uint256 i = 0; i < market.collateralParams.length; i++) {
                address collateralToken = market.collateralParams[i].token;
                require(collateralToken > previousCollateralToken, CollateralParamsNotSorted());

                require(isLltvAllowed(lltv), LltvNotAllowed());
```
