### Title
Malicious oracle in permissionlessly-created market permanently blocks liquidation of all borrowers — (`src/Midnight.sol`)

### Summary

`touchMarket` performs no validation on `CollateralParams.oracle`, allowing any caller to register a market whose oracle is a contract that always reverts on `price()`. The `liquidate()` function calls `IOracle(_collateralParam.oracle).price()` inside an unconditional while loop over every bit in `_position.collateralBitmap` with no try/catch, so a single reverting oracle in that bitmap causes every liquidation attempt on every borrower in that market to revert permanently. This directly violates the core invariant that unhealthy positions remain liquidatable.

### Finding Description

**Root cause — no oracle validation in `touchMarket`:**

`touchMarket` validates `collateralToken`, `lltv`, and `maxLif` for each `CollateralParams` entry, but performs zero validation on the `oracle` field: [1](#0-0) 

Any address — including a contract whose `price()` unconditionally reverts — is accepted.

**Unconditional oracle call in `liquidate()` with no error handling:**

Inside `liquidate()`, the collateral health loop at lines 607–618 calls `IOracle(_collateralParam.oracle).price()` for every bit set in `_position.collateralBitmap`: [2](#0-1) 

There is no `try/catch`, no fallback, and no skip path. If any oracle in the bitmap reverts, the entire `liquidate()` call reverts. The protocol's own inline comment explicitly acknowledges this: [3](#0-2) 

**`supplyCollateral` does not call the oracle**, so a borrower can activate a collateral slot whose oracle already reverts — the protocol's test suite explicitly verifies this: [4](#0-3) 

**Exploit flow:**

1. Attacker deploys `RevertOracle` (a contract whose `price()` always reverts).
2. Attacker calls `touchMarket` with `CollateralParams[i].oracle = address(RevertOracle)`. No check stops this.
3. Victim borrower calls `supplyCollateral(market, i, assets, borrower)` — oracle is not called, succeeds, sets bit `i` in `_position.collateralBitmap`.
4. Victim takes a sell offer; debt increases.
5. Price drops (or maturity passes); position becomes unhealthy.
6. Any liquidator calls `liquidate()` → the while loop reaches bit `i` → `IOracle(RevertOracle).price()` reverts → entire transaction reverts.
7. No liquidation of this borrower is ever possible again.

**Why existing checks are insufficient:**

- `touchMarket` checks `token != address(0)` (implicitly via the sort requirement `collateralToken > previousCollateralToken` starting from `address(0)`), `lltv`, and `maxLif` — but never touches `oracle`.
- `liquidate` has no `try/catch` and no circuit-breaker to skip a reverting oracle.
- The `liquidatorGate` check (line 598–600) is irrelevant; the revert happens before it matters.
- Post-maturity mode (`postMaturityMode = true`) still executes the same oracle loop before the `NotLiquidatable` check, so it provides no escape. [5](#0-4) 

### Impact Explanation

Every borrower in the attacker-created market who has activated collateral slot `i` (bit `i` set in `collateralBitmap`) has their position permanently frozen: no liquidator can ever seize their collateral or reduce their debt, regardless of how undercollateralized they become. Bad debt accumulates without resolution, lenders cannot recover funds through liquidation, and the `lossFactor` accounting that depends on liquidation-triggered bad-debt write-downs is never triggered. The invariant "unhealthy positions remain liquidatable" is permanently broken for all affected borrowers.

### Likelihood Explanation

**Preconditions:** (1) attacker deploys one contract; (2) attacker calls `touchMarket` once — both are permissionless and cost only gas. **Victim precondition:** at least one borrower supplies collateral to the attacker's market and takes debt. This is realistic because the market can appear legitimate (valid `lltv`, valid `maxLif`, real collateral token) and the oracle can return a valid price initially (e.g., using a `RevertingOracle`-style contract with a `stopOracle()` toggle), only reverting after borrowers are locked in. The attack is repeatable across any number of markets and borrowers at negligible ongoing cost.

### Recommendation

Two complementary fixes:

1. **Validate the oracle at market creation:** In `touchMarket`, call `IOracle(market.collateralParams[i].oracle).price()` and require it returns a non-zero value. This prevents markets with broken oracles from being registered.

2. **Defensive oracle call in `liquidate()`:** Wrap the `IOracle(...).price()` call in a `try/catch`. On revert, either skip that collateral (treating its contribution as zero, which is conservative for the liquidator) or revert with a dedicated error that signals the oracle is broken, allowing governance to intervene. The former preserves liveness; the latter at least gives a clear signal.

Option 1 alone is insufficient if the oracle can be toggled post-deployment (as `RevertingOracle.stopOracle()` demonstrates); option 2 alone changes the health accounting semantics. Together they provide defense in depth.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import {Test} from "forge-std/Test.sol";
import {Midnight} from "src/Midnight.sol";
import {IMidnight, Market, CollateralParams} from "src/interfaces/IMidnight.sol";
import {IOracle} from "src/interfaces/IOracle.sol";

contract AlwaysRevertOracle {
    function price() external pure returns (uint256) { revert("no price"); }
}

contract MaliciousOracleLiquidationDoSTest is Test {
    Midnight midnight;
    Market market;

    function setUp() public {
        // deploy protocol (use existing BaseTest setup or deploy fresh)
        midnight = new Midnight();
        AlwaysRevertOracle badOracle = new AlwaysRevertOracle();

        CollateralParams[] memory cp = new CollateralParams[](1);
        cp[0] = CollateralParams({
            token: address(someERC20),       // any valid token address
            lltv: 0.77e18,
            maxLif: midnight.maxLif(0.77e18, 0.25e18),
            oracle: address(badOracle)       // malicious oracle
        });
        market.loanToken = address(loanToken);
        market.maturity = block.timestamp + 100;
        market.collateralParams = cp;

        // Step 1+2: attacker creates market — must succeed
        midnight.touchMarket(market);
    }

    function testLiquidationPermanentlyDosedByRevertingOracle() public {
        // Step 3: victim supplies collateral (oracle NOT called — succeeds)
        deal(cp[0].token, borrower, 1e18);
        vm.prank(borrower);
        midnight.supplyCollateral(market, 0, 1e18, borrower);

        // Step 4: victim takes debt (setup via offer/take)
        // ... (standard take setup) ...

        // Step 5: position is unhealthy (price dropped / maturity passed)

        // Step 6: liquidation MUST revert
        vm.expectRevert();
        midnight.liquidate(market, 0, 0, 1, borrower, false, address(this), address(0), "");

        // Also fails in post-maturity mode
        vm.warp(market.maturity + 1);
        vm.expectRevert();
        midnight.liquidate(market, 0, 0, 1, borrower, true, address(this), address(0), "");

        // Assert: borrower debt is unchanged — position permanently frozen
        assertGt(midnight.debtOf(toId(market), borrower), 0);
    }
}
```

**Expected assertions:** both `liquidate` calls revert; borrower debt remains non-zero; no collateral is ever seized. The test confirms the invariant "unhealthy positions remain liquidatable" is broken.
