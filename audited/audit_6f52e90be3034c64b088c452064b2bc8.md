### Title
Cross-chain replay attack via chainid-free deterministic salt in `createPool` - (File: src/ConfidencePoolFactory.sol)

### Summary
`createPool` computes the CREATE2 salt without `block.chainid`. If the factory and pool implementation are deployed at the same addresses on a second chain (standard practice with deterministic deployers), an attacker can pre-deploy a pool at the identical address with a malicious `recoveryAddress`, draining stakers who interact with the wrong chain.

### Finding Description
The salt used in `Clones.cloneDeterministic` is:

```solidity
bytes32 salt = keccak256(abi.encode(agreement, _poolsByAgreement[agreement].length));
pool = Clones.cloneDeterministic(poolImplementation, salt);
``` [1](#0-0) 

No `block.chainid` is included. The CREATE2 pool address is fully determined by `factory_address ++ salt ++ keccak256(clone_bytecode)`. The clone bytecode embeds `poolImplementation`, so if the factory and implementation are deployed at the same addresses on two chains (achievable with any deterministic deployer such as the canonical CREATE2 factory at `0x4e59b44847b379578588920cA78FbF26c0B4956C`), and the agreement contract lands at the same address on both chains, the resulting pool address is identical across chains.

An attacker who controls the agreement on the second chain can call `createPool` there with the same `agreement` address and the same pool-count index, satisfying both guards:

```solidity
if (!safeHarborRegistry.isAgreementValid(agreement)) revert InvalidAgreement();
if (IAgreement(agreement).owner() != msg.sender) revert UnauthorizedCreator();
``` [2](#0-1) 

but supplying an attacker-controlled `recoveryAddress`. The pool is initialized at the same address as the legitimate pool on the primary chain.

### Impact Explanation
Stakers who accidentally connect to the wrong chain (or are phished into doing so) deposit into a pool that looks identical on-chain but routes all CORRUPTED-path funds — and, under bad-faith resolution, the entire stake + bonus — to the attacker's `recoveryAddress` via `claimCorrupted`. [3](#0-2) 

Principal and bonus are fully at risk; there is no recovery path once the outcome is flagged and `claimsStarted` is set.

### Likelihood Explanation
BattleChain is currently single-chain, which limits immediate exposure. However, deterministic deployment is standard practice and BattleChain is an EVM-compatible L2, making same-address deployment on a fork or a parallel EVM chain straightforward. The judge in the analogous Stakehouse finding explicitly noted that a single-chain protocol can fork or expand, and the same reasoning applies here. The attacker's only prerequisite is owning an agreement at the same address on the target chain, which is achievable by front-running agreement creation on that chain before the legitimate sponsor does.

### Recommendation
Include `block.chainid` in the salt so pool addresses are chain-specific:

```solidity
bytes32 salt = keccak256(abi.encode(block.chainid, agreement, _poolsByAgreement[agreement].length));
```

This ensures a pool created on BattleChain mainnet cannot be replayed at the same address on any fork or parallel deployment.

### Proof of Concept
1. Factory deployed at address `F` and pool implementation at address `I` on BattleChain mainnet and on a second EVM chain, both via a deterministic deployer (same addresses on both).
2. Agreement contract `A` is deployed at the same address on both chains; attacker owns `A` on the second chain.
3. Sponsor calls `createPool(A, stakeToken, expiry, minStake, sponsor_recovery, accounts)` on mainnet. Salt = `keccak256(abi.encode(A, 0))` → pool deployed at address `P`.
4. Attacker calls `createPool(A, stakeToken, expiry, minStake, attacker_address, accounts)` on the second chain with the same `A` and pool count 0. Identical salt and identical bytecode hash → pool also deploys at address `P` on the second chain, but with `recoveryAddress = attacker_address`.
5. Stakers, deceived by the identical address, stake tokens into `P` on the second chain.
6. Attacker (as moderator on the second chain, or after expiry via `claimExpired` auto-CORRUPTED) triggers resolution. `claimCorrupted` sweeps the full balance to `attacker_address`. [4](#0-3) [5](#0-4)
