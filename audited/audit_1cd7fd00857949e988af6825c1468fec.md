### Title
Fee-on-Transfer Loan Token Causes Lenders to Receive Less Than Credit Entitlement on Withdraw - (File: src/Midnight.sol)

### Summary
The `withdraw` function in `src/Midnight.sol` commits all state decrements (`credit`, `withdrawable`, `totalUnits`) for the full `units` amount before calling `SafeTransferLib.safeTransfer`, which only verifies the call succeeded and returned `true` — it does not verify the receiver's actual balance delta. If the market's `loanToken` is a fee-on-transfer token, the receiver receives `units * (1 - fee_rate)` while the protocol's accounting is reduced by the full `units`, causing a permanent, per-withdrawal loss for every lender.

`SafeTransferLib.safeTransfer` in `src/libraries/SafeTransferLib.sol` lines 12–22 only checks `success` and the boolean return value of `transfer`; it performs no balance-before/balance-after check and cannot detect that the receiver received fewer tokens than `units`.

**Market creation has no loan token restriction:**

`touchMarket` in `src/Midnight.sol` lines 755–791 validates collateral params, maturity, LLTV, and maxLif, but imposes zero constraints on `market.loanToken`. Any ERC20 — including fee-on-transfer tokens — is accepted. `touchMarket` is permissionless; any user can create a market.

**Compound accounting break across the full lifecycle:**

- On `repay` (line 520): `safeTransferFrom(market.loanToken, payer, address(this), units)` — the contract receives `units * (1 - fee_rate)` but credits the full `units` to `withdrawable`. The contract is immediately underfunded.
- On `withdraw` (line 499): `safeTransfer(market.loanToken, receiver, units)` — the contract's balance decreases by `units` but the receiver only receives `units * (1 - fee_rate)`. The fee is silently absorbed by the token contract.

**Exploit flow:**

1. Any user calls `touchMarket` with a fee-on-transfer token as `loanToken`.
2. A lender lends N units via `take`; the contract receives `buyerAssets * (1 - fee_rate)` but credits `buyerAssets` to the lender's position.
3. A borrower repays N units; the contract receives `N * (1 - fee_rate)` but credits N to `withdrawable`.
4. The lender calls `withdraw(market, N, lender, receiver)`:
   - `credit`, `withdrawable`, `totalUnits` all decrease by N (full amount).
   - `safeTransfer` succeeds (token returns `true`).
   - Receiver's balance increases by only `N * (1 - fee_rate)`.
5. No revert, no check, no recovery path.

**Why existing checks fail:**

`SafeTransferLib.safeTransfer` only checks `success` and the ABI-decoded boolean return — it cannot detect fee-on-transfer shortfalls. [1](#0-0)  There is no balance-before/after guard anywhere in `withdraw`. [2](#0-1)  `touchMarket` has no loan token allowlist or fee-on-transfer check. [3](#0-2) 

### Impact Explanation

Every lender withdrawal in a fee-on-transfer loan token market results in the receiver receiving `units * (1 - fee_rate)` tokens instead of `units`. The credit accounting is permanently reduced by the full `units`, so the lender cannot re-withdraw the shortfall. The loss is proportional to the fee rate and repeats on every withdrawal, draining lender value systematically. The contract's actual token balance also diverges from its tracked `withdrawable` state, eventually causing later withdrawals to revert due to insufficient balance, freezing remaining lenders' funds.

### Likelihood Explanation

`touchMarket` is permissionless — any user can create a market with any ERC20 as the loan token. Fee-on-transfer tokens (e.g., tokens with deflationary mechanics, tokens with protocol fees) exist in production. A lender who participates in such a market — whether knowingly or not — will suffer the loss on every `withdraw` call. The condition is fully attacker-controllable (market creation) and repeatable on every withdrawal.

### Recommendation

Add a balance-before/after check in `withdraw` (and symmetrically in `repay`, `take`, `liquidate`) to verify the actual amount received/sent matches `units`. For example, in `withdraw`:

```solidity
uint256 balanceBefore = IERC20(market.loanToken).balanceOf(receiver);
SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
require(IERC20(market.loanToken).balanceOf(receiver) - balanceBefore == units, FeeOnTransferToken());
```

Alternatively, maintain an explicit allowlist of approved loan tokens and reject fee-on-transfer tokens at market creation time in `touchMarket`.

contract FeeOnTransferToken {
    mapping(address => uint256) public balanceOf;
    mapping(address => mapping(address => uint256)) public allowance;
    uint256 public constant FEE_RATE = 10; // 10%

    function mint(address to, uint256 amount) external { balanceOf[to] += amount; }
    function approve(address spender, uint256 amount) external returns (bool) {
        allowance[msg.sender][spender] = amount; return true;
    }
    function transfer(address to, uint256 amount) external returns (bool) {
        uint256 fee = amount * FEE_RATE / 100;
        balanceOf[msg.sender] -= amount;
        balanceOf[to] += amount - fee;
        return true;
    }
    function transferFrom(address from, address to, uint256 amount) external returns (bool) {
        allowance[from][msg.sender] -= amount;
        uint256 fee = amount * FEE_RATE / 100;
        balanceOf[from] -= amount;
        balanceOf[to] += amount - fee;
        return true;
    }
}

    function safeTransfer(address token, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transfer, (to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
            }
        }
        require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
    }
