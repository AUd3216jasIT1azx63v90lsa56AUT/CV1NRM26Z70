### Title
`sellerAssetsToUnits` division-by-zero when `settlementFee == offerPrice` on a buy offer, blocking all bundler sell flows for affected offers - (`File: src/periphery/TakeAmountsLib.sol`)

### Summary
`TakeAmountsLib.sellerAssetsToUnits` computes `sellerPrice = offerPrice - settlementFee` for buy offers and immediately uses it as the denominator in `mulDivUp(targetSellerAssets, WAD, sellerPrice)` with no guard against `sellerPrice == 0`. When the feeSetter legitimately sets the index-6 settlement fee to `MAX_SETTLEMENT_FEE_360_DAYS = 0.005e18` and a buy offer exists at the tick whose price equals exactly `0.005e18`, `sellerPrice` becomes zero and every call to `sellerAssetsToUnits` for that offer panics. Because this call in `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` sits outside the `try/catch` that wraps `IMidnight.take`, the panic propagates and reverts the entire bundler transaction for any user who attempts to sell against that offer through the bundler.

### Finding Description
**Code path:**

`src/periphery/TakeAmountsLib.sol`, `sellerAssetsToUnits`, lines 41–46:
```solidity
uint256 offerPrice = TickLib.tickToPrice(offer.tick);
uint256 settlementFee =
    IMidnight(midnight).settlementFee(id, UtilsLib.zeroFloorSub(offer.market.maturity, block.timestamp));
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
return
    offer.buy ? targetSellerAssets.mulDivUp(WAD, sellerPrice) : targetSellerAssets.mulDivDown(WAD, sellerPrice);
```

**Root cause:** No `require(sellerPrice > 0)` guard before the division. The sibling function `buyerAssetsToUnits` has an analogous guard (`require(buyerPrice <= WAD)`), but `sellerAssetsToUnits` has none.

**Why `sellerPrice == 0` is reachable:**

- `MAX_TICK = 5820`, `tickToPrice(MAX_TICK) = 1e18`. The price function is monotonically increasing with `PRICE_ROUNDING_STEP = 1e12`. Since `0.005e18 = 5e15` is a multiple of `1e12`, there exists a tick `t*` where `tickToPrice(t*) = 0.005e18` exactly.
- `MAX_SETTLEMENT_FEE_360_DAYS = 0.005e18`. `setMarketSettlementFee` enforces `newSettlementFee <= maxSettlementFee(index)`, so setting index 6 to `0.005e18` passes the `<=` check and is a valid feeSetter action.
- For a market with TTM ≥ 360 days, `settlementFee(id, ttm) = settlementFeeCbp6 * CBP = 0.005e18`.
- Therefore `sellerPrice = 0.005e18 − 0.005e18 = 0`.

**Why `Midnight.take()` does NOT protect against this:** `Midnight.take()` computes `sellerAssets = units.mulDivDown(0, WAD) = 0` — no revert. The NatDoc comment "Taking buy offers with price < settlement fee will revert" is accurate only for strict less-than (arithmetic underflow); the equal-to case silently produces `sellerAssets = 0` and succeeds.

**Why the bundler `try/catch` does NOT protect against this:** In `MidnightBundles.supplyCollateralAndSellWithAssetsTarget` (lines 285–300), `TakeAmountsLib.sellerAssetsToUnits` is called at line 286, *before* the `try { IMidnight(MIDNIGHT).take(...) } catch {}` block at line 292. The panic from `mulDivUp(..., 0)` is thrown at line 286 and propagates directly to the caller, reverting the entire bundler call.

**Exploit flow:**
1. feeSetter calls `setMarketSettlementFee(id, 6, 0.005e18)` — valid, within bounds.
2. A buy offer exists at tick `t*` where `tickToPrice(t*) = 0.005e18`, with TTM ≥ 360 days.
3. Any unprivileged user calls `supplyCollateralAndSellWithAssetsTarget(...)` including that offer.
4. `sellerAssetsToUnits` is called → `sellerPrice = 0` → `mulDivUp(target, WAD, 0)` → Solidity 0.8 arithmetic panic → entire bundler call reverts.

### Impact Explanation
Every call to `supplyCollateralAndSellWithAssetsTarget` that includes the affected buy offer reverts unconditionally. Users cannot use the bundler to sell against that offer. Because `ConsumableUnitsLib.consumableUnits` also calls `sellerAssetsToUnits` for buy offers with `maxAssets > 0`, any bundler path that routes through `consumableUnits` for such an offer is equally blocked. Direct `Midnight.take()` still works (sellerAssets = 0), but the bundler sell path — the primary interface for asset-targeted selling — is frozen for all affected offers.

### Likelihood Explanation
The feeSetter setting the 360-day fee to its maximum (`0.005e18`) is a routine governance action with no indication it is unsafe. The tick whose price equals exactly `0.005e18` is a valid, accessible tick (price is a multiple of `PRICE_ROUNDING_STEP`). Any maker who placed a buy offer at that tick before or after the fee change is affected. The condition is deterministic and repeatable: every call to the bundler with that offer will panic until the fee is lowered or the offer expires.

### Recommendation
Add a guard in `sellerAssetsToUnits` mirroring the existing guard in `buyerAssetsToUnits`:

```solidity
// After computing sellerPrice:
uint256 sellerPrice = offer.buy ? offerPrice - settlementFee : offerPrice;
require(sellerPrice > 0, /* e.g. SellerPriceIsZero() */);
```

This makes the revert explicit and consistent with the NatDoc claim "Reverts if offerPrice < settlementFee in case of a buy offer", extending it to cover the equal-to case. Alternatively, the `Midnight.take()` core could enforce `sellerPrice > 0` for buy offers, which would make the bundler's `try/catch` absorb the revert naturally.

### Proof of Concept
```solidity
// Foundry unit test
function testSellerAssetsToUnitsZeroDivision() public {
    // Find tick t* where tickToPrice(t*) == 0.005e18
    // (binary search or known value; 0.005e18 = 5e15 is a valid PRICE_ROUNDING_STEP multiple)
    uint256 targetPrice = 0.005e18;
    uint256 tick = TickLib.priceToTick(targetPrice, 1);
    assertEq(TickLib.tickToPrice(tick), targetPrice, "exact price tick");

    // Create market with TTM >= 360 days
    market.maturity = block.timestamp + 361 days;
    bytes32 id = midnight.touchMarket(market);

    // feeSetter sets index-6 fee to MAX_SETTLEMENT_FEE_360_DAYS
    midnight.setMarketSettlementFee(id, 6, 0.005e18); // valid: equals max

    // Construct buy offer at tick t*
    Offer memory offer;
    offer.buy = true;
    offer.tick = tick;
    offer.market = market;
    // ... fill other fields ...

    // Assert sellerAssetsToUnits panics (division by zero)
    vm.expectRevert(stdError.divisionError);
    TakeAmountsLib.sellerAssetsToUnits(address(midnight), id, offer, 1e18);
}
```

Expected assertion: `vm.expectRevert(stdError.divisionError)` passes, confirming the panic. A follow-up test should confirm `Midnight.take()` itself does NOT revert with the same setup (sellerAssets = 0), proving the divergence between core and periphery behavior.
