Audit Report

## Title
Fee-on-Transfer Token Flash Loan Drains Protocol Reserves Without Balance Invariant Check - (File: src/Midnight.sol)

## Summary
The `flashLoan` function transfers `assets[i]` out to the callback and pulls `assets[i]` back with no pre/post balance snapshot on the protocol's own holdings. When the token is fee-on-transfer, each leg silently deducts a fee, so the protocol receives `assets[i]*(1-f)` on repayment while having disbursed `assets[i]`, producing a net loss of `assets[i]*f` per call. The only formal exclusion of this token class is a Certora spec-level assumption at line 31 of `Solvency.spec`, not an on-chain guard.

## Finding Description
**Exact code path — `flashLoan` lines 737–752:**

Outbound leg: `SafeTransferLib.safeTransfer(tokens[i], callback, assets[i])` at line 743. With fee rate `f`, the protocol's balance decreases by `assets[i]` while the callback receives only `assets[i]*(1-f)`. [1](#0-0) 

Inbound leg: `SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i])` at line 750. The call requests `assets[i]` from the callback; the fee is deducted in transit, so the protocol receives only `assets[i]*(1-f)`. [2](#0-1) 

**Root cause:** There is no `balanceOf(address(this))` snapshot before the outbound transfer compared against a snapshot after the inbound transfer. A grep across `src/Midnight.sol` confirms zero occurrences of any balance check in or around `flashLoan`. The only formal exclusion is the Certora `Solvency.spec` assumption at line 31:

> "Assume ERC20 tokens transfer correctly: no fee taking from sender or receiver, no rebasing, no blacklisting, no transfer limits." [3](#0-2) 

This is a prover-level modeling assumption (`CVL_transferFrom` models lossless transfers), not an on-chain enforcement. The `tokenBalanceCorrect` strong invariant proven in the spec is therefore unsound for fee-on-transfer tokens in production. [4](#0-3) 

**Why existing checks fail:** The only post-inbound check is that `safeTransferFrom` does not revert. It succeeds as long as the callback holds `assets[i]` tokens (satisfied by pre-funding with `assets[i]*f`). There is no `require(balanceAfter >= balanceBefore)` guard anywhere in `flashLoan`. [5](#0-4) 

## Impact Explanation
The protocol's token balance for any fee-on-transfer loan token decreases by `assets*f` per flash loan call. Repeated calls drain the protocol's reserves for that token, directly violating the core solvency invariant that contract balances must cover collateral, withdrawable, and claimable settlement fee amounts. Lenders and borrowers whose positions are denominated in that token face insolvency as backing reserves are eroded. This constitutes direct, concrete loss of protocol-held assets. [6](#0-5) 

## Likelihood Explanation
**Preconditions:**
- A fee-on-transfer token must be used as a market's loan token (USDT has a dormant fee-on-transfer switch; many DeFi-native tokens carry transfer fees by design).
- The attacker must pre-fund their callback with `assets*f` tokens per iteration.

**Feasibility:** `flashLoan` is callable by any unprivileged address with no token allowlist, no role gate, and no market-membership check on the supplied token addresses. [7](#0-6) 

**Cost model:** This is a 1:1 griefing attack — the attacker spends `assets*f` tokens per iteration and the protocol loses the same amount. It is not directly profitable, but it is a credible griefing/DoS vector against any market whose loan token carries a transfer fee. Neither `SECURITY.md` nor `RESEARCHER.md` excludes fee-on-transfer tokens or griefing attacks from scope. [8](#0-7) 

## Recommendation
Add a pre/post balance invariant check inside `flashLoan`:

```solidity
uint256[] memory balancesBefore = new uint256[](tokens.length);
for (uint256 i = 0; i < tokens.length; i++) {
    balancesBefore[i] = IERC20(tokens[i]).balanceOf(address(this));
    SafeTransferLib.safeTransfer(tokens[i], callback, assets[i]);
}
// ... callback ...
for (uint256 i = 0; i < tokens.length; i++) {
    SafeTransferLib.safeTransferFrom(tokens[i], callback, address(this), assets[i]);
    require(
        IERC20(tokens[i]).balanceOf(address(this)) >= balancesBefore[i],
        InsufficientRepayment()
    );
}
```

This ensures the protocol's balance is at least restored to its pre-loan level regardless of token transfer mechanics, and is the standard pattern used by Morpho Blue and Aave v3 flash loan implementations. [5](#0-4) 

## Proof of Concept
1. Deploy a mock ERC20 with a 1% fee-on-transfer (both send and receive legs deduct 1%).
2. Create a market using this token as the loan token; deposit 10,000 tokens into the protocol.
3. Deploy an attacker callback contract pre-funded with 100 tokens (= `assets * f` for `assets = 10,000`, `f = 0.01`).
4. Call `midnight.flashLoan([feeToken], [10_000e18], attackerCallback, "")`.
   - Protocol sends 10,000; callback receives 9,900.
   - Callback tops up to 10,000 using its 100 pre-funded tokens and calls `approve`.
   - Protocol pulls 10,000; receives 9,900 after fee.
   - Net: protocol balance −100, attacker balance −100.
5. Repeat until protocol reserves for that token are exhausted.
6. Assert `feeToken.balanceOf(address(midnight)) < initialBalance` after each iteration.

A Foundry fuzz test parameterizing `assets` and `feeRate` over `[1, 1e6]` and `[1, 100]` bps respectively will confirm the invariant break on every non-zero fee rate. [9](#0-8)
