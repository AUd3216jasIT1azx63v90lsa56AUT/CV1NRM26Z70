### Title
Fee-on-Transfer Loan Token Inflates `claimableSettlementFee` Without Actual Balance Coverage - (`src/Midnight.sol`)

### Summary

In `take`, `claimableSettlementFee[offer.market.loanToken]` is incremented by `buyerAssets - sellerAssets` at line 418 before the inbound transfer at line 455. `SafeTransferLib.safeTransferFrom` does not verify the actual amount received. When the loan token is a fee-on-transfer token, the protocol's real balance increases by less than the credited amount, permanently inflating `claimableSettlementFee` above the actual token balance and creating a solvency gap that can drain lender-owned `withdrawable` funds when the fee claimer calls `claimSettlementFee`.

### Finding Description

**Exact code path** (`src/Midnight.sol`):

- Line 418: `claimableSettlementFee[offer.market.loanToken] += buyerAssets - sellerAssets;` — accounting updated unconditionally before any transfer.
- Line 420–422: when `offer.buy == true` and `offer.callback != address(0)`, `buyerCallback = offer.callback` and `payer = buyerCallback`.
- Line 455: `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, address(this), buyerAssets - sellerAssets);` — protocol receives the fee portion.
- Line 456: `SafeTransferLib.safeTransferFrom(offer.market.loanToken, payer, receiver, sellerAssets);` — seller assets go directly to `receiver`, bypassing the protocol.

`SafeTransferLib.safeTransferFrom` (`src/libraries/SafeTransferLib.sol` lines 24–34) only checks that the call did not revert and returned `true`. It performs no before/after balance comparison. There is no fee-on-transfer guard anywhere in `touchMarket` or `take`.

**Exploit flow:**

1. Attacker deploys a fee-on-transfer ERC20 (e.g., 1% fee on every transfer) and uses it as `loanToken`.
2. Attacker calls `touchMarket` (permissionless) to create a market with this token.
3. Attacker (as maker) creates a buy offer with `offer.buy = true` and `offer.callback = attackerBuyCallback`.
4. A second attacker-controlled address acts as taker and calls `take`.
5. At line 418, `claimableSettlementFee[feeToken] += buyerAssets - sellerAssets` (e.g., 100 units).
6. At line 455, `safeTransferFrom(feeToken, payer, address(this), 100)` executes; the token deducts 1% fee, so the protocol receives only 99 units.
7. `claimableSettlementFee[feeToken]` now reads 100 but the actual balance increase is 99. Shortfall = 1 unit per take, accumulates across repeated takes.
8. When `claimSettlementFee(feeToken, 100, receiver)` is called (line 305–310), the protocol transfers 100 units out. If the contract's balance of `feeToken` is exactly 99 (from this take alone), the transfer reverts. If other lenders have deposited `feeToken` via `repay` (increasing `withdrawable`), the fee claimer drains their funds.

**Why existing checks fail:**

- `touchMarket` validates LLTV, maxLif, collateral sorting, and maturity — no token type validation.
- The Certora `Solvency.spec` `tokenBalanceCorrect` invariant explicitly assumes "no fee taking from sender or receiver" (line 31–33 of `Solvency.spec`), so the formal proof does not cover this case.
- The `pendingFeeReceipt` ghost in `Solvency.spec` (lines 142–151) tracks the gap between the `claimableSettlementFee` increment and the inbound transfer, but the CVL model assumes the transfer delivers exactly the requested amount — it does not model fee-on-transfer deductions.

### Impact Explanation

`claimableSettlementFee[loanToken]` exceeds the protocol's actual token balance by `(buyerAssets - sellerAssets) * feeRate` per take. Repeated takes compound the shortfall. When `claimSettlementFee` is called for the full credited amount, the protocol either reverts (DoS on fee collection) or, if the same token is used across multiple markets with lender `withdrawable` balances, the fee claimer drains lender-redeemable assets — direct protocol insolvency matching the scoped impact.

### Likelihood Explanation

Preconditions: (1) loan token is fee-on-transfer; (2) `offer.buy = true`; (3) `offer.callback != address(0)`. Market creation is permissionless and accepts any `loanToken`. The attacker controls all three preconditions. The attack is repeatable on every `take` call and requires no privileged access. Legitimate fee-on-transfer tokens (e.g., PAXG) used as loan tokens by honest market creators also trigger this path without any attacker involvement.

### Recommendation

After each inbound `safeTransferFrom` to `address(this)`, verify the actual received amount using a before/after balance check:

```solidity
uint256 balanceBefore = IERC20(loanToken).balanceOf(address(this));
SafeTransferLib.safeTransferFrom(loanToken, payer, address(this), buyerAssets - sellerAssets);
require(IERC20(loanToken).balanceOf(address(this)) - balanceBefore == buyerAssets - sellerAssets, FeeOnTransferToken());
```

Apply the same pattern to `repay` (line 520), `supplyCollateral` (line 545), `liquidate` (line 717), and `flashLoan` (line 750). Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported as loan or collateral tokens, and add a token validation hook in `touchMarket`.

    function safeTransferFrom(address token, address from, address to, uint256 value) internal {
        require(token.code.length > 0, NoCode());

        (bool success, bytes memory returndata) = token.call(abi.encodeCall(IERC20.transferFrom, (from, to, value)));
        if (!success) {
            assembly ("memory-safe") {
                revert(add(returndata, 0x20), mload(returndata))
