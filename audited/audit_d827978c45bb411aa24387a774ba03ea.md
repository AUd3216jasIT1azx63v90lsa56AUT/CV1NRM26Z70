### Title
Silent permit failure for DAI-style tokens causes `safeTransferFrom` revert in `pullToken` - (File: src/periphery/MidnightBundles.sol)

### Summary
`pullToken` calls `IERC20Permit(token).permit(...)` using the ERC2612 function selector (`0xd505accf`) inside a `try/catch` that unconditionally swallows all failures. For tokens with a DAI-style permit interface (`bool allowed` instead of `uint256 value`, selector `0x8fcbaf0c`), the call dispatches to a non-existent function, the token reverts, the catch discards it, and the subsequent `safeTransferFrom` reverts due to zero allowance. The user's entire bundle reverts with no meaningful error.

### Finding Description
**Code path:**

`src/periphery/interfaces/IERC20Permit.sol` lines 5–7 define the interface with the ERC2612 selector only:
```solidity
function permit(address owner, address spender, uint256 value, uint256 deadline, uint8 v, bytes32 r, bytes32 s) external;
``` [1](#0-0) 

`src/periphery/MidnightBundles.sol` lines 379–384 in `pullToken`:
```solidity
if (permit.kind == PermitKind.ERC2612) {
    (uint256 deadline, uint8 v, bytes32 r, bytes32 s) =
        abi.decode(permit.data, (uint256, uint8, bytes32, bytes32));
    // Tolerate revert: a third party may have already consumed the permit.
    try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
    SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
``` [2](#0-1) 

**Root cause:** The `try/catch` on line 383 was designed to tolerate front-running (a third party consuming the permit before the bundler call). However, it also silently swallows failures caused by ABI/selector mismatch. A DAI-style token's permit has selector `0x8fcbaf0c`; calling it with selector `0xd505accf` (ERC2612) causes the token to revert with no matching function. The catch discards this, no allowance is set, and `safeTransferFrom` on line 384 reverts.

**Attacker-controlled inputs:** None required. Any unprivileged user (lender, borrower, repayer) who holds a DAI-style token as the market's loan token and submits `PermitKind.ERC2612` with no pre-existing allowance triggers this path. The user is acting exactly as the bundler's API invites them to.

**Why existing checks fail:** The only "protection" is the `try/catch`, which is precisely the mechanism that enables the silent failure. There is no post-permit allowance check (e.g., `IERC20(token).allowance(from, address(this)) >= amount`) to verify the permit actually succeeded before proceeding to `safeTransferFrom`.

### Impact Explanation
Every bundler entry point that calls `pullToken` with `PermitKind.ERC2612` — `buyWithUnitsTargetAndWithdrawCollateral`, `buyWithAssetsTargetAndWithdrawCollateral`, `repayAndWithdrawCollateral`, and collateral supply paths — reverts for any DAI-style token. Users cannot lend, borrow, or repay through the bundler using the permit path, even though the API exposes `PermitKind.ERC2612` as a valid option. The revert surfaces at `safeTransferFrom`, not at the permit call, giving the user no actionable error. [3](#0-2) 

### Likelihood Explanation
DAI is one of the most widely used ERC20 tokens and uses a non-standard permit. Any market created with DAI (or any DAI-fork) as `loanToken` exposes this path to every user who attempts to use the bundler's permit feature without a pre-existing allowance. The precondition (no pre-existing allowance, `PermitKind.ERC2612` selected) is the normal, expected usage of the bundler's permit API. It is repeatable on every call.

### Recommendation
After the `try/catch`, verify that the permit actually granted sufficient allowance before proceeding:

```solidity
try IERC20Permit(token).permit(from, address(this), amount, deadline, v, r, s) {} catch {}
require(
    IERC20(token).allowance(from, address(this)) >= amount,
    PermitFailed()
);
SafeTransferLib.safeTransferFrom(token, from, address(this), amount);
```

Alternatively, add a `PermitKind.DAI` variant that encodes the DAI-style permit ABI (`bool allowed`, `uint256 nonce`) and dispatches to the correct selector. Either fix ensures that a failed permit surfaces as a clear, actionable revert rather than a silent no-op followed by a confusing transfer failure.

### Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {MidnightBundles} from "src/periphery/MidnightBundles.sol";
import {PermitKind, TokenPermit} from "src/periphery/interfaces/IMidnightBundles.sol";

// DAI-style permit mock: different selector (bool allowed, uint256 nonce)
contract DaiStyleToken is MockERC20 {
    mapping(address => mapping(address => uint256)) public allowance;
    function permit(
        address holder, address spender,
        uint256 nonce, uint256 expiry,
        bool allowed, uint8 v, bytes32 r, bytes32 s
    ) external {
        allowance[holder][spender] = allowed ? type(uint256).max : 0;
    }
    // No ERC2612 permit(address,address,uint256,uint256,uint8,bytes32,bytes32)
}

contract DaiPermitBugTest is Test {
    MidnightBundles bundler;
    DaiStyleToken dai;
    address user = address(0xBEEF);

    function setUp() public {
        dai = new DaiStyleToken();
        bundler = new MidnightBundles(address(mockMidnight));
        deal(address(dai), user, 100e18);
        // user has ZERO allowance to bundler
    }

    function testDaiPermitSilentlyFailsThenTransferReverts() public {
        // Encode as ERC2612 permit (wrong selector for DAI)
        TokenPermit memory permit = TokenPermit({
            kind: PermitKind.ERC2612,
            data: abi.encode(block.timestamp + 1, uint8(27), bytes32(0), bytes32(0))
        });

        vm.prank(user);
        // Expected: reverts at safeTransferFrom (not at permit), because
        // the try/catch swallowed the DAI revert and no allowance was set.
        vm.expectRevert(); // safeTransferFrom reverts: insufficient allowance
        bundler.repayAndWithdrawCollateral(
            market, 100e18, user, permit,
            new CollateralWithdrawal[](0), address(0), 0, address(0)
        );

        // Assert: allowance is still zero (permit never executed)
        assertEq(dai.allowance(user, address(bundler)), 0);
    }
}
```

**Expected assertion:** The call reverts at `safeTransferFrom` (not at the permit step), and `dai.allowance(user, address(bundler)) == 0`, confirming the permit silently no-oped and the transfer had no allowance to draw on.
