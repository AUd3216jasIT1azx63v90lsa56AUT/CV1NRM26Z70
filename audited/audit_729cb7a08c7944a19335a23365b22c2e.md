Audit Report

## Title
`buyerAssetsToUnits` unconditionally reverts for valid high-tick sell offers with nonzero settlement fees, DoS-ing `buyWithAssetsTargetAndWithdrawCollateral` - (File: src/periphery/TakeAmountsLib.sol)

## Summary
`TakeAmountsLib.buyerAssetsToUnits` contains a hard `require(buyerPrice <= WAD)` guard that fires for any sell offer where `tickToPrice(tick) + settlementFee > WAD`. `Midnight.take` has no equivalent guard for sell offers and succeeds in exactly this scenario. Because `MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` calls `buyerAssetsToUnits` outside its try/catch block, the entire bundle transaction reverts for any sell offer at a high tick in a market with a nonzero settlement fee, permanently blocking the bundle flow for those offers.

## Finding Description

**Root cause — asymmetric guard in `buyerAssetsToUnits`:**

In `src/periphery/TakeAmountsLib.sol` lines 22–28, for a sell offer (`offer.buy = false`):
- `sellerPrice = offerPrice` (no subtraction)
- `buyerPrice = offerPrice + settlementFee`
- `require(buyerPrice <= WAD, TickLib.PriceGreaterThanOne())` — reverts when `offerPrice + settlementFee > WAD` [1](#0-0) 

In `src/Midnight.sol` lines 358–364, the identical arithmetic is performed for sell offers, but **no `require(buyerPrice <= WAD)` check exists**. The function proceeds to compute `buyerAssets = units.mulDivUp(buyerPrice, WAD)` and continues normally. [2](#0-1) 

**Trigger condition:**

`tickToPrice(MAX_TICK)` evaluates to exactly `WAD = 1e18`. At `tick = MAX_TICK = 5820`, the exponent argument `LN_ONE_PLUS_DELTA * (2910 - 5820)` is large and negative, making `wExp(...)` negligible, so `1e36 / (1e18 + ~0) = 1e18`. Any nonzero `settlementFee` then pushes `buyerPrice` above `WAD`. [3](#0-2) [4](#0-3) 

**Confirmed by test `testSnappedBuyerAssetsBuyerIsLender`** (lines 185–205 of `test/TakeAmountsTest.sol`): it explicitly sets `buyerPrice = tickToPrice(MAX_TICK) + settlementFee`, calls `take` directly, and asserts success — proving `take` handles `buyerPrice > WAD` for sell offers without reverting. [5](#0-4) 

**DoS path in `MidnightBundles`:**

In `src/periphery/MidnightBundles.sol` lines 208–214, `TakeAmountsLib.buyerAssetsToUnits` is called to compute `unitsToTake`. This call is **outside** the try/catch block that wraps `IMidnight(MIDNIGHT).take(...)` at lines 215–221. A revert from `buyerAssetsToUnits` propagates uncaught, reverting the entire `buyWithAssetsTargetAndWithdrawCollateral` transaction. [6](#0-5) 

The NatSpec at line 175 documents this: "Reverts if TakeAmountsLib or ConsumableUnitsLib reverts" — confirming the revert is not caught. [7](#0-6) 

## Impact Explanation
`buyWithAssetsTargetAndWithdrawCollateral` is permanently DoS'd for any sell offer where `tickToPrice(tick) + settlementFee > WAD`. Lenders cannot use the bundle flow to lend against these offers; they must call `Midnight.take` directly, bypassing the bundle's collateral-withdrawal and referral-fee logic. No funds are lost, but a core periphery function is rendered non-functional for a valid and reachable class of offers.

## Likelihood Explanation
`MAX_TICK = 5820` is a valid, accessible tick requiring no special permissions. Settlement fees are a standard protocol feature enabled by market creators. Any market with a nonzero settlement fee and a sell offer at or near `MAX_TICK` triggers this deterministically. No attacker action is required — the condition arises from normal protocol usage.

## Recommendation
Wrap the `TakeAmountsLib.buyerAssetsToUnits` call inside `buyWithAssetsTargetAndWithdrawCollateral` in a try/catch, consistent with how `take` failures are already handled. If `buyerAssetsToUnits` reverts (e.g., `buyerPrice > WAD`), skip the offer rather than reverting the entire transaction. Alternatively, modify `buyerAssetsToUnits` to handle the `buyerPrice > WAD` case for sell offers by returning a units value computed via `mulDivUp(WAD, buyerPrice)` directly, since the function's stated invariant ("not all buyerAssets are reachable") applies only to exact targeting, not to the bundle's fill loop.

## Proof of Concept
1. Deploy a market with any nonzero settlement fee.
2. A borrower posts a sell offer with `tick = MAX_TICK` (valid, no special permissions needed).
3. A lender calls `MidnightBundles.buyWithAssetsTargetAndWithdrawCollateral` targeting that offer.
4. Inside the loop, `TakeAmountsLib.buyerAssetsToUnits` is called at line 209; `buyerPrice = WAD + settlementFee > WAD` triggers `require(buyerPrice <= WAD)` → reverts with `PriceGreaterThanOne`.
5. The revert propagates (no try/catch around `buyerAssetsToUnits`); the entire bundle call reverts.
6. Calling `Midnight.take` directly with the same offer and units succeeds — confirmed by `testSnappedBuyerAssetsBuyerIsLender` in `test/TakeAmountsTest.sol`. [8](#0-7) [9](#0-8) [10](#0-9)
