### Title
Zero-Amount `withdraw()` Reverts with `TransferReturnedFalse()` on `UintReturnToken` Loan Token - (File: src/libraries/SafeTransferLib.sol)

### Summary
`withdraw()` in `src/Midnight.sol` has no `units > 0` guard and unconditionally calls `SafeTransferLib.safeTransfer(market.loanToken, receiver, units)` even when `units == 0`. `SafeTransferLib.safeTransfer` decodes any 32-byte return value as `bool` via `abi.decode(returndata, (bool))`; a loan token whose `transfer()` returns the transferred amount as `uint256` will return `uint256(0)` for a zero-amount call, which ABI-decodes as `bool(false)`, causing an unconditional revert with `TransferReturnedFalse()`. An unprivileged market creator can deploy such a token and create a market, making every `withdraw(market, 0, ...)` call on that market permanently revert.

### Finding Description
**Code path:**

`withdraw()` — `src/Midnight.sol` lines 481–500 — has no non-zero guard on `units`. It always reaches:

```solidity
SafeTransferLib.safeTransfer(market.loanToken, receiver, units);   // line 499
```

`SafeTransferLib.safeTransfer` — `src/libraries/SafeTransferLib.sol` line 21:

```solidity
require(returndata.length == 0 || abi.decode(returndata, (bool)), TransferReturnedFalse());
```

**Root cause:** The check passes only if `returndata` is empty OR decodes as `true`. A token that returns `uint256(transferredAmount)` produces exactly 32 bytes. For `value == 0`, those 32 bytes are all zeros. `abi.decode(zeros32, (bool))` == `false`. The `require` fires.

**Attacker inputs:**
- Deploy `UintReturnToken`: `transfer(address, uint256 v) external returns (uint256) { return v; }`
- Call `touchMarket` / any market-creation path with `loanToken = address(UintReturnToken)`
- Call `withdraw(market, 0, onBehalf, receiver)` — no authorization barrier for `msg.sender == onBehalf`

**Why existing checks fail:**
- `withdraw()` has no `require(units > 0)` — confirmed by grep (no match in `src/Midnight.sol`)
- `SafeTransferLib` only short-circuits on `returndata.length == 0`; a 32-byte `uint256(0)` is not length-zero
- No market-creation validation rejects tokens with non-boolean return types [1](#0-0) [2](#0-1) 

### Impact Explanation
Every call to `withdraw(market, 0, onBehalf, receiver)` on a market whose `loanToken` returns `uint256` from `transfer()` reverts with `TransferReturnedFalse()`. The zero-amount path is permanently broken for that market. Any integration, multicall bundle, or off-chain script that issues a zero-unit withdraw (e.g., to trigger `_updatePosition` side-effects or as a no-op flush) will fail. State changes before the transfer (credit/withdrawable/totalUnits decremented by 0, event emitted) are rolled back, so there is no lasting state corruption — the impact is confined to the unexpected revert. [3](#0-2) 

### Likelihood Explanation
**Preconditions:**
1. Market creator deploys a `UintReturnToken` (trivial, ~5 lines of Solidity).
2. Market creator calls `touchMarket` with that token as `loanToken` (permissionless).
3. Any caller invokes `withdraw(market, 0, self, receiver)` — requires only that `onBehalf == msg.sender` or authorization.

The attack is repeatable on every such market and requires no privileged access, no oracle manipulation, and no existing liquidity. The only limiting factor is that the DoS is scoped to zero-amount withdrawals on attacker-created markets; users who independently choose to interact with such a market are the affected population. [4](#0-3) 

### Recommendation
Add a zero-amount short-circuit in `withdraw()` before the transfer, consistent with how the protocol already handles zero-amount liquidations:

```solidity
function withdraw(Market memory market, uint256 units, address onBehalf, address receiver) external {
    require(onBehalf == msg.sender || isAuthorized[onBehalf][msg.sender], Unauthorized());
+   if (units == 0) return;   // no-op guard; avoids token-specific return-value edge cases
    bytes32 id = touchMarket(market);
    ...
    SafeTransferLib.safeTransfer(market.loanToken, receiver, units);
}
```

Alternatively, add a `value == 0` early-return inside `SafeTransferLib.safeTransfer` to skip the external call entirely when transferring zero tokens, which would also protect `claimSettlementFee`, `claimContinuousFee`, and `flashLoan` from the same class of issue. [5](#0-4) [2](#0-1)

        require(token.code.length > 0, NoCode());
