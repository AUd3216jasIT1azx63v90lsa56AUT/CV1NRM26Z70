### Title
Staker principal permanently locked when claim transfer fails with no alternative recovery path - (File: src/ConfidencePool.sol)

### Summary
After a pool resolves to SURVIVED or EXPIRED, stakers must call `claimSurvived()` or `claimExpired()` to retrieve their principal and bonus. Both functions use a push-payment pattern that transfers exclusively to `msg.sender`. If the staker's address cannot receive the stake token (e.g., blacklisted by USDC after staking), the transfer reverts on every retry and the staker has no alternative recovery path. `sweepUnclaimedBonus()` explicitly reserves the staker's principal, preventing it from being swept. The funds are permanently locked.

### Finding Description
`claimSurvived()` and `claimExpired()` both transfer only to `msg.sender`:

```solidity
// claimSurvived – line 403
stakeToken.safeTransfer(msg.sender, payout);

// claimExpired – line 601
stakeToken.safeTransfer(msg.sender, payout);
``` [1](#0-0) [2](#0-1) 

Once `riskWindowStart != 0`, `withdraw()` is permanently disabled via the one-way latch:

```solidity
if (riskWindowStart != 0 || (state != ... NOT_DEPLOYED ... && ...)) {
    revert WithdrawsDisabled();
}
``` [3](#0-2) 

`sweepUnclaimedBonus()` explicitly reserves the full remaining `totalEligibleStake` for non-claimers, so a blacklisted staker's principal can never be swept:

```solidity
if (totalEligibleStake != 0) {
    reserved = totalEligibleStake;
    if (riskWindowStart != 0) {
        reserved += snapshotTotalBonus - claimedBonus;
    }
}
uint256 amount = freeBalance > reserved ? freeBalance - reserved : 0;
if (amount == 0) revert NothingToSweep();
``` [4](#0-3) 

No "claim on behalf of" or recipient-redirect mechanism exists anywhere in the contract. The staker's `eligibleStake` mapping entry is keyed to their address and can only be claimed by that exact address.

### Impact Explanation
A staker whose address cannot receive the stake token after `riskWindowStart` is sealed loses their entire principal and bonus permanently. The funds remain in the contract indefinitely: `claimSurvived()`/`claimExpired()` always revert, `withdraw()` is permanently disabled, and `sweepUnclaimedBonus()` reserves the principal and cannot sweep it. There is no escape hatch.

### Likelihood Explanation
Low-medium. The scenario requires a token with an address-level blocklist (USDC, USDT, and several other common ERC20s have this feature) to be on the factory allowlist, and the staker's address to be blocked after staking but before claiming. USDC blocklisting of specific addresses (e.g., Tornado Cash-linked addresses, sanctioned entities) is a documented real-world occurrence. The factory allowlist is intended to exclude "defective" tokens, but USDC is a standard ERC20 and the DESIGN.md's stated exclusion targets only tokens that "freeze the pool's own balance" — a per-address blocklist does not freeze the pool balance, only the recipient transfer. [5](#0-4) 

### Recommendation
Replace the push-payment pattern with a recipient parameter or a two-step pull pattern:

- Add `claimSurvivedTo(address recipient)` / `claimExpiredTo(address recipient)` variants that allow the staker (`msg.sender`) to direct their payout to any address they control.
- Alternatively, implement a separate `withdrawStake(address recipient)` escape path that is gated on `hasClaimed[msg.sender] == false` and `eligibleStake[msg.sender] > 0` post-resolution, allowing a staker to redirect principal to a non-blocked address.

### Proof of Concept
1. Factory owner calls `setStakeTokenAllowed(USDC, true)`.
2. Sponsor creates a pool with USDC as the stake token.
3. Alice (EOA) calls `stake(1_000e6)` — 1,000 USDC transferred to the pool.
4. Registry transitions to `UNDER_ATTACK`; `riskWindowStart` is sealed. Alice's `withdraw()` is now permanently disabled (`WithdrawsDisabled`).
5. Registry transitions to `PRODUCTION`; moderator calls `flagOutcome(SURVIVED, false, address(0))`.
6. USDC issuer blocks Alice's address (e.g., regulatory sanction).
7. Alice calls `claimSurvived()` — `stakeToken.safeTransfer(Alice, payout)` reverts because Alice is blocked. Transaction reverts; `hasClaimed[Alice]` is never set.
8. Alice retries indefinitely — every call reverts for the same reason.
9. Alice checks alternatives:
   - `withdraw()` → `WithdrawsDisabled` (permanent).
   - `sweepUnclaimedBonus()` → `reserved = totalEligibleStake` includes Alice's 1,000 USDC; `amount == 0`; reverts `NothingToSweep`.
   - No `claimSurvivedTo` or equivalent exists.
10. Alice's 1,000 USDC is permanently locked in the pool contract with no recovery path.

    function claimSurvived() external nonReentrant {
        if (outcome != PoolStates.Outcome.SURVIVED) revert OutcomeNotSet();
        if (hasClaimed[msg.sender]) revert InvalidAmount();

        uint256 userEligible = eligibleStake[msg.sender];
        if (userEligible == 0) revert InvalidAmount();

        _clampUserSums(msg.sender);

        hasClaimed[msg.sender] = true;

        uint256 bonusShare = _bonusShare(msg.sender, userEligible);
        uint256 payout = userEligible + bonusShare;
        totalEligibleStake -= userEligible;
        claimedBonus += bonusShare;

        delete eligibleStake[msg.sender];
        delete userSumStakeTime[msg.sender];
        delete userSumStakeTimeSq[msg.sender];

        if (!claimsStarted) claimsStarted = true;
        stakeToken.safeTransfer(msg.sender, payout);
        emit ClaimSurvived(msg.sender, userEligible, bonusShare);
    }
